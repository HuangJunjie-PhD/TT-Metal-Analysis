This page documents the error reporting and validation infrastructure in the TensorIR schedule system. It covers the error class hierarchy, validation mechanisms that ensure schedule transformations preserve program semantics, and the configurable error rendering system that provides detailed diagnostic information.

For information about the Schedule API itself and how to use transformation primitives, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api). For information about the internal state management during scheduling, see [Schedule State Management](https://deepwiki.com/tenstorrent/tt-tvm/5.4-schedule-state-management).

## Purpose and Scope

The error reporting and validation system serves two critical functions:

1.   **Pre-condition Validation**: Checks that schedule primitives can be safely applied before transforming the IR
2.   **Error Reporting**: Provides detailed, context-aware error messages with source location information when validation fails

This system ensures that invalid schedule transformations are caught early with clear diagnostic messages, helping users understand what went wrong and how to fix it.

## Validation System

### Validation Categories

The schedule system performs extensive validation across multiple categories:

**Sources**: [src/tir/schedule/analysis/analysis.cc 263-459](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L263-L459)[src/tir/schedule/analysis.h 42-57](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis.h#L42-L57)[src/tir/schedule/primitive/cache_read_write.cc 34-65](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/primitive/cache_read_write.cc#L34-L65)

### Complete Block Validation

A complete block must satisfy three conditions to ensure it can be safely transformed:

**Validation Logic** ([src/tir/schedule/analysis/analysis.cc 211-237](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L211-L237)):

| Condition # | Check | Purpose |
| --- | --- | --- |
| 1 | All block vars are data parallel | Ensures no reduction dependencies |
| 2 | Block is dominant (only writer) | Prevents data races |
| 3 | No read/write buffer overlap | Ensures independence |

**Sources**: [src/tir/schedule/analysis/analysis.cc 211-293](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L211-L293)

The error message includes the complete block definition ([src/tir/schedule/analysis/analysis.cc 239-242](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L239-L242)):

```
Definition of a complete block:
1) All block vars are data parallel
2) Dominant: the block is the only writer of its output, dominating the reader of its output buffers
3) No overlap between the buffers the block reads and writes
```

### Reduction Block Validation

Reduction blocks have additional requirements beyond complete blocks:

**Validation Conditions** ([src/tir/schedule/analysis/analysis.cc 304-327](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L304-L327)):

1.   The block has an `init` statement
2.   All block bindings are quasi-affine expressions
3.   Block vars are either data parallel or reduction
4.   Block is dominant (only writer)
5.   Reduction vars don't index output buffers

**Sources**: [src/tir/schedule/analysis/analysis.cc 304-359](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/analysis/analysis.cc#L304-L359)

| --- | --- |
| No blocks found | `"Cannot find a block with the name: {name}"` |

Schedule transformations preserve span information to enable precise error reporting:

*   **Span Propagation**: IR nodes carry span information through transformations
*   **LocationsOfInterest**: Error classes return specific IR nodes to highlight

## Configuration and Best Practices

### Choosing an Error Render Level

| Use Case | Recommended Level | Rationale |
| --- | --- | --- |
| Interactive development | `"detail"` | Full diagnostics aid debugging |
| Automated testing | `"detail"` | Detailed logs for CI failures |
| Production deployment | `"fast"` | Balance error info with overhead |
| Performance tuning | `"none"` | Minimize error handling cost |
| Meta-schedule search | `"fast"` or `"none"` | Thousands of schedules tested |

### Debug Mask for Additional Validation

The schedule constructor also accepts a `debug_mask` parameter for runtime verification:

The debug mask controls:

*   SRef tree verification after each transformation
*   Cached flag consistency checks
*   Additional invariant assertions

**Sources**: [python/tvm/tir/schedule/schedule.py 144-150](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/tir/schedule/schedule.py#L144-L150)[src/tir/schedule/state.cc 107-124](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/state.cc#L107-L124)

### Enable/Disable Prerequisite Checks

The `enable_check` parameter controls whether prerequisite validation runs:

When `enable_check=False`, transformations may fail with less informative errors, but validation overhead is reduced. This is useful when applying known-good schedule recipes.

**Sources**: [python/tvm/tir/schedule/schedule.py 156-164](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/tir/schedule/schedule.py#L156-L164)[include/tvm/tir/schedule/state.h 85-88](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/schedule/state.h#L85-L88)
