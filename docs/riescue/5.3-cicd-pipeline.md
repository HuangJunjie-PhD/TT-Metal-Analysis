This document describes the CI/CD pipeline implementation in RiESCUE using GitHub Actions. The pipeline enforces code quality, runs automated tests, and deploys documentation. For information about local code quality tools and pre-commit hooks, see [Code Quality and Linting](https://deepwiki.com/tenstorrent/riescue/5.2-code-quality-and-linting). For documentation build configuration, see [Documentation Deployment](https://deepwiki.com/tenstorrent/riescue/5.3.2-documentation-deployment).

**Sources:**[.github/workflows/smoke.yml 1-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L1-L100)

## Overview

The CI/CD pipeline is defined in the `smoke` workflow, which runs on every push to the repository. It implements a multi-stage validation process with conditional execution based on the target branch. The pipeline ensures that code meets quality standards before merging and automatically deploys documentation updates from the main branch.

**Key characteristics:**

*   **Multi-stage validation**: Sequential jobs with dependency management
*   **Branch-aware execution**: Different behavior for main vs. feature branches
*   **Containerized testing**: Uses Rocky Linux 9.1 for reproducible test environments
*   **Automated deployment**: Documentation published to GitHub Pages on main branch

**Sources:**[.github/workflows/smoke.yml 1-5](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L1-L5)

## Pipeline Architecture

**Pipeline Job Execution Flow**

The pipeline uses a sequential dependency chain where each job depends on the success of its predecessor. The `lint` job depends on `check-spdx-headers`, and both `test` and `deploy-docs` depend on `lint`. However, only one of the final two jobs executes based on the branch condition.

**Sources:**[.github/workflows/smoke.yml 6-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L6-L100)

## Job Stages

### SPDX Header Check Job

The first job validates that all source files contain proper SPDX license headers. This ensures compliance with licensing requirements before any other validation occurs.

**Configuration:**

*   **Runner**: `ubuntu-latest`
*   **License validation**: Apache-2.0
*   **Action**: `enarx/spdx@master`

**Sources:**[.github/workflows/smoke.yml 7-14](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L7-L14)

### Lint Job

The `lint` job performs three validation checks in sequence: dependency installation, code linting, and documentation build verification.

**Job Configuration:**

| Aspect | Value |
| --- | --- |
| Runner | `ubuntu-latest` |
| Python Version | 3.9 |
| Depends On | `check-spdx-headers` |

**Steps:**

1.   **Repository checkout** - Uses `actions/checkout@v4`
2.   **Python setup** - Configures Python 3.9 environment
3.   **Dependency installation** - Installs wheel and RiESCUE package in editable mode
4.   **Linting** - Runs `flake8` and `black --check`
5.   **Documentation check** - Builds docs with `--check` flag for validation

**Lint Job Step Execution**

The lint checks enforce code style consistency and verify that documentation can build successfully. The `--check` flag on the documentation build ensures that any Sphinx warnings or errors will cause the job to fail.

**Sources:**[.github/workflows/smoke.yml 16-33](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L16-L33)

### Test Job

The `test` job runs comprehensive integration tests in a containerized Rocky Linux 9.1 environment. This job only executes for non-main branches due to the conditional: `if: github.ref != 'refs/heads/main'`.

**Job Configuration:**

| Aspect | Value |
| --- | --- |
| Runner | `ubuntu-latest` |
| Container | `rockylinux:9.1` |
| Condition | Non-main branches only |
| Depends On | `lint` |

**Test Job Detailed Execution Flow**

#### Step 1: Install Base Dependencies

Executes the `build_base_deps.sh` script to install Rocky Linux system packages required for building C/C++ dependencies.

**Packages installed:**

*   Build tools: `gcc-toolset-12`, `make`, `bc`, `perl`
*   Libraries: `boost`, `boost-devel`, `openssl-devel`, `zlib-devel`
*   Python: `python3-pip`, `python3-devel`, `python-sphinx`
*   Other: `git`, `wget`, `sudo`, `dtc`, `numactl-libs`

**Sources:**[.github/workflows/smoke.yml 43-45](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L43-L45)[infra/build_base_deps.sh 1-39](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_base_deps.sh#L1-L39)

#### Step 2: Download and Configure RISC-V Toolchain

Downloads a pre-built RISC-V GNU toolchain from the official releases and configures environment variables.

**Toolchain details:**

*   **Version**: `2025.05.30` nightly build
*   **Target**: `riscv64-elf-ubuntu-22.04`
*   **Installation path**: `$HOME/RV_TOOLCHAIN/riscv/bin`
*   **Environment variables set**: 
    *   `RV_TOOLCHAIN`: Points to toolchain bin directory
    *   `PATH`: Updated to include toolchain binaries

**Sources:**[.github/workflows/smoke.yml 48-54](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L48-L54)

#### Step 3: Install Python Dependencies

Installs the RiESCUE package and its Python dependencies using pip.

**Commands executed:**

`python3 -m pip install wheelpython3 -m pip install -e .`
This performs an editable installation, making all entry points (`riescued`, `riescuec`, `ctk`) available.

**Sources:**[.github/workflows/smoke.yml 56-59](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L56-L59)

#### Step 4: Build Whisper Simulator

Builds the Whisper instruction set simulator from source using the build script.

**Build configuration:**

*   **Environment variable**: `NO_SINGULARITY=1` (skips container checks)
*   **Install location**: `/usr/bin/whisper`
*   **Environment variable set**: `WHISPER_PATH=/usr/bin/whisper`

The build script clones the Whisper repository and compiles it with appropriate flags. See [Building Whisper](https://deepwiki.com/tenstorrent/riescue/3.2.2-building-whisper) for more details on the build process.

**Sources:**[.github/workflows/smoke.yml 61-66](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L61-L66)

#### Step 5: Build Spike Simulators

Builds both the standard Spike and TT-specific `tt_spike` simulators from source.

**Build configuration:**

*   **Environment variable**: `NO_SINGULARITY=1`
*   **Install locations**: 
    *   `/usr/local/bin/spike`
    *   `/usr/local/bin/tt_spike`

See [Building Spike](https://deepwiki.com/tenstorrent/riescue/3.2.3-building-spike) for detailed information about the build process and configuration.

**Sources:**[.github/workflows/smoke.yml 68-70](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L68-L70)

#### Step 6: Run Unit Tests

Executes all test files in the `tests/` directory using Python's unittest discovery mechanism.

**Test discovery pattern:**

`python3 -m unittest discover -s tests/ -p "*_test.py"`
This finds and runs all test files matching the pattern `*_test.py` in the tests directory. For more information about the test structure and how to run tests locally, see [Automated Testing](https://deepwiki.com/tenstorrent/riescue/5.3.1-automated-testing).

**Sources:**[.github/workflows/smoke.yml 72-73](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L72-L73)

### Deploy Documentation Job

The `deploy-docs` job builds and publishes documentation to GitHub Pages. It only executes on the main branch: `if: github.ref == 'refs/heads/main'`.

**Job Configuration:**

| Aspect | Value |
| --- | --- |
| Runner | `ubuntu-latest` |
| Python Version | 3.9 |
| Condition | Main branch only |
| Depends On | `lint` |
| Permissions | `contents: read`, `pages: write`, `id-token: write` |

**Documentation Deployment Flow**

The deployment process builds the Sphinx documentation and uploads it as a GitHub Pages artifact. The final deploy step publishes the artifact to the GitHub Pages site.

**Build command:**

`./docs/build.py --build_dir=public --source_dir=docs/source`
This differs from the lint job's documentation check by omitting the `--check` flag, allowing the full build to proceed even with warnings. The built HTML is output to the `./public` directory.

**Sources:**[.github/workflows/smoke.yml 75-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L75-L100)

## Branch-Based Conditional Execution

The pipeline implements different execution paths based on the target branch, optimizing CI resources and ensuring appropriate validation for each context.

**Conditional Logic Summary**

| Job | Main Branch | Feature Branch |
| --- | --- | --- |
| `check-spdx-headers` | ✓ Runs | ✓ Runs |
| `lint` | ✓ Runs | ✓ Runs |
| `test` | ✗ Skipped | ✓ Runs |
| `deploy-docs` | ✓ Runs | ✗ Skipped |

**Rationale:**

*   **Feature branches**: Full test suite validates changes before merge
*   **Main branch**: Documentation deployment occurs immediately; tests are assumed to have passed during PR review

The conditions are implemented using GitHub Actions `if` clauses:

*   Test job: `if: github.ref != 'refs/heads/main'`
*   Deploy docs job: `if: github.ref == 'refs/heads/main'`

**Sources:**[.github/workflows/smoke.yml 38-39](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L38-L39)[.github/workflows/smoke.yml 78](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L78-L78)

## Integration with Development Workflow

**Complete Development and CI/CD Flow**

The CI/CD pipeline integrates with local development practices to create a comprehensive quality assurance process:

1.   **Local development**: Developers write code and use pre-commit hooks for immediate feedback
2.   **Pre-commit validation**: Automated local checks catch basic issues before commit
3.   **CI validation**: GitHub Actions pipeline provides additional validation in clean environments
4.   **Branch-specific actions**: Tests run on feature branches; docs deploy from main
5.   **Feedback loop**: Failures trigger local fixes and re-submission

**Sources:**[.github/workflows/smoke.yml 1-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L1-L100)[.github/CONTRIBUTING.md 36-63](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L36-L63)

## Testing Environment Details

### Container vs. Native Execution

The CI pipeline uses different execution environments for different job types:

| Job | Environment | Reason |
| --- | --- | --- |
| SPDX Check | Native Ubuntu | Lightweight license validation |
| Lint | Native Ubuntu | Python-only, minimal dependencies |
| Test | Rocky Linux Container | Requires compiled C++ dependencies |
| Deploy | Native Ubuntu | Python-only documentation build |

The test job requires a Rocky Linux container because:

*   Whisper and Spike must be compiled from source
*   System-level dependencies (gcc-toolset-12, boost libraries) need specific versions
*   Reproducibility requires matching the production container environment

**Sources:**[.github/workflows/smoke.yml 37](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L37-L37)

### Environment Variables

The test job sets several environment variables to configure tool paths:

`# RISC-V toolchain locationRV_TOOLCHAIN=$HOME/RV_TOOLCHAIN/riscv/bin # Whisper simulator pathWHISPER_PATH=/usr/bin/whisper # NO_SINGULARITY flag for build scriptsNO_SINGULARITY=1`
The `NO_SINGULARITY=1` flag tells build scripts to skip container-related checks since the job is already running inside a container. See [Container Management Commands](https://deepwiki.com/tenstorrent/riescue/3.1.2-container-management-commands) for more information about this flag.

**Sources:**[.github/workflows/smoke.yml 53-54](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L53-L54)[.github/workflows/smoke.yml 66](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L66-L66)

## Pull Request Workflow

Pull requests trigger the CI pipeline automatically on every push. The pipeline provides quality gates that must pass before code can be merged.

**PR Validation Process:**

1.   **Initial validation**: SPDX headers and lint checks provide fast feedback (~2-3 minutes)
2.   **Comprehensive testing**: Full test suite runs in containerized environment (~15-20 minutes)
3.   **Review required**: PR cannot merge until all checks pass and review is approved
4.   **Post-merge**: Documentation automatically deploys when PR merges to main

**Required checks for merge:**

*   ✓ SPDX header validation
*   ✓ Code formatting (black)
*   ✓ Linting (flake8)
*   ✓ Documentation build succeeds
*   ✓ All unit tests pass
*   ✓ At least one approving review

For more information about contributing and PR requirements, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/riescue/5.1-contributing-guidelines).

**Sources:**[.github/CONTRIBUTING.md 38-47](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L38-L47)[.github/ISSUE_TEMPLATE/pull_request_template.md 1-19](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/pull_request_template.md?plain=1#L1-L19)

## Workflow Permissions

The `deploy-docs` job requires elevated GitHub permissions to publish to GitHub Pages:

`permissions:  contents: read       # Read repository contents  pages: write         # Write to GitHub Pages  id-token: write      # Generate OIDC token for deployment`
These permissions follow the principle of least privilege:

*   Other jobs have default permissions (read-only)
*   Only the deployment job can write to GitHub Pages
*   Permissions are explicitly declared for security transparency

**Sources:**[.github/workflows/smoke.yml 79-82](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L79-L82)

## Failure Handling and Debugging

When the CI pipeline fails, developers should follow this debugging process:

1.   **Check the failing job**: GitHub Actions UI shows which job failed
2.   **Examine step logs**: Each step's output is individually viewable
3.   **Reproduce locally**: Use the same commands from the workflow file
4.   **Fix and re-push**: Pushing new commits triggers a new pipeline run

**Common failure scenarios:**

| Failure Type | Job | Typical Cause | Solution |
| --- | --- | --- | --- |
| SPDX header missing | `check-spdx-headers` | New file without license header | Add SPDX comment to file |
| Linting error | `lint` | Code style violation | Run `black .` locally |
| Flake8 error | `lint` | PEP-8 violation | Fix reported issues |
| Doc build failure | `lint` | Sphinx error or warning | Check `docs/build.py` output |
| Test failure | `test` | Logic error or broken dependency | Debug failing test locally |
| Deploy failure | `deploy-docs` | Sphinx error in full build | Run docs build locally |

For local testing procedures, see [Running Tests Locally](https://deepwiki.com/tenstorrent/riescue/5.3.1-automated-testing).

**Sources:**[.github/workflows/smoke.yml 1-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L1-L100)[.github/CONTRIBUTING.md 65-71](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L65-L71)

## Relationship to Local Development Tools

The CI pipeline mirrors local development tools to ensure consistency:

| Tool | Local Usage | CI Usage |
| --- | --- | --- |
| `black` | Pre-commit hook | Lint job step |
| `flake8` | Pre-commit hook | Lint job step |
| `pyright` | Manual/IDE integration | Not in CI (optional) |
| `unittest` | Manual execution | Test job step |
| `docs/build.py` | Manual build | Lint and deploy jobs |

The pre-commit configuration should match the CI lint configuration to catch issues before pushing. For setup instructions, see [Pre-commit Hooks](https://deepwiki.com/tenstorrent/riescue/5.2.4-pre-commit-hooks).
