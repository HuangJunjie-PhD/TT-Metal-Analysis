# Test Report Parsers

Relevant source files
*   [.github/actions/collect_data/src/parsers/parser.py](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parser.py)
*   [.github/actions/collect_data/src/test_parser.py](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/test_parser.py)

The parser subsystem in `tt-github-actions` is responsible for transforming raw test output files—such as JUnit XML, JSON, or TAR archives—into standardized Pydantic models (`Test` or `OpTest`). This abstraction allows the data collection pipeline to handle diverse test frameworks (Pytest, Unittest) and specialized report formats used across Tenstorrent projects.

### Parser Architecture

All parsers inherit from a common interface defined in the `Parser` abstract base class. This design ensures that the collection logic can treat all report formats uniformly through a two-step process: identification and extraction.

#### The Parser Interface

The `Parser` class [.github/actions/collect_data/src/parsers/parser.py 14-39](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/%20.github/actions/collect_data/src/parsers/parser.py#L14-L39) defines two mandatory methods:

*   `can_parse(filepath)`: A heuristic check (usually based on file extension or internal structure) to determine if the parser is compatible with the file.
*   `parse(filepath, ...)`: The execution logic that reads the file and returns a list of test objects. If an unrecoverable error occurs during this phase, it raises a `ParserError`[.github/actions/collect_data/src/parsers/parser.py 8-11](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/%20.github/actions/collect_data/src/parsers/parser.py#L8-L11)

#### Parser Dispatch Chain

The `test_parser.py` module acts as the entry point for file processing. It maintains a prioritized list of concrete parser instances and iterates through them until a suitable match is found.

### Available Parsers

The system includes several concrete implementations tailored to different testing environments. These are categorized into standard JUnit XML parsers and specialized parsers for internal tools.

#### JUnit XML Parsers (Pytest & Unittest)

These parsers handle standard XML reports generated by Python testing frameworks. They extract test case names, durations, and statuses (Passed, Failed, Skipped, Error), and map them to the internal `TestStatus` enum.

*   **PythonPytestParser**: Optimized for Pytest-specific XML structures and property tagging.
*   **PythonUnittestParser**: Handles standard library `unittest` XML outputs.

For details on XML parsing logic and status mapping, see [JUnit XML Parsers (Pytest & Unittest)](https://deepwiki.com/tenstorrent/tt-github-actions/3.1-junit-xml-parsers-(pytest-and-unittest)).

#### Specialized Parsers

Some Tenstorrent projects require custom parsing logic to handle complex data, such as operation compilation statuses or hardware-specific metadata.

*   **TTTorchModelTestsParser**: Extracts detailed operation and tensor metadata from TAR archives.
*   **ParameterSupportTestParser**: Parses JSON reports detailing parameter support across different hardware backends.
*   **BuilderPytestParser**: An extension for OpTests that captures hardware properties like `card_type`, `backend`, and `failure_stage`.

For details on these specialized formats, see [Specialized Parsers (TTTorch, ParameterSupport, BuilderPytest)](https://deepwiki.com/tenstorrent/tt-github-actions/3.2-specialized-parsers-(tttorch-parametersupport-builderpytest)).

### Parser Selection Registry

The registry in `test_parser.py` determines the order in which parsers are attempted. Specialized parsers are generally placed higher in the list to prevent generic XML parsers from consuming files that contain richer, specific metadata.
