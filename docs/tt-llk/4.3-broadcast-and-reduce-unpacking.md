This page documents the unpacker subsystem's support for **broadcast** and **reduce** operations. Broadcasting enables element-wise operations between tiles of different effective sizes by replicating data along specific dimensions (rows, columns, or scalar). Reduce operations require specialized unpacker configuration to transpose and pool data along specified dimensions.

For general unpacker configuration and operation, see [Core Unpacker Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/4.1-core-unpacker-operations-and-configuration). For the corresponding math operations that consume broadcast/reduce data, see [Element-wise Operations](https://deepwiki.com/tenstorrent/tt-llk/5.2-element-wise-operations) and [Reduction and Pooling Operations](https://deepwiki.com/tenstorrent/tt-llk/5.5-reduction-and-pooling-operations).

## Broadcast Types and Data Patterns

The unpacker supports four broadcast modes defined by the `BroadcastType` enumeration:

| Broadcast Type | Description | Data Replication Pattern | Source Register |
| --- | --- | --- | --- |
| `NONE` | No broadcasting | Full tile unpacked normally | SrcA and SrcB |
| `ROW` | Row broadcast | Single row replicated across tile | SrcB |
| `COL` | Column broadcast | Single column replicated across tile | SrcB |
| `SCALAR` | Scalar broadcast | Single value replicated across tile | SrcB |

**Key principle**: When broadcasting is enabled, the data to be broadcast is always loaded into **SrcB**, while the full-size operand goes into **SrcA**. The unpacker then configures special address increment patterns to reuse the broadcast data.

## Broadcast Unpacking Implementation

### Row Broadcast (BroadcastType::ROW)

For row broadcasting, a single row of data is unpacked into SrcB and reused across multiple face rows by manipulating the Z address counter.

**Configuration characteristics:**

*   Unpacks to **SrcB** register
*   **Z counter** is set to 0 and not incremented during unpacking
*   Uses replay buffer instructions to unpack the same row multiple times
*   Outer loop iterates once, inner loop handles face replication

**Diagram: Row Broadcast Address Control**

**Implementation details:**

The key instructions:

*   `unpack_srcb_unpack_srcb` - Replay buffer that unpacks SrcB twice
*   `srcb_clear_z` - `TT_OP_SETADCZW(p_setadc::UNP_B, 0, 0, 0, 0, 0b0001)` sets Z counter to 0

### Column Broadcast (BroadcastType::COL)

Column broadcasting unpacks a column of data and replicates it horizontally across the tile by using a Z counter increment of 2, which causes the unpacker to read alternating faces.

**Configuration characteristics:**

*   Unpacks to **SrcB** register
*   **Z counter increment** is set to 2 (accesses faces 0, 2 or 1, 3)
*   Outer loop runs twice (for 4-face tiles) to handle both columns
*   Math unit clears SrcB manually between columns

**Diagram: Column Broadcast Z Counter Pattern**

**Implementation details:**

Key operation: `srcb_set_z_2 = TT_OP_SETADCZW(p_setadc::UNP_B, 0, 0, 0, 2, 0b0001)` sets the Z counter to 2.

**Sources:**

### Scalar Broadcast (BroadcastType::SCALAR)

Scalar broadcast unpacks a single value and replicates it across the entire tile by not incrementing the Z counter at all.

**Configuration characteristics:**

*   Unpacks to **SrcB** register
*   **Z counter increment** is 0 (stays at same position)
*   Single outer loop, multiple inner loops for face count
*   Simplest broadcast pattern

**Implementation:**

Key instruction: `unpack_srcb_inc_z_0 = TT_OP_UNPACR(SrcB, 0b0 /*Z inc*/, ...)` - Z increment is 0.

## Unpack-to-Dest Broadcast for 32-bit Formats

For 32-bit data formats (Float32, Int32, Tf32), the unpacker can bypass source registers and write directly to destination registers. This requires different broadcast handling.

**Special considerations:**

*   Row and scalar broadcasts write single rows/values directly to destination
*   Column broadcasts use `D2B` (dest-to-B) and `B2D` (B-to-dest) moves with transpose
*   Math operations handle the actual broadcast replication

## MOP Programming for Broadcast Operations

Micro-operation (MOP) templates are programmed differently for each broadcast type to achieve the required data replication patterns.

**MOP Template Structure:**

### MOP Instruction Types for Broadcasting

| Broadcast Type | Outer Loop | Inner Loop | Key Instructions |
| --- | --- | --- | --- |
| `NONE` | 1 | num_faces | `unpack_srca`, `unpack_srcb` |
| `ROW` | 1 | 1 | `unpack_srcb_unpack_srcb` (replay), `srcb_clear_z` |
| `COL` | 1 | 1 | `unpack_srcb`, `srcb_set_z_2` |
| `SCALAR` | 1 | 1 | `unpack_srcb_inc_z_0` |

## Address Counter and Register Configuration

Broadcast and reduce operations rely on careful management of address counters, particularly the **Z counter** which controls which face is accessed.

### Address Counter Registers

**Key registers:**

*   **X counter** (`SETADCXX`) - Controls data within a face (rows × columns)
*   **Y counter** (`SETADCXY`) - Controls stride between rows
*   **Z counter** (`SETADCZW`) - Controls face selection (0-3)
*   **W counter** (`SETADCZW`) - Controls tile selection in multi-tile operations
