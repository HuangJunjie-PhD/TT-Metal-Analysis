## CUDA Graphs

CUDA graphs reduce kernel launch overhead by capturing a sequence of GPU operations into a reusable graph structure. This eliminates the per-iteration overhead of launching individual kernels from the CPU.

### CUDA Graph Capture Process

The model runner captures CUDA graphs for different batch sizes to handle varying request loads efficiently:

**Sources:**[vllm/v1/worker/gpu_model_runner.py 1343-1468](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1343-L1468)[vllm/v1/worker/gpu_worker.py 320-344](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L320-L344)

### Graph Capture Implementation

The `GPUModelRunner.capture_model()` method captures CUDA graphs for predefined batch sizes:

**Key steps:**

1.   **Dispatcher Initialization**: Creates `CudagraphDispatcher` to manage multiple graph instances [vllm/v1/worker/gpu_model_runner.py 1356](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1356-L1356)
2.   **Size Iteration**: Loops through `cudagraph_capture_sizes` in descending order [vllm/v1/worker/gpu_model_runner.py 1361](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1361-L1361)
3.   **Graph Capture Context**: Uses `torch.cuda.graph()` context manager [vllm/v1/worker/gpu_model_runner.py 1386](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1386-L1386)
4.   **Dummy Run**: Executes `_dummy_run()` to capture operations [vllm/v1/worker/gpu_model_runner.py 1387](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1387-L1387)
5.   **Warmup Replays**: Replays the graph multiple times to ensure stability [vllm/v1/worker/gpu_model_runner.py 1398-1401](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1398-L1401)
6.   **Memory Tracking**: Calculates memory used by captured graphs [vllm/v1/worker/gpu_model_runner.py 1441-1461](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1441-L1461)

**Configuration:**

*   Batch sizes to capture are determined by `compilation_config.cudagraph_capture_sizes`[vllm/v1/worker/gpu_model_runner.py 340-342](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L340-L342)
*   Sorted in descending order: larger batches first [vllm/v1/worker/gpu_model_runner.py 341](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L341-L341)

**Sources:**[vllm/v1/worker/gpu_model_runner.py 1343-1468](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L1343-L1468)[vllm/v1/worker/gpu_model_runner.py 339-343](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L339-L343)

### Runtime Graph Dispatching

The `CudagraphDispatcher` selects the appropriate captured graph at runtime based on the current batch size:

The dispatcher maintains a dictionary mapping batch sizes to their corresponding CUDA graph objects. When a new batch arrives, it selects the largest captured graph that doesn't exceed the current batch size.

**Sources:**[vllm/v1/cudagraph_dispatcher.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/cudagraph_dispatcher.py)[vllm/v1/worker/gpu_model_runner.py 426-428](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L426-L428)

## Compilation and Kernel Warmup

Model compilation and kernel warmup optimize execution speed by pre-compiling frequently used code paths and tuning kernel parameters.

### Compilation Levels

vLLM supports multiple compilation levels through `CompilationConfig`:

| Level | Description | Use Case |
| --- | --- | --- |
| `NO_COMPILATION` | No torch.compile usage | Debugging, LoRA |
| `DYNAMO_AS_IS` | Basic dynamo without optimizations | Testing |
| `DYNAMO_ONCE` | Compile entire model once | TPU, XPU |
| `PIECEWISE` | Compile individual components | GPU (default) |

**Sources:**[vllm/config/compilation.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/config/compilation.py)[vllm/platforms/cpu.py 201-232](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L201-L232)

## Speculative Decoding

Speculative decoding improves generation speed by using a smaller draft model to predict multiple tokens ahead, which are then verified by the target model.

### Architecture Overview

**Supported methods:**

*   **Ngram**: Statistical prediction based on n-gram patterns [vllm/v1/spec_decode/ngram_proposer.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/spec_decode/ngram_proposer.py)

### Rejection Sampling

The `RejectionSampler` verifies draft tokens against the target model's distribution:

**Algorithm:**

1.   Compute target probabilities for draft tokens [vllm/v1/sample/rejection_sampler.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/sample/rejection_sampler.py)
2.   Accept token if `draft_prob <= target_prob` (with random threshold)
3.   Resample rejected position using adjusted target distribution
4.   Discard all tokens after first rejection

**Key optimizations:**

*   **Batched verification**: Processes multiple requests simultaneously
*   **Early termination**: Stops at first rejected token
*   **State updates**: Efficiently updates `num_accepted_tokens`[vllm/v1/worker/gpu_input_batch.py 223-228](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_input_batch.py#L223-L228)

**Sources:**[vllm/v1/sample/rejection_sampler.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/sample/rejection_sampler.py)[tests/v1/sample/test_rejection_sampler.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/v1/sample/test_rejection_sampler.py)

| --- | --- | --- |

| `block_size` | KV cache granularity | 16 for GPU, 64 for XPU, 128 for MLA |
| `enable_chunked_prefill` | Prefill optimization | Enable for long contexts |
| `cudagraph_capture_sizes` | CUDA graph batch sizes | Align with typical request sizes |
