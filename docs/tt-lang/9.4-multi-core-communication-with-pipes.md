This page explains how to write kernels that execute across multiple cores and communicate between them using pipes. Multi-core programming enables parallel execution of operations across a grid of cores, with pipes providing the communication mechanism for data transfer between cores.

## Overview

tt-lang supports multi-core kernel execution where operations are distributed across a grid of cores. Each core executes the same kernel code but can behave differently based on its position in the grid and its role in pipe communication patterns.

Key concepts:

*   **Grid**: A multi-dimensional arrangement of cores specified in the `@ttl.operation` decorator (e.g., `grid="full"` to use the entire device grid). [examples/eltwise_pipe.py 15-16](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/eltwise_pipe.py#L15-L16)[examples/eltwise_add.py 12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/eltwise_add.py#L12-L12)
*   **Pipe**: A unidirectional communication channel from a source core to one or more destination cores. [python/sim/pipe.py 42-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L42-L62)
*   **PipeNet**: A network of pipes used to manage complex communication patterns like multicasts or all-to-all transfers. [python/sim/pipe.py 210-225](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L210-L225)
*   **Conditional Execution**: Functions like `if_src()` and `if_dst()` enable role-based execution within a `PipeNet` context. [python/sim/pipe.py 255-285](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L255-L285)

## The Pipe and PipeNet Abstraction

A `Pipe` represents a unidirectional communication channel between cores. It specifies a source core (`src`) and a destination specification (`dst`), which can be a single coordinate (unicast) or a `NodeRange` with slices (multicast). [python/sim/pipe.py 42-65](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L42-L65)

## Operational Semantics and Synchronization

Pipe transfers in tt-lang follow a specific protocol to ensure data integrity across the NoC (Network on Chip).

*   **No In-transit Buffering**: A pipe has no hidden buffer. The destination storage is the `DataflowBuffer` (DFB) block reserved by the receiver. [docs/development/PipeNets.md 60-61](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L60-L61)
*   **Receive Posting**: `ttl.copy(pipe, dst_blk)` posts a receive, publishing the address where the sender should write. [docs/development/PipeNets.md 62-64](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L62-L64)
*   **Send Execution**: `ttl.copy(src_blk, pipe)` starts a send. It waits until the destination addresses are posted, writes directly to the receiver's SRAM, and signals completion. [docs/development/PipeNets.md 67-71](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L67-L71)
*   **Synchronization**: Receivers wait on the transfer handle returned by `ttl.copy(pipe, dst_blk)`, which blocks until the sender signals that the payload write is complete. [docs/development/PipeNets.md 65-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L65-L66)

## Conditional Execution Based on Pipe Roles

The `PipeNet` class provides `if_src()` and `if_dst()` methods. These allow a single kernel to define different logic for the sender and the receivers of a data transfer.

### Role-Based Execution Flow

**Key API Methods:**

*   `pipe_net.is_active()`: Returns `True` if the current node is a participant in any pipe within the network. [python/sim/pipe.py 227-237](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L227-L237)
*   `pipe_net.if_src(callback)`: Executes `callback` only on the source core. The callback receives a `SrcPipeIdentity`. [python/sim/pipe.py 255-269](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L255-L269)
*   `pipe_net.if_dst(callback)`: Executes `callback` only on destination cores. The callback receives a `DstPipeIdentity`. [python/sim/pipe.py 271-285](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/pipe.py#L271-L285)

## Compilation and Resource Planning

The `PipeLowering` pass transforms logical pipe operations into low-level `TTKernel` commands involving semaphores and NoC writes. [lib/Dialect/TTL/Transforms/PipeLowering.cpp 1-100](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeLowering.cpp#L1-L100)

*   **PipeGraph**: Analyzes the module to validate that each logical pipe has a consistent destination DFB and sufficient blocks for overlapping writes. [lib/Dialect/TTL/Transforms/PipeGraph.h 22-25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeGraph.h#L22-L25)
*   **Gather Slots**: Pipes sharing a receiver DFB are assigned distinct "gather slots" (indices within the DFB) to prevent overwriting data. [lib/Dialect/TTL/Transforms/PipeGraph.cpp 58-126](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeGraph.cpp#L58-L126)
*   **Resource Limits**: The compiler verifies that the number of required semaphores fits within hardware limits. [lib/Dialect/TTL/Transforms/PipeLowering.h 151-153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeLowering.h#L151-L153)

## Common Communication Patterns

### Pattern 1: Gather

Multiple sources send data to a single destination core. This requires the receiver DFB to have a `block_count` at least equal to the number of incoming pipes. [test/ttlang/Dialect/TTL/Transforms/convert_pipe_ops_invalid.mlir 10-36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/Transforms/convert_pipe_ops_invalid.mlir#L10-L36)

### Pattern 2: Multicast

A single source sends data to multiple destination cores. The compiler ensures that all receivers publish the same destination SRAM address for hardware NoC multicast compatibility. [test/ttlang/Dialect/TTL/Transforms/convert_pipe_ops_invalid.mlir 43-71](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/Transforms/convert_pipe_ops_invalid.mlir#L43-L71)

### Pattern 3: Complex Liveness (K-Split)

In advanced kernels like K-split matmul, multiple `PipeNet` routes (row/column/helper) are active concurrently. The compiler performs liveness-based resource allocation to manage semaphores. [test/python/pipe/pipenet_resource_allocation.py 68-104](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/pipe/pipenet_resource_allocation.py#L68-L104)
