## Purpose and Scope

This document explains the automated release workflow that manages version bumping, changelog generation, artifact building, and publication to multiple distribution channels (GitHub Releases, PyPI, TestPyPI, and Debian repositories). The workflow implements a temporary branch isolation pattern to safely modify version files and generate changelogs before merging back to the main branch.

For information about building Debian packages specifically, see [Debian Package Build System](https://deepwiki.com/tenstorrent/tt-tools-common/5.1-debian-package-build-system). For Python package building details, see [Python Package Build](https://deepwiki.com/tenstorrent/tt-tools-common/5.2-python-package-build). For PyPI publication mechanics, see [Publishing to PyPI and TestPyPI](https://deepwiki.com/tenstorrent/tt-tools-common/5.4-publishing-to-pypi-and-testpypi).

## Workflow Overview

The release workflow is defined in [.github/workflows/release.yml 1-453](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L1-L453) and can be triggered manually via `workflow_dispatch` with a required input parameter specifying the version bump type (patch, minor, or major). The workflow orchestrates seven sequential jobs that collectively handle all aspects of creating and publishing a new release.

| Job Name | Primary Responsibility | Key Dependencies |
| --- | --- | --- |
| `create-temp-branch` | Creates isolation branch `rc-temp-<sha>-<timestamp>` | None |
| `versionchange` | Bumps version in `pyproject.toml` using `bump2version` | `create-temp-branch` |
| `changelogs` | Generates `debian/changelog` using `gbp dch` | `versionchange` |
| `build_all_depends` | Orchestrates Debian and Python builds | `changelogs` |
| `tagrelease` | Creates Git tag `v<version>` | `build_all_depends` |
| `generate-release` | Creates GitHub Release with artifacts | `tagrelease` |
| `mergeback` | Merges temp branch to main and deletes temp branch | `generate-release` |

Sources: [.github/workflows/release.yml 1-453](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L1-L453)

## Workflow Trigger and Inputs

The workflow can be triggered manually through the GitHub Actions UI or programmatically via workflow calls. The version bump type determines which semantic version component gets incremented.

The `version_bump_type` parameter is passed to the `bump2version` command in [.github/workflows/release.yml 116-121](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L116-L121) which modifies [pyproject.toml](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml) according to semantic versioning rules:

*   **patch**: Increments the patch version (1.4.32 → 1.4.33)
*   **minor**: Increments the minor version, resets patch (1.4.32 → 1.5.0)
*   **major**: Increments the major version, resets minor and patch (1.4.32 → 2.0.0)

Sources: [.github/workflows/release.yml 4-16](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L4-L16)

## Temporary Branch Isolation Pattern

The workflow implements a temporary branch strategy to isolate version changes and changelog generation before merging to main. This prevents conflicts and ensures atomic commits.

The temporary branch name is generated using the current Git SHA and timestamp in [.github/workflows/release.yml 54-58](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L54-L58):

This naming convention ensures uniqueness and provides context about when and from which commit the release was initiated.

Sources: [.github/workflows/release.yml 41-62](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L41-L62)

## Version Management System

Version management centralizes on `pyproject.toml` as the single source of truth. The workflow extracts the current version, bumps it, and propagates the new version to all subsequent jobs.

The version extraction uses `toml-cli` in [.github/workflows/release.yml 102-107](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L102-L107):

The `bump2version` tool modifies the version in place in [.github/workflows/release.yml 115-121](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L115-L121) After bumping, the workflow commits the change with a standardized message format in [.github/workflows/release.yml 145-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L145-L149):

The workflow also calculates the number of commits since the last tag in [.github/workflows/release.yml 138-142](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L138-L142) which is used for Debian package versioning.

Sources: [.github/workflows/release.yml 64-160](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L64-L160)

## Changelog Generation

The changelog generation job uses `gbp dch` (Git-Buildpackage Debian Changelog) to automatically generate `debian/changelog` entries from Git commit history.

The `gbp dch` command in [.github/workflows/release.yml 209-213](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L209-L213) uses several key flags:

*   `--debian-branch`: Specifies the temporary branch to work from
*   `-R`: Marks this as a release (not a snapshot)
*   `-N ${MAJOR}.${MINOR}.${PATCH}`: Sets the version number explicitly
*   `--spawn-editor=never`: Runs non-interactively

The generated changelog follows Debian conventions as shown in [debian/changelog 1-236](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/debian/changelog#L1-L236) Each entry includes:

*   Version number with distribution target (`noble`)
*   Commit messages grouped by author
*   Timestamp in RFC 2822 format
*   Maintainer information from environment variables

Sources: [.github/workflows/release.yml 165-237](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L165-L237)

## Build Orchestration

After version and changelog updates, the workflow triggers the `build-all.yml` workflow to create artifacts for all distribution targets.

The workflow call in [.github/workflows/release.yml 241-253](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L241-L253) passes the temporary branch reference and version components to [.github/workflows/build-all.yml 1-53](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-all.yml#L1-L53) which then dispatches to:

1.   **build-debian.yml** - Creates Debian packages for multiple Ubuntu versions with matrix builds
2.   **build-pypi.yml** - Builds Python wheel packages

Both build workflows upload their artifacts using GitHub Actions artifact storage, making them available for the release creation job.

Sources: [.github/workflows/release.yml 241-253](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L241-L253)[.github/workflows/build-all.yml 1-53](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-all.yml#L1-L53)

## GitHub Release Creation

The `generate-release` job downloads all build artifacts, generates a changelog from Git history, and creates a GitHub Release with attached files.

The changelog builder in [.github/workflows/release.yml 305-321](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L305-L321) uses a hybrid mode that includes both pull requests and direct commits. The template formats entries with author attribution:

The .deb file renaming logic in [.github/workflows/release.yml 328-344](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L328-L344) prepends the distribution name to prevent filename collisions when multiple Ubuntu versions are built.

The release creation in [.github/workflows/release.yml 346-357](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L346-L357) uploads files matching these patterns:

*   `${{ github.workspace }}/${{ env.project_name }}*/*.deb`
*   `${{ github.workspace }}/[!artifacts-]*/*.deb`
*   `${{ github.workspace }}/*/*.whl`

Sources: [.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L288-L357)

## PyPI Publication Strategy

The workflow implements a staged publication strategy that validates packages on TestPyPI before publishing to production PyPI.

Both publication jobs use OIDC trusted publishing, which requires specific permissions in [.github/workflows/release.yml 404-405](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L404-L405) and [.github/workflows/release.yml 439](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L439-L439):

The TestPyPI publication job in [.github/workflows/release.yml 391-419](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L391-L419) publishes to the test environment, allowing verification before production deployment. The production PyPI publication in [.github/workflows/release.yml 424-452](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L424-L452) depends on successful TestPyPI publication.

Both jobs download artifacts using the pattern `${{ env.project_name }}*.whl` to collect all built wheel files.

Sources: [.github/workflows/release.yml 391-452](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L391-L452)

## Merge Back Strategy

The final job merges the temporary branch back to main using rebase to maintain a linear history, then deletes the temporary branch.

The merge back sequence in [.github/workflows/release.yml 380-386](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L380-L386) uses rebase instead of merge to avoid merge commits:

This ensures the version bump and changelog commits appear as direct commits on main, maintaining a clean linear history.

Sources: [.github/workflows/release.yml 362-386](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L362-L386)

## Workflow Environment Variables

The workflow uses a centralized environment variable for the project name in [.github/workflows/release.yml 18-19](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L18-L19):

This variable is referenced throughout the workflow for:

*   Artifact path construction
*   File pattern matching
*   PyPI project URL generation
*   GitHub Release file uploads

Job-specific environments set Git configuration:

*   `EMAIL: releases@tenstorrent.com`
*   `NAME: Tenstorrent Releases`

These credentials are used for automated commits during version bumping and changelog generation.
