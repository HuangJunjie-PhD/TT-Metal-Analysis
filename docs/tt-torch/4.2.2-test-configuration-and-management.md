The test configuration and management system coordinates test execution through [tests/runner/test_config.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_config.py) which defines a `test_config` dictionary mapping test identifiers to configuration dictionaries. The system implements pytest integration via [tests/runner/conftest.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/conftest.py) and supports architecture-specific overrides for different hardware targets.

## 1. Overview and Purpose

The test configuration system consists of:

*   `test_config` dictionary in [tests/runner/test_config.py 8-1500](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_config.py#L8-L1500) mapping test IDs to configuration dicts
*   `ModelStatus` enum in [tests/runner/test_utils.py 15-24](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L15-L24) defining test lifecycle states
*   `ModelTestConfig` class in [tests/runner/test_utils.py 26-68](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L26-L68) for configuration resolution
*   pytest hooks in [tests/runner/conftest.py 14-111](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/conftest.py#L14-L111) for test collection and validation
*   Test discovery utilities in [tests/runner/test_utils.py 70-228](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L70-L228) for filesystem scanning

The configuration system operates during pytest collection to attach metadata to test items, enable filtering by markers, and validate configuration consistency against discovered tests.

### 3.2 ModelStatus Enumeration

The `ModelStatus` enum at [tests/runner/test_utils.py 15-24](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L15-L24) defines four lifecycle states:

| Status Value | pytest Marker | Behavior | Use Case |
| --- | --- | --- | --- |
| `EXPECTED_PASSING` | `pytest.mark.expected_passing` | Test runs normally | Stable, passing tests |
| `KNOWN_FAILURE_XFAIL` | `pytest.mark.xfail(strict=True)` | Expected to fail, fails test if passes | Tracked regressions |
| `NOT_SUPPORTED_SKIP` | `pytest.mark.not_supported_skip` | Skipped via `skip_full_eval_test()` | Unsupported configs |
| `UNSPECIFIED` | `pytest.mark.unspecified` | Test runs normally | Untriaged new tests |

## 8. Troubleshooting and Best Practices

When working with Op-by-Op testing, consider the following best practices:

1.   **Test Organization**: Group related models in the same test file to leverage common testing infrastructure
2.   **Backend Selection**: 
    *   Use PyTorch backend (`OpByOpBackend.TORCH`) for most cases as it has the most mature operation support
    *   Use StableHLO backend (`OpByOpBackend.STABLEHLO`) for testing ONNX model compatibility

3.   **Assertions**: 
    *   `assert_atol`: Control absolute tolerance assertion
    *   `assert_pcc`: Control Pearson correlation coefficient assertion
    *   Disable these assertions when testing complex models initially

4.   **Test Logs**: Review test logs to identify which specific operations failed
5.   **Configuration Adjustments**: 
    *   Disable constant evaluation (`enable_consteval = False`) if there are issues with constant folding
    *   Control parameter evaluation with `consteval_parameters` for weight-heavy models

By following these practices, developers can effectively use Op-by-Op testing to validate and debug operations across the entire model ecosystem.
