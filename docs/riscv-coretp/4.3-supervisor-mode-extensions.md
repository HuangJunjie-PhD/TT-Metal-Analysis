## Purpose and Scope

This page documents test coverage for RISC-V supervisor-mode privileged extensions that enhance timer/counter functionality and performance monitoring. These extensions enable supervisor software to manage timers, control counter access, and handle performance monitoring unit (PMU) overflow events across privilege levels.

The coretp framework provides comprehensive test scenarios for three supervisor-mode extensions:

*   **SSTC (Supervisor-mode Timer Counter)**: Timer compare CSRs and interrupt generation
*   **Zicntr/Zihpm/Sscounterenw**: Counter enable control across privilege modes
*   **Sscofpmf**: Counter overflow and local overflow interrupt management

For detailed test scenario documentation, see:

*   SSTC-specific tests: [SSTC Timer Counter Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.1-sstc-timer-counter-tests)
*   Counter enable tests: [Counter Enable Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.2-counter-enable-tests)
*   PMU overflow tests: [PMU Overflow Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.3-pmu-overflow-tests-(sscofpmf))

For information about test step primitives used in these scenarios, see [CSR Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.2-csr-operations).

## Hierarchical CSR Access Control Model

Supervisor-mode extensions implement a hierarchical access control model where privilege levels control access to timer and counter CSRs through enable bits in `mcounteren`, `scounteren`, and `hcounteren`.

**Diagram: Hierarchical CSR Access Control**

The diagram illustrates how enable bits gate access to timer and counter CSRs. Access from a lower privilege mode requires all higher-level enable bits to be set. For example, accessing `time` from U-mode requires `mcounteren.TM=1`, `hcounteren.TM=1` (if H-extension present), and `scounteren.TM=1`.

## Exception Behavior

Access violations result in different exception types depending on the context:

| Violation Type | Exception Cause | Example Scenario |
| --- | --- | --- |
| `mcounteren` bit clear in S/U mode | `ILLEGAL_INSTRUCTION` | Access `time` with `mcounteren.TM=0` |
| `hcounteren` bit clear in VS/VU mode | `VIRTUAL_INSTRUCTION` | Access `vstimecmp` with `hcounteren.TM=0` |
| `scounteren` bit clear in U mode | `ILLEGAL_INSTRUCTION` | Access `time` with `scounteren.TM=0` |
| `menvcfg.STCE=0` in non-M mode | `ILLEGAL_INSTRUCTION` | Access `stimecmp` with `menvcfg.STCE=0` |
| `henvcfg.STCE=0` in VS/VU mode | `VIRTUAL_INSTRUCTION` | Access `vstimecmp` with `henvcfg.STCE=0` |

## TestStep Composition Pattern

Supervisor-mode test scenarios compose multiple `TestStep` primitives to verify CSR behavior across privilege modes:

**Diagram: TestStep Composition in Supervisor-Mode Scenarios**

Test scenarios follow a pattern: configure CSRs via `CsrWrite`, attempt accesses via `CsrRead`, and verify results using `AssertException` or `AssertEqual`. The `direct_read` and `direct_write` parameters control whether CSR operations execute directly in the test's privilege mode or require mode switching.

## Common Testing Patterns

### Counter Increment Verification

Scenarios verify counter behavior by reading before/after execution of filler instructions:

**Diagram: Counter Increment Verification Pattern**

This pattern appears in scenarios like `SID_SSCOFPMF_03A`, `SID_SSCOFPMF_03B`, `SID_SSCOFPMF_04B`, and `SID_SSCOFPMF_04D`. The number of filler instructions (`NUM_FILL=32`) ensures observable counter changes.

**Sources:**

*   [coretp/plans/sscofpmf/sscofpmf_scenarios.py 104-134](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sscofpmf/sscofpmf_scenarios.py#L104-L134)

### Counter Inhibit Verification

Scenarios verify that `mcountinhibit` stops counters by checking equality after a settling period:

**Diagram: Counter Inhibit Verification Pattern**

The settling period (`NUM_SETTLE=8` instructions) allows out-of-order processors to complete in-flight instructions before the counter stops incrementing.

**Sources:**

*   [coretp/plans/sscofpmf/sscofpmf_scenarios.py 170-203](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sscofpmf/sscofpmf_scenarios.py#L170-L203)

### WARL Bit Verification

Write-Any-Read-Legal (WARL) behavior is verified by writing all ones and checking reserved bits remain clear:

**Diagram: WARL Bit Verification Pattern**

This pattern verifies that reserved or read-only bits cannot be set. `mcountinhibit[1]` must remain 0 (cannot inhibit time counter), and `mhpmevent[57:56]` are reserved WPRI bits.

**Sources:**

## PMU Overflow and LCOFIP Interrupt

The Sscofpmf extension provides local counter overflow interrupt handling:

**Diagram: PMU Overflow and LCOFIP Mechanism**

When a hardware performance counter overflows from 63 bits to 64 bits, the `mhpmeventN.OF` bit is set, which triggers the local counter overflow interrupt pending (`LCOFIP`) bit in `mip` and `sip`. The `scountovf` CSR provides supervisor-mode read-only access to overflow bits for counters enabled via `mcounteren` or `scounteren`.

## Implementation Notes

### SSTC Interrupt Handler Limitations

Several SSTC test scenarios are commented out because they require interrupt handler injection, which is not yet supported by the test framework:

```
# SID_SSTC_07: Unable to be implemented
# SID_SSTC_09-15: Unable to be implemented - Requires interrupt handler to be injected
```

These scenarios would test timer interrupt generation and handling when `time >= stimecmp` or `time >= vstimecmp`.

**Sources:**

*   [coretp/plans/sstc/sstc_scenarios.py 491-601](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L491-L601)

### Counter Implementation Detection

Sscofpmf scenarios must detect whether hardware performance counters are implemented before testing them. This is done by writing a non-zero value and reading back:

`write_mhpmcounter3_test = CsrWrite(csr_name="mhpmcounter3", value=1)read_mhpmcounter3_test = CsrRead(csr_name="mhpmcounter3")is_implemented = AssertNotEqual(src1=read_mhpmcounter3_test, src2=zero)`
If the read-back value is zero, the counter is not implemented and subsequent test steps may be skipped.

**Sources:**

*   [coretp/plans/sscofpmf/sscofpmf_scenarios.py 383-457](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sscofpmf/sscofpmf_scenarios.py#L383-L457)
