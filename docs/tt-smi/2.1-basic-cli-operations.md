## Purpose and Scope

This page documents the fundamental command-line interface operations of TT-SMI, including version display, device listing, local mode, and compact mode flags. These operations provide essential information and configuration options for using TT-SMI without modifying system state.

For device reset operations, see [Reset Commands](https://deepwiki.com/tenstorrent/tt-smi/2.2-reset-commands). For snapshot generation and advanced device listing, see [Snapshot and Listing Commands](https://deepwiki.com/tenstorrent/tt-smi/2.3-snapshot-and-listing-commands). For interactive GUI usage, see [Interactive Text User Interface (TUI)](https://deepwiki.com/tenstorrent/tt-smi/3-interactive-text-user-interface-(tui)).

## Device Listing Operation

The list flag (`-ls` or `--list`) displays all detected Tenstorrent devices on the host system, including their PCI device IDs, board types, device series, and board numbers. This operation is handled in `tt_smi_main()` before launching the GUI.

### Execution Flow

### Backend Method

The `print_all_available_devices()` method in `TTSMIBackend` formats and displays device information in two tables:

1.   **All available boards on host** - Shows every detected device
2.   **Boards that can be reset** - Shows only devices with PCI device IDs (local devices)

Remote devices on multi-chip boards (e.g., n300 Remote chip) are listed without PCI device IDs, as they cannot be directly accessed or reset via PCI.

### Output Format

The method uses the `rich` library to create formatted tables. Each device row includes:

*   **PCI Dev ID**: Index for PCI operations (N/A for remote devices)
*   **Board Type**: e.g., Wormhole, Blackhole, Galaxy variants
*   **Device Series**: e.g., n300 L, n300 R, p-series
*   **Board Number**: Unique hardware identifier

## Local Mode Operation

The local flag (`-l` or `--local`) restricts device detection to local chips only, specifically for Wormhole systems. This mode is useful when operating on multi-chip boards where remote chips should be ignored.

### Effect on Device Detection

When `args.local` is `True`, it is passed to `detect_chips_with_callback()` as both the `local_only` and `ignore_ethernet` parameters:

**Code Location:**[tt_smi/tt_smi.py 874-876](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L874-L876)

### Behavior Implications

| Parameter | Value | Effect |
| --- | --- | --- |
| `local_only` | `True` | Only detect chips with direct PCI connections |
| `ignore_ethernet` | `True` | Skip Ethernet fabric discovery for remote chips |

This is particularly relevant for Wormhole n300 boards, which have two chips (Local and Remote) connected via Ethernet. With local mode enabled:

*   Only the local chip is detected
*   The remote chip is not discovered via the Ethernet fabric
*   Reset operations only affect the local chip
*   Telemetry is only collected from the local chip

### Usage Restrictions

The help text explicitly states "Wormhole only" because:

*   Blackhole systems use different architectures
*   Grayskull is deprecated as of v3.0.35
*   Galaxy systems manage multi-chip connectivity differently

## Compact Mode Operation

The compact flag (`-c` or `--compact`) modifies the GUI layout to hide the sidebar containing host information, maximizing screen space for device data tables. This mode affects only the Textual TUI interface, not CLI operations.

### Implementation Flow

### GUI Layout Structure

The GUI is composed using Textual containers defined in the `compose()` method at [tt_smi/tt_smi.py 114-146](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L114-L146) The layout has:

*   **Header**: TTHeader widget showing app name and version
*   **Left Column** (`#left_col`): Vertical container with TTHostCompatibilityMenu
*   **Tab Container** (`#tab_container`): TabbedContent with three data tables
*   **Footer**: Footer widget with keyboard shortcuts

The `#left_col` container's visibility is controlled by the `display` property.

### Sidebar Toggle Action

In addition to the CLI flag, users can toggle compact mode interactively by pressing `c` or `C` in the GUI. This is implemented by the `action_toggle_compact()` method at [tt_smi/tt_smi.py 509-512](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L509-L512):

## Default GUI Launch

When TT-SMI is invoked without flags (or with only mode-modifying flags like `-l` or `-c`), it launches the interactive Textual TUI application. This is the primary user interface for monitoring device telemetry and system information.

### Launch Conditions

The GUI is launched by `tt_smi_main()` at [tt_smi/tt_smi.py 685-731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L685-L731) when none of the following early-exit flags are present:

*   `args.list` (device listing)
*   `args.glx_list_tray_to_device` (Galaxy tray mapping)
*   `args.snapshot` or `args.filename` (snapshot operations)

### TTSMI Application Initialization

The `TTSMI` class is a Textual `App` subclass defined at [tt_smi/tt_smi.py 67-575](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L67-L575) It is initialized with several parameters:

| Parameter | Source | Purpose |
| --- | --- | --- |
| `backend` | `TTSMIBackend(devices)` | Provides device data and collection methods |
| `snapshot` | `args.snapshot` | Enables snapshot mode (normally False for GUI) |
| `result_filename` | `args.filename` | Snapshot filename (normally None for GUI) |
| `show_sidebar` | `not args.compact` | Controls sidebar visibility |

### Application Lifecycle

Once initialized, calling `tt_smi_app.run()` at [tt_smi/tt_smi.py 731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L731-L731) starts the Textual event loop, which:

1.   **Mounts widgets**: Calls `on_mount()` to initialize tables and set sidebar visibility
2.   **Displays UI**: Renders the three-tab interface with header and footer
3.   **Handles events**: Processes keyboard input and tab switching
4.   **Updates telemetry**: Spawns background thread when telemetry tab is activated
5.   **Responds to actions**: Executes action methods for user commands

The application continues running until the user quits via `q` key or Ctrl+C.

### CSS Styling

The application loads two CSS files for styling:

*   **common_style.css**: Shared styles from `tt_tools_common` library
*   **tt_smi_style.css**: TT-SMI-specific styles

These are specified in the `CSS_PATH` class attribute at [tt_smi/tt_smi.py 89](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L89-L89)

### Argument Processing and Execution Flow

### Argument Flag Priority

The order of execution in `main()` and `tt_smi_main()` establishes implicit priority for CLI flags:

1.   **Version flag** (`-v`): Handled by argparse, exits before any TT-SMI code runs
2.   **Reset flags** (`-r`, `-glx_reset*`): Handled in `main()` before backend creation, covered in [Reset Commands](https://deepwiki.com/tenstorrent/tt-smi/2.2-reset-commands)
3.   **List flag** (`-ls`): First check in `tt_smi_main()`, exits before GUI launch
4.   **Snapshot flags** (`-s`, `-f`): Checked after list, covered in [Snapshot and Listing Commands](https://deepwiki.com/tenstorrent/tt-smi/2.3-snapshot-and-listing-commands)
5.   **Mode flags** (`-l`, `-c`): Affect device detection and GUI rendering, not exit conditions

This priority ensures that quick information queries exit early without unnecessary initialization.

## TTY Detection and Output Formatting

TT-SMI detects whether it is running in a TTY (terminal) environment to adjust its output formatting. This affects progress indicators and color output throughout the application.

### Detection Implementation

At [tt_smi/tt_smi.py 787](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L787-L787) the application checks if stdout is a TTY:

The `is_tty` boolean is passed to multiple components:

*   **Device detection**: Controls progress bar display in `detect_chips_with_callback(print_status=is_tty)`
*   **Backend creation**: Controls output formatting in `TTSMIBackend(devices, pretty_output=is_tty)`
*   **Reset operations**: Controls status messages in reset functions

### Override Flag

The `--snapshot_no_tty` flag at [tt_smi/tt_smi.py 638-642](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L638-L642) forces non-TTY behavior even when running in a terminal. This is useful for:

*   Piping output to files or other programs
*   Scripting environments where ANSI codes should be avoided
*   CI/CD systems that don't handle TTY output well
