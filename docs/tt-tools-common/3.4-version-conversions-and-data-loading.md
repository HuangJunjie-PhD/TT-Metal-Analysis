# Version Conversions and Data Loading

Relevant source files
*   [CHANGELOG.md](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1)

## Purpose and Scope

This page documents the utility functions for converting firmware version formats, identifying board types from hardware identifiers, and loading chip-specific data files. These utilities support runtime version parsing, hardware identification, and configuration loading throughout the tt-tools-common library.

For driver version checking and semantic version parsing with extraversion fields, see [Driver Version Management](https://deepwiki.com/tenstorrent/tt-tools-common/3.1-driver-version-management). For chip detection and hardware communication, see [Chip Detection and Communication](https://deepwiki.com/tenstorrent/tt-tools-common/3.3-chip-detection-and-communication).

## Version Conversion Architecture

The version conversion system handles multiple firmware types, each with distinct hexadecimal encoding schemes. All conversion functions follow a consistent pattern: they accept a hexadecimal integer and return a semantic version string, handling invalid values (0x0 or 0xFFFFFFFF) appropriately.

### Version Conversion Flow

**Sources:**[tt_tools_common/utils_common/tools_utils.py 136-190](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L136-L190)

### Standard Semantic Version Conversion

The `hex_to_semver()` function converts standard firmware versions encoded in 32-bit hexadecimal format. The encoding uses the lower 24 bits for version components.

**Bit Layout:**

| Bits | Field | Purpose |
| --- | --- | --- |
| 31-24 | Reserved | Unused (0x00) |
| 23-16 | Major | Major version number |
| 15-8 | Minor | Minor version number |
| 7-0 | Patch | Patch version number |

**Example:**

*   Input: `0x0A0F0100` (hex)
*   Extraction: Major=10, Minor=15, Patch=1
*   Output: `"10.15.1"`

The function raises `ValueError` for invalid inputs (0x0 or 0xFFFFFFFF) rather than returning "N/A", as these represent uninitialized or corrupted firmware metadata.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 136-145](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L136-L145)

### Ethernet Firmware Version Conversion

The `hex_to_semver_eth()` function handles ethernet firmware versions with a non-standard bit layout that allocates more bits to the patch version field.

**Bit Layout:**

| Bits | Field | Purpose |
| --- | --- | --- |
| 23-16 | Major | Major version number |
| 15-12 | Minor | Minor version number (4 bits) |
| 11-0 | Patch | Patch version number (12 bits) |

**Example:**

*   Input: `0x061000` (hex)
*   Extraction: Major=6, Minor=1, Patch=0
*   Output: `"6.1.0"`

Invalid values return `"N/A"` instead of raising exceptions, allowing graceful handling of ethernet firmware absence.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 148-157](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L148-L157)

### M3 Firmware Version Conversion

The `hex_to_semver_m3_fw()` function converts M3 firmware versions which include a fourth version component, producing a four-part semantic version string.

**Bit Layout:**

| Bits | Field | Purpose |
| --- | --- | --- |
| 31-24 | Major | Major version number |
| 23-16 | Minor | Minor version number |
| 15-8 | Patch | Patch version number |
| 7-0 | Ver | Additional version component |

**Example:**

*   Input: `0x0A0F0100` (hex)
*   Extraction: Major=10, Minor=15, Patch=1, Ver=0
*   Output: `"10.15.1.0"`

The four-component format distinguishes M3 firmware from standard ARC firmware versions.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 160-170](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L160-L170)

### Zephyr Firmware Version Conversion

The `hex_to_semver_zephyr_fw()` function converts Zephyr RTOS firmware versions, which support release candidate suffixes for pre-release builds.

**Bit Layout and RC Handling:**

| Bits | Field | Purpose |
| --- | --- | --- |
| 31-24 | Major | Major version number |
| 23-16 | Minor | Minor version number |
| 15-8 | Patch | Patch version number |
| 7-0 | RC | Release candidate number (0=final) |

**RC Encoding:**

*   `rc_num = 0`: Final release (no suffix)
*   `rc_num > 0`: Release candidate with `-rcN` suffix

**Examples:**

*   Input: `0x0A0F0100` → Output: `"10.15.1"` (final release)
*   Input: `0x0A0F0101` → Output: `"10.15.1-rc1"` (release candidate 1)
*   Input: `0x0A0F0105` → Output: `"10.15.1-rc5"` (release candidate 5)

This format enables pre-release testing while maintaining semantic version compatibility.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 172-189](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L172-L189)

## Date and Time Conversion

The `hex_to_date()` function converts firmware build timestamps from a compact hexadecimal format to human-readable date strings.

**Encoding Format (0xYMDDHHMM):**

| Bits | Field | Range | Purpose |
| --- | --- | --- | --- |
| 31-28 | Year offset | 0-15 | Years since 2020 (0x0=2020, 0xF=2035) |
| 27-24 | Month | 1-12 | Calendar month |
| 23-16 | Day | 1-31 | Day of month |
| 15-8 | Hour | 0-23 | Hour in 24h format |
| 7-0 | Minute | 0-59 | Minute |

**Example Conversion:**

*   Input: `0x51A0830` (hex)
*   Decoding: Year=2025 (5+2020), Month=1, Day=10, Hour=8, Minute=48
*   Output: `"2025-01-10 08:48"`

The `include_time` parameter controls whether the time portion is appended. Invalid dates (0x0 or 0xFFFFFFFF) return `"N/A"`.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 93-109](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L93-L109)

## Board Type Identification System

The board type identification system decodes hardware serial numbers to determine board models. Board IDs are 56-bit values encoded as hexadecimal strings.

### Board ID Structure

**Sources:**[tt_tools_common/utils_common/tools_utils.py 46-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L46-L90)

### UPI to Board Type Mapping

The `get_board_type()` function extracts the Unique Part Identifier (UPI) and revision fields from the board ID, then maps them to specific board types.

**Extraction Logic:**

`serial_num = int(f"0x{board_id}", base=16)upi = (serial_num >> 36) & 0xFFFFF  # Bits 47-32rev = (serial_num >> 32) & 0xF       # Bits 35-32`
**UPI Mapping Table:**

| UPI (hex) | Revision | Board Type | Notes |
| --- | --- | --- | --- |
| 0x1 | 0x2 | E300_R2 | Early E300 revision 2 |
| 0x1 | 0x3, 0x4 | E300_R3 | E300 revision 3/4 |
| 0x3 | Any | e150 | Formerly E300_105 |
| 0x7 | Any | e75 | Entry-level board |
| 0x8 | Any | NEBULA_CB | Nebula carrier board |
| 0xA | Any | e300 | Formerly E300_X2 |
| 0xB | Any | GALAXY | Multi-chip galaxy system |
| 0x14 | Any | n300 | Formerly NEBULA_X2 |
| 0x18 | Any | n150 | Formerly NEBULA_X1 |
| Other | Any | N/A | Unknown/invalid UPI |

The function returns `"N/A"` for unrecognized UPI values or invalid revision combinations (e.g., UPI 0x1 with revision outside 0x2-0x4).

**Sources:**[tt_tools_common/utils_common/tools_utils.py 46-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L46-L90)

## Chip Data File Loading

The chip data loading system provides access to chip-specific configuration files, firmware definitions, and other static data assets.

### Data File Organization

**Sources:**[tt_tools_common/utils_common/tools_utils.py 17-43](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L17-L43)

### Loading Firmware Definitions

The `init_fw_defines()` function loads chip-specific firmware message definitions from YAML files. These definitions include ARC message types, register addresses, and protocol specifications.

**Function Signature:**

`def init_fw_defines(chip_name: str = "wormhole", tool_name: str = "tt_smi"):`
**Parameters:**

*   `chip_name`: Chip architecture (`"wormhole"` or `"grayskull"`)
*   `tool_name`: Package name for path resolution (default: `"tt_smi"`)

**Usage Pattern:**

1.   Resolves package path using `importlib.resources`
2.   Constructs path: `data/{chip_name}/fw_defines.yaml`
3.   Loads YAML with `safe_load()`
4.   Returns dictionary with firmware definitions

**Example:**

`fw_defines = init_fw_defines(chip_name="wormhole", tool_name="tt_smi")# Returns: {'MSG_TYPE_RESET': 0x10, 'MSG_TYPE_...': ...}`
The returned dictionary is typically used for ARC message protocol encoding/decoding.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 17-24](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L17-L24)

### Generic Data File Access

The `get_chip_data()` function provides generic access to any file in the chip data directories, supporting both public and internal data locations.

**Function Signature:**

`def get_chip_data(chip_name, file, internal: bool, tool_name="tt_smi"):`
**Parameters:**

*   `chip_name`: Chip architecture (`"wormhole"` or `"grayskull"`)
*   `file`: Filename within the chip directory
*   `internal`: If `True`, loads from `.ignored/` directory
*   `tool_name`: Package name for path resolution

**Path Resolution Logic:**

| Condition | Path Template |
| --- | --- |
| `internal=True` | `.ignored/{chip_name}/{file}` |
| `internal=False` | `data/{chip_name}/{file}` |
| Tool name contains `.data` | `{chip_name}/{file}` (special case) |

**Example Usage:**

`# Load public configurationconfig = get_chip_data("wormhole", "reset_config.yaml", False) # Load internal/proprietary datainternal_data = get_chip_data("wormhole", "proprietary.yaml", True)`
The function returns a file handle opened in read mode, raising an `Exception` if the chip name is not supported (only Wormhole and Grayskull are valid).

**Sources:**[tt_tools_common/utils_common/tools_utils.py 31-43](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L31-L43)

## Utility Functions Reference

### Reverse Conversion Functions

Two utility functions support conversion from semantic version strings back to hexadecimal format:

**`semver_to_hex(semver: str)`**

*   Converts semantic version string to hexadecimal
*   Input: `"10.15.1"` → Output: `"0a0f0100"`
*   Used for firmware upgrade validation and version comparison

**`date_to_hex(date: int)`**

*   Converts date string to hexadecimal timestamp
*   Input: `"202501100848"` (YYYYMMDDHHMM) → Output: `"051a0830"`
*   Used for firmware build date encoding

**Sources:**[tt_tools_common/utils_common/tools_utils.py 118-133](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L118-L133)

### Integration with Other Systems

Version conversion functions are primarily consumed by:

*   **Chip telemetry parsing**: Converting raw hardware registers to human-readable versions
*   **Firmware upgrade validation**: Comparing current vs. target firmware versions
*   **Reset logging**: Recording firmware versions in `HostResetLog` structures (see [Reset Configuration and Logging](https://deepwiki.com/tenstorrent/tt-tools-common/2.6-reset-configuration-and-logging))
*   **Compatibility checking**: Validating firmware version requirements

Board type identification is used by:

*   **Hardware detection**: Identifying board models during `detect_chips_with_callback()` (see [Chip Detection and Communication](https://deepwiki.com/tenstorrent/tt-tools-common/3.3-chip-detection-and-communication))
*   **Configuration selection**: Loading board-specific reset configurations
*   **Compatibility validation**: Ensuring operations are supported on detected hardware
