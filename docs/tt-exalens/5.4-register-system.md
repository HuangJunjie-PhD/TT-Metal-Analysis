## Purpose and Scope

The Register System provides an abstraction layer for accessing hardware registers on Tenstorrent devices. It defines register descriptions that specify address offsets, bit masks, and data types, and provides a unified interface for reading and writing registers through various access methods (NOC, BAR0, or private memory). This system handles both named registers with predefined descriptions and dynamic register access through runtime-specified indices and offsets.

The primary implementation lives in [ttexalens/register_store.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py) Architecture-specific register maps are defined in `ttexalens/hardware/<arch>/`. The `TensixRegisterDescription` grouping structure lives in [ttexalens/hardware/tensix_registers_description.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/tensix_registers_description.py)

For memory access operations beyond registers, see page 3.3. For hardware block abstraction, see page 5.2. For platform-specific register implementations, see page 5.5. For the Python library `read_register`/`write_register` API, see page 3.4.

## Register Description Class Hierarchy

The register system uses a hierarchy of dataclasses to describe different types of registers. Each description type captures the addressing and access characteristics specific to that register category.

**Register description class hierarchy:**

Sources: [ttexalens/register_store.py 63-161](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L63-L161)

### Base RegisterDescription

The `RegisterDescription` dataclass defines the common attributes for all register types:

| Attribute | Type | Default | Description |
| --- | --- | --- | --- |
| `base_address` | `DeviceAddress | None` | `None` | Base address for the register bank |
| `offset` | `int` | `0` | Byte offset from base address |
| `mask` | `int` | `0xFFFFFFFF` | Bit mask for multi-field registers |
| `shift` | `int` | `0` | Bit shift for extracting field values |
| `data_type` | `REGISTER_DATA_TYPE` | `INT_VALUE` | How to interpret the register value |

The class provides computed properties for different address spaces:

*   `noc_address`: NOC-accessible address (`base + offset`)
*   `private_address`: RISC-local private memory address
*   `bar0_address`: PCIe BAR0 address for direct host access
*   `noc_id`: Which NOC to use for access

Helper methods:

*   `clone(base_address)`: Deep copy with a new base address applied
*   `change_offset(offset)`: Deep copy with an additional offset applied

Sources: [ttexalens/register_store.py 63-103](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L63-L103)

### ConfigurationRegisterDescription

Configuration registers use indirect addressing through control/data register pairs. The `index` parameter specifies which configuration register to access:

```
ConfigurationRegisterDescription(index=42, mask=0xFF, shift=8)
```

The offset is automatically computed in `__post_init__` as `base_offset + index * 4` (4 bytes per register). The maximum valid index is determined at runtime from the `config_regs` memory block size in the RISC's memory map.

Sources: [ttexalens/register_store.py 115-121](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L115-L121)

### TensixGeneralPurposeRegisterDescription

Tensix cores have 64 general-purpose registers per thread. The offset is computed as:

```
offset = base_offset + (index + thread_id * 64) * 4
```

This allows access to any GPR on any Tensix thread through a single description type.

Sources: [ttexalens/register_store.py 123-131](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L123-L131)

## Register Data Types

The `REGISTER_DATA_TYPE` enum defines how register values should be interpreted and displayed:

| Enum Value | Format | Example Output |
| --- | --- | --- |
| `INT_VALUE` | Decimal integer | `42` |
| `ADDRESS` | Hexadecimal | `0x1a000` |
| `MASK` | Hexadecimal | `0xff00` |
| `FLAGS` | Comma-separated booleans | `True,False,True,True` |
| `TENSIX_DATA_FORMAT` | `TensixDataFormat` enum name | `TensixDataFormat.Float16` |

Sources: [ttexalens/register_store.py 20-27](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L20-L27)

### Value Formatting and Parsing

`format_register_value(value, data_type, number_of_bits)` converts a raw integer to a human-readable string based on the `REGISTER_DATA_TYPE`.

`parse_register_value(value)` converts a string to an integer. Accepted formats: hexadecimal (`0x...`), binary (`0b...`), decimal, comma-separated boolean list (`True,False,...`), or a `TensixDataFormat` member name.

Sources: [ttexalens/register_store.py 29-61](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L29-L61)

## RegisterStoreInitialization and RegisterStore

### RegisterStoreInitialization

`RegisterStoreInitialization` is a dataclass holding the pre-built register map and the address resolver callback:

| Field | Type | Description |
| --- | --- | --- |
| `registers_map` | `dict[str, RegisterDescription]` | Name-to-description mapping with resolved addresses |
| `get_register_base_address` | `Callable[[RegisterDescription], DeviceAddress]` | Resolves base address for a description type |

The `RegisterStore.create_initialization()` static method builds a `RegisterStoreInitialization` from one or more raw offset maps by calling `get_register_base_address` for each entry:

```
initialization = RegisterStore.create_initialization(
    register_offset_maps,       # dict or list of dicts
    get_register_base_address   # block-type callback
)
```

This object is created once per block type and reused for every instance of that block, avoiding redundant address resolution.

Sources: [ttexalens/register_store.py 163-167](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L163-L167)[ttexalens/register_store.py 363-383](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L363-L383)

### Register Access Methods

| Method | Parameters | Returns | Description |
| --- | --- | --- | --- |
| `read_register()` | `register: str | RegisterDescription, safe_mode=None` | `int` | Read value (masked and shifted) |

### Register Naming and Parsing

Register stores support two access patterns:

1.   **Named registers** — pre-defined in the register map:

```
store.read_register("RISCV_DEBUG_REG_CFGREG_RD_CNTL")
```
2.   **Dynamic register descriptions** — runtime-specified syntax parsed by `parse_register_description()`:

```
# Configuration register: cfg(index [, mask [, shift]])
store.read_register("cfg(42, 0xFF, 8)")

# Debug register: dbg(address [, mask [, shift]])
store.read_register("dbg(0x1000, 0xFFFF)")
```

The parser validates the `cfg` index against the runtime-determined maximum (`_max_config_register_index`) derived from the `config_regs` memory block size.

Sources: [ttexalens/register_store.py 242-284](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L242-L284)

## Register Access Mechanisms

`read_register` and `write_register` select the appropriate access path based on which address fields are populated in the `RegisterDescription`.

**Access path selection in read_register / write_register:**

Sources: [ttexalens/register_store.py 286-362](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L286-L362)

### BAR0 Access

When `bar0_address` is not None, the register is directly accessible via the PCIe BAR0 window. Available only on local MMIO-capable devices. For masked writes, a read-modify-write is performed.

Sources: [ttexalens/register_store.py 304-347](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L304-L347)

### NOC Access

When `noc_address` is not None, the register is accessible through the NOC fabric using `OnChipCoordinate.noc_read32()` / `noc_write32()`. Works for both local and remote devices. The `noc_id` from the register description specifies which NOC (0 or 1) to use. Device-level NOC failover retries on the alternate NOC if timeouts occur.

Sources: [ttexalens/register_store.py 306-352](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L306-L352)

### Indirect Configuration Access

`ConfigurationRegisterDescription` registers use a control/data register pair. The `RegisterStore` caches the addresses of these two registers as `_control_register_address` and `_data_register_address`, resolved from the named registers `RISCV_DEBUG_REG_CFGREG_RD_CNTL` and `RISCV_DEBUG_REG_CFGREG_RDDATA`.

Access sequence:

1.   Write `register.index` to `_control_register_address`
2.   Read from / write to `_data_register_address`

Sources: [ttexalens/register_store.py 186-210](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L186-L210)[ttexalens/register_store.py 308-310](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L308-L310)

### Private Memory Access

When no other addressing mode is available, registers are accessed via `RiscDebug.read_memory()` / `write_memory()` after entering the `ensure_private_memory_access()` context manager, which temporarily halts the RISC core if necessary.

Sources: [ttexalens/register_store.py 313-361](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L313-L361)

## RegisterStore Integration with Device Architecture

### Address Resolution

Register descriptions initially hold only offsets. During `create_initialization()`, the `get_register_base_address` callback resolves each raw `RegisterDescription` into a full `DeviceAddress` (with `noc_address`, `private_address`, and/or `bar0_address` fields populated). This is done once at initialization time, so runtime accesses are fast.

For example, a `DebugRegisterDescription(offset=0x100)` is cloned with a resolved base:

```
noc_address    = base.noc_address + 0x100
private_address = base.private_address + 0x100
```

Sources: [ttexalens/register_store.py 71-98](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L71-L98)[ttexalens/register_store.py 363-383](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L363-L383)

### Configuration Register Bounds Checking

The `_max_config_register_index` cached property determines the valid range at runtime:

1.   Call `block.get_default_risc_debug()` for the RISC at this location
2.   Find the `config_regs` named block in `risc_info.memory_map`
3.   Compute `max_index = (block_size_bytes // 4) - 1`

Both `read_register` and `write_register` validate the index against this limit when given a `ConfigurationRegisterDescription`.

Sources: [ttexalens/register_store.py 198-210](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L198-L210)[ttexalens/register_store.py 292-300](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L292-L300)

## TensixRegisterDescription and TensixDebugBusDescription

The `TensixRegisterDescription` dataclass in [ttexalens/hardware/tensix_registers_description.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/tensix_registers_description.py) groups named register keys by functional area. Each field is a `list[dict[str, str]]` — a list of register sets, where each dict maps a human-readable field name to a register name in the `RegisterStore`.

| Field | Description |
| --- | --- |
| `alu_config` | ALU rounding and format configuration |
| `unpack_tile_descriptor` | Input tile descriptor fields |
| `unpack_config` | Unpacker section configuration |
| `pack_config` | Packer configuration |
| `relu_config` | ReLU post-processing config |
| `pack_dest_rd_ctrl` | Destination read control |
| `pack_edge_offset` | Edge offset settings |
| `pack_counters` | Pack counter fields |
| `pack_strides` | Stride configuration |
| `general_purpose_registers` | Per-thread GPR mappings (`list[dict[str,str]]`) |

`TensixDebugBusDescription` groups debug bus signal group names by function:

| Field | Description |
| --- | --- |
| `register_window_counter_groups` | RWC signal group names (`list[str]`) |
| `address_counter_groups` | ADC signal group names (`list[str]`) |

Both are used by the `dump-tensix-state` CLI command to look up which register names or signal group names to read for each sub-section.

Sources: [ttexalens/hardware/tensix_registers_description.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/tensix_registers_description.py)[ttexalens/cli_commands/dump-tensix-state.py 152-277](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/dump-tensix-state.py#L152-L277)

## Usage Examples

### Reading Named Registers

```
store = device.get_block(location).get_register_store()
value = store.read_register("RISCV_DEBUG_REG_SOFT_RESET_0")
```

### Writing with Bit Masks (Read-Modify-Write)

```
# mask=0xFF00, shift=8 — sets bits [15:8] without touching other bits
store.write_register("SOME_CONTROL_REG", 0x42)
```

### Dynamic Configuration Register Access

```
# cfg(index [, mask [, shift]])
cfg_value  = store.read_register("cfg(42)")
field_value = store.read_register("cfg(42, 0xFF, 8)")

# dbg(address [, mask [, shift]])
dbg_value = store.read_register("dbg(0x1000, 0xFFFF)")
```

### Accessing All Register Names

```
for name in store.get_register_names():
    print(name)
```

## Error Handling

The register system performs validation and provides detailed error messages:

| Error Condition | Exception Type | Example Message |
| --- | --- | --- |
| Unknown register name | `ValueError` | `Unknown register name 'FOO' on (1,2) for device 0` |
| Invalid register index | `ValueError` | `Register index must be less than or equal to 63, but got 100` |
| Invalid mask value | `ValueError` | `Invalid mask value. Mask must be between 0 and 0xFFFFFFFF` |
| Invalid shift value | `ValueError` | `Invalid shift value. Shift must be between 0 and 31` |
| Value out of range | `ValueError` | `Value must be greater than 0 and inside the mask 0xff00` |
| Invalid parse format | `ValueError` | `Invalid input string format: foo(1,2,3,4,5)` |

**Sources:**[ttexalens/register_store.py 219-341](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/register_store.py#L219-L341)

## Platform-Specific Register Maps

While `RegisterStore` provides a unified interface, the actual register maps are platform-specific and defined in architecture subdirectories:

*   `ttexalens/hardware/wormhole/` - Wormhole register definitions
*   `ttexalens/hardware/blackhole/` - Blackhole register definitions
*   `ttexalens/hardware/quasar/` - Quasar register definitions

Each platform provides its own `RegisterStoreInitialization` objects tailored to its hardware blocks. For details on platform-specific implementations, see [Platform-Specific Implementations](https://deepwiki.com/tenstorrent/tt-exalens/5.5-platform-specific-implementations).
