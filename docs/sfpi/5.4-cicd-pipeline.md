# CI/CD Pipeline

Relevant source files
*   [.github/workflows/build-sfpi.yaml](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml)

This document describes the automated build, test, and release infrastructure for the SFPI toolchain. The CI/CD system uses GitHub Actions to build the GCC/Binutils toolchain across multiple architectures and distributions, execute test suites, package artifacts in multiple formats, and publish them for consumption by downstream projects.

For information about the manual build process and available build options, see [Configuration and Compilation](https://deepwiki.com/tenstorrent/sfpi/5.2-configuration-and-compilation). For details on packaging and release artifact formats, see [Packaging and Release](https://deepwiki.com/tenstorrent/sfpi/5.3-packaging-and-release).

## Workflow Architecture

The CI/CD pipeline is implemented as a reusable GitHub Actions workflow defined in [.github/workflows/build-sfpi.yaml 1-113](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L1-L113) This workflow can be triggered both manually via `workflow_dispatch` and programmatically via `workflow_call` from other workflows.

### Workflow Triggers and Inputs

`workflow_dispatch:  inputs:    tag: Version tag to checkout (optional)    arch: x86_64 | aarch64    box: none | Alma workflow_call:  inputs:    tag, arch, box, key`
The workflow accepts four key inputs that control the build environment:

| Input | Required | Purpose | Values |
| --- | --- | --- | --- |
| `tag` | No | Specific git tag/commit to build | Any valid git ref |
| `arch` | Yes | Target architecture | `x86_64`, `aarch64` |
| `box` | Yes | Container environment | `none`, `Alma` |
| `key` | No | Unique identifier for matrix builds | String identifier |

**Sources:**[.github/workflows/build-sfpi.yaml 3-17](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L3-L17)

### Build Matrix Configuration

The workflow supports a 2×2 matrix of build configurations combining architecture and container environment:

**Architecture Selection:** The `arch` input determines which GitHub-hosted runner is selected via the expression `ubuntu-22.04${{ inputs.arch == 'aarch64' && '-arm' || '' }}`, which evaluates to either `ubuntu-22.04` (x86_64) or `ubuntu-22.04-arm` (aarch64).

**Container Selection:** When `box` is set to `Alma`, the workflow runs inside a `quay.io/pypa/manylinux_2_34_{arch}` container, which provides a Fedora-based environment for building RPM packages. When `box` is `none`, builds execute directly on the Ubuntu runner.

**Sources:**[.github/workflows/build-sfpi.yaml 26-41](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L26-L41)[.github/workflows/build-sfpi.yaml 49-51](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L49-L51)

### Dynamic Environment Variables

The workflow sets the `SUDO` environment variable conditionally based on container context:

`env:  SUDO: ${{ inputs.box == 'none' && 'sudo' || '' }}`
This allows package installation commands to work correctly both in bare-metal runners (which require `sudo`) and containers (which run as root).

**Sources:**[.github/workflows/build-sfpi.yaml 44](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L44-L44)

## Build Process Sequence

The workflow executes a series of steps that checkout source code, install dependencies, build the toolchain, run tests, package artifacts, and upload results.

**Sources:**[.github/workflows/build-sfpi.yaml 53-112](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L53-L112)

### Step 1: Repository Checkout

The workflow uses `actions/checkout@v4` with specific configuration to ensure complete access to version history and submodules:

*   `fetch-depth: 64` - Retrieves the last 64 commits (sufficient for version detection via `git describe`)
*   `fetch-tags: true` - Downloads all repository tags
*   `submodules: recursive` - Initializes gcc, binutils, newlib, qemu, and riscv-dejagnu submodules

The subsequent `git config --global --add safe.directory '*'` step is required because the workflow may run in containers where the working directory ownership differs from the git user.

**Sources:**[.github/workflows/build-sfpi.yaml 53-61](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L53-L61)

### Step 2: Dependency Installation

Package installation varies by distribution type, detected via the presence of `apt-get` or `dnf`:

**Common packages:**`autoconf automake bison expect flex gawk python3 texinfo patchutils ruby`

**Debian-specific:**`rpm libexpat1-dev libgmp-dev libmpc-dev libmpfr-dev gcc g++`

**Fedora-specific:**`rpm-build expat-devel gmp-devel libmpc-devel mpfr-devel gcc gcc-c++`

The FPM (Effing Package Management) tool is installed via RubyGems to enable `.deb` and `.rpm` package creation.

**Sources:**[.github/workflows/build-sfpi.yaml 62-75](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L62-L75)

### Step 3: Toolchain Build

The build step invokes [scripts/build.sh 77](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L77-L77) with two critical flags:

*   `--gdb` - Enables GDB debugger compilation (sets `enable_gdb=--enable-gdb`)
*   `--tt-built` - Indicates this is an official Tenstorrent build (affects bug URLs and package metadata)

This triggers the complete toolchain build sequence documented in [Configuration and Compilation](https://deepwiki.com/tenstorrent/sfpi/5.2-configuration-and-compilation):

1.   Configure stage: `configure --prefix=build/sfpi/compiler --with-mfc=tt --enable-gcc-checking=release`
2.   Binutils build
3.   GCC stage 1 build
4.   Newlib C library build
5.   GCC stage 2 build

The `--tt-built` flag causes the configure step to set:

*   `--with-bugurl='https://github.com/tenstorrent/sfpi'`
*   `--with-pkgversion="tenstorrent/sfpi:$tt_version"`

**Sources:**[.github/workflows/build-sfpi.yaml 76-77](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L76-L77)[scripts/build.sh 129-145](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L129-L145)

### Step 4: Test Harness Build

The `--dejagnu` flag triggers compilation of the DejaGnu test framework:

`make -C build build-dejagnu -j$NCPUS`
This builds the testing infrastructure required by the upstream GCC/Binutils test suites.

**Sources:**[.github/workflows/build-sfpi.yaml 78-79](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L78-L79)[scripts/build.sh 151-153](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L151-L153)

## Testing and Validation

### Test Execution

The `--test-tt` flag runs the Tenstorrent-specific SFPU test suite:

`SFPI=$(pwd) make -C build -j$NCPUS NEWLIB_TARGET_BOARDS="riscv-sim/" check-gcc-tt`
This executes the test suite defined in the repository's `tests/*/sfpi/` directories, validating:

1.   **Instruction-level coverage** - [tests/wormhole/sfpi/coverage.cc](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/coverage.cc) and [tests/grayskull/sfpi/coverage.cc](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/coverage.cc)
2.   **Kernel function implementations** - [tests/wormhole/sfpi/ckernel.cc](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/ckernel.cc)
3.   **Compiler optimization passes** - [tests/wormhole/sfpi/regmov.cc](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc) and [tests/grayskull/sfpi/regmov.cc](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/regmov.cc)

Test results are collected from the build directory and placed in `build/tests/` with naming convention `{gcc,g++}-{arch}_{dist}.{sum,log}`.

**Sources:**[.github/workflows/build-sfpi.yaml 80-81](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L80-L81)[scripts/build.sh 200-213](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L200-L213)

### Test Result Processing

The build script extracts test statistics from DejaGnu `.sum` files:

`fails=$((fails + $(grep -c '^FAIL:' $tests/$dst.sum || true)))unresolveds=$((unresolveds + $(grep -c '^UNRESOLVED:' $tests/$dst.sum || true)))errors=$((errors + $(grep -c '^ERROR:' $tests/$dst.sum || true)))`
The build fails if any of these counters are non-zero, ensuring quality gates are enforced.

**Sources:**[scripts/build.sh 209-211](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L209-L211)[scripts/build.sh 217-234](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L217-L234)

## Artifact Generation and Publishing

### Release Package Creation

The release step invokes [scripts/release.sh 83](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L83-L83) with two flags:

*   `--ci` - Skips source hash verification (assumes clean checkout from CI)
*   `--tt-built` - Enables official Tenstorrent package metadata

The release script performs these operations:

**Sources:**[scripts/release.sh 48-159](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L48-L159)

### Package Types and Naming

The release script generates three artifact types for each architecture/distribution combination:

| Format | Extension | Tool | Purpose |
| --- | --- | --- | --- |
| Tarball | `.txz` | `tar cJf` | Universal distribution, used by CMake FetchContent |
| Debian | `.deb` | `fpm -t deb` | Installation on Debian/Ubuntu systems |
| RPM | `.rpm` | `fpm -t rpm` | Installation on Fedora/RHEL/Alma systems |

Filename format: `sfpi_{VERSION}_{ARCH}_{DIST}.{txz,deb,rpm}`

Example: `sfpi_0.9.0_x86_64_debian.deb`

**Sources:**[scripts/sfpi-info.sh 108-110](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L108-L110)[scripts/release.sh 59-150](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L59-L150)

### Dependency Resolution

The release script analyzes shared library dependencies of built executables using `ldd` and maps them to distribution-specific package names:

`# Example mapping for common librarieslibc.so*      -> libc6:glibclibgmp.so*    -> libgmp10:gmplibmpfr.so*   -> libmpfr6:mpfrlibstdc++.so* -> libstdc++6:libstdc++`
Format: `debian_name:fedora_name`

Package versions are determined by querying the build system:

*   Debian: `dpkg-query -f '${Version}' -W package_name`
*   Fedora: `rpm -q --qf '%{VERSION}' package_name`

These become `--depends` arguments to FPM, ensuring package managers can enforce correct runtime dependencies.

**Sources:**[scripts/release.sh 63-137](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L63-L137)

### Artifact Upload

The workflow uploads three categories of artifacts using `actions/upload-artifact@v4`:

`# Build artifacts (packages and checksums)name: build-{arch}-{box}path: build/release/*compression-level: 0  # Already compressed # Test results (.sum and .log files)name: tests-{arch}-{box}path: build/tests/* # Configuration metadataname: config-{arch}-{box}path: outputs/config-*`
The `compression-level: 0` setting for build artifacts prevents double-compression of `.txz` and `.rpm` files.

**Sources:**[.github/workflows/build-sfpi.yaml 84-112](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L84-L112)

## Version Management

### Version Detection Logic

Version determination follows a hierarchical strategy implemented in [scripts/build.sh 48-107](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L48-L107):

**Version formats:**

*   Tagged release: `1.2.3`
*   Development: `1.2.3-15-gdeadbeef-feature-branch`
*   Fork: `user-1.2.3-feature`

**Sources:**[scripts/build.sh 48-108](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L48-L108)

### Configuration Tracking

The `sfpi-info.sh` script provides a single source of truth for version information and system detection:

`# Usage patternseval $(./scripts/sfpi-info.sh VERSION 1.2.3)  # Release modeeval $(./scripts/sfpi-info.sh SHELL)           # User mode./scripts/sfpi-info.sh CMAKE                   # CMake mode`
**Generated variables:**

*   `sfpi_version` - Full version string
*   `sfpi_arch` - `x86_64` or `aarch64` (from `uname -m`)
*   `sfpi_dist` - `debian` or `fedora` (from `/etc/os-release`)
*   `sfpi_pkg` - `deb`, `rpm`, or `txz`
*   `sfpi_filename` - Complete artifact name
*   `sfpi_hashtype` - `sha256`
*   `sfpi_url` - GitHub release download URL

The workflow captures this configuration and stores it in the `outputs/config-{key}` file for multi-matrix builds.

**Sources:**[scripts/sfpi-info.sh 78-140](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L78-L140)[.github/workflows/build-sfpi.yaml 95-106](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L95-L106)

### Source Integrity Verification

During release, the script verifies that the source tree matches the build-time state by comparing git commit hashes:

`# At build timescripts/git-hash.sh "$tt_version" > build/sfpi/README.txt # At release timescripts/git-hash.sh "$tt_version" > build/README.postcmp -s build/sfpi/README.txt build/README.post`
This check is skipped in CI mode (`--ci` flag) since GitHub Actions ensures a clean checkout.

**Sources:**[scripts/release.sh 37-46](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L37-L46)[scripts/build.sh 113](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L113-L113)

## Workflow Integration Patterns

### Reusable Workflow Invocation

The `workflow_call` trigger allows this workflow to be invoked from other workflows (e.g., release automation):

`jobs:  build-matrix:    strategy:      matrix:        arch: [x86_64, aarch64]        box: [none, Alma]    uses: ./.github/workflows/build-sfpi.yaml    with:      tag: ${{ github.ref }}      arch: ${{ matrix.arch }}      box: ${{ matrix.box }}      key: ${{ matrix.arch }}-${{ matrix.box }}`
The `key` input enables parallel matrix builds to upload uniquely named configuration artifacts.

**Sources:**[.github/workflows/build-sfpi.yaml 4-17](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L4-L17)

### Artifact Consumption

Downstream projects (like tt-metal) consume these artifacts via CMake's `FetchContent` mechanism, using the checksums stored in `sfpi-version` files to verify integrity. For details on this integration, see [Integration with tt-metal](https://deepwiki.com/tenstorrent/sfpi/7.1-integration-with-tt-metal).
