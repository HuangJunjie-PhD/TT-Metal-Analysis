This page explains debugging techniques for TT-XLA compilation and runtime issues. It covers the `TTXLA_LOGGER_LEVEL` environment variable for enabling debug logging and verbose IR printing, the `export_path` and `export_model_name` compile options for IR file dumps, test-time debugging with `--serialize` and `@pytest.mark.filecheck`, source location metadata via `XLA_HLO_DEBUG`, and common debugging workflows.

For background on the compilation pipeline stages, see page [4.1](https://github.com/tenstorrent/tt-xla/blob/c77995f6/4.1) For the test infrastructure that uses these debugging tools, see page [6.3](https://github.com/tenstorrent/tt-xla/blob/c77995f6/6.3)

## Logger Configuration: TTXLA_LOGGER_LEVEL

The `TTXLA_LOGGER_LEVEL` environment variable is the primary entry point for enabling debug output in TT-XLA. It controls both the verbosity of Python-side logging (via `loguru`) and C++-side MLIR pass manager diagnostics.

**Log Levels** (defined in `python_package/ttxla_tools/logging.py`):

| Level | Usage |
| --- | --- |
| `TRACE` | Finest granularity; rarely used |
| `DEBUG` | Detailed diagnostic information useful for debugging |
| `INFO` | Confirmation that things are working as expected |
| `SUCCESS` | Highlighted success messages |
| `WARNING` | Default level; indicates potential issues |
| `ERROR` | Serious problems that prevent execution |
| `CRITICAL` | Very serious errors |

**Setting the logger level:**

**Effect on MLIR Pass Manager:**

When `TTXLA_LOGGER_LEVEL=VERBOSE`, the MLIR pass manager's `enableVerboseIRPrinting` is activated, causing the full IR text to be printed after **every individual MLIR pass**. This is useful when debugging mid-pipeline transformations:

The verbose IR printing is triggered in `ModuleBuilder::buildModule` before running each `PassManager`. For debug-level logging without full IR dumps, use `DEBUG` instead of `VERBOSE`.

## Enabling IR Export via CompileOptions

IR dumping is controlled by two fields in `CompileOptions`:

| Field | Type | Description |
| --- | --- | --- |
| `export_path` | `std::optional<std::string>` | Directory for all exported artifacts. Setting this **enables** IR dumping. When using a non-flatbuffer backend, this field is mandatory. |
| `export_model_name` | `std::string` | Human-readable model identifier. When set, the graph counter (`g0`, `g1`, …) is appended automatically. The counter resets when `export_model_name` changes between compilations. |

Sources: [pjrt_implementation/inc/api/compile_options.h 120-127](https://github.com/tenstorrent/tt-xla/blob/c77995f6/pjrt_implementation/inc/api/compile_options.h#L120-L127)[pjrt_implementation/src/api/compile_options.cc 69-73](https://github.com/tenstorrent/tt-xla/blob/c77995f6/pjrt_implementation/src/api/compile_options.cc#L69-L73)

### File Naming Convention

IR files land in `{export_path}/irs/` and follow this pattern:

```
<stage>_<model_name>_g<N>_<timestamp>.mlir
```

Where:

*   `<stage>` is one of: `vhlo`, `shlo`, `shlo_frontend`, `shlo_compiler`, `shlo_compiler_cleaned`, `ttir`, `ttnn`
*   `<model_name>` is the sanitized `export_model_name` (forward slashes replaced with underscores)
*   `<N>` is the graph counter (0-indexed, increments per compilation call within the same model name)
*   `<timestamp>` is milliseconds since epoch

The graph counter logic is in `ModuleBuilder::buildModule`:

## Common Debugging Workflows

### Debugging a Compilation Failure

**Workflow:**

**Steps:**

1.   Set `TTXLA_LOGGER_LEVEL=DEBUG` to see diagnostic messages
2.   If the error is vague, enable IR dumps:  
3.   Examine the last successfully generated IR file to identify where the pipeline failed
4.   If the failure is mid-pass, use `TTXLA_LOGGER_LEVEL=VERBOSE` to see IR after each MLIR pass

Sources: [python_package/tt_torch/backend/backend.py 34-96](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/backend.py#L34-L96)[docs/src/getting_started_build_from_source.md 123](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/getting_started_build_from_source.md?plain=1#L123-L123)

### Debugging Incorrect Results

**Workflow:**

**Steps:**

1.   Verify the issue by comparing against CPU execution
2.   Enable IR dumps to inspect compiled operations
3.   Disable fusion passes or composite ops selectively to isolate the bug:  
4.   Write a minimal reproduction test with FileCheck assertions (see next section)

Sources: [python_package/tt_torch/backend/backend.py 43-59](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/backend.py#L43-L59)[python_package/tt_torch/backend/passes.py 13-32](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/passes.py#L13-L32)[tests/torch/ops/test_fusion_ops.py 16-39](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/torch/ops/test_fusion_ops.py#L16-L39)

### Debugging Performance Issues

**Approach:**

1.   **Identify bottlenecks:**

    *   Check if compilation time is excessive (set `TTXLA_LOGGER_LEVEL=DEBUG` to see compilation stages timing)
    *   Check if runtime is slow compared to expectations

2.   **Compilation performance:**

    *   Excessive recompilation: Check if `torch.compile` is retracing the graph on each call
    *   For vLLM: Ensure `dummy_run` pre-compilation is working (see page [5.3.1](https://github.com/tenstorrent/tt-xla/blob/c77995f6/5.3.1))
    *   Use `tt_experimental_compile` / `tt_legacy_compile` options to compare compilation flows

3.   **Runtime performance:**

    *   Profile device execution using Tracy (if available; not currently documented in provided files)
    *   Check tensor shapes and memory layout in TTNN IR dumps
    *   Look for unexpected data movement operations in the IR

4.   **Graph optimization:**

    *   Verify fusion passes are enabled (`tt_enable_torch_fx_fusion_pass: True`)
    *   Verify composite ops are being used where appropriate (`tt_enable_composite_ops: True`)
    *   Inspect `ttir_*.mlir` to confirm high-level operations (e.g., `ttir.rms_norm`) appear instead of decomposed ops

Sources: [python_package/tt_torch/backend/backend.py 254-269](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/backend.py#L254-L269)[python_package/tt_torch/fusion_providers.py 75-91](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/fusion_providers.py#L75-L91)

## FileCheck: IR Pattern Verification in Tests

FileCheck allows tests to assert that specific MLIR operation names appear in a compiled IR output. The mechanism has two parts: a **pattern file** and a **test decorator**.

### Pattern Files

Pattern files live in `tests/filecheck/` and contain `//CHECK:` directives compatible with LLVM's FileCheck tool:

| File | Pattern |
| --- | --- |
| `tests/filecheck/add.ttnn.mlir` | `//CHECK: "ttnn.add"` |
| `tests/filecheck/rms_norm.ttir.mlir` | `//CHECK: "ttir.rms_norm"` |

The filename encodes which stage's IR is matched: `<name>.<stage>.mlir`.

## Source Location Metadata (XLA_HLO_DEBUG)

The PyTorch backend can embed source-file location information from the FX graph into the XLA HLO IR, making it easier to trace compiled HLO operations back to the original Python code.

**Activation:**

This is gated in `XLAExecutor.__init__`:

**How it works:**

The `extract_nodes_info` function reads `stack_trace` and `nn_module_stack` from each FX node's `.meta` dict, resolves the enclosing function using `_find_enclosing_function`, and returns a `dict[str, str]` mapping node names to serialized `EmitLoc` strings.

`MetadataInterpreter` (a `torch.fx.Interpreter` subclass) sets a `ContextVar` before each node executes. `MetadataDispatchMode` (a `TorchDispatchMode`) reads that `ContextVar` when any aten operation is dispatched and calls `torch_xla._XLAC._set_xla_custom_op_name_prefix` on the result tensor.

This is a debug-only feature. When `XLA_HLO_DEBUG` is not set, `XLAExecutor` runs the module directly without the interpreter or dispatch mode overhead.

Sources: [python_package/tt_torch/backend/backend.py 119-251](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/backend.py#L119-L251)[python_package/tt_torch/backend/metadata_propagation.py 1-491](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/tt_torch/backend/metadata_propagation.py#L1-L491)

## Summary: Debug Options Reference

| Goal | Mechanism | How to activate |
| --- | --- | --- |
| Dump IR at each pipeline stage to files | `export_path` + `export_model_name` compile options | `set_custom_compile_options({"export_path": "..."})` |
| Print full IR after every MLIR pass | `enableVerboseIRPrinting` | `TTXLA_LOGGER_LEVEL=VERBOSE` |
| General debug logs | `TTXLA_LOGGER_LEVEL` | `TTXLA_LOGGER_LEVEL=DEBUG` |
| Assert IR operation names in tests | `@pytest.mark.filecheck([...])` | Decorator + `request` fixture |
| Dump IR artifacts from any test | `--serialize` pytest flag | `pytest ... --serialize` |
| Embed FX source locations in HLO IR | `MetadataInterpreter` + `MetadataDispatchMode` | `XLA_HLO_DEBUG=1` |
