This document covers the containerization system that provides consistent, reproducible environments for executing RISC-V architectural tests. The container infrastructure uses Podman to package prebuilt Whisper and Spike instruction set simulators along with their dependencies, enabling standardized test execution across different environments.

For information about the test execution runners that utilize these containers, see [Test Execution](https://deepwiki.com/tenstorrent/riscv_arch_tests/3.1-test-execution). For CI/CD pipeline integration details, see [CI/CD Integration](https://deepwiki.com/tenstorrent/riscv_arch_tests/3.4-cicd-integration).

## Purpose and Architecture

The container infrastructure serves as the foundation for consistent test environments by packaging RISC-V simulators and their dependencies into portable containers. The system is built around Podman as the primary container runtime, with Docker compatibility as a secondary goal.

## Container Build Process

The container build system automatically retrieves specific SHA commits from the Whisper and Spike submodules and compiles them into the container image.

### Build Script Organization

The build process follows this sequence:

1.   **Submodule SHA Retrieval**: The `get_submodule_sha()` function extracts commit SHAs from git submodules [infra/container/container_common.sh 11-23](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_common.sh#L11-L23)
2.   **Build Argument Preparation**: SHA values are passed as build arguments to Podman [infra/container-build 11-16](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-build#L11-L16)
3.   **Container Build**: Podman builds the image using the Containerfile with the specified commit SHAs [infra/container-build 20](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-build#L20-L20)

**Sources:**[infra/container-build 6-20](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-build#L6-L20)[infra/container/container_common.sh 11-23](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_common.sh#L11-L23)[infra/container/container_build.sh 10-20](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_build.sh#L10-L20)

## Container Image Definition

The `Containerfile` defines a comprehensive RISC-V development environment based on Ubuntu 22.04:

| Component | Purpose | Installation Method |
| --- | --- | --- |
| Base System | Ubuntu 22.04 foundation | FROM directive |
| Build Tools | C++ compilation support | apt packages: `build-essential`, `g++` |
| Whisper Dependencies | Boost libraries for Whisper | `libboost-all-dev` |
| Spike Dependencies | Device tree compilation | `device-tree-compiler` |
| RISC-V Toolchain | Cross-compilation support | `gcc-riscv64-unknown-elf` |
| Python Environment | Test script support | `python3-pip`, `prettytable` |

### Simulator Build Process

Both simulators are built from source using their respective commit SHAs:

**Whisper Build**[infra/container/Containerfile 34-38](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/Containerfile#L34-L38):

*   Clones from `https://github.com/tenstorrent/whisper.git`
*   Builds with Boost library support
*   Installs binary to `/usr/bin/whisper`

**Spike Build**[infra/container/Containerfile 43-50](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/Containerfile#L43-L50):

*   Clones from `https://github.com/tenstorrent/spike.git`
*   Configures with Tenstorrent-specific extensions
*   Builds with ISA extensions: `RV64IMAFDCV_ZBA_ZBB_ZBC_ZBS`
*   Installs binary to `/usr/bin/spike`

**Sources:**[infra/container/Containerfile 1-53](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/Containerfile#L1-L53)

## Container Execution

The container runtime system provides both interactive and command execution modes through volume mounting and working directory preservation.

### Runtime Configuration

The execution scripts implement these key features:

*   **Volume Mounting**: Current directory mounted at same path inside container [infra/container-run 9](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-run#L9-L9)
*   **Working Directory**: Container working directory matches host [infra/container-run 9](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-run#L9-L9)
*   **Interactive Mode**: TTY allocation for interactive sessions [infra/container-run 12](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-run#L12-L12)
*   **Command Pass-through**: Arguments forwarded to container [infra/container-run 12](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-run#L12-L12)

### Execution Modes

| Mode | Command | Behavior |
| --- | --- | --- |
| Interactive | `./infra/container-run` | Enters bash shell in container |
| Command Execution | `./infra/container-run <cmd>` | Executes command and exits |
| Test Execution | `./infra/container-run python3 infra/quals.py` | Runs test suite in container |

**Sources:**[infra/README.md 14-23](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/README.md?plain=1#L14-L23)[infra/container-run 9-12](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container-run#L9-L12)[infra/container/container_run.sh 9-12](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_run.sh#L9-L12)

## Registry Integration

The container push system integrates with GitLab's container registry for CI/CD workflows.

### Push Process

The `container_push.sh` script performs these operations:

1.   **Container ID Extraction**: Retrieves the built image ID [infra/container/container_push.sh 14](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_push.sh#L14-L14)
2.   **Registry Push**: Uploads to GitLab registry with container ID as tag [infra/container/container_push.sh 17](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_push.sh#L17-L17)
3.   **CI Configuration Generation**: Creates `gitlab-container.yml` for CI use [infra/container/container_push.sh 19-22](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_push.sh#L19-L22)

The generated configuration file enables CI pipelines to reference the specific container image:

`image:    name: $CI_REGISTRY_IMAGE:<container_id>`
**Note**: This script is marked as internal-only and excluded from public releases [infra/container/container_push.sh 3](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_push.sh#L3-L3)

**Sources:**[infra/container/container_push.sh 1-24](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_push.sh#L1-L24)

The `container_common.sh` provides shared functionality:

*   **Containerfile Path**: Resolves to `Containerfile` in script directory [infra/container/container_common.sh 2](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/container_common.sh#L2-L2)
