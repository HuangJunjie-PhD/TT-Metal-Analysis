This document covers the integration of Tenstorrent hardware devices with TT-Studio, including automatic hardware detection, Docker device mounting, and configuration management. This integration enables AI models to run directly on Tenstorrent hardware through the TT-Metal framework.

For general installation and setup procedures, see [Installation and Setup](https://deepwiki.com/tenstorrent/tt-studio/2.1-installation-and-setup). For environment configuration details, see [Environment Configuration](https://deepwiki.com/tenstorrent/tt-studio/2.3-environment-configuration).

## Overview

TT-Studio integrates with Tenstorrent hardware by detecting devices at `/dev/tenstorrent` and conditionally mounting them into Docker containers. The integration uses a layered Docker Compose configuration approach, where hardware-specific settings are applied through an overlay compose file. To use deployment features, users must have access to a Tenstorrent AI accelerator or connect to remote endpoints [README.md 7-8](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L7-L8)

## Hardware Detection Process

The system performs automatic hardware detection during startup to determine if Tenstorrent devices are available and should be mounted.

### Detection Flow

The detection logic is implemented in `startup.sh`:

*   Checks for existence of `/dev/tenstorrent` device [startup.sh 158-160](https://github.com/tenstorrent/tt-studio/blob/a6d347af/startup.sh#L158-L160)
*   Provides the `--tt-hardware` flag to force hardware mode [startup.sh 158-160](https://github.com/tenstorrent/tt-studio/blob/a6d347af/startup.sh#L158-L160)
*   Updates `.env` file with `ENABLE_TT_HARDWARE` setting based on user interaction or flags.

**Sources**: [startup.sh 158-160](https://github.com/tenstorrent/tt-studio/blob/a6d347af/startup.sh#L158-L160)[README.md 159-160](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L159-L160)

### Environment Configuration

Hardware detection results are stored in environment variables that control Docker Compose behavior. These variables are defined in the `.env` file, using `.env.default` as a template [app/docker-compose.yml 30-31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L30-L31)

| Variable | Purpose | Values |
| --- | --- | --- |
| `ENABLE_TT_HARDWARE` | Controls hardware mounting | `true`, `false` |
| `TT_STUDIO_ROOT` | Base directory path | Absolute path [app/docker-compose.yml 34](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L34-L34) |
| `VITE_ENABLE_DEPLOYED` | Enables deployment UI features | `true`, `false`[app/docker-compose.yml 13](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L13-L13) |

## Docker Compose Integration

The hardware integration uses a multi-file Docker Compose strategy with conditional overlays to provide modularity between base services and hardware-specific requirements.

### Compose File Architecture

**Sources**: [app/docker-compose.yml 5-147](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L5-L147)[app/docker-compose.tt-hardware.yml 1-5](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.tt-hardware.yml#L1-L5)[app/README.md 48-51](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/README.md?plain=1#L48-L51)

### Base Configuration

The base Docker Compose configuration in `app/docker-compose.yml` defines the `tt_studio_backend` service. While it contains commented-out device lines for manual developer use [app/docker-compose.yml 14-17](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L14-L17) the production-ready hardware integration relies on the overlay file.

### Hardware Overlay

The hardware-specific configuration in `app/docker-compose.tt-hardware.yml` provides the specific device mapping required for the TT-Metal framework to communicate with the chips:

`services:  tt_studio_backend:    devices:      - /dev/tenstorrent:/dev/tenstorrent`
This overlay adds the `/dev/tenstorrent` device to the `tt_studio_backend` container, allowing it to manage hardware-accelerated model deployments [app/docker-compose.tt-hardware.yml 1-5](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.tt-hardware.yml#L1-L5)

## Service Configuration

### Backend Service Integration

The `tt_studio_backend` service is the central hub for hardware interaction. It orchestrates model containers and requires hardware access to validate chip availability.

The backend service configuration includes:

*   **Port Mapping**: `8000:8000`[app/docker-compose.yml 22](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L22-L22)
*   **Healthchecks**: Verifies the `/up/` endpoint before allowing dependent services (like frontend) to start [app/docker-compose.yml 65-73](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L65-L73)
*   **Docker Control**: Communicates with the Docker Control Service via `DOCKER_CONTROL_SERVICE_URL` to manage model containers without mounting the raw docker socket [app/docker-compose.yml 55-57](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L55-L57)

**Sources**: [app/docker-compose.yml 6-76](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L6-L76)[app/docker-compose.tt-hardware.yml 1-5](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.tt-hardware.yml#L1-L5)

### Compose Command Generation

The `run.py` (and deprecated `startup.sh`) scripts dynamically build the Docker Compose command. For development on hardware, the following files are combined:

1.   `docker-compose.yml` (Base)
2.   `docker-compose.dev-mode.yml` (Development overrides)
3.   `docker-compose.tt-hardware.yml` (Hardware access)

Example manual command for hardware development:

`docker compose -f app/docker-compose.yml -f app/docker-compose.dev-mode.yml -f app/docker-compose.tt-hardware.yml up`

## Integration with Model Execution

### Hardware Access Path

Models are deployed as separate Docker containers. Each model container typically runs on ports 7000+ [README.md 77](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L77-L77)

The integration ensures that:

1.   The backend detects available Tenstorrent chips.
2.   The `tt_studio_backend` requests the creation of model containers via the Docker Control Service [app/docker-compose.yml 55-57](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L55-L57)
3.   Model containers are launched with the necessary device mappings to access `/dev/tenstorrent`.

## Network Architecture

All TT-Studio services communicate over a dedicated internal network named `tt_studio_network`. This network is defined as `external` to prevent Docker Compose from prefixing the name, which is critical for the backend's ability to programmatically manage other containers on the same network [app/docker-compose.yml 149-154](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L149-L154)
