## Purpose and Scope

Stress and long-duration tests validate the reliability and stability of the firmware under sustained operation and repeated reset cycles. These tests run 100-1000 iterations of critical operations to uncover race conditions, memory leaks, and edge cases that may not manifest in smoke tests. The test suite focuses on reset mechanisms, firmware upgrade paths, hardware sensor reliability, and performance under load.

For basic functionality validation, see [Hardware Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests). For TT-Metal runtime integration testing, see [TT-Metal Integration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.5-tt-metal-integration-tests).

## Test Categories

### Reset Mechanism Tests

These tests validate that the firmware can reliably recover from various reset scenarios:

**Test:**`test_arc_watchdog` validates the DMC watchdog timer will reset the ARC firmware when it hangs. Runs up to 100 iterations. Allows 0% failure rate.

**Test:**`test_smi_reset` validates `tt-smi -r` command reliability. Runs up to 200 iterations (reduced from 1000 due to longer UMD health check duration). Allows 1% failure rate. Tracks average and maximum DMC ping latency after reset.

**Test:**`test_dirty_reset` simulates a "dirty" reset where the DMC resets without SMC coordination (similar to NOC hang recovery). Runs up to 1000 iterations. Allows 1% failure rate.

**Sources:**[app/smc/pytest/e2e_stress.py 76-218](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L76-L218)[app/smc/pytest/e2e_smoke.py 551-636](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L551-L636)[app/smc/pytest/e2e_smoke.py 707-738](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L707-L738)[app/smc/pytest/e2e_smoke.py 740-787](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L740-L787)

### Performance Validation Tests

Performance tests measure timing characteristics under repeated execution:

**Test:**`test_pcie_fw_load_time` validates PCIe firmware loading completes within 40ms from ARC start to PCIe link training completion. Runs 10 iterations with 0% allowed failure rate.

**Test:**`test_dmc_ping` measures SMC-to-DMC communication latency by pinging the DMC 1000 times without resets. Reports average and maximum ping duration. Allows 0% failure rate.

**Sources:**[app/smc/pytest/e2e_stress.py 102-131](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L102-L131)[app/smc/pytest/e2e_stress.py 222-252](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L222-L252)[app/smc/pytest/e2e_smoke.py 647-682](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L647-L682)

### Hardware Sensor Tests

Sensor tests validate PVT (Process, Voltage, Temperature) monitoring under repeated access:

**Test:**`test_temperature_sensors` reads all 8 temperature sensors repeatedly, validating responses fall within 25-70°C range.

**Test:**`test_voltage_monitors` reads all 8 voltage monitors, validating 0.65-0.95V range.

**Test:**`test_process_detectors` reads 16 process detectors across 3 delay chains (19, 20, 21), validating 95-240 MHz frequency range.

**Test:**`test_pvt_comprehensive` performs combined validation of temperature sensors, process detectors, and voltage monitors in a single pass.

All sensor tests allow 0% failure rate.

**Sources:**[app/smc/pytest/e2e_stress.py 287-341](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L287-L341)[app/smc/pytest/e2e_smoke.py 326-411](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L326-L411)

### Power Stress Test

The power virus test validates firmware stability under maximum sustained power draw:

**Test:**`test_power_virus` runs the `tt-burnin` workload for 180 seconds while continuously reading temperature sensors. This validates that the firmware can monitor hardware sensors while the ASIC is under maximum power load. The test verifies the `tt-burnin` process remains alive and all sensor reads succeed. Skipped on Galaxy boards due to stability issues.

**Sources:**[app/smc/pytest/e2e_stress.py 344-436](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L344-L436)

### Firmware Upgrade Tests

Upgrade tests validate migration paths from previous firmware versions:

**Test:**`test_upgrade_from_18x` validates upgrade paths from firmware versions 18.10.0, 18.11.0, and 18.12.0 to the current development version. The test:

1.   Flashes recovery firmware (v18.12.2)
2.   Flashes the base version being tested
3.   Verifies base version telemetry matches expected values
4.   Flashes the current development version
5.   Verifies current version telemetry
6.   Confirms all ASICs enumerate correctly after each flash

Galaxy boards skip the recovery flash step as they lack DMC hardware.

**Sources:**[app/smc/pytest/e2e_stress.py 255-284](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L255-L284)[app/smc/pytest/e2e_smoke.py 211-324](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L211-L324)

### Failure Tolerance Configuration

Each stress test defines its acceptable failure rate:

| Test | Total Iterations | Allowed Failures | Failure Rate |
| --- | --- | --- | --- |
| `test_arc_watchdog` | 100 | 0 | 0% |
| `test_pcie_fw_load_time` | 10 | 0 | 0% |
| `test_smi_reset` | 200 | 2 | 1% |
| `test_dirty_reset` | 1000 | 10 | 1% |
| `test_dmc_ping` | 1000 | 0 | 0% |
| `test_temperature_sensors` | 100 | 0 | 0% |
| `test_process_detectors` | 50 | 0 | 0% |
| `test_voltage_monitors` | 100 | 0 | 0% |
| `test_pvt_comprehensive` | 20 | 0 | 0% |
| `test_power_virus` | 180s | 0 | 0% |

The formula for allowed failures: `failure_fail_count = total_tries // 100` (applied to tests with 1% tolerance).

## Metrics Collection

### DMC Ping Statistics

The SMI reset and DMC ping tests collect latency statistics:

The `PING_DMFW_DURATION_REG_ADDR` (0x80030448) contains the last ping duration in milliseconds. The tests log:

*   Average DMFW ping time after reset
*   Maximum DMFW ping time after reset
*   Average DMFW ping time without reset
*   Maximum DMFW ping time without reset

**Sources:**[app/smc/pytest/e2e_stress.py 146-181](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L146-L181)[app/smc/pytest/e2e_stress.py 229-250](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L229-L250)

## CI Integration

### Workflow Dependencies

The long-duration test workflow has specific dependencies and concurrency controls:

This ensures only one instance of the workflow runs per PR or commit, canceling any in-progress runs when new commits are pushed.

The workflow requires:

1.   `long-test-check` job to verify trigger conditions
2.   `build-fw-artifact` job to produce signed firmware
3.   `gen-config` job to generate board test matrix

**Sources:**[.github/workflows/hardware-long.yml 12-14](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-long.yml#L12-L14)[.github/workflows/hardware-long.yml 17-29](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-long.yml#L17-L29)

### Container Environment

Tests run in the Zephyr CI container with specific device access:

This provides access to:

*   `/dev/tenstorrent`: Hardware device interface
*   `/dev/bus/usb`: USB/JTAG access for firmware flashing
*   `/dev/ipmi0`: IPMI interface for power management
*   HugePages: Required for tt-kmd driver
