## Purpose and Scope

This document presents a comprehensive performance evaluation of multicast write operations on the TT-Metal NOC infrastructure. The study analyzes how sender placement relative to the destination grid, NoC instance selection (NoC0 vs NoC1), and loopback configuration impact multicast throughput. The analysis covers 10 distinct sender placement schemes across varying grid sizes on both Wormhole B0 and Blackhole P100 platforms.

This study provides empirical data and recommendations for optimizing multicast performance. For general NOC architecture information, see [NOC Architecture](https://deepwiki.com/tenstorrent/tt-low-level-documentation/2.2-noc-architecture). For baseline performance metrics of all data movement primitives, see [Baseline Performance Metrics](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.1-baseline-performance-metrics). For API usage patterns, see [API Usage Guide](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3-api-usage-guide).

## Executive Summary

The study evaluated multicast throughput (bytes per cycle) across 10 sender placement schemes with the following key finding: **performance is primarily governed by whether write-acknowledge traffic traverses the same congested physical links as the multicast writes**. When acknowledge paths share congested routes, bandwidth degrades sharply with increasing grid size. Although writes and acknowledges use different virtual channels, they share the same physical links, so virtual channel separation does not eliminate underlying link contention.

**Critical Finding:** Self-interference occurs when high-volume multicast write traffic and returning acknowledge traffic converge on the same congested physical links, creating a positive feedback loop where delayed acknowledges prevent new writes, compounding the bottleneck.

**Sources:**[data_movement_doc/multicast_schemes/Multicast Schemes.md 4-10](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/multicast_schemes/Multicast%20Schemes.md?plain=1#L4-L10)

### Key Observations

**Inside-the-Grid Schemes (1-4):**

*   Slightly higher performance when loopback is disabled
*   Performance relatively consistent across NoC instances

**Outside-the-Grid with Shared Geometry:**

*   **NoC 0:** Column-shared placements (schemes 6, 9) show noticeable performance drops
*   **NoC 1:** Row-shared placements (schemes 5, 8) show noticeable performance drops
*   Performance degradation increases with grid size

**Outside-the-Grid with No Sharing (7, 10):**

*   More resilient as grid size increases
*   Avoid the self-interference problem by not sharing congested paths

## Root Cause Analysis: Self-Interference Mechanism

### Traffic Contention Architecture

### NoC-Specific Routing Emphasis

### Congestion Path Analysis

The markdown document includes detailed congestion diagrams showing write paths versus acknowledge paths for representative schemes:

**NoC 0 Behavior:**

*   **Scheme 5 (row-shared):** Congested path NOT shared by writes and acknowledges → Good performance
*   **Scheme 6 (column-shared):** Congested path IS shared by writes and acknowledges → Poor performance

This demonstrates that the worst-case schemes differ by NoC instance due to their complementary routing emphasis.

## Detailed Root Cause Explanation

### Physical Link Contention vs Virtual Channels

The disproportionate bandwidth loss occurs when:

1.   **High multicast write volume** saturates links leading to the destination grid
2.   **Acknowledge traffic** must traverse the same saturated links to return to sender
3.   **Self-interference feedback loop:** Congested write path delays acknowledges → Delayed acknowledges restrict forward progress → Further compounds bottleneck

### Grid Size Effect

As the subordinate grid grows, more multicast traffic converges on shared links. If the acknowledge path also uses those links, the system experiences increasingly severe self-interference.

**Sources:**[data_movement_doc/multicast_schemes/Multicast Schemes.md 54-60](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/multicast_schemes/Multicast%20Schemes.md?plain=1#L54-L60)

## Recommendations

### Configuration Guidelines

### Recommended Configurations

| NoC Instance | **Avoid** | **Prefer** | Rationale |
| --- | --- | --- | --- |
| **NoC 0** | Column-shared placements (Schemes 6, 9) | Schemes 7, 10 (no sharing) Scheme 5 (row-shared is OK) | NoC 0 emphasizes column routing; column-shared causes self-interference |
| **NoC 1** | Row-shared placements (Schemes 5, 8) | Schemes 7, 10 (no sharing) Scheme 6 (column-shared is OK) | NoC 1 emphasizes row routing; row-shared causes self-interference |
| **Inside Grid** | Loopback enabled | Loopback disabled | Slight performance improvement |

### Design Principle

**Core Recommendation:** Try to use configurations that avoid sharing the congested write path with the acknowledge path. The optimal scheme depends on NoC instance due to their complementary routing emphasis (column vs row).

## Integration with Best Practices

This performance study directly informs the multicast usage recommendations in the Best Practices guide. When implementing multicast operations:

1.   **Choose appropriate sender placement** based on NoC instance selection (avoid row/column sharing as described above)
2.   **Consider disabling loopback** for inside-grid senders for marginal performance gains
3.   **Monitor grid size impact** - larger grids exacerbate self-interference in poorly configured schemes
4.   **Use schemes 7 or 10** when possible as they are resilient across both NoC instances

For detailed API usage patterns and additional optimizations, see the [Best Practices](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3-api-usage-guide) section.

## Platform-Specific Considerations

### Wormhole B0 vs Blackhole P100

Both platforms exhibit the same fundamental self-interference behavior, but differ in scale:

| Characteristic | Wormhole B0 | Blackhole P100 |
| --- | --- | --- |
| **Grid Size Range** | 2×2 to 7×7 | 2×2 to 9×9 |
| **Interface Width** | 256 bits/cycle | 512 bits/cycle |
| **Core Count** | 64 Tensix cores | 110 Tensix cores |

The larger grid sizes and higher throughput on Blackhole amplify the impact of self-interference for poorly configured schemes, making proper sender placement even more critical.

For comprehensive platform comparison details, see [Platform Comparison](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.2-platform-comparison).
