This document describes the hierarchical versioning scheme used in tt-system-firmware, explaining how product, SMC, and DMC firmware versions are independently managed while being packaged together. It covers the VERSION file format, firmware signing and verification mechanisms, build artifact generation, and the release packaging process that produces combined firmware bundles.

For information about the overall system architecture, see [System Architecture](https://deepwiki.com/tenstorrent/tt-system-firmware/1.1-system-architecture). For details on the build system and firmware packaging, see [Build System](https://deepwiki.com/tenstorrent/tt-system-firmware/6-build-system).

* * *

## Version Number Hierarchy

The tt-system-firmware repository employs a three-level hierarchical versioning strategy where a top-level product version encompasses independently versioned firmware applications.

### Version Components

Each version follows a four-component semantic versioning scheme:

**Format**: `MAJOR.MINOR.PATCHLEVEL.TWEAK`

*   **MAJOR**: Incompatible API/interface changes
*   **MINOR**: Backward-compatible functionality additions
*   **PATCHLEVEL**: Backward-compatible bug fixes
*   **TWEAK**: Build metadata or patch-level increments

## VERSION File Format

Each VERSION file contains six fields that define the version number and optional metadata:

| Field | Description | Example |
| --- | --- | --- |
| `VERSION_MAJOR` | Major version number | `19` |
| `VERSION_MINOR` | Minor version number | `7` |
| `PATCHLEVEL` | Patch level | `99` |
| `VERSION_TWEAK` | Tweak/build number | `0` |
| `EXTRAVERSION` | Optional suffix (e.g., `-rc1`, `-beta`) | Empty or `-rc1` |

### VERSION File Locations

| Path | Purpose | Current Version |
| --- | --- | --- |
| [VERSION 1-5](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/VERSION#L1-L5) | Product-level version for entire release | `19.7.99.0` |
| [app/smc/VERSION 1-5](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/VERSION#L1-L5) | SMC (System Management Controller) firmware | `0.29.99.0` |

### Version File Structure

Each VERSION file follows this format:

```
VERSION_MAJOR = <number>
VERSION_MINOR = <number>
PATCHLEVEL = <number>
VERSION_TWEAK = <number>
EXTRAVERSION = <suffix>
```

Note: The trailing blank line is required by the parsing logic.

## Binary Blob Integrity

In addition to application firmware, the system includes pre-compiled binary blobs for hardware subsystems (MRISC, ERISC, SerDes, bootrom). These blobs must maintain integrity across releases.

### Blob Verification Workflow

**Sources**: [scripts/verify_blob.py 34-56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L34-L56)[.github/workflows/verify-blob.yml 44-47](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/verify-blob.yml#L44-L47)

### Blob Verification Algorithm

The verification script [verify_blob.py](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/verify_blob.py) implements the following logic:

1.   Parse `zephyr/module.yml` to extract blob definitions
2.   For each blob entry: 
    *   Read the declared SHA256 checksum from `module.yml`
    *   Compute actual SHA256 of the blob file
    *   Compare declared vs. computed checksums

3.   Exit with status 1 if any mismatch is detected

**Key Functions**:

*   `get_cur_blobs()`: [scripts/verify_blob.py 14-22](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L14-L22) - Parses blob definitions from YAML
*   `generate_expected_blob_sha256()`: [scripts/verify_blob.py 25-31](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L25-L31) - Computes SHA256 of blob file
*   `verify_blobs()`: [scripts/verify_blob.py 34-56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L34-L56) - Main verification loop

## Recovery Firmware

Recovery firmware provides a fail-safe mechanism for board recovery when normal firmware fails to boot or becomes corrupted.

### Recovery Build Configuration

Recovery firmware is built with special configuration overrides:

| Configuration | Value | Purpose |
| --- | --- | --- |
| Board Target | `tt_blackhole@p100a/tt_blackhole/smc` | Generic board support |
| Extra Config | `sysbuild/recovery.conf` | Recovery-specific options |
| Signature | Required (`-DCONFIG_MCUBOOT_SIGNATURE_KEY_FILE`) | Must be signed |
| Unsigned Images | Disabled (`-DCONFIG_MCUBOOT_GENERATE_UNSIGNED_IMAGE=n`) | Force signing |

### Recovery Build Process

**Sources**: [.github/workflows/build-fw.yml 140-203](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L140-L203)

### Recovery Firmware Usage

Recovery firmware is deployed when:

*   Normal firmware fails signature verification
*   Firmware update is interrupted or corrupted
*   Manual recovery mode is triggered via JTAG or hardware pins
*   Watchdog timeout occurs during boot

## Version Retrieval in Runtime

At runtime, firmware versions can be queried through multiple interfaces:

### SMC Telemetry Interface

The SMC firmware exposes version information through telemetry registers (TAG system):

*   **Product Version**: `TAG_FW_BUNDLE_VERSION_MAJOR/MINOR/PATCH/TWEAK`
*   **SMC Version**: `TAG_APP_FW_VERSION_MAJOR/MINOR/PATCH/TWEAK`
*   **DMC Version**: Available through CM2DM protocol

### Host Query Methods

Host software can retrieve versions using:

1.   **pyluwen**: Python library for hardware management
2.   **tt-smi**: System management interface tool
3.   **Direct register access**: Via PCIe BAR or SMBus

### Version Validation

During firmware updates, the system validates:

*   Signature authenticity using MCUBoot verification
*   Version compatibility (preventing downgrades if configured)
*   Board variant matching
*   Blob checksums for binary components

## Version Change Management

When updating versions, developers must:

1.   **Update VERSION files**: Modify [VERSION 1-5](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/VERSION#L1-L5)[app/smc/VERSION 1-5](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/VERSION#L1-L5)[app/dmc/VERSION 1-5](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/VERSION#L1-L5)
2.   **Verify builds**: Run local `west build` to ensure no build breaks
3.   **Update blobs**: If binary blobs change, update checksums in `zephyr/module.yml`
4.   **Test signing**: Verify signature generation works with test keys
5.   **Update documentation**: Document version changes in release notes

### Version Bump Guidelines

| Change Type | Version Component | Example |
| --- | --- | --- |
| Breaking API changes | MAJOR | `19.7.99.0` → `20.0.0.0` |
| New features | MINOR | `19.7.99.0` → `19.8.0.0` |
| Bug fixes | PATCHLEVEL | `19.7.99.0` → `19.7.100.0` |
| Build tweaks | TWEAK | `19.7.99.0` → `19.7.99.1` |

### Sub-Application Independence

SMC and DMC versions can be incremented independently of the product version:

*   **SMC update only**: Product 19.7.99.0 → 19.7.100.0, SMC 0.29.99.0 → 0.30.0.0, DMC unchanged
*   **DMC update only**: Product 19.7.99.0 → 19.7.100.0, DMC 0.23.99.0 → 0.24.0.0, SMC unchanged
*   **Coordinated update**: All three versions increment for major releases
