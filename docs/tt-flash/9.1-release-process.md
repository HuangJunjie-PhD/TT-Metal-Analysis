# Release Process

Relevant source files
*   [.github/workflows/build-pypi.yml](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-pypi.yml)

This page documents the automated release workflow that orchestrates version bumping, changelog generation, package building, Git tagging, and distribution publishing. The release process is implemented as a GitHub Actions workflow that coordinates all steps required to produce and publish a new version of tt-flash.

For detailed information about Debian package building, see [Debian Packaging](https://deepwiki.com/tenstorrent/tt-flash/9.2-debian-packaging). For Python wheel building and PyPI publication specifics, see [PyPI Distribution](https://deepwiki.com/tenstorrent/tt-flash/9.3-pypi-distribution).

* * *

## Release Workflow Architecture

The release process is implemented in [.github/workflows/release.yml 1-453](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L1-L453) and follows a multi-stage pipeline with explicit dependency ordering. The workflow can be manually triggered via `workflow_dispatch` with a version bump type selection (patch, minor, or major).

**Workflow Stages**:

| Stage | Job Name | Purpose | Key Outputs |
| --- | --- | --- | --- |
| 1 | `create-temp-branch` | Isolates release changes on temporary branch | `temp_branch_ref` |
| 2 | `versionchange` | Increments version number in `pyproject.toml` | `package_version_new`, `git_hash` |
| 3 | `changelogs` | Generates Debian changelog entries | Updated `debian/changelog` |
| 4 | `build_all_depends` | Triggers parallel Debian and PyPI builds | `.deb` and `.whl` artifacts |
| 5 | `tagrelease` | Creates annotated Git tag | Git tag `v{version}` |
| 6 | `generate-release` | Creates GitHub Release with artifacts | GitHub Release with attachments |
| 7 | `mergeback` | Merges temporary branch back to main | Updated main branch |
| 8 | `publish-to-testpypi` | Publishes to TestPyPI for validation | TestPyPI publication |
| 9 | `publish-to-pypi` | Publishes to production PyPI | PyPI publication |

**Sources**: [.github/workflows/release.yml 1-453](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L1-L453)

## Temporary Branch Strategy

The release process uses a temporary branch to isolate all release-related commits before merging back to the main branch. This prevents conflicts with concurrent development work and allows atomic release operations.

The temporary branch naming follows the pattern `rc-temp-{short_sha}-{timestamp}`[.github/workflows/release.yml 56-58](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L56-L58) This ensures uniqueness and provides traceability back to the triggering commit.

**Sources**: [.github/workflows/release.yml 41-62](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L41-L62)

## Version Bumping Process

Version management uses the `bump2version` tool to automatically increment version numbers in `pyproject.toml`. The workflow reads the current version, applies the requested bump type, and commits the change.

**Implementation Details**:

The `versionchange` job [.github/workflows/release.yml 64-160](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L64-L160) performs the following operations:

1.   **Environment Setup**: Creates Python virtual environment and installs `toml-cli`[.github/workflows/release.yml 96-100](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L96-L100)
2.   **Current Version Extraction**: Uses `toml get project.version --toml-path pyproject.toml`[.github/workflows/release.yml 105](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L105-L105)
3.   **Version Increment**: Invokes `bump2version` with the selected bump type [.github/workflows/release.yml 119-121](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L119-L121)
4.   **New Version Extraction**: Reads the updated version from `pyproject.toml`[.github/workflows/release.yml 129](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L129-L129)
5.   **Version Parsing**: Uses `release-kit/semver@v2` to extract major, minor, patch components [.github/workflows/release.yml 132-136](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L132-L136)
6.   **Commit Count**: Calculates commits since last tag using `git rev-list`[.github/workflows/release.yml 140](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L140-L140)

**Output Variables**:

| Variable | Description | Example |
| --- | --- | --- |
| `package_version_new` | Full new version string | `"3.5.0"` |
| `version_major` | Major version component | `3` |
| `version_minor` | Minor version component | `5` |
| `version_patch` | Patch version component | `0` |
| `git_hash` | Commit hash after version bump | `abc123...` |
| `number_of_commits_since_tag` | Commits since previous tag | `42` |

**Sources**: [.github/workflows/release.yml 64-160](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L64-L160)

## Changelog Generation

Debian-style changelog entries are automatically generated using Git BuildPackage tools (`gbp dch`). This creates properly formatted entries in [debian/changelog](https://github.com/tenstorrent/tt-flash/blob/60c06032/debian/changelog) that comply with Debian packaging standards.

**Command Invocation**:

The `changelogs` job [.github/workflows/release.yml 165-236](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L165-L236) executes:

`gbp dch \  --debian-branch ${{ temp_branch_ref }} \  -R \  -N ${MAJOR}.${MINOR}.${PATCH} \  --spawn-editor=never`
**Parameters**:

*   `--debian-branch`: Specifies the temporary release branch [.github/workflows/release.yml 210](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L210-L210)
*   `-R`: Releases the changelog (removes UNRELEASED status)
*   `-N`: Sets the new version number [.github/workflows/release.yml 212](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L212-L212)
*   `--spawn-editor=never`: Automated mode without interactive editing [.github/workflows/release.yml 213](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L213-L213)

The generated changelog follows the format visible in [CHANGELOG.md 1-96](https://github.com/tenstorrent/tt-flash/blob/60c06032/CHANGELOG.md?plain=1#L1-L96) with dated entries documenting changes between releases.

**Sources**: [.github/workflows/release.yml 165-236](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L165-L236)[CHANGELOG.md 1-96](https://github.com/tenstorrent/tt-flash/blob/60c06032/CHANGELOG.md?plain=1#L1-L96)

## Build Orchestration

After version bumping and changelog generation, the workflow triggers `build-all.yml`[.github/workflows/release.yml 241-253](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L241-L253) which orchestrates parallel builds for Debian packages and Python wheels. This workflow is invoked using `uses: ./.github/workflows/build-all.yml`.

**Passed Parameters**:

`with:  ref: ${{ temp_branch_ref }}           # Build from temp branch  MAJOR: ${{ version_major }}  MINOR: ${{ version_minor }}  PATCH: ${{ version_patch }}  NUMBER_OF_COMMITS_SINCE_TAG: ${{ num_commits }}secrets: inherit                        # Pass all secrets`
The build workflow produces:

*   **Debian packages**: `.deb` files for Ubuntu 22.04, 24.04, and latest
*   **Python wheels**: `.whl` files compatible with Python 3.10+

All artifacts are uploaded to GitHub Actions artifact storage for later retrieval during release creation.

**Sources**: [.github/workflows/release.yml 241-253](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L241-L253)

## Git Tagging Strategy

Once builds complete successfully, the workflow creates an annotated Git tag pointing to the commit containing the version bump and changelog. Tags follow the format `v{major}.{minor}.{patch}`.

**Implementation**[.github/workflows/release.yml 257-283](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L257-L283):

`git tag v${{ package_version_new }}git taggit push --tags`
The tag is created on the commit hash from the `changelogs` job [.github/workflows/release.yml 274](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L274-L274) ensuring it includes all release-related changes. This tag serves as the stable reference point for the release.

**Sources**: [.github/workflows/release.yml 257-283](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L257-L283)

## GitHub Release Creation

The `generate-release` job [.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L288-L357) creates a public GitHub Release with all build artifacts and auto-generated release notes.

**Release Notes Generation**:

The workflow uses `mikepenz/release-changelog-builder-action@v5`[.github/workflows/release.yml 307](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L307-L307) with HYBRID mode to include both pull requests and direct commits. The template formats entries as:

```
- {TITLE} (#{NUMBER}) by @{AUTHOR}
```

**Artifact Attachment**:

The release includes [.github/workflows/release.yml 351-354](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L351-L354):

*   Debian packages: `${{ github.workspace }}/${{ project_name }}*/*.deb`
*   Python wheels: `${{ github.workspace }}/*/*.whl`

Debian packages are renamed before attachment to prevent filename conflicts across different Ubuntu versions [.github/workflows/release.yml 328-344](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L328-L344)

**Sources**: [.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L288-L357)

## Distribution Pipeline

After the GitHub Release is created, packages are published to Python package repositories in a two-stage validation pipeline: TestPyPI first, then production PyPI.

**TestPyPI Publication**[.github/workflows/release.yml 391-419](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L391-L419):

`publish-to-testpypi:  needs:    - build_all_depends    - generate-release  environment:    name: testpypi    url: https://test.pypi.org/p/tt-flash`
The job downloads wheel artifacts and publishes using `pypa/gh-action-pypi-publish@release/v1` with `repository-url: https://test.pypi.org/legacy/`[.github/workflows/release.yml 418](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L418-L418)

**PyPI Publication**[.github/workflows/release.yml 424-452](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L424-L452):

`publish-to-pypi:  needs:    - build_all_depends    - generate-release    - publish-to-testpypi  environment:    name: pypi    url: https://pypi.org/p/tt-flash`
The PyPI job is dependent on successful TestPyPI publication [.github/workflows/release.yml 432](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L432-L432) ensuring the test environment validates the package before production release.

**Trusted Publishing**: Both jobs use `id-token: write` permissions [.github/workflows/release.yml 405-439](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L405-L439) for OIDC-based trusted publishing, eliminating the need for API tokens.

**Sources**: [.github/workflows/release.yml 391-452](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L391-L452)

## Branch Merge Strategy

The final step merges the temporary release branch back to the main branch and cleans up the temporary branch.

**Implementation**[.github/workflows/release.yml 362-386](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L362-L386):

The `mergeback` job checks out the original main branch (not the temporary branch) [.github/workflows/release.yml 376](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L376-L376) and rebases it onto the temporary branch containing all release commits [.github/workflows/release.yml 382](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L382-L382) After pushing the updated main branch, it deletes the temporary branch [.github/workflows/release.yml 386](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L386-L386)

This ensures:

*   Linear commit history on main
*   All release commits are preserved
*   Temporary branches don't accumulate in the repository

**Sources**: [.github/workflows/release.yml 362-386](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L362-L386)

**Version Bump Guidelines**:

*   **Major**: Architecture changes, removed platform support (e.g., 1.0→2.0 for WH, 2.0→3.0 for new generation)
*   **Minor**: New features, significant dependency updates, new platform support
*   **Patch**: Bug fixes, minor dependency updates

## Manual Release Trigger

To trigger a release manually:

1.   Navigate to **Actions** → **Create Release** in the GitHub repository
2.   Click **Run workflow**
3.   Select version bump type: 
    *   `patch`: Bug fixes (3.4.0 → 3.4.1)
    *   `minor`: New features (3.4.0 → 3.5.0)
    *   `major`: Breaking changes (3.4.0 → 4.0.0)

4.   Workflow executes all stages automatically

The workflow requires appropriate repository permissions for creating tags, releases, and publishing packages.

**Sources**: [.github/workflows/release.yml 4-16](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/release.yml#L4-L16)
