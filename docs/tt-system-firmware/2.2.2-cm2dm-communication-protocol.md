## Overview

The CM2DM (Chip Management to Device Management) Communication Protocol defines the message-passing interface between the Chip Management Firmware (CMFW) running on the ARC processor inside the Blackhole ASIC and the Device Management Controller (DMC) firmware running on the RISC-V processor on the board. This protocol enables the DMC to monitor and control chip-level operations based on instructions and status updates from the CMFW.

For information about the DMC's main event loop and overall architecture, see [DMC Main Loop and Event Processing](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.1-dmc-main-loop-and-event-processing). For details about JTAG-based chip reset operations managed by the DMC, see [JTAG and Chip Reset Management](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.3-jtag-and-chip-reset-management).

**Key Characteristics:**

*   **Transport:** SMBus-based communication with CMFW as target (slave) and DMC as controller (master)
*   **Direction:** Primarily CMFW → DMC, with DM2CM feedback for specific operations
*   **Message Types:** 9 distinct message types including reset requests, ping, fan control, and heartbeat
*   **Reliability:** Sequence-numbered messages with explicit acknowledgment and retry logic
*   **Scheduling:** Round-robin message selection to prevent starvation

## Protocol Architecture

The CM2DM protocol operates as an asymmetric master-slave architecture over SMBus. The CMFW acts as an SMBus target device, exposing registers that the DMC polls to retrieve pending messages. The DMC runs a periodic event loop that checks for new messages, processes them, and sends acknowledgments.

### Message State Management

The `Cm2DmMsgState` structure maintains the protocol state on the CMFW side:

*   **`pending_messages`**: Atomic bitmask where bit N indicates message type N has pending data
*   **`next_id_rr`**: Round-robin cursor for fair message scheduling
*   **`next_seq_num`**: Auto-incrementing sequence number for new messages
*   **`curr_msg_valid`**: Indicates if `curr_msg` is awaiting acknowledgment
*   **`curr_msg`**: The most recently transmitted message pending ACK
*   **`next_msgs[]`**: Array storing the latest data for each message type

## Communication Flow

### Message Posting and Selection

The CMFW side uses a round-robin scheduler to ensure fair delivery when multiple message types are pending:

**Round-Robin Selection Algorithm:**

The `next_id_rr()` function implements a fair scheduling policy to prevent message starvation:

1.   Search `pending_messages` bitmask starting from `next_id_rr` position
2.   If no messages found from cursor to MSB, wrap and search from LSB
3.   Use `LOG2(LSB_GET())` to find the lowest set bit efficiently
4.   Update `next_id_rr = (selected_id + 1) % kCm2DmMsgCount` for next iteration

This ensures that if message types 2 and 5 are repeatedly posted, the protocol alternates `2 → 5 → 2 → 5` rather than always prioritizing the lower ID.

### Acknowledgment and Duplicate Detection

After processing a message, the DMC must acknowledge it to prevent retransmission:

**Duplicate Detection:**

The DMC tracks `last_cm2dm_seq_num` per chip to detect repeated messages. When a duplicate is detected (lines 233-243 in main.c), the processor logs a warning and continues polling without reprocessing. This handles the case where the ACK write failed or was not received by the CMFW.

## SMBus Transport Layer

### SMBus Register Map

The CM2DM protocol uses two primary SMBus registers on the CMFW target device:

**Command Definitions:**

Both commands are registered with the SMBus target driver in the CMFW initialization:

```
smbus_req_cmd_def:
  - pec: 1
  - trans_type: kSmbusTransBlockRead
  - send_handler: Cm2DmMsgReqSmbusHandler

smbus_ack_cmd_def:
  - pec: 1
  - trans_type: kSmbusTransWriteWord
  - rcv_handler: Cm2DmMsgAckSmbusHandler
```

### SMBus Transaction Details

| Operation | Function | Protocol Details | Size |
| --- | --- | --- | --- |
| **Read Request** | `bharc_smbus_block_read()` | SMBus Block Read with PEC | 1 byte count + 6 bytes data + 1 byte PEC = 8 bytes total |
| **Write ACK** | `bharc_smbus_word_data_write()` | SMBus Write Word with PEC | 1 byte cmd + 2 bytes data + 1 byte PEC = 4 bytes total |

The SMBus controller driver automatically handles:

*   Start/stop conditions
*   Address byte transmission (0x4C write for read, 0x4C write for write)
*   PEC calculation and verification
*   Clock stretching if CMFW is busy

## DM2CM Protocol (Reverse Direction)

While CM2DM is the primary command channel, the protocol also supports DM2CM messages for feedback and data reporting from the DMC to the CMFW. These use additional SMBus registers where the CMFW acts as the SMBus target.

### DM2CM Message Types

**Key DM2CM Operations:**

| SMBus Register | Purpose | When Sent | Data Structure |
| --- | --- | --- | --- |
| `CMFW_SMBUS_DM_STATIC_INFO` | DMC version info, boot timing | Once per reset | `dmStaticInfo` (20 bytes) |
| `CMFW_SMBUS_PING` / `CMFW_SMBUS_PING_V2` | Respond to ping request | On CM2DM ping message | 0xA5A5 (2 bytes) |
| `CMFW_SMBUS_FAN_SPEED` | Broadcast final fan speed | After fan speed update | Percentage 0-100 (2 bytes) |
| `CMFW_SMBUS_FAN_RPM` | Report fan tachometer | Periodic (20ms) | RPM value (2 bytes) |
| `CMFW_SMBUS_POWER_INSTANT` | Current board power draw | Periodic (1ms) | Watts (2 bytes) |
| `CMFW_SMBUS_DMC_LOG` | Forward DMC logs to CMFW | As logs are generated | Up to 32 bytes of log data |

## Integration with DMC Event Loop

The CM2DM protocol is integrated into the DMC's main event loop through periodic timers and event flags:

**Event Loop Integration Points:**

1.   **Timer Setup** (lines 674-676 in main.c): A 20ms periodic timer posts `TT_EVENT_CM2DM_POLL | TT_EVENT_FAN_RPM_TO_SMC | TT_EVENT_LOGS_TO_SMC`

2.   **Event Processing** (lines 680-709 in main.c): The main loop waits for events and dispatches handlers

3.   **Message Processing** (lines 529-534 in main.c): `handle_cm2dm_messages()` iterates over all chips and processes pending messages

4.   **Processor Dispatch** (lines 205-254 in main.c): `process_cm2dm_message()` retrieves up to `kCm2DmMsgCount` messages per chip and dispatches them to registered processors

## Special Message Behaviors

### Reset Request Handling

The reset request message (`kCm2DmMsgIdResetReq`) has special timing considerations:

**Reset Levels:**

*   **`kCm2DmResetLevelAsic` (0)**: ASIC-only reset via JTAG bootrom sequence, DMC continues running
*   **`kCm2DmResetLevelDmc` (3)**: Full system reset including DMC firmware via `sys_reboot()`

**Bus Transfer Cancellation:**

Before executing a reset, the DMC calls `bh_chip_cancel_bus_transfer_clear()` to abort any pending SMBus transactions. This prevents the reset sequence from hanging if the I2C bus is in an inconsistent state.

### Watchdog and Heartbeat

The auto-reset and heartbeat messages work together to implement a software watchdog:

| Message | Purpose | DMC Action |
| --- | --- | --- |
| `kCm2DmMsgIdAutoResetTimeoutUpdate` | Set watchdog timeout | Start/stop `k_timer` with specified duration |
| `kCm2DmMsgTelemHeartbeatUpdate` | Heartbeat from CMFW | Restart timer if heartbeat value changed |

If the heartbeat stops incrementing (indicating CMFW hang), the timer expires and triggers:

1.   Read ARC PC register via JTAG to capture hang location
2.   Store PC value in `chip->data.arc_hang_pc`
3.   Execute chip reset sequence
4.   Fan forced to 100% until system recovers

### Ping Protocol Variants

The ping message supports two modes distinguished by the `data` field:

*   **Legacy Mode** (`data == 1`): DMC performs block write to `CMFW_SMBUS_PING`, CMFW responds by processing the write and DMC expects write to succeed
*   **V2 Mode** (`data == 0`): DMC performs block read from `CMFW_SMBUS_PING_V2`, CMFW returns 0xA5A5 in the read response

Both modes signal a semaphore (`dmfw_ping_sem`) that the host-facing ping handler waits on to measure round-trip latency. The latency is recorded in `PING_DMFW_DURATION_REG_ADDR` for telemetry.
