This document describes the DRAM tile architecture in Wormhole B0, including the GDDR6 memory interface, NoC integration, and addressing mechanisms. DRAM tiles provide the primary off-chip memory capacity for the ASIC. For information about NoC transactions and addressing, see [NoC Memory Map](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface) and [Transaction Types](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing). For details on other tile types, see [Chip Architecture and Tile Types](https://deepwiki.com/tenstorrent/tt-isa-documentation/1.2-chip-architecture-and-tile-types).

## Purpose and Scope

DRAM tiles expose GDDR6 memory to the NoC, allowing all tiles in the ASIC to read and write data to external memory. Each ASIC contains 18 DRAM tiles organized into 6 independent memory groups (D0 through D5), collectively providing 12 GiB of capacity (6 groups × 2 GiB per group). Each 2 GiB memory bank is accessed through 6 NIUs distributed across 3 tiles. This document covers:

*   DRAM tile internal architecture and components
*   Dual-channel GDDR6 controller and PHY organization
*   NoC Interface Unit (NIU) connectivity patterns (6 NIUs per 2 GiB bank)
*   Memory addressing and tile identification
*   Configuration interfaces and control paths
*   Register maps and memory controller configuration

## Architecture Overview

The ASIC contains 18 DRAM tiles arranged in 6 groups of 3 tiles each (groups D0 through D5). Each group connects to a single 2 GiB GDDR6 memory bank. All 3 tiles in a group access the same physical memory, providing redundant access paths and load balancing across NoC #0 and NoC #1.

**ASIC-Wide DRAM Organization**

**Key architectural characteristics:**

| Property | Value | Notes |
| --- | --- | --- |
| Total DRAM tiles | 18 | Organized into 6 groups |
| Memory groups | 6 (D0-D5) | Each group = 3 tiles |
| Capacity per group | 2 GiB | Shared by 3 tiles |
| Total capacity | 12 GiB | 6 groups × 2 GiB |
| NIUs per memory bank | 6 | 3 tiles × 2 NIUs per tile |
| NIUs per tile | 2 | One for NoC #0, one for NoC #1 |
| Channels per bank | 2 | Each channel: 1 GiB |
| Tile type ID | `8` | In `NOC_ENDPOINT_ID[16:23]` |

## DRAM Tile Internal Architecture

Each DRAM tile contains dual-channel GDDR6 memory controllers, PHYs, and 2 NoC Interface Units (one per NoC). The three tiles within a group provide redundant access to the same physical 2 GiB GDDR6 memory.

**Single DRAM Tile Block Diagram**

**Component details:**

### NoC Interface Units (NIUs)

Each DRAM tile contains exactly 2 NIUs:

*   **NIU #0:** Connected to NoC #0 router, base address `NIU_BASE + 0x0000`
*   **NIU #1:** Connected to NoC #1 router, base address `NIU_BASE + 0x0400`
*   Both NIUs are **slave-only** (cannot initiate NoC transactions)
*   All 6 NIUs across 3 tiles in a group access the same 2 GiB memory

### Dual-Channel GDDR6 Controllers

Each 2 GiB memory bank is divided into two 1 GiB channels:

| Channel | Address Range | Controller | Physical Device |
| --- | --- | --- | --- |
| Channel 0 | `0x0000_0000` - `0x3FFF_FFFF` | Dedicated controller + PHY | 1 GiB GDDR6 |
| Channel 1 | `0x4000_0000` - `0x7FFF_FFFF` | Dedicated controller + PHY | 1 GiB GDDR6 |

**Address mapping:** Bit 30 of the memory address selects the channel:

*   `address[30] == 0` → Channel 0
*   `address[30] == 1` → Channel 1

**Controller responsibilities:**

*   GDDR6 command scheduling (activate, read, write, precharge)
*   Refresh management (auto-refresh, self-refresh)
*   Timing parameter enforcement (tRCD, tRP, tRAS, etc.)
*   Write leveling and read training
*   Error detection and correction (ECC, if enabled)

### AXI Crossbar

The AXI crossbar receives requests from both NIUs and routes them to the appropriate GDDR6 channel based on address bit 30. It also handles:

*   Request arbitration when both NIUs access the same channel
*   AXI protocol conversion (NoC → AXI)
*   Response routing back to the originating NIU

### Configuration Interface

DRAM tiles can be configured via:

1.   **Direct APB path (D0 group only):** ARC tile has dedicated APB connection for boot-time initialization
2.   **NoC-based configuration (all groups):** Any tile can write to configuration registers via NoC

## NoC Integration and Transaction Handling

DRAM tiles are **passive responders** in the NoC fabric—they can only respond to requests initiated by other tiles. They cannot initiate NoC transactions themselves.

**Transaction types supported:**

| Transaction Type | Supported | Notes |
| --- | --- | --- |
| Read | Yes | Reads data from GDDR6, returns to initiator |
| Write (Posted) | Yes | Writes data to GDDR6, no acknowledgement |
| Write (Non-Posted) | Yes | Writes data to GDDR6, sends acknowledgement |
| Atomic | Yes | Atomic operations on GDDR6 data |
| Initiate | No | DRAM tiles cannot initiate transactions |

**Key behavioral characteristics:**

1.   **Receive-Only Operation:**

    *   DRAM tile NIUs have no `NOC_CMD_CTRL` write capability
    *   Cannot configure `NOC_TARG_ADDR`, `NOC_RET_ADDR`, or other initiator registers
    *   Exclusively respond to incoming packets

2.   **Multiple Access Paths:**

    *   All 6 NIUs (across 3 tiles in a group) access the same 2 GiB
    *   Load balancing across NIUs improves throughput
    *   Software can distribute traffic across NoC #0 and NoC #1

3.   **Address Space:**

    *   Each DRAM group presents a 2 GiB address space (0x0000_0000 to 0x7FFF_FFFF)
    *   Same physical address accessible from any of the 3 tiles in the group
    *   See [Alignment Requirements](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Alignment%20Requirements)

**Sources:**[WormholeB0/NoC/README.md 15-16](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L15-L16)[WormholeB0/NoC/MemoryMap.md 1-269](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L1-L269)

## Tile Identification and Addressing

DRAM tiles are identified through their `NOC_ENDPOINT_ID` register, which software can read to determine tile type, group, and index within the group.

### NOC_ENDPOINT_ID Register Layout

The `NOC_ENDPOINT_ID` register (at `NIU_BASE + 0x0030`) provides tile identification:

| Bits | Field Name | DRAM Tile Values |
| --- | --- | --- |
| 0:7 | Tile index | `0` to `2` (position within group) |
| 8:15 | Group index | `0` to `5` (D0 through D5) |
| 16:23 | Tile type | `8` (DRAM tile) |
| 24:31 | NoC index | `0` (NoC #0) or `1` (NoC #1) |

### NoC Coordinate Space

DRAM tiles occupy specific coordinates in the NoC 2D torus:

**Addressing rules:**

1.   **Group Selection:** Choose which 2 GiB memory bank to access (D0-D5)
2.   **Tile Selection:** Choose one of 3 tiles in the group (any tile provides same data)
3.   **NoC Selection:** Choose NoC #0 or NoC #1 based on routing and congestion
4.   **Memory Address:** Specify address within 2 GiB range (31-bit address)

## Memory Access and Alignment Requirements

DRAM tiles enforce strict alignment requirements due to GDDR6 protocol constraints and the AXI crossbar architecture.

### Address Alignment Rules

DRAM addresses are treated as "other address" type in NoC alignment rules (not L1, not MMIO):

**Length-Based Transactions (NOC_CMD_WR_BE=false)**

| Data Flow | Alignment Requirement | Notes |
| --- | --- | --- |
| L1 → DRAM | `SrcAddr % 32 == DstAddr % 32` (C32) | Both source and dest must be congruent mod 32 |
| DRAM → L1 | `SrcAddr % 16 == DstAddr % 16` (C16) | Congruent mod 16 sufficient |
| DRAM → DRAM | `SrcAddr % 32 == DstAddr % 32` (C32) | Between different DRAM groups |

**Byte-Enable Transactions (NOC_CMD_WR_BE=true)**

| Data Flow | Alignment Requirement | Notes |
| --- | --- | --- |
| L1 → DRAM | Dest 32-byte aligned (DA32), Src 32-byte aligned (SA32) | Up to 32 bytes per transaction |
| DRAM → L1 | Dest 32-byte aligned (DA32), Src 32-byte aligned (SA32) | Rarely used for DRAM |

**Performance considerations:**

1.   **32-byte alignment (minimum):** Required for correctness
2.   **128-byte alignment (recommended):** Matches GDDR6 burst size for optimal throughput
3.   **Channel distribution:** Alternate between addresses in Channel 0 (bit 30 = 0) and Channel 1 (bit 30 = 1) for parallel access

### Automatic Transaction Splitting

NoC packets have a maximum payload of 8192 bytes (256 flits × 32 bytes/flit). Larger transfers automatically split into multiple packets.

**Transaction Splitting Diagram**

**Splitting mechanism:**

1.   **Initial counter update:**`NIU_MST_REQS_OUTSTANDING_ID` and `NIU_MST_WRITE_REQS_OUTGOING_ID` are incremented by `ceil(length / 8192)` immediately
2.   **Packet generation:** Hardware generates packets sequentially: 
    *   Packet 1: addresses as specified, length = min(8192, remaining)
    *   Packet 2: both addresses += 8192, length = min(8192, remaining)
    *   Continue until all data sent

3.   **NOC_CMD_CTRL clearing:** Remains `1` until final packet is generated
4.   **Counter updates:** Counters decrement as each response arrives

**Alignment requirement for splitting:** Both `NOC_TARG_ADDR_LO` and `NOC_RET_ADDR_LO` must be 32-byte aligned before splitting begins. Hardware adds 8192 (which is 32-byte aligned) to both addresses.

**Maximum split size:** Counters are 8-bit, so maximum outstanding requests = 255. With 8192 bytes per packet:

*   Maximum split transaction size ≈ 2 MiB (255 × 8192 bytes)

**Sources:**[WormholeB0/NoC/Counters.md 167-174](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L167-L174)[WormholeB0/NoC/MemoryMap.md 73-92](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L73-L92)

## Configuration and Control Paths

DRAM tiles have limited configuration interfaces compared to Tensix tiles, primarily for memory controller setup.

### Configuration Register Access

**Configuration paths:**

1.   **Direct ARC Connection (D0 group only):**

    *   ARC tile has direct APB/AXI path to D0 DRAM tiles
    *   Used during boot and initialization
    *   Bypasses NoC for early memory setup

2.   **NoC-based Configuration (all groups):**

    *   Standard NoC write transactions to configuration registers
    *   Accessible from any tile with NoC access
    *   Used for runtime reconfiguration

### NIU Register Map

DRAM tile NIUs contain standard NIU registers but with **slave-only** functionality (no `NOC_CMD_CTRL` or initiator registers):

**Identification Registers**

| Register | Address | Access | Purpose |
| --- | --- | --- | --- |
| `NOC_NODE_ID` | `NIU_BASE + 0x002C` | Read | NoC X/Y coordinates, torus dimensions |
| `NOC_ENDPOINT_ID` | `NIU_BASE + 0x0030` | Read | Tile type (`8`), group index (`0`-`5`), tile index (`0`-`2`) |

**Configuration Registers**

| Register | Address | Bits | Access | Purpose |
| --- | --- | --- | --- | --- |
| `NIU_CFG_0` | `NIU_BASE + 0x0100` | 14 | R/W | Coordinate translation enable |
| `NIU_CFG_0` | `NIU_BASE + 0x0100` | 12 | R/W | Tile clock disable (for fused-off tiles) |
| `ROUTER_CFG_1` | `NIU_BASE + 0x0108` | 0:9 | R/W | Broadcast opt-out mask (columns) |
| `ROUTER_CFG_3` | `NIU_BASE + 0x0110` | 0:11 | R/W | Broadcast opt-out mask (rows) |
| `NOC_X_ID_TRANSLATE_TABLE_0` | `NIU_BASE + 0x0118` | - | R/W | X coordinate translation table (128 bits) |
| `NOC_Y_ID_TRANSLATE_TABLE_0` | `NIU_BASE + 0x0128` | - | R/W | Y coordinate translation table (128 bits) |

**Counter Registers (Slave-Side Only)**

Starting at `NIU_BASE + 0x0200`, DRAM tiles provide slave-side counters only:

| Counter | Index | Description |
| --- | --- | --- |
| `NIU_SLV_REQ_ACCEPTED` | 52 | Total requests accepted by this NIU |
| `NIU_SLV_RD_REQ_RECEIVED` | 53 | Read requests received |
| `NIU_SLV_NONPOSTED_ATOMIC_RECEIVED` | 54 | Atomic requests (non-posted) received |
| `NIU_SLV_POSTED_ATOMIC_RECEIVED` | 55 | Atomic requests (posted) received |
| `NIU_SLV_NONPOSTED_WR_REQ_RECEIVED` | 58 | Write requests (non-posted) received |
| `NIU_SLV_POSTED_WR_REQ_RECEIVED` | 59 | Write requests (posted) received |
| `NIU_SLV_RD_RESP_SENT` | 50 | Read responses sent back to initiator |
| `NIU_SLV_WR_ACK_SENT` | 49 | Write acknowledgements sent |
| `NIU_SLV_ATOMIC_RESP_SENT` | 48 | Atomic responses sent |

**Note:** Master-side counters (`NIU_MST_*`) always read as zero because DRAM tiles cannot initiate transactions.

## Memory Bandwidth and Performance Characteristics

DRAM tiles provide high-bandwidth access to GDDR6 memory, with performance influenced by access patterns and NoC routing.

### Bandwidth Considerations

**Theoretical maximum per channel:**

*   GDDR6 interface width: varies by implementation
*   Typical per-channel bandwidth: ~20-30 GB/s per direction
*   6 independent channels provide aggregate bandwidth

**Practical considerations:**

1.   **Multiple NIU Access:**

    *   6 NIUs per 2 GiB bank (across 3 tiles × 2 NoCs)
    *   Distribute traffic across NIUs for maximum throughput
    *   Load balancing reduces hotspots

2.   **NoC Congestion:**

    *   DRAM tiles at edges of NoC torus
    *   Cross-chip traffic must traverse many hops
    *   Use both NoC #0 and NoC #1 to reduce congestion

3.   **Alignment Impact:**

    *   Misaligned accesses cause undefined behavior
    *   128-byte alignment recommended for best performance
    *   Sequential access patterns optimal for GDDR6

### Access Pattern Recommendations

**Best practices:**

1.   **Use large sequential transfers** (prefer 8 KiB+ per transaction)
2.   **Balance load across DRAM groups** when possible
3.   **Alternate between NoC #0 and NoC #1** to avoid congestion
4.   **Align to 128-byte boundaries** for optimal controller performance
5.   **Monitor `NIU_SLV_*` counters** to identify bottlenecks

## Clock Domains and Timing

DRAM tiles operate across multiple clock domains with appropriate clock domain crossing logic.

**Clock domains:**

1.   **AI Clock (NoC):** Standard chip clock for NoC fabric
2.   **AXI Clock:** Intermediate clock for AXI crossbar and NIU interfaces
3.   **GDDR Clock:** Memory controller and PHY clock (synchronous to GDDR6)

**CDC (Clock Domain Crossing) Logic:**

*   FIFOs and synchronizers between clock domains
*   Latency added by CDC logic (~5-10 cycles per crossing)
*   All crossings handled transparently by hardware

**Sources:**[Diagrams/Src/EdgeTile.lua 438-513](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L438-L513)

## Initialization and Training

DRAM tiles require initialization before use, typically performed by firmware running on the ARC tile.

### Initialization Sequence

**Initialization steps:**

1.   **Controller Configuration:** Set GDDR6 timing parameters (CAS latency, refresh rate, etc.)
2.   **PHY Calibration:** ZQ calibration, write leveling, read training
3.   **ECC Configuration:** Enable error correction (if supported)
4.   **Scrubber Activation:** Start background ECC scrubber
5.   **Temperature Monitoring:** Configure thermal throttling

**Typical initialization time:** 10-100 milliseconds per channel

**Sources:**[Diagrams/Src/EdgeTile.lua 438-513](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L438-L513)

## Summary

DRAM tiles provide the ASIC's primary external memory capacity through the following architecture:

*   **18 tiles** organized into **6 groups**, each exposing **2 GiB** of GDDR6 (12 GiB total)
*   **3 redundant tiles per group** for load balancing and fault tolerance
*   **6 NIUs per group** (across 3 tiles × 2 NoCs) for high aggregate bandwidth
*   **Passive responders** only—cannot initiate NoC transactions
*   **Strict alignment requirements** (32-byte congruence, 128-byte recommended)
*   **Direct ARC connection** for D0 group initialization
*   **NoC-accessible configuration** for all groups during operation
