## Purpose and Scope

This document describes the `setup-and-test.yml` reusable GitHub Actions workflow, which implements the two-phase test execution strategy that separates compilation from hardware execution. This workflow is the core testing infrastructure used by [on-pr.yml workflow](https://deepwiki.com/tenstorrent/tt-llk/10.2-on-pr.yml-workflow-and-change-detection) to run tests on different architectures.

For information about the environment setup script that this workflow depends on, see [Environment Setup and SFPI Toolchain](https://deepwiki.com/tenstorrent/tt-llk/9.2-environment-setup-and-sfpi-toolchain). For details about the overall CI/CD pipeline architecture, see [GitHub Actions Pipeline Overview](https://deepwiki.com/tenstorrent/tt-llk/10.1-github-actions-pipeline-overview).

## Two-Phase Architecture Overview

The workflow implements a **PRODUCE/CONSUME** pattern that separates test compilation from test execution:

**Phase 1 (PRODUCE)**: Compile all test variants into ELF binaries without executing them on hardware. This phase uses the `--compile-producer` flag and runs with 10 parallel workers to maximize compilation throughput.

**Phase 2 (CONSUME)**: Execute pre-compiled ELF binaries on actual hardware without re-compiling. This phase uses the `--compile-consumer` flag and runs serially with the `-x` (fail-fast) option to catch hardware issues quickly.

This separation provides several benefits:

*   **Isolation**: Compilation failures don't block other tests from compiling
*   **Parallelism**: Compilation can use many workers; execution runs serially for deterministic hardware access
*   **Efficiency**: Pre-compiled artifacts can be reused if only test parameters change
*   **Diagnostics**: Separate JUnit reports for compilation vs execution failures

## Workflow Structure

The `setup-and-test.yml` workflow consists of three jobs:

| Job | Runs On | Purpose |
| --- | --- | --- |
| `generate-matrix` | ubuntu-latest | Generate test group numbers [1..N] for parallel execution |
| `setup-and-test` | Hardware runners | Compile and execute tests for one group |
| `merge-coverage` | ubuntu-latest | Combine coverage data from all groups |

## Phase 1: Compilation (PRODUCE)

The compilation phase runs with `pytest --compile-producer` and these characteristics:

| Parameter | Value | Rationale |
| --- | --- | --- |
| `-n 10` | 10 parallel workers | Maximize CPU utilization for compilation |
| `--timeout=60` | 60 second timeout | Detect hanging compilation |
| No `-x` flag | Continue on failure | Compile all variants to discover all compilation errors |

### Variant Hash Calculation

Each test variant is identified by a SHA256 hash of its compilation parameters. This ensures that different format combinations, template parameters, and compilation flags produce separate ELF binaries:

**Parameters included in hash:**

*   Test name
*   Template parameters (e.g., `APPROX_MODE`, `MATH_OP`)
*   Formats configuration (if `compile_time_formats=True`)
*   Boot mode
*   Profiler build flag
*   Coverage build flag
*   L1-to-L1 iterations
*   Destination accumulation mode
*   Unpack-to-dest flag

**Parameters excluded from hash:**

*   Runtime parameters (these vary per execution)
*   Stimuli configuration (input/output data)
*   Format configuration (if `compile_time_formats=False`)

## Phase 2: Execution (CONSUME)

The execution phase runs with `pytest --compile-consumer` and these characteristics:

| Parameter | Value | Rationale |
| --- | --- | --- |
| `-x` | Fail fast | Stop on first hardware failure for quick feedback |
| No `-n` flag | Serial execution | One test at a time for deterministic hardware access |
| `--timeout=60` | 60 second timeout | Detect hanging hardware operations |

### Runtime Parameter Layout

Runtime parameters are written to L1 memory at a fixed address. The layout is generated by `generate_runtime_args_struct()`:

The address differs based on coverage mode:

*   **Non-coverage**: `0x20000` (128 KB offset)
*   **Coverage**: `0x6E000` (440 KB offset, avoids coverage instrumentation data)
