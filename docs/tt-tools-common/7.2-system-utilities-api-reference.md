## Purpose and Scope

This page provides complete API documentation for the system utilities modules in tt-tools-common. These modules provide foundational functionality for driver management, host information collection, hardware detection, and version conversions used across all Tenstorrent tools.

The system utilities are organized into two modules:

*   **`system_utils`**: Driver version checking, host information gathering, and software version reporting
*   **`tools_utils`**: Chip detection, firmware data loading, version conversions, and reference clock monitoring

For documentation of the hardware reset system that uses these utilities, see [Reset API Reference](https://deepwiki.com/tenstorrent/tt-tools-common/7.1-reset-api-reference). For UI components that display system information, see [UI Components API Reference](https://deepwiki.com/tenstorrent/tt-tools-common/7.3-ui-components-api-reference).

## system_utils Module API

The `system_utils` module provides system-level utilities for driver management, host information collection, and compatibility checking.

### Driver Version Management

#### `get_driver_version() -> Union[str, None]`

Returns the version string of the currently installed Tenstorrent kernel driver.

**Returns:**

*   `str`: Version string (e.g., "1.26.0", "2.4.1-rc1")
*   `None`: If no driver is detected

**Implementation Details:**

*   Reads from `/sys/module/tenstorrent/version`
*   Returns `None` on any exception (file not found, permission denied, etc.)

**Example:**

`from tt_tools_common.utils_common.system_utils import get_driver_version version = get_driver_version()if version:    print(f"Driver version: {version}")else:    print("No Tenstorrent driver detected")`
**Sources:**[tt_tools_common/utils_common/system_utils.py 35-45](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/system_utils.py#L35-L45)

#### `is_driver_version_at_least(current_version: str, minimum_version: str) -> bool`

Compares two semantic version strings to determine if the current version meets or exceeds the minimum required version.

**Parameters:**

*   `current_version` (str): Current driver version to check
*   `minimum_version` (str): Minimum required version

**Returns:**

*   `bool`: `True` if current version >= minimum version, `False` otherwise

**Raises:**

*   `ValueError`: If `current_version` is `None` or if version strings are malformed

**Supported Version Formats:**

*   Standard SemVer: `"1.34.0"`
*   Short version: `"1.34"` (patch defaults to 0)
*   Pre-release: `"1.34.0-alpha"` (pre-release suffix stripped)
*   Build metadata: `"1.2.3+build456"` (build suffix stripped)
*   Combined: `"1.4.0-rc1+build42"`

**Version Comparison Logic:** Version tuples are compared lexicographically: `(major, minor, patch)`

*   `(1, 34, 0) >= (1, 26, 0)` → `True`
*   `(1, 26, 0) >= (1, 26, 0)` → `True`
*   `(2, 0, 0) >= (1, 35, 0)` → `True`
*   `(1, 25, 0) >= (1, 26, 0)` → `False`

**Sources:**[tt_tools_common/utils_common/system_utils.py 47-97](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/system_utils.py#L47-L97)

#### `check_driver_version(operation: str, minimum_required_version_str: str = "1.26.0")`

Validates that the installed driver version is sufficient for the specified operation. Exits the program with status code 1 if validation fails.

**Parameters:**

*   `operation` (str): Description of the operation requiring the driver (e.g., "chip reset", "firmware update")
*   `minimum_required_version_str` (str, optional): Minimum required version. Defaults to `"1.26.0"` (`MINIMUM_DRIVER_VERSION_LDS_RESET`)

**Exit Conditions:** The function calls `sys.exit(1)` if:

*   No driver is detected (`current_version` is `None`)
*   Version strings are malformed
*   Current driver version < minimum required version

**Output:** Prints colored error messages to stdout indicating:

*   Current driver version
*   Required minimum version for the operation
*   Instructions to update driver via tt-kmd

**Example:**

`from tt_tools_common.utils_common.system_utils import check_driver_version # Will exit if driver version < 1.26.0check_driver_version("chip reset") # Check for custom minimum versioncheck_driver_version("advanced feature", "2.4.1")`

### Software Version Reporting

#### `get_sw_ver_info(show_sw_ver: bool, board_ids: str) -> dict`

Retrieves software version information for Tenstorrent tools and frameworks, optionally querying a remote server based on board serial numbers.

**Parameters:**

*   `show_sw_ver` (bool): Whether to fetch version information. If `False`, returns default "N/A" values
*   `board_ids` (str): List of board serial numbers to query

**Returns:** Dictionary with keys for various software components:

| Key | Description |
| --- | --- |
| `"Firmware Bundle"` | Firmware bundle version |
| `"tt-smi"` | tt-smi tool version |
| `"tt-flash"` | tt-flash tool version |
| `"tt-kmd"` | Kernel driver version |
| `"TT-Buda"` | TT-Buda framework version |
| `"TT-Metalium"` | TT-Metalium framework version |

**Behavior:**

1.   Checks for `~/.config/tenstorrent/reset_config.json` for reporting flags
2.   If `disable_sw_version_report` flag is set, returns default values
3.   If `disable_serial_report` flag is set, queries server with empty board ID
4.   Otherwise, queries server with provided board IDs

**Remote Query:**

*   Endpoint: `https://cereal.tenstorrent.com?SerialNumber={board_id}`
*   Handles HTTP errors, connection errors, timeouts gracefully
*   Returns error messages in `"Failed to fetch"` key on failure

### Board Type Identification

#### `get_board_type(board_id: str) -> str`

Extracts board type from a board ID serial number string.

**Parameters:**

*   `board_id` (str): Board ID in hex format (e.g., `"01000012345678"`)

**Returns:**

*   `str`: Board type identifier or `"N/A"` if unknown

**Board ID Format:**

```
AA-BBBBB-C-D-EE-FF-XXX
│   │    │ │  │  │  └─ XXX: Additional info
│   │    │ │  │  └──── FF: Manufacturer
│   │    │ │  └─────── EE: Region
│   │    │ └────────── D: Reserved
│   │    └──────────── C: Revision
│   └───────────────── BBBBB: Unique Part Identifier (UPI)
└────────────────────── AA: Reserved
```

**UPI to Board Type Mapping:**

| UPI (hex) | Revision | Board Type |
| --- | --- | --- |
| 0x1 | 0x2 | `"E300_R2"` |
| 0x1 | 0x3, 0x4 | `"E300_R3"` |
| 0x3 | Any | `"e150"` |
| 0x7 | Any | `"e75"` |
| 0x8 | Any | `"NEBULA_CB"` |
| 0xA | Any | `"e300"` |
| 0xB | Any | `"GALAXY"` |
| 0x14 | Any | `"n300"` |
| 0x18 | Any | `"n150"` |
| Other | Any | `"N/A"` |

**Example:**

`from tt_tools_common.utils_common.tools_utils import get_board_type board_type = get_board_type("0A00001234567890")print(f"Board type: {board_type}")  # "e300"`

### Reference Clock Monitoring

These functions monitor the reference clock counter hardware register to validate clock operation.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 198-253](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L198-L253)

#### `read_refclk_counter(chip) -> int`

Reads the 64-bit reference clock counter from hardware registers.

**Parameters:**

*   `chip`: PciChip object with AXI access capabilities

**Returns:**

*   `int`: 64-bit counter value
*   `None`: For Grayskull chips (not supported)

**Implementation:**

1.   Reads `ARC_RESET.REFCLK_COUNTER_HIGH` (first read)
2.   Reads `ARC_RESET.REFCLK_COUNTER_LOW`
3.   Reads `ARC_RESET.REFCLK_COUNTER_HIGH` (second read)
4.   If high values differ, re-reads low register to handle overflow
5.   Returns `(high2 << 32) | low`

**Overflow Handling:** The double-read of the high register detects overflow that occurs between reading low and high registers, ensuring atomic 64-bit reads.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 202-214](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L202-L214)

* * *

#### `refclk_counter_rate(chip, delay_interval: float = 0.1) -> float`

Calculates the reference clock counter rate by measuring counter increments over time.

**Parameters:**

*   `chip`: PciChip object
*   `delay_interval` (float, optional): Measurement interval in seconds. Defaults to `0.1`

**Returns:**

*   `float`: Counter rate in MHz

**Calculation:**

```
rate_MHz = (counter_after - counter_before) * 1000 / (time_ns_after - time_ns_before)
```

**Accuracy:** Longer delay intervals provide more accurate measurements but take more time. Default 0.1s provides good balance.

**Example:**

`from tt_tools_common.utils_common.tools_utils import refclk_counter_rate rate = refclk_counter_rate(chip, delay_interval=0.5)print(f"Reference clock rate: {rate:.2f} MHz")`
**Sources:**[tt_tools_common/utils_common/tools_utils.py 217-227](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L217-L227)

#### `check_refclk_counter_rate(chip, expected_refclk: float, accuracy: float)`

Validates that the reference clock counter is incrementing at the expected frequency.

**Parameters:**

*   `chip`: PciChip object
*   `expected_refclk` (float): Expected clock rate in MHz
*   `accuracy` (float): Allowed deviation percentage (e.g., 1.0 for ±1%)

**Returns:**

*   `str`: Error message if rate is outside tolerance
*   `None`: If rate is within tolerance

**Validation:** Checks if: `|expected - observed| / min(expected, observed) <= accuracy / 100`

**Example:**

### Chip Detection

#### `detect_chips_with_callback(local_only: bool = False, ignore_ethernet: bool = False, print_status: bool = True) -> List[PciChip]`

Detects all Tenstorrent chips in the system with real-time status updates via callback mechanism.

**Parameters:**

*   `local_only` (bool, optional): If `True`, only detect local PCIe chips. Defaults to `False`
*   `ignore_ethernet` (bool, optional): If `True`, skip Ethernet neighbor detection. Defaults to `False`
*   `print_status` (bool, optional): If `True`, print detection progress. Defaults to `True`

**Returns:**

*   `List[PciChip]`: List of detected chips with verified communication

**Raises:**

*   `Exception`: If any detected chip fails communication verification

**Detection Process:**

**Callback Events:** The internal `chip_detect_callback` function handles these event types:

1.   **`new_chip()`**: New chip detected, increment counter
2.   **`correct_down()`**: Previously detected chip removed, decrement counter
3.   **`status_string()`**: Provides detailed detection progress messages

**Status Display:** When `print_status=True`, shows:

*   **Chip count**: Total detected chips (purple text)
*   **ARC detection**: Rotating spinner (blue text)
*   **DRAM detection**: Rotating spinner (blue text)
*   **Ethernet detection**: Per-chip progress (purple/blue text)

**Communication Verification:** After detection, each chip is checked with `have_comms()`. If communication fails, raises:

```
Exception: Do not have communication with {device}, you should reset or remove this device from your system before continuing.
```

**Firmware Upgrade:** Successfully detected chips are upgraded via `force_upgrade()` to ensure latest firmware capabilities.

**Example:**


## Version String Parsing Details

### Internal Version Parser

The `_parse_version_string()` function implements robust semantic version parsing used by driver version checking.

**Syntax Support:**

| Format | Example | Parsed As |
| --- | --- | --- |
| Major.Minor.Patch | `"1.26.0"` | `(1, 26, 0)` |
| Major.Minor | `"1.26"` | `(1, 26, 0)` |
| Major only | `"2"` | `(2, 0, 0)` |
| Pre-release | `"1.26.0-alpha"` | `(1, 26, 0)` |
| Build metadata | `"1.26.0+build123"` | `(1, 26, 0)` |
| Combined | `"1.26.0-rc1+build42"` | `(1, 26, 0)` |

**Parsing Algorithm:**

1.   Strip build metadata (text after `+`)
2.   Strip pre-release identifier (text after `-`)
3.   Split remaining string on `.`
4.   Parse as integers with defaults: `(major, minor or 0, patch or 0)`

**Error Cases:**

*   Empty string → `ValueError`
*   Non-numeric parts → `ValueError`
*   Invalid format → `ValueError`
