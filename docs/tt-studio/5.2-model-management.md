This document covers the backend systems for model deployment, lifecycle management, container orchestration, and hardware allocation in TT-Studio. It details the Django-based model control logic, Docker orchestration via the Docker Control Service, and the health monitoring systems that manage Tenstorrent-accelerated AI models.

## System Architecture

The model management system bridges high-level model abstractions with low-level Docker and hardware operations. It uses a combination of Django for state management and a FastAPI-based Docker Control Service for secure container orchestration.

## Deployment Orchestration

Model deployment is handled differently depending on the model type. The system distinguishes between models managed by the `TT Inference Server` (standard LLMs/VLLM) and "direct" containers (e.g., Face Recognition, Object Detection).

### Deployment Logic

1.   **TT Inference Server Path**: For supported LLMs, `start_chat_deployment` in `tt_inference_client.py` sends a request to the FastAPI-based inference server [app/backend/docker_control/tt_inference_client.py 40-100](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/tt_inference_client.py#L40-L100)
2.   **Direct Container Path**: For standalone models, `_run_direct_container` in `docker_utils.py` configures mounts, ports, and environment variables before calling the Docker Control Service [app/backend/docker_control/docker_utils.py 157-210](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L157-L210)

### Hardware & Chip Allocation

The system maps board types to specific device names for the inference server using `map_board_type_to_device_name`[app/backend/docker_control/docker_utils.py 122-155](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L122-L155) It also handles multi-chip boards by mapping them to their constituent single-chip equivalents (e.g., T3K to N300) via the `_BOARD_TO_SINGLE_CHIP_DEVICE` mapping [app/backend/docker_control/docker_utils.py 97-119](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L97-L119)

## Lifecycle Operations

### Container Management

The system provides a wrapper around Docker operations to ensure Tenstorrent hardware state remains consistent.

*   **Stop & Reset**: When a container is stopped via `StopView`, the system first terminates the Docker container and then performs a hardware reset using `perform_reset()` to clear the Tenstorrent chip state [app/backend/docker_control/views.py 75-148](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/views.py#L75-L148)
*   **Deployment Cache**: To avoid frequent expensive Docker API calls, the system maintains a `deploy_cache` using Django's cache framework [app/backend/model_control/model_utils.py 107-141](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L107-L141) This cache is refreshed by `update_deploy_cache()`[app/backend/docker_control/docker_utils.py 20-21](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L20-L21)

### Health Monitoring

The `health_check` function in `model_utils.py` performs periodic checks on deployed models [app/backend/model_control/model_utils.py 144-173](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L144-L173) It handles several states:

*   **200 OK**: Healthy.
*   **503 Service Unavailable**: Often indicates the model is still loading (common for large LLMs or media-server models) [app/backend/model_control/model_utils.py 162-170](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L162-L170)
*   **Connection Refused**: The container is starting, but the internal server is not yet listening [app/backend/model_control/model_utils.py 149-152](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L149-L152)

## Performance Tracking

The `InferenceMetricsTracker` class provides standardized measurement of inference performance, following the vLLM measurement approach [app/backend/model_control/metrics_tracker.py 22-39](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/metrics_tracker.py#L22-L39)

### Key Metrics Tracked

*   **TTFT (Time to First Token)**: Calculated as the interval between the request start and the first content token [app/backend/model_control/metrics_tracker.py 101-107](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/metrics_tracker.py#L101-L107)
*   **ITL (Inter-Token Latency)**: A list of intervals between consecutive tokens [app/backend/model_control/metrics_tracker.py 30-31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/metrics_tracker.py#L30-L31)
*   **TPOT (Time Per Output Token)**: Calculated as the mean of the ITL [app/backend/model_control/metrics_tracker.py 109-118](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/metrics_tracker.py#L109-L118)
*   **Thinking Time**: For reasoning models, the tracker separates `thinking_duration` from content generation [app/backend/model_control/metrics_tracker.py 132-136](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/metrics_tracker.py#L132-L136)

## Backend Utilities

### Model Metadata Enrichment

The system lazily enriches deployment information by querying running containers. The `get_deploy_cache` function automatically fetches `max_model_len` and the actual `cached_model_name` from the vLLM `/v1/models` endpoint when a container is detected [app/backend/model_control/model_utils.py 123-141](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L123-L141)

### Model Compatibility

The system uses `model_compatibility_overrides.json` to manually force compatibility for specific models on hardware where auto-detection might fail or is not yet implemented [app/backend/docker_control/views.py 62-70](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/views.py#L62-L70)
