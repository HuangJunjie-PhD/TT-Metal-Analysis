## Purpose and Scope

This document describes the hardware platforms supported by tt-system-firmware, including Blackhole, Wormhole, and Grayskull ASIC variants. Each platform has specific board configurations with corresponding firmware parameter sets, test infrastructure, and CI/CD support.

For information about firmware versioning across platforms, see [Versioning and Release Strategy](https://deepwiki.com/tenstorrent/tt-system-firmware/1.2-versioning-and-release-strategy). For details on the firmware applications that run on these platforms, see [Firmware Applications](https://deepwiki.com/tenstorrent/tt-system-firmware/2-firmware-applications).

## CI/CD Board Configuration Matrix

The CI/CD infrastructure uses a centralized board configuration file to define test matrices across all hardware platforms.

### Board Configuration File Structure

The file `.github/ci_boards.json` defines the complete test matrix:

**Sources:**[.github/ci_boards.json 1-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L1-L46)[.github/workflows/hardware-smoke.yml 29-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml#L29-L46)

### Board Filtering in Workflows

CI workflows dynamically filter boards based on product type:

`# Blackhole boards filter (excludes galaxy from smoke tests)MATRIX_CONFIG_BH="$(jq -c '[.[] | select((.product == "bh") and (.board != "galaxy")) | del(.product)]' .github/ci_boards.json)" # Wormhole boards filterMATRIX_CONFIG_WH="$(jq -c '[.[] | select(.product == "wh") | del(.product)]' .github/ci_boards.json)"`
**Sources:**[.github/workflows/hardware-smoke.yml 39-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml#L39-L46)

**Note:** The Galaxy board is excluded from smoke tests due to lack of recovery solution, but is included in long-duration and Metal integration tests.

## Binary Blob Verification

All hardware-specific firmware blobs are verified during CI setup using SHA256 checksums:

`# verify blobs (for tt-system-firmware only - we do not depend on other blobs)west blobs fetch tt-system-firmware`
**Sources:**[.github/workflows/prepare-container/action.yml 107-111](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/prepare-container/action.yml#L107-L111)

The blob verification ensures that the correct firmware variants are loaded for each board configuration. The SHA256 hashes are declared in the Zephyr module manifest and automatically verified during the build process.

**Sources:**[zephyr/module.yml 36-235](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L36-L235)

## Summary

The tt-system-firmware project supports:

*   **12 Blackhole board variants** with full source-based firmware and comprehensive CI/CD testing
*   **2 Wormhole board variants** with pre-compiled firmware and E2E smoke tests
*   **1 Grayskull platform** with pre-compiled firmware (legacy support)

Each platform variant has specific GDDR memory parameters, runner configurations, and test requirements defined in centralized configuration files that drive the CI/CD matrix execution.

For details on how these platforms are tested, see [Testing and Validation](https://deepwiki.com/tenstorrent/tt-system-firmware/7-testing-and-validation). For information on building firmware for specific platforms, see [Build System](https://deepwiki.com/tenstorrent/tt-system-firmware/6-build-system).
