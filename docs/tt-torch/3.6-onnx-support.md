This page documents tt-torch's support for compiling and executing ONNX models on Tenstorrent hardware. ONNX (Open Neural Network Exchange) provides an alternative frontend to PyTorch for model ingestion, allowing users to compile pre-exported ONNX models without requiring the original PyTorch model definition.

For information about the main PyTorch compilation pipeline, see [Compilation Pipeline](https://deepwiki.com/tenstorrent/tt-torch/3.1-compilation-pipeline). For the general executor framework, see [Executor Framework](https://deepwiki.com/tenstorrent/tt-torch/3.3-executor-framework).

## Overview

tt-torch provides a complete ONNX compilation path that converges with the PyTorch path at the StableHLO IR stage. The ONNX frontend accepts `onnx.ModelProto` objects and transforms them through multiple MLIR dialects before producing executable flatbuffer binaries. This enables users to:

*   Compile ONNX models exported from various frameworks (PyTorch, TensorFlow, etc.)
*   Leverage existing ONNX model zoos and ecosystems
*   Execute models in op-by-op or full execution modes
*   Verify ONNX model outputs against CPU golden references

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 1-122](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L1-L122)

## Compilation Pipeline

The ONNX compilation pipeline transforms models through the following stages:

**Diagram: ONNX Compilation Pipeline with Code Entities**

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 35-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L35-L121)[tt_torch/dynamo/backend.py 228-270](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L228-L270)

### Pipeline Stages

| Stage | Function | Input | Output | Description |
| --- | --- | --- | --- | --- |
| ONNX Import | `generate_torch_onnx_ir()` | `onnx.ModelProto` | Torch ONNX IR | Uses `torch_mlir.onnx_importer` to convert ONNX protobuf to MLIR |
| Backend Lowering | `torch_onnx_to_torch_backend_ir()` | Torch ONNX IR | Torch Backend IR | Applies `torch-onnx-to-torch-backend-pipeline` pass |
| StableHLO Conversion | `torch_backend_ir_to_stablehlo()` | Torch Backend IR | StableHLO | Lowers to StableHLO using `lower_mlir_module()` |
| Flatbuffer Generation | `shlo_to_flatbuffer()` | StableHLO | Binary | Compiles through TTIR/TTNN to executable binary |

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 35-67](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L35-L67)[tt_torch/dynamo/backend.py 228-270](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L228-L270)

### Entry Point Function

The main entry point is `compile_onnx()`, which orchestrates the entire pipeline:

`compile_onnx(model_proto: onnx.ModelProto, compiler_config: CompilerConfig = None)`
This function [tt_torch/onnx_compile/onnx_compile.py 71-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L71-L121):

1.   Applies shape inference to the ONNX model
2.   Preprocesses the model using `onnxruntime.quantization.shape_inference.quant_pre_process()`
3.   Routes to op-by-op execution mode if configured
4.   Otherwise, compiles through the full pipeline to produce an `OnnxExecutor`

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 71-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L71-L121)

### Preprocessing

Before compilation, ONNX models undergo preprocessing [tt_torch/onnx_compile/onnx_compile.py 82-91](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L82-L91):

1.   **Shape Inference:**`onnx.shape_inference.infer_shapes()` propagates tensor shapes throughout the graph
2.   **Quantization Preprocessing:**`onnxruntime.quantization.shape_inference.quant_pre_process()` optimizes the model and handles quantization-related transformations
3.   **Model Serialization:** Temporary files (`model.onnx`, `processed_model.onnx`) are created during preprocessing

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 82-91](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L82-L91)

## ONNX Executor

The `OnnxExecutor` class manages execution of compiled ONNX models. The executor handles:

*   Binary storage and device communication
*   Input/output format conversion between PyTorch and ONNX
*   Integration with the runtime for device execution

**Diagram: OnnxExecutor Lifecycle and Runtime Integration**

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 103-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L103-L121)[tt_torch/tools/utils.py 838-895](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L838-L895)

### Execution Modes

The ONNX compilation path supports two primary execution modes, controlled by `CompilerConfig.compile_depth`:

| Mode | CompileDepth | Description | Executor Type |
| --- | --- | --- | --- |
| **Op-by-Op** | `COMPILE_OP_BY_OP` or `EXECUTE_OP_BY_OP` | Compiles to StableHLO and executes operations individually | `StablehloExecutor` |
| **Full Execution** | `EXECUTE` (default) | Compiles to flatbuffer and executes complete model | `OnnxExecutor` |

For op-by-op modes [tt_torch/onnx_compile/onnx_compile.py 92-101](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L92-L101):

*   The pipeline stops at StableHLO generation
*   A `StablehloExecutor` is created with the ONNX model proto attached
*   Operations are executed one at a time for debugging
*   The `OpByOpBackend.STABLEHLO` backend is automatically set

For full execution [tt_torch/onnx_compile/onnx_compile.py 102-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L102-L121):

*   The complete pipeline runs through flatbuffer generation
*   An `OnnxExecutor` receives the compiled binary
*   The model executes as a single unit on device

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 71-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L71-L121)[tt_torch/tools/utils.py 68-71](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L68-L71)

## Input/Output Handling

ONNX models use different tensor formats than PyTorch. tt-torch provides conversion utilities to handle this impedance mismatch:

**Diagram: Input/Output Format Conversion**

**Sources:**[tt_torch/tools/utils.py 838-895](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L838-L895)

### torch_input_to_onnx()

Converts PyTorch inputs to ONNX-compatible format [tt_torch/tools/utils.py 838-870](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L838-L870):

**Supported input formats:**

*   Single `torch.Tensor` → Dict with single entry
*   Dictionary of tensors → Maps tensor names to numpy arrays
*   List/tuple of tensors → Maps positional inputs to named inputs

**Type handling:**

*   Converts `torch.bfloat16` to `torch.float32` before numpy conversion (ONNX doesn't support bfloat16)
*   Detaches tensors from computation graph with `.detach().cpu()`
*   Maps PyTorch tensors to ONNX input names from `InferenceSession`

**Sources:**[tt_torch/tools/utils.py 838-870](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L838-L870)

### onnx_output_to_torch()

Converts ONNX outputs back to PyTorch tensors [tt_torch/tools/utils.py 873-876](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L873-L876):

`def onnx_output_to_torch(output):    for i in range(len(output)):        output[i] = torch.tensor(output[i])    return output`
This simple conversion wraps each numpy array in the output list as a `torch.Tensor`.

**Sources:**[tt_torch/tools/utils.py 873-876](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L873-L876)

### InferenceSession Utilities

Two utility functions support ONNX model execution:

| Function | Purpose | Key Parameters |
| --- | --- | --- |
| `prepare_inference_session()` | Creates ONNXRuntime session | `model_proto`, `intra_op_num_threads`, `inter_op_num_threads` |
| `run_model_proto()` | Executes ONNX model | `model_proto`, `inputs`, `sess`, `input_data_types` |

The `prepare_inference_session()` function [tt_torch/tools/utils.py 879-885](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L879-L885) configures threading for ONNX execution. The `run_model_proto()` function [tt_torch/tools/utils.py 888-895](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L888-L895) handles the complete execution workflow including input conversion, session creation, and output retrieval.

**Sources:**[tt_torch/tools/utils.py 879-895](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L879-L895)

## Configuration Differences

ONNX compilation has specific configuration requirements that differ from PyTorch:

`compiler_config.op_by_op_backend = OpByOpBackend.STABLEHLO  # Always set for ONNXcompiler_config.typecast_inputs = False  # No automatic typecasting`
[tt_torch/onnx_compile/onnx_compile.py 79-80](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L79-L80)

These settings reflect that:

*   ONNX models must use the StableHLO backend for op-by-op execution (no Torch IR backend available)
*   ONNX models handle their own type management through the model proto definition
*   Input typecasting is disabled to preserve ONNX's explicit type semantics

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 79-80](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L79-L80)

## Verification and Testing

### Verification Flow

The verification flow for ONNX models follows a similar pattern to PyTorch models but uses ONNX-specific utilities:

**Diagram: ONNX Model Verification Flow**

The verification process [tt_torch/tools/verify.py 248-310](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L248-L310):

1.   Creates an `InferenceSession` for CPU reference execution
2.   Runs the model on CPU to generate golden outputs
3.   Compiles the model using `compile_onnx()`
4.   Executes the compiled model on device
5.   Compares outputs using PCC (Pearson Correlation Coefficient) and ATOL (Absolute Tolerance)

**Sources:**[tt_torch/tools/verify.py 248-310](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L248-L310)

### verify_module() Integration

The `verify_module()` function supports both PyTorch and ONNX modules [tt_torch/tools/verify.py 313-361](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L313-L361):

`verify_module(    mod,  # Can be nn.Module or onnx.ModelProto    inputs=None,    input_shapes=None,  # Must be None for ONNX (inferred from proto)    compiler_config=CompilerConfig(),    required_pcc=0.99,    required_atol=1e-2)`
When `mod` is an `onnx.ModelProto`, the function automatically:

*   Routes to `_verify_onnx_module()` instead of `_verify_torch_module()`
*   Infers input shapes from the ONNX model's input specifications
*   Handles format conversions for inputs and outputs
*   Verifies that the number of outputs matches between golden and compiled versions

**Sources:**[tt_torch/tools/verify.py 313-361](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L313-L361)

### Output Handling During Verification

ONNX verification requires special handling based on compilation depth [tt_torch/tools/verify.py 289-293](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L289-L293):

`if compiler_config.compile_depth not in [    CompileDepth.EXECUTE,    CompileDepth.EXECUTE_OP_BY_OP,]:    ret = onnx_output_to_torch(ret)`
This ensures that outputs are converted to PyTorch tensors when compilation stops before execution, allowing consistent verification across all compilation depths.

**Sources:**[tt_torch/tools/verify.py 289-293](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L289-L293)

## MLIR Dumping

The ONNX pipeline supports MLIR dumping at each stage via `dump_module()`[tt_torch/dynamo/backend.py 126-150](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L126-L150) controlled by environment variables:

*   `TT_TORCH_IR_LOG_LEVEL=DEBUG` or `INFO` - Enables dumping to stderr
*   `TT_TORCH_SAVE_MLIR=<dialect1>,<dialect2>` - Saves specific dialects to files

Available dialects for ONNX:

*   **Torch Onnx module** - Initial import from ONNX protobuf
*   **Torch Backend module** - After backend lowering
*   **StableHLO module** - After StableHLO conversion
*   **TTIR** - After TTIR compilation
*   **TTNN** - Final TTNN dialect

**Sources:**[tt_torch/onnx_compile/onnx_compile.py 43-66](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/onnx_compile/onnx_compile.py#L43-L66)[tt_torch/dynamo/backend.py 126-150](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L126-L150)
