The `test-gate.yml` workflow serves as the primary CI gate for the `tt-inference-server` repository. It is designed to validate code integrity across the Python orchestration layer and the C++ backend on every pull request to the `main` and `dev` branches [.github/workflows/test-gate.yml 1-7](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L1-L7)

## Workflow Overview

The pipeline uses a tiered execution strategy to optimize resource usage, starting with change detection and linting before proceeding to heavy build and test jobs.

## Core Jobs and Implementation

### 1. Change Detection (`detect-changes`)

This job uses `dorny/paths-filter` to determine which subsystems require testing.

*   **Python Changes**: Triggered by `.py`, `requirements*.txt`, `pyproject.toml`, or workflow changes [.github/workflows/test-gate.yml 22-26](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L22-L26)
*   **C++ Changes**: Triggered by modifications within `tt-media-server/cpp_server/**`[.github/workflows/test-gate.yml 27-28](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L27-L28)

### 2. Linting and Formatting (`lint`)

Ensures code quality using `ruff` (version 0.15.0) [.github/workflows/test-gate.yml 44](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L44-L44) It performs both a logic check (`ruff check .`) and a formatting check (`ruff format --check .`) [.github/workflows/test-gate.yml 47-50](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L47-L50)

### 3. Python Unit Testing (`unit-tests`)

Executes the test suites for both the orchestration layer and the media server.

*   **Scope**: Runs tests in `tests/` (root) and `tt-media-server/tests/`[.github/workflows/test-gate.yml 84-99](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L84-L99)
*   **Mocking**: Since CI runners lack Tenstorrent hardware, these tests rely on the configuration in `pyproject.toml` to ignore integration-bound tests [.pyproject.toml 23-34](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.pyproject.toml#L23-L34)
*   **Reporting**: Parses `pytest` output to generate a GitHub Step Summary table containing Passed, Failed, Skipped, and Collection Error counts [.github/workflows/test-gate.yml 108-182](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L108-L182)

### 4. C++ Build and Validation

When C++ changes are detected, the pipeline initiates a build process for the `tt_media_server_cpp` backend.

*   **Build**: Compiles the server using CMake. The build process uses `tt-media-server/cpp_server/build.sh` which manages dependencies like Drogon and tokenizers [.tt-media-server/cpp_server/build.sh 1-161](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.tt-media-server/cpp_server/build.sh#L1-L161)
*   **Benchmarks**: Runs `cpp-server-benchmarks` to ensure no performance regressions in the C++ scheduler or sequence management logic [.github/workflows/test-gate.yml 230-240](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L230-L240)
*   **Prefill/Decode Split**: Validates the disaggregation logic where prefill and decode operations are handled by separate server instances [.github/workflows/test-gate.yml 242-254](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L242-L254)

### 5. Forge Runner CPU Tests (`forge-runner-cpu-tests`)

Specifically targets the `FORGE` inference engine integration [.github/workflows/test-gate.yml 214-228](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L214-L228) It validates the bridge using CPU-based execution to ensure the implementation remains functional without requiring a physical N150/N300 device.

## CI Configuration Validation

The repository maintains a complex CI configuration for hardware-specific testing in `.github/workflows/models-ci-config.json`. This file is validated against a JSON schema to prevent syntax errors that could break nightly or release pipelines.

### Validation Logic

The `validate-models-ci-config.yml` workflow runs on any PR that modifies the config file [.github/workflows/validate-models-ci-config.yml 4-7](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/validate-models-ci-config.yml#L4-L7) It uses `jsonschema` to verify:

*   **Device Names**: Must be valid hardware identifiers like `N150`, `N300`, `T3K`, `GALAXY`, etc. [.github/workflows/models-ci-config-schema.json 7-10](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/models-ci-config-schema.json#L7-L10)
*   **Inference Engines**: Must be one of `vLLM`, `MEDIA`, or `FORGE`[.github/workflows/models-ci-config-schema.json 76-81](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/models-ci-config-schema.json#L76-L81)
*   **Schedules**: Validates configurations for `nightly`, `weekly`, `bi_weekly`, and `release`[.github/workflows/models-ci-config-schema.json 85-99](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/models-ci-config-schema.json#L85-L99)

## Aggregation Gate (`ci-gate`)

The `ci-gate` job acts as the final synchronization point [.github/workflows/test-gate.yml 256-276](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/test-gate.yml#L256-L276) It is a "Required" check in GitHub branch protection rules. It only passes if all dependent jobs (`lint`, `unit-tests`, `cpp-build`, etc.) have completed successfully or were correctly skipped by the `detect-changes` logic.
