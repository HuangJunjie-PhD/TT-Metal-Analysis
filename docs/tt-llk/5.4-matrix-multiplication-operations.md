Matrix multiplication operations compute the product of two input matrices using the hardware's MVMUL instruction. This page covers the math unit's matmul functionality, including the reuse logic that determines which operand is cached, address modifier configuration, MOP programming, narrow tile support, and throttling mechanisms.

For unpacker operations that feed data to matmul, see [Unpacker System](https://deepwiki.com/tenstorrent/tt-llk/4-unpacker-system). For element-wise operations, see [Element-wise Operations](https://deepwiki.com/tenstorrent/tt-llk/5.2-element-wise-operations). For overall math unit configuration, see [Core Math Unit Configuration](https://deepwiki.com/tenstorrent/tt-llk/5.1-core-math-unit-configuration).

## Operation Overview

Matrix multiplication in the LLK computes `C = A × B` where operands are organized in tiles. The operation is implemented using the `MVMUL` (matrix-vector multiply) instruction, which processes data in 8-row chunks from one operand against 16×16 faces from the other operand.

**Operand Assignment:**

*   Input A (in0) → Loaded to **SrcB** registers
*   Input B (in1) → Loaded to **SrcA** registers
*   Result → Written to **Dest** registers

The key initialization functions are `_llk_math_matmul_init_`[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 678-731](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L678-L731) and the execution function is `_llk_math_matmul_`[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 733-905](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L733-L905)

## Reuse Logic and Dimension Parameters

The matmul operation uses a critical optimization called **reuse logic** that determines which operand (A or B) remains cached in source registers while the other operand is streamed through. This decision is based on the tile dimensions of the matrix multiplication.

**Key Parameters:**

*   `ct_dim` - Column tile dimension (number of tiles in K dimension for operand A, or M dimension for result)
*   `rt_dim` - Row tile dimension (number of tiles in K dimension for operand B, or N dimension for result)
*   `kt_dim` - K tile dimension (number of tiles in reduction dimension)
*   `t_dim` - The smaller of ct_dim or rt_dim (determines loop count)

**Reuse Decision:**`reuse_a = (ct_dim >= rt_dim)`[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 819-820](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L819-L820)

| Condition | Reused Operand | Streamed Operand | Loop Dimension |
| --- | --- | --- | --- |
| `ct_dim >= rt_dim` | Input A (SrcB) | Input B (SrcA) | `t_dim = rt_dim` |
| `ct_dim < rt_dim` | Input B (SrcA) | Input A (SrcB) | `t_dim = ct_dim` |

The unpacker coordinates with this logic by managing which operand's address is incremented between iterations [tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 230-376](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L230-L376)

**Sources:**[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 819-826](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L819-L826)[tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 30-31](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L30-L31)[tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 235-236](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L235-L236)

## Address Modifier Configuration

Address modifiers control how the hardware increments pointers to SrcA, SrcB, and Dest registers during execution. The function `matmul_configure_addrmod`[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 24-297](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L24-L297) configures seven address modifier slots (ADDR_MOD_0 through ADDR_MOD_6) based on tile dimensions and transpose settings.

**Primary Address Modifiers:**

| Slot | Purpose | SrcA | SrcB | Dest | Fidelity |
| --- | --- | --- | --- | --- | --- |
| **ADDR_MOD_0** | Inner loop increment | incr=0 | incr=8 | incr=8 | - |
| **ADDR_MOD_1** | Face boundary handling | varies | varies | varies | - |
| **ADDR_MOD_2** | Tile boundary handling | varies | varies | varies | - |
| **ADDR_MOD_3** | Copy of ADDR_MOD_0 | incr=0 | incr=8 | incr=8 | incr=1 |
| **ADDR_MOD_4** | Reset between iterations | varies | varies | varies | incr=1 |
| **ADDR_MOD_5** | Reset all, inc fidelity | clr=1 | clr=1 | clr=1 | incr=1 (if HiFi) |
| **ADDR_MOD_6** | Reset all inc fidelity | clr=1 | clr=1 | clr=1 | clr=1 |

The `MVMUL` instruction computes `D[8,16] = B[8,16] * A[16,16]`, processing 8 destination rows at a time [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 38-49](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L38-L49)

**Narrow Tile Handling:**

Special configurations exist for non-standard tile sizes:

*   `is_in0_16x32` - Input A has 16 rows × 32 cols [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 34](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L34-L34)
*   `is_in0_32x16` - Input A has 32 rows × 16 cols [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 35](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L35-L35)
*   `is_in1_32x16` - Input B has 32 rows × 16 cols [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 36](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L36-L36)

## MOP Programming and Instruction Sequences

The `matmul_configure_mop` function [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 299-454](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L299-L454) programs the replay buffer with sequences of `MVMUL` instructions that execute repeatedly. The instruction sequence depends on tile dimensions and transpose settings.

**Standard 32×32 Tile Sequence (16 instructions):**

The replay buffer length varies based on tile configuration [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 327-328](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L327-L328):

*   16 instructions for standard 32×32 tiles
*   8 instructions for narrow tiles (16×32, 32×16)
*   4 instructions for both-narrow combinations (16×32 × 32×16)

**Template Programming:**

The `ckernel_template` class [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 433](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L433-L433) organizes the instruction sequence into nested loops:

`// Outer loop: 1 iteration// Inner loop: fidelity phases (1 for LoFi, 2-4 for HiFi2-4)ckernel_template tmp(1, inner_loops,                      lltt::replay_insn(replay_buf_offset, replay_buf_len));`
For high fidelity modes, the inner loop executes multiple fidelity phases, with `ADDR_MOD_5` incrementing the fidelity counter between phases [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 432-442](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L432-L442)

**End-of-Loop Operations:**

Special handling clears source registers after the final iteration [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 407-429](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L407-L429):

*   If `reuse_a` and `t_dim == 1`: Clear SrcA with `CLR_A`
*   If `!reuse_a` and `t_dim == 1`: Clear SrcB with `CLR_B`
*   For high fidelity, always reset fidelity phase counter

**Sources:**[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 299-454](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L299-L454)[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 378-429](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L378-L429)

## Narrow Tile and Partial Face Support

Narrow tiles handle non-standard matrix dimensions where one or both operands have fewer than 4 faces or non-standard face dimensions. This enables efficient processing of matrices that don't align to 32×32 tile boundaries.

**Supported Narrow Tile Configurations:**

| Configuration | Input A Dim | Input B Dim | Description |
| --- | --- | --- | --- |
| `is_in0_16x32` | 16×32 | Any | Input A has 16 rows, 32 cols |
| `is_in0_32x16` | 32×16 | Any | Input A has 32 rows, 16 cols |
| `is_in1_32x16` | Any | 32×16 | Input B has 32 rows, 16 cols |
| `is_in1_16x32` | Any | 16×32 | Input B has 16 rows, 32 cols |

**Partial Face Unpacking:**

The unpacker supports partial face operations via the `unpA_partial_face` and `unpB_partial_face` flags [tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 165-166](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L165-L166):

*   When enabled, unpacking processes faces one at a time with explicit Z-counter resets
*   Uses `CH0_Z=0, CH1_Z=0` pattern via `0b0101` mask [tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L43-L43)
*   Configures `unpA_face_r_dim` and `unpB_face_r_dim` to specify face dimensions [tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h 161-162](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB_matmul.h#L161-L162)

**MOP Adjustments for Narrow Tiles:**

The MOP configuration differs for narrow tiles [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 334-397](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L334-L397):

## Throttling and Performance Control

Throttling reduces matmul compute throughput by inserting `NOP` instructions between `MVMUL` operations. This enables power management and thermal control. The `run_throttled_sequence` function [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 456-578](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L456-L578) implements five throttling levels.

**Throttling Levels:**

| Level | Throughput | NOP Pattern | Instructions |
| --- | --- | --- | --- |
| 0 | 100% (no throttling) | None | 16 MVMUL |
| 1 | 73% | 1 NOP every ~3 MVMUL | 10 insn total |
| 2 | 67% | 1 NOP every ~2 MVMUL | 11 insn total |
| 3 | 50% | 1 NOP per MVMUL | 13 insn total |
| 4 | 40% | 2 NOP per MVMUL | 16 insn total |
| 5 | 33% | 2-3 NOP per MVMUL | 16 insn total |

**Example: Level 1 Throttling Sequence [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 459-471](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L459-L471):**

`TTI_NOP;TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_0, 0);TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_1, 0);TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_0, 0);TTI_NOP;TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_2, 0);// ... continues with mixed MVMUL and NOP`
**Throttled MOP Configuration:**

The `matmul_configure_mop_throttled` function [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 592-676](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L592-L676) programs throttled sequences into the replay buffer:

`template <MathFidelity math_fidelity, int THROTTLE_LEVEL>inline void matmul_configure_mop_throttled(    const std::uint32_t ct_dim,    const std::uint32_t rt_dim,    const std::uint32_t in0_tile_r_dim = TILE_R_DIM,    const std::uint32_t in0_tile_c_dim = TILE_C_DIM,    const std::uint32_t in1_tile_r_dim = TILE_R_DIM,    const std::uint32_t in1_tile_c_dim = TILE_C_DIM,    const bool partial_face = false)`
**Restrictions:**

*   Throttling only works for full 32×32 tiles [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 611-613](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L611-L613)
*   Valid `THROTTLE_LEVEL` range is 1-5 [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 610](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L610-L610)

**Replay Buffer Organization for Levels 4-5:**

Higher throttling levels use a 2-outer-loop structure with multiple replay instructions [tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 634-642](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L634-L642):

## Fidelity Control and Precision Phases

Math fidelity controls precision by executing multiple computation phases with different mantissa bit patterns. The `MathFidelity` enum defines four levels [tests/python_tests/helpers/golden_generators.py 772-777](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L772-L777):

| Fidelity | Iterations | Description | Golden Model |
| --- | --- | --- | --- |
| **LoFi** | 0 | Single phase, masked mantissa | Phase 0 only |
