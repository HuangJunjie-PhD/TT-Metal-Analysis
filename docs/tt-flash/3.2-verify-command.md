# verify Command

## Purpose and Scope

The `verify` command provides read-only validation of firmware stored in SPI flash on Tenstorrent devices. It displays currently running firmware versions, reads the flashed firmware version from SPI, and checksums the firmware image to confirm integrity. Unlike the `flash` command (see [3.1](https://deepwiki.com/tenstorrent/tt-flash/3.1-flash-command)), `verify` performs no write operations and is safe to run on production systems.

The command can validate firmware against either:

*   A provided firmware bundle file
*   Previously stored flash records in known filesystem locations

For information about the flashing process itself, see [5.2](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages) Flash Stages.

## Verification Objectives

Based on the command help text, the `verify` command is designed to perform the following operations:

### 1. Version Display

*   **Running Version**: Query the currently executing firmware version from the chip's ARC processor
*   **Flashed Version**: Read the firmware version stored in SPI flash memory

### 2. Checksum Validation

The verification process checksums the firmware image in SPI flash and compares it against:

| Verification Source | Description | Priority |
| --- | --- | --- |
| Provided Bundle | When `fwbundle` argument is supplied, validate against this bundle's contents | Primary |
| Filesystem Flash Record | When no bundle provided, search for previously stored flash records | Secondary |
| Default Locations | Search known locations where flash records may be stored | Fallback |

### 3. Board Coverage

When `--skip-missing-fw` is specified, verification continues even if the firmware bundle lacks images for certain detected boards.

## Flash Record Search Locations

When no firmware bundle is provided, the `verify` command searches for flash records in known filesystem locations. The command help text indicates it will "search known locations that the flash record may have been written to and exit with an error if it cannot be found or read" [tt_flash/main.py 111](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L111-L111)

Expected search locations (inferred from typical system patterns):

*   `/etc/tenstorrent/` - System-wide configuration directory
*   `~/.config/tenstorrent/` - User-specific configuration
*   `/var/lib/tenstorrent/` - System state directory
*   Current working directory - Local flash records

If no valid flash record is found, the command exits with an error.

**Sources**: [tt_flash/main.py 111](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L111-L111)

### Current State

The `verify` command is **defined in the argument parser but not fully implemented** in the execution path. Examining the `main()` function [tt_flash/main.py 204-242](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L204-L242):

At [tt_flash/main.py 236](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L236-L236) the else clause catches all non-flash commands and raises:

This means invoking `tt-flash verify` will result in an error message stating no handler exists for the verify command.

### Expected Implementation

Based on the defined arguments and help text, a complete implementation would:

1.   Load manifest from provided bundle or search for flash records
2.   Detect local chips using `detect_local_chips()`[tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#LNaN-LNaN)
3.   For each detected chip: 
    *   Read SPI flash contents using `chip.spi_read()` methods
    *   Extract and parse flashed firmware version
    *   Query running firmware version via ARC messages
    *   Compute checksums of SPI flash contents
    *   Compare against reference (bundle or flash record)

4.   Display results in formatted output using `tabulate` and `CConfig.COLOR` utilities

## Comparison with flash Command

| Aspect | `flash` Command | `verify` Command |
| --- | --- | --- |
| **Primary Operation** | Write firmware to SPI flash | Read and validate SPI flash contents |
| **Destructive** | Yes - modifies SPI flash | No - read-only operations |
| **Requires Bundle** | Yes - mandatory | Optional - can use flash records |
| **Reset Required** | Yes (unless `--no-reset`) | No - no hardware state changes |
| **Version Enforcement** | Prevents downgrades (unless `--allow-major-downgrades`) | Reports version differences only |
| **Implementation** | Fully implemented in `flash_chips()` | Argument parser only, no handler |
| **Use Case** | Update firmware on devices | Validate firmware integrity and versions |

**Sources**: [tt_flash/main.py 212-234](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L212-L234)[tt_flash/main.py 109-127](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L109-L127)

## Example Usage Patterns

### Verify Against Provided Bundle

Validates SPI flash contents against the specified firmware bundle.

### Verify Using Stored Flash Records

Searches default locations for previously written flash records and validates against them.

### Verify with Missing Firmware Tolerance

Continues verification even if some detected boards lack firmware images in the bundle.

### Verify Without Color Output

Produces verification output without ANSI color codes, useful for log files or non-TTY environments.

## Integration with Hardware Abstraction

When implemented, the `verify` command would leverage the same hardware abstraction layer used by the `flash` command:

This would mirror the architecture used in `flash_chips()`[tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py) using:

*   `detect_local_chips()` for hardware enumeration [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#LNaN-LNaN)
*   `TTChip.spi_read()` for SPI flash access
*   `TTChip.get_bundle_version()` or similar for version queries
*   ARC message protocols for running firmware interrogation
