## Purpose and Scope

This document explains the runtime execution layer of the Forge compiler, focusing on how compiled models are executed on Tenstorrent devices. It covers the `CompiledModel` class, runtime state management, forward/backward/optimizer execution, and tensor lifecycle management during inference and training.

For information about the compilation pipeline that produces these runtime artifacts, see [Compilation Pipeline Overview](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.1-compilation-pipeline-overview). For details about the MLIR backend that generates the executable binaries, see [MLIR Backend and Code Generation](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.3-mlir-backend-and-code-generation). For verification of runtime results, see [Verification System](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.4-verification-system).

## CompiledModel Architecture

The `CompiledModel` class ([forge/forge/compiled_graph_state.py 165-238](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compiled_graph_state.py#L165-L238)) is the primary interface for executing compiled models on Tenstorrent devices. ### CompiledGraphState

Each execution program (forward, backward, optimizer) has an associated `CompiledGraphState` that tracks:

| Attribute | Description |
| --- | --- |
| `graph` | The compiled computation graph |
| `ordered_input_names` | Ordered list of input tensor names |
| `ordered_output_names` | Ordered list of output tensor names |
| `ordered_parameter_node_names` | Model parameters (weights) |
| `ordered_constant_node_names` | Constant tensors |
| `consteval_trace` | JSON trace for constant evaluation |
| `post_const_eval_constants` | Constants after constant folding |
| `post_const_eval_parameters` | Parameters after constant folding |
| `aliased_outputs` | Mapping of aliased output names |

## Integration with Framework Models

### PyTorch Parameter Sharing

In training mode, `CompiledModel` maintains shared parameter references with the original PyTorch module:

This ensures that:

*   Parameter updates on the device are visible to PyTorch optimizers
*   PyTorch optimizer updates are visible to the device
*   Gradient accumulation works correctly

Sources: [forge/forge/compiled_graph_state.py 299-310](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compiled_graph_state.py#L299-L310)

### Training Mode Considerations

**Training vs Inference Differences:**

| Aspect | Inference | Training |
| --- | --- | --- |
| Backward graph | Not compiled | Compiled if `training=True` |
| Parameter sharing | Optional | Required for gradients |
| Gradient tracking | Disabled | Enabled for trainable parameters |
| Memory footprint | Lower (forward only) | Higher (forward + backward) |
| Optimizer on device | N/A | Optional via `opt_compiled_graph_state` |

Sources: [forge/forge/compiled_graph_state.py 299-313](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compiled_graph_state.py#L299-L313)[forge/forge/compile.py 186-260](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L186-L260)

## Execution Stage Tracking

Runtime execution is tracked using the property handler system for debugging and analytics:

Sources: [forge/forge/verify/verify.py 369-557](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L369-L557)[forge/forge/forge_property_utils.py 279-347](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L279-L347)

**Key Execution Stages:**

*   `FAILED_TTNN_BINARY_EXECUTION`: Set before runtime execution
*   `FAILED_VERIFICATION`: Set during output verification
*   `PASSED`: Set after successful execution and verification

These stages are recorded in test metadata for CI/CD analysis and failure classification.

Sources: [forge/forge/forge_property_utils.py 279-308](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L279-L308)
