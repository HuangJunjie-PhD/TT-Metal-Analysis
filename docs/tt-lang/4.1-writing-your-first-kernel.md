This page provides a step-by-step tutorial for writing your first `tt-lang` kernel. We will build a simple element-wise addition kernel that adds two tensors on a Tenstorrent core. This tutorial covers:

*   Creating circular buffers (Dataflow Buffers) for data staging [[test/python/simple_add.py 28-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L28-L30)]

## Kernel Architecture Overview

A `tt-lang` kernel consists of three key components that work together to perform computation on Tenstorrent hardware:

**Kernel Component Diagram**

**Thread Execution Pattern**: The data movement threads (`dm_read`, `dm_write`) and compute thread (`add_compute`) run concurrently on different RISC processors within the Tensix core (MATH, BRISC, and NCRISC), synchronized via circular buffers (referred to as Dataflow Buffers or DFBs in the DSL) [[docs/sphinx/specs/TTLangSpecification.md 53-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[docs/sphinx/specs/TTLangSpecification.md?plain=1#L53-L58)]. `dm_read` loads input data into DFBs, `add_compute` consumes from input DFBs and produces to the output DFB, and `dm_write` writes results back to device memory [[test/python/simple_add.py 32-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L32-L62)].

## Step 1: Kernel Definition and Grid Specification

Every kernel begins with the `@ttl.operation` decorator, which specifies the computational grid [[test/python/simple_add.py 26-27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L26-L27)]:

`@ttl.operation(grid=(1, 1))def add_kernel(lhs, rhs, out):    # Kernel body...`
### Key Concepts

| Parameter | Description | Example |
| --- | --- | --- |
| `grid` | 2D tuple specifying core layout `(cols, rows)` | `(1, 1)`[[test/python/simple_add.py 26](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L26-L26)] |
| Function parameters | Input/output tensors (typically `ttnn.Tensor`) | `lhs`, `rhs`, `out`[[test/python/simple_add.py 27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L27-L27)] |

### Validation Rules

*   Grid must be a 2D tuple for single-chip execution [[docs/sphinx/specs/TTLangSpecification.md 100-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L103)].
*   All tensors must be in a tiled format (32x32 tiles) for hardware efficiency [[test/python/simple_add.py 182](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L182-L182)] [[docs/sphinx/specs/TTLangSpecification.md 55-57](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[docs/sphinx/specs/TTLangSpecification.md?plain=1#L55-L57)].

## Step 2: Creating Dataflow Buffers

Dataflow buffers (DFBs) act as staging areas in L1 memory between device memory and compute registers. Create one DFB per input/output tensor [[test/python/simple_add.py 28-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L28-L30)]:

`@ttl.operation(grid=(1, 1))def add_kernel(lhs, rhs, out):    lhs_dfb = ttl.make_dataflow_buffer_like(lhs, shape=(1, 1), block_count=2)    rhs_dfb = ttl.make_dataflow_buffer_like(rhs, shape=(1, 1), block_count=2)    out_dfb = ttl.make_dataflow_buffer_like(out, shape=(1, 1), block_count=2)`
### DFB Parameters [[test/python/simple_add.py 28-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L28-L30)]

| Parameter | Description | Typical Value |
| --- | --- | --- |
| `tensor` | Template tensor to inherit layout/dtype | Input/output tensor |
| `shape` | Number of tiles `(rows, cols)` to process per iteration | `(1, 1)` for single tile |
| `block_count` | Double-buffering multiplier (Capacity = shape × count) | `2` for double-buffering |

**Block Count Purpose**: `block_count=2` allows one tile to be consumed by the compute thread while another is being fetched by the data movement thread, enabling pipelined execution. This is also referred to as `buffer_factor` in earlier versions of the specification [[docs/sphinx/specs/TTLangSpecification.md 47](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[docs/sphinx/specs/TTLangSpecification.md?plain=1#L47-L47)].

Sources: [[test/python/simple_add.py 28-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L28-L30)], [[test/python/simple_add_multitile.py 30-32](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_multitile.py#L30-L32)]

## Step 3: Define the Compute Thread

The compute thread performs math operations on tiles. It waits for data to arrive in DFBs, performs the math, and stores results to the output DFB [[test/python/simple_add.py 32-41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L32-L41)]:

`@ttl.compute()def add_compute():    # 1. Wait for input tiles to be available in DFBs    l = lhs_dfb.wait()    r = rhs_dfb.wait()        # 2. Reserve space in output DFB    o = out_dfb.reserve()        # 3. Perform computation (operator overloading maps to ttl.add)    result = l + r        # 4. Store result to output DFB    o.store(result)        # 5. Release DFBs (inputs: pop consumed data, output: push new data)    l.pop()    r.pop()    o.push()`
### DFB Lifecycle Operations

**Code Entity to System Space: Compute Thread**

| Operation | Purpose | Returns | When to Use |
| --- | --- | --- | --- |
| `dfb.wait()` | Wait for data to be available (consumer) | `TensorBlock` | Reading inputs [[test/python/simple_add.py 34-35](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L34-L35)] |
| `dfb.reserve()` | Wait for space to be available (producer) | `TensorBlock` | Writing outputs [[test/python/simple_add.py 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L36-L36)] |
| `block.pop()` | Release consumed data | None | After processing inputs [[test/python/simple_add.py 39-40](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L39-L40)] |
| `block.push()` | Make produced data visible | None | After storing outputs [[test/python/simple_add.py 41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L41-L41)] |

## Step 4: Define Data Movement Threads

Data movement threads handle DMA transfers between device memory (DRAM/L1) and circular buffers using the Network-on-Chip (NOC) [[test/python/simple_add.py 43-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L43-L62)].

### Read Thread: Memory → DFB [[test/python/simple_add.py 44-54](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L44-L54)]

`@ttl.datamovement()def dm_read():    # Read first input    lhs_blk = lhs_dfb.reserve()              # 1. Reserve L1 space    tx_lhs = ttl.copy(lhs[0, 0], lhs_blk)   # 2. Start async DMA (Memory -> L1)    tx_lhs.wait()                            # 3. Wait for DMA completion    lhs_blk.push()                           # 4. Signal compute thread`
### Write Thread: DFB → Memory [[test/python/simple_add.py 56-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L56-L62)]

`@ttl.datamovement()def dm_write():    out_blk = out_dfb.wait()                 # 1. Wait for compute to finish    tx = ttl.copy(out_blk, out[0, 0])        # 2. Start async DMA (L1 -> Memory)    tx.wait()                                 # 3. Wait for DMA completion    out_blk.pop()                             # 4. Release L1 space`
### Copy Operation Semantics

| Copy Pattern | Source | Destination | Direction | Use Case |
| --- | --- | --- | --- | --- |
| `ttl.copy(tensor[i,j], dfb)` | Device memory | Circular buffer | Memory → CB | Read inputs [[test/python/simple_add.py 47-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L47-L48)] |
| `ttl.copy(dfb, tensor[i,j])` | Circular buffer | Device memory | CB → Memory | Write outputs [[test/python/simple_add.py 60-61](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L60-L61)] |

Sources: [[test/python/simple_add.py 43-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L43-L62)], [[test/python/simple_add_dram.py 46-65](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_dram.py#L46-L65)]

## Alternative Pattern: With-Statement for DFB Lifecycle

The `with` statement provides automatic DFB lifecycle management. It handles `wait`/`reserve` on entry and `pop`/`push` on exit in reverse order [[test/python/simple_add_with_stmt.py 15-18](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_with_stmt.py#L15-L18)] [[test/python/simple_add_with_stmt.py 41-46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_with_stmt.py#L41-L46)]:

`@ttl.compute()def compute():    # 'with' automatically handles wait/reserve at entry, pop/push at exit    with lhs_dfb.wait() as l, rhs_dfb.wait() as r, out_dfb.reserve() as o:        result = l + r        o.store(result)    # Automatic: o.push(), r.pop(), l.pop() (reverse order)`
Sources: [[test/python/simple_add_with_stmt.py 12-18](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_with_stmt.py#L12-L18)], [[test/python/simple_add_with_stmt.py 41-46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_with_stmt.py#L41-L46)]

## Multi-Tile and Multi-Node Extensions

For more complex kernels, you can extend these patterns to process multiple tiles per core or distribute work across a grid of cores.

### Multi-Tile Processing

When processing multiple tiles, use nested loops and linearized indexing to access specific tiles within the `TensorBlock`[[test/python/simple_add_multitile.py 114-122](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add_multitile.py#L114-L122)]:

`for i in range(2):    for j in range(2):        # Linearized index: i * cols + j        lin_idx = i * 2 + j        # ... perform tile operation ...`
### Multi-Node Parallelism

Use `ttl.node(dims=2)` to retrieve the current core's coordinates and partition the global tensor accordingly [[examples/elementwise-tutorial/step_3_multinode.py 93-96](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[examples/elementwise-tutorial/step_3_multinode.py#L93-L96)]:

`node_col, node_row = ttl.node(dims=2)# Calculate start_row_tile and start_col_tile based on node coordinates`

## Execution and Validation

To execute your kernel, you can use the functional simulator or run it on hardware using `ttnn` to manage device memory [[test/python/simple_add.py 164-193](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/python/simple_add.py#L164-L193)].

### Running on Hardware

`import torchimport ttnnimport ttl device = ttnn.open_device(device_id=0) # Create tensors in TILE_LAYOUT (32x32 tiles)lhs_torch = torch.full((32, 32), 2.0, dtype=torch.bfloat16)rhs_torch = torch.full((32, 32), 3.0, dtype=torch.bfloat16) lhs = ttnn.from_torch(lhs_torch, layout=ttnn.TILE_LAYOUT, device=device, memory_config=ttnn.DRAM_MEMORY_CONFIG)rhs = ttnn.from_torch(rhs_torch, layout=ttnn.TILE_LAYOUT, device=device, memory_config=ttnn.DRAM_MEMORY_CONFIG)out = ttnn.from_torch(torch.zeros((32, 32)), layout=ttnn.TILE_LAYOUT, device=device, memory_config=ttnn.DRAM_MEMORY_CONFIG) # Call the kernel functionadd_kernel(lhs, rhs, out) # Extract resultsresult = ttnn.to_torch(out)print(result)ttnn.close_device(device)`
