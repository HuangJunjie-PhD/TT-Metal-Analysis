This document covers the standardized processes and templates for reporting bugs and requesting new features in TT-Studio. It provides guidance on using the GitHub issue templates and following established procedures for community contributions, including versioning and branching strategies.

For information about development workflows and CI processes, see [Testing and CI](https://deepwiki.com/tenstorrent/tt-studio/8.2-testing-and-ci). For general development setup, see [Development Environment](https://deepwiki.com/tenstorrent/tt-studio/8.1-development-environment).

## Overview

TT-Studio uses GitHub issue templates to standardize bug reports and feature requests, ensuring consistent information collection and efficient issue triage. The system provides structured templates that guide users through providing necessary details for effective issue resolution. All contributions must begin with filing a feature request or bug report in the Issues section [CONTRIBUTING.md 11-13](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L11-L13)

## Contribution Standards and Review

To ensure quality, all code changes must adhere to specific technical standards before an issue is considered resolved [CONTRIBUTING.md 21-25](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L21-L25)

### Acceptance Criteria

*   **Approvals**: A PR must be approved by a maintaining team member and relevant codeowners [CONTRIBUTING.md 27-29](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L27-L29)
*   **Validation**: Changes must pass pre-commit hooks and automated GitHub Actions workflow tests [CONTRIBUTING.md 30-32](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L30-L32)
*   **Issue Alignment**: The implementation must pass all acceptance criteria mandated in the original GitHub issue [CONTRIBUTING.md 31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L31-L31)

## Versioning and Branching Strategy

When resolving issues and feature requests, developers must follow specific versioning and branching rules to maintain project integrity.

### Semantic Versioning (SemVer)

TT-Studio follows semantic versioning principles for all releases [CONTRIBUTING.md 105-108](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L105-L108):

*   **MAJOR**: Breaking changes to APIs or functionality (e.g., removing features, networking redesign) [CONTRIBUTING.md 109-114](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L109-L114)
*   **MINOR**: New backward-compatible features (e.g., supporting new models like YOLOv4) [CONTRIBUTING.md 116-120](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L116-L120)
*   **PATCH**: Backward-compatible bug fixes and minor improvements [CONTRIBUTING.md 122-123](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L122-L123)

### Branching Workflow

1.   **Feature Branches**: Created from `dev` with naming convention `dev-name/feature` or `dev/github-issue-number`[CONTRIBUTING.md 58-62](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L58-L62)
2.   **Merging to Dev**: Feature branches are **squash merged** into `dev`[CONTRIBUTING.md 64-65](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L64-L65)
3.   **Release Preparation**: A release cut branch (`rc-vx.x.x`) is created from `main`, and features are **cherry-picked** from `dev`[CONTRIBUTING.md 73-81](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L73-L81)
4.   **Production**: The release branch is merged into `main` and tagged [CONTRIBUTING.md 89-94](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L89-L94)

Sources: [CONTRIBUTING.md 37-103](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CONTRIBUTING.md?plain=1#L37-L103)

## Code of Conduct

All interactions regarding issues and feature requests must adhere to the **Contributor Covenant Code of Conduct**. This includes:

*   **Positive Behavior**: Demonstrating empathy, being respectful of differing opinions, and giving/accepting constructive feedback [CODE_OF_CONDUCT.md 15-27](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CODE_OF_CONDUCT.md?plain=1#L15-L27)
*   **Unacceptable Behavior**: Trolling, harassment, or the use of sexualized language [CODE_OF_CONDUCT.md 28-37](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CODE_OF_CONDUCT.md?plain=1#L28-L37)
*   **Enforcement**: Community leaders have the right to remove or reject issues and comments not aligned with these standards [CODE_OF_CONDUCT.md 41-49](https://github.com/tenstorrent/tt-studio/blob/a6d347af/CODE_OF_CONDUCT.md?plain=1#L41-L49)
