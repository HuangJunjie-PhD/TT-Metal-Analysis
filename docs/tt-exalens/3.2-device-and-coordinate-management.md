This page documents the Device abstraction layer and coordinate management system in TTExaLens. The Device class provides hardware abstraction for different chip architectures (Wormhole, Blackhole, Quasar), while the OnChipCoordinate system handles conversion between five different coordinate systems used throughout the hardware and software stack. For information about device communication and UMD integration, see [UMD Integration Layer](https://deepwiki.com/tenstorrent/tt-exalens/5.6-umd-integration-layer). For information about NOC operations and memory access using coordinates, see [Memory Access Operations](https://deepwiki.com/tenstorrent/tt-exalens/3.3-memory-access-operations).

## Device Abstraction Overview

The `Device` class is the core hardware abstraction layer in TTExaLens. It provides a unified interface to chip hardware while hiding architecture-specific details. Device instances are created through a factory method and provide access to chip topology, coordinate conversion, and hardware blocks.

### Device Factory and Properties

Device instances are created using the `Device.create()` factory method, which detects the chip architecture and instantiates the appropriate platform-specific subclass:

`# Factory method creates architecture-specific Device instancesdevice = Device.create(device_id=0, context=context)`
The factory method inspects the architecture reported by UMD and returns one of three concrete implementations:

*   `WormholeDevice` for `tt_umd.ARCH.WORMHOLE_B0`
*   `BlackholeDevice` for `tt_umd.ARCH.BLACKHOLE`
*   `QuasarDevice` for `tt_umd.ARCH.QUASAR`

Key Device properties:

| Property | Type | Description |
| --- | --- | --- |
| `id` | `int` | Chip ID within the cluster |
| `unique_id` | `int` | Hardware unique identifier |
| `_arch` | `tt_umd.ARCH` | Chip architecture |
| `is_local` | `bool` | Whether device is MMIO-capable (directly accessible via PCIe) |
| `board_type` | `tt_umd.BoardType` | Board type (e.g., p100, e150) |
| `firmware_version` | `FirmwareVersion` | ARC firmware version |
| `active_noc` | `int` | Currently active NOC (0 or 1) for operations |

## OnChipCoordinate System

The `OnChipCoordinate` class represents a single location on the chip and provides conversion between five different coordinate systems. Internally, coordinates are stored as NOC0 coordinates with a reference to their device, enabling conversion to any supported coordinate system.

### The Five Coordinate Systems

TTExaLens supports five coordinate systems, each serving a different purpose in the hardware/software stack:

| Coordinate System | Purpose | Notation | Harvesting Dependent |
| --- | --- | --- | --- |
| **noc0** | NOC 0 routing coordinates. Distance of 1 = 1 hop on NOC. Wraps around. | `X-Y` | No |
| **noc1** | NOC 1 routing coordinates. Same as noc0 but opposite direction. | `X-Y` | No |
| **die** | Geographic/physical layout on die. Used in marketing materials. | `X,Y` | No |
| **logical** | User-facing Cartesian grid. Starts at 0,0, hides harvesting. Includes core type. | `X,Y` or `qX,Y` (q=core type prefix) | Yes |
| **translated** | NOC coordinates adjusted for harvesting. Used by NOC hardware. | `X-Y` | Yes |

The logical coordinate system includes core type information, so the same `(X,Y)` can refer to different core types:

*   `0,0` or `t0,0` - Tensix core at logical position (0,0)
*   `e0,0` - Ethernet core at logical position (0,0)
*   `d0,0` - DRAM core at logical position (0,0)

**Sources:**[ttexalens/coordinate.py 1-62](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/coordinate.py#L1-L62)[ttexalens/device.py 399-458](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L399-L458)

### Creating OnChipCoordinate Objects

Coordinates can be created programmatically or parsed from strings:

`# Programmatic creationcoord = OnChipCoordinate(x=1, y=2, input_type="noc0", device=device) # From string (auto-detects format)coord = OnChipCoordinate.create("1-2", device)      # noc0 or translatedcoord = OnChipCoordinate.create("0,0", device)      # logical tensixcoord = OnChipCoordinate.create("e0,0", device)     # logical ethernetcoord = OnChipCoordinate.create("ch0", device)      # DRAM channel # Using the public API (string only)from ttexalens import convert_coordinatecoord = convert_coordinate("1-2", device_id=0, context=context)`
String format detection rules:

*   Contains `-` → NOC coordinate (noc0 or translated, auto-detected)
*   Contains `,` → Logical coordinate
*   Starts with core type letter → Logical with explicit core type
*   Starts with `CH` or `ch` → DRAM channel (converts to logical DRAM)

### Coordinate Conversion Methods

**OnChipCoordinate conversion methods:**

`# Convert to different coordinate systemsnoc0_tuple = coord.to("noc0")              # Returns (x, y)noc1_tuple = coord.to("noc1")              # Returns (x, y)die_tuple = coord.to("die")                # Returns (x, y)logical = coord.to("logical")              # Returns ((x, y), core_type)translated_tuple = coord.to("translated")  # Returns (x, y) # Convert to string representationsnoc0_str = coord.to_str("noc0")           # "1-2"logical_str = coord.to_str("logical")     # "0,0" or "e0,0"user_str = coord.to_user_str()            # "1-2 (0,0)" # Change to same location on different devicecoord_dev1 = coord.change_device(device1)`
**Device-level conversion methods:**

The Device class maintains pre-computed lookup tables for fast coordinate conversion:

`# Convert any coordinate system to noc0noc0_coord = device.to_noc0((0, 0), "logical", "tensix") # Convert from noc0 to any coordinate system(logical_coord, core_type) = device.from_noc0((1, 2), "logical")`
These methods use the internal lookup tables `_to_noc0` and `_from_noc0` which are populated during device initialization by querying UMD's coordinate manager.

## Block Location Management

The Device class manages NOC blocks (functional units on the chip) and their locations. Each block type represents a different functional unit type: Tensix cores, Ethernet cores, DRAM controllers, ARC processor, etc.

### Block Types

| `functional_workers` | `.` | Functional Tensix worker | `tensix` | `False` | Green |
| `eth` | `E` | Ethernet core | `eth` | `False` | Yellow |

### Querying Block Locations

**Block location query methods:**

`# Get all locations of a specific block typetensix_locations = device.get_block_locations("functional_workers")eth_locations = device.get_block_locations("eth")dram_locations = device.get_block_locations("dram") # Get the NocBlock object at a specific locationblock = device.get_block(coordinate) # Get all NocBlock objects of a specific typetensix_blocks = device.get_blocks("functional_workers") # Get the block type at a specific locationblock_type = device.get_block_type(coordinate)  # Returns "functional_workers", etc. # Special accessorsarc = device.arc_block                          # Single ARC blockactive_eth = device.active_eth_block_locations  # Ethernet cores in useidle_eth = device.idle_eth_block_locations      # Ethernet cores not in use`
The `_block_locations` dictionary is lazily computed from the UMD SOC descriptor during device initialization. It queries the descriptor for each core type and creates `OnChipCoordinate` objects in NOC0 space:

1.   For each block type defined in `block_types`
2.   Query `_soc_descriptor.get_cores()` or `get_harvested_cores()` based on `core_harvesting` flag
3.   Convert each core coordinate to an `OnChipCoordinate` in NOC0 space
4.   Store in `_block_locations[block_type]`

## Device Topology Inspection

The Device class provides a `render()` method for visualizing chip topology as an ASCII grid. This is used extensively by the CLI `device` command.

### Rendering System

**Render method signature:**

`def render(    self,     axis_coordinate: str = "die",    cell_renderer: Callable[[OnChipCoordinate], str] | None = None,    legend: list[str] | None = None) -> str`
**Parameters:**

*   `axis_coordinate`: The coordinate system to use for the grid axes. Options: `"die"`, `"noc0"`, `"noc1"`, `"logical"`, `"logical-tensix"`, `"logical-eth"`, `"logical-dram"`, `"translated"`
*   `cell_renderer`: Optional function that takes an `OnChipCoordinate` and returns a string for that cell. If `None`, defaults to `location.to_str("logical")`
*   `legend`: Optional list of strings to display as a legend column on the right side

**Example custom cell renderers:**

`# Show RISC-V status (CLI default)def riscv_status_renderer(location):    block = device.get_block(location)    riscs = block.debuggable_riscs    if not riscs:        # No RISC cores, show block symbol        block_type = device.get_block_type(location)        return device.block_types[block_type].symbol    status = ""    for risc in riscs:        status += "R" if risc.is_running() else "-"    return status # Show NOC coordinates on logical griddef show_noc_coords(location):    return location.to_str("noc0") # Show block typedef show_block_type(location):    return device.get_block_type(location)`
**Usage examples:**

`# Default rendering (logical coordinates in each cell)print(device.render()) # Show RISC-V status on die coordinate axesprint(device.render("die", riscv_status_renderer)) # Show NOC0 coordinates on logical-tensix axesprint(device.render("logical-tensix", show_noc_coords))`
The rendering algorithm:

1.   **Compute extents**: Iterate through all block locations, convert to target axis coordinate system, find min/max for horizontal and vertical axes
2.   **Build grid dictionary**: Create `all_block_locs` mapping from `(ui_hor, ui_ver)` grid positions to `OnChipCoordinate` objects
3.   **Render cells**: For each grid position, call `cell_renderer(location)` if location exists, or leave empty
4.   **Add axis labels**: Prepend row labels, append column headers
5.   **Add legend**: Optionally append legend column on the right
6.   **Format with tabulate**: Use `tabulate` library to create aligned ASCII table

**Sources:**[ttexalens/device.py 596-670](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L596-L670)[docs/ttexalens-app-docs.md 403-595](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/docs/ttexalens-app-docs.md?plain=1#L403-L595)

### CLI Device Command Integration

The `device` command in the CLI uses the rendering system extensively. Examples:

`# Show RISC-V status on logical-tensix axes (default)tt-exalens> device # Show RISC-V status on noc0 axes> device noc0 # Show block types on noc0 axes> device noc0 block # Show NOC0 coordinates on logical-tensix axes> device logical-tensix noc0 # Show for specific device only> device -d 0 die`
The command translates these arguments into appropriate `axis_coordinate` and `cell_renderer` parameters for the `render()` method.

## Coordinate System Initialization

During device initialization, coordinate translation tables are pre-computed for fast lookup:

The initialization process:

1.   **Populate `_noc0_to_block_type`**: Map each NOC0 coordinate to its block type by iterating `_block_locations`

2.   **Build conversion tables**: For each NOC0 location and block type:

    *   Query UMD's coordinate manager for conversions to `noc1`, `logical`, `translated`
    *   Store bidirectional mappings in `_from_noc0` and `_to_noc0` dictionaries
    *   Add `die` coordinate mappings (not supported by UMD, computed using `DIE_X_TO_NOC_0_X` and similar lookup tables)

3.   **Handle multiple core types**: `logical` and `die` coordinates include core type as a third dimension. `translated` and `noc1` are unique per coordinate (no ambiguity), so they map with `core_type="any"`

**Lookup table structures:**

`# Map from (noc0_coord, target_system) to (converted_coord, core_type)_from_noc0: dict[tuple[tuple[int, int], str], tuple[tuple[int, int], str]] # Map from (coord, source_system, core_type) to noc0_coord_to_noc0: dict[tuple[tuple[int, int], str, str], tuple[int, int]] # Map from noc0_coord to block_type string_noc0_to_block_type: dict[tuple[int, int], str]`

## Summary

The Device and Coordinate Management system provides:

1.   **Unified hardware abstraction** through the `Device` class and architecture-specific subclasses
2.   **Five coordinate systems** (noc0, noc1, die, logical, translated) unified by `OnChipCoordinate`
3.   **Fast coordinate conversion** through pre-computed lookup tables
4.   **Block topology management** with queries for block locations and types
5.   **Visual device inspection** through the ASCII rendering system
6.   **Public API** via `convert_coordinate()` for string-based coordinate creation and conversion

This abstraction layer isolates higher-level code from hardware-specific details like harvesting, NOC routing, and architecture differences, while providing efficient access to low-level hardware operations.
