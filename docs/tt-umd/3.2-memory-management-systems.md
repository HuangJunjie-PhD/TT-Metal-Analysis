The memory management layer in tt-umd provides address translation, host memory management, and optimization strategies for device memory access. The system consists of three primary components: `TLBManager` for PCIe address translation, `SysmemManager` for host memory allocation (including hugepages), and `SysmemBuffer` for DMA-backed memory transfers.

## Overview

Memory management enables host-to-device and device-to-device communication through multiple pathways depending on access patterns and device connectivity (Silicon vs. Simulation).

## TLBManager and TlbWindow

`TLBManager` manages the lifecycle of Translation Lookaside Buffer (TLB) windows. It abstracts the differences between hardware architectures (Wormhole, Blackhole) and execution environments (Silicon, Simulation).

### TLB Allocation and Configuration

TLBs are requested through `configure_tlb`, which allocates a `TlbWindow` and records the mapping in internal maps.

### Silicon vs. Simulation TLB Handles

The system uses the Bridge pattern to support different backends via `TlbHandle` subclasses:

*   **SiliconTlbHandle**: Interfaces with physical PCIe BARs using `mmap`. [device/pcie/silicon_tlb_handle.cpp 10-40](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/pcie/silicon_tlb_handle.cpp#L10-L40)
*   **TTSimTlbHandle**: Used for simulation; writes TLB configuration registers via a `TTSimCommunicator`. [device/pcie/tt_sim_tlb_handle.cpp 24-51](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/pcie/tt_sim_tlb_handle.cpp#L24-L51)
*   **RtlSimTlbHandle**: Interfaces with RTL simulators via FlatBuffers. [device/pcie/rtl_sim_tlb_handle.cpp 15-35](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/pcie/rtl_sim_tlb_handle.cpp#L15-L35)

### Memory Access Optimization: device_memcpy

For Silicon devices, memory access is optimized through `device_memcpy.cpp`. This implementation uses specialized logic for `memcpy_to_device` and `memcpy_from_device` to handle alignment requirements and maximize PCIe throughput.

*   **Alignment**: Logic in `SiliconTlbWindow::memcpy_to_device` performs Read-Modify-Write (RMW) for partial words at the start or end of a transfer to maintain 4-byte alignment for device writes. [device/pcie/silicon_tlb_window.cpp 175-220](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/pcie/silicon_tlb_window.cpp#L175-L220)
*   **Safe Access**: `SiliconTlbWindow` includes a `sigbus_handler` to catch `SIGBUS` errors during device access (e.g., if a device is warm-reset during an I/O operation) and recover gracefully using `siglongjmp`. [device/pcie/silicon_tlb_window.cpp 32-67](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/pcie/silicon_tlb_window.cpp#L32-L67)

## SysmemManager

`SysmemManager` handles host-side memory that is accessible by the device. It is split into `SiliconSysmemManager` and `SimulationSysmemManager`.

### Hugepage Management

On Silicon, `SiliconSysmemManager` attempts to allocate 1GB hugepages for host-device shared memory. If 1GB hugepages are unavailable, it falls back to 512MB (AArch64), 2MB, or regular system pages.

### Silicon vs. Simulation Split

| Feature | SiliconSysmemManager | SimulationSysmemManager |
| --- | --- | --- |
| **Backing Store** | Physical Hugepages via `hugetlbfs` | Anonymous `mmap` with `MADV_HUGEPAGE` |
| **IOMMU** | Supports IOMMU pinning/mapping | Uses a `MappedBufferRegistry` for tracking |
| **DMA** | Direct PCIe DMA via KMD | Simulated DMA via `memcpy` |
| **Channels** | Up to 4 host memory channels | Up to 4 host memory channels |

## SysmemBuffer and DMA

`SysmemBuffer` represents a pinned/mapped region of host memory that can be used for zero-copy DMA transfers.

### DMA Operations

The `SysmemBuffer` class provides `dma_write_to_device` and `dma_read_from_device` methods. These methods:

1.   Validate the buffer offset. [device/chip_helpers/sysmem_buffer.cpp 79](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_buffer.cpp#L79-L79)
2.   Configure a transient TLB window for the target device address. [device/chip_helpers/sysmem_buffer.cpp 88-96](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_buffer.cpp#L88-L96)
3.   Invoke `tt_device_->dma_h2d_zero_copy` or `dma_d2h_zero_copy` to initiate the transfer. [device/chip_helpers/sysmem_buffer.cpp 113](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_buffer.cpp#L113-L113)

### IOMMU and Memory Pinning

If IOMMU is enabled, `SiliconSysmemManager` uses `pin_or_map_iommu()` to map host memory into the device's IOVA space. This allows the device to access host memory using the addresses returned by `pci_device_->map_buffer_to_noc()`. [device/chip_helpers/silicon_sysmem_manager.cpp 124-131](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/silicon_sysmem_manager.cpp#L124-L131)

## Memory Access Optimization Strategies

1.   **TLB Window Caching**: `SysmemBuffer` caches its TLB window to avoid repeated allocation overhead during DMA loops. [device/chip_helpers/sysmem_buffer.cpp 95](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_buffer.cpp#L95-L95)
2.   **Hugepage Alignment**: `SysmemManager` ensures that addresses are wrapped around the hugepage size (typically 1GB) to simulate hardware circular buffers. [device/chip_helpers/sysmem_manager.cpp 46](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_manager.cpp#L46-L46)
3.   **Zero-Copy DMA**: By pinning host memory and mapping it to the device's NOC or PCIe address space, UMD avoids intermediate copies between host and device. [device/chip_helpers/sysmem_buffer.cpp 41-45](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/sysmem_buffer.cpp#L41-L45)
4.   **Write-Combining (WC)**: Silicon TLB windows are typically mapped as `TlbMapping::WC` to allow the CPU to buffer multiple writes into a single PCIe burst. [device/chip_helpers/tlb_manager.cpp 46](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip_helpers/tlb_manager.cpp#L46-L46)
