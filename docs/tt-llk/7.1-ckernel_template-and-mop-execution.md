## Purpose and Scope

This document describes the `ckernel_template` system for programming Micro-Operation Programs (MOPs) that execute repetitive instruction sequences on Tensix hardware. MOPs enable efficient looping constructs by recording instruction sequences into hardware replay buffers, which can then be executed with minimal overhead. This page focuses on template creation, configuration, and execution patterns. For information about address modifiers used within MOPs, see [Address Modifiers (ADDR_MOD_0-7)](https://deepwiki.com/tenstorrent/tt-llk/7.2-address-modifiers-(addr_mod_0-7)). For replay buffer mechanics, see [Replay Buffers and Instruction Recording](https://deepwiki.com/tenstorrent/tt-llk/7.3-replay-buffers-and-instruction-recording).

## Overview

**Micro-Operation Programs (MOPs)** are hardware-level loop constructs that execute pre-programmed instruction sequences stored in replay buffers. Instead of issuing individual TTI (Tensix Tensor Instruction) commands repeatedly, MOPs allow kernels to record a sequence once and replay it multiple times with varying address counters and loop iterations.

The `ckernel_template` class provides a C++ abstraction for creating, configuring, and executing MOPs across the three pipeline stages:

*   **Unpacker** (TRISC0): Uses `ckernel_unpack_template`
*   **Math** (TRISC1): Uses `ckernel_template`
*   **Packer** (TRISC2): Uses `ckernel_template`

### Key Concepts

| Concept | Description |
| --- | --- |
| **Outer Loop** | Number of major iterations (e.g., faces in a tile) |
| **Inner Loop** | Number of minor iterations within each outer iteration (e.g., rows in a face) |
| **Replay Buffer** | Hardware buffer storing instruction sequences |
| **Instruction Slots** | Positions for start_op, loop instructions, end_op, etc. |
| **Address Modifiers** | ADDR_MOD_0 through ADDR_MOD_7 control counter increments |

## Template Configuration Methods

### Instruction Slot Customization

Templates provide methods to customize instruction placement at specific points in the loop execution:

| Method | Purpose | Typical Use Case |
| --- | --- | --- |
| `set_start_op()` | Instruction before first iteration | Initialize counters, zero sources |
| `set_end_op()` | Instruction after last iteration | Clear source registers |
| `set_end_ops()` | Two instructions after last iteration | Reset counters + increment Z/W |
| `set_last_inner_loop_instr()` | Last instruction of inner loop | Increment fidelity phase |
| `set_last_outer_loop_instr()` | Last instruction of outer loop | Reset fidelity + clear sources |

## MOP Execution Flow

### Programming Phase: `program()`

The `program()` method writes the configured instruction sequence into the hardware replay buffer. This is a one-time setup operation performed during initialization.

### Execution Phase: `run()`

The `run()` method triggers MOP execution, causing the hardware to replay the stored instruction sequence according to the configured loop structure.

## Interaction with Hardware Synchronization

MOPs execute alongside synchronization primitives to coordinate between pipeline stages:

The typical execution sequence:

1.   **Wait for context**: `wait_for_next_context(2)` ensures hardware configuration slots are available
2.   **Configure addresses**: Write L1 memory addresses to configuration registers
3.   **Post semaphore**: Signal that configuration is complete
4.   **Stall for config**: Wait for configuration writes to propagate
5.   **Execute MOP**: `ckernel_template::run()` executes the replay buffer
6.   **Release context**: `t6_semaphore_get()` releases configuration context

## Performance Considerations

### Replay Buffer Length vs Instruction Count

Longer replay buffers reduce loop overhead but consume limited replay buffer memory:

| Operation | Replay Buffer Length | Rationale |
| --- | --- | --- |
| Simple Unpacker | 0 (inline) | Minimal instructions, no replay needed |
| Matmul (Standard) | 16 | Complex face reordering requires sequence |
| Matmul (Throttled) | 10-16 | Varies by throttle level |
| Untilize | 5-6 | Address counter manipulation per row |
| Tilize | 9 (Blackhole) | Multi-channel counter coordination |

### Inner vs Outer Loop Trade-offs

Outer loops allow `set_last_outer_loop_instr()` for cleanup between major iterations, while inner loops with `set_last_inner_loop_instr()` enable fidelity phase progression in high-fidelity math operations.
