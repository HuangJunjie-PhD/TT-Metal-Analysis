## Purpose and Scope

This document describes how the Device Management Controller (DMC) manages chip resets and bootrom loading using the JTAG interface. The DMC is responsible for orchestrating all reset operations on Blackhole ASIC chips, including loading bootrom code via JTAG, managing reset sequences, and implementing recovery mechanisms when the System Management Controller (SMC) firmware becomes unresponsive.

For information about the DMC's main event loop and how reset events are processed, see [DMC Main Loop and Event Processing](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.1-dmc-main-loop-and-event-processing). For details on how the SMC requests resets from the DMC, see [CM2DM Communication Protocol](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.2-cm2dm-communication-protocol).

## Reset Types and Triggers

The DMC supports multiple reset mechanisms, each triggered by different events and handled through the event system defined in [lib/tenstorrent/event.h](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/event.h) The reset level is specified using the `Cm2DmResetLevel` enum:

| Reset Level | Value | Description | Scope |
| --- | --- | --- | --- |
| `kCm2DmResetLevelAsic` | 0 | Reset only the ASIC and ARC cores | Chip-level reset |
| `kCm2DmResetLevelDmc` | 3 | Reset both ASIC and DMC firmware | System-level reset |

### Reset Request Processing

The DMC receives reset requests from the SMC via the CM2DM message protocol. The `reset_dm_handler()` function processes these requests:

**Message Handler Registration**

The handler is registered to process `TT_SMC_MSG_TRIGGER_RESET` (0x88) messages from the host:

`REGISTER_MESSAGE(TT_SMC_MSG_TRIGGER_RESET, reset_dm_handler);`
**Reset Request Flow**

1.   Host sends `TT_SMC_MSG_TRIGGER_RESET` message to SMC
2.   SMC processes message and calls `reset_dm_handler()` in [lib/tenstorrent/bh_arc/cm2dm_msg.c 183-202](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/cm2dm_msg.c#L183-L202)
3.   Handler validates reset level and starts 5ms timer to allow response to be sent
4.   Timer callback invokes `IssueChipReset()` which posts a CM2DM message
5.   DMC's main loop receives the message via `process_reset_req()` in [app/dmc/src/main.c 101-120](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L101-L120)
6.   DMC executes `bh_chip_reset_chip()` based on reset level

## Bootrom Loading Process

The bootrom is a small firmware image loaded into the ARC processor's instruction memory (ICCM) via JTAG. This bootrom initializes the ARC and prepares it to execute the main SMC firmware from SPI flash.

### Bootrom Loading Sequence

Sources: [lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 86-134](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L86-L134)[lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 191-225](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L191-L225)[lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 269-313](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L269-L313)

### Bootrom Code Source

The bootrom binary is embedded in the DMC firmware image at compile time. The binary is generated from a separate build process and included via preprocessor directive:

`__aligned(sizeof(uint32_t)) static const uint8_t bootcode[] = {#include "bootcode.h"};`
The bootcode is patched to ICCM starting at offset `0x80` (not address 0x0) to preserve certain memory regions. The total length is obtained via `get_bootcode_len()` which returns `sizeof(bootcode) / sizeof(uint32_t)`.

Sources: [lib/tenstorrent/jtag_bootrom/reset.c 15-30](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/reset.c#L15-L30)[include/tenstorrent/jtag_bootrom.h 22-38](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/jtag_bootrom.h#L22-L38)

## Reset Sequence Implementation

The reset sequence coordinates multiple subsystems to safely reset the ASIC while maintaining proper state. The sequence varies based on the reset trigger but follows a common pattern.

### Core Reset Functions

### Strapping Configuration

Strapping GPIOs configure the ASIC boot mode during reset. The straps are set to `GPIO_OUTPUT_ACTIVE` during the reset sequence and released to `GPIO_INPUT` (high-impedance) after the ASIC samples them.

The `bh_straps` structure defines four strapping pins:

*   `gpio6` - Boot mode strap 0
*   `gpio38` - Boot mode strap 1
*   `gpio39` - Boot mode strap 2
*   `gpio40` - Boot mode strap 3

The strapping functions include I2C bus recovery logic to handle cases where the I2C mux may conflict with GPIO configuration.

Sources: [include/tenstorrent/bh_chip.h 20-26](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_chip.h#L20-L26)[lib/tenstorrent/bh_chip/strapping.c 12-62](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/strapping.c#L12-L62)

### Bus Transfer Cancellation

Before and after reset operations, the DMC manages SMBus transfer cancellation to prevent communication errors:

**Before Reset:**

`bh_chip_cancel_bus_transfer_set(chip);`
Calls `smbus_cancel()` to abort any pending SMBus operations. This is critical when the ARC is hanging or being reset.

**After Reset:**

`bh_chip_cancel_bus_transfer_clear(chip);`
Calls `smbus_uncancel()` to resume normal SMBus operations once the ARC is ready.

## Recovery Mechanisms

The DMC implements multiple autonomous recovery mechanisms to handle failure scenarios without host intervention. These mechanisms monitor chip health and automatically trigger resets when necessary.

### Watchdog Timer Recovery

Sources: [lib/tenstorrent/bh_chip/bh_chip.c 128-136](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L128-L136)[app/dmc/src/main.c 419-449](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L419-L449)[app/dmc/src/main.c 168-182](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L168-L182)

**Watchdog Flow:**

1.   Host sets watchdog timeout via `TT_SMC_MSG_SET_WDT` (0xC1) message
2.   SMC processes message and posts `kCm2DmMsgIdAutoResetTimeoutUpdate` to DMC
3.   DMC starts `auto_reset_timer` with the specified timeout
4.   SMC periodically updates telemetry heartbeat via `kCm2DmMsgTelemHeartbeatUpdate`
5.   Each heartbeat update restarts the timer
6.   If timer expires, `bh_chip_auto_reset()` callback fires: 
    *   Sets `arc_wdog_triggered` flag
    *   Cancels bus transfers (ARC likely hung)
    *   Posts `TT_EVENT_WATCHDOG_EXPIRED` event

7.   DMC main loop processes event via `handle_watchdog_reset()`: 
    *   Uses JTAG to read ARC program counter at `RESET_UNIT_ARC_PC_CORE_0` (0x80030C00)
    *   Stores PC in `chip->data.arc_hang_pc` for post-mortem debugging
    *   Clears `auto_reset_timeout` to prevent repeated resets
    *   Forces fan to 100% speed
    *   Calls `bh_chip_reset_chip()` to perform full reset

The stored `arc_hang_pc` value is later sent to the SMC via `dmStaticInfo` structure so it can be exposed in telemetry for debugging.

Sources: [app/dmc/src/main.c 419-449](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L419-L449)[lib/tenstorrent/bh_chip/bh_chip.c 78-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L78-L88)

### Thermal Trip Recovery

Thermal trip events trigger immediate chip resets when the ASIC's thermal protection activates. The `therm_trip` GPIO is monitored via interrupt:

**Thermal Trip Handler:**

`void therm_trip_detected(const struct device *dev, struct gpio_callback *cb, uint32_t pins){    struct bh_chip *chip = CONTAINER_OF(cb, struct bh_chip, therm_trip_cb);        chip->data.therm_trip_triggered = true;    bh_chip_cancel_bus_transfer_set(chip);    tt_event_post(TT_EVENT_THERM_TRIP);}`
The main loop handles the event in `handle_therm_trip()`:

*   Cancels bus transfers
*   Sets board fault LED
*   Forces fan to 100% speed
*   Increments `therm_trip_count`
*   Performs chip reset via `bh_chip_reset_chip()`

The therm trip count is tracked persistently and sent to the SMC so it can be monitored in telemetry.

Sources: [lib/tenstorrent/bh_chip/bh_chip.c 192-224](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L192-L224)[app/dmc/src/main.c 375-417](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L375-L417)

### PERST (PCIe Reset) Handling

On boards with PCIe, the `preset_trigger` GPIO monitors the PCI Express reset signal. When PERST is asserted, the DMC must reload bootrom and reset the ASIC:

**PERST GPIO Callback:**

`void gpio_asic_reset_callback(const struct device *port, struct gpio_callback *cb, uint32_t pins){    perst_seen = true;    perst_start_time = k_cycle_get_32();        ARRAY_FOR_EACH_PTR(BH_CHIPS, chip) {        atomic_set(&chip->data.trigger_reset, true);        if (!chip->data.performing_reset) {            bh_chip_cancel_bus_transfer_set(chip);        }    }    tt_event_post(TT_EVENT_PERST);}`
The `handle_perst()` function in main.c processes the event by calling the JTAG bootrom reset sequence. The DMC tracks timing information:

*   `perst_start_time` - Timestamp when PERST asserted (in DMC cycles)
*   `arc_start_time` - Timestamp when bootrom execution began (in ASIC refclk cycles)
*   `dm_init_duration` - Time from PERST to DMC init complete

These timestamps are sent to the SMC via `dmStaticInfo` for performance monitoring.

Sources: [lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 67-81](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L67-L81)[app/dmc/src/main.c 452-482](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L452-L482)[lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 258-267](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L258-L267)

### PGOOD (Power Good) Handling

The `pgood` GPIO monitors ASIC power rail status. When power fails, the DMC:

*   Asserts board fault LED
*   Asserts ASIC reset to prevent undefined behavior
*   Tracks trip frequency to detect severe faults

When power returns:

*   Performs full reset sequence via `bh_chip_reset_chip()`
*   Clears board fault LED (unless in assembly test mode)

If PGOOD falls twice within 1 second, a severe fault is flagged and the chip is left in reset.

Sources: [lib/tenstorrent/bh_chip/bh_chip.c 264-295](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L264-L295)[app/dmc/src/main.c 484-489](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L484-L489)

### Dirty Reset Test

The "dirty reset" test simulates a scenario where the DMC resets itself without the SMC requesting it, similar to what might occur during a NOC hang or watchdog reset:

**Test Implementation:**

`def dirty_reset_test():    timeout = 60  # seconds to wait for SMC boot        # Reset DMC via JTAG using dmc_reset script    ret = dmc_reset.reset_dmc(args)    if ret != os.EX_OK:        return False            # Wait for SMC to boot and enumerate    ret = dmc_reset.wait_for_smc_boot(timeout)    if ret != os.EX_OK:        return False            return True`
The test uses the `dmc_reset` script (similar to `scripts/dmc-reset`) which:

1.   Opens JTAG connection via OpenOCD
2.   Asserts DMC reset via JTAG commands
3.   Releases reset and waits for boot
4.   Polls for SMC chip enumeration via pyluwen

This validates the reset sequence can recover from unexpected DMC resets without host intervention.

### Watchdog Reset Test

The watchdog test validates autonomous recovery when the ARC firmware hangs:

**Test Flow:**

1.   Configure watchdog for 1000ms timeout via `TT_SMC_MSG_SET_WDT`
2.   Wait 1500ms (longer than timeout)
3.   Verify ARC still responds (watchdog hasn't triggered yet)
4.   Clear `ARC_HANG_PC_REG_ADDR` to prepare for hang detection
5.   Halt ARC cores by writing to `ARC_MISC_CTRL` register
6.   Wait 500ms - verify ARC still enumerated (grace period)
7.   Wait 600ms more - verify `arc_hang_pc` was set (watchdog triggered)
8.   Wait for PCIe re-enumeration
9.   Verify ARC boots successfully
10.   Verify DMC still responds to ping

The test confirms:

*   Watchdog timeout is properly enforced (1000ms +/- tolerance)
*   Hung PC is captured for debugging
*   Full reset sequence completes
*   System returns to operational state

Sources: [app/smc/pytest/e2e_smoke.py 551-645](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L551-L645)

### PCIe Firmware Load Time Test

This test measures the time from PERST assertion to PCIe subsystem ready:

**Timing Registers:**

*   `ARC_START_TIME_REG_ADDR` (0x80030440) - ASIC refclk timestamp when bootrom started
*   `PCIE_INIT_CPL_TIME_REG_ADDR` (0x80030438) - ASIC refclk timestamp when PCIe init completed
*   `CMFW_START_TIME_REG_ADDR` (0x8003043C) - ASIC refclk timestamp when CMFW started

The test reads these registers and calculates:

`duration_in_ms = (pcie_init_cpl_time - arc_start_time) / REFCLK_HZ * 1000`
Where `REFCLK_HZ = 50_000_000` (50 MHz ASIC reference clock).

The test asserts that firmware load time is under 40ms to ensure PCIe link training completes within spec.

Sources: [app/smc/pytest/e2e_smoke.py 647-690](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L647-L690)[app/smc/pytest/e2e_stress.py 103-131](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_stress.py#L103-L131)

*   Average DMC ping latency after reset
*   Maximum DMC ping latency after reset
*   Boot time from reset to operational
*   Consecutive failure patterns

**Example Output:**

```
INFO: tt-smi reset test completed. Failed 1/200 times.
INFO: Average DMFW ping time: 12.34 ms, Max DMFW ping time: 45.67 ms.
```

These metrics help identify performance regressions and reliability issues in reset handling.
