## Overview

The Llama fine-tuning experiments apply LoRA (Low-Rank Adaptation) to adapt pre-trained Llama models for downstream tasks. LoRA reduces the number of trainable parameters by injecting trainable rank decomposition matrices into Transformer layers, making it feasible to fine-tune large language models on resource-constrained hardware.

The implementation supports:

*   Three model sizes: Llama 3.2 1B, Llama 3.2 3B, Llama 3.1 8B
*   Multiple datasets: SST-2, Text-to-SQL, SQuAD v2
*   Single-chip and multi-chip configurations
*   Data parallel, tensor parallel, and hybrid parallelism strategies
*   Integration with HuggingFace `transformers` and `peft` libraries

**Sources:**[blacksmith/experiments/torch/llama/xla/lora/README.md 9-13](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L9-L13)[docs/src/experiments.md 15-21](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L15-L21)

### LoRA Configuration Parameters

The LoRA configuration is specified in `TrainingConfig` and controls the adaptation behavior:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `lora_r` | int | 4 | Rank of LoRA decomposition matrices (controls capacity) |
| `lora_alpha` | int | 8 | Scaling factor for LoRA updates (typically 2×rank) |
| `lora_target_modules` | list[str] | `["all-linear"]` | Target modules for LoRA injection (supports regex patterns) |
| `lora_task_type` | str | `"CAUSAL_LM"` | Training task type for LoRA configuration |

**LoRA Rank and Alpha:**

*   Lower rank (r=2-4): Fewer parameters, faster training, may limit adaptation capacity
*   Higher rank (r=8-16): More parameters, better adaptation, longer training
*   Alpha controls the magnitude of LoRA updates: `scaling = alpha / r`

**Target Modules:**

*   `["all-linear"]`: Applies LoRA to all linear layers (default)
*   Specific patterns: `["q_proj", "v_proj"]` for attention-only adaptation
*   Custom regex: `[".*attn.*"]` for all attention-related layers

**Sources:**[blacksmith/experiments/torch/llama/configs.py 63-67](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/configs.py#L63-L67)[blacksmith/models/torch/huggingface/hf_models.py 37-45](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/models/torch/huggingface/hf_models.py#L37-L45)[blacksmith/experiments/torch/llama/xla/lora/README.md 154-157](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L154-L157)

#### 2. Training Loop Implementation

The `train()` function [blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 103-216](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L103-L216) implements the core training logic:

1.   **Model Loading**: Calls `get_model()` to load Llama with LoRA
2.   **Optimizer Setup**: Creates `AdamW` optimizer with configured learning rate
3.   **Dataset Loading**: Uses `get_dataset()` with `collate_fn_for_causal_lm`
4.   **Epoch Loop**: Iterates through `config.num_epochs`
5.   **Batch Processing**: 
    *   Zero gradients with `optimizer.zero_grad()`
    *   Prepare batch with `device_manager.prepare_batch()` (data sharding)
    *   Shard model with `device_manager.shard_model()` (tensor sharding)
    *   Forward pass through `training_step_inner()`
    *   Backward pass with `loss.backward()`
    *   XLA sync if on TT hardware: `torch_xla.sync(wait=True)`
    *   Optimizer step via `device_manager.optimizer_step()`

6.   **Periodic Actions**: 
    *   Log metrics every `steps_freq` steps
    *   Run validation every `steps_freq` steps
    *   Save checkpoints based on `save_strategy`

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 103-216](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L103-L216)

### Model Sharding Patterns

The `model_sharding_patterns` configuration uses regex-based pattern matching to apply sharding specifications to model layers:

| Pattern | Modules Matched | Sharding Spec | Purpose |
| --- | --- | --- | --- |
| `\.self_attn\.q_proj\.base_layer$` | Query projection (LoRA base layer) | `["model", null]` | Column-wise sharding |
| `\.self_attn\.v_proj\.base_layer$` | Value projection (LoRA base layer) | `["model", null]` | Column-wise sharding |
| `\.self_attn\.o_proj$` | Output projection | `[null, "model"]` | Row-wise sharding |
| `\.mlp\.gate_proj$` | MLP gate projection | `["model", null]` | Column-wise sharding |
| `\.mlp\.up_proj$` | MLP up projection | `["model", null]` | Column-wise sharding |
| `\.mlp\.down_proj$` | MLP down projection | `[null, "model"]` | Row-wise sharding |

**Column-wise sharding** (`["model", null]`): Splits weight matrix along the output dimension. Used for projections that expand dimensionality.

**Row-wise sharding** (`[null, "model"]`): Splits weight matrix along the input dimension. Used for projections that reduce dimensionality.

**Note**: Patterns target LoRA base layers (`.base_layer$`) to shard the original pretrained weights, not the LoRA adaptation matrices.

**Sources:**[blacksmith/experiments/torch/llama/xla/lora/README.md 27-34](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L27-L34)[blacksmith/experiments/torch/llama/configs.py 80-82](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/configs.py#L80-L82)

### Data Collation: collate_fn_for_causal_lm

The training pipeline uses `collate_fn_for_causal_lm`[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 128-129](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L128-L129) to prepare batches for causal language modeling:

1.   **Tokenization**: Converts text to input IDs using model tokenizer
2.   **Padding**: Pads sequences to `max_length`
3.   **Attention Masks**: Creates masks for padded tokens
4.   **Label Shifting**: Shifts labels for next-token prediction
5.   **Label Transformation**: Converts labels to one-hot encoding (for TT hardware compatibility)

The batch dictionary contains:

*   `input_ids`: Tokenized input sequences
*   `attention_mask`: Mask for padding tokens
*   `labels`: Target tokens (shifted for causal LM)
*   `expected_output`: One-hot encoded labels (after transform)
*   `labels_mask`: Mask for ignored indices

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 128-156](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L128-L156)

## Implementation Details and Best Practices

### Memory Optimization: training_step_inner

The training step is extracted into a separate function to manage memory efficiently:

**Purpose**: Large vocabulary-sized tensors (32K+ vocab) consume significant memory. Keeping logits scoped locally prevents them from propagating beyond the step via the computation graph, avoiding unnecessary and expensive collective communications (CCLs) in multi-chip setups.

**Key Points**:

*   Returns `loss.detach()` to break computation graph
*   Logits are garbage collected after function returns
*   Critical for multi-chip configurations with large models
*   Reduces memory pressure and communication overhead

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 89-101](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L89-L101)

### XLA Synchronization Points

The training loop includes explicit synchronization for Tenstorrent hardware:

**Synchronization Points**:

1.   After backward pass: Ensures gradients are computed before optimizer step
2.   After optimizer step: Ensures parameter updates complete before next iteration
3.   During validation: Ensures all predictions are computed before collecting results

**Purpose**: XLA defers execution to enable graph optimization. `sync(wait=True)` forces immediate execution and data transfer back to host, necessary for:

*   Accurate loss monitoring
*   Checkpoint saving
*   Validation metrics
*   WandB logging

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 165-167](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L165-L167)[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 191-193](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L191-L193)

### Computation Cache Management

After validation, the XLA computation cache is cleared:

**Purpose**: XLA caches compiled computation graphs. During training with periodic validation, different graph structures accumulate in memory. Clearing the cache prevents memory leaks and ensures consistent performance.

**When to Clear**:

*   After validation (different graph than training)
*   After checkpoint loading
*   When switching between train/eval modes frequently

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 191-193](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L191-L193)

### Label Transformation for TT Hardware

The training loop includes a workaround for label handling:

**Purpose**: Converts sparse labels to one-hot encoded format and creates a mask for ignored indices. This is a temporary workaround for TT hardware compatibility.

**Implementation Note**: See issue [#327](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/#327) for planned refactoring.

**Sources:**[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 147-156](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L147-L156)

## Troubleshooting and Common Issues

### Model Size Does Not Fit on Single Chip

**Error**: OOM (Out of Memory) or allocation failure on single chip

**Solution**:

*   Use multi-chip configuration with tensor parallelism
*   Llama 3B requires `mesh_shape: [1, 4]` minimum
*   Llama 8B requires model dimension > 1

**Sources:**[blacksmith/experiments/torch/llama/xla/lora/README.md 62-85](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L62-L85)

### Mesh Configuration Errors

**Error**: Invalid mesh shape or axis names

**Requirements**:

*   For Llama 8B: model dimension must be > 1
*   If one dimension is 1, it must be the data dimension
*   Axis names must match mesh shape dimensions
*   Valid orderings: `["data", "model"]` or `["model", "data"]`

**Example Fix**:

**Sources:**[blacksmith/experiments/torch/llama/xla/lora/README.md 84](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L84-L84)

### Sharding Pattern Mismatches

**Error**: Layers not sharded correctly or performance degradation

**Solution**:

*   Verify regex patterns match actual module names
*   Check LoRA layers use `.base_layer$` suffix
*   Ensure column-wise and row-wise sharding are balanced
*   Use `model.named_modules()` to inspect actual layer names
