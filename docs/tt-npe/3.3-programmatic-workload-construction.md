## Purpose and Scope

This page documents how to construct workloads programmatically using the tt-npe Python API. It covers the object model exposed by the `tt_npe_pybind` module, the construction workflow, and detailed examples of building workloads from scratch in Python code.

For information about the underlying workload data model and validation rules, see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For creating workloads from JSON files, see [Creating Workloads from JSON](https://deepwiki.com/tenstorrent/tt-npe/3.2-creating-workloads-from-json). For converting NoC traces to workloads, see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads).

## Configuration: npe.Config

The `npe.Config` class contains all simulation parameters. All fields have default values.

### Constructor

`cfg = npe.Config()`
### Key Configuration Fields

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `device_name` | `str` | `"wormhole_b0"` | Device to simulate: `"wormhole_b0"`, `"blackhole"`, `"N150"`, `"N300"`, `"T3K"`, `"P100"`, `"P150"` |
| `congestion_model_name` | `str` | `"fast"` | Congestion model: `"none"` or `"fast"` |
| `cycles_per_timestep` | `int` | `128` | Number of cycles per simulation timestep |
| `infer_injection_rate_from_src` | `bool` | `True` | Automatically infer injection rate based on source core type |
| `emit_timeline_file` | `bool` | `False` | Generate timeline JSON for visualization |
| `timeline_filepath` | `str` | `""` | Path for timeline output file |
| `compress_timeline_output_file` | `bool` | `False` | Compress timeline output using zstd |
| `use_legacy_timeline_format` | `bool` | `False` | Use v0 timeline format instead of v1 |
| `scale_workload_schedule` | `float` | `0.0` | Scale factor for workload schedule (experimental) |
| `topology_json` | `str` | `""` | Path to topology JSON for multi-chip simulation |

### Verbosity Control

`cfg.set_verbosity_level(level)  # 0 (normal), 1 (verbose), 2 (more), 3 (most)`

## Coordinate Types

### npe.Coord

Represents a single coordinate as a tuple of `{device_id, row, col}`. **Note:** This is NOT an `{x, y}` pair; it uses row/column indexing.

`# Constructorcoord = npe.Coord(device_id, row, col) # Fieldscoord.device_id  # int: Device ID (0 for single-chip)coord.row        # int: Physical row indexcoord.col        # int: Physical column index`
### npe.MulticastCoordSet

Represents a multicast destination as one or more rectangular grids of coordinates.

`# Constructor for empty setmcast = npe.MulticastCoordSet() # Constructor for single gridstart = npe.Coord(0, 1, 1)end = npe.Coord(0, 4, 6)mcast = npe.MulticastCoordSet(start, end) # Query sizetotal_cores = mcast.grid_size()  # Returns total number of coordinates`
### npe.NocType

Enum specifying which NoC is used for a transfer.

`npe.NocType.NOC_0  # NoC 0npe.NocType.NOC_1  # NoC 1`

## Transfer Construction: npe.Transfer

A `npe.Transfer` represents a contiguous series of packets from a single source to one or more destinations, equivalent to a single `noc_async_read()` or `noc_async_write()` call in tt-metal.

### Constructor Signature

`transfer = npe.Transfer(    packet_size,          # int: Size of each packet in bytes    num_packets,          # int: Number of packets (≥1)    src,                  # npe.Coord: Source location    dst,                  # npe.Coord | npe.MulticastCoordSet: Destination(s)    injection_rate,       # float: Injection rate in GB/s (0 for auto-infer)    phase_cycle_offset,   # int: Earliest cycle this transfer can start    noc_type             # npe.NocType: NOC_0 or NOC_1)`
### Field Constraints and Guidelines

| Parameter | Constraint | Notes |
| --- | --- | --- |
| `packet_size` | Must be > 0 | In bytes. For best accuracy, split large transfers into max-sized packets (e.g., 8KB on Wormhole) |
| `num_packets` | Must be ≥ 1 | Multiple packets are transferred back-to-back in series |
| `src` | Valid coordinate on device | Source is where data flows FROM (for both reads and writes) |
| `dst` | Valid coordinate(s) on device | Can be `npe.Coord` (unicast) or `npe.MulticastCoordSet` (multicast) |
| `injection_rate` | ≥ 0 | Set to 0 to auto-infer from source core type; otherwise in GB/s |
| `phase_cycle_offset` | ≥ 0 | Relative to containing phase's start; actual start may be later due to congestion |
| `noc_type` | `NOC_0` or `NOC_1` | Selects which physical NoC to use |

### Unicast Transfer Example

`src = npe.Coord(0, 2, 3)  # Device 0, row 2, col 3dst = npe.Coord(0, 5, 7)  # Device 0, row 5, col 7 transfer = npe.Transfer(    packet_size=8192,           # 8KB packet    num_packets=1,              # Single packet    src=src,    dst=dst,    injection_rate=0,           # Auto-infer    phase_cycle_offset=100,     # Start at cycle 100 or later    noc_type=npe.NocType.NOC_0)`
### Multicast Transfer Example

`src = npe.Coord(0, 1, 1)dst_start = npe.Coord(0, 3, 3)dst_end = npe.Coord(0, 6, 8)mcast_dst = npe.MulticastCoordSet(dst_start, dst_end) transfer = npe.Transfer(    packet_size=4096,    num_packets=2,              # Two packets back-to-back    src=src,    dst=mcast_dst,              # Multicast to 4x6 grid    injection_rate=0,    phase_cycle_offset=0,    noc_type=npe.NocType.NOC_1)`
**Sources:**[tt_npe/cpp/pybind/bindings.cpp 246-275](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L246-L275)[tt_npe/py/examples/programmatic_workload_generation.py 67-71](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/examples/programmatic_workload_generation.py#L67-L71)

## Phase Construction: npe.Phase

A `npe.Phase` is a logical grouping of transfers that have no mutual dependencies. Phases execute sequentially, but transfers within a phase can overlap.

### Creating a Phase

`phase = npe.Phase()  # Create empty phase`
### Adding Transfers

`# Add transfers one at a timephase.addTransfer(transfer1)phase.addTransfer(transfer2)phase.addTransfer(transfer3)`
### Phase Semantics

Transfers within a phase can overlap in execution. The `phase_cycle_offset` field specifies the earliest possible start time relative to the phase's beginning, but actual start times depend on congestion and NoC availability.

**Sources:**[tt_npe/cpp/pybind/bindings.cpp 277-289](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L277-L289)

## Workload Construction: npe.Workload

A `npe.Workload` is the top-level container for all transfers to be simulated. It consists of one or more phases.

### Creating a Workload

`workload = npe.Workload()  # Create empty workload`
### Adding Phases

`# Add phases in execution orderworkload.addPhase(phase1)workload.addPhase(phase2)workload.addPhase(phase3)`
### Setting Golden Result Cycles (Optional)

For validation purposes, you can provide known execution times:

`golden_cycles = {    0: (start_cycle, end_cycle),  # Device 0    1: (start_cycle, end_cycle),  # Device 1 (if multi-chip)}workload.setGoldenResultCycles(golden_cycles)`
**Sources:**[tt_npe/cpp/pybind/bindings.cpp 291-308](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L291-L308)

## API Initialization and Execution

### Initializing the API

`npe_api = npe.InitAPI(cfg)if npe_api is None:    # Initialization failed (invalid config)    print("Failed to initialize NPE API")    sys.exit(1)`
The `InitAPI()` function returns `Optional[npe.API]`:

*   Returns `npe.API` handle if successful
*   Returns `None` if configuration is invalid

### Running Simulation

`result = npe_api.runNPE(workload) # Check result typematch type(result):    case npe.Stats:        # Simulation succeeded        print(f"Completed in {result.estimated_cycles} cycles")        print(f"Link utilization: {result.overall_avg_link_util}")    case npe.Exception:        # Simulation failed        print(f"Error: {result.err()}")`
The `runNPE()` method returns a `Union[npe.Stats, npe.Exception]`:

*   `npe.Stats` if simulation completes successfully
*   `npe.Exception` if an error occurs (timeout, malformed workload, etc.)

**Sources:**[tt_npe/cpp/pybind/bindings.cpp 18-37](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L18-L37)[tt_npe/py/examples/programmatic_workload_generation.py 87-100](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/examples/programmatic_workload_generation.py#L87-L100)

## Advanced Patterns

### Multiple Phases for Dependency Modeling

Use multiple phases when transfers have clear sequential dependencies:

`workload = npe.Workload() # Phase 1: Initial data distributionphase1 = npe.Phase()for i in range(10):    phase1.addTransfer(create_distribution_transfer(i))workload.addPhase(phase1) # Phase 2: Compute phase (depends on phase 1 completion)phase2 = npe.Phase()for i in range(20):    phase2.addTransfer(create_compute_transfer(i))workload.addPhase(phase2) # Phase 3: Gather results (depends on phase 2 completion)phase3 = npe.Phase()for i in range(10):    phase3.addTransfer(create_gather_transfer(i))workload.addPhase(phase3)`
### Multicast Pattern

When broadcasting data to multiple destinations:

`# Define source and destination gridsrc = npe.Coord(0, 0, 1)  # DRAM locationdest_grid_start = npe.Coord(0, 1, 1)dest_grid_end = npe.Coord(0, 8, 10)mcast_set = npe.MulticastCoordSet(dest_grid_start, dest_grid_end) # Create multicast transfertransfer = npe.Transfer(    packet_size=8192,    num_packets=4,    src=src,    dst=mcast_set,    injection_rate=0,    phase_cycle_offset=0,    noc_type=npe.NocType.NOC_1)`
### Manual Injection Rate Control

To override automatic injection rate inference:

`cfg = npe.Config()cfg.infer_injection_rate_from_src = False  # Disable auto-inference # Now manually specify injection ratestransfer = npe.Transfer(    packet_size=4096,    num_packets=1,    src=src,    dst=dst,    injection_rate=25.0,  # 25 GB/s explicit rate    phase_cycle_offset=0,    noc_type=npe.NocType.NOC_0)`

## Key Considerations

### Source and Destination Semantics

The `src` coordinate always represents where data flows FROM:

*   For a **WRITE** operation: `src` is the Tensix core initiating the write
*   For a **READ** operation: `src` is the memory location being read from (the initiating core is `dst`)

This matches the data flow direction in the NoC, not necessarily the initiating core.

### Packet Size Guidelines

For accurate modeling:

*   Break large transfers into maximum-sized packets (8KB on Wormhole B0)
*   Example: A 20KB transfer should be modeled as 3 packets: [8KB, 8KB, 4KB]
*   This accounts for per-packet overheads in the simulation

### Phase Cycle Offsets

The `phase_cycle_offset` field specifies the earliest possible start time:

*   Actual start may be delayed by congestion or NoC contention
*   Offsets are relative to the containing phase's start cycle
*   Use offset 0 if transfers should start as soon as possible

### Injection Rate Inference

When `cfg.infer_injection_rate_from_src = True` (default):

*   Rates are automatically determined based on source core type
*   DRAM cores use DRAM bandwidth limits
*   Worker cores use Tensix bandwidth limits
*   ETH cores use Ethernet bandwidth limits
*   Inference happens during workload preprocessing in `npeAPI::preprocessWorkload()`
