This page documents the mathematical fidelity and precision control mechanisms in the Low-Level Kernel (LLK) library. Math fidelity determines the accuracy of operations by controlling the number of iterative refinement phases, while precision control governs the bit-width and format of accumulation.

For general information about math operations, see [Math Unit Operations](https://deepwiki.com/tenstorrent/tt-llk/5-math-unit-operations). For SFPU-specific operations, see [SFPU Operations Library](https://deepwiki.com/tenstorrent/tt-llk/5.3-sfpu-operations-library). For matrix multiplication details, see [Matrix Multiplication Operations](https://deepwiki.com/tenstorrent/tt-llk/5.4-matrix-multiplication-operations).

## Overview and Purpose

The LLK library provides multiple levels of mathematical accuracy and precision control to balance between computational throughput and numerical accuracy:

*   **Math Fidelity Levels**: Control the number of iterative refinement phases (0-3 iterations) for operations like matrix multiplication
*   **Precision Modes**: Toggle between 16-bit and 32-bit destination accumulation
*   **Approximation Modes**: Enable fast approximations for certain SFPU operations
*   **Throttling**: Reduce compute throughput to control power and thermal behavior

These mechanisms are implemented through hardware address modifiers, instruction sequencing, and mantissa masking in the golden model validation system.

## MathFidelity Enumeration and Modes

**MathFidelity Mode Mapping**

| Mode | Iterations | Throughput | Use Case | Hardware Phases |
| --- | --- | --- | --- | --- |
| `LoFi` | 0 | 100% | Maximum throughput, acceptable accuracy | Single pass with mask[0] |
| `HiFi2` | 1 | ~50% | Balanced accuracy/performance | Two passes with masks[0,1] |
| `HiFi3` | 2 | ~33% | High accuracy requirements | Three passes with masks[0,1,2] |
| `HiFi4` | 3 | ~25% | Maximum accuracy | Four passes with masks[0,1,2,3] |

The `is_high_fidelity` helper function determines whether fidelity iteration is needed:

`// HiFi2, HiFi3, HiFi4 are considered high fidelityconstexpr bool high_fidelity = is_high_fidelity(math_fidelity);constexpr std::uint32_t fidelity_increment = high_fidelity ? 1 : 0;`

## Fidelity Implementation in Matrix Multiplication

### Address Modifier Configuration

The hardware uses address modifiers to control fidelity phase progression. Each modifier specifies how source/destination register addresses and the fidelity counter are updated after instructions execute.

The `matmul_configure_addrmod` function sets up these address modifiers with fidelity control:

`// ADDR_MOD_3: Same as ADDR_MOD_0 but with bias increment for fidelity accumulationaddr_mod_t {    .srca = {.incr = 0, .clr = 0, .cr = 0},    .srcb = {.incr = 8, .clr = 0, .cr = 0},    .dest = {.incr = 8, .clr = 0, .cr = 0},    .bias = {.incr = 1},}.set(ADDR_MOD_3); // ADDR_MOD_5: Reset all counters, increment fidelity phaseaddr_mod_t {    .srca     = {.incr = 0, .clr = 1, .cr = 1},    .srcb     = {.incr = 0, .clr = 1, .cr = 1},    .dest     = {.incr = 0, .clr = 1, .cr = 1},    .fidelity = {.incr = fidelity_increment, .clr = 0},    .bias     = {.incr = 1},}.set(ADDR_MOD_5);`

### MOP Programming for Fidelity Iterations

The Micro-Operation (MOP) programming differs based on fidelity level. The `matmul_configure_mop` function programs the replay buffer with the appropriate instruction sequence.

The MOP template configuration:

`constexpr std::uint32_t inner_loops = high_fidelity ? to_underlying(math_fidelity) : 1;ckernel_template tmp(1 /* outer loop */, inner_loops,                      lltt::replay_insn(ckernel::math::replay_buf_offset, replay_buf_len)); if constexpr (high_fidelity) {    // Reset fidelity phase after all iterations    tmp.set_end_op(TT_OP_SETRWC(p_setrwc::CLR_NONE, 0, 0, 0, 0, p_setrwc::SET_F));}tmp.program();`

## Fidelity Masking in Golden Models

The golden model system simulates hardware fidelity behavior by applying mantissa masks to operands before computation. This ensures test validation matches the actual hardware precision.

### Mantissa Masking Mechanism

**Fidelity Mask Tables**

For Wormhole B0 and Blackhole architectures:

`FP_FIDELITY_ITER_MASK = [    (0b11111000000, 0b11111110000),  # Iteration 0: 5 MSBs of A, 7 MSBs of B    (0b00000111110, 0b11111110000),  # Iteration 1: 4 LSBs of A, 7 MSBs of B    (0b11111000000, 0b00000001111),  # Iteration 2: 5 MSBs of A, 4 LSBs of B    (0b00000111110, 0b00000001111),  # Iteration 3: 4 LSBs of A, 4 LSBs of B]`
For Quasar architecture:

`FP_FIDELITY_ITER_MASK = [    (0b11111111000, 0b11111111000),  # Iteration 0: 8 MSBs of both    (0b00000000111, 0b11111111000),  # Iteration 1: 3 LSBs of A, 8 MSBs of B    (0b11111111000, 0b00000000111),  # Iteration 2: 8 MSBs of A, 3 LSBs of B    (0b00000000111, 0b00000000111),  # Iteration 3: 3 LSBs of both]`

### MatmulGolden Fidelity Implementation

The `MatmulGolden` class demonstrates how fidelity masking is applied for each mode:

The implementation iteratively applies masks and accumulates results:

`MATH_FIDELITY_TO_ITER_COUNT = {    MathFidelity.LoFi: 0,    MathFidelity.HiFi2: 1,    MathFidelity.HiFi3: 2,    MathFidelity.HiFi4: 3,} fidelity_iter_count = MATH_FIDELITY_TO_ITER_COUNT[math_fidelity]res = 0 if fidelity_iter_count == 0:    # LoFi: Single iteration with mask[0]    t1, t2 = self._apply_fidelity_masking(data_format, t1, t2, 0)    res = torch.matmul(t1.view(M, K1), t2.view(K2, N))elif fidelity_iter_count > 0:    # HiFi2/HiFi3: Accumulate masked results    for i in range(fidelity_iter_count + 1):        t1 = to_tensor(operand1, data_format)        t2 = to_tensor(operand2, data_format)        t1, t2 = self._apply_fidelity_masking(data_format, t1, t2, i)        res += torch.matmul(t1.view(M, K1), t2.view(K2, N))`

## Destination Accumulation and Precision Control

### DestAccumulation Mode

The `DestAccumulation` parameter controls whether the destination registers use 32-bit or 16-bit precision:

**Destination Register Capacity Impact**

| Mode | Precision | Tiles per Half (16-bit format) | Tiles per Half (32-bit format) | Use Case |
| --- | --- | --- | --- | --- |
| `DestAccumulation::No` | 16-bit | 8 | 4 | Standard operations, maximum throughput |
| `DestAccumulation::Yes` | 32-bit | 4 | 2 | High precision accumulation, reduced capacity |

Code example from test sources:

`const int MAX_TILES_DEST = is_fp32_dest_acc_en ?     (BIT32_DEST_REGISTER_HALF_SIZE / (TILE_NUM_FACES * FACE_R_DIM)) :     (DEST_REGISTER_HALF_SIZE / (TILE_NUM_FACES * FACE_R_DIM));`

### ApproximationMode

The `ApproximationMode` enum controls whether SFPU operations use fast approximations:

**Exponential Clamping Behavior**

The `CLAMP_NEGATIVE` parameter specifically affects exponential operations:

*   `CLAMP_NEGATIVE=true`: Large negative inputs (< -88) are clamped to produce zero
*   `CLAMP_NEGATIVE=false`: Large negative inputs produce negative values (hardware-dependent behavior)

Example test configuration:

`configuration = TestConfig(    templates=[        APPROX_MODE(ApproximationMode.Yes),        FAST_MODE(FastMode.Yes),        CLAMP_NEGATIVE(clamp_negative),        MATH_OP(mathop=MathOperation.Exp),    ])`

## Throttling and Throughput Control

Throttling reduces matrix multiplication throughput to control power consumption and thermal behavior. Five throttling levels are supported.

### Throttle Level Configuration

**Throttle Level Throughput Table**

| Level | Throughput | NOP Pattern | Use Case |
| --- | --- | --- | --- |
| 0 | 100% | None | Maximum performance |
| 1 | 73% | 1 NOP between some MVMUL | Light throttling |
| 2 | 67% | 2 NOPs between some MVMUL | Moderate throttling |
| 3 | 50% | 1 NOP between each MVMUL | Balanced power/performance |
| 4 | 40% | 2 NOPs between each MVMUL | Heavy throttling |
| 5 | 33% | 2-3 NOPs between each MVMUL | Maximum power reduction |

### Throttled Instruction Sequencing

The `run_throttled_sequence` function inserts NOPs to slow down computation:

`template <int THROTTLE_LEVEL, bool high_fidelity>void run_throttled_sequence(const std::uint32_t t_dim, const bool reuse_a) {    if constexpr (THROTTLE_LEVEL == 1) {        TTI_NOP;        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_0, 0);        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_1, 0);        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_0, 0);        TTI_NOP;        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_2, 0);        // ... additional pattern    }    else if constexpr (THROTTLE_LEVEL == 5) {        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_3, 0);        TTI_NOP;        TTI_NOP;        TTI_MVMUL(p_setrwc::CLR_NONE, 0, ADDR_MOD_0, 0);        TTI_NOP;        TTI_NOP;        // ... maximum NOP insertion    }}`
Throttled configuration requires additional address modifier setup:

`if constexpr (THROTTLE_LEVEL) {    // ADDR_MOD_6: Complete reset including fidelity counter    addr_mod_t {        .srca     = {.incr = 0, .clr = 1, .cr = 1},        .srcb     = {.incr = 0, .clr = 1, .cr = 1},        .dest     = {.incr = 0, .clr = 1, .cr = 1},        .fidelity = {.incr = 0, .clr = 1},        .bias     = {.incr = 1},    }.set(ADDR_MOD_6);}`

## Architecture-Specific Fidelity Differences

### Wormhole B0, Blackhole, and Quasar Variations

Different architectures use different fidelity mask patterns to optimize for their hardware characteristics:

The golden model checks architecture and applies the appropriate masks:

`if get_chip_architecture() == ChipArchitecture.QUASAR:    FP_FIDELITY_ITER_MASK = [        (0b11111111000, 0b11111111000),        (0b00000000111, 0b11111111000),        (0b11111111000, 0b00000000111),        (0b00000000111, 0b00000000111),    ]else:    FP_FIDELITY_ITER_MASK = [        (0b11111000000, 0b11111110000),        (0b00000111110, 0b11111110000),        (0b11111000000, 0b00000001111),        (0b00000111110, 0b00000001111),    ]`
**Mantissa Bit Distribution**

| Architecture | Iteration | Operand A Bits | Operand B Bits | Total Mantissa Coverage |
| --- | --- | --- | --- | --- |
| WH/BH | 0 | 5 MSBs | 7 MSBs | 12 bits total, 5 overlap |
| WH/BH | 1 | 5 LSBs | 7 MSBs | Completes A, maintains B |
| WH/BH | 2 | 5 MSBs | 4 LSBs | Maintains A, starts B LSBs |
| WH/BH | 3 | 5 LSBs | 4 LSBs | Full 10-bit coverage |
| Quasar | 0 | 8 MSBs | 8 MSBs | 16 bits total, 8 overlap |
| Quasar | 1 | 3 LSBs | 8 MSBs | Completes A, maintains B |
| Quasar | 2 | 8 MSBs | 3 LSBs | Maintains A, starts B LSBs |
| Quasar | 3 | 3 LSBs | 3 LSBs | Full 10-bit coverage (with overlap) |

## Summary of Key Code Entities

**Enumerations and Types**:

*   `MathFidelity`: `LoFi`, `HiFi2`, `HiFi3`, `HiFi4` - Controls iteration count
*   `DestAccumulation`: `Yes`, `No` - Controls 32-bit vs 16-bit destination precision
*   `ApproximationMode`: `Yes`, `No` - Controls use of fast approximations
*   `FastMode`: `Yes`, `No` - Enables fast-mode variants of select operations

**Hardware Configuration**:

*   `matmul_configure_addrmod()`: Sets up address modifiers with fidelity increment
*   `matmul_configure_mop()`: Programs MOP with fidelity iterations
*   `matmul_configure_mop_throttled()`: Programs MOP with throttling NOPs
*   `run_throttled_sequence()`: Inserts NOPs between MVMUL instructions
*   `ADDR_MOD_3`: Address modifier with bias increment for accumulation
*   `ADDR_MOD_5`: Reset with fidelity increment
*   `ADDR_MOD_6`: Complete reset with fidelity clear (throttled mode)
*   `is_fp32_dest_acc_en`: Template parameter controlling 32-bit accumulation

**Golden Model Classes**:

*   `FidelityMasking`: Base class with `_apply_fidelity_masking()` method
*   `MatmulGolden`: Implements fidelity-aware matrix multiplication
*   `SrcFormatModel`: Converts between DataFormat and TF32 representation
*   `FP_FIDELITY_ITER_MASK`: Array of (mask_a, mask_b) tuples per iteration

**Test Configuration**:

*   `MATH_FIDELITY()`: Template parameter for fidelity level
*   `APPROX_MODE()`: Template parameter for approximation mode
*   `CLAMP_NEGATIVE()`: Template parameter for exponential clamping
*   `TestConfig`: Orchestrates compilation and execution with fidelity settings
