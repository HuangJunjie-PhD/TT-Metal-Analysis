## Purpose and Scope

This document describes the Blackhole ASIC Ethernet subsystem, which provides 14 independent Ethernet channels for chip-to-chip communication. Each Ethernet channel is managed by an ERISC (Ethernet RISC) controller running dedicated firmware. This page covers the hardware architecture, firmware components, SerDes multiplexing, and initialization sequence for the Ethernet subsystem.

For information about the PCIe subsystem (which shares SerDes lanes with some Ethernet channels), see [PCIe Subsystem](https://deepwiki.com/tenstorrent/tt-system-firmware/3.2-pcie-subsystem). For the overall hardware initialization sequence, see [Boot and Initialization Sequence](https://deepwiki.com/tenstorrent/tt-system-firmware/3.1-boot-and-initialization-sequence).

* * *

## Hardware Architecture

### Ethernet Instances and ERISC Controllers

Blackhole provides 14 Ethernet instances (ETH 0-13), each with a dedicated ERISC controller and 512KB of L1 memory. The ERISC is a RISC processor that executes Ethernet protocol firmware to handle link training, packet processing, and SerDes management.

**Ethernet Instance Architecture**

**Sources:**[lib/tenstorrent/bh_arc/eth.c 31-36](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L31-L36)[lib/tenstorrent/bh_arc/eth.c 56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L56-L56)

### Ethernet Instance Distribution

The 14 Ethernet instances are distributed across the chip and share SerDes resources with PCIe controllers:

| Ethernet Instances | SerDes Assignment | Conflicts With |
| --- | --- | --- |
| ETH 0-3 | SerDes 0, 1 | PCIe Instance 0 (if configured for x8 or x16) |
| ETH 4-6 | SerDes 2, 3 (multiplexed) | None (dedicated Ethernet SerDes) |
| ETH 7-9 | SerDes 5 (multiplexed) | None (dedicated Ethernet SerDes) |
| ETH 10-13 | SerDes 3, 4 | PCIe Instance 1 (if configured for x8 or x16) |

**Sources:**[lib/tenstorrent/bh_arc/eth.c 87-184](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L87-L184)

### L1 Memory Layout

Each Ethernet instance has 512KB of L1 memory organized as follows:

| Address Range | Size | Contents |
| --- | --- | --- |
| `0x70000 - 0x7BFFF` | ~48KB | ERISC firmware binary |
| `0x7C000 - 0x7FFFF` | 16KB | Configuration parameters |
| `0x70188` | 4B | Firmware version (offset from base) |

**Sources:**[lib/tenstorrent/bh_arc/eth.c 31-34](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L31-L34)

## Initialization Sequence

The Ethernet subsystem is initialized by the `EthInit()` function at system init priority 106, following PCIe initialization (priority 103) and preceding telemetry initialization (priority 111).

### High-Level Initialization Flow

**Sources:**[lib/tenstorrent/bh_arc/eth.c 424-476](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L424-L476)

### Detailed Initialization Steps

#### 1. L1 Memory Wiping

Before loading firmware, all ERISC L1 memory is cleared using NOC DMA transfers:

```
Function: wipe_l1()
- Uses DMA channel 1 on NOC
- Clears 512KB (ERISC_L1_SIZE) per instance
- Source: Zero-filled Tensix L1 (broadcast)
- Destination: Each enabled Ethernet instance
```

**Sources:**[lib/tenstorrent/bh_arc/eth.c 381-422](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L381-L422)

#### 2. Firmware Loading

The `LoadEthFw()` function loads the ERISC firmware binary:

```
Function: LoadEthFw(eth_inst, ring, buf, buf_size, spi_address, image_size)
- Load address: ETH_FW_BASE_ADDR (0x70000)
- Uses spi_arc_dma_transfer_to_tile()
- Sets RESET_PC_0 to firmware load address
- Sets END_PC_0 to parameter base - 4
```

**Sources:**[lib/tenstorrent/bh_arc/eth.c 228-248](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L228-L248)

#### 3. Configuration Parameters

The `LoadEthFwCfg()` function loads configuration parameters and injects board-specific data:

**Configuration Parameter Injection**

| Offset (32-bit words) | Parameter | Source |
| --- | --- | --- |
| 0 | `eth_sel` (enabled instances bitmask) | `GetEthSel()` |
| 1 | `speed_override` (40/100/200/400G) | FW table or default |
| 32 | `pcb_type` | FW table |
| 33 | `asic_location` | FW table |
| 34-35 | `board_id` (64-bit) | FW table |
| 36-37 | `mac_addr_base` (48-bit) | Generated from ASIC ID |
| 38-39 | `asic_id` (64-bit) | Functional eFuse |
| 40 | `eth_enabled` | Tile enable mask |

**Sources:**[lib/tenstorrent/bh_arc/eth.c 260-317](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L260-L317)

#### 4. ERISC Reset Release

After firmware and configuration are loaded, the ERISC is released from reset:

```
Function: ReleaseEthReset(eth_inst, ring)
- Clear bit 11 in ETH_RISC_DEBUG_SOFT_RESET_0 (0xFFB121B0)
- Releases RISC0 reset only (RISC1 remains in reset)
- ERISC begins executing from RESET_PC_0
```

**Sources:**[lib/tenstorrent/bh_arc/eth.c 219-226](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L219-L226)

## SerDes Multiplexing and Configuration

### SerDes Mux Architecture

Blackhole uses SerDes multiplexers to share high-speed lanes between PCIe and Ethernet. The mux configuration is controlled by the `RESET_UNIT_PCIE_MISC_CNTL3` registers.

**SerDes Mux Configuration**

**Sources:**[lib/tenstorrent/bh_arc/eth.c 87-114](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L87-L114)[lib/tenstorrent/bh_arc/eth.c 64-75](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L64-L75)

### Ethernet Selection Logic

The `GetEthSel()` function determines which Ethernet instances are active based on:

1.   **Mux configuration** (which instances are routed to SerDes)
2.   **PCIe mode** (whether PCIe is using SerDes lanes)
3.   **Harvesting mask** (defective instances)
4.   **Board-specific disable mask** (from FW table)

**Ethernet Selection Decision Tree**

**Sources:**[lib/tenstorrent/bh_arc/eth.c 116-184](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L116-L184)

## MAC Address Generation

Each Ethernet instance is assigned a unique MAC address derived from the ASIC ID stored in functional eFuses.

### MAC Address Algorithm

```
Base MAC Address Format:
  Organizationally Unique Identifier: 0x208C47 (24 bits)
  Device-Specific ID: asic_id_low[15:0] * 12 (24 bits)

Full MAC Address (48 bits):
  [47:24] = 0x208C47
  [23:0]  = (asic_id_low & 0xFFFF) * 12 + ethernet_instance_offset
```

**Function:**`GetMacAddressBase()` returns the base MAC address (without instance offset). The ERISC firmware adds instance-specific offsets.

**Sources:**[lib/tenstorrent/bh_arc/eth.c 44](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L44-L44)[lib/tenstorrent/bh_arc/eth.c 186-198](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L186-L198)

## Firmware Version Retrieval

The CMFW can query the ERISC firmware version from any running Ethernet instance:

```
Function: GetEthFwVersion(ring)
- Searches for first enabled Ethernet instance
- Reads 32-bit version from L1 offset 0x188 (ETH_FW_VERSION_ADDR_OFFSET)
- Returns version as 32-bit integer (major.minor.patch format)
- Returns 0 if no Ethernet instances enabled
```

**Sources:**[lib/tenstorrent/bh_arc/eth.c 200-217](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L200-L217)

## Configuration Override Mechanisms

### Speed Configuration

Ethernet link speed can be overridden via the FW table:

`uint32_t speed_override = fwtable->eth_property_table.eth_speed_override; // Valid values: 40, 100, 200, 400 (Gbps)if (speed_override == 40 || speed_override == 100 ||     speed_override == 200 || speed_override == 400) {    fw_cfg_32b[1] = speed_override;}`
**Sources:**[lib/tenstorrent/bh_arc/eth.c 276-284](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L276-L284)

### Instance Disable Mask

Specific Ethernet instances can be forcibly disabled:

`if (fwtable->eth_property_table.eth_disable_mask_en) {    eth_sel &= ~fwtable->eth_property_table.eth_disable_mask;}`
This allows disabling individual Ethernet channels for debugging or board-specific constraints.

**Sources:**[lib/tenstorrent/bh_arc/eth.c 167-170](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L167-L170)
