This page documents the automated quality assurance (QA) workflows that validate code quality, licensing compliance, and artifact integrity in the tt-system-firmware repository. These checks run automatically on pull requests and pushes to ensure code meets project standards before merging.

For information about testing firmware functionality, see [Testing and Validation](https://deepwiki.com/tenstorrent/tt-system-firmware/7-testing-and-validation). For build system details, see [CI/CD Infrastructure](https://deepwiki.com/tenstorrent/tt-system-firmware/8-cicd-infrastructure).

## Overview

The tt-system-firmware repository implements four categories of automated QA checks:

| Check Type | Workflow | Purpose | Trigger Events |
| --- | --- | --- | --- |
| Compliance | `compliance.yml` | Code style, commit format, documentation linting | Pull requests |
| Copyright | `copyright-check.yml` | SPDX header validation | Pushes, pull requests, merge groups |
| License | `license-check.yml` | Third-party license scanning with scancode | Pushes, pull requests, merge groups |
| Blob Verification | `verify-blob.yml` | SHA256 integrity checks for binary blobs | Pull requests |
| Build Verification | `build-fw.yml` | Compiler warnings as errors, signature validation | Pushes, pull requests, merge groups |

All workflows enforce quality gates that must pass before code can be merged to `main`, `v*-branch`, or collaboration branches.

## Compliance Checks

The `compliance.yml` workflow validates code quality standards using Zephyr's `check_compliance.py` tool. This check runs only on pull requests to validate patch series against the target branch.

### Compliance Workflow Details

### Compliance Checks Performed

The workflow validates the following aspects of code changes:

| Check Category | Description | Enforcement Level |
| --- | --- | --- |
| ClangFormat | C/C++ code formatting style | Warning |
| GitLint | Commit message format validation | Error |
| Identity | Signed-off-by line presence | Error (skipped for dependabot) |
| Checkpatch | Linux kernel coding style | Error |
| Kconfig | Kconfig symbol validation | Error |
| DeviceTree | DTS linting and validation | Error |
| YAML | YAML file syntax validation | Error |

### Key Environment Variables

*   `ZEPHYR_BASE`: Path to Zephyr repository (`.../zephyr`)
*   `checkout_path`: PR checkout directory (`tt-system-firmware`)
*   `compliance_excludes`: `-e BinaryFiles -e KconfigBasicNoModules -e SysbuildKconfigBasic -e LicenseAndCopyrightCheck`
*   `manifest_group_filter`: `+ci,+optional`

### Rebase Strategy

The compliance workflow enforces a linear history by rebasing pull request commits onto the target branch:

1.   Ensures no merge commits exist in the PR ([.github/workflows/compliance.yml 48-49](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L48-L49))
2.   Performs `git rebase origin/${BASE_REF}` ([.github/workflows/compliance.yml 52](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L52-L52))
3.   Fails the check if rebase conflicts occur

### Artifacts

The workflow uploads the following artifacts on completion:

*   `compliance.xml`: JUnit-formatted test results
*   `dts_linter.patch`: Automatically generated devicetree formatting fixes (if applicable)

Sources: [.github/workflows/compliance.yml 1-157](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L1-L157)[.github/workflows/compliance.yml 6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L6-L6)

## Copyright Check

The `copyright-check.yml` workflow validates that all source files contain proper SPDX license headers. This check runs on pushes, pull requests, and merge group events.

### Copyright Validation Process

### Concurrency Control

The workflow uses GitHub's concurrency control to cancel in-progress checks when new commits are pushed:

*   **Concurrency Group**: `${{ github.workflow }}-${{ github.event.pull_request.number || github.sha }}`
*   **Cancel In Progress**: `true`

This ensures that only the most recent version of a PR is being validated, reducing CI resource usage.

### Script Behavior

The `check-copyright.py` script ([.github/workflows/copyright-check.yml 37-38](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/copyright-check.yml#L37-L38)):

1.   Scans all tracked files in the repository
2.   Validates SPDX-License-Identifier presence
3.   Verifies copyright year and organization format
4.   Returns non-zero exit code if any files are non-compliant

Sources: [.github/workflows/copyright-check.yml 1-39](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/copyright-check.yml#L1-L39)

## License Scanning with Scancode

The `license-check.yml` workflow performs deep license analysis using the scancode-toolkit to detect third-party code and licensing issues.

### Scancode Workflow Architecture

### Scan Configuration

The workflow scans the `scan/` directory ([.github/workflows/license-check.yml 40](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/license-check.yml#L40-L40)), which typically contains:

*   Source code directories
*   Third-party library integrations

### Output Artifacts

The scancode action generates the following artifacts in the `artifacts/` directory:

| File | Format | Purpose |
| --- | --- | --- |
| `report.txt` | Plain text | Human-readable license violations |
| `scancode.json` | JSON | Machine-readable scan results |
| `summary.txt` | Plain text | High-level summary of findings |

### Failure Conditions

The check fails if `report.txt` is non-empty ([.github/workflows/license-check.yml 49-56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/license-check.yml#L49-L56)), indicating one or more license compliance issues were detected. The report content is printed to the GitHub Actions error log for debugging.

## Blob Verification

The `verify-blob.yml` workflow ensures the integrity of pre-compiled binary firmware blobs by validating their SHA256 checksums against the declarations in `zephyr/module.yml`.

### Blob Verification Process

### verify_blob.py Implementation

The verification script implements the following logic:

**Module Configuration Parsing** ([scripts/verify_blob.py 14-22](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L14-L22)):

**SHA256 Calculation** ([scripts/verify_blob.py 25-31](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L25-L31)):

**Verification Logic** ([scripts/verify_blob.py 34-56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/verify_blob.py#L34-L56)):

*   Iterates through all blobs declared in `module.yml`
*   Skips blobs where file no longer exists
*   Compares declared SHA256 with calculated SHA256
*   Returns exit code 1 if any mismatch is detected

### Protected Blobs

The workflow protects the integrity of critical firmware binaries including:

| Blob Category | Examples | Purpose |
| --- | --- | --- |
| MRISC Firmware | `tt_blackhole_gddr_init.bin` | GDDR memory training |
| ERISC Firmware | `tt_blackhole_erisc.bin` | Ethernet subsystem |
| SerDes Firmware | `tt_blackhole_libpciesd.a` | PCIe PHY configuration |
| Boot ROM | `tt_blackhole_nano_bootcode.bin` | ARC boot code |
| Legacy Bundles | `fw_pack-*.tar.gz` | Wormhole/Grayskull firmware |

### Failure Scenarios

The check fails when:

1.   A blob file has been modified without updating `module.yml` SHA256
2.   A blob listed in `module.yml` has an incorrect SHA256 hash
3.   The verification script encounters a file read error

## Build-Time Quality Checks

The `build-fw.yml` workflow enforces quality standards during the firmware build process through compiler flags and post-build validation.

### Build-Time Validation Flow

### Warnings as Errors

All firmware builds enable `-DCONFIG_COMPILER_WARNINGS_AS_ERRORS=y` ([.github/workflows/build-fw.yml 95](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L95-L95)), which promotes compiler warnings to errors. This prevents code with warnings from being merged.

**Affected Build Targets**:

*   Main firmware builds ([.github/workflows/build-fw.yml 100-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L100-L106))
*   Recovery firmware ([.github/workflows/build-fw.yml 171-176](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L171-L176))
*   Assembly test firmware ([.github/workflows/build-fw.yml 258-265](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L258-L265))

### Signature Key Management

The signing key is stored as a GitHub secret and decoded during the workflow:

1.   Secret `SIGNATURE_KEY` is base64-encoded ([.github/workflows/build-fw.yml 79-84](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L79-L84))
2.   Decoded to temporary file during build
3.   File size checked to ensure valid key (> 42 bytes)
4.   Used for both signing (MCUBoot) and verification (imgtool)

If no valid signing key is available, signature verification is skipped with a warning message.

### Build Matrix Validation

The workflow builds firmware for all supported board revisions in parallel ([.github/workflows/build-fw.yml 46-52](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L46-L52)):

Each board variant must compile without warnings and produce valid signed binaries before the workflow succeeds.
