### Job Dependency Graph in run-tests.yaml

**Complete Job Execution Flow**

**Job Dependencies and Execution Order**

The `run-tests.yaml` workflow implements a complex dependency graph with six execution phases:

1.   **Initial Phase**: `tools-tests` and `count-test-files` run in parallel

## Permissions Model

### Workflow Permission Requirements

Different workflows require different permission scopes:

| Workflow | Permission Scope | Required Permissions | Justification |
| --- | --- | --- | --- |
| `before_merge.yaml` | Job-level | `actions: read`, `contents: write`, `pages: write`, `pull-requests: write`, `id-token: write` | Delegates to `run-tests.yaml` |
| `run-tests.yaml` | Workflow-level | Same as above | Creates PRs, publishes pages |
| `update-ttnn-wheel.yaml` | Step-level | `pull-requests: write`, `contents: read` (job) + `GH_TOKEN` secret | Creates and merges PRs |
| `community-issue-tagging.yml` | Job-level | `issues: write`, `pull-requests: write` | Adds labels |
| `update-readme.yaml` | Job-specific | Split between jobs | Separation of concerns |

**Token Usage Patterns**

The workflows use two types of authentication tokens:

1.   **`secrets.GITHUB_TOKEN`**: Default GitHub Actions token for standard operations ([.github/workflows/update-ttnn-wheel.yaml 70](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L70-L70))
2.   **`secrets.GH_TOKEN`**: Custom PAT (Personal Access Token) for operations requiring elevated permissions ([.github/workflows/update-ttnn-wheel.yaml 65](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L65-L65)[.github/workflows/update-ttnn-wheel.yaml 80](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L80-L80))

The `GH_TOKEN` is necessary for:

*   Auto-approving PRs (requires separate authentication context)

## Accuracy Testing Workflow

### run-accuracy-tests.yaml Architecture

The accuracy testing workflow follows a three-phase execution model distinct from the main test pipeline:

**Phase 1: Model Test Execution and Export**

This phase runs model tests and exports standalone test scripts to `tests/export_code/accuracy/`.

**Phase 2: Dynamic Group Calculation**

Dynamically creates a matrix based on the number of generated accuracy test files.

**Phase 3: Parallel Accuracy Validation**

Each matrix job runs one standalone accuracy test script.
