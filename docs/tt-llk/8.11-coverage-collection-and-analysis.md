This page documents the code coverage collection and analysis system in the tt-llk testing framework. Coverage data tracks which lines of C++ kernel code are executed during test runs, providing visibility into test suite completeness. For information about performance profiling and benchmarking, see [8.10](https://deepwiki.com/tenstorrent/tt-llk/8.10-performance-testing-and-profiling).

## Purpose and Scope

The coverage system instruments compiled kernel code with gcov profiling hooks, collects execution data during hardware test runs, and generates LCOV-format reports that are aggregated and uploaded to Codecov. This enables:

*   Line-level execution tracking of LLK kernel implementations
*   Identification of untested code paths across different architectures
*   Coverage trending over time via CI/CD integration
*   Per-architecture coverage metrics (Wormhole, Blackhole, Quasar)

The system handles the special requirements of RISC-V cross-compiled code running on Tensix hardware, including custom memory layouts and data extraction from L1 memory.

### Linker Script Memory Layout

Coverage builds use expanded debug linker scripts to accommodate instrumentation overhead:

#### Wormhole Debug Layout

```
TRISC0_CODE:            64K   (origin: 0x12000)
TRISC0_LOCAL_DATA_MEM:  32K
TRISC0_GCOV_MEM:        32K   ← Coverage data storage

TRISC1_CODE:            64K
TRISC1_LOCAL_DATA_MEM:  32K  
TRISC1_GCOV_MEM:        32K   ← Coverage data storage

TRISC2_CODE:            64K
TRISC2_LOCAL_DATA_MEM:  32K
TRISC2_GCOV_MEM:        32K   ← Coverage data storage

BRISC_GCOV_MEM:         8K    ← Coverage data storage
```

#### Blackhole Debug Layout

```
TRISC0_GCOV_MEM:        8K    (smaller due to memory constraints)
TRISC1_GCOV_MEM:        8K
TRISC2_GCOV_MEM:        8K
BRISC_GCOV_MEM:         8K
```

**Sources:**[tests/helpers/ld/memory.wormhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L1-L68)[tests/helpers/ld/memory.blackhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L1-L68)

## Runtime Coverage Data Collection

During test execution, instrumented code writes coverage counters to the allocated `GCOV_MEM` sections in L1 memory. The test infrastructure extracts this data after each test variant completes.

### Coverage Data Flow

**Sources:**[tests/python_tests/helpers/test_config.py 1087-1141](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1087-L1141)

### Coverage Data Extraction Implementation

The `read_coverage_data()` function in `TestConfig` handles extraction:

| Step | Function/Method | Description |
| --- | --- | --- |
| 1 | `parse_elf()` | Parse ELF to locate `.gcov` section addresses |
| 2 | `read_from_device()` | Read coverage data from L1 memory at extracted addresses |
| 3 | File I/O | Write raw binary `.gcda` files to variant directory |
| 4 | Per-RISC | Separate files for `brisc.gcda`, `main_unpack.gcda`, `main_math.gcda`, `main_pack.gcda` |

The coverage data must be extracted before the next test variant executes, as each test overwrites the L1 memory regions.

**Sources:**[tests/python_tests/helpers/test_config.py 1087-1141](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1087-L1141)

## Coverage Processing Pipeline

After test execution completes, coverage data is processed to generate consolidated reports:

### Processing Function: `process_coverage_run_artefacts()`

**Sources:**[tests/python_tests/helpers/test_config.py 1143-1272](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1143-L1272)

### Processing Commands

The processing pipeline executes these shell commands:

```
# Step 1: Merge per-variant .gcda files
riscv-tt-elf-gcov-tool merge-stream {variant1}/coverage/*.gcda 
                                    {variant2}/coverage/*.gcda 
                                    -o {merged_dir}/

# Step 2: Generate coverage info for each test
riscv-tt-elf-gcov -i {test_name}.gcda
# Produces: {test_name}.info

# Step 3: Merge all test coverage
lcov -q -a test1.info -a test2.info -a test3.info -o merged.info

# Step 4: Filter out test infrastructure code
lcov -q --remove merged.info '*tests/*' -o merged_coverage.info
```

The final `merged_coverage.info` file is placed at `/tmp/tt-llk-build/merged_coverage.info` and contains line-level coverage data for all kernel code.

**Sources:**[tests/python_tests/helpers/test_config.py 1185-1269](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1185-L1269)

## Coverage Data File Structure

The coverage system generates multiple file types:

### File Types and Locations

```
/tmp/tt-llk-build/
├── {test_name}/
│   └── {variant_hash}/
│       └── coverage/
│           ├── brisc.gcda              # BRISC coverage counters
│           ├── main_unpack.gcda        # TRISC0 coverage counters
│           ├── main_math.gcda          # TRISC1 coverage counters
│           └── main_pack.gcda          # TRISC2 coverage counters
├── coverage_info/
│   ├── {test_name1}.info               # Per-test LCOV format
│   ├── {test_name2}.info
│   └── ...
└── merged_coverage.info                # Final aggregated report
```

### GCDA File Format

The `.gcda` files are binary gcov data files containing:

*   Magic number identifying gcov version
*   Function summaries
*   Arc transition counts (branch execution counts)
*   Line execution counts

These files must be processed with gcov tools matching the compiler version that generated the instrumentation.

**Sources:**[tests/python_tests/helpers/test_config.py 270](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L270-L270)[tests/python_tests/helpers/test_config.py 1095-1140](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1095-L1140)

## Usage in Test Development

### Enabling Coverage for Local Testing

`cd tests/python_tests/ # Run tests with coveragepytest --coverage -m "not perf" . # Coverage data is automatically processed# Output: /tmp/tt-llk-build/merged_coverage.info`
### Viewing Coverage Reports Locally

`# Generate HTML report from merged coveragegenhtml /tmp/tt-llk-build/merged_coverage.info \        --output-directory coverage_html # Open in browseropen coverage_html/index.html`
### Coverage Considerations

When developing tests, consider:

1.   **Build Time**: Coverage builds take ~2-3x longer due to instrumentation
2.   **Memory Constraints**: Debug linker scripts require more L1 space
3.   **Incompatibility**: Cannot enable both `--coverage` and profiler builds simultaneously
4.   **Filtering**: Test infrastructure code is filtered from final reports

## Coverage Metrics and Reporting

### Codecov Integration

Coverage reports are uploaded to [codecov.io](https://codecov.io/) where they provide:

*   Line coverage percentages per file
*   Diff coverage (coverage of changed lines in PRs)
*   Coverage trends over time
*   Architecture-specific coverage (separate reports for Wormhole/Blackhole)

The upload is authenticated via `CODECOV_TOKEN` secret and includes flags for architecture identification:

**Sources:**[.github/workflows/setup-and-test.yml 267-274](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L267-L274)

### Coverage Report Structure

The final LCOV info file contains entries for each source file:

```
SF:tt_llk_wormhole_b0/llk_lib/llk_math_eltwise_unary_sfpu.h
FN:42,llk_math_eltwise_unary_sfpu_exponential
FNDA:156,llk_math_eltwise_unary_sfpu_exponential
DA:43,156
DA:44,156
DA:45,0
...
```

Where:

*   `SF`: Source file path
*   `FN`: Function definition at line 42
*   `FNDA`: Function executed 156 times
*   `DA`: Line 43 executed 156 times (0 = not executed)
