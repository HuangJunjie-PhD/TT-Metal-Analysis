## Overview

Attention mechanisms are a core component of transformer-based language models, allowing the model to focus on different parts of input when generating output. vLLM implements multiple attention backends optimized for different hardware platforms and use cases, enabling high-performance inference across a wide range of environments.

## Attention Types

vLLM supports different types of attention for various model architectures:

These attention types are defined in the `AttentionType` class:

| Attention Type | Description | Used In |
| --- | --- | --- |
| DECODER | Self-attention for decoder models | GPT, LLaMA, etc. |
| ENCODER | Self-attention in encoder part | Encoder component of T5, BART |
| ENCODER_ONLY | Bidirectional attention | BERT, RoBERTa |
| ENCODER_DECODER | Cross-attention between decoder Q and encoder KV | T5, BART |

Sources: [vllm/attention/backends/abstract.py 19-32](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/attention/backends/abstract.py#L19-L32)

## Attention Layer

The `Attention` class in vLLM is the main module that performs attention computation. It takes query, key, and value tensors as input, stores the key and value tensors in the KV cache, and performs attention computation using the selected backend.

During initialization, the appropriate attention backend is selected based on the model configuration and available hardware. The forward pass then performs the following steps:

1.   Reshape the query, key, and value tensors
2.   Calculate KV scales if needed (for FP8 quantization)
3.   Call the backend's forward method to compute attention
4.   Return the attention output

## Attention Backends

vLLM provides multiple attention backends optimized for different hardware platforms and specific use cases:

### 1. Flash Attention Backend

The Flash Attention backend implements the FlashAttention algorithm, which optimizes attention computation by avoiding materializing the full attention matrix.

Key features:

*   Supports head sizes: 32, 64, 96, 128, 160, 192, 224, 256
*   Efficient tiled computation to maximize GPU cache usage
*   Fused softmax operation
*   Support for alibi positional embeddings
*   Support for sliding window attention
*   Support for FP8 KV cache

Sources: [vllm/attention/backends/flash_attn.py 40-366](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/attention/backends/flash_attn.py#L40-L366)

### 2. Flash Infer Backend

The Flash Infer backend uses the FlashInfer library for optimized attention computation.

Key features:

*   Supports head sizes: 64, 128, 256
*   Optimized for auto-regressive generation
*   Support for paged KV cache
*   CUDA graph support for decode phase

Sources: [vllm/attention/backends/flashinfer.py 52-360](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/attention/backends/flashinfer.py#L52-L360)

### 3. xFormers Backend

The xFormers backend uses the xFormers library for memory-efficient attention.

Key features:

*   Memory-efficient attention implementation
*   Support for different attention patterns

### 4. Torch SDPA Backend

The Torch SDPA backend uses PyTorch's `scaled_dot_product_attention` function.

Key features:

## Attention Metadata

Attention metadata is used to track information about the sequences being processed and is essential for correct attention computation. It includes:

*   Sequence lengths
*   Block tables (mapping from sequence to KV cache blocks)
*   Slot mappings (mapping from tokens to KV cache slots)
*   Context lengths
*   Query and sequence start locations

The metadata is structured differently for prefill and decode phases:

| Phase | Key Metadata |
| --- | --- |
| Prefill | Sequence lengths, query start locations, block tables for entire prompt |
| Decode | Context lengths, block tables for tokens generated so far |

The metadata builders construct this metadata based on the input batch, ensuring that each backend has the information it needs for attention computation.

Sources: [vllm/attention/backends/utils.py 121-267](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/attention/backends/utils.py#L121-L267)

## Prefill and Decode Phases

vLLM's attention computation has two distinct phases:

### Prefill Phase

During the prefill phase, the entire prompt is processed at once:

1.   The model processes all tokens in the prompt simultaneously
2.   Key and value tensors are computed and stored in the KV cache
3.   For each token, attention is computed over all previous tokens
4.   The attention patterns can be causal (decoder-only) or bidirectional (encoder-only)

### Decode Phase

During the decode phase, one new token is generated at a time:

1.   The model processes only the new token
2.   Key and value tensors for the new token are computed and stored in the KV cache
3.   Attention is computed over all stored key-value pairs
4.   The attention pattern is always causal (looking at previous tokens only)

This separation allows for different optimizations in each phase. The prefill phase optimizes for batch processing, while the decode phase optimizes for single-token generation and reuse of cached key-value pairs.

Sources: [vllm/attention/backends/flash_attn.py 604-606](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/attention/backends/flash_attn.py#L604-L606)

## Conclusion

vLLM's attention mechanisms provide a flexible and efficient foundation for high-performance LLM inference. The modular backend architecture allows for optimization across diverse hardware platforms, while the paged attention mechanism enables efficient memory usage for long sequences.

The system supports a wide range of model architectures, including decoder-only models (like GPT and LLaMA), encoder-only models (like BERT), and encoder-decoder models (like T5 and BART). The ability to handle different attention types and phases (prefill and decode) makes vLLM a versatile toolkit for LLM inference.

Understanding these attention mechanisms is crucial for maximizing the performance of LLM inference in vLLM and for extending the system to new hardware platforms and model architectures.
