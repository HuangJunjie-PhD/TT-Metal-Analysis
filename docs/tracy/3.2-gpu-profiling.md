This page covers Tracy's GPU profiling capabilities, which enable detailed performance analysis of graphics and compute workloads. Tracy supports GPU profiling across multiple graphics APIs including Direct3D 11/12, Vulkan, OpenGL, and OpenCL.

For general CPU profiling features, see [Core Profiling API](https://deepwiki.com/tenstorrent/tracy/3.1-core-profiling-api). For memory profiling, see [Memory & System Tracking](https://deepwiki.com/tenstorrent/tracy/3.3-memory-and-system-tracking).

## Overview

Tracy's GPU profiling system captures precise timing information from GPU operations by leveraging each graphics API's native timestamp query mechanisms. The system establishes synchronized time domains between CPU and GPU, allowing for accurate correlation of CPU and GPU work in the timeline view.

### Clock Calibration

GPU profiling requires precise synchronization between CPU and GPU time domains. Each API implements calibration differently:

**Direct3D 12** uses `GetClockCalibration()` to correlate timestamps:

*   Queries both CPU and GPU timestamps simultaneously
*   Calculates drift compensation over time
*   Sends `QueueType::GpuCalibration` events for synchronization

**Vulkan** supports calibrated timestamps via `VK_EXT_calibrated_timestamps`:

*   Uses `vkGetCalibratedTimestampsEXT()` for precise correlation
*   Falls back to device-only timestamps if extension unavailable
*   Measures calibration deviation to ensure accuracy

### Collection Process

The `Collect()` function retrieves completed timestamp data:

**Direct3D 12** uses a payload queue system:

1.   Groups queries into payloads per frame
2.   Uses fences to track completion
3.   Maps readback buffer to retrieve results
4.   Resolves queries in batches for efficiency

**Vulkan** polls query availability:

1.   Checks `vkGetQueryPoolResults()` for completion
2.   Retrieves timestamp values in nanoseconds
3.   Resets consumed queries for reuse
4.   Performs periodic clock recalibration

**OpenCL** processes event profiling info:

1.   Checks `cl_event` completion status
2.   Extracts `CL_PROFILING_COMMAND_START/END` timestamps
3.   Releases events after data extraction
4.   Handles begin/end event pairs

## API-Specific Implementation Details

### Direct3D 12 Implementation

The D3D12 implementation (`D3D12QueueCtx`) provides comprehensive queue-based profiling:

**Key Features:**

*   Per-queue context isolation
*   Copy queue timestamp support via `D3D12_OPTIONS3`
*   Automatic query heap sizing with fallback
*   Fence-based synchronization for data collection
*   Clock drift compensation

**Query Resolution:**

*   Uses `ResolveQueryData()` to copy timestamps to readback buffer
*   Maps buffer memory for CPU access
*   Processes query pairs (begin/end) atomically

Sources: [public/tracy/TracyD3D12.hpp 116-146](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/tracy/TracyD3D12.hpp#L116-L146)[public/tracy/TracyD3D12.hpp 419-435](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/tracy/TracyD3D12.hpp#L419-L435)

### Vulkan Implementation

The Vulkan implementation (`VkCtx`) supports multiple construction modes:

**Standard Mode:**

*   Requires command buffer for query pool reset
*   Uses `vkQueueSubmit()` for synchronization
*   Supports device-only timestamps

**Host Query Reset Mode:**

*   Uses `VK_EXT_host_query_reset` extension
*   Eliminates command buffer dependency
*   Requires calibrated timestamps for accuracy

**Symbol Table Option:**

*   Optional dynamic function loading via `TRACY_VK_USE_SYMBOL_TABLE`
*   Enables profiling in environments without static linking

Sources: [public/tracy/TracyVulkan.hpp 96-223](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/tracy/TracyVulkan.hpp#L96-L223)[public/tracy/TracyVulkan.hpp 175-222](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/tracy/TracyVulkan.hpp#L175-L222)

### OpenCL Event-Based Profiling

OpenCL profiling uses the event profiling mechanism:

**Event Handling:**

*   Retains `cl_event` objects for begin/end timestamps
*   Associates events with Tracy query IDs
*   Processes events in chronological order
*   Validates command queue profiling is enabled

**Example Usage Pattern:**

`TracyCLZone(ctx, "Kernel Execution");clEnqueueNDRangeKernel(queue, kernel, 1, nullptr, &global_size, nullptr, 0, nullptr, &event);TracyCLZoneSetEvent(event);`
