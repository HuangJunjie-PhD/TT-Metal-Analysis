## Purpose and Scope

This document describes the user-space resource locking mechanism provided by the Tenstorrent kernel driver via the `TENSTORRENT_IOCTL_LOCK_CTL` command. Resource locks enable inter-process coordination for shared device resources without requiring additional user-space synchronization primitives. Each device provides 64 independent locks that can be acquired, released, and tested via IOCTL operations.

For information about kernel-level synchronization primitives used within the driver, see [9.1 Synchronization Primitives](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/9.1%20Synchronization%20Primitives) For device reset coordination, see [9.4 Reset and Recovery](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/9.4%20Reset%20and%20Recovery)

* * *

## Overview

The resource lock system provides a kernel-mediated locking mechanism for coordinating access to device resources between multiple processes. Each Tenstorrent device maintains 64 independent locks (indices 0-63) that can be acquired and released by file descriptors.

### Key Characteristics

| Property | Value |
| --- | --- |
| Lock Count | 64 per device [ioctl.h 43](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L43-L43) |
| Lock Scope | Per-device, shared across all file descriptors [chardev.c 288-289](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L288-L289) |
| Lock Type | Binary (acquired or free) [chardev.c 270-329](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L270-L329) |
| Ownership Tracking | Per file descriptor (`chardev_private`) [chardev_private.h 29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L29-L29) |
| Cleanup | Automatic on file descriptor close [chardev.c 702-707](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L702-L707) |
| Blocking Support | Yes, via `TENSTORRENT_LOCK_CTL_ACQUIRE_BLOCKING`[ioctl.h 213](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L213-L213) |
| Signal Safety | Blocking operations are interruptible [chardev.c 296-298](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L296-L298) |

The system implements a two-level tracking structure:

*   **Global locks** (`tenstorrent_device.resource_lock`): tracks which locks are held by any file descriptor on this hardware instance.
*   **Local locks** (`chardev_private.resource_lock`): tracks which locks are held by each specific file descriptor.

Sources: [ioctl.h 43](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L43-L43)[chardev.c 270-329](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L270-L329)[chardev.c 702-707](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L702-L707)[chardev_private.h 29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L29-L29)

## Architecture

### Two-Level Lock Structure

**Diagram: Resource Lock Architecture**

The architecture implements ownership tracking at two levels. Each file descriptor maintains a local bitmap of locks it holds (`chardev_private.resource_lock`), while the device maintains a global bitmap of all held locks (`tenstorrent_device.resource_lock`). This design enables fast local ownership checks and automatic cleanup when file descriptors close.

Sources: [chardev.c 95](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L95-L95)[chardev.c 288-289](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L288-L289)[chardev.c 304-305](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L304-L305)[chardev_private.h 29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L29-L29)

## Lock Operations

### Operation Types

The `TENSTORRENT_IOCTL_LOCK_CTL` IOCTL supports four operation types specified in the `flags` field of `tenstorrent_lock_ctl_in`:

| Flag | Value | Behavior | Return Value (`out.value`) |
| --- | --- | --- | --- |
| `TENSTORRENT_LOCK_CTL_ACQUIRE` | 0 | Non-blocking acquire | 1 = acquired, 0 = held by another |
| `TENSTORRENT_LOCK_CTL_RELEASE` | 1 | Release held lock | 1 = released, 0 = not held by us |
| `TENSTORRENT_LOCK_CTL_TEST` | 2 | Check lock status | Bit 0 = held by us, Bit 1 = held by any |
| `TENSTORRENT_LOCK_CTL_ACQUIRE_BLOCKING` | 3 | Blocking acquire | 1 = acquired, -ERESTARTSYS if interrupted |

Sources: [ioctl.h 209-213](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L209-L213)[chardev.c 286-318](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L286-L318)

### IOCTL Structure

The IOCTL uses `struct tenstorrent_lock_ctl`[ioctl.h 244-247](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L244-L247) which contains input and output sub-structures.

**Input Structure (`tenstorrent_lock_ctl_in`):**

*   `output_size_bytes`: Size of output buffer for versioning.
*   `flags`: Operation type (ACQUIRE, RELEASE, TEST, ACQUIRE_BLOCKING).
*   `index`: Lock index (0-63).

**Output Structure (`tenstorrent_lock_ctl_out`):**

*   `value`: Operation-specific return value.

Sources: [ioctl.h 232-242](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L232-L242)

## Operation Details

### Non-Blocking Acquire (`ACQUIRE`)

**Diagram: Non-Blocking Acquire Flow**

The non-blocking acquire uses atomic `test_and_set_bit()` to attempt acquiring the global lock. If successful, the local lock bit is set to track ownership. The ordering (global then local) ensures the global bit is always set while the local bit is set.

Sources: [chardev.c 287-293](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L287-L293)

### Blocking Acquire (`ACQUIRE_BLOCKING`)

The blocking acquire uses `wait_event_interruptible()` on `tt_dev->resource_lock_waitqueue`[chardev.c 296-298](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L296-L298) It attempts to acquire the lock using `test_and_set_bit()` as the exit condition. If the process is interrupted by a signal while waiting, the IOCTL returns `-ERESTARTSYS`.

Sources: [chardev.c 295-301](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L295-L301)

### Release Operation

Release first clears the local bit using `test_and_clear_bit()`. If the bit was set (meaning this FD held the lock), it clears the global bit and calls `wake_up_interruptible()` to notify waiters [chardev.c 302-310](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L302-L310)

Sources: [chardev.c 302-310](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L302-L310)

### Test Operation

The `TEST` operation returns a two-bit value encoding lock ownership:

*   **Bit 0**: `test_bit(index, priv->resource_lock)` (Held by this FD).
*   **Bit 1**: `test_bit(index, priv->device->resource_lock)` (Held by any FD).

Sources: [chardev.c 311-315](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L311-L315)

## Implementation Details

### Ordering Invariant

The implementation maintains a critical ordering invariant [chardev.c 268-269](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L268-L269):

*   **Acquire**: Set global bit **then** local bit.
*   **Release**: Clear local bit **then** global bit.

This ensures the global bit is always set while the local bit is set, preventing race conditions where the global lock appears free while a file descriptor believes it holds the lock.

### Atomic Operations

All bit operations use Linux kernel atomic bit manipulation functions (`test_and_set_bit`, `test_and_clear_bit`, `set_bit`, `clear_bit`, `test_bit`). These provide the necessary memory barriers for correct multi-processor behavior [chardev.c 288-315](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L288-L315)

## Lifecycle and Cleanup

### Automatic Release on File Descriptor Close

When a file descriptor is closed, the driver's release function `tt_cdev_release()` iterates through the local `resource_lock` bitmap. For every bit set, it clears the corresponding global bit on the device [chardev.c 702-707](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L702-L707)

**Diagram: Automatic Lock Release on Close**

This ensures that abnormal process termination (segfault, kill signal, etc.) automatically releases all held locks, preventing deadlock across processes.

Sources: [chardev.c 702-707](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L702-L707)
