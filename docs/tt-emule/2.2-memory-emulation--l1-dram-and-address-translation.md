This page details how `tt-emule` models the memory hierarchy of Tenstorrent hardware, including per-core L1 SRAM, banked DRAM, and the translation mechanisms that allow JIT-compiled kernels to access these regions using standard host pointers.

## Core Memory Model: L1 SRAM

Each emulated core (represented by the `tt_emule::Core` class) owns a dedicated L1 region in host memory [include/tt\_emule/device.hpp40-42](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L40-L42) The primary convention used throughout the emulator is **host-pointer-everywhere**: every L1 address provided to a kernel (e.g., via `get_write_ptr` or `get_semaphore`) is a real host pointer truncated to a `uint32_t` [docs/l1-emulation.md21-25](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L21-L25)

### Backing Store and L1Pool

For worker cores, L1 memory is allocated using `mmap` with the `MAP_32BIT` flag [docs/l1-emulation.md60-63](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L60-L63) This ensures that the host pointer fits within 32 bits, allowing kernels to perform pointer arithmetic that round-trips losslessly to the host's 64-bit address space [docs/l1-emulation.md24-26](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L24-L26)

To manage address space pressure and ensure alignment, `tt-emule` utilizes an `L1Pool` [include/tt\_emule/l1\_pool.hpp](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/l1_pool.hpp)

* **L1Pool Mode**: A single contiguous `MAP_32BIT` region is carved into 2 MB-aligned slots [docs/l1-emulation.md79-83](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L79-L83)
* **Slot Masking**: Offsets are extracted using `addr & 0x1FFFFF` (where `0x1FFFFF` is the `L1_SLOT_MASK`) [docs/l1-emulation.md81-83](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L81-L83)
* **Fallback**: If the pool is exhausted, cores fall back to individual `MAP_32BIT` mappings [docs/l1-emulation.md143-144](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L143-L144)

### L1 Memory Initialization

L1 memory is initialized using `MAP_ANONYMOUS`, which guarantees the region is zeroed by the OS upon first access [docs/mem-zeros-handling.md80-83](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L80-L83) This matches the silicon behavior where `wzeromem` is called once at boot [docs/mem-zeros-handling.md22-24](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L22-L24) Crucially, `tt-emule` does **not** re-zero L1 between program runs to avoid masking latent kernel bugs that might corrupt firmware-reserved regions [docs/mem-zeros-handling.md63-68](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L63-L68)

**Sources:** `include/tt_emule/device.hpp`, `include/tt_emule/l1_pool.hpp`, `docs/l1-emulation.md`, `docs/mem-zeros-handling.md`

## DRAM Emulation

DRAM banks are emulated as `tt_emule::Core` objects with the `CoreRole::DRAM` role [docs/dram-emulation.md20-21](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L20-L21) Unlike worker cores, DRAM cores do not use `MAP_32BIT` because they are accessed via 64-bit NOC addresses rather than truncated 32-bit pointers [docs/l1-emulation.md61-63](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L61-L63)

### Bank Topology

The DRAM topology is populated at runtime from the `metal_SocDescriptor` [docs/dram-emulation.md22-24](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L22-L24)

* **Channels**: The number of banks matches `get_num_dram_views()` (e.g., 12 for Wormhole N150, 8 for Blackhole) [docs/dram-emulation.md52-53](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L52-L53)
* **Mapping**: The runner populates `dram_bank_to_noc_xy` and `bank_to_dram_offset` tables used by the address generator [docs/dram-emulation.md32-44](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L32-L44)

### Interleaved Address Generation

Kernels use an `InterleavedAddrGen` to map a flat page ID to a specific bank and offset [docs/dram-emulation.md64-67](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L64-L67)

1. **Stripe/Bank Calculation**: `bank_index = id % NUM_DRAM_BANKS` [docs/dram-emulation.md69-70](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L69-L70)
2. **Address Packing**: The bank-local offset is combined with the bank's NOC coordinates into a 64-bit NOC address [docs/dram-emulation.md72-76](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L72-L76)
3. **Resolution**: `__emule_resolve_noc_addr` decodes the NOC address, identifies the target DRAM `Core`, and returns a host pointer via `core->l1_ptr(offset)` [docs/dram-emulation.md90-94](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/dram-emulation.md?plain=1#L90-L94)

**Sources:** `docs/dram-emulation.md`, `tests/dram/test_dram_model.cpp`, `include/jit_hw/api/dataflow/endpoints.h`

## Address Translation & Resolution

The emulator bridges the gap between the kernel's view of memory (offsets or truncated pointers) and the host's physical memory using several key functions.

### Key Translation Functions

| Function | Direction | Description |
| --- | --- | --- |
| `__emule_addr_to_offset(addr)` | Host Ptr → Offset | Encodes a host pointer into an L1 offset. Uses `addr & 0x1FFFFF` in L1Pool mode [docs/l1-emulation.md89](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L89-L89) |
| `__emule_local_l1_to_ptr(addr)` | Offset → Host Ptr | Kernel-side helper. If `addr` looks like an absolute pointer (>= `l1_base`), it is used directly; otherwise, it is treated as an offset from `__emule_bridge_l1` [docs/l1-emulation.md90](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L90-L90) |
| `__emule_resolve_noc_addr(noc)` | NOC → Host Ptr | Decodes 64-bit NOC address. Extracts `(x,y)` to find the `Core`, then uses the lower 36 bits as the offset into that core's memory [docs/l1-emulation.md91](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L91-L91) |
| `__emule_local_l1_ptr(offset)` | Offset → Host Ptr | Legacy `extern "C"` fast path for local L1 access [docs/l1-emulation.md92](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L92-L92) |

## MEM\_ZEROS Handling

The `MEM_ZEROS` region is a 512-byte L1 area reserved by firmware as a source for zero-filling (e.g., `zero_tile`) [docs/mem-zeros-handling.md3-5](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L3-L5)

### Arch-Specific Constants

Because the `tt-emule` JIT include path does not include arch-specific firmware subdirectories, it uses a shim `include/jit_hw/dev_mem_map.h` to provide these constants based on JIT defines [include/jit\_hw/dev\_mem\_map.h8-14](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L8-L14)

| Architecture | `MEM_ZEROS_BASE` | Source |
| --- | --- | --- |
| Wormhole | `0x3280` | [include/jit\_hw/dev\_mem\_map.h42](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L42-L42) |
| Blackhole | `0x32E0` | [include/jit\_hw/dev\_mem\_map.h38](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L38-L38) |
| Quasar | `0xE180` | [include/jit\_hw/dev\_mem\_map.h40](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L40-L40) |

### Preservation of Invariant

The region remains zero because:

1. **Initial Zeroing**: `MAP_ANONYMOUS` ensures it starts at zero [docs/mem-zeros-handling.md80-83](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L80-L83)
2. **Allocator Guard**: The `tt-metal` allocator only hands out addresses above the `l1_unreserved_base`, which is safely above the `MEM_ZEROS` region [docs/mem-zeros-handling.md124-131](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L124-L131)
3. **No Re-zeroing**: By not re-zeroing per program, `tt-emule` ensures that if a kernel accidentally writes to this region, the corruption persists, allowing developers to catch the bug [docs/mem-zeros-handling.md63-68](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L63-L68)

**Sources:** `include/jit_hw/dev_mem_map.h`, `docs/mem-zeros-handling.md`
