## Purpose and Scope

This document describes the `Chip` abstraction layer, which provides a uniform interface for interacting with individual Tenstorrent chips regardless of their connection method (PCIe, JTAG, Ethernet, or Simulation). The Chip abstraction sits between the high-level [Cluster Management](https://deepwiki.com/tenstorrent/tt-umd/2.1-cluster-management) and the low-level [TTDevice Interface](https://deepwiki.com/tenstorrent/tt-umd/2.4-ttdevice-interface), implementing connection-specific memory access strategies and resource management.

For information about multi-chip topology and device enumeration, see [Cluster Descriptor & Topology](https://deepwiki.com/tenstorrent/tt-umd/2.2-cluster-descriptor-and-topology). For architecture-specific device implementations, see [TTDevice Interface](https://deepwiki.com/tenstorrent/tt-umd/2.4-ttdevice-interface).

## LocalChip: Direct Hardware Access

`LocalChip` represents chips with direct PCIe or JTAG connections to the host. These chips have memory-mapped I/O (MMIO) capabilities and manage their own hardware resources.

### LocalChip Construction

`LocalChip` instances are created via the `LocalChip::create` factory method, which initializes the `TLBManager` and `SiliconSysmemManager` if a PCIe device is present [device/chip/local_chip.cpp 44-62](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L44-L62)

**Sources:**[device/chip/local_chip.cpp 44-62](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L44-L62)

### Resource Management Components

#### TLBManager and SysmemManager

`LocalChip` owns a `TLBManager` to manage PCIe Translation Lookaside Buffer (TLB) windows and a `SysmemManager` for host memory (hugepages) [device/api/umd/device/chip/local_chip.hpp 86-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/chip/local_chip.hpp#L86-L87)

*   **TLBManager**: Handles mapping of address ranges to PCIe BAR windows [device/chip/local_chip.cpp 56](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L56-L56)
*   **SysmemManager**: Manages hugepage allocation and pinning for device DMA [device/chip/local_chip.cpp 57](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L57-L57)

**Sources:**[device/chip/local_chip.cpp 44-62](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L44-L62)[device/api/umd/device/chip/local_chip.hpp 86-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/chip/local_chip.hpp#L86-L87)

### Device Lifecycle

#### Device Startup

`void LocalChip::start_device(uint32_t dram_membar_subchannel)`
The startup sequence:

1.   Acquires a `CHIP_IN_USE` robust mutex to prevent concurrent driver access [device/chip/local_chip.cpp 147](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L147-L147)
2.   Pins or maps system memory to the device [device/chip/local_chip.cpp 149](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L149-L149)
3.   Initializes PCIe iATUs if not supported by the kernel driver [device/chip/local_chip.cpp 152](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L152-L152)
4.   Resets and initializes memory barrier flags on Tensix, Ethernet, and DRAM cores [device/chip/local_chip.cpp 111-129](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L111-L129)[device/chip/local_chip.cpp 154](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L154-L154)

**Sources:**[device/chip/local_chip.cpp 139-155](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L139-L155)

#### Device Shutdown

`void LocalChip::close_device()`
Shutdown performs:

1.   Sets power state to `LONG_IDLE`[device/chip/local_chip.cpp 162](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L162-L162)
2.   Asserts RISC reset on all cores [device/chip/local_chip.cpp 163](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L163-L163)
3.   Unpins/unmaps system memory [device/chip/local_chip.cpp 166](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L166-L166)
4.   Clears mapped TLBs [device/chip/local_chip.cpp 169](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L169-L169)
5.   Releases the `CHIP_IN_USE` lock [device/chip/local_chip.cpp 172](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L172-L172)

**Sources:**[device/chip/local_chip.cpp 157-173](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L157-L173)

## RemoteChip: Ethernet-Routed Access

`RemoteChip` represents chips accessed via Ethernet routing through a gateway `LocalChip`.

### RemoteChip Construction

`RemoteChip` is created using the `RemoteChip::create` factory method, which sets up `RemoteCommunication` and a corresponding `TTDevice`[device/chip/remote_chip.cpp 31-50](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L31-L50)

**Sources:**[device/chip/remote_chip.cpp 31-50](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L31-L50)

### Memory Access and Limitations

RemoteChip delegates all memory operations (`write_to_device`, `read_from_device`) to its internal `TTDevice`, which routes commands through the Ethernet fabric [device/chip/remote_chip.cpp 76-82](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L76-L82)

**Limitations:**

*   Does not support direct DMA (`dma_write_to_device` throws `RuntimeError`) [device/chip/remote_chip.cpp 92-94](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L92-L94)
*   Does not support system memory access (`write_to_sysmem` throws `RuntimeError`) [device/chip/remote_chip.cpp 124-126](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L124-L126)
*   No local `TLBManager` or `SysmemManager`[device/chip/remote_chip.cpp 148-154](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L148-L154)

**Sources:**[device/chip/remote_chip.cpp 76-154](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L76-L154)

## Simulation and Mock Chips

### SimulationChip

`SimulationChip` provides an interface for chips running in simulation (either RTL simulation or software-based TTSim).

*   **Backend Delegation**: Backend variance (RTL subprocess vs TTSim `.so`) is handled by the `TTDevice` subtype held by the chip [device/api/umd/device/simulation/simulation_chip.hpp 36-38](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/simulation/simulation_chip.hpp#L36-L38)
*   **Multicast Workaround**: Currently implements `noc_multicast_write` as a series of unicast writes to each core in the range [device/simulation/simulation_chip.cpp 104-125](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_chip.cpp#L104-L125)
*   **Locking**: Uses a `device_lock` mutex to ensure safe multi-threaded access [device/simulation/simulation_chip.cpp 61](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_chip.cpp#L61-L61)

**Sources:**[device/simulation/simulation_chip.cpp 23-54](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_chip.cpp#L23-L54)[device/simulation/simulation_chip.cpp 99-125](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_chip.cpp#L99-L125)

### MockChip

`MockChip` is used for testing purposes where hardware interaction is not required. All I/O operations are empty stubs [device/chip/mock_chip.cpp 18-55](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/mock_chip.cpp#L18-L55)
