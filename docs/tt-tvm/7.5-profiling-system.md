## Purpose and Scope

The TVM Profiling System provides infrastructure for measuring and analyzing runtime performance of compiled models. It collects per-operator timing, hardware performance counters (PAPI), and custom metrics during model execution. The system integrates with all three TVM execution engines: Graph Executor, VM Executor, and AOT Executor.

**Related pages:** For information about the runtime execution engines themselves, see [Graph Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.1-graph-executor), [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor), and [AOT Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.3-aot-executor). For auto-tuning workflows that use profiling data, see [Auto-Tuning System](https://deepwiki.com/tenstorrent/tt-tvm/8-auto-tuning-system).

## Architecture Overview

The profiling system consists of three main layers: timer infrastructure, profiler coordination, and executor integration.

**Key Components:**

*   **Timer Infrastructure**: Device-aware timing with automatic fallback to default timer
*   **Profiler**: Coordinates profiling sessions, manages call stack, invokes metric collectors
*   **Executors**: Hook into profiler at operator boundaries
*   **Report**: Aggregates and formats profiling results

## Timer Infrastructure

The timer system provides device-specific timing with automatic synchronization. Each device type can register its own timer implementation for accurate measurement.

### Timer Class Hierarchy

### Timer Registration and Selection

The `Timer::Start(Device)` static factory method selects the appropriate timer based on device type:

1.   Looks up registered timer via `Registry::Get("profiling.timer." + device_type)`
2.   Falls back to `DefaultTimer` if no device-specific timer exists
3.   Automatically starts the timer before returning

**Key Implementation Details:**

*   `DefaultTimerNode` calls `TVMSynchronize()` before starting/stopping to ensure accurate measurement across asynchronous device operations
*   Device-specific timers (e.g., CUDA) avoid unnecessary synchronization overhead
*   Warning is logged once per unseen device type when falling back to default timer

## Profiler Class

The `Profiler` class coordinates profiling sessions by managing a call stack and invoking metric collectors.

### Profiler State Machine

### CallFrame Structure

Each profiling call is represented by a `CallFrame`:

| Field | Type | Description |
| --- | --- | --- |
| `dev` | `Device` | Device where operation executes |
| `name` | `String` | Operation name (e.g., "fused_nn_conv2d") |
| `timer` | `Timer` | Timer started for this call |
| `extra_metrics` | `Map<String, ObjectRef>` | Additional metrics (shapes, counts, etc.) |
| `extra_collectors` | `Vector<Pair<Collector, ObjectRef>>` | Active collector state |

**Profiling Workflow:**

1.   **Initialization**: `Profiler(devices, collectors, configuration)`

    *   Resets thread pool to ensure PAPI hooks are set in all threads
    *   Stores configuration metadata (number of threads, etc.)

2.   **Start**: `Start()`

    *   Sets `is_running_ = true`
    *   Calls `StartCall("Total", dev)` for each device to measure total time

3.   **Per-Operation**: `StartCall(name, dev, extra_metrics)` and `StopCall(extra_metrics)`

    *   Creates timer and invokes `collector->Start(dev)` for each collector
    *   Pushes `CallFrame` onto `in_flight_` stack
    *   On stop, collects metrics from collectors and moves frame to `calls_` vector

4.   **Finalization**: `Stop()`

    *   Calls `StopCall()` for each device to close "Total" measurements
    *   Sets `is_running_ = false`

5.   **Report Generation**: Internally constructs `Report` from `calls_` vector

## Report Generation

The `Report` class aggregates profiling data and provides multiple output formats.

### Report Structure

### Output Formats

**CSV Format** (`AsCSV()`):

*   Header row with all unique metric names across calls
*   One row per call with metric values
*   Missing metrics left empty
*   Numbers formatted without thousand separators for parsing

**JSON Format** (`AsJSON()`):

`{  "calls": [    {      "Name": {"string": "fused_add"},      "Duration (us)": {"microseconds": 123.45},      "Count": {"count": 1},      "Hash": {"string": "abc123"}    }  ],  "device_metrics": {    "cpu": {"Total Duration": {"microseconds": 500.0}}  },  "configuration": {    "Executor": {"string": "VM"},    "Number of threads": {"count": 4}  }}`
**Table Format** (`AsTable(aggregate, sort, compute_col_sums)`):

*   Human-readable ASCII table with columns aligned
*   Optional aggregation by operation hash + argument shapes
*   Optional sorting by duration (descending)
*   Optional column summations for numeric metrics

### Metric Types

| Type | Usage | Aggregation |
| --- | --- | --- |
| `DurationNode` | Timing in microseconds | Sum |
| `CountNode` | Operation counts | Sum |
| `PercentNode` | Percentage values | Sum |
| `RatioNode` | Ratios/rates | Average |
| `String` | Names, hashes | First (if all equal) |

## MetricCollector System

The `MetricCollector` interface allows custom metrics beyond basic timing.

### Collector Interface

### PAPI Integration

The PAPI (Performance Application Programming Interface) collector measures hardware performance counters.

**Supported Metrics (examples):**

*   CPU: `PAPI_FP_OPS`, `PAPI_TOT_CYC`, `PAPI_L1_DCM`, `PAPI_L2_DCM`
*   CUDA: `cuda:::event:shared_load:device=0`, `cuda:::gpu__compute_memory_access_throughput.max.pct_of_peak_sustained_region:device=0`

**Usage Pattern:**

`from tvm.runtime.profiling import PAPIMetricCollector collectors = [    PAPIMetricCollector({        tvm.cpu(): ["PAPI_FP_OPS"],        tvm.cuda(): ["cuda:::event:shared_load:device=0"]    })] report = vm.profile(data, collectors=collectors)`
**Implementation Notes:**

*   Thread pool is reset during `Profiler` initialization to ensure PAPI event sets are registered in all worker threads
*   Collectors maintain per-device state
*   Metrics are automatically added to the report alongside timing data

## Integration with VM Executor

The VM integrates profiling through the `VirtualMachineDebug` subclass, which overrides operator execution hooks.

### VM Profiling Architecture

### Profiled VM Instructions

The VM profiles various instruction types with operation-specific names:

| Instruction | Profiling Name | Metrics Collected |
| --- | --- | --- |
| `Opcode::LoadConst` | `"VM::LoadConst"` | Device index |
| `Opcode::DeviceCopy` | `"VM::DeviceCopy"` | Source/destination devices |
| `Opcode::ReshapeTensor` | `"VM::ReshapeTensor"` | - |
| `Opcode::AllocTensor` | `"VM::AllocTensor"` | Argument shapes, dtype |
| `Opcode::AllocTensorReg` | `"VM::AllocTensorReg"` | Argument shapes (from register), dtype |
| `Opcode::AllocStorage` | `"VM::AllocStorage"` | Size, dtype hint, device |
| `Opcode::InvokePacked` | Primitive function name | Argument shapes (extracted from NDArrays) |

**Key Implementation:**

*   `OpStartHook()` and `OpStopHook()` wrap each instruction execution
*   Packed function calls use the actual primitive name from `packed_index_map_`
*   Shape information is extracted from tensors for better profiling granularity

## Integration with Graph Executor

The Graph Executor provides profiling through `GraphExecutorDebug`, which can measure individual operators or the entire graph.

### Graph Executor Profiling Flow

### Profiling Modes

**Individual Operator Profiling** (`RunIndividual()`):

*   Measures each operator separately with multiple repeats
*   Uses `WrapTimeEvaluator()` for accurate measurement
*   Supports min_repeat_ms, cooldown intervals
*   Returns timing blob with per-operator, per-iteration data

**Integrated Profiling** (`profile()`):

*   Uses `Profiler` class for unified profiling
*   Collects per-operator timing with hardware counters
*   Generates structured `Report` object
*   Accessible via `debug_executor.profile()` in Python

**RPC Support:**

*   `RunOpRPC()` delegates timing to remote machine
*   Uses `runtime.RPCTimeEvaluator` for remote measurement
*   Avoids synchronization overhead of local timing

## Implementation Details

### Synchronization Strategy

The profiling system carefully manages synchronization to avoid measurement overhead:

1.   **Timer Start/Stop**: Lightweight operations that record time points
2.   **SyncAndGetElapsedNanos**: Expensive synchronization deferred until after all timers stop
3.   **Device Synchronization**: `TVMSynchronize()` called before starting/stopping default timers

**Rationale**: Starting and stopping multiple timers without intervening synchronization allows parallel timing of overlapping operations.

### Thread Safety

The profiling system is **not thread-safe**. Key considerations:

*   `Profiler::in_flight_` stack assumes single-threaded call/return pattern
*   Thread pool is reset during initialization to register PAPI hooks in worker threads
*   Parallel operator execution is timed as a unit, not per-thread

### Performance Overhead

Profiling introduces overhead from:

*   Timer start/stop calls (~1-10 μs depending on device)
*   Metric collector invocations (PAPI: ~1-5 μs)
*   Stack operations and bookkeeping (~0.1 μs)
*   Synchronization barriers (device-dependent: 1-100 μs)

**Mitigation:**

*   Use device-specific timers to avoid unnecessary synchronization
*   Warmup runs (typically 3) before profiling to eliminate JIT/cache effects
*   Minimal per-call bookkeeping during profiling; expensive processing deferred to report generation

### Memory Management

*   `CallFrame` objects stored in `std::vector<CallFrame> calls_`
*   Timers hold device synchronization state until `SyncAndGetElapsedNanos()`
*   Report generation aggregates data without copying raw timing arrays
*   PAPI collectors maintain per-thread event sets (managed by PAPI library)

## Build Configuration

Profiling features are controlled by CMake options:

| Option | Default | Impact |
| --- | --- | --- |
| `USE_PROFILING` | ON | Enables profiling infrastructure |
| `USE_PAPI` | OFF | Enables PAPI hardware counter support |
| `USE_VM_PROFILER` | ON | Enables VM profiling hooks |

**Conditional Compilation:**

*   PAPI collectors only available when `USE_PAPI=ON`
*   VM profiler requires `VirtualMachineDebug` linked into runtime
