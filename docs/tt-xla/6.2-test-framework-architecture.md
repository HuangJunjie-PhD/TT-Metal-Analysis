## High-Level Architecture

The test framework is built on pytest and consists of four major subsystems:

1.   **Test Discovery**: Dynamically discovers model loaders from `third_party/tt_forge_models`
2.   **Test Execution**: Runs models on both CPU and TT hardware using framework-specific testers
3.   **Result Validation**: Compares outputs using PCC, ATOL, and allclose metrics
4.   **Failure Analysis**: Classifies failures using pattern matching on exceptions and logs

**Architecture Overview**: The test framework follows a layered architecture. At the top, `DynamicLoader` classes discover models from the `tt_forge_models` repository. Test entry points in `test_models.py` parametrize these models with run modes (inference/training) and parallelism modes (single-device/data-parallel/tensor-parallel). The execution layer uses framework-specific `ModelTester` subclasses to compile and run models on both CPU and TT hardware. Results flow through the validation layer (`Evaluator` and `ComparisonResult`) which applies PCC and ATOL thresholds. On failure, the `FailingReasonsFinder` classifies exceptions using 50+ predefined patterns.

### Discovery Process

**Discovery Flow**: At pytest collection time, the `DynamicLoader.setup_test_discovery()` method walks the `tt_forge_models` directory tree looking for framework-specific `loader.py` files. Each loader is dynamically imported using `importlib.util.spec_from_file_location()` with proper module packaging setup to support relative imports. The loader's `query_available_variants()` method returns available model configurations (e.g., different sizes or quantization levels). Each variant becomes a `ModelTestEntry` containing the loader path, variant tuple, and framework identifier.

## Test Execution Pipeline

The test execution follows a standardized pipeline regardless of framework:

**Execution Flow**: The `_run_model_test_impl()` function in `test_models.py` orchestrates the test lifecycle. It first fixes venv isolation issues and installs per-model requirements using `RequirementsManager`. The loader instantiates the model and inputs, which are wrapped in a framework-specific tester. The tester's `test()` method (inherited from `ModelTester`) runs the core pipeline: cache inputs, initialize a `Workload` object, compile for CPU (using torch.compile with inductor backend), run on CPU, compile for TT device (using torch_xla/JAX backend), run on TT device, and compare results. The `Evaluator` validates comparison results against configured thresholds. Exceptions are caught and analyzed by `FailingReasonsFinder`. Finally, `record_model_test_properties()` is always called to record test metadata for CI reporting.

## Test Entry Points and Parameterization

### Primary Test Functions

The framework provides four main test entry points in `test_models.py`:

| Test Function | Purpose | Parametrization |
| --- | --- | --- |
| `test_all_models_torch()` | PyTorch models (general) | `test_entry` × `run_mode` (inference/training) × `parallelism` (single/data/tensor) |
| `test_all_models_jax()` | JAX models | `test_entry` × `run_mode` (inference/training) × `parallelism` (single/data/tensor) |
| `test_llms_torch()` | PyTorch LLMs (decode/prefill) | `test_entry_and_phase` × `sequence_length` × `batch_size` × `run_mode` × `parallelism` |
| `test_all_models_op_by_op()` | Op-by-op validation | `test_entry` × `run_mode` × `parallelism` (single-device only) |

### Parametrization Details

**test_all_models_torch()** and **test_all_models_jax()**: These tests use standard pytest parametrization with discovered test entries from `TorchDynamicLoader` and `JaxDynamicLoader`. Each test entry is crossed with run modes (inference/training) and parallelism modes (single-device, data-parallel, tensor-parallel). The `DynamicLoader.create_test_id_generator()` method generates human-readable test IDs like `vision/resnet/pytorch-resnet50-inference-single_device`.

**test_llms_torch()**: This specialized test focuses on LLM workloads with decode and prefill phases. It filters test entries to only those with `load_inputs_decode()` or `load_inputs_prefill()` methods and generates separate test cases for each phase. Additional parametrization includes sequence lengths (128, 1024, 2048, 4096, 8192) and batch sizes (1, 2) for prefill tests.

**test_all_models_op_by_op()**: This test spawns a subprocess running the normal model test with `--dump-irs` flag to collect StableHLO IR. It then parses the IR, splits it into individual operations, and tests each operation in isolation. This enables fine-grained debugging of compilation failures.

### Bringup Status Tracking

The `BringupStatus` enum tracks how far a test progressed before failure:

*   `NOT_STARTED`: Placeholder for models not yet integrated
*   `FAILED_FE_COMPILATION`: Failed during frontend passes (VHLO→SHLO)
*   `FAILED_TTMLIR_COMPILATION`: Failed during TTMLIR passes (SHLO→TTIR→TTNN)
*   `FAILED_RUNTIME`: Failed during tt-metal runtime execution
*   `INCORRECT_RESULT`: Execution succeeded but output comparison failed
*   `PASSED`: Test passed all checks
*   `UNKNOWN`: Failure stage could not be determined

The C++ compilation pipeline writes stage markers to `._bringup_stage.txt` when `ENABLE_BRINGUP_STAGE_LOGGING=1` is set. The `parse_last_bringup_stage()` function reads this file to determine the failure stage.

## Result Validation and Comparison

### ComparisonResult Flow

The validation pipeline processes comparison results through multiple stages:

**Comparison Validation**: The tester's `_compare()` method returns a list of `ComparisonResult` objects (one per output tensor). The `Evaluator._assert_on_results()` method checks if all comparisons passed and raises an assertion if not. Regardless of outcome, `_process_comparison_results()` analyzes the results to determine `BringupStatus`. If any PCC value is below the threshold, the status is `INCORRECT_RESULT`. The `_get_first_failing_or_last_result()` helper selects the first failed comparison for reporting, or the last comparison if all passed.

**Guidance Tags**: The framework generates guidance tags for CI automation:

*   `ENABLE_PCC` / `ENABLE_PCC_099`: PCC check is disabled but measured PCC is safely above threshold
*   `RAISE_PCC` / `RAISE_PCC_099`: Measured PCC exceeds the next centesimal step
*   `RM_XFAIL`: Test marked as KNOWN_FAILURE_XFAIL but is passing
*   `ADD_CONFIG`: Test marked as UNSPECIFIED but is passing

These tags help identify tests that need configuration updates.

## Failure Classification System

### FailingReasonsFinder Architecture

The failure classification system automatically triages test failures by matching exception patterns:

**Classification Process**: When a test raises an exception, the `update_test_metadata_for_exception()` function captures stdout/stderr and builds an `ExceptionData` object containing the exception class name, message, and full traceback. The `FailingReasonsFinder.find_reason_by_exception()` method converts the exception to `ExceptionData` and calls `find_reason_by_ex_data()`. This iterates through all patterns in the `FailingReasons` enum, calling `FailingReason.check()` on each. An `ExceptionCheck` matches if all its conditions pass: class name matches, component checker passes, and all message/log/stdout/stderr matchers return true. The first matching pattern is returned as an `ExceptionReason` with the failing reason enum and extracted summary.

### Component Checkers

The `ComponentChecker` enum identifies which stack component caused the failure by analyzing stack traces:

| Component | Description | Detection Heuristics |
| --- | --- | --- |
| `XLA` | XLA/torch_xla frontend | Last line contains `tt_torch/backend/`, no Metal/TTNN libs in trace |
| `METAL` | tt-metal runtime | Stack contains `lib/libtt_metal.so` |
| `TTNN` | TTNN operations | Stack contains `lib/_ttnncpp.so` or `lib/libTTMLIRCompiler.so`, no Metal |
| `TORCH` | Pure PyTorch issues | No XLA/Metal/TTNN libs, last line in torch utils |
| `SWEEPS` | Test infrastructure | Last line in test/operators/utils |

### Failing Reason Patterns

The `FailingReasons` enum defines 50+ specific failure patterns. Key categories include:

**Data Mismatch**:

*   `DATA_MISMATCH_WRONG_PCC`: PCC below threshold
*   `DATA_MISMATCH_PCC_IS_NAN`: PCC is NaN (invalid comparison)
*   `DATA_MISMATCH_ALL_CLOSE`: Allclose check failed
*   `DATA_MISMATCH_PCC_LOW_RANGE` / `MEDIUM_RANGE` / `HIGH_RANGE`: PCC in specific range

**Compilation Failures**:

*   `FATAL_ERROR`: RuntimeError with "Fatal error" message
*   `SEG_FAULT`: Segmentation fault during compilation/execution
*   `XLA_SYNC_MULTI_FATAL_ERROR`: Fatal error in xla_sync_multi call

**Resource Issues**:

*   `OUT_OF_MEMORY`: Allocation failure with detailed memory report
*   `HIGH_MEMORY`: Memory usage too high
*   `DATA_TYPE_NOT_SUPPORTED`: Unsupported tensor dtype

**Configuration Issues**:

*   `NOT_SUPPORTED_AND_SKIPPED`: Model marked as not supported
*   `INCORRECT_RESULT_PCC_DISABLED`: Test shows incorrect result but PCC check is disabled
*   `TOKENIZER_CHAT_TEMPLATE_NOT_SET`: Missing chat template for LLM
*   `SEGMENTATION_MODELS_PYTORCH_NOT_FOUND`: Missing dependency

**Unknown**:

*   `UNCLASSIFIED`: No pattern matched the exception

Each pattern includes detailed `ExceptionCheck` objects with class name, component, and multiple message matchers. Some patterns also include summary extractors using regex to pull out specific error details.

### MessageChecker DSL

The `MessageChecker` class (aliased as `M`) provides a combinator library for building exception matchers:

| Method | Purpose | Example |
| --- | --- | --- |
| `M.contains(s)` | Check if string contains substring | `M.contains("Out of Memory")` |
| `M.starts_with(s)` | Check if string starts with substring | `M.starts_with("Comparison result 0 failed")` |
| `M.equals(s)` | Check if string equals exactly | `M.equals("Error code: 13")` |
| `M.regex(pattern)` | Match regex pattern | `M.regex(r"PCC=\d+\.\d+")` |
| `M.any(*checkers)` | Logical OR of checkers | `M.any(M.contains("lib/a.so"), M.contains("lib/b.so"))` |
| `M.neg(checker)` | Logical NOT | `M.neg(M.contains("lib/libtt_metal.so"))` |
| `M.last_line(checker)` | Apply checker to last line only | `M.last_line(M.contains("infra/"))` |

These combinators enable expressive pattern matching without complex logic:

## Pytest Integration

### Fixtures and Plugins

The test framework integrates with pytest through fixtures defined in `conftest.py` (not provided but referenced in code):

**test_metadata**: Fixture that loads `ModelTestConfig` for the current test from YAML files based on test ID and architecture. This provides per-test configuration without hardcoding.

**captured_output_fixture**: Fixture that captures stdout/stderr during test execution for failure analysis. Implemented using `pytest.capture_fixture`.

**clear_torchxla_computation_cache**: Fixture that clears torch_xla's computation cache between tests to ensure clean state.

**record_property**: Built-in pytest fixture for recording test properties that appear in JUnit XML output. Used by `record_model_test_properties()`.

### Custom Markers

Tests are marked with `@pytest.mark.no_auto_properties` to prevent pytest's automatic property recording, since `record_model_test_properties()` provides complete custom reporting.

The `@pytest.mark.filecheck` marker (applied dynamically when `test_metadata.filechecks` is non-empty) triggers FileCheck validation of generated MLIR against pattern files.

### Test Collection Hooks

The test framework uses pytest's collection hooks to:

1.   Discover models dynamically using `DynamicLoader.setup_test_discovery()`
2.   Generate parametrized test cases for all discovered models
3.   Apply markers based on test metadata (status, markers, supported_archs)
4.   Filter tests based on architecture and markers
