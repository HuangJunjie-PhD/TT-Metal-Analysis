## Purpose and Scope

This document describes **Test Plan (TP) Mode** of RiescueC, which generates directed tests from test plans defined in the `riscv-coretp` library. TP mode targets architectural and privilege scenarios by consuming structured test scenarios and solving for appropriate execution environments.

For datapath instruction compliance testing, see [Bringup Mode](https://deepwiki.com/tenstorrent/riescue/2.2.1-bringup-mode). For general information about RiescueC, see [RiescueC: Compliance Test Generator](https://deepwiki.com/tenstorrent/riescue/2.2-riescuec:-compliance-test-generator).

* * *

## Overview

TP mode is a compliance test generation mode that:

*   Consumes **TestPlans** from the `riscv-coretp` library
*   Defines architectural test scenarios as structured Python data
*   Solves for execution environments using constraint predicates
*   Generates test assembly using RiescueD's underlying framework
*   Supports privilege modes, paging, virtualization, and exception delegation

## Test Plans and Scenarios

### Test Plan Concept

A **TestPlan** is a structured definition of test scenarios for a specific RISC-V extension or architectural feature. Test plans are maintained in the `riscv-coretp` library and accessed through the plan registry.

Each test plan contains:

*   **TestScenarios**: Sequences of instructions with environmental constraints
*   **Extension requirements**: ISA features needed for the test
*   **Environment constraints**: Privilege modes, paging modes, virtualization requirements

### Accessing Test Plans

Test plans are retrieved via the `get_plan()` function from `coretp.plans.test_plan_registry`:

The `TpMode.generate()` method retrieves test plans at generation time [riescue/compliance/tp.py 84-86](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L84-L86)

### TestScenario Structure

A **TestScenario** describes:

*   Instruction sequences to execute
*   Register initialization requirements
*   Memory state requirements
*   Expected outcomes and side effects
*   Environmental constraints (privilege, paging, etc.)

The `TestPlanGenerator` processes these scenarios and generates corresponding assembly code.

## Environment Solving with Predicates

### Predicate Concept

**Predicates** are constraint functions that filter valid execution environments for test scenarios. Each predicate takes a `TestEnv` object and returns `True` if the environment satisfies the constraint.

The environment solving process:

1.   Test scenarios define required environmental features
2.   Predicates filter valid environments based on `FeatMgr` configuration
3.   Solver selects a compatible environment
4.   Generator creates assembly for the selected environment

### Predicate Implementation

The following diagram shows the predicate evaluation flow:

### Available Predicates

The `TpMode.get_predicates()` method creates these predicate functions [riescue/compliance/tp.py 139-182](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L139-L182):

#### 1. Privilege Mode Predicate

Filters environments matching the configured privilege mode:

*   Machine mode: `env.priv == PrivilegeMode.M`
*   Supervisor mode: `env.priv == PrivilegeMode.S`
*   User mode: `env.priv == PrivilegeMode.U`

#### 2. Paging Mode Predicate

Filters environments matching the configured paging mode:

*   Bare mode: `env.paging_mode == PagingMode.DISABLED`
*   Sv39: `env.paging_mode == PagingMode.SV39`
*   Sv48: `env.paging_mode == PagingMode.SV48`
*   Sv57: `env.paging_mode == PagingMode.SV57`

#### 3. Virtualization Predicate

Filters environments based on virtualization:

*   Virtualized: `env.virtualized == True`
*   Non-virtualized: `env.virtualized == False`

#### 4. Exception Delegation Predicate

Filters environments matching exception delegation configuration:

*   Delegate to machine mode: `env.deleg_excp_to == PrivilegeMode.M`
*   Delegate to supervisor mode: `env.deleg_excp_to == PrivilegeMode.S`

## Integration with RiescueD

TP mode leverages the RiescueD framework for compilation and simulation. The integration provides:

### Assembly Processing

*   RiescueD's `Parser` processes TP-generated assembly [riescue/riescued.py 76-86](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L76-L86)
*   Directives like `;#random_data()`, `;#page_mapping()` are resolved

### Feature Management

*   `FeatMgr` from TP configuration is passed to RiescueD [riescue/compliance/tp.py 104](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L104-L104)
*   Environmental settings (privilege, paging) are preserved

### Compilation

*   Uses RISC-V toolchain via `Toolchain.compiler`[riescue/riescued.py 281-298](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L281-L298)
*   Generates ELF executables with proper linker scripts

### Simulation

*   Runs on Whisper ISS by default [riescue/compliance/tp.py 108-111](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L108-L111)
*   Configures ISS with appropriate privilege and paging settings

This reuse of RiescueD's infrastructure ensures consistency between TP mode tests and manually written RiescueD tests.

**Sources:**[riescue/compliance/tp.py 103-112](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L103-L112)[riescue/riescued.py 76-86](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L76-L86)[riescue/riescued.py 239-267](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L239-L267)
