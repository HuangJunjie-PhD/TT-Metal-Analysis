Each Network Interface Unit (NIU) provides hardware counters for monitoring transaction progress and enforces address alignment constraints that vary by transaction type. The counter array at `NIU_BASE + 0x200` enables software to poll for completion of asynchronous NoC operations without requiring interrupt-driven notification. Alignment requirements are enforced at packet formation time and violations result in undefined hardware behavior.

This page documents:

*   Counter array organization and access patterns
*   Counter increment/decrement semantics for each transaction type
*   Address alignment rules for reads, writes, and atomics
*   Hardware limitations and known bugs affecting counter behavior

Related pages: [NoC Memory Map and NIU Interface](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface), [Transaction Types and Routing](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing), [Virtual Channels and Ordering Guarantees](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.3-virtual-channels-and-ordering-guarantees).

## NoC Counter System

### Counter Array Organization

Each NIU exposes 62 counters as read-only 32-bit registers in a 256-byte MMIO region starting at `NIU_BASE + 0x200` (indices 0-61 are defined; indices 62-63 are reserved). For Tensix and Ethernet tiles, the base addresses are:

*   NoC #0 NIU: `0xFFB2_0200`
*   NoC #1 NIU: `0xFFB3_0200`

The counter array tracks transaction lifecycle events including command acceptance, packet transmission, data word movement, and response arrival.

**Sources:**[WormholeB0/NoC/Counters.md 1-39](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L1-L39)[WormholeB0/NoC/MemoryMap.md 20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L20-L20)

### Counter Types and Behavior

The 62 defined counters fall into three functional categories with different width and overflow characteristics:

| Category | Indices | Behavior | Width | Overflow Behavior |
| --- | --- | --- | --- | --- |
| **Global Event Counters** | 0-15, 48-61 | Increment only | 32 bits | Wraps at 2³² (4,294,967,296) |
| **Outstanding Request Counters** | 16-31 | Increment and decrement | 8 bits | Software bug if over/underflows |
| **Outgoing Write Counters** | 32-47 | Increment and decrement | 8 bits | Software bug if over/underflows |

The 8-bit counters at indices 16-47 are organized as 16 pairs, each pair indexed by the 4-bit `NOC_PACKET_TRANSACTION_ID` field in bits [13:10] of the `NOC_PACKET_TAG` register. Counter at index `16 + i` tracks outstanding requests for transaction ID `i`, while counter at index `32 + i` tracks outgoing writes for transaction ID `i`.

```
Counter[16 + txn_id] = NIU_MST_REQS_OUTSTANDING_ID(txn_id)     // 0 ≤ txn_id ≤ 15
Counter[32 + txn_id] = NIU_MST_WRITE_REQS_OUTGOING_ID(txn_id)  // 0 ≤ txn_id ≤ 15
```

The 8-bit width limits the maximum number of simultaneously outstanding operations per transaction ID to 255. Software must ensure that increments do not cause overflow (exceeding 255) and that decrements do not cause underflow (going below 0), as either condition indicates a software bug. For large transfers using automatic request splitting, the initial increment is `ceil(length / 8192)`, which effectively limits per-transaction transfers to approximately 2 MiB (255 × 8192 bytes).

**Sources:**[WormholeB0/NoC/Counters.md 5-39](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L5-L39)[WormholeB0/NoC/Counters.md 167-174](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L167-L174)

### Transaction ID Indexing

The 4-bit `NOC_PACKET_TRANSACTION_ID` field in the `NOC_PACKET_TAG` register selects which of 16 transaction ID counter pairs to manipulate. Software can use different transaction IDs to track different classes of requests independently.

> **Hardware Bug (WH-B0):** When initiating non-posted write requests (with `NOC_CMD_RESP_MARKED=1` in `NOC_CTRL`) to the PCIe tile or ARC tile, the `NOC_PACKET_TRANSACTION_ID` field in bits [13:10] of `NOC_PACKET_TAG` must be set to zero. If non-zero transaction ID `i` is used:
> 
> 
> *   At the initiating NIU: `NIU_MST_REQS_OUTSTANDING_ID(i)` is incremented
> *   At the PCIe/ARC NIU: Write acknowledgement is sent correctly
> *   At the initiating NIU: `NIU_MST_REQS_OUTSTANDING_ID(0)` is decremented instead of `NIU_MST_REQS_OUTSTANDING_ID(i)`
> 
> 
> This leaves counter `i` permanently incremented and counter 0 erroneously decremented (likely underflowing). Workaround: Always use transaction ID 0 for non-posted writes to PCIe/ARC tiles.

**Sources:**[WormholeB0/NoC/MemoryMap.md 145-147](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L145-L147)[WormholeB0/NoC/Counters.md 23-24](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L23-L24)

### Counter Usage Patterns

Software typically polls counters to determine transaction completion. The RISC-V memory model allows loads and stores to be reordered, so an intervening read-back is required to enforce ordering between the `NOC_CMD_CTRL` write and subsequent counter reads.

`// Example: Initiate non-posted write with transaction ID = 3#define NIU0_BASE 0xFFB20000#define NIU0_CMD_BUF_0 (NIU0_BASE + 0x000) volatile uint32_t *noc_packet_tag = (volatile uint32_t*)(NIU0_CMD_BUF_0 + 0x18);volatile uint32_t *noc_ctrl = (volatile uint32_t*)(NIU0_CMD_BUF_0 + 0x1C);volatile uint32_t *noc_cmd_ctrl = (volatile uint32_t*)(NIU0_CMD_BUF_0 + 0x28); volatile uint32_t *counter_outstanding = (volatile uint32_t*)(NIU0_BASE + 0x200 + 4*(16 + 3));volatile uint32_t *counter_outgoing = (volatile uint32_t*)(NIU0_BASE + 0x200 + 4*(32 + 3)); // Configure request (destination, length, etc.)// ... // Set transaction ID = 3 in bits [13:10]*noc_packet_tag = (3 << 10) | /* other flags */; // Set NOC_CMD_RESP_MARKED=1 for non-posted write*noc_ctrl = (2 << 0) | (1 << 4) | /* other flags */; // Initiate request*noc_cmd_ctrl = 1; // Enforce write ordering: read back NOC_CMD_CTRL before polling counters// This prevents RISC-V from reordering the counter read before the NOC_CMD_CTRL write(void)*noc_cmd_ctrl; // Poll for virtual channel assignment (NOC_CMD_CTRL returns to 0)while (*noc_cmd_ctrl & 1) { } // Poll for source data read completion (non-inline writes only)// Counter at [32 + txn_id] tracks number of packets still reading source datawhile (*counter_outgoing != 0) { } // Poll for write acknowledgement arrival (non-posted requests only)// Counter at [16 + txn_id] tracks number of responses not yet receivedwhile (*counter_outstanding != 0) { }`
The read-back of `NOC_CMD_CTRL` creates a data dependency that prevents the RISC-V core from reordering subsequent counter reads ahead of the `NOC_CMD_CTRL` write. Without this read-back, the counter reads could execute before the NIU has incremented the counters, causing software to incorrectly conclude the transaction has already completed.

**Sources:**[WormholeB0/NoC/Counters.md 42-43](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L42-L43)[WormholeB0/NoC/MemoryMap.md 148-157](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L148-L157)

### Clearing Transaction ID Counters

The write-only register at `NIU_BASE + 0x050` clears `NIU_MST_REQS_OUTSTANDING_ID` counters. Each bit in the written value controls one counter:

`// Hardware behavior when writing to (NIU_BASE + 0x050)for (unsigned i = 0; i < 16; ++i) {  if (written_value & (1 << i)) {    NIU_MST_REQS_OUTSTANDING_ID(i) = 0;  // Counter at index [16 + i]  }}`
Example usage:

`#define NIU0_BASE 0xFFB20000#define NIU0_CLEAR_COUNTERS (NIU0_BASE + 0x050) // Clear counters for transaction IDs 3 and 7uint32_t clear_mask = (1 << 3) | (1 << 7);*(volatile uint32_t*)NIU0_CLEAR_COUNTERS = clear_mask;// Result: NIU_MST_REQS_OUTSTANDING_ID(3) = 0//         NIU_MST_REQS_OUTSTANDING_ID(7) = 0`
**When Clearing is Required:**

1.   **Redirected Responses:** When `NOC_RET_ADDR` specifies a different NIU than the initiating NIU:

    *   Initiating NIU: `NIU_MST_REQS_OUTSTANDING_ID(i)` incremented when request starts
    *   Different NIU: Receives response and decrements its own `NIU_MST_REQS_OUTSTANDING_ID(i)`
    *   Result: Both NIUs have non-zero counters; manually clear at initiating NIU

2.   **Broadcast with Acknowledgements:** When `NOC_CMD_BRCST_PACKET=1` and `NOC_CMD_RESP_MARKED=1`:

    *   Initiating NIU: `NIU_MST_REQS_OUTSTANDING_ID(i)` incremented once
    *   Each recipient: Sends write acknowledgement back
    *   Initiating NIU: Decrements counter once per acknowledgement received
    *   Result: Counter goes negative (underflows) after first acknowledgement; manually clear after all acknowledgements

> **Note:** There is no corresponding clear register for `NIU_MST_WRITE_REQS_OUTGOING_ID` counters at indices 32-47. These counters must naturally return to zero through normal transaction completion.

**Sources:**[WormholeB0/NoC/Counters.md 152-166](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L152-L166)[WormholeB0/NoC/MemoryMap.md 17](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L17-L17)

### Automatic Request Splitting

Individual NoC packets are limited to 256 data flits (8192 bytes maximum payload). The NIU can automatically split larger transfers into multiple packets without software intervention.

**Splitting Mechanics:**

When software writes `NOC_CMD_CTRL = 1` with `NOC_AT_LEN_BE > 8192`:

1.   Hardware increments transaction counters by `ceil(NOC_AT_LEN_BE / 8192.0)`:

```
NIU_MST_REQS_OUTSTANDING_ID(txn_id) += ceil(length / 8192.0)  // if non-posted
NIU_MST_WRITE_REQS_OUTGOING_ID(txn_id) += ceil(length / 8192.0)  // if write
```
2.   Hardware assigns virtual channel and initiates first 8192-byte packet

3.   Before clearing `NOC_CMD_CTRL` to 0, hardware modifies the request initiator registers:

```
NOC_AT_LEN_BE -= 8192
NOC_TARG_ADDR_LO += 8192
NOC_RET_ADDR_LO += 8192
```
4.   If `NOC_AT_LEN_BE > 0`, `NOC_CMD_CTRL` remains 1 and steps 2-3 repeat

5.   When `NOC_AT_LEN_BE` reaches ≤ 8192, final packet is sent and `NOC_CMD_CTRL` clears to 0

**Requirements:**

*   `NOC_AT_LEN_BE` at `(NOC_CMD_BUF_BASE + 0x20)` contains total transfer length in bytes
*   `NOC_TARG_ADDR_LO` at `(NOC_CMD_BUF_BASE + 0x00)` must be 32-byte aligned
*   `NOC_RET_ADDR_LO` at `(NOC_CMD_BUF_BASE + 0x0C)` must be 32-byte aligned
*   Software must not write to any `NOC_CMD_CTRL` at the same NIU while splitting is active
*   Software must not modify request initiator registers while `NOC_CMD_CTRL = 1`

**Counter Behavior:**

Each automatically-split packet triggers independent counter updates:

*   `NIU_MST_*_SENT` counters increment once per packet
*   `NIU_SLV_*_RECEIVED` counters increment once per packet at each recipient
*   `NIU_MST_*_OUTSTANDING_ID` and `NIU_MST_WRITE_REQS_OUTGOING_ID` decrement once per packet completion

**Size Limits:**

The 8-bit width of transaction ID counters limits the initial increment:

```
max_increment = 255
max_length = 255 * 8192 = 2,088,960 bytes ≈ 2 MiB
```

Attempting to initiate a transfer larger than 2 MiB with automatic splitting will overflow the 8-bit counter, causing incorrect behavior.

**Sources:**[WormholeB0/NoC/Counters.md 167-174](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L167-L174)[WormholeB0/NoC/MemoryMap.md 122-127](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L122-L127)[WormholeB0/NoC/Counters.md 72-80](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L72-L80)

## Address Alignment Requirements

### Overview

The NIU enforces address alignment constraints when forming NoC packets. Constraints depend on three factors:

*   **Transaction type:** Read, write with length, write with byte-enable, or atomic
*   **Source address type:** MMIO (memory-mapped I/O), L1 (scratchpad RAM), or other
*   **Destination address type:** MMIO, L1, or other

Alignment violations result in undefined hardware behavior including data corruption, incorrect byte lane selection, or NoC protocol violations.

**Address Space Classification:**

In Tensix tiles (tile type `0x01` in `NOC_ENDPOINT_ID`) and Ethernet tiles (tile type `0x02`):

*   **L1 addresses:**`0x0000_0000` to `0xFEFF_FFFF` (scratchpad RAM, up to 1464 KiB per tile)
*   **MMIO addresses:**`0xFF00_0000` to `0xFFFF_FFFF` (NIU registers, debug interfaces, configuration)

In DRAM tiles (tile type `0x08`), PCIe tile (tile type `0x03`), ARC tile (tile type `0x05`), and empty tiles:

*   **All addresses are "other"** (not L1, not MMIO in the Tensix/Ethernet sense)
*   Some addresses cross AXI/APB bridges which impose 4-byte alignment

**Sources:**[WormholeB0/NoC/Alignment.md 1-9](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Alignment.md?plain=1#L1-L9)[WormholeB0/NoC/Alignment.md 30-33](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Alignment.md?plain=1#L30-L33)[WormholeB0/NoC/MemoryMap.md 196-204](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L196-L204)

### Length-Based Transfer Alignment

This applies when `NOC_AT_LEN_BE` contains a length value:

*   All read requests
*   Write requests with `NOC_CMD_WR_BE=false` and `NOC_CMD_WR_INLINE=false`

#### Alignment Notation

The alignment requirement tables use the following notation:

| Notation | Mathematical Constraint | Example |
| --- | --- | --- |
| **Any** | No restrictions | Any address, any length (PCIe host) |
| **AL=4** | `(Address & 0x3) == 0` `Length == 4` | Address=0x1000, Length=4 ✓ Address=0x1002, Length=4 ✗ |
| **AL≤4** | `Length ≤ (4 - (Address & 0x3))` | Address=0x1001, Length=3 ✓ Address=0x1001, Length=4 ✗ |
| **C4** | `(SrcAddr & 0x3) == (DstAddr & 0x3)` | Src=0x1001, Dst=0x2005 ✓ Src=0x1001, Dst=0x2006 ✗ |
| **C16** | `(SrcAddr & 0xF) == (DstAddr & 0xF)` | Src=0x1003, Dst=0x2013 ✓ Src=0x1003, Dst=0x2014 ✗ |
| **C32** | `(SrcAddr & 0x1F) == (DstAddr & 0x1F)` | Src=0x1007, Dst=0x2027 ✓ Src=0x1007, Dst=0x2028 ✗ |
| **DA16** | `(DstAddr & 0xF) == 0` | 0x1000 ✓, 0x1010 ✓ 0x1008 ✗ |
| **DA32** | `(DstAddr & 0x1F) == 0` | 0x1000 ✓, 0x1020 ✓ 0x1010 ✗ |
| **SA32** | `(SrcAddr & 0x1F) == 0` | 0x2000 ✓, 0x2020 ✓ 0x2010 ✗ |

**Sources:**[WormholeB0/NoC/Alignment.md 21-29](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Alignment.md?plain=1#L21-L29)

### Byte-Enable Transfer Alignment

When `NOC_CMD_WR_BE=1` (bit 2 of `NOC_CTRL`), the `NOC_AT_LEN_BE` register contains a 32-bit byte-enable mask instead of a length. When `NOC_CMD_WR_INLINE=1` (bit 3 of `NOC_CTRL`), data comes from the 32-bit `NOC_AT_DATA` register and byte-enable behavior varies by destination.

| Source Type | Dest: MMIO | Dest: L1 | Dest: Other |
| --- | --- | --- | --- |
| **MMIO address** | DA4, SA4 | DA32, SA4 | DA32, SA4 |
| **L1 address** | DA4, C16 | DA32, SA32 | DA32, SA32 |
| **Inline (`NOC_AT_DATA`)** | DA4 | DA16 | N/A |

#### Byte-Enable Semantics

**DA4 (Destination Aligned 4):**

*   Destination address must satisfy `(DstAddr & 0x3) == 0`
*   Byte-enable mask in `NOC_AT_LEN_BE` is ignored
*   Hardware always performs a 4-byte write
*   Used for MMIO register writes (which typically don't support sub-word writes)

**DA16 (Destination Aligned 16):**

*   Destination address must satisfy `(DstAddr & 0xF) == 0`
*   If destination is not 16-byte aligned, hardware rounds down: `DstAddr &= ~0xF`
*   Hardware can perform up to two separate writes to the same 16-byte region: 
    *   First write: Uses low 16 bits of `NOC_AT_LEN_BE` as byte-enable mask
    *   Second write: Uses high 16 bits of `NOC_AT_LEN_BE` as byte-enable mask

*   Both writes broadcast the same 4-byte value from `NOC_AT_DATA`
*   For maximum performance, set either low 16 bits or high 16 bits to zero (not both non-zero)

**DA32 (Destination Aligned 32):**

*   Destination address must satisfy `(DstAddr & 0x1F) == 0`
*   All 32 bits of `NOC_AT_LEN_BE` control which bytes within the 32-byte region are written
*   Bit `i` of `NOC_AT_LEN_BE` controls byte at address `(DstAddr + i)`

**SA4 (Source Aligned 4):**

*   Source address must satisfy `(SrcAddr & 0x3) == 0`
*   Hardware reads 4 bytes from source address
*   The 4-byte value is broadcast to fill all byte positions in the destination write

**SA32 (Source Aligned 32):**

*   Source address must satisfy `(SrcAddr & 0x1F) == 0`
*   Hardware reads 32 bytes from source address

**C16 (Congruent mod 16):**

*   When transferring from L1 to MMIO with byte-enable, addresses must satisfy: 
```
(SrcAddr & 0xF) == (DstAddr & 0xF)
```

> **Restrictions:**
> 
> 
> *   `NOC_CMD_WR_INLINE=1` cannot be used when writing to "other" tile types (DRAM, PCIe, ARC)
> *   When crossing an AXI/APB bridge, byte-enable must select all or none of each 4-byte group

**Sources:**[WormholeB0/NoC/Alignment.md 34-56](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Alignment.md?plain=1#L34-L56)[WormholeB0/NoC/MemoryMap.md 124-128](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L124-L128)

### Atomic Operation Alignment

Atomic requests (selected by setting bits [1:0] of `NOC_CTRL` to `0b01`) operate on L1 memory and have specific alignment constraints.

#### Basic Constraints

| Register/Field | Constraint | Rationale |
| --- | --- | --- |
| **`NOC_TARG_ADDR` destination** | Must be L1 address `< 0xFF00_0000` | Atomics only supported in L1 |
| **Target tile** | Tensix (type `0x01`) or Ethernet (type `0x02`) | Other tiles lack L1 |
| **Atomic operation region** | Within single aligned 16-byte boundary | L1 atomic hardware operates on 16-byte units |
| **`NOC_RET_ADDR` (if used)** | Must satisfy `(Addr & 0x3) == 0` | Response is always 4 bytes |
| **`NOC_RET_ADDR` (if used)** | Must be L1 address `< 0xFF00_0000` | Response write targets L1 |

#### Operation Region Alignment

The low 36 bits of `NOC_TARG_ADDR` specify the operation target address. The atomic operation acts on the aligned 16-byte region containing that address:

`uint64_t target_addr = (NOC_TARG_ADDR_MID << 32) | NOC_TARG_ADDR_LO;uint64_t region_base = target_addr & ~0xF;  // Round down to 16-byte boundary// Atomic operates on bytes [region_base, region_base + 16)`
Individual atomic opcodes encode additional sub-alignment in bits of `NOC_AT_LEN_BE`:

*   Some opcodes operate on 4-byte words at specific offsets within the 16-byte region
*   Some opcodes operate on 2-byte halfwords
*   Some opcodes operate on 1-byte quantities

For example, an atomic increment of a 4-byte word at address `0x1234_5678`:

```
target_addr = 0x1234_5678
region_base = 0x1234_5670  (aligned 16-byte boundary)
word_offset = (0x1234_5678 - 0x1234_5670) / 4 = 2  (third word in region)
```

The word offset is typically encoded in bits [5:4] of `NOC_AT_LEN_BE` for word-based atomics.

> **See Also:**[NoC Atomic Operations](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/NoC%20Atomic%20Operations) for opcode-specific alignment and encoding details.

**Sources:**[WormholeB0/NoC/Alignment.md 57-66](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Alignment.md?plain=1#L57-L66)[WormholeB0/NoC/MemoryMap.md 79-80](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L79-L80)[WormholeB0/NoC/MemoryMap.md 122](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L122-L122)

#### Byte-Enable Transfers (Write with `NOC_CMD_WR_BE=true` or `NOC_CMD_WR_INLINE=true`)

| Source → Destination | Alignment Requirements |
| --- | --- |
| L1 → L1 | `DA32, SA32`: Both 32-byte aligned |
| L1 → MMIO | `DA4, C16`: Dest 4-byte aligned, congruent mod 16 |
| L1 → Other | `DA32, SA32`: Both 32-byte aligned |
| MMIO → L1 | `DA32, SA4`: Dest 32-byte aligned, source 4-byte aligned |
| MMIO → MMIO | `DA4, SA4`: Both 4-byte aligned |
| MMIO → Other | `DA32, SA4`: Dest 32-byte aligned, source 4-byte aligned |
| Inline → L1 | `DA16`: Dest 16-byte aligned |
| Inline → MMIO | `DA4`: Dest 4-byte aligned |
