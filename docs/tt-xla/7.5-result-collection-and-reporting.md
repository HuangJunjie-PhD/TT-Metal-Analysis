This page covers the mechanisms by which test results are collected, stored, summarized, and fed back into the CI system after test jobs complete in tt-xla. It covers the JUnit XML report format, the GitHub Actions artifact upload pattern, the SFTP-based CI/CD data collection workflow, and the scripts that post-process XML reports for test-splitting and failure analysis.

For the test matrix generation that drives which tests run, see [7.3](https://deepwiki.com/tenstorrent/tt-xla/7.3-test-matrix-generation-and-execution). For the benchmark-specific reporting and the `superset-api` action, see [7.4](https://deepwiki.com/tenstorrent/tt-xla/7.4-performance-benchmarking). For how individual test properties (`pcc`, `bringup_status`, etc.) are produced inside tests, see [6.5](https://deepwiki.com/tenstorrent/tt-xla/6.5-failure-analysis-and-classification).

## JUnit XML Report Production

Each `run-tests` job in `call-test.yml` invokes pytest with `--junitxml`, writing a single XML file whose path is computed from the GitHub job ID:

[.github/workflows/call-test.yml 126](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L126-L126)

```
test_report_path=report_$JOB_ID.xml
```

The file name embeds the job ID fetched by the `tenstorrent/tt-github-actions/.github/actions/job_id` composite action, making each report uniquely identifiable across matrix jobs.

The XML content is enriched by `conftest.py`. The `pytest_collection_modifyitems` hook attaches `user_properties` to every collected test at collection time:

[tests/conftest.py 135-179](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/conftest.py#L135-L179)

These properties become `<properties>` elements in the JUnit XML. The full set of allowed property keys is:

| Key | Type | Description |
| --- | --- | --- |
| `category` | `utils.Category` | Test category |
| `model_name` | `str` | Model under test |
| `model_group` | `str` | Model grouping |
| `model_info` | `ModelInfo` dict | Full model metadata |
| `run_mode` | `infra.RunMode` | Inference / training |
| `bringup_status` | `utils.BringupStatus` | Pass/fail classification |
| `pcc` | `float` | Pearson correlation coefficient |
| `atol` | `float` | Absolute tolerance |
| `jax_op_name` | `str` | JAX op identifier |
| `torch_op_name` | `str` | PyTorch op identifier |
| `shlo_op_name` | `str` | StableHLO op identifier |

Additionally, every test receives:

*   `tags` dict: `{"test_name": ..., "specific_test_case": ...}` plus the above properties
*   `owner`: always `"tt-xla"`
*   `group`: model group string (model tests only)

The `record_test_properties` marker is the public API for attaching these fields:

[tests/conftest.py 27-57](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/conftest.py#L27-L57)

## Report Artifacts in CI

**Title: Report artifact naming and upload flow in `call-test.yml`**

Sources: [.github/workflows/call-test.yml 357-407](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L357-L407)

The artifact name format for test reports is:

```
test-reports-{test_matrix_hash}-{run_attempt}.{job_index}-{runs-on}-{test_mark}-{job_id}
```

The `test_matrix_hash` is an MD5 of the expanded JSON matrix (computed in `call-generate-matrix.yml`). This hash scopes artifact downloads: the `test-summary` job uses `--pattern "test-reports-{hash}-*"` to download only reports from the same matrix run, even across re-run attempts.

> **Note:** The `test-mark` field is sanitized before use in artifact names — spaces are replaced with underscores. This is because `produce_cicd_data` (the SFTP collection action) drops artifacts with spaces in their names.

[.github/workflows/call-test.yml 349-355](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L349-L355)

Sources: [.github/workflows/call-test.yml 357-407](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L357-L407)

## In-Workflow Test Summary

After all matrix jobs complete, the `test-summary` job runs on `ubuntu-latest` and produces a GitHub check annotation:

[.github/workflows/call-test.yml 382-407](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L382-L407)

It uses the `mikepenz/action-junit-report@v5` action with:

| Parameter | Value |
| --- | --- |
| `report_paths` | `test_reports/report_*.xml` |
| `check_name` | `TT-XLA {test_matrix_name}` |
| `detailed_summary` | `true` |
| `include_passed` | from workflow input |
| `group_suite` | `true` |

The `include_passed` input is propagated from the calling workflow. The experimental nightly workflow sets it to `true` to show all results; other workflows default to `false` (failures only).

[.github/workflows/schedule-nightly-experimental.yml 36](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/schedule-nightly-experimental.yml#L36-L36)

## CI/CD Data Collection via SFTP

`workflow-run-collect-data.yml` is a dedicated internal workflow that runs automatically after any of the listed workflows complete:

[.github/workflows/workflow-run-collect-data.yml 1-61](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/workflow-run-collect-data.yml#L1-L61)

**Title: `workflow-run-collect-data.yml` trigger and data flow**

Sources: [.github/workflows/workflow-run-collect-data.yml 1-61](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/workflow-run-collect-data.yml#L1-L61)

The `collect_data` action receives the `run_id` and `run_attempt` either from the triggering `workflow_run` event or from `workflow_dispatch` inputs (for manual backfills).

### Secrets Used

| Secret | Purpose |
| --- | --- |
| `SFTP_CICD_WRITER_HOSTNAME` | Host for CI/CD telemetry uploads |
| `SFTP_CICD_WRITER_USERNAME` | Username for CI/CD SFTP |
| `SFTP_PERF_WRITER_HOSTNAME` | Host for performance data uploads |
| `SFTP_PERF_WRITER_USERNAME` | Username for performance SFTP |
| `SFTP_CICD_WRITER_KEY` | SSH private key (shared for both) |

The action downloads artifacts from the completed workflow run and uploads them to the SFTP servers. The `collect_data` action is maintained externally in the `tenstorrent/tt-github-actions` repository.

## Test Duration Aggregation

`get_test_duration_from_junit_xmls.py` converts a directory of JUnit XML reports into a JSON duration file used by `pytest-split`'s `least_duration` splitting algorithm.

[.github/scripts/get_test_duration_from_junit_xmls.py 1-101](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/scripts/get_test_duration_from_junit_xmls.py#L1-L101)

**Title: `get_test_duration_from_junit_xmls.py` data flow**

Sources: [.github/scripts/get_test_duration_from_junit_xmls.py 1-101](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/scripts/get_test_duration_from_junit_xmls.py#L1-L101)

### Key Implementation Details

`extract_test_case_info` maps each `<testcase>` element to a duration entry:

*   **Key format:**`{classname_with_slashes}.py::{name}` — the `classname` attribute uses `.` separators which are converted to `/` to reconstruct the file path.
*   **Value:**`float(testcase.get("time", 0))` in seconds.

[.github/scripts/get_test_duration_from_junit_xmls.py 20-42](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/scripts/get_test_duration_from_junit_xmls.py#L20-L42)

After aggregation, tests with exactly `0.0` duration are assigned a default of `0.01` seconds. This prevents `pytest-split`'s `least_duration` algorithm from treating zero-duration tests as free and packing them all into a single group.

[.github/scripts/get_test_duration_from_junit_xmls.py 84-91](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/scripts/get_test_duration_from_junit_xmls.py#L84-L91)

The output file is `.test_durations` (a JSON file at the repo root). This file is checked into the repository and checked out by test jobs in `call-test.yml`:

[.github/workflows/call-test.yml 96-110](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L96-L110)

### Pytest Splitting Integration

The `run-tests` job uses `--splitting-algorithm least_duration` when not in a re-run context:

[.github/workflows/call-test.yml 260-263](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L260-L263)

The timeout for each test group is estimated using a separate `calculate_test_timeout.py` script that also reads from `.test_durations`:

[.github/workflows/call-test.yml 307-316](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L307-L316)

## Re-run and Failure Recovery

On re-run attempts (`github.run_attempt > 1`), the `run-tests` job downloads prior test reports and uses `find_all_failed_tests.py` to produce a `.pytest_tests_to_run` filter file.

[.github/workflows/call-test.yml 185-224](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L185-L224)

`conftest.py` reads this file during `pytest_collection_modifyitems` and removes any test not in the allowed set:

[tests/conftest.py 127-133](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/conftest.py#L127-L133)

When this filter is active, `--splits` / `--group` are omitted from the pytest command (no re-splitting needed since only failed tests run).

[.github/workflows/call-test.yml 255-263](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L255-L263)

## Failure Analysis and Nightly Status Scripts

Two additional scripts operate on JUnit XML data downstream of CI runs. They are not included as workflow steps but are referenced by the project's CI tooling:

| Script | Input | Output | Purpose |
| --- | --- | --- | --- |
| `summarize_junit_xmls.py` | Directory of `report_*.xml` | Summary text/JSON | Categorizes failures by error type; used for dashboards |
| `nightly_status.py` | Historical JUnit XML archives | Ranked flaky test list | Scores tests by their pass/fail volatility across runs |

These scripts parse the same JUnit XML format produced by `conftest.py`'s `user_properties` enrichment, reading the `<properties>` elements (`bringup_status`, `model_name`, etc.) to drive their analysis.
