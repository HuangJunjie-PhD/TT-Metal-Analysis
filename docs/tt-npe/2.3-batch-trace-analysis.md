The batch trace analysis system processes multiple NoC trace files captured during workload execution, grouping them by operation and executing NPE simulations in parallel. This system is implemented in `npe_analyze_noc_trace_dir.py` and serves as the primary analysis tool for profiling workflows. For information about trace collection, see [Trace Collection with tt-metal Profiler](https://deepwiki.com/tenstorrent/tt-npe/2.1-trace-collection-with-tt-metal-profiler). For details on the preprocessing step, see [Trace Preprocessing and Fabric Routing](https://deepwiki.com/tenstorrent/tt-npe/2.2-trace-preprocessing-and-fabric-routing). For output formats, see [Statistics and Visualization Output](https://deepwiki.com/tenstorrent/tt-npe/2.4-statistics-and-visualization-output).

## Overview

The batch trace analyzer scans a directory of NoC trace JSON files, groups them by operation, runs fabric post-processing for multi-chip workloads, executes NPE simulations in parallel, and aggregates performance statistics across all operations. The tool supports both TTNN (TT-NN framework) and Metal (direct Metal API) trace formats, each using different grouping strategies.

**Sources:**[tt_npe/py/util/npe_analyze_noc_trace_dir.py 1-456](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L1-L456)

## Batch Processing Workflow

The complete batch processing pipeline consists of several stages that transform raw trace files into aggregated performance metrics and visualization artifacts.

### End-to-End Pipeline

**Pipeline Stages:**

| Stage | Function | Purpose |
| --- | --- | --- |
| Directory Scan | `glob.glob()` | Locate all `noc_trace*.json` files, excluding merged files |
| Metadata Extraction | `extractDataFromFilename()` | Parse device ID, op name, op ID, trace ID from filenames |
| Operation Grouping | `group_traces_ttnn()` / `group_traces_metal()` | Collect traces belonging to same operation across devices |
| Fabric Processing | `process_traces()` | Merge multi-chip traces with cross-device routing |
| Parallel Execution | `process_trace()` worker | Run NPE simulation for each operation |
| Statistics Collection | `Stats.addDatapoint()` | Accumulate metrics across all operations |
| Output Generation | Multiple functions | Produce summary table, manifest, timelines |

## Trace Grouping Strategies

The analyzer supports two distinct trace grouping strategies depending on the profiling framework used. The strategy determines how individual trace files are associated with logical operations.

### TTNN Trace Grouping

TTNN traces use a compound identifier system where each trace file encodes a `program_runtime_id` that combines a TTNN operation ID with a device ID. Operations may also have a `metal_trace_id` for traced/replayed executions.

**TTNN program_runtime_id Encoding:**

```
program_runtime_id (32 bits):
  [31:10] = ttnn_op_id (22 bits)
  [9:0]   = device_id (10 bits)
```

The `OpUID` class encapsulates both the TTNN operation ID and the optional Metal trace ID for uniqueness:

**Sources:**[tt_npe/py/util/npe_analyze_noc_trace_dir.py 126-139](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L126-L139)[tt_npe/py/util/npe_analyze_noc_trace_dir.py 321-362](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L321-L362)

### Metal Trace Grouping

Metal traces lack the TTNN operation ID structure, so grouping relies on aligning traces by position across devices. All traces for a given program execution must appear in the same sequential order across devices.

**Alignment Requirements:**

*   All devices must have the same number of trace files
*   Trace files must be in execution order
*   Missing traces cause alignment errors

## Statistics Aggregation

The `Stats` class accumulates performance metrics across all operations, enabling both per-operation analysis and aggregate workload characterization.

### Stats Data Model

**Aggregation Metrics:**

| Metric | Formula | Purpose |
| --- | --- | --- |
| Total Cycles | `sum(golden_cycles)` | Overall workload duration |
| Avg Link Util | `mean(link_util)` | Simple average across operations |
| Weighted Avg Link Util | `sum(link_util * golden_cycles) / total_cycles` | Cycle-weighted average |
| Weighted Avg DRAM BW | `sum(dram_bw_util * golden_cycles) / total_cycles` | Cycle-weighted DRAM utilization |
| Weighted Avg ETH BW | `sum(eth_bw_util * golden_cycles) / total_cycles` | Cycle-weighted Ethernet utilization |
| Avg Prediction Error | `mean(abs(cycle_prediction_error))` | Average accuracy |
| Error Percentiles | `percentile(errors, [25, 50, 75, 100])` | Error distribution |

**Datapoint Key Structure:** Each datapoint is indexed by `(op_uid, device_id)` tuple, allowing per-device statistics for multi-device operations:

[tt_npe/py/util/npe_analyze_noc_trace_dir.py 57-58](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L57-L58)

## Output Artifacts

The batch analyzer generates multiple output artifacts depending on configuration, including summary tables, timeline files, and visualization manifests.

### Summary Table Format

The console summary table displays key metrics for each operation, sorted by cycle contribution:

```
-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------
Opname                                      Op ID Metal Trace ID       NoC Util   McastWr NoC Util   DRAM BW Util   Cong Impact   % Overall Cycles   ETH BW Util (per core)
-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------
matmul_op                                      42                      45.23%             12.45%        67.12%         8.34%              15.67%  (0,8,0):23.4% (0,9,0):24.1%
reshape_op                                     43                      12.56%              0.00%        23.45%         2.11%               8.23%  -
...
-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------
average link util                : 28.9% 
cycle-weighted overall link util : 34.2% 
cycle-weighted dram bw util      : 45.6% 
cycle-weighted eth bw util       : 18.3%
```

**Column Descriptions:**

| Column | Data Source | Description |
| --- | --- | --- |
| Opname | Extracted from filename | Operation name (e.g., `matmul`, `layernorm`) |
| Op ID | TTNN op ID component | Unique operation identifier |
| Metal Trace ID | Optional trace replay ID | Present for traced/replayed operations |
| NoC Util | `overall_avg_link_util` | Average NoC link utilization percentage |
| McastWr NoC Util | `overall_avg_mcast_write_link_util` | Multicast write link utilization |
| DRAM BW Util | `dram_bw_util` | DRAM bandwidth utilization percentage |
| Cong Impact | `getCongestionImpact()` | Performance impact of congestion |
| % Overall Cycles | `golden_cycles / total_cycles` | Contribution to total execution time |
| ETH BW Util | `getEthBwUtilPerCoreStr()` | Per-core Ethernet bandwidth utilization |

### Timeline File Organization

Timeline files are written to the `npe_viz/` subdirectory adjacent to the trace directory. File naming follows the pattern `{opname}_{op_uid}.npeviz[.zst]`.

**Timeline Split Mechanism:**

For workloads exceeding `timeline_split_threshold_timesteps`, the system generates split timeline files to manage memory usage during visualization:

```
npe_viz/
├── manifest.json
├── matmul_op_ID42.npeviz           # Full timeline
├── matmul_op_ID42_split_0.npeviz   # First 10K timesteps
├── matmul_op_ID42_split_1.npeviz   # Next 10K timesteps
└── matmul_op_ID42_split_2.npeviz   # Remaining timesteps
```

Each split file contains:

*   Complete `common_info` metadata with split metadata
*   Filtered `noc_transfers` (partially overlapping with region)
*   Filtered `timestep_data` (fully contained in region)
*   Filtered `zones` (fully contained in region)

**Split Metadata Structure:**

For detailed timeline format specification, see [Timeline Visualization Format](https://deepwiki.com/tenstorrent/tt-npe/7.1-timeline-visualization-format).

## Error Handling and Validation

The analyzer includes multiple validation checks to ensure data integrity and provide actionable error messages.

### Pre-Flight Validation

**Validation Checks:**

| Check | Location | Action on Failure |
| --- | --- | --- |
| Directory exists | [line 374](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20374) | Raise `FileNotFoundError` |
| Device sync enabled | [line 378](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20378) | Log warning (may cause inaccurate results) |
| Trace files present | [line 392](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20392) | Exit with error message |
| Filename format valid | [line 266](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20266) | Skip file with warning |
| No duplicate devices per op | [line 349](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20349) | Log error, request cleanup |
| Op names consistent | [line 357](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/line%20357) | Log error, request cleanup |

**Common Error Scenarios:**

1.   **Multiple trace runs mixed**: Detected when same device appears twice for one operation

```
Error: There seem to trace files from multiple runs in the trace directory, 
please remove old traces and run again.
```
2.   **Missing device sync**: Warning when `sync_device_info.csv` absent

```
WARN: Device sync isn't turned on. This will lead to inaccurate traces for 
mesh device workloads! ... set TT_METAL_DEVICE_PROFILER=1 and 
TT_METAL_PROFILER_SYNC=1
```
3.   **Invalid filename format**: Files not matching expected pattern are skipped

```
Invalid name for noc trace file: {filename}. Skipping...
```
