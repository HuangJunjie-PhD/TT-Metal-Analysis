## Purpose and Scope

The Remote Communication & Ethernet system enables access to Tenstorrent chips that are not directly connected to the host via PCIe. In multi-chip configurations (such as Galaxy or T3K), only a subset of chips have direct PCIe connections (MMIO-capable chips). The remaining chips are accessed through Ethernet links between chips, routing communication through a gateway chip. This document covers the `RemoteChip` and `RemoteCommunication` abstractions, ethernet-based data transfer protocols, and legacy firmware support.

For information about multi-chip topology, see [Cluster Descriptor & Topology](https://deepwiki.com/tenstorrent/tt-umd/2.2-cluster-descriptor-and-topology). For PCIe-based direct access patterns, see [PCIDevice & Hardware Communication](https://deepwiki.com/tenstorrent/tt-umd/3.1-pcidevice-and-hardware-communication).

## Architecture Overview

The remote communication system follows a gateway pattern where a PCIe-connected `LocalChip` acts as a proxy for ethernet-connected `RemoteChip` instances.

## RemoteCommunication Class Hierarchy

The `RemoteCommunication` class is the abstract base for all ethernet-based communication. It is instantiated by a `LocalChip` and shared with `RemoteChip` instances that route through that local gateway.

### Core Implementation: RemoteCommunicationLegacyFirmware

## Ethernet Command Structures

## Ethernet Broadcast

The system supports `ethernet_broadcast_write` to send data to multiple cores across multiple chips in a single operation.

*   **Broadcast Headers:** The `broadcast_header` vector is used to define the target grid of chips. [device/api/umd/device/tt_device/remote_communication_legacy_firmware.hpp 41](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/remote_communication_legacy_firmware.hpp#L41-L41)
*   **Implementation:** Within `write_to_non_mmio`, if the `broadcast` flag is set, the `routing_cmd_t` is configured with broadcast flags and sent to the Ethernet cores. [device/api/umd/device/tt_device/remote_communication_legacy_firmware.hpp 40-41](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/remote_communication_legacy_firmware.hpp#L40-L41)

## Synchronization and Mutex Usage

Because the Ethernet command queues do not have hardware-level mutual exclusion for host-side write pointer updates, UMD implements a software-level inter-process mutex.

*   **NON_MMIO Mutex:** All `read_non_mmio` and `write_to_non_mmio` calls must acquire `MutexType::NON_MMIO`. [device/tt_device/remote_communication_legacy_firmware.cpp 110](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/remote_communication_legacy_firmware.cpp#L110-L110)
*   **Scope:** The lock is held for the duration of the Ethernet core interaction to prevent corruption of the command queue by multiple threads or processes. [device/tt_device/remote_communication_legacy_firmware.cpp 88-94](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/remote_communication_legacy_firmware.cpp#L88-L94)
*   **Lock Management:** Handled by the `LockManager` class initialized within the `RemoteCommunication` constructor. [device/tt_device/remote_communication.cpp 23](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/remote_communication.cpp#L23-L23)
