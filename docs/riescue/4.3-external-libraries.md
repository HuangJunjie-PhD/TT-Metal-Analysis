This document describes the external Python libraries that RiESCUE depends on, focusing on the `riscv-coretp` test plan library and other supporting dependencies. For information about external tools like the RISC-V toolchain and instruction set simulators, see [RISC-V Toolchain Integration](/tenstorrent/riescue/4.1-risc-v-toolchain-integration) and [Instruction Set Simulators](/tenstorrent/riescue/4.2-instruction-set-simulators).

## Overview

RiESCUE depends on several external Python libraries that provide essential functionality:

* **`coretp` (riscv-coretp)**: Test plan definitions for RISC-V compliance testing, consumed by RiescueC's TP mode
* **Data structure libraries**: `sortedcontainers`, `intervaltree`, `numpy` for efficient data manipulation
* **Configuration libraries**: `pyyaml` for configuration file parsing
* **Development tools**: `sphinx`, `black`, `flake8`, `pyright`, `coverage` for documentation and code quality

All dependencies are declared in `pyproject.toml` and installed via pip when RiESCUE is installed.

## Core External Library: riscv-coretp

### Purpose and Integration

The `coretp` library (from the `riscv-coretp` repository) is RiESCUE's primary external dependency. It provides a framework for defining RISC-V architectural test plans as structured Python data structures called `TestScenario` objects. These scenarios describe:

* Sequences of instructions to execute
* Environmental configurations (privilege modes, paging modes, CSR states)
* Pre-conditions and post-conditions for testing
* Predicates that constrain test generation

RiescueC's TP mode consumes these test plans to generate directed compliance tests.

**Sources:** [pyproject.toml28](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L28-L28) [README.md54-64](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L54-L64)

### Version Pinning

The `coretp` dependency is pinned to a specific Git commit to ensure reproducibility:

```
coretp @ git+https://git@github.com/tenstorrent/riscv-coretp@5da80d20b9eb6327d3f272fdaf7fca310da13fc8 
```

This approach provides several benefits:

* **Reproducibility**: Tests generated with the same RiESCUE version use the same test plan definitions
* **Stability**: Changes to test plans in the upstream repository don't unexpectedly affect RiESCUE
* **Traceability**: The specific commit hash documents exactly which test plan version is being used

To update the `coretp` version, the commit hash in [pyproject.toml28](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L28-L28) must be updated to a newer commit from the `riscv-coretp` repository.

**Sources:** [pyproject.toml28](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L28-L28)

### Usage in TP Mode

When RiescueC is invoked with `--mode tp --test_plan` , the following workflow occurs:

**Diagram: TP Mode Test Generation Flow**

The test plan name (e.g., `zicond`) corresponds to a module in the `coretp` package that defines test scenarios for that RISC-V extension. RiescueC imports these scenarios, evaluates their predicates to determine valid instruction sequences and environmental configurations, then uses the underlying RiescueD generator to produce test assembly and ELF files.

**Sources:** [README.md53-64](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L53-L64)

### Supported Test Plans

The list of supported test plans varies by release. As of the documented version, RiescueC supports TP mode for the following extensions (see [README.md162-246](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L162-L246) for the complete table):

| Extension | Description | TP Mode Support |
| --- | --- | --- |
| Zicsr | CSR instructions | All modes |
| Zicntr | Base counters and timers | TP |
| Zihpm | Hardware performance counters | TP |
| Zkt | Data-independent execution latency | TP |
| Zicond | Integer conditional operations | TP |
| Zimop | May-be-operations | TP |
| Zcmop | Compressed may-be-operations | TP |
| Zifencei | Instruction-Fetch Fence | TP |
| Sv39 | Page-based 39-bit virtual-memory | TP |
| Sv48 | Page-based 48-bit virtual-memory | TP |
| Sv57 | Page-based 57-bit virtual-memory | TP |
| Svpbmt | Page-based memory types | TP |
| Svinval | Fine-grained TLB invalidation | TP |
| Sstc | Supervisor-mode timer interrupts | TP |
| Svadu | Hardware A/D bit updates | TP |
| Sscounterenw | Writable counter enables | TP |
| Svbare | Bare mode virtual-memory | All modes |
| H | Hypervisor extension | All modes |

**Sources:** [README.md162-246](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L162-L246)

## Supporting Python Libraries

### Data Structure and Algorithm Libraries

**Diagram: Data Structure Library Usage**

#### sortedcontainers

The `sortedcontainers` library provides sorted list, sorted dict, and sorted set implementations with better performance than standard library alternatives. Used in RiESCUE's `Pool` class and other components that maintain sorted collections of test data.

**Package:** `sortedcontainers` (latest version, unpinned)

#### intervaltree

The `intervaltree` library provides interval tree data structures for efficient querying of overlapping intervals. Used in memory management to track memory regions and detect address conflicts.

**Package:** `intervaltree` (latest version, unpinned)

#### numpy

The `numpy` library provides efficient numerical array operations. Used for generating random data arrays and performing mathematical operations on test data.

**Package:** `numpy` (latest version, unpinned)

**Sources:** [pyproject.toml17-20](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L17-L20) [pyproject.toml25](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L25-L25)

### Configuration Library: PyYAML

**Diagram: PyYAML Usage in Configuration**

The `pyyaml` library provides YAML parsing and generation. While RiESCUE primarily uses JSON for configuration files, YAML is used in some test directives and can be used as an alternative configuration format.

**Package:** `pyyaml` (latest version, unpinned)

**Sources:** [pyproject.toml19](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L19-L19)

### Development Tool Libraries

The following libraries support development, testing, and documentation workflows but are not used in the core test generation functionality:

**Diagram: Development Tool Libraries in Build Pipeline**

#### sphinx and sphinx-rtd-theme

* **sphinx**: Documentation generator that converts reStructuredText to HTML
* **sphinx-rtd-theme**: Read the Docs theme for Sphinx documentation
* **Minimum version**: 7.4.0 for Sphinx (theme version unpinned)
* **Usage**: Generating documentation website from source files in `docs/`

See [Documentation System](/tenstorrent/riescue/6-documentation-system) for detailed information on documentation building.

#### black

* **Purpose**: Python code formatter that enforces consistent style
* **Version**: Pinned to `25.1.0` for consistency across all contributors
* **Configuration**: [pyproject.toml1-4](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L1-L4) defines line length (200) and excluded paths

#### flake8

* **Purpose**: Python linter that checks code style and finds errors

#### pyright

* **Version**: Latest with Node.js bundled (`pyright[nodejs]`)
* **Configuration**: [pyproject.toml51-78](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L51-L78) defines type checking mode and included files
* **Usage**: Pre-commit hooks, CI pipeline, and IDE integration

#### coverage

* **Purpose**: Code coverage measurement for Python tests

## Dependency Management

### pyproject.toml Configuration

All Python dependencies are declared in the `[project]` section of [pyproject.toml11-29](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L11-L29):

```
[project] name = "riescue" version = "1.2.0" description = "RISC-V Directed Test Framework and Compliance Suite" readme = "README.md" requires-python = ">= 3.9" dependencies = [ "sortedcontainers", "pyyaml", "numpy", "sphinx>=7.4.0", "sphinx-rtd-theme", "flake8", "black==25.1.0", "intervaltree", "coverage", "pyright[nodejs]", "coretp @ git+https://git@github.com/tenstorrent/riscv-coretp@5da80d20b9eb6327d3f272fdaf7fca310da13fc8", ] 
```

**Sources:** [pyproject.toml11-29](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L11-L29)

### Versioning Strategy

RiESCUE uses different versioning strategies for different types of dependencies:

| Dependency Type | Strategy | Example | Rationale |
| --- | --- | --- | --- |
| Core external library | Git commit pinning | `coretp @ git+...@5da80d20` | Reproducibility of test plans |
| Code formatter | Exact version pinning | `black==25.1.0` | Consistent formatting across contributors |
| Documentation | Minimum version | `sphinx>=7.4.0` | Ensure required features available |
| Utilities | Latest version | `numpy`, `pyyaml` | Get bug fixes and improvements |

This mixed strategy balances stability (for critical dependencies) with flexibility (for utilities that don't affect test generation).

**Sources:** [pyproject.toml17-29](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L17-L29)

### Installation

When RiESCUE is installed via pip, all dependencies are automatically installed:

The `coretp` dependency is automatically cloned from GitHub and installed from the specified commit. This requires Git to be available in the environment and access to the GitHub repository.

For development setups using the containerized environment, all dependencies are pre-installed in the container image. See [Container System](/tenstorrent/riescue/3.1-container-system) for details on the development container.
