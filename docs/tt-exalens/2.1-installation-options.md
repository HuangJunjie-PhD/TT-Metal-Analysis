## Installation Methods

**Installation method decision diagram:**

Sources: [README.md 17-74](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/README.md?plain=1#L17-L74)

* * *

### Option 1: Install from PyPI

The simplest method. Installs a pre-built wheel that includes the bundled `riscv-tt-elf-gdb` binary.

```
pip install tt-exalens
```

To install a specific version:

```
pip install tt-exalens==x.y.z
```

After installation, the `tt-exalens` CLI entry point is available directly on the shell path. This entry point maps to `ttexalens.cli:main` as declared in `pyproject.toml` under `[project.scripts]`.

Sources: [README.md 19-27](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/README.md?plain=1#L19-L27)

* * *

### Option 2: Install from GitHub

Installs directly from the `main` branch HEAD. Useful when the latest code has not yet been published to PyPI.

```
pip install git+https://github.com/tenstorrent/tt-exalens.git
```

This triggers a source build via `scikit-build-core`, which invokes CMake with `BUILD_PYTHON_WHEEL=ON` and `STRIP_SYMBOLS=ON` as configured in `pyproject.toml` under `[tool.scikit-build.cmake.define]`.

Sources: [README.md 28-31](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/README.md?plain=1#L28-L31)

* * *

### Option 3: Install from Local Source Checkout

Used for development, testing, or contributions.

**Step 1: Clone the repository**

```
git clone https://github.com/tenstorrent/tt-exalens.git
cd tt-exalens
```

**Step 2: Create and activate a virtual environment (recommended)**

The `create-venv.sh` script creates a virtual environment in `.venv`, activates it, ensures pip is up-to-date, and then calls `install-deps.sh`:

```
./scripts/create-venv.sh
source .venv/bin/activate
```

**Step 3 (alternative to Step 2): Install dependencies into an existing environment**

If you already have a Python environment, use `install-deps.sh` directly:

```
./scripts/install-deps.sh
```

**Dependency installation sequence:**

`install-deps.sh` installs four dependency sets. The script detects and prefers `uv` for faster installation, falling back to `pip` if `uv` is unavailable.

| Dependency Set | File | Purpose | Script Location |
| --- | --- | --- | --- |
| Runtime | `ttexalens/requirements.txt` | Required to run the library and CLI | [scripts/install-deps.sh 25-26](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L25-L26) |
| Development | `ttexalens/dev-requirements.txt` | Required for static checks, linting | [scripts/install-deps.sh 28-29](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L28-L29) |
| Test | `test/test_requirements.txt` | Required to run the test suite | [scripts/install-deps.sh 31-32](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L31-L32) |
| Build tools | `wheel`, `build`, `setuptools` | Required to build wheel artifacts | [scripts/install-deps.sh 34-35](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L34-L35) |

The `uv` vs `pip` selection logic is in [scripts/install-deps.sh 16-23](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L16-L23)

Sources: [README.md 57-74](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/README.md?plain=1#L57-L74)[scripts/create-venv.sh 1-23](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/create-venv.sh#L1-L23)[scripts/install-deps.sh 1-39](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/scripts/install-deps.sh#L1-L39)

## Package and Entry Points

**How installation artifacts map to code entities:**

The `tt-exalens` shell command is a console script entry point defined in `pyproject.toml` under `[project.scripts]`, mapping to the `main()` function in `ttexalens/cli.py`. The bundled GDB binary is copied from the SFPI toolchain during wheel builds.

Sources: pyproject.toml (entry point mapping), [CMakeLists.txt 27-46](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/CMakeLists.txt#L27-L46) (GDB bundling)

## Build System Integration (Wheel Packaging)

When building from source (either from GitHub or locally via `make wheel`), the build chain is:

**Wheel build pipeline (source to distribution):**

When `BUILD_PYTHON_WHEEL=ON` is set, the CMake configuration skips RISC-V kernel compilation ([CMakeLists.txt 19-21](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/CMakeLists.txt#L19-L21)). Instead, it only:

1.   Downloads the SFPI toolchain via `cmake/sfpi_release.cmake`
2.   Extracts the `riscv-tt-elf-gdb` binary from the toolchain
3.   Strips debug symbols via `install-gdb-for-wheel` target
4.   Packages the stripped binary into the wheel

The SFPI toolchain URL and version are defined in [cmake/sfpi_release.cmake 6-7](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/cmake/sfpi_release.cmake#L6-L7) The GDB binary extraction and installation logic is in [CMakeLists.txt 27-46](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/CMakeLists.txt#L27-L46)

Sources: [Makefile 39-41](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/Makefile#L39-L41)[CMakeLists.txt 12-46](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/CMakeLists.txt#L12-L46) cmake/sfpi_release.cmake

## System-Level Dependencies (Development Only)

When running tests or building RISC-V kernels from source, additional system packages are required. These are installed by `scripts/setup-dev-env.sh`:

| Package | Purpose |
| --- | --- |
| `ninja-build` | Build system backend for CMake |
| `cmake` | Build configuration |
| `make` | Makefile runner |
| `rsync` | File sync utilities |
| `gdb` | Native GDB (separate from bundled RISC-V GDB) |
| `libyaml-cpp-dev` | YAML C++ library for kernel builds |
