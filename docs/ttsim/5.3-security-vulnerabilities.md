# Security Vulnerabilities

## Purpose and Scope

This page details the process for reporting security vulnerabilities in ttsim. It covers the private reporting mechanism, required information, response timelines, and coordinated disclosure policy. Security vulnerabilities must be reported through GitHub's private security advisory feature, not through public issues.

For reporting non-security bugs, feature requests, or general questions, see [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues). For general contribution guidelines, see [Contribution Process](https://deepwiki.com/tenstorrent/ttsim/5.1-contribution-process). For community behavioral standards, see [Code of Conduct](https://deepwiki.com/tenstorrent/ttsim/6.1-code-of-conduct).

**Sources:**[SECURITY.md 1-54](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L1-L54)[CONTRIBUTING.md 10-11](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L10-L11)

## Private Reporting Requirement

Security vulnerabilities **must not** be reported through public GitHub issues. Public disclosure of vulnerabilities before a fix is available puts all users at risk. The ttsim project uses GitHub's private vulnerability reporting feature to handle security issues confidentially.

**Sources:**[SECURITY.md 5](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L5-L5)[CONTRIBUTING.md 11](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L11-L11)

## Required Information

Security reports must include comprehensive details to enable rapid triage and remediation. The following table lists required and recommended information:

| Information Category | Requirement Level | Description |
| --- | --- | --- |
| Type of issue | Required | Classification (e.g., buffer overflow, memory corruption, injection) |
| Full paths of source files | Required | Absolute or relative paths to affected code |
| Source code location | Required | Tag/branch/commit or direct URL to affected code |
| Special configuration | Required | Any non-default settings needed to reproduce |
| Step-by-step instructions | Required | Detailed reproduction steps |
| Proof-of-concept code | Recommended | Working exploit or PoC demonstrating the vulnerability |
| Impact assessment | Required | Explanation of how an attacker might exploit the issue |

**Sources:**[SECURITY.md 16-26](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L16-L26)

### Response Timeline

### Response Phases

| Phase | Actions | Timeline | Communication Channel |

## Coordinated Disclosure Policy

The ttsim project follows a coordinated disclosure approach to protect users while acknowledging security researchers:

### Disclosure Workflow

**Sources:**[SECURITY.md 39-43](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L39-L43)

### Disclosure Principles

The coordinated disclosure policy is built on two key principles:

1.   **User Safety First**: No public disclosure before a fix is available and deployed
2.   **Researcher Cooperation**: Work with reporters to understand scope and develop fixes before disclosure

The security team works with reporters through the private advisory channel to:

*   Understand the full scope of the vulnerability
*   Develop an effective fix
*   Coordinate timing of public disclosure
*   Ensure proper attribution for responsible disclosure

**Sources:**[SECURITY.md 39-43](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L39-L43)

## Supported Versions

Security updates are provided according to the following policy:

| Version Category | Security Support | Recommendation |
| --- | --- | --- |
| Most recent release | Full security support | Keep up to date |
| Older releases | No security patches | Upgrade to latest |

Users are strongly encouraged to keep their ttsim installations up to date to receive security patches. Security fixes are not backported to older releases.

**Sources:**[SECURITY.md 45-48](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L45-L48)

## Reporting Channel Comparison

Understanding when to use which reporting channel is critical for appropriate handling:

**Sources:**[SECURITY.md 5](https://github.com/tenstorrent/ttsim/blob/67b1733e/SECURITY.md?plain=1#L5-L5)[CONTRIBUTING.md 10-11](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L10-L11)[CONTRIBUTING.md 15-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L15-L29)

### Channel Selection Criteria

| Issue Type | Characteristics | Reporting Channel |
| --- | --- | --- |
| Security Vulnerability | Could allow unauthorized access, data corruption, denial of service, or other security impact | GitHub Security Advisory (private) |
| Functional Bug | Incorrect behavior but no security implications | GitHub Issues (public) |
| Feature Request | Suggestion for new functionality | GitHub Issues (public) |
| Configuration Issue | Setup or usage question | GitHub Issues (public) |
