## Purpose and Scope

CTK (Compliance Test Kit) is a high-level command-line tool that generates comprehensive test suites from RISC-V ISA strings. It provides a simplified interface for creating compliance test directories by automatically invoking RiescueC for each extension specified in the ISA string. This page documents CTK's architecture, workflow, and usage patterns.

For detailed information about the underlying test generation mechanisms, see [RiescueC: Compliance Test Generator](/tenstorrent/riescue/2.2-riescuec:-compliance-test-generator). For information about the base directed test framework, see [RiescueD: Directed Test Framework](/tenstorrent/riescue/2.1-riescued:-directed-test-framework).

## Overview

CTK serves as a wrapper around RiescueC's bringup mode, automating the process of generating test suites for multiple RISC-V extensions. Instead of manually invoking RiescueC for each extension, users can specify an ISA string (e.g., `rv64imfv`) and CTK will generate a complete test directory structure with tests for all included extensions.

**Key Capabilities:**

* Parses RISC-V ISA strings to identify base ISA and extensions
* Automatically invokes RiescueC for each supported extension
* Generates organized directory structures with test files
* Creates self-checking compliance tests for datapath validation
* Supports the full range of RiescueC-compatible extensions

CTK is particularly useful for:

* Generating comprehensive test suites for new core designs
* Creating baseline compliance test sets
* Automated regression testing across multiple extensions
* Rapid bring-up validation of RISC-V implementations

## Test Generation Workflow

CTK follows a multi-stage workflow to transform an ISA string into a complete test directory:

**Workflow Stages:**

1. **Parsing Stage**: CTK parses the ISA string to extract the base ISA (e.g., RV64I) and individual extensions (e.g., M, F, V)
2. **Orchestration Stage**: Creates the output directory structure and iterates over each identified extension
3. **Generation Stage**: For each extension:

   * Invokes RiescueC with `--mode bringup` and the extension name
   * Generates self-checking test assembly
   * Compiles tests using the RISC-V toolchain
   * Creates disassembly files for verification
4. **Organization Stage**: Organizes generated artifacts into a structured directory and produces a summary report

## ISA String Format

CTK accepts standard RISC-V ISA strings following the canonical format:

**Format:** `rv{XLEN}{base}{extensions}`

**Examples:**

* `rv64imfv` тЖТ RV64I base with M, F, V extensions
* `rv32imafc` тЖТ RV32I base with M, A, F, C extensions
* `rv64gc` тЖТ RV64I base with IMAFD extensions and Zicsr, Zifencei (G shorthand)

**Supported Base ISAs:**

* `rv32i` - 32-bit base integer ISA
* `rv64i` - 64-bit base integer ISA

**Extension Parsing:** CTK recognizes both single-letter and multi-letter extensions:

| Extension Format | Example | Description |
| --- | --- | --- |
| Single-letter | `M`, `A`, `F`, `D`, `V`, `C` | Standard extensions |
| Multi-letter (Z) | `Zba`, `Zbb`, `Zbc`, `Zbs` | Bit manipulation |
| Multi-letter (Zf) | `Zfh`, `Zfbfmin` | Floating-point variants |
| Multi-letter (Zv) | `Zvbb`, `Zvbc`, `Zvfh` | Vector variants |
| Combined (G) | `G` expands to `IMAFD_Zicsr_Zifencei` | General-purpose combination |

## Test Directory Structure

CTK generates an organized directory structure containing all test artifacts:

```
/ тФЬтФАтФА / тФВ тФЬтФАтФА test__001.elf тФВ тФЬтФАтФА test__001.dis тФВ тФЬтФАтФА test__001.log тФВ тФЬтФАтФА test__002.elf тФВ тФФтФАтФА ... тФЬтФАтФА / тФВ тФЬтФАтФА test__001.elf тФВ тФФтФАтФА ... тФЬтФАтФА summary.txt тФФтФАтФА ctk_config.json 
```

**File Types:**

| File Type | Extension | Purpose |
| --- | --- | --- |
| ELF Executable | `.elf` | Compiled test binary for ISS execution |
| Disassembly | `.dis` | Human-readable disassembly for debugging |
| Test Log | `.log` | Execution log from ISS simulation |
| Configuration | `.json` | Test generation parameters and settings |
| Summary Report | `.txt` | High-level summary of generated tests |

**Directory Organization:**

* Tests are grouped by extension in separate subdirectories
* Each extension directory contains all test variants for that extension
* Configuration files preserve the generation parameters for reproducibility
* Summary files provide overview of test counts and coverage

## Relationship to RiescueC and RiescueD

CTK operates as a high-level orchestration layer over RiescueC:

**Layer Interactions:**

1. **CTK тЖТ RiescueC**: CTK invokes RiescueC multiple times with `--mode bringup --extension`  for each extension in the ISA string
2. **RiescueC тЖТ RiescueD**: RiescueC uses RiescueD's `Generator` and `Pool` classes to create assembly code with self-checking mechanisms
3. **RiescueD тЖТ Toolchain**: RiescueD compiles generated assembly using the RISC-V toolchain and optionally simulates on ISS

**Key Differences:**

| Aspect | CTK | RiescueC | RiescueD |
| --- | --- | --- | --- |
| **Interface** | ISA string | Extension name + mode | Assembly test file |
| **Scope** | Multi-extension suite | Single extension tests | Single test file |
| **Automation** | Full automation | Semi-automated generation | Manual test writing |
| **Output** | Test directory tree | Test files per extension | Single ELF + artifacts |
| **Use Case** | Batch compliance testing | Extension compliance | Directed test cases |

## Extension Support

CTK supports all extensions available in RiescueC's bringup mode. The complete list includes:

**Standard Extensions (Bringup Mode):**

* **I**: Base integer instruction set (RV32I, RV64I)
* **M**: Integer multiplication and division
* **A**: Atomic instructions
* **F**: Single-precision floating-point
* **D**: Double-precision floating-point
* **C**: Compressed instructions
* **V**: Vector extension

**Bit Manipulation Extensions:**

* **B**: Combined bit manipulation (expands to Zba, Zbb, Zbc, Zbs)
* **Zba**, **Zbb**, **Zbc**, **Zbs**: Individual bit manipulation subsets

**Floating-Point Extensions:**

* **Zfh**: Half-precision floating-point
* **Zfhmin**: Minimal half-precision floating-point (converts only)
* **Zfbfmin**: BF16 floating-point converts

**Vector Extensions:**

* **Zvbb**: Vector basic bit manipulation
* **Zvbc**: Vector carryless multiplication
* **Zvfh**: Vector half-precision floating-point
* **Zvfhmin**: Vector minimal half-precision floating-point
* **Zvfbfmin**: Vector BF16 converts
* **Zvfbfwma**: Vector BF16 widening multiply-add

**Additional Extensions:**

* **Zcb**: Additional compressed instructions
* **Zfa**: Additional floating-point instructions

For privileged and system-level extensions requiring test plan scenarios, use RiescueC's TP mode directly instead of CTK. See [Test Plan (TP) Mode](/tenstorrent/riescue/2.2.2-test-plan-(tp)-mode) for details.

## Performance and Scaling

CTK's generation time scales linearly with the number of extensions:

**Typical Generation Times:**

| ISA String | Extensions | Approximate Time |
| --- | --- | --- |
| `rv64i` | 1 (base) | ~10 seconds |
| `rv64im` | 2 | ~20 seconds |
| `rv64imafdc` | 6 | ~60 seconds |
| `rv64imafdc_zba_zbb_zbc_zbs` | 10 | ~100 seconds |

Times vary based on:

* Number of test variants per extension
* Complexity of extension instructions
* System performance (toolchain compilation speed)

**Optimization Strategies:**

1. **Parallel Generation**: Extensions can be generated independently
2. **Selective Testing**: Use `--extensions` to generate only needed tests
3. **Incremental Updates**: Regenerate only changed extensions
