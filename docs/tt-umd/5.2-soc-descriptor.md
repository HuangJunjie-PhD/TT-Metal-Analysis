## Purpose and Scope

The `SocDescriptor` class provides chip-level architectural information that describes the physical layout and capabilities of a single Tenstorrent device. It encapsulates core locations, types, connectivity, harvesting configurations, and coordinate translation rules specific to each chip architecture.

This document covers the structure and usage of `SocDescriptor`, its YAML representation, and the coordinate manager integration. For information about coordinate system definitions and the rationale behind multiple coordinate spaces, see [Coordinate Translations](https://deepwiki.com/tenstorrent/tt-umd/5.1-coordinate-translations).

## Overview

`SocDescriptor` serves as the architectural blueprint for a single chip. It is constructed from a static `SocArchDescriptor` (which defines the fixed architecture layout) and runtime `ChipInfo` (which provides chip-specific data like harvesting masks).

Key responsibilities include:

*   **Core enumeration** by type (TENSIX, DRAM, ETH, ARC, PCIE, etc.) and coordinate system.
*   **Coordinate translation** between hardware (NOC0/NOC1), logical, and translated spaces via an internal `CoordinateManager`.
*   **Harvesting awareness** indicating which cores are disabled and how they affect logical indexing.
*   **Architecture parameters** such as L1 sizes, DRAM bank sizes, and feature versions (unpacker/packer/overlay).

## YAML Format and Schema

The SoC architecture is typically defined in a YAML file. This file describes the physical grid and the locations of various core types in the `NOC0` coordinate system.

### Key YAML Sections

*   **grid**: Defines `x_size` and `y_size` of the physical NOC.
*   **core types**: Sections for `arc`, `pcie`, `dram`, `eth`, `functional_workers` (Tensix), `router_only`, `security`, `l2cpu`, and `dispatch`.
*   **noc0_x_to_noc1_x / noc0_y_to_noc1_y**: Mapping tables for translating between NOC0 and NOC1 coordinates.
*   **L1/DRAM Sizes**: Constants for memory sizes.
*   **features**: Versioning for hardware engines (unpacker, packer, overlay).

### Example: Blackhole SoC Descriptor

The Blackhole architecture (e.g., `blackhole_140_arch.yaml`) defines a 17x12 grid.

`grid:  x_size: 17  y_size: 12 arc: [ 8-0 ]pcie: [ 2-0, 11-0 ]dram: [    [0-0, 0-1, 0-11], # Bank 0 with multiple NOC ports    [0-2, 0-10, 0-3], # Bank 1    # ...]functional_workers: [   1-2, 2-2, 3-2, ... # Tensix cores]arch_name: BLACKHOLE`

## Core Enumeration and Types

`SocDescriptor` categorizes on-chip cores into distinct functional types defined in the `CoreType` enum.

### Core Types

| Core Type | Description |
| --- | --- |
| `TENSIX` / `WORKER` | General purpose compute cores. |
| `DRAM` | Memory controllers. |
| `ETH` | Ethernet cores for inter-chip links. |
| `ARC` | Adaptive Runtime Controller (system management). |
| `PCIE` | PCIe host interface. |
| `ROUTER_ONLY` | Cores serving only as NOC routers without functional logic. |

**Sources:**[device/api/umd/device/types/core_coordinates.hpp 23-41](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/core_coordinates.hpp#L23-L41)

### Core Queries

The descriptor provides methods to retrieve lists of cores based on type and coordinate system.

## Coordinate Translation

`SocDescriptor` delegates coordinate transformations to an internal `CoordinateManager`. This handles the complex logic of mapping logical software indices to physical hardware coordinates while accounting for harvesting.

### Translation Methods

*   `translate_coord_to`: Converts a `CoreCoord` from its current system to a target `CoordSystem`.
*   `get_coord_at`: Returns a full `CoreCoord` (including type) for a raw `tt_xy_pair` in a specific system.
*   `translate_chip_coord_to_translated`: A helper for the common task of converting chip-relative coordinates to the `TRANSLATED` system used for device access.

## Harvesting Support

Harvesting allows the UMD to operate on chips where certain rows, columns, or channels are disabled.

### Harvesting Masks

The `HarvestingMasks` structure contains bitmasks for different resources:

*   `tensix_harvesting_mask`: Disables Tensix cores (rows in Wormhole, columns in Blackhole).
*   `dram_harvesting_mask`: Disables specific DRAM channels.
*   `eth_harvesting_mask`: Disables specific Ethernet links.

### Impact on Coordinates

When harvesting is applied, the `LOGICAL` coordinate system shifts to provide a contiguous view of the remaining resources. For example, if Tensix row 0 is harvested on Wormhole, logical row 0 will map to physical NOC row 1.

`// Example: Checking if a core is harvestedbool is_harvested = soc_desc.is_core_of_type(xy, CoreType::HARVESTED, CoordSystem::NOC0);`

## Serialization

`SocDescriptor` can be serialized back into a YAML format. This is useful for debugging or for passing the runtime state of a chip (including its harvesting configuration) to other tools.

### Serialization Methods

*   `serialize()`: Returns a YAML string.
*   `serialize_to_file(path)`: Writes the descriptor to a file. If no path is provided, it defaults to a temporary directory.

The serialization process uses `YAML::Emitter` to write out the grid size, core locations for all types, and the NOC0-to-NOC1 mapping tables.
