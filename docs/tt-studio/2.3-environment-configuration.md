This document provides a comprehensive guide to the environment variables and configuration options used in TT-Studio. The system relies on a central `.env` file (derived from `.env.default`) to manage service URLs, security secrets, feature flags, and hardware-specific settings across the backend, frontend, agent, and vector database services.

## Environment File Structure

TT-Studio uses a template-based approach for environment management. The application reads configuration from a `.env` file at runtime, which is typically generated during the initial setup.

### Configuration Lifecycle

**Sources**: [app/.env.default 1-3](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L1-L3)[app/docker-compose.yml 30-31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L30-L31)[app/docker-compose.yml 100-111](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L100-L111)

## Core Configuration Reference

The configuration is divided into functional categories that control different aspects of the system.

### Core Application Paths

These variables define the workspace and storage locations used by Docker volumes.

| Variable | Description | Implementation Detail |
| --- | --- | --- |
| `TT_STUDIO_ROOT` | Absolute path to the repository root. | Used for relative volume mapping in `docker-compose.yml`[app/docker-compose.yml 34](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L34-L34) |
| `HOST_PERSISTENT_STORAGE_VOLUME` | Path on the host machine for data persistence. | Maps to `${TT_STUDIO_ROOT}/tt_studio_persistent_volume`[app/.env.default 6](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L6-L6) |
| `INTERNAL_PERSISTENT_STORAGE_VOLUME` | Mount point inside containers. | Standardized as `/tt_studio_persistent_volume`[app/.env.default 7](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L7-L7) |
| `BACKEND_API_HOSTNAME` | Internal network alias for the backend. | Set to `tt-studio-backend-api` for inter-container communication [app/.env.default 8](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L8-L8) |

### Artifact and Version Control

TT-Studio allows targeting specific versions of the Tenstorrent inference stack.

*   `TT_INFERENCE_ARTIFACT_BRANCH`: Specifies the git branch for the inference engine (e.g., `tt_qb2_launch_branch`) [app/.env.default 15](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L15-L15)
*   `TT_INFERENCE_ARTIFACT_VERSION`: Recommended for production; allows pinning to a specific release tag like `v0.10.0`[app/.env.default 13-14](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L13-L14)

### Security and Authentication

These secrets are critical for securing the communication between the distributed services.

*   `JWT_SECRET`: Secret key used for signing tokens between the Backend and the Agent/Inference services [app/.env.default 18](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L18-L18)
*   `DJANGO_SECRET_KEY`: Standard Django security key for session management and CSRF protection [app/.env.default 19](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L19-L19)
*   `DOCKER_CONTROL_JWT_SECRET`: Specific secret for the **Docker Control Service** (port 8002) to authorize container lifecycle operations [app/.env.default 27](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L27-L27)
*   `RAG_ADMIN_PASSWORD`: Credentials required to access the RAG management interface when `VITE_ENABLE_RAG_ADMIN` is active [app/.env.default 40](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L40-L40)

## Service URLs and Integration

TT-Studio can operate in a hybrid mode, utilizing local Tenstorrent hardware or falling back to cloud-based APIs.

### Docker Control Service

Unlike standard Docker setups that mount `/var/run/docker.sock`, TT-Studio uses a secure proxy:

*   `DOCKER_CONTROL_SERVICE_URL`: Defaults to `http://host.docker.internal:8002`[app/.env.default 26](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L26-L26) This allows the backend to manage containers without root socket access inside the container [app/docker-compose.yml 74-76](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L74-L76)

### Feature Flags (VITE_*)

These variables are injected into the frontend build process via Vite.

*   `VITE_ENABLE_DEPLOYED`: Boolean flag. If `true`, enables the "Models Deployed" interface and local hardware management [app/.env.default 36](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L36-L36)
*   `VITE_ENABLE_RAG_ADMIN`: Boolean flag. If `true`, enables the document ingestion and vector database management UI [app/.env.default 37](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L37-L37)

**Sources**: [app/.env.default 32-37](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L32-L37)[app/docker-compose.yml 12-13](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L12-L13)

### External Cloud APIs

Used primarily when local hardware is unavailable or when `VITE_ENABLE_DEPLOYED` is set to false.

| Service | URL Variable | Token Variable |
| --- | --- | --- |
| **Chat/LLM** | `CLOUD_CHAT_UI_URL` | `CLOUD_CHAT_UI_AUTH_TOKEN` |
| **Vision (YOLO)** | `CLOUD_YOLOV4_API_URL` | `CLOUD_YOLOV4_API_AUTH_TOKEN` |
| **Speech (STT)** | `CLOUD_SPEECH_RECOGNITION_URL` | `CLOUD_SPEECH_RECOGNITION_AUTH_TOKEN` |
| **Images (SD)** | `CLOUD_STABLE_DIFFUSION_URL` | `CLOUD_STABLE_DIFFUSION_AUTH_TOKEN` |

### Agent Service Polling Configuration

The `tt_studio_agent` uses specific environment variables to manage its circuit breaker and polling logic when waiting for LLM containers to become healthy:

*   `AGENT_LLM_POLLING_TIMEOUT`: Maximum time to wait for a model container [app/docker-compose.yml 108](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L108-L108)
*   `AGENT_LLM_POLLING_MAX_ATTEMPTS`: Number of retries before failing the request [app/docker-compose.yml 109](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L109-L109)
*   `AGENT_LLM_POLLING_CIRCUIT_BREAKER_THRESHOLD`: Threshold for disabling requests to a failing model [app/docker-compose.yml 110](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L110-L110)

### LLM Thinking Blocks Configuration

The frontend component `StreamingMessage.tsx` processes the content stream based on tokens that are often influenced by the model's configuration. While not a direct `.env` variable, the logic in `processContent` relies on the model outputting `<think>` tags, which can be configured via model-specific environment overrides during deployment [app/frontend/src/components/chatui/StreamingMessage.tsx 22-49](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/chatui/StreamingMessage.tsx#L22-L49)
