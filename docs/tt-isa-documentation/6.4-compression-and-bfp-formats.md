This document describes the compression mechanisms and Block Floating-Point (BFP) formats used for bandwidth-efficient data storage and transfer in L1 memory. These mechanisms are primarily utilized by the [Unpacker System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.3-unpacker-system-(unpacr)) when moving data from L1 to the compute registers.

For details on the format conversion pipeline that applies after decompression, see [Format Conversion Pipeline](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.3-format-conversion-pipeline). For information about data type encodings and their computational properties, see [Data Type Catalog](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.2-data-type-catalog).

## Overview

The Tensix architecture supports two orthogonal bandwidth optimization techniques:

1.   **Zero-Run Compression (Delta Encoding)**: Sparse data representation where runs of zeroes between non-zero values are encoded using 4-bit delta values, achieving up to 32× compression for sparse data
2.   **Block Floating-Point (BFP)**: Shared-exponent formats where groups of 16 datums share a single exponent, reducing per-datum storage from 16 bits to 2, 4, or 8 bits

These mechanisms can be combined: compressed BFP data stores both the compressed mantissas and the shared exponents. The Unpacker hardware transparently handles decompression and format conversion during data movement from L1 to registers.

## Zero-Run Compression

### Compression Mechanism

Zero-run compression encodes sparse data by storing only the non-zero datums alongside 4-bit "delta" values that indicate the number of zeroes preceding each non-zero datum. The delta value can represent 0-15 consecutive zeroes.

**Key Properties:**

*   **Delta Range**: 0-15 zeroes per non-zero datum (4 bits per delta)
*   **Chunk Size**: Compression operates on 32-datum chunks (16 bytes of deltas per chunk)
*   **Maximum Sparsity**: If all deltas are 15, this represents 16 non-zero values among 496 total positions (32× compression ratio)
*   **Storage Overhead**: 0.5 bytes per datum for delta storage

### Row Start Table

Compressed tiles include a "row start table" that enables random access to individual rows without sequential decompression. Each entry in this table is a 16-bit offset indicating where a particular row's data begins within the compressed datum stream.

The number of rows depends on whether the tile uses "blobs" or standard row-major layout:

*   **Standard mode**: `NumRows = YDim × ZDim × WDim`
*   **Blob mode**: `NumBlobs = BlobsPerXYPlane × ZDim × WDim`

The final entry (at index `NumRows` or `NumBlobs`) stores the offset to the end of the compressed data, allowing software to calculate the compressed size of the last row.

### Decompression Algorithm

The decompression process reads datums and their corresponding deltas, expanding each into the datum followed by 0-15 zeroes:

**Address Calculation:**

*   Datums and deltas are interleaved in 32-element chunks
*   Within each chunk: 32 datums followed by 32 deltas (16 bytes)
*   Address wrapping occurs at configurable FIFO boundaries

## Block Floating-Point Formats

### BFP Format Family

Block Floating-Point formats reduce storage requirements by sharing exponents across groups of 16 datums. Two BFP families exist, differing in their target conversion format:

| Format | Bits per Datum | Exponent Storage | Target Format | Typical Use |
| --- | --- | --- | --- | --- |
| `BFP8` | 8 bits | 1 byte per 16 datums | BF16 or TF32 | General reduced-precision |
| `BFP4` | 4 bits | 1 byte per 16 datums | BF16 or TF32 | High sparsity or very low precision |
| `BFP2` | 2 bits | 1 byte per 16 datums | BF16 or TF32 | Extreme compression |
| `BFP8a` | 8 bits | 1 byte per 16 datums | FP16 | IEEE-compatible path |
| `BFP4a` | 4 bits | 1 byte per 16 datums | FP16 | IEEE-compatible path |
| `BFP2a` | 2 bits | 1 byte per 16 datums | FP16 | IEEE-compatible path |

**Storage Savings:**

*   BFP8: 8 + 1/16 = 8.0625 bits per datum vs 16 bits (≈50% reduction)
*   BFP4: 4 + 1/16 = 4.0625 bits per datum vs 16 bits (≈75% reduction)
*   BFP2: 2 + 1/16 = 2.0625 bits per datum vs 16 bits (≈87% reduction)

### BFP Encoding

BFP formats store each datum as a sign bit plus magnitude bits, with a shared exponent:

**Reconstruction Algorithm (BFP8 to BF16):**

**Key Steps:**

1.   Extract sign and magnitude from datum
2.   Count leading zeros in magnitude
3.   Normalize magnitude by left-shifting
4.   Adjust shared exponent by subtracting leading zero count
5.   Assemble BF16 result: sign (1 bit) | exponent (8 bits) | mantissa (7 bits)

### BFPa to FP16 Conversion

The "a" variants (BFP8a, BFP4a, BFP2a) convert to FP16 instead of BF16, using a similar algorithm with different exponent field width:

**Difference from BFP→BF16:**

*   FP16 uses 5-bit exponent vs BF16's 8-bit exponent
*   FP16 has 10-bit mantissa vs BF16's 7-bit mantissa
*   Exponent overflow check required (FP16 can't represent as wide a range)

**Sources:**

*   [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 508-520](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L508-L520)

### Exponent Storage

BFP formats require one 8-bit exponent per 16 datums. The exponent section layout depends on the format:

**Configuration Options:**

*   **Normal mode**: Exponents stored in dedicated section after row start table
*   **Forced exponent mode**: Single fixed exponent for entire tile (no storage required)
*   **NoBFPExpSection**: For BFP2/BFP4 only, omits exponent section (must use forced mode)

## Combined Compression and BFP

When both compression and BFP are used together, the data layout becomes:

**Address Calculation:**

**Processing Order:**

1.   Use row start table to find compressed row offset
2.   Navigate to appropriate 32-datum chunk
3.   For each datum: read BFP datum (2/4/8 bits) + delta (4 bits) + exponent (8 bits per 16 datums)
4.   Convert BFP datum to target format using shared exponent
5.   Write converted datum followed by zero-run indicated by delta

## Unpacker Integration

### Format Configuration

The `UNPACR` instruction accesses format information from the tile descriptor and configuration registers:

| Configuration Field | Location | Purpose |
| --- | --- | --- |
| `InDataFormat` | `THCON_SEC[n].TileDescriptor.InDataFormat` or `THCON_SEC[n].Unpack_data_format_cntx[i]` | Specifies input format (BFP2/4/8, etc.) |
| `IsUncompressed` | `THCON_SEC[n].TileDescriptor.IsUncompressed` or `THCON_SEC[n].REG2_Disable_zero_compress_cntx[i]` | Enables/disables decompression |
| `NoBFPExpSection` | `THCON_SEC[n].TileDescriptor.NoBFPExpSection` | Omits exponent section for BFP2/4 |
| `REG2_Force_shared_exp` | `THCON_SEC[n].REG2_Force_shared_exp` | Uses fixed exponent |
| `FORCED_SHARED_EXP_shared_exp` | `UNP[n].FORCED_SHARED_EXP_shared_exp` | Fixed exponent value |

### Decompression Cache

The Unpacker maintains a small cache of row start table entries to avoid redundant L1 reads. Software can flush this cache using the `UNPACR` flush variant:

**Cache Properties:**

*   4 one-entry caches per unpacker
*   Caches row start table entries
*   Must be flushed when overwriting compressed data in place

**Sources:**

### Performance Considerations

Compression affects unpacker throughput:

| Configuration | L1 Bandwidth | Notes |
| --- | --- | --- |
| Uncompressed, normal | x1/x2/x4 speed | Configurable via `Throttle_mode` |
| Compressed (`!IsUncompressed`) | x1 speed | Always limited to 16 bytes/cycle |
| BFP2/BFP2a uncompressed | x1 speed | Sub-byte datums constrain throughput |
| Upsampling (≥3 zeroes) | x1 speed | Output bandwidth limited |

**Address Calculation Overhead:**

*   Uncompressed: 2 cycles minimum
*   Compressed: 2+ cycles (table lookup required)

During address calculation, the issuing thread stalls and other threads cannot start new `UNPACR` instructions.
