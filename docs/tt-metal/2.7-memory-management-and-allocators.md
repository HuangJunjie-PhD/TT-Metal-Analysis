This document describes the memory management system in tt-metal, including the allocator architecture, memory layouts, and address translation mechanisms. Memory management handles allocation and organization of on-chip (L1) and off-chip (DRAM) memory across Tensix cores and multi-device meshes.

## Memory Architecture Overview

The tt-metal memory system manages two primary memory types on Tenstorrent devices:

*   **L1 Memory**: Fast, on-chip memory local to each Tensix core (typically 1-1.5 MB per core).
*   **DRAM**: Larger, off-chip memory accessible via memory controllers (multiple GB per channel).

The memory system partitions L1 and DRAM into several regions with specific purposes. For L1, the space is divided to accommodate RISC binaries, kernel arguments, and user buffers.

## Allocator System Architecture

The allocator system uses a layered abstraction where `AllocatorImpl` handles the core logic and `BankManager` manages specific memory pools.

**Allocator Class Hierarchy**

### Key Entities

*   **`AllocatorImpl`**: The base implementation class that coordinates between different memory managers (DRAM, L1, TRACE) [[tt_metal/impl/allocator/allocator.cpp 28-29](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L28-L29)]. It maintains the mapping between bank IDs, logical cores, and DRAM channels [[tt_metal/impl/allocator/allocator.cpp 31-35](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L31-L35)].
*   **`BankManager`**: Manages a collection of memory banks for a specific `BufferType`. It handles the mapping of bank IDs to offsets and invokes the underlying allocation algorithm [[tt_metal/impl/allocator/allocator.cpp 153-170](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L153-L170)].
*   **`FreeListOpt`**: The primary allocation algorithm used within `BankManager` to manage contiguous blocks of memory.

Sources: [[tt_metal/impl/allocator/allocator.cpp 115-151](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L115-L151)], [[tt_metal/impl/allocator/bank_manager.hpp 1](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/bank_manager.hpp#L1-L1)]

## Buffer Management

Buffers are the primary abstraction for device memory. A `Buffer` object represents a contiguous or interleaved memory region on the device.

### Buffer Types and Sharding

Buffers are categorized by `BufferType` (DRAM, L1, L1_SMALL, TRACE) and `TensorMemoryLayout`.

*   **Interleaved**: Pages are distributed across all banks in a round-robin fashion [[tt_metal/impl/allocator/allocator.cpp 51](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L51-L51)].
*   **Sharded**: Data is partitioned into shards, where each shard is mapped to a specific core. Sharding types include `HEIGHT_SHARDED`, `WIDTH_SHARDED`, and `BLOCK_SHARDED`[[tt_metal/impl/buffers/buffer.cpp 98-146](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/buffers/buffer.cpp#L98-L146)].
*   **ND Sharding**: A newer sharding specification allows for flexible sharding across multiple dimensions, supported by `BufferDistributionSpec` which computes complex page mappings [[tt_metal/impl/buffers/buffer_distribution_spec.cpp 88-120](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/buffers/buffer_distribution_spec.cpp#L88-L120)].
*   **Per-Core Allocation**: A mode where each core in a shard grid receives an independent address, requiring `AllocatorMode::HYBRID`. This is currently only supported for L1 buffers [[tt_metal/impl/allocator/allocator.cpp 129-151](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L129-L151)].

### Mesh Buffers and Distributed Memory

In distributed systems, memory is managed across a `MeshDevice`.

*   **Global Semaphores**: Used for synchronization across the mesh, allocated as sharded L1 buffers [[tt_metal/impl/buffers/global_semaphore.cpp 55-65](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/buffers/global_semaphore.cpp#L55-L65)].
*   **Global Circular Buffers**: Facilitate data movement between sender and receiver cores across the mesh, supporting both Worker-to-Worker and DRAM-to-Worker transfers [[tt_metal/impl/buffers/global_circular_buffer.cpp 109-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/buffers/global_circular_buffer.cpp#L109-L155)].

## Address Translation and Banking

The system uses a banking scheme to simplify address generation for kernels. Each bank is associated with a specific logical core or DRAM channel.

### L1 Banking

L1 memory is divided into banks, where each compute core typically represents one bank. The `AllocatorImpl` initializes these banks during device setup [[tt_metal/impl/allocator/allocator.cpp 76-98](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L76-L98)]. Space is reserved for RISC binaries and kernel arguments by starting user allocations at `l1_unreserved_base`[[tt_metal/impl/allocator/allocator.cpp 88](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L88-L88)].

### DRAM Banking

DRAM is partitioned by channels. The allocator assigns one bank per DRAM channel [[tt_metal/impl/allocator/allocator.cpp 52-57](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L52-L57)]. It also manages a separate `trace_buffer_manager_` for trace regions at the top of DRAM [[tt_metal/impl/allocator/allocator.cpp 60-67](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L60-L67)].

## Memory Allocation Strategies

### Hybrid Allocation

In `AllocatorMode::HYBRID`, the allocator supports per-core allocation where each core in a grid gets a unique address [[tt_metal/impl/allocator/allocator.cpp 130-132](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/allocator/allocator.cpp#L130-L132)]. This is critical for sharded buffers where memory pressure might vary across the grid.

### Data Movement and Async Runtime

Buffer data is moved to/from the device via the command queue system or direct buffer APIs.

*   **Async Execution**: Tensors can be allocated and operated upon asynchronously. The `Buffer` class tracks unique IDs for graph tracking and resource management [[tt_metal/impl/buffers/buffer.cpp 182-185](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/impl/buffers/buffer.cpp#L182-L185)].
*   **Read/Write Operations**: `WriteToBuffer` and `ReadFromBuffer` facilitate host-device data transfers [[tt_metal/api/tt-metalium/tt_metal.hpp 98-143](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/api/tt-metalium/tt_metal.hpp#L98-L143)]. For sharded buffers, `ReadShard` allows reading data from specific cores [[tt_metal/api/tt-metalium/tt_metal.hpp 145-161](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/api/tt-metalium/tt_metal.hpp#L145-L161)].
