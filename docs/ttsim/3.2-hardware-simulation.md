# Hardware Simulation

Relevant source files
*   [README.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1)

## Purpose and Scope

This page documents the hardware components simulated by ttsim, focusing on the Tensix core architecture and the fidelity levels of different functional units. It explains which hardware features are bit-accurately simulated versus approximate implementations, the numerical accuracy model, and the timing behavior of the simulator.

For information about how ttsim integrates with TT-Metalium and the environment configuration required to use the simulator, see [TT-Metalium Integration](https://deepwiki.com/tenstorrent/ttsim/3.3-tt-metalium-integration). For details on the overall system architecture and layering, see [System Architecture](https://deepwiki.com/tenstorrent/ttsim/3.1-system-architecture). For practical guidance on achieving deterministic results in your workloads, see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy).

## Simulated Hardware Components

ttsim provides a virtual implementation of Tenstorrent chip architectures, currently supporting Wormhole and Blackhole devices. The simulator executes on Linux/x86_64 host systems and emulates the behavior of Tenstorrent silicon without requiring physical hardware.

Each architecture has its own simulator binary:

*   `libttsim_wh.so` - Wormhole simulator
*   `libttsim_bh.so` - Blackhole simulator

The simulator implements SOC (System on Chip) topology defined by YAML descriptor files:

*   `wormhole_b0_80_arch.yaml` - Wormhole B0 configuration
*   `blackhole_140_arch.yaml` - Blackhole configuration

## Tensix Core Architecture

The core computational element simulated by ttsim is the Tensix processor. The Tensix pipeline consists of four major functional units that process data in sequence:

### Tensix Pipeline Diagram

Sources: [README.md 99-102](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L99-L102)

### Unpacker

The Unpacker is the first stage in the Tensix pipeline, responsible for unpacking and preparing data from memory for processing by downstream units.

**Implementation Status**: Fully bit-accurate. The Unpacker simulation is believed to match silicon behavior bit-for-bit across all operations.

Sources: [README.md 100](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L100-L100)

### SFPU (Special Function Processing Unit)

The SFPU executes special mathematical functions including transcendental operations, activation functions, and other specialized computations that require dedicated hardware support.

**Implementation Status**: Fully bit-accurate. The SFPU simulation is believed to match silicon behavior bit-for-bit across all operations.

Sources: [README.md 100](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L100-L100)

### FPU (Floating Point Unit)

The FPU handles general floating-point arithmetic operations. The simulation fidelity varies by opcode category:

| Opcode Category | Implementation Status | Notes |
| --- | --- | --- |
| `MOV*` | Bit-Accurate | Move operations fully implemented |
| `GMPOOL` | Bit-Accurate | Global max pooling operations fully implemented |
| `ELW*` | Work in Progress | Element-wise operations not yet bit-accurate |
| `MVMUL` | Work in Progress | Matrix-vector multiply not yet bit-accurate |
| `GAPOOL` | Work in Progress | Global average pooling not yet bit-accurate |

The FPU is under active development to achieve full bit-accuracy across all opcode categories.

Sources: [README.md 101-102](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L101-L102)

### Packer

The Packer is the final stage in the Tensix pipeline, responsible for packing processed data and writing it back to memory in the appropriate format.

**Implementation Status**: Fully bit-accurate. The Packer simulation is believed to match silicon behavior bit-for-bit across all operations.

Sources: [README.md 100](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L100-L100)

## Numerical Accuracy Model

ttsim is designed with a **bit-exact numerical accuracy** goal. The target is to match silicon behavior bit-for-bit across all instructions, opcodes, functional units, and special cases, including the precise bit representation of NaNs (Not-a-Number values) produced by operations.

### Bit-Exactness Goals

The simulation aims to provide:

*   Identical results for all floating-point computations
*   Exact matching of integer arithmetic
*   Precise reproduction of special floating-point values (infinities, NaNs, denormals)
*   Matching NaN bit patterns (payload and signaling/quiet bit)
*   Identical behavior for edge cases and corner cases

Sources: [README.md 78-83](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L78-L83)

### Sources of Divergence

While the goal is bit-exactness, certain execution patterns can produce divergent results between simulator and silicon:

| Divergence Source | Description | Mitigation |
| --- | --- | --- |
| **Timing-Dependent Operand Order** | Operations with non-deterministic execution order may compute in different sequences | Explicitly serialize operations in deterministic order |
| **Hardware Entropy Sources** | Random number generators and entropy sources cannot be reproduced | Avoid or seed deterministically |
| **Performance Counters** | Cycle counters and timers will differ due to simulation overhead | Do not use for correctness checks |
| **Missing Synchronization** | Insufficient cache flushes, memory fences, or barriers | Add proper synchronization primitives |
| **Undefined Behavior** | ISA violations or unpredictable value usage | Avoid ISA violations |
| **Unimplemented Operations** | Operations not yet fully simulated | Check current implementation status |

Sources: [README.md 85-91](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L85-L91)

### Floating-Point Reduction Example

For algorithms like floating-point reductions, non-associativity of floating-point addition means that operand order affects results:

```
# Non-deterministic (may diverge)
sum = 0.0
for i in parallel:
    sum += values[i]  # Order depends on race conditions

# Deterministic (matches simulator and silicon)
sum = 0.0
for i in sequential_order:
    sum += values[i]  # Fixed order, reproducible
```

The simulator may evaluate operations in any order permitted by software synchronization, including orders that are extremely unlikely on silicon due to timing characteristics.

Sources: [README.md 93-97](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L93-L97)

## Timing Model

ttsim does not provide cycle-accurate timing simulation. The simulator executes operations in any order that satisfies the synchronization constraints defined by the software. This means:

*   Operation completion order is not guaranteed to match silicon
*   Performance counter values will differ from silicon
*   Race conditions may manifest differently than on hardware
*   Timing-dependent bugs may behave differently in simulation

The simulator focuses on functional correctness rather than performance modeling. For accurate performance characterization, testing on physical silicon is required.

## Error Detection and Reporting

ttsim implements comprehensive error detection to catch ISA violations, unsupported operations, and simulator bugs. Errors are categorized to help users understand the nature of issues:

### Error Category Hierarchy

Sources: [README.md 63-76](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L63-L76)

### Error Category Definitions

| Category | Meaning | Action Required |
| --- | --- | --- |
| `UndefinedBehavior` | ISA specification violation with unpredictable results | Fix software to comply with ISA |
| `UnpredictableValueUsed` | Using value that ISA does not define | Fix software to avoid undefined values |
| `NonContractualBehavior` | Behavior not guaranteed by ISA contract | Modify software to not depend on this behavior |
| `UntestedFunctionality` | Feature implemented but lacks test coverage | Feature disabled until testing complete |
| `UnimplementedFunctionality` | Feature planned but not yet implemented | Wait for future release or use silicon |
| `UnsupportedFunctionality` | Feature unlikely to be implemented | Requires strong justification to add |
| `MissingSpecification` | Feature awaits internal specification | Requires internal work before implementation |
| `SystemError` | Operating system error (file I/O, memory, etc.) | Check OS configuration and resources |
| `ConfigurationError` | Environment variables or configuration files incorrect | Fix configuration (see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration)) |
| `AssertionFailure` | Internal simulator bug | Report issue to ttsim team |

For ISA-related errors (`UndefinedBehavior`, `UnpredictableValueUsed`, `NonContractualBehavior`), consult the [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) for detailed explanations of these concepts.

For complete error handling guidance, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference). For reporting simulator bugs, see [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues).

Sources: [README.md 68-76](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L68-L76)

## Architecture-Specific Simulation

ttsim provides separate simulator binaries for different chip architectures:

### Supported Architectures

| Architecture | Simulator Binary | SOC Descriptor | Maturity Level |
| --- | --- | --- | --- |
| Wormhole B0 | `libttsim_wh.so` | `wormhole_b0_80_arch.yaml` | More mature |
| Blackhole | `libttsim_bh.so` | `blackhole_140_arch.yaml` | Less mature |

Both architectures can run many tt-metal and ttnn examples/tests in slow dispatch mode, as well as numerous LLK (Low-Level Kernel) tests. Wormhole is currently in more mature shape than Blackhole.

### SOC Descriptor Requirements

Each simulator binary must be co-located with a `soc_descriptor.yaml` file that defines the chip topology. The simulator loads this descriptor at runtime to understand the device configuration, including:

*   Core grid dimensions
*   Memory hierarchy
*   NOC (Network on Chip) topology
*   Device capabilities

The SOC descriptor must be copied from the TT-Metalium repository (`$TT_METAL_HOME/tt_metal/soc_descriptors/`) to the same directory as the simulator binary, and must be named exactly `soc_descriptor.yaml`.
