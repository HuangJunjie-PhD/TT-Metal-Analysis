This document covers the coordinate generation algorithms used by tt-topology to assign spatial coordinates to Tenstorrent chips based on their physical connections. These algorithms analyze chip connectivity patterns and generate appropriate (x,y) coordinate mappings for different topology layouts including mesh, linear, torus, and isolated configurations.

For information about how these algorithms integrate with hardware flashing operations, see [Hardware Interface](https://deepwiki.com/tenstorrent/tt-topology/4.2-hardware-interface). For details on the backend systems that orchestrate these algorithms, see [Backend Systems](https://deepwiki.com/tenstorrent/tt-topology/4.1-backend-systems).

## Algorithm Overview

The topology algorithms in tt-topology transform detected chip connectivity into spatial coordinate systems. The system supports four primary topology types, each requiring different algorithmic approaches:

| Layout Type | Algorithm Approach | Coordinate Pattern | Primary Function |
| --- | --- | --- | --- |
| mesh | Breadth-First Search (BFS) | 2D grid coordinates | `generate_coordinates_mesh` |
| linear | Cycle detection + longest path | 1D sequential coordinates | `generate_coordinates_torus_or_linear` |
| torus | Cycle detection | 1D ring coordinates | `generate_coordinates_torus_or_linear` |
| isolated | Static assignment | (0,0) or (1,0) | Direct assignment |

## Data Structures and Connection Analysis

The algorithms operate on connection maps generated from hardware detection. The `generate_connection_map` function creates a comprehensive adjacency representation of chip connectivity.

**Connection Data Processing Pipeline**

The connection map structure stores chip connectivity information:

*   **chip_data**: Maps ethernet board info to chip metadata including connections
*   **connections**: List of tuples `(neighbor_chip_id, connection_type)` where connection types are "X" (regular) or "T" (Tfly)
*   **adjacency_map**: Simplified representation mapping chip IDs to lists of connected chip IDs

## Mesh Coordinate Generation

The mesh algorithms use breadth-first search to assign 2D grid coordinates while respecting physical connection constraints. Two mesh algorithm variants are implemented.

### Connection-Independent Mesh Algorithm

The `generate_mesh_connection_independent` function implements a general BFS approach that works with any connection topology:

**BFS Mesh Coordinate Assignment**

**Coordinate Validation Rules** (lines 542-558):

1.   Candidate coordinates must not be already assigned to another node
2.   Coordinates cannot be negative
3.   For each neighbor with assigned coordinates, candidate must be exactly ±1 in X or Y direction

### Connection-Type-Aware Mesh Algorithm

The `generate_coordinates_mesh` function implements mesh coordinate generation with specific rules for connection types:

**Connection Type Rules**:

*   **Tfly connections ("T")**: Always increment in Y direction (lines 618-620)
*   **Local-to-Remote chip connections**: Always increment in X direction (lines 622-624)
*   **Regular connections ("X")**: Use unused direction from parent node (lines 628-644)

## Linear and Torus Coordinate Generation

The `generate_coordinates_torus_or_linear` function uses cycle detection to generate coordinates for ring and chain topologies.

**Cycle Detection Algorithm Flow**

**Longest Path Fallback Algorithm**: When no complete cycle exists, the system falls back to `find_longest_simple_path` which uses depth-first search to find the longest simple path in the graph. This accommodates linear topologies that don't form complete rings.

**DFS Path Finding Process** (lines 725-741):

1.   Try DFS from each node as starting point
2.   Track longest path found across all starting points
3.   Use backtracking to explore all possible paths
4.   Return longest simple path for coordinate assignment

## Connection Validation

The `check_num_available_connections` function validates that the detected topology matches expected connection patterns using a mathematical formula.

**Connection Count Validation**:

*   **Expected connections formula**: `((3 * total_chips) - 4) // 2`
*   **Assumption**: Even number of chips in 2×N configuration
*   **Validation logic**: Compare actual unique connections against expected count

**Example for 4×2 Configuration** (lines 450-454):

## Algorithm Integration and Coordinate Flashing

The coordinate generation algorithms integrate with hardware programming through the `flash_to_specified_state` function, which translates logical coordinates into hardware register values.

**Coordinate to Hardware Mapping**:

*   **Mesh layouts**: Use 2D coordinates directly for `ETH_PARAM_CHIP_COORD`
*   **Linear/Torus layouts**: Use (0, sequence_index) coordinate pattern
*   **Port disable calculations**: Determine which ethernet ports to enable based on topology connections

**Port Disable Logic** (lines 812-849):

1.   For mesh: Enable all ports (`port_disable = 0x0`)
2.   For torus/linear: Start with all disabled (`port_disable = 0xFFFF`)
3.   Enable only ports connected to adjacent chips in the topology sequence
4.   Check each of 16 ethernet ports against remote chip information
5.   Clear disable bit for ports connecting to adjacent topology neighbors

The algorithms ensure that generated coordinates create the intended physical topology by carefully managing both spatial positioning and network connectivity.
