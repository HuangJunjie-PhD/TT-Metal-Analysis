The JavaScript test suite provides unit and integration testing for the Eleventy-based frontend of the `tt-awesome` project. It ensures that custom Nunjucks filters, data aggregation logic, and the processing of project entries function correctly before deployment. These tests run in a Node.js environment and complement the Python-based validation suite.

## Overview and Implementation

The suite is divided into two primary files:

1.   **`tests/test_eleventy_filters.js`**: Unit tests for complex data transformation filters used in templates.
2.   **`tests/test_data_files.js`**: Integration tests verifying the Eleventy data pipeline, specifically how GitHub metadata is merged with entry JSON files.

## Nunjucks Filter Unit Tests

The `tests/test_eleventy_filters.js` file validates the logic within `.eleventy.js` without requiring a full Eleventy build cycle. It achieves this by passing a `stubConfig` object to the exported configuration function to capture filter definitions [tests/test_eleventy_filters.js 14-28](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_eleventy_filters.js#L14-L28)

## Data Pipeline Integration Tests

The `tests/test_data_files.js` script verifies the logic in `src/_data/entries.js` and `src/_data/recentReleases.js`. It ensures that raw metadata from the GitHub crawler is correctly associated with project entries.

### Mocking and Injection

Because these data files rely on the filesystem and external JSON, the test suite patches `Module._load` to intercept requests for `github_meta.json` and returns controlled test fixtures [tests/test_data_files.js 16-44](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_data_files.js#L16-L44) It also stubs `fs.readdirSync` and `fs.readFileSync` to simulate the presence of entry files in the `entries/` directory [tests/test_data_files.js 62-74](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_data_files.js#L62-L74)

### Release Logic Verification

The suite specifically targets the `latestStableRelease` logic, which is critical for the "Recent Releases" pane on the website:

*   **Stable Detection**: Verifies that the logic correctly identifies the most recent non-prerelease tag [tests/test_data_files.js 87-91](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_data_files.js#L87-L91)
*   **Prerelease Filtering**: Ensures that projects with only alpha/beta/RC tags do not show a "stable" release [tests/test_data_files.js 93-96](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_data_files.js#L93-L96)
*   **Feed Aggregation**: Confirms that `recentReleases.js` only includes entries that actually have a stable release and sorts them newest-first [tests/test_data_files.js 130-146](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_data_files.js#L130-L146)

## Running Tests

Tests are executed using standard Node.js without a heavy test runner:

`# Run filter unit testsnode tests/test_eleventy_filters.js # Run data pipeline integration testsnode tests/test_data_files.js`
Successful execution is indicated by prefixed checkmarks (e.g., `✓ articleFeedItems: only article-type links included`) [tests/test_eleventy_filters.js 78](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_eleventy_filters.js#L78-L78) Failure results in an `AssertionError` which halts the CI process.
