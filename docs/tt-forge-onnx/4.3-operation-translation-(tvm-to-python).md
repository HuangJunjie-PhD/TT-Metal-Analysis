## Purpose and Scope

Operation Translation is the mechanism by which operations from ML frameworks (PyTorch, ONNX, TensorFlow, etc.) are mapped and transformed into Forge operations, and ultimately lowered to MLIR for hardware execution. This page covers:

*   TVM Relay operation name mapping to PyTorch/Forge operations
*   Attribute extraction and population from TVM nodes
*   Pattern-based recognition of composite operations
*   MLIR lowering and attribute type conversion

For details on the overall framework integration pipeline, see [Framework Integration](https://deepwiki.com/tenstorrent/tt-forge-onnx/4-framework-integration). For information on the operation system architecture and individual operation implementations, see [Operation System](https://deepwiki.com/tenstorrent/tt-forge-onnx/5-operation-system).

The `tvm_to_pytorch_op_map` dictionary provides the primary mapping from TVM Relay operation names to PyTorch operation names used internally by Forge.

### Core Mapping Dictionary

The mapping is defined in `tvm_to_pytorch_op_map` at [forge/forge/tvm_to_python.py 416-470](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L416-L470):

| TVM Operation | PyTorch/Forge Operation | Notes |
| --- | --- | --- |
| `nn.dense` | `linear` | Dense layer → linear |
| `nn.matmul` | `matmul` | Matrix multiplication |
| `nn.batch_matmul` | `matmul` | Batched matmul → matmul |
| `nn.softmax` | `softmax` | Softmax activation |
| `nn.log_softmax` | `log_softmax` | Log-softmax |
| `strided_slice` | `index_select` | Slicing with strides |
| `scatter_elements` | `scatter_add` | Scatter with reduction |
| `forge_cpudevice.adv_index` | `adv_index` | Advanced indexing |

## Attribute Population and Translation

Each operation type requires specific attributes to be extracted from TVM Relay nodes and formatted for PyTorch/Forge consumption. The `populate_torch_*_args` functions handle this translation.

### Attribute Population Architecture

**Sources:**[forge/forge/tvm_to_python.py 406-549](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L406-L549)

### Common Attribute Population Patterns

The base function `_populate_torch_init_args` extracts inputs from the TVM graph [forge/forge/tvm_to_python.py 406-413](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L406-L413):

`def _populate_torch_init_args(graph, nid):    curr_node = graph["nodes"][nid]    args = {"inp": [], "attr": {}, "inplace": False}    for i in range(0, int(curr_node["attrs"]["num_inputs"])):        args["inp"].append(graph["nodes"][curr_node["inputs"][i][0]]["forge_name"])    return curr_node, args`
Operation-specific functions then add attributes. For example, `populate_torch_softmax_args`[forge/forge/tvm_to_python.py 160-170](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L160-L170):

`def populate_torch_softmax_args(graph, nid, compiler_cfg):    curr_node, args = _populate_torch_init_args(graph, nid)    args["attr"] = {        "dim": {            "val": int(curr_node["attrs"]["axis"][0][0]),            "inp_pos": 1,        }    }    return args`
### Attribute Format

The returned `args` dictionary follows this structure:

`{    "inp": ["input_node_name", ...],  # Input node names    "attr": {        "attr_name": {            "val": actual_value,      # Attribute value            "inp_pos": position,      # Position in function args            "as_named": True/False    # Whether to pass as named arg        }    },    "inplace": True/False             # Whether operation is in-place}`
**Sources:**[forge/forge/tvm_to_python.py 406-413](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L406-L413)[forge/forge/tvm_to_python.py 160-170](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L160-L170)

### Complex Attribute Transformations

Some operations require complex attribute transformations. For example, `populate_torch_transpose_args` converts TVM's axis permutation format to PyTorch's two-dimension format [forge/forge/tvm_to_python.py 233-263](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L233-L263):

`def populate_torch_transpose_args(graph, nid, compiler_cfg):    curr_node, args = _populate_torch_init_args(graph, nid)        # TVM: [0, 2, 1, 3] → PyTorch: [1, 2]    changed_axes = []    axes = [int(axis) for axis in curr_node["attrs"]["axes"][0]]    for i, axis in enumerate(axes):        if i != axis:            changed_axes.append(str(i))        args["attr"] = {        "dim0": {"val": int(changed_axes[0]), "inp_pos": 1},        "dim1": {"val": int(changed_axes[1]), "inp_pos": 2},    }    return args`
**Sources:**[forge/forge/tvm_to_python.py 233-263](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L233-L263)

## Pattern-Based Operation Recognition

TVM Relay uses pattern matching to identify and transform composite operations before they are translated to Forge operations.

### Pattern Table Architecture

**Sources:**[forge/forge/tvm_calls/relay/op/forge.py 122-142](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L122-L142)

### Pattern Definitions

Patterns are defined using TVM's dataflow pattern DSL [forge/forge/tvm_calls/relay/op/forge.py 60-119](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L60-L119):

`def dense_to_matmul():    data = wildcard()    weight = wildcard()    weight_t = is_op("transpose")(weight)    return is_op("nn.dense")(data, weight_t) def merge_conv2d_with_bias():    input = wildcard()    weight = wildcard()    bias = wildcard()    conv2d = is_op("nn.conv2d")(input, weight)    bias_add = is_op("nn.bias_add")(conv2d, bias)    return bias_add`
The pattern table registers these patterns as composite operations [forge/forge/tvm_calls/relay/op/forge.py 122-142](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L122-L142):

| Pattern Name | TVM Pattern | Forge Operation |
| --- | --- | --- |
| `forge.matmul` | `dense_to_matmul()` | Matrix multiplication |
| `forge.forge_conv2d_with_bias` | `merge_conv2d_with_bias()` | Conv2d with bias |
| `forge.forge_conv2d_transpose_with_bias` | `merge_conv2d_transpose_with_bias()` | Transpose conv with bias |
| `forge.adv_index` | `decompose_adv_index_input_tuple()` | Advanced indexing |
| `forge.concatenate` | `decompose_concatenate()` | Concatenation |
| `forge.dropout` | `dropout_tuple_get_item()` | Dropout (single output) |

**Sources:**[forge/forge/tvm_calls/relay/op/forge.py 60-142](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L60-L142)

### TVM Compile Passes with Patterns

Custom TVM passes use these patterns to decompose and transform operations. For example, `ConvertLayout`[forge/forge/tvm_calls/relay/op/forge_passes.py 19-96](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L19-L96) converts data layout for convolution and pooling operations, while `ResolveConvChannels`[forge/forge/tvm_calls/relay/op/forge_passes.py 99-157](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L99-L157) resolves channel layout issues.

**Sources:**[forge/forge/tvm_calls/relay/op/forge_passes.py 19-157](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L19-L157)

After operations are translated to Forge's internal representation, they must be lowered to MLIR TTIR dialect for hardware execution.

`void init_lowering_handler_map() {    lowering_handler_map["abs"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::AbsOp>;    lowering_handler_map["add"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::AddOp>;    lowering_handler_map["matmul"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::MatmulOp>;    lowering_handler_map["conv2d"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::Conv2dOp>;    lowering_handler_map["softmax"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::SoftmaxOp>;    // ... 80+ operation mappings}`

## Operation Translation Flow

The complete operation translation flow spans multiple stages from framework input to MLIR output:

**Sources:**[forge/forge/tvm_to_python.py 416-549](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L416-L549)[forge/csrc/passes/lower_to_mlir.cpp 512-530](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L512-L530)[forge/csrc/passes/lower_to_mlir.cpp 534-559](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L534-L559)

### Stage-by-Stage Translation Details

#### Stage 1: TVM Frontend Import

Framework models are converted to TVM Relay IR using TVM's frontend converters (e.g., `relay.frontend.from_pytorch`). This produces a Relay Module containing operations like `nn.conv2d`, `nn.dense`, `nn.softmax`.

#### Stage 2: Pattern Matching

The pattern table identifies composite operations and rewrites them into Forge-specific patterns. For example, `nn.dense` followed by `transpose` becomes `forge.matmul`.

#### Stage 3: TVM to PyTorch Translation

The TVM JSON graph representation is processed:

1.   Each node's operation name is looked up in `tvm_to_pytorch_op_map`
2.   If the operation needs attributes, the corresponding `populate_torch_*_args` function extracts and formats them
3.   The result is a PyTorch-compatible operation with properly formatted arguments

#### Stage 4: Forge Graph Construction

PyTorch operations are converted to Forge's internal graph representation with `OpNode` instances containing `OpType` enum values.

#### Stage 5: MLIR Lowering

Each Forge operation is:

1.   Checked for MLIR support via `extract_unique_op_configuration`
2.   Looked up in `lowering_handler_map` to find its MLIR generation handler
3.   Attributes are remapped via `AttributeMapper`
4.   MLIR operation is emitted via `emit_mlir_ttforge_op`

**Sources:**[forge/forge/tvm_to_python.py 1-1200](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L1-L1200)[forge/csrc/passes/lower_to_mlir.cpp 512-559](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L512-L559)

## Example: Conv2D Translation

To illustrate the complete translation flow, consider a Conv2D operation:

### Conv2D Attribute Population

The `populate_conv2d_args` function [forge/forge/tvm_to_python.py 552-592](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L552-L592) extracts Conv2D-specific attributes:

`def populate_conv2d_args(graph, nid, compiler_cfg):    node = graph["nodes"][nid]        strides = [int(stride) for stride in node["attrs"]["strides"][0]]    padding = [int(padding) for padding in node["attrs"]["padding"][0]]    dilation = [int(dilation) for dilation in node["attrs"]["dilation"][0]]    groups = int(node["attrs"]["groups"][0][0])    channel_last = node["attrs"]["data_layout"][0][0] == "NHWC"        # Padding format: [top, left, bottom, right]    return [        ("stride", f"{strides}"),        ("padding", f"{padding}"),        ("dilation", f"{dilation[0]}"),        ("groups", f"{groups}"),        ("channel_last", f"{channel_last}"),    ]`
### Conv2D MLIR Attribute Mapping

The `AttributeMapper` applies these transformations for Conv2D [forge/csrc/passes/lower_to_mlir.cpp 134-138](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L134-L138):

| Forge Attribute | MLIR Name | Target Type |
| --- | --- | --- |
| `stride` | `stride` | `DenseI32ArrayAttr` |
| `padding` | `padding` | `DenseI32ArrayAttr` |
| `dilation` | `dilation` | `DenseI32ArrayAttr` |
| `groups` | `groups` | `I32Attr` |
