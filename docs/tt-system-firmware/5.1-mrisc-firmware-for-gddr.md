## Purpose and Scope

This page documents the MRISC (Memory RISC) firmware that runs on dedicated RISC processors within each GDDR memory controller instance in Blackhole ASICs. The MRISC firmware is responsible for GDDR memory initialization, training, built-in self-test (BIST), and runtime power management. This firmware is loaded by the Chip Management Firmware (CMFW) during the boot sequence and operates independently on each of the 8 GDDR controller instances.

For information about the overall hardware initialization sequence, see [Boot and Initialization Sequence](https://deepwiki.com/tenstorrent/tt-system-firmware/3.1-boot-and-initialization-sequence). For details about ERISC firmware that manages Ethernet controllers, see [ERISC Firmware for Ethernet](https://deepwiki.com/tenstorrent/tt-system-firmware/5.2-erisc-firmware-for-ethernet).

## MRISC Firmware Architecture

### Hardware Context

Each Blackhole ASIC contains 8 GDDR memory controller instances, and each controller includes a dedicated MRISC processor with its own 128KB L1 memory. The MRISC processors execute firmware that handles memory-specific operations independently of the main ARC processor running CMFW.

**MRISC Hardware Specifications**

```
Number of GDDR Instances: 8 (instance 0-7)
MRISC L1 Memory Size:     128KB per instance
NOC2AXI Ports per MRISC:  3 (for NOC access)
L1 Base Address:          0x1_0000_0000 (bit 37 set)
Register Base Address:    0x10_0000_0000 (bit 40 set)
```

### Firmware Components

The MRISC firmware consists of two binary components stored in SPI flash:

| Component | Description | Size | Flash Tag |
| --- | --- | --- | --- |
| `tt_blackhole_gddr_init.bin` | Main MRISC executable code | Variable | `memfw` |
| `tt_blackhole_gddr_params_*.bin` | Board-specific configuration parameters | Variable | `memfwcfg` |

## Initialization Sequence

### Boot Flow Overview

The MRISC firmware initialization occurs as part of the ordered hardware initialization sequence managed by Zephyr's `SYS_INIT` mechanism. The initialization is performed by the `InitMrisc()` function registered with priority 105, placing it after Tensix initialization (104) and before Ethernet initialization (106).

The MRISC L1 memory is organized with specific regions for firmware code, configuration, messaging, and telemetry.

**MRISC L1 Memory Map**

## GDDR Training Process

### Training Overview

After the MRISC firmware is loaded and reset is released, each MRISC independently executes a memory training sequence to calibrate the GDDR interface. The CMFW monitors the training progress by polling status registers written by the MRISC.

### Status Registers

The MRISC communicates its training status through scratch registers accessible via NOC2AXI:

**MRISC Status Register Definitions**

| Register | Address | Purpose | Values |
| --- | --- | --- | --- |
| `MRISC_INIT_STATUS` | `0xFFB14010` (RISC_CTRL_A_SCRATCH_0) | Training state | See below |
| `MRISC_POST_CODE` | `0xFFB14014` (RISC_CTRL_A_SCRATCH_1) | Debug post code | Implementation-specific |
| `MRISC_MSG_REGISTER` | `0xFFB14018` (RISC_CTRL_A_SCRATCH_2) | Message passing | Message type codes |

**MRISC_INIT_STATUS Values**

| Value | Constant | Meaning |
| --- | --- | --- |
| `0x11111111` | `MRISC_INIT_BEFORE` | Set by CMFW before reset release |
| `0x00000000` | `MRISC_INIT_STARTED` | MRISC has started execution |
| `0xDEADBEEF` | `MRISC_INIT_FINISHED` | Training completed successfully |
| `0x0000FA11` | `MRISC_INIT_FAILED` | Training failed |

### Training Monitoring

The `CheckGddrTraining()` function [gddr.c 444-463](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L444-L463) polls `MRISC_INIT_STATUS` with a 1000ms timeout:

`// Polling loop structuredo {    uint32_t poll_val = MriscRegRead32(gddr_inst, MRISC_INIT_STATUS);        if (poll_val == MRISC_INIT_FINISHED) {        return 0;  // Success    }    if (poll_val == MRISC_INIT_FAILED) {        LOG_ERR("Training failed");        return -EIO;    }    k_msleep(1);} while (!sys_timepoint_expired(timeout));`
### Training Execution

The `gddr_training()` function [gddr.c 509-544](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L509-L544) coordinates training across all enabled GDDR instances:

1.   **Initialize Timeout:** 1000ms timeout per instance (`MRISC_INIT_TIMEOUT`)
2.   **Poll All Instances:** Check training status for each enabled GDDR
3.   **Handle Errors:** Log timeout or failure errors but continue checking others
4.   **Proceed to BIST:** If no training errors, execute hardware memory test

## Runtime Communication Protocol

### Message-Based Interface

After initialization, the CMFW communicates with MRISC firmware through a message-passing protocol using the `MRISC_MSG_REGISTER` and a message structure in L1 memory.

### Message Types

**MRISC Message Protocol Constants**

| Message Type | Value | Purpose | Function |
| --- | --- | --- | --- |
| `MRISC_MSG_TYPE_NONE` | 0 | No active message / completion signal | - |
| `MRISC_MSG_TYPE_PHY_POWERDOWN` | 1 | Put PHY in power-down state | `set_mrisc_power_setting(false)` |
| `MRISC_MSG_TYPE_PHY_WAKEUP` | 2 | Wake PHY from power-down | `set_mrisc_power_setting(true)` |
| `MRISC_MSG_TYPE_RUN_MEMTEST` | 8 | Execute memory test | `StartHwMemtest()` |

### Communication Functions

**Message Send/Receive Pattern**

The `mrisc_message()` helper function [gddr.c 546-574](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L546-L574) implements the generic message protocol:

1.   **Check Busy:** Verify `MRISC_MSG_REGISTER == 0` for each target instance
2.   **Send Command:** Write message type to `MRISC_MSG_REGISTER`
3.   **Wait Completion:** Poll until `MRISC_MSG_REGISTER` returns to 0
4.   **Timeout Handling:** Return `-ETIMEDOUT` if operation exceeds deadline

**Memory Access Functions**

Low-level register/memory access is provided through NOC2AXI wrapper functions:

| Function | Purpose | TLB Used |
| --- | --- | --- |
| `MriscL1Read32()`[gddr.c 81-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L81-L88) | Read from MRISC L1 memory | `MRISC_SETUP_TLB` (13) |
| `MriscL1Write32()`[gddr.c 90-97](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L90-L97) | Write to MRISC L1 memory | `MRISC_SETUP_TLB` (13) |
| `MriscRegRead32()`[gddr.c 99-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L99-L106) | Read MRISC control register | `MRISC_SETUP_TLB` (13) |
| `MriscRegWrite32()`[gddr.c 108-115](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L108-L115) | Write MRISC control register | `MRISC_SETUP_TLB` (13) |

All functions use NOC logical coordinates obtained via `GetGddrNocCoords()` to address the correct GDDR instance and NOC2AXI port.

## Power Management

### PHY Power States

The MRISC firmware supports power management of the GDDR PHY through power-down and wake-up commands. This is used during system power state transitions.

### Power Management API

**set_mrisc_power_setting() Function**

Located at [gddr.c 576-582](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L576-L582) this function broadcasts power state changes to all active MRISC instances:

`int32_t set_mrisc_power_setting(bool on) {    uint32_t op_code = on ? MRISC_MSG_TYPE_PHY_WAKEUP                            : MRISC_MSG_TYPE_PHY_POWERDOWN;        return mrisc_message(op_code, GetDramMask(),                          MRISC_POWER_SETTING_TIMEOUT_MS,                          "power_setting");}`
**Parameters:**

*   `on`: `true` to wake PHY, `false` to power down
*   Timeout: 1000ms (`MRISC_POWER_SETTING_TIMEOUT_MS`)
*   Target instances: Determined by `GetDramMask()` (accounts for harvesting and masking)

### GDDR Reset and Recovery

The CMFW provides a mechanism to reset individual GDDR instances and re-run training and BIST through the `toggle_gddr_reset()` message handler [gddr.c 610-678](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L610-L678):

**Reset Sequence:**

1.   Wake PHY if currently powered down
2.   Assert MRISC soft reset via `assert_mrisc_soft_reset()`[gddr.c 586-598](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L586-L598)
3.   Initialize status register and release reset
4.   Wait for training completion
5.   Execute BIST
6.   Restore previous power state

**Error Codes:**

| Error Code | Constant | Meaning |
| --- | --- | --- |
| 0 | Success | Operation completed successfully |
| Various | `GDDR_RESET_ERR_INVALID_INST` | Invalid GDDR instance number |
| Various | `GDDR_RESET_ERR_HARVESTED` | Instance is harvested |
| Various | `GDDR_RESET_ERR_MASKED` | Instance is masked in DRAM table |
| Various | `GDDR_RESET_ERR_POWERDOWN` | Power state change failed |
| Various | `GDDR_RESET_ERR_TRAINING` | Memory training failed |
| Various | `GDDR_RESET_ERR_BIST` | Memory test failed |

## Telemetry Interface

### Telemetry Table Structure

Each MRISC maintains a telemetry table in its L1 memory at `GDDR_TELEMETRY_TABLE_ADDR` (0x8000) that provides runtime status and version information.

**read_gddr_telemetry_table() Function**

The function [gddr.c 117-143](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L117-L143) reads the telemetry table from MRISC L1 memory:

`int read_gddr_telemetry_table(uint8_t gddr_inst,                                gddr_telemetry_table_t *gddr_telemetry) {    // 1. Setup TLB for MRISC L1 access    volatile uint8_t *mrisc_l1 = SetupMriscL1Tlb(gddr_inst);        // 2. DMA transfer if available, else word-by-word read    #ifdef CONFIG_DMA_ARC_HS        dma_arc_hs_transfer(arc_dma_dev, ...);    #endif        // 3. Verify version matches expectation    if (gddr_telemetry->telemetry_table_version !=         GDDR_TELEMETRY_TABLE_T_VERSION) {        return -ENOTSUP;    }        return 0;}`
### Version Information

The telemetry table includes firmware version fields used for compatibility checking:

*   `mrisc_fw_version_major` - Major version (e.g., 2)
*   `mrisc_fw_version_minor` - Minor version (e.g., 13)
*   `telemetry_table_version` - Table structure version (`GDDR_TELEMETRY_TABLE_T_VERSION`)

## Board-Specific Configuration

### DRAM Masking

The firmware supports selective enabling/disabling of GDDR instances through the firmware table's DRAM table:

**GetDramMask() Function**[gddr.c 200-209](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L200-L209)

`static uint32_t GetDramMask(void) {    uint32_t dram_mask = tile_enable.gddr_enabled;  // Harvesting mask        if (tt_bh_fwtable_get_fw_table(fwtable_dev)->has_dram_table &&        tt_bh_fwtable_get_fw_table(fwtable_dev)->dram_table.dram_mask_en) {        dram_mask &= tt_bh_fwtable_get_fw_table(fwtable_dev)                         ->dram_table.dram_mask;    }    return dram_mask;}`
This combines:

1.   **Harvesting mask** (`tile_enable.gddr_enabled`) - Physically disabled instances
2.   **Software mask** (`dram_table.dram_mask`) - Administratively disabled instances

### NOC Coordinate Mapping

Each GDDR instance has specific NOC coordinates that must be used for addressing. The MRISC firmware is accessed through the middle NOC2AXI port (port 0 of 3) on each GDDR controller.

**GDDR NOC Addressing Constants**

`#define MRISC_FW_NOC2AXI_PORT 0    // Middle port#define NUM_MRISC_NOC2AXI_PORT 3   // Total ports per GDDR#define ARC_NOC0_X 8               // ARC's NOC X coordinate#define ARC_NOC0_Y 0               // ARC's NOC Y coordinate#define MRISC_L1_ADDR (1ULL << 37) // L1 address bit#define MRISC_REG_ADDR (1ULL << 40) // Register address bit`
The `GetGddrNocCoords()` function (from `harvesting.h`) translates GDDR instance numbers to physical NOC (X,Y) coordinates accounting for harvesting.
