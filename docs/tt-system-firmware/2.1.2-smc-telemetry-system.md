## Purpose and Scope

The SMC Telemetry System provides real-time monitoring data about the Blackhole ASIC and its surrounding hardware to the host system. This system collects and exposes approximately 70 different metrics including temperatures, voltages, frequencies, power consumption, firmware versions, and hardware status information. The telemetry data is updated periodically (every 100ms by default) and made available through memory-mapped registers that the host can read via PCIe.

For information about how the host sends control commands to the SMC, see [SMC Message Handling](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.1-smc-message-handling). For information about DVFS (Dynamic Voltage and Frequency Scaling) which consumes telemetry data to make control decisions, see [DVFS and Throttling](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.3-dvfs-and-throttling).

## Telemetry Table Architecture

### Memory Layout

**Telemetry Table Memory Layout**

The SMC firmware maintains a `telemetry_table` structure that contains both metadata and actual telemetry data. The address of this table is published in scratch registers so the host can discover and read it.

Sources: [lib/tenstorrent/bh_arc/telemetry.c 68-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L68-L88)[lib/tenstorrent/bh_arc/telemetry.c 483-484](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L483-L484)

### Data Structures

The telemetry system uses three primary data structures:

**`struct telemetry_entry`** - Maps a TAG identifier to its offset in the telemetry data array:

*   `tag`: The TAG_* constant identifying this metric
*   `offset`: Index into the telemetry[] array where this metric's value is stored

**`struct telemetry_table`** - The main telemetry container:

*   `version`: Telemetry interface version (0x00000100 for v0.1.0)
*   `entry_count`: Number of valid telemetry entries (70)
*   `tag_table[TAG_COUNT]`: Array mapping TAG identifiers to offsets
*   `telemetry[TAG_COUNT]`: Array containing actual telemetry values

**`telemetry_table`** - Global instance initialized at [lib/tenstorrent/bh_arc/telemetry.c 106-173](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L106-L173)**`telemetry`** - Pointer to the telemetry data array at [lib/tenstorrent/bh_arc/telemetry.c 181](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L181-L181)

Sources: [lib/tenstorrent/bh_arc/telemetry.c 53-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L53-L88)

## TAG System

The telemetry system uses a TAG-based indexing scheme where each metric has a unique TAG identifier. There are 70 tags (TAG_COUNT = 70) organized into several categories.

### TAG Categories

| Category | TAG Range | Description |
| --- | --- | --- |
| **Identification** | 1-4, 61-62 | Board ID, ASIC ID (high/low parts) |
| **Configuration** | 5, 9-10, 34-37, 40, 53, 55-56, 59, 63-64 | Update speed, limits, enabled hardware |
| **Voltages & Power** | 6-8, 54 | VCore, TDP, TDC, input power |
| **Temperatures** | 11-13, 42-45, 48, 51 | ASIC, board, GDDR temperatures |
| **Frequencies** | 14-20 | AICLK, AXICLK, ARCCLK, L2CPUCLK |
| **Hardware Status** | 21-23, 38 | Ethernet, GDDR, PCIe status |
| **Firmware Versions** | 24-30 | Firmware versions for all subsystems |
| **Fan Control** | 31, 41 | Fan speed percentage and RPM |
| **Error Tracking** | 46-47, 49-50, 60 | GDDR errors, thermal trip count |
| **AICLK Arbitration** | 65-69 | AICLK arbiter status and limits |

### Key Telemetry Tags

**Static Tags** (set once at initialization):

*   `TAG_BOARD_ID_HIGH/LOW` (1-2): 64-bit board identifier
*   `TAG_ASIC_ID_HIGH/LOW` (61-62): 64-bit ASIC identifier from eFuses
*   `TAG_CM_FW_VERSION` (29): SMC firmware version
*   `TAG_FLASH_BUNDLE_VERSION` (28): Combined firmware bundle version
*   `TAG_ENABLED_TENSIX_COL` (34): Bitmask of enabled Tensix columns
*   `TAG_ENABLED_GDDR` (36): Bitmask of enabled GDDR controllers

**Dynamic Tags** (updated every 100ms):

*   `TAG_VCORE` (6): Core voltage in millivolts
*   `TAG_TDP` (7): Thermal Design Power in watts
*   `TAG_TDC` (8): Thermal Design Current in amperes
*   `TAG_ASIC_TEMPERATURE` (11): ASIC temperature in signed 16.16 fixed-point format
*   `TAG_AICLK` (14): AI clock frequency in MHz
*   `TAG_GDDR_STATUS` (22): GDDR training/BIST status (see bit layout below)
*   `TAG_FAN_SPEED` (31): Fan speed as percentage (0-100)
*   `TAG_TIMER_HEARTBEAT` (32): Incremented on each update
*   `TAG_INPUT_POWER` (54): Total input power in watts

**GDDR Status Bit Layout** (TAG_GDDR_STATUS):

```
Bits [15:0]  - Training status (2 bits per GDDR instance)
  [0] = GDDR 0 training complete
  [1] = GDDR 0 training error
  [2] = GDDR 1 training complete
  [3] = GDDR 1 training error
  ... (continues for GDDR 2-7)
  
Bits [31:16] - BIST status (2 bits per GDDR instance)
  [16] = GDDR 0 BIST complete
  [17] = GDDR 0 BIST failed
  [18] = GDDR 1 BIST complete
  [19] = GDDR 1 BIST failed
  ... (continues for GDDR 2-7)
```

### Periodic Update Mechanism

The telemetry system uses Zephyr's timer and work queue infrastructure to perform periodic updates:

**Timer Infrastructure**:

*   `K_TIMER_DEFINE(telem_update_timer, telemetry_timer_handler, NULL)` at [lib/tenstorrent/bh_arc/telemetry.c 474](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L474-L474)

**Update Flow**:

1.   Timer expires every 100ms
2.   `telemetry_timer_handler()` submits work item to system work queue
3.   `telemetry_work_handler()` calls `update_telemetry()`
4.   `update_telemetry()` collects fresh data from all subsystems

### GDDR Telemetry Update

The GDDR subsystem provides detailed telemetry that is aggregated from 8 GDDR controller instances:

**GDDR Telemetry Collection Flow**

The GDDR telemetry packing is space-efficient, combining data from pairs of GDDR controllers:

*   Temperature tags: Two controllers per tag (8 temperature values per tag)
*   Error tags: Two controllers per tag (4 error counts per tag)
*   Status: All controllers in single TAG_GDDR_STATUS (32 bits total)

Sources: [lib/tenstorrent/bh_arc/telemetry.c 219-289](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L219-L289)

## Host Interface

### Reading Telemetry from Host

The host system (via pyluwen or other tools) reads telemetry through a multi-step process:

**Host Telemetry Read Flow**

Example from test code at [app/smc/pytest/e2e_smoke.py 108-116](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L108-L116):

## Data Format Conversions

### Fixed-Point Representation

Some telemetry values (particularly temperatures) use a signed 16.16 fixed-point format to represent floating-point values in 32-bit integers:

```
| Sign bit | Integer part (15 bits) | Fractional part (16 bits) |
|    1     |         15             |           16              |
```

**Conversion Functions**:

`ConvertFloatToTelemetry()` at [lib/tenstorrent/bh_arc/telemetry.c 189-207](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L189-L207):

*   Handles special value `FLT_MAX` → `0x80000000` (error condition)
*   Splits float into integer and fractional parts
*   Packs into 32-bit value with sign preservation

`ConvertTelemetryToFloat()` at [lib/tenstorrent/bh_arc/telemetry.c 209-217](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L209-L217):

*   Detects error condition `INT32_MIN` → `FLT_MAX`
*   Divides by 65536.0 to restore floating-point value

**Python-side conversion** at [app/smc/pytest/e2e_smoke.py 119-126](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L119-L126):

`def convert_telemetry_to_float(value):    INT32_MIN = -2147483648        if value == INT32_MIN:        return sys.float_info.max    else:        return value / 65536.0`
### Integer Value Tags

Most telemetry tags use direct integer representation:

*   Voltages: millivolts (uint32_t)
*   Power: watts (uint32_t)
*   Current: amperes (uint32_t)
*   Frequencies: MHz (uint32_t)
*   Percentages: 0-100 (uint32_t)
*   Status bits: packed bit fields (uint32_t)

### Static vs Dynamic Telemetry

**Static Telemetry** (set once during initialization):

*   Board and ASIC identifiers
*   Firmware versions (SMC, DMC, GDDR, Ethernet, bundle)
*   Hardware configuration (enabled Tensix, GDDR, Ethernet, L2CPU)
*   Limit values (FMAX, VDD limits, thermal limits, TDP/TDC limits)
*   Features (NOC translation, PCIe usage)

**Dynamic Telemetry** (updated every 100ms):

*   Real-time measurements (voltage, current, power, temperature)
*   Clock frequencies (AICLK, AXICLK, ARCCLK, L2CPUCLK)
*   AICLK arbiter states and limits
*   GDDR training/BIST status, temperatures, error counts
*   Fan speed and RPM
*   Heartbeat counter

### AICLK Arbiter Telemetry

The telemetry system exposes detailed information about the AICLK (AI Clock) frequency arbitration system:

**`TAG_AICLK_ARB_MIN` (65)**: Effective minimum AICLK frequency

*   Bits [15:0]: Frequency in MHz
*   Bits [31:16]: Which min arbiter is effective (from `enum aiclk_arb_min`)

**`TAG_AICLK_ARB_MAX` (66)**: Effective maximum AICLK frequency

*   Bits [15:0]: Frequency in MHz
*   Bits [31:16]: Which max arbiter is limiting (from `enum aiclk_arb_max`)

**`TAG_ENABLED_MIN_ARB` (67)**: Bitmask of enabled minimum arbiters

*   Each bit represents whether a min arbiter is enabled
*   Bit positions match `enum aiclk_arb_min` values

**`TAG_ENABLED_MAX_ARB` (68)**: Bitmask of enabled maximum arbiters

*   Each bit represents whether a max arbiter is enabled
*   Bit positions match `enum aiclk_arb_max` values (TDP, TDC, thermal, etc.)

**`TAG_AICLK_PPM_INFO` (69)**: Target AICLK frequency decision information

*   Contains why a particular target frequency was chosen
*   Includes which arbiter determined the frequency
*   Encodes the reason (min_arb, max_arb, fmin, sweep, forced)

These tags allow the host to understand exactly why the AICLK is at its current value and which limits are active.

Sources: [lib/tenstorrent/bh_arc/telemetry.c 400-410](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L400-L410)[lib/tenstorrent/bh_arc/aiclk_ppm.h 12-73](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.h#L12-L73)

### Update Interval Configuration

The telemetry update interval is configurable via the `telem_update_interval` variable (default 100ms). The actual interval is exposed to the host via `TAG_UPDATE_TELEM_SPEED` (tag 5).

The update mechanism uses Zephyr's timer API with both duration and period set to `telem_update_interval`:

`k_timer_start(&telem_update_timer, K_MSEC(telem_update_interval),              K_MSEC(telem_update_interval));`
Sources: [lib/tenstorrent/bh_arc/telemetry.c 187](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L187-L187)[lib/tenstorrent/bh_arc/telemetry.c 332](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L332-L332)[lib/tenstorrent/bh_arc/telemetry.c 494-495](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L494-L495)

### Post Code Integration

The telemetry update process sets post codes to track its execution:

*   `POST_CODE_TELEMETRY_START` at the beginning of `update_telemetry()`
*   `POST_CODE_TELEMETRY_END` at the end

This allows debugging of telemetry update timing and hangs.
