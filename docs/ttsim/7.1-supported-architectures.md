# Supported Architectures

Relevant source files
*   [README.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1)

This page documents the Tenstorrent chip architectures that ttsim supports, including the binary files, SOC descriptor files, and architecture-specific considerations. For general configuration information including environment variables, see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration). For feature implementation status across architectures, see [Feature Status](https://deepwiki.com/tenstorrent/ttsim/7.3-feature-status).

## Overview

ttsim provides simulation support for two Tenstorrent chip architectures:

| Architecture | Binary File | SOC Descriptor File | Status |
| --- | --- | --- | --- |
| **Wormhole** | `libttsim_wh.so` | `wormhole_b0_80_arch.yaml` | More mature, recommended for initial development |
| **Blackhole** | `libttsim_bh.so` | `blackhole_140_arch.yaml` | Less mature, under active development |

Each architecture requires a separate simulator binary and a corresponding SOC descriptor YAML file. The simulator binary implements the architecture-specific hardware behavior, while the SOC descriptor defines the chip topology and configuration.

## Wormhole Architecture

### Binary Details

*   **File:**`libttsim_wh.so`
*   **Download:** Available from [releases page](https://github.com/tenstorrent/ttsim/blob/67b1733e/releases%20page)
*   **Target chip:** Wormhole B0 with 80-core configuration

### SOC Descriptor

*   **Source file:**`$TT_METAL_HOME/tt_metal/soc_descriptors/wormhole_b0_80_arch.yaml`
*   **Installation name:** Must be copied and renamed to `soc_descriptor.yaml` in the same directory as `libttsim_wh.so`

### Configuration Example

`# Set simulator locationexport TT_METAL_SIMULATOR=~/sim/libttsim_wh.so # Copy SOC descriptorcp $TT_METAL_HOME/tt_metal/soc_descriptors/wormhole_b0_80_arch.yaml ~/sim/soc_descriptor.yaml # Run example (slow dispatch mode required)cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 ./build/programming_examples/metal_example_add_2_integers_in_riscv`
### Maturity Status

Wormhole is the most mature architecture in ttsim. It can run many tt-metal and ttnn examples/tests in slow dispatch mode, as well as numerous LLK tests. Wormhole is recommended for initial development and testing.

**Sources:**[README.md 45-52](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L45-L52)[README.md 17-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L17-L19)

## SOC Descriptor Files

**Diagram: SOC Descriptor File Flow**

The SOC descriptor file defines the chip topology, including core counts, memory configurations, and interconnect details. The simulator binary reads this file at runtime to configure the simulated hardware.

### Requirements

1.   **Co-location:** The SOC descriptor file must be in the same directory as the simulator binary
2.   **File name:** The file must be named exactly `soc_descriptor.yaml`
3.   **Source:** Copy the architecture-specific descriptor from `$TT_METAL_HOME/tt_metal/soc_descriptors/`

### File Location

The simulator locates the SOC descriptor using this algorithm:

1.   Determine the directory containing the `libttsim.so` file pointed to by `TT_METAL_SIMULATOR`
2.   Look for `soc_descriptor.yaml` in that directory
3.   Fail with `ConfigurationError` if the file is not found

**Sources:**[README.md 42-43](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L42-L43)[README.md 48](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L48-L48)[README.md 57](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L57-L57)

## Architecture Selection

**Diagram: Architecture Selection Flow**

The architecture is selected implicitly by setting `TT_METAL_SIMULATOR` to point to the appropriate binary. TT-Metalium loads the simulator dynamically and communicates with it through the ttsim API.

### Selection Process

1.   Set `TT_METAL_SIMULATOR` to the full path of the desired simulator binary
2.   Ensure the corresponding SOC descriptor is named `soc_descriptor.yaml` in the same directory
3.   Launch TT-Metalium workload with `TT_METAL_SLOW_DISPATCH_MODE=1`

### Common Errors

| Error | Cause | Solution |
| --- | --- | --- |
| `ConfigurationError: SOC descriptor not found` | Missing or misnamed SOC descriptor | Ensure `soc_descriptor.yaml` exists in same directory as binary |
| `ConfigurationError: Cannot load simulator` | Wrong binary for host architecture | Download Linux/x86_64 binary |
| Wrong architecture behavior | SOC descriptor mismatch | Verify SOC descriptor matches simulator binary (wormhole vs blackhole) |

**Sources:**[README.md 41-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L41-L61)

## Maturity and Capability Comparison

| Capability | Wormhole | Blackhole |
| --- | --- | --- |
| **Overall maturity** | More mature | Less mature |
| **tt-metal examples** | Many supported | Many supported |
| **ttnn examples/tests** | Many supported | Many supported |
| **LLK tests** | Numerous passing | Numerous passing |
| **Dispatch mode** | Slow dispatch only | Slow dispatch only |
| **Bit-exact operations** | Unpacker, SFPU, Packer, some FPU | Same as Wormhole |
| **Recommended for** | Initial development, stable testing | Advanced testing, future hardware preparation |

Both architectures share the same fundamental limitations and capabilities, including:

*   Fast dispatch not yet supported (must use `TT_METAL_SLOW_DISPATCH_MODE=1`)
*   Same numerical accuracy guarantees (see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy))
*   Same error reporting categories (see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference))
*   Same bit-exact implementation for Unpacker, SFPU, and Packer
*   Same FPU implementation status (MOV*/GMPOOL bit-exact, ELW*/MVMUL/GAPOOL not yet bit-exact)

The primary difference is maturity: Wormhole has been tested more extensively and is more likely to work correctly for a given workload.

## Installation Layout Example

A typical installation directory structure for both architectures:

```
~/sim/
├── libttsim_wh.so              # Wormhole simulator
├── wormhole_soc_descriptor.yaml # Wormhole descriptor (not used, just for reference)
├── libttsim_bh.so              # Blackhole simulator  
├── blackhole_soc_descriptor.yaml # Blackhole descriptor (not used, just for reference)
└── soc_descriptor.yaml         # Active descriptor (copy of either wh or bh)
```

To switch architectures, update both `TT_METAL_SIMULATOR` and `soc_descriptor.yaml`:

`# Switch to Wormholeexport TT_METAL_SIMULATOR=~/sim/libttsim_wh.socp ~/sim/wormhole_soc_descriptor.yaml ~/sim/soc_descriptor.yaml # Switch to Blackholeexport TT_METAL_SIMULATOR=~/sim/libttsim_bh.socp ~/sim/blackhole_soc_descriptor.yaml ~/sim/soc_descriptor.yaml`
**Note:** Only one architecture can be active at a time per installation directory because both use the same `soc_descriptor.yaml` filename. To run both simultaneously, use separate installation directories with separate `TT_METAL_SIMULATOR` settings.
