## Purpose and Scope

This guide explains how to extend the instruction catalog system to support new RISC-V instructions and extensions. The instruction catalog provides a centralized, queryable database of all RISC-V instructions supported by the coretp framework. This system enables test scenarios to filter and select appropriate instructions based on ISA configuration, extension membership, operand structure, and instruction category.

For information about using instructions in test scenarios, see [Writing Test Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/6.2-writing-test-scenarios). For details about the TestStep primitives that use instructions, see [Test Step System](https://deepwiki.com/tenstorrent/riscv-coretp/3.3-test-step-system).

**Sources:**[coretp/isa/catalog.py 1-125](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L1-L125)

## Instruction Catalog Architecture

The instruction catalog system consists of three main layers:

**Diagram: Instruction Catalog System Layers**

The **Definition Layer** creates frozen templates (`InstructionDef`) with operand specifications (`OperandSlot`). The **Catalog Layer** aggregates all definitions into `ALL_INSTRS` and provides `InstructionCatalog` for ISA-aware filtering, with `IMPLIED_EXTENSIONS` resolving extension hierarchies. The **Runtime Layer** produces mutable `Instruction` instances from templates for use in test code.

**Sources:**[coretp/isa/catalog.py 37-125](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L37-L125)[coretp/isa/instructions/crypto.py 1-1340](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L1-L1340)

## Defining New Instructions

### InstructionDef Structure

Each instruction is defined as an `InstructionDef` object with the following properties:

| Property | Type | Description |
| --- | --- | --- |
| `name` | `str` | Instruction mnemonic (e.g., "andn", "sha256sig0") |
| `extension` | `Extension` | Extension flag(s) this instruction belongs to |
| `xlen` | `Xlen` | Register width compatibility (XLEN32, XLEN64, or both) |
| `category` | `Category` | Instruction category (LOGIC, ARITHMETIC, etc.) |
| `destination` | `OperandSlot` or `None` | Destination operand specification |
| `source` | `list[OperandSlot]` | List of source operand specifications |
| `formatter` | `str` | Assembly format string with operand placeholders |

**Sources:**[coretp/isa/catalog.py 9-10](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L9-L10)[coretp/isa/instructions/crypto.py 9-20](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L9-L20)

### OperandType Enum

Operand types define the kind of registers or values an instruction accepts:

| OperandType | Description |
| --- | --- |
| `NONE` | No operand (empty) |
| `GPR` | General Purpose Register (x0-x31) |
| `FPR` | Floating Point Register (f0-f31) |
| `VEC` | Vector Register (v0-v31) |
| `CSR` | Control and Status Register |
| `IMM` | Immediate value |
| `SYMBOL` | Symbol for pseudoinstructions (la, li) |

**Sources:**[coretp/rv_enums/instruction.py 7-25](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/rv_enums/instruction.py#L7-L25)

### Category Flags

The `Category` enum uses flag-based membership, allowing instructions to belong to multiple categories:

| Category | Description |
| --- | --- |
| `ARITHMETIC` | Arithmetic operations (add, mul, clmul) |
| `LOGIC` | Logical operations (and, or, xor, andn) |
| `SHIFT` | Shift operations (sll, srl, rol, ror) |
| `ATOMIC` | Atomic memory operations |
| `CONTROL` | Control flow (branches, jumps) |
| `LOAD` | Memory load operations |
| `STORE` | Memory store operations |
| `ENCRYPTION` | Cryptographic operations (AES, SHA) |
| `VECTOR` | Vector operations |
| `CACHE_OPERATION` | Cache management operations |
| `SYSTEM` | System instructions |
| `PSEUDO` | Pseudoinstructions |

**Sources:**[coretp/rv_enums/instruction.py 27-58](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/rv_enums/instruction.py#L27-L58)

## Creating Instruction Definitions

### Basic Scalar Instruction

Example of defining a scalar bitwise instruction:

**Diagram: InstructionDef Component Structure**

Reference implementation from cryptographic extension:

`andn = InstructionDef(    name="andn",    extension=Extension.ZBKB,    xlen=Xlen.XLEN32,    category=Category.LOGIC,    destination=OperandSlot(name="rd", type=OperandType.GPR),    source=[        OperandSlot(name="rs1", type=OperandType.GPR),        OperandSlot(name="rs2", type=OperandType.GPR),    ],    formatter="andn {rd}, {rs1}, {rs2}",)`
**Sources:**[coretp/isa/instructions/crypto.py 9-20](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L9-L20)

### Instructions with Immediate Operands

For instructions with immediate values, use `OperandType.IMM`:

`rori = InstructionDef(    name="rori",    extension=Extension.ZBKB,    xlen=Xlen.XLEN64,    category=Category.LOGIC,    destination=OperandSlot(name="rd", type=OperandType.GPR),    source=[        OperandSlot(name="rs1", type=OperandType.GPR),        OperandSlot(name="shamtd", type=OperandType.IMM),    ],    formatter="rori {rd}, {rs1}, {shamtd}",)`
**Sources:**[coretp/isa/instructions/crypto.py 1031-1042](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L1031-L1042)

### Vector Instructions

Vector instructions use `OperandType.VEC` for vector register operands:

`vaesdf_vs = InstructionDef(    name="vaesdf.vs",    extension=Extension.ZVKNED | Extension.ZVKN,    xlen=Xlen.XLEN32,    category=Category.ENCRYPTION,    destination=OperandSlot("vd", OperandType.VEC),    source=[        OperandSlot("vs2", OperandType.VEC),    ],    formatter="vaesdf.vs {vd}, {vs2}",)`
**Sources:**[coretp/isa/instructions/crypto.py 256-266](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L256-L266)

### Multiple Extension Membership

Instructions can belong to multiple extensions using the bitwise OR operator (`|`):

`pack = InstructionDef(    name="pack",    extension=Extension.ZKS | Extension.ZKN | Extension.ZK | Extension.ZBKB,    xlen=Xlen.XLEN32,    category=Category.LOGIC,    destination=OperandSlot("rd", OperandType.GPR),    source=[        OperandSlot("rs1", OperandType.GPR),        OperandSlot("rs2", OperandType.GPR),    ],    formatter="pack {rd}, {rs1}, {rs2}",)`
This instruction will be included in the catalog if **any** of the specified extensions (ZKS, ZKN, ZK, or ZBKB) are present in the target ISA.

**Sources:**[coretp/isa/instructions/crypto.py 92-103](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L92-L103)

## Extension Hierarchy and Implied Extensions

### IMPLIED_EXTENSIONS Dictionary

The `IMPLIED_EXTENSIONS` dictionary defines hierarchical relationships between extensions. When a parent extension is enabled, all implied extensions are automatically included:

**Diagram: Extension Hierarchy via IMPLIED_EXTENSIONS**

The dictionary definition:

`IMPLIED_EXTENSIONS = {    Extension.G: Extension.I | Extension.M | Extension.A | Extension.F | Extension.D | Extension.ZICSR | Extension.ZIFENCEI,    Extension.ZKT: Extension.ZKR | Extension.ZKN | Extension.ZKS,    Extension.ZK: Extension.ZKR | Extension.ZKN | Extension.ZKS,    Extension.ZKN: Extension.ZBKB | Extension.ZBKC | Extension.ZBKX | Extension.ZKNE | Extension.ZKND | Extension.ZKNH,    Extension.ZKS: Extension.ZBKB | Extension.ZBKC | Extension.ZBKX | Extension.ZKSED | Extension.ZKSH,    Extension.ZVE32F: Extension.ZVE32X,    Extension.ZVE64F: Extension.ZVE64D | Extension.ZVE64X,    Extension.ZBC: Extension.ZBKC,    Extension.ZBB: Extension.ZBKB,}`
**Sources:**[coretp/isa/catalog.py 12-23](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L12-L23)

### Extension Expansion Algorithm

The `expand_implied_extensions` function recursively resolves all implied extensions:

`def expand_implied_extensions(mask: Extension) -> Extension:    changed = True    while changed:        changed = False        for k, v in IMPLIED_EXTENSIONS.items():            if mask & k and not mask & v:                mask |= v                changed = True    return mask`
This iterative algorithm ensures transitive closure: if Extension A implies Extension B, and Extension B implies Extension C, then Extension A will ultimately include Extension C.

**Sources:**[coretp/isa/catalog.py 26-34](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L26-L34)

## Instruction Catalog Filtering

### InstructionCatalog Initialization

The `InstructionCatalog` class filters `ALL_INSTRS` based on the target ISA configuration:

**Diagram: InstructionCatalog Initialization Flow**

The filtering predicate checks:

1.   **XLEN compatibility**: Instruction's xlen must be compatible with ISA's xlen
2.   **Extension presence**: All required extensions must be present in the ISA

`def matching_instr(i: InstructionDef) -> bool:    return i.xlen.compatible_with(self.isa.xlen) and (i.extension & ~self.isa.extensions) == Extension(0)`
**Sources:**[coretp/isa/catalog.py 42-61](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L42-L61)

### Filter Method Parameters

The `filter()` method provides comprehensive instruction querying:

| Parameter | Type | Description |
| --- | --- | --- |
| `extension` | `Extension` | Filter by specific extension(s) |
| `category` | `Category` | Filter by instruction category |
| `has_destination` | `bool` | Whether instruction has destination operand |
| `source_reg_count` | `int` | Minimum number of source register operands |
| `has_immediate` | `bool` | Whether instruction has immediate operand |
| `destination_type` | `OperandType` | Required destination operand type (e.g., FPR) |
| `source_type` | `OperandType` | Required source operand type (e.g., VEC) |
| `exclude_extensions` | `Extension` | Extensions to exclude from results |
| `xlen` | `Xlen` | Filter by specific xlen |

**Sources:**[coretp/isa/catalog.py 68-116](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L68-L116)

### Filter Usage Examples

**Example 1: Find all floating-point arithmetic instructions**

`catalog.filter(    category=Category.ARITHMETIC,    destination_type=OperandType.FPR)`
**Example 2: Find vector encryption instructions with at least 2 source operands**

`catalog.filter(    extension=Extension.ZVKNED,    category=Category.ENCRYPTION,    source_reg_count=2)`
**Example 3: Find instructions from cryptographic extensions excluding specific ones**

`catalog.filter(    extension=Extension.ZKN,    exclude_extensions=Extension.ZBKB)`
**Sources:**[coretp/isa/catalog.py 68-116](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L68-L116)

## Adding Instructions to the Catalog

### Step-by-Step Process

**Diagram: Process for Adding New Instructions**

### Creating a New Instruction Module

1.   **Create instruction definition file**: `coretp/isa/instructions/your_extension.py`

2.   **Define instructions**:

`from coretp.rv_enums import Extension, Xlen, Categoryfrom coretp.isa.instruction import InstructionDeffrom coretp.isa.operands import OperandSlotfrom coretp.rv_enums import OperandType custom_instr1 = InstructionDef(    name="custom.op1",    extension=Extension.YOUR_EXTENSION,    xlen=Xlen.XLEN32,    category=Category.ARITHMETIC,    destination=OperandSlot("rd", OperandType.GPR),    source=[        OperandSlot("rs1", OperandType.GPR),        OperandSlot("rs2", OperandType.GPR),    ],    formatter="custom.op1 {rd}, {rs1}, {rs2}",) custom_instr2 = InstructionDef(    name="custom.op2",    extension=Extension.YOUR_EXTENSION,    xlen=Xlen.XLEN32,    category=Category.LOGIC,    destination=OperandSlot("rd", OperandType.GPR),    source=[        OperandSlot("rs1", OperandType.GPR),    ],    formatter="custom.op2 {rd}, {rs1}",) # Export all instructionsyour_extension_instrs = [    custom_instr1,    custom_instr2,]`
1.   **Add to ALL_INSTRS**: Update `coretp/isa/instructions/__init__.py` to include the new module:

`from .your_extension import your_extension_instrs ALL_INSTRS = [    # ... existing instruction lists ...    *your_extension_instrs,]`
**Sources:**[coretp/isa/instructions/crypto.py 1249-1339](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L1249-L1339)

## Formatter Syntax

### Placeholder Format

The `formatter` string uses curly-brace placeholders matching operand names:

| Pattern | Description | Example |
| --- | --- | --- |
| `{rd}` | Destination register | `add {rd}, {rs1}, {rs2}` → `add x1, x2, x3` |
| `{rs1}`, `{rs2}` | Source registers | `andn {rd}, {rs1}, {rs2}` → `andn x5, x6, x7` |
| `{vd}`, `{vs1}` | Vector registers | `vadd.vv {vd}, {vs2}, {vs1}` → `vadd.vv v0, v1, v2` |
| `{imm}`, `{zimm5}` | Immediate values | `addi {rd}, {rs1}, {imm}` → `addi x1, x2, 100` |

### Complex Formatter Examples

**Single-operand instruction:**

`formatter="rev8 {rd}, {rs1}"`
**Three-operand with immediate:**

`formatter="sm4ed {rd}, {rs1}, {rs2}, {bs}"`
**Vector instruction with mask:**

`formatter="vandn.vv {vd}, {vs2}, {vs1}"`
**Sources:**[coretp/isa/instructions/crypto.py 19](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L19-L19)[coretp/isa/instructions/crypto.py 237](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L237-L237)[coretp/isa/instructions/crypto.py 412](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L412-L412)

## Best Practices

### Naming Conventions

1.   **Instruction names**: Use lowercase with dots for prefixes (e.g., `cbo.flush`, `vaes.vs`)
2.   **Operand names**: Follow RISC-V convention (`rd`, `rs1`, `rs2`, `vd`, `vs1`, etc.)
3.   **Module naming**: Group related instructions by extension family (e.g., `crypto.py`, `vector.py`)

### Extension Assignment

1.   **Single extension**: Assign the most specific extension

`extension=Extension.ZBKB`
2.   **Multiple extensions**: Use OR operator for instructions in multiple specs

`extension=Extension.ZKS | Extension.ZKN | Extension.ZK`
3.   **Composite extensions**: Don't manually add implied extensions (handled automatically)

`# GOODextension=Extension.ZKN # BAD - implied extensions added automaticallyextension=Extension.ZKN | Extension.ZBKB | Extension.ZBKC`

### Category Selection

1.   **Primary category**: Choose the most specific category

2.   **Multiple categories**: Use OR operator if instruction truly spans categories

`category=Category.ARITHMETIC | Category.ATOMIC`
3.   **Encryption instructions**: Always include `Category.ENCRYPTION` for cryptographic operations

### XLEN Specification

1.   **32-bit only**: `xlen=Xlen.XLEN32`
2.   **64-bit only**: `xlen=Xlen.XLEN64`
3.   **Both**: `xlen=Xlen.XLEN32 | Xlen.XLEN64` (for RV32/RV64 compatible instructions)

**Sources:**[coretp/isa/instructions/crypto.py 9-1340](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/crypto.py#L9-L1340)

## Testing New Instructions

### Verification Checklist

1.   **Catalog inclusion**: Verify instruction appears in `InstructionCatalog` for appropriate ISA

`catalog = InstructionCatalog("rv64gc_zk")assert "custom.op1" in [i.name for i in catalog]`
2.   **Extension filtering**: Test that extension filters work correctly

`instrs = catalog.filter(extension=Extension.YOUR_EXTENSION)assert len(instrs) > 0`
3.   **Operand type filtering**: Verify operand-based queries

`instrs = catalog.filter(    destination_type=OperandType.GPR,    source_reg_count=2)`
4.   **Formatter output**: Test that `Instruction.format()` produces valid assembly

`instr = catalog.get_instruction("custom.op1")instr.operands = ["x1", "x2", "x3"]assert instr.format() == "custom.op1 x1, x2, x3"`
5.   **XLEN compatibility**: Verify instruction availability for target register widths

`catalog_32 = InstructionCatalog("rv32gc")catalog_64 = InstructionCatalog("rv64gc")# Verify instruction appears in appropriate catalog(s)`
