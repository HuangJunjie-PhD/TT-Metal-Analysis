## Purpose and Scope

The AOT (Ahead-of-Time) Executor is TVM's execution runtime designed for embedded and resource-constrained devices. It generates fully standalone C code with static memory allocation and minimal runtime dependencies, making it suitable for microcontrollers and embedded systems where dynamic memory allocation is undesirable or unavailable.

This document covers the AOT executor's architecture, compilation process, and key features. For information about the Graph Executor for static execution on server/mobile devices, see [7.1](https://deepwiki.com/tenstorrent/tt-tvm/7.1-graph-executor). For the Virtual Machine executor with dynamic shape support, see [7.2](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor). For the minimal C Runtime used with AOT, see [7.4](https://deepwiki.com/tenstorrent/tt-tvm/7.4-c-runtime-(crt)).

## Architecture Overview

The AOT executor is part of TVM's runtime layer, providing one of three execution models for compiled neural networks. Unlike the Graph Executor and VM which require runtime support for dynamic memory allocation and operator dispatch, the AOT executor pre-compiles everything into static C code.

## Compilation Process

### Packed Call Legalization

A critical step in AOT compilation is legalizing packed calls. The Graph and VM executors use `tvm_call_cpacked` which expects arguments in a packed format using the `TVMValue` structure. For AOT, these calls are transformed to explicitly construct the packed arguments.

The `LegalizePackedCalls` pass performs the following transformations:

1.   **Identifies packed calls**: Searches for `tvm_call_cpacked` builtin calls
2.   **Looks up function signature**: Retrieves the `PrimFunc` from the IRModule to determine buffer parameters
3.   **Wraps non-input arguments**: For each non-input argument, wraps it with `tvm_stack_make_array` that includes: 
    *   Data pointer
    *   Shape information via `tvm_stack_make_shape`
    *   Strides (null for compact arrays)
    *   Number of dimensions (`ndim`)
    *   Data type
    *   Element offset

4.   **Preserves input arguments**: Input parameters to the function are passed through unchanged
5.   **Maintains device context**: Device context parameter is passed at the end

Sources: [src/tir/transforms/legalize_packed_calls.cc 44-118](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/transforms/legalize_packed_calls.cc#L44-L118)[tests/python/unittest/test_aot_legalize_packed_call.py 28-111](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/python/unittest/test_aot_legalize_packed_call.py#L28-L111)

## Memory Management

AOT executor uses **static memory planning** to eliminate runtime allocation:

### Static Memory Allocation Strategy

Key benefits:

*   **Predictable memory usage**: All memory requirements known at compile time
*   **No heap fragmentation**: Uses stack or static arrays
*   **Faster execution**: No allocation overhead during inference
*   **Embedded-friendly**: Works without dynamic memory allocator

## Comparison with Other Executors

### Execution Model Differences

### Use Case Matrix

| Scenario | Recommended Executor | Reason |
| --- | --- | --- |
| Microcontroller (< 256KB RAM) | **AOT** | Minimal runtime, predictable memory |
| Mobile/Embedded Linux | Graph | Good balance of size and flexibility |
| Server deployment | Graph or VM | Full runtime available |
| Dynamic batch sizes | **VM** | Runtime shape handling |
| Research/Experimentation | **VM** | Easiest to debug |
| Production embedded (static shapes) | **AOT** | Maximum performance, minimal overhead |

## Test Example

The test suite demonstrates the legalization transformation:

### Before Transformation

The original TIR contains a raw packed call:

```
T.evaluate(
    T.tvm_call_cpacked(
        "tvm_test_cpacked",
        A,  # Raw handle
        B,  # Raw handle  
        C,  # Raw handle
        device_context,
        dtype="int32",
    )
)
```

### After Transformation

Each handle is wrapped in a `tvm_stack_make_array` call:

```
T.evaluate(
    T.tvm_call_cpacked(
        "tvm_test_cpacked",
        T.tvm_stack_make_array(
            A,
            T.tvm_stack_make_shape(1, dtype="handle"),
            T.reinterpret(T.uint64(0), dtype="handle"),
            T.uint32(1),
            T.Cast("float32", 0),
            0,
            dtype="handle",
        ),
        # Similar wrapping for B and C...
        device_context,
        dtype="int32",
    )
)
```

## Limitations

The AOT executor has specific constraints:

1.   **Static shapes only**: All tensor shapes must be known at compile time
2.   **No control flow**: Conditional execution not supported (must be unrolled at compile time)
3.   **Single model per executable**: Cannot load multiple models dynamically
4.   **Fixed batch size**: Batch dimension must be constant
5.   **No runtime optimization**: All decisions made at compile time

These limitations are by design to enable the minimal runtime footprint required for embedded systems.
