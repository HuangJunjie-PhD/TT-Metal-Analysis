## Purpose and Scope

This page documents the ReLU (Rectified Linear Unit) configuration system in the packer hardware, including the `relu_config_t` structure, configuration encoding formats, and the functions used to program ReLU thresholding behavior during pack operations. ReLU is applied as data is packed from destination registers to L1 memory.

For general packer configuration, see [Core Packer Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/6.1-core-packer-operations-and-configuration). For destination register management, see [Destination Register Management](https://deepwiki.com/tenstorrent/tt-llk/6.2-destination-register-management).

* * *

## ReLU Types and Operation Modes

The packer supports four ReLU operation modes that control how values are processed during the pack operation:

| ReLU Type | Numeric Value | Behavior |
| --- | --- | --- |
| `NO_RELU` | 0 | Pass-through mode; no modification of values |
| `ZERO_RELU` | 1 | Standard ReLU: `max(0, x)` |
| `MIN_THRESHOLD_RELU` | 2 | Minimum threshold: `max(threshold, x)` |
| `MAX_THRESHOLD_RELU` | 3 | Maximum threshold: `min(threshold, x)` |

The ReLU operation is performed by the hardware as data is read from destination registers and written to L1 memory. This allows efficient application of activation functions without additional compute cycles.

## Configuration Structure: `relu_config_t`

### Structure Layout

The `relu_config_t` structure defines the hardware register format for configuring ReLU behavior:

### Field Definitions

*   **`STACC_RELU_ApplyRelu`** (4 bits): Specifies the ReLU operation mode (0-3 as per table above)
*   **`STACC_RELU_ReluThreshold`** (16 bits): Threshold value encoded in the source data format
*   **Zero Flag Control** (2 bits): Controls whether zero flags are disabled for source and destination
*   **RISC Backpressure** (10 bits): Controls backpressure for various RISC cores (not typically used)

## Configuration Functions

### `_llk_pack_relu_config_()`

Dynamically reconfigures ReLU settings without reinitializing the packer:

**Diagram: Configuration Process**

**Implementation Details:**

1.   Extracts mode from lower 4 bits: `(config & 0xf)`
2.   Maps mode value to `ReluType` enum
3.   Extracts threshold from upper 16 bits: `(config >> 16)`
4.   Combines into hardware register format: 
    *   `val = (threshold << STACC_RELU_ReluThreshold_SHAMT) | (mode << STACC_RELU_ApplyRelu_SHAMT)`

5.   Loads `val` into GPR register `p_gpr_pack::TMP0` (split into LO_16/HI_16)
6.   Stalls until packer is idle: `TTI_STALLWAIT(p_stall::STALL_CFG, p_stall::PACK)`
7.   Writes configuration: `TTI_WRCFG(p_gpr_pack::TMP0, p_cfg::WRCFG_32b, STACC_RELU_ApplyRelu_ADDR32)`
8.   Inserts NOPs for timing

## ReLU Application Timing and Data Flow

### When ReLU is Applied

ReLU is applied by the packer hardware during the destination-to-L1 data transfer:

**Key Points:**

*   ReLU is applied **during the pack operation**, not as a separate math operation
*   The destination register contains the original (pre-ReLU) values
*   L1 memory contains the post-ReLU values
*   Zero overhead - no additional compute cycles required
*   Hardware automatically applies the configured ReLU mode to each datum

## Threshold Encoding

### Threshold Value Format

The 16-bit `STACC_RELU_ReluThreshold` field stores the threshold in the **source data format** used by the packer. This means:

*   For Float16_b: The threshold is encoded as a 16-bit bfloat16 value
*   For Float16: The threshold is encoded as a 16-bit float16 value
*   For Float32: Only the upper 16 bits are used (requires careful encoding)
*   For integer formats: The threshold is encoded as a signed/unsigned integer

## Configuration Register Mapping

### Hardware Register Details

### Bit Masks

The following bit masks are used for read-modify-write operations:

*   `STACC_RELU_ApplyRelu_MASK`: Masks the 4-bit mode field
*   `STACC_RELU_ReluThreshold_MASK`: Masks the 16-bit threshold field
*   Combined mask: `STACC_RELU_ApplyRelu_MASK | STACC_RELU_ReluThreshold_MASK`

These masks ensure that only the ReLU-related fields are modified, preserving other control bits in the register.

## Summary

The ReLU configuration system provides hardware-accelerated activation function application during packing operations. Key characteristics:

*   **Four operation modes**: NO_RELU, ZERO_RELU, MIN_THRESHOLD_RELU, MAX_THRESHOLD_RELU
*   **Configurable threshold**: 16-bit value in source data format
*   **Zero-overhead application**: Applied during dest-to-L1 transfer
*   **Simple encoding**: 32-bit parameter with mode in lower 4 bits, threshold in upper 16 bits
*   **Dynamic reconfiguration**: Can be changed without full packer reinitialization
*   **Hardware-controlled**: No software intervention during each pack operation

The configuration is set once and applies to all subsequent pack operations until explicitly changed, making it efficient for processing large batches of tiles with consistent activation requirements.
