## Purpose and Scope

This page provides practical guidance on using NOC virtual channels (VCs) correctly in TT-Metal kernels. Virtual channels are a hardware feature designed to prevent head-of-line blocking, but counter-intuitively, using multiple VCs to parallelize the same type of traffic often degrades performance rather than improving it.

The key principle: **Use virtual channels to separate different traffic classes, not to parallelize identical traffic to the same destination.**

For detailed performance evaluation and experimental results, see [Virtual Channels Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.3-virtual-channels-performance-study). For general NOC API usage patterns, see [Core API Concepts](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.1-core-api-concepts).

## The Common Misconception

### Intuitive (but Wrong) Approach

Many developers assume that alternating traffic across multiple virtual channels will increase throughput by enabling parallel data movement:

### Why This Fails

The NOC architecture has two implementation details that prevent this optimization from working:

**Key architectural constraints:**

1.   **Router VC Allocation Ordering** - The router arbitrates among VCs before arbitrating among ports. If a blocked VC is selected, idle cycles occur even if other VCs could route traffic.

2.   **Input Buffer Sharing** - Input buffers are shared across VCs and sized to support 3 active VCs at 100% bandwidth. Using 4 VCs creates buffer pressure, dropping performance to 1/16th bandwidth when multiple VCs stall.

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 79-86](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L79-L86)[data_movement_doc/general/dm_best_practices.md 152-160](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L152-L160)

## Correct Usage: Traffic Class Separation

Virtual channels should separate traffic with **different bandwidth characteristics** or **different quality-of-service requirements**, not parallelize identical traffic.

### What Constitutes Different Traffic Classes

**Bandwidth-based separation:**

*   **DRAM traffic** has different (slower) bandwidth characteristics than L1/Ethernet → Use separate VC
*   **L1 and Ethernet traffic** have similar bandwidth characteristics → Can share the same VC

**Purpose-based separation:**

*   **Flow control/credit messages** indicate buffer space availability → Use separate VC from data
*   **High-priority vs low-priority** operations → Use separate VCs to prevent priority inversion

**Sources:**[data_movement_doc/general/dm_best_practices.md 157-159](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L157-L159)[data_movement_doc/virtual_channels/Virtual Channels.md 94-96](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L94-L96)

## Available Virtual Channels

The following virtual channels are available for software use:

| VC Number | Operation Type | Purpose |
| --- | --- | --- |
| 0 | Unicast | Available for writes/reads |
| 1 | Unicast | Available for writes/reads |
| 2 | Unicast | Available for writes/reads (historically used for credits) |
| 3 | Unicast | Available for writes/reads |
| 4 | Multicast | Only VC available for multicast operations |
| 5-15 | System | Reserved for internal protocols |

**Important constraints:**

*   **Unicast operations:** Valid VC values are 0, 1, 2, 3
*   **Multicast operations:** VC 4 is the only VC that can be used
*   **Other VCs:** Reserved for system use (e.g., dateline VCs, internal protocols)

**Sources:**[data_movement_doc/general/dm_best_practices.md 161-166](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L161-L166)

## Response and Acknowledgement VCs

A critical limitation of virtual channel control:

**Software has no way of controlling virtual channel number assignment for response/acknowledgement packets.** The hardware automatically assigns VCs for:

*   Read response data packets
*   Write acknowledgement packets (for non-posted writes)

This means that even if you carefully assign request VCs to separate traffic classes, the responses may share VCs and potentially interfere. This is an architectural constraint that cannot be worked around in software.

**Sources:**[data_movement_doc/general/dm_best_practices.md 205-206](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L205-L206)

## When Manual VC Assignment Helps

Manual VC assignment provides tangible benefits in these scenarios:

### Scenario 1: DRAM vs Compute Traffic Separation

Without VC separation, slow DRAM traffic could block fast L1 operations in the router queues.

### Scenario 2: Flow Control Message Separation

Flow control messages indicating buffer availability should not compete with data traffic for VC resources.

**Sources:**[data_movement_doc/general/dm_best_practices.md 183-204](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L183-L204)

## Practical Guidelines Summary

### Do:

✓ Use separate VCs for traffic with **different bandwidth characteristics** (DRAM vs L1)

 ✓ Use separate VCs for **flow control/credit messages** vs data

 ✓ Use separate VCs for **high-priority vs low-priority** traffic

 ✓ Use the same VC for traffic with **similar characteristics** (L1 and Ethernet)

### Don't:

✗ Split identical traffic across multiple VCs expecting parallelization

 ✗ Separate debug and production data into different VCs if targeting the same destination

 ✗ Use more than 3 VCs concurrently at high load (buffer pressure)

 ✗ Expect to control response/ACK VC selection (hardware-controlled)
