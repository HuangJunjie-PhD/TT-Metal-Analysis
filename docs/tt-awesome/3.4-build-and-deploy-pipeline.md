This page details the full CI/CD lifecycle of the `tt-awesome` project. The pipeline is designed to transform a collection of raw JSON files into both a human-readable `README.md` and a high-performance static website hosted on GitHub Pages. The architecture follows a "unidirectional data flow" where `entries/*.json` serves as the single source of truth for all downstream artifacts.

## CI/CD Lifecycle Overview

The automation infrastructure is divided into two primary workflows: **Validation** (for pull requests) and **Build & Deploy** (for the `main` branch).

## The Build Pipeline (`deploy.yml`)

The `deploy.yml` workflow is the core engine of the project. It triggers automatically on every push to the `main` branch or via manual `workflow_dispatch`.

### 1. Python Preprocessing Phase

Before the website is built, two Python scripts execute to update the repository's root artifacts:

*   **`generate_readme.py`**: Iterates through all entries to rebuild the main `README.md`. It applies the "community-first" sorting logic where community projects appear before official Tenstorrent projects to emphasize ecosystem growth `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/generate_readme.py#L80-L91" min=80 max=91 file-path="scripts/generate_readme.py">Hii</FileRef>`.
*   **`generate_data_json.py`**: Aggregates all entry data into a single `data.json` file. This file acts as a local API for the frontend build and external consumers `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/generate_data_json.py#L59-L84" min=59 max=84 file-path="scripts/generate_data_json.py">Hii</FileRef>`.

### 2. Eleventy Phase

Once the data artifacts are ready, the workflow transitions to the Node.js environment:

*   **Environment Setup**: Uses `actions/setup-node@v4` with `node-version: "22"` and `npm` caching enabled `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L31-L32" min=31 max=32 file-path=".github/workflows/deploy.yml">Hii</FileRef>`.
*   **Dependency Installation**: Runs `npm ci` to ensure a clean, reproducible install of `@11ty/eleventy``<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L34-L34" min=34  file-path=".github/workflows/deploy.yml">Hii</FileRef>`.
*   **Static Site Generation**: Executes `npm run build` (aliased to `eleventy` in `package.json`). Eleventy consumes the generated `data.json` and templates in `src/` to produce the final HTML in the `_site/` directory `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/package.json#L6-L12" min=6 max=12 file-path="package.json">Hii</FileRef>`.

### 3. Deployment Phase

The workflow uses official GitHub Actions to upload and deploy the site:

*   **Upload**: `actions/upload-pages-artifact@v3` packages the `./_site` directory `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L37-L39" min=37 max=39 file-path=".github/workflows/deploy.yml">Hii</FileRef>`.
*   **Deploy**: `actions/deploy-pages@v4` pushes the artifact to the `github-pages` environment `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L41-L50" min=41 max=50 file-path=".github/workflows/deploy.yml">Hii</FileRef>`.

## PR Gating and Validation (`validate.yml`)

To maintain data integrity, every Pull Request that modifies the `entries/` directory must pass the `validate.yml` workflow.

*   **Trigger**: Filters for changes in `entries/**` or the validation script itself `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/validate.yml#L2-L4" min=2 max=4 file-path=".github/workflows/validate.yml">Hii</FileRef>`.
*   **Execution**: Runs `scripts/validate.py`, which checks for: 
    *   Schema compliance (required fields like `name`, `description`, `affiliation`).
    *   ID-to-filename matching (e.g., `entries/ai-models/my-proj.json` must have `"id": "my-proj"`).
    *   URL formatting and enum constraints for categories and hardware.

## Jekyll Suppression Strategy

GitHub Pages by default attempts to build sites using Jekyll. Because `tt-awesome` uses Eleventy, Jekyll must be explicitly suppressed to prevent build conflicts and ensure files starting with underscores (like `_site/`) are handled correctly.

The `_config.yml` file in the root directory is used solely for this purpose. It instructs Jekyll's build engine to ignore the entire project structure:

*   `src/`, `entries/`, `scripts/`, `node_modules/`, and `.eleventy.js` are all excluded `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/_config.yml#L4-L15" min=4 max=15 file-path="_config.yml">Hii</FileRef>`.
*   This ensures that GitHub Pages simply serves the pre-built files uploaded by the `deploy.yml` workflow without further modification `<FileRef file-url="https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/_config.yml#L1-L3" min=1 max=3 file-path="_config.yml">Hii</FileRef>`.
