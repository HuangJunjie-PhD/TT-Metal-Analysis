## Purpose and Scope

The Jenkins Pipeline System is a template-based infrastructure that generates and maintains Jenkins pipeline definitions (`Jenkinsfile`) for continuous integration and testing across multiple platforms. The system uses Jinja2 templates to generate declarative Jenkins pipelines, implements smart change detection to preserve timestamps when only Docker image tags change, and orchestrates pytest execution with test sharding support.

For information about the Docker environments used by these pipelines, see [Docker CI Environments](https://deepwiki.com/tenstorrent/tt-tvm/10.2-docker-ci-environments). For details about the testing framework executed by the pipelines, see [Testing Framework](https://deepwiki.com/tenstorrent/tt-tvm/10.4-testing-framework).

## Template-Based Generation

### Template Structure

The Jenkins pipeline definitions are generated from Jinja2 templates stored in `ci/jenkins/templates/`. The system uses template inheritance and variable substitution to create maintainable pipeline definitions.

| Component | Location | Purpose |
| --- | --- | --- |
| Template Sources | `ci/jenkins/templates/*_jenkinsfile.groovy.j2` | Pipeline definition templates |
| Base Template | `ci/jenkins/templates/base.groovy.j2` | Shared template components |
| Configuration Data | `ci/jenkins/data.py` | Variables for template rendering |
| Generated Output | `ci/jenkins/generated/` | Final Jenkinsfile without extension |

**Template Discovery Process:**

**Sources:**[ci/jenkins/generate.py 35-38](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L35-L38)[ci/jenkins/generate.py 163-164](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L163-L164)

### Jinja2 Environment Configuration

The generator creates a strict Jinja2 environment with specific settings to ensure clean template rendering:

`environment = jinja2.Environment(    loader=jinja2.FileSystemLoader(TEMPLATES_DIR),    undefined=jinja2.StrictUndefined,  # Fail on undefined variables    lstrip_blocks=True,                # Remove leading whitespace    trim_blocks=True,                  # Remove trailing newlines    keep_trailing_newline=True,        # Preserve final newline)`
**Key Configuration Parameters:**

*   **`StrictUndefined`**: Raises exceptions for undefined template variables, preventing silent errors
*   **`lstrip_blocks=True`**: Removes indentation before block tags for cleaner output
*   **`trim_blocks=True`**: Removes newlines after block tags
*   **`keep_trailing_newline=True`**: Ensures generated files end with newline

## Change Detection System

### Smart Timestamp Preservation

The system implements intelligent change detection to avoid unnecessary Jenkins pipeline rebuilds when only Docker image tags change. This optimization prevents pipeline churn from routine image updates.

**Sources:**[ci/jenkins/generate.py 55-103](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L55-L103)[ci/jenkins/generate.py 105-153](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L105-L153)

### Change Type Classification

The `change_type()` function analyzes diff lines to classify changes into three categories:

| Change Type | Condition | Behavior |
| --- | --- | --- |
| `Change.NONE` | No diff lines found | Preserve original timestamp |
| `Change.IMAGES_ONLY` | Only Docker image tag changes (`ci_* = '...'`) | Preserve original timestamp |
| `Change.FULL` | Any non-image changes detected | Update to new timestamp |

**Detection Algorithm:**

1.   Parse diff lines starting with `+` or `-`
2.   Extract image assignments matching pattern: `^(ci_[a-zA-Z0-9]+) = '.*'$`
3.   Track added and removed image variables separately
4.   Classify as `IMAGES_ONLY` if added and removed lists match exactly
5.   Otherwise classify as `FULL` change requiring timestamp update

**Sources:**[ci/jenkins/generate.py 61-103](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L61-L103)

### Regex Pattern Matching

**Timestamp Extraction:**

```
Pattern: ^// Generated at (.*)$
Flags:   re.MULTILINE
Purpose: Extract ISO timestamp from existing Jenkinsfile
```

**Image Tag Detection:**

```
Pattern: ^(ci_[a-zA-Z0-9]+) = '.*'$
Flags:   re.MULTILINE
Purpose: Identify Docker image variable assignments
```

## Generation Process

### Command-Line Interface

The generator script supports two modes of operation:

| Mode | Flag | Description |
| --- | --- | --- |
| Generate | (default) | Render templates and write to `generated/` |
| Check | `--check` | Verify generated files match templates without writing |
| Force | `--force` | Always update timestamp even for image-only changes |

**Usage Examples:**

`# Generate Jenkinsfiles from templatespython3 ci/jenkins/generate.py # Verify templates are up-to-date (CI check)python3 ci/jenkins/generate.py --check # Force timestamp update for all filespython3 ci/jenkins/generate.py --force`
**Sources:**[ci/jenkins/generate.py 156-161](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L156-L161)

### Check Mode Workflow

**Error Messages:**

When check mode fails, the script outputs:

```
Newly generated Jenkinsfiles did not match the ones on disk! If you have made
edits to the Jenkinsfiles in generated/, move them to the corresponding source and
regenerate the Jenkinsfiles from the templates with

    python3 -m pip install -r jenkins/requirements.txt
    python3 jenkins/generate.py
```

**Sources:**[ci/jenkins/generate.py 166-191](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L166-L191)

## Integration with Pytest

### Environment Setup Script

The `setup-pytest-env.sh` script configures the test environment and provides the `run_pytest()` function for executing test suites with sharding support.

**Environment Variables Set:**

| Variable | Value | Purpose |
| --- | --- | --- |
| `TVM_PATH` | `$(pwd)` | Repository root directory |
| `PYTHONPATH` | `${TVM_PATH}/python` | Python module search path |
| `TVM_PYTEST_RESULT_DIR` | `${TVM_PATH}/build/pytest-results` | Test result XML output directory |
| `PYTEST_ADDOPTS` | `-s -vv [additional]` | Default pytest options |
| `TVM_FFI` | `<ffi_type>` | FFI implementation to test |

**Sources:**[tests/scripts/setup-pytest-env.sh 22-32](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L22-L32)

### Test Execution with Sharding

**Sources:**[tests/scripts/setup-pytest-env.sh 49-97](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L49-L97)

### JUnit XML Output Format

Test results are written in JUnit XML format with structured naming:

```
File: ${TVM_PYTEST_RESULT_DIR}/${suite_name}.xml
Suite Name: ${test_suite_name}-${current_shard}-${ffi_type}
Prefix: ${ffi_type}
```

**Example Suite Names:**

*   `integration-tests-no-shard-ctypes`
*   `relay-tests-shard-3-cython`
*   `tir-tests-shard-1-cppnative`

**Sources:**[tests/scripts/setup-pytest-env.sh 76-89](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L76-L89)

### Error Collection and Reporting

The script uses a Bash trap to collect all pytest failures and report them at script exit:

`pytest_errors=() function cleanup() {    if [ "${#pytest_errors[@]}" -gt 0 ]; then        echo "These pytest invocations failed..."        python3 ci/scripts/jenkins/pytest_wrapper.py "${pytest_errors[@]}"        exit 1    fi}trap cleanup 0`
This ensures all test suites complete before reporting failures, enabling parallel test execution while preserving error context.

**Sources:**[tests/scripts/setup-pytest-env.sh 34-47](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L34-L47)

### Pytest Rerun Configuration

The system automatically adds test retry capability if the `pytest-rerunfailures` plugin is available:

`has_reruns=$(python3 -m pytest --help 2>&1 | grep 'reruns=' || true)if [ -n "$has_reruns" ]; then    extra_args+=('--reruns=3')fi`
This provides resilience against flaky tests by automatically retrying failed tests up to 3 times.

**Sources:**[tests/scripts/setup-pytest-env.sh 71-74](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L71-L74)

## Validation and Error Handling

### Generation Validation

The system performs several validation steps during generation:

1.   **Template Existence**: Verifies source template files exist
2.   **Timestamp Extraction**: Ensures existing Jenkinsfiles have valid timestamps
3.   **Strict Undefined Variables**: Jinja2 fails on undefined template variables
4.   **Diff Computation**: Validates diff generation for change detection

**Timestamp Validation Error:**

`if not timestamp_match:    raise RuntimeError(        f"Could not find timestamp in Jenkinsfile: {destination.relative_to(TEMPLATES_DIR)}"    )`
**Sources:**[ci/jenkins/generate.py 113-119](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/ci/jenkins/generate.py#L113-L119)

### Pytest Exit Code Handling

The pytest wrapper handles specific exit codes with semantic meaning:

`# Pytest will return error code -5 if no test is collected.if [ "$exit_code" -ne "0" ] && [ "$exit_code" -ne "5" ]; then    pytest_errors+=("${suite_name}: $@")fi`
**Exit Code Meanings:**

*   `0`: All tests passed
*   `5`: No tests collected (not treated as failure)
*   Other non-zero: Test failures (recorded as error)

**Sources:**[tests/scripts/setup-pytest-env.sh 91-94](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L91-L94)

## Summary

The Jenkins Pipeline System provides a maintainable, template-based approach to CI/CD pipeline definition with the following key characteristics:

**Core Components:**

*   **Template Generator**: `ci/jenkins/generate.py` with Jinja2 rendering
*   **Change Detection**: Smart timestamp preservation for image-only changes
*   **Test Orchestration**: `setup-pytest-env.sh` with sharding and retry support
*   **Build Configuration**: Task-specific CMake configuration scripts

**Key Features:**

*   Declarative pipeline definitions from version-controlled templates
*   Intelligent diff analysis to minimize unnecessary rebuilds
*   Parallel test execution with automatic result collection
*   Integration with JUnit XML reporting for Jenkins test dashboard
*   Flexible test retry mechanism for flaky test resilience
