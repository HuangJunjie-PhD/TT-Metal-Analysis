## Purpose and Scope

This document describes the **tilize** and **untilize** operations that transform data between linear row-major memory layout and the tiled face-based layout used by Tensix hardware. Tilize operations are performed by the unpacker when reading data from L1 memory, while untilize operations are performed by the packer when writing results back to L1 memory.

For general unpacker operations beyond tilize, see [Core Unpacker Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/4.1-core-unpacker-operations-and-configuration). For general packer operations beyond untilize, see [Core Packer Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/6.1-core-packer-operations-and-configuration). For information about the tile and face memory structure, see [Tile and Face Memory Layout](https://deepwiki.com/tenstorrent/tt-llk/3.3-tile-and-face-memory-layout).

## Memory Layout Transformation

### Layout Conversion Overview

Sources: [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 1-217](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L1-L217)[tt_llk_wormhole_b0/llk_lib/llk_pack_untilize.h 1-212](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_untilize.h#L1-L212)

The standard 32x32 tile contains **4 faces** (each 16x16), organized as:

*   **Face 0**: Rows 0-15, Columns 0-15 (top-left)
*   **Face 1**: Rows 0-15, Columns 16-31 (top-right)
*   **Face 2**: Rows 16-31, Columns 0-15 (bottom-left)
*   **Face 3**: Rows 16-31, Columns 16-31 (bottom-right)

For narrow tiles (16x32), only **2 faces** are used (top two faces). Some operations support **1 face** (16x16) tiles for specific workloads like reduce operations.

Sources: [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 46-79](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L46-L79)[tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 43-92](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L43-L92)

### MOP Programming for Tilize

The MOP (Micro-Operation Program) for tilize unpacking differs based on whether data is unpacked to source registers or directly to destination registers:

**Standard Tilize (to Source Registers):**

`// Wormhole B0 implementationunpack_srca = TT_OP_UNPACR(SrcA, 0b1 /*Z inc*/, 0, 0, 0, 1, 1, ...);unpack_srcb_zerosrc = TT_OP_UNPACR_NOP(SrcB, p_unpacr_nop::UNP_ZEROSRC);unpack_srcb_set_dvalid = TT_OP_UNPACR_NOP(SrcB, p_unpacr_nop::UNP_SET_DVALID); ckernel_template tmp(outerloop, innerloop, unpack_srcb_zerosrc, unpack_srcb_set_dvalid);tmp.set_start_op(unpack_srca);`
**Tilize to Destination (for 32-bit formats):**

`unpack_srca_to_dest = TT_OP_UNPACR(SrcA, 0b00010001 /*CH0/CH1 Z inc*/, 0, 0, 0, 1, 0, ...);ckernel_template tmp(outerloop, innerloop, unpack_srca_to_dest);`

### Unpack-to-Dest Mode for 32-bit Formats

When unpacking 32-bit integers (`UInt32`, `Int32`) or lossless `Float32` data, the unpacker writes directly to destination registers, bypassing source registers. This mode is required because 32-bit data cannot fit in the 16-bit source register format.

The decision logic is:

`const bool unpack_to_dest = (unpack_src_format == to_underlying(DataFormat::UInt32)) ||                             (unpack_src_format == to_underlying(DataFormat::Int32)) ||                            (unpack_dst_format == to_underlying(DataFormat::Float32));`
Key differences in unpack-to-dest mode:

*   **Destination availability check**: `wait_for_dest_available()` before unpacking [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 129](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L129-L129)
*   **Set destination write address**: `set_dst_write_addr(unp_cfg_context, unpack_src_format)`[tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 128](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L128-L128)
*   **Different MOP instruction**: Uses `CH0/CH1 Z inc` to write both channels simultaneously [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 26](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L26-L26)

### Fast Tilize Operation

The **fast tilize** operation (`_llk_unpack_fast_tilize_*`) is an optimized tilize path that uses both unpackers (SrcA and SrcB) and the packer simultaneously to achieve higher throughput. It supports processing 1, 2, or 3 tiles per iteration.

Key characteristics:

*   **Dual unpacker usage**: UNPA reads top rows, UNPB reads bottom rows or different sections
*   **Unit dimensions**: 
    *   `unit_dim = 1`: Process one full tile per iteration
    *   `unit_dim = 2`: Process two tiles side-by-side per iteration
    *   `unit_dim = 3`: Process three tiles side-by-side per iteration

*   **Custom tile dimensions**: Sets x/y/z dims to cover multiple tiles in a row [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 517-532](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L517-L532)
*   **CH1 Z stride**: Configured to 32 datums (one row) for unit_dim 2/3 [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 538-540](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L538-L540)

Address calculation for dual unpacker:

`std::uint32_t address = base_address + SCALE_DATUM_SIZE(unpack_src_format, tile_index * TILE_C_DIM) >> 4;std::uint32_t unpB_row_offset = unit_dim == 2 ? FACE_R_DIM : (FACE_R_DIM / 2);std::uint32_t unpB_address = address + SCALE_DATUM_SIZE(unpack_src_format, full_dim * TILE_C_DIM * unpB_row_offset) >> 4;`
Sources: [tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 466-544](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L466-L544)[tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 562-693](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L562-L693)

### Address Modifier Configuration

The address modifiers control how the packer increments through destination registers and L1 memory:

**Standard Untilize Mode:**

`// ADDR_MOD_0: Increment y by 15, then add 1 with INCADCXY instructionaddr_mod_pack_t { .y_src = {.incr = 15} }.set(ADDR_MOD_0); // ADDR_MOD_1: Complete row with carry-resetaddr_mod_pack_t { .y_src = {.incr = 0, .clr = 0, .cr = 1} }.set(ADDR_MOD_1); // ADDR_MOD_2: Clear y counter for next tileaddr_mod_pack_t { .y_src = {.incr = 0, .clr = 1, .cr = 0} }.set(ADDR_MOD_2);`

`// Increment only by 1 per row for diagonal elementsaddr_mod_pack_t { .y_src = {.incr = 1} }.set(ADDR_MOD_0);`
**Narrow Row Mode:**

`// No increment needed for narrow rowsaddr_mod_pack_t { .y_src = {.incr = 0, .clr = 0} }.set(ADDR_MOD_0);`

### MOP Programming for Untilize

The MOP for untilize packing varies based on configuration:

**Standard Block Untilize (Wormhole B0):**

`// Outer loop: iterate over column tiles (block_ct_dim)// Inner loop: single iteration, or multiple for narrow_row// Start: PACR instruction to begin packing// Inner loop body: INCADCXY to move to next row// End: PACR with Last=1, increment W counter for next tile`
**Dense Packing (Blackhole):** Dense mode packs two 16x32 tiles (2 faces each) into contiguous L1 memory by using all 4 packer interfaces simultaneously. This improves performance when `block_ct_dim` is even and `num_faces == 2`:

`const std::uint32_t PACK_INTF_SEL = dense ? p_pacr::ALL_INTF_ACTIVE : ...;constexpr std::uint32_t MOP_INNER_LOOP = dense ? block_ct_dim / 2 : block_ct_dim;`

### Strided Destination Access (Blackhole)

Blackhole introduces **strided destination access mode** for improved untilize performance. Each packer interface has a stride of `16 * block_size` where block_size is the size of a row within a face.

The striding mechanism:

*   **First outer loop iteration**: Pack row 0 and row 16 from tile 0, row 64 and row 80 from tile 1, etc.
*   **Second iteration**: Pack row 1 and row 17 from tile 0, row 65 and row 81 from tile 1, etc.
*   **Continue** for all `face_r_dim` rows

Configuration:

`// Set CH0 Zstride = 2 * face_r_dim * y_strideconst std::uint32_t z_stride = 2 * face_r_dim * y_stride;cfg_reg_rmw_tensix<PCK0_ADDR_CTRL_ZW_REG_0_Zstride_RMW>(z_stride); // PACR instruction with strided modeTT_OP_PACR(    p_pacr::CFG_CTXT_0,    p_pacr::NO_ROW_PAD_ZERO,    p_pacr::DST_ACCESS_STRIDED_MODE,  // Strided mode enabled    ADDR_MOD_0,    ...)`
Sources: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 164-170](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L164-L170)[tt_llk_blackhole/llk_lib/llk_pack_untilize.h 66-83](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L66-L83)

## Untilize Unpacker Operations (Blackhole)

In addition to untilize packing, Blackhole supports **untilize unpacking** operations that read tiled data from L1 and convert it to linear format in source registers. This is useful for operations that need row-major data in the unpacker.

Sources: [tt_llk_blackhole/llk_lib/llk_unpack_untilize.h 57-99](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_untilize.h#L57-L99)[tt_llk_blackhole/llk_lib/llk_unpack_untilize.h 126-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_untilize.h#L126-L206)

The untilize unpack MOP uses a replay buffer with sophisticated address management:

`load_replay_buf(0, replay_buf_len, [] {    TTI_DMANOP; // Wait for WRCFG completion    TTI_UNPACR(SrcA, 0b01000001 /*CH1_Y+=1, CH0_Z+=1*/, ...); // Unpack two rows    TTI_UNPACR(SrcA, 0b01000001 /*CH1_Y+=1, CH0_Z+=1*/, ...); // Unpack two more rows    TTI_ADDDMAREG(0, p_gpr_unpack::TILE_OFFSET, p_gpr_unpack::TILE_OFFSET, p_gpr_unpack::TILE_SIZE); // Move to next tile    TTI_STALLWAIT(p_stall::STALL_CFG, p_stall::THCON);    TTI_ADDRCRZW(0b001, 0, 0, 0, 0, 0b0001 /*CH0_Z*/); // Reset Z counter with carry-reset});`
Key aspects:

*   **Two passes**: First pass reads top faces (z=0), second pass reads bottom faces (z=2)
*   **Row-by-row processing**: Each MOP iteration reads 2 rows from each tile in the block
*   **Tile offset management**: TILE_SIZE is added to move between tiles horizontally
*   **Y counter increment**: Moves down to the next row in L1 memory

Sources: [tt_llk_blackhole/llk_lib/llk_unpack_untilize.h 21-55](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_untilize.h#L21-L55)[tt_llk_blackhole/llk_lib/llk_unpack_untilize.h 126-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_untilize.h#L126-L206)

## Architecture-Specific Differences

### Wormhole B0 vs Blackhole Comparison

| Feature | Wormhole B0 | Blackhole |
| --- | --- | --- |
| **Tilize Z dim handling** | Manual loop for 2 passes (top/bottom faces) | Single pass with extended Z dimension |
| **Tilize X dim** | Set to face dimension (16) | Set to full tile width (face_r_dim * num_faces * 16) |
| **TilizeA_B implementation** | Loop-based with manual address calc | Replay buffer with CFGSHIFTMASK |
| **Untilize striding** | Manual Y increment per row | Hardware strided mode with z_stride |
| **Untilize dense packing** | Not available | Supported for even block_ct_dim |
| **Untilize unpacker** | Not available | Supported with replay buffer |
| **Context management** | Both contexts used | Both contexts used |
| **Max block_ct_dim** | 8 tiles | 8 tiles (16 with dense) |

### Blackhole Tilize Improvements

Blackhole simplifies tilize operations by:

1.   **Extended tile dimensions**: Sets `Tile_x_dim` to cover the entire tile width, eliminating the need to loop over faces [tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 80-89](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L80-L89)
2.   **Single Z dimension**: Sets `Tile_z_dim = 1` since X dimension covers everything [tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 86](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L86-L86)
3.   **Replay buffer for TilizeA_B**: Uses `CFGSHIFTMASK` instruction to increment base address automatically [tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 214-222](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L214-L222)

Example Blackhole tilize configuration:

`const std::uint32_t Tile_x_dim = face_r_dim * num_faces * FACE_C_DIM;const std::uint32_t Tile_z_dim = 1;cfg_reg_rmw_tensix<THCON_SEC0_REG5_Tile_x_dim_cntx0_ADDR32, 0, 0xffffffff>(Tile_x_dim | (Tile_x_dim << 16));cfg_reg_rmw_tensix<THCON_SEC0_REG0_TileDescriptor_ADDR32 + 1, 0, 0xffff0000>(Tile_z_dim << 16);`
Sources: [tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 43-92](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L43-L92)

### Blackhole Dense Packing

Dense packing mode is unique to Blackhole and allows packing two narrow tiles (16x32, 2 faces each) side-by-side as if they were a single 32x32 region:

`template <std::uint32_t block_ct_dim, ..., bool dense = false>inline void _llk_pack_untilize_init_(...) {    static_assert(!dense || (block_ct_dim % 2 == 0), "block_ct_dim must be even when dense");    static_assert(!dense || (!narrow_row), "narrow_row must be false when dense");    LLK_ASSERT(!dense || (num_faces == 2), "num_faces must be 2 when dense");        const std::uint32_t PACK_INTF_SEL = dense ? p_pacr::ALL_INTF_ACTIVE : ...;    constexpr std::uint32_t MOP_INNER_LOOP = dense ? block_ct_dim / 2 : block_ct_dim;}`
The dense mode leverages all four packer interfaces to output data in the correct order without gaps, improving throughput for narrow tile workloads.

### Memory Layout Implications

The relationship between parameters and memory layout:

```
Standard Tile (32x32):
  num_faces = 4, face_r_dim = 16
  Total datums = 1024
  Layout: [Face0 Face1 Face2 Face3] in destination

Narrow Tile (16x32):
  num_faces = 2, face_r_dim = 16  
  Total datums = 512
  Layout: [Face0 Face1] in destination
  
Partial Face (e.g., 8x32):
  num_faces = 4, face_r_dim = 8
  Total datums = 256 valid (768 padding)
  Layout: [Face0(8x16) Face1(8x16) Face2(8x16) Face3(8x16)]
```
