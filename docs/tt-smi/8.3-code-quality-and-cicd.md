## Purpose and Scope

This page documents the code quality standards, automated checks, and continuous integration/continuous deployment (CI/CD) pipeline for TT-SMI. It covers code formatting enforcement using `black`, pre-commit hooks for local validation, and the GitLab CI pipeline that runs automated checks and builds artifacts. For information about the testing framework and test execution, see [Testing](https://deepwiki.com/tenstorrent/tt-smi/8.2-testing). For package configuration and release workflows, see [Build and Release System](https://deepwiki.com/tenstorrent/tt-smi/7-build-and-release-system).

## Code Formatting Standards

TT-SMI enforces consistent code formatting across the codebase using the `black` Python code formatter. The formatter configuration is version-specific based on the Python version in use.

### Black Formatter Configuration

The project specifies different `black` versions depending on Python version compatibility:

| Python Version | Black Version | Configuration Location |
| --- | --- | --- |
| >= 3.9 | 24.10.0 | [pyproject.toml 39](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L39-L39) |
| 3.8 | 24.8.0 | [pyproject.toml 40](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L40-L40) |
| 3.7 | 23.3.0 | [pyproject.toml 41](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L41-L41) |

The `black` formatter is defined as a development dependency in the `[project.optional-dependencies.dev]` section [pyproject.toml 38-42](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L38-L42)

### Formatting Rules

Black is configured to run with the following options in the CI pipeline:

*   `--check`: Verify files are formatted without making changes
*   `--verbose`: Display detailed output
*   `--diff`: Show differences if formatting is incorrect
*   `--color`: Enable colored output for readability

These options are applied in the GitLab CI pipeline [.gitlab-ci.yml 29](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L29-L29) to ensure formatting compliance before merge.

**Sources:**[pyproject.toml 38-42](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L38-L42)[.gitlab-ci.yml 19-29](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L19-L29)

## Pre-commit Hook System

TT-SMI uses pre-commit hooks to enforce quality checks locally before code is committed to the repository. This provides immediate feedback to developers and prevents non-compliant code from entering the CI pipeline.

### Hook Configuration

The pre-commit configuration is defined in `.pre-commit-config.yaml` and includes two repository sources:

#### Standard Pre-commit Hooks

From `https://github.com/pre-commit/pre-commit-hooks` (v4.5.0):

*   `check-yaml`: Validates YAML file syntax
*   `end-of-file-fixer`: Ensures files end with a newline
*   `trailing-whitespace`: Removes trailing whitespace
*   `check-toml`: Validates TOML file syntax (for `pyproject.toml`)

All hooks use Python 3.8 as the language version [.pre-commit-config.yaml 9](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L9-L9)

#### Black Formatting Hook

From `https://github.com/psf/black` (23.11.0):

*   `black`: Automatically formats Python code to match black standards
*   Language version: Python 3.8 [.pre-commit-config.yaml 14](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L14-L14)

### Installation and Usage

The pre-commit package is included as a required dependency in [pyproject.toml 34](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L34-L34):

```
'pre-commit>=3.5.0'
```

To enable pre-commit hooks locally, developers must install them after cloning the repository:

`pre-commit install`
Once installed, hooks run automatically before each `git commit` operation. Developers can also manually run hooks on all files:

`pre-commit run --all-files`
**Sources:**[.pre-commit-config.yaml 1-15](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L1-L15)[pyproject.toml 34](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L34-L34)

## GitLab CI/CD Pipeline

TT-SMI uses GitLab CI for automated quality checks and artifact generation. The pipeline is defined in `.gitlab-ci.yml` and consists of two stages executed sequentially.

### Pipeline Architecture

**Sources:**[.gitlab-ci.yml 15-52](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L15-L52)

### Stage Definitions

The pipeline defines two sequential stages [.gitlab-ci.yml 15-17](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L15-L17):

1.   `format` - Code quality validation
2.   `build` - Artifact generation

### Format Stage: Black Job

The `black` job runs in the `format` stage and enforces code formatting standards.

**Job Configuration:**

*   **Stage:**`format`[.gitlab-ci.yml 20](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L20-L20)
*   **Tags:**`8-core`[.gitlab-ci.yml 21-22](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L21-L22)
*   **Execution:** Runs on every commit/merge request

**Script Steps:**

1.   Install system dependencies: `curl`, `git`, `python3.8`, `python3.8-venv`, `python3-pip`[.gitlab-ci.yml 24](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L24-L24)
2.   Create and activate Python 3.8 virtual environment [.gitlab-ci.yml 25-26](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L25-L26)
3.   Upgrade pip [.gitlab-ci.yml 27](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L27-L27)
4.   Install `black==24.8.0`[.gitlab-ci.yml 28](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L28-L28)
5.   Run black with validation flags [.gitlab-ci.yml 29](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L29-L29)

**Failure Conditions:** The job fails if any Python file does not conform to black formatting standards. The `--check` flag ensures black does not modify files, only validates them. The `--diff` flag displays the required changes, allowing developers to identify and fix formatting issues.

**Sources:**[.gitlab-ci.yml 19-29](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L19-L29)

### Build Stage: Build Job

The `build` job runs in the `build` stage and generates distributable artifacts.

**Job Configuration:**

*   **Stage:**`build`[.gitlab-ci.yml 32](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L32-L32)
*   **Tags:**`8-core`[.gitlab-ci.yml 33-34](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L33-L34)
*   **Execution:** Only runs if the `format` stage passes

**Script Steps:**

1.   Install system dependencies [.gitlab-ci.yml 36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L36-L36)
2.   Install Rust toolchain using rustup [.gitlab-ci.yml 37-38](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L37-L38)

3.   Create and activate Python 3.8 virtual environment [.gitlab-ci.yml 39-40](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L39-L40)
4.   Install `build` package [.gitlab-ci.yml 41-42](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L41-L42)
5.   Build Python wheel [.gitlab-ci.yml 43](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L43-L43)
6.   Extract wheel file to project root [.gitlab-ci.yml 44](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L44-L44)

## Quality Gate Enforcement

TT-SMI implements multiple layers of quality enforcement to ensure code consistency and correctness.

### Quality Gate Hierarchy

**Sources:**[.pre-commit-config.yaml 1-15](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L1-L15)[.gitlab-ci.yml 15-52](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L15-L52)

### Enforcement Layers

| Layer | Enforcement Point | Tools | Blocking |
| --- | --- | --- | --- |
| **Local (Optional)** | Pre-commit hook | check-yaml, end-of-file-fixer, trailing-whitespace, check-toml, black | Yes (if installed) |
| **CI (Required)** | GitLab format stage | black | Yes (blocks merge) |
| **CI (Required)** | GitLab build stage | Python build tools | Yes (blocks merge) |

### Pre-commit vs CI Enforcement

**Pre-commit Hooks:**

*   Optional but recommended
*   Provides immediate feedback
*   Catches issues before push
*   Can be bypassed with `--no-verify` flag
*   Uses `black==23.11.0`[.pre-commit-config.yaml 11](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L11-L11)

**CI Pipeline:**

*   Mandatory for all commits
*   Cannot be bypassed
*   Runs on GitLab infrastructure
*   Uses `black==24.8.0`[.gitlab-ci.yml 28](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L28-L28)
*   Blocks merge requests on failure

**Version Discrepancy Note:** The pre-commit hook uses `black==23.11.0` while the CI pipeline uses `black==24.8.0`. This may cause differences in formatting between local and CI checks. Developers should ensure their local environment matches the CI version for consistency.

**Sources:**[.pre-commit-config.yaml 10-14](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L10-L14)[.gitlab-ci.yml 28](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L28-L28)

## Best Practices for Contributors

### Before Committing

1.   **Install pre-commit hooks:**

`pip install pre-commit>=3.5.0pre-commit install`
2.   **Run checks manually:**

`pre-commit run --all-files`
3.   **Format code with black:**

`pip install black==24.8.0  # Match CI versionblack .`

### During Code Review

1.   **Check CI status:** Ensure both `format` and `build` stages pass
2.   **Review artifacts:** Verify wheel file is generated correctly
3.   **Address failures:** Fix any formatting or build issues before merge

### Version Consistency

To avoid discrepancies between local and CI environments:

*   Use `black==24.8.0` for local development (CI version)
*   Ensure Python 3.8 or higher is available
*   Install development dependencies: `pip install -e ".[dev]"`
