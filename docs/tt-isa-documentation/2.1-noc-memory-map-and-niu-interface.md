## Purpose and Scope

This document details the memory-mapped register interface of Network Interface Units (NIUs), which provide the primary mechanism for software running on Tensix and Ethernet tiles to initiate and monitor NoC transactions. Each NIU exposes configuration registers, request initiators, status registers, and identification information through a well-defined memory map.

For information about NoC transaction types, routing mechanisms, and addressing modes, see [Transaction Types and Routing](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing). For details on virtual channel architecture and ordering guarantees, see [Virtual Channels and Ordering Guarantees](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.3-virtual-channels-and-ordering-guarantees). For counter semantics and transaction monitoring, see [NoC Counters and Alignment Requirements](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements).

Sources: [WormholeB0/NoC/MemoryMap.md 1-11](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L1-L11)

## NIU Base Addresses

Each tile contains two NIUs: NIU #0 connected to NoC #0, and NIU #1 connected to NoC #1. The base address (`NIU_BASE`) where NIU registers are mapped depends on the tile type:

| Tile Type | NIU #0 Base Address | NIU #1 Base Address |
| --- | --- | --- |
| Tensix or Ethernet | `0x0_FFB2_0000` or `0x?_FFB2_0000` | `0x0_FFB3_0000` or `0x?_FFB3_0000` |
| Other (via NoC #0) | `0xF_FFB2_0000` or `0xF_????_?000` | Cannot access |
| Other (via NoC #1) | Cannot access | `0xF_FFB3_0000` or `0xF_????_?000` |

**Important constraint**: Software can only use NIU request initiators in Tensix tiles and Ethernet tiles. Other tile types can receive NoC transactions but cannot initiate them through their NIUs. Host software uses a different mechanism (via the PCIe tile) to send NoC requests.

Sources: [WormholeB0/NoC/MemoryMap.md 3-10](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L3-L10)[WormholeB0/NoC/MemoryMap.md 46-47](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L46-L47)

## Memory Map Overview

**Figure 1: NIU Memory Map Structure**

The NIU memory map is organized into several functional regions:

1.   **Request Initiators (4 total)**: Primary interface for initiating NoC transactions
2.   **Identification Registers**: Read-only information about the NIU and its location
3.   **Counter Control**: Write-only register for clearing transaction ID counters
4.   **Combined Status**: Convenient read-only view of all initiator busy states
5.   **Configuration Registers**: Control NIU behavior, coordinate translation, and broadcast filtering
6.   **Counters**: Transaction monitoring counters (see [NoC Counters](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements))
7.   **Debug Registers**: Router and NIU diagnostic information

Sources: [WormholeB0/NoC/MemoryMap.md 11-28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L11-L28)

### NOC_CTRL Register

`NOC_CTRL` specifies the request type and commonly used flags:

| Bits | Field Name | Description |
| --- | --- | --- |
| [1:0] | Request type | `0` = Read (`NOC_CMD_RD`) `1` = Atomic (`NOC_CMD_AT`) `2` = Write (`NOC_CMD_WR`) `3` = Reserved |
| [2] | `NOC_CMD_WR_BE` | For writes: `1` = `NOC_AT_LEN_BE` is byte-enable mask, `0` = length |
| [3] | `NOC_CMD_WR_INLINE` | For writes: `1` = data in `NOC_AT_DATA`, `0` = data from memory |
| [4] | `NOC_CMD_RESP_MARKED` | For writes/atomics: `1` = request acknowledgement, `0` = posted |
| [5] | `NOC_CMD_BRCST_PACKET` | `1` = broadcast to rectangle, `0` = unicast |
| [6] | `NOC_CMD_VC_LINKED` | `1` = part of multi-request transaction, `0` = single request |
| [7] | `NOC_CMD_VC_STATIC` | `1` = software controls virtual channel, `0` = hardware controls |
| [8] | `NOC_CMD_PATH_RESERVE` | For broadcasts: `1` = reserve path before sending data |
| [9] | `NOC_CMD_MEM_RD_DROP_ACK` | For writes: NoC Overlay acknowledgement control |
| [13] | `NOC_CMD_STATIC_VC` | When `NOC_CMD_VC_STATIC=1`: buddy bit |
| [15:14] | `NOC_CMD_STATIC_VC` | When `NOC_CMD_VC_STATIC=1`: class bits (`0b00`/`0b01` unicast, `0b10` multicast) |
| [16] | `NOC_CMD_BRCST_XY` | When broadcasting: `0` = X-major axis, `1` = Y-major axis |
| [17] | `NOC_CMD_BRCST_SRC_INCLUDE` | When broadcasting: `1` = include source in recipients |
| [30:27] | `NOC_CMD_ARB_PRIORITY` | Priority for virtual channel assignment (0 = default) |

Sources: [WormholeB0/NoC/MemoryMap.md 49-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L49-L71)

### NOC_TARG_ADDR and NOC_RET_ADDR Registers

The meaning of `NOC_TARG_ADDR` (combined `_LO` and `_MID`) and `NOC_RET_ADDR` depends on request type:

**Figure 3: Address Register Usage by Request Type**

#### Unicast Address Encoding

For unicast requests, the combined 64-bit address register contains:

| Bits | Field | Description |
| --- | --- | --- |
| [35:0] | Memory address | 36-bit physical address (high 4 bits usually zero) |
| [41:36] | X coordinate | 6-bit X coordinate (0-9 in practice) |
| [47:42] | Y coordinate | 6-bit Y coordinate (0-11 in practice) |
| [59:48] | Reserved | No effect on request/response |
| [63:60] | Reserved | Ignored by hardware |

#### Broadcast Address Encoding

For broadcast requests (`NOC_CMD_BRCST_PACKET=1`), `NOC_TARG_ADDR` uses a different encoding:

| Bits | Field | Description |
| --- | --- | --- |
| [35:0] | Memory address | 36-bit physical address at all recipients |
| [41:36] | EndX | 6-bit ending X coordinate |
| [47:42] | EndY | 6-bit ending Y coordinate |
| [53:48] | StartX | 6-bit starting X coordinate |
| [59:54] | StartY | 6-bit starting Y coordinate |
| [63:60] | Reserved | Ignored by hardware |

The broadcast rectangle spans all tiles `(x, y)` where:

*   X range: `StartX ≤ x ≤ EndX` (or wrapping: `x ≤ EndX || StartX ≤ x` if `StartX > EndX`)
*   Y range: `StartY ≤ y ≤ EndY` (or wrapping: `y ≤ EndY || StartY ≤ y` if `StartY > EndY`)

When coordinate translation is enabled, all four coordinate fields are translated before determining the broadcast span.

Sources: [WormholeB0/NoC/MemoryMap.md 73-114](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L73-L114)

### NOC_AT_LEN_BE Register

The interpretation of `NOC_AT_LEN_BE` depends on request type:

| Request Type | NOC_AT_LEN_BE Content |
| --- | --- |
| Atomic | Atomic opcode and operands (see [Transaction Types and Routing](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing)) |
| Read | Number of bytes to read (max 8192 for L1, 4 for MMIO) |
| Write (non-inline, `NOC_CMD_WR_BE=0`) | Number of bytes to write (max 8192 for L1, 4 for MMIO) |
| Write (non-inline, `NOC_CMD_WR_BE=1`) | Byte-enable mask for up to 32 bytes |
| Write (inline) | Byte-enable mask for up to 16 bytes |

For writes with byte-enable masks to L1 addresses:

*   Inline writes: Up to 16 bytes written to `NOC_TARG_ADDR & ~0xf`. Each byte address `i` in this range receives byte `i & 3` of `NOC_AT_DATA` when bit `i & 15` or bit `16 + (i & 15)` of `NOC_AT_LEN_BE` is set.
*   Non-inline writes: Up to 32 bytes written to destination. All 32 bits of the byte-enable mask select written bytes.

For writes to MMIO addresses, byte-enable masks are ignored and 4-byte writes are always performed.

Sources: [WormholeB0/NoC/MemoryMap.md 116-128](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L116-L128)

### NOC_PACKET_TAG Register

`NOC_PACKET_TAG` contains transaction identification and special-purpose flags:

| Bits | Field Name | Description |
| --- | --- | --- |
| [5:0] | Stream ID | When `DeliverToReceiverOverlay=1`: NoC Overlay stream ID |
| [6] | `DeliverToReceiverOverlay` | `1` = deliver to receiver's NoC Overlay (in addition to memory write) |
| [7] | First packet flag | When delivered to NoC Overlay: first packet of message |
| [8] | Last packet flag | When delivered to NoC Overlay: last packet of message |
| [9] | `NOC_PACKET_TAG_HEADER_STORE` | For posted writes: also write first 128 bits to `NOC_AT_DATA << 4` |
| [13:10] | `NOC_PACKET_TRANSACTION_ID` | 4-bit ID affecting which counters are incremented/decremented |
| [15:14] | Reserved | - |
| [31:16] | Reserved | Writes ignored, always read as zero |

**Hardware Bug**: When sending write requests to PCIe or ARC tiles with `NOC_CMD_RESP_MARKED=1`, `NOC_PACKET_TRANSACTION_ID` must be zero. Otherwise, counters at the transaction ID will be incremented, but counters at index zero will be decremented when the acknowledgement arrives.

Sources: [WormholeB0/NoC/MemoryMap.md 130-146](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L130-L146)

### NOC_CMD_CTRL Register and Request Initiation

Software initiates a NoC request by:

1.   Writing configuration to the initiator's registers (`NOC_TARG_ADDR`, `NOC_RET_ADDR`, `NOC_CTRL`, etc.)
2.   Writing `1` to bit 0 of `NOC_CMD_CTRL`

When software writes `1` to `NOC_CMD_CTRL`, the NIU immediately:

*   Increments relevant counters (`NIU_MST_REQS_OUTSTANDING_ID`, `NIU_MST_WRITE_REQS_OUTGOING_ID`)
*   Applies coordinate translation if enabled, updating `NOC_TARG_ADDR_MID` and `NOC_RET_ADDR_MID`
*   Writes values to some reserved fields in `NOC_CTRL`
*   Appends the initiator to an internal queue awaiting virtual channel assignment

Hardware continuously attempts to assign virtual channels to queued initiators. Once assignment succeeds, hardware clears bit 0 of `NOC_CMD_CTRL` back to `0`, indicating the initiator is available for reuse.

**Critical constraint**: While bit 0 of `NOC_CMD_CTRL` is `1`, software must not write to any fields of that initiator. Software should always check this bit before reconfiguring an initiator.

## Status and Identification Registers

### NIU Combined Request Initiator Status

The `NIU_BASE + 0x054` register provides a convenient way to check all four initiators' busy status simultaneously:

| Bits | Field | Description |
| --- | --- | --- |
| [0] | Initiator #0 status | Bit 0 of initiator #0's `NOC_CMD_CTRL` |
| [1] | Initiator #1 status | Bit 0 of initiator #1's `NOC_CMD_CTRL` |
| [2] | Initiator #2 status | Bit 0 of initiator #2's `NOC_CMD_CTRL` |
| [3] | Initiator #3 status | Bit 0 of initiator #3's `NOC_CMD_CTRL` |
| [31:4] | Reserved | Always zero |

Software can poll this single register instead of checking each initiator's `NOC_CMD_CTRL` individually.

Sources: [WormholeB0/NoC/MemoryMap.md 158-168](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L158-L168)

### NOC_NODE_ID Register

`NOC_NODE_ID` at `NIU_BASE + 0x02C` provides static information about the NIU and its attached NoC router:

| Bits | Field | Description |
| --- | --- | --- |
| [5:0] | X coordinate | NIU's X coordinate (0-9 in practice) |
| [11:6] | Y coordinate | NIU's Y coordinate (0-11 in practice) |
| [18:12] | NoC width | Number of NIUs per row (always 10) |
| [25:19] | NoC height | Number of NIUs per column (always 12) |
| [26] | X dateline enable | `1` if dateline bit can flip at X-axis output port |
| [27] | Y dateline enable | `1` if dateline bit can flip at Y-axis output port |
| [28] | Routing mode | `1` = X-major routing (NoC #0), `0` = Y-major (NoC #1) |
| [31:29] | Reserved | Always zero |

The X/Y coordinates in this register are in NoC coordinate space. When coordinate translation is enabled, software should read `NOC_ID_LOGICAL` to get translated coordinates.

Sources: [WormholeB0/NoC/MemoryMap.md 179-192](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L179-L192)

### NOC_ENDPOINT_ID Register

`NOC_ENDPOINT_ID` at `NIU_BASE + 0x030` uniquely identifies each NIU within an ASIC:

| Bits | Field | Description |
| --- | --- | --- |
| [7:0] | Tile index | Index within tile type (see below) |
| [15:8] | Group index | For DRAM tiles: 0-5, otherwise: 0 |
| [23:16] | Tile type | `1` = Tensix, `2` = Ethernet, `3` = PCIe/Empty, `5` = ARC, `8` = DRAM |
| [31:24] | NoC index | `0` = NoC #0, `1` = NoC #1 |

Tile index interpretation:

*   **Tensix tiles**: 0-79
*   **Ethernet tiles**: 0-15
*   **PCIe tile**: Always 2
*   **Empty tiles**: Some value other than 2
*   **ARC tile**: Always 0
*   **DRAM tiles**: 0-2 (within each group of 6 NIUs accessing the same 2 GiB)

Sources: [WormholeB0/NoC/MemoryMap.md 194-203](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L194-L203)

## Configuration Registers

### NIU_CFG_0 Register

`NIU_CFG_0` at `NIU_BASE + 0x100` controls NIU-level configuration:

| Bits | Field Name | Description |
| --- | --- | --- |
| [11:0] | Reserved | Software should preserve when modifying other bits |
| [12] | Tile clock disable | `1` = disable tile (set by firmware for fused-off tiles) |
| [13] | Double store disable | `1` = ignore `NOC_PACKET_TAG_HEADER_STORE` (non-Tensix/Ethernet tiles) |
| [14] | Coordinate translation enable | `1` = enable translation via `NOC_X_ID_TRANSLATE_TABLE_0` and `NOC_Y_ID_TRANSLATE_TABLE_0` |
| [31:15] | Available for general use | Software-defined |

Sources: [WormholeB0/NoC/MemoryMap.md 221-229](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L221-L229)

### Router Configuration Registers

Router configuration registers control NoC router behavior:

| Register | Address | Purpose |
| --- | --- | --- |
| `ROUTER_CFG_0` | `NIU_BASE + 0x104` | Bits [18:0] reserved, bits [31:19] available for general use |
| `ROUTER_CFG_1` | `NIU_BASE + 0x108` | Bits [9:0] = broadcast opt-out mask (columns), bits [31:10] general use |
| `ROUTER_CFG_2` | `NIU_BASE + 0x10C` | All 32 bits available for general use |
| `ROUTER_CFG_3` | `NIU_BASE + 0x110` | Bits [11:0] = broadcast opt-out mask (rows), bits [31:12] general use |
| `ROUTER_CFG_4` | `NIU_BASE + 0x114` | All 32 bits available for general use |

#### Broadcast Opt-Out Masks

`ROUTER_CFG_1` (bits [9:0]) and `ROUTER_CFG_3` (bits [11:0]) form broadcast opt-out masks:

*   **Column mask**: One bit per X coordinate. If the bit matching this NIU's X coordinate is set, the NIU is opted out of broadcasts.
*   **Row mask**: One bit per Y coordinate. If the bit matching this NIU's Y coordinate is set, the NIU is opted out of broadcasts.

Firmware configures these masks so that only Tensix tiles receive broadcast packets. The same masks should be set at all NIUs (required for correct operation at the PCIe tile).

Sources: [WormholeB0/NoC/MemoryMap.md 231-253](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L231-L253)

### Coordinate Translation Tables

When bit 14 of `NIU_CFG_0` is set, coordinate translation is enabled. As software writes to `NOC_CMD_CTRL`, hardware translates X/Y coordinates in `NOC_TARG_ADDR_MID` and `NOC_RET_ADDR_MID` from logical space to NoC coordinate space using lookup tables:

*   **X translation table**: 128 bits starting at `NOC_X_ID_TRANSLATE_TABLE_0` (`NIU_BASE + 0x118`)
*   **Y translation table**: 128 bits starting at `NOC_Y_ID_TRANSLATE_TABLE_0` (`NIU_BASE + 0x128`)

Each 128-bit table is:

*   Exposed as four consecutive 32-bit MMIO registers
*   Organized as 32 entries of 4 bits each for table lookup
*   Configured by firmware to map logical coordinates (16-25 for X, 16-27 for Y) to NoC coordinates (0-9 for X, 0-11 for Y)

For details on coordinate translation semantics, see [Transaction Types and Routing](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing).

Sources: [WormholeB0/NoC/MemoryMap.md 256-258](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L256-L258)

### NOC_ID_LOGICAL Register

`NOC_ID_LOGICAL` at `NIU_BASE + 0x138` provides the NIU's coordinates in translated space:

| Bits | Field | Description |
| --- | --- | --- |
| [5:0] | Logical X coordinate | X coordinate in translated space |
| [11:6] | Logical Y coordinate | Y coordinate in translated space |
| [31:12] | Available for general use | Software-defined |

Upon ASIC power-on, this register contains the NIU's NoC coordinates (same as `NOC_NODE_ID`). When firmware configures coordinate translation, firmware updates this register with logical coordinates (16-25 for X, 16-27 for Y).

Sources: [WormholeB0/NoC/MemoryMap.md 260-268](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L260-L268)

### Memory Ordering Considerations

When software writes to `NOC_CMD_CTRL` to initiate a request and then immediately reads from an NIU counter, RISC-V memory ordering rules must be considered to prevent reordering. To ensure correct ordering, software should read back from `NOC_CMD_CTRL` before performing the first counter read.

Sources: [WormholeB0/NoC/Counters.md 42-43](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L42-L43)

### Multi-Initiator Usage

Software can use multiple initiators simultaneously to overlap NoC transactions:

**Figure 6: Multi-Initiator Usage Strategy**

Best practices:

*   Use different initiators for concurrent requests to avoid waiting for `NOC_CMD_CTRL` to clear
*   Statically assign initiators to different transaction types to minimize register reconfiguration
*   Reserve one initiator for high-priority requests
*   Use transaction IDs to distinguish counter increments from different initiators
