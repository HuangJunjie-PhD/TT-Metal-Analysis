# Wormhole

This page documents the Wormhole chip implementation in the Luwen codebase. Wormhole is one of the Tenstorrent AI accelerator chip architectures supported by Luwen, alongside Grayskull and Blackhole. This document focuses specifically on how the Wormhole chip type is implemented, initialized, and interacted with through the Luwen interface.

For information about the common chip interface that all chip types implement, see [Common Interface (ChipImpl)](https://deepwiki.com/tenstorrent/luwen/3.1-common-interface-(chipimpl)).

## Overview

The Wormhole implementation provides a concrete implementation of the `ChipImpl` trait for the Wormhole architecture. It supports both local (PCI-connected) and remote (Ethernet-connected) chip access, handles chip initialization, and provides interfaces for communication, telemetry gathering, and inter-chip networking.

## Initialization

Initializing a Wormhole chip involves setting up the communication interfaces and address structures. The process varies depending on whether it's a local or remote chip.

The initialization process has several key steps:

1.   The `init` method creates the basic structure with communication interfaces
2.   For remote chips, `open_remote` method is used to establish a connection
3.   After basic initialization, `update_init_state` is called to check component status: 
    *   ARC (Application-specific RISC Cores) status
    *   DRAM training status
    *   Ethernet training status

4.   Each component is checked in sequence, and initialization waits until all are ready or times out

## Communication Methods

### ARC Messaging

The primary method for communicating with the Wormhole chip is through ARC (Application-specific RISC Core) messaging. This involves writing commands to specific memory-mapped registers and reading responses.

The Wormhole implementation includes safety checks (`check_arc_msg_safe`) to ensure ARC messaging is available and won't conflict with other operations.

Sources: [crates/luwen-if/src/chip/wormhole.rs 219-315](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/wormhole.rs#L219-L315)[crates/luwen-if/src/chip/wormhole.rs 831-849](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/wormhole.rs#L831-L849)

### SPI Access

Wormhole provides methods for SPI flash read/write access:

```
spi_read(addr, buffer) → Read data from SPI flash
spi_write(addr, data) → Write data to SPI flash
```

The implementation can use either direct SPI access or ARC-mediated access based on the `use_arc_for_spi` flag.

Sources: [crates/luwen-if/src/chip/wormhole.rs 317-331](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/wormhole.rs#L317-L331)

### NOC (Network-on-Chip) Communications

The Wormhole implements the `HlComms` interface, providing access to the chip's Network-on-Chip (NOC) for memory access:

*   `noc_read` - Read from NOC memory at specified coordinates
*   `noc_write` - Write to NOC memory at specified coordinates

These methods are fundamental for accessing the chip's memory space and are used extensively throughout the implementation.

Sources: [crates/luwen-if/src/chip/wormhole.rs 43-53](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/wormhole.rs#L43-L53)

## Ethernet Connectivity

Wormhole chips feature extensive Ethernet connectivity capabilities, allowing them to form networks of interconnected chips.

Key Ethernet features include:

1.   **Ethernet Cores**: Each Wormhole chip has 16 Ethernet cores positioned at specific NOC coordinates, defined in the `eth_locations` array.

2.   **Remote Chip Access**: The `open_remote` method enables creating a connection to another chip via Ethernet.

3.   **Neighbor Discovery**: `get_neighbouring_chips` method discovers and returns information about connected chips.

4.   **Ethernet Training**: Before Ethernet connections can be used, they must complete a training process that establishes link parameters.

5.   **Firmware Version Management**: The implementation includes methods to check and verify Ethernet firmware versions.

The `EthAddresses` structure maintains addresses that vary based on the firmware version, allowing compatibility with different firmware releases.

## Telemetry Collection

Wormhole chips provide extensive telemetry data through the `get_telemetry` method. This collects various metrics about the chip's state and operation.

The telemetry data includes:

| Category | Fields |
| --- | --- |
| Identification | `device_id`, `board_id`, `asic_ro`, `asic_idd` |
| Firmware Versions | `arc0_fw_version` through `arc3_fw_version`, `eth_fw_version`, `spibootrom_fw_version` |
| Component Status | `ddr_status`, `eth_status0`, `eth_status1`, `pcie_status` |
| Performance Metrics | `aiclk`, `axiclk`, `arcclk`, `fan_speed` |
| Health Metrics | `asic_temperature`, `vreg_temperature`, `board_temperature`, `vcore` |
| Power Management | `tdp`, `tdc`, `vdd_limits`, `thm_limits` |

## Ethernet Mapping Utility

The Wormhole implementation is used by the `create-ethernet-map` utility to discover and map connections between chips. This helps build a comprehensive view of the chip network topology.

The utility:

1.   Detects all available chips
2.   For Wormhole chips, gets local coordinates and identifies neighboring chips
3.   Establishes connections between chips
4.   Generates a detailed map of the Ethernet network

## Error Handling

The Wormhole implementation includes comprehensive error handling for various failure conditions:

1.   **ARC Status Errors**:

    *   `ArcReadyError::NoAccess` - Cannot access ARC registers
    *   `ArcReadyError::WatchdogTriggered` - ARC watchdog has triggered
    *   `ArcReadyError::Asleep` - ARC is in sleep mode
    *   `ArcReadyError::BootIncomplete` - ARC boot is not complete
    *   `ArcReadyError::OutstandingPcieDMA` - DMA operation in progress

2.   **Component Initialization Errors**:

    *   DRAM training failures
    *   Ethernet training failures
    *   AXI access errors

3.   **Firmware Version Errors**:

    *   `UnsupportedFwVersion` - Firmware version incompatibility

This structured approach to error handling helps diagnose issues during chip initialization and operation.

## Usage Examples

Working with a Wormhole chip typically involves these steps:

1.   **Chip Detection**: Use `detect_chips()` to find available chips
2.   **Type Identification**: Use `as_wh()` method to check if a chip is a Wormhole
3.   **Initialization**: Wait for initialization to complete
4.   **Communication**: Use ARC messaging, NOC access, or SPI functions
5.   **Network Access**: For multi-chip setups, use `get_neighbouring_chips()` and `open_remote()`
6.   **Telemetry**: Get chip status with `get_telemetry()`

When working with the Ethernet functionality, it's important to ensure that training is complete using `check_ethernet_training_complete()` before attempting communication.
