## Overview

This page documents Whisper's I/O device integration system, which allows simulated RISC-V harts to interact with peripheral devices through memory-mapped interfaces. Memory-mapped devices occupy regions of the physical address space and respond to load/store operations, providing a standardized mechanism for peripheral communication.

For information about interrupt controllers (ACLINT, IMSIC, APLIC) that deliver interrupts to harts, see [7.1](https://deepwiki.com/tenstorrent/whisper/7.1-interrupt-controllers-(aclint-imsic-aplic)). For IOMMU device address translation, see [7.2](https://deepwiki.com/tenstorrent/whisper/7.2-iommu-(inputoutput-memory-management-unit)).

**Key Components:**

*   **Memory-Mapped Registers (MMR)**: Word-aligned registers with configurable masks defined through the PMA system
*   **IoDevice Interface**: Base class that devices implement to handle read/write operations
*   **Device Registration**: Mechanisms to register devices with the memory subsystem
*   **Supported Devices**: UART (8250, sifive), block devices, frame buffers, PCI devices

## Architecture Overview

Memory-mapped devices integrate into Whisper's memory hierarchy through two primary mechanisms: the Physical Memory Attribute (PMA) system for defining register properties, and the `IoDevice` interface for implementing device behavior.

**Sources:**[Memory.hpp 60-517](https://github.com/tenstorrent/whisper/blob/733fd921/Memory.hpp#L60-L517)[PmaManager.hpp 155-517](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.hpp#L155-L517)[System.cpp 49-261](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L49-L261)

## Memory-Mapped Register Interface

Memory-mapped registers are defined through the `PmaManager` class, which associates specific physical addresses with register properties including read/write masks and access constraints.

### MemMappedReg Structure

The `PmaManager` maintains a map of memory-mapped registers, each described by:

| Field | Type | Description |
| --- | --- | --- |
| `value_` | `uint64_t` | Current register value |
| `mask_` | `uint64_t` | Write mask (1 = writable bit) |
| `size_` | `unsigned` | Register size (4 or 8 bytes) |
| `pma_` | `Pma` | Physical memory attributes |

**Sources:**[PmaManager.hpp 479-485](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.hpp#L479-L485)

### Defining Memory-Mapped Registers

Registers are defined using `PmaManager::defineMemMappedReg`:

`bool defineMemMappedReg(uint64_t addr, uint64_t mask, unsigned size, Pma pma);`
*   **addr**: Must be word-aligned (4 bytes) or double-word aligned (8 bytes)
*   **mask**: Bits set to 1 are writable; bits set to 0 are read-only or reserved
*   **size**: Must be 4 (word) or 8 (double-word)
*   **pma**: Physical memory attributes for the register

**Sources:**[PmaManager.cpp 89-100](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.cpp#L89-L100)[PmaManager.hpp 263-265](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.hpp#L263-L265)

### Register Access Operations

The `PmaManager` provides templated read/write methods that enforce alignment and masking:

**Register Read:**

*   Supports byte, half-word, word, and double-word reads
*   Multi-byte reads may span adjacent word-aligned registers
*   Returns current register value after applying any read-side effects

**Register Write:**

*   Only writable bits (mask = 1) are modified
*   Read-only bits remain unchanged
*   Expression: `value = (value & ~mask & ~write_mask) | (new_value & write_mask)`

**Sources:**[PmaManager.cpp 136-231](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.cpp#L136-L231) (read), [PmaManager.cpp 235-354](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.cpp#L235-L354) (write), [PmaManager.hpp 395-461](https://github.com/tenstorrent/whisper/blob/733fd921/PmaManager.hpp#L395-L461)

### Configuration from JSON

Memory-mapped registers can be configured via JSON configuration files:

The `processMemMappedMasks` function parses JSON entries containing:

*   **attribs**: Array of PMA attributes (e.g., "read", "write", "mem_mapped")
*   **masks**: Array of triplets `[lowAddr, highAddr, mask]` defining register ranges
*   **size**: Register size (4 or 8 bytes)

**Sources:**[HartConfig.cpp 1260-1308](https://github.com/tenstorrent/whisper/blob/733fd921/HartConfig.cpp#L1260-L1308)[HartConfig.cpp 1218-1254](https://github.com/tenstorrent/whisper/blob/733fd921/HartConfig.cpp#L1218-L1254)

## IoDevice Base Class

The `IoDevice` class provides the abstract interface that all memory-mapped devices implement. Devices respond to memory accesses within their configured address ranges.

### IoDevice Interface

| Method | Purpose |
| --- | --- |
| `isAddressInRange(uint64_t addr)` | Check if address falls within device range |
| `read(uint64_t addr)` | Read device register at address |
| `write(uint64_t addr, T value)` | Write value to device register |
| `type()` | Return device type string |
| `enable()` / `disable()` | Enable/disable device operation |
| `saveSnapshot()` / `loadSnapshot()` | Snapshot support for device state |

## Device Integration

Devices are registered with the `Memory` object through the `System` class, which maintains a list of all active devices.

### Device Registration

Devices are registered in two locations:

1.   **Memory::ioDevs_**: Used during memory access operations to route reads/writes
2.   **System::ioDevs_**: Used for lifecycle management (enable/disable, snapshots)

**Sources:**[System.cpp 103-208](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L103-L208)[Memory.hpp 464-465](https://github.com/tenstorrent/whisper/blob/733fd921/Memory.hpp#L464-L465)

### Device Lifecycle

The `System` class manages device lifecycle:

**Creation:**

*   Devices created via `System::defineUart()`, `System::defineFrameBuffer()`, etc.
*   Device constructors typically receive base address, size, and channel configuration

**Operation:**

*   Devices respond to memory accesses automatically
*   May generate interrupts via APLIC (e.g., UART interrupt on data ready)

**Snapshot Support:**

*   `System::saveSnapshot()` iterates devices and calls `device->saveSnapshot()`
*   Device state serialized to filesystem for later restoration

**Sources:**[System.cpp 559-698](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L559-L698) (snapshot save), [System.cpp 749-916](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L749-L916) (snapshot load)

## UART Support

Whisper supports two UART device types for serial communication with simulated programs.

### UART Types

| Type | Class | Description |
| --- | --- | --- |
| `uart8250` | `Uart8250` | Industry-standard 16550-compatible UART |
| `uartsf` | `Uartsf` | SiFive-specific UART implementation |

### UART Configuration

UARTs are defined via `System::defineUart()`:

`bool defineUart(const std::string& type, uint64_t addr, uint64_t size,                uint32_t iid, const std::string& channel,                 unsigned regShift = 2);`
**Parameters:**

*   **type**: "uart8250" or "uartsf"
*   **addr**: Base address in physical memory
*   **size**: Size of UART register region
*   **iid**: Interrupt identifier for APLIC
*   **channel**: Channel specification (see below)
*   **regShift**: Register address shift (default 2 = word-aligned)

**Sources:**[System.cpp 103-208](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L103-L208)[System.hpp 269-273](https://github.com/tenstorrent/whisper/blob/733fd921/System.hpp#L269-L273)

### UART Channel Types

UART channels connect the simulated UART to host I/O:

**Channel Specifications:**

*   **stdio**: Direct connection to simulator stdin/stdout
*   **pty**: Allocates a pseudo-terminal (accessible via /dev/pts/N)
*   **unix:<path>**: Creates Unix domain socket at specified path
*   **Semicolon-separated**: Fork to multiple channels (e.g., "stdio;pty")

**Channel Implementation:** The `createChannel` lambda function recursively parses channel specifications:

`auto createChannel = [](std::string_view channel_type) -> std::unique_ptr<UartChannel> {  // Handle fork (semicolon-separated)  auto pos = channel_type.find(';');  if (pos != std::string::npos) {    // Create read-write and write-only channels    return std::make_unique<ForkChannel>(...);  }  // Handle individual channel types  if (channel_type == "stdio") return std::make_unique<FDChannel>(...);  if (channel_type == "pty") return std::make_unique<PTYChannel>();  if (channel_type.find("unix:") == 0) { /* socket setup */ }};`
**Sources:**[System.cpp 109-187](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L109-L187)

### Uart8250 Specifics

The `Uart8250` class implements a 16550-compatible UART:

*   **Register Offset**: Configurable via `regShift` (typically 2 for word-alignment)
*   **Interrupts**: Generates APLIC interrupts when data is available or transmit buffer empty
*   **Channel**: Configurable I/O channel for host communication

**Constructor:**

`Uart8250(uint64_t addr, uint64_t size, std::shared_ptr<TT_APLIC::Aplic> aplic,         uint32_t iid, std::unique_ptr<UartChannel> channel,          bool hasExtInterrupt, unsigned regShift);`

## Other Device Types

### Block Devices (VirtIO)

Whisper supports VirtIO block devices for disk I/O:

**Class:**`pci::virtio::Blk`

*   Implements VirtIO block device specification
*   Accessible via PCI bus
*   Supports read/write operations on backing storage

**Integration:** Block devices are registered through the PCI subsystem rather than directly as IO devices.

### Frame Buffer (Remote Display)

When compiled with `REMOTE_FRAME_BUFFER` support, Whisper can define frame buffer devices:

`bool defineFrameBuffer(const std::string& type, uint64_t addr,                        uint64_t width, uint64_t height,                       uint64_t bytes_per_pixel, int port);`
**Type:**`rfb` (Remote Frame Buffer) **Class:**`RemoteFrameBuffer`

*   Memory-mapped frame buffer accessible to simulated programs
*   Streams frame buffer contents over network for display
*   Configured with dimensions and pixel format

**Sources:**[System.cpp 212-229](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L212-L229)[System.hpp 283-288](https://github.com/tenstorrent/whisper/blob/733fd921/System.hpp#L283-L288)

### PCI Device Support

Whisper provides PCI host-root-complex emulation for PCI-based devices:

`bool configPci(uint64_t configBase, uint64_t mmioBase,                uint64_t mmioSize, unsigned buses, unsigned slots);`
**Configuration:**

*   **configBase**: PCI configuration space base address
*   **mmioBase**: MMIO region base address for PCI devices
*   **mmioSize**: Size of MMIO region
*   **buses**: Number of PCI buses
*   **slots**: Number of slots per bus

Devices are added via `System::addPciDevices()` with device specifications.

## Configuration from JSON

Memory-mapped devices and registers can be fully configured via JSON configuration files processed by the `HartConfig` class.

### Memory-Mapped Region Configuration

JSON structure for defining memory-mapped regions:

`{  "memmap": [    {      "addr": "0x10000000",      "size": "0x1000",      "attribs": ["read", "write", "mem_mapped"],      "masks": [        ["0x10000000", "0x10000004", "0xffffffff"]      ],      "size": 4    }  ]}`
### Configuration Processing Flow

**Processing Steps:**

1.   **Parse PMA Attributes**: `getConfigPma()` converts attribute strings to `Pma::Attrib` flags

    *   Valid attributes: "read", "write", "exec", "mem_mapped", "amo", "rsrv", "idempotent", "io"

2.   **Define PMA Region**: `PmaManager::defineRegion()` creates a memory region with specified attributes

    *   Regions checked in order; first matching region applies

3.   **Process Masks**: `processMemMappedMasks()` creates memory-mapped registers

    *   Each mask entry defines register address range and writable bits
    *   Registers inherit PMA attributes from parent region

**Sources:**[HartConfig.cpp 1218-1254](https://github.com/tenstorrent/whisper/blob/733fd921/HartConfig.cpp#L1218-L1254) (getConfigPma), [HartConfig.cpp 1260-1308](https://github.com/tenstorrent/whisper/blob/733fd921/HartConfig.cpp#L1260-L1308) (processMemMappedMasks), [HartConfig.cpp 1330-1513](https://github.com/tenstorrent/whisper/blob/733fd921/HartConfig.cpp#L1330-L1513) (applyMemoryConfig)

### UART Configuration Example

While UARTs are typically defined programmatically via `System::defineUart()`, the memory regions they occupy can be configured via JSON:

`{  "memmap": [    {      "addr": "0x10000000",      "size": "0x1000",      "attribs": ["read", "write", "io"]    }  ]}`
The UART device is then instantiated in code:

`system.defineUart("uart8250", 0x10000000, 0x1000,                   iid, "stdio", 2);`
**Sources:**[System.cpp 103-208](https://github.com/tenstorrent/whisper/blob/733fd921/System.cpp#L103-L208)
