The BOOM processor implements a sophisticated branch prediction system to reduce pipeline stalls caused by control flow changes. This document describes the branch prediction architecture, components, and mechanisms used to predict and resolve branches in the BOOM core.

For information about the instruction fetch pipeline, see [Instruction Fetch Pipeline](https://deepwiki.com/tenstorrent/riscv-ocelot/4-instruction-fetch-pipeline).

## Branch Prediction Architecture Overview

BOOM employs a two-level branch prediction system:

1.   **Next-Line Predictor (NLP)**: A fast, single-cycle predictor that provides quick predictions for the next instruction fetch.
2.   **Backing Predictor (BPD)**: A more sophisticated predictor that provides higher accuracy but requires multiple cycles.

Sources: [src/main/scala/ifu/frontend.scala 332-334](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/frontend.scala#L332-L334)[docs/sections/branch-prediction/index.rst 1-37](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/branch-prediction/index.rst#L1-L37)

## Next-Line Predictor (NLP)

The Next-Line Predictor is a combination of three components that work together to make fast, single-cycle predictions:

1.   **Branch Target Buffer (BTB)**: Stores branch target addresses
2.   **Return Address Stack (RAS)**: Handles function call/return prediction
3.   **Bi-Modal Table (BIM)**: Provides branch direction prediction (taken/not-taken)

The NLP makes a prediction based on the current fetch PC:

1.   The BTB is checked for a tag match
2.   If a match is found, the branch type is determined (branch, jump, or return)
3.   For branches, the BIM determines if the branch is taken
4.   For returns, the RAS provides the target address
5.   For jumps, the target from the BTB is used directly

Sources: [src/main/scala/ifu/bpd/btb.scala 23-197](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/bpd/btb.scala#L23-L197)[docs/sections/branch-prediction/nl-predictor.rst 1-115](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/branch-prediction/nl-predictor.rst#L1-L115)

## Backing Predictor (BPD)

The Backing Predictor provides more accurate predictions but takes multiple cycles. BOOM supports multiple BPD implementations:

| Predictor Type | Description | Configuration |
| --- | --- | --- |
| TAGE-L | Tagged Geometric Length predictor | Default predictor |
| GShare | Uses global history to index counters | Alternative predictor |
| Alpha21264 | Tournament predictor with local/global history | Alternative predictor |
| Software | Interface for custom predictors | For experimentation |

The BPD makes predictions throughout the fetch stages and can override the NLP's prediction:

The TAGE-L predictor is the default and most sophisticated predictor, using multiple tables of different history lengths to make predictions.

Sources: [src/main/scala/ifu/bpd/tage.scala 16-360](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/bpd/tage.scala#L16-L360)[src/main/scala/common/config-mixins.scala 462-489](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/config-mixins.scala#L462-L489)[src/main/scala/ifu/bpd/predictor.scala 14-373](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/bpd/predictor.scala#L14-L373)

## Global History Register (GHR)

The Global History Register (GHR) tracks the outcome of previous branches to improve prediction accuracy:

Key characteristics of GHR management:

*   Updated speculatively during fetch when a branch is predicted
*   Reset and corrected during execute when a misprediction is detected
*   Committed version maintained for recovery from exceptions

Sources: [src/main/scala/ifu/frontend.scala 45-127](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/frontend.scala#L45-L127)[docs/sections/branch-prediction/backing-predictor.rst 99-128](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/branch-prediction/backing-predictor.rst#L99-L128)

## Fetch Target Queue (FTQ)

The Fetch Target Queue (FTQ) maintains a record of all in-flight branch predictions:

The FTQ serves several purposes:

*   Stores branch prediction metadata for in-flight instructions
*   Provides PC information to execution units
*   Used to update branch predictors after branch resolution
*   Deallocated entries when all instructions in a fetch packet are committed

Sources: [src/main/scala/ifu/fetch-target-queue.scala 12-365](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/fetch-target-queue.scala#L12-L365)[docs/sections/branch-prediction/backing-predictor.rst 130-168](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/branch-prediction/backing-predictor.rst#L130-L168)

## Branch Prediction Pipeline Integration

The branch prediction system is integrated throughout the BOOM pipeline:

Key branch prediction events in the pipeline:

1.   **F0-F1**: NLP makes a prediction for the next fetch address
2.   **F1-F3**: BPD makes its prediction and may override the NLP
3.   **F3**: Frontend may be redirected based on BPD prediction
4.   **Execute**: Branch resolution occurs, mispredictions trigger pipeline flushes
5.   **Commit**: Branch predictors are updated with final branch outcomes

Sources: [src/main/scala/ifu/frontend.scala 344-507](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/frontend.scala#L344-L507)[src/main/scala/ifu/frontend.scala 816-965](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/frontend.scala#L816-L965)

## Branch Misprediction Recovery

When a branch misprediction is detected in the execute stage, the processor must recover:

The misprediction recovery process involves:

1.   Detecting the misprediction in the Branch Unit
2.   Flushing the pipeline of incorrect-path instructions
3.   Redirecting the frontend to fetch from the correct path
4.   Restoring the speculative GHR to its correct state
5.   Restoring the rename map to its state at the branch

Sources: [src/main/scala/ifu/frontend.scala 954-1037](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/ifu/frontend.scala#L954-L1037)

## Branch Predictor Configuration

BOOM provides several configuration options for its branch predictors:

| Configuration | Description |
| --- | --- |
| WithTAGELBPD | Default TAGE-L predictor configuration |
| WithBoom2BPD | Simplified GShare-based predictor |
| WithAlpha21264BPD | Tournament predictor similar to Alpha 21264 |
| WithSWBPD | Software-based predictor interface |

These configurations can be selected when building the processor, allowing for different performance and area tradeoffs.

Sources: [src/main/scala/common/config-mixins.scala 462-564](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/config-mixins.scala#L462-L564)

## Conclusion

BOOM's branch prediction system is a sophisticated multi-level design that balances speed and accuracy. The NLP provides fast single-cycle predictions while the BPD provides higher accuracy predictions with a longer latency. The FTQ and GHR management systems ensure that branch predictions are tracked, updated, and recovered correctly throughout the pipeline.
