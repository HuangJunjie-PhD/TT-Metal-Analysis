## Purpose and Scope

This document describes tt-torch's comprehensive hardware matrix testing infrastructure, which validates model execution across multiple Tenstorrent hardware architectures and parallelism configurations. The system uses GitHub Actions workflows, pytest parameterization, and a centralized configuration system to execute tests on different hardware types with architecture-specific overrides.

For information about the testing framework itself (ModelTester, verification system), see [4.2.1](https://deepwiki.com/tenstorrent/tt-torch/4.2.1-model-testing-framework). For test configuration and management details, see [4.2.2](https://deepwiki.com/tenstorrent/tt-torch/4.2.2-test-configuration-and-management).

## Hardware Architecture Support Matrix

tt-torch tests models across multiple Tenstorrent hardware architectures, each with different capabilities and configurations. The following table summarizes the supported hardware:

| Architecture | Runner Name | Type | Memory Variant | Primary Use Case |
| --- | --- | --- | --- | --- |
| Wormhole B0 | `wormhole_b0` | Single-chip | Standard | General model testing |
| Wormhole B0 | `high-memory-wormhole` | Single-chip | High memory | Large models (Phi-2, Mamba) |
| Blackhole | `p150` | Single-chip | Standard | Next-gen architecture validation |
| N300 | `n300` | Multi-chip (2x4) | Standard | Batch/pipeline parallelism |
| N300 | `high-memory-n300` | Multi-chip (2x4) | High memory | Large model pipeline parallel |
| LLMBox | `llmbox` | Multi-chip | Custom | Tensor parallelism for LLMs |

## Test Execution Modes and Parallelism

Tests can be executed in different modes depending on the model characteristics and hardware configuration:

### Execution Mode Matrix

| Mode | Description | Hardware | CompilerConfig Setting |
| --- | --- | --- | --- |
| `single_device` | Single chip execution | wormhole_b0, p150 | `data_parallel_mode=False` |
| `data_parallel` | Batch distributed across chips | wormhole_b0, p150, n300 | `data_parallel_mode=True` |
| `batch_parallel` | N300-specific batch distribution | n300 | `automatic_parallelization=True` `mesh_shape=[1,2]` |
| `pipeline_parallel` | Multi-stage model execution | high-memory-n300 | Custom split configuration |
| `tensor_parallel` | Tensor sharding for LLMs | llmbox | XLA SPMD configuration |

**Compilation Depth Options:**

| Value | Description | Usage |
| --- | --- | --- |
| `CompileDepth.EXECUTE` | Full compilation to binary | Production, performance testing |
| `CompileDepth.EXECUTE_OP_BY_OP` | Per-operation compilation | Debugging, verification |
| `CompileDepth.TORCH_FX` | Stop at FX graph capture | Graph inspection |
| `CompileDepth.STABLEHLO` | Stop at StableHLO IR | IR-level debugging |
| `CompileDepth.TTNN_IR` | Stop at TTNN IR | Hardware IR inspection |

### Matrix Strategy and Job Distribution

The workflows use GitHub Actions matrix strategy to parallelize test execution:

**Nightly Full Model Execution (example matrix configuration):**

`strategy:  fail-fast: false  matrix:    runner: [wormhole, blackhole]    build: [      {wh-runner: wormhole_b0, bh-runner: p150, name: "eval_1", tests: "..."},      {wh-runner: n300, name: "eval_1_data_parallel", tests: "..."},      {wh-runner: high-memory-wormhole, bh-runner: p150, name: "eval_7_high_mem", pytest-args: "--forked", tests: "..."}    ]`
The matrix creates jobs for:

*   Each runner type (wormhole, blackhole)
*   Each build configuration (test group)
*   Approximately 10-15 parallel jobs per workflow run

**ModelStatus Enum:**

`class ModelStatus(Enum):    EXPECTED_PASSING = "expected_passing"    KNOWN_FAILURE_XFAIL = "known_failure_xfail"    NOT_SUPPORTED_SKIP = "not_supported_skip"    UNSPECIFIED = "unspecified"`

Configuration for Phi-2 model showing architecture-specific overrides:

`"phi2/causal_lm/pytorch-microsoft/phi-2-full-eval": {    "status": ModelStatus.EXPECTED_PASSING,    "assert_pcc": False,    "arch_overrides": {        "wormhole": {            "status": ModelStatus.NOT_SUPPORTED_SKIP,            "skip_reason": "Out of host memory - issue #1224",            "skip_bringup_status": "FAILED_RUNTIME",        },    },}`

## Test Categorization and Filtering

### pytest Markers and Filtering

Tests are marked based on their `ModelStatus` and can be filtered in CI:

| Marker | Description | CI Usage |
| --- | --- | --- |
| `expected_passing` | Models that should pass | Primary nightly/PR tests |
| `known_failure_xfail` | Expected failures with tickets | Tracked but not blocking |

## Container Configuration for Hardware Testing

All hardware tests run in Docker containers with specific device access and volume mounts:

`container:  image: ${{ inputs.docker-image }}  options: --user root --device /dev/tenstorrent/0 --shm-size=4gb  volumes:    - /dev/hugepages:/dev/hugepages    - /dev/hugepages-1G:/dev/hugepages-1G    - /etc/udev/rules.d:/etc/udev/rules.d    - /lib/modules:/lib/modules    - /mnt/dockercache:/mnt/dockercache`
**Key configurations:**

*   `--device /dev/tenstorrent/0`: Device access (single chip) or `--device /dev/tenstorrent` (multi-chip)
*   `--shm-size=4gb`: Shared memory for inter-process communication
*   `/dev/hugepages`: Huge page support for memory efficiency
*   `/mnt/dockercache`: Persistent cache for model downloads (HF_HOME, TORCH_HOME)

**Multi-chip device access:**

For N300 and LLMBox tests, the device option changes to allow access to all chips:

`options: --user root --device /dev/tenstorrent --shm-size=4gb`
**Sources:**[.github/workflows/run-full-model-execution-tests-nightly.yml 323-332](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-full-model-execution-tests-nightly.yml#L323-L332)[.github/workflows/run-full-model-execution-tests.yml 165-174](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-full-model-execution-tests.yml#L165-L174)

## Test Result Artifacts and Reporting

Each workflow job produces multiple artifacts for debugging and analysis:

| Artifact | Content | Usage |
| --- | --- | --- |
| `test-log-{runner}-{group}-{job_id}` | Complete pytest output | Debugging test failures |
| `memory-usage-{runner}-{group}-{job_id}` | CSV with memory consumption | Performance analysis |
| `test-reports-models-{runner}-{group}-{job_id}` | JUnit XML report | CI test result display |
| `model-mlir-{workflow}-{runner}-{group}-{job_id}` | STABLEHLO, TTIR, TTNN IRs | Compiler debugging |

**MLIR artifact collection:**

`TT_TORCH_SAVE_MLIR=STABLEHLO,TTIR,TTNN pytest ...`
This saves intermediate compilation artifacts to `model_mlir/` directory, which gets uploaded as an artifact.

**Artifact merging:**

Multi-job workflows merge artifacts using a separate job:

`merge_mlir_artifacts:  needs: [tests]  if: ${{ always() && inputs.run-dump-mlir == 'true' }}  runs-on: ubuntu-latest  steps:    - name: Merge Artifacts      uses: actions/upload-artifact/merge@v4      with:        name: model-mlir-execute-nightly        pattern: model-mlir-execute-nightly-*        delete-merged: true`
