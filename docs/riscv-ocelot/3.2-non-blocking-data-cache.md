The Non-Blocking Data Cache (NBDC) in the BOOM RISC-V processor is a key component of the memory hierarchy that provides fast access to frequently used data while supporting high memory-level parallelism. Unlike a blocking cache that stalls on a miss, this non-blocking implementation can continue to service other requests while waiting for outstanding misses to be resolved, significantly improving performance in the presence of cache misses.

This document explains the architecture and operation of the Non-Blocking Data Cache. For information about the Load/Store Unit that interfaces with this cache, see [Load/Store Unit](https://deepwiki.com/tenstorrent/riscv-ocelot/3.1-loadstore-unit).

## Architecture Overview

The BoomNonBlockingDCache is instantiated as part of the BOOM tile and provides the data cache functionality for the processor. It connects to the memory system through a TileLink interface and interfaces with the Load/Store Unit (LSU) through a custom interface.

## Core Components

The Non-Blocking Data Cache consists of several key components that work together to provide efficient data access:

### Meta and Data Arrays

The cache uses separate arrays for metadata (tags, coherence state) and data:

The data array can be configured in two different ways:

*   **Duplicated Data Array**: Each way has a complete copy of the data, allowing parallel access.
*   **Banked Data Array**: The data is split into banks that can be accessed in parallel.

## Non-Blocking Operation

The key feature of this cache is its ability to handle multiple outstanding misses. This is enabled by the MSHR system, which tracks and manages multiple in-flight cache misses.

When a request misses in the cache:

1.   An MSHR is allocated to track the miss
2.   The MSHR sends a request to memory through TileLink
3.   Meanwhile, the cache continues to service other requests
4.   When the memory response arrives, the MSHR refills the cache
5.   Finally, the original request is completed or replayed

Secondary misses (requests to a line that already has an MSHR allocated) are handled by either:

*   Queuing the request in the existing MSHR (if permissions are compatible)
*   Blocking the request and triggering a replay (if different permissions are needed)

## Banked vs. Duplicated Data Arrays

The data array can be configured in two ways:

### Duplicated Data Array

*   Each way has a separate array copy for each memory width
*   Simpler implementation but higher area cost
*   Good for narrow caches with few ways

```
for (j <- 0 until memWidth) {
  for (w <- 0 until nWays) {
    val array = DescribedSRAM(...)
    when (io.write.bits.way_en(w) && io.write.valid) {
      array.write(waddr, data, io.write.bits.wmask.asBools)
    }
    io.resp(j)(w) := RegNext(array.read(...))
  }
}
```

Sources: [src/main/scala/lsu/dcache.scala 281-302](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/dcache.scala#L281-L302)

### Banked Data Array

*   Data array is divided into banks that can be accessed in parallel
*   More complex but more area-efficient
*   Better for wide caches with many ways

```
for (i <- 0 until nWays) {
  val s0_ren = s0_valid
  val wen = (refill_one_beat && !invalidated) && repl_way === i.U
  
  var mem_idx0: UInt = null
  var mem_idx1: UInt = null
  
  // Calculate bank indices...
  
  when (wen) {
    dataArraysB0(i).write(mem_idx0, data(wordBits/2-1, 0))
    dataArraysB1(i).write(mem_idx1, data(wordBits-1, wordBits/2))
  }
}
```

## TileLink Interface

The Non-Blocking Data Cache interfaces with the memory system through TileLink:

The cache uses TileLink to:

*   Request cache lines from memory (Acquire)
*   Write back dirty lines (Release)
*   Respond to coherence requests (ProbeAck)
*   Receive requested data (Grant)
*   Acknowledge received data (GrantAck)

## Conclusion

The Non-Blocking Data Cache is a sophisticated component that provides high-performance data access for the BOOM processor. Its non-blocking design allows multiple outstanding misses to be handled in parallel, enabling the processor to continue executing other instructions even in the presence of cache misses. The modular architecture with separate meta and data arrays, configurable as banked or duplicated, provides flexibility in designing for different performance and area requirements.

The integration with the Load/Store Unit, TLB, and vector processing support makes it a versatile cache implementation suitable for high-performance RISC-V processors like BOOM.
