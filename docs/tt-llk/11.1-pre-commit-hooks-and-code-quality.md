## Pre-commit Framework Overview

The repository uses the [pre-commit](https://pre-commit.com/) framework to automate code quality checks before commits are finalized. The framework executes a series of hooks that format code, check for common issues, and enforce project standards across both C++ and Python files.

**Key Characteristics:**

*   **Multi-language support**: Handles C++, Python, YAML, Markdown, and other file types
*   **Automatic execution**: Runs on `git commit` after local installation
*   **CI enforcement**: GitHub Actions validates all PRs using the same hooks [.github/workflows/pre-commit.yml 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/pre-commit.yml#L1-L68)
*   **Auto-fixing**: Many hooks automatically correct issues (formatting, trailing whitespace, etc.)
*   **Manual execution**: Can be run on-demand via `pre-commit run --all-files`

## Hook Configuration and Checks

The pre-commit framework executes multiple hooks that cover different aspects of code quality. While the `.pre-commit-config.yaml` file is not provided, the documentation references the following tools:

| Tool | Purpose | Language | Auto-fix |
| --- | --- | --- | --- |
| **clang-format** | C++ code formatting according to `.clang-format` | C++ | Yes |
| **black** | Python code formatting (PEP 8 style) | Python | Yes |
| **mypy** | Python static type checking | Python | No |
| **codespell** | Spell checking across all files and commit messages | All | Yes |
| **trailing-whitespace** | Remove trailing whitespace | All | Yes |
| **end-of-file-fixer** | Ensure files end with newline | All | Yes |
| **check-yaml** | Validate YAML syntax | YAML | No |
| **check-merge-conflict** | Detect unresolved merge conflicts | All | No |

**clang-format Configuration:**

The C++ formatting style is defined in `.clang-format`[CONTRIBUTING.md 128-129](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L128-L129) The `.clangd` configuration file supplements this with compiler flags for IDE integration [.clangd 1-3](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.clangd#L1-L3):

```
CompileFlags:
  Add: [-std=c++17, -xc++, -ferror-limit=100]
  CompilationDatabase: .
```

**codespell Ignore List:**

False positives can be added to `.codespellignore` to prevent valid words from being flagged [CONTRIBUTING.md 218-228](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L218-L228)

## Local Development Setup

Pre-commit hooks are automatically installed when developers run the testing environment setup script:

`cd tests./setup_testing_env.sh`
This script installs the pre-commit framework and configures git hooks [CONTRIBUTING.md 148-156](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L148-L156)

**Automated Installation via setup_testing_env.sh:**

The `setup_precommit()` function in [tests/setup_testing_env.sh 144-170](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L144-L170) handles pre-commit installation:

`# Function to setup pre-commit hookssetup_precommit() {    # Navigate to repository root    local repo_root="${SCRIPT_DIR}/.."        # Check if in git repository    if ! git -C "$repo_root" rev-parse --git-dir >/dev/null 2>&1; then        echo "WARNING: Not in a git repository, skipping pre-commit setup"        return    fi        # Install pre-commit if not available    if ! command -v pre-commit >/dev/null 2>&1; then        pip install pre-commit    fi        # Install hooks    (cd "$repo_root" && pre-commit install)}`
This function is called automatically in the main setup flow [tests/setup_testing_env.sh 203](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L203-L203)

**Manual Installation Process:**

**Manual Setup Commands:**

1.   **Install pre-commit**: `pip install pre-commit` (included in virtual environment)
2.   **Install git hooks**: `pre-commit install` (configures `.git/hooks/pre-commit`)
3.   **Verify setup**: `pre-commit run --all-files` (optional, checks all files)

**Commit Workflow with Pre-commit:**

When a developer runs `git commit`, the following occurs:

1.   Pre-commit framework executes all configured hooks
2.   If hooks make changes (e.g., formatting), commit is aborted
3.   Developer must `git add` the modified files
4.   Developer re-runs `git commit`
5.   Commit succeeds if all hooks pass

**Sources:**[tests/setup_testing_env.sh 144-170](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L144-L170)[tests/setup_testing_env.sh 203](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L203-L203)

## CI/CD Integration

The pre-commit checks are enforced in GitHub Actions via the `pre-commit.yml` workflow, which is called by the main `on-pr.yml` workflow as the first validation step.

**Workflow Invocation from on-pr.yml:**

The pre-commit job is defined in [.github/workflows/on-pr.yml 28-30](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L28-L30):

`jobs:  pre-commit:    name: "🔍 Static checks"    uses: ./.github/workflows/pre-commit.yml`
This job must complete successfully before other jobs run. The `detect-changes` job explicitly depends on it [.github/workflows/on-pr.yml 35](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L35-L35):

`detect-changes:  name: "🔍 Detect changes"  runs-on: ubuntu-latest  needs: pre-commit  # Blocks on pre-commit completion`
**Workflow Structure:**

**Key Implementation Details:**

1.   **Workflow call**: Reusable workflow accepting `python_version` parameter (default: "3.10")
2.   **Checkout configuration**: `fetch-depth: 0` ensures full git history for range comparison
3.   **Pre-commit action**: Uses `pre-commit/action@v3.0.1` with `--from-ref origin/main --to-ref HEAD` to check only changed files
4.   **PR comment job**: Posts welcome message to first-time contributors 
    *   Only runs for non-fork PRs (`github.event.pull_request.head.repo.full_name == github.repository`)
    *   Checks for existing comment before posting to avoid duplicates
    *   Includes information about `metal-post-commit-tests` label

**Concurrency Control:**

The workflow uses concurrency groups to cancel obsolete runs [.github/workflows/on-pr.yml 17-25](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L17-L25):

`concurrency:  # Use github.run_id on main branch (no cancellation)  # Use github.event.pull_request.number on PRs (cancel old runs)  group: ${{ github.workflow }}-${{ github.ref_protected && github.run_id ||    github.event.pull_request.number || github.ref }}  cancel-in-progress: true`
**Failure Handling:**

When pre-commit checks fail in CI:

*   The workflow job fails, blocking all downstream jobs
*   GitHub Actions UI shows which specific hook failed
*   The `check-all-green` aggregation job ensures pre-commit must pass [.github/workflows/on-pr.yml 207-223](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L207-L223)
*   Developers must fix issues locally and push new commits
*   CI automatically re-runs on new pushes

**Sources:**[.github/workflows/on-pr.yml 17-30](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L17-L30)[.github/workflows/on-pr.yml 35](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L35-L35)[.github/workflows/on-pr.yml 207-223](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L207-L223)

## IDE Integration with clangd

The repository provides `clangd` language server integration for real-time code checking and completion in IDEs.

**Setup Process:**

**Setup Commands**[CONTRIBUTING.md 190-217](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L190-L217):

`# 1. Install clangd extension in your IDE # 2. Download dependenciescd tests./setup_testing_env.shcd .. # 3. Generate compilation flags for specific architecture./setup_clangd.sh blackhole  # or wormhole, quasar # 4. Reload file in editor (if needed)`
**Generated Configuration:**

*   **compile_flags.txt**: Contains architecture-specific compiler flags, include paths, and definitions (generated by `setup_clangd.sh`)
*   **.clangd**: Contains base compiler configuration [.clangd 1-3](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.clangd#L1-L3): 
    *   `-std=c++17`: C++17 language standard
    *   `-xc++`: Treat files as C++
    *   `-ferror-limit=100`: Limit error messages
    *   `CompilationDatabase: .`: Look for `compile_flags.txt` in repository root

**Sources:**[tests/setup_testing_env.sh 144-170](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L144-L170)

## Spell Checking with codespell

The `codespell` tool automatically detects and fixes common spelling errors in source code, documentation, and commit messages.

**Codespell Workflow:**

**Handling False Positives**[CONTRIBUTING.md 218-228](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L218-L228):

When `codespell` incorrectly flags a valid word:

1.   Open `.codespellignore` in repository root
2.   Add the word on a new line
3.   Commit the updated ignore file
4.   Re-run pre-commit hooks

**Example:**

```
# .codespellignore
tensix
unpacker
matmul
sfpu
```

**Sources:**[CONTRIBUTING.md 218-228](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L218-L228)

## Git Blame Configuration

To maintain clean `git blame` output after mass formatting changes, the repository uses `.git-blame-ignore-revs` to exclude specific commits.

**Configuration Setup:**

`# Configure git to use the ignore filegit config blame.ignoreRevsFile .git-blame-ignore-revs # Or globally for all repositoriesgit config --global blame.ignoreRevsFile .git-blame-ignore-revs`
**Usage:**

After configuration, `git blame <file>` automatically skips commits listed in `.git-blame-ignore-revs`, providing a clearer view of meaningful changes [CONTRIBUTING.md 230-246](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L230-L246)

**Commit Selection:**

The `.git-blame-ignore-revs` file typically contains:

*   Mass formatting commits (initial clang-format application)
*   Automated refactoring commits
*   Whitespace-only changes

**Sources:**[CONTRIBUTING.md 230-246](https://github.com/tenstorrent/tt-llk/blob/366f58e2/CONTRIBUTING.md?plain=1#L230-L246)

## Summary: Code Quality Enforcement Points

The code quality system operates at multiple enforcement points to ensure consistency:

| Enforcement Point | Mechanism | When | Blocking |
| --- | --- | --- | --- |
| **Local commits** | Pre-commit hooks via `.git/hooks/pre-commit` | On `git commit` | Yes (aborts commit) |
| **Manual checks** | `pre-commit run --all-files` | Developer-initiated | No (informational) |
| **CI validation** | GitHub Actions `pre-commit.yml` workflow | On PR creation/update | Yes (blocks merge) |
| **IDE integration** | clangd language server with `.clangd` config | Real-time while editing | No (warnings only) |
