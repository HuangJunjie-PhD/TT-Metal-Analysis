The Virtual Machine Executor provides a register-based bytecode interpreter for executing compiled Relay models with support for dynamic shapes, control flow, closures, and multi-device execution. It is one of three execution backends in TVM, alongside the Graph Executor ([7.1](https://deepwiki.com/tenstorrent/tt-tvm/7.1-graph-executor)) and AOT Executor ([7.3](https://deepwiki.com/tenstorrent/tt-tvm/7.3-aot-executor)).

For information about compiling models to VM bytecode, see VM Backend ([6.7](https://deepwiki.com/tenstorrent/tt-tvm/6.7-virtual-machine-backend)). For profiling VM execution, see Profiling System ([7.5](https://deepwiki.com/tenstorrent/tt-tvm/7.5-profiling-system)).

## Architecture Overview

The VM executor consists of three major components: the compiler that generates bytecode, the executable container, and the runtime interpreter.

**Diagram: VM Executor Architecture**

The `VMCompiler` ([src/relay/backend/vm/compiler.cc 240-826](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L240-L826)) translates Relay functions into sequences of VM instructions. Each compiled function becomes a `VMFunction` containing bytecode, parameter metadata, and register allocation information. The `Executable` ([include/tvm/runtime/vm/executable.h 61-159](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/runtime/vm/executable.h#L61-L159)) packages these functions with constants, device mappings, and primitive function references. At runtime, the `VirtualMachine` ([include/tvm/runtime/vm/vm.h 112-277](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/runtime/vm/vm.h#L112-L277)) interprets bytecode using a register-based execution model with call stack management.

## Register-Based Execution Model

The VM uses a register machine architecture where each function has a fixed-size register file and instructions operate on register operands.

**Diagram: VM Register Machine State**

Each `VMFrame` ([src/runtime/vm/vm.cc 399-413](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L399-L413)) maintains its own register file, return address, and saved execution state. The `register_file` is a vector of `ObjectRef` that can hold any TVM runtime object (NDArray, ADT, Closure, etc.). Instructions reference registers by index, and the compiler ([src/relay/backend/vm/compiler.cc 301-334](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L301-L334)) allocates registers as needed during code generation.

## Instruction Set

The VM instruction set supports tensor operations, control flow, memory allocation, and function calls.

| Category | Instructions | Description |
| --- | --- | --- |
| **Data Movement** | `Move`, `LoadConst`, `LoadConsti` | Move data between registers, load constants |
| **Memory Allocation** | `AllocTensor`, `AllocTensorReg`, `AllocStorage` | Allocate tensor storage and views |
| **Shape Operations** | `ShapeOf`, `ReshapeTensor` | Dynamic shape introspection and reshaping |
| **Control Flow** | `If`, `Goto`, `Ret` | Conditional branching, jumps, function return |
| **Function Calls** | `Invoke`, `InvokePacked`, `InvokeClosure` | Call VM functions, primitives, closures |
| **ADT Operations** | `AllocADT`, `GetField`, `GetTag` | Algebraic data type construction and access |
| **Device Operations** | `DeviceCopy` | Cross-device tensor transfer |
| **Register Management** | `KillRegister` | Explicit register lifetime management |

**Table: VM Instruction Categories**

The instruction dispatch loop ([src/runtime/vm/vm.cc 628-1008](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L628-L1008)) uses a switch statement on the opcode. Each instruction reads operands from registers, performs its operation, and writes results to destination registers.

## Function Invocation

The VM supports three types of function calls: VM function invocation, primitive packed function calls, and closure invocation.

### VM Function Calls

The `Invoke` instruction ([src/runtime/vm/vm.cc 686-687](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L686-L687)) calls a VM function by index. The runtime:

1.   Looks up the `VMFunction` from the `Executable` ([src/runtime/vm/vm.cc 415-427](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L415-L427))
2.   Calls `PushFrame()` to create a new stack frame ([src/runtime/vm/vm.cc 399-402](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L399-L402))
3.   Copies arguments into the new frame's register file
4.   Sets `pc_` and `code_` to the target function's entry point

### Primitive Calls

The `InvokePacked` instruction ([src/runtime/vm/vm.cc 452-495](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L452-L495)) calls a compiled primitive function. Arguments are extracted from registers (including unpacking ADTs for tuple arguments) and marshalled into TVMArgs format for the `PackedFunc` call ([src/runtime/vm/vm.cc 463-494](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L463-L494)).

### Closure Calls

The `InvokeClosure` instruction ([src/runtime/vm/vm.cc 696-698](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L696-L698)) handles higher-order functions. A `VMClosure` object ([include/tvm/runtime/vm/vm.h 59-79](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/runtime/vm/vm.h#L59-L79)) contains a function index and captured free variables. The runtime unpacks the closure, prepends free variables to the argument list, and invokes the underlying function.

## Dynamic Shape Support

Unlike the Graph Executor which requires static shapes, the VM supports dynamic shapes through runtime shape computation.

**Diagram: Dynamic Shape Flow**

The `ShapeOf` instruction ([src/runtime/vm/vm.cc 635-636](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L635-L636)) extracts the shape of a tensor into a new 1D integer array stored in a register. The `AllocTensorReg` instruction ([src/runtime/vm/vm.cc 586-591](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L586-L591)) allocates a tensor using a shape stored in a register (as opposed to `AllocTensor` which uses a compile-time constant shape).

The `ReshapeTensor` instruction ([src/runtime/vm/vm.cc 643-644](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L643-L644)) performs zero-copy shape changes using shape values from registers, enabling efficient dynamic reshaping.

## Memory Management

The VM uses TVM's memory manager with separate allocators for different devices.

### Storage and Tensor Allocation

Memory allocation is a two-step process:

1.   **AllocStorage**: Allocates raw storage buffer ([src/runtime/vm/vm.cc 665-695](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L665-L695))

    *   Takes size, alignment, dtype hint, device index
    *   Returns `Storage` object wrapping allocated buffer
    *   Storage can be reused for multiple tensors

2.   **AllocTensor/AllocTensorReg**: Creates tensor view on storage ([src/runtime/vm/vm.cc 577-591](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L577-L591))

    *   Takes storage, offset, shape, dtype
    *   Returns `NDArray` view into the storage
    *   Multiple tensors can share storage

**Diagram: Storage and Tensor Allocation**

The VM initializes allocators per device during `Init()` ([src/runtime/vm/vm.cc 524-550](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L524-L550)). Users can configure allocator types (naive or pooled) per device via the Python interface ([python/tvm/runtime/vm.py 355-423](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/runtime/vm.py#L355-L423)).

## Multi-Device Execution

The VM supports heterogeneous execution across multiple devices with explicit device placement and cross-device copying.

### Device Indexing

The `Executable` contains a `virtual_devices` vector ([src/runtime/vm/executable.cc 196-203](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L196-L203)) mapping device indices used in bytecode to actual device types and IDs. At runtime, `VirtualMachine::Init()` ([src/runtime/vm/vm.cc 524-550](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L524-L550)) maps these to physical devices provided by the user.

### Device Copy

The `DeviceCopy` instruction ([src/runtime/vm/vm.cc 536-543](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L536-L543)) transfers tensors between devices. The compiler emits these automatically when data flows between operations on different devices ([src/relay/backend/vm/compiler.cc 533-545](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L533-L545)).

### Device Placement

*   Constants are placed on specific devices tracked by `const_device_indexes` ([src/runtime/vm/vm.cc 368-378](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L368-L378))
*   Function parameters have device assignments in `param_device_indexes` ([include/tvm/runtime/vm/vm.h 96-97](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/runtime/vm/vm.h#L96-L97))
*   Allocations specify target device via device index ([src/runtime/vm/vm.cc 621-623](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/vm.cc#L621-L623))

## Serialization and Deployment

The VM executable can be serialized to bytecode and deployed independently from the compilation environment.

### Executable Serialization

**Diagram: Executable Serialization Flow**

The `Executable::Save()` method ([src/runtime/vm/executable.cc 283-310](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L283-L310)) serializes the executable to a binary format with sections:

1.   **Header**: Magic number and TVM version
2.   **Virtual Devices**: Device type/ID mappings ([src/runtime/vm/executable.cc 312-315](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L312-L315))
3.   **Global Section**: Function name to index mapping ([src/runtime/vm/executable.cc 388-401](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L388-L401))
4.   **Constant Section**: Serialized NDArray constants ([src/runtime/vm/executable.cc 409-437](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L409-L437))
5.   **Primitive Section**: Packed function names ([src/runtime/vm/executable.cc 302](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L302-L302))
6.   **Code Section**: VM functions and bytecode ([src/runtime/vm/executable.cc 304](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L304-L304))

### Late-Bound Constants

Large constants can be moved to a separate file to reduce bytecode size ([src/runtime/vm/executable.cc 317-386](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L317-L386)). The `move_late_bound_consts()` method extracts constants above a size threshold, and `load_late_bound_consts()` reloads them before execution.
