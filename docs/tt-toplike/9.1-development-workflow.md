This page details the standard development procedures for `tt-toplike`, including the feature-flag-based build system, testing strategies with mock hardware, and the utility scripts used to validate visualization modes and frontends.

## Build System and Feature Flags

The project uses a modular architecture where major subsystems (TUI, GUI, and specific backends) are guarded by Cargo feature flags. This allows for lightweight builds in environments without specific dependencies (e.g., building the TUI on a server without Vulkan/GL libraries).

### Core Feature Flags

| Feature Flag | Description | Included in Default |
| --- | --- | --- |
| `tui` | Enables the `ratatui`-based Terminal User Interface. | Recommended |
| `gui` | Enables the `iced`-based Graphical User Interface. | Optional |
| `json-backend` | Enables the subprocess-based backend that parses `tt-smi` JSON. | Recommended |
| `linux-procfs` | Enables workload detection by scanning `/proc` for Tenstorrent file descriptors. | Recommended |
| `luwen-backend` | Enables direct PCI BAR0 access via the `luwen` library. Requires `ttkmd`. | **No** |

### Build Commands

The following commands represent the standard development entry points:

`# Standard TUI build (safe for CI and most dev environments)cargo build --release --bin tt-toplike-tui --features tui,json-backend,linux-procfs # GUI build (requires desktop environment dependencies)cargo build --release --bin tt-toplike-gui --features gui # Development build with Luwen (for low-level hardware debugging)cargo build --release --bin tt-toplike-tui --features tui,json-backend,linux-procfs,luwen-backend`
[CONTRIBUTING.md 50-63](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L50-L63)

> **Warning**: Using `--all-features` is discouraged as several features are mutually exclusive or require hardware/drivers (like `luwen-backend`) that may not be present on build machines [CONTRIBUTING.md 65-66](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L65-L66)

## Offline Dependency Strategy

To support restricted environments and reproducible Debian packaging, `tt-toplike` utilizes a `vendor/` directory strategy. This ensures that all Rust crate dependencies are bundled with the source code, allowing for `cargo build --offline` execution during the package creation process.

## Mock Mode and Hardware Simulation

For developers without immediate access to Tenstorrent hardware, the `MockBackend` provides a simulated telemetry stream. This is essential for testing UI layouts and animations.

### Using Mock Mode

The mock backend can be triggered via CLI flags on any frontend binary:

*   `--mock`: Enables the simulation backend.
*   `--mock-devices <N>`: Specifies the number of virtual devices to generate (e.g., simulating a Galaxy system with 32 devices).

`# Run the TUI with 4 mock devicescargo run -- --mock --mock-devices 4`
[CONTRIBUTING.md 82](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L82-L82)

## Helper Scripts

The repository includes shell scripts to automate the testing of various visualization modes and frontend configurations.

### `test-modes.sh`

This script iterates through all available visualization modes in the TUI using a 3-second timeout for each. It is the primary tool for verifying that changes to the animation engine do not cause regressions across different views.

**Supported Modes Tested:**

1.   `normal`: Standard table view.
2.   `starfield`: Tensix core brightness/twinkle visualization.
3.   `castle`: Memory hierarchy particle system.
4.   `flow`: DRAM-to-Grid data motion.
5.   `arcade`: Unified compositor with the "Hero" character.

[test-modes.sh 8-13](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/test-modes.sh#L8-L13)[test-modes.sh 20-30](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/test-modes.sh#L20-L30)

### `test-egui.sh`

This script launches the `egui`-based dashboard (`tt-toplike-egui`) using the mock backend. It serves as a quick-start for testing the lightweight GUI frontend.

[test-egui.sh 11-12](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/test-egui.sh#L11-L12)

## Testing and Quality Standards

### Unit Testing

Developers are expected to write unit tests for new logic. The test suite is executed via:

`cargo test --lib --features tui`
[CONTRIBUTING.md 61-63](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L61-L63)

### Code Style and Linting

The project enforces strict formatting and linting rules:

1.   **Toolchain**: Uses Rust `1.93.0` as specified in `rust-toolchain.toml`[rust-toolchain.toml 1-4](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/rust-toolchain.toml#L1-L4)
2.   **Formatting**: `cargo fmt` must be run before submission [CONTRIBUTING.md 70](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L70-L70)
3.   **Linting**: `cargo clippy` is used to catch common errors and enforce idiomatic patterns [CONTRIBUTING.md 71](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L71-L71)
4.   **Licensing**: Every new file must include the Apache-2.0 SPDX header [CONTRIBUTING.md 72-76](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L72-L76)

## Pull Request Process

The development workflow follows a standard fork-and-pull model:

1.   **Branching**: Create feature branches from `main`[CONTRIBUTING.md 27](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L27-L27)
2.   **Review**: Maintainers review PRs typically on a weekly cadence [CONTRIBUTING.md 36](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L36-L36)
3.   **Validation**: PRs must pass CI (which runs `rustfmt`, `clippy`, and `cargo test`) before merging [CONTRIBUTING.md 29](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/CONTRIBUTING.md?plain=1#L29-L29)
