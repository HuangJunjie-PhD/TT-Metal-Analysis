This page documents the testing infrastructure for the OpModel system, which validates operation constraints and runtime estimates. The OpModel testing framework ensures that operations correctly report resource constraints (L1 memory, circular buffers, output buffer sizes), runtime estimates are accurate, and tensor layouts are legal for given hardware configurations.

* * *

## Testing Architecture Overview

The testing architecture bridges high-level MLIR operation definitions with low-level hardware constraints and reference implementations. It utilizes a combination of C++ unit tests, mock device contexts, and conversion validation.

## OpModel C++ Unit Tests

C++ unit tests validate the OpModel API's ability to query operation constraints and runtime estimates. These tests use the `OpModelFixture` to manage MLIR contexts and device states.

### OpModelFixture and Device Management

The `OpModelFixture` manages the lifecycle of the `SingletonDeviceContext`, ensuring a device (mock or real) is available for queries. It initializes the `TTCore` and `TTNN` dialects and registers the device in the MLIR module.

### Constraint Validation Testing

Constraint tests verify that `getOpConstraints` returns valid resource usage for various input/output layout combinations. The system uses a `WRAP_OP` macro to forward arguments to the correct `ttnn` operation overload [lib/OpModel/TTNN/TTNNOpModel.cpp 45-55](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L45-L55)

Title: Constraint Query and Validation

**Key Validation Functions:**

*   `getOpConstraints`: Queries peak L1 memory, circular buffer (CB) usage, and output tensor layouts. It utilizes `executeConstraintQuery` to perform the actual call to the backend [lib/OpModel/TTNN/TTNNOpModel.cpp 139-163](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L139-L163)
*   `executeConstraintQuery`: Wraps the low-level `ttnn::graph` query, handling exceptions and validating the `ExecutionStatus`. It also manages `ProgramCacheState` to ensure clean measurements [lib/OpModel/TTNN/TTNNOpModel.cpp 86-120](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L86-L120)

## OpModel Interface Validation

Interface tests validate that MLIR operations correctly implement the `OpModel` interface, allowing the compiler to query properties directly from an `Operation*`.

### Interface Testing Pattern

The tests cast generic `mlir::Operation*` to the `OpModel` interface to ensure the dispatch logic works correctly across different operation types.

**Interface Helper Methods:**

## Shard Solver and Scheduler Tests

The validation infrastructure extends to higher-level optimization components that rely on OpModel data and hardware descriptors.

### Shard Solver Validation

The `ShardSolverBase` fixture provides a controlled environment for testing sharding propagation. It allows for manual injection of "legal" configurations to verify the solver's logic in resolving shard chains [test/unittests/Optimizer/TestShardSolver.cpp 32-45](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/Optimizer/TestShardSolver.cpp#L32-L45)

**Verification Targets:**

*   **Max Core Usage**: Validates that the solver correctly accumulates and propagates core usage across sharded chains [test/unittests/Optimizer/TestShardSolver.cpp 152-156](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/Optimizer/TestShardSolver.cpp#L152-L156)
*   **Grid Selection**: Verifies that the solver selects valid hardware grids for `WidthSharded`, `HeightSharded`, and `BlockSharded` layouts [test/unittests/Optimizer/TestShardSolver.cpp 166-181](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/Optimizer/TestShardSolver.cpp#L166-L181)

### Scheduler Validation

The Scheduler tests verify the execution order of operations based on dependency constraints and resource availability.

*   **Dependency Handling**: Tests ensure that operations are scheduled only after their operands are available [lib/Scheduler/Scheduler.cpp 21-35](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Scheduler/Scheduler.cpp#L21-L35)
*   **Resource Constraints**: Validates that the scheduler respects device memory and compute limits when dispatching ops.

### Greedy L1 Interleaved Policy Tests

The `GreedyL1InterleavedPolicyBase` validates the logic for falling back to interleaved DRAM/L1 layouts when memory is constrained.

*   **Usage Caps**: Tests how the policy reacts to `tensorL1UsageCap` attributes [test/unittests/Optimizer/TestGreedyL1InterleavedPolicy.cpp 49-52](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/Optimizer/TestGreedyL1InterleavedPolicy.cpp#L49-L52)
*   **Policy Choice**: Verifies that the policy correctly identifies which operations should stay in L1 versus falling back to DRAM [test/unittests/Optimizer/TestGreedyL1InterleavedPolicy.cpp 136-187](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/Optimizer/TestGreedyL1InterleavedPolicy.cpp#L136-L187)

## Error Handling and Mock Devices

The system handles cases where hardware is not present or specific queries are unsupported.

### Mock Device Mode

When `SingletonDeviceContext` is in mock mode, certain queries like `getOpRuntime` are explicitly disabled as they require physical hardware timing [lib/OpModel/TTNN/TTNNOpModel.cpp 167-170](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L167-L170)

*   `OpNotSupportedError`: A custom error class used when an operation lacks OpModel support for specific reasons, such as memory write limitations during trace capture [lib/Dialect/TTNN/Interfaces/TTNNOpModelInterface.cpp 35-56](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Interfaces/TTNNOpModelInterface.cpp#L35-L56)

### Resource Guarding

The `ProgramCacheState` RAII helper is used during constraint queries to disable and clear the program cache, ensuring that resource measurements are not affected by cached state. It restores the original cache state upon destruction [lib/OpModel/TTNN/TTNNOpModel.cpp 60-74](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L60-L74)

### Constraint Validation Pass

The `TTNNOperationValidationAndFallback` pass uses `validateOperation` to check if ops are legal under current memory constraints. If an op fails validation, the pass attempts to apply fallbacks (e.g., changing memory layout or data types) [lib/Dialect/TTNN/Transforms/OptimizerPasses/OperationValidationAndFallback.cpp 119-128](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Transforms/OptimizerPasses/OperationValidationAndFallback.cpp#L119-L128)

### OpModel Exemptions

Certain operations are explicitly exempt from implementing the OpModel interface via the `OpModelExempt` trait [include/ttmlir/Dialect/TTNN/IR/TTNNTraits.h 59-60](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/IR/TTNNTraits.h#L59-L60) The validation system recognizes this trait and returns a `NotImplemented` status instead of throwing a fatal error, allowing the optimizer to fall back gracefully [lib/Dialect/TTNN/Validation/OpConstraintValidation.cpp 172-177](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Validation/OpConstraintValidation.cpp#L172-L177)
