# Project Orchestration

Relevant source files
*   [config/runcfg_all.yaml](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/config/runcfg_all.yaml)
*   [polproj.py](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polproj.py)
*   [ttsim/config/runcfgmodel.py](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/config/runcfgmodel.py)

This document covers the project orchestration system that manages and coordinates simulation runs through the `polproj.py` script and `PolarisRunConfig` model. The system provides a higher-level abstraction over the core `polaris.py` simulation orchestrator, adding features like Git integration, templating, and reproducible run management.

For information about the core simulation orchestrator itself, see [Polaris Main Orchestrator](https://deepwiki.com/tenstorrent/polaris/2.1-polaris-main-orchestrator). For details about the underlying configuration management, see [Configuration Management](https://deepwiki.com/tenstorrent/polaris/2.3-configuration-management).

## System Architecture

The project orchestration system is built around three main components: the `polproj.py` orchestration script, the `PolarisRunConfig` validation model, and a templating system for dynamic configuration generation.

### PolarisRunConfig Structure

Sources: [ttsim/config/runcfgmodel.py 16-120](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/config/runcfgmodel.py#L16-L120)

### Configuration Validation

The `PolarisRunConfig` model includes field validators for critical parameters:

| Parameter | Type | Validation | Description |
| --- | --- | --- | --- |
| `frequency` | `TYPE_frequency` | Positive values, start ≤ end | Frequency range in MHz (start, end, step) |
| `batchsize` | `TYPE_batchsize` | Positive values, step ≠ 1, start < end | Batch size geometric sequence |
| `filterrun` | `TYPE_RUN` | Literal validation | Must be 'inference' or 'training' |
| `log_level` | `TYPE_LOGLEVEL` | Literal validation | Must be valid log level |
| `outputformat` | `TYPE_OUTPUTFORMAT` | Literal validation | Must be 'json', 'yaml', 'pickle', or 'none' |

## Git Integration

The orchestration system provides comprehensive Git integration for reproducibility and version control. It extracts repository metadata and supports running simulations at specific commits.

### Git Repository Management

Sources: [polproj.py 81-106](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polproj.py#L81-L106)[polproj.py 109-121](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polproj.py#L109-L121)[polproj.py 199-210](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polproj.py#L199-L210)

### Git Metadata Variables

The system extracts comprehensive Git metadata for use in Jinja2 templates:

| Variable | Type | Description |
| --- | --- | --- |
| `giturl` | `str` | Remote origin URL |
| `githash` | `str` | Current commit hash |
| `gitdirty` | `bool` | Repository has uncommitted changes |
| `gituntracked` | `bool` | Repository has untracked files |
| `gitroot` | `str` | Repository working directory |
| `gitbranch` | `str` | Active branch name |
| `author` | `str` | Author email |
| `committer` | `str` | Committer email |
| `tstamp` | `str` | Current timestamp |
| `commit_tstamp` | `str` | Commit timestamp |
| `commit_year` | `int` | Commit year |
| `commit_month` | `str` | Commit month (zero-padded) |

Sources: [polproj.py 81-106](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polproj.py#L81-L106)
