## Purpose and Scope

This page explains how tt-lang kernels execute on Tenstorrent hardware. It covers the thread-based execution model, coordination mechanisms, and how execution scales from single-core to multi-core grids.

For detailed technical breakdowns, see the child pages:

*   [Single-core Execution](https://deepwiki.com/tenstorrent/tt-lang/2.4.1-single-core-execution) — Details on thread coordination and Dataflow Buffer (DFB) synchronization.
*   [Multi-core Grid Execution and Work Distribution](https://deepwiki.com/tenstorrent/tt-lang/2.4.2-multi-core-grid-execution-and-work-distribution) — Details on grid specification, core coordinates, and work partitioning.

## Kernel Execution Overview

A tt-lang operation is a Python function decorated with `@ttl.operation` (aliased from `ttl.kernel` in earlier versions) that defines computation to be executed on Tenstorrent hardware [docs/sphinx/specs/TTLangSpecification.md 45](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L45-L45)[docs/sphinx/specs/TTLangSpecification.md 60-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L60-L62) When invoked, the operation follows a specific lifecycle:

1.   **Grid Determination**: Defines the space of nodes (Tensix cores) to which the operation is submitted [docs/sphinx/specs/TTLangSpecification.md 100-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L102)
2.   **Resource Instantiation**: Creates Dataflow Buffers (DFB), pipes, and semaphores shared by all kernel functions [docs/sphinx/specs/TTLangSpecification.md 57-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L57-L58)
3.   **Thread Launch**: Executes compute and data movement threads defined via `@ttl.compute()` and `@ttl.datamovement()`[docs/sphinx/specs/TTLangSpecification.md 62-63](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L62-L63)
4.   **Explicit Coordination**: Threads synchronize via DFBs to manage dataflow between computation and movement [docs/sphinx/specs/TTLangSpecification.md 52-55](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L52-L55)

## Thread-Based Execution Model

Each Tensix core runs multiple hardware threads that execute concurrently to overlap computation with data movement [docs/sphinx/specs/TTLangSpecification.md 53-55](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L53-L55) The DSL abstracts these into three primary roles:

| Thread Type | Decorator | Hardware Thread | Primary Role |
| --- | --- | --- | --- |
| **Compute** | `@ttl.compute()` | TRISC (MATH) | Execute tile math operations (FPU/SFPU) [docs/sphinx/specs/TTLangSpecification.md 74-75](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L74-L75) |
| **Data Movement** | `@ttl.datamovement()` | NCRISC | Typically used for reading data from DRAM/L1 [docs/sphinx/specs/TTLangSpecification.md 78-81](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L78-L81) |
| **Data Movement** | `@ttl.datamovement()` | BRISC | Typically used for writing data to DRAM/L1 [docs/sphinx/specs/TTLangSpecification.md 82-84](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L82-L84) |

### Thread Coordination via DFB

Thread functions communicate through `DataflowBuffer` objects [python/ttl/ttl_api.py 74](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L74-L74) Synchronization is enforced via `reserve`, `wait`, `push`, and `pop` operations [docs/sphinx/specs/TTLangSpecification.md 38-39](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L38-L39) While the specification allows explicit calls, the compiler can also automatically insert missing synchronization ops via the `TTLInsertCBSync` and `TTLInsertCopyWait` passes [include/ttlang/Dialect/TTL/Passes.td 6-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L6-L118)

## Single-Core Execution

In single-core execution, a kernel runs on exactly one Tensix core with `grid=(1, 1)`[examples/elementwise-tutorial/step_1_single_node_single_tile_block.py](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/elementwise-tutorial/step_1_single_node_single_tile_block.py) (by default). All thread functions execute on the same core, coordinating through local dataflow buffers.

*   **Blocking Operations**: `reserve()` and `wait()` block execution until buffer space or data is available [include/ttlang/Dialect/TTL/Passes.td 8-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L8-L12)
*   **Signaling**: `push()` and `pop()` are non-blocking signals that update the buffer state machine for the consumer or producer [include/ttlang/Dialect/TTL/Passes.td 8-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L8-L12)
*   **Compiler Materialization**: If compute ops require DFB inputs but receive fused expression results, the `TTLInsertIntermediateDFBs` pass materializes them to L1 via double-buffered DFBs [lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp 9-14](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp#L9-L14)

For details, see [Single-core Execution](https://deepwiki.com/tenstorrent/tt-lang/2.4.1-single-core-execution).

## Multi-Core Grid Execution

Multi-core execution distributes work across a grid of Tensix cores. In this model, the same kernel code is executed on multiple cores (SPMD mode), with each core identifying its unique role via coordinates [docs/sphinx/specs/TTLangSpecification.md 100-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L103)

### Grid Coordinates

Cores access their position and total grid size using:

*   `ttl.grid_size(dims)`: Returns the total dimensions of the execution grid [docs/sphinx/specs/TTLangSpecification.md 107-114](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L107-L114)
*   `ttl.node(dims)`: Returns the coordinates of the current core [docs/sphinx/specs/TTLangSpecification.md 127-129](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L127-L129)

### Inter-Core Communication

Multi-core kernels can communicate using **Pipes** and **PipeNets**, which wrap the complexity of node-to-node communication and memory layout [docs/sphinx/specs/TTLangSpecification.md 57-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L57-L58) The compiler handles lowering these high-level pipe abstractions through the `PipeLowering` and `PipeGraph` passes [lib/Dialect/TTL/Transforms/CMakeLists.txt 4-5](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/CMakeLists.txt#L4-L5)

For details, see [Multi-core Grid Execution and Work Distribution](https://deepwiki.com/tenstorrent/tt-lang/2.4.2-multi-core-grid-execution-and-work-distribution).

## Execution Lifecycle and Profiling

When `TTLANG_AUTO_PROFILE=1` is enabled, the execution model includes automatic instrumentation to measure performance at the line level [python/ttl/ttl_api.py 53-61](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L53-L61)

1.   **Instrumentation**: The compiler inserts profiling signposts during the compilation pipeline via the `LowerSignpostToEmitC` pass [lib/Dialect/TTL/Transforms/CMakeLists.txt 13](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/CMakeLists.txt#L13-L13)
2.   **Hardware Execution**: The kernel runs on the device, emitting timing data to a device-side buffer.
3.   **Collection**: `ttnn.ReadDeviceProfiler` fetches profile logs after execution [python/ttl/ttl_api.py 205](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L205-L205)
4.   **Reporting**: Data is parsed from CSV and mapped back to Python source lines using the `line_mapper` for performance analysis [python/ttl/ttl_api.py 228-231](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L228-L231)
