## Purpose and Scope

This document provides a practical guide for writing individual test scenario functions within the coretp framework. A test scenario function is a Python function that defines a sequence of test steps and their execution environment. This page covers the mechanics of composing TestStep primitives, configuring test environments with `TestEnvCfg`, and assembling the final `TestScenario` object.

For information about creating and registering new test plans, see [Creating New Test Plans](https://deepwiki.com/tenstorrent/riscv-coretp/6.1-creating-new-test-plans). For details on the TestStep primitive types themselves, see [Test Steps Reference](https://deepwiki.com/tenstorrent/riscv-coretp/5-test-steps-reference).

* * *

## Test Scenario Function Structure

A test scenario function follows a standard pattern:

1.   **Decorator registration** - Associates the function with a test plan using a decorator
2.   **Step composition** - Creates and configures TestStep objects
3.   **Scenario assembly** - Returns a `TestScenario` object via `TestScenario.from_steps()`

**Basic Template:**

The minimal structure appears in [coretp/plans/svadu/svadu_scenarios.py 32-62](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L32-L62):

`@svadu_scenariodef SID_SVADU_01_fault_on_a_bit_cleared():    """    Docstring describing test purpose    """    # Step 1: Create test steps    mem = Memory(size=0x1000, ...)    disable_svadu = CsrWrite(csr_name="menvcfg", clear_mask=1 << 61)    load_op = Load(memory=mem)    assert_fault = AssertException(cause=..., code=[load_op])        # Step 2: Assemble scenario    return TestScenario.from_steps(        id="1",        name="SID_SVADU_01_fault_on_a_bit_cleared",        description="When SVADU disabled, memory access faults when pte.a=0",        env=TestEnvCfg(paging_modes=[...]),        steps=[mem, disable_svadu, assert_fault],    )`
**Sources:**[coretp/plans/svadu/svadu_scenarios.py 32-62](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L32-L62)[coretp/plans/sstc/sstc_scenarios.py 11-101](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L11-L101)

## Decorator Registration

Test scenario functions must be decorated with a test plan decorator to register them with the test plan registry. The decorator is created by the test plan's `__init__.py` file using `new_test_plan()`.

**Example from SSTC test plan:**

[coretp/plans/sstc/__init__.py 8-12](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/__init__.py#L8-L12) creates the decorator:

`sstc_scenario = new_test_plan(    name="sstc",    description="Covers SSTC (Supervisor-mode Timer Counter) behavior",    tags=["sstc", "timer", "supervisor", "interrupt"],)`
Then [coretp/plans/sstc/sstc_scenarios.py 11-12](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L11-L12) uses it:

`@sstc_scenariodef SID_SSTC_01():    # ... scenario implementation`
The decorator automatically registers the scenario with the test plan registry and associates it with the plan's metadata.

**Sources:**[coretp/plans/sstc/__init__.py 1-14](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/__init__.py#L1-L14)[coretp/plans/sstc/sstc_scenarios.py 11-12](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L11-L12)

## Composing Test Steps

Test scenarios are built by composing TestStep primitives. Each step represents an operation or verification point in the test.

### Memory Management Steps

**Memory Step:**

The `Memory` step allocates a memory region with specified page table attributes. Example from [coretp/plans/svadu/svadu_scenarios.py 37-43](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L37-L43):

`mem = Memory(    size=0x1000,                    # Region size in bytes    page_size=PageSize.SIZE_4K,     # Page granularity    flags=PageFlags.VALID | PageFlags.READ | PageFlags.WRITE,    exclude_flags=PageFlags.ACCESSED | PageFlags.DIRTY,    modify=True,                    # Allow PTE modifications)`
**Key Parameters:**

*   `size` - Memory region size
*   `page_size` - Page size (SIZE_4K, SIZE_2M, SIZE_1G)
*   `flags` - Page table entry flags to set
*   `exclude_flags` - Flags to explicitly clear
*   `modify` - Whether the test will modify PTEs

**Advanced Page Table Operations:**

[coretp/plans/svadu/svadu_scenarios.py 118-122](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L118-L122) shows reading PTE entries:

`first_read_leaf_pte = ReadLeafPTE(memory=mem)load_first_immediate_mask_check = LoadImmediateStep(imm=1 << 6)and_first_op = Arithmetic(op="and", src1=first_read_leaf_pte, src2=load_first_immediate_mask_check)`
**Sources:**[coretp/plans/svadu/svadu_scenarios.py 37-43](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L37-L43)[coretp/plans/svadu/svadu_scenarios.py 106-112](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L106-L112)[coretp/plans/svadu/svadu_scenarios.py 161-167](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L161-L167)

### CSR Operations

CSR operations configure processor state and verify register values.

**CsrWrite Step:**

[coretp/step/csr.py 12-42](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/csr.py#L12-L42) defines three write modes:

`# Set specific bitsenable_svadu = CsrWrite(csr_name="menvcfg", set_mask=1 << 61) # Clear specific bits  disable_svadu = CsrWrite(csr_name="menvcfg", clear_mask=1 << 61) # Write entire valuewrite_value = CsrWrite(csr_name="mstatus", value=0x1800)`
Only one of `set_mask`, `clear_mask`, or `value` can be provided per operation.

**CsrRead Step:**

[coretp/plans/sstc/sstc_scenarios.py 22](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L22-L22) shows CSR reading:

`time_read = CsrRead(csr_name="time", direct_read=True)`
The `direct_read` parameter controls whether the read uses a direct `csrr` instruction or transitions through privilege levels.

**Verification Pattern:**

[coretp/plans/sstc/sstc_scenarios.py 457-461](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L457-L461) demonstrates verifying CSR bits:

`henvcfg_read = CsrRead(csr_name="henvcfg")top = LoadImmediateStep(imm=(1 << 63))henvcfg_masked = Arithmetic(op="and", src1=henvcfg_read, src2=top)zero = LoadImmediateStep(imm=0)assert_equal = AssertEqual(src1=henvcfg_masked, src2=zero)`
**Sources:**[coretp/step/csr.py 12-61](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/csr.py#L12-L61)[coretp/plans/svadu/svadu_scenarios.py 46](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L46-L46)[coretp/plans/sstc/sstc_scenarios.py 19-27](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L19-L27)

### Load and Store Operations

Memory access operations reference a `Memory` step and optionally specify data values or sources.

**Load Step:**

[coretp/plans/svadu/svadu_scenarios.py 49](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L49-L49) shows basic load:

`load_op = Load(memory=mem)`
The load operation reads from the memory region and returns a value that can be used in subsequent steps.

**Store Step:**

[coretp/plans/svadu/svadu_scenarios.py 82-83](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L82-L83) shows store with immediate value:

`store_val = LoadImmediateStep(imm=0xDEAD)store_op = Store(memory=mem, value=store_val)`
The `value` parameter accepts either a TestStep (for step chaining) or an immediate integer.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 49](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L49-L49)[coretp/plans/svadu/svadu_scenarios.py 82-83](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L82-L83)[coretp/plans/svadu/svadu_scenarios.py 180-181](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L180-L181)

### Assertions and Verification

Assertions verify expected behavior and detect faults.

**AssertException:**

[coretp/plans/svadu/svadu_scenarios.py 50](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L50-L50) verifies page faults:

`load_op = Load(memory=mem)assert_load_fault = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,     code=[load_op])`
The `code` parameter contains the TestStep(s) expected to trigger the exception.

**AssertEqual and AssertNotEqual:**

[coretp/plans/svadu/svadu_scenarios.py 131](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L131-L131) verifies bit patterns:

`assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
Both `src1` and `src2` accept TestStep objects or immediate integers.

### Arithmetic and Computation Steps

Arithmetic steps perform computations on values from previous steps.

**LoadImmediateStep:**

[coretp/plans/svadu/svadu_scenarios.py 119](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L119-L119) loads constants:

`load_immediate_mask_check = LoadImmediateStep(imm=1 << 6)`

[coretp/plans/svadu/svadu_scenarios.py 120](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L120-L120) performs bitwise AND:

## Configuring Test Environments

`TestEnvCfg` specifies the parameter space for test execution. The framework generates concrete test environments by expanding these parameters.

### Parameter Expansion

The framework expands `TestEnvCfg` parameters into multiple concrete test instances:

| Parameter | Type | Effect |
| --- | --- | --- |
| `priv_modes` | List of PrivilegeMode | Tests run in each specified privilege level |
| `paging_modes` | List of PagingMode | Tests run with each paging configuration |
| `hypervisor` | List of bool | Tests run with/without H-extension |
| `virtualized` | List of bool | Tests run in HS-mode vs VS-mode |
| `min_num_harts` | int | Ensures multi-hart configurations |

### Step Ordering

Steps execute in the order specified in the `steps` list. Each step can reference outputs from previous steps:

[coretp/plans/svadu/svadu_scenarios.py 138-151](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L138-L151) shows step dependencies:

`steps=[    mem,                           # 1. Allocate memory    enable_svadu,                  # 2. Configure CSR    first_read_leaf_pte,          # 3. Read PTE    load_first_immediate_mask_check,  # 4. Load mask constant    and_first_op,                  # 5. Mask operation (uses steps 3, 4)    zero,                          # 6. Load zero constant    assert_equal_first,            # 7. Verify result (uses steps 5, 6)    load_op,                       # 8. Perform load (uses step 1)    read_leaf_pte,                 # 9. Read PTE again    load_immediate_mask_check,     # 10. Load mask    and_op,                        # 11. Mask operation    assert_equal,                  # 12. Final verification]`
**Sources:**[coretp/plans/svadu/svadu_scenarios.py 52-62](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L52-L62)[coretp/plans/svadu/svadu_scenarios.py 133-152](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L133-L152)[coretp/plans/sstc/sstc_scenarios.py 78-101](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L78-L101)

## Common Patterns and Examples

### Pattern: CSR Configuration and Verification

This pattern configures a CSR, performs an operation, then verifies the CSR state.

[coretp/plans/sstc/sstc_scenarios.py 451-487](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L451-L487) demonstrates WARL bit verification:

`# Clear CSR bitmenvcfg_clear = CsrWrite(csr_name="menvcfg", clear_mask=(1 << 63)) # Try to set dependent CSR bithenvcfg_set = CsrWrite(csr_name="henvcfg", set_mask=(1 << 63)) # Read back and verify bit remained clear (WARL behavior)henvcfg_read = CsrRead(csr_name="henvcfg")top = LoadImmediateStep(imm=(1 << 63))henvcfg_masked = Arithmetic(op="and", src1=henvcfg_read, src2=top)zero = LoadImmediateStep(imm=0)assert_equal = AssertEqual(src1=henvcfg_masked, src2=zero)`
### Pattern: Exception Verification

Test that operations cause expected exceptions under specific conditions.

[coretp/plans/sstc/sstc_scenarios.py 19-27](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L19-L27) tests privilege-based access control:

`# Configure to block accessmcounteren_clear = CsrWrite(csr_name="mcounteren", clear_mask=0x2) # Attempt access - should faulttime_read_1 = CsrRead(csr_name="time", direct_read=True)assert_time_exception_1 = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[time_read_1])`
### Pattern: PTE Bit Verification

Verify page table entry bits are set/cleared as expected.

[coretp/plans/svadu/svadu_scenarios.py 127-131](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L127-L131) checks the Accessed bit:

`read_leaf_pte = ReadLeafPTE(memory=mem)# A bit is bit 6 in the PTE entryload_immediate_mask_check = LoadImmediateStep(imm=1 << 6)and_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
### Pattern: Before/After State Comparison

Verify that an operation changes state as expected.

[coretp/plans/svadu/svadu_scenarios.py 118-151](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L118-L151) verifies hardware A bit update:

`# Read PTE before operationfirst_read_leaf_pte = ReadLeafPTE(memory=mem)# ... verify A bit is 0 ... # Perform memory accessload_op = Load(memory=mem) # Read PTE after operation  read_leaf_pte = ReadLeafPTE(memory=mem)# ... verify A bit is now 1 ...`
### Pattern: Multi-Configuration Testing

Single scenario function tests multiple CSR configurations.

## Step Composition Reference Table

| Step Type | Purpose | Key Parameters | Output Type |
| --- | --- | --- | --- |
| `Memory` | Allocate memory region | `size`, `page_size`, `flags` | Memory reference |
| `Load` | Read from memory | `memory` | Data value |
| `Store` | Write to memory | `memory`, `value` | None |
| `CsrRead` | Read CSR | `csr_name`, `direct_read` | CSR value |
| `CsrWrite` | Write CSR | `csr_name`, `set_mask`/`clear_mask`/`value` | None |
| `LoadImmediateStep` | Load constant | `imm` | Constant value |
| `Arithmetic` | Compute operation | `op`, `src1`, `src2` | Computed value |
| `ReadLeafPTE` | Read page table entry | `memory` | PTE value |
| `AssertEqual` | Verify equality | `src1`, `src2` | None |
| `AssertNotEqual` | Verify inequality | `src1`, `src2` | None |
| `AssertException` | Expect exception | `cause`, `code` | None |
