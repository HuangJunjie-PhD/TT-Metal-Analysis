The `show_telemtery` action is a diagnostic utility designed to monitor system resource consumption during the execution of a GitHub Actions workflow. It samples data directly from the Linux `/proc` filesystem to track CPU load, memory usage, disk availability, and network throughput. The action provides visual feedback by generating charts in the GitHub Job Summary and optionally preserves raw data as workflow artifacts.

## Overview and Data Flow

The action operates as a background process that spans the duration of other workflow steps. It is typically invoked twice: once with `start_collection` to initialize the background sampler, and once with `finish_collection` to terminate the process and generate reports [action.yml 18-23](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L18-L23)

## Implementation Details

### Data Collection (`collect_telemetry.py`)

The `ProcTelemetryCollector` class is responsible for the low-level parsing of the `/proc` filesystem. It utilizes standard Python libraries to avoid heavy dependencies during the collection phase.

*   **Process Filtering**: To prevent log bloat, the collector only records processes exceeding specific thresholds: 
    *   `memory_threshold_kb`: 100 MB [collect_telemetry.py 19](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L19-L19)
    *   `cpu_threshold`: 1.0% [collect_telemetry.py 22](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L22-L22)

*   **Metrics Gathered**: 
    *   **Memory**: Parsed from `/proc/[pid]/status` (specifically `VmRSS`) [collect_telemetry.py 40-85](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L40-L85)
    *   **CPU**: Calculated as an approximation using `/proc/[pid]/stat` fields (utime, stime, cutime, cstime) relative to system uptime [collect_telemetry.py 87-146](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L87-L146)
    *   **Disk**: Collected via `os.statvfs` on mount points found in `/proc/mounts`[collect_telemetry.py 148-185](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L148-L185)
    *   **Network**: Derived from `/proc/net/dev`[collect_telemetry.py 187-203](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/collect_telemetry.py#L187-L203)

### Data Processing (`process_telemetry.py`)

The `TelemetryProcessor` class consumes the JSONL (JSON Lines) file produced by the collector. It aggregates samples and formats them for display.

*   **Chart Generation**: The processor generates charts using GitHub's native support for Mermaid.js or Markdown tables. Key functions include: 
    *   `process_memory_usage()`: Aggregates total and used memory [process_telemetry.py 52-80](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/process_telemetry.py#L52-L80)
    *   `process_top_processes_by_cpu()`: Identifies the top N resource consumers over the duration of the run [process_telemetry.py 146-186](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/process_telemetry.py#L146-L186)

*   **Output**: The results are written to a `summary.md` file, which the composite action then appends to the `$GITHUB_STEP_SUMMARY`[action.yml 72-73](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L72-L73)

## Action Configuration

### Inputs

| Input | Description | Default |
| --- | --- | --- |
| `proc_path` | Path to the proc folder. Use `/host/proc` in Docker [README.md 38](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/README.md?plain=1#L38-L38) | `/proc` |
| `sampling_rate` | Frequency of data collection in seconds [action.yml 13](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L13-L13) | `5` |
| `generate_chart` | Whether to include visual charts in the summary [action.yml 17](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L17-L17) | `true` |
| `start_collection` | Boolean flag to trigger the start of the background process [action.yml 18](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L18-L18) | N/A |
| `finish_collection` | Boolean flag to trigger the stop and report generation [action.yml 21](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L21-L21) | N/A |
| `telemetry_path` | Directory to store temporary logs and data [action.yml 27](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/action.yml#L27-L27) | `telemetry` |

### Docker Container Usage

When running within a Docker container, the runner's host `/proc` must be mounted into the container to see the actual hardware resource usage. The recommended pattern is to mount `/proc` to `/host/proc` as read-only and point the action to that path [README.md 81-87](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/README.md?plain=1#L81-L87)

`container:  image: ubuntu:22.04  volumes:    - /proc:/host/proc:rosteps:  - uses: ./.github/actions/show_telemtery    with:      start_collection: true      proc_path: '/host/proc'`

## Local Testing

The repository includes a `local_test.sh` script to verify telemetry collection and processing outside of the GitHub environment. This script:

1.   Starts the collector in the background [local_test.sh 17](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/local_test.sh#L17-L17)
2.   Simulates CPU load using `dd`[local_test.sh 37](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/local_test.sh#L37-L37)
3.   Simulates memory pressure by allocating a large Python array (~500MB) [local_test.sh 52](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/local_test.sh#L52-L52)
4.   Simulates network activity via `curl`[local_test.sh 72](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/local_test.sh#L72-L72)
5.   Stops the collector and runs the processor to verify the output [local_test.sh 81-87](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/local_test.sh#L81-L87)
