## Purpose and Scope

The guard system is a critical safety mechanism in the torch_ttnn compilation pipeline that prevents operations with problematic input signatures from being lowered to TTNN operations. When certain operation and input combinations are known to cause runtime failures, produce incorrect results, or exceed hardware resource limits, the guard system blocks their conversion and forces them to fall back to PyTorch's native implementation.

This document covers the architecture of the guard checking system, the structure of operation blocklists (both autogenerated and manual), and how guards integrate with the compilation pipeline. For information about how guards are generated from test failures, see [Guard Blocklist Management](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.4-guard-blocklist-management). For details on the lowering process that guards protect, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)).

## Guard System Architecture

The guard system consists of two main components: an autogenerated blocklist file derived from execution metrics and a manual blocklist file containing hand-curated entries for specific failure cases discovered during model testing.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 1-311](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L1-L311)[torch_ttnn/passes/lowering/to_tt_guard.py 1-114](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L1-L114)

## Guard Function Entry Point

The primary interface to the guard system is the `can_lowering_to_ttnn()` function, which determines whether a graph node can safely be lowered to TTNN.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L109-L113)[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 288-310](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L288-L310)

The function performs a simple lookup:

1.   Check if the node's target operation exists in the `GUARD` dictionary
2.   If present, invoke the registered guard function for that operation
3.   If not present, return `True` (allow lowering by default)

## Input Variation Extraction

The guard system identifies problematic operations by their complete input signatures, including tensor shapes, dtypes, and scalar parameter values. The `get_inputs()` function extracts these signatures from graph nodes using the metrics collection infrastructure.

| Function | Purpose | Return Type |
| --- | --- | --- |
| `get_inputs(node)` | Extract input variation from node metadata | `List[str]` or `None` |
| `guard_aten(blocklist, node)` | Check if node's inputs match blocklist entries | `bool` |

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 288-295](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L288-L295)

The input extraction logic handles two metric types:

*   **InputVariation**: Direct capture of original PyTorch operation inputs
*   **ConvertedInput**: Wrapper that preserves the original input variation after conversion

Each input is stringified to create a canonical representation like:

```
"Tensor<[1, 3, 224, 224]> input = ?"
"Tensor<[768, 3, 32, 32]> weight = ?"
"List[int] stride = [32, 32]"
```

## Autogenerated Blocklists

The autogenerated blocklists are produced by the guard generation tool (see [Guard Blocklist Management](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.4-guard-blocklist-management)) that analyzes test execution failures. Each blocklist is a Python list of lists, where each inner list represents a complete input signature that should be blocked.

### Convolution Blocklists

The largest autogenerated blocklist is for `aten.convolution.default`, containing over 20 entries that primarily block operations that cause out-of-memory (OOM) errors on device or have problematic tensor dimensions.

**Example entries:**

`aten_convolution_default_blocklist = [    # Large kernel convolutions that OOM    [        "Tensor<[1, 3, 224, 224]> input = ?",        "Tensor<[768, 3, 32, 32]> weight = ?",        "Optional[Tensor] bias = ?",        "List[int] stride = [32, 32]",        ...    ],    # Large spatial dimension convolutions    [        "Tensor<[1, 64, 480, 640]> input = ?",        "Tensor<[64, 64, 3, 3]> weight = ?",        ...    ],    # Depthwise convolutions with high group count    [        "Tensor<[1, 816, 19, 19]> input = ?",        "Tensor<[816, 1, 5, 5]> weight = ?",        ...        "int groups = 816",    ],]`
**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 13-279](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L13-L279)

## Manual Blocklists

The manual blocklist file (`to_tt_guard.py`) contains hand-curated entries discovered during model-specific testing. Each section is documented with the model name and specific failure reason.

### Structure of Manual Blocklists

Manual blocklists follow the same structure as autogenerated ones but include extensive comments explaining the failure mode:

`############################################################# EXTRA BLOCKLIST OF codegen############################################################# ttnn.from_torch(torch.tensor(False)) => ttnn.gt# ttnn.from_torch not support scalar of bool dtype# RuntimeError: TT_FATAL @ tensor/types.cpp:209: normalized_index >= 0 and normalized_index < rank# not lowering ttnn.gt to avoid ttnn.from_torch of scalar of bool dtypeaten_gt_Scalar_blocklist = [["Tensor<[]> self = ?", "number other = 0"]]`

## Guard Registration and Lookup

The `GUARD` dictionary maps PyTorch operation targets to their corresponding guard functions. Both autogenerated and manual guards contribute to this registry.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 306-310](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L306-L310)[torch_ttnn/passes/lowering/to_tt_guard.py 104-106](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L104-L106)

The registration uses Python's `functools.partial` to bind each blocklist to the `guard_aten` function:

`GUARD = {    torch.ops.aten._log_softmax.default: partial(guard_aten, aten__log_softmax_default_blocklist),    torch.ops.aten.convolution.default: partial(guard_aten, aten_convolution_default_blocklist),    torch.ops.aten.argmax.default: partial(guard_aten, aten_argmax_default_blocklist),} # Extended in to_tt_guard.pyGUARD[torch.ops.aten.gt.Scalar] = partial(guard_aten, aten_gt_Scalar_blocklist)GUARD[torch.ops.aten.cumsum.default] = partial(guard_aten, aten_cumsum_default_blocklist)GUARD[torch.ops.aten.stack.default] = partial(guard_aten, aten_aten_stack_default)`
## Integration with Compilation Pipeline

The guard system is invoked at two key points in the compilation pipeline:

### 1. Operation Lowering (ToTtPass)

During the main lowering pass, guards prevent problematic operations from being converted to TTNN. When a guard returns `False`, the operation remains in its original aten form and will be executed by PyTorch's fallback mechanism.

### 2. Constant Folding Pass

The constant folding pass checks guards before folding operations with constant inputs. This prevents the creation of constant tensors that would fail when materialized on device.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 51-54](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L54)[torch_ttnn/passes/constant_folding_pass.py 39-49](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L39-L49)

## Blocklist Entry Format

Each blocklist entry is a list of strings representing the complete function signature with concrete values:

| Component | Example | Purpose |
| --- | --- | --- |
| Tensor argument | `"Tensor<[1, 3, 224, 224]> input = ?"` | Shape and parameter name |
| Optional tensor | `"Optional[Tensor]<[768]> bias = ?"` | Nullable tensors with shape if present |
| List argument | `"List[int] stride = [32, 32]"` | List-valued parameters with concrete values |
| Scalar argument | `"int groups = 1"` | Scalar parameters with concrete values |
| Boolean argument | `"bool transposed = False"` | Boolean flags |

**Example complete entry:**

`[    "Tensor<[1, 3, 224, 224]> input = ?",    "Tensor<[768, 3, 32, 32]> weight = ?",    "Optional[Tensor]<[768]> bias = ?",    "List[int] stride = [32, 32]",    "List[int] padding = [0, 0]",    "List[int] dilation = [1, 1]",    "bool transposed = False",    "List[int] output_padding = [0, 0]",    "int groups = 1",]`

## File Generation and Maintenance

The autogenerated guard file contains a prominent warning header:

`# THIS FILE IS AUTOGENERATED BY tools/generate_guard_function_from_input_var_metric.py# DO NOT EDIT THIS FILE MANUALLY# YOU CAN DEBUG THESE INPUT VAR VIA tests/autogen-op/ALL`
**Key maintenance points:**

*   **Autogenerated file**: `to_tt_guard_autogen.py` should never be edited manually
*   **Manual file**: `to_tt_guard.py` is for hand-curated entries with detailed explanations
*   **Merge tool**: `merge_to_tt_guard_autogen.py` combines guard files (see [Guard Blocklist Management](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.4-guard-blocklist-management))
*   **Test debugging**: Failed input variations can be debugged via `tests/autogen-op/` directory

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 4-6](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L4-L6)

| Component | File | Lines | Purpose |

The guard system serves as an essential safety net, ensuring that the compilation pipeline only attempts to lower operations that are known to work correctly on Tenstorrent hardware, while gracefully falling back to PyTorch's native implementation for problematic cases.
