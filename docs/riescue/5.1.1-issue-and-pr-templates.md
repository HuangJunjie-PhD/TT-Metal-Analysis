# Issue and PR Templates

This document describes the GitHub issue and pull request templates available in the RiESCUE repository. These templates standardize how contributors report bugs, propose features, and start discussions. For broader contributing guidelines and code standards, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/riescue/5.1-contributing-guidelines). For information about the code review and CI/CD process that follows after creating issues or PRs, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline).

## Purpose and Structure

The RiESCUE repository provides pre-configured templates stored in `.github/ISSUE_TEMPLATE/` and `.github/DISCUSSION_TEMPLATE/` directories. When users create new issues or discussions on GitHub, they are prompted to select from these templates, which provide structured forms to capture necessary information.

## Bug Report Template

The bug report template ([.github/ISSUE_TEMPLATE/bug_report.md](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1)) uses GitHub's YAML front matter format to configure issue metadata and provides a structured markdown body.

### Front Matter Configuration

The template header specifies:

*   **name**: "Report a bug" - Display name in template selector
*   **about**: Brief description shown to users
*   **title**: Pre-filled title with "[Bug Report] ..." prefix
*   **labels**: Automatically applies `bug` label
*   **assignees**: Left empty for manual assignment

Sources: [.github/ISSUE_TEMPLATE/bug_report.md 1-8](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1#L1-L8)

### Required Sections

The bug report template requests four key pieces of information:

#### 1. Test Case Attachment

`**Can you attach an example test case and command that shows the issue?**`
This prompts users to provide `.rasm` files or other test inputs that reproduce the bug.

#### 2. Environment Information

`**Please complete the following environment information:**- OS: [e.g. Ubuntu 20.04]- Version of software (eg. commit)- Container: [Yes / No]`
Captures whether the user is running in a container (see [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system)) and the specific commit hash from the repository.

#### 3. Minimum Reproduction Steps

`**Minimum Reproduction Steps**command line to reproduce:`
./riescued.py --foo

Provides a code fence for exact command-line invocation. This is critical for tools like `riescued`, `riescuec`, or `ctk` (see [Core Testing Framework](https://deepwiki.com/tenstorrent/riescue/2-core-testing-framework)).

#### 4. Expected Behavior

Captures what the user anticipated versus actual results.

Sources: [.github/ISSUE_TEMPLATE/bug_report.md 10-24](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1#L10-L24)

## Feature Proposal Template

The feature proposal template ([.github/ISSUE_TEMPLATE/feature_proposal.md](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/feature_proposal.md?plain=1)) follows a similar structure but focuses on forward-looking improvements rather than bug fixes.

### Front Matter Configuration

*   **name**: "Propose a feature"
*   **title**: "[Feature Request] ..." prefix
*   **labels**: `feature-request`

Sources: [.github/ISSUE_TEMPLATE/feature_proposal.md 1-8](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/feature_proposal.md?plain=1#L1-L8)

### Required Sections

#### 1. Problem Description

`**Is your feature request related to a problem? Please describe.**A clear and concise description of what the problem is. Ex. I'm always frustrated when [...]`
Encourages users to articulate the underlying problem, not just the solution.

#### 2. Proposed Solution

`**Describe the solution you'd like**A clear and concise description of what you want to happen.`
Details the desired feature implementation.

#### 3. Alternative Approaches

`**Describe alternatives you've considered**A clear and concise description of any alternative solutions or features you've considered.`
Documents design trade-offs and rejected approaches.

#### 4. Additional Context

Open-ended section for screenshots, references to similar features in other projects, or use cases.

Sources: [.github/ISSUE_TEMPLATE/feature_proposal.md 10-20](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/feature_proposal.md?plain=1#L10-L20)

## Discussion Template

Unlike issue templates which use markdown with YAML front matter, the discussion template ([.github/DISCUSSION_TEMPLATE/general.yml](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml)) uses pure YAML with GitHub Forms syntax. This provides a more structured, form-based interface with validation.

### YAML Forms Structure

**Discussion Template YAML Forms Architecture**

Sources: [.github/DISCUSSION_TEMPLATE/general.yml 1-55](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L1-L55)

### Field Types and Validation

The discussion template demonstrates all available GitHub Forms field types:

| Field Type | ID | Purpose | Validation |
| --- | --- | --- | --- |
| `markdown` | N/A | Static text/headers | None |
| `textarea` | `improvements` | Multi-line input for top 3 improvements | Required |
| `input` | `has-id` | Single-line suggestions | Required |
| `dropdown` | `download` | Select improvement area | Required |
| `checkboxes` | N/A | Confirmation checkboxes | One required |

Sources: [.github/DISCUSSION_TEMPLATE/general.yml 4-50](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L4-L50)

### Textarea Configuration Example

`- type: textarea  id: improvements  attributes:    label: Top 3 improvements    description: "What are the top 3 improvements we could make to this project?"    value: |      1.      2.      3.      ...    render: bash  validations:    required: true`
The `render: bash` attribute provides syntax highlighting for the input ([.github/DISCUSSION_TEMPLATE/general.yml 8-20](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L8-L20)).

### Dropdown Options

The dropdown field provides four predefined options reflecting common areas for improvement:

*   Documentation
*   Pull request review time
*   Bug fix time
*   Release cadence

These align with the project's development workflow concerns (see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline) for automated review processes).

Sources: [.github/DISCUSSION_TEMPLATE/general.yml 33-43](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L33-L43)

## Best Practices for Using Templates

### For Bug Reports

1.   **Always include reproduction command**: Reference the specific tool (`riescued`, `riescuec`, `ctk`) and all command-line arguments. Example from template:

`./riescued.py --foo`
2.   **Specify container usage**: The "Container: [Yes / No]" field is critical because the container environment (see [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system)) has specific versions of Whisper, Spike, and the RISC-V toolchain.

3.   **Attach test files**: If the bug involves test generation, attach the `.rasm` source file or provide the exact test plan name for RiescueC (see [RiescueC](https://deepwiki.com/tenstorrent/riescue/2.2-riescuec:-compliance-test-generator)).

4.   **Commit hash over version numbers**: Use Git commit SHA instead of vague version descriptions for reproducibility.

Sources: [.github/ISSUE_TEMPLATE/bug_report.md 12-21](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1#L12-L21)

### For Feature Proposals

1.   **Start with the problem**: Template explicitly asks "Is your feature request related to a problem?" before asking for solutions. This helps maintainers understand context.

2.   **Document alternatives**: The "Describe alternatives you've considered" section helps avoid duplicate discussions and shows design thinking.

3.   **Link to related issues**: If proposing enhancements to existing features like RiescueD directives (see [Test Directives and Syntax](https://deepwiki.com/tenstorrent/riescue/2.1.2-test-directives-and-syntax)), reference existing issue numbers.

Sources: [.github/ISSUE_TEMPLATE/feature_proposal.md 10-17](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/feature_proposal.md?plain=1#L10-L17)

### For Discussions

1.   **Use structured fields**: The YAML form enforces structure, making it easier to scan multiple discussions for common themes.

2.   **Required vs. optional**: Note which fields have `validations: required: true` and must be completed.

3.   **Dropdown selections**: Pre-defined options in the dropdown field ([.github/DISCUSSION_TEMPLATE/general.yml 36-41](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L36-L41)) help categorize feedback consistently.

Sources: [.github/DISCUSSION_TEMPLATE/general.yml 19-43](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L19-L43)

## Template Customization

Both markdown and YAML templates can be customized by modifying the respective files in `.github/ISSUE_TEMPLATE/` and `.github/DISCUSSION_TEMPLATE/`.

### Markdown Template Syntax

Markdown templates use YAML front matter delimited by `---`:

`---name: Display Nameabout: Brief descriptiontitle: "[PREFIX] ..."labels: label-nameassignees: ''--- Body content in markdown...`
Sources: [.github/ISSUE_TEMPLATE/bug_report.md 1-8](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1#L1-L8)

### YAML Forms Syntax

YAML templates use a structured format with `body` array containing field definitions:

`title: "[Prefix] "labels: ["label1", "label2"]body:  - type: field_type    id: unique_id    attributes:      label: Field Label      description: Help text    validations:      required: true/false`
The `type` field supports: `markdown`, `textarea`, `input`, `dropdown`, and `checkboxes` ([.github/DISCUSSION_TEMPLATE/general.yml 3-50](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L3-L50)).

Sources: [.github/DISCUSSION_TEMPLATE/general.yml 1-55](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/DISCUSSION_TEMPLATE/general.yml#L1-L55)

## Integration with CI/CD

While templates don't directly trigger CI/CD workflows, they support the workflow by:

1.   **Automated labeling**: The `labels` field in templates automatically categorizes issues, which can be used in GitHub Actions filters (see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline)).

2.   **Structured data**: Consistent formatting makes it easier to parse issues programmatically if needed.

3.   **Environment capture**: The environment information requested in bug reports aligns with the CI/CD environment details:

    *   Container usage matches [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system)
    *   Commit hashes enable bisection for regression tracking
    *   OS information helps identify platform-specific issues

The smoke workflow ([.github/workflows/smoke.yml](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml)) runs automatically on PRs created to fix issues filed with these templates.

Sources: [.github/ISSUE_TEMPLATE/bug_report.md 12-15](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/ISSUE_TEMPLATE/bug_report.md?plain=1#L12-L15)
