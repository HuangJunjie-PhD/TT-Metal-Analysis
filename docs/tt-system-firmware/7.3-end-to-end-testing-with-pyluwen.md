This page documents the end-to-end (E2E) testing infrastructure that validates the complete firmware stack (SMC→DMC→CMFW) using the pyluwen hardware abstraction library. These tests run on physical hardware platforms and verify the communication chains between management controllers, chip firmware, and hardware subsystems.

For unit testing on simulated platforms, see [Unit Testing with Twister](https://deepwiki.com/tenstorrent/tt-system-firmware/7.1-unit-testing-with-twister). For basic smoke tests, see [Hardware Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests). For long-duration reliability testing, see [Stress and Long-Duration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.4-stress-and-long-duration-tests).

* * *

## Purpose and Scope

E2E tests verify the complete firmware stack by:

1.   **Flashing firmware bundles** to physical hardware via `tt-flash`
2.   **Validating communication** between host, SMC, DMC, and CMFW
3.   **Testing hardware operations** including PCIe, GDDR, Ethernet, and Tensix initialization
4.   **Verifying reset mechanisms** including watchdog, SMI reset, and dirty reset scenarios
5.   **Validating firmware upgrades** from previous release versions
6.   **Testing telemetry systems** and monitoring functionality
7.   **Exercising hardware sensors** for temperature, voltage, and process detection

These tests run on actual Blackhole and Wormhole boards with JTAG access, providing comprehensive validation before firmware release.

## pyluwen Library Integration

The `pyluwen` library provides Python bindings to interact with Tenstorrent hardware. E2E tests use pyluwen as the primary interface to communicate with the SMC firmware and validate system state.

### Core pyluwen APIs Used in E2E Tests

| API Function | Purpose | Example Usage |
| --- | --- | --- |
| `pyluwen.detect_chips()` | Enumerate PCIe devices and return chip handles | Chip detection after reset |
| `chip.arc_msg(msg_type, wait, error_ok, arg0, arg1, timeout_ms)` | Send ARC message to SMC and receive response | Test message handler: `TT_SMC_MSG_TEST` |
| `chip.axi_read32(addr)` / `chip.axi_write32(addr, data)` | Read/write 32-bit AXI registers | Read `ARC_STATUS` at `0x80030060` |
| `chip.noc_read32(noc_id, x, y, addr)` / `chip.noc_write32(...)` | Access NOC-connected registers | Tensix scratch register access |
| `chip.noc_broadcast32(noc_id, addr, data)` | Broadcast write to all NOC nodes | Tensix soft reset during reset sequence |
| `chip.get_telemetry()` | Retrieve telemetry structure | Verify firmware version and status |
| `chip.as_bh().spi_read(addr, buf)` / `chip.as_bh().spi_write(addr, buf)` | SPI flash read/write operations | Firmware update and MCUboot testing |
| `chip.set_power_state(state)` | Control power management state | Set "high" power before Tensix tests |

## Test Framework Components

### Pytest Fixtures

E2E tests use pytest fixtures to manage test lifecycle and hardware state:

**Fixture Details:**

1.   **`unlaunched_dut`** (session scope): Creates `DeviceAdapter` instance with firmware bundle path. Does not flash immediately.

2.   **`launched_arc_dut`** (session scope): Flashes firmware once per test session via `unlaunched_dut.launch()`, which internally calls `tt-flash`.

3.   **`arc_chip_dut`** (function scope): Calls `wait_arc_boot(asic_id)` to verify chip is ready before each test. Deletes chip handle to avoid stale file descriptors.

**Sources:**[app/smc/pytest/e2e_smoke.py 128-198](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L128-L198)

### DeviceAdapter and Firmware Flashing

The `DeviceAdapter` class from Twister harness integrates with `tt-flash` for firmware deployment:

When running under Twister harness in CI, `DeviceAdapter` is provided by the harness. When running standalone, a fallback implementation uses `subprocess` to invoke `tt-flash`.

**Sources:**[app/smc/pytest/e2e_smoke.py 25-62](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L25-L62)

### Boot Validation Helper

The `wait_arc_boot()` function polls for chip detection and validates boot status:

The function:

1.   Attempts chip detection with 500ms polling interval
2.   Triggers PCIe rescan on detection failure
3.   Validates `ARC_STATUS` register contains magic value `0xC0DE????`
4.   Checks post code is at least `0x1D` (message queue ready)
5.   Invokes recovery procedure on timeout

## CI Pipeline Integration

### Hardware Smoke Workflow

**Job Configuration:**

The workflow:

1.   Generates test matrix from `ci_boards.json` for boards: p100a, p150a, p300a (galaxy excluded from smoke)
2.   Runs in Docker container with hardware device access
3.   Uses JTAG-capable runners (e.g., `p100a-jtag`, `p150a-jtag`)
4.   Installs tt-kmd driver matching `matrix.config.kmd_build` (2.5.0)
5.   Executes `scripts/ci/run-e2e.sh` with board name and signature key
6.   Uploads Twister logs and firmware bundle artifacts
