## Purpose and Scope

This page documents the YAML-based test configuration system that declares expected test outcomes, comparison thresholds, hardware restrictions, and pytest markers for model tests in TT-XLA. The system manages test status using three primary states: `EXPECTED_PASSING` for tests that must pass, `KNOWN_FAILURE_XFAIL` for expected failures, and `NOT_SUPPORTED_SKIP` for unsupported configurations. Architecture-specific overrides allow per-chip customization of test behavior.

The configuration system consists of:

*   **YAML config files** in `tests/runner/test_config/` organized by framework (torch, jax, torch_llm) and parallelism mode
*   **`config_loader.py`** which parses YAML files and merges configurations
*   **`constants.py`** defining `ModelTestStatus`, `BringupStatus`, and valid field names
*   **`conftest.py`** which applies pytest markers based on configuration at collection time
*   **`validate_test_config.py`** which validates YAML entries against discoverable model variants

For test execution and the `ModelTester` class hierarchy, see page 6.2. For PCC/ATOL comparison logic, see page 6.4. For CI matrix generation from pytest markers, see page 7.3.

## YAML Config File Format

Every YAML file has a single top-level key `test_config` whose value is a mapping of **test IDs** to **entry dicts**.

### Test ID Format

Test IDs are constructed from model identity and execution context and must match the pytest node ID exactly. The patterns are:

| Category | Pattern |
| --- | --- |
| Standard (inference/training) | `<family>/<task>/<fw>-<variant>-<parallelism>-<run_mode>` |
| LLM decode/prefill | `<family>/<task>/<fw>-<variant>-<phase>-seq_<N>-batch_<N>-<parallelism>-<run_mode>` |

**Examples:**

```
gpt_neo/causal_lm/pytorch-125M-single_device-inference
falcon/pytorch-3_7B_Base-tensor_parallel-inference
llama/causal_lm/pytorch-3.2_1B-llm_decode-seq_1-batch_1-single_device-inference
```

LLM parametrization values are defined in `constants.py`:

*   `LLM_SEQUENCE_LENGTHS = (128, 1024, 2048, 4096, 8192)`
*   `LLM_BATCH_SIZES = (1, 2)`
*   `LLM_PHASES = {"load_inputs_decode": "llm_decode", "load_inputs_prefill": "llm_prefill"}`

Sources: [tests/runner/test_config/constants.py 40-54](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_config/constants.py#L40-L54)

### Entry Fields

All valid field names for a config entry are defined as `ALLOWED_FIELDS` in `constants.py`. Unknown fields cause a validation error.

| Field | Type | Description |
| --- | --- | --- |
| `status` | `ModelTestStatus` | Required. Controls how pytest handles the test. |
| `reason` | `str` | Human-readable explanation for non-passing status. |
| `bringup_status` | `BringupStatus` | Optional. Categorises the failure stage. |
| `required_pcc` | `float` | Minimum PCC threshold; overrides the default of `0.99`. |
| `assert_pcc` | `bool` | Set to `false` to skip PCC comparison entirely. |
| `assert_atol` | `bool` | Enable/disable ATOL assertion. |
| `required_atol` | `float` | ATOL threshold. |
| `assert_allclose` | `bool` | Enable/disable `allclose` assertion. |
| `allclose_rtol` / `allclose_atol` | `float` | Tolerances for `allclose`. |
| `markers` | `list[str]` | Extra pytest markers (e.g., `push`, `nightly`, `extended`). |
| `supported_archs` | `list[str]` | Restricts test to named hardware targets. |
| `arch_overrides` | `dict` | Per-arch field overrides (see below). |
| `filechecks` | `list[str]` | FileCheck pattern filenames expected in `tests/filecheck/`. |
| `batch_size` | `int` | Override batch size for the test. |
| `execution_pass` | `str` | Used by training tests (`FORWARD` or `BACKWARD`). |

## Status Management System

The test configuration system uses the `ModelTestStatus` enum to control how pytest handles each test. This is the primary mechanism for managing test expectations across the codebase.

### `ModelTestStatus` Enum

Defined in [tests/runner/test_config/constants.py 19-24](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_config/constants.py#L19-L24) this enum determines the pytest behavior for each test entry. Values can be specified in YAML as enum names (e.g., `EXPECTED_PASSING`) or lowercase values (e.g., `expected_passing`).

| Status Value | Pytest Marker | Behavior | Use Case |
| --- | --- | --- | --- |
| `EXPECTED_PASSING` | `expected_passing` | Test must pass; failure blocks CI | Stable, validated tests |
| `KNOWN_FAILURE_XFAIL` | `known_failure_xfail` | Test expected to fail; xfail marker applied | Known issues tracked by GitHub issue links |
| `NOT_SUPPORTED_SKIP` | `not_supported_skip` | Test skipped via pytest.skip | Unsupported hardware/feature combinations |
| `UNSPECIFIED` | `unspecified` | No expectation enforced | Experimental or newly added tests |
| `EXCLUDE_MODEL` | _(test deselected)_ | Test never collected | Models removed from test suite |

### `BringupStatus` Classification

The optional `bringup_status` field (defined in [tests/runner/test_config/constants.py 27-33](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_config/constants.py#L27-L33)) categorizes the failure stage for non-passing tests. This classification is exposed as a pytest marker for CI filtering and triage.

| Bringup Status | Marker Applied | Failure Stage |
| --- | --- | --- |
| `FAILED_FE_COMPILATION` | `failed_fe_compilation` | Framework frontend (torch.compile / JAX tracing) |
| `FAILED_TTMLIR_COMPILATION` | `failed_ttmlir_compilation` | TT-MLIR dialect lowering (TTIR/TTNN passes) |
| `FAILED_RUNTIME` | `failed_runtime` | On-device execution (TT-Metal runtime) |
| `INCORRECT_RESULT` | `incorrect_result` | Produces wrong numerical output (PCC/ATOL failure) |

**Example YAML entry with bringup classification:**

`pocket_tts/pytorch-Pocket_tts-single_device-inference:  assert_pcc: false  status: NOT_SUPPORTED_SKIP  reason: "Aborted - https://github.com/tenstorrent/tt-mlir/issues/6786"  bringup_status: FAILED_RUNTIME`
The `bringup_status` marker allows CI to run subsets like `pytest -m failed_ttmlir_compilation` to isolate compiler issues.

### Enum Normalization

The `_normalize_enums_in_cfg` function ([tests/runner/test_config/config_loader.py 47-73](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_config/config_loader.py#L47-L73)) converts string values to Python enum members for type safety. It supports both uppercase enum names and lowercase enum values:

| YAML Input | Normalized To |
| --- | --- |
| `status: EXPECTED_PASSING` | `ModelTestStatus.EXPECTED_PASSING` |
| `status: expected_passing` | `ModelTestStatus.EXPECTED_PASSING` |
| `bringup_status: FAILED_RUNTIME` | `BringupStatus.FAILED_RUNTIME` |
| `bringup_status: failed_runtime` | `BringupStatus.FAILED_RUNTIME` |

**Enum mapping functions:**

`def _enum_map_status(value: str | ModelTestStatus) -> ModelTestStatus:    if isinstance(value, ModelTestStatus):        return value    try:        return ModelTestStatus[value.upper()]  # Try name lookup    except KeyError:        return ModelTestStatus(value)  # Fall back to value lookup`
The normalization is applied recursively to `arch_overrides` sub-dictionaries, ensuring consistent enum usage throughout the configuration tree.

The `arch_overrides` field enables per-architecture customization of test configuration. When `--arch <name>` is passed to pytest, the `ModelTestConfig` class in `conftest.py` overlays the architecture-specific configuration on top of the base entry.

### Valid Architecture Names

The `--arch` option is constrained to `ALLOWED_ARCHES` defined in [tests/runner/test_config/constants.py 7-8](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_config/constants.py#L7-L8):

`ALLOWED_ARCHES = {"n150", "p150", "n300", "n300-llmbox", "galaxy-wh-6u"}`
These values correspond to physical hardware targets:

*   `n150` / `p150`: Single-chip Grayskull and Wormhole B0 devices
*   `n300`: Two-chip Wormhole configurations (for data parallelism)
*   `n300-llmbox`: Eight-chip Wormhole configuration (for tensor parallelism)
*   `galaxy-wh-6u`: Large-scale multi-rack configurations

### Override Use Cases

**1. Per-chip PCC thresholds:**

`deit/pytorch-Base-single_device-inference:  status: EXPECTED_PASSING  required_pcc: 0.98  arch_overrides:    p150:      required_pcc: 0.97  # Lower threshold for Wormhole`
**2. Architecture-specific test status:**

`clip/pytorch-Large_Patch14-single_device-inference:  status: EXPECTED_PASSING  arch_overrides:    p150:      status: KNOWN_FAILURE_XFAIL      reason: "PCC comparison failed. Calculated: pcc=0.9832..."`
**3. Architecture restrictions via `supported_archs`:**

`falcon/pytorch-3_7B_Base-single_device-inference:  supported_archs: ["p150"]  # Only Wormhole has sufficient DRAM  status: EXPECTED_PASSING`
When `--arch n150` is specified, this test is skipped because `n150` is not in `supported_archs`. The default for tests without `supported_archs` is `["n150", "p150"]`.

## Static Validation (`validate_test_config.py`)

`validate_test_config.py` is a standalone script (no framework imports, pure AST parsing) that cross-checks YAML config entries against the universe of actually-discoverable model variants. It is invoked in CI by the `inspect-changes` action when YAML-only changes are detected in a PR.

The `ValidationResult` dataclass captures:

| Field | Meaning |
| --- | --- |
| `yaml_key_count` | Number of entries loaded from all YAML files |
| `structure_errors` | Unknown fields, bad arch_overrides keys, etc. |
| `torch_model_count` | PyTorch variants found via AST |
| `jax_model_count` | JAX variants found via AST |
| `llm_model_count` | LLM variants found via AST |
| `discovered_id_count` | Total expected test IDs generated |
| `discovered_ids` | Full set of generated IDs |

When an entry in a YAML file does not match any discovered model ID, the validator reports it as an orphan (possibly a misspelling or a deleted model). Fuzzy matching via `difflib` suggests the closest valid ID.

The script is invoked as:

```
python tests/runner/validate_test_config.py
```

It is also invoked automatically via the `inspect-changes` GitHub Action when a PR modifies only YAML config files:

`python tests/runner/validate_test_config.py`
Sources: [tests/runner/validate_test_config.py 1-60](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/validate_test_config.py#L1-L60)[.github/actions/inspect-changes/action.yml 119-122](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/actions/inspect-changes/action.yml#L119-L122)

## pytest Integration (`conftest.py`)

`pytest_collection_modifyitems` in `tests/runner/conftest.py` runs after test collection. It applies markers to every collected test item based on its config entry.

**Full marker application flow:**

Sources: [tests/runner/conftest.py 83-178](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/conftest.py#L83-L178)

### Marker Sources and Priority

| Marker type | Source | Example values |
| --- | --- | --- |
| Status marker | `meta.status` | `expected_passing`, `known_failure_xfail`, `not_supported_skip`, `unspecified` |
| Custom markers | `meta.markers` list in YAML | `push`, `nightly`, `weekly`, `large`, `extended`, `notimeout` |
| Schedule marker | `ModelGroup` of the test (RED/PRIORITY → nightly, GENERALITY → weekly) | `nightly`, `weekly` |
| Red marker | `ModelGroup.RED` | `red` |
| Bringup marker | `meta.bringup_status` (lower-cased) | `failed_runtime`, `incorrect_result` |
| Arch markers | `meta.supported_archs` or default `["n150", "p150"]` | `n150`, `p150`, `n300_llmbox` |

Custom `markers` from YAML take precedence over the schedule marker derived from `ModelGroup`; if `nightly` or `weekly` is present in `meta.markers`, the group-based schedule marker is not added.

Arch names containing hyphens (e.g., `n300-llmbox`) receive both the hyphen form (`n300-llmbox`) and an underscore-substituted form (`n300_llmbox`) as markers, since pytest marker names must be valid Python identifiers.

### `--arch` CLI Option

The `--arch` option is registered via `pytest_addoption`:

`parser.addoption(    "--arch",    action="store",    default=None,    choices=sorted(ALLOWED_ARCHES),    ...)`
Valid choices are the members of `ALLOWED_ARCHES`: `n150`, `p150`, `n300`, `n300-llmbox`, `galaxy-wh-6u`.

When provided, this value is passed to `ModelTestConfig` to resolve `arch_overrides` for all collected tests.

## PR Change Detection (`model_diff.py`)

`tests/runner/test_config/model_diff.py` is a lightweight script invoked by the `inspect-changes` action to determine whether a YAML config PR requires running push tests. It:

1.   Loads the new and old YAML files via `yaml.safe_load`
2.   Computes the symmetric difference of `test_config` keys
3.   Checks whether any modified or added entry has `push` in its `markers` list
4.   Exits with code `1` (indicating push tests are needed) if any push-marked test was added or modified

This is used to decide whether CI should run push-level tests for PRs that only touch config files without touching source code.
