## Purpose and Scope

This page documents data parallel training in TT-Blacksmith, where training data is partitioned across multiple devices while each device maintains a complete copy of the model. This approach increases training throughput by processing larger effective batch sizes across distributed hardware.

For tensor parallel training where model weights are sharded across devices, see [Tensor Parallelism](https://deepwiki.com/tenstorrent/tt-blacksmith/6.2-jax-experiments). For combining both strategies, see [Hybrid Parallelism for Large Models](https://deepwiki.com/tenstorrent/tt-blacksmith/8.1-data-parallelism#6.3). For the overall distributed training architecture, see [Multi-Chip Parallelism Architecture](https://deepwiki.com/tenstorrent/tt-blacksmith/8.1-data-parallelism#3.6).

## Conceptual Overview

Data parallelism replicates the model on each device and partitions the training batch across devices. Each device:

1.   Receives a shard of the batch (1/N of the total batch size)
2.   Performs forward and backward passes on its local data
3.   Computes gradients based on its data shard
4.   Synchronizes gradients via all-reduce operations
5.   Updates model parameters (identical across all devices)

This strategy is effective when:

*   The model fits entirely in a single device's memory
*   Batch sizes can be increased for better throughput
*   Communication overhead is acceptable (gradient synchronization per step)

**Key Advantages:**

*   Simple to implement - minimal code changes from single-device training
*   Scales linearly with number of devices (up to communication limits)
*   No model-specific sharding patterns required

**Key Trade-offs:**

*   Requires full model replication (memory overhead per device)
*   All-reduce gradient synchronization adds latency
*   Limited to models that fit on a single device

## Device Mesh Configuration for Data Parallelism

### Mesh Structure

Data parallelism uses a 2D device mesh with a dedicated "data" axis. The mesh configuration specifies:

*   `mesh_shape`: Dimensions of the device grid
*   `mesh_axis_names`: Named axes for sharding specifications

**Example Configuration:**

`mesh_shape: [1, 2]           # [model_axis, data_axis]mesh_axis_names: ["model", "data"]`
This creates a 1×2 mesh with no model parallelism and 2-way data parallelism.

### Mesh Creation Flow

**DeviceManager Initialization:**

`DeviceManager._create_mesh`[blacksmith/tools/device_manager.py 51-69](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L51-L69) reads `config.mesh_shape` and `config.mesh_axis_names`, calls `xr.global_runtime_device_count()` to enumerate available devices, and constructs an `xs.Mesh`. If `mesh_shape` is `None`, it returns `None` and single-device mode is used.

The `is_data_parallel()` method checks if the mesh has a "data" axis with more than one device:

## Batch Sharding Mechanics

### The prepare_batch Method

The `DeviceManager.prepare_batch()` method implements batch sharding for data parallelism. It marks tensors in the batch with sharding specifications that partition along the first (batch) dimension.

**Implementation:**`DeviceManager.prepare_batch`[blacksmith/tools/device_manager.py 115-125](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L115-L125) transfers each tensor to the device and, when data parallelism is active, calls `xs.mark_sharding` with a `("data", None, ...)` partition spec to shard the batch dimension across the data mesh axis.

**Sharding Specifications:**

| Tensor Shape | PartitionSpec | Meaning |
| --- | --- | --- |
| `[B]` | `("data",)` | Shard dimension 0 across data axis |
| `[B, H]` | `("data", None)` | Shard dimension 0, replicate dimension 1 |
| `[B, H, W]` | `("data", None, None)` | Shard dimension 0, replicate dimensions 1-2 |

For a 2-device data parallel setup with batch size 256:

*   Original: `[256, 784]`
*   Device 0: `[128, 784]`
*   Device 1: `[128, 784]`

## Gradient Aggregation

### PyTorch/XLA Automatic All-Reduce

In PyTorch with XLA backend, gradient synchronization occurs automatically during the optimizer step when `XLA_ALWAYS_ALLREDUCE=1` is set:

**Optimizer Step Implementation:** See `DeviceManager.optimizer_step`[blacksmith/tools/device_manager.py 127-136](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L127-L136) When a mesh is present, it calls `xm.optimizer_step(optimizer, barrier=True)` instead of `optimizer.step()`. The `barrier=True` flag forces XLA graph execution, performs the all-reduce across mesh devices, and ensures all devices are synchronized before continuing.

The `xm.optimizer_step()` function:

1.   Triggers XLA graph compilation
2.   Executes all pending operations
3.   Performs all-reduce on gradients across the mesh
4.   Applies the averaged gradients to parameters
5.   Uses a barrier to ensure synchronization

Sources: [blacksmith/tools/device_manager.py 127-136](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L127-L136)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 115](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L115-L115)

### JAX Explicit All-Gather Pattern

**JAX Gradient Aggregation Code:**

## Framework-Specific Implementation Comparison

### PyTorch vs JAX Data Parallelism

| Aspect | PyTorch/XLA | JAX |
| --- | --- | --- |
| **Batch Sharding** | `DeviceManager.prepare_batch()` with `xs.mark_sharding()` | `jax.device_put()` with `NamedSharding` |
| **Gradient Sync** | Implicit via `xm.optimizer_step()` | Explicit via `lax.all_gather()` + `jnp.mean()` |
| **Mesh Creation** | `xs.Mesh(device_ids, mesh_shape, axis_names)` | `Mesh(jax.devices("tt"), axis_names)` |
| **Sharding Spec** | `("data", None, ...)` tuple | `PartitionSpec("dp")` |
| **Compilation** | `torch_xla.sync()` triggers | `jax.jit()` decorator |
| **Loss Workaround** | Custom `mse_loss()` with keepdim | Custom aggregation in training step |

## Loss Function Considerations

### Scalar Loss Issue

Both PyTorch and JAX implementations require custom loss functions to work around tensor shape requirements in data parallel mode:

**Problem:** Standard loss functions (e.g., `nn.MSELoss`, `nn.CrossEntropyLoss`) with `reduction='mean'` return scalars, but data parallel operations require losses with shape `[1, 1]` to maintain proper gradients across devices.

**PyTorch Solution (MSE Loss):**

`# Lines 7-14 in data_parallel/utils.pydef mse_loss(outputs: torch.Tensor, targets: torch.Tensor) -> torch.Tensor:    # Workaround for nn.MSELoss - it returns a scalar (reduction='mean'),    # but data parallel operations require loss shape [1, 1] (keepdim=True).    # github issue: https://github.com/tenstorrent/tt-xla/issues/1993    loss = (outputs - targets).pow(2)    loss = loss.mean(dim=1, keepdim=True)    loss = loss.mean(dim=0, keepdim=True)    return loss`
**JAX Solution (Cross-Entropy):**

`# Lines 72-73 in jax data parallel exampledef cross_entropy(logits, y):    return -jnp.mean(jnp.sum(y * jax.nn.log_softmax(logits), axis=-1))`
The JAX implementation uses explicit `lax.all_gather` on the loss value to ensure proper aggregation across devices.

Sources: [blacksmith/experiments/torch/mnist/data_parallel/utils.py 7-14](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/utils.py#L7-L14)[blacksmith/experiments/jax/mnist/multi_chip/data_parallel/test_pure_jax_mnist.py 72-89](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/jax/mnist/multi_chip/data_parallel/test_pure_jax_mnist.py#L72-L89)

## Performance Considerations

### Effective Batch Size

In data parallel training, the **effective batch size** is:

```
effective_batch_size = per_device_batch_size × num_data_parallel_devices
```

For a configuration with `mesh_shape: [1, 2]` and `batch_size: 256`:

*   Device 0 processes: 128 samples
*   Device 1 processes: 128 samples
*   Total effective batch: 256 samples per step

**Configuration trade-off:** Larger per-device batch sizes reduce communication overhead but may not fit in device memory.

### Communication Overhead

Data parallelism introduces two communication operations per training step:

1.   **Gradient all-reduce:** After backward pass, gradients must be averaged across devices
2.   **Parameter synchronization:** Ensures all devices have identical model parameters

**Scaling efficiency** depends on:

*   Gradient size (number of model parameters)
*   Network bandwidth between devices
*   Compute-to-communication ratio

**Best practices:**

*   Use gradient accumulation to increase compute time relative to communication
*   Ensure batch size per device is large enough to saturate compute
*   Monitor all-reduce time vs. forward/backward time

Sources: [blacksmith/tools/device_manager.py 128-137](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L128-L137)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training_dp.yaml 15](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training_dp.yaml#L15-L15)

`for epoch in range(num_epochs):    for i in range(num_batches):        # Create batch on CPU, transfer with sharding        x_batch = jax.device_put(x_batch_host, sharding_config.data_sharding)        y_batch = jax.device_put(y_batch_host, sharding_config.data_sharding)                # Training step with explicit gradient aggregation        loss, grads, logits = training_step_jit(params, x_batch, y_batch, learning_rate)                # Optimizer step        params = optimizer_step_jit(params, grads, learning_rate)`

## Troubleshooting

### Common Issues

**1. Gradients not synchronizing**

*   Symptom: Models diverge across devices
*   Check: `XLA_ALWAYS_ALLREDUCE=1` environment variable is set
*   Check: Using `xm.optimizer_step()` instead of `optimizer.step()`

**2. Shape mismatches in loss computation**

*   Symptom: Runtime errors about incompatible tensor shapes
*   Solution: Use custom loss functions with `keepdim=True` (see Loss Function Considerations)

**3. Device mesh creation fails**

*   Symptom: `RuntimeError` about device count
*   Check: `mesh_shape` product matches available devices
*   Example: `mesh_shape: [1, 2]` requires exactly 2 devices

**4. Batch size issues**

*   Symptom: Out-of-memory errors or incorrect gradients
*   Remember: `batch_size` in config is per-device, not total
*   Total samples per step = `batch_size × num_data_parallel_devices`

**5. JAX all_gather errors**

*   Symptom: Errors about collective operations
*   Check: `axis_name` in `lax.all_gather()` matches mesh axis name
*   Check: Shardy partitioner is enabled
