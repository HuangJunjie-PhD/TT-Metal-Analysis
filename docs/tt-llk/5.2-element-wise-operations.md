## Purpose and Scope

This page documents element-wise (eltwise) operations in the math unit, which perform operations on corresponding elements across tiles. Element-wise operations include binary operations (add, subtract, multiply) and unary operations (datacopy). These operations support various broadcast modes and can reuse destination registers as sources.

For SFPU-based unary operations (exp, log, etc.), see [SFPU Operations Library](https://deepwiki.com/tenstorrent/tt-llk/5.3-sfpu-operations-library). For reduction operations, see [Reduction and Pooling Operations](https://deepwiki.com/tenstorrent/tt-llk/5.5-reduction-and-pooling-operations).

## Overview of Element-wise Operations

Element-wise operations process tiles by operating on corresponding elements in source registers and writing results to destination registers. The math unit supports two categories:

| Operation Category | Operations | Template Parameter | Use Case |
| --- | --- | --- | --- |
| **Binary** | Add, Subtract, Multiply | `EltwiseBinaryType` | Combine two input tiles |
| **Unary Datacopy** | A→D, B→D | `DataCopyType` | Move/broadcast single input tile |

All element-wise operations support:

*   **Multiple broadcast modes**: None, Row, Column, Scalar
*   **Destination reuse**: Using previous output as an input
*   **Partial tiles**: Operating on 1, 2, or 4 faces
*   **High fidelity modes**: Multi-phase computation for precision control

Sources: [llk_math_eltwise_binary.h 1-408](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L1-L408)[llk_math_eltwise_unary_datacopy.h 1-626](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L1-L626)

## Element-wise Unary Datacopy Operations

### Datacopy Types

Unary datacopy operations move data from source registers to destination:

| DataCopyType | Source | Description |
| --- | --- | --- |
| `A2D` | SrcA → Dest | Move from SrcA register to destination |
| `B2D` | SrcB → Dest | Move from SrcB register to destination |

### Unary Datacopy Data Flow

**Sources:**[llk_math_eltwise_unary_datacopy.h 22-174](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L22-L174)[llk_math_eltwise_unary_datacopy.h 258-363](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L258-L363)

### A2D Datacopy Implementation

For `A2D` datacopy (SrcA to Dest):

**Non-FP32 mode**: Uses `MOVA2D` instruction with 8-row granularity [llk_math_eltwise_unary_datacopy.h 278](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L278-L278)

**FP32 dest accumulation mode**: Uses `ELWADD` with SrcB cleared to handle unpacking FP16 data into FP32 destination [llk_math_eltwise_unary_datacopy.h 272](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L272-L272)

**UInt8 format**: Also uses `ELWADD` for proper datum handling [llk_math_eltwise_unary_datacopy.h 269-274](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L269-L274)

### B2D Datacopy Implementation

For `B2D` datacopy (SrcB to Dest):

**Standard operation**: Uses `MOVB2D` instruction with 4-row granularity [llk_math_eltwise_unary_datacopy.h 358](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L358-L358)

**Column broadcast**: Requires two separate MOP runs for full 32x32 tile [llk_math_eltwise_unary_datacopy.h 290-302](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L290-L302)

*   First run processes 2 outer loops (16 rows)
*   Manual `SrcB` counter clear between runs
*   Second run processes remaining 16 rows

**Scalar broadcast**: Single MOP run covering all faces, uses `ELWADD` for non-UInt16 formats [llk_math_eltwise_unary_datacopy.h 308-333](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L308-L333)

**UInt16 special case**: Uses `MOVB2D` instead of `ELWADD` because FPU interprets some UInt16 values as float with exp=0 [llk_math_eltwise_unary_datacopy.h 336-342](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L336-L342)

**Sources:**[llk_math_eltwise_unary_datacopy.h 258-363](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L258-L363)[llk_math_eltwise_unary_datacopy.h 176-256](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L176-L256)

## Broadcast Modes

### Broadcast Type Enumeration

The `BroadcastType` enum defines broadcast patterns:

| Mode | Description | Unpacker Behavior |
| --- | --- | --- |
| `NONE` | No broadcast, 1:1 element pairing | Both SrcA and SrcB advance normally |
| `ROW` | Single row broadcast to all rows | SrcB Z counter doesn't increment |
| `COL` | Single column broadcast to all columns | SrcB uses special Z addressing |
| `SCALAR` | Single element broadcast to all elements | SrcB Z counter stays at 0 |

### Row Broadcast Pattern

**Unpacker configuration for row broadcast:**

*   MOP uses `unpack_srcb_inc_z_0` to prevent Z counter increment [llk_unpack_A.h 50-51](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L50-L51)
*   End operation sets `srcb_clear_z` to reset counter [llk_unpack_A.h 56](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L56-L56)[llk_unpack_A.h 111-117](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L111-L117)
*   For narrow tiles, `unpack_srcb_no_z_inc` is used [llk_unpack_AB.h 67](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_AB.h#L67-L67)

**Sources:**[llk_unpack_A.h 107-118](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L107-L118)[llk_unpack_AB.h 64-72](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_AB.h#L64-L72)

### Column Broadcast Pattern

Column broadcast unpacks a single column and broadcasts it across all columns of the output tile.

**Unpacker configuration:**

*   Sets `srcb_set_z_2` to advance Z counter by 2 faces [llk_unpack_A.h 55](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L55-L55)
*   Uses specialized unpack pattern for 16x16 faces [llk_unpack_A.h 97-106](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L97-L106)

**Math execution:**

*   Requires two separate MOP runs due to 2 outer loop limitation [llk_math_eltwise_binary.h 142-163](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L142-L163)
*   Manual SrcB counter clearing between runs via `TTI_SETRWC(p_setrwc::CLR_B, ...)`[llk_math_eltwise_binary.h 152](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L152-L152)

**Sources:**[llk_unpack_A.h 97-106](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L97-L106)[llk_math_eltwise_binary.h 142-163](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L142-L163)

### Scalar Broadcast Pattern

Scalar broadcast uses a single element from SrcB and broadcasts it to all elements of the tile.

**Unpacker configuration:**

*   Single outer loop, multiple inner loops [llk_unpack_A.h 119-127](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L119-L127)
*   Uses `unpack_srcb_inc_z_0` to prevent counter increment [llk_unpack_A.h 124](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L124-L124)
*   Sets `unpack_srca_zerosrc_set_dvalid` as start operation [llk_unpack_A.h 126](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L126-L126)

**Math instruction:**

*   Uses `p_elwise::SRCB_BCAST_ALL` broadcast type [llk_math_eltwise_binary.h 227](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L227-L227)
*   For non-UInt16: `ELWADD` instruction [llk_math_eltwise_binary.h 329](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L329-L329)
*   For UInt16: `MOVB2D` with `MOV_8_ROW_BRCST_D0_BRCST`[llk_math_eltwise_binary.h 315-316](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L315-L316)

**Sources:**[llk_unpack_A.h 119-128](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L119-L128)[llk_math_eltwise_binary.h 308-333](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L308-L333)

## Destination Reuse Pattern

### Destination Zeroing for ELWMUL

When reusing destination with `ELWMUL`, the destination must be cleared before the operation [llk_math_eltwise_binary.h 92-106](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L92-L106):

```

- Calculate base_address from dest_buffer_base and dst_index
- For FP32 mode: Clear 8x16 datums twice per 16x16 face (line 96-101)
- For FP16 mode: Clear full 16x16 face once (line 104)
```

## Address Modifier Configuration

### Binary Operation Address Modifiers

The `eltwise_binary_configure_addrmod` function [llk_math_eltwise_binary.h 19-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L19-L43) configures four address modifiers:

| ADDR_MOD | SrcA Incr | SrcB Incr | Dest Incr | Purpose |
| --- | --- | --- | --- | --- |
| `ADDR_MOD_0` | 8 | 0/8 (broadcast dependent) | 8 | Inner loop operation |
| `ADDR_MOD_1` | 0 | 0 | 0 | No modification (reuse) |
| `ADDR_MOD_2` | 0, clr=1 | 0, clr=1 | 0, cr=1 | Reset + carry return + fidelity incr |
| `ADDR_MOD_3` | 0, clr=1 | 0, clr=1 | 8, c_to_cr=1 | Reset + dest carry to CR + fidelity clr |

**SrcB increment variations:**

*   `NONE` or `COL` broadcast: `incr = 8`[llk_math_eltwise_binary.h 22](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L22-L22)
*   `ROW` or `SCALAR` broadcast: `incr = 0`[llk_math_eltwise_binary.h 22](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L22-L22)

**Fidelity control:**

*   `ADDR_MOD_2`: Increments fidelity phase by `FIDELITY_INCREMENT`[llk_math_eltwise_binary.h 37](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L37-L37)
*   `ADDR_MOD_3`: Clears fidelity phase counter [llk_math_eltwise_binary.h 41](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L41-L41)

**Sources:**[llk_math_eltwise_binary.h 19-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L19-L43)

## MOP Programming

### Binary Operation MOP Structure

The `eltwise_binary_configure_mop` function [llk_math_eltwise_binary.h 210-253](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L210-L253) programs the MOP replay buffer:

**Outer loop calculation**[llk_math_eltwise_binary.h 223](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L223-L223):

```
- binary_reuse_dest != NONE: outerloop = 1
- COL broadcast: outerloop = 2
- Otherwise: outerloop = num_faces
```

**Inner loop**: Fixed at 2 iterations (16 rows / 8 rows per operation) [llk_math_eltwise_binary.h 220](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L220-L220)

**ELWADD/ELWSUB**[llk_math_eltwise_binary.h 233-237](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L233-L237):

*   Standard MOP with end operation `SETRWC(CLR_SRC, CR_AB, ...)`
*   `CLR_SRC` is `CLR_A` for normal operation, `CLR_AB` except for COL/SCALAR [llk_math_eltwise_binary.h 231](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L231-L231)

**ELWMUL with high fidelity**[llk_math_eltwise_binary.h 239-252](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L239-L252):

*   Inner loop count = `NUM_FIDELITY_PHASES`[llk_math_eltwise_binary.h 241](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L241-L241)
*   Last inner loop instruction uses `ADDR_MOD_2` (increment fidelity) [llk_math_eltwise_binary.h 244](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L244-L244)
*   Last outer loop instruction uses `ADDR_MOD_3` (reset and clear fidelity) [llk_math_eltwise_binary.h 245](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L245-L245)

**Sources:**[llk_math_eltwise_binary.h 210-253](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_binary.h#L210-L253)

### Unary Datacopy MOP Structure

The `eltwise_unary_configure_mop` function [llk_math_eltwise_unary_datacopy.h 258-363](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L258-L363) programs datacopy MOPs:

**A2D MOP**[llk_math_eltwise_unary_datacopy.h 264-281](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L264-L281):

```
outerloop = num_faces
innerloop = 16 >> 3 (rows_per_inst==MOV_8_ROWS) or 16 (rows_per_inst==MOV_1_ROW)

Standard path: MOVA2D with ADDR_MOD_2
FP32/Int mode: ELWADD with ADDR_MOD_2 (to handle format conversion)
UInt8: ELWADD (special handling required)
```

**B2D MOP**[llk_math_eltwise_unary_datacopy.h 283-362](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L283-L362):

```
outerloop = 4 (for standard, processes 4 faces)
innerloop = 16 >> 2 (rows_per_inst==MOV_4_ROWS) or 16 (rows_per_inst==MOV_1_ROW)

Special broadcast configurations:
- COL: innerloop=2, outerloop=2, uses MOVB2D or ELWADD
- ROW: innerloop=2, broadcast_type=MOV_8_ROW_BRCST
- SCALAR: outerloop=1, innerloop=num_faces*2, broadcast_type varies
```

**End operations:**

*   A2D: `SETRWC(CLR_AB, ...)`[llk_math_eltwise_unary_datacopy.h 273](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L273-L273)
*   B2D: `SETRWC(CLR_B, CR_B, ...)` or `SETRWC(CLR_AB, ...)` for scalar [llk_math_eltwise_unary_datacopy.h 324](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L324-L324)

**Sources:**[llk_math_eltwise_unary_datacopy.h 258-363](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L258-L363)

## Fast Tilize Operation

Fast tilize is an optimized datacopy operation that uses both unpackers and the packer simultaneously to process input data more efficiently.

### Fast Tilize Architecture

**Sources:**[llk_math_eltwise_unary_datacopy.h 395-523](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L395-L523)

### Fast Tilize Address Modifier Setup

The `_llk_math_fast_tilize_addrmod_config_` function [llk_math_eltwise_unary_datacopy.h 406-467](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L406-L467) configures address modifiers for jumping between top and bottom face offsets:

| ADDR_MOD | Purpose | Configuration |
| --- | --- | --- |
| `ADDR_MOD_1` | MOVB2D increment | srcb.incr=4, dest.incr=4 |
| `ADDR_MOD_2` | MOVA2D increment | srca.incr=8, dest.incr=8 |
| `ADDR_MOD_3` | Forward jump to bottom faces | Computed based on unit_dim |
| `ADDR_MOD_0` | Backward jump to next tile | Negative offset to return |

**Bottom face offset calculation**[llk_math_eltwise_unary_datacopy.h 424](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L424-L424):

```
bottom_face_offset = (unpack_dst_format == Tf32 ? 256 : 512) / 2
- Tf32: 128 rows offset (256/2)
- Other formats: 256 rows offset (512/2)
Uses DstSync::SyncHalf, so actual capacity is half of normal
```

**Forward jump computation**[llk_math_eltwise_unary_datacopy.h 429-430](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L429-L430):

```
unit_dim 1: bottom_face_offset - (16 - 8) = offset - 8
unit_dim 2: bottom_face_offset - (32 - 8) = offset - 24
```

**Backward jump computation**[llk_math_eltwise_unary_datacopy.h 434-435](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L434-L435):

```
unit_dim 1: -bottom_face_offset + 8 (compensate for MOVA2D)
unit_dim 2: -bottom_face_offset + 4 (compensate for MOVB2D)
```

**Sources:**[llk_math_eltwise_unary_datacopy.h 406-467](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L406-L467)

### Fast Tilize Execution Pattern

The `_llk_math_fast_tilize_block_` function [llk_math_eltwise_unary_datacopy.h 524-625](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L524-L625) executes the fast tilize operation:

**unit_dim=1 execution**[llk_math_eltwise_unary_datacopy.h 534-555](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L534-L555):

```
1. MOP loop: 3 MOVA2D iterations (3-1 count)
2. MOVA2D with ADDR_MOD_3 (jump to bottom faces)
3. MOP loop: 3 MOVA2D iterations for bottom faces
4. MOVA2D with ADDR_MOD_0 (jump back)
5. SETRWC(CLR_A) to clear SrcA dvalid and reset counters
```

**unit_dim=2 execution**[llk_math_eltwise_unary_datacopy.h 556-578](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L556-L578):

```
1. MOP loop: 7 MOVA2D iterations for top faces (7-1 count)
2. MOVA2D with ADDR_MOD_3 (jump to bottom faces)
3. MOP loop: 15 MOVB2D iterations for bottom faces (15-1 count)
4. MOVB2D with ADDR_MOD_0 (jump back)
5. SETRWC(CLR_AB) to clear both dvalids
```

**unit_dim=3 execution**[llk_math_eltwise_unary_datacopy.h 579-622](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L579-L622):

```
1. Process top faces:
   - 6 MOVA2D for top 8 rows of top faces
   - 12 MOVB2D for bottom 8 rows of top faces
   - SETRWC(CLR_AB, SET_ABD) to clear sources and dest counter
   
2. Calculate bottom_face_offset manually with set_dst_write_addr
   
3. Process bottom faces:
   - 6 MOVA2D for top 8 rows of bottom faces
   - 11 MOVB2D for bottom 8 rows (11-1 for loop)
   - MOVB2D with ADDR_MOD_0 (final iteration with jump back)
   - SETRWC(CLR_AB) to clear both dvalids
```

**Sources:**[llk_math_eltwise_unary_datacopy.h 524-625](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L524-L625)

### Fast Tilize Initialization

The `_llk_math_fast_tilize_init_` function [llk_math_eltwise_unary_datacopy.h 485-507](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L485-L507) sets up the math unit:

**FP32 destination handling**[llk_math_eltwise_unary_datacopy.h 491-497](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L491-L497):

```
Non-Tf32 formats:
1. Set CFG_STATE_ID to 1
2. Wait with TTI_NOP (2 cycles)
3. Clear ALU_ACC_CTRL_Fp32_enabled
```

This manipulation is required because `MOVA2D` and `MOVB2D` partially respect the FP32 flag, creating a hardware quirk [llk_math_eltwise_unary_datacopy.h 487-489](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L487-L489)

**Cleanup in `_llk_math_fast_tilize_uninit_`**[llk_math_eltwise_unary_datacopy.h 509-522](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L509-L522):

```
1. STALLWAIT for math/SFPU completion
2. Restore ALU_ACC_CTRL_Fp32_enabled to original state
3. Reset CFG_STATE_ID to 0
4. Wait with TTI_NOP (2 cycles)
```

**Sources:**[llk_math_eltwise_unary_datacopy.h 485-522](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_eltwise_unary_datacopy.h#L485-L522)
