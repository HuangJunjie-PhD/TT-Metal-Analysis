# Reset Testing Framework

Relevant source files
*   [pyproject.toml](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml)
*   [tests/test_reset.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py)

## Purpose and Scope

This document describes the pytest-based automated testing framework for validating reset operations across all supported Tenstorrent hardware platforms. The testing framework verifies both standard PCI resets and Galaxy 6U IPMI-based resets through single-execution and stress testing scenarios.

For information about the reset operations themselves, see [Reset Architecture and Types](https://deepwiki.com/tenstorrent/tt-smi/5.1-reset-architecture-and-types), [Standard PCI Device Resets](https://deepwiki.com/tenstorrent/tt-smi/5.2-standard-pci-device-resets), and [Galaxy System Resets](https://deepwiki.com/tenstorrent/tt-smi/5.3-galaxy-system-resets). For general development and testing practices, see [Testing](https://deepwiki.com/tenstorrent/tt-smi/8.2-testing).

* * *

## Test Suite Overview

The reset testing framework is implemented in [tests/test_reset.py 1-104](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L1-L104) and provides automated validation of the reset operations documented in section 5. The suite uses pytest fixtures to detect hardware configuration and conditionally execute platform-specific tests.

The framework validates two critical aspects of reset operations:

1.   **Functional correctness**: Reset operations complete without errors
2.   **Device persistence**: All devices remain detectable after reset with matching device counts

### Test Configuration

`NUM_RESETS_STRESS_TEST = 10`
The stress test constant [tests/test_reset.py 11](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L11-L11) defines the number of consecutive reset iterations performed during stress testing. This value ensures sufficient stress testing while maintaining reasonable test execution time.

## Test Fixtures

The framework provides two pytest fixtures with session scope to characterize the hardware environment and enable conditional test execution.

### Device Detection Fixture

`@pytest.fixture(scope="session")def devices() -> List[int]:    """    Return a list of PCI indices of Tenstorrent devices.    """    return pci_scan()`
The `devices` fixture [tests/test_reset.py 13-18](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L13-L18) scans for Tenstorrent devices using `pyluwen.pci_scan()` and returns a list of PCI indices. The session scope ensures the scan occurs once per test session, improving test performance.

### Galaxy 6U Detection Fixture

`@pytest.fixture(scope="session")def is_galaxy_6u() -> bool:    """    Return True if the system is a Galaxy 6U (32 devices, all Galaxy board type).    """    devices = detect_chips_with_callback()    if len(devices) != 32:        return False     board_types = {get_board_type(f"{device.board_id():x}") for device in devices}    return board_types <= {"tt-galaxy-wh", "tt-galaxy-bh"}`
The `is_galaxy_6u` fixture [tests/test_reset.py 21-31](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L21-L31) determines if the test system is a Galaxy 6U configuration by validating:

*   **Device count**: Exactly 32 devices must be present
*   **Board types**: All devices must be either `tt-galaxy-wh` or `tt-galaxy-bh`

This detection logic uses `detect_chips_with_callback()` from `tt_tools_common` rather than `pci_scan()` to access board type information through the chip objects.

## Test Cases

The suite implements four test functions covering both PCI and Galaxy reset operations in single and stress test configurations.

### Standard PCI Reset Test

`def test_pci_reset_all_devices(devices, is_galaxy_6u):    """    Test resetting all PCI devices (invoked by tt-smi -r).     Passes if the reset is successful and the same number of devices are    detected before and after.    """    if is_galaxy_6u:        pytest.skip("Skipping PCI reset test on Galaxy")     pci_board_reset(devices, reinit=True)    post_reset_devices = pci_scan()    assert len(post_reset_devices) == len(devices)`
The `test_pci_reset_all_devices` function [tests/test_reset.py 34-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L34-L46) validates the standard PCI reset operation invoked by `tt-smi -r`. The test:

1.   Skips execution on Galaxy 6U systems (which use IPMI-based resets)
2.   Calls `pci_board_reset()` with `reinit=True` to reset all devices
3.   Re-scans for devices after reset
4.   Asserts that the post-reset device count matches the pre-reset count

### PCI Reset Stress Test

`def test_pci_reset_all_devices_stress(devices, is_galaxy_6u):    """    Test resetting all PCI devices NUM_RESETS_STRESS_TEST times in a row.     Passes if the resets are successful and the same number of devices are    detected before and after each reset.    """    if is_galaxy_6u:        pytest.skip("Skipping PCI reset stress test on Galaxy")     for _ in range(NUM_RESETS_STRESS_TEST):        pci_board_reset(devices, reinit=True)        post_reset_devices = pci_scan()        assert len(post_reset_devices) == len(devices)`
The `test_pci_reset_all_devices_stress` function [tests/test_reset.py 49-62](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L49-L62) performs `NUM_RESETS_STRESS_TEST` (10) consecutive reset cycles to validate reset stability and detect intermittent failures. Each iteration performs the same validation as the single reset test.

### Galaxy 6U Reset Test

`def test_glx_reset(devices, is_galaxy_6u):    """    Test galaxy 6U trays reset (invoked by tt-smi -glx_reset).     Passes if the reset is successful and the same number of devices are    detected before and after.    """    if not is_galaxy_6u:        pytest.skip("Skipping Galaxy reset test on non-Galaxy system")     # Expect SystemExit with return code 0 on successful reset    with pytest.raises(SystemExit) as exc_info:        glx_6u_trays_reset()     assert exc_info.value.code == 0     post_reset_devices = pci_scan()    assert len(post_reset_devices) == len(devices)`
The `test_glx_reset` function [tests/test_reset.py 65-82](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L65-L82) validates Galaxy 6U IPMI-based reset operations. The test:

1.   Skips execution on non-Galaxy systems
2.   Calls `glx_6u_trays_reset()` within a `pytest.raises(SystemExit)` context manager
3.   Validates that the exit code is 0 (successful reset)
4.   Re-scans for devices and verifies device count persistence

The `SystemExit` expectation reflects the behavior of `glx_6u_trays_reset()`, which calls `sys.exit(0)` upon successful completion [tt_smi/tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py)

### Galaxy Reset Stress Test

`def test_glx_reset_stress(devices, is_galaxy_6u):    """    Test galaxy 6U trays reset NUM_RESETS_STRESS_TEST times in a row.     Passes if the resets are successful and the same number of devices are    detected before and after each reset.    """    if not is_galaxy_6u:        pytest.skip("Skipping Galaxy reset test on non-Galaxy system")     for _ in range(NUM_RESETS_STRESS_TEST):        # Expect SystemExit with return code 0 on successful reset        with pytest.raises(SystemExit) as exc_info:            glx_6u_trays_reset()         assert exc_info.value.code == 0         post_reset_devices = pci_scan()        assert len(post_reset_devices) == len(devices)`
The `test_glx_reset_stress` function [tests/test_reset.py 85-103](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L85-L103) performs `NUM_RESETS_STRESS_TEST` (10) consecutive Galaxy reset cycles, validating IPMI command stability and device re-initialization across multiple reset operations.

## Running the Tests

### Installation

The pytest package is specified as an optional test dependency in [pyproject.toml 44-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L44-L46):

`optional-dependencies.test = [  "pytest",]`
Install test dependencies with:

`pip install -e ".[test]"`
### Execution

Run all reset tests:

`pytest tests/test_reset.py`
Run specific tests:

`pytest tests/test_reset.py::test_pci_reset_all_devicespytest tests/test_reset.py::test_glx_reset_stress`
Run with verbose output:

`pytest tests/test_reset.py -v`
### Expected Behavior

The test suite automatically detects hardware configuration and executes appropriate tests:

| System Type | Tests Executed | Tests Skipped |
| --- | --- | --- |
| Standard Wormhole/Blackhole | `test_pci_reset_all_devices` `test_pci_reset_all_devices_stress` | `test_glx_reset` `test_glx_reset_stress` |
| Galaxy 6U (32 devices) | `test_glx_reset` `test_glx_reset_stress` | `test_pci_reset_all_devices` `test_pci_reset_all_devices_stress` |

Pytest's conditional skip mechanism ensures tests run only on appropriate hardware, preventing test failures due to incompatible reset operations.

## Validation Strategy

The testing framework validates reset operations through device count persistence checks rather than detailed telemetry validation. This approach balances test reliability with execution speed:

### Device Count Assertion

All test functions use the same validation pattern:

`pre_reset_count = len(devices)  # From fixture# ... perform reset operation ...post_reset_devices = pci_scan()assert len(post_reset_devices) == pre_reset_count`
This assertion verifies that:

*   All devices successfully complete the reset sequence
*   Device re-initialization completes without failures
*   The PCIe bus topology remains stable
*   No devices become inaccessible after reset

### Exit Code Validation (Galaxy Only)

Galaxy reset tests additionally validate the `SystemExit` return code:

`with pytest.raises(SystemExit) as exc_info:    glx_6u_trays_reset() assert exc_info.value.code == 0`
This check ensures the reset operation reports successful completion before proceeding to device count validation.

### Stress Test Validation

Stress tests perform the same validation in a loop, catching intermittent failures that may not appear in single-execution tests. Common failure modes detected by stress testing include:

*   Incomplete device re-initialization
*   Race conditions in driver loading
*   Thermal management issues causing device dropout
*   IPMI command timing sensitivities
