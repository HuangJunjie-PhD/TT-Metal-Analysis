This page describes the core performance estimation algorithm implemented in `npeEngine`. The algorithm simulates NoC transfer execution using a timestep-based discrete event simulation approach, modeling congestion effects and computing bandwidth utilization. For information about transfer state lifecycle management, see [Transfer State Lifecycle](https://deepwiki.com/tenstorrent/tt-npe/4.2-transfer-state-lifecycle). For dependency tracking details, see [Dependency Tracking System](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system). For congestion modeling specifics, see [Congestion Modeling](https://deepwiki.com/tenstorrent/tt-npe/4.4-congestion-modeling).

## Algorithm Overview

The performance estimation algorithm operates as a discrete timestep simulator that processes NoC transfers over time. The simulation advances in fixed-size timesteps (configurable via `npeConfig::cycles_per_timestep`), computing the effective bandwidth and bytes transferred for each active transfer during each timestep. The algorithm accounts for congestion by computing link and NIU demand across the device grid and derating transfer rates accordingly.

## Entry Point and Configuration

The simulation is invoked through `npeEngine::runPerfEstimation()`, which may execute one or two simulations depending on configuration:

*   **Single simulation mode**: Executes with the provided `npeConfig` settings
*   **Congestion impact estimation mode**: If `cfg.estimate_cong_impact` is enabled, runs two simulations: 
    1.   Full simulation with congestion modeling enabled
    2.   Congestion-free simulation (`congestion_model_name = "none"`) to establish a baseline

The dual-simulation approach enables reporting of congestion impact as a percentage of recoverable runtime.

## Initialization Phase

Before the main simulation loop begins, the algorithm performs several initialization steps to prepare data structures and establish transfer ordering and dependencies.

### Transfer State Initialization

The `initTransferState()` method converts the hierarchical `npeWorkload` (phases containing transfers) into a flat vector of `PETransferState` objects. Each transfer state includes:

*   Transfer parameters (source, destination, size, NoC type)
*   Computed route through the NoC (obtained from `model->route()`)
*   Initial start cycle based on `phase_cycle_offset`
*   Dependency checkpoint identifiers

Sources: [tt_npe/cpp/src/npeEngine.cpp 25-47](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L25-L47)

### Transfer Queue Creation

The `createTransferQueue()` method creates a sorted queue of transfers ordered by their scheduled start time (`phase_cycle_offset`). This queue is sorted in descending order so that transfers can be efficiently popped from the end as they become ready.

**Queue Sorting Logic:**

*   Primary key: `start_cycle` (descending)
*   Secondary key: `id` (descending, for stable ordering)

The simulation loop iterates through this queue from the end, activating transfers whose `start_cycle` has been reached and whose dependencies are satisfied.

Sources: [tt_npe/cpp/src/npeEngine.cpp 49-65](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L49-L65)

### Dependency Generation

The `genDependencies()` method creates an `npeTransferDependencyTracker` and establishes ordering constraints between transfers. Dependencies are generated in two phases:

**Phase 1: Virtual Channel (VC) Dependencies**

Transfers sharing the same NIU source and first-hop link type are bucketed together. Within each bucket, transfers are sorted by start cycle, and each transfer depends on the transfer two positions earlier (stride-2 dependency). This approximates 2-VC effects where transfers must wait for earlier transfers to clear the injection point.

**Phase 2: Transfer Group Dependencies**

For multi-chip fabric transfers, explicit parent-child relationships are defined via `transfer_group_id`, `transfer_group_index`, and `transfer_group_parent` fields. Child transfers depend on their parent completing, with additional delay modeling:

*   NoC write latency (architecture-specific)
*   Ethernet hop delay for cross-device hops: `ETH_HOP_CYCLE_DELAY_BASE + ETH_HOP_CYCLE_DELAY_PER_BYTE * packet_size`

Sources: [tt_npe/cpp/src/npeEngine.cpp 67-170](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L67-L170)

### Device State Initialization

The device model initializes `npeDeviceState` with grids for tracking link and NIU demand. Grid dimensions are architecture-specific:

| Architecture | Link Grid Dimensions | NIU Grid Dimensions |
| --- | --- | --- |
| WormholeB0 | `11×9×2×2` (H-links, V-links per NOC) | `12×10×2` (cores per NOC) |
| Blackhole | `16×11×2×2` | `17×12×2` |

## Main Simulation Loop

The main loop advances time in discrete steps, activating transfers and updating their state based on computed bandwidth.

### Timestep Iteration

Each iteration of the main loop (lines 225-350) represents one timestep spanning `[start_of_timestep, curr_cycle)` where:

*   `start_of_timestep = curr_cycle - cycles_per_timestep`
*   `curr_cycle` starts at `cycles_per_timestep` and increments by `cycles_per_timestep` each iteration

**Loop Termination Conditions:**

1.   All transfers complete: `live_transfer_ids.size() == 0 && transfer_queue.size() == 0`
2.   Cycle limit exceeded: `curr_cycle > MAX_CYCLE_LIMIT` (returns error)

Sources: [tt_npe/cpp/src/npeEngine.cpp 220-351](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L220-L351)

### Transfer Activation

At the start of each timestep, the algorithm checks the `transfer_queue` (sorted by descending start cycle) and activates transfers that meet two conditions:

1.   **Time condition**: `transfer_queue[i].start_cycle <= curr_cycle`
2.   **Dependency condition**: `dep_tracker.done(transfer.depends_on, curr_cycle)`

Activated transfers are moved to `live_transfer_ids` and removed from `transfer_queue`. If a transfer has dependencies, its `start_cycle` is adjusted to be no earlier than `dep_tracker.end_cycle_plus_delay(depends_on)`.

**Activation Algorithm:**

Sources: [tt_npe/cpp/src/npeEngine.cpp 235-256](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L235-L256)

### Bandwidth Computation

The algorithm delegates bandwidth computation to the device model via `model->computeCurrentTransferRate()`. This method:

1.   Computes demand on each link and NIU from all live transfers
2.   Applies congestion modeling to derate bandwidth (if enabled)
3.   Updates `PETransferState::curr_bandwidth` for each live transfer

The device model operates on the entire `transfer_state` vector and `live_transfer_ids`, modifying bandwidth values in place. The `device_state` parameter contains grids that are populated with demand values and can be queried by subsequent operations.

**Key Data Structures:**

| Structure | Purpose | Modified By |
| --- | --- | --- |
| `transfer_state` | Vector of all transfers with current bandwidth | Device model (in-place update) |
| `live_transfer_ids` | Indices of active transfers | Main loop |
| `device_state` | Link/NIU demand grids | Device model |

### Statistics Update

After bandwidth computation, `updateSimulationStats()` aggregates demand values from `device_state` into per-timestep statistics stored in `npeStats::per_device_stats[device_id].per_timestep_stats`. Metrics include:

*   Average and maximum link demand (overall, NOC0, NOC1)
*   Average link utilization (cycles-active vs. cycles-available)
*   Average and maximum NIU demand
*   List of live transfer IDs

Sources: [tt_npe/cpp/src/npeEngine.cpp 267-273](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L267-L273)

### Transfer State Update

For each live transfer, the algorithm computes bytes transferred during the current timestep:

**Computation Steps:**

1.   **Determine active cycles**:

```
cycles_active = min(cycles_per_timestep, curr_cycle - start_cycle)
```

Adjust if dependency completed within previous timestep

2.   **Compute maximum transferable bytes**:

```
max_bytes = cycles_active * curr_bandwidth
```
3.   **Limit by remaining bytes**:

```
bytes_transferred = min(remaining_bytes, max_bytes)
total_bytes_transferred += bytes_transferred
```
4.   **Compute end cycle if complete**:

```
cycles_transferring = ceil(bytes_transferred / curr_bandwidth)
end_cycle = start_cycle_within_timestep + cycles_transferring
```
5.   **Update dependency checkpoints**: For completed transfers, call `dep_tracker.updateCheckpoint()` for each checkpoint in `required_by`

6.   **Update worst-case statistics**: Track the latest `end_cycle` per device and for the mesh as a whole

Sources: [tt_npe/cpp/src/npeEngine.cpp 276-318](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L276-L318)

### Live Transfer Compaction

After updating all transfers, completed transfers (where `total_bytes_transferred == total_bytes`) are removed from `live_transfer_ids` using `std::remove_if()` and `erase()`. This compaction ensures the next timestep only processes still-active transfers.

Sources: [tt_npe/cpp/src/npeEngine.cpp 320-327](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L320-L327)

## Simulation Termination and Finalization

The simulation terminates when `live_transfer_ids` and `transfer_queue` are both empty, indicating all transfers have completed. At termination:

1.   **Dependency validation**: Verify `dep_tracker.sanityCheck()` and `dep_tracker.allComplete()` return true

2.   **Finalize statistics**: Call `stats.finishSimulation()` to:

    *   Compute `estimated_cycles` as the worst-case `end_cycle` across all transfers
    *   Set `wallclock_runtime_us` from the simulation timer
    *   Mark device completion status

3.   **Compute summary statistics**: Call `stats.computeSummaryStats()` to aggregate per-timestep metrics into overall averages and maxima

4.   **Emit timeline file**: If `cfg.emit_timeline_file` is enabled, write visualization data via `stats.emitSimTimelineToFile()`

## Key Algorithmic Properties

### Timestep Granularity

The `cycles_per_timestep` parameter controls the time resolution of the simulation. Smaller values increase accuracy but require more iterations:

*   **Default**: 16 cycles
*   **Effect on accuracy**: Finer granularity captures intra-timestep dependency completion more accurately
*   **Effect on performance**: Simulation runtime scales inversely with timestep size

### Bandwidth Modeling

Transfer bandwidth (`curr_bandwidth` in bytes/cycle) is recomputed every timestep by the device model. Key characteristics:

*   **Congestion-sensitive**: Bandwidth decreases when links/NIUs are oversubscribed
*   **Transfer-specific**: Each transfer may have different bandwidth based on its route congestion
*   **Time-varying**: As transfers complete, congestion eases and bandwidth may increase

### Dependency Semantics

Dependencies are modeled as checkpoints that must be satisfied before a transfer activates:

*   **VC dependencies**: Ensure serial ordering at injection points
*   **Transfer group dependencies**: Model multi-hop fabric routing where later hops depend on earlier hops
*   **Checkpoint delays**: Account for NoC latency and inter-device link delays

For detailed dependency tracking implementation, see [Dependency Tracking System](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system).

### Error Conditions

The simulation can fail with the following error codes:

| Error Code | Condition | Recovery |
| --- | --- | --- |
| `EXCEEDED_SIM_CYCLE_LIMIT` | `curr_cycle > MAX_CYCLE_LIMIT` | Increase limit or check for infinite loops |
| `DEPENDENCY_GEN_FAILED` | Unsupported architecture in dependency generation | Ensure device model is fully implemented |

## Performance Characteristics

The algorithm's computational complexity is:

*   **Time per timestep**: O(T × L) where T is the number of live transfers and L is the average route length
*   **Total timesteps**: O(C / S) where C is the maximum transfer completion cycle and S is `cycles_per_timestep`
*   **Memory**: O(N) where N is the total number of transfers in the workload

The algorithm is highly efficient for typical workloads, with most simulations completing in milliseconds to seconds of wall-clock time.
