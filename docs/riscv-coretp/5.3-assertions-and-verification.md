## Purpose and Scope

This document describes the assertion and verification primitives available in the coretp framework for validating test behavior. Assertions are specialized TestStep types that enable declarative verification of processor state, memory values, and exception conditions within test scenarios.

For information about other test step types, see Memory Operations ([5.1](https://deepwiki.com/tenstorrent/riscv-coretp/5.1-memory-operations)), CSR Operations ([5.2](https://deepwiki.com/tenstorrent/riscv-coretp/5.2-csr-operations)), and Control Flow and Multi-Hart Operations ([5.4](https://deepwiki.com/tenstorrent/riscv-coretp/5.4-control-flow-and-multi-hart-operations)).

## Assertion Step Types

The coretp framework provides three assertion primitives that inherit from the `TestStep` base class:

| Assertion Type | Purpose | Primary Use Cases |
| --- | --- | --- |
| `AssertEqual` | Verify two values are equal | Memory value checks, CSR field validation, PTE bit verification |
| `AssertNotEqual` | Verify two values are not equal | Negative testing, state transition validation |
| `AssertException` | Verify specific exceptions occur | Privilege violation testing, page fault validation, illegal instruction detection |

## AssertEqual: Value Equality Verification

`AssertEqual` verifies that two values are identical. It accepts two operands that can be either immediate values or results from other TestStep operations.

### Parameters

*   **`src1`**: First operand (TestStep result or immediate value)
*   **`src2`**: Second operand (TestStep result or immediate value)

### Common Usage Patterns

**Verifying Memory Values:**

`# SID_ZICBO_018 example: verify cache block zeroingstore_data = Store(memory=mem, value=0xDEADBEEF, offset=0)cbo_zero = MemAccess(op="cbo.zero", memory=mem)load_verify = Load(op="lw", memory=mem, offset=0)assert_zero = AssertEqual(src1=load_verify, src2=0)`
**Verifying PTE Bit Updates:**

`# SID_SVADU_02 example: verify A bit set after memory accessread_leaf_pte = ReadLeafPTE(memory=mem)load_immediate_mask_check = LoadImmediateStep(imm=1 << 6)  # A bit is bit 6and_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
**Verifying Virtual Address Aliasing:**

`# SID_ZICBO_019 example: verify same PA accessed through different VAsload_va1 = Load(op="lw", memory=mem_va1, offset=0)load_va2 = Load(op="lw", memory=mem_va2, offset=0)assert_zero_va1 = AssertEqual(src1=load_va1, src2=0)assert_zero_va2 = AssertEqual(src1=load_va2, src2=0)`
**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 617-795](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L617-L795)[coretp/plans/svadu/svadu_scenarios.py 122-207](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L122-L207)

## AssertNotEqual: Value Inequality Verification

`AssertNotEqual` verifies that two values are different. This is commonly used in negative testing and state change validation.

### Parameters

*   **`src1`**: First operand (TestStep result or immediate value)
*   **`src2`**: Second operand (TestStep result or immediate value)

### Usage Example

`# Verify PTE bits have changed (A or D bit set)read_leaf_pte = ReadLeafPTE(memory=mem)load_immediate_mask_check = LoadImmediateStep(imm=(1 << 6) | (1 << 7))  # A and D bitsand_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)li_zero = LoadImmediateStep(imm=0)assert_not_equal = AssertNotEqual(src1=and_op, src2=li_zero)`

## AssertException: Exception Verification

`AssertException` verifies that executing specific operations raises the expected exception. This is critical for testing privilege violations, page faults, and illegal instruction handling.

### Parameters

*   **`cause`**: Expected exception cause (from `ExceptionCause` enum)
*   **`code`**: List of TestStep operations expected to raise the exception

### Exception Cause Types

The framework defines exception causes in the `ExceptionCause` enum:

*   `ILLEGAL_INSTRUCTION`: Opcode not recognized or disabled by configuration
*   `LOAD_PAGE_FAULT`: Page fault during load operation
*   `STORE_AMO_PAGE_FAULT`: Page fault during store/AMO operation
*   `LOAD_ACCESS_FAULT`: Access violation during load
*   `STORE_AMO_ACCESS_FAULT`: Access violation during store/AMO

### Common Usage Patterns

**Illegal Instruction Exception:**

`# SID_ZICBO_001 example: cbo.inval with menvcfg.CBIE=00menvcfg_write = CsrWrite(csr_name="menvcfg", clear_mask=0x30)  # Clear CBIE bitscbo_inval = MemAccess(op="cbo.inval", memory=mem)assert_exception = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[cbo_inval])`
**Store Page Fault:**

`# SID_ZICBO_022 example: cbo.zero with read-only pagemem = Memory(    size=0x1000,    page_size=PageSize.SIZE_4K,    flags=PageFlags.READ,  # No write permission)cbo_zero_page_fault = MemAccess(op="cbo.zero", memory=mem)assert_exception_page_fault = AssertException(    cause=ExceptionCause.STORE_AMO_PAGE_FAULT,     code=[cbo_zero_page_fault])`
**Load Page Fault (SVADU Testing):**

`# SID_SVADU_01 example: access with pte.a=0 and SVADU disabledmem = Memory(    size=0x1000,    page_size=PageSize.SIZE_4K,    flags=PageFlags.VALID | PageFlags.READ | PageFlags.WRITE,    exclude_flags=PageFlags.ACCESSED | PageFlags.DIRTY,)disable_svadu = CsrWrite(csr_name="menvcfg", clear_mask=1 << 61)load_op = Load(memory=mem)assert_load_fault = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,     code=[load_op])`
**Multiple Operations in Exception Block:**

`# SID_ZICBO_003 example: multiple cache operations raising exceptionscbo_clean = MemAccess(op="cbo.clean", memory=mem)cbo_flush = MemAccess(op="cbo.flush", memory=mem)assert_exception_clean = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[cbo_clean])assert_exception_flush = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[cbo_flush])`

## Integration with Test Scenarios

Assertions integrate seamlessly into the test scenario composition pipeline through the `TestScenario.from_steps()` method.

### Typical Test Scenario Structure

### Complete Example: Cache Block Zero Test

`# SID_ZICBO_018: Verify cache block zeroingdef test_cache_block_zero():    # Setup phase    menvcfg_write = CsrWrite(csr_name="menvcfg", set_mask=0xF0)    mem = Memory(        size=0x1000,        page_size=PageSize.SIZE_4K,        flags=PageFlags.VALID | PageFlags.READ | PageFlags.WRITE | PageFlags.EXECUTE,    )        # Operation phase    store_data = Store(memory=mem, value=0xDEADBEEF, offset=0)    cbo_zero = MemAccess(op="cbo.zero", memory=mem)        # Verification phase    load_verify = Load(op="lw", memory=mem, offset=0)    assert_zero = AssertEqual(src1=load_verify, src2=0)        return TestScenario.from_steps(        id="18",        name="SID_ZICBO_018",        description="Ensure all bytes of cache block are zeroed in UP",        env=TestEnvCfg(),        steps=[            menvcfg_write,            mem,            store_data,            cbo_zero,            load_verify,            assert_zero,  # Assertion integrated into step list        ],    )`

## Assertion Usage by Test Plan

Different test plans utilize assertions for their specific verification needs:

| Test Plan | Primary Assertions | Verification Focus |
| --- | --- | --- |
| **ZICBOM/ZICBOZ/ZICBOP** | `AssertException` (90%) `AssertEqual` (10%) | Illegal instruction detection for disabled cache ops Memory value verification after cache operations |
| **SVADU** | `AssertException` (40%) `AssertEqual` (60%) | Page fault conditions when A/D bits not set PTE bit update verification |
| **SSTC** | `AssertEqual` (primary) | Timer CSR value validation Counter enable hierarchies |
| **Paging** | `AssertException` (primary) | Page table walk fault conditions Permission violation detection |

## Multi-Step Assertion Patterns

### Pattern 1: Compute-Then-Assert

Many assertions require intermediate computation steps before verification:

`# Read PTE and mask specific bitsread_leaf_pte = ReadLeafPTE(memory=mem)load_immediate_mask_check = LoadImmediateStep(imm=1 << 6)  # A bit maskand_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
### Pattern 2: Before-After State Verification

`# Verify initial statefirst_read_leaf_pte = ReadLeafPTE(memory=mem)load_first_immediate_mask_check = LoadImmediateStep(imm=1 << 6)and_first_op = Arithmetic(op="and", src1=first_read_leaf_pte, src2=load_first_immediate_mask_check)zero = LoadImmediateStep(imm=0)assert_equal_first = AssertEqual(src1=and_first_op, src2=zero) # Perform operation that modifies stateload_op = Load(memory=mem) # Verify final stateread_leaf_pte = ReadLeafPTE(memory=mem)load_immediate_mask_check = LoadImmediateStep(imm=1 << 6)and_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
### Pattern 3: Multiple Exception Conditions

`# Test multiple operations for the same exception typecbo_clean = MemAccess(op="cbo.clean", memory=mem)cbo_flush = MemAccess(op="cbo.flush", memory=mem)assert_exception_clean = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[cbo_clean])assert_exception_flush = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[cbo_flush])`

## Best Practices

### 1. Explicit State Verification

Always verify both initial and final states when testing state transitions:

`# Good: Verify A bit is initially 0first_read = ReadLeafPTE(memory=mem)assert_initial = AssertEqual(src1=first_read & (1<<6), src2=0) # Perform operationload_op = Load(memory=mem) # Verify A bit is now 1second_read = ReadLeafPTE(memory=mem)assert_final = AssertEqual(src1=second_read & (1<<6), src2=(1<<6))`
### 2. Specific Exception Causes

Use the most specific exception cause available:

`# Good: Specific page fault typeAssertException(cause=ExceptionCause.STORE_AMO_PAGE_FAULT, code=[store_op]) # Avoid: Generic fault specification# (not shown - this is a guideline)`
### 3. Single Responsibility

Each assertion should verify one specific property:

`# Good: Separate assertions for each cache operationassert_exception_clean = AssertException(cause=ExceptionCause.ILLEGAL_INSTRUCTION, code=[cbo_clean])assert_exception_flush = AssertException(cause=ExceptionCause.ILLEGAL_INSTRUCTION, code=[cbo_flush]) # Avoid: Combining unrelated verifications in a single assertion`
