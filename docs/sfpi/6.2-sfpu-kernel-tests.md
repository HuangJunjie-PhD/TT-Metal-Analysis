## Purpose and Scope

This document describes the SFPU kernel test suite, which validates the correctness of SFPI C++ API code generation and SFPU instruction execution. These tests ensure that the SFPI compiler correctly translates high-level C++ constructs into SFPU assembly instructions and that the resulting code produces expected computational results.

For information about the overall test infrastructure and execution framework, see [Test Infrastructure Overview](https://deepwiki.com/tenstorrent/sfpi/6.1-test-infrastructure-overview). For compiler-specific optimization validation, see [Compiler Optimization Tests](https://deepwiki.com/tenstorrent/sfpi/6.3-compiler-optimization-tests).

## Test Suite Overview

The SFPU kernel tests are organized into three primary categories, each stored in architecture-specific directories (`tests/wormhole/sfpi/`, `tests/grayskull/sfpi/`):

| Test File | Purpose | Gold Standard |
| --- | --- | --- |
| `coverage.cc` | Instruction-level validation of all SFPU operations | `gold/coverage.S` |
| `ckernel.cc` | High-level mathematical kernel functions | `gold/ckernel.S` |
| `kernels.cc` | Integration tests with runtime validation | `gold/kernels_int`, `gold/kernels_float` |

### Assembly Validation Pattern

Each coverage test follows a consistent pattern where the generated assembly is compared against the gold standard. For example, `test_load_store`:

**Expected Assembly Pattern (coverage.S:10-29):**

`_Z15test_load_storev:    SFPLOAD  L2, 0, 0, 3    SFPLOAD  L0, 2, 0, 3    SFPLOAD  L1, 4, 0, 3    SFPSTORE 6, L2, 0, 3    SFPSTORE 8, L0, 0, 3    SFPSTORE 10, L1, 0, 3`
The test validates:

*   Correct instruction selection (`SFPLOAD`, `SFPSTORE`)
*   Proper register allocation (`L0`, `L1`, `L2`)
*   Correct addressing modes (immediate offsets: 0, 2, 4, 6, 8, 10)
*   Proper mode flags (mode 3 = `ADDR_MODE_NOINC`)

## Kernel Function Tests

The `ckernel.cc` file implements production-quality mathematical operations as they would appear in real SFPU kernels. These tests validate that complex algorithms compile correctly and produce accurate results.

### Mathematical Kernel Categories

| Function Template | Purpose | Key Instructions | Variants |
| --- | --- | --- | --- |
| `calculate_exponential<APPROX, ZERO_NEG, SCALE>` | e^x computation | `SFPEXEXP`, `SFPSETEXP`, `SFPMAD` | Fast/accurate, optional scaling |
| `calculate_log<APPROX, HAS_SCALE>` | ln(x) computation | `SFPEXEXP`, `SFPEXMAN`, `SFPSETEXP` | Base scaling support |
| `calculate_gelu<APPROX>` | GELU activation | `SFPLUT`, `SFPMUL`, `SFPMAD` | Tanh/erf approximations |
| `calculate_sigmoid<APPROX>` | Sigmoid activation | `SFPLUT`, `SFPADDI` | LUT-based |
| `calculate_tanh<APPROX>` | Tanh activation | `SFPLUT` | LUT-based |
| `calculate_sqrt<APPROX, ITER, RECIP_ITER>` | √x computation | `SFPMUL`, `SFPMAD` | Newton-Raphson iterations |
| `calculate_reciprocal<APPROX, ITER>` | 1/x computation | `SFPSETSGN`, `SFPEXEXP`, `SFPMAD` | Newton-Raphson iterations |
| `calculate_power<APPROX>` | x^n computation | `SFPMUL` loop | Integer exponents |

### GELU Function Test Structure

The GELU implementation showcases how kernel tests validate complex mathematical operations:

**Code Structure (ckernel.cc:278-323):**

`template <bool APPROXIMATION_MODE>inline vFloat calculate_gelu_core(vFloat in) {    if constexpr (APPROXIMATION_MODE) {        result = in;  // Use erf approximation via LUT    } else {        // Tanh approximation: sqrt(2/pi) * (x + 0.044715*x^3)        result = (in * in) * (in * s2vFloat16b(0.044715f)) + in;        result *= s2vFloat16b(0.79788f);    }    return result;} template <bool APPROXIMATION_MODE>void calculate_gelu() {    for (int d = 0; d < 8; d++) {        vFloat in = dst_reg[0];        vFloat result = calculate_gelu_core<APPROXIMATION_MODE>(in);        vFloat half_in = in * half;        result = lut(result, l0, l1, l2);  // LUT lookup        result = half_in * result + half_in;        dst_reg[0] = result;        dst_reg++;    }}`
**Expected Assembly (ckernel.S:929-952):**

`_Z14calculate_geluILb0EEvv:    TTREPLAY    0, 15, 1, 1    SFPLOAD     L4, 0, 0, 3    SFPMUL      L3, L4, L4, L9, 0       # in * in    SFPLOADI    L5, 15671, 0    SFPMUL      L5, L4, L5, L9, 0       # in * 0.044715    SFPNOP    SFPMAD      L3, L3, L5, L4, 0       # in*in * (in*0.044715) + in    SFPNOP    SFPMULI     L3, 16204, 0            # *= 0.79788    SFPMULI     L4, 16128, 0            # half_in = in * 0.5    SFPLUT      L3, 4                   # LUT lookup    SFPNOP    SFPMAD      L4, L4, L3, L4, 0       # half_in * lut_result + half_in    SFPNOP    SFPSTORE    0, L4, 0, 3`

## Architecture-Specific Test Organization

Tests are organized by Tenstorrent architecture, with each platform maintaining its own gold standards to account for instruction set differences:

| Architecture | Test Directory | Coverage Tests | Kernel Tests | Notes |
| --- | --- | --- | --- | --- |
| Wormhole | `tests/wormhole/sfpi/` | Full suite (35 tests) | Full suite (25+ kernels) | 51 SFPU operations, `ADDR_MODE_NOINC=3` |
| Blackhole | `tests/blackhole/sfpi/` | Full suite | Full suite + enhanced ops | 54 operations, `ADDR_MODE_NOINC=7` |
| Grayskull | `tests/grayskull/sfpi/` | Reduced suite | Core kernels | Limited to essential operations |

The test framework uses preprocessor macros (`__riscv_xtttensixwh`, `__riscv_xtttensixbh`, `__riscv_xtttensixgs`) to conditionally compile architecture-specific code paths.

This test ensures proper condition stack management across multiple nesting levels.
