## Purpose and Scope

This document describes the memory management subsystem within RiESCUE's test generation framework, specifically how tests can declare memory requirements, generate random addresses, and create page table mappings. This system is part of RiescueD's test generation internals and provides directives that allow test authors to specify memory layouts and virtual-to-physical address mappings.

For information about the overall test generation architecture, see [Test Generation Internals](https://deepwiki.com/tenstorrent/riescue/7.1-test-generation-internals). For configuration of CPU memory maps, see [CPU Configuration](https://deepwiki.com/tenstorrent/riescue/7.2.1-cpu-configuration).

## Overview

The memory management system in RiESCUE enables test authors to:

1.   **Generate random addresses** with constraints (alignment, size, address type)
2.   **Reserve memory regions** to prevent allocation conflicts
3.   **Define page table mappings** between linear and physical addresses
4.   **Configure page table attributes** (permissions, page sizes, security)
5.   **Initialize memory contents** at specific addresses

These capabilities are exposed through custom assembly directives that are parsed by the `Parser` class, stored in the `Pool`, and processed by the `Generator` to produce actual memory structures and page tables in the generated test.

Sources: [riescue/dtest_framework/README.md 13-16](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/README.md?plain=1#L13-L16)[riescue/dtest_framework/parser.py 20-38](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L20-L38)

## Random Address Generation

### Directive Syntax

The `;#random_addr()` directive generates random addresses with specified constraints:

`;#random_addr(name=addr1, type=linear, size=0x1000, and_mask=0xFFFFF000);#random_addr(name=addr2, type=physical32, size=0x2000, align=0x1000);#random_addr(name=io_addr, type=physical, size=0x100, io=true)`
### Parameters

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `name` | string | Unique identifier for the address | Required |
| `type` | string | Address type: `linear`, `linear32`, `physical`, `physical32` | Required |
| `size` | hex | Size of memory region | `0x1000` |
| `align` | hex | Alignment requirement (older parameter) | `0xFFFFFFFFFFFFF000` |
| `and_mask` | hex | Mask to apply to generated address | `0xFFFFFFFFFFFFFFFF` |
| `or_mask` | hex | Bits to set in generated address | `0x0` |
| `io` | bool | Whether this is an I/O address | `false` |
| `in_pma` | bool | Whether address must be in PMA region | `false` |
| `pma_*` | various | PMA (Physical Memory Attributes) settings | None |
| `secure` | bool | Whether this is a secure address | `false` |

### Address Types

The `type` parameter supports the following formats:

*   **`linear`**: Virtual/linear address (full address width)
*   **`linear32`**: 32-bit linear address
*   **`physical`**: Physical address (full address width)
*   **`physical32`**: 32-bit physical address

The address width can be explicitly specified by appending a number: `linear48`, `physical39`, etc.

This dataclass stores all parameters parsed from a `random_addr` directive and is used during generation to allocate actual addresses.

## Memory Reservation

### Directive Syntax

The `;#reserve_memory()` directive prevents the generator from allocating addresses in specific memory regions:

`;#reserve_memory(addr_type=linear, start_addr=0x80000000, size=0x10000);#reserve_memory(addr_type=physical, start_addr=0x40000000, size=0x1000)`
### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `addr_type` | string | Type of address: `linear` or `physical` |
| `start_addr` | hex | Starting address of reserved region |
| `size` | hex | Size of reserved region |

### ParsedReserveMemory Structure

The parsed reservation is stored as:

| Field | Type | Description |
| --- | --- | --- |
| `addr` | string | Address (not used in current implementation) |
| `addr_type` | string | Address space type |
| `size` | int | Size in bytes |
| `and_mask` | int | Mask value (default `0xFFFFFFFFFFFFF000`) |
| `name` | string | Auto-generated: `__res_mem_{type}{start_addr}` |
| `start_addr` | int | Starting address |

## Page Mappings

### Directive Syntax

The `;#page_mapping()` directive defines virtual-to-physical address translations with page table attributes:

`;#page_mapping(lin_name=vaddr1, phys_name=paddr1, v=1, r=1, w=1, x=0);#page_mapping(lin_name=vaddr2, phys_name=paddr2, pagesize=[4kb, 2mb]);#page_mapping(lin_name=vaddr3, phys_name=paddr3, u=1, secure=1)`
### Core Parameters

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `lin_name` | string | Name of linear/virtual address | Required |
| `phys_name` | string | Name of physical address | Required |
| `lin_addr` | hex | Explicit linear address | None |
| `phys_addr` | hex | Explicit physical address | None |

### Permission Attributes

Page table entry permissions can be specified for different page table levels:

| Attribute | Levels | Description |
| --- | --- | --- |
| `v` | leaf, nonleaf, level0-4 | Valid bit |
| `r` | leaf, nonleaf, level0-4 | Read permission |
| `w` | leaf, nonleaf, level0-4 | Write permission |
| `x` | leaf, nonleaf, level0-4 | Execute permission |
| `u` | leaf, nonleaf, level0-4 | User mode accessible |
| `a` | leaf, nonleaf, level0-4 | Accessed bit |
| `d` | leaf, nonleaf, level0-4 | Dirty bit |
| `g` | leaf, nonleaf, level0-4 | Global bit |

Attributes can be specified with suffixes for different page table contexts:

*   `v_nonleaf`: Valid bit for non-leaf entries
*   `v_leaf_gleaf`: Valid bit for leaf entries where guest entry is also leaf
*   `v_nonleaf_gnonleaf`: Valid bit for non-leaf entries where guest entry is non-leaf

### Page Size Configuration

The `pagesize` parameter accepts a list of allowed page sizes:

`;#page_mapping(lin_name=addr1, phys_name=paddr1, pagesize=[4kb, 2mb, 1gb])`
Supported sizes: `4kb`, `2mb`, `1gb`, `512gb`, `256tb`, `any`

For virtualized environments, additional parameters control guest-stage page sizes:

*   `gstage_vs_leaf_pagesize`: Guest-stage leaf page sizes
*   `gstage_vs_nonleaf_pagesize`: Guest-stage non-leaf page sizes

### Additional Attributes

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `pbmt` | int | Page-based memory type | `0` |
| `n` | bool | NAPOT (Naturally Aligned Power-of-Two) | `false` |
| `rsw` | int | Reserved for software (2 bits) | `0` |
| `reserved` | int | Reserved bits | `0` |
| `secure` | bool | Secure mapping (marks addresses as secure) | `false` |
| `page_maps` | list | List of page map names to include this mapping | `[]` |

## Memory Initialization

### Directive Syntax

The `;#init_memory` directive marks addresses that need to be initialized:

`;#init_memory @data_addr;#init_memory @buffer:map1,map2`
### Syntax Formats

1.   **Simple format**: `@address_name` - Initialize the address in all page maps
2.   **Map-specific format**: `@address_name:map1,map2` - Initialize only in specified maps

### Processing

The parser extracts the address name and optional map list, then adds each to the pool:

**Memory Initialization Parsing**

When maps are specified, the parser creates separate init entries for each map by appending the map name to the address name.

Sources: [riescue/dtest_framework/parser.py 625-641](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L625-L641)

## Processing Flow

### Parsing Phase

**Memory Directive Parsing Sequence**

The parser processes each line sequentially, detecting memory-related directives through string prefix matching and routing them to specialized parsing methods.

Sources: [riescue/dtest_framework/parser.py 52-98](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L52-L98)

### Regex-Based Extraction

All memory directives use regex patterns to extract parameters:

| Directive | Pattern |
| --- | --- |
| `reserve_memory` | `^;#(reserve_memory)\((.+)\)` |
| `random_data` | `^;#(random_data)\((.+)\)` |
| `random_addr` | `^;#(random_addr)\((.+)\)` |
| `page_mapping` | `^;#(page_mapping)\((.+)\)` |
| `page_map` | `^;#(page_map)\((.+)\)` |

After matching, arguments are split by commas and parsed into key-value pairs using `split("=")`.

Sources: [riescue/dtest_framework/parser.py 100-101](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L100-L101)[riescue/dtest_framework/parser.py 117-118](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L117-L118)[riescue/dtest_framework/parser.py 133-134](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L133-L134)[riescue/dtest_framework/parser.py 169](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L169-L169)[riescue/dtest_framework/parser.py 575-576](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L575-L576)

### Generation Phase Integration

The parsed memory structures are consumed by the `Generator` during test generation:

**Memory Generation Pipeline**

The generator uses parsed structures from the pool to resolve dependencies, allocate addresses avoiding reserved regions, construct page tables, and initialize memory contents.

### Secure Address Propagation

When a page mapping is marked as secure, the parser propagates this attribute to the referenced addresses:

`# From parse_page_mappings() when secure=1if ppm_inst.secure:    parsed_addrs = self.pool.get_parsed_addrs()    for lin_addr, phys_addr in generate_time_process_args:        if lin_addr in parsed_addrs:            parsed_addr = parsed_addrs[lin_addr]            parsed_addr.secure = True            self.pool.add_parsed_addr(parsed_addr, force_overwrite=True)`
This ensures that addresses used in secure mappings are marked appropriately for security-aware memory allocation.
