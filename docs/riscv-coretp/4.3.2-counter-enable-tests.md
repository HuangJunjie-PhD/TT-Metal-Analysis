# Counter Enable Tests

Relevant source files
*   [coretp/plans/zicntr_zihpm_sscounterenw/__init__.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicntr_zihpm_sscounterenw/__init__.py)

## Purpose and Scope

This document describes the Counter Enable test plan for the RISC-V Core Test Plan (coretp) framework. These tests validate the behavior of the `mcounteren` and `scounteren` Control and Status Registers (CSRs), which control access to hardware performance counters across different privilege modes.

The Counter Enable tests focus specifically on CSR-based access control mechanisms. For timer-specific counter tests including `stimecmp` and counter enable hierarchies, see [SSTC Timer Counter Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.1-sstc-timer-counter-tests). For performance monitoring unit overflow behavior, see [PMU Overflow Tests (SSCOFPMF)](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.3-pmu-overflow-tests-(sscofpmf)).

**Sources:**[coretp/plans/zicntr_zihpm_sscounterenw/__init__.py 1-15](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicntr_zihpm_sscounterenw/__init__.py#L1-L15)

## Counter Enable Architecture

Counter enable CSRs implement a hierarchical access control mechanism for hardware performance monitoring counters in RISC-V processors. The architecture consists of two primary control registers:

*   **`mcounteren`**: Machine-mode CSR that controls whether Supervisor-mode and User-mode can access specific counters
*   **`scounteren`**: Supervisor-mode CSR that controls whether User-mode can access specific counters

**Diagram: Counter Enable Privilege Hierarchy**

Each bit in the `mcounteren` and `scounteren` registers corresponds to a specific counter:

*   Bit 0: `cycle` counter
*   Bit 2: `instret` counter
*   Bits 3-31: `hpmcounter[3-31]` (Hardware Performance Monitor counters)

When a bit is clear (0) in the appropriate enable register, attempts to access the corresponding counter from a lower privilege mode result in an illegal instruction exception.

## Test Coverage Areas

The Counter Enable test plan covers the following functional areas:

### CSR Access Control

Tests validate that access control mechanisms work correctly across privilege mode boundaries:

| Scenario | mcounteren | scounteren | Access Mode | Expected Behavior |
| --- | --- | --- | --- | --- |
| M-mode access | Any | Any | M-mode | Always succeeds |
| S-mode blocked by M | Bit clear | Any | S-mode | Illegal instruction exception |
| S-mode allowed by M | Bit set | Any | S-mode | Access succeeds |
| U-mode blocked by M | Bit clear | Any | U-mode | Illegal instruction exception |
| U-mode blocked by S | Bit set | Bit clear | U-mode | Illegal instruction exception |
| U-mode fully enabled | Bit set | Bit set | U-mode | Access succeeds |

### Hardware Performance Monitor Counters

Tests verify that the counter enable mechanism applies uniformly to all counter types:

*   **Base counters**: `cycle` (bit 0), `instret` (bit 2)
*   **HPM counters**: `hpmcounter[3-31]` (bits 3-31)
*   **Counter independence**: Each counter bit can be independently controlled

## Test Scenario Categories

Counter Enable tests are organized into several categories based on the privilege mode interactions being validated:

### M-Mode to S/U-Mode Delegation

These scenarios test the `mcounteren` CSR's control over Supervisor and User mode access:

**Diagram: mcounteren Test Scenarios**

### S-Mode to U-Mode Delegation

These scenarios test the `scounteren` CSR's control over User mode access, with `mcounteren` configured to allow S-mode access:

*   **Hierarchical blocking**: U-mode can only access counters if both `mcounteren` and `scounteren` bits are set
*   **S-mode independence**: `scounteren` configuration does not affect S-mode's own access rights
*   **Bit-wise granularity**: Each counter can be independently delegated to U-mode

### Illegal Access Exception Generation

Tests verify that appropriate illegal instruction exceptions are raised when counter access is blocked:

| Test Aspect | Verification Method |
| --- | --- |
| Exception code | `AssertException` with expected cause |
| Trap delegation | Verify exception handled at correct privilege level |
| CSR state preservation | Ensure enable CSRs unchanged after exception |
| No side effects | Verify counter values unmodified on blocked access |

### Counter Enable Bit Behavior

Tests validate the behavior of individual bits within the enable registers:

*   **WARL semantics**: Write-Any-Read-Legal behavior for unsupported counters
*   **Bit independence**: Setting/clearing one bit does not affect others
*   **Reset values**: Verify default enable state after reset
*   **Read-back consistency**: Written values read back correctly for supported counters

## Integration with Test Step Framework

Counter Enable test scenarios compose TestStep primitives to implement their validation logic. The primary steps used are:

### CSR Operations

Counter Enable tests extensively use CSR read and write operations:

**Diagram: Counter Enable Test Step Composition**

For detailed information on CSR operation steps, see [CSR Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.2-csr-operations).

### Assertion Steps

Verification uses two primary assertion types:

*   **`AssertEqual`**: Validates that counter reads succeed and return valid values when access is enabled
*   **`AssertException`**: Validates that illegal instruction exceptions occur when access is blocked by enable bits

For more information on assertion steps, see [Assertions and Verification](https://deepwiki.com/tenstorrent/riscv-coretp/5.3-assertions-and-verification).

### Multi-Privilege Mode Testing

Some test scenarios may use Hart control steps to coordinate testing across multiple privilege modes:

*   **`Hart`**: Configure separate harts running at different privilege levels
*   **`Call`**: Transition between privilege modes to test delegation boundaries
*   **`HartExit`**: Synchronize multi-privilege mode test completion

For details on control flow steps, see [Control Flow and Multi-Hart Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.4-control-flow-and-multi-hart-operations).

## Related Test Plans

The Counter Enable test plan is part of a broader suite of supervisor-mode extension tests:

*   **[SSTC Timer Counter Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.1-sstc-timer-counter-tests)**: Tests timer functionality and counter enable hierarchies for timer CSRs (`mcounteren.TM`, `hcounteren.TM`, `scounteren.TM`)
*   **[PMU Overflow Tests (SSCOFPMF)](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.3-pmu-overflow-tests-(sscofpmf))**: Tests performance monitoring unit overflow behavior, which depends on counter accessibility controlled by enable CSRs
*   **[Supervisor Mode Extensions](https://deepwiki.com/tenstorrent/riscv-coretp/4.3-supervisor-mode-extensions)**: Parent section containing all supervisor-mode test plans

The Counter Enable test plan provides foundational validation that other counter-related test plans depend on, ensuring that the privilege mode access control mechanisms work correctly before testing more advanced counter features.
