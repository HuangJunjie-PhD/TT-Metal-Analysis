This page provides practical examples demonstrating how to compile and execute PyTorch models using tt-torch. It covers basic model compilation with `torch.compile()`, the `verify_module()` utility for rapid testing, configuration options, and multi-device execution patterns.

For detailed installation instructions, see [Installation and Setup](https://deepwiki.com/tenstorrent/tt-torch/2.1-installation-and-setup). For information on controlling compiler behavior through environment variables and programmatic configuration, see [Configuration and Control](https://deepwiki.com/tenstorrent/tt-torch/3.1.3-configuration-and-control).

## Core Usage Pattern: torch.compile() with tt-torch Backend

The primary interface for using tt-torch is PyTorch's `torch.compile()` function with the `"tt-legacy"` or `"tt-experimental"` backend. This compiles a PyTorch model to run on Tenstorrent hardware.

### Basic Compilation Flow

**Basic Compilation Flow**: Shows the path from a PyTorch module through torch.compile with backend selection to execution.

### Minimal Example

The simplest usage pattern involves importing `tt_torch`, defining a PyTorch model, and compiling it:

`import torchimport tt_torch class AddTensors(torch.nn.Module):    def forward(self, x, y):        return x + y model = AddTensors()compiled_model = torch.compile(model, backend="tt-legacy") x = torch.ones(5, 5)y = torch.ones(5, 5)result = compiled_model(x, y)`
The backend is registered through the `@register_backend` decorator at [tt_torch/dynamo/backend.py 313](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L313-L313)

## Verification Workflow: verify_module()

The `verify_module()` function provides a convenient way to test models by automatically comparing hardware execution results against CPU golden outputs.

**Verification Flow**: The verify_module function automates golden reference comparison with configurable PCC and ATOL thresholds.

### Basic verify_module Usage

`import torchfrom torch import nnfrom tt_torch.tools.verify import verify_module class Basic(nn.Module):    def __init__(self):        super().__init__()     def forward(self, x, y):        return torch.add(x, y) verify_module(Basic(), input_shapes=[(256, 256), (256, 256)])`
The function is defined at [tt_torch/tools/verify.py 313-362](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L313-L362) and handles:

*   Input generation based on shapes or provided tensors
*   Compilation via `compile_model()` at [tt_torch/tools/verify.py 25-36](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L25-L36)
*   Automatic golden reference generation
*   Numerical comparison using `verify_against_golden()` at [tt_torch/tools/verify.py 59-201](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L59-L201)

### Configuration Parameters

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `mod` | `torch.nn.Module` or `onnx.ModelProto` | Required | Model to verify |
| `input_shapes` | `List[Tuple]` | `None` | Shape tuples for generated inputs |
| `inputs` | `List[torch.Tensor]` | `None` | Pre-generated input tensors |
| `input_data_types` | `List[torch.dtype]` | `[torch.float32]` | Data types for inputs |
| `required_pcc` | `float` | `0.99` | Minimum Pearson Correlation Coefficient |
| `required_atol` | `float` | `1e-2` | Maximum absolute tolerance |
| `input_range` | `Tuple[float, float]` | `(-0.5, 0.5)` | Range for float input generation |
| `compiler_config` | `CompilerConfig` | `CompilerConfig()` | Compilation configuration |
| `do_assert` | `bool` | `True` | Whether to assert on verification failure |
| `device` | Device | `None` | Specific device for execution |

## CompilerConfig: Controlling Compilation Behavior

The `CompilerConfig` class provides programmatic control over compilation depth, verification, optimization, and debugging features.

**CompilerConfig Structure**: The main configuration categories and key options available for controlling compilation.

### Basic Configuration Example

`from tt_torch.tools.utils import CompilerConfig, CompileDepthfrom tt_torch.dynamo.backend import BackendOptionsimport torchimport tt_torch class MyModel(torch.nn.Module):    def forward(self, x):        return x * 2 model = MyModel() # Create and configure compiler configcc = CompilerConfig()cc.compile_depth = CompileDepth.EXECUTEcc.enable_consteval = True # Pass config through BackendOptionsoptions = BackendOptions()options.compiler_config = cc compiled_model = torch.compile(model, backend="tt-legacy", options=options)`
The `CompilerConfig` class is defined at [tt_torch/tools/utils.py 300-572](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L300-L572) and `BackendOptions` at [tt_torch/dynamo/backend.py 21-54](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L21-L54)

### Key CompileDepth Values

The `CompileDepth` enum at [tt_torch/tools/utils.py 48-54](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L48-L54) controls how far through the compilation pipeline to proceed:

| Value | Description |
| --- | --- |
| `TORCH_FX` | Stop after FX graph capture, return uncompiled graph |
| `STABLEHLO` | Compile through StableHLO IR, stop before TTIR |
| `TTNN_IR` | Compile through TTNN dialect, return IR |
| `COMPILE_OP_BY_OP` | Compile operations individually without execution |
| `EXECUTE_OP_BY_OP` | Compile and execute operations individually |
| `EXECUTE` | Full end-to-end compilation and execution |

### Environment Variable Overrides

The `apply_environment_overrides()` method at [tt_torch/tools/utils.py 419-468](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L419-L468) allows configuration via environment variables:

| Environment Variable | CompilerConfig Property | Effect |
| --- | --- | --- |
| `TT_TORCH_COMPILE_DEPTH` | `compile_depth` | Sets maximum compilation depth |
| `TT_TORCH_VERIFY_OP_BY_OP` | `verify_op_by_op` | Enables per-op verification |
| `TT_TORCH_CONSTEVAL` | `enable_consteval` | Enables constant evaluation |
| `TT_TORCH_OPTIMIZER` | `enable_optimizer` | Enables TTNN optimizer passes |
| `TT_TORCH_IR_LOG_LEVEL` | `dump_debug`, `dump_info` | Controls IR dumping verbosity |

## Multi-Device Execution

tt-torch supports both asynchronous execution and data-parallel execution across multiple devices.

**Multi-Device Execution Patterns**: Three modes for device utilization: single device, async multi-device, and data parallel.

### Async Execution Pattern

The `verify_torch_module_async()` function at [tt_torch/tools/verify.py 364-419](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L364-L419) demonstrates async execution:

`from tt_torch.tools.verify import verify_torch_module_asyncfrom tt_torch.tools.device_manager import DeviceManager class Basic(nn.Module):    def forward(self, x, y):        return torch.add(x, y) # Automatically acquires all available devicesverify_torch_module_async(Basic(), input_shapes=[(256, 256), (256, 256)])`
The function:

1.   Acquires all available devices via `DeviceManager.acquire_available_devices()` at [tt_torch/tools/verify.py 380](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L380-L380)
2.   Compiles the model for each device with `async_mode=True` at [tt_torch/tools/verify.py 387](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L387-L387)
3.   Executes on all devices and uses `tt_mlir.to_host()` to retrieve results at [tt_torch/tools/verify.py 401](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L401-L401)

### Manual Multi-Device Control

`from tt_torch.tools.device_manager import DeviceManager # Acquire all devicesparent_device, device_list = DeviceManager.acquire_available_devices() # Use devices for compilation (see ModelTester)# ... # Release when doneDeviceManager.release_parent_device(parent_device, cleanup_sub_devices=True)`

## Backend Selection: tt-legacy vs tt-experimental

tt-torch provides two backend implementations with different compilation approaches.

### Backend Comparison

| Feature | tt-legacy | tt-experimental |
| --- | --- | --- |
| Implementation | Dynamo-based with torch-mlir | XLA-based integration |
| Registration | `@register_backend(name="tt-legacy")` | XLA device plugin |
| Compile API | `torch.compile(backend="tt-legacy")` | `torch.compile(backend="tt-experimental")` or `.to("xla")` |
| Op-by-op Support | Yes (TORCH and STABLEHLO) | Limited |
| Multi-chip Support | Yes | Experimental |

### tt-legacy Backend

The primary backend, registered at [tt_torch/dynamo/backend.py 313](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L313-L313):

`import torchimport tt_torch model = MyModel()compiled = torch.compile(model, backend="tt-legacy")`
The backend function at [tt_torch/dynamo/backend.py 314-369](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L314-L369) routes to different executors based on `CompileDepth`:

*   `_torch_backend()` for op-by-op with TORCH backend at [tt_torch/dynamo/backend.py 182-194](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L182-L194)
*   `_shlo_backend()` for op-by-op with STABLEHLO backend at [tt_torch/dynamo/backend.py 165-179](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L165-L179)
*   `_base_backend()` for full compilation at [tt_torch/dynamo/backend.py 273-309](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L273-L309)

### tt-experimental Backend (XLA)

The experimental XLA-based backend supports both `torch.compile()` and eager execution:

`import torchimport tt_torch # Method 1: torch.compilemodel = MyModel()compiled = torch.compile(model, backend="tt-experimental") # Method 2: Eager executionmodel = MyModel().to("xla")x = torch.randn(5, 5, device="xla")result = model(x).to("cpu")`
The XLA integration is documented at [docs/src/test.md 96-137](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/docs/src/test.md?plain=1#L96-L137)

## Common Pitfalls and Debugging

### Input Shape Requirements

Operations may have specific shape requirements. For example, TTNN operations often require dimensions to be multiples of 32 for optimal performance.

### Data Type Handling

The `typecast_inputs` flag at [tt_torch/tools/utils.py 321](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L321-L321) controls automatic type conversion. When enabled (default), inputs are converted to hardware-supported types.

### Verification Metrics

Two metrics are used for verification:

*   **PCC (Pearson Correlation Coefficient)**: Measures output correlation (default threshold: 0.99)
*   **ATOL (Absolute Tolerance)**: Measures maximum absolute difference (default: 0.01)

Calculated by `calculate_pcc()` at [tt_torch/tools/utils.py 785-801](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L785-L801) and `calculate_atol()` at [tt_torch/tools/utils.py 743-782](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L743-L782)

### Op-by-Op Debugging

For models that fail end-to-end compilation, use op-by-op execution to identify problematic operations:

`cc = CompilerConfig()cc.compile_depth = CompileDepth.EXECUTE_OP_BY_OPcc.verify_op_by_op = True  # Verify each op against golden verify_module(model, input_shapes=shapes, compiler_config=cc, do_assert=False)`
Results are saved to JSON at [tt_torch/tools/utils.py 509-540](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L509-L540) for post-processing analysis.
