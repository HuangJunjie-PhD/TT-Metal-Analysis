## Purpose and Scope

This page documents the documentation build system implemented in `docs/build.py`, which provides a command-line interface for building, validating, and serving Sphinx-generated documentation. The build system supports multiple modes including standard builds, validation with strict error checking, local HTTP serving, and interactive development with hot-reloading.

For details about the documentation structure and Sphinx configuration, see [Documentation Structure](https://deepwiki.com/tenstorrent/riescue/6.2-documentation-structure). For information about CI/CD deployment of documentation, see [Documentation Deployment](https://deepwiki.com/tenstorrent/riescue/5.3.2-documentation-deployment).

## Overview

The documentation build system is implemented as a single Python script (`docs/build.py`) containing the `DocBuilder` class. This class wraps the `sphinx-build` command with additional functionality for local development and CI/CD integration.

**Key Features:**

*   Standard Sphinx HTML documentation generation
*   Strict validation mode for CI/CD pipelines
*   Built-in HTTP server for local documentation preview
*   Interactive mode with rebuild-on-demand capability
*   Support for both public and internal documentation trees

## Documentation Source Selection

The build system supports two documentation source trees, selected via the `BUILD_TT_DOCS` environment variable:

| Environment Variable | Source Directory | Purpose |
| --- | --- | --- |
| `BUILD_TT_DOCS` not set | `docs/source` | Public API documentation |
| `BUILD_TT_DOCS` set | `docs/tt_source` | Internal documentation for TT developers |

This selection happens at initialization:

The source directory is stored in `DocBuilder.defualt_src` (note the typo in the variable name) and can be overridden via the `--source_dir` command-line argument.

## Local Development Modes

### Standard Local Host

The `--local_host` flag builds the documentation and starts an HTTP server:

This:

1.   Cleans the build directory
2.   Builds the documentation
3.   Starts an HTTP server on `localhost:8888`
4.   Waits for `CTRL+C` to stop

### Interactive Mode

Interactive mode adds rebuild capability while the server runs:

Commands available during interactive mode:

*   `r` or `reload`: Rebuild documentation and restart server
*   `s` or `stop`: Stop the server and exit
*   `CTRL+C`: Stop the server and exit

## Sphinx Build Command Construction

The `build_docs()` method constructs the Sphinx command dynamically:

**Locale Configuration:** The build process sets `LC_ALL=C` and `LANG=C` in the environment to avoid locale-related errors in containerized builds [docs/build.py 117-121](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L117-L121)

**Command Example:**

```
sphinx-build -W -E docs/source docs/_build
```

Where:

*   `-W`: Treat warnings as errors (in check mode)
*   `-E`: Don't use cached environment, force full rebuild (in check mode)

Sources: [docs/build.py 102-122](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L102-L122)

## HTTP Server Implementation

The local HTTP server uses Python's `http.server.SimpleHTTPRequestHandler` with a custom `ReusableTCPServer` class:

**Key Implementation Details:**

1.   **Address Reuse:**`allow_reuse_address = True` prevents "Address already in use" errors when restarting the server [docs/build.py 194](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L194-L194)

2.   **Interactive Mode:** The server runs in a daemon thread, allowing the main thread to handle user input [docs/build.py 140-142](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L140-L142)

3.   **Working Directory:** The server changes to the build directory before serving, so paths are relative to the documentation root [docs/build.py 190](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L190-L190)

Sources: [docs/build.py 179-218](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L179-L218)

## Build Modes Comparison

| Mode | Clean | Strict | HTTP Server | Reload | Use Case |
| --- | --- | --- | --- | --- | --- |
| Default | No | No | No | N/A | Quick build for manual inspection |
| `--clean` | Yes | No | No | N/A | Ensure fresh build without old artifacts |
| `--check` | No | Yes | No | N/A | CI/CD validation |
| `--local_host` | Yes | No | Yes | No | Local preview during development |
| `--interactive` | Yes | No | Yes | Yes | Active documentation writing |

**Common Patterns:**

*   **CI/CD:**`./docs/build.py --check --build_dir public`
*   **Development:**`./docs/build.py --interactive`
*   **Quick Preview:**`./docs/build.py --local_host`

## Interactive Mode Architecture

Interactive mode implements a command loop with rebuild capability:

**Implementation Details:**

1.   **Threading:** Server runs in a daemon thread to avoid blocking user input [docs/build.py 140](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L140-L140)

2.   **Stop Event:** A `threading.Event` signals the server thread to shut down gracefully [docs/build.py 136](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L136-L136)

3.   **Reload Loop:** After rebuilding, the outer loop restarts, creating a new server instance [docs/build.py 130-177](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L130-L177)

4.   **Timeout:** In interactive mode, the server uses a 1-second timeout for `handle_request()` to check the stop event periodically [docs/build.py 206](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L206-L206)

Sources: [docs/build.py 124-177](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L124-L177)

## Error Handling

### Build Errors

The `build_docs()` method uses `subprocess.run(check=True)`, which raises `CalledProcessError` if Sphinx returns a non-zero exit code [docs/build.py 122](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L122-L122)

### Missing Source Directory

If the source directory doesn't exist, a `FileNotFoundError` is raised with a descriptive message [docs/build.py 105-106](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L105-L106)

### Server Errors

Server errors in non-interactive mode are propagated to the caller. In interactive mode, they're caught and printed, allowing the user to retry [docs/build.py 174-176](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L174-L176)[docs/build.py 212-216](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L212-L216)

### Keyboard Interrupt

`KeyboardInterrupt` exceptions are handled gracefully:

*   In standard local host mode: prints "Local host server stopped" [docs/build.py 209-211](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L209-L211)
*   In interactive mode: stops the server and returns to the prompt [docs/build.py 157-160](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L157-L160)

Sources: [docs/build.py 102-218](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L102-L218)

## Integration with CI/CD

The GitHub Actions workflow uses the build script in check mode:

This configuration:

1.   Treats Sphinx warnings as errors (`-W` flag)
2.   Forces a full rebuild (`-E` flag)
3.   Outputs to the `public` directory for GitHub Pages artifact upload

The strict error checking ensures that documentation builds cleanly before deployment, preventing broken documentation from reaching users.
