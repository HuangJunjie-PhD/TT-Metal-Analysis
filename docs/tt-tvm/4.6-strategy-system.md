## Purpose and Scope

The Strategy System is Relay's mechanism for selecting and executing target-dependent operator implementations. It provides a framework for registering multiple implementations of the same operator optimized for different hardware targets (CUDA, CPU, x86, ARM, etc.) and automatically selecting the most appropriate implementation based on the target and operator attributes.

This page covers the strategy registration infrastructure, target-specific strategy modules, and the selection process. For information about the operators themselves, see [Neural Network Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.1-neural-network-operators) and [Tensor Transformation Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.2-tensor-transformation-operators). For scheduling primitives used within strategies, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api).

## Core Components

### OpStrategy Class

The `OpStrategy` class is the central container for operator implementations. It holds multiple implementation choices, each consisting of:

| Component | Description | Type |
| --- | --- | --- |
| **Compute Function** | Generates tensor expressions for the operator | `FTVMCompute` |
| **Schedule Function** | Applies optimizations to the computation | `FTVMSchedule` |
| **Name** | Identifier for the implementation | String |
| **Priority Level (plevel)** | Selection priority (higher is better) | Integer |

**OpStrategy Structure**: Each implementation within an OpStrategy contains compute and schedule functions with associated metadata.

Sources: [python/tvm/relay/op/strategy/cuda.py 49-57](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L49-L57)[python/tvm/relay/op/strategy/cuda.py 81-97](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L81-L97)

## Strategy Selection Process

### Priority-Based Selection

Implementations within an OpStrategy are selected based on their priority level (plevel):

| Priority Level | Typical Use Case |
| --- | --- |
| **25+** | External optimized libraries (cuDNN, OneDNN) |
| **20** | Hardware-specific intrinsics (TensorCore, AVX-512) |
| **15** | Auto-scheduler/Meta-scheduler implementations |
| **10** | Default optimized implementations |
| **5** | Alternative algorithms (Winograd, FFT-based) |
| **1-3** | Generic fallback implementations |

Sources: [python/tvm/relay/op/strategy/cuda.py 373-381](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L373-L381)[python/tvm/relay/op/strategy/cuda.py 246-281](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L246-L281)

### Selection Flow

**Strategy Selection Process**: During compilation, the appropriate strategy is selected based on target, then the best implementation is chosen based on tuning or priority level.

Sources: [python/tvm/relay/op/strategy/generic.py 34-46](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/generic.py#L34-L46)[python/tvm/relay/op/strategy/cuda.py 138-460](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L138-L460)

## Common Strategy Patterns

### Pattern 1: Multi-Algorithm Support

Many operators support multiple algorithms with different trade-offs:

**Multi-Algorithm Pattern**: Different algorithms excel in different scenarios, with priorities guiding automatic selection.

## Strategy vs Direct Compute Registration

### Comparison

| Aspect | Strategy System | Direct FTVMCompute |
| --- | --- | --- |
| **Flexibility** | Multiple implementations per target | Single implementation |
| **Target Selection** | Automatic based on target | Manual conditional logic |
| **Priority** | Explicit plevel for selection | No priority mechanism |
| **Auto-Tuning** | Integrated support | Manual integration |
| **Use Case** | Complex ops with multiple algorithms | Simple element-wise ops |

The strategy system is used for operators with significant implementation variation across targets (conv2d, dense, matmul), while simple operators may use direct `FTVMCompute` registration:
