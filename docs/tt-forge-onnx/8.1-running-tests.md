## Test Execution Overview

The tt-forge-onnx test infrastructure uses pytest as the execution engine with a custom filtering and selection system built on top of it. Tests can be run locally via pytest commands or automatically through GitHub Actions workflows.

**Sources:**[forge/test/operators/utils/plan.py 392-444](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/plan.py#L392-L444)[forge/test/operators/pytorch/test_all.py 449-478](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L449-L478)[forge/test/operators/utils/utils.py 143-208](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/utils.py#L143-L208)[.github/workflows/test-sub.yml 198-244](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L198-L244)

## Running Tests Locally

### Basic pytest Execution

The simplest way to run tests is directly through pytest:

`# Activate the environmentsource env/activate # Run all operator testspytest forge/test/operators/pytorch/ # Run tests for a specific operatorpytest forge/test/operators/pytorch/eltwise_unary/test_abs.py # Run a single test by namepytest forge/test/operators/pytorch/eltwise_unary/test_abs.py::test_abs`
### Using Test Filtering with Environment Variables

The test system provides extensive filtering capabilities through environment variables:

`# Run tests for specific operatorsexport OPERATORS=add,mulpytest forge/test/operators/pytorch/test_all.py # Exclude operators (note the leading dash)export OPERATORS=-add,mulpytest forge/test/operators/pytorch/test_all.py # Filter by input shapesexport INPUT_SHAPES="[(1,4),(1,17)]"pytest forge/test/operators/pytorch/test_all.py # Apply custom filtersexport FILTERS=QUICK,NOT_FAILINGpytest forge/test/operators/pytorch/test_all.py # Sample 10% of tests randomlyexport SAMPLE=10export RANDOM_SEED=42pytest forge/test/operators/pytorch/test_all.py`
**Sources:**[forge/test/operators/pytorch/test_all.py 60-140](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L60-L140)[forge/test/operators/pytorch/test_all.py 449-478](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L449-L478)

## Environment Variables for Test Control

The following tables document all environment variables that control test selection and execution:

### Test Selection Variables

| Variable | Description | Example Values |
| --- | --- | --- |
| `OPERATORS` | Comma-separated list of operators to test. Prefix with `-` to exclude. | `add,mul` or `-conv2d` |
| `FILTERS` | Comma-separated list of filter lambdas from `VectorLambdas` or `QueryLambdas` | `QUICK,NOT_FAILING,SINGLE_SHAPE` |
| `INPUT_SOURCES` | Filter by input source type | `FROM_HOST,FROM_ANOTHER_OP,CONST_EVAL_PASS` |
| `INPUT_SHAPES` | Filter by specific tensor shapes (Python list syntax) | `[(1,4),(2,3,4)]` |
| `DEV_DATA_FORMATS` | Filter by data format | `Float16_b,Int8` |
| `MATH_FIDELITIES` | Filter by math fidelity | `HiFi4,HiFi3` |
| `KWARGS` | Filter by operator parameters (Python dict syntax) | `[{'dim': 0}, {'dim': 1}]` |

### Test Range and Sampling Variables

| Variable | Description | Example Values |
| --- | --- | --- |
| `SAMPLE` | Percentage of tests to randomly sample | `10` (for 10%) |
| `RANDOM_SEED` | Seed for random sampling | `42` |
| `RANGE` | Limit test indices (start,end) | `0,100` or `100` |
| `TEST_ID` | Run a single test by its ID string | `add-FROM_HOST-{}-...` |
| `ID_FILES` | Comma-separated paths to files containing test IDs | `ids/passing.txt,ids/new.txt` |
| `ID_FILES_IGNORE` | Files containing test IDs to skip | `ids/blacklist.txt` |

### Execution Control Variables

| Variable | Description | Example Values |
| --- | --- | --- |
| `SKIP_FORGE_VERIFICATION` | Skip Forge compilation and verification | `true` / `false` |
| `DRY_RUN` | Skip test execution, only list tests | `true` / `false` |
| `IGNORE_XFAIL_FILES` | Ignore expected failure files | `true` / `false` |
| `IGNORE_SKIP_FILES` | Ignore skip files | `true` / `false` |
| `HF_TOKEN` | HuggingFace token for model downloads | `hf_...` |

**Sources:**[forge/test/operators/pytorch/test_all.py 31-140](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L31-L140)[forge/test/operators/utils/features.py 11-58](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/features.py#L11-L58)

## CI/CD Test Execution

### GitHub Actions Workflows

The repository uses three main workflows for test execution:

### Test Matrix Strategy

The CI system uses a matrix strategy to parallelize test execution across hardware and test groups:

| Parameter | Description | Example Values |
| --- | --- | --- |
| `runs-on` | Hardware targets | `n150`, `n300`, `p150` |
| `test_group_id` | Test group number | `1`, `2`, `3`, ..., `6` |
| `test_group_cnt` | Total number of groups | `6` (default) |
| `test_mark` | pytest marker to select tests | `push`, `pr_models_regression`, `nightly` |

**Example Matrix Expansion:**

*   `runs-on: [n150, n300]` and `test_group_cnt: 6` creates **12 parallel jobs**
*   Each job runs approximately 1/6 of the test suite on its assigned hardware

**Sources:**[.github/workflows/test-sub.yml 79-104](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L79-L104)[.github/workflows/on-pr.yml 108-130](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/on-pr.yml#L108-L130)

### Test Collection and Splitting

The test collection process uses pytest-split to divide tests across groups:

`# Collect tests for group 2 out of 6 total groupspytest \  --splits 6 \  --group 2 \  --splitting-algorithm least_duration \  -m "push" \  --collect-only`
The `least_duration` algorithm uses historical test durations from [.test_durations](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.test_durations) to balance load across groups.

**Sources:**[.github/workflows/test-sub.yml 208-234](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L208-L234)

## Test Runner and Crash Handling

### Test Runner Script

The CI uses a custom test runner (`test_runner.py`) that provides:

*   **Crash Recovery:** Continues after pytest crashes
*   **Memory Logging:** Tracks memory usage per test
*   **JUnit XML Generation:** Structured test reports

`# Run with crash recovery and memory loggingpython .github/workflows/test_runner.py \  --junit-xml=reports/report.xml \  --log-memory-usage \  --continue-after-crash`
When a test crashes, the runner:

1.   Logs the crashed test to `crashed_pytest.log`
2.   Restarts pytest with remaining tests
3.   Appends crash information to the job summary

**Sources:**[.github/workflows/test-sub.yml 253-288](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L253-L288)

### Verbose Output Mode

Enable verbose pytest output to see print statements and detailed execution:

`# Local execution with verbose outputpytest -svv forge/test/operators/pytorch/test_abs.py # CI execution (set in workflow dispatch inputs)# Sets enable-verbose-output: true in on-pr.yml or test.yml`
**Sources:**[.github/workflows/test-sub.yml 272-275](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L272-L275)[.github/workflows/on-pr.yml 24-28](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/on-pr.yml#L24-L28)

## Test Result Interpretation

### pytest Outcomes

Tests can have the following outcomes:

| Outcome | Symbol | Description |
| --- | --- | --- |
| **PASSED** | `.` | Test passed successfully |
| **FAILED** | `F` | Test failed with an exception |
| **XFAIL** | `x` | Expected failure occurred (test marked with `@pytest.mark.xfail`) |
| **XPASS** | `X` | Unexpected pass (expected to fail but passed) |
| **SKIPPED** | `s` | Test was skipped (marked with `@pytest.mark.skip`) |
| **ERROR** | `E` | Error during test setup or teardown |

### Expected Failures (xfail)

Tests with known issues are marked with xfail reasons from `FailingReasons`:

`# From plan.py - TestResultFailingpytest.mark.xfail(reason="Data mismatch (PCC < 0.99)")pytest.mark.xfail(reason="Unsupported TTNN operation")pytest.mark.xfail(reason="MLIR lowering failed")`
The conftest hook validates that xfail reasons match actual exceptions:

**Sources:**[forge/test/operators/pytorch/conftest.py 22-86](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/conftest.py#L22-L86)[forge/test/operators/utils/failing_reasons_validation.py 15-53](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons_validation.py#L15-L53)

### JUnit XML Reports

Test results are exported to JUnit XML format with custom properties:

`<testcase name="test_name">  <properties>    <property name="operator" value="add"/>    <property name="input_shape" value="(1, 4)"/>    <property name="dev_data_format" value="Float16_b"/>    <property name="xfail_reason" value="Data mismatch"/>    <property name="pcc" value="0.9876"/>  </properties></testcase>`

## Advanced Features

### Running Tests from ID Files

Test IDs can be loaded from files for reproducibility:

`# Create a file with test IDs (one per line)cat > my_tests.txt << EOFadd-FROM_HOST-{}-[(1, 4)]-Float16_b-HiFi4mul-FROM_ANOTHER_OP-{}-[(2, 3, 4)]-None-NoneEOF # Run only those testsexport ID_FILES=my_tests.txtpytest forge/test/operators/pytorch/test_all.py`
Test IDs follow the format:

```
{operator}-{input_source}-{kwargs}-{input_shape}-{dev_data_format}-{math_fidelity}
```

**Sources:**[forge/test/operators/utils/plan.py 620-720](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/plan.py#L620-L720)

Maintain a blacklist of problematic tests:

`# Default blacklist locationcat ids/blacklist.txt# add-FROM_HOST-{}-[(10000, 10000)]-None-None  # OOM error# ... # Blacklist is applied by default via ID_FILES_IGNOREexport ID_FILES_IGNORE=ids/blacklist.txt,ids/fatal_errors.txtpytest forge/test/operators/pytorch/test_all.py`
**Sources:**[forge/test/operators/pytorch/test_all.py 72](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L72-L72)[forge/test/operators/pytorch/test_all.py 224-226](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/test_all.py#L224-L226)

## Summary

Running tests in tt-forge-onnx involves:

1.   **Local Execution:** Use pytest directly with environment variables for filtering
2.   **Test Selection:** Apply filters through `OPERATORS`, `FILTERS`, `INPUT_SHAPES`, etc.
3.   **CI/CD:** Tests run automatically on PRs and nightly through GitHub Actions
4.   **Parallelization:** Matrix strategy splits tests across hardware and groups
5.   **Result Interpretation:** JUnit reports, xfail validation, and crash recovery
6.   **Advanced Features:** ID files, blacklisting, duration tracking, and verbose output
