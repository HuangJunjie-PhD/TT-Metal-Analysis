## Overview

Tensix compute cores are the primary computational units in the Blackhole ASIC, organized as a 14-column by 10-row grid providing the main tensor processing fabric. Each Tensix core contains three RISC processors (TRISCs), 1.5 MB of local L1 memory, and specialized tensor processing hardware. This page documents the initialization, memory management, and runtime control of the Tensix grid.

For information about the Network-on-Chip that interconnects Tensix cores, see [NOC Translation and Harvesting](https://deepwiki.com/tenstorrent/tt-system-firmware/4.1-noc-translation-and-harvesting). For GDDR memory controllers (MRISC) and Ethernet controllers (ERISC) that share similar initialization patterns, see [GDDR Memory Controllers](https://deepwiki.com/tenstorrent/tt-system-firmware/3.4-gddr-memory-controllers) and [Ethernet Subsystem](https://deepwiki.com/tenstorrent/tt-system-firmware/3.5-ethernet-subsystem).

### Clock Gating Configuration

Clock gating reduces power consumption by disabling clocks to unused functional blocks. The firmware configures 17 functional blocks with hysteresis values and enables clock gating for all blocks simultaneously via broadcast.

**Clock Gating Block Assignment**

| Bit | Functional Block | Description |
| --- | --- | --- |
| 0 | Register Blocks | Configuration registers |
| 1 | FPU | Floating Point Unit |
| 2 | FPU M Tile | Matrix multiplication tile |
| 3 | FPU SFPU | Special Function Processing Unit |
| 4 | Mover | Data movement engine |
| 5 | Packers | Output data packing |
| 6 | Unpacker 0 | Input data unpacking (port 0) |
| 7 | Unpacker 1 | Input data unpacking (port 1) |
| 8 | X Search | Crossbar search logic |
| 9 | Thread Controller | Thread scheduling |
| 10 | TRISC 0 | Compute control RISC |
| 11 | TRISC 1 | Data movement RISC |
| 12 | TRISC 2 | Memory interface RISC |
| 13 | L1 Return Muxes | L1 read path multiplexers |
| 14 | Instruction Thread | Instruction fetch/decode |
| 15 | L1 Banks | L1 memory banks |
| 16 | Src B | Operand B source logic |

The configuration uses NOC broadcast to set three hysteresis registers and one enable register across all Tensix cores:

```
CG_CTRL_HYST0 (0xFFB12070) = 0x02020202  // Hysteresis = 2 for blocks 0-3
CG_CTRL_HYST1 (0xFFB12074) = 0x02020202  // Hysteresis = 2 for blocks 4-7
CG_CTRL_HYST2 (0xFFB1207C) = 0x02020202  // Hysteresis = 2 for blocks 8-11
CG_CTRL_EN    (0xFFB12244) = 0xFFFFFFFF  // Enable all 17 blocks
```

## L1 Memory Wiping

L1 memory wiping ensures all Tensix cores start with clean memory state. The implementation uses an optimized broadcast strategy to minimize initialization time.

**L1 Wipe Algorithm**

**Key Implementation Details:**

*   **Total L1 Size**: 1,536 KB per Tensix (`TENSIX_L1_SIZE`)
*   **Scratchpad Buffer**: Configurable via `CONFIG_TT_BH_ARC_SCRATCHPAD_SIZE`
*   **Doubling Strategy**: After zeroing the initial scratchpad, the algorithm repeatedly doubles the cleared region by copying from offset 0 to later offsets
*   **Broadcast Optimization**: Once a single Tensix is fully cleared, one broadcast operation clears all remaining Tensix cores simultaneously
*   **Alignment Requirement**: NOC2AXI to Tensix L1 transactions must be 64-byte aligned

Sources: [lib/tenstorrent/bh_arc/tensix_init.c 117-178](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L117-L178)[lib/tenstorrent/bh_arc/tensix_init.c 37](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L37-L37)

## DEST Register Clearing

The DEST (destination) registers hold accumulation results from tensor operations. Unlike L1 memory, DEST registers can only be written by code executing on the local TRISC processor. The firmware loads a specialized TRISC firmware to clear these registers.

**DEST Wipe Sequence**

**Global Synchronization Mechanism:**

The synchronization uses an atomic counter in one Tensix core's L1 memory:

*   **Counter Location**: `COUNTER_L1_ADDR` (0x110000) in chosen Tensix
*   **Coordinate Location**: `COUNTER_COORD_L1_ADDR` (0x100000) stores (X, Y) of counter Tensix
*   **Expected Count**: `POPCOUNT(tile_enable.tensix_col_enabled) * NUM_TENSIX_ROWS` (10 rows per column)
*   **Timeout**: 10 ms (`WIPE_DEST_TIMEOUT_US`)

Each TRISC firmware instance:

1.   Clears its local DEST registers using 32-bit stores
2.   Reads counter coordinates from L1
3.   Performs atomic increment to counter location via NOC

Sources: [lib/tenstorrent/bh_arc/tensix_init.c 241-315](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L241-L315)[lib/tenstorrent/bh_arc/tensix_init.c 52-56](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L52-L56)[lib/tenstorrent/bh_arc/tensix_init.c 187-201](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L187-L201)

### TRISC Reset PC Configuration

The reset PC (program counter) determines where TRISC execution begins after reset:

| Register | Address | Purpose |
| --- | --- | --- |
| `TRISC0_RESET_PC` | 0xFFB12228 | TRISC 0 reset entry point |
| `TRISC_RESET_PC_OVERRIDE` | 0xFFB12234 | Enable override (1=use RESET_PC) |
| `SOFT_RESET_0` | 0xFFB121B0 | Soft reset control register |

The soft reset value `ALL_RISC_SOFT_RESET` (0x47800) holds all three TRISCs in reset. Clearing bit 12 releases only TRISC 0.

## Runtime Power Management

Tensix cores can be dynamically enabled or disabled to save power during idle periods or when compute fabric is not needed.

### Tensix Enable/Disable

The `set_tensix_enable()` function controls the tile clock for all Tensix cores via the `NIU_CFG_0_TILE_CLK_OFF` bit in the NOC interface unit configuration.

**Implementation Details:**

*   **Broadcast Address**: `NIU_CFG_0` register at 0xFFB20100
*   **Ordering Mode**: `kNoc2AxiOrderingStrict` ensures sequential completion
*   **ZBus Publication**: Publishes `tensix_state_msg` with enable state for telemetry tracking

Sources: [lib/tenstorrent/bh_arc/noc_init.c 193-219](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L193-L219)

### Tile Clock Disable During Initialization

During NOC initialization, the firmware sets `NIU_CFG_0_TILE_CLK_OFF` for harvested Tensix columns, Ethernet tiles, and GDDR tiles based on the harvesting configuration:

## NOC Integration and Broadcast

Tensix cores are accessed via the Network-on-Chip using TLB (Translation Lookaside Buffer) mappings. The firmware uses broadcast mechanisms for efficient bulk operations.

### Broadcast Exclusion Configuration

To prevent unintended writes to non-Tensix tiles, the firmware programs broadcast exclusion masks in router configuration registers:

**Broadcast Exclusion Setup**

The exclusion masks are 64-bit values programmed into four router configuration registers per NOC:

*   `ROUTER_CFG_1,2`: Column exclusion (bits 0-63 map to X coordinates)
*   `ROUTER_CFG_3,4`: Row exclusion (bits 0-63 map to Y coordinates)

A node will not receive broadcasts if it is in an excluded row OR excluded column.

Sources: [lib/tenstorrent/bh_arc/noc_init.c 114-155](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L114-L155)[lib/tenstorrent/bh_arc/noc_init.c 267-269](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L267-L269)

### Tensix Broadcast TLB Setup

The `NOC2AXITensixBroadcastTlbSetup()` function configures a TLB for broadcasting to all enabled Tensix cores while avoiding the ARC's own row (to work around hardware issue #3401):

**Broadcast Rectangle Selection**

| NOC ID | Translation | X Range | Y Range | Rationale |
| --- | --- | --- | --- | --- |
| 0 | Any | 2→1 | 2→11 | Skip row 0 (ARC row in NOC0) |
| 1 | Enabled | 1→2 | 11→2 | Backwards rect, skip row 0 |
| 1 | Disabled | 0→16 | 0→10 | Skip row 11 (ARC row in NOC1) |

The broadcast uses multicast TLB configuration with `kNoc2AxiOrderingStrict` to ensure sequential completion.

Sources: [lib/tenstorrent/bh_arc/noc2axi.c 146-177](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.c#L146-L177)[lib/tenstorrent/bh_arc/noc_init.c 162](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L162-L162)

### TLB Window Addressing

ARC firmware accesses Tensix registers and memory through TLB windows mapped into the ARC's address space:

```
TLB Base Addresses:
  NOC0: 0xC0000000
  NOC1: 0xE0000000

Window Address Calculation:
  address = NOC_BASE + (tlb_entry << 24) + (addr & 0xFFFFFF)
  
Window Size: 16 MB per TLB entry
Number of TLBs: 16 per NOC ring
```

**Example TLB Configurations:**

| Purpose | Ring | TLB | X | Y | Base Address | Usage |
| --- | --- | --- | --- | --- | --- | --- |
| Enabled Tensix | 0 | Varies | 1-16 | 2-11 | Varies | Access arbitrary Tensix |
| Broadcast | 0 | Varies | 2 | 2 | NIU_CFG_0 | Bulk register writes |
| Multicast FW load | 0 | 0 | 2→1 | 2→11 | 0x6000 | Load TRISC firmware |

Sources: [lib/tenstorrent/bh_arc/noc2axi.h 34-42](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.h#L34-L42)[lib/tenstorrent/bh_arc/noc2axi.h 16-17](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.h#L16-L17)[lib/tenstorrent/bh_arc/noc2axi.c 93-112](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.c#L93-L112)

## Message Queue Integration

Host software controls Tensix power via the message queue system using the `TT_SMC_MSG_POWER_SETTING` message type. The `tensix_power` flag in the power settings request controls Tensix enable state.

**Power Setting Message Flow**

The power setting message allows hosts to control multiple power domains simultaneously, including Tensix, MRISC (GDDR), L2CPU, and AICLK frequency targets.
