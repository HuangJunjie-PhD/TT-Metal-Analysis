## Overview

Address modifiers are 8 hardware configuration slots (`ADDR_MOD_0` through `ADDR_MOD_7`) that control automatic address counter updates during TTI instruction execution. Each slot stores increment amounts, clear flags, and carry-return behavior for multi-dimensional address counters tracking positions within the tile structure (X, Y, Z, W dimensions).

Hardware instructions include an ADDR_MOD slot index parameter (e.g., `TTI_PACR(ADDR_MOD_1, ...)`, `TTI_MVMUL(..., ADDR_MOD_0, ...)`). After the instruction executes, the hardware automatically applies the slot's configuration to update the address counters. This enables complex access patterns—face-by-face traversal, strided rows, broadcast operations—without explicit counter manipulation in each MOP iteration.

**Key Mechanism**:

1.   Configuration phase writes `addr_mod_pack_t` or `addr_mod_t` structures to ADDR_MOD slots via `.set(ADDR_MOD_N)` method
2.   MOP template embeds ADDR_MOD slot references in TTI instructions
3.   Hardware applies slot configuration to counters after each TTI instruction execution

## Address Counter Dimensions

### Tile Organization

Hardware address counters track positions within the 4-dimensional tile structure. Each dimension serves a specific purpose in accessing tile data:

| Dimension | Purpose | Typical Range | Usage |
| --- | --- | --- | --- |
| **X** | Column within a face | 0-15 (FACE_C_DIM-1) | Horizontal position in 16x16 face |
| **Y** | Row within a face | 0-15 (FACE_R_DIM-1) | Vertical position in 16x16 face |
| **Z** | Face index within tile | 0-3 | Which of 4 faces (for 32x32 tile) |
| **W** | Tile index | 0-15 | Which tile in destination register |

## Control Field Semantics

Each address counter in an address modifier configuration has three control fields that define its update behavior:

| Field | Type | Range | Hardware Behavior |
| --- | --- | --- | --- |
| `incr` | uint8_t | 0-15 | `counter = counter + incr` (applied first) |
| `clr` | uint8_t | 0-1 | If 1: `counter = 0` (overrides incr) |
| `cr` | uint8_t | 0-1 | If 1: `counter = base_value; next_dimension_counter++` |

**Field Application Order** ([llk_pack.h 23-27](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack.h#L23-L27)[llk_pack_untilize.h 40-47](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack_untilize.h#L40-L47)):

### Increment Field (incr)

The `incr` field is a 4-bit value (0-15) added to the counter after the instruction executes. Some operations apply additional implicit increments:

**Example:** Packer row increment with hardware addition ([llk_pack.h 23-27](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack.h#L23-L27)):

The `TTI_INCADCXY` instruction combines with this modifier to achieve the 16-row stride.

### Clear Flag (clr)

The `clr` flag forces the counter to 0, overriding the `incr` value. Used to reset a dimension at loop boundaries.

**Example:** Reset Y counter after face completion ([llk_pack.h 39-46](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack.h#L39-L46)):

When this ADDR_MOD is applied: `y_src = 0; y_dst = 0;` (spatial counters reset, face counters preserved).

### Carry/Reset Flag (cr)

The `cr` flag implements wrap-around: the current counter resets to its base value and the next higher dimension counter increments. Enables automatic face-to-face or tile-to-tile progression.

**Example:** Wrap Y and advance Z (face) counter ([llk_pack_untilize.h 40-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack_untilize.h#L40-L43)):

**Hardware Behavior When cr=1:**

```
if (y_src == y_src_end_value) {
    y_src = 0;      // Reset to base
    z_src++;        // Advance next dimension
}
```

This creates the untilize access pattern: rows 0,1,2,...,15 (face 0) → rows 0,1,2,...,15 (face 1).

**Common Usage**: Face-to-face transitions in untilize operations, tile-to-tile advancement in matmul outer loops.

Sources: [tt_llk_wormhole_b0/llk_lib/llk_pack_untilize.h 40-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_untilize.h#L40-L43)[tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 64-71](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L64-L71)

## Address Modifier Slot Allocation

The hardware provides 8 address modifier configuration slots (`ADDR_MOD_0` through `ADDR_MOD_7`). Operations configure these slots during initialization and reference them by index in TTI instructions. Slot allocation conventions prevent conflicts when multiple operation types run in sequence (e.g., matmul followed by element-wise operations).

**Slot Allocation Strategy**:

*   ADDR_MOD_0-2: Core operation patterns (inner loops, transitions)
*   ADDR_MOD_3-4: Operation-specific extensions (bias, outer loops)
*   ADDR_MOD_5-6: High-fidelity multi-phase operations
*   ADDR_MOD_7: Reserved/unused

**Key Constraint**: When chaining operations in a single kernel, operations must use non-overlapping ADDR_MOD slots or reconfigure slots between operations with proper synchronization.

**Address Modifier Slot Usage Conventions**

**Slot Allocation Table:**

| Slot | Matmul | Element-wise | Pack | Reduce |
| --- | --- | --- | --- | --- |
| ADDR_MOD_0 | Inner loop body (reuse SrcA) | Single element increment | Row increment within face | Pool with carry |
| ADDR_MOD_1 | Column block transition | Dest-to-SrcA/B move | Face boundary (clear/carry) | Unused |
| ADDR_MOD_2 | Row block transition | Batch (8 elements) | Tile flush | Transpose move (B2D) |
| ADDR_MOD_3 | ADDR_MOD_0 + bias | Unused | Unused | Unused |
| ADDR_MOD_4 | Face/outer loop completion | Unused | Unused | Unused |
| ADDR_MOD_5 | Fidelity phase increment | Unused | Unused | Unused |
| ADDR_MOD_6 | Complete reset + fidelity clear | Unused | Unused | Unused |
| ADDR_MOD_7 | Reserved | Unused | Unused | Unused |

## Address Modifier Usage in Hardware Instructions

TTI instructions include an ADDR_MOD slot index parameter specifying which configuration to apply after execution. The parameter position varies by instruction type.

**TTI Instruction Address Modifier Parameter Positions**

**Key Difference**: Unpacker instructions encode counter control directly (Z increment bitmask) rather than referencing ADDR_MOD slots. Math and packer instructions reference pre-configured ADDR_MOD slots.

### Packer TTI_PACR

`TTI_PACR` packs data from destination registers to L1 memory, applying the specified ADDR_MOD configuration.

**Instruction Format** ([llk_pack.h 75](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_pack.h#L75-L75) generated via `TT_OP_PACR` macro):

**Hardware Execution Sequence**:

1.   Read destination register at current `y_src`, `z_src` position
2.   Write to L1 memory at current `y_dst`, `z_dst` position
3.   Apply ADDR_MOD_1 configuration: `y_src += incr; y_dst += incr; /* apply clr/cr flags */`

### Math TTI_MVMUL

`TTI_MVMUL` performs 8-row matrix-vector multiply `D[8,16] = B[8,16] × A[16,16]`, applying ADDR_MOD configuration to SrcA, SrcB, Dest counters.

**Instruction Format** ([llk_math_matmul.h 380-384](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_math_matmul.h#L380-L384)):

**Hardware Execution Sequence**:

1.   Compute 8 destination rows: `dest[0:7] = srcb[0:7] × srca[0:15]` (hardware performs 16×16→8 reduction)
2.   Apply `p_setrwc::CLR_NONE`: Don't clear SrcA/SrcB counters
3.   Apply ADDR_MOD_0 configuration: `srcb += 8; dest += 8; srca += 0;`

### Unpacker TTI_UNPACR

Unpacker instructions encode address counter control directly in the instruction rather than referencing ADDR_MOD slots. Counter updates are specified via the `z_inc` parameter.

**Instruction Format** ([llk_unpack_A.h 39](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L39-L39)):

**Z Increment Encoding**:

*   `0b00000000`: No Z increment
*   `0b00000001`: Increment ch0_z (face counter for unpacker channel 0)
*   `0b00010000`: Increment ch1_z (face counter for unpacker channel 1)
*   `0b00010001`: Increment both ch0_z and ch1_z

**Example**: Standard tile unpack ([llk_unpack_A.h 32-33](https://github.com/tenstorrent/tt-llk/blob/366f58e2/llk_unpack_A.h#L32-L33)):

**Key Difference from Math/Packer**: Unpacker counter control is embedded in instruction encoding, not configured via ADDR_MOD slots. This allows unpacker MOP templates to avoid ADDR_MOD slot allocation conflicts with math/packer operations.

## Blackhole Architecture Differences

### Enhanced Packer Instruction Control

Blackhole extends `TTI_PACR` with additional parameters for strided memory access and multi-context operation.

**Blackhole TTI_PACR Format** ([tt_llk_blackhole/llk_lib/llk_pack_untilize.h 69-80](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L69-L80)):

**DST_ACCESS_STRIDED_MODE**: In strided mode, each packer interface uses stride `16 * block_size` bytes, where `block_size = SCALE_DATUM_SIZE(format, full_ct_dim * TILE_C_DIM * (TILE_R_DIM/4))`. This enables efficient block-untilize operations where multiple destination tiles are interleaved.

**Address Modifier Structures**: Blackhole uses the same `addr_mod_pack_t` structure as Wormhole but interprets fields in strided mode differently:

*   `y_src.incr`: Row increment within source tile face
*   `y_dst.incr`: Row increment within destination block (scaled by stride in strided mode)

**Example**: Block untilize configuration ([tt_llk_blackhole/llk_lib/llk_pack_untilize.h 34-47](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L34-L47)):

## Summary

Address modifiers provide fine-grained control over hardware address counter behavior during pipeline operations. By configuring increment amounts, clear flags, and carry behavior for multiple dimensions, they enable complex data access patterns without explicit software management on each iteration. The 8 available ADDR_MOD slots allow operations to switch between different addressing modes as needed within MOP sequences.

Key concepts:

*   **Multi-dimensional counters**: X, Y, Z, W dimensions track positions in tile structure
*   **Three control fields**: `incr`, `clr`, `cr` control counter update behavior
*   **Stage-specific structures**: `addr_mod_pack_t` for packer, `addr_mod_t` for math
*   **Slot-based selection**: Operations reference ADDR_MOD_0 through ADDR_MOD_7 by index
*   **Common patterns**: Sequential face processing, row striding, broadcast handling, multi-phase resets
