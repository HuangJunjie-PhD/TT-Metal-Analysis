## Purpose and Scope

The Backend Configuration System provides memory-mapped registers that control the behavior of the Tensix coprocessor's execution units (unpackers, packers, Matrix Unit, Vector Unit, etc.). This system consists of two primary configuration arrays: `Config` (thread-agnostic) and `ThreadConfig` (thread-specific), which can be accessed by both RISC-V control cores and Tensix coprocessor instructions.

For configuration of the Replay Expanders and MOP Expanders, see [Instruction Pipeline and Frontend](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend). For Vector Unit-specific configuration (`LaneConfig` and `LoadMacroConfig`), see [Vector Unit (SFPU) Architecture](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.5-vector-unit-(sfpu)-architecture). For packer and unpacker configuration via TDMA-RISC, see [Data Movement: Mover and TDMA-RISC](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.4-programmable-interrupt-controller-(pic)).

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 1-88](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L1-L88)

## Configuration Memory Layout

The backend configuration system exposes two main data structures mapped into the RISC-V address space starting at `TENSIX_CFG_BASE`:

**Configuration Variable Structure**

| Variable | Type | Banks | Purpose |
| --- | --- | --- | --- |
| `Config` | `uint32_t[2][CFG_STATE_SIZE * 4]` | 2 state banks | Thread-agnostic configuration for all execution units |
| `ThreadConfig` | `struct {uint16_t Value, Padding[7];}[3][THD_STATE_SIZE]` | 3 thread banks | Thread-specific configuration for T0, T1, T2 |

**Address Space Layout:**

*   Base address: `TENSIX_CFG_BASE` (typically `0x1FC00000`)
*   `Config[0]` starts at offset `0x00000`
*   `Config[1]` starts at offset `CFG_STATE_SIZE * 4 * sizeof(uint32_t)`
*   `ThreadConfig` follows immediately after `Config[1]`

**Field Access Notation**

Configuration fields are accessed using a shorthand notation in instruction descriptions:

*   `Config[i].Field` means `(Config[i][Field_ADDR32] & Field_MASK) >> Field_SHAMT`
*   `ThreadConfig[i].Field` means `(ThreadConfig[i][Field_ADDR32].Value & Field_MASK) >> Field_SHAMT`
*   `ThreadConfig[CurrentThread].Field` automatically indexes the correct thread (T0=0, T1=1, T2=2)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 3-20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L3-L20)

## Configuration Access Methods

**Access Method Summary**

| Client | Config (Thread-agnostic) | ThreadConfig (Thread-specific) |
| --- | --- | --- |
| **Tensix implicit read** | Various instructions | Various instructions |
| **Tensix explicit read** | `RDCFG` | Not supported |
| **Tensix write** | `WRCFG`, `RMWCIB`, `REG2FLOP` | `SETC16` |
| **RISC-V read** | `lw`, `lh`, `lhu`, `lb`, `lbu` | `lw`, `lh`, `lhu`, `lb`, `lbu` |
| **RISC-V write** | `sw` only (32-bit) | Not supported |

**Important Restrictions:**

*   RISC-V cores can only write to `Config` using 32-bit `sw` instructions (no 16-bit or 8-bit stores)
*   RISC-V cores cannot write directly to `ThreadConfig`; they must push `SETC16` instructions instead
*   `REG2FLOP` can only write to `Config[i][j]` when `THCON_CFGREG_BASE_ADDR32 ≤ j < GLOBAL_CFGREG_BASE_ADDR32`

## Configuration State Banks

The `Config` array provides two state banks (indices 0 and 1) that allow rapid context switching between different configuration sets. Each Tensix thread selects which bank to use via the `CFG_STATE_ID_StateID` field in its `ThreadConfig`:

**Packer Configuration Access**

Packer configuration uses a special shorthand notation that maps logical packer indices to specific configuration register locations:

| Shorthand | Actual Register Location |
| --- | --- |
| `Config[i].Packers[0].Field` | `Config[i].THCON_SEC0_REG1_Field` |
| `Config[i].Packers[1].Field` | `Config[i].THCON_SEC0_REG8_Field` |
| `Config[i].Packers[2].Field` | `Config[i].THCON_SEC1_REG1_Field` |
| `Config[i].Packers[3].Field` | `Config[i].THCON_SEC1_REG8_Field` |

This notation is used throughout instruction functional models to access packer-specific configuration, such as:

*   `PackerIConfig.In_data_format` - Input data format
*   `PackerIConfig.Out_data_format` - Output data format
*   `PackerIConfig.Addr_cnt_context` - ADC context selection
*   `PackerIConfig.Disable_zero_compress` - Zero compression control
*   `PackerIConfig.L1_Dest_addr` - L1 destination base address

**Global Configuration Region**

Writes to `Config[i][j]` where `j >= GLOBAL_CFGREG_BASE_ADDR32` will automatically write to **both**`Config[0][j]` and `Config[1][j]`. This region is used for configuration that should apply to all threads regardless of their state bank selection. Instruction descriptions use the shorthand `Config.Field` (without bank index) when `Field_ADDR32 >= GLOBAL_CFGREG_BASE_ADDR32`.

## Race Conditions and Synchronization

RISC-V execution and Tensix execution proceed asynchronously. Once a Tensix instruction is pushed, it will eventually execute, but a variable number of RISC-V instructions might execute first. Additionally, RISC-V cores can reorder memory writes to different memory regions. This creates two primary race condition scenarios:

**Danger 1: Configuration Write Before Instruction Push**

When an RISC-V core writes to `Config` followed by pushing a Tensix instruction intended to consume that configuration:

**Problem:** The two `sw` instructions can be reordered, causing the Tensix instruction to execute with old configuration.

**Solutions:**

1.   **Use Tensix configuration instructions instead:** Push a `WRCFG`, `RMWCIB`, or `REG2FLOP` instruction to perform the configuration write (these instructions execute in the Tensix pipeline and maintain proper ordering)
2.   **Use `STALLWAIT` synchronization:** Push a `STALLWAIT` instruction after writing `Config` and before pushing the target instruction, specifying condition `C13` with appropriate block mask to ensure configuration is visible
3.   **Use RISC-V memory ordering fence sequence:**

```
sw   t0, 0(t1)   # Write to Config at TENSIX_CFG_BASE
lw   t2, 0(t1)   # Read back from same address (guaranteed to finish after store)
addi x0, t2, 0   # Consume result of load (blocks subsequent instructions)
sw   t3, 0(t4)   # Push Tensix instruction (now guaranteed to see Config write)
```

**Danger 2: Instruction Push Before Configuration Write**

When an RISC-V core pushes a Tensix instruction followed by writing to `Config`, where that configuration is intended only for future instructions:

**Problem:** The pushed instruction might execute after the configuration write, consuming the new configuration when it wasn't meant to.

**Solution:** Ensure the pushed Tensix instruction has started (or finished) execution before performing the `sw` to write to `Config`.

## Immediate-Effect Configuration Writes

Most writes to `Config` or `ThreadConfig` are consumed by subsequent Tensix instructions. However, certain configuration writes trigger immediate hardware actions independent of instruction execution:

**`Config` Immediate-Effect Registers:**

| Address/Field | Immediate Effect | Notes |
| --- | --- | --- |
| `Config[i][STATE_RESET_EN_ADDR32]` | Zeros all `Config[i][j]` where `j < GLOBAL_CFGREG_BASE_ADDR32` | Does not apply when written via `RMWCIB`; equivalent to soft reset of configuration |
| `Config.PRNG_SEED_Seed_Val_ADDR32` | Re-seeds all PRNGs (Matrix Unit, Vector Unit, Packers) with the written value | Affects stochastic rounding behavior |
| `Config.RISCV_IC_INVALIDATE_InvalidateAll` | Invalidates instruction caches of masked RISC-V cores | Bit mask: 0=B, 1=T0, 2=T1, 3=T2, 4=NC |

**`ThreadConfig` Immediate-Effect Registers (via `SETC16`):**

| Address/Field | Immediate Effect | Purpose |
| --- | --- | --- |
| `ThreadConfig[i][CG_CTRL_EN_Regblocks_ADDR32]` | Enable/disable clock gating for register blocks | Power management |
| `ThreadConfig[i][CG_CTRL_EN_Hyst_ADDR32]` | Configure clock gating hysteresis | Power management |
| `ThreadConfig[i][CG_CTRL_KICK_Regblocks_ADDR32]` | Manually kick clock gating state machine | Power management |
| `ThreadConfig[i][UNPACK_MISC_CFG_CfgContextCntReset_0_ADDR32]` | Reset unpacker 0 configuration context counter for thread i | Multi-context unpacker control |
| `ThreadConfig[i][UNPACK_MISC_CFG_CfgContextCntReset_1_ADDR32]` | Reset unpacker 1 configuration context counter for thread i | Multi-context unpacker control |

**RISC-V Control Configuration**

Several fields affect RISC-V cores rather than the Tensix backend:

**Branch Predictor Control:**

*   `Config[0].DISABLE_RISC_BP_Disable_main` - Disable B core branch predictor
*   `Config[0].DISABLE_RISC_BP_Disable_trisc` - Disable T0/T1/T2 branch predictors
*   `Config[0].DISABLE_RISC_BP_Disable_ncrisc` - Disable NC core branch predictor
*   `Config[0].DISABLE_RISC_BP_Disable_bmp_clear_*` - Partial disable (not recommended)

**Reset PC Override:**

*   `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en` - Enable custom reset PC for T0/T1/T2
*   `Config.TRISC_RESET_PC_SEC0_PC` - Reset PC for T0
*   `Config.TRISC_RESET_PC_SEC1_PC` - Reset PC for T1
*   `Config.TRISC_RESET_PC_SEC2_PC` - Reset PC for T2
*   `Config.NCRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en` - Enable custom reset PC for NC
*   `Config.NCRISC_RESET_PC_PC` - Reset PC for NC

**Instruction Cache Prefetcher:**

*   `Config.BRISC_END_PC_PC` - Address upper bound for B core prefetcher
*   `Config.TRISC_END_PC_SEC0_PC` - Address upper bound for T0 prefetcher
*   `Config.TRISC_END_PC_SEC1_PC` - Address upper bound for T1 prefetcher
*   `Config.TRISC_END_PC_SEC2_PC` - Address upper bound for T2 prefetcher
*   `Config.NOC_RISC_END_PC_PC` - Address upper bound for NC prefetcher
*   `Config.RISC_PREFETCH_CTRL_Enable_Brisc` - Enable/disable B core prefetcher
*   `Config.RISC_PREFETCH_CTRL_Enable_Trisc` - Enable/disable T0/T1/T2 prefetchers
*   `Config.RISC_PREFETCH_CTRL_Enable_NocRisc` - Enable/disable NC prefetcher
*   `Config.RISC_PREFETCH_CTRL_Max_Req_Count` - Limit in-flight prefetch requests

## Configuration Outside the Backend System

Not all Tensix coprocessor configuration resides in `Config` or `ThreadConfig`. Several subsystems maintain separate configuration spaces:

**Configuration Location Reference:**

| Subsystem | Configuration Method | Access Pattern | Reference |
| --- | --- | --- | --- |
| Replay Expanders | In-band via `REPLAY` instruction | Per-thread, in-band | See [#4.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend) |
| MOP Expanders | `MopCfg[9]` at `TENSIX_MOP_CFG_BASE` | Write-only MMIO, per-thread | See [#4.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend) |
| Vector Unit (SFPU) | `LaneConfig` and `LoadMacroConfig` | In-band via `SFPCONFIG` | See [#4.5](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.5-vector-unit-(sfpu)-architecture) |
| TDMA-RISC | Packer/unpacker-specific config | Command-based MMIO | See [#3.4](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.4-programmable-interrupt-controller-(pic)) |
| ADCs (Address Counters) | Auto-increment addressing counters | Implicit state, instructions modify | See [#4.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.3-unpacker-system-(unpacr)), [#4.6](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.6-packer-system-(pacr)) |
| RWCs (Row/Column Counters) | Row/column addressing counters | Implicit state, instructions modify | See [#4.4](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.4-matrix-unit-(fpu)-architecture), [#4.5](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.5-vector-unit-(sfpu)-architecture) |

**MOP Expander Configuration Details**

Each MOP Expander has a `uint32_t MopCfg[9]` array mapped at `TENSIX_MOP_CFG_BASE` in the corresponding RISC-V core's address space:

| RISC-V Core | MOP Expander Target | Address Base |
| --- | --- | --- |
| RISC-V T0 | Tensix thread T0 MOP Expander | `TENSIX_MOP_CFG_BASE` |
| RISC-V T1 | Tensix thread T1 MOP Expander | `TENSIX_MOP_CFG_BASE` |
| RISC-V T2 | Tensix thread T2 MOP Expander | `TENSIX_MOP_CFG_BASE` |

This address range is **write-only**; reads trigger undefined behavior. Configuration should not be changed while a MOP expansion is in progress. Use TTSync to wait for expansions to finish before modifying `MopCfg`.

## Debug Access to Configuration

`Config` and `ThreadConfig` are not mapped into the RISC-V NC address space, nor are they accessible via the NoC. However, debug functionality exists to allow external access:

**Debug Read Procedure:**

1.   Write value `x` to `RISCV_DEBUG_REG_CFGREG_RD_CNTL`
2.   Wait a few cycles for hardware to process
3.   Hardware performs: `RISCV_DEBUG_REG_CFGREG_RDDATA = ((uint32_t*)TENSIX_CFG_BASE)[x & 0x7ff]`
4.   Read result from `RISCV_DEBUG_REG_CFGREG_RDDATA`

**Address Translation:**

*   The index `x & 0x7ff` provides 11 bits of address space
*   Addressable range: 0-2047 in 32-bit word units
*   This covers both `Config[0][]`, `Config[1][]`, and `ThreadConfig[0..2][]` arrays
*   The debug registers exist in the "Tile control / debug / status registers" memory region
*   This region is accessible to: 
    *   All RISC-V cores (B, T0, T1, T2, NC)
    *   External clients via NoC

**Use Cases:**

*   Remote inspection of Tensix backend configuration
*   Debugging configuration issues from RISC-V NC (which cannot directly access `TENSIX_CFG_BASE`)
*   External monitoring and debugging via NoC

## Configuration Field Reference

Complete field definitions (`Field_ADDR32`, `Field_MASK`, `Field_SHAMT`) are available in `cfg_defines.h`:

**Structure of `cfg_defines.h`:**

*   `// Registers for THREAD` section: Fields for indexing into `ThreadConfig`
*   All other sections: Fields for indexing into `Config`

The `cfg_defines.h` file contains 354-398 configuration registers covering:

*   Packer configuration (data format, addressing, compression)
*   Unpacker configuration (data format, addressing, decompression)
*   Matrix Unit (FPU) configuration
*   Address generation parameters (strides, base addresses)
*   Clock gating control
*   RISC-V prefetcher control
*   Synchronization and control registers
