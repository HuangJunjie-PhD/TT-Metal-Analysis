## Purpose and Scope

This document describes the `ClusterDescriptor` class and the `TopologyDiscovery` system in tt-umd. The `ClusterDescriptor` serves as the authoritative source of hardware topology information, containing mappings between logical chip IDs, unique ASIC IDs, Ethernet connections, physical locations, and board types. It handles the parsing of YAML descriptor files and represents the connectivity graph of the Tenstorrent cluster.

For information about how the `Cluster` uses the `ClusterDescriptor` at runtime, see [Cluster Management](https://deepwiki.com/tenstorrent/tt-umd/2.1-cluster-management). For details on coordinate systems and spatial addressing, see [Coordinate Systems & SOC Descriptors](https://deepwiki.com/tenstorrent/tt-umd/5.1-coordinate-translations).

## ClusterDescriptor Data Structures

The `ClusterDescriptor` class maintains several critical data structures that define the system topology:

| Data Structure | Type | Purpose |
| --- | --- | --- |
| `chip_unique_ids` | `std::unordered_map<ChipId, uint64_t>` | Maps logical `ChipId` (0, 1, 2...) to unique ASIC ID. |
| `ethernet_connections` | `std::unordered_map<ChipId, std::unordered_map<EthernetChannel, std::tuple<ChipId, EthernetChannel>>>` | Bidirectional Ethernet link mappings between chips within the cluster. |
| `chip_locations` | `std::unordered_map<ChipId, EthCoord>` | Physical coordinates (rack, shelf, x, y) for multi-rack systems. |
| `chips_with_mmio` | `std::unordered_map<ChipId, ChipId>` | Maps logical chip to its PCIe device index. |
| `chip_board_type` | `std::unordered_map<ChipId, BoardType>` | Board type (N150, N300, P150, P300, UBB, etc.). |
| `harvesting_masks` | `std::unordered_map<ChipId, HarvestingMasks>` | Disabled cores (Tensix, DRAM, ETH, PCIe) due to harvesting. |

### EthCoord Structure

The `EthCoord` structure provides spatial addressing for large-scale deployments:

`struct EthCoord {    int cluster_id;  // Unique ID for the connected cluster    int x;           // X coordinate within the topology    int y;           // Y coordinate within the topology    int rack;        // Rack number in datacenter    int shelf;       // Shelf within rack};`
This coordinate system enables routing in deployments where chips are distributed across multiple shelves and racks. The `cluster_id` is used to group chips that are physically connected via Ethernet.

## Cluster Descriptor YAML Schema

The `ClusterDescriptor` can be instantiated from YAML files that describe the hardware configuration. This is particularly useful for simulation or when hardware discovery is performed offline.

### Key YAML Properties

*   **arch**: Maps Chip ID to architecture type (`wormhole_b0`, `blackhole`, `quasar`).
*   **chips**: Ethernet coordinates `[rack, shelf, y, x]` for each chip (primarily Wormhole).
*   **ethernet_connections**: A list of arrays defining links between `chip` and `chan`.
*   **chips_with_mmio**: List of chips accessible via PCIe and their logical index.
*   **harvesting**: Per-chip masks for `noc_translation`, `harvest_mask`, and subsystem-specific masks.
*   **boards**: List of boards, each containing `board_id`, `board_type`, and the `chips` residing on it.

**Sources:**[docs/yaml_schemas/cluster_descriptor.yaml 20-230](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/docs/yaml_schemas/cluster_descriptor.yaml#L20-L230)[device/cluster_descriptor.cpp 130-145](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L130-L145)

## Topology Discovery & Constraints

### TT_VISIBLE_DEVICES Filtering

The UMD supports the `TT_VISIBLE_DEVICES` environment variable to restrict the chips the driver will interact with. This can be specified using logical indices or PCIe BDF (Bus:Device.Function) addresses.

When a constrained descriptor is created, UMD filters the `chips_with_mmio`, `ethernet_connections`, and `all_chips` sets to include only the requested devices and their immediate dependencies.

**Sources:**[device/cluster_descriptor.cpp 81-82](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L81-L82)[tests/api/test_tt_visible_devices.cpp 76-81](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/api/test_tt_visible_devices.cpp#L76-L81)[tests/baremetal/test_cluster_descriptor_offline.cpp 140-162](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/baremetal/test_cluster_descriptor_offline.cpp#L140-L162)

### Closest MMIO Chip Logic

For remote chips (accessed via Ethernet), the `ClusterDescriptor` provides logic to find the "gateway" chip that has a physical PCIe connection to the host.

`ChipId ClusterDescriptor::get_closest_mmio_capable_chip(const ChipId chip) {    if (this->is_chip_mmio_capable(chip)) return chip;        // Search for MMIO chips on the same board    const auto chips_on_the_same_board = get_board_chips(get_board_id_for_chip(chip));    for (const ChipId &candidate_mmio_chip : chips_on_the_same_board) {        if (is_chip_mmio_capable(candidate_mmio_chip)) return candidate_mmio_chip;    }    // ... throws if not found}`

## Ethernet Link Management

The `ClusterDescriptor` tracks active Ethernet links to facilitate multi-chip communication.

### Key Connection Functions

*   **ethernet_core_has_active_ethernet_link**: Checks if a specific local chip and channel have a defined connection in the descriptor. [device/cluster_descriptor.cpp 38-47](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L38-L47)
*   **get_chip_and_channel_of_remote_ethernet_core**: Returns the `(ChipId, EthernetChannel)` pair for the remote end of a link. [device/cluster_descriptor.cpp 49-65](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L49-L65)
*   **get_directly_connected_ethernet_channels_between_chips**: Returns all channels that form direct links between two specific chips. [device/cluster_descriptor.cpp 69-84](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L69-L84)

### Ethernet Bidirectionality

The descriptor ensures that connections are bidirectional. If chip A channel 1 connects to chip B channel 2, the internal map stores both directions to allow O(1) lookups from either side.

**Sources:**[device/cluster_descriptor.cpp 1111-1125](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster_descriptor.cpp#L1111-L1125)[tests/baremetal/test_cluster_descriptor_offline.cpp 122-128](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/baremetal/test_cluster_descriptor_offline.cpp#L122-L128)
