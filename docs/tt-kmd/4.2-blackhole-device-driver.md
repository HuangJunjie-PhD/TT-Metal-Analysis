## Purpose and Scope

This document describes the Blackhole-specific device driver implementation, which provides hardware support for Tenstorrent Blackhole ASICs (PCI Device ID: 0xB140). The Blackhole driver implements the generic `tenstorrent_device_class` interface to provide 58-bit DMA addressing, two-tier TLB configuration, ARC firmware communication via message queues, and advanced power management features.

For information about the generic device abstraction layer, see [Device Abstraction Layer](https://deepwiki.com/tenstorrent/tt-kmd/3.2-device-abstraction-layer). For comparison with the previous hardware generation, see [Wormhole Device Driver](https://deepwiki.com/tenstorrent/tt-kmd/4.1-wormhole-device-driver). For details on ARC firmware communication protocols, see [ARC Firmware Communication](https://deepwiki.com/tenstorrent/tt-kmd/4.3-arc-firmware-communication).

**Sources:**[blackhole.c 1-1123](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1-L1123)[blackhole.h 1-32](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.h#L1-L32)

## Hardware Characteristics

The Blackhole device class defines the following hardware characteristics:

| Property | Value | Description |
| --- | --- | --- |
| **DMA Address Bits** | 58 | Maximum addressable space for DMA operations |
| **NOC DMA Limit** | `(1ULL << 58) - 1` | Maximum address for NOC DMA transactions |
| **NOC PCIe Offset** | `4ULL << 58` | Offset added to NOC addresses for PCIe access |
| **TLB Kinds** | 2 | Two types of TLB windows (2M and 4G) |
| **2M TLB Count** | 202 | Number of 2MB translation windows |
| **4G TLB Count** | 8 | Number of 4GB translation windows (limited by BAR4 size) |
| **2M Window Size** | `1 << 21` (2MB) | Size of small TLB windows |
| **4G Window Size** | `1UL << 32` (4GB) | Size of large TLB windows |
| **Outbound IATU Regions** | 16 | Number of outbound address translation regions |
| **Max IATU Region Size** | 1TB | Maximum size per IATU region |

The device class is instantiated in `blackhole_class` at [blackhole.c 1099-1122](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1099-L1122) which populates the `tenstorrent_device_class` function pointer table.

**Sources:**[blackhole.c 19-30](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L19-L30)[blackhole.c 1099-1122](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1099-L1122)

## Device Structure and Memory Layout

**Diagram: Blackhole Device Structure and MMIO Mappings**

The `blackhole_device` structure extends `tenstorrent_device` with device-specific state:

*   **TLB registers:** Memory-mapped at BAR0 offset 0x1FC00000, providing access to all 210 TLB configuration registers (202 x 2M + 8 x 4G) [blackhole.c 33-34](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L33-L34)
*   **Kernel TLB:** Reserved last 2M TLB window (index 201) for kernel NOC access, protected by dedicated mutex [blackhole.c 40-42](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L40-L42)
*   **NOC2AXI configuration:** Access to NOC status counters and PCIe performance metrics at BAR0 offset 0x1FD00000 [blackhole.c 44-45](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L44-L45)
*   **BAR2 mapping:** Full BAR2 access for IATU configuration [blackhole.h 17](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.h#L17-L17)

**Sources:**[blackhole.h 10-23](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.h#L10-L23)[blackhole.c 32-48](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L32-L48)

## TLB Configuration System

### TLB Window Types and Layout

**Diagram: TLB Window Allocation and Layout**

The Blackhole device provides two tiers of TLB windows:

**2M Windows (TLB 0-201):**

*   Total count: 202 windows [blackhole.c 20](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L20-L20)
*   Mapped in BAR0.
*   User-accessible: 0-200.
*   Kernel-reserved: 201 (index `KERNEL_TLB_INDEX`) [blackhole.c 40](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L40-L40)
*   First 32 windows support strided access patterns for non-rectangular multicast [blackhole.c 36](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L36-L36)

**4G Windows (TLB 202-209):**

*   Total count: 8 windows [blackhole.c 25](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L25-L25)
*   Mapped in BAR4.
*   Final count determined at runtime based on BAR4 size [blackhole.c 861](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L861-L861)

**Sources:**[blackhole.c 19-41](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L19-L41)[blackhole.c 853-862](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L853-L862)

### TLB Register Structure

Both 2M and 4G TLB windows use 12-byte register structures with packed bitfields:

**2M TLB Register Layout (`struct TLB_2M_REG`):**[blackhole.c 112-136](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L112-L136)

*   **address**: 43 bits (shifted right by 21)
*   **x_start / x_end**: 6 bits each
*   **y_start / y_end**: 6 bits each
*   **noc**: 2 bits
*   **multicast**: 1 bit
*   **ordering**: 2 bits

**4G TLB Register Layout (`struct TLB_4G_REG`):**[blackhole.c 139-162](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L139-L162)

*   **address**: 32 bits (shifted right by 32)
*   Fields identical to 2M for coordinates and NOC flags.

**Sources:**[blackhole.c 112-163](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L112-L163)

## Kernel NOC Access

The kernel reserves TLB window 201 for its own NOC access needs, protecting it with `kernel_tlb_mutex`. Each NOC access:

1.   Locks the kernel TLB mutex [blackhole.c 253](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L253-L253)
2.   Configures TLB 201 via `bh_configure_kernel_tlb`[blackhole.c 233-246](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L233-L246)
3.   Performs the MMIO read/write at `kernel_tlb + offset`[blackhole.c 256](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L256-L256)
4.   Unlocks the mutex [blackhole.c 258](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L258-L258)

**CSM (Core System Memory) Access:** Helper functions `csm_read32` and `csm_write32`[blackhole.c 275-291](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L275-L291) wrap NOC access with range checking via `is_range_within_csm()`[telemetry.h 75-78](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.h#L75-L78)

**Sources:**[blackhole.c 233-291](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L233-L291)[telemetry.h 73-80](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.h#L73-L80)

## ARC Firmware Message Queue Protocol

### Message Queue Structure

**Diagram: ARC Message Queue Structure**

The ARC message queue protocol uses a dual-ring buffer architecture. The `ARC_MSG_QCB_PTR`[blackhole.c 64](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L64-L64) points to the Control Block which defines the `queue_base` address.

**Message Transmission ([blackhole.c 781-821](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L781-L821)):**

1.   Poll `ARC_BOOT_STATUS` for `ARC_BOOT_STATUS_READY_FOR_MSG`[blackhole.c 73-74](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L73-L74)
2.   Discovery of queue parameters from CSM.
3.   Push message to request queue via `push_arc_msg`[blackhole.c 676-724](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L676-L724)
4.   Trigger ARC via `ARC_MSI_FIFO`[blackhole.c 65](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L65-L65)
5.   Pop response via `pop_arc_msg`[blackhole.c 726-779](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L726-L779)

**Sources:**[blackhole.c 63-79](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L63-L79)[blackhole.c 670-821](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L670-L821)

## Telemetry and Monitoring

### Tag Discovery and Resolution

The driver dynamically discovers sensor addresses from ARC firmware during `telemetry_probe`[blackhole.c 619-668](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L619-L668)

1.   Reads pointers from `ARC_TELEMETRY_PTR` and `ARC_TELEMETRY_DATA`[blackhole.c 60-61](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L60-L61)
2.   Iterates through the tag array to resolve `tag_id` to CSM offsets.
3.   Matches tags against `bh_hwmon_attrs`[blackhole.c 407-413](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L407-L413) and `bh_sysfs_attributes`[blackhole.c 421-432](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L421-L432)

### HWMON Integration

Exposes sensors via `tt_hwmon_ops`[telemetry.c 228-232](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.c#L228-L232):

*   **Temperature**: `TELEMETRY_ASIC_TEMP` (16.16 fixed-point) [telemetry.c 184-189](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.c#L184-L189)
*   **Power/Current**: `TELEMETRY_POWER`, `TELEMETRY_CURRENT`[telemetry.c 194-197](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.c#L194-L197)
*   **Fan**: `TELEMETRY_FAN_RPM`[telemetry.c 202-204](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.c#L202-L204)

### PCIe Performance Counters

Exposes counters for NOC0/NOC1 at `NOC_STATUS_OFFSET`[blackhole.c 47](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L47-L47):

*   Master/Slave read/write data words [blackhole.c 350-362](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L350-L362)
*   Accessible via sysfs in `pcie_perf_counters` subdirectory [docs/sysfs-attributes.md 93-97](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/docs/sysfs-attributes.md?plain=1#L93-L97)

**Sources:**[blackhole.c 407-432](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L407-L432)[telemetry.c 149-232](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/telemetry.c#L149-L232)[docs/sysfs-attributes.md 33-142](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/docs/sysfs-attributes.md?plain=1#L33-L142)

## Device Lifecycle and Initialization

### Initialization Sequence

**`blackhole_init`[blackhole.c 853-901](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L853-L901):**

*   Maps MMIO regions: `tlb_regs`, `kernel_tlb`, `noc2axi_cfg`, `bar2_mapping`.
*   Reserves `KERNEL_TLB_INDEX` (201).
*   Determines 4G TLB count based on BAR4 size.

**`blackhole_init_hardware`[blackhole.c 903-928](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L903-L928):**

*   Detects Galaxy boards (subsystem 0x0047) and retrains links [blackhole.c 908-912](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L908-L912)
*   Sets A0 power state via ARC message.
*   Configures watchdog timeout.

**`blackhole_cleanup_hardware`[blackhole.c 983-994](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L983-L994):**

*   Transitions device to A3 state if not detached.

**Sources:**[blackhole.c 853-994](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L853-L994)

## IATU Configuration

Blackhole provides 16 outbound IATU regions for host-to-NOC translation.

**`blackhole_configure_outbound_atu`[blackhole.c 1041-1074](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1041-L1074):**

*   Configures base, limit, and target addresses.
*   Writes to BAR2 mapping at `IATU_BASE` (0x1000) [blackhole.c 76](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L76-L76)
*   Uses `INCREASE_REGION_SIZE` flag for large mappings [blackhole.c 91](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L91-L91)

**Sources:**[blackhole.c 76-110](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L76-L110)[blackhole.c 1041-1074](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1041-L1074)

## Reset Implementation

The driver supports two reset types in `blackhole_reset`[blackhole.c 823-851](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L823-L851):

1.   **ASIC + DMC Reset**: Sends `ARC_MSG_TYPE_TRIGGER_RESET` with payload 3 (ASIC+M3) [blackhole.c 841-843](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L841-L843)
2.   **ASIC Reset**: Triggers PCIe timer interrupt [blackhole.c 847](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L847-L847)

Both types set the reset marker in PCIe config space via `set_reset_marker`[blackhole.c 838-846](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L838-L846)

**Sources:**[blackhole.c 823-851](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L823-L851)
