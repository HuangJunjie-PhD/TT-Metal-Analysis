## Purpose and Scope

This document explains the data collection pipeline in TT-SMI, which manages the flow of hardware telemetry and device information from Tenstorrent devices through the backend to the user interface. The pipeline operates in two phases: initial data collection during backend initialization, and continuous telemetry updates via a background thread.

For information about the backend class structure, see [Backend System (TTSMIBackend)](https://deepwiki.com/tenstorrent/tt-smi/6.2-backend-system-(ttsmibackend)). For details on data models and logging, see [Data Models and Logging](https://deepwiki.com/tenstorrent/tt-smi/6.4-data-models-and-logging). For the application layer that consumes this data, see [Application Layer Architecture](https://deepwiki.com/tenstorrent/tt-smi/6.1-application-layer-architecture).

During backend initialization, the `TTSMIBackend.__init__()` method collects all device data once for each detected device. This phase populates the backend storage arrays that serve as the data source for the UI.

## Continuous Telemetry Collection

After initialization, a background thread continuously updates telemetry data every 100 milliseconds. This thread is spawned only when the user navigates to the telemetry tab.

### Telemetry Thread Lifecycle

**Thread Management:**

*   **Global flag `RUNNING_TELEM_THREAD`**[58](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/58)(): Prevents multiple thread spawns

*   **Thread list `TELEM_THREADS`**[57](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/57)(): Tracks active threads for cleanup

## Data Transformation Pipeline

Data undergoes multiple transformations from raw hardware registers to formatted UI text. Each transformation stage adds semantic meaning and presentation formatting.

### Raw Telemetry to Backend Storage

The `get_smbus_board_info()` method reads raw telemetry registers and stores hexadecimal values:

**Method:**`get_smbus_board_info(board_num)`[241-256](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/241-256)()

**Process:**

1.   Call `pyluwen_chip.as_wh().get_telemetry()` or `.as_bh().get_telemetry()`
2.   Convert telemetry struct to dict via `dict_from_public_attrs()`[713-721](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/713-721)()
3.   Filter fields using `constants.SMBUS_TELEMETRY_LIST`[8-64](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/8-64)()
4.   Store as hexadecimal strings (e.g., `"VCORE": "0x32a"`)

**Sources:**[tt_smi/tt_smi_backend.py 241-256](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L241-L256)[tt_smi/constants.py 8-64](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L64)

### Backend Storage to Formatted Telemetry

Platform-specific methods convert hex strings to human-readable floating-point values:

**Wormhole Telemetry Processing**[471-499](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/471-499)():

*   **Voltage:**`int(hex_str, 16) / 1000` → Volts with 2 decimal places
*   **Current:**`int(hex_str, 16) & 0xFFFF` → Amperes with 1 decimal place
*   **Power:**`int(hex_str, 16) & 0xFFFF` → Watts with 1 decimal place
*   **Temperature:**`(int(hex_str, 16) & 0xFFFF) / 16` → Celsius with 1 decimal place
*   **AICLK:**`int(hex_str, 16) & 0xFFFF` → MHz with 0 decimal places
*   **Fan Speed:** Direct value as percentage
*   **Heartbeat:**`int(hex_str, 16) // 5` → Heartbeat counter (~2 per second)

**Blackhole Telemetry Processing**[423-469](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/423-469)():

*   **Voltage:**`int(hex_str, 16) / 1000` → Volts with 2 decimal places
*   **Current:**`int(hex_str, 16) & 0xFFFF` → Amperes with 1 decimal place
*   **Power:**`int(hex_str, 16) & 0xFFFF` → Watts with 1 decimal place
*   **Temperature:**`convert_signed_16_16_to_float()` → Celsius with 1 decimal place (handles signed fixed-point)
*   **AICLK:**`int(hex_str, 16) & 0xFFFF` → MHz with 0 decimal places
*   **Fan Speed:**`min(int(hex_str, 16), 5000) / 50` → Percent conversion from RPM
*   **Heartbeat:**`int(hex_str, 16) // 6` → Heartbeat counter (~2 per second)

**Sources:**[tt_smi/tt_smi_backend.py 423-499](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L423-L499)[tt_smi/tt_smi_backend.py 501-513](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L501-L513)

## Platform-Specific Data Handling

The pipeline includes platform-specific handlers for Wormhole and Blackhole architectures, which have different telemetry register layouts and data formats.

### Platform Dispatch Table

| Data Type | Dispatch Method | Wormhole Handler | Blackhole Handler |
| --- | --- | --- | --- |
| Telemetry | `get_chip_telemetry()` | `get_wh_chip_telemetry()` | `get_bh_chip_telemetry()` |
| Chip Limits | `get_chip_limits()` | `get_wh_chip_limits()` | `get_bh_chip_limits()` |
| DRAM Speed | `get_dram_speed()` | Decode from DDR_STATUS bits [289-306](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/289-306)() | Read DDR_SPEED field [279-288](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/279-288)() |
| DRAM Training | `get_dram_training_status()` | Check for `0x222222` pattern [354-366](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/354-366)() | Check for `0x5555` pattern [367-380](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/367-380)() |

### Telemetry Field Mappings

**Wormhole SMBus Fields** (via `SMBUS_TELEMETRY_LIST`[8-64](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/8-64)()):

*   Temperature: `ASIC_TEMPERATURE` → 16-bit value, divide by 16
*   Heartbeat: `ARC3_HEALTH` → Integer, divide by 5
*   Fan: `FAN_SPEED` → Direct percentage value
*   Voltage: `VCORE` → Millivolts, divide by 1000

**Blackhole TAG Fields** (via `BH_TELEMETRY_LIST`[66-100](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/66-100)()):

*   Temperature: `TAG_ASIC_TEMPERATURE` → Signed 16.16 fixed-point
*   Heartbeat: `TAG_TIMER_HEARTBEAT` → Integer, divide by 6
*   Fan: `TAG_FAN_RPM` → RPM, convert to percentage (RPM / 50, max 5000)
*   Voltage: `TAG_VCORE` → Millivolts, divide by 1000

**Sources:**[tt_smi/tt_smi_backend.py 423-513](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L423-L513)[tt_smi/constants.py 8-100](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L100)

## Update Mechanisms

The pipeline supports two update patterns: one-time initialization and continuous telemetry polling.

### One-Time Data Collection

Static device properties are collected once during initialization and never updated:

**Static Data Types:**

*   **Firmware Versions**[588-649](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/588-649)(): ARC, ETH, M3 bootloader/app, TT Flash, FW bundle
*   **Device Information**[384-415](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/384-415)(): Bus ID, board type, board ID, coordinates
*   **PCI Properties**[310-347](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/310-347)(): Link speed/width from sysfs (read from `/sys/bus/pci/devices/{bdf}/`)
*   **Chip Limits**[515-586](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/515-586)(): VDD limits, TDP/TDC limits, thermal limits, FMAX

These values are displayed in the Information tab (Tab 1) and Firmware tab (Tab 3), which do not refresh.

**Sources:**[tt_smi/tt_smi_backend.py 84-96](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L84-L96)[tt_smi/tt_smi.py 148-160](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L148-L160)

### Continuous Telemetry Polling

Dynamic telemetry values are refreshed every 100ms when the telemetry tab is active:

**Update Method:**`backend.update_telem()`[258-262](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/258-262)()

`def update_telem(self):    """Update telemetry in a given interval"""    for i, _ in enumerate(self.devices):        self.smbus_telem_info[i] = self.get_smbus_board_info(i)        self.device_telemetrys[i] = self.get_chip_telemetry(i)`
**Dynamic Data Types:**

*   Voltage, current, power
*   AICLK frequency
*   ASIC temperature

## Error Handling and Edge Cases

The pipeline includes handling for common failure modes:

### Missing or Invalid Data

**Telemetry Collection:**

*   **None values:** Handled with conditional checks before conversion [426-428](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/426-428)(), [473-477](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/473-477)()
*   **Default values:** Zero or high sentinel values (e.g., voltage defaults to 10000 if None)
*   **Format strings:** Always produce valid output, showing "0" for missing data

**UI Formatting:**

*   **N/A display:** Gray text for unavailable data [494-497](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/494-497)()
*   **NoMatches exception:** Silently caught when help menu obscures telemetry table [175-176](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/175-176)()

### Remote Devices

Remote devices (Wormhole N300 remote ASICs) have limited data availability:

*   **PCI properties:** Return "N/A" dict [312-313](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/312-313)()
*   **Bus ID:** Caught exception returns "N/A" [388-391](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/388-391)()
*   **PCIe speed/width:** Display "N/A" in gray text [363-370](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/363-370)(), [400-407](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/400-407)()

## Performance Considerations

### Backend Initialization

The initial data collection phase iterates through all devices sequentially with a progress bar:

`# Lines 84-96for i, device in track(    enumerate(self.devices),    total=len(self.devices),    description="Gathering Information",    update_period=0.01,    disable=not self.pretty_output,):`
**Per-device collection time:** Approximately 100-200ms for all six data collection methods

**Total initialization time:**`num_devices * 150ms` (e.g., 4.8 seconds for 32 Galaxy devices)

### Telemetry Thread Performance

**Update cycle:** 100ms per iteration

*   **Hardware reads:** ~10ms per device (pyluwen telemetry read)
*   **Processing:** ~1ms per device (conversion and formatting)
*   **UI update:** ~5ms (Textual table update)

**Total per cycle:** For N devices: `(10ms + 1ms) * N + 5ms`

**Maximum devices before lag:** ~8 devices maintain 100ms cycle without drift
