## Purpose and Scope

This document describes the property and metadata management system in tt-forge-onnx, which tracks execution stages, model properties, compilation configurations, and test results throughout the compilation pipeline. The `ForgePropertyHandler` system provides a centralized mechanism for recording and retrieving metadata without requiring explicit parameter passing through the entire call stack.

For information about the verification system that consumes these properties, see [Verification System](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.4-verification-system). For details on compiler configuration, see [Compilation Configuration](https://deepwiki.com/tenstorrent/tt-forge-onnx/8.2-compilation-configuration).

## System Overview

The property management system serves three primary functions:

1.   **Execution Tracking**: Records progress through compilation stages and identifies failure points
2.   **Metadata Collection**: Captures model characteristics, configurations, and operational parameters
3.   **Result Recording**: Stores numerical comparison metrics (PCC, ATOL, RTOL) and tensor descriptors

The system is designed around a context variable pattern, allowing property recording from any point in the codebase without threading handler objects through function calls.

### ForgePropertyHandler Methods

The handler provides a dot-separated key API for nested property access:

| Method | Parameters | Purpose |
| --- | --- | --- |
| `add(key, value)` | Dot-separated key path, value | Sets nested property (creates intermediate objects) |
| `get(key)` | Dot-separated key path | Retrieves nested property (returns None if missing) |
| `record_execution(stage)` | `ExecutionStage` enum | Records both stage and depth |
| `record_execution_stage(stage)` | `ExecutionStage` enum | Records fine-grained stage |
| `record_execution_depth(depth)` | `ExecutionDepth` enum | Records high-level categorization |
| `record_execution_run_mode(mode)` | `ExecutionRunMode` enum | Records inference/training mode |
| `record_execution_pass(pass)` | `ExecutionPass` enum | Records forward/backward pass |
| `record_emitc_status(success)` | bool | Records EmitC verification result |
| `to_dict()` | - | Converts store to dictionary |
| `clean_store()` | - | Removes empty values (except compiler/verify) |

### Execution Metadata

Execution metadata tracks the compilation/runtime state:

| Property Path | Type | Purpose |
| --- | --- | --- |
| `tags.execution_stage` | str (ExecutionStage name) | Fine-grained stage (20 stages) |
| `tags.bringup_status` | str (ExecutionDepth name) | High-level status (6 categories) |
| `tags.run_mode` | str | "inference" or "training" |
| `tags.execution_pass` | str | "FORWARD" or "BACKWARD" |
| `tags.emitc_status` | bool | EmitC verification success |
| `tags.pcc` | float | Pearson Correlation Coefficient |
| `tags.atol` | float | Absolute tolerance |
| `tags.rtol` | float | Relative tolerance |

**Recording Points**:

*   `compile.py` records execution stages at each compilation phase
*   `tvm_calls/forge_compile.py` records TVM-specific stages
*   `verify/verify.py` records verification pass, consistency limits, and emitc status

### Tensor and Binary Information

The `FlatbufferDetailsExtractor` class parses compiled binary metadata:

**Title**: Flatbuffer Binary Metadata Extraction

**Key Features**:

*   Extracts tensor shapes, data types, buffer types, memory layouts, and grid shapes
*   Handles both device tensors (with `memory_config`) and host tensors (with `storage_type`)

## Integration in Compilation Pipeline

### Stage Recording Flow

**Title**: Property Recording in Compilation Pipeline

At each stage, the code records progressively more advanced `ExecutionStage` values. If compilation fails at any stage, the last recorded stage indicates where the failure occurred.

Sources: [forge/forge/compile.py 182-416](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L182-L416)[forge/forge/tvm_calls/forge_compile.py 514-602](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L514-L602)

### Verification Integration

**Title**: Property Recording in Verification Flow

The verification system records execution passes, stages, and EmitC status. The `record_consistency_limits()` function (called from verify) computes and records PCC/ATOL/RTOL values.

## Property Store Structure

The complete property store hierarchy:

```
ForgePropertyStore
├── owner: str = "tt-forge-onnx"
├── group: str (ModelGroup.GENERALITY | RED)
├── priority: str (ModelPriority.P1 | P2)
├── tags: Tags
│   ├── model_name: str
│   ├── bringup_status: str (ExecutionDepth)
│   ├── execution_stage: str (ExecutionStage)
│   ├── run_mode: str ("inference" | "training")
│   ├── execution_pass: str ("FORWARD" | "BACKWARD")
│   ├── pcc: float
│   ├── atol: float
│   ├── rtol: float
│   ├── op_info: OpInfo
│   │   ├── forge_op_name: str
│   │   ├── args: Dict[str, Any]
│   │   ├── operands: List[Operand]
│   │   └── model_names: List[str]
│   ├── inputs: List[TensorDesc]
│   ├── outputs: List[TensorDesc]
│   ├── model_info: ModelInfo
│   │   ├── framework: str
│   │   ├── model_arch: str
│   │   ├── variant_name: str
│   │   ├── task: str
│   │   └── source: str
│   ├── sweeps: SweepsTags (for operator tests)
│   ├── failure_category: str
│   ├── refined_error_message: str
│   ├── group: str
│   ├── parallelism: str (Parallelism enum)
│   └── emitc_status: bool
└── config: Config
    ├── compiler: Dict[str, Any]
    └── verify: Dict[str, Any]
```

Sources: [forge/forge/forge_property_utils.py 528-559](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L528-L559)
