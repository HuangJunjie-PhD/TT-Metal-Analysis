This page provides practical guidance on structuring compute kernels for the Tensix Coprocessor and efficiently using the MOP (Macro-Op) Expander to reduce RISC-V instruction overhead. It covers the three-phase kernel architecture (UNPACK/MATH/PACK), RISC-V core assignment, MOP Expander templates, and coordination mechanisms between phases.

For instruction-level scheduling and hazard management within each phase, see [Instruction Scheduling and Hazards](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.2-instruction-scheduling-and-hazards). For details on the MOP Expander hardware mechanism, see [Instruction Pipeline and Frontend](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend).

## Kernel Three-Phase Architecture

Tensix compute kernels are structured as three concurrent phases that form a pipeline:

**Diagram: Three-Phase Kernel Pipeline**

Each phase runs on a dedicated RISC-V core and can operate concurrently with the others, with synchronization points enforced through hardware mechanisms. The phases communicate through shared register files (`SrcA`, `SrcB`, `Dst`) rather than through L1, minimizing memory traffic.

## RISC-V Core Assignment

**Diagram: RISC-V Core to Kernel Phase Mapping**

| Core | Primary Role | Tensix Access | Memory | Coordination Capabilities |
| --- | --- | --- | --- | --- |
| **RISCV B** | Data movement, orchestration | Push to all Tensix threads via PCBufs | 4 KiB LRAM | Full control of other cores via `PCBufs`, `Mailboxes` |
| **RISCV T0** | UNPACK kernel execution | Direct push to T0 frontend, can push to T1/T2 via PCBufs | 2 KiB LRAM | `TTSync`, `Mailboxes`, semaphores |
| **RISCV T1** | MATH kernel execution | Direct push to T1 frontend | 2 KiB LRAM (½ KiB I$) | `TTSync`, `Mailboxes`, semaphores |
| **RISCV T2** | PACK kernel execution | Direct push to T2 frontend | 2 KiB LRAM | `TTSync`, `Mailboxes`, semaphores |
| **RISCV NC** | Additional data movement | Debug bus only | 4 KiB LRAM + 16 KiB IRAM | Limited; primarily independent |

## MOP Expander Overview

The MOP Expander enables efficient kernel execution by allowing a single `MOP` instruction to expand into up to 32,639 backend instructions. This addresses a fundamental mismatch: RISC-V cores can push instructions at sub-cycle rates, but the Tensix backend expects one instruction per cycle per thread.

**Diagram: MOP Expander in Instruction Pipeline**

**Key benefits:**

*   **RISC-V efficiency**: A single 32-bit `sw` instruction pushes a `MOP`, which can generate thousands of backend operations
*   **Backend throughput**: Backend receives one instruction per cycle despite RISC-V operating at sub-cycle push rates
*   **Code density**: Kernel code stored in 2 KiB LRAM or L1 cache can drive extensive computation
*   **Latency hiding**: MOP expansion occurs asynchronously, allowing RISC-V to perform other tasks

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 1-5](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L1-L5)[WormholeB0/TensixTile/BabyRISCV/README.md 23-33](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L23-L33)

## MOP Template 0: Masked Emission

Template 0 provides **masked conditional emission** of instructions, allowing selective execution based on a 32-bit mask. This is useful for:

*   Skipping computation on zero tiles or invalid data
*   Handling boundary conditions in tensor operations
*   Conditional unpacking/packing based on data characteristics

**Diagram: Template 0 Execution Flow**

### Configuration Registers (Template 0)

| Register | Purpose | Usage |
| --- | --- | --- |
| `MopCfg[1]` | `Flags` (bits 1:0) | `Flags & 1`: emit `InsnB` `Flags & 2`: emit `InsnA1`, `InsnA2`, `InsnA3` |
| `MopCfg[2]` | `InsnB` | Instruction emitted when mask bit is 0 (if `HasB`) |
| `MopCfg[3]` | `InsnA0` | Always emitted when mask bit is 0 |
| `MopCfg[4]` | `InsnA1` | Emitted when mask bit is 0 (if `HasA123`) |
| `MopCfg[5]` | `InsnA2` | Emitted when mask bit is 0 (if `HasA123`) |
| `MopCfg[6]` | `InsnA3` | Emitted when mask bit is 0 (if `HasA123`) |
| `MopCfg[7]` | `SkipA0` | Emitted when mask bit is 1 |
| `MopCfg[8]` | `SkipB` | Emitted when mask bit is 1 (if `HasB`) |

## MOP Template 1: Nested Loops

Template 1 provides **nested loop structures** for repetitive operations. This is the most commonly used template for regular tensor operations where the same sequence of instructions repeats with different addressing.

**Diagram: Template 1 Nested Loop Structure**

### Configuration Registers (Template 1)

| Register | Purpose | Typical Usage |
| --- | --- | --- |
| `MopCfg[0]` | `OuterCount` (bits 6:0) | Number of outer loop iterations (1-128) |
| `MopCfg[1]` | `InnerCount` (bits 6:0) | Number of inner loop iterations (0-127) Doubled if `LoopOp1` is not `NOP` |
| `MopCfg[2]` | `StartOp` | Emitted once at start of each outer iteration Use for initialization (e.g., `SETRWC`) |
| `MopCfg[3]` | `EndOp0` | Emitted once at end of each outer iteration Use for finalization (e.g., `PACR`) |
| `MopCfg[4]` | `EndOp1` | Second end operation (only if `EndOp0` is not `NOP`) |
| `MopCfg[5]` | `LoopOp` | Primary inner loop body instruction Typically the compute instruction (e.g., `MVMUL`) |
| `MopCfg[6]` | `LoopOp1` | Alternate with `LoopOp` if not `NOP` Use for interleaving operations |
| `MopCfg[7]` | `Loop0Last` | Overrides last inner iteration (last outer loop) |
| `MopCfg[8]` | `Loop1Last` | Overrides last inner iteration (not last outer loop) |

### Typical Usage: Matrix Multiply Tile Loop

### LoopOp Alternation

When `MopCfg[6]` (`LoopOp1`) is not a `NOP`, the inner loop alternates between `LoopOp` and `LoopOp1`:

```
Inner iteration 0: LoopOp
Inner iteration 1: LoopOp1
Inner iteration 2: LoopOp
Inner iteration 3: LoopOp1
...
```

Additionally, `InnerCount` is effectively doubled. This is useful for interleaving operations (e.g., alternating between `UNPACR` instructions for two unpackers).

## MOP Configuration Management

### Configuration Address Mapping

Each Tensix thread's `MopCfg` array is mapped into its corresponding RISC-V core's address space:

| RISC-V Core | Configures | Address | Access |
| --- | --- | --- | --- |
| RISCV T0 | T0 MOP Expander | `TENSIX_MOP_CFG_BASE` (0xFFB8_0000) | Write-only, 9×32-bit words |
| RISCV T1 | T1 MOP Expander | `TENSIX_MOP_CFG_BASE` (0xFFB8_0000) | Write-only, 9×32-bit words |
| RISCV T2 | T2 MOP Expander | `TENSIX_MOP_CFG_BASE` (0xFFB8_0000) | Write-only, 9×32-bit words |

### Configuration Timing Constraints

**Critical rule:** Do not modify `MopCfg` while a MOP expansion is in progress. The MOP Expander samples configuration values during expansion, not just at the start.

## Combining MOP and REPLAY

The MOP Expander sits before the Replay Expander in the frontend pipeline. A typical pattern is:

1.   MOP expands to multiple instructions, including one or more `REPLAY` instructions
2.   Each `REPLAY` instruction expands to its own sequence

**Diagram: MOP and REPLAY Combination**

### Example: Matrix Operation with REPLAY

This pattern is efficient because:

*   RISC-V pushes only ~10 instructions total (REPLAY loads + MOP_CFG + MOP)
*   Backend receives 128 instructions at 1 per cycle
*   Hides the one-cycle transition penalty after MOP expansion

## Best Practices

### 1. Kernel Structure Guidelines

### 2. MOP Configuration Best Practices

| Guideline | Rationale |
| --- | --- |
| Use Template 1 for regular loops | Most operations have regular structure (tiles, rows) |
| Use Template 0 for sparse/conditional | Mask-based execution for irregular patterns |
| Combine MOP with REPLAY | Amortizes MOP transition penalty |
| Always use TTSync before reconfiguring | Prevents races with in-flight expansions |
| Prefer larger InnerCount | Reduces per-tile overhead (StartOp/EndOp) |
| Use Loop0Last/Loop1Last for edge cases | Clean handling of last iteration differences |

### 3. Synchronization Best Practices

| Mechanism | When to Use |
| --- | --- |
| `PCBufs` | RISCV B controlling T0/T1/T2 start/stop |
| `Mailboxes` | Any RISC-V signaling to any other RISC-V |
| `TTSync` | Within Ti, wait for MOP/REPLAY/backend completion |
| Semaphores | Cross-phase handoff (UNPACK→MATH→PACK) |
| `CLEARDVALID` | Transfer SrcA/SrcB ownership to Matrix Unit |
| `STALLWAIT` | Backend instruction waiting on specific conditions |

### 4. Performance Optimization

**Diagram: Configuration Batching**

**Key optimization strategies:**

1.   **Maximize expansion ratio:** Configure MOP to expand to as many instructions as possible (up to 32,639)
2.   **Minimize reconfiguration:** Set up MOP once per kernel phase, not per tile
3.   **Use REPLAY for instruction variety:** MOP provides looping, REPLAY provides the actual operations
4.   **Pipeline the three phases:** Ensure UNPACK, MATH, and PACK run concurrently on different data
5.   **Hide RISC-V latency:** While MOP/REPLAY expand, RISC-V can perform other work (address calculation, synchronization)

**Summary:** Efficient Tensix programming requires structuring kernels into UNPACK/MATH/PACK phases, leveraging the MOP Expander to reduce RISC-V instruction overhead (1 push → 1000s of operations), and coordinating phases through semaphores, PCBufs, and register file ownership mechanisms. Template 1 (nested loops) handles most regular tensor operations, while Template 0 (masked emission) addresses conditional execution. Combining MOP with REPLAY maximizes backend instruction throughput while freeing RISC-V cores for orchestration tasks.
