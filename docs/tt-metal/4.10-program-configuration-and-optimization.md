This page documents the configuration and optimization mechanisms available for TTNN operations and model implementations, focusing on how programs are tuned for performance and accuracy, how precision is controlled, and how runtime configurations are managed using graph-based analysis.

* * *

## Overview

TTNN operations support extensive configuration to balance accuracy, performance, and hardware resource utilization. The configuration system enables:

*   **Precision Control**: Selecting data types (e.g., `BFLOAT16`, `BFP8`, `BFP4`), accumulation modes (e.g., FP32 accumulation), and math fidelity levels per operator group.
*   **Program Caching**: Hashing operation attributes and tensor arguments to reuse compiled programs, significantly reducing host-side overhead [ttnn/api/ttnn/device_operation.hpp 58-69](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L58-L69)
*   **Graph-Based Optimization**: Capturing operation traces to analyze memory footprint, peak L1 usage, and resource constraints via the `GraphProcessor`[ttnn/core/graph/graph_processor.cpp 160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/graph/graph_processor.cpp#L160-L160)
*   **Layout Management**: Managing transitions between `ROW_MAJOR` and `TILE` layouts, and optimizing sharding strategies [ttnn/api/ttnn/tensor/tensor_utils.hpp 35-37](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor_utils.hpp#L35-L37)

* * *

## Program Caching and Hashing

To optimize execution, TTNN employs a program cache. Before an operation is executed, a hash is computed based on its `operation_attributes` and `tensor_args`[ttnn/api/ttnn/device_operation.hpp 58-69](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L58-L69)

*   **Default Hashing**: Uses `ttsl::hash::hash_objects_with_default_seed` to combine the operation type and its parameters [ttnn/api/ttnn/device_operation.hpp 66-68](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L66-L68)
*   **Custom Hashing**: Operations can implement `compute_program_hash` to provide specialized caching logic, particularly useful when certain attributes do not affect the generated kernel binary [ttnn/api/ttnn/device_operation.hpp 61-63](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L61-L63)

### Trace Profiling

The system integrates with Tracy for profiling. The `TracyOpTTNNDevice` macro captures operation IDs, program pointers, and cache hit/miss status, allowing developers to identify redundant compilations [ttnn/api/tools/profiler/op_profiler.hpp 168-176](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/tools/profiler/op_profiler.hpp#L168-L176)

## Graph Tracing and Resource Analysis

TTNN includes a `GraphProcessor` and `GraphTrace` system to analyze program behavior. This is critical for determining if a specific configuration will fit within hardware limits (e.g., L1 memory) before actual execution.

### Resource Usage Analysis

The `query_op_constraints` function dry-runs operations using a `MockDevice` to extract peak resource usage per core [ttnn/api/ttnn/graph/graph_query_op_constraints.hpp 129-152](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/graph/graph_query_op_constraints.hpp#L129-L152) It collects metrics such as `cb_peak_size_per_core` (Circular Buffer usage) and `peak_memory_usage_per_core`[ttnn/api/ttnn/graph/graph_query_op_constraints.hpp 129-134](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/graph/graph_query_op_constraints.hpp#L129-L134)

### Graph Capture Modes

The `GraphProcessor` supports different `RunMode` settings:

*   `NORMAL`: Standard execution with tracing enabled.
*   `NO_DISPATCH`: Traces the graph without actually dispatching kernels to the hardware, allowing for fast resource estimation [ttnn/core/graph/graph_processor.cpp 160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/graph/graph_processor.cpp#L160-L160)

## Layout and Data Movement Optimization

Optimizing the layout of tensors is critical for minimizing the overhead of "tilization" (converting Row-Major to Tile layout) and "untilization".

### Sharded Tensor Optimization

The utility `get_optimal_worker_cores_for_sharded_tensor` determines the best core grid for a sharded tensor to minimize NOC (Network-on-Chip) congestion [ttnn/api/ttnn/tensor/tensor_utils.hpp 35-36](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor_utils.hpp#L35-L36)

### Automated CB Configuration

For sharded tensors, `cb_descriptor_from_sharded_tensor` automatically derives `CBDescriptor` fields (page size, core ranges, and buffer addresses), reducing the manual boilerplate required to set up circular buffers for complex sharding patterns [ttnn/api/ttnn/tensor/tensor_utils.hpp 72-77](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor_utils.hpp#L72-L77)

## Generic Operation Configuration

The `generic_op` API allows for high-level program configuration using `ProgramDescriptor`. This structure bundles kernels, semaphores, and circular buffers into a single execution unit [tests/ttnn/unit_tests/gtests/test_generic_op.cpp 134-140](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/gtests/test_generic_op.cpp#L134-L140)

### Kernel Customization

Developers can optimize generic operations by providing `CompileTimeArgs` and `RuntimeArgs` per core [tests/ttnn/unit_tests/gtests/test_generic_op.cpp 106-123](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/gtests/test_generic_op.cpp#L106-L123)

*   **Compile-Time Args**: Used for hardware-specific optimizations like loop unrolling or static buffer addressing.
*   **Runtime Args**: Used for dynamic parameters like buffer addresses that change between operation calls [tests/ttnn/unit_tests/gtests/test_generic_op.cpp 119-123](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/gtests/test_generic_op.cpp#L119-L123)
