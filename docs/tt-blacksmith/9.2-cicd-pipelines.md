## Purpose and Scope

This document describes the GitHub Actions-based CI/CD infrastructure that automates testing, validation, and dependency management for TT-Blacksmith. The pipeline system orchestrates test execution across multiple hardware configurations (N150, N300, Galaxy), manages nightly dependency uplifts, and enforces code quality checks on pull requests.

For information about the test framework itself (test definitions, pytest markers, golden file validation), see [Test Framework](https://deepwiki.com/tenstorrent/tt-blacksmith/9.1-test-framework). For details on the test cases and their configuration, refer to [Test Framework](https://deepwiki.com/tenstorrent/tt-blacksmith/9.1-test-framework).

## CI/CD Architecture Overview

The CI/CD system follows a modular, reusable workflow pattern with three layers:

**Sources:**[.github/workflows/pr-main.yml 1-48](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L1-L48)[.github/workflows/push-main.yml 1-26](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/push-main.yml#L1-L26)[.github/workflows/manual-test.yml 1-34](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/manual-test.yml#L1-L34)[.github/workflows/call-test.yml 1-78](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/call-test.yml#L1-L78)

The architecture separates concerns into trigger workflows (when to run), orchestration workflows (what to run), and test presets (which tests to run). All test execution flows through the reusable `call-test.yml` workflow, which dynamically generates a test matrix and executes tests in parallel across hardware configurations.

## Trigger Workflows

### Pull Request Workflow

The `pr-main.yml` workflow runs on every pull request to the main branch and includes concurrency control to cancel in-progress runs when new commits are pushed.

**Sources:**[.github/workflows/pr-main.yml 1-48](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L1-L48)

Key features:

*   **Concurrency control**: `group: ${{ github.workflow }}-${{ github.ref }}` with `cancel-in-progress: true`[.github/workflows/pr-main.yml 8-10](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L8-L10)
*   **Conditional uplift tests**: Only runs `uplift-test.json` when the PR branch is named `uplift`[.github/workflows/pr-main.yml 18](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L18-L18)
*   **Always-run basic tests**: The `basic-test` job runs with `if: always()`[.github/workflows/pr-main.yml 27](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L27-L27)
*   **Green check validation**: Uses `re-actors/alls-green@release/v1` to ensure all required jobs pass, with `uplift-test` marked as `allowed-skips`[.github/workflows/pr-main.yml 35-47](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L35-L47)

### Scheduled Nightly Uplift

The `schedule-uplift.yml` workflow automates daily updates to the `pjrt-plugin-tt` dependency:

**Sources:**[.github/workflows/schedule-uplift.yml 1-85](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/schedule-uplift.yml#L1-L85)

**Version Detection Logic:**

The workflow queries the Tenstorrent PyPI index and filters for nightly versions matching the pattern `[0-9]+\.[0-9]+\.[0-9]+\.dev[0-9]{14}` (e.g., `1.0.0.dev20240115120000`):

**Auto-merge Process:**

The workflow uses two separate GitHub tokens:

*   `GH_APPROVE_TOKEN`: Approves the PR [.github/workflows/schedule-uplift.yml 73](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/schedule-uplift.yml#L73-L73)
*   `GH_TOKEN`: Enables auto-merge [.github/workflows/schedule-uplift.yml 82](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/schedule-uplift.yml#L82-L82)

This separation is necessary because GitHub requires PR approval from a different account than the one that creates the PR. The PR will only merge after all CI checks (specifically `uplift-test.json`) pass.

## Test Matrix System

### Test Matrix Presets

Test matrix presets are JSON files defining hardware configurations and test markers. Each entry specifies:

| Field | Description |
| --- | --- |
| `runs-on` | Self-hosted runner label (e.g., `n150`, `n300`, `galaxy-wh-6u`) |
| `frontend` | Environment to activate (`xla`, `gpu`, `ffe`) |
| `marks` | Pytest marker expression to filter tests |

**Basic Test Preset** (`basic-test.json`):

**Sources:**[.github/workflows/test-matrix-presets/basic-test.json 1-7](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/basic-test.json#L1-L7)

The basic test preset runs on PR and push events, covering fundamental single-chip and multi-chip scenarios on N150 and N300 hardware.

**Uplift Test Preset** (`uplift-test.json`):

**Sources:**[.github/workflows/test-matrix-presets/uplift-test.json 1-10](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/uplift-test.json#L1-L10)

Key features of the uplift preset:

*   **Split execution**: N150 PyTorch tests are split into `split_0` and `split_1` to parallelize long-running tests [.github/workflows/test-matrix-presets/uplift-test.json 5-6](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/uplift-test.json#L5-L6)
*   **LLM-specific hardware**: Uses `n300-llmbox` runner for data parallel PyTorch tests [.github/workflows/test-matrix-presets/uplift-test.json 7](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/uplift-test.json#L7-L7)
*   **Galaxy testing**: Includes 32-chip Galaxy hardware for large-scale experiments [.github/workflows/test-matrix-presets/uplift-test.json 8](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/uplift-test.json#L8-L8)
*   **Framework coverage**: Tests both JAX and PyTorch experiments

### Pytest Marker Combinations

Common marker expressions used in test matrices:

| Expression | Filters For |
| --- | --- |
| `single_chip and push and n150 and torch` | Single-chip PyTorch tests on N150 for PR validation |
| `data_parallel and uplift and n300 and jax` | Data parallel JAX tests on N300 for uplift validation |
| `galaxy and uplift and torch` | Galaxy-scale PyTorch tests for uplift validation |
| `n150 and torch and split_0` | First half of N150 PyTorch test suite |

For the complete pytest marker system, see [Test Framework](https://deepwiki.com/tenstorrent/tt-blacksmith/9.1-test-framework).

## Workflow Permissions

Different workflows require different permission levels:

**Pull Request Workflow**[.github/workflows/pr-main.yml 1-11](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/pr-main.yml#L1-L11):

*   Inherits default repository permissions
*   No explicit permissions defined

**Push to Main Workflow**[.github/workflows/push-main.yml 7-12](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/push-main.yml#L7-L12):

*   `contents: read` - Read repository contents
*   `checks: write` - Write check run results
*   `pages: write` - Deploy to GitHub Pages
*   `id-token: write` - OIDC token for deployment
*   `deployments: write` - Create deployments

**Scheduled Uplift Workflow**:

*   Uses `GH_TOKEN` secret for PR creation and auto-merge [.github/workflows/schedule-uplift.yml 68-82](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/schedule-uplift.yml#L68-L82)
*   Uses `GH_APPROVE_TOKEN` secret for PR approval [.github/workflows/schedule-uplift.yml 73](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/schedule-uplift.yml#L73-L73)
