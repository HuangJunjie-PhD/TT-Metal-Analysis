## Purpose and Scope

This document covers memory-related test steps used to construct test scenarios in the coretp framework. Memory operations include:

*   **Memory region allocation** (`Memory`, `CodePage`)
*   **Memory access operations** (`Load`, `Store`, `MemAccess`)
*   **Page table manipulation** (`ModifyPte`, `ReadLeafPTE`)

These steps form the foundation for testing memory subsystems, virtual memory, cache operations, and multi-processor synchronization. For CSR-based memory configuration (e.g., `menvcfg`, `satp`), see [CSR Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.2-csr-operations). For assertion-based memory verification, see [Assertions and Verification](https://deepwiki.com/tenstorrent/riscv-coretp/5.3-assertions-and-verification).

## Memory Allocation Steps

### Memory Class

The `Memory` class defines memory regions for test scenarios. It specifies physical and virtual address ranges, page sizes, memory protection flags, and alignment requirements.

**Class Definition:**[coretp/step/memory.py 11-50](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L11-L50)

#### Key Parameters

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `size` | `int` | `0x1000` | Size of memory region in bytes |
| `page_size` | `PageSize` | `SIZE_4K` | Page size (4K, 2M, 1G) |
| `flags` | `PageFlags` | `VALID|READ|WRITE|EXECUTE` | Memory protection flags |
| `exclude_flags` | `Optional[PageFlags]` | `None` | Flags to explicitly exclude |
| `page_cross_en` | `bool` | `False` | Whether page boundary crossing is enabled |
| `alignment` | `Optional[int]` | `None` | Memory alignment requirement |
| `base_pa` | `Optional[int]` | `None` | Requested physical page number (PPN) |
| `base_va` | `Optional[int]` | `None` | Requested virtual page number (VPN) |
| `num_pages` | `Optional[int]` | `1` | Number of pages to allocate |
| `modify` | `bool` | `False` | Whether memory can be modified post-allocation |
| `needs_io` | `bool` | `False` | Whether memory requires IO support |

Memory regions are typically allocated at the beginning of a test scenario, then referenced by subsequent memory access operations:

#### Physical and Virtual Address Control

The `base_pa` and `base_va` parameters allow precise control over memory placement:

*   `base_pa`: Requests a specific physical page number (PPN). Useful for testing device memory regions or specific physical address ranges.
*   `base_va`: Requests a specific virtual page number (VPN). Useful for testing virtual address aliasing or specific VA layouts.

### CodePage Class

The `CodePage` class extends `Memory` to represent executable memory regions containing test instructions. It defines the memory region where code exists but does not automatically branch to it.

**Class Definition:**[coretp/step/memory.py 52-73](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L52-L73)

#### Key Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `code` | `list[TestStep]` | List of test steps (instructions) to place in the code page |

All `Memory` parameters are also available.

**Class Definition:**[coretp/step/load_store/store.py 15-45](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/load_store/store.py#L15-L45)

| `extension` | `Optional[Extension]` | `None` | RISC-V extension required for this operation |

#### Dependency Management

The `deps()` method [coretp/step/load_store/store.py 38-44](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/load_store/store.py#L38-L44) returns dependencies for this step:

*   The `memory` region being written to
*   The `value` being stored (if it's a `TestStep` result)

This ensures proper ordering in the test execution pipeline.

## Page Table Operations

Page table operations manipulate virtual memory structures for testing paging behavior, fault conditions, and PTE (Page Table Entry) attributes.

### ModifyPte Class

The `ModifyPte` class modifies page table entries associated with a memory region. This is used to test scenarios involving modified PTEs, recursive page tables, and specific paging configurations.

**Class Definition:**[coretp/step/memory.py 76-88](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L76-L88)

### ReadLeafPTE Class

The `ReadLeafPTE` class reads the leaf page table entry for a memory region. The implementation uses a riesc (RISC-V escape) mechanism to handle this via machine mode, returning the PTE entry in register `T2`.

**Class Definition:**[coretp/step/memory.py 92-101](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L92-L101)

## Memory Step Parameter Reference

### PageSize Enumeration

Valid `page_size` values for `Memory` and `CodePage`:

| Value | Description |
| --- | --- |
| `PageSize.SIZE_4K` | 4 KiB pages (0x1000 bytes) |
| `PageSize.SIZE_2M` | 2 MiB pages (0x200000 bytes) |
| `PageSize.SIZE_1G` | 1 GiB pages (0x40000000 bytes) |

### PageFlags Bitfield

Valid `flags` values for `Memory` and `CodePage` (can be combined with `|` operator):

| Flag | Description |
| --- | --- |
| `PageFlags.VALID` | Page table entry is valid |
| `PageFlags.READ` | Read permission |
| `PageFlags.WRITE` | Write permission |
| `PageFlags.EXECUTE` | Execute permission |
| `PageFlags.USER` | User-mode accessible |
| `PageFlags.GLOBAL` | Global mapping (not affected by ASID) |
| `PageFlags.ACCESSED` | Accessed bit (A) |
| `PageFlags.DIRTY` | Dirty bit (D) |

## Integration with Other Test Steps

Memory operations integrate with other test step types to form complete test scenarios:

**Integration Points:**

*   **CSR Operations**: Configure memory-related CSRs (`satp`, `menvcfg`, `senvcfg`) before memory access
*   **Assertions**: Verify memory access results and exception conditions
*   **Hart Control**: Coordinate memory operations across multiple harts for cache coherence testing
*   **Arithmetic**: Compute addresses and values for memory operations
