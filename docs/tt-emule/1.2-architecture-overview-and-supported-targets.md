The `tt-emule` software emulator provides a high-fidelity execution environment for Tenstorrent kernels by modeling the specific threading, memory, and synchronization constraints of different hardware generations. It supports three primary architectures: **Wormhole (N150)**, **Blackhole (P100)**, and **Quasar (Q1)**.

## Supported Architectures & Thread Models

The emulator differentiates between architectures primarily through their processor topology and the threading model used to execute JIT-compiled kernels.

| Architecture | Core Type | Threading Model | Memory (L1) | Sync Mechanism |
| --- | --- | --- | --- | --- |
| **Wormhole (N150)** | Tensix | 3-thread (NCR, Comp, NCW) | 1.5 MB | Circular Buffers (CB) |
| **Blackhole (P100)** | Tensix | 3-thread (NCR, Comp, NCW) | 2.0 MB | Circular Buffers (CB) |
| **Quasar (Q1)** | Tensix Neo | 12-thread (8 DM + 4 Comp) | 4.0 MB | Dataflow Buffers (DFB) |

### Wormhole & Blackhole (Tensix)

These architectures utilize a 3-RISC model per core. In `tt-emule`, each core is represented by three host threads mirroring the hardware:

1.   **NOC Reader (NCR):** Handles data movement from DRAM/L1 to the core.
2.   **Compute:** Executes the UNPACK-MATH-PACK pipeline.
3.   **NOC Writer (NCW):** Handles data movement from the core to DRAM/L1.

These threads synchronize via **Circular Buffers (CBs)**, which are Single-Producer Single-Consumer (SPSC) FIFOs [docs/cb-emulation.md 28-29](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/cb-emulation.md?plain=1#L28-L29)

### Quasar (Tensix Neo)

The Quasar "Neo" core represents a significant architectural shift. A single Neo core contains **8 Data Movement (DM)** processors and **4 Compute engines**[docs/QUASAR_EMULATION.md 22-23](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L22-L23)

*   **Threading:** A Neo core is emulated using up to 12 host threads (8 DM + 4 Compute) [docs/QUASAR_EMULATION.md 40-41](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L40-L41)
*   **Shared L1:** All 12 threads share a single 4 MB L1 address space [docs/QUASAR_EMULATION.md 23](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L23-L23)
*   **Dataflow Buffers (DFB):** Replaces CBs with a Multi-Producer Multi-Consumer (MPMC) tile-counting system [docs/DFB_EMULATION.md 11-12](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L11-L12)

## Synchronization Models

### CB-Based Synchronization (WH/BH)

Circular Buffers rely on a simple `occupied` semaphore and a mutex/condition variable pair.

*   **CBSyncState:** Tracks the shared geometry and occupancy [include/tt_emule/cb_sync_state.hpp 59-62](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/cb_sync_state.hpp#L59-L62)
*   **SPSC Pattern:** Only one producer and one consumer are allowed per CB.
*   **Fast Path:** The `cb_reserve_back` and `cb_wait_front` functions check atomics before acquiring a mutex [include/tt_emule/cb_sync_state.hpp 63-66](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/cb_sync_state.hpp#L63-L66)

### DFB-Based Synchronization (Quasar)

Dataflow Buffers (DFBs) utilize **Tile Counters (TC)** to support MPMC access patterns.

*   **TileCounter:** Contains independent `posted` (producer) and `acked` (consumer) atomic counters [include/tt_emule/tile_counter.hpp 24-26](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/tile_counter.hpp#L24-L26)
*   **MPMC Support:** Each consumer in a DFB tracks its own progress via a dedicated TC [docs/DFB_EMULATION.md 13](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L13-L13)
*   **Access Patterns:**
    *   **STRIDED:** Producers and consumers round-robin through specific memory slots [docs/DFB_EMULATION.md 102-104](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L102-L104)
    *   **BLOCKED:** Producers broadcast to all consumers, and consumers round-robin through producer memory blocks [docs/DFB_EMULATION.md 107-109](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L107-L109)

## Hardware Emulation Specifics

### Memory Sizes & Layout

All device memory is owned by `tt_emule::Core` objects inside the `SWEmuleChip`. L1 memory is mmap'd from a central `L1Pool` to ensure host-pointer consistency [docs/l1-emulation.md 18-19](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/l1-emulation.md?plain=1#L18-L19)

| Feature | Wormhole | Blackhole | Quasar |
| --- | --- | --- | --- |
| **L1 Size** | 1.5 MB | 2.0 MB | 4.0 MB |
| **DRAM Banks** | 6 | 8 | 2 |
| **DRAM Layout** | Banked/Interleaved | Banked/Interleaved | Banked/Interleaved |

Sources: [docs/QUASAR_EMULATION.md 22-25](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L22-L25)[docs/QUASAR_EMULATION.md 71-73](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L71-L73)

### CSR & HartID Patching

Because kernels are JIT-compiled as standard C++ for the host, hardware-specific Control and Status Register (CSR) reads must be intercepted. The `emulated_program_runner` uses regex-patching to redirect these to thread-local storage (TLS) [docs/QUASAR_EMULATION.md 43-48](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L43-L48)

*   `mhartid` (RISC-V Hart ID) → `__processor_id`
*   `NEO_ID` → `__emule_neo_id`
*   `TRISC_ID` → `__emule_trisc_id`

This allows the same kernel source code to behave differently depending on which host thread is executing it [docs/QUASAR_EMULATION.md 50-51](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L50-L51)
