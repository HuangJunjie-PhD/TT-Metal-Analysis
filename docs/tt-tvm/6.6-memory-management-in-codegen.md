## Purpose and Scope

This page documents the memory management subsystem within TVM's LLVM code generation backend. It covers how the code generators handle buffer allocation, pointer arithmetic, memory alignment, address spaces, and type-based alias analysis (TBAA) when translating TIR (TensorIR) programs into LLVM IR.

For information about the overall LLVM code generation architecture, see [LLVM Backend Architecture](https://deepwiki.com/tenstorrent/tt-tvm/6.1-llvm-backend-architecture). For details on platform-specific code generation features, see [CPU Code Generation](https://deepwiki.com/tenstorrent/tt-tvm/6.2-cpu-code-generation), [GPU Code Generation](https://deepwiki.com/tenstorrent/tt-tvm/6.3-gpu-code-generation), and [Specialized Target Code Generation](https://deepwiki.com/tenstorrent/tt-tvm/6.4-specialized-target-code-generation).

## Core Data Structures

The LLVM codegen maintains several key data structures to track memory-related information during code generation:

### TypedPointer

The `TypedPointer` struct bridges the gap between TVM's typed pointer semantics and LLVM's opaque pointer model (required for LLVM 15+). It maintains both the pointer value and its pointee type, enabling proper pointer arithmetic and dereferencing.

**Sources:**[src/target/llvm/codegen_llvm.h 253-259](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.h#L253-L259)

### StorageInfo

The `StorageInfo` struct tracks memory alignment requirements for allocated buffers. This information is propagated to LLVM allocation instructions and load/store operations to enable vectorization and satisfy hardware alignment constraints.

**Sources:**[src/target/llvm/codegen_llvm.h 260-264](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.h#L260-L264)[src/target/llvm/codegen_llvm.cc 240-246](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L240-L246)

### Storage Tracking Map

The `alloc_storage_info_` member variable maps buffer variables to their `StorageInfo`, maintaining alignment metadata throughout code generation:

| Map Key | Map Value | Purpose |

| `const VarNode*` | `StorageInfo` | Buffer variable to alignment mapping |

## Buffer Access and Pointer Management

### CreateBufferPtr Function

The `CreateBufferPtr` function is the central mechanism for translating TIR buffer accesses into LLVM pointer arithmetic. It handles address space mapping, type conversion, and index computation.

The function performs these operations:

1.   Converts buffer element type and value type to LLVM types
2.   Determines address space from buffer pointer type
3.   Casts buffer pointer to element pointer type if needed
4.   Performs bounds-checked pointer arithmetic using `CreateInBoundsGEP`
5.   Casts to target value type if element type differs
6.   Returns `TypedPointer` with type and address

**Sources:**[src/target/llvm/codegen_llvm.cc 961-994](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L961-L994)

### Address Space Handling

Different memory regions use distinct address spaces in LLVM IR. The base implementation uses address space 0 (global), but GPU backends override `GetGlobalAddressSpace()`:

| Platform | Address Space | Memory Type |
| --- | --- | --- |
| CPU | 0 | Global/system memory |
| CUDA/NVPTX | 0 | Global device memory |
| CUDA/NVPTX | 3 | Shared memory |
| ROCm/AMDGPU | 1 | Global device memory |
| ROCm/AMDGPU | 3 | Shared memory |
| Hexagon | 0 | Default |

**Sources:**[src/target/llvm/codegen_llvm.cc 564](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L564-L564)[src/target/llvm/codegen_nvptx.cc 99](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L99-L99)[src/target/llvm/codegen_amdgpu.cc 217](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L217-L217)

## Memory Alignment

### Alignment Computation

The `GetAlignment` function computes the alignment of buffer accesses using modular arithmetic analysis. This information is critical for generating efficient vectorized loads/stores.

The algorithm:

1.   Looks up maximum alignment from `alloc_storage_info_`
2.   Uses `arith::Analyzer` to compute modular set `{base, coeff}` of index expression
3.   Iteratively divides by 2 while both base and coefficient are even
4.   Returns alignment in bytes (minimum 8 bits = 1 byte)

**Sources:**[src/target/llvm/codegen_llvm.cc 691-718](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L691-L718)

## Memory Allocation

### Stack Allocation

For local memory on CPU and GPU, the codegen uses LLVM's `alloca` instruction to allocate stack memory in the function entry block:

**CPU Implementation:**

*   Allocates on default address space
*   Sets alignment from `StorageInfo`
*   Always allocates at function entry for SSA form

**GPU Implementations (CUDA/ROCm):**

*   Uses address space 0 (function-local)
*   Applies maximum alignment of 16 bytes on NVPTX
*   Applies maximum alignment of 16 bytes on AMDGPU

**Sources:**[src/target/llvm/codegen_nvptx.cc 87-140](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L87-L140)[src/target/llvm/codegen_amdgpu.cc 102-156](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L102-L156)

### Shared Memory Allocation

GPU shared memory is allocated as module-level global variables in address space 3:

**Dynamic Shared Memory:**

*   Uses linkage `ExternalLinkage` with size 0
*   Tagged with storage scope `.dyn`
*   Name becomes external symbol for runtime allocation

**Static Shared Memory:**

*   Uses linkage `PrivateLinkage` with fixed size
*   Allocated at compile time

## Type-Based Alias Analysis (TBAA)

### AddAliasInfo Function

The `AddAliasInfo` function attaches TBAA metadata to load and store instructions, enabling LLVM's optimizer to reason about pointer aliasing and perform aggressive optimizations:

**TBAA Metadata Hierarchy:**

The metadata forms a tree representing memory regions:

*   Root: `md_tbaa_root_` ("tvm-tbaa")
*   Buffer level: One node per buffer variable (by pointer address)
*   Range level: Hierarchical nodes for power-of-2 aligned ranges (e.g., `.w1024.b0`, `.w512.b0`)

This allows LLVM to determine that accesses to different aligned regions don't alias.

**Sources:**[src/target/llvm/codegen_llvm.cc 636-689](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L636-L689)[src/target/llvm/codegen_llvm.cc 164-165](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L164-L165)

## Platform-Specific Memory Management

### CPU Memory Model

CPU codegen uses a simple memory model with a single address space and stack-based allocation for temporary buffers.

**Sources:**[src/target/llvm/codegen_cpu.cc 364-456](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L364-L456)

### CUDA/NVPTX Memory Model

**Key Characteristics:**

*   Address space 3 for shared memory (`__shared__` in CUDA)
*   Maximum alignment of 16 bytes enforced
*   Dynamic shared memory uses external linkage

**Sources:**[src/target/llvm/codegen_nvptx.cc 87-140](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L87-L140)

### ROCm/AMDGPU Memory Model

**Key Differences from NVPTX:**

*   Address space 1 for global (not 0)
*   `GetGlobalAddressSpace()` returns 1
*   Dynamic shared memory is experimental with warning

**Sources:**[src/target/llvm/codegen_amdgpu.cc 102-156](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L102-L156)[src/target/llvm/codegen_amdgpu.cc 217](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L217-L217)

### Hexagon Memory Model

The Hexagon backend includes unique support for 2D buffer indexing:

This enables efficient access to buffers partitioned across multiple memory regions (e.g., HVX vector memory).
