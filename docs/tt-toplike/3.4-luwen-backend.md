# Luwen Backend

Relevant source files
*   [src/backend/luwen.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs)

The Luwen backend provides high-performance, direct hardware access to Tenstorrent devices by utilizing the official `luwen` library. Unlike other backends that rely on Linux sysfs or external process parsing, the Luwen backend communicates directly with hardware via PCIe memory-mapped I/O (MMIO).

## Architecture and Design

The Luwen backend is designed for environments where the lowest latency and highest telemetry sampling frequency are required. It leverages the `luwen-if` and `luwen-ref` crates to handle low-level PCIe transactions and chip-specific communication protocols.

### Key Capabilities

*   **Direct PCIe BAR0 Access**: Bypasses the overhead of the Linux kernel's `hwmon` subsystem and external CLI tools [src/backend/luwen.rs 7-15](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L7-L15)
*   **Multi-Architecture Support**: Native support for Grayskull, Wormhole, and Blackhole architectures [src/backend/luwen.rs 16](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L16-L16)
*   **Non-Invasive Monitoring**: Uses `noc_safe` mode by default to ensure telemetry reads do not interfere with active workloads running on the Network-on-Chip (NoC) [src/backend/luwen.rs 74-77](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L74-L77)

## Implementation Details

### Device Discovery and Initialization

The initialization process is handled in `detect_devices`. It specifically requests `local_only` and `noc_safe` access to prevent accidental interference with remote chips or active compute kernels [src/backend/luwen.rs 71-80](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L71-L80)

The backend maps `all_smi_luwen_core::Arch` variants to the internal `Architecture` enum:

*   `Arch::Grayskull` → `Architecture::Grayskull`[src/backend/luwen.rs 102](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L102-L102)
*   `Arch::Wormhole` → `Architecture::Wormhole`[src/backend/luwen.rs 103](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L103-L103)
*   `Arch::Blackhole` → `Architecture::Blackhole`[src/backend/luwen.rs 104](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L104-L104)

### Telemetry Update Loop

During the `update()` cycle, the backend iterates through its stored `Chip` handles and executes `get_telemetry()`. This call performs direct MMIO reads from the device's ARC (Argon RISC Core) power management registers [src/backend/luwen.rs 166-174](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L166-L174)

## Safety and Invasiveness

The Luwen backend is inherently more invasive than the Sysfs backend because it requires direct access to PCIe device memory (`/dev/mem` or specific `/dev/tenstorrent/` nodes).

### Exclusion from Auto-Detection

The Luwen backend is **excluded** from the automatic backend detection chain. This is a deliberate design choice to prevent `tt-toplike` from attempting direct hardware access unless explicitly requested by the user via `--backend luwen`. This prevents:

1.   Permissions errors (Luwen usually requires `root` or `plugdev` group access).
2.   Potential conflicts with other tools using the `luwen` library or `tt-smi`.

### Panic-Catch Wrapping

Direct hardware access carries a risk of hardware hangs or illegal register access if the device is in a bad state. While not shown in the truncated source, the `update` loop is designed to be resilient, wrapping hardware calls to ensure that a single failing chip does not crash the entire TUI/GUI application [src/backend/luwen.rs 78](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L78-L78)

## Feature Flagging

The Luwen backend is optional and gated behind the `luwen-backend` feature flag. If the crate is compiled without this feature, the `detect_devices` function will return an error explaining that a rebuild is required [src/backend/luwen.rs 146-151](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/luwen.rs#L146-L151)

| Feature Flag | Effect |
| --- | --- |
| `luwen-backend` | Enables `all_smi_luwen_if` dependencies and `LuwenBackend` implementation. |
| (default) | Luwen backend is disabled to minimize binary size and dependency tree. |
