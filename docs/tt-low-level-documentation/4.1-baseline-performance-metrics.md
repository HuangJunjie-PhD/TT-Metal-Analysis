## Purpose and Scope

This document presents the baseline performance characteristics for all fundamental data movement primitives on Tenstorrent hardware platforms. These metrics represent the ideal, maximum achievable throughput for each data movement pattern when the NOC is operating under optimal conditions with no contention or congestion.

The baseline metrics serve as:

*   **Reference values** for evaluating real-world kernel performance
*   **Upper bounds** for throughput expectations in application code
*   **Validation targets** for regression testing and hardware verification

For detailed analysis of multicast performance variations based on sender placement and grid configuration, see [Multicast Schemes Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study). For virtual channel performance characteristics, see [Virtual Channels Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.3-virtual-channels-performance-study).

| DRAM Read | 22 | 33 |

| One To All (Multicast) | 15 | 24 |

**Note:** "All" refers to 64 Tensix cores on Wormhole B0 and 110 Tensix cores on Blackhole P100.

## Primitive Categories and Definitions

### DRAM Operations

**DRAM Read/Write Primitives**

*   **DRAM Read:** Single Tensix core reads from DRAM bank to local L1 memory
*   **DRAM Write:** Single Tensix core writes from local L1 memory to DRAM bank
*   These represent the fundamental memory hierarchy traversal for off-core storage
*   Performance limited by DRAM controller bandwidth and NOC interface width

### Core-to-Core Operations

**Point-to-Point Communication**

*   **One To One:** Single sender performs NOC write to single remote receiver
*   **One From One:** Single receiver performs NOC read from single remote sender
*   Represent the simplest, most efficient core-to-core transfer patterns
*   Achieves near-maximum interface bandwidth utilization

**Broadcast Patterns**

*   **One To All (Unicast):** Sender issues separate write transaction to each of N receivers
*   **One To All (Multicast):** Sender issues single multicast packet that hardware replicates to N receivers
*   **One To All (Multicast + Linked):** Multicast packet with linked write operations for improved throughput
*   Multicast variants reduce sender-side overhead but may experience NOC congestion

**Aggregation Pattern**

*   **One From All:** N senders each write to a single receiver's L1 buffer
*   Commonly used for reduction operations and result collection
*   Performance depends on receiver buffer size and write interleaving

### Performance Scaling Analysis

The Blackhole platform demonstrates consistent 2x scaling for point-to-point operations:

| Primitive | WH B0 | BH P100 | Scaling Factor |
| --- | --- | --- | --- |
| One To One | 29 | 60 | 2.07x |
| One From One | 28 | 60 | 2.14x |
| One To All (UC) | 31 | 62 | 2.00x |
| One From All | 30 | 60 | 2.00x |

**Observations:**

1.   Point-to-point operations scale linearly with interface width (2x improvement)
2.   DRAM operations show sub-linear scaling (1.5-1.6x) due to memory controller bottlenecks
3.   Multicast operations show improved but less-than-2x scaling due to NOC congestion patterns

## Testing Methodology

### Test Framework Architecture

### Test Execution Process

Each baseline metric is derived from directed ideal tests that:

1.   **Isolate Single Primitive:** Test focuses on one data movement pattern in isolation
2.   **Maximize Throughput:** Configuration parameters chosen to achieve peak NOC utilization
3.   **Minimize Overhead:** Eliminate synchronization barriers and CPU overhead from measurement
4.   **Measure Sustained Performance:** Execute sufficient iterations to reach steady-state throughput

## Interpreting Baseline Metrics

### Relationship to Interface Width

The theoretical maximum NOC throughput is determined by the interface width:

*   **Wormhole B0:** 32 bytes/cycle per NOC instance
*   **Blackhole P100:** 64 bytes/cycle per NOC instance

Comparing measured baseline to theoretical maximum reveals efficiency:

| Primitive (WH B0) | Measured | Theoretical Max | Efficiency |
| --- | --- | --- | --- |
| One To One | 29 | 32 | 90.6% |
| One To All (Unicast) | 31 | 32 | 96.9% |
| DRAM Read | 22 | 32 | 68.8% |
| One To All (Multicast) | 15 | 32 | 46.9% |

**Key Insights:**

1.   **Point-to-point operations** achieve >90% of theoretical maximum
2.   **DRAM operations** are bottlenecked by memory controllers, not NOC
3.   **Multicast operations** show lower efficiency due to packet replication overhead and self-interference (detailed in [4.2](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study))

### Using Baselines for Performance Evaluation

When profiling application kernels:

1.   **Compare to Baseline:** Calculate `actual_throughput / baseline_throughput` ratio

2.   **Expected Ranges:**

    *   80-100%: Excellent performance, minimal overhead
    *   60-80%: Good performance, some NOC contention or synchronization overhead
    *   40-60%: Moderate performance, investigate batching and barrier placement
    *   <40%: Poor performance, likely excessive barriers or suboptimal data movement patterns

3.   **Common Degradation Sources:**

    *   Barrier placement after each transaction (see [3.2](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.2-asynchronous-operations-and-batching))
    *   NOC instance contention from multiple concurrent operations
    *   Self-interference in multicast patterns (see [4.2](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study))
    *   Virtual channel misuse (see [4.3](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.3-virtual-channels-performance-study))

## Multicast Performance Considerations

The baseline multicast metrics represent **optimal placement** scenarios. In practice, multicast performance varies significantly based on:

*   Sender location relative to receiver grid
*   NOC instance selection (NOC0 vs NOC1)
*   Loopback configuration
*   Grid dimensions and aspect ratio

**Example Performance Range (Wormhole B0):**

*   Best case: 22 B/cycle (Multicast + Linked, optimal placement)
*   Worst case: <5 B/cycle (Self-interference scenario)

For comprehensive analysis of multicast placement strategies and self-interference mechanisms, see [Multicast Schemes Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study).

## Virtual Channel Impact

The baseline metrics assume **default virtual channel usage** where each operation uses a single VC appropriate for its traffic class.

**Important:** Splitting the same traffic class across multiple virtual channels **does not improve** these baseline metrics. Virtual channels are designed to separate different traffic classes, not parallelize identical traffic.

For detailed explanation of VC arbitration behavior and proper usage patterns, see [Virtual Channels Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.3-virtual-channels-performance-study) and [Virtual Channels Usage](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.4-virtual-channels-usage).

## Summary

The baseline performance metrics provide:

1.   **Reference throughput values** for all fundamental data movement primitives
2.   **Platform comparison data** showing 2x scaling from Wormhole B0 to Blackhole P100
3.   **Validation targets** for regression testing and performance analysis
4.   **Foundation for advanced studies** of multicast schemes and virtual channel behavior

These metrics represent ideal conditions. Real-world kernel performance depends on:

*   API usage patterns (batching, barriers, stateful operations)
*   NOC contention from concurrent operations
*   Memory access patterns and buffer management
*   Synchronization overhead

For guidance on achieving performance close to these baselines in application code, see [API Usage Guide](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3-api-usage-guide) and [Best Practices](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.1-core-api-concepts).
