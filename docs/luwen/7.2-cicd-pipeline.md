This page documents the Continuous Integration and Continuous Deployment (CI/CD) pipeline for the Luwen project. It covers the pipeline configuration, build process, testing approach, deployment mechanisms, and local development tools that ensure code quality.

For details on building and testing the codebase locally, see [Building and Testing](https://deepwiki.com/tenstorrent/luwen/7.1-building-and-testing).

## Pipeline Overview

The Luwen project utilizes GitLab CI/CD for automating the build, test, and deployment processes. The pipeline is designed to ensure that code changes meet quality standards and can be safely deployed to production.

Sources: [.gitlab-ci.yml 17-20](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L17-L20)[.pre-commit-config.yaml 3-19](https://github.com/tenstorrent/luwen/blob/55e35616/.pre-commit-config.yaml#L3-L19)

## Pipeline Stages

The CI/CD pipeline is organized into three primary stages:

| Stage | Purpose | Key Jobs |
| --- | --- | --- |
| Build | Compile the codebase | build:stable:release, build:stable:debug, build:nightly |
| Test | Execute test suite | test:cargo:gs, test:cargo:wh |
| Deploy | Package and deploy | deploy, deploy-cross |

Sources: [.gitlab-ci.yml 17-20](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L17-L20)

## Build Process

The build stage compiles the Luwen codebase using different Rust toolchains and configurations.

### Base Build Configuration

The `.build-base` job template defines common settings for all build jobs:

*   Uses a custom Docker image: `$CI_REGISTRY/syseng-platform/luwen/rust-ci-build:latest`
*   Runs on runners with the `8-core` tag
*   Sets a timeout of 1 hour
*   Installs required dependencies (protobuf-compiler)
*   Builds the project using `cargo build`
*   Preserves build artifacts for use in subsequent stages

### Build Variants

The pipeline includes three build variants:

1.   **build:stable:release**: Builds the project with stable Rust in release mode
2.   **build:stable:debug**: Builds the project with stable Rust in debug mode
3.   **build:nightly**: Builds the project with nightly Rust in release mode (allowed to fail)

Sources: [.gitlab-ci.yml 44-78](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L44-L78)

## Testing Strategy

The test stage verifies that the codebase functions correctly on different hardware platforms.

### Test Base Configuration

All test jobs extend from `.cargo_test_setup`, which:

*   Inherits from `.build-base`
*   Depends on the successful completion of `build:stable:release`
*   Runs cargo test across the workspace

### Hardware-Specific Tests

The pipeline includes two hardware-specific test jobs:

1.   **test:cargo:gs**: Runs tests on machines with Grayskull cards
2.   **test:cargo:wh**: Runs tests on machines with Wormhole (Nebula x2) cards

This ensures that the codebase works correctly with different Tenstorrent hardware architectures.

Sources: [.gitlab-ci.yml 81-98](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L81-L98)

## Deployment Process

The deployment stage packages the application and prepares it for distribution.

### Debian Package Creation

The `deploy` job:

*   Runs on runners with the `8-core` tag
*   Installs `cargo-deb` for generating Debian packages
*   Executes `make deb` to create a Debian package
*   Deploys to the production environment

### RISC-V Cross-Compilation Support

The `deploy-cross` job:

*   Is manually triggered
*   Sets up Docker-in-Docker
*   Clones the cross-compilation tools repository
*   Builds and pushes a Docker image for RISC-V cross-compilation

Sources: [.gitlab-ci.yml 110-136](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L110-L136)

## Pre-Commit Hooks

In addition to the CI/CD pipeline, the Luwen project uses pre-commit hooks to catch issues before code is committed:

The pre-commit hooks include:

1.   **Code Style Checks**:

    *   Trailing whitespace removal
    *   End-of-file fixing

2.   **File Validation**:

    *   YAML validation
    *   Large file checking

3.   **Rust-specific Checks**:

    *   Code formatting (`cargo fmt`)
    *   Compilation checking (`cargo check`)
    *   Linting with Clippy (`cargo clippy`)

These hooks ensure code quality and consistency before changes are committed to the repository.

Sources: [.pre-commit-config.yaml 3-19](https://github.com/tenstorrent/luwen/blob/55e35616/.pre-commit-config.yaml#L3-L19)

## RPM Packaging

The project also includes support for RPM packaging using Tito:

*   The `.tito` directory contains configuration for the Tito build system
*   Uses the standard Builder and VersionTagger
*   Configured for changelog management

This allows the project to be packaged as RPMs for distribution on Red Hat-based systems.

Sources: [.tito/tito.props 1-6](https://github.com/tenstorrent/luwen/blob/55e35616/.tito/tito.props#L1-L6)[.tito/packages/.readme 1-4](https://github.com/tenstorrent/luwen/blob/55e35616/.tito/packages/.readme#L1-L4)

## Environment Variables and Configuration

The CI/CD pipeline uses several key environment variables and configuration settings:

| Variable/Setting | Purpose |
| --- | --- |
| `RUST_BACKTRACE: 1` | Enables detailed backtraces for Rust errors |
| `CHANNEL` | Specifies the Rust channel (stable, nightly) |
| `PROFILE` | Build profile (debug, release) |
| `TARGET` | Build target (x86_64-unknown-linux-gnu) |
| `CARGO_OPTS` | Additional Cargo options |

Sources: [.gitlab-ci.yml 41-54](https://github.com/tenstorrent/luwen/blob/55e35616/.gitlab-ci.yml#L41-L54)

## Runner Requirements

The CI/CD pipeline relies on specific GitLab runner tags to ensure that jobs run on appropriate hardware:

| Tag | Purpose |
| --- | --- |
| `8-core` | General-purpose builds and deployments |
| `gs-1-card` | Tests requiring Grayskull hardware |
| `wh-nebula_x2-card` | Tests requiring Wormhole hardware |

This configuration ensures that hardware-specific tests can access the required Tenstorrent chips.

## Workflow for Developers

For developers working on the Luwen project, the typical workflow involves:

1.   Make code changes locally
2.   Pre-commit hooks automatically check code quality
3.   Push changes to GitLab
4.   CI/CD pipeline builds and tests the changes
5.   If all tests pass, changes can be merged
6.   Deployment jobs package the application for distribution
