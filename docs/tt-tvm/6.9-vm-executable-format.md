## Purpose and Scope

This document describes the structure, serialization format, and in-memory representation of the VM (Virtual Machine) executable. The VM executable is the compiled output from the VMCompiler that contains all the information needed to execute a Relay program at runtime, including bytecode, constants, and metadata.

For information about the VM compilation process that produces executables, see [VM Compiler](https://deepwiki.com/tenstorrent/tt-tvm/6.7-virtual-machine-backend). For information about the bytecode instruction set contained within executables, see [VM Instruction Set](https://deepwiki.com/tenstorrent/tt-tvm/6.8-vm-instruction-set). For information about how executables are executed, see [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor).

## Executable Overview

The VM executable (`Executable` class) is a self-contained module that encapsulates:

*   **Bytecode functions**: Compiled Relay functions represented as sequences of VM instructions
*   **Constants**: Pre-computed tensor values (model weights, etc.)
*   **Global mappings**: Function name to index mappings
*   **Primitive operations**: External packed functions (kernels)
*   **Virtual device information**: Device placement metadata

The executable can be serialized to disk, transferred over RPC, and deserialized for execution on target devices.

Sources: [include/tvm/runtime/vm/executable.h 61-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/runtime/vm/executable.h#L61-L100)[src/runtime/vm/executable.cc 61-124](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L61-L124)

## In-Memory Representation

**Key Data Members**

| Member | Type | Description |
| --- | --- | --- |
| `functions` | `vector<VMFunction>` | Array of compiled VM functions indexed by function ID |
| `constants` | `vector<ObjectRef>` | Constant pool containing NDArray values |
| `global_map` | `unordered_map<string,Index>` | Maps global function names to function indices |
| `primitive_map` | `unordered_map<string,Index>` | Maps primitive operation names to packed function indices |
| `virtual_devices` | `vector<pair<Device,String>>` | Virtual devices with device type, ID, and memory scope |
| `const_device_indexes` | `vector<Index>` | Device index for each constant in the pool |
| `late_bound_constant_names` | `vector<String>` | Names for constants that are late-bound (loaded separately) |
| `host_device_index` | `Index` | Index of the host device (always 0) |
| `lib` | `Module` | Runtime module containing compiled kernel code |

## Serialization Format

The VM executable is serialized as a binary blob with the following structure:

### Header Section

The header identifies the file format and version:

| Field | Size | Description |
| --- | --- | --- |
| Magic Number | 8 bytes | `0x6E6F73646D766D54` (`kTVMVMBytecodeMagic`) |
| Version String | Variable | TVM version string (e.g., "0.10.0") |

Sources: [src/runtime/vm/executable.cc 276-281](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L276-L281)

### Virtual Devices Section

Contains device placement information:

Each virtual device pair contains a `Device` (device_type + device_id) and a memory scope string (e.g., "global", "shared").

Sources: [src/runtime/vm/executable.cc 312-315](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L312-L315)

### Global Section

Maps function names to indices. Serialized as a sorted vector of function name strings, where the array index becomes the function's global index.

Sources: [src/runtime/vm/executable.cc 388-401](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L388-L401)

### Constant Section

The constant section supports two types of constants:

**Format:**

```
uint64_t: total_constant_count
For each constant:
  uint32_t: tag (0=immediate, 1=late-bound)
  If tag == 0 (immediate):
    DLTensor: serialized tensor data
  If tag == 1 (late-bound):
    string: constant name (e.g., "const_42")
vector<Index>: const_device_indexes
```

**Immediate Constants** (tag=0): Full NDArray data is embedded in the executable.

**Late-Bound Constants** (tag=1): Only a name is stored; actual data is loaded separately via `LoadLateBoundConstantsFromFile()` or `LoadLateBoundConstantsFromMap()`.

Sources: [src/runtime/vm/executable.cc 409-437](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L409-L437)[src/runtime/vm/executable.cc 439-475](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L439-L475)

### Primitive Names Section

Serializes the mapping of primitive operation names to their indices:

Sources: [src/runtime/vm/executable.cc 484-495](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L484-L495)

### Code Section

Serializes all VM functions with their bytecode:

```
uint64_t: num_functions
For each function:
  string: function_name
  vector<string>: parameter_names
  Index: register_file_size
  vector<Index>: param_device_indexes
  uint64_t: num_instructions
  For each instruction:
    VMInstructionSerializer: serialized_instruction
```

Each instruction is serialized with its opcode and operand fields.

## Late-Bound Constants

Late-bound constants enable memory-efficient deployment by storing large constants separately from the main executable file.

**Workflow:**

1.   **Separation**: `MoveLateBoundConstantsToFile(path, byte_limit)` extracts constants larger than `byte_limit` bytes
2.   **Serialization**: Constants are saved as a parameter map to disk
3.   **Loading**: At runtime, `LoadLateBoundConstantsFromFile(path)` restores the constants

**Key Methods:**

| Method | Description |
| --- | --- |
| `GetLateBoundConstants(byte_limit)` | Returns map of constants >= byte_limit |
| `MoveLateBoundConstantsToFile(path, byte_limit)` | Saves large constants to file |
| `LoadLateBoundConstantsFromFile(path)` | Restores constants from file |
| `LoadLateBoundConstantsFromMap(map)` | Restores constants from map |

When constants are late-bound:

*   The `constants` vector contains `nullptr` for late-bound entries
*   The `late_bound_constant_names` vector stores the name for each constant (or `nullptr` for immediate)
*   Names follow the pattern `"const_{index}"`

## Deserialization Process

**Loading Steps:**

1.   **Header Validation**: Verify magic number matches `kTVMVMBytecodeMagic` and version is compatible
2.   **Virtual Devices**: Restore device placement information
3.   **Globals**: Rebuild `global_map` from function names
4.   **Constants**: Load constants, marking late-bound ones with names
5.   **Primitives**: Rebuild `primitive_map` from primitive names
6.   **Code**: Deserialize all `VMFunction` objects and their bytecode
7.   **Library**: Attach the runtime module containing compiled kernels
8.   **Late-Bound Constants** (if any): Load from separate file or map

Sources: [src/runtime/vm/executable.cc 552-662](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/runtime/vm/executable.cc#L552-L662)

## Integration with Runtime

When loaded into a `VirtualMachine`:

The VM uses the executable's metadata to:

*   Index into the function array by global index
*   Access constants by index (with device caching)
*   Invoke primitive operations by packed function index
*   Determine device placement using virtual device indices
