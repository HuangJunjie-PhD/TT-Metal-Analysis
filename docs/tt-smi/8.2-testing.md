# Testing

Relevant source files
*   [pyproject.toml](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml)
*   [tests/test_reset.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py)

This document covers the testing framework for TT-SMI, specifically the pytest-based test suite that validates device reset functionality across different hardware platforms. The tests ensure that reset operations work correctly for both standard PCI devices and Galaxy 6U systems, including stress testing scenarios.

For information about the reset operations being tested, see [Reset Operations](https://deepwiki.com/tenstorrent/tt-smi/5-reset-operations). For details about the overall development workflow and CI/CD, see [Code Quality and CI/CD](https://deepwiki.com/tenstorrent/tt-smi/8.3-code-quality-and-cicd).

## Test Framework Configuration

TT-SMI uses pytest as its testing framework. The test dependencies are declared in `pyproject.toml`:

| Dependency | Version | Purpose |
| --- | --- | --- |
| `pytest` | (latest) | Core testing framework |

The test configuration is minimal and relies on pytest's default discovery patterns. Test files are located in the `tests/` directory and follow the `test_*.py` naming convention.

**Sources:**[pyproject.toml 44-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L44-L46)

## Test Architecture

The testing framework is organized around validating hardware reset operations, which are critical to system reliability. The test suite uses pytest fixtures for device discovery and system identification, then executes both single-reset and stress-test scenarios.

**Sources:**[tests/test_reset.py 1-104](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L1-L104)

## Test Fixtures

The test suite defines two session-scoped fixtures that provide environment information to all test functions.

### devices() Fixture

The `devices()` fixture returns a list of PCI indices for all detected Tenstorrent devices in the system. This fixture uses `pci_scan()` from the pyluwen library to enumerate devices.

The fixture is session-scoped, meaning device detection occurs once per test session to minimize overhead.

**Sources:**[tests/test_reset.py 13-18](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L13-L18)

### is_galaxy_6u() Fixture

The `is_galaxy_6u()` fixture determines whether the test system is a Galaxy 6U configuration. This detection logic is critical for routing tests to the appropriate hardware platform.

The Galaxy 6U detection algorithm:

1.   Calls `detect_chips_with_callback()` to enumerate all devices
2.   Checks if exactly 32 devices are present (Galaxy 6U has 4 trays × 8 devices)
3.   Extracts board type for each device using `get_board_type()`
4.   Verifies all boards are either `tt-galaxy-wh` or `tt-galaxy-bh`

**Sources:**[tests/test_reset.py 21-32](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L21-L32)

## PCI Reset Test Suite

The PCI reset tests validate standard board-level reset operations for non-Galaxy systems. These tests are automatically skipped on Galaxy 6U systems.

### test_pci_reset_all_devices()

This test validates a single reset cycle of all detected devices, simulating the behavior of `tt-smi -r`:

| Test Step | Operation | Validation |
| --- | --- | --- |
| 1. Pre-reset | Get device list from fixture | Baseline device count |
| 2. Reset | Call `pci_board_reset(devices, reinit=True)` | Reset all devices |
| 3. Post-reset scan | Call `pci_scan()` | New device list |
| 4. Assertion | Compare counts | `len(post_reset_devices) == len(devices)` |

The test passes if the same number of devices are detected after reset, indicating successful reset and re-initialization.

**Sources:**[tests/test_reset.py 34-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L34-L46)

### test_pci_reset_all_devices_stress()

This stress test performs 10 consecutive reset cycles (configurable via `NUM_RESETS_STRESS_TEST`):

The stress test validates:

*   Reset operation stability under repeated cycles
*   Consistent device re-enumeration
*   Absence of hardware state corruption

**Sources:**[tests/test_reset.py 11-62](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L11-L62)

## Galaxy Reset Test Suite

The Galaxy reset tests validate IPMI-based reset operations specific to Galaxy 6U systems. These tests are automatically skipped on non-Galaxy systems.

### test_glx_reset()

This test validates a single Galaxy 6U trays reset, simulating the behavior of `tt-smi -glx_reset`:

The Galaxy reset test has special handling for `SystemExit`:

*   `glx_6u_trays_reset()` calls `sys.exit(0)` on successful reset
*   Test uses `pytest.raises(SystemExit)` to catch the exit
*   Validates exit code is 0 (success)
*   Confirms device count matches pre-reset count

**Sources:**[tests/test_reset.py 65-82](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L65-L82)

### test_glx_reset_stress()

This stress test performs 10 consecutive Galaxy reset cycles:

| Iteration | Operations | Validations |
| --- | --- | --- |
| Each of 10 cycles | Call `glx_6u_trays_reset()` | Exit code == 0 |
|  | Call `pci_scan()` | Device count unchanged |

The Galaxy stress test validates:

*   IPMI command reliability under repeated execution
*   Driver reload consistency (30-second waits per reset)
*   Ethernet link re-establishment across cycles

**Sources:**[tests/test_reset.py 85-103](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L85-L103)

This pattern ensures:

### Test Output Interpretation

| Result | Meaning |
| --- | --- |
| `PASSED` | Test completed successfully, device count matched |
| `SKIPPED` | Test skipped due to hardware type mismatch |
| `FAILED` | Reset failed or device count mismatch detected |

```
tests/test_reset.py::test_glx_reset_stress SKIPPED (non-Galaxy system)
```

## Test Coverage and Scope

The current test suite focuses exclusively on reset operations, which are critical for system reliability. The tests validate:

| Category | Coverage |
| --- | --- |
| Reset Types | Board-level (PCI), IPMI-based (Galaxy) |
| Platforms | Wormhole, Blackhole, Galaxy WH, Galaxy BH |
| Scenarios | Single reset, stress test (10 cycles) |
| Validations | Device count preservation, successful re-initialization |

The test suite does **not** currently cover:

*   Telemetry collection accuracy
*   GUI rendering and interactions
*   Firmware version reporting
*   PCI property detection
*   Snapshot generation

These areas may be candidates for future test expansion.

**Sources:**[tests/test_reset.py 1-104](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L1-L104)

## Configuration Constants

The test suite uses a single configurable constant:

| Constant | Value | Purpose | Location |
| --- | --- | --- | --- |
| `NUM_RESETS_STRESS_TEST` | 10 | Number of reset cycles in stress tests | [tests/test_reset.py 11](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L11-L11) |

This value can be modified to adjust stress test duration. Higher values provide more thorough validation but increase test execution time (Galaxy resets require ~30 seconds per cycle).

**Sources:**[tests/test_reset.py 11](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L11-L11)
