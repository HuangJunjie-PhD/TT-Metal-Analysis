# Container Definition and Build

Relevant source files
*   [infra/Container.def](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def)

## Purpose and Scope

This document describes the Singularity container definition file (`Container.def`) and the build process that creates the RiESCUE development environment. The container encapsulates all system-level dependencies, RISC-V tools, and Python packages required for test generation and simulation.

For information about managing built containers (build, run, push, pull operations), see [Container Management Commands](https://deepwiki.com/tenstorrent/riescue/3.1.2-container-management-commands). For container runtime configuration (registry, bind mounts, authentication), see [Container Configuration](https://deepwiki.com/tenstorrent/riescue/3.1.3-container-configuration). For details on the individual build scripts executed during container creation, see [Build Scripts and Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2-build-scripts-and-dependencies).

## Container Definition Overview

The container is defined in `Container.def`, a Singularity definition file that specifies a multi-stage build process. The container uses Rocky Linux 9.1 as the base operating system and installs the complete RiESCUE development stack including compilers, simulators, and Python dependencies.

**Key Design Principles:**

*   **Reproducibility**: All dependencies are installed from known sources with pinned versions
*   **Self-contained**: The container includes all tools needed for test generation and simulation
*   **Layer caching**: Build steps are ordered to maximize Docker layer cache efficiency
*   **Portability**: Works across different host systems (Linux, macOS with appropriate Singularity support)

Sources: [infra/Container.def 1-57](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L1-L57)

## Definition File Structure

The Singularity definition file is organized into several sections that control different aspects of the container build and runtime behavior.

**Container Definition Structure**

Sources: [infra/Container.def 1-57](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L1-L57)

### Bootstrap and Base Image

The container bootstraps from a Docker image and uses Rocky Linux 9.1 as the base operating system:

| Directive | Value | Purpose |
| --- | --- | --- |
| `bootstrap` | `docker` | Use Docker Hub as the image source |
| `From` | `rockylinux:9.1` | Base image with stable RHEL-compatible OS |
| `Stage` | `devel` | Build stage identifier |

Rocky Linux 9.1 was chosen for its stability, long-term support, and compatibility with enterprise RISC-V development tools.

Sources: [infra/Container.def 1-3](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L1-L3)

### Environment Configuration

The `%environment` section defines environment variables that are set each time the container runs:

**Environment Variables:**

*   `PATH`: Prepends `/usr/local/bin` to ensure locally-built tools (Whisper, Spike) are found first
*   `LD_LIBRARY_PATH`: Adds `/usr/lib64` and `/shared` for shared library discovery, supporting both system libraries and custom-built dependencies

Sources: [infra/Container.def 5-7](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L5-L7)

This removes stack size limitations, which is important for RISC-V simulators that may use deep recursion or large stack frames during instruction emulation.

### Container Metadata

Labels store metadata about the container:

| Label | Value | Purpose |
| --- | --- | --- |
| `DESCRIPTION` | Warning message | Documents that version is auto-updated by `container-push` |
| `VERSION` | `1.30` | Current container version number |

The version number is automatically incremented by the `container-push` script during container publication. Manual changes to this field should be avoided.

## File Staging

The `%files` section copies files from the host build context into the container filesystem before the `%post` installation stage:

**File Staging Table:**

| Source (Host) | Destination (Container) | Purpose |
| --- | --- | --- |
| `pyproject.toml` | `/app/pyproject.toml` | Python project metadata and dependencies |
| `README.md` | `/app/README.md` | Project documentation |
| `riescue/` | `/app/riescue/` | RiESCUE Python package source code |
| `infra/build_spike.sh` | `/build_spike.sh` | Spike simulator build script |
| `infra/build_whisper.sh` | `/build_whisper.sh` | Whisper simulator build script |
| `infra/build_base_deps.sh` | `/build_base_deps.sh` | System dependencies build script |

Note that `pyproject.toml` appears twice in the definition file, which is redundant but harmless.

Sources: [infra/Container.def 19-26](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L19-L26)

The `%post` section executes a multi-stage installation process that builds the complete development environment. This is the most complex part of the container definition.

### Stage 2: Python Environment

The second stage sets up the Python environment and installs all Python dependencies:

**Python Packages Installed:**

| Package | Purpose |
| --- | --- |
| `wheel` | Python packaging format support |
| `sortedcontainers` | Sorted list/dict data structures |

Note that the full RiESCUE package with additional dependencies is not installed during container build. The container only includes these core dependencies. The actual RiESCUE package is typically installed in editable mode after the container is built, or via the Python package management system. ### Stage 3: RISC-V Simulators

The final stage builds the RISC-V instruction set simulators from source:

The `NO_SINGULARITY=1` environment variable is set to prevent the build scripts from attempting to use Singularity recursively (since they're already running inside a container build).

**Simulators Built:**

*   **Whisper**: Tenstorrent's RISC-V ISS (installed to `/usr/bin/whisper`)
*   **Spike**: Standard RISC-V ISS (installed to `/usr/local/bin/spike`)
*   **tt_spike**: Tenstorrent-specific Spike variant (installed to `/usr/local/bin/tt_spike`)

For details on how these simulators are built, see [Building Whisper](https://deepwiki.com/tenstorrent/riescue/3.2.2-building-whisper) and [Building Spike](https://deepwiki.com/tenstorrent/riescue/3.2.3-building-spike).

## Version Management

The container version is stored in the `VERSION` label and follows a simple integer versioning scheme:

```
VERSION 1.30
```

**Versioning Strategy:**

*   Version numbers are incremented sequentially (1.29 → 1.30 → 1.31)
*   The version is automatically updated by the `container-push` script when publishing to a registry
*   Manual changes to the version field are discouraged and will be overwritten on the next push
*   The version serves as a tag for the container image in the registry

The container definition file includes a warning comment:

```
DESCRIPTION Please do not change the version, ./infra/container-push will update the version number here
```

This automated versioning ensures consistency between the definition file and the published container image. For information on how versioning is managed during container publication, see [Container Management Commands](https://deepwiki.com/tenstorrent/riescue/3.1.2-container-management-commands).

Sources: [infra/Container.def 12-14](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L12-L14)

## Integration with Build Scripts

The container definition delegates complex installation tasks to dedicated build scripts. This separation of concerns provides several benefits:

**Benefits of Delegated Build Scripts:**

*   **Modularity**: Each build script can be tested and maintained independently
*   **Reusability**: Build scripts can be executed outside the container for local development
*   **Clarity**: The container definition remains clean and high-level
*   **Version control**: Build scripts can reference specific commits of external repositories

**Build Script Integration Architecture**

Each build script is self-contained and can be invoked independently for testing or local development. The scripts handle:

*   Repository cloning from GitHub
*   Checkout of specific commits for reproducibility
*   Configuration of build options
*   Compilation and installation
*   Error handling and logging

For detailed documentation of each build script, see [Build Scripts and Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2-build-scripts-and-dependencies).

Sources: [infra/Container.def 19-56](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L19-L56)

## Container Image Size and Optimization

The resulting container image is substantial due to the complete development toolchain:

**Approximate Size Breakdown:**

*   Base Rocky Linux image: ~200 MB
*   System packages (gcc, cmake, boost, etc.): ~1.5 GB
*   Python packages and dependencies: ~500 MB
*   Whisper simulator: ~50 MB
*   Spike simulators (spike + tt_spike): ~100 MB
*   Source files and build artifacts: ~200 MB

**Total container size**: Approximately 2-3 GB

The container definition is optimized for build reproducibility and completeness rather than size. No size reduction techniques (multi-stage builds, layer squashing) are currently applied, as the development workflow prioritizes having all tools available in a single container.

Sources: [infra/Container.def 1-57](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L1-L57)

## Common Build Issues and Solutions

**Issue: Build fails during `build_base_deps.sh`**

*   **Cause**: Network issues or Rocky Linux repository unavailability
*   **Solution**: Retry the build or check network connectivity

**Issue: Build fails during Whisper/Spike compilation**

*   **Cause**: Insufficient memory or CPU resources
*   **Solution**: Allocate more resources to the build process

**Issue: Version conflict with existing container**

*   **Cause**: Attempting to build over an existing container with the same version
*   **Solution**: Use `container-push` to automatically increment version, or manually delete old container

**Issue: Python package installation failures**

*   **Cause**: PyPI connection issues or package version conflicts
*   **Solution**: Check network connectivity to PyPI; build script uses unpinned versions for flexibility

For troubleshooting container management operations, see [Container Management Commands](https://deepwiki.com/tenstorrent/riescue/3.1.2-container-management-commands).
