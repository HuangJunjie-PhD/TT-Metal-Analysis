## Purpose and Scope

This page documents the token sampling pipeline in the `vllm_tt` plugin: how per-request sampling parameters are collected from `InputBatch`, how they are packaged into an XLA-compatible metadata object, and how the `Sampler` module applies logit modifications and sampling strategies to produce output tokens on Tenstorrent hardware.

This page focuses on the on-device sampling logic. For the model runners that call this pipeline (`TTModelRunner`, `TTPoolingModelRunner`) and their KV cache and input preparation responsibilities, see [Platform, Worker, and Model Runners](https://deepwiki.com/tenstorrent/tt-xla/5.3.1-platform-worker-and-model-runners).

## InputBatch: Persistent CPU Sampling Tensors

`InputBatch` ([integrations/vllm_plugin/vllm_tt/input_batch.py 24-210](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/input_batch.py#L24-L210)) maintains one CPU tensor per sampling parameter, sized `[max_num_reqs]`. These tensors are updated incrementally as requests are added, removed, or modified by the scheduler. They serve as the source for building device tensors each step.

| Field | Dtype | Shape | Default |
| --- | --- | --- | --- |
| `temperature_cpu_tensor` | float32 | `[max_num_reqs]` | set per request |
| `top_p_cpu_tensor` | float32 | `[max_num_reqs]` | 1.0 |
| `top_k_cpu_tensor` | int32 | `[max_num_reqs]` | vocab_size |
| `min_p_cpu_tensor` | float32 | `[max_num_reqs]` | 0.0 |
| `frequency_penalties_cpu_tensor` | float32 | `[max_num_reqs]` | 0.0 |
| `presence_penalties_cpu_tensor` | float32 | `[max_num_reqs]` | 0.0 |
| `repetition_penalties_cpu_tensor` | float32 | `[max_num_reqs]` | 1.0 |
| `logit_bias` | `list[dict[int,float] | None]` | `[max_num_reqs]` |
| `bad_words_token_ids` | `dict[int, list[list[int]]]` | sparse | {} |

`InputBatch` also tracks set membership for each feature (`greedy_reqs`, `random_reqs`, `top_p_reqs`, `top_k_reqs`, `min_p_reqs`, `frequency_penalties_reqs`, etc.) which is used to derive the batch-level flags `all_greedy`, `all_random`, and `no_penalties`.

Sources: [integrations/vllm_plugin/vllm_tt/input_batch.py 94-200](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/input_batch.py#L94-L200)[integrations/vllm_plugin/vllm_tt/input_batch.py 259-336](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/input_batch.py#L259-L336)

## XLASupportedSamplingMetadata

`XLASupportedSamplingMetadata` ([integrations/vllm_plugin/vllm_tt/metadata.py 24-328](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/metadata.py#L24-L328)) is the metadata object passed into the compiled sampler graph. It is designed to satisfy XLA compilation constraints:

*   **No `Optional` tensor fields with branching on `None`** — all tensor fields are either always present or guarded by a non-tensor boolean flag (`no_penalties`, `no_logit_bias`, `no_bad_words`).
*   **Fixed shapes** — all tensors are padded to `padded_num_reqs` before being moved to device. This prevents recompilation caused by variable batch sizes.
*   **No per-request generators** — `_generators` always returns an empty dict; `torch.Generator` objects are not portable across XLA devices.

### Fields

| Field | Type | Notes |
| --- | --- | --- |
| `temperature` | `Tensor [batch]` | float32; per-request |
| `min_p` | `Tensor [batch]` | float32 |
| `top_k` | `Tensor [batch]` | int32 |
| `top_p` | `Tensor [batch]` | float32 |
| `all_greedy` | `bool` | True when every request uses greedy |
| `all_random` | `bool` | True when no request uses greedy |
| `logprobs` | `bool` | Always False (not yet implemented) |
| `no_penalties` | `bool` | Skips penalty computation entirely |
| `output_token_counts` | `Tensor [batch, vocab]` or None | float32; counts of output tokens |
| `prompt_token_mask` | `Tensor [batch, vocab]` or None | bool; True for prompt-present tokens |
| `presence_penalties` | `Tensor [batch]` or None |  |
| `frequency_penalties` | `Tensor [batch]` or None |  |
| `repetition_penalties` | `Tensor [batch]` or None |  |
| `no_logit_bias` | `bool` | Skips logit bias |
| `logit_bias_tensor` | `Tensor [batch, vocab]` or None | float32; additive bias |
| `no_bad_words` | `bool` | Skips bad-words masking |
| `bad_words_mask` | `Tensor [batch, vocab]` or None | float32; -inf for banned tokens, 0 elsewhere |

### Construction via `from_input_batch()`

`XLASupportedSamplingMetadata.from_input_batch()` ([integrations/vllm_plugin/vllm_tt/metadata.py 157-328](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/metadata.py#L157-L328)) is the factory method called each step. The sequence is:

1.   **Guard checks** — raises `NotImplementedError` if `logprobs` or `generators` are requested (tracked issues #3366 and #3365 respectively).
2.   **Logit bias tensor** — iterates `input_batch.logit_bias[:num_reqs]`, builds a `[padded_num_reqs, vocab_size]` float32 tensor on CPU, then moves to device.
3.   **Bad-words mask** — calls `_compute_bad_words_mask()` to build a `[padded_num_reqs, vocab_size]` float32 mask.
4.   **Early return** — if `all_greedy` is True and no logit-modifying features are active (no penalties, no logit bias, no bad words), returns a minimal metadata object immediately to skip unnecessary CPU-to-device copies.
5.   **Pad sampling tensors** — fills `input_batch.*_cpu_tensor[num_reqs:padded_num_reqs]` with default values.
6.   **Penalty tensors** — if `has_penalties`, calls `_compute_token_counts()` and `_compute_prompt_mask()` on CPU, then moves all penalty tensors to device.
7.   **Move base tensors to device** — slices all `*_cpu_tensor[:padded_num_reqs]` and calls `.to(xla_device)`.

## Bad-Words Mask Construction

`_compute_bad_words_mask()` supports both single-token and multi-token bad words:

*   **Single-token**`[T]`: always sets `mask[req_idx, T] = -inf`.
*   **Multi-token**`[t0, t1, ..., tN]`: bans `tN` only when the most recent `N` output tokens equal `[t0, ..., t(N-1)]` (prefix matching against `req_output_token_ids`). If there is insufficient output history for the prefix length, the ban is skipped for this step.

This logic runs entirely on CPU during `from_input_batch()`. The resulting mask is a dense `[batch, vocab]` float32 tensor that is added to logits directly.

Sources: [integrations/vllm_plugin/vllm_tt/metadata.py 113-155](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/metadata.py#L113-L155)

## Sampler

`Sampler` ([integrations/vllm_plugin/vllm_tt/sampler.py 14-244](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L14-L244)) is an `nn.Module` that applies all logit modifications and selects output tokens.

### Compilation

In `TTModelRunner.__init__()` ([integrations/vllm_plugin/vllm_tt/model_runner.py 438-446](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/model_runner.py#L438-L446)):

```
if not self.enable_tensor_parallel:
    self.sample_from_logits_func = torch.compile(
        self.sample_from_logits,
        backend="tt",
        fullgraph=True,
        dynamic=False,
    )
else:
    self.sample_from_logits_func = self.sample_from_logits
```

When tensor parallelism is disabled, the sampler is compiled as a separate XLA subgraph with `dynamic=False` (fixed shapes) and `fullgraph=True`. This ensures the sampler graph is compiled once and reused across all steps.

### forward() and sample() Pipeline

`Sampler.forward()` casts logits to float32 then delegates to `Sampler.sample()`. `sample()` applies operations in the following fixed order:

### Individual Operations

| Method | Input/Output | Description |
| --- | --- | --- |
| `apply_bad_words(logits, bad_words_mask)` | `Tensor + Tensor → Tensor` | Elementwise addition; banned positions receive -inf |
| `apply_logit_bias(logits, logit_bias_tensor)` | `Tensor + Tensor → Tensor` | Elementwise addition of per-token bias values |
| `apply_penalties(logits, output_token_counts, prompt_token_mask, presence, frequency, repetition)` | `Tensor → Tensor` | See penalty section below |
| `greedy_sample(logits)` | `Tensor → Tensor [batch]` | `argmax(dim=-1)` |
| `apply_temperature(logits, temp, all_random)` | `Tensor → Tensor` | `logits / temp`; guards against division by zero for greedy requests |
| `apply_min_p(logits, min_p)` | `Tensor → Tensor` | Computes softmax, finds max prob per row, masks tokens below `min_p * max_prob` |
| `apply_top_k_top_p(logits, k, p)` | `Tensor → Tensor` | Module-level function; XLA-optimized to avoid `torch.scatter` |
| `random_sample(probs, generators)` | `Tensor → Tensor [batch]` | `(probs / Exponential()).argmax(dim=-1)` |

Sources: [integrations/vllm_plugin/vllm_tt/sampler.py 39-244](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L39-L244)

### Penalty Application

`apply_penalties()` matches the vLLM GPU specification from `vllm/model_executor/layers/utils.py`. The three penalties are applied in order: repetition → frequency → presence.

```
rep_mask = (output_token_counts > 0) | prompt_token_mask   # prompt ∪ output
penalty_factor = where(logits > 0, 1/rep, rep)
logits = where(rep_mask, logits * penalty_factor, logits)  # repetition

logits -= frequency_penalties * output_token_counts         # frequency
logits -= presence_penalties * (output_token_counts > 0)   # presence
```

The `prompt_token_mask` ensures repetition penalty covers tokens that appeared in the prompt but not yet in output — matching the GPU behavior.

Only element-wise masking operations are used; no `sort`, `scatter`, or `gather` calls, which would be expensive on XLA.

Sources: [integrations/vllm_plugin/vllm_tt/sampler.py 53-93](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L53-L93)

### Top-k / Top-p Filtering

`apply_top_k_top_p()` ([integrations/vllm_plugin/vllm_tt/sampler.py 247](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L247-LNaN)) avoids `torch.scatter` because it is slow on TPU/XLA backends. Instead it finds a scalar "cutoff" value in the sorted logits and applies threshold masking to retain only the top-k and top-p tokens.

## Greedy vs. Random Token Selection

The sampler computes both the greedy token (`argmax`) and the random token (exponential gumbel sampling) regardless of the sampling mode. The final selection uses `torch.where`:

```
sampled = where(temperature < 1e-5, greedy_sampled, random_sampled)
```

This avoids data-dependent branching inside the compiled graph, which would cause XLA recompilation. Both paths execute on every call, and the `where` selects the appropriate result per request.

Sources: [integrations/vllm_plugin/vllm_tt/sampler.py 155-163](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L155-L163)

## XLA Compilation Constraints

The design of both `XLASupportedSamplingMetadata` and `Sampler` is driven by the need to produce a stable, recompilation-free XLA graph.

| Constraint | Reason | Implementation |
| --- | --- | --- |
| No `Optional` tensors inside the compiled graph | Dynamic dispatch on `None` causes graph branching and recompilation | Boolean flags (`no_penalties`, `no_logit_bias`, `no_bad_words`) guard tensor access outside the compiled subgraph |
| Fixed `padded_num_reqs` shape | XLA specializes on tensor shapes; variable batch size triggers recompilation | Tensors padded to `padded_num_reqs` with default values before device transfer |
| No per-request `torch.Generator` | Generators are CPU objects and cannot be captured in XLA graphs | `generators` field always returns `{}` |
| CPU-side dynamic logic | Slice indices and Python-level control flow cannot appear in XLA trace | `from_input_batch()` runs entirely on CPU, including `_compute_token_counts()`, `_compute_prompt_mask()`, `_compute_bad_words_mask()` |
| Both greedy and random paths always execute | `torch.where` avoids data-dependent branching | Both `greedy_sample()` and `random_sample()` are called; `torch.where` selects per-element |
| `fullgraph=True, dynamic=False` in `torch.compile` | Forces a single static graph with no graph breaks | Applied to `sample_from_logits` in `TTModelRunner.__init__()` |

Sources: [integrations/vllm_plugin/vllm_tt/metadata.py 24-70](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/metadata.py#L24-L70)[integrations/vllm_plugin/vllm_tt/model_runner.py 438-446](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/model_runner.py#L438-L446)[integrations/vllm_plugin/vllm_tt/sampler.py 109-163](https://github.com/tenstorrent/tt-xla/blob/c77995f6/integrations/vllm_plugin/vllm_tt/sampler.py#L109-L163)
