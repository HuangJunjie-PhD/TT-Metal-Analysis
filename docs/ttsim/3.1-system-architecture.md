## Purpose and Scope

This document explains the overall system design of ttsim, including its layered architecture, shared library structure, and relationship with TT-Metalium. For details on the simulated hardware components and Tensix core architecture, see [Hardware Simulation](https://deepwiki.com/tenstorrent/ttsim/3.2-hardware-simulation). For information about API interfaces, environment variables, and SOC descriptors, see [TT-Metalium Integration](https://deepwiki.com/tenstorrent/ttsim/3.3-tt-metalium-integration).

**Sources:**[README.md 1-121](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L1-L121)

## Overview

ttsim is a full-system simulator that emulates Tenstorrent hardware as a shared library (`libttsim.so`) running on Linux/x86_64 host systems. The simulator provides a virtual device that TT-Metalium can program and execute workloads on, without requiring physical silicon. Each simulator binary is compiled for a specific chip architecture (Wormhole or Blackhole) and provides bit-exact numerical results for most computations.

The core architectural principle is **host-based simulation**: the simulator runs as a user-space shared library on x86_64 Linux, interpreting and executing workloads that would otherwise run on specialized Tenstorrent hardware. This approach enables rapid development and testing without hardware dependencies.

**Sources:**[README.md 3-10](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L3-L10)

## Layered Architecture

**Layered Architecture Diagram**

The system consists of four primary layers:

| Layer | Components | Responsibility |
| --- | --- | --- |
| **Application** | User workloads, tt-metal examples, ttnn tests, LLK tests | Execute computational workloads on virtual device |
| **Software Stack** | TT-Metalium, environment variables | Orchestrate device programming and workload execution |
| **Simulation** | `libttsim_wh.so`, `libttsim_bh.so`, API | Emulate hardware behavior and expose virtual device |
| **Hardware Model** | Virtual device, SOC descriptors, simulated Tensix cores | Model chip architecture and compute units |
| **Host Environment** | Linux/x86_64, user-space process | Provide execution environment for simulator |

## Shared Library Design

**Shared Library Architecture**

ttsim implements a **shared library plugin architecture** where:

*   **Single file per architecture**: Each simulator is a single `.so` file (e.g., `libttsim_wh.so`, `libttsim_bh.so`)
*   **Dynamic loading**: TT-Metalium discovers and loads the simulator via the `TT_METAL_SIMULATOR` environment variable
*   **Co-location requirement**: The `.so` file must reside in the same directory as `soc_descriptor.yaml`
*   **API boundary**: The simulator exports a defined API that TT-Metalium calls to program and interact with the virtual device

This design provides several advantages:

*   **Architecture isolation**: Different chip architectures are completely separate binaries
*   **Binary distribution**: Can distribute compiled binaries without source code
*   **Version management**: Users can maintain multiple simulator versions side-by-side
*   **Drop-in replacement**: No recompilation of TT-Metalium required to use simulator

## Architecture-Specific Binaries

**Architecture-Specific Binary Organization**

ttsim provides separate binaries for each supported chip architecture:

| Binary | SOC Descriptor | Target Architecture |
| --- | --- | --- |
| `libttsim_wh.so` | `wormhole_b0_80_arch.yaml` | Wormhole B0 (80 cores) |
| `libttsim_bh.so` | `blackhole_140_arch.yaml` | Blackhole (140 cores) |

**Key characteristics:**

*   **Independent binaries**: Each architecture has a completely separate `.so` file
*   **Maturity levels**: Wormhole is in more mature shape than Blackhole
*   **SOC descriptor pairing**: Each binary requires its corresponding YAML descriptor file
*   **Configuration-driven selection**: The `TT_METAL_SIMULATOR` environment variable determines which architecture to use

The SOC descriptor file defines the chip architecture details that the simulator needs:

*   Core count and grid layout
*   Memory topology and sizes
*   NOC configuration
*   Functional unit capabilities

## File Structure and Dependencies

**File Structure and Configuration Dependencies**

The simulator requires a specific file organization:

**Directory structure:**

```
~/sim/                           # Simulator installation directory
├── libttsim_wh.so              # Wormhole simulator binary
├── libttsim_bh.so              # Blackhole simulator binary
└── soc_descriptor.yaml         # SOC descriptor (copied from TT_METAL_HOME)

$TT_METAL_HOME/                 # TT-Metalium installation
├── tt_metal/
│   └── soc_descriptors/
│       ├── wormhole_b0_80_arch.yaml
│       └── blackhole_140_arch.yaml
└── build/programming_examples/
    └── metal_example_add_2_integers_in_riscv
```

**Co-location requirement:** The simulator `.so` file and `soc_descriptor.yaml` must be in the same directory. This design simplifies path resolution and ensures the simulator always has access to its configuration.

## Execution Model

**Execution Flow Diagram**

The execution model operates as follows:

1.   **Configuration phase**: Environment variables are set to configure TT-Metalium for simulation mode
2.   **Discovery phase**: TT-Metalium reads `TT_METAL_SIMULATOR` to locate the `.so` file
3.   **Loading phase**: The shared library is dynamically loaded via `dlopen()`
4.   **Initialization phase**: Simulator reads the co-located SOC descriptor and creates a virtual device
5.   **Programming phase**: TT-Metalium programs the virtual device using the simulator API
6.   **Execution phase**: Workload runs on simulated Tensix cores with emulated hardware behavior
7.   **Results phase**: TT-Metalium retrieves results from virtual device memory

**Slow dispatch requirement:** Currently, fast dispatch is not implemented. The `TT_METAL_SLOW_DISPATCH_MODE=1` environment variable must be set for execution to succeed.

## Bit-Exact Numerical Model

The simulator implements a **bit-exact numerical model** for hardware computations:

| Component | Status | Description |
| --- | --- | --- |
| **Unpacker** | Fully bit-accurate | Data unpacking operations match silicon exactly |
| **SFPU** | Fully bit-accurate | Special function unit operations match silicon exactly |
| **Packer** | Fully bit-accurate | Data packing operations match silicon exactly |
| **FPU MOV*** | Fully bit-accurate | Move operations match silicon exactly |
| **FPU GMPOOL** | Fully bit-accurate | Global max pool operations match silicon exactly |
| **FPU ELW*** | Not yet bit-accurate | Element-wise operations planned for improvement |
| **FPU MVMUL** | Not yet bit-accurate | Matrix-vector multiply planned for improvement |
| **FPU GAPOOL** | Not yet bit-accurate | Global average pool planned for improvement |

The bit-exact goal means that for supported operations, the simulator produces:

*   Identical floating-point bit patterns
*   Identical integer results
*   Identical NaN representations
*   Deterministic behavior matching silicon

**Divergence sources** that can cause differences from silicon:

*   Timing-dependent operation ordering
*   Hardware entropy sources or RNGs
*   Performance counters and cycle timers
*   Missing synchronization primitives
*   Execution of undefined behavior
*   ISA specification violations

## Component Relationships

**System Component Relationships**

The diagram shows how the major components interact:

*   **Configuration drives discovery**: Environment variables tell TT-Metalium where to find the simulator
*   **Co-location enforces consistency**: SOC descriptor must be adjacent to simulator binary
*   **Dynamic loading enables flexibility**: Simulator is loaded at runtime, not compile-time
*   **API provides abstraction**: Simulator implements a device interface that TT-Metalium programs
*   **Hardware model is internal**: Virtual cores and functional units are implementation details hidden behind the API
