# Synchronization Primitives

Relevant source files
*   [chardev.c](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c)
*   [device.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h)

## Purpose and Scope

This document describes the kernel-level synchronization primitives used throughout the Tenstorrent kernel driver to protect shared data structures and coordinate concurrent operations. It covers mutexes, read-write semaphores, wait queues, and the locking patterns employed to ensure safe access across multiple contexts (user-space clients, reset operations, device removal, interrupt handlers).

For user-space resource locking mechanisms, see [Resource Locks](https://deepwiki.com/tenstorrent/tt-kmd/9.2-resource-locks). For reset-specific coordination and generation tracking, see [Reset and Recovery](https://deepwiki.com/tenstorrent/tt-kmd/9.4-reset-and-recovery). For power state aggregation locking, see [Power Management](https://deepwiki.com/tenstorrent/tt-kmd/9.3-power-management).

## Overview

The driver must coordinate access from multiple sources:

*   Multiple concurrent file descriptors from different processes.
*   Device reset operations that invalidate hardware state.
*   Device removal (hot-unplug) events.
*   Interrupt handlers and deferred work queues.
*   PCI configuration space operations.

The synchronization strategy uses a combination of:

*   **Read-write semaphores** for reset coordination (allowing concurrent normal operations).
*   **Mutexes** for protecting device-global and per-FD state.
*   **Wait queues** for blocking operations.
*   **Atomic counters** for generation-based invalidation.

## Device-Level Synchronization Primitives

### reset_rwsem - Read-Write Semaphore

**Code-to-System Mapping: Reset Coordination**

**Purpose**: Coordinates device reset operations with normal hardware access.

**Type**: `struct rw_semaphore reset_rwsem` declared in [device.h 35](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L35-L35)

**Locking Pattern**:

*   **Read lock**: Acquired by all non-reset IOCTLs to allow concurrent normal operations [chardev.c 475](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L475-L475)
*   **Write lock**: Acquired exclusively by reset IOCTLs to prevent any hardware access during reset [chardev.c 473](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L473-L473)

**Key Implementation Details**:

In `tt_cdev_ioctl`, the driver determines the lock type based on the command [chardev.c 472-475](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L472-L475):

**ABBA Deadlock Avoidance**: In `tt_cdev_mmap`, the driver uses `down_read_trylock()`[chardev.c 587](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L587-L587) This is necessary because `mmap()` is called with the kernel's `mmap_lock` held. Conversely, `tenstorrent_vma_zap()` (called during reset while holding `reset_rwsem`) eventually attempts to acquire `mmap_lock` to invalidate pages. Using a blocking `down_read()` in `mmap` would create a circular dependency [chardev.c 587-590](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L587-L590)

Sources: [device.h 35](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L35-L35)[chardev.c 472-475](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L472-L475)[chardev.c 587-590](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L587-L590)

### chardev_mutex - Character Device Mutex

**Purpose**: Protects the per-device state related to open file descriptors and lifecycle callbacks.

**Type**: `struct mutex chardev_mutex` declared in [device.h 43](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L43-L43)

**Protected Resources**:

*   `chardev_open_count`: Number of open file descriptors [device.h 44](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L44-L44)
*   `open_fds_list`: List of active file descriptors (`struct chardev_private`) [device.h 55](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L55-L55)
*   Invocation of `first_open_cb` and `last_release_cb`[device.h 97-98](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L97-L98)

**Usage Pattern**: In `tt_cdev_open`, the mutex ensures that `first_open_cb` is called exactly once when the first user opens the device [chardev.c 613-619](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L613-L619) Similarly, it serializes the addition of new `chardev_private` structures to the `open_fds_list`[chardev.c 661-663](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L661-L663)

Sources: [device.h 43-44](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L43-L44)[device.h 55](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L55-L55)[chardev.c 613-627](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L613-L627)[chardev.c 661-663](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L661-L663)

### iatu_mutex - IATU Configuration Mutex

**Purpose**: Serializes access to the outbound iATU (Internal Address Translation Unit) region configuration.

**Type**: `struct mutex iatu_mutex` declared in [device.h 66](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L66-L66)

**Protected Resources**:

*   `outbound_iatus` array: Configuration state for outbound mapping regions [device.h 67](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L67-L67)
*   Physical iATU hardware registers (accessed via `dev_class->configure_outbound_atu`) [device.h 104](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L104-L104)

**Why Needed**: Multiple threads may attempt to map different segments of device memory (e.g., peer-to-peer BAR mappings). The mutex ensures that the allocation of an iATU window and the subsequent hardware register writes are atomic relative to other translation requests.

Sources: [device.h 66-67](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L66-L67)[device.h 104](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L104-L104)

## File Descriptor-Level Synchronization

### chardev_private Mutex

**Purpose**: Protects per-file-descriptor state that may be accessed concurrently (e.g., by a reset thread iterating all open FDs).

**Type**: `struct mutex mutex` in `struct chardev_private` (initialized in `tt_cdev_open`) [chardev.c 639](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L639-L639)

**Protected Resources**:

*   `power_state`: The power requirements requested by this specific FD.
*   `noc_cleanup`: Configuration for cleaning up NOC resources on close.

**Usage Example**: When the driver aggregates power state across all clients, it iterates `open_fds_list` while holding `chardev_mutex`, and then locks each `priv->mutex` to safely read the individual power state [chardev.c 384-434](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L434)

Sources: [chardev.c 384-434](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L434)[chardev.c 639](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L639-L639)

### vma_lock - VMA List Mutex

**Purpose**: Protects the list of Virtual Memory Areas (VMAs) associated with a file descriptor.

**Type**: `struct mutex vma_lock` in `struct chardev_private`[chardev.c 645](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L645-L645)

**Usage**: This mutex is used to protect the `vma_list` which tracks all active `mmap` regions. During a device reset, the driver must "zap" (invalidate) these mappings. The lock ensures that the list is not modified (by a concurrent `munmap` or `mmap`) while the reset thread is iterating through it to clear page tables.

Sources: [chardev.c 644-645](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L644-L645)

## Wait Queues

### resource_lock_waitqueue

**Purpose**: Facilitates blocking acquisition of user-space resource locks.

**Type**: `wait_queue_head_t resource_lock_waitqueue` declared in [device.h 49](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L49-L49)

**Usage Pattern**: When a user-space process requests a resource lock via `TENSTORRENT_IOCTL_LOCK_CTL` with the `ACQUIRE_BLOCKING` flag, the driver uses `wait_event_interruptible` on this queue [chardev.c 296-300](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L296-L300) When a lock is released (either explicitly or via FD closure), `wake_up_interruptible` is called to notify pending waiters [chardev.c 305](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L305-L305)[chardev.c 707](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L707-L707)

## Synchronization Patterns

### Lock Hierarchy and Ordering

To prevent deadlocks, the driver follows a strict lock acquisition order.

**Entity-to-Primitive Hierarchy**

| Order | Primitive | Scope | Notes |
| --- | --- | --- | --- |
| 1 | `reset_rwsem` | Device | Must be held for any hardware access. |
| 2 | `chardev_mutex` | Device | Held for list management and open/close tracking. |
| 3 | `iatu_mutex` | Device | Held for hardware address translation setup. |
| 4 | `priv->mutex` | Per-FD | Protects individual client state. |
| 5 | `priv->vma_lock` | Per-FD | Protects mapping lists. |

**Special Case**: `mmap()` operations must use `down_read_trylock()` on `reset_rwsem` because the kernel already holds `mmap_lock` before calling the driver's mmap file operation [chardev.c 587](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L587-L587)

Sources: [chardev.c 472-475](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L472-L475)[chardev.c 587](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L587-L587)[chardev.c 661](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L661-L661)

### Generation-Based Synchronization (`reset_gen`)

The driver uses an atomic generation counter to invalidate stale file descriptors after a hardware reset.

1.   **Increment**: During a reset operation (e.g., `TENSTORRENT_RESET_DEVICE_ASIC_RESET`), the `reset_gen` is incremented while holding the `reset_rwsem` write lock [chardev.c 232-234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L232-L234)
2.   **Capture**: When an FD is opened, the current `reset_gen` is saved in `chardev_private`[chardev.c 649](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L649-L649)
3.   **Validation**: Every IOCTL (except those specifically allowed after reset) compares the FD's captured generation against the device's current generation [chardev.c 484-487](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L484-L487) If they differ, the FD is considered stale, and the IOCTL returns `-ENODEV`.

Sources: [device.h 34](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L34-L34)[chardev.c 232](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L232-L232)[chardev.c 484-487](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L484-L487)[chardev.c 649](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L649-L649)

### Deferred Work Synchronization

The driver supports deferred idle power-down via `power_down_work`[device.h 61](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L61-L61)

*   **Arming**: Armed in `tt_cdev_release` when the last FD is closed [chardev.c 719](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L719-L719)
*   **Cancellation**: Must be cancelled using `cancel_delayed_work_sync` during device reset or removal to prevent the ARC firmware message from racing with hardware cleanup [chardev.c 210](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L210-L210)
