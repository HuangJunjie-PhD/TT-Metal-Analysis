## Purpose and Scope

This document explains the guard blocklist management system, which prevents specific operations with problematic input variations from being lowered to TTNN operations during compilation. The guard system acts as a safety mechanism that blocks known-failing or unsupported operation patterns before they reach hardware execution. For information about the overall operation lowering process, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)). For details on how conversion status is tracked and reported, see [Input Variation Analysis and Conversion Status](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.2-input-variation-analysis-and-conversion-status).

## System Overview

The guard blocklist management system provides fine-grained control over which operations are allowed to lower to TTNN. Rather than blocking entire operation types, the system blocks specific input signatures that are known to cause failures, out-of-memory errors, or produce incorrect results. This enables partial support for operations where some input patterns work correctly while others do not.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard.py 1-114](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L1-L114)[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 1-328](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L1-L328)[torch_ttnn/passes/constant_folding_pass.py 51-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L53)

## Guard Blocklist Structure

Blocklists are organized as Python lists of input signature strings, grouped by operation. Each blocklist entry represents a specific combination of tensor shapes, data types, and parameter values that should not be lowered.

### Blocklist Format

Each entry in a blocklist is a list of strings representing the full input signature. The format follows the pattern from `InputVariation` serialization:

*   **Tensors**: `Tensor<[shape]> param_name = ?`
*   **Scalars**: `type value = value` (e.g., `int dim = 1`)
*   **Lists**: `List[type] name = [values]`
*   **Optional values**: `Optional[type]<[shape]> name = ?`

**Example from autogenerated file:**

[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 12-28](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L12-L28)

| Operation | Blocklist Name | Entry Count | Purpose |
| --- | --- | --- | --- |
| `aten._log_softmax.default` | `aten__log_softmax_default_blocklist` | 1 | Blocks specific large tensor dimensions |
| `aten.native_layer_norm.default` | `aten_native_layer_norm_default_blocklist` | 2 | Blocks specific normalized shape configurations |
| `aten.convolution.default` | `aten_convolution_default_blocklist` | 22 | Blocks OOM cases and large spatial dimensions |
| `aten.argmax.default` | `aten_argmax_default_blocklist` | 3 | Blocks unsupported dtype or dimension combinations |

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 12-301](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L12-L301)

## Autogenerated vs Manual Blocklists

The system maintains two types of blocklists: autogenerated entries derived from test metrics, and manual entries added by developers for specific known issues.

### Autogenerated Blocklists

The file `to_tt_guard_autogen.py` is automatically generated by `tools/generate_guard_function_from_input_var_metric.py` and should not be manually edited. The header clearly states:

```
# THIS FILE IS AUTOGENERATED BY tools/generate_guard_function_from_input_var_metric.py
# DO NOT EDIT THIS FILE MANUALLY
# YOU CAN DEBUG THESE INPUT VAR VIA tests/autogen-op/ALL
```

**Key characteristics:**

*   Generated from metrics collected during test runs
*   Contains input variations that failed in model tests
*   Organized by operation with blocklist variables named `aten_<op_name>_blocklist`
*   Includes helper function `get_inputs(node)` to extract input strings from nodes
*   Provides `guard_aten(blocklist, node)` function for blocklist checking

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 1-328](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L1-L328)

### Manual Blocklists

The file `to_tt_guard.py` imports autogenerated blocklists and adds manual entries for known issues discovered during development. These manual entries include:

**Categories of manual blocks:**

1.   **Type casting errors** (Whisper, GPTNeo, codegen, OPT models):

    *   `RuntimeError: aten::clone() Expected a value of type 'Tensor' for argument 'self' but instead found type 'SymInt'`
    *   Example: [torch_ttnn/passes/lowering/to_tt_guard.py 36-44](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L36-L44)

2.   **Scalar handling issues**:

    *   `ttnn.from_torch` not supporting boolean or float scalars
    *   Example: `aten_gt_Scalar_blocklist`[torch_ttnn/passes/lowering/to_tt_guard.py 36](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L36-L36)

3.   **Kernel allocation failures**:

    *   Out of L1 memory errors
    *   Example: `aten_aten_stack_default`[torch_ttnn/passes/lowering/to_tt_guard.py 85-87](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L85-L87)

4.   **Unsupported storage types**:

    *   Operations with specific keepdim or dtype combinations
    *   Example: argmax with keepdim=True [torch_ttnn/passes/lowering/to_tt_guard.py 94](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L94-L94)

The guard system centers on the `GUARD` dictionary and the `can_lowering_to_ttnn` function, which together determine whether an operation can be safely lowered.

## Integration with Compilation Passes

Guard checks are integrated at critical points in the compilation pipeline to prevent problematic operations from lowering.

### Usage in ConstantFoldingPass

The `ConstantFoldingPass` checks guards before folding constant operations:

[torch_ttnn/passes/constant_folding_pass.py 51-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L53)

The `_can_fold` method verifies:

1.   All arguments are constants or simple values
2.   The operation passes the guard check via `can_lowering_to_ttnn(node)`

This prevents folding operations that would later fail during lowering.

### Usage in ToTtPass

While not explicitly shown in the provided files, the `ToTtPass` uses guards when deciding whether to convert ATen operations to TTNN equivalents. Operations that fail the guard check remain as ATen operations, effectively falling back to CPU execution.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 51-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L53)[torch_ttnn/passes/lowering/to_tt_guard.py 109-114](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L109-L114)

The guard blocklist system operates within a feedback loop that captures test failures and automatically generates appropriate blocklist entries.

### Metrics Collection Process

When operations are executed during model tests, the metrics system captures their input variations:

1.   **During compilation**: `InputVariation` objects are created for each operation via `collect_input_variation_from_node(node)`[torch_ttnn/metrics.py 170-191](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L170-L191)

2.   **After conversion**: If an operation converts to TTNN, a `ConvertedInput` object wraps the original variation [torch_ttnn/metrics.py 107-117](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L107-L117)

3.   **Serialization**: Input variations are serialized to pickle files in `metrics/{model}/original-schema_list.pickle` and `metrics/{model}/compiled-schema_list.pickle`

### Status Determination

The `collect_metrics.py` script processes these pickle files and determines conversion status:

[tools/collect_metrics.py 151-159](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L151-L159)

The `InputVarPerOp` class tracks status for each input variation:

| Status | Meaning | Next Action |
| --- | --- | --- |
| `DONE` | Successfully converted to TTNN | None - working correctly |
| `FALLBACK` | Partially converted (aten→aten) | Investigate if full conversion possible |
| `BUG` | Known issue with conversion | Add to manual blocklist if needed |
| `NONE` | No conversion attempted | Check if guard is blocking it |
| `REMOVED` | Operation removed from graph | None - intentionally optimized away |
| `UNKNOWN` | Status could not be determined | Investigate metrics collection |

## Adding New Blocklist Entries

Developers can add new blocklist entries in two ways: through the autogenerated process or by manual addition.

### Adding Entries via Autogeneration

The autogeneration process should be used for entries discovered through systematic model testing:

1.   **Run model tests** that fail with specific input patterns
2.   **Collect metrics** - metrics are automatically captured during test execution
3.   **Run generation script**: `python tools/generate_guard_function_from_input_var_metric.py`
4.   **Review changes** to `to_tt_guard_autogen.py`
5.   **Commit both** the metrics changes and the regenerated guard file

### Adding Manual Entries

Manual entries should be added to `to_tt_guard.py` when:

*   The issue affects multiple models with similar patterns
*   The root cause is a known limitation in TTNN
*   The failure is systematic rather than input-specific

**Steps for manual additions:**

1.   **Create the blocklist variable**:

`aten_operation_name_blocklist = [    ["Tensor<[shape]> param_name = ?", "int param = value"],]`
2.   **Add to GUARD dictionary**:

`GUARD[torch.ops.aten.operation.variant] = partial(guard_aten, aten_operation_name_blocklist)`
3.   **Document the reason** with comments explaining:

    *   Which models trigger this issue
    *   What error occurs (with error message)
    *   Link to tracking issue if applicable

### Testing Blocklist Changes

After adding blocklist entries:

1.   **Verify the operation is blocked**:

    *   Run the previously failing test
    *   Confirm operation falls back to PyTorch instead of crashing

2.   **Check conversion metrics**:

    *   Run `python tools/collect_metrics.py`
    *   Verify the operation shows as `NONE` or `FALLBACK` in conversion status

3.   **Test isolated operation** (if autogen tests exist):

`pytest tests/autogen-op/ALL/test_<operation>.py`
