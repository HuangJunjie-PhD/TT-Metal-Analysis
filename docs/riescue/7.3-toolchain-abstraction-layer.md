# Toolchain Abstraction Layer

Relevant source files
*   [riescue/lib/toolchain/tool.py](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py)

This document describes the toolchain abstraction layer that provides a unified Python interface for invoking external RISC-V tools. The abstraction layer handles executable discovery, subprocess management, error classification, and command-line integration.

The toolchain abstraction is implemented in `riescue.lib.toolchain` and provides wrapper classes for:

*   RISC-V GNU toolchain components (compiler, disassembler, objcopy)
*   Instruction Set Simulators (Spike, tt_spike)

For documentation of the underlying external tools themselves, see [RISC-V Toolchain Integration](https://deepwiki.com/tenstorrent/riescue/4.1-risc-v-toolchain-integration) and [Instruction Set Simulators](https://deepwiki.com/tenstorrent/riescue/4.2-instruction-set-simulators). For documentation of tool subclass implementations, see [Tool Subclasses](https://deepwiki.com/tenstorrent/riescue/7.3.2-tool-subclasses). For broader toolchain integration architecture, see [Toolchain and External Dependencies](https://deepwiki.com/tenstorrent/riescue/4-toolchain-and-external-dependencies).

## Tool Base Class Architecture

The `Tool` base class [riescue/lib/toolchain/tool.py 18-126](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L18-L126) provides the foundation for all external tool wrappers. It is an abstract base class (ABC) that defines the common interface and behavior for executable invocation.

### Class Hierarchy

**Sources:**[riescue/lib/toolchain/tool.py 18-335](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L18-L335)

### Key Responsibilities

The `Tool` class handles:

| Responsibility | Method | Description |
| --- | --- | --- |
| Executable Location | `find_executable()` | Three-tier discovery: explicit path → environment variable → PATH search |
| File Resolution | `check_filepath()` | Resolves data files relative to CWD or package installation |
| Subprocess Management | `run()` | Executes tool with arguments, timeout, and output redirection |
| Error Handling | `_classify()` | Abstract method for tool-specific error detection |
| Error Reporting | `_raise_toolchain_error()` | Standardized exception raising with context |

**Sources:**[riescue/lib/toolchain/tool.py 18-126](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L18-L126)

## Executable Discovery Mechanism

The `find_executable()` method implements a three-tier fallback mechanism for locating external tools:

**Sources:**[riescue/lib/toolchain/tool.py 36-55](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L36-L55)

### Discovery Priority

1.   **Explicit Path**: If `tool_path` is provided and exists, use it immediately
2.   **Environment Variable**: Check for tool-specific environment variable (e.g., `RV_GCC`, `SPIKE_PATH`)
3.   **PATH Search**: Use `shutil.which()` to search system PATH for the tool name

This design allows flexibility for different deployment scenarios:

*   Development: Tools in non-standard locations via environment variables
*   Container: Tools installed to standard locations found via PATH
*   Testing: Explicit paths for specific tool versions

### Environment Variables by Tool

| Tool Class | Environment Variable | Default Tool Name |
| --- | --- | --- |
| `Compiler` | `RV_GCC` | `riscv64-unknown-elf-gcc` |
| `Disassembler` | `RV_OBJDUMP` | `riscv64-unknown-elf-objdump` |
| `Objcopy` | `RV_OBJCOPY` | `riscv64-unknown-elf-objcopy` |
| `Spike` | `SPIKE_PATH` | `tt_spike` or `spike` |

## File Path Resolution

The `check_filepath()` method provides a flexible mechanism for locating data files that works across both repository and package installations:

The `package_path` class variable [riescue/lib/toolchain/tool.py 29](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L29-L29) is set to `Path(__file__).parents[2].resolve()`, which resolves to the `riescue/` directory. This allows data files to be specified relative to the package root, enabling the same configuration files to work whether RiESCUE is:

*   Installed as a Python package (pip install)
*   Used from a cloned repository
*   Run from an arbitrary working directory

**Sources:**[riescue/lib/toolchain/tool.py 57-72](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L57-L72)

## Error Handling and Classification

Tool-specific error classification is implemented via the abstract `_classify()` method, which each subclass must implement to detect and report tool-specific failure modes.

### ToolchainError Exception

The base class provides `_raise_toolchain_error()`[riescue/lib/toolchain/tool.py 110-114](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L110-L114) to standardize exception raising. This method wraps errors in `ToolchainError` exceptions (defined in `riescue.lib.toolchain.exceptions`) with structured metadata:

`raise ToolchainError(    tool_name=self.__class__.__name__,    cmd=process.args,    kind=ToolFailureType,  # enum: COMPILE_FAILURE, SEGFAULT, NONZERO_EXIT    returncode=process.returncode,    error_text=error_text,    **kwargs)`
**Sources:**[riescue/lib/toolchain/tool.py 110-114](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L110-L114)

## Tool Subclasses

### Compiler

The `Compiler` class [riescue/lib/toolchain/tool.py 128-204](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L128-L204) wraps `riscv64-unknown-elf-gcc` with RiESCUE-specific defaults.

**Default Configuration:**

*   `-static`: Static linking
*   `-mcmodel=medany`: Medium any code model
*   `-fvisibility=hidden`: Hidden symbol visibility
*   `-nostdlib`, `-nostartfiles`: Bare-metal compilation
*   `-march`: ISA string (default: `rv64imafdcvh_svinval_zfh_zba_zbb_zbc_zbs_zifencei_zicsr_zvkned_zicbom_zicbop_zicboz_zawrs_zihintpause_zvbb1_zicond_zvkg_zvkn_zvbc_zfa_zk`)

**Key Features:**

| Feature | Parameter | Purpose |
| --- | --- | --- |
| ISA String | `compiler_march` | Override default ISA extensions |
| ABI | `abi` | Set calling convention (e.g., `lp64d`) |
| Test Equates | `test_equates` | Define preprocessor macros (`-D<name>=<value>`) |
| Extra Options | `compiler_opts` | Additional GCC flags |

**Sources:**[riescue/lib/toolchain/tool.py 128-204](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L128-L204)

### Spike

The `Spike` class [riescue/lib/toolchain/tool.py 273-335](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L273-L335) wraps the Spike ISS with special handling for both third-party Spike and Tenstorrent's `tt_spike`.

**Configuration Differences:**

| Configuration | Third-Party Spike | tt_spike |
| --- | --- | --- |
| Tool Name | `spike` | `tt_spike` |
| Default ISA | `RV64IMAFDCVH_zba_zbb_zbc_zfh_zbs_zfbfmin_zvfh_zvbb_zvbc_zvfbfmin_zvfbfwma_zvkg_zvkned_zvl256b_zve64d_svpbmt` | `RV64IMAFDCVH_zba_zbb_zbc_zfh_zbs_zfbfmin_zvfh_zvbb_zvbc_zvfbfmin_zvfbfwma_zvkg_zvkned_zvknhb_svpbmt_sstc_zicntr` |
| Vector Arch | Not set | `vlen:256,elen:64` |
| Max Instructions | Not set | `--max-instrs=2000000` (default) |

**Special Execution Method:**

Spike overrides `run()` to raise `NotImplementedError`[riescue/lib/toolchain/tool.py 321-322](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L321-L322) Instead, callers must use `run_iss()`[riescue/lib/toolchain/tool.py 324-334](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L324-L334) which:

1.   Takes an `elf_file` parameter
2.   Automatically appends `--isa=<spike_isa>`
3.   Appends `--varch=vlen:<vlen>,elen:<elen>` for tt_spike
4.   Appends the ELF file path to arguments

This design ensures ISA and architecture parameters are always correctly specified.

**Sources:**[riescue/lib/toolchain/tool.py 273-335](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L273-L335)

## Command-Line Integration

Each tool class provides static methods for argparse integration, following a consistent pattern:

**Sources:**[riescue/lib/toolchain/tool.py 169-178](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L169-L178)[riescue/lib/toolchain/tool.py 217-222](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L217-L222)[riescue/lib/toolchain/tool.py 256-261](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L256-L261)[riescue/lib/toolchain/tool.py 304-311](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L304-L311)

### Pattern Implementation

Each tool implements:

| Method | Type | Purpose |
| --- | --- | --- |
| `add_arguments(parser)` | `@staticmethod` | Register argparse arguments in an argument group |
| `from_clargs(args)` | `@classmethod` | Construct tool instance from parsed arguments |

This pattern enables:

*   **Composability**: Multiple tools can add arguments to the same parser without conflicts
*   **Documentation**: Argument groups organize help text by tool
*   **Consistency**: Standard naming conventions (e.g., `--<tool>_path`, `--<tool>_opts`)

### Example: Compiler Arguments

```
Compiler:
  --compiler_path PATH          Path to compiler executable
  --compiler_opts [OPTS ...]    Additional args to pass to compiler
  --compiler_march STR          march to pass to compiler
  --compiler_mabi STR           ABI to use for compiler
  --test_equates KEY=VALUE      Variables to define with -D
  --config_json PATH            Path to config.json
```

**Sources:**[riescue/lib/toolchain/tool.py 169-178](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L169-L178)
