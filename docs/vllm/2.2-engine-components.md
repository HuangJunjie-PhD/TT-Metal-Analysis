This page documents vLLM's V1 engine components. The V1 architecture is the current default (`VLLM_USE_V1=True`) and consists of several layers that handle text generation from request intake to token output.

## Component Overview

The V1 engine is organized into the following layers:

| Layer | Components | Purpose |
| --- | --- | --- |
| **API Layer** | `LLM`, `AsyncLLM` | User-facing interfaces for offline and online inference |
| **Processing Layer** | `Processor`, `OutputProcessor` | Input validation/preprocessing and output assembly |
| **Core Layer** | `EngineCore`, `EngineCoreClient` | Scheduling, resource management, IPC abstraction |
| **Execution Layer** | `Executor`, `Worker`, `ModelRunner` | Model execution and worker management |

## LLM and LLMEngine

### LLM Class (Offline Interface)

The `LLM` class in [vllm/entrypoints/llm.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/llm.py) provides a synchronous interface for offline batch inference. It creates an `LLMEngine` internally and provides methods like `generate()`, `chat()`, `encode()`, `score()`, and `classify()`.

Sources: [vllm/entrypoints/llm.py 167-343](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/llm.py#L167-L343)[vllm/v1/engine/llm_engine.py 46-138](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/llm_engine.py#L46-L138)

### LLMEngine Structure

The V1 `LLMEngine` class in [vllm/v1/engine/llm_engine.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/llm_engine.py) is a synchronous wrapper that:

*   Holds `Processor` for input validation/preprocessing
*   Holds `OutputProcessor` for output assembly
*   Holds `EngineCoreClient` for communication with the core
*   Provides `step()` method for synchronous execution

Key methods:

*   `generate()`: Generates text from prompts
*   `add_request()`: Adds a request via the processor
*   `abort_requests()`: Aborts requests by ID
*   `step()`: Gets outputs from engine core and processes them

Sources: [vllm/v1/engine/llm_engine.py 46-250](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/llm_engine.py#L46-L250)

## AsyncLLM (Online Interface)

The `AsyncLLM` class in [vllm/v1/engine/async_llm.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/async_llm.py) provides an asynchronous interface for online serving. Unlike the V0 implementation, V1's `AsyncLLM` uses multiprocessing by default with the `EngineCore` running in a separate process.

Key components:

*   **Processor**: Validates inputs and creates `EngineCoreRequest` objects
*   **EngineCoreClient**: Handles IPC with the engine process (see [EngineCoreClient section](https://github.com/tenstorrent/vllm/blob/cd9e5b83/EngineCoreClient%20section))
*   **OutputProcessor**: Assembles outputs from `EngineCoreOutputs`
*   **Output Handler Task**: Asyncio task that continuously pulls outputs

The `AsyncLLM` constructor parameters:

*   `vllm_config`: Configuration object
*   `executor_class`: Executor implementation class
*   `log_stats`: Whether to log statistics
*   `client_addresses`: Optional dict for external load balancing
*   `client_count`/`client_index`: For data parallel setups

Sources: [vllm/v1/engine/async_llm.py 52-242](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/async_llm.py#L52-L242)

## Processor (Input Layer)

The `Processor` class in [vllm/v1/engine/processor.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/processor.py) validates and preprocesses inputs before they reach the engine core.

**Key responsibilities:**

1.   **Input validation**: Validates `SamplingParams`, `PoolingParams`, structured output constraints
2.   **Tokenization**: Tokenizes prompts if needed
3.   **Multi-modal processing**: Processes images, audio, video through `InputPreprocessor`
4.   **Request construction**: Creates `EngineCoreRequest` objects with validated inputs

The `process_inputs()` method flow:

1.   Validate parameters (logprobs, sampling params, LoRA, etc.)
2.   Process multi-modal data and assign UUIDs
3.   Tokenize prompts
4.   Create `EngineCoreRequest` with arrival time and validated data

Sources: [vllm/v1/engine/processor.py 36-450](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/processor.py#L36-L450)

## EngineCore (Orchestration Layer)

The `EngineCore` class in [vllm/v1/engine/core.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py) is the inner loop that orchestrates scheduling and execution. It runs in its own process when using multiprocess mode.

**Key responsibilities:**

1.   **Request management**: Adds/aborts requests in the scheduler
2.   **Scheduling**: Calls `scheduler.schedule()` each iteration
3.   **Execution**: Calls `model_executor.execute_model()`
4.   **Output processing**: Calls `scheduler.update_from_output()`
5.   **KV cache initialization**: Profiles memory and initializes cache

**Step execution flow:**

**Batch queue mode**: When `batch_queue_size > 1`, the engine supports pipeline parallelism by scheduling multiple batches asynchronously. The `step_with_batch_queue()` method manages this queue.

Sources: [vllm/v1/engine/core.py 64-335](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py#L64-L335)

## EngineCoreClient (IPC Abstraction)

The `EngineCoreClient` in [vllm/v1/engine/core_client.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core_client.py) provides an abstraction layer for communicating with the `EngineCore`. It has three implementations:

| Implementation | Use Case | IPC Method |
| --- | --- | --- |
| `InprocClient` | Single process, debugging | Direct method calls |
| `SyncMPClient` | Synchronous multiprocess | ZeroMQ DEALER/PULL sockets |
| `AsyncMPClient` | Asynchronous multiprocess | ZeroMQ asyncio sockets |

**Data parallel variants:**

*   `DPAsyncMPClient`: Connects to a single DP rank (external load balancer mode)
*   `DPLBAsyncMPClient`: Balances requests across all DP ranks (internal load balancer mode)

The client uses `MsgpackEncoder`/`MsgpackDecoder` for efficient serialization.

Sources: [vllm/v1/engine/core_client.py 49-242](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core_client.py#L49-L242)

## EngineCoreProc (Process Management)

When using multiprocess mode, the `EngineCoreProc` class in [vllm/v1/engine/core.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py) manages the engine process lifecycle:

**Key components:**

*   **Input thread**: Receives requests via ZMQ ROUTER socket
*   **Output thread**: Sends outputs via ZMQ PUSH socket
*   **Main loop**: Busy loop that calls `engine_core.step()` and checks for requests
*   **Handshake**: Sends metadata to client on startup

The process uses a busy loop (`POLLING_TIMEOUT_S`) to minimize latency.

Sources: [vllm/v1/engine/core.py 378-615](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py#L378-L615)

## OutputProcessor (Output Assembly)

The `OutputProcessor` class in [vllm/v1/engine/output_processor.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/output_processor.py) assembles outputs from the engine core into `RequestOutput` objects.

**Key responsibilities:**

1.   **Detokenization**: Converts token IDs to text using `Detokenizer`
2.   **Output aggregation**: Collects outputs across multiple iterations
3.   **Metrics tracking**: Records completion stats, finish reasons
4.   **Parallel sampling**: Handles `n > 1` sampling via `ParentRequest`

The processor maintains a `RequestOutputCollector` for each active request, which accumulates outputs until the request finishes.

Sources: [vllm/v1/engine/output_processor.py 86-393](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/output_processor.py#L86-L393)

## Environment Variables for Engine Configuration

vLLM's engine behavior can be configured through various environment variables:

| Environment Variable | Description | Default |
| --- | --- | --- |
| `VLLM_USE_V1` | Whether to use V1 engine architecture | `True` |
| `VLLM_ATTENTION_BACKEND` | Backend for attention computation (FLASHINFER, FLASH_ATTN, etc.) | Auto-detected |
| `VLLM_ENGINE_ITERATION_TIMEOUT_S` | Timeout for each engine iteration in seconds | 60 |
| `VLLM_USE_ASYNC_OUTPUT_PROC` | Whether to use async output processor | `True` |
| `VLLM_WORKER_MULTIPROC_METHOD` | Method for worker multiprocessing (fork/spawn) | `"fork"` |

These environment variables can be used to fine-tune the engine's behavior without changing code.

Sources: [vllm/envs.py 11-110](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/envs.py#L11-L110)

## Conclusion

vLLM's engine components form a sophisticated system for efficient large language model inference. The LLMEngine orchestrates the process, using the Scheduler to manage resource allocation, the Executor to communicate with Workers, and various supporting components to handle tokenization, output processing, and more.

This architecture enables vLLM to deliver high-performance inference with features like:

1.   **Efficient memory management** through block allocation and swapping
2.   **Parallel execution** across multiple GPUs and nodes
3.   **Continuous batching** to maximize throughput
4.   **Asynchronous API** for online serving

The V1 architecture further improves on this design by separating concerns and providing better isolation between components.
