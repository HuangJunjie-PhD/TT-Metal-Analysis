## Purpose and Scope

This document covers vLLM's support for non-CUDA/ROCm hardware platforms: CPU, TPU (Tensor Processing Units), and XPU (Intel Data Center GPUs). These platforms extend vLLM's reach beyond NVIDIA and AMD GPUs, enabling inference on diverse hardware ecosystems including edge devices, cloud TPUs, and Intel accelerators.

For the platform abstraction layer that enables this multi-hardware support, see [Platform Abstraction Layer](https://deepwiki.com/tenstorrent/vllm/5.1-platform-abstraction-layer). For CUDA and ROCm GPU platforms, see [CUDA and ROCm Support](https://deepwiki.com/tenstorrent/vllm/5.2-cuda-and-rocm-support).

## Platform Detection and Registration

vLLM automatically detects available platforms through a plugin system at import time. Each platform provides a detection function that checks for hardware availability and required dependencies.

**Platform Detection Logic**

The detection functions verify hardware availability before activating a platform:

| Platform | Detection Method | Key Dependencies |
| --- | --- | --- |
| **TPU** | `libtpu` import or `VLLM_TPU_USING_PATHWAYS` env var | `libtpu`, `torch_xla`, `tpu_info` |
| **CPU** | `vllm` version contains "cpu" substring or Darwin OS | None (always available on CPU builds) |
| **XPU** | `torch.xpu.is_available()` | `intel_extension_for_pytorch`, `oneccl_bindings_for_pytorch` |

## CPU Platform

The `CpuPlatform` enables vLLM inference on x86, ARM, PowerPC, RISC-V, and other CPU architectures. It's the fallback platform for systems without GPU accelerators and is commonly used on MacOS or in CPU-only deployments.

### Architecture and Device Configuration

**Sources:**[vllm/platforms/cpu.py 68-89](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L68-L89)[vllm/platforms/interface.py 334-352](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/interface.py#L334-L352)

### Configuration and Environment Setup

The `check_and_update_config()` method enforces CPU-specific requirements and sets up the execution environment:

**Key Configuration Details:**

1.   **Block Size:** 128 with IPEX, 16 without (required for CPU-optimized kernels)
2.   **Worker Class:**`vllm.v1.worker.cpu_worker.CPUWorker`
3.   **Distributed Backend:**`mp` (multiprocessing) with `spawn` method
4.   **Compilation:** DYNAMO_ONCE with inductor backend (eager for CI tests)
5.   **FP8 Support:**`fp8_e5m2` only (not `fp8_e4m3`)
6.   **KV Cache Space:** Configurable via `VLLM_CPU_KVCACHE_SPACE` (default 4 GiB)

**Sources:**[vllm/platforms/cpu.py 140-267](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L140-L267)

### NUMA and CPU Topology

For multi-socket CPU systems, vLLM can query NUMA topology to optimize worker placement:

**Sources:**[vllm/platforms/cpu.py 39-66](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L39-L66)[vllm/platforms/cpu.py 279-315](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L279-L315)

### Attention Backend

CPU platform exclusively uses PyTorch's native Scaled Dot-Product Attention (SDPA):

**Attention Backend Selection:**

*   **V1 Engine (required):**`vllm.v1.attention.backends.cpu_attn.TorchSDPABackend`
*   **MLA:** Not supported (raises `NotImplementedError`)
*   **Sparse Attention:** Not supported

**Sources:**[vllm/platforms/cpu.py 95-110](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cpu.py#L95-L110)

## TPU Platform

The `TpuPlatform` enables vLLM on Google Cloud TPUs using JAX/XLA compilation. TPUs are optimized for large-scale training and inference with high memory bandwidth.

### TPU Architecture and Characteristics

**Sources:**[vllm/platforms/tpu.py 32-102](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/tpu.py#L32-L102)

### TPU Configuration Requirements

**Key TPU-Specific Constraints:**

1.   **Compilation:** Must use `DYNAMO_ONCE` level with `openxla` backend
2.   **Data Types:** Prefers `bfloat16` (automatically converts fp16/fp32)
3.   **CUDA Graphs:** Not supported (disabled)
4.   **Speculative Decoding:** Not supported
5.   **Multimodal:** Requires `disable_chunked_mm_input = True`
6.   **Worker Class:**`vllm.v1.worker.tpu_worker.TPUWorker`
7.   **Block Size:** Computed by `PallasAttentionBackend.get_page_size()`

**Sources:**[vllm/platforms/tpu.py 104-167](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/tpu.py#L104-L167)

## XPU Platform

The `XPUPlatform` enables vLLM on Intel Data Center GPUs (formerly Xe-HPG/Xe-HPC), including Ponte Vecchio and future Intel accelerators. XPU uses Intel's oneAPI stack with extensions for PyTorch.

### XPU Architecture and Device Configuration

**Sources:**[vllm/platforms/xpu.py 26-247](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/xpu.py#L26-L247)

### XPU Configuration and Setup

**Key XPU Configuration Points:**

1.   **Block Size:** Fixed at 64 (optimized for XPU kernels)
2.   **Worker Class:**`vllm.v1.worker.xpu_worker.XPUWorker`
3.   **Distributed Backend:** Prefers `ray`, supports `mp` with `spawn` method
4.   **CUDA Graphs:** Not supported (must be NONE)
5.   **KV Cache Layout:** Must use `NHD` (not `HND`)
6.   **Multiprocess Method:** Must use `spawn` (not `fork`)
7.   **Compilation:** No static compile sizes, LoRA disables compilation

**Sources:**[vllm/platforms/xpu.py 112-171](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/xpu.py#L112-L171)

## Platform Comparison

The following table summarizes key differences between CPU, TPU, and XPU platforms:

| Feature | CPU | TPU | XPU |
| --- | --- | --- | --- |
| **Device Type** | `cpu` | `tpu` | `xpu` |
| **Dispatch Key** | `CPU` | `XLA` | `XPU` |
| **Distributed Backend** | `gloo` | `gloo` | `ccl` / `xccl` |
| **Attention Backend** | TorchSDPA | Pallas | FlashAttention / Triton |
| **Worker Class** | `CPUWorker` | `TPUWorker` | `XPUWorker` |
| **Default Block Size** | 128 (IPEX) / 16 | 16 (computed) | 64 |
| **KV Cache Layout** | Default | Default | `NHD` (required) |
| **Compilation Backend** | `inductor` | `openxla` | `inductor` |
| **CUDA Graphs** | Not supported | Not supported | Not supported |
| **MLA Support** | No | No (with workarounds) | No (with workarounds) |
| **Sparse Attention** | No | No | No |
| **FP8 Support** | `fp8_e5m2` only | Yes | Triton backend only |
| **Inference Mode** | `torch.no_grad()` | `torch.no_grad()` | `torch.no_grad()` |
| **Pin Memory** | No | No | Yes |
| **Hybrid KV Cache** | Yes | No | Yes |
| **Inplace Updates** | Yes | No | Yes |

## Troubleshooting

### Common CPU Issues

**Issue:**`RuntimeError: --block-size=128 requires intel_extension_for_pytorch`

*   **Solution:** Install IPEX (`pip install intel-extension-for-pytorch`) or use `--block-size=16`

**Issue:** FP8 KV cache fails

*   **Solution:** CPU only supports `fp8_e5m2`, not `fp8_e4m3`. Use `--kv-cache-dtype=fp8_e5m2`

**Issue:** Distributed execution fails

*   **Solution:** CPU requires `mp` backend with `spawn` method. Set `VLLM_WORKER_MULTIPROC_METHOD=spawn`

### Common TPU Issues

**Issue:**`ValueError: TPU backend only supports V1`

*   **Solution:** Set `VLLM_USE_V1=1` environment variable

**Issue:** Compilation errors with multimodal models

*   **Solution:** Add `--disable_chunked_mm_input` flag

**Issue:**`AssertionError: TPU does not support speculative decoding`

*   **Solution:** Remove speculative decoding configuration

### Common XPU Issues

**Issue:**`AssertionError: CUDA graph mode should be NONE on XPU`

*   **Solution:** CUDA graphs not supported. Remove `--cudagraph-capture-sizes` or similar flags

**Issue:** KV cache layout error

*   **Solution:** XPU only supports `NHD` layout (automatically set, but check for overrides)

**Issue:** bfloat16 accuracy issues on Intel Arc A770

*   **Solution:** Use `--dtype=half` instead of bfloat16 for client GPUs
