This document describes the distribution channels through which TT-SMI packages are published and made available to end users. It covers PyPI publishing (both test and production), Debian repository distribution, GitHub release assets, and the automated workflows that manage these publications.

For information about the source package configuration, see [Package Configuration (pyproject.toml)](https://deepwiki.com/tenstorrent/tt-smi/7.1-package-configuration-(pyproject.toml)). For details on the release automation that triggers distribution, see [GitHub Release Workflow](https://deepwiki.com/tenstorrent/tt-smi/7.2-github-release-workflow). For Debian package building specifics, see [Debian Package Building](https://deepwiki.com/tenstorrent/tt-smi/7.3-debian-package-building).

## PyPI Distribution

### TestPyPI Validation

Before publishing to production PyPI, TT-SMI artifacts are first published to TestPyPI for validation. This allows verification that the package metadata, dependencies, and installation process work correctly without affecting the production repository.

The TestPyPI publication workflow is defined in the `publish-to-testpypi` job:

The workflow downloads artifacts matching the pattern `tt-smi*.whl` and publishes them to TestPyPI using trusted publishing (no API tokens required). The publication is configured at [.github/workflows/release.yml 391-419](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L391-L419):

| Configuration | Value |
| --- | --- |
| Job name | `publish-to-testpypi` |
| Environment | `testpypi` |
| Repository URL | `https://test.pypi.org/legacy/` |
| Permission required | `id-token: write` (for trusted publishing) |
| Artifact pattern | `tt-smi*.whl` |
| Download path | `dist/` |

**Sources:**[.github/workflows/release.yml 391-419](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L391-L419)

### Production PyPI Publishing

After successful TestPyPI validation, the package is published to production PyPI. This is the primary distribution method for users installing via `pip install tt-smi`.

The PyPI publication workflow [.github/workflows/release.yml 424-452](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L424-L452) uses GitHub's trusted publishing mechanism, eliminating the need for long-lived API tokens:

| Configuration | Value |
| --- | --- |
| Job name | `publish-to-pypi` |
| Environment | `pypi` |
| Environment URL | `https://pypi.org/p/tt-smi` |
| Permission required | `id-token: write` |
| Dependencies | `build_all_depends`, `generate-release`, `publish-to-testpypi` |
| Publish action | `pypa/gh-action-pypi-publish@release/v1` |

**Sources:**[.github/workflows/release.yml 424-452](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L424-L452)[pyproject.toml 1-26](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L1-L26)

## Package Metadata and Entry Points

The PyPI package includes metadata from `pyproject.toml` that defines how the package appears in the Python Package Index and how it integrates with the Python environment:

Key metadata defined in [pyproject.toml 1-54](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L1-L54):

| Field | Value | Purpose |
| --- | --- | --- |
| `name` | `"tt-smi"` | Package name on PyPI |
| `version` | `"3.1.1"` | Current version, auto-bumped by release workflow |
| `description` | `"ncurses based hardware monitoring for Tenstorrent silicon"` | Short package description |
| `requires-python` | `">=3.10"` | Minimum Python version |
| `[project.scripts]` | `tt-smi = "tt_smi:main"` | Creates `tt-smi` executable that calls `main()` |
| `[project.urls]` | Homepage, Bug Reports, Source | Links displayed on PyPI page |

The package includes non-Python data files specified in [pyproject.toml 59-65](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L59-L65):

*   `data/version.txt` - Version information
*   `*.css` - Textual TUI stylesheets
*   `data/*/*.yaml` and `data/*/*/*.yaml` - Configuration files

**Sources:**[pyproject.toml 1-87](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L1-L87)

## Debian Repository Distribution

Debian packages (`.deb` files) are built for multiple Ubuntu distributions and made available through a Debian repository. These packages allow installation via `apt install tt-smi` and include proper system integration.

Debian packages are built by the workflow referenced in [.github/workflows/build-all.yml 34-45](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-all.yml#L34-L45) and configured to support multiple Ubuntu distributions. The packages are GPG-signed for security and include proper dependency resolution for system-level dependencies.

**Sources:**[.github/workflows/build-all.yml 34-45](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-all.yml#L34-L45)[.github/workflows/release.yml 328-344](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L328-L344)

## GitHub Release Assets

Every release publishes both wheel and Debian packages as GitHub release assets, providing a direct download option for users who prefer not to use package managers or need specific versions.

The release creation workflow [.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L288-L357) performs several operations:

1.   **Artifact Download**: Downloads all build artifacts from previous jobs
2.   **File Discovery**: Locates `.deb` and `.whl` files in the workspace [.github/workflows/release.yml 323-326](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L323-L326)
3.   **Debian Renaming**: Renames Debian packages to prevent filename conflicts [.github/workflows/release.yml 328-344](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L328-L344)
4.   **Changelog Generation**: Uses `mikepenz/release-changelog-builder-action@v5` to build changelog from commits and PRs [.github/workflows/release.yml 305-321](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L305-L321)
5.   **Release Creation**: Uses `softprops/action-gh-release@v1` to create the GitHub release [.github/workflows/release.yml 346-357](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L346-L357)

Files attached to the release:

| File Pattern | Description |
| --- | --- |
| `${{ github.workspace }}/tt-smi*/*.deb` | Debian packages from tt-smi artifacts |
| `${{ github.workspace }}/[!artifacts-]*/*.deb` | Debian packages (excluding certain artifact dirs) |
| `${{ github.workspace }}/*/*.whl` | Python wheel files |

**Sources:**[.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L288-L357)[.github/workflows/release.yml 305-344](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L305-L344)

## Trusted Publishing

Both TestPyPI and PyPI publications use OpenID Connect (OIDC) based trusted publishing, which eliminates the need for long-lived API tokens. The workflow configuration includes:

Configuration requirements for trusted publishing [.github/workflows/release.yml 404-405](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L404-L405)[.github/workflows/release.yml 439](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L439-L439):

| Element | Configuration |
| --- | --- |
| Permission | `id-token: write` (mandatory) |
| Environment | `testpypi` or `pypi` (must be configured in repository settings) |
| Action | `pypa/gh-action-pypi-publish@release/v1` |
| Repository URL | `https://test.pypi.org/legacy/` (for TestPyPI only) |

This approach provides enhanced security by using short-lived tokens that are automatically validated against the PyPI trusted publisher configuration.
