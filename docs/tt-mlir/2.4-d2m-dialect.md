The D2M (Data-to-Metal) dialect is an intermediate representation in the `tt-mlir` compilation pipeline that bridges high-level tensor operations (TTIR) to hardware-specific execution (TTMetal/TTNN/TTKernel). D2M introduces device-aware abstractions for grid-based execution, memory layout transformations, tile-level compute, and explicit data movement modeling.

**Scope:** This page covers the D2M dialect's operations, transformation passes, and role in the lowering pipeline. For high-level tensor operations, see [TTIR Dialect](https://deepwiki.com/tenstorrent/tt-mlir/2.1-ttir-dialect). For final metal-level lowering, see [TTMetal Backend Pipeline](https://deepwiki.com/tenstorrent/tt-mlir/3.3-ttir-to-ttmetal-backend-pipeline) and [TTKernel Dialect](https://deepwiki.com/tenstorrent/tt-mlir/2.3-ttkernel-and-sfpi-dialects). For memory optimization strategies, see [D2M Memory Allocation and Grid Selection](https://deepwiki.com/tenstorrent/tt-mlir/5.3-d2m-memory-allocation-and-grid-selection).

## Generic Operation Model

The central abstraction in D2M is `d2m::GenericOp`, which encapsulates compute or datamovement operations with explicit grid mapping, memory layout, and thread assignment.

**Generic Operation Structure**

**Key Attributes:**

*   `grid`: Physical core grid shape `[rows, cols]`.
*   `threads`: Thread assignment (e.g., `unified`, `compute`, `datamovement`).
*   `block_factors`: Blocking factors for iteration space tiling [lib/Dialect/D2M/IR/D2MOps.cpp 77-79](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/IR/D2MOps.cpp#L77-L79)
*   `indexing_maps`: Affine maps from iteration space to operands [lib/Dialect/D2M/IR/D2MOps.cpp 84-118](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/IR/D2MOps.cpp#L84-L118)
*   `iterator_types`: Parallel or reduction iterators.

## Tile-Level Compute Operations

D2M provides tile-granularity arithmetic and math operations that operate within generic compute regions. These map to FPU or SFPU hardware instructions.

**Compute Operation Hierarchy**

**Compute Interfaces:** D2M compute operations are classified by their operand requirements and hardware mapping:

*   `OperandLoadStoreRegisterOpInterface`: Interface for operations that load/store tiles to DST registers [lib/Dialect/D2M/Utils/Utils.cpp 43-45](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/Utils/Utils.cpp#L43-L45)
*   `D2M_GenericRegionComputeUnaryDstOp`: Unary operations that load/store from DST.
*   `D2M_GenericRegionComputeBinaryDstOp`: Binary operations where both operands are tiles.

**Hardware Mapping:**

*   `tile_add`: Element-wise addition on FPU or SFPU.
*   `tile_regs_acquire/release`: Managed via the `TTKernel` dialect during lowering.
*   `MathFidelity`: D2M-to-TTNN conversion handles fidelity levels like `LoFi`, `HiFi2`, `HiFi3`, `HiFi4`[lib/Conversion/D2MToTTNN/D2MToTTNN.cpp 143-156](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/D2MToTTNN/D2MToTTNN.cpp#L143-L156)
