## Purpose and Scope

This document describes the Docker container infrastructure used by the pytorch2.0_ttnn repository for CI/CD testing and development. It covers the base image, Dockerfile configuration, image versioning strategy using SHA1 hashes, and the automated container build and update workflow.

For information about how containers are used in the testing pipeline, see [Testing Pipeline and Parallelization](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.2-testing-pipeline-and-parallelization). For details on Python environment setup within containers, see [Development Environment Setup](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.5-development-environment-setup).

* * *

## Base Image and Container Architecture

The pytorch2.0_ttnn Docker container is built on top of the Tenstorrent tt-metalium base image, which provides the foundational Tenstorrent hardware support and dependencies.

### Base Image

The container uses the following base image:

```
ghcr.io/tenstorrent/tt-metal/tt-metalium/ubuntu-22.04-dev-amd64:latest
```

This base image provides:

*   Ubuntu 22.04 operating system
*   Tenstorrent metal driver and libraries
*   Development tools for building and testing TTNN operations
*   Hardware interface libraries for Tenstorrent accelerators

Sources: [dockerfile/Dockerfile 1](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/dockerfile/Dockerfile#L1-L1)

### System Dependencies

The Dockerfile adds several system-level dependencies on top of the base image:

| Dependency | Purpose |
| --- | --- |
| `sudo` | Administrative privilege management |
| `libgl1-mesa-glx` | OpenGL support for visualization models |
| `git-lfs` | Large file storage for model weights |
| `libsndfile1` | Audio file processing for speech models |
| `docker.io` | Docker-in-docker capability for nested workflows |

These dependencies are installed using `apt-get` with cache cleanup to minimize image size [dockerfile/Dockerfile 4-11](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/dockerfile/Dockerfile#L4-L11)

### Git LFS Initialization

Git LFS is initialized in the container to enable automatic handling of large files (model weights, test data):

This ensures that any `git clone` or `git pull` operations within the container automatically fetch large files stored in Git LFS [dockerfile/Dockerfile 14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/dockerfile/Dockerfile#L14-L14)

## Container Versioning Strategy

The repository uses a content-based versioning strategy where container image tags are computed from the SHA1 hash of dependency files. This ensures containers are rebuilt only when dependencies change.

This produces a tag like:

```
ghcr.io/tenstorrent/pytorch2.0_ttnn/ubuntu22.04:{SHA1}
```

Sources: [.github/workflows/update-docker-container.yaml 28-34](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L28-L34)

### Tag Format and Repository Structure

| Component | Value | Example |
| --- | --- | --- |
| Registry | `ghcr.io` | GitHub Container Registry |
| Organization | `tenstorrent` | Tenstorrent organization |
| Repository | `pytorch2.0_ttnn` | This project |
| Base OS | `ubuntu22.04` | Ubuntu 22.04 |
| Version | `{SHA1}` | `a1b2c3d4e5f...` |
| Latest | `latest` | Alias for latest passing version |

### Latest Tag Management

The `latest` tag is only applied to container images that:

1.   Successfully build from the Dockerfile
2.   Pass all test suites in the `run-tests` workflow
3.   Have no test failures

This ensures the `latest` tag always points to a stable, validated container image [.github/workflows/update-docker-container.yaml 98-131](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L98-L131)

## Container Build and Update Workflow

The `update-docker-container.yaml` workflow orchestrates container building, testing, and publishing. It can be triggered manually via `workflow_dispatch` or called by other workflows using `workflow_call`.

### Workflow Architecture

**Diagram: Container Build and Update Workflow Flow**

### Job Breakdown

#### 1. check-docker-images

This job computes the SHA1 tag and checks if the container already exists in the registry:

**Outputs:**

*   `dev-exists`: Boolean indicating if image exists
*   `dev-tag`: Full image tag (ghcr.io/tenstorrent/pytorch2.0_ttnn/ubuntu22.04:{SHA1})

The image existence check uses `docker manifest inspect`:

Sources: [.github/workflows/update-docker-container.yaml 15-52](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L15-L52)

#### 2. create-new-container-and-upload

This job runs conditionally only if the container does not exist:

**Build Process:**

1.   Checkout repository
2.   Login to GitHub Container Registry using `GH_TOKEN` secret
3.   Build Docker image using `docker/build-push-action@v6`
4.   Push to registry with computed SHA1 tag

**Build Configuration:**

| Parameter | Value |
| --- | --- |
| `file` | `dockerfile/Dockerfile` |
| `platforms` | `linux/amd64` |
| `pull` | `true` (always pull base image) |
| `push` | `true` (push after build) |
| `tags` | SHA1-based tag from step 1 |
| `context` | `.` (repository root) |
| `target` | `release` (Dockerfile stage) |

Sources: [.github/workflows/update-docker-container.yaml 54-82](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L54-L82)

#### 3. run-tests

After the container is built (or confirmed to exist), the workflow invokes the `run-tests.yaml` workflow to validate the container:

This runs the full test suite inside the new container to ensure it works correctly. See [Testing Pipeline and Parallelization](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.2-testing-pipeline-and-parallelization) for details.

Sources: [.github/workflows/update-docker-container.yaml 83-96](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L83-L96)

#### 4. tag-latest

This job runs after testing completes (even if previous steps are skipped) and updates the `latest` tag if all conditions are met:

**Conditions for tagging `latest`:**

**Tagging Process:**

1.   Pull the SHA1-tagged image
2.   Tag it as `ghcr.io/tenstorrent/pytorch2.0_ttnn/ubuntu-22.04-amd64:latest`
3.   Push the `latest` tag

Sources: [.github/workflows/update-docker-container.yaml 98-131](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L98-L131)

### Container Selection in Workflows

Workflows can specify which container to use via the `docker_tag` parameter:

When `docker_tag` is not specified, workflows default to the `latest` tag [.github/workflows/update-docker-container.yaml 95](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L95-L95)

### Environment Setup Inside Containers

Once a container is running, the `common_repo_setup` action performs additional setup:

**Diagram: Environment Setup Steps in common_repo_setup Action**

**Key Setup Steps:**

1.   **Pip Cache Management**: Creates `PIP_CACHE_DIR` if defined to enable caching between runs
2.   **Python 3.10 Setup**: Uses `actions/setup-python@v5` to ensure Python 3.10 is available
3.   **PyTorch CPU Index**: Configures pip to use PyTorch CPU-only wheels from `https://download.pytorch.org/whl/cpu`
4.   **Requirements Installation**: Installs all dependencies from `requirements-dev.txt`
5.   **Pytest Extensions**: Adds `pytest-github-report` for GitHub integration

## Container Image Registry and Authentication

All container images are stored in GitHub Container Registry (GHCR) with access controlled by GitHub authentication.

### Registry Structure

```
ghcr.io/
└── tenstorrent/
    └── pytorch2.0_ttnn/
        ├── ubuntu-22.04-amd64:latest
        └── ubuntu22.04:{SHA1_HASH}
```

### Authentication

Workflows authenticate to GHCR using the `docker/login-action`:

The `GH_TOKEN` secret must have `write:packages` permission to push images [.github/workflows/update-docker-container.yaml 36-111](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L36-L111)

### Public Accessibility

Container images are publicly readable, allowing:

*   External contributors to pull images for testing
*   CI/CD runners to pull without authentication
*   Docker clients to use images with standard `docker pull` commands

Sources: [.github/workflows/update-docker-container.yaml 36-111](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L36-L111)

## Summary

The Docker container infrastructure provides:

1.   **Reproducible Environments**: Content-based versioning ensures containers are rebuilt only when dependencies change
2.   **Base Image Integration**: Leverages tt-metalium for Tenstorrent hardware support
3.   **Automated Testing**: New containers are validated before being tagged as `latest`
4.   **Efficient CI/CD**: Containers are reused across workflow runs when dependencies haven't changed
5.   **Version Tracking**: SHA1-based tags provide precise version tracking and rollback capability

This infrastructure ensures consistent, validated testing environments across all CI/CD workflows while minimizing rebuild overhead.
