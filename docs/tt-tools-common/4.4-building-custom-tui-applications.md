## Purpose and Scope

This document provides a practical guide to building terminal user interface (TUI) applications using tt-tools-common's widget library and theming system. It covers the application structure, widget composition, layout strategies, event handling, and styling integration needed to create custom TUI tools.

For detailed reference on individual widgets and their APIs, see [Widget Library Reference](https://deepwiki.com/tenstorrent/tt-tools-common/4.1-widget-library-reference). For information about modal dialogs, see [Modal Screens and Dialogs](https://deepwiki.com/tenstorrent/tt-tools-common/4.2-modal-screens-and-dialogs). For styling customization, see [Theming and Styling](https://deepwiki.com/tenstorrent/tt-tools-common/4.3-theming-and-styling).

## Application Architecture

### Textual App Foundation

TUI applications built with tt-tools-common inherit from `textual.app.App` and integrate the common widget library. The basic structure follows Textual's lifecycle model with tt-tools-common extensions.

**Sources:**[tt_tools_common/tests/test_widget_app.py 26-89](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_widget_app.py#L26-L89)[tt_tools_common/ui_common/widgets.py 1-365](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/ui_common/widgets.py#L1-L365)

### Basic Application Template

The minimal application structure consists of:

| Component | Purpose | Required |
| --- | --- | --- |
| `__init__()` | Initialize app metadata and bindings | Yes |
| `CSS_PATH` | Link to styling system | Yes |
| `BINDINGS` | Define keyboard shortcuts | Optional |
| `compose()` | Create widget hierarchy | Yes |
| `on_mount()` | Initialize widget data | Optional |
| `action_*` methods | Handle user interactions | Optional |

**Sources:**[tt_tools_common/tests/test_widget_app.py 26-50](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_widget_app.py#L26-L50)

tt-tools-common widgets work with Textual's container system for flexible layouts:

| Container | Use Case | Example |
| --- | --- | --- |

| `Vertical()` | Stacked placement | Multiple TTMenu widgets |
| `Grid()` | Cell-based layout | Modal dialogs (TTConfirmBox) |
| `ScrollableContainer` | Scrollable content | Built into TTDataTable |

## Event Handling and Actions

### Key Bindings

Applications define keyboard shortcuts through the `BINDINGS` class attribute:

[tt_tools_common/tests/test_widget_app.py 29-33](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_widget_app.py#L29-L33)

Each binding is a tuple: `(key, action_name, description)`

| Element | Purpose | Example |
| --- | --- | --- |
| Key | Keyboard key(s) | `"d"`, `"q"`, `"ctrl+c"` |
| Action name | Method name without `action_` prefix | `"toggle_dark"` → `action_toggle_dark()` |
| Description | Footer display text | `"Toggle dark mode"` |

## Best Practices

### Widget ID Management

Always assign unique IDs to widgets that need programmatic access:

| Pattern | Usage |
| --- | --- |
| `id="menu_1"` | Single widget instance |
| `id=f"menu_{i}"` | Multiple similar widgets |
| `id="dt_example"` | Descriptive names for data tables |
| `id=self.id + "_table"` | Child widget IDs based on parent |

### Layout Responsiveness

Leverage Textual's layout system for responsive designs:

| Container | Behavior | Use When |
| --- | --- | --- |
| `Horizontal()` | Side-by-side, fills width | Multiple columns needed |
| `Vertical()` | Top-to-bottom, fills height | Stacking widgets |
| `Grid()` | Cell-based, fixed structure | Forms, dialogs |
| `ScrollableContainer` | Overflow scrolling | Large content areas |

The CSS controls sizing with `height: 1fr` (fill remaining space) and explicit values.

## Integration with Consumer Applications

### Importing tt-tools-common Widgets

Consumer applications (like tt-smi or tt-flash) import widgets as a library:

`from tt_tools_common.ui_common.widgets import (    TTHeader,    TTFooter,    TTDataTable,    TTMenu,    TTHostCompatibilityMenu,    TTConfirmBox,)`
The widgets are designed for reusability across different Tenstorrent tools.

Applications can subclass tt-tools-common widgets to add custom behavior:

`class CustomDataTable(TTDataTable):    def on_mount(self) -> None:        super().on_mount()        # Custom initialization            def custom_method(self) -> None:        # Additional functionality        pass`
This pattern maintains the base widget's styling and behavior while adding application-specific features.

## Summary

Building TUI applications with tt-tools-common requires:

1.   **Inherit from `textual.app.App`** and set `CSS_PATH` to load common styling
2.   **Define `compose()`** to yield widget hierarchy using containers and tt-tools-common widgets
3.   **Implement `on_mount()`** to populate widgets with initial data
4.   **Configure `BINDINGS`** and `action_*` methods for user interactions
5.   **Use modal screens** via `push_screen()` and `pop_screen()` for dialogs
6.   **Access widgets by ID** using `get_widget_by_id()` for dynamic updates
7.   **Follow layout patterns** with Horizontal/Vertical containers for responsive designs

The test widget application [tt_tools_common/tests/test_widget_app.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_widget_app.py) demonstrates all these patterns in a working example that can serve as a template for new applications.
