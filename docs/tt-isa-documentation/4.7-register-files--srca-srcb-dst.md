## Purpose and Scope

The Tensix coprocessor maintains three primary register files—**SrcA**, **SrcB**, and **Dst**—that serve as high-throughput intermediary storage between L1 memory and the execution units. These register files form the core of the Tensix compute pipeline:

*   **SrcA** and **SrcB** hold input operands for the Matrix Unit (FPU) in low-precision 19-bit format
*   **Dst** accumulates computation results and can be viewed as either 16-bit or 32-bit data

Data flows from L1 memory through Unpackers into these register files, then through compute units (Matrix Unit or Vector Unit), and back to L1 via Packers. The register files provide the low-latency storage necessary to feed the high-throughput Matrix Unit, which can execute matrix multiplications at up to 4.096 TFLOP/s per tile.

For instruction-level details on operations that use these registers, see the Matrix Unit instruction reference ([#5.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.2-matrix-unit-instructions)) and Vector Unit instruction reference ([#5.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.3-vector-unit-core-instructions), [#5.4](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.4-vector-unit-data-movement-instructions)). For data format specifics, see the data format catalog ([#6.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.2-data-type-catalog)).

### SrcA and SrcB Register Files

Both **SrcA** and **SrcB** are organized as dual-banked register files, each bank containing a 64×16 matrix of 19-bit elements:

The Matrix Unit maintains bank selection state (`MatrixUnit.SrcABank` and `MatrixUnit.SrcBBank`) that alternates between banks 0 and 1. This dual-banking enables:

*   Concurrent read from one bank while the other bank is being written by an unpacker
*   Double-buffering for higher throughput operations

The 19-bit element width accommodates internal floating-point formats with extended precision during computation, supporting various fidelity phases for matrix operations. Each bank also tracks an `AllowedClient` field to coordinate access between the Matrix Unit and Unpackers (see [AllowedClient Coordination](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/AllowedClient%20Coordination)).

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 23-58](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L23-L58)

### Dst Register File

The **Dst** register file provides flexible storage that can be viewed as either 16-bit or 32-bit data:

**Underlying Storage:**

**Two Views of the Same Memory:**

| View | Dimensions | Element Width | Access Pattern | Architecture |
| --- | --- | --- | --- | --- |
| `Dst16b[Row][Col]` | 1024 rows × 16 columns | 16 bits | `DstBits[Adj16(Row)][Col]` | Both |
| `Dst32b[Row][Col]` | 512 rows × 16 columns | 32 bits | `(DstBits[Adj32(Row)][Col] << 16)` | `DstBits[Adj32(Row) + 8][Col]` | Both |

Software must choose one view and use it consistently. Mixing `Dst16b` and `Dst32b` accesses requires careful attention to address mapping. The `Config.ALU_ACC_CTRL_Fp32_enabled`, `Config.ALU_ACC_CTRL_INT8_math_enabled`, or `RISCV_DEBUG_REG_DBG_FEATURE_DISABLE[11]` settings determine which view is in effect for certain operations.

**Addressing Functions:**

Blackhole A0 introduces configurable address remapping via `Config.DEST_ACCESS_CFG_remap_addrs` and `Config.DEST_ACCESS_CFG_swizzle_32b`:

Wormhole B0 uses fixed addressing without remapping:

The `Adj32()` function maps 512 logical 32-bit rows to 1024 physical 16-bit rows by interleaving: row `N` maps to physical rows `Adj32(N)` and `Adj32(N) + 8`.

## Supported Data Types

**Diagram: Data Type Support Across Register Files**

### SrcA and SrcB Data Types

SrcA and SrcB use an internal 19-bit format that extends standard formats with guard bits for higher precision during computation. These registers do not directly expose their bit layout to software; data enters through Unpackers with automatic format conversion from L1.

### Dst16b Data Types

| Type | Range/Precision | Bit Layout | Notes |
| --- | --- | --- | --- |
| **BF16** | 8-bit exponent, 7-bit mantissa | `S[15] Exp[14:7] Mant[6:0]` | Lossless from BFP8/BFP4/BFP2 |
| **FP16** | 5-bit exponent, 10-bit mantissa | `S[15] Exp[14:10] Mant[9:0]` | Non-IEEE infinity format; lossless from FP8 |
| **Integer "8"** | -1023 to +1023 | `S[15] RawExp[14:10] Mag[9:5] RawExp[4:0]` | Overlaid on FP16; typical range -127 to +127 or 0 to 255 |
| **Integer "16"** | Opaque 16-bit | `S[15] Mag[14:0]` | Sign-magnitude; no computation ops, data transfer only |

### Dst32b Data Types

| Type | Format | Notes |
| --- | --- | --- |
| **FP32** | IEEE-like 32-bit float | `S[31] Exp[30:23] Mant[22:0]` (with some IEEE divergences) |
| **Integer "32"** | Sign-magnitude | `S[31] Mag[30:0]` (not two's complement) |

**Important:** The coprocessor diverges from IEEE 754 in several ways. See [#6.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.1-floating-point-models-and-fma) for floating-point model details and [#6.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.2-data-type-catalog) for complete data type specifications.

Matrix Unit (FPU) compute instructions that read from register files:

*   `MVMUL`, `DOTPV`, `GAPOOL`: Read `SrcA[MatrixUnit.SrcABank]` and `SrcB[MatrixUnit.SrcBBank]`, accumulate to `Dst`
*   `ELWMUL`, `ELWADD`, `ELWSUB`: Read `SrcA`, `SrcB`, and optionally `Dst`, write/accumulate to `Dst`
*   `GMPOOL`: Reads `SrcA`, accumulates to `Dst`

### Vector Unit Data Movement

The Vector Unit (SFPU) accesses `Dst` through the `LReg[0-16]` intermediate register file:

*   `SFPLOAD`: Reads 32 lanes from `Dst[Row][0-15]` into `LReg[n]` (latency varies)
*   `SFPLOADMACRO`: Reads multiple rows from `Dst` and applies lookup table, stores in `LReg`
*   `SFPSTORE`: Writes 32 lanes from `LReg[n]` to `Dst[Row][0-15]` (latency varies)

These instructions use RWC addressing but do not update `RWCs[CurrentThread].FidelityPhase`.

### Unpacker/Packer Access

Unpackers move data from L1 to register files via `UNPACR` instruction variants:

*   **Unpacker 0**: Can write to `SrcA[0]`, `SrcA[1]`, `Dst`, or `SrcB` (target selected by instruction fields)
*   **Unpacker 1**: Can write to `SrcB[0]`, `SrcB[1]`, or `Dst` (target selected by instruction fields)

Packers move data from `Dst` back to L1 via `PACR` instruction:

*   **Packers 0-3**: All read from `Dst` and write to L1 (packer 0 can also read from L1 for L1→L1 packing)

## AllowedClient Coordination Mechanism

To prevent read-after-write hazards, each register file bank tracks which execution unit currently "owns" write access:

### Wait Gate Arbitration

Before dispatching an instruction to an execution unit, the Wait Gate checks whether the required register bank allows access from that unit:

**Diagram: AllowedClient Wait Gate Arbitration Flow**

### Example Instruction Constraints

The `MOVB2A` instruction demonstrates asymmetric wait behavior defined at [WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 22-28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L22-L28):

This asymmetry exists because:

*   **Reading** from `SrcB`: Automatically waits for `AllowedClient == MatrixUnit` (handled by Wait Gate)
*   **Writing** to `SrcA`: Software must explicitly synchronize with `STALLWAIT` if an unpacker was recently writing

Similar patterns apply to other Matrix Unit instructions: automatic waits for reads, manual synchronization for writes.

## Validity Tracking and ZEROACC

Each row in `Dst` has an associated validity bit that tracks whether the row contains defined data:

### Marking Rows as Undefined

The `ZEROACC` instruction marks rows as undefined without actually writing zeros:

### Undefined Row Semantics

Different execution units interpret undefined rows differently:

| Unit | Behavior When Reading Undefined Row |
| --- | --- |
| **Packers** | Read undefined rows as **zero** |
| **Matrix Unit (FPU)** | Read as **identity element** for the operation: • Negative infinity for `GMPOOL` • Zero for all other operations |
| **Vector Unit (SFPU)** | **Undefined behavior**—software must avoid |
| **Unpackers** | **Undefined behavior** if partially overwriting undefined row |

### Common Usage Pattern

Typical accumulation loop structure using `ZEROACC` with different execution unit behaviors:

**Critical Rules:**

*   Matrix Unit: Reads undefined as identity element (zero for most ops, -∞ for `GMPOOL`)
*   Packers: Read undefined rows as zero
*   Vector Unit (SFPU): **Undefined behavior** if reading undefined rows with `SFPLOAD`
*   Unpackers: **Undefined behavior** if partially writing to undefined row

## Instruction Scheduling and Hazards

### Read-After-Write Hazard (4-Cycle Latency)

After any instruction writes to `Dst`, a 4-cycle hazard window prevents reading the same 8×16 block:

**Diagram: Four-Cycle Read-After-Write Hazard Window**

### Automatic Stalling (Blackhole A0)

Blackhole A0 introduces automatic hardware stalling for read-after-write hazards. Hardware automatically stalls threads that attempt to read from `Dst` during the hazard window:

**Note:** Wormhole B0 does not have automatic stalling. Software must manually schedule instructions to avoid the 4-cycle hazard window, or accept reduced performance from manual synchronization.

### Performance Optimization: Loop Over Blocks

To avoid stalls from the 4-cycle read-after-write hazard, software should loop over at least **5 distinct 8×16 blocks** of `Dst`. The hazard applies to aligned 8-row blocks: rows `[N & ~7]` through `[(N & ~7) + 7]`.

**Key Rules:**

1.   When using multiple fidelity phases, make the **fidelity loop inner** and the **block loop outer**
2.   Ensure at least 5 distinct 8×16 blocks in rotation
3.   Instructions accumulating to `Dst` include: `MVMUL`, `GAPOOL`, `DOTPV`, `GMPOOL`, `ELWMUL`, `ELWADD`, `ELWSUB`

This optimization applies to both architectures, though Blackhole's automatic stalling reduces the performance penalty of poor scheduling.

## RISC-V Core Access to Dst

RISC-V cores T0, T1, and T2 can access `Dst` via memory-mapped I/O starting at address `0xFFBD_8000`. RISC-V cores B and NC cannot access `Dst` directly.

### Memory-Mapped Address Range

### Format Control via Configuration

The mapping behavior is controlled by per-thread configuration fields in the `Config[StateID].RISC_DEST_ACCESS_CTRL_SEC[CurrentThread]` structure:

| `fmt` Value | C Type View | Conversion Performed |
| --- | --- | --- |
| 0 | `float Dst32b[512][16]` or `uint32_t Dst32b[512][16]` | IEEE754 FP32 ↔ Dst FP32 bit layout Conventional int ↔ Dst Integer "32" |
| 1 | `int32_t Dst32b[512][16]` | Two's complement ↔ Sign-magnitude Conventional int ↔ Dst Integer "32" |
| 2 | `__fp16 Dst16b[1024][16]` | IEEE754 FP16 ↔ Dst FP16 bit layout |
| 3 | `__bf16 Dst16b[1024][16]` | Truncated IEEE754 FP32 ↔ Dst BF16 |
| 4 | `int16_t Dst16b[1024][16]` or `uint16_t Dst16b[1024][16]` | Two's complement ↔ Sign-magnitude |
| 5 | `int8_t Dst16b[1024][16]` or `uint8_t Dst16b[1024][16]` | Two's complement ↔ Sign-magnitude Conventional int ↔ Dst Integer "8" |
| ≥6 | Reserved | N/A |

### Access Width Constraints

*   **RISC-V T0 and T1**: Can only access one array element at a time

    *   16-bit types: Use `sh`/`lhu` instructions
    *   32-bit types: Use `sw`/`lw` instructions

*   **RISC-V T2**: Can access multiple elements at once

    *   Supports both scalar and vector load/store instructions
    *   Still requires alignment to the overall size of the load/store

### Bit Layout Conversion Example

When `no_swizzle == false` (default), hardware performs automatic bit rearrangement between CPU-expected bit layout and Dst internal layout. The conversion functions are defined at [BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 118-146](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L118-L146) and [WormholeB0/TensixTile/TensixCoprocessor/Dst.md 118-146](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L118-L146):

The `no_swizzle` flag allows software to bypass conversion and access raw Dst bit patterns directly, useful for debugging or specialized operations.
