This page documents the vector memory operations in the RISCV-Ocelot system, focusing on how vector load and store instructions are processed through the memory hierarchy. For information about the Vector Processing Unit (VPU) internals, see [Vector Processing Unit](https://deepwiki.com/tenstorrent/riscv-ocelot/5.2-vector-processing-unit).

## Overview

Vector memory operations in the RISCV-Ocelot implementation allow for efficient data movement between memory and vector registers. The vector memory subsystem supports various addressing modes including unit-stride, strided, indexed, whole-register, and masked operations as defined by the RISC-V Vector Extension (RVV).

The system handles these operations through a collaboration between the OviWrapper interface component, the Load/Store Unit (LSU), and specialized vector memory handling components within the VPU.

## Vector Memory Operation Types

The RISCV-Ocelot implements several types of vector memory operations as defined by the RISC-V Vector Extension:

| Operation Type | Description | Addressing | Encoding |
| --- | --- | --- | --- |
| Unit-stride | Elements are contiguous in memory | Base + i*esize | mop=0 |
| Strided | Elements are separated by a stride | Base + i*stride | mop=2 |
| Indexed | Elements' addresses from an index vector | Base + Vx[i] | mop=1/3 |
| Whole Register | Transfer entire vector register(s) | Unit-stride with special umop | umop=8, mop=0 |
| Masked | Conditionally access elements | Any of the above with mask | vm=0 |

The addressing modes are detected in the OviWrapper's instruction decoding logic, which sets appropriate control signals for the Vector Address Generator (VAgen).

## Memory Interface Components

### OviWrapper Interface

The OviWrapper serves as the bridge between the scalar core's execution units and the vector processing unit. For vector memory operations, it:

1.   Receives vector load/store instructions and decodes them
2.   Routes them to the VPU and LSU appropriately
3.   Handles vector memory synchronization via memory sync signals
4.   Manages data movement between the scalar core, LSU, and VPU

The OviWrapper contains the Vector Address Generator (VAgen) which calculates memory addresses for vector load/store operations based on the addressing mode.

### Load/Store Unit (LSU) Vector Interface

The LSU handles both scalar and vector memory operations. For vector operations, it:

1.   Maintains separate queues for vector loads (DLQ) and stores (DSQ)
2.   Coordinates with the OviWrapper through the vGenIO interface
3.   Ensures proper ordering between scalar and vector memory operations
4.   Executes the actual memory accesses

Vector operations use the following queue structures:

*   **Vector Load Queue (DLQ)**: Holds entries for vector load operations
*   **Vector Store Queue (DSQ)**: Holds entries for vector store operations

These are distinguished from the scalar Load Queue (LDQ) and Store Queue (STQ).

## Vector Memory Operation Flow

### Vector Load Flow

Vector load operations follow this processing flow:

1.   A vector load instruction is dispatched to the OviWrapper
2.   The OviWrapper decodes the instruction and determines the addressing mode
3.   The Vector Address Generator calculates memory addresses
4.   For masked operations, the Mask FSM generates mask information
5.   The LSU allocates DLQ entries and issues memory requests
6.   Memory returns data, which the LSU routes back to the VPU
7.   The VPU writes the data to appropriate vector registers
8.   Synchronization signals are exchanged upon completion

### Vector Store Flow

Vector store operations follow this processing flow:

1.   A vector store instruction is dispatched to the OviWrapper
2.   The OviWrapper decodes the instruction and determines the addressing mode
3.   The Store FSM retrieves store data from vector registers
4.   The Vector Address Generator calculates memory addresses
5.   For masked operations, the Mask FSM generates mask information
6.   The LSU allocates DSQ entries and issues memory requests
7.   Store data is written to memory
8.   Synchronization signals are exchanged upon completion

## Special Vector Memory Operation Cases

### Masked Memory Operations

Vector memory operations can be conditional using a mask vector, where only elements corresponding to '1' mask bits are accessed:

1.   The MaskFSM component handles mask processing for vector loads/stores
2.   It receives the mask vector and generates a sequence of mask items
3.   For each memory transaction, it provides mask information
4.   Masked-off elements (mask=0) are handled specially: 
    *   For loads, fake responses are generated
    *   For stores, the memory request is suppressed

### Whole-Register Operations

Whole-register operations transfer entire vector registers to/from memory:

1.   Special decoding identifies whole-register operations (umop=8)
2.   The VWhLSDecoder calculates the appropriate vector length based on nf (number of fields)
3.   The operation is processed as a sequence of unit-stride accesses
4.   The entire register(s) are transferred regardless of the current vector length (vl)

### Vector Length Handling

Vector memory operations respect the current vector length (vl):

1.   The OviWrapper receives the vector configuration including vl
2.   For normal operations, only vl elements are processed
3.   For whole-register operations, the decoder overrides vl
4.   When vl=0, special handling: 
    *   A "fake" flag is set in queue entries
    *   The operation completes without actual memory accesses

## Memory Operation Synchronization

Vector memory operations use a synchronization protocol between the VPU and LSU:

1.   The VPU initiates a memory operation with `memop_sync_start`
2.   The LSU processes the operation, tracking completion
3.   When all memory requests are complete, the LSU signals `memop_sync_end`
4.   The VPU continues execution once synchronization is complete

A scoreboarding mechanism tracks outstanding memory operations and ensures proper completion:

## Interaction with Memory Consistency Model

Vector memory operations must maintain proper ordering with respect to other memory operations:

1.   The LSU tracks dependencies between scalar and vector memory operations: 
    *   Vector loads (vld) are tracked with a counter (vld_count)
    *   Vector stores (vst) are tracked with a counter (vst_count)

2.   Scalar loads check if they are blocked by older vector operations
3.   Vector operations check if they can proceed based on ordering constraints

The system ensures scalar and vector operations maintain proper ordering, preventing hazards while allowing concurrency where possible.

## Implementation Optimizations

### Load Data Buffering

The system includes a load data buffering mechanism to handle vector loads efficiently:

1.   The scoreboard allocates load data buffer entries for vector loads
2.   The buffer collects load data from multiple memory transactions
3.   When all data is collected, it's transferred to the vector register file
4.   A credit-based system manages buffer resources

This allows efficient handling of vector loads with element sizes smaller than the memory datapath width.

### Address Generation Optimizations

The Vector Address Generator (VAgen) includes several optimizations:

1.   Special handling for different addressing modes (unit-stride, strided, indexed)
2.   Support for fractional EMUL (effective element multiplier)
3.   Optimization for stride magnitude and direction
4.   Address alignment handling to match memory system requirements

These optimizations ensure efficient memory access patterns for vector operations.

## Conclusion

The vector memory operations system in RISCV-Ocelot provides a flexible and efficient mechanism for transferring data between memory and vector registers. It supports all standard RISC-V vector memory operations including different addressing modes and masking capabilities. The implementation balances performance with correctness, ensuring proper memory ordering while enabling parallelism where possible.

The system consists of multiple interacting components across the OviWrapper, LSU, and VPU, with specialized mechanisms for address generation, masking, data buffering, and synchronization. This design enables efficient execution of vector memory operations that forms a crucial part of the overall vector processing capabilities.
