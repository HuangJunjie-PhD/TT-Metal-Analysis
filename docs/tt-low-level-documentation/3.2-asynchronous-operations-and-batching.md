## Purpose and Scope

This document explains how to use asynchronous NOC operations and batching patterns to maximize data movement throughput in TT-Metal kernels. The NOC hardware is designed to handle multiple in-flight transactions concurrently, but achieving this performance requires understanding when to issue operations, when to synchronize, and how to batch operations effectively.

## Asynchronous Execution Model

NOC operations in TT-Metal are **asynchronous by design**. When you call `noc_async_read()` or `noc_async_write()`, the function returns immediately after queuing the operation—the actual data transfer happens in the background while your kernel continues executing.

### How Asynchronous Operations Work

**Sources:**[data_movement_doc/general/dm_best_practices.md 7-62](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L7-L62)

### Key Characteristics

| Aspect | Behavior |
| --- | --- |
| **Function Return** | Returns immediately after queuing the operation |
| **Data Availability** | Data arrives at destination after an undefined delay |
| **Concurrent Operations** | Multiple operations can be in-flight simultaneously |
| **Hardware Queues** | NOC maintains internal command buffers for pending operations |
| **Completion Detection** | Requires explicit barrier calls to ensure completion |

The asynchronous model enables **overlapping compute with communication**: while the NOC transfers data, your kernel can perform other work, maximizing overall throughput.

**Sources:**[data_movement_doc/general/dm_best_practices.md 9-10](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L9-L10)

## Barriers and Synchronization Points

Barriers are synchronization primitives that **block execution until all pending NOC operations complete**. Proper barrier placement is critical for both correctness and performance.

### Core Barrier APIs

**Sources:**[data_movement_doc/general/dm_best_practices.md 52-62](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L52-L62)[data_movement_doc/general/dm_best_practices.md 239-248](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L239-L248)

### Critical Synchronization Points

Barriers are **required** at these points:

| Synchronization Point | Reason | API Used |
| --- | --- | --- |
| **Before Data Use** | Ensure data has arrived before processing it | `noc_async_read_barrier()` |
| **Buffer Full** | Wait for space before issuing more operations | Barrier or check buffer status |
| **Context Switch** | Flush operations before changing NOC state | `noc_async_write_barrier()` |
| **Kernel End** | Ensure all writes complete before termination | `noc_async_write_barrier()` |

**Sources:**[data_movement_doc/general/dm_best_practices.md 52-57](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L52-L57)

## The Batching Pattern

The fundamental performance optimization for NOC operations is **batching**: issue multiple operations before placing a barrier, allowing the hardware to process them concurrently.

### Anti-Pattern: Barrier Per Operation

This pattern **destroys performance** by preventing any concurrency:

Example of **incorrect code** from [data_movement_doc/general/dm_best_practices.md 16-22](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L16-L22):

**Why this is slow:**

*   Each barrier waits for one operation to complete
*   NOC sits idle most of the time
*   No overlap between operations
*   No overlap between NOC and compute

### Performance Benefits

| Pattern | NOC Utilization | Compute/NOC Overlap | Relative Throughput |
| --- | --- | --- | --- |
| Barrier per operation | Very Low (~10%) | None | 1x (baseline) |
| Small batches (2-4) | Medium (~40%) | Limited | ~3-4x |
| Optimal batches (4-8) | High (~80%) | Good | ~8-10x |
| Excessive batches (>16) | High (~80%) | Poor (compute waits) | ~6-8x |

The exact optimal batch size depends on operation size, compute time, and memory constraints.

The batch size represents a **fundamental trade-off** between NOC utilization and compute parallelism:

### Selection Criteria

From [data_movement_doc/general/dm_best_practices.md 24-28](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L24-L28):

### Factors Affecting Optimal Batch Size

| Factor | Small Batch Size | Large Batch Size |
| --- | --- | --- |
| **Operation latency** | Long latency → bigger batches hide latency | Short latency → smaller batches sufficient |
| **Compute time per tile** | Short compute → bigger batches keep NOC busy | Long compute → smaller batches prevent idle time |
| **Available memory** | Large L1 → can buffer more | Limited L1 → smaller batches |
| **Transfer size** | Small transfers → more ops in batch | Large transfers → fewer ops in batch |

## Best Practices Summary

| Practice | Why It Matters | Performance Impact |
| --- | --- | --- |
| **Batch operations** | Enables concurrent NOC operations | 3-10x throughput increase |
| **Choose batch size carefully** | Balances NOC utilization vs compute idle time | 20-40% additional gain |
| **Single barrier per batch** | Minimizes synchronization overhead | Critical for batching benefits |
| **Never omit barriers before data use** | Prevents data races and corruption | Correctness requirement |
| **Use `writes_flushed()` when possible** | Faster than `write_barrier()` | 10-20% barrier overhead reduction |
| **Combine with stateful APIs** | Reduces per-operation overhead | Additional 10-30% gain |
