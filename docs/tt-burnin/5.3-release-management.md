This document covers the automated release management system for the tt-burnin project, including version management, changelog generation, package building, and distribution to multiple channels. The release system handles semantic versioning, maintains multiple changelog formats, and coordinates distribution to GitHub Releases, PyPI, and Debian package repositories.

For information about the CI/CD workflows that support release management, see [CI/CD Workflows](https://deepwiki.com/tenstorrent/tt-burnin/5.2-cicd-workflows). For details about Debian packaging specifics, see [Debian Packaging](https://deepwiki.com/tenstorrent/tt-burnin/5.1-debian-packaging).

## Version Management

The tt-burnin project uses semantic versioning with automated patch version bumping during releases. Version information is centrally managed in the `pyproject.toml` file and propagated throughout the build system.

### Version Bumping Process

The release workflow automatically increments patch versions using the `bump2version` tool. The version management process reads the current version from `pyproject.toml`, increments the patch number, and commits the change back to the repository.

**Sources:**[.github/workflows/release.yml 92-144](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L92-L144)[pyproject.toml](https://github.com/tenstorrent/tt-burnin/blob/809f293d/pyproject.toml)

### Version Components

The version management system extracts and uses multiple version components throughout the release process:

| Component | Description | Usage |
| --- | --- | --- |
| `major` | Major version number | Debian changelog, package metadata |
| `minor` | Minor version number | Debian changelog, package metadata |
| `patch` | Patch version number | Automatic increment target |
| `full` | Complete version string | Git tags, GitHub releases |
| `prerelease` | Pre-release identifier | Future use for alpha/beta releases |
| `build` | Build metadata | Future use for build identifiers |

**Sources:**[.github/workflows/release.yml 68-73](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L68-L73)[.github/workflows/release.yml 120-124](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L120-L124)

## Changelog Management

The release system maintains three different changelog formats to serve different audiences and packaging requirements.

### User-Facing Changelog

The main `CHANGELOG.md` follows the Keep a Changelog format with semantic versioning. This changelog is manually maintained and serves as the authoritative record of user-visible changes.

**Sources:**[CHANGELOG.md 1-40](https://github.com/tenstorrent/tt-burnin/blob/809f293d/CHANGELOG.md?plain=1#L1-L40)

### Debian Package Changelog

The Debian changelog is automatically generated using `git-buildpackage` (`gbp dch`) during the release process. This changelog follows Debian packaging standards and includes commit-level changes since the last release.

The generation process uses the number of commits since the last tag to create appropriate changelog entries for the Debian packaging system.

**Sources:**[.github/workflows/release.yml 190-204](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L190-L204)[debian/changelog 1-39](https://github.com/tenstorrent/tt-burnin/blob/809f293d/debian/changelog#L1-L39)

### GitHub Release Changelog

GitHub releases include automatically generated changelogs created by the `mikepenz/release-changelog-builder-action`. This changelog includes both pull requests and direct commits in a hybrid format.

**Sources:**[.github/workflows/release.yml 293-309](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L293-L309)

## Release Workflow Process

The release management system orchestrates a complex workflow involving version management, building, testing, and distribution across multiple channels.

### Release Workflow Architecture

**Sources:**[.github/workflows/release.yml 1-442](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L1-L442)

### Temporary Branch Management

The release process creates temporary branches to isolate release changes from the main development branch. This approach allows for atomic releases while maintaining a clean commit history.

The temporary branch naming follows the pattern: `rc-temp-{short-sha}-{timestamp}`, ensuring uniqueness and traceability.

## Distribution Channels

The release system distributes packages to multiple channels to serve different user communities and installation methods.

### Distribution Channel Architecture

**Sources:**[.github/workflows/release.yml 276-346](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L276-L346)[.github/workflows/release.yml 379-441](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L379-L441)

### GitHub Releases

GitHub releases serve as the primary distribution point, providing:

*   Tagged releases with semantic version numbers
*   Release notes generated from commit and PR history
*   Downloadable artifacts including Debian packages and Python wheels
*   GPG-signed packages for security verification

The release artifact collection includes packages for multiple Ubuntu versions and Python wheel distributions.

**Sources:**[.github/workflows/release.yml 334-346](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L334-L346)

### PyPI Distribution

Python package distribution follows a two-stage process:

1.   **TestPyPI**: Initial publication to the test repository for validation
2.   **PyPI**: Production publication after successful testing

Both distributions use trusted publishing with OIDC tokens for secure, keyless authentication.

**Sources:**[.github/workflows/release.yml 379-441](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L379-L441)

## Git Management

The release system includes sophisticated Git repository management to ensure clean release workflows and proper version tracking.

### Commit History Management

The release process creates a specific commit history pattern:

1.   Version bump commit in `pyproject.toml`
2.   Changelog generation commit for Debian packaging
3.   Merge back to main branch with all release changes

### Tag Management

Git tags follow the semantic versioning format `v{major}.{minor}.{patch}` and are created after successful building and testing. Tags serve as the authoritative version markers for release identification.

**Sources:**[.github/workflows/release.yml 245-272](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L245-L272)

### Branch Cleanup

The release workflow includes automatic cleanup of temporary branches after successful merge back to the main branch, maintaining repository hygiene.

**Sources:**[.github/workflows/release.yml 350-375](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/release.yml#L350-L375)
