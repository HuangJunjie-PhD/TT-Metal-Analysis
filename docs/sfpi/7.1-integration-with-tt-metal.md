This document explains how the SFPI toolchain is consumed by the tt-metal repository, including version management, CMake-based dependency fetching, issue tracking, and debugging workflows. This covers the distribution and consumption side of SFPI releases. For information about building SFPI releases, see [Building the SFPI Toolchain](https://deepwiki.com/tenstorrent/sfpi/5-building-the-sfpi-toolchain). For information about writing SFPU kernels using the SFPI API, see [Writing SFPU Kernels](https://deepwiki.com/tenstorrent/sfpi/7.2-writing-sfpu-kernels).

## Overview

The tt-metal repository consumes pre-built SFPI toolchain releases via CMake's FetchContent mechanism. Version management is centralized in a shell script that defines the required SFPI version, architecture-specific download URLs, and checksum verification hashes. The integration supports multiple package formats (`.txz`, `.deb`, `.rpm`) across different architectures (x86_64, aarch64) and Linux distributions (Debian-based, Fedora-based).

## Release Artifact Distribution

SFPI releases are published to GitHub as versioned releases containing multiple artifact formats. The release process creates platform-specific packages with appropriate dependencies and checksums for integrity verification.

**Sources:**[scripts/build.sh 49-115](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L49-L115)[scripts/release.sh 29-60](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L29-L60)[README.md 135-161](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L135-L161)

### Release Package Formats

The SFPI build system generates three package formats to support different deployment scenarios:

| Format | Extension | Use Case | Distribution Method |
| --- | --- | --- | --- |
| Tarball | `.txz` | Universal, CMake FetchContent | Direct extraction to target directory |
| Debian | `.deb` | Debian/Ubuntu systems | `dpkg -i` or `apt install` |
| RPM | `.rpm` | Fedora/RHEL/AlmaLinux systems | `rpm -i` or `dnf install` |

Each package includes:

*   SFPI header files from `include/` directory [scripts/release.sh 49](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L49-L49)
*   Compiled RISC-V toolchain binaries (stripped) [scripts/release.sh 51-52](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L51-L52)
*   Version information in `sfpi/README.txt`[scripts/build.sh 113](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L113-L113)
*   Platform-specific dependencies resolved from shared library requirements [scripts/release.sh 63-89](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L63-L89)

**Sources:**[scripts/release.sh 59-156](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L59-L156)[README.md 122-128](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L122-L128)

## Version Management via sfpi-version.sh

The canonical source for SFPI version information in tt-metal is the `sfpi-version.sh` script. This script serves multiple roles: defining release metadata, computing platform-specific download URLs, and providing version information in both shell and CMake syntax.

**Sources:**[scripts/sfpi-info.sh 1-211](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L1-L211)

### Script Operation Modes

The script operates in four distinct modes:

1.   **CREATE mode**: Generates `sfpi-version` file from checksum files [scripts/sfpi-info.sh 40-76](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L40-L76)

    *   Scans directories for `.sha256` files
    *   Extracts version, architecture, and distribution from filenames
    *   Produces shell-sourceable variable definitions

2.   **VERSION mode**: Accepts version string and computes release metadata [scripts/sfpi-info.sh 78-81](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L78-L81)

    *   Used by release scripts during package creation
    *   Sets `sfpi_version` and `sfpi_hashtype` directly

3.   **SHELL/CMAKE modes**: Reads `sfpi-version` and emits platform-specific variables [scripts/sfpi-info.sh 83-140](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L83-L140)

    *   Detects host architecture via `uname -m`
    *   Detects distribution via `/etc/os-release`
    *   Computes download URL: `$sfpi_repo/releases/download/$sfpi_version/sfpi_${sfpi_version}_${sfpi_arch}_${sfpi_dist}.$sfpi_pkg`
    *   Looks up integrity hash from stored values

4.   **BUILD mode**: Clones repository and builds toolchain from source [scripts/sfpi-info.sh 114-211](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L114-L211)

    *   Alternative to downloading pre-built releases
    *   Clones at specified version tag
    *   Executes `scripts/build.sh --test-tt` and `scripts/release.sh --txz-only`

**Sources:**[scripts/sfpi-info.sh 26-141](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L26-L141)

### Version String Format

Version strings follow a structured format computed during the build process:

```
[owner-]version[-branch]
```

*   Base version from `git describe --tags`[scripts/build.sh 53-62](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L53-L62)
*   Branch suffix added if not on `main` branch or not at tagged commit [scripts/build.sh 91-99](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L91-L99)
*   Owner prefix added if repository URL is not `tenstorrent`[scripts/build.sh 101-106](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L101-L106)

**Sources:**[scripts/build.sh 48-108](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/build.sh#L48-L108)

## CMake FetchContent Integration

The tt-metal repository uses CMake's `FetchContent` module to automatically download and extract SFPI releases. This approach ensures reproducible builds with verified toolchain versions.

**Sources:**[README.md 135-161](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L135-L161)

### CMake Configuration Example

The typical CMake integration pattern:

`# Source version informationexecute_process(    COMMAND ${CMAKE_CURRENT_SOURCE_DIR}/../sfpi-version.sh CMAKE    OUTPUT_VARIABLE SFPI_VARS)eval(${SFPI_VARS}) # Declare and fetch SFPIinclude(FetchContent)FetchContent_Declare(    sfpi    URL ${SFPI_url}/${SFPI_filename}    URL_HASH ${SFPI_HASHTYPE}=${SFPI_hash}    SOURCE_DIR ${INSTALL_LOCATION})FetchContent_MakeAvailable(sfpi)`
Key variables provided by `sfpi-version.sh CMAKE`:

*   `SFPI_version`: Version string (e.g., `1.2.3`, `1.2.3-feature-branch`)
*   `SFPI_repo`: GitHub repository URL
*   `SFPI_url`: Download base URL (repo + `/releases/download/` + version)
*   `SFPI_filename`: Platform-specific filename
*   `SFPI_HASHTYPE`: Hash algorithm (`SHA256`)
*   `SFPI_hash`: Expected checksum for integrity verification

**Sources:**[README.md 138-156](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L138-L156)[scripts/sfpi-info.sh 123-139](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L123-L139)

## Debugging Integration

tt-metal provides debugging facilities that interact with the SFPI toolchain to generate diagnostic information for issue reporting.

**Sources:**[README.md 28-36](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L28-L36)

### Debugging Workflow

When encountering compiler errors (ICE - Internal Compiler Error) or unexpected behavior:

1.   **Enable map file generation**: `export TT_METAL_KERNEL_MAP=1`[README.md 31](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L31-L31)

    *   Instructs build system to preserve intermediate files
    *   Generates `.map` files with symbol tables and link information

2.   **Capture execution output**: `pytest ... |& tee bug.log`[README.md 32](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L32-L32)

    *   Saves complete compiler diagnostic messages
    *   Records build commands with full paths

3.   **Collect build artifacts**: `cp bug.log ~/.cache/tt-metal-cache`[README.md 33](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L33-L33)

    *   Centralizes all relevant files
    *   Includes compiled objects, assembly, and logs

4.   **Create reproducible test case**: `tar cf bug.tar -C ~/.cache tt-metal-cache`[README.md 34](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L34-L34)

    *   Bundles complete build state
    *   Enables offline debugging by SFPI developers

5.   **File issue**: Upload to `https://github.com/tenstorrent/tt-metal` with `sfpi` label [README.md 23-25](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L23-L25)

**Sources:**[README.md 28-36](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L28-L36)

## Issue Tracking and Feedback Loop

SFPI issue tracking is centralized in the tt-metal repository to provide a unified interface for users. Issues are tagged with the `sfpi` label to route them to the appropriate development team.

**Sources:**[README.md 23-25](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L23-L25)

### Issue Reporting Guidelines

For effective issue resolution, include:

1.   **Reproducer**: Complete test case or minimal example demonstrating the issue
2.   **Environment**: SFPI version from `build/version` or package metadata
3.   **Architecture**: Host architecture and target Tensix core (Wormhole, Blackhole, Grayskull)
4.   **Build artifacts**: Tarball created following debugging workflow above
5.   **Expected vs. actual behavior**: Clear description of the problem

**Sources:**[README.md 23-36](https://github.com/tenstorrent/sfpi/blob/ed989b9d/README.md?plain=1#L23-L36)

## Build System Integration Points

The SFPI toolchain integrates with tt-metal's build system at multiple levels:

### Compiler Path Configuration

The compiler path is determined by examining the installation location:

```
$INSTALL_LOCATION/compiler/bin/riscv32-unknown-elf-gcc
$INSTALL_LOCATION/compiler/bin/riscv32-unknown-elf-g++
$INSTALL_LOCATION/compiler/bin/riscv32-unknown-elf-as
$INSTALL_LOCATION/compiler/bin/riscv32-unknown-elf-ld
```

Header files are located at:

```
$INSTALL_LOCATION/include/sfpi.h
$INSTALL_LOCATION/include/sfpi_fp16.h
$INSTALL_LOCATION/include/lltt.h
$INSTALL_LOCATION/include/{wormhole,blackhole,grayskull}/sfpi_hw.h
$INSTALL_LOCATION/include/{wormhole,blackhole,grayskull}/sfpi_imp.h
$INSTALL_LOCATION/include/{wormhole,blackhole,grayskull}/sfpi_lib.h
```

### Build Cache Location

Compiled kernel artifacts are cached at `~/.cache/tt-metal-cache/` to accelerate incremental builds and facilitate debugging. When `TT_METAL_KERNEL_MAP=1` is set, this directory contains complete compilation traces.

## Continuous Integration Pipeline

The SFPI CI pipeline ensures all releases are tested before publication. The workflow executes on every push and for tagged releases:

**Sources:**[.github/workflows/build-sfpi.yaml 1-113](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L1-L113)

### CI Build Stages

1.   **Environment Setup**[.github/workflows/build-sfpi.yaml 53-75](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L53-L75)

    *   Checkout repository with full submodule recursion
    *   Configure Git safe directories
    *   Install build dependencies (autoconf, automake, bison, flex, gcc, g++, etc.)
    *   Install FPM for package generation

2.   **Build Execution**[.github/workflows/build-sfpi.yaml 76-79](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L76-L79)

    *   Run `scripts/build.sh --gdb --tt-built`
    *   Build DejaGnu test infrastructure with `--dejagnu`
    *   Version string from git tags recorded in `build/version`

3.   **Test Validation**[.github/workflows/build-sfpi.yaml 80-81](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L80-L81)

    *   Execute `scripts/build.sh --test-tt`
    *   Run TT-specific compiler tests
    *   Verify zero failures, unresolved tests, and errors

4.   **Artifact Generation**[.github/workflows/build-sfpi.yaml 82-83](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L82-L83)

    *   Run `scripts/release.sh --ci --tt-built`
    *   Create `.txz`, `.deb`, `.rpm` packages with checksums
    *   Strip binaries to reduce package size

5.   **Upload to GitHub**[.github/workflows/build-sfpi.yaml 84-112](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L84-L112)

    *   Upload build artifacts with zero compression (already compressed)
    *   Upload test results for analysis
    *   Upload configuration summary with version/arch/dist information

**Sources:**[.github/workflows/build-sfpi.yaml 43-113](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L43-L113)

### Matrix Build Configuration

The CI workflow builds across multiple configurations:

`arch: x86_64, aarch64box: none, Alma`
*   `box: none` → Ubuntu 22.04 native environment
*   `box: Alma` → AlmaLinux container (`manylinux_2_34_*`) for RPM testing

This produces 4 build variants per commit, ensuring compatibility across deployment environments.

**Sources:**[.github/workflows/build-sfpi.yaml 26-51](https://github.com/tenstorrent/sfpi/blob/ed989b9d/.github/workflows/build-sfpi.yaml#L26-L51)
