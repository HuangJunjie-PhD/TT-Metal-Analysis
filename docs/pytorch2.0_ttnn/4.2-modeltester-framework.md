## Purpose and Scope

The ModelTester framework provides a standardized base class for testing PyTorch models with the torch_ttnn backend. It enforces a consistent interface where model-specific tests only need to implement model loading and input preparation logic, while the framework handles test execution orchestration, mode configuration (train/eval), and integration with the pytest fixture system.

For information about the broader testing infrastructure including pytest fixtures and device management, see [pytest Infrastructure and Fixtures](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.1-pytest-infrastructure-and-fixtures). For details on test generation from metrics, see [Automated Test Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.4-automated-test-generation).

## Base Class Architecture

The `ModelTester` class is defined in [tests/utils.py 15-153](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L15-L153) and serves as an abstract base class that enforces a standard interface for all model tests in the repository. All model-specific test implementations inherit from this class and follow the naming convention `ThisTester` to provide model loading and input preparation logic.

### Constructor and Initialization

The constructor [tests/utils.py 16-23](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L16-L23) accepts three parameters:

*   `model_name`: String identifier for the model being tested (e.g., "BERT", "ResNet18", "Mnist")
*   `mode`: Either "train" or "eval" to control test behavior
*   `batch_size`: Number of samples to process (default 1)

The constructor validates that `mode` is either "train" or "eval" [tests/utils.py 17-18](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L17-L18) then immediately calls the abstract methods `_load_model()`[tests/utils.py 21](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L21-L21) and `_load_inputs()`[tests/utils.py 23](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L23-L23) to initialize the model and test data. This ensures that all required components are available when the tester instance is created.

### Abstract Methods Contract

Two abstract methods must be implemented by any concrete subclass:

| Method | Signature | Purpose | Return Type |
| --- | --- | --- | --- |
| `_load_model()` | `def _load_model(self)` | Load and configure the specific model | `torch.nn.Module` |
| `_load_inputs()` | `def _load_inputs(self, batch_size)` | Generate or load test inputs | Tensor, Dict, or Sequence |

These methods are defined at [tests/utils.py 25-29](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L25-L29) and will raise `NotImplementedError` if not overridden.

Sources: [tests/utils.py 15-29](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L15-L29)

## Model and Input Configuration Methods

The framework provides four configuration methods that prepare models and inputs for their respective modes:

### Evaluation Mode Configuration

For evaluation, the model is set to `eval()` mode and inputs are returned unchanged.

### Training Mode Configuration

Training mode configuration is more complex:

*   Sets model to `train()` mode
*   Zeros gradients via `zero_grad()`
*   **Eliminates randomness from Dropout** by setting `layer.p = 0` for all `torch.nn.Dropout` layers [tests/utils.py 40-42](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L40-L42)
*   Sets `requires_grad=True` on input tensors [tests/utils.py 55](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L55-L55)

The dropout elimination ensures deterministic behavior when comparing native PyTorch execution against torch_ttnn compiled execution. Setting `requires_grad` on inputs allows gradient-based verification (explained below).

Sources: [tests/utils.py 31-61](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L31-L61)

## Test Execution Flow

The framework provides three primary execution methods:

### Main Entry Point

This method dispatches to either `test_model_train()` or `test_model_eval()` based on the mode specified in the constructor.

### Evaluation Execution

The evaluation flow [tests/utils.py 135-144](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L135-L144):

1.   Sets random seed to 0 for reproducibility
2.   Configures model for evaluation
3.   Optionally compiles model with torch_ttnn backend
4.   Runs model on inputs
5.   Returns model outputs directly

### Training Execution

The training flow [tests/utils.py 116-133](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L116-L133):

1.   Sets random seed to 99 for reproducibility
2.   Configures model and inputs for training
3.   Optionally compiles model
4.   Executes forward pass
5.   Applies fake loss function (`torch.mean` of outputs) [tests/utils.py 78-91](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L78-L91)
6.   Executes backward pass via `loss.backward()`
7.   Returns **input gradients** rather than outputs

**Rationale for Input Gradient Verification**: Instead of checking gradients for every model parameter (which is expensive for large models), the framework only verifies the gradient of the input tensor [tests/utils.py 93-111](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L93-L111) Since input gradients depend on all other gradients via the chain rule, any incorrect gradient computation in the model will manifest in the input gradient. This provides comprehensive verification with minimal computational cost.

Sources: [tests/utils.py 78-152](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L78-L152)

## Helper Methods

### Model Compilation

Wraps `torch.compile()` with the `torch_ttnn.backend` and provided options.

### Model Execution

Handles three input types:

*   `Mapping` (dict): Unpacks as keyword arguments via `model(**inputs)`
*   `Sequence` (list/tuple): Unpacks as positional arguments via `model(*inputs)`
*   Single tensor: Passes directly via `model(inputs)`

This flexibility accommodates different model input conventions (e.g., transformers often use dict inputs, vision models use single tensors).

Sources: [tests/utils.py 63-76](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L63-L76)

## Integration with Pytest Fixtures

The ModelTester framework integrates with the pytest fixture system through the `compile_and_run` autouse fixture defined in [tests/conftest.py 191-410](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L191-L410)

### Communication via record_property

Test functions use pytest's `record_property` mechanism to pass the ModelTester instance and native execution results to the `compile_and_run` autouse fixture:

The `record_property` call stores data in `request.node.user_properties` as a list of tuples. The `compile_and_run` fixture converts this to a dictionary at [tests/conftest.py 235](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L235-L235) using `dict(request.node.user_properties)` and then checks if the key `"torch_ttnn"` exists at [tests/conftest.py 261](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L261-L261) If present, it unpacks the tuple into `model_tester` and `outputs` variables at [tests/conftest.py 262](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L262-L262)

This mechanism allows test functions to remain simple while enabling the fixture to perform complex compilation and comparison logic automatically.

### Compilation and Execution

After validation, the fixture [tests/conftest.py 269-348](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L269-L348):

1.   Creates a `TorchTtnnOption` with device and configuration parameters
2.   Handles `load_params_once` optimization conflicts with native integration
3.   Optionally moves model and inputs to ttnn device for native integration
4.   Executes `model_tester.test_model(as_ttnn=True, option=option)` multiple times
5.   Collects timing metrics (first iteration vs. warm iterations)
6.   Calculates accuracy by comparing native vs. compiled outputs using `calculate_accuracy()`

### Fallback on Compilation Failure

If compilation fails [tests/conftest.py 367-394](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L367-L394) the fixture:

1.   Catches the exception
2.   Resets torch dynamo cache
3.   Retries with `option.bypass_compile = True` to collect aten operation metrics
4.   If the bypass run succeeds but compilation failed, raises a descriptive error
5.   Honors `@pytest.mark.compilation_xfail` marker to allow expected failures

This fallback mechanism ensures that even failed compilations produce useful diagnostic metrics about which operations were attempted.

## Mode-Specific Behavior

The framework behaves differently based on the mode parameter:

| Aspect | Eval Mode | Train Mode |
| --- | --- | --- |
| Model configuration | `model.eval()` | `model.train()`, zero gradients, disable dropout |
| Input configuration | Unchanged | Set `requires_grad=True` |
| Random seed | 0 | 99 |
| Decorator | `@torch.no_grad()` | None (gradients enabled) |
| Forward pass | Standard inference | Standard inference |
| Backward pass | Not executed | `loss.backward()` with `torch.mean` loss |
| Return value | Model outputs | **Input gradients** (`inputs.grad`) |
| Purpose | Test inference accuracy | Test gradient computation correctness |

Sources: [tests/utils.py 116-144](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L116-L144)

### Training Mode Rationale

The training mode implementation makes specific design choices for efficiency:

**Dropout Elimination**[tests/utils.py 35-42](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L35-L42): Setting dropout probability to 0 ensures that two runs (native PyTorch and torch_ttnn) produce identical results for comparison. Without this, dropout's randomness would cause differences unrelated to compilation correctness.

**Input Gradient Verification**[tests/utils.py 93-111](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L93-L111): Rather than checking all parameter gradients (expensive for large models), the framework only checks input gradients. The documentation explains: "Based on the chain rule, the input gradient depends on all other gradients, so any incorrect gradient computation should reflect here."

**Mean as Loss Function**[tests/utils.py 78-91](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L78-L91): Using `torch.mean(outputs)` as a universal loss function simplifies testing. While not optimal for actual training, it produces a scalar loss suitable for backpropagation and is sufficient to verify gradient computation correctness.

Sources: [tests/utils.py 31-133](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L31-L133)

### Common Patterns Across All Implementations

All examples follow these standard patterns:

1.   **Class naming**: Use `ThisTester` as the class name inheriting from `ModelTester`
2.   **Method implementation**: Override `_load_model()` and `_load_inputs(batch_size)`
3.   **State management**: Store preprocessing objects (tokenizers, processors, image data) as instance variables
4.   **Data type consistency**: Convert models and inputs to `torch.bfloat16`
5.   **Parametrization**: Use `@pytest.mark.parametrize` for mode and batch_size variants
6.   **Markers**: Apply `@pytest.mark.converted_end_to_end` for fully converted models
7.   **Record properties**: Set `model_name` and `mode` via `record_property()`
8.   **Fixture integration**: Pass `(tester, results)` tuple via `record_property("torch_ttnn", ...)`

### Input Type Patterns

The framework supports three input patterns via the `run_model()` method [tests/utils.py 70-76](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L70-L76):

*   **Single Tensor**: Vision models (MNIST, ResNet, U-Net) - passed directly as `model(inputs)`
*   **Dict (Mapping)**: Transformer models (BERT, GPT-2, Llama) - unpacked as `model(**inputs)`
*   **Sequence**: Less common, but supported - unpacked as `model(*inputs)`

## Supporting Utilities

The framework relies on several utility functions also defined in [tests/utils.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py):

### Accuracy Calculation

The `calculate_accuracy()` function [tests/utils.py 476-510](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L476-L510) recursively processes nested structures:

*   Tensors: Computes PCC (Pearson Correlation Coefficient) via `comp_pcc()`
*   Lists/Tuples: Recursively processes each element and returns mean accuracy
*   Dicts: Recursively processes each value and returns mean accuracy

This handles models with complex outputs (e.g., transformers returning dicts of logits, hidden states, etc.).

### Input Batching

The `repeat_inputs()` function [tests/utils.py 155-169](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L155-L169) replicates inputs to match a desired batch size:

*   For dict inputs: Repeats each tensor value along dimension 0
*   For tensor inputs: Repeats along dimension 0
*   Raises TypeError for unsupported input types

This is useful when a model expects batched inputs but test data is unbatched.

### Assertion Helpers

Three assertion functions validate tensor accuracy:

*   `assert_with_pcc()`[tests/utils.py 266-272](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L266-L272): Validates Pearson correlation coefficient
*   `assert_with_ulp()`[tests/utils.py 395-440](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L395-L440): Validates Units of Least Precision
*   `assert_allclose()`[tests/utils.py 455-471](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L455-L471): Validates absolute/relative tolerance

These are used in operation-level tests (see [Operation Lowering Test Coverage](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6-operation-lowering-test-coverage)) but not typically in ModelTester-based tests, which use `calculate_accuracy()` instead.

Sources: [tests/utils.py 155-510](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py#L155-L510)

## Relationship to Operation Tests

While ModelTester is used for full model tests, the repository also contains operation-level tests (e.g., [tests/lowering/eltwise/binary/test_pow_scalar.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py)[tests/lowering/eltwise/unary/test_hardsigmoid.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/unary/test_hardsigmoid.py)) that do NOT use ModelTester. These tests:

*   Directly create small `torch.nn.Module` classes for specific operations
*   Manually call `torch.compile()` with torch_ttnn backend
*   Use assertion helpers like `assert_with_ulp()` for validation
*   Do not use the `compile_and_run` fixture

The ModelTester framework is specifically designed for end-to-end model testing, not isolated operation testing. For operation testing patterns, see [Element-wise and Pointwise Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.1-element-wise-and-pointwise-operation-tests) and [Tensor Manipulation Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.2-tensor-manipulation-operation-tests).
