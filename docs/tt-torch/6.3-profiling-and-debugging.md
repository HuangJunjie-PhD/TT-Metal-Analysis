## Code Style and Standards

tt-torch follows strict code style and formatting requirements enforced through automated pre-commit hooks.

### Pre-commit Hooks

All contributions must pass pre-commit checks before submission. To set up pre-commit hooks locally:

`pip install pre-commitpre-commit install`
These hooks enforce:

*   Code formatting (black)
*   Import sorting
*   Line length constraints
*   SPDX license header presence
*   Trailing whitespace removal

### SPDX License Headers

All source files must contain SPDX license headers. The standard format is:

`# SPDX-FileCopyrightText: (c) 2024 Tenstorrent AI ULC## SPDX-License-Identifier: Apache-2.0`
These headers are automatically checked by CI and can be added by pre-commit hooks.

Sources: [.github/workflows/spdx.yml 1-20](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/spdx.yml#L1-L20)[tests/torch/test_interpolation.py 1-4](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_interpolation.py#L1-L4)

### Python Coding Conventions

*   Use consistent naming: snake_case for functions and variables, PascalCase for classes
*   Document functions, classes, and modules with docstrings
*   Use type hints where appropriate
*   Keep functions focused on a single responsibility
*   Follow PEP 8 guidelines

## Testing Requirements

Testing is critical for ensuring code quality in tt-torch. All contributions must include appropriate tests.

### Test Structure

Tests should follow the established patterns:

1.   For PyTorch models, create a class that inherits from `ModelTester`
2.   For ONNX models, create a class that inherits from `OnnxModelTester`
3.   Implement the required abstract methods: 
    *   `_load_model()`: Load the model to be tested
    *   `_load_inputs()`: Prepare input data
    *   `_extract_outputs()` (optional): Extract output tensors

### Test Parameters and Configuration

Use pytest's parametrization to test various configurations:

`@pytest.mark.parametrize(    "op_by_op",    [OpByOpBackend.STABLEHLO, OpByOpBackend.TORCH, None],    ids=["op_by_op_stablehlo", "op_by_op_torch", "full"],)def test_your_model(record_property, op_by_op):    cc = CompilerConfig()    if op_by_op:        cc.compile_depth = CompileDepth.EXECUTE_OP_BY_OP        cc.op_by_op_backend = op_by_op        # Test implementation`
### Verification Requirements

Tests should verify model outputs using appropriate validation methods:

*   For PyTorch models, use `verify_module()` or `ModelTester.test_model()` with appropriate tolerance settings
*   For ONNX models, use `OnnxModelTester.test_model()`

For accuracy validation, set appropriate tolerance thresholds:

`verify_module(    model,    inputs=[input_tensor],    compiler_config=cc,    required_atol=0.07,  # Absolute tolerance)`

## Pull Request Process

### PR Workflow Diagram

### PR Requirements

1.   **Branch Naming**: Use descriptive names that reflect the changes being made
2.   **PR Description**: Clearly describe the changes, the motivation, and any related issues
3.   **CI Checks**: All automated checks must pass
4.   **Code Reviews**: PRs require approval from at least one reviewer
5.   **Test Coverage**: New functionality must include tests

### CI/CD Pipeline

The CI/CD pipeline for PRs includes:

| Stage | Description | Required Status |
| --- | --- | --- |
| Pre-commit | Runs pre-commit hooks to check formatting | Required |
| SPDX | Verifies license headers | Required |
| Docker Build | Builds a Docker image for testing | Required |
| Build | Compiles the code | Required |
| Tests | Runs unit tests | Required |
| Full Model Tests | Runs integration tests with full models | Required |

tt-torch is licensed under the Apache License 2.0. All contributions must comply with this license.

### License Requirements

1.   Every source file must include the SPDX license header

3.   Third-party dependencies must have compatible licenses

## Contribution Workflow

### Creating Custom Decompositions

When implementing new operators or extending existing functionality, you may need to create custom decompositions:

Example of adding a custom decomposition:

1.   Create a decomposition function in [tt_torch/dynamo/decompositions.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/decompositions.py)
2.   Add the function to the `_get_custom_decopositions()` table

### Adding Tests to CI

When adding new models or tests, follow these guidelines for including them in CI:

1.   For fully functional tests:

    *   Add to "nightly full model execute list" in `.github/workflows/run-full-model-execution-tests-nightly.yml`
    *   Add to "weekly op-by-op-flow list" in `.github/workflows/run-op-by-op-flow-tests-weekly.yml`

2.   For models with PCC/ATOL errors:

    *   Open a GitHub issue with the `models_pcc_issue` label
    *   Reference the issue in the test code
    *   Add to the same lists as fully functional tests

3.   For models with compilation failures:

    *   Add to "nightly op-by-op flow list" in `.github/workflows/run-op-by-op-flow-tests-nightly.yml`
    *   Generate and include `results/models_op_per_op.xlsx` in your PR
