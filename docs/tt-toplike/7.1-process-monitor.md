# Process Monitor

Relevant source files
*   [docs/PROCESS_MONITORING.md](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1)
*   [src/workload/process_monitor.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs)

The Process Monitor is a Linux-specific subsystem responsible for identifying which host processes are actively utilizing Tenstorrent hardware [docs/PROCESS_MONITORING.md 1-3](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L1-L3) It operates by inspecting the Linux `/proc` filesystem (procfs) to detect open file descriptors, memory-mapped regions, and hugepage allocations associated with Tenstorrent device drivers [src/workload/process_monitor.rs 7-13](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L7-L13)

## Detection Logic and Implementation

The monitoring logic is encapsulated in the `ProcessMonitor` struct, which maintains the current state of hardware-to-process associations [src/workload/process_monitor.rs 59-65](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L59-L65)

### Scanning Mechanism

The monitor performs a scan of all system processes via `procfs::process::all_processes()`[src/workload/process_monitor.rs 92-95](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L92-L95) For each accessible process, the `scan_process` function evaluates three primary indicators:

1.   **Device File Descriptors**: It iterates through `/proc/[pid]/fd/` looking for paths starting with `/dev/tenstorrent/`. The trailing integer in the path is parsed as the device index [src/workload/process_monitor.rs 116-131](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L116-L131)
2.   **Hugepage Allocation**: It inspects `/proc/[pid]/maps` for entries containing `hugepages-1G`, `pagesize-1GB`, or generic `hugepages` (assumed to be 2MB) [src/workload/process_monitor.rs 138-150](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L138-L150)
3.   **Memory Mappings**: It checks memory map paths for Tenstorrent-specific keywords such as `tenstorrent`, `tt_`, `blackhole`, `wormhole`, or `grayskull`[src/workload/process_monitor.rs 151-159](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L151-L159)

## Execution and Performance

*   **Cadence**: The scan runs every 2 seconds, which is independent of the telemetry refresh rate [docs/PROCESS_MONITORING.md 43](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L43-L43)
*   **Latency**: Scans typically complete in under 10ms [docs/PROCESS_MONITORING.md 44](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L44-L44)
*   **Permissions**: The monitor silently skips processes where access is denied (e.g., processes owned by other users when `tt-toplike` is run as non-root) [src/workload/process_monitor.rs 97](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L97-L97) If `/proc` is entirely inaccessible, a warning is logged but the application continues [src/workload/process_monitor.rs 101-104](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L101-L104)

## UI Integration and Display

The TUI displays process information in the **Hardware Usage** section [docs/PROCESS_MONITORING.md 7-11](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L7-L11)

*   **Display Limits**: To prevent UI bloat, the interface typically displays up to 2 processes per device [docs/PROCESS_MONITORING.md 51](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L51-L51)
*   **Dynamic Visibility**: The "Hardware Usage" block is hidden if no Tenstorrent-related processes are detected [docs/PROCESS_MONITORING.md 21](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L21-L21)
*   **Shared Resources**: Processes that have Tenstorrent memory mappings but no direct file descriptors (common in certain containerized or shared-memory workloads) are tracked in `shared_processes`[src/workload/process_monitor.rs 62-64](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/process_monitor.rs#L62-L64)

## Feature Configuration

Process monitoring is enabled via the `linux-procfs` Cargo feature, which is included in the default feature set [docs/PROCESS_MONITORING.md 31](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L31-L31)

`# Enable process monitoring (Default)cargo build --features tui # Disable process monitoringcargo build --no-default-features --features tui,json-backend`

## Limitations

*   **Non-root Access**: When run as a standard user, `tt-toplike` can only inspect file descriptors and maps for processes owned by that same user [docs/PROCESS_MONITORING.md 49](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L49-L49)
*   **Containers**: The monitor identifies the host-side process (e.g., `containerd-shim` or `python3` running in the host PID namespace) rather than the internal containerized process names if namespaces are isolated [docs/PROCESS_MONITORING.md 50](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/PROCESS_MONITORING.md?plain=1#L50-L50)
