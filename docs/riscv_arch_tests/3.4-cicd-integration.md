## Container Integration

The CI pipeline uses a containerized environment managed through the GitLab container registry.

### Container Image Management

The container image is specified using a content-addressable hash to ensure reproducible builds:

```
$CI_REGISTRY_IMAGE:53a0f902fa8ee7fa34c1b16d65c248f6ea1cd25ca1058c5f492f9f617ea4f2a4
```

This approach provides:

*   **Immutable environments**: Hash-based image references prevent unexpected changes
*   **Reproducible builds**: Same image hash guarantees identical container environment
*   **Version control**: Image updates require explicit hash changes

Sources: [infra/container/gitlab-container.yml 1-3](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/container/gitlab-container.yml#L1-L3)

## Pipeline Stages

The CI configuration defines two primary stages for different types of automated activities:

| Stage | Purpose | Execution Trigger |
| --- | --- | --- |
| `test` | Core test execution | Every MR and branch push |
| `scheduled_event` | Periodic maintenance | Scheduled intervals |

The `scheduled_event` stage is designed for periodic tasks such as checking for updates from public repositories and performing maintenance operations.
