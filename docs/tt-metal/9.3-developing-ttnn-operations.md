This document provides a comprehensive guide for developers adding new operations to the TTNN library. It covers the architecture of TTNN operations, the implementation process from Python API to device kernels, and testing requirements.

For information about using existing TTNN operations in models, see [Model Development Workflow](https://deepwiki.com/tenstorrent/tt-metal/7.1-model-development-workflow). For details on the underlying device operation infrastructure, see [Device Abstraction Layer](https://deepwiki.com/tenstorrent/tt-metal/2.3-device-abstraction-layer) and [Program and Kernel System](https://deepwiki.com/tenstorrent/tt-metal/2.4-program-and-kernel-system).

## Operation Architecture Overview

TTNN operations follow a layered architecture spanning from Python bindings through C++ device operations down to hardware SFPU kernels. This layered design ensures efficient execution on Tenstorrent hardware with opportunities for optimization and multi-device scaling.

**TTNN Operation Architecture**

This architecture enables operations written in C++ to be accessed directly from Python, validated against golden PyTorch computations, dispatched through a device operation layer, and compiled down to efficient SFPU machine instructions.

## Operation Types

TTNN operations fall into diverse categories based on their functional and execution requirements.

| Operation Type | Description | Examples | Implementation Complexity |
| --- | --- | --- | --- |
| **Unary** | Single tensor input | `exp`, `relu`, `abs` | Low - uses standard SFPU kernels |
| **Binary** | Two tensor/scalar inputs | `add`, `multiply`, `atan2` | Medium - requires broadcasting |
| **Data Movement** | Reshaping and reordering | `concat`, `split`, `slice` | Medium - reader/writer heavy |
| **Backward** | Gradient computation | `gelu_bw`, `tanh_bw` | High - complex math, multi-input |
| **Reduction** | Reduce dimension operations | `sum`, `mean`, `argmax` | High - inter-core communication |
| **CCL** | Collective communication ops | `all_gather`, `reduce_scatter` | High - mesh and fabric intensive |

**Sources:**[docs/source/ttnn/ttnn/api.rst 112-371](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/docs/source/ttnn/ttnn/api.rst#L112-L371)[ttnn/cpp/ttnn/operations/eltwise/unary/unary_nanobind.cpp 27-86](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/eltwise/unary/unary_nanobind.cpp#L27-L86)

## Implementation Components

### 1. Python Bindings (Nanobind)

Operations are exposed to Python via `nanobind`, which provides:

*   **Overload Dispatch:** Different C++ overloads, e.g., Tensor-Tensor vs Tensor-Scalar, are bound to a single Python function name.
*   **Docstrings:** Rich, Sphinx-style docstrings are provided for API documentation and usability.
*   **GIL Management:** Releasing the Python Global Interpreter Lock during execution for efficiency.

The bindings are implemented inside files like `unary_nanobind.cpp` and `binary_nanobind.cpp`. The common pattern uses `ttnn::bind_function` providing Python-callable wrappers around C++ implementations.

### 2. Device Operation (C++)

The device operation layer encapsulates the core logic of validation, program creation, and launching for the operation. This includes:

*   **Validation:** Check tensor allocations and shape compatibility with helpers like `CheckDeviceBufferIsAllocated`.
*   **Program Hashing:** Compute a unique program hash from operation attributes and tensor arguments to assist caching.
*   **Program Factory Invocation:** Generate the hardware execution plan (a `tt::tt_metal::Program`), often producing reader, compute, and writer kernels.

`device_operation.hpp` defines the templates and utilities to implement device operations and integrate with the TT-Metal runtime.

### 3. Program Factory

The program factory produces an executable program representation. Key tasks:

*   **Circular Buffer (CB) Setup:** Using `cb_descriptor_from_sharded_tensor` to automatically configure L1 circular buffers for sharded tensors simplifies kernel execution setup.
*   **Kernel Creation:** Create data movement kernels (reader/writer) and compute kernels, assembling them into a program descriptor.

This factory encapsulates knowledge of operation dispatch and hardware specifics.

### 4. Mesh Adaptation

To enable multi-device execution on a mesh, TTNN supports adapting single-device operations to mesh-wide workloads. Using functions like `create_mesh_workload_from_workload_factory`, operations are mapped to mesh coordinates producing `CachedMeshWorkload` objects managing per-core execution plans.

## Program Caching and Hashing

To avoid redundant recompilations and program construction:

*   A program hash is computed using operation attributes and tensor arguments by `compute_program_hash`.
*   Operations can customize hashing by implementing `compute_program_hash` themselves.
*   Cached programs are reused when hashes match, improving runtime efficiency.
*   On cache hits, runtime arguments may be overridden without rebuilding the program.

Caching and hash computation are primarily implemented in `device_operation.hpp`.

## Debugging and Inspection

TTNN includes facilities to inspect and debug operations:

*   **Logging:** Detailed debug logs for operation launch, including operation name and program hash.
*   **Reflection:** Operation type names obtained via `ttsl::get_type_name` aid debugging.
*   **Graph Capture:** The framework supports capturing computation graphs with `GraphProcessor`.
*   **Constraint Querying:** Developers can query operation resource constraints without full dispatching.

These capabilities help developers verify correctness and performance behavior during development.

## Summary of Implementation Steps

1.   **Define Operation Attributes and Tensor Arguments**

    *   Implement `operation_attributes_t` and `tensor_args_t` within the operation's namespace.

2.   **Implement Device Operation**

    *   Define validation and program creation using the device operation templates in `device_operation.hpp`.

3.   **Create Program Factory**

    *   Implement program construction with kernels and configure circular buffers using helpers like `cb_descriptor_from_sharded_tensor`.

4.   **Kernel Implementation**

    *   Write reader, writer, and compute kernels in C++ (e.g., inside `reader_*.cpp` or `compute/*.cpp`).

5.   **Bind to Python**

    *   Register the operation with `ttnn::bind_function` in a `nanobind` source file such as `unary_nanobind.cpp`.

6.   **Testing**

    *   Write unit tests (GTests) comparing device operation outputs with golden PyTorch equivalents.
    *   Use tools such as `query_op_constraints` to validate resource usage and limits.
