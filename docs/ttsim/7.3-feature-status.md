# Feature Status

Relevant source files
*   [README.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1)

This page documents the current implementation status of features in ttsim, including which capabilities are fully implemented and bit-accurate, which are partially implemented, and which are planned for future releases. This reference helps developers understand what to expect when running workloads on the simulator.

For configuration requirements and how to enable the simulator, see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration). For detailed error messages and troubleshooting, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference). For information on numerical accuracy guarantees and divergence sources, see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy).

## Overall Implementation Status

ttsim is actively under development. The simulator can successfully run many tt-metal and ttnn examples and tests, as well as numerous LLK (Low-Level Kernel) tests. The core simulation engine is functional, but several features remain incomplete or require additional testing.

**Sources:**[README.md 17-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L17-L19)

## Architecture Support Status

The simulator supports two Tenstorrent chip architectures with different maturity levels:

| Architecture | Status | SOC Descriptor | Library Name |
| --- | --- | --- | --- |
| **Wormhole** | More mature, recommended for development | `wormhole_b0_80_arch.yaml` | `libttsim_wh.so` |
| **Blackhole** | Functional but less mature | `blackhole_140_arch.yaml` | `libttsim_bh.so` |

Both architectures support running tt-metal examples and tests in slow dispatch mode. Wormhole has undergone more extensive testing and is generally more stable.

**Sources:**[README.md 17-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L17-L19)[README.md 45-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L45-L61)

## FPU Opcode Status

The Floating Point Unit (FPU) has different implementation maturity levels across its opcode families:

### Fully Bit-Accurate Opcodes

*   _\_MOV\_ family_*: All move operations are bit-exact
*   **GMPOOL opcodes**: Pooling operations are bit-exact

### Partially Implemented Opcodes

The following opcode families are functional but not yet bit-accurate. They produce approximately correct results but may diverge from silicon in precise bit representation:

*   _\_ELW\_ family_*: Element-wise operations
*   **MVMUL opcodes**: Matrix-vector multiplication operations
*   **GAPOOL opcodes**: Another pooling operation family

These opcodes are planned to be fixed to achieve bit-accuracy to the extent possible.

## Numerical Accuracy Goals and Status

### Design Goal

ttsim is designed to provide **bit-exact** numerical results relative to silicon for all computations. The goal is to match all hardware computations bit-for-bit across all instructions, opcodes, functional units, and special cases, including the precise bit representation of NaNs produced by operations.

**Sources:**[README.md 78-83](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L78-L83)

### Current Accuracy Status

While most code achieves bit-exact results, the following components are known to have accuracy limitations:

#### Fully Bit-Accurate Components

*   Unpacker: Complete bit-accuracy achieved
*   SFPU: Complete bit-accuracy achieved
*   Packer: Complete bit-accuracy achieved
*   FPU MOV* opcodes: Complete bit-accuracy achieved
*   FPU GMPOOL opcodes: Complete bit-accuracy achieved

#### Components Under Development

*   FPU ELW* opcodes: Not yet bit-accurate
*   FPU MVMUL opcodes: Not yet bit-accurate
*   FPU GAPOOL opcodes: Not yet bit-accurate

**Sources:**[README.md 99-102](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L99-L102)

### Divergence Sources

Cases that can produce divergent results from silicon even when implementation is bit-accurate:

| Divergence Source | Description | Mitigation |
| --- | --- | --- |
| Timing-dependent operand order | Computations where operand evaluation order affects result | Use deterministic serialization |
| Hardware entropy sources | Random number generators | Not applicable for deterministic workloads |
| Performance counters | Cycle counters, timers | Expect different values |
| Missing synchronization | Cache flushes, memory fences | Add proper synchronization |
| UndefinedBehavior | ISA specification violations | Avoid undefined operations |
| UnpredictableValue | Non-deterministic hardware states | Avoid unpredictable operations |

For timing-dependent computations, ttsim may evaluate operations in any order permitted by software synchronization, including orders extremely unlikely on silicon. To ensure exact match with silicon, avoid algorithms where runtime operation order affects results (e.g., unserialized floating-point reductions).

**Sources:**[README.md 85-97](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L85-L97)

## Feature Categories and Error Codes

When encountering unimplemented or problematic features, ttsim reports errors with one of the following categories. Understanding these categories helps determine whether a feature will be supported:

### Error Category Definitions

#### ISA-Related Categories

| Category | Meaning | Reference | Action Required |
| --- | --- | --- | --- |
| `UndefinedBehavior` | Operation violates ISA specification | [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) | Fix code to avoid undefined operation |
| `UnpredictableValueUsed` | Using value without defined state | [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) | Initialize values properly |
| `NonContractualBehavior` | Behavior not guaranteed by ISA | [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) | Don't depend on unspecified behavior |

#### Implementation Categories

| Category | Meaning | Future Support | Action Required |
| --- | --- | --- | --- |
| `UntestedFunctionality` | Feature implemented but lacks test coverage | Will be enabled after testing | Report issue, wait for update |
| `UnimplementedFunctionality` | Feature not yet implemented | Planned for future release | Report issue, prioritize if needed |
| `UnsupportedFunctionality` | Feature unlikely to be implemented | Requires strong justification | Consider alternative approach |
| `MissingSpecification` | Feature requires internal specification work | Depends on spec completion | Report use case if critical |

#### Configuration and System Categories

| Category | Meaning | Action Required |
| --- | --- | --- |
| `SystemError` | Operating system error | Check OS configuration |
| `ConfigurationError` | Invalid environment variables or configuration files | Review configuration setup |
| `AssertionFailure` | Internal simulator bug | Report as bug with reproduction steps |

## Summary: Current Feature Maturity

### Production-Ready Features

*   ✅ Slow dispatch mode execution
*   ✅ Wormhole architecture support
*   ✅ Unpacker bit-accuracy
*   ✅ SFPU bit-accuracy
*   ✅ Packer bit-accuracy
*   ✅ FPU MOV* opcodes bit-accuracy
*   ✅ FPU GMPOOL opcodes bit-accuracy
*   ✅ tt-metal examples execution
*   ✅ ttnn tests execution
*   ✅ LLK tests execution

### Features Under Active Development

*   ⚠️ Blackhole architecture (functional but less mature)
*   ⚠️ FPU ELW* opcodes bit-accuracy
*   ⚠️ FPU MVMUL opcodes bit-accuracy
*   ⚠️ FPU GAPOOL opcodes bit-accuracy

### Known Limitations

*   ❌ Fast dispatch mode (not working, slow dispatch required)
*   ❌ Various features categorized as UntestedFunctionality
*   ❌ Various features categorized as UnimplementedFunctionality
*   ❌ Features categorized as UnsupportedFunctionality
*   ❌ Features categorized as MissingSpecification

### Overall Assessment

ttsim is suitable for development and testing of workloads on Wormhole architecture in slow dispatch mode. Most core computational components are bit-accurate, with FPU element-wise operations being the primary area still under development. The simulator can run substantial workloads with good productivity, though some advanced features remain unimplemented.
