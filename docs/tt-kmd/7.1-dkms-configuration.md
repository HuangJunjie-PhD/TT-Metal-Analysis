## Purpose and Scope

This document describes the Dynamic Kernel Module Support (DKMS) configuration for the Tenstorrent kernel driver (`tt-kmd`). DKMS is the recommended installation method for Debian, Ubuntu, Fedora, RHEL, and other Linux distributions that support DKMS. It automates kernel module compilation and installation, ensuring the driver remains functional across kernel updates.

For Alpine Linux installations, see [AKMS Configuration](https://deepwiki.com/tenstorrent/tt-kmd/7.2-akms-configuration). For manual compilation without package management, see [Manual Build with Make](https://deepwiki.com/tenstorrent/tt-kmd/7.3-manual-build-with-make). For runtime module parameter configuration, see [Module Parameters](https://deepwiki.com/tenstorrent/tt-kmd/7.4-module-parameters).

## DKMS Overview

DKMS (Dynamic Kernel Module Support) is a framework that enables automatic rebuilding of kernel modules when a new kernel is installed. For the Tenstorrent driver, DKMS provides:

*   **Automatic Rebuilds**: Module is recompiled for each installed kernel version.
*   **Kernel Update Safety**: Driver remains available after kernel upgrades.
*   **Version Tracking**: Multiple driver versions can coexist for different kernels.
*   **Dependency Management**: Ensures build dependencies (like kernel headers) are met.
*   **Standardized Installation**: Uses distribution package management integration.

The driver module `tenstorrent.ko` is typically installed to `/lib/modules/<kernel-version>/extra/` as specified in the DKMS configuration [dkms.conf 5](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms.conf#L5-L5)

## Version Management

The driver version is managed through a coordinated update process across multiple files to ensure consistency between the build system, package managers, and the kernel module itself.

### Version Synchronization

The version string `"2.9.1-pre"` in `dkms.conf` serves as the primary source of truth for the build system [dkms.conf 2](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms.conf#L2-L2) Inside the driver code, the version is composed from components defined in `module.h` and used in `module.c`:

*   **Major**: `TENSTORRENT_DRIVER_VERSION_MAJOR`
*   **Minor**: `TENSTORRENT_DRIVER_VERSION_MINOR`
*   **Patch**: `TENSTORRENT_DRIVER_VERSION_PATCH`
*   **Suffix**: `TENSTORRENT_DRIVER_VERSION_SUFFIX` (e.g., `-pre`)

The version string is constructed via the `TENSTORRENT_DRIVER_VERSION_STRING` macro [module.c 23-27](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L23-L27) and registered with the kernel via `MODULE_VERSION()`[module.c 31](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L31-L31)

### Version Update Automation

The `release.yml` GitHub workflow automates the synchronization of version files [release.yml 88-118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/release.yml#L88-L118):

1.   Updates `PACKAGE_VERSION` in `dkms.conf`[release.yml 99](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/release.yml#L99-L99)
2.   Updates `modver` in `AKMBUILD`[release.yml 102](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/release.yml#L102-L102)
3.   Updates `TENSTORRENT_DRIVER_VERSION_MAJOR/MINOR/PATCH` and `SUFFIX` in `module.h`[release.yml 105-116](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/release.yml#L105-L116)

The `build_rpms.sh` script includes a sanity check that enforces consistency between `module.h` and `dkms.conf` before allowing a build [tools/build_rpms.sh 19-28](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/tools/build_rpms.sh#L19-L28)

## Post-Installation Hook

The `POST_INSTALL` directive in `dkms.conf` points to `dkms-post-install`[dkms.conf 9](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms.conf#L9-L9) This script is critical for ensuring that user-space applications can interact with the driver immediately after installation.

The script performs two primary actions:

1.   **Rule Deployment**: Copies the Tenstorrent udev rules to the system directory [dkms-post-install 3](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms-post-install#L3-L3)
2.   **Udev Refresh**: Triggers a reload of the udev control interface to apply the new rules without a reboot [dkms-post-install 4](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms-post-install#L4-L4)

**Sources:**[dkms.conf 9](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms.conf#L9-L9)[dkms-post-install 1-5](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms-post-install#L1-L5)

## RPM Package Integration

For RHEL/Fedora-based systems, the `tools/build_rpms.sh` script creates a `tenstorrent-dkms` package that encapsulates the DKMS source and installation logic.

### RPM Build Workflow

1.   **Version Normalization**: RPM does not allow hyphens in the version field. The script converts `-pre` or `-rc1` suffixes to `~pre` or `~rc1` to ensure correct package sorting [tools/build_rpms.sh 30-34](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/tools/build_rpms.sh#L30-L34)
2.   **Source Staging**: Files are filtered using `tools/exclude-from-release` to keep the package lightweight [tools/build_rpms.sh 55](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/tools/build_rpms.sh#L55-L55)
3.   **DKMS Registration**: The RPM `%post` script automatically adds, builds, and installs the module for all kernels currently having headers installed on the system [tools/build_rpms.sh 99-113](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/tools/build_rpms.sh#L99-L113)
4.   **Auto-Load**: The script attempts to `modprobe tenstorrent` immediately after installation [tools/build_rpms.sh 120](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/tools/build_rpms.sh#L120-L120)

## Module Parameters and Configuration

While DKMS manages the build, the driver behavior is controlled via module parameters defined in `module.c`. Users can configure these parameters using a configuration file in `/etc/modprobe.d/`.

A template is provided in `modprobe.d-tenstorrent.conf`:

| Parameter | Default | Code Entity | Purpose |
| --- | --- | --- | --- |
| `max_devices` | 32 | `max_devices` | Maximum chips to support [module.c 36-37](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L36-L37) |
| `dma_address_bits` | 0 | `dma_address_bits` | 0 for automatic detection [module.c 40-41](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L40-L41) |
| `reset_limit` | 10 | `reset_limit` | Boot-time reset retry count [module.c 44-45](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L44-L45) |
| `auto_reset_timeout` | 10 | `auto_reset_timeout` | Seconds for M3 auto-reset [module.c 48-49](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L48-L49) |
| `power_policy` | true | `power_policy` | Enable power management [module.c 52-53](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L52-L53) |
