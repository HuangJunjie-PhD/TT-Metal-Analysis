## Purpose and Scope

This document describes the synchronization primitives provided by the ckernel layer for coordinating operations between different hardware units within a Tensix core. These primitives include hardware semaphores for stage coordination (unpack/math/pack), mutexes for atomic resource access, and memory ordering operations. For higher-level operation sequencing, see [Micro-Operation Programming (MOPs)](https://deepwiki.com/tenstorrent/tt-llk/7-micro-operation-programming-(mops)). For configuration state management, see [Configuration State Management](https://deepwiki.com/tenstorrent/tt-llk/2.3-configuration-state-management).

## Hardware Semaphore Architecture

The Tensix hardware provides 8 hardware semaphores accessed through a special memory region in the PC_BUF address space. Each semaphore maintains a 4-bit counter value (0-15) and supports atomic increment (SEMPOST) and decrement (SEMGET) operations.

## RISC-V Semaphore Operations

### Semaphore Read

Reads the current value of a semaphore (0-15). Returns the semaphore counter value without modifying it.

**Implementation:**[tt_llk_wormhole_b0/common/inc/ckernel.h 202-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L202-L206)

### Semaphore Post (Increment)

Atomically increments the semaphore counter by 1, up to a maximum of 15. Writing 0 (LSB clear) to the semaphore access register triggers the hardware to perform the increment. The operation silently has no effect if the semaphore is already at 15, so the function includes an assertion to prevent misuse.

**Implementation:**[tt_llk_wormhole_b0/common/inc/ckernel.h 208-217](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L208-L217)

### Semaphore Get (Decrement)

Atomically decrements the semaphore counter by 1, down to a minimum of 0. Writing 1 (LSB set) to the semaphore access register triggers the hardware to perform the decrement. The operation silently has no effect if the semaphore is already at 0, so the function includes an assertion to prevent misuse.

**Implementation:**[tt_llk_wormhole_b0/common/inc/ckernel.h 219-228](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L219-L228)

### Atomic Semaphore Behavior

The hardware semaphore access registers exhibit special read/write behavior:

```
Read:  Returns Semaphores[i].Value

Write: atomic {
         if (new_val & 1) { // SEMGET
             if (Value > 0) Value -= 1;
         } else {           // SEMPOST
             if (Value < 15) Value += 1;
         }
       }
```

## Tensix Thread Semaphores

In addition to RISC-V semaphores accessed through PC_BUF, the Tensix hardware provides native semaphore instructions that operate on a separate set of hardware semaphores.

### Tensix Semaphore Index Conversion

Converts a semaphore index (0-7) into a bitmask for Tensix semaphore instructions. This allows the hardware to operate on multiple semaphores simultaneously.

## Memory Ordering Primitives

### Load Blocking

Issues a blocking load transaction that prevents the core from proceeding until the data is available. This is essential for ensuring correct memory ordering in the weakly-ordered RISC-V memory model.

**Wormhole B0 Implementation:**

```
1. Issue LOAD transaction to address
2. Issue instruction that depends on loaded data (AND)
3. Memory clobber to prevent compiler reordering
```

**Blackhole Implementation:**

```
1. Issue LOAD transaction to address  
2. Issue instruction that depends on loaded data (AND with x0)
3. Memory clobber to prevent compiler reordering
```

**Sources:**

*   [tt_llk_wormhole_b0/common/inc/ckernel.h 92-122](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L92-L122)
*   [tt_llk_blackhole/common/inc/ckernel.h 101-129](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L101-L129)

### Store Blocking

Issues a blocking store transaction that prevents the core from proceeding until the store is complete.

**Wormhole B0 Implementation:**

```
1. Issue STORE transaction to address
2. Issue LOAD transaction to same address (must complete after STORE)
3. Issue instruction that depends on loaded data
4. Memory clobber to prevent compiler reordering
```

**Blackhole Implementation:**

```
1. Issue STORE transaction to address (also flushes L0/DCACHE line)
2. Issue LOAD transaction to same address (reads from L1 memory)
3. Issue instruction that depends on loaded data
4. Memory clobber to prevent compiler reordering
```

The Blackhole version includes an additional guarantee: the STORE to L1 flushes the L0 (DCACHE) line, ensuring the subsequent LOAD reads from L1 memory.

## Blackhole-Specific: TRISC Synchronization Tracking

Blackhole introduces automatic hardware tracking of dependencies between TRISC memory accesses and Tensix instructions, reducing the need for explicit synchronization.

### TRISC Sync Control Flags

| Flag | Purpose |
| --- | --- |
| `TRACK_GLOBAL_CFG` | Automatically synchronize CFG register accesses between TRISC and Tensix |
| `EN_SUBDIVIDED_CFG_FOR_UNPACR` | Enable fine-grained tracking for unpacker 0, unpacker 1, and other CFG registers separately |
| `TRACK_GPR` | Track general-purpose register dependencies |
| `TRACK_TDMA` | Track TDMA (tile DMA) dependencies |
| `TRACK_TENSIX_INSTRUCTIONS` | Track general Tensix instruction dependencies |

### Enabling Synchronization Tracking

When enabled, the hardware automatically stalls TRISC memory accesses and/or Tensix instructions to guarantee correct ordering, eliminating most cases where explicit `tensix_sync()` or `sync_regfile_write()` calls were previously required.

## Cache Invalidation

### Wormhole B0

Wormhole B0 RISC-V cores do not have a data cache, so cache invalidation is a no-op. The function exists for forward compatibility.

**Implementation:**[tt_llk_wormhole_b0/common/inc/ckernel.h 599-602](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L599-L602)

### Blackhole

Blackhole RISC-V cores have an L0 data cache (DCACHE). The `FENCE` instruction invalidates the cache as a side effect, with a memory clobber to prevent compiler reordering.
