# Operations and Graph Construction

*   [ttsim/config/simconfig.py](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/config/simconfig.py)

This document covers the core operational building blocks and graph construction mechanisms within the TTSIM framework. It focuses on the `SimOp` class hierarchy for representing computational operations and the `WorkloadGraph` system for building and managing computational graphs.

For tensor-specific operations and functional interfaces, see [Tensor Operations](https://deepwiki.com/tenstorrent/polaris/2.2.2-tensor-operations). For hardware modeling and device specifications, see [Hardware Modeling](https://deepwiki.com/tenstorrent/polaris/2.2.3-hardware-modeling).

## SimOp Operation System

The foundation of the TTSIM framework is the `SimOp` base class, which provides a unified interface for all computational operations. Each operation encapsulates both the functional behavior and performance characteristics needed for simulation.

### Base SimOp Architecture

The `SimOp` class defines the essential interface that all operations must implement:

*   **Operation Identity**: Each operation has a unique `name`, `optype`, and `attrs` dictionary for configuration
*   **Graph Connectivity**: `inList` and `outList` specify input and output tensor names
*   **Performance Modeling**: `get_perf_counts()` computes instruction counts and memory usage
*   **Hardware Execution**: `execute()` calculates cycles on target device
*   **Gradient Support**: `backward()` generates gradient operations for backpropagation

Sources: [ttsim/ops/op.py 205-261](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/ops/op.py#L205-L261)

### Performance Counting System

Each `SimOp` implementation must provide performance statistics through the `get_perf_counts()` method. This returns a standardized dictionary containing:

`perf_stats = {    'inElems': <input_element_count>,    'outElems': <output_element_count>,     'inBytes': <input_byte_count>,    'outBytes': <output_byte_count>,    'instrs': <instruction_count_dict>}`
The instruction count dictionary maps instruction types to their execution counts, enabling accurate performance modeling across different hardware targets.

Sources: [ttsim/ops/op.py 259-276](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/ops/op.py#L259-L276)

### Concrete Operation Implementations

The framework provides implementations for all major neural network operations:

| Operation Class | Description | Key Features |
| --- | --- | --- |
| `ConstantOp` | Constant value operations | Generates tensor data from attributes |
| `EltwiseUnaryOp` | Element-wise unary operations | Identity, activation functions |
| `EltwiseBinaryOp` | Element-wise binary operations | Add, multiply with broadcasting |
| `MatMulOp` | Matrix multiplication | Batched matmul with gradient support |
| `ConvOp` | Convolution operations | 2D convolution with flexible padding |
| `LayerNormalizationOp` | Layer normalization | Full forward/backward implementation |
| `GatherOp` | Embedding lookups | Efficient indexing operations |
| `PoolingOp` | Pooling operations | Max and average pooling variants |

Sources: [ttsim/ops/op.py 310-1600](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/ops/op.py#L310-L1600)

## WorkloadGraph Construction

The `WorkloadGraph` class manages the computational graph using NetworkX as the underlying graph representation. It provides a high-level interface for constructing, analyzing, and executing computational graphs.

### Graph Structure

The graph construction process follows these steps:

1.   **Tensor Registration**: `add_tensor()` registers `SimTensor` objects in the graph
2.   **Operation Registration**: `add_op()` validates and registers `SimOp` objects
3.   **Graph Construction**: `construct_graph()` builds the NetworkX graph from tensor connectivity
4.   **Topological Analysis**: Identifies input/output nodes and tensors for execution planning

Sources: [ttsim/graph/wl_graph.py 24-81](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L24-L81)

### Graph Building Process

The graph construction process ensures proper connectivity and validation:

Key validation checks during construction:

*   **Tensor Uniqueness**: No duplicate tensor names allowed
*   **Operation Uniqueness**: No duplicate operation names allowed
*   **Connectivity Validation**: All input/output tensors must exist before adding operations
*   **Graph Consistency**: Topological analysis ensures valid execution order

Sources: [ttsim/graph/wl_graph.py 52-81](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L52-L81)

## Graph Optimization Features

The `WorkloadGraph` provides several optimization capabilities for improving performance:

### Precision Setting

The `set_precision()` method applies data type configurations across operations:

`def set_precision(self, o2dt):    for op_type_x, data_type_x in o2dt:        for opname in nx.topological_sort(self._graph):            op_type = self._ops[opname].optype.upper()            if op_type_x == '*' or op_type == op_type_x:                self._ops[opname].set_precision(data_type_x)`
This enables mixed-precision execution where different operation types can use different data types (fp32, fp16, int8, etc.).

Sources: [ttsim/graph/wl_graph.py 134-149](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L134-L149)

### Node Removal Optimization

Operations can be marked for removal during optimization:

`def remove_nodes(self, op_type_list):    for op_type_x in op_type_list:        for opname in nx.topological_sort(self._graph):            op_type = self._ops[opname].optype.upper()            if op_type == op_type_x:                self._ops[opname].remove_in_optimization()`
This supports optimization passes that eliminate redundant or no-op operations.

Sources: [ttsim/graph/wl_graph.py 151-157](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L151-L157)

### Operation Fusion

The framework supports automatic fusion of operation sequences for improved performance:

The fusion process:

1.   **Pattern Detection**: Identifies fusable operation sequences based on provided patterns
2.   **Dependency Analysis**: Ensures fusion candidates have simple linear dependencies
3.   **Performance Modeling**: Calculates fused operation performance with overlap costs
4.   **Graph Transformation**: Marks intermediate operations as fused while preserving the primary operation

Sources: [ttsim/graph/wl_graph.py 159-242](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L159-L242)

## Resource Mapping and Execution

### Hardware Resource Mapping

The `map_resources()` method assigns operations to hardware compute pipes:

`def map_resources(self, op2rsrc_map):    for node in nx.topological_sort(self._graph):        optype = self._ops[node].optype.upper()        self._ops[node].uses_compute_pipe = op2rsrc_map[optype]`
This mapping connects abstract operations to specific hardware execution units, enabling accurate performance modeling.

Sources: [ttsim/graph/wl_graph.py 244-251](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L244-L251)

### Graph Execution

The execution process follows topological order to ensure proper data dependencies:

`def execute(self, device):    for opname in nx.topological_sort(self._graph):        op = self._ops[opname]        op.execute(device)`
Each operation's `execute()` method computes hardware-specific performance metrics including compute cycles, memory read/write cycles, and instruction counts.

Sources: [ttsim/graph/wl_graph.py 123-127](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L123-L127)

## ONNX Export Capability

The framework provides ONNX export functionality for interoperability:

The `graph2onnx()` method handles:

*   **Type Mapping**: Converts NumPy dtypes to ONNX tensor types
*   **Tensor Classification**: Separates constants, parameters, and activations
*   **Node Generation**: Creates ONNX nodes from `SimOp` instances
*   **Graph Assembly**: Builds complete ONNX model with proper input/output specification

Sources: [ttsim/graph/wl_graph.py 253-323](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/graph/wl_graph.py#L253-L323)
