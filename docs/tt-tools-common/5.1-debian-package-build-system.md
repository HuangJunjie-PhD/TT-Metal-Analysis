## Purpose and Scope

This document explains the Debian packaging infrastructure for tt-tools-common, covering the multi-distribution build matrix, package configuration files, GPG signing, and artifact generation. The system produces `.deb` packages for multiple Ubuntu versions (22.04, 24.04, and latest) with automated changelog generation and version management.

For Python wheel creation and PyPI publishing, see [Python Package Build](https://deepwiki.com/tenstorrent/tt-tools-common/5.2-python-package-build). For the overall release workflow that orchestrates both Debian and Python builds, see [Automated Release Workflow](https://deepwiki.com/tenstorrent/tt-tools-common/5.3-automated-release-workflow).

* * *

## Build Matrix Architecture

The Debian build system employs a parallel matrix strategy to generate packages for multiple Ubuntu distributions simultaneously. This ensures compatibility across different LTS releases and accommodates varying toolchain requirements.

### Matrix Configuration

**Sources:**[.github/workflows/build-debian.yml 28-34](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L28-L34)

The `fail-fast: false` configuration ensures that build failures in one Ubuntu version do not halt the other builds, allowing partial success and easier debugging of distribution-specific issues.

## Debian Package Configuration Files

### Package Metadata and Dependencies

The `debian/control` file defines the source and binary package metadata:

**Sources:**[debian/control 1-36](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/debian/control#L1-L36)

| Field | Value | Purpose |
| --- | --- | --- |
| **Source** | `tt-tools-common` | Package source name |
| **Section** | `utils` | Debian archive section classification |
| **Priority** | `optional` | Installation priority level |
| **Architecture** | `all` | Architecture-independent Python package |
| **Build-depends** | `pybuild-plugin-pyproject` | Modern pyproject.toml build support |
| **Runtime depends** | `python3-pyluwen` | Critical hardware abstraction dependency |

The package depends on `python3-pyluwen` as the sole explicit runtime dependency beyond Python standard libraries, as pyluwen provides the hardware communication layer required by the reset system.

### Version History and Changelog

The `debian/changelog` follows Debian changelog format with structured release entries:

```
tt-tools-common (1.6.0) noble; urgency=medium

  * pyproject.toml- updating version to 1.6.0

 -- Tenstorrent Releases <releases@tenstorrent.com>  Tue, 23 Dec 2025 20:59:35 +0000
```

**Sources:**[debian/changelog 1-5](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/debian/changelog#L1-L5)

Each entry contains:

*   **Version number** in parentheses (e.g., `1.6.0`)
*   **Distribution target** (e.g., `noble` for Ubuntu 24.04)
*   **Urgency level** (typically `medium`)
*   **Change description** with contributor attribution
*   **Maintainer signature** with RFC 2822 timestamp

## GPG Signing Infrastructure

All Debian packages and changes files are cryptographically signed using GPG to ensure authenticity and integrity.

### Signing Process Flow

**Sources:**[.github/workflows/build-debian.yml 67-73](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L67-L73)[.github/workflows/build-debian.yml 132-133](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L132-L133)

The GPG key import step outputs a fingerprint that is passed to `gbp buildpackage` via the `DEBSIGN_KEYID` environment variable. The build tool automatically signs both the `.deb` binary package and the `.changes` metadata file during package creation.

## Ubuntu 22.04 Compatibility Layer

Ubuntu 22.04 ships with older packaging tools that lack native pyproject.toml support, requiring a compatibility shim.

### Compatibility Setup Script

**Sources:**[setup.py 1-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/setup.py#L1-L21)

The `setup.py` script serves exclusively as a fallback for Ubuntu 22.04's `dpkg-buildpackage` tool, which invokes `setup.py` when `pybuild-plugin-pyproject` is unavailable. The script:

1.   Loads `pyproject.toml` using the `tomli` library
2.   Extracts the version from `toml_data['project']['version']`
3.   Passes minimal metadata to `setuptools.setup()`

On Ubuntu 24.04 and later, the `pybuild-plugin-pyproject` package enables direct pyproject.toml consumption, making `setup.py` unused.

## Artifact Naming and Collection

Build artifacts follow a structured naming convention to distinguish packages built for different Ubuntu distributions.

### Artifact Naming Transformation

**Sources:**[.github/workflows/build-debian.yml 143-178](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L143-L178)

### Artifact Upload Table

| Artifact Type | Naming Pattern | Example | Purpose |
| --- | --- | --- | --- |
| **Debian Package** | `$DEB_FILENAME_DISTRO` | `tt-tools-common_1.6.0_all-ubuntu-22.04.deb` | Installable binary package |
| **Changes File** | `$DEB_FILENAME_CHANGES_DISTRO` | `tt-tools-common_1.6.0_amd64-ubuntu-22.04.changes` | Package metadata and checksums |
| **Changelog** | `debian-changelog-$DISTRO_IDENT` | `debian-changelog-ubuntu-22.04` | Version history documentation |
| **Full Archive** | `artifacts-$DISTRO_IDENT.zip` | `artifacts-ubuntu-22.04.zip` | Complete artifact bundle |

**Sources:**[.github/workflows/build-debian.yml 143-178](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L143-L178)

The distro identifier is extracted from `${{ matrix.os }}` and appended to artifact names, ensuring that packages from parallel matrix builds do not collide when uploaded to GitHub Actions artifacts or GitHub Releases. The `.changes` file contains cryptographic checksums and build metadata required for Debian repository integration.

## Build Package Dependencies

The `gbp buildpackage` process requires specific package dependencies to successfully build and sign Debian packages.

### Dependency Categories

| Category | Packages | Purpose |
| --- | --- | --- |
| **Core Build** | `build-essential`, `debhelper` | Fundamental Debian packaging tools |
| **Python Integration** | `dh-python`, `dh-sequence-python3`, `pybuild-plugin-pyproject` | Python-specific packaging helpers |
| **Python Runtime** | `python3-all`, `python3-pip`, `python3-setuptools`, `python3-wheel` | Python interpreter and packaging tools |
| **Security** | `gnupg` | GPG signing for package authentication |
| **Version Control** | `git-buildpackage` | Git integration for Debian packaging |
| **Development Headers** | `libpython3-all-dev` | Python C extension compilation support |
| **Configuration Parsing** | `python3-tomli` | TOML parser for Ubuntu 22.04 setup.py |

**Sources:**[.github/workflows/build-debian.yml 54-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L54-L66)[debian/control 5-13](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/debian/control#L5-L13)

The build dependencies in `debian/control` mirror the runtime installation requirements in the workflow, ensuring that the Debian package can be rebuilt on any system meeting the declared build-depends specifications.
