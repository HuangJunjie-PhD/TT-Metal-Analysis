The device model interface defines the abstract contract that all hardware implementations must satisfy. The `npeDeviceModel` class provides a hardware abstraction layer that enables the simulation engine to estimate NoC performance across different Tenstorrent architectures without architecture-specific code paths.

This page documents the abstract interface itself. For details on specific hardware implementations, see:

*   [Wormhole B0 Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface#2.2.2)
*   [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface#2.2.3)
*   [Blackhole Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface#2.2.4)

For information on how the simulation engine uses device models during execution, see [Simulation Engine](https://deepwiki.com/tenstorrent/tt-npe/2.1-trace-collection-with-tt-metal-profiler).

## Routing Interface

The `route()` method is the primary routing interface. It computes the link-by-link path through the NoC from a source to one or more destinations.

### Method Signature

### Parameters and Return Type

| Parameter | Type | Description |
| --- | --- | --- |
| `noc_type` | `nocType` | NOC0 or NOC1 (determines routing dimension order) |
| `startpoint` | `const Coord&` | Source coordinate (device_id, row, col) |
| `destination` | `const NocDestination&` | Either `Coord` for unicast or `MulticastCoordSet` for multicast |
| **Returns** | `nocRoute` | Vector of `nocLinkID` representing hop-by-hop route |

The `NocDestination` type uses `std::variant`:

### Routing Semantics

**Example from Wormhole B0:**

*   NOC0: Dimension-ordered routing (X-then-Y, East-then-South)
*   NOC1: Dimension-ordered routing (Y-then-X, North-then-West)

For multicast, the implementation computes routes to all destinations and returns the set of unique links that form the multicast tree.

## Performance Characteristics

Device models provide bandwidth and rate information for different resource types.

### Bandwidth Query Methods

| Method | Return Type | Description |
| --- | --- | --- |
| `getSrcInjectionRate(Coord)` | `BytesPerCycle` | Maximum injection rate at source NIU for given coordinate |
| `getSinkAbsorptionRate(Coord)` | `BytesPerCycle` | Maximum absorption rate at sink NIU for given coordinate |
| `getLinkBandwidth(nocLinkID)` | `float` | Bandwidth (GB/s) of NoC link |
| `getAggregateDRAMBandwidth()` | `float` | Total DRAM bandwidth across all banks |
| `getEthBandwidth()` | `float` | Bandwidth of single-direction Ethernet link |

These rates vary by core type. For example, in Wormhole B0:

| Core Type | Injection Rate | Absorption Rate |
| --- | --- | --- |
| WORKER | 28.1 GB/s | 28.1 GB/s |
| DRAM | 23.2 GB/s | 24.0 GB/s |
| ETH | 28.1 GB/s | 24.0 GB/s |

## Device State Initialization

The `initDeviceState()` method allocates resource tracking grids used during congestion modeling.

### Method Signature

The returned `npeDeviceState` contains:

*   `LinkDemandGrid`: Tracks bandwidth demand on each NoC link
*   `NIUDemandGrid`: Tracks bandwidth demand on each NIU (source and sink)

Grid sizes are determined by the number of resources:

For multi-chip models, the grid sizes scale with the number of chips to track resources across all devices.

The `computeCurrentTransferRate()` method updates transfer bandwidth based on packet size effects and congestion.

### Method Signature

### Two-Phase Computation

### TransferBandwidthTable

Implementations maintain a `TransferBandwidthTable` that maps packet size to effective bandwidth. Small packets achieve lower bandwidth due to overhead:

**Wormhole B0 Example:**

| Packet Size (bytes) | Bandwidth (GB/s) |
| --- | --- |
| 0 | 0 |
| 128 | 5.5 |
| 256 | 10.1 |
| 512 | 18.0 |
| 1024 | 27.4 |
| 2048 | 30.0 |
| 8192 | 30.0 |

The `interpolateBW()` utility function performs linear interpolation between table entries.

## Congestion Modeling

Congestion modeling is implemented within the device model to allow architecture-specific congestion behavior.

### Congestion Algorithm

The congestion model identifies bottleneck resources (links or NIUs) and proportionally reduces bandwidth when demand exceeds capacity. This first-order congestion model typically converges in a single iteration.

**Sources:**[tt_npe/cpp/include/device_models/wormhole_b0.hpp 55-176](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L55-L176)[tt_npe/cpp/include/device_models/blackhole.hpp 67-188](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L67-L188)

## Integration with Simulation Engine

The simulation engine interacts with device models at specific points in the simulation pipeline.

### Engine-Model Interaction Flow

### Key Integration Points

| Phase | Device Model Methods Called | Purpose |
| --- | --- | --- |
| Initialization | `initDeviceState()` | Allocate resource tracking grids |
| Validation | `isValidDeviceID()`, `getCoreType()`, `getSrcInjectionRate()` | Validate workload against device topology |
| Routing | `route()` | Compute link-by-link routes for all transfers |
| Simulation | `computeCurrentTransferRate()` | Update bandwidth each timestep |
| Statistics | `getLinkAttributes()`, `getNIUAttributes()`, `getLinkBandwidth()` | Translate resource IDs to attributes for reporting |
