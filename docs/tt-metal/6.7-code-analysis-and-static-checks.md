This document describes the code analysis infrastructure and static checking mechanisms integrated into the `tt-metal` repository. These systems enforce code quality, detect potential bugs, and ensure adherence to C++ modernization standards across the TT-Metalium, TTNN, and LLK layers.

## Purpose and Scope

The code analysis system provides:

*   **Static Analysis**: Compile-time checks for code quality and bug detection using `clang-tidy`[.clang-tidy 1-38](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L1-L38)
*   **CI/CD Integration**: Automated linting, code analysis workflows, and specialized checks within GitHub Actions [.github/workflows/code-analysis.yaml 1-78](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/code-analysis.yaml#L1-L78)
*   **Dockerized Environments**: Consistent analysis environments using pre-built Docker images containing the LLVM 20 toolchain [dockerfile/Dockerfile 148-160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L148-L160)
*   **Incremental Scanning**: Parallelized workflows that analyze changed components (Metalium, TTNN, tt-train) to optimize CI turnaround time [.github/workflows/code-analysis.yaml 172-215](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/code-analysis.yaml#L172-L215)

## Static Analysis Tools

### Clang-Tidy Integration

The project uses `clang-tidy` (targeting version 20) to enforce rules defined in the root `.clang-tidy` file [.clang-tidy 20](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L20-L20) The analysis is configured to include nearly all check categories while explicitly excluding irrelevant platforms (e.g., `abseil-*`, `android-*`, `fuchsia-*`) and specifically noisy checks [.clang-tidy 125-176](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L125-L176)

**Clang-Tidy Configuration Flow**

**Key Check Categories:**

*   **Modernization**: Enforces `#pragma once` via `modernize-use-pragma-once`[.clang-tidy 18-20](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L18-L20)
*   **Bug Prevention**: Targeted checks for `bugprone-*` and `cppcoreguidelines-*`[.clang-tidy 4-5](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L4-L5)
*   **Complexity Management**: Uses `readability-function-cognitive-complexity` with a threshold of `312` to flag overly complex logic [.clang-tidy 181-182](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L181-L182)

**Intentionally Disabled Checks:** Some checks are disabled due to high noise or lack of value in the low-level hardware context, such as `concurrency-mt-unsafe` (flags `std::getenv`) and `performance-avoid-endl` (due to concerns about flush dependencies) [.clang-tidy 33-38](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L33-L38)

Sources: [.clang-tidy 1-198](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.clang-tidy#L1-L198)

## CI/CD Static Analysis Workflows

The repository uses specialized GitHub Action workflows to manage code analysis at scale.

### Code Analysis Workflow (`code-analysis.yaml`)

This workflow determines whether to perform a full scan or an incremental scan based on the nature of the changes [.github/workflows/code-analysis.yaml 84-127](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/code-analysis.yaml#L84-L127)

1.   **Full Scan**: Triggered on the `main` branch, when the `.clang-tidy` configuration itself changes, or when manually forced [.github/workflows/code-analysis.yaml 106-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/code-analysis.yaml#L106-L112)
2.   **Incremental Scan**: Parallelized jobs target specific sub-projects like `tt_metal` (Metalium) and `ttnn` to reduce latency [.github/workflows/code-analysis.yaml 173-195](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/code-analysis.yaml#L173-L195)
3.   **Docker Support**: The workflow utilizes `ci-build-docker-image` which contains pre-built tools including `ccache`, `doxygen`, and `mold`[dockerfile/Dockerfile 148-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L148-L155)

### Build Artifact Pipeline Integration

The `build-artifact.yaml` workflow supports static analysis by providing specialized toolchains, such as `cmake/x86_64-linux-clang-20-libstdcpp-toolchain.cmake`[.github/workflows/build-artifact.yaml 36-40](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L36-L40) It also manages the creation of Docker images used for analysis [dockerfile/Dockerfile 136-160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L136-L160)
