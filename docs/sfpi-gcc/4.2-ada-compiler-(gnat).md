This document covers the GNAT Ada compiler implementation within the GCC infrastructure, focusing on the frontend components responsible for Ada language processing. GNAT (GNU NYU Ada Translator) provides complete Ada language support including semantic analysis, expansion, and integration with GCC's optimization and code generation phases.

For information about SFPU extensions and target-specific backends, see other sections of this wiki. This document focuses specifically on the language-independent Ada frontend components.

## Overall Architecture

GNAT follows a multi-phase compilation model typical of modern compilers, with distinct phases for parsing, semantic analysis, expansion, and code generation.

### Compilation Pipeline

Sources: [gcc/ada/sem_util.adb 1-100](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_util.adb#L1-L100)[gcc/ada/exp_util.adb 1-100](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/exp_util.adb#L1-L100)[gcc/ada/einfo.ads 1-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/einfo.ads#L1-L50)[gcc/ada/sinfo.ads 1-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sinfo.ads#L1-L50)

The compilation process maintains a sophisticated abstract syntax tree representation throughout all phases, with nodes defined in `Sinfo` and entity information tracked in `Einfo`. The `sem_util.adb` provides common utilities used across semantic analysis phases.

## Semantic Analysis Architecture

The semantic analysis phase is organized according to chapters of the Ada Reference Manual, with each major language construct handled by a dedicated module. Analysis proceeds in two phases: a top-down analysis phase that constructs candidate interpretations, followed by a bottom-up resolution phase that selects the correct interpretation.

### Semantic Analysis Organization

Sources: [gcc/ada/sem_util.adb 77-100](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_util.adb#L77-L100)[gcc/ada/sem_ch3.adb 89-127](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch3.adb#L89-L127)[gcc/ada/sem_res.adb 92-106](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L92-L106)[gcc/ada/sem_type.adb 1-30](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_type.adb#L1-L30)

Each semantic analysis module implements the rules for its corresponding Ada Reference Manual chapter. The `sem_util.adb` provides fundamental operations like accessibility level calculation, type utilities, and entity management that are used throughout the semantic analysis process.

### Type System Processing

Type processing in GNAT involves several stages, handled primarily by `sem_ch3.adb`:

#### Type Declaration Processing

Sources: [gcc/ada/sem_ch3.adb 111-134](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch3.adb#L111-L134)[gcc/ada/sem_ch3.adb 486-510](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch3.adb#L486-L510)[gcc/ada/sem_ch3.adb 553-594](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch3.adb#L553-L594)

The type system handles:

*   **Derived types**: `Build_Derived_Type` processes type derivation, copying characteristics from parent types
*   **Record types**: `Build_Derived_Record_Type` handles tagged type inheritance and component copying via `Inherit_Components`

*   **Array constraints**: `Constrain_Array` and `Constrain_Index` process array type constraints
*   **Component subtypes**: `Constrain_Component_Type` creates constrained component subtypes based on discriminant values

## Overload Resolution

GNAT implements a sophisticated overload resolution system in `sem_res.adb` and `sem_type.adb` that handles Ada's complex overloading rules.

### Resolution Architecture

Sources: [gcc/ada/sem_res.adb 102-112](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L102-L112)[gcc/ada/sem_type.adb 75-115](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_type.adb#L75-L115)[gcc/ada/sem_res.adb 222-262](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L222-L262)[gcc/ada/sem_res.adb 269-286](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L269-L286)

#### Interpretation Collection

The `sem_type.adb` module maintains interpretation lists for overloaded names:

| Function | Purpose |
| --- | --- |
| `Collect_Interps` | Gather all possible interpretations for a name |
| `Add_One_Interp` | Add an interpretation to the list with type information |
| `Covers` | Determine if one type covers another for overload resolution |
| `Compatible_Types` | Check type compatibility between operands |
| `Intersect_Types` | Find common type for mixed-mode operations |
| `Get_First_Interp` / `Get_Next_Interp` | Iterate through interpretation list |

Sources: [gcc/ada/sem_type.adb 1-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_type.adb#L1-L50)[gcc/ada/sem_type.ads 1-30](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_type.ads#L1-L30)

#### Resolution Process

The resolution phase in `sem_res.adb` selects the correct interpretation based on context:

1.   **Context matching**: `Resolve(N, Typ)` resolves node N to expected type Typ
2.   **Actuals resolution**: `Resolve_Actuals` matches actual parameters to formal parameters, handling named notation and defaults
3.   **Conformance checking**: `Check_Conformance` (in `sem_ch6.adb`) verifies parameter mode and type conformance
4.   **Implicit conversions**: Automatic conversions inserted where Ada rules permit
5.   **Ambiguity detection**: Reports errors when multiple valid interpretations remain

Sources: [gcc/ada/sem_res.adb 390-443](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L390-L443)[gcc/ada/sem_res.adb 269-296](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_res.adb#L269-L296)[gcc/ada/sem_ch6.adb 138-158](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch6.adb#L138-L158)

## Freeze Mechanism

The freeze mechanism in `freeze.adb` delays certain aspects of entity processing until sufficient information is available. Freezing is a critical phase where representation and layout become fixed.

### Freeze Point Determination

Sources: [gcc/ada/freeze.adb 80-107](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/freeze.adb#L80-L107)[gcc/ada/freeze.adb 1-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/freeze.adb#L1-L50)

### Freeze Operations by Entity Kind

| Entity Kind | Freeze Actions | Key Functions |
| --- | --- | --- |
| Array types | Initialize `Component_Size`, build initialization procedures | `Freeze_Array_Type`, `Build_Array_Init_Proc` (exp_ch3.adb) |
| Record types | Build discriminant checking functions, dispatch tables, equality ops | `Freeze_Record_Type`, `Build_Discr_Checking_Funcs` |
| Access types | Set designated type size, check accessibility | `Freeze_Entity` (access type handling) |
| Subprograms | Build convention wrappers, check bodies present | `Freeze_Subprogram` |
| Enumeration types | Build representation arrays for non-standard reps | `Freeze_Enumeration_Type` |

Sources: [gcc/ada/freeze.adb 1-100](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/freeze.adb#L1-L100)[gcc/ada/exp_ch3.adb 82-96](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/exp_ch3.adb#L82-L96)[gcc/ada/exp_ch3.adb 167-189](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/exp_ch3.adb#L167-L189)

### Freeze Timing Rules

Freezing occurs at specific points defined by Ada rules:

*   When an entity is used in ways that require knowing its representation
*   At the end of a declarative part
*   Before generating code that depends on representation details
*   When completing a private type with its full view

The `Freeze_Before` procedure in `freeze.adb` inserts freeze nodes at the earliest possible freezing point, while `Freeze_All` processes all entities at the end of declarative regions.

Sources: [gcc/ada/freeze.adb 1-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/freeze.adb#L1-L50)

## Advanced Language Features

GNAT implements several sophisticated Ada language features that require specialized processing.

### Contract and Aspect Processing

Sources: [gcc/ada/aspects.ads 1-30](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/aspects.ads#L1-L30)[gcc/ada/contracts.adb 1-30](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/contracts.adb#L1-L30)[gcc/ada/exp_spark.adb 1-25](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/exp_spark.adb#L1-L25)

GNAT provides comprehensive support for Ada 2012+ contracts and aspects, including pre/postconditions, type invariants, and SPARK verification conditions.

### Generic Instantiation

The generic system in GNAT handles one of Ada's most complex features through the `sem_ch12.adb` module, which manages:

*   Generic declaration analysis
*   Instantiation processing
*   Generic body compilation
*   Template management and specialization

Sources: [gcc/ada/sem_ch12.adb 25-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/ada/sem_ch12.adb#L25-L50)

### Tasking Support

GNAT's tasking implementation spans multiple modules:
