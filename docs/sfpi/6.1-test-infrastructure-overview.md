This document describes the test infrastructure for the SFPI toolchain, including the organization of test files, the DejaGnu test framework, simulator configuration (QEMU, Spike, GDB), gold standard validation, and the failure management system. The infrastructure validates both SFPI-specific functionality and upstream GCC/Binutils compatibility across three Tenstorrent architectures (Wormhole, Blackhole, Grayskull).

For detailed information about specific test categories and content, see [SFPU Kernel Tests](https://deepwiki.com/tenstorrent/sfpi/6.2-sfpu-kernel-tests). For information about compiler optimization tests, see [Compiler Optimization Tests](https://deepwiki.com/tenstorrent/sfpi/6.3-compiler-optimization-tests). For details about the known failures system, see [Known Issues and Failures](https://deepwiki.com/tenstorrent/sfpi/6.4-known-issues-and-failures).

## Test Directory Organization

The test suite is organized hierarchically by architecture, with each Tenstorrent chip generation having its own test directory:

| Directory | Architecture | Description |
| --- | --- | --- |
| `tests/wormhole/sfpi/` | Wormhole | Tests for Wormhole-specific SFPU instructions (51 operations) |
| `tests/blackhole/sfpi/` | Blackhole | Tests for Blackhole-specific SFPU instructions (54 operations) |
| `tests/grayskull/sfpi/` | Grayskull | Tests for Grayskull-specific SFPU instructions |
| `tests/{arch}/sfpi/gold/` | All | Gold standard reference files for each architecture |
| `xfails/` | All | Expected failure definitions for upstream test suites |

Each architecture directory contains:

*   **Source files (.cc)**: C++ test implementations using SFPI library
*   **Gold assembly files (.S)**: Expected assembly output from compilation
*   **Gold data files (_int, _float)**: Expected runtime execution results
*   **Test configuration**: Architecture-specific test parameters

## DejaGnu Test Framework

The SFPI toolchain uses DejaGnu as its test orchestration framework. DejaGnu is a test framework written in Expect/Tcl that executes test suites and collects results.

**DejaGnu Test Configuration**

Sources: [Makefile.in 459-467](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L459-L467)[Makefile.in 469-472](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L469-L472)[configure.ac 73-78](https://github.com/tenstorrent/sfpi/blob/ed989b9d/configure.ac#L73-L78)

The DejaGnu build and configuration is managed through:

*   **Build target**: `stamps/build-dejagnu` builds DejaGnu from `riscv-dejagnu` submodule
*   **Installation**: DejaGnu installs to `$(test_infradir)` (typically `build/infra`)
*   **Test invocation**: Tests are invoked via `make check-gcc-tt` or similar targets
*   **RUNTESTFLAGS**: Passes configuration to DejaGnu, including target board configuration

Test execution example from Makefile:

`check-gcc-tt: stamps/build-gcc-newlib-stage2 stamps/build-dejagnu	$(MAKE) -C build-gcc-newlib-stage2 check-gcc \	  "RUNTESTFLAGS=--target_board='$(NEWLIB_TARGET_BOARDS)' tt.exp sfpi.exp"`
Sources: [Makefile.in 469-472](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L469-L472)

## Simulator Configuration

The test infrastructure supports three RISC-V simulators for test execution, configurable via the `--with-sim` option during configuration.

**Simulator Support Matrix**

| Simulator | Configuration | Description | When Used |
| --- | --- | --- | --- |
| QEMU | `--with-sim=qemu` (default) | QEMU user-mode emulation (v7.2.15) | Default for all tests |
| Spike | `--with-sim=spike` | Spike ISA simulator with pk proxy kernel | Alternative backend |
| GDB | `--with-sim=gdb` | GDB's built-in simulator | Alternative backend |

**Simulator Selection Flow**

Sources: [Makefile.in 124-158](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L124-L158)[configure.ac 73-78](https://github.com/tenstorrent/sfpi/blob/ed989b9d/configure.ac#L73-L78)

### QEMU Configuration

QEMU is built with the following configuration:

*   **Targets**: `riscv32-linux-user` (user-mode emulation only by default)
*   **Build directory**: `build-qemu/`
*   **Installation**: `infra/` directory
*   **Wrapper script**: `scripts/riscv-*-elf-run` provides execution wrapper

QEMU build configuration from Makefile:

`stamps/build-qemu: $(QEMU_SRCDIR) $(QEMU_SRC_GIT)	rm -rf $@ $(notdir $@)	mkdir $(notdir $@)	cd $(notdir $@) && $</configure \		--prefix=$(test_infradir) \		--target-list=$(QEMU_TARGETS) \		--interp-prefix=$(INSTALL_DIR)/sysroot \		--python=python3	$(MAKE) -C $(notdir $@)	$(MAKE) -C $(notdir $@) install`
Sources: [Makefile.in 446-457](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L446-L457)

### Spike Configuration

When using Spike, the infrastructure builds both the simulator and the proxy kernel (pk):

*   **Spike**: RISC-V ISA simulator
*   **PK32/PK64**: Proxy kernels for rv32 and rv64 execution
*   **Wrapper**: Custom wrapper in `scripts/wrapper/spike/` sets up execution environment

### Assembly Gold Files

Assembly gold files (`.S`) contain the expected assembly output from compiling SFPI test sources. These validate:

*   Correct instruction selection
*   Proper register allocation
*   Expected optimization passes
*   Instruction sequencing

Example from `coverage.S`:

`_Z15test_load_storev:	SFPLOAD	L2, 0, 0, 3	SFPLOAD	L0, 2, 0, 3	SFPLOAD	L1, 4, 0, 3	SFPSTORE	6, L2, 0, 3	SFPSTORE	8, L0, 0, 3	SFPSTORE	10, L1, 0, 3`
Sources: [tests/wormhole/sfpi/gold/coverage.S 10-29](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/coverage.S#L10-L29)

### Runtime Gold Files

Runtime gold files contain expected execution results in two formats:

**Integer Format (`kernels_int`)**:

*   32-bit hexadecimal values
*   One row per test iteration
*   Validates bit-exact results

**Float Format (`kernels_float`)**:

*   Decimal floating-point values
*   One row per test iteration
*   Validates numerical results

Example formats:

```
# kernels_int format
Row: 0
0000 3F800000 40000000 40400000 40800000 40A00000 40C00000 40E00000 41000000

# kernels_float format
Row: 0
0.000000 1.000000 2.000000 3.000000 4.000000 5.000000 6.000000 7.000000
```

### Makefile Test Targets

The Makefile provides multiple test targets for different test suites:

| Target | Description | Test Suite |
| --- | --- | --- |
| `check-gcc-tt` | TT-specific GCC tests | `tt.exp`, `sfpi.exp` |
| `check-gcc-newlib` | Full GCC test suite | All GCC tests with newlib |
| `check-binutils-newlib` | Binutils test suite | gas, ld, binutils tests |
| `check-gdb-newlib` | GDB test suite | GDB tests with newlib |

Example test invocation:

`.PHONY: check-gcc-ttcheck-gcc-tt: stamps/build-gcc-newlib-stage2 stamps/build-dejagnu	$(MAKE) -C build-gcc-newlib-stage2 check-gcc \	  "RUNTESTFLAGS=--target_board='$(NEWLIB_TARGET_BOARDS)' tt.exp sfpi.exp"`

### Test Board Configuration

Tests are executed with specific target board configurations that specify the architecture and ABI:

`NEWLIB_TARGET_BOARDS = riscv-sim/-march=rv32i-ilp32/-mabi=ilp32/@cmodel@ \                       riscv-sim/-march=rv64imafdc-lp64d/-mabi=lp64d/@cmodel@`
The target board configuration includes:

*   **Board name**: `riscv-sim` (simulator board)
*   **Architecture flags**: `-march=<arch>` specifies ISA extensions
*   **ABI flags**: `-mabi=<abi>` specifies calling convention
*   **Code model**: `@cmodel@` specifies memory model (medlow/medany)

Sources: [Makefile.in 69-72](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L69-L72)

## Test Result Files

Test execution generates two types of result files managed by DejaGnu:

**Summary Files (`.sum`)**:

*   Contains test case results: PASS, FAIL, UNRESOLVED, XPASS, XFAIL
*   One line per test case
*   Used for automated result processing
*   Example: `build-gcc-newlib-stage2/gcc/testsuite/gcc/gcc.sum`

**Log Files (`.log`)**:

*   Contains detailed test output
*   Includes compilation output, execution traces, and error messages
*   Used for debugging test failures
*   Example: `build-gcc-newlib-stage2/gcc/testsuite/gcc/gcc.log`

Test result aggregation:

`report-gcc-newlib: stamps/check-gcc-newlib	$(srcdir)/scripts/testsuite-filter gcc newlib $(srcdir)/test/allowlist \	  `find build-gcc-newlib-stage2/gcc/testsuite/ -name *.sum | paste -sd "," -``
Sources: [Makefile.in 547-552](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L547-L552)

## Expected Failure Management

The test infrastructure includes a comprehensive system for managing expected test failures, allowing the toolchain to track known issues without blocking CI/CD.

**XFAIL System Architecture**

Sources: [Makefile.in 547-606](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L547-L606)

The XFAIL files contain test names or patterns that are known to fail. This system ensures:

1.   **Known issues are tracked** without blocking development
2.   **Regressions are caught** when new failures appear
3.   **Progress is visible** when known failures are fixed (XPASS)
4.   **CI stability** by accepting expected failures

For detailed information about the XFAIL system and how to manage expected failures, see [Known Issues and Failures](https://deepwiki.com/tenstorrent/sfpi/6.4-known-issues-and-failures).

Sources: [Makefile.in 589-606](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L589-L606)

## Test Function Approach

The SFPI test suite uses a systematic approach to test various aspects of the library. Tests are designed to be:

1.   **Comprehensive**: Covering all SFPI instructions and operations
2.   **Isolated**: Testing specific functionality in isolation
3.   **Combined**: Testing interactions between different operations
4.   **Architecture-specific**: Targeting specific hardware implementations

## Conclusion

The SFPI test suite is designed to comprehensively verify the correctness of the SFPI interface and its implementations across different architectures. Through a combination of coverage tests, kernel tests, combination tests, and specialized tests for specific functionality, it ensures that SFPI operates as expected in all contexts.

For working with the test suite, users should:

1.   Understand the organization by architecture and test type
2.   Examine gold files to understand expected outputs
3.   Use the test harness for running tests
4.   Leverage the XFAIL system for managing expected failures (see [Expected Failures System](https://deepwiki.com/tenstorrent/sfpi/6.2-sfpu-kernel-tests))
