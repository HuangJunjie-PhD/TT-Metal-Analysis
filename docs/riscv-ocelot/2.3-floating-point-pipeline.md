The Floating-Point Pipeline in the RISC-V Ocelot processor handles the execution of all floating-point instructions. This document describes the architecture and operation of the FP pipeline, including its issue logic, register file, execution units, and overall dataflow. For information about vector processing operations, see [Vector Processing](https://deepwiki.com/tenstorrent/riscv-ocelot/5-vector-processing).

## Overview

The floating-point pipeline operates in parallel with the integer pipeline, providing computational capabilities for floating-point operations. It consists of several key components, including a dedicated issue queue, floating-point register file (FPRF), and specialized execution units. The FP pipeline supports both single-precision (32-bit) and double-precision (64-bit) operations as defined by the RISC-V "F" and "D" extensions.

Sources: [src/main/scala/exu/fp-pipeline.scala:29-62], [src/main/scala/exu/core.scala:76-77], [src/main/scala/common/parameters.scala:73-74]

## FP Pipeline Components

### FP Issue Queue

The FP issue queue schedules floating-point instructions for execution when their operands are ready. It operates similarly to the integer issue queue, but is specifically configured for floating-point operations through specialized issue parameters.

Key features:

*   Collapsing queue structure for compact storage
*   Wakeup logic for tracking operand readiness
*   Issue width configured by `fpIssueParams.issueWidth`
*   Support for age-ordered or unordered issue policies (configurable)

Sources: [src/main/scala/exu/fp-pipeline.scala:67-70], [src/main/scala/exu/issue-units/issue-unit.scala:103-116], [src/main/scala/exu/issue-units/issue-unit-age-ordered.scala:29-35]

### FP Register File

The floating-point register file stores floating-point values and provides read/write ports for the execution units.

Key characteristics:

*   Size: `numFpPhysRegisters` (default 64) physical registers
*   Width: `fLen+1` bits (typically 33 bits for single-precision or 65 bits for double-precision, including an extra bit for internal representation)
*   Multiple read ports for parallel operand fetch
*   Write ports to receive results from execution units and memory loads

Sources: [src/main/scala/exu/fp-pipeline.scala:71-80], [src/main/scala/exu/register-read/regfile.scala:74-95], [src/main/scala/common/parameters.scala:190]

### FP Register Read Stage

The register read stage fetches source operands from the FP register file and forwards them to the execution units.

Implementation details:

*   Multi-ported to read multiple operands simultaneously
*   Single-cycle access (register address in, data out)
*   Instantiated by `fregister_read` module in the FP pipeline
*   Processes up to 3 operands per instruction (for fused multiply-add operations)

Sources: [src/main/scala/exu/fp-pipeline.scala:81-88], [src/main/scala/exu/register-read/register-read.scala:34-64]

### FP Execution Units

The floating-point pipeline includes several specialized execution units for different types of operations.

Each execution unit type has different characteristics:

1.   **FP ADD/SUB**: Handles floating-point addition and subtraction

    *   Microops: `uopFADD_S`, `uopFSUB_S`, `uopFADD_D`, `uopFSUB_D`
    *   Latency: Usually 3-4 cycles

2.   **FP MUL**: Handles floating-point multiplication

    *   Microops: `uopFMUL_S`, `uopFMUL_D`
    *   Latency: Usually 3-4 cycles

3.   **FP FMA**: Handles fused multiply-add operations (a×b+c in a single operation)

    *   Microops: `uopFMADD_S`, `uopFMSUB_S`, `uopFNMADD_S`, `uopFNMSUB_S`, `uopFMADD_D`, etc.
    *   Latency: Configured by `sfmaLatency`/`dfmaLatency` (usually 3-4 cycles)

4.   **FP DIV/SQRT**: Handles division and square root operations

    *   Microops: `uopFDIV_S`, `uopFDIV_D`, `uopFSQRT_S`, `uopFSQRT_D`
    *   Latency: Variable (implemented in FDivSqrt module when `usingFDivSqrt` is true)

5.   **FP Conversion Units**: Handle conversions between integer and floating-point formats

    *   INT→FP (I2F): `uopFCVT_S_X`, `uopFCVT_D_X`
    *   FP→INT (F2I): `uopFCVT_X_S`, `uopFCVT_X_D`
    *   FP→FP: `uopFCVT_S_D`, `uopFCVT_D_S`
    *   Latency: Usually 2-3 cycles

6.   **FP Misc Operations**: Handles comparisons, sign injection, and classifications

    *   Microops: `uopCMPR_S`, `uopCMPR_D`, `uopFSGNJ_S`, `uopFSGNJ_D`, `uopFCLASS_S`, `uopFCLASS_D`, etc.
    *   Latency: Usually 1-2 cycles

Sources: [src/main/scala/exu/execution-units/functional-unit.scala:34-56], [src/main/scala/common/consts.scala:209-255], [src/main/scala/common/parameters.scala:204-208]

## FP Pipeline Operation

The floating-point pipeline processes instructions through several stages, from dispatch to writeback.

### Dispatch Phase

FP instructions are dispatched from the decode stage to the FP issue queue. The `fpIssueParams` determines how many instructions can be dispatched per cycle (usually 1).

`for (w <- 0 until dispatchWidth) {  issue_unit.io.dis_uops(w) <> io.dis_uops(w)}`
Sources: [src/main/scala/exu/fp-pipeline.scala:116-118], [src/main/scala/exu/issue-units/issue-unit.scala:74]

### Issue Phase

Instructions in the issue queue wait until all their operands are ready, then compete for the available execution resources.

Key operations:

*   Track source register readiness via a ready table
*   Wake up instructions when their source operands become available
*   Select ready instructions for execution based on policy (age-ordered or unordered)
*   Route instructions to appropriate execution units

Sources: [src/main/scala/exu/fp-pipeline.scala:124-130], [src/main/scala/exu/issue-units/issue-slot.scala:96-142]

### Register Read Phase

When an instruction is issued, its source operands are read from the floating-point register file:

`fregister_read.io.rf_read_ports <> fregfile.io.read_ports`
The register read stage fetches up to 3 operands for each issued instruction (for instructions like FMA that need 3 inputs).

Sources: [src/main/scala/exu/fp-pipeline.scala:137-149], [src/main/scala/exu/register-read/register-read.scala:66-80]

### Execute Phase

The execution units perform the actual floating-point operations. Each type of operation has a specific latency:

| Operation Type | Typical Latency | Configuration Parameter |
| --- | --- | --- |
| ADD/SUB | 3-4 cycles | Fixed in implementation |
| MUL | 3-4 cycles | Fixed in implementation |
| FMA | 3-4 cycles | sfmaLatency/dfmaLatency |
| DIV/SQRT | Variable | Based on implementation |
| Conversions | 2-3 cycles | Fixed in implementation |
| Comparisons | 1-2 cycles | Fixed in implementation |

Sources: [src/main/scala/exu/fp-pipeline.scala:151-172], [src/main/scala/common/parameters.scala:204-208]

### Writeback Phase

Results from the execution units are written back to the floating-point register file and broadcast to the issue queue to wake up dependent instructions.

`for (eu <- exe_units) {  if (eu.writesFrf) {    fregfile.io.write_ports(wakeup_port_idx) := WritePort(eu.io.fresp, fpPregSz, fLen+1, RT_FLT)    io.wakeups(wakeup_port_idx) := ear_wb_uop    io.debug_wb_wdata(wakeup_port_idx) := ear_wb_uop.bits.data    wakeup_port_idx += 1  }}`
Sources: [src/main/scala/exu/fp-pipeline.scala:261-276], [src/main/scala/exu/register-read/regfile.scala:51-63]

## Floating-Point Control and Status

The floating-point pipeline interacts with the FCSR (Floating-Point Control and Status Register) for:

1.   **Rounding Modes**: Determines how floating-point results are rounded

    *   The `fcsr_rm` signal from the CSR file is passed to the FP units

2.   **Exception Flags**: Tracks floating-point exceptions

    *   Exception flags are captured by execution units and passed to the ROB
    *   The ROB accumulates flags for committed instructions
    *   When instructions commit, flags are written to the FCSR

Sources: [src/main/scala/exu/fp-pipeline.scala:41], [src/main/scala/exu/rob.scala:618-643]

## Integration with Other Pipelines

The FP pipeline integrates with other parts of the processor:

### Memory Interface

Load instructions can return floating-point values directly to the FP register file:

`for (w <- 0 until numLlPorts) {  fregfile.io.write_ports(ll_wport_idx) <> WritePort(io.ll_wports(w), fpPregSz, fLen+1, RT_FLT)  io.wakeups(wakeup_port_idx).valid := io.ll_wports(w).valid && io.ll_wports(w).bits.uop.dst_rtype === RT_FLT  io.wakeups(wakeup_port_idx).bits := io.ll_wports(w).bits  wakeup_port_idx += 1  ll_wport_idx += 1}`
Store instructions need to send FP values to the Store Data Queue:

`fp_sdq_unit.io.req <> fpiu_unit.io.reqio.to_sdq <> fp_sdq_unit.io.fresp`
Sources: [src/main/scala/exu/fp-pipeline.scala:285-293], [src/main/scala/exu/fp-pipeline.scala:343-346]

### Integer Interface

The FP pipeline can receive values from the integer pipeline for INT→FP conversions:

`if (intToFpLatency == 0) {  fp_pipeline.io.from_int <> exe_units.ifpu_unit.io.fresp} else {  fp_pipeline.io.from_int <> Queue(exe_units.ifpu_unit.io.fresp, intToFpLatency)}`
Conversely, FP→INT conversions send results to the integer pipeline:

`io.to_int <> fpiu_unit.io.iresp`
Sources: [src/main/scala/exu/core.scala:239-243], [src/main/scala/exu/fp-pipeline.scala:317-318], [src/main/scala/common/parameters.scala:210]

## Configuration Parameters

The floating-point pipeline can be configured through several parameters in the BoomCoreParams:

`case class BoomCoreParams(  ...  usingFPU: Boolean = true,  fpu: Option[FPUParams] = Some(FPUParams(sfmaLatency=4, dfmaLatency=4)),  numFpPhysRegisters: Int = 64,  ...)`
| Parameter | Description | Default Value |
| --- | --- | --- |
| usingFPU | Whether to include FP pipeline | true |
| fpu | FPU parameters | Some(FPUParams(sfmaLatency=4, dfmaLatency=4)) |
| numFpPhysRegisters | Number of physical FP registers | 64 |
| intToFpLatency | Latency between INT and FP pipelines | 2 |
| issueParams | Includes FP issue queue configuration | IssueParams(iqType=IQT_FP) |

Additionally, each FP execution unit can be configured with specific parameters:

1.   Latencies for operations like FMA are configurable
2.   Pipeline depth can be adjusted for timing/area tradeoffs
3.   Number of read/write ports can be configured based on performance needs

Sources: [src/main/scala/common/parameters.scala:25-130], [src/main/scala/common/parameters.scala:190-208]
