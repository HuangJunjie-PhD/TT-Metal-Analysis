The Control Plane and Fabric System provides the infrastructure for multi-device communication in TT-Metalium. The **Control Plane** manages topology discovery, logical-to-physical device mapping, and routing table generation. The **Fabric** provides the data plane for multi-hop packet communication between devices via Ethernet cores configured as routers. Together, they enable scaling workloads across multiple chips in T3000 (8-chip), Galaxy (32-chip), and multi-host configurations.

For information about the overall device abstraction and management, see [Device Abstraction Layer](https://deepwiki.com/tenstorrent/tt-metal/2.3-device-abstraction-layer). For details on how the fabric integrates with dispatch and command queues, see [Fast Dispatch and Command Queue System](https://deepwiki.com/tenstorrent/tt-metal/2.5-fast-dispatch-and-command-queue-system). For multi-device execution orchestration, see [Multi-Device Execution and MeshWorkload](https://deepwiki.com/tenstorrent/tt-metal/2.10-multi-device-execution-and-meshworkload).

## System Overview

The Control Plane and Fabric System consists of three main layers:

1.   **Control Plane Layer**: Manages topology configuration, device discovery, and routing table generation. The `ControlPlane` class [tt_metal/fabric/control_plane.cpp 309-419](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/control_plane.cpp#L309-L419) orchestrates mesh graph creation, topology mapping from logical to physical devices, and routing table computation.
2.   **Fabric Configuration Layer**: Provides APIs to configure fabric modes (1D/2D topologies, reliability modes) and manages the lifecycle of fabric routers on Ethernet cores.
3.   **Fabric Data Plane**: Executes packet routing via EDM (Ethernet Data Mover) kernels running on ERISC (Ethernet RISC) cores, enabling multi-hop unicast and multicast communication.

## Routing and Data Plane

### Routing Table Generation

The `RoutingTableGenerator` computes routing tables that ERISC cores use to forward packets. On-device routing tables contain direction tables for both intra-mesh and inter-mesh routing [tt_metal/fabric/control_plane.cpp 119-141](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/control_plane.cpp#L119-L141)

*   **Intra-mesh**: Directions to chips within the same `MeshId`.
*   **Inter-mesh**: Directions to different meshes.
*   **Direction Mapping**: Device-side code uses `get_next_hop_router_direction` to query these tables based on `dst_mesh_id` and `dst_dev_id`.

### Fabric Erisc Data Mover (EDM)

The `fabric_erisc_router.cpp` kernel manages packet transmission over Ethernet links. It consists of:

*   **Sender Channel 0**: Accepts packets from local workers [tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp 64-65](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp#L64-L65)
*   **Sender Channel 1**: Accepts packets from upstream EDM routers [tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp 66-67](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp#L66-L67)
*   **Receiver Channel**: Accepts packets from the Ethernet link and either writes to local memory or forwards them [tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp 69-73](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp#L69-L73)

### 2-ERISC Mode (Blackhole)

On Blackhole architecture, the fabric can distribute sender and receiver responsibilities across two RISC cores:

*   **ERISC0**: Handles sender channels [tt_metal/fabric/erisc_datamover_builder.cpp 131-135](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/fabric/erisc_datamover_builder.cpp#L131-L135)
