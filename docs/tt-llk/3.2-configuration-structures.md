This page documents the low-level configuration structures that control hardware behavior in the unpacker, math, and packer subsystems. These structures define bit-field layouts and register mappings that configure Tensix core hardware for data processing operations.

For information about how these configurations are initialized and used in high-level operations, see:

*   Unpacker operations: [4.1](https://deepwiki.com/tenstorrent/tt-llk/4.1-core-unpacker-operations-and-configuration)
*   Math operations: [5.1](https://deepwiki.com/tenstorrent/tt-llk/5.1-core-math-unit-configuration)
*   Packer operations: [6.1](https://deepwiki.com/tenstorrent/tt-llk/6.1-core-packer-operations-and-configuration)

For information about data formats themselves (as opposed to how they're configured), see [3.1](https://deepwiki.com/tenstorrent/tt-llk/3.1-dataformat-enumeration-and-support).

The LLK library uses C++ bit-field structures to provide type-safe access to hardware configuration registers. These structures are organized by pipeline stage and map directly to hardware register layouts.

## Unpacker Configuration Structures

### unpack_tile_descriptor_t

The unpacker tile descriptor defines the dimensions and format of data being read from L1 memory. It consists of 4 words (128 bits) and controls how the unpacker interprets tile data.

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `in_data_format` | 4 | Input data format (see DataFormat enum) |
| `uncompressed` | 1 | Always 1 for uncompressed tiles |
| `blobs_per_xy_plane` | 4 | Number of blobs per XY plane |
| `x_dim` | 16 | Tile width in datums (e.g., 16 for 16x16 face) |
| `y_dim` | 16 | Tile height in datums |
| `z_dim` | 16 | Number of faces (1, 2, or 4) |
| `w_dim` | 16 | W dimension (typically 1) |

**Diagram: unpack_tile_descriptor_t Layout**

The tile descriptor is written to configuration registers `THCON_SEC0_REG0_TileDescriptor` for unpacker 0 (SrcA) and `THCON_SEC1_REG0_TileDescriptor` for unpacker 1 (SrcB).

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 22-49](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L22-L49)

### unpack_config_t

The unpacker configuration structure controls operational parameters including output format, throttling, and context management. It also consists of 4 words (128 bits).

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `out_data_format` | 4 | Output data format to source registers |
| `throttle_mode` | 2 | Throttling mode (typically 2) |
| `context_count` | 2 | Number of contexts (0 or 1) |
| `haloize_mode` | 1 | XY transpose enable for SrcA |
| `tileize_mode` | 1 | Tilization mode |
| `unpack_src_reg_set_update` | 1 | Source register set update |
| `unpack_if_sel` | 1 | Interface selection |
| `upsample_rate` | 2 | Upsampling rate |
| `uncompress_cntx0_3` | 4 | Uncompress enable for contexts 0-3 |
| `unpack_if_sel_cntx0_3` | 4 | Interface selection for contexts 0-3 |
| `force_shared_exp` | 1 | Force shared exponent mode |
| `uncompress_cntx4_7` | 4 | Uncompress enable for contexts 4-7 |
| `limit_addr` | 17 | Address limit for unpacker |
| `fifo_size` | 17 | FIFO size configuration |

The configuration is programmed into `THCON_SEC0_REG2_Out_data_format` and `THCON_SEC1_REG2_Out_data_format` registers. The function `configure_unpack_AB()` at [tt_llk_wormhole_b0/common/inc/cunpack_common.h 208-377](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L208-L377) demonstrates how to populate and write these structures.

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 52-88](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L52-L88)

## Math/ALU Configuration Structures

### alu_config_t

The ALU configuration structure controls data formats and operational modes for the math unit. It packs all configuration into a single 32-bit word.

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `ALU_ROUNDING_MODE_Fpu_srnd_en` | 1 | FPU stochastic rounding enable |
| `ALU_ROUNDING_MODE_Gasket_srnd_en` | 1 | Gasket stochastic rounding enable |
| `ALU_ROUNDING_MODE_Packer_srnd_en` | 1 | Packer stochastic rounding enable |
| `ALU_FORMAT_SPEC_REG0_SrcAUnsigned` | 1 | SrcA unsigned data flag |
| `ALU_FORMAT_SPEC_REG0_SrcBUnsigned` | 1 | SrcB unsigned data flag |
| `ALU_FORMAT_SPEC_REG0_SrcA` | 4 | SrcA data format |
| `ALU_FORMAT_SPEC_REG1_SrcB` | 4 | SrcB data format |
| `ALU_FORMAT_SPEC_REG2_Dstacc` | 4 | Destination accumulator format |
| `ALU_ACC_CTRL_Fp32_enabled` | 1 | FP32 accumulation mode enable |
| `ALU_ACC_CTRL_SFPU_Fp32_enabled` | 1 | SFPU FP32 mode enable |
| `ALU_ACC_CTRL_INT8_math_enabled` | 1 | Int8 math mode enable |

**Diagram: alu_config_t Bit Fields**

This structure is written to `ALU_FORMAT_SPEC_REG0_SrcA_ADDR32` register. The configuration function `_llk_math_hw_configure_()` at [tt_llk_wormhole_b0/llk_lib/llk_math_common.h 30-55](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_common.h#L30-L55) shows how to set these fields based on source data formats.

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 91-115](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L91-L115)

## Packer Configuration Structures

### pack_config_t

The packer configuration structure is the most complex, controlling data format conversion, compression settings, and L1 destination addressing. It consists of 4 words (128 bits).

**Structure Definition:**

| Field | Bits | Word | Description |
| --- | --- | --- | --- |
| `row_ptr_section_size` | 16 | 0 | Row pointer section size (unused) |
| `exp_section_size` | 16 | 0 | Exponent section size for BFP formats |
| `l1_dest_addr` | 32 | 1 | L1 destination address |
| `uncompress` | 1 | 2 | Uncompressed output (always 1) |
| `add_l1_dest_addr_offset` | 1 | 2 | Add L1 address offset |
| `out_data_format` | 4 | 2 | Output data format |
| `in_data_format` | 4 | 2 | Input format from dest registers |
| `src_if_sel` | 1 | 2 | Source interface selection |
| `pack_per_xy_plane` | 7 | 2 | Packs per XY plane |
| `l1_src_addr` | 8 | 2 | L1 source address offset |
| `downsample_mask` | 16 | 3 | Downsampling mask |
| `downsample_shift_count` | 3 | 3 | Downsample shift count |
| `read_mode` | 1 | 3 | Read mode |
| `exp_threshold_en` | 1 | 3 | Exponent threshold enable |
| `pack_l1_acc_disable_pack_zero_flag` | 2 | 3 | L1 accumulation flags |
| `exp_threshold` | 8 | 3 | Exponent threshold value |

The function `set_packer_config()` at [tt_llk_wormhole_b0/common/inc/cpack_common.h 224-352](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L224-L352) demonstrates how to configure these fields for different data formats and modes.

### relu_config_t

The ReLU configuration structure controls ReLU activation application during packing. It packs into a single 32-bit word.

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `ALU_ACC_CTRL_Zero_Flag_disabled_src` | 1 | Disable zero flag for source |
| `ALU_ACC_CTRL_Zero_Flag_disabled_dst` | 1 | Disable zero flag for destination |
| `STACC_RELU_ApplyRelu` | 4 | ReLU application mode (0=none, 3=max, others=min) |
| `STACC_RELU_ReluThreshold` | 16 | ReLU threshold value |
| `DISABLE_RISC_BP_*` | 10 | RISC breakpoint disable flags |

### dest_rd_ctrl_t

The destination read control structure configures how the packer reads from destination registers.

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `PCK_DEST_RD_CTRL_Read_32b_data` | 1 | Read 32-bit data (for FP32/Int32) |
| `PCK_DEST_RD_CTRL_Read_unsigned` | 1 | Read unsigned data |
| `PCK_DEST_RD_CTRL_Read_int8` | 1 | Read Int8 data |
| `PCK_DEST_RD_CTRL_Round_10b_mant` | 1 | Round to 10-bit mantissa |

This configuration is set within `set_packer_config()` and `reconfig_packer_data_format()` functions based on source and destination formats.

Sources: [tt_llk_wormhole_b0/common/inc/cpack_common.h 94-109](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L94-L109)

### pck_edge_offset_t

The edge offset structure is used for reduce operations to control which datums are written in each row of a tile face.

**Structure Definition:**

| Field | Bits | Description |
| --- | --- | --- |
| `mask` | 16 | Per-datum mask for row (1=write, 0=skip) |
| `mode` | 1 | Edge offset mode |
| `tile_row_set_select_pack0` | 2 | Row set mapping for packer 0 |
| `tile_row_set_select_pack1` | 2 | Row set mapping for packer 1 |
| `tile_row_set_select_pack2` | 2 | Row set mapping for packer 2 |
| `tile_row_set_select_pack3` | 2 | Row set mapping for packer 3 |

**Diagram: Reduce Masking Configuration**

The function `_llk_pack_reduce_mask_config_()` at [tt_llk_wormhole_b0/llk_lib/llk_pack_common.h 178-260](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_common.h#L178-L260) demonstrates how to configure these structures for row, column, and scalar reduce operations.

## Configuration Workflow

The typical workflow for configuring hardware involves:

1.   **Initialization**: Create and zero-initialize configuration structures
2.   **Field population**: Set required fields based on operation parameters
3.   **Register writing**: Write configuration to hardware registers
4.   **Validation** (optional): Read back and verify configuration

## Configuration Register Addresses

Configuration structures are written to specific hardware register addresses. The mapping is defined in hardware header files.

**Key Register Addresses:**

| Configuration | Register Base | Description |
| --- | --- | --- |
| Unpacker 0 Tile Descriptor | `THCON_SEC0_REG0_TileDescriptor_ADDR32` | SrcA tile descriptor |
| Unpacker 1 Tile Descriptor | `THCON_SEC1_REG0_TileDescriptor_ADDR32` | SrcB tile descriptor |
| Unpacker 0 Config | `THCON_SEC0_REG2_Out_data_format_ADDR32` | SrcA config |
| Unpacker 1 Config | `THCON_SEC1_REG2_Out_data_format_ADDR32` | SrcB config |
| Unpacker 0 Base Address | `THCON_SEC0_REG3_Base_address_ADDR32` | SrcA L1 address |
| ALU Format | `ALU_FORMAT_SPEC_REG0_SrcA_ADDR32` | Math format config |
| Packer 0 Config | `THCON_SEC0_REG1_Row_start_section_size_ADDR32` | PCK0 config |

| Packer 2 Config | `THCON_SEC1_REG1_Row_start_section_size_ADDR32` | PCK2 config |
| Packer 3 Config | `THCON_SEC1_REG8_Row_start_section_size_ADDR32` | PCK3 config |
| ReLU Config | `STACC_RELU_ApplyRelu_ADDR32` | ReLU threshold |

## Runtime Reconfiguration

Configuration structures can be efficiently reconfigured at runtime to support dynamic format changes without full reinitialization.

**Reconfiguration Functions:**

| Function | Purpose | Key Operations |
| --- | --- | --- |
| `_llk_unpack_reconfig_data_format_srca_impl_()` | Change SrcA format | Update tile descriptor, config format, tile size |
| `_llk_unpack_reconfig_data_format_srcb_impl_()` | Change SrcB format | Update tile descriptor, config format, tile size |
| `_llk_math_reconfig_data_format_()` | Change math formats | Update ALU_FORMAT_SPEC registers |
| `reconfig_packer_data_format()` | Change packer format | Update pack_config, dest_rd_ctrl, exp sizes |

**Diagram: Reconfiguration vs Full Configuration**

The reconfiguration functions use `cfg_reg_rmw_tensix()` to perform read-modify-write operations, updating only necessary fields while preserving others. This is more efficient than full reconfiguration.

## Special Configuration Cases

### BFP Format Exponential Section Sizes

Block Floating Point (BFP) formats require special configuration for exponent section sizes. The `exp_section_size` field in `pack_config_t` varies based on:

*   BFP data format (Bfp2, Bfp4, or Bfp8)
*   Number of faces being packed (1, 2, or 4)
*   Packer index (0-3)

The function `cache_exponential_section_sizes_in_gprs()` at [tt_llk_wormhole_b0/common/inc/cpack_common.h 169-201](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L169-L201) pre-computes and caches these values in GPRs for efficient reconfiguration.

### FP32 Destination Accumulator Mode

When FP32 destination accumulation is enabled (`is_fp32_dest_acc_en = true`), several configuration fields require special handling:

*   `dest_rd_ctrl.PCK_DEST_RD_CTRL_Read_32b_data` must be set
*   `alu_config.ALU_ACC_CTRL_Fp32_enabled` must be set
*   `alu_config.ALU_ACC_CTRL_SFPU_Fp32_enabled` must be set
*   Packer `exp_threshold` workaround may be needed (see [tt_llk_wormhole_b0/common/inc/cpack_common.h 249-257](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L249-L257))

### Narrow Tile Support (32x16)

Narrow tiles (32x16 instead of standard 32x32) require adjusted configuration:

*   Unpacker `face_r_dim` set to 8 instead of 16
*   Packer offset calculations modified in `_llk_init_packer_dest_offset_registers_()` at [tt_llk_wormhole_b0/llk_lib/llk_pack_common.h 62-109](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_common.h#L62-L109)
