The Direct-to-Metal (D2M) workflows automate the integration testing of `tt-emule` against the `tt-mlir` compiler stack. These workflows ensure that the emulation layer remains compatible with the kernels and programs generated by the MLIR-based compiler. This is achieved through a cascading dependency model that synchronizes specific versions of `tt-emule`, `tt-mlir`, and `tt-metal`.

## Workflow Overview and Dependency Resolution

The D2M system utilizes two primary workflows:

1.   **PR D2M Regression**: Triggered on pull requests to `main`. It validates the PR against a specific "pinned" version of `tt-mlir` defined in `tt-mlir-pin.txt`[.github/workflows/pr-d2m-regression.yml 1-10](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L1-L10)
2.   **Nightly D2M Upstream**: Runs daily against the `main` tip of `tt-mlir` to detect upstream breakages and warm the build cache [.github/workflows/nightly-d2m-upstream.yml 1-12](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/nightly-d2m-upstream.yml#L1-L12)

### Cascading Dependency Logic

To ensure a consistent environment, the CI resolves dependencies in a three-tier cascade:

1.   **tt-emule**: The source of truth for the current PR or branch.
2.   **tt-mlir**: The SHA is read from `tt-mlir-pin.txt`[.github/workflows/pr-d2m-regression.yml 52-57](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L52-L57)
3.   **tt-metal**: The SHA is dynamically extracted from the `tt-mlir` source tree by inspecting `third_party/CMakeLists.txt` for the `TT_METAL_VERSION` variable [.github/workflows/pr-d2m-regression.yml 59-69](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L59-L69)

## Build and Preparation Pipeline

The build process is handled by `ci-build-mlir.sh`. It compiles the entire stack with `TT_METAL_USE_EMULE=ON` to redirect hardware calls to the emulator.

### Build Steps

1.   **Checkout**: All three repositories are checked out into sibling directories [.github/workflows/pr-d2m-regression.yml 43-88](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L43-L88)
2.   **Submodule Init**: Minimal submodules (UMD and Tracy) are initialized for `tt-metal`[.github/workflows/pr-d2m-regression.yml 89-98](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L89-L98)
3.   **Compilation**: `ci-build-mlir.sh` invokes CMake to build `tt-mlir`, which in turn builds its internal `tt-metal` dependency using the provided `tt-emule` source [.github/workflows/pr-d2m-regression.yml 109-116](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L109-L116)
4.   **Artifact Staging**: Build outputs are tarred and uploaded to avoid re-compilation in the test stage [.github/workflows/pr-d2m-regression.yml 117-135](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L117-L135)

### Test Preparation

Before running tests, `ci-d2m-prepare.sh` (invoked within the `test` job) prepares the environment:

*   **ttrt Installation**: The `tt-runtime-tools` (ttrt) wheel is installed from the build directory [.github/workflows/nightly-d2m-upstream.yml 180-199](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/nightly-d2m-upstream.yml#L180-L199)
*   **System Description**: Generates `system_desc.ttsys` which describes the emulated hardware topology to the `tt-mlir` runtime.

## Test Execution and Regression Analysis

Testing is performed by `run_d2m_regression.sh`. This script executes the `tt-mlir` "golden" pytest suite against the emulator.

### Execution Model

The runner supports both serial and parallel execution of test files [.run_d2m_regression.sh 108-112](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.run_d2m_regression.sh#L108-L112) It configures the environment to force emulation mode:

*   `TT_METAL_EMULE_MODE=1`
*   `TT_METAL_SLOW_DISPATCH_MODE=1`
*   `TT_METAL_MOCK_CLUSTER_DESC_PATH` points to the architecture-specific YAML (e.g., `wormhole_N150.yaml`) [.run_d2m_regression.sh 95-98](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.run_d2m_regression.sh#L95-L98)

### Hang and Timeout Protection

The workflow implements a two-tier timeout strategy to handle potential deadlocks in the emulator:

1.   **`PYTEST_TIMEOUT`**: A per-test-case timeout enforced via the `pytest-timeout` plugin [.run_d2m_regression.sh 24-27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.run_d2m_regression.sh#L24-L27)
2.   **`FILE_TIMEOUT`**: A bash-level ceiling (default 3600s) that kills the entire process if the Python interpreter hangs in native code [.run_d2m_regression.sh 28](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.run_d2m_regression.sh#L28-L28)

### Results Classification

After execution, `classify-d2m-results.py` parses the JUnit XML outputs and compares them against architecture-specific allowlists (e.g., `known-failures-d2m-wormhole.txt`).

| Category | Description | CI Action |
| --- | --- | --- |
| **New Failures** | Tests that failed but are not in the allowlist. | **Fail** |
| **Newly Passing** | Tests that are allowlisted but passed in this run. | **Fail** (Requires allowlist cleanup) |
| **Stale Entries** | Allowlist patterns that matched no tests in the suite. | **Fail** |
| **Expected Failures** | Tests that failed and match an allowlist pattern. | **Pass** |
