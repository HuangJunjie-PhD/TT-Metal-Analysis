This page provides instructions for setting up a local development environment to build, test, and contribute to the tt-system-firmware project. It covers initial setup, building firmware for different board variants, running tests locally, and generating documentation.

For information about the overall system architecture and firmware components, see [System Architecture](https://deepwiki.com/tenstorrent/tt-system-firmware/1.1-system-architecture). For coding standards and contribution guidelines, see [Coding Guidelines and Standards](https://deepwiki.com/tenstorrent/tt-system-firmware/9.3-coding-guidelines-and-standards).

## Prerequisites

Before setting up the development environment, ensure the following dependencies are installed on your system:

### Required Tools

| Tool | Minimum Version | Purpose |
| --- | --- | --- |
| Python | 3.12+ | Build scripts, west, documentation |
| CMake | 3.20+ | Build system |
| Ninja or Make | - | Build backend |
| Git | 2.25+ | Version control and west |
| Devicetree Compiler (`dtc`) | 1.4.6+ | Board definitions |

### Python Dependencies

The project requires several Python packages for building and testing:

`pip install west>=0.14.0pip install -r scripts/requirements-actions.txt`
Key Python packages include `west` (Zephyr meta-tool), `imgtool` (firmware signing), `intelhex` (firmware formats), `pytest` (testing), and Sphinx/Doxygen dependencies for documentation.

**Sources:**[scripts/requirements-actions.txt](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/requirements-actions.txt)[scripts/requirements-actions.in](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/requirements-actions.in)

### Zephyr SDK

The Zephyr SDK provides cross-compilation toolchains for ARC and RISC-V architectures used by SMC and DMC firmware:

`# Download and install Zephyr SDK (example for Linux)wget https://github.com/zephyrproject-rtos/sdk-ng/releases/download/v0.16.5/zephyr-sdk-0.16.5_linux-x86_64.tar.xztar xf zephyr-sdk-0.16.5_linux-x86_64.tar.xzcd zephyr-sdk-0.16.5./setup.sh`
Refer to the [Zephyr Getting Started Guide](https://docs.zephyrproject.org/latest/develop/getting_started/index.html) for platform-specific installation instructions.

**Sources:**[.github/workflows/build-fw.yml 57-59](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L57-L59)

## Repository Setup

### Cloning and Initializing

**Figure 1: West Workspace Initialization Flow**

The repository uses `west` as a meta-tool to manage dependencies defined in the project manifest:

`# Clone the repositorygit clone https://github.com/tenstorrent/tt-system-firmware.gitcd tt-system-firmware # Initialize west workspacewest init -l . # Fetch all dependencieswest update # Optional: Fetch full git history for versioningwest forall -c 'git fetch --unshallow || true'`
The `west.yml` manifest defines the following key dependencies:

*   **zephyr**: Tenstorrent fork of Zephyr RTOS at commit `01cee78c8305`
*   **librpmi**: RISC-V Power Management Interface library
*   **mcuboot**: Bootloader for firmware signing
*   **mbedtls**: Cryptography library

**Sources:**[west.yml 1-33](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/west.yml#L1-L33)[.github/workflows/build-fw.yml 65-68](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L65-L68)

### Workspace Structure

After initialization, the workspace structure is:

```
.
├── tt-system-firmware/        # This repository (application code)
│   ├── app/                   # SMC and DMC applications
│   ├── lib/                   # Board-specific libraries
│   ├── boards/                # Board definitions
│   ├── zephyr/                # Zephyr module integration
│   └── west.yml               # Project manifest
├── zephyr/                    # Zephyr RTOS
├── modules/                   # Dependencies from manifest
└── bootloader/                # MCUBoot (if using sysbuild)
```

## Building Firmware

### Board Selection

Firmware can be built for different board variants defined in `.github/boards.json`. Common boards include:

| Board Identifier | Hardware | Description |
| --- | --- | --- |
| `p100a` | Blackhole P100A | Single-chip development board |
| `p150a` | Blackhole P150A | Dual-chip board |
| `p300a` | Blackhole P300A | Quad-chip board |
| `galaxy-32` | Galaxy | 32-chip system |

**Sources:**[.github/workflows/build-fw.yml 36-44](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L36-L44)

### Build Process

**Figure 2: Firmware Build Pipeline**

### Basic Build Commands

To build firmware for a specific board:

`# Build for P100A boardBOARD=$(./scripts/rev2board.sh "p100a")west build -d build-p100a -p -b $BOARD app/smc --sysbuild # Build output locationls build-p100a/update.fwbundlels build-p100a/smc/zephyr/zephyr.signed.binls build-p100a/dmc/zephyr/zephyr.signed.bin`
The `--sysbuild` flag enables building multiple applications (SMC, DMC, recovery) and MCUBoot bootloader in a single invocation.

**Sources:**[.github/workflows/build-fw.yml 86-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L86-L106)

### Build with Signing

Production builds require firmware signing. The build system uses MCUBoot's `imgtool` for signature generation:

`# Generate or provide signing keySIGNATURE_KEY_FILE="path/to/signing-key.pem" west build \  -d build-p100a \  -p \  -b tt_blackhole@p100a/tt_blackhole/smc \  app/smc \  --sysbuild \  -- \  -DCONFIG_COMPILER_WARNINGS_AS_ERRORS=y \  -DSB_CONFIG_BOOT_SIGNATURE_KEY_FILE="$SIGNATURE_KEY_FILE" # Verify signaturesimgtool verify --key "$SIGNATURE_KEY_FILE" build-p100a/smc/zephyr/zephyr.signed.binimgtool verify --key "$SIGNATURE_KEY_FILE" build-p100a/dmc/zephyr/zephyr.signed.bin`
**Sources:**[.github/workflows/build-fw.yml 77-121](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L77-L121)

### Build Artifacts

Each build produces the following artifacts in `build-<board>/`:

| Artifact | Description |
| --- | --- |
| `update.fwbundle` | Combined firmware bundle for all apps |
| `smc/zephyr/zephyr.elf` | SMC firmware ELF (with debug symbols) |
| `smc/zephyr/zephyr.signed.bin` | Signed SMC binary |
| `dmc/zephyr/zephyr.signed.bin` | Signed DMC binary |
| `recovery/zephyr/zephyr.signed.bin` | Signed recovery firmware |
| `**/.config` | Kconfig configuration |
| `**/zephyr.dts` | Devicetree configuration |

**Sources:**[.github/workflows/build-fw.yml 123-138](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L123-L138)

## Running Tests

### Unit Tests

Unit tests run on the `native_sim` platform without hardware:

`# Build and run unit testswest twister -i -p native_sim -p unit_testing -T tests/ -T samples/ # Run specific testwest twister -i -p native_sim -T tests/lib/tenstorrent/test_telemetry/`
The `twister` tool (Zephyr's test runner) compiles tests, executes them, and generates reports:

**Sources:**[.github/workflows/run-unit-tests.yml 26-66](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/run-unit-tests.yml#L26-L66)

### Test Organization

**Figure 3: Test Execution Flow**

### Hardware Tests

Hardware tests require physical boards connected via JTAG. These tests are typically run in CI but can be executed locally if hardware is available:

`# Build-only tests for specific boardwest twister \  -p tt_blackhole@p150a/tt_blackhole/dmc \  --build-only \  -T app/smc \  --outdir twister-smc-tests-p150a`
See the [Hardware Test Infrastructure](https://deepwiki.com/tenstorrent/tt-system-firmware/8.2-hardware-test-infrastructure) documentation for details on running tests on physical hardware.

**Sources:**[.github/workflows/build-fw.yml 398-454](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L398-L454)

### Test Results

Test results are stored in:

*   `twister-out/twister.json`: Machine-readable test results
*   `twister-out/twister.log`: Human-readable test log
*   `twister-out/**/handler.log`: Individual test logs

**Sources:**[.github/workflows/run-unit-tests.yml 51-66](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/run-unit-tests.yml#L51-L66)

## Common Build Configurations

### Debug Build

`west build -d build-debug -b tt_blackhole@p100a/tt_blackhole/smc app/smc --sysbuild \  -- \  -DCONFIG_DEBUG=y \  -DCONFIG_ASSERT=y \  -DCONFIG_LOG_DEFAULT_LEVEL=4`
### RTT Console Build

For debugging with Segger RTT:

`west build -d build-rtt -b tt_blackhole@p100a/tt_blackhole/smc app/smc --sysbuild \  -S rtt-console \  -- \  -DCONFIG_USE_SEGGER_RTT=y \  -DCONFIG_RTT_CONSOLE=y`
**Sources:**[.github/workflows/build-fw.yml 258-265](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L258-L265)

Special builds for board assembly testing:

`west build \  -d build-assembly-test-p150 \  -b tt_blackhole@p150a/tt_blackhole/dmc \  app/dmc \  --sysbuild \  -S rtt-console \  -- \  -DCONFIG_TT_ASSEMBLY_TEST=y`
**Sources:**[.github/workflows/build-fw.yml 242-265](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L242-L265)
