## Purpose and Scope

The OpModel Analysis System provides compile-time hardware constraint checking and runtime estimation for TTNN operations. It enables the compiler to query resource usage (L1 memory, circular buffer size) and performance characteristics (execution cycles) before lowering operations to hardware. This information drives optimization decisions such as layout selection, memory allocation, and operation validation.

For information about TTNN layout optimization passes that consume OpModel data, see [TTNN Layout and Memory Optimization](https://deepwiki.com/tenstorrent/tt-mlir/5.2-ttnn-layout-and-memory-optimization). For runtime execution of compiled programs, see [Runtime Architecture and Multi-Backend Support](https://deepwiki.com/tenstorrent/tt-mlir/4.1-runtime-architecture-and-multi-backend-support).

## System Architecture

Title: OpModel System Data Flow

The system bridges MLIR operations to tt-metal hardware queries through a conversion and caching layer. Operations implement the `OpModel` interface, which dispatches to specialized templates that query the hardware using the tt-metal graph API.

## Core Components

### OpModel Template Hierarchy

The OpModel system uses C++ template specialization to provide operation-specific query implementations. Base templates like `UnaryEltwiseOpModel` or `BinaryEltwiseOpModel` provide default logic for common operation signatures.

Title: OpModel Template Specialization

Operations are grouped by signature patterns into base templates. Each operation type specializes `OpModel<OpTy>` to inherit from the appropriate base template or provide custom implementations for complex operations.

Sources: [include/ttmlir/OpModel/TTNN/TTNNOpModel.h 56-181](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/OpModel/TTNN/TTNNOpModel.h#L56-L181)[include/ttmlir/OpModel/TTNN/TTNNOpModel.h 234-245](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/OpModel/TTNN/TTNNOpModel.h#L234-L245)

### OpConstraints Structure

The `OpConstraints` structure encapsulates resource usage information returned from hardware queries:

| Field | Type | Description |
| --- | --- | --- |
| `cb_peak_size_per_core` | `size_t` | Peak circular buffer size per core in L1 memory |
| `l1_buffers_peak_per_core` | `size_t` | Peak tensor buffer size per core in L1 memory |
| `peak_memory_usage_per_core` | `size_t` | Total peak memory usage per core |
| `l1_output_buffer_per_core` | `size_t` | Output tensor buffer size per core in L1 memory |
| `outputLayouts` | `llvm::SmallVector<TTNNLayoutAttr>` | Actual output tensor layouts determined by hardware |

Sources: [lib/OpModel/TTNN/TTNNOpModel.cpp 158-163](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/OpModel/TTNN/TTNNOpModel.cpp#L158-L163)[test/unittests/OpModel/TTNN/Lib/TestOpModelLib.cpp 91-93](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/unittests/OpModel/TTNN/Lib/TestOpModelLib.cpp#L91-L93)

## Validation and Fallback

The `TTNNOperationValidationAndFallback` pass uses OpModel to ensure operations are legal for hardware execution. It leverages the `OpModel` interface to check constraints and applies transformations if an operation fails validation [lib/Dialect/TTNN/Transforms/OptimizerPasses/OperationValidationAndFallback.cpp 149-185](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Transforms/OptimizerPasses/OperationValidationAndFallback.cpp#L149-L185)

Title: Validation and Fallback Logic
