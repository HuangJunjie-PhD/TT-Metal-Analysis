Combined firmware bundles are comprehensive distribution packages that consolidate all firmware components required to run Tenstorrent Blackhole hardware across multiple board variants. A combined bundle packages together SMC firmware, DMC firmware, recovery firmware, and binary firmware blobs for all supported hardware subsystems into a single versioned `.fwbundle` file for distribution and deployment.

For information about individual firmware components within bundles, see [Binary Firmware Components](https://deepwiki.com/tenstorrent/tt-system-firmware/5-binary-firmware-components). For details on how bundles are signed and verified, see [Firmware Signing and MCUBoot](https://deepwiki.com/tenstorrent/tt-system-firmware/6.3-firmware-signing-and-mcuboot).

## Bundle Purpose and Scope

Combined firmware bundles serve as the primary distribution format for production firmware releases. A single `fw_pack-blackhole.fwbundle` file contains:

*   **SMC firmware** for all board variants (P100A, P150A, P300A, Galaxy)
*   **DMC firmware** for multi-chip boards (P150A, P300A, Galaxy)
*   **Recovery firmware** for emergency boot scenarios
*   **Binary firmware blobs** for MRISC, ERISC, SerDes, and bootrom
*   **Version metadata** tracking product version and component versions

This approach ensures consistent firmware deployment across all hardware platforms and simplifies firmware updates by providing a single atomic package.

## Bundle Contents Structure

The following diagram shows the hierarchical organization of components within a combined firmware bundle:

**Component Categories:**

| Category | Components | Purpose |
| --- | --- | --- |
| **SMC Firmware** | `smc/zephyr.signed.bin` per board | System Management Controller firmware running on ARC processor |
| **DMC Firmware** | `dmc/zephyr.signed.bin` per board | Device Management Controller firmware for multi-chip boards |
| **Recovery Firmware** | `recovery/zephyr.signed.bin` | Emergency boot firmware for recovery scenarios |
| **MRISC Blobs** | `tt_blackhole_gddr_init.bin` + 13 param variants | Memory controller firmware for GDDR training |
| **ERISC Blobs** | `tt_blackhole_erisc.bin` + params + SerDes | Ethernet controller firmware |
| **PCIe Blobs** | `tt_blackhole_libpciesd.a` | PCIe SerDes initialization library |
| **Bootrom** | `tt_blackhole_nano_bootcode.bin` | Modified bootrom for Blackhole chip initialization |
| **Legacy Platforms** | `fw_pack-wormhole.tar.gz`, `fw_pack-grayskull.tar.gz` | Pre-packaged firmware for Wormhole and Grayskull ASICs |

## Build Pipeline

The combined firmware bundle is created through a multi-stage GitHub Actions pipeline that builds individual board firmwares and then packages them together:

**Build Workflow Steps:**

1.   **Board Configuration Generation** (`genboards` job)

    *   Reads board variants from `.github/boards.json`
    *   Outputs matrix strategy for parallel builds

2.   **Matrix Firmware Builds** (`build` job)

    *   Runs for each board variant in parallel
    *   Executes `west build --sysbuild` with board-specific configuration
    *   Signs firmware using `imgtool` with `SIGNATURE_KEY_FILE`
    *   Produces `update.fwbundle` containing SMC, DMC (if applicable), and recovery firmware
    *   Uploads as `firmware-{board}` artifact

3.   **Recovery Firmware Build** (`build-recovery` job)

    *   Builds standalone recovery firmware for emergency scenarios
    *   Targets `tt_blackhole@p100a/tt_blackhole/smc`
    *   Uses `sysbuild/recovery.conf` configuration overlay

4.   **Combined Bundle Creation** (`build-combined-fwbundle` job)

    *   Downloads all `firmware-*` artifacts
    *   Sets `BUNDLE_TEMP_PREFIX` environment variable
    *   Invokes `scripts/build-combined-fwbundle.sh`
    *   Packages individual bundles with binary blobs
    *   Produces final `fw_pack-blackhole.fwbundle`

**Sources:**[.github/workflows/build-fw.yml 36-380](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L36-L380)

## Bundle Creation Script

The `build-combined-fwbundle.sh` script orchestrates the final packaging step:

**Key Operations:**

| Operation | Description | Files Involved |
| --- | --- | --- |
| **Artifact Download** | Download all `firmware-*` artifacts from previous jobs | `firmware-{p100a,p150a,p300a,galaxy}/update.fwbundle` |
| **Dependency Installation** | Install Python packages for bundle creation | `pykwalify`, `intelhex`, `imgtool` |
| **Binary Blob Collection** | Gather blobs declared in `zephyr/module.yml` | All `tt_blackhole_*.bin` and `*.a` files |
| **Bundle Assembly** | Combine per-board firmwares with shared blobs | Creates unified `.fwbundle` structure |
| **Metadata Generation** | Generate version manifest and checksums | Product version, component versions, SHA256 sums |

The script uses `BUNDLE_TEMP_PREFIX` to locate downloaded firmware artifacts and combines them with binary blobs from `zephyr/blobs/` directory.

**Sources:**[.github/workflows/build-fw.yml 365-370](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L365-L370)

## Binary Blob Integration

Binary firmware blobs are declared in `zephyr/module.yml` and automatically included in combined bundles:

**Blob Declaration Structure:**

Each blob entry in `module.yml` contains:

*   `path`: Relative path in `zephyr/blobs/` directory
*   `sha256`: Integrity checksum for verification
*   `type`: Blob type (`img` or `lib`)
*   `version`: Component version string
*   `license-path`: License file reference
*   `url`: Source URL for blob
*   `description`: Human-readable description

**Example ERISC Blob Declaration:**

`- path: tt_blackhole_erisc.bin  sha256: c797a2f75823c5499791c4c7d02a9196ede520d11e03bd4fc1ffd583767ad28f  type: img  version: '1.9.0'  license-path: zephyr/blobs/license.txt  url: "https://github.com/tenstorrent/tt-system-firmware/..."  description: "Ethernet FW"`
**Blob Integrity Verification:**

The `verify_blob.py` script validates blob checksums before bundle creation:

1.   Reads blob declarations from `MODULE_YAML_PATH` (`zephyr/module.yml`)
2.   Computes SHA256 hash of each file in `BLOBS_BASE_DIR` (`zephyr/blobs/`)
3.   Compares computed hash with declared `sha256` value
4.   Fails build if any blob has been modified

This ensures bundle integrity and prevents accidental inclusion of modified binary components.

## Board Variant Parameterization

Combined bundles support multiple board variants through parameterized GDDR configurations:

**Board Variant Strategy:**

*   **Single Base Firmware**: One MRISC binary (`tt_blackhole_gddr_init.bin`) for all boards
*   **Parameter Files**: Board-specific GDDR training parameters (13 variants)
*   **Runtime Selection**: Firmware loads appropriate parameters based on board detection
*   **Revision Support**: Multiple revisions (A, B, C) for iterative hardware improvements

This parameterization approach minimizes firmware duplication while supporting hardware diversity.

**Sources:**[zephyr/module.yml 82-198](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L82-L198)

## Firmware Signing Integration

All firmware components within combined bundles are cryptographically signed:

**Signing Process:**

1.   **Key Management**

    *   `SIGNATURE_KEY` stored as GitHub Actions secret
    *   Base64-decoded to temporary file during build
    *   Passed to builds via `-DSB_CONFIG_BOOT_SIGNATURE_KEY_FILE`

2.   **Signing Verification**

    *   `imgtool verify` executed for each signed binary
    *   Validates SMC, DMC, and recovery firmware signatures
    *   Build fails if any signature is invalid

3.   **Signed Artifacts**

    *   `zephyr.signed.bin`: Signed binary with MCUBoot header
    *   `zephyr.signed.hex`: Intel HEX format for programming

**Signature Verification Example:**

`imgtool verify --key "$SIGNATURE_KEY_FILE" build-p100a/smc/zephyr.signed.bin`
**Sources:**[.github/workflows/build-fw.yml 77-121](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L77-L121)

## Bundle Distribution and Usage

Combined firmware bundles are distributed through GitHub Actions artifacts:

**Artifact Upload:**

`- name: Upload Combined Firmware Bundle Assets  uses: actions/upload-artifact@v4  with:    name: combined-fwbundle    path: |      fw_pack*.fwbundle`
**Output Configuration:**

The `build-combined-fwbundle` job exposes the artifact name for downstream workflows:

`outputs:  combined-fwbundle-artifact: ${{ steps.set_combined_fwbundle_output.outputs.combined_fwbundle_artifact }}`
This allows other workflows (e.g., hardware testing, deployment) to reference and download the combined bundle for firmware updates.

**Bundle Naming Convention:**

*   Format: `fw_pack-{platform}.fwbundle`
*   Examples: `fw_pack-blackhole.fwbundle`, `fw_pack-wormhole.tar.gz`
*   Platform identifier matches target ASIC architecture

**Sources:**[.github/workflows/build-fw.yml 372-380](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L372-L380)

## CI/CD Integration

Combined firmware bundle creation is integrated into the standard build workflow:

**Workflow Call Interface:**

`on:  workflow_call:    secrets:      SIGNATURE_KEY:        required: true    outputs:      combined-fwbundle-artifact:        description: "The combined firmware bundle artifact name"        value: ${{ jobs.build-combined-fwbundle.outputs.combined-fwbundle-artifact }}`
This interface allows the build workflow to be called from other workflows (e.g., release, deployment) and receive the bundle artifact name for downstream processing.

**Trigger Conditions:**

*   Push to `main`, `v*-branch`, or `collab*` branches
*   Pull requests to these branches
*   Merge group checks
*   Manual workflow dispatch
