# Numerical Accuracy

Relevant source files

## Purpose and Scope

This page documents ttsim's numerical accuracy characteristics, explaining which operations produce bit-exact results matching Tenstorrent silicon and which operations may diverge. It covers the accuracy guarantees for different hardware components, sources of potential divergence, and strategies for ensuring deterministic results.

For general configuration requirements, see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration). For information about error messages related to undefined behavior, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference). For implementation status of specific features, see [Feature Status](https://deepwiki.com/tenstorrent/ttsim/7.3-feature-status).

## Bit-Exactness Goal

ttsim is designed to provide **bit-exact** numerical results relative to Tenstorrent silicon for all computations, floating-point and otherwise. The goal is to match all hardware computations bit-for-bit across:

*   All instructions and opcodes
*   All functional units (Unpacker, FPU, SFPU, Packer)
*   All special cases and edge conditions
*   Precise bit representation of NaNs produced by operations

While bugs are inevitable and some code paths are not yet bit-exact, this fidelity is the intended target. The simulator aims to reproduce not just numerically close results, but the exact bit patterns produced by silicon hardware.

This level of accuracy allows developers to:

*   Debug numerical issues without access to physical hardware
*   Verify algorithmic correctness with confidence
*   Validate functional correctness before silicon runs
*   Reproduce silicon behavior deterministically

## Component Accuracy Status

The Tensix core pipeline consists of multiple functional units, each with different accuracy implementation status. The following diagram shows the data flow through these components and their current accuracy levels:

### Current Implementation Status

| Component | Accuracy Status | Notes |
| --- | --- | --- |
| **Unpacker** | Bit-Exact ✓ | Believed to be fully bit-accurate |

| **FPU: GMPOOL** | Bit-Exact ✓ | Global max pooling operations are bit-accurate |

The majority of the Tensix pipeline produces bit-exact results. The primary source of numerical divergence is a subset of FPU operations that are implemented but not yet fully bit-accurate. Work is ongoing to fix these operations "to the extent possible."

## Sources of Divergence

Even when all implemented operations are bit-exact, several categories of program behavior can produce results that diverge from silicon. Understanding these sources is critical for writing deterministic code.

### Timing-Dependent Operations

For timing-dependent computations, ttsim may evaluate operations in **any order permitted by software synchronization**. This may include operation orders that are extremely unlikely on silicon but are technically allowed by the memory model.

**Example: Floating-Point Reduction**

Consider a parallel floating-point sum:

```
// Non-deterministic - result depends on execution order
result = a + b + c + d  // parallel additions, no synchronization
```

Since floating-point addition is not associative, different execution orders produce different results:

*   `((a + b) + c) + d` may differ from
*   `(a + (b + c)) + d` or
*   `((a + c) + (b + d))`

To ensure an exact match with silicon, avoid algorithms where the runtime order of operations affects the result, or explicitly serialize operations in a deterministic order.

### Non-Deterministic Hardware Features

| Feature | Divergence Reason | Mitigation |
| --- | --- | --- |
| **Hardware entropy sources** | Random number generators produce different sequences | Use seeded PRNG or avoid random operations |
| **Performance counters** | Cycle counts differ between simulation and silicon | Do not use cycle counts in computation |
| **Timers** | Time measurements differ due to simulation overhead | Do not use time values in computation |
| **Cycle counters** | Timing characteristics fundamentally different | Do not use timing in algorithms |

### Memory Consistency Issues

| Issue | Divergence Reason | Mitigation |
| --- | --- | --- |
| **Missing synchronization** | Unordered memory operations may execute differently | Add proper barriers/fences |
| **Missing cache flushes** | Stale data in cache may be read | Flush caches at appropriate points |
| **Missing memory fences** | Memory ordering may differ from expectations | Insert memory fences |

### ISA Specification Violations

| Violation | Divergence Reason | Detection |
| --- | --- | --- |
| **UndefinedBehavior** | No guarantees about result | Simulator reports error |
| **UnpredictableValue** | Result is unpredictable per ISA spec | Simulator reports error |
| **NonContractualBehavior** | Behavior not guaranteed by specification | Simulator reports error |

When code executes operations with UndefinedBehavior or reads UnpredictableValues, the simulator may produce different results than silicon, or may report an error. See the [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) for precise definitions of these terms.

## Determinism Guarantees

The following diagram illustrates which execution patterns provide deterministic results:

### Execution Determinism Model

### Ensuring Deterministic Results

To guarantee bit-exact results matching silicon:

1.   **Serialize Order-Dependent Operations**

    *   For floating-point reductions, explicitly order each operation
    *   Use barriers to enforce operation ordering
    *   Document assumptions about execution order

2.   **Add Proper Synchronization**

    *   Insert memory barriers between dependent operations
    *   Flush caches when required by ISA
    *   Use memory fences to enforce ordering

3.   **Avoid Timing Dependencies**

    *   Do not use cycle counters in computation logic
    *   Do not use timer values in algorithms
    *   Do not rely on performance counter values

4.   **Avoid Random Sources**

    *   Do not read hardware random number generators
    *   Use deterministic, seeded PRNGs if randomness needed
    *   Avoid entropy sources

5.   **Follow ISA Specifications**

    *   Avoid UndefinedBehavior patterns
    *   Do not use UnpredictableValues in computations
    *   Follow all ISA requirements for cache coherence

## Accuracy Testing and Verification

When developing code for ttsim, you can verify numerical accuracy through comparison with silicon runs:

### Verification Process

### Investigating Divergence

If ttsim results diverge from silicon:

1.   **Check for Known Non-Bit-Exact Operations**

    *   Verify if divergent operation uses FPU ELW*, MVMUL, or GAPOOL opcodes
    *   These are known to be approximate currently
    *   Check release notes for accuracy improvements

2.   **Look for Timing Dependencies**

    *   Review code for parallel operations without synchronization
    *   Check for floating-point reductions with undefined order
    *   Add barriers to enforce deterministic ordering

3.   **Verify Synchronization**

    *   Ensure all memory barriers are present
    *   Check cache flush operations
    *   Validate memory fence placement

4.   **Check for ISA Violations**

    *   Review ttsim error messages for UndefinedBehavior reports
    *   Look for UnpredictableValue usage
    *   Verify all operations follow ISA specification

5.   **Report Suspected Bugs**

    *   If none of above explain divergence, may be simulator bug
    *   File issue with minimal reproducible example
    *   Include both ttsim and silicon outputs

## Summary

ttsim provides bit-exact numerical accuracy for most operations, with the following characteristics:

| Aspect | Status |
| --- | --- |
| **Unpacker, SFPU, Packer** | Bit-exact ✓ |
| **FPU MOV* and GMPOOL** | Bit-exact ✓ |
| **FPU ELW*, MVMUL, GAPOOL** | Approximate (work in progress) ⚠ |
| **Deterministic Code** | Bit-exact when properly synchronized ✓ |
| **Timing-Dependent Code** | May diverge due to execution order ⚠ |
| **ISA Violations** | No guarantees (error reported) ✗ |

For maximum confidence in numerical results:

*   Use proper synchronization for all dependent operations
*   Avoid timing-dependent logic
*   Follow ISA specifications strictly
*   Test on silicon when possible to verify results
