This document enumerates all data formats supported by the tt-llk library, specifying their bit widths, memory layouts, hardware constraints, and intended use cases. The `DataFormat` enumeration defines the fundamental data types that can flow through the three-stage LLK pipeline (Unpack-Math-Pack).

For information about format conversion mechanisms and the internal TF32 representation used during computation, see [Format Conversion Pipeline](https://deepwiki.com/tenstorrent/tt-llk/3.4-format-conversion-and-inference-pipeline). For details on runtime format reconfiguration within kernels, see [Runtime Format Reconfiguration](https://deepwiki.com/tenstorrent/tt-llk/3.5-runtime-format-reconfiguration). For block floating-point exponential section calculations, see [Block Floating-Point and MX Format Handling](https://deepwiki.com/tenstorrent/tt-llk/3.6-block-floating-point-and-mx-format-handling).

## DataFormat Enumeration Overview

The `DataFormat` enum categorizes supported data types into four families: floating-point formats (standard IEEE 754 variants), block floating-point formats (shared exponent compression), integer formats (signed and unsigned), and MX formats (microscaling floating-point). Each format defines the bit representation, memory footprint, and computational characteristics for tensor data.

## Floating-Point Formats

Floating-point formats follow IEEE 754 conventions with explicit sign, exponent, and mantissa fields. These formats provide the widest dynamic range and are the primary data types for most computation.

### Float32

32-bit IEEE 754 single-precision floating-point format with 1 sign bit, 8 exponent bits (bias 127), and 23 mantissa bits. Provides the highest precision but doubles memory footprint compared to 16-bit formats.

*   **Tile Size**: 4096 bytes (1024 elements × 4 bytes)
*   **Dest Capacity**: 4 tiles per half when `is_fp32_dest_acc_en=false`, 2 tiles when `true`
*   **Use Cases**: High-precision accumulation, intermediate results requiring extended range
*   **Pipeline Behavior**: When unpacking to source registers, mantissa truncated to 10 bits (TF32)

### Float16

16-bit IEEE 754 half-precision floating-point format with 1 sign bit, 5 exponent bits (bias 15), and 10 mantissa bits. Provides standard half-precision with symmetric exponent range.

*   **Tile Size**: 2048 bytes (1024 elements × 2 bytes)
*   **Dest Capacity**: 8 tiles per half when `is_fp32_dest_acc_en=false`
*   **Hardware Constraints**: On Blackhole, Float16 requires `dest_acc=DestAccumulation.Yes` when used as input or output format
*   **Conversion to TF32**: Mantissa preserved (10 bits), exponent bias adjusted from 15 to 127

### Float16_b (bfloat16)

16-bit brain floating-point format with 1 sign bit, 8 exponent bits (bias 127), and 7 mantissa bits. Shares exponent range with Float32 while halving memory requirements.

*   **Tile Size**: 2048 bytes (1024 elements × 2 bytes)
*   **Dest Capacity**: 8 tiles per half when `is_fp32_dest_acc_en=false`
*   **Use Cases**: ML training and inference, format compatible with Float32 exponent range
*   **Conversion to TF32**: Mantissa zero-padded to 10 bits (3 LSBs implied 0)

## Block Floating-Point Formats

Block floating-point formats store a shared exponent for groups of 16 mantissa values, reducing memory while maintaining relative precision within blocks. Each 16×1 row of a face shares one exponent.

### BFP Format Variants

Two variant families exist: "A" variants (Bfp8, Bfp4, Bfp2) and "B" variants (Bfp8_b, Bfp4_b, Bfp2_b). The distinction affects unpacker data path selection and exponent handling.

#### Bfp8 and Bfp8_b

8-bit mantissas with shared exponents.

*   **Tile Size**: 1088 bytes (1024 mantissa bytes + 64 exponent bytes)
*   **Exponent Storage**: 64 exponents per tile (4 faces × 16 rows/face)
*   **Precision**: ~7-8 effective bits depending on dynamic range
*   **Special Handling**: Infinity or NaN in a row zeros entire 16-element group

#### Bfp4 and Bfp4_b

4-bit mantissas with shared exponents.

*   **Tile Size**: 576 bytes (512 mantissa bytes + 64 exponent bytes)
*   **Precision**: ~4-5 effective bits
*   **Use Cases**: Extreme compression for weight storage

#### Bfp2 and Bfp2_b

2-bit mantissas with shared exponents.

*   **Tile Size**: 320 bytes (256 mantissa bytes + 64 exponent bytes)
*   **Precision**: ~2-3 effective bits
*   **Use Cases**: Specialized quantization schemes

## Integer Formats

Integer formats store whole numbers without fractional components. Supported for specific operations like indexing, counters, and integer arithmetic.

### Int8 and UInt8

8-bit signed and unsigned integers.

*   **Tile Size**: 1024 bytes
*   **Range**: Int8 [-128, 127], UInt8 [0, 255]
*   **Use Cases**: Quantized inference, indexing operations

### Int16 and UInt16

16-bit signed and unsigned integers.

*   **Tile Size**: 2048 bytes
*   **Range**: Int16 [-32768, 32767], UInt16 [0, 65535]
*   **Use Cases**: Intermediate quantization, address calculations

### Int32 and UInt32

32-bit signed and unsigned integers.

*   **Tile Size**: 4096 bytes
*   **Range**: Int32 [-2³¹, 2³¹-1], UInt32 [0, 2³²-1]
*   **Pipeline Constraints**: 32-bit formats require `unpack_to_dest=true` for lossless tilize operations
*   **Dest Capacity**: When `is_fp32_dest_acc_en=true`, capacity is 2 tiles per half (same as Float32)

## MX Formats (Microscaling Floating-Point)

MX formats implement microscaling floating-point quantization where values are stored as compressed 8-bit representations but unpacked to bfloat16 for computation. The golden model must apply quantization via pack→unpack roundtrip to match hardware behavior.

### MxFp8R (E5M2 Variant)

Microscaling format with 5 exponent bits and 2 mantissa bits, using nearest rounding.

*   **Storage**: Compressed in L1 memory, unpacked to bfloat16 in registers
*   **Quantization**: `pack_mxfp8r()` followed by `unpack_mxfp8r()`
*   **Golden Model**: `quantize_mx_stimuli(tensor, DataFormat.MxFp8R, num_faces)`
*   **Chunk Processing**: 1024 elements (4 faces), 512 elements (2 faces), or 256 elements (1 face)

### MxFp8P (E4M3 Variant)

Microscaling format with 4 exponent bits and 3 mantissa bits, using rounding toward positive.

*   **Storage**: Compressed in L1 memory, unpacked to bfloat16 in registers
*   **Quantization**: `pack_mxfp8p()` followed by `unpack_mxfp8p()`
*   **Golden Model**: `quantize_mx_stimuli(tensor, DataFormat.MxFp8P, num_faces)`
*   **Use Cases**: Higher mantissa precision at cost of reduced exponent range compared to MxFp8R

## Hardware Constraints and Architecture Differences

### Destination Register Capacity

The number of tiles that can be held in destination registers depends on format bit width and `is_fp32_dest_acc_en` setting:

*   **16-bit formats (Float16, Float16_b, Bfp8_b, Int16, UInt16)**:

    *   `is_fp32_dest_acc_en=false`: 8 tiles per half (16 tiles full)
    *   `is_fp32_dest_acc_en=true`: 4 tiles per half (8 tiles full)

*   **32-bit formats (Float32, Int32, UInt32)**:

    *   `is_fp32_dest_acc_en=false`: 4 tiles per half (8 tiles full)
    *   `is_fp32_dest_acc_en=true`: 2 tiles per half (4 tiles full)

### Architecture-Specific Format Support

**Blackhole-Specific Constraints**:

*   Float16 format requires `dest_acc=DestAccumulation.Yes` when used as input or output
*   InputOutputFormat(Float32, Float16) combination not supported when `dest_acc=DestAccumulation.No`
*   Fp8_e4m3 format available only on Blackhole (not on Wormhole)

**Format Variant Selection**:

*   "A" variants (Bfp8, Bfp4, Bfp2, Lf8, Float16) checked via `IS_A_FORMAT()` predicate
*   "B" variants use alternative unpacker data paths
*   Affects `IS_BFP_A_FORMAT()` checks in unpacker configuration

## Format Usage in Pipeline Stages

Each pipeline stage has format-specific configuration requirements:

### Unpacker Format Configuration

*   `unpack_A_src`: Source format in L1 memory
*   `unpack_A_dst`: Destination format in source registers (typically matches `unpack_A_src`)
*   Special case: 32-bit formats use `unpack_to_dest=true` to write directly to destination registers

### Math Format Configuration

*   `math`: Format used for ALU operations (set via `ALU_FORMAT_SPEC_REG`)
*   Fidelity masking applied to mantissa bits based on `MathFidelity` setting
*   TF32 internal representation: 1 sign, 8 exponent, 10 mantissa bits

### Packer Format Configuration

*   `pack_src`: Source format in destination registers
*   `pack_dst`: Destination format in L1 memory
*   Format conversion occurs during pack operation
*   BFP formats require exponential section size configuration per packer
