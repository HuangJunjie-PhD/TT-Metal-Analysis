This page provides a detailed technical overview of the JUnit XML parsing subsystem within the `collect_data` action. These parsers are responsible for transforming standard and augmented JUnit XML reports into structured `Test` and `OpTest` Pydantic models for CI/CD and operation-level telemetry.

## Overview of JUnit Parsers

The system employs three primary parsers for XML-based test reports, all inheriting from the `Parser` base class defined in `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parser.py" undefined  file-path=".github/actions/collect_data/src/parsers/parser.py">Hii</FileRef>`.

| Parser Class | Target Format | Primary Use Case |
| --- | --- | --- |
| `PythonPytestParser` | Standard Pytest JUnit XML | General Python unit and integration tests. |
| `PythonUnittestParser` | Standard Unittest XML | Legacy or standard unittest-style XML reports. |
| `BuilderPytestParser` | Augmented Pytest XML | ML kernel operation tests with hardware/tensor metadata. |

## Shared Utility: `junit_xml_utils`

The `junit_xml_utils.py` module provides a functional interface for interacting with the XML DOM, specifically optimized for Pytest's flavor of JUnit XML.

### Key Functions

*   **`is_pytest_junit_xml(root_element)`**: Identifies if a report was generated by Pytest by checking if the first `testsuite` name is "pytest" `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/junit_xml_utils.py#L31-L37" min=31 max=37 file-path=".github/actions/collect_data/src/parsers/junit_xml_utils.py">Hii</FileRef>`.
*   **`get_pytest_testcase_properties(testcase_element)`**: Flattens the `<properties>` block into a Python dictionary. It uses `toolz.dicttoolz.merge` to reduce property elements into a single map `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/junit_xml_utils.py#L52-L62" min=52 max=62 file-path=".github/actions/collect_data/src/parsers/junit_xml_utils.py">Hii</FileRef>`.
*   **Status Helpers**: Functions like `get_pytest_testcase_is_skipped`, `_is_failed`, and `_is_error` check for the existence of specific child tags within a `<testcase>``<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/junit_xml_utils.py#L69-L78" min=69 max=78 file-path=".github/actions/collect_data/src/parsers/junit_xml_utils.py">Hii</FileRef>`.

## PythonPytestParser

This parser handles standard Pytest XML output. It is selected if `can_parse` detects the "pytest" name in the XML root `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_pytest_parser.py#L22-L28" min=22 max=28 file-path=".github/actions/collect_data/src/parsers/python_pytest_parser.py">Hii</FileRef>`.

### Implementation Details

1.   **Timestamp Reconstruction**: Pytest XML often lacks per-test start times. The parser uses the `testsuite` start timestamp as a base. If the testcase has a `start_timestamp` property, it uses that; otherwise, it calculates the end time by adding the `time` attribute (duration) to the base `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_pytest_parser.py#L84-L91" min=84 max=91 file-path=".github/actions/collect_data/src/parsers/python_pytest_parser.py">Hii</FileRef>`.
2.   **Category Classification**: Tests are categorized based on their `classname`. It searches for keywords like `models`, `ttnn`, `tt_eager`, or `tt_metal``<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_pytest_parser.py#L98-L106" min=98 max=106 file-path=".github/actions/collect_data/src/parsers/python_pytest_parser.py">Hii</FileRef>`.
3.   **Property Extraction**: It attempts to extract `tags` and `config` from XML properties, using `ast.literal_eval` to safely parse string-encoded Python literals `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_pytest_parser.py#L120-L132" min=120 max=132 file-path=".github/actions/collect_data/src/parsers/python_pytest_parser.py">Hii</FileRef>`.

## PythonUnittestParser

Used for XML files that do not match the Pytest signature. It uses `xmltodict` for parsing instead of `ElementTree``<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_unittest_parser.py#L4-L33" min=4 max=33 file-path=".github/actions/collect_data/src/parsers/python_unittest_parser.py">Hii</FileRef>`.

### Key Behaviors

*   **Sequential Timing**: Since standard unittest XML lacks individual timestamps, the parser maintains a `previous_test_end_ts` and increments it by the test duration for each subsequent test to ensure unique start times, which is a requirement for the downstream data warehouse `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_unittest_parser.py#L80-L83" min=80 max=83 file-path=".github/actions/collect_data/src/parsers/python_unittest_parser.py">Hii</FileRef>`.
*   **Status Mapping**: Maps `<skipped>`, `<error>`, and `<failure>` tags to the `Test` model's `success` and `skipped` boolean fields `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_unittest_parser.py#L65-L78" min=65 max=78 file-path=".github/actions/collect_data/src/parsers/python_unittest_parser.py">Hii</FileRef>`.

## BuilderPytestParser (OpTest XML)

The `BuilderPytestParser` is a specialized parser for "OpTests" — tests that validate specific ML kernel operations and include rich metadata about tensors and hardware.

### Detection Logic

The parser identifies compatible files by checking for a `card` property in the XML `<properties>` block, which indicates a report from the hardware-aware builder suite `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L37-L44" min=37 max=44 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.

### Metadata Extraction

Unlike standard parsers, this parser populates the `OpTest` model with:

*   **Hardware Context**: Extracts `card` (e.g., N300, N150) and `git_sha` from the suite properties `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L65-L66" min=65 max=66 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.
*   **Failure Stages**: Maps the `failure_stage` property to a `TestStatus` enum (e.g., `compile_failed`, `golden_failed`, `run_failed`) `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L17-L22" min=17 max=22 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.
*   **Tensor Descriptions**: Parses `input_shapes` and `input_dtypes` to create `TensorDesc` objects `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L163-L168" min=163 max=168 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.

## Validation and Error Handling

All parsers utilize a shared failure reporting mechanism. If a `ValidationError` occurs during the instantiation of a Pydantic `Test` or `OpTest` model, the parser calls `failure_happened()` from the `shared` module `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/shared.py#L10-L11" min=10 max=11 file-path=".github/actions/collect_data/src/shared.py">Hii</FileRef>`. This sets a global flag that the `collect_data` action uses to signal a partial failure in data collection `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_pytest_parser.py#L150-L153" min=150 max=153 file-path=".github/actions/collect_data/src/parsers/python_pytest_parser.py">Hii</FileRef>`.

Additionally, `is_valid_testcase_` is used to filter out malformed XML entries, such as those caused by GitHub Action timeouts where a `<testcase>` tag might be present but empty of attributes `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/shared.py#L19-L32" min=19 max=32 file-path=".github/actions/collect_data/src/shared.py">Hii</FileRef>`.
