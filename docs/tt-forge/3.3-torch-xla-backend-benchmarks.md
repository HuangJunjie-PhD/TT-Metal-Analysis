## Purpose and Scope

This document provides an overview of the torch-xla backend benchmark implementations in the TT-Forge repository. These benchmarks use PyTorch with the torch-xla library to compile and execute models on Tenstorrent hardware via the PJRT (Plugin-based Just-in-Time Runtime) interface. The benchmarks validate the TT-XLA frontend → StableHLO → TT-MLIR compilation path and measure performance characteristics of various neural network architectures.

The torch-xla benchmarks are organized into three model families, each documented in detail on child pages:

*   [CNN and Vision Model Benchmarks](https://deepwiki.com/tenstorrent/tt-forge/3.3.1-cnn-and-vision-model-benchmarks) — Document ResNet, EfficientNet, VoVNet, and MobileNetV2 benchmarking implementations using torch-xla
*   [Segmentation and Transformer Benchmarks](https://deepwiki.com/tenstorrent/tt-forge/3.3.2-segmentation-and-transformer-benchmarks) — Document UNet, Segformer, and Vision Transformer (ViT) benchmarking implementations using torch-xla
*   [LLM Benchmarks](https://deepwiki.com/tenstorrent/tt-forge/3.3.3-llm-benchmarks) — Document Large Language Model benchmarking infrastructure, including text generation, static cache management, and performance measurement

For information about the benchmark infrastructure and execution workflows, see [Benchmark Infrastructure and Workflows](https://deepwiki.com/tenstorrent/tt-forge/3.1-benchmark-infrastructure-and-workflows) and [Test Matrix Configuration](https://deepwiki.com/tenstorrent/tt-forge/3.2-test-matrix-configuration).

## torch-xla Backend Architecture

torch-xla benchmarks use PyTorch with the torch-xla library to compile and execute neural networks on Tenstorrent hardware. All benchmarks follow a standardized pattern that leverages the PJRT interface and StableHLO intermediate representation. The torch-xla backend is part of the TT-XLA frontend layer, which provides multi-chip support and JAX/PyTorch compatibility.

## Framework Specifics

### PyTorch (torch-xla)

PyTorch models are integrated using the `tt` backend for `torch.compile`. The runtime is explicitly set to Tenstorrent hardware using `xr.set_device_type("TT")`[demos/tt-xla/cnn/resnet_demo.py 26](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/cnn/resnet_demo.py#L26-L26) Tensors and models are moved to the device using `xm.xla_device()`[demos/tt-xla/nlp/pytorch/bge3_demo.py 32-33](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/nlp/pytorch/bge3_demo.py#L32-L33)

### JAX

JAX models (such as GPT-2) utilize the PJRT interface to target Tenstorrent devices. Compilation is handled via `jax.jit`[demos/tt-xla/nlp/jax/gpt_demo.py 39-41](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/nlp/jax/gpt_demo.py#L39-L41) Device management uses `jax.devices()` and `jax.sharding.Mesh` to define the execution environment [demos/tt-xla/nlp/jax/gpt_demo.py 27-28](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/nlp/jax/gpt_demo.py#L27-L28)

## Performance Measurement and Validation

TT-XLA benchmarks measure performance and validate correctness through several mechanisms:

1.   **Warmup Phase**: Initial iterations to trigger JIT compilation and hardware state initialization.
2.   **FPS Measurement**: Calculating throughput (frames per second) over a defined loop count.
3.   **PCC Validation**: Comparing Tenstorrent device outputs against a CPU-based golden reference using Pearson Correlation Coefficient.
4.   **Post-processing**: Model-specific validation, such as `loader.output_postprocess()` for vision models [demos/tt-xla/cnn/resnet_demo.py 53](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/cnn/resnet_demo.py#L53-L53) or token rank analysis for LLMs [demos/tt-xla/nlp/jax/gpt_demo.py 73-81](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/demos/tt-xla/nlp/jax/gpt_demo.py#L73-L81)
