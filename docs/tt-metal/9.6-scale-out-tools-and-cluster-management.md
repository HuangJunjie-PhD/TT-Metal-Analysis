This page documents the suite of tools and utilities used to manage, validate, and describe large-scale Tenstorrent clusters. These tools bridge the gap between physical hardware cabling and the logical fabric used by the TT-Metalium runtime.

## Overview of Scale-Out Infrastructure

The scale-out infrastructure consists of descriptors that define the physical system, tools to generate these descriptors from cabling specifications, and validation utilities to ensure the live hardware matches the intended topology.

### Key Components

| Component | Description | Primary Code Entities |
| --- | --- | --- |

| **Cabling Generator** | Generates system descriptors and cabling guides from high-level cluster and deployment configurations. | `CablingGenerator`[[tools/scaleout/cabling_generator/cabling_generator.cpp 141-154](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.cpp#L141-L154)], `Node` classes [[tools/scaleout/node/node.cpp 70-131](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/node/node.cpp#L70-L131)] |

## Physical System Descriptor (PSD)

The `PhysicalSystemDescriptor` is the central data structure for cluster topology. It maintains a graph of ASICs and hosts, their Ethernet interconnections, and PCIe locations.

### Implementation and Data Flow

The PSD can be initialized through several pathways:

*   By hardware discovery, assembling the current cluster topology using live fabric queries [[tt_metal/fabric/physical_system_discovery.cpp 30-135](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_discovery.cpp#L30-L135)].
*   By deserializing from a `.textproto` file for simulation or testing purposes [[tt_metal/fabric/physical_system_descriptor.cpp 42-58](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L42-L58)].

All data members of the PSD are directly moved during deserialization or merging, avoiding costly copy operations [[tt_metal/fabric/physical_system_descriptor.cpp 46-57](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L46-L57)]. Support for merging multiple PSDs allows building a unified view of multi-host clusters, which is crucial for distributed system initialization [[tt_metal/fabric/physical_system_descriptor.cpp 102-143](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L102-L143)]. The `clear()` method resets all topology data for a fresh discovery [[tt_metal/fabric/physical_system_descriptor.cpp 86-100](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L86-L100)].

### System Graph and Connectivity

The PSD holds a `system_graph_` that nests two main graphs:

*   `asic_connectivity_graph`: Mapping of local ASIC-to-ASIC connections within each host [[tt_metal/fabric/physical_system_descriptor.cpp 103-105](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L103-L105)].
*   `host_connectivity_graph`: Inter-host Ethernet connection mappings [[tt_metal/fabric/physical_system_descriptor.cpp 106-108](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L106-L108)].

Additional metadata maps hostnames to motherboard names, ranks, and PCIe device layouts by tray [[tt_metal/fabric/physical_system_descriptor.cpp 112-126](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/physical_system_descriptor.cpp#L112-L126)].

## Cabling Generator and Deployment Tools

The `CablingGenerator` toolchain translates high-level cluster and deployment specifications into concrete cabling guides, factory system descriptors, and graph models for use by the runtime and validation tools.

### Input Descriptors

*   **Cluster Descriptor**: Logical description of node topology and inter-node connections as a protobuf `ClusterDescriptor`[[tools/scaleout/cabling_generator/cabling_generator.cpp 77-79](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.cpp#L77-L79)
*   **Deployment Descriptor**: Maps logical host IDs to physical hostnames, rack, aisle, and shelf U information, as protobuf `DeploymentDescriptor`[[tools/scaleout/cabling_generator/cabling_generator.cpp 81-83](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.cpp#L81-L83)].

### Node Templates

`Node` classes represent individual compute nodes with a set of boards and internal cabling connections. Common node classes include `N300T3KNode` for Wormhole architectures and `WHGalaxyNode` for Galaxy-scale systems [[tools/scaleout/node/node.cpp 70-131](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/node/node.cpp#L70-L131)].

Node templates specify motherboard types, board count and types, and port-level inter-board connections like WARP100 or QSFP_DD ports [[tools/scaleout/node/node.cpp 23-96](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/node/node.cpp#L23-L96)].

### Implementation Features

*   **Port Management:** Ports on boards are marked used once wired for inter-board connections to prevent conflicts during generation [[tools/scaleout/cabling_generator/cabling_generator.cpp 57-73](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.cpp#L57-L73)].
*   **Merge & Validation:** Cluster descriptors can come from multiple cabling files and are merged carefully with conflict detection, ensuring consistent motherboard, board count, and connection definitions [[tools/scaleout/cabling_generator/cabling_generator.cpp 125-163](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.cpp#L125-L163)].
*   **Factory System Descriptor (FSD) Generation:**`CablingGenerator` can output a unified factory system descriptor protobuf object and export to YAML/CSV formats for documentation and runtime input [[tools/scaleout/cabling_generator/cabling_generator.hpp 177-180](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/cabling_generator/cabling_generator.hpp#L177-L180)].

## Cluster Validation Utilities

The `run_cluster_validation` is a critical tool that verifies cluster health, cabling consistency, and link integrity across multi-node Tenstorrent clusters.

### Validation Modes

*   **Connectivity Validation:** Compares the live discovered topology with the provided cabling and deployment descriptors to ensure hardware cabling matches specifications [[tools/scaleout/validation/run_cluster_validation.cpp 159-161](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/run_cluster_validation.cpp#L159-L161)].
*   **Traffic Stress Test:** Sends test packets over links to verify bandwidth, latency, and retransmit behavior. Configured using packet sizes, data size, and iteration counts [[tools/scaleout/validation/run_cluster_validation.cpp 101-102](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/run_cluster_validation.cpp#L101-L102)].
*   **Link Metrics Logging:** Collects and logs live Ethernet link metrics including errors, bandwidth, and retrain counts [[tools/scaleout/validation/run_cluster_validation.cpp 97-98](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/run_cluster_validation.cpp#L97-L98)].
*   **Link Reset:** Supports restarting specific links on a target host/tray/channel via a dedicated subcommand interface [[tools/scaleout/validation/run_cluster_validation.cpp 117-126](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/run_cluster_validation.cpp#L117-L126)].

### Implementation Details

Traffic injection kernels are deployed locally per ASIC using ERISC cores. Kernels use a reserved L1 memory address range for source and destination buffers [[tools/scaleout/validation/utils/cluster_validation_utils.cpp 181-182](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/utils/cluster_validation_utils.cpp#L181-L182)]. For Blackhole architectures, the tool selects the appropriate Network on Chip (NoC) and ERISC core based on configuration for transmitting test packets [[tools/scaleout/validation/utils/cluster_validation_utils.cpp 189-190](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/utils/cluster_validation_utils.cpp#L189-L190)].

Batch execution of concurrent traffic tests and metrics collection is carefully handled using async futures and parallel batch execution utilities [[tools/scaleout/validation/utils/cluster_validation_utils.cpp 108-156](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tools/scaleout/validation/utils/cluster_validation_utils.cpp#L108-L156)].

## Mesh Graph and Routing Configuration

Scale-out systems use `MeshGraph` and accompanying descriptors (MGD files) to define logical fabric topologies that map onto physical hardware.

### Key Concepts

*   **MeshGraphDescriptor (MGD):** A protobuf-based representation defining node and fabric connection topology used for routing and fabric initialization [[tt_metal/fabric/mesh_graph.cpp 112-118](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/mesh_graph.cpp#L112-L118)].
*   **TopologyMapper:** Binds logical fabric nodes defined in MGD to physical devices by using the physical system descriptor and runtime cluster information [[tt_metal/fabric/fabric_host_utils.cpp 157-160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/fabric_host_utils.cpp#L157-L160)] .
*   **Fixed ASIC Position Pinning:** For Galaxy systems, a mapping pins key ASICs at mesh corners for optimal QSFP alignment and minimal cable complexity [[tt_metal/fabric/control_plane.cpp 85-117](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/control_plane.cpp#L85-L117)].
*   **Fabric Types:** The topology mapper classifies fabric types, e.g., `FABRIC_1D_RING`, `FABRIC_2D_TORUS_XY`, or `FABRIC_1D_NEIGHBOR_EXCHANGE` based on configuration and hardware type [[tt_metal/fabric/fabric_host_utils.cpp 36-53](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/fabric/fabric_host_utils.cpp#L36-L53)].

## Build and Testing

The scale-out tools are tightly integrated into the build system, with specific executables and extensive tests for validating descriptor generation, merging, and cluster validation logic.

### Build Targets

# Summary

The scale-out and cluster management tools provide comprehensive support for defining, validating, and managing large multi-host Tenstorrent clusters. They enable abstract description of the complex physical cabling, provide robust discovery and validation tooling, and integrate tightly with runtime fabric and device drivers for consistent and optimized cluster operation.
