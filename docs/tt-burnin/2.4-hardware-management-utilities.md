# Hardware Management Utilities

Relevant source files
*   [tt_burnin/utils.py](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py)

This document covers the hardware management utility functions provided by the `tt_burnin.utils` module. These utilities handle device reset operations, hardware detection, telemetry collection, and board identification for Tenstorrent AI hardware. For information about the core hardware abstraction layer and chip-specific implementations, see [Hardware Abstraction Layer](https://deepwiki.com/tenstorrent/tt-burnin/2.2-hardware-abstraction-layer). For details about the main application workflow, see [Main Application Logic](https://deepwiki.com/tenstorrent/tt-burnin/2.1-main-application-logic).

## Device Reset Management

The reset management system provides centralized control for resetting different types of Tenstorrent chips through a unified interface that delegates to chip-specific reset implementations.

### Reset Workflow Architecture

Sources: [tt_burnin/utils.py 30-85](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L30-L85)

The `pci_board_reset` function [tt_burnin/utils.py 30-85](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L30-L85) serves as the primary entry point for device reset operations. It accepts a list of PCI indices and an optional reinit flag, then:

1.   Creates `PciChip` instances for each device
2.   Categorizes devices by chip type using `as_wh()`, `as_gs()`, and `as_bh()` methods
3.   Delegates to appropriate reset implementations from `tt_tools_common.reset_common`
4.   Optionally re-initializes devices after reset using `detect_chips_with_callback()`

### Reset Configuration Management

The system supports both direct PCI index lists and JSON configuration files for complex reset scenarios:

| Function | Purpose | Input Format |
| --- | --- | --- |
| `parse_reset_input` | Validates and parses reset inputs | JSON file path or comma-separated integers |
| `pci_indices_from_json` | Extracts PCI indices from JSON config | JSON dictionary with chip-specific sections |
| `mobo_reset_from_json` | Handles motherboard-level reset operations | JSON with `wh_mobo_reset` configuration |

Sources: [tt_burnin/utils.py 87-155](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L87-L155)

The JSON configuration format supports different reset types:

*   `gs_tensix_reset`: Grayskull Tensix core reset
*   `wh_link_reset`: Wormhole link-level reset
*   `wh_mobo_reset`: Wormhole motherboard reset
*   `re_init_devices`: Flag for device re-initialization

## Device Detection and Listing

### Device Discovery and Display

Sources: [tt_burnin/utils.py 157-184](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L157-L184)[tt_burnin/utils.py 186-233](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L186-L233)

The `print_all_available_devices` function [tt_burnin/utils.py 157-184](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L157-L184) creates a comprehensive display table showing:

*   **PCI Dev ID**: Hardware interface identifier or "N/A" for remote devices
*   **Device Series**: Architecture type (Grayskull, Wormhole, Blackhole)
*   **Board Type**: Specific board model determined by UPI parsing
*   **Board Number**: Hexadecimal board identifier
*   **Coordinates**: Device grid coordinates

### Board Type Identification

The `get_board_type` function [tt_burnin/utils.py 186-233](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L186-L233) implements a lookup table based on the Unique Part Identifier (UPI) extracted from the board ID:

| UPI Value | Board Type | Architecture |
| --- | --- | --- |
| 0x3, 0xA, 0x7 | e150, e300, e75 | Grayskull |
| 0x8, 0xB, 0x14, 0x18, 0x35 | nb_cb, wh_4u, n300, n150, tt-galaxy-wh | Wormhole |
| 0x36, 0x43, 0x40, 0x41 | bh-scrappy, p100a, p150a, p150b | Blackhole |

Sources: [tt_burnin/utils.py 186-233](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L186-L233)

## Telemetry Collection and Display

### Telemetry Data Processing

Sources: [tt_burnin/utils.py 297-330](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L297-L330)[tt_burnin/utils.py 241-247](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L241-L247)[tt_burnin/utils.py 235-239](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L235-L239)

The telemetry system extracts and displays real-time hardware metrics:

| Metric | Extraction Method | Display Format |
| --- | --- | --- |
| Core Voltage | `telem["vcore"] / 1000` | Volts with 2 decimal places |
| Core Current | `telem["tdc"] & 0xFFFF` | Amperes with 1 decimal place |
| AICLK | `telem["aiclk"] & 0xFFFF` | MHz as integer |
| Power | `telem["tdp"] & 0xFFFF` | Watts with 1 decimal place |
| Temperature | `asic_temperature_parser()` | Celsius with 1 decimal place |

The `prefix_color_picker` function [tt_burnin/utils.py 235-239](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L235-L239) applies color coding based on utilization thresholds:

*   Green: Values below 85% of maximum
*   Orange: Values at or above 85% of maximum

Sources: [tt_burnin/utils.py 235-330](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L235-L330)

## Specialized Reset Operations

### Galaxy System Reset

The `reset_6u_glx` function [tt_burnin/utils.py 261-295](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L261-L295) provides specialized reset capabilities for Wormhole Galaxy systems:

Sources: [tt_burnin/utils.py 261-295](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L261-L295)[tt_burnin/utils.py 249-259](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/utils.py#L249-L259)

The Galaxy reset sequence includes:

1.   IPMI-based reset command with specific parameters
2.   30-second timed wait with progress indication via `timed_wait()`
3.   Driver reinitialization wait
4.   Chip detection with fallible detection mode
5.   Success confirmation with detected device count

### Utility Helper Functions

| Function | Purpose | Key Parameters |
| --- | --- | --- |
| `timed_wait` | Progress-indicated delay | `seconds`: Wait duration with console updates |
| `asic_temperature_parser` | Architecture-specific temperature parsing | `temp`: Raw value, `dev`: Device for architecture check |
| `prefix_color_picker` | Threshold-based color selection | `current_value`, `max_value`: For utilization calculation |
