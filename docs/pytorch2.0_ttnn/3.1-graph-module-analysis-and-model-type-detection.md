## Purpose and Scope

This document describes the **GraphModuleAnalysisPass** system, which classifies compiled FX graphs as either inference, training forward, or training backward passes. This classification enables downstream compilation passes to apply mode-specific optimizations and transformations. The pass analyzes structural patterns in the FX graph to determine the computation type, storing the result in the graph module's metadata for use by subsequent passes.

For information about other analysis passes such as input classification, see [Input Analysis and Classification](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3.3-input-analysis-and-classification). For optimization passes that consume this analysis, see [Additional Optimization Passes](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3.5-memory-management-and-optimization-passes).

## ModelType Enumeration

The system defines three distinct graph execution modes through the `ModelType` enumeration:

| ModelType | Value | Description | Use Case |
| --- | --- | --- | --- |
| `INFERENCE` | 1 | Forward pass for inference only | Model evaluation, prediction |
| `TRAIN_FORWARD` | 2 | Forward pass during training | Forward propagation with gradient tracking |
| `TRAIN_BACKWARD` | 3 | Backward pass during training | Gradient computation and backpropagation |

The `ModelType` enum is defined in [torch_ttnn/passes/analysis/graph_module_analysis_pass.py 9-19](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/graph_module_analysis_pass.py#L9-L19) and serves as the canonical type system for differentiating computation modes throughout the compilation pipeline.

### Training Forward Pass Detection

The `is_train_forward()` function identifies training forward passes by checking if any output node directly returns an input placeholder unchanged. This pattern occurs because PyTorch's autograd system requires access to the original inputs for gradient computation:

| Graph Pattern | Interpretation |
| --- | --- |
| Output node returns placeholder(s) | Training forward - inputs preserved for backward pass |
| Output node returns only computed values | Inference - inputs not needed after forward |

The detection algorithm:

1.   Finds all nodes with `op == "output"`
2.   Examines the arguments of each output node
3.   Checks if any argument is a placeholder node (input)
4.   Returns `True` if placeholders are found in outputs

**Implementation:**[torch_ttnn/passes/analysis/graph_module_analysis_pass.py 31-40](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/graph_module_analysis_pass.py#L31-L40)

### Inference Detection

Inference mode is detected by exclusion: if the graph is neither a backward pass nor a training forward pass, it must be an inference graph. This includes both `eval()` mode and `torch.no_grad()` contexts.

## Usage in Compilation Pipeline

The `GraphModuleAnalysisPass` executes early in the compilation pipeline, providing critical mode information to downstream passes.

### Metadata Storage and Access

The detected model type is stored in the graph module's metadata dictionary:

Downstream passes access this metadata to make mode-specific decisions:

| Accessing Pass | Decision Based on ModelType |
| --- | --- |
| Optimization passes | Apply aggressive optimizations only in `INFERENCE` mode |
| Memory passes | Handle gradient buffers differently in training modes |
| Multi-device passes | Different sharding strategies for training vs inference |

### Pass Pipeline Position

The pass runs before any graph transformations, ensuring that the original graph structure (containing training indicators like placeholder returns or backward ops) is analyzed before modifications occur.

**Sources:**[torch_ttnn/passes/analysis/graph_module_analysis_pass.py 43-67](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/graph_module_analysis_pass.py#L43-L67)

## Limitations and Design Considerations

The detection heuristics have known limitations documented in the implementation:

### Current Assumptions

1.   **Training forward detection:** Assumes training graphs return inputs unchanged for gradient computation. This pattern is reliable for PyTorch 2.0's `torch.compile` but may fail for custom training loops with modified output structure.

2.   **Backward detection:** Relies on function names containing "backward". This works for standard autograd operations but may miss custom backward implementations.

3.   **Timing dependency:** The pass documentation notes that `torch.is_grad_enabled()` and `gm.training` flags are unreliable by the time the backend receives the graph module, necessitating structural analysis instead.

### Implementation Comment

The code includes this note at [torch_ttnn/passes/analysis/graph_module_analysis_pass.py 50-51](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/graph_module_analysis_pass.py#L50-L51):

> "It would be cleaner if we could just use something like `torch.is_grad_enabled()` or `gm.training` instead, but these appear to be inaccurate by the time the GraphModule is passed to our backend."

This explains why structural heuristics are necessary rather than direct mode queries.

**Sources:**[torch_ttnn/passes/analysis/graph_module_analysis_pass.py 48-55](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/graph_module_analysis_pass.py#L48-L55)

For a simple inference call like `model(input)` in eval mode, the graph contains only forward operations with no special output structure:

    %output : [num_users=0] = output(%relu)
```

Detection: No "backward" in any target, no placeholder in output → `INFERENCE`

### Training Forward Graph Pattern

For a training forward pass, the graph preserves inputs in the output:

```

    %input : [num_users=2] = placeholder[target=input]

Detection: No "backward" in any target, placeholder `%input` in output → `TRAIN_FORWARD`
