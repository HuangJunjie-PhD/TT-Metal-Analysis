This document describes how the Tenstorrent kernel driver discovers and manages hardware devices on the PCI bus. It covers the complete device lifecycle: discovery, probe initialization, power management, and removal. This includes handling hotplug events where devices may be added or removed from the system at runtime.

The driver implements a standard Linux PCI driver that registers with the kernel PCI subsystem to receive notifications when compatible Tenstorrent devices are discovered. Each device is assigned a unique ordinal and registered as a character device at `/dev/tenstorrent/N`.

For device-specific initialization details, see pages 4.1 (Wormhole) and 4.2 (Blackhole). For the device abstraction layer that enables polymorphic device support, see page 3.2.

## PCI Driver Registration

The driver registers with the Linux PCI subsystem using a `pci_driver` structure that defines supported device IDs and callback functions for device lifecycle events.

**PCI Driver Structure**

The `tenstorrent_pci_driver` structure defined in [enumerate.c 380-388](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L380-L388) provides the following callbacks:

| Callback | Function | Purpose |
| --- | --- | --- |
| `probe` | `tenstorrent_pci_probe()` | Initialize newly discovered devices |
| `remove` | `tenstorrent_pci_remove()` | Clean up device being removed |
| `shutdown` | `tenstorrent_pci_remove()` | Handle system shutdown |
| `driver.pm` | `tenstorrent_pm_ops` | Suspend/resume operations |

The driver is registered during module initialization via `tenstorrent_pci_register_driver()`[module.c 75](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L75-L75) and unregistered during module cleanup [module.c 95](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L95-L95)

**Supported Device IDs**

The driver defines a PCI device ID table in [module.c 45-53](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L45-L53) that matches specific vendor/device ID combinations:

| Device | Vendor ID | Device ID | Device Class |
| --- | --- | --- | --- |
| Grayskull | `PCI_VENDOR_ID_TENSTORRENT` | `PCI_DEVICE_ID_GRAYSKULL` | Deprecated |
| Wormhole | `PCI_VENDOR_ID_TENSTORRENT` | `PCI_DEVICE_ID_WORMHOLE` | `wormhole_class` |
| Blackhole | `PCI_VENDOR_ID_TENSTORRENT` | `PCI_DEVICE_ID_BLACKHOLE` | `blackhole_class` |

Each entry contains a `driver_data` field that points to the device-specific implementation class. The `MODULE_DEVICE_TABLE` macro [module.c 55](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L55-L55) exports this table to enable automatic module loading when matching hardware is detected.

## Device Discovery and Ordinal Assignment

When the PCI subsystem detects a matching device, it invokes `tenstorrent_pci_probe()` to initialize it. The driver uses an XArray data structure to assign each device a unique ordinal number.

**Device Discovery Flow**

**Ordinal Assignment with XArray**

The driver maintains a global XArray structure `tenstorrent_dev_xa`[enumerate.c 33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L33-L33) to manage device ordinals:

`static DEFINE_XARRAY_ALLOC(tenstorrent_dev_xa);`
Device ordinal allocation occurs in [enumerate.c 229-234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L229-L234):

`err = xa_alloc(&tenstorrent_dev_xa, &ordinal, tt_dev, xa_limit_31b, GFP_KERNEL);if (err) {    kfree(tt_dev);    pci_disable_device(dev);    return err;}`
The XArray provides internal locking and thread-safe ordinal allocation. The `xa_limit_31b` constraint limits ordinals to 31-bit values, matching the character device minor number space.

**Galaxy System Ordinal Mapping**

For Galaxy systems (large-scale topologies with 32 chips across 4 boards), the driver implements a specific BDF-to-ordinal mapping in `galaxy_bdf_to_ordinal()`[enumerate.c 48-76](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L48-L76) It uses the high nibble of the PCI bus number to identify the board (UBB) and the low nibble for the chip index [enumerate.c 35-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L35-L38)

| System Type | Subsystem ID | Mapping Table |
| --- | --- | --- |
| Wormhole Galaxy | `PCI_SUBSYSTEM_ID_GALAXY_WH` (0x0035) | `wh_galaxy_ubb_bus_prefix`[enumerate.c 45](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L45-L45) |
| Blackhole Galaxy | `PCI_SUBSYSTEM_ID_GALAXY_BH` (0x0047) | `bh_galaxy_ubb_bus_prefix`[enumerate.c 46](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L46-L46) |

## Device Probe Lifecycle

Once a device is discovered and enabled, `tenstorrent_pci_probe()` executes a structured initialization sequence that configures PCI features and delegates device-specific initialization to the device class.

**Probe Sequence Diagram**

**Initialization Steps**

The probe function [enumerate.c 198-293](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L198-L293) performs the following operations:

| Step | Code Location | Operation |
| --- | --- | --- |
| Validate device class | [enumerate.c 205-208](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L205-L208) | Check `id->driver_data` is not NULL, return `-ENODEV` if unsupported |
| Fix PCI class code | [enumerate.c 217-220](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L217-L220) | Assign class `0x120000` if `PCI_CLASS_NOT_DEFINED` |
| Enable PCI device | [enumerate.c 222-223](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L222-L223) | `pci_enable_device()`, return `-EIO` on failure |
| Allocate device structure | [enumerate.c 225-227](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L225-L227) | `kzalloc(device_class->instance_size, GFP_KERNEL)` |
| Assign ordinal | [enumerate.c 229-234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L229-L234) | `xa_alloc(&tenstorrent_dev_xa, &ordinal, tt_dev, xa_limit_31b)` |
| Initialize kref | [enumerate.c 237](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L237-L237) | `kref_init(&tt_dev->kref)` - persists until remove |
| Configure DMA masks | [enumerate.c 259-260](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L259-L260) | `dma_set_mask(64-bit)`, `dma_set_coherent_mask()` |
| Enable interrupts | [enumerate.c 272](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L272-L272) | `tenstorrent_enable_interrupts()`[interrupt.c 21-37](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/interrupt.c#L21-L37) |
| Initialize device | [enumerate.c 274-275](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L274-L275) | `dev_class->init_device()` - map BARs |
| Initialize hardware | [enumerate.c 275](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L275-L275) | `dev_class->init_hardware()` - returns `needs_hw_init` flag |
| Register chardev | [enumerate.c 280](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L280-L280) | `tenstorrent_register_device()`[chardev.h 13](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.h#L13-L13) creates `/dev/tenstorrent/N` |

**DMA Configuration**

The probe function configures dual DMA masks [enumerate.c 259-264](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L259-L264) to support both 64-bit streaming DMA and configurable coherent DMA. The `dma_address_bits` module parameter [module.c 33-35](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L33-L35) overrides the device class default. Setting it to 32 allows legacy software while 64-bit masks enable IOMMU support.

**Interrupt Setup**

The driver allocates a single interrupt vector using `pci_alloc_irq_vectors()`[interrupt.c 23](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/interrupt.c#L23-L23) and requests a shared IRQ handler `irq_handler()`[interrupt.c 13-19](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/interrupt.c#L13-L19)

## PCIe Link Management and Thunderbolt Handling

Tenstorrent devices, particularly Wormhole, require specialized PCIe link training and reset handling to ensure stable high-speed operation (Gen4/Gen5).

**PCIe Link Training**

The `wormhole_complete_pcie_init()` function [pcie.c 92-131](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L92-L131) manages link retraining. It sends a `FW_MSG_PCIE_RETRAIN` (0xB6) message to the ARC firmware [pcie.c 16-112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L16-L112) If training fails, the driver performs a PCIe hot reset and retries up to `reset_limit` times [pcie.c 101-128](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L101-L128)

**PCIe Hot Reset**

`pcie_hot_reset_and_restore_state()`[pcie.c 61-90](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L61-L90) implements a secondary bus reset via the upstream bridge:

1.   Calls `pci_ignore_hotplug()`[pcie.c 70](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L70-L70) to prevent the kernel from removing the device during reset.
2.   Sets `PCI_BRIDGE_CTL_BUS_RESET` in the bridge's `PCI_BRIDGE_CONTROL` register [pcie.c 74](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L74-L74)
3.   Clears the bit after 2ms [pcie.c 77](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L77-L77)
4.   Polls for link up using `poll_pcie_link_up()`[pcie.c 24-41](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L24-L41)
5.   Restores PCI configuration space using `safe_pci_restore_state()`[pcie.c 43-59](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L43-L59)

**Thunderbolt and External Enclosures**

Handling external PCIe (Thunderbolt/eGPU) requires careful management of hotplug events. The driver uses `pci_ignore_hotplug()`[pcie.c 70](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L70-L70) during resets to ensure the device node remains stable even if the link momentarily drops.

Sources: [pcie.c 16-131](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L16-L131)[pcie.h 10-15](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.h#L10-L15)

## Power Management

The driver implements Linux device power management through suspend and resume callbacks that put devices into low-power states and restore them to operational state.

**Power Management Callbacks**

The driver registers power management operations via `tenstorrent_pm_ops`[enumerate.c 377](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L377-L377) using the `SIMPLE_DEV_PM_OPS` macro:

| Callback | Function | Purpose |
| --- | --- | --- |
| `suspend` | `tenstorrent_suspend()` | Transition device to low-power state |
| `resume` | `tenstorrent_resume()` | Restore device to operational state |

**Suspend and Resume Flow**

1.   **Suspend**[enumerate.c 355-362](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L355-L362): Calls `tt_dev->dev_class->cleanup_hardware(tt_dev)`. This typically instructs the ARC firmware to enter a low-power state.
2.   **Resume**[enumerate.c 364-375](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L364-L375): Calls `tt_dev->dev_class->init_hardware(tt_dev)`. After successful hardware re-initialization, it calls `pci_save_state(pdev)`[enumerate.c 372](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L372-L372) to refresh the saved PCI configuration.

Sources: [enumerate.c 355-375](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L355-L375)[enumerate.c 377](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L377-L377)

## Device Removal and Cleanup

Device removal handles both graceful shutdown and hotplug scenarios where the device may no longer be accessible.

**Hotplug Detection**

The driver distinguishes between normal removal and hotplug events [enumerate.c 325-327](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L325-L327) by reading the vendor ID. If the read returns `0xFFFF`, the device is no longer accessible, and hardware cleanup is skipped to prevent system hangs.

**Cleanup Sequence**

The `tenstorrent_pci_remove()` function [enumerate.c 323-335](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L323-L335) ensures proper resource deallocation:

1.   **Hardware Cleanup**: Skips if device is detached [enumerate.c 327](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L327-L327)
2.   **Unregistration**: Calls `tenstorrent_unregister_device()`[enumerate.c 331](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L331-L331) to remove `/dev/tenstorrent/N`.
3.   **Interrupts**: Calls `tenstorrent_disable_interrupts()`[enumerate.c 332](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L332-L332)
4.   **XArray**: Erases the ordinal entry [enumerate.c 333](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L333-L333)
5.   **Reference Counting**: Calls `tenstorrent_device_put()`[enumerate.c 334](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L334-L334) When the refcount reaches zero, `tt_dev_release()`[enumerate.c 337-347](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L337-L347) is invoked to `kfree()` the device structure.

Sources: [enumerate.c 323-347](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L323-L347)[interrupt.c 39-46](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/interrupt.c#L39-L46)
