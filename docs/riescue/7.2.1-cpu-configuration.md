## Purpose and Scope

This document describes the CPU configuration file format (`cpu_config.json`) used by RiescueD to define the target processor's memory map, architectural features, and other system-level properties. This configuration determines how tests are generated, assembled, and executed.

For information about how FeatMgr consolidates CPU configuration with test directives and command-line arguments, see [Feature Manager](https://deepwiki.com/tenstorrent/riescue/7.1.2-feature-manager-(featmgr)). For command-line usage of the `--cpuconfig` option, see [RiescueD Command-Line Interface](https://deepwiki.com/tenstorrent/riescue/2.1.1-riescued-command-line-interface).

## Configuration File Structure

The CPU configuration file is a JSON document that defines system properties required for test generation and execution. The file is typically named `cpu_config.json` and is passed to RiescueD via the `--cpuconfig` command-line argument.

### Core Sections

The configuration file contains the following primary sections:

| Section | Purpose | Required |
| --- | --- | --- |
| `memory_map` | Defines addressable memory regions | Yes |
| `features` | Specifies enabled/disabled ISA extensions | Yes |
| Additional properties | System-specific configuration | Optional |

## Memory Map Configuration

### Memory Map Structure

The `memory_map` section defines the physical memory regions available to the test environment. Each region is identified by a name and contains addressing information.

**Memory Map: Configuration Structure**

### Memory Region Properties

Each memory region is defined by the following properties:

| Property | Type | Description | Example |
| --- | --- | --- | --- |
| `start` | String (hex) | Base address of the memory region | `"0x80000000"` |
| `size` | String (hex) | Size of the memory region in bytes | `"0x10000000"` |

### Example Memory Map

### Memory Map Usage in Test Generation

The memory map configuration is used by RiescueD components for:

1.   **Random Address Generation**: Ensures generated addresses fall within valid memory regions
2.   **Page Table Generation**: Creates page tables mapping virtual to physical addresses within defined regions
3.   **Linker Script Generation**: Places test sections at valid addresses
4.   **Simulator Configuration**: Configures memory regions in Whisper and Spike

## Feature Configuration

### Features Section Structure

The `features` section specifies which RISC-V extensions and architectural features are enabled or disabled for the target CPU. This configuration affects:

*   Test generation constraints
*   Instruction availability
*   CSR accessibility
*   Simulator configuration

**Feature Configuration: Extension Flags**

### Feature Flag Format

Features are specified as key-value pairs where:

*   **Key**: Feature identifier following the naming convention `ext_<extension>`
*   **Value**: Boolean indicating whether the feature is enabled

| Feature Key | Description | Type |
| --- | --- | --- |
| `ext_v` | RISC-V Vector Extension (V) | Boolean |
| `ext_fp` | Floating-Point Extensions (F, D, Q) | Boolean |
| `ext_a` | Atomic Extension (A) | Boolean |
| `ext_c` | Compressed Extension (C) | Boolean |
| `ext_m` | Integer Multiplication/Division (M) | Boolean |

## Integration with Test Generation Pipeline

### Configuration Flow Through RiescueD

The CPU configuration file flows through multiple components in the test generation pipeline:

**CPU Configuration Integration: Data Flow**

### Component Interactions

The CPU configuration is consumed by the following RiescueD components:

1.   **Parser**: Loads and validates the JSON configuration file
2.   **FeatMgr**: Consolidates feature flags with test header directives and CLI arguments
3.   **Memory Management**: Uses memory map for random address generation and page tables
4.   **Generator**: Applies feature constraints to generated assembly
5.   **Linker Script Generator**: Places sections according to memory map

## Configuration Precedence and Feature Resolution

### Feature Consolidation Hierarchy

When multiple sources specify feature configuration, FeatMgr resolves conflicts using the following precedence (highest to lowest):

**Configuration Precedence: Resolution Hierarchy**

### Precedence Rules

| Priority | Source | Example | Override Behavior |
| --- | --- | --- | --- |
| 1 (Highest) | Command-line arguments | `--paging_mode sv39` | Overrides all other sources |
| 2 | Test header directives | `;#test.features ext_v.enable` | Overrides CPU config and defaults |
| 3 | CPU configuration file | `"ext_v": true` | Overrides defaults only |
| 4 (Lowest) | System defaults | Built-in values | Used when not specified elsewhere |

This precedence model allows:

*   Global defaults via CPU configuration
*   Per-test overrides via test headers
*   Runtime overrides via command-line arguments

## Memory Map and Address Generation

### Random Address Generation Constraints

The memory map configuration directly constrains random address generation for test directives:

**Address Generation: Memory Map Constraints**

### Page Table Generation

For paged memory modes (sv39, sv48, sv57), the memory map guides page table construction:

1.   **Physical Address Ranges**: Determined by memory map regions
2.   **Virtual Address Mapping**: Maps to physical addresses within configured regions
3.   **Page Table Placement**: Placed in RAM regions defined in memory map

## Feature Flags and Test Constraints

### Feature-Dependent Test Generation

The `features` section controls which instructions and behaviors are available during test generation:

| Feature Disabled | Impact on Test Generation |
| --- | --- |
| `ext_v: false` | Vector instructions excluded from generation |
| `ext_fp: false` | Floating-point instructions excluded |
| `ext_a: false` | Atomic instructions excluded |
| `ext_c: false` | Compressed instructions excluded |

### Test Header Feature Overrides

Test headers can enable features locally even if disabled in CPU config:

This test-level directive:

*   Enables vector extension regardless of CPU config

## Best Practices

### Configuration Management

1.   **Version Control**: Store CPU configuration files in version control alongside tests
2.   **Naming Convention**: Use descriptive names (e.g., `cpu_rv64imafdc_256mb.json`)
3.   **Documentation**: Add comments to clarify non-obvious configuration choices
4.   **Validation**: Always validate configuration changes by running test generation
5.   **Reusability**: Create reusable configurations for common target platforms
