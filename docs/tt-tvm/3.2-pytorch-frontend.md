## Purpose and Scope

The PyTorch Frontend is responsible for converting PyTorch models (represented as `torch.jit.ScriptModule`) into TVM Relay IR. This frontend supports over 150 PyTorch operators, quantized models, and preserves source location information for debugging. The converter produces a Relay `IRModule` and parameters dictionary that can be optimized and compiled for various hardware targets.

For information about the general frontend conversion infrastructure shared across all frameworks, see [Common Frontend Infrastructure](https://deepwiki.com/tenstorrent/tt-tvm/3.1-common-frontend-infrastructure). For Relay IR details, see [Relay IR System](https://deepwiki.com/tenstorrent/tt-tvm/4-relay-ir-system).

## Main Components

### Entry Point: `from_pytorch()`

The primary entry point for converting PyTorch models is the `from_pytorch()` function, which accepts a traced PyTorch model and returns a Relay IRModule with parameters.

**Function Signature**:

`def from_pytorch(script_module, input_shapes, custom_convert_map=None,                  default_dtype="float32", use_parser_friendly_name=False,                 keep_quantized_weight=False)`
**Key Parameters**:

*   `script_module`: TorchScript traced model (`torch.jit.ScriptModule`)
*   `input_shapes`: List of (name, shape) tuples for model inputs
*   `custom_convert_map`: Optional custom operator converters
*   `keep_quantized_weight`: Whether to preserve quantized weights or dequantize

**Location**: [python/tvm/relay/frontend/pytorch.py 3690-3810](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L3690-L3810)

### Example: Convolution Operator Conversion

The conversion of PyTorch `conv2d` demonstrates key patterns:

**PyTorch Op**: `aten::conv2d(input, weight, bias, stride, padding, dilation, groups)`

**Conversion Steps**:

2.   Convert PyTorch data layout conventions (NCHW) to Relay
3.   Handle optional bias term
4.   Map to `relay.nn.conv2d()` with appropriate attributes
5.   Add bias using `relay.nn.bias_add()` if present

### Quantization Schemes

The PyTorch frontend supports two quantization schemes:

| Scheme | Description | PyTorch Type | Zero Point Constraint |
| --- | --- | --- | --- |
| **Per-Tensor** | Single scale/zero_point for entire tensor | `torch.per_tensor_affine` | Scalar integer |
| **Per-Channel** | Different scale/zero_point per output channel | `torch.per_channel_affine` | Must be all zeros |

**Implementation**: [python/tvm/relay/frontend/qnn_torch.py 63-77](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/qnn_torch.py#L63-L77)

## Type Inference and Shape Handling

### Incremental Type Inference

The PyTorch frontend uses an incremental type inference system that avoids re-inferring types for the entire module on each operator conversion:

**Diagram: Incremental Type Inference System**

### _TypeFinder Helper Class

The `_TypeFinder` class is an `ExprMutator` that replaces expressions with unknown types with fresh variables so that Relay's type inference can work:

**Key Methods**:

*   `visit()`: Main visitor that checks cache and creates variables for unknown types
*   `visit_let()`: Marks let-bound variables as not inputs
*   `visit_function()`: Marks function parameters as not inputs

**Implementation**: [python/tvm/relay/frontend/pytorch.py 73-104](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L73-L104)

### PyTorch Type Promotion

The `pytorch_promote_types()` method implements PyTorch's type promotion rules to ensure operands have compatible types:

**Rules**:

1.   Extract actual data types from tensor inputs
2.   Determine result type based on PyTorch's promotion rules
3.   Insert casts for inputs that don't match the result type
4.   Promote scalar constants to the result type

**Implementation**: [python/tvm/relay/frontend/pytorch.py 260-282](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L260-L282)

## Advanced Features

### Dynamic Shape Support

The PyTorch frontend handles dynamic shapes using Relay's `Any()` type:

**Shape Inference with Dynamic Dimensions**:

*   Uses `relay.Any()` for unknown dimensions
*   Employs `relay.op.shape_of()` for runtime shape queries
*   Supports operators that can work with dynamic shapes (reshape, concatenate, etc.)

**Implementation**: [python/tvm/relay/frontend/pytorch.py 468-570](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L468-L570)

### List ADT Support

For dynamic list operations, the frontend uses Relay's Algebraic Data Type (ADT) system:

**When Lists are Dynamic**:

*   Element access at runtime (`__getitem__`)
*   In-place modifications (`add_`)
*   Used inside loops

**List Construction**:

`_, cons, nil = self.prelude.mod.get_type("List")adt_lst = nil()for elem in reversed(py_lst):    adt_lst = cons(elem, adt_lst)`
**Implementation**: [python/tvm/relay/frontend/pytorch.py 118-225](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L118-L225)

### Control Flow Support

The PyTorch frontend supports TorchScript control flow through Relay's control flow constructs:

**Supported Constructs**:

*   `if` statements → `relay.If`
*   `while` loops → Recursive functions via `relay.loops.while_loop`
*   `for` loops → Unrolled or converted to `while`

Sources: [python/tvm/relay/frontend/pytorch.py 118-570](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/pytorch.py#L118-L570)

### Operator Converter Pattern

Most operator converters follow this pattern:

1.   **Extract Inputs**: Get input expressions from the graph node
2.   **Extract Attributes**: Parse operator attributes (stride, padding, etc.)
3.   **Type Promotion**: Apply PyTorch type promotion if needed
4.   **Create Relay Op**: Construct the equivalent Relay operator
5.   **Record Type**: Infer and cache the output type
6.   **Return Expression**: Return the Relay expression
