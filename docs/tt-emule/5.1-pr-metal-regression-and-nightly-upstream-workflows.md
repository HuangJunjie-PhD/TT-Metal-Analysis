The `tt-emule` repository utilizes GitHub Actions to maintain high-fidelity emulation of Tenstorrent hardware. The CI system is divided into two primary pipelines: the **PR Metal Regression**, which gates every pull request against a stable pinned version of `tt-metal`, and the **Nightly Metal Upstream Regression**, which tracks the tip of the `tt-metal` main branch to detect breaking changes early and warm the build cache.

## Workflow Change Detection

To optimize CI resource usage, the `pr-metal-regression.yml` workflow implements a custom runtime change detection job rather than relying on top-level GitHub `paths-ignore` filters. This approach ensures that documentation-only changes skip heavy build and test jobs while still reporting a successful status to the `all-passed` aggregator gate, satisfying branch protection rules without unnecessary computation.

### Implementation Logic

The `changes` job classifies files by comparing the PR head against the base SHA [.github/workflows/pr-metal-regression.yml 40-88](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/pr-metal-regression.yml#L40-L88)

*   **Full Suite Trigger:** Always runs for `merge_group` events or when a usable base SHA is missing [.github/workflows/pr-metal-regression.yml 62-75](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/pr-metal-regression.yml#L62-L75)
*   **Skip Criteria:** The suite is skipped if _every_ changed file matches documentation patterns (`*.md`, `docs/*`, `.claude/*`), license files, or `CODEOWNERS`[.github/workflows/pr-metal-regression.yml 82-86](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/pr-metal-regression.yml#L82-L86)

## Build Job & Environment

The build process compiles `tt-metal` with the `-DTT_METAL_USE_EMULE=ON` flag, integrating the local `tt-emule` source code into the `tt-metal` hardware abstraction layer.

### CI Environment

*   **Container:**`ghcr.io/tenstorrent/tt-mlir/tt-mlir-ci-ubuntu-24-04:latest`, which provides a pre-baked toolchain including `clang-20` and `gcc-13`[.github/workflows/pr-metal-regression.yml 96-100](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/pr-metal-regression.yml#L96-L100)
*   **CCache:** Persistent caching is keyed by the pinned `tt-metal` SHA and the compiler version to accelerate incremental builds [.github/workflows/pr-metal-regression.yml 137-145](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/pr-metal-regression.yml#L137-L145)

### Build Execution (`ci-build.sh`)

The `ci-build.sh` script handles the configuration and compilation of the metal project.

1.   **UMD Patching:** It applies an idempotent patch to `umd/device/cluster.cpp` to include `sw_emule_chip.hpp`, a requirement for the emulation build in current `tt-metal` versions [.github/scripts/ci-build.sh 36-45](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/scripts/ci-build.sh#L36-L45)
2.   **CMake Configuration:** Configures the build with `Ninja`, using the `x86_64-linux-clang-20-libstdcpp-toolchain.cmake` file [.github/scripts/ci-build.sh 57-70](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/scripts/ci-build.sh#L57-L70)
3.   **Target Selection:** Only builds essential targets: `unit_tests_api`, `unit_tests_integration`, `unit_tests_data_movement`, `unit_tests_ttnn`, and the `ttnn` Python bindings [.github/scripts/ci-build.sh 79-86](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/scripts/ci-build.sh#L79-L86)
4.   **Artifact Staging:** Post-build symlinks for `libtt_metal.so` and `_ttnn.so` are created so that the uploaded artifact is self-contained [.github/scripts/ci-build.sh 89-99](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/scripts/ci-build.sh#L89-L99)

## Test Matrix & Sharding

The test phase executes the regression suite across a matrix of three architectures: `wormhole`, `blackhole`, and `quasar`.

### Architecture Policies

*   **Wormhole & Blackhole:** Zero-tolerance policy; 100% of tests must pass.
*   **Quasar:** Utilizes an allowlist (`.github/known-failures-quasar.txt`) for known architectural gaps, such as tests exceeding the 6-user-DM limit or matmul block issues related to DFB format emission [.github/known-failures-quasar.txt 1-33](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/known-failures-quasar.txt#L1-L33)

### Pytest Sharding

For `ttnn` Python tests, the workflow utilizes sharding to parallelize execution. The suite is split into 5 shards, which are executed in parallel across the matrix to reduce total wall-clock time.

### Result Classification

The `classify-results.py` script parses the GTest XML outputs. It enforces that:

1.   No new failures are introduced (unless in the Quasar allowlist).
2.   No stale entries remain in the allowlist (if a test starts passing, the allowlist must be updated).

## Nightly Upstream Workflow

The `nightly-metal-upstream.yml` workflow runs daily at 07:00 UTC [.github/workflows/nightly-metal-upstream.yml 18](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/nightly-metal-upstream.yml#L18-L18)

### Objectives

1.   **Early Warning:** Detects if the latest changes in `tt-metal/main` break the `tt-emule` integration [.github/workflows/nightly-metal-upstream.yml 8](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/nightly-metal-upstream.yml#L8-L8)
2.   **Cache Warming:** Populates the GitHub Actions `ccache` with the latest `tt-metal` objects. When a developer later updates `tt-metal-pin.txt` to the current main tip, their PR will benefit from a warm cache, reducing build times from ~20 minutes to <5 minutes [.github/workflows/nightly-metal-upstream.yml 9-10](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/nightly-metal-upstream.yml#L9-L10)

### Implementation Differences

Unlike the PR workflow, the nightly run:

*   Resolves the `tt-metal` SHA dynamically using `git ls-remote`[.github/workflows/nightly-metal-upstream.yml 48](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/nightly-metal-upstream.yml#L48-L48)
*   Sets `continue-on-error: true` so that failures do not block other maintenance tasks but still trigger notifications [.github/workflows/nightly-metal-upstream.yml 30](https://github.com/tenstorrent/tt-emule/blob/a57cf072/%20.github/workflows/nightly-metal-upstream.yml#L30-L30)
