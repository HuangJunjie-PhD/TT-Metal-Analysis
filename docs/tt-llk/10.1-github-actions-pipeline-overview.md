## Producer/Consumer Test Execution Pattern

The test infrastructure implements a two-phase execution model that separates compilation (producer) from hardware execution (consumer). This pattern enables parallel compilation of test artifacts followed by sequential hardware execution, optimizing resource utilization.

### TestMode Enumeration

**Sources:** [tests/python\_tests/helpers/test\_config.py78-82](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L78-L82) [tests/python\_tests/conftest.py343-359](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L343-L359) [.github/workflows/setup-and-test.yml136-150](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L136-L150)

### Artifact Generation and Reuse

| Phase | Mode | Parallelization | Output Artifacts | Hardware Required |
| --- | --- | --- | --- | --- |
| Compilation | `PRODUCE` | `-n 10` workers | ELF binaries in `/tmp/tt-llk-build/` | No |
| Execution | `CONSUME` | `-x` sequential | Coverage `.info`, perf CSV, JUnit XML | Yes |

The `/tmp/tt-llk-build/` directory structure contains:

* `shared/obj/` - Shared object files (brisc.o, trisc.o, coverage.o)
* `shared/elf/` - Shared ELF files (brisc.elf)
* `//elf/` - Variant-specific ELFs (unpack.elf, math.elf, pack.elf)

**Sources:** [tests/python\_tests/helpers/test\_config.py119-127](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L119-L127) [.github/workflows/setup-and-test.yml136-150](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L136-L150)

## Test Splitting and Matrix Strategy

Tests are distributed across multiple parallel jobs using pytest-split, with optional duration-based balancing to achieve even load distribution across workers.

### Matrix Generation and Job Distribution

**Configuration Parameters:**

| Workflow | test\_splits | use\_durations | pytest\_markers |
| --- | --- | --- | --- |
| Wormhole PR | 2 | false | `not perf and not nightly and not quasar` |
| Blackhole PR | 2 | false | `not perf and not nightly and not quasar` |
| Performance tests | 5 | false | `perf` |
| Coverage nightly | Variable | true | All tests |

**Sources:** [.github/workflows/setup-and-test.yml14-47](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L14-L47) [.github/workflows/on-pr.yml144-205](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L205)

## Change Detection System

The `detect-changes` job uses file pattern matching to determine which architecture-specific tests should execute, minimizing resource usage for focused changes.

### Detection Patterns

| Output Variable | File Patterns | Triggered Jobs |
| --- | --- | --- |
| `wormhole` | `tt_llk_wormhole_b0/**` | setup-and-test-wormhole |
| `blackhole` | `tt_llk_blackhole/**` | setup-and-test-blackhole |
| `quasar` | `tt_llk_quasar/**`, `tests/sources/quasar/**`, `tests/python_tests/quasar/**` | build-quasar |
| `tests` | `tests/**` (excluding quasar) | All test jobs |
| `github` | `.github/**` | All test jobs |
| `performance` | `**/*perf**`, `**/*profiler**` | perf-tests matrix |

## Concurrency Control

The pipeline implements intelligent concurrency management to prevent resource conflicts while allowing useful parallel execution:

**Behavior:**

* **Protected branches (main):** Uses `github.run_id` - never cancels runs
* **Pull requests:** Uses PR number - cancels obsolete runs on new pushes
* **Other branches:** Uses branch ref - cancels previous runs on new commits

**Sources:** [.github/workflows/on-pr.yml17-25](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L17-L25)
