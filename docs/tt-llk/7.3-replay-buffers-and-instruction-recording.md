## Purpose and Scope

Replay buffers are hardware-accelerated instruction storage mechanisms in the Blackhole architecture that enable efficient execution of repetitive instruction sequences. This document covers:

*   The replay buffer architecture and organization
*   Recording instruction sequences using `load_replay_buf()`
*   Invoking replay instructions with `lltt::replay_insn()`
*   Common usage patterns in tilize, untilize, and data movement operations

For information about the broader MOP (Micro-Operation Programming) system that utilizes replay buffers, see [7.1](https://deepwiki.com/tenstorrent/tt-llk/7.1-ckernel_template-and-mop-execution). For address modifier configuration used within replay sequences, see [7.2](https://deepwiki.com/tenstorrent/tt-llk/7.2-address-modifiers-(addr_mod_0-7)).

**Note**: Replay buffers are a Blackhole-specific optimization feature. Wormhole B0 uses direct instruction programming in `ckernel_template` without replay buffer support.

## Replay Buffer Architecture

Replay buffers provide hardware storage for instruction sequences that can be efficiently replayed without repeated instruction fetch overhead. The buffer space is partitioned between different functional units.

**Buffer Organization**

| Component | Offset Range | Purpose | Typical Users |
| --- | --- | --- | --- |
| SFPU Space | 0-15 | Special function unit operations | Math SFPU operations |
| FPU/General Space | 16-31 | General purpose operations | Packer, Unpacker operations |

## Recording Instruction Sequences

The `load_replay_buf()` function records instruction sequences into hardware replay buffers. It takes three parameters:

1.   **offset**: Starting position in the replay buffer (0-31)
2.   **replay_buf_len**: Number of instructions to record
3.   **lambda function**: Contains the instruction sequence to record

### Lambda-Based Recording Interface

The lambda-based interface provides several advantages:

*   **Type Safety**: Captures instructions at compile time
*   **Inline Execution**: Instructions are written directly to replay buffer during kernel initialization
*   **Flexible Composition**: Multiple instruction types can be sequenced together

## Tilize Packer Replay Pattern

The tilize packer uses replay buffers to optimize face packing operations with alternating interface masks.

**Key Characteristics**

*   **Variable Length**: Replay buffer length = `face_r_dim - 1` instructions
*   **Interface Alternation**: Mask 0b1010 for even faces, 0b0101 for odd faces
*   **Address Modifiers**: ADDR_MOD_0 for iteration, ADDR_MOD_1 for face transition, ADDR_MOD_2 for tile completion
*   **Override Control**: Last instruction kept outside buffer to allow conditional tile closing

**Instruction Sequence**

| Phase | Instruction Count | Address Modifier | Interface Mask |
| --- | --- | --- | --- |
| Face 0 loop | `(face_r_dim/2) - 1` | ADDR_MOD_0 | 0b1010 |
| Face 0 end | 1 | ADDR_MOD_1 | 0b1010 |
| Face 1 loop | `(face_r_dim/2) - 1` | ADDR_MOD_0 | 0b0101 |
| Face 1 end | 1 (outside buffer) | ADDR_MOD_2 | 0b0101 |

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack.h 148-253](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack.h#L148-L253)

## TilizeA_B Unpacker Replay Pattern

The tilizeA_B unpacker uses replay buffers to efficiently unpack row-major data into tiled format while updating L1 addresses across contexts.

**Dual-Context Strategy**

The replay buffer is organized to support context switching:

*   **First Half (offset 0-2)**: Operations for configuration context 0

    *   Unpack 1×16 row to SrcA
    *   Increment `THCON_SEC0_REG3_Base_address`
    *   NOP for timing

*   **Second Half (offset 3-5)**: Operations for configuration context 1

    *   Unpack 1×16 row to SrcA
    *   Increment `THCON_SEC0_REG3_Base_cntx1_address`
    *   NOP for timing

**Address Increment Mechanism**

Uses `TTI_CFGSHIFTMASK` to atomically update L1 base addresses:

*   Reads value from `SCRATCH_SEC0_val` register (set during init)
*   Adds to current base address
*   Writes back to context-specific address register

**Sources**: [tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 201-269](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L201-L269)

## Untilize Replay Patterns

### Unpacker Untilize Replay

The untilize unpacker uses replay buffers to efficiently read tiled data in row-major order:

**Offset Management**

The replay sequence maintains a running tile offset in `p_gpr_unpack::TILE_OFFSET`:

1.   Unpacks two rows (Y increment) across faces (Z increment)
2.   Adds tile size to offset for next tile
3.   Waits for address update to complete
4.   Resets Z counter to process next face pair

**Sources**: [tt_llk_blackhole/llk_lib/llk_unpack_untilize.h 21-54](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_untilize.h#L21-L54)

### Packer Untilize Replay

The packer untilize uses a shorter replay buffer to update L1 output addresses:

**Address Offset Calculation**

The `OUTPUT_ADDR_OFFSET` is precomputed during initialization:

*   For normal tiles: `full_ct_dim × (num_faces/2) × FACE_C_DIM × datum_size`
*   For narrow rows: `full_ct_dim × row_num_datums × datum_size`
*   Stored in GPR for efficient replay access

## Replay Buffer Offset Management

### Namespace Partitioning

To avoid conflicts, different subsystems use distinct offset ranges:

| Subsystem | Constant | Value | Usage |
| --- | --- | --- | --- |
| Math/SFPU | N/A | 0-15 | SFPU operations (reserved) |
| Packer | `ckernel::packer::replay_buf_offset` | 16 | Packer operations |
| Unpacker | N/A | 0-15 | Unpacker operations |

**Critical Constraints**

*   **Non-overlapping**: Different operation types must not share replay buffer space
*   **Fixed Allocation**: Offset values are compile-time constants to prevent runtime conflicts
*   **Subsystem Isolation**: Each subsystem (pack/unpack/math) manages its own portion

## Performance Considerations

### When to Use Replay Buffers

Replay buffers provide performance benefits when:

1.   **Repetitive Sequences**: The same instruction pattern repeats multiple times
2.   **Variable Parameters**: Only data addresses or counters change between iterations
3.   **Complex Patterns**: Multi-instruction sequences that would otherwise bloat code size

**Overhead Analysis**

| Approach | Instruction Fetch | Setup Cost | Best For |
| --- | --- | --- | --- |
| Direct MOP | Per iteration | Low | Simple, few iterations |
| Replay Buffer | One-time | Higher initial recording | Complex, many iterations |
