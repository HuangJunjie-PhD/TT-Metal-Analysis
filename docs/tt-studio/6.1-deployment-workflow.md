This document describes the step-by-step model deployment process in TT-Studio using the stepper-based user interface. The deployment workflow guides users through hardware detection, model selection, chip allocation, and asynchronous job monitoring.

For information about model configuration parameters and deployment options, see [Model Configuration](https://deepwiki.com/tenstorrent/tt-studio/6.2-model-configuration). For details about the inference system that receives deployed models, see [Inference System](https://deepwiki.com/tenstorrent/tt-studio/7-inference-system).

## Workflow Overview

The TT-Studio deployment workflow is implemented using a multi-step process in the `SelectionSteps.tsx` component [app/frontend/src/components/SelectionSteps.tsx 34-83](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L34-L83) It adapts dynamically based on the detected hardware (e.g., single-chip vs. multi-chip boards like QB2/P300Cx2) [app/frontend/src/components/SelectionSteps.tsx 47-72](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L47-L72)

## Step-by-Step Breakdown

### 1. Hardware Detection & Path Selection

On mount, `SelectionSteps` fetches chip status from `/docker-api/chip-status/`[app/frontend/src/components/SelectionSteps.tsx 53-68](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L53-L68)

*   **Single-Chip/QB2**: Defaults to a simplified 2-step flow (Model Selection -> Deploy) [app/frontend/src/components/SelectionSteps.tsx 80-83](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L80-L83)
*   **Multi-Chip (Non-QB2)**: Defaults to a 3-step flow including "Hardware Configuration" [app/frontend/src/components/SelectionSteps.tsx 74-79](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L74-L79)

### 2. Model Selection (`FirstStepForm`)

Users select a model from the list fetched via `getModelsUrl`[app/frontend/src/components/FirstStepForm.tsx 125-140](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/FirstStepForm.tsx#L125-L140)

*   **Compatibility Check**: The UI validates if the selected model is compatible with the `current_board`[app/frontend/src/components/FirstStepForm.tsx 156-170](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/FirstStepForm.tsx#L156-L170)
*   **Auto-Deploy**: Supports `auto-deploy` URL parameters to bypass manual selection [app/frontend/src/components/FirstStepForm.tsx 204-220](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/FirstStepForm.tsx#L204-L220)

### 3. Chip Allocation (`ChipConfigStep`)

For multi-chip systems, this step allows users to specify which device ID to target. If omitted (as in the QB2 simplified flow), the backend auto-allocates the best available slot [app/frontend/src/components/SelectionSteps.tsx 193-195](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L193-L195)

### 4. Deployment Execution (`DeployModelStep`)

This step triggers the asynchronous deployment job via a POST request to `/docker-api/deploy/`[app/frontend/src/components/SelectionSteps.tsx 196-205](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L196-L205)

*   **Job Monitoring**: The frontend receives a `job_id` and uses the `useDeploymentProgress` hook to monitor status [app/frontend/src/components/DeployModelStep.tsx 53-54](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L53-L54)
*   **Asynchronous Feedback**: The system attempts to use Server-Sent Events (SSE) via `/docker-api/deploy/progress/stream/{jobId}/` and falls back to standard polling if SSE fails [app/frontend/src/hooks/useDeploymentProgress.ts 106-169](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/hooks/useDeploymentProgress.ts#L106-L169)
*   **Logs**: Users can fetch real-time deployment logs from `/docker-api/deploy/logs/{jobId}/` to debug failures [app/frontend/src/components/DeployModelStep.tsx 64-89](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L64-L89)

Sources: [app/frontend/src/components/DeployModelStep.tsx 98-144](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L98-L144)[app/frontend/src/hooks/useDeploymentProgress.ts 59-104](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/hooks/useDeploymentProgress.ts#L59-L104)[app/frontend/src/components/SelectionSteps.tsx 171-205](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/SelectionSteps.tsx#L171-L205)

## Error Recovery and Monitoring

The workflow includes several mechanisms to handle deployment failures:

| Feature | Implementation |
| --- | --- |
| **Progress Tracking** | `DeploymentProgress.tsx` component displays stages like `model_preparation` (weights download) and `container_setup`[app/frontend/src/components/ui/DeploymentProgress.tsx 26-52](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/ui/DeploymentProgress.tsx#L26-L52) |
| **Grace Periods** | Polling includes a 90-second grace period for "job not found" errors to account for race conditions during container startup [app/frontend/src/hooks/useDeploymentProgress.ts 41-42](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/hooks/useDeploymentProgress.ts#L41-L42) |
| **Error Persistance** | `deploymentError` state in `DeployModelStep` ensures errors remain visible even if polling stops [app/frontend/src/components/DeployModelStep.tsx 44-50](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L44-L50) |
| **Log Inspection** | A "View Logs" button appears on failure, calling `fetchDeploymentLogs`[app/frontend/src/components/DeployModelStep.tsx 64-89](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L64-L89) |

## Completion Actions

Once `hasCompletedAllSteps` is true, the `StepperFooter` provides two paths:

1.   **Deploy Another**: Calls `resetSteps()` to return to the beginning [app/frontend/src/components/StepperFooter.tsx 37-40](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/StepperFooter.tsx#L37-L40)
2.   **View Deployed Models**: Navigates the user to the `/models-deployed` route [app/frontend/src/components/StepperFooter.tsx 60-64](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/StepperFooter.tsx#L60-L64)
