## Purpose and Scope

The `tt-lang` test framework validates correctness across three dimensions:

*   **Compilation correctness**: MLIR transformations produce valid IR and C++ code.
*   **Runtime correctness**: Kernels compute correct results on tensors.
*   **Hardware behavior**: Dataflow Buffer (DFB) lifecycle, synchronization, and communication protocols work correctly.

The framework supports testing without hardware through a functional simulator, enabling rapid development and CI/CD integration.

## Test Categories

The test suite is organized into several distinct categories, each with dedicated CMake targets and runners:

| Category | CMake Target | Runner | Path | Hardware Required |
| --- | --- | --- | --- | --- |
| **MLIR Dialect Tests** | `check-ttlang-mlir` | `llvm-lit` | `test/ttlang/` | No |
| **Python Lit Tests** | `check-ttlang-python-lit` | `llvm-lit` (`-j1`) | `test/python/` (non-`test_*`) | No (compile-only) |
| **Python Binding Tests** | `check-ttlang-python-bindings` | `llvm-lit` | `test/bindings/python/` | No |
| **Parametric Tests** | `check-ttlang-pytest` | `pytest` | `test/python/test_*.py` | Optional (auto-detects) |
| **Middle E2E Tests** | `check-ttlang-me2e` | `pytest` | `test/me2e/` | Yes |
| **Simulator Tests** | N/A | `pytest` | `test/sim/` | No |

**Combined targets:**

*   `check-ttlang`: Runs MLIR + Python bindings + packaging tests (no hardware required) [[test/TESTING.md 26](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L26-L26)
*   `check-ttlang-all`: Runs all test suites sequentially to avoid device resource contention [[test/TESTING.md 27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L27-L27)

**File naming convention**: Files matching `test_*.py` under `test/python/` are excluded from `lit` collection and run exclusively via `pytest`[[test/lit.cfg.py 68-74](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/lit.cfg.py#L68-L74) All other `.py` files in that directory are treated as `lit` tests [[test/lit.cfg.py 27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/lit.cfg.py#L27-L27)

### MLIR Dialect and Pass Tests

These validate MLIR transformation passes and dialect operations using `FileCheck` assertions. They verify IR structure after each pass, type conversions, and attribute propagation without requiring a Tenstorrent device. These are located in `test/ttlang/` and executed via the `ttlang-opt` and `ttlang-translate` tools [[test/lit.cfg.py 98-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/lit.cfg.py#L98-L102)

### Python Lit Tests

Execute Python DSL kernels in compile-only mode and validate:

*   **Initial IR**: MLIR before pipeline execution (`ttl` dialect operations).
*   **Final C++ output**: Generated kernel code after full compilation via `EmitC`.

**Execution pattern**:

1.   Run test with `TTLANG_COMPILE_ONLY=1` and `TTLANG_INITIAL_MLIR=%t.initial.mlir`[[test/TESTING.md 65-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L65-L66)
2.   Validate initial IR with `FileCheck`[[test/TESTING.md 71](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L71-L71)
3.   Validate generated C++ with `FileCheck --check-prefix=CHECK-CPP` (optional) [[test/TESTING.md 76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L76-L76)

**Constraint**: Each `lit` test file should contain exactly one `@ttl.operation` (or `@ttl.kernel`) decorated function to avoid overwriting temporary files [[test/TESTING.md 39-42](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L39-L42)

### Parametric Tests (pytest)

Systematic testing across parameter spaces (shapes, ops, grid sizes) using `pytest`'s parametrization. These tests often use dynamic kernel generation to verify behavior across different hardware configurations. These tests are defined in `test/python/test_*.py` and run via the `check-ttlang-pytest` target [[test/TESTING.md 24-29](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L24-L29)

### Simulator Tests

Validate hardware behavior without physical devices. The functional simulator executes kernels as pure Python code, enabling DFB lifecycle validation and deadlock detection. Simulator tests are run via `pytest` on the `test/sim` directory [[test/lit.cfg.py 62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/lit.cfg.py#L62-L62) Command line utilities like `tt-lang-sim` and `tt-lang-sim-stats` are used to run and analyze these simulations [[.github/workflows/call-test-sim.yml 52-53](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[.github/workflows/call-test-sim.yml#L52-L53)

Testing behavior is heavily influenced by environment variables defined in the build system and `lit` configuration.

| Variable | Role | Source |
| --- | --- | --- |

### FileCheck Patterns

Tests use specific prefixes to target different stages of the pipeline:

*   `CHECK`: Targets the initial `ttl` dialect MLIR, verifying circular buffer binding, wait/reserve logic, and thread attributes [[test/TESTING.md 100-132](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L100-L132)
*   `CHECK-CPP`: Targets the final generated C++ kernel source code, verifying DST register lifecycles (`tile_regs_acquire`) and hardware-specific compute calls (`add_binary_tile`) [[test/TESTING.md 149-174](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L149-L174)
*   `PRETTY` / `VERBOSE`: Targets error messages in negative (invalid) tests to verify compiler rejection of malformed input [[test/TESTING.md 88-89](https://github.com/tenstorrent/tt-lang/blob/d76e6233/[test/TESTING.md?plain=1#L88-L89)
