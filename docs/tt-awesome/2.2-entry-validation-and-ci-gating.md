The `tt-awesome` project maintains high data quality through a strict validation layer that ensures every project entry adheres to the system's schema. This process is governed by a dedicated Python validation engine and enforced by GitHub Actions to block any malformed data from entering the `main` branch.

## Validation Architecture

The validation system acts as a gatekeeper between raw JSON contributions and the downstream data pipeline (README generation and the Eleventy website).

## The Validation Engine (`validate.py`)

The core logic resides in `scripts/validate.py`. It performs two distinct types of checks: **Fatal Errors** (which block builds) and **Soft Warnings** (which suggest improvements).

### Key Validation Constants

The script enforces strict adherence to predefined enums to maintain consistency across the ecosystem:

*   **`VALID_AFFILIATIONS`**: `{"official", "affiliated", "community"}`[scripts/validate.py 11](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L11-L11)
*   **`VALID_CATEGORY_SLUGS`**: Includes 12 specific slugs like `ai-models`, `kernels`, and `riscv-arch`[scripts/validate.py 12-16](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L12-L16)
*   **`VALID_LINK_TYPES`**: Restricts links to `repo`, `article`, `talk`, `video`, `website`, `demo`, `lesson`, or `paper`[scripts/validate.py 17](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L17-L17)
*   **`VALID_HARDWARE`**: Validates hardware targets such as `grayskull`, `wormhole`, `blackhole`, `quietbox`, `galaxy`, and `ttsim`[scripts/validate.py 18](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L18-L18)

### Validation Logic Implementation

| Function | Responsibility |
| --- | --- |
| `validate_entry(path, data)` | Checks required fields (`id`, `name`, `description`, `affiliation`), URL formats via `URL_RE`, and enum constraints. [scripts/validate.py 24-93](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L24-L93) |
| `validate_entry_warnings(path, data)` | Identifies non-breaking issues, such as a missing `added_at` field. [scripts/validate.py 96-101](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L96-L101) |
| `main()` | Crawls the `entries/` directory, detects duplicate IDs, and manages the final exit status. [scripts/validate.py 104-151](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L104-L151) |

### Schema Enforcement Details

1.   **ID/Filename Matching**: The `id` field inside the JSON must exactly match the filename (stem) to ensure consistent lookups [scripts/validate.py 26-28](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L26-L28)
2.   **URL Security**: All URLs must use the `https://` protocol, enforced by `URL_RE`[scripts/validate.py 20](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L20-L20)[scripts/validate.py 49-50](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L49-L50)
3.   **Data Integrity**: Fields like `featured`, `hidden`, and `hardware` are type-checked (e.g., ensuring `featured` is a boolean) [scripts/validate.py 80-84](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L80-L84)
4.   **Global Uniqueness**: The `main()` function maintains a `seen` set of IDs across all files to prevent duplicate project identifiers [scripts/validate.py 136-143](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/validate.py#L136-L143)

## CI Gating & GitHub Actions

The validation process is integrated into the development workflow via two primary GitHub Actions.

### PR Validation (`validate.yml`)

This workflow triggers on any pull request that modifies files in the `entries/` directory or the validation script itself [.github/workflows/validate.yml 3-4](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/validate.yml#L3-L4) It runs `scripts/validate.py` and will fail the PR check if any fatal errors are found [.github/workflows/validate.yml 12](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/validate.yml#L12-L12)

### Deployment Safety (`deploy.yml`)

While `validate.yml` gates PRs, the `deploy.yml` workflow (which runs on `push` to `main`) performs the actual build. It executes `scripts/generate_readme.py` and `scripts/generate_data_json.py` before building the Eleventy site [.github/workflows/deploy.yml 26-30](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L26-L30) Because these generation scripts rely on the data being valid, the PR gate is the critical line of defense.

**Sources:**[.github/workflows/validate.yml 1-13](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/validate.yml#L1-L13)[.github/workflows/deploy.yml 1-40](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/deploy.yml#L1-L40)

## Data Sanitization in the Build Layer

Even after validation, the Eleventy data layer (`src/_data/entries.js`) applies additional safety filters during the build process to ensure the live website remains secure and well-formatted.

**Data Layer Logic Flow**

### Sanitization Functions

*   **`isSafeHttpsUrl`**: A JavaScript utility that verifies URL protocol and hostname existence, providing a second layer of defense against malformed links [src/_data/entries.js 16-24](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/entries.js#L16-L24)
*   **Hidden Entry Filtering**: Entries with `hidden: true` are permitted in the repository for historical reasons but are explicitly filtered out of the website's `visibleEntries` array [src/_data/entries.js 82](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/entries.js#L82-L82)
*   **Automated Author Links**: If `author_url` is missing, the script attempts to derive it from the GitHub repository owner [src/_data/entries.js 53-61](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/entries.js#L53-L61)
