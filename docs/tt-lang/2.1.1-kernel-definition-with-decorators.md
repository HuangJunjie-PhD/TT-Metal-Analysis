This page explains the decorator-based API for defining tt-lang kernels. It covers the primary decorators (`@ttl.operation`, `@ttl.compute`, `@ttl.datamovement`), their parameters, and how they work together to define kernel execution on Tenstorrent hardware.

For information about the conceptual differences between compute and data movement execution contexts, see [Compute vs Data Movement Threads (2.1.2)](https://github.com/tenstorrent/tt-lang/blob/d76e6233/Compute%20vs%20Data%20Movement%20Threads%20(2.1.2)) For details about the memory architecture, see [Memory Architecture (2.3)](https://github.com/tenstorrent/tt-lang/blob/d76e6233/Memory%20Architecture%20(2.3))

## Overview of the Decorator Hierarchy

tt-lang kernels are defined using a multi-level decorator structure that maps Python functions to specific hardware RISC processors. The top-level entry point is `@ttl.operation`, which orchestrates the underlying thread functions.

### Decorator to Hardware Mapping

"Decorator to Hardware Mapping"

**Decorator Hierarchy:**

| Decorator Level | Purpose | Scope | Key Functions |
| --- | --- | --- | --- |
| `@ttl.operation` | Top-level kernel entry point | Defines grid and tensor mapping | `ttl.grid_size`, `ttl.node` |
| `@ttl.compute` | Compute thread function | Executes mathematical operations on FPU/SFPU | `ttl.math.*`, `store()` |
| `@ttl.datamovement` | Data movement thread function | Handles DMA/NOC operations | `ttl.copy()`, `wait()` |

## The `@ttl.operation` Decorator

The `@ttl.operation` decorator (historically referred to as `@ttl.kernel` in early versions) is the entry point for tt-lang [docs/sphinx/specs/TTLangSpecification.md 45](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L45-L45) It handles the transition from standard Python execution to the MLIR-based compilation pipeline via `TTLGenericCompiler`[python/ttl/ttl_api.py 70](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L70-L70)

### Decorator Behavior

1.   **Argument Capture**: The function takes input and output `ttnn.Tensor` objects [docs/sphinx/specs/TTLangSpecification.md 68-71](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L68-L71)
2.   **Thread Registration**: When the operation is called, it triggers the collection of all nested functions decorated with `@ttl.compute` and `@ttl.datamovement` into a `_thread_registry`[python/ttl/ttl_api.py 101-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L101-L118)
3.   **Caching**: Compilations are cached based on tensor properties (shape, dtype, layout) and compiler options to avoid redundant MLIR processing [python/ttl/ttl_api.py 133-158](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L133-L158)

### Grid Specification

Inside the operation function, developers specify the execution grid. A grid defines a space of nodes (Tensix Cores) to which the operation is submitted [docs/sphinx/specs/TTLangSpecification.md 100-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L102)

| Function | Purpose |
| --- | --- |
| `ttl.grid_size(dims)` | Returns the size of the grid in specified dimensionality [docs/sphinx/specs/TTLangSpecification.md 106-114](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L106-L114) |
| `ttl.node(dims)` | Returns the coordinates of the current node in the grid [docs/sphinx/specs/TTLangSpecification.md 43](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L43-L43) |

## Thread Decorators

Kernel functions are Python functions with no arguments and returning `None`, annotated by `@ttl.compute` or `@ttl.datamovement`[docs/sphinx/specs/TTLangSpecification.md 74-84](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L74-L84)

### The `@ttl.compute` Decorator

Identifies a function intended for the TRISC processors. These functions perform mathematical operations on tiles residing in local L1 memory (Dataflow Buffers).

*   **Mathematical Operations**: Uses `ttl.math` for operations like `abs`, `neg`, and `pow`[docs/sphinx/specs/TTLangSpecification.md 46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L46-L46)
*   **DST Allocation**: The compiler performs DST register assignment for these operations during the `ttl-assign-dst` pass [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L36)

### The `@ttl.datamovement` Decorator

Defines a function for the BRISC or NCRISC processors. These threads handle data movement between the NOC and local L1 buffers.

*   **DMA Operations**: Typically uses `ttl.copy` to move data between tensors and blocks [docs/sphinx/specs/TTLangSpecification.md 15-16](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L15-L16)
*   **Synchronization**: Uses `wait()` on transfer handles returned by copy operations [include/ttlang/Dialect/TTL/Passes.td 108-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L108-L118)

## Compilation Pipeline

When an `@ttl.operation` is invoked, the `ttl-to-ttkernel-pipeline` transforms the Python DSL into executable hardware code [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 19-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L19-L76)

### Key Transformation Passes

"Compilation Data Flow"

| Pass Name | Role |
| --- | --- |
| `ttl-insert-intermediate-dfbs` | Materializes intermediate values to L1 via compiler-allocated DFBs at fusion split points [lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp 9-13](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp#L9-L13) |
| `ttl-insert-cb-sync` | Inserts missing `cb_push`/`cb_pop` for unmatched `cb_reserve`/`cb_wait`[include/ttlang/Dialect/TTL/Passes.td 6-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L6-L12) |
| `ttl-coalesce-dfb-acquires` | Collapses consecutive same-DFB acquires into a single multi-tile acquire [include/ttlang/Dialect/TTL/Passes.td 28-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L28-L48) |
| `ttl-assign-dst` | Maps mathematical operations to hardware DST registers [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L36) |
