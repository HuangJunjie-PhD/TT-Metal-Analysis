The Model Tracer is a specialized toolset designed to bridge the gap between high-level model execution and low-level operation validation. It automatically extracts real-world operation configurations—including tensor shapes, data types, memory layouts, and exact sharding specifications—from production model tests and integrates them into the `sweep_framework` for exhaustive testing [[tests/sweep_framework/README.md:25-29]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[model_tracer/README.md:3-6]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

## System Architecture

The extraction process transforms execution data from **Natural Language Space** (model names and intent) into **Code Entity Space** (specific `ttnn` operation parameters and sweep vectors).

### Data Flow Overview

The tracer operates by intercepting operation calls during model execution. When a model is run with the `--trace-params` flag, the `ttnn` backend serializes operation metadata into individual JSON files [[model_tracer/README.md:15-22]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[tests/sweep_framework/sweeps_runner.py:65-88]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction). These traces are then aggregated into a master configuration file, `ttnn_operations_master.json`, which serves as the primary source for model-traced sweep tests [[tests/sweep_framework/README.md:31-38]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[model_tracer/README.md:20-25]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

## Key Components and Implementation

### 1. The Tracer (`generic_ops_tracer.py`)

`generic_ops_tracer.py` is the core tool for operation extraction. It executes a model test, either as a pytest target or a standalone Python script, with the `--trace-params` flag enabled to activate lightweight parameter tracing without the overhead of graph tracing [[model_tracer/generic_ops_tracer.py:9-25]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

*   It dynamically detects the test type (pytest or standalone) and invokes the test accordingly.
*   Extracts detailed machine and software metadata using tools like `pyluwen` or `tt-smi` snapshot for accurate hardware identification [[model_tracer/generic_ops_tracer.py:171-202]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).
*   Captured operation parameters are serialized as individual JSON files to the directory `generated/ttnn/reports/operation_parameters/`.
*   Post-tracing, the individual JSONs are filtered by a curated list of active operations (in `Allops.txt`) and merged, with deduplication, into a master JSON file named `ttnn_operations_master.json`.

By running `generic_ops_tracer.py` with a model test path (e.g., `models/tt_transformers/demo/simple_text_demo.py::test_demo_text`), developers can capture real-world operation parameters reflecting actual model workloads [[model_tracer/README.md:13-24]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

### 2. Database Integration and Trace Management (`load_ttnn_ops_data_v2.py` and `trace_selection_registry.yaml`)

The lifecycle of traced data extends beyond file capture:

*   `load_ttnn_ops_data_v2.py` handles loading the master JSON trace files into a PostgreSQL database for persistent storage, allowing CI pipelines to query and reconstruct test configurations dynamically [[model_tracer/README.md:88-92]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).
*   It supports commands like `load`, `reconstruct-manifest`, and `list-traces` to manage trace lifecycle.
*   The `trace_selection_registry.yaml` maintains: 
    *   A **"targets"** section, specifying which models and trace IDs are selected for reconstruction into CI sweep manifests.
    *   A **"registry"** section, recording metadata for all loaded traces, their status (draft/active), associated models, and hardware [[model_tracer/trace_selection_registry.yaml:1-40]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

This architecture enables selective promotion and filtering of trace data for CI usage, ensuring only validated and approved traces are integrated into test vectors.

### 3. Vector Generation (`sweeps_parameter_generator.py`)

The `sweeps_parameter_generator.py` is responsible for converting trace data into test vectors consumable by the sweep tests. It:

*   Parses the master JSON or database outputs.
*   Groups traced vectors by **mesh device shape** or **hardware configuration**, recognizing physical constraints of target test runners [[tests/sweep_framework/sweeps_parameter_generator.py:36-106]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).
*   Extracts hardware tuples and mesh shapes robustly, including legacy and new format compatibility [[tests/sweep_framework/sweeps_parameter_generator.py:140-162]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).
*   Supports grouping policies such as "mesh" or "hardware" for different CI routing strategies.
*   Obeys routing policies to ensure appropriate vector sets run on compatible devices and clusters [[tests/sweep_framework/framework/matrix_runner_config.py:7-20]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

This process ensures real model configurations run on representative hardware in CI tests.

## Lead Models and CI Routing

Certain priority models, called **Lead Models** (e.g., Llama-3, DeepSeek V3, GPT-OSS), are given dedicated CI coverage with specialized routing and batching policies [[tests/sweep_framework/README.md:42-44]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[model_tracer/trace_selection_registry.yaml:43-56]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

These models are typically traced from demo or production scripts and have their operation parameters extracted to guide CI testing. For reference, demo tests like `simple_text_demo.py` produce trace data for many lead models [[models/tt_transformers/demo/simple_text_demo.py:38-39]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

### Routing Policy Overview

Routing assignments are managed by `matrix_runner_config.py` which defines:

*   Physical runner profiles (e.g., N150, N300, Galaxy)
*   Logical test groups (e.g., `lead-models-single-chip`, `lead-models-galaxy`)
*   Mappings from hardware groups and mesh shapes to test groups for sweep vector routing
*   Batching policies for parallel jobs per CI lane [[tests/sweep_framework/framework/matrix_runner_config.py:7-20]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

| Model Type | Hardware Group | Logical Test Group | Runner Profile |
| --- | --- | --- | --- |
| Lead Model | Single-chip | `lead-models-single-chip` | `n150` |
| Lead Model | Multi-chip | `lead-models-galaxy` | `galaxy-topology-6u` |
| Model Traced | Wormhole N150 | `wormhole-n150-sweeps` | `n150` |
| Model Traced | Blackhole P150b | `blackhole-p150b-sweeps` | `p150b` |

This ensures that traced operations relevant to these models are run on representative and resource-appropriate hardware in CI [[tests/sweep_framework/README.md:52-65]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[tests/sweep_framework/framework/matrix_runner_config.py:133-143]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction), [[model_tracer/trace_selection_registry.yaml:42-58]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

## Usage Workflow

The Model Tracer integration into sweep testing follows a clear 4-step approach:

1.   **Tracing**

Run a model test with the operation tracer enabled via:

```
python model_tracer/generic_ops_tracer.py <test_path>
```

where `<test_path>` can be a pytest test (e.g., `models/tt_transformers/demo/simple_text_demo.py::test_demo_text`) or a standalone Python script [[model_tracer/generic_ops_tracer.py:13-25]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

2.   **Aggregation**

Individual trace JSON files created during execution are merged into `ttnn_operations_master.json` in the `model_tracer/traced_operations/` directory. This master file can be manually curated or auto-reconstructed through the database [[model_tracer/README.md:20-25]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

3.   **Vector Generation**

Run the sweep parameter generator pointing at the master JSON or database reconstructed manifest:

```
python tests/sweep_framework/sweeps_parameter_generator.py --model-traced all
```

This generates JSON vector files grouped with routing suffixes for appropriate hardware targeting [[tests/sweep_framework/README.md:34-38]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

4.   **Execution**

Launch sweep tests with:

```
python tests/sweep_framework/sweeps_runner.py --vector-source vectors_export --result-dest results_export
```

enabling validation and performance testing of real model operation parameters [[tests/sweep_framework/README.md:17-21]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction).

### Automated CI Sweeps

The CI pipeline triggers model-traced sweeps automatically according to this workflow:

*   Workflow file: `.github/workflows/ttnn-run-sweeps.yaml`
*   Triggered sweep: `ALL SWEEPS (Model Traced)` and `ALL SWEEPS (Lead Models)`
*   Scheduled run times: 
    *   Model Traced sweeps daily at 3 AM
    *   Lead Model sweeps daily at 2 AM [[.github/workflows/ttnn-run-sweeps.yaml:1-30]](https://deepwiki.com/tenstorrent/tt-metal/9.7-model-tracer-and-operation-extraction)

This guarantees continuous testing of real operation configurations from production models integrated with hardware runners.

# Summary

The **Model Tracer and Operation Extraction** subsystem of tt-metal automates the extraction of realistic operation parameters from model executions, enabling highly relevant and comprehensive sweep testing. By capturing rich metadata and integrating with the sweep framework's vector generation and execution pipelines, it ensures that operation validation and regression testing closely reflect production workloads on appropriate hardware platforms.
