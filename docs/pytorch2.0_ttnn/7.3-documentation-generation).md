## Purpose and Scope

The Documentation Generation system automatically transforms collected metrics from test runs into comprehensive, human-readable documentation. This system analyzes runtime performance data, operation conversion status, and input variations to generate the main README.md, operation reports, and model-specific documentation pages.

For information about how metrics are collected during test execution, see [Metrics Collection and Schema Tracking](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.1-metrics-collection-and-schema-tracking). For details on input variation analysis, see [Input Variation Analysis and Scoring](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.2-input-variation-analysis-and-conversion-status).

* * *

## Overview

The documentation generation system is centered on the `collect_metrics.py` script, which reads pickled metrics files, processes them through the `InputVarPerOp` analysis framework, and outputs multiple documentation artifacts. The system uses a template-based approach for the main README and dynamically generates operation-specific reports.

**Key Characteristics:**

*   Fully automated - runs as part of CI/CD workflows
*   Reads from standardized pickle file format
*   Generates both human-readable markdown and machine-readable JSON
*   Tracks operation conversion status across all tested models
*   Maintains cumulative statistics across the entire test suite

## Input Data Sources

The documentation generation system reads metrics from a standardized directory structure. Each model test generates multiple pickle files containing different aspects of the execution.

### Metrics Directory Structure

```
metrics/
├── {model_name}/
│   ├── original-run_time_metrics.pickle    # Native PyTorch execution metrics
│   ├── compiled-run_time_metrics.pickle    # Compiled backend execution metrics
│   ├── original-schema_list.pickle         # Operation schemas before compilation
│   ├── compiled-schema_list.pickle         # Operation schemas after compilation
│   └── model_run.json                      # Generated structured output
└── ...

metrics-autogen-op/
└── {model_name}/
    └── *.pickle                            # Individual operation test results
```

### Pickle File Contents

| File Type | Key Contents | Purpose |
| --- | --- | --- |
| `run_time_metrics.pickle` | `run_time`, `success`, `batch_size`, `accuracy` | Performance and validation metrics |
| `schema_list.pickle` | List of `InputVariation.dict()` objects | Operation schemas and input shapes |
| `single_metrics/*.pickle` | `opname`, `input_strings`, `run`, `accuracy`, `convert_to_ttnn` | Individual operation test results |

## The `InputVarPerOp` System

The `InputVarPerOp` class is the core data structure for tracking operation conversion status across different input variations. It maintains a two-level nested dictionary: operation name → input variation → conversion status.

### Class Hierarchy

**Sources:**[tools/collect_metrics.py 152-449](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L152-L449)

### Conversion Status Classification

The system classifies each input variation into one of six states:

| Status | Meaning | How Determined |
| --- | --- | --- |
| `DONE` | Successfully converted to TTNN | `opname` changed and new op doesn't start with "aten" |
| `FALLBACK` | Converted to another aten op | `opname` changed but new op still starts with "aten" |
| `BUG` | Known issue preventing conversion | Manually marked (future use) |
| `NONE` | No conversion attempted | `opname` and input variation unchanged in compiled graph |
| `UNKNOWN` | Not yet processed | Default state before analysis |
| `REMOVED` | Operation eliminated by optimization | Operation not present in compiled graph + successful run |

**Sources:**[tools/collect_metrics.py 152-159](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L152-L159)[tools/collect_metrics.py 237-283](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L237-L283)

### Processing Logic

The `InputVarPerOp.__init__()` method classifies each operation's conversion status by comparing original and compiled schemas.

**Sources:**[tools/collect_metrics.py 222-300](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L222-L300)

### Generality Score Calculation

Each operation receives a "generality score" representing the fraction of input variations successfully converted:

```
Generality Score = (DONE + REMOVED) / Total Input Variations
```

This score helps identify operations that work well for some inputs but fail for others.

**Sources:**[tools/collect_metrics.py 202-210](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L202-L210)

### Operations Report Structure

The Operations Report (`docs/OperationsReport.md`) contains a high-level summary table:

**Columns:**

*   **Operations**: Operation name (with link to detail page)
*   **Input Variations**: Total number of input variations tested
*   **Converted**: Count with `ConversionStatus.DONE`
*   **Removed**: Count with `ConversionStatus.REMOVED`
*   **Fallback**: Count with `ConversionStatus.FALLBACK`
*   **Completed**: ✅ if all variations converted, 🚧 if partial, ✘ if none
*   **Score**: Generality score (0.0 to 1.0)

**Sources:**[tools/collect_metrics.py 389-422](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L389-L422)

### Individual Operation Pages

Each `operations/{opname}.md` file contains a detailed table showing:

*   **ATen Input Variations**: The specific input schema (types and shapes)
*   **Status**: Conversion status (Done, Fallback, None, Removed, etc.)
*   **Isolated**: Single operation test result (Done, Failed, Unknown)
*   **PCC**: Pearson Correlation Coefficient for accuracy
*   **Host**: Number of fallbacks to host

**Sources:**[tools/collect_metrics.py 423-433](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L423-L433)[tools/collect_metrics.py 324-368](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L324-L368)

## Integration with CI/CD

The documentation generation runs as part of the CI/CD pipeline. Tests generate metrics pickle files, and `collect_metrics.py` processes them to update documentation.

### Workflow Integration

| Workflow File | Frequency | Trigger | Action |
| --- | --- | --- | --- |
| `run-tests.yaml` | On PR / Push | Test execution | Generates metrics pickle files via pytest fixtures |
| CI collect job | After test completion | Test success | Runs `python tools/collect_metrics.py` |
| Automated PR | After metrics collection | Commit changes | Creates PR with updated README.md and docs/ |

The metrics collection job runs after all test matrices complete, aggregating results from multiple test runs into unified documentation.

**Sources:**[.github/workflows/run-tests.yaml](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml) (referenced in Diagram 3 from high-level architecture)
