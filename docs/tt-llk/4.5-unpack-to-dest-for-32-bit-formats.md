## Purpose and Scope

This document explains the unpack-to-dest optimization available in the unpacker system when working with 32-bit data formats (Float32, Int32, UInt32). This optimization allows the unpacker to write data directly to destination registers, bypassing the intermediate source registers (SrcA/SrcB) that are normally used in the three-stage pipeline. This page covers the conditions under which this optimization is enabled, the synchronization mechanisms required, configuration specifics, and special handling for broadcast operations.

For general unpacker configuration and operations, see [Core Unpacker Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/4.1-core-unpacker-operations-and-configuration). For information about destination register management in the packer, see [Destination Register Management](https://deepwiki.com/tenstorrent/tt-llk/6.2-destination-register-management).

* * *

## Overview of Unpack-to-Dest Optimization

### Normal vs. Unpack-to-Dest Data Flow

In the standard LLK pipeline, data flows through three stages: L1 memory → Unpacker → Source Registers (SrcA/SrcB) → Math Unit → Destination Registers → Packer → L1 memory. For simple datacopy operations with 32-bit formats, this introduces unnecessary intermediate steps.

The unpack-to-dest optimization allows the unpacker to write directly to destination registers, eliminating the source register stage. This is particularly beneficial for datacopy operations where no actual computation is needed.

## 32-Bit Format Requirement

The unpack-to-dest optimization is only available when both the input format (from L1) and output format (written to destination) are 32-bit formats. This is validated by the `is_32bit_input()` helper function.

### Supported Format Combinations

| Input Format | Output Format | Unpack-to-Dest Support |
| --- | --- | --- |
| Float32 | Float32 | ✓ Yes |
| Int32 | Int32 | ✓ Yes |
| Int32 | Float32 | ✓ Yes |
| Float32 | Int32 | ✓ Yes |
| Float16 | Float32 | ✗ No |
| Float32 | Float16 | ✗ No |

## Configuration and MOP Programming

### MOP Configuration for Unpack-to-Dest

The Micro-Operation Programming (MOP) for unpack-to-dest mode differs significantly from normal unpacking. Instead of incrementing the Z counter in channel 0 only, it must increment both channel 0 and channel 1 Z counters to write to destination registers.

### MOP Instruction Patterns

Different patterns are used based on operation type and broadcast mode:

#### Standard Tile Unpack

```
static constexpr std::uint32_t unpack_srca_to_dest =
    TT_OP_UNPACR(SrcA, 0b00010001, 0, 0, 0, 1, 0, 
                 p_unpacr::RAREFYB_DISABLE, 0, 0, 0, 0, 1);
// ch0_z_inc=1, ch1_z_inc=1
```

#### Broadcast Row

```
constexpr std::uint32_t outerloop = 2;
constexpr std::uint32_t innerloop = 1;
ckernel_template tmp(outerloop, innerloop, unpack_srca_to_dest);
```

#### Broadcast Column

```
static constexpr std::uint32_t unpack_srca_to_dest_column =
    TT_OP_UNPACR(SrcA, 0b00100010, 0, 0, 0, 1, 0,
                 p_unpacr::RAREFYB_DISABLE, 0, 0, 0, 0, 1);
// Different Z increment pattern for column broadcast
```

## Configuration State Restoration

After completing unpack-to-dest operations, the unpacker configuration must be restored to its default state for normal operations:

### Restoration Steps

1.   **Restore Stride**: The Z-stride register is restored from the saved GPR value
2.   **Restore Context Configuration**: The `Unpack_if_sel_cntx` and `Dest_cntx_address` are set back to normal values
3.   **Re-enable Address Swizzle**: The `SRCA_SET_Base` register is set to `0x4` to re-enable the standard address bit swizzle pattern
4.   **Restore X-dimension**: The X counter end value is restored to the full face dimension

This restoration ensures that subsequent unpack operations that do not use unpack-to-dest will function correctly.

**Sources:**[tt_llk_wormhole_b0/common/inc/cunpack_common.h 419-440](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L419-L440)

## Summary

The unpack-to-dest optimization provides significant performance benefits for datacopy operations with 32-bit formats by eliminating intermediate register stages. Key points:

*   **Format Requirement**: Both input and output must be 32-bit formats (Float32, Int32, UInt32)
*   **Synchronization**: Requires explicit semaphore coordination (`UNPACK_TO_DEST`)
*   **MOP Configuration**: Uses special Z-increment pattern (`0b00010001`) to write to destination
*   **Broadcast Support**: Manual implementation required using MOVD2B/MOVB2D instructions
*   **Hardware Workaround**: Must execute one unpack-to-srcA instruction after last unpack-to-dest (TEN-3868)
*   **State Restoration**: Configuration must be carefully restored after unpack-to-dest operations

This optimization is most beneficial in datacopy scenarios where the only operation is moving data from one location to another without transformation, making it particularly useful for tensor reshape and layout operations on 32-bit data.
