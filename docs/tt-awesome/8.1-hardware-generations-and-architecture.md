This page serves as a technical reference for the Tenstorrent hardware generations and architectural components tracked within the `tt-awesome` ecosystem. It covers the evolution from first-generation Grayskull to the switch-free scaling of Blackhole and the multi-chip Galaxy configurations.

## Overview of Hardware Generations

Tenstorrent hardware is categorized into distinct generations, each identified in the `entries/*.json` data files via the `hardware` array field. These generations represent shifts in compute density, interconnect technology, and host integration.

| Generation | Key Models | Primary Interconnect | Notable Features |
| --- | --- | --- | --- |
| **Grayskull** | e75, e150 | PCIe Gen4 | First-gen, Tensix cores, no native Ethernet for chip-to-chip. |
| **Wormhole** | n150, n300 | Ethernet (100GbE) | Integrated 100GbE for scaling, GDDR6, ERISC processors. |
| **Blackhole** | p150, p300 | Ethernet (400GbE) | Switch-free scaling, SiFive x280 L2CPUs, improved NoC. |
| **Galaxy** | 32-chip systems | Ethernet Fabric | Large-scale multi-card configurations (e.g., 32-chip). |

## Architectural Components

Tenstorrent's "Tile-based" architecture revolves around a Network-on-Chip (NoC) that connects heterogeneous processors and memory controllers.

### Tensix Compute Cores

The Tensix core is the fundamental compute unit. Each core contains:

*   **Five RISC-V Processors**: Dedicated to data movement (TRISC) and compute orchestration.
*   **FPU/SFPU**: Floating point units and Special Function Floating Point Units for non-linear operations (e.g., Exp, Log, Reciprocal) [entries/riscv-arch/blog-anuraagw-blackhole-arch.json 4-5](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/entries/riscv-arch/blog-anuraagw-blackhole-arch.json#L4-L5)
*   **SRAM**: Local scratchpad memory for tile storage.
*   **MOP (Micro-Op Processor)**: Handles instruction sequencing and replay buffers [entries/riscv-arch/blog-anuraagw-blackhole-arch.json 17-18](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/entries/riscv-arch/blog-anuraagw-blackhole-arch.json#L17-L18)

### Network-on-Chip (NoC)

The NoC is a 2D-torus interconnect that allows any core to communicate with any other core, DRAM, or PCIe/Ethernet interfaces via memory-mapped I/O (MMIO).

*   **Data Flow**: Managed via `NCRISC` (NoC RISC) processors within the Tensix tile.
*   **Multicast**: Hardware support for sending data from one source to multiple destination cores simultaneously [src/_data/planet_feeds.json 64-65](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L64-L65)

### Specialized Processors

*   **ERISC (Ethernet RISC)**: Found in Wormhole and Blackhole, these processors manage the 100GbE/400GbE links for chip-to-chip communication [src/_data/planet_feeds.json 8-9](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L8-L9)
*   **ARC (Admin RISC)**: A system management processor that handles power states, thermal monitoring, and firmware messaging (e.g., `ARC_MSG_QCB_PTR`) [src/_data/planet_feeds.json 11-12](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L11-L12)
*   **SiFive x280**: Integrated into Blackhole as "L2CPUs" to handle complex control logic and scalar processing alongside the Tensix grid [entries/riscv-arch/blog-anuraagw-blackhole-arch.json 15-16](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/entries/riscv-arch/blog-anuraagw-blackhole-arch.json#L15-L16)

## System Configurations

### QuietBox (QB)

QuietBox refers to workstation-class systems designed for silent operation in office environments.

*   **QuietBox 2 (QB2)**: Typically utilizes Blackhole P300X2 or Wormhole N300 cards [src/_data/planet_feeds.json 22](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L22-L22)
*   **Stability**: Features like PCIe Gen4 and enhanced Ethernet firmware are critical for QB2 stability [src/_data/planet_feeds.json 78-80](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L78-L80)

### Galaxy & Multi-Chip Scaling

Galaxy systems represent the pinnacle of Tenstorrent's current scaling architecture, connecting up to 32 chips (e.g., `wh_x32` or `bh_x32`) in a unified fabric [src/_data/planet_feeds.json 64-65](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L64-L65)

*   **Switch-free**: Leveraging native Ethernet on-chip to avoid external InfiniBand or Ethernet switches for chip-to-chip traffic.
*   **Management**: Handled via `tt-smi` for reset logic (e.g., `RESET_PCIE_LINK`) and thermal tracking across the fabric [src/_data/planet_feeds.json 50-52](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L50-L52)

## Simulation: ttsim

For developers without physical hardware, `ttsim` provides a bit-exact, full-system simulator.

### ttsim Implementation

`ttsim` simulates the hardware at a functional level, allowing `TT-Metalium` workloads to run on x86_64 Linux systems [entries/riscv-arch/ttsim.json 4-5](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/entries/riscv-arch/ttsim.json#L4-L5)

**Key ttsim Capabilities:**

*   **Architecture Targets**: Supports `wh_x32` (Wormhole) and `bh_x32` (Blackhole) topologies [src/_data/planet_feeds.json 64-65](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L64-L65)
*   **Firmware Interaction**: Simulates ERISC and ARC behavior, including mailbox registers and outbound iATU (Internal Address Translation Unit) configurations [src/_data/planet_feeds.json 8-9](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L8-L9)
*   **Memory Management**: Removes "MMIO DMA hacks" in favor of realistic iATU and DMA simulation to match silicon behavior [src/_data/planet_feeds.json 12-14](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/src/_data/planet_feeds.json#L12-L14)

## Summary of Hardware Metadata in tt-awesome

The `tt-awesome` project tracks hardware compatibility using the `hardware` tag in entry JSON files. This allows the frontend to filter projects by their supported generation.

| Code Entity | Description |
| --- | --- |
| `hardware: ["grayskull"]` | Compatible with e75/e150 cards. |
| `hardware: ["wormhole"]` | Compatible with n150/n300/Galaxy. |
| `hardware: ["blackhole"]` | Compatible with p150/p300/QuietBox 2. |
| `hardware: ["ttsim"]` | Supports execution via the `ttsim` software simulator [entries/riscv-arch/ttsim.json 27-29](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/entries/riscv-arch/ttsim.json#L27-L29) |
