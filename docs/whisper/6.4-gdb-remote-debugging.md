## Purpose and Scope

This page documents Whisper's support for the GDB Remote Serial Protocol (RSP), which enables source-level debugging of RISC-V programs using a cross-compiled `riscv-gdb` debugger. This allows developers to set breakpoints, inspect registers and memory, single-step through code, and perform other standard debugging operations on RISC-V binaries running in Whisper.

For information about the server API protocol used for lock-step verification, see [Server API and Message Protocol](https://deepwiki.com/tenstorrent/whisper/6.1-server-api-and-message-protocol). For interactive command-line debugging, see [Interactive Command-Line Interface](https://deepwiki.com/tenstorrent/whisper/6.2-interactive-command-line-interface). For debug trigger configuration, see [Debug Triggers and Breakpoints](https://deepwiki.com/tenstorrent/whisper/8.1-debug-triggers-and-breakpoints).

* * *

## Overview

Whisper can operate as a GDB remote target by implementing the GDB Remote Serial Protocol. When started with the `--gdb` command-line option, Whisper communicates with a GDB client over standard input/output, translating GDB protocol packets into Whisper's internal peek, poke, step, and control operations. This enables full source-level debugging capabilities for RISC-V programs.

## GDB Remote Serial Protocol (RSP)

The GDB Remote Serial Protocol is a packet-based communication protocol that GDB uses to control remote debugging targets. The protocol defines commands for:

| Protocol Command | Purpose | Whisper Implementation |
| --- | --- | --- |
| `g` (read registers) | Read all general-purpose registers | Calls `Hart::peekIntReg()` for each register |
| `G` (write registers) | Write all general-purpose registers | Calls `Hart::pokeIntReg()` for each register |
| `m` (read memory) | Read memory at specified address | Calls `Hart::peekMemory()` |
| `M` (write memory) | Write memory at specified address | Calls `Hart::pokeMemory()` |
| `c` (continue) | Resume execution | Calls `Hart::run()` |
| `s` (step) | Single-step one instruction | Calls `Hart::singleStep()` |
| `Z0` (insert breakpoint) | Set software breakpoint | Uses trigger infrastructure |
| `z0` (remove breakpoint) | Remove software breakpoint | Clears trigger |
| `?` (halt reason) | Query why target stopped | Returns exception/interrupt information |

When Whisper is started with `--gdb`, it reads RSP packets from stdin and writes responses to stdout, enabling GDB to communicate with it through a pipe.

## Starting a GDB Debugging Session

### Basic Usage

To debug a RISC-V program using GDB and Whisper, follow these steps:

**Step 1:** Compile your RISC-V program with debug symbols:

`riscv32-unknown-elf-gcc -g -march=rv32imc -mabi=ilp32 -o myprogram myprogram.c`
**Step 2:** Start GDB with your binary:

`riscv32-unknown-elf-gdb myprogram`
**Step 3:** At the GDB prompt, connect to Whisper using a pipe:

```
(gdb) target remote | whisper --gdb myprogram
```

This command launches Whisper in GDB mode, where GDB and Whisper communicate through stdin/stdout. The `|` operator creates a bidirectional pipe between GDB and Whisper.

### Alternative Connection Methods

For more complex setups, you can use TCP sockets instead of pipes:

**Terminal 1 (Whisper):**

`whisper --gdb --server localhost:5555 myprogram`
**Terminal 2 (GDB):**

`riscv32-unknown-elf-gdb myprogram(gdb) target remote localhost:5555`

## Register and Memory Access

### Register Operations

When GDB requests register values, Whisper translates these to internal peek/poke operations:

The `peekCommand()` and `pokeCommand()` methods in the `Server` class handle register access:

*   **Integer registers** (`resource='r'`): Accesses via `Hart::peekIntReg()` / `Hart::pokeIntReg()`
*   **Floating-point registers** (`resource='f'`): Accesses via `Hart::peekFpReg()` / `Hart::pokeFpReg()`
*   **CSRs** (`resource='c'`): Accesses via `Hart::peekCsr()` / `Hart::externalPokeCsr()`
*   **Program counter** (`resource='p'`): Accesses via `Hart::peekPc()` / `Hart::pokePc()`

**Sources:**[Server.cpp 286-439](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L286-L439)[Server.cpp 143-281](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L143-L281)

### Memory Operations

Memory reads and writes are handled similarly:

| `m addr,len` | `Hart::peekMemory(addr, value, usePma=false)` | [Server.cpp 354-359](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L354-L359) |

## Execution Control

### Single-Stepping

The GDB `step` (`s`) and `stepi` (`si`) commands map to Whisper's `stepCommand()`:

The `stepCommand()` method:

1.   Calls `Hart::singleStep()` to execute one instruction
2.   Collects register and memory changes via `processStepChanges()`
3.   Returns changed resources to GDB
4.   Reports if triggers were hit

## Breakpoints and Watchpoints

GDB breakpoints are implemented using Whisper's hardware trigger system.

### Software Breakpoints

When GDB sets a breakpoint using `Z0,addr`, Whisper configures a trigger:

The trigger configuration for instruction breakpoints:

*   **Trigger type**: `Mcontrol` or `Mcontrol6`
*   **tdata1.execute**: Set to 1 (trigger on instruction execution)
*   **tdata1.action**: `TriggerAction::EnterDebug` to halt on hit
*   **tdata2**: Target address
*   **Timing**: `TriggerTiming::Before` (trigger before instruction executes)

**Sources:**[Triggers.hpp 27-42](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L27-L42)[Triggers.cpp 283-309](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L283-L309)

### Hardware Watchpoints

Data watchpoints are implemented using address/data match triggers:

| Watchpoint Type | Trigger Configuration |
| --- | --- |
| Read watchpoint | `tdata1.load = 1` |
| Write watchpoint | `tdata1.store = 1` |
| Access watchpoint | `tdata1.load = 1, tdata1.store = 1` |

## Trigger System Integration

The `Triggers` class manages multiple hardware triggers that can be used for breakpoints:

| Class | Purpose | Key Methods |
| --- | --- | --- |
| `Triggers<URV>` | Manages collection of triggers | `instAddrTriggerHit()`, `ldStAddrTriggerHit()` |
| `Trigger<URV>` | Individual trigger configuration | `matchInstAddr()`, `matchLdStAddr()`, `writeData1()`, `writeData2()` |
| `TriggerType` | Trigger type enumeration | `Mcontrol`, `Mcontrol6`, `Icount`, `Etrigger`, `Itrigger` |
| `TriggerAction` | Action on trigger hit | `RaiseBreak`, `EnterDebug` |

### Trigger Configuration

Triggers are configured through CSR-like interfaces:

*   **TSELECT**: Selects which trigger to configure
*   **TDATA1**: Control register (type, action, match conditions)
*   **TDATA2**: Match value (address or data)
*   **TDATA3**: Additional configuration (context)

## GDB Command Translation Table

The following table shows how common GDB commands map to Whisper operations:

| GDB Command | Purpose | Whisper Method | WhisperMessage Type |
| --- | --- | --- | --- |
| `info registers` | Display all registers | `peekCommand(resource='r')` | `Peek` |
| `print $pc` | Show program counter | `Hart::peekPc()` | `Peek` |
| `x/4wx 0x1000` | Examine memory | `Hart::peekMemory()` | `Peek` |
| `set $x1=0x42` | Set register | `Hart::pokeIntReg()` | `Poke` |
| `set {int}0x1000=100` | Write memory | `Hart::pokeMemory()` | `Poke` |
| `stepi` | Single step | `Hart::singleStep()` | `Step` |
| `continue` | Resume execution | `Hart::run()` | - |
| `break *0x1000` | Set breakpoint | Configure trigger | `Poke` CSR |
| `watch *0x2000` | Set watchpoint | Configure data trigger | `Poke` CSR |
| `info break` | List breakpoints | Read trigger registers | `Peek` CSR |

## Limitations and Considerations

### Supported Features

*   ✅ Register read/write (integer, FP, CSR)
*   ✅ Memory read/write
*   ✅ Single-stepping
*   ✅ Continue execution
*   ✅ Software breakpoints
*   ✅ Hardware watchpoints (via triggers)
*   ✅ Source-level debugging (with debug symbols)
*   ✅ Backtrace/stack unwinding

### Limitations

*   ⚠️ **Limited number of breakpoints**: Whisper supports a finite number of hardware triggers (typically 4-16), which limits simultaneous breakpoints
*   ⚠️ **No software breakpoint memory patching**: Unlike some debuggers, Whisper doesn't patch instructions with breakpoint instructions
*   ⚠️ **Trigger resource sharing**: Breakpoints and watchpoints share the same trigger resources
*   ⚠️ **No multi-hart debugging coordination**: Each hart must be debugged independently
*   ⚠️ **Performance**: Single-stepping is slower than native execution due to trigger checking overhead

### Trigger Availability

To check available triggers:

```
(gdb) p/x $tdata1
(gdb) info registers tselect tdata1 tdata2
```

The number of triggers is configured via the JSON configuration file.

## Summary

GDB remote debugging in Whisper provides a powerful interface for source-level debugging of RISC-V programs. The implementation leverages Whisper's existing `Server` infrastructure and `Triggers` system to translate GDB Remote Serial Protocol commands into native Whisper operations. This enables standard GDB workflows including breakpoints, watchpoints, register/memory inspection, and single-stepping.
