This page documents the operational skills required for working with Tenstorrent hardware within the TT-Forge ecosystem. It covers remote device connectivity, performance profiling, the workflow for exporting TT-Lang kernels to TT-Metal C++, and procedures for tracing and bug reporting.

## 1. Remote Device Connectivity

The `tt-connect-remote-device` skill provides a standardized set of scripts for interacting with Tenstorrent hardware located on remote servers. This infrastructure allows developers to run kernels in either a functional simulator or on real hardware via an SSH and Docker-based chain.

### 1.1 Implementation and Data Flow

The remote connection is managed through a configuration file `remote.conf` and a shared library `_lib.sh`[skills/tt-connect-remote-device/scripts/_lib.sh 1-61](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-connect-remote-device/scripts/_lib.sh#L1-L61) The library implements `remote_run`, `remote_copy_file`, and `remote_copy_from` functions that abstract the underlying `ssh`, `scp`, and `docker exec` commands.

**Remote Connection Data Flow** "Local Workspace" -> "remote_copy_file" -> "Remote Host (/tmp)" -> "docker cp" -> "Remote Container" "Remote Container" -> "remote_run" -> "Execution Output" -> "Local Terminal/Log"

## 2. Performance Profiling and Optimization

Profiling on Tenstorrent hardware involves a multi-layered approach using built-in device profilers and TT-Lang specific instrumentation.

### 2.1 Optimization Workflow

The optimization process follows a strict iteration flow:

1.   **Baseline**: Run with `TT_METAL_DEVICE_PROFILER=1` and `TTLANG_PERF_DUMP=1` to establish wall time [skills/tt-lang-profile-optimize/SKILL.md 50-67](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-profile-optimize/SKILL.md?plain=1#L50-L67)
2.   **Bottleneck Identification**: Analyze core utilization, DRAM traffic, and DFB (Dataflow Buffer) block sizes [skills/tt-lang-profile-optimize/SKILL.md 68-76](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-profile-optimize/SKILL.md?plain=1#L68-L76)
3.   **Implementation**: Make one change at a time and measure wall time impact [skills/tt-lang-profile-optimize/SKILL.md 87-97](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-profile-optimize/SKILL.md?plain=1#L87-L97)

### 2.2 Auto-Profiling and Signposts

TT-Lang supports line-by-line cycle profiling via `TTLANG_AUTO_PROFILE=1`. This instruments the kernel with signposts to report hotspots [skills/tt-lang-profile-optimize/performance-tools.md 81-102](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-profile-optimize/performance-tools.md?plain=1#L81-L102) Alternatively, `ttl.signpost("name")` can be used as a context manager for targeted measurement [skills/tt-lang-profile-optimize/performance-tools.md 130-144](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-profile-optimize/performance-tools.md?plain=1#L130-L144)

## 3. TT-Lang to TT-Metal Export

The export workflow transforms high-level TT-Lang Python kernels into standalone TT-Metal C++ code. This is essential for moving from rapid prototyping to production-ready low-level kernels.

### 3.1 Export Process

1.   **Emission**: Run the TT-Lang script with `TTLANG_EMIT_RUNNER=1`[skills/tt-lang-to-tt-metal/SKILL.md 41-48](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L41-L48)
2.   **Artifact Generation**: The compiler generates four primary files in `/tmp/$USER/`: 
    *   `ttlang_kernel_compute_<hash>.cpp` (Compute kernel)
    *   `ttlang_kernel_dm_read_<hash>.cpp` (Reader)
    *   `ttlang_kernel_dm_write_<hash>.cpp` (Writer)
    *   `ttlang_kernel_compute_<hash>_runner.py` (Python runner using `ttnn.generic_op`) [skills/tt-lang-to-tt-metal/SKILL.md 50-55](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L50-L55)

3.   **Adaptation**: Update the generated runner to match the original program's `main()` setup and grid dimensions [skills/tt-lang-to-tt-metal/SKILL.md 84-103](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L84-L103)

### 3.2 Safe Transformations (Beautification)

While logic changes are forbidden, developers should perform "P2 Beautification":

*   Rename generated variables (e.g., `v1` to `input_lhs`) [skills/tt-lang-to-tt-metal/SKILL.md 122-131](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L122-L131)
*   Collapse redundant casts from `get_compile_time_arg_val`[skills/tt-lang-to-tt-metal/SKILL.md 135-142](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L135-L142)
*   Define `constexpr` for Circular Buffer (CB) indices [skills/tt-lang-to-tt-metal/SKILL.md 144-157](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-lang-to-tt-metal/SKILL.md?plain=1#L144-L157)

## 4. TTNN Tracing

Trace capture is a performance optimization that records a sequence of TTNN operations to eliminate host dispatch overhead. It is critical for real-time inference.

### 4.1 Implementation Rules

To ensure a valid trace replay, the following constraints must be met:

1.   **Pure Device Work**: Remove all `ttnn.from_torch` and `ttnn.to_torch` calls from the traced region [skills/tt-enable-tracing/SKILL.md 30-32](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L30-L32)
2.   **No Host Logic**: Remove all Python conditionals or CPU-side tensor creation from the capture block [skills/tt-enable-tracing/SKILL.md 33](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L33-L33)
3.   **Pre-allocation**: All tensors must be pre-allocated with fixed addresses using `ttnn.allocate_tensor_on_device`[skills/tt-enable-tracing/SKILL.md 34-36](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L34-L36)[skills/tt-enable-tracing/advanced_perf_optimizations.md 78-83](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/advanced_perf_optimizations.md?plain=1#L78-L83)

### 4.2 API Sequence

1.   `ttnn.open_device(..., trace_region_size=...)`: Reserves DRAM for trace buffers [skills/tt-enable-tracing/SKILL.md 16-20](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L16-L20)
2.   `ttnn.begin_trace_capture(device, cq_id)`: Starts recording [skills/tt-enable-tracing/SKILL.md 46](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L46-L46)
3.   `ttnn.end_trace_capture(device, tid, cq_id)`: Stops recording [skills/tt-enable-tracing/SKILL.md 49](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L49-L49)
4.   `ttnn.execute_trace(device, tid, cq_id)`: Replays the sequence [skills/tt-enable-tracing/SKILL.md 55](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-enable-tracing/SKILL.md?plain=1#L55-L55)

## 5. Bug Reporting and Repro-Scripts

When hardware or compiler issues are identified, they must be reported to the correct repository with specific context.

### 5.1 Component Mapping

| Component | Repository |
| --- | --- |
| TT-Lang (DSL, MLIR Compiler) | `tenstorrent/tt-lang` |
| TTNN / TT-Metal (Ops, Runtime) | `tenstorrent/tt-metal` |
| TT-XLA (Frontend) | `tenstorrent/tt-xla` |

### 5.2 Reporting Requirements

For compiler errors, reports must include:

*   Initial MLIR: `/tmp/ttlang_initial.mlir`[skills/tt-bug-report/SKILL.md 39](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/tt-bug-report/SKILL.md?plain=1#L39-L39)
