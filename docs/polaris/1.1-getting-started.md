## Prerequisites and Environment Setup

### System Requirements

Polaris requires Python 3.13 or higher and uses the Miniforge package manager for dependency management. The system supports both user and developer environments depending on your intended usage.

**Sources:**[README.md 42-47](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/README.md?plain=1#L42-L47)[envdev.yaml 7](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/envdev.yaml#L7-L7)

### Installing Miniforge

First, download and install Miniforge to manage the Python environment:

`curl -L -O "https://github.com/conda-forge/miniforge/releases/latest/download/Miniforge3-$(uname)-$(uname -m).sh"bash Miniforge3-$(uname)-$(uname -m).sh"conda update -n base -c conda-forge conda`
**Sources:**[README.md 50-58](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/README.md?plain=1#L50-L58)

### Development Environment Setup

For development work, create the developer environment which includes additional tools for code quality, testing, and compliance:

`conda env create --file envdev.yamlconda activate polarisdevpre-commit install`
The development environment includes essential dependencies such as:

| Component | Version | Purpose |

| `python` | 3.13.2 | Core runtime |
| `pydantic` | 2.10.6 | Configuration validation |
| `loguru` | latest | Logging framework |
| `pytest` | 8.3.4 | Testing framework |
| `mypy` | 1.15.0 | Static type checking |
| `pre-commit` | latest | Code quality hooks |

## Running Your First Simulation

### Basic Command Structure

The primary entry point for running simulations is `polaris.py` with three required configuration files:

`python polaris.py --archspec <architecture_config> --wlspec <workload_spec> --wlmapspec <mapping_spec>`
### Essential Command Line Options

| Option | Short | Description |
| --- | --- | --- |
| `--archspec` | `-a` | Architecture specification YAML file |
| `--wlspec` | `-w` | Workload specification YAML file |
| `--wlmapspec` | `-m` | Workload mapping specification YAML file |
| `--study` | `-s` | Simulation study name (default: "study") |
| `--odir` | `-o` | Output directory (default: ".") |
| `--outputformat` |  | Output format (none/yaml/json/pickle) |
| `--dryrun` | `-n` | Validate configuration without execution |

**Sources:**[README.md 84-101](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/README.md?plain=1#L84-L101)

### Configuration File Structure

The simulation requires three types of configuration files:

1.   **Architecture Specification (`archspec`)**: Defines hardware configuration including device specifications, memory hierarchy, and compute resources
2.   **Workload Specification (`wlspec`)**: Defines AI model configurations, batch sizes, and operator specifications
3.   **Workload Mapping Specification (`wlmapspec`)**: Maps operators to datatypes and defines resource requirements

**Sources:**[README.md 108-130](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/README.md?plain=1#L108-L130)

### Example Simulation Run

To validate your setup, start with a dry run to check configuration validity:

`python polaris.py --dryrun --archspec configs/all_archs.yaml --wlspec configs/all_workloads.yaml --wlmapspec configs/wl2archmapping.yaml`
For a complete simulation with CSV output:

`python polaris.py --archspec configs/all_archs.yaml --wlspec configs/all_workloads.yaml --wlmapspec configs/wl2archmapping.yaml --dump_stats_csv --outputformat yaml`
**Sources:**[README.md 159-168](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/README.md?plain=1#L159-L168)

## Output Structure and Analysis

### Generated Output Directory

Polaris creates a structured output directory containing simulation results:

```
output_dir/
├── study_name/
│   ├── SUMMARY/
│   │   └── study-summary.csv
│   └── stats/
│       ├── device-workload-stats.csv
│       └── device-workload-opstats.[yaml|json|pickle]
```

### Key Performance Metrics

The simulation generates various performance metrics including:

*   Execution cycles and time
*   Memory usage and requirements
*   Resource bottlenecks
*   Input/output parameter counts
*   Resource utilization

## Development Tools and Code Quality

### Pre-commit Hooks

The development environment includes automated code quality checks:

`pre-commit install`
This enables automatic execution of:

*   SPDX license and copyright checking via `spdxchecker.py`
*   Static type analysis with `mypy`
*   Unit test execution with `pytest`

**Sources:**[.pre-commit-config.yaml 11-30](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/.pre-commit-config.yaml#L11-L30)[tools/spdxchecker.py 28-37](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L28-L37)

### SPDX Compliance

All source files must include proper SPDX headers. The system automatically validates:

*   License identifier: `Apache-2.0`
*   Copyright holder: `Tenstorrent AI ULC`

Files can be excluded from checking via the ignore configuration in `.github/spdxchecker-ignore.yml`.

**Sources:**[.github/spdxchecker-ignore.yml 3-16](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/.github/spdxchecker-ignore.yml#L3-L16)[tools/spdxchecker.py 61-67](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L61-L67)
