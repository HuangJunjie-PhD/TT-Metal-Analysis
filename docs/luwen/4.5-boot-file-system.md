## Introduction

The Boot File System (Boot FS) is a specialized file system interface that provides access to chip firmware and configuration data stored in the SPI flash memory of Tenstorrent AI accelerator chips. It is primarily used for reading firmware images, board configurations, and other critical system data required during chip initialization and operation.

Boot FS is predominantly used in Blackhole chip implementations, providing a structured way to access tagged data blocks within the SPI ROM.

Sources: [crates/luwen-if/src/chip/blackhole/boot_fs.rs](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/blackhole/boot_fs.rs)

## Boot File System Architecture

The Boot File System uses a simple but effective structure to organize data in the SPI flash memory:

The Boot FS consists of a series of file descriptors stored sequentially in memory, each containing metadata about a tagged data block. The system scans these descriptors to locate specific entries based on their tags.

Sources: [crates/luwen-if/src/chip/blackhole/boot_fs.rs 4-143](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/blackhole/boot_fs.rs#L4-L143)[tests/boot_fs_test.rs 8-13](https://github.com/tenstorrent/luwen/blob/55e35616/tests/boot_fs_test.rs#L8-L13)

## File Descriptor Structure

Each file in the Boot FS is represented by a `TtBootFsFd` structure that contains metadata about the file:

The key fields of a file descriptor include:

*   `spi_addr`: Physical address in SPI flash where the file data is stored
*   `copy_dest`: Destination address where the data should be copied during boot
*   `flags`: Contains the file size, validity flag, and executable flag
*   `image_tag`: 8-byte identifier string for the file (e.g., "boardcfg")
*   CRC values for data integrity verification

Sources: [crates/luwen-if/src/chip/blackhole/boot_fs.rs 4-116](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/blackhole/boot_fs.rs#L4-L116)

## Common Boot FS Entries

The Boot FS contains several standard entries that are used during chip initialization and operation:

| Tag Name | Description | Purpose |
| --- | --- | --- |
| boardcfg | Board Configuration | Contains board-specific settings and parameters |
| flshinfo | Flash Information | Provides details about the SPI flash memory configuration |
| cmfwcfg | Firmware Configuration | Contains configuration data for the chip firmware |
| origcfg | Original Configuration | Stores the original factory configuration (may not be present on all chips) |

These entries can be accessed using the `get_boot_fs_tables_spi_read` method in the Blackhole chip implementation.

Sources: [tests/boot_fs_test.rs 22-96](https://github.com/tenstorrent/luwen/blob/55e35616/tests/boot_fs_test.rs#L22-L96)

## Accessing the Boot File System

The Boot FS provides two primary functions for accessing data:

1.   `read_fd`: Reads a file descriptor from a specific address in SPI memory
2.   `read_tag`: Scans the Boot FS sequentially to find a file descriptor with a specific tag

## Protobuf Data Structures

The Boot FS entries often contain structured data defined using Protocol Buffers. Three main protobuf schemas are used for parsing Boot FS data:

1.   `fw_table`: Firmware table schema
2.   `flash_info`: Flash memory information schema
3.   `read_only`: Read-only configuration data schema

These protobuf definitions are compiled during the build process and included in the codebase, allowing structured access to the data contained in Boot FS entries.

## Integration with Chip Architecture

The Boot File System primarily serves the Blackhole chip implementation, but its concepts are applicable to other chip architectures in the Luwen project:

The Boot FS functionality is implemented specifically for the Blackhole chip architecture through the `get_boot_fs_tables_spi_read` method. This method allows reading tagged data from the SPI ROM using the Boot FS structure.
