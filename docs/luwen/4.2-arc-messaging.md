## Purpose and Scope

ARC Messaging is the communication protocol used to interact with the ARC (Auxiliary Risc Controller) processors embedded within Tenstorrent AI accelerator chips. This document explains the architecture, components, and operation of the ARC Messaging system in Luwen. For information about memory access mechanisms like AXI and NOC, see [Memory Access](https://deepwiki.com/tenstorrent/luwen/4.3-memory-access). For information about telemetry data that can be retrieved using ARC Messaging, see [Telemetry](https://deepwiki.com/tenstorrent/luwen/4.4-telemetry).

## Overview

ARC Messaging provides a standardized way to send commands to and receive responses from the ARC processor on Tenstorrent chips. These messages enable various operations such as:

*   Querying chip state and firmware versions
*   Controlling power states
*   Reading/writing SPI flash memory
*   Triggering resets
*   Collecting telemetry data

The system abstracts the low-level details of chip communication, offering a consistent interface across different chip architectures (Wormhole, Grayskull, and Blackhole).

## Message Types and Operation

ARC messages are categorized into several functional groups:

### Control Messages

*   `Nop`: No operation
*   `ArcGoToSleep`: Put the ARC processor into sleep mode
*   `SetPowerState`: Configure power state (Busy, ShortIdle, LongIdle)
*   `TriggerReset`: Trigger a system reset
*   `SetArcState`: Set the ARC to a specific state (A0, A1, A3, A5)

### Hardware Management

*   `ResetSafeClks`: Reset safe clocks with given argument
*   `ToggleTensixReset`: Toggle reset for Tensix cores
*   `DeassertRiscVReset`: Deassert reset on RISC-V
*   `GetAiclk`: Get AI clock frequency

### Information Retrieval

*   `FwVersion`: Get firmware version for the specified type
*   `GetSmbusTelemetryAddr`: Get SMBus telemetry address
*   `GetHarvesting`: Get harvesting information
*   `Test`: Test functionality with a given argument

### SPI Flash Operations

*   `TriggerSpiCopyLtoR`: Trigger SPI copy from left to right
*   `GetSpiDumpAddr`: Get SPI dump address
*   `SpiRead`: Read data from SPI at specified address
*   `SpiWrite`: Write data to SPI

## Messaging Protocol

The diagram above illustrates the ARC messaging protocol flow. The key steps are:

1.   The host system sends a command through the `ChipImpl` interface
2.   The `arc_msg` function converts the message to a code and arguments
3.   It writes the message code to the message register and arguments to the return register
4.   It triggers a firmware interrupt to notify the ARC processor
5.   If waiting for completion is requested, it polls the status register until completion or timeout
6.   On completion, it reads the response from the return register and returns it

The messaging function accepts parameters including:

*   The high-level communications object (`comms`)
*   The message to send (`msg`)
*   Whether to wait for completion (`wait_for_done`)
*   Timeout duration (`timeout`)
*   Register addresses (`msg_reg`, `return_reg`, `addrs`)

## Address Structure

The `ArcMsgAddr` struct defines the memory addresses used for ARC messaging:

```
struct ArcMsgAddr {
    scratch_base: u64,   // Base address of scratch registers
    arc_misc_cntl: u64,  // Address of ARC misc control register
}
```

These addresses are chip-specific and are initialized when the chip is detected and initialized.

## Error Handling

The ARC Messaging system provides comprehensive error handling through several error types:

1.   `ArcMsgProtocolError`: Low-level protocol errors such as:

    *   Unrecognized messages
    *   Timeouts waiting for ARC response
    *   ARC being asleep
    *   Failed firmware interrupts
    *   Invalid mailbox selections
    *   Unknown error codes

2.   `ArcMsgError`: Wraps protocol errors with backtraces and also handles AXI errors

3.   `ArcReadyError`: Indicates issues that prevent ARC communication:

    *   No register access
    *   Watchdog triggers
    *   Incomplete boots
    *   ARC being asleep
    *   Outstanding PCIe DMA requests
    *   Messages already queued or being handled
    *   Invalid post codes

The error types use Rust's `thiserror` library to provide detailed error messages and maintain backtraces for debugging.

## Blackhole Chip Message Queue Implementation

For Blackhole chips, ARC messaging is implemented using a message queue system:

The `MessageQueue` implementation provides:

1.   A queue-based system for request/response communication
2.   Methods to push requests, pop responses, and send messages
3.   Queue status reporting through `get_queue_info`
4.   Error handling for timeouts and out-of-range indices

The queue system maintains separate request and response queues with their own read and write pointers.

## Integration with Chip Implementations

Each chip implementation (Wormhole, Grayskull, Blackhole) uses ARC Messaging through its `ChipImpl` interface. The interface defines an `arc_msg` method that each implementation must provide.

The specific details of ARC Messaging may vary between chip architectures, but the interface ensures consistent behavior from the caller's perspective.

## Best Practices

When using ARC Messaging:

1.   **Handle timeouts appropriately**: ARC messages may take time to process, so set reasonable timeouts based on the operation.

2.   **Check for ARC readiness**: Before sending messages, verify that the ARC is ready to avoid errors.

3.   **Use typed messages when possible**: Prefer `TypedArcMsg` over raw messages for better type safety and code clarity.

4.   **Handle errors robustly**: ARC messaging can fail for various reasons, so implement proper error handling.

5.   **Minimize messaging frequency**: ARC messaging involves hardware interactions, which can be relatively slow. Avoid unnecessary messages and batch operations when possible.
