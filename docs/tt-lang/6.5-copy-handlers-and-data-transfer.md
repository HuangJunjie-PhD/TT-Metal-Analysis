## Purpose and Scope

This document describes the copy handler system used by the tt-lang simulator to transfer data between different memory types and storage locations. The copy handler architecture uses a registry-based strategy pattern to support extensible data transfer operations between Tensors, Dataflow Buffers (via Blocks), and inter-core Pipes.

The simulation framework models these transfers as `CopyTransaction` objects that follow a strict `can_wait` protocol to enable cooperative scheduling and deadlock detection within the `GreenletScheduler`.

## Copy Handler Architecture

The copy handler system implements a registry-based strategy pattern where each handler supports transfers between a specific (source_type, destination_type) pair. This architecture enables adding new transfer types without modifying existing core logic.

### Handler Registry Pattern

The `HANDLER_REGISTRY` is a static lookup table populated at import time via `@register_copy_handler` decorators.

## Core Handler Implementations

### Tensor ↔ Block Handlers

These handlers manage data movement between `ttnnsim.Tensor` (DRAM/L1 storage) and `Block` objects (L1 staging buffers within a `DataflowBuffer`).

#### TensorToBlockHandler Implementation

This handler validates that the total tile count in the source tensor matches the capacity of the destination block and that layouts are compatible.

*   **Validation**: [python/sim/copyhandlers.py 212-230](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L212-L230)
    *   Checks for layout mismatch (e.g., TILE vs ROW_MAJOR).
    *   Uses `tile_count_from_tensor` to determine source tiles.
    *   Compares against block shape via `check_count_match`.

*   **Transfer**: [python/sim/copyhandlers.py 255-272](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L255-L272)
    *   Iterates through tiles in row-major order.
    *   Extracts slices from the source tensor.
    *   Stores them into the block using `dst[tile_idx] = tile`.

#### BlockToTensorHandler Implementation

The reverse operation extracts data from L1 blocks and places it back into the destination tensor.

*   **Transfer**: [python/sim/copyhandlers.py 297-315](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L297-L315)
    *   Iterates through block tiles.
    *   Retrieves tile via `src[tile_idx]`.
    *   Assigns to tensor slice using calculated coordinates.

Sources: [python/sim/copyhandlers.py 209-230](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L209-L230)[python/sim/copyhandlers.py 255-272](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L255-L272)[python/sim/copyhandlers.py 275-318](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L275-L318)

### PipeToBlockHandler (Pipe Receive)

Receiving from a pipe is a blocking operation if data is not yet available.

*   **can_wait Protocol**: [python/sim/copyhandlers.py 329-338](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L329-L338)
    *   Returns `True` if the pipe's queue is not empty.
    *   Returns `False` otherwise, triggering a scheduler yield.

*   **Transfer**: [python/sim/copyhandlers.py 381-408](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L381-L408)
    *   Retrieves the message from the head of the queue.
    *   Records the current core in the receiver set to prevent double-consumption by the same core.
    *   Decrements the remaining `num_receivers`.
    *   If `num_receivers` reaches 0, the message is popped from the queue via `entry["queue"].popleft()`.

## CopyTransaction and Synchronization

The `CopyTransaction` class manages the lifecycle of a data transfer. It interacts with the `Block` state machine to lock resources during the transfer.

### State Machine Integration

When a `CopyTransaction` is initialized, it immediately transitions any involved blocks into "in-flight" states to prevent illegal concurrent access.

*   **Source Locking**: [python/sim/copy.py 99-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L99-L103) calls `src.mark_copy_as_source()`.
*   **Destination Locking**: [python/sim/copy.py 104-108](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L104-L108) calls `dst.mark_copy_as_dest()`.
*   **Wait Completion**: [python/sim/copy.py 180-191](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L180-L191) calls `mark_tx_wait_complete()`, releasing the locks after the transfer is finished.

Sources: [python/sim/copy.py 73-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L73-L118)[python/sim/copy.py 154-197](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L154-L197)

### can_wait Protocol and Scheduling

The `wait()` method on a transaction is a blocking call that integrates with the cooperative scheduler.

`# python/sim/copy.py:170-172from .greenlet_scheduler import block_if_needed block_if_needed(self, "wait")`
1.   `wait()` calls `block_if_needed(self, "wait")`.
2.   `block_if_needed` checks `tx.can_wait()`.
3.   If `can_wait()` is `False` (e.g., Pipe is empty), the scheduler yields control.
4.   The transfer is performed immediately after the block is lifted via `self._handler.transfer(self._src, self._dst)`.

Sources: [python/sim/copy.py 154-179](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L154-L179)[python/sim/copy.py 199-203](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L199-L203)

## Group Transfers

The `GroupTransfer` class allows multiple `CopyTransaction` objects to be managed and synchronized together. This is useful for complex data movement patterns like upsampling or multicasting to multiple destinations.

*   **Usage**: [python/sim/copy.py 205-213](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L205-L213)
    *   Create a `GroupTransfer` instance.
    *   Add transactions using `add(tx)`.
    *   Call `wait_all()` to synchronize on all transfers.

## Summary of Transfer Patterns

| Pattern | Source | Destination | Handler | can_wait Behavior |
| --- | --- | --- | --- | --- |
| Load | `Tensor` | `Block` | `TensorToBlockHandler` | Always `True` |
| Store | `Block` | `Tensor` | `BlockToTensorHandler` | Always `True` |
| Send | `Block` | `Pipe` | `BlockToPipeHandler` | Always `True` |
| Receive | `Pipe` | `Block` | `PipeToBlockHandler` | `True` if queue has data |
