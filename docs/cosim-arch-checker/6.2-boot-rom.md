## Purpose and Scope

The Boot ROM component provides the initial system initialization code for RISC-V multi-hart processors in the cosim-arch-checker verification environment. This documentation covers the boot ROM implementation, its role in system startup, and how it integrates with the Chipyard/Boom processor environment.

For information about the broader integration with Chipyard, see [Chipyard Integration](https://deepwiki.com/tenstorrent/cosim-arch-checker/6.1-chipyard-integration). For details about the overall system architecture, see [System Architecture](https://deepwiki.com/tenstorrent/cosim-arch-checker/2-system-architecture).

## Overview

The boot ROM is a small piece of assembly code that handles the initial system boot process for multi-hart RISC-V systems. It coordinates the startup of multiple hardware threads (harts), establishes interrupt vectors, and transfers control to the main program loaded into system memory.

The boot ROM serves as the first code executed after system reset and is responsible for:

*   Initializing hart 0 as the primary boot hart
*   Coordinating the startup of secondary harts via inter-processor interrupts
*   Setting up interrupt vectors and machine-mode CSRs
*   Loading the target program address and transferring execution control

## Key Components

### Reset Vector Handler (`_hang`)

The `_hang` function serves as the initial reset vector that all harts execute upon system reset:

The reset vector performs these initialization steps:

*   Sets `mtvec` CSR to point to `_start` for interrupt handling
*   Clears interrupt delegation (`mideleg`) if supervisor mode is supported
*   Enables machine software interrupts (MSIP) in `mie` and `mstatus` CSRs
*   Enters a wait-for-interrupt loop until boot is triggered

Sources: [bootrom/bootrom.S 22-39](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bootrom/bootrom.S#L22-L39)

### Primary Boot Handler (`_start`)

The `_start` function handles the actual boot process, with different paths for hart 0 and secondary harts:

The boot process implements these key behaviors:

*   Hart 0 sends MSIP interrupts to wake up all other harts
*   Secondary harts wait for hart 0 to clear its MSIP before proceeding
*   All harts load the program entry point from the boot address register
*   Each hart clears its own MSIP and jumps to the loaded program

Sources: [bootrom/bootrom.S 6-18](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bootrom/bootrom.S#L6-L18)[bootrom/bootrom.S 42-60](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bootrom/bootrom.S#L42-L60)

## Memory Layout and Key Addresses

The boot ROM operates with a specific memory layout that integrates with the broader system:

| Address | Symbol | Purpose |
| --- | --- | --- |
| `0x10000` | `_start` | Primary boot entry point |
| `0x10040` | `_hang` | Reset vector handler |
| `0x2000000` | CLINT base | Core Local Interrupt controller |
| `0x4000` | `BOOTADDR_REG` | Boot address register |
| `0x100B0` | `_dtb` | Device tree blob location |

The boot address register at `0x4000` contains the entry point address for the main program, typically `0x80000000` for DRAM-based execution.

## Integration with cosim-arch-checker

The boot ROM integrates with the cosim-arch-checker system by providing a standardized initialization sequence that both the DUT and Whisper reference model can execute consistently:

The boot ROM ensures that:

*   Both DUT and Whisper start execution from the same reset vector
*   Multi-hart coordination follows identical patterns in both models
*   The transition to the test program occurs at the same instruction boundaries
*   All CSR initialization is performed consistently across both models

This standardized boot sequence is essential for maintaining lockstep execution between the DUT and reference model during architectural verification.
