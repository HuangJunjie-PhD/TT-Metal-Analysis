## Purpose and Scope

This page documents the **target wrapper system** that provides unified interfaces to TTNN operations, and the **categorization of TTNN operations** used throughout the compilation pipeline. Target wrappers serve as an abstraction layer between PyTorch operations and TTNN operations, handling preprocessing, multi-device coordination, and complex operation patterns that require multiple TTNN calls.

For information about how PyTorch operations are lowered to these TTNN operations, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)). For details on how data movement operations are inserted around these wrappers, see [Data Movement and Alignment (AddDataMovePass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.3-data-movement-and-alignment-(adddatamovepass)).

## Target Wrapper System Overview

Target wrappers are special functions that provide preprocessing, composition, or specialized handling of TTNN operations. They serve three primary purposes:

1.   **Preprocessing**: Operations like parameter loading and weight preprocessing that should run only once
2.   **Composition**: Complex operations that require multiple TTNN calls (e.g., `roll`, `stack`, `native_layer_norm`)
3.   **Multi-Device Coordination**: Placeholder functions for shape propagation in data-parallel execution

## The `@target_wrapper` Decorator

The `@target_wrapper` decorator modifies function names to prevent naming conflicts when exporting code. It applies a prefix to the function's `__name__` attribute, which PyTorch's FX graph uses when generating operation names.

### Naming Convention

[torch_ttnn/passes/lowering/target_wrappers.py 13-26](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L13-L26)

This decorator is combined with `@torch.fx.wrap` to ensure wrapped functions can be traced by PyTorch's FX graph system.

**Example:** A function named `conv` becomes `target_wrapper_conv` in the graph, and the full qualified name in the exported code becomes `torch_ttnn_passes_lowering_target_wrappers_target_wrapper_conv`.

**Sources:**[torch_ttnn/passes/lowering/target_wrappers.py 13-26](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L13-L26)

## Preprocessing Wrappers

### `run_once` and `RunOnceIdx`

The `run_once` wrapper enables parameter preprocessing and weight loading optimization. It executes a sequence of operations only on the first iteration, caching the results for subsequent iterations. This is critical for operations like convolution weight preprocessing.

[torch_ttnn/passes/lowering/target_wrappers.py 29-165](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L29-L165)

**Key Features:**

| Feature | Description |
| --- | --- |
| **Function Specification Format** | `(should_return, func_name, args, kwargs)` tuples |
| **Result Referencing** | Previous results referenced via pickled `RunOnceIdx` objects |
| **Special Handling** | Convolution weight preprocessing with automatic result substitution |
| **Memory Management** | Deallocates temporary tensors that don't alias return values |

**Example Usage Pattern:** The `LoadOncePass` generates calls to `run_once` with specifications for preprocessing model parameters. Each parameter goes through operations like `ttnn.from_torch` and `ttnn.to_device`, with results cached for reuse.

**Sources:**[torch_ttnn/passes/lowering/target_wrappers.py 29-165](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L29-L165)

#### `roll` - Tensor Rolling

[torch_ttnn/passes/lowering/target_wrappers.py 276-297](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L276-L297)

Implements tensor rolling by slicing and concatenating:

**Algorithm:**

1.   For each dimension to roll: calculate `end = (input_shape[dim] - shift) % input_shape[dim]`
2.   Slice tensor into two parts at the end position
3.   Concatenate slices in reverse order `[part2, part1]`

**Special Requirements:**

*   Input must be row-major layout ([torch_ttnn/passes/lowering/add_data_move_pass.py 352-357](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L352-L357))

#### `stack` - Tensor Stacking

[torch_ttnn/passes/lowering/target_wrappers.py 300-321](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L300-L321)

Stacks tensors along a new dimension by reshaping and concatenating:

**Process:**

1.   Handle negative dimension by wrapping around
2.   Create unsqueezed shape with size 1 in stack dimension
3.   Reshape each input tensor to add new dimension
4.   Handle layout constraints (convert tiled uint32 to row-major)
5.   Concatenate reshaped tensors along stack dimension

**Special Handling:**

*   Converts tiled uint32 tensors to row-major layout before concatenation ([torch_ttnn/passes/lowering/target_wrappers.py 316-317](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L316-L317))

#### `all` - Boolean Reduction

[torch_ttnn/passes/lowering/target_wrappers.py 324-341](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L324-L341)

Checks if all elements in a tensor are non-zero:

**Implementation:**

1.   Compare tensor elements to zero: `neq_zero = ttnn.ne(tensor, 0)`
2.   Sum non-zero elements: `total_none_zero = ttnn.sum(neq_zero)`
3.   Compare sum to total number of elements: `ttnn.eq(total_none_zero, num_elements)`

Used in [torch_ttnn/passes/lowering/to_tt_pass.py 1071-1080](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L1071-L1080) for lowering `aten.all.default`.

#### `native_layer_norm` - Layer Normalization

[torch_ttnn/passes/lowering/target_wrappers.py 369-413](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L369-L413)

Provides conditional execution of layer normalization based on whether mean and rstd outputs are needed:

**Optimization Strategy:**

| Condition | Implementation | Reason |
| --- | --- | --- |
| Only output needed | `ttnn.layer_norm` | Marginally faster for single output |
| Mean and/or rstd needed | `ttnn.operations.moreh.layer_norm` | Can produce all three outputs |

**Shape Handling:**

*   PyTorch `aten.native_layer_norm` keeps normalized dimensions in mean/rstd shapes
*   TTNN `moreh.layer_norm` does not, requiring reshape operations ([torch_ttnn/passes/lowering/target_wrappers.py 392-393](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L392-L393))

## Integration with Compilation Passes

### ToTtPass Integration

The `ToTtPass` uses target wrappers in two contexts:

1.   **Direct wrapping in ReplaceMoreTt transformer:** Simple operations are lowered directly to wrappers
2.   **Manual wrapping in ReplaceMoreTtManually:** Complex operations requiring graph manipulation use wrappers through `GraphWrapper.call_function`

[torch_ttnn/passes/lowering/to_tt_pass.py 31](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L31-L31)

The convolution lowering extracts parameters from `aten.convolution.default` and forwards them to `target_wrappers.conv`, which dispatches to the appropriate TTNN convolution function.

### AddDataMovePass Integration

The `AddDataMovePass` treats target wrappers as compute operations and applies data movement around them:

**Special Alignment Rules:**[torch_ttnn/passes/lowering/add_data_move_pass.py 345-386](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L345-L386)

| Target Wrapper | Input Site | Special Rule |
| --- | --- | --- |
| `conv` | Input 1 (weight) | Must be on host, row-major layout |
| `stack` | All inputs | Coerce int32/int64 to uint32 if needed |
| `roll` | Input 0 | Must be row-major layout |

**Multi-Device Substitution:**[torch_ttnn/passes/lowering/add_data_move_pass.py 443-464](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L443-L464)

When multi-device mode is active, `AddDataMovePass` detects placeholder wrappers (`shard_tensor`, `replicate_tensor`, `concat_tensor`) and substitutes them with actual mesh mapper calls during `from_torch` and `to_torch` insertion.

## Operation Classification Functions

Three key functions determine how the compilation pipeline handles operations:

### `is_tt_compute(node)`

[torch_ttnn/passes/lowering/add_data_move_pass.py 201-241](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L201-L241)

Returns `True` if the node is a TTNN compute operation (any operation from the categories above, including target wrappers). Used to determine when data movement operations need to be inserted.

**Special handling:**

*   If node is `getitem`, checks the input of the split operation
*   Excludes pure data movement operations like `ttnn.from_device`, `ttnn.to_device`

### `is_tt_data_move(node)`

[torch_ttnn/passes/lowering/add_data_move_pass.py 244-256](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L244-L256)

Returns `True` if the node is a TTNN data movement operation:

*   `ttnn.from_device`, `ttnn.to_device`
*   `ttnn.from_torch`, `ttnn.to_torch`
*   `ttnn.to_layout`
*   `ttnn.MemoryConfig`, `ttnn.Shape`

### `is_tt(node)`

[torch_ttnn/passes/lowering/add_data_move_pass.py 258-259](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L258-L259)

Returns `True` if the node is any TTNN operation (compute or data movement).
