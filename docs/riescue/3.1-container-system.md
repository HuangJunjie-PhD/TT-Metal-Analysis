## Purpose and Scope

The Container System provides a reproducible, isolated development environment for RiESCUE using Singularity/Apptainer containerization. This system encapsulates all system-level dependencies, external tools (Whisper ISS, Spike ISS, RISC-V GNU Toolchain), and Python packages required to develop and run RiESCUE tests. The container eliminates environment inconsistencies by ensuring identical runtime environments across different host systems.

For information about the specific build scripts that install dependencies within the container, see [Build Scripts and Dependencies](/tenstorrent/riescue/3.2-build-scripts-and-dependencies). For Python package configuration details, see [Python Package Configuration](/tenstorrent/riescue/3.3-python-package-configuration). For details on the CI/CD integration, see [CI/CD Pipeline](/tenstorrent/riescue/5.3-cicd-pipeline).

**Sources:** [infra/Container.def](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def) [docs/container.md](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1)

## Architecture Overview

The Container System consists of four primary components: the container definition file, a Python management class, wrapper scripts, and a configuration system.

**Diagram: Container System Architecture**

The `Container` class provides lifecycle management methods that interact with Singularity/Apptainer through subprocess calls. Wrapper scripts provide convenient command-line interfaces to these methods. The `.container_config` file enables optional registry integration and custom bind mounts.

## Container Definition

### Container.def Structure

The container definition file specifies the complete build process for the RiESCUE development environment.

| Section | Purpose | Key Contents |
| --- | --- | --- |
| `bootstrap` | Base image specification | Rocky Linux 9.1 from Docker Hub |
| `%environment` | Runtime environment variables | PATH and LD\_LIBRARY\_PATH configuration |
| `%runscript` | Container startup commands | Sets unlimited stack size with `ulimit` |
| `%labels` | Container metadata | VERSION label for tracking (currently 1.30) |
| `%setup` | Pre-build setup | Enables strict error handling |
| `%files` | Copy files into container | Project files, build scripts, pyproject.toml |
| `%post` | Build-time installation | Install dependencies, build external tools |

### Environment Configuration

The container configures runtime environment variables in the `%environment` section:

* **PATH modification**: Prepends `/usr/local/bin` to ensure container-built tools take precedence over system tools [infra/Container.def6](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L6-L6)

| --- | --- | --- | --- |

| `remote_token_path` | string | No | Path to authentication token file, supports `~` expansion |

## Lifecycle Operations

### Building Containers

The `build()` method creates a new container image from the definition file:

**Diagram: Container Build Process**

The build process includes automatic version management. If the definition version matches the tracked container version, the definition version is incremented to indicate a new build [infra/singularity/container.py93-96](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L93-L96) The version string combines the numeric version with a SHA-256 hash of the container's filesystem payload [infra/singularity/container.py97-99](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L97-L99)

**Sources:** [infra/singularity/container.py85-112](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L85-L112)

### Running Containers

The `run()` method executes commands within the container environment:

**Diagram: Container Run Process**

The run method implements intelligent version checking. If the local SIF file's version doesn't match the tracked version in `singularity-id`, it automatically pulls from the registry or rebuilds locally [infra/singularity/container.py132-146](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L132-L146) Bind mounts are filtered to include only paths that exist on the host system [infra/singularity/container.py149-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L149-L156)

**Special Argument Handling:**

* `--singularity-sif=/path/to/custom.sif`: Use a specific SIF file, bypassing version checks
* `--singularity-binds=/path1:/mount1,/path2:/mount2`: Add additional bind mounts
* All arguments starting with `--singularity-` are removed before command execution [infra/singularity/container.py319-325](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L319-L325)

**Sources:** [infra/singularity/container.py114-171](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L114-L171) [infra/singularity/container.py305-325](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L305-L325)

### Version Management and Hash Verification

The container system implements a two-part versioning scheme: semantic version + content hash.

#### Version Properties

| Property | Source | Format | Purpose |
| --- | --- | --- | --- |
| `def_version` | Container.def VERSION label | Integer (e.g., 30) | Manually incremented build version |
| `sif_version` | Built SIF file labels | Integer | Version embedded in container image |
| `container_version` | singularity-id file | Integer | Version of tracked/deployed container |
| `container_string` | singularity-id file | `{version}-{sha}` | Full version identifier with hash |

**Sources:** [infra/singularity/container.py204-224](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L204-L224)

### Registry Operations

#### Pushing Containers

The `push()` method uploads a container image to a remote registry:

1. Constructs version string: `{def_version}-{sha_payload()}`
2. Executes `singularity push riescue.sif {registry_uri}:{version_string}`
3. Requires registry authentication via `_check_singularity_remotes()`

**Sources:** [infra/singularity/container.py172-176](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L172-L176)

#### Pulling Containers

The `pull()` method downloads the tracked container version:

1. Verifies registry configuration exists [infra/singularity/container.py180-181](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L180-L181)
2. Removes existing local SIF if present [infra/singularity/container.py182-184](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L182-L184)
3. Checks and adds Singularity remote if needed [infra/singularity/container.py185](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L185-L185)
4. Executes `singularity pull riescue.sif {registry_uri}:{container_string}` [infra/singularity/container.py186-187](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L186-L187)

The pulled container matches the version string stored in `singularity-id`, ensuring reproducible environments across machines.

**Sources:** [infra/singularity/container.py178-187](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L178-L187)

## Bind Mounts

Bind mounts allow the container to access host filesystem locations. The container system manages bind mounts from multiple sources:

### Default Binds

The base bind configuration is hardcoded: `/usr/lib64:/shared` [infra/singularity/container.py41](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L41-L41) This ensures shared libraries on the host are available in the container at `/shared`.

### Configuration Binds

Additional binds can be specified in `.container_config`:

The bind format is `source:target` where `source` is the host path and `target` is the container path. If no `:` separator is present, the same path is used for both [docs/container.md25-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L25-L26)

### Runtime Binds

Command-line binds override configuration:

### Bind Resolution

Before container execution, the `run()` method filters bind mounts to include only paths that exist on the host system [infra/singularity/container.py149-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L149-L156):

This prevents container launch failures due to missing host directories.

## Environment Variables

### .env File Support

The `Container` class automatically loads environment variables from a `.env` file in the repository root [infra/singularity/container.py73](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L73-L73):

**Example .env file:**

```
RV_TOOLCHAIN=/opt/riscv WHISPER_CONFIG=/path/to/whisper_config.json 
```

**Sources:** [infra/singularity/container.py291-303](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L291-L303)

### RV\_TOOLCHAIN Path Export

If `RV_TOOLCHAIN` is defined, the `Container.__init__()` method automatically appends it to the `PATH` environment variable [infra/singularity/container.py75-82](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L75-L82):

This enables the container to locate RISC-V toolchain binaries without requiring explicit path configuration.

**Sources:** [infra/singularity/container.py75-82](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L75-L82)

### Container Runtime Environment

Within the container, the `%environment` section sets persistent environment variables [infra/Container.def5-7](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L5-L7):

* `PATH=/usr/local/bin:${PATH}`: Prioritizes container-built tools
* `LD_LIBRARY_PATH=/usr/lib64:${LD_LIBRARY_PATH:+$LD_LIBRARY_PATH:}/shared`: Adds library search paths

**Sources:** [infra/Container.def5-7](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L5-L7)

## Advanced Topics

### Fakeroot Builds

The build command uses `--fakeroot` to allow building containers without root privileges [infra/singularity/container.py96](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L96-L96):

This requires Singularity/Apptainer 3.5+ with user namespace support configured on the host system.

### Force Rebuilds

The `--force` flag ensures the build overwrites existing SIF files [infra/singularity/container.py96](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L96-L96) Combined with version incrementation, this prevents accidental overwrites of tracked versions.

### NO\_SINGULARITY Environment Variable

During container build, the `%post` section sets `NO_SINGULARITY=1` [infra/Container.def50](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/Container.def#L50-L50) This flag is checked by nested build scripts to prevent attempting container operations within containers, which would fail or cause infinite recursion.
