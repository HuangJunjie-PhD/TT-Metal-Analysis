## Purpose and Scope

This document describes the Blackhole-specific architecture implementation within the `tt-umd` codebase. It covers the `BlackholeTTDevice` class, Blackhole topology discovery, hardware communication mechanisms, and key architectural differences from Wormhole. For general device management concepts, see [Device Management System](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/Device%20Management%20System) For the architecture abstraction pattern, see [Architecture-Specific Features](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/Architecture-Specific%20Features) For Wormhole-specific details, see [Wormhole Architecture](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/Wormhole%20Architecture)

## BlackholeTTDevice Class Implementation

The `BlackholeTTDevice` class provides the Blackhole-specific implementation of the `TTDevice` interface. Unlike Wormhole, Blackhole has distinct hardware characteristics including no DMA support, configurable iATU regions, and conditional ARC accessibility.

## iATU Configuration

Blackhole requires explicit iATU (Internal Address Translation Unit) configuration for PCIe address mapping. This is necessary to map host memory addresses to NOC-accessible address ranges. The iATU configuration is done directly through BAR2 registers.

## ARC Communication and Access Paths

Blackhole has unique ARC access characteristics. The ARC processor may not be accessible via AXI (PCIe) depending on the PCIe tile's x-coordinate. When ARC is unavailable over AXI, UMD falls back to NOC-based access.

### ARC Accessibility Determination

| PCIe X-Coordinate | ARC over AXI | Access Method |
| --- | --- | --- |
| 2 (Type 2) | No | NOC-based read/write |
| 11 (Type 1) | Yes | AXI/BAR0-based read/write |

**Sources:**[device/tt_device/blackhole_tt_device.cpp 211-253](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L211-L253)[device/tt_device/blackhole_tt_device.cpp 304-312](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L304-L312)

## Hardware-Specific Features

### DMA Operations

Blackhole does **not** support DMA operations. All DMA methods throw runtime errors:

| Method | Behavior |
| --- | --- |
| `dma_d2h()` | Throws "D2H DMA is not supported on Blackhole." |
| `dma_h2d()` | Throws "H2D DMA is not supported on Blackhole." |
| `dma_h2d_zero_copy()` | Throws "H2D DMA is not supported on Blackhole." |
| `dma_d2h_zero_copy()` | Throws "D2H DMA is not supported on Blackhole." |

**Sources:**[device/tt_device/blackhole_tt_device.cpp 195-209](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L195-L209)

### CSM Access

Blackhole does not support CSM (Chip-to-chip Synchronization Module) read/write operations:

| Method | Behavior |
| --- | --- |
| `read_from_arc_csm()` | Throws "CSM read not supported for Blackhole." |
| `write_to_arc_csm()` | Throws "CSM write not supported for Blackhole." |

**Sources:**[device/tt_device/blackhole_tt_device.cpp 255-261](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L255-L261)

## Coordinate Manager

The `BlackholeCoordinateManager` handles the mapping between `LOGICAL`, `TRANSLATED`, and `NOC0` coordinate systems, accounting for harvesting effects.

### Tensix Harvesting Logic

Blackhole implements a specific column-shuffling logic for Tensix cores when harvesting is enabled.

**Sources:**[device/coordinates/blackhole_coordinate_manager.cpp 75-149](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/blackhole_coordinate_manager.cpp#L75-L149)

### Coordinate Manager Constraints

The `BlackholeCoordinateManager` enforces architecture-specific constraints during construction:

*   At most 1 DRAM bank can be harvested. [device/coordinates/blackhole_coordinate_manager.cpp 60-62](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/blackhole_coordinate_manager.cpp#L60-L62)
