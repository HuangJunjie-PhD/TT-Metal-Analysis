## Purpose and Scope

This document describes the stimuli generation and data packing/unpacking infrastructure used by the tt-llk testing framework. This system bridges the gap between high-level Python test code (PyTorch tensors) and low-level hardware data formats stored in Tensix L1 memory.

**Scope:**

*   Generation of test input data (stimuli) with various patterns and formats
*   Packing functions that serialize PyTorch tensors into hardware byte representations
*   Unpacking functions that deserialize hardware data back to PyTorch tensors
*   Memory layout management for organizing data in L1 memory
*   Format-specific handling for BFP8_b, MXFP8, and standard formats

**Related Pages:**

*   For data format definitions and inference, see [Data Formats and Configuration](https://deepwiki.com/tenstorrent/tt-llk/3-data-formats-and-configuration)
*   For device communication and hardware execution, see [Device Interface and Hardware Execution](https://deepwiki.com/tenstorrent/tt-llk/8.5-device-interface-and-hardware-execution)
*   For golden model generation that uses this infrastructure, see [Golden Model System](https://deepwiki.com/tenstorrent/tt-llk/8.4-golden-model-system)
*   For test orchestration that coordinates these components, see [TestConfig and Test Orchestration](https://deepwiki.com/tenstorrent/tt-llk/8.2-testconfig-and-test-orchestration)

### Tile and Face Structure

The stimuli generation system works with the hardware's tile-based memory layout:

**Key Constants:**

*   `FACE_DIM = 16` - Face width (columns)
*   `FACE_R_DIM = 16` - Default face height (rows)
*   `ELEMENTS_PER_FACE = 256` - 16×16
*   `FACES_PER_TILE = 4` - Full tile has 4 faces
*   `ELEMENTS_PER_TILE = 1024` - 4 faces × 256 elements
*   `TILE_SIZE = 32` - Standard tile dimension

**Tile Generation Modes:**

1.   **Full Tiles (`face_r_dim=16, num_faces=4`)**: Generate complete 32×32 tiles with all 1024 elements
2.   **Partial Faces (`face_r_dim<16`)**: Generate tiles with reduced face dimensions (1, 2, 4, 8 rows per face)
3.   **Reduced Face Count (`num_faces<4`)**: Generate only 1 or 2 faces per tile (256 or 512 elements)
4.   **Dense Tile Dimensions**: Generate tiles with non-standard dimensions like 8×32 or 16×16

### MX Format Stimuli Generation

MX (Microscaling) formats require special handling to generate data within their representable range:

**MX Format Constants:**

| Format | Element Type | Max Normal Value | Scale Factor Used |
| --- | --- | --- | --- |
| `MxFp8R` | E5M2 (5 exp, 2 mant) | ±57,344 | 2,867.2 (5%) |
| `MxFp8P` | E4M3 (4 exp, 3 mant) | ±448 | 22.4 (5%) |

**Rationale for 5% Scaling:**

*   Ensures values are well within representable range
*   Avoids saturation during quantization
*   Maintains realistic dynamic range for testing
*   Creates diverse test data without edge cases

**Integer Sign-Magnitude Format:**

| Format | Sign Bit | Magnitude Bits | Special Notes |
| --- | --- | --- | --- |
| `Int32` | bit 31 | bits 30:0 | Cannot represent -2³¹, clips to [-2³¹+1, 2³¹-1] |

| `Int8` | bit 7 | bits 6:0 | Cannot represent -2⁷, clips to [-2⁷+1, 2⁷-1] |

### BFP8_b Format Packing

Block Floating-Point 8-bit (BFP8_b) uses 16-element blocks with a shared exponent:

**BFP8_b Packing Algorithm:**

1.   **Block Division**: Split input into 16-element blocks (based on `face_r_dim` and `num_faces`)
2.   **Find Max Exponent**: Determine maximum exponent across block [tests/python_tests/helpers/pack.py 80-113](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/pack.py#L80-L113)
3.   **Normalize Mantissas**: Shift each mantissa by delta from max exponent
4.   **Pack Mantissas**: Combine sign bit (1 bit) + shifted mantissa (7 bits)
5.   **Hardware Padding**: Pad to `MIN_BFP_EXPONENTS = 16` total exponents

**Memory Layout:**

```
[exponent_block0][exponent_block1]...[padding_to_16]
[mantissa_block0_elem0][mantissa_block0_elem1]...[mantissa_block0_elem15]
[mantissa_block1_elem0]...
```

**Key Implementation Details:**

`def pack_bfp8_b(tensor, block_size=16, num_faces=4, face_r_dim=16):    elements_per_face = face_r_dim * FACE_C_DIM  # 16    elements_to_pack = elements_per_face * num_faces    flattened_tensor = tensor.flatten()[:elements_to_pack]        num_blocks = len(flattened_tensor) // block_size    exponents = []    mantissas = []        for i in range(num_blocks):        block = flattened_tensor[i * block_size : (i + 1) * block_size]        shared_exponent, bfp8_mantissas = float_to_bfp8_block(block)        exponents.append(shared_exponent)        mantissas.extend(bfp8_mantissas)        # Hardware requires minimum 16 exponents    if len(exponents) < MIN_BFP_EXPONENTS:        exponents.extend([0] * (MIN_BFP_EXPONENTS - len(exponents)))        return exponents + mantissas`

### MX Format Packing

Microscaling (MX) formats use 32-element blocks with E8M0 scale factors per OCP MX specification:

**MX Format Comparison:**

| Feature | MxFp8R (E5M2) | MxFp8P (E4M3) |
| --- | --- | --- |
| Element Format | 1 sign + 5 exp + 2 mant | 1 sign + 4 exp + 3 mant |
| Element Max Normal | ±57,344 | ±448 |
| Inf Support | Yes | No |
| NaN Encoding | Standard | 0bS1111111 |
| Precision | Lower (2-bit mantissa) | Higher (3-bit mantissa) |
| Dynamic Range | Higher (5-bit exponent) | Lower (4-bit exponent) |
| ml_dtypes Type | `float8_e5m2` | `float8_e4m3fn` |

**E8M0 Scale Encoding:**

The E8M0 (8-bit exponent, no mantissa) scale encoding follows the OCP MX specification:

`def encode_e8m0_scale(max_abs_value, element_max_normal):    # Special cases    if max_abs_value == 0 or np.isnan(max_abs_value):        return 127  # 2^0 = 1 (neutral scale)    if np.isinf(max_abs_value):        return 254  # Max representable scale        # Calculate exponent: ceil(log2(max / element_max))    scale_ratio = max_abs_value / element_max_normal    exponent = math.ceil(math.log2(scale_ratio))        # Clamp and add bias    return int(max(-127, min(127, exponent)) + 127)`
The "round up" (ceiling) approach ensures post-scaling values never exceed the representable FP8 range [tests/python_tests/helpers/format_config.py 133-162](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/format_config.py#L133-L162)

**Memory Layout:**

MX formats use a **fully separated** layout (similar to BFP8_b):

```
[all_scales_32_bytes][all_fp8_elements_1024_bytes]
```

*   **Scales**: 32 bytes (1 per 32-element block, 32 blocks for 1024 elements)
*   **Elements**: 1024 bytes (32 blocks × 32 elements)

**Vectorized Implementation:**

The packing uses NumPy vectorization for performance:

`def _pack_mxfp8(tensor, fp8_dtype, element_max_normal, num_faces=4, face_r_dim=16):    # Reshape into blocks: (num_blocks, 32)    blocks = fp32_array[:num_blocks * 32].reshape(num_blocks, 32)        # Vectorized scale encoding    max_abs_values = np.max(np.abs(blocks), axis=1)    scale_ratio = max_abs_values / element_max_normal    exponents = np.ceil(np.log2(scale_ratio, where=(scale_ratio > 0)))        # Handle special cases vectorially    exponents = np.where((max_abs_values == 0) | np.isnan(max_abs_values), 0,                        np.where(np.isinf(max_abs_values), 127, exponents))        # Convert to E8M0    scales_e8m0 = np.clip(exponents, -127, 127).astype(np.int32) + 127        # Scale blocks and convert to FP8    scale_factors = np.exp2(scales_e8m0.astype(np.float32) - 127.0)    scaled_blocks = blocks / scale_factors[:, np.newaxis]    fp8_blocks = scaled_blocks.astype(fp8_dtype)        # Return separated layout    return scales_e8m0.tolist() + list(fp8_blocks.tobytes())`

## Unpacking System

The unpacking system reverses the packing transformations, converting hardware byte formats back to PyTorch tensors:

### BFP8_b Unpacking

BFP8_b unpacking reverses the shared exponent encoding:

**Unpacking Algorithm:**

The implementation uses memoization to avoid recomputing the same (exponent, mantissa) pairs:

`def bfp8_to_float_block(exponent, bfp8_mantissas, unpacked_bfp8):    bfloat16_values = []    exponent = exponent - 127  # Remove bias        for mantissa in bfp8_mantissas:        # Check cache first        if (exponent, mantissa) in unpacked_bfp8:            bfloat16_values.append(unpacked_bfp8[(exponent, mantissa)])            continue                # Extract sign and mantissa bits        sign_mantissa = str(format(mantissa, "08b"))        sign = int(sign_mantissa[0], 2)        mantissa_value = sign_mantissa[1:]                # Calculate fractional value        fract_value = 0.0        for i in range(len(mantissa_value)):            if mantissa_value[i] == "1":                fract_value += 1 / (2 ** i)                value = ((-1.0) ** sign) * (2**exponent) * fract_value        bfloat16_values.append(value)        unpacked_bfp8[(exponent, mantissa)] = value        return bfloat16_values def unpack_bfp8_b(bfp8_block, sfpu=False, num_faces=4, face_r_dim=16):    actual_exponents = face_r_dim * num_faces    exponents_in_packed = max(actual_exponents, MIN_BFP_EXPONENTS)        all_exponents = bfp8_block[:exponents_in_packed]    mantissas = bfp8_block[exponents_in_packed:]    exponents = all_exponents[:actual_exponents]  # Only use actual        unpacked_bfp8 = {}    bfloat16_values = []        for i in range(len(exponents)):        exponent = exponents[i]        bfp8_mantissas = mantissas[i * 16 : (i + 1) * 16]        block_values = bfp8_to_float_block(exponent, bytes(bfp8_mantissas), unpacked_bfp8)        bfloat16_values.extend(block_values)        return torch.tensor(bfloat16_values, dtype=torch.bfloat16)`

### Integer Sign-Magnitude Unpacking

Hardware integer formats use sign-magnitude representation, which must be converted back to two's complement:

## Memory Layout Management

The `StimuliConfig` class manages L1 memory layout and coordinates data transfers:

### StimuliConfig Architecture

**Key Memory Addresses:**

| Configuration | Base Address | Notes |
| --- | --- | --- |
| Performance mode | `0x21000` | Normal testing, smaller L1 footprint |

### Write Operations

The `write()` method handles two modes: backward-compatible (32×32 tiles) and dense (variable dimensions):

**Backward-Compatible Write:**

`@staticmethoddef write_matrix(    buffer,    tile_count: int,    pack_function,    base_address: int,    tile_size: int,    num_faces: int,    face_r_dim: int,    location: str = "0,0",    write_full_tiles: bool = False,):    # Always stride at MAX_TILE_ELEMENTS (1024)    if write_full_tiles:        tile_elements = MAX_TILE_ELEMENTS    else:        tile_elements = num_faces * face_r_dim * FACE_C_DIM        for ind in range(tile_count):        start_idx = MAX_TILE_ELEMENTS * ind  # Fixed stride        tile_data = buffer[start_idx : start_idx + tile_elements]        packed_data = pack_function(tile_data, num_faces=num_faces, face_r_dim=face_r_dim)        write_to_device(location, base_address + ind * tile_size, packed_data)`

## Integration with Test Pipeline

The following diagram shows how stimuli generation and packing integrate with the complete test flow:

**Integration Points:**

1.   **Test Function** → `generate_stimuli()`: Creates input tensors
2.   **Tensors** → `StimuliConfig`: Configures memory layout
3.   **StimuliConfig** → `pack.py`: Serializes tensors to bytes
4.   **Packed bytes** → `write_to_device()`: Transfers to L1
5.   **Hardware execution** → `read_from_device()`: Fetches results
6.   **Result bytes** → `unpack.py`: Deserializes to tensors
7.   **Result tensor** → `passed_test()`: Validates against golden
