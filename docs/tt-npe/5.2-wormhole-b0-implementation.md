## Purpose and Scope

This page documents the `WormholeB0DeviceModel` class, which implements the device model abstraction for Tenstorrent's Wormhole B0 architecture. This model encapsulates hardware-specific characteristics including grid topology, core type mapping, NoC routing algorithms, bandwidth characteristics, and congestion modeling. The implementation enables the simulation engine to accurately predict NoC performance for workloads targeting Wormhole B0 devices.

For information about the device model interface that this class implements, see [Device Model Interface](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface). For multi-chip extensions of this model, see [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.4-multi-chip-device-models). For the Blackhole architecture implementation, see [Blackhole Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.3-blackhole-implementation).

## Physical Grid Architecture

The Wormhole B0 architecture uses a **12x10 grid** (`_num_rows = 12`, `_num_cols = 10`) organized as a 2D mesh with wraparound (torus) topology. Each grid location can contain different core types that determine its capabilities and bandwidth characteristics.

**Grid Dimensions and Topology:**

| Property | Value | Description |
| --- | --- | --- |
| Rows | 12 | `_num_rows` constant |
| Columns | 10 | `_num_cols` constant |
| Total Cores | 120 | rows × cols |
| Topology | Torus | Wraparound in both dimensions |
| Device ID | 0 | Single-chip model uses `_device_id = 0` |

## Core Type Mapping

The `WormholeB0DeviceModel` maintains a complete mapping from grid coordinates to core types via the `coord_to_core_type_map`. This mapping is initialized in the constructor and stored in a `Grid2D<CoreType>` for efficient lookup.

**Core Type Distribution:**

| Core Type | Count | Locations | Primary Function |
| --- | --- | --- | --- |
| `CoreType::DRAM` | 12 | Columns 0, 5 at specific rows | Memory access |
| `CoreType::ETH` | 16 | Rows 0, 6 at columns 1-4, 6-9 | Ethernet links for multi-chip |
| `CoreType::WORKER` | 80 | Interior grid positions | Tensix compute cores |
| `CoreType::UNDEF` | 12 | Row 2-4, 8-10 at column 0 | Unused/harvested positions |

**Lookup Implementation:**

The `getCoreType()` method provides O(1) lookup through the `coord_to_core_type` grid:

`CoreType getCoreType(const Coord &c) const override {     return coord_to_core_type(c.row, c.col); }`

## Network-on-Chip (NOC) Routing

Wormhole B0 features two independent NoCs (NOC0 and NOC1) with dimension-ordered routing. The routing algorithm is deterministic and implements torus wraparound for both NoCs.

**Routing Characteristics:**

| NoC | Order | Link Types | Wraparound Function |
| --- | --- | --- | --- |
| NOC0 | East, then South | `NOC0_EAST`, `NOC0_SOUTH` | `wrapToRange(x+1, limit)` |
| NOC1 | North, then West | `NOC1_NORTH`, `NOC1_WEST` | `wrapToRange(x-1, limit)` |

**Unicast Routing Implementation:**

The `unicastRoute()` method implements dimension-ordered routing with torus wraparound:

*   **NOC0**: Moves east until column matches, then south until row matches
*   **NOC1**: Moves north until row matches, then west until column matches

The routing produces a `nocRoute` vector containing the sequence of `nocLinkID` values traversed.

**Multicast Routing:**

For multicast destinations, the router computes a union of unicast routes to all destination cores:

*   **NOC0 Multicast**: Routes to each column in the destination range at the final row
*   **NOC1 Multicast**: Routes to each row in the destination range at the final column

The implementation uses `boost::unordered_flat_set<nocLinkID>` to deduplicate links that appear in multiple unicast paths, producing a single combined route.

**Hop Count Calculation:**

Static utility function `route_hops()` computes the number of hops for a route:

```
NOC0 hops = modulo(dst.col - src.col, 10) + modulo(dst.row - src.row, 12)
NOC1 hops = modulo(src.col - dst.col, 10) + modulo(src.row - dst.row, 12)
```

## Bandwidth Characteristics

The Wormhole B0 model uses a packet-size-dependent bandwidth table (`TransferBandwidthTable`) to model realistic NoC performance, accounting for packetization overhead and steady-state throughput.

**Transfer Bandwidth Table:**

| Packet Size (bytes) | Bandwidth (B/cycle) | Notes |
| --- | --- | --- |
| 0 | 0.0 | Zero transfer |
| 128 | 5.5 | Small packet overhead |
| 256 | 10.1 |  |
| 512 | 18.0 |  |
| 1024 | 27.4 |  |
| 2048 | 30.0 | Peak bandwidth reached |
| 8192 | 30.0 | Large packet performance |

**Bandwidth Interpolation:**

The `interpolateBW()` function in `npeDeviceModelUtils.hpp` performs linear interpolation between table entries and accounts for first-packet latency:

1.   **Find bracketing entries**: Locate two consecutive table entries that bracket the actual packet size
2.   **Linear interpolation**: Compute steady-state bandwidth based on position within range
3.   **First-packet adjustment**: Weight the effective bandwidth between first-packet (peak) and steady-state rates based on `num_packets`

```
effective_bw = (first_transfer_ratio × max_bw) + (steady_state_ratio × interpolated_bw)
where: steady_state_ratio = (num_packets - 1) / num_packets
```

**Aggregate Bandwidth Metrics:**

`// Maximum link bandwidth (any single link)float getLinkBandwidth(const nocLinkID &link_id) const override { return 30; } // Aggregate DRAM bandwidth (all 12 banks)float getAggregateDRAMBandwidth() const override {     return NUM_BANKS * ((core_type_to_inj_rate.at(CoreType::DRAM) +                          core_type_to_abs_rate.at(CoreType::DRAM)) / 2); } // Single-direction Ethernet link bandwidthfloat getEthBandwidth() const override { return 12.5; }`

## Injection and Absorption Rates

Each core type has specific injection (source) and absorption (sink) rates that constrain the maximum bandwidth for transfers originating from or terminating at that core.

**Injection and Absorption Rate Table:**

| Core Type | Injection Rate (B/cycle) | Absorption Rate (B/cycle) | Notes |
| --- | --- | --- | --- |
| `CoreType::DRAM` | 23.2 | 24.0 | Asymmetric DRAM access |
| `CoreType::ETH` | 28.1 | 24.0 | Ethernet core limits |
| `CoreType::WORKER` | 28.1 | 28.1 | Symmetric Tensix core rates |
| `CoreType::UNDEF` | 28.1 | 28.1 | Default rates for undefined cores |

**Rate Inference:**

The `inferInjectionRates()` method in `npeWorkload` automatically populates transfer injection rates based on source core types:

`void npeWorkload::inferInjectionRates(const npeDeviceModel &device_model) {    for (auto &ph : phases) {        for (auto &tr : ph.transfers) {            auto inferred_ir = device_model.getSrcInjectionRate(tr.src);            tr.injection_rate = inferred_ir;        }    }}`
This enables workload specifications to omit explicit injection rates, with the device model providing hardware-accurate defaults. See [Injection Rate Inference](https://deepwiki.com/tenstorrent/tt-npe/7.3-injection-rate-inference) for details.

**Fallback Behavior:**

If a core type is not found in the rate maps, the implementation logs an error and defaults to the `CoreType::WORKER` rate to prevent simulation failure.

## Congestion Modeling

The `modelCongestion()` method implements a first-order congestion model that adjusts transfer bandwidths based on resource contention. The algorithm runs once per timestep (`NUM_ITERS = 1`) and derates bandwidth when demand exceeds capacity.

**Congestion Model Characteristics:**

| Parameter | Value | Description |
| --- | --- | --- |
| Iterations | 1 | `NUM_ITERS = 1` - first-order congestion only |
| Gradient Factor | 1.0 | `grad_fac = 1.0` - full bandwidth adjustment |
| Link Bandwidth | 30.0 B/cycle | Capacity for derate calculation |
| Demand Calculation | Time-weighted | Accounts for mid-timestep starts |

**Demand Accumulation:**

For each active transfer, the model computes `effective_demand`:

```
effective_demand = (end_timestep - max(start_timestep, transfer.start_cycle)) / 
                   (end_timestep - start_timestep) × transfer.curr_bandwidth
```

This time-weighting accounts for transfers that begin mid-timestep, reducing their effective demand proportionally.

**Bottleneck Identification:**

For each transfer, three potential bottlenecks are identified:

1.   **Link Bottleneck**: Maximum demand on any link in the route divided by link capacity
2.   **Source NIU Bottleneck**: Source injection rate divided by total demand at source NIU
3.   **Sink NIU Bottleneck**: Sink absorption rate divided by total demand at destination NIU(s)

The minimum derate factor across all three bottlenecks determines the bandwidth adjustment.

**Multicast Handling:**

For multicast transfers, the algorithm:

*   Accumulates demand at **all** destination NIUs (only `WORKER` cores for multicast writes)
*   Uses the **slowest** (maximum demand) sink NIU to compute the derate factor
*   Marks multicast write links separately in `multicast_write_link_demand_grid` for statistics

**Integration with Simulation Loop:**

The congestion model is invoked via `computeCurrentTransferRate()`:

`void computeCurrentTransferRate(...) {    // First, compute baseline bandwidth from transfer parameters    updateTransferBandwidth(&transfer_state, live_transfer_ids, ...);        // Then model congestion and derate if enabled    if (enable_congestion_model) {        modelCongestion(...);    }}`

## Link and NIU Management

The device model maintains bidirectional lookup tables for NoC links and Network Interface Units (NIUs), enabling efficient translation between coordinate-based attributes and numeric IDs.

**Link Types:**

The model defines four link types corresponding to NoC routing directions:

*   `nocLinkType::NOC0_EAST` - NOC0 horizontal links (eastward)
*   `nocLinkType::NOC0_SOUTH` - NOC0 vertical links (southward)
*   `nocLinkType::NOC1_NORTH` - NOC1 vertical links (northward)
*   `nocLinkType::NOC1_WEST` - NOC1 horizontal links (westward)

**Total link count**: 12 rows × 10 cols × 4 types = **480 links**

**NIU Types:**

The model defines four NIU types corresponding to NoC source/sink ports:

*   `nocNIUType::NOC0_SRC` - NOC0 source (injection) port
*   `nocNIUType::NOC0_SINK` - NOC0 sink (absorption) port
*   `nocNIUType::NOC1_SRC` - NOC1 source (injection) port
*   `nocNIUType::NOC1_SINK` - NOC1 sink (absorption) port

**Total NIU count**: 12 rows × 10 cols × 4 types = **480 NIUs**

**Lookup Table Population:**

Both link and NIU lookup tables are populated in the constructor via nested loops:

`void populateNoCLinkLookups() {    for (size_t r = 0; r < getRows(); r++) {        for (size_t c = 0; c < getCols(); c++) {            for (const auto &link_type : getLinkTypes()) {                nocLinkAttr attr = {{getDeviceID(), r, c}, link_type};                link_id_to_attr_lookup.push_back(attr);  // Sequential ID assignment                link_attr_to_id_lookup[attr] = link_id_to_attr_lookup.size() - 1;            }        }    }}`
The ordering ensures IDs are assigned sequentially: row-major order, then column-major within row, then link type within column.

**Lookup Methods:**

The device model provides bidirectional lookup:

| Method | Purpose | Complexity |
| --- | --- | --- |
| `getLinkID(nocLinkAttr)` | Attribute → ID | O(1) via hash map |
| `getLinkAttributes(nocLinkID)` | ID → Attribute | O(1) via vector indexing |
| `getNIUID(nocNIUAttr)` | Attribute → ID | O(1) via hash map |
| `getNIUAttributes(nocNIUID)` | ID → Attribute | O(1) via vector indexing |

**Device State Initialization:**

The lookup table sizes are used to initialize device state grids:

`std::unique_ptr<npeDeviceState> initDeviceState() const override {    size_t num_niu_types = niu_id_to_attr_lookup.size();  // 480    size_t num_links = link_id_to_attr_lookup.size();      // 480    return std::make_unique<npeDeviceState>(num_niu_types, num_links);}`

## Latency Calculations

The `WormholeB0DeviceModel` provides static methods for computing hardcoded latencies for read and write operations based on source and destination coordinates.

**Read Latency Table:**

| Condition | Latency (cycles) | Description |
| --- | --- | --- |
| Same core (sx=dx, sy=dy) | 70 | Local read within core |
| Same row (sx=dx, sy≠dy) | 154 | Vertical transfer |
| Same column (sy=dy, sx≠dx) | 170 | Horizontal transfer |
| Diagonal (sx≠dx, sy≠dy) | 270 | Both horizontal and vertical |

These are hardcoded empirical values representing the minimum latency for read operations on Wormhole B0 hardware.

**Write Latency Formula:**

Write latency is computed as a linear function of the number of hops:

```
write_latency = 40 + (hops × 10)
```

Where:

*   **Startup latency**: 40 cycles (fixed overhead)
*   **Per-hop latency**: 10 cycles (incremental cost per router hop)
*   **Hops**: Computed via `route_hops()` based on the NoC type

**Example Latency Calculations:**

| Transfer | NoC | Distance | Hops | Write Latency |
| --- | --- | --- | --- | --- |
| (0,0) → (0,0) | Either | Same core | 0 | 40 cycles |
| (0,0) → (5,0) | NOC0 | 5 rows south | 5 | 90 cycles |
| (0,0) → (5,0) | NOC1 | 7 rows north (wrap) | 7 | 110 cycles |
| (0,0) → (0,5) | NOC0 | 5 cols east | 5 | 90 cycles |
| (0,0) → (5,5) | NOC0 | 5+5 = 10 hops | 10 | 140 cycles |

**Usage Context:**

These static methods are provided for standalone latency estimation and are **not** directly used by the simulation engine, which models transfers at the bandwidth level rather than individual packet latencies. They are useful for:

*   External latency analysis tools
*   Validation of transfer scheduling
*   Performance comparison with hardware measurements

## Device State Initialization

The device model provides the `initDeviceState()` method to create properly-sized state objects for tracking runtime resource utilization during simulation.

**Device State Structure:**

The `npeDeviceState` object contains three primary grids for tracking resource demand:

| Grid | Type | Size (single-chip) | Purpose |
| --- | --- | --- | --- |
| `NIUDemandGrid` | `vector<float>` | 480 | Per-NIU bandwidth demand accumulation |
| `LinkDemandGrid` | `vector<float>` | 480 | Per-link bandwidth demand accumulation |
| `MulticastWriteLinkDemandGrid` | `vector<float>` | 480 | Multicast-specific link demand tracking |

**Initialization Logic:**

`std::unique_ptr<npeDeviceState> initDeviceState() const override {    size_t num_niu_types = niu_id_to_attr_lookup.size();    size_t num_links = link_id_to_attr_lookup.size();    return std::make_unique<npeDeviceState>(num_niu_types, num_links);}`
The sizes are derived from the lookup tables populated during construction, ensuring the state grids align with the device topology.

**Usage in Simulation:**

The device state is:

1.   **Initialized once** at the start of simulation by the engine
2.   **Reset to zero** at the beginning of each timestep
3.   **Populated** by `modelCongestion()` with bandwidth demands
4.   **Read** by `updateSimulationStats()` to compute utilization metrics

The state object is owned by the simulation engine and passed by reference to device model methods, avoiding repeated allocation/deallocation.

**Multi-chip Considerations:**

For multi-chip models (see [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.4-multi-chip-device-models)), the state grids scale proportionally:

```
Multi-chip NIU count = num_chips × 480
Multi-chip link count = num_chips × 480
```

This ensures each chip's resources are tracked independently within a unified state object.

## Class Hierarchy and Interface Compliance

The `WormholeB0DeviceModel` implements the `npeDeviceModel` interface, providing all required abstract methods for simulation engine integration.

**Key Implementation Methods:**

| Category | Methods | Purpose |
| --- | --- | --- |
| Routing | `route()`, `unicastRoute()` | Compute link-by-link NoC paths |
| Bandwidth | `computeCurrentTransferRate()`, `modelCongestion()` | Calculate effective transfer rates |
| Topology | `getRows()`, `getCols()`, `getCoreType()` | Query grid structure |
| Rates | `getSrcInjectionRate()`, `getSinkAbsorptionRate()` | Query core bandwidth limits |
| Lookups | `getLinkID()`, `getLinkAttributes()`, `getNIUID()`, `getNIUAttributes()` | Translate between IDs and attributes |
| State | `initDeviceState()` | Create runtime tracking structures |
| Architecture | `getArch()` | Return `DeviceArch::WormholeB0` |

**Static Utility Methods:**

The class also provides static utility methods for external use:

*   `route_hops()`: Calculate hop count for a given route
*   `get_read_latency()`: Compute read operation latency
*   `get_write_latency()`: Compute write operation latency

These are not part of the interface but are useful for analysis tools.

**Constructor Behavior:**

The default constructor performs full initialization:

1.   Allocates `coord_to_core_type` grid (12×10)
2.   Populates grid from `coord_to_core_type_map`
3.   Calls `populateNoCLinkLookups()` to create link ID mappings
4.   Calls `populateNoCNIULookups()` to create NIU ID mappings
