This page documents the two implementations of the Schedule interface in TVM's TensorIR system: `ConcreteScheduleNode` and `TracedScheduleNode`. These implementations provide different trade-offs between execution speed and schedule introspection/replay capabilities.

For information about the Schedule API methods themselves, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api). For details on the internal state management, see [Schedule State Management](https://deepwiki.com/tenstorrent/tt-tvm/5.4-schedule-state-management). For information on how traces are used in auto-tuning, see [Auto-Tuning System](https://deepwiki.com/tenstorrent/tt-tvm/8-auto-tuning-system).

## Purpose and Design Goals

The TensorIR schedule system provides two distinct implementations of the `Schedule` interface to serve different use cases:

*   **ConcreteSchedule**: Optimized for direct schedule execution with minimal overhead
*   **TracedSchedule**: Designed for recording, serialization, and replay of schedule transformations

Both implementations share the same public API defined in `ScheduleNode` but differ fundamentally in how they handle transformation history.

**Sources:**[include/tvm/tir/schedule/schedule.h 104-219](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/schedule/schedule.h#L104-L219)[src/tir/schedule/concrete_schedule.h 34-232](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L34-L232)[src/tir/schedule/traced_schedule.h 30-145](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.h#L30-L145)

## Class Hierarchy

**Inheritance Structure:**

*   `TracedScheduleNode` inherits from `ConcreteScheduleNode`, not directly from `ScheduleNode`
*   This means `TracedScheduleNode` reuses all the transformation logic from `ConcreteScheduleNode`
*   The only difference is that `TracedScheduleNode` wraps each operation to record it in the trace

**Sources:**[include/tvm/tir/schedule/schedule.h 104-112](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/schedule/schedule.h#L104-L112)[src/tir/schedule/concrete_schedule.h 34-65](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L34-L65)[src/tir/schedule/traced_schedule.h 30-48](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.h#L30-L48)

## ConcreteSchedule Implementation

`ConcreteScheduleNode` provides direct schedule transformation execution without recording history.

### Internal State

| Component | Type | Purpose |
| --- | --- | --- |
| `state_` | `ScheduleState` | The IR module and sref tree being transformed |
| `symbol_table_` | `Map<ObjectRef, ObjectRef>` | Maps random variables (BlockRV/LoopRV/ExprRV) to concrete entities (StmtSRef/Integer) |
| `analyzer_` | `std::unique_ptr<arith::Analyzer>` | Stateful arithmetic analyzer for simplification |
| `rand_state_` | `TRandState` | Random state for sampling operations |
| `error_render_level_` | `ScheduleErrorRenderLevel` | Controls error message detail (detail/fast/none) |
| `func_working_on_` | `Optional<GlobalVar>` | The function currently being scheduled |

**Sources:**[src/tir/schedule/concrete_schedule.h 42-53](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L42-L53)

### Execution Flow

**Direct Transformation:** Each schedule primitive directly modifies the `state_` without intermediate recording.

**Sources:**[src/tir/schedule/concrete_schedule.cc 400-504](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.cc#L400-L504)

### Symbol Table Management

The symbol table maps random variables to their concrete representations:

*   **BlockRV → StmtSRef**: Points to a block in the IR
*   **LoopRV → StmtSRef**: Points to a loop in the IR (or special markers: `InlineMark`, `RootMark`)
*   **ExprRV → Integer**: Evaluates to an integer value

**Sources:**[src/tir/schedule/concrete_schedule.h 39-49](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L39-L49)[src/tir/schedule/concrete_schedule.h 238-290](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L238-L290)[src/tir/schedule/concrete_schedule.h 338-380](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L338-L380)

### Copy Operation

When copying a `ConcreteSchedule`, the system performs a deep copy:

1.   **SRef Tree Reconstruction**: The entire StmtSRef tree is rebuilt with new objects
2.   **Symbol Table Update**: All random variables are remapped to new srefs
3.   **State Independence**: The copied schedule is completely independent

**Sources:**[src/tir/schedule/concrete_schedule.cc 50-207](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.cc#L50-L207)

## TracedSchedule Implementation

`TracedScheduleNode` extends `ConcreteScheduleNode` by adding trace recording capabilities.

### Additional State

| Component | Type | Purpose |
| --- | --- | --- |
| `trace_` | `Trace` | Records the sequence of schedule instructions and decisions |

The trace contains:

*   **Instructions**: Each schedule primitive call (kind, inputs, attrs, outputs)
*   **Decisions**: Sampling decisions for replay determinism

**Sources:**[src/tir/schedule/traced_schedule.h 34-48](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.h#L34-L48)

### Execution Flow with Tracing

**Wrapping Pattern:** Every schedule primitive in `TracedScheduleNode` follows this pattern:

1.   Call the base `ConcreteScheduleNode` implementation
2.   Create an `Instruction` object with the operation details
3.   Append the instruction to `trace_`
4.   Return the result

**Sources:**[src/tir/schedule/traced_schedule.cc 230-248](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.cc#L230-L248)

### Copy Operation with Trace

When copying a `TracedSchedule`:

1.   Performs all the copy operations of `ConcreteSchedule`
2.   **Copies the trace**: `Trace(this->trace_->insts, this->trace_->decisions)`
3.   Both schedules maintain independent traces

## Key Differences

### Comparison Table

| Aspect | ConcreteSchedule | TracedSchedule |
| --- | --- | --- |
| **Inheritance** | Directly inherits `ScheduleNode` | Inherits `ConcreteScheduleNode` |
| **Trace Property** | Returns `NullOpt` | Returns the recorded `Trace` object |
| **Memory Overhead** | Minimal (just state + symbol table) | Additional memory for instruction list |
| **Execution Speed** | Fastest (direct transformations) | Slightly slower (instruction recording) |
| **Serialization** | Not supported | Full serialization via trace |
| **Replay** | Not supported | Can replay from trace |
| **Auto-Tuning** | Not suitable | Designed for auto-tuning workflows |
| **Use in Production** | Manual scheduling scripts | Search-based optimization |

**Sources:**[src/tir/schedule/concrete_schedule.h 68-69](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L68-L69)[src/tir/schedule/traced_schedule.h 48](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.h#L48-L48)

### Performance Implications

**Overhead Analysis:**

*   **ConcreteSchedule**: O(1) overhead per operation (symbol table operations)
*   **TracedSchedule**: O(1) + trace recording overhead
*   For typical workloads, the difference is negligible compared to IR transformation cost

**Sources:**[src/tir/schedule/concrete_schedule.cc 400-504](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.cc#L400-L504)[src/tir/schedule/traced_schedule.cc 230-248](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.cc#L230-L248)

## Trace Structure

The `Trace` object in `TracedScheduleNode` consists of:

**Instruction Components:**

*   **kind**: Identifies the schedule primitive (e.g., "Split", "Fuse", "CacheRead")
*   **inputs**: Random variables passed as arguments
*   **attrs**: Primitive-specific attributes (e.g., split factors, storage scope)
*   **outputs**: Random variables produced by the primitive

**Decisions:**

*   Used for sampling primitives (`SamplePerfectTile`, `SampleCategorical`, etc.)
*   Ensures deterministic replay of stochastic choices
*   Essential for reproducible auto-tuning

**Sources:**[src/tir/schedule/traced_schedule.cc 59-116](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/traced_schedule.cc#L59-L116)

## Error Handling

Both implementations use the same error handling mechanism with different levels:

| Level | Description | Use Case |
| --- | --- | --- |
| `kDetail` | Full error with IR rendering | Development and debugging |
| `kFast` | Simple error message | Production auto-tuning |
| `kNone` | No error details | High-performance scenarios |

The error rendering is controlled by the `TVM_TIR_SCHEDULE_BEGIN()` and `TVM_TIR_SCHEDULE_END()` macros.
