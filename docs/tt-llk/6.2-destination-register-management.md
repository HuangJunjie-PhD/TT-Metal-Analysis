## Overview

The destination register management system controls addressing and synchronization for the Tensix destination register file, which stores computation results before packing to L1 memory. The system implements double-buffering via the global `dest_offset_id` variable, supports two synchronization modes (`DstSync::SyncFull` and `DstSync::SyncHalf`), and adjusts capacity based on `is_fp32_dest_acc_en` mode.

**Key Code Entities:**

*   Global variable: `dest_offset_id` (0 or 1)
*   Constant: `DEST_REGISTER_HALF_SIZE` (marks the halfway point in destination register space)
*   Enum: `DstSync` with values `SyncFull` and `SyncHalf`
*   Functions: `flip_packer_dest_offset_id()`, `get_packer_dest_offset()`, `select_packer_dest_registers<>()`
*   GPR indices: `p_gpr_pack::DEST_OFFSET_LO`, `p_gpr_pack::DEST_OFFSET_HI`

Related topics: Packer operations [6.1](https://deepwiki.com/tenstorrent/tt-llk/6.1-core-packer-operations-and-configuration), Math unit output [5.1](https://deepwiki.com/tenstorrent/tt-llk/5.1-core-math-unit-configuration).

**Sources:**[tt_llk_wormhole_b0/common/inc/cpack_common.h 594-607](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L594-L607)[tt_llk_wormhole_b0/common/inc/ckernel_globals.h](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel_globals.h)

## Destination Register Architecture and Key Constants

**Diagram: Destination Register File Structure**

**Core Constants:**

*   `DEST_REGISTER_HALF_SIZE`: Offset value separating the two halves of the destination register file. Exact value depends on architecture and accumulation mode.
*   In FP32 mode (`is_fp32_dest_acc_en=true`), effective capacity is halved.

**Sources:**[tt_llk_wormhole_b0/common/inc/cpack_common.h 594-625](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L594-L625)[tt_llk_wormhole_b0/common/inc/cmath_common.h 193-216](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cmath_common.h#L193-L216)[tt_llk_wormhole_b0/common/inc/ckernel_defs.h](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel_defs.h)

### Capacity Constraints by Mode

| `is_fp32_dest_acc_en` | `DstSync` Mode | Available Capacity per Section | `ZEROACC` Clear Mode |
| --- | --- | --- | --- |
| `false` (FP16) | `SyncFull` | Full register | `p_zeroacc::CLR_ALL` |
| `false` (FP16) | `SyncHalf` | `DEST_REGISTER_HALF_SIZE` tiles | `p_zeroacc::CLR_HALF` |
| `true` (FP32) | `SyncFull` | Half register (total) | `p_zeroacc::CLR_ALL_32B` |
| `true` (FP32) | `SyncHalf` | Quarter register (per section) | `p_zeroacc::CLR_HALF_32B` |

FP32 accumulation mode reduces capacity by 2x because each datum occupies 32 bits instead of 16 bits. The `dest_offset_id` modulo operation in `p_zeroacc::CLR_HALF` mode: `(dest_offset_id) % 2`.

**Sources:**[tt_llk_wormhole_b0/common/inc/cpack_common.h 224-316](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L224-L316)[tt_llk_wormhole_b0/llk_lib/llk_pack_common.h 34-59](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_common.h#L34-L59)

## DstSync Enum and Synchronization Modes

The `DstSync` enum (defined in `ckernel_defs.h`) controls synchronization behavior with two values: `SyncFull` and `SyncHalf`.

**Diagram: DstSync Mode Comparison**

### DstSync::SyncFull Mode Details

*   **Packer Offset:** Always `p_gpr_pack::DEST_OFFSET_LO` (offset 0x0)
*   **ZEROACC Mode:**`p_zeroacc::CLR_ALL` (16-bit) or `p_zeroacc::CLR_ALL_32B` (32-bit)
*   **Semaphore Init:**`SEMINIT(1, 0, SEMAPHORE_1)` - single buffer, math waits when packer is active
*   **No Flipping:**`dest_offset_id` not toggled; entire register used as one buffer

**Sources:**[tt_llk_wormhole_b0/llk_lib/llk_pack_common.h 39-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_common.h#L39-L43)[tt_llk_wormhole_b0/llk_lib/llk_math_common.h 84-88](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_common.h#L84-L88)

### DstSync::SyncHalf Mode Details

*   **Packer Offset:** Toggles based on `get_packer_dest_offset_index()`
*   **ZEROACC Mode:**`p_zeroacc::CLR_HALF` (16-bit) or `p_zeroacc::CLR_HALF_32B` (32-bit) with `(dest_offset_id) % 2` parameter
*   **Semaphore Init:**`SEMINIT(2, 0, SEMAPHORE_1)` - double buffer, allows overlap
*   **Flipping:**`flip_packer_dest_offset_id()` called in `_llk_pack_dest_section_done_()` and math's `dest_section_flip()`

**Sources:**[tt_llk_wormhole_b0/llk_lib/llk_pack_common.h 46-58](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_pack_common.h#L46-L58)[tt_llk_wormhole_b0/common/inc/cmath_common.h 193-201](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cmath_common.h#L193-L201)

**Diagram: Packer Offset Initialization Patterns**

## Unpack-to-Dest Direct Write Mode

For 32-bit input formats (`Int32`, `UInt32`, `Float32`) with lossless tilize, the unpacker writes directly to destination registers, bypassing source registers (SrcA/SrcB). This requires specialized destination address management.

**Diagram: Unpack-to-Dest Control Flow**

### Key Functions

**`set_dst_write_addr()`** ([tt_llk_wormhole_b0/common/inc/cunpack_common.h 442-463](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L442-L463)):

*   Calculates destination byte address with fixed offset: `16 * (4 + ThreadId)`
*   Disables address bit swizzle for direct addressing
*   Configures unpacker stride for destination write mode
*   Updates `THCON_SEC0_REG5_Dest_cntx0/1_address` and `THCON_SEC0_REG2_Unpack_if_sel_cntx0/1`

**`get_dest_max_matmul_tiles()`** - Calculates maximum dest index for matmul:

This function is critical for ensuring matmul operations don't exceed destination register capacity. The addressing pattern always results in a maximum offset of `ct_dim * rt_dim - 1` tiles from the starting index.
