# Page Table Walk Scenarios

Relevant source files
*   [coretp/isa/instruction.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instruction.py)

## Purpose and Scope

This document describes the comprehensive test scenarios for RISC-V virtual memory page table walks implemented in the coretp framework. These scenarios verify correct hardware behavior for address translation across all supported paging modes (SV39, SV48, SV57) and various fault conditions.

The scenarios test page table walk (PTW) behavior including: successful translations across different page sizes, boundary conditions for Virtual Page Numbers (VPN) and Physical Page Numbers (PPN), fault detection for misaligned superpages, reserved PTE encodings, permission violations, and global bit handling. For hardware-managed Accessed/Dirty bit testing, see [SVADU Test Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.1-svadu-test-scenarios). For general paging configuration concepts, see [Test Environment Configuration](https://deepwiki.com/tenstorrent/riscv-coretp/3.2-test-environment-configuration).

**Sources:**[coretp/plans/paging/page_table_walks.py 1-380](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L1-L380)

## Page Table Walk Overview

A page table walk is the hardware process of translating a virtual address (VA) to a physical address (PA) by traversing hierarchical page table structures. RISC-V defines three Sv modes for 64-bit systems:

| Paging Mode | VA Bits | Levels | Page Sizes Supported |
| --- | --- | --- | --- |
| SV39 | 39 | 3 | 4K, 2M, 1G |
| SV48 | 48 | 4 | 4K, 2M, 1G, 512G |
| SV57 | 57 | 5 | 4K, 2M, 1G, 512G, 256T |

The virtual address is decomposed into VPN (Virtual Page Number) fields and a page offset. Each VPN field indexes into a page table level to retrieve a Page Table Entry (PTE). PTEs contain:

*   PPN (Physical Page Number) - physical address of next level or final page
*   Permission bits - R (Read), W (Write), X (Execute), U (User)
*   Status bits - V (Valid), A (Accessed), D (Dirty), G (Global)

**Sources:**[coretp/plans/paging/page_table_walks.py 1-30](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L1-L30)[coretp/rv_enums.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/rv_enums.py) (PagingMode, PageSize, PageFlags enums)

## Test Scenario Structure

**Scenario Construction Pattern:** Each test scenario function is decorated with `@paging_scenario` and returns a `TestScenario` object. The function instantiates `Memory` objects defining virtual address regions with specific page sizes and permission flags, then creates operation steps (`Load`, `Store`, `Call`) that operate on those regions. Verification steps (`AssertException`) wrap operations expected to fault. Finally, `TestScenario.from_steps()` assembles all components with metadata (ID, name, description) and environment configuration specifying which paging modes and privilege levels to test.

**Sources:**[coretp/plans/paging/page_table_walks.py 32-56](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L32-L56)[coretp/plans/paging/page_table_walks.py 8](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L8-L8) (decorator import)

## Basic PTW Scenarios (No Faults)

### All Access Types (SID_PBVMS_001)

**Test Objective:** Verify successful page table walks for all three access types (instruction fetch, load, store) with valid permissions. The `Memory` object with `PageFlags.READ | PageFlags.WRITE | PageFlags.EXECUTE` ensures all access types are permitted. The scenario tests across all three paging modes (SV39, SV48, SV57).

**Sources:**[coretp/plans/paging/page_table_walks.py 32-56](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L32-L56)

### All Page Size Combinations (SID_PBVMS_002)

This scenario verifies PTW for all supported page sizes in each paging mode:

| Paging Mode | Page Sizes Tested |
| --- | --- |
| SV39 | 1G (`SIZE_1G`), 2M (`SIZE_2M`), 4K (`SIZE_4K`) |
| SV48 | 512G, 1G, 2M, 4K |
| SV57 | 256T, 512G, 1G, 2M, 4K |
`# Three Memory objects with different page sizesmem_1 = Memory(size=0x40000000, page_size=PageSize.SIZE_1G, flags=PageFlags.VALID | PageFlags.READ)mem_2 = Memory(size=0x200000, page_size=PageSize.SIZE_2M, flags=PageFlags.VALID | PageFlags.READ)mem_3 = Memory(size=0x10000, page_size=PageSize.SIZE_4K, flags=PageFlags.VALID | PageFlags.READ)`
The test creates separate `Load` operations for each memory region, forcing the hardware to perform PTW at different page table levels (leaf at level 2 for 1G, level 1 for 2M, level 0 for 4K in SV39).

**Sources:**[coretp/plans/paging/page_table_walks.py 59-80](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L59-L80)

## Boundary Value Testing

### VPN Boundary Values (SID_PBVMS_004)

Virtual Page Numbers are 9-bit fields in each page table level. This scenario tests VPN values at boundaries (0, max=511) and intermediate values:

The scenario generates `Load` operations at carefully calculated offsets to exercise VPN fields at their extremes. For example, `vpn2_max = (1 << 9) - 1` calculates the maximum 9-bit VPN value (511).

**Sources:**[coretp/plans/paging/page_table_walks.py 83-129](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L83-L129)

### PPN Boundary Values (SID_PBVMS_005)

Physical Page Numbers have different bit widths depending on page size (44-bit physical addresses in RISC-V):

| Page Size | PPN Width | Max PPN Value |
| --- | --- | --- |
| 1G | 14 bits (44-30) | `(1 << 14) - 1` |
| 2M | 23 bits (44-21) | `(1 << 23) - 1` |
| 4K | 32 bits (44-12) | `(1 << 32) - 1` |

The test creates `Memory` objects with `base_pa` set to minimum, maximum, and intermediate PPN values converted to physical addresses via the helper function:

`def ppn_to_pa(ppn, page_size):    return ppn * page_size`
This ensures PTW operates correctly across the full physical address space.

**Sources:**[coretp/plans/paging/page_table_walks.py 132-194](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L132-L194)

### Page Boundary Crossing (SID_PBVMS_006)

Tests memory accesses that span page boundaries to verify correct multi-page PTW behavior:

`mem = Memory(size=0x4000, page_size=PageSize.SIZE_4K, flags=..., num_pages=4)load1 = Load(memory=mem, offset=0xFF8)  # Access near end of first pageload2 = Load(memory=mem, offset=0x1000) # Access start of second pagestore = Store(memory=mem, offset=0xFF8, value=0xCAFE)`
The `num_pages=4` parameter creates a multi-page contiguous virtual region. Accesses at offset `0xFF8` (near 4K boundary) ensure the PTW must correctly translate adjacent pages.

**Sources:**[coretp/plans/paging/page_table_walks.py 197-214](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L197-L214)

## Fault Condition Testing

### Non-Leaf Recursive Walk Fault (SID_PBVMS_007)

Tests detection of invalid PTE pointer structures. The `ModifyPte` step with `make_recursive=True` alters a PTE to point back to itself, creating an infinite loop structure:

`mem0 = Memory(size=0x1000, page_size=PageSize.SIZE_4K, flags=..., modify=True)modify_pte = ModifyPte(memory=mem0, level=0, make_recursive=True)assert_exception = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,     code=[Load(memory=mem0, offset=0x1000)])`
The `modify=True` flag indicates this Memory object's PTEs will be modified after creation. Hardware must detect the invalid structure and raise `LOAD_PAGE_FAULT`.

**Sources:**[coretp/plans/paging/page_table_walks.py 217-233](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L217-L233)

### VA Sign Extension Fault (SID_PBVMS_008)

RISC-V requires that all bits above the valid VA range must equal the most significant valid VA bit (sign extension). For SV57, VA[63:57] must equal VA[56]:

`mem = Memory(size=0x10000, page_size=PageSize.SIZE_4K, flags=PageFlags.VALID | PageFlags.READ)# 0x8000000000000000 has bit 63 set but bit 56 clear - invalid sign extensionassert_exception = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,     code=[Load(memory=mem, offset=0x8000000000000000)])`
This scenario only tests SV57 mode where the sign extension check is most constrained.

**Sources:**[coretp/plans/paging/page_table_walks.py 236-250](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L236-L250)

### Superpage Alignment Fault (SID_PBVMS_009)

Superpages (2M, 1G, etc.) require PPN fields to be zero for bits corresponding to the page size. A 2M leaf PTE must have PPN[0] = 0 (9 zero bits), and a 1G leaf must have PPN[1:0] = 0 (18 zero bits):

`mem = Memory(size=0x200000, page_size=PageSize.SIZE_2M, flags=...)# offset 0x200001 creates misalignment - PPN[0] would be non-zeroassert_exception = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,    code=[Load(memory=mem, offset=0x200001)])`
Hardware must check superpage alignment and fault if PPN low bits are non-zero.

**Sources:**[coretp/plans/paging/page_table_walks.py 253-266](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L253-L266)

### Reserved PTE Values Fault (SID_PBVMS_010)

Certain PTE permission bit combinations are reserved and must cause faults. The encoding `V=1, R=0, W=1, X=0` (writable but not readable) is reserved:

`mem = Memory(size=0x10000, page_size=PageSize.SIZE_4K,              flags=PageFlags.VALID | PageFlags.WRITE)  # V=1, W=1, R=0, X=0assert_exception = AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,    code=[Load(memory=mem, offset=0x1000)])`
**Reserved PTE Encodings:**

| V | R | W | X | Interpretation |
| --- | --- | --- | --- | --- |
| 0 | x | x | x | Invalid page |
| 1 | 0 | 0 | 0 | Pointer to next level |
| 1 | 0 | 1 | x | **Reserved** (tested) |
| 1 | 1 | x | x | Leaf page |

**Sources:**[coretp/plans/paging/page_table_walks.py 269-282](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L269-L282)

### Boundary Crossing Fault Combinations (SID_PBVMS_011)

Tests fault behavior when multi-byte accesses span pages with different permission settings:

**Combination 1:** Both pages valid - access succeeds

`mem1_first = Memory(num_pages=2, size=0x2000, page_size=PageSize.SIZE_4K,                    flags=PageFlags.VALID | PageFlags.READ | PageFlags.WRITE)load1 = Load(memory=mem1_first, offset=0xFFA, access_size=8)  # Spans two pages`
**Combination 2:** First page lacks read permission - access faults

`mem2_first = Memory(num_pages=2, size=0x2000, page_size=PageSize.SIZE_4K,                    flags=PageFlags.VALID)  # No READ permissionload2 = Load(memory=mem2_first, offset=0xFFA, access_size=8)AssertException(cause=ExceptionCause.LOAD_PAGE_FAULT, code=[load2])`
The 8-byte load at offset `0xFFA` accesses bytes `0xFFA-0x1001`, spanning the 4K boundary at `0x1000`.

**Sources:**[coretp/plans/paging/page_table_walks.py 285-305](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L285-L305)

## Permission and Access Control Testing

### Permission Encodings (SID_PBVMS_013)

This scenario creates three memory regions with different permission combinations and performs operations consistent with those permissions. The `TestEnvCfg` includes all privilege modes (`U`, `S`, `M`) to verify permission checks work correctly across different execution contexts.

Note: The scenario focuses on valid permission cases. Invalid cases (e.g., writing to read-only memory) would use `AssertException` wrappers.

**Sources:**[coretp/plans/paging/page_table_walks.py 326-351](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L326-L351)

### Global Bit Handling (SID_PBVMS_015)

The Global bit (G) in PTEs indicates the translation is valid for all Address Space IDs (ASIDs). When G=1, TLB entries should remain valid across ASID changes:

The `satp` CSR format in Sv39 mode:

*   Bits [63:60]: MODE (paging mode)
*   Bits [59:44]: ASID (Address Space ID)
*   Bits [43:0]: PPN (Page table base physical address)

The test manipulates the ASID field (bits 59:44) using bit masks `0xF << 44` while the Global bit ensures the translation remains valid without SFENCE.VMA.

**Sources:**[coretp/plans/paging/page_table_walks.py 354-379](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L354-L379)

## Test Step Composition Patterns

### Memory-Centric Pattern

Most PTW scenarios follow this pattern:

1.   **Define memory regions** with specific properties
2.   **Create operations** that reference those regions
3.   **Optionally modify PTEs** for fault injection
4.   **Wrap faulting operations** in `AssertException`
5.   **Configure environment** for paging modes and privilege levels

**Key Classes Used:**

*   `Memory`[coretp/step/memory.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py) - Virtual memory region definition
*   `CodePage`[coretp/step/memory.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py) - Executable code regions
*   `Load`, `Store`[coretp/step/memory.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py) - Memory access operations
*   `ModifyPte`[coretp/step/memory.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py) - PTE manipulation for fault injection
*   `AssertException`[coretp/step/assertions.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/assertions.py) - Exception verification
*   `CsrRead`, `CsrWrite`[coretp/step/csr.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/csr.py) - CSR manipulation
*   `TestEnvCfg`[coretp/env/cfg.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/env/cfg.py) - Environment configuration

**Sources:**[coretp/plans/paging/page_table_walks.py 1-380](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/paging/page_table_walks.py#L1-L380)[coretp/step/__init__.py 1-10](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/__init__.py#L1-L10)

### Common Parameters

**Memory Object Parameters:**

`Memory(    size=<bytes>,                    # Total size of virtual region    page_size=PageSize.SIZE_4K,      # Page granularity    flags=PageFlags.VALID | ...,     # Permission/status bits    base_va=<optional_va>,           # Specific virtual address    base_pa=<optional_pa>,           # Specific physical address    num_pages=<count>,               # Multi-page regions    modify=<bool>                    # Allow PTE modifications)`
**Load/Store Parameters:**

`Load(memory=<Memory>, offset=<int>, access_size=<bytes>)Store(memory=<Memory>, offset=<int>, value=<int>, access_size=<bytes>)`
**Exception Assertion Parameters:**

`AssertException(    cause=ExceptionCause.LOAD_PAGE_FAULT,  # Expected exception type    code=[<operation_list>]                # Operations that should fault)`
