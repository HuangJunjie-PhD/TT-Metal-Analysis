The Worker Architecture in vLLM is responsible for managing model execution on GPU devices. This page documents the `Worker` and `GPUModelRunner` classes that handle model execution and memory management on GPUs. These components form the execution layer that runs the actual inference computation.

For information about the memory management system including PagedAttention and KV cache, see [Memory Management](https://deepwiki.com/tenstorrent/vllm/2.2-engine-components).

## Worker Component Overview

Sources:

*   [vllm/v1/worker/gpu_worker.py 34-89](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L34-L89)

The Worker Architecture consists of two main components:

1.   **Worker**: Manages the GPU device, initializes the distributed environment, and handles memory management
2.   **GPUModelRunner**: Executes the model, prepares input batches, and manages KV cache

These components work together to efficiently execute inference requests on GPU hardware.

## Worker Class

The `Worker` class serves as the main interface for model execution on GPU devices. It inherits from `WorkerBase` and provides methods for device initialization, model loading, and inference execution.

Sources:

*   [vllm/v1/worker/gpu_worker.py 34-85](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L34-L85)
*   [vllm/v1/worker/gpu_worker.py 90-268](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L90-L268)

### Key Methods

*   **init_device()**: Initializes the GPU device, sets up CUDA environment, and initializes the distributed environment for multi-GPU execution.

*   **load_model()**: Loads the model into GPU memory, optionally using a memory pool when sleep mode is enabled.

*   **determine_available_memory()**: Profiles peak memory usage of the model to determine how much memory is available for KV cache.

*   **initialize_from_config()**: Allocates GPU KV cache with the configuration determined by the memory profiling.

*   **execute_model()**: The main execution method that receives scheduler output and forwards it to the model runner.

*   **sleep()** and **wake_up()**: Enable efficient memory management by releasing and restoring model weights.

## GPUModelRunner Class

The `GPUModelRunner` class is responsible for the actual model execution, managing KV cache, preparing input batches, and handling token generation. It's the core class that performs the computational work.

Sources:

*   [vllm/v1/worker/gpu_model_runner.py 60-246](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L60-L246)

### Key Methods

*   **_update_states()**: Updates cached request states and the persistent batch with scheduler output. Handles request additions, removals, and updates.

*   **_prepare_inputs()**: Prepares input tensors for model execution, including token IDs, positions, and slot mappings for the KV cache.

*   **execute_model()**: Executes the model with prepared inputs and produces outputs for token generation.

*   **initialize_kv_cache()**: Initializes KV cache tensors according to the provided configuration.

## Execution Flow

The execution flow in the Worker architecture involves several steps to process batched inference requests efficiently:

Sources:

*   [vllm/v1/worker/gpu_worker.py 238-243](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L238-L243)
*   [vllm/v1/worker/gpu_model_runner.py 275-462](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L275-L462)
*   [vllm/v1/worker/gpu_model_runner.py 464-617](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_model_runner.py#L464-L617)
*   [vllm/v1/executor/abstract.py 73-79](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/executor/abstract.py#L73-L79)

The execution process involves these steps:

1.   The Executor sends a scheduler output to the Worker via RPC
2.   The Worker forwards it to the ModelRunner
3.   The ModelRunner updates its cached states based on scheduler output
4.   The ModelRunner prepares input tensors for model execution
5.   The model processes the inputs and produces logits
6.   The ModelRunner processes the logits and returns a ModelRunnerOutput
7.   The Worker returns the output to the Executor

## Input Batch Preparation

The ModelRunner prepares input batches for efficient model execution through the `_prepare_inputs()` method. This is a critical step that organizes the computation for maximum efficiency.

Key operations performed during input preparation:

1.   **Batch Reordering**: The batch may be reordered based on attention backend requirements

 
2.   **Block Table Setup**: The KV cache block table is set up to map token positions to memory locations

 
3.   **Position Calculation**: Token positions are calculated based on computed tokens and offsets

 
4.   **Slot Mapping**: Maps tokens to their corresponding slots in the KV cache

 
5.   **Attention Metadata**: Creates attention metadata for efficient computation

### Sleep/Wake Functionality

One of the unique features of the Worker architecture is its ability to conserve GPU memory through sleep/wake functionality:

*   **sleep(level)**: Releases GPU memory to increase available memory

    *   level=1: Offloads only model weights
    *   level=2: Offloads all memory including KV cache

*   **wake_up(tags)**: Restores offloaded memory

    *   tags=['weights']: Restores only model weights
    *   tags=['kv_cache']: Restores only KV cache memory

This functionality is particularly useful in deployment scenarios where multiple models need to share the same GPU.

### Memory Profiling

The Worker performs memory profiling to determine how much memory is available for KV cache:

1.   Empty GPU cache and reset peak memory stats
2.   Run a profile forward pass with dummy inputs
3.   Measure peak memory usage and non-torch allocations
4.   Calculate available memory for KV cache based on total GPU memory and utilization factor
