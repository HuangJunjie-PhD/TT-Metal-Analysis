# Chip Detection and Initialization

This page documents the process by which Luwen detects Tenstorrent AI accelerator chips and initializes them for operation. It covers the detection of local (PCI-attached) and remote (Ethernet-connected) chips, the initialization sequence for chip components, and error handling mechanisms.

For details on the communication protocols used after initialization, see [Communication Interfaces](https://deepwiki.com/tenstorrent/luwen/2.3-communication-interfaces).

## Overview

Chip detection and initialization is a multi-stage process that begins with scanning for local PCI devices, initializing those devices, and then using those devices to discover additional remote chips accessible via Ethernet connections.

Sources: [crates/luwen-if/src/detect_chips.rs 175-247](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L175-L247)[crates/luwen-ref/src/detect.rs 238-307](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/detect.rs#L238-L307)

## Detection Architecture

The detection system is built around several key components that work together to discover and initialize all available chips.

Sources: [crates/luwen-if/src/detect_chips.rs 14-173](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L14-L173)[crates/luwen-if/src/chip/init/status.rs 371-478](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L371-L478)

## Detection Flow

### 1. PCI Device Discovery

The detection process begins with scanning for physical PCI devices connected to the system.

The detection begins with `PciDevice::scan()` which returns a list of device IDs for PCI devices. For each device ID, an `ExtendedPciDevice` is opened and a `Chip` object is created. Basic PCI communication is tested, and if successful, the chip is added to the list of root chips.

Sources: [crates/luwen-ref/src/detect.rs 238-285](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/detect.rs#L238-L285)

### 2. Initialization of Root Chips

Once the PCI devices are identified, each root chip undergoes an initialization process.

The `wait_for_init` function orchestrates the initialization process for a single chip. It repeatedly calls `update_init_state` until initialization is complete or an error occurs.

Sources: [crates/luwen-if/src/chip/init/mod.rs 52-122](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/mod.rs#L52-L122)

### 3. Component Initialization Status Tracking

Each chip has multiple components that must be initialized. The status of each component is tracked using the `InitStatus` structure.

The `InitStatus` structure contains status information for all components of a chip (ARC, DRAM, Ethernet, CPU). Each component's status is tracked using a `ComponentStatusInfo` structure, which contains an array of `WaitStatus` values.

Sources: [crates/luwen-if/src/chip/init/status.rs 371-478](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L371-L478)[crates/luwen-if/src/chip/init/status.rs 167-339](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L167-L339)

### 4. Remote Chip Discovery

After root chips are initialized, they can be used to discover remote chips accessible via Ethernet.

For each root chip, the system retrieves a list of neighbouring chips. It then iterates through these chips, opening a remote connection to each one and initializing it. It also retrieves the neighbouring chips of each remote chip, continuing the discovery process recursively.

Sources: [crates/luwen-if/src/detect_chips.rs 251-385](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L251-L385)

## Initialization Components

### ARC Initialization

The ARC (Asynchronous RISC Core) is a critical component that must be initialized for the chip to function properly.

ARC initialization involves checking access to the ARC, verifying the firmware status, checking the firmware version, and waiting for the initialization to complete.

Sources: [crates/luwen-if/src/chip/init/status.rs 41-71](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L41-L71)

### DRAM Initialization

DRAM must be properly trained for reliable operation.

DRAM initialization primarily involves checking the training status of each DRAM channel. Channels can be in various states, such as `TrainingPass` (successful), `TrainingFail` (failed), or `TrainingNone` (not yet trained).

Sources: [crates/luwen-if/src/chip/init/status.rs 73-128](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L73-L128)

### Ethernet Initialization

Ethernet initialization is required for remote chip discovery and communication.

Ethernet initialization involves checking the firmware integrity and verifying that the Ethernet interfaces are properly trained.

Sources: [crates/luwen-if/src/chip/init/status.rs 10-38](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/init/status.rs#L10-L38)

## UninitChip API

The `UninitChip` enum provides a safe way to handle chips that may not be fully initialized.

The `UninitChip` enum can be in one of two states:

1.   `Partially` - The chip is partially initialized and may not be safe to use.
2.   `Initialized` - The chip is fully initialized and ready for use.

The API provides methods to check the status of the chip and to safely upgrade a `UninitChip` to a `Chip` when it's ready.

Sources: [crates/luwen-if/src/detect_chips.rs 21-173](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L21-L173)

## Detection API

The Luwen codebase provides several functions for detecting and initializing chips:

| Function | Description |
| --- | --- |
| `detect_chips` | Detects all chips, both local and remote, and returns a list of `UninitChip` objects. |
| `detect_initialized_chips` | Detects and initializes all chips, returning a list of fully initialized `Chip` objects. |
| `detect_chips_silent` | Detects chips without producing progress output, returning a list of `Chip` objects. |
| `detect_chips_options` | Detects chips with custom options, returning a list of `UninitChip` objects. |
| `detect_local_chips` | Detects only local (PCI-attached) chips, returning a list of `Chip` objects. |

These functions provide different levels of control over the detection and initialization process.

## C++ Bindings

Luwen provides C++ bindings for chip detection and initialization. These bindings allow C++ applications to use the Luwen API to detect and initialize Tenstorrent chips.

The C++ bindings provide functions for opening local and remote chips:

*   `luwen_open`: Opens a local chip
*   `luwen_open_remote`: Opens a remote chip accessible via a local chip

These functions use a callback mechanism (`LuwenGlue`) to communicate with the hardware, allowing for flexibility in how the hardware is accessed.

Sources: [crates/luwencpp/src/lib.rs 68-276](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwencpp/src/lib.rs#L68-L276)[crates/luwencpp/src/lib.rs 277-320](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwencpp/src/lib.rs#L277-L320)

## Summary

The chip detection and initialization process in Luwen provides a flexible and robust way to discover and initialize Tenstorrent AI accelerator chips. The process includes:

1.   Scanning for local PCI devices
2.   Initializing these devices to create root chips
3.   Using the root chips to discover remote chips via Ethernet
4.   Tracking the initialization status of each component (ARC, DRAM, Ethernet, CPU)
5.   Providing a safe API to handle chips that may not be fully initialized

The system is designed to handle errors gracefully, with options to continue detection even when some chips fail to initialize. It also provides a callback mechanism to report progress and status during the initialization process.
