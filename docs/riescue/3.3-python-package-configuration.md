## Purpose and Scope

This document provides a comprehensive reference for the Python package configuration in RiESCUE, as defined in `pyproject.toml`. It covers the build system, project metadata, runtime dependencies, package structure, included data files, command-line entry points, and development tool configurations.

For information about the container environment and system-level dependencies, see [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system). For details on the build scripts that compile external tools, see [Build Scripts and Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2-build-scripts-and-dependencies). For information about using the command-line tools, see [Core Testing Framework](https://deepwiki.com/tenstorrent/riescue/2-core-testing-framework).

## Package Metadata and Build System

The RiESCUE package uses modern Python packaging standards with `pyproject.toml` as the single source of configuration truth. The package is named `riescue` with version `1.2.0` and requires Python 3.9 or later.

### Build Backend Configuration

The build system uses `setuptools` as the backend:

`[build-system]requires = ["setuptools >= 64.0.3", "pip", "wheel"]build-backend = "setuptools.build_meta"`
This configuration specifies that setuptools version 64.0.3 or higher is required to build the package, along with `pip` and `wheel` as build dependencies.

**Sources:**[pyproject.toml 7-9](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L7-L9)

### Project Metadata

The project metadata section defines core package information:

| Field | Value | Description |
| --- | --- | --- |
| `name` | `riescue` | PyPI package name |
| `version` | `1.2.0` | Current package version |
| `description` | `RISC-V Directed Test Framework and Compliance Suite` | Short description |
| `readme` | `README.md` | Long description source |
| `requires-python` | `>= 3.9` | Minimum Python version |

## Package Structure and Discovery

The package structure defines how Python modules are organized and discovered during installation.

### Package Directory Configuration

**Package Directory Mapping Diagram**

The configuration maps the source directory to the installed package name:

`[tool.setuptools.package-dir]riescue = "riescue/" [tool.setuptools.packages.find]where = ["."]`
This tells setuptools to:

1.   Look for packages starting at the repository root (`where = ["."]`)
2.   Map the `riescue/` source directory to the `riescue` package name
3.   Automatically discover all subpackages recursively

**Sources:**[pyproject.toml 31-35](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L31-L35)

### Package Data Inclusion

Non-Python files must be explicitly declared to be included in the installed package. The configuration specifies data files for various submodules:

**Package Data Files Diagram**

The package data configuration:

## Command-Line Entry Points

The package exposes three command-line tools via console script entry points. These are automatically created as executable scripts when the package is installed.

### Entry Point Configuration

**Entry Point to Module Mapping Diagram**

### Entry Point Details

| Entry Point | Module Path | Function | Description |
| --- | --- | --- | --- |
| `riescued` | `riescue.riescued` | `main()` | RiescueD directed test framework CLI |
| `riescuec` | `riescue.riescuec` | `main()` | RiescueC compliance test generator CLI |
| `ctk` | `riescue.ctk` | `main()` | CTK compliance test kit high-level interface |

When a user runs `riescued`, Python:

1.   Locates the installed `riescued` script in the PATH
2.   Imports the `riescue.riescued` module
3.   Executes the `main()` function from that module

This mechanism provides a clean separation between the installed command names and the underlying Python module structure. For detailed usage of these commands, see:

*   [RiescueD Command-Line Interface](https://deepwiki.com/tenstorrent/riescue/2.1.1-riescued-command-line-interface)
*   [RiescueC: Compliance Test Generator](https://deepwiki.com/tenstorrent/riescue/2.2-riescuec:-compliance-test-generator)
*   [CTK: Compliance Test Kit](https://deepwiki.com/tenstorrent/riescue/2.3-ctk:-compliance-test-kit)

**Sources:**[pyproject.toml 46-49](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L46-L49)

## Tool Configuration

The `pyproject.toml` file also contains configuration for development tools. These tools are configured here for centralized management and consistency across the development team.

### Black Code Formatter Configuration

**Black Formatter Configuration Diagram**

The Black formatter is configured with:

*   **Line length**: 200 characters (significantly longer than the default 88)
*   **Workers**: 8 parallel processes for faster formatting
*   **Excluded paths**: External dependencies, generated code, and virtual environments

The exclusion pattern uses a regular expression to ignore directories that contain external code or generated artifacts.

**Sources:**[pyproject.toml 1-4](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L1-L4)

### Pyright Type Checker Configuration

Pyright is configured to perform static type checking on a subset of the codebase:

**Pyright Type Checking Scope Diagram**

Key configuration details:

| Setting | Value | Purpose |
| --- | --- | --- |
| `typeCheckingMode` | `"standard"` | Balanced type checking strictness (not `"strict"`) |
| `pythonVersion` | `"3.9"` | Matches minimum required Python version |
| `pythonPlatform` | `"Linux"` | Target platform for type checking |
| `extraPaths` | `["riescue"]` | Additional paths for import resolution |
| `reportMissingTypeStubs` | `false` | Don't warn about missing type stubs for dependencies |

**Included Paths:**

*   `riescue/lib`: Core library utilities
*   `riescue/*.py`: Top-level modules (riescued, riescuec, ctk)
*   `riescue/dtest_framework`: RiescueD implementation
*   `tests/dtest_framework`: RiescueD tests
*   `riescue/compliance/*.py` and submodules: RiescueC implementation

**Excluded Paths:**

*   `tests/compliance`: Compliance test outputs (not source code)
*   `riescue/dtest_framework/generator/generator.py`: Temporarily excluded due to type issues (marked with FIXME)

The configuration includes a note about potentially migrating to strict mode in the future using per-file strictness settings.

**Sources:**[pyproject.toml 51-80](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L51-L80)

## Package Installation and Editable Mode

The package is typically installed in editable mode during development, which allows changes to source files to take effect immediately without reinstalling:

`pip install -e .`
This command:

1.   Reads the `pyproject.toml` configuration
2.   Installs all dependencies from the `dependencies` array
3.   Creates symlinks to the source directory instead of copying files
4.   Installs entry point scripts (`riescued`, `riescuec`, `ctk`) in the PATH

After installation, the three command-line tools are available system-wide (or within the active virtual environment), and any changes to Python source files in `riescue/` are immediately reflected when running the commands.

For container-based development, the package is installed during the container build process (see [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system)). For local development outside containers, see [Installation and Quick Start](https://deepwiki.com/tenstorrent/riescue/1.1-installation-and-quick-start).
