## Purpose and Scope

This page documents the type system and type conversion infrastructure used during the TTL to TTKernel compilation pipeline. The type converter (`TTLToTTKernelTypeConverter`) transforms high-level TTL dialect types into low-level TTKernel types that map directly to hardware operations. Understanding this conversion is essential for comprehending how Python DSL types are lowered to executable kernel code and how data formats are managed across the compute engines.

For the broader compilation pipeline context, see [Pipeline Overview](https://deepwiki.com/tenstorrent/tt-lang/3.1-pipeline-overview). For TTL dialect type definitions, see [TTL Dialect Specification](https://deepwiki.com/tenstorrent/tt-lang/11.1-ttl-dialect-specification). For TTKernel dialect types, see [TTKernel Dialect Specification](https://deepwiki.com/tenstorrent/tt-lang/11.2-ttkernel-dialect-specification).

## TTLToTTKernelTypeConverter

The `TTLToTTKernelTypeConverter` class defines conversion rules for TTL types. It is instantiated during the TTL to TTKernel lowering pass and used by all conversion patterns.

### Conversion Rules

#### CircularBufferType Conversion

The `CircularBufferType` represents a multi-dimensional circular buffer in L1. It is flattened to a TTKernel `CBType` with a total element count.

*   **Logic**: Uses `CircularBufferType::getTotalElements()`, which multiplies shape dimensions by the `blockCount`[include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 46-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L46-L49)
*   **Target**: Lowered to `!ttkernel.cb<total_elements, element_type>`[include/ttlang/Dialect/Utils/ConversionUtils.h 79-109](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/Utils/ConversionUtils.h#L79-L109)

#### RankedTensorType Conversion

Tensors carrying a `tt::ttl::LayoutAttr` encoding [include/ttlang/Dialect/TTL/IR/TTLOpsAttrs.td 44-74](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsAttrs.td#L44-L74) are identified by the `TTLLayoutTensor` constraint [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 18-25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L18-L25) These are lowered to `!ttkernel.tensor_accessor` for DMA access.

#### TransferHandleType Preservation

The `TransferHandleType` represents an asynchronous transfer handle. It is preserved during conversion to allow `ttl.wait` operations to inspect the `TransferKind` (read vs write) [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 52-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L52-L58)

## Data Type Conversion (Typecasting)

The TTL dialect supports explicit data type conversion via `ttl.typecast` (tensor-level) and `ttl.tile_typecast` (tile-level).

### Tile Typecasting Lowering

The `ttl.tile_typecast` operation changes the element data type of a tile resident in a DST register. It is restricted to floating-point formats supported by the SFPU [test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir 30-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir#L30-L49)

**Key Behaviors:**

1.   **In-place Operation**: Typecasting is performed in-place within a DST register slot [test/ttlang/Conversion/TTLToTTKernel/typecast.mlir 57-74](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/typecast.mlir#L57-L74)
2.   **Initialization**: Lowering inserts `ttkernel.typecast_tile_init` which carries both input and output dtypes [test/ttlang/Conversion/TTLToTTKernel/typecast.mlir 5-9](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/typecast.mlir#L5-L9)
3.   **Deduplication**: Consecutive typecasts with the same dtype pair share a single init op [test/ttlang/Conversion/TTLToTTKernel/typecast.mlir 25-35](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/typecast.mlir#L25-L35)
4.   **Identity Folding**: Typecasts where input and output dtypes match are treated as no-ops and folded [test/ttlang/Dialect/TTL/IR/typecast.mlir 55-65](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast.mlir#L55-L65)

Sources: [test/ttlang/Conversion/TTLToTTKernel/typecast.mlir 1-96](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/typecast.mlir#L1-L96)[test/ttlang/Dialect/TTL/IR/typecast.mlir 1-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast.mlir#L1-L66)[test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir 1-50](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir#L1-L50)

## Operation Verification and Indexing

### Accumulation Scopes

The `ttl.accumulation_scope` defines an accumulation policy for one or more destination tensors. It supports three initial modes:

1.   **overwrite**: The first contribution defines the accumulator.
2.   **accumulate_existing**: Existing values in output participate.
3.   **init**: An explicit init operand seeds the accumulator.

The verifier ensures that the number of initial modes, init operands, and body block arguments match the number of outputs [test/ttlang/Dialect/TTL/IR/accumulation_scope_invalid.mlir 7-18](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/accumulation_scope_invalid.mlir#L7-L18)
