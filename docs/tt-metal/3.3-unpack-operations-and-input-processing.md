## Purpose and Scope

This page documents the **unpack operations** in the Low-Level Kernel (LLK) API, which are responsible for reading input data from circular buffers in L1 memory and preparing it for compute operations. Unpack operations transfer data into the SRCA and SRCB input registers of the Tensix compute pipeline, applying necessary format conversions and layout transformations.

Unpack operations complement **pack operations** (covered in [Pack Operations and Output Processing](https://deepwiki.com/tenstorrent/tt-metal/3.4-pack-operations-and-output-processing)) which write computed results back to memory. For information about the math operations that consume unpacked data, see [Math and Compute Operations](https://deepwiki.com/tenstorrent/tt-metal/3.5-math-and-compute-operations). The initialization sequence for compute kernels is described in [Compute Kernel Initialization and Startup](https://deepwiki.com/tenstorrent/tt-metal/3.2-compute-kernel-initialization-and-startup).

**Key Responsibilities:**

*   Reading tiles from circular buffers.
*   Converting between data formats (source format to compute format).
*   Handling broadcast patterns (ROW, COL, SCALAR).
*   Tilization (row-major to tile layout conversion) and Untilization.
*   Configuring hardware unpacker units.

Single operand unpacking reads data from a single circular buffer into the SRCA register. It is primarily used for unary operations or as a source operand in more complex computation kernels.

### Address Calculation for Tiles

The L1 SRAM address for the target tile is computed using circular buffer metadata:

| Component | Calculation | Purpose |
| --- | --- | --- |
| `base_address` | `get_local_cb_interface(id).fifo_rd_ptr - 1` | Starting read address (header removed) |
| `offset_address` | `fifo_page_size * tile_index` | Byte offset for the tile index |
| `address` | `base_address + offset_address` | Final address of tile data |

## Tilization and Untilization

### Untilization Operations

Untilization converts data from tile layout back to row-major format. This is often used when the output needs to be in row-major for host consumption or specific ops.

*   **`llk_unpack_untilize_init`**: Sets up the hardware for the specified operand [tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_untilize_api.h 13-19](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_untilize_api.h#L13-L19)
*   **`llk_unpack_untilize`**: Performs untilization passes by calling the underlying implementation `_llk_unpack_untilize_`[tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_untilize_api.h 21-41](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_untilize_api.h#L21-L41)

## Data Format and Broadcast Handling

The LLK system handles a wide variety of data formats (BFP8, Float16, Float32, Int8) and broadcast patterns to support diverse tensor operations.

### Broadcast Types

Broadcast operations are handled during the unpack stage, allowing a smaller tensor (e.g., a vector or scalar) to be applied to a larger tensor.

| Broadcast Type | Code Identifier | Behavior |
| --- | --- | --- |
| **Row** | `BroadcastType::ROW` | Replicates a row across the tile [tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_unpack_AB_api.h 67-68](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_unpack_AB_api.h#L67-L68) |
| **Column** | `BroadcastType::COL` | Replicates a column across the tile [tests/tt_metal/tt_metal/llk/test_broadcast.cpp 77](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tt_metal/llk/test_broadcast.cpp#L77-L77) |
| **Scalar** | `BroadcastType::SCALAR` | Replicates a single element across the entire tile [tests/tt_metal/tt_metal/llk/test_broadcast.cpp 78](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tt_metal/llk/test_broadcast.cpp#L78-L78) |

## Configuration and Reconfiguration

### Hardware Configuration

The primary entry point for configuring unpacker hardware is `llk_unpack_hw_configure`. This function derives face geometry and tile size directly from the Circular Buffer (CB) metadata.

### Data Format Reconfiguration

Kernels can reconfigure the unpacker data format mid-execution if different circular buffers have different formats.

*   **`are_unpackers_AB_configured_correctly`**: A block assertion used in dual-operand APIs to ensure format and dimension compatibility between SRCA and SRCB [tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_AB_api.h 19-27](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/ckernels/blackhole/metal/llk_api/llk_unpack_AB_api.h#L19-L27)
*   **`WAYPOINT`**: Used for tracking kernel execution progress through unpack stages (e.g., "UPAW" for Unpack A Wait, "UPAD" for Unpack A Done) [tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_unpack_A_api.h 64-67](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_unpack_A_api.h#L64-L67)
