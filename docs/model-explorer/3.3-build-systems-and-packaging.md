This document covers the build systems and packaging infrastructure for the Model Explorer backend components. This includes Bazel workspace configuration for C++ compilation, Python setuptools packaging for wheel distribution, and the build automation scripts that coordinate the entire process.

For information about the Python server runtime and APIs, see [Python Server and APIs](https://deepwiki.com/tenstorrent/model-explorer/3.2-python-server-and-apis). For frontend build processes, see [Frontend Build and Configuration](https://deepwiki.com/tenstorrent/model-explorer/4.4-frontend-build-and-configuration).

## Build System Architecture

The Model Explorer backend uses a hybrid build system combining Bazel for C++ compilation and Python setuptools for distribution packaging. The system produces installable Python wheels containing native C++ extensions.

Sources: [src/builtin-adapter/python/pip_package/build_pip_package.sh 1-133](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/python/pip_package/build_pip_package.sh#L1-L133)[src/builtin-adapter/WORKSPACE 1-166](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/WORKSPACE#L1-L166)[src/builtin-adapter/python/pip_package/setup_with_binary.py 1-72](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/python/pip_package/setup_with_binary.py#L1-L72)

## Bazel Workspace Configuration

The Bazel workspace is configured through the `WORKSPACE` file which defines external dependencies, primarily TensorFlow and its ecosystem. The workspace uses hermetic Python initialization to ensure reproducible builds across environments.

Sources: [src/builtin-adapter/WORKSPACE 18-20](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/WORKSPACE#L18-L20)[src/builtin-adapter/WORKSPACE 43-52](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/WORKSPACE#L43-L52)[src/builtin-adapter/WORKSPACE 64-98](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/builtin-adapter/WORKSPACE#L64-L98)

### Bazel Configuration Options

The `.bazelrc` file defines comprehensive build configurations for different platforms and optimization levels. Key configurations include compiler flags, platform-specific settings, and build optimization options.

| Configuration | Purpose | Key Options |
| --- | --- | --- |
| `monolithic` | Single shared object build | `--define framework_shared_object=false` |
| `dbg` | Debug builds | `-c dbg`, debug symbol control |
| `android` | Android cross-compilation | `--crosstool_top=//external:android/crosstool` |
| `windows` | Windows-specific settings | `/std:c++20`, MSVC compatibility |
| `linux_arm64` | ARM64 Linux builds | `--copt="-mtune=generic" --copt="-march=armv8-a"` |

The build system supports multiple platforms with specific naming conventions for wheel distribution:

| --- | --- | --- | --- |
| macOS | x86_64 | `macosx_10_15_x86_64` | `.so` |

| Linux | x86_64 | `manylinux_2_17_x86_64` | `.so` |
| Linux | aarch64 | `manylinux_2_17_aarch64` | `.so` |
| Windows | x86_64 | (platform-default) | `.pyd` |

## Dependency Management

The build system manages complex dependencies through multiple layers, ensuring compatibility between TensorFlow, XLA, and Model Explorer components.

### TensorFlow Integration

The system pins to specific TensorFlow commits to ensure reproducible builds:

*   **TensorFlow Commit**: `1e03bc9bb2fdccdb58a1a809e3358030b9e498b2`
*   **SHA256 Verification**: `0641196b87046cbaefcc4a627cb2d97a3eca6a41ec72f3447655cf45557071f0`
*   **Python Version Support**: 3.9, 3.10, 3.11, 3.12, 3.13

The workspace configuration automatically downloads and configures TensorFlow with its dependencies through a series of workspace initialization functions: `tf_workspace3()`, `tf_workspace2()`, `tf_workspace1()`, and `tf_workspace0()`.
