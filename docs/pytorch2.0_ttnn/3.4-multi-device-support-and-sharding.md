## Purpose and Scope

This document covers the multi-device support system that enables data-parallel execution across multiple Tenstorrent devices. The system consists of two primary passes: `MultiDeviceShardAnalysisPass` for propagating sharding metadata through the computation graph, and `MultiDevicePass` for inserting mesh operations that distribute tensors across devices.

For information about input classification that feeds into multi-device decisions, see [Input Analysis and Classification](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3.3-input-analysis-and-classification). For integration with data movement operations, see [Data Movement and Alignment (AddDataMovePass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.3-data-movement-and-alignment-(adddatamovepass)).

## Multi-Device Architecture Overview

The multi-device system enables data-parallel execution by distributing input tensors across a mesh of devices while replicating model parameters. The architecture distinguishes between three types of inputs based on their persistence and training semantics:

*   **Parameters** (weights, biases): Replicated across all devices for consistency
*   **Buffers** (batch norm statistics): Replicated across all devices
*   **Arguments** (input data): Sharded across the batch dimension for data parallelism

## Input Classification Strategy

The multi-device system relies on input classification performed by `InputAnalysisPass`, which annotates placeholder nodes with `PrimalTag` values:

| PrimalTag | Description | Multi-Device Strategy |
| --- | --- | --- |
| `PARAMETER` | Model weights/biases tracked by optimizer | **Replicated** across all devices |
| `BUFFER` | Fixed model state (batch norm statistics) | **Replicated** across all devices |
| `ARGUMENT` | Dynamic input data | **Sharded** across batch dimension |

This classification determines how tensors are distributed in the multi-device transformation.

## MultiDevicePass Transformation

The `MultiDevicePass` performs the actual graph transformation by inserting mesh operations. It operates on four types of nodes:

1.   **Placeholder nodes**: Insert replication or sharding
2.   **Output nodes**: Insert concatenation
3.   **Call function nodes**: Rewrite constant arguments for sharded operations

## Replication to Mesh

Parameters and buffers are replicated to all devices using `target_wrappers.replicate_tensor`. This ensures each device has a complete copy of the model weights.

### Replication Implementation

The operation is inserted before the first non-placeholder node to ensure proper execution order.

**Sources:**[torch_ttnn/passes/multi_device_pass.py 31-34](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L31-L34)

## Sharding to Mesh

Argument inputs (dynamic data) are sharded across devices along the batch dimension (dimension 0). Each device receives a portion of the batch.

### Sharding Implementation

The sharding operation takes:

*   `node`: Tensor to shard
*   `batch_dimension`: Dimension to shard along (0)
*   `num_devices`: Number of devices in the mesh

**Sources:**[torch_ttnn/passes/multi_device_pass.py 36-42](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L36-L42)

## Concatenation from Mesh

Output tensors that were sharded during computation must be concatenated back into a single tensor. The `concat_to_tensor` method handles this by wrapping sharded outputs with concatenation operations.

### Concatenation Implementation

Only tensors marked with `is_sharded=True` in their metadata are concatenated; non-sharded outputs pass through unchanged.

**Sources:**[torch_ttnn/passes/multi_device_pass.py 44-60](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L44-L60)

## Target Wrappers for Multi-Device Operations

The multi-device pass inserts calls to placeholder wrapper functions that will be replaced with actual TTNN mesh operations later in the compilation pipeline:

| Wrapper Function | Purpose | Arguments |
| --- | --- | --- |
| `target_wrappers.replicate_tensor` | Replicate tensor to all devices | `(tensor)` |
| `target_wrappers.shard_tensor` | Shard tensor across devices | `(tensor, dim, num_devices)` |
| `target_wrappers.concat_tensor` | Concatenate sharded tensors | `(tensor, dim, num_devices)` |

These wrappers serve as placeholders during shape propagation and are later lowered to actual TTNN mesh operations. For details on target wrappers, see [Target Wrappers and TTNN Operations](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.5-target-wrappers-and-ttnn-operations).

## Limitations and Requirements

### Current Constraints

1.   **Divisibility Requirement**: The batch dimension size must be divisible by the number of devices in the mesh. This constraint is enforced in `MultiDeviceShardAnalysisPass`:

1.   **Fixed Sharding Dimension**: Currently hardcoded to batch dimension (dimension 0). Other sharding strategies are not supported.

2.   **MeshDevice Requirement**: Multi-device execution requires a `MeshDevice` with at least 2 devices. Single-device execution skips all multi-device passes.

### Supported Operations

Operations that modify tensor shapes must have their constant arguments rewritten:

*   `torch.ops.aten.view.default`
*   `torch.ops.aten._unsafe_view.default`
*   `torch.ops.aten.expand.default`

Operations that modify the sharding dimension require special handling:

*   `torch.ops.aten.permute.default` updates the `shard_dim` metadata

**Sources:**[torch_ttnn/passes/analysis/multi_device_shard_analysis_pass.py 57-60](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/multi_device_shard_analysis_pass.py#L57-L60)[torch_ttnn/passes/multi_device_pass.py 106-112](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L106-L112)
