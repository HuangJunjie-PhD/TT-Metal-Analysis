Blackhole A0 introduces significant improvements to the Vector Unit (SFPU) and Fused Multiply-Add (FMA) operations compared to Wormhole B0. The primary changes affect instruction scheduling, FMA computation semantics, and instruction behavior.

## Overview of Improvements

The major SFPU and FMA improvements in Blackhole A0:

| Category | Improvement | Location |
| --- | --- | --- |
| **Instruction Scheduling** | Automatic hazard detection and stalling | Vector Unit wait gate |
| **SFPMAD Modifiers** | `SFPMAD_MOD1_NEGATE_VA` and `SFPMAD_MOD1_NEGATE_VC` flags | `SFPMAD` instruction encoding |
| **FMA Semantics** | Canonical NaN output (`0x7fc00000`), sign-preserved zeros | `fma_model_bh` function |
| **SFPSHFT2 Bug Fix** | Correct zero fill in `SFPSHFT2_MOD1_SUBVEC_SHFLSHR1` | `SFPSHFT2` instruction |
| **SFPLUTFP32** | New instruction for FP32/FP16 piecewise linear evaluation | Vector Unit MAD sub-unit |
| **Baby RISC-V FMA** | FMA instructions use `fma_model_bh` semantics | `fmadd.s`, `fmsub.s`, `fnmsub.s`, `fnmadd.s` |

## Automatic Instruction Scheduling

### Wormhole B0 Manual Scheduling

In Wormhole B0, software must manually insert `SFPNOP` instructions to avoid read-after-write hazards on `LReg` registers. The `SFPMAD` instruction requires 2 cycles to compute its result.

**Diagram: Wormhole B0 Manual Instruction Spacing**

Sources: [WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 61-63](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L61-L63)

### Blackhole A0 Automatic Stalling

Blackhole A0 implements automatic hazard detection in the Vector Unit wait gate. When an instruction reads from an `LReg` index that has a pending write from a previous instruction, the wait gate automatically stalls the thread for one cycle.

**Diagram: Blackhole A0 Automatic Stall Insertion**

The automatic stalling logic tracks:

*   `LReg` indices written by instructions with multi-cycle latency
*   `LReg` indices read by incoming instructions
*   Cycle count until results become available

**SFPMAD latency:** 2 cycles from execution to result availability

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L74)

### Exceptions and Limitations

**SFPLOADMACRO sequences:** Automatic stalling does not apply to instructions executed as part of an `SFPLOADMACRO` sequence. Software must manually ensure proper instruction spacing within macro sequences.

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 72-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L72-L74)

**Indirect addressing modes:** When `SFPMAD_MOD1_INDIRECT_VA` is used, the stalling logic conservatively assumes that `SFPMAD` reads from every `LReg`. When `SFPMAD_MOD1_INDIRECT_VD` is used, the stalling logic conservatively assumes that `SFPMAD` writes to every `LReg`. This can decrease performance.

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L71)

### Hardware Bugs in Automatic Stalling

The following instruction combinations are not correctly detected by the automatic stalling logic:

| Instruction | Bug Description | Required Workaround |
| --- | --- | --- |
| `SFPAND` with `SFPAND_MOD1_USE_VB` | Stalling logic ignores `SFPAND_MOD1_USE_VB`, assumes read from `VD` not `VB` | Manual `SFPNOP` insertion |
| `SFPOR` with `SFPOR_MOD1_USE_VB` | Stalling logic ignores `SFPOR_MOD1_USE_VB`, assumes read from `VD` not `VB` | Manual `SFPNOP` insertion |
| `SFPIADD` | Stalling logic does not detect read from `VD` | Manual `SFPNOP` insertion |
| `SFPSHFT` | Stalling logic does not detect read from `VD` | Manual `SFPNOP` insertion |
| `SFPCONFIG` | Stalling logic does not detect potential read from `LReg[0]` | Manual `SFPNOP` insertion |
| `SFPSWAP` (non-swap modes) | First cycle reads from `VC` and `VD` not detected | Manual `SFPNOP` insertion |
| `SFPSHFT2` with certain modes | Read operands incorrectly identified | Manual `SFPNOP` insertion |

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 75-86](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L75-L86)

## SFPMAD Instruction Enhancements

### New Modifier Flags

Blackhole A0 adds two modifier flags to the `SFPMAD` instruction encoding:

```
SFPMAD_MOD1_NEGATE_VA = 1  // XOR sign bit of operand a
SFPMAD_MOD1_NEGATE_VC = 2  // XOR sign bit of operand c
```

The existing flags remain:

```
SFPMAD_MOD1_INDIRECT_VA = 4  // Read VA index from LReg[7] & 15
SFPMAD_MOD1_INDIRECT_VD = 8  // Read VD index from LReg[7] & 15
```

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 49-54](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L49-L54)

### Functional Model Differences

**Diagram: SFPMAD Execution Flow Comparison**

**Wormhole B0 execution:**

```
1. va = (Mod1 & SFPMAD_MOD1_INDIRECT_VA) ? LReg[7].u32 & 15 : VA
2. a = LReg[va].u32
3. b = LReg[VB].u32
4. c = LReg[VC].u32
5. d = fma_model_wh(a, b, c)
6. vd = (Mod1 & SFPMAD_MOD1_INDIRECT_VD && VD != 16) ? LReg[7].u32 & 15 : VD
7. LReg[vd].u32 = d
```

**Blackhole A0 execution:**

```
1. va = (Mod1 & SFPMAD_MOD1_INDIRECT_VA) ? LReg[7].u32 & 15 : VA
2. a = LReg[va].u32
3. b = LReg[VB].u32
4. c = LReg[VC].u32
5. if (Mod1 & SFPMAD_MOD1_NEGATE_VA) a ^= 0x80000000
6. if (Mod1 & SFPMAD_MOD1_NEGATE_VC) c ^= 0x80000000
7. d = fma_model_bh(a, b, c)
8. vd = (Mod1 & SFPMAD_MOD1_INDIRECT_VD && VD != 16) ? LReg[7].u32 & 15 : VD
9. LReg[vd].u32 = d
```

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 22-44](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L22-L44)[WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 19-39](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L19-L39)

### Supported Operations

The modifier flags enable four FMA variants in a single instruction:

| Mod1 Bits | Computation | Use Case |
| --- | --- | --- |
| `0` | `VD = VA * VB + VC` | Standard FMA |
| `SFPMAD_MOD1_NEGATE_VA` | `VD = (-VA) * VB + VC` | Negated multiplicand |
| `SFPMAD_MOD1_NEGATE_VC` | `VD = VA * VB + (-VC)` | FMA subtraction |
| `SFPMAD_MOD1_NEGATE_VA | SFPMAD_MOD1_NEGATE_VC` | `VD = (-VA) * VB + (-VC)` | Double negation |

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 1-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L1-L13)[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 26-32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L26-L32)

### Semantic Differences

**Wormhole B0 (`fma_model_wh`):**

1.   **Zero handling:** All zero outputs become positive zero (sign discarded)
2.   **NaN output:** Non-canonical bit patterns (minimum `0x7f800001`, can have additional mantissa bits)
3.   **Denormal flush:** Denormals flushed before rounding, sign discarded
4.   **Sticky bit bug:** When right-shifting by 2+ bits, one bit lost instead of contributing to sticky

Sources: [Miscellaneous/FMA/README.md 23-31](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/README.md?plain=1#L23-L31)[Miscellaneous/FMA/fma.c 191-287](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L191-L287)[WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 49-59](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L49-L59)

**Blackhole A0 (`fma_model_bh`):**

1.   **Zero handling:** Sign-preserved zero output
2.   **NaN output:** Canonical bit pattern `0x7fc00000`
3.   **Denormal flush:** Denormals flushed after rounding, sign preserved
4.   **Sticky bit:** Correct sticky bit computation

Sources: [Miscellaneous/FMA/README.md 12-22](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/README.md?plain=1#L12-L22)[Miscellaneous/FMA/fma.c 102-189](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L102-L189)[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 56-65](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L56-L65)

### Shared Characteristics

Both `fma_model_wh` and `fma_model_bh` differ from `fma_model_ieee` in these ways:

1.   **Denormal inputs:** Treated as zero (not gradual underflow)
2.   **Partial fusion:** Product `p = x * y` stored as 64-bit, then shifted right by 23 bits with sticky, reducing to ~28-bit precision before addition
3.   **Underflow handling:** If `x * y` alone would underflow to zero, result is as if `x * y` was exactly zero
4.   **Overflow handling:** If `x * y` alone would overflow to infinity, result is infinity (even if properly fused result would be finite)

Sources: [Miscellaneous/FMA/README.md 14-22](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/README.md?plain=1#L14-L22)

### Baby RISC-V FMA Instructions

Blackhole A0 Baby RISC-V cores implement FMA instructions using `fma_model_bh` semantics:

| Instruction | Computation | FMA Model |
| --- | --- | --- |
| `fmadd.s` | `rd = (rs1 * rs2) + rs3` | `fma_model_bh` |
| `fmsub.s` | `rd = (rs1 * rs2) - rs3` | `fma_model_bh` |
| `fnmsub.s` | `rd = -(rs1 * rs2) + rs3` | `fma_model_bh` |
| `fnmadd.s` | `rd = -(rs1 * rs2) - rs3` | `fma_model_bh` |

All variants: denormal inputs treated as zero, denormal outputs flushed to zero, canonical NaN output, sign-preserved zeros.

### SFPSHFT2_MOD1_SUBVEC_SHFLSHR1 Mode

The `SFPSHFT2` instruction with mode `SFPSHFT2_MOD1_SUBVEC_SHFLSHR1` shifts lanes right by one position within each 8-lane subvector. In Wormhole B0, this mode contains a hardware bug affecting lanes 0, 8, 16, and 24.

### Performance Considerations

**Automatic Stalling Overhead:**

The automatic stalling mechanism introduces conservative assumptions for indirect addressing modes:

| Condition | Conservative Assumption | Performance Impact |
| --- | --- | --- |
| `SFPMAD_MOD1_INDIRECT_VA` used | Reads from every `LReg` | May stall unnecessarily |
| `SFPMAD_MOD1_INDIRECT_VD` used | Writes to every `LReg` | May stall unnecessarily |

For performance-critical code using indirect modes, consider manual scheduling with explicit `SFPNOP` placement.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L71)

**SFPLOADMACRO Sequences:**

When using `SFPLOADMACRO`, automatic stalling does not apply. Software must ensure proper instruction spacing within macro sequences, just as in Wormhole B0.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 72-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L72-L74)
