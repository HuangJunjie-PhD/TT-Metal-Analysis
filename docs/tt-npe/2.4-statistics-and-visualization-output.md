## Purpose and Scope

This page documents the output formats produced by tt-npe simulations, including statistical summaries, timeline visualization files, and batch analysis results. These outputs enable both programmatic analysis and visual exploration of NoC performance characteristics.

For information about running batch analysis that generates these outputs, see [Batch Trace Analysis](https://deepwiki.com/tenstorrent/tt-npe/2.3-batch-trace-analysis). For details on the simulation process that collects this data, see [Performance Estimation Algorithm](https://deepwiki.com/tenstorrent/tt-npe/4.1-performance-estimation-algorithm).

## npeStats Class Architecture

The `npeStats` class serves as the central container for all simulation results. It maintains statistics at multiple granularities: per-device, per-timestep, and per-transfer.

**Sources:**[tt_npe/cpp/src/npeStats.cpp 24-30](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L24-L30)[tt_npe/cpp/include/npeStats.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeStats.hpp)

### Statistics Collection Lifecycle

Statistics are populated throughout the simulation process via three key methods:

| Method | When Called | Purpose |
| --- | --- | --- |
| `insertTimestep()` | Start of each timestep | Create new timestep entry for data collection |
| `updateWorstCaseTransferEndCycle()` | When transfer completes | Track latest completion time per device |
| `finishSimulation()` | End of simulation | Compute estimated cycles, filter timesteps |
| `computeSummaryStats()` | After simulation | Aggregate per-timestep data into summary metrics |

The simulation engine calls these methods as shown in [tt_npe/cpp/src/npeEngine.cpp 232-354](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L232-L354):

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 172-361](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L172-L361)[tt_npe/cpp/src/npeStats.cpp 42-85](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L42-L85)

### Per-Device Statistics

The `per_device_stats` map contains entries for each physical device plus a special `MESH_DEVICE` entry (ID = -1) representing the entire multi-chip system. For single-chip workloads, only device 0 and `MESH_DEVICE` are populated.

Key fields in `deviceStats`:

*   **Cycle Counts**: `estimated_cycles`, `golden_cycles`, `estimated_cong_free_cycles`
*   **Prediction Accuracy**: `cycle_prediction_error` (percentage)
*   **Bandwidth Utilization**: `dram_bw_util`, `dram_bw_util_sim`, `eth_bw_util_per_core`
*   **Link Metrics**: `overall_avg_link_util`, `overall_max_link_demand`, `overall_avg_link_demand`
*   **NIU Metrics**: `overall_avg_niu_demand`, `overall_max_niu_demand`
*   **NoC-Specific**: `overall_avg_noc0_link_demand`, `overall_avg_noc1_link_demand`

**Sources:**[tt_npe/cpp/src/npeStats.cpp 122-201](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L122-L201)

## Timeline Output Formats

Timeline files enable visualization in ttnn-visualizer by providing detailed transfer routing and congestion data. tt-npe supports two schema versions and optional file splitting for large workloads.

### Schema Versions

#### V0 Schema (Legacy)

The V0 format uses a flat structure with per-transfer routes stored as simple arrays. Each transfer lists its complete route independently.

Key characteristics:

*   Single-file output only
*   Transfer routes are complete paths (no segmentation)
*   Simpler structure but larger file sizes for multi-chip workloads
*   No zone information
*   Limited topology metadata

**Sources:**[tt_npe/cpp/src/npeStats.cpp 203-331](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L203-L331)

#### V1 Schema (Current)

The V1 format introduces transfer groups, zone tracking, and multi-chip topology support. It is the default for all new outputs.

Key improvements over V0:

*   **Transfer Groups**: Multi-hop fabric transfers represented as single logical transfers with route segments
*   **Zone Tracking**: Nested zone hierarchies for temporal analysis (see [Zone Tracking and Transfer Groups](https://deepwiki.com/tenstorrent/tt-npe/3.5-zone-tracking-and-transfer-groups))
*   **Topology Information**: Device coordinates in fabric for multi-chip visualization
*   **Per-NoC Statistics**: Separate metrics for NOC0 and NOC1
*   **Split File Support**: Large workloads can be divided into multiple timeline files

The V1 schema version identifier is defined as [tt_npe/cpp/include/npeStats.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeStats.hpp#LNaN-LNaN)

**Sources:**[tt_npe/cpp/src/npeStats.cpp 359-745](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L359-L745)

### Transfer Group Serialization

Transfer groups enable multi-chip transfers to be visualized as single logical operations. The serialization process:

1.   **Group Construction**: Build mapping of `transfer_group_id` → component transfers

    *   Transfers without explicit groups assigned dummy group IDs
    *   Component transfers sorted by `transfer_group_index`

2.   **Logical Transfer Emission**: Each group becomes one timeline entry with:

    *   Start point from first component transfer
    *   End point from last component transfer
    *   Array of route segments (one per component)

3.   **Route Segment Details**: Each segment includes:

    *   Device ID, source, destination
    *   NoC type and injection rate
    *   Start/end cycles
    *   Complete link path

This transformation is implemented in [tt_npe/cpp/src/npeStats.cpp 456-604](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L456-L604)

**Sources:**[tt_npe/cpp/src/npeStats.cpp 450-604](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L450-L604)

### Split File Generation

When `timeline_split_threshold_timesteps` is set and the workload exceeds the threshold, tt-npe generates multiple timeline files:

**Region Filtering Rules:**

| Data Type | Inclusion Criterion |
| --- | --- |
| Transfers | `partiallyContainedInRegion()` - starts OR ends in region |
| Zones | `fullyContainedInRegion()` - both start AND end in region |
| Timesteps | `fullyContainedInRegion()` - entire timestep in region |

The filtering logic is defined in [tt_npe/cpp/src/npeStats.cpp 337-357](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L337-L357) with application in [tt_npe/cpp/src/npeStats.cpp 548-679](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L548-L679)

**Sources:**[tt_npe/cpp/src/npeStats.cpp 746-845](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L746-L845)

### Compression Support

Timeline files can be compressed using Zstandard (zstd) compression by setting `compress_timeline_output_file = true`. Compressed files receive a `.zst` extension and are decompressed transparently by ttnn-visualizer.

The compression implementation uses [tt_npe/cpp/src/npeCompressionUtil.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeCompressionUtil.cpp):

Compression typically achieves 5-10x size reduction for timeline files, which is significant given their verbosity.

**Sources:**[tt_npe/cpp/src/npeStats.cpp 853-867](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L853-L867)[tt_npe/cpp/src/npeCompressionUtil.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeCompressionUtil.cpp)

## Statistics Categories

### Summary Statistics

Summary statistics provide high-level performance metrics aggregated across the entire simulation. These are computed in `computeSummaryStats()` after all timesteps have been processed.

#### Cycle Metrics

*   **`estimated_cycles`**: Simulated execution time from first transfer start to last transfer completion
*   **`golden_cycles`**: Actual hardware execution time from trace data
*   **`estimated_cong_free_cycles`**: Simulated time with congestion modeling disabled (if `estimate_cong_impact` enabled)
*   **`cycle_prediction_error`**: `100.0 * (estimated - golden) / golden`
*   **`congestion_impact`**: `100.0 * (estimated - cong_free) / estimated`

Cycle metrics are finalized in [tt_npe/cpp/src/npeStats.cpp 62-84](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L62-L84):

**Sources:**[tt_npe/cpp/src/npeStats.cpp 55-84](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L55-L84)

#### Bandwidth Utilization

DRAM bandwidth utilization is computed by:

1.   Summing total bytes transferred to/from DRAM cores
2.   Dividing by theoretical maximum bandwidth over the active cycle window
3.   Computing both golden-based and simulation-based percentages

Ethernet bandwidth utilization is tracked **per ETH core** since different cores may have different utilization patterns:

**Sources:**[tt_npe/cpp/src/npeStats.cpp 159-201](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L159-L201)

#### Link and NIU Metrics

Link and NIU metrics are averaged across all timesteps:

| Metric | Description | Significance |
| --- | --- | --- |
| `overall_avg_link_util` | Average link utilization | Actual bandwidth used / available |
| `overall_avg_link_demand` | Average link demand | Requested bandwidth / available (can exceed 1.0) |
| `overall_max_link_demand` | Maximum link demand | Peak congestion indicator |
| `overall_avg_niu_demand` | Average NIU demand | Injection point congestion |
| `overall_max_niu_demand` | Maximum NIU demand | Peak injection congestion |

Separate metrics are maintained for NOC0 and NOC1 to identify NoC-specific bottlenecks.

**Sources:**[tt_npe/cpp/src/npeStats.cpp 122-154](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L122-L154)

### Demand Grid Interpretation

Demand grids are flattened 2D/3D arrays indexed by link or NIU identifiers. Demand values represent bandwidth pressure:

*   **< 1.0**: Link/NIU operating below capacity
*   **= 1.0**: Link/NIU at full capacity
*   **> 1.0**: Congestion - multiple transfers competing for same resource

In timeline JSON output, only links/NIUs with demand above `DEMAND_SIGNIFICANCE_THRESHOLD` (0.001) are emitted to reduce file size [tt_npe/cpp/src/npeStats.cpp 298-698](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L298-L698)

## Integration with ttnn-visualizer

Timeline files are consumed by the ttnn-visualizer web application, which provides interactive exploration of NoC traffic patterns.

### Visualization Features Enabled

| Timeline Data | Visualization Capability |
| --- | --- |
| `noc_transfers` with routes | Animated transfer playback showing data movement |
| `timestep_data` with demand grids | Heatmap overlays showing congestion hotspots |
| `zones` hierarchy | Nested timeline tracks for operation phases |
| `chips` topology | Multi-chip fabric layout visualization |
| Transfer groups | Logical operation grouping for fabric transfers |

### File Discovery Process

ttnn-visualizer discovers timeline files through two mechanisms:

1.   **Manifest-based** (batch analysis): Reads `manifest.json` to find all timeline files and their ordering
2.   **Single-file**: Directly opens individual `.npeviz` or `.npeviz.zst` files

The split file naming convention (`timeline_part_0.npeviz`, `timeline_part_1.npeviz`, etc.) enables automatic loading of multi-part timelines.

**Sources:**[tt_npe/cpp/src/npeStats.cpp 829-845](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L829-L845)[tt_npe/py/util/npe_analyze_noc_trace_dir.py 437-446](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L437-L446)

### Timeline JSON Schema Reference

For complete schema documentation, see [tt_npe/scripts/stats_schema.json](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/stats_schema.json) which documents the legacy V0 format. The V1 format adds:

*   `version` field in `common_info` (set to `CURRENT_TIMELINE_SCHEMA_VERSION`)
*   `arch` field: `"wormhole_b0"` or `"blackhole"`
*   `mesh_device` field: device name string
*   `chips` object: device ID → `[x, y, rack, shelf]` fabric coordinates
*   `zones` array: nested zone hierarchies per core/proc
*   `fabric_event_type` boolean on transfers
*   `route` as array of route segments (vs. single route in V0)
*   Per-NoC statistics in timestep data

**Sources:**[tt_npe/scripts/stats_schema.json 1-104](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/stats_schema.json#L1-L104)[tt_npe/cpp/src/npeStats.cpp 359-745](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L359-L745)

## Output File Locations

Default output locations vary by execution mode:

| Execution Mode | Output Directory | Files Generated |
| --- | --- | --- |
