# Reset Procedures

Relevant source files
*   [tt_flash/data/wormhole/fw_defines.yaml](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml)

## Purpose and Scope

This document describes the platform-specific reset mechanisms used to activate newly flashed firmware in tt-flash. After firmware has been successfully written to SPI flash and verified, the system must trigger a reset to load the new firmware into the ARC processors and M3 controller. The reset procedures vary significantly by platform and board type, ranging from IPMI-based system resets for Galaxy installations to direct chip resets for standalone boards.

For information about the overall flashing workflow, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages). For details on write and verification operations that precede reset, see [Write and Verify Cycle](https://deepwiki.com/tenstorrent/tt-flash/5.5-write-and-verify-cycle).

## Reset Trigger Conditions

Resets are conditionally triggered based on multiple factors evaluated after successful firmware flashing:

| Condition | Behavior | Code Reference |
| --- | --- | --- |
| `--no-reset` flag provided | Skip automatic reset entirely | [tt_flash/flash.py 902-910](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L902-L910) |
| Flash errors detected (`rc != 0`) | Skip automatic reset for safety | [tt_flash/flash.py 912-913](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L912-L913) |

| NEBULA_X2 local-to-remote copy active | Wait 15 seconds before reset | [tt_flash/flash.py 884-888](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L884-L888) |
| All chips are Galaxy boards | Use IPMI reset instead of standard | [tt_flash/flash.py 916-925](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L916-L925) |

## Galaxy IPMI Reset

Galaxy systems (multi-board installations using UBB interconnects) require IPMI-based reset commands to reset the entire tray of boards simultaneously.

### Galaxy Reset Procedure

**Sources:**[tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789)[tt_flash/flash.py 700-735](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L735)

### IPMI Parameters

The `glx_6u_trays_reset` function accepts the following parameters for fine-grained control:

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `ubb_num` | str | UBB bitmap (0x0-0xF) | `"0xF"` (all) |
| `dev_num` | str | Device bitmap (0x0-0xFF) | `"0xFF"` (all) |
| `op_mode` | str | 0x0=toggle, 0x1=assert, 0x2=deassert | `"0x0"` |
| `reset_time` | str | Reset duration (10ms resolution) | `"0xF"` (150ms) |

**Sources:**[tt_flash/flash.py 738-754](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L754)

### Ethernet Link Validation

After Galaxy reset, the system validates Ethernet connectivity between boards by checking 16 Ethernet ports per chip:

**Ethernet Port Locations (NOC 0):**

```
(9,0) (1,0) (8,0) (2,0) (7,0) (3,0) (6,0) (4,0)  # Ports 0-7
(9,6) (1,6) (8,6) (2,6) (7,6) (3,6) (6,6) (4,6)  # Ports 8-15
```

**Sources:**[tt_flash/flash.py 700-735](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L735)

## Standard Wormhole Chip Reset

For Wormhole chips not in Galaxy configurations, the system uses `WHChipReset.full_lds_reset()` from `tt_tools_common`.

### WH Reset Implementation

| Parameter | Value | Purpose |
| --- | --- | --- |
| `pci_interfaces` | List of interface IDs | Specific chips to reset |
| `reset_m3` | `True` | Reset M3 controller to load new firmware |

**When WH Reset is Triggered:**

*   Board type is `WH_UBB` but not all chips are Galaxy boards
*   Board type is other WH variants (E300 series, E75)
*   Board auto-reset capability detected (`can_reset=True`)

**Sources:**[tt_flash/flash.py 927-930](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L927-L930)[tt_flash/flash.py 844-868](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L844-L868)

## Standard Blackhole Chip Reset

Blackhole chips use `BHChipReset.full_lds_reset()` with adaptive M3 boot delay configuration.

### BH Reset Implementation

| Parameter | Value | Purpose |
| --- | --- | --- |
| `pci_interfaces` | List of interface IDs | Specific chips to reset |
| `reset_m3` | `True` | Reset M3 controller |
| `m3_delay` | 20 or 60 seconds | Adaptive boot timeout |

**When BH Reset is Triggered:**

*   Any Blackhole board type (GALAXY-1, NEBULA series, P-series)
*   `isinstance(chip, BhChip)` evaluates to `True`

**Sources:**[tt_flash/flash.py 932-936](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L932-L936)[tt_flash/flash.py 480-481](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L480-L481)

## Adaptive M3 Boot Delays

The M3 controller requires different boot times depending on whether the firmware update crosses a major version boundary.

### M3 Delay Decision Logic

**Implementation:**

| Condition | M3 Delay | Reason |
| --- | --- | --- |
| Running version unknown | 60 seconds | Unknown state, be conservative |
| Major version change (e.g., v18→v19) | 60 seconds | M3 performs additional initialization |
| Same major version | 20 seconds | Standard boot time |

**Sources:**[tt_flash/flash.py 893-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L893-L900)

## Chip Re-enumeration

After reset operations complete, the system re-enumerates PCI devices to obtain fresh chip handles:

**Purpose:**

*   PCI device handles become stale after hardware reset
*   Fresh enumeration ensures post-validation messages reach the correct devices
*   Provides updated chip state for subsequent operations

**Sources:**[tt_flash/flash.py 938-939](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L938-L939)
