## Purpose and Scope

This page explains how to use `ttl.Pipe` and `ttl.PipeNet` for multicast data transfer and synchronization between cores. Pipes enable one source core to efficiently broadcast data to multiple destination cores, leveraging the hardware's Network-on-Chip (NoC) capabilities. This is a critical abstraction for operations where shared data (like bias vectors or weight blocks) needs to be distributed across a core grid. [docs/development/PipeNets.md 18-26](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L18-L26)

* * *

## Overview

A **Pipe** is a communication primitive that describes a data transfer from a source core to one or more destination cores. It supports both unicast (one-to-one) and multicast (one-to-many) patterns. [include/ttlang/Dialect/TTL/IR/TTLOps.td 631-640](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L631-L640)

**Key Concepts**:

*   **Source core**: The single core coordinate (`srcX`, `srcY`) that sends data. [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 96-97](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L96-L97)
*   **Destination range**: A range of coordinates (`dstStartX`, `dstStartY` to `dstEndX`, `dstEndY`) that receive the data. [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 98-101](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L98-L101)
*   **PipeNet**: A logical network of pipes that provides conditional execution based on a core's role (source or destination). [docs/development/PipeNets.md 38-46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L38-L46)
*   **Pipe Role**: An enumeration (`Source`, `Destination`, `Active`) used by predicate ops to guard execution. [include/ttlang/Dialect/TTL/IR/TTL.h 70-74](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L70-L74)

**Sources**: [include/ttlang/Dialect/TTL/IR/TTLOps.td 631-640](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L631-L640)[include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 95-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L95-L103)[docs/development/PipeNets.md 18-46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L18-L46)

## Pipe and PipeNet Creation

### Creating a Pipe

Pipes are defined by their source and destination coordinates. In the `tt-lang` DSL, these are typically instantiated via the `Pipe` class which maps to `ttl.create_pipe` in MLIR. [python/ttl/pipe.py 12-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/pipe.py#L12-L30)

`# Example: Source at (0,0), broadcasting to a range of nodes# NodeRange format: (row_index, slice(start_col, end_col))pipe = ttl.Pipe((0, 0), (0, slice(1, 4))) # Targets nodes (0,1), (0,2), (0,3)`
### Creating a PipeNet

A `PipeNet` wraps one or more pipes. It acts as the single source of truth for communication patterns within an operation and is represented in MLIR as `ttl.pipenet_scope`. [include/ttlang/Dialect/TTL/IR/TTLOps.td 645-655](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L645-L655)

`pipe_net = ttl.PipeNet([pipe])`

## Core Participation and Roles

### Determining Participation

Not every core in the grid necessarily participates in a specific communication pattern. `net.is_active()` checks if the current core is either the source or part of the destination range. This lowers to `ttl.pipenet_is_role` with `PipeRole::Active`. [include/ttlang/Dialect/TTL/IR/TTLOps.td 677-685](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L677-L685)

### Conditional Execution (if_src / if_dst)

`PipeNet` provides `if_src` and `if_dst` methods. These methods take a callback function that is only executed on cores matching that specific role. Internally, these use `ttl.pipenet_is_role` predicates to guard the block. [include/ttlang/Dialect/TTL/IR/TTLOps.td 658-675](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L658-L675)

## Data Transfer with Pipes

Communication via pipes follows a "Push-Pull" model using `ttl.copy()`. [include/ttlang/Dialect/TTL/IR/TTLOps.td 122-133](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L122-L133)

### Source Operations (Push)

The source core copies data from a local `CircularBuffer` (`!ttl.cb`) into a `!ttl.pipe`. This lowers to a `ttl.copy` operation where the destination is a pipe. [include/ttlang/Dialect/TTL/IR/TTLOps.td 122-133](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L122-L133)

`def pipe_src(pipe_id: ttl.SrcPipeIdentity):    # Push from local CB to the pipe    tx = ttl.copy(cb_input, pipe_id)    ttl.wait(tx)`
### Destination Operations (Pull)

Destination cores copy data from the `!ttl.pipe` into a local `CircularBuffer`. This requires a `cb_reserve` destination to provide storage. [lib/Dialect/TTL/IR/TTLOps.cpp 189-193](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L189-L193)

`def pipe_dst(pipe_id: ttl.DstPipeIdentity):    # Pull from pipe to local reserved CB block    with ttl.reserve(cb_output) as out_block:        tx = ttl.copy(pipe_id, out_block)        ttl.wait(tx)`
**Sources**: [include/ttlang/Dialect/TTL/IR/TTLOps.td 122-133](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L122-L133)[lib/Dialect/TTL/IR/TTLOps.cpp 189-193](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L189-L193)[docs/development/PipeNets.md 61-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L61-L66)

## Implementation and Lowering

### PipeGraph and Slot Allocation

When multiple pipes target the same receiver DFB, the compiler must prevent data corruption. `PipeGraph::assignGatherSlotIndices` performs greedy coloring to assign each pipe a unique `gatherSlotIdx`. This index is used to calculate distinct SRAM offsets in the receiver's L1 buffer. [lib/Dialect/TTL/Transforms/PipeGraph.cpp 58-126](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeGraph.cpp#L58-L126)

### TTKernel Lowering

The `ConvertTTLToTTKernel` pass lowers high-level pipe operations to low-level hardware commands:

*   **Send**: Lowered to direct NoC writes to the receiver's SRAM addresses. [docs/development/PipeNets.md 67-71](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L67-L71)
*   **Synchronization**: Uses hardware semaphores to signal when data is ready (sender to receiver) and when buffers are free (receiver to sender). [lib/Dialect/TTL/Transforms/PipeLowering.cpp 189-199](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeLowering.cpp#L189-L199)

**Pipe Multicast Sequence**

### Verification

The compiler verifies user-written guards: each pipe-coupled operation must be reachable only from the nodes permitted by its role. [docs/development/PipeNets.md 48-52](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/PipeNets.md?plain=1#L48-L52)

*   `ttl.copy(cb, pipe)` is only valid in `Source` nodes.
*   `ttl.copy(pipe, cb)` is only valid in `Destination` nodes.
*   `PipeGraph::verifyReceiverDFBBlockCounts` ensures the receiver's DFB has enough blocks to accommodate all assigned slots. [lib/Dialect/TTL/Transforms/PipeGraph.cpp 128-143](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/PipeGraph.cpp#L128-L143)
