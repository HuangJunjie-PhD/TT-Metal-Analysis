# Documentation Structure

Relevant source files
*   [docs/source/conf.py](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py)
*   [docs/source/index.rst](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/index.rst)

## Purpose and Scope

This page describes the organizational structure of RiESCUE's documentation system, including the three-tier content hierarchy (tutorials, user guides, reference), directory layout, and navigation architecture. For information about building and serving the documentation locally, see [Building Documentation](https://deepwiki.com/tenstorrent/riescue/6.1-building-documentation). For details on Sphinx configuration settings and extensions, see [Sphinx Configuration](https://deepwiki.com/tenstorrent/riescue/6.2.1-sphinx-configuration). For guidance on writing and contributing documentation, see [Writing Documentation](https://deepwiki.com/tenstorrent/riescue/6.2.2-writing-documentation).

## Documentation Architecture

RiESCUE's documentation follows a three-tier organizational model designed to serve different user needs:

| Documentation Tier | Purpose | Target Audience | Content Type |
| --- | --- | --- | --- |
| **Tutorials** | Learning-oriented, hands-on lessons | New users | Step-by-step exercises with concrete examples |
| **User Guides** | Task-oriented, goal-focused workflows | Practitioners | How-to guides for specific tasks |
| **Reference** | Information-oriented, comprehensive details | All users | API documentation, syntax reference, configuration schemas |

This structure follows the [Diátaxis framework](https://diataxis.fr/) for technical documentation, separating content by user intent rather than tool boundaries.

**Sources:**[docs/source/index.rst 1-46](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/index.rst#L1-L46)

## Directory Structure

The documentation source tree is organized into three primary directories under `docs/source/`:

```
docs/source/
├── index.rst                    # Main documentation entry point
├── conf.py                      # Sphinx configuration
├── tutorials/
│   ├── index.rst               # Tutorials landing page
│   └── riescuec/               # Tool-specific tutorials
│       └── bringup_mode_tutorial.rst
├── user_guides/
│   ├── index.rst               # User guides landing page
│   └── [task-oriented guides]
├── reference/
│   ├── index.rst               # Reference landing page
│   └── python_api/             # Auto-generated API docs
│       ├── RiescueD.rst
│       ├── RiescueC.rst
│       └── CTK.rst
├── common/                      # Shared resources
│   ├── images/
│   │   ├── tt_logo.svg
│   │   └── favicon.png
│   └── _templates/              # Shared Sphinx templates
└── _static/
    └── tt_theme.css             # Custom CSS styling
```

### Public vs Internal Documentation

RiESCUE supports two documentation source trees:

*   **Public Documentation**: Located at `docs/source/`, this is the default documentation tree that is built and deployed to GitHub Pages
*   **Internal Documentation**: Located at `docs/tt_source/`, this tree is only built when the `BUILD_TT_DOCS` environment variable is set

The internal documentation tree follows the same structural conventions but may contain proprietary information not suitable for public release.

**Sources:**[docs/source/conf.py 121-127](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L121-L127) documentation system overview from diagrams

## Main Entry Point: index.rst

The `docs/source/index.rst` file serves as the documentation homepage and defines the top-level navigation structure through Sphinx's `toctree` directive:

`.. toctree::   :maxdepth: 2   :caption: Contents:   :hidden:    /tutorials/index   /user_guides/index   /reference/index`
This toctree configuration:

*   **maxdepth: 2** - Displays up to two levels of nested sections in the sidebar
*   **caption: Contents:** - Labels the navigation section as "Contents"
*   **hidden** - Hides the toctree from the main page content (navigation appears only in sidebar)
*   References three index files that serve as landing pages for each documentation tier

The index.rst also provides:

*   Project introduction and purpose statement
*   Overview of the three main tools (RiescueD, RiescueC, CTK)
*   Quick links to getting started resources
*   Cross-references to key tutorial and reference pages

**Sources:**[docs/source/index.rst 1-46](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/index.rst#L1-L46)

## Content Organization by Tier

### Tutorials Section

Location: `docs/source/tutorials/`

The tutorials section provides hands-on learning experiences for new users. Each tutorial:

*   Follows a narrative structure with clear learning objectives
*   Provides complete, executable examples
*   Shows expected outputs and verification steps
*   Builds on previous tutorials when appropriate

Example structure:

```
tutorials/
├── index.rst                           # Tutorial landing page
├── riescuec/
│   └── bringup_mode_tutorial.rst      # RiescueC bringup mode walkthrough
└── [additional tutorials by tool/topic]
```

The tutorials index provides a curated learning path, directing users to start with foundational concepts before moving to advanced topics.

### User Guides Section

Location: `docs/source/user_guides/`

User guides focus on accomplishing specific tasks. Each guide:

*   Starts with a clear goal or use case
*   Provides step-by-step instructions
*   Includes troubleshooting and best practices
*   Links to relevant reference documentation

Guides are organized by workflow rather than by tool, reflecting real-world usage patterns.

### Reference Section

Location: `docs/source/reference/`

The reference section provides comprehensive technical documentation including:

```
reference/
├── index.rst                    # Reference landing page
├── python_api/                  # Python API documentation
│   ├── RiescueD.rst            # RiescueD module reference
│   ├── RiescueC.rst            # RiescueC module reference
│   └── CTK.rst                 # CTK module reference
├── test_syntax/                 # Test directive syntax
├── configuration/               # Configuration file schemas
└── cli/                         # Command-line interface reference
```

The Python API documentation uses Sphinx's `autodoc` extension to generate reference material directly from code docstrings, ensuring documentation stays synchronized with implementation.

**Sources:**[docs/source/index.rst 7-46](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/index.rst#L7-L46)

## Navigation Architecture

RiESCUE's documentation navigation is implemented through a hierarchy of toctrees:

Each index file acts as a landing page for its section, providing:

*   Section overview and purpose
*   Organized list of documents in that section
*   toctree directive to define navigation structure
*   Links to recommended starting points

The theme configuration in `conf.py` controls navigation behavior:

| Configuration | Value | Effect |
| --- | --- | --- |
| `collapse_navigation` | False | Keeps all navigation sections expanded |
| `sticky_navigation` | True | Navigation sidebar scrolls with page |
| `navigation_depth` | 4 | Shows up to 4 levels in sidebar |
| `titles_only` | False | Shows section headings, not just page titles |

**Sources:**[docs/source/conf.py 110-119](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L110-L119)[docs/source/index.rst 7-14](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/index.rst#L7-L14)

## Integration with Code Documentation

The documentation structure bridges natural language documentation with code through several mechanisms:

### Custom Directive: ArgparseDirective

The `ArgparseDirective` class enables automatic documentation generation from `argparse`-based command-line interfaces:

**Implementation:**[docs/source/conf.py 21-56](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L21-L56)

This directive:

1.   Imports a class that defines CLI arguments
2.   Creates an `ArgumentParser` instance
3.   Calls the class's `add_arguments()` method
4.   Extracts option strings and help text
5.   Generates formatted documentation as definition lists

Usage in RST files:

`.. argparse:: riescue.dtest_framework.riescued.RiescueD`
This automatically generates documentation for all command-line options defined in the `RiescueD.add_arguments()` method.

### Autodoc Configuration

The `autodoc` extension configuration controls API documentation generation:

**Configuration:**[docs/source/conf.py 93-98](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L93-L98)

Key settings:

*   `autodoc_default_options`: Disables inheritance display to keep docs focused
*   `autodoc_member_order`: Orders members by source code order (bysource) rather than alphabetically
*   Custom `MockedClassDocumenter`: Removes "Bases: object" lines to reduce clutter [docs/source/conf.py 59-66](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L59-L66)

**Sources:**[docs/source/conf.py 21-98](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L21-L98)

## Excluded Content

Certain directories and patterns are explicitly excluded from documentation builds:

**Exclusions:**[docs/source/conf.py 96](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L96-L96)

`exclude_patterns = ["public", "_build", "**/_build_api/**", "**/_templates"]`
*   `public/`: Previously generated static site output (no longer used)
*   `_build/`: Sphinx build directory
*   `**/_build_api/**`: Temporary API documentation generation directory
*   `**/_templates/`: Sphinx template files (not content)

This prevents Sphinx from attempting to process build artifacts or template files as documentation source.

**Sources:**[docs/source/conf.py 96](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L96-L96)

## Version Management

Documentation version information is dynamically extracted from `pyproject.toml`:

**Version Extraction:**[docs/source/conf.py 72-83](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L72-L83)

The configuration:

1.   Locates `pyproject.toml` relative to `conf.py`
2.   Parses the file to find the `version` field
3.   Sets both `version` and `release` variables for Sphinx
4.   Raises an error if version cannot be determined

This ensures documentation version always matches the package version, preventing version drift between code and documentation.

**Sources:**[docs/source/conf.py 72-83](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L72-L83)

## Template System

The documentation uses Sphinx templates for consistent formatting:

**Template Path Configuration:**[docs/source/conf.py 97](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L97-L97)

`templates_path = ["_templates", "../common/_templates"]`
Templates are searched in two locations:

1.   `docs/source/_templates/`: Source-tree-specific templates
2.   `docs/common/_templates/`: Shared templates for both public and internal docs

This allows internal documentation to reuse public templates while maintaining the ability to override them when necessary.

**Sources:**[docs/source/conf.py 97](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L97-L97)
