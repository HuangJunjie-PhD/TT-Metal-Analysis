This document describes the automated workflows that manage community contributions and issue tracking in the tt-flash repository. These GitHub Actions workflows automatically label incoming issues and pull requests from community contributors, and maintain project board status synchronization based on priority labels.

For information about build and release automation, see [Build Workflows](https://deepwiki.com/tenstorrent/tt-flash/10.1-build-workflows) and [Release Process](https://deepwiki.com/tenstorrent/tt-flash/9.1-release-process).

* * *

## Overview

The tt-flash repository uses two primary automation workflows to streamline community management:

1.   **Automatic Community Labeling** - Tags issues and pull requests from external contributors
2.   **Project Board Status Updates** - Synchronizes GitHub Project board status fields with priority labels

These automations reduce manual triage overhead and ensure consistent categorization of community contributions.

**Sources:**[.github/workflows/community-issue-tagging.yml](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml)[.github/workflows/bh-update-status-by-label.yml](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml)

## Community Issue Tagging Workflow

### Workflow Overview

The `community-issue-tagging.yml` workflow automatically labels issues and pull requests opened by community contributors (non-organization members). This provides immediate visibility to maintainers about which items require community engagement and may need additional context or guidance.

**Diagram: Community Issue Tagging Flow**

**Sources:**[.github/workflows/community-issue-tagging.yml 1-19](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L1-L19)

### Trigger Configuration

The workflow activates on two event types:

| Event Type | Activity Types | Description |
| --- | --- | --- |
| `issues` | `opened` | Triggered when a new issue is created |
| `pull_request` | `opened` | Triggered when a new pull request is created |

**Sources:**[.github/workflows/community-issue-tagging.yml 3-8](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L3-L8)

### Permissions

The workflow requires specific GitHub API permissions to function:

*   `contents: read` - Access repository metadata
*   `issues: write` - Add labels to issues
*   `pull-requests: write` - Add labels to pull requests

**Sources:**[.github/workflows/community-issue-tagging.yml 9-12](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L9-L12)

### Job Structure

The workflow uses a single job `community-labeling` that delegates to a reusable workflow maintained in the `tenstorrent/tt-github-actions` repository. This centralized approach ensures consistent community labeling behavior across all Tenstorrent repositories.

**Diagram: Job Delegation to Reusable Workflow**

The reusable workflow is invoked with the `AUTOLABEL_COMMUNITY_ISSUES` secret, which likely contains authentication credentials or configuration data required for the labeling operation.

**Sources:**[.github/workflows/community-issue-tagging.yml 14-18](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L14-L18)

## Project Board Status Automation

### Workflow Overview

The `bh-update-status-by-label.yml` workflow automatically synchronizes GitHub Project board status fields based on priority labels (`P0`, `P1`, `P2`, `P3`). This workflow runs on a daily schedule and can also be manually triggered to ensure project boards remain up-to-date with issue priorities.

**Diagram: Project Board Status Update Flow**

**Sources:**[.github/workflows/bh-update-status-by-label.yml 1-83](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L1-L83)

### Schedule Configuration

The workflow executes automatically on a daily schedule:

| Schedule Type | Expression | Execution Time |
| --- | --- | --- |
| Cron schedule | `'0 0 * * *'` | Daily at midnight UTC |
| Manual trigger | `workflow_dispatch` | On-demand execution |

**Sources:**[.github/workflows/bh-update-status-by-label.yml 2-5](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L2-L5)

### Status Update Logic

The inline JavaScript implementation follows this algorithm:

**Diagram: Priority Label to Status Field Mapping Algorithm**

The script implements priority detection with a hierarchical check (P0 → P1 → P2 → P3), meaning if multiple priority labels exist on an issue, the highest priority (P0) takes precedence.

**Sources:**[.github/workflows/bh-update-status-by-label.yml 44-82](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L44-L82)

### API Interactions

The workflow interacts with the GitHub Projects API (Classic Projects) through two endpoints:

#### Fetch Project Cards

Retrieves all issue cards associated with the project board identified by `BH_SW_BOARD_ID`.

**Sources:**[.github/workflows/bh-update-status-by-label.yml 46-49](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L46-L49)

#### Update Card Status

Updates the card's note field to reflect the current priority status. This approach uses the card note as a status indicator, a common pattern when working with GitHub Projects (Classic).

**Sources:**[.github/workflows/bh-update-status-by-label.yml 68-73](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L68-L73)

## Configuration Requirements

### Secrets Management

Both workflows require GitHub Secrets to be configured in the repository settings:

| Secret Name | Used By | Purpose |
| --- | --- | --- |
| `AUTOLABEL_COMMUNITY_ISSUES` | `community-issue-tagging.yml` | Authentication for community labeling workflow |
| `BH_SW_BOARD_ID` | `bh-update-status-by-label.yml` | Project board identifier for status updates |
| `GITHUB_TOKEN` | `bh-update-status-by-label.yml` | GitHub API authentication (automatically provided) |

**Note:** The `GITHUB_TOKEN` is automatically provided by GitHub Actions and does not require manual configuration.

**Sources:**[.github/workflows/community-issue-tagging.yml 18](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L18-L18)[.github/workflows/bh-update-status-by-label.yml 30-31](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml#L30-L31)

### Permissions Configuration

The repository must have the following permissions enabled for Actions:

*   **Issues:** Write access for adding labels
*   **Pull Requests:** Write access for adding labels
*   **Projects:** Write access for updating project board cards

These are configured through the `permissions` key in workflow files or repository settings.

**Sources:**[.github/workflows/community-issue-tagging.yml 9-12](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/community-issue-tagging.yml#L9-L12)

## Error Handling

### Community Tagging Error Handling

Error handling is delegated to the reusable workflow `issues-community-autotag.yml` maintained in the `tenstorrent/tt-github-actions` repository. The tt-flash workflow does not implement explicit error handling, relying on GitHub Actions' default failure behavior.
