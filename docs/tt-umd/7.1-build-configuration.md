## Purpose and Scope

This document describes the CMake-based build system for `tt-umd`, covering build options, third-party dependency management via CPM, compilation configurations, and packaging outputs. The build system supports building C++ libraries, Python bindings, tests, tools, examples, and simulation components for multiple platforms.

For information about the CI/CD pipeline and automated builds, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/tt-umd/7.2-cicd-pipeline). For testing infrastructure details, see [Testing Infrastructure](https://deepwiki.com/tenstorrent/tt-umd/7.3-testing-infrastructure).

* * *

## CMake Build System Structure

The `tt-umd` project uses CMake 3.25+ as its build system with a modular structure organized into multiple `CMakeLists.txt` files. The root `CMakeLists.txt` orchestrates the overall build configuration while delegating to subdirectories for specific components.

### Root CMakeLists.txt Organization

The main build configuration file [CMakeLists.txt 1-180](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L1-L180) follows this structure:

1.   **Project Declaration**[CMakeLists.txt 23-31](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L23-L31) - Reads version from `VERSION` file and declares the project `umd`.
2.   **Build Type Selection**[CMakeLists.txt 43-53](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L43-L53) - Sets default build type to `Release` when building as the master project.
3.   **Build Options**[CMakeLists.txt 55-63](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L55-L63) - Defines CMake options for enabling/disabling components (tests, simulation, python, tools).
4.   **Logging Configuration**[CMakeLists.txt 101-112](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L101-L112) - Controls compile-time logging via `SPDLOG_ACTIVE_LEVEL`.
5.   **Third-Party Dependencies**[CMakeLists.txt 113](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L113-L113) - Includes `third_party/` subdirectory.
6.   **Clang-Tidy Configuration**[CMakeLists.txt 120-141](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L120-L141) - Sets up static analysis.
7.   **Compiler Warnings**[CMakeLists.txt 144-163](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L144-L163) - Configures warning flags and architecture optimizations.
8.   **Component Subdirectories**[CMakeLists.txt 168-177](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L168-L177) - Adds subdirectories for `device`, `nanobind`, `src`, and `tools`.

**Sources:**[CMakeLists.txt 1-180](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L1-L180)

### Version Management

The project version is read from the `VERSION` file at configuration time. CMake `project()` versioning only accepts numeric formats, so any prerelease suffixes (e.g., `-rc1`) are stripped for the CMake project version but preserved for packaging [CMakeLists.txt 12-31](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L12-L31)

**Sources:**[CMakeLists.txt 12-31](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L12-L31)[VERSION 1](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/VERSION#L1-L1)

## Build Options

The build system provides multiple CMake options to control component selection. These options follow the `TT_UMD_BUILD_*` naming pattern.

### Available Build Options

| Option | Default | Description |
| --- | --- | --- |
| `TT_UMD_BUILD_TESTS` | OFF | Enables build of `tt_umd` tests [CMakeLists.txt 55](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L55-L55) |
| `TT_UMD_BUILD_SIMULATION` | OFF | Enables simulation harnessing (FlatBuffers, NNG, libuv) [CMakeLists.txt 56](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L56-L56) |
| `TT_UMD_BUILD_PYTHON` | OFF | Enables build of Python bindings via nanobind [CMakeLists.txt 57](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L57-L57) |
| `TT_UMD_BUILD_TOOLS` | ON | Enables build of `tt_umd` tools (topology, telemetry, etc.) [CMakeLists.txt 58](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L58-L58) |
| `TT_UMD_BUILD_EXAMPLES` | OFF | Enables build of `tt_umd` examples [CMakeLists.txt 59](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L59-L59) |
| `TT_UMD_BUILD_ALL` | OFF | Enables all components (tests, simulation, python, tools, examples) [CMakeLists.txt 60](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L60-L60) |
| `TT_UMD_BUILD_STATIC` | OFF | Builds `tt-umd` as a static library [CMakeLists.txt 61](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L61-L61) |
| `TT_UMD_BUILD_EMULE` | OFF | Enables software emulation chip (`SWEmuleChip`) [CMakeLists.txt 63](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L63-L63) |
| `TT_UMD_BUILD_JTAG` | OFF | Enables JTAG support [CMakeLists.txt 3](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L3-L3) |
| `TT_UMD_ENABLE_LOGGING` | Auto | Controls `SPDLOG_ACTIVE_LEVEL` (ON for Debug) [CMakeLists.txt 108](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L108-L108) |

**Sources:**[CMakeLists.txt 3-63](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L3-L63)[CMakeLists.txt 91-98](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L91-L98)[CMakeLists.txt 101-112](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L101-L112)

## Device Library Build Configuration

The core library `tt-umd` is defined in [device/CMakeLists.txt 1-138](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L1-L138)

### Library Type and Properties

The library can be built as `SHARED` or `STATIC` based on `TT_UMD_BUILD_STATIC`[device/CMakeLists.txt 31-38](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L31-L38) It uses `POSITION_INDEPENDENT_CODE ON`[device/CMakeLists.txt 3](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L3-L3)

### Source Selection

The library includes architecture-specific implementations for Wormhole and Blackhole [device/CMakeLists.txt 85-86](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L85-L86) coordinate management [device/CMakeLists.txt 110-112](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L110-L112) and communication protocols (PCIe, JTAG, Remote) [device/CMakeLists.txt 100-102](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L100-L102)

If `TT_UMD_BUILD_SIMULATION` is enabled:

*   It generates FlatBuffers headers from `simulation_device.fbs`[device/CMakeLists.txt 6-24](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L6-L24)
*   It adds simulation-specific sources like `simulation_chip.cpp` and `rtl_sim_communicator.cpp`[device/CMakeLists.txt 140-157](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L140-L157)

**Sources:**[device/CMakeLists.txt 1-159](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/CMakeLists.txt#L1-L159)

## Python Build Configuration

Python bindings are built using `nanobind` and `scikit-build-core`.

### Nanobind Module

The module `nanobind_tt_umd` (exported as `tt_umd`) binds key C++ APIs including `Cluster`, `TTDevice`, and `TopologyDiscovery`[nanobind/CMakeLists.txt 28](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L28-L28)[nanobind/CMakeLists.txt 67-72](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L67-L72)

### Type Stubs and Initialization

The build system automatically generates Python type stubs (`.pyi`) for type checking [nanobind/CMakeLists.txt 34-41](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L34-L41) and a dynamic `__init__.py` based on detected submodules [nanobind/CMakeLists.txt 44-54](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L44-L54)

### Pip Wheel vs. FHS

*   **Pip Wheel (`TT_UMD_BUILD_PIP=ON`)**: Configures `RPATH` to `$ORIGIN` and co-locates `libtt-umd.so` within the Python package [nanobind/CMakeLists.txt 145-184](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L145-L184)
*   **FHS Package**: Installs to system `site-packages` (e.g., `/usr/lib/python3/dist-packages`) and links against system-installed `libtt-umd.so`[nanobind/CMakeLists.txt 186-231](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L186-L231)

**Sources:**[nanobind/CMakeLists.txt 1-231](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/nanobind/CMakeLists.txt#L1-L231)[pyproject.toml 1-78](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/pyproject.toml#L1-L78)

## Installation and Packaging

The project uses `GNUInstallDirs` for standard path conventions [CMakeLists.txt 37](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/CMakeLists.txt#L37-L37)

### Components and CPack

Packaging is handled by `cmake/packaging.cmake` which defines three primary components for DEB/RPM generation:

1.   **umd-runtime**: Contains the shared library `libtt-umd.so`[cmake/packaging.cmake 84-88](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/cmake/packaging.cmake#L84-L88)

3.   **umd-python**: Contains the Python FHS bindings [cmake/packaging.cmake 99-106](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/cmake/packaging.cmake#L99-L106)
