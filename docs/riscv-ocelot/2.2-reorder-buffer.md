The Reorder Buffer (ROB) is a critical component in BOOM's out-of-order execution pipeline that tracks all inflight instructions. Its primary purpose is to provide the illusion of sequential program execution while allowing instructions to execute out-of-order for performance. The ROB ensures architectural state is updated in program order, handles exceptions, and facilitates precise recovery during branch mispredictions. For information about the interface between the Load/Store Unit and the ROB, see [Load/Store Unit](https://deepwiki.com/tenstorrent/riscv-ocelot/3.1-loadstore-unit).

## Organization and Structure

The ROB is implemented as a circular buffer that maintains program order of dispatched instructions. To support superscalar dispatch and commit, the ROB is organized into rows with multiple banks.

In this organization:

*   Each ROB row can hold up to `coreWidth` instructions (one per bank)
*   Dispatch fills a row from left to right before moving to the next row
*   Instructions within a fetch packet are placed across a single row
*   Since instructions in a fetch packet are consecutive in memory, we can store a single PC for the entire row

Sources: [src/main/scala/exu/rob.scala 130-248](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L130-L248)[docs/sections/reorder-buffer.rst 21-50](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/reorder-buffer.rst#L21-L50)

### Key Indices

The ROB maintains several key indices:

*   `rob_head`: Points to the oldest instruction, ready for commit
*   `rob_tail`: Points to the next row where instructions will be dispatched
*   `rob_pnr`: "Point of No Return" - indicates instructions that are ready to commit

Sources: [src/main/scala/exu/rob.scala 237-247](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L237-L247)

### ROB Entry State

Each ROB entry contains the following state:

| Field | Description |
| --- | --- |
| `rob_val` | Is the entry valid? |
| `rob_bsy` | Is the instruction still executing? |
| `rob_unsafe` | Is this instruction potentially unsafe (may cause exceptions)? |
| `rob_uop` | The microop stored in this entry |
| `rob_exception` | Has the instruction raised an exception? |
| `rob_predicated` | Was this instruction predicated out? |
| `rob_fflags` | Floating-point status flags for this instruction |

## Operation

### Dispatch Stage

During dispatch:

1.   Up to `coreWidth` instructions are written from the Fetch Packet into an ROB row
2.   Each instruction is assigned an ROB entry with initial state: 
    *   `rob_val` = true
    *   `rob_bsy` = true (unless it's a fence instruction)
    *   `rob_unsafe` = instruction's `unsafe` bit
    *   `rob_exception` = instruction's `exception` bit

Sources: [src/main/scala/exu/rob.scala 334-353](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L334-L353)

### Writeback and Wakeup

When an instruction completes execution:

1.   Execution units send results to the ROB via `wb_resps`
2.   The ROB clears the `rob_bsy` bit for the corresponding entry
3.   For store instructions, the LSU uses `lsu_clr_bsy` to mark stores as not busy
4.   Memory ordering hazards are cleared using `lsu_clr_unsafe`

Sources: [src/main/scala/exu/rob.scala 355-389](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L355-L389)

### Commit Stage

The commit logic:

1.   Examines entries at the head of the ROB
2.   Only commits instructions that are valid and not busy
3.   Commits instructions in program order
4.   Stops commit if any instruction can throw an exception
5.   Updates architectural state only when instructions commit

On exception, the pipeline is flushed and control transfers to the exception handler.

Sources: [src/main/scala/exu/rob.scala 552-604](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L552-L604)

### Branch Misprediction Handling

When a branch is mispredicted:

1.   The `brupdate` signal carries misprediction information
2.   ROB entries with matching branch masks are invalidated
3.   The ROB coordinates with the frontend to redirect fetch to the correct path

For every instruction in the ROB:

*   Its `br_mask` is compared against the `mispredict_mask`
*   If there's a match, the instruction is invalidated
*   Otherwise, its `br_mask` is updated with the new branch resolution information

Sources: [src/main/scala/exu/rob.scala 465-486](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L465-L486)

## Exception Handling

The ROB maintains information about the oldest excepting instruction:

*   `r_xcpt_val`: Indicates if there's a pending exception
*   `r_xcpt_uop`: Stores the excepting microop
*   `r_xcpt_badvaddr`: Holds the bad virtual address for memory exceptions

When an excepting instruction reaches the ROB head:

*   The pipeline is flushed
*   The `com_xcpt` signal is sent to the CSR file
*   The `flush` signal redirects the frontend to the exception handler

Sources: [src/main/scala/exu/rob.scala 644-687](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L644-L687)

## Interface and Integration

The ROB connects to various pipeline components through the `RobIo` interface:

Key interface signals:

*   **Inputs**:

    *   `enq_valids`/`enq_uops`: Instructions to be added to the ROB
    *   `wb_resps`: Execution results from functional units
    *   `brupdate`: Branch resolution information
    *   `lsu_clr_bsy`: Clear busy bits for stores that have been dispatched to memory
    *   `fflags`: Floating-point status flag updates

*   **Outputs**:

    *   `commit`: Signals for committed instructions
    *   `com_xcpt`: Exception information for the CSR file
    *   `flush`: Flush signal sent to the pipeline on exceptions/mispredictions
    *   `empty`/`ready`: Flow control signals for the decode stage

Sources: [src/main/scala/exu/rob.scala 43-116](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L43-L116)[src/main/scala/exu/core.scala 741-751](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/core.scala#L741-L751)

## Implementation Details

### Handling Speculative Execution

The ROB is central to BOOM's speculative execution:

1.   Instructions execute speculatively under branch predictions
2.   Each instruction has a `br_mask` indicating which branches it depends on
3.   If a branch mispredicts, all instructions with the corresponding bit in their `br_mask` are killed
4.   The `rob_unsafe` bit tracks instructions that could cause imprecise exceptions

Sources: [src/main/scala/exu/rob.scala 504-508](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L504-L508)

### Relationship with Register Renaming

The ROB works closely with register renaming:

1.   When instructions commit, their destination physical registers become part of the architectural state
2.   On exceptions or mispredictions, the ROB coordinates with the rename stage to restore the correct register mapping
3.   The `commit.rbk_valids` and `commit.rollback` signals facilitate this coordination

Sources: [src/main/scala/exu/rob.scala 442-446](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L442-L446)

### Performance Considerations

The ROB design includes several performance optimizations:

*   Bank-based organization reduces dispatch and commit complexity
*   Per-row PC storage reduces implementation area
*   Exception state is only fully tracked for the oldest excepting instruction
*   The point-of-no-return (PNR) index allows memory operations to execute as early as possible

Sources: [src/main/scala/exu/rob.scala 270-286](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rob.scala#L270-L286)

## Summary

The ROB is a crucial component in BOOM's out-of-order execution engine. It maintains program order, handles exceptions, and enables precise state recovery during speculation. By tracking the state of all inflight instructions, the ROB creates the illusion of sequential execution while allowing high-performance out-of-order execution.
