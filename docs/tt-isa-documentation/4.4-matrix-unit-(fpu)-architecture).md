## Purpose and Scope

This document describes the Matrix Unit (FPU), one of the three computation units within the Tensix Coprocessor backend. The Matrix Unit performs high-throughput, low-precision (â‰¤19-bit) matrix operations including matrix multiplication, element-wise operations, and pooling. It achieves peak performance of 4.096 TFLOP/s per tile by operating on data in the internal register files [`SrcA`](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst), [`SrcB`](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst), and [`Dst`](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst).

For detailed instruction-level documentation, see [Matrix Unit Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.2-matrix-unit-instructions). For information on the higher-precision Vector Unit alternative, see [Vector Unit (SFPU) Architecture](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.5-vector-unit-(sfpu)-architecture). For the complete frontend pipeline that feeds instructions to the Matrix Unit, see [Instruction Pipeline and Frontend](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend).

## Architectural Position

The Matrix Unit (FPU) is one of nine execution units in the Tensix Coprocessor backend. It receives instructions from the frontend (after MOP expansion, replay expansion, and wait gate synchronization) and executes matrix operations with single-instruction throughput and 1-5 cycle latencies.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 1-6](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L1-L6)

## Register Files and Data Path

The Matrix Unit operates on three internal register files: `SrcA`, `SrcB`, and `Dst`. Each register file is structured as a 2D array of rows and columns, with each datum being â‰¤19 bits in `SrcA`/`SrcB` and either 16 or 32 bits in `Dst`.

### Register File Structure

| Register File | Banks | Precision | Primary Role | Access Pattern |
| --- | --- | --- | --- | --- |
| `SrcA` | 2 | â‰¤19-bit | Matrix multiplication left operand | Row-based addressing via `RWCs[CurrentThread].SrcA` |
| `SrcB` | 2 | â‰¤19-bit | Matrix multiplication right operand | Row-based addressing via `RWCs[CurrentThread].SrcB` |
| `Dst` | 1 | 16 or 32-bit | Accumulator and result storage | Row-based addressing via `RWCs[CurrentThread].Dst` |

Each register file has two banks to enable double-buffering: while the Matrix Unit reads from one bank, unpackers can write to the other bank. The active bank is tracked by `MatrixUnit.SrcABank`, `MatrixUnit.SrcBBank`, and `MatrixUnit.DstBank` state variables.

For detailed information on register file structure, address remapping, and data views, see [Register Files: SrcA, SrcB, Dst](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst).

## RWC Addressing Mechanism

Matrix Unit instructions use **RWCs (Read-Write Counters)** as auto-incrementing addressing counters. Each Tensix thread (T0, T1, T2) maintains its own set of RWCs, which specify the current row addresses for `Dst`, `SrcA`, and `SrcB`, as well as the active fidelity phase.

### RWC Structure

`struct {  uint10_t Dst, Dst_Cr;       // Destination row address and checkpoint  uint6_t SrcA, SrcA_Cr;      // SrcA row address and checkpoint  uint6_t SrcB, SrcB_Cr;      // SrcB row address and checkpoint  uint2_t FidelityPhase;      // Current fidelity phase (0-3)  uint1_t ExtraAddrModBit;    // Additional addressing mode selection} RWCs[3];  // Indexed by CurrentThread (T0=0, T1=1, T2=2)`
### Address Modification via AddrMod Bits

Most Matrix Unit instructions include a 2-bit `AddrMod` field that specifies how to update the RWCs after instruction execution. The `AddrMod` bits index into pre-configured increment tables stored in `ThreadConfig[CurrentThread]`:

The `ApplyAddrMod()` function implements this logic by reading configuration from the indexed entries and applying increments, clears, or checkpoint operations to each RWC field. This indirection allows complex addressing patterns (e.g., nested loops over tiles) without requiring explicit `SETRWC`/`INCRWC` instructions on every iteration.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/RWCs.md 1-57](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/RWCs.md?plain=1#L1-L57)[WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 61-62](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L61-L62)

### RWC Manipulation Instructions

| Instruction | Purpose | Latency |
| --- | --- | --- |
| `SETRWC` | Directly set RWC fields | 1 cycle |
| `INCRWC` | Directly increment RWC fields | 1 cycle |
| Instructions with `AddrMod` | Implicitly update RWCs via `ApplyAddrMod()` | Varies |

Software is encouraged to use the `AddrMod` mechanism rather than standalone `SETRWC`/`INCRWC` instructions, as this avoids consuming additional instruction cycles.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 7](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L7-L7)[WormholeB0/TensixTile/TensixCoprocessor/RWCs.md 60-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/RWCs.md?plain=1#L60-L74)

## Fidelity Phases

**Fidelity phases** enable the Matrix Unit to perform multi-precision arithmetic by decomposing operations across multiple instruction invocations. Each phase processes a subset of the operand bits, with results accumulated in `Dst`.

### Floating-Point Fidelity Phases

For floating-point multiplication (e.g., `MVMUL`), the number of fidelity phases required depends on the operand precisions:

*   **1 phase:** BF8 Ă— BF8, FP8 Ă— FP8
*   **2 phases:** BF16 Ă— BF16, FP16 Ă— FP16
*   **3 phases:** BF16 Ă— BF8, FP16 Ă— FP8
*   **4 phases:** TF32 (19-bit) Ă— TF32

The `RWCs[CurrentThread].FidelityPhase` counter (0-3) tracks which phase is currently executing. Instructions automatically increment this counter via the `AddrMod` mechanism, allowing software to loop over phases:

```
// Example: 2-phase BF16Ă—BF16 multiply-accumulate
SETRWC Dst=0, SrcA=0, SrcB=0, FidelityPhase=0
MVMUL AddrMod=1  // Phase 0, AddrMod increments FidelityPhase
MVMUL AddrMod=2  // Phase 1, AddrMod resets FidelityPhase and advances rows
```

### Integer Fidelity Phases

For integer operations, fidelity phases relate to the **maximum magnitude** of the inputs:

*   **1 phase:** INT4 range (-7 to +7)
*   **2 phases:** INT8 range (-127 to +127)
*   **3 phases:** INT8 range (-255 to +255)
*   **4 phases:** INT8 full range (-128 to +127) or INT16

Software must ensure that operands stay within the assumed range for the selected number of fidelity phases.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 33](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L33-L33)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 60-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L60-L74)[WormholeB0/TensixTile/TensixCoprocessor/RWCs.md 17](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/RWCs.md?plain=1#L17-L17)

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 21-35](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L21-L35)

### Instruction Scheduling Rules

#### Automatic Stalling

The Matrix Unit enforces pipeline hazards through automatic stalling:

1.   **After `MOVB2A` or `MOVD2A`:** On the next cycle, only `MOVD2A` or `MOVB2A` can execute. Any other Matrix Unit instruction will cause the thread to stall for one cycle.

2.   **Wait Gate Checks:** Some instructions (e.g., `MOVB2A`) automatically wait at the Wait Gate until the destination register file's `AllowedClient` field indicates the Matrix Unit has ownership.

#### Manual Stalling with STALLWAIT

Software can use the `STALLWAIT` instruction to explicitly wait for pipeline conditions before issuing Matrix Unit instructions. For example, before `MOVB2A`, software may use `STALLWAIT` with block bit B6 and condition code C10 to ensure `SrcA[MatrixUnit.SrcABank].AllowedClient == MatrixUnit`.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 64-68](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L64-L68)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 35](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L35-L35)

## Performance Characteristics

At Wormhole's standard 1 GHz clock rate, theoretical maximum performance per Matrix Unit:

### Floating-Point Performance

| Instruction | 1 Phase | 2 Phases | 3 Phases | 4 Phases |
| --- | --- | --- | --- | --- |
| `MVMUL` (no broadcast) | 4.096 TFLOP/s | 2.048 TFLOP/s | 1.366 TFLOP/s | 1.024 TFLOP/s |
| `DOTPV` | 4.096 TFLOP/s | 2.048 TFLOP/s | 1.366 TFLOP/s | 1.024 TFLOP/s |
| `GAPOOL` | 2.048 TFLOP/s | 1.024 TFLOP/s | 0.683 TFLOP/s | 0.512 TFLOP/s |
| `MVMUL` (with broadcast) | 0.560 TFLOP/s | 0.280 TFLOP/s | 0.187 TFLOP/s | 0.140 TFLOP/s |
| `ELWMUL` | 0.256 TFLOP/s | 0.128 TFLOP/s | 0.085 TFLOP/s | 0.064 TFLOP/s |
| `GMPOOL` | 0.256 TFLOP/s | 0.256 TFLOP/s | 0.256 TFLOP/s | 0.256 TFLOP/s |
| `ELWADD` / `ELWSUB` (with add) | 0.256 TFLOP/s | 0.256 TFLOP/s | 0.256 TFLOP/s | 0.256 TFLOP/s |
| `ELWADD` / `ELWSUB` (no add) | 0.128 TFLOP/s | 0.128 TFLOP/s | 0.128 TFLOP/s | 0.128 TFLOP/s |

**Note:**`GMPOOL`, `ELWADD`, and `ELWSUB` do not require multiple fidelity phases regardless of data type, so performance is constant across columns.

### Comparison to Vector Unit

The Matrix Unit significantly outperforms the Vector Unit (SFPU) for operations it supports:

*   **Matrix Unit:** 4.096 TFLOP/s peak (1-phase operations)
*   **Vector Unit:** 0.064 TFLOP/s peak (`SFPMAD` at FP32)

The Matrix Unit achieves 64Ă— higher throughput, but only supports â‰¤19-bit operand precision. For operations requiring full 32-bit precision on all operands, the Vector Unit must be used despite its lower performance.

## Instruction Encoding Example

Matrix Unit instructions use a standard 32-bit encoding format. The `MOVB2A` instruction illustrates the typical structure:

![Instruction Encoding](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Instruction%20Encoding#LNaN-LNaN)

### Field Breakdown

| Field | Bits | Purpose |
| --- | --- | --- |
| Opcode | [31:24] | Instruction identifier (`0x0B` for `MOVB2A`) |
| `SrcARow` | [22:17] | Row index in `SrcA` (added to `RWCs[CurrentThread].SrcA`) |
| `AddrMod` | [16:15] | Index into `ThreadConfig` increment tables |
| `Move4Rows` | [14] | If set, move 4 aligned rows; else move 1 row |
| `SrcBRow` | [5:0] | Row index in `SrcB` (added to `RWCs[CurrentThread].SrcB`) |

The functional model for this instruction reads from `SrcB`, applies optional denormal flushing, respects lane masking via `LaneConfig`, and writes to `SrcA`. After execution, it calls `ApplyAddrMod(AddrMod)` to update RWCs.

## Legacy Instructions

The following instructions exist in the instruction set but perform no useful operation in Wormhole B0:

| Instruction | Behavior |
| --- | --- |
| `MFCONV3S1`, `CONV3S1`, `CONV3S2`, `APOOL3S1`, `APOOL3S2` | Compute `Dst += 0` |
| `MPOOL3S1`, `MPOOL3S2` | Similar to `GMPOOL` with zero `SrcA` |

These instructions were functional in earlier architectures but were neutered for Wormhole rather than being fully removed. Their opcode values are likely to be repurposed in future implementations. This behavior is a **NonContractualBehavior**.
