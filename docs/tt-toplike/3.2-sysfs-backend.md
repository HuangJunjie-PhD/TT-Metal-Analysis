# Sysfs Backend

Relevant source files
*   [docs/SYSFS_BACKEND.md](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1)
*   [src/backend/sysfs.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs)

The Sysfs backend provides **non-invasive hardware monitoring** by reading from Linux's `hwmon` subsystem (`/sys/class/hwmon/`). It is designed for monitoring Tenstorrent hardware that is actively running workloads (LLMs, training, inference) where direct PCI access via other backends might cause resource conflicts or crashes [docs/SYSFS_BACKEND.md 3-5](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L3-L5)

## Overview and Implementation

The `SysfsBackend` struct implements the `TelemetryBackend` trait [src/backend/sysfs.rs 44-60](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L44-L60) Unlike the Luwen backend, which requires direct PCI BAR0 memory mapping, the Sysfs backend uses **kernel-mediated access**[docs/SYSFS_BACKEND.md 10-17](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L10-L17) This ensures:

*   **Zero PCI interference**: Reads are performed from kernel-maintained buffers [docs/SYSFS_BACKEND.md 17](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L17-L17)
*   **No special permissions**: Most `sysfs` files are world-readable, removing the need for root or `ttkmd` module permissions [docs/SYSFS_BACKEND.md 19](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L19-L19)
*   **Concurrency**: Multiple monitoring tools can read from `sysfs` simultaneously [docs/SYSFS_BACKEND.md 18](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L18-L18)

### Data Flow and Initialization

During initialization, the backend scans `/sys/class/hwmon/` for directories containing a `name` file with Tenstorrent-related strings such as `"tenstorrent"`, `"grayskull"`, `"wormhole"`, or `"blackhole"`[src/backend/sysfs.rs 108-120](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L108-L120)

## Device Discovery and Mapping

The backend maps `hwmon` entries to internal `Device` models. It attempts to determine the `Architecture` by parsing the `name` string [src/backend/sysfs.rs 128-136](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L128-L136)

### PCI Address Extraction

To uniquely identify devices, the backend extracts the PCI bus address by reading the symlink at `/sys/class/hwmon/hwmon*/device`[docs/SYSFS_BACKEND.md 111-118](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L111-L118) This allows the backend to correlate `hwmon` data with other system tools.

### Board Type Enrichment

Because the kernel driver often only exposes the chip name (e.g., "blackhole"), the backend optionally runs `tt-smi -s` during `init()` to "enrich" the `board_type` field with the specific SKU (e.g., "P300c") [src/backend/sysfs.rs 172-182](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L172-L182)

## Sensor Layout and Unit Conversion

The `SensorPaths` struct caches the location of specific files to avoid repeated filesystem lookups during the high-frequency `update()` loop [src/backend/sysfs.rs 31-39](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L31-L39)

| Metric | Sysfs File Pattern | Unit Conversion |
| --- | --- | --- |
| **Temperature** | `temp*_input` | Millicelsius → Celsius (/ 1000) |
| **Voltage** | `in*_input` | Millivolts → Volts (/ 1000) |
| **Power** | `power*_input` | Microwatts → Watts (/ 1000000) |
| **Current** | `curr*_input` | Milliamps → Amperes (/ 1000) |

## Limitations

While non-invasive, the Sysfs backend has restricted visibility compared to the Hybrid or Luwen backends:

1.   **No SMBUS Data**: Information like firmware versions, ARC heartbeats, and DDR training status is unavailable because it requires direct hardware access not exposed by standard `hwmon`[docs/SYSFS_BACKEND.md 62-70](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L62-L70)
2.   **Missing Clocks**: AICLK (Clock frequency) is not a standard `hwmon` attribute [docs/SYSFS_BACKEND.md 67](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L67-L67)
3.   **Driver Dependency**: Availability of power and current sensors depends entirely on whether the specific kernel driver version exposes them [src/backend/sysfs.rs 20-21](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/sysfs.rs#L20-L21)

## Troubleshooting

### Common Issues

| Symptom | Potential Cause | Solution |
| --- | --- | --- |
| **No devices found** | Driver not loaded or `hwmon` support disabled [docs/SYSFS_BACKEND.md 208-212](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L208-L212) | Check `lsmod | grep tt` and verify `/sys/class/hwmon/` contents [docs/SYSFS_BACKEND.md 214-226](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L214-L226) |
| **Power shows 0W** | Driver does not expose `power*_input`[docs/SYSFS_BACKEND.md 234](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L234-L234) | Use JSON backend (requires `tt-smi`) or Luwen (if idle) [docs/SYSFS_BACKEND.md 237-239](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L237-L239) |
| **Incorrect Values** | Unrecognized unit scaling in driver | Verify raw values via `cat /sys/class/hwmon/hwmon*/temp1_input`[docs/SYSFS_BACKEND.md 141-143](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1#L141-L143) |
