## Purpose and Scope

This page documents the hardware architecture details of Tenstorrent accelerators that are directly relevant to `tt-lang` kernel programming. It covers the Tensix core architecture, DST register file organization, memory hierarchy, and tile-based data layout. This information is essential for understanding DST allocation constraints, performance characteristics, and the three-thread execution model.

For language-level abstractions (dataflow buffers, blocks, pipes), see **Core Concepts**. For compilation of these abstractions to hardware operations, see **Compilation Pipeline**.

* * *

## Tensix Core Architecture

A **Tensix core** is the fundamental compute unit in Tenstorrent hardware. Each core executes kernels using a three-thread architecture with distinct responsibilities. These threads are mapped to specific hardware processors: **MATH** (compute), **BRISC** (Data Movement - Write), and **NCRISC** (Data Movement - Read).

### Thread Model and DSL Mapping

| Thread | DSL Decorator | Hardware Mapping | Primary Responsibility |
| --- | --- | --- | --- |
| **MATH** | `@ttl.compute()` | TRISC processors | Execute tile math operations (SFPU/FPU) on DST registers. |
| **BRISC** | `@ttl.datamovement()` | BRISC processor | Data movement: L1 → DRAM, Core → Core (transmit). |
| **NCRISC** | `@ttl.datamovement()` | NCRISC processor | Data movement: DRAM → L1, Core → Core (receive). |

The threads synchronize through **Circular Buffers (CBs)** in L1 SRAM. The `tt-lang` compiler manages this by lowering `ttl.cb` types to `ttkernel::CBType` with flattened element counts [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 70-73](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L70-L73)

## DST Register File

The **DST (Destination) Register File** is the MATH thread's scratchpad for tile computations. It has strict physical and logical capacity constraints that drive the compiler's subblocking and unrolling strategies [docs/development/DST_Utilization.md 5-25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Utilization.md?plain=1#L5-L25)

### Physical and Logical Capacity

The physical hardware contains **16 tile slots**[include/ttlang/Dialect/TTL/IR/TTL.h 98-99](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L98-L99) However, the logical capacity available to a kernel depends on the target architecture (e.g., Blackhole vs Wormhole_B0), the data type, and synchronization mode [include/ttlang/Dialect/TTL/IR/TTL.h 46-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L46-L66)

| Category | Input Source | DST Slot Usage | In-Place | Example Op |
| --- | --- | --- | --- | --- |
| **SFPU Binary** | DST (both) | 2 inputs, 1 fresh output | No | `mul_binary_tile`[docs/development/DST_Allocation.md 19-24](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Allocation.md?plain=1#L19-L24) |
| **FPU Binary** | CB (both) | 0 inputs, 1 fresh output | No | `add_tiles`[docs/development/DST_Allocation.md 25-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Allocation.md?plain=1#L25-L30) |

| **Matmul** | CB (A+B) | 0 inputs, M*N footprint | No (Accumulates) | `ttl.tile_matmul_block`[docs/development/DST_Allocation.md 82-93](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Allocation.md?plain=1#L82-L93) |

## Tile Architecture

The **Tile** is the fundamental unit of computation. A standard tile is a **32x32** grid of elements [include/ttlang/Dialect/TTL/IR/TTL.h 23-24](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L23-L24)

### Hardware Constraints on Tiling

*   **Innermost Dimensions:** Tensors in tile layout typically have dimensions divisible by 32.
*   **Data Types:** A single tile's memory footprint varies by element type (e.g., `bf16`, `f32`).
*   **Subblocking:** To fit computations within the 16-tile DST limit, the compiler partitions large blocks into "subblocks" defined by the `unroll_factor`[lib/Dialect/TTL/Transforms/TTLSubblockComputeForDST.cpp 113-115](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLSubblockComputeForDST.cpp#L113-L115)

## NOC (Network-on-Chip) and Addressing

The **NOC** enables asynchronous data transfers between cores and between cores and DRAM. `tt-lang` exposes this through data movement threads and tensor accessors.

### Tensor Accessors and Addressing

Data movement threads (NCRISC/BRISC) use `TensorAccessor` to calculate hardware-specific addresses for NOC transfers. The `TensorAccessor` handles CTA (Compile-Time Argument) offsets to find the correct bank and page regardless of whether a tensor is interleaved or sharded [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 161-182](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L161-L182)
