This page provides practical guidance for debugging compilation failures, runtime errors, and accuracy issues in the pytorch2.0_ttnn compilation system. It covers common error patterns, debugging flags, interactive debugging techniques, and accuracy troubleshooting workflows.

For information about the guard system that prevents problematic operations from compiling, see [Guard System and Operation Blocklists](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.4-guard-system-and-operation-blocklists). For details on test infrastructure and automated test generation, see [Testing Framework and Infrastructure](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4-testing-framework-and-infrastructure). For code export utilities, see [Code Export and Standalone Script Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/8.1-code-export-and-standalone-script-generation).

## Overview of Debugging Approaches

The pytorch2.0_ttnn system provides multiple debugging mechanisms depending on the type of issue:

| Issue Type | Primary Tool | Secondary Tools |
| --- | --- | --- |
| Compilation Failures | `--trace` flag, `pdb` | Pass Manager logs |
| Incorrect Arguments | Error logs, TTNN docs | Operation wrappers inspection |
| Type Mismatches | Error messages | AddDataMovePass analysis |
| Accuracy Issues | `--export_code=accuracy` | Per-operation comparison |
| Memory Issues | `--trace` | DeallocationPass inspection |
| Multi-Device Issues | `--data_parallel` logs | Mesh device configuration |

## Common Error Patterns

### Incompatible Function Arguments

This error occurs when TTNN operations receive arguments in an incorrect format or order.

**Error Signature:**

```
TypeError: __call__(): incompatible function arguments. The following argument types are supported:
```

**Example Error:**

```
self = FastOperation(python_fully_qualified_name='ttnn.clone', ...)
function_args = (ttnn.Tensor([1, 256, 1024], bfloat16, tile), MemoryConfig, bfloat16)
```

**Diagnostic Process:**

**Resolution Steps:**

1.   **Identify the operation** from the error message (e.g., `ttnn.clone`)
2.   **Locate the binding** in the tt-metal repository at `ttnn/cpp/ttnn/operations/*/pybind.cpp`
3.   **Find the `py::kw_only()` marker** that separates positional from keyword-only arguments
4.   **Search tt-metal codebase** for correct usage patterns
5.   **Fix the operation wrapper** to match the expected signature

**Code Reference:** The operation wrappers that may need correction are in [torch_ttnn/passes/lowering/target_wrappers.py 1-500](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L1-L500)

Sources: [docs/ProblemSolving.md 15-91](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/ProblemSolving.md?plain=1#L15-L91)

### Backend Compiler Failure

This error indicates that one of the compilation passes failed during graph transformation.

**Error Signature:**

```
torch._dynamo.exc.BackendCompilerFailed: backend='ttnn_backend' raised:
Exception: An error occurred when running the 'ToTtPass' pass after the following passes: []
```

#### Interactive Debugging with pdb

**Debugging Workflow:**

1.   **Run with trace flag:**

`pytest tests/models/bert/test_bert.py --trace`
2.   **Set breakpoint at failure point:**

`(Pdb) b ../tt-metal/python_env/lib/python3.8/site-packages/torch/fx/passes/infra/pass_manager.py:296`
3.   **Run and wait for breakpoint:**

`(Pdb) r`
4.   **Enter interactive mode:**

`(Pdb) interact>>> import pdb>>> pdb.post_mortem(e.__traceback__)`
5.   **Inspect the exception context** to determine which operation or graph transformation caused the failure

Sources: [docs/ProblemSolving.md 92-145](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/ProblemSolving.md?plain=1#L92-L145)

### Tensor Type Mismatch

This error occurs when a `torch.Tensor` is passed to a TTNN operation that expects a `ttnn.Tensor`.

**Error Signature:**

```
TypeError: conv2d(): incompatible function arguments. The following argument types are supported:
    1. (..., input_tensor: tt_lib.tensor.Tensor, ...)
Invoked with: kwargs: input_tensor=tensor(..., dtype=torch.bfloat16), ...
```

**Analysis:**

**Root Cause:** The AddDataMovePass failed to insert a `ttnn.from_torch` operation before the problematic operation.

**Common Causes:**

1.   **Missing target wrapper** - Operation not registered in target_wrappers
2.   **Incorrect input classification** - InputAnalysisPass misclassified the tensor
3.   **Guard blocked conversion** - Operation was blocked by guard system

**Resolution:** Examine [torch_ttnn/passes/lowering/add_data_move_pass.py 1-500](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L1-L500) to understand why data movement was not inserted for this specific operation.

Sources: [docs/ProblemSolving.md 146-177](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/ProblemSolving.md?plain=1#L146-L177)

## Accuracy Debugging with Export Code

The `--export_code=accuracy` flag generates standalone Python scripts that compare each PyTorch operation with its TTNN lowered counterpart, identifying the first operation pair that produces low accuracy.

### Accuracy Debugging Workflow

### Example Usage

**Generate Accuracy Debugger:**

`pytest tests/models/mobilenet_ssd/test_mobilenet_ssd.py --export_code=accuracy`
**Execute Debugger:**

`python3 tests/export_code/accuracy/MobileNetSSD_code.py`
**Example Output:**

```
Traceback (most recent call last):
  File "MobileNetSSD_code.py", line 5650, in <module>
    forward(*inputs)
  File "MobileNetSSD_code.py", line 1347, in forward
    test_accuracy(mean, ttnn_mean)
  File "MobileNetSSD_code.py", line 225, in test_accuracy
    assert_with_pcc(expected, actual, pcc = 0.90)
AssertionError: 0.7265316050734197
```

**Analysis:** Line 1347 shows that `ttnn.mean` operation produced accuracy of only 0.72, indicating a lowering issue with this operation.

### Generated Script Structure

The exported script contains:

1.   **Input data serialization** - All test inputs as pickled tensors
2.   **Forward function assembly** - Both PyTorch and TTNN execution paths
3.   **Per-operation comparison** - PCC check after each operation pair
4.   **Early termination** - Script stops at first low-accuracy operation

**Script Generation:** See [torch_ttnn/export_code.py 1-300](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/export_code.py#L1-L300) for the implementation.

## Memory Debugging

Memory issues typically manifest as out-of-memory errors or performance degradation due to L1 memory pressure on Tenstorrent devices.

### Deallocation Pass Analysis

**DeallocationPass Behavior:**

1.   **Tracks last use** of each tensor using `TrackUnusedValues`[torch_ttnn/passes/deallocation_pass.py 21-75](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L21-L75)
2.   **Inserts deallocate calls** after the last user of each tensor [torch_ttnn/passes/deallocation_pass.py 82-112](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L82-L112)
3.   **Skips certain node types:**
    *   `pack_to_tuple` - References to other nodes
    *   `operator.getitem` - References to tuple elements
    *   Cached nodes - Marked with `is_cached=True` metadata

**Common Issues:**

| Issue | Symptom | Solution |
| --- | --- | --- |
| Deallocate not called | Memory accumulation | Check `torch.fx.node.has_side_effect(deallocate)` |
| Premature deallocation | Tensor use after free | Verify last-use tracking logic |
| L1 pressure | Slow execution | Review tensor sharding strategy |

Sources: [torch_ttnn/passes/deallocation_pass.py 1-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L1-L113)

## Multi-Device Debugging

Multi-device tests introduce additional complexity with mesh devices, sharding, and replication.

### Multi-Device Test Execution

**Common Multi-Device Issues:**

1.   **Incorrect sharding** - Arguments not properly distributed

    *   Check `MultiDeviceShardAnalysisPass` output
    *   Verify `concat_size` calculation

2.   **Replication failures** - Parameters not replicated correctly

    *   Examine `replicate_to_mesh` calls
    *   Check parameter classification in `InputAnalysisPass`

3.   **Mesh device configuration** - Wrong device mesh shape

    *   Verify `TorchTtnnOption(data_parallel=True)`
    *   Check `ttnn.open_mesh_device(MeshShape(1, 2))`

**Multi-Device Test Example:** See [tests/multi_device/test_multi_bert.py 1-93](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/multi_device/test_multi_bert.py#L1-L93) and [tests/multi_device/test_multi_mnist.py 1-70](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/multi_device/test_multi_mnist.py#L1-L70) for reference implementations.

## Troubleshooting Checklist

### Compilation Failures

*   - [x]  Run with `--trace` flag for interactive debugging
*   - [x]  Set breakpoint at `PassManager` exception handler
*   - [x]  Use `pdb.post_mortem()` to inspect exception context
*   - [x]  Check which pass failed (ToTtPass, AddDataMovePass, etc.)
*   - [x]  Verify operation is in target_wrappers
*   - [x]  Check guard blocklists for operation

### Type Mismatch Errors

*   - [x]  Identify which argument has wrong type
*   - [x]  Check if `ttnn.from_torch` was inserted
*   - [x]  Examine AddDataMovePass logs
*   - [x]  Verify NodeInputAligner execution
*   - [x]  Check operation signature in tt-metal pybind
*   - [x]  Ensure keyword-only args use `py::kw_only()`

### Accuracy Issues

*   - [x]  Run with `--export_code=accuracy`
*   - [x]  Execute generated script
*   - [x]  Identify first low-accuracy operation
*   - [x]  Compare operation lowering with tt-metal tests
*   - [x]  Check operation wrapper implementation
*   - [x]  Verify input tensor layouts (TILE vs ROW_MAJOR)
*   - [x]  Consider adding to guard blocklist if unfixable

### Memory Issues

*   - [x]  Check DeallocationPass execution
*   - [x]  Verify `deallocate()` calls after last uses
*   - [x]  Review tensor sizes and L1 memory capacity
*   - [x]  Check for memory leaks (tensors never deallocated)
*   - [x]  Examine cached tensor strategy

### Multi-Device Issues

*   - [x]  Verify `--data_parallel` flag used
*   - [x]  Check mesh device opened correctly
*   - [x]  Examine input classification (PARAMETER vs ARGUMENT)
*   - [x]  Verify sharding strategy for arguments
*   - [x]  Check replication for parameters
*   - [x]  Ensure ConcatMeshToTensor for outputs
