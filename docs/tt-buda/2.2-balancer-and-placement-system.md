The Balancer and Placement System is a critical component in the PyBuda compilation pipeline responsible for optimizing operation placement on Tenstorrent hardware and efficiently managing computational resources. It takes an optimized computational graph and transforms it into a balanced graph with optimal operation placements, which is then lowered to a Buda Netlist.

## 1. System Overview

The Balancer and Placement System sits between graph optimization and backend compilation in the PyBuda framework. Its primary responsibilities include:

1.   Determining optimal grid sizes for each operation
2.   Placing operations on hardware efficiently
3.   Managing resources across epochs and chips
4.   Handling fork-join patterns in the computational graph
5.   Ensuring valid constraints for hardware execution

**Key Components:**

*   **Legalizer**: Ensures operations have valid configurations for hardware execution
*   **Graph Solver**: Solves constraints across the graph to find valid operation configurations
*   **Policy Manager**: Applies balancing policies to generate optimal grid shapes
*   **Interactive Placer**: Places operations on the hardware grid with specific constraints
*   **Epoch Planning**: Organizes operations into execution epochs
*   **Fork-Join Buffering**: Handles divergent and convergent paths in the computational graph

## 2. Operation Models and Constraints

Operation models (OpModels) are the fundamental unit of work in the balancer. Each OpModel represents an operation with specific grid shape, block shape, buffering requirements, and execution characteristics.

### 2.1 OpModel Structure

The `OpModel` contains:

*   `grid_shape`: The 2D (row × column) shape of cores assigned to the operation
*   `block_shape`: The detailed shape information for data blocks and tiles
*   Input, output, and parameter buffer models describing memory requirements
*   Execution cycle information to estimate operation performance

*   [pybuda/csrc/balancer/types.hpp 60-145](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/types.hpp#L60-L145)
*   [pybuda/csrc/balancer/types.cpp](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/types.cpp)

### 2.2 The Graph Solver

The Graph Solver is responsible for finding valid OpModel configurations that satisfy all constraints across the computational graph. It works by:

1.   Taking a graph and generating legal operation models for each node
2.   Solving constraints between connected operations
3.   Cutting edges when constraints cannot be satisfied
4.   Providing valid OpModel selections to the balancer

The Graph Solver uses different constraint solvers based on the target hardware architecture (Grayskull, Wormhole, etc.), as seen in [pybuda/csrc/balancer/balancer.cpp 87-107](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/balancer.cpp#L87-L107)

## 3. Balancing Policies

The Policy Manager applies specific balancing policies to determine optimal grid shapes for operations. Different policies are optimized for different types of workloads.

### 3.1 Available Policies

| Policy Type | Description | Best For |
| --- | --- | --- |
| Ribbon | Organizes operations in "ribbons" of consistent height | General purpose, default for most workloads |
| CNN | Optimized for convolutional neural networks | Vision models with heavy convolutions |
| NLP | Optimized for natural language processing models | Transformer-based models |
| MinimizeGrid | Minimizes grid size for each operation | Maximizing operation packing |
| Random | Randomly selects grid shapes (for testing) | Testing and debugging |
| SingleOpPerEpoch | Places each operation in its own epoch | Testing and isolation |

### 3.2 The Ribbon Policy

The Ribbon policy is the most sophisticated and commonly used policy. It:

1.   Attempts to maintain a consistent "ribbon" height (number of rows) across operations
2.   Picks ribbon size based on the dimensions of matrix multiplication operations
3.   Optimizes operation placement within epochs
4.   Specially handles sparse-dense matrix multiplication pairs

The Ribbon policy uses a dynamic target cycles calculation to optimize operation performance within epochs. It can also dynamically adjust ribbon size as matrix multiplication dimensions change.

### 3.3 Policy Selection and Management

The Policy Manager makes the final decisions about which OpModels to use based on the selected policy and optimization criteria, including:

1.   Matching target ribbon sizes
2.   Optimizing for execution cycles
3.   Balancing memory usage vs. performance
4.   Ensuring operation models are valid for hardware

## 4. Interactive Placement

The Interactive Placer is responsible for the actual placement of operations on the hardware grid. It works closely with the Policy Manager to implement the placement decisions.

### 4.1 Placement Process

The Interactive Placer maintains the state of the current epoch and chip, and makes decisions about:

*   Where to place each operation on the grid
*   When to start a new epoch
*   When to move to a different chip
*   Whether to transpose operation grid shapes

Sources:

*   [pybuda/csrc/balancer/policies/policy_utils.cpp 50-133](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/policies/policy_utils.cpp#L50-L133)

### 4.2 Epoch Planning

Epochs are groups of operations that are executed together. The system plans epochs based on:

1.   Explicit epoch breaks defined in configuration
2.   Node epoch type changes (e.g., Forward → Backward → Optimizer)
3.   Grid space availability
4.   Chip breaks for multi-chip scenarios

The scheduler provides an initial ordering of operations, and the balancer may modify this based on placement constraints. When an epoch is full or an epoch break is encountered, the system finalizes the current epoch and starts a new one.

## 5. Fork-Join Buffering

Fork-Join patterns occur when a single operation sends data to multiple operations (fork), and those paths later converge at a single operation (join). These patterns require special handling to ensure proper data flow and synchronization.

### 5.1 Fork-Join Detection

The system detects fork-join patterns by identifying nodes with multiple data users and tracing the paths until they converge. This is implemented in [pybuda/csrc/passes/fork_join.cpp 132-238](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/fork_join.cpp#L132-L238)

Sources:

*   [pybuda/csrc/passes/fork_join.cpp 132-238](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/fork_join.cpp#L132-L238)
*   [pybuda/csrc/passes/fork_join.cpp 59-81](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/fork_join.cpp#L59-L81)

### 5.2 Fork-Join Buffering

When fork-join patterns are detected, the system calculates the amount of buffering needed to ensure both paths can make forward progress at maximum speed. Buffering is inserted in the form of:

1.   NOP operations that consume cycles but don't perform computation
2.   Buffering queues that store intermediate results

This buffering process is integrated into the balancing policy through methods like `buffer_fork_joins()` in [pybuda/csrc/balancer/policies/policy_ribbon2.cpp 539-620](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/policies/policy_ribbon2.cpp#L539-L620)

## 7. Special Case Handling

The Balancer includes special handling for certain operation types and patterns to optimize performance.

1.   Detection of sparse-dense matrix pairs

## 8. Performance Estimation

The Balancer estimates operation performance to make informed placement decisions:

Performance estimation takes into account:

1.   Operation execution cycles based on grid shape and data format
2.   Memory bandwidth limitations
3.   PCIe bandwidth limitations
4.   The slowest of these factors becomes the "limiter cycles"

This estimation is implemented in methods like `get_limiter_cycles()` and `get_op_cycles_estimates()` in [pybuda/csrc/balancer/policies/policy_utils.hpp 153-283](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/policies/policy_utils.hpp#L153-L283)
