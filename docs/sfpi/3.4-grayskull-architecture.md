## Purpose and Scope

This document describes the Grayskull-specific implementation of the SFPI programming interface. Grayskull represents one of three Tenstorrent chip architectures supported by the SFPI system, alongside Wormhole ([3.2](https://deepwiki.com/tenstorrent/sfpi/3.2-wormhole-architecture)) and Blackhole ([3.3](https://deepwiki.com/tenstorrent/sfpi/3.3-blackhole-architecture)). This page covers the hardware abstraction layer structure, architectural differences, conditional compilation mechanisms, and testing infrastructure specific to Grayskull.

For information about hardware intrinsics common across all architectures, see [3.1](https://deepwiki.com/tenstorrent/sfpi/3.1-hardware-intrinsics-overview). For the core SFPI programming interface that sits above these architecture-specific layers, see [2](https://deepwiki.com/tenstorrent/sfpi/2-core-sfpi-programming-interface).

## Hardware Abstraction Layer Structure

### File Organization

The Grayskull architecture follows the standard SFPI file organization pattern:

| File Path | Purpose | Key Contents |
| --- | --- | --- |
| `include/grayskull/sfpi_hw.h` | Hardware intrinsics and constants | `__builtin_rvtt_sfp*` macros, addressing modes, format constants |
| `include/grayskull/sfpi_imp.h` | Type implementations | `vFloat`, `vInt`, `vUInt` constructors, operators, register access |
| `include/grayskull/sfpi_lib.h` | Math library functions | Lookup tables, float manipulation, type conversions |

**Sources:** Diagram 1 (Repository Structure), Diagram 5 (Platform-Specific Architecture Variants)

### Conditional Compilation

Grayskull-specific code is selected at compile-time using the `__riscv_xtttensixgs` preprocessor macro. The compiler defines this macro when targeting the Grayskull architecture, allowing the SFPI interface to include the appropriate platform-specific headers.

**Sources:** Diagram 5 (Conditional Compilation section), Diagram 1 (Hardware Abstraction Layers)

### Key Architectural Differences

#### Register File

Grayskull implements the standard SFPU register file organization:

*   **LRegs (Local Registers):** 8 scalar registers (LReg0-LReg7) for temporary storage
*   **DRegs (Destination Registers):** 512 addressable 32-bit vector registers (1024 rows with stride 2)
*   **CRegs (Constant Registers):** Hardware constant registers for common values

**Sources:**[include/wormhole/sfpi_hw.h 92-98](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_hw.h#L92-L98)[include/wormhole/sfpi_imp.h 162-172](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_imp.h#L162-L172)

#### Addressing Modes

The Grayskull architecture supports load/store operations with various addressing modes. The `SFPLOAD_ADDR_MODE_NOINC` and `SFPSTORE_ADDR_MODE_NOINC` constants define no-increment addressing behavior, though the specific values may differ from Wormhole (3) and Blackhole (7).

**Sources:**[include/wormhole/sfpi_hw.h 104](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_hw.h#L104-L104)[include/blackhole/sfpi_hw.h 108](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/blackhole/sfpi_hw.h#L108-L108)

## Hardware Intrinsics

### Intrinsic Categories

Grayskull hardware intrinsics follow the same organizational pattern as other architectures:

**Sources:**[include/wormhole/sfpi_hw.h 18-91](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_hw.h#L18-L91)

### Macro Definition Pattern

Grayskull intrinsics are defined as preprocessor macros that wrap GCC built-in functions. The pattern follows this structure for Grayskull-specific implementations:

*   `__builtin_rvtt_gs_*` - Grayskull-specific built-in function names
*   Macros abstract the platform prefix, providing uniform `__builtin_rvtt_sfp*` names
*   Arguments typically include: instruction buffer, vector operands, modifiers, immediate values

**Sources:**[include/wormhole/sfpi_hw.h 18-91](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_hw.h#L18-L91)[include/blackhole/sfpi_hw.h 18-94](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/blackhole/sfpi_hw.h#L18-L94)

## Type Implementations

### Vector Type Constructors

The implementation layer provides constructors for SFPI vector types:

**Sources:**[include/wormhole/sfpi_imp.h 97-129](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_imp.h#L97-L129)

### Operator Overloading

The Grayskull implementation provides operator overloading for vector types, enabling intuitive C++ syntax:

| Operator | Type | Implementation | Hardware Intrinsic |
| --- | --- | --- | --- |
| `+` | `vFloat` | Addition | `__builtin_rvtt_sfpadd` |
| `-` | `vFloat` | Subtraction (via MAD) | `__builtin_rvtt_sfpmad` with `-1` |
| `*` | `vFloat` | Multiplication | `__builtin_rvtt_sfpmul` |
| `==`, `!=`, `<`, etc. | Comparison | Conditional creation | `__builtin_rvtt_sfpxfcmps/v` |
| `&`, ` | `,`^` | `vInt/vUInt` | Bitwise operations |
| `<<`, `>>` | `vInt/vUInt` | Shift operations | `__builtin_rvtt_sfpshft_i/v` |

**Sources:**[include/wormhole/sfpi_imp.h 27-142](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_imp.h#L27-L142)

### Register Access

The `__vDReg` class provides type-safe access to destination registers with assignment operators:

*   **Store Float:**`dst_reg[n] = vFloat(value)` → stores using `SFPSTORE_MOD0_FMT_SRCB`
*   **Store Int:**`dst_reg[n] = vInt(value)` → stores using `SFPSTORE_MOD0_FMT_INT32_TO_SM`
*   **Load:**`vFloat v = dst_reg[n]` → loads using `SFPLOAD_MOD0_FMT_SRCB`

**Sources:**[include/wormhole/sfpi_imp.h 27-81](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_imp.h#L27-L81)

## Math Library Functions

The Grayskull math library (`include/grayskull/sfpi_lib.h`) provides high-level mathematical operations built on top of hardware intrinsics. While specific Grayskull implementations may differ, the library typically includes:

### Function Categories

| Category | Functions | Purpose |
| --- | --- | --- |
| **Lookup Tables** | `lut()`, `lut2()`, `lut_sign()` | Table-based approximations |
| **Float Manipulation** | `exexp()`, `exman()`, `setexp()`, `setman()`, `setsgn()` | Exponent/mantissa extraction and modification |
| **Exponent Operations** | `addexp()` | Exponent arithmetic |
| **Type Conversions** | `int32_to_float()`, `float_to_fp16a()`, `float_to_fp16b()`, `float_to_int8()`, `float_to_uint8()` | Format conversions with rounding |
| **Vector Operations** | `subvec_transp()`, `vec_swap()`, `vec_min_max()` | Data movement and comparison |

## Testing Infrastructure

### Test Suite Organization

Grayskull maintains platform-specific tests to validate instruction generation and compiler optimization:

**Sources:** Diagram 5 (Grayskull Architecture section), Diagram 4 (Test Source Code)

### Test Coverage

The Grayskull test suite includes:

1.   **Instruction-Level Tests** (`coverage.cc`): Validates that SFPI API calls generate correct assembly instructions for the Grayskull architecture
2.   **Register Move Optimization** (`regmov.cc`): Ensures the compiler's move pass optimization works correctly for Grayskull, eliminating unnecessary register-to-register moves

Note that compared to Wormhole and Blackhole, Grayskull test infrastructure notably does not include `ckernel.cc` kernel function tests or corresponding integer/float output validation files.

## Usage Considerations

### When to Target Grayskull

Target the Grayskull architecture when:

*   Developing kernels for first-generation Tenstorrent hardware
*   Maintaining compatibility with legacy Grayskull deployments
*   Working with codebases that need to support multiple architecture generations

### Cross-Architecture Compatibility

Code written using the common SFPI interface (`include/sfpi.h`) compiles for all architectures including Grayskull. Platform-specific optimizations are only needed when:

*   Leveraging architecture-specific features (e.g., Blackhole's `sfparecip` or `sfpmul24`)
*   Optimizing for specific hardware characteristics
*   Working around architecture-specific limitations

**Sources:** Diagram 2 (User Code Level), Diagram 5 (Common SFPI Interface)

### Build Configuration

To build for Grayskull, configure the toolchain with:

```
--with-arch=rv32im...xtttensixgs
```

This sets the target architecture, causing the compiler to define `__riscv_xtttensixgs` and select Grayskull-specific implementations.

**Sources:** Diagram 3 (Configuration Phase), section 5.2 for build configuration details
