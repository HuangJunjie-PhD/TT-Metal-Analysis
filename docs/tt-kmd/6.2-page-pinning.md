## Purpose and Scope

This document describes the page pinning subsystem in the Tenstorrent kernel driver, which enables user-space applications to pin virtual memory pages into physical memory for direct DMA access. Page pinning ensures that user-allocated memory remains resident and physically contiguous (or IOMMU-mapped to contiguous IOVA space) for the duration needed by hardware operations.

For kernel-allocated DMA buffers, see [DMA Buffer Allocation](https://deepwiki.com/tenstorrent/tt-kmd/6.1-dma-buffer-allocation). For address translation mechanisms that allow the device to access pinned memory, see [IATU and Address Translation](https://deepwiki.com/tenstorrent/tt-kmd/6.4-iatu-and-address-translation).

## Overview

Page pinning allows user-space applications to use their own memory allocations for DMA operations rather than relying on kernel-allocated DMA buffers. The driver pins the specified pages into physical memory, optionally maps them through the IOMMU, and can configure an outbound IATU region to make them accessible via the device's Network-on-Chip (NOC).

Key capabilities:

*   Pin up to 4GB of user memory per operation [memory.c 240](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L240-L240)
*   Support for both IOMMU-enabled and IOMMU-disabled systems [memory.c 652-720](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L652-L720)
*   Automatic IOVA mapping when IOMMU is active [memory.c 652-680](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L652-L680)
*   Optional NOC address mapping via outbound IATU [memory.c 159-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L159-L187)
*   Physical contiguity verification [memory.c 702-710](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L702-L710)
*   Per-file-descriptor resource tracking via `priv->pinnings`[chardev_private.h 59](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L59-L59)

## IOCTL Interface

### PIN_PAGES

The `TENSTORRENT_IOCTL_PIN_PAGES` command pins user-space pages for DMA access [memory.c 757-791](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L757-L791)

`struct tenstorrent_pin_pages_in {    u32 output_size_bytes;    // Size of output structure    u32 flags;                // Control flags    u64 virtual_address;      // User virtual address (page-aligned)    u64 size;                 // Size in bytes (page-aligned)}; struct tenstorrent_pin_pages_out_extended {    u64 physical_address;     // Physical address or IOVA    u64 noc_address;          // NOC address (if NOC_DMA flag set)};`
**Flags:**[ioctl.h 185-188](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L185-L188)

| Flag | Value | Description |
| --- | --- | --- |
| `TENSTORRENT_PIN_PAGES_CONTIGUOUS` | 1 | Application attests pages are physically contiguous (unused) |
| `TENSTORRENT_PIN_PAGES_NOC_DMA` | 2 | Allocate outbound IATU region for NOC access |
| `TENSTORRENT_PIN_PAGES_NOC_TOP_DOWN` | 4 | Allocate IATU region from top of address space (vs. bottom-up) |

### UNPIN_PAGES

The `TENSTORRENT_IOCTL_UNPIN_PAGES` command unpins previously pinned pages [memory.c 793-820](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L793-L820)

`struct tenstorrent_unpin_pages_in {    u64 virtual_address;      // Original VA used to pin (not current VA if remapped)    u64 size;                 // Original size in bytes    u64 reserved;};`
**Note:** Partial unpinning is not supported. The virtual address and size must exactly match a previous PIN_PAGES operation [memory.c 806-810](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L806-L810)

**Sources:**[ioctl.h 167-206](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L167-L206)[memory.c 575-820](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L575-L820)

## Data Structures

### pinned_page_range

The driver tracks each pinned memory range with a `pinned_page_range` structure [memory.h 21-33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L21-L33):

`struct pinned_page_range {    struct list_head list;           // Linked list node    unsigned long page_count;        // Number of pinned pages    struct page **pages;             // Array of page pointers (vmalloc'd)    struct sg_table dma_mapping;     // Scatter-gather table for IOMMU    u64 virtual_address;             // Original virtual address    int outbound_iatu_region;        // IATU region index (-1 if none)    bool read_only;                  // IOMMU forbids device writes};`
These structures are tracked per-file-descriptor in `chardev_private.pinnings`[chardev_private.h 59](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L59-L59) and cleaned up automatically on file descriptor close [memory.c 1318-1348](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1318-L1348)

**Sources:**[memory.h 21-33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L21-L33)[chardev_private.h 59](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L59-L59)

## Kernel Version Compatibility

The driver includes compatibility shims for multiple kernel versions, as the page pinning API has evolved significantly over time:

### pin_user_pages_fast_longterm

**Key compatibility issues:**

| Kernel Version | API Used | vmas Array | Notes |
| --- | --- | --- | --- |
| >= 5.11.0 | `pin_user_pages_fast()` | Not needed | Kernel handles vmas allocation [memory.c 192-194](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L192-L194) |
| 5.6.0 - 5.10.x | `pin_user_pages()` | `kvmalloc_array` | Must allocate vmas to avoid contiguous memory allocation failure [memory.c 203-210](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L203-L210) |
| < 5.6.0 | `get_user_pages()` | `kvmalloc_array` | Older API, same vmas allocation requirement [memory.c 220-227](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L220-L227) |

**Rationale:** The vmas array allocation is necessary because older kernels would allocate it as physically contiguous memory, which frequently fails for large pinnings. By allocating it ourselves with `kvmalloc_array`, we allow virtually-contiguous allocation [memory.c 198-200](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L198-L200)

**Sources:**[memory.c 189-229](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L189-L229)

Similar compatibility shims exist for unpinning:

## IOMMU Handling

When an IOMMU is active, the driver must map pinned pages through the IOMMU to obtain I/O Virtual Addresses (IOVAs):

**IOMMU Detection:**

The driver uses `is_iommu_translated()` to check if an IOMMU is active for the device [memory.c 557-561](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L557-L561):

`bool is_iommu_translated(struct device *dev) {    struct iommu_domain *domain = iommu_get_domain_for_dev(dev);    return domain && domain->type != IOMMU_DOMAIN_IDENTITY;}`
**Sources:**[memory.c 557-561](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L557-L561)[memory.c 652-720](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L652-L720)

## Scatter-Gather Table Allocation

When IOMMU is active, the driver must create a scatter-gather table. For large pinnings, standard allocation may fail due to contiguous memory requirements. The driver uses `alloc_chained_sgt_for_pages()` to create scatter-gather tables that can be allocated using single-page allocations and scatterlist chaining for unlimited scaling [sg_helpers.c 15-17](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.c#L15-L17)

**Chaining Mechanism:**

*   Reserves the last entry of every page for the chain [sg_helpers.c 6-10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.c#L6-L10)
*   Uses `sg_chain()` to connect pages of scatterlists [sg_helpers.c 40](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.c#L40-L40)
*   Uses `sg_mark_end()` on the final valid entry [sg_helpers.c 66](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.c#L66-L66)

**Sources:**[sg_helpers.c 1-118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.c#L1-L118)[sg_helpers.h 36-37](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/sg_helpers.h#L36-L37)

## Outbound IATU Integration

When the `TENSTORRENT_PIN_PAGES_NOC_DMA` flag is set, the driver allocates an outbound IATU region via `setup_noc_dma()`[memory.c 159-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L159-L187)

The `setup_noc_dma` function calculates the final address by adding the device-specific `noc_pcie_offset` to the allocated IATU base [memory.c 183](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L183-L183)

**Sources:**[memory.c 123-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L123-L187)[memory.c 54-120](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L54-L120)

## Lifecycle Management

### Cleanup on File Descriptor Close

When a file descriptor is closed, `tenstorrent_memory_cleanup()` automatically unpins all pages [memory.c 1318-1348](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1318-L1348):

`void tenstorrent_memory_cleanup(struct chardev_private *priv) {    // ...    list_for_each_entry_safe(pinning, tmp_pinning, &priv->pinnings, list) {        unpin_pinned_page_range(priv, pinning);    }}`
**Cleanup sequence (`unpin_pinned_page_range`) [memory.c 333-356](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L333-L356):**

1.   Release outbound IATU region (if index >= 0) [memory.c 338-341](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L338-L341)
2.   Unmap and free scatter-gather table [memory.c 343-346](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L343-L346)
3.   Unpin pages with `unpin_user_pages_dirty_lock(pinning->pages, pinning->page_count, true)`[memory.c 348](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L348-L348)
4.   Free the `pages` array via `vfree()`[memory.c 350](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L350-L350)
5.   Free the `pinned_page_range` structure [memory.c 355](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L355-L355)

### Duplicate Prevention

The driver prevents duplicate pinnings of the same virtual address range [memory.c 614-623](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L614-L623):

`list_for_each_entry(pinning, &priv->pinnings, list) {    if (pinning->virtual_address == in.virtual_address &&        pinning->page_count == PAGE_ALIGN(in.size) >> PAGE_SHIFT) {        return -EEXIST;    }}`
**Sources:**[memory.c 333-356](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L333-L356)[memory.c 1318-1348](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1318-L1348)[memory.c 614-623](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L614-L623)

## Limitations and Constraints

### Size and Alignment

*   **Max Buffer Size:** 256MB by default (`MAX_DMA_BUF_SIZE_LOG2` = 28) [memory.h 10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L10-L10)[memory.c 240](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L240-L240)
*   **Alignment:** Both `virtual_address` and `size` must be page-aligned [memory.c 587-591](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L587-L591)

### Kernel Version Specific Limits

On kernels <= 5.4.0, pinnings larger than 1GB are blocked to avoid soft lockups during process exit when IOMMU is active [memory.c 563-573](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L563-L573)

`static bool is_pin_pages_size_safe(u64 size) {#if LINUX_VERSION_CODE <= KERNEL_VERSION(5, 4, 0)    return size <= 1 << 30;  // 1GB limit#else    return true;#endif}`
**Sources:**[memory.c 563-573](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L563-L573)[memory.h 10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L10-L10)
