## Purpose and Scope

This document describes the automated testing system in RiESCUE's CI/CD pipeline, focusing on how tests are discovered, executed, and validated in continuous integration. For information about writing and organizing tests locally, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/riescue/5.1-contributing-guidelines). For the complete CI/CD workflow including linting and deployment, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline).

The automated testing system ensures that all code changes pass unit and integration tests before merge, running in a clean containerized environment that mirrors the production setup. Tests run automatically on every push to non-main branches.

## Test Job Overview

The automated testing workflow is defined in the `smoke` GitHub Actions workflow, which executes multiple quality gates in sequence.

### Smoke Workflow Test Job Architecture

**Sources:**[.github/workflows/smoke.yml 35-73](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L35-L73)

### Test Job Execution Flow

The test job (`test`) runs only when:

1.   The branch is **not**`main` (checked via `if: github.ref != 'refs/heads/main'`)
2.   The `check-spdx-headers` job passes
3.   The `lint` job passes

This conditional execution prevents redundant test runs on the main branch, where tests have already been validated in pull requests.

**Sources:**[.github/workflows/smoke.yml 35-39](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L35-L39)

## Container Environment

### Base Container Configuration

The test job executes inside a Rocky Linux 9.1 container, specified in the workflow:

`container: rockylinux:9.1`
This ensures tests run in a consistent, reproducible environment matching the Singularity container used for development. The Rocky Linux base provides the necessary system libraries and package management.

**Sources:**[.github/workflows/smoke.yml 37](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L37-L37)

## Dependency Installation

The test environment requires multiple dependency layers installed in a specific order.

### System Dependencies

The first step installs system-level dependencies via `infra/build_base_deps.sh`, which uses `yum` to install packages required for building C++ projects and Python development:

**Key package categories:**

*   **Build toolchain:**`gcc-toolset-12`, `make`, `which`
*   **Development headers:**`openssl-devel`, `zlib-devel`, `libffi-devel`, `python3-devel`
*   **Boost libraries:**`boost`, `boost-static`, `boost-devel`, `boost-program-options`
*   **Python tools:**`python3-pip`, `python-sphinx`
*   **Utilities:**`bc`, `perl`, `dtc`, `wget`, `git`, `sudo`, `zstd`

These dependencies are needed to compile Whisper and Spike from source.

**Sources:**[.github/workflows/smoke.yml 43-45](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L43-L45)[infra/build_base_deps.sh 1-39](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_base_deps.sh#L1-L39)

### RISC-V Toolchain Installation

The CI workflow downloads a pre-built RISC-V GNU toolchain rather than building from source (which would take too long):

| Step | Action | Environment Configuration |
| --- | --- | --- |
| Create directory | `mkdir -p $HOME/RV_TOOLCHAIN` | Prepare installation location |
| Download | `wget riscv64-elf-ubuntu-22.04-gcc-nightly-2025.05.30-nightly.tar.xz` | Fetch pre-built toolchain |
| Extract | `tar -xJf ... -C $HOME/RV_TOOLCHAIN` | Unpack toolchain |
| Set `RV_TOOLCHAIN` | `$HOME/RV_TOOLCHAIN/riscv/bin` | Export to `$GITHUB_ENV` |
| Update PATH | Append to `$GITHUB_PATH` | Make binaries available |

The specific toolchain version (`2025.05.30 nightly`) is pinned for reproducibility. The `RV_TOOLCHAIN` environment variable is used throughout RiESCUE to locate the compiler and related tools.

**Sources:**[.github/workflows/smoke.yml 48-54](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L48-L54)

### Python Package Installation

Python dependencies are installed via pip in editable mode:

`python3 -m pip install wheelpython3 -m pip install -e .`
The `-e .` flag installs RiESCUE in editable/development mode, which:

*   Installs dependencies specified in `pyproject.toml`
*   Creates entry points for `riescued`, `riescuec`, `ctk` commands
*   Links the package to the source directory for immediate code changes

This ensures all Python test modules can import from `riescue.dtest_framework`, `riescue.compliance`, and `riescue.lib`.

**Sources:**[.github/workflows/smoke.yml 56-59](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L56-L59)

### Simulator Installation

Both instruction set simulators are built from source during test setup:

#### Whisper Installation Flow

The Whisper simulator is built via the `build_whisper.sh` script with `NO_SINGULARITY=1` to skip container-specific logic:

The resulting binary is installed to `/usr/bin/whisper`, and the `WHISPER_PATH` environment variable is set for tests to locate it.

## Test Discovery and Execution

### Test Discovery Mechanism

The CI workflow uses Python's `unittest` discovery mechanism to automatically find and run all tests:

`python3 -m unittest discover -s tests/ -p "*_test.py"`
**Discovery parameters:**

*   `-s tests/`: Start directory for test discovery
*   `-p "*_test.py"`: Pattern matching test file names

The unittest framework:

1.   Recursively searches the `tests/` directory
2.   Imports all Python files matching `*_test.py`
3.   Identifies all classes inheriting from `unittest.TestCase`
4.   Collects all methods starting with `test_`
5.   Executes each test method as an independent test case

**Sources:**[.github/workflows/smoke.yml 72-73](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L72-L73)

### Test Execution Environment

During test execution, the following resources are available:

| Resource | Location | Purpose |
| --- | --- | --- |
| RISC-V Compiler | `$RV_TOOLCHAIN/riscv64-unknown-elf-gcc` | Compile test assembly to ELF |
| Whisper Simulator | `/usr/bin/whisper` | Execute and trace RISC-V instructions |
| Spike Simulator | `/usr/local/bin/spike` | Reference ISS for comparison |
| tt_spike | `/usr/local/bin/tt_spike` | Tenstorrent-specific Spike variant |
| RiESCUE Package | Editable install | Import framework classes |

Tests can use the `Toolchain` class to automatically discover these tools via the `RV_TOOLCHAIN` and `WHISPER_PATH` environment variables.

### Test Failure Handling

The GitHub Actions workflow will fail the entire job if:

*   Any individual test case fails (assertion error)
*   Any test raises an unhandled exception
*   The unittest runner exits with a non-zero status code

Failed tests prevent the PR from being merged, ensuring code quality gates are enforced.

## Integration with CI Pipeline

### Workflow Dependencies

The test job is part of a dependency chain in the smoke workflow:

1.   **check-spdx-headers** must pass first
2.   **lint** must pass after SPDX check
3.   **test** runs only on non-main branches after lint passes
4.   **deploy-docs** runs only on main branch after lint passes (skipping tests since they ran in the PR)

This structure ensures:

*   All PRs are tested before merge
*   Main branch only runs documentation deployment (tests already validated)
*   No redundant test execution on main

**Sources:**[.github/workflows/smoke.yml 7-78](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L7-L78)

### Test Artifacts and Reporting

The workflow does not currently produce test artifacts or detailed reports. Test results are visible in the GitHub Actions logs, where:

*   Each test case shows as passed or failed
*   Failed assertions display the error message and traceback
*   The job summary shows the total test count and pass/fail status

Developers can view these logs in the "Actions" tab of the GitHub repository for each commit.

## Local Test Execution

While this page focuses on automated CI testing, developers can run the same test suite locally using the Singularity container:

`# Inside the containerpython3 -m unittest discover -v -s tests -p "*_test.py"`
The `-v` flag adds verbose output for debugging. For more details on local testing practices, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/riescue/5.1-contributing-guidelines).

**Sources:**[.github/CONTRIBUTING.md 65-71](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L65-L71)

## Summary

The automated testing system provides:

*   **Reproducible environment**: Rocky Linux 9.1 container with pinned dependencies
*   **Complete toolchain**: RISC-V compiler, Whisper, and Spike installed from source
*   **Automatic discovery**: unittest finds all test cases in `tests/`
*   **Quality gates**: Tests must pass before merge
*   **Efficient execution**: Only runs on non-main branches to avoid redundancy

This infrastructure ensures that all code changes are validated against the full test suite in an environment that closely mirrors production deployment.
