## L1 Memory Architecture

The L1 scratchpad RAM in each Tensix Tile is organized as 16 banks of 91.5 KiB each (1464 KiB total). Each bank supports one 128-bit read **or** one 128-bit write per cycle. Access to these banks is managed through 16 access ports, where any port can access any bank.

**Critical concept:** When multiple ports attempt to access the same bank in the same cycle, a **bank conflict** occurs. Only one port will succeed; others must wait. This is the primary performance bottleneck for L1 access.

**Sources:**[WormholeB0/TensixTile/L1.md 1-19](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L1-L19)

### Port Assignment Details

Understanding which clients share which ports is essential for avoiding contention:

| Port | Primary Client(s) | Notes |
| --- | --- | --- |
| 0 | RISC-V cores | Shared by all 5 RISC-V cores via mux |
| 1 | Unpacker 0, Mover read | Shared via mux |
| 2-4 | Unpacker 0, Unpacker 1 | Shared between both unpackers |
| 5 | Unpacker 1, Mover write | Shared via mux |
| 6 | Packer 0 (write), Packer 0 read (L1-to-L1 mode) | Independent connections |
| 7 | Packer 1 | Dedicated |
| 8 | Packer 2 | Dedicated |
| 9 | Packer 3 | Dedicated |
| 10-11 | NoC 0 read | Independent connections |
| 12-13 | NoC 0 write | Independent connections |
| 14-15 | NoC 1 (read/write) | Similar to NoC 0 |

**Key observation:** Ports 2-4 are shared between both unpackers. When both unpackers operate simultaneously, they will interfere with each other on these ports, potentially reducing effective bandwidth.

**Sources:**[WormholeB0/TensixTile/L1.md 16-19](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L16-L19)

## Addressing Mechanisms: ADCs

ADCs (Auto-incrementing Address Counters) are the primary mechanism for generating addresses for Unpackers and Packers. Each ADC contains four dimensional counters: **X**, **Y**, **Z**, and **W**.

### ADC Organization

**Critical distinction:** Each unpacker has **two channels** (Channel[0] and Channel[1]), serving different purposes:

*   **Channel[0]:** Used for input address calculation (where to read from L1)
*   **Channel[1]:** Used for output address calculation (where to write in SrcA/SrcB/Dst)

For Packers:

*   **Channel[0]:** Used for input address calculation (where to read from Dst or L1)
*   **Channel[1]:** Used for output address calculation (where to write to L1)

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 134-163](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L134-L163)[WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 1-102](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L1-L102)

## Unpacker Memory Access Optimization

### Speed Modes and Bandwidth

Each unpacker can operate at three speeds, fetching from L1 at different rates:

| Speed Mode | L1 Bandwidth | Cycles for 16 bytes |
| --- | --- | --- |
| **x1** | 16 bytes/cycle | 1 cycle |
| **x2** | 32 bytes/cycle | 0.5 cycles |
| **x4** | 64 bytes/cycle | 0.25 cycles |

The speed mode is controlled by `ConfigState.THCON_SEC[WhichUnpacker].Throttle_mode`:

*   `0` = x1 speed
*   `1` = x2 speed
*   `2` = x4 speed

However, certain configurations **force** specific speed modes regardless of `Throttle_mode`:

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 572-598](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L572-L598)

### Two-Unpacker Interference

When both unpackers operate simultaneously, they share ports 2-4, causing interference:

| Unpacker 0 wants | Unpacker 1 wants | Result |
| --- | --- | --- |
| x1 | x1 | Both get x1 (no interference) |
| x1 | x2 | U0 gets x1, U1 gets x2 |
| x1 | x4 | U0 gets x1, U1 gets x4 |
| x2 | x1 | U0 gets x2, U1 gets x1 |
| x2 | x2 | **U0 gets x1, U1 gets x1** ⚠️ |
| x2 | x4 | **U0 gets x2, U1 gets x1** ⚠️ |
| x4 | x1 | U0 gets x4, U1 gets x1 |
| x4 | x2 | **U0 gets x1, U1 gets x2** ⚠️ |
| x4 | x4 | **U0 gets x2, U1 gets x2** ⚠️ |

**Optimization strategy:** When both unpackers must run simultaneously, configure them so that one wants x1 and the other wants higher speed. Avoid both wanting x2 or both wanting x4.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 581-587](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L581-L587)

### Discontiguous Input Rows

The `DiscontiguousInputRows` mode (controlled by `THCON_SEC[WhichUnpacker].REG2_Tileize_mode`) allows unpacking from non-contiguous memory:

**Important constraints:**

*   This mode always forces x4 speed (highest bandwidth)
*   It is incompatible with upsampling (`UpsampleZeroes > 0`)
*   It is incompatible with compressed data (`!IsUncompressed`)
*   Input address must be 16-byte aligned

**Use case:** Accessing tiles stored in interleaved or strided layouts in L1.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 204-277](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L204-L277)

### Compression and Decompression

When unpacking compressed data (delta-encoded with zero-run-length encoding):

1.   **Always operates at x1 speed** (one of the forced constraints)
2.   Requires a row-start table in L1 before the data
3.   Uses a cache for row-start entries (4 entries per unpacker)

**Cache management:** If you're rapidly reusing memory containing compressed data, flush the row-start cache using `UNPACR` with flush mode to avoid stale entries.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 109-192](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L109-L192)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_FlushCache.md 1-32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_FlushCache.md?plain=1#L1-L32)

### Multi-Context Mode

Multi-context mode allows rapid switching between up to 8 different configurations without reconfiguring:

**Use cases:**

*   Processing multiple input tensors with different formats/layouts
*   Reducing configuration overhead when alternating between patterns

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 43-88](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L43-L88)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_IncrementContextCounter.md 1-42](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_IncrementContextCounter.md?plain=1#L1-L42)

## Packer Memory Access Optimization

### Packer Bandwidth and Alignment

Each packer can theoretically write one 128-bit (16-byte) value to L1 per cycle, but output is **always buffered and aligned to 16-byte boundaries**:

**Key implications:**

1.   Output addresses are calculated as `(Addr & 0x1ffff) << 4` - only bits that are multiples of 16 matter
2.   Buffers persist between consecutive pack instructions
3.   Use `Last` or `Flush` flags to force buffer flush (padded with zeros if needed)

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 106-115](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L106-L115)

### Packer 0 L1-to-L1 Mode

Packer 0 uniquely supports reading from L1 (instead of Dst):

**Use case:** L1-to-L1 copy operations with format conversion or compression.

**Important:** When packer 0 performs L1-to-L1 operations, the other three packers should remain idle to avoid port contention (packer 0's read uses port 6, packer 2's write also uses port 8, but in practice port 2 contention should be avoided).

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 45-52](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L45-L52)[WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)

### Atomic Accumulate Operations

Packers can perform near-memory accumulation into L1 (4×FP32, 4×INT32, 8×FP16, or 8×BF16):

*   **Atomic mode:** Blocks port and bank for 5 cycles per operation
*   **Non-atomic mode:** Only blocks for 2 cycles, but requires software synchronization

These operations are configured via packer settings and are useful for reduction operations.

**Sources:**[WormholeB0/TensixTile/L1.md 13-14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L13-L14)

## Bank Conflict Avoidance Strategies

### Understanding Bank Selection

The bank accessed for any L1 address is determined by specific address bits. For 128-bit (16-byte) accesses, the bank is selected by bits that determine which 16-byte chunk within L1 is being accessed.

**General principle:** Addresses that differ by `16 × 16 = 256` bytes will access different banks (assuming sequential bank assignment).

### Scheduling to Avoid Conflicts

**Practical techniques:**

1.   **Offset input regions by non-power-of-2 amounts** to ensure different clients access different banks
2.   **Serialize operations when possible** - if unpackers don't need to run simultaneously, run them sequentially
3.   **Use NoC for large transfers** rather than Mover when possible, as NoC has more L1 ports (4 per NoC vs 2 for Mover)
4.   **Avoid RISC-V L1 access** - delegate to specialized units (Mover, NoC, Unpackers/Packers) whenever possible

**Sources:**[WormholeB0/TensixTile/L1.md 5-6](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L5-L6)

### Round-Robin Arbitration

When conflicts occur, hardware uses round-robin arbitration at each mux level:

```
// From L1.md lines 17-18
Each mux can be modelled as performing round-robin allocation in the case of conflicts.
Where an access port has multiple levels of muxing, there is independent conflict 
resolution and round-robin allocation at each mux.
```

This means sustained conflicts will distribute bandwidth fairly, but **any conflict still reduces total throughput**.

**Sources:**[WormholeB0/TensixTile/L1.md 17-18](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L17-L18)

## Performance Guidelines and Bandwidth Budgets

### Bandwidth by Client Type

| Client | Peak Read Bandwidth | Peak Write Bandwidth | Notes |
| --- | --- | --- | --- |
| **RISC-V cores** | ~18.3 bits/cycle (sustained) 4 bits/cycle (dependent loads) | 6.4 bits/cycle | Always use narrow accesses (5× penalty for writes) |
| **Mover** | ~93.1 bits/cycle | ~93.1 bits/cycle | Measured performance (8 reads + 8 writes per 11 cycles) |
| **Unpacker 0** | 512 bits/cycle (x4 mode) | N/A | Can drop to 128 or 256 bits/cycle based on mode |
| **Unpacker 1** | 512 bits/cycle (x4 mode) | N/A | Shares ports 2-4 with Unpacker 0 |
| **Each Packer** | 128 bits/cycle (Packer 0 only, L1-to-L1) | 128 bits/cycle 25.6 bits/cycle (atomic) | Atomic accumulate is 5× slower |
| **Each NoC** | 256 bits/cycle | 256 bits/cycle | Theoretical; 4 ports per NoC (2 read, 2 write) |
| **Scalar Unit** | 42.7 bits/cycle (128-bit reads) | 42.7 bits/cycle (128-bit writes) 6.4 bits/cycle (32-bit writes) | Issues one request per 3 cycles |

**Sources:**[WormholeB0/TensixTile/L1.md 22-54](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L22-L54)

### Client Selection Guidelines

**Key recommendations:**

1.   **Never use RISC-V for bulk transfers** - 6.4 bits/cycle write bandwidth is 20× slower than Unpacker/Packer
2.   **Prefer Mover for large L1-to-L1 copies** without format conversion (93 bits/cycle bidirectional)
3.   **Use NoC for inter-tile transfers** even on same tile (256 bits/cycle, can write to same tile without network transit)
4.   **Use Unpackers/Packers for data requiring format conversion** - they integrate decompression/compression
5.   **Use Scalar Unit for atomics** when NoC is unavailable (RISC-V cores cannot perform atomics directly)

**Sources:**[WormholeB0/TensixTile/L1.md 24-29](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L24-L29)
