This page documents how vLLM implements streaming response generation and tool calling in the OpenAI-compatible API server. Streaming allows tokens to be sent to the client as they are generated using Server-Sent Events (SSE), while tool calling enables models to invoke external functions during generation. Both features support chat completions (`/v1/chat/completions`) and text completions (`/v1/completions`) endpoints.

## Overview

vLLM provides two primary features for enhanced interaction with language models:

1.   **Streaming**: Tokens are delivered incrementally via SSE as they are generated, reducing perceived latency
2.   **Tool Calling**: Models can generate structured function calls that the client can execute, with automatic parsing via `ToolParserManager`

These features can be used independently or combined - for example, streaming tool calls as they are generated. The implementation also integrates with `ReasoningParserManager` for models that generate reasoning steps alongside their responses.

## Streaming Response Generation

### SSE Protocol

vLLM uses Server-Sent Events (SSE) to stream responses. Each chunk is formatted as `data: {json}\n\n`, with a final `data: [DONE]\n\n` message to signal completion. The response has `Content-Type: text/event-stream`.

### Chat Completion Streaming

The `chat_completion_stream_generator()` method in `OpenAIServingChat` implements streaming for `/v1/chat/completions`. It processes each `RequestOutput` from the engine and converts it to SSE chunks.

**Streaming Steps:**

1.   First chunk: Send role (`delta: {"role": "assistant"}`)
2.   Token chunks: Send each generated token as `delta: {"content": "..."}`
3.   Tool call chunks: If tool calling is enabled, stream tool calls (see Tool Calling section)
4.   Final chunk: If `include_usage=true`, send usage statistics
5.   Completion: Send `data: [DONE]`

## Tool Calling System

Tool calling allows models to generate structured function calls that clients can execute. vLLM provides automatic tool call parsing through the `ToolParserManager`.

### Tool Parser Architecture

Title: Tool Parser System Components

The `ToolParserManager` maintains a registry of parsers, each implementing the `ToolParser` interface. Parsers are selected based on the `--tool-call-parser` CLI argument or registered via `--tool-parser-plugin`.

*   [vllm/entrypoints/openai/serving_chat.py 115-128](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/openai/serving_chat.py#L115-L128)

### ToolParserManager

`ToolParserManager` is a registry that stores and retrieves tool parsers by name. It's initialized at server startup:

**Built-in Parsers:**

*   `hermes`: For Nous-Hermes models
*   `mistral`: For Mistral models with native tool calling
*   `granite-20b-fc`: For IBM Granite models
*   `pythonic`: For Llama-3.2 and similar models
*   `internlm2`: For InternLM2 models

Each parser implements two key methods:

*   `extract_tool_calls()`: Parse model output to extract tool calls
*   `extract_tool_calls_streaming()`: Parse streaming output incrementally

### Reasoning in Streaming

When a reasoning parser is enabled, the `chat_completion_stream_generator()` processes reasoning content separately:

1.   Detect reasoning start tokens
2.   Stream reasoning content (if `include_reasoning=true`)
3.   Detect reasoning end and transition to assistant response
4.   Stream final response separately

Title: Reasoning Parser Flow in Streaming

The reasoning parser is invoked on each token to determine its type, allowing proper streaming of both reasoning and response content.

## Advanced Topics

### Non-Streamable Requests

Some requests with `stream=true` return a non-streaming response:

1.   **Beam Search**: When `use_beam_search=true`, generation completes before returning results
2.   **Multiple Completions with best_of**: When `n != best_of`, all candidates are evaluated before selection

In these cases, vLLM generates the complete response, then returns it as a single SSE chunk:

```
data: {full_response}\n\n
data: [DONE]\n\n
```

### Error Handling

Errors during streaming are returned as SSE chunks with error information:

**Error Response Format:**

```
data: {"error": {"message": "Error occurred", "type": "InternalError", "code": 500}}\n\n
data: [DONE]\n\n
```

The `create_streaming_error_response()` method formats errors for SSE delivery, ensuring clients can handle errors even in streaming mode.

## Error Handling in Streaming

Errors that occur during streaming are sent as JSON-formatted SSE chunks with an `error` field:

```
data: {"error": {"message": "Error message", "type": "ErrorType", "code": 400}}\n\n
data: [DONE]\n\n
```

This allows clients to properly handle errors even in streaming mode.

## Conclusion

vLLM's streaming implementation allows for efficient token-by-token delivery of model outputs to clients, improving user experience by providing immediate feedback. The implementation follows the SSE protocol and is compatible with the OpenAI API format, making it easy to integrate with existing clients.

This streaming capability applies to both chat completions and text completions, with special handling for tool calls and reasoning steps. The modular design allows for flexible configuration through request parameters and supports various use cases.
