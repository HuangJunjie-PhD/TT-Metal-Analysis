## Purpose and Scope

This document describes Pyright, the static type checker used in the RiESCUE project to enforce type safety and catch errors before runtime. This page covers Pyright's configuration, usage patterns, IDE integration, and the project's type checking strategy. For information about other code quality tools, see [Black Code Formatter](https://deepwiki.com/tenstorrent/riescue/5.2.1-black-code-formatter), [Flake8 Linter](https://deepwiki.com/tenstorrent/riescue/5.2.2-flake8-linter), and [Pre-commit Hooks](https://deepwiki.com/tenstorrent/riescue/5.2.4-pre-commit-hooks). For the overall CI/CD pipeline that runs Pyright, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline).

## Overview

Pyright is a static type checker for Python that analyzes code without executing it. RiESCUE uses Pyright to:

*   Catch type errors, undefined variables, and incorrect type usage before runtime
*   Enforce statically-typed code style for improved readability
*   Enable better IDE support for code navigation and autocomplete
*   Reduce debugging time by catching errors during development

The project uses the `pyright[nodejs]` package, which bundles Node.js and Pyright together, eliminating external dependencies. Pyright was specifically chosen for its integration with the Pylance VS Code extension, making it the default type checker for many Python developers.

## Configuration

Pyright reads its configuration from the `[tool.pyright]` section in `pyproject.toml`. This approach centralizes all Python tool configurations in a single file and is compatible with VS Code's default behavior.

### Configuration Structure

**Sources:**[pyproject.toml 51-80](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L51-L80)

### Key Configuration Options

| Option | Value | Purpose |
| --- | --- | --- |
| `typeCheckingMode` | `"standard"` | Enables standard type checking rules (not as strict as `"strict"` mode) |
| `pythonVersion` | `"3.9"` | Targets Python 3.9 compatibility |
| `pythonPlatform` | `"Linux"` | Assumes Linux platform for platform-specific type stubs |
| `reportMissingTypeStubs` | `false` | Suppresses warnings for external packages without type stubs |
| `extraPaths` | `["riescue"]` | Adds `riescue` directory to Python import resolution path |

**Sources:**[pyproject.toml 67-80](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L67-L80)

### Type Checking Mode

The project currently uses `typeCheckingMode = "standard"`, which provides a balance between strictness and pragmatism. A comment in the configuration notes plans to eventually migrate to `"strict"` mode, potentially on a file-by-file basis using Pyright's per-file configuration feature.

**Sources:**[pyproject.toml 67-70](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L67-L70)

## Included and Excluded Paths

### Included Paths

Pyright checks the following directories and files:

| Path Pattern | Description |
| --- | --- |
| `riescue/lib` | Core library utilities and toolchain abstractions |
| `riescue/*.py` | Top-level entry point scripts (`riescued.py`, `riescuec.py`, `ctk.py`) |
| `riescue/dtest_framework` | RiescueD directed testing framework |
| `tests/dtest_framework` | Unit tests for RiescueD |
| `riescue/compliance/*.py` | Top-level RiescueC compliance files |
| `riescue/compliance/config` | Compliance configuration modules |
| `riescue/compliance/src` | Compliance source modules |
| `riescue/compliance/test_plan` | Test plan handling modules |

**Sources:**[pyproject.toml 52-65](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L52-L65)

### Excluded Paths

Two paths are explicitly excluded from type checking:

| Path | Reason |
| --- | --- |
| `tests/compliance` | Generated compliance test outputs, not source code |
| `riescue/dtest_framework/generator/generator.py` | Marked with `#FIXME` for future type safety improvements |

The `generator.py` exclusion indicates technical debt that will be addressed in future refactoring.

**Sources:**[pyproject.toml 74-77](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L74-L77)

## Usage

### Running Pyright Locally

Execute Pyright from within the container:

The `-p` flag specifies the configuration file to use. While VS Code defaults to `pyrightconfig.json`, RiESCUE uses `pyproject.toml` to maintain a single source of truth for all tool configurations.

**Sources:**[docs/lint.md 58-63](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L58-L63)

### Pyright Output

When type errors are found, Pyright outputs diagnostic messages in the format:

```
path/to/file.py:line:column - error: Error message
```

A successful run with no errors exits with status code 0. Any type errors result in a non-zero exit code, which causes CI pipeline failures.

## IDE Integration

### VS Code and Pylance

Pyright is the underlying type checker for Microsoft's [Pylance](https://marketplace.visualstudio.com/items?itemName=ms-python.vscode-pylance) VS Code extension. Pylance provides:

*   Real-time type checking as code is written
*   Intelligent autocomplete based on type information
*   "Go to Definition" functionality that respects types
*   Inline error highlighting and hover documentation

To enable Pylance in VS Code:

1.   Install the Python extension (which includes Pylance)
2.   Ensure `pyproject.toml` is at the workspace root
3.   Pylance automatically reads the `[tool.pyright]` configuration

### Configuration Discovery

VS Code's Pylance extension automatically detects type checking configuration in this priority order:

1.   `pyrightconfig.json` (not used in RiESCUE)
2.   `[tool.pyright]` section in `pyproject.toml` (used by RiESCUE)

By using `pyproject.toml`, RiESCUE ensures consistency between local IDE checks and CI pipeline checks.

**Sources:**[docs/lint.md 50-59](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L50-L59)

## Type Checking Strategy

### Incremental Type Coverage

RiESCUE follows an incremental approach to type safety:

1.   **Core modules first**: The `lib` directory and entry point scripts have full type checking
2.   **Framework modules**: RiescueD framework is included with one known exception (`generator.py`)
3.   **Future expansion**: Plans to enable strict mode on a per-file basis as code matures

This strategy allows the project to benefit from type checking while gradually improving type coverage across the codebase.

### Rationale for Exclusions

*   `tests/compliance`: Contains generated test directories and outputs, not source code requiring type checking
*   `generator.py`: Contains complex code that requires refactoring for proper type annotations (marked for future work)

**Sources:**[pyproject.toml 74-77](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L74-L77)

## Benefits of Type Checking

### Runtime Error Prevention

Pyright catches common errors before execution:

*   **Undefined variables**: References to variables that don't exist
*   **Type mismatches**: Passing wrong types to functions
*   **Attribute errors**: Accessing non-existent attributes on objects
*   **Import errors**: Incorrect or missing imports

### Code Maintainability

Type annotations serve as inline documentation:

*   Function signatures clearly specify input and output types
*   IDEs can provide better refactoring support
*   New contributors can understand code interfaces more quickly

### Static Analysis

Pyright enables static analysis without execution:

*   No need to run tests to discover basic type errors
*   Faster feedback cycle during development
*   Reduces debugging time by catching errors early

**Sources:**[docs/lint.md 50-55](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L50-L55)

## Common Issues and Solutions

### Missing Type Stubs

External packages without type stubs may generate warnings. RiESCUE sets `reportMissingTypeStubs = false` to suppress these warnings for dependencies that lack complete type information.

**Sources:**[pyproject.toml 80](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L80-L80)

### Import Resolution

The `extraPaths = ["riescue"]` configuration ensures Pyright can resolve imports correctly when modules reference other parts of the package. This is necessary because Pyright performs static analysis without running code or installing packages.

**Sources:**[pyproject.toml 73](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L73-L73)

### Generator.py Exclusion

The file `riescue/dtest_framework/generator/generator.py` is currently excluded due to complexity. The FIXME comment indicates this is known technical debt. When working on this file, developers should be aware that type checking is disabled and should manually verify type correctness.

**Sources:**[pyproject.toml 76](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L76-L76)

## Summary

Pyright provides static type checking for RiESCUE's Python codebase with configuration centralized in `pyproject.toml`. The project uses standard mode type checking with plans for stricter checking in the future. Key components like the library utilities, entry points, and framework modules are type-checked, while generated outputs and one complex module are excluded. Integration with Pylance enables real-time type checking during development, and Pyright runs as a required check in the CI pipeline. This multi-layered approach catches type errors early, improving code quality and maintainability.
