# Hardware Abstraction Layer

Relevant source files
*   [tt_burnin/chip.py](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py)

The Hardware Abstraction Layer (HAL) provides a unified interface for interacting with different Tenstorrent chip architectures while handling the specific implementation details of each hardware platform. This layer abstracts away architecture-specific differences in coordinate systems, telemetry collection, and hardware communication protocols, allowing higher-level components to work with any supported chip type through a common interface.

For information about TTX workload loading and execution, see [TTX Workload System](https://deepwiki.com/tenstorrent/tt-burnin/2.3-ttx-workload-system). For details about hardware management utilities, see [Hardware Management Utilities](https://deepwiki.com/tenstorrent/tt-burnin/2.4-hardware-management-utilities).

## Architecture Overview

The HAL is built around an abstract base class `TTChip` with concrete implementations for each supported Tenstorrent architecture. The system uses the `pyluwen` library as the underlying hardware interface and provides additional abstraction layers specific to burn-in testing requirements.

## Hardware Communication Interfaces

The HAL provides standardized interfaces for different hardware communication protocols used by Tenstorrent chips.

### Communication Protocol Support

| Protocol | Purpose | Methods | Supported Operations |
| --- | --- | --- | --- |
| NOC (Network-on-Chip) | Inter-core communication | `noc_read32`, `noc_write32`, `noc_broadcast32` | Read/Write to specific cores, Broadcast to all cores |
| AXI | Memory-mapped register access | `axi_read32`, `axi_write32`, `axi_read`, `axi_write` | Register and memory access |
| SPI | Serial peripheral interface | `spi_read`, `spi_write` | Firmware and configuration access |
| ARC | Processor messaging | `arc_msg` | Command/response communication |

## Architecture-Specific Implementations

Each chip architecture has specific requirements for coordinate systems, core layouts, and hardware features.

### Grayskull (GsChip)

*   **Grid Size**: 13x12 grid with 12x10 Tensix cores
*   **Coordinate Mapping**: Physical to NOC coordinate translation via lookup tables
*   **Harvesting**: Row-based harvesting with bit manipulation for disabled cores
*   **Special Features**: Simplified coordinate system with "N/A" cluster coordinates

Sources: [tt_burnin/chip.py 317-364](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py#L317-L364)

### Wormhole (WhChip and RemoteWhChip)

*   **Grid Size**: 10x12 grid with 8x10 Tensix cores
*   **Coordinate Mapping**: Bidirectional NOC 0/1 coordinate translation
*   **Harvesting**: Row-based harvesting with physical-to-NOC coordinate mapping
*   **Remote Support**: `RemoteWhChip` overrides broadcast methods to iterate over individual cores
*   **Cluster Coordinates**: 4-tuple shelf/rack positioning via `coord()` method

Sources: [tt_burnin/chip.py 253-300](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py#L253-L300)[tt_burnin/chip.py 302-315](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py#L302-L315)

### Blackhole (BhChip)

*   **Grid Size**: 17x12 grid with 14x10 Tensix cores
*   **Coordinate Mapping**: Complex physical-to-NOC translation with coordinate flipping
*   **NOC Translation**: Runtime-configurable NOC translation mode affects core layout
*   **Column Harvesting**: Column-based harvesting via `enabled_tensix_columns` bitmask
*   **Advanced Features**: `noc_coord_flip()` and `phys_to_noc()` coordinate transformation

Sources: [tt_burnin/chip.py 173-251](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py#L173-L251)[tt_burnin/chip.py 208-240](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/chip.py#L208-L240)

### Harvesting and Core Discovery

Each architecture implements `get_tensix_locations()` to discover functional compute cores:

*   **Harvest Bit Retrieval**: Uses ARC message `0x57` to get harvesting information
*   **Physical-to-NOC Mapping**: Translates disabled physical rows/columns to NOC coordinates
*   **Core Filtering**: Returns only functional cores as coordinate sets
*   **Architecture-Specific Logic**: Each chip type handles harvesting differently (rows vs columns)
