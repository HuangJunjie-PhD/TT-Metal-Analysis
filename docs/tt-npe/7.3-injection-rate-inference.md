## Purpose and Scope

This document explains the automatic injection rate inference system in tt-npe. Injection rate inference automatically determines the maximum data injection rate for each transfer based on the source core type and device architecture. This feature simplifies workload creation by eliminating the need to manually specify injection rates for every transfer.

For information about creating workloads programmatically, see [Programmatic Workload Construction](https://deepwiki.com/tenstorrent/tt-npe/3.3-programmatic-workload-construction). For details on workload ingestion from traces, see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads).

## Injection Rate Fundamentals

The **injection rate** of a transfer represents the maximum bandwidth at which the source core can inject data into its Network Interface Unit (NIU). This rate is measured in GB/cycle and directly affects simulation accuracy:

*   **Physical Meaning**: Different core types (Tensix workers, DRAM, Ethernet) have different maximum injection capabilities based on their hardware design
*   **Simulation Impact**: The injection rate acts as a bandwidth ceiling that limits how quickly a transfer can complete, even if the NoC fabric has available capacity
*   **Default Value**: If not specified, transfers use a default injection rate of 28.1 GB/cycle (as defined in [npeWorkload.hpp 62](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npeWorkload.hpp#L62-L62))

The simulation engine uses injection rate in combination with congestion modeling to compute actual transfer bandwidth at each timestep.

### Inference Logic

The `inferInjectionRates` method iterates through all transfers in the workload and queries the device model for the appropriate injection rate based on each transfer's source coordinate:

**Injection Rate Inference Process**

The device model determines the injection rate based on:

*   **Source coordinate** (device_id, row, col)
*   **Core type** at that location (WORKER, DRAM, ETH)
*   **Architecture-specific parameters** (e.g., AI clock frequency for Blackhole)

## Device Model Integration

Each device model implementation provides injection rate information through its interface. The device model maintains lookup tables or computation logic for different core types:

| Core Type | Typical Injection Rate | Notes |
| --- | --- | --- |
| Tensix Worker | Architecture-dependent | Based on NIU design and clock frequency |
| DRAM | Architecture-dependent | DRAM interface bandwidth limits |
| Ethernet | Architecture-dependent | Fabric router capabilities |

Different device architectures have different injection rates. For example:

*   **Wormhole B0**: Standard injection rates per core type
*   **Blackhole**: May have different rates due to AI clock scaling

The device model abstraction (see [Device Model Interface](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface)) ensures that injection rate queries work uniformly across all supported architectures.

## Configuration Control

### The `infer_injection_rate_from_src` Flag

Injection rate inference is controlled by the `infer_injection_rate_from_src` boolean field in the `npeConfig` structure:

*   **Default behavior**: When `true` (typical default), the system automatically infers injection rates
*   **Manual control**: When `false`, the system uses injection rates specified in the workload

Example configuration in Python:

`cfg = npe.Config()cfg.infer_injection_rate_from_src = True  # Enable automatic inference`
### Preprocessing Order

The preprocessing pipeline applies transformations in this order:

1.   **Injection rate inference** (if enabled)
2.   **Workload schedule scaling** (if configured)
3.   **Local transfer removal** (if enabled)

This order ensures that injection rates are set before any other transformations that might depend on them.

## Manual Override Capability

### When to Use Manual Injection Rates

Manual injection rate specification is appropriate when:

*   **Benchmarking specific scenarios**: Testing with controlled injection rates
*   **Modeling throttled sources**: Simulating cores that are artificially rate-limited
*   **Custom hardware configurations**: Non-standard device setups

### Setting Manual Injection Rates

#### In JSON Workloads

When loading workloads from JSON, include the `injection_rate` field for each transfer:

`{  "phases": [{    "transfers": [{      "src_x": 1,      "src_y": 2,      "dst_x": 3,      "dst_y": 4,      "packet_size": 2048,      "num_packets": 1,      "injection_rate": 25.5,  // Manual rate in GB/cycle      "phase_cycle_offset": 0,      "noc_type": "NOC_0"    }]  }]}`
If the `injection_rate` field is missing or zero, it will be overwritten during inference if enabled.

#### In Programmatic Workloads

When constructing workloads programmatically, pass the injection rate to the transfer constructor:

`import tt_npe_pybind as npe cfg = npe.Config()cfg.infer_injection_rate_from_src = False  # Disable inference wl = npe.Workload()phase = npe.Phase() # Manually specify injection ratephase.addTransfer(npe.Transfer(    packet_size=2048,    num_packets=1,    src=npe.Coord(0, 1, 2),    dst=npe.Coord(0, 3, 4),    injection_rate=25.5,  # Manual rate    start_offset=0,    noc_type=npe.NocType.NOC_0)) wl.addPhase(phase)`

When converting NoC traces to workloads (see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads)), the injection rate field is initialized to 0.0 for all transfers:

**Trace Ingestion and Inference Integration**

This ensures that:

1.   Trace-derived workloads start with zero injection rates
2.   The inference system will populate appropriate rates if enabled
3.   Manual rate specification in traces is optional

## Validation Interaction

Injection rate validation occurs during workload validation, which happens after preprocessing:

**Workload Validation Flow**

The validation system checks:

*   Source and destination coordinates are within device bounds
*   Injection rates are non-negative
*   Transfer sizes are valid

If inference is disabled and injection rates are not manually set, transfers will have default values that may not be realistic for the actual hardware.

## Best Practices

### Recommended Approach

For most use cases, **enable automatic inference**:

*   Set `cfg.infer_injection_rate_from_src = True`
*   Omit `injection_rate` field in JSON workloads or set to 0
*   Let the device model provide accurate rates

### When to Disable Inference

Consider disabling inference when:

*   **Comparing with external benchmarks**: Match specific injection rates
*   **Sensitivity analysis**: Test various injection rate scenarios
*   **Custom device models**: Using modified or experimental device configurations

### Debugging Injection Rates

To verify injection rates are correctly set:

1.   Enable verbose output (`cfg.set_verbosity_level(2)`)
2.   Check transfer initialization logs during preprocessing
3.   Examine the device model's injection rate lookup tables
