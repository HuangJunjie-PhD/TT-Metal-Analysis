## Purpose and Scope

This guide walks through writing and executing a first tt-lang kernel: a simple element-wise addition operation. It demonstrates the basic structure of tt-lang kernels, the circular buffer lifecycle, and how to run code in both simulation and on hardware.

tt-lang joins the Tenstorrent software ecosystem as an ergonomic middle ground between high-level **TT-NN** and low-level **TT-Metalium**[docs/sphinx/specs/TTLangSpecification.md 53-55](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L53-L55) It allows developers to author custom kernels with explicit control over data movement and compute threads while the compiler handles resource management like DST register allocation and NOC addressing [docs/sphinx/specs/TTLangSpecification.md 57-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L57-L58)

## Your First Kernel: Element-wise Addition

### Complete Example

The following kernel adds two tensors element-wise. It defines a grid and specific thread functions for computation and data movement. The `with` statement is the recommended pattern as it automatically handles the `wait`/`pop` and `reserve`/`push` lifecycle of buffers [test/python/simple_add_with_stmt.py 12-18](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_add_with_stmt.py#L12-L18)

`import ttlimport ttnn @ttl.operation(grid=(1, 1))def add_kernel(lhs, rhs, out):    # Create circular buffers (Dataflow Buffers) for each tensor    # block_count=2 enables double-buffering    lhs_dfb = ttl.make_dataflow_buffer_like(lhs, shape=(1, 1), block_count=2)    rhs_dfb = ttl.make_dataflow_buffer_like(rhs, shape=(1, 1), block_count=2)    out_dfb = ttl.make_dataflow_buffer_like(out, shape=(1, 1), block_count=2)     @ttl.compute()    def add_compute():        # 'with' handles wait/reserve at entry, pop/push at exit        with lhs_dfb.wait() as l, rhs_dfb.wait() as r, out_dfb.reserve() as o:            # Compute: tile-based addition using operator overloading            result = l + r            # Result is stored into the reserved output block            o.store(result)     @ttl.datamovement()    def dm_read():        # Read from DRAM/L1 into Dataflow Buffer        with lhs_dfb.reserve() as lhs_blk:            tx_lhs = ttl.copy(lhs[0, 0], lhs_blk)            tx_lhs.wait()                with rhs_dfb.reserve() as rhs_blk:            tx_rhs = ttl.copy(rhs[0, 0], rhs_blk)            tx_rhs.wait()     @ttl.datamovement()    def dm_write():        # Wait for compute to finish, then write back to DRAM        with out_dfb.wait() as out_blk:            tx = ttl.copy(out_blk, out[0, 0])            tx.wait()`

### Kernel Structure and Hardware Mapping

tt-lang maps Python functions to specific hardware threads on a Tenstorrent Tensix core. The `@ttl.operation` decorator (aliased from `ttl.kernel` in earlier versions) serves as the entry point [docs/sphinx/specs/TTLangSpecification.md 60-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L60-L62) while `@ttl.compute` and `@ttl.datamovement` target the TRISC (Math/Pack/Unpack) and BRISC/NCRISC processors respectively [docs/sphinx/specs/TTLangSpecification.md 74-84](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L74-L84)

### Circular Buffer (Dataflow Buffer) Lifecycle

The Dataflow Buffer (DFB) is the primary synchronization mechanism between threads. Producers `reserve` and `push` blocks [test/python/simple_add.py 46-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_add.py#L46-L49) while consumers `wait` and `pop` them [test/python/simple_add.py 34-41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_add.py#L34-L41)

## Running on Hardware

To run on Tenstorrent hardware, the kernel is compiled through an MLIR-based pipeline into C++.

### Execution Example with TTNN

A standard execution script uses `ttnn` to manage device lifecycle and tensor transfers [test/python/simple_add.py 179-195](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_add.py#L179-L195)

`import torchimport ttnnimport ttl device = ttnn.open_device(device_id=0)try:    # Create tensors on host and move to device    dim = 64    a_torch = torch.rand((dim, dim), dtype=torch.bfloat16)    b_torch = torch.rand((dim, dim), dtype=torch.bfloat16)        a = ttnn.from_torch(a_torch, dtype=ttnn.bfloat16, layout=ttnn.TILE_LAYOUT, device=device)    b = ttnn.from_torch(b_torch, dtype=ttnn.bfloat16, layout=ttnn.TILE_LAYOUT, device=device)    out = ttnn.from_torch(torch.zeros_like(a_torch), dtype=ttnn.bfloat16, layout=ttnn.TILE_LAYOUT, device=device)     # Execute the compiled kernel    add_kernel(a, b, out)     # Verify results    result = ttnn.to_torch(out)    expected = a_torch + b_torch    assert torch.allclose(result, expected, rtol=1e-2, atol=1e-2)finally:    ttnn.close_device(device)`

### Compilation and Multi-tile Handling

For larger tensors, kernels use loops to iterate over tiles. The compiler generates nested loops and linearized indexing to handle multi-tile grids [test/python/simple_add_multitile.py 114-121](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_add_multitile.py#L114-L121) The compilation pipeline transforms the Python AST into TTL MLIR, which is then lowered to TTKernel and finally EmitC (C++) [test/me2e/builder/pipeline.py 43-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/me2e/builder/pipeline.py#L43-L76)

### Pass Pipeline Overview

The `compile_ttl_to_ttkernel` function orchestrates the transformation from high-level DSL to hardware-specific C++ code [test/me2e/builder/pipeline.py 18-23](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/me2e/builder/pipeline.py#L18-L23) Key stages include:

1.   **Sync Insertion**: `ttl-insert-cb-sync` and `ttl-insert-copy-wait` ensure all buffers are properly pushed/popped and transfers are waited on [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 26-27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L26-L27)
2.   **Compute Fusion**: `convert-ttl-to-compute` fuses element-wise ops into unified compute kernels [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 29](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L29-L29)
3.   **DST Management**: `ttl-assign-dst` and `ttl-subblock-compute-for-dst` manage the limited DST register file [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36-41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L41)
4.   **Loop Lowering**: `ttl-lower-to-loops` generates the final nested loop structures [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 44-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L44-L48)
