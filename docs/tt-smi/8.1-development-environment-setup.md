## Prerequisites

### Python Version Requirements

TT-SMI requires **Python 3.10 or higher** as specified in [pyproject.toml 6](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L6-L6) The project supports Python versions through 3.11, with the following classifiers defined in [pyproject.toml 17-23](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L17-L23):

| Python Version | Support Status |
| --- | --- |
| 3.7 - 3.9 | Listed in classifiers but below minimum |
| 3.10 | Minimum required version |
| 3.11 | Fully supported |

### Rust Installation

TT-SMI depends on `pyluwen` (version ~0.8.0) which requires Rust compilation. Rust must be installed before attempting to install TT-SMI dependencies.

#### Distribution Package Installation (Preferred)

**Fedora / EL9:**

`sudo dnf install cargo`
**Ubuntu / Debian:**

`sudo apt install cargo`
#### Rustup Installation (Alternative)

`curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | shsource "$HOME/.cargo/env"`

## Installation Modes

### Standard Installation

Standard installation copies package files to the Python environment's site-packages directory. Changes to source code require reinstallation:

`pip install .`
This mode is suitable for:

*   Testing the package before contributing
*   Installing a specific development version
*   CI/CD build validation

### Editable Installation (Recommended for Development)

Editable mode creates symbolic links to the source directory. Code changes are immediately reflected without reinstallation:

`pip install --editable .`
This mode is **recommended for active development** because:

*   No rebuild required after code changes
*   Immediate testing of modifications
*   Simplified debugging workflow

The editable installation is configured through the `setuptools` backend in [pyproject.toml 77-84](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L77-L84)

## Dependency Version Constraints

The project uses semantic versioning constraints defined in [pyproject.toml 26-36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L36):

| Package | Version Constraint | Rationale |
| --- | --- | --- |
| `pyluwen` | `~=0.8.0` | Compatible release, allows 0.8.x updates |
| `tt_tools_common` | `~=1.6.0` | Compatible release, allows 1.6.x updates |
| `textual` | `>=0.59.0` | Minimum version, allows any newer release |
| `rich` | `>=13.7.0` | Minimum version for required features |
| `pydantic` | `>=1.2` | Minimum version for data validation |
| `pre-commit` | `>=3.5.0` | Minimum version for hook support |

The tilde (`~=`) operator ensures tight coupling with Tenstorrent-specific libraries (`pyluwen`, `tt_tools_common`) to maintain hardware compatibility, while the greater-than-or-equal (`>=`) operator provides flexibility for common libraries.

**Sources:**[pyproject.toml 26-36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L36)

## Pre-Commit Hook Configuration

### Installing Pre-Commit Hooks

After repository cloning and package installation, install pre-commit hooks:

`pre-commit install`
This command installs Git hooks that automatically run before each commit, as defined in [.pre-commit-config.yaml](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml)

### Hook Functionality

The pre-commit configuration includes the following hooks:

**Hook Definitions:**

| Hook | Purpose | Source Repository |
| --- | --- | --- |
| `check-yaml` | Validates YAML file syntax | pre-commit/pre-commit-hooks |
| `end-of-file-fixer` | Ensures files end with newline | pre-commit/pre-commit-hooks |
| `trailing-whitespace` | Removes trailing whitespace | pre-commit/pre-commit-hooks |
| `check-toml` | Validates TOML file syntax | pre-commit/pre-commit-hooks |
| `black` | Python code formatting | psf/black |

The `black` formatter uses Python 3.8 as the language version for compatibility, as specified in [.pre-commit-config.yaml 9](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L9-L9) and [.pre-commit-config.yaml 14](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.pre-commit-config.yaml#L14-L14)

## Package Entry Points

The package defines a single command-line entry point in [pyproject.toml 53-54](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L53-L54):

`[project.scripts]tt-smi = "tt_smi:main"`
This configuration creates the `tt-smi` executable that invokes the `main()` function from the `tt_smi` module. After installation, users can run:

`tt-smi --versiontt-smi --helptt-smi  # Launch GUI`
**Sources:**[pyproject.toml 53-54](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L53-L54)

## Package Data Inclusion

The development environment must account for non-Python resources included in the package. The [pyproject.toml 56-72](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L56-L72) configuration specifies:

`[tool.setuptools.package-data]"*" = [  'data/version.txt',  "*.css",  "data/*/*.yaml",  "data/*/*/*.yaml",]`
These patterns include:

*   `data/version.txt` - Version information file
*   `*.css` - Textual TUI stylesheets
*   `data/*/*.yaml` and `data/*/*/*.yaml` - Configuration data for hardware platforms

The following directories are explicitly excluded from packaging:

*   `build/` - Build artifacts
*   `debian/` - Debian packaging files
*   `images/` - Documentation images

**Sources:**[pyproject.toml 56-72](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L56-L72)

## Verification Steps

### Verify Installation

After completing the setup, verify the installation:

`# Check that tt-smi is accessiblett-smi --version # Expected output format:# tt-smi version 3.1.1`
### Verify Development Mode

For editable installations, verify that code changes are reflected:

1.   Edit a source file (e.g., add a print statement)
2.   Run `tt-smi` to see the change without reinstalling
3.   Remove the test change

### Verify Pre-Commit Hooks

Test that pre-commit hooks are active:

`# Make a dummy change with trailing whitespaceecho "test   " >> test_file.txtgit add test_file.txtgit commit -m "Test commit" # Expected: pre-commit hooks run and fix trailing whitespace`

## CI/CD Environment Setup

The CI/CD pipeline defined in [.gitlab-ci.yml](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml) demonstrates the automated build environment setup. Understanding this configuration helps developers replicate the CI environment locally.

The CI pipeline runs two stages:

1.   **Format Stage** ([.gitlab-ci.yml 19-29](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L19-L29)): Validates Python code formatting using `black==24.8.0`
2.   **Build Stage** ([.gitlab-ci.yml 31-51](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L31-L51)): Builds the wheel package with Rust compilation

Both stages use Python 3.8 virtual environments and install dependencies via pip. The build stage additionally installs Rust via `rustup` and creates distributable wheel artifacts.

**Sources:**[.gitlab-ci.yml 1-52](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.gitlab-ci.yml#L1-L52)

## Virtual Environment Recommendations

While not strictly required, using a virtual environment is recommended to isolate TT-SMI dependencies:

`# Create virtual environmentpython -m venv .venv # Activate virtual environmentsource .venv/bin/activate  # Linux/macOS# or.venv\Scripts\activate  # Windows # Install in virtual environmentpip install --editable .`
Benefits of virtual environments:

*   Isolates TT-SMI dependencies from system Python
*   Prevents version conflicts with other projects
*   Allows clean environment teardown (delete `.venv` directory)
*   Matches CI/CD environment isolation pattern

## Development Workflow Summary

A typical development workflow after setup:

1.   **Activate virtual environment** (if used): `source .venv/bin/activate`
2.   **Make code changes** in `tt_smi/` directory
3.   **Test changes immediately** (editable mode): `tt-smi` commands work without reinstall
4.   **Run tests** (see [Testing](https://deepwiki.com/tenstorrent/tt-smi/8.2-testing)): `pytest tests/`
5.   **Commit changes**: Pre-commit hooks automatically run formatting and validation
6.   **Push changes**: CI/CD pipeline validates build and formatting
