# Performance Metrics Extraction

*   [tests/test_tools/test_parse_metal_tensix_results.py](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tests/test_tools/test_parse_metal_tensix_results.py)

## Purpose and Scope

The Performance Metrics Extraction system is responsible for extracting structured performance metrics from TT-Metal Tensix HTML reports and converting them into a standardized format for analysis. This system parses HTML tables containing benchmark results, normalizes column names and data formats, and outputs structured YAML files containing performance metrics.

This system serves as the foundation for correlation analysis - for information about comparing extracted metrics against simulation results, see [Correlation Analysis](https://deepwiki.com/tenstorrent/polaris/3.2-correlation-analysis). For details on visualization and reporting of these metrics, see [Visualization and Reports](https://deepwiki.com/tenstorrent/polaris/3.4-visualization-and-reports).

## System Overview

The performance metrics extraction system operates as a standalone tool that fetches HTML content from TT-Metal repositories, parses tabular performance data, and converts it into structured `TensixNwPerfMetricModel` objects. The extracted metrics are then saved as YAML files organized by workload type.

## Data Model

The system uses the `TensixNwPerfMetricModel` Pydantic model to structure performance metrics with standardized field names and types.

**TensixNwPerfMetricModel Structure**

The model includes the following key fields:

*   `benchmark`: Standardized benchmark name (e.g., "Benchmark.BERT", "Benchmark.ResNet50")
*   `wlname`: Original workload name from the source data
*   `gpu_batch_size`: Batch size used for the benchmark
*   `perf`: Actual performance value
*   `target_perf`: Target performance value for comparison
*   `system`: Hardware system identifier (e.g., "n150", "n300")
*   `metric`: Performance metric type (e.g., "Samples/s")

Sources: [tools/parse_metal_tensix_results.py 23-48](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/parse_metal_tensix_results.py#L23-L48)

## HTML Parsing Pipeline

The system implements a robust HTML parsing pipeline that handles various table formats and column naming conventions found in TT-Metal reports.

**HTML Table Parsing Workflow**

The `parse_html_table` function processes HTML table elements by:

1.   Extracting header rows (`th` elements) to determine column names
2.   Applying column name normalization via `get_colname_map`
3.   Extracting data rows (`td` elements)
4.   Returning structured column names and row data

The `COLNAME_MAP` dictionary handles various column naming conventions:

*   `'ttft_(ms)'` → `'time_to_first_token_ms'`
*   `'t/s/u'` → `'tokens_per_second_per_user'`
*   `'sen/sec'` → `'perf'`
*   `'target_sen/sec'` → `'target_perf'`
*   `'model'` → `'benchmark'`

## Metric Processing and Normalization

The system applies workload-specific processing to normalize benchmark data into a consistent format.

**Workload Processing Pipeline**

The `process_bert_row` and `process_resnet_row` functions standardize workload-specific data:

*   Extract workload name from the benchmark field
*   Set standardized benchmark identifiers
*   Normalize metric types to "Samples/s"
*   Clean numeric values by removing commas and asterisks

The system also filters results based on system identifiers, only processing entries where the system field matches the pattern `[a-z][0-9][0-9][0-9]` (e.g., "n150", "n300").

Sources: [tools/parse_metal_tensix_results.py 123-153](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/parse_metal_tensix_results.py#L123-L153)[tools/parse_metal_tensix_results.py 188-214](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/parse_metal_tensix_results.py#L188-L214)

## Output Generation

The system organizes extracted metrics by workload type and generates separate YAML files for each workload category.

**Output Organization**

The `save_metrics` function:

1.   Groups metrics by workload name (converting variations like "resnet-50" to "resnet50")
2.   Creates separate YAML files for each workload type
3.   Uses Pydantic's `model_dump(mode='yaml')` for consistent serialization
4.   Saves files to the specified output directory (default: `data/metal/inf`)

The system also provides logging and reporting functionality through:

*   `setup_logger`: Configures structured logging output
*   `report_systems_of_interest`: Reports unique hardware systems found in the data

Sources: [tools/parse_metal_tensix_results.py 236-266](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/parse_metal_tensix_results.py#L236-L266)[tools/parse_metal_tensix_results.py 217-234](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/parse_metal_tensix_results.py#L217-L234)

## Command Line Interface

The system provides a command-line interface through the `parse_metal_tensix_results` function with configurable output directory and caching options.

**CLI Usage**

| Argument | Short | Description | Default |
| --- | --- | --- | --- |
| `--output-dir` | `-o` | Directory to save extracted metrics | `data/metal/inf` |

The main execution flow:

1.   `create_args`: Parses command line arguments
2.   `setup_logger`: Configures logging
3.   `extract_table_from_html_link`: Fetches and processes HTML content
4.   `save_metrics`: Outputs structured YAML files
5.   `report_systems_of_interest`: Logs system analysis
