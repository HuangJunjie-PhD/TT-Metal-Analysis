# Stateful vs Stateless APIs

## Purpose and Scope

This document explains the difference between stateful and stateless NOC data movement APIs in TT-Metal, their register state preservation mechanisms, and when to use each pattern for optimal performance. Stateful APIs amortize register programming overhead across multiple operations by preserving NOC register state, while stateless APIs reconfigure all necessary registers on every call.

For general NOC API usage patterns, see [Core API Concepts](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.1-core-api-concepts). For specific synchronization requirements, see [Asynchronous Operations and Batching](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.2-asynchronous-operations-and-batching). For platform-specific register availability, see [Register State Preservation](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.1-register-state-preservation).

## Overview

NOC APIs execute by programming hardware registers in command buffers. The key distinction between stateful and stateless APIs is **whether these registers preserve their values across multiple API calls**:

*   **Stateful APIs**: Program registers once with `_set_state()`, then issue multiple operations with `_with_state()` that reuse those register values
*   **Stateless APIs**: Program all necessary registers on every call, providing complete configuration independence

The performance impact can be significant: stateful APIs eliminate redundant register writes when performing multiple operations with similar parameters.

**Key observations:**

*   Address registers (`NOC_TARG_ADDR_*`, `NOC_RET_ADDR_*`) are stateful - they maintain their values
*   Control registers (`NOC_CMD_CTRL`, `NOC_NODE_ID`, `NOC_ENDPOINT_ID`) are not stateful - they must be set on each transaction
*   Blackhole has additional stateful registers (`NOC_BRCST_EXCLUDE`, `NOC_L1_ACC_AT_INSTRN`, `NOC_SEC_CTRL`) not present in Wormhole B0

## Stateful API Pattern

### When to Use Stateful APIs

Stateful APIs provide performance benefits when:

1.   **Multiple operations to the same destination**: Target NOC address (`NOC_TARG_ADDR_*`) remains constant
2.   **Fixed transfer sizes**: Packet size registers (`NOC_AT_LEN_BE`) don't change
3.   **Repeated operation type**: Control flags (`NOC_CTRL`) remain the same
4.   **Sequential operations**: The overhead of state setup can be amortized across N operations

**Do NOT use stateful APIs when:**

*   Destinations change significantly between operations
*   Transfer sizes vary per operation
*   Operations are isolated (single transaction)
*   Context switches occur between operations

Sources: [data_movement_doc/general/dm_best_practices.md 93-99](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L93-L99)

### Stateful API Functions

The following API pairs implement the stateful pattern:

**Read Operations:**

*   `noc_async_read_set_state()` - Configure stateful registers for reads
*   `noc_async_read_with_state()` - Execute read using preserved state

**Write Operations:**

*   `noc_async_write_one_packet_set_state()` - Configure stateful registers for single-packet writes
*   `noc_async_write_one_packet_with_state()` - Execute write using preserved state

## Stateless API Pattern

### When to Use Stateless APIs

Stateless APIs are appropriate when:

1.   **Destinations vary per operation**: Each transaction targets a different NOC address
2.   **Transfer sizes differ**: Packet sizes change between operations
3.   **Isolated transactions**: Single operations where setup overhead isn't amortized
4.   **Maximum flexibility needed**: Complete independence between operations
5.   **Context switches occur**: When NOC state may be modified by other code paths

Sources: [data_movement_doc/general/dm_best_practices.md 98-99](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L98-L99)

### Stateless API Functions

The standard NOC APIs are stateless:

**Read Operations:**

*   `noc_async_read()` - Programs all necessary registers for each read

**Write Operations:**

*   `noc_async_write()` - Programs all necessary registers for each write
*   `noc_async_write_multicast()` - Programs all necessary registers for each multicast

These APIs are simpler to use but incur register programming overhead on every call.

## Performance Impact Analysis

### Register Programming Overhead

**Performance characteristics:**

| Pattern | Register Writes per Operation | Best Use Case |
| --- | --- | --- |
| Stateless | ~6-8 registers | Single operations, varying destinations |
| Stateful (first call) | ~6-8 registers (setup) | N/A |
| Stateful (subsequent) | ~2 registers | Multiple operations, same destination |

The performance gain from stateful APIs increases with the number of operations sharing the same configuration. For N operations:

*   **Stateless overhead**: `N × (6-8 register writes)`
*   **Stateful overhead**: `1 × (6-8 register writes) + N × (2 register writes)`

For N > 2, stateful APIs reduce register programming overhead.

## Critical Synchronization Requirements

### State Preservation Across Context Switches

When using stateful APIs, be aware that **NOC register state is not preserved across context switches** or when other code modifies NOC state. Always call `_set_state()` before `_with_state()` operations if:

1.   Other kernel code may have used NOC APIs
2.   Control flow has branched significantly
3.   State was set in a different function context

## Decision Flow: Choosing Stateful vs Stateless

**Key decision criteria:**

1.   **Destination consistency**: Same target address across operations
2.   **Size consistency**: Same transfer size across operations
3.   **Locality**: Operations in tight loop without context switches
4.   **State preservation**: No intervening code that modifies NOC state

If all criteria are met, stateful APIs provide measurable performance benefits. Otherwise, use stateless APIs for simplicity and safety.
