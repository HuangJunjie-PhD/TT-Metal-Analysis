This page documents how documentation is automatically built and deployed to GitHub Pages through the CI/CD pipeline. It focuses specifically on the deployment job, trigger conditions, and GitHub Pages integration.

For information about building documentation locally, see [Building Documentation](https://deepwiki.com/tenstorrent/riescue/6.1-building-documentation). For the overall documentation system architecture, see [Documentation System](https://deepwiki.com/tenstorrent/riescue/6-documentation-system).

## Overview

Documentation deployment is handled by the `deploy-docs` job in the smoke workflow, which runs exclusively on commits to the `main` branch. After passing linting checks, the job builds Sphinx documentation and deploys it to GitHub Pages using GitHub Actions.

The deployment process uses the `DocBuilder` class from [docs/build.py](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py) in CI mode with the `--check` flag disabled to generate HTML output that is then uploaded as a GitHub Pages artifact.

## Deployment Job Configuration

The `deploy-docs` job is defined in the smoke workflow with specific trigger conditions and permissions.

### Trigger Condition

The job only executes on the `main` branch using a conditional check:

`if: github.ref == 'refs/heads/main'`
This prevents documentation deployment from feature branches or pull requests, ensuring only canonical documentation from `main` is published.

**Sources:**[.github/workflows/smoke.yml 78](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L78-L78)

### Required Permissions

The job requires elevated permissions to deploy to GitHub Pages:

| Permission | Access Level | Purpose |
| --- | --- | --- |
| `contents` | `read` | Access repository files for building docs |
| `pages` | `write` | Upload and deploy Pages artifacts |
| `id-token` | `write` | Authenticate with GitHub Pages service |

These permissions are explicitly declared in the workflow file at [.github/workflows/smoke.yml 79-82](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L79-L82)

**Sources:**[.github/workflows/smoke.yml 79-82](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L79-L82)

## Build Process in CI

**Sources:**[.github/workflows/smoke.yml 84-93](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L84-L93)

### Build Command

The documentation build command in CI:

`./docs/build.py --build_dir=public --source_dir=docs/source`
This invocation differs from the linting phase's documentation check:

| Phase | Command | Flags | Purpose |
| --- | --- | --- | --- |
| Lint | `./docs/build.py --build_dir=public --source_dir=docs/source --check` | `--check` | Fail on Sphinx warnings |
| Deploy | `./docs/build.py --build_dir=public --source_dir=docs/source` | None | Generate HTML for deployment |

The deploy phase omits the `--check` flag because warnings should already be caught in the lint job. The build outputs to `public/` directory for GitHub Pages deployment.

**Sources:**[.github/workflows/smoke.yml 33](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L33-L33)[.github/workflows/smoke.yml 92](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L92-L92)

### DocBuilder Integration

The `DocBuilder` class handles the build process:

1.   **Initialization**: [docs/build.py 29-52](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L29-L52) creates a `DocBuilder` instance with paths and options
2.   **Directory Setup**: [docs/build.py 102-106](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L102-L106) ensures `build_dir` exists
3.   **Sphinx Execution**: [docs/build.py 110-122](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L110-L122) constructs and runs the `sphinx-build` command
4.   **Locale Configuration**: [docs/build.py 118-120](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L118-L120) sets `LC_ALL=C` and `LANG=C` to avoid locale errors in containers

The CI invocation maps to:

`DocBuilder(    clean=False,    source_dir=Path("docs/source"),    build_dir=Path("public"),    check=False,    local_host=False,    port=8888,    host="0.0.0.0",    interactive=False)`
**Sources:**[docs/build.py 29-52](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L29-L52)[docs/build.py 102-122](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L102-L122)

## GitHub Pages Deployment

### Upload Artifact

The build output is uploaded using the `actions/upload-pages-artifact@v3` action:

`- name: Deploy to GitHub Pages  uses: actions/upload-pages-artifact@v3  with:    path: ./public`
This action packages the `public/` directory contents into a GitHub Pages artifact for deployment.

**Sources:**[.github/workflows/smoke.yml 94-97](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L94-L97)

### Deploy to Pages

The deployment step uses `actions/deploy-pages@v4`:

`- name: Deploy  uses: actions/deploy-pages@v4`
This action takes the uploaded artifact and publishes it to the repository's GitHub Pages site. The action handles:

*   Artifact retrieval from the upload step
*   Deployment to the Pages environment
*   Cache invalidation for updated content

**Sources:**[.github/workflows/smoke.yml 99-100](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L99-L100)

## Dependency Chain

The deployment job has explicit dependencies declared:

`needs: lint`
This ensures the `lint` job completes successfully before documentation deployment begins. Since `lint` itself depends on `check-spdx-headers`, the complete dependency chain is:

1.   SPDX header validation
2.   Code linting and documentation build check
3.   Documentation deployment (main branch only)

**Sources:**[.github/workflows/smoke.yml 77](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L77-L77)

## Build vs. Test Branch Behavior

The smoke workflow implements branch-specific behavior:

| Branch | Job Executed | Purpose |
| --- | --- | --- |
| `main` | `deploy-docs` | Deploy documentation to GitHub Pages |
| Non-main | `test` | Run unit tests in containerized environment |

Both jobs depend on the `lint` job, but only one executes based on the branch condition. This prevents redundant test execution on `main` while ensuring feature branches are validated before merge.

The `test` job explicitly excludes `main`:

`if: github.ref != 'refs/heads/main'`
While the `deploy-docs` job explicitly requires `main`:

`if: github.ref == 'refs/heads/main'`
**Sources:**[.github/workflows/smoke.yml 38](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L38-L38)[.github/workflows/smoke.yml 78](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L78-L78)

## Source Directory Selection

The deployment always builds from the public documentation source:

`--source_dir=docs/source`
This differs from the optional internal documentation mode (`BUILD_TT_DOCS` environment variable) used by `DocBuilder` for local builds. The `DocBuilder` class selects the source directory in [docs/build.py 23-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L23-L26):

`if os.getenv("BUILD_TT_DOCS"):    defualt_src = docs_dir / "tt_source"else:    defualt_src = docs_dir / "source"`
However, the CI deployment explicitly specifies `docs/source`, ensuring only public documentation is deployed regardless of environment variables.

**Sources:**[docs/build.py 23-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/build.py#L23-L26)[.github/workflows/smoke.yml 92](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/workflows/smoke.yml#L92-L92)

## Deployment URL

After successful deployment, documentation is accessible at the repository's GitHub Pages URL. For the Tenstorrent RiESCUE repository, this would be:

```
https://tenstorrent.github.io/riescue/
```

The actual URL depends on the repository owner and name. GitHub Pages automatically serves the `index.html` file from the deployed artifact.
