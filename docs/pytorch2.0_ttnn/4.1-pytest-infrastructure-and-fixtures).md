## Purpose and Scope

This document details the pytest infrastructure defined in [tests/conftest.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py) The conftest module provides the foundational testing framework including command-line options, device management fixtures, test lifecycle management, and integration with the metrics collection system. This infrastructure supports all model and operation tests in the repository.

For information about the ModelTester framework that tests use, see [ModelTester Framework](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.2-modeltester-framework). For details on test utilities and assertion helpers, see [Test Utilities and Assertion Helpers](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.3-test-utilities-and-assertion-helpers).

**Sources:**[tests/conftest.py 1-430](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L1-L430)

## Command-Line Options

The pytest framework is extended with several command-line options via `pytest_addoption` that control test behavior and compilation settings.

### Configuration Options

| Option | Type | Default | Purpose |
| --- | --- | --- | --- |
| `--input_var_only_native` | flag | False | Run tests only on native PyTorch without TTNN compilation |
| `--input_var_check_ttnn` | flag | False | Verify TTNN conversion for input variations |
| `--input_var_check_accu` | flag | False | Check accuracy metrics for input variations |
| `--report_nth_iteration` | int | 1 | Number of iterations to run; metrics reported on final iteration |
| `--export_code` | string | None | Export standalone Python code (see [Code Export](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/8.1-code-export-and-standalone-script-generation)) |
| `--data_parallel` | flag | False | Enable data parallel execution across multiple devices |
| `--native_integration` | flag | False | Use native device integration (incompatible with data_parallel) |
| `--disable_load_params_once` | flag | False | Disable LoadParamsOnce optimization |

## Device Fixture and Hardware Configuration

The `device` fixture is a session-scoped fixture that manages Tenstorrent hardware initialization and cleanup for the entire test session.

### Device Configuration Flow

**Sources:**[tests/conftest.py 77-110](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L77-L110)[tests/conftest.py 112-128](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L112-L128)

### Dispatch Core Configuration

The dispatch core configuration is standardized across all tests to ensure consistent hardware resource allocation:

*   **Core Type:** Always `ttnn.device.DispatchCoreType.ETH` to maximize available cores on N300 devices
*   **Core Axis:**`ttnn.DispatchCoreAxis.ROW` (should be `COL` for Blackhole architecture)
*   **L1 Small Size:** 65536 bytes (manually determined per tt-metal#13746)

**Sources:**[tests/conftest.py 112-128](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L112-L128)

## The compile_and_run Fixture

The `compile_and_run` fixture is the central orchestrator for test execution, compilation, and metrics collection. It is an autouse fixture that runs for every test function.

### Execution Flow

**Sources:**[tests/conftest.py 192-411](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L192-L411)

### Key Responsibilities

The `compile_and_run` fixture performs the following operations:

1.   **Metrics Initialization:** Sets up `runtime_metrics` and `comp_runtime_metrics` dictionaries with test metadata
2.   **Test Execution:** Yields control to the test function body and measures execution time
3.   **Compilation:** If test provides `torch_ttnn` user property, compiles model with `TorchTtnnOption`
4.   **Iteration Control:** Runs model for `--report_nth_iteration` iterations to measure warm performance
5.   **Native Integration:** Moves model/inputs to TTNN device if `--native_integration` is enabled
6.   **Accuracy Calculation:** Compares original outputs with compiled outputs using `calculate_accuracy`
7.   **Schema Collection:** Saves both original and compiled schema lists to pickle files
8.   **Error Handling:** On compilation failure, attempts bypass run to collect aten operation schemas
9.   **Validation:** Checks that models marked `converted_end_to_end` contain no aten operations

**Sources:**[tests/conftest.py 192-411](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L192-L411)

### TorchTtnnOption Configuration

The fixture constructs `TorchTtnnOption` from command-line arguments:

**Sources:**[tests/conftest.py 286-297](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L286-L297)

### Load Params Once Logic

The `load_params_once` optimization is enabled by default but can be disabled via:

*   Command-line: `--disable_load_params_once`

## Metrics Collection Integration

The `compile_and_run` fixture integrates with the metrics collection system (see [Metrics Collection and Schema Tracking](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.1-metrics-collection-and-schema-tracking)) to save comprehensive test results.

### Metrics File Structure

**Sources:**[tests/conftest.py 200-260](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L200-L260)[tests/conftest.py 356-407](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L356-L407)

### Original Runtime Metrics

Saved to `metrics/{model_name}/original-run_time_metrics.pickle`:

| Field | Description |
| --- | --- |
| `success` | Boolean indicating test success |
| `start_ts` | Test start timestamp (UTC) |
| `end_ts` | Test end timestamp (UTC) |
| `test_name` | Full test name from pytest |
| `test_filepath` | Path to test file |
| `model_name` | Extracted model name |
| `model_path` | Path to model directory |
| `run_time` | Execution time in milliseconds |
| `error_message` | Error message if test failed |

**Sources:**[tests/conftest.py 200-260](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L200-L260)

### Compiled Runtime Metrics

Saved to `metrics/{model_name}/compiled-run_time_metrics.pickle`:

| Field | Description |
| --- | --- |
| `success` | Boolean indicating compilation success |
| `avg_run_time` | Average runtime from warm iterations (excluding first two) |
| `run_time` | Last iteration runtime in milliseconds |
| `run_time_first_iter` | First iteration runtime (includes compilation overhead) |
| `has_aten` | Boolean indicating presence of aten operations |
| `batch_size` | Batch size used in test |
| `accuracy` | Accuracy metric from `calculate_accuracy` |
| `fits_in_memory` | Memory fit status (N/A on failure) |
| `peak_sram_usage` | Peak SRAM usage in bytes |

**Sources:**[tests/conftest.py 338-372](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L338-L372)[tests/conftest.py 403-407](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L403-L407)

### Schema Lists

Two schema list pickle files are saved:

1.   **original-schema_list.pickle:** Contains `OpSchema.dict_for_pickle()` entries for operations observed during bypass run
2.   **compiled-schema_list.pickle:** Contains `OpSchema.dict()` entries for operations after compilation

These schemas feed into the metrics aggregation system for input variation tracking and guard generation.

**Sources:**[tests/conftest.py 356-360](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L356-L360)[tests/conftest.py 398-402](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L398-L402)

## Pytest Markers

The infrastructure recognizes several custom pytest markers for test behavior control:

| Marker | Purpose |
| --- | --- |
| `@pytest.mark.skip_platform("arch")` | Skip test on specified architecture (grayskull, wormhole_b0) |
| `@pytest.mark.e2e_with_native_integration` | Mark test as compatible with `--native_integration` |
| `@pytest.mark.compilation_xfail` | Expected compilation failure; xfail instead of raising error |
| `@pytest.mark.converted_end_to_end` | Asserts that compiled model contains no aten operations |
