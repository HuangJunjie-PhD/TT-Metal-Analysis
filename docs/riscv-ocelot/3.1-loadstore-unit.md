The Load/Store Unit (LSU) is responsible for managing memory operations in the RISC-V Ocelot processor. It handles the execution of loads, stores, atomic operations, and fence instructions in an out-of-order execution context while maintaining correct memory ordering semantics. This document covers the architecture, organization, and operation of the LSU, including its support for vector memory operations.

For information about the overall memory hierarchy, see [Memory System](https://deepwiki.com/tenstorrent/riscv-ocelot/3-memory-system), and for vector memory operations, see [Vector Memory Operations](https://deepwiki.com/tenstorrent/riscv-ocelot/5.3-vector-memory-operations).

## LSU Architecture Overview

The LSU maintains multiple queues to track in-flight memory operations and ensure correct program behavior. It optimistically executes loads as soon as possible while ensuring correctness through various detection mechanisms, and it executes stores to memory only after they have been committed.

Sources: [src/main/scala/lsu/lsu.scala 12-43](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L12-L43)[src/main/scala/lsu/lsu.scala 232-316](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L232-L316)

## Queue Structures

The LSU manages four primary queue structures to track memory operations:

### Load Queue (LDQ)

The Load Queue tracks all in-flight load operations. Each entry contains:

*   Valid bit
*   Memory address (when available)
*   Execution status (executed, succeeded, ordered)
*   Store dependency mask (tracks older stores that might write to the same address)
*   Forwarding information (if data was forwarded from a store)
*   Vector-specific state for vector loads

Sources: [src/main/scala/lsu/lsu.scala 234-265](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L234-L265)

### Store Queue (STQ)

The Store Queue tracks all in-flight store operations. Each entry contains:

*   Valid bit
*   Memory address (when available)
*   Store data (when available)
*   Commitment status (committed, succeeded)
*   Vector-specific state for vector stores

Sources: [src/main/scala/lsu/lsu.scala 291-308](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L291-L308)

### Vector Data Queues (DLQ and DSQ)

For vector operations, additional queues track vector memory operations:

*   Data Load Queue (DLQ): Tracks vector load operations
*   Data Store Queue (DSQ): Tracks vector store operations

These queues contain metadata needed for vector execution, including vector register IDs, element counts, and mask bits.

Sources: [src/main/scala/lsu/lsu.scala 267-290](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L267-L290)[src/main/scala/lsu/lsu.scala 309-320](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L309-L320)

## Queue Management

The LSU manages multiple pointers to track entries within each queue:

### Load Queue Pointers

*   `ldq_head`: Points to the oldest load entry ready for commitment
*   `ldq_v_head`: Similar to ldq_head but doesn't need ROB commit
*   `ldq_tail`: Points to where new load entries will be allocated

### Store Queue Pointers

*   `stq_head`: Points to the next store to clear from STQ (send to memory)
*   `stq_v_head`: Vector-specific store head pointer
*   `stq_tail`: Points to where new store entries will be allocated
*   `stq_commit_head`: Points to the next store to commit
*   `stq_execute_head`: Points to the next store to execute

Sources: [src/main/scala/lsu/lsu.scala 339-355](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L339-L355)

## Memory Operation Execution

### Dispatch and Allocation

When a memory instruction is decoded and dispatched to the ROB, it is also allocated an entry in either the LDQ or STQ. The allocation process includes:

1.   Checking if there's space in the respective queue
2.   Setting up appropriate dependency tracking
3.   Initializing queue entry fields
4.   For stores, updating the "live store mask" that tracks all active stores

Sources: [src/main/scala/lsu/lsu.scala 462-625](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L462-L625)

### Load Execution

Load operations are executed optimistically as soon as possible. The execution process includes:

1.   Address calculation and TLB lookup
2.   Checking for conflicts with earlier stores (for forwarding or ordering)
3.   Accessing the data cache
4.   Forwarding from store queue if needed
5.   Handling memory order violations if detected

For loads, the LSU can "wake up" sleeping loads or retry loads that had TLB misses once those conditions are cleared.

Sources: [src/main/scala/lsu/lsu.scala 716-816](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L716-L816)

### Store Execution

Stores have a two-phase execution:

1.   **Execute Phase:** Address and data calculation
2.   **Commit Phase:** After ROB commitment, the store can be sent to memory

The store execution process includes:

1.   Address calculation and TLB lookup
2.   Data generation (getting the data to be stored)
3.   Checking for memory ordering violations with younger loads
4.   After commitment by the ROB, writing to memory

Sources: [src/main/scala/lsu/lsu.scala 784-797](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L784-L797)

## Memory Ordering and Forwarding

The LSU implements memory ordering enforcement through several mechanisms:

### Store-to-Load Forwarding

When a load's address matches an earlier store's address in the STQ, the store's data can be forwarded to the load if available. This avoids having to access memory and provides correct program behavior:

1.   When a load executes, it searches the STQ for address matches with older stores
2.   If a match is found and the store data is available, the data is forwarded
3.   If data isn't available, the load is put to sleep until the store data arrives

Sources: [src/main/scala/lsu/lsu.scala 19-27](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L19-L27)

### Memory Ordering Violations

The LSU detects and handles memory ordering violations:

1.   When a store executes, it searches the LDQ for address matches with younger loads that have already executed
2.   If a match is found, a memory ordering violation has occurred
3.   The pipeline is flushed and execution restarts from the violating store

Sources: [src/main/scala/lsu/lsu.scala 28-36](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L28-L36)[src/main/scala/lsu/lsu.scala 117-127](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L117-L127)

## Vector Memory Operations

The LSU includes specialized support for vector memory operations, coordinating with the Vector Processing Unit (VPU) through the OviWrapper interface.

### Vector Load/Store Flow

Vector memory operations follow a different flow than scalar operations:

1.   Vector instructions are issued to the OviWrapper
2.   The OviWrapper enqueues the operation in the Vector Load/Store Instruction Queue (vLSIQueue)
3.   The VAgen (Vector Address Generator) produces memory addresses
4.   For stores, the VDB (Vector Data Buffer) handles data
5.   The LSU processes vector operations through the DLQ and DSQ queues
6.   Results are returned to the VPU for register file updates

Sources: [src/main/scala/exu/ovi.scala 127-471](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/ovi.scala#L127-L471)[src/main/scala/lsu/lsu.scala 632-695](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L632-L695)

### Vector Load/Store Queue Management

The vector LSU components include special handling for:

*   Vector strided memory accesses
*   Vector indexed memory accesses
*   Vector masked memory operations
*   Vector element tracking for proper register file updates

Vector operations can block scalar operations until they complete, especially when address overlaps might occur. The LSU tracks age relationships between vector and scalar operations to ensure correct execution.

Sources: [src/main/scala/lsu/lsu.scala 492-618](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L492-L618)

## LSU Interfaces

The LSU connects to other processor components through several interfaces:

### Core Interface (LSUCoreIO)

Connects to the core execution units, receiving memory operation requests and returning load data.

Key components:

*   `exe`: Connects to execution units for memory operations
*   `VGen`: Vector generator interface
*   `dis_uops`: Dispatched micro-ops
*   `fp_stdata`: Floating-point store data
*   `commit`: Commitment signals from ROB

Sources: [src/main/scala/lsu/lsu.scala 170-215](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L170-L215)

### Memory Interface (LSUDMemIO)

Connects to the data cache, sending memory requests and receiving responses.

Key components:

*   `req`: Memory request channel
*   `resp`: Memory response channel
*   `nack`: Request rejection channel
*   `release`: Cache line release channel

Sources: [src/main/scala/lsu/lsu.scala 141-166](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L141-L166)

### TLB Interface

Connects to the Data TLB for virtual to physical address translation.

Key components:

*   DTLB module handles address translation
*   TLB miss handling and retry mechanisms

Sources: [src/main/scala/lsu/lsu.scala 444-450](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/lsu/lsu.scala#L444-L450)

## Memory Model Compliance

The LSU implements the RVWMO (RISC-V Weak Memory Ordering) memory consistency model. Key compliance features include:

1.   Relaxed Write→Read ordering (younger loads may execute before older stores)
2.   Maintained Read→Read ordering for same address (loads to same address appear in program order)
3.   Early reading of a thread's own writes (store forwarding)

The memory ordering enforcement mechanisms described earlier ensure compliance with the RVWMO model while allowing high performance through speculation.

## Summary

The Load/Store Unit is a critical component of the RISC-V Ocelot processor, handling all memory operations while maintaining memory ordering correctness. It supports both scalar and vector memory operations through various queue structures and complex ordering logic. The LSU's ability to speculate and forward data helps reduce memory access latency while providing mechanisms to detect and recover from memory ordering violations.
