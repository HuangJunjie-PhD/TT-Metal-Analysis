This page documents the supported Tenstorrent hardware configurations, the test matrix system used to organize tests across different hardware and configuration combinations, and the pytest marker system used to filter and execute tests. For information about how tests are implemented and validated, see [Test Framework](https://deepwiki.com/tenstorrent/tt-blacksmith/9.1-test-framework). For details about CI/CD pipelines that trigger these tests, see [CI/CD Pipelines](https://deepwiki.com/tenstorrent/tt-blacksmith/9.2-cicd-pipelines).

## Overview

TT-Blacksmith supports multiple Tenstorrent hardware configurations and uses a declarative test matrix system to ensure experiments work across different hardware, parallelism strategies, and frameworks. The system uses pytest markers to categorize and filter tests, and JSON preset files to define test suites for different CI/CD contexts (e.g., push vs. nightly uplift).

## Supported Hardware

TT-Blacksmith supports the following Tenstorrent hardware configurations:

| Hardware Type | Chips | Architecture | pytest Marker | Primary Use Case |
| --- | --- | --- | --- | --- |
| N150 | 1 | Blackhole | `n150` | Single-chip training, development |
| N150 High Memory | 1 | Blackhole | `n150` | Single-chip LLM fine-tuning |
| N300 | 2 | Wormhole | `n300` | Data/tensor parallel training |
| N300 LLMBox | 2 | Wormhole | `n300_llmbox` | Data parallel LLM training (quietbox) |
| Galaxy | 32 | Wormhole | `galaxy` | Hybrid parallel large-scale training |

**Hardware Capabilities:**

*   **N150**: Single Blackhole chip suitable for small models (MNIST, 1B parameter LLMs with LoRA). Used as baseline for single-chip validation.
*   **N300**: Two Wormhole chips enabling data parallelism (replicate model, shard data) or tensor parallelism (shard model across chips).
*   **N300 LLMBox**: Specialized quietbox configuration optimized for LLM workloads with 2 chips.
*   **Galaxy**: 32 Wormhole chips supporting hybrid parallelism (both data and tensor parallelism simultaneously) for large models.

## Pytest Marker System

All tests in TT-Blacksmith are categorized using pytest markers. These markers enable fine-grained test filtering and are used by the test matrix system to route tests to appropriate hardware.

### Marker Categories

Sources: [tests/pytest.ini 1-16](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/pytest.ini#L1-L16)[tests/training_test_cases.py 1-280](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/training_test_cases.py#L1-L280)

### Marker Definitions

| Marker | Category | Purpose |
| --- | --- | --- |
| `uplift` | Frequency | Test runs on nightly uplift (daily dependency updates) |
| `push` | Frequency | Test runs on every push to repository |
| `n150` | Hardware | Test compatible with N150 (Blackhole single chip) |
| `n300` | Hardware | Test compatible with N300 (Wormhole 2-chip) |
| `n300_llmbox` | Hardware | Test compatible with N300 quietbox configuration |
| `galaxy` | Hardware | Test compatible with Galaxy (32-chip) |
| `torch` | Framework | Test uses PyTorch framework |
| `jax` | Framework | Test uses JAX framework |
| `single_chip` | Parallelism | Test runs on single chip (no parallelism) |
| `data_parallel` | Parallelism | Test uses data parallelism (replicate model, shard data) |
| `tensor_parallel` | Parallelism | Test uses tensor parallelism (shard model) |
| `split_0` | Split | Test in first split group (for parallel test execution) |
| `split_1` | Split | Test in second split group (for parallel test execution) |

**Split Markers**: Used to divide long-running N150 tests into two groups that can execute in parallel on different runners. For example, Llama tests are in `split_0` while Gemma and Albert are in `split_1`.

## Test Matrix Presets

Test matrix presets are JSON files that define which test combinations to run in different CI/CD contexts. Each preset specifies hardware configurations, frontends, and pytest marker expressions.

### Available Presets

| Preset File | Purpose | Typical Trigger |
| --- | --- | --- |
| `uplift-test.json` | Comprehensive nightly tests after dependency updates | Daily at 06:00 UTC |
| `basic-test.json` | Fast smoke tests for PRs | Push to main, pull requests |
| `regressed-test.json` | Tests for known regressions | Manual trigger |

### Preset Structure

Each preset is a JSON array of test configurations:

`{  "runs-on": "n300",           // CI runner label  "frontend": "xla",           // Environment type (xla/gpu/ffe)  "marks": "data_parallel and uplift and n300 and jax"  // pytest marker expression}`
Sources: [.github/workflows/test-matrix-presets/uplift-test.json 1-10](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/uplift-test.json#L1-L10)[.github/workflows/test-matrix-presets/basic-test.json 1-7](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.github/workflows/test-matrix-presets/basic-test.json#L1-L7)

### Uplift Test Matrix

The uplift test matrix provides comprehensive coverage across all hardware and parallelism strategies:

| runs-on | frontend | marks | Tests |
| --- | --- | --- | --- |
| `n150` | `xla` | `single_chip and uplift and n150 and jax` | JAX single-chip (pure JAX, Flax) |
| `n300` | `xla` | `data_parallel and uplift and n300 and jax` | JAX data parallel |
| `n300` | `xla` | `tensor_parallel and uplift and n300 and jax` | JAX tensor parallel |

The split markers (`split_0`, `split_1`) enable parallel execution of long-running N150 tests by dividing them into two groups.

## Running Tests with Markers

### Filtering by Hardware

`# Run all N150 testspytest -m "n150" tests/ # Run all N300 testspytest -m "n300" tests/ # Run all Galaxy testspytest -m "galaxy" tests/`
### Filtering by Framework

`# Run all PyTorch testspytest -m "torch" tests/ # Run all JAX testspytest -m "jax" tests/`
### Filtering by Parallelism Strategy

`# Run single-chip tests onlypytest -m "single_chip" tests/ # Run data parallel testspytest -m "data_parallel" tests/ # Run tensor parallel testspytest -m "tensor_parallel" tests/`
### Combining Markers

`# Run PyTorch tests on N300 with data parallelismpytest -m "n300 and torch and data_parallel" tests/ # Run JAX single-chip tests on N150pytest -m "n150 and jax and single_chip" tests/ # Run uplift tests for N300 tensor parallelpytest -m "uplift and n300 and tensor_parallel" tests/`
### CI/CD Usage

In CI/CD workflows, the test matrix preset defines the marker expression:

`# From call-test.ymlrun: |  source env/activate --${{ matrix.build.frontend }}  pytest -m '${{ matrix.build.marks }}' tests/`
The `matrix.build.marks` value comes from the JSON preset file, for example:

*   `"single_chip and uplift and n150 and jax"` from uplift-test.json
*   `"data_parallel and push and n300 and torch"` from basic-test.json

## Test Validation

Tests are validated using golden files that store expected loss values. The test framework compares actual training and validation losses against these golden references with a configurable tolerance (default 0.3).

### Golden File Example

Golden files are CSV files with step and loss columns:

`_step,train/loss5,5.39192724227905310,4.49270844459533715,2.990625095367432`
Each test generates two golden files:

*   `{test_id}_train.csv` - Training loss progression
*   `{test_id}_val.csv` - Validation loss progression

### Tolerance and Validation

The validation logic in [tests/test_all.py 29-33](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L29-L33) compares test logs against golden files:

`def assert_loss_with_tolerance(log_file: str, golden_file: str, tolerance: float):    log_df = pd.read_csv(log_file)    golden_df = pd.read_csv(golden_file)    pd.testing.assert_frame_equal(golden_df, log_df, rtol=tolerance)`
Tests can skip loss validation by setting `"skip_loss_checks": True` in their test case dictionary (common for JAX tests that are still under development).

## Test Configuration Override

Tests use a two-level configuration system:

1.   **Experiment Config**: Full training configuration (e.g., `test_llama_3_2_1b_sst2.yaml`)
2.   **Test Config**: Overrides for faster testing (e.g., `test_training_fast.yaml`)

The test configuration typically limits training steps for CI/CD:

`# tests/configs/test_training_fast.yamltest_config:  max_steps_per_epoch: 15 steps_freq: 5       # Log metrics every 5 stepsval_steps_freq: 5   # Run validation every 5 steps use_tt: truelog_on_wandb: false`
This allows full experiments to run for many epochs while CI/CD tests complete quickly with just 15 steps.
