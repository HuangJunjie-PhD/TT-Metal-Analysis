## Purpose and Scope

Operator Testing validates individual operations (unary, binary, n-ary) across diverse configurations including input shapes, data formats, math fidelities, and input sources. The system contains 1000+ test cases covering element-wise operations, tensor manipulation, convolutions, pooling, reductions, and other fundamental operations.

This page focuses on **operator-level testing**. For framework-level integration testing, see [Model Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.3-model-testing). For the underlying test planning abstractions (`TestPlan`, `TestCollection`, `TestVector`), see [Test Planning Framework](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.1-test-planning-framework). For failure classification, see [Failure Classification](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.5-failure-classification-system).

**Sources:**[forge/test/operators/pytorch/eltwise_unary/test_unary.py 1-446](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L1-L446)[forge/test/operators/pytorch/eltwise_binary/test_binary.py 1-545](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_binary/test_binary.py#L1-L545)

## InputSource Types

Tests validate three distinct input source patterns to ensure operators work correctly regardless of how their inputs are provided:

### Diagram: InputSource Patterns

**Sources:**[forge/test/operators/utils/__init__.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/__init__.py) (InputSource enum definition), [forge/test/operators/pytorch/eltwise_unary/test_unary.py 8-20](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L8-L20)

### InputSource Definitions

**`InputSource.FROM_HOST`**: Operator receives input tensors directly from host memory. This pattern validates that operators handle network inputs correctly.

**`InputSource.FROM_ANOTHER_OP`**: Operator receives inputs from intermediate computation results. Pre-processing operations (e.g., `torch.add(x, x)`) feed the target operator to simulate real network execution paths.

**`InputSource.CONST_EVAL_PASS`**: Operator receives constant tensors registered as model buffers. This pattern validates constant folding optimizations where the compiler can evaluate operations at compile time.

**Sources:**[forge/test/operators/pytorch/eltwise_unary/test_unary.py 76-83](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L76-L83)

## Model Wrapper Patterns

Each `InputSource` requires a specific `torch.nn.Module` wrapper that structures computation appropriately:

### Diagram: Model Wrapper Classes

**Sources:**[forge/test/operators/pytorch/eltwise_unary/test_unary.py 73-74](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L73-L74)

### ModelDirect Pattern

Used for `InputSource.FROM_HOST`. Applies operator directly to input tensor:

**Example:** Testing `relu` with direct input creates `ModelDirect(torch.relu, {})` and passes `torch.Tensor` as input.

**Sources:**[forge/test/operators/pytorch/eltwise_unary/test_unary.py 73-74](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L73-L74) (model selection logic)

### ModelFromAnotherOp Pattern

Used for `InputSource.FROM_ANOTHER_OP`. Preprocesses input before applying target operator:

**Example:** Testing `sigmoid` with intermediate input runs `torch.add(x, x)` first, then applies `sigmoid` to the result.

**Sources:**[forge/test/operators/pytorch/eltwise_nary/test_where.py 27-38](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_nary/test_where.py#L27-L38)

### ModelConstEvalPass Pattern

Used for `InputSource.CONST_EVAL_PASS`. Registers constant tensors and applies operator at compile time:

**Example:** Testing `exp` with constant input registers `exp(constant)` which gets evaluated during compilation's constant folding pass.

**Sources:**[forge/test/operators/pytorch/eltwise_unary/test_unary.py 73-74](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L73-L74)[forge/test/operators/pytorch/eltwise_nary/test_where.py 53-86](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_nary/test_where.py#L53-L86)

## Failing Rules System

Operators often have known failures that are expected until bugs are fixed. The failing rules system marks these tests with `xfail` or `skip` markers.

### Diagram: Failing Rules Application

**Sources:**[forge/test/operators/pytorch/eltwise_binary/failing_rules.py 1-206](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_binary/failing_rules.py#L1-L206)

### Defining Failing Rules

Failing rules are `TestCollection` objects with `failing_reason` or `skip_reason` attributes:

**Sources:**[forge/test/operators/pytorch/eltwise_binary/failing_rules.py 34-46](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_binary/failing_rules.py#L34-L46)

### Operator-Specific Failing Reasons

Each operator maintains a register of detected failing reasons:

**Sources:**[forge/test/operators/pytorch/ids/failing_reasons_register.py 10-134](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/ids/failing_reasons_register.py#L10-L134)

### Rule Matching Algorithm

The test framework matches each `TestVector` against failing rules:

1.   For each failing rule collection:

    *   Check if `operator` matches
    *   Check if `input_source` matches
    *   Check if `input_shape` matches
    *   Check if `dev_data_format` matches
    *   Check if `math_fidelity` matches

2.   If all specified fields match:

    *   Apply `@pytest.mark.xfail(reason=failing_reason)` if `failing_reason` is set
    *   Apply `@pytest.mark.skip(reason=skip_reason)` if `skip_reason` is set

3.   If no rules match:

    *   Execute test normally

**Sources:**[forge/test/operators/utils/test_plan.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/test_plan.py) (rule matching implementation)

## Test Duration Tracking

The `.test_durations` file records historical test execution times for intelligent test splitting across CI runners:

### File Format

Each entry maps a test identifier to execution time in seconds.

**Sources:**[.test_durations 1-100](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.test_durations#L1-L100)

### Usage in CI/CD

The CI pipeline uses duration data to:

1.   **Split Tests Evenly**: Distribute tests across multiple runners to minimize total runtime
2.   **Prioritize Fast Tests**: Run quick tests first for faster feedback
3.   **Identify Slow Tests**: Flag tests with execution time > threshold for optimization

Example distribution for 4 runners:

| Runner | Test Count | Total Duration |
| --- | --- | --- |
| Runner 1 | 250 tests | ~300 seconds |
| Runner 2 | 250 tests | ~295 seconds |
| Runner 3 | 250 tests | ~310 seconds |
| Runner 4 | 250 tests | ~305 seconds |

**Sources:**[.test_durations 1-1000](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.test_durations#L1-L1000)
