This document covers the Python package structure for `tt-burnin` and its dynamic versioning system. It explains how the package is organized as a Python module and how version information is dynamically resolved at runtime to support both development and distribution environments.

## Dynamic Versioning System

The package implements a sophisticated dynamic versioning system through the `__init__.py` module that automatically detects the correct version depending on whether the package is running from an installed distribution or a development environment.

### Version Resolution Architecture

**Sources:**[tt_burnin/__init__.py 18-51](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L18-L51)[tt_burnin/__init__.py 54-59](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L54-L59)

### Version Detection Implementation

The version detection system uses a multi-layered approach implemented in the `__get_package_version()` function:

| Priority | Method | Description | Indicator |
| --- | --- | --- | --- |
| 1 | Cached | Previously resolved version | No modification |
| 2 | Distribution | `importlib_metadata.version()` | Clean version |
| 3 | Development | `pyproject.toml` parsing | Version with `+` suffix |

The system uses lazy evaluation through the `__getattr__()` method, which intercepts attribute access for `__version__` and delegates to the version resolution function.

**Sources:**[tt_burnin/__init__.py 54-59](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L54-L59)[tt_burnin/__init__.py 18-51](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L18-L51)

## Version Source Mechanisms

### Distribution Environment Detection

When the package is installed from a distribution (PyPI, Debian package, etc.), the system attempts to resolve the version using `importlib_metadata.version("tt-burnin")`. This approach leverages Python's standard metadata system for installed packages.

**Sources:**[tt_burnin/__init__.py 28-32](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L28-L32)

### Development Environment Fallback

When running in a development environment where the package hasn't been installed, the system falls back to reading version information directly from `pyproject.toml`. This fallback includes several safeguards:

*   **TOML Library Compatibility**: Handles both `tomli` (external) and `tomllib` (Python 3.11+)
*   **File Existence Check**: Verifies `pyproject.toml` exists before attempting to read
*   **Development Indicator**: Appends `+` to indicate potential local modifications

**Sources:**[tt_burnin/__init__.py 36-50](https://github.com/tenstorrent/tt-burnin/blob/809f293d/tt_burnin/__init__.py#L36-L50)

## Import and Compatibility Handling

The versioning system includes robust import handling to support different Python versions and installation scenarios:

### TOML Library Resolution

The system first attempts to import the `tomli` library (required for Python < 3.11), then falls back to the built-in `tomllib` module available in Python 3.11+.

## Version String Format

The package produces different version string formats depending on the detection method:

| Source | Format | Example | Use Case |
| --- | --- | --- | --- |
| Distribution | `x.y.z` | `1.2.3` | Installed package |
| Development | `x.y.z+` | `1.2.3+` | Local development |
| Unknown | `"unknown"` | `unknown` | Fallback case |

The `+` suffix in development environments indicates that the version may include uncommitted changes or local modifications not present in the official release.
