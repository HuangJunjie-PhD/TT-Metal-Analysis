## Purpose and Scope

This page documents Vector Unit (SFPU) instructions that transfer data between register files and perform data layout transformations. The primary data movement patterns are:

*   **`Dst` ↔ `LReg`:** Interface between Matrix Unit results and Vector Unit processing
*   **`LReg` ↔ `LReg`:** Intra-Vector-Unit data reorganization and comparison operations
*   **Immediate → `LReg`:** Constant loading

These instructions execute in the Load sub-unit, Store sub-unit, or Simple sub-unit of the Vector Unit backend.

For Vector Unit computational instructions, see [Vector Unit Core Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.3-vector-unit-core-instructions). For Vector Unit configuration and control, see [Vector Unit Configuration and Control Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.5-vector-unit-configuration-and-control-instructions).

## Instruction Summary

| Instruction | Backend Unit | Data Path | Format Conversion | Lane Predication | RWC Update |
| --- | --- | --- | --- | --- | --- |
| `SFPLOAD` | Load | `Dst` → `LReg[VD]` | FP16/BF16/FP32/INT8/INT16/INT32 → FP32/SignMag | Yes | Via `AddrMod` |
| `SFPLOADI` | Load | `Imm16` → `LReg[VD]` | FP16/INT/RAW → FP32 | Yes | No |
| `SFPLOADMACRO` | Load | `Dst` → `LReg[VD]` + schedule | Same as `SFPLOAD` | Yes | Via `AddrMod` |
| `SFPSTORE` | Store | `LReg[VD]` → `Dst` | FP32/SignMag → FP16/BF16/FP32/INT8/INT16/INT32 | Yes | Via `AddrMod` |
| `SFPMOV` | Simple | `LReg[VC]` → `LReg[VD]` | None (bitcast) | Yes | No |
| `SFPCAST` | Simple | `LReg[VC]` → `LReg[VD]` | FP32 ↔ SignMag ↔ TwosComp | Yes | No |
| `SFPSWAP` | Simple | `LReg[VC]` ↔ `LReg[VD]` | None | Yes | No |

**Key abbreviations:**

*   **SignMag:** Sign-magnitude integer format (1 sign bit + 31 magnitude bits)
*   **TwosComp:** Two's complement integer format
*   **RWC:** Read/Write Counter (addressing state for `Dst`, `SrcA`, `SrcB`)

## SFPLOAD: Load from Dst to LReg

### Overview

`SFPLOAD` moves up to 32 datums from four consecutive rows of `Dst` (even or odd columns) to a single `LReg` register. The instruction performs automatic format conversion from `Dst` data types (FP16, BF16, FP32, Integer "8", Integer "16", Integer "32") to `LReg` data types (FP32 or sign-magnitude integers).

### Cross-Lane Data Movement Pattern

Each `Dst` row contributes 8 datums to 8 consecutive lanes of `LReg[VD]`. The `AddrMod` field controls whether even columns (0, 2, 4, ..., 14) or odd columns (1, 3, 5, ..., 15) are selected.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 54-59](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L54-L59)

### Supported Format Conversions

The `Mod0` field specifies the data type conversion:

| Mode | Source (`Dst` type) | Destination (`LReg` type) | Notes |
| --- | --- | --- | --- |
| `MOD0_FMT_FP16` | FP16 | FP32 | Configurable infinity handling via `LaneConfig.ENABLE_FP16A_INF` |
| `MOD0_FMT_BF16` | BF16 | FP32 | Standard BF16 to FP32 extension |
| `MOD0_FMT_FP32` | FP32 or Integer "32" | FP32 or sign-magnitude | Preserves bit pattern |
| `MOD0_FMT_INT32` | FP32 or Integer "32" | FP32 or sign-magnitude | Identical to `MOD0_FMT_FP32` |
| `MOD0_FMT_INT8` | Integer "8" (±255) | Sign-magnitude | Blackhole range increased from Wormhole's ±127 |
| `MOD0_FMT_INT16` | Integer "16" | Sign-magnitude | 16-bit sign-magnitude conversion |
| `MOD0_FMT_UINT16` | Integer "16" (opaque) | Unsigned (zero-extend) | Treats as opaque 16-bit value |
| `MOD0_FMT_LO16_ONLY` | Integer "16" (opaque) | Unsigned | Writes low 16 bits, preserves high 16 |
| `MOD0_FMT_HI16_ONLY` | Integer "16" (opaque) | Unsigned | Writes high 16 bits, preserves low 16 |
| `MOD0_FMT_SRCB` | (resolved at runtime) | (resolved at runtime) | Resolves to FP16/BF16/FP32 based on config |

**Note:** In Blackhole, `MOD0_FMT_INT8_COMP` and `MOD0_FMT_INT32_SM` are deprecated and no longer perform two's complement conversions.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 26-50](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L26-L50)

### Address Calculation

The `Dst` address is computed as follows:

*   **Bits [9:2]** of `Addr` select the aligned group of four `Dst` rows
*   **Bit [1]** selects even (0) or odd (1) columns
*   **Bit [0]** is unused
*   `AddrMod` field advances RWCs (Dst/SrcA/SrcB) after the load completes

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 81-95](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L81-L95)

### Instruction Scheduling

At least **three unrelated Tensix instructions** must execute between:

1.   A Matrix Unit (FPU) instruction that writes to `Dst`
2.   An `SFPLOAD` instruction that reads that same region of `Dst`

This requirement exists because Matrix Unit write operations have a multi-cycle pipeline. Software can insert NOPs or use [`STALLWAIT`](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend) (block B8, condition C7), though explicit instruction insertion is recommended.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 214-217](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L214-L217)

## SFPLOADI: Load Immediate to LReg

### Overview

`SFPLOADI` loads a 16-bit immediate value into all 32 lanes of an `LReg` register. The immediate value is interpreted and optionally converted based on the `Mod0` field.

**Syntax:**

The `Mod0` field determines how the 16-bit immediate is interpreted:

*   Format conversion modes (similar to `SFPLOAD`)
*   Lane mask modes (for conditional execution setup)
*   Special constant modes

This instruction provides a fast path for initializing `LReg` registers with constants without requiring a `Dst` read.

**Sources:**[Diagrams/Src/Bits32.lua 965-972](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/Bits32.lua#L965-L972)

## SFPCAST: Type Conversion Within LReg

`SFPCAST` performs type conversions within the `LReg` register file, operating lanewise. The `Mod1` field selects the conversion type.

**Syntax:**

**Supported conversions (selected via `Mod1`):**

| `Mod1` | Conversion | Description |
| --- | --- | --- |
| 0 | FP32 → SignMag | Reinterpret FP32 as sign-magnitude integer |
| 1 | SignMag → FP32 | Reinterpret sign-magnitude integer as FP32 |
| 2 | TwosComp → SignMag | Convert two's complement to sign-magnitude |
| 3 | SignMag → TwosComp | Convert sign-magnitude to two's complement |
| 4-15 | Reserved | Behavior undefined |

**Functional model (example for `Mod1 = 2`):**

This instruction is critical in Blackhole A0 because `SFPLOAD`/`SFPSTORE` with `MOD0_FMT_INT8_COMP` and `MOD0_FMT_INT32_SM` were deprecated. Software must now explicitly use `SFPCAST` for these conversions.

### Configuration via SFPCONFIG

The `LoadMacroConfig` state is written using [`SFPCONFIG`](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.5-vector-unit-configuration-and-control-instructions) instructions:

Each `Sequence[MacroIndex]` byte encodes:

*   **Bits [2:0]:** Instruction selection (0=none, 1=undef, 2=NOP, 3=SFPSTORE, 4-7=template 0-3)
*   **Bits [5:3]:** Delay (0-6 cycles/instructions, 7=cancel)
*   **Bit [6]:** Result destination (`VD` override: 0=use load's VD, 1=use VD=16)
*   **Bit [7]:** Input source (for non-Store sub-units: 0=VC from load, 1=VB from load)

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 49-61](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L49-L61)[WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md 133-135](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md?plain=1#L133-L135)

`SFPSWAP` performs lanewise comparison between `LReg[VC]` and `LReg[VD]`, conditionally swapping the values based on the comparison result. This implements min/max or argmin/argmax operations.

**Basic operation:**

**Argmin/argmax mode:**

| 2 | `ENABLE_DEST_INDEX` | Enables argmin/argmax (index tracking) mode |
| 3 | `CAPTURE_DEFAULT_DEST_INDEX` | If true with `ENABLE_DEST_INDEX`, captures index to additional register |
| 8 | `EXCHANGE_SRCB_SRCC` | If true, inverts comparison (computes max/min instead of min/max) |

**Execution:** Simple sub-unit, 1 cycle latency

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md 115-132](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md?plain=1#L115-L132)

## Addressing Modes and RWC Updates

### AddrMod Field

The 3-bit `AddrMod` field (2-bit in Wormhole) in `SFPLOAD`, `SFPSTORE`, and `SFPLOADMACRO` controls how Read/Write Counters (RWCs) advance after the instruction completes:

Common patterns:

*   **No increment:** RWC stays at current position
*   **+1:** Advance to next group of 4 rows
*   **+Cr:** Advance by configured increment (from `SETRWC` or `INCRWC`)
*   **+CtoCr:** Special mode for dual-buffer addressing

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 133-137](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L133-L137)[BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md 131-133](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md?plain=1#L131-L133)

## Architectural Differences: Wormhole vs Blackhole

### Blackhole Enhancements

**SFPLOAD changes:**

*   `MOD0_FMT_INT32_ALL`: Overhauled stack-like addressing mode
*   `MOD0_FMT_INT8`: Range increased from ±127 to ±255
*   `MOD0_FMT_INT8_COMP` and `MOD0_FMT_INT32_SM` deprecated (no longer convert from two's complement)
*   `AddrMod` encoding expanded from 2 bits to 3 bits

**SFPSTORE changes:**

*   `MOD0_FMT_FP32`: Now flushes denormals to signed zero (was identity in Wormhole)
*   `MOD0_FMT_SRCB`: Behavior changed due to `MOD0_FMT_FP32` change
*   `MOD0_FMT_INT8_COMP` and `MOD0_FMT_INT32_SM` deprecated
*   `MOD0_FMT_INT32_ALL`: Overhauled addressing mode
*   `AddrMod` encoding expanded from 2 bits to 3 bits

**SFPLOADMACRO changes:**

*   `AddrMod` encoding expanded from 2 bits to 3 bits
*   Additional instructions now schedulable: `SFPARECIP`, `SFPGT`, `SFPLE`, `SFPMUL24`

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 7-14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L7-L14)[BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md 8-15](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md?plain=1#L8-L15)

## Programming Considerations

### Data Hazard Timing

**Critical timing constraint:** At least **3 unrelated Tensix instructions** must execute between:

1.   Matrix Unit instruction writing to `Dst`
2.   `SFPLOAD` or `SFPLOADMACRO` reading that region of `Dst`

This applies to both Wormhole and Blackhole. Software should prefer explicit instruction insertion over `STALLWAIT`.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md 214-217](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOAD.md?plain=1#L214-L217)[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 151-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L151-L156)

### SFPLOADMACRO Scheduling Conflicts

When using `SFPLOADMACRO`, if a scheduled instruction arrives at a sub-unit on the same cycle as a regular Vector Unit instruction issued by software, the **scheduled instruction takes priority** and the regular instruction is silently discarded. Software must carefully track outstanding scheduled instructions to avoid conflicts.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 149-150](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L149-L150)

### LaneConfig Backdoor Load Behavior

When `LaneConfig.DISABLE_BACKDOOR_LOAD` is false (default), instructions with `VD >= 12` are treated as writes to `LoadMacroConfig.InstructionTemplate[VD - 12]` rather than executing the intended operation. This "backdoor" mechanism allows efficient configuration but requires care when using:

*   `SFPSTORE` with `VD >= 12`
*   `SFPSWAP` with `VD >= 12`

Software should set `DISABLE_BACKDOOR_LOAD` to true if using these instruction/register combinations.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md 117-121](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPCONFIG.md?plain=1#L117-L121)

### Format Conversion Precision Loss

When converting from FP32 to lower-precision formats:

*   **FP32 → FP16:** Denormals flushed to zero, mantissa truncated toward zero, overflow saturates
*   **FP32 → BF16:** Denormals flushed to zero, mantissa truncated toward zero
*   NaN inputs to FP16 conversion become infinity (software should avoid NaN inputs)

For better rounding control, use [`SFPSTOCHRND`](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.5-vector-unit-configuration-and-control-instructions) or [`SFPCAST`](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.5-vector-unit-configuration-and-control-instructions) before `SFPSTORE`.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md 50-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPSTORE.md?plain=1#L50-L51)
