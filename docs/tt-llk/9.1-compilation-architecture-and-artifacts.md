## Purpose and Scope

This page documents the architecture of the LLK compilation system, explaining how test kernels are compiled into ELF binaries and how build artifacts are organized. It covers the two-tier artifact structure (shared vs variant-specific), directory organization, and the build coordination mechanisms that enable parallel compilation. For details on compilation commands and linking, see [Kernel Compilation and Linking](https://deepwiki.com/tenstorrent/tt-llk/9.3-kernel-compilation-and-linking). For toolchain setup, see [Environment Setup and SFPI Toolchain](https://deepwiki.com/tenstorrent/tt-llk/9.2-environment-setup-and-sfpi-toolchain).

## Overview of Compilation Architecture

The LLK build system uses a **two-tier artifact architecture** to optimize compilation time when running thousands of test variants. Each test variant represents a unique combination of template parameters, runtime parameters, and data formats.

**Sources:**[tests/python_tests/helpers/test_config.py 119-127](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L119-L127)[tests/python_tests/helpers/test_config.py 263-273](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L263-L273)

### Key Design Principles

1.   **Shared artifacts** are built once per build session and reused across all test variants
2.   **Variant-specific artifacts** are built per unique configuration and cached by hash
3.   **File locking** prevents concurrent builds of the same artifact
4.   **Marker files** (`.build_complete`, `.shared_complete`) indicate completion
5.   **Profiler builds** maintain separate shared artifacts because `trisc.cpp` compiles differently with `-DLLK_PROFILER`

## Artifact Directory Structure

The build system uses `/tmp/tt-llk-build/` as the root artifact directory by default.

**Sources:**[tests/python_tests/helpers/test_config.py 263-273](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L263-L273)[tests/python_tests/helpers/test_config.py 276-294](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L276-L294)

### Directory Definitions

| Directory | Purpose | Created When |
| --- | --- | --- |
| `ARTEFACTS_DIR` | Root build directory | Always (default: `/tmp/tt-llk-build/`) |
| `SHARED_DIR` | Shared artifacts for non-profiler builds | First test build |
| `SHARED_OBJ_DIR` | Shared object files | First test build |
| `SHARED_ELF_DIR` | Shared ELF binaries | First test build |
| `PROFILER_SHARED_DIR` | Shared artifacts for profiler builds | First profiler test |
| `COVERAGE_INFO_DIR` | Merged coverage `.info` files | Coverage build |
| `SYNC_DIR` | FileLock synchronization files | First parallel build |
| `PERF_DATA_DIR` | Performance data CSVs | Performance tests |
| `{test_name}/{variant_id}/` | Test variant-specific artifacts | Per variant |

**Sources:**[tests/python_tests/helpers/test_config.py 119-127](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L119-L127)[tests/python_tests/helpers/test_config.py 263-273](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L263-L273)

## Shared Artifacts

Shared artifacts are compiled once per build session and reused across all test variants. This significantly reduces compilation time since these components do not depend on test-specific parameters.

### Shared Object Files

**Sources:**[tests/python_tests/helpers/test_config.py 681-722](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L681-L722)

### Shared Artifact Compilation Details

Each shared artifact is compiled with specific flags:

1.   **tmu-crt0.o** - Assembly startup code

    *   Compiled with: `ARCH_NON_COMPUTE` (e.g., `-mcpu=tt-wh`)
    *   Source: [tests/helpers/tmu-crt0.S](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/tmu-crt0.S)
    *   Used by: All RISC cores for initialization

2.   **brisc.o** - BRISC control processor code

    *   Compiled with: `ARCH_NON_COMPUTE` + `-DCOVERAGE` (if enabled)
    *   Source: [tests/helpers/src/brisc.cpp](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/src/brisc.cpp)
    *   Skipped for: Quasar architecture (uses TRISC boot mode)

3.   **main_unpack.o, main_math.o, main_pack.o** - TRISC kernel wrappers

    *   Compiled with: `ARCH_COMPUTE` (e.g., `-mcpu=tt-wh-tensix`)
    *   Source: [tests/helpers/src/trisc.cpp](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/src/trisc.cpp) (same file, different `-DCOMPILE_FOR_TRISC` values)
    *   Includes: `-DLLK_PROFILER` for profiler builds
    *   Includes: `-DRUNTIME_FORMATS` if runtime format configuration enabled

4.   **coverage.o** - Coverage infrastructure

    *   Compiled with: `ARCH_NON_COMPUTE` + `-fno-strict-aliasing`
    *   Source: [tests/helpers/src/coverage.cpp](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/src/coverage.cpp)
    *   Only built when: `--coverage` flag is set

5.   **brisc.elf** - BRISC executable

    *   Links: `tmu-crt0.o` + `brisc.o` + `coverage.o` (if coverage)
    *   Linker scripts: `memory.{arch}.ld`, `brisc.ld`, `sections.ld`

**Sources:**[tests/python_tests/helpers/test_config.py 681-729](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L681-L729)

## Build Coordination and Locking

The build system uses file-based locking to coordinate parallel compilation across pytest workers.

**Sources:**[tests/python_tests/helpers/test_config.py 872-1002](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L872-L1002)

### Locking Strategy

The build system uses three types of locks:

1.   **Shared artifact locks**

    *   `/tmp/tt-llk-build-shared.lock` - Non-profiler shared artifacts
    *   `/tmp/tt-llk-build-shared-profiler.lock` - Profiler shared artifacts
    *   Prevents concurrent builds of shared artifacts

2.   **Variant-specific locks**

    *   `SYNC_DIR/{variant_id}.lock` - Per-variant locks
    *   Prevents duplicate builds of the same variant
    *   One lock per unique variant ID

3.   **Lock implementation**

    *   Uses `filelock.FileLock` from the `filelock` library
    *   Context manager ensures lock release even on exceptions
    *   Double-check pattern minimizes lock contention

**Sources:**[tests/python_tests/helpers/test_config.py 636-736](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L636-L736)[tests/python_tests/helpers/test_config.py 872-1002](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L872-L1002)

### CI/CD Workflow Integration

The PRODUCE/CONSUME pattern is used in CI to separate compilation (CPU-intensive) from execution (requires hardware):

**Sources:**[.github/workflows/setup-and-test.yml 137-150](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L137-L150)

### Mode Behavior Details

| Mode | build_elfs() | Hardware Execution | Golden Generator | Use Case |
| --- | --- | --- | --- | --- |
| DEFAULT | Runs normally | Executes test | Real golden generator | Local development |
| PRODUCE | Runs normally | Skipped | Dummy generator | CI compilation phase |
| CONSUME | Skipped (uses existing) | Executes test | Real golden generator | CI execution phase |

**Sources:**[tests/python_tests/helpers/test_config.py 342-363](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L342-L363)[tests/python_tests/helpers/test_config.py 84-96](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L84-L96)

The dummy golden generator in PRODUCE mode returns dummy data because no validation occurs during compilation-only mode. This allows tests to "pass" the compilation phase quickly.
