## Purpose and Scope

This page explains the separation between **compute threads** and **data movement threads** in tt-lang kernels. These thread types execute concurrently on distinct hardware units within a Tenstorrent Tensix core and communicate through Dataflow Buffers (DFBs). This architecture models the physical separation between mathematical execution units (TRISC) and the Network-on-Chip (NOC) data movement engines (BRISC/NCRISC).

Understanding this thread separation is essential for writing correct tt-lang kernels, as each thread type has different responsibilities, available operations, and access patterns to memory.

## Thread Type Overview

tt-lang kernels are composed of threads defined using specific decorators inside a `@ttl.kernel` function. The `TTLGenericCompiler` identifies these threads during AST traversal and assigns them a `kernel_type` based on the decorator used [python/ttl/_src/ttl_ast.py 128-135](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L128-L135)

| Thread Type | DSL Decorator | Hardware Mapping | Primary Responsibility | Key Code Entities |
| --- | --- | --- | --- | --- |
| **Compute** | `@ttl.compute()` | TRISC (Tensix MATH) | Mathematical operations on tiles | `ttl.math`, `ttl.tile_add`, `ttl.tile_mul` |
| **Data Movement** | `@ttl.datamovement()` | BRISC / NCRISC (NOC) | Transfers between L1 and DRAM/NOC | `ttl.copy`, `noc_async_read_tile` |

These threads execute concurrently. In the functional simulator, this is implemented using cooperative multitasking where threads yield control at blocking operations like `wait()`, `reserve()`, or `tx.wait()`. The compiler tracks these threads via a `_thread_registry` during the decoration process [python/ttl/ttl_api.py 100-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L100-L118)

## Thread Communication and Dataflow

The programming model requires explicit synchronization between threads using Dataflow Buffers (circular buffers in L1 memory). Compute threads consume data produced by Data Movement threads and vice versa.

**Diagram: Thread Communication Pattern**

### Key Interaction Protocol:

1.   **Reader (DM)**: Calls `reserve()` to get a `TensorBlock`, uses `ttl.copy()` to move data from a `ttnn.Tensor` into L1, and calls `push()` to signal the Compute thread [python/ttl/operators.py 96-97](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/operators.py#L96-L97)
2.   **Compute**: Calls `wait()` on the input DFB to block until data is ready, performs math (e.g., `ttl.tile_add`), calls `reserve()` on the output DFB, writes results via `store()`, and calls `push()`[python/ttl/operators.py 122-212](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/operators.py#L122-L212)
3.   **Writer (DM)**: Calls `wait()` on the output DFB, uses `ttl.copy()` to move data from L1 back to the destination `ttnn.Tensor`, and calls `pop()` to release the buffer space.

The compiler automatically inserts missing `cb_push`/`cb_pop` or `ttl.wait` operations during the `TTLInsertCBSync` and `TTLInsertCopyWait` passes if they are not explicitly provided in the DSL [include/ttlang/Dialect/TTL/Passes.td 6-26](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L6-L26)[include/ttlang/Dialect/TTL/Passes.td 108-118](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L108-L118)

## Compute Threads

Compute threads are defined using `@ttl.compute()` and are the only threads permitted to perform complex tile-based mathematics. They are lowered to TRISC kernels.

### Characteristics

*   **Math Operations**: Access to tile-based math such as `ttl.add`, `ttl.mul`, and matrix multiplication `ttl.matmul`[python/ttl/operators.py 135-208](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/operators.py#L135-L208)
*   **Register Access**: Operations move data from L1 (Circular Buffers) to DST registers for computation. The `TTLAssignDST` pass performs register assignment [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L36)
*   **Fusion**: Multiple element-wise operations are fused into single `ttl.compute` operations via the `ConvertTTLToCompute` pass [lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp 27-146](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp#L27-L146)

### Compute Thread Implementation

In the MLIR representation, a compute thread is identified by the `ttl.kernel_thread` attribute set to `compute`. It uses hardware-specific abstractions like `tile_regs_acquire` and `tile_regs_release` to guard math operations. During lowering, the compiler traces operand indices to generate `tensor.extract` ops for each input [lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 133-153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L133-L153)

## Data Movement Threads

Data movement threads are defined using `@ttl.datamovement()` and handle the "plumbing" of the kernel. They are mapped to the BRISC and NCRISC processors.

### Characteristics

*   **Copy Engine**: Use `ttl.copy()` to interface with the NOC for DRAM or inter-core transfers [include/ttlang/Dialect/TTL/IR/TTLOps.td 122-162](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L122-L162)
*   **NOC Operations**: Lowered to `TTKernel` NOC operations. The `ConvertTTLToTTKernel` pass transforms `ttl.copy` into asynchronous NOC transfers [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 146-154](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L146-L154)
*   **Synchronization**: Uses `ttl.wait` which blocks until the asynchronous transfer identified by a `!ttl.transfer_handle` is complete [include/ttlang/Dialect/TTL/IR/TTLOps.td 164-172](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L164-L172)

### Data Movement Lowering

Data movement threads iterate through tensor tiles using `ttl.tensor_slice` to create views at specific tile positions [include/ttlang/Dialect/TTL/IR/TTLOps.td 80-120](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L80-L120) These slices are then passed to `ttl.copy`. The compiler converts these into `TTKernel` address calculations and NOC commands [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 161-175](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L161-L175)

## Summary of Differences

| Feature | Compute Thread (`@ttl.compute`) | Data Movement Thread (`@ttl.datamovement`) |
| --- | --- | --- |
| **Hardware** | TRISC (MATH) | BRISC / NCRISC (NOC) |
| **Key Instructions** | `add_tiles`, `mul_tiles`, `matmul` | `noc_async_read_tile`, `noc_async_write_tile` |
| **Lowering Pass** | `ConvertTTLToCompute` | `ConvertTTLToTTKernel` |
| **Buffer Role** | Consumer (Input) / Producer (Output) | Producer (Input) / Consumer (Output) |
| **Sync Mechanism** | `cb_wait`, `cb_reserve` | `ttl.wait`, `cb_push`, `cb_pop` |
| **Register Usage** | DST Register File (via `TTLAssignDST`) | N/A (Direct L1 Access) |
