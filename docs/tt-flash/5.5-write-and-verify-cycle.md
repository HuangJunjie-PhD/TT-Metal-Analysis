# Write and Verify Cycle

Relevant source files
*   [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py)

## Purpose and Scope

This document describes the write and verify cycle used during the SPI flash programming stage of firmware updates. After firmware images and parameters have been prepared (see [Parameter Handling and Masks](https://deepwiki.com/tenstorrent/tt-flash/5.4-parameter-handling-and-masks)), the system must write data to SPI flash and verify it was written correctly. This cycle includes critical SIGINT protection, automatic retry logic, and detailed progress reporting.

For information about the overall flash stages, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages). For post-flash reset procedures, see [Reset Procedures](https://deepwiki.com/tenstorrent/tt-flash/5.6-reset-procedures).

* * *

## Overview

The write and verify cycle is implemented in the `flash_chip_stage2` function, which receives prepared `FlashData` objects containing lists of `FlashWrite` operations. The cycle follows this pattern:

1.   **Write Phase**: Write firmware data to SPI flash with SIGINT protection
2.   **Initial Verify**: Read back and compare data byte-by-byte
3.   **Retry (if needed)**: If verification fails, repeat write and verify once
4.   **Final Status**: Report success or critical failure

**Sources**: [tt_flash/flash.py 495-651](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L495-L651)

## SIGINT Protection Mechanism

During write and verify operations, the system must not be interrupted to prevent SPI flash corruption. The protection mechanism uses Python's `signal` module to temporarily block Ctrl-C interrupts.

### Signal Handler Implementation

The `signal_handler` function at [tt_flash/flash.py 500-501](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L500-L501) prints a warning message but does not raise an exception, effectively blocking the interrupt during critical operations.

### Protected Operation Pattern

Both `perform_write` and `perform_verify` follow the same protection pattern:

| Step | Action | Line Reference |
| --- | --- | --- |
| 1. Save original handler | `original_sigint_handler = signal.getsignal(signal.SIGINT)` | [tt_flash/flash.py 504](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L504-L504) |
| 2. Install protective handler | `signal.signal(signal.SIGINT, signal_handler)` | [tt_flash/flash.py 505](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L505-L505) |
| 3. Perform critical operations | Loop through writes/reads | [tt_flash/flash.py 508-509](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L508-L509) |
| 4. Restore original handler | `signal.signal(signal.SIGINT, original_sigint_handler)` | [tt_flash/flash.py 511](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L511-L511) |

The handler restoration occurs in a `finally` block, ensuring it executes even if exceptions occur during the critical operations.

**Sources**: [tt_flash/flash.py 500-511](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L500-L511)[tt_flash/flash.py 514-531](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L514-L531)

## Write Phase

The `perform_write` function executes the actual SPI flash writes. It receives a list of `FlashWrite` objects, each containing an offset and data payload.

### Write Operation Structure

The function iterates through all `FlashWrite` objects and calls `chip.spi_write(write.offset, write.write)` for each entry at [tt_flash/flash.py 508-509](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L508-L509) The underlying `spi_write` implementation in the chip classes handles the low-level hardware communication through pyluwen.

**Sources**: [tt_flash/flash.py 503-511](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L503-L511)

## Verify Phase

The `perform_verify` function reads back data from SPI flash and compares it byte-by-byte with the expected data. It returns detailed mismatch information if verification fails.

### Verification Logic

### Mismatch Detection Algorithm

At [tt_flash/flash.py 521-528](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L521-L528) the verification function performs detailed analysis when a mismatch is detected:

1.   **Byte-by-byte comparison**: Iterate through `zip(base_data, write.write)`
2.   **First mismatch tracking**: Record the index of the first byte that differs
3.   **Total count**: Increment counter for every mismatched byte
4.   **Early return**: Return tuple `(first_mismatch, mismatch_count)` immediately

The function returns `None` at [tt_flash/flash.py 533](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L533-L533) if all verifications pass successfully.

**Sources**: [tt_flash/flash.py 513-533](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L513-L533)

## Retry Logic

The write/verify cycle includes automatic retry logic to handle transient SPI flash errors. The system attempts one additional write/verify cycle if the initial verification fails.

### Retry State Machine

### Retry Implementation Details

| Phase | Action | Output | Line Reference |
| --- | --- | --- | --- |
| Initial write | `perform_write(chip, data.write)` | Progress message | [tt_flash/flash.py 544](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L544-L544) |
| Initial verify | `perform_verify(chip, data.write)` | Success or failure | [tt_flash/flash.py 560](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L560-L560) |
| Report failure | Print mismatch details | First mismatch, count | [tt_flash/flash.py 566-570](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L566-L570) |
| Retry write | `perform_write(chip, data.write)` | Progress message | [tt_flash/flash.py 583](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L583-L583) |
| Retry verify | `perform_verify(chip, data.write)` | Success or failure | [tt_flash/flash.py 599](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L599-L599) |
| Critical failure | Return `None` | Contact support message | [tt_flash/flash.py 606-611](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L606-L611) |

If the second verification also fails at [tt_flash/flash.py 600-611](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L600-L611) the function returns `None`, indicating a critical error that requires manual intervention. The error message instructs the user not to reset or power off the board and to contact support.

## Error Detection and Reporting

When verification fails, the system provides detailed diagnostic information to aid troubleshooting.

### Mismatch Information Structure

At [tt_flash/flash.py 562-570](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L562-L570) the system reports:

```
Initial verification: failed
    First Mismatch at: <offset>
    Found <count> mismatches
```

After retry failure at [tt_flash/flash.py 601-610](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L601-L610) the system reports:

```
Second verification failed, please do not reset or poweroff the board and contact support for further assistance.
    First Mismatch at: <offset>
    Found <count> mismatches
```

### Color-Coded Messages

The system uses `CConfig.COLOR` constants for visual feedback:

| Status | Color | Code Reference |
| --- | --- | --- |
| Success | GREEN | [tt_flash/flash.py 549](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L549-L549)[tt_flash/flash.py 616](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L616-L616) |
| Failure | RED | [tt_flash/flash.py 567](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L567-L567)[tt_flash/flash.py 606](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L606-L606) |

### Return Values

The `flash_chip_stage2` function returns:

*   `bool` (True/False): For NEBULA_X2 boards indicating if local-to-remote SPI copy was triggered at [tt_flash/flash.py 619-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L619-L650)
*   `None`: Indicates critical failure requiring manual intervention at [tt_flash/flash.py 611](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L611-L611)

The caller in `flash_chips` increments an error counter `rc` when `None` is returned at [tt_flash/flash.py 878-881](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L878-L881) ultimately determining overall flash success or failure.

**Sources**: [tt_flash/flash.py 560-617](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L560-L617)[tt_flash/flash.py 873-881](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L873-L881)
