## Purpose and Scope

The tt-topology tool supports a specific set of Tenstorrent hardware boards for ethernet coordinate flashing and topology configuration. This page documents which boards are supported, their capabilities, detection mechanisms, and important limitations.

## Supported Hardware Overview

The tt-topology tool supports three primary board types, each with different capabilities and use cases:

| Board Type | Model | Primary Use Case | Topology Modes | Special Features |
| --- | --- | --- | --- | --- |
| `n300` | N300 | Standard multi-chip topologies | mesh, linear, torus, isolated | Dual-chip configuration |
| `n150` | N150 | Galaxy/Octopus configurations | octopus mode | Single-chip, galaxy connectivity |
| `GALAXY` | WH 4U Galaxy | Large-scale systems | octopus mode | Remote chip hosting |

## N300 Boards

The N300 board is the primary target for standard topology configurations in tt-topology. Each N300 board contains two chips and supports all standard layout types.

**Key Characteristics:**

*   Dual-chip configuration (local and remote chips)
*   Supports mesh, linear, torus, and isolated layouts
*   Default coordinate assignment: local chip (0,0), remote chip (1,0)
*   Full ethernet connectivity capabilities

## N150 Boards

N150 boards are specifically designed for Galaxy/Octopus configurations where multiple N150s connect to Galaxy systems for large-scale deployments.

**Key Characteristics:**

*   Single-chip configuration
*   Primarily used in octopus mode with Galaxy systems
*   Requires specialized reset configuration files
*   Supports TG (4 N150s + 1 Galaxy) and TGG (8 N150s + 2 Galaxies) configurations

## Galaxy Boards

Galaxy boards (WH 4U only) serve as the backbone for large-scale Octopus configurations, hosting remote chips that N150 boards connect to.

**Key Characteristics:**

*   Designated as `GALAXY` board type
*   Hosts 32 remote chips per Galaxy unit
*   Requires coordination with N150 boards through reset configuration
*   Only WH 4U Galaxy systems are supported

## Board Detection Mechanism

The tool uses a systematic approach to detect and validate supported hardware:

**Detection Process:**

1.   **Initial Discovery:**`detect_chips_with_callback()` discovers all Tenstorrent devices
2.   **Board ID Extraction:**`dev.board_id()` returns hardware-specific board identifier
3.   **Type Resolution:**`get_board_type()` maps board ID to human-readable type
4.   **Support Validation:** Check against `supported_boards = ["n300", "n150", "GALAXY"]`
5.   **Device Filtering:** Separate supported and unsupported devices

Sources: [tt_topology/tt_topology.py 452-470](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L452-L470)[tt_topology/tt_topology.py 20-22](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L20-L22)

## Unsupported Hardware

The following hardware types are explicitly unsupported and will cause the tool to either filter them out or exit with an error:

**Explicitly Unsupported:**

*   **BH PCIe cards** - Blackhole architecture not supported
*   **WH 6U Galaxy systems** - Only WH 4U Galaxy systems supported
*   **BH 6U Galaxy systems** - Blackhole 6U architecture not supported

**Warning and Error Handling:**

**Error Message Output:** When unsupported devices are detected, users see:

```
TT-Topology will only run on n300/n150/GALAXY(WH 4U only) boards.
Ignoring these devices: [list of unsupported board types].
```

Different board types enable different operational modes within tt-topology:

| Board Type | Standard Layouts | Octopus Mode | Reset Requirements |
| --- | --- | --- | --- |
| `n300` | ✅ mesh, linear, torus, isolated | ❌ | None |
| `n150` | ❌ | ✅ | reset_config.json required |
| `GALAXY` | ❌ | ✅ (passive) | reset_config.json required |
