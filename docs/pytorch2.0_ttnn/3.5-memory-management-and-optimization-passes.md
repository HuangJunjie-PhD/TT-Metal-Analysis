This document covers the memory management and optimization passes that run after operation lowering and data movement in the compilation pipeline. These passes improve runtime performance through caching, eliminate redundant operations, and manage device memory efficiently. The primary passes covered are:

*   **LoadOncePass**: Caches iteration-invariant computations (parameters, weights, biases)
*   **EliminateCoreopsPass**: Removes redundant data movement operations
*   **DeallocationPass**: Manages L1 memory by deallocating tensors after last use
*   **Other optimization passes**: FusionPass and CSEPass

For information about the overall pass pipeline orchestration, see [Backend Entry Points and Pass Pipeline](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.1-backend-entry-points-and-pass-pipeline). For lowering passes that transform PyTorch operations to TTNN, see [Operation Lowering](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)).

## LoadOncePass: Parameter Preprocessing and Caching

The `LoadOncePass` is a performance optimization that moves iteration-invariant operations (those that don't depend on input arguments) into a cached initialization function called `run_once`. This is particularly important for inference models where parameters and buffers remain constant across iterations.

### Purpose and Applicability

The pass only executes when:

1.   The model is converted end-to-end (`is_end_to_end=True`)
2.   Parameter caching is enabled (`load_params_once=True`)
3.   The graph is classified as inference (`ModelType.INFERENCE`)

Sources: [torch_ttnn/passes/lowering/load_once_pass.py 186-188](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/load_once_pass.py#L186-L188)

### Iteration Invariance Analysis

The pass marks all nodes as either **iteration-invariant** (constant) or **variable** (depends on arguments):

**Diagram: Iteration Invariance Marking Algorithm**

The algorithm uses an "infection" strategy where argument nodes mark all their descendants as variable:

| Node Type | Classification | Reasoning |
| --- | --- | --- |
| `placeholder` with `PrimalTag.PARAMETER` | Invariant | Model weights/parameters don't change |
| `placeholder` with `PrimalTag.BUFFER` | Invariant | Model buffers don't change |
| `placeholder` with `PrimalTag.ARGUMENT` | Variable | Input data changes each iteration |
| Any descendant of `ARGUMENT` | Variable | Depends on input data |
| `output`, `get_attr`, `root` | Variable | Must be recomputed |

Sources: [torch_ttnn/passes/lowering/load_once_pass.py 17-39](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/load_once_pass.py#L17-L39)

## EliminateCoreopsPass: Redundant Data Movement Elimination

The `EliminateCoreopsPass` removes redundant patterns of paired data movement operations. These patterns commonly arise during the lowering and data movement passes when tensors are unnecessarily moved between host and device or between PyTorch and TTNN representations.

### Redundant Patterns

The pass detects and eliminates the following redundant patterns:

**Diagram: Redundant Data Movement Patterns**

Sources: [torch_ttnn/passes/lowering/eliminate_coreops_pass.py 9-33](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/eliminate_coreops_pass.py#L9-L33)

### Implementation Details

The elimination logic is implemented in the `_eliminate_pair` helper function:

| Check | Condition | Result |
| --- | --- | --- |
| Function validity | Both `func` and `pre_func` in allowed list | Continue |
| Node has users | `len(node.users) > 0` | Continue |
| Current node target | `node.target == func` | Continue |
| Previous node type | `type(pre_node) == torch.fx.node.Node` | Continue |
| Previous node target | `pre_node.target == pre_func` | Continue |
| Elimination | All checks pass | `node.replace_all_uses_with(pre_node.args[0])` |

The allowed function list ensures only core data movement operations are considered:

*   `ttnn.from_device`
*   `ttnn.to_device`
*   `ttnn.from_torch`
*   `ttnn.to_torch`

Sources: [torch_ttnn/passes/lowering/eliminate_coreops_pass.py 9-33](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/eliminate_coreops_pass.py#L9-L33)

### Pass Execution

**Diagram: EliminateCoreopsPass Execution Flow**

After eliminating redundant pairs, the pass calls `eliminate_dead_code()` to remove the now-unused intermediate nodes.

Sources: [torch_ttnn/passes/lowering/eliminate_coreops_pass.py 36-46](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/eliminate_coreops_pass.py#L36-L46)

### Common Scenarios

This pass is particularly useful in the following scenarios:

1.   **After NodeInputAligner**: The aligner may insert data movement operations that create round-trips
2.   **Multi-device graphs**: Device assignment changes can create unnecessary device transfers
3.   **Fallback handling**: When operations fallback to PyTorch, round-trip conversions may occur

Sources: [torch_ttnn/passes/lowering/eliminate_coreops_pass.py 1-46](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/eliminate_coreops_pass.py#L1-L46)

## DeallocationPass: Memory Management

The `DeallocationPass` is critical for managing limited L1 memory on Tenstorrent devices. It inserts explicit `deallocate()` calls to free TTNN tensors after their last use, preventing memory exhaustion.

### Memory Management Motivation

TTNN graphs can have high memory pressure because:

*   L1 memory on device is limited
*   Intermediate tensors accumulate if not explicitly freed
*   Unlike PyTorch's automatic garbage collection, TTNN requires explicit deallocation

Sources: [torch_ttnn/passes/deallocation_pass.py 21-26](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L21-L26)

### TrackUnusedValues: Last-Use Analysis

The `TrackUnusedValues` class tracks the last use of each node in the graph:

**Diagram: Last-Use Tracking Algorithm**

Sources: [torch_ttnn/passes/deallocation_pass.py 39-74](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L39-L74)

### Tracking Algorithm

The algorithm builds two key data structures:

| Data Structure | Type | Purpose |
| --- | --- | --- |
| `node_to_last_use` | `Dict[Node, Node]` | Maps each node to the last node that uses it |
| `user_to_last_uses` | `Dict[Node, List[Node]]` | Maps each node to the list of nodes whose last use is this node |

The tracking is performed by reverse-iterating through the graph:

### Deallocation Insertion

The pass inserts `deallocate()` calls after each node that represents the last use of one or more tensors:

**Diagram: Deallocation Insertion Logic**

Sources: [torch_ttnn/passes/deallocation_pass.py 89-105](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L89-L105)

### Special Cases

The pass includes special handling for certain node types:

| Node Type | Handling | Reason |
| --- | --- | --- |
| `pack_to_tuple` | Skip deallocation | Node is just a reference wrapper |
| `operator.getitem` | Skip deallocation | Node is just a reference extraction |
| Cached nodes (`is_cached=True`) | Skip deallocation | Node is reused across runs |
| Unused outputs | Add to deletion list | Nodes with no users but have side effects |

Sources: [torch_ttnn/passes/deallocation_pass.py 96-101](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/deallocation_pass.py#L96-L101)

The `deallocate()` wrapper function handles type checking:

The function is marked with `@torch.fx.wrap` to ensure it's treated as a side-effect operation that won't be eliminated by dead code elimination. Additionally, `torch.fx.node.has_side_effect(deallocate)` is called in the pass constructor.

## Other Optimization Passes

While the primary memory management and optimization passes are detailed above, the compilation pipeline includes additional optimization passes:

### FusionPass

The `FusionPass` combines compatible operations into single fused operations for improved performance. While the implementation is not provided in the source files, it's referenced in the compilation pipeline.

**Common fusion patterns include:**

*   Element-wise operation chains (e.g., add + relu)
*   Broadcast operations with computation
*   Layout transformation with operations
