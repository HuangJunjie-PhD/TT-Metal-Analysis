This page describes the integration between the **vLLM** inference engine and the **tt-metal** backend. This integration allows Tenstorrent hardware (N150, N300, T3K, Galaxy) to serve as an acceleration platform for vLLM's OpenAI-compatible API server.

## Overview

The integration is primarily delivered via a specialized Docker image and a vLLM platform plugin (`tt-vllm-plugin`). The system allows vLLM to offload model execution to Tenstorrent devices while maintaining compatibility with standard vLLM features like PagedAttention and the OpenAI API.

## The vLLM tt-metal Docker Image

The Docker image provides a pre-configured environment containing `tt-metal`, a fork of `vllm` installed from source, and the `tt-vllm-plugin`.

### Container Entrypoint: `run_vllm_api_server.py`

The script `run_vllm_api_server.py` serves as the primary entrypoint for the container. It handles model resolution, environment setup, and launches the vLLM server process.

**Key Responsibilities:**

1.   **Model Spec Resolution**: Loads the model configuration from a bundled JSON catalog or a pre-resolved runtime spec [vllm-tt-metal/src/run_vllm_api_server.py 120-186](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/src/run_vllm_api_server.py#L120-L186) It prioritizes `RUNTIME_MODEL_SPEC_JSON_PATH` if set by the `run.py` orchestration [vllm-tt-metal/src/run_vllm_api_server.py 140-151](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/src/run_vllm_api_server.py#L140-L151)
2.   **Weight Management**: Downloads model weights from HuggingFace using `snapshot_download` if they are not already mapped into the container [vllm-tt-metal/src/run_vllm_api_server.py 16-17](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/src/run_vllm_api_server.py#L16-L17)
3.   **Authorization**: Supports optional API key authorization via `JWT_SECRET` or disabling it with `--no-auth`[vllm-tt-metal/src/run_vllm_api_server.py 69-72](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/src/run_vllm_api_server.py#L69-L72)
4.   **Trace Capture**: Optionally triggers a background trace capture using `run_background_trace_capture` to warm up the Tenstorrent device and capture execution graphs [vllm-tt-metal/src/run_vllm_api_server.py 22](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/src/run_vllm_api_server.py#L22-L22)

## Model Registration and Plugin System

The `tt-vllm-plugin` utilizes vLLM's plugin architecture to register Tenstorrent-specific platforms and models.

### TTPlatform and ModelRegistry

The plugin registers a custom platform and maps Tenstorrent-optimized model implementations to vLLM model names.

| Model Name | Implementation Path | Use Case |
| --- | --- | --- |
| `TTLlamaForCausalLM` | `models.tt_transformers.tt.generator_vllm:LlamaForCausalLM` | Llama 3/3.1/3.2 Causal LM |
| `TTBertModel` | `models.demos.wormhole.bge_large_en.demo.generator_vllm:BGEForEmbedding` | BGE Embedding |
| `TTQwen3Model` | `models.demos.wormhole.qwen3_embedding_8b.demo.generator_vllm:Qwen3ForEmbedding` | Qwen3 Embedding |

**Plugin Entrypoints** The plugin is registered via `pyproject.toml` entry points:

*   `vllm.platform_plugins`: Maps `tt` to `tt_vllm_plugin:register`[tt-vllm-plugin/pyproject.toml 44-45](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-vllm-plugin/pyproject.toml#L44-L45)
*   `vllm.general_plugins`: Maps `tt_model_registry` to `tt_vllm_plugin:register_models`[tt-vllm-plugin/pyproject.toml 47-48](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-vllm-plugin/pyproject.toml#L47-L48)

## Multi-Host Deployment

For large-scale deployments like the Galaxy system, the integration uses a specialized entrypoint and Dockerfile to manage distributed execution across multiple Tenstorrent-equipped hosts.

### `multihost_entrypoint.sh`

This script orchestrates the launch across multiple hosts. It is designed to be the entrypoint for distributed containers [vllm-tt-metal/vllm.tt-metal.src.multihost.Dockerfile 1-10](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/vllm.tt-metal.src.multihost.Dockerfile#L1-L10) It typically handles:

1.   **Environment Sync**: Ensuring `TT_METAL_HOME`, `PYTHONPATH`, and device-specific paths are consistent across nodes [vllm-tt-metal/multihost_entrypoint.sh 1-10](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/vllm-tt-metal/multihost_entrypoint.sh#L1-L10)
2.   **Distributed Backend**: Launching the vLLM engine with the appropriate distributed configuration for multi-node inference [docs/multihost_deployment.md 31-32](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/multihost_deployment.md?plain=1#L31-L32)

## Model Specification Expansion

Models are defined using `ModelSpecTemplate` which allows for expanding a single definition across multiple devices and weights, ensuring consistent configuration across the vLLM ecosystem.

**Expansion Logic:**

1.   A template is defined with a list of supported `weights` (HuggingFace repos) and `device_model_specs`[tests/test_model_specification.py 61-84](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tests/test_model_specification.py#L61-L84)
2.   The `expand_to_specs()` function generates individual `ModelSpec` objects for every weight/device combination [tests/test_model_specification.py 93-94](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tests/test_model_specification.py#L93-L94)
3.   During release, `update_model_spec.py` updates these templates with specific `tt_metal_commit` and `vllm_commit` hashes based on CI results [scripts/release/update_model_spec.py 9-10](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/update_model_spec.py#L9-L10)
4.   These specs are then exported to `model_spec.json` using `export_model_specs_json`, which is bundled into the Docker image for the `run_vllm_api_server.py` to use for resolution [scripts/build_docker_images.py 28](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/build_docker_images.py#L28-L28)
