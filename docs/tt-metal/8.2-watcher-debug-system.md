## Purpose and Scope

The Watcher debug system in the tt-metal codebase provides a runtime monitoring and error detection framework for Tenstorrent devices. It instruments firmware and kernels running on device cores to monitor device state, kernel execution progress, and catch runtime errors such as assertion failures and hardware faults. When such errors occur, Watcher collects detailed debugging information (including core states, waypoints, NoC accesses, and execution context) and captures it for post-mortem or live inspection.

The system consists of two main parts:

*   A **device-side instrumentation** framework that embeds macros such as `ASSERT()` and `WAYPOINT()` into kernels and firmware to report runtime conditions.

*   A **host-side WatcherServer** thread that periodically polls device L1 mailboxes and logging memory buffers via the UMD interface to detect hangs, asserts, and errors.

Watcher is complementary to and distinct from other debugging subsystems:

*   **DPRINT Server (Section 8.3)**: Kernel print (`printf`) support using circular buffers.

*   **Inspector**: Host-side runtime querying of device state.

*   **Triage Tools (Section 8.6)**: Post-mortem hang and fault analysis tooling.

## System Architecture

Watcher employs a client-server model where device firmware writes status and event information to predefined L1 memory "mailboxes" and ring buffers. The host-side WatcherServer runs a background thread that polls these buffers over the device management layer (UMD), interprets the data, and logs comprehensive debug information to disk.

The WatcherServer holds one or more `WatcherDeviceReader` instances — each corresponding to a Tenstorrent compute device (chip). These readers access device memory through the `MetalContext` HAL abstraction and interpret kernel- and hardware-generated messages.

### Code Entity Space Mapping

Explanation:

*   **MetalContext**: Central singleton managing device clusters and hardware abstraction.

*   **WatcherServer::Impl**: Host-side server thread managing polling and debug state.

*   **WatcherDeviceReader**: Reads per-device debug mailboxes and interprets them.

*   **HAL / HalProcessorIdentifier**: Defines core types and provides addressing information.

*   **Device L1 Memory**: Holds debug structures written by device side, like asserts and waypoints.

*   **ASSERT() / WAYPOINT() macros**: Embedded in device kernels for runtime error and progress reporting.

## WatcherServer and Polling Logic

The `WatcherServer` runs asynchronously on the host to periodically fetch debug data from every active device. It supports lifecycle methods to attach/detach devices and to dump accumulative watch data.

### Execution Flow

1.   **Initialization**

    *   The WatcherServer is created based on runtime options storing in `MetalEnv`. It uses HAL info to discover devices.

2.   **Attach Devices**

    *   For each device, a `WatcherDeviceReader` is instantiated.

    *   Log files such as `watcher.log` and kernel info files are created.

    *   DMA library calls are disabled because watcher access is not thread-safe.

3.   **Polling Loop**

    *   Runs in a separate thread.

    *   Iteratively queries devices via `WatcherDeviceReader::Dump` which reads device memory mailboxes over UMD.

    *   Uses HAL to decode core and processor identifiers.

    *   Detects tripped asserts and errors, sets internal flags.

    *   Flushes log output to disk periodically.

4.   **Detach Devices**

    *   Stops the polling thread, closes files.

    *   Re-enables DMA calls.

### Polling Data Flow

*   The device kernel writes to designated L1 mailbox areas.

*   `WatcherDeviceReader` reads these memory regions via UMD calls embedded in `MetalContext` and HAL interfaces.

*   HAL maps raw core IDs to human-readable names.

## Detection Features

The Watcher system can detect multiple forms of runtime anomalies and provide rich logging and assertion data:

### 1. Assert Detection

*   The kernel-side `ASSERT()` macro writes assert line numbers and type codes to the debug mailbox (`debug_assert_msg_t`).

*   Message includes details like faulting line, core/hardware thread, and error type.

*   The helper `get_debug_assert_message()` maps assert codes to readable messages.

*   For example, it detects inter-kernel data races, bad NOC packet states, and out-of-bound runtime arg access.

### 2. Hardware Faults

*   On architectures like Quasar, Watcher captures hardware faults (e.g. illegal instruction or memory fault).

*   It records program counter (PC) and fault cause from RISC-V registers in mailbox.

*   Specialized handling exists to decode Quasar-specific error registers.

### 3. Network-on-Chip (NoC) Sanitization

*   Watcher examines NoC transactions to detect malformed addresses, out-of-bound multicast ranges, and invalid linked transactions.

*   Uses HAL to validate multicast/unicast coordinates and memory types.

*   Supports reporting bad alignment, zero-length writes, and bad write state errors.

### 4. Processor Identification

*   Watcher supports multiple core types including:

    *   **TENSIX** (e.g. DM, COMPUTE cores),

    *   **Ethernet cores** (ACTIVE_ETH, IDLE_ETH),

    *   **DRAM cores** (e.g. DRISC processor).

*   HAL provides string names for cores and processors.

*   Watcher uses these to specify exact fault location.

## Stack Usage Monitoring

Watcher also tracks the kernel stack usage for the device cores to detect possible overflows or unsafe stack conditions.

### Stack Usage Data Flow

*   Stack usage data is collected during kernel execution.

*   Read back by WatcherDeviceReader and logged for diagnostics.

## Configuration and JIT Integration

### RunTimeOptions and Environment Variables

*   Watcher activation is controlled via `RunTimeOptions` interface.

*   Typical environment enabling flag is `TT_METAL_WATCHER`.

*   Watcher enabled flag controls whether the background thread spawns during testing.

### JIT Kernel Compilation

*   The JIT build system incorporates conditional defines (e.g. `WATCHER_ENABLED`) to compile kernel instrumentation macros like `ASSERT()` and `WAYPOINT()`.

*   This allows the runtime to embed Watcher hooks only when debugging is desired.

## Watcher Dump and Post-Mortem

Watcher provides facilities to dump the collected debug information:

*   `WatcherServer::Impl::dump(FILE* f)` iterates over each device reader and dumps detailed per-device debug data, including assert and waypoint logs, into the provided stream or file pointer.

*   `isolated_dump()` allows dumping selected devices without affecting the general Watcher state.

*   These dumps capture device states at the time of error or hang and are critical for debugging.

# Summary

The Watcher debug system is a comprehensive runtime monitoring system that uses host-side polling of device state mailboxes and logging buffers to detect kernel asserts, hardware faults, NoC transaction errors, and kernel hangs. It provides detailed per-device logs about all critical runtime events. Integration with the JIT kernel compilation and CI workflows ensures automatic activation during tests, facilitating rapid detection and diagnosis of hardware and kernel software errors on Tenstorrent devices.

Its design leverages HAL abstractions for processor and memory addressing, and interacts with device instrumentation macros to provide a tightly coupled debug experience. Watcher is a core debugging tool within tt-metal for improving kernel reliability and fault transparency.
