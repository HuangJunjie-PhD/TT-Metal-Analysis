## Purpose and Scope

The Device Information Tab is the first of three tabs in the TT-SMI interactive text user interface (TUI). This tab displays static and semi-static hardware configuration information for all detected Tenstorrent devices in a tabular format. The information includes PCI bus topology, board identification, memory training status, and PCIe link properties.

For navigation and keyboard shortcuts to access this tab, see [TUI Navigation and Layout](https://deepwiki.com/tenstorrent/tt-smi/3.1-tui-navigation-and-layout). For dynamic telemetry monitoring (voltage, current, temperature), see [Telemetry Tab](https://deepwiki.com/tenstorrent/tt-smi/3.3-telemetry-tab). For firmware version information, see [Firmware Version Tab](https://deepwiki.com/tenstorrent/tt-smi/3.4-firmware-version-tab).

## Table Structure and Columns

The Device Information tab presents data in a `TTDataTable` widget with a two-row header and color-coded cells. The table is defined at initialization and populated on mount with static device information that does not require periodic updates.

### Column Definitions

| Column | Header | Data Source | Description |
| --- | --- | --- | --- |
| 1 | # | Device index | Sequential device number (0-based) |
| 2 | Bus ID | `device.get_pci_bdf()` | PCI bus:device.function identifier |
| 3 | Board Type | `get_board_type(board_id)` | Board model identifier (e.g., n300, p150a) |
| 4 | Board ID | `get_board_id()` | 16-character hex board serial number |
| 5 | Coords | `device.as_wh().get_local_coord()` | Galaxy shelf/rack coordinates (Wormhole only) |
| 6 | DRAM Trained | `get_dram_training_status()` | Memory training pass/fail status |
| 7 | DRAM Speed | `get_dram_speed()` | Memory frequency (e.g., 16G, 14G) |
| 8 | Link Speed | `pci_properties["current_link_speed"]` | PCIe generation (Gen1-Gen5) |
| 9 | Link Width | `pci_properties["current_link_width"]` | PCIe lane count (x1-x16) |

### Initialization Sequence

Device information collection occurs during `TTSMIBackend` initialization in a tracked progress loop:

1.   **SMBus Telemetry Collection**[tt_smi/tt_smi_backend.py 91](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L91-L91): Reads raw hardware telemetry registers via `get_telemetry()` from pyluwen
2.   **PCI Properties Collection**[tt_smi/tt_smi_backend.py 93](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L93-L93): Reads sysfs files under `/sys/bus/pci/devices/{bdf}/`
3.   **Device Info Assembly**[tt_smi/tt_smi_backend.py 94](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L94-L94): Aggregates data from multiple sources into `device_infos[i]` dictionary

The collected data is stored in backend arrays (`device_infos`, `pci_properties`, `smbus_telem_info`) and remains static for the lifetime of the TUI session.

## Display Formatting and Color Coding

The `format_device_info_rows()` method applies conditional formatting based on operational status and board type. The method creates `rich.Text` objects with style attributes from the `text_theme` dictionary.

### Board Type Display

**Wormhole n300 Remote/Local Indicators**[tt_smi/tt_smi.py 326-352](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L326-L352):

*   Local chip: Displays `"n300 L"` with " L" in yellow_bold
*   Remote chip: Displays `"n300 R"` with " R" in yellow_bold
*   Determination: `device.is_remote()` method from pyluwen

### PCIe Link Width Display

**Status Indication**[tt_smi/tt_smi.py 361-397](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L361-L397):

*   **Remote devices**: Display `"N/A"` in gray (no PCIe connection)
*   **Below maximum**: Display `"x{current}"` in attention (orange) + `" / x{max}"` in yellow_bold
*   **At maximum**: Display `"x{current}"` in text_green + `" / x{max}"` in yellow_bold

Example outputs:

*   `x8 / x16` (orange/yellow) - Link trained below capacity
*   `x16 / x16` (green/yellow) - Link at full capacity
*   `N/A` (gray) - Remote device, no local PCIe

### PCIe Link Speed Display

**Generation-Based Formatting**[tt_smi/tt_smi.py 398-460](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L398-L460):

*   **Remote devices**: Display `"N/A"` in gray
*   **Current below maximum**: Display `"Gen{current}"` in attention + `" / Gen{max}"` in yellow_bold
*   **Current at maximum**: Display `"Gen{current}"` in text_green + `" / Gen{max}"` in yellow_bold
*   **N/A current speed**: Display `"N/A"` in red_bold (link down)
*   **N/A max speed**: Display `"Gen{current}"` in attention + `" / N/A"` in gray (sysfs read failure)

The PCIe generation mapping [tt_smi/tt_smi_backend.py 321-333](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L321-L333):

*   Gen 5: 32.0 GT/s
*   Gen 4: 16.0 GT/s
*   Gen 3: 8.0 GT/s
*   Gen 2: 5.0 GT/s
*   Gen 1: 2.5 GT/s

### DRAM Status Display

**Binary Training Status**[tt_smi/tt_smi.py 461-475](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L461-L475):

*   **Trained**: Display `"Y"` in text_green
*   **Not trained**: Display `"N"` in attention (orange)

**DRAM Speed Display**[tt_smi/tt_smi.py 476-492](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L476-L492):

*   **Valid speed**: Display frequency (e.g., `"16G"`) in text_green
*   **Invalid/Not available**: Display `"N/A"` in red_bold

## Platform-Specific Behavior

### Wormhole-Specific Features

**Galaxy Coordinates**[tt_smi/tt_smi.py 399-405](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L399-L405)[tt_smi/tt_smi_backend.py 399-405](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L399-L405):

*   Format: `(shelf_x, shelf_y, rack_x, rack_y)`
*   Only available on Wormhole devices
*   Blackhole devices display `"N/A"`

**DRAM Status Register Layout**[tt_smi/tt_smi_backend.py 354-366](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L354-L366):

*   Bit [31:24]: DRAM speed
*   Bit [23:0]: Per-channel status (6 channels)
*   Success value: `0x222222` (all channels passed training)
*   Each channel: `0x2` = DRAM_TRAINING_PASS, `0x1` = DRAM_TRAINING_FAIL

**DRAM Speed Encoding**[tt_smi/tt_smi_backend.py 289-305](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L289-L305):

*   Raw value from `(DDR_STATUS >> 24)`
*   Mapping: 0→16G, 1→14G, 2→12G, 3→10G, 4→8G

### Blackhole-Specific Features

**DRAM Status Register Layout**[tt_smi/tt_smi_backend.py 367-380](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L367-L380):

*   16-bit field with bit pairs per channel
*   8 GDDR channels (bits [15:0])
*   Bit pairs: [even]=training_complete, [odd]=error
*   Success value: `0x5555` = `0b0101010101010101` (all channels trained without error)

**DRAM Speed Encoding**[tt_smi/tt_smi_backend.py 279-288](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L279-L288):

*   Direct frequency value from `DDR_SPEED` register
*   Divisible by 1000: Display in GHz (e.g., `16G`)
*   Not divisible by 1000: Display in MHz (e.g., `1600`)

### Remote Device Handling

Remote devices (detected via `device.is_remote()`) have limited data availability:

*   **Bus ID**: Returns `"N/A"` if `get_pci_bdf()` fails
*   **PCIe Speed**: Always `"N/A"` (no local PCIe interface)
*   **PCIe Width**: Always `"N/A"` (no local PCIe interface)
*   **Coords**: Available for Wormhole Galaxy systems
*   **Board Type**: Includes " R" suffix for n300 remote chips

Sources: [tt_smi/tt_smi_backend.py 312-313](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L312-L313)[tt_smi/tt_smi.py 363-370](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L363-L370)

## Table Rendering

### Widget Configuration

The Device Information table is created during the `compose()` lifecycle method:

`TTDataTable(    title="Device Information",    id="tt_smi_device_info",    header=constants.INFO_TABLE_HEADER,    header_height=2,)`
Key properties:

*   **Widget type**: `TTDataTable` from tt_tools_common.ui_common.widgets
*   **ID**: `"tt_smi_device_info"` for query_one() selection
*   **Header**: Two-row header from `INFO_TABLE_HEADER` constant
*   **Cursor**: Disabled (`cursor_type = "none"`) - table is read-only

Sources: [tt_smi/tt_smi.py 128-133](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L128-L133)[tt_smi/tt_smi.py 150-152](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L150-L152)

### Population and Updates

The table is populated once during the `on_mount()` event handler and does not update automatically:

Unlike the Telemetry tab which updates every 100ms, the Device Information tab displays static data captured at initialization. To see updated information after a hardware change (e.g., PCIe link retraining), users must restart tt-smi.

Sources: [tt_smi/tt_smi.py 148-152](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L148-L152)

## Color Theme Reference

The formatting uses color styles from `create_tt_tools_theme()`:

| Style Name | Usage |
| --- | --- |
| `yellow_bold` | Device index (#), maximum values, suffixes |
| `text_green` | Normal operational values |
| `attention` | Warning states (below-spec PCIe, DRAM not trained) |
| `red_bold` | Critical failures (N/A speeds, link down) |
| `gray` | N/A values, unavailable data |
