# Hypervisor Virtual Memory

Relevant source files
*   [hypervisor.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/hypervisor.cpp)

## Purpose and Scope

This document describes the hypervisor virtual memory subsystem in Whisper, which implements two-stage address translation as defined by the RISC-V Hypervisor Extension. This includes VS-stage (first-stage) and G-stage (second-stage/Stage-2) translation, TLB management for both stages, HFENCE instructions for selective TLB invalidation, and hypervisor load/store (HLV/HSV) instructions for accessing virtual machine memory.

For general virtual memory and address translation concepts, see [Virtual Memory and Address Translation](https://deepwiki.com/tenstorrent/whisper/4.2-virtual-memory-and-address-translation). For an overview of the Hypervisor Extension and VS-mode execution, see [Hypervisor Extension (H)](https://deepwiki.com/tenstorrent/whisper/5.2-hypervisor-extension-(h)). For physical memory protection that applies after address translation, see [Physical Memory and Protection](https://deepwiki.com/tenstorrent/whisper/4.1-physical-memory-and-protection).

* * *

## Two-Stage Address Translation Overview

When the hypervisor extension is enabled and a hart is executing in VS-mode (virtual supervisor mode) or VU-mode (virtual user mode), memory accesses undergo two-stage translation:

1.   **VS-stage (First-stage)**: Translates virtual address (VA) to guest physical address (GPA) using VS-mode page tables
2.   **G-stage (Stage-2)**: Translates guest physical address (GPA) to supervisor physical address (PA) using hypervisor page tables

### Translation Path

**Two-Stage Translation Flow Diagram**

Both TLBs are accessed when translating addresses in virtual mode. The VS-stage TLB translates VA to GPA using the address space identifier (ASID) and virtual machine identifier (VMID), while the Stage-2 TLB translates GPA to PA using VMID only.

## TLB Organization

The `VirtMem` class contains two separate TLB structures for hypervisor support:

| TLB Structure | Purpose | Key Fields | Indexing |
| --- | --- | --- | --- |
| `vsTlb_` | VS-stage translation | VA → GPA mappings | VPN, ASID, VMID, WID |
| `stage2Tlb_` | G-stage translation | GPA → PA mappings | GPN, VMID, WID |

### TLB Entry Identification

Each TLB entry is identified by:

*   **VPN/GPN**: Virtual/Guest Page Number (address bits)
*   **ASID**: Address Space Identifier (for VS-stage only)
*   **VMID**: Virtual Machine Identifier (for both stages)
*   **WID**: World Identifier (for STEE secure world isolation, if enabled)

## HFENCE Instructions

HFENCE instructions provide selective TLB invalidation for hypervisor environments. Whisper implements two primary HFENCE instructions and their SINVAL extension variants.

### HFENCE.VVMA

The `HFENCE.VVMA` instruction invalidates VS-stage TLB entries. Implementation in `execHfence_vvma()`:

**Privilege Requirements:**

*   Must have hypervisor extension enabled (`isRvh()`)
*   Cannot execute in virtual mode (causes virtual instruction exception)
*   Cannot execute in User mode (causes illegal instruction exception)

**Invalidation Patterns:**

| rs1 | rs2 | Behavior | TLB Invalidation |
| --- | --- | --- | --- |
| x0 | x0 | All entries for current VMID | `vsTlb_.invalidateVmid(vmid, wid)` |
| x0 | !x0 | All entries for ASID and VMID | `vsTlb_.invalidateAsidVmid(asid, vmid, wid)` |
| !x0 | x0 | Single page for VMID | `vsTlb_.invalidateVirtualPageVmid(vpn, vmid, wid)` |
| !x0 | !x0 | Single page for ASID | `vsTlb_.invalidateVirtualPageAsid(vpn, asid, wid)` |

**HFENCE.VVMA Execution Flow Diagram**

Sources: [hypervisor.cpp 28-82](https://github.com/tenstorrent/whisper/blob/733fd921/hypervisor.cpp#L28-L82)

### HFENCE.GVMA

The `HFENCE.GVMA` instruction invalidates G-stage (Stage-2) TLB entries. Implementation in `execHfence_gvma()`:

**Privilege Requirements:**

*   Must have hypervisor extension enabled (`isRvh()`)
*   Cannot execute in virtual mode (causes virtual instruction exception)
*   Cannot execute in User mode
*   Cannot execute in Supervisor mode if `mstatus.TVM=1` (causes illegal instruction exception)

**Invalidation Patterns:**

| rs1 | rs2 | Behavior | Notes |
| --- | --- | --- | --- |
| x0 | x0 | All entries for current VMID | Over-invalidates VS-stage TLB as well |
| x0 | !x0 | All entries for specified VMID | VMID value from `x[rs2]` |
| !x0 | x0 | Single GPA for current VMID | GPA = `x[rs1] << 2`, may over-invalidate if `hfenceGvmaIgnoresGpa_` |
| !x0 | !x0 | Single GPA for specified VMID | GPA = `x[rs1] << 2`, VMID from `x[rs2]` |

**Important Implementation Detail:** The boolean flag `hfenceGvmaIgnoresGpa_` controls whether the implementation stores guest-physical addresses in the TLB. If false, the implementation over-invalidates by ignoring the GPA parameter and invalidating all entries for the VMID.

**HFENCE.GVMA Execution Flow Diagram**

Sources: [hypervisor.cpp 87-153](https://github.com/tenstorrent/whisper/blob/733fd921/hypervisor.cpp#L87-L153)

### SINVAL Extension Variants

The Supervisor Memory-Management Fence Invalidation (SINVAL) extension provides additional HFENCE variants:

*   `HINVAL.VVMA`: Implemented by calling `execHfence_vvma()` if `isRvsvinval()` is enabled
*   `HINVAL.GVMA`: Implemented by calling `execHfence_gvma()` if `isRvsvinval()` is enabled

## Hypervisor Load/Store Instructions (HLV/HSV)

HLV (Hypervisor Load Virtual) and HSV (Hypervisor Store Virtual) instructions allow the hypervisor to explicitly access memory as if it were running in VS-mode or VU-mode. These instructions perform two-stage address translation but use different privilege and configuration settings.

### Common Execution Logic

Both HLV and HSV instructions share common setup logic implemented in `hyperLoad()` and `hyperStore()` templates:

**Privilege Checks:**

1.   Hypervisor extension must be enabled (`isRvh()`)
2.   Cannot execute in virtual mode (causes virtual instruction exception)
3.   Can execute in User mode only if `hstatus.HU=1`

**Translation Configuration:**

The instructions temporarily modify translation state to use VS-mode settings:

| Setting | Source CSR | Purpose |
| --- | --- | --- |
| Big-endian mode | `hstatus.VSBE` | Controls byte ordering for translation |
| Stage-1 MXR | `vsstatus.MXR` | Make executable readable for first-stage |
| SUM bit | `vsstatus.SUM` | Supervisor access to user pages |
| Pointer masking | `hstatus.HUPMM` | Pointer masking mode for User-mode access |

**HLV/HSV Common Execution Logic Diagram**

### HLV Instructions

Whisper implements the following HLV (Hypervisor Load Virtual) instructions:

| Instruction | Implementation | Data Type | Sign Extension |
| --- | --- | --- | --- |
| `HLV.B` | `execHlv_b()` | int8_t | Yes |
| `HLV.BU` | `execHlv_bu()` | uint8_t | No |
| `HLV.H` | `execHlv_h()` | int16_t | Yes |
| `HLV.HU` | `execHlv_hu()` | uint16_t | No |
| `HLV.W` | `execHlv_w()` | int32_t | Yes |
| `HLV.WU` | `execHlv_wu()` | uint32_t | No (RV64 only) |
| `HLV.D` | `execHlv_d()` | uint64_t | No (RV64 only) |

**Execute-as-Read Variants:**

Two special variants use executable pages for reads:

*   `HLVX.HU`: Sets `virtMem_.useExecForRead(true)` before calling `hyperLoad<uint16_t>()`
*   `HLVX.WU`: Sets `virtMem_.useExecForRead(true)` before calling `hyperLoad<uint32_t>()`

These instructions allow the hypervisor to read instruction memory even when it's marked execute-only.

## Related CSRs and State

### Control and Status Registers

The hypervisor load/store instructions interact with several CSRs:

| CSR | Field | Purpose in HLV/HSV |
| --- | --- | --- |
| `hstatus` | `VSBE` | VS-mode big-endian translation |
| `hstatus` | `HU` | Allow HLV/HSV in User mode |
| `hstatus` | `HUPMM` | User-mode pointer masking |
| `vsstatus` | `MXR` | Make executable readable (stage-1) |
| `vsstatus` | `SUM` | Supervisor user memory access |
| `vsatp` | - | VS-stage page table base (implicit) |
| `hgatp` | - | G-stage page table base (implicit) |

### VMID Management

The Virtual Machine Identifier (VMID) is retrieved via `virtMem_.vmid()` and used to isolate TLB entries between different virtual machines. HFENCE instructions can target specific VMIDs or the current VMID.

### World Identifier (WID)

When the STEE (Secure Trusted Execution Environment) extension is enabled, a World Identifier is used in addition to VMID:

*   `uint32_t wid = steeEnabled_ ? stee_.secureWorld() : 0`
*   WID provides additional isolation for secure vs. non-secure worlds

### TLB Invalidation Methods

The TLB classes provide several invalidation methods used by HFENCE instructions:

*   `invalidateVmid(vmid, wid)`: Invalidate all entries for VMID
*   `invalidateAsidVmid(asid, vmid, wid)`: Invalidate all entries for ASID and VMID
*   `invalidateVirtualPageVmid(vpn, vmid, wid)`: Invalidate specific page for VMID
*   `invalidateVirtualPageAsid(vpn, asid, wid)`: Invalidate specific page for ASID
