## Purpose and Scope

This document details the core abstractions that form the foundation of the Luwen framework. It covers the key interfaces, classes, and their relationships that enable interacting with Tenstorrent AI accelerator chips in a unified way. These components provide the architecture for chip detection, initialization, and communication across different chip types.

This page focuses on the core abstractions themselves. For more information about:

*   Communication methods, see [Communication Interfaces](https://deepwiki.com/tenstorrent/luwen/2.3-communication-interfaces)
*   Specific chip implementations, see [Chip Support](https://deepwiki.com/tenstorrent/luwen/3-chip-support)
*   Detection and initialization process, see [Chip Detection and Initialization](https://deepwiki.com/tenstorrent/luwen/4.1-chip-detection-and-initialization)

## Key Abstractions

The Luwen framework is built on a set of core abstractions that enable interaction with different chip architectures through a unified interface.

### Chip Interface Hierarchy

The following diagram shows the main interfaces and their relationships:

Sources: [crates/luwen-if/src/chip/mod.rs 282-385](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L282-L385)

The core interfaces in the hierarchy are:

*   **HlComms**: A low-level interface for basic communication with a chip, providing access to communication objects like `ChipComms` and `ChipInterface`

*   **ChipImpl**: The main interface that defines common functionality for all chip types, extending `HlComms` with chip-specific operations such as:

    *   Initialization state management
    *   Architecture identification
    *   Telemetry collection
    *   ARC messaging
    *   Chip discovery in mesh configurations

*   **Chip**: A wrapper struct that contains a `Box<dyn ChipImpl>`, providing a unified interface for all chip types while allowing downcasting to specific implementations when needed

### Specific Chip Implementations

Three main chip implementations are provided:

1.   **Wormhole**: Implementation for Wormhole chips with Ethernet support for mesh configurations
2.   **Grayskull**: Implementation for Grayskull chips
3.   **Blackhole**: Implementation for Blackhole chips

Each implementation is tailored to the specific capabilities and characteristics of its respective chip architecture.

### Chip Struct and Downcasting

The `Chip` struct provides a convenient abstraction over the various chip implementations:

Sources: [crates/luwen-if/src/chip/mod.rs 321-346](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L321-L346)

The `Chip` struct provides methods for downcasting to specific chip implementations:

*   `as_wh()`: Attempts to downcast to a Wormhole chip
*   `as_gs()`: Attempts to downcast to a Grayskull chip
*   `as_bh()`: Attempts to downcast to a Blackhole chip

This design allows code to work with chips generically through the `ChipImpl` interface, but also access architecture-specific functionality when needed.

## Chip Status and Telemetry

### Initialization Status

The `InitStatus` structure tracks the initialization state of a chip, providing detailed information about different components:

Sources: [crates/luwen-if/src/chip/mod.rs 23-26](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L23-L26)

The `InitStatus` tracks:

*   Basic communication status
*   ARC (Accelerated Routing Core) status
*   DRAM initialization status
*   Ethernet subsystem status
*   CPU subsystem status

### Telemetry Information

The `Telemetry` struct provides comprehensive information about a chip's state:

| Category | Examples of Telemetry Data |
| --- | --- |
| Identification | board_id, device_id, asic_id |
| Firmware Versions | arc*_fw_version, eth_fw_version, ddr_fw_version |
| Status Indicators | ddr_status, eth_status*, pcie_status |
| Performance Metrics | aiclk, axiclk, arcclk |
| Temperature/Power | asic_temperature, vreg_temperature, board_temperature, tdp, tdc |
| Health Indicators | arc*_health, faults |

Sources: [crates/luwen-if/src/chip/mod.rs 66-261](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L66-L261)

The `Telemetry` struct provides methods to access formatted information:

*   `firmware_date()`: Returns the firmware date in YYYY-MM-DD format
*   `arc_fw_version()`: Returns ARC firmware version in MAJOR.MINOR.PATCH format
*   `eth_fw_version()`: Returns Ethernet firmware version
*   `board_type()`: Returns the board type based on serial number
*   Various temperature/power/clock methods to access formatted data

## Chip Detection and Initialization

### UninitChip Abstraction

The `UninitChip` enum represents a chip that may or may not be fully initialized:

Sources: [crates/luwen-if/src/detect_chips.rs 21-173](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L21-L173)

The `UninitChip` can be in one of two states:

*   `Partially`: The chip has been detected but not fully initialized. Contains status information about the initialization progress.
*   `Initialized`: The chip is fully initialized and ready for use.

This abstraction allows for safe handling of chips that may have initialization issues, providing methods to check component status before attempting operations.

### Chip Detection Process

The chip detection process is implemented in the `detect_chips` function:

Sources: [crates/luwen-if/src/detect_chips.rs 251-386](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L251-L386)[crates/luwen-ref/src/detect.rs 19-308](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/detect.rs#L19-L308)

The detection process involves:

1.   Scanning for PCI devices to find root chips
2.   Initializing each discovered root chip
3.   For chips with Ethernet capability, discovering and initializing remote chips
4.   Managing duplicates to ensure each physical chip is only represented once
5.   Returning a collection of `UninitChip` objects representing all discovered chips

### Chip Detection Options

The `ChipDetectOptions` struct provides configuration for the chip detection process:

| Option | Purpose |
| --- | --- |
| `continue_on_failure` | Whether to continue searching for chips after encountering a recoverable error |
| `local_only` | Whether to search only for directly connected chips (vs. including remote chips) |
| `chip_filter` | Filter for specific chip architectures |
| `noc_safe` | Whether to avoid operations that might cause a NoC (Network on Chip) hang |

Sources: [crates/luwen-if/src/detect_chips.rs 175-218](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L175-L218)

These options allow for customizing the detection process based on the needs of the application.

## Error Handling

The core components use a robust error handling system:

Sources: [crates/luwen-if/src/chip/mod.rs 263-273](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L263-L273)[crates/luwen-if/src/detect_chips.rs 251-416](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/detect_chips.rs#L251-L416)

Three levels of error handling are provided:

1.   `ChipInitResult`: Used during initialization to indicate whether it's safe to continue
2.   `InitError`: Wraps platform errors and callback errors
3.   `PlatformError`: Represents low-level errors in the platform

This multi-layered approach allows for fine-grained control over error handling, enabling recovery from non-critical errors while ensuring safety when critical errors occur.
