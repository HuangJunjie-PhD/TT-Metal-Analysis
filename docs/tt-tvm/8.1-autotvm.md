## Purpose and Scope

AutoTVM is TVM's template-based auto-tuning system that automatically searches for optimal operator implementations by exploring parameterized schedule templates. It is one of three auto-tuning approaches in TVM, alongside AutoScheduler ([8.2](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler)) which is template-free, and Meta Schedule ([8.3](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule)) which provides advanced scheduling infrastructure.

AutoTVM focuses on finding optimal configuration parameters for manually-written schedule templates defined in the TOPI library. These templates expose tunable parameters (e.g., tile sizes, unroll factors, vectorization widths) that AutoTVM explores through search algorithms and performance measurement.

**Key characteristics:**

*   **Template-based**: Requires manually written schedule templates with tunable parameters
*   **TOPI Integration**: Works with existing operator implementations in `tvm.topi`
*   **Task-level Tuning**: Tunes individual operators or kernels independently
*   **Cross-platform**: Supports local and remote measurement via RPC ([8.4](https://deepwiki.com/tenstorrent/tt-tvm/8.4-rpc-infrastructure))

For template-free auto-tuning that generates schedules automatically from compute definitions, see AutoScheduler ([8.2](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler)).

## System Architecture

AutoTVM operates as a layer between Relay IR optimization and TIR code generation. It tunes TIR-level operators by exploring template parameter configurations and measuring their actual performance on target hardware.

## Core Components

### Task Extraction

Tasks are extracted from a Relay program by identifying all TOPI operator calls that have registered schedule templates. Each unique operator configuration (shape, dtype, target) becomes a tuning task.

Task weights represent the number of times an operator appears in the network, used to prioritize tuning effort.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 258-278](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L258-L278) (AutoScheduler equivalent shows similar pattern)

### Template and ConfigSpace

Templates are schedule implementations with tunable parameters exposed through a `ConfigSpace`. Each template defines:

1.   **Compute Definition**: The TOPI compute (e.g., `topi.nn.conv2d`)
2.   **Schedule Template**: Parameterized schedule transformations
3.   **ConfigSpace**: Valid ranges for tunable parameters

**Example ConfigSpace parameters:**

*   `tile_x`, `tile_y`, `tile_k`: Tile sizes for computation tiling
*   `num_output`: Output vectorization width
*   `unroll_explicit`: Loop unrolling factor
*   `use_tensorcore`: Whether to use Tensor Core instructions

Sources: [NEWS.md 172-178](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/NEWS.md?plain=1#L172-L178) TOPI library structure from high-level diagrams

## Search Algorithms

AutoTVM provides multiple search algorithms (tuners) that explore the ConfigSpace efficiently:

| Tuner Class | Strategy | Best Use Case |
| --- | --- | --- |
| `XGBTuner` | Machine learning cost model (XGBoost) | Large search spaces (default) |
| `GATuner` | Genetic algorithm | Medium search spaces |
| `RandomTuner` | Random sampling | Quick exploration |
| `GridSearchTuner` | Exhaustive grid search | Small search spaces |

## Measurement Infrastructure

AutoTVM measures actual execution time on target hardware through a builder-runner architecture:

### Builder Options

| Builder | Description | Use Case |
| --- | --- | --- |
| `LocalBuilder` | Compiles on local machine | Same-architecture tuning |
| `LocalBuilder(build_func="ndk")` | Uses Android NDK compiler | Cross-compilation for Android |
| `LocalBuilder(build_func="default")` | Uses default compiler | Standard cross-compilation |

### Runner Options

| Runner | Description | Use Case |
| --- | --- | --- |
| `LocalRunner` | Executes on local machine | CPU tuning on same host |
| `RPCRunner` | Executes on remote device via RPC | Remote device tuning (ARM, GPU, etc.) |

## Tuning Logs and Replay

### Log File Format

AutoTVM stores tuning results in JSON format with one record per line (JSON Lines format). Each record contains:

*   **Configuration**: Parameter values for the schedule template
*   **Result**: Measured execution time and status
*   **Workload**: Operator signature and arguments
*   **Target**: Hardware target specification

### Log Management Operations

| Operation | Purpose | Usage |
| --- | --- | --- |
| `RecordToFile(log_file)` | Save measurements during tuning | Callback in TuningOptions |
| `ApplyHistoryBest(log_file)` | Load best schedules for compilation | Context manager during build |
| `load_from_file(log_file)` | Query historical results | Programmatic access |
| Distill logs | Remove redundant records | `python -m tvm.autotvm.measure_record --mode distill` |

### Replay Workflow

**Example replay:**

`# Apply previously tuned scheduleswith autotvm.apply_history_best(log_file):    with tvm.transform.PassContext(opt_level=3):        lib = relay.build(mod, target=target, params=params)`

## RPC Integration

AutoTVM integrates with TVM's RPC infrastructure ([8.4](https://deepwiki.com/tenstorrent/tt-tvm/8.4-rpc-infrastructure)) to enable tuning on remote devices, which is essential for:

*   Cross-compilation scenarios (tune x86 host for ARM device)
*   GPU tuning from CPU host
*   Mobile device tuning (Android, iOS)
*   Embedded device tuning

### RPC Setup Components

### Starting RPC Infrastructure

**1. Start RPC Tracker:**

`python -m tvm.exec.rpc_tracker --host=0.0.0.0 --port=9190`
**2. Register Device:**

`# On remote devicepython -m tvm.exec.rpc_server --tracker=[HOST_IP]:9190 --key=device_name`
**3. Query Available Devices:**

`python -m tvm.exec.query_rpc_tracker --host=0.0.0.0 --port=9190`
**4. Configure RPCRunner:**

`runner = autotvm.RPCRunner(    key='device_name',           # Device key from tracker    host='127.0.0.1',           # Tracker host    port=9190,                  # Tracker port    timeout=30,                 # Measurement timeout    repeat=3,                   # Repeat measurements    min_repeat_ms=200,          # Minimum repeat duration    enable_cpu_cache_flush=True # Flush cache for accuracy)`
Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 154-222](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L154-L222) (RPC tracker setup), [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 309-318](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L309-L318) (RPCRunner configuration), [gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 150-160](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L150-L160) (device key setup)

## Comparison with AutoScheduler

AutoTVM and AutoScheduler serve similar purposes but with different approaches:

| Aspect | AutoTVM | AutoScheduler ([8.2](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler)) |
| --- | --- | --- |
| **Approach** | Template-based | Template-free |
| **Search Space** | Defined by template parameters | Generated from compute DAG |
| **Manual Effort** | Requires writing schedule templates | Only requires compute definition |
| **Flexibility** | Limited to template parameters | Explores wider space |
| **Operator Coverage** | Requires template per operator | Handles any TOPI compute |
| **Maturity** | Mature, production-ready | Newer, evolving |
| **Best For** | Well-understood operators | Novel operators, research |

### Migration Path

For operators with existing TOPI templates, AutoTVM provides immediate tuning capability. For new operators or research workloads, AutoScheduler ([8.2](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler)) can generate schedules without template writing. Meta Schedule ([8.3](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule)) unifies both approaches in a modern framework.

Sources: [NEWS.md 26](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/NEWS.md?plain=1#L26-L26) (AutoTVM, AutoScheduler, Meta Schedule section), [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 42-45](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L42-L45) (comparison explanation)

## Integration with Compilation Pipeline

AutoTVM integrates at multiple points in the TVM compilation flow:

### Dispatch Context Mechanism

The dispatch context determines which schedule template is used for each operator based on:

1.   Target hardware (CPU, GPU, DSP)
2.   Operator type and shape
3.   Available tuning logs

**Without tuning:**

*   Uses default TOPI schedule (generic, not optimized)

**With tuning (ApplyHistoryBest):**

*   Looks up operator in tuning logs
*   Applies best measured configuration
*   Falls back to default if not found

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 323-329](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L323-L329) shows compilation with history, High-level architecture (Diagram 1)

## Best Practices

### Tuning Budget Allocation

For network tuning, recommended number of trials:

```
num_measure_trials = 800 * len(tasks)
```

This typically allows the search to converge. Adjust based on:

*   Search space size (more parameters = more trials needed)
*   Time budget constraints
*   Hardware availability

### Multi-Device Parallelization

Scale tuning by registering multiple devices to RPC tracker:

All devices run measurements concurrently, sharing results through the tracker.

### Log File Management

**During tuning:**

*   Use `RecordToFile` callback to continuously save results
*   Allows resumption if tuning interrupted
*   Provides checkpoints for partial results

**After tuning:**

*   Distill logs to remove redundant records
*   Version control tuned logs for reproducibility
*   Share logs across team for consistent performance

**Resume tuning:**

`# Resume from previous logtuner = autotvm.tuner.XGBTuner(task, load_log_file='previous.json')`

## Debugging and Troubleshooting

### Common Issues

**1. Compilation Errors During Tuning:**

*   Invalid configurations may cause build failures
*   AutoTVM isolates these errors and continues
*   Check `MeasureErrorNo` codes in logs

**2. Slow Measurement:**

*   Increase `min_repeat_ms` for stable measurements
*   Use `enable_cpu_cache_flush=True` for CPU accuracy
*   Check network latency for RPC measurements

**3. No Performance Improvement:**

*   Search space may be too small (add more parameters)
*   Default schedule may already be optimal
*   Try different tuner (XGBTuner vs GATuner)

### Measurement Callbacks

AutoTVM supports custom callbacks for monitoring:

`measure_callbacks=[    autotvm.callback.RecordToFile(log_file),    autotvm.callback.progress_bar(total_trials),    autotvm.callback.log_to_file(log_file)]`
### Error Isolation

Failed measurements are logged with error codes and isolated from the main tuning process.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 413-416](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L413-L416) (error handling note), [gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 333-337](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L333-L337) (error isolation)

## Advanced Features

### Multi-Objective Tuning

Beyond latency optimization, AutoTVM can tune for:

*   **Memory footprint**: Minimize workspace allocation
*   **Power consumption**: Optimize for energy efficiency
*   **Multi-metric trade-offs**: Balance latency vs. memory

### Task Scheduling

For network-level tuning, the task scheduler allocates measurement trials across operators based on their impact on end-to-end latency:

```
Objective = Σ(latency[task] × weight[task])
```

where `weight[task]` = number of operator occurrences in network.

### Hierarchical Search

Some templates support hierarchical search:

1.   **Coarse search**: Explore major structural choices
2.   **Fine search**: Optimize tile sizes within structure
3.   **Local search**: Refine around best configuration

This reduces search space exploration time.

## Historical Context

AutoTVM was TVM's original auto-tuning system, contributed by the team including Lianmin Zheng and Eddie Yan. It established the foundation for:

*   Template-based performance optimization
*   RPC-based remote measurement
*   Log-based schedule management
*   Task extraction and scheduling

While AutoScheduler ([8.2](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler)) and Meta Schedule ([8.3](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule)) have expanded TVM's auto-tuning capabilities, AutoTVM remains widely used for:

*   Production deployments with mature templates
*   Operators with well-understood optimization patterns
*   Scenarios requiring predictable tuning time

The template system continues to be actively maintained with new hardware support (e.g., int4 Tensor Cores) added in recent releases.
