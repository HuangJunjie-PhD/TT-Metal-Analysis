This document describes the SFPI system for conditional execution on the SFPU hardware, including the `v_if`, `v_elseif`, `v_else`, and `v_endif` macro constructs, condition code management, and how comparisons and boolean operations work. For general vector operations, see [Vector Types and Operations](https://deepwiki.com/tenstorrent/sfpi/2.1-vector-types-and-operations). For instruction recording and replay, see [Instruction Replay Interface](https://deepwiki.com/tenstorrent/sfpi/2.3-instruction-replay-interface).

## Overview

The SFPU implements predicated execution rather than traditional branching. All code paths execute, but instructions are conditionally applied based on a condition code (CC) register that controls which vector elements are active. The SFPI API provides a C++-style conditional syntax that compiles to SFPU predicated execution at zero runtime overhead.

The predication system operates on a per-element basis across the 64-element SFPU vector. Each comparison generates a bitmask indicating which elements satisfy the condition, and this mask controls subsequent operations within the conditional block.

**Sources:**[include/sfpi.h 54-116](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L54-L116)

## Macro-Based Control Flow Interface

### Core Macros

The SFPI provides four macros that implement conditional control flow:

| Macro | Purpose | C++ Analog |
| --- | --- | --- |
| `v_if(condition)` | Begin conditional block | `if (condition) {` |
| `v_elseif(condition)` | Alternative conditional block | `} else if (condition) {` |
| `v_else` | Default block | `} else {` |
| `v_endif` | End conditional block | `}` |

These macros automatically manage the lifecycle of the `__vCCCtrl` object and balance braces for proper scoping.

## Condition Code Control Classes

### The `__vCCCtrl` Class

The `__vCCCtrl` class manages the hardware condition code stack. It tracks the nesting level of conditionals and ensures proper stack management.

| Method | Purpose |
| --- | --- |
| `cc_if(const __vCond& op)` | Begin conditional based on condition result |
| `cc_elseif(const __vCond& cond)` | Add alternative condition |
| `cc_else()` | Invert current condition (else block) |
| `mark_top()` | Record current stack depth |
| `push()` | Push current CC state to hardware stack |
| `pop()` | Pop CC state from hardware stack |
| `enable_cc()` | Enable condition code control globally |

The constructor marks the top of the CC stack, and the destructor automatically pops any pushed condition states, ensuring stack balance even if `v_endif` is omitted (though this generates compiler warnings).

## Boolean Operations

### AND, OR, NOT

The `__vCond` class supports boolean composition through operator overloading:

| Operator | Operation | Hardware Intrinsic | Modifier |
| --- | --- | --- | --- |
| `&&` | Logical AND | `__builtin_rvtt_sfpxbool` | `SFPXBOOL_MOD1_AND` |
| ` |  | ` | Logical OR |
| `!` | Logical NOT | `__builtin_rvtt_sfpxbool` | `SFPXBOOL_MOD1_NOT` |

### Complex Boolean Expressions

Boolean expressions are evaluated at compile-time into a tree structure, then emitted as a sequence of hardware instructions:

## Constraints and Limitations

### Compiler Warnings and Errors

The compiler enforces several constraints on predicated execution:

| Error/Warning | Cause | Solution |
| --- | --- | --- |
| `malformed program, sfpsetcc_v outside of pushc/popc` | Comparison instruction outside conditional context | Only use comparisons within `v_if` blocks or store in variables |
| `cannot write sfpu vector to memory` | Vector spilled to stack | Reduce register pressure or split function |
| `Illegal rvtt builtin found in conditional tree` | Invalid instruction in boolean expression | Move operation outside conditional expression |

**Sources:**[tests/wormhole/sfpi/gold/warn.txt 42-75](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L42-L75)

### Register Pressure

Excessive nesting or complex conditionals can exhaust the 8 available LREGs. The compiler will issue an error if it cannot allocate registers:

```
fatal error: cannot store sfpu register (register spill)
```

To avoid this, minimize the number of live variables within conditional blocks or split complex functions.

**Sources:**[tests/wormhole/sfpi/gold/warn_spill.txt 1-11](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn_spill.txt#L1-L11)

## Summary

The SFPI predicated execution system provides:

*   **Natural C++-style syntax** through `v_if`/`v_elseif`/`v_else`/`v_endif` macros
*   **Zero runtime overhead** via compile-time expansion to hardware intrinsics
*   **Automatic CC stack management** through RAII-based `__vCCCtrl` class
*   **Rich comparison operations** for floating-point and integer types
*   **Boolean composition** with `&&`, `||`, and `!` operators
*   **Per-element execution control** across 64-element SFPU vectors

Understanding these mechanisms enables efficient implementation of complex kernel logic while maintaining clean, readable source code.
