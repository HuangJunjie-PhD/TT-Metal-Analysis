This document describes how tt-llk is integrated as a submodule within the tt-metal repository and the coordination mechanisms required to maintain compatibility between the two codebases. It covers dependency management, header synchronization, version coordination, and the workflows for handling breaking changes.

For information about the breaking change workflow process, see [Breaking Change Workflow](https://deepwiki.com/tenstorrent/tt-llk/11.2-breaking-change-workflow). For general contribution guidelines, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/tt-llk/11-contributing-guidelines).

## Overview of the Submodule Relationship

tt-llk serves as a low-level kernel library that is consumed by tt-metal through a git submodule integration. This architecture allows tt-llk to maintain independent development while providing stable interfaces for tt-metal's higher-level operations.

**Sources:**[.github/workflows/on-pr.yml 1-224](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L1-L224)[tests/python_tests/conftest.py 1-421](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L1-L421)

## Dependency Architecture

### External Package Dependencies

tt-llk requires several packages from the Tenstorrent ecosystem that are published through PyPI and test.pypi.org:

| Package | Version | Source | Purpose |
| --- | --- | --- | --- |
| `tt-exalens` | 0.3.6 | test.pypi.org | Hardware control and device communication |
| `tt-umd` | 0.9.1 | test.pypi.org | Unified Memory Descriptor for device access |
| `tt-smi` | 3.1.1 | pypi.org | System Management Interface |

These dependencies provide the runtime interface for controlling Tenstorrent hardware and are version-pinned to ensure compatibility.

**Sources:**[tests/requirements.txt 1-49](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/requirements.txt#L1-L49)

### Hardware-Specific Headers

tt-llk requires architecture-specific hardware headers from tt-metal that define register layouts, memory maps, and hardware configurations. These are not part of the tt-llk repository but must be downloaded during setup:

The verification function `check_hardware_headers()` runs during pytest initialization to ensure all required headers are present:

**Sources:**[tests/python_tests/conftest.py 29-83](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L29-L83)

#### Header Verification Process

The function checks for architecture-specific headers based on the `CHIP_ARCH` environment variable:

**Wormhole/Blackhole Required Headers:**

*   `core_config.h`
*   `cfg_defines.h`
*   `dev_mem_map.h`
*   `tensix.h`
*   `tensix_types.h`

**Quasar Additional Headers:**

*   `t6_debug_map.h`
*   `t6_mop_config_map.h`
*   `tt_t6_trisc_map.h`

If headers are missing, pytest exits with an error message directing the user to run `./setup_testing_env.sh`.

**Sources:**[tests/python_tests/conftest.py 29-83](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L29-L83)

## Build System Integration

### Compilation Include Paths

The build system must coordinate include paths to access both tt-llk internal headers and externally-provided headers from tt-metal:

**Sources:**[tests/python_tests/helpers/test_config.py 314-325](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L314-L325)

The include paths are structured as:

| Path Pattern | Purpose |
| --- | --- |
| `-Isfpi/include` | SFPI compiler headers |
| `-I../{ARCH_LLK_ROOT}/llk_lib` | High-level LLK API headers |
| `-I../{ARCH_LLK_ROOT}/common/inc` | Core ckernel abstractions |
| `-I../{ARCH_LLK_ROOT}/common/inc/sfpu` | SFPU operation definitions |
| `-I../common` | Shared common headers |
| `-I{HEADER_DIR}` | Downloaded hardware-specific headers |
| `-Ihw_specific/{ARCH}/` | Architecture-specific overrides |
| `-Ihw_specific/{ARCH}/metal_sfpu` | Metal SFPU implementations |
| `-Ifirmware/riscv/common` | RISC-V firmware interfaces |
| `-Ihelpers/include` | Test infrastructure headers |

### Linker Script Coordination

Memory layout definitions differ between normal and coverage builds, requiring separate linker scripts:

**Normal Build:**

*   `memory.{arch}.ld` - Standard memory layout

**Coverage Build:**

*   `memory.{arch}.debug.ld` - Extended layout with gcov data sections

**Sources:**[tests/python_tests/helpers/test_config.py 608-634](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L608-L634)[tests/helpers/ld/memory.wormhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L1-L68)[tests/helpers/ld/memory.blackhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L1-L68)

The debug linker scripts allocate additional memory regions for coverage instrumentation:

**Wormhole Debug Layout:**

*   TRISC0_GCOV_MEM: 32K @ 0x16000
*   TRISC1_GCOV_MEM: 32K @ 0x30000
*   TRISC2_GCOV_MEM: 32K @ 0x4A000

**Blackhole Debug Layout:**

*   TRISC0_GCOV_MEM: 8K @ 0x6F000
*   TRISC1_GCOV_MEM: 8K @ 0x8D000
*   TRISC2_GCOV_MEM: 8K @ 0xAB000

## Version Coordination Strategy

### Submodule Update Process

When tt-metal needs to update its tt-llk submodule:

1.   **Developer makes changes in tt-llk repository**

2.   **Changes are reviewed and merged to main**

3.   **tt-metal developer updates submodule reference:**

`cd tt-metal/tt_llkgit pull origin maincd ..git add tt_llkgit commit -m "Update tt-llk submodule to <commit-hash>"`
4.   **tt-metal CI validates integration**

5.   **PR merged to tt-metal main**

### API Compatibility Requirements

To maintain compatibility during independent development:

1.   **Interface Stability:** Public LLK headers maintain backward compatibility
2.   **Versioned Dependencies:**`requirements.txt` pins specific versions of tt-exalens and tt-umd
3.   **Feature Flags:** New features use compile-time flags for opt-in adoption
4.   **Deprecation Process:** Breaking changes follow the workflow described in [Breaking Change Workflow](https://deepwiki.com/tenstorrent/tt-llk/11.2-breaking-change-workflow)

## Development Workflow

### Local Development Setup

Developers working on tt-llk within a tt-metal context should:

1.   **Clone tt-metal with submodules:**

`git clone --recurse-submodules https://github.com/tenstorrent/tt-metal.gitcd tt-metal/tt_llk`
2.   **Set up tt-llk environment:**

`export CHIP_ARCH=wormhole  # or blackhole, quasarcd tests./setup_testing_env.sh`
3.   **Run tests to validate:**

`cd python_testspytest -n 4 -m "not perf and not nightly"`

### Making Breaking Changes

When changes to tt-llk require corresponding changes in tt-metal:

1.   **Create feature branch in tt-llk**
2.   **Open draft PR in tt-llk describing changes**
3.   **Create corresponding branch in tt-metal** that updates submodule reference
4.   **Test integration in tt-metal branch**
5.   **Mark tt-llk PR as ready when tt-metal changes are validated**
6.   **Coordinate merge timing** (see [Breaking Change Workflow](https://deepwiki.com/tenstorrent/tt-llk/11.2-breaking-change-workflow))

### Testing Integration

Developers can test their tt-llk changes against tt-metal without merging:

`# In tt-metal repositorycd tt_llkgit remote add dev-fork https://github.com/user/tt-llk.gitgit fetch dev-forkgit checkout dev-fork/feature-branch # Test tt-metal with the development branchcd .../build_metal.sh  # or equivalent build command./run_tests.sh`

### Performance Data Collection

Performance test artifacts are architecture-specific and collected independently:

**Sources:**[.github/workflows/setup-and-test.yml 167-188](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L167-L188)

The artifacts are named with architecture prefixes:

*   `perf-data-wormhole-{group}`
*   `perf-data-blackhole-{group}`

### Coverage Data Coordination

Coverage builds generate `.info` files that are merged and uploaded:

**Sources:**[.github/workflows/setup-and-test.yml 217-275](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L217-L275)

The merge process:

1.   Each test group generates `code-coverage-{arch}-{group}.info`
2.   `merge-coverage` job combines all groups for an architecture
3.   Final artifact uploaded as `code_coverage_for_{arch}`
4.   Uploaded to Codecov with architecture-specific naming

## Gitignore Coordination

The `.gitignore` file explicitly excludes hardware-specific headers and build artifacts:

**Sources:**[.gitignore 18-19](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.gitignore#L18-L19)

This ensures that:

*   Downloaded headers from tt-metal are not committed
*   Build artifacts remain local
*   Submodule status stays clean

Key ignored patterns:

*   `**/hw_specific/**` - All hardware headers
*   `tests/helpers/include/build.h` - Generated configuration header
*   `**/fused_tests/**` - Generated test code
*   `perf_data/` - Performance test outputs
