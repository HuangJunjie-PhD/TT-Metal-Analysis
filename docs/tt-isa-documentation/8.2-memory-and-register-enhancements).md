This page documents memory and register system improvements introduced in Blackhole A0 compared to Wormhole B0. The primary enhancements center on the `Dst` register addressing system, with additional improvements to the `LReg` register file architecture.

For information about SFPU computational improvements, see [SFPU and FMA Improvements](https://deepwiki.com/tenstorrent/tt-isa-documentation/8.1-sfpu-and-fma-improvements). For general information about the `Dst` and `LReg` register architectures and data types, see [Register Files: SrcA, SrcB, Dst](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst).

## Overview of Enhancements

Blackhole A0 introduces the following memory and register enhancements:

### Dst Register Address Transformations

1.   **Address Remapping** via `Config.DEST_ACCESS_CFG_remap_addrs` - Bit-level address transformation for improved memory access patterns
2.   **32-bit Swizzling** via `Config.DEST_ACCESS_CFG_swizzle_32b` - Additional transformation for 32-bit data modes

These transformations affect all accesses to `Dst`:

*   Matrix Unit (FPU) operations via `MVMUL`, `MOVA2D`, `MOVD2A`, etc.
*   Vector Unit (SFPU) operations via `SFPLOAD`, `SFPSTORE`
*   Packer input address generation via `PACR`
*   RISC-V memory-mapped access at `0xFFBD_8000`

### LReg Register File Enhancements

Blackhole adds `LReg[16]`, a 32-lane register accessible only through `SFPLOADMACRO`-scheduled instructions. This provides dedicated storage for macro-expanded vector operations without interfering with the standard `LReg[0:7]` working registers.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 1-40](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L1-L40)[BlackholeA0/TensixTile/TensixCoprocessor/LReg.md 1-17](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/LReg.md?plain=1#L1-L17)

## Dst Address Transformation Architecture

### Address Mapping Functions

In Blackhole A0, the relationship between logical row indices and physical storage has been enhanced with configurable transformation functions:

**Wormhole B0 Behavior:** In Wormhole, `Adj16(r)` simply returns `r` unchanged, and `Adj32(r)` performs only the basic interleaving transformation:

`// Wormhole B0uint10_t Adj32(uint10_t Row) {  return ((Row & 0x1f8) << 1) | (Row & 0x207);}`
**Blackhole A0 Behavior:** In Blackhole, both functions support additional transformations:

`// Blackhole A0uint10_t Adj16(uint10_t r) {  if (Config.DEST_ACCESS_CFG_remap_addrs) {    r = (r & 0x3c7) ^ ((r & 0x030) >> 1) ^ ((r & 0x008) << 2);  }  return r;} uint10_t Adj32(uint10_t r) {  r = Adj16(r);  if (Config.DEST_ACCESS_CFG_swizzle_32b) {    r = (r & 0x3f3) ^ ((r & 0x018) >> 1) ^ ((r & 0x004) << 1);  }  return ((r & 0x1f8) << 1) | (r & 0x207);}`
**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 21-37](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L21-L37)[WormholeB0/TensixTile/TensixCoprocessor/Dst.md 21-26](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L21-L26)

### Address Remap Transformation Detail

The `remap_addrs` transformation applies XOR-based bit permutation to the row index before physical addressing:

This transformation preserves bits 9-6 and bits 2-0 unchanged, while bits 5-3 undergo controlled XOR mixing with each other. The specific bit manipulation pattern is designed to improve bank conflict patterns for certain access sequences.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 23-26](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L23-L26)

### 32-bit Swizzle Transformation Detail

The `swizzle_32b` transformation applies an additional XOR-based permutation after address remapping, specifically tailored for 32-bit data access patterns:

| Configuration | Transformation Applied |
| --- | --- |
| `remap_addrs=0, swizzle_32b=0` | None (Wormhole-compatible) |
| `remap_addrs=1, swizzle_32b=0` | Address remapping only |
| `remap_addrs=0, swizzle_32b=1` | 32-bit swizzle only (uncommon) |
| `remap_addrs=1, swizzle_32b=1` | Both transformations |

The swizzle operates on the row index after `Adj16` has been applied:

`// Applied within Adj32, after Adj16if (Config.DEST_ACCESS_CFG_swizzle_32b) {  r = (r & 0x3f3) ^ ((r & 0x018) >> 1) ^ ((r & 0x004) << 1);}`
This preserves bits 9-6 and bits 1-0, while mixing bits 5-2 through XOR operations.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 30-35](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L30-L35)

## Impact on Different Access Paths

### Dst16b vs Dst32b Access

The transformation functions are invoked differently depending on whether 16-bit or 32-bit view is being used:

**Key Points:**

*   `Dst16b[Row][Col]` uses `Adj16(Row)` to compute physical address
*   `Dst32b[Row][Col]` uses `Adj32(Row)`, which internally calls `Adj16(Row)` first
*   Both transformation modes can be enabled independently
*   The `swizzle_32b` bit only affects `Dst32b` access paths

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 15-17](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L15-L17)

### Impact on Packer Input Address Generation

The configuration bits affect how packers compute input addresses when reading from `Dst`:

`// From packer input address generatorif (i == 0 && PackerIConfig.Source_interface_selection == 1) {  // Packer 0 reading from L1 - transformations don't apply  PackerI.InputSource = INPUT_SOURCE_L1;} else {  // Packers reading from Dst - transformations DO apply  Addr = ((Addr / BytesPerDatum) & ~ADC_X_Mask) + (ADC.X & ADC_X_Mask);  Addr += Config.DEST_TARGET_REG_CFG_PACK_SEC[i].Offset << 4;  PackerI.InputSource = INPUT_SOURCE_Dst;  PackerI.InputSourceAddr = Addr & 0x3fff; // Will be transformed by Adj16/Adj32}`
When packers read from `Dst`, they use the same `Adj16`/`Adj32` functions based on the data format, meaning the address transformations are transparent to the packer logic but affect which physical rows are accessed.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 38-40](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L38-L40)[WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 46-59](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L46-L59)

### Impact on RISC-V Access

RISC-V cores access `Dst` through memory-mapped interface starting at `0xFFBD_8000`. The transformation functions are applied transparently during these accesses:

`// RISC-V Load16 (simplified)uint16_t Load16(uint32_t Addr) {  Addr = (Addr - 0xFFBD8000) / 2;  uint16_t Value = Dst16b[Addr / 16][Addr % 16]; // Adj16 applied here  // ... format conversion based on fmt ...  return Value;} // RISC-V Load32 (simplified)uint32_t Load32(uint32_t Addr) {  Addr = (Addr - 0xFFBD8000) / 4;  uint32_t Value = Dst32b[Addr / 16][Addr % 16]; // Adj32 applied here  // ... format conversion based on fmt ...  return Value;}`
The RISC-V core sees a linear address space, but the `remap_addrs` and `swizzle_32b` configuration affects which physical rows are accessed. This is completely transparent to RISC-V code—the same address will access the same logical data, but the physical storage location may differ.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 99-146](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L99-L146)

## Configuration and Control

### Backend Configuration Fields

The two new configuration bits are part of the Tensix backend configuration system:

| Field | Location | Width | Purpose |
| --- | --- | --- | --- |
| `Config.DEST_ACCESS_CFG_remap_addrs` | `Config[StateID]` | 1 bit | Enable address remapping transformation |
| `Config.DEST_ACCESS_CFG_swizzle_32b` | `Config[StateID]` | 1 bit | Enable 32-bit swizzle transformation |

These fields are accessed through the standard configuration mechanism:

**Configuration Access:**

*   **RISC-V writes:** Use `sw` instructions to `TENSIX_CFG_BASE` + offset
*   **Tensix writes:** Use `WRCFG`, `RMWCIB`, or `REG2FLOP` instructions
*   **Implicit reads:** Automatically applied during any `Dst` access
*   **Explicit reads:** Use `RDCFG` instruction from Tensix threads

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 23-40](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L23-L40)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 1-33](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L1-L33)

### Race Condition Considerations

As with all backend configuration, care must be taken when modifying these bits:

**Danger:** Writing to `Config` from RISC-V, followed by pushing a Tensix instruction that will access `Dst`, can result in the instruction executing with stale configuration if memory ordering is not enforced.

**Recommended approach:**

`// RISC-V codesw t0, DEST_ACCESS_CFG_OFFSET(t1);  // Write new configlw t2, DEST_ACCESS_CFG_OFFSET(t1);  // Read back (ensures write finished)addi x0, t2, 0;                      // Consume result (prevents reordering)sw t3, TENSIX_PUSH_ADDR(t4);        // Now safe to push Tensix instruction`
**Alternative:** Use Tensix `WRCFG` or `RMWCIB` instructions instead of RISC-V `sw` to avoid race conditions, as Tensix instructions execute in-order within a thread.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 34-50](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L34-L50)

## Practical Use Cases

### Memory Access Pattern Optimization

The address remapping features enable optimization of specific access patterns:

1.   **Bank Conflict Reduction:** The XOR-based remapping can distribute consecutive logical addresses to different physical banks, reducing conflicts when multiple clients access nearby rows
2.   **Stride Access Optimization:** Certain strided access patterns benefit from the bit mixing, as logical strides map to physical patterns that avoid hotspots
3.   **32-bit Data Alignment:** The `swizzle_32b` transformation specifically optimizes for 32-bit data access patterns where pairs of 16-bit rows need to be accessed together

### Example: Dst32b with Swizzling

Consider accessing `Dst32b` with both transformations enabled:

| Logical Row | `Adj16(r)` with remap | After `swizzle_32b` | Final after interleave | Physical Rows Accessed |
| --- | --- | --- | --- | --- |
| 0x000 | 0x000 | 0x000 | 0x000 | 0x000, 0x008 |
| 0x001 | 0x001 | 0x001 | 0x201 | 0x201, 0x209 |
| 0x004 | 0x006 | 0x004 | 0x008 | 0x008, 0x010 |
| 0x008 | 0x028 | 0x020 | 0x040 | 0x040, 0x048 |

The complex remapping ensures that sequential logical accesses map to physical addresses with better distribution characteristics.

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 21-37](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L21-L37)

## Compatibility Considerations

### Wormhole B0 Compatibility Mode

To maintain compatibility with Wormhole B0 behavior, simply ensure both configuration bits are set to 0:

`Config.DEST_ACCESS_CFG_remap_addrs = 0;Config.DEST_ACCESS_CFG_swizzle_32b = 0;`
In this mode:

*   `Adj16(r)` returns `r` unchanged
*   `Adj32(r)` performs only the standard interleaving
*   All logical row indices map directly to physical rows (for 16-bit mode)
*   Blackhole behaves identically to Wormhole for all `Dst` accesses

### Software Considerations

**Kernel Code:**

*   Kernels must be aware of the active configuration when accessing `Dst`
*   RISC-V code reading/writing `Dst` sees the same logical view regardless of configuration
*   Tensix instructions (FPU, SFPU, Packers) automatically apply transformations

**Debugging:**

*   When examining `Dst` contents via debug interfaces, remember that addresses are transformed
*   Use `RDCFG` to check current configuration state
*   Consider disabling transformations during initial debugging
