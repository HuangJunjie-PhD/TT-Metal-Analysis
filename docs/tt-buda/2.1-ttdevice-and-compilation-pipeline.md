This page documents the core PyBuda TTDevice class and the complete compilation pipeline that transforms high-level models (PyTorch, TensorFlow, etc.) into executable code for Tenstorrent hardware. For information about the Balancer and Placement system that optimizes resource usage, see [Balancer and Placement System](https://deepwiki.com/tenstorrent/tt-buda/2.2-balancer-and-placement-system).

## TTDevice Overview

The `TTDevice` class is the primary interface between PyBuda and Tenstorrent hardware. It represents one or more physical Tenstorrent chips and is responsible for compiling and executing PyBuda modules.

Sources: [pybuda/pybuda/ttdevice.py 45-166](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/ttdevice.py#L45-L166)[pybuda/pybuda/backend.py 33-166](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/backend.py#L33-L166)

### Key Properties of TTDevice

| Property | Description |
| --- | --- |
| name | User-defined name for the device |
| num_chips | Number of chips the device spans |
| chip_ids | Specific chip IDs the device uses |
| arch | Hardware architecture (Grayskull, Wormhole_B0, etc.) |
| devtype | Backend type (Silicon, Golden, Versim, etc.) |
| device_mode | Compile and/or run mode |
| graph | Current computational graph |
| modules | Modules placed on the device |
| backend_api | Interface to backend compiler/runtime |

Sources: [pybuda/pybuda/ttdevice.py 50-165](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/ttdevice.py#L50-L165)

## Compilation Pipeline Flow

The compilation pipeline transforms high-level PyBuda modules into executable code for Tenstorrent hardware through a series of stages, from initial graph generation to backend compilation.

Sources: [pybuda/pybuda/compile.py 205-276](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L205-L276)[pybuda/pybuda/compile.py 723-1082](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L723-L1082)

## Detailed Compilation Stages

### 1. Initial Graph Generation

The compilation begins with generating a computational graph from PyBuda modules. This process converts module operations into a graph representation that can be optimized and transformed.

Key functions:

*   `generate_graph()` - Creates a graph from module operations
*   `create_op_node()` - Creates operation nodes in the graph
*   `create_data_edge()` - Creates edges between nodes
*   `register_module_inputs/outputs()` - Registers the I/O tensors

Sources: [pybuda/pybuda/ttdevice.py 368-720](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/ttdevice.py#L368-L720)

### 2. Post Initial Graph Passes

After the initial graph is created, several optimization passes are applied to transform the graph for better performance.

Sources: [pybuda/pybuda/compile.py 605-666](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L605-L666)

### 3. Graph Optimization

The optimization phase applies various transformations to improve performance, such as operator fusion, algebraic simplifications, and common subexpression elimination.

Sources: [pybuda/pybuda/compile.py 693-731](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L693-L731)

### 4. Autograd (Training)

For training models, the autograd pass adds operations for computing gradients.

Sources: [pybuda/pybuda/compile.py 733-763](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L733-L763)

### 5. Pre-Lowering Passes

These passes prepare the graph for lowering to Buda operations, handling operations specific to the Tenstorrent architecture.

Key passes:

*   Converting broadcast operations to tensor manipulations (TMs)
*   Inserting tile broadcast operations
*   Fusing bias operations
*   Fusing other operations like GELU

Sources: [pybuda/csrc/passes/pre_lowering_passes.cpp 1-147](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/pre_lowering_passes.cpp#L1-L147)[pybuda/csrc/passes/pre_lowering_passes.hpp 1-48](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/pre_lowering_passes.hpp#L1-L48)

### 6. Balancer and Placer

The balancer and placer determine how operations are distributed across hardware resources and placed on the chip grid.

Key components:

*   `BalancerConfig` - Controls balancing behavior
*   `OpOverride` - Allows manual override of operation parameters
*   `PolicyType` - Different placement policies (CNN, NLP, etc.)

Sources: [pybuda/csrc/balancer/balancer_config.hpp 1-143](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/balancer/balancer_config.hpp#L1-L143)[pybuda/pybuda/compile.py 866-1034](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L866-L1034)

### 7. Netlist Generation

The netlist generation phase creates a Buda netlist representation of the balanced and placed graph.

Sources: [pybuda/pybuda/compile.py 1063-1086](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L1063-L1086)

### 8. Backend Compilation

The backend compilation phase compiles the netlist to an executable form for the Tenstorrent hardware.

Sources: [pybuda/pybuda/compile.py 1088-1145](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/compile.py#L1088-L1145)[pybuda/pybuda/torch_compile.py 78-181](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/torch_compile.py#L78-L181)

## Runtime Execution Process

Once a model is compiled, it can be executed on the target hardware. This section describes the runtime execution process.

Key functions:

*   `push_tensor()` - Pushes input tensors to device memory
*   `dispatch()` - Runs compiled programs on the device
*   `pop_tensor()` - Retrieves output tensors from device memory

Sources: [pybuda/csrc/tt_torch_device/tt_device.cpp 193-465](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/tt_torch_device/tt_device.cpp#L193-L465)

### Program Execution

The TTDevice and Compilation Pipeline form the core of PyBuda's ability to transform high-level models into efficient executables for Tenstorrent hardware.
