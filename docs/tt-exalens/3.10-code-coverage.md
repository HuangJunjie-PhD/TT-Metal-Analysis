This page documents the code coverage extraction feature in TTExaLens: how firmware is compiled with GCOV-compatible instrumentation, how instrumented ELFs are loaded and run on device, and how the `coverage` API function extracts live coverage data from device memory into standard `.gcda` files that can be processed by external tools (gcov, lcov, genhtml).

For details on ELF loading mechanics, see [ELF Management](https://deepwiki.com/tenstorrent/tt-exalens/3.5-elf-management). For the underlying ElfLoader implementation, see [ELF Loader Implementation](https://deepwiki.com/tenstorrent/tt-exalens/6.6-elf-loader-implementation).

* * *

## Overview

RISC-V firmware running on Tenstorrent hardware cannot use the standard GCOV runtime (which depends on libc and a full C runtime). Instead, TTExaLens uses the GCC `-fprofile-info-section` flag to embed coverage metadata directly into a special ELF section (`.gcov_info`). After the firmware runs on device, TTExaLens reads the live coverage counters from device memory and serializes them into a standard `.gcda` file.

## Coverage Build Type

The firmware CMake build system (`riscv-src/CMakeLists.txt`) defines three build types: `debug`, `release`, and `coverage`. The `coverage` type compiles source files with the following flags:

| CMake Variable | Flags |
| --- | --- |
| `GCOV_FLAGS` | `-fprofile-arcs -ftest-coverage -fprofile-info-section -DCOVERAGE` |

The flag `-fprofile-info-section` is critical. Normally, GCC's GCOV instrumentation requires libc startup code to register coverage data structures with the runtime. On bare-metal RISC-V firmware this is unavailable. `-fprofile-info-section` instead embeds a pointer to the raw coverage data in a special named ELF section (`.gcov_info`), which can then be located and read by TTExaLens.

Two additional C source files are compiled and linked into every coverage build:

| Source file | Build flags | Role |
| --- | --- | --- |
| `riscv-src/coverage/gcov.c` | `release` (no instrumentation) | Implements the GCOV data serialization primitives on the firmware side |
| `riscv-src/coverage/coverage.c` | `release` + `-fno-strict-aliasing` | Traverses the `.gcov_info` structures and formats them for extraction |

Coverage builds use the `debug` linker script (`memory.${ARCH}.debug.ld`) rather than the release one, since the debug layout accommodates the additional data sections.

## `.gcov_info` ELF Section

When firmware is compiled in the `coverage` build type, GCC adds a `.gcov_info` section to the resulting ELF. This section contains pointers to all per-translation-unit GCOV data structures (counters, filenames, function checksums, etc.).

The `ElfLoader` class explicitly includes `.gcov_info` in its list of sections to be loaded to device memory:

```
SECTIONS_TO_LOAD = [".init", ".text", ".ldm_data", ".gcov_info"]
```

This means that when a coverage ELF is loaded with `load_elf` or `run_elf`, the `.gcov_info` data is written to L1 memory alongside the firmware code. The firmware's coverage counter arrays are also written into L1, and GCC's instrumentation updates them in-place as code paths execute.

Sources: [ttexalens/elf_loader.py 23](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/elf_loader.py#L23-L23)

## `coverage` API Function

The `coverage` function is defined in `ttexalens/tt_exalens_lib.py` and is decorated with `@trace_api`.

**Signature:**

**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `location` | `str | OnChipCoordinate` | NOC coordinate of the core whose coverage data is to be extracted |
| `elf` | `str | ParsedElfFile` | The coverage-instrumented ELF that was loaded onto the core |
| `gcda_path` | `str` | Output path for the generated `.gcda` file |
| `gcno_copy_path` | `str | None` | Optional path to copy the `.gcno` file (generated at compile time alongside the object) |

The function locates the `.gcov_info` section in the running firmware's L1 memory via the parsed ELF's symbol table, reads the live counter values from the device, and writes a binary `.gcda` file in the format expected by the standard `gcov` tool.

## Relationship to ELF Loading

The `ElfLoader.SECTIONS_TO_LOAD` list [ttexalens/elf_loader.py 23](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/elf_loader.py#L23-L23) is the join point between the build system and the coverage extraction. A regular (non-coverage) ELF simply has no `.gcov_info` section; it is silently skipped. A coverage ELF has this section and it is written to device memory as part of the standard load process with no extra user steps.

The `load_elf` and `run_elf` library functions (see page [3.5](https://deepwiki.com/tenstorrent/tt-exalens/3.5-elf-management)) work identically for both coverage and non-coverage ELFs. The only caller-visible difference is that after a coverage ELF runs, the `coverage()` function can be used to extract the results.

## Build System Details

The `riscv-src/CMakeLists.txt` creates one ELF target per combination of architecture × application × core × build type. For coverage builds specifically:

*   Object files are compiled with `GCOV_FLAGS` appended to the standard compile options.
*   `gcov.o` and `coverage.o` are appended to `LINK_OBJECTS`.
*   The linker script used is `memory.${ARCH}.debug.ld` (not the coverage-specific one, since there is none).

The `cov_test` application in `RISCV_APPS` is the dedicated coverage test firmware. It is built for all supported architectures and cores in all three build types.

| Variable | Value |
| --- | --- |
| `RISCV_APPS` (includes) | `cov_test` |
| `RISCV_ARCHITECTURES` | `wormhole`, `blackhole` |
| Coverage-only objects | `gcov.o`, `coverage.o` |
| ELF output pattern | `build/riscv-src/{arch}/{app}.coverage.{core}.elf` |
