# Error Reference

This page provides a comprehensive reference for all error categories and messages produced by ttsim. Understanding these error types helps diagnose issues, determine appropriate remediation steps, and decide when to report bugs versus adjusting workload code.

For configuration-specific errors and environment variable issues, see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration). For issues related to numerical divergence from silicon, see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy). For reporting bugs or unimplemented features, see [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues).

## Error Categories Reference

The following table provides a complete reference of all error categories, their meanings, and recommended actions:

| Category | Severity | Meaning | Recommended Action |
| --- | --- | --- | --- |
| `UndefinedBehavior` | Critical | ISA specification violation; behavior is undefined per architecture documentation | Fix workload code to avoid violating ISA requirements |
| `UnpredictableValueUsed` | Critical | Workload reads or uses a value whose content is unpredictable per ISA specification | Fix workload code to avoid reading unpredictable values |
| `NonContractualBehavior` | Warning | Workload relies on behavior not guaranteed by ISA specification | Update workload to rely only on contractual guarantees |
| `UntestedFunctionality` | Blocked | Feature implemented but disabled due to insufficient test coverage | Wait for future release or report need for prioritization |
| `UnimplementedFunctionality` | Blocked | Feature not yet implemented but planned for future support | Wait for future release or report priority need |
| `UnsupportedFunctionality` | Blocked | Feature unlikely to be implemented without strong justification | Redesign workload to avoid this feature or provide use case |
| `MissingSpecification` | Blocked | Feature requires internal specification work before implementation | Report use case to help prioritize specification work |
| `SystemError` | Variable | Operating system error (file I/O, memory allocation, etc.) | Check system resources and OS error messages |
| `ConfigurationError` | Blocked | Environment variables, command-line options, or configuration files incorrect | Fix configuration per [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) page |
| `AssertionFailure` | Critical | Internal simulator bug; should never occur | Report bug with reproduction steps |

## ISA Specification Violations

These error categories indicate that the workload violates requirements defined in the Tenstorrent instruction set architecture specification. These are critical errors that must be fixed in the workload code.

### UndefinedBehavior

`UndefinedBehavior` indicates the workload has executed an operation or instruction sequence that violates ISA requirements. Per the ISA specification, the simulator is permitted to behave in any manner when undefined behavior occurs, including terminating execution.

**Common causes:**

*   Invalid instruction encoding
*   Illegal operand combinations
*   Constraint violations on instruction sequences
*   Out-of-bounds memory accesses
*   Violation of documented preconditions

**Remediation:** Review workload code against the ISA specification and eliminate the violating operation. See the [tt-isa-documentation glossary](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation%20glossary) for the precise definition of undefined behavior.

### UnpredictableValueUsed

`UnpredictableValueUsed` indicates the workload has read or computed with a value whose content is unpredictable according to the ISA specification. This commonly occurs when reading uninitialized memory, registers, or hardware state.

**Common causes:**

*   Reading from uninitialized registers
*   Reading memory that was never written
*   Reading hardware state before initialization
*   Using results from operations with unpredictable outputs

**Remediation:** Ensure all values are properly initialized before use. Trace data flow to identify where the unpredictable value originates.

### NonContractualBehavior

`NonContractualBehavior` warns that the workload relies on behavior not guaranteed by the ISA specification. While the behavior may work consistently in ttsim or on current silicon, it is not contractual and may change in future hardware or simulator versions.

**Common causes:**

*   Relying on specific timing behavior not guaranteed by spec
*   Assuming specific undefined behavior outcomes
*   Using undocumented features or side effects

**Remediation:** Refactor workload to rely only on contractual guarantees documented in the ISA specification. Consider this a portability issue.

## Feature Availability Errors

These error categories indicate that the requested functionality is not currently available in ttsim, but with different implications for future availability and recommended actions.

**Feature Availability Decision Tree**

### UntestedFunctionality

The requested feature has been implemented in ttsim but is disabled because it lacks sufficient test coverage to guarantee correctness. This is a conservative stance to prevent exposing potentially buggy functionality.

**Typical timeline:** May be enabled in future releases as test coverage improves.

**Remediation options:**

1.   Wait for future release with feature enabled
2.   Report priority need via [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues) to expedite testing
3.   Consider alternative approaches using tested features

### UnimplementedFunctionality

The requested feature is not yet implemented but is planned for future support. This is the most common category for missing features with clear specification and implementation path.

**Typical timeline:** Implementation timeline depends on internal prioritization.

**Remediation options:**

1.   Wait for future release with feature implemented
2.   Report use case via [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues) to help prioritize
3.   Use alternative approaches or silicon hardware if timeline is critical

### UnsupportedFunctionality

The requested feature is unlikely to be implemented without strong justification. This typically applies to hardware features that are difficult to simulate accurately or have limited practical value in a simulator context.

**Examples:**

*   Features requiring cycle-accurate timing
*   Hardware debugging features
*   Performance counter edge cases
*   Chip-to-chip network behaviors

**Remediation options:**

1.   Provide strong use case justification via [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues)
2.   Accept that feature requires silicon hardware
3.   Redesign workload to avoid the unsupported feature

### MissingSpecification

The requested feature requires additional internal specification work before implementation can proceed. This indicates the feature exists in hardware but lacks sufficient documentation or behavioral specification for accurate simulation.

**Typical timeline:** Depends on specification prioritization and complexity.

**Remediation:**

*   Report use case via [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues) to help prioritize specification work
*   Providing detailed expected behavior descriptions can accelerate specification

## Configuration and System Errors

These errors indicate problems with the execution environment, configuration, or operating system resources.

### ConfigurationError

`ConfigurationError` indicates incorrect environment variables, command-line options, or configuration files. These must be corrected before ttsim can execute workloads.

**Common configuration errors:**

| Error Condition | Cause | Remediation |
| --- | --- | --- |
| `TT_METAL_HOME` not set | Environment variable missing | `export TT_METAL_HOME=/path/to/tt-metal` |
| `TT_METAL_SIMULATOR` not set | Environment variable missing | `export TT_METAL_SIMULATOR=/path/to/libttsim.so` |
| SOC descriptor not found | Missing `soc_descriptor.yaml` co-located with `libttsim.so` | Copy appropriate SOC descriptor from `$TT_METAL_HOME/tt_metal/soc_descriptors/` |
| SOC descriptor architecture mismatch | Wormhole SOC descriptor with Blackhole simulator | Ensure `wormhole_b0_80_arch.yaml` with `libttsim_wh.so` or `blackhole_140_arch.yaml` with `libttsim_bh.so` |
| Fast dispatch mode used | `TT_METAL_SLOW_DISPATCH_MODE` not set to `1` | `export TT_METAL_SLOW_DISPATCH_MODE=1` |
| Invalid path in environment variable | Path points to non-existent file or directory | Verify paths exist and are accessible |

**Diagnostic approach:**

1.   Verify all required environment variables are set: `TT_METAL_HOME`, `TT_METAL_SIMULATOR`, `TT_METAL_SLOW_DISPATCH_MODE`
2.   Verify paths point to existing, accessible files/directories
3.   Verify SOC descriptor architecture matches simulator binary
4.   Consult [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) page for complete requirements

### SystemError

`SystemError` indicates an operating system error such as file I/O failure, memory allocation failure, or permission denied. These are not ttsim bugs but environmental issues.

**Common causes:**

*   Insufficient disk space
*   Insufficient memory
*   Permission denied on files/directories
*   File descriptors exhausted
*   Corrupted or truncated files

**Remediation:**

1.   Check OS error message for specific failure
2.   Verify system resources (disk space, memory, file descriptors)
3.   Verify file permissions
4.   Check for corrupted downloads (re-download `libttsim.so` if necessary)

## Internal Simulator Bugs

### AssertionFailure

`AssertionFailure` indicates an internal simulator bug. This should never occur during normal operation and represents a defect in ttsim itself.

**Characteristics:**

*   Error message includes assertion condition and source location
*   Indicates violation of internal invariant
*   May include stack trace or diagnostic information

**Remediation:**

1.   **Do not modify workload code** - this is a simulator bug, not a workload bug
2.   Report via [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues) with: 
    *   Complete error message and stack trace
    *   Minimal reproduction steps
    *   Environment details (ttsim version, TT-Metalium version, OS)
    *   Workload code if possible

3.   For security-sensitive crashes, use [private security reporting](https://deepwiki.com/tenstorrent/ttsim/5.3-security-vulnerabilities)

**Priority:** These are treated as high-priority bugs and typically fixed in subsequent releases.

Sources: [README.md 76](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L76-L76)

## Error Message Structure

ttsim error messages follow a consistent structure to provide maximum diagnostic information:

**Typical Error Message Format**

**Example error message components:**

*   **Category**: Classification (e.g., `UnimplementedFunctionality`)
*   **Context**: Where in execution (e.g., during unpacker operation, during SFPU computation)
*   **Description**: What went wrong (e.g., specific instruction or feature)
*   **Location**: Hardware location if applicable (e.g., core coordinates, memory address)
*   **Suggestion**: Recommended remediation or reference to documentation

## Common Error Scenarios

The following table maps common error scenarios to their likely categories and remediation approaches:

| Scenario | Likely Category | First Diagnostic Step |
| --- | --- | --- |
| Simulator crashes immediately on launch | `ConfigurationError` or `SystemError` | Verify environment variables per [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| Error mentions missing SOC descriptor | `ConfigurationError` | Copy SOC descriptor to simulator directory |
| Error mentions fast dispatch | `ConfigurationError` | Set `TT_METAL_SLOW_DISPATCH_MODE=1` |
| Results differ from silicon | `NonContractualBehavior` or see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy) | Check for timing-dependent operations |
| Feature listed in ISA specification but fails | `UnimplementedFunctionality` or `UntestedFunctionality` | Check [Feature Status](https://deepwiki.com/tenstorrent/ttsim/7.3-feature-status) page |
| Exotic hardware feature fails | `UnsupportedFunctionality` | Consider if feature is simulator-compatible |
| Random crashes with no pattern | `AssertionFailure` | Report bug with reproduction steps |
| Workload works on silicon but fails in simulator | `UndefinedBehavior` or `UnimplementedFunctionality` | Simulator is stricter; check ISA compliance |
| Permission denied or file not found | `SystemError` | Check file permissions and paths |
| Out of memory | `SystemError` | Check system memory availability |

## Error Remediation Workflow

The following diagram illustrates the decision process for handling ttsim errors:

**Workflow steps:**

1.   **Identify category**: Read error message to determine error category
2.   **Consult appropriate resource**: 
    *   ISA violations → [tt-isa-documentation](https://github.com/tenstorrent/ttsim/blob/67b1733e/tt-isa-documentation)
    *   Feature unavailable → [Feature Status](https://deepwiki.com/tenstorrent/ttsim/7.3-feature-status)
    *   Configuration errors → [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration)
    *   Internal bugs → [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues)

3.   **Take appropriate action**: 
    *   ISA violations: Fix workload code
    *   Configuration errors: Fix environment
    *   Feature unavailable: Wait, report priority, or use alternative
    *   Internal bugs: Report with reproduction steps

4.   **Verify fix**: Re-run workload after remediation

## Error Reporting Best Practices

When reporting errors via the [issue tracker](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues), include:

**For ISA-related errors:**

*   Complete error message
*   Minimal code snippet demonstrating the issue
*   Expected behavior per ISA specification
*   Actual behavior observed

**For feature availability errors:**

*   Feature name and description
*   Use case explaining why feature is needed
*   Current workaround if any
*   Impact of feature absence on workflow

**For configuration errors:**

*   Complete error message
*   Environment variable values (`TT_METAL_HOME`, `TT_METAL_SIMULATOR`, etc.)
*   OS and version
*   ttsim version
*   Directory structure showing `libttsim.so` and `soc_descriptor.yaml` placement

**For assertion failures:**

*   Complete error message including stack trace
*   ttsim version and architecture (Wormhole/Blackhole)
*   TT-Metalium version
*   Minimal reproduction steps
*   Full workload code if possible
