## Purpose and Scope

This document describes the artifact generation and packaging system in `tt-metal`, covering how build outputs are produced, versioned, and distributed. It details the different artifact types (tarballs, Python wheels, Debian packages, Docker images), the workflows that generate them, naming conventions, and publishing mechanisms.

For information about the CMake build process that produces these artifacts, see [5.1 CMake Configuration and Compilation](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/5.1%20CMake%20Configuration%20and%20Compilation) For Docker build environment details, see [5.2 Docker Build Environments](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/5.2%20Docker%20Build%20Environments) For dependency management, see [5.4 Dependency Management](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/5.4%20Dependency%20Management)

## Artifact Types Overview

The `tt-metal` build system generates several types of artifacts for different deployment scenarios:

| Artifact Type | File Pattern | Purpose | Primary Workflow |
| --- | --- | --- | --- |
| Build Tarball | `ttm_any.tar.zst` | Compressed archive of built binaries, libraries, and tools | `.github/workflows/build-artifact.yaml` |
| Python Wheels | `*.whl` | Python package distribution for `pip` installation | `.github/workflows/wheels.yaml` |
| Debian Packages | `*.deb`, `*.ddeb` | System packages for Ubuntu/Debian installation | `.github/workflows/build-artifact.yaml` |
| Docker Images | `ghcr.io/tenstorrent/tt-metal/*` | Container images for development and runtime | `.github/workflows/publish-release-image.yaml` |

## Build Artifact Tarball Generation

### Tarball Contents and Creation

The `build-artifact` job produces `ttm_any.tar.zst`, a `zstd`-compressed tarball. The tarball includes compiled binaries, shared libraries, and runtime files required for execution without a full source tree.

**Tarball Structure**

Sources: [.github/workflows/build-artifact.yaml 544-551](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L544-L551)

### Tarball Creation Implementation

The `Tar files` step conditionally includes `tt-train` and `data` directories based on the `skip-tt-train` input [.github/workflows/build-artifact.yaml 51-56](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L51-L56):

**Compression Configuration:**

*   `zstd --adapt=min=9`: Adaptive compression starting at level 9 (high compression ratio) [.github/workflows/build-artifact.yaml 550](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L550-L550)
*   `-T0`: Multi-threaded compression using all available CPU cores [.github/workflows/build-artifact.yaml 550](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L550-L550)
*   `-cvhf`: Create archive, verbose output, follow symlinks, specify output file [.github/workflows/build-artifact.yaml 550](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L550-L550)

### Artifact Naming Convention

The `Set build artifact name` step generates unique artifact names using shell variable construction [.github/workflows/build-artifact.yaml 553-576](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L553-L576):

```
TTMetal_build_any_{VERSION}_{ARCH}_{TOOLCHAIN}{TRACY_SUFFIX}{DIST_SUFFIX}_{REF}_{RUN_ID}
```

**Naming Components:**

*   `VERSION`: Ubuntu version extracted from parsed platform ("22.04", "24.04") [.github/workflows/build-artifact.yaml 556](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L556-L556)
*   `ARCH`: Architecture from `inputs.architecture` (default "amd64") [.github/workflows/build-artifact.yaml 32-35](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L32-L35)
*   `TOOLCHAIN_CLEANED`: Toolchain path with `cmake/` prefix and `-toolchain.cmake` suffix removed [.github/workflows/build-artifact.yaml 559](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L559-L559)
*   `TRACY_SUFFIX`: `_profiler` when `inputs.tracy=true`[.github/workflows/build-artifact.yaml 17-562](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L17-L562)
*   `DIST_SUFFIX`: `_no_distributed` when `inputs.distributed=false`[.github/workflows/build-artifact.yaml 22-565](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L22-L565)

Sources: [.github/workflows/build-artifact.yaml 553-576](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L553-L576)

### Caching and Optimization

To optimize build times, the wheel workflow utilizes:

*   **LTO**: Link Time Optimization can be enabled via `inputs.enable-lto` for production wheels [.github/workflows/wheels.yaml 26-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/wheels.yaml#L26-L112)

### Validation and Tagging

Before an image is tagged as `latest` or `latest-rc`, it must pass a smoke test [.github/workflows/publish-release-image.yaml 108-162](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/publish-release-image.yaml#L108-L162):

2.   **Import Test**: Verifies `python -c "import ttnn"` inside the container [.github/workflows/publish-release-image.yaml 151-154](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/publish-release-image.yaml#L151-L154)
3.   **Tagging**: If tests pass, the image is tagged in GHCR [.github/workflows/publish-release-image.yaml 163-195](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/publish-release-image.yaml#L163-L195)

## Release Workflow Architecture

### Versioning and Tagging

The `package-and-release.yaml` workflow orchestrates the creation of release tags using the `calculate-version` action [.github/workflows/package-and-release.yaml 137-149](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/package-and-release.yaml#L137-L149)

*   **Release Types**: Supports `dev` (daily cron), `rc` (release candidate), and `prod`[.github/workflows/package-and-release.yaml 10-116](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/package-and-release.yaml#L10-L116)
*   **Automated Tagging**: The `create-tag` job creates a git tag and ref on the `stable` branch [.github/workflows/release-verify-or-create-tag.yaml 36-127](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/release-verify-or-create-tag.yaml#L36-L127)
*   **Release Promotion**: When promoting an `rc` to `prod`, the workflow skips redundant tests and focuses on artifact publishing [.github/workflows/package-and-release.yaml 58-61](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/package-and-release.yaml#L58-L61)
