This page documents the system description generation and hardware query tools in `tt-mlir`. The `SystemDesc` structure captures complete hardware topology including chip configurations, memory hierarchies, and inter-chip connectivity. The primary tool for querying hardware is the `ttrt query` command, which generates `.ttsys` files containing serialized system descriptions.

For runtime device management, see page 4.2. For compilation usage of system descriptions, see page 3.2 and 3.3.

## ttrt query Command

The `ttrt query` command is the primary tool for generating system descriptions. It queries the local hardware and produces a `.ttsys` file containing the serialized `SystemDesc`.

**Basic Usage:**

`# Query system and print to stdoutttrt query # Save system description to filettrt query --save-artifacts # Save to custom directoryttrt query --save-artifacts --artifact-dir /path/to/dir # Quiet mode (suppress output)ttrt query --quiet # Save results to JSONttrt query --result-file result.json`
The command generates `system_desc.ttsys` under the artifacts directory (default: `ttrt-artifacts`).

## SystemDesc Structure

The `SystemDesc` is a runtime type that wraps a flatbuffer containing the complete hardware description. It is essential for cross-compilation, allowing the compiler to target specific hardware configurations without being physically connected to them. The schema is defined in `system_desc.fbs` and includes descriptors for CPUs, chips, and their interconnects.

## Topology and Connectivity

### ChipChannel: Ethernet Connectivity

The `ChipChannel` table describes physical ethernet connections between chips in a cluster. This is used by the compiler to plan multi-device communication and collective operations.

The connectivity graph is built by `getAllDeviceConnections`, which iterates over active ethernet cores of all devices in the mesh. It uses `get_connected_ethernet_core` to discover neighbors [runtime/lib/common/system_desc.cpp 50-85](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/common/system_desc.cpp#L50-L85) The resulting connections are serialized into `ChipChannel` structs containing `device_id` and `ethernet_core_coord` for both ends of the link [include/ttmlir/Target/Common/types.fbs 109-114](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Target/Common/types.fbs#L109-L114)

### ChipCapability: Feature Flags

The `ChipCapability` enum defines device feature flags, such as whether a device is a PCIe controller (`HostMMIO`) [include/ttmlir/Target/Common/types.fbs 59-61](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Target/Common/types.fbs#L59-L61) In the MLIR dialect, this is exposed via `ChipCapabilityAttr`[include/ttmlir/Dialect/TTCore/IR/TTCoreOpsTypes.td 131-132](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTCore/IR/TTCoreOpsTypes.td#L131-L132)

### ChipCoord: Physical Location

The `ChipCoord` table stores the physical location of each chip in a cluster, including rack, shelf, and grid coordinates (`y`, `x`) [include/ttmlir/Target/Common/types.fbs 102-107](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Target/Common/types.fbs#L102-L107) This is populated during system discovery to provide a mapping of the logical mesh to physical hardware [runtime/lib/common/system_desc.cpp 153-154](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/common/system_desc.cpp#L153-L154)

## Integration with Compilation Pipeline

System descriptions are integrated into the MLIR compilation flow through device attributes and custom type definitions in the `TTCore` dialect.

**Flatbuffer to MLIR Conversion:** The `deviceToFlatbufferMeshShape` utility converts `DeviceAttr` from the MLIR dialect into the flatbuffer representation used in the final binary. It defaults to a 1x1 grid if no mesh shape is specified [include/ttmlir/Target/Utils/MLIRToFlatbuffer.h 52-66](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Target/Utils/MLIRToFlatbuffer.h#L52-L66)

**Mock System Descriptions:** For testing and development without hardware, the compiler can generate default system descriptions for specific architectures. For example, `createDefaultBlackholeSystemDesc` populates a mock `SystemDescAttr` with standard Blackhole parameters:

*   L1 size: 1,572,864 bytes [lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp 71](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp#L71-L71)
*   8 DRAM channels [lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp 72](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp#L72-L72)
*   Standard coordinate translation offsets {18, 18} [lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp 105](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp#L105-L105) Similar logic exists for Quasar [lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp 42-65](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTCore/IR/TTCoreOpsTypes.cpp#L42-L65)

**Registration Pass:** The `TTCoreRegisterDevicePass` allows registering a device via a system description path or a mock architecture (WormholeB0, Blackhole, Quasar) during the compilation pipeline [include/ttmlir/Dialect/TTCore/Transforms/Passes.td 54-86](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTCore/Transforms/Passes.td#L54-L86)

## Runtime System Description API

The runtime provides a high-level API to query the current system configuration and manage device capabilities.

**Key Functions:**

*   `tt::runtime::system_desc::getCurrentSystemDesc`: Dispatches to the active backend to generate a system description [runtime/lib/common/system_desc.cpp 265](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/common/system_desc.cpp#L265-L265)
*   `calculateDRAMUnreservedEnd`: A conservative estimator for the end of the DRAM region usable by the compiler, accounting for program carve-outs by calculating the space needed for kernels across all cores and aligning to DRAM channel constraints [runtime/lib/common/system_desc.cpp 115-139](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/common/system_desc.cpp#L115-L139)
*   `toFlatbuffer`: Overloaded functions that convert internal `tt-metal` types (like `CoreCoord` or `ARCH`) into flatbuffer-compatible structures [runtime/lib/common/system_desc.cpp 30-48](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/common/system_desc.cpp#L30-L48)
