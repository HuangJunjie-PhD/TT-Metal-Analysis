## Purpose and Scope

This page documents circular buffer (CB) operations in `tt-lang`, which provide the low-level synchronization primitives for producer-consumer communication between thread functions. Circular buffers are the hardware-level construct that underlies the user-facing **Dataflow Buffer (DFB)** abstraction.

While the Python DSL uses `ttl.make_dataflow_buffer_like()`, the underlying MLIR representation uses explicit CB operations (`ttl.cb_reserve`, `ttl.cb_wait`, etc.) to manage L1 memory and synchronization [include/ttlang/Dialect/TTL/IR/TTLOps.td 638-733](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L638-L733)

**Key Distinctions:**

*   **Dataflow Buffer (DFB)**: A high-level abstraction representing a sharded tensor in L1.
*   **Circular Buffer (CB)**: The hardware mechanism (slots 0-31) that manages a FIFO of tiles in L1 [include/ttlang/Dialect/TTL/IR/TTLOps.td 34-45](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L34-L45)[include/ttlang/Dialect/TTL/IR/TTL.h 25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L25-L25)
*   **Handshaking**: Synchronization is achieved through blocking `reserve`/`wait` calls and non-blocking `push`/`pop` calls [include/ttlang/Dialect/TTL/IR/TTLOps.td 643-717](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L643-L717)

## Circular Buffer Lifecycle

A circular buffer follows a well-defined lifecycle through the compilation pipeline, transitioning from high-level tensor associations to low-level hardware pointer management.

**Title: Circular Buffer Lifecycle and State Transitions**

**Lifecycle Stages:**

1.   **Binding** (`ttl.bind_cb`): Declares that a kernel will use a specific hardware slot (0-31). This op produces an SSA handle (`!ttl.cb`) but does not allocate memory [include/ttlang/Dialect/TTL/IR/TTLOps.td 25-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L25-L49)
2.   **Attachment** (`ttl.attach_cb`): Associates a tensor value with a CB handle. This allows the compiler to trace back from a `ttl.compute` or `ttl.copy` operation to the specific hardware CB slot [include/ttlang/Dialect/TTL/IR/TTLOps.td 52-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L52-L76)
3.   **Acquisition**: 
    *   **Producers** call `ttl.cb_reserve` to get writable space in L1 [include/ttlang/Dialect/TTL/IR/TTLOps.td 638-654](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L638-L654)
    *   **Consumers** call `ttl.cb_wait` to ensure data is available for reading [include/ttlang/Dialect/TTL/IR/TTLOps.td 687-703](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L687-L703)

4.   **Use**: Operations like `ttl.tile_store` or `ttl.copy` act on the "view" (tensor) returned by the acquisition phase [test/ttlang/Dialect/TTL/IR/cb_ops.mlir 95-101](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/cb_ops.mlir#L95-L101)
5.   **Release**: 
    *   **Producers** call `ttl.cb_push` to signal data is ready for the consumer [include/ttlang/Dialect/TTL/IR/TTLOps.td 664-678](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L664-L678)
    *   **Consumers** call `ttl.cb_pop` to free the space in L1 [include/ttlang/Dialect/TTL/IR/TTLOps.td 712-726](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L712-L726)

## Core CB Operations

### Operation Summary Table

| Operation | Blocking | Hardware Lowering (TTKernel) | Role |
| --- | --- | --- | --- |
| `ttl.bind_cb` | No | `ttkernel.get_compile_time_arg_val` | Metadata Declaration |
| `ttl.attach_cb` | No | (Erased/Metadata) | SSA Association |
| `ttl.cb_reserve` | **Yes** | `ttkernel.cb_reserve_back` | Producer Acquire |
| `ttl.cb_push` | No | `ttkernel.cb_push_back` | Producer Release |
| `ttl.cb_wait` | **Yes** | `ttkernel.cb_wait_front` | Consumer Acquire |
| `ttl.cb_pop` | No | `ttkernel.cb_pop_front` | Consumer Release |

**Sources:**[test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir 3-63](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir#L3-L63)[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 1-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L1-L12)

### ttl.bind_cb

Declares usage of a hardware slot. It defines the `block_count` (e.g., 2 for double buffering) and the `cb_index`[include/ttlang/Dialect/TTL/IR/TTLOps.td 34-45](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L34-L45) During verification, the compiler ensures the `cb_index` is in the valid non-negative range [lib/Dialect/TTL/IR/TTLOps.cpp 101-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L101-L103)

### ttl.attach_cb

An identity operation on a tensor that records a mapping for the compiler. Exactly one CB should be attached per tensor SSA value used in compute or data movement [include/ttlang/Dialect/TTL/IR/TTLOps.td 57-61](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L57-L61) This mapping is critical for identifying output CBs during elementwise fusion, where `collectOutputCBs` queries the CB attached to a store view [lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp 101-119](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp#L101-L119)

### ttl.cb_reserve and ttl.cb_wait

These operations return a `tensor` view representing the L1 memory window. For a 2D block shape (e.g., `[2, 2]`), these operations effectively manage `2*2=4` pages at a time [test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir 118-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir#L118-L130) The result tensor must match the CB's element type and shape rank [test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir 5-34](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir#L5-L34)

## Automatic Synchronization Insertion

The `ttl-insert-cb-sync` pass is responsible for auto-inserting `cb_push` and `cb_pop` operations when they are missing from the input IR [lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp 9-15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp#L9-L15)

*   **Producer Rules**: For a `ttl.cb_reserve`, the pass inserts a `ttl.cb_push` after the last use of the acquired slot (typically a `ttl.store` or `ttl.copy`) [lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp 74-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp#L74-L76)
*   **Consumer Rules**: For a `ttl.cb_wait`, the pass inserts a `ttl.cb_pop` after the last transitive use of the data (e.g., after a `ttl.add` or `ttl.copy` is complete) [lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp 77-79](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertCBSync.cpp#L77-L79)

**Sources:**[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 1-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L1-L12)[include/ttlang/Dialect/TTL/IR/TTLOps.td 25-733](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L25-L733)[test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir 3-63](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/cb_ops.mlir#L3-L63)

### Compute and Data Movement Integration

When a `ttl.compute` operation is lowered, the compiler must resolve which CB index corresponds to each input/output tensor. This is done by tracing the SSA values back through `ttl.attach_cb` or `ttl.bind_cb`[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 59-112](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L59-L112)

1.   **Linearization**: The compiler linearizes tile indices (e.g., in a loop) to calculate the specific page index within the circular buffer using `affine.AffineLinearizeIndexOp`[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 133-153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L133-L153)
2.   **Store Integration**: In the `ConvertTTLToCompute` pass, block-level stores are converted to `tile_store` operations inside the compute body. The pass collects output CBs by identifying `StoreOp` users whose views are defined by `CBReserveOp`[lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp 101-119](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToCompute.cpp#L101-L119)
3.   **Data Movement Trace**: For `ttl.copy` and tile operations, the compiler uses `getAttachedCB` and `traceUnrealizedCasts` to find the underlying `!ttl.cb` handle [include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h 41-54](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h#L41-L54)[include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h 116-133](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h#L116-L133)

## Common Validation Rules

The `tt-lang` compiler enforces several invariants during the `verify()` stage of CB operations:

*   **Index Range**: `cb_index` must be non-negative and is checked against `kMaxCircularBuffers` (32) [lib/Dialect/TTL/IR/TTLOps.cpp 101-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L101-L103)[include/ttlang/Dialect/TTL/IR/TTL.h 25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L25-L25)
*   **Type Consistency**: The element type of the tensor being attached must match the element type defined in the `!ttl.cb` handle [lib/Dialect/TTL/IR/TTLOps.cpp 122-126](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L122-L126)
*   **Copy Operand Constraint**: A `ttl.copy` operation must have exactly one operand that is a circular buffer [lib/Dialect/TTL/IR/TTLOps.cpp 196-200](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L196-L200)
*   **View Source**: A `ttl.tile_store` view must originate from a `ttl.cb_reserve` operation; it cannot be a raw function argument [test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir 5-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir#L5-L12)
*   **Capacity Checks**: The `num_tiles` attribute on `cb_reserve`, `cb_wait`, and `cb_push` must not exceed the total capacity of the CB (elements per block * block count) [test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir 116-122](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir#L116-L122)[test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir 127-133](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/cb_ops_invalid.mlir#L127-L133)
