This document describes the `UNPACR` instruction family, which controls the Unpacker execution units within the Tensix Coprocessor. These instructions move data from L1 memory to the internal register files (`SrcA`, `SrcB`, or `Dst`), performing format conversion and decompression as needed. For details on the Unpacker system architecture and data path, see [Unpacker System (UNPACR)](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.3-unpacker-system-(unpacr)). For configuration management, see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system).

## Overview

The `UNPACR` instruction family consists of three variants, all sharing the same opcode but differentiated by bit fields in the instruction encoding:

| Variant | Purpose | Primary Use Case |
| --- | --- | --- |
| **Regular UNPACR** | Move datums from L1 to `SrcA`/`SrcB`/`Dst` | Primary data ingestion operation |
| **Increment Context Counter** | Advance the context counter | Switching between multiple input configurations in multi-context mode |
| **Flush Cache** | Clear decompression row start cache | Ensuring correctness when reusing L1 memory containing compressed data |

## Regular UNPACR: Data Movement

The regular `UNPACR` instruction is the primary data ingestion operation in the Tensix Coprocessor pipeline. It fetches data from L1 memory and writes it to one of three destinations: `SrcA` (Matrix Unit input), `SrcB` (Matrix Unit input), or `Dst` (accumulator/output buffer).

### Syntax

`TT_OP_UNPACR(/* u1 */ WhichUnpacker,            ((/* u2 */ Ch1YInc) << 6) +            ((/* u2 */ Ch1ZInc) << 4) +            ((/* u2 */ Ch0YInc) << 2) +              /* u2 */ Ch0ZInc,            false,            /* u3 */ ContextNumber,            /* u2 */ ContextADC,            /* bool */ MultiContextMode,            /* bool */ FlipSrc,            false,            /* bool */ AllDatumsAreZero,            /* bool */ UseContextCounter,            /* bool */ RowSearch,            false,            false)`
| Parameter | Bits | Description |
| --- | --- | --- |
| `WhichUnpacker` | 1 | Select unpacker 0 or 1 |
| `Ch0YInc`, `Ch0ZInc` | 2 each | Increment values for ADC Channel 0 Y and Z coordinates |
| `Ch1YInc`, `Ch1ZInc` | 2 each | Increment values for ADC Channel 1 Y and Z coordinates |
| `ContextNumber` | 3 | Context index for multi-context mode |
| `ContextADC` | 2 | ADC selection for multi-context mode |
| `MultiContextMode` | 1 | Enable multi-context addressing |
| `FlipSrc` | 1 | Swap SrcA/SrcB banks and reset row counter |
| `AllDatumsAreZero` | 1 | Force all output datums to zero |
| `UseContextCounter` | 1 | Use hardware context counter instead of `ContextNumber` |
| `RowSearch` | 1 | Enable row-based addressing for compressed data |

### Format Conversion

The unpacker performs format conversion from L1 data types to internal register file formats. Supported conversions differ based on whether unpacking to `SrcA`/`SrcB` or to `Dst`:

#### Unpacking to SrcA or SrcB

| L1 Format | → | Internal Format (19-bit) |
| --- | --- | --- |
| FP32, TF32 | → | TF32, BF16, or FP16 |
| BF16 | → | TF32 or BF16 |
| BFP8/4/2 | → | TF32 or BF16 |
| BFP8a/4a/2a | → | FP16 |
| FP16, FP8 | → | FP16 |
| INT8 (signed/unsigned) | → | Integer "8" |
| INT16 | → | Integer "16" |

#### Unpacking to Dst

| L1 Format | → | Dst Format (16/32-bit) |
| --- | --- | --- |
| FP32, TF32 | → | FP32, BF16, or FP16 |
| BF16 | → | BF16 |
| BFP8/4/2 | → | BF16 |
| BFP8a/4a/2a | → | FP16 |
| FP16, FP8 | → | FP16 |
| INT32 | → | Integer "32" |
| INT16 | → | Integer "16" |
| INT8 (signed/unsigned) | → | Integer "8" |

### Decompression

For compressed data formats (BFP with compression enabled via `IsUncompressed = false`), the unpacker performs run-length decompression using delta encoding:

**Compressed Data Layout in L1:**

1.   **Row Start Table:** Array of `uint16_t` specifying byte offset for each row or blob
2.   **Exponent Array:** One 8-bit exponent per 16 datums (BFP formats only)
3.   **Data Array:** Compressed datums (BFP8=1 byte, BFP4=0.5 byte, BFP2=0.25 byte per datum)
4.   **Delta Array:** 4-bit deltas specifying number of zero datums following each non-zero datum

**Decompression Algorithm (simplified):**

`for each datum in input:    DatumBits = ReadL1(DataAddr)    DecompressionZeroes = ReadL1(DeltaAddr) // 4-bit delta    Output(DatumBits)    for (i = 0; i < DecompressionZeroes; ++i):        Output(0)`
**Address Wrapping:** All input addresses wrap at the FIFO boundary:

`if (addr > Unpack_limit_address * 16):    addr -= Unpack_fifo_size * 16`
Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 108-202](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L108-L202)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 302-312](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L302-L312)

### Special Modes

#### Transpose Mode (`Haloize_mode`)

When `THCON_SEC[0].REG2_Haloize_mode = true` for unpacker 0, the unpacker transposes the low 4 bits of the row address with the column address:

`uint4_t RowLowBits = Row & 0xf;std::swap(RowLowBits, Col);Row = (Row & ~0xf) | RowLowBits;`
**Restrictions:**

*   Only applies to unpacker 0
*   Requires aligned 16-byte input address boundaries
*   Incompatible with `UnpackToDst` and `ColShift`

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 218-232](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L218-L232)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 349-353](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L349-L353)

#### Tileize Mode (`Tileize_mode`)

When `THCON_SEC[WhichUnpacker].REG2_Tileize_mode = true`, input data is read from discontiguous rows with configurable stride:

`RowStride = (Shift_amount_cntx[0] << 4) |            (Shift_amount_cntx[1] << 8) |            (Shift_amount_cntx[2] << 12);  // Max 65520 bytes`
After reading 16 datums, the input address advances by `RowStride` instead of `DatumSizeBytes * 16`.

**Restrictions:**

*   Requires aligned 16-byte input address boundaries
*   Incompatible with upsampling (`UpsampleZeroes > 0`)
*   Incompatible with compression (`!IsUncompressed`)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 204-213](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L204-L213)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 266-274](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L266-L274)

#### Row Search Mode (`RowSearch`)

Enables blob-based addressing when `BlobsPerXYPlane != 0`. Instead of regular X/Y coordinates, the ADC X coordinate selects a blob index, and addressing uses the `BlobsYStart` array to determine input ranges.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 144-158](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L144-L158)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 167-178](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L167-L178)

### Output Address Generation

The output address to `SrcA`/`SrcB`/`Dst` is calculated from base addresses and ADC Channel 1 coordinates:

`OutAddr = ADDR_BASE_REG_1_Base        + ADC_Out.Y * ADDR_CTRL_XY_REG_1_Ystride        + ADC_Out.Z * ADDR_CTRL_ZW_REG_1_Zstride        + ADC_Out.W * ADDR_CTRL_ZW_REG_1_Wstride;`
For multi-context mode with unpacker 0, an additional context-specific offset can be added or replace the base:

`if (MultiContextMode && WhichUnpacker == 0) {    CtxOutAddr = REG5_Dest_cntx[WhichContext & 3].address;    if (UnpackToDst || ADD_DEST_ADDR_CNTR_add_dest_addr_cntr) {        OutAddr += CtxOutAddr;    } else {        OutAddr = CtxOutAddr;    }}`
Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 234-253](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L234-L253)

### Bank Flipping and Row Management

The `FlipSrc` parameter controls bank swapping behavior:

`if (FlipSrc) {    // Make current bank available to Matrix Unit    SrcBuffer[CurrentUnpacker.SrcBank].AllowedClient = MatrixUnit;    // Switch to other bank    CurrentUnpacker.SrcBank ^= 1;    // Reset row counter    CurrentUnpacker.SrcRow[CurrentThread] = SrcRowBase;} else if (Unpack_Src_Reg_Set_Upd) {    // Advance row counter without flipping    CurrentUnpacker.SrcRow[CurrentThread] += 16 + SrcRowBase;}`
This double-buffering mechanism allows the Matrix Unit to consume data from one bank while the unpacker fills the other.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 394-401](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L394-L401)

### Performance Characteristics

The unpacker instruction performance depends on L1 fetch bandwidth:

| Speed Mode | Bandwidth | Conditions |
| --- | --- | --- |
| x1 speed | 16 bytes/cycle | Default; required for compressed data, BFP2/2a, `UpsampleZeroes ≥ 3` |
| x2 speed | 32 bytes/cycle | Configurable via `Throttle_mode = 1` when allowed |
| x4 speed | 64 bytes/cycle | Configurable via `Throttle_mode = 2`; required for `Tileize_mode` |

**Two-Unpacker Interference:**

When both unpackers are active simultaneously, they share L1 bandwidth according to this priority table:

|  | Unpacker 0: x1 | Unpacker 0: x2 | Unpacker 0: x4 |
| --- | --- | --- | --- |
| **Unpacker 1: x1** | Both get x1 | U0 gets x2, U1 gets x1 | U0 gets x4, U1 gets x1 |
| **Unpacker 1: x2** | U0 gets x1, U1 gets x2 | Both get x1 ⚠️ | U0 gets x2, U1 gets x1 ⚠️ |
| **Unpacker 1: x4** | U0 gets x1, U1 gets x4 | U0 gets x1, U1 gets x2 ⚠️ | Both get x2 ⚠️ |

⚠️ = Degraded performance from requested speed

**Startup Latency:** Each `UNPACR` instruction incurs at least 2 cycles of address calculation latency (more for compressed data), during which no other `UNPACR` instruction from any thread can start.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 572-598](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L572-L598)

## Increment Context Counter

This variant increments the hardware context counter used in multi-context mode with `UseContextCounter = true`. It enables rapid context switching without RISC-V intervention.

### Syntax

`TT_OP_UNPACR(/* u1 */ WhichUnpacker,             0,             true,  // Distinguishing bit             0,             0,             false,             false,             false,             false,             false,             false,             false)`
### Functional Model

`uint1_t StateID = ThreadConfig[CurrentThread].CFG_STATE_ID_StateID;auto& ConfigState = Config[StateID];auto& CurrentUnpacker = Unpackers[WhichUnpacker]; uint3_t WhichContext = CurrentUnpacker.ContextCounter[CurrentThread];uint3_t IncrementedCounter = WhichContext + 1;if (IncrementedCounter >= (1 << ConfigState.THCON_SEC[WhichUnpacker].Context_count)) {    IncrementedCounter = 0;  // Wraparound}CurrentUnpacker.ContextCounter[CurrentThread] = IncrementedCounter;`
The counter wraps around based on the `Context_count` configuration field (1-8 contexts supported).

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_IncrementContextCounter.md 1-41](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_IncrementContextCounter.md?plain=1#L1-L41)

## Flush Decompression Cache

This variant flushes the row start table cache used for compressed data addressing. It is necessary when L1 memory containing row start tables is being rapidly reused.

### Syntax

`TT_OP_UNPACR(/* u1 */ WhichUnpacker,             0,             false,             0,             0,             /* bool */ MultiContextMode,             false,             false,             false,             false,             false,             true,   // Distinguishing bit             false)`
### Behavior

Each unpacker has 4 one-entry caches for row start table lookups:

*   When `MultiContextMode = true`: Flushes all 4 caches
*   When `MultiContextMode = false`: Flushes only the cache for the current thread

This instruction completes immediately with no data movement.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_FlushCache.md 1-32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_FlushCache.md?plain=1#L1-L32)

## ADC Update Behavior

At instruction completion, the instruction updates ADC (Auto-incrementing Addressing Counter) coordinates based on the `Ch0YInc`, `Ch0ZInc`, `Ch1YInc`, and `Ch1ZInc` parameters:

`for (unsigned i = 0; i < 3; ++i) {    if (i == CurrentThread || i == WhichADC) {        ADCs[i].Unpacker[WhichUnpacker].Channel[0].Y += Ch0YInc;        ADCs[i].Unpacker[WhichUnpacker].Channel[0].Z += Ch0ZInc;        ADCs[i].Unpacker[WhichUnpacker].Channel[1].Y += Ch1YInc;        ADCs[i].Unpacker[WhichUnpacker].Channel[1].Z += Ch1ZInc;    }}`
This allows software to advance through multi-dimensional tensors by incrementing higher-dimensional coordinates after processing lower-dimensional slices.
