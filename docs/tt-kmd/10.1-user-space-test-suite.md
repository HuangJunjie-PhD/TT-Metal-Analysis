This document describes the user-space test suite for the Tenstorrent Kernel Mode Driver (tt-kmd). The test suite, titled `ttkmd_test`, is a comprehensive set of C++ programs designed to verify the functionality, correctness, and robustness of the kernel mode driver by exercising its user-space interfaces.

## Overview

The user-space test suite validates the various interfaces provided by the tt-kmd driver. It performs tests on all detected Tenstorrent devices, verifying operations such as memory mapping, DMA buffer allocation, TLB management, PCI configuration, and IOCTL behavior. The suite ensures driver stability across hardware generations (Wormhole and Blackhole) and provides detailed failure diagnostics.

## Device Enumeration and Abstraction

The `EnumeratedDevice` structure provides a unified view of a device by correlating information from `/dev/tenstorrent` and `/sys/bus/pci/devices`.

*   **`EnumerateDevices()`**: Cross-references character device nodes with PCI vendor IDs (0x1E52) to ensure the driver is correctly bound to the hardware [test/enumeration.cpp 142-167](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/enumeration.cpp#L142-L167)
*   **`IdentityDeviceType()`**: Determines if the hardware is `Wormhole` (0x401e) or `Blackhole` (0xb140) based on the PCI device ID [test/enumeration.cpp 128-140](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/enumeration.cpp#L128-L140)
*   **`IsIommuTranslated()`**: Checks the IOMMU group status in sysfs to determine if DMA addresses require translation [test/enumeration.cpp 115-126](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/enumeration.cpp#L115-L126)

## Key Test Categories

### PCI Configuration Space

`TestConfigSpace` validates that the kernel has correctly initialized the device's PCI registers. It verifies:

*   **Command Register**: Memory Space and Bus Master enables [test/config_space.cpp 88-98](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/config_space.cpp#L88-L98)
*   **MSI Capability**: Ensures Message Signaled Interrupts are enabled and have a non-zero address [test/config_space.cpp 100-118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/config_space.cpp#L100-L118)
*   **AER (Advanced Error Reporting)**: Checks that error reporting is enabled, unless `--skip-aer` is provided [test/config_space.cpp 120-138](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/config_space.cpp#L120-L138)

### IOCTL Robustness

The suite includes specialized tests to ensure the driver handles malicious or malformed user-space inputs:

*   **`TestIoctlOverrun`**: Uses `EndOfPage<T>` to place IOCTL structures at the very end of a mapped page followed by an unmapped page. This triggers a segmentation fault or `EFAULT` if the driver reads/writes beyond the structure size [test/ioctl_overrun.cpp 32-86](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/ioctl_overrun.cpp#L32-L86)
*   **`TestIoctlZeroing`**: Verifies that the driver properly zeroes out output structures to prevent kernel memory leaks [test/main.cpp 52](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/main.cpp#L52-L52)

### Diagnostic Interface Testing

Newer tests verify the driver's transparency through debugfs and procfs:

*   **`TestMappingsDebugfs`**: Verifies that `/sys/kernel/debug/tenstorrent/<id>/mappings` correctly lists open FDs, pinned pages, and DMA buffers [test/mappings_debugfs.cpp 4-10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/mappings_debugfs.cpp#L4-L10)
*   **`TestProcfsPids`**: Ensures that the driver correctly tracks which PIDs are using which device [test/main.cpp 55](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/main.cpp#L55-L55)

## Building and Running

### Build System

The suite is built via a standalone Makefile in the `test/` directory. It compiles core utilities and individual test modules into a single `ttkmd_test` binary.

`# Build the test suitemake -C test -j $(nproc)`
Sources: [test/Makefile 1-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/Makefile#L1-L38)

### Hardware Test Script

The `test/run-hardware-tests.sh` script is the primary entry point for automated testing. It performs the following:

1.   Removes existing driver and DKMS versions [test/run-hardware-tests.sh 78-89](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/run-hardware-tests.sh#L78-L89)
2.   Builds and loads the fresh driver [test/run-hardware-tests.sh 91-99](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/run-hardware-tests.sh#L91-L99)
3.   Allocates 2MB hugepages required for certain DMA tests [test/run-hardware-tests.sh 132-143](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/run-hardware-tests.sh#L132-L143)
4.   Executes `ttkmd_test` with a 120-second timeout and collects diagnostics (dmesg, lscpu) on failure [test/run-hardware-tests.sh 145-148](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/run-hardware-tests.sh#L145-L148)

## CI Integration

The test suite is integrated into GitHub Actions via `.github/workflows/hardware-test.yml`. It runs on a variety of Tenstorrent hardware configurations, including:

*   **N150/N300** (Wormhole)
*   **P150b** (Blackhole)
*   **vIOMMU** configurations to test IOMMU translation logic.
