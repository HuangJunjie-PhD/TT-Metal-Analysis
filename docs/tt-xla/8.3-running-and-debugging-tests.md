## Running Test Suites

### Basic Test Execution

TT-XLA uses pytest as its test runner. Tests are organized in the `tests/` directory and can be run with standard pytest commands:

`# Run all testspytest tests/ # Run specific test filepytest tests/runner/test_models.py # Run specific test functionpytest tests/runner/test_models.py::test_all_models_torch # Run with verbose outputpytest -v tests/runner/test_models.py # Run with extra verbose output and show print statementspytest -svv tests/runner/test_models.py`
The `-svv` flags are recommended for debugging as they show all output including print statements and detailed error traces.

**Sources:**[docs/src/getting_started_build_from_source.md 190](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/getting_started_build_from_source.md?plain=1#L190-L190)[docs/src/test_infra.md 103-114](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/test_infra.md?plain=1#L103-L114)

### Test Discovery and Parametrization

**Figure: Test Discovery and Parametrization Flow**

The test system uses dynamic loaders to discover models and variants at runtime:

1.   **Discovery:**`TorchDynamicLoader.discover_loader_paths()` walks the `third_party/tt_forge_models/` directory tree looking for `pytorch/loader.py` files
2.   **Variant Loading:** For each loader, `get_model_variants()` queries available variants from `ModelLoader.query_available_variants()`
3.   **Test Entry Creation:** Each (loader_path, variant) pair becomes a `ModelTestEntry`
4.   **Parametrization:**`@pytest.mark.parametrize` generates test instances for each combination of (test_entry, run_mode, parallelism)

## Inspecting Compilation Artifacts

### Using the --serialize Flag

The `--serialize` flag instructs the test framework to dump compilation artifacts (MLIR IRs, TTNN IRs) to disk:

`# Serialize all IR artifacts for a testpytest tests/runner/test_models.py::test_name --serialize # Serialize with verbose outputpytest -svv tests/runner/test_models.py::test_name --serialize`
For op and graph tests, you must pass the `request` fixture:

`def test_my_op(request):    run_op_test(MyOp(), [torch.randn(32, 32)], request=request)`
For model tests, pass the `request` fixture to `tester.test()`:

`def test_my_model(model_tester: MyModelTester, request):    model_tester.test(request=request)`
**Sources:**[docs/src/test_infra.md 118-140](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/test_infra.md?plain=1#L118-L140)[tests/infra/testers/single_chip/model/model_tester.py 174-176](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/infra/testers/single_chip/model/model_tester.py#L174-L176)

### IR Artifact Directory Structure

When `--serialize` is used, artifacts are written to:

```
output_artifact/<sanitized_test_name>/
├── shlo_compiler_0.mlir          # StableHLO IR
├── ttir_0.mlir                    # TT-MLIR TTIR dialect
└── ttnn_0.mlir                    # TT-MLIR TTNN dialect
```

The test name is sanitized by replacing `/` and special characters with `_`.

For `--dump-irs` (used by op-by-op tests), artifacts go to:

```
collected_irs/<model_name>/
└── irs/
    └── shlo_compiler*.mlir
```

## FileCheck Validation

### Overview

FileCheck is a pattern-matching tool used to verify that compiler transformations produce expected IR. It searches for patterns in MLIR files and fails if patterns are not found or appear in the wrong order.

### Using the @pytest.mark.filecheck Decorator

To validate IR transformations, use the `@pytest.mark.filecheck` decorator with a list of pattern files:

`@pytest.mark.filecheck(["add.ttnn.mlir", "rms_norm.ttir.mlir"])def test_my_op(request):    run_op_test(MyOp(), [torch.randn(32, 32)], request=request)`
FileCheck automatically:

1.   Serializes compilation artifacts (no need for `--serialize` flag)
2.   Runs pattern matching against the generated IR
3.   Fails the test if patterns don't match

**Sources:**[docs/src/test_infra.md 143-154](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/test_infra.md?plain=1#L143-L154)[tests/runner/test_models.py 157-160](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_models.py#L157-L160)

### Pattern File Syntax

Pattern files live in `tests/filecheck/` and use FileCheck syntax. A simple example:

`//CHECK: "ttir.rms_norm"`
This checks that the string `"ttir.rms_norm"` appears in the TTIR output.

More complex example:

`//CHECK: "ttnn.add"`
**FileCheck Directives:**

| Directive | Purpose | Example |
| --- | --- | --- |
| `//CHECK:` | Must match line in order | `//CHECK: "ttnn.add"` |
| `//CHECK-NEXT:` | Must match on next line | `//CHECK-NEXT: %result` |
| `//CHECK-DAG:` | Must match in any order (within block) | `//CHECK-DAG: "ttnn.multiply"` |
| `//CHECK-NOT:` | Must NOT appear | `//CHECK-NOT: "ttnn.softmax"` |
| `//CHECK-SAME:` | Must match on same line as previous | `//CHECK-SAME: dtype = f32` |

**Sources:**[tests/filecheck/add.ttnn.mlir 1-2](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/filecheck/add.ttnn.mlir#L1-L2)[tests/filecheck/rms_norm.ttir.mlir 1-2](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/filecheck/rms_norm.ttir.mlir#L1-L2)[docs/src/test_infra.md 154](https://github.com/tenstorrent/tt-xla/blob/c77995f6/docs/src/test_infra.md?plain=1#L154-L154)

### Dynamic Pattern Injection

Test metadata can specify FileCheck patterns dynamically:

`class ModelTestConfig:    def __init__(self, data, arch=None):        # ...        self.filechecks = self._normalize_markers(            self._resolve("filechecks", default=[])        )`
The test runner injects the marker at runtime:

`if hasattr(test_metadata, "filechecks") and test_metadata.filechecks:    request.node.add_marker(        pytest.mark.filecheck(test_metadata.filechecks)    )`

## Debugging Test Failures

### Understanding Test Output

When a test fails, pytest displays several types of information:

1.   **Test ID:** Full pytest node ID (e.g., `test_models.py::test_all_models_torch[resnet18-single_device-inference]`)
2.   **Exception Traceback:** Stack trace showing where the failure occurred
3.   **Captured Output:** stdout/stderr from the test
4.   **Test Report Properties:** Metadata recorded by `record_model_test_properties()`

**Figure: Test Failure Capture and Analysis Flow**

**Sources:**[tests/runner/test_models.py 171-182](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_models.py#L171-L182)[tests/runner/test_utils.py 221-263](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_utils.py#L221-L263)

### Bringup Status Tracking

The test framework tracks where failures occur in the compilation/execution pipeline using bringup status markers:

`class BringupStatus(Enum):    NOT_STARTED = "not_started"    FAILED_FE_COMPILATION = "failed_fe_compilation"    FAILED_TTMLIR_COMPILATION = "failed_ttmlir_compilation"    FAILED_RUNTIME = "failed_runtime"    INCORRECT_RESULT = "incorrect_result"    PASSED = "passed"    UNKNOWN = "unknown"`
The status is determined by:

1.   Reading `._bringup_stage.txt` file (written by C++ compilation pipeline when `ENABLE_BRINGUP_STAGE_LOGGING=1`)
2.   Mapping stage names to status: 
    *   `FE_COMPILATION_START` → `FAILED_FE_COMPILATION`
    *   `TTMLIR_COMPILATION_START` → `FAILED_TTMLIR_COMPILATION`
    *   `RUNTIME_EXECUTION_START` → `FAILED_RUNTIME`

3.   Comparing outputs to determine `PASSED` or `INCORRECT_RESULT`

**Sources:**[tests/runner/test_utils.py 191-218](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_utils.py#L191-L218)[tests/runner/test_utils.py 555-567](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_utils.py#L555-L567)[.gitignore 36](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.gitignore#L36-L36)

### Failure Classification System

**Figure: Failure Classification and Component Detection System**

The failure classification system uses pattern matching on exception data to categorize failures:

1.   **Exception Data Extraction:** Extract class name, message, traceback from exception
2.   **Pattern Matching:** Match against `FailingReasons` enum using `ExceptionCheck` patterns
3.   **Component Detection:** Determine failing component (METAL, TTNN, XLA, TORCH)
4.   **Summary Extraction:** Extract relevant error details from stdout/stderr

Common failing reasons include:

| Failing Reason | Description | Component |
| --- | --- | --- |
| `DATA_MISMATCH_WRONG_PCC` | PCC below threshold | XLA |
| `DATA_MISMATCH_PCC_IS_NAN` | PCC is NaN | XLA |
| `OUT_OF_MEMORY` | L1 buffer allocation failed | METAL |
| `SEG_FAULT` | Segmentation fault | - |
| `XLA_SYNC_MULTI_FATAL_ERROR` | Fatal error in xla_sync_multi | XLA |

**Sources:**[tests/infra/utilities/failing_reasons/finder.py 15-101](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/infra/utilities/failing_reasons/finder.py#L15-L101)[tests/infra/utilities/failing_reasons/checks_xla.py 13-155](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/infra/utilities/failing_reasons/checks_xla.py#L13-L155)[tests/runner/test_utils.py 221-263](https://github.com/tenstorrent/tt-xla/blob/c77995f6/tests/runner/test_utils.py#L221-L263)

### Comparison Results and Metrics

When tests complete execution but produce incorrect results, comparison metrics are recorded:

`comparison_results: list[ComparisonResult] # Each ComparisonResult contains:result.pcc           # Pearson correlation coefficientresult.atol          # Absolute toleranceresult.passed        # Boolean pass/failresult.error_message # Description of failure`
The test framework records:

*   **First failing result** or last result if all passed
*   **PCC/ATOL thresholds** from comparison config
*   **Whether assertion is enabled** for each comparator
