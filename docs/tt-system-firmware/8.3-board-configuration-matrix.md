The Board Configuration Matrix is a centralized configuration system that defines hardware platforms, test runners, and firmware variants for the CI/CD infrastructure. It enables dynamic test matrix generation across multiple board types and firmware configurations.

For information about how these boards are used in hardware testing, see [Hardware Test Infrastructure](https://deepwiki.com/tenstorrent/tt-system-firmware/8.2-hardware-test-infrastructure). For details on firmware binary selection for different boards, see [MRISC Firmware for GDDR](https://deepwiki.com/tenstorrent/tt-system-firmware/5.1-mrisc-firmware-for-gddr).

## Overview

The board configuration matrix is defined in a single JSON file that serves as the source of truth for all hardware-related CI operations. This configuration drives test matrix generation, firmware binary selection, and runner assignment across GitHub Actions workflows.

**Sources:**[.github/ci_boards.json 1-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L1-L46)

## Configuration File Structure

The board configuration is stored in `.github/ci_boards.json` as a JSON array. Each entry defines a complete board configuration with hardware and software attributes.

### Board Configuration Schema

| Field | Type | Purpose |
| --- | --- | --- |
| `board` | string | Board identifier (e.g., `"p100a"`, `"galaxy"`) |
| `runs-on` | string | GitHub Actions runner label for job assignment |
| `product` | string | Product family: `"bh"` (Blackhole) or `"wh"` (Wormhole) |
| `metal-image` | string | Docker container image for TT-Metal integration tests |
| `metal-target` | string | Target platform identifier for Metal runtime |
| `kmd_build` | string | Kernel module driver version (e.g., `"2.5.0"`) |

**Sources:**[.github/ci_boards.json 1-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L1-L46)

*   `${{ matrix.config.metal-image }}` - Metal container (if applicable)

## Board-Specific Firmware Binary Selection

The board identifier from `ci_boards.json` maps directly to GDDR parameter binary files defined in `zephyr/module.yml`. This enables automatic selection of board-specific memory training parameters.

### GDDR Parameter Binary Mapping

The board name is uppercased and used to construct the firmware binary filename pattern:

*   Board: `p100a` → Binary: `tt_blackhole_gddr_params_P100A.bin`
*   Board: `p150a` → Binary: `tt_blackhole_gddr_params_P150A.bin`
*   Board: `p300a` → Binary: `tt_blackhole_gddr_params_P300A.bin`
*   Board: `galaxy` → Binary: `tt_blackhole_gddr_params_GALAXY.bin`

**Sources:**[.github/ci_boards.json 3-27](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L3-L27)[zephyr/module.yml 100-189](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L100-L189)

## Runner Label Assignment

The `runs-on` field maps boards to GitHub Actions self-hosted runner pools with JTAG access and specific hardware configurations.

### Runner Architecture

| Runner Label | Hardware Access | Purpose |
| --- | --- | --- |
| `p100a-jtag` | P100A board + JTAG | Single-chip Blackhole tests |
| `p150a-jtag` | P150A board + JTAG | Single-chip Blackhole tests |
| `p300a-jtag` | P300A board + JTAG | Dual-chip Blackhole tests |
| `tt-galaxy-bh` | Galaxy (32-chip) + JTAG | Multi-chip system tests |
| `n150-jtag` | N150 board + JTAG | Single-chip Wormhole tests |
| `n300-jtag` | N300 board + JTAG | Dual-chip Wormhole tests |

All runners require:

*   Physical hardware access via `/dev/tenstorrent`
*   USB JTAG access via `/dev/bus/usb`
*   IPMI interface via `/dev/ipmi0` (Blackhole boards)
*   KMD version selection capability via `/opt/tenstorrent/kmd-builds/`

**Sources:**[.github/ci_boards.json 4-42](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L4-L42)[.github/workflows/hardware-smoke.yml 58-63](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml#L58-L63)

## Configuration Validation

The configuration file is validated through several mechanisms:

1.   **JSON Schema Validation**: GitHub Actions validates JSON syntax on load
2.   **Required Fields**: Workflows fail if required fields are missing from `matrix.config`
3.   **Runner Availability**: Jobs fail if specified runner label has no available runners
4.   **Binary Existence**: West blob verification ensures firmware binaries exist for configured boards

**Sources:**[.github/workflows/prepare-container/action.yml 107-111](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/prepare-container/action.yml#L107-L111)[.github/workflows/prepare-zephyr/action.yml 82-86](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/prepare-zephyr/action.yml#L82-L86)
