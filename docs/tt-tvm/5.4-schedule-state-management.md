This document describes the internal state management system of TensorIR schedules, focusing on how the schedule tracks and maintains references to IR nodes during transformations. For information about the scheduling primitives themselves, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api). For details on traced vs concrete schedules, see [ConcreteSchedule vs TracedSchedule](https://deepwiki.com/tenstorrent/tt-tvm/5.5-concreteschedule-vs-tracedschedule).

## Overview

Schedule state management is the foundation of TensorIR's scheduling system. It maintains:

1.   A mapping from IR statements to internal references (`StmtSRef`)
2.   A symbol table mapping random variables to IR constructs
3.   Metadata about blocks including scope information and cached analysis results
4.   The current IRModule being transformed

The state management system ensures that all schedule operations can efficiently locate and modify the correct IR nodes while preserving the ability to copy and fork schedules.

## Core State Components

### ScheduleState

The `ScheduleState` class is the central data structure that tracks the current state of the IR being scheduled.

**Key Responsibilities:**

| Responsibility | Description | Key Methods |
| --- | --- | --- |
| IR Tracking | Maintains the IRModule and maps statements to references | `mod`, `stmt2ref` |
| Block Analysis | Caches analysis results for blocks | `GetBlockScope()`, `GetBlockInfo()` |
| Affine Binding | Tracks whether block bindings are affine | `IsAffineBlockBinding()` |
| Verification | Optionally verifies state consistency | `DebugVerify()` |

### StmtSRef: The Reference Tree

`StmtSRef` (Statement Structured Reference) is a lightweight reference to a statement in the IR that forms a tree structure parallel to the IR tree.

**Key Properties:**

*   **Immutable References**: StmtSRefs are stable references that don't change when the IR is mutated
*   **Tree Structure**: Each sref has a parent pointer, mirroring the IR's nesting structure
*   **Lightweight**: Contains only a pointer to the statement and tree metadata
*   **Special Values**: `InlineMark()` and `RootMark()` are sentinel values for special cases

### Symbol Table: Random Variables to SRefs

The symbol table in `ConcreteScheduleNode` maps random variables (RVs) to their actual values in the IR.

**Mapping Rules:**

| Random Variable Type | Maps To | Purpose |
| --- | --- | --- |
| `BlockRV` | `StmtSRef` pointing to `BlockNode` | Identifies a block in the IR |
| `LoopRV` | `StmtSRef` pointing to `ForNode`, or special marks | Identifies a loop, or inline/root markers |
| `ExprRV` | `Integer` value | Represents sampled integer values |

**Key Operations:**

```
// Creating a random variable from an sref
template <class T>
T CreateRV(const StmtSRef& sref) {
  T rv;
  symbol_table_.Set(rv, sref);
  return rv;
}

// Looking up the sref for a random variable
StmtSRef GetSRef(const BlockRV& block_rv) const {
  auto it = symbol_table_.find(block_rv);
  return it->second.as<StmtSRef>();
}
```

## ConcreteScheduleNode vs TracedScheduleNode

The schedule system has two implementations that manage state differently:

**Key Differences:**

| Aspect | ConcreteScheduleNode | TracedScheduleNode |
| --- | --- | --- |
| Base Class | `ScheduleNode` | `ConcreteScheduleNode` |
| Trace | Returns `NullOpt` | Maintains and returns `Trace` object |
| Use Case | Direct transformation, faster | Recording for replay/serialization |
| Overhead | Minimal | Records each operation |

## Schedule State Copying

Copying a schedule creates a completely independent copy of the state, including reconstructing the entire SRef tree.

**Copy Guarantees:**

1.   **Complete SRef Tree Reconstruction**: All srefs are new objects with correct parent pointers
2.   **IRModule Untouched**: The actual IR is not modified during copy
3.   **Valid Random Variables**: All RVs in the symbol table point to valid new srefs
4.   **Fresh Analyzer**: A new stateless analyzer is created
5.   **Forked Random State**: Random seed is forked for independent sampling

**Implementation Details:**

The `ScheduleCopier` class uses an `old2new_` map to track the correspondence between old and new srefs:

```
// Simplified pseudocode
class ScheduleCopier {
  unordered_map<const StmtSRefNode*, StmtSRef> old2new_;
  
  void Copy(ScheduleState* new_state, TSymbolTable* new_symbol_table) {
    // Phase 1: Create all new srefs without parents
    for (auto& [stmt, old_sref] : state->stmt2ref) {
      old2new_[old_sref] = StmtSRef(stmt, nullptr, old_sref->seq_index);
    }
    
    // Phase 2: Fill in parent pointers
    for (auto& [old_sref, new_sref] : old2new_) {
      if (old_sref->parent) {
        new_sref->parent = old2new_[old_sref->parent];
      }
    }
    
    // Phase 3-5: Copy other structures with new srefs
  }
}
```

## State Verification and Debug Checking

The schedule state supports optional verification to catch errors early during development.

**Verification Checks:**

1.   **SRef Tree Verification** (`VerifySRefTree`):

    *   Parent pointers are consistent
    *   All statements in IRModule have corresponding srefs
    *   Tree structure matches IR nesting

2.   **Cached Flags Verification** (`VerifyCachedFlags`):

    *   `affine_binding` flag matches actual binding analysis
    *   `region_cover` flag matches coverage analysis
    *   `stage_pipeline` flag matches pipeline property

**Debug Mask Options:**

| Mask Bit | Check | Purpose |
| --- | --- | --- |
| `kVerifySRefTree` | SRef tree consistency | Catch invalid tree structure |
| `kVerifyCachedFlags` | Cached flag accuracy | Detect stale analysis results |

## Random Variables: User API Bridge

Random variables (RVs) provide a stable, user-facing handle to IR constructs that can survive IR mutations.

**Why Random Variables?**

| Problem | Solution |
| --- | --- |
| IR nodes change after mutations | RVs provide stable handles |
| Users shouldn't deal with internal srefs | RVs abstract away implementation |
| Need to track multiple schedule versions | Each schedule has its own symbol table |
| Sampling produces integer values | ExprRV wraps sampled integers |

**Random Variable Lifecycle:**

1.   **Creation**: Schedule creates BlockRV/LoopRV when returning blocks/loops to user
2.   **Storage**: RV stored in symbol table with mapping to StmtSRef
3.   **Lookup**: When user passes RV back, schedule looks up the corresponding sref
4.   **Transformation**: Primitive operates on srefs, returns new srefs
5.   **Update**: New RVs created for new srefs, old RVs may be invalidated

## State Replacement and IR Mutation

When a schedule primitive modifies the IR, the state must be updated to reflect the changes.

**State.Replace() Parameters:**

| Parameter | Type | Purpose |
| --- | --- | --- |
| `src_sref` | `StmtSRef` | The statement being replaced |
| `tgt_stmt` | `Stmt` | The new statement (or subtree) |
| `block_sref_reuse` | `Map<Block, Block>` | Maps old blocks to new blocks to preserve analysis |

**Replacement Process:**

1.   Find the PrimFunc containing the statement being replaced
2.   Use an IR mutator to replace the old statement with the new one
3.   Reconstruct srefs for new statements
4.   Update `stmt2ref` mapping to point to new srefs
5.   If blocks are reused (via `block_sref_reuse`), preserve their `BlockInfo`
6.   Run verification if debug mode is enabled
