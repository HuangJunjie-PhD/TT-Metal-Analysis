## Purpose and Scope

This document describes TT-SMI's dependency architecture, including runtime libraries, development tools, and version management strategies. It covers the external packages that TT-SMI relies upon, their version requirements, and how they integrate with the application code. For information about the build and release pipeline that packages these dependencies, see [Build and Release System](https://deepwiki.com/tenstorrent/tt-smi/7-build-and-release-system). For details about the backend's use of hardware interface libraries, see [Backend System](https://deepwiki.com/tenstorrent/tt-smi/6.2-backend-system-(ttsmibackend)).

## Dependency Tiers

TT-SMI organizes dependencies into five distinct functional tiers, each serving a specific architectural role. The dependency declarations in `pyproject.toml` establish the complete dependency graph.

**Dependency Tier Classification**

| Tier | Dependencies | Version Strategy | Purpose |
| --- | --- | --- | --- |
| Critical Hardware | `pyluwen`, `tt_tools_common` | Compatible release (`~=`) | Hardware abstraction and device management |
| UI Framework | `textual`, `rich` | Minimum version (`>=`) | Text user interface rendering |
| Data/Config | `pydantic`, `distro`, `tomli` | Minimum version (`>=`) | Data validation and system detection |
| Optional Runtime | `elasticsearch` | Minimum version (`>=`) | Telemetry logging (not required) |
| Build/Dev | `setuptools`, `wheel`, `pre-commit`, `black`, `pytest` | Mixed | Package building and code quality |

Sources: [pyproject.toml 26-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L46)

## Critical Hardware Dependencies

### pyluwen

`pyluwen` provides the low-level hardware interface for Tenstorrent devices. It is the primary abstraction layer for PCI device access and telemetry collection.

**Key Integration Points in TTSMIBackend:**

**pyluwen Usage Pattern:**

The backend interacts with `pyluwen.PciChip` objects throughout device lifecycle:

*   **Device Discovery**: [tests/test_reset.py 17](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L17-L17) uses `pci_scan()` to enumerate PCI devices
*   **Chip Abstraction**: [tt_smi/tt_smi_backend.py 55](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L55-L55) accepts `List[PciChip]` in constructor
*   **Telemetry Collection**: [tt_smi/tt_smi_backend.py 244-250](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L244-L250) calls `chip.as_wh().get_telemetry()` or `chip.as_bh().get_telemetry()`
*   **PCI Properties**: [tt_smi/tt_smi_backend.py 316](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L316-L316) calls `device.get_pci_bdf()` for sysfs path construction
*   **Reset Operations**: [tt_smi/tt_smi_backend.py 889-891](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L889-L891) calls `run_wh_ubb_ipmi_reset()` and `run_ubb_wait_for_driver_load()`

**Version Requirement**: `~0.8.0` (compatible release operator)

*   Allows versions >= 0.8.0 and < 0.9.0
*   Ensures API stability within minor version
*   Coordinated with `tt_tools_common` releases

Sources: [pyproject.toml 31](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L31-L31)[tt_smi/tt_smi_backend.py 28-32](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L28-L32)[tt_smi/tt_smi_backend.py 244-250](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L244-L250)[tests/test_reset.py 7-8](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py#L7-L8)

## Optional Dependencies

### elasticsearch

`elasticsearch` enables optional telemetry logging to Elasticsearch clusters for long-term monitoring and analytics.

**Integration Pattern:**

The dependency is declared at [pyproject.toml 28](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L28-L28) but marked as optional in the architecture. The `tt_smi.log` module provides Elasticsearch integration through Pydantic models that can serialize to Elasticsearch-compatible JSON format.

**Optional Behavior:**

*   TT-SMI functions fully without Elasticsearch
*   When configured, logs can be sent to Elasticsearch via `TTSMILog` models
*   JSON snapshots remain functional without Elasticsearch installation

**Version Requirement**: `>=8.11.0` (minimum version)

*   Supports Elasticsearch 8.x API
*   Modern cluster compatibility

Sources: [pyproject.toml 28](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L28-L28)

## Version Requirements Strategy

TT-SMI employs different version constraint strategies based on dependency criticality and stability requirements.

**Semantic Versioning Constraints:**

| Operator | Constraint | Example | Allows | Use Case |
| --- | --- | --- | --- | --- |
| `~=` | Compatible release | `pyluwen~=0.8.0` | ≥0.8.0, <0.9.0 | Hardware interface stability |
| `>=` | Minimum version | `textual>=0.59.0` | ≥0.59.0 | Feature requirements |
| `==` | Exact version | `black==24.10.0` | Exact match | Deterministic formatting |
| (none) | Any version | `tomli` | Any | Standard backports |

**Coordinated Release Pattern:**

**Rationale for Version Strategies:**

1.   **Hardware Dependencies (`~=`):**

    *   `pyluwen` and `tt_tools_common` use compatible release operator
    *   Ensures ABI compatibility with hardware drivers
    *   Synchronized releases across Tenstorrent toolchain
    *   Prevents breaking changes within minor versions

2.   **UI Dependencies (`>=`):**

    *   Minimum version ensures required features exist
    *   Allows security updates and non-breaking enhancements
    *   Application code handles API variations

3.   **Build Tools (`==` for black):**

    *   Exact versions ensure consistent formatting across team
    *   Different versions per Python version for compatibility
    *   Deterministic builds and code style

4.   **Standard Backports (no constraint):**

    *   `tomli` is a stable standard library backport
    *   No version-specific features required

Sources: [pyproject.toml 26-46](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L46)

## Integration Points Summary

**Critical Integration Points by Module:**

**Dependency Resolution Order:**

1.   Build-time: `setuptools`, `wheel` construct package
2.   Runtime-critical: `pyluwen`, `tt_tools_common` enable hardware access
3.   Runtime-UI: `textual`, `rich` enable interactive mode
4.   Runtime-data: `pydantic`, `distro`, `tomli` provide data handling
5.   Optional: `elasticsearch` enables telemetry logging
6.   Development: `pre-commit`, `black`, `pytest` ensure quality
