## Purpose and Scope

This document describes the CMake-based build system for tt-npe, including configuration structure, dependency management, compilation artifacts, and environment setup. It covers how C++ sources are compiled into shared libraries, how Python bindings are generated, and how the build orchestration scripts simplify the build process.

For information about running tests after building, see [Testing Framework](https://deepwiki.com/tenstorrent/tt-npe/8.2-testing-framework). For details on integrating the built artifacts into applications, see [Python API and CLI](https://deepwiki.com/tenstorrent/tt-npe/6-python-api-and-cli).

### Library Configuration

The [tt_npe/CMakeLists.txt 1-102](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L1-L102) defines three primary build targets:

#### Common Dependencies Interface

[tt_npe/CMakeLists.txt 26-38](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L26-L38) creates `npe_common_libs` interface target:

`add_library(npe_common_libs INTERFACE)target_link_libraries(npe_common_libs INTERFACE    fmt::fmt-header-only    magic_enum    nlohmann_json::nlohmann_json    simdjson    Boost::container    Boost::unordered    Boost::program_options    libzstd_static)`
This interface is linked by all tt-npe targets, ensuring consistent dependency availability.

#### libtt_npe.so (Core Library)

Defined in [tt_npe/CMakeLists.txt 42-59](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L42-L59):

*   **Sources**: All `.cpp` files from `tt_npe/cpp/src/` (glob pattern)
*   **Headers**: Public headers from `tt_npe/cpp/include/*.hpp`
*   **RPATH**: Set to `$ORIGIN/../lib:$ORIGIN/` for relocatable installation
*   **Include Directories**: 
    *   Build interface: `${CMAKE_CURRENT_SOURCE_DIR}/cpp/include`
    *   Install interface: `include`
    *   Private: `${zstd_SOURCE_DIR}/lib` (workaround for zstd export issue)

#### tt_npe_pybind.so (Python Module)

Defined in [tt_npe/CMakeLists.txt 64-65](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L64-L65):

`pybind11_add_module(tt_npe_pybind cpp/pybind/bindings.cpp)target_link_libraries(tt_npe_pybind PRIVATE tt_npe)`
Uses `pybind11_add_module` to create a Python extension module that exposes C++ API via Python bindings.

#### tt_npe_ut (Unit Tests)

Defined in [tt_npe/CMakeLists.txt 70-80](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L70-L80):

*   **Sources**: All `.cpp` files from `tt_npe/cpp/test/`
*   **Dependencies**: Links both `npe_common_libs`, `test_common_libs` (googletest), and `tt_npe`

**Sources:**[tt_npe/CMakeLists.txt 1-102](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L1-L102)

## Dependency Management

### CPM-Based Dependencies

The [cmake/dependencies.cmake 1-101](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L1-L101) uses CPM (CMake Package Manager) for most external libraries:

| Dependency | Repository | Version/Tag | Purpose |
| --- | --- | --- | --- |
| `googletest` | `google/googletest` | `v1.16.0` | Unit testing framework |
| `reflect` | `boost-ext/reflect` | `v1.1.1` | Reflection utilities |
| `magic_enum` | `Neargye/magic_enum` | `v0.9.6` | Enum to string conversion |
| `fmt` | `fmtlib/fmt` | `11.1.4` | String formatting library |
| `json` | `nlohmann/json` | `v3.12.0` | JSON parsing/serialization |
| `simdjson` | `simdjson/simdjson` | `3.12.3` | High-performance JSON parsing |

CPM configuration sets cache location via [cmake/dependencies.cmake 5](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L5-L5):

`set(ENV{CPM_SOURCE_CACHE} "${PROJECT_SOURCE_DIR}/.cpmcache")`
This ensures downloaded dependencies are cached and reused across builds.

**Sources:**[cmake/dependencies.cmake 43-88](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L43-L88)

### FetchContent Dependencies

Two dependencies use CMake's `FetchContent` module:

#### zstd (Compression Library)

[cmake/dependencies.cmake 12-24](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L12-L24) configures zstd build:

`set(ZSTD_BUILD_STATIC ON)set(ZSTD_BUILD_SHARED OFF)set(ZSTD_BUILD_PROGRAMS OFF) FetchContent_Declare(    zstd    URL "https://github.com/facebook/zstd/releases/download/v1.5.5/zstd-1.5.5.tar.gz"    DOWNLOAD_EXTRACT_TIMESTAMP TRUE    SOURCE_SUBDIR build/cmake)FetchContent_MakeAvailable(zstd)`
Downloads tarball from GitHub releases rather than cloning repository, and builds only static library without programs.

#### pybind11 (Python Bindings)

[cmake/dependencies.cmake 92-99](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L92-L99) fetches pybind11:

`find_package(Python COMPONENTS Interpreter Development REQUIRED)FetchContent_Declare(    pybind11    GIT_REPOSITORY https://github.com/pybind/pybind11.git    GIT_TAG        v2.13.6)FetchContent_MakeAvailable(pybind11)`
First finds system Python installation, then fetches pybind11 to match Python version.

**Sources:**[cmake/dependencies.cmake 12-100](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L12-L100)

### Boost Library Management

Boost libraries are acquired via custom `fetch_boost_library` function defined in [cmake/fetch_boost.cmake](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/fetch_boost.cmake) The [cmake/dependencies.cmake 30-40](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L30-L40) fetches:

*   `Boost::core` - Core utilities
*   `Boost::smart_ptr` - Smart pointer types
*   `Boost::container` - Container implementations
*   `Boost::unordered` - Unordered containers
*   `Boost::tokenizer` - String tokenization
*   `Boost::program_options` - Command-line parsing

Additionally creates a `span` interface target linking `Boost::core` for span support.

**Sources:**[cmake/dependencies.cmake 28-40](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/dependencies.cmake#L28-L40)[cmake/fetch_boost.cmake](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/cmake/fetch_boost.cmake)

## Build Artifacts

### Compilation Output Structure

**Sources:**[tt_npe/CMakeLists.txt 85-101](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L85-L101)

### libtt_npe.so

The core C++ shared library containing:

*   **npeAPI**: Entry point functions (`runNPE`, `InitAPI`)
*   **npeEngine**: Simulation loop implementation
*   **npeStats**: Statistics collection and timeline generation
*   **npeWorkloadIngest**: Workload parsing and validation
*   **Device Models**: `WormholeB0DeviceModel`, `BlackholeDeviceModel`, `WormholeMultichipDeviceModel`
*   **Utility Classes**: `npeTransferDependencyTracker`, `npeDeviceState`, etc.

Exports public API headers defined in [tt_npe/CMakeLists.txt 19-21](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L19-L21)

### tt_npe_pybind.so

Python extension module built from [tt_npe/cpp/pybind/bindings.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp) Provides Python access to:

*   `npe.Config` class
*   `npe.Workload`, `npe.Phase`, `npe.Transfer` classes
*   `npe.Stats` class
*   `npe.Exception` exception type
*   `npe.InitAPI()` and `npe_api.runNPE()` functions

Linked against `libtt_npe.so` and installed to `install/lib/` where Python's `PYTHONPATH` can find it.

### tt_npe_ut

Unit test executable linking googletest framework. Contains test cases from `tt_npe/cpp/test/*.cpp`:

*   Device model tests
*   Workload construction tests
*   Engine simulation tests
*   Statistics tests

Executed via [tt_npe/scripts/run_ut.sh](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/run_ut.sh) during CI/CD.

**Sources:**[tt_npe/CMakeLists.txt 42-80](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L42-L80)

## Build Orchestration

### build-npe.sh

The [build-npe.sh 1-34](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/build-npe.sh#L1-L34) script automates the complete build process:

**Key Features:**

*   **Debug Mode**: Pass `debug` as first argument to build with `-O0 -g -DDEBUG`
*   **Ninja Generator**: Uses Ninja for faster incremental builds
*   **Parallel Compilation**: `NINJA_STATUS` environment variable shows progress, `-j16` parallelizes across 16 cores
*   **Clean Install**: Removes previous installation before installing new build
*   **No-Confirm**: Installation output redirected to `/dev/null` for cleaner output

**Usage:**

`# Release build./build-npe.sh # Debug build./build-npe.sh debug`
**Sources:**[build-npe.sh 1-34](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/build-npe.sh#L1-L34)

## Installation and Environment Setup

### Installation Structure

The [tt_npe/CMakeLists.txt 85-101](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L85-L101) defines installation targets:

```
install/
├── bin/
│   ├── tt_npe.py                            # CLI entry point
│   ├── npe_analyze_noc_trace_dir.py        # Batch trace analysis

│   ├── libtt_npe.so                        # Core C++ library

```

#### RPATH Configuration

[tt_npe/CMakeLists.txt 4](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L4-L4) sets:

`set(CMAKE_INSTALL_RPATH "$ORIGIN/../lib:$ORIGIN/")`
This allows `tt_npe_pybind.so` to find `libtt_npe.so` at runtime without requiring `LD_LIBRARY_PATH` configuration. The RPATH is relative to the binary location, supporting relocatable installations.

**Sources:**[tt_npe/CMakeLists.txt 4-101](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/CMakeLists.txt#L4-L101)

### Virtual Environment Creation

Two scripts support Python virtual environment creation:

#### create_venv_uv.sh (Modern Approach)

[create_venv_uv.sh 1-30](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/create_venv_uv.sh#L1-L30) uses `uv` (Astral's fast Python package installer):

`export UV_CACHE_DIR=.uv_cacheexport VIRTUAL_ENV=./python_venvuv venv --quiet $VIRTUAL_ENVuv pip install --quiet -r requirements.txt`
Significantly faster than traditional `pip` for dependency resolution and installation.

#### create_venv.sh (Traditional Approach)

[create_venv.sh 1-44](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/create_venv.sh#L1-L44) uses standard Python `venv` module:

`python3 -m venv $PYTHON_ENV_DIRsource $PYTHON_ENV_DIR/bin/activatepython3 -m pip install -r requirements.txt`
Provides fallback when `uv` is not available.

## Code Formatting

The [.clang-format 1-147](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/.clang-format#L1-L147) defines project-wide C++ formatting rules:

| Rule | Value |
| --- | --- |
| **Style Base** | Google |
| **Column Limit** | 100 characters |
| **Indent Width** | 4 spaces |
| **C++ Standard** | C++11 (minimum for clang-format) |
| **Pointer Alignment** | Right (`int *ptr`) |
| **Break Before Braces** | Attach |
| **Bin Pack Arguments** | false (one per line) |
| **Allow Short Functions** | All (inline single-line functions) |

Developers should run `clang-format` on modified C++ files before committing to maintain consistent code style.
