## Purpose and Scope

This document describes the memory mapping interface provided by the Tenstorrent kernel driver through the `mmap()` system call. The driver multiplexes several distinct mapping types (PCI BARs, TLB windows, and DMA buffers) through a single character device node using an offset encoding scheme. Applications use `mmap()` to map device memory and DMA buffers into their virtual address space for direct hardware access.

The driver also manages the lifecycle of these mappings during device resets, including "zapping" VMAs to prevent illegal access to hardware that is being reinitialized.

## Offset Encoding Scheme

The driver uses a multiplexing scheme where different memory regions are accessed through different offset ranges in the `mmap()` call. The offset parameter encodes both the resource type and caching attributes.

### Offset Constants

| Offset Range (bits 36-39) | Resource Type | Caching |
| --- | --- | --- |
| `0 << 36` | BAR 0 (RESOURCE0) | UC (Uncached) |
| `1 << 36` | BAR 0 (RESOURCE0) | WC (Write-Combining) |
| `2 << 36` | BAR 2 (RESOURCE1) | UC |
| `3 << 36` | BAR 2 (RESOURCE1) | WC |
| `4 << 36` | BAR 4 (RESOURCE2) | UC |
| `5 << 36` | BAR 4 (RESOURCE2) | WC |
| `6 << 36` | TLB windows | UC |
| `7 << 36` | TLB windows | WC |
| `(PAGE_SIZE-256-1) << 32` | DMA buffers | Coherent |

*   **UC**: Uncached, configured via `pgprot_device`[memory.c 1251](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1251-L1251)
*   **WC**: Write-Combining, configured via `pgprot_writecombine`[memory.c 1248](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1248-L1248)

Each resource type occupies a 2^36 byte (64 GB) address window in the mmap offset space. Within each window, the lower bits represent the offset within the resource.

## Query Mappings IOCTL

Before calling `mmap()`, applications typically use `TENSTORRENT_IOCTL_QUERY_MAPPINGS` to discover available mappings and their properties.

### Query Mappings Structure

The `tenstorrent_query_mappings` structure ([ioctl.h 82-85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L82-L85)) returns an array of `tenstorrent_mapping` descriptors.

| Field | Type | Description |
| --- | --- | --- |
| `mapping_id` | `__u32` | Identifier (e.g., `TENSTORRENT_MAPPING_RESOURCE0_UC`) [ioctl.h 33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L33-L33) |
| `mapping_base` | `__u64` | Encoded offset to use in `mmap()` |
| `mapping_size` | `__u64` | Size of the mappable region |

### Implementation

The `ioctl_query_mappings` function ([memory.c 362-440](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L362-L440)) iterates through PCI BARs 0, 2, and 4. It reads the size of each BAR using `pci_resource_len`[memory.c 380](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L380-L380) and populates both UC and WC entries if the BAR is present.

Sources: [memory.c 362-440](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L362-L440)[ioctl.h 65-85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L65-L85)

## PCI BAR Mappings

PCI Base Address Registers (BARs) provide access to device memory and registers. The driver supports mapping BARs with `bar_vm_ops` for reference tracking.

## TLB Window Mappings

Translation Lookaside Buffer (TLB) windows provide configurable address translation for accessing the Network-on-Chip (NOC). Applications must first allocate a TLB via `TENSTORRENT_IOCTL_ALLOCATE_TLB`[ioctl.h 25](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L25-L25)

### TLB Mapping Implementation

The `map_tlb_window` function ([memory.c 1168-1234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1168-L1234)):

1.   **Validates Ownership**: Checks `test_bit(id, priv->tlbs)` to ensure the calling process owns the TLB [memory.c 1196](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1196-L1196)
2.   **Determines Physical Address**: Calls `tt_dev->dev_class->describe_tlb` to get the hardware address of the window [memory.c 1210](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1210-L1210)
3.   **Maps Memory**: Uses `io_remap_pfn_range` to map the physical address into the VMA [memory.c 1224](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1224-L1224)
4.   **Reference Counting**: Increments `tt_dev->tlb_refs[id]` via `tlb_vma_open`[memory.c 1158-1161](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1158-L1161)

Sources: [memory.c 1168-1234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1168-L1234)[memory.c 1107-1166](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1107-L1166)

## DMA Buffer Mappings

DMA buffers are coherent memory allocations. They are mapped using `dma_mmap_coherent`[memory.c 1282](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1282-L1282)

### DMA Buffer Lookup

The `vma_dmabuf_target` function ([memory.c 1016-1040](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1016-L1040)) extracts the `buf_index` from the mmap offset and performs a hashtable lookup in `priv->dmabufs`.

Sources: [memory.c 1016-1040](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1016-L1040)[chardev_private.h 29-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L29-L38)

## VMA Tracking and Zapping

A critical safety feature of the driver is **VMA Zapping**. During a device reset, all user-space mappings to the hardware must be invalidated to prevent the CPU from accessing the device while it is in an unstable state.

### Implementation of Zapping

The `tenstorrent_vma_zap` function ([memory.c 1321-1335](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1321-L1335)) iterates through all open file descriptors (`chardev_private` instances) associated with a `tenstorrent_device`. For each instance, it calls `zap_vma_ptes` on every active BAR mapping.

Sources: [memory.c 1321-1335](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1321-L1335)[chardev.c 569-605](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L569-L605)

## Memory Mapping Lifecycle

### Cleanup on File Descriptor Close

When a file descriptor is closed, `tenstorrent_memory_cleanup` ([memory.c 1289-1319](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1289-L1319)) is called. It performs the following:

1.   **DMA Buffers**: Iterates through `priv->dmabufs` and calls `dma_free_coherent`[memory.c 1296](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1296-L1296)
2.   **Pinned Pages**: Iterates through `priv->pinned_pages` and releases them [memory.c 1303](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1303-L1303)
3.   **Peer BARs**: Unmaps any peer-to-peer mappings [memory.c 1310](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1310-L1310)

Sources: [memory.c 1289-1319](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1289-L1319)[chardev.c 501-541](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L501-L541)
