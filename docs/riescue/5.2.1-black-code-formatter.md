## Overview

Black is a deterministic code formatter that targets Python whitespace and formatting rules. It removes debates about code style by enforcing a single, consistent format automatically. The RiESCUE project uses Black to ensure all Python code follows uniform styling, making code reviews faster and reducing merge conflicts related to formatting.

Black operates by:

1.   Parsing Python source code into an Abstract Syntax Tree (AST)
2.   Reformatting the AST according to its style rules
3.   Writing the reformatted code back to disk

The project pins Black to version `25.1.0` to ensure consistent formatting across all development environments and CI runs.

## Configuration

Black reads its configuration from the `[tool.black]` section in `pyproject.toml`. The RiESCUE project defines three primary configuration options:

| Configuration Key | Value | Purpose |
| --- | --- | --- |
| `line-length` | `200` | Maximum characters per line before wrapping |
| `workers` | `8` | Number of parallel processes for formatting |
| `exclude` | Pattern (see below) | Directories and files to skip |

### Line Length

The project uses a line length of 200 characters, significantly longer than Black's default of 88 characters. This accommodates the project's coding style and reduces forced line breaks in complex expressions.

### Parallel Workers

The `workers = 8` setting enables Black to format files in parallel using 8 worker processes, improving performance on multi-core systems when formatting large numbers of files.

### Exclusion Pattern

The exclude pattern prevents Black from formatting files in external dependencies, generated code, and development artifacts:

`/(.git|riscv-dv|riscv-isa-sim|whisper|fpgen|riescue_config|.*_clone|opensrc.*|.*-scripts|.venv)/`
**Excluded Directories:**

*   `.git` - Git metadata
*   `riscv-dv` - External RISC-V DV framework
*   `riscv-isa-sim` - Spike simulator source
*   `whisper` - Whisper simulator source
*   `fpgen` - Floating-point generator
*   `riescue_config` - Configuration data
*   `*_clone` - Cloned repositories (pattern match)
*   `opensrc*` - Open source dependencies (pattern match)
*   `*-scripts` - Script directories (pattern match)
*   `.venv` - Python virtual environments

**Sources:**[pyproject.toml 1-5](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml#L1-L5)

## Local Usage

### Installation

Black is installed as a project dependency and is available within the Singularity container. For local development outside the container, install Black using pip:

`python3 -m pip install black --user`
The project pins to `black==25.1.0` in its dependencies to ensure version consistency.

### Formatting Files

To format all Python files in the repository:

`./infra/container-run black .`
This command:

1.   Executes `black .` inside the container
2.   Reads configuration from `pyproject.toml`
3.   Recursively finds all `.py` files (except excluded paths)
4.   Reformats them in-place
5.   Reports which files were changed

### Preview Changes

To see what changes Black would make without modifying files:

`./infra/container-run black --diff .`
The `--diff` flag shows a unified diff of proposed changes, allowing you to review formatting modifications before applying them.

### Checking Format Compliance

To verify files are properly formatted without making changes:

`./infra/container-run black --check .`
This command:

*   Returns exit code `0` if all files are properly formatted
*   Returns non-zero exit code if any files need formatting
*   Prints the names of files that would be reformatted

## CI/CD Integration

The GitHub Actions workflow runs Black in check mode as part of the lint phase. This ensures that all code merged into the repository is properly formatted.

### CI Workflow Execution

The CI pipeline replicates the check command:

`./infra/container-run black --check`
If Black detects any files that need formatting:

1.   The command returns a non-zero exit code
2.   The CI job fails
3.   The pipeline stops before running tests
4.   The developer must reformat code and push again

### Workflow Position

Black runs early in the CI pipeline, immediately after SPDX header checks. This fail-fast approach prevents wasting CI resources on improperly formatted code.

## Pre-commit Hook Integration

Pre-commit hooks automate Black execution before code is committed, catching formatting issues earlier in the development workflow. This is optional but recommended for active contributors.

### Setup

To install pre-commit hooks:

1.   Install the pre-commit framework:

`python3 -m pip install pre-commit --user`
1.   Install hooks for the repository:

`pre-commit install`
This reads `.pre-commit-config.yaml` and configures Git to run Black automatically on staged Python files before each commit.

### Behavior

When pre-commit hooks are enabled:

1.   Developer runs `git commit`
2.   Pre-commit hook intercepts the commit
3.   Black formats any staged `.py` files
4.   If files were modified: 
    *   The commit is aborted
    *   Developer must stage the formatted files
    *   Developer re-runs `git commit`

5.   If no changes needed: 
    *   The commit proceeds normally

### Local vs. CI Black

Pre-commit can use either:

*   **Isolated Black**: Pre-commit downloads Black from GitHub, ensuring reproducibility
*   **Local Black**: Uses the pip-installed Black from the local environment

The project recommends the default isolated approach for consistency with CI.

## Integration with RiESCUE Package Structure

Black formats all Python code in the RiESCUE package hierarchy, ensuring consistent style across:

*   **riescue/dtest_framework/** - RiescueD directed test framework
*   **riescue/compliance/** - RiescueC compliance test generator
*   **riescue/lib/** - Shared utilities and toolchain abstractions
*   **riescue/*.py** - Top-level command-line entry points
*   **tests/** - Unit and integration tests
*   **infra/** - Container and build scripts
*   **docs/** - Documentation build scripts

The exclusion pattern ensures Black does not format:

*   External dependencies cloned during builds (Whisper, Spike, riscv-dv)
*   Generated configuration files
*   Third-party open source components

This selective formatting maintains code quality standards for project-owned code while preserving the original formatting of external dependencies.

## Best Practices

### Before Committing

Always run Black locally before committing changes:

`./infra/container-run black .`
This prevents CI failures and speeds up the review process.

### Reviewing Black Changes

When Black reformats code during development:

1.   Review the changes using `git diff`
2.   Ensure formatting changes don't obscure logical changes
3.   Commit formatting changes separately from functional changes when practical
4.   This makes code reviews easier to understand

### Version Consistency

The project pins Black to version `25.1.0` in `pyproject.toml`. Never modify this version locally, as it will cause formatting inconsistencies between developers and CI.

### Handling Merge Conflicts

If Black changes cause merge conflicts:

1.   Resolve the conflict normally
2.   Run `black .` on the resolved files
3.   Stage and commit the resolution

Black's deterministic formatting will produce consistent results regardless of merge order.
