# Application Layer Architecture

Relevant source files
*   [tt_smi/constants.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py)
*   [tt_smi/tt_smi.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py)

This document describes the presentation layer of TT-SMI: the `TTSMI` Textual application class, its widget composition, action handlers, and the background telemetry thread that enables real-time monitoring. For backend data collection mechanisms, see [Backend System (TTSMIBackend)](https://deepwiki.com/tenstorrent/tt-smi/6.2-backend-system-(ttsmibackend)). For the complete data flow from hardware to UI, see [Data Collection Pipeline](https://deepwiki.com/tenstorrent/tt-smi/6.3-data-collection-pipeline).

## Overview

The application layer is implemented as a single Textual `App` subclass named `TTSMI` that orchestrates the entire user interface. It manages three primary responsibilities:

1.   **Widget Composition**: Assembling the UI hierarchy with header, sidebar, tabbed content, and data tables
2.   **Action Handling**: Responding to user keyboard inputs and tab switching
3.   **Background Updates**: Running a daemon thread that refreshes telemetry data every 100ms

The `TTSMI` class serves as the presentation controller, delegating all hardware interaction to the `TTSMIBackend` while focusing exclusively on UI concerns.

**Sources:**[tt_smi/tt_smi.py 67-575](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L67-L575)

## TTSMI Application Class Structure

The `TTSMI` class inherits from Textual's `App` base class and holds a reference to a `TTSMIBackend` instance. The separation of concerns is clear: `TTSMI` handles presentation and user interaction, while `TTSMIBackend` manages all hardware communication.

**Sources:**[tt_smi/tt_smi.py 67-112](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L67-L112)[tt_smi/tt_smi.py 91-112](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L91-L112)

## Widget Hierarchy and Composition

The UI is composed in the `compose()` method, which yields widgets in a specific hierarchical structure. Textual's context managers (`with` statements) establish parent-child relationships.

### Widget Assembly

The `compose()` method at [tt_smi/tt_smi.py 114-146](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L114-L146) constructs the following layout:

| Widget | ID | Purpose | Configuration |
| --- | --- | --- | --- |
| `TTHeader` | (none) | Application title bar | Shows app name and version |
| `Container` | `app_grid` | Main layout container | Holds sidebar and tabs |
| `Vertical` | `left_col` | Sidebar column | Contains host info menu |
| `TTHostCompatibilityMenu` | `host_info` | Host system details | Driver, OS, kernel info |
| `TabbedContent` | `tab_container` | Tab switcher | Three tabs for different views |
| `TTDataTable` | `tt_smi_device_info` | Device info table | 9 columns, 2 header rows |
| `TTDataTable` | `tt_smi_telem` | Telemetry table | 8 columns, 2 header rows |
| `TTDataTable` | `tt_smi_firmware` | Firmware table | 8 columns, 2 header rows |
| `Footer` | (none) | Keybinding hints | Shows available actions |

The sidebar visibility is controlled by the `show_sidebar` parameter, which can be toggled via the `-c/--compact` CLI flag or the `C` key at runtime.

## Telemetry Thread Management

Real-time telemetry updates are achieved through a background daemon thread that runs continuously while the telemetry tab is active. This design keeps the UI responsive while updating data at 100ms intervals.

### Thread Lifecycle

**Initialization**[tt_smi/tt_smi.py 568-574](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L568-L574)

*   The `on_tabbed_content_tab_activated()` event handler detects when tab-2 (telemetry) becomes active
*   Calls `dispatch_telem_thread()` to start the background thread

**Thread Creation**[tt_smi/tt_smi.py 550-566](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L550-L566)

*   Checks global `RUNNING_TELEM_THREAD` flag to prevent duplicate threads
*   Registers signal handlers for `SIGTERM` and `SIGINT`
*   Creates a nested function `update_telem()` that runs in the thread
*   Sets thread as daemon so it doesn't prevent app exit
*   Appends thread to global `TELEM_THREADS` list for cleanup

**Update Loop**[tt_smi/tt_smi.py 556-560](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L556-L560)

**Update Execution**[tt_smi/tt_smi.py 166-176](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L166-L176)

*   Queries the telemetry table widget by ID
*   Calls `backend.update_telem()` to refresh data from hardware
*   Formats data using `format_telemetry_rows()`
*   Updates table with new rows via `update_data()`
*   Catches `NoMatches` exception when help menu obscures the table

**Termination**[tt_smi/tt_smi.py 516-525](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L516-L525)

*   Sets `INTERRUPT_RECEIVED = True` to signal thread to exit
*   Joins all threads in `TELEM_THREADS` with 0.1s timeout
*   Allows graceful shutdown without blocking

**Sources:**[tt_smi/tt_smi.py 550-574](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L550-L574)[tt_smi/tt_smi.py 166-176](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L166-L176)[tt_smi/constants.py 159](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L159-L159)

## Data Formatting Pipeline

The application layer transforms raw backend data into styled `rich.text.Text` objects for display in the TUI. Three formatting methods handle the three data tables:

### Device Information Formatting

The `format_device_info_rows()` method at [tt_smi/tt_smi.py 318-507](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L318-L507) handles complex conditional styling:

**Special Cases:**

*   **Board Type** (lines 325-360): Appends " L" (local) or " R" (remote) suffix for n300 boards
*   **PCIe Width** (lines 361-397): Compares current vs max width, applies attention style if degraded
*   **PCIe Speed** (lines 398-460): Compares current vs max speed, handles "N/A" cases
*   **DRAM Status** (lines 461-475): Shows "Y" (green) or "N" (attention) for boolean
*   **DRAM Speed** (lines 476-492): Shows speed or "N/A" in red

Each field value is wrapped in a `rich.text.Text` object with appropriate styling from `self.text_theme` (created via `create_tt_tools_theme()` from tt_tools_common).

**Sources:**[tt_smi/tt_smi.py 318-507](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L318-L507)

### Telemetry Formatting with Limits

The `format_telemetry_rows()` method at [tt_smi/tt_smi.py 196-302](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L196-L302) compares current values against limits to color-code warnings:

**Limit Comparisons:**

| Telemetry Field | Limit Source | Comparison Logic |
| --- | --- | --- |
| `voltage` | `chip_limits["vdd_max"]` | Green if < limit, attention otherwise |
| `current` | `chip_limits["tdc_limit"]` | Green if < limit, attention otherwise |
| `power` | `chip_limits["tdp_limit"]` | Green if < limit, attention otherwise |
| `aiclk` | `chip_limits["asic_fmax"]` | Green if < limit, attention otherwise |
| `asic_temperature` | `chip_limits["thm_limit"]` | Green if < limit, attention otherwise |
| `fan_speed` | Fixed 100% | Green if 0 < val <= 100, gray otherwise |

**Special Field: Heartbeat**

*   Calls `get_heartbeat_spinner()` at [tt_smi/tt_smi.py 304-316](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L304-L316)
*   Converts heartbeat seconds into rotating spinner symbols: `["●∙∙", "∙●∙", "∙∙●", "∙●∙"]`
*   Updates every ~0.5 seconds, creating animation effect

Each value is displayed with its limit: `"42.5 / 50.0"` where the current value is colored green or attention, and the limit is yellow_bold.

**Sources:**[tt_smi/tt_smi.py 196-302](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L196-L302)[tt_smi/tt_smi.py 304-316](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L304-L316)

### Firmware Formatting

The `format_firmware_rows()` method at [tt_smi/tt_smi.py 178-194](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L178-L194) is the simplest formatter:

*   Iterates through `constants.FW_LIST` fields
*   Applies gray style for "N/A" values
*   Applies text_green style for actual version strings
*   No complex comparisons or conditional logic needed

**Sources:**[tt_smi/tt_smi.py 178-194](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L178-L194)

## CSS Styling and Themes

The application uses a two-file CSS approach:

1.   **Common Styles** from tt_tools_common library: `common_style.css` (located via `importlib.resources`)
2.   **Application-specific Styles**: `tt_smi_style.css` in the tt_smi package

The CSS files are loaded at [tt_smi/tt_smi.py 80-89](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L80-L89):

The `text_theme` dictionary is created using `create_tt_tools_theme()` from tt_tools_common, providing consistent color schemes across Tenstorrent tools. Key theme styles used:

*   `yellow_bold`: Headers, limits, suffixes
*   `text_green`: Normal healthy values
*   `attention`: Warning values (degraded PCIe, over limits)
*   `red_bold`: Error values (N/A when expected)
*   `gray`: Unavailable data
*   `purple`: Log file paths

**Sources:**[tt_smi/tt_smi.py 80-89](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L80-L89)[tt_smi/tt_smi.py 109](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L109-L109)

## Initialization and Mount Sequence

The application follows a specific initialization sequence:

**Key Points:**

1.   **Backend Pre-initialization**[tt_smi/tt_smi.py 891](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L891-L891): The backend must be fully initialized before creating the TTSMI app, as it needs device data immediately

2.   **Widget Composition**[tt_smi/tt_smi.py 114-146](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L114-L146): Textual calls `compose()` to build the widget tree, but widgets are not yet mounted

3.   **Data Population**[tt_smi/tt_smi.py 148-164](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L148-L164): The `on_mount()` method is where tables are actually populated:

    *   Queries each table by ID
    *   Disables cursor: `dt.cursor_type = "none"`
    *   Formats data for that table
    *   Adds all rows at once: `dt.add_rows()`

4.   **Sidebar Control**[tt_smi/tt_smi.py 162-163](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L162-L163): Sets initial sidebar visibility based on `show_sidebar` parameter

## Snapshot Mode and Non-TTY Handling

When run with `-s` (snapshot) or `-f` (file output) flags, the application skips the TUI and directly outputs formatted data. The `snapshot` parameter is stored during initialization and checked during the quit action:

[tt_smi/tt_smi.py 518-522](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L518-L522):

However, for pure CLI snapshot operations without launching the TUI, the main entry point at [tt_smi/tt_smi.py 685-731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L685-L731) handles these cases before creating the TTSMI app:

*   **`-ls` flag**[tt_smi/tt_smi.py 700-702](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L700-L702): Calls `backend.print_all_available_devices()` and exits
*   **`-s` flag**[tt_smi/tt_smi.py 714-716](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L714-L716): Calls `backend.print_logs_to_stdout()` and exits
*   **`-f <file>` flag**[tt_smi/tt_smi.py 717-724](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L717-L724): Calls `backend.save_logs_to_file()` and exits

Only if none of these early-exit conditions match does the code create and run the TUI at [tt_smi/tt_smi.py 725-731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L725-L731)

**Sources:**[tt_smi/tt_smi.py 518-522](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L518-L522)[tt_smi/tt_smi.py 685-731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L685-L731)
