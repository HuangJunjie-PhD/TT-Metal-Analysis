## Purpose and Scope

The `Cluster` class serves as the primary API entrypoint for tt-umd, providing a unified interface for managing single or multi-chip Tenstorrent device configurations. It abstracts hardware topology, orchestrates I/O operations across heterogeneous chip types (PCIe-connected, Ethernet-connected, mock, and simulation), and provides memory management, synchronization primitives, and power control.

For information about the underlying topology representation, see [Cluster Descriptor & Topology (2.2)](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/Cluster%20Descriptor%20&%20Topology%20(2.2)) For details on individual chip abstractions, see [Chip Abstractions (2.3)](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/Chip%20Abstractions%20(2.3)) For architecture-specific device implementations, see [TTDevice Interface (2.4)](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/TTDevice%20Interface%20(2.4))

## Cluster Class Overview

The `Cluster` class manages a collection of `Chip` objects representing Tenstorrent devices. It maintains:

*   **Chip Collections**: Maps of all chips, local (MMIO-capable) chips, and remote (Ethernet-connected) chips.
*   **Topology Information**: A `ClusterDescriptor` defining chip connectivity and capabilities.
*   **Broadcast Infrastructure**: Cached Ethernet broadcast headers for efficient multi-chip writes.
*   **Configuration State**: Architecture type, firmware versions, and broadcast feature flags.

## Device I/O Operations

The `Cluster` class provides a high-level abstraction for memory and register I/O, routing requests to the appropriate chip and handling coordinate translation.

### Primary I/O Functions

*   `write_to_device`: Writes a block of data to a specific core and address. [device/cluster.cpp 1155-1174](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L1155-L1174)
*   `read_from_device`: Reads a block of data from a specific core and address. [device/cluster.cpp 1176-1192](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L1176-L1192)
*   `write_to_device_reg`: Optimized register write, often bypassing certain caches or using specific TLB windows. [device/cluster.cpp 1194-1217](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L1194-L1217)
*   `broadcast_write_to_cluster`: Performs an Ethernet-based multicast write to multiple chips simultaneously. [device/cluster.cpp 736-834](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L736-L834)

### I/O Implementation Detail (Local vs Remote)

When `write_to_device` is called, the `Cluster` retrieves the `Chip` object. If it is a `LocalChip`, it uses the `TLBManager` and `TTDevice` (PCIe) to perform the write. If it is a `RemoteChip`, the request is forwarded through a `RemoteCommunication` interface which uses Ethernet links on a gateway local chip.

**Sources:**[device/chip/local_chip.cpp 267-293](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L267-L293)[device/chip/remote_chip.cpp 76-82](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/remote_chip.cpp#L76-L82)

## Multi-Chip Orchestration

### Ethernet Broadcast

Multi-chip orchestration is primarily handled via `broadcast_write_to_cluster`. This function calculates exclusion masks for chips, rows, and columns to target specific cores across the fabric.

*   **Coordinate Translation**: It translates logical coordinates to `TRANSLATED` coordinates using the `SocDescriptor`. [device/cluster.cpp 173-177](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L173-L177)
*   **Header Caching**: To minimize overhead, the cluster caches broadcast headers in `bcast_header_cache`. [device/cluster.cpp 688-734](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L688-L734)
*   **Firmware Interaction**: It relies on Ethernet firmware versions (checked via `get_cluster_eth_fw_version`) to ensure compatibility with broadcast features. [device/cluster.cpp 154](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L154-L154)

**Sources:**[device/cluster.cpp 568-663](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L568-L663)[device/cluster.cpp 736-834](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/cluster.cpp#L736-L834)

### Refreshing Topology

The `refresh_cluster_description` method allows the cluster to re-run topology discovery without recreating `Chip` objects. This is critical for detecting changes in Ethernet link status or firmware updates in a live system.

## Lifecycle Management

### Device Start and Stop

The `Cluster` manages the lifecycle of all chips through `start_device` and `close_device`.

*   **`start_device`**:

    *   Acquires a `CHIP_IN_USE` robust mutex to prevent concurrent access from other processes. [device/chip/local_chip.cpp 147](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L147-L147)
    *   Pins or maps system memory hugepages via `SysmemManager`. [device/chip/local_chip.cpp 149](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L149-L149)
    *   Initializes PCIe iATUs (Address Translation Units) if not handled by the kernel driver. [device/chip/local_chip.cpp 150-153](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L150-L153)
    *   Resets memory barriers on the device cores. [device/chip/local_chip.cpp 154](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L154-L154)

*   **`close_device`**:

    *   Puts the device into a `LONG_IDLE` power state. [device/chip/local_chip.cpp 161](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L161-L161)
    *   Asserts RISC reset on all cores. [device/chip/local_chip.cpp 162](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L162-L162)
    *   Unmaps system memory and clears TLB mappings. [device/chip/local_chip.cpp 164-169](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L164-L169)
    *   Releases the `CHIP_IN_USE` lock. [device/chip/local_chip.cpp 172](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/chip/local_chip.cpp#L172-L172)
