This page documents instructions used to configure and control the Tensix coprocessor backend. These instructions manage the configuration state that controls how compute units (Matrix Unit, Vector Unit, Unpackers, Packers) operate, as well as synchronization primitives for coordinating execution.

For data movement instructions, see [Data Movement and DMA Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.7-packer-instructions-(pacr)). For synchronization primitives like semaphores and mutexes, see [Sync Unit](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.8-soft-reset-and-system-control). For Scalar Unit instructions that access L1, see [Scalar Unit (ThCon) and Utilities](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.8-soft-reset-and-system-control).

## Configuration System Overview

The Tensix backend maintains two primary configuration structures that control execution behavior: `Config` and `ThreadConfig`. These structures are mapped into memory-accessible regions but have different access patterns and update semantics.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 1-88](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L1-L88)

### Configuration Variables

The configuration system is defined by two global variables:

**`Config`** - Thread-agnostic configuration with two banks (`Config[0]` and `Config[1]`). Any Tensix thread can access either bank. This contains configuration for:

*   Packer settings (data format, addressing, compression)
*   Unpacker settings (tile descriptors, format conversion)
*   Global backend parameters
*   RISC-V core settings (branch predictors, prefetchers, reset PCs)

**`ThreadConfig`** - Thread-specific configuration with one bank per Tensix thread (`ThreadConfig[0]`, `ThreadConfig[1]`, `ThreadConfig[2]`). Each thread accesses its own bank indexed by `CurrentThread`. This contains:

*   Thread state ID (selects which `Config` bank to use)
*   Address modifiers for pack/unpack operations
*   Context selection for multi-context mode
*   Clock gating control per thread

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 3-20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L3-L20)

### Address Space Mapping

Both configuration structures are mapped starting at `TENSIX_CFG_BASE` in the address space of RISC-V B, T0, T1, and T2 cores. The mapping order is `Config` immediately followed by `ThreadConfig`.

**Special addressing rules:**

*   Writes to `Config[i][j]` where `j >= GLOBAL_CFGREG_BASE_ADDR32` write to **both**`Config[0][j]` and `Config[1][j]` simultaneously
*   These dual-bank fields use shorthand `Config.Field` in documentation since only one value exists
*   Writing to `Config[i][STATE_RESET_EN_ADDR32]` resets `Config[i][0..GLOBAL_CFGREG_BASE_ADDR32-1]` to zero

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 56-61](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L56-L61)

## Configuration Write Instructions

### SETC16

**Purpose:** Write a 16-bit value to a `ThreadConfig` field for the current thread.

**Syntax:**

```
SETC16(/* u16 */ Data, /* u16 */ Addr)
```

**Execution Unit:** Configuration Unit (executes in-band with other Tensix instructions)

**Behavior:**

The `Addr` parameter is an offset into the `ThreadConfig[CurrentThread]` array. Common uses include:

*   Setting address modifiers (`ADDR_MOD_PACK_SEC`, `ADDR_MOD_UNPACK_SEC`)
*   Configuring clock gating (`CG_CTRL_EN_Regblocks`, `CG_CTRL_EN_Hyst`)
*   Selecting configuration state ID (`CFG_STATE_ID_StateID`)
*   Resetting unpacker context counters (`UNPACK_MISC_CFG_CfgContextCntReset`)

**Special cases with immediate effect:**

*   Writes to clock gating fields affect clock gaters immediately
*   Writes to context reset fields reset unpacker counters immediately

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 53](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L53-L53)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 63-64](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L63-L64)

### WRCFG

**Purpose:** Write a 32-bit value to a `Config` field.

**Syntax:**

```
WRCFG(/* u32 */ Data, /* u10 */ Addr, /* u1 */ StateID)
```

**Execution Unit:** Configuration Unit

**Behavior:**

The `StateID` parameter selects which configuration bank (0 or 1) to write. The `Addr` parameter is an offset into the `Config[StateID]` array.

**Usage for writes that must be atomic with Tensix execution:** When a RISC-V core needs to update configuration that a Tensix instruction will consume, using `WRCFG` (pushed as a Tensix instruction) guarantees the configuration write completes before subsequent Tensix instructions execute. This avoids race conditions that can occur with RISC-V `sw` instructions.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L28-L28)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 39](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L39-L39)


**Execution Unit:** Configuration Unit

**Behavior:**

**Use case:** Read current configuration state for inspection or conditional logic within Tensix kernel code.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 27](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L27-L27)

### RMWCIB

**Purpose:** Atomically set or clear individual bits within a byte of `Config`.

**Syntax:**

```
RMWCIB(/* u8 */ SetMask, /* u8 */ ClearMask, /* u10 */ ByteAddr, /* u1 */ StateID)
```

**Execution Unit:** Configuration Unit

**Behavior:**

**Key feature:** Provides atomic read-modify-write for individual bits, avoiding race conditions when multiple instruction sequences need to manipulate different bits of the same configuration byte.

**Note:** Writing to `Config[i][STATE_RESET_EN_ADDR32]` via `RMWCIB` does **not** trigger the special reset behavior (unlike `WRCFG` or RISC-V `sw`).

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L28-L28)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L60-L60)[WormholeB0/TensixTile/L1.md 68](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L68-L68)

### REG2FLOP

**Purpose:** Write configuration values from Tensix GPRs to a limited subset of `Config` fields.

**Restrictions:**

*   Can only write to `Config[i][j]` where `THCON_CFGREG_BASE_ADDR32 ≤ j < GLOBAL_CFGREG_BASE_ADDR32`
*   This covers packer-related and unpacker-related configuration
*   Cannot write to global configuration fields

**Use case:** Efficiently update multiple related configuration fields from values computed in Tensix GPRs, typically for packer/unpacker address generation parameters.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L28-L28)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 55](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L55-L55)

## Configuration Read Instructions

### SETDMAREG (Special Mode)

**Purpose:** Read packer state and configuration into Tensix GPRs.

**Execution Unit:** Scalar Unit (ThCon)

**Key input sources:**

| InputSource | Data Read |
| --- | --- |
| 0 | `AccTileSize` and `LastTileSize` from all four packers |
| 1 | `AllZeroFlags` from all four packers |
| 2-5 | Full tile header structure from packer (InputSource - 2) |
| 6-7 | Exponent histogram data (16 bytes per source) |
| 8 | Compressed `AllZeroFlags` from all packers; optionally reset `AccTileSize` |
| 9 | `ExponentHistogramMaxExponent` from packer 0 |

**Functional model excerpt:**

**Use case:** Inspect packer output progress, tile metadata, or compression statistics. This is commonly used when managing L1 memory layout or when constructing tile headers for cross-tile communication.

## Synchronization and Control Instructions

### STALLWAIT

**Purpose:** Stall execution until specified condition(s) are met.

**Common conditions:**

*   **C12**: Wait for Mover to complete outstanding requests
*   **C13**: Wait for writes to `Config` (from RISC-V `sw`) to become visible
*   Backend ready conditions for each execution unit (Unpacker, Matrix, Vector, Packer)

**Interaction with configuration updates:**

**Configuration synchronization pattern:** When a RISC-V core writes to `Config` via `sw`, the write is asynchronous relative to Tensix execution. To ensure a Tensix instruction sees the updated configuration, use one of:

1.   Push `WRCFG`/`RMWCIB`/`REG2FLOP` instead of using RISC-V `sw`
2.   Push `STALLWAIT` with condition C13 after `sw` but before the consuming instruction
3.   Use RISC-V memory ordering (read-back pattern) to ensure `sw` completes before pushing the instruction

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 36-47](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L36-L47)

## Special Configuration Behaviors

### Immediate-Effect Configuration Writes

Most configuration writes are sampled by the next instruction that consumes them. However, some configuration writes have immediate effects:

| Configuration Field | Immediate Effect |
| --- | --- |
| `Config[i][STATE_RESET_EN_ADDR32]` | Zeros `Config[i][0..GLOBAL_CFGREG_BASE_ADDR32-1]` immediately (except via `RMWCIB`) |
| `Config.PRNG_SEED_Seed_Val_ADDR32` | Re-seeds all PRNGs immediately |
| `Config.RISCV_IC_INVALIDATE_InvalidateAll` | Invalidates instruction caches of specified RISC-V cores |
| `ThreadConfig[i][CG_CTRL_*]` | Affects clock gating immediately |
| `ThreadConfig[i][UNPACK_MISC_CFG_CfgContextCntReset_*]` | Resets unpacker context counters immediately |

Sources: [WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 59-64](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L59-L64)

### RISC-V Configuration Fields

Some `Config` fields affect RISC-V cores rather than backend execution:

**Branch predictor control:**

*   `Config[0].DISABLE_RISC_BP_Disable_main`
*   `Config[0].DISABLE_RISC_BP_Disable_trisc`
*   `Config[0].DISABLE_RISC_BP_Disable_ncrisc`

**Reset PC overrides:**

*   `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en`

*   `Config.NCRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en`

## Alternative Configuration Mechanisms

Not all configuration lives in `Config` or `ThreadConfig`:

| Configuration System | Access Method | Purpose |
| --- | --- | --- |
| **Replay Expanders** | `REPLAY` instruction | Configure instruction replay loops |
| **MOP Expanders** | Write-only MMIO at `TENSIX_MOP_CFG_BASE` | Configure macro-op expansion |
| **Vector Unit (SFPU)** | `SFPCONFIG` instruction | `LaneConfig` and `LoadMacroConfig` |
| **TDMA-RISC** | Write to TDMA-RISC memory region | Packer/unpacker parameters, Mover commands |
| **ADCs** | Various ADC instructions | Address counters (input/output address generation) |
| **RWCs** | Implicit in Matrix/Vector instructions | Row/column counters for register file access |
