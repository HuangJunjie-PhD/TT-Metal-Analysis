This page documents the infrastructure for collecting, processing, and reporting CI/CD pipeline results in the `tt-metal` repository. It covers the aggregation of test results into structured JSON data, failure classification logic, the generation of JUnit XML reports, Slack notifications, and the automated triage system that analyzes CI/CD pipeline data within the `infra/data_collection` subsystem for CI/CD pipeline data analysis.

* * *

## System Overview

The result collection system operates in two primary phases: **Data Production** and **Data Analysis & Reporting**. After tracked CI/CD workflows complete, a producer workflow fetches raw artifacts and logs. Subsequently, an aggregator workflow analyzes this data to detect regressions and notify developers through Slack and trigger automatic triage jobs.

### System Data Flow

This system supports broad coverage across many workflows including Sanity tests, Blackhole post-commit tests, model performance evaluations, and T3000 integration [`.github/workflows/_produce-data.yaml 3-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/_produce-data.yaml#L3-L112)[`infra/data_collection/cicd.py 31-125](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/cicd.py#L31-L125)

## Data Production and Collection Pipeline

Data collection involves the following key steps:

### 1. Workflow Triggering and Log Downloading

*   The `_produce-data.yaml` workflow is triggered upon completion (`workflow_run` event) of several priority workflows tracked by the Tenstorrent CI system [`.github/workflows/_produce-data.yaml 35-97](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/_produce-data.yaml#L35-L97)
*   The workflow uses the GitHub CLI in the script `download_cicd_logs_and_artifacts.sh` to fetch all related job logs and test report artifacts for the specific workflow run and attempt.
*   The implementation handles pagination issues in the GitHub API and gracefully retries with fallback schemes ensuring reliable log fetching [`infra/data_collection/github/download_cicd_logs_and_artifacts.sh 1-150](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/github/download_cicd_logs_and_artifacts.sh#L1-L150)

### 2. Test Report Parsing and JSON Aggregation

*   Collected logs include JUnit XML reports (from gtest and pytest) stored as artifacts.
*   The function `create_cicd_json_for_data_analysis` in `infra/data_collection/cicd.py` aggregates logs, annotations, and test data into a full structured JSON representation called a _pipeline_.
*   Inside, it calls helper functions to: 
    *   Parse JUnit XML test results.
    *   Extract job-level details like job steps, status, failure signatures.
    *   Collect hardware tool (`tt_smi`) version info and reset telemetry data.

*   The result is a comprehensive `Pipeline` python model instance loaded with: 
    *   Jobs and tests with status and timing details.
    *   Reset metadata about hardware resets tracked per-job.

*   This JSON summary is then used for offline analysis or uploaded to a centralized datastore [`infra/data_collection/cicd.py 32-126](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/cicd.py#L32-L126)[`infra/data_collection/github/workflows.py 20-200](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/github/workflows.py#L20-L200)[`infra/data_collection/pydantic_models.py 15-148](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/pydantic_models.py#L15-L148)[`infra/data_collection/github/utils.py 19-97](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/github/utils.py#L19-L97)

## Pydantic Data Models for Structured Data

The data collected from GitHub workflows and artifacts are typed strictly using Pydantic models defined in `infra/data_collection/pydantic_models.py`. These models represent the hierarchy of data for a pipeline run.

| Model Name | Description |
| --- | --- |
| `Pipeline` | Represents a CI/CD pipeline; holds metadata and associated list of Jobs. |
| `Job` | Represents a CI/CD job with start/end time, success, failure signatures, and associated tests. |
| `Test` | Represents individual tests executed sequentially within a job, with pass/fail and error info. |
| `Step` | Represents CI/CD job steps detailing actions inside a job with timestamps and outcome. |
| `TTSmiReset` | Tracks hardware tool (`tt_smi`) reset attempts inside a job with status, duration, and errors. |

The `Job` model also captures failure signature fields to classify error types, which helps in automated triage and filtering failures efficiently [`infra/data_collection/pydantic_models.py 15-193](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/pydantic_models.py#L15-L193)[`infra/data_collection/models.py 1-29](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/models.py#L1-L29)

## Failure Classification and Automated Triage

### Failure Signature Extraction

*   Failures in jobs are automatically classified using the function `get_job_failure_signature_` in `infra/data_collection/github/utils.py`.
*   This uses a dictionary mapping known error snippets to failure types enumerated in the `InfraErrorV1` enum (e.g., timeout, runner communication failure, disk space issues) [`infra/data_collection/github/utils.py 99-166](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/github/utils.py#L99-L166)[`infra/data_collection/models.py 4-18](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60infra/data_collection/models.py#L4-L18)

Example signature mapping:

| Error Snippet | Failure Signature |
| --- | --- |
| `"has timed out"` | `InfraErrorV1.JOB_UNIT_TIMEOUT_FAILURE` |
| `"lost communication with the server"` | `InfraErrorV1.RUNNER_COMM_FAILURE` |
| `"No space left on device"` | `InfraErrorV1.DISK_SPACE_FAILURE` |
| `"device timeout, potential hang detected, the device is unrecoverable"` | `InfraErrorV1.TT_TRIAGE_JOB_HANG` |

*   Additionally, the system detects setup failures, checkout failures, and clang-tidy failures by inspecting job step logs.

### Integration with Triage Workflows

*   Certain workflows include an input flag `auto-triage` (e.g., `cpp-post-commit.yaml`), when enabled, the job setup installs the triage environment.
*   Upon failure, the system triggers the `tt-auto-triage` workflow via dispatch to perform advanced debug data collection and analysis automatically [`.github/workflows/cpp-post-commit.yaml 130-141](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/cpp-post-commit.yaml#L130-L141)[`.github/workflows/ttnn-post-commit.yaml 148-150](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/ttnn-post-commit.yaml#L148-L150)

## Result Reporting and Notifications

Effective communication of testing outcomes is crucial for developer and infrastructure teams to monitor system health.

### JUnit XML Test Reports

*   C++ tests invoke Google Test (`gtest`) and configure XML output by setting environment variable:

`GTEST_OUTPUT=xml:/work/generated/test_reports/` producing JUnit-compatible XML files [`.github/workflows/cpp-post-commit.yaml 112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/cpp-post-commit.yaml#L112-L112)
*   Python test frameworks (`pytest`) produce similar XML outputs that are uploaded as artifacts [`.github/workflows/tt-train-post-commit.yaml 60-62](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/tt-train-post-commit.yaml#L60-L62)
*   The `upload-artifact-with-job-uuid` action is leveraged to collect test reports regardless of job success or failure, ensuring completeness [`.github/workflows/cpp-post-commit.yaml 151-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/cpp-post-commit.yaml#L151-L155)[`.github/workflows/build-and-unit-tests.yaml 122-126](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/build-and-unit-tests.yaml#L122-L126)

### Power and Performance Telemetry

*   The `tt_power_sidecar.py` script runs concurrent with tests collecting hardware power data, which is uploaded under `power-report` artifacts [`.github/workflows/tt-train-post-commit.yaml 128-153](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/tt-train-post-commit.yaml#L128-L153)
*   Performance profiling workflows gather LLVM profiler data and metrics for deeper analysis and performance trend monitoring [`.github/workflows/ttnn-post-commit.yaml 158-183](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/%60.github/workflows/ttnn-post-commit.yaml#L158-L183)

### Slack Notifications and Owner Mapping

*   Notification of regressions or failures is automated via Slack messages targeting assigned owners fetched from the `owners.json` mapping which correlates jobs by regexed job-name components to GitHub users.
*   This ensures timely alerting of responsible engineers for test failures and regression analysis.

# Summary

This `infra/data_collection` subsystem implements a robust, automated framework in the `tt-metal` repository for:

*   Capturing detailed pipeline test results, job logs, and hardware telemetry.
*   Parsing and aggregating this data into strict Pydantic JSON models.
*   Classifying failure types automatically.
*   Reporting in standard XML formats for test results.
*   Sending Slack notifications to responsible owners based on regression detection.
*   Triggering automatic triage workflows for in-depth incident analysis.

Together, this system forms the foundation for monitoring and improving reliability across Tenstorrent's hardware and software test pipelines.
