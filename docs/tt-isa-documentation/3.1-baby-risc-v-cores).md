## Purpose and Scope

This document describes the five "Baby" RISC-V cores present in each Tensix tile: their roles, pipeline architecture, instruction set implementation, memory access patterns, and performance characteristics. These are RV32IM cores optimized for area and power efficiency that orchestrate the Tensix coprocessor and NoC operations rather than performing heavy computation themselves.

For instruction set reference, see page 5.1. For inter-core communication mechanisms (mailboxes, PCBufs, TTSync), see page 3.3. For Ethernet tile's RISC-V E core, see page 9.1.

* * *

## Core Overview and Roles

Each Tensix tile contains five distinct RISC-V cores running at 1 GHz, each single-issue and in-order, implementing the RV32IM instruction set. The cores differ in their access capabilities and resource allocations:

**Core Comparison**

| Core | Primary Role | Local Data RAM | Instruction Cache | Instruction RAM | Tensix Access | Debug Interface |
| --- | --- | --- | --- | --- | --- | --- |
| **RISC-V B** (Brisc) | Orchestration & control | 4 KiB | 2 KiB | None | OK | Yes |
| **RISC-V T0** (Trisc 0) | UNPACK control | 2 KiB | 2 KiB | None | Great | Yes |
| **RISC-V T1** (Trisc 1) | MATH control | 2 KiB | 512 B | None | Great | Yes |
| **RISC-V T2** (Trisc 2) | PACK control | 2 KiB | 2 KiB | None | Great | Yes |
| **RISC-V NC** (Ncrisc) | NoC data movement | 4 KiB | 512 B | 16 KiB | Debug bus only | No |

In TT-Metalium kernels, the typical division of labor is:

*   **RISC-V B** and **RISC-V NC**: Data movement kernels (NoC operations)
*   **RISC-V T0**: UNPACK kernel portion
*   **RISC-V T1**: MATH kernel portion
*   **RISC-V T2**: PACK kernel portion

## Instruction Set Architecture

### RV32IM Implementation

Each Baby RISC-V core implements the full **RV32I Base Integer Instruction Set** and **M Standard Extension for Integer Multiplication and Division** (collectively RV32IM), with the following exceptions:

*   **`fence`**: Executed as `nop`; cannot enforce memory ordering
*   **`ebreak` / `ecall`**: Trigger a debug pause (equivalent to hardware breakpoint)
*   **Invalid/unsupported instructions**: Silently executed as some other instruction (usually `nop`)

All cores run at **1 GHz** and are designed to execute **one instruction per cycle** under ideal conditions (though multiply, divide, and branch instructions may take longer).

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 15-25](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L15-L25)

### Custom `.ttinsn` Extension

The cores implement a custom instruction set extension called **`.ttinsn`** that allows RISC-V cores to push Tensix coprocessor instructions. When a RISC-V core writes to specific MMIO addresses in the `INSTRN_BUF_BASE` range, the written data is interpreted as a Tensix instruction and pushed into the Tensix coprocessor's instruction buffer.

**Instruction Buffer Addresses:**

| Address Range | Access From | Destination |
| --- | --- | --- |
| `INSTRN_BUF_BASE` `0xFFE4_0000` - `0xFFE4_FFFF` | RISC-V B | T0 instruction buffer (after T0 MOP expander) |
| `INSTRN_BUF_BASE` `0xFFE4_0000` - `0xFFE4_FFFF` | RISC-V T0 | T0 instruction buffer (before T0 MOP expander) |
| `INSTRN_BUF_BASE` `0xFFE4_0000` - `0xFFE4_FFFF` | RISC-V T1 | T1 instruction buffer (before T1 MOP expander) |
| `INSTRN_BUF_BASE` `0xFFE4_0000` - `0xFFE4_FFFF` | RISC-V T2 | T2 instruction buffer (before T2 MOP expander) |
| `INSTRN1_BUF_BASE` `0xFFE5_0000` - `0xFFE5_FFFF` | RISC-V B | T1 instruction buffer (after T1 MOP expander) |
| `INSTRN2_BUF_BASE` `0xFFE6_0000` - `0xFFE6_FFFF` | RISC-V B | T2 instruction buffer (after T2 MOP expander) |

This mechanism enables RISC-V code to control the Tensix coprocessor by issuing matrix operations, vector operations, and data movement commands. See page 5 for complete Tensix instruction documentation.

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 23](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L23-L23)[WormholeB0/TensixTile/BabyRISCV/README.md 92-94](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L92-L94)

## Pipeline Architecture

All five Baby RISC-V cores share the same pipeline architecture: a four-stage in-order pipeline with speculative execution in the frontend and out-of-order completion within the Load/Store Unit.

**Pipeline Characteristics:**

*   **In-order execution**: Instructions enter and exit each stage in program order
*   **Exception**: Load/Store Unit can reorder internally (see [Memory Ordering](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Memory%20Ordering))
*   **Forwarding paths**: Results available before retirement for dependent instructions
*   **Minimum instruction latency**: 3 cycles (1 per unit)

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 27-31](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L27-L31)

### Frontend

The Frontend fetches instructions from memory and sends them to the Integer Unit at a rate of **one instruction per cycle**. It performs **branch prediction** on all control flow instructions, meaning all fetched instructions are speculative until validated by the Integer Unit.

**Instruction Sources:**

Instructions can only be fetched from:

*   **L1 scratchpad RAM** (`0x0000_0000` - `0x0016_FFFF`): Via instruction cache
*   **Core-local instruction RAM** (RISC-V NC only, `0xFFC0_0000` - `0xFFC0_FFFF`): Direct access

Each core (except NC) has a small **instruction cache** between the frontend and L1:

*   **RISC-V B, T0, T2**: 2 KiB I-cache
*   **RISC-V T1, NC**: 512 B I-cache
*   **RISC-V NC only**: Additional 16 KiB instruction RAM (not accessible by Load/Store Unit)

**Branch Prediction Penalty:**

When the Integer Unit detects a mispredicted branch, the Frontend is notified and the pipeline is flushed. This creates a **2-cycle bubble** visible to the Integer Unit (appears as 3 cycles total from external perspective).

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 33-37](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L33-L37)[WormholeB0/TensixTile/BabyRISCV/README.md 49](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L49-L49)

### Integer Unit

The Integer Unit decodes instructions, executes arithmetic/logical operations, and validates branch predictions. Instructions can enter the Integer Unit once all register operands are available (via forwarding or retirement).

**Execution Latencies:**

| Instruction Type | Integer Unit Occupancy | Notes |
| --- | --- | --- |
| Most ALU operations | 1 cycle | Result available via forwarding (latency 1, throughput 1) |
| `mul`, `mulh`, `mulhsu`, `mulhu` | 2 cycles | Blocks next instruction |
| `div`, `divu`, `rem`, `remu` | 2-33 cycles | Variable based on operand values: • Divide by 0 or 1: 2 cycles • -2³¹ ÷ -1: 2 cycles • Other cases: 6-33 cycles |
| Branches (predicted correctly) | 1 cycle |  |
| Branches (mispredicted) | 3 cycles | Includes 2-cycle pipeline bubble |

**Forwarding Path:**

Results from the Integer Unit can be forwarded directly to the next instruction, enabling back-to-back dependent integer operations with no stalls (apparent latency of 1 cycle).

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 39-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L39-L51)

### Load/Store Unit

The Load/Store Unit handles all memory access and MMIO operations. It enforces **in-order entry and exit** but allows **internal reordering** to hide latency. Up to **8 instructions** can reside in this unit simultaneously: the oldest non-retired load plus up to 7 subsequent instructions in program order.

**Memory Access Architecture:**

**Load/Store Unit Memory Access Architecture**

**Load Latencies by Memory Region:**

| Address Range | Latency (cycles) | Max Loads In-Flight |
| --- | --- | --- |
| `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_0FFF` Local data RAM | 2 | 8 |
| `MEM_L1_BASE` `0x0000_0000` - `0x0016_FFFF` L1 scratchpad RAM | ≥8 (with conflicts) | 4 |
| Mailboxes, PCBufs, TTSync, Tensix semaphores | ≥3 (more if waiting) | 4 (aggregate) |
| Tensix GPRs (`REGFILE_BASE`), Tensix backend config (`TENSIX_CFG_BASE`) | ≥4 (more with conflicts) | 4 (aggregate) |
| TDMA-RISC, Tile control/debug, PIC, NoC 0/1, NoC overlay | ≥7 (more with conflicts) | 4 (aggregate) |

**Throughput Characteristics:**

*   **Sustained loads**: 
    *   If latency < 5 cycles: 1 load per cycle
    *   If latency = N ≥ 5 cycles: 4 loads every (N-1) cycles

*   **Sustained stores to L1**: At most 1 store every 5 cycles
*   **Sustained stores to other regions**: 1 store per cycle (barring conflicts)

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 53-69](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L53-L69)

### Retire Unit

Every instruction spends exactly **1 cycle** in the Retire Unit, where results are written to the register file (GPRs). This unit is transparent from an external perspective because:

*   Results are available for forwarding as instructions move from LSU to Retire Unit (1 cycle)
*   The Retire Unit itself makes results permanently available (1 cycle)

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 71-73](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L71-L73)

## Memory Hierarchy

### Local Data RAM

Each Baby RISC-V core has private local data RAM accessible only by that core:

| Core | Size | Address Range | Typical Usage |
| --- | --- | --- | --- |
| RISC-V B | 4 KiB | `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_0FFF` | Call stack, thread-local variables |
| RISC-V T0 | 2 KiB | `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_07FF` | Call stack, thread-local variables |
| RISC-V T1 | 2 KiB | `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_07FF` | Call stack, thread-local variables |
| RISC-V T2 | 2 KiB | `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_07FF` | Call stack, thread-local variables |
| RISC-V NC | 4 KiB | `MEM_LOCAL_BASE` `0xFFB0_0000` - `0xFFB0_0FFF` | Call stack, thread-local variables |

**Performance Characteristics:**

*   **Load latency**: 2 cycles (fully hidden by 1 independent instruction)
*   **No contention**: Private to each core
*   **Comparison**: L1 loads require ≥8 cycles (7 independent instructions to hide latency)

**Recommended Usage:**

Software should place the following in local data RAM:

*   **Call stack** (highest priority)
*   **Thread-local variables**
*   **Frequently-accessed read-only globals**

### Sustained Memory Throughput

**Load Throughput Formula:**

*   If latency < 5 cycles: **1 load per cycle**
*   If latency = N ≥ 5 cycles: **4 loads per (N-1) cycles**

For L1 loads with minimum latency of 8 cycles:

*   Sustained throughput = 4 loads / 7 cycles ≈ **0.57 loads per cycle**

**Store Throughput:**

*   **L1 stores**: Maximum 1 store every 5 cycles = **0.2 stores per cycle**
*   **Other stores**: 1 store per cycle (no conflicts)

**Bandwidth to L1:**

Each RISC-V core has **6.4 bytes per cycle** (bpc) write bandwidth to L1. At 1 GHz clock frequency:

*   Per-core bandwidth: **6.4 GB/s** write
*   Aggregate (5 cores): **32 GB/s** write to L1

## Exception Handling

Baby RISC-V cores **cannot fault**. All potentially exceptional conditions are handled gracefully:

| Condition | Behavior | Classification |
| --- | --- | --- |
| Unaligned memory access | Silently rounded down to nearest aligned address | NonContractualBehavior |
| Load from unmapped memory | Usually stalls forever waiting for response | Undefined |
| Store to unmapped memory | May stall forever or be silently discarded | Undefined |
| `ebreak` / `ecall` instruction | Pauses core (equivalent to GDB breakpoint) | Debug feature |
| Invalid/unsupported instruction | Executed as some other instruction (usually `nop`) | Undefined |

**No Exceptions Means:**

*   No interrupt vectors
*   No exception handlers
*   No privilege levels (always M-mode equivalent)
*   Simplifies pipeline (no exception state to track)

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 107-114](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L107-L114)

## Memory Ordering

Instructions enter and exit the Load/Store Unit in program order, but **significant reordering can occur within the unit**. This is critical for hiding memory latency but has important implications for synchronization:

**Key Properties:**

1.   **In-order queue entry/exit**: Instructions flow through LSU in program order
2.   **Out-of-order completion**: Memory operations may complete in arbitrary order
3.   **Fence ineffective**: `fence` instruction executes as `nop`, providing no ordering guarantees
4.   **8-entry queue**: Up to 8 instructions can be in-flight in LSU simultaneously

**Implications for Synchronization:**

Software must use explicit synchronization mechanisms rather than relying on memory ordering:

*   **Mailboxes** (see page 3.3)
*   **Semaphores** at `0xFFE8_0020` - `0xFFE8_FFFF`
*   **TTSync** at `0xFFE8_0004` - `0xFFE8_001F`

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 31](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L31-L31)[WormholeB0/TensixTile/BabyRISCV/README.md 54-55](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L54-L55)

## Clock Gating and Power Management

RISC-V NC (and RISC-V E in Ethernet tiles) can have their clocks gated when idle using the PIC (Programmable Interrupt Controller). The clock will run if any of the following occurred in the previous `PIC_CLKGT.HYST` cycles:

*   `PIC_CLKGT.GATING_ENABLED` is `false` (default)
*   `PIC_CLKGT.FORCE_ACTIVE` is `true`
*   Any IRQ is pending
*   Any PIC register was accessed

**Clock Gating Registers:**

| Register | Address | Purpose |
| --- | --- | --- |
| `PIC_CLKGT_EN` | `0xFFB1_300C` | Enable gating, force active, read active status |
| `PIC_CLKGT_HYST` | `0xFFB1_3010` | Hysteresis: cycles to remain active after event |

**Typical Usage Pattern:**

1.   Configure `PIC_CLKGT.HYST` (default 3 cycles)
2.   Set `PIC_CLKGT.GATING_ENABLED = true`
3.   Before clearing IRQs: `PIC_CLKGT.FORCE_ACTIVE = true`
4.   After servicing IRQs: `PIC_CLKGT.FORCE_ACTIVE = false`

This mechanism allows RISC-V NC to sleep when waiting for NoC operations or stream activity, reducing power consumption during idle periods.

**Sources:**[WormholeB0/TensixTile/PIC.md 27-40](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/PIC.md?plain=1#L27-L40)[WormholeB0/TensixTile/PIC.md 73-89](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/PIC.md?plain=1#L73-L89)
