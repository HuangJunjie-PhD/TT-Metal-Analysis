The TTNN Operations Library is the high-level functional interface for Tenstorrent hardware, providing a PyTorch-like experience for tensor manipulation and computation. It abstracts the underlying hardware complexities while allowing fine-grained control over memory layouts, data types, and multi-device execution through a Single-Program Multiple-Device (SPMD) model.

## Core Architecture and Data Flow

TTNN operates by dispatching individual operation kernels to the hardware. While each op call typically corresponds to a single kernel launch, developers can use [TT-Lang](https://deepwiki.com/tenstorrent/tt-forge/8.1-tt-lang-kernel-dsl) for manual kernel fusion [skills/ttnn/SKILL.md 97-100](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L97-L100)

### Data Flow: Host to Device

The standard workflow involves moving data from PyTorch/Host space into the TTNN device environment. Tensors must be explicitly moved to the device before computation can occur [skills/ttnn/SKILL.md 103](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L103-L103)

## Device and Memory Management

### Device Abstractions

TTNN supports two primary device abstractions:

1.   **Single Device**: Managed via `ttnn.open_device` and `ttnn.close_device`[skills/ttnn/api.rst 12-13](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/api.rst#L12-L13)
2.   **MeshDevice**: A virtual abstraction representing a logical 2D mesh of physical chips (e.g., N300 1x2, Galaxy 8x4) [skills/ttnn/multi_device.md 55-58](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/multi_device.md?plain=1#L55-L58)

### Memory Configurations

Tensors can be stored in different memory regions:

*   **DRAM**: Large capacity, default storage [skills/ttnn/SKILL.md 105](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L105-L105)
*   **L1**: High-speed, on-chip SRAM (~1.5MB per core). Critical for performance-sensitive buffers [skills/ttnn/SKILL.md 105](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L105-L105)

### Layouts and Data Types

TTNN requires specific layouts for hardware acceleration:

*   **`ttnn.ROW_MAJOR_LAYOUT`**: Standard contiguous row storage [skills/ttnn/tensor.rst 40-42](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor.rst#L40-L42)
*   **`ttnn.TILE_LAYOUT`**: Data organized into 32x32 tiles. This is required for most high-performance compute ops like `ttnn.matmul`[skills/ttnn/tensor.rst 60-64](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor.rst#L60-L64)

**Supported Data Types**[skills/ttnn/tensor.rst 117-124](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor.rst#L117-L124):

| Data Type | Width Multiple (Row Major) | Description |
| --- | --- | --- |
| `ttnn.bfloat16` | 2 | Standard training/inference type |
| `ttnn.bfloat8_b` | 32 | Compressed 8-bit format (requires TILE_LAYOUT) |
| `ttnn.float32` | 1 | High precision |
| `ttnn.uint32` | 1 | Unsigned integer |

## Tensor Sharding Strategies

Sharding distributes a single tensor across multiple L1 memory banks (cores) to enable parallel processing and minimize data movement [skills/ttnn/tensor_sharding.md 14-18](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor_sharding.md?plain=1#L14-L18)

### 2D Sharding Types

| Strategy | Description | Best Use Case |
| --- | --- | --- |
| **Height Sharding** | Divides tensor along the H dimension; each core gets full width [skills/ttnn/tensor_sharding.md 23-25](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor_sharding.md?plain=1#L23-L25) | Row-wise operations |
| **Width Sharding** | Divides tensor along the W dimension; each core gets full height [skills/ttnn/tensor_sharding.md 88-89](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/tensor_sharding.md?plain=1#L88-L89) | Column-wise operations |

## Multi-Device Mesh (SPMD)

TTNN employs Single-Program Multiple-Device (SPMD) execution. The `MeshDevice` coordinates command queues to distribute work across all physical devices in the mesh simultaneously [skills/ttnn/multi_device.md 44-47](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/multi_device.md?plain=1#L44-L47)

### Data Distribution Mappers

When moving a tensor to a `MeshDevice`, a `mesh_mapper` defines the distribution [skills/ttnn/SKILL.md 30-37](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L30-L37):

*   **`ttnn.ReplicateTensorToMesh`**: Every device receives a full copy of the tensor.
*   **`ttnn.ShardTensorToMesh`**: The tensor is split along a specified dimension across the devices.

### Collective Communication (CCL)

Ops for inter-device synchronization [skills/ttnn/SKILL.md 42-46](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L42-L46):

*   **`ttnn.all_gather`**: Collect shards from all devices to form a full tensor on each.
*   **`ttnn.all_reduce`**: Sum/Reduce data across all devices.
*   **`ttnn.reduce_scatter`**: Reduce data and then distribute results as shards.

## PyTorch Model Conversion

The recommended path for porting models to TTNN follows a three-step process to ensure functional correctness and performance [skills/ttnn/converting_torch_model_to_ttnn.rst 12-16](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/converting_torch_model_to_ttnn.rst#L12-L16)

### Conversion Workflow

1.   **Functional Rewrite**: Rewrite the `torch.nn.Module` using functional `torch` APIs. This separates weights (parameters) from the compute logic [skills/ttnn/converting_torch_model_to_ttnn.rst 17-20](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/converting_torch_model_to_ttnn.rst#L17-L20)
2.   **TTNN Translation**: Replace functional torch calls (e.g., `torch.nn.functional.gelu`) with TTNN equivalents (`ttnn.gelu`). Use `ttnn.from_torch` to move parameters to the device [skills/ttnn/converting_torch_model_to_ttnn.rst 99-102](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/converting_torch_model_to_ttnn.rst#L99-L102)
3.   **Optimization**: Introduce specialized memory configs (L1), sharding, and lower-precision data types like `bfloat8_b`[skills/ttnn/converting_torch_model_to_ttnn.rst 160-163](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/converting_torch_model_to_ttnn.rst#L160-L163)

### Pre-allocation and Tracing

To eliminate host overhead in hot loops, TTNN supports **Tracing**. This requires pre-allocating tensors using the `output_tensor` parameter in operations to ensure memory addresses remain static during trace replay [skills/ttnn/SKILL.md 79-84](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/skills/ttnn/SKILL.md?plain=1#L79-L84)
