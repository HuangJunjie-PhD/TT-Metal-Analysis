## Overview

The packaging and release system transforms a built toolchain into distributable artifacts suitable for various Linux distributions. The primary entry point is `scripts/release.sh`, which creates tarball, Debian, and RPM packages from the built toolchain in the `build` directory. The release process includes:

*   Source integrity verification to ensure packages match the built source tree
*   Inclusion of SFPI header files from the `include` directory
*   Binary stripping to reduce package size
*   Automatic dependency detection from shared library requirements
*   Platform-specific package creation with proper metadata
*   Checksum generation for distribution verification

## Package Creation Process

The release system creates three types of distribution artifacts: tarball (.txz), Debian package (.deb), and RPM package (.rpm). Each package type has specific characteristics and use cases.

### Tarball Package (.txz)

The tarball is the universal distribution format, created unconditionally for all platforms. It contains the complete toolchain hierarchy compressed with xz:

**Creation command**: `tar cJf build/release/sfpi_VERSION_ARCH_DIST.txz -C build sfpi`

The tarball includes:

*   `compiler/` - Complete RISC-V toolchain binaries
*   `include/` - SFPI header files copied from the repository
*   `README.txt` - Source hashes and version information

All executable ELF binaries are stripped with `strip -g` to remove debug symbols, reducing package size while preserving necessary symbol information for linking.

Sources: [scripts/release.sh 48-60](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L48-L60)

### Debian Package (.deb)

Debian packages are created on systems where `dpkg-query` is available and the distribution is detected as Debian-based (debian, ubuntu, etc.). The package uses FPM (Effing Package Management) for creation:

**Base FPM options**:

```
-s dir                          # Source type: directory
-t deb                          # Target type: Debian
-n sfpi                         # Package name
-v VERSION                      # Version (underscores converted to hyphens)
-C build/sfpi                   # Source directory
--prefix /opt/tenstorrent/sfpi  # Installation prefix
--architecture native           # Use build host architecture
```

**Metadata** (when `--tt-built`):

*   License: "Apache 2.0/GPL 2,3/LGPL 2.1"
*   URL: [https://github.com/tenstorrent/sfpi](https://github.com/tenstorrent/sfpi)
*   Maintainer: Tenstorrent [support@tenstorrent.com](mailto:support@tenstorrent.com)
*   Vendor: Tenstorrent

Sources: [scripts/release.sh 92-143](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L92-L143)

### RPM Package (.rpm)

RPM packages are created on systems where `rpm` is available and the distribution is detected as Fedora-based (fedora, rhel, centos, almalinux, etc.):

**Additional RPM-specific options**:

```
--rpm-auto-add-directories      # Automatically create parent directories
--rpm-rpmbuild-define "_build_id_links none"  # Disable build-id links
```

Version strings with hyphens are converted to underscores for RPM compatibility (hyphens are forbidden in RPM version strings).

### Shared Library Mapping Table

The script maintains a hardcoded mapping from shared object names to package names in both Debian and Fedora ecosystems:

| Shared Object | Debian Package | Fedora Package | Purpose |
| --- | --- | --- | --- |
| `libc.so*` | `libc6` | `glibc` | C standard library |
| `libexpat.so*` | `libexpat1` | `expat` | XML parsing |
| `libgmp.so*` | `libgmp10` | `gmp` | Arbitrary precision arithmetic |
| `libisl.so*` | `libisl23` | `isl` | Integer set library |
| `liblzma.so*` | `liblzma5` | `xz-libs` | XZ compression |
| `libmpc.so*` | `libmpc3` | `libmpc` | Multiple precision complex |
| `libmpfr.so*` | `libmpfr6` | `mpfr` | Multiple precision floating-point |
| `libncursesw.so*` | `libncursesw6` | `ncurses-libs` | Terminal UI library |
| `libstdc++.so*` | `libstdc++6` | `libstdc++` | C++ standard library |
| `libtinfo.so*` | `libtinfo6` | `ncurses-libs` | Terminal info library |
| `libz.so*` | `zlib1g` | `zlib-ng-compat` | Compression library |
| `libzstd.so*` | `libzstd1` | `libzstd` | Zstandard compression |

Unmapped shared objects trigger a warning but do not halt the release process.

### Version Constraint Generation

The script queries the installed version of each dependency package and generates minimum version constraints. This ensures that users installing the SFPI package will have compatible library versions:

**Debian query**: `dpkg-query -f '${Version}' -W package_name`

*   Removes epoch prefix (e.g., `1:2.35` → `2.35`) via regex matching
*   Generates: `--depends "package >= version"`

**Fedora query**: `rpm -q --qf '%{VERSION}' package_name`

*   Direct version string extraction
*   Generates: `--depends "package >= version"`

If version detection fails for a package, a warning is emitted but processing continues.

Sources: [scripts/release.sh 106-137](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/release.sh#L106-L137)

## Platform Detection and Metadata

The `scripts/sfpi-info.sh` script serves as the "single source of truth" for platform detection, version formatting, and metadata generation. It is invoked by both `release.sh` and user-facing CMake integration.

### Platform Variables

The script evaluates and exports the following variables:

| Variable | Description | Example |
| --- | --- | --- |
| `sfpi_version` | Release version string | `1.2.3` |
| `sfpi_arch` | Host architecture from `uname -m` | `x86_64`, `aarch64` |
| `sfpi_dist` | Distribution identifier | `debian`, `fedora`, `ubuntu` |
| `sfpi_pkg` | Package format for distribution | `deb`, `rpm`, `txz` |
| `sfpi_filename` | Base filename for artifacts | `sfpi_1.2.3_x86_64_debian` |
| `sfpi_url` | Download URL base | `https://github.com/tenstorrent/sfpi/releases/download/1.2.3` |
| `sfpi_hashtype` | Hash algorithm used | `sha256` |
| `sfpi_hash` | Expected hash for package | `abc123...` |

## Release Checklist

To create an official release:

1.   **Update version** (if not already on a tagged release):

    *   Commit version changes to `version` file or create git tag
    *   Ensure main branch for stable releases

2.   **Build toolchain**:

 
3.   **Run tests**:

 
4.   **Create packages**:

 
5.   **Verify artifacts** in `build/release/`:

    *   Check file sizes are reasonable
    *   Verify checksums: `sha256sum -c *.sha256`
    *   Test installation of .deb or .rpm on target systems

6.   **Create GitHub Release**:

    *   Create tag matching version string
    *   Upload all files from `build/release/`
    *   Document changes in release notes

7.   **Update tt-metal integration**:

    *   Update version in `tt_metal/sfpi-version.sh`
    *   Add hash entries for new packages
    *   Test integration build
