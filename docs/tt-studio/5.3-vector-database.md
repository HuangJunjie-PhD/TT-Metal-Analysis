## Purpose and Scope

The Vector Database system in TT-Studio provides storage and retrieval capabilities for document embeddings, enabling Retrieval-Augmented Generation (RAG) functionality. Built on **ChromaDB**, the system provides a comprehensive infrastructure for document ingestion, text chunking, embedding generation, and semantic search. It includes a dedicated administrative interface for managing collections across different user sessions.

## Architecture Overview

The Vector Database system integrates a dedicated ChromaDB container with the Django backend. The frontend interacts with these services through a specialized API layer to manage document lifecycles and query context.

## Core Components

### ChromaDB Service

The system utilizes the `chromadb/chroma:0.5.3` Docker image, configured as a persistent service named `tt_studio_chroma`.

*   **Port**: 8111 [app/docker-compose.yml 136](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L136-L136)
*   **Persistence**: Data is stored in the host volume `${HOST_PERSISTENT_STORAGE_VOLUME}/chroma` mapped to `/chroma/chroma` in the container [app/docker-compose.yml 127](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L127-L127)
*   **Environment**: `IS_PERSISTENT=TRUE` ensures data remains across restarts [app/docker-compose.yml 130](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L130-L130)

### Document Processing and Ingestion

Document processing is handled in the backend, primarily focusing on PDF and text extraction.

1.   **Extraction**: PDF text is extracted using `pypdf.PdfReader`.
2.   **Chunking**: Text is split using LangChain's `RecursiveCharacterTextSplitter` to maintain context window limits [app/api/vector_db_control/documents.py 9-25](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/api/vector_db_control/documents.py#L9-L25)
3.   **Embedding**: Chunks are transformed into vectors using the `all-MiniLM-L6-v2` model [app/api/api/settings.py 153](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/api/api/settings.py#L153-L153)
4.   **Batching**: To prevent memory overflow, documents are inserted into ChromaDB in batches of 166 [app/api/vector_db_control/chroma.py 84-105](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/api/vector_db_control/chroma.py#L84-L105)

## Frontend Management Interfaces

### RAG Management

The `RagManagement` component provides the primary user interface for managing personal or session-based document collections.

*   **Session Tracking**: Uses a `X-Browser-ID` header (stored in `localStorage` as `tt_studio_browser_id`) to isolate collections between different users/browsers [app/frontend/src/components/rag/RagManagement.tsx 64-97](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/rag/RagManagement.tsx#L64-L97)
*   **Operations**: Supports creating collections, deleting collections, and uploading documents via `GentleFileUpload`[app/frontend/src/components/rag/RagManagement.tsx 17-31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/rag/RagManagement.tsx#L17-L31)

### RAG Admin Interface

The `RagAdmin` component allows administrators to oversee all collections stored in the database, regardless of the originating session.

*   **Authentication**: Requires a password defined by `RAG_ADMIN_PASSWORD` in the environment [app/docker-compose.yml 45](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L45-L45)
*   **Capabilities**: 
    *   List all global collections with metadata.
    *   Identify user types (Authenticated vs Anonymous/Browser Session) [app/frontend/src/components/rag/RagAdmin.tsx 196-204](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/rag/RagAdmin.tsx#L196-L204)

## Integration with Inference

When a user interacts with the Chat UI, the system performs the following to achieve RAG:

1.   **Retrieval**: The backend queries the selected ChromaDB collection using the user's prompt as the search vector.
2.   **Augmentation**: The top-k relevant text chunks are retrieved and prepended to the LLM prompt as "Context".
3.   **Generation**: The inference server (FastAPI) receives the augmented prompt and generates a response based on the provided documents.
4.   **Stats**: Performance metrics including `tokens_prefilled` (reflecting context size) and `user_ttft_s` are tracked and displayed in the `InferenceStats` component [app/frontend/src/components/chatui/InferenceStats.tsx 183-220](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/chatui/InferenceStats.tsx#L183-L220)
