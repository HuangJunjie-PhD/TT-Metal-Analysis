## Purpose and Scope

This page documents the complete syntax and usage of RiescueD test directives. These directives are special comment-based annotations that users embed in their assembly test files (`.s` or `.rasm` files) to control test generation, memory management, page table creation, and environment configuration.

For information about RiescueD's command-line interface, see [RiescueD Command-Line Interface](https://deepwiki.com/tenstorrent/riescue/2.1.1-riescued-command-line-interface). For details on RiescueD's internal architecture and how these directives are processed, see [RiescueD Internal Architecture](https://deepwiki.com/tenstorrent/riescue/2.1.3-riescued-internal-architecture).

## Directive Overview

RiescueD directives are written as special comments beginning with `;#` in assembly files. The `Parser` class in [riescue/dtest_framework/parser.py 20-698](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L20-L698) reads the test file line-by-line and identifies these directives using regex pattern matching.

**Directive Parsing Flow**

Sources: [riescue/dtest_framework/parser.py 20-698](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L20-L698)

### Directive Categories

| Category | Directives | Purpose |
| --- | --- | --- |
| Test Metadata | `;#test.*` | Define test properties and environment constraints |
| Data Generation | `;#random_data()` | Generate random data values |
| Address Management | `;#random_addr()`, `;#reserve_memory()` | Manage memory addresses and reservations |
| Page Tables | `;#page_mapping()`, `;#page_map()` | Configure virtual memory page table entries |
| Test Structure | `;#discrete_test()` | Define independent test cases within a file |
| Memory Init | `;#init_memory` | Mark addresses for initialization |
| Interrupts | `;#vectored_interrupt()` | Configure vectored interrupt handlers |
| CSR Access | `;#csr_rw()` | Generate CSR read/write/set/clear code |
| PTE Access | `;#read_leaf_pte()` | Generate leaf PTE read code |

Sources: [riescue/dtest_framework/parser.py 25-38](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L25-L38)

## Test Header Directives

Test header directives define metadata and environment constraints for the test. All test headers use the `;#test.<property>` syntax and are parsed by `parse_test_header()` at [riescue/dtest_framework/parser.py 590-598](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L590-L598)

### Syntax

`;#test.<property>    <value>`
### Available Properties

| Property | Type | Description | Example Values |
| --- | --- | --- | --- |
| `name` | string | Test name identifier | `sample_test` |
| `author` | string | Test author email | `name@email.com` |
| `arch` | string | Architecture specification | `rv64`, `rv32` |
| `priv` | list | Privilege mode constraints | `machine super user any` |
| `env` | list | Test environment modes | `virtualized bare_metal` |
| `cpus` | int | Number of CPUs/harts | `1`, `2`, `4` |
| `paging` | list | Paging mode constraints | `sv39 sv48 sv57 disable any` |
| `category` | string | Test category | `arch`, `system` |
| `class` | string | Test class | `vector`, `integer` |
| `features` | list | Feature enable/disable | `ext_v.enable ext_fp.disable` |
| `tags` | list | Test tags for filtering | `vectors vector_ld_st` |
| `secure_mode` | string | Secure mode specification | `secure non_secure` |

## Random Data Directives

The `;#random_data()` directive generates random data values with configurable masking. Parsed by `parse_random_data()` at [riescue/dtest_framework/parser.py 116-130](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L116-L130)

### Syntax

`;#random_data(name=<identifier>, type=<data_type>, and_mask=<mask>, or_mask=<mask>)`
### Parameters

| Parameter | Required | Type | Description | Default |
| --- | --- | --- | --- | --- |
| `name` | Yes | string | Identifier for the random data | - |
| `type` | Yes | string | Data type specification | - |
| `and_mask` | No | hex/int | Bitwise AND mask applied to random value | `0xFFFFFFFFFFFFF000` |
| `or_mask` | No | hex/int | Bitwise OR mask applied to random value | `0x0` |

### Example

`;#random_data(name=rdata1, type=32bits, and_mask=0xfffff000, or_mask=0x00000001)`
This generates a 32-bit random value, masks bits [11:0] to zero, and sets bit 0 to 1.

### Parsed Structure

`@dataclassclass ParsedRandomData:    name: str = ""    type: str = ""    and_mask: int = 0xFFFFFFFFFFFFF000    or_mask: int = 0x0`
Sources: [riescue/dtest_framework/parser.py 116-130](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L116-L130)[riescue/dtest_framework/parser.py 701-707](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L701-L707)

#### Parameters

| Parameter | Required | Type | Description | Default |
| --- | --- | --- | --- | --- |
| `name` | Yes | string | Address identifier | - |
| `type` | Yes | string | `linear` or `physical`, optionally with bit width (e.g., `linear32`) | - |

### Page Mapping Directive

The `;#page_mapping()` directive creates a virtual-to-physical memory mapping. This is the most complex directive with extensive support for page table entry attributes. Parsed by `parse_page_mappings()` at [riescue/dtest_framework/parser.py 168-572](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L168-L572)

#### Syntax

`;#page_mapping(lin_name=<linear_id>, phys_name=<physical_id>, v=<0|1>, r=<0|1>, w=<0|1>, x=<0|1>, ...)`
#### Basic Parameters

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `lin_name` | string | Linear (virtual) address name | Required |
| `phys_name` | string | Physical address name | Required |
| `v` | bool | Valid bit | `1` |
| `r` | bool | Read permission | `1` |
| `w` | bool | Write permission | `0` |
| `x` | bool | Execute permission | `1` |
| `u` | bool | User mode accessible | - |
| `g` | bool | Global mapping | `0` |
| `a` | bool | Accessed bit | - |
| `d` | bool | Dirty bit | - |

#### Advanced Parameters

The directive supports fine-grained control over page table entries at different levels:

**Leaf vs Non-Leaf PTEs:**

*   `v_leaf`, `v_nonleaf` - Valid bit for leaf/non-leaf entries
*   `r_leaf_gleaf`, `r_leaf_gnonleaf` - Read permission for leaf entries with guest-stage variations
*   `w_nonleaf_gleaf` - Write permission for non-leaf with guest-leaf
*   Similar patterns for `x`, `u`, `a`, `d`, `g` bits

**Page Size Control:**

`pagesize=[4kb, 2mb, 1gb, 512gb, 256tb, any]gstage_vs_leaf_pagesize=[4kb, 2mb]gstage_vs_nonleaf_pagesize=[4kb, 2mb]`
**Security and PMA:**

*   `secure=<0|1>` - Mark as secure memory
*   `pbmt=<value>` - Page-based memory type
*   `n=<0|1>` - No-translate bit
*   `rsw=<value>` - Reserved for supervisor software

Sources: [riescue/dtest_framework/parser.py 168-572](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L168-L572)[riescue/dtest_framework/parser.py 736-845](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L736-L845)

#### Example

`;#random_addr(name=lin1, type=linear, size=0x1000);#random_addr(name=phys1, type=physical, size=0x1000);#page_mapping(lin_name=lin1, phys_name=phys1, v=1, r=1, w=1, x=0, u=1, pagesize=[4kb, 2mb])`
#### Parsed Structure

The `ParsedPageMapping` dataclass at [riescue/dtest_framework/parser.py 736-845](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L736-L845) stores over 80 fields to represent all possible PTE configurations, including:

## Memory Initialization Directive

The `;#init_memory` directive marks an address for memory initialization. This is parsed by `parse_init_mem()` at [riescue/dtest_framework/parser.py 625-640](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L625-L640)

### Syntax

`;#init_memory @<addr_name>;#init_memory @<addr_name>:<map1>,<map2>,...`
The second form is used when an address is mapped through multiple page maps.

### Example

`;#random_addr(name=data_addr, type=linear, size=0x1000);#init_memory @data_addr ;# With multiple maps;#init_memory @data_addr:map1,map2`
