## Testing Framework Overview

The tt-torch test suite uses a structured framework built around the `ModelTester` base class and the `ModelLoader` pattern from the `tt-forge-models` repository. Tests are automatically discovered, configured via `test_config.py`, and executed across multiple hardware architectures.

### ModelTester Class Hierarchy

Sources: [tests/runner/test_utils.py 152-168](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L152-L168)[tests/utils.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/utils.py) (referenced), [docs/src/adding_models.md 12-32](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/docs/src/adding_models.md?plain=1#L12-L32)

The `ModelTester` class requires derived classes to implement:

*   **`_load_model()`**: Returns the PyTorch model to test
*   **`_load_inputs()`**: Returns input data (torch tensors or dictionaries)
*   **`_extract_outputs()`**: (Optional) Extracts outputs when default extraction fails

The `DynamicTester` class (used by the unified test runner) automatically handles `dtype_override` parameters if the `ModelLoader` supports them, providing seamless integration with tt-forge-models.

Models in tt-torch follow the `ModelLoader` pattern from the `tt-forge-models` repository. This pattern standardizes model loading, input generation, and metadata handling.

The `ModelInfo` class provides metadata about a model:

*   **`name`**: Unique identifier for the model (used in test configuration)
*   **`group`**: Model category (e.g., "nlp", "vision")
*   **`variants`**: Available model variants (e.g., different sizes)

### CompilerConfig Options

The `CompilerConfig` class controls compilation behavior:

| Option | Description | Common Values |
| --- | --- | --- |
| `enable_consteval` | Enable constant evaluation | `True` (recommended) |
| `consteval_parameters` | Include parameters in consteval | `True` (recommended) |
| `compile_depth` | Compilation depth level | `CompileDepth.EXECUTE` (default), `CompileDepth.EXECUTE_OP_BY_OP` |
| `op_by_op_backend` | Backend for op-by-op mode | `OpByOpBackend.TORCH`, `OpByOpBackend.STABLEHLO` |

### Configuration Resolution Flow

### ModelStatus Enumeration

Tests are classified using the `ModelStatus` enum [tests/runner/test_utils.py 15-23](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L15-L23):

| Status | Description | Pytest Marker |
| --- | --- | --- |
| `EXPECTED_PASSING` | Test should pass | `expected_passing` |
| `KNOWN_FAILURE_XFAIL` | Known failure, expected to fail | `known_failure_xfail` + `xfail` |
| `NOT_SUPPORTED_SKIP` | Not supported on architecture | `not_supported_skip` |
| `UNSPECIFIED` | New model, not yet triaged | `unspecified` |

### Workflow Descriptions

| Workflow File | Purpose | Execution Frequency | Test Depth |
| --- | --- | --- | --- |
| `run-full-model-execution-tests-nightly.yml` | Models that pass end-to-end | Nightly | `EXECUTE` (full compilation) |
| `run-op-by-op-flow-tests-weekly.yml` | Models with all ops passing | Weekly | `EXECUTE_OP_BY_OP` |
| `run-op-by-op-flow-tests-nightly.yml` | Models with compile/runtime failures | Nightly | `EXECUTE_OP_BY_OP` |
| `run-e2e-compile-tests.yml` | Models that compile but fail at runtime | Nightly | `TTNN_IR` (compile only) |

## Step-by-Step Guide

### 1. Create Model in tt-forge-models

First, create your model implementation in the `tt-forge-models` repository:

**Directory structure**:

```
third_party/tt_forge_models/
└── my_model_name/
    └── pytorch/
        └── loader.py
```

*   `ModelLoader` class with `load_model()`, `load_inputs()`, and `get_model_info()` methods

## Advanced Topics

### Writing Standalone Tests

While the unified test runner is recommended, you can also write standalone test files for models that require custom handling [tests/models/codegen/test_codegen.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/models/codegen/test_codegen.py)[tests/models/opt/test_opt.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/models/opt/test_opt.py):

1.   Create `tests/models/my_model/test_my_model.py`
2.   Define a custom tester class inheriting from `ModelTester`
3.   Implement `_load_model()` and `_load_inputs()`
4.   Write test function with pytest parametrization

This approach gives more control but doesn't integrate with the automated test discovery system.

### Loading Files from LFS

## Summary

Adding a new model to tt-torch involves:

1.   **Implement ModelLoader** in tt-forge-models repository
2.   **Add test configuration** to `test_config.py`
3.   **Run test locally** to verify functionality
4.   **Update status** based on test results
5.   **Add to CI/CD** workflows based on status
6.   **Configure architecture overrides** if needed

The unified test runner handles test discovery, configuration resolution, and execution automatically, minimizing boilerplate code while maintaining flexibility for custom requirements.
