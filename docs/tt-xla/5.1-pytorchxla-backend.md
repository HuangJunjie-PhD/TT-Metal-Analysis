For detailed coverage of the `torch.compile` graph transformation passes, `XLAExecutor`, and SPMD handling, see [torch.compile Backend Integration](https://deepwiki.com/tenstorrent/tt-xla/5.1.1-graph-transformation-pipeline). ## Integration Overview

The PyTorch/XLA backend connects `torch.compile` to Tenstorrent hardware through two cooperating layers:

1.   **`TTPlugin`** ([`python_package/torch_plugin_tt/__init__.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/torch_plugin_tt/__init__.py%60)) — a `DevicePlugin` subclass that registers the `pjrt_plugin_tt.so` binary with `torch_xla`, configures required environment variables, and is discovered at import time via the `torch_xla.plugins` entry point.

2.   **`xla_backend`** ([`python_package/tt_torch/backend/backend.py 254-269](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/backend/backend.py#L254-L269)) — registered as the `"tt"` backend for `torch.compile`. It accepts a `torch.fx.GraphModule`, applies a series of graph transformation passes, and returns an `XLAExecutor` that drives compilation and execution through torch-xla.

## TTPlugin — Device Plugin

`TTPlugin` in [`python_package/torch_plugin_tt/__init__.py 19-45](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/torch_plugin_tt/__init__.py#L19-L45) extends `torch_xla.experimental.plugins.DevicePlugin`. torch-xla discovers it at import via the `torch_xla.plugins` entry point declared in the wheel's `setup.py`.

### Initialization sequence

When `TTPlugin.__init__()` runs:

| Step | Call | Effect |
| --- | --- | --- |
| 1 | `setup_tt_pjrt_plugin_dir()` | Sets `TT_PJRT_PLUGIN_DIR` to the directory containing `pjrt_plugin_tt.so` |
| 2 | `setup_tt_metal_home()` | Sets `TT_METAL_RUNTIME_ROOT` to the tt-metal installation path |
| 3 | `os.environ["XLA_STABLEHLO_COMPILE"] = "1"` | Tells torch-xla to use the StableHLO compilation path instead of HLO |
| 4 | `_set_xla_all_numbers_special_scalars(True)` | Configures XLA scalar handling |

`library_path()` returns the resolved path to `pjrt_plugin_tt.so` via `get_library_path()`.

### Module-level configuration

At import time, [`python_package/torch_plugin_tt/__init__.py 52](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/torch_plugin_tt/__init__.py#L52-L52) sets:

```
torch._dynamo.config.inline_inbuilt_nn_modules = False
```

This keeps model parameters bound to the `GraphModule`'s `state_dict` rather than treating them all as graph inputs. The `torch_pass_pipeline` depends on this to distinguish parameters from user inputs when inserting argument type markers.

### Environment and path helpers

Both path setup functions live in [`python_package/pjrt_plugin_tt/__init__.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/pjrt_plugin_tt/__init__.py%60):

*   **`setup_tt_pjrt_plugin_dir()`** — checks for a user-set `TT_PJRT_PLUGIN_DIR`; if absent, resolves the directory relative to the installed package.
*   **`setup_tt_metal_home()`** — checks `TT_METAL_RUNTIME_ROOT`; searches first in the wheel package (`pjrt_plugin_tt/tt-metal`), then in the source tree (`third_party/tt-mlir/.../tt-metal`).
*   **`get_library_path()`** — returns the `Path` to `pjrt_plugin_tt.so`; raises `FileNotFoundError` if absent.

Sources: [`python_package/torch_plugin_tt/__init__.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/torch_plugin_tt/__init__.py%60)[`python_package/pjrt_plugin_tt/__init__.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/pjrt_plugin_tt/__init__.py%60)

### xla_backend entry point

`xla_backend` ([`python_package/tt_torch/backend/backend.py 254-269](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/backend/backend.py#L254-L269)) is registered with `@register_backend(name="tt")`. When `torch.compile(backend='tt')` is called, dynamo invokes it with the traced `GraphModule`, example inputs, and an optional `options` dictionary.

**Recognized options:**

| Option key | Type | Effect |
| --- | --- | --- |
| `tt_enable_torch_fx_fusion_pass` | `bool` (default `True`) | Enables pattern-based fusion passes via `FusionProvider` |
| `tt_enable_composite_ops` | `bool` (default `True`) | Enables replacement of torch ops with StableHLO composite ops |
| `tt_legacy_compile` | `bool` (default `False`) | Falls back to the older graph execution path |
| `tt_experimental_compile` | `bool` | Deprecated alias; inverse of `tt_legacy_compile` |

### XLAExecutor

`XLAExecutor` ([`python_package/tt_torch/backend/backend.py 119-251](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/backend/backend.py#L119-L251)) holds the compiled graph module and its export signature. When called, it chooses between two execution paths:

| Path | Condition | Mechanism |
| --- | --- | --- |
| Experimental (default) | `legacy_compile_enabled = False` | Calls `_call_experimental_compile`, which invokes `bridge.extract_compiled_graph` from torch-xla on first call; subsequent calls reuse the compiled graph directly |
| Legacy | `legacy_compile_enabled = True` | Runs `MetadataInterpreter` + `MetadataDispatchMode` (if `XLA_HLO_DEBUG=1`) or directly calls the module, then signals XLA to cut the graph with `_xla_sync_multi` or `torch_xla.sync()` |

In SPMD mode, `_mark_unsharded_args_replicated` ([`python_package/tt_torch/backend/backend.py 99-116](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/backend/backend.py#L99-L116)) is called before execution to mark tensors with no sharding annotation as `REPLICATED`. This prevents spurious graph retracing in `dynamo_bridge`.

### Composite ops and fusion

**Composite ops** ([`python_package/tt_torch/composite_ops.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/composite_ops.py%60)) use `StableHLOCompositeBuilder` to wrap high-level operations (gelu, rms_norm, layer_norm) into named StableHLO composite ops. This allows tt-mlir to match and handle them with native implementations instead of decomposing them into many low-level StableHLO ops.

**Fusion providers** ([`python_package/tt_torch/fusion_providers.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/fusion_providers.py%60)) use `torch.fx.subgraph_rewriter.replace_pattern_with_filters` to match multi-node patterns (e.g., the LlamaRMSNorm expand-pow-mean-rsqrt-mul pattern) and replace them with a single op before the composite op substitution pass runs.

| Class | Pattern | Replacement |
| --- | --- | --- |
| `RMSNormFusionProvider` | `to(float32) → pow(2) → mean → add(eps) → rsqrt → mul → mul(weight)` | `torch.nn.functional.rms_norm` |

Sources: [`python_package/tt_torch/composite_ops.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/composite_ops.py%60)[`python_package/tt_torch/fusion_providers.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/fusion_providers.py%60)

### Metadata propagation (debug only)

When `XLA_HLO_DEBUG=1`, the legacy compile path activates `MetadataInterpreter` and `MetadataDispatchMode` from [`python_package/tt_torch/backend/metadata_propagation.py`](https://github.com/tenstorrent/tt-xla/blob/c77995f6/%60python_package/tt_torch/backend/metadata_propagation.py%60):

*   **`extract_nodes_info(graph_module)`** — iterates FX nodes after all passes, extracts source file paths, line numbers, and `nn_module_stack` hierarchy from `node.meta`, and returns a `dict[node_name → metadata_string]`.
*   **`MetadataInterpreter`** — an `fx.Interpreter` subclass that sets a `ContextVar` (`_current_node_metadata`) before executing each FX node.
*   **`MetadataDispatchMode`** — a `TorchDispatchMode` that reads `_current_node_metadata` and attaches it to XLA tensors via `torch_xla._XLAC._set_xla_custom_op_name_prefix`. This labels operations in the generated HLO IR with their source locations for debugging.

This mechanism correctly handles the case where a single FX node decomposes into multiple aten ops at dispatch time: all dispatched ops inherit the metadata from the enclosing FX node.
