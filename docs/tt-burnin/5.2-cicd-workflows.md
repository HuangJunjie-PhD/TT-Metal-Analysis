This document covers the automated build, test, and release workflows implemented through GitHub Actions for the tt-burnin project. The CI/CD system handles building Debian packages, Python wheels, and managing the complete release lifecycle including version management, changelog generation, and distribution to multiple package repositories.

For information about the actual package structures and build rules, see [Debian Packaging](https://deepwiki.com/tenstorrent/tt-burnin/5.1-debian-packaging). For details about version management and distribution channels, see [Release Management](https://deepwiki.com/tenstorrent/tt-burnin/5.3-release-management).

## Build Workflows

### Debian Package Building

The `build-debian.yml` workflow creates signed Debian packages for multiple Ubuntu versions using the `git-buildpackage` toolchain:

| Component | Purpose | Implementation |
| --- | --- | --- |
| **Matrix Strategy** | Multi-OS builds | `ubuntu-22.04`, `ubuntu-24.04`, `ubuntu-latest` |
| **Package Builder** | `gbp buildpackage` | Debian package generation with GPG signing |
| **Version Management** | `gbp dch` | Debian changelog generation |
| **Artifact Upload** | GitHub Actions | Distributes `.deb` files and changelogs |

## Workflow Input Parameters

The workflows accept standardized input parameters for version control and build customization:

| Parameter | Type | Purpose | Used By |
| --- | --- | --- | --- |
| **`ref`** | `string` | Git reference to build from | All workflows |
| **`MAJOR`** | `string` | Major version number | `build-debian.yml`, `release.yml` |
| **`MINOR`** | `string` | Minor version number | `build-debian.yml`, `release.yml` |
| **`PATCH`** | `string` | Patch version number | `build-debian.yml`, `release.yml` |
| **`NUMBER_OF_COMMITS_SINCE_TAG`** | `string` | Commit count for development builds | `build-debian.yml`, `release.yml` |

## Security and Signing

The build system implements package signing for security and authenticity:

*   **GPG Signing**: Debian packages are signed using `DEBSIGN_KEYID` from GitHub secrets
*   **Trusted Publishing**: PyPI uploads use OpenID Connect for secure authentication without API tokens
*   **Secret Management**: GPG keys stored in `PKG_SIGNING_KEY_DEB` repository secret
