## Purpose and Scope

This document describes the physical organization of the SFPI repository, including the layout of source files, header directories, test suites, build scripts, and toolchain components. It provides a navigational guide to understand where different types of files are located and how the repository's directory structure maps to the system's functional components.

For information about the logical architecture and how components interact, see [SFPI System Components](https://deepwiki.com/tenstorrent/sfpi/1.2-sfpi-system-components). For build system details, see [Build System Architecture](https://deepwiki.com/tenstorrent/sfpi/5.1-build-system-architecture). For integration with downstream projects, see [Integration with tt-metal](https://deepwiki.com/tenstorrent/sfpi/7.1-integration-with-tt-metal).

## Top-Level Repository Organization

The SFPI repository follows a hybrid structure combining custom SFPI headers with standard GNU toolchain components. The root directory contains configuration files, while subdirectories organize platform-specific implementations, tests, and build infrastructure.

**Directory Structure Overview**

| Directory/File | Purpose |
| --- | --- |
| `include/` | Platform-independent SFPI C++ API headers |
| `wormhole/`, `blackhole/`, `grayskull/` | Platform-specific hardware abstractions and math libraries |
| `gcc/` | TT-enhanced RISC-V GCC compiler (submodule) |
| `binutils/` | TT-enhanced RISC-V binutils (submodule) |
| `newlib/` | Standard C library implementation (submodule) |
| `riscv-dejagnu/` | DejaGnu test harness for RISC-V (submodule) |
| `tests/` | SFPI-specific test suites by architecture |
| `xfails/` | Known test failure tracking and filtering |
| `scripts/` | Build, release, and utility scripts |
| `.github/workflows/` | GitHub Actions CI/CD pipeline definitions |
| `configure`, `Makefile.in` | Autoconf/Automake build system files |

## Core SFPI Headers (include/)

The `include/` directory contains the platform-independent programming interface that provides the primary C++ API for SFPU programming.

**Core Header Files**

| File | Purpose | Key Contents |
| --- | --- | --- |
| `sfpi.h` | Primary programming interface | Vector types, operators, control flow constructs, conditional compilation directives |
| `sfpi_fp16.h` | 16-bit float conversions | `s2vFloat16`, `s2vFloat16a`, `s2vFloat16b` classes for dual FP16 formats |
| `lltt.h` | Instruction replay interface | `lltt::record()`, `lltt::replay()`, `lltt::replay_insn()` functions |

The headers use conditional compilation based on architecture-specific preprocessor macros to include the appropriate platform implementations. User code typically includes only `sfpi.h`, which transitively includes the necessary platform-specific headers.

## Platform-Specific Implementations

Each Tenstorrent chip architecture has a dedicated directory containing three-layer hardware abstraction: hardware intrinsics, implementation details, and math library functions.

**Platform Directory Structure**

Each architecture directory follows the same three-file pattern:

| File | Layer | Purpose |
| --- | --- | --- |
| `sfpi_hw.h` | Hardware Intrinsics | GCC built-in functions (`__builtin_rvtt_sfp*`) mapping directly to SFPU instructions |
| `sfpi_imp.h` | Implementation | C++ class implementations for `vFloat`, `vInt`, `vUInt`, register access patterns |
| `sfpi_lib.h` | Math Library | High-level mathematical operations built on hardware intrinsics |

**Architecture Differences**

*   **Wormhole**: 51 operations, `ADDR_MODE_NOINC = 3`, baseline feature set
*   **Blackhole**: 54 operations, `ADDR_MODE_NOINC = 7`, adds `sfparecip`, `sfpmul24`, `rand()`, `approx_recip()`, `approx_exp()`
*   **Grayskull**: Earlier architecture with core feature set

## Toolchain Components (Submodules)

The repository includes four Git submodules containing the enhanced RISC-V toolchain components required to compile SFPU code.

**Submodule Details**

| Submodule | Repository Type | Purpose |
| --- | --- | --- |
| `gcc/` | Git submodule | Forked RISC-V GCC with Tenstorrent SFPU extensions and intrinsics |
| `binutils/` | Git submodule | Forked RISC-V binutils with SFPU instruction encoding support |
| `newlib/` | Git submodule | Standard newlib C library for bare-metal RISC-V targets |
| `riscv-dejagnu/` | Git submodule | DejaGnu test infrastructure with RISC-V board definitions |

**QEMU**: Not a submodule but cloned on-demand during testing. Version 7.2.15 is used for RISC-V user-mode emulation.

**Initializing Submodules**

The submodules must be initialized after cloning the repository:

`git submodule update --init --recursive`

## Testing Infrastructure

The `tests/` directory contains platform-specific test suites that validate SFPI functionality and compiler correctness.

**Test Organization by Architecture**

| Architecture | Test Files | Gold Standard Files |
| --- | --- | --- |
| Wormhole | `coverage.cc`, `ckernel.cc`, `kernels.cc`, `regmov.cc` | `gold/coverage.S`, `gold/ckernel.S`, `gold/regmov.S`, `gold/kernels_int`, `gold/kernels_float` |
| Grayskull | `coverage.cc`, `regmov.cc` | `gold/regmov.S` |
| Blackhole | (Future implementation) | (Future implementation) |

**Test Types**

*   **coverage.cc**: Instruction-level tests exercising individual SFPU intrinsics
*   **ckernel.cc**: Kernel function tests for mathematical operations (exp, log, gelu, reciprocal, etc.)
*   **kernels.cc**: Integration tests combining multiple operations
*   **regmov.cc**: Compiler optimization validation focusing on register move elimination

**Building and Running Tests**

Tests are built by linking them against the compiled toolchain:

`ln -s ../tests buildCC_PATH=$(pwd)/build/sfpi/compiler make -C build/tests allCC_PATH=$(pwd)/build/sfpi/compiler make -C build/tests test`
The `test` target compares generated assembly against gold standard files in `gold/` subdirectories.

## Expected Test Failures (xfails/)

The `xfails/` directory tracks known test failures from the upstream GCC and binutils test suites that are expected due to platform limitations or test environment constraints.

**Expected Failure Files**

| File | Purpose |
| --- | --- |
| `gcc.xfail` | C compiler test cases expected to fail |
| `g++.xfail` | C++ compiler test cases expected to fail |
| `ld.xfail` | Linker test cases expected to fail |
| `binutils.xfail` | Binutils (assembler, etc.) test cases expected to fail |
| `local-xfails.py` | Python script that filters `.sum` files by removing expected failures |

The `local-xfails.py` script is run automatically after DejaGnu test execution to post-process summary files. It removes failures listed in the `.xfail` files, leaving only unexpected failures for investigation. The filtered results are placed in the `build/` directory while originals remain unchanged.

## Build and Release Scripts (scripts/)

The `scripts/` directory contains shell scripts that orchestrate building, testing, packaging, and releasing the SFPI toolchain.

**Script Descriptions**

| Script | Purpose | Key Operations |
| --- | --- | --- |
| `build.sh` | Main build orchestrator | Configure toolchain, compile binutils→GCC→newlib, optionally run test suites |
| `release.sh` | Release package creator | Verify source hashes, strip binaries, create `.txz`, `.deb`, `.rpm`, generate checksums |
| `sfpi-info.sh` | Version management | Extract version from `version` file and Git metadata (tags, branch) |
| `riscv32-unknown-elf-run` | Test runner wrapper | Parse QEMU arguments and execute tests under QEMU RISC-V emulation |

**Build Script Options**

*   `--dir=$DIR`: Specify build directory (default: `build`)
*   `--checking=VALUE`: GCC internal checking level (default: `release`)
*   `--test`: Run full GCC/binutils test suites
*   `--test-binutils`: Run only binutils tests
*   `--test-gcc`: Run only GCC tests
*   `--test-tt`: Run only TT-specific tests
*   `--dejagnu`: Build DejaGnu test infrastructure
*   `--sim`: Build QEMU simulator
*   `--gdb`: Build GDB debugger
*   `--tt-built`: Use pre-built toolchain instead of building

**Release Script Options**

*   `--dir=$DIR`: Specify build directory
*   `--force`: Skip source hash verification
*   `--ci`: CI mode (additional checks)
*   `--tt-built`: Package pre-built toolchain

**Version Numbering**

The version is determined by combining the contents of the `version` file with Git branch information. Branch name (last component after `/`) is appended unless on `main` branch.

## Configuration Files

The repository root contains Autoconf/Automake configuration files inherited from the upstream RISC-V GNU toolchain project.

**Configuration File Purposes**

| File | Purpose | Origin |
| --- | --- | --- |
| `configure.ac` | Autoconf source defining build options, dependencies, and configuration logic | RISC-V GNU toolchain upstream |
| `Makefile.in` | Make template with rules for building binutils, GCC, newlib, tests | RISC-V GNU toolchain upstream |
| `configure` | Generated shell script for system detection and Makefile generation | Generated from `configure.ac` |
| `version` | Plain text file containing base version number | SFPI-specific |

**Build Configuration Process**

1.   `configure.ac` defines available options (`--with-arch`, `--with-abi`, etc.)
2.   `autoconf` generates `configure` script from `configure.ac`
3.   User runs `./configure` with desired options
4.   `configure` processes `Makefile.in` to generate `Makefile`
5.   `make` executes build rules from generated `Makefile`

The `build.sh` script automates this process with appropriate defaults for SFPI.

## CI/CD Pipeline

The `.github/workflows/` directory contains GitHub Actions workflow definitions for automated building, testing, and releasing.

**CI/CD Workflow Structure**

The primary workflow file `build-sfpi.yaml` orchestrates the complete build-test-release cycle:

**Build Matrix**

*   Architectures: `x86_64`, `aarch64`
*   Containers: `none` (native), `Alma` (AlmaLinux)

**Job Steps**

1.   **Checkout**: Clone repository with submodules and fetch Git tags for version detection
2.   **Install Dependencies**: Install build prerequisites (compilers, libraries) and FPM gem for package creation
3.   **Build**: Execute `scripts/build.sh --gdb --tt-built` to compile toolchain
4.   **Test**: Run `scripts/build.sh --test-tt` for TT-specific test validation
5.   **Release**: Create packages with `scripts/release.sh --ci --tt-built`
6.   **Upload Artifacts**: Publish build products, test results, and configuration for distribution

**Artifacts Generated**

*   Toolchain binaries (`.txz` tarball)
*   Debian package (`.deb`)
*   RPM package (`.rpm`)
*   Checksums (`.sha256`)
*   Test summaries
*   Build configuration

## Generated Build Artifacts

During the build process, several directories are created to contain intermediate and final build products.

**Generated Directories**

| Directory | Purpose |
| --- | --- |
| `build/stamps/` | Checkpoint markers for idempotent build steps |
| `build/build-binutils-newlib/` | Binutils compilation intermediate files |
| `build/build-gcc-newlib-stage1/` | GCC Stage 1 (bootstrap) compilation |
| `build/build-gcc-newlib-stage2/` | GCC Stage 2 (full) compilation |
| `build/infra/` | Test infrastructure (QEMU, DejaGnu) binaries |
| `build/sfpi/compiler/` | Final installed toolchain (binaries, libraries, headers) |
| `build/sfpi/README.txt` | Source provenance information (submodule hashes) |

**Build Tracking Files**

*   `build/src-hashes`: SHA checksums of source directories at build start
*   `build/version`: Full version string including branch suffix
*   `build/*.sum`: Processed test summary files (filtered by `local-xfails.py`)
*   `build/*.log`: Original DejaGnu test log files

The `release.sh` script verifies that `build/src-hashes` matches current source state to ensure build integrity before packaging.
