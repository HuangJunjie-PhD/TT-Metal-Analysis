## Purpose and Scope

This document provides a high-level overview of the CI/CD infrastructure for the `tt-metal` repository. It describes the pipeline stages from code commit through testing to release, the test matrix organization across different hardware platforms, and the mechanisms for change detection and quality enforcement.

The CI/CD system is designed to validate the two-tier architecture (TT-Metalium and TTNN) across supported Tenstorrent hardware, including Wormhole (N150/N300) and Blackhole (P100/P150) [[.github/workflows/ttsim.yaml:48-53]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).

## Test Matrix Organization

Tests are organized into dynamic matrices generated at runtime to optimize hardware utilization and provide coverage across different SKUs.

### Matrix Generation Implementation

*   **Demo Tests Matrix**: `single-card-demo-tests-impl.yaml` defines a static list of demo tests (e.g., `vgg`, `resnet`, `stable_diffusion`) mapped to runner labels like `N150` or `N300`[[.github/workflows/single-card-demo-tests-impl.yaml:40-90]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Performance Matrix**: `perf-device-models-impl.yaml` filters an `original_matrix` based on selected models and architectures like `wormhole_b0` (N300) and `blackhole` (P150) [[.github/workflows/perf-device-models-impl.yaml:142-178]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).

## Container and Build System

The CI/CD pipeline relies on a sophisticated Docker-based build system to provide reproducible environments for compilation and testing.

### Docker Image Strategy

*   **Multi-stage Builds**: `dockerfile/Dockerfile` uses stages like `base`, `ci-build`, and `ci-test` to optimize image size and caching [[dockerfile/Dockerfile:136-154]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Tool Layers**: Tools like `cmake`, `ccache`, `mold`, and `sfpi` are injected from specialized layers to ensure consistent toolchains [[dockerfile/Dockerfile:48-60, 98-101]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Platform Support**: Images are built for both `Ubuntu 22.04` and `Ubuntu 24.04`[[.github/workflows/pr-gate.yaml:89-95]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).

### Build Artifacts

*   **Debian Packages**: The build produces `.deb` packages for `tt-metalium`, `tt-metalium-jit`, and `tt-metalium-dev`[[.github/workflows/ttsim.yaml:92]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Python Wheels**: Wheels are generated to support `ttnn` and model testing [[.github/workflows/build-artifact.yaml:27-31]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Ccache Integration**: Builds use `ccache` to significantly reduce compilation times in incremental CI runs [[CMakeLists.txt:112-115]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).

## Quality Gates and Static Analysis

The pipeline includes several non-functional checks to maintain code quality and stability.

### Code Analysis

*   **Clang-Tidy**: Incremental and full scans are performed to enforce C++ coding standards [[.github/workflows/code-analysis.yaml:156-200]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Static Checks**: The `merge-gate.yaml` workflow runs a suite of static checks before allowing code into the merge queue [[.github/workflows/merge-gate.yaml:56-62]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Sanitizers**: Specialized builds for AddressSanitizer (ASan) and ThreadSanitizer (TSan) are used to detect memory errors and data races [[.github/workflows/pr-gate.yaml:121-133]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview), [[.github/workflows/merge-gate.yaml:138-156]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).

### Quality Metrics

*   **Performance Regressions**: Tracked via `perf-device-models-impl.yaml` which runs end-to-end model performance tests [[.github/workflows/perf-device-models-impl.yaml:179-210]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
*   **Change Detection**: `find-changed-files` action is used to skip unnecessary build or test steps if relevant files haven't changed [[.github/workflows/pr-gate.yaml:146-156]](https://deepwiki.com/tenstorrent/tt-metal/6.1-cicd-architecture-overview).
