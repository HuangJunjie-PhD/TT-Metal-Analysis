This document covers the graph optimization passes applied to the computational graph in the PyBuda framework. Graph optimization transforms the initial computation graph through a series of passes that eliminate redundant operations, fuse compatible operations, optimize data formats, and make the graph more efficient for execution on Tenstorrent hardware. For information about the balancer and placement system that follows graph optimization, see [Balancer and Placement System](https://deepwiki.com/tenstorrent/tt-buda/2.2-balancer-and-placement-system).

## Overview of Graph Optimization in the Compilation Pipeline

Graph optimization sits between the initial graph creation and the balancing/placement phases in the PyBuda compilation pipeline. It applies a series of transformations to improve the execution efficiency of the model.

Sources: [pybuda/csrc/buda_passes.cpp 129-211](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/buda_passes.cpp#L129-L211)

## Main Optimization Pass Categories

The graph optimization system includes several categories of passes, each addressing different aspects of graph efficiency:

1.   **Transformation Management (TM)**: Optimize reshape, transpose, and other data transformation operations
2.   **Quantization Optimization**: Improve and enable efficient quantized operations
3.   **Operation Fusion**: Combine compatible operations to reduce overhead
4.   **Pattern Replacement**: Replace operation sequences with more efficient equivalents
5.   **Fork-Join Optimization**: Optimize parallel branches in the computation graph

Sources: [pybuda/csrc/buda_passes.cpp 129-211](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/buda_passes.cpp#L129-L211)

## TM Operation Optimizations

Transformation operations (TMs) like reshape and transpose are critical for adjusting tensor dimensions but can add significant overhead if not optimized properly. The PyBuda framework includes several passes focused on optimizing these operations.

### Inverse Operation Elimination

One of the most effective optimizations is identifying and removing pairs of operations that cancel each other out (inverse operations). For example, a reshape followed by its inverse reshape can be entirely removed.

The `erase_inverse_ops` pass identifies these patterns and eliminates them when possible. If direct elimination isn't possible, the system tries other approaches like:

1.   Inserting inverse operations on outputs
2.   Inserting inverse operations on inputs
3.   Replacing incommutable patterns with equivalent operations

Sources: [pybuda/csrc/buda_passes.cpp 154-174](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/buda_passes.cpp#L154-L174)[pybuda/csrc/passes/erase_inverse_ops.cpp 18-56](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/erase_inverse_ops.cpp#L18-L56)

### Commuting Transformations

The commutation utilities enable moving transformation operations through other operations when possible. This is critical for creating optimization opportunities.

The system can commute operations through:

*   Elementwise operations
*   Reduce operations (with specific constraints)
*   Concatenate and Select operations
*   Quantization operations

## Fork-Join Optimization

The fork-join pattern occurs when a node's output is used by multiple operations (fork) and then those branches converge later (join). This creates potential issues if the branches have different execution times.

The fork-join optimization identifies these patterns and inserts buffering operations to balance the execution paths:

1.   First, the system identifies fork-join patterns in the graph
2.   It calculates the execution time difference between paths
3.   It inserts NOPs or buffers on faster paths to balance execution

Sources: [pybuda/csrc/passes/fork_join.cpp 60-254](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/fork_join.cpp#L60-L254)[pybuda/csrc/passes/fork_join.hpp 76-463](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/fork_join.hpp#L76-L463)

## Optimization Pass Implementation Details

The graph optimization passes are implemented in C++ and follow a structured pattern:

### Core Optimization Algorithm

Most optimization passes follow this general pattern:

1.   Traverse the graph in topological order
2.   Identify patterns that can be optimized
3.   Apply transformations
4.   Repeat until no more changes are possible

For example, the `erase_inverse_ops` pass repeatedly searches for patterns until no more changes are possible:

```
bool attempt_update = true;
while (attempt_update)
{
    attempt_update = passes::erase_inverse_ops(graph);
    if (not attempt_update) {
        attempt_update = passes::insert_inverse_on_outputs(graph);
        // ... more attempts
    }
}
```

## Optimization Execution Order

The optimization passes are executed in a specific order to maximize effectiveness. The main optimization sequence is:

1.   **Lower concat to runtime transform**
2.   **Bypass NOP transforms**
3.   **Fuse reshape transpose pairs into slice/stack**
4.   **Erase inverse operations** (iterative process with fallbacks)
5.   **Move transforms outside quantized regions**
6.   **Hoist transforms to inputs**
7.   **Pad output buffer**
8.   **Lower reinterpret shape**
9.   **Bind reshape to IO**
10.   **Fuse per-channel operations**
11.   **Constant folding** (optional)
12.   **Move select after matmul** (optional)
13.   **Fuse transform sequences**

Sources: [pybuda/csrc/buda_passes.cpp 129-211](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/buda_passes.cpp#L129-L211)

## Configuration Options

Graph optimization can be configured through environment variables:

| Environment Variable | Purpose | Default |
| --- | --- | --- |
| `PYBUDA_DISABLE_CONSTANT_FOLDING` | Disable constant folding optimization | `false` |
| `PYBUDA_DISABLE_SINGLE_REDUCE_COMMUTE` | Disable commuting through single reduce ops | `false` |
| `PYBUDA_DISABLE_CONV_BIAS_QDQ_INSERTION` | Disable quantization-dequantization insertion for conv biases | `false` |
| `PYBUDA_SPARSE_NO_MATH` | Disable sparse math optimization | `false` |
| `PYBUDA_FUSE_MATMUL_GELU` | Enable fusion of matmul and GELU operations | `false` |

## Advanced Feature: Commutation Through Different Operation Types

The graph optimization system includes specialized handlers for commuting operations through various operation types:

Each operation type has specific constraints that determine whether commutation is possible:

*   **Elementwise**: Almost always allows commutation
*   **Reduce**: Requires special handling for reduction dimensions
*   **Concat/Select**: Requires alignment of dimensions
*   **Quantization**: Requires maintaining quantization semantics
*   **Squeeze**: Only supported in specific cases (e.g., squeeze dim 0)

Sources: [pybuda/csrc/passes/commute_utils.cpp 543-911](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/commute_utils.cpp#L543-L911)[pybuda/csrc/passes/commute_utils.cpp 989-1019](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/csrc/passes/commute_utils.cpp#L989-L1019)

## Conclusion

Graph optimization is a critical phase in the PyBuda compilation pipeline that significantly improves model execution efficiency on Tenstorrent hardware. Through a series of passes that eliminate redundant operations, fuse compatible patterns, optimize quantization, and handle fork-join patterns, the framework transforms the initial computational graph into a form that can be efficiently balanced, placed, and executed.

The optimization system is highly configurable and extensible, allowing for customization based on model characteristics and hardware targets. Understanding these optimizations can help users get the best performance from their models when running on Tenstorrent hardware.
