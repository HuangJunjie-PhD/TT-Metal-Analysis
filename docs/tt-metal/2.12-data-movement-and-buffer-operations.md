## Purpose and Scope

This page documents the data movement and buffer operations system in TT-Metalium, which handles transferring data between host memory and device memory (L1/DRAM). This includes the APIs for writing data to buffers on Tenstorrent devices and reading data back to the host, as well as the underlying dispatch mechanisms that execute these transfers.

For information about buffer allocation and memory management, see [Memory Management and Allocators](https://deepwiki.com/tenstorrent/tt-metal/2.7-memory-management-and-allocators). For details on the command queue system that executes these operations, see [Fast Dispatch and Command Queue System](https://deepwiki.com/tenstorrent/tt-metal/2.5-fast-dispatch-and-command-queue-system).

* * *

## System Overview

Data movement in TT-Metalium operates at two levels: direct synchronous transfers and asynchronous queued transfers. The system supports both single-device operations (via `IDevice`) and multi-device operations (via `MeshDevice`), with specialized handling for different buffer memory layouts.

## Buffer Memory Layouts and Validation

The data transfer logic varies based on buffer memory layout. TT-Metalium supports several layouts defined in `TensorMemoryLayout`[tt_metal/impl/buffers/buffer.cpp 59-61](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L59-L61)

| Layout Type | Description | Use Case | Memory Distribution |
| --- | --- | --- | --- |
| **INTERLEAVED** | Pages distributed round-robin across memory banks | General purpose tensors | Sequential across DRAM channels or L1 banks |
| **HEIGHT_SHARDED** | Shards divided by height across cores | Parallel compute | Specific worker core grid [tt_metal/impl/buffers/buffer.cpp 98-115](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L98-L115) |
| **WIDTH/BLOCK_SHARDED** | Shards divided by width or 2D blocks | Parallel compute | Complex 2D core mapping [tt_metal/impl/buffers/buffer.cpp 116-146](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L116-L146) |

### Buffer Parameter Validation

Before any data movement occurs, the system validates buffer parameters in `validate_buffer_parameters`[tt_metal/impl/buffers/buffer.cpp 55-82](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L55-L82):

*   Ensures sharded buffers have a `shard_spec` or `buffer_distribution_spec`[tt_metal/impl/buffers/buffer.cpp 62-70](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L62-L70)
*   Verifies that the total buffer size is divisible by the page size [tt_metal/impl/buffers/buffer.cpp 76-81](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L76-L81)
*   Validates `sub_device_id` to ensure shard cores match the allocated sub-device worker cores [tt_metal/impl/buffers/buffer.cpp 150-168](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L150-L168)
*   Validates `sub_device_manager_id` to ensure the buffer is being used with the active manager on the device [tt_metal/impl/buffers/buffer.cpp 170-178](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L170-L178)

## Host-to-Device Write Operations

### Asynchronous Write API (Fast Dispatch)

Asynchronous writes are enqueued to the command queue. This involves `BufferWriteDispatchParams` and its specialized derived classes to compute NOC offsets and assemble command packets.

**Key Dispatch Parameters:**

*   `InterleavedBufferWriteDispatchParams`: Handles logic for interleaved buffers. It includes logic to move bank base addresses if the `dst_page_index` exceeds the maximum uint16_t offset allowed in dispatch commands (`CQ_DISPATCH_CMD_PAGED_WRITE_MAX_PAGE_INDEX`) [tt_metal/impl/buffers/dispatch.cpp 130-141](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/dispatch.cpp#L130-L141)
*   `InterleavedBufferWriteLargePageDispatchParams`: Specialized for pages that require partial page specifications [tt_metal/impl/buffers/dispatch.cpp 161-180](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/dispatch.cpp#L161-L180)
*   `ShardedBufferWriteDispatchParams`: Maps host pages to specific cores in a shard grid [tt_metal/impl/buffers/dispatch.cpp 182-210](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/dispatch.cpp#L182-L210)

**Alignment and Pinned Memory:** For pinned memory transfers, the system calculates an `alignment_prefix_bytes` to align host memory writes to the HAL-defined host alignment (e.g., `HalMemType::HOST`) [tt_metal/impl/buffers/dispatch.cpp 110-116](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/dispatch.cpp#L110-L116)

### Synchronous and Multi-Device Support

In a `MeshDevice` context, writes are coordinated by the command queue system. While Fast Dispatch (`FDMeshCommandQueue`) is preferred for performance, Slow Dispatch (`SDMeshCommandQueue`) is supported for specific legacy or validation scenarios [tt_metal/distributed/sd_mesh_command_queue.cpp 71-106](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/sd_mesh_command_queue.cpp#L71-L106)`SDMeshCommandQueue` performs synchronous writes using `tt::tt_metal::detail::WriteToBuffer`[tt_metal/api/tt-metalium/tt_metal.hpp 88-121](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/tt_metal.hpp#L88-L121) after waiting for cores to be idle [tt_metal/distributed/sd_mesh_command_queue.cpp 87-103](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/sd_mesh_command_queue.cpp#L87-L103)

## Device-to-Host Read Operations

### Asynchronous Read API

Asynchronous reads utilize the completion queue mechanism. The host enqueues a read command, and the dispatcher writes requested data into the completion region in host system memory.

**Completion Queue Reader:** The `FDMeshCommandQueue` manages a `completion_queue_reader_thread_` that processes entries in the completion queue [tt_metal/distributed/fd_mesh_command_queue.cpp 167](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/fd_mesh_command_queue.cpp#L167-L167) It uses a `MultiProducerSingleConsumerQueue` of `MeshCompletionReaderVariant` to track outstanding reads [tt_metal/distributed/fd_mesh_command_queue.hpp 126-128](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/fd_mesh_command_queue.hpp#L126-L128)

**Multi-Device Read Handling:** In `MeshDevice` environments, `distributed::ReadShard` provides a high-level interface to read data from a specific mesh coordinate back to host memory [tt_metal/api/tt-metalium/tt_metal.hpp 145-161](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/tt_metal.hpp#L145-L161)

### Synchronous Read API

Synchronous reads perform a blocking read from the device. In `SDMeshCommandQueue`, `read_shard_from_device` waits for cores to be idle before performing the read via `tt::tt_metal::detail::ReadFromBuffer`[tt_metal/api/tt-metalium/tt_metal.hpp 122-139](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/tt_metal.hpp#L122-L139)[tt_metal/distributed/sd_mesh_command_queue.cpp 129-138](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/sd_mesh_command_queue.cpp#L129-L138) For sharded buffers, `core_to_host_pages` is used to map the sharded core data back into a contiguous host-side representation [tt_metal/impl/buffers/buffer.cpp 84-148](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/buffers/buffer.cpp#L84-L148)

## Mesh Buffer Operations

Multi-device data movement is abstracted via the `MeshDevice` and its command queue. Operations can be performed on `MeshBuffer` objects, which represent a buffer distributed across a `MeshDevice`.

### MeshBuffer Creation and Allocation

`MeshBuffer::create` handles the allocation of buffers across a mesh [tt_metal/distributed/mesh_buffer.cpp 84-107](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_buffer.cpp#L84-L107) It supports different allocation modes:

*   **Per-core allocation**: Each device allocates independently [tt_metal/distributed/mesh_buffer.cpp 110-130](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_buffer.cpp#L110-L130)
*   **Lockstep/Hybrid allocation**: The `MeshDevice` allocator provides a synchronized address across the mesh [tt_metal/distributed/mesh_buffer.cpp 131-157](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_buffer.cpp#L131-L157)

### Shard Data Transfer

The `distributed::ShardDataTransfer` structure is used to specify host data and the target `BufferRegion` when enqueuing writes or reads to a mesh [tt_metal/api/tt-metalium/mesh_command_queue.hpp 164-173](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/mesh_command_queue.hpp#L164-L173) This allows for granular control over which parts of a mesh-resident buffer are updated or retrieved.

### FDMeshCommandQueue Execution

The `FDMeshCommandQueue` implements multi-device dispatch by managing a completion queue reader thread [tt_metal/distributed/fd_mesh_command_queue.cpp 167](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/fd_mesh_command_queue.cpp#L167-L167) It uses a `RingbufferCacheManager` to manage prefetcher cache space for commands [tt_metal/distributed/fd_mesh_command_queue.cpp 154-158](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/fd_mesh_command_queue.cpp#L154-L158) When reading from the mesh, it populates `MeshCompletionReaderVariant` descriptors which are then processed by the reader thread to perform memory copies back to the host [tt_metal/distributed/fd_mesh_command_queue.hpp 33-35](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/fd_mesh_command_queue.hpp#L33-L35)
