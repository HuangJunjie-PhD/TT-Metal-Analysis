## Purpose and Scope

This document describes the profiling and performance analysis infrastructure within the `tt-metal` repository. It details the integration of the **Tracy Profiler**, the **Device Profiler** for on-chip kernel timing, and the Python-based performance analysis scripts. These tools are essential for identifying host-side bottlenecks, device-side kernel stalls, and NOC (Network-on-Chip) congestion to optimize model execution on Tenstorrent hardware.

* * *

## Profiling Infrastructure Overview

The profiling system spans the entire stack, from low-level RISC-V kernels to high-level Python host code. It is designed to provide a unified timeline of execution across multiple asynchronous processors.

### System Architecture

Title: Profiling Data Flow and Component Interaction

**Key Components:**

*   **Kernel Profiler**: A header-only library used within device kernels to record timestamps and custom markers into L1 memory. It timestamps kernel events using RISC-V registers and manages buffer flushing to DRAM to prevent overflow. It also supports tracing NOC events. [tt_metal/tools/profiler/kernel_profiler.hpp 40-116](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L40-L116)
*   **NOC Event Profiler**: Instrumentation for tracking Network-on-Chip read/write and barrier events, recording source/destination coordinates, byte counts, and posting information for detailed congestion analysis. [tt_metal/tools/profiler/noc_event_profiler.hpp 7-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/noc_event_profiler.hpp#L7-L112)
*   **ProfilerStateManager**: The host-side manager that orchestrates profiling sessions, reading data from devices, aggregating results, and interfacing with the Tracy timeline. [tt_metal/impl/profiler/profiler_state_manager.hpp 1-30](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler_state_manager.hpp#L1-L30)
*   **Tracy Integration**: Handles the translation and forwarding of device profiling events into the Tracy profiler context, enabling live, cross-device profiling visualization. [tt_metal/impl/profiler/profiler.cpp 20-60](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler.cpp#L20-L60)

## Device-Side Profiling (Kernel Profiler)

The Device Profiler enables nanosecond-resolution timing of RISC-V kernels running on various cores (BRISC, NCRISC, TRISC, ERISC).

### Implementation Details

Device-side profiling uses a dedicated profiling message buffer (`profiler_msg_buffer_t`) located in the core's L1 mailbox area. Kernels record timestamped events into this buffer which can be flushed to DRAM if needed.

| Feature | Description | Code Reference |
| --- | --- | --- |
| **Timestamping** | Uses RISC-V wall clock registers (low/high) accessible via special debug registers. | [tt_metal/tools/profiler/kernel_profiler.hpp 178-185](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L178-L185) |
| **Markers** | Unique 16-bit marker IDs generated by compile-time hashing of source code location strings. | [tt_metal/tools/profiler/kernel_profiler.hpp 34-36](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L34-L36) |
| **Buffer Flushing** | Supports flushing L1 buffer data to DRAM to manage profiling data size and prevent data loss. | [tt_metal/tools/profiler/kernel_profiler.hpp 127-156](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L127-L156) |
| **NOC Tracing** | Records NoC (Network-on-Chip) transfer events including read, write, multicast and barrier events. | [tt_metal/tools/profiler/noc_event_profiler.hpp 88-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/noc_event_profiler.hpp#L88-L112) |

### Data Flow: Device to Host

1.   **Initialization**: The `init_profiler()` function clears device buffers and sets up initial profiling state including the run counter. [tt_metal/tools/profiler/kernel_profiler.hpp 117-156](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L117-L156)
2.   **Event Recording**: Each kernel uses `mark_time_at_index_inlined()` to log 32-bit timer IDs and timestamps into the buffer. [tt_metal/tools/profiler/kernel_profiler.hpp 178-185](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/tools/profiler/kernel_profiler.hpp#L178-L185)
3.   **Host-Device Sync**: The `syncDeviceHost()` function launches a sync kernel (`sync_kernel.cpp`) that aligns device timebase with host CPU time for accurate correlation. [tt_metal/impl/profiler/tt_metal_profiler.cpp 102-180](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/tt_metal_profiler.cpp#L102-L180)
4.   **Data Readback**: The profiling buffers are read back on the host using slow MMIO dispatch or fast command queue dispatch, processed and interpreted by the `DeviceProfiler` class. [tt_metal/impl/profiler/profiler.hpp 197-205](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler.hpp#L197-L205)

## Tracy Profiler Integration

The Tracy Profiler is integrated to provide unified visualization of asynchronous host and device execution timelines.

### Entity Mapping and Interaction

Title: Profiler Host-Side Class Relationships

### Key Functions

*   **`setControlBuffer`**: Configures the profiling control buffer for each device core, setting parameters like the number of cores per DRAM buffer bank. It also triggers device write of this control data. [tt_metal/impl/profiler/tt_metal_profiler.cpp 76-100](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/tt_metal_profiler.cpp#L76-L100)
*   **`make_noc_debug_event`**: Converts raw NOC event logs from device memory into `NOCDebugEvent` structures, which are then used to annotate Tracy timelines with communication events. This helps visualize network traffic and potential bottlenecks. [tt_metal/impl/profiler/profiler.cpp 113-168](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler.cpp#L113-L168)
*   **`risc_type_to_control_buffer_dram_address_offset`**: Maps RISC core types (BRISC, NCRISC, etc.) to offsets within the profiler DRAM buffer, enabling precise location of profiling data for each RISC core type. [tt_metal/impl/profiler/profiler.cpp 170-190](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler.cpp#L170-L190)
*   **`risc_type_to_control_buffer_host_index_offset`**: Maps RISC types to indices for internal host buffers managing profiling data readback states. [tt_metal/impl/profiler/profiler.cpp 191-210](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/profiler/profiler.cpp#L191-L210)

## Performance Analysis and Benchmarking

The repository includes Python-based tools and tests for analyzing raw profiling logs and generating performance reports.

### Post-Processing Pipeline

Profiling data captured from devices is processed via scripts such as `process_ops_logs.py` and `process_device_log.py`. This pipeline aggregates raw logs, computes timing and hardware utilization metrics, and generates human-readable CSV and JSON reports.

Key Steps:

1.   **Log Import**: Reads JSON and CSV artifacts, like `device_analysis_data.json` and `cpp_device_perf_report.csv`, to collect raw performance data. [tools/tracy/process_ops_logs.py 170-210](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/tracy/process_ops_logs.py#L170-L210)
2.   **Metric Computation**: Calculates kernel firmware durations, op-to-op latencies, and kernel execution times, including minimum, maximum, and average per-core durations. [tools/tracy/process_ops_logs.py 62-128](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/tracy/process_ops_logs.py#L62-L128)
3.   **Efficiency Analysis**: Evaluates utilization metrics such as FPU use and NOC efficiencies using performance counters collected during profiling. [tools/tracy/process_ops_logs.py 39-47](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/tracy/process_ops_logs.py#L39-L47)

### Performance Regression Tests

To ensure regression safety and performance monitoring, automated tests run as part of the CI pipeline validate profiling data against known reference values for architectures like Wormhole and Blackhole.

**Example Test Cases:**

*   **`test_multi_op`**: Confirms the expected number of kernel start/end markers are collected over repeated launches, matching architecture-specific expected counts. [tests/tt_metal/tools/profiler/test_device_profiler.py 162-181](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tools/profiler/test_device_profiler.py#L162-L181)
*   **`test_multi_op_buffer_overflow`**: Verifies the profiler correctly handles buffer overflow scenarios where the L1 buffer is flushed multiple times to DRAM. It checks compute and data movement marker counts. [tests/tt_metal/tools/profiler/test_device_profiler.py 184-211](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tools/profiler/test_device_profiler.py#L184-L211)

## Summary

The `tt-metal` profiling infrastructure provides high-resolution kernel timing and system event tracing through integrated device-side code instrumentation and advanced host-side tools. This infrastructure feeds data into the Tracy Profiler for real-time visualization, and post-processing scripts for deeper performance analysis. Together, these components enable developers and researchers to optimize Tenstorrent hardware workloads with precise measurement and analysis.
