## Purpose and Scope

This document describes the debugging and performance monitoring capabilities available in Wormhole B0 and Blackhole A0 architectures. It covers mechanisms for:

*   Monitoring NoC transaction status through hardware counters
*   Inspecting Tensix backend configuration state
*   Performing component-level isolation via soft reset
*   Reading packer/unpacker operational state
*   Accessing router and NIU debug information

For information about programming the Tensix coprocessor efficiently, see [7.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.3-kernel-structure-and-mop-usage). For details on instruction scheduling and hazards, see [7.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.2-instruction-scheduling-and-hazards). For memory access optimization, see [7.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.1-memory-access-patterns-and-optimization).

## NoC Transaction Monitoring

### NIU Counter Architecture

Each NIU (NoC Interface Unit) exposes an array of 32-bit hardware counters at memory-mapped addresses starting at `NIU_BASE + 0x200`. These counters automatically track transaction lifecycle events without software intervention.

**Sources:**[WormholeB0/NoC/Counters.md 1-174](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Counters.md?plain=1#L1-L174)[WormholeB0/NoC/MemoryMap.md 148-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L148-L156)

### Counter Types and Usage

The NIU counter array contains three categories of counters:

| Index | Counter Name | Width | Purpose |
| --- | --- | --- | --- |
| 0-15 | Command/Response Tracking | 32-bit | Total packets sent/received (wrap-around) |
| 16-31 | `NIU_MST_REQS_OUTSTANDING_ID(i)` | 8-bit | Outstanding requests by transaction ID |
| 32-47 | `NIU_MST_WRITE_REQS_OUTGOING_ID(i)` | 8-bit | In-flight write data by transaction ID |
| 48-61 | Slave-side Tracking | 32-bit | Received requests/sent responses |

The most useful counters for monitoring transaction completion are `NIU_MST_REQS_OUTSTANDING_ID` and `NIU_MST_WRITE_REQS_OUTGOING_ID`, which are indexed by the 4-bit `NOC_PACKET_TRANSACTION_ID` field in the request.

## Component-Level Soft Reset

### Soft Reset Register Overview

The `RISCV_DEBUG_REG_SOFT_RESET_0` register at address `0xFFB1_2000` provides fine-grained control for resetting individual Tensix components. This is valuable for debugging by isolating problematic components or forcing known-good states. Each bit controls a specific component, with `1` asserting reset and `0` releasing reset.

**RISCV_DEBUG_REG_SOFT_RESET_0 Bit Assignments:**

**Sources:**[WormholeB0/TensixTile/SoftReset.md 1-32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L1-L32)

### Reset Bit Functions

| Bit(s) | Component | Key Effects |
| --- | --- | --- |
| 0,1,7 | Unpackers | Abort `UNPACR`, block new unpacks |
| 2-5 | Packers 0-3 | Abort `PACR`, clear output buffers, reset `AccTileSize`/`LastTileSize` |
| 6 | Mover | Abort DMA operations, discard `XMOV` instructions |
| 8 | TDMA-RISC & Glue | Empty mover command queue, discard TDMA writes |
| 9 | ThCon Config & Scalar Unit | Zero `Config[i][j]` for `THCON_CFGREG_BASE_ADDR32 ≤ j < GLOBAL_CFGREG_BASE_ADDR32` |
| 10 | Matrix & Vector Units | Abort FPU/SFPU ops, reset PRNGs, zero/initialize `LReg` |
| 11-14,18 | RISC-V cores | Invalidate I-cache, clear GPRs, set PC to reset value |
| 15,19-22 | SrcA | Zero register data, reset `AllowedClient` |
| 16 | SrcB | Zero register data, reset `AllowedClient` |
| 17 | Packer-Dst & ZEROACC | Reset exponent histograms, revert `ZEROACC` effects |

### RISC-V Reset Behavior

When a RISC-V core is taken out of soft reset, the following actions occur:

1.   All 32 general-purpose registers (GPRs) are cleared to zero
2.   Instruction cache is invalidated
3.   Program counter (PC) is set to the reset vector

The reset vector for each core is determined by backend configuration fields:

| Core | Reset Vector Determination | Default Address |
| --- | --- | --- |
| RISC-V B | Always `0x00000` | `0x00000` (L1 base) |
| RISC-V T0 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[0] ? Config.TRISC_RESET_PC_SEC0_PC : 0x06000` | `0x06000` |
| RISC-V T1 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[1] ? Config.TRISC_RESET_PC_SEC1_PC : 0x0A000` | `0x0A000` |
| RISC-V T2 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[2] ? Config.TRISC_RESET_PC_SEC2_PC : 0x0E000` | `0x0E000` |
| RISC-V NC | `Config.NCRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en ? Config.NCRISC_RESET_PC_PC : 0x12000` | `0x12000` |

Configuration fields are located at specific offsets within the `Config` array at `TENSIX_CFG_BASE` (`0xFFEF_0000`).

**Sources:**[WormholeB0/TensixTile/SoftReset.md 33-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L33-L51)

## Packer and Unpacker State Inspection

### Reading Packer State via SETDMAREG

The `SETDMAREG` instruction (when used with `WhichOp == 1`) allows reading packer runtime state into GPRs. This is useful for monitoring tile processing progress and compression effectiveness.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md 1-113](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md?plain=1#L1-L113)

### Tile Size Tracking

Packers maintain per-thread tile size accumulators:

*   `AccTileSize`: Accumulated size across multiple pack operations
*   `LastTileSize`: Size from the most recent pack operation (only valid if `LastThread == CurrentThread`)

## NoC Debug Information Regions

### Router and NIU Debug Registers

Each NIU exposes read-only debug information at two memory regions:

| Address Range | Contents | Access |
| --- | --- | --- |
| `NIU_BASE + 0x300` to `0x37F` | NoC router debug information | Read-only via MMIO/NoC |
| `NIU_BASE + 0x380` to `0x3FF` | NIU debug information | Read-only via MMIO/NoC |

These regions contain internal hardware state useful for low-level debugging of NoC behavior, packet routing, and flow control.

**Sources:**[WormholeB0/NoC/MemoryMap.md 21-22](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L21-L22)

### NIU Identification Registers

Two identification registers provide topology and tile type information:

**`NOC_NODE_ID` (`NIU_BASE + 0x02C`):**

| Bits | Field | Description |
| --- | --- | --- |
| 0-5 | X coordinate | NoC X position (0-9) |
| 6-11 | Y coordinate | NoC Y position (0-11) |
| 12-18 | NoC width | Always 10 |
| 19-25 | NoC height | Always 12 |
| 26 | X dateline | True if dateline flips on X axis |
| 27 | Y dateline | True if dateline flips on Y axis |
| 28 | X-first routing | True for NoC #0, false for NoC #1 |

**`NOC_ENDPOINT_ID` (`NIU_BASE + 0x030`):**

| Bits | Field | Values |
| --- | --- | --- |
| 0-7 | Tile index | Tensix: 0-79, Ethernet: 0-15, etc. |
| 8-15 | Group index | DRAM: 0-5, others: 0 |
| 16-23 | Tile type | 1=Tensix, 2=Ethernet, 3=PCIe/Empty, 5=ARC, 8=DRAM |
| 24-31 | NoC index | 0 for NoC #0, 1 for NoC #1 |
