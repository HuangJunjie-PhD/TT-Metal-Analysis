## Purpose and Scope

This page documents the kernel debugging infrastructure within the `tt-metal` repository. It focuses on on-device debugging capabilities, including the **DPRINT** (Device Print) server for kernel-side logging and the **Watcher** system for monitoring device state and detecting hangs or errors. These tools allow developers to inspect the execution of kernels across multiple RISC-V processors (Data Movement, Unpack, Math, Pack) on Tenstorrent hardware.

## Architecture Overview

The kernel debugging system consists of host-side servers that poll device-side mailboxes and buffers. Because hardware threads (RISCs) cannot directly access host standard output, they write to reserved sections of L1 SRAM defined by the Hardware Abstraction Layer (HAL). The `MetalContext` manages these singletons and their lifecycle during device initialization and teardown.

## DPRINT System

The DPRINT system provides a `std::ostream`-like interface for kernels. It supports basic types, hex/dec formatting, and specialized tile-slice printing for Circular Buffers (CBs).

### Implementation Details

The host-side `DPrintServer` manages the polling of device buffers. It uses a "magic value" handshake to synchronize with kernels during startup.

*   **Initialization**: The host writes `DEBUG_PRINT_SERVER_STARTING_MAGIC` to the buffer's write pointer address to signal it is ready to receive data. This is done by `WriteInitMagic()` which initializes the buffer on kernel startup and waits for the kernel to acknowledge readiness by clearing the magic value [tt_metal/impl/debug/dprint_server.cpp 122-150](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/dprint_server.cpp#L122-L150)

*   **Kernel Handshake**: The kernel clears this magic value when it first attempts to print, signaling the host that the kernel is running and the write pointer (`wpos`) now holds valid circular buffer state [tt_metal/impl/debug/dprint_server.cpp 141-150](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/dprint_server.cpp#L141-L150)

*   **Buffer Layout and Identification**: Each RISC on a core has a dedicated DPRINT buffer segment. The server identifies RISCs and their indices using HAL API (`hal.get_processor_class_name`) and cluster coordinate translation [tt_metal/impl/debug/dprint_server.cpp 93-104](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/dprint_server.cpp#L93-L104)

*   **Polling Loop**: The server polls each registered device and core for new debug prints, decoding the raw data into human-readable logs [tt_metal/impl/debug/dprint_server.cpp 185-200](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/dprint_server.cpp#L185-L200)

### Runtime Configuration

*   The DPRINT server behavior is controlled through `RunTimeOptions`. It supports filtering by device, core, and particular RISC processors.

*   The output log file and muting behavior are configurable. Muting effectively disables output while still polling to keep interfaces alive.

### Environment Variables

| Environment Variable | Description |
| --- | --- |
| `TT_METAL_DPRINT` | Enables DPRINT and specifies the target selection such as `worker` or specific cores/RISCs |
| `TT_METAL_DPRINT_LOG_FILE` | Specifies the path to the output log file (`generated/dprint/dprint.log` by default) |

## Watcher Debug System

The Watcher is a background host-side service that monitors device cores to detect kernel hangs, out-of-bounds memory accesses, invalid NOC transactions, and assert trap events.

### Key Components

*   **WatcherServer**: Runs a dedicated thread to poll all devices periodically, reading the Watcher mailboxes from device L1 memory and collecting debug data [tt_metal/impl/debug/watcher_server.cpp 46-107](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_server.cpp#L46-L107)

*   **WatcherDeviceReader**: Parses and decodes the binary mailbox buffers for each device. Uses HAL and cluster descriptions to interpret core coordinates and processor types [tt_metal/impl/debug/watcher_device_reader.cpp 36-41](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_device_reader.cpp#L36-L41)[tt_metal/impl/debug/watcher_device_reader.cpp 64-106](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_device_reader.cpp#L64-L106)

*   **Kernel Registration**: Kernels are registered with the watcher server by name and associated unique IDs. This allows mapping instruction pointers from core reports to source code or ELF debug info [tt_metal/impl/debug/watcher_server.cpp 62-64](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_server.cpp#L62-L64)

### Watcher Mailbox and Reporting

*   Each device has mailboxes (shared memory structures) that RISCs write to report status, waypoint hits, and asserts.

*   The WatcherServer reads these mailboxes, interprets error events, and logs detailed information including core, RISC, kernel ID, and assert line numbers.

*   The system relies on coordinate translation between NOC coordinates and virtual coordinates to identify the exact core and processor reporting errors [tt_metal/impl/debug/watcher_device_reader.cpp 135-151](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_device_reader.cpp#L135-L151)

### Sentinel and Sanitization Checks

*   To detect memory errors and out-of-bounds access, the firmware uses sanitizer hooks that mark invalid uses with special sentinel values.

*   The Watcher detects these through inspecting mailboxes for flags/events indicating sanitization failures.

*   The NOC address sanitization logic prevents devices from issuing invalid NOC transactions by checking physical coordinates and alignment [tt_metal/hw/inc/internal/debug/sanitize.h 5-15](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/inc/internal/debug/sanitize.h#L5-L15)[tests/tt_metal/tt_metal/debug_tools/watcher/test_sanitize.cpp 50-64](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tt_metal/debug_tools/watcher/test_sanitize.cpp#L50-L64)

## Hardware Abstraction Layer (HAL) Integration

Kernel debugging depends on HAL for memory address mapping, processor identification, and architecture-specific details.

*   **Memory Mapping**: DPRINT buffers and Watcher mailboxes reside in device local L1 SRAM at HAL-designated offsets. Host reads/writes to these regions are performed via `tt::Cluster::read_core` and `write_core` interfaces using virtual core coordinates [tt_metal/impl/debug/dprint_server.cpp 143-145](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/dprint_server.cpp#L143-L145)[tt_metal/impl/debug/watcher_server.cpp 113-117](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_server.cpp#L113-L117)

*   **Processor Naming and Identification**: The HAL provides APIs like `get_processor_class_name` which are used to map numerical processor indices/type to human-readable RISC names (e.g., `brisc`, `ncrisc`, `trisc0`) helping correlate debug events to hardware threads [tt_metal/impl/debug/watcher_device_reader.cpp 64-106](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_device_reader.cpp#L64-L106)

*   **Coordinate Translation**: Different hardware generations (Blackhole, Wormhole, Quasar) use different conventions for NoC coordinate translations. The Watcher and DPRINT readers use `MetalContext` cluster queries to translate physical coordinates back to virtual system coordinates [tt_metal/impl/debug/watcher_device_reader.cpp 135-151](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_device_reader.cpp#L135-L151)

## Post-Mortem Analysis Tools

### Watcher Dump Tool

The `WatcherServer::dump()` function and its helpers allow the host user to extract and persist the current debug state of all devices' mailboxes. This is used for offline inspection of kernel hangs, asserts, or sanitization failures [tt_metal/impl/debug/watcher_server.cpp 193-210](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/watcher_server.cpp#L193-L210)

### Hardware Fault Detection

*   The system detects hardware faults through tripped asserts logged by RISCs.

*   Test code in `tests/tt_metal/tt_metal/debug_tools/watcher/test_assert.cpp` runs kernels instruments to intentionally trip asserts and verifies the Watcher server detects these errors, confirms kernel names, core and RISC info, and logs the reason [tests/tt_metal/tt_metal/debug_tools/watcher/test_assert.cpp 52-140](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tt_metal/debug_tools/watcher/test_assert.cpp#L52-L140)

*   Hardware faults are mapped via error enums like `DmError` and `TriscError` to enable high-level debugging messages [tt_metal/impl/debug/debug_helpers.hpp 134-148](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/debug/debug_helpers.hpp#L134-L148)

### NOC Sanitization Error Handling

*   Firmware runs address sanitization to catch invalid NoC operations (e.g., wrong offsets, out-of-range coordinates).

*   If sanitization fails, the kernel soft-hangs in a spin loop preventing further corruption, allowing the host Watcher to gather debug info safely [tt_metal/hw/inc/internal/debug/sanitize.h 5-15](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/hw/inc/internal/debug/sanitize.h#L5-L15)[tests/tt_metal/tt_metal/debug_tools/watcher/test_sanitize.cpp 50-64](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/tt_metal/tt_metal/debug_tools/watcher/test_sanitize.cpp#L50-L64)

## Summary of Debugging Workflow

1.   **Enable Debugging Tools**: Set runtime options or environment variables (`TT_METAL_WATCHER=1`, `TT_METAL_DPRINT=worker`) to enable watcher and print servers before running kernels.

2.   **Instrument Kernels**: Use `DPRINT` macros or assertion macros (`WATCHER_ASSERT`) within kernel code to emit logs or trigger asserts.

3.   **Run Kernel on Target Hardware**: The execution emits debug data into reserved L1 memory buffers.

4.   **Host Polling**: The `DPrintServer` and `WatcherServer` poll devices for debug output buffers and mailboxes, parsing and translating messages.

5.   **Inspect Logs**: View logs written to `generated/dprint/dprint.log` and `generated/watcher/watcher.log`. Inspect assert failures or sanitizer traps including core, RISC, kernel, and instruction pointer context.

6.   **Post-Test Cleanup**: Watcher detaches devices, flushes logs, and restores runtime states.
