## Purpose and Scope

The Constant Folding Pass evaluates operations with constant inputs at compile time, replacing them with pre-computed values. This optimization reduces runtime overhead and simplifies the computation graph for downstream passes. This pass operates early in the compilation pipeline, after the graph has been captured but before operation lowering to TTNN.

For information about the overall pass pipeline orchestration, see [Backend Entry Points and Pass Pipeline](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.1-backend-entry-points-and-pass-pipeline). For details about guard system integration, see [Guard System and Operation Blocklists](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.4-guard-system-and-operation-blocklists).

* * *

## Overview

The `ConstantFoldingPass` is implemented as a `PassBase` subclass that traverses the FX graph and identifies operations whose inputs are all constants (scalars, graph attributes, or previously folded constants). When such operations are found, the pass evaluates them immediately using PyTorch's eager mode execution and replaces them with `get_attr` nodes that reference pre-computed parameter values.

**Diagram: Constant Folding Pass Execution Flow**

The pass transforms computation sequences like:

*   `arange(0, 100)` → `get_attr("_folded_arange")`
*   `pow(tensor_a, tensor_b)` where both are constants → `get_attr("_folded_pow")`
*   `view(constant_tensor, [shape])` → `get_attr("_folded_view")`

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 10-138](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L10-L138)

## Folding Eligibility Checks

The `_can_fold()` method determines whether a node is eligible for constant folding through a multi-stage check:

**Diagram: Folding Eligibility Decision Tree**

The eligibility criteria are:

1.   **Guard Check**: The node must pass the `can_lowering_to_ttnn()` guard check, ensuring the operation is not blocklisted for TTNN conversion
2.   **Constant Arguments**: All node arguments must be: 
    *   Scalars (`int`, `float`) or `None`
    *   FX nodes with `op` in `("get_attr", "constant")`
    *   Lists/tuples containing only the above types

This ensures that folding only occurs when all inputs can be resolved at compile time and the operation is safe to execute.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 51-87](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L87)[torch_ttnn/passes/lowering/to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L109-L113)

## Node Evaluation Mechanism

The `_evaluate_node()` method executes the operation eagerly to produce a folded value:

**Diagram: Node Evaluation Process**

Special handling is provided for `lift_fresh_copy`, which simply returns its input unchanged. All other operations are executed by calling `node.target(*args, **node.kwargs)` with the resolved arguments.

The method uses `unset_fake_temporarily()` context (in `call()`) to ensure eager execution occurs with real tensors rather than fake tensors used during graph tracing.

## Graph Transformation

The `_replace_with_constant()` method performs the actual graph modification:

**Diagram: Constant Replacement Transformation**

The transformation involves:

1.   **Node Creation**: A new `get_attr` node is created with target name `_folded_{original_name}`
2.   **Metadata Preservation**: Critical metadata is copied from the original node: 
    *   `seq_nr`: Maintains execution order
    *   `original_aten`: Preserves operation type information
    *   `tensor_meta`: Contains shape and dtype information

3.   **Parameter Registration**: The folded value is registered as a non-trainable parameter in the graph module
4.   **Use Replacement**: All downstream uses of the original node are redirected to the new `get_attr` node
5.   **Node Removal**: The original computation node is erased from the graph

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 126-137](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L126-L137)

## Integration with Guard System

The constant folding pass integrates with the guard system to ensure folded operations are compatible with TTNN lowering. The `_can_fold()` method calls `can_lowering_to_ttnn()` before attempting to fold any operation.

**Diagram: Guard System Integration**

This integration prevents folding of operations that would later be blocked during TTNN lowering. For example, certain `aten.arange` or `aten.cumsum` configurations may be blocklisted due to TTNN limitations, and the constant folding pass respects these constraints.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 52-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L52-L53)[torch_ttnn/passes/lowering/to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L109-L113)[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 298-303](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L298-L303)

## Error Handling

The pass includes defensive error handling to ensure compilation robustness:

`try:    folded_value = self._evaluate_node(gm, node)    self._replace_with_constant(gm, node, folded_value)except Exception as e:    print(f"Warning: Could not fold node {node.name}: {e}")`
If evaluation or replacement fails for any reason, the pass prints a warning and continues processing the graph. The original node remains in the graph unchanged, ensuring that compilation can proceed even if some constant folding opportunities are missed.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 43-47](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L43-L47)

## Pass Execution Context

The `call()` method executes within the `unset_fake_temporarily()` context manager, which temporarily disables fake tensor mode. This is necessary because:

1.   During graph tracing, PyTorch may use fake tensors to avoid materializing actual data
2.   Constant folding requires executing operations with real tensors to produce concrete values
3.   The context manager ensures eager execution occurs correctly
4.   After folding, the pass returns control and fake tensor mode may resume for subsequent passes

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 40-48](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L40-L48)

## Summary

The Constant Folding Pass provides the following benefits:

| Benefit | Description |
| --- | --- |
| **Runtime Performance** | Eliminates redundant computations by pre-computing constant expressions |
| **Graph Simplification** | Reduces graph complexity for downstream optimization passes |
| **Memory Efficiency** | Converts repeated constant computations into single parameter references |
| **Guard Compliance** | Respects TTNN operation blocklists to prevent invalid folding |
| **Robustness** | Gracefully handles folding failures without interrupting compilation |

The pass operates as an early optimization in the compilation pipeline, transforming constant expressions into pre-computed parameters before operation lowering and data movement insertion occur.
