**Purpose**: This document explains the `InputAnalysisPass` which classifies placeholder nodes in compiled FX graphs as `PARAMETER`, `BUFFER`, or `ARGUMENT` based on their source in the original PyTorch model. This classification enables downstream passes to apply different strategies for parameters (model weights), buffers (non-trainable state), and arguments (runtime inputs).

For information about how this classification is used in multi-device scenarios, see [Multi-Device Support and Sharding](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3.4-multi-device-support-and-sharding). For the broader context of analysis passes in the compilation pipeline, see [Graph Analysis and Optimization Passes](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3-graph-analysis-and-optimization-passes).

* * *

## Overview

The `InputAnalysisPass` solves a critical problem in the compilation pipeline: determining whether each placeholder node in the FX graph represents a model parameter, a buffer, or a runtime input argument. This distinction is essential because:

*   **Parameters** (e.g., `conv.weight`, `linear.bias`) are optimizer-tracked tensors that should be cached and replicated across devices
*   **Buffers** (e.g., batch normalization running statistics) are non-trainable but persistent model state
*   **Arguments** (e.g., input images, text sequences) are runtime inputs that may vary between calls and should be sharded for data parallelism

After PyTorch 2.7.1, the AOT Autograd system no longer guarantees a specific ordering of these input types. The `InputAnalysisPass` inspects PyTorch's internal source metadata to reliably classify each placeholder regardless of ordering.

**Sources**: [torch_ttnn/passes/analysis/input_analysis_pass.py 1-111](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py#L1-L111)

## PrimalTag Enumeration

The `PrimalTag` enum defines the three classification categories:

`class PrimalTag(Enum):    PARAMETER = 1    BUFFER = 2    ARGUMENT = 3`
| Tag | Description | Examples | Training Behavior |
| --- | --- | --- | --- |
| `PARAMETER` | Optimizer-tracked tensors representing weights and biases | `conv1.weight`, `linear.bias` | Updated during backward pass |
| `BUFFER` | Non-trainable model state that persists between calls | `bn.running_mean`, `bn.running_var` | Not updated by optimizer |
| `ARGUMENT` | Runtime inputs provided by user, not persistent | Input images, text tokens, batch data | Changes between calls |

The key distinction: `PARAMETER` and `BUFFER` tensors are expected to remain constant across multiple inference calls, while `ARGUMENT` tensors are expected to change.

**Sources**: [torch_ttnn/passes/analysis/input_analysis_pass.py 61-73](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py#L61-L73)

## Classification Algorithm

### Helper Function: find_first_source_member

The `find_first_source_member` function recursively traverses the source chain to find a specific member string:

**Diagram**: Recursive traversal algorithm for finding specific members in source chains.

This function searches for `_parameters` or `_buffers` strings in the source chain. If found, the placeholder originates from model state. If neither is found, it's a runtime argument.

**Sources**: [torch_ttnn/passes/analysis/input_analysis_pass.py 10-21](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py#L10-L21)

### Function: get_placeholder_types

The `get_placeholder_types` function applies the classification logic to all placeholder nodes:

**Diagram**: Classification logic flow showing priority order: buffers checked first, then parameters, then default to arguments.

**Example source metadata** for a parameter:

```
DictGetItemSource(
    base=UnspecializedBuiltinNNModuleSource(
        base=UnspecializedParamBufferSource(
            base=DictGetItemSource(
                base=UnspecializedNNModuleSource(
                    base=AttrSource(
                        base=LocalSource(local_name='self', is_input=True),
                        member='_modules'
                    )
                ),
                index='conv1'
            ),
            member='_parameters'  # <-- This identifies it as PARAMETER
        )
    ),
    index='weight'
)
```

**Example source metadata** for an argument:

```
LocalSource(
    local_name='x',
    is_input=True,
    dynamism=None,
    is_derefed_cell_contents=False
)
# No '_parameters' or '_buffers' member -> ARGUMENT
```

### Pass Execution Flow

The `call` method performs the annotation:

**Diagram**: Execution flow of `InputAnalysisPass.call` method showing the annotation process.

Key implementation details:

1.   **Input extraction**: Uses `get_input_nodes(gm.graph)` utility to get placeholder nodes in order
2.   **Length validation**: Only annotates if the placeholder count matches the type list length
3.   **Metadata storage**: Stores `PrimalTag` enum values in `node.meta["primal_tag"]`
4.   **Immutability**: Returns `modified=False` because no graph structure changes occur (only metadata)

## Usage in Downstream Passes

### MultiDevicePass Integration

The `MultiDevicePass` uses `PrimalTag` annotations to determine tensor distribution strategy across multiple devices:

**Diagram**: Distribution strategy based on PrimalTag classification in multi-device scenarios.

**Code example from MultiDevicePass**:

`for node in node_list:    if node.op == "placeholder":        try:            if (node.meta["primal_tag"] == PrimalTag.PARAMETER) or (                node.meta["primal_tag"] == PrimalTag.BUFFER            ):                self.replicate_to_mesh(node, first_node)            elif node.meta["primal_tag"] == PrimalTag.ARGUMENT:                self.shard_to_mesh(node, first_node)        except Exception as e:            logging.error(f"Could not distribute node {node.name}: {e}")`
The rationale:

*   **Parameters and buffers** are replicated because they're shared state that all devices need
*   **Arguments** are sharded to enable data parallelism (each device processes a subset of the batch)

**Sources**: [torch_ttnn/passes/multi_device_pass.py 88-98](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L88-L98)

## Key Metadata Fields

After `InputAnalysisPass` executes, each placeholder node has the following metadata structure:

| Metadata Field | Type | Description | Set By |
| --- | --- | --- | --- |
| `node.meta["primal_tag"]` | `PrimalTag` | Classification as PARAMETER, BUFFER, or ARGUMENT | `InputAnalysisPass` |
| `node.meta["grapharg"]` | `GraphArg` | Original source metadata from AOT Autograd | AOT Autograd |
| `node.meta["grapharg"].source` | `Source` | Chain of source objects for tracing origin | PyTorch Dynamo |

**Example inspection in test**:

`# From test files showing metadata accessfor node in graph_module.graph.nodes:    if node.op == "placeholder":        tag = node.meta["primal_tag"]        if tag == PrimalTag.PARAMETER:            # Apply parameter-specific logic            ...`

## Error Handling and Edge Cases

### Length Mismatch

The pass only annotates nodes if the placeholder count matches the type list length. This prevents misalignment:

`if len(node_list) == len(self.placeholder_types):    # Only annotate if counts match    for node, placeholder_type in zip(node_list, self.placeholder_types):        # ... annotate`
If there's a mismatch, the pass silently returns without modifications, allowing the compilation to continue but potentially causing issues in downstream passes that expect annotations.

**Sources**: [torch_ttnn/passes/analysis/input_analysis_pass.py 101-109](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py#L101-L109)

* * *

### Multi-Device Exception Handling

The `MultiDevicePass` wraps primal_tag access in exception handlers to gracefully handle missing annotations:

`try:    if (node.meta["primal_tag"] == PrimalTag.PARAMETER) or (        node.meta["primal_tag"] == PrimalTag.BUFFER    ):        self.replicate_to_mesh(node, first_node)    elif node.meta["primal_tag"] == PrimalTag.ARGUMENT:        self.shard_to_mesh(node, first_node)except Exception as e:    logging.error(f"Could not distribute node {node.name}: {e}")`
This ensures that missing `primal_tag` metadata doesn't crash the compilation, though it may result in incorrect tensor distribution.

**Sources**: [torch_ttnn/passes/multi_device_pass.py 90-98](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L90-L98)

## Test Coverage

### Multi-Device Test Example

The test `test_torch_scalar_sub_fresh` demonstrates how parameter classification affects multi-device execution:

`class FreshLiftCopyModule(torch.nn.Module):    def __init__(self):        super().__init__()     def forward(self, x):        tensor = torch.tensor(1.0, dtype=torch.bfloat16)  # Constant (buffer-like)        return tensor - x  # x is ARGUMENT`
In this test:

*   `x` is classified as `ARGUMENT` and would be sharded across devices
*   The constant tensor is lifted and treated as a buffer, replicated to all devices

**Sources**: [tests/multi_device/test_multi_folded_lift_fresh_copy.py 1-43](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/multi_device/test_multi_folded_lift_fresh_copy.py#L1-L43)

### Scaled Dot Product Attention Test

The `test_sdpa` test shows classification of complex inputs with multiple placeholders:

`def test_sdpa(device, input_shape, is_causal, attn_mask_shape):    # query, key, value are all ARGUMENT placeholders    query = fa_rand(*input_shape, dtype=torch.bfloat16)    key = fa_rand(*input_shape, dtype=torch.bfloat16)    value = fa_rand(*input_shape, dtype=torch.bfloat16)    attn_mask = torch.rand(attn_mask_shape) < 0.5 if attn_mask_shape else None`
All these tensors are runtime inputs, so `InputAnalysisPass` classifies them as `ARGUMENT`. Any model parameters (if present) would be classified separately.

## Summary

The `InputAnalysisPass` provides essential metadata for the compilation pipeline:

1.   **Classification**: Distinguishes parameters, buffers, and arguments through source metadata inspection
2.   **Persistence awareness**: Enables passes to know which tensors change between calls
3.   **Distribution strategy**: Allows multi-device passes to replicate model state while sharding data
4.   **Pipeline integration**: Executes early to provide metadata for downstream optimization passes

This classification is fundamental to correct multi-device execution and efficient memory management in the compiled model.
