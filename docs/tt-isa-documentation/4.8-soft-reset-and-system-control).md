## Purpose and Scope

This page documents the soft reset mechanism and system control capabilities available in the Tensix tile. The soft reset register (`RISCV_DEBUG_REG_SOFT_RESET_0`) provides fine-grained control for resetting individual hardware units within the Tensix coprocessor and RISC-V cores. This mechanism allows software to selectively reset specific execution units, data paths, and control structures without affecting the entire tile.

For information about backend configuration management, see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system). For RISC-V core architecture and initialization, see [Baby RISC-V Cores](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores).

## Soft Reset Register Overview

The soft reset register is a 32-bit memory-mapped control register providing per-component reset control within the Tensix tile. Each bit in `RISCV_DEBUG_REG_SOFT_RESET_0` controls reset functionality for one or more hardware units.

### Register Location and Access

```
Register Symbol: RISCV_DEBUG_REG_SOFT_RESET_0
Memory Region: Tile control / debug / status registers
Address: Within tile's NoC-accessible MMIO space
Width: 32 bits (write-any, read-back)
Access: All RISC-V cores (B/T0/T1/T2/NC) + NoC clients
```

This register resides in the "Tile control / debug / status registers" memory region, which is mapped into the address space of all five RISC-V cores within the tile. External agents (other tiles, host via PCIe, ARC) can also access it via NoC memory-mapped transactions targeting this tile's MMIO region. The register supports both read and write operations, allowing software to query the current reset state as well as modify it.

Sources: [WormholeB0/TensixTile/SoftReset.md 1-4](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L1-L4)

### Critical Atomicity Constraint

> **CAUTION**: There is no hardware support for atomic bit manipulation (set/clear individual bits) of `RISCV_DEBUG_REG_SOFT_RESET_0`.

Software must implement read-modify-write (RMW) sequences:

When multiple agents (RISC-V cores or NoC clients) need to manipulate different bits concurrently, software-level mutual exclusion is required to prevent race conditions. Available synchronization primitives include:

*   Mailboxes (FIFO between RISC-V cores)
*   Tensix semaphores (for Tensix coprocessor threads)
*   NoC atomic operations (compare-and-swap, atomic increment)
*   Scalar Unit `ATCAS` instruction (compare-and-set on L1 memory)

## Reset Phase Model

Soft reset operates in three distinct phases for each controlled unit:

**Figure 2: Soft Reset State Machine**

### Phase 1: Entering Soft Reset

When a bit transitions from 0 to 1, immediate actions occur:

*   In-flight operations are typically aborted (though software should not rely on this if reset is only asserted briefly)
*   Internal buffers and queues are cleared
*   State variables are reset to default values

### Phase 2: Held in Soft Reset

While a bit remains 1:

*   New operations targeting the unit will not start
*   Incoming requests may be silently discarded
*   The unit maintains its reset state

### Phase 3: Leaving Soft Reset

When a bit transitions from 1 to 0:

*   Registers are initialized to their reset values
*   Caches are invalidated if applicable
*   Default configuration is applied
*   The unit returns to operational state

## RISC-V Soft Reset

Bits 11-14 and 18 control the five Baby RISC-V cores: B, T0, T1, T2, and NC.

### Bit Assignments and Constants

Individual RISC-V cores are controlled by separate bits:

| Bit | Core | Constant Name |
| --- | --- | --- |
| 11 | RISC-V B (Brisc) | `RISCV_SOFT_RESET_0_BRISC` |
| 12 | RISC-V T0 (Trisc0) | Part of `RISCV_SOFT_RESET_0_TRISCS` |
| 13 | RISC-V T1 (Trisc1) | Part of `RISCV_SOFT_RESET_0_TRISCS` |
| 14 | RISC-V T2 (Trisc2) | Part of `RISCV_SOFT_RESET_0_TRISCS` |
| 18 | RISC-V NC (NCrisc) | `RISCV_SOFT_RESET_0_NCRISC` |

The constant `RISCV_SOFT_RESET_0_TRISCS` is a bitmask covering bits 12-14 (all three Trisc cores).

**Figure 3: RISC-V Core Reset Phases**

### Reset Phase Behaviors

**Upon entering soft reset:**

*   Any in-flight RISC-V instructions are likely aborted (though this is not guaranteed, especially for brief reset assertions)
*   For RISC-V T i: Instructions in the MOP Expander or preceding FIFO are discarded
*   Mailboxes writable by the RISC-V core have their contents discarded
*   All debug `DR` registers are set to zero

**Whilst held in soft reset:**

*   New RISC-V instructions will not start
*   The frontend stops performing instruction reads against L1 and core-local instruction RAM
*   Writes to debug `DR` registers are discarded

**Upon leaving soft reset:**

*   For RISC-V T i: The PCBuf from RISC-V B is cleared
*   Branch predictor history is forgotten
*   Instruction cache is invalidated
*   All GPRs are set to zero
*   `pc` is set according to the table below

### Reset Vectors

Upon leaving soft reset, each RISC-V core's program counter (`pc`) is initialized as follows:

| Core | Reset Vector Expression | Default Value | Override Config Field |
| --- | --- | --- | --- |
| RISC-V B | `0x00000` | `0x00000` | N/A (fixed) |
| RISC-V T0 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[0] ?` `Config.TRISC_RESET_PC_SEC0_PC : 0x06000` | `0x06000` | `TRISC_RESET_PC_SEC0_PC` |
| RISC-V T1 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[1] ?` `Config.TRISC_RESET_PC_SEC1_PC : 0x0A000` | `0x0A000` | `TRISC_RESET_PC_SEC1_PC` |
| RISC-V T2 | `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en.Bit[2] ?` `Config.TRISC_RESET_PC_SEC2_PC : 0x0E000` | `0x0E000` | `TRISC_RESET_PC_SEC2_PC` |
| RISC-V NC | `Config.NCRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en ?` `Config.NCRISC_RESET_PC_PC : 0x12000` | `0x12000` | `NCRISC_RESET_PC_PC` |

These reset vectors are byte addresses into L1 memory (`0x00000` - `0x16FFFF` range), where firmware/kernel code is expected to be loaded. The `Config` variables reference the global backend configuration array `Config[2][CFG_STATE_SIZE * 4]` (see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system)). The relevant configuration fields are:

*   `Config.TRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en` (field symbol): Enable bits `[2:0]` for T0/T1/T2 override
*   `Config.TRISC_RESET_PC_SEC0_PC`, `Config.TRISC_RESET_PC_SEC1_PC`, `Config.TRISC_RESET_PC_SEC2_PC` (field symbols): Override reset vectors for T0/T1/T2
*   `Config.NCRISC_RESET_PC_OVERRIDE_Reset_PC_Override_en` (field symbol): Enable bit for NC override
*   `Config.NCRISC_RESET_PC_PC` (field symbol): Override reset vector for NC

These configuration fields can be written via RISC-V `sw` instructions or via `WRCFG` instructions before asserting soft reset, allowing software to control where each core begins execution.

## Execution Unit Soft Reset

Bit 10 controls the Matrix Unit (FPU), Vector Unit (SFPU), and all columns of SrcA data simultaneously.

**Figure 4: Execution Unit Reset Interconnections**

### Matrix Unit (FPU) Reset

**Upon entering soft reset:** Any in-flight Matrix Unit instructions are aborted.

**Whilst held in soft reset:**

*   New Matrix Unit instructions will not start (may be silently discarded)
*   Unpacker 0 cannot write to `Dst`

**Upon leaving soft reset:** PRNGs used for stochastic rounding are reset.

### Vector Unit (SFPU) Reset

**Upon entering soft reset:**

*   Any in-flight Vector Unit instructions are aborted
*   Instructions scheduled via `SFPLOADMACRO` are forgotten

**Whilst held in soft reset:** New Vector Unit instructions will not start (may be silently discarded).

**Upon leaving soft reset:**

*   Vector Unit PRNG is reset
*   All `LaneConfig` and `LoadMacroConfig` are set to zero
*   Conditional execution stack is emptied
*   Conditional execution mode is set to "unconditional"
*   Mutable `LReg` registers are initialized per the table below:

| Register | Reset Value | Format |
| --- | --- | --- |
| `LReg[0]` - `LReg[7]` | Zero | - |
| `LReg[11]` | `-1.0` | FP32 |
| `LReg[12]` | `1.0/65536` | FP32 |
| `LReg[13]` | `-0.67487759` | FP32 |
| `LReg[14]` | `-0.34484843` | FP32 |
| `LReg[16]` | Zero | - |

## Data Path Soft Reset

### Unpacker Soft Reset

Bits 0, 1, and 7 collectively control both unpackers. If any bit is set, all three should be set. The effects described apply when all bits are set.

**Upon entering soft reset:** Any in-flight `UNPACR` and `UNPACR_NOP` instructions are aborted.

**Whilst held in soft reset:**

*   New `UNPACR` and `UNPACR_NOP` instructions will not start (may be silently discarded)
*   In-flight `STOREIND` instructions writing to `SrcA` or `SrcB` might behave strangely

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 136-144](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L136-L144)

### Packer Soft Reset

Bits 2-5 individually control the four packers (packer 0 through packer 3). Bit 17 also affects all packers (see [Packer Dst and ZEROACC Soft Reset](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Packer%20Dst%20and%20ZEROACC%20Soft%20Reset)).

**Figure 5: Packer Reset Control Structure**

**Upon entering soft reset:**

*   Any in-flight `PACR` and `PACR_SETREG` instructions on the packer are aborted
*   Non-empty output buffers before L1 have their contents discarded
*   `AccTileSize` and `LastTileSize` are set to zero (observable via `SETDMAREG` and TDMA-RISC)
*   The metadata output FIFO is emptied

**Whilst held in soft reset:** New `PACR` and `PACR_SETREG` instructions will not start on the packer (may be silently discarded).

**Upon leaving soft reset:**`l1_dest_addr_offset` is set to zero (can be set again via TDMA-RISC).

Sources: [WormholeB0/TensixTile/SoftReset.md 146-154](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L146-L154)

### Mover Soft Reset

Bit 6 controls the Mover DMA engine.

**Upon entering soft reset:** Any in-flight Mover command/instruction is aborted.

**Whilst held in soft reset:**

*   Tensix `XMOV` instructions are discarded rather than sent to the mover
*   TDMA-RISC mover commands are discarded
*   `STALLWAIT` condition C12 observes the mover as having no requests outstanding
*   TDMA-RISC observes the mover as not busy

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 104-110](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L104-L110)

## Register File Soft Reset

### SrcA Data Soft Reset

Bits 10, 19-22 control SrcA register data. Bit 10 resets all columns, while bits 19-22 control groups of four columns.

| Bit | Columns Affected |
| --- | --- |
| 10 | All columns (0-15) |
| 19 | Columns 0-3 |
| 20 | Columns 4-7 |
| 21 | Columns 8-11 |
| 22 | Columns 12-15 |

Software is encouraged to use `ZEROSRC` rather than relying heavily on SrcA data soft reset.

**Upon entering soft reset:** The relevant columns of `SrcA` (all banks, all rows) have their data set to zero.

**Whilst held in soft reset:** Instructions writing to the relevant columns of `SrcA` (any bank, any row) have the writes to those columns discarded.

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 52-60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L52-L60)

### SrcB Soft Reset

Bit 16 controls both SrcB AllowedClient coordination and SrcB data.

Software is encouraged to use `ZEROSRC` for data clearing and `CLEARDVALID` for AllowedClient reset.

**Upon entering soft reset (data):** All columns of `SrcB` (all banks, all rows) have their data set to zero.

**Upon entering soft reset (AllowedClient):**

**Whilst held in soft reset:**

*   Instructions writing to `SrcB` have the writes discarded
*   Instructions cannot change `MatrixUnit.SrcBBank`, `Unpackers[1].SrcBank`, `SrcB[0].AllowedClient`, or `SrcB[1].AllowedClient`

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 62-102](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L62-L102)

### SrcA AllowedClient Soft Reset

Bit 15 controls SrcA AllowedClient coordination only (not data). Software is encouraged to use `CLEARDVALID` rather than relying heavily on this soft reset.

**Upon entering soft reset:**

**Whilst held in soft reset:** Instructions cannot change `MatrixUnit.SrcABank`, `Unpackers[0].SrcBank`, `SrcA[0].AllowedClient`, or `SrcA[1].AllowedClient`.

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 72-86](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L72-L86)

### Packer Dst and ZEROACC Soft Reset

Bit 17 affects the connection between all packers and `Dst`, as well as the `ZEROACC` mechanism.

**Upon entering soft reset:**

*   Invokes the `Reset` function on the exponent histogram of all four packers (software should use `CLREXPHIST` instead)
*   Behaves as if a `ZEROACC` instruction was executed with `Mode == ZEROACC_MODE_ALL_OF_DST` and `Revert == true`
    *   Any rows of `Dst` previously set to `Undefined` revert to their pre-undefined values

**Whilst held in soft reset:**

*   `PACR` instructions reading from `Dst` will behave strangely
*   `ZEROACC` instructions are silently discarded

**Upon leaving soft reset:** PRNGs used for stochastic rounding in early format conversion are reset.

## Configuration and Infrastructure Reset

### Configuration and Scalar Soft Reset

Bit 9 controls both THCON configuration and the Scalar Unit.

Software is encouraged to use `STATE_RESET_EN_ADDR32` for configuration reset rather than relying heavily on soft reset.

**Upon entering soft reset (THCON configuration):**

The configuration reset clears all thread-agnostic configuration registers in the THCON range. This is equivalent to executing:

This affects configuration registers at MMIO address range:

*   Start: `TENSIX_CFG_BASE + THCON_CFGREG_BASE_ADDR32 * 4` (byte address)
*   End (exclusive): `TENSIX_CFG_BASE + GLOBAL_CFGREG_BASE_ADDR32 * 4` (byte address)

The cleared range includes:

*   Packer configuration: `Config[i].Packers[0].*` through `Config[i].Packers[3].*`
*   Unpacker configuration (partial)
*   ADC (addressing counter) base configurations
*   Various other THCON-related state

The reset excludes global configuration fields at addresses `>= GLOBAL_CFGREG_BASE_ADDR32`, which include RISC-V branch predictor controls, PRNG seed registers, and other chip-wide settings.

**Upon entering soft reset (Scalar Unit):** Any in-flight Scalar Unit instruction taking more than one cycle to execute is likely aborted (though not guaranteed for brief reset assertions).

**Whilst held in soft reset:**

*   All writes to `Config[i][j]` with `THCON_CFGREG_BASE_ADDR32 ≤ j < GLOBAL_CFGREG_BASE_ADDR32` are silently discarded
*   `DMANOP` and `SETDMAREG` instructions can execute
*   Other Scalar Unit instructions will not start

**Upon leaving soft reset:** No additional action.

Sources: [WormholeB0/TensixTile/SoftReset.md 190-212](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L190-L212)

### TDMA and Glue Soft Reset

Bit 8 controls both TDMA-RISC and glue logic.

**Upon entering soft reset (TDMA-RISC):**

*   Some value is assigned to `Unpackers.SetRegBase`, `Packers.SetRegBase`, and `Packers.SetRegHiScaler`
*   The command queue for instructing the mover is emptied
*   The related `CmdParams` array is set to zero

**Upon entering soft reset (Glue):** No particular action.

**Whilst held in soft reset:**

*   All writes to the TDMA-RISC memory region are silently discarded
*   New Scalar Unit, `PACR`, `PACR_SETREG`, `UNPACR`, `UNPACR_NOP`, and `XMOV` instructions will not start
*   In-flight instructions accessing L1 might behave strangely

**Upon leaving soft reset:** No additional action.

## Usage Patterns and Alternatives

**Figure 6: Soft Reset Usage Decision Tree**

### When to Use Soft Reset

Soft reset is most appropriate when:

1.   **Cold initialization**: Bringing a RISC-V core from power-on to a known state
2.   **Error recovery**: Recovering from a hung or malfunctioning unit
3.   **Context switching**: Preparing a unit for execution by a different workload
4.   **Debugging**: Forcing a unit into a known state for test purposes

### Synchronization Considerations

Before asserting or deasserting soft reset:

1.   **Check for in-flight operations**: Use `STALLWAIT` or TDMA-RISC status queries to ensure no critical operations are in progress
2.   **Coordinate across RISC-V cores**: Use mailboxes or PCBufs to synchronize reset actions
3.   **Handle atomicity**: Implement read-modify-write sequences with proper mutual exclusion
4.   **Wait for effects to complete**: Some reset actions may take multiple cycles to fully complete

### Preferred Alternatives

Many reset operations have preferred alternatives:

| Reset Purpose | Soft Reset Approach | Preferred Alternative |
| --- | --- | --- |
| Clear SrcA/SrcB data | Bits 10, 16, 19-22 | `ZEROSRC` instruction |
| Clear Dst undefined rows | Bit 17 | `ZEROACC` instruction |
| Reset AllowedClient | Bits 15, 16 | `CLEARDVALID` instruction |
| Clear THCON configuration | Bit 9 | Write to `STATE_RESET_EN_ADDR32` |
| Clear exponent histogram | Bit 17 | `CLREXPHIST` instruction |

These alternatives typically:

*   Provide more precise control
*   Avoid side effects on unrelated state
*   Execute atomically without requiring read-modify-write sequences
*   Are better documented in their specific instruction references

## System Control Beyond Soft Reset

### Clock Gating Control

The Tensix coprocessor supports dynamic clock gating for power management. Clock gating is controlled per-thread via the `ThreadConfig[3][THD_STATE_SIZE]` array (not via the soft reset register):

| Field Symbol | Purpose | Write Method |
| --- | --- | --- |
| `ThreadConfig[i][CG_CTRL_EN_Regblocks_ADDR32]` | Enable clock gating for specific register blocks (bitmask) | `SETC16` |
| `ThreadConfig[i][CG_CTRL_EN_Hyst_ADDR32]` | Configure hysteresis for clock gating transitions | `SETC16` |
| `ThreadConfig[i][CG_CTRL_KICK_Regblocks_ADDR32]` | Manually kick (wake) clock-gated blocks (bitmask) | `SETC16` |

These fields are written using `SETC16` instructions and **take effect immediately** upon write (no instruction pipeline delay). The clock gating mechanism automatically gates off clocks to idle functional units to reduce dynamic power consumption. The hysteresis configuration prevents excessive toggling when activity is bursty.

Thread index `i` selects:

*   `i=0`: Configuration for Tensix thread T0 (controlled by RISC-V T0)
*   `i=1`: Configuration for Tensix thread T1 (controlled by RISC-V T1)
*   `i=2`: Configuration for Tensix thread T2 (controlled by RISC-V T2)

### PRNG Seeding

Multiple hardware units in the Tensix coprocessor use Pseudo-Random Number Generators (PRNGs):

*   **Matrix Unit (FPU)**: For stochastic rounding in MVMUL operations
*   **Vector Unit (SFPU)**: Internal PRNG (reset via soft reset, not via this mechanism)
*   **Packers** (all 4): For stochastic rounding in early format conversion stage

All PRNGs (except SFPU) can be simultaneously reseeded by writing to the global configuration field:

This write can be performed via:

*   RISC-V `sw` instruction to `TENSIX_CFG_BASE + PRNG_SEED_Seed_Val_ADDR32 * 4`
*   Tensix `WRCFG` instruction

The write **takes effect immediately**, resynchronizing all PRNGs in the tile to a deterministic state based on the provided seed. Use cases:

*   Reproducible testing (same seed → same stochastic behavior across runs)
*   Initialization after soft reset or power-on
*   Changing randomization behavior between kernel phases

Note: The Vector Unit PRNG is reset only via bit 10 of `RISCV_DEBUG_REG_SOFT_RESET_0` (Vector Unit soft reset), not via this seed register.

### Instruction Cache Invalidation

Each RISC-V core has a small instruction cache (I-cache) to reduce L1 memory access latency during instruction fetch. The I-caches can be selectively invalidated without performing a full soft reset by writing to the global configuration field:

| Bit Position | Core | I-Cache Size | Notes |
| --- | --- | --- | --- |
| 0 | RISC-V B | 2 KiB | Main data movement orchestrator |
| 1 | RISC-V T0 | 2 KiB | UNPACK phase thread |
| 2 | RISC-V T1 | 0.5 KiB | MATH phase thread (smaller cache) |
| 3 | RISC-V T2 | 2 KiB | PACK phase thread |
| 4 | RISC-V NC | 0.5 KiB | Additional data movement (smaller cache) |

**Example**: To invalidate T0 and T1 caches only:

This write can be performed via RISC-V `sw` instruction or Tensix `WRCFG` instruction. Invalidation **takes effect immediately** and is necessary when:

*   Code in L1 has been modified by another agent (DMA, NoC write, another core)
*   After loading new kernel code into L1
*   Context switching between different code segments
*   Self-modifying code patterns

Note: Full soft reset (bits 11-14, 18) also invalidates I-caches but additionally resets all core state.

### Unpacker Context Counter Reset

Unpacker configuration context counters can be reset per-thread without full unpacker reset:

*   `ThreadConfig[i][UNPACK_MISC_CFG_CfgContextCntReset_0_ADDR32]`
*   `ThreadConfig[i][UNPACK_MISC_CFG_CfgContextCntReset_1_ADDR32]`
