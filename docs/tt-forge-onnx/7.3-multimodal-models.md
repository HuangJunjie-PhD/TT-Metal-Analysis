This document describes the support for multimodal models in the tt-forge-onnx compilation framework. Multimodal models process multiple input modalities (e.g., text and images) and include architectures like Stable Diffusion, LLaVA, Phi-3.5 Vision, and PaddleOCR.

For text-only language models, see [Text Models](https://deepwiki.com/tenstorrent/tt-forge-onnx/7.2-text-and-llm-models). For vision-only models, see [Vision Models](https://deepwiki.com/tenstorrent/tt-forge-onnx/7.1-vision-models).

## Purpose and Scope

Multimodal models in tt-forge-onnx handle inputs from multiple modalities (typically text and images) and perform tasks such as:

*   **Conditional generation**: Generating images from text prompts (Stable Diffusion)
*   **Visual question answering**: Answering questions about images (LLaVA)
*   **Multimodal causal language modeling**: Text generation conditioned on images (Phi-3.5 Vision)
*   **Optical character recognition**: Detecting and recognizing text in images (PaddleOCR)

All multimodal models follow the standard Forge compilation pipeline but require specialized input handling and model wrappers to accommodate multiple input types.

### Wrapper Class Pattern

Multimodal HuggingFace models accept `**kwargs` dictionaries and return structured output objects. Each model is wrapped in a `torch.nn.Module` subclass before compilation. The wrapper performs three functions:

1.   Converts positional tensor arguments into the keyword arguments expected by the underlying model
2.   Extracts a single tensor output (e.g., `output.logits`) from the model's structured return value
3.   Allows pre-compilation configuration (e.g., `return_dict=False`, `use_cache=False`)

**Wrapper implementations**

The `StableDiffusionWrapper` wraps `pipe.transformer` (an `SD3Transformer2DModel`) and maps the four positional arguments to the transformer's keyword arguments, returning only the first element `[0]` of the output tuple.

The LLaVA `Wrapper` packs inputs into a dictionary and returns `output.logits`, making the output a plain tensor suitable for verification.

The Phi-3.5 Vision `Wrapper` passes `None` for position IDs, past key values, and inputs embeddings, as these are unused in the test configuration.

### Key Compilation Steps

1.   **Property Recording**: `record_model_properties` captures metadata including `framework`, `model` (`ModelArch` enum), `variant`, `task`, `source`, `group`, and `priority`. This metadata is used for tracking and CI/CD organization. See page 3.5 for full details.

2.   **Test Skip Logic**: All multimodal tests call `pytest.xfail(reason="Requires multi-chip support")` because these models exceed single-chip memory capacity.

3.   **Model Loading**: Instantiate `ModelLoader` with the appropriate `ModelVariant` enum value and call `load_model()`. For Stable Diffusion 3.5, `dtype_override=torch.bfloat16` is passed.

4.   **Model Configuration**: Set model configuration flags before wrapping. For LLaVA and Phi-3.5 Vision: `model.config.return_dict = False` and `model.config.use_cache = False`.

5.   **Wrapper Creation**: Wrap the model in a custom `torch.nn.Module` subclass that provides a positional-argument forward signature compatible with `forge.compile`.

6.   **Input Preparation**: Call `loader.load_inputs()` to generate sample inputs. For Stable Diffusion 3.5, `dtype_override=torch.bfloat16` is passed. Inputs are returned as a list or dictionary depending on the loader.

7.   **Compiler Configuration**: For Stable Diffusion 3.5, create `CompilerConfig(default_df_override=DataFormat.Float16_b)` to use FP16 data format.

8.   **Compilation**: Call `forge.compile(framework_model, sample_inputs=inputs, module_name=module_name, compiler_cfg=compiler_cfg)`.

9.   **Verification**: Call `verify(inputs, framework_model, compiled_model)` to compare hardware outputs against the reference framework implementation using PCC (Pearson Correlation Coefficient) metrics.

## Stable Diffusion 3.5 Specifics

### Model Architecture

Stable Diffusion 3.5 uses a transformer-based denoising architecture. The test framework compiles only the transformer component:

*   **Transformer** (`pipe.transformer`): An `SD3Transformer2DModel` that performs the core denoising computation. This is the only component currently compiled.
*   **VAE Decoder**: Not yet compiled separately. A TODO comment at line 86 notes that post-processing using VAE decode should be implemented after obtaining the transformer output.

### Input Structure

| Input | Description | Shape |
| --- | --- | --- |
| `latent_model_input` | Noisy latent representation of the image | `[batch, channels, height, width]` |
| `timestep` | Current diffusion timestep (scalar) | `[1]` |
| `prompt_embeds` | Text prompt embeddings (encoder hidden states) | `[batch, seq_len, hidden_dim]` |
| `pooled_prompt_embeds` | Pooled text embeddings (pooled projections) | `[batch, pooled_dim]` |

### Data Type Configuration

Both model and inputs use `torch.bfloat16`:

*   `loader.load_model(dtype_override=torch.bfloat16)` at line 80
*   `loader.load_inputs(dtype_override=torch.bfloat16)` at line 81

The compiler is configured with `CompilerConfig(default_df_override=DataFormat.Float16_b)` at lines 89-90 to use FP16 representation in the compiled binary.

### Wrapper Implementation

The `StableDiffusionWrapper` class at lines 27-43 performs the following transformations:

1.   Accepts four positional arguments: `latent_model_input`, `timestep`, `prompt_embeds`, `pooled_prompt_embeds`
2.   Calls `self.model()` with keyword arguments: `hidden_states`, `timestep`, `encoder_hidden_states`, `pooled_projections`
3.   Passes additional configuration: `joint_attention_kwargs` and `return_dict` (both configurable via constructor)
4.   Returns only the first element `[0]` of the output tuple, which is the noise prediction

## LLaVA Specifics

### Model Architecture

LLaVA (`LlavaForConditionalGeneration`) is a multimodal model that combines vision and language understanding:

**LLaVA architecture components**

*   **Vision Encoder**: CLIP-based ViT that processes `pixel_values` into visual embeddings
*   **Projection Layer**: Linear layer that maps visual embeddings into the language model's embedding space
*   **Language Model**: Phi or Vicuna-based transformer that processes concatenated text and visual tokens

### Input Structure

| Input | Description | Source |
| --- | --- | --- |
| `input_ids` | Tokenized text input (questions/prompts) | `loader.load_inputs()["input_ids"]` |
| `attention_mask` | Padding mask for variable-length sequences | `loader.load_inputs()["attention_mask"]` |
| `pixel_values` | Preprocessed image tensor (normalized, resized) | `loader.load_inputs()["pixel_values"]` |

### Wrapper Implementation

The `Wrapper` class at lines 26-34 implements:

`def forward(self, input_ids, attention_mask, pixel_values):    inputs = {        "input_ids": input_ids,        "attention_mask": attention_mask,        "pixel_values": pixel_values    }    output = self.model(**inputs)    return output.logits`
This wrapper:

1.   Converts three positional arguments into a keyword argument dictionary
2.   Calls the underlying `LlavaForConditionalGeneration` model
3.   Extracts and returns only `output.logits` (the next-token prediction logits)

## Phi-3.5 Vision Specifics

### Model Architecture

`Phi3VForCausalLM` extends the Phi-3 language model with vision understanding capabilities:

**Phi-3.5 Vision architecture components**

*   **Image Encoder**: ViT-based sub-model that processes `pixel_values` into visual token embeddings
*   **Language Model**: Phi-3 transformer with cross-attention layers that process both text and visual tokens
*   **Variable Resolution**: `image_sizes` parameter enables handling of images with different dimensions

### Input Structure

| Input | Description | Purpose |
| --- | --- | --- |
| `input_ids` | Tokenized text input | Question or prompt tokens |
| `attention_mask` | Padding mask | Masks out padding tokens |
| `pixel_values` | Preprocessed image tensor | Image data (normalized, resized) |
| `image_sizes` | Original image dimensions | Enables variable-resolution processing |

### Model Configuration

Before wrapping, the model is configured at lines 58-59:

`model.config.return_dict = Falsemodel.config.use_cache = False`
*   `return_dict = False`: Forces the model to return tuples instead of dict-like objects
*   `use_cache = False`: Disables KV cache (required for compilation, see page 7.4)

### Wrapper Implementation

The `Wrapper` class at lines 25-31 implements:

`def forward(self, input_ids, attention_mask, pixel_values, image_sizes):    return self.model(input_ids, attention_mask, None, None, None, pixel_values, image_sizes)`
The wrapper passes `None` for three unused arguments:

1.   Position IDs (not needed with attention mask)
2.   Past key values (KV cache disabled)
3.   Inputs embeds (using `input_ids` instead)

### Verification

All three models use the default `verify` function:

`verify(inputs, framework_model, compiled_model)`
The verification process (see page 3.4):

1.   Runs the framework model on CPU/GPU with the given inputs
2.   Runs the compiled model on Tenstorrent hardware with the same inputs
3.   Compares outputs using PCC (Pearson Correlation Coefficient)
4.   Fails if PCC is below threshold (default: 0.99)

No custom verification configuration is used for these models. Verification is performed at lines:

*   Stable Diffusion 3.5: Line 98
*   LLaVA: Line 73
*   Phi-3.5 Vision: Line 75

## Current Limitations

All multimodal models currently have `pytest.xfail` markers due to:

1.   **Multi-chip requirement**: Models exceed single-chip memory capacity

    *   Stable Diffusion 3.5: All variants marked out of memory
    *   LLaVA 1.5-7B: Requires multi-chip support
    *   Phi-3.5 Vision: Requires multi-chip support

2.   **Model size**: Large parameter counts (typically 7B+) require distributed execution

3.   **Memory constraints**: High-resolution image processing and large embedding spaces
