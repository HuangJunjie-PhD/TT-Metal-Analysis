## Purpose and Scope

This page documents the IOMMU (Input/Output Memory Management Unit) implementation in Whisper, which provides address translation and protection for device memory accesses. The IOMMU translates I/O Virtual Addresses (IOVAs) from devices to System Physical Addresses (SPAs), enforcing access control and supporting virtualization through two-stage address translation.

The IOMMU is implemented in the `TT_IOMMU` namespace and follows the RISC-V IOMMU specification. It is an independent component that integrates with the system memory subsystem to translate device memory accesses.

For information about hart-side address translation (satp/hgatp), see [Virtual Memory and Address Translation](https://deepwiki.com/tenstorrent/whisper/4.2-virtual-memory-and-address-translation). For interrupt delivery mechanisms, see [Interrupt Controllers](https://deepwiki.com/tenstorrent/whisper/7.1-interrupt-controllers-(aclint-imsic-aplic)).

## High-Level Architecture

The IOMMU architecture consists of several major subsystems: memory-mapped configuration registers, translation context tables (Device and Process Directory Tables), command/fault/page-request queues, and the address translation engine with multiple stages.

**Diagram: IOMMU Component Architecture**

The `Iommu` class [iommu/Iommu.hpp 406-1300](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L406-L1300) is the main component. It provides:

1.   **Memory-mapped registers** at a configured base address for software control
2.   **Translation contexts** loaded from Device Directory Table (DDT) and Process Directory Table (PDT)
3.   **Address translation engine** supporting 1-stage, 2-stage, and MSI translations
4.   **Command queue processing** for cache invalidation and ATS operations
5.   **Fault/page-request queues** for error reporting and demand paging
6.   **Interrupt generation** via MSI or wired interrupts

**Sources:**[iommu/Iommu.hpp 406-1300](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L406-L1300)[iommu/Iommu.cpp 1-100](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1-L100)

## Device and Process Contexts

Device and Process contexts contain translation configuration for each device and process. They are loaded from memory on-demand and cached.

### Device Context Structure

**Diagram: Device Context Loading Flow**

The `loadDeviceContext()` method [iommu/Iommu.cpp 913-1029](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L913-L1029) performs a multi-level directory walk:

1.   **Cache lookup**: Check `ddtCache_`[iommu/Iommu.hpp 1192-1193](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L1192-L1193) for existing entry
2.   **Directory walk**: For each level from `levels-1` to 1: 
    *   Read DDTE at `base_addr + DDI[level] * 8`
    *   Verify DDTE.V bit and reserved fields
    *   Extract PPN for next level

3.   **Leaf read**: Read device context (32 or 64 bytes) at level 0
4.   **Validation**: Check `TC.V` and call `misconfiguredDc()`[iommu/Iommu.cpp 1158-1377](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1158-L1377)
5.   **Cache update**: Store in `ddtCache_` via `updateDdtCache()`[iommu/Iommu.cpp 2179-2208](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2179-L2208)

**Device Context Fields** (from `DeviceContext` class):

*   `tc`: Translation control (EN_ATS, EN_PRI, PDTV, etc.)
*   `iohgatp`: Second-stage (G-stage) page table pointer
*   `ta`: Translation attributes (RCID, MCID)
*   `fsc`: First-stage context (PDTP or IOSATP based on PDTV)
*   `msiptp`: MSI page table pointer (extended format only)

**Sources:**[iommu/Iommu.cpp 913-1029](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L913-L1029)[iommu/Iommu.hpp 1192-1193](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L1192-L1193)[iommu/DeviceContext.hpp 1-500](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/DeviceContext.hpp#L1-L500)

### Process Context Structure

**Diagram: Process Context Loading Flow**

The `loadProcessContext()` method [iommu/Iommu.cpp 1033-1154](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1033-L1154) walks the Process Directory Table:

1.   **Cache lookup**: Check `pdtCache_`[iommu/Iommu.hpp 1208-1209](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L1208-L1209)
2.   **PDT walk**: For each level determined by `dc.processTableLevels()`: 
    *   If `dc.iohgatp.MODE != Bare`, perform G-stage translation of directory address

3.   **Leaf read**: Read 16-byte process context
4.   **Validation**: Check `PC.TA.V` and call `misconfiguredPc()`[iommu/Iommu.cpp 1381-1421](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1381-L1421)

**Process Context Fields**:

*   `ta`: Translation attributes
*   `fsc`: First-stage context (IOSATP register value)

## Address Translation

The IOMMU performs multi-stage address translation from I/O Virtual Address (IOVA) to System Physical Address (SPA).

### Translation Overview

**Diagram: Complete Address Translation Flow**

The main `translate()` method [iommu/Iommu.cpp 1425-1579](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1425-L1579) orchestrates the translation:

1.   **Load Device Context**: Call `loadDeviceContext()` with device ID
2.   **Determine translation type**: 
    *   **Untranslated requests** (`UntransRead/Write/Exec`): Require stage-1 translation
    *   **Translated requests** (`TransRead/Write/Exec`): Already translated, may need G-stage only if `T2GPA=1`

3.   **Stage-1 (VS-stage)**: If needed, translate VA→GPA using: 
    *   Process context loaded via `loadProcessContext()` if `PDTV=1`
    *   Direct IOSATP from device context if `PDTV=0`

4.   **Stage-2 (G-stage)**: If `iohgatp.MODE != Bare`, translate GPA→SPA
5.   **MSI Translation**: If address matches MSI region, translate through MSI page table
6.   **PMP/PMA checks**: Verify physical memory protection and attributes

**Sources:**[iommu/Iommu.cpp 1425-1579](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1425-L1579)[iommu/Iommu.cpp 1581-1795](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1581-L1795)

### Stage-1 Translation

The `stage1Translate()` method [iommu/Iommu.cpp 1951-2046](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1951-L2046) performs VS-stage address translation:

1.   **Configure translation**: Call `stage1Config_` callback with IOSATP mode, ASID, PPN, SUM bit
2.   **Invoke page walk**: Call `stage1_` callback to perform Sv32/Sv39/Sv48/Sv57 page table walk
3.   **Hardware A/D updates**: Configure `setFaultOnFirstAccess_` based on `dc.sade()`
4.   **Return GPA**: On success, return guest physical address

The actual page table walk is delegated to the hart's virtual memory system via callbacks [iommu/Iommu.hpp 673-691](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L673-L691)

## Command Queue Processing

The IOMMU command queue allows software to issue commands for cache invalidation, ATS operations, and synchronization.

### Command Queue Architecture

**Diagram: Command Queue Processing**

The command queue is a circular buffer in memory. Software writes commands to the tail (`cqt_`), and the IOMMU processes from the head (`cqh_`). The `processCommandQueue()` method [iommu/Iommu.cpp 3094-3146](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L3094-L3146) is invoked when `cqt_` is written [iommu/Iommu.cpp 327-333](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L327-L333)

### IOTINVAL Commands

**Diagram: IOTINVAL Command Processing**

The `executeIotinvalCommand()` method [iommu/Iommu.cpp 3059-3092](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L3059-L3092) invalidates page table caches:

*   **IOTINVAL.VMA**: Invalidates first-stage (VS-stage) TLB entries

*   **IOTINVAL.GVMA**: Invalidates second-stage (G-stage) TLB entries

    *   `GV=1`: Guest-specific invalidation matching `GSCID`
    *   `PSCV` must be 0 for GVMA per specification

### IOFENCE.C Command

The `executeIofenceCCommand()` method [iommu/Iommu.cpp 2986-3056](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2986-L3056) provides synchronization:

1.   **Wait for ATS invalidations**: If any ITAG is busy, block until completions arrive
2.   **Check timeout**: If any invalidation timed out, set `cmd_to` error bit
3.   **Memory operations**: If `PR=1` or `PW=1`, ensure prior reads/writes complete
4.   **Write to memory**: If `AV=1`, write `DATA` to address `ADDR` (must be 4-byte aligned)
5.   **Interrupt**: If `WSI=1`, generate wired interrupt after completion

This command is used to ensure ordering between commands and memory operations.

**Sources:**[iommu/Iommu.cpp 2986-3056](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2986-L3056)[iommu/Ats.hpp 154-170](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Ats.hpp#L154-L170)

### ATS Commands

#### ATS.INVAL Command

**Diagram: ATS.INVAL Command Flow**

The `executeAtsInvalCommand()` method [iommu/Iommu.cpp 2621-2715](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2621-L2715) sends PCIe ATS Invalidation Requests:

1.   **ITAG allocation**: Find free ITAG from `itagBusy_` bitmap (32 ITAGs available)
2.   **Build message**: Construct invalidation request with routing ID, PASID, address
3.   **Send to device**: Call `sendInvalReq_` callback with invalidation parameters
4.   **Track completion**: Mark ITAG as busy and wait for device completion

The device must respond by calling `atsInvalidationCompletion()`[iommu/Iommu.cpp 2719-2755](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2719-L2755) to clear the ITAG.

**Sources:**[iommu/Iommu.cpp 2621-2715](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2621-L2715)[iommu/Iommu.cpp 2719-2755](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2719-L2755)[iommu/Ats.hpp 96-115](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Ats.hpp#L96-L115)

#### ATS.PRGR Command

The `executeAtsPrgrCommand()` method [iommu/Iommu.cpp 2758-2764](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2758-L2764) sends Page Request Group Responses:

1.   **Extract fields**: Get RID, DSEG, PID, PRGI, response code
2.   **Send response**: Call `sendPrgr_` callback to deliver response to device
3.   **Response codes**: 
    *   `SUCCESS (0x0)`: Pages successfully made resident
    *   `INVALID (0x1)`: Pages don't exist or access denied
    *   `FAILURE (0xF)`: Catastrophic error

This responds to page request messages received via `atsPageRequest()`[iommu/Iommu.cpp 2584-2618](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2584-L2618)

## Fault and Page Request Handling

The IOMMU reports translation faults and page requests through dedicated queues in memory.

### Fault Queue

**Diagram: Fault Queue Write Flow**

The `writeFaultRecord()` method [iommu/Iommu.cpp 2393-2478](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2393-L2478) writes faults to the fault queue:

1.   **Check capacity**: If queue is full (`fqFull()`), set `fqmf` error bit
2.   **Build record**: Create 32-byte `FaultRecord`[iommu/FaultQueue.hpp 100-200](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/FaultQueue.hpp#L100-L200) with: 
    *   `cause`: Fault cause code (e.g., 257=DDT load fault, 260=page fault)
    *   `iotval`, `iotval2`: Faulting address and additional info
    *   `ttyp`: Transaction type (read/write/exec)
    *   `DID`: Device ID
    *   `PID`: Process ID (if valid)

3.   **Write to memory**: Write record to `fqb_.PPN * pageSize + fqt_ * 32`
4.   **Advance tail**: Increment `fqt_` modulo queue size
5.   **Signal interrupt**: Call `updateIpsr(NewFault)` and `signalInterrupt(fiv)`

**Sources:**[iommu/Iommu.cpp 2393-2478](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2393-L2478)[iommu/FaultQueue.hpp 100-200](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/FaultQueue.hpp#L100-L200)

### Page Request Queue

The page request queue handles PCIe ATS Page Request messages when devices need demand paging:

**Diagram: Page Request Queue Write**

The `atsPageRequest()` method [iommu/Iommu.cpp 2584-2618](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2584-L2618) receives page requests from devices:

1.   **Check capacity**: Verify queue is not full
2.   **Write request**: Write 16-byte `PageRequest`[iommu/Iommu.hpp 332-357](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L332-L357) containing: 
    *   `DID`: Device ID
    *   `PID`: Process ID
    *   `address`: Requested page address
    *   `r`, `w`, `l`: Read, write, last-in-group flags
    *   `prgi`: Page request group index

3.   **Signal interrupt**: Notify software via interrupt

Software responds by faulting in the page and sending an `ATS.PRGR` command.

## ATS (Address Translation Services) Protocol

ATS enables PCIe devices to cache translated addresses and request translations directly.

### ATS Translation Request

**Diagram: ATS Translation Request Flow**

The `atsTranslate()` method [iommu/Iommu.cpp 2481-2581](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2481-L2581) handles ATS Translation Requests:

1.   **Check ATS enabled**: Verify `dc.tc.EN_ATS == 1` in device context
2.   **Perform translation**: 
    *   If `T2GPA=1`: Translate to GPA (hypervisor containment)
    *   Otherwise: Translate to SPA

3.   **Build response**: Create `AtsResponse`[iommu/Iommu.hpp 620-633](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L620-L633) with: 
    *   `success`: True if translation succeeded
    *   `translatedAddr`: Resulting PA or GPA
    *   `readPerm`, `writePerm`, `execPerm`: Access permissions from PTE
    *   `privMode`: Privilege bit
    *   `untranslatedOnly`: U bit for MRIF mode

4.   **Count event**: Call `countEvent(HpmEventId::AtsTransReq)`

The device caches the translation and uses it for future memory accesses.

**Sources:**[iommu/Iommu.cpp 2481-2581](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L2481-L2581)[iommu/Iommu.hpp 620-633](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L620-L633)

### T2GPA Mode

When `dc.tc.T2GPA=1`, ATS translations return Guest Physical Addresses instead of System Physical Addresses. This enables hypervisor containment where the hypervisor controls the final GPA→SPA mapping.

The `t2gpaTranslate()` method [iommu/Iommu.cpp 1797-1875](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1797-L1875) performs this translation:

*   Executes stage-1 translation to get GPA
*   **Does not** perform stage-2 translation
*   Returns GPA in the response
*   Sets `untranslatedOnly=1` to prevent device from using address for DMA

**Sources:**[iommu/Iommu.cpp 1797-1875](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L1797-L1875)

## Interrupt Generation

The IOMMU generates interrupts for command completion, faults, page requests, and counter overflows.

### Interrupt Modes

**Diagram: Interrupt Generation Architecture**

The IOMMU supports two interrupt modes:

**MSI Mode** (`capabilities.igs != WSI`):

*   Uses MSI configuration table with 16 entries [iommu/Iommu.hpp 315-322](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L315-L322)
*   Each interrupt source (CQ, FQ, PQ, HPM) uses a configurable vector from `icvec_`
*   `sendMsi()`[iommu/Iommu.cpp 720-734](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L720-L734) writes `msi_data` to `msi_addr`
*   Supports per-vector masking via `msi_vec_ctl` register

**WSI Mode** (`fctl.wsi=1` when `capabilities.igs` allows):

*   Uses wired interrupts through callback
*   `signalWiredInterrupt_()` callback [iommu/Iommu.hpp 734-735](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.hpp#L734-L735) delivers to external interrupt controller
*   Four separate interrupt lines for CQ, FQ, PQ, HPM

The `updateIpsr()` method [iommu/Iommu.cpp 747-783](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L747-L783) checks conditions and calls `signalInterrupt()`[iommu/Iommu.cpp 696-717](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L696-L717) when appropriate.

## Debug Translation Interface

The IOMMU provides a debug interface for software to test translations without triggering side effects.

**Diagram: Debug Translation Flow**

The debug translation interface [iommu/Iommu.cpp 558-610](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L558-L610) allows testing translations:

1.   **Write IOVA**: Software writes virtual address to `tr_req_iova_`[iommu/Iommu.cpp 518-531](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L518-L531)
2.   **Write control**: Write `tr_req_ctl_` with device ID, process ID, access type, and set `go_busy=1`[iommu/Iommu.cpp 534-555](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L534-L555)
3.   **Process translation**: On `go_busy` 0→1 transition, call `processDebugTranslation()`[iommu/Iommu.cpp 558-610](https://github.com/tenstorrent/whisper/blob/733fd921/iommu/Iommu.cpp#L558-L610)

    *   `fault`: 1 if translation failed
    *   `ppn`: Translated physical page number
    *   `pbmt`: Page-based memory type

5.   **Clear busy**: Set `go_busy=0` to signal completion

This interface is useful for IOMMU driver development and debugging.

*   `setStage1ConfigCb()`: Configure hart's IOSATP-equivalent registers
*   `setStage2ConfigCb()`: Configure hart's IOHGATP-equivalent registers

*   `setMemReadCb()`: Read physical memory with PMP/PMA checks
*   `setMemWriteCb()`: Write physical memory with PMP/PMA checks
*   `setIsReadableCb()`: Check if address is readable

**Device Communication Callbacks**:

*   `setSendInvalReqCb()`: Send ATS invalidation to device
*   `setSendPrgrCb()`: Send page request group response to device

**Interrupt Callback**:

*   `setSignalWiredInterruptCb()`: Signal wired interrupt (WSI mode)

## Summary

The IOMMU implementation provides comprehensive I/O address translation and protection for RISC-V systems:

*   **Memory-mapped registers** control IOMMU operation and configuration
*   **Two-level context tables** (DDT and PDT) store per-device and per-process translation configuration with caching
*   **Multi-stage translation** supports VS-stage, G-stage, and MSI address translation
*   **Command queue** processes cache invalidation and ATS commands
*   **Fault and page request queues** handle errors and demand paging
*   **ATS protocol support** enables device-side translation caching
*   **Performance monitoring** tracks translation behavior with 32 configurable counters
*   **PMP/PMA integration** enforces physical memory protection and attributes
*   **Flexible interrupt generation** via MSI or wired interrupts

The implementation follows the RISC-V IOMMU specification and integrates with Whisper's memory and interrupt subsystems through a callback-based interface.
