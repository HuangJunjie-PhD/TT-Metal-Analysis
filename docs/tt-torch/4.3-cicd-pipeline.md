## Purpose and Scope

This document describes the GitHub Actions-based CI/CD pipeline that automates building, testing, and validating tt-torch across multiple hardware architectures and test configurations. The pipeline handles Docker image management, parallel test execution, artifact generation, and result reporting. For information about the test framework itself, see [Testing Framework](https://deepwiki.com/tenstorrent/tt-torch/4.2-testing-framework). For build system details, see [Build System](https://deepwiki.com/tenstorrent/tt-torch/4.1-build-system).

## Docker Image Management

The CI/CD system uses a content-addressed Docker image caching strategy to avoid rebuilding images when dependencies haven't changed.

### Docker Tag Computation

**Diagram: Docker Tag Derivation**

The tag computation script [.github/get-docker-tag.sh 1-86](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/get-docker-tag.sh#L1-L86) performs:

1.   **Extract TT_MLIR_VERSION**: Reads from `third_party/CMakeLists.txt` or `tt-xla`'s CMakeLists
2.   **Clone tt-mlir**: Checks out exact version specified
3.   **Get MLIR docker tag**: Executes tt-mlir's tag script
4.   **Hash Dockerfiles**: SHA256 of base and CI Dockerfiles
5.   **Combine hashes**: Final tag is `dt-{combined_hash}`

**Build Process:**

The [.github/workflows/build-image.yml 1-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/build-image.yml#L1-L121) workflow:

1.   **Check existence**: Uses `docker manifest inspect` to check if image exists [.github/workflows/build-image.yml 26-55](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/build-image.yml#L26-L55)
2.   **Build if missing**: Builds base and CI images sequentially [.github/workflows/build-image.yml 78-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/build-image.yml#L78-L121)
3.   **Tag as latest**: Tags image as `latest` when pushed to main [.github/workflows/build-image.yml 58-76](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/build-image.yml#L58-L76)

Sources: [.github/get-docker-tag.sh 1-86](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/get-docker-tag.sh#L1-L86)[.github/build-docker-images.sh 1-50](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/build-docker-images.sh#L1-L50)[.github/workflows/build-image.yml 1-121](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/build-image.yml#L1-L121)

## Summary

The tt-torch CI/CD pipeline provides:

*   **Multi-stage workflow orchestration**: Pre-commit checks, Docker builds, compilation, and testing
*   **Hardware matrix testing**: Parallel execution across Wormhole, Blackhole, N300, and specialized hardware
*   **Intelligent artifact management**: Content-addressed Docker images, build caching, and MLIR dumps
*   **Comprehensive coverage**: Unit tests, full model execution, op-by-op validation, and performance benchmarks
*   **Flexible configuration**: Workflow inputs for MLIR overrides, backend selection, and artifact control
*   **Robust reporting**: Codecov integration, JUnit XML reports, and model test documentation generation

The system balances fast PR feedback (120-minute workflows) with comprehensive nightly validation (340+ minutes), ensuring code quality while supporting development velocity.
