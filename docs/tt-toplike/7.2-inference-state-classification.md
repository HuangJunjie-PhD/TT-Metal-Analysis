# Inference State Classification

Relevant source files
*   [src/workload/inference.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs)
*   [src/workload/serving.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs)

The `InferenceEngine` is responsible for transforming raw telemetry data (power, temperature, clock speeds) into high-level, human-readable hardware states. It uses a combination of rolling-window statistics, trend analysis, and external process probing to classify Tenstorrent device activity into one of 13 distinct states.

## Inference Engine Architecture

The classification logic resides in `src/workload/inference.rs`. The engine maintains a rolling history of 30 samples per device to calculate variances and slopes, allowing it to distinguish between steady-state workloads (like LLM decoding) and bursty workloads (like LLM prefill).

### Data Flow and Ingestion

Telemetry is fed into the engine via the `ingest` method. Each sample is stored in a `VecDeque` within a `HashMap` keyed by device index.

1.   **Baseline Update**: The engine updates an `AdaptiveBaseline`[src/workload/inference.rs 163](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L163-L163) which learns the "idle" characteristics of the device over the first 20 samples.
2.   **History Management**: Samples are pushed to a 30-sample rolling window [src/workload/inference.rs 169-172](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L169-L172)
3.   **Health Tracking**: The engine monitors ARC (Active Resource Controller) health counters. If counters remain identical for ~30 consecutive frames (~3 seconds), a `Stalled` state is flagged [src/workload/inference.rs 130-132](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L130-L132)

## Device Inference States

The `DeviceInferenceState` enum defines 13 evocative states. These are prioritized such that critical hardware issues (Faults, Thermal Stress) override operational classifications (Thinking, Generating).

| State | Label | Criteria |
| --- | --- | --- |
| `Stalled` | **GONE SILENT** | ARC counters frozen or heartbeat is 0 [src/workload/inference.rs 42](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L42-L42) |
| `HardwareFault` | **IN DISTRESS** | Faults register is non-empty [src/workload/inference.rs 43](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L43-L43) |
| `ThermalStress` | **SCORCHING** | Temperature exceeds 82°C [src/workload/inference.rs 44](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L44-L44) |
| `Throttling` | **REINED IN** | Throttler active or AICLK dropped >20% under load [src/workload/inference.rs 45](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L45-L45) |
| `MaxedOut` | **BURNING BRIGHT** | Power > 85% TDP or power_change > 1.5 [src/workload/inference.rs 47](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L47-L47) |
| `Thinking` | **DREAMING DEEP** | High-variance power (typical of LLM prefill/attention) [src/workload/inference.rs 50](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L50-L50) |
| `Generating` | **CONJURING** | Low-variance, high current-to-power ratio (LLM decode) [src/workload/inference.rs 51](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L51-L51) |
| `MovingData` | **IN FULL FLOW** | Current change significantly exceeds power change [src/workload/inference.rs 52](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L52-L52) |
| `Idle` | **BREATHING** | Telemetry within 15% of learned baseline [src/workload/inference.rs 53](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L53-L53) |

## Power Trend Analysis

The `PowerTrend` struct is calculated from the rolling history to provide context for the classifier.

*   **Slope**: Measured in Watts per second (W/s). Positive values indicate rising activity [src/workload/inference.rs 28](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L28-L28)
*   **Variance**: Standard deviation of power over the last 10 samples. High variance distinguishes "Thinking" from "Generating" [src/workload/inference.rs 30](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L30-L30)
*   **Monotonicity**: `is_rising` and `is_falling` flags are set if the last 8 samples move strictly in one direction [src/workload/inference.rs 31-34](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/inference.rs#L31-L34)

## Inference Server Detection

While the `InferenceEngine` classifies hardware behavior, the `InferenceServerProbe` identifies the specific software stack driving the workload.

### Detection Pipeline

The probe scans the system for known Tenstorrent-compatible inference servers:

1.   **Process Scanning**: Uses `ProcessMonitor` to find candidates like `tt-inference-server`, `vLLM` (with `VLLM_TARGET_DEVICE=tt`), or `prompt_server.py`[src/workload/serving.rs 12-13](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L12-L13)
2.   **Port Discovery**: Extracts the listening port from command-line arguments or by decoding `/proc/[PID]/net/tcp` (little-endian hex) [src/workload/serving.rs 14-15](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L14-L15)
3.   **Health & Metrics Probing**: 
    *   `GET /health`: Checks if the model is loaded and ready [src/workload/serving.rs 16](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L16-L16)
    *   `GET /metrics`: Parses Prometheus-format metrics to extract token throughput (TPS) and KV cache utilization [src/workload/serving.rs 17](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L17-L17)

4.   **Environment Inspection**: Reads `MESH_DEVICE` and `VLLM_TARGET_DEVICE` from `/proc/[PID]/environ`[src/workload/serving.rs 164-179](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L164-L179)

### Serving Metrics Collected

The `ServingMetrics` struct aggregates data from multiple sources to provide a complete picture of the serving state:

*   **Throughput**: `generation_tps` and `prompt_tps` calculated via counter deltas [src/workload/serving.rs 74-77](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L74-L77)
*   **Resource Usage**: `kv_cache_utilization` and `prefix_cache_hit_rate`[src/workload/serving.rs 84-87](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L84-L87)
*   **Latency**: p50 and p99 Time-To-First-Token (TTFT) [src/workload/serving.rs 88-91](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L88-L91)
*   **Logs**: The probe attempts to locate the server's log file by scanning `/proc/[PID]/fd/` for symlinks [src/workload/serving.rs 18-19](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/workload/serving.rs#L18-L19)
