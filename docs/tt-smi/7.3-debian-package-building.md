# Debian Package Building

## Purpose and Scope

This document describes the Debian package building system for TT-SMI. It covers the GitHub Actions workflow that produces signed `.deb` packages for multiple Ubuntu distributions, the package configuration files in the `debian/` directory, and the integration with the release workflow. For information about the overall release process, see [GitHub Release Workflow](https://deepwiki.com/tenstorrent/tt-smi/7.2-github-release-workflow). For Python wheel building and PyPI distribution, see [Distribution Channels](https://deepwiki.com/tenstorrent/tt-smi/7.4-distribution-channels).

* * *

## Build Workflow Architecture

TT-SMI uses the `build-debian.yml` GitHub Actions workflow to produce Debian packages for multiple Ubuntu distributions. The workflow supports both manual dispatch and integration as a callable workflow from the main release process.

**Sources:**[.github/workflows/build-debian.yml 1-179](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L1-L179)

The workflow implements a matrix strategy to build packages for three Ubuntu versions in parallel. Each job runs independently and produces distribution-specific artifacts.

* * *

## Workflow Inputs and Version Handling

The workflow accepts several inputs that control the build process:

| Input | Required | Type | Purpose |
| --- | --- | --- | --- |
| `ref` | Yes | string | Git reference to build from (default: `github.ref`) |
| `MAJOR` | No | string | Major version component |
| `MINOR` | No | string | Minor version component |
| `PATCH` | No | string | Patch version component |
| `NUMBER_OF_COMMITS_SINCE_TAG` | No | string | Commit count since last tag |

**Sources:**[.github/workflows/build-debian.yml 6-23](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L6-L23)

Version extraction occurs in two stages:

1.   **Tag-based extraction** using `damienaicheh/extract-version-from-tag-action@v1.3.0`[.github/workflows/build-debian.yml 80-81](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L80-L81)
2.   **Input parameter override** where explicitly passed version components take precedence [.github/workflows/build-debian.yml 84-116](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L84-L116)

The final version is used in the `gbp dch` command to generate the Debian changelog with the correct version number [.github/workflows/build-debian.yml 120-124](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L120-L124)

## Build Environment Setup

### System Dependencies

The workflow installs a comprehensive set of Debian build tools:

**Sources:**[.github/workflows/build-debian.yml 52-66](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L52-L66)

Key components:

*   **`git-buildpackage`**: Debian native packaging tool that integrates with Git
*   **`pybuild-plugin-pyproject`**: Enables building Python packages using `pyproject.toml`
*   **`dh-sequence-python3`**: Debhelper sequence for Python 3 packages
*   **`python3-tomli`**: TOML parser required for reading `pyproject.toml`

### GPG Key Management

Package signing is performed using a GPG key stored in GitHub Secrets:

**Sources:**[.github/workflows/build-debian.yml 67-73](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L67-L73)[.github/workflows/build-debian.yml 133](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L133-L133)

The GPG key fingerprint is captured as an output from the import action and passed to the build process via the `DEBSIGN_KEYID` environment variable [.github/workflows/build-debian.yml 133](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L133-L133)

## Package Configuration

### Control File Structure

The `debian/control` file defines package metadata and dependencies:

**Sources:**[debian/control 1-40](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/control#L1-L40)

Key aspects:

*   **Architecture: all**[debian/control 20](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/control#L20-L20) indicates the package contains only Python code and data, no compiled binaries
*   **Tenstorrent-specific dependencies** include `python3-pyluwen` and `python3-tt-tools-common`[debian/control 31-32](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/control#L31-L32)
*   **Variable substitutions**`${misc:Depends}` and `${python3:Depends}` allow debhelper to inject computed dependencies [debian/control 22-23](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/control#L22-L23)

### Build Rules

The `debian/rules` file specifies how to build the package:

`#!/usr/bin/make -f %:	dh $@ \		--buildsystem=pybuild \		--with=python3 override_dh_auto_test:	# Skip tests during package build`
**Sources:**[debian/rules 1-12](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/rules#L1-L12)

This minimal configuration:

*   Uses **pybuild** as the build system [debian/rules 5](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/rules#L5-L5) which automatically handles Python package building from `pyproject.toml`
*   Integrates with Python 3 debhelper sequences [debian/rules 6](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/rules#L6-L6)
*   **Skips tests** during package build [debian/rules 9-10](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/rules#L9-L10) to avoid test failures in the build environment

## Build Process Pipeline

The build process follows a structured pipeline:

**Sources:**[.github/workflows/build-debian.yml 74-178](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L74-L178)

### Changelog Generation

The `gbp dch` command generates Debian changelog entries:

*   **`--debian-branch ${ref_name}`**: Specifies which Git branch to use [.github/workflows/build-debian.yml 121](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L121-L121)
*   **`-R`**: Release mode, marks the version as released [.github/workflows/build-debian.yml 122](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L122-L122)
*   **`-N ${MAJOR}.${MINOR}.${PATCH}`**: Sets the new version number [.github/workflows/build-debian.yml 123](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L123-L123)
*   **`--spawn-editor=never`**: Prevents interactive editor from launching [.github/workflows/build-debian.yml 124](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L124-L124)

Environment variables set the committer identity:

*   `EMAIL: releases@tenstorrent.com`
*   `NAME: Tenstorrent Releases`

**Sources:**[.github/workflows/build-debian.yml 120-127](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L120-L127)

### Package Building

The `gbp buildpackage` command performs the actual build:

*   **`--git-ignore-new`**: Ignores uncommitted changes in the Git repository [.github/workflows/build-debian.yml 130](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L130-L130)
*   **`DEBSIGN_KEYID`** environment variable provides the GPG key fingerprint for signing [.github/workflows/build-debian.yml 133](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L133-L133)

**Sources:**[.github/workflows/build-debian.yml 129-133](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L129-L133)

The build process:

1.   Reads package configuration from `debian/control`
2.   Executes build rules from `debian/rules`
3.   Uses pybuild to build the Python package from `pyproject.toml`
4.   Creates the `.deb` binary package
5.   Signs the package and generates `.changes` file

## Integration with Release Workflow

The Debian build workflow integrates with the main release workflow as a callable workflow:

**Sources:**[.github/workflows/build-debian.yml 5-23](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L5-L23)

The callable workflow pattern:

*   Receives version information from the parent release workflow
*   Executes builds in parallel for all Ubuntu distributions
*   Returns artifacts that can be attached to the GitHub release

This ensures consistent versioning across all build artifacts and maintains a single source of truth for release versions.

## Build System Dependencies

The Debian package build relies on a specific set of tools and their relationships:

**Sources:**[debian/control 1-40](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/control#L1-L40)[debian/rules 1-12](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/debian/rules#L1-L12)[.github/workflows/build-debian.yml 54-66](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml#L54-L66)

The integration points:

*   **git-buildpackage** bridges Git version control with Debian packaging conventions
*   **pybuild-plugin-pyproject** enables direct building from `pyproject.toml` without requiring a separate `setup.py`
*   **debhelper** sequences automate the build process based on `debian/rules`
*   **debsign** integrates with the build process through the `DEBSIGN_KEYID` environment variable

This architecture allows TT-SMI to maintain a single source of truth for package configuration (`pyproject.toml`) while generating proper Debian packages with all required metadata and signatures.
