## Overview

The RISC-V Ocelot parameterization system enables the configuration of different processor variants with specific characteristics. This system allows users to customize various aspects of the processor, including pipeline width, execution resources, cache configurations, and branch prediction capabilities, without modifying the core implementation code.

This document explains how the parameterization system is structured, how predefined configurations work, and how to create custom configurations. For information about building the system, see [Building and Configuration](https://deepwiki.com/tenstorrent/riscv-ocelot/6-building-and-configuration).

Sources: [src/main/scala/common/config-mixins.scala 22-336](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/config-mixins.scala#L22-L336)

## Configuration Framework

The RISC-V Ocelot uses the Chipyard/Rocket Chip configuration system based on the Parameters framework. This allows for modular, composable configurations through a technique called "cake pattern" or "mixin composition".

### Config Classes and Fragments

Configurations are built by composing multiple configuration fragments. Each fragment typically modifies a specific aspect of the processor by manipulating parameters.

Sources: [src/main/scala/common/config-mixins.scala 26-107](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/config-mixins.scala#L26-L107)

The RISC-V Ocelot provides several predefined core configurations representing different performance and area points. Each configuration defines a complete set of parameters for a specific BOOM core variant.

## Component-Specific Parameters

Each major component in the BOOM core has its own parameter class that controls its behavior and implementation details.

### Branch Predictor Parameters

Branch predictors are highly configurable, with different types available:

The configuration allows selecting from different branch predictor types:

*   TAGE-L: Tournament Adaptive GEometric history length predictor with Loop predictor
*   Gshare: Global history-based predictor
*   Alpha21264-style: Tournament predictor used in the Alpha 21264 processor
*   Software-based: For custom branch prediction implementations

### Cache Parameters

Caches are configured via `DCacheParams` and `ICacheParams`:

The cache parameters scale with core size:

*   Small BOOM: 4-way caches with 2 MSHRs
*   Medium BOOM: 4-way caches with 2 MSHRs
*   Large BOOM: 8-way caches with 4 MSHRs
*   Mega BOOM: 8-way caches with 8 MSHRs

## Creating Custom Configurations

Custom configurations can be created by composing configuration fragments to achieve the desired processor characteristics.

### Configuration Composition

Configurations are composed by applying multiple fragments together, with each subsequent fragment potentially overriding settings from previous ones:

### Example: Custom Configuration

To create a custom configuration, combine existing fragments and override specific parameters:

`// Example of custom configuration (not actual code)class MyCustomBoomConfig extends Config(  new WithNLargeBooms(1) ++            // Start with Large BOOM base  new WithVector(3) ++                 // Add vector extension with 3-wide dispatch  new WithNBoomPerfCounters(8) ++      // Increase performance counters  new WithSynchronousBoomTiles ++      // Use synchronous crossing  new WithBoomCommitLogPrintf ++       // Enable commit log printing  new Config((site, here, up) => {    case TilesLocated(InSubsystem) => up(TilesLocated(InSubsystem), site) map {      case tp: BoomTileAttachParams => tp.copy(tileParams = tp.tileParams.copy(core = tp.tileParams.core.copy(        // Custom overrides        numLdqEntries = 32,            // Increase load queue entries        numStqEntries = 32,            // Increase store queue entries        numFetchBufferEntries = 28     // Custom fetch buffer size      )))      case other => other    }  }))`

## Vector Extension Configuration

Vector processing capabilities can be added through the `WithVector` configuration fragment:

The vector extension parameters include:

*   Vector register length (`setvLen`): 256 bits by default
*   Memory data width: matches the DCache row bits
*   Dedicated vector issue queue with 32 entries

Sources: [src/main/scala/common/config-mixins.scala 91-105](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/config-mixins.scala#L91-L105)

## Parameter Implementation

The parameterization system is implemented using Scala case classes that hold configuration values, and Config classes that modify these parameters based on a hierarchical lookup system.

Each module or bundle in the BOOM core inherits from either `BoomModule` or `BoomBundle`, which provides access to the full set of core parameters.

Sources: [src/main/scala/common/types.scala 1-23](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/common/types.scala#L1-L23)

## Conclusion

The RISC-V Ocelot parameterization system provides a flexible framework for configuring processors with different performance, area, and power characteristics. By composing configuration fragments, users can create customized processor variants tailored to specific applications without modifying the core implementation code.

For information on how to build the system with your chosen configuration, see [Building and Configuration](https://deepwiki.com/tenstorrent/riscv-ocelot/6-building-and-configuration). For details on the continuous integration system used to test different configurations, see [Continuous Integration](https://deepwiki.com/tenstorrent/riscv-ocelot/6.2-continuous-integration).
