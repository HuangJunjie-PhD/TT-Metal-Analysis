# Reporting Issues

Relevant source files
*   [CONTRIBUTING.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1)

## Purpose and Scope

This page guides users through the process of reporting bugs, requesting features, asking questions, and providing feedback via GitHub issues. It covers what information to include in issue reports and best practices for effective communication with the development team.

For information about the overall contribution model and why pull requests are not accepted, see [Contribution Process](https://deepwiki.com/tenstorrent/ttsim/5.1-contribution-process). For reporting security vulnerabilities through private channels, see [Security Vulnerabilities](https://deepwiki.com/tenstorrent/ttsim/5.3-security-vulnerabilities). If you need help understanding error messages before reporting them, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference).

## Issue Types and When to Report

### Bug Reports

Report a bug when ttsim exhibits unexpected behavior, crashes, or produces incorrect results. Common scenarios include:

*   Simulator crashes or hangs during execution
*   Numerical results that differ unexpectedly from expected values
*   Error messages that indicate incorrect simulator behavior
*   Configuration failures despite following documented steps
*   ISA violations or undefined behavior errors that seem incorrect

### Feature Requests

Request a feature when you need functionality that ttsim does not currently provide. Examples include:

*   Support for additional Tensix operations
*   Improved error diagnostics
*   Performance optimizations
*   Additional configuration options
*   Enhanced debugging capabilities

### Questions

Ask questions when the existing documentation does not clarify how to use ttsim or interpret its behavior. Appropriate questions include:

*   How to configure ttsim for specific use cases
*   Interpretation of error messages (after checking [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference))
*   Expected behavior for edge cases
*   Compatibility questions with TT-Metalium versions

### General Feedback

Provide feedback on documentation quality, user experience issues, or suggestions for improvement that don't fit cleanly into the above categories.

## Required Information for Issue Reports

### For Bug Reports

When reporting a bug, include the following information to enable effective triage and reproduction:

**Diagram: Required Information for Bug Reports**

#### Reproduction Steps Template

Provide step-by-step instructions that allow developers to reproduce the issue:

```
1. Set environment variables:
   - TT_METAL_HOME=/path/to/tt-metal
   - TT_METAL_SIMULATOR=/path/to/libttsim_wh.so
   - TT_METAL_SLOW_DISPATCH_MODE=1

2. Navigate to directory: cd $TT_METAL_HOME/tests/ttnn/unit_tests

3. Execute command: pytest test_eltwise_binary.py::test_add

4. Observe error: [paste error message or unexpected output]
```

#### System Information Collection

Collect system information using these commands:

| Information | Command |
| --- | --- |
| Linux distribution | `cat /etc/os-release` |
| Kernel version | `uname -r` |
| Architecture | `uname -m` |
| Environment variables | `env | grep TT_METAL` |
| Simulator file info | `ls -la $TT_METAL_SIMULATOR` |
| SOC descriptor location | `find $TT_METAL_HOME -name "*_arch.yaml"` |

**Sources:**[CONTRIBUTING.md 22-26](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L22-L26)

### For Feature Requests

When requesting a feature, provide context about the use case and desired behavior:

**Diagram: Feature Request Structure**

Include:

*   **Use case**: What are you trying to accomplish?
*   **Motivation**: Why is this feature needed? What problem does it solve?
*   **Proposed solution**: How do you envision the feature working?
*   **Alternatives**: What workarounds have you tried?
*   **Impact**: How would this feature improve your workflow?

**Sources:**[CONTRIBUTING.md 15-20](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L15-L20)

### For Questions

When asking questions, provide context about what you've already tried:

*   **Question**: State your question clearly
*   **Context**: What are you trying to accomplish?
*   **Attempted solutions**: What have you tried so far?
*   **Documentation consulted**: Which wiki pages or README sections have you reviewed?
*   **Related configuration**: Relevant environment variables and setup

**Sources:**[CONTRIBUTING.md 15-20](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L15-L20)

## Issue Reporting Workflow

**Diagram: Issue Reporting and Processing Workflow**

### Step-by-Step Process

1.   **Search existing issues** at `https://github.com/tenstorrent/ttsim/issues`

    *   Use search keywords related to your problem
    *   Check both open and closed issues
    *   Review similar issues to see if your case is covered

2.   **Determine issue type**

    *   If security vulnerability → use private Security Advisory (see [Security Vulnerabilities](https://deepwiki.com/tenstorrent/ttsim/5.3-security-vulnerabilities))
    *   If Code of Conduct violation → email `opensource@tenstorrent.com`
    *   Otherwise → proceed to create public issue

3.   **Create new issue**

    *   Navigate to `https://github.com/tenstorrent/ttsim/issues/new`
    *   Choose appropriate issue type if templates are available
    *   Fill in all required information sections

4.   **Provide complete information**

    *   Follow the templates in the "Required Information" section above
    *   Attach logs, error messages, or minimal reproduction cases if applicable
    *   Use code blocks for formatting command output and error messages

5.   **Monitor issue for responses**

    *   Watch for questions from maintainers
    *   Provide additional information promptly when requested
    *   Test proposed fixes or workarounds and report results

**Sources:**[CONTRIBUTING.md 28-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L28-L29)[CONTRIBUTING.md 13-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L13-L34)

## Best Practices

### Before Reporting

**Diagram: Pre-Report Checklist**

Perform these checks before reporting an issue:

| Check | Description | Reference |
| --- | --- | --- |
| Search existing issues | Verify your issue hasn't been reported | [CONTRIBUTING.md 28-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L28-L29) |
| Review documentation | Check README and relevant wiki pages | [Getting Started](https://deepwiki.com/tenstorrent/ttsim/2-getting-started), [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| Verify configuration | Ensure `TT_METAL_HOME`, `TT_METAL_SIMULATOR`, `TT_METAL_SLOW_DISPATCH_MODE=1` are set correctly | [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| Check SOC descriptors | Confirm `*_arch.yaml` files are co-located with simulator binary | [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| Consult error reference | See if your error is documented | [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference) |

**Sources:**[CONTRIBUTING.md 28-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L28-L29)

### Writing Effective Issue Reports

#### Use Clear, Specific Titles

Good titles enable quick scanning of the issue tracker:

| ❌ Poor Title | ✅ Good Title |
| --- | --- |
| "Simulator doesn't work" | "Simulator crashes on ttnn::add with UndefinedBehavior error" |
| "Need help" | "How to configure TT_METAL_SIMULATOR for Blackhole architecture?" |
| "Feature request" | "Request: Support for additional SFPU operations in libttsim" |
| "Error" | "ConfigurationError: SOC descriptor not found at expected path" |

#### Provide Minimal Reproducible Examples

When possible, reduce the problem to the smallest code or command sequence that demonstrates the issue:

```
# Minimal reproduction for bug report
export TT_METAL_HOME=/opt/tt-metal
export TT_METAL_SIMULATOR=/opt/tt-metal/libttsim_wh.so
export TT_METAL_SLOW_DISPATCH_MODE=1

cd $TT_METAL_HOME/tests/ttnn/unit_tests
pytest test_specific_operation.py::test_specific_case

# Expected: Test passes
# Actual: UndefinedBehavior error at line X
```

#### Format Technical Information

Use markdown formatting to make technical information readable:

*   **Code blocks** for commands, error messages, and configuration:

````bashexport TT_METAL_HOME=/path/to/tt-metal````
*   **Inline code** for variable names, file paths, function names:

`The error occurs when `TT_METAL_SIMULATOR` points to `libttsim_bh.so``
*   **Tables** for structured comparisons or environment information

*   **Lists** for reproduction steps or multiple related issues

**Sources:**[CONTRIBUTING.md 22-26](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L22-L26)

### Common Mistakes to Avoid

| Mistake | Why It's Problematic | Better Approach |
| --- | --- | --- |
| Reporting multiple unrelated issues in one report | Makes tracking and fixing harder | Create separate issues for each distinct problem |
| Omitting reproduction steps | Developers cannot reproduce to fix | Always include step-by-step reproduction |
| Using vague descriptions like "doesn't work" | Lacks actionable information | Describe specific behavior and error messages |
| Not including environment information | Configuration differences affect behavior | Always include OS, versions, environment variables |
| Reporting security issues publicly | Exposes vulnerability before fix available | Use private Security Advisory feature |
| Not searching existing issues | Creates duplicate reports | Search before creating new issue |
| Demanding immediate fixes | Not constructive feedback | Describe impact professionally |

## What Happens After Reporting

Once you report an issue through GitHub issues, the following process occurs:

**Diagram: Issue Lifecycle from Submission to Resolution**

### Timeline Expectations

The internal team processes issues on their own schedule. Response times vary based on:

*   **Issue severity**: Crashes and critical bugs receive higher priority
*   **Reproducibility**: Issues with clear reproduction steps are easier to address
*   **Impact scope**: Issues affecting many users receive more attention
*   **Available resources**: Implementation depends on team capacity and priorities

Note that even after implementation in the internal repository, changes may not appear immediately in public releases due to the filtered export process described in [Contribution Process](https://deepwiki.com/tenstorrent/ttsim/5.1-contribution-process).

**Sources:**[CONTRIBUTING.md 32-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L32-L34)

### Your Role After Reporting

After creating an issue, you can help the process by:

1.   **Monitoring the issue**: Enable notifications for updates and questions
2.   **Responding to requests**: Provide additional information when maintainers ask
3.   **Testing fixes**: If a fix is released, test and confirm it resolves your issue
4.   **Updating the issue**: If you find a workaround or additional information, add it as a comment
5.   **Closing if resolved**: If the issue is resolved by a release or workaround, confirm and close
