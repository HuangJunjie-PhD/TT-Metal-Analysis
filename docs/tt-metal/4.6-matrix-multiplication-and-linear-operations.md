## Purpose and Scope

This page documents matrix multiplication (matmul) and linear layer operations in the TTNN library. Matrix multiplication is a fundamental primitive for neural network models, particularly in transformer architectures and dense layers. Linear operations combine matrix multiplication with bias addition and optional activation functions.

For related operations:

*   Convolution operations: see [Convolution Operations](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Convolution%20Operations)
*   Attention mechanisms (which heavily use matmul): see [Attention and Transformer Operations](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Attention%20and%20Transformer%20Operations)
*   Element-wise operations: see [Element-wise and Unary Operations](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Element-wise%20and%20Unary%20Operations)

## Matrix Multiplication Operations

### Basic Matmul Operation

The primary matmul operation computes $C = A \times B$.

*   **Tiled execution**: Operations work on tiles, typically 32x32, though "tiny tiles" (e.g., 16x16, 16x32) are supported for specific configurations [tests/ttnn/unit_tests/operations/matmul/test_matmul.py 46-69](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/matmul/test_matmul.py#L46-L69)
*   **Data format support**: `BFLOAT16`, `BFLOAT8_B`, `BFLOAT4_B`, and `FLOAT32` are supported [tests/ttnn/unit_tests/operations/matmul/test_matmul.py 83](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/matmul/test_matmul.py#L83-L83)
*   **Zero Volume Handling**: The system explicitly handles cases where input tensors have a dimension of size 0 by creating a zero-filled output tensor via `handle_zero_volume_matmul`[ttnn/cpp/ttnn/operations/matmul/matmul.cpp 58-83](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/matmul.cpp#L58-L83)

## Linear Layer Operations

Linear layers extend matmul by adding a bias tensor and optionally fusing an activation function.

### Bias Fusion and Post-Processing

TTNN attempts to fuse bias addition into the matmul kernel. The function `get_post_process_bias` determines if fusion is possible. If the bias shape is incompatible (e.g., `bias_padded_shape[-2] != tile_height`) or the program config doesn't support it (e.g., `MatmulMultiCoreProgramConfig`), the system falls back to post-processing bias as a separate `ttnn::add` operation [ttnn/cpp/ttnn/operations/matmul/matmul.cpp 98-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/matmul.cpp#L98-L155)

### Activation Integration

Common activations like `GELU`, `RELU`, and `SILU` can be fused into the matmul/linear operation by passing a `fused_activation` attribute [ttnn/cpp/ttnn/operations/matmul/matmul.cpp 87-96](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/matmul.cpp#L87-L96)

## GEMM Optimization Strategies

### Tiling and Sub-blocking

To maximize hardware utilization, matmuls use sub-blocking. The `find_max_subblock` utility determines the best `out_subblock_h` and `out_subblock_w` such that their product is $\le 8$ (a hardware constraint for destination register capacity) [tests/ttnn/unit_tests/operations/matmul/test_matmul.py 28-43](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/matmul/test_matmul.py#L28-L43)

### Memory Configuration and Sharding

Matmul performance is heavily dependent on memory layout. TTNN supports various sharding strategies to minimize NOC (Network-on-Chip) traffic:

*   **Width Sharding**: Often used for weights in unbatched paths (batch == 1), allowing for multicasting across a core row [tests/ttnn/unit_tests/operations/matmul/test_matmul_deepseek.py 100-134](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/matmul/test_matmul_deepseek.py#L100-L134)
*   **Height Sharding**: Used for weights in batched paths (batch > 1) or for activations to distribute work across core columns [tests/ttnn/unit_tests/operations/matmul/test_matmul_deepseek.py 147-178](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/matmul/test_matmul_deepseek.py#L147-L178)
*   **DRAM Sharding**: Specific optimizations for reading sharded data directly from DRAM banks to L1. The `MatmulMultiCoreReuseMultiCastDRAMShardedProgramFactory` assigns readers to optimal DRAM banks to maximize bandwidth [ttnn/cpp/ttnn/operations/matmul/device/factory/matmul_multicore_reuse_mcast_dram_sharded_program_factory.cpp 107-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/factory/matmul_multicore_reuse_mcast_dram_sharded_program_factory.cpp#L107-L112)

## Kernel Implementation Details

### Dataflow Kernels

*   **Multicasting (Mcast)**: Kernels implement 1D or 2D multicasting to share tiles across cores. `reader_bmm_tile_layout_in0_sender_padding.cpp` handles the sender side of Input A multicast, including NOC semaphore synchronization [ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in0_sender_padding.cpp 106-107](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in0_sender_padding.cpp#L106-L107)
*   **Sparsity Support**: Kernels can skip batches based on a sparsity tensor to optimize execution for sparse workloads [ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in0_sender_padding.cpp 69-73](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in0_sender_padding.cpp#L69-L73)
*   **DRAM Sharded Weights**: Specialized readers handle DRAM-width sharded weights, where each core reads from specific DRAM banks directly using bank IDs and offsets provided via runtime arguments [ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in1_sender_writer_padding.cpp 146-151](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/kernels/dataflow/reader_bmm_tile_layout_in1_sender_writer_padding.cpp#L146-L151)

### Compute Kernels

1.   **Tile Transposition**: Supports transposing tiles in L1 before math operations via `transpose_tile_block`, which acquires registers, performs `transpose_wh_tile`, and packs the result [ttnn/cpp/ttnn/operations/matmul/device/kernels/compute/bmm_large_block_zm_fused_bias_activation.cpp 45-87](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/kernels/compute/bmm_large_block_zm_fused_bias_activation.cpp#L45-L87)

## Advanced Optimization: Prefetching

For large-scale models, TTNN uses a prefetcher strategy to separate data movement from compute. This is particularly relevant for operations involving large weights that do not fit in L1.

*   **Global Circular Buffer**: Acts as a shared L1 memory space between a prefetcher (sender) and workers (receivers) [ttnn/cpp/ttnn/operations/matmul/device/matmul_device_operation.cpp 13](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/matmul/device/matmul_device_operation.cpp#L13-L13)
*   **Sub-device Partitioning**: The device can be partitioned into sub-devices (e.g., prefetcher and worker) to isolate resources and manage execution flow for complex transformer blocks [tests/ttnn/unit_tests/operations/prefetcher_common.py 24-25](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tests/ttnn/unit_tests/operations/prefetcher_common.py#L24-L25)

## Summary

Matrix multiplication and linear operations in TTNN are highly optimized primitives that:

1.   **Leverage Tiered Dispatch**: From high-level `ttnn::linear` to architecture-specific program factories based on sharding and grid topology.
2.   **Optimize Data Movement**: Use 1D/2D multicasting and DRAM sharding to saturate compute and minimize NOC congestion.
3.   **Support Fusion**: Integrate bias and activations directly into compute kernels, with fallback post-processing for incompatible shapes.
4.   **Scale for Large Models**: Utilize advanced sharding strategies and precision controls to handle massive weight tensors efficiently across Tensix grids.
