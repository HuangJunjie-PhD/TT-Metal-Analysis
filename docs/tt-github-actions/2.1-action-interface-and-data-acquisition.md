The `collect_data` action serves as the primary entry point for aggregating telemetry from GitHub Actions workflows. It manages the lifecycle of data acquisition—from fetching raw artifacts and metadata via the GitHub API to orchestrating Python-based processing and finally uploading structured results to remote SFTP storage.

### Action Interface (`action.yml`)

The `collect_data` action is a composite action that requires specific inputs to target a workflow run and credentials to export the resulting data.

#### Inputs

| Input | Description | Required |
| --- | --- | --- |
| `repository` | The full name of the repository (e.g., `tenstorrent/tt-metal`). | Yes |
| `run_id` | The unique ID of the workflow run to collect data from. | Yes |
| `run_attempt` | The specific attempt number of the workflow run. | Yes |
| `sftp_host` / `sftp_user` | Credentials for the main CICD telemetry SFTP server. | No |
| `sftp_perf_host` / `sftp_perf_user` | Credentials for performance benchmark data storage. | No |
| `sftp_optest_host` / `sftp_optest_user` | Credentials for OpTest (kernel test) data storage. | No |
| `ssh-private-key` | Private key used for all SFTP authentication. | No |

**Sources:**[.github/actions/collect_data/action.yml 5-35](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/action.yml#L5-L35)

### Data Acquisition (`download_workflow_data.sh`)

This bash script handles the initial retrieval of raw data from the GitHub API using the `gh` CLI and `jq`. It standardizes the local directory structure before the Python processing stage begins.

1.   **Directory Setup**: It creates a workspace at `generated/cicd/${RUN_ID}/` with subdirectories for `artifacts` and `logs`[[.github/actions/collect_data/download_workflow_data.sh:25-31]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
2.   **Artifact Retrieval**: It paginates through workflow artifacts and downloads only those containing the string "report" in their name [[.github/actions/collect_data/download_workflow_data.sh:35-40]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
3.   **Metadata Fetching**: 
    *   Saves the workflow run details to `workflow.json`[[.github/actions/collect_data/download_workflow_data.sh:60-60]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
    *   Aggregates all job metadata (handling pagination) into a single `workflow_jobs.json` file [[.github/actions/collect_data/download_workflow_data.sh:61-61]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).

**Sources:**[.github/actions/collect_data/download_workflow_data.sh 1-64](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/download_workflow_data.sh#L1-L64)

### Processing Entrypoint (`generate_data.py`)

The `generate_data.py` script acts as the CLI entrypoint that orchestrates three distinct sub-pipelines. It transforms the raw JSON and artifacts into structured Pydantic models.

#### Execution Logic

The script executes the following functions in order:

1.   **`create_pipeline_json`**: Calls `cicd.create_cicd_json_for_data_analysis` to generate the high-level `Pipeline` model. This model is a prerequisite for the other pipelines as it provides context (e.g., commit hash, branch) [[.github/actions/collect_data/src/generate_data.py:16-32]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
2.   **`create_benchmark_jsonl`**: Calls `benchmark.create_json_from_report`. It filters and sorts performance data, writing it to a `.jsonl` file [[.github/actions/collect_data/src/generate_data.py:35-47]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
3.   **`create_optest_json`**: Calls `optests.create_optest_reports`. It generates individual JSON files for every job containing operation-level test results [[.github/actions/collect_data/src/generate_data.py:50-65]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).

**Sources:**[.github/actions/collect_data/src/generate_data.py 68-105](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/generate_data.py#L68-L105)

### SFTP Upload Mechanism

Data is exported to remote storage using the `sftp` command-line tool with batch files. The action uses a temporary `id_key` file created from the `ssh-private-key` input [[.github/actions/collect_data/action.yml:63-68]](https://deepwiki.com/tenstorrent/tt-github-actions/2.1-action-interface-and-data-acquisition).
