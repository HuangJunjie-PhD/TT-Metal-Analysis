## Purpose and Scope

The Platform Abstraction Layer provides hardware-agnostic interfaces that enable vLLM to run on diverse computing platforms including NVIDIA CUDA GPUs, AMD ROCm GPUs, Intel XPUs, Google TPUs, and CPUs. This layer encapsulates platform-specific behavior, device detection, capability querying, and backend selection logic, allowing the rest of the codebase to operate without direct hardware dependencies.

For details on specific hardware backends and their optimizations, see [CUDA and ROCm Support](https://deepwiki.com/tenstorrent/vllm/5.2-cuda-and-rocm-support) and [CPU, TPU, and XPU Support](https://deepwiki.com/tenstorrent/vllm/5.3-cpu-tpu-and-xpu-support).

## Platform Detection and Initialization

vLLM automatically detects the available hardware platform at runtime using a plugin-based architecture. The detection occurs lazily when `current_platform` is first accessed.

### Detection Flow

**Sources:**[vllm/platforms/__init__.py 180-221](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/__init__.py#L180-L221)[vllm/platforms/__init__.py 231-256](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/__init__.py#L231-L256)

### Plugin System

The platform detection supports both built-in and out-of-tree (OOT) platform plugins. Built-in plugins are defined in `builtin_platform_plugins` dictionary, mapping platform names to detection functions:

| Platform | Detection Function | Detection Criteria |
| --- | --- | --- |
| `tpu` | `tpu_platform_plugin()` | `VLLM_TPU_USING_PATHWAYS` env var or `libtpu` library |
| `cuda` | `cuda_platform_plugin()` | NVML device count > 0, not CPU build |
| `rocm` | `rocm_platform_plugin()` | AMD SMI processor handles exist |
| `xpu` | `xpu_platform_plugin()` | Intel XPU available via IPEX |
| `cpu` | `cpu_platform_plugin()` | CPU build or Darwin platform |

### CUDA Platform Variants

CUDA has two implementation variants depending on NVML availability:

*   **`NvmlCudaPlatform`**: Uses NVIDIA Management Library (NVML) for device queries without initializing CUDA context. This is the preferred implementation for production.
*   **`NonNvmlCudaPlatform`**: Falls back to PyTorch's CUDA APIs when NVML is unavailable (e.g., on Jetson devices).

The selection occurs at module load time:

`# Simplified from vllm/platforms/cuda.py:673-685nvml_available = Falsetry:    pynvml.nvmlInit()    nvml_available = Trueexcept Exception:    nvml_available = Falsefinally:    if nvml_available:        pynvml.nvmlShutdown() CudaPlatform = NvmlCudaPlatform if nvml_available else NonNvmlCudaPlatform`
