This document covers the Tracy profiler's main GUI application, which serves as the primary interface for visualizing and analyzing profiling data. The application can connect to live profiling clients or load saved trace files for offline analysis.

For information about the underlying data processing engine, see [Data Processing Engine](https://deepwiki.com/tenstorrent/tracy/4.2-data-processing-engine). For details about timeline visualization components, see [Timeline Visualization](https://deepwiki.com/tenstorrent/tracy/4.3-timeline-visualization).

## Application Architecture

The Tracy profiler application follows a client-server architecture where the GUI application acts as the server that receives profiling data from instrumented client applications. The application is built around several key components that handle initialization, UI rendering, client discovery, and data visualization.

### Application Entry Point and Initialization

The main application entry point is implemented in [profiler/src/main.cpp 215-368](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L215-L368) The `main()` function handles command line argument parsing, backend initialization, and application setup:

The application supports three primary startup modes:

*   **File Mode**: Direct loading of `.tracy` files via `tracy::FileRead`
*   **Connect Mode**: Direct connection to a specified client address
*   **Discovery Mode**: UDP broadcast-based client discovery

**Sources:**[profiler/src/main.cpp 215-368](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L215-L368)

### Backend Abstraction Layer

The application uses a `Backend` class to abstract different UI rendering implementations. This allows Tracy to run on multiple platforms with different windowing systems:

The `Backend` class provides a consistent interface regardless of the underlying windowing system, handling window creation, event processing, and frame rendering coordination.

**Sources:**[profiler/src/Backend.hpp 11-32](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/Backend.hpp#L11-L32)[profiler/src/BackendGlfw.cpp 63-157](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/BackendGlfw.cpp#L63-L157)[profiler/src/BackendWayland.cpp 682-990](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/BackendWayland.cpp#L682-L990)

### Client Discovery and Connection Management

The application implements UDP broadcast-based client discovery to automatically find running Tracy clients on the network:

The `UpdateBroadcastClients()` function maintains a map of discovered clients with their connection details:

*   **ClientData Structure**: Contains protocol version, port, process name, and resolved address
*   **Connection Tracking**: Clients are identified by IP address and port combination
*   **Timeout Management**: Clients are removed if no broadcast received within 4 seconds

**Sources:**[profiler/src/main.cpp 370-514](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L370-L514)[profiler/src/main.cpp 69-78](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L69-L78)

## Core Application Components

### Main UI Loop

The primary user interface is rendered in the `DrawContents()` function, which implements different UI states based on the application's current mode:

The UI switches between a startup/connection screen and the main profiling visualization interface based on whether a `TracyView` instance exists.

**Sources:**[profiler/src/main.cpp 525-1666](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L525-L1666)

### Global State Management

The application maintains several global variables that coordinate between different subsystems:

| Variable | Type | Purpose |
| --- | --- | --- |
| `view` | `std::unique_ptr<tracy::View>` | Main profiling data visualization |
| `clients` | `tracy::unordered_flat_map<uint64_t, ClientData>` | Discovered client information |
| `broadcastListen` | `std::unique_ptr<tracy::UdpListen>` | UDP discovery listener |
| `connHist` | `ConnectionHistory*` | Connection history management |
| `filt` | `Filters*` | Client filtering system |
| `mainThreadTasks` | `RunQueue` | Cross-thread task execution |

**Sources:**[profiler/src/main.cpp 82-115](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L82-L115)

### Threading and Async Operations

The application uses several background threads for non-blocking operations:

The `RunQueue` class enables safe execution of callbacks on the main thread from background operations.

**Sources:**[profiler/src/main.cpp 288-311](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/main.cpp#L288-L311)[profiler/src/RunQueue.hpp 9-21](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/RunQueue.hpp#L9-L21)[profiler/src/RunQueue.cpp 8-31](https://github.com/tenstorrent/tracy/blob/68c5cbd6/profiler/src/RunQueue.cpp#L8-L31)
