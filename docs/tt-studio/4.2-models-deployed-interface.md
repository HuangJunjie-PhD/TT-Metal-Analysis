## Purpose and Scope

The Models Deployed Interface is a centralized management dashboard in TT-Studio for monitoring and controlling AI models running on Tenstorrent hardware. It provides a table-based view of active Docker containers, real-time health monitoring via specialized badges, and a stepper-based deployment wizard for launching new models.

Sources: [app/frontend/src/pages/ModelsDeployed.tsx 1-14](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/pages/ModelsDeployed.tsx#L1-L14)[app/frontend/src/components/ModelsDeployedTable.tsx 1-10](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/ModelsDeployedTable.tsx#L1-L10)

## Interface Architecture

The interface is built around the `ModelsDeployedCard` (rendered via `ModelsDeployedTable`), which aggregates data from the Docker Control Service and the Model Management backend.

## Key Components

### Health Monitoring (`HealthBadge`)

The `HealthBadge` component provides real-time health status for a specific deployment. It performs an initial check and then enters a monitoring loop for any model not yet in a "healthy" state.

*   **Status Polling**: Checks `/models-api/health/?deploy_id={id}` every 3 seconds if the status is not `healthy`.
*   **States**: `healthy` (200 OK), `starting` (202 Accepted), `unavailable` (503 Service Unavailable), and `unknown`.
*   **Visuals**: Uses an `animate-pulse` dot for active transitions.

Sources: [app/frontend/src/components/HealthBadge.tsx 27-58](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/HealthBadge.tsx#L27-L58)[app/frontend/src/components/HealthBadge.tsx 108-112](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/HealthBadge.tsx#L108-L112)[app/frontend/src/components/HealthBadge.tsx 158-160](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/HealthBadge.tsx#L158-L160)

### Status Management (`StatusBadge`)

While `HealthBadge` monitors the application-level readiness, `StatusBadge` reflects the Docker container state.

*   **Mappings**: 
    *   `running`/`available` -> Green (Success)
    *   `stopped`/`not downloaded` -> Red (Destructive)
    *   Others -> Yellow (Warning)

Sources: [app/frontend/src/components/StatusBadge.tsx 16-36](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/StatusBadge.tsx#L16-L36)

### Deployment Wizard (`Stepper` System)

For new deployments, TT-Studio uses a multi-step wizard implemented via a specialized Stepper UI.

*   **StepperProvider**: Manages `activeStep`, `initialStep`, and navigation logic (`nextStep`, `prevStep`).
*   **StepperFormActions**: Provides "Prev", "Next", and "Skip" (for optional steps) buttons, integrating with `react-hook-form`.
*   **DeploymentProgress**: A specialized component that displays during the deployment job, showing stages like `model_preparation` (with download speeds) and `container_setup`.

## Deployment Progress Tracking

During the deployment phase, the `DeploymentProgress` component tracks the asynchronous job status. It includes specific logic for handling model weight downloads.

*   **Weight Download Metrics**: Displays `downloaded_bytes`, `speed_bps`, and `eta_seconds` when the stage is `model_preparation`.
*   **Stage Mapping**: Maps internal stages (e.g., `container_setup`) to user-friendly labels (e.g., "Creating container") and icons.
*   **Persistence**: Notifies users that they can leave the page while downloads continue in the background.

Sources: [app/frontend/src/components/ui/DeploymentProgress.tsx 26-52](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/ui/DeploymentProgress.tsx#L26-L52)[app/frontend/src/components/ui/DeploymentProgress.tsx 96-112](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/ui/DeploymentProgress.tsx#L96-L112)[app/frontend/src/components/ui/DeploymentProgress.tsx 193-197](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/ui/DeploymentProgress.tsx#L193-L197)

## API Integration Detail

The frontend interacts with the backend via `axios` calls defined in `modelsDeployedApis.ts`.

1.   **`fetchModels()`**: Calls `/docker-api/status/`. It parses the returned `ContainerData` and maps `port_bindings` into a readable string format (e.g., `0.0.0.0:8000->80/tcp`).
2.   **`deleteModel()`**: Sends a POST request to `/docker-api/stop/` with the `container_id`. It expects a `StopResponse` indicating success for both the stop and reset operations.
3.   **`Health Check`**: `HealthBadge` uses the native `fetch` API to hit `/models-api/health/` to avoid axios interceptor overhead for frequent polling.
