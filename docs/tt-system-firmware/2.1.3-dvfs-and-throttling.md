This document describes the Dynamic Voltage and Frequency Scaling (DVFS) system and throttling mechanisms implemented in the SMC firmware. DVFS dynamically adjusts the AICLK frequency and VCORE voltage to balance performance with power and thermal constraints. Throttling mechanisms enforce limits on maximum frequency based on power consumption (TDP/TDC), temperature, board power limits, and GDDR thermal conditions.

For information about the broader telemetry collection system that reports DVFS state, see [SMC Telemetry System](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.2-smc-telemetry-system). For information about voltage regulation hardware control, see the voltage arbiter implementation in `lib/tenstorrent/bh_arc/voltage.c`.

## System Architecture

The DVFS system consists of three primary subsystems that work together to manage AICLK frequency:

1.   **AICLK Power Performance Management (PPM)**: Arbitration system that determines target AICLK frequency based on minimum and maximum frequency requests from multiple sources
2.   **Throttlers**: PID-controlled feedback loops that dynamically adjust maximum frequency limits based on measured power, current, and temperature
3.   **DVFS Coordinator**: 1ms periodic loop that orchestrates voltage changes and frequency transitions

**Figure 1: DVFS System Architecture**

The DVFS system operates on a 1ms timer that triggers a work handler to recalculate throttlers, arbitrate AICLK frequency, and coordinate voltage-frequency transitions. Throttlers continuously monitor telemetry data and adjust arbiter limits using PID control.

**Sources:**[lib/tenstorrent/bh_arc/dvfs.c 1-89](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L1-L89)[lib/tenstorrent/bh_arc/aiclk_ppm.c 1-413](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L1-L413)[lib/tenstorrent/bh_arc/throttler.c 1-381](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L1-L381)

## AICLK Arbitration System

The AICLK arbitration system determines the target AICLK frequency by collecting minimum and maximum frequency requests from multiple sources and applying a priority resolution algorithm.

### Arbiter Types

The system maintains two categories of arbiters:

| Arbiter Type | Enum | Purpose | Default State |
| --- | --- | --- | --- |
| **Minimum Arbiters** | `aiclk_arb_min` | Set lower bounds on frequency |  |
| `aiclk_arb_min_fmin` | 0 | Absolute minimum (chip Fmin) | Enabled at Fmin |
| `aiclk_arb_min_busy` | 1 | Performance requirement when busy | Enabled at Fmin (idle) or Fmax (busy) |
| **Maximum Arbiters** | `aiclk_arb_max` | Set upper bounds on frequency |  |
| `aiclk_arb_max_fmax` | 0 | Absolute maximum (chip Fmax) | Enabled at Fmax |
| `aiclk_arb_max_tdp` | 1 | Thermal Design Power limit | Controlled by TDP throttler |
| `aiclk_arb_max_fast_tdc` | 2 | Fast Thermal Design Current limit | Controlled by fast TDC throttler |
| `aiclk_arb_max_tdc` | 3 | Thermal Design Current limit | Controlled by TDC throttler |
| `aiclk_arb_max_thm` | 4 | Thermal temperature limit | Controlled by thermal throttler |
| `aiclk_arb_max_board_power` | 5 | Board-level power limit | Controlled by board power throttler |
| `aiclk_arb_max_voltage` | 6 | Voltage-constrained frequency | Statically set based on VF curve |
| `aiclk_arb_max_gddr_thm` | 7 | GDDR thermal limit | Controlled by GDDR thermal throttler |
| `aiclk_arb_max_doppler_slow` | 8 | Doppler slow power limit | Controlled by Doppler throttler |
| `aiclk_arb_max_doppler_critical` | 9 | Doppler critical power limit | Enabled on T2/T3 trigger |

**Sources:**[include/tenstorrent/bh_arc/aiclk_ppm.h 12-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_arc/aiclk_ppm.h#L12-L46)

### Target Frequency Calculation

**Figure 2: Target Frequency Calculation Algorithm**

The `CalculateTargAiclk()` function implements a priority-based arbitration algorithm [lib/tenstorrent/bh_arc/aiclk_ppm.c 91-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L91-L140):

1.   **Start with minimum arbiters**: Find the highest enabled minimum arbiter value
2.   **Apply maximum arbiters**: If the result exceeds the lowest enabled maximum arbiter, clamp to that maximum
3.   **Apply Fmin floor**: Ensure frequency is at least `fmin` (chip-specific minimum)
4.   **Apply sweep override**: If frequency sweep is enabled for testing, use random frequency in range
5.   **Apply forced override**: If `forced_freq` is set, use that value regardless of arbiters

The result is stored in `aiclk_ppm.targ_freq` and recorded in telemetry as `TAG_AICLK_PPM_INFO`.

**Sources:**[lib/tenstorrent/bh_arc/aiclk_ppm.c 91-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L91-L140)[lib/tenstorrent/bh_arc/aiclk_ppm.c 288-320](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L288-L320)

## Throttler System

Throttlers are PID-controlled feedback loops that dynamically adjust maximum AICLK frequency limits based on measured telemetry. Each throttler monitors a specific metric (power, current, or temperature) and adjusts its corresponding arbiter to keep the metric below a configured limit.

### Throttler Implementation

**Figure 3: Throttler PID Control Loop**

Each throttler follows this update sequence [lib/tenstorrent/bh_arc/throttler.c 239-258](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L239-L258):

1.   **Filter measurement**: Apply alpha filter to smooth telemetry reading
2.   **Calculate error**: `error = (limit - value) / limit` (normalized error)
3.   **PID output**: `output = p_gain * error + d_gain * (error - prev_error)`
4.   **Update arbiter**: `arb_val += output * kThrottlerAiclkScaleFactor` (500 MHz scale factor)
5.   **Apply to arbiter**: Call `SetAiclkArbMax()` with the calculated frequency

**Sources:**[lib/tenstorrent/bh_arc/throttler.c 65-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L65-L140)[lib/tenstorrent/bh_arc/throttler.c 239-258](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L239-L258)

### Throttler Types and Tuning

| Throttler | Metric Monitored | Limit Source | Alpha | P-Gain | D-Gain | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| `kThrottlerTDP` | `vcore_power` (W) | `chip_limits.tdp_limit` | 1.0 | 0.015 | 0.1 | Thermal Design Power |
| `kThrottlerFastTDC` | `vcore_current` (A) | `chip_limits.tdc_fast_limit` | 1.0 | 0.5 | 0 | Fast response to current spikes |
| `kThrottlerTDC` | `vcore_current` (A) | `chip_limits.tdc_limit` | 0.1 | 0.2 | 0 | Slower, filtered TDC |
| `kThrottlerThm` | `asic_temperature` (°C) | `chip_limits.thm_limit` | 1.0 | 0.2 | 0 | Die temperature |
| `kThrottlerBoardPower` | `GetInputPower()` (W) | `DEFAULT_BOARD_POWER_LIMIT` (150W) | 1.0 | 0.1 | 0.1 | Board-level power limit |
| `kThrottlerGDDRThm` | `GetMaxGDDRTemp()` (°C) | `chip_limits.gddr_thm_limit` | 1.0 | 0.2 | 0 | GDDR thermal |
| `kThrottlerDopplerSlow` | Moving avg power (W) | Board power limit from DMC | 1.0 | 0.0025 | 0.3 | Slow Doppler power control |

**Sources:**[lib/tenstorrent/bh_arc/throttler.c 83-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L83-L140)

The tuning parameters control the throttler response characteristics:

*   **`alpha_filter`**: Smoothing factor for measurements (1.0 = no filtering, 0.1 = heavy filtering)
*   **`p_gain`**: Proportional gain - how aggressively to respond to current error
*   **`d_gain`**: Derivative gain - how much to respond to rate of change of error

Note that fast TDC has higher P-gain (0.5) for rapid response to current transients, while slow TDC uses heavy filtering (alpha=0.1) with lower P-gain (0.2) for stability.

**Sources:**[lib/tenstorrent/bh_arc/throttler.c 83-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L83-L140)

### Doppler Throttling System

Doppler throttling is an advanced power management mode that uses multiple tiers of power limiting and can inject NOPs into kernel execution when AICLK throttling alone is insufficient.

**Figure 4: Doppler Throttling State Machine**

Doppler throttling has three tiers [lib/tenstorrent/bh_arc/throttler.c 289-337](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L289-L337):

1.   **T1 (Slow)**: PID-controlled throttler operating on 1000-sample moving average of input power. Uses very low P-gain (0.0025) for gradual adjustments. Updates `aiclk_arb_max_doppler_slow`.

2.   **T2 (Fast)**: Triggered when instantaneous power exceeds 2x the board power limit for 10 consecutive samples. Enables `aiclk_arb_max_doppler_critical` arbiter set to Fmin and broadcasts kernel throttle state to inject NOPs.

3.   **T3 (Critical)**: Triggered when instantaneous power exceeds 2.5x the board power limit for 2 consecutive samples. Same response as T2 but with lower latency.

When T2 or T3 is triggered, the system writes to address `0x10` on all Tensix cores via NOC broadcast [lib/tenstorrent/bh_arc/throttler.c 156-188](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L156-L188) The kernel throttle counter is incremented, and the LLK (Low-Level Kernel) uses the parity of this counter to determine whether to run at full speed (even) or insert NOPs (odd).

**Sources:**[lib/tenstorrent/bh_arc/throttler.c 202-237](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L202-L237)[lib/tenstorrent/bh_arc/throttler.c 289-337](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L289-L337)[lib/tenstorrent/bh_arc/throttler.c 156-188](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L156-L188)

## DVFS Coordination Loop

The DVFS loop coordinates voltage and frequency transitions to ensure safe operation. The key constraint is that voltage must be set before increasing frequency, and frequency must be decreased before lowering voltage.

### DVFS Execution Sequence

**Figure 5: DVFS Loop Execution Sequence**

The `DVFSChange()` function [lib/tenstorrent/bh_arc/dvfs.c 15-30](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L15-L30) orchestrates the following steps:

1.   **`CalculateThrottlers()`**: Update all throttler PID loops based on current telemetry
2.   **`CalculateTargAiclk()`**: Arbitrate AICLK frequency from min/max arbiters
3.   **`VFCurve(targ_freq)`**: Determine required voltage for target frequency
4.   **`VoltageArbRequest(VoltageReqAiclk, voltage)`**: Request voltage from voltage arbiter
5.   **`CalculateTargVoltage()`**: Calculate final target voltage (may be higher if other requesters need more)
6.   **Sequenced transitions**: 
    *   **`DecreaseAiclk()`**: If `targ_freq < curr_freq`, decrease frequency first
    *   **`VoltageChange()`**: Adjust voltage via I2C to voltage regulator
    *   **`IncreaseAiclk()`**: If `targ_freq > curr_freq`, increase frequency after voltage is set

**Sources:**[lib/tenstorrent/bh_arc/dvfs.c 15-30](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L15-L30)[lib/tenstorrent/bh_arc/aiclk_ppm.c 142-162](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L142-L162)

### DVFS Timer Management

The DVFS timer runs at a fixed 1ms period but can be phase-adjusted to align with board power updates from the DMC [lib/tenstorrent/bh_arc/dvfs.c 73-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L73-L88):

*   **Base period**: 1ms (`DVFS_MSEC`)
*   **Adjustment**: When a board power message arrives from DMC, if DVFS is scheduled more than 10% of its period away, the next execution is moved 1% closer
*   **Purpose**: Minimize latency between receiving updated board power limit and responding with throttling adjustment

This adaptive timing helps synchronize DVFS with the DMC's power telemetry updates.

**Sources:**[lib/tenstorrent/bh_arc/dvfs.c 44-88](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L44-L88)

## Initialization and Startup

DVFS initialization occurs in phases during system startup:

**Figure 6: DVFS Initialization Sequence**

### Initialization Steps

1.   **`InitAiclkPPM()`**[lib/tenstorrent/bh_arc/aiclk_ppm.c 200-239](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L200-L239): Called via `SYS_INIT_APP` macro

    *   Reads current frequency from PLL as `boot_freq`
    *   Sets `fmax` and `fmin` from FW table with bounds checking
    *   Initializes all arbiters to default values (max arbiters at fmax, min arbiters at fmin)
    *   Disables `forced_freq` and frequency sweep

2.   **`InitDVFS()`**[lib/tenstorrent/bh_arc/dvfs.c 44-51](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L44-L51): Called from application main

    *   `InitVFCurve()`: Loads voltage-frequency curve from FW table
    *   `InitVoltagePPM()`: Initializes voltage arbitration system
    *   `InitArbMaxVoltage()`: Sets `aiclk_arb_max_voltage` based on VF curve and max voltage
    *   `InitThrottlers()`: Configures throttler limits and enables appropriate arbiters based on Doppler mode
    *   Sets `dvfs_enabled = true`

3.   **`StartDVFSTimer()`**[lib/tenstorrent/bh_arc/dvfs.c 55-58](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L55-L58): Called from application main

    *   Starts 1ms periodic timer that submits `dvfs_worker` to system work queue

**Sources:**[lib/tenstorrent/bh_arc/aiclk_ppm.c 200-239](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L200-L239)[lib/tenstorrent/bh_arc/dvfs.c 44-58](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L44-L58)[lib/tenstorrent/bh_arc/throttler.c 202-237](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L202-L237)

### Doppler Mode Selection

Throttler initialization selects between traditional throttling and Doppler mode [lib/tenstorrent/bh_arc/throttler.c 202-237](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L202-L237):

**Traditional Mode** (`doppler = false`):

*   Enables: `aiclk_arb_max_tdp`, `aiclk_arb_max_fast_tdc`, `aiclk_arb_max_tdc`, `aiclk_arb_max_board_power`
*   Thermal throttlers always enabled: `aiclk_arb_max_thm`, `aiclk_arb_max_gddr_thm`

**Doppler Mode** (`doppler = true`):

*   Disables traditional power throttlers (TDP, TDC, board power)
*   Enables: `aiclk_arb_max_doppler_slow`
*   Thermal throttlers remain enabled
*   T2/T3 critical throttling enabled on demand

The Doppler enable flag comes from the FW table: `tt_bh_fwtable_get_fw_table()->feature_enable.doppler_en`.

**Sources:**[lib/tenstorrent/bh_arc/throttler.c 202-237](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/throttler.c#L202-L237)
