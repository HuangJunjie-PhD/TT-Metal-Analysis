This page provides a detailed reference for all configuration options available in the `npeConfig` class. These options control simulation behavior, output generation, and performance modeling parameters. For information about using the command-line interface, see [Command Line Interface](https://deepwiki.com/tenstorrent/tt-npe/6.1-command-line-interface). For details on the Python API, see [Python API Reference](https://deepwiki.com/tenstorrent/tt-npe/6.2-python-api-reference).

## Overview

The `npeConfig` struct contains all parameters needed to initialize and run an NPE simulation. It can be populated either through command-line arguments (via [tt_npe.py](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe.py#LNaN-LNaN)), through direct Python API manipulation, or by setting fields in C++ code.

## Configuration Fields Reference

All fields in `npeConfig` have sensible defaults, so only workload-specific options typically need to be set.

### Device Configuration

These options specify the target hardware architecture for simulation.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `device_name` | `std::string` | `"wormhole_b0"` | `-d`, `--device` | Target device architecture. Valid values: `wormhole_b0`, `N150`, `N300`, `T3K`, `blackhole`, `P100`, `P150` |
| `topology_json` | `std::string` | `""` | `--topology-json` | Path to topology JSON file for multi-chip configurations. Required for fabric-enabled devices. See [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.4-multi-chip-device-models) for details. |

**Device Selection Behavior:**

*   Single-chip devices: `wormhole_b0`, `blackhole`, `P100`, `P150`
*   Multi-chip devices: `N150` (2 chips), `N300` (4 chips), `T3K` (8+ chips)
*   Multi-chip configurations require a valid `topology_json` file specifying chip interconnect layout

**Sources:**[tt_npe/cpp/include/npeConfig.hpp 17-32](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeConfig.hpp#L17-L32)[tt_npe/py/pycli/tt_npe.py 27-107](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/pycli/tt_npe.py#L27-L107)

### Simulation Parameters

These options control the core simulation algorithm behavior.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `cycles_per_timestep` | `uint32_t` | `128` | `-c`, `--cycles-per-timestep` | Number of NoC clock cycles represented by each simulation timestep. Smaller values increase accuracy but reduce performance. |
| `congestion_model_name` | `std::string` | `"fast"` | `--cong-model` | Congestion modeling algorithm. Valid values: `fast` (default, models link/NIU contention), `none` (no congestion modeling) |
| `estimate_cong_impact` | `bool` | `true` | _(not exposed via CLI)_ | Calculate congestion-free cycle estimate for comparison with actual estimate. Enables `getCongestionImpact()` metrics. |

### Injection Rate Configuration

Controls how source injection rates are determined for transfers.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `infer_injection_rate_from_src` | `bool` | `true` | `--no-injection-rate-inference` | Automatically infer injection rate from source core type (WORKER, DRAM, ETH). When disabled, uses explicitly specified rates. See [Injection Rate Inference](https://deepwiki.com/tenstorrent/tt-npe/7.3-injection-rate-inference) for details. |

**Inference Behavior:**

*   When `true`: Engine queries device model for core-type-specific injection rates
*   When `false`: Uses `injection_rate` field from each `npeWorkloadTransfer`
*   DRAM cores typically have higher injection rates than worker cores
*   See [WormholeB0DeviceModel](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/WormholeB0DeviceModel#LNaN-LNaN) for device-specific rates

**Sources:**[tt_npe/cpp/include/npeConfig.hpp 23](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeConfig.hpp#L23-L23)[tt_npe/py/pycli/tt_npe.py 48-129](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/pycli/tt_npe.py#L48-L129)

### Workload Loading Options

Configure how workload JSON files are interpreted.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `workload_json` | `std::string` | `""` | `-w`, `--workload` | Path to workload JSON file (required unless constructing workload programmatically) |
| `workload_is_noc_trace` | `bool` | `false` | `-t`, `--workload-is-noc-trace` | Parse workload as tt-metal NoC trace format instead of standard NPE JSON format. See [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads) |
| `scale_workload_schedule` | `float` | `0.0` | `--scale-workload-schedule` | **(Experimental)** Scale all transfer timing by this factor. `0.0` disables scaling. See [Workload Schedule Scaling](https://deepwiki.com/tenstorrent/tt-npe/7.4-workload-schedule-scaling) |
| `remove_localized_unicast_transfers` | `bool` | `false` | _(not exposed via CLI)_ | Filter out unicast transfers where source and destination are the same core. Useful for analyzing only cross-core traffic. |

### Timeline Output Options

Control generation and formatting of timeline visualization files.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `emit_timeline_file` | `bool` | `false` | `-e`, `--emit-timeline-file` | Enable generation of `.npeviz` timeline file for visualization in ttnn-visualizer |
| `timeline_filepath` | `std::string` | `""` | `--timeline-json-filepath` | Output path for timeline file. Defaults to `npe_timeline.json` if empty |
| `compress_timeline_output_file` | `bool` | `false` | `--compress-timeline-output-file` | Compress timeline file using zstd. Creates `.npeviz.zst` file instead of `.npeviz` |
| `use_legacy_timeline_format` | `bool` | `false` | `--use-legacy-timeline-format` | Use v0 timeline format instead of v1.0.0 format. See [Timeline Visualization Format](https://deepwiki.com/tenstorrent/tt-npe/7.1-timeline-visualization-format) |
| `timeline_split_threshold_timesteps` | `size_t` | `10000` | _(not exposed via CLI)_ | Maximum timesteps per timeline file before splitting into multiple files. Prevents memory issues with very long traces. |

### Verbosity Control

Control the amount of diagnostic output during simulation.

| Field | Type | Default | CLI Argument | Description |
| --- | --- | --- | --- | --- |
| `verbosity` | `VerbosityLevel` | `Normal` | `-v`, `--verbose` | Logging verbosity level. Values: `Normal` (0), `Verbose` (1), `MoreVerbose` (2), `MostVerbose` (3) |

**Verbosity Levels:**

| Level | Value | Output Includes |
| --- | --- | --- |
| `Normal` | 0 | Essential messages, errors, final statistics |
| `Verbose` | 1 | Configuration dump, workload summary, phase transitions |
| `MoreVerbose` | 2 | Transfer-level details, dependency resolution logs |
| `MostVerbose` | 3 | Timestep-by-timestep state, link utilization per step |

## Configuration Validation

The `npeAPI` constructor validates the configuration when `InitAPI()` is called. Validation includes:

1.   **Device name validation:** Ensures `device_name` matches a supported device model
2.   **File existence checks:** Verifies `workload_json` and `topology_json` (if specified) exist
3.   **Parameter range checks:** Validates `cycles_per_timestep > 0`, valid congestion model names
4.   **Multi-chip consistency:** Requires `topology_json` for multi-chip device names

**Sources:**[tt_npe/cpp/pybind/bindings.cpp 28-37](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L28-L37)
