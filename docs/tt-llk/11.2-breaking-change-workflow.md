## Purpose and Scope

This document describes the process for introducing breaking changes to the LLK (Low-Level Kernel) API surface. Breaking changes require careful coordination across multiple chip architectures, validation through comprehensive testing, and communication with downstream consumers, particularly the tt-metal repository which depends on tt-llk as a submodule.

For information about maintaining code quality standards during development, see [Pre-commit Hooks and Code Quality](/tenstorrent/tt-llk/11.1-pre-commit-hooks-and-code-quality). For details on coordinating submodule updates with tt-metal, see [Submodule Coordination with tt-metal](/tenstorrent/tt-llk/11.3-submodule-coordination-with-tt-metal).

## What Constitutes a Breaking Change

A breaking change in tt-llk is any modification that alters the public API surface or behavior in a way that requires downstream code modifications. Breaking changes include:

| Change Type | Examples | Impact |
| --- | --- | --- |
| **API Signature Changes** | Modifying function parameters in `llk_unpack_*`, `llk_math_*`, `llk_pack_*` functions | Compilation failures in downstream kernels |
| **Data Format Modifications** | Changes to `DataFormat` enum, format inference logic, or tile layout | Incorrect data processing or runtime failures |
| **Configuration Structure Changes** | Modifications to `pack_config_t`, `unpack_config_t`, or other config structures | Compilation failures or undefined behavior |
| **Hardware Abstraction Changes** | Changes to `ckernel` namespace functions or synchronization primitives | Runtime failures or deadlocks |
| **Build System Changes** | Modifications to compilation flags, linker scripts, or `TestConfig` setup | Build failures or test infrastructure breakage |
| **Architecture-Specific Behavior** | Changes affecting Wormhole B0, Blackhole, or Quasar implementations differently | Architecture-specific failures |

| **SFPU Operations** | `tt_llk_wormhole_b0/common/inc/sfpu/` | `tt_llk_blackhole/common/inc/sfpu/` | `tt_llk_quasar/common/inc/sfpu/` | N/A |

### Test Failure Investigation

When breaking changes cause test failures, the following diagnostic information is available:

1. **Compilation Failures**: Check JUnit XML reports for compilation errors in the compile-producer phase
2. **Runtime Failures**: Examine stack traces in the compile-consumer phase, including LLK assertion failures
3. **Architecture-Specific Issues**: Compare test results across Wormhole B0, Blackhole, and Quasar
4. **Format-Related Issues**: Review format inference logic and golden model outputs
5. **Synchronization Issues**: Check for timeouts indicating deadlocks or missing synchronization

## Version Considerations

The tt-llk repository uses Git commit-based versioning rather than semantic versioning. Breaking changes are tracked through:

1. **Git Commit History**: Each breaking change is documented in commit messages
2. **Submodule References**: The tt-metal repository pins specific tt-llk commits
3. **Migration Path**: Breaking changes require coordinated updates to tt-metal submodule references
4. **No Backward Compatibility**: tt-llk does not maintain backward compatibility across breaking changes

When introducing breaking changes:

* **Atomic Updates**: Coordinate tt-llk and tt-metal updates to merge at approximately the same time
* **Feature Flags**: For complex migrations, consider feature flags to enable gradual rollout
* **Deprecation Period**: For widely-used APIs, consider a deprecation period with warnings before removal
* **Documentation**: Maintain clear documentation of API evolution in commit messages and PR descriptions

## Common Breaking Change Scenarios

### Scenario 1: Adding Required Function Parameter

**Example**: Adding a new required parameter to `llk_unpack_A()`.

**Required Changes**:

1. Update function signature in all architecture-specific headers
2. Update all call sites in test sources
3. Update `TestConfig.generate_build_header()` to expose parameter as compile-time constant
4. Update golden models if parameter affects output
5. Document migration path in PR description

### Scenario 2: Modifying Configuration Structure

**Example**: Changing fields in `pack_config_t`.

**Required Changes**:

1. Update structure definition in common headers
2. Update initialization code in `_llk_pack_hw_configure_` implementations
3. Update test infrastructure that constructs configuration structures
4. Verify all architecture-specific implementations handle new fields correctly
5. Update serialization/deserialization if configuration is passed to hardware

### Scenario 3: Changing Data Format Handling

**Example**: Modifying format inference rules or adding new `DataFormat` enum value.

**Required Changes**:

1. Update `DataFormat` enum in `helpers/llk_params.py`
2. Update format inference logic in `data_format_inference.py`
3. Update golden models to handle new format
4. Add packing/unpacking functions for new format in `pack.py` and `unpack.py`
5. Update format combination sweeps in test parameterization
6. Verify format outlier detection logic in `is_format_combination_outlier()`

## Rollback Procedures

If a breaking change causes unexpected issues after merge:

1. **Immediate Rollback**: Revert the breaking change commit in tt-llk
2. **Submodule Downgrade**: Update tt-metal submodule reference to previous stable commit
3. **Investigation**: Diagnose root cause using CI artifacts and test logs
4. **Fix Forward**: Prepare corrected version with proper validation
5. **Re-coordination**: Re-coordinate merge with tt-metal repository

The two-phase testing approach (compile-producer, compile-consumer) catches most issues before merge, but post-merge issues may still occur due to:

* Integration issues with tt-metal not covered by tt-llk tests
* Hardware-specific edge cases not covered by test suite
* Interactions between multiple concurrent changes
