## Purpose and Scope

Meta Schedule (also known as AutoTIR) is TVM's third-generation automated performance tuning system. It provides advanced search capabilities for optimizing tensor programs by exploring the space of TensorIR schedules. This document covers the architecture, core components, and usage of the Meta Schedule infrastructure.

For template-based tuning, see [AutoTVM](https://deepwiki.com/tenstorrent/tt-tvm/8.1-autotvm). For template-free sketch-based tuning, see [AutoScheduler](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler). For the schedule primitives that Meta Schedule explores, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api). For the RPC measurement infrastructure used by all auto-tuners, see [RPC Infrastructure](https://deepwiki.com/tenstorrent/tt-tvm/8.4-rpc-infrastructure).

## Overview

Meta Schedule represents a significant evolution in TVM's auto-tuning capabilities. Unlike its predecessors, it is designed to work directly with the TensorIR (S-TIR) scheduling interface, enabling more flexible and powerful optimization strategies.

### Key Characteristics

| Feature | AutoTVM | AutoScheduler | Meta Schedule |
| --- | --- | --- | --- |
| **Approach** | Template-based | Template-free (sketch-based) | TensorIR schedule exploration |
| **Search Space** | Manual templates | Auto-generated from compute DAG | Schedule primitive combinations |
| **IR Level** | TE (Tensor Expression) | TE with sketch rules | TensorIR (S-TIR) |
| **Extensibility** | Limited to templates | Fixed sketch rules | Pluggable search strategies |
| **State Tracking** | Implicit | Implicit | Explicit via database |
| **Integration** | Legacy | Mid-level | Native TIR integration |

Meta Schedule operates at the TensorIR level, making it the natural choice for workloads expressed in TVMScript and providing seamless integration with the modern TVM compilation pipeline.

## Core Components

### TuneContext

The `TuneContext` serves as the central coordinator for the tuning process. It maintains references to all subsystems and orchestrates the search workflow.

**Responsibilities:**

*   Holds the target `IRModule` and hardware `Target`
*   Manages the search space and strategy
*   Coordinates builder, runner, and database
*   Tracks tuning progress and statistics

### SearchSpace

The `SearchSpace` defines what schedule transformations are available for exploration. It typically includes schedule rules that can be applied to TensorIR blocks and loops.

**Key Methods:**

*   Generate initial schedules
*   Apply transformation rules (split, fuse, reorder, cache_read, etc.)
*   Validate schedule legality
*   Sample from available transformations

### SearchStrategy

The `SearchStrategy` determines how the search space is explored. Meta Schedule supports multiple strategies:

**Strategy Types:**

*   **Evolutionary Search**: Mutation and crossover of schedules
*   **Random Search**: Uniform sampling baseline
*   **Replay Search**: Apply known good schedules

The strategy maintains a population of candidate schedules and uses the cost model to guide exploration toward better regions of the search space.

### CostModel

The `CostModel` predicts the performance of schedules without measuring them on hardware. This enables efficient pruning of the search space.

**Implementation Approaches:**

*   **XGBoost-based**: Train on measured schedules
*   **MLP-based**: Neural network prediction
*   **Random**: Baseline for comparison

## Builder and Runner System

The measurement pipeline compiles candidate schedules and executes them to obtain actual performance data.

### Builder Subsystem

**Builder System Architecture**

The builder handles:

*   Applying schedule transformations to the base TensorIR
*   Lowering through the compilation pipeline
*   Cross-compilation for remote targets
*   Error handling for invalid schedules

### Runner Subsystem

The runner executes compiled binaries and reports measurements.

**Runner Configuration:**

*   `LocalRunner`: Execute on the host machine
*   `RPCRunner`: Execute on remote devices via RPC
*   Measurement parameters: 
    *   `number`: Number of executions per measurement
    *   `repeat`: Number of measurement trials
    *   `min_repeat_ms`: Minimum time per trial
    *   `timeout`: Maximum execution time
    *   `enable_cpu_cache_flush`: Flush cache between runs

## Integration with TensorIR Schedule Primitives

Meta Schedule's power comes from its ability to explore the space of TensorIR schedule transformations. It leverages all schedule primitives available in the S-TIR system.

### Schedule Transformation Space

The search space includes combinations of the following primitives:

**Loop Transformations:**

*   `split(loop, factors)` - Split loop into nested loops
*   `fuse(loops)` - Fuse multiple loops into one
*   `reorder(loops)` - Change loop nesting order
*   `parallel(loop)` - Mark loop for parallel execution
*   `vectorize(loop)` - Mark loop for SIMD vectorization
*   `unroll(loop)` - Unroll loop iterations

**Memory Transformations:**

*   `cache_read(buffer, scope)` - Create cached read buffer
*   `cache_write(buffer, scope)` - Create cached write buffer
*   `storage_align(buffer, axis, factor, offset)` - Align buffer storage

**Compute Transformations:**

*   `compute_at(block, loop)` - Compute block within loop scope
*   `compute_inline(block)` - Inline block computation
*   `reverse_compute_at(block, loop)` - Reverse computation placement

**Block Transformations:**

*   `decompose_reduction(block, loop)` - Decompose reduction computation
*   `rfactor(loop, factor_axis)` - Factor out reduction

For detailed information on these primitives, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api).

## Performance Considerations

### Tuning Resource Requirements

Meta Schedule tuning is computationally intensive:

**CPU Requirements:**

*   High core count (16+ cores) for parallel compilation
*   Each trial requires compiling and feature extraction
*   Multiple workers can process trials in parallel

**Measurement Requirements:**

*   Access to target hardware (local or via RPC)
*   Stable thermal conditions for consistent measurements
*   Network bandwidth for RPC-based tuning

### Tuning Time Guidelines

| Network | Tasks | Trials/Task | Est. Time (Single Device) |
| --- | --- | --- | --- |
| MobileNet | ~30 | 1000 | 8-12 hours |
| ResNet-50 | ~50 | 1000 | 12-18 hours |
| BERT-Base | ~80 | 1000 | 20-30 hours |

Time can be reduced by:

*   Using multiple RPC devices in parallel
*   Setting `num_measure_trials` to a smaller value for initial exploration
*   Resuming from previous tuning logs
*   Distilling logs to remove redundant trials

### Result Quality

Meta Schedule typically achieves:

*   10-30% better performance than hand-written schedules
*   Comparable or better results than AutoScheduler
*   More consistent optimization across diverse operators
*   Better handling of complex fusion patterns

### Common Issues

**Invalid Schedules:**

*   Some sampled schedules may violate legality constraints
*   These are filtered out during the build phase
*   Reported as build failures but don't affect overall tuning

**Measurement Timeout:**

*   Can occur with slow schedules or network issues
*   Adjust `timeout` parameter in runner configuration
*   Check RPC connection stability

**Search Not Converging:**

*   Increase `max_trials_per_task`
*   Try different search strategy
*   Check if cost model is training properly

## Comparison with Previous Auto-Tuners

### AutoTVM vs AutoScheduler vs Meta Schedule

**When to Use Each:**

**AutoTVM:**

*   Legacy TE-based operators
*   Well-defined search spaces with known templates
*   Operators with limited schedule variations

**AutoScheduler:**

*   Template-free optimization needed
*   Working with TE compute definitions
*   Sketch-based search is sufficient

**Meta Schedule:**

*   Modern TensorIR-based workflows
*   Complex fusion patterns
*   Need for extensible search strategies
*   Working with TVMScript programs
*   Integration with latest TVM features

### Migration Path

For users migrating from AutoScheduler to Meta Schedule:

1.   **Express operators in TensorIR** instead of TE where possible
2.   **Update task extraction** to use Meta Schedule's API
3.   **Configure database** instead of log files
4.   **Use new builder/runner API** which provides more control
5.   **Leverage schedule replay** from existing logs if available
