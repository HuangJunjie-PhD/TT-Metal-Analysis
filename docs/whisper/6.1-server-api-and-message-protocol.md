## Purpose and Scope

This document describes the Server interface and WhisperMessage protocol used for programmatic control of the Whisper simulator. The Server API enables external tools to communicate with Whisper through socket or shared-memory connections, supporting use cases such as lock-step verification with RTL testbenches, automated testing frameworks, and remote debugging.

For interactive debugging with human-friendly commands, see [Interactive Command-Line Interface](https://deepwiki.com/tenstorrent/whisper/6.2-interactive-command-line-interface). For GDB remote debugging protocol support, see [GDB Remote Debugging](https://deepwiki.com/tenstorrent/whisper/6.4-gdb-remote-debugging). For detailed pipeline control and performance modeling, see [Performance Modeling API](https://deepwiki.com/tenstorrent/whisper/6.3-performance-modeling-api).

## Server Architecture

The Server interface provides a command-response communication model where clients send `WhisperMessage` requests and receive `WhisperMessage` replies. The `Server<URV>` class processes commands and delegates operations to the `System` and `Hart` objects.

## Communication Mechanisms

### Socket Communication

Socket-based communication uses TCP/IP for client-server interaction. The server listens on a socket and processes requests sequentially.

**Key Functions:**

*   `receiveMessage(int soc, WhisperMessage& msg)` - [Server.cpp 36-66](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L36-L66)
*   `sendMessage(int soc, const WhisperMessage& msg)` - [Server.cpp 69-94](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L69-L94)
*   `Server<URV>::interact(int soc, FILE* traceFile, FILE* commandLog)` - [Server.cpp 1065-1084](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L1065-L1084)

The socket interface performs blocking reads/writes with EINTR retry logic and supports graceful disconnection.

### Shared Memory Communication

Shared memory communication provides lower-latency IPC using a memory-mapped region with atomic guard variables for synchronization.

**Key Functions:**

*   `receiveMessage(std::span<char> shm, WhisperMessage& msg)` - [Server.cpp 98-109](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L98-L109)
*   `sendMessage(std::span<char> shm, const WhisperMessage& msg)` - [Server.cpp 112-125](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L112-L125)
*   `Server<URV>::interact(std::span<char> shm, FILE* traceFile, FILE* commandLog)` - [Server.cpp 1089-1108](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L1089-L1108)

**Protocol:**

1.   First byte serves as atomic lock ('s' = server ready, 'c' = client ready)
2.   Message starts at 4-byte aligned offset after guard byte
3.   Client writes message and sets guard to 's'
4.   Server reads message, processes, writes reply, sets guard to 'c'

**Sources:**[Server.cpp 98-125](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L98-L125)[Server.cpp 1089-1108](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L1089-L1108)

## Peek Command Details

The peek command reads simulator state. The `resource` field determines what to read, and the `address` field specifies which instance (register number, CSR number, memory address, etc.).

### Integer Register Peek

```
Request:
  type = Peek
  resource = 'r'
  address = register number (0-31)

Reply:
  value = register value
```

### CSR Peek

```
Request:
  type = Peek
  resource = 'c'
  address = CSR number
  flags.virt = virtual mode flag (for VS-mode CSRs)

Reply:
  value = CSR value
  address = write mask
  time = poke mask
  instrTag = read mask
```

The reply for CSR peek includes masks in unused fields for configuration validation.

### Vector Register Peek

```
Request:
  type = Peek
  resource = 'v'
  address = register number (0-31)

Reply:
  buffer[0..n] = vector register data (reversed byte order)
```

Vector register data is packed into the `buffer` field with most significant byte first.

### Memory Peek

```
Request:
  type = Peek
  resource = 'm'
  address = memory address

Reply:
  value = memory value (word-sized)
```

### Special Resource Peek

```
Request:
  type = Peek
  resource = 's'
  address = WhisperSpecialResource enum value

Reply:
  value = resource-specific value
  (for EffMemAttr: value=PMA1, address=PMA2)
```

**Sources:**[Server.cpp 284-439](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L284-L439)

## Poke Command Details

The poke command writes simulator state. The format is similar to peek but includes a value to write.

### Integer/FP Register Poke

```
Request:
  type = Poke
  resource = 'r' or 'f'
  address = register number
  value = new value
```

### CSR Poke

```
Request:
  type = Poke
  resource = 'c'
  address = CSR number
  value = new value
  flags.virt = virtual mode flag
```

Special handling for `MVIP` CSR to preserve aliased bits [Server.cpp 176-190](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L176-L190)

### Vector Register Poke

```
Request:
  type = Poke
  resource = 'v'
  address = register number
  size = byte count
  buffer[0..size-1] = vector data (reversed)
```

### Memory Poke

```
Request:
  type = Poke
  resource = 'm'
  address = memory address
  value = new value
  size = byte count (1, 2, 4, or 8)
  flags.cache = true to access MCM cache
  flags.skipMem = true to skip actual memory write
```

The `cache` flag enables direct cache access for A/D bit updates and I/O coherence. The `skipMem` flag updates cache without updating memory [Server.cpp 217-220](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L217-L220)

### Program Counter Poke

```
Request:
  type = Poke
  resource = 'p'
  value = new PC value
```

**Sources:**[Server.cpp 143-281](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L143-L281)

## Step Command Details

The step command executes a single instruction and returns all resulting state changes.

### Request Format

```
Request:
  type = Step
  hart = hart ID
  time = time stamp (for MCM)
  instrTag = instruction tag (for MCM)
```

### Reply Format

The reply to a step command is a `ChangeCount` message:

```
Reply:
  type = ChangeCount
  address = PC of executed instruction
  resource = instruction encoding
  value = number of change records
  flags = WhisperFlags with:
    - privMode: privilege mode (2 bits)
    - fpFlags: incremental FP flags (5 bits)
    - trap: instruction trapped (1 bit)
    - stop: target program finished (1 bit)
    - interrupt: instruction interrupted (1 bit)
    - virt: virtual mode (1 bit)
    - debug: debug mode (1 bit)
    - load: instruction is a load (1 bit)
    - cancelled: instruction cancelled (1 bit)
  buffer[0..n] = disassembly text
```

### Retrieving Changes

After receiving the `ChangeCount` reply, the client must send `Change` requests to retrieve each change record:

```
Client Request:
  type = Change

Server Reply:
  type = Change
  resource = 'r'/'f'/'c'/'v'/'m'
  address = register/CSR number or memory address
  value = new value
  size = byte count
  time = previous value (reused field)
```

Changes are returned in reverse order (use `pendingChanges_.pop_back()`) and include:

1.   Integer register changes
2.   Floating-point register changes
3.   Vector register changes (multiple changes for grouped registers)
4.   CSR changes
5.   Memory changes (store operations, including vector stores)

**Sources:**[Server.cpp 666-738](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L666-L738)[Server.cpp 478-634](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L478-L634)

## Memory Consistency Model Commands

When MCM is enabled (`System::isMcmEnabled()`), additional commands support out-of-order memory simulation with merge buffers and load-to-store forwarding.

### MCM Read Command

Records a load operation in the MCM infrastructure:

```
Request:
  type = McmRead
  hart = hart ID
  time = time stamp
  instrTag = instruction tag
  address = memory address
  size = access size (1, 2, 4, 8, or >8)
  value = data read (for size <= 8)
  buffer[0..n] = data read (for size > 8)
  resource = (elem << 16) | field  // vector element/field
  flags.cache = true for cache access
```

The MCM read command validates memory ordering rules and checks for store-to-load forwarding. For large vector loads, data is passed in the `buffer` field.

**Implementation:**[Server.cpp 780-846](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L780-L846)

### MCM Insert Command

Inserts a store operation into the merge buffer:

```
Request:
  type = McmInsert
  hart = hart ID
  time = time stamp
  instrTag = instruction tag
  address = memory address
  size = access size
  value = data to store (for size <= 8)
  buffer[0..n] = data to store (for size > 8)
  resource = (elem << 16) | field
```

Store data is held in the merge buffer until explicitly drained or retired.

**Implementation:**[Server.cpp 851-915](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L851-L915)

### MCM Bypass Command

Writes directly to memory while updating the MCM cache:

```
Request:
  type = McmBypass
  hart = hart ID
  time = time stamp
  instrTag = instruction tag
  address = memory address
  size = access size
  value = data to write
  resource = (elem << 16) | field
  flags.cache = true for cache update
```

Used for stores that bypass the merge buffer.

**Implementation:**[Server.cpp 920-985](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L920-L985)

## Additional Commands

### Reset Command

Resets the specified hart:

```
Request:
  type = Reset
  hart = hart ID
  address = optional reset PC (0 for default)
```

Memory-mapped registers are reset only after the initial reset [Server.cpp 1116-1118](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L1116-L1118)

### Translate Command

Translates a virtual address to physical address:

```
Request:
  type = Translate
  hart = hart ID
  address = virtual address
  flags = (r | (w<<1) | (x<<2) | (pm<<3) | (twoStage<<4))
    - r: read access
    - w: write access  
    - x: execute access
    - pm: privilege mode (User/Supervisor)
    - twoStage: two-stage translation

Reply:
  address = physical address
  type = Invalid if translation fails
```

**Implementation:**[Server.cpp 744-775](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L744-L775)

### Debug Mode Commands

```
EnterDebug: Enter debug mode
  hart = hart ID

ExitDebug: Exit debug mode  
  hart = hart ID

CancelLr: Cancel LR reservation
  hart = hart ID

CancelDiv: Cancel pending division
  hart = hart ID
```

**Sources:**[Server.cpp 1243-1252](https://github.com/tenstorrent/whisper/blob/733fd921/Server.cpp#L1243-L1252)

### Page Table Walk Command

Retrieves page table walk entries for address translation debugging:

```
Request:
  type = PageTableWalk
  hart = hart ID
  flags = (isInstr | (isAddr<<1))
  address = walk index

Reply:
  size = number of entries
  buffer[0..n] = 64-bit entries or addresses
```

## Error Handling

Invalid commands return a reply with `type = Invalid`. Common error cases:

| Error Condition | Detection | Reply |
| --- | --- | --- |
| Invalid hart ID | `checkHartId()` | `type = Invalid` |
| Out-of-bounds register | `peekIntReg()` returns false | `type = Invalid` |
| Invalid CSR number | `peekCsr()` returns false | `type = Invalid` |
| Memory access violation | `peekMemory()` returns false | `type = Invalid` |
| Translation failure | `transAddrNoUpdate()` returns exception | `type = Invalid` |

## Integration with Interactive Mode

The `Interactive` class shares similar command structure but provides human-friendly syntax. The server mode uses binary `WhisperMessage` protocol, while interactive mode uses text commands.

**Comparison:**

| Operation | Server Mode | Interactive Mode |
| --- | --- | --- |
| Read register | `Peek, resource='r', address=5` | `peek r x5` |
| Write CSR | `Poke, resource='c', address=0x300, value=0x1800` | `poke c mstatus 0x1800` |
| Execute instruction | `Step, hart=0` | `step` |
| Reset | `Reset, hart=0` | `reset` |

Both interfaces ultimately call the same `Hart` and `System` methods for state access and execution.

**Sources:**[Interactive.cpp 505-828](https://github.com/tenstorrent/whisper/blob/733fd921/Interactive.cpp#L505-L828)[Interactive.cpp 832-1067](https://github.com/tenstorrent/whisper/blob/733fd921/Interactive.cpp#L832-L1067)

## Summary

The Server API provides a binary message protocol for programmatic control of Whisper through socket or shared-memory communication. Key features include:

*   **WhisperMessage Protocol**: Fixed-size binary structure with serialization support
*   **Command Types**: Peek/poke state, step execution, MCM operations, debug control
*   **Resource Identifiers**: Access registers, CSRs, memory, PC, and special resources
*   **Change Tracking**: Step command returns all state changes for verification
*   **MCM Support**: Specialized commands for memory consistency model simulation
*   **Dual Transport**: Socket (TCP/IP) or shared memory (IPC)

This interface is primarily used by Verilog testbenches for cycle-accurate lock-step verification and by automated testing frameworks for RISC-V compliance validation.
