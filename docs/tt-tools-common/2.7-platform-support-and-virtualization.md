## Purpose and Scope

This document covers the platform-specific requirements and virtualization support for the tt-tools-common hardware reset system. It explains supported CPU architectures, ARM platform restrictions, Xen HVM hypervisor integration, and PCI device hotplug handling during reset operations.

For information about the overall reset architecture and chip-specific implementations, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns), [ChipReset: Modern Unified Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset), [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation), and [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation).

## Supported Platforms and Architecture Restrictions

The reset system has **strict platform compatibility requirements** due to PCIe device management differences across CPU architectures.

### x86_64 Platform Support

The reset system is **fully supported** on x86_64 (Intel/AMD) platforms. All reset operations, including:

*   Secondary bus reset (SBR)
*   ASIC reset
*   M3/DMC reset
*   PCI device hotplug/rescan

work correctly on x86_64 systems running Linux with the tt-kmd kernel driver version >= 2.4.1.

### ARM Platform Restrictions

**ARM platforms (ARM and AArch64) are NOT supported** for chip reset operations. The system explicitly checks the platform architecture and exits with an error if ARM is detected.

**Technical Reason**: ARM-based systems handle PCIe device rescans differently than x86_64. After issuing a PCIe reset, the PCI subsystem may not properly re-enumerate devices, leaving chips in an inaccessible state. The only reliable way to reset chips on ARM platforms is to **reboot the entire system**.

The platform check is performed in `get_host_info()["Platform"]` and compared against prefixes `"arm"` and `"aarch"`.

## Platform Detection Implementation

**Platform Detection Flow**: Every reset operation validates platform architecture before proceeding

The `get_host_info()` function from `system_utils` returns a dictionary containing platform information using Python's `platform.machine()` API. Common return values include:

*   `"x86_64"` for Intel/AMD 64-bit
*   `"aarch64"` for ARM 64-bit
*   `"armv7l"` for ARM 32-bit

## Xen HVM Virtualization Support

The reset system provides **full support for Xen HVM (Hardware Virtual Machine) guests** running on paravirtualized hypervisors. This enables Tenstorrent accelerators to be passed through to virtual machines using PCI passthrough while maintaining reset functionality.

### Xen HVM Detection

The system detects Xen HVM mode by reading two sysfs files:

| File Path | Expected Value |
| --- | --- |
| `/sys/hypervisor/type` | `"xen"` |
| `/sys/hypervisor/guest_type` | `"HVM"` |

Both conditions must be true for the system to operate in Xen HVM mode. The detection is implemented in the `check_xen_hvm()` function.

**Xen HVM Detection Logic**: Two sysfs files must both exist and contain expected values

The function handles `FileNotFoundError` and `OSError` gracefully, returning `False` if either file is missing or inaccessible (indicating bare-metal execution).

## Xenstore Coordination Protocol

When running in Xen HVM mode, the reset system uses **Xenstore** (Xen's shared configuration/state database) to coordinate PCIe device hot-plug operations with the Dom0 (host) hypervisor.

### Why Xenstore Coordination is Necessary

During a chip reset in a virtual machine:

1.   The guest initiates a reset via IOCTL
2.   The device undergoes PCIe hot-remove
3.   The device reappears on the PCI bus
4.   The guest must wait for Dom0 to complete PCI re-enumeration
5.   Only after Dom0 updates PCI state can the guest access the device

**Without coordination**, the guest might attempt to access the device before Dom0 has finished PCI rescanning, resulting in I/O errors or device inaccessibility.

### Xenstore File Naming Convention

For each device being reset, a Xenstore path is constructed from the PCI BDF (Bus:Device.Function) address:

```
pci_hard_reset/<bus>_<device>-<function>
```

**Example**: For device at PCI address `0000:81:00.0`, the Xenstore path is:

```
pci_hard_reset/81_00-0
```

The path construction logic splits the BDF string on `:` and `.` separators.

Sources: [tt_tools_common/reset_common/chip_reset.py 178-180](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L178-L180)

### Xenstore Protocol State Machine

**Xenstore Protocol State Machine**: Guest and Dom0 coordinate through xenstore file lifecycle

The guest polls the Xenstore path using `xenstore-read` commands until it returns a non-zero exit code (indicating the file no longer exists). This signals that Dom0 has completed PCI re-enumeration.

Sources: [tt_tools_common/reset_common/chip_reset.py 214-257](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L214-L257)

### Xenstore Implementation Details

#### Sudo Privileges Requirement

All xenstore operations (`xenstore-write`, `xenstore-read`) require **root/sudo privileges**. The system:

1.   Prompts the user for sudo credentials before reset
2.   Validates credentials with `sudo -v`
3.   Credentials are cached for 5 minutes by sudo
4.   Revalidates before polling operations to ensure long-running operations don't timeout

#### Parallel Xenstore Polling

When resetting multiple devices, the system creates **one thread per device** to poll xenstore files in parallel. This prevents a slow Dom0 response for one device from blocking other devices.

**Parallel Xenstore Polling**: One thread per device polls xenstore independently

Each thread has a **5-minute timeout**. If the xenstore file is not removed within this timeout, the system prints an error and exits.

Sources: [tt_tools_common/reset_common/chip_reset.py 228-256](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L228-L256)

#### Error Handling

The xenstore integration handles several error conditions:

| Error Condition | Behavior |
| --- | --- |
| `xenstore-write` fails | Print warning, continue with reset (non-fatal) |
| `xenstore-read` timeout (5 min) | Print error, `sys.exit(1)` (fatal) |
| No xenstore filenames stored | Print error, `sys.exit(1)` (fatal) |
| Sudo authentication failure | Subprocess raises exception (fatal) |

Sources: [tt_tools_common/reset_common/chip_reset.py 189-193](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L189-L193)[tt_tools_common/reset_common/chip_reset.py 245-248](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L245-L248)

## PCI Hotplug and Device Re-enumeration

After a chip reset, the device **temporarily disappears from the PCI bus** during hot-remove, then **reappears with a potentially different PCI interface ID**. The system must wait for the device to reappear before completing the reset sequence.

### Device Reappearance Detection

The `wait_for_device_to_reappear()` function monitors sysfs for device re-enumeration:

**Device Reappearance Detection**: Polls sysfs until device node reappears with 10-second timeout

The function:

1.   Uses the **PCI BDF address** (which remains constant) to locate the device in sysfs
2.   Extracts the **new interface ID** from the tenstorrent device symlink
3.   Verifies the `/dev/tenstorrent/{interface_id}` character device exists
4.   Returns the new interface ID (which may differ from the original)

**Why Interface ID Changes**: During PCI hot-remove/hot-add cycles, the kernel driver may assign a different interface ID when the device reappears. The BDF address is stable, but the tt-kmd interface ID is not guaranteed to remain the same.

Sources: [tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111)

### Post-Reset Wait Times

After issuing the reset IOCTL, the system waits for a calculated duration before checking for device reappearance:

| Reset Type | Wait Time |
| --- | --- |
| ASIC reset (no M3) | `max(2, 0.4 * len(pci_interfaces))` seconds |
| ASIC+DMC reset (M3) | `m3_delay` seconds (default 20) |

The non-M3 reset uses a **per-device scaling factor** (0.4 seconds per device) to account for systems with many devices, where PCI rescanning takes longer.

Sources: [tt_tools_common/reset_common/chip_reset.py 259-265](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L259-L265)

## Integration with Modern Reset Path

The Xen HVM detection and platform checks integrate into the **delegation pattern** used by chip-specific reset classes.

**Reset Path Delegation**: Xen HVM detection routes to modern `ChipReset` even if driver < 2.4.1

The **key decision logic** in both `BHChipReset` and `WHChipReset`:

`if is_driver_version_at_least(get_driver_version(), "2.4.1") or check_xen_hvm():    return ChipReset().full_lds_reset(...)`
This ensures that:

*   **Xen HVM guests always use the modern `ChipReset` path**, which includes xenstore coordination
*   **Bare-metal systems use legacy paths if driver < 2.4.1**, maintaining backward compatibility
*   **Both paths validate platform (reject ARM)** before proceeding

**Important**: The `ChipReset` class still requires driver >= 2.4.1 even in Xen mode. The Xen check bypasses the version check in the chip-specific classes, but `ChipReset.full_lds_reset()` calls `check_driver_version(operation="reset", minimum_required_version_str="2.4.1")`, which will fail if the driver is too old.

Sources: [tt_tools_common/reset_common/bh_reset.py 92-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L92-L99)[tt_tools_common/reset_common/wh_reset.py 89-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L89-L90)[tt_tools_common/reset_common/chip_reset.py 137](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L137-L137)

## Driver Version Requirements

Virtualization support requires specific minimum driver versions:

| Feature | Minimum Driver Version |
| --- | --- |
| Basic reset operations | 1.26.0 |
| Modern unified reset (ChipReset) | 2.4.1 |
| Xen HVM support | 2.4.1 |
| ASIC_RESET and ASIC_DMC_RESET IOCTLs | 2.4.1 |
| POST_RESET IOCTL | 2.4.1 |

The modern `ChipReset` implementation uses **new IOCTL flags** (`IoctlResetFlags.ASIC_RESET`, `IoctlResetFlags.ASIC_DMC_RESET`, `IoctlResetFlags.POST_RESET`) that were introduced in driver version 2.4.1. These IOCTLs provide better integration with the kernel's PCI hotplug subsystem and are required for Xen HVM coordination.

For details on driver version checking, see [Driver Version Management](https://deepwiki.com/tenstorrent/tt-tools-common/3.1-driver-version-management).

## Summary Table: Platform and Virtualization Support

| Platform/Environment | Supported | Notes |
| --- | --- | --- |
| x86_64 bare-metal | ✅ Yes | Full support with driver >= 1.26.0 |
| x86_64 + driver >= 2.4.1 | ✅ Yes | Modern unified reset path |
| ARM/AArch64 bare-metal | ❌ No | PCIe rescan issues; requires system reboot |
| Xen HVM (x86_64) | ✅ Yes | Requires driver >= 2.4.1, uses xenstore coordination |
| Xen HVM (ARM) | ❌ No | ARM restriction applies even in virtualization |
| KVM/QEMU | ⚠️ Untested | No explicit support code; may work like bare-metal |
| VMware ESXi | ⚠️ Untested | No explicit support code |
| Windows Hyper-V | ❌ No | Linux-only driver |

The reset system is **explicitly designed and tested** for x86_64 bare-metal and Xen HVM environments. Other virtualization platforms may work but are not officially supported or tested.
