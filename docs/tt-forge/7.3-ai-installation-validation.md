# AI Installation Validation

Relevant source files
*   [.github/workflows/call-claude.yml](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/call-claude.yml)

The AI Installation Validation system provides an automated, autonomous mechanism for auditing the `tt-forge` installation process and demo execution using AI agents. It utilizes the Claude Code agent to simulate a user attempting to follow repository documentation, identify friction points, and validate hardware accessibility within containerized environments.

## System Architecture

The validation infrastructure is built on a reusable workflow pattern that bridges high-level natural language instructions with low-level hardware interactions. It ensures that the software stack—from Python dependencies to Tenstorrent kernel drivers—is correctly configured for end-users.

## Core Workflows

### Manual AI Installation Test (`manual-ai-install.yml`)

This workflow acts as the high-level orchestrator for the installation audit. It defines the "Persona" and "Mission" for the AI agent, providing specific constraints to prevent infinite loops while allowing for "common-sense" troubleshooting.

Key constraints defined in the prompt:

*   **Hardware Access:** The agent is informed that `/dev/tenstorrent` is available for hardware-accelerated demos [.github/workflows/manual-ai-install.yml 15-17](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L15-L17)
*   **Troubleshooting Limit:** The agent is permitted 2-3 safe attempts to fix environment issues (e.g., missing system packages) but must document persistent failures rather than brute-forcing them [.github/workflows/manual-ai-install.yml 43-48](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L43-L48)
*   **Safety Boundaries:** The agent is strictly prohibited from modifying repository code, opening PRs, or pushing changes [.github/workflows/manual-ai-install.yml 49](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L49-L49)

### Reusable Claude Caller (`call-claude.yml`)

The `call-claude.yml` workflow provides a standardized interface for invoking the AI agent across different hardware targets (e.g., `n150`, `n300`).

#### Hardware and Container Configuration

The workflow ensures the agent has direct access to Tenstorrent silicon by mapping device nodes and hugepages into the container:

*   **Device Mapping:**`--device /dev/tenstorrent` allows the agent to run actual hardware kernels [.github/workflows/call-claude.yml 68](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/call-claude.yml#L68-L68)
*   **Memory Configuration:** Volumes for `/dev/hugepages` and `/dev/hugepages-1G` are mounted to support the high-performance memory requirements of the TT-MLIR compiler and runtime [.github/workflows/call-claude.yml 69-72](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/call-claude.yml#L69-L72)

#### Agent Permissions

The agent is granted a specific set of capabilities via the `settings` input to `anthropics/claude-code-action`:

| Permission | Description |
| --- | --- |
| `Bash(*)` | Execute shell commands to install packages and run demos |
| `Read(*)` | Inspect source code and documentation |
| `Write(*)` | Create the final markdown report |
| `Edit(*)` | Modify local environment files (e.g., `.bashrc` or local configs) |
| `Glob(*)` / `Grep(*)` | Search the codebase for scripts and error patterns |

## Data Flow and Reporting

The validation process follows a strict data flow to ensure results are captured even if the agent encounters a fatal error during the installation process.

**Data Flow: From Prompt to Artifact**

### Reporting Requirements

The agent is instructed to produce a structured report at `/github/workspace/tt-forge-ai-report.md` containing:

1.   **Success/Fail Summary:** Whether the installation completed and demos ran on hardware [.github/workflows/manual-ai-install.yml 51](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L51-L51)
2.   **Version Tracking:** Identification of the specific `tt-forge` and Tenstorrent package versions installed [.github/workflows/manual-ai-install.yml 53](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L53-L53)
3.   **Friction Log:** A bulleted list of failures and the specific fixes attempted [.github/workflows/manual-ai-install.yml 54](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L54-L54)
4.   **Documentation Feedback:** Qualitative feedback on the clarity of `README.md` and linked documentation [.github/workflows/manual-ai-install.yml 56-58](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/manual-ai-install.yml#L56-L58)

## Cleanup and Maintenance

Because the AI agent operates with elevated permissions inside the container, it may leave behind files with restrictive permissions. The `call-claude.yml` workflow includes a cleanup step that recursively modifies permissions (`chmod -R 777`) on the agent's working directory to ensure subsequent GitHub Action runs are not blocked by permission errors [.github/workflows/call-claude.yml 131-141](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/call-claude.yml#L131-L141)
