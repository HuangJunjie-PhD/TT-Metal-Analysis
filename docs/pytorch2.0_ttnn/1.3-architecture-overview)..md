## System Overview

The pytorch2.0_ttnn repository implements a PyTorch compiler backend that transforms models for execution on Tenstorrent AI accelerators. The architecture follows a standard compiler pipeline: input preprocessing → multi-pass graph transformation → execution on hardware. The system centers on converting PyTorch `aten` operations to Tenstorrent `ttnn` operations while managing data movement, device allocation, and operation compatibility.

The system operates in three major layers:

**Compilation Backend**: When a user calls `torch.compile(model, backend=torch_ttnn.backend)`[backend.py 271-323](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L271-L323) the `ttnn_backend()` function wraps the graph with input aliasing handling and AOT Autograd, then delegates to `aten_backend()`[backend.py 101-259](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L101-L259) The `aten_backend()` function receives a `torch.fx.GraphModule` containing ATen operations and a `TorchTtnnOption` configuration object. A `PassManager`[backend.py 196](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L196-L196) executes 15 sequential transformation passes [backend.py 166-179](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L166-L179): analysis passes gather metadata, `ToTtPass` lowers operations to TTNN (consulting `can_lowering_to_ttnn()` for compatibility), and `AddDataMovePass` inserts `ttnn.from_torch`, `ttnn.to_device`, and `ttnn.to_layout` calls.

**Execution Layer**: The compiled graph executes via the `ttnn` library on `ttnn.Device` (single accelerator) or `ttnn.MeshDevice` (multi-accelerator mesh). TTNN operations perform computation directly on device memory, minimizing data movement.

**Development Infrastructure**: The `compile_and_run` pytest fixture [tests/conftest.py 233-391](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py#L233-L391) orchestrates test execution and captures runtime/schema metrics via `torch_ttnn.metrics`[metrics.py 1-206](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/metrics.py#L1-L206) The `collect_metrics.py` script [tools/collect_metrics.py 1-511](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L1-L511) aggregates these metrics into `InputVarPerOp` objects tracking conversion status per operation variant. Failed conversions feed `generate_guard_function_from_input_var_metric.py` to update blocklists in `to_tt_guard_autogen.py`, which `can_lowering_to_ttnn()`[to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_guard.py#L109-L113) consults during compilation.

## Compilation Pipeline Execution Flow

The compilation pipeline transforms PyTorch FX graphs through approximately 15 sequential passes organized into preprocessing, analysis, optimization, lowering, and post-processing phases. The `PassManager`[backend.py 196](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L196-L196) executes passes in the order defined in `aten_backend()`[backend.py 166-196](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L166-L196) The pipeline follows PyTorch's standard compilation flow with AOT Autograd integration.

### Pass Execution Phases

The compilation pipeline in `aten_backend()`[backend.py 166-196](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L166-L196) executes passes in five phases:

**Preprocessing**: The `insert_clones_for_input_aliasing()`[preprocessing/handle_input_aliasing.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/preprocessing/handle_input_aliasing.py) function addresses input aliasing issues before AOT Autograd processes the graph. The `remove_clones_for_input_aliasing()` function [backend.py 117](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L117-L117) removes these clones before the main pass pipeline begins.

**Analysis Phase**: Three passes gather metadata without modifying operations [backend.py 167-169](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L167-L169):

| Pass Class | Purpose | Key Metadata Set |
| --- | --- | --- |
| `GraphModuleAnalysisPass` | Classifies graph as INFERENCE, TRAIN_FORWARD, or TRAIN_BACKWARD | `graph_module.meta["model_type"]` |
| `InputAnalysisPass` | Tags placeholders with PrimalTag enum | `node.meta["primal_tag"]` (PARAMETER/BUFFER/ARGUMENT) |
| `MultiDeviceShardAnalysisPass` | Determines sharding strategy | `node.meta["is_sharded"]`, `node.meta["shard_dim"]` |

**Optimization Phase**[backend.py 170-173](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L170-L173): Prepares the graph for lowering:

*   `ConstantFoldingPass`[constant_folding_pass.py 39-49](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/constant_folding_pass.py#L39-L49): Evaluates operations with constant inputs at compile time, consulting `can_lowering_to_ttnn()`[constant_folding_pass.py 52-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/constant_folding_pass.py#L52-L53) first
*   `MultiDevicePass`[multi_device_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/multi_device_pass.py): Inserts `shard_tensor`, `replicate_tensor`, `concat_tensor` wrappers
*   `FusionPass`[fusion_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/fusion_pass.py): Merges compatible operations to reduce kernel launches

**Lowering Phase**[backend.py 172-174](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L172-L174): The critical transformation phase:

*   `ToTtPass`[to_tt_pass.py 1308-1348](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_pass.py#L1308-L1348): Converts `aten.*` operations to `ttnn.*` operations via `ReplaceMoreTt.transform()`[to_tt_pass.py 172-176](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_pass.py#L172-L176) and `ReplaceMoreTtManually()`[to_tt_pass.py 440-1307](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_pass.py#L440-L1307)
*   Guard System: `can_lowering_to_ttnn(node)`[to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_guard.py#L109-L113) blocks problematic operation/input combinations by checking `GUARD` dictionary [to_tt_guard_autogen.py 306-310](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_guard_autogen.py#L306-L310)
*   `AddDataMovePass`[add_data_move_pass.py 568-611](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/add_data_move_pass.py#L568-L611): Inserts `ttnn.from_torch`, `ttnn.to_device`, `ttnn.to_layout`, `ttnn.to_torch` calls via `NodeInputAligner.align()`[add_data_move_pass.py 549-565](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/add_data_move_pass.py#L549-L565)

**Post-Processing Phase**[backend.py 175-179](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/backend.py#L175-L179): Cleanup and optimization:

*   `LoadOncePass`[load_once_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/load_once_pass.py): Wraps iteration-invariant operations (parameter loading) in `run_once()` calls
*   `EliminateCoreopsPass`[eliminate_coreops_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/eliminate_coreops_pass.py): Removes redundant helper operations
*   `CSEPass`[torch.fx.passes.dialect.common](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch.fx.passes.dialect.common): Common subexpression elimination
*   `DeallocationPass`[deallocation_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/deallocation_pass.py): Inserts explicit `ttnn.deallocate()` calls for memory management

The metrics pipeline operates in four stages:

1.   **Capture**: The `compile_and_run` fixture [tests/conftest.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py) captures four metric types during test execution:

    *   Runtime metrics (success, execution time, throughput)
    *   Original operation schemas (ATen operations with input shapes)
    *   Compiled operation schemas (TTNN operations with input shapes)
    *   Single-operation test results

2.   **Storage**: Metrics are serialized to `.pickle` files in a structured directory hierarchy:

    *   `metrics/ModelName/*.pickle` for model-level metrics
    *   `metrics-autogen-op/ModelName/*.pickle` for operation-level metrics

        *   `BUG`: Known issue prevents conversion
        *   `UNKNOWN`: Not yet processed

    *   `README.md`: Model support table with status, throughput, accuracy
    *   `docs/OperationsReport.md`: Cumulative operation status across all models

    *   `docs/models/*/input_variations.md`: Per-model operation analysis
    *   `metrics/*/model_run.json`: Structured Pydantic data for automation

This creates transparency into compilation support and automatically generates guard functions for operations that fail conversion.

## Operation Lowering and Guard System

Operation lowering converts PyTorch `aten.*` operations to `ttnn.*` operations through `ToTtPass`[to_tt_pass.py 1308-1348](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/to_tt_pass.py#L1308-L1348) The guard system prevents lowering of operations known to fail or produce incorrect results.

For mesh devices with multiple Tenstorrent accelerators, the system implements data parallel execution by sharding inputs across devices and replicating model parameters.

### Multi-Device Data Flow

**Sources:**

*   [torch_ttnn/passes/analysis/input_analysis_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py)

The multi-device pipeline operates in three stages:

1.   **Input Classification**: `InputAnalysisPass` tags each input using `PrimalTag` enum:

    *   `PARAMETER`: Model weights/biases (replicated to all devices)
    *   `BUFFER`: Batch normalization statistics (replicated to all devices)
    *   `ARGUMENT`: Input data (sharded along batch dimension)

2.   **Distribution Strategy**: `MultiDevicePass` inserts wrapper operations based on tags:

    *   `target_wrappers.replicate_tensor()` for parameters and buffers
    *   `target_wrappers.shard_tensor()` for arguments with `shard_dim=0`

3.   **Hardware Mapping**: `AddDataMovePass` replaces wrappers with actual TTNN mesh operations:

    *   `ttnn.ShardTensorToMesh` with `mesh_mapper`[add_data_move_pass.py 445-450](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/add_data_move_pass.py#L445-L450)
    *   `ttnn.ReplicateTensorToMesh` with `mesh_mapper`[add_data_move_pass.py 453-458](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/add_data_move_pass.py#L453-L458)
    *   `ttnn.ConcatMeshToTensor` for output aggregation [add_data_move_pass.py 475-478](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/add_data_move_pass.py#L475-L478)

The `MultiDeviceShardAnalysisPass`[multi_device_shard_analysis_pass.py 39-65](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/multi_device_shard_analysis_pass.py#L39-L65) propagates sharding information through the graph via node metadata (`is_sharded`, `shard_dim`, `concat_size`) to ensure correct tensor dimensions throughout the computation.
