The benchmark data processing pipeline is responsible for collecting performance metrics from various Tenstorrent projects and normalizing them into a unified format. It transforms project-specific JSON reports into a standardized `CompleteBenchmarkRun` JSONL format suitable for downstream telemetry and analysis.

## Overview

The processing logic resides in `benchmark.py` and is orchestrated by the `collect_data` action. It supports diverse report formats from `tt-forge`, `tt-shield`, `tt-inference-server`, and `vllm`. The pipeline handles model name normalization, hardware metadata extraction, and metric mapping for standard LLM performance indicators such as TTFT (Time to First Token) and TPOT (Time Per Output Token).

## Core Implementation

### Report Discovery and Loading

The process begins with `_get_model_reports`, which walks through the workflow artifacts directory to find JSON files. It expects filenames in the format `<report_name>_<job_id>.json` and extracts the `job_id` to correlate performance data with CI/CD job metadata [.github/actions/collect_data/src/benchmark.py 64-89](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L64-L89)

Key functions include:

*   `create_json_from_report`: The main entry point that iterates through found reports and invokes the mapping logic [.github/actions/collect_data/src/benchmark.py 35-56](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L35-L56)
*   `_load_model_spec_json`: A helper for `tt-shield` benchmarks to load external model configuration metadata that isn't yet embedded in the primary report [.github/actions/collect_data/src/benchmark.py 20-32](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L20-L32)

### Benchmark Data Mappers

The system uses an abstract base class `_BenchmarkDataMapper` to define the interface for project-specific transformations [.github/actions/collect_data/src/benchmark.py 92-95](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L92-L95)

| Class | Purpose | Key Metric Extractions |
| --- | --- | --- |
| `ForgeBenchmarkDataMapper` | Processes reports from the `tt-forge` compiler. | `total_samples`, `total_time`. |
| `ShieldBenchmarkDataMapper` | Processes LLM benchmarks from `tt-shield`. | `mean_ttft_ms`, `mean_tpot_ms`, `mean_tps`, `mean_e2el_ms`. |
| `VllmBenchmarkDataMapper` | Processes vLLM-based inference reports. | Inherits standard LLM metrics; handles specific vLLM metadata. |

### Metric Extraction and Normalization

Metrics are encapsulated in `BenchmarkMeasurement` objects. The `_create_measurements` helper ensures that every metric (e.g., ITL, TPS) is timestamped with the job's start and end times and validated against the Pydantic schema [.github/actions/collect_data/src/benchmark.py 106-139](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L106-L139)

Commonly extracted metrics include:

*   **TTFT**: Time to First Token (ms)
*   **TPOT**: Time Per Output Token (ms)
*   **TPS**: Tokens Per Second
*   **ITL**: Inter-Token Latency
*   **E2EL**: End-to-End Latency (ms)

## Data Models

The output is a `CompleteBenchmarkRun` object, which flattens pipeline, job, and performance metadata into a single record.

### CompleteBenchmarkRun

Key fields include:

*   `github_pipeline_id`: Linkage to the CI/CD run [.github/actions/collect_data/src/pydantic_models.py 168](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/pydantic_models.py#L168-L168)
*   `ml_model_name`: Normalized model name (e.g., "Llama-3.1-70B-Instruct") [.github/actions/collect_data/src/pydantic_models.py 178](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/pydantic_models.py#L178-L178)
*   `device_info`: Hardware configuration details (e.g., `{"device_name": "t3k"}`) [.github/actions/collect_data/src/pydantic_models.py 175-177](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/pydantic_models.py#L175-L177)
*   `measurements`: A list of `BenchmarkMeasurement` objects containing the actual performance numbers [.github/actions/collect_data/src/pydantic_models.py 192](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/pydantic_models.py#L192-L192)

### Model Name Formatting

The `ShieldBenchmarkDataMapper` includes logic in `_format_model_name` to strip repository prefixes (like `meta-llama/`) from model identifiers to ensure consistency across different benchmarking tools [.github/actions/collect_data/src/benchmark.py 369-378](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/benchmark.py#L369-L378)
