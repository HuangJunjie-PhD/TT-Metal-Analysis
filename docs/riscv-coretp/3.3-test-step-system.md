## Purpose and Scope

The Test Step System provides the primitive building blocks for constructing test scenarios in the coretp framework. This document explains the `TestStep` abstraction, the hierarchy of concrete step types, and how steps compose together to form executable test sequences.

For information about how test steps are assembled into complete test scenarios, see [Test Scenario Composition](https://deepwiki.com/tenstorrent/riscv-coretp/3-core-architecture). For detailed reference documentation on specific step types, see [Test Steps Reference](https://deepwiki.com/tenstorrent/riscv-coretp/5-test-steps-reference).

## TestStep Base Abstraction

The `TestStep` class serves as the abstract base for all test step types in the framework. All concrete steps are implemented as frozen (immutable) dataclasses that inherit from `TestStep`, ensuring that step definitions cannot be modified after creation.

Steps are composed into test scenarios by passing them as lists to `TestScenario.from_steps()`. During test execution, steps are processed sequentially, with each step potentially depending on the output of previous steps.

## Memory Management Steps

Memory management steps define and manipulate memory regions within test scenarios. These steps configure virtual memory layouts, page table entries, and memory protection attributes.

**Diagram: Memory Management Step Types**

### Memory

The `Memory` step allocates a memory region with specified size, page size, and protection flags. It supports both physical and virtual address specification.

Key parameters:

*   `size`: Memory region size in bytes
*   `page_size`: Page size enumeration (SIZE_4K, SIZE_2M, SIZE_1G)
*   `flags`: Memory protection flags (VALID, READ, WRITE, EXECUTE)
*   `exclude_flags`: Flags to explicitly exclude
*   `alignment`: Memory alignment requirements
*   `num_pages`: Number of pages to allocate

**Sources:**[coretp/step/memory.py 11-50](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L11-L50)

### CodePage

`CodePage` extends `Memory` to define executable code regions. It contains a list of test steps that represent instructions to be placed in the code page.

**Important:**`CodePage` cannot contain another `CodePage`. Use `Call` to jump to a `CodePage` from another code region.

**Sources:**[coretp/step/memory.py 52-73](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L52-L73)

### ModifyPte

`ModifyPte` alters page table entries at a specific level, enabling tests to create invalid mappings, recursive entries, or custom permission configurations.

**Sources:**[coretp/step/memory.py 75-89](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L75-L89)

### ReadLeafPTE

`ReadLeafPTE` reads the leaf page table entry for a given memory region, returning the PTE value for verification.

**Sources:**[coretp/step/memory.py 91-102](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/memory.py#L91-L102)

## CSR Operation Steps

CSR (Control and Status Register) operations manipulate processor state through privileged registers. The framework provides two CSR step types: `CsrRead` and `CsrWrite`.

**Diagram: CSR Operation Step Structure**

### CsrWrite

`CsrWrite` modifies CSR values using set masks, clear masks, or direct value writes. Only one modification mode can be used per step.

## Computation Steps

Computation steps perform arithmetic operations and load immediate values or addresses into registers.

### Arithmetic

`Arithmetic` performs ALU operations on source operands. It supports both register-register and register-immediate operations.

Common operations include: `add`, `sub`, `and`, `or`, `xor`, `andi`, `ori`, `xori`, `sll`, `srl`, `sra`.

**Sources:**[coretp/step/arithmetic.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/arithmetic.py)

### LoadImmediateStep

`LoadImmediateStep` loads a constant immediate value.

**Sources:**[coretp/step/arithmetic.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/arithmetic.py)[coretp/plans/sstc/sstc_scenarios.py 458-460](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L458-L460)

## Step Integration with Test Scenarios

Steps are integrated into test scenarios using `TestScenario.from_steps()`, which accepts:

*   `id`: Unique scenario identifier
*   `name`: Scenario name
*   `description`: Human-readable description
*   `env`: TestEnvCfg specifying test environment parameters
*   `steps`: Ordered list of TestStep instances
