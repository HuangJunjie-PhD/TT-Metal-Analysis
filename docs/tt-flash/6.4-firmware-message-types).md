## Purpose and Scope

This document catalogs the firmware message types used for communication between tt-flash and the ARC processor on Tenstorrent chips. These message types are defined in platform-specific `fw_defines.yaml` files and establish the protocol for querying chip state, retrieving firmware versions, triggering operations, and validating flash operations.

For information about how these messages are sent and received through hardware interfaces, see [Hardware Communication Interfaces](https://deepwiki.com/tenstorrent/tt-flash/6.3-hardware-communication-interfaces). For chip-specific implementations that use these message types, see [TTChip Class Hierarchy](https://deepwiki.com/tenstorrent/tt-flash/6.1-ttchip-class-hierarchy).

* * *

## Message Protocol Architecture

The firmware message protocol provides a standardized communication channel between tt-flash host software and the ARC processor firmware running on Tenstorrent chips. Each chip architecture (Wormhole, Blackhole, Grayskull) defines its own message type constants in YAML configuration files.

**Sources:**[tt_flash/data/wormhole/fw_defines.yaml 1-8](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml#L1-L8)[tt_flash/data/blackhole/fw_defines.yaml 1-7](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L1-L7)[tt_flash/data/grayskull/fw_defines.yaml 1-7](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/grayskull/fw_defines.yaml#L1-L7)

## Message Type Details

### MSG_TYPE_ARC_STATE0 (0xa0)

**Platform Support:** All (Wormhole, Blackhole, Grayskull)

Queries the ARC processor's state register 0, which contains boot status and readiness information. This message type is used during chip initialization and post-reset validation to determine if the ARC processor has completed its boot sequence.

**Usage Context:** Sent during chip detection and after reset operations to verify the ARC processor is responsive and has reached the expected boot state.

**Sources:**[tt_flash/data/wormhole/fw_defines.yaml 4](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml#L4-L4)[tt_flash/data/blackhole/fw_defines.yaml 4](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L4-L4)[tt_flash/data/grayskull/fw_defines.yaml 4](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/grayskull/fw_defines.yaml#L4-L4)

### MSG_TYPE_ARC_STATE3 (0xa3)

**Platform Support:** All (Wormhole, Blackhole, Grayskull)

Queries the ARC processor's state register 3, which provides extended status information about the firmware's operational state. This message type complements `MSG_TYPE_ARC_STATE0` by providing additional diagnostic information.

**Usage Context:** Used in conjunction with `MSG_TYPE_ARC_STATE0` to perform comprehensive state verification during boot and post-flash validation sequences.

**Sources:**[tt_flash/data/wormhole/fw_defines.yaml 5](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml#L5-L5)[tt_flash/data/blackhole/fw_defines.yaml 5](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L5-L5)[tt_flash/data/grayskull/fw_defines.yaml 5](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/grayskull/fw_defines.yaml#L5-L5)

### MSG_TYPE_FW_VERSION (0xb9)

**Platform Support:** Wormhole, Grayskull

Requests the currently running firmware version from the ARC processor. The response contains version information that tt-flash uses for version checking and compatibility validation before performing flash operations.

**Usage Context:**

*   Called during chip detection to retrieve the running firmware version
*   Used in version comparison logic to determine if flashing is needed
*   Critical for preventing incompatible firmware downgrades

**Not Supported On:** Blackhole (uses alternative mechanism for version retrieval)

**Sources:**[tt_flash/data/wormhole/fw_defines.yaml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml#L6-L6)[tt_flash/data/grayskull/fw_defines.yaml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/grayskull/fw_defines.yaml#L6-L6)

### MSG_TRIGGER_SPI_COPY_LtoR (0x50)

**Platform Support:** Wormhole only

Triggers a hardware-assisted SPI flash copy operation from local flash to remote flash. This message type is specific to multi-die configurations (e.g., NEBULA_X2 boards) where one die's flash must be copied to another die's flash.

**Usage Context:**

*   Sent after successfully flashing the local SPI flash on NEBULA_X2 boards
*   Instructs the firmware to perform an internal copy operation to the remote die
*   Eliminates the need for tt-flash to directly access the remote flash

**Implementation Note:** The firmware handles the copy operation internally, providing progress feedback through state registers.

**Sources:**[tt_flash/data/wormhole/fw_defines.yaml 7](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/wormhole/fw_defines.yaml#L7-L7)

### MSG_CONFIRM_FLASHED_SPI (0xc4)

**Platform Support:** Blackhole only

Performs cryptographic challenge-response validation to confirm that new firmware has been successfully flashed and is currently executing. This message type is part of Blackhole's enhanced security and validation mechanisms.

**Usage Context:**

*   Sent during post-flash validation on Blackhole chips with firmware version 19 or higher
*   Used after chip reset to cryptographically verify the new firmware is running
*   Provides stronger assurance than simple version checking

**Implementation Note:** The challenge-response mechanism prevents false positives from cached version information or incomplete flash operations.

**Sources:**[tt_flash/data/blackhole/fw_defines.yaml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L6-L6)

## Configuration File Format

The `fw_defines.yaml` files follow a simple key-value format where each message type is mapped to its hexadecimal byte value. These files are loaded at runtime by the chip-specific classes (WhChip, BhChip) to obtain the correct message constants for the target platform.

## Platform Evolution

The message type definitions reflect the evolution of Tenstorrent chip architectures:

| Feature | Wormhole | Blackhole | Evolution Notes |
| --- | --- | --- | --- |
| **Basic State Query** | `MSG_TYPE_ARC_STATE0/3` | `MSG_TYPE_ARC_STATE0/3` | Common across all platforms |
| **Version Query** | `MSG_TYPE_FW_VERSION` | Alternative mechanism | BH uses different version retrieval |
| **Multi-Die Support** | `MSG_TRIGGER_SPI_COPY_LtoR` | N/A | WH-specific for NEBULA_X2 |
| **Security Validation** | N/A | `MSG_CONFIRM_FLASHED_SPI` | BH introduces crypto validation |

## Integration with Hardware Communication

The firmware message types are sent through the pyluwen library's ARC messaging interface. The chip-specific classes (WhChip, BhChip) load the appropriate `fw_defines.yaml` file and construct message payloads using the defined constants.
