This document defines the coding standards, style guidelines, and contribution requirements for tt-system-firmware. These standards ensure code quality, maintainability, and consistency across the codebase. For general contribution workflow information, see [Contributing to TT System Firmware](https://deepwiki.com/tenstorrent/tt-system-firmware/9-development-guide). For build and testing setup, see [Getting Started](https://deepwiki.com/tenstorrent/tt-system-firmware/9.1-getting-started).

## Overview

The tt-system-firmware project follows Zephyr RTOS coding standards with additional project-specific requirements. All code contributions must pass automated compliance checks, maintain bisectability, and adhere to specific formatting and documentation standards. The build system enforces warnings-as-errors to maintain code quality.

**Key Requirements:**

*   Follow Zephyr coding style with `clang-format`
*   Use snake_case naming conventions
*   Include copyright headers in all source files
*   Sign all commits with `Signed-off-by` line
*   Pass all compliance checks before merge
*   Build without warnings or errors

## Code Style Requirements

### Formatting Standard

The project uses `clang-format` for automatic code formatting. All C/C++ source files must conform to the Zephyr coding style defined in the upstream Zephyr project.

**Formatting Commands:**

`# Format a single fileclang-format -i path/to/file.c # Format all changed files in a commitgit diff --name-only HEAD^ | grep '\.[ch]$' | xargs clang-format -i`
**Important:** Create separate commits for formatting changes. Do not mix formatting changes with functional changes, as this makes code review difficult.

## Naming Conventions

### Snake Case Standard

Snake case (`snake_case`) shall be used for all identifiers including:

*   Variable names
*   Function names
*   Structure member names
*   File names (except for special cases like CMakeLists.txt)

Camel case (`camelCase` or `PascalCase`) is non-compliant except for external library compatibility.

**Examples:**

| Non-Compliant | Compliant |
| --- | --- |
| `myVariableName` | `my_variable_name` |
| `calculateMaxValue()` | `calculate_max_value()` |
| `struct myDataStruct` | `struct my_data_struct` |
| `fieldValue` | `field_value` |

Sources: [doc/contribute/coding_guidelines/index.rst 83-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/doc/contribute/coding_guidelines/index.rst#L83-L106)

## Type Definitions

### Struct and Enum Definitions

Direct `struct` and `enum` definitions are preferred over `typedef` aliases. This improves code clarity by making the type nature explicit.

**Non-Compliant:**

`typedef struct {    uint32_t field;} foo_t;  /* Avoid this pattern */ typedef enum {    ENTRY} my_enum_e;  /* Avoid this pattern */`
**Compliant:**

`struct foo {    uint32_t field;};  /* Preferred */ enum my_enum {    ENTRY};  /* Preferred */`
### Exception: Function Pointer Typedefs

Function pointer typedefs are acceptable and preferred for readability, particularly in callback interfaces and dispatch tables.

**Acceptable:**

`typedef int (*message_handler_t)(const union request *req, struct response *rsp);typedef void (*callback_func_t)(void *data); struct message_dispatcher {    message_handler_t handler;    callback_func_t callback;};`
Sources: [doc/contribute/coding_guidelines/index.rst 28-82](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/doc/contribute/coding_guidelines/index.rst#L28-L82)

## Commit Standards

### Signed-off-by Requirement

All commits must include a `Signed-off-by` line certifying the Developer Certificate of Origin (DCO). This can be added automatically with the `-s` flag.

**Add to new commit:**

`git commit -s -m "subsystem: description of change"`
**Add to existing commit:**

`git commit --amend -s`
The compliance workflow validates the presence of `Signed-off-by` lines through the Identity check.

**Diagram: Commit Signing Workflow**

Sources: [.github/workflows/compliance.yml 102-104](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L102-L104)[doc/contribute/index.rst 40-43](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/doc/contribute/index.rst#L40-L43)

### Commit Message Format

Follow the Zephyr commit message format:

```
<subsystem>: <short description>

<detailed description explaining what changed and why>

Signed-off-by: Your Name <your.email@example.com>
```

**Example:**

```
smc: add telemetry collection for GDDR temperature

Implement periodic polling of GDDR temperature sensors using
the MRISC firmware interface. Temperature readings are stored
in telemetry registers TAG_GDDR_TEMP_0 through TAG_GDDR_TEMP_7.

This enables host software to monitor memory thermal status
for thermal management decisions.

Signed-off-by: John Developer <jdeveloper@tenstorrent.com>
```

### Bisectability

Each commit must maintain bisectability - the firmware must build successfully after applying each commit. This allows `git bisect` to function correctly when debugging regressions.

**Requirements per commit:**

*   SMC firmware builds without errors
*   DMC firmware builds without errors
*   All unit tests pass (if applicable)
*   No compiler warnings (due to warnings-as-errors)

Sources: [doc/contribute/index.rst 34-44](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/doc/contribute/index.rst#L34-L44)

## Pull Request Guidelines

### PR Size and Scope

PRs should contain the smallest logical change possible. Large PRs are difficult to review and prone to introducing defects.

**Recommended approach:**

*   One feature per PR
*   Split implementation across multiple commits
*   Each commit adds incremental functionality
*   Final commit enables the feature

**Anti-patterns to avoid:**

*   Mixing unrelated changes
*   Reformatting existing code in feature PRs
*   Combining bug fixes with new features
*   Large refactoring changes

### Rebase Workflow

The project does not allow merge commits. All PRs must be rebased onto the target branch before merging.

**Diagram: PR Rebase Workflow**

**Commands for interactive rebase:**

`# Rebase onto target branchgit fetch origingit rebase origin/main # Interactive rebase to fix specific commitgit rebase -i <commit-hash>^ # After making changesgit add <modified-files>git rebase --continue # Force push to update PRgit push --force`

## Compliance Checks

### Automated Check Pipeline

The compliance workflow runs on all pull requests and validates multiple aspects of code quality and standards adherence.

**Diagram: Compliance Check Pipeline**

### Check Categories

| Check | Type | Description | Action on Failure |
| --- | --- | --- | --- |
| ClangFormat | Warning | Validates code formatting | Warning annotation |
| Identity | Error | Verifies Signed-off-by presence | Blocks merge |
| Gitlint | Error | Validates commit message format | Blocks merge |
| Checkpatch | Error | Linux kernel style checks | Blocks merge |
| DeviceTree | Error | Validates device tree syntax | Blocks merge |
| Kconfig | Error | Validates Kconfig syntax | Blocks merge |

**Excluded checks:**

*   `BinaryFiles` - Binary blobs are allowed
*   `KconfigBasicNoModules` - Module Kconfig is allowed
*   `SysbuildKconfigBasic` - Sysbuild Kconfig is allowed
*   `LicenseAndCopyrightCheck` - Handled by separate workflows

Sources: [.github/workflows/compliance.yml 1-157](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L1-L157)

### Running Compliance Checks Locally

`# Set up environmentwest init -l tt-system-firmwarewest update # Run compliance checks on current branch against maincd tt-system-firmware../zephyr/scripts/ci/check_compliance.py -c origin/main.. # Check specific commit range../zephyr/scripts/ci/check_compliance.py -c HEAD~3..HEAD # Exclude specific checks../zephyr/scripts/ci/check_compliance.py -e Identity -c origin/main..`
The script generates report files for each check (e.g., `ClangFormat.txt`, `Identity.txt`). Review these files to identify issues.

Sources: [.github/workflows/compliance.yml 89-108](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml#L89-L108)
