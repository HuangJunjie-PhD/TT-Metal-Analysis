# Board Identification

Relevant source files
*   [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py)
*   [tt_flash/utility.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py)

This document describes how tt-flash identifies and names different Tenstorrent board types through a two-stage resolution system that parses 64-bit Board IDs into codenames and then translates them into user-facing display names. For information about platform-specific chip implementations and features, see [Wormhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.3-wormhole-platform) and [Blackhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.4-blackhole-platform). For a complete list of supported board variants, see [Platform Support Matrix](https://deepwiki.com/tenstorrent/tt-flash/4.1-platform-support-matrix).

## Overview

Board identification enables tt-flash to distinguish between 14+ distinct hardware variants across Wormhole and Blackhole architectures. The system extracts a Unique Part Identifier (UPI) and revision from a 64-bit Board ID value, maps it to an internal codename, and optionally translates that to a marketing-aligned public name for user display.

**Sources**: [tt_flash/utility.py 39-106](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L39-L106)[tt_flash/chip.py 202-203](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L202-L203)

## Board ID Structure

The 64-bit Board ID follows a structured format where specific bit ranges encode different hardware attributes:

**Bit Field Extraction**:

*   **UPI** (Unique Part Identifier): 20 bits at positions 36-55, extracted via `(board_id >> 36) & 0xFFFFF`
*   **Revision**: 4 bits at positions 32-35, extracted via `(board_id >> 32) & 0xF`

The UPI serves as the primary lookup key, with revision providing additional disambiguation for board families like E300 that have multiple hardware revisions.

**Sources**: [tt_flash/utility.py 39-58](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L39-L58)

## Resolution Flow

**Function Signatures**:

*   `get_board_type(board_id: int, from_type: bool = False) -> Optional[str]` - Primary resolution function
*   `change_to_public_name(codename: str) -> str` - Optional public name translation

**Sources**: [tt_flash/utility.py 39-106](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L39-L106)[tt_flash/utility.py 108-130](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L108-L130)

## Codename to Public Name Translation

The `change_to_public_name()` function provides a user-friendly translation layer for display purposes:

### Translation Table

| Codename | Public Name | Notes |
| --- | --- | --- |
| E300_105 | e150 | Marketing name |
| E300_X2 | e300 | X2 variant |
| E75 | e75 | Direct mapping |
| NEBULA_X1 | n150 | Blackhole NEBULA |
| NEBULA_X2 | n300 | Higher-capacity NEBULA |
| WH_UBB | Galaxy Wormhole | Galaxy system with WH chips |
| P100-1 | p100 | P-series naming |
| P150A-1 | p150a | Revision-specific |
| P150B-1 | p150b | Revision-specific |
| P150C-1 | p150c | Revision-specific |
| P300A | p300 | P300 variants map to p300 |
| P300B | p300 | P300 variants map to p300 |
| P300C | p300 | P300 variants map to p300 |
| GALAXY-1 | Galaxy Blackhole | Galaxy system with BH chips |

**Unmapped Behavior**: If no public name mapping exists, `change_to_public_name()` returns the codename unchanged.

**Sources**: [tt_flash/utility.py 108-130](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L108-L130)

## Code Integration

### TTChip Interface

All chip objects expose board identification through the `TTChip` base class:

**Usage Pattern**:

`chip = WhChip(pci_chip)  # or BhChipboard_id = chip.board_type()  # Returns 64-bit int from PciChipcodename = get_board_type(board_id)  # Resolves to "E300_R2", "WH_UBB", etc.public_name = change_to_public_name(codename)  # Translates to "e150", "Galaxy Wormhole", etc.`
**Sources**: [tt_flash/chip.py 113-203](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L113-L203)

### Detection and Enumeration

The chip detection flow automatically retrieves Board IDs during enumeration:

During detection, each chip's Board ID is accessible but not automatically parsed. Applications call `chip.board_type()` and `get_board_type()` as needed for display or filtering.

**Sources**: [tt_flash/chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L308-L374)

## Special Cases

### E300 Revision Handling

The E300 board family (UPI 0x01) requires revision-based disambiguation:

**Implementation Details**:

*   Revision field is **required** for UPI 0x01
*   Revisions 0x3 and 0x4 both map to E300_R3
*   If `from_type=True` parameter is set, revision logic is skipped (returns None)
*   E300_R2 and E300_R3 codenames have no public name mappings

**Sources**: [tt_flash/utility.py 60-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L60-L69)

### P-Series Consolidation

P300 variants (P300A, P300B, P300C) all map to the same public name "p300":

`"P300": "p300","P300A": "p300", "P300C": "p300",`
This consolidation suggests hardware variants that are functionally similar from a user perspective.

**Sources**: [tt_flash/utility.py 120-122](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L120-L122)

### Unknown Board IDs

When an unrecognized UPI is encountered:

*   `get_board_type()` returns `None`
*   Downstream code must handle `None` gracefully
*   No exception is raised, allowing forward compatibility with new board types

**Sources**: [tt_flash/utility.py 104-105](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L104-L105)

## Integration with Flashing Operations

Board identification affects firmware flashing in several ways:

1.   **Firmware Selection**: The manifest.json in firmware bundles may contain board-specific images indexed by codename
2.   **Parameter Masking**: Different board types may have different parameter requirements (see [Parameter Handling and Masks](https://deepwiki.com/tenstorrent/tt-flash/5.4-parameter-handling-and-masks))
3.   **Reset Procedures**: Platform-specific resets vary by board type (see [Reset Procedures](https://deepwiki.com/tenstorrent/tt-flash/5.6-reset-procedures))
4.   **Validation**: Post-flash validation may check that the correct firmware is running for the identified board type
