## Purpose and Scope

This document explains the input variation tracking system and conversion status classification used to assess the quality and completeness of operation lowering in the pytorch2.0_ttnn compiler. The system tracks each unique combination of input types, shapes, and values observed during model execution, classifies the conversion success for each variation, and computes generality metrics. This information drives test generation, guard blocklist creation, and documentation reporting.

For details on how input variations are initially collected during graph execution, see [Metrics Collection and Schema Tracking](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.1-metrics-collection-and-schema-tracking). For how this data is transformed into user-facing documentation, see [Documentation Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.3-documentation-generation). For how failed conversions feed into the guard system, see [Guard Blocklist Management](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.4-guard-blocklist-management).

## Input Variation Concept

An **input variation** represents a unique signature of an operation invocation, capturing the operation name along with the types, shapes, and values of all arguments. During model compilation and execution, the system collects these variations to understand the diversity of operation usage patterns.

### Why Input Variations Matter

Operations in deep learning models can be invoked with widely varying inputs. For example, `aten.add` might be called with:

*   Different tensor ranks: `<[1, 2, 3]>` vs `<[4, 5, 6, 7]>`
*   Different scalar types: `int` vs `float` vs `Tensor`
*   Different broadcasting patterns
*   Different memory layouts

Each variation may require different lowering logic or expose different bugs. The input variation system enables fine-grained tracking of which patterns are successfully converted and which require fallback to PyTorch.

### Input Variation Collection

Input variations are collected using the `InputVariation` and `ConvertedInput` classes from the metrics module. During graph transformation, these objects capture operation schemas with actual runtime metadata.

## Conversion Status Classification

The system uses a `ConversionStatus` enum to classify the outcome of lowering each input variation from PyTorch operations to TTNN operations.

### ConversionStatus Enum

`class ConversionStatus(Enum):    DONE = (1,)      # Conversion successful    FALLBACK = (2,)  # Partial conversion (aten -> aten)    BUG = (3,)       # Known issue    NONE = (4,)      # No conversion occurred    UNKNOWN = (5,)   # Not yet processed    REMOVED = (6,)   # Op eliminated by optimization`
**Sources:**[tools/collect_metrics.py 152-159](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L152-L159)

### Status Definitions

| Status | Meaning | Example Scenario |
| --- | --- | --- |
| `DONE` | Operation successfully converted to TTNN | `aten.add.Tensor` → `ttnn.add` |
| `FALLBACK` | Operation converted to another aten operation | `aten.mul.Scalar` → `aten.mul.Tensor` |
| `BUG` | Known conversion issue (reserved for future use) | Tracked issue blocks conversion |
| `NONE` | Operation unchanged in compiled graph | Same `aten.add.Tensor` in both graphs |
| `UNKNOWN` | Status not yet determined | Initial state before analysis |
| `REMOVED` | Operation eliminated during optimization | Constant folding removed the op |

**Sources:**[tools/collect_metrics.py 152-159](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L152-L159)

## Status Determination Logic

The system determines conversion status by comparing the original and compiled graphs, examining whether operations have been transformed and whether they still exist.

### Status Classification Flow

**Sources:**[tools/collect_metrics.py 237-289](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L237-L289)

### Key Decision Points

1.   **Original Inputs Metadata**: Compiled nodes preserve `original_inputs` metadata linking back to the original operation signature. This enables tracking transformations.

2.   **Opname Comparison**: If the operation name changed, the system checks whether it was converted to TTNN or remained as aten.

3.   **Unprocessed Ops Set**: Operations that appear unchanged in the compiled graph are held in an `unprocessed_compiled_ops` set for later analysis.

4.   **Removal Detection**: If an input variation disappears from the compiled graph and the compilation succeeded, it's classified as `REMOVED` rather than `UNKNOWN`.

**Sources:**[tools/collect_metrics.py 246-283](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L246-L283)

### Example Scenarios

`# Scenario 1: Successful conversion# Original: aten.sqrt.default with Tensor<[1, 256, 768]># Compiled: ttnn.sqrt with original_inputs = {opname: "aten.sqrt.default", ...}# Result: Status = DONE # Scenario 2: Partial conversion# Original: aten.mul.Scalar with Tensor<[1, 512]>, scalar = 2.0# Compiled: aten.mul.Tensor with original_inputs = {opname: "aten.mul.Scalar", ...}# Result: Status = FALLBACK # Scenario 3: No conversion# Original: aten.view.default with Tensor<[1, 768]># Compiled: aten.view.default (same opname + input variation)# Result: Status = NONE # Scenario 4: Operation removed# Original: aten.add.Tensor with constant operands# Compiled: Operation not present (constant folding)# Result: Status = REMOVED`

## Generality Score and Completion Metrics

The `InputVarStatus` class provides methods to compute aggregate metrics that quantify how well an operation is supported across its input variations.

### Completion Status Symbol

The system assigns a visual symbol to each operation based on its overall conversion success:

`def get_completion_status_for(self) -> str:    done = self.get_conversion_status_count_for(ConversionStatus.DONE)    removed = self.get_conversion_status_count_for(ConversionStatus.REMOVED)    completed = done + removed    total = len(self.values())        if completed == total:        return "✅"  # All variations converted or removed    elif completed < total and completed > 0:        return "🚧"  # Partial conversion    else:        return "✘"  # No successful conversions`
**Sources:**[tools/collect_metrics.py 174-190](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L174-L190)

### Generality Score

The **generality score** quantifies the fraction of input variations that are successfully handled:

```
Generality Score = (DONE count + REMOVED count) / Total variations
```

This metric ranges from 0.0 (no variations converted) to 1.0 (all variations converted). It appears in the operations report and helps prioritize which operations need additional lowering support.

**Sources:**[tools/collect_metrics.py 202-210](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L202-L210)

### Metrics Computation Flow

**Sources:**[tools/collect_metrics.py 192-210](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L192-L210)[tools/collect_metrics.py 389-409](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L389-L409)

### Example Metrics Output

For an operation `aten.add.Tensor` with 10 input variations:

*   8 variations: Status = `DONE`
*   1 variation: Status = `FALLBACK`
*   1 variation: Status = `NONE`

The metrics would be:

*   **Converted**: 8
*   **Completed**: 8 (since REMOVED = 0)
*   **Generality Score**: 0.8
*   **Completion Symbol**: 🚧 (partial)

## Integration with Guard Blocklist System

Input variations that fail testing feed into the automated guard generation system, preventing future compilation attempts of known problematic patterns.

### Guard Generation Flow

**Sources:**[tools/generate_guard_function_from_input_var_metric.py 11-97](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/generate_guard_function_from_input_var_metric.py#L11-L97)

### Blocklist Threshold Logic

The guard generator applies a threshold to determine which variations should be blocked:

`def need_block(input_var: dict, threshold: list):    for key in threshold:        if isinstance(input_var[key], bool) and input_var[key] == False:            return True  # Compilation failed        elif not (float(input_var[key]) >= 0.99):            return True  # Low accuracy`
By default, the threshold checks `["run", "accuracy"]`, meaning variations are blocked if:

*   The test run failed (`run == False`)
*   The accuracy is below 99% (`accuracy < 0.99`)

**Sources:**[tools/generate_guard_function_from_input_var_metric.py 40-57](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/generate_guard_function_from_input_var_metric.py#L40-L57)

### Single Metrics Integration

The `InputVarPerOp` class includes methods to append isolated test results to the input variation reports:

`def append_single_status(self, opname, input_variations, input_vars_dict):    # Adds columns: "Isolated", "PCC", "Host"    # Shows per-variation test results`
This enables side-by-side comparison of:

*   **E2E Status**: Conversion status in full model compilation
*   **Isolated Status**: Whether the variation passes in isolation
*   **PCC**: Pearson correlation coefficient for accuracy
*   **Host**: Number of fallbacks to host operations

**Sources:**[tools/collect_metrics.py 324-369](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L324-L369)

## Merging Input Variations Across Models

The `merge` method allows combining input variation data from multiple models to build a comprehensive view of operation support:

`def merge(self, other: "InputVarPerOp"):    for opname, input_var in other.items():        for input, status in input_var.items():            # Don't overwrite existing with UNKNOWN status            if self[opname][input] == ConversionStatus.UNKNOWN:                self[opname][input] = status`
This enables:

*   **Cumulative tracking**: Build a complete picture across all tested models
*   **Cross-model analysis**: Identify patterns that work in some contexts but not others
*   **Test consolidation**: Generate merged test suites covering all observed variations

**Sources:**[tools/collect_metrics.py 307-319](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L307-L319)

## Summary

The input variation analysis system provides fine-grained tracking of operation lowering quality by:

1.   **Collecting** unique input signatures from model execution
2.   **Classifying** conversion success using the `ConversionStatus` enum
3.   **Computing** generality scores and completion metrics
4.   **Generating** automated tests for observed patterns
5.   **Creating** guard blocklists for failed variations
6.   **Producing** comprehensive markdown reports

This feedback loop continuously improves the compiler's robustness by ensuring newly discovered usage patterns are tested, documented, and either successfully converted or explicitly guarded against.
