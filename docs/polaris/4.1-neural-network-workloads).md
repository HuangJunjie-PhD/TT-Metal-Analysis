This document covers the neural network workload models available in the Polaris simulation framework. It explains the structure and implementation of TTSIM workloads, IP workloads, and their configuration systems. For information about ONNX model integration, see [ONNX Integration](https://deepwiki.com/tenstorrent/polaris/4.2-onnx-integration). For detailed examples of specific models, see [Specific Model Examples](https://deepwiki.com/tenstorrent/polaris/4.3-specific-model-examples).

## Workload Architecture Overview

The Polaris framework supports two primary types of neural network workloads: TTSIM workloads that are implemented directly using the simulation framework, and IP workloads that are pre-configured models with specific parameter sets.

### Workload Type Hierarchy

**Sources:**[config/ip_workloads.yaml 1-50](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/config/ip_workloads.yaml#L1-L50)[tests/test_simnn.py 1-36](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tests/test_simnn.py#L1-L36)

## TTSIM Framework Workloads

TTSIM workloads are neural network models implemented directly using the simulation framework's operations and graph construction system. These workloads provide fine-grained control over model architecture and simulation behavior.

### BasicLLM Implementation

The `BasicLLM` class represents a configurable large language model implementation that demonstrates the TTSIM workload pattern:

**Sources:**[tests/test_simnn.py 10-36](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tests/test_simnn.py#L10-L36)[workloads/BasicLLM.py](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/workloads/BasicLLM.py)

### TTSIM Workload Configuration

TTSIM workloads are configured with the following parameters:

| Parameter | Description | Example |
| --- | --- | --- |
| `vocab_sz` | Vocabulary size | 100 |
| `drop_prob` | Dropout probability | 0.1 |
| `bs` | Batch size | 7 |
| `nL` | Number of layers | 4 |
| `nH` | Number of attention heads | 3 |
| `dE` | Embedding dimension | 15 |
| `nW` | Window size | 11 |
| `attn_type` | Attention type | 'bidir' |
| `norm_type` | Normalization type | 'layer' |
| `positional_encoding` | Positional encoding method | 'learned' |
| `use_segment_embedding` | Enable segment embeddings | true |

**Sources:**[tests/test_simnn.py 11-23](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tests/test_simnn.py#L11-L23)

## IP Workloads

IP workloads are pre-configured neural network models that provide specific implementations for common architectures. These are defined in the `ip_workloads.yaml` configuration file.

### Available IP Models

**Sources:**[config/ip_workloads.yaml 1-50](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/config/ip_workloads.yaml#L1-L50)

### IP Workload Configuration Structure

Each IP workload follows a consistent configuration pattern:

`- api: TTSIM  name: MODEL_NAME  basedir: workloads  module: ClassName@filename.py  params: { shared_parameters }  instances:    instance_name: { specific_parameters }`
**Sources:**[config/ip_workloads.yaml 4-50](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/config/ip_workloads.yaml#L4-L50)

ResNet50 workloads use the following configuration pattern:

`params:   layers: [3,4,6,3]  num_classes: 1000  num_channels: 3  use_adaptive_pool: true  init_stride: 4  bs: 1instances:  rn50_b1_hd: { img_height: 1024, img_width: 1024 }  rn50_b1_uhd: { img_height: 2828, img_width: 2828 }`
**Sources:**[config/ip_workloads.yaml 27-33](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/config/ip_workloads.yaml#L27-L33)

`params:  is_train_depth: false  training: false  num_cameras: 1  num_sweeps: 2  head_conf_yaml: config/bevdepth_cfgs/bevdepth_head.yaml  backbone_conf_yaml: config/bevdepth_cfgs/bevdepth_backbone.yaml`

## Integration with Simulation Framework

Neural network workloads integrate with the broader simulation system through:

1.   **Functional Operations**: Using `ttsim.front.functional.op` for tensor operations
2.   **Graph Construction**: Building `WorkloadGraph` objects for simulation execution
3.   **Type System**: Leveraging the simulation type system for data formats and math fidelity
4.   **Configuration Management**: Integration with the YAML-based configuration system
