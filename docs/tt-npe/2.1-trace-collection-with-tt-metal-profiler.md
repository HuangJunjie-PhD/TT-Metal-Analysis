## Purpose and Scope

This page documents how to instrument tt-metal code and collect NoC event traces using the built-in device profiler. Trace collection is the first mandatory step in the NoC trace analysis pipeline (section 2). The captured traces contain detailed records of all NoC read/write/multicast operations executed during kernel runtime, which tt-npe ingests via `createWorkloadFromJSON` for performance simulation.

This page focuses on the **Input & Preprocessing** stage of the data flow pipeline. For subsequent stages, see page 2.2 (Trace Preprocessing) and page 2.3 (Batch Analysis).

* * *

## Overview

The tt-metal device profiler captures NoC events by instrumenting the dataflow API (`noc_async_read`, `noc_async_write`, `noc_async_write_multicast`, etc.) at runtime. Event records are buffered in device DRAM during execution, then extracted to JSON files via `tt::tt_metal::DumpDeviceProfileResults()`.

**Diagram: NoC Event Capture and Ingestion Flow**

The profiler operates in two modes:

1.   **TTNN Integration**: Automatic trace collection using `profile_this.py` wrapper (recommended)
2.   **Raw tt-metal**: Manual environment variable configuration for standalone executables

## Environment Variables

The device profiler is controlled by environment variables that must be set before running the workload:

| Variable | Required | Purpose | Recommended Value |
| --- | --- | --- | --- |
| `TT_METAL_DEVICE_PROFILER` | Yes | Enable device profiler | `1` |
| `TT_METAL_DEVICE_PROFILER_NOC_EVENTS` | Yes | Enable NoC event capture | `1` |
| `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT` | Yes | DRAM buffer size for events | `10000` |
| `TT_METAL_PROFILER_SYNC` | Recommended | Enable device synchronization | `1` |

### Buffer Sizing

The `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT` parameter controls the amount of DRAM allocated per device for storing event records. The value `10000` provides sufficient space for approximately 5-100 ops, depending on NoC activity per op. Workloads with more ops may exhaust DRAM and lose events.

## TTNN Integration (profile_this.py)

For TTNN-based workloads, `profile_this.py` (located at `tt_metal/tools/profiler/profile_this.py`) provides integrated trace collection with automatic environment variable configuration and post-processing:

`TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT=10000 \  tt_metal/tools/profiler/profile_this.py \  --collect-noc-traces \  -c 'pytest path/to/test.py' \  -o output_directory`
### Command Line Arguments

| Argument | Purpose |
| --- | --- |
| `--collect-noc-traces` | Enables NoC event capture mode (sets `TT_METAL_DEVICE_PROFILER_NOC_EVENTS=1`) |
| `-c '<command>'` | Specifies the workload command to profile (e.g., `pytest`, Python script) |
| `-o <dir>` | Output directory for all profiler results |

## Raw tt-metal Executable Tracing

Standalone tt-metal programs (without TTNN) can be traced by manually configuring environment variables:

### Prerequisites

1.   **Build with profiler support**: tt-metal must be compiled with profiler enabled:

`./build_metal.sh --enable-profiler ...`
2.   **Instrument code**: The executable must call the profiler dump function:

`tt::tt_metal::DumpDeviceProfileResults(device, program);`

### Execution

`TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT=10000 \TT_METAL_DEVICE_PROFILER_NOC_EVENTS=1 \TT_METAL_DEVICE_PROFILER=1 \  ./path/to/executable`
### Output Location

Raw traces are written to:

```
<tt-metal-root>/generated/profiler/.logs/noc_trace_dev*.json
```

**Note:** Raw metal traces lack operation IDs and names that TTNN provides, making batch analysis more limited. For batch processing, use `--group_as_metal_traces` flag with `npe_analyze_noc_trace_dir.py`.

**Sources:**[docs/src/getting_started.md 91-106](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/docs/src/getting_started.md?plain=1#L91-L106)[README.md 98-107](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/README.md?plain=1#L98-L107)

## NoC Trace File Format

Each device generates a JSON file containing an array of NoC event objects. The filename encodes device ID, operation name, and operation ID:

```
noc_trace_dev<device_id>_<op_name>_ID<op_id>.json
noc_trace_dev<device_id>_<op_name>_ID<op_id>_traceID<trace_id>.json  # For traced runs
```

### Filename Parsing and Operation Grouping

The filename structure is parsed by `extractDataFromFilename` in [tt_npe/py/util/npe_analyze_noc_trace_dir.py 264-274](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L264-L274):

**Diagram: Trace Filename Parsing and Operation Grouping**

**program_runtime_id Encoding:**

| Bits | Field | Description |
| --- | --- | --- |
| 31-10 (21 bits) | `ttnn_op_id` | TTNN operation identifier |
| 9-0 (10 bits) | `device_id` | Device identifier (0-1023) |

The `OpUID` class ([tt_npe/py/util/npe_analyze_noc_trace_dir.py 126-138](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L126-L138)) combines `ttnn_op_id` and `metal_trace_id` to uniquely identify operations across trace collection runs. The grouping logic differs for TTNN vs raw metal traces:

*   **TTNN traces** (`group_traces_ttnn`): Groups by `OpUID`, checking for duplicate device IDs ([tt_npe/py/util/npe_analyze_noc_trace_dir.py 333-362](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L333-L362))
*   **Metal traces** (`group_traces_metal`): Groups by aligned `program_runtime_id` indices ([tt_npe/py/util/npe_analyze_noc_trace_dir.py 277-319](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L277-L319))

### Event Schema

Each event record contains:

| Field | Type | Description |
| --- | --- | --- |
| `timestamp` | int64 | Cycle count when event occurred |
| `type` | string | Event type (READ, WRITE_, WRITE_MULTICAST, etc.) |
| `sx`, `sy` | int64 | Source core coordinates |
| `dx`, `dy` | int64 | Destination core coordinates (unicast) |
| `mcast_start_x/y`, `mcast_end_x/y` | int64 | Multicast rectangle (if applicable) |
| `num_bytes` | int64 | Transfer size in bytes |
| `noc` | string | NoC network used (NOC_0 or NOC_1) |
| `proc` | string | Processor type (BRISC, NCRISC, TRISC0, etc.) |
| `src_device_id` | int64 | Source device ID (multi-chip) |
| `dst_device_id` | int64 | Destination device ID (multi-chip) |
| `fabric_send` | object | Fabric routing metadata (if cross-chip) |
| `zone`, `zone_phase` | string | Profiling zone information |

### Supported Event Types

The trace ingestion pipeline recognizes the following NoC event types:

`const boost::unordered_flat_set<std::string_view> SUPPORTED_NOC_EVENTS = {    "READ",    "READ_SET_STATE",    "READ_WITH_STATE",    "READ_WITH_STATE_AND_TRID",    "READ_DRAM_SHARDED_SET_STATE",    "READ_DRAM_SHARDED_WITH_STATE",    "WRITE_",    "WRITE_MULTICAST",    "WRITE_SET_STATE",    "WRITE_WITH_STATE",    "FABRIC_UNICAST_WRITE",    "FABRIC_UNICAST_INLINE_WRITE",    "FABRIC_UNICAST_ATOMIC_INC",    "FABRIC_FUSED_UNICAST_ATOMIC_INC",    "FABRIC_UNICAST_SCATTER_WRITE"};`
**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 314-330](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L314-L330)

### State Management Events

Traces include special events for reducing JSON size by saving repeated coordinates:

*   **SET_STATE events**: Store source/destination coordinates for subsequent transfers

`{"type": "READ_SET_STATE", "sx": 1, "sy": 2, "dx": 5, "dy": 6, ...}`
*   **WITH_STATE events**: Reuse previously saved coordinates

`{"type": "READ_WITH_STATE", "num_bytes": 4096, ...}`

The ingestion pipeline maintains separate state for READ and WRITE event families:

## Device Synchronization

Multi-device workloads require synchronized device clocks for accurate trace analysis. Device synchronization is enabled via `TT_METAL_PROFILER_SYNC=1`.

### Synchronization Check

The trace analysis pipeline checks for the presence of `sync_device_info.csv`:

`if not os.path.isfile(os.path.join(noc_trace_dir, "sync_device_info.csv")):    log_warning("Device sync isn't turned on. This will lead to inaccurate traces " +                "for mesh device workloads! If you are tracing a workload on a mesh " +                "device, please turn on device sync by setting TT_METAL_DEVICE_PROFILER=1 " +                "and TT_METAL_PROFILER_SYNC=1 then re-collect traces")`
Without synchronization, timestamp alignment across devices may be incorrect, leading to inaccurate multi-chip simulation results.

**Sources:**[tt_npe/py/util/npe_analyze_noc_trace_dir.py 370-374](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L370-L374)

## Important Considerations

### DRAM Buffer Limitations

**Constraint:** Each device has finite DRAM allocated for event storage. Workloads with high NoC activity can exhaust this buffer.

**Symptoms:**

*   Missing events at the end of execution
*   Incomplete traces for later operations
*   Events dropped silently

**Mitigation:**

*   Keep `iterations=1` in pytest fixtures when possible
*   Profile subgraphs separately rather than entire models
*   Increase `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT` if DRAM allows
*   Limit the number of profiled operations (approximately 5-100 ops per run)

**Sources:**[docs/src/getting_started.md 42-46](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/docs/src/getting_started.md?plain=1#L42-L46)

### Operation Grouping

The trace analysis pipeline groups traces by operation ID to correlate events across devices:

`def get_ttnn_op_id(program_runtime_id):    # program_runtime_id: ttnn_op_id (21 bits) | device_id (10 bits)    return program_runtime_id >> 10 def get_device_id(program_runtime_id):    # program_runtime_id: ttnn_op_id (21 bits) | device_id (10 bits)    return program_runtime_id & 0x3FF  # mask lower 10 bits`
This encoding allows automatic grouping of per-device traces for the same logical operation, which is essential for multi-device simulations.

**Sources:**[tt_npe/py/util/npe_analyze_noc_trace_dir.py 314-324](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L314-L324)[tt_npe/py/util/npe_analyze_noc_trace_dir.py 326-355](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L326-L355)

### Golden Cycle Computation in Workload Ingestion

The trace ingestion pipeline computes "golden cycles" from trace data - the actual execution time observed on hardware. This is performed by `computeGoldenCyclesAndT0` during workload creation from NoC traces.

**Diagram: Golden Cycle Extraction and Accuracy Computation**

**Implementation Details:**

The `t0_timestamp` (global minimum timestamp) serves as the time origin for all transfers in the workload. Per-device execution windows are computed and stored via `npeWorkload::setGoldenResultCycles` ([tt_npe/cpp/src/npeWorkload.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp)).

**20-Cycle Overhead:** A constant 20-cycle overhead is subtracted from `max_kernel_cycles` to account for the latency between the last NoC event being logged and kernel completion. This prevents overestimation of execution time.

**Accuracy Calculation:** During simulation finalization, `npeStats::finishSimulation` ([tt_npe/cpp/src/npeStats.cpp 62-85](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L62-L85)) computes `cycle_prediction_error` by comparing simulated `estimated_cycles` against hardware-observed `golden_cycles`.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 221-269](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L221-L269)[tt_npe/cpp/src/npeStats.cpp 62-85](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L62-L85)[tt_npe/cpp/src/npeStats.cpp 148-149](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L148-L149)

### Zone Profiling and Hierarchical Context

Traces may include hierarchical profiling zones that mark logical execution phases (e.g., loop iterations, operator sub-phases). Zones provide temporal context for transfers in timeline visualization.

**Diagram: Zone Extraction and Timeline Integration**

**npeZone Structure:**

`struct npeZone {    CycleCount timestamp;     // Cycle count when zone starts/ends    std::string zone;         // Zone identifier (e.g., "matmul_inner_loop")    ZonePhase phase;          // ZONE_START or ZONE_END};`
**Zone Processing Pipeline:**

1.   **Extraction** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 271-289](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L271-L289)): `extractZones` parses `zone` and `zone_phase` fields from trace events, grouping by `CoreProc` (device_id, row, col, processor).

2.   **Storage** ([tt_npe/cpp/include/npeWorkload.hpp 112-114](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L112-L114)): Zones are stored in `npeWorkload::zones_per_core_proc` map for later retrieval.

3.   **Transfer Attachment**: Each transfer's `enclosing_zone_path` ([tt_npe/cpp/src/npeWorkloadIngest.cpp 447-475](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L447-L475)) records the nested zone hierarchy at the time of the NoC event.

4.   **Serialization** ([tt_npe/cpp/src/npeStats.cpp 596-658](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L596-L658)): `v1TimelineSerialization` emits zones as nested JSON using `ZoneIterator` to traverse the hierarchy.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 271-289](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L271-L289)[tt_npe/cpp/include/npeWorkload.hpp 112-114](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L112-L114)[tt_npe/cpp/src/npeStats.cpp 596-658](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L596-L658)
