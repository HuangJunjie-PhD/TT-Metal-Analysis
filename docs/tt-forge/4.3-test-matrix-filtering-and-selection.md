## Purpose and Scope

This document describes the test matrix filtering and selection system, specifically the `filter-test-matrix.py` script that processes test configurations from `models-matrix.json` (and similar files) and produces filtered test matrices for execution in GitHub Actions workflows. This system provides intelligent test selection, runner mapping, and matrix expansion capabilities for both benchmarking and demo test suites.

For information about the benchmark configuration format, see [Test Matrix Configuration](https://deepwiki.com/tenstorrent/tt-forge/3.2-test-matrix-configuration). For information about how filtered matrices are executed, see [Demo Tests Workflow](https://deepwiki.com/tenstorrent/tt-forge/4.2-demo-tests-workflow).

## Overview

The test matrix filtering system serves as an adapter between declarative test configurations and GitHub Actions matrix strategies. It transforms a hierarchical JSON structure containing project-level defaults and individual test specifications into a flat list of test configurations, applies filtering rules, and maps runner names based on deployment context.

## Input Matrix Structure

The filter script expects a JSON array where each element represents a project with test configurations. The structure supports two-level inheritance: project-level defaults that apply to all tests within a project, and test-specific overrides.

| Field | Level | Type | Description |
| --- | --- | --- | --- |
| `project` | Project | String | Project identifier (e.g., `tt-forge-onnx`, `tt-xla`, `tt-torch`) |
| `test-defaults` | Project | Object | Default values inherited by all tests in the project |
| `tests` | Project | Array | List of test configurations |
| `name` | Test | String | Unique test identifier |
| `path` | Test | String | Path to the demo/test script relative to the project directory |
| `runs-on` | Test/Default | String or Array | Runner name(s) where test executes (e.g., `n150`, `p150`) |
| `pyreq` | Test | String | Inline pip requirements for the specific test |

**Example Structure:**[.github/workflows/models-matrix.json 1-15](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/models-matrix.json#L1-L15)

`[  {    "project": "tt-forge-onnx",    "test-defaults": {      "runs-on": ["n150", "p150"]    },    "tests": [      { "name": "resnet50_onnx", "path": "onnx/cnn/resnet_demo.py" }    ]  }]`
**Sources:**[.github/workflows/models-matrix.json 1-45](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/models-matrix.json#L1-L45)

## Filter Script Components

### flatten_matrix Function

The `flatten_matrix()` function performs test expansion and inheritance resolution. It processes each project's tests and merges project-level defaults with test-specific configurations using Python's dictionary merge semantics `{**test_defaults, **test}`.

**Key Behavior:** If `runs-on` is a list (e.g., `["n150", "p150"]`), the function creates separate matrix entries for each runner, enabling parallel execution across different hardware architectures.

**Sources:**[.github/workflows/filter-test-matrix.py 10-24](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L10-L24)

### filter_matrix Function

The `filter_matrix()` function applies filtering based on project and optional test name criteria.

**Special Case - "tt-forge" Project Filter:** When `project_filter` is set to `"tt-forge"`, the function implements an exclusion logic where it includes only items where project is `tt-xla`. For any other `project_filter` value, it performs an exact match.

**Name Filtering:** Supports comma-separated filters. A test is included if any of the filter strings (case-insensitive) match the test `name`.

### update_runners Function

The `update_runners()` function transforms generic runner names into specific infrastructure pool names based on the `--sh-runner` flag.

| sh_runner Flag | Input Runner | Output Runner | Use Case |
| --- | --- | --- | --- |
| `True` | `p150` | `p150b` | Shared runner pool |
| `False` | `n150` | `n150-perf` | Performance-specific runner pool |

The original runner name is preserved in the `runs-on-original` field.

**Sources:**[.github/workflows/filter-test-matrix.py 51-60](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L51-L60)

## Command-Line Interface

The script provides a CLI for invoking the filtering pipeline, typically called from a GitHub Actions `run` step.

`python .github/workflows/filter-test-matrix.py \  .github/workflows/models-matrix.json \  <project_filter> \  [--test-filter "<name>"] \  [--sh-runner]`
### Output Format

The script prints a JSON object to stdout containing the expanded matrix and a skip flag: [.github/workflows/filter-test-matrix.py 85-86](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/filter-test-matrix.py#L85-L86)

## Integration with Workflows

### Demo Tests Workflow

The `demo-tests.yml` workflow uses the script to generate a dynamic matrix of demo scripts to run on hardware.

1.   **Job `set-matrix`**: Calls `filter-test-matrix.py` with `models-matrix.json`.
2.   **Job `demo-test`**: Uses `strategy.matrix.include` to ingest the generated list.
3.   **Execution**: Maps the `runs-on` value to the environment-specific label: `tt-ubuntu-2204-{0}-stable`.

**Sources:**[.github/workflows/demo-tests.yml 69-94](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L69-L94)

### Basic Tests Workflow

The `basic-tests.yml` workflow uses an inline bash/jq approach rather than the python script to generate a simpler matrix for smoke tests. It loops through `frontends` and `runs_on` inputs to build a JSON array of objects containing `frontend` and `runs_on` keys.

## Error Handling

The `filter-test-matrix.py` script includes a `try-except` block to handle processing errors. If no matching tests are found after filtering, it sets `matrix_skip: "true"` and prints an error message to `sys.stderr`. If a hard exception occurs, it exits with status code 1.
