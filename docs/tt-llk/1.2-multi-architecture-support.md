## Purpose and Scope

This document describes how the tt-llk repository supports multiple Tenstorrent hardware architectures through a combination of common interfaces and architecture-specific implementations. The system enables the same high-level kernel operations to run on Wormhole B0, Blackhole, and Quasar hardware with minimal code changes, while allowing each architecture to leverage its unique hardware capabilities.

For information about the three-stage pipeline architecture that is common across all architectures, see [Three-Stage Pipeline Architecture](/tenstorrent/tt-llk/1.1-three-stage-pipeline-architecture). For repository organization and directory structure, see [Repository Structure and File Organization](/tenstorrent/tt-llk/1.3-repository-structure-and-file-organization).

## Supported Architectures

The tt-llk library supports three Tenstorrent hardware generations, each with distinct capabilities and maturity levels:

| Architecture | Status | Key Characteristics | Directory |
| --- | --- | --- | --- |
| **Wormhole B0** | Mature | Reference implementation, standard TRISC pipeline | `tt_llk_wormhole_b0/` |
| **Blackhole** | Production | Enhanced features (LLTT, CSR, 64-row unpack, gathering control) | `tt_llk_blackhole/` |
| **Quasar** | Early stage | Minimal header set, architecture features TBD | `tt_llk_quasar/` |

Each architecture maintains its own implementation of the core kernel API while adhering to common function signatures defined in the `ckernel` namespace.

**Sources:** Diagram 5 from high-level architecture analysis

## Architecture Directory Structure

Each architecture directory contains a parallel structure with architecture-specific implementations of:

* **common/inc/**: Core abstractions (`ckernel.h`, packer/unpacker configuration)
* **llk\_lib/**: Low-level kernel operation implementations
* **sfpu/**: Special function processing unit operations

**Sources:** [tt\_llk\_wormhole\_b0/common/inc/ckernel.h1-472](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L1-L472) [tt\_llk\_blackhole/common/inc/ckernel.h1-687](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L1-L687)

## Architecture-Specific Implementations

### Wormhole B0: Reference Implementation

Wormhole B0 serves as the baseline architecture with standard implementations of all kernel operations.

#### Packer Architecture: Four-Packer Design

Wormhole B0 implements a four-packer architecture where each packer handles one face of a tile:

Key characteristics:

* **NUM\_PACKERS = 4** ([tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h23](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L23-L23))
* **PACK\_SEL() macro** generates selection bitmask for active packers ([tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h25-28](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L25-L28))
* **Face-based L1 addressing**: Each packer writes to a different L1 offset ([tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h354-380](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L354-L380))
* **Configuration per packer**: Four configuration register sets (SEC0\_REG1, SEC0\_REG8, SEC1\_REG1, SEC1\_REG8) ([tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h283-294](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L283-L294))

**Sources:** [tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h23-28](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L23-L28) [tt\_llk\_wormhole\_b0/common/inc/cpack\_common.h354-380](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cpack_common.h#L354-L380)

#### Tilize Operations

Wormhole's tilize implementation supports narrow tiles and multiple loop iterations:

**Key characteristics:**

* **Narrow tile support**: Configurable face dimensions with `narrow_tile` parameter ([tt\_llk\_wormhole\_b0/llk\_lib/llk\_unpack\_tilize.h21-44](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L21-L44))
* **Multi-loop pattern**: Iterates over face pairs, 2 outer loops for narrow tiles ([tt\_llk\_wormhole\_b0/llk\_lib/llk\_unpack\_tilize.h204](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L204-L204))
* **Context switching**: Uses dual contexts with `wait_for_next_context(2)` ([tt\_llk\_wormhole\_b0/llk\_lib/llk\_unpack\_tilize.h95](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L95-L95))
* **32-bit optimization**: Direct unpack-to-dest for Float32/Int32/UInt32 ([tt\_llk\_wormhole\_b0/llk\_lib/llk\_unpack\_tilize.h57-58](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L57-L58))

### Blackhole: Enhanced Architecture

Blackhole extends the base architecture with hardware enhancements that require software support.

#### Packer Architecture: Single-Packer Design

Blackhole consolidates packing functionality into a single hardware unit that processes all faces sequentially:

Key characteristics:

* **NUM\_PACKERS = 1** ([tt\_llk\_blackhole/common/inc/cpack\_common.h24](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L24-L24))
* **No PACK\_SEL macro**: Single packer handles all faces
* **Stride-based addressing**: Z-counter increments automatically ([tt\_llk\_blackhole/common/inc/cpack\_common.h177](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L177-L177))
* **Template parameters**: `set_packer_strides()` adapts stride for different modes ([tt\_llk\_blackhole/common/inc/cpack\_common.h167](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L167-L167))
* **Simplified configuration**: One configuration register set instead of four

**Sources:** [tt\_llk\_blackhole/common/inc/cpack\_common.h24](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L24-L24) [tt\_llk\_blackhole/common/inc/cpack\_common.h167-199](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L167-L199)

## Runtime Architecture Detection

The test infrastructure detects the target architecture at runtime through multiple strategies:

**Detection methods:**

1. **Environment variable**: `CHIP_ARCH` explicitly sets the target
2. **Hardware detection**: `tt-smi` queries installed devices
3. **CI labels**: GitHub Actions runner labels determine architecture
4. **Header download**: Sparse git clone fetches architecture-specific headers from tt-metal repository

The `setup_arch()` function in test configuration orchestrates this detection and configures compilation paths accordingly.
