This page documents the host-side object model and API of `tt-emule`. It covers the core entities used to represent Tenstorrent hardware in software, the memory layout of emulated cores, and the mechanisms for program execution and buffer management.

## System Architecture

`tt-emule` integrates into the `tt-metal` ecosystem by providing an emulation-specific implementation of the Device and Command Queue layers. When built with `-DTT_METAL_USE_EMULE=ON`, the emulator substitutes the hardware-bound Universal Memory Device (UMD) with `SWEmuleChip`, a memory-backed software implementation [docs/metal-integration.md 24-28](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/metal-integration.md?plain=1#L24-L28)

### Integration Layer

The integration operates across four layers to ensure that `tt-metal` host code and `ttnn` operations run unchanged:

1.   **UMD Device Injection**: `SWEmuleChip` provides memory-backed I/O where all non-memory operations (like resets or power management) are no-ops [docs/metal-integration.md 24-28](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/metal-integration.md?plain=1#L24-L28)
2.   **Runtime Activation**: Controlled via the `TT_METAL_EMULE_MODE=1` environment variable, which selects the emulation target device [docs/metal-integration.md 49-58](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/metal-integration.md?plain=1#L49-L58)
3.   **JIT Kernel Execution**: Kernels are compiled into shared objects (`.so`) and executed as OS threads [docs/metal-integration.md 65-78](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/metal-integration.md?plain=1#L65-L78)
4.   **Dispatch Interception**: `LaunchProgram()` in `tt_metal.cpp` is branched to call `emule::execute_program_emulated` when the target device is an emulator [docs/metal-integration.md 118-124](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/metal-integration.md?plain=1#L118-L124)

## Core Components

### Device (`SWEmuleChip` / `tt_emule::Device`)

The `Device` represents the entire SoC. It manages the global DRAM memory and a collection of `Core` objects.

*   **DRAM Emulation**: DRAM is typically backed by a large `std::vector<uint8_t>` or `mmap` region [src/host_api.cpp 22-25](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/host_api.cpp#L22-L25)
*   **Allocator**: It includes a `MockAllocator` to handle buffer placements in L1 and DRAM [include/tt_emule/device.hpp 90](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L90-L90)
*   **Sub-devices**: Supports sub-device IDs and stall groups for compatibility with `tt-metal`[src/host_api.cpp 29-30](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/host_api.cpp#L29-L30)

### Core (`tt_emule::Core`)

The `Core` represents a single Tensix or Neo core. Each core has its own L1 SRAM and synchronization state.

*   **L1 Memory**: Backed by `mmap` with `MAP_32BIT` for worker cores, ensuring that pointers fit into 32-bit registers as they would on hardware [include/tt_emule/device.hpp 58-61](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L58-L61)[include/tt_emule/device.hpp 201](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L201-L201)
*   **Roles**: Cores are assigned roles, such as `WORKER` or `DRAM`[include/tt_emule/device.hpp 38](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L38-L38)
*   **Synchronization**: Holds the `CBSyncState` array for Wormhole/Blackhole circular buffers and `DFBSyncState` for Quasar dataflow buffers [include/tt_emule/device.hpp 138-170](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L138-L170)

### Program (`tt_emule::Program`)

A `Program` is a container for kernels and their configurations.

*   **KernelDescriptor**: Defines the `KernelType` (Compute, DataMovement, or Quasar variants), source path, and target processor ID [include/tt_emule/program.hpp 144](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/program.hpp#L144-L144)
*   **Buffer Configs**: Contains lists of `CircularBufferConfig` and `DataflowBufferConfig` that define how cores communicate [include/tt_emule/program.hpp 144-145](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/program.hpp#L144-L145)

### Data Flow: Program Execution

When `EnqueueProgram` is called, the emulator performs the following steps:

1.   **Thread Spawning**: Launches a thread for every kernel role on every active core (e.g., 3 threads for a Tensix core: Reader, Writer, Compute) [README.md 21-23](https://github.com/tenstorrent/tt-emule/blob/a57cf072/README.md?plain=1#L21-L23)
2.   **State Injection**: Before the kernel entry point is called, the runner injects thread-local pointers (`thread_local`) for `__core`, `__device`, and `__rt_args`[src/kernel_runner.cpp 16-20](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/kernel_runner.cpp#L16-L20)
3.   **DFB/CB Setup**: For Quasar, it builds `EmuleDFBInterface` objects to manage STRIDED or BLOCKED access patterns [src/kernel_runner.cpp 36-136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/kernel_runner.cpp#L36-L136)
4.   **Execution**: The dlopen'd kernel function is invoked.

## Memory Model and Addressing

`tt-emule` uses a "host-pointer-everywhere" convention. Because L1 is mmap'd below the 4GB boundary (`MAP_32BIT`), the absolute host address can be cast to a `uint32_t` and used directly within the JIT-compiled kernels as a local L1 pointer [include/tt_emule/device.hpp 197-201](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L197-L201)

*   **L1 Access**: `Core::l1_ptr(offset)` returns the host address for a given L1 offset. It includes bounds checking to prevent silent corruption [include/tt_emule/device.hpp 97-108](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L97-L108)
*   **DRAM Access**: DRAM is accessed via `__emule_dram_ptr(offset)`, which resolves to the device's DRAM backing store [src/kernel_runner.cpp 26-28](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/kernel_runner.cpp#L26-L28)
*   **Bump Allocation**: A simple bump allocator (`l1_alloc`) is used for dynamic L1 requirements, such as Quasar DFB backing stores [include/tt_emule/device.hpp 123-129](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/device.hpp#L123-L129)
