This document covers the `AddDataMovePass`, which handles the insertion of data movement and tensor alignment operations in the TTNN compilation pipeline. This pass ensures tensors are properly converted between PyTorch and TTNN formats, placed on appropriate devices, and configured with correct layouts and data types.

For information about the initial operation lowering from PyTorch to TTNN, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)). For details about the overall compilation pipeline, see [Backend Architecture and Pass Pipeline](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.1-backend-entry-points-and-pass-pipeline).

## Purpose and Data Movement Requirements

The `AddDataMovePass` addresses the fundamental requirement that TTNN operations expect tensors in specific formats that differ from standard PyTorch tensors. This pass automatically inserts the necessary conversion operations to bridge between PyTorch and TTNN tensor representations while optimizing for performance and correctness.

**Title: Core Data Movement and Alignment Operations**

**Data Movement Triggers:**

*   **PyTorch → TTNN**: Detected by `is_torch_to_ttnn(src_node, dst_node)`[torch_ttnn/passes/lowering/add_data_move_pass.py 261-264](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L261-L264) triggers when a non-TTNN node feeds into a TTNN compute operation
*   **TTNN → PyTorch**: Detected by `is_ttnn_to_torch(src_node, dst_node)`[torch_ttnn/passes/lowering/add_data_move_pass.py 267-268](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L267-L268) triggers when a TTNN node feeds into a non-TTNN operation
*   **TTNN → TTNN**: Detected by `is_ttnn_to_ttnn(src_node, dst_node)`[torch_ttnn/passes/lowering/add_data_move_pass.py 271-274](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L271-L274) triggers for layout or device alignment between TTNN operations

## Special Case Handling and Operation-Specific Alignment

Many TTNN operations have specific requirements that override the default alignment rules. The pass implements extensive special case handling through the `_align_special_cases` method.

**Key Special Cases:**

| Operation | Input | Requirement | Reason |
| --- | --- | --- | --- |
| `ttnn.embedding` | First input | `dtype=TtnnUint32`, `layout=TtnnRowMajorLayout` | Embedding indices must be uint32 and row major |
| `ttnn.slice` | First input | `layout=TtnnRowMajorLayout` | Can only unpad tilized tensors with full tiles |
| `ttnn.split` | First input | `layout=TtnnRowMajorLayout` | Runtime error for tilized inputs |
| `ttnn.argmax` | First input | `dtype=TtnnBfloat16`, `layout=TtnnRowMajorLayout` | Documentation requirement |
| `target_wrappers.conv` | Weight input | `layout=TtnnRowMajorLayout`, `device="host"` | Weight must be on host for preprocessing |

**Additional Special Cases:**

*   `target_wrappers.stack` with `int32`/`int64` output → coerce inputs to `TtnnUint32`[torch_ttnn/passes/lowering/add_data_move_pass.py 365-372](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L365-L372)
*   `ttnn.subtract` from PyTorch `int32`/`int64` → use `TtnnBfloat16` to handle negative values [torch_ttnn/passes/lowering/add_data_move_pass.py 377-384](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L377-L384)
*   `target_wrappers.roll` input → `TtnnRowMajorLayout` (calls `ttnn.slice` internally) [torch_ttnn/passes/lowering/add_data_move_pass.py 352-356](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L352-L356)

The method `_reset_to_default_layout()` also handles post-operation conversions, such as converting back to tile layout after operations that output row major tensors [torch_ttnn/passes/lowering/add_data_move_pass.py 312-342](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L312-L342)

## Multi-Device Support and Mesh Operations

The pass provides comprehensive multi-device support for both sharding and replication scenarios, handling the complex tensor distribution patterns required for efficient multi-device execution. The mesh mappers and composers are cached per dimension to avoid redundant graph nodes.

**Title: Multi-Device Tensor Distribution Flow**

**Mesh Mapper and Composer Caching:**

*   `self.shard_to_mesh` - Dictionary mapping shard dimension to `ttnn.ShardTensorToMesh` node [torch_ttnn/passes/lowering/add_data_move_pass.py 285](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L285-L285)
*   `self.replicate_to_mesh` - Single `ttnn.ReplicateTensorToMesh` node for all replications [torch_ttnn/passes/lowering/add_data_move_pass.py 286](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L286-L286)

## Integration with Compilation Pipeline

The `AddDataMovePass` executes at a specific point in the compilation pipeline, after operation lowering but before optimization passes.

**Pipeline Integration Points:**

*   **Input**: FX graph with TTNN operations but inconsistent tensor formats
*   **Processing**: Inserts `ttnn.from_torch`, `ttnn.to_torch`, `ttnn.to_device`, `ttnn.from_device`, and `ttnn.to_layout` operations
*   **Output**: FX graph with all necessary data movement operations for correct execution
*   **Caching**: Uses `aligned_node_dict` to prevent duplicate conversion operations
