## Purpose and Scope

This page documents `PETransferState`, the core data structure that tracks the execution state of individual NoC transfers during simulation. `PETransferState` manages the complete lifecycle of a transfer from initialization through completion, including start/end cycle computation, bandwidth tracking, byte transfer accounting, and dependency resolution.

For information about the overall simulation algorithm that manages these transfer states, see [Performance Estimation Algorithm](https://deepwiki.com/tenstorrent/tt-npe/4.1-performance-estimation-algorithm). For details on how dependencies between transfers are created and tracked, see [Dependency Tracking System](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system). For how bandwidth is computed based on congestion, see [Congestion Modeling](https://deepwiki.com/tenstorrent/tt-npe/4.4-congestion-modeling).

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 1-362](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L1-L362)

## PETransferState Structure

`PETransferState` maintains both static parameters (from the workload definition) and dynamic runtime state that evolves during simulation:

| Field Category | Field Name | Type | Description |
| --- | --- | --- | --- |
| **Static Parameters** | `params` | `npeWorkloadTransfer` | Immutable transfer definition from workload (src, dst, bytes, noc_type, etc.) |
|  | `route` | `std::vector<nocLinkID>` | Computed NoC path from source to destination |
| **Timing State** | `start_cycle` | `CycleCount` | Actual cycle when transfer begins (may differ from phase_cycle_offset) |
|  | `end_cycle` | `CycleCount` | Cycle when transfer completes |
| **Bandwidth State** | `curr_bandwidth` | `double` | Current effective bandwidth in bytes/cycle (updated each timestep) |
| **Transfer Progress** | `total_bytes_transferred` | `size_t` | Running total of bytes transferred so far |
| **Dependency State** | `depends_on` | `npeCheckpointID` | Checkpoint this transfer waits for before starting |
|  | `required_by` | `std::vector<npeCheckpointID>` | Checkpoints that wait for this transfer to complete |

The `params` field contains the original transfer specification from the workload, including `phase_cycle_offset` (the earliest cycle the transfer could start), while `start_cycle` tracks when it actually starts after dependency resolution.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 25-46](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L25-L46)[tt_npe/cpp/src/npeEngine.cpp 277-319](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L277-L319)

## Initialization Phase

Transfer state initialization occurs in `npeEngine::initTransferState()`, which creates a flat vector of all transfers from the hierarchical workload structure:

The initialization process:

1.   **Size Calculation** - Count total transfers across all phases [27-30](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/27-30)
2.   **Vector Allocation** - Pre-allocate `transfer_state` vector [32-33](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/32-33)
3.   **Per-Transfer Initialization** - For each transfer [35-44](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/35-44): 
    *   Extract transfer ID from workload transfer
    *   Compute NoC route using device model's `route()` function
    *   Create `PETransferState` with: 
        *   Original transfer parameters (`wl_transfer`)
        *   Initial start cycle set to `phase_cycle_offset`[41](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/41)
        *   Computed route through the NoC [42](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/42)

**Key Implementation Detail:** All phases are assumed to start at cycle 0 in the current implementation [40](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/40) The `phase_cycle_offset` field in each transfer determines when it becomes eligible to start.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 25-47](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L25-L47)

## Transfer Queue and Activation

After initialization, transfers are organized into a sorted queue for efficient dispatch during simulation:

**Queue Creation** ([49-65](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/49-65)):

*   Create `TransferQueuePair` for each transfer containing `{start_cycle, id}`
*   Sort in descending order so simulation can pop from the end efficiently
*   Uses stable sort to preserve original ordering for ties

**Activation Logic** ([236-253](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/236-253)):

1.   **Eligibility Check** - Transfer's `start_cycle <= curr_cycle`
2.   **Dependency Check** - Call `dep_tracker.done(transfer.depends_on, curr_cycle)`
3.   **Start Cycle Adjustment** - If dependency exists and completes later than `phase_cycle_offset`, adjust: 
```
start_cycle = max(start_cycle, dep_tracker.end_cycle_plus_delay(depends_on))
```
4.   **Queue Management** - Move activated transfer to end of queue and resize to discard it

This two-phase check ensures transfers respect both temporal ordering (cycle-based) and causal ordering (dependency-based).

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 49-65](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L49-L65)[tt_npe/cpp/src/npeEngine.cpp 236-256](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L236-L256)

## Bandwidth Computation

Once a transfer is active, its bandwidth is recomputed each timestep by the device model:

The bandwidth computation is delegated to the device model through the polymorphic `computeCurrentTransferRate()` interface ([258-264](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/258-264)):

```
model->computeCurrentTransferRate(
    start_of_timestep,
    curr_cycle,
    transfer_state,
    live_transfer_ids,
    *device_state,
    enable_congestion_model);
```

**Key Aspects:**

*   **Timestep Granularity** - Bandwidth is recomputed once per timestep, not every cycle
*   **Congestion Impact** - Bandwidth reflects current network congestion if enabled
*   **Per-Transfer** - Each active transfer gets an independent bandwidth value in its `curr_bandwidth` field
*   **Device-Specific** - The device model determines how congestion affects bandwidth (see [Congestion Modeling](https://deepwiki.com/tenstorrent/tt-npe/4.4-congestion-modeling))

The computed `curr_bandwidth` is used in the next phase to calculate bytes transferred.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 258-264](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L258-L264)

## Byte Transfer Accounting

After bandwidth is computed, the simulation updates how many bytes each transfer has moved:

**Byte Calculation Steps** ([277-319](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/277-319)):

1.   **Compute Remaining Bytes**[281](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/281)

```
remaining_bytes = params.total_bytes - total_bytes_transferred
```
2.   **Calculate Active Cycles**[282-292](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/282-292)

    *   Base case: `min(cycles_per_timestep, curr_cycle - start_cycle)`
    *   Special case: If dependency completed in previous timestep, adjust for mid-timestep start
    *   This handles transfers that start partway through a timestep due to dependency delays

3.   **Apply Bandwidth Limit**[293](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/293)

```
max_transferrable_bytes = cycles_active * curr_bandwidth
```
4.   **Enforce Total Bytes Constraint**[297](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/297)

```
bytes_transferred = min(remaining_bytes, max_transferrable_bytes)
```
5.   **Update Running Total**[298](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/298)

```
total_bytes_transferred += bytes_transferred
```

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 277-298](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L277-L298)

## Completion and End Cycle Calculation

When a transfer completes (`total_bytes_transferred == params.total_bytes`), the simulation computes its exact end cycle and updates dependencies:

**End Cycle Computation** ([301-309](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/301-309)):

The algorithm accounts for transfers that both start and end within the same timestep:

1.   **Calculate transfer duration**[302](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/302)

```
cycles_transferring = ceil(bytes_transferred / curr_bandwidth)
```
2.   **Find actual start within timestep**[305-306](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/305-306)

```
start_cycle_of_transfer_within_timestep = max(start_cycle, start_of_timestep)
```

This handles cases where the transfer started in a previous timestep

3.   **Compute end cycle**[307-309](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/307-309)

```
transfer_end_cycle = start_cycle_of_transfer_within_timestep + cycles_transferring
lt.end_cycle = transfer_end_cycle
```

**Dependency and Statistics Updates** ([311-318](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/311-318)):

After computing `end_cycle`:

*   Update all checkpoints in `required_by` list so dependent transfers can proceed
*   Update worst-case transfer end cycle for the device and mesh device in statistics
*   This tracks the latest completing transfer to determine total simulation duration

**Cleanup** ([322-328](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/322-328)):

Completed transfers are removed from `live_transfer_ids` using `std::remove_if` with a predicate that checks if `total_bytes_transferred == params.total_bytes`.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 300-328](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L300-L328)

## Interaction with Simulation Components

The following diagram shows how `PETransferState` interacts with other simulation components:

**Component Responsibilities:**

*   **npeEngine** - Orchestrates lifecycle, manages state transitions
*   **PETransferState** - Maintains per-transfer state and progress
*   **Device Model** - Computes routes and bandwidth based on hardware characteristics
*   **Dependency Tracker** - Coordinates transfer ordering through checkpoints
*   **npeStats** - Collects metrics at each lifecycle stage for analysis

The transfer state serves as the central data structure that all components read from and write to during simulation.

## Timeline Example

The following diagram illustrates how a transfer's state evolves over multiple timesteps:

**Key Events:**

*   **Cycle 0-100**: Transfer created during initialization, exists in `transfer_state`
*   **Cycle 100-300**: Transfer in `transfer_queue`, waiting for start cycle and dependencies
*   **Cycle 320**: Dependency satisfied, transfer activated mid-timestep, `start_cycle` adjusted to 320
*   **Cycle 320-640**: Transfer active across 4 timesteps, bandwidth updated each timestep, bytes accumulate
*   **Cycle 640**: Transfer completes mid-timestep 6, `end_cycle` computed as 640, dependent checkpoints updated
*   **After cycle 640**: Transfer removed from `live_transfer_ids`, no longer processed

This example shows how transfers can start and end at arbitrary cycle boundaries, not just timestep boundaries, enabling accurate cycle-level timing resolution.
