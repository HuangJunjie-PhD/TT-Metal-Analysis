## Purpose and Scope

This document explains how the pytorch2.0_ttnn project automatically generates operator-level tests from metrics collected during model execution. The system extracts input variation patterns observed across 40+ model tests and creates focused tests that validate individual ATen-to-TTNN operation conversions across all observed input configurations.

This automated approach provides comprehensive operator coverage without requiring manual test authoring for every operation and input combination. For information about the model tests that produce the source metrics, see [Model Test Coverage](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5-model-test-coverage). For details on how metrics are collected and analyzed, see [Metrics Collection and Schema Tracking](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.1-metrics-collection-and-schema-tracking).

## Input Data: Schema Metrics from Model Tests

The test generator consumes schema metrics captured during model test execution. These metrics record every operation encountered during compilation, along with its input characteristics.

### Schema List Structure

During model execution, the `compile_and_run` fixture serializes schema information to pickle files:

| File | Content |
| --- | --- |
| `original-schema_list.pickle` | Operations from the original PyTorch graph before compilation |
| `compiled-schema_list.pickle` | Operations from the compiled TTNN graph after transformation |

### Data Classes for Schema Capture

The metrics system uses structured data classes defined in `torch_ttnn/metrics.py`:

**InputVariation Class** (`torch_ttnn/metrics.py:58-105`)

*   Captures operation name and input characteristics
*   Stores list of `Inputs` objects containing dtype, name, shape, and value
*   Provides `dict()` method for serialization with schema information
*   Provides `dict_for_pickle()` method that extracts tensor metadata including shape, dtype, layout, and buffer type

**ConvertedInput Class** (`torch_ttnn/metrics.py:107-117`)

*   Tracks operations after conversion to TTNN
*   Links converted operation back to original `InputVariation`
*   Enables tracing of which ATen operations successfully converted

**Collection Functions**

*   `collect_input_variation()`[torch_ttnn/metrics.py 119-167](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L119-L167) - Extracts schema from target function using `_schema` attribute
*   `collect_input_variation_from_node()`[torch_ttnn/metrics.py 170-190](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L170-L190) - Collects from FX graph nodes, detecting if operation was converted
*   `collect_input_variations_from_list_nodes()`[torch_ttnn/metrics.py 193-205](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L193-L205) - Aggregates variations from node list

Each schema entry captures:

*   **Operation target**: The ATen operation (e.g., `torch.ops.aten.add.Tensor`)
*   **Input schemas**: Mapped from `target._schema.arguments` to actual args/kwargs
*   **Tensor metadata**: Extracted from `node.meta["val"]` for `FakeTensor` objects
*   **Input values**: Concrete values for non-tensor arguments (scalars, booleans, etc.)

**Sources:**[torch_ttnn/metrics.py 1-206](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L1-L206)[.github/workflows/run-tests.yaml 147-152](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L147-L152)

## Test Generation Pipeline

### Job: push-autogen-op-tests

The test generation occurs in the `push-autogen-op-tests` job after all model tests complete. This job is defined at [.github/workflows/run-tests.yaml 206-265](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L206-L265)

**Job Dependencies and Execution Flow**

The job performs two generation passes:

1.   **Per-Model Tests**: First invocation [.github/workflows/run-tests.yaml 239](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L239-L239) generates tests organized by model name in `tests/autogen_op/model_name/`
2.   **Merged Tests**: Second invocation with `--merge=True`[.github/workflows/run-tests.yaml 240](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L240-L240) creates consolidated tests in `tests/autogen_op/ALL/` that combine variations across all models

The `commit-glob` output is determined by the `commit_report` input [.github/workflows/run-tests.yaml 241-248](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L241-L248):

*   `"All"` commits all generated files
*   `"Tests"` commits only `tests/*` directory
*   Any other value skips PR creation

## Conditional Execution

Test generation only occurs when `commit_report` is not `'None'`:

| Input Value | Behavior |
| --- | --- |
| `'None'` | Skip test generation entirely |
| `'Tests'` | Generate and commit tests to `tests/*` |
| `'Docs'` | Generate but only commit documentation |
| `'All'` | Generate and commit all files |

This allows the workflow to run model tests without generating autogen tests in contexts where they're not needed (e.g., before-merge validation).

### Integration with Metrics Collection and Documentation

The generated tests serve multiple purposes in the overall system:

**Metrics Flow to Documentation**

**Integration Points**

1.   **Validation**: Generated tests verify that operations work correctly across observed input patterns
2.   **Status Tracking**: Test results populate the `InputVarPerOp` data structure [tools/collect_metrics.py 161-449](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tools/collect_metrics.py#L161-L449) with conversion status: 
    *   `ConversionStatus.DONE` - Operation successfully converted to TTNN
    *   `ConversionStatus.FALLBACK` - Operation partially converted (still uses some ATen ops)
    *   `ConversionStatus.NONE` - Operation not converted
    *   `ConversionStatus.REMOVED` - Operation optimized away

3.   **Blocklist Generation**: Failures trigger guard function generation via `generate_guard_function_from_input_var_metric.py`
4.   **Documentation**: The `collect_metrics.py` script aggregates test results and generates operation reports showing which input variations are supported

## Summary

The automated test generation system creates a feedback loop:

1.   **Model tests** execute diverse workloads, capturing operation patterns
2.   **Generator** synthesizes focused operator tests from observed patterns
3.   **Generated tests** validate operations across all input variations
4.   **Metrics** from generated tests inform guard blocklist updates
5.   **Updated guards** improve compilation success rates

This automation reduces manual test maintenance burden while ensuring comprehensive coverage of real-world operation usage patterns across the entire model test corpus.
