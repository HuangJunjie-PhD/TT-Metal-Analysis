## Purpose and Scope

This page demonstrates how to process large tensors that exceed L1 memory capacity by streaming data through small Dataflow Buffers (DFBs). The streaming pattern allows kernels to operate on multi-gigabyte DRAM tensors using only kilobytes of L1 buffer space by processing the tensor in chunks.

For basic tensor slicing and multi-tile access patterns, see [Multi-tile Processing](https://deepwiki.com/tenstorrent/tt-lang/9.2-multi-tile-processing). For general data movement strategies, see [Data Movement Patterns](https://deepwiki.com/tenstorrent/tt-lang/4.2-data-movement-patterns). For information on the memory hierarchy, see [Memory Hierarchy: DRAM, L1, and DST Registers](https://deepwiki.com/tenstorrent/tt-lang/2.3.1-memory-hierarchy:-dram-l1-and-dst-registers).

## Problem: Memory Hierarchy Constraints

Tenstorrent hardware has a three-tier memory hierarchy:

*   **DRAM**: Large capacity (gigabytes), high latency.
*   **L1 SRAM**: Small capacity (~1MB per core), low latency, used for circular buffers.
*   **DST registers**: Very small capacity (16 tiles = 32KB), zero latency.

A naive approach might attempt to allocate DFBs sized to hold the entire input tensor, but this fails for large tensors because **DFB size is allocated in L1**, not DRAM. L1 capacity on a single core is typically around 1MB, making it impossible to buffer entire large tensors (e.g., a 200MB tensor).

Sources: [test/python/test_large_dram_streaming.py 10-13](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L10-L13)[test/python/test_large_dram_streaming.py 112-115](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L112-L115)

## Solution: Streaming Architecture

The streaming pattern decouples DFB size from tensor size by processing the tensor in small blocks. The key insight is that **DFB size depends only on the `shape` parameter** provided to `ttl.make_dataflow_buffer_like`, not the total tensor volume.

## DFB Sizing and Memory Calculation

### Buffer Size Formula

Each DFB allocates L1 memory based on the `shape` and `block_count` parameters:

```
L1_bytes = tile_rows * tile_cols * tile_size * block_count
         = shape[0] * shape[1] * 2KB * block_count
```

Where:

*   `shape` = DFB tile dimensions (from `ttl.make_dataflow_buffer_like`).
*   `tile_size` = 32x32xbf16 = 2KB per tile.
*   `block_count` = number of pages for buffering (typically 2 for double-buffering).

### Example Calculation

For the streaming kernel in `test_large_dram_streaming.py`: [test/python/test_large_dram_streaming.py 32-43](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L32-L43)

`# Tensor: 10240 x 10240 elements = 320 x 320 tiles# DFB: shape=(2, 2), block_count=2 dfb_size = 2 * 2 * 2KB * 2 = 16KB per DFBtotal_l1 = 4 DFBs * 16KB = 64KB # vs. tensor size in DRAMtensor_size = 10240 * 10240 * 2 bytes = 200MB`
This achieves a **3000:1 memory ratio** - streaming 200MB through 64KB of buffers.

| Component | Size Calculation | Memory |
| --- | --- | --- |
| Single tile | 32x32x2 bytes | 2 KB |
| DFB (2x2 tiles) | 4 tiles * 2 KB * block_count(2) | 16 KB |
| Full tensor (320x320 tiles) | 102,400 tiles * 2 KB | 200 MB |
| Total L1 needed | 4 DFBs * 16 KB | 64 KB |

## Implementation Patterns

### Pattern 1: Block-Based Iteration

The core streaming pattern uses nested loops to iterate over blocks of tiles. The number of blocks is determined by the total tensor shape divided by the DFB block shape.

`# Calculate block dimensionsrow_tiles = 2  # DFB shapecol_tiles = 2rows = a.shape[0] // 32 // row_tiles  # Number of row blockscols = a.shape[1] // 32 // col_tiles  # Number of col blocks # Process each blockfor row in range(rows):    for col in range(cols):        with a_dfb.reserve() as a_blk:            tx_a = ttl.copy(                a[row * row_tiles : (row + 1) * row_tiles,                  col * col_tiles : (col + 1) * col_tiles],                a_blk            )            tx_a.wait()`
Sources: [test/python/test_large_dram_streaming.py 26-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L26-L30)[test/python/test_large_dram_streaming.py 59-89](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L59-L89)

### Pattern 2: Slice Notation for Block Access

Tensor slicing uses Python slice syntax `tensor[start:end, start:end]` to extract tile blocks. The slice indices are in **tile coordinates**, not scalar element coordinates.

`# Single tile accesstile = inp[r, c] # Block access (2x2 tiles)block = a[    row * row_tiles : (row + 1) * row_tiles,    col * col_tiles : (col + 1) * col_tiles]`

### Pattern 3: Context Manager for DFB Lifecycle

Using Python `with` statements simplifies DFB management by automatically handling `wait`/`reserve` at entry and `pop`/`push` at exit.

`# [test/python/test_large_dram_streaming.py:49-55]with (    a_dfb.wait() as a_block,    b_dfb.wait() as b_block,    c_dfb.wait() as c_block,    y_dfb.reserve() as y_block,):    y_block.store(a_block * b_block + c_block)# Automatic: y_block.push(), c_block.pop(), b_block.pop(), a_block.pop()`

## Example: Fused Multiply-Add Streaming

This example shows a complete streaming kernel for computing `y = a * b + c` on large DRAM tensors.

### Entity Relationship Diagram

This diagram shows how the kernel threads interact with the DFBs and DRAM tensors.

### Thread Implementations

**Compute Thread**: Processes blocks in order using the `with` pattern.

`# [test/python/test_large_dram_streaming.py:45-55]@ttl.compute()def compute_kernel():    for _ in range(rows):        for _ in range(cols):            with (                a_dfb.wait() as a_block,                b_dfb.wait() as b_block,                c_dfb.wait() as c_block,                y_dfb.reserve() as y_block,            ):                y_block.store(a_block * b_block + c_block)`
**Data Movement Read Thread**: Fetches input blocks from DRAM.

`# [test/python/test_large_dram_streaming.py:57-90]@ttl.datamovement()def read_kernel():    for row in range(rows):        for col in range(cols):            with (a_dfb.reserve() as a_blk, b_dfb.reserve() as b_blk, c_dfb.reserve() as c_blk):                tx_a = ttl.copy(a[row*2:(row+1)*2, col*2:(col+1)*2], a_blk)                tx_b = ttl.copy(b[row*2:(row+1)*2, col*2:(col+1)*2], b_blk)                tx_c = ttl.copy(c[row*2:(row+1)*2, col*2:(col+1)*2], c_blk)                tx_a.wait()                tx_b.wait()                tx_c.wait()`

## Performance Considerations

### Block Count Selection

The `block_count` parameter controls double-buffering depth. `block_count=2` is the standard choice to allow the data movement thread to fetch the next block while the compute thread processes the current one.

| block_count | L1 per DFB (2x2 tiles) | Behavior |
| --- | --- | --- |
| 1 | 8 KB | Single buffering - threads will stall waiting for each other. |
| 2 | 16 KB | Double buffering - typical choice for pipelining. |

Sources: [test/python/test_large_dram_streaming.py 33-43](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L33-L43)[test/python/test_elementwise_shapes.py 163-169](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L163-L169)

### Block Size Tuning

Block size (DFB `shape` parameter) affects the trade-off between loop overhead and L1 memory usage.

*   **Smaller blocks** (e.g., `shape=(1, 1)`): Minimize L1 usage but increase loop overhead and reduce NOC bandwidth efficiency.
*   **Larger blocks** (e.g., `shape=(4, 4)`): Improve NOC efficiency but increase L1 pressure.

Sources: [test/python/test_large_dram_streaming.py 26-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L26-L30)[test/python/test_elementwise_shapes.py 44-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L44-L49)

### Loop Generation and Multi-Tile DMA

When processing multi-tile blocks, the compiler generates nested tile loops to handle the transfer. If multiple copies use the same tile grid, they may be batched to minimize barriers.
