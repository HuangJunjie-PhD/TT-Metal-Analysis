## Purpose and Scope

This page documents the core backend entry points that integrate PyTorch's `torch.compile` with the TTNN compilation system. It covers the `TorchTtnnOption` configuration class, the `ttnn_backend` and `aten_backend` functions that orchestrate compilation, and the `PassManager`-based pipeline that transforms PyTorch ATen operations into TTNN operations.

For details on individual passes, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)), [Data Movement and Alignment (AddDataMovePass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.3-data-movement-and-alignment-(adddatamovepass)), and [Graph Analysis and Optimization Passes](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3-graph-analysis-and-optimization-passes). For guard system integration, see [Guard System and Operation Blocklists](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.4-guard-system-and-operation-blocklists).

## TorchTtnnOption Configuration

The `TorchTtnnOption` class encapsulates all configuration parameters for the compilation backend. Users create an instance of this class and pass it to the backend.

### Configuration Parameters

| Parameter | Type | Default | Purpose |
| --- | --- | --- | --- |
| `device` | `ttnn.Device` | Required | Target TTNN device (single or mesh) |
| `gen_graphviz` | `bool` | `False` | Generate graphviz visualizations after each pass |
| `run_mem_analysis` | `bool` | `False` | Run memory analysis pass to track SRAM usage |
| `run_eviction_opt` | `bool` | `False` | Run eviction optimization to manage memory overflows |
| `verbose` | `bool` | `True` | Enable verbose logging output |
| `metrics_path` | `str` | `""` | Path for saving compilation metrics |
| `bypass_compile` | `bool` | `False` | Skip compilation and return original graph |
| `use_less_ttnn_op_types` | `bool` | `True` | Use fewer TTNN operation types (performance optimization) |
| `export_code` | `str` or `None` | `None` | Path to export standalone Python scripts for debugging |
| `total_num_iterations` | `int` | `1` | Total number of iterations for memory analysis |
| `data_parallel` | `bool` | `False` | Enable data parallel execution across multiple devices |
| `load_params_once` | `bool` | `True` | Preprocess model parameters once before execution |
| `native_integration` | `bool` | `False` | Enable native PyTorch-TTNN tensor integration |

**Sources:**[torch_ttnn/backend.py 22-77](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L22-L77)

## aten_backend: The Main Compilation Function

The `aten_backend` function is the core compilation entry point that receives a traced ATen graph from `aot_autograd` and transforms it through a series of passes.

### Compilation Stages

### Preprocessing and Graph Capture

Before pass execution, `aten_backend` performs several preprocessing steps:

1.   **Input Aliasing Cleanup** - Removes clone operations inserted to handle PyTorch input aliasing constraints: [torch_ttnn/backend.py 115-117](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L115-L117)

2.   **Graph Export** (optional) - When `export_code` is enabled, captures the ATen graph for standalone script generation: [torch_ttnn/backend.py 120-126](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L120-L126)

3.   **Metrics Collection** (optional) - Performs constant folding and collects input variation schemas from the original graph: [torch_ttnn/backend.py 129-136](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L129-L136)

4.   **Bypass Check** - If `bypass_compile` is True, returns immediately without transformation: [torch_ttnn/backend.py 139-141](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L139-L141)

## Registration of TTNN Objects

Before pass execution, TTNN-specific objects must be registered with the FX graph tracer so they can be embedded in the compiled code. The `register_ttnn_objects` function adds these objects to the graph's global namespace using unique string identifiers.

### Registered Object Categories

These registered names appear in the generated FX graph code. For example, a `from_torch` call might reference `ttnn_Specified_Device` as the device argument, which PyTorch's `exec()` call will resolve to the actual device object.

**Sources:**[torch_ttnn/backend.py 79-96](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L79-L96)

### Object Registration Mechanism

The registration uses `get_add_custom_object_in_graph` utility function to associate string keys with Python objects. During graph code generation, FX replaces the string keys with the actual objects:

**Sources:**[torch_ttnn/backend.py 79-96](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L79-L96)[torch_ttnn/utils.py 61-72](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/utils.py#L61-L72)

## Memory Analysis and Eviction Optimization

When `run_mem_analysis=True`, the backend includes memory tracking to monitor SRAM usage and optionally perform eviction optimization to prevent memory overflows.

### Memory Analysis Flow

**Sources:**[torch_ttnn/backend.py 182-243](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L182-L243)

### Eviction Loop Details

The eviction optimization runs iteratively:

1.   **Check SRAM Overflow** - `mem_utils.check_sram_overflow` determines if any operation exceeds SRAM capacity: [torch_ttnn/backend.py 218](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L218-L218)

2.   **Identify Guilty Operation** - `mem_utils.which_tensors_to_evict` finds the operation causing overflow and which tensors to evict: [torch_ttnn/backend.py 222](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L222-L222)

3.   **Insert Evictions** - `EvictionPass` inserts `from_device` and `to_device` calls around the guilty operation: [torch_ttnn/backend.py 232](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L232-L232)

4.   **Reanalyze** - Runs `MemoryPass` again to verify the evictions resolved the overflow: [torch_ttnn/backend.py 230-242](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L230-L242)

5.   **Loop** - Continues until no overflows remain or max evictions reached: [torch_ttnn/backend.py 218-243](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L218-L243)

**Sources:**[torch_ttnn/backend.py 210-243](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L210-L243)

## Compilation Output and Metrics

After pass execution completes, `aten_backend` finalizes the compilation and optionally saves metrics.

### Output Artifacts

| Artifact | Condition | Storage Location | Purpose |
| --- | --- | --- | --- |
| Compiled graph | Always | `option._out_fx_graphs` | Final TTNN graph |
| TTNN graph (export) | `export_code=True` | `option._ttnn_fx_graphs` | Standalone script generation |
| Compiled schemas | `metrics_path` set | `option.compiled_schema_list` | Operation coverage analysis |
| Memory manager | `run_mem_analysis=True` | `option.memory_manager` | SRAM usage tracking |

**Sources:**[torch_ttnn/backend.py 206-250](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L206-L250)

### Final Processing Steps

Before returning, `aten_backend` performs:

1.   **Graph Cleanup** - `GraphCleanup(gm)` removes dead nodes and normalizes the graph: [torch_ttnn/backend.py 203](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L203-L203)

2.   **Schema Collection** - If metrics enabled, collects input variation schemas from the compiled graph: [torch_ttnn/backend.py 245-246](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L245-L246)

3.   **Graph Storage** - Appends the compiled graph to `_out_fx_graphs` for later analysis: [torch_ttnn/backend.py 248](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L248-L248)

4.   **Debug Output** - Prints placeholder metadata and generated code when verbose: [torch_ttnn/backend.py 252-257](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L252-L257)

5.   **Boxing** - Wraps the compiled GraphModule in a callable function via `make_boxed_func`: [torch_ttnn/backend.py 259](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L259-L259)

**Sources:**[torch_ttnn/backend.py 203-259](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L203-L259)
