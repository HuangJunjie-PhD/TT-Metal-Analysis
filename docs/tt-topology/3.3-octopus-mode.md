## Purpose and Scope

Octopus Mode is a specialized operational mode in tt-topology designed for configuring Tenstorrent Galaxy systems with N150 boards. This mode handles the complex coordination required to establish connectivity between local N150 boards and remote Galaxy systems in TGG (Tenstorrent Galaxy Gateway) and TG (Tenstorrent Galaxy) configurations.

For standard topology configurations using mesh, linear, or torus layouts, see [Layout Types](https://deepwiki.com/tenstorrent/tt-topology/3.2-layout-types). For general command-line usage, see [Command Options](https://deepwiki.com/tenstorrent/tt-topology/3.1-command-options).

## System Overview

Octopus Mode operates fundamentally differently from standard topology modes. Instead of calculating coordinates based on physical connections between boards, it programs N150 boards to communicate with Galaxy systems through a carefully orchestrated reset and configuration sequence.

### Supported Configurations

| Configuration | Local Boards | Remote Systems | Description |
| --- | --- | --- | --- |
| TGG | 8 N150s | 2 Galaxy systems | Full dual-Galaxy setup |
| TG | 4 N150s | 1 Galaxy system | Single Galaxy configuration |

## Command-Line Usage

Octopus Mode requires both the octopus flag and a reset configuration file:

### Required Arguments

| Argument | Description | Required |
| --- | --- | --- |
| `-o, --octopus` | Enable Octopus mode | Yes |
| `-r config.json` | Reset configuration file | Yes |

The system validates that a reset JSON file is provided and exits with an error if missing.

## Reset Configuration

The reset configuration file defines the mapping between local N150 boards and Galaxy systems. The configuration must be generated using `tt-topology -g` and then manually edited.

### Configuration Structure

### Configuration Fields

| Field | Type | Description |
| --- | --- | --- |
| `nb_host_pci_idx` | Array | PCI indices of local N150 boards |
| `mobo` | String | Galaxy motherboard identifier |
| `credo` | Array | Active port connections to Galaxy |
| `disabled_ports` | Array | Ports to disable during operation |

## Programming Flow

The `program_galaxy()` function orchestrates the complete configuration sequence through multiple phases.

**Programming State Machine**

### Phase Details

1.   **Modem Enable**: Configure `ETH_PARAM_MOBO_ETH_EN` on all N150 boards
2.   **Rack/Shelf Setup**: Program Galaxy systems with shelf identifiers
3.   **Initial Coordinates**: Set all N150s to (R0, S0, X0, Y0)
4.   **First Reset**: Apply retimer selection with minimal port disables
5.   **Remote Reading**: Read Galaxy coordinates and set local coordinates accordingly
6.   **Final Reset**: Apply final configuration with specified port disables
7.   **Validation**: Verify expected device counts

## Hardware Configuration Process

### Memory Address Configuration

The system writes to specific SPI register addresses to configure board behavior:

| Constant | Address | Purpose |
| --- | --- | --- |
| `ETH_PARAM_MOBO_ETH_EN` | Hardware-defined | Enable Galaxy communication |
| `ETH_PARAM_CHIP_COORD` | Hardware-defined | X/Y coordinate storage |
| `ETH_PARAM_RACK_SHELF` | Hardware-defined | Rack and shelf identifiers |

### Coordinate Assignment Logic

The `read_remote_set_local()` method implements the coordinate mapping logic:

1.   **Remote Detection**: Read coordinates from connected Galaxy systems
2.   **Shelf Mapping**: Map Galaxy shelf numbers to N150 shelf assignments 
    *   Galaxy shelf 1 → N150 shelf 0
    *   Galaxy shelf 2 → N150 shelf 3

3.   **Sorting**: Order N150s by remote coordinates for consistent assignment
4.   **Local Assignment**: Set X/Y coordinates based on sorted position

**Coordinate Flow**

### Device Validation

After configuration, the system validates that all expected devices are detected:

*   **Local Devices**: Must match pre-configuration count
*   **Remote Devices**: Each Galaxy system should expose 32 devices (expected count)
