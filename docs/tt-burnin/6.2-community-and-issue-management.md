# Community and Issue Management

Relevant source files
*   [.github/workflows/bh-update-status-by-label.yml](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/bh-update-status-by-label.yml)

This document covers the automated workflows and systems used to manage community contributions, issue tracking, and project board maintenance within the tt-burnin repository. These systems handle automatic labeling of community issues and pull requests, as well as priority-based status updates for project management.

For information about the CI/CD build and release workflows, see [CI/CD Workflows](https://deepwiki.com/tenstorrent/tt-burnin/5.2-cicd-workflows).

## Community Issue Tagging System

The repository implements an automated community issue and pull request labeling system through the `community-issue-tagging.yml` workflow. This system automatically processes new issues and pull requests from community contributors to ensure proper categorization and routing.

### Workflow Triggers

The community tagging workflow activates on two specific GitHub events:

*   `issues.opened` - when new issues are created
*   `pull_request.opened` - when new pull requests are opened

The workflow delegates the actual labeling logic to a centralized workflow maintained in the `tenstorrent/tt-github-actions` repository at [.github/workflows/community-issue-tagging.yml 11](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/community-issue-tagging.yml#L11-L11)

### Integration with Central Workflow

The workflow uses the `uses` syntax to call the central community issue handler:

`uses: tenstorrent/tt-github-actions/.github/workflows/on-community-issue.yml@main`
This design allows for consistent community management across all Tenstorrent repositories while maintaining individual repository flexibility.

## Priority-Based Status Management

The repository includes an automated system for managing issue statuses based on priority labels through the `bh-update-status-by-label.yml` workflow. This system runs daily to maintain consistency between issue labels and project board status fields.

### Scheduled Execution

The status update workflow operates on a daily schedule using cron syntax:

`schedule:  - cron: '0 0 * * *'  # Runs daily at midnight UTC`
The workflow can also be triggered manually through `workflow_dispatch` for immediate updates [.github/workflows/bh-update-status-by-label.yml 5](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/bh-update-status-by-label.yml#L5-L5)

### Priority Label Mapping

The system recognizes four priority levels and maps them to corresponding status updates:

| Priority Label | Status Update |
| --- | --- |
| P0 | Critical Priority |
| P1 | High Priority |
| P2 | Medium Priority |
| P3 | Low Priority |

### GitHub API Integration

The workflow uses Node.js with the `axios` library to interact with the GitHub API. The core logic iterates through project board issues and updates status fields based on detected priority labels [.github/workflows/bh-update-status-by-label.yml 44-82](https://github.com/tenstorrent/tt-burnin/blob/809f293d/.github/workflows/bh-update-status-by-label.yml#L44-L82)

The system requires two environment variables:

*   `GITHUB_TOKEN` - for API authentication
*   `BH_SW_BOARD_ID` - specifies the target project board

## Project Board Management Architecture

The community and issue management system integrates with GitHub project boards to provide comprehensive tracking and organization of issues and pull requests.

### Environment Configuration

Both workflows rely on GitHub Secrets for secure configuration:

*   `GITHUB_TOKEN` - Standard GitHub Actions token for API access
*   `BH_SW_BOARD_ID` - Project board identifier stored as a repository secret

### API Interaction Patterns

The workflows demonstrate two distinct patterns for GitHub API integration:

1.   **Delegated Processing**: The community tagging workflow delegates to a central repository for consistent cross-repository behavior
2.   **Direct API Manipulation**: The status update workflow directly manipulates project board cards using REST API calls
