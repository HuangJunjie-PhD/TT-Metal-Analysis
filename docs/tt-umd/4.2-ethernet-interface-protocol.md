## Purpose and Scope

This document describes the low-level Ethernet interface protocol used for inter-chip communication in Tenstorrent devices. It covers the command structures, queue mechanisms, and memory layout defined in firmware headers that enable remote device access over the Ethernet fabric. It also documents the erisc firmware heartbeat and training mechanisms used to monitor link health and version compatibility.

For higher-level abstractions that use this protocol, including `RemoteChip`, `RemoteCommunication`, and `ethernet_broadcast_write`, see [Remote Communication & Ethernet](https://deepwiki.com/tenstorrent/tt-umd/4.1-remote-communication-and-ethernet).

**Sources:**[src/firmware/riscv/wormhole/eth_interface.h 1-98](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/src/firmware/riscv/wormhole/eth_interface.h#L1-L98)[device/api/umd/device/firmware/erisc_firmware.hpp 1-27](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/firmware/erisc_firmware.hpp#L1-L27)

## Protocol Overview

The Ethernet interface protocol implements a command queue-based communication system between chips connected via Ethernet links. Each Ethernet core maintains bidirectional command queues (request and response) that enable remote memory operations across the chip fabric.

The protocol operates on a request-response model:

*   **Request Queue**: Host sends commands to remote chip.
*   **Response Queue**: Remote chip sends acknowledgments and data back.
*   Commands are fixed-size structures (`routing_cmd_t`) placed in circular buffers.
*   Read/write pointers track queue state with remote updates for synchronization.

## Command Structure

The `routing_cmd_t` structure defines the format of commands transmitted over Ethernet. Each command is exactly 32 bytes and contains all information needed to route and execute a memory operation on a remote chip.

**Diagram: routing_cmd_t Structure Layout**

## Blackhole Ethernet Training Status

On Blackhole devices, the Ethernet firmware provides detailed training and status information via the `boot_results_t` structure located at `BOOT_RESULTS_ADDR` (0x7CC00) [device/api/umd/device/types/blackhole_eth.hpp 176](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/blackhole_eth.hpp#L176-L176)

### Link Training Status (`link_train_status_e`)

The `train_status` field [device/api/umd/device/types/blackhole_eth.hpp 82](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/blackhole_eth.hpp#L82-L82) reports the outcome of the SerDes training process:

*   `LINK_TRAIN_TRAINING`: Training in progress.
*   `LINK_TRAIN_PASS`: Link training successful.
*   `LINK_TRAIN_TIMEOUT_LINK_UP`: Failed to bring link up within timeout.
*   `LINK_TRAIN_TIMEOUT_CDR_LOCK`: Failed to achieve Clock and Data Recovery (CDR) lock.

### Port Status (`port_status_e`)

The `port_status` field [device/api/umd/device/types/blackhole_eth.hpp 81](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/blackhole_eth.hpp#L81-L81) indicates the physical port state:

*   `PORT_UP`: Link is active and operational.
*   `PORT_DOWN`: Physical link is down.
*   `PORT_UNUSED`: Port is not configured for use.

**Sources:**[device/api/umd/device/types/blackhole_eth.hpp 25-44](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/blackhole_eth.hpp#L25-L44)[device/api/umd/device/types/blackhole_eth.hpp 78-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/blackhole_eth.hpp#L78-L87)

## Semantic Versioning (SemVer)

The UMD uses a specialized `SemVer` class to handle firmware versioning. This class is more permissive than standard SemVer 2.0.0 to accommodate legacy Tenstorrent firmware versioning schemes [device/api/umd/device/utils/semver.hpp 17-20](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/utils/semver.hpp#L17-L20)

### Version Unpacking

*   **Wormhole ETH FW**: Unpacked from `0xEERRCDDD` where Major is `RR`, Minor is `C`, and Patch is `DDD`[device/api/umd/device/utils/semver.hpp 33-42](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/utils/semver.hpp#L33-L42)
*   **Blackhole ETH FW**: Unpacked from `0xAAAABBCC` where Major is `AAAA`, Minor is `BB`, and Patch is `CC`[device/api/umd/device/utils/semver.hpp 46-54](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/utils/semver.hpp#L46-L54)
*   **Legacy Bundle Versions**: Versions with Major >= 80 are treated as legacy and considered smaller than non-legacy versions during comparison [device/api/umd/device/utils/semver.hpp 91-104](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/utils/semver.hpp#L91-L104)
