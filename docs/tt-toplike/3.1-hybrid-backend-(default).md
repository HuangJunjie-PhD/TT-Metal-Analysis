# 3.1. Hybrid Backend (Default)

Relevant source files
*   [docs/SYSFS_BACKEND.md](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/SYSFS_BACKEND.md?plain=1)
*   [src/backend/hybrid.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs)
*   [src/backend/smbus_smooth.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs)

The Hybrid backend combines the strengths of the Sysfs and JSON backends to provide comprehensive and efficient telemetry. It leverages the speed and non-invasiveness of `sysfs` for real-time metrics while enriching this data with detailed SMBUS telemetry obtained through a persistent streaming `tt-smi` process. This approach ensures a smooth user experience with minimal overhead and rich data.

## Purpose and Scope

The primary purpose of the Hybrid backend is to offer a robust and performant telemetry solution that overcomes the limitations of individual backends. It aims to provide:

*   **Real-time responsiveness**: Achieved by using `sysfs` for frequently updated metrics like temperature, power, voltage, and current. These reads are fast and non-blocking, ensuring the UI remains fluid.
*   **Rich SMBUS data**: Obtained from `tt-smi` for information not available via `sysfs`, such as DDR status, ARC health, board IDs, and firmware versions.
*   **Efficient data acquisition**: By employing a persistent streaming `tt-smi` process, avoiding the overhead of repeatedly spawning new processes.
*   **Smooth visual transitions**: Through Exponential Moving Average (EMA) smoothing of numeric SMBUS fields, preventing sudden jumps in displayed values.
*   **Zero-allocation render path**: Utilizing `Arc<HashMap>` for efficient data sharing between threads, minimizing heap allocations during rendering.
*   **Resilience**: Degraded mode fallback if `tt-smi` is unavailable or fails.

This page details the implementation, data flow, key functions, and classes involved in the Hybrid backend.

## Architecture and Data Flow

The Hybrid backend operates with two main data sources and processing paths:

1.   **Sysfs Real-time Path**: The `SysfsBackend` is used to read real-time hardware monitoring data from `/sys/class/hwmon/`. This path is fast, non-invasive, and provides essential metrics like temperature, voltage, power, and current. These reads occur on every `update()` call of the `HybridBackend`[src/backend/hybrid.rs 9-10](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L9-L10)
2.   **Streaming tt-smi Enrichment Path**: A separate background thread manages a persistent `tt-smi` subprocess. This subprocess streams RS-delimited JSON records containing rich SMBUS telemetry. The reader thread parses these records and makes them available to the main thread.

### Why Persistent Streaming `tt-smi`?

Traditional polling by repeatedly spawning `tt-smi -s` can introduce significant overhead [src/backend/hybrid.rs 18-21](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L18-L21):

*   Process creation overhead (~50–200 ms).
*   `tt-smi` SMBUS probe startup (~100–500 ms).
*   Sudden "surge" as all device data updates simultaneously.

By contrast, a persistent shell loop keeps the `tt-smi` process "warm." New data arrives approximately every 1.5 seconds, and the reader thread efficiently picks it up using a `read_until()` call on an open pipe [src/backend/hybrid.rs 23-26](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L23-L26)

### Data Sharing and Smoothing

The SMBUS data is shared between the background reader thread and the main render thread using `Arc<ArcSwap<HashMap<usize, SmbusTelemetry>>>`[src/backend/hybrid.rs 71-74](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L71-L74) This allows the reader thread to atomically swap the `Arc` pointer to a new `HashMap` containing the latest data. The render thread then performs an `Arc::clone()` to adopt the new snapshot, which is a single atomic increment of the reference count, resulting in zero heap allocations during the render path [src/backend/hybrid.rs 39-42](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L39-L42)

Numeric fields from the SMBUS data (e.g., ARC health counters, DDR speed, clock frequencies) are smoothed using an Exponential Moving Average (EMA) with an alpha value of 0.25 [src/backend/hybrid.rs 29-30](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L29-L30) This distributes visual changes over approximately 4 render frames (around 400 ms at 100 ms/frame), eliminating sudden "pops" in the UI [src/backend/smbus_smooth.rs 15-26](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L15-L26) Discrete fields (like board IDs, firmware versions, DDR status bitmask) are copied verbatim as they have no meaningful interpretation as floats [src/backend/hybrid.rs 32-33](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L32-L33)[src/backend/smbus_smooth.rs 88-90](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L88-L90)

### Degraded Mode

If the `sh` shell is absent or `tt-smi` fails to start, the backend falls back to a 5-second blocking poll loop, similar to the previous implementation [src/backend/hybrid.rs 46-47](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L46-L47) If `tt-smi` is entirely unavailable, the backend operates in a `sysfs`-only mode [src/backend/hybrid.rs 47-48](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L47-L48)

## Key Components and Implementation Details

### `HybridBackend` Struct

The `HybridBackend` struct [src/backend/hybrid.rs 64-115](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L64-L115) encapsulates the state and logic for this backend.

*   `sysfs`: An instance of `SysfsBackend`[src/backend/hybrid.rs 65](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L65-L65) responsible for real-time `sysfs` reads.
*   `tt_smi_path`: The resolved absolute path to the `tt-smi` executable [src/backend/hybrid.rs 68](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L68-L68)
*   `smbus_shared`: An `Arc<ArcSwap<HashMap<usize, SmbusTelemetry>>>`[src/backend/hybrid.rs 71-74](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L71-L74) that holds the latest SMBUS telemetry snapshot. `ArcSwap` enables lock-free atomic updates by the reader thread and efficient cloning by the render thread.
*   `device_meta_shared`: Similar to `smbus_shared`, but stores firmware information and device limits [src/backend/hybrid.rs 77-88](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L77-L88)
*   `smbus_latest`: The render thread's private `Arc` reference to the latest SMBUS snapshot [src/backend/hybrid.rs 91](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L91-L91)
*   `smbus_generation`: An `Arc<AtomicU64>`[src/backend/hybrid.rs 98](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L98-L98) incremented by the reader thread to signal new data.
*   `smbus_snapshot_gen`: The generation count reflected in `smbus_latest`[src/backend/hybrid.rs 101](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L101-L101)
*   `smbus_blended`: A `HashMap` storing the EMA-smoothed SMBUS telemetry, which is what `smbus_telemetry()` actually returns [src/backend/hybrid.rs 105](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L105-L105)
*   `smbus_ema`: A `HashMap` storing the `SmbusEmaState` for each device, used to maintain EMA accumulators [src/backend/hybrid.rs 109](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L109-L109)
*   `stop_flag`: An `Arc<AtomicBool>`[src/backend/hybrid.rs 112](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L112-L112) to signal the background thread to terminate.
*   `refresh_handle`: A `Mutex<Option<thread::JoinHandle<()>>>`[src/backend/hybrid.rs 115](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L115-L115) holding the handle to the background reader thread.

### `TelemetryBackend` Implementation

The `HybridBackend` implements the `TelemetryBackend` trait [src/backend/hybrid.rs 152](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L152-L152)

*   **`init()`**: 
    *   Initializes the `sysfs` backend for primary device detection [src/backend/hybrid.rs 155-157](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L155-L157)
    *   Resolves the `tt-smi` path to an absolute path for robustness [src/backend/hybrid.rs 163-169](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L163-L169)
    *   Spawns the background reader thread (`spawn_refresh_thread`) [src/backend/hybrid.rs 170-171](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L170-L171)
    *   The reader thread attempts to start a persistent `tt-smi -s` subprocess. If this fails, it falls back to a blocking `tt-smi` poll or `sysfs`-only mode [src/backend/hybrid.rs 200-204](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L200-L204)

*   **`update()`**: 
    *   Calls `sysfs.update()` to get real-time metrics [src/backend/hybrid.rs 260](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L260-L260)
    *   Checks `smbus_generation` against `smbus_snapshot_gen` to see if new SMBUS data is available [src/backend/hybrid.rs 263-264](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L263-L264)
    *   If new data is available, it updates `smbus_latest` and `smbus_snapshot_gen`[src/backend/hybrid.rs 265-267](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L265-L267)
    *   Iterates through the devices, applying EMA smoothing to numeric SMBUS fields using `smbus_smooth::apply_ema`[src/backend/hybrid.rs 274-276](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L274-L276) and enriching the `Device` telemetry with both `sysfs` and smoothed SMBUS data [src/backend/hybrid.rs 277-280](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L277-L280)
    *   Updates firmware info and device limits from `device_meta_shared`[src/backend/hybrid.rs 281-286](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L281-L286)

*   **`device_count()`**: Returns the number of devices detected by the `sysfs` backend [src/backend/hybrid.rs 291](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L291-L291)
*   **`get_device_info()`**: Retrieves device information from the `sysfs` backend [src/backend/hybrid.rs 296](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L296-L296)
*   **`get_telemetry()`**: Returns the combined `Telemetry` for a specific device [src/backend/hybrid.rs 301](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L301-L301)

### `spawn_refresh_thread` Function

This private function [src/backend/hybrid.rs 310-312](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L310-L312) is responsible for launching the background thread that manages the `tt-smi` subprocess.

*   It constructs the `tt-smi` command, preferring `sh -c "while true; do tt-smi -s; echo -e '\x1e'; sleep 1.5; done"` for persistent streaming [src/backend/hybrid.rs 320-324](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L320-L324) The `\x1e` character is the Record Separator (RS) used to delimit JSON records.
*   If the streaming command fails, it falls back to `tt-smi -s` (blocking poll) [src/backend/hybrid.rs 330-331](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L330-L331)
*   The thread continuously reads from the `tt-smi` stdout, parsing RS-delimited JSON records [src/backend/hybrid.rs 350-352](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L350-L352)
*   Parsed `SmbusTelemetry` and device metadata are stored in new `HashMap`s, wrapped in `Arc`, and then atomically swapped into `smbus_shared` and `device_meta_shared`[src/backend/hybrid.rs 360-367](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L360-L367)
*   `smbus_generation` is incremented to signal new data [src/backend/hybrid.rs 368](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/hybrid.rs#L368-L368)

### `smbus_smooth` Module

The `smbus_smooth` module [src/backend/smbus_smooth.rs 1-3](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L1-L3)[src/backend/smbus_smooth.rs 5-7](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L5-L7) handles the Exponential Moving Average (EMA) smoothing of numeric SMBUS telemetry fields.

*   **`try_parse_numeric(s: &str) -> Option<f64>`**: Parses a string into an `f64`, supporting decimal, integer, and hex formats. Returns `None` for non-numeric strings [src/backend/smbus_smooth.rs 40-51](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L40-L51)
*   **`EMA_ALPHA`**: A constant `f64` value (0.25) determining the fraction of the new value applied per frame [src/backend/smbus_smooth.rs 54](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L54-L54)
*   **`DeviceEmaState`**: A struct holding `FieldEma` (an `Option<f64>`) for each smoothable numeric field of a device [src/backend/smbus_smooth.rs 60-81](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L60-L81)
*   **`SmbusEmaState`**: A `HashMap` mapping device indices to their `DeviceEmaState`[src/backend/smbus_smooth.rs 84](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L84-L84)
*   **`apply_ema(ema: &mut SmbusEmaState, device_idx: usize, incoming: &SmbusTelemetry, existing: &mut SmbusTelemetry)`**: This is the core smoothing function [src/backend/smbus_smooth.rs 93-97](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L93-L97)
    *   It retrieves or creates the `DeviceEmaState` for the given `device_idx`[src/backend/smbus_smooth.rs 99](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L99-L99)
    *   Discrete fields are copied directly from `incoming` to `existing`[src/backend/smbus_smooth.rs 102-131](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L102-L131)
    *   For numeric fields, it calls the `blend` helper function, which calculates the new EMA value and updates both the `DeviceEmaState` and the `existing``SmbusTelemetry`[src/backend/smbus_smooth.rs 133-154](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L133-L154)

### `blend` Helper Function

This function [src/backend/smbus_smooth.rs 160-179](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L160-L179) performs the actual EMA calculation. If an `incoming` value is numeric, it blends it with the `current_ema` (if available) or initializes the EMA. The blended value is then formatted back into a string and stored in `existing`. Non-numeric strings are copied directly.

## Performance Characteristics

The Hybrid backend is designed for optimal performance:

*   **Latency**: `sysfs` reads are typically <1ms per device. The `tt-smi` streaming provides updates every ~1.5s, with EMA smoothing distributing visual changes over ~400ms [src/backend/smbus_smooth.rs 25-26](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/smbus_smooth.rs#L25-L26)
*   **CPU Usage**: Minimal. `sysfs` reads are very light. The background `tt-smi` process and parsing are offloaded to a separate thread, preventing blocking of the main render loop.
*   **Memory**: Efficient `Arc<HashMap>` usage and `ArcSwap` minimize memory allocations and overhead.

## Usage

The Hybrid backend is the default choice for `tt-toplike`. It will be automatically selected if `tt-smi` is found and `sysfs` devices are detected.

`# Default behavior, will auto-detect and use Hybrid if possible./target/debug/tt-toplike-tui # Explicitly select the hybrid backend (if tt-smi is in PATH)./target/debug/tt-toplike-tui --backend hybrid # If tt-smi is not in PATH, specify its full path./target/debug/tt-toplike-tui --backend hybrid --tt-smi-path /usr/bin/tt-smi`
