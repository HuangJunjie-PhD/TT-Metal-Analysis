## Purpose and Scope

CPU Code Generation is the backend component of TVM that translates optimized TensorIR (TIR) programs into executable machine code for CPU targets. This subsystem uses LLVM as its intermediate representation and provides a layered architecture supporting multiple CPU architectures including x86-64, ARM, and specialized DSPs.

This page focuses specifically on CPU code generation. For GPU code generation (CUDA/ROCm), see [6.3](https://deepwiki.com/tenstorrent/tt-tvm/6.3-gpu-code-generation). For the LLVM backend architecture overview, see [6.1](https://deepwiki.com/tenstorrent/tt-tvm/6.1-llvm-backend-architecture). For the overall code generation backend system, see [6](https://deepwiki.com/tenstorrent/tt-tvm/6-code-generation-backends).

## Core Components

### CodeGenLLVM Base Class

`CodeGenLLVM` provides the foundation for all LLVM-based code generation in TVM. It inherits from both `ExprFunctor` and `StmtFunctor` to traverse TIR expressions and statements.

**Key Responsibilities:**

*   **LLVM IR Construction**: Uses `llvm::IRBuilder` to emit LLVM instructions
*   **Type System**: Converts TVM DataTypes to LLVM types via `DTypeToLLVMType()`
*   **Memory Operations**: Handles buffer loads/stores with TBAA (Type-Based Alias Analysis) metadata
*   **Control Flow**: Generates basic blocks for loops, conditionals, and branches
*   **Function Management**: Creates and links LLVM functions from PrimFunc

### Platform-Specific Backends

#### CodeGenX86_64

Handles x86-64 specific optimizations, particularly for vector operations and half-precision floating point.

**Optimizations:**

*   **FP16 Conversion**: Uses AVX-512 `vcvtph2ps_512` intrinsic for vectorized half→float conversion
*   **Vector Intrinsics**: Calls platform intrinsics when beneficial over LLVM's default lowering

**Sources:**[src/target/llvm/codegen_x86_64.cc 47-127](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_x86_64.cc#L47-L127)

#### CodeGenARM

Provides ARM/NEON-specific optimizations.

**Optimizations:**

*   **Popcount**: Custom lowering using NEON `vcnt` (8-bit popcount) + `vpaddlu` (pairwise add-widen) to achieve efficient popcount for wider types
*   **Vector Width**: Sets `native_vector_bits_ = 128` (16 bytes for NEON)

**Sources:**[src/target/llvm/codegen_arm.cc 43-122](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_arm.cc#L43-L122)

## Code Generation Flow

**Detailed Steps:**

1.   **Target Initialization** ([codegen_llvm.cc 144-230](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L144-L230))

    *   Create `llvm::TargetMachine` with CPU architecture, features, and optimization level
    *   Set module target triple and data layout
    *   Detect native vector width (512-bit for x86-64 AVX-512, 128-bit for ARM)

2.   **Runtime Setup** ([codegen_cpu.cc 77-189](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L77-L189))

    *   Define LLVM struct types for `DLTensor`, `TVMValue`, `DLDevice`, etc.
    *   Declare external runtime functions (`TVMFuncCall`, `TVMBackendParallelLaunch`, etc.)
    *   Initialize global context pointer `gv_mod_ctx_`

3.   **Function Declaration** ([codegen_llvm.cc 264-298](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L264-L298))

    *   Convert PrimFunc parameters to LLVM function signature
    *   Create `llvm::Function` with appropriate linkage (external or private)
    *   Mark pointer parameters with `noalias` attribute if `tir::attr::kNoAlias` is set

4.   **Body Emission** ([codegen_llvm.cc 300-352](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L300-L352))

    *   Create entry basic block
    *   Visit TIR body using `StmtFunctor` pattern
    *   Generate return instruction

5.   **Optimization** ([codegen_llvm.cc 413-558](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L413-L558))

    *   Run LLVM optimization passes based on opt level (O0-O3)
    *   Apply inlining, loop vectorization, SLP vectorization
    *   Target-specific adjustments via `TargetMachine::adjustPassManager()`

6.   **Module Finalization** ([codegen_llvm.cc 362-373](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L362-L373))

    *   Add startup function for system library mode
    *   Link imported LLVM modules
    *   Verify and return `llvm::Module`

**Sources:**[src/target/llvm/codegen_llvm.cc 144-558](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L144-L558)[src/target/llvm/codegen_cpu.cc 77-189](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L77-L189)

## Packed Function Interface

The packed function ABI is TVM's unified calling convention for dynamically-typed functions that can be called from Python, C++, or other languages.

**Key Implementation Details:**

*   **Argument Unpacking**: `CreateStructRefPtr()` with `builtin::kTVMValueContent` to access union members ([codegen_cpu.cc 437-451](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L437-L451))
*   **Type Checking**: Generated code checks `type_codes` array matches expected types
*   **Pointer Arguments**: Handled as `void*` in `TVMValue` union, cast to correct pointer type
*   **Return Values**: Single return value stored in `out_ret_value` with type code in `out_ret_tcode`

**Sources:**[src/target/llvm/codegen_cpu.cc 77-189](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L77-L189)[src/target/llvm/codegen_cpu.cc 364-456](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L364-L456)

## Parallel Execution Support

TVM generates code for multi-threaded execution using the `TVMBackendParallelLaunch` runtime API.

**Implementation Details:**

*   **Parallel Pragma Detection** ([codegen_cpu.cc 806-835](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L806-L835)):

    *   `pragma::kParallelLaunchPoint` marks outer loop
    *   `pragma::kParallelStridePattern` enables strided iteration
    *   `pragma::kParallelBarrierWhenFinish` adds barrier after loop

*   **Closure Packing** ([codegen_cpu.cc 653-677](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L653-L677)):

    *   Creates LLVM struct type containing all captured variables
    *   Allocates on stack using `CreateAlloca`
    *   Stores captured values with `CreateStore`

*   **Lambda Generation** ([codegen_cpu.cc 691-790](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L691-L790)):

    *   Creates internal function with signature `int(int, TVMParallelGroupEnv*, void*)`
    *   Unpacks closure data to restore captured variables
    *   Computes iteration range based on `task_id` and `num_tasks`
    *   Executes loop body for assigned range

*   **Runtime Call** ([codegen_cpu.cc 780-789](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L780-L789)):

    *   Calls `TVMBackendParallelLaunch` (or context-injected function pointer)
    *   Passes lambda function pointer, closure data pointer, and num_tasks
    *   Checks return code for errors

**Sources:**[src/target/llvm/codegen_cpu.cc 653-790](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L653-L790)[src/target/llvm/codegen_cpu.cc 806-895](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L806-L895)

## Memory Management

### Buffer Access and Pointer Arithmetic

TVM generates efficient buffer access code with proper alignment and alias information.

**Buffer Pointer Creation** ([codegen_llvm.cc 961-994](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L961-L994)):

**TBAA Metadata** ([codegen_llvm.cc 629-689](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L629-L689)):

*   Creates hierarchical TBAA tree based on buffer variable address
*   Enables LLVM to reason about pointer aliasing
*   Improves optimization opportunities (e.g., load hoisting, dead store elimination)

**Alignment Handling** ([codegen_llvm.cc 691-718](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L691-L718)):

*   Uses modular arithmetic analysis to determine natural alignment
*   Propagates alignment information to load/store instructions
*   Higher alignment enables vectorization and improved performance

**Sources:**[src/target/llvm/codegen_llvm.cc 629-718](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L629-L718)[src/target/llvm/codegen_llvm.cc 961-994](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L961-L994)

### Allocation Strategies

**Stack Allocation** ([codegen_cpu.cc 566-651](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L566-L651)):

*   Uses `alloca` instruction for thread-local buffers

## External Function Calls

TVM supports calling external C functions for operations not implemented in TIR.

**Implementation** ([codegen_cpu.cc 458-489](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_cpu.cc#L458-L489)):

1.   **Function Declaration**:

    *   Check if function pointer is injected via context (`gv_func_map_`)
    *   Check if function already exists in module
    *   Otherwise, create external function declaration with inferred signature

2.   **Argument Conversion**:

    *   Convert TIR expressions to LLVM values using `MakeValue()`
    *   Collect argument types to build `llvm::FunctionType`

3.   **Type Casting**:

    *   Cast function pointer to expected signature if needed
    *   Ensure pointer types match for call instruction

4.   **Call Emission**:

    *   Use `IRBuilder::CreateCall` with function and argument values

**Sources:**[src/target/llvm/codegen_cpu.cc 458-489](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L458-L489)

## Platform-Specific Optimizations

### x86-64 Optimizations

**Half-Precision Conversion** ([codegen_x86_64.cc 56-94](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_x86_64.cc#L56-L94)):

| Vector Width | LLVM Version | CPU Feature | Intrinsic Used |
| --- | --- | --- | --- |
| 16+ lanes | Any | AVX-512 | `x86_avx512_mask_vcvtph2ps_512` |
| 8+ lanes | ≤10 | F16C | `x86_vcvtph2ps_256` |
| Other | Any | Any | Default LLVM lowering |

The implementation detects available CPU features and selects the most efficient vectorized conversion intrinsic. For vectors larger than the intrinsic width, it splits the computation, processes each chunk, and concatenates results.

**Sources:**[src/target/llvm/codegen_x86_64.cc 47-127](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_x86_64.cc#L47-L127)

### ARM/NEON Optimizations

**Popcount Optimization** ([codegen_arm.cc 70-122](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_arm.cc#L70-L122)):

ARM lacks native popcount for 16/32/64-bit types, but has efficient 8-bit `vcnt`. The optimization:

1.   Reinterpret input as vector of 8-bit values
2.   Apply 8-bit popcount: `vcnt.8`
3.   Pairwise add and double width iteratively using `vpaddlu`: 
    *   8-bit → 16-bit: `vpaddlu.u8`
    *   16-bit → 32-bit: `vpaddlu.u16`
    *   32-bit → 64-bit: `vpaddlu.u32`

This sequence is significantly faster than LLVM's default scalar lowering.

**Vector Width Configuration** ([codegen_arm.cc 48-52](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_arm.cc#L48-L52)):

*   Sets `native_vector_bits_ = 128` (16 bytes) for NEON
*   Enables automatic vectorization of loops up to this width

**Sources:**[src/target/llvm/codegen_arm.cc 43-122](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_arm.cc#L43-L122)

## Registry and Factory Pattern

CPU code generators are registered in TVM's global function registry for dynamic dispatch based on target architecture.

**Implementation** ([codegen_llvm.cc 126-142](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/codegen_llvm.cc#L126-L142)):

1.   Extract target architecture name from `TargetMachine`
2.   Try to find registered factory: `"tvm.codegen.llvm.target_" + target`
3.   Fallback to generic CPU codegen: `"tvm.codegen.llvm.target_cpu"`
4.   Return `unique_ptr<CodeGenLLVM>` to caller

*   `test_llvm_void_intrin`: Void-returning intrinsics (e.g., `llvm.va_start`)
*   `test_llvm_large_uintimm`: Large 64-bit unsigned immediate values

**Vectorization** ([test_target_codegen_llvm.py 183-208](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/test_target_codegen_llvm.py#L183-L208)):

*   `test_llvm_vadd_pipeline`: Vector addition with parallelization
*   `test_llvm_flip_pipeline`: Vector slicing and reversal
*   `test_llvm_madd_pipeline`: Multi-dimensional buffer access

*   `test_alignment`: Verifies alignment attributes on loads/stores
