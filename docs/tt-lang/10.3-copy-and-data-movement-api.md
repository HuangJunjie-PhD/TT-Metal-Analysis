## Purpose and Scope

This page provides a technical reference for the **tt-lang** data movement API, specifically focusing on `ttl.copy`, tensor slicing, and inter-core communication via pipes. These abstractions allow developers to manage explicit data transfers between the memory hierarchy (DRAM, L1), circular buffers (Dataflow Buffers), and across the Tensix core grid.

Key components covered:

*   **`ttl.copy()`**: The primary interface for asynchronous data transfer.
*   **Tensor Slicing**: Tile-based coordinate indexing for tensor views.
*   **Transfer Handles**: Objects used for synchronization via `wait()`.
*   **Pipes and PipeNet**: High-level abstractions for multicast and point-to-point inter-core communication.
*   **GroupTransfer**: Batch synchronization of multiple data movement operations.

## Data Movement Overview

Data movement in tt-lang is centered around the `ttl.copy` operation, which moves data between different "endpoints." In the tt-lang programming model, these transfers are typically asynchronous to allow for double-buffering and overlapping of compute and data movement.

## Tensor Slicing API

Before moving data from a large tensor, a slice is created. Unlike standard Python slicing, tt-lang slicing operates on **tile coordinates** (32x32 blocks) when the tensor is in `TILE_LAYOUT`.

### DSL Syntax

`# Create a slice of 2x2 tiles starting at tile index (0, 1)# Resulting slice has shape (2, 2) in tile unitsslice_view = my_tensor[0:2, 1:3]`
The slicing mechanism allows kernels to process large tensors in smaller, L1-resident chunks. The indexing logic supports both `TILE` and `ROW_MAJOR` layouts. In MLIR, this is represented by the `ttl.tensor_slice` operation, which takes variadic indices representing tile coordinates. The result rank may be less than the tensor rank (rank-reducing slice), where leading dimensions are squeezed by scalar subscripts.

Sources: [include/ttlang/Dialect/TTL/IR/TTLOps.td 80-120](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L80-L120)[lib/Dialect/TTL/IR/TTLOps.cpp 140-166](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/IR/TTLOps.cpp#L140-L166)

## The `ttl.copy` Operation

`ttl.copy` initiates a transfer between supported endpoints. It returns a `CopyTransaction` (in simulation) or a transfer handle (`!ttl.transfer_handle`) in the MLIR dialect that must be synchronized.

### Supported Transfer Patterns

The `CopyTransaction` supports specific endpoint combinations via a registry-based strategy pattern defined in `HANDLER_REGISTRY`:

| Source | Destination | Handler Class |
| --- | --- | --- |
| `Tensor` | `Block` | `TensorToBlockHandler` |
| `Block` | `Tensor` | `BlockToTensorHandler` |
| `Block` | `Pipe` | `BlockToPipeHandler` (Multicast Send) |
| `Pipe` | `Block` | `PipeToBlockHandler` (Receive) |

### Simulation State Machine and Locking

In the simulator, `copy` operations interact with the `Block` state machine to prevent race conditions. When a transaction is created, it marks the involved blocks to transition their internal state.

**Diagram: Copy Entity Mapping**

**State Transitions:**

*   **Source Blocks**: Transition via `mark_copy_as_source()`. This prevents writes to the block while a copy is reading from it.
*   **Destination Blocks**: Transition via `mark_copy_as_dest()`. This locks the block against all user access (reads or writes) during the transfer.
*   **Completion**: Calling `tx.wait()` triggers `mark_tx_wait_complete()`, transitioning blocks back to accessible states once the `transfer()` method completes.

## Inter-Core Communication: Pipes

Pipes provide a high-level abstraction for moving data between Tensix cores without manually calculating NOC addresses.

### Pipe and PipeNet

*   **`Pipe`**: Defines a source core and a destination core range (multicast).
*   **`PipeNet`**: A collection of pipes. It provides `if_src` and `if_dst` context managers to execute code only on relevant cores.
*   **`SrcPipeIdentity` / `DstPipeIdentity`**: Used to identify specific endpoints in a pipe network.

### Multicast Logic

When a `Block` is copied to a `Pipe`, the `BlockToPipeHandler` stores data in a shared buffer (`pipe_buffer`) accessible by all nodes. It calculates the number of receivers based on the `dst_node_range`, which can be a single `NodeCoord` or a `NodeRange`.

## Group Transfers

The `GroupTransfer` class allows batching multiple `CopyTransaction` handles. This is particularly useful for complex data movement patterns where multiple buffers must be filled or flushed simultaneously before computation can proceed.

### Usage Pattern

1.   Create a `GroupTransfer` instance.
2.   Add handles returned by `ttl.copy()` using `add()`.
3.   Call `wait_all()` to synchronize all transfers.

Sources: [python/sim/copy.py 228-237](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L228-L237)

## MLIR Lowering and Transformation

During compilation, `ttl.copy` operations are lowered to hardware-specific instructions. The `ConvertTTLTileOpsToTTKernel` pass handles the translation of tile-level movement.

*   **CB Index Lookup**: The compiler traces back from the copy source to find the original `ttl.bind_cb` operation to determine the hardware circular buffer slot via `lookupCBByIndex`.
*   **Tile Index Calculation**: For transfers involving circular buffers, the `computeCBTileIndex` function linearizes multi-dimensional indices into a flat CB offset by tracing defining `tensor.extract` ops.
*   **Synchronization**: `ttl.wait` operations block until the asynchronous transfer identified by the input transfer handle is complete. In current MVP lowering, this uses TTKernel global NOC barriers.

Sources: [lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 59-82](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L59-L82)[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 133-153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L133-L153)[include/ttlang/Dialect/TTL/IR/TTLOps.td 164-171](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L164-L171)
