## Purpose and Scope

`ToTtPass` is the core operation lowering pass that transforms PyTorch ATen operations into TTNN equivalents. It converts FX graph nodes with targets like `torch.ops.aten.add.Tensor`, `torch.ops.aten.mm.default` into `ttnn.add`, `ttnn.matmul` that execute on Tenstorrent hardware.

The pass implements a two-stage lowering strategy:

*   **Stage 1:**`ReplaceMoreTt` transformer handles straightforward one-to-one operation mappings
*   **Stage 2:**`ReplaceMoreTtManually` function handles complex operations requiring multi-step decomposition, type casting, or shape-dependent logic

Every operation conversion is guarded by `can_lowering_to_ttnn` which checks autogenerated and manual blocklists to prevent compilation failures.

Related pages:

*   See page 2.3 for `AddDataMovePass` which inserts data movement operations after lowering
*   See page 2.4 for the guard system and blocklist management
*   See page 2.5 for target wrapper functions used in lowering

## Architecture and Execution Flow

The `ToTtPass` is executed as part of the pass pipeline orchestrated by `PassManager` in the `aten_backend` function. It operates on a PyTorch FX graph that has already been decomposed into ATen primitives by AOT Autograd and undergone analysis passes.

### Pass Pipeline Position

**Title:** ToTtPass Position in aten_backend Compilation Pipeline

The `PassManager` in `aten_backend` instantiates `ToTtPass(device, use_less_ttnn_op_types)` and calls `call(gm)` which executes both transformation stages sequentially. The pass returns a `PassResult(gm, modified)` where `modified=True` if any operations were lowered.

**Sources:**[torch_ttnn/backend.py 166-180](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L166-L180)[torch_ttnn/passes/lowering/to_tt_pass.py 1280-1295](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L1280-L1295)

## Two-Stage Transformation Mechanism

`ToTtPass` implements operation lowering using two complementary mechanisms that handle different complexity levels of operation conversions.

### Stage 1: ReplaceMoreTt Transformer

**Title:** ReplaceMoreTt Transformer Flow

`ReplaceMoreTt` extends `torch.fx.Transformer` and overrides `call_function` to intercept PyTorch operations. For each call, it:

1.   Creates a `PseudoNode` with the operation's target, args, and kwargs
2.   Checks `can_lowering_to_ttnn(pseudo_node)` which queries blocklists and constraints
3.   If allowed, maps the operation using dictionaries like `TTNN_POINTWISE_UNARY_OPS` or hardcoded logic
4.   Calls `call_function_prop_meta` to create the new node with preserved metadata
5.   Collects `original_input_variations` metrics for documentation/testing

The transformer also overrides `placeholder` and `get_attr` to restore original metadata for input and parameter nodes.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 163-191](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L163-L191)[torch_ttnn/passes/lowering/to_tt_pass.py 204-221](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L204-L221)[torch_ttnn/passes/lowering/to_tt_pass.py 224-372](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L224-L372)

### Stage 2: ReplaceMoreTtManually Function

**Title:** ReplaceMoreTtManually Manual Lowering Flow

`ReplaceMoreTtManually` handles operations requiring complex transformations:

*   **Binary elementwise:**`lower_binary_eltwise` checks shapes, casts int to bfloat16, swaps scalar operands, handles sub/div special cases
*   **Layer norm:** Analyzes `getitem` users to determine if mean/rstd are needed, chooses `ttnn.layer_norm` or `ttnn.moreh.layer_norm`
*   **Shape operations:**`expand` chooses between `ttnn.expand` (aligned last dim) or `target_wrappers.repeat` based on constraints
*   **High-rank transpose:** Reshapes to ≤4 dims, transposes, reshapes back to original output shape

Each operation handler returns `new_node` on success or `None` to keep the original ATen operation.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 440-450](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L440-L450)[torch_ttnn/passes/lowering/to_tt_pass.py 473-530](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L473-L530)[torch_ttnn/passes/lowering/to_tt_pass.py 548-604](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L548-L604)[torch_ttnn/passes/lowering/to_tt_pass.py 687-845](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L687-L845)

## GraphWrapper and Metadata Propagation

`ReplaceMoreTtManually` uses `GraphWrapper` to create new nodes while preserving metadata from the original ATen node.

**Title:** GraphWrapper Metadata and Shape Inference

`GraphWrapper` wraps the FX graph to provide three key functions:

1.   **Metadata preservation:** Each new node inherits `dict(self.node.meta)` from the original ATen node
2.   **Metrics collection:** Stores `original_input_variations` using `metrics.collect_input_variation_from_node(self.node)`
3.   **Shape inference:**`_get_output_val` computes output shape by mapping TTNN ops to equivalent torch ops on fake tensors (e.g., `ttnn.add` → `torch.add(fake_a, fake_b)`)

For `get_attr` nodes, it retrieves the actual tensor with `getattr(self.gm, target)` and converts to fake tensor using `fake_mode.fake_tensor_converter`.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 386-438](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L386-L438)

## Guard Integration and Blocklist Checking

Both transformation stages check `can_lowering_to_ttnn` before attempting to lower operations.

**Title:** Guard System Flow in ToTtPass

`can_lowering_to_ttnn` queries the `GUARD` dictionary from `to_tt_guard.py`:

*   If `node.target` has an entry, it calls the associated guard function
*   Guard functions like `guard_aten(blocklist, node)` extract input variations and check against blocklists
*   Blocklists contain stringified input signatures like `["Tensor<[1, 3, 224, 224]> input = ?", "Tensor<[768, 3, 32, 32]> weight = ?", ...]`
*   If inputs match a blocklist entry, the operation is blocked (returns `False`)
*   If no guard exists or inputs don't match blocklist, lowering is allowed (returns `True`)

Additional checks in `ReplaceMoreTt`:

*   `are_args_from_int_output_ops(args)` blocks if arguments come from `argmax` or `argmin`
*   `is_target_incompatible_with_grayskull(target, device)` blocks operations unsupported on Grayskull hardware

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 44-70](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L44-L70)[torch_ttnn/passes/lowering/to_tt_pass.py 210-221](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L210-L221)[torch_ttnn/passes/lowering/to_tt_guard.py 109-113](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L109-L113)[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 288-310](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L288-L310)

## Special Case Handling

### Binary Elementwise with Type Casting and Scalar Handling

`lower_binary_eltwise(fn, args)` handles complex transformations for `torch.ops.aten.add.Tensor`, `mul.Tensor`, `sub.Tensor`, `div.Tensor`:

**Title:** lower_binary_eltwise Transformation Logic

**Constraints checked:**

*   Returns `None` if shapes are `None` or incompatible (rank > 4 with mismatched -3:-2 dims)

**Type casting:**

*   `cast_bf16(target, other)` casts int32/int64 to bfloat16 when the other operand is float to avoid mixed-type errors

**Scalar position workaround (tt-metal#31247):**

*   TTNN binary ops only accept scalar in second operand
*   If first operand is scalar, swap operands
*   For `ttnn.sub`, negate both operands after swap
*   For `ttnn.div`, compute `ttnn.mul(ttnn.reciprocal(new_args[0]), new_args[1])`

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 473-514](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L473-L514)

### Native Layer Norm with Usage Analysis

`torch.ops.aten.native_layer_norm.default` returns `(output, mean, rstd)`, but not all consumers use mean/rstd.

**Title:** Native Layer Norm Lowering with Conditional Statistics

The lowering performs usage analysis to choose the appropriate TTNN operation:

1.   Counts `getitem` users of the layer norm node
2.   If `getitem[1]` or `getitem[2]` exist, sets `use_mean`/`use_rstd` flags
3.   Calls `target_wrappers.native_layer_norm` which: 
    *   Uses `ttnn.layer_norm` if only output is needed (faster)
    *   Uses `ttnn.moreh.layer_norm` if mean/rstd are needed (returns all three)
    *   Reshapes mean/rstd to match ATen's expected shape (keeps normalized dims)

**Key difference:** ATen's mean/rstd keep normalized dimensions (e.g., `[batch, hidden_dim]`), but TTNN removes them (e.g., `[batch]`). The wrapper reshapes to match ATen's shape.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 548-604](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L548-L604)[torch_ttnn/passes/lowering/target_wrappers.py 372-413](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/target_wrappers.py#L372-L413)

### Expand with Shape-Dependent Logic

`torch.ops.aten.expand.default` has different semantics than `ttnn.expand` and `ttnn.repeat`.

**Title:** Expand Operation Lowering Decision Tree

**Semantics difference:**

*   `aten.expand(input, [2, 3, 4])` → output shape is `[2, 3, 4]`
*   `ttnn.repeat(input, [2, 3, 4])` → repeat each dim by `[2, 3, 4]` times

**Constraints:**

*   `ttnn.expand` requires `input_shape[-1] % 2 == 0` and contiguous expansion dims (tt-metal#31270)
*   If input and output have different ranks, use `ttnn.experimental.view` to add leading dims
*   Falls back to ATen if constraints can't be met

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 687-744](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L687-L744)

## View and Reshape Operations

### View vs Reshape Decision Logic

**Title:** View vs Reshape Selection Logic

The distinction is important because:

*   `ttnn.experimental.view` preserves existing padding, faster but more restrictive
*   `ttnn.reshape` may re-pad tensors, slower but more flexible
*   Tile layout requires special attention to padding in last two dimensions

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 879-907](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L879-L907)

### Transpose with Rank > 4 via Reshape Strategy

`ttnn.transpose` only supports tensors with rank ≤ 4. For higher ranks, reshape to collapse dimensions.

**Title:** High-Rank Transpose via Dimension Collapsing

For `torch.ops.aten.transpose.int` with rank > 4:

1.   Collapse dimensions between and outside the transposed dims to reduce rank to ≤ 4
2.   Example: `[A, B, C, D, E]` transposing dims 1 and 3 → reshape to `[A, B, (C), D, (E)]`
3.   Transpose on the reshaped tensor
4.   Reshape back to original output shape

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 793-867](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L793-L867)

## Workarounds and Limitations

### Grayskull Hardware Incompatibilities

Certain operations are incompatible with Grayskull architecture and are blocked:

The `is_target_incompatible_with_grayskull` function checks the device type and blocks these operations during lowering.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 52-62](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L52-L62)

### Negative Exponent Workaround

This workaround decomposes `pow(x, -n)` into `reciprocal(pow(x, n))` to support negative exponents.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 282-296](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L282-L296)

### Scalar Position Limitation

TTNN binary operations only support scalars in the second operand position. When the first operand is scalar, the manual pass swaps operands and adjusts the operation accordingly (negating both operands for `sub`, using reciprocal+multiply for `div`).

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 495-511](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L495-L511)

### Padding and Layout Constraints

Several operations have layout and padding requirements:

*   `constant_pad_nd`: Cannot handle negative padding (#516), switches to row-major for non-tile-aligned tensors
*   `slice`: Only unpad tilized tensors with full tiles
*   `split`: Requires row-major input to avoid runtime errors
*   `embedding`: More accurate with row-major layout

These constraints are handled by returning `None` (keeping ATen op) when constraints cannot be met, or by inserting layout conversions.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py 847-877](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py#L847-L877)[torch_ttnn/passes/lowering/add_data_move_pass.py 344-384](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py#L344-L384)

## ToTtPass Class Structure

**Title:** ToTtPass Implementation Classes

**ToTtPass** (PassBase subclass):

*   Stores `device` and `use_less_ttnn_op_types` configuration
*   `call(gm)` executes Stage 1 (`ReplaceMoreTt`) then Stage 2 (`ReplaceMoreTtManually`)
*   Returns `PassResult(gm, modified)` where `modified=True` if any ops lowered

**ReplaceMoreTt** (torch.fx.Transformer subclass):

*   Saves original metadata in `_input_node_meta` and `_get_attr_meta`
*   `transform()` traverses graph and calls overridden methods
*   `call_function()` creates `PseudoNode` for guard check, then attempts lowering
*   `call_function_prop_meta()` creates new node with metadata preservation

**GraphWrapper** (helper class):

*   Wraps graph and node for manual lowering stage
*   `call_function()` creates new node with copied metadata and computed output shape
*   `_get_output_val()` uses fake tensor computation for shape inference
