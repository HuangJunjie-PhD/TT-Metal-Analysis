This page documents the execution control subsystem for RISC-V cores in TTExaLens: the operations that halt, step, continue, and reset individual cores, plus related helpers for branch prediction and instruction cache management. This covers the `RiscDebug` abstract interface and its `BabyRiscDebug` implementation family.

For information about the abstract `RiscDebug` class structure and how instances are obtained, see [6.1](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/6.1) For memory and register access operations available after a core is halted, see [6.3](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/6.3) For watchpoint and breakpoint management, see [6.4](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/6.4) For platform-specific behavioral differences, see [6.5](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/6.5)

## Core State Model

A RISC-V core managed by TTExaLens can be in one of three mutually exclusive states at any given time:

**Reset State** — The core is held by a hardware reset signal and does not execute instructions. Debug hardware commands (halt, step, continue, GPR access) are not available while in reset.

**Running State** — The core fetches and executes instructions normally. The halted flag in the debug status register reads `0`.

**Halted State** — The core is paused at a specific instruction. GPR access, memory access through the debug interface, and single-stepping are available. The `STATUS_HALTED` bit in `REG_STATUS` reads `1`.

Sources: [ttexalens/hardware/risc_debug.py 108-145](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L108-L145)[ttexalens/hardware/baby_risc_debug.py 299-349](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L299-L349)

## Reset Control

### `set_reset_signal(value: bool)`

Defined in `RiscDebug` and implemented in `BabyRiscDebug`. Reads and modifies the `RISCV_DEBUG_REG_SOFT_RESET_0` register. Each core has a specific bit position given by `BabyRiscInfo.reset_flag_shift`.

```
reset_reg = read(RISC_DBG_SOFT_RESET0)
reset_reg = (reset_reg & ~(1 << shift)) | (value << shift)
write(RISC_DBG_SOFT_RESET0, reset_reg)
```

Passing `True` asserts reset (stops the core). Passing `False` deasserts reset (allows execution).

Sources: [ttexalens/hardware/baby_risc_debug.py 509-518](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L509-L518)

### `is_in_reset() -> bool`

Reads the same `RISCV_DEBUG_REG_SOFT_RESET_0` register and checks the bit at `reset_flag_shift`. Returns `True` when the core is held in reset.

Sources: [ttexalens/hardware/baby_risc_debug.py 505-507](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L505-L507)

### `assert_not_in_reset(message="")`

Raises `ValueError` if the core is currently in reset. Called internally by `halt()`, `step()`, `cont()`, `is_halted()`, and `is_ebreak_hit()` when `enable_asserts` is `True`.

Sources: [ttexalens/hardware/baby_risc_debug.py 520-528](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L520-L528)

## Halt, Step, and Continue

These three operations form the primary execution control interface. All three are guarded by `assert_not_in_reset` and `assert_debug_hardware` checks, which verify the core is not in reset and that debug hardware is present before issuing any command.

### Hardware Mechanism

`BabyRiscDebugHardware` issues commands by writing to a pair of hardware control registers via NOC:

| Register Name | Offset | Purpose |
| --- | --- | --- |
| `RISC_DBG_CNTL0` | — | Trigger register address and direction |
| `RISC_DBG_CNTL1` | — | Write data for next command |
| `RISC_DBG_STATUS0` | — | Read valid flag |
| `RISC_DBG_STATUS1` | — | Read data from last command |

Command constants written to `REG_COMMAND` (register address `1`):

| Constant | Value | Operation |
| --- | --- | --- |
| `COMMAND_HALT` | `0x00000001` | Stop core |
| `COMMAND_STEP` | `0x00000002` | Execute one instruction |
| `COMMAND_CONTINUE` | `0x00000004` | Resume execution |
| `COMMAND_DEBUG_MODE` | `0x80000000` | Must be OR'd with all debug commands |

All normal debug commands combine with `COMMAND_DEBUG_MODE`: e.g., `COMMAND_DEBUG_MODE | COMMAND_HALT`.

Sources: [ttexalens/hardware/baby_risc_debug.py 49-58](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L49-L58)[ttexalens/hardware/baby_risc_debug.py 293-348](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L293-L348)

### `halt()`

Sends `COMMAND_DEBUG_MODE | COMMAND_HALT`. If the core is already halted, a warning is logged and the method returns immediately. After sending the command, `is_halted()` is checked; if the core did not halt, `RiscHaltError` is raised.

Sources: [ttexalens/hardware/baby_risc_debug.py 299-306](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L299-L306)[ttexalens/hardware/baby_risc_debug.py 629-634](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L629-L634)

### `step()`

Sends `COMMAND_DEBUG_MODE | COMMAND_STEP`. This executes exactly one instruction. The core remains in the halted state afterwards.

**Platform difference — Blackhole:** Due to a hardware bug, `BlackholeBabyRiscDebug.step()` calls `super().step()` twice.

Sources: [ttexalens/hardware/baby_risc_debug.py 337-339](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L337-L339)[ttexalens/hardware/blackhole/baby_risc_debug.py 15-18](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L15-L18)

### `cont()`

Sends `COMMAND_DEBUG_MODE | COMMAND_CONTINUE`. If the core is already running, a warning is logged and the method returns immediately.

**Platform difference — Wormhole:**`WormholeBabyRiscDebug.cont()` disables branch prediction before calling `super().cont()` for functional worker cores (those with a `branch_prediction_register`). For ERISC cores, which have no branch prediction register, it falls back to `continue_without_debug()`, which omits the `COMMAND_DEBUG_MODE` flag. This avoids a hardware-specific lockup issue.

Sources: [ttexalens/hardware/baby_risc_debug.py 341-348](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L341-L348)[ttexalens/hardware/wormhole/baby_risc_debug.py 13-24](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L13-L24)

### `continue_without_debug()`

An alternative continuation path on `BabyRiscDebugHardware` that sends only `COMMAND_CONTINUE` without `COMMAND_DEBUG_MODE`. This can cause core lockup in certain situations and should not be used directly unless specifically required by a platform workaround.

Sources: [ttexalens/hardware/baby_risc_debug.py 350-368](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L350-L368)

## Status Queries

### `is_halted() -> bool`

Reads `REG_STATUS` (register address `0`) via the debug hardware protocol and checks whether `STATUS_HALTED` (`0x1`) is set. Reading status is itself a hardware protocol operation: it calls `__trigger_read(REG_STATUS)` then reads from `RISC_DBG_STATUS1`.

Sources: [ttexalens/hardware/baby_risc_debug.py 375-377](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L375-L377)[ttexalens/hardware/baby_risc_debug.py 615-620](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L615-L620)

### `is_ebreak_hit() -> bool`

Reads `REG_STATUS` and checks `STATUS_EBREAK_HIT` (`0x8`). Returns `True` when the core halted because the CPU executed an `ebreak` instruction, as opposed to a programmatic halt issued via `halt()`.

This distinction matters for callstack unwinding: when `is_ebreak_hit()` returns `True`, the PC register points to the instruction _after_ the `ebreak`, so the callstack logic subtracts 4 from the PC before resolving symbols.

Sources: [ttexalens/hardware/baby_risc_debug.py 379-395](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L379-L395)[ttexalens/hardware/baby_risc_debug.py 622-627](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L622-L627)[ttexalens/hardware/risc_debug.py 486-489](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L486-L489)

### `read_status() -> RiscDebugStatus`

Returns a full `BabyRiscDebugStatus` (a `RiscDebugStatus` subclass) with all status bits decoded:

| Field | Status Bit | Description |
| --- | --- | --- |
| `is_halted` | `STATUS_HALTED = 0x1` | Core is stopped |
| `is_pc_watchpoint_hit` | `STATUS_PC_WATCHPOINT_HIT = 0x2` | PC watchpoint triggered |
| `is_memory_watchpoint_hit` | `STATUS_MEMORY_WATCHPOINT_HIT = 0x4` | Memory watchpoint triggered |
| `is_ebreak_hit` | `STATUS_EBREAK_HIT = 0x8` | `ebreak` instruction was executed |
| `watchpoints_hit` | bits 8–15 | Per-watchpoint hit flags |

Sources: [ttexalens/hardware/baby_risc_debug.py 192-201](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L192-L201)[ttexalens/hardware/baby_risc_debug.py 370-373](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L370-L373)

## `ensure_halted()` Context Manager

Defined on both `BabyRiscDebugHardware` and `BabyRiscDebug`. Guarantees that the body of the `with` block executes while the core is halted, and restores the previous execution state on exit.

```
with risc_debug.ensure_halted():
    # core is guaranteed to be halted here
    value = risc_debug.read_memory(addr)
# core resumes if it was running before
```

If the core was already halted when the context is entered, no halt or continue commands are sent. If the core was running, it is halted on entry and resumed with `cont()` on exit (even if an exception occurs).

This is used extensively inside TTExaLens for watchpoint operations and other operations that require a halted core:

Sources: [ttexalens/hardware/baby_risc_debug.py 323-335](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L323-L335)[ttexalens/hardware/baby_risc_debug.py 650-657](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L650-L657)

## Branch Prediction Control

### `set_branch_prediction(enable: bool)`

Writes to a platform-specific register named in `BabyRiscInfo.branch_prediction_register` using the mask `BabyRiscInfo.branch_prediction_mask`. The value `0` enables branch prediction; `1` disables it (the register is an inversion).

This is needed as a hardware workaround on Wormhole, where branch prediction must be disabled before `step()` and `cont()` to prevent incorrect execution behavior.

The Quasar platform also configures a `branch_prediction_register` per RISC core in `QuasarFunctionalNeoBlock`.

| Platform | Register | Mask (per-core) |
| --- | --- | --- |
| Wormhole (functional) | arch-specific | arch-specific |
| Quasar TRISC0 | `RISC_BRANCH_PREDICTION_CTRL` | `0b0001` |
| Quasar TRISC1 | `RISC_BRANCH_PREDICTION_CTRL` | `0b0010` |
| Quasar TRISC2 | `RISC_BRANCH_PREDICTION_CTRL` | `0b0100` |
| Quasar TRISC3 | `RISC_BRANCH_PREDICTION_CTRL` | `0b1000` |

Sources: [ttexalens/hardware/baby_risc_debug.py 602-613](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L602-L613)[ttexalens/hardware/wormhole/baby_risc_debug.py 26-30](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L26-L30)[ttexalens/hardware/quasar/functional_neo_block.py 86-90](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/functional_neo_block.py#L86-L90)

## Instruction Cache Invalidation

### `invalidate_instruction_cache()`

Writes a bitmask to `RISCV_IC_INVALIDATE_InvalidateAll` with the bit position corresponding to `risc_id`. This forces the instruction cache to discard all cached lines, which is required when code has been written to L1 memory while the core was halted and the new code must be executed on resume.

**Platform difference — Quasar:** After the cache invalidation write, `QuasarBabyRiscDebug.invalidate_instruction_cache()` reads the current PC and calls `debug_hardware.flush(pc)`, which executes `COMMAND_FLUSH` to reset the pipeline to the current PC address. This additional step is required on Quasar hardware to re-activate the instruction cache.

Sources: [ttexalens/hardware/baby_risc_debug.py 530-534](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L530-L534)[ttexalens/hardware/quasar/baby_risc_debug.py 14-27](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L14-L27)[ttexalens/hardware/baby_risc_debug.py 308-321](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L308-L321)

## Platform Behavior Differences Summary

| Operation | Base (`BabyRiscDebug`) | Wormhole | Blackhole | Quasar |
| --- | --- | --- | --- | --- |
| `halt()` | Send `COMMAND_HALT` | Same | Same | Same |
| `step()` | Send `COMMAND_STEP` once | Disable branch pred first | Send `COMMAND_STEP`**twice** | Same as base |
| `cont()` | Send `COMMAND_CONTINUE` | Disable branch pred (functional); `continue_without_debug()` for ERISC | Same as base | Same as base |
| `read_gpr(32)` (PC) | Read via debug hardware | Read from debug bus | Read from debug bus | Read from NEO debug bus |
| `invalidate_instruction_cache()` | Write `RISCV_IC_INVALIDATE_InvalidateAll` | Same | Same | Also calls `flush(pc)` |
