## Purpose and Scope

Fused operation testing validates multi-stage LLK operations where the output of one pipeline stage (Unpack-Math-Pack) becomes the input of the next stage, with data remaining in L1 memory between stages. This enables testing realistic computational patterns such as matmul followed by activation functions, or tilize followed by computation followed by untilize. This document explains the `L1_to_L1_iterations` parameter, intermediate buffer management, format reconfiguration, and semaphore coordination required for multi-stage testing.

For single-stage operation testing (basic Unpack-Math-Pack execution), see [8.6](https://deepwiki.com/tenstorrent/tt-llk/8.6-stimuli-generation-and-data-packing). For test architecture and golden model generation, see [8.4](https://deepwiki.com/tenstorrent/tt-llk/8.4-golden-model-system).

## Overview: Multi-Stage Pipeline Execution

Fused operation testing chains multiple pipeline runs where:

*   **Stage 0**: Input data → Operation A → Intermediate buffer in L1
*   **Stage 1**: Intermediate buffer → Operation B → Output data

Each stage executes a complete Unpack-Math-Pack pipeline, but instead of reading from/writing to host-controlled memory locations, intermediate results remain in L1 memory at predictable addresses for consumption by the next stage.

## L1_to_L1_iterations Parameter

The `L1_to_L1_iterations` parameter in `TestConfig` specifies the number of pipeline stages to execute. This controls:

1.   The number of format configuration arrays (`formats_array[0]`, `formats_array[1]`, etc.)
2.   Golden model computation (accumulated precision loss tracking)
3.   Test validation tolerance adjustment
4.   RuntimeParams structure generation with multiple `FormatConfig` entries

**Table: L1_to_L1_iterations Values**

| Value | Meaning | Use Cases | Example Tests |
| --- | --- | --- | --- |
| `None` (default) | Single-stage operation | Basic operation testing | `test_eltwise_unary_sfpu`, `test_matmul` |
| `2` | Two-stage pipeline | Operation + activation, tilize + compute + untilize | `test_matmul_and_unary_sfpu`, `test_tilize_calculate_untilize_L1` |
| `3+` | Multi-stage pipeline | Complex fused operations | Not currently implemented in tests |

**Python Test Configuration Example**:

`configuration = TestConfig(    "sources/matmul_and_unary_sfpu_test.cpp",    formats,    templates=[        generate_input_dim(input_dimensions, input_dimensions),        MATH_FIDELITY(math_fidelity),        APPROX_MODE(approx_mode),        MATH_OP(mathop=mathop),    ],    runtimes=[TILE_COUNT(tile_cnt_A)],    variant_stimuli=StimuliConfig(...),    dest_acc=dest_acc,    L1_to_L1_iterations=2,  # Two-stage fused operation)`
**C++ Access Pattern in Kernels**:

`#ifdef RUNTIME_FORMATS    const volatile FormatConfig* formats_array = params->formats;#endifint run = 0; // First stage: formats_array[0]// ... stage 0 operations ... run = 1; // Second stage: formats_array[1]// ... stage 1 operations with formats_array[1] ...`
The `run` variable indexes into `formats_array` to select the appropriate format configuration for each pipeline stage. The test framework automatically generates an array of `FormatConfig` structures sized according to `L1_to_L1_iterations`.

## Intermediate Buffer Management

Fused operations require explicit intermediate buffers in L1 memory to store results between stages. The buffer address must be chosen to avoid collision with runtime parameters, coverage data, and other test infrastructure.

**Table: L1 Memory Layout for Fused Operations**

| Address Range | Purpose | Symbol | Placement Strategy |
| --- | --- | --- | --- |
| `0x20000` | Runtime parameters (non-coverage) | `RuntimeParams` | Standard test infrastructure |
| `0x64000` | Runtime parameters (coverage mode) | `RuntimeParams` | Extended for coverage data |
| `0x66000` | Intermediate buffer | `buffer_A_tilized` | Safe for both coverage and non-coverage modes |
| Dynamic | Input/output buffers | `params->buffer_A/B/Res` | Allocated by test framework via `StimuliConfig` |

**C++ Intermediate Buffer Declaration**:

`constexpr std::uint32_t buffer_A_tilized = 0x66000; // L1 address placed so both coverage and non-coverage elfs don't overrun`
**Buffer Usage Pattern**:

*   **Stage 0 Pack**: Writes to `L1_ADDRESS(buffer_A_tilized)`
*   **Stage 1 Unpack**: Reads from `L1_ADDRESS(buffer_A_tilized)`
*   **Final Pack**: Writes to `L1_ADDRESS(params->buffer_Res[0])` for host readback

The intermediate buffer size must accommodate the full tile data (1024 elements × format datum size). For a 32×32 tile in Float16_b format, this requires 2048 bytes (1024 elements × 2 bytes/element).

**Sources**: [tests/sources/matmul_and_unary_sfpu_test.cpp 21](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L21-L21)[tests/sources/matmul_and_unary_sfpu_test.cpp 138](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L138-L138)[tests/sources/matmul_and_unary_sfpu_test.cpp 57](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L57-L57)

## Format Reconfiguration Between Stages

Each pipeline stage can use different data formats. Format reconfiguration functions must be called before processing data with new formats to update hardware configuration registers.

**Diagram: Format Reconfiguration Flow**

**Reconfiguration Function Summary**:

| Pipeline Component | Reconfiguration Function | Parameters | Purpose |
| --- | --- | --- | --- |
| Unpacker | `_llk_unpack_reconfig_data_format_srca_impl_` | `is_fp32_dest_acc_en`, `transpose`, `new_src_format`, `new_dst_format`, `tile_size` | Update unpacker format registers |
| Math | `_llk_math_reconfig_data_format_srca_` | `is_fp32_dest_acc_en`, `transpose`, `new_format` | Update ALU format registers |
| Packer | `_llk_pack_reconfig_data_format_` | `is_fp32_dest_acc_en`, `new_src_format`, `new_dst_format`, `tile_size` | Update packer format registers |

## Semaphore Coordination Between Stages

Semaphores synchronize the completion of stage N packer before stage N+1 unpacker begins, preventing race conditions where unpacker reads incomplete data.

**Diagram: Semaphore-Based Stage Synchronization**

**Semaphore Operations**:

| Function | Behavior | Stage Usage |
| --- | --- | --- |
| `t6_semaphore_post<>(semaphore::PACK_DONE)` | Increment semaphore | Called by stage N packer after writing intermediate buffer |
| `t6_semaphore_wait_on_zero<p_stall::STALL_SYNC>(semaphore::PACK_DONE)` | Wait until semaphore reaches 0 | Called by stage N+1 unpacker before reading intermediate buffer |
| `t6_semaphore_get<>(semaphore::PACK_DONE)` | Decrement semaphore | Called by stage N+1 unpacker after wait completes |

**C++ Synchronization Example**:

`// Stage 0 Pack completest6_semaphore_post<>(semaphore::PACK_DONE); // Stage 1 Unpack waitst6_semaphore_wait_on_zero<p_stall::STALL_SYNC>(semaphore::PACK_DONE);t6_semaphore_get<>(semaphore::PACK_DONE);`
**Sources**: [tests/sources/matmul_and_unary_sfpu_test.cpp 45-46](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L45-L46)[tests/sources/matmul_and_unary_sfpu_test.cpp 132](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L132-L132)

## Test Implementation: Matmul + Unary SFPU

The `matmul_and_unary_sfpu_test.cpp` demonstrates a complete two-stage fused operation: matrix multiplication followed by SFPU activation function.

**Diagram: Matmul + SFPU Kernel Structure**

**Stage Execution Details**:

**Stage 0 - Matrix Multiplication**:

*   **Unpack**: `_llk_unpack_AB_matmul_<>()` reads `buffer_A[0]` and `buffer_B[0]` into source registers
*   **Math**: `_llk_math_matmul_<MATH_FIDELITY>(0)` performs matrix multiplication with fidelity masking
*   **Pack**: `_llk_pack_<DstSync::SyncHalf, is_fp32_dest_acc_en, false>(0, L1_ADDRESS(buffer_A_tilized))` writes result to intermediate buffer

**Stage 1 - SFPU Activation**:

*   **Unpack**: `_llk_unpack_A_<>()` reads from `buffer_A_tilized` (matmul output) into source registers
*   **Math**: Two operations: 
    1.   `_llk_math_eltwise_unary_datacopy_<DataCopyType::A2D>()` copies source to destination
    2.   `test_utils::call_sfpu_operation<>()` applies SFPU operation (exp, relu, gelu, etc.) to destination

*   **Pack**: `_llk_pack_<>(0, L1_ADDRESS(params->buffer_Res[0]))` writes final result to output buffer

**Sources**: [tests/sources/matmul_and_unary_sfpu_test.cpp 23-55](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L23-L55)[tests/sources/matmul_and_unary_sfpu_test.cpp 59-106](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L59-L106)[tests/sources/matmul_and_unary_sfpu_test.cpp 109-151](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sources/matmul_and_unary_sfpu_test.cpp#L109-L151)

## Golden Model Computation

Golden models for fused operations must match the hardware execution order, applying operations sequentially and accumulating format-specific precision loss at each stage. Each stage may apply different format conversions, quantization, and approximations that compound across the pipeline.

**Python Golden Computation Example (Matmul + SFPU)**:

`# Stage 0: Matmul with fidelity maskinggenerate_matmul_golden = get_golden_generator(MatmulGolden)golden_tensor = generate_matmul_golden(    src_A,    src_B,    formats.output_format,    math_fidelity,  # LoFi, HiFi2, HiFi3, or HiFi4    input_A_dimensions=input_dimensions,    input_B_dimensions=input_dimensions,)# Tilize output to match hardware formatgolden_tensor = tilize(golden_tensor, formats.output_format) # Stage 1: SFPU operation on matmul resultgenerate_sfpu_golden = get_golden_generator(UnarySFPUGolden)golden_tensor = generate_sfpu_golden(    mathop,    golden_tensor,  # Input is previous stage output    formats.output_format,    dest_acc,    formats.input_format,    input_dimensions,)golden_tensor = golden_tensor.to(torch_format)`
**Tolerance Adjustment**: The `passed_test()` function accepts an `L1_to_L1_iterations` parameter to scale tolerance for accumulated precision loss. Each stage introduces quantization errors that accumulate multiplicatively:

`assert passed_test(    golden_tensor,     res_tensor,     formats.output_format,     L1_to_L1_iterations=2  # Relaxed tolerance: base_tol × iteration_factor), "Assert against golden failed"`
**Precision Loss Accumulation Factors**:

*   **Single operation** (`L1_to_L1_iterations=None`): Standard format-specific tolerance
*   **Two-stage operation** (`L1_to_L1_iterations=2`): Tolerance multiplied by iteration-dependent factor
*   **Multi-stage operation**: Tolerance scales with number of format conversions and quantization steps

## Implementation Checklist

When implementing a new fused operation test:

**Required Components**:

*   ☐ Set `L1_to_L1_iterations` parameter in `TestConfig`
*   ☐ Declare intermediate buffer addresses (`buffer_A_tilized`, etc.) avoiding collision with 0x20000-0x66000 range
*   ☐ Define `formats_array[]` with format configurations for each stage
*   ☐ Implement semaphore coordination (`t6_semaphore_post`, `t6_semaphore_wait_on_zero`, `t6_semaphore_get`)
*   ☐ Call format reconfiguration functions between stages
*   ☐ Implement golden model matching hardware execution order
*   ☐ Adjust validation tolerance using `L1_to_L1_iterations` parameter

**Common Pitfalls**:

*   **Buffer collision**: Intermediate buffer overlaps with runtime params or coverage data
*   **Missing reconfiguration**: Format changes without calling `_llk_*_reconfig_data_format_*` functions
*   **Race condition**: Unpacker reads intermediate buffer before packer completes writing
*   **Format mismatch**: `formats_array[run]` index doesn't match current stage
*   **Tolerance too strict**: Golden model comparison fails due to accumulated precision loss
