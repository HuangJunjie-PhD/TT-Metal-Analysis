## Purpose and Scope

The testing infrastructure for `tt-umd` provides a comprehensive suite of validation tools designed to ensure hardware abstraction reliability across diverse Tenstorrent architectures (Wormhole, Blackhole) and system topologies (N150, N300, Galaxy). It covers low-level PCIe/JTAG communication, memory management (TLBs), architecture-specific features (harvesting, ARC telemetry), and multi-process safety.

The infrastructure is built primarily on **Google Test (gtest)** for C++ unit and integration tests, complemented by a suite of **UMD Tools** for system-level diagnostics and **Microbenchmarks** for performance regression tracking.

## API and Cluster Testing

The API tests validate the `Cluster` and `TTDevice` abstractions. These tests ensure that the UMD can discover chips, parse SoC descriptors, and perform basic I/O regardless of the underlying board type.

### Key API Test Patterns

*   **Device Discovery**: `OpenAllSiliconChips` verifies that `make_cluster_for_test()` can successfully initialize all detected hardware [tests/api/test_cluster.cpp 45](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/api/test_cluster.cpp#L45-L45)
*   **SoC Descriptor Integrity**: `PrintAllSiliconChipsAllCores` iterates through every chip and core type (TENSIX, DRAM, ETH) to verify coordinate mapping [tests/api/test_cluster.cpp 47-85](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/api/test_cluster.cpp#L47-L85)
*   **Clock Control**: `TestClusterAICLKControl` validates that the UMD can transition devices between `DevicePowerState::BUSY` and `DevicePowerState::LONG_IDLE`, checking that AICLK values match architectural expectations [tests/api/test_cluster.cpp 87-115](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/api/test_cluster.cpp#L87-L115)
*   **Thread Safety**: `TTDeviceMultipleThreadsIO` launches parallel threads performing simultaneous reads/writes to different L1 addresses on the same `TTDevice` to ensure driver-level reentrancy [tests/api/test_tt_device.cpp 112-164](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/api/test_tt_device.cpp#L112-L164)

## Multi-Process and Multi-Threaded Safety

A critical component of the infrastructure is validating that multiple processes or threads can share Tenstorrent hardware without corruption. The `unified_tests` suite focuses on these concurrency patterns.

### Parallel I/O Implementation

The function `test_read_write_all_tensix_cores_impl` partitions L1 memory into chunks based on `NUM_PARALLEL`[tests/unified/multiprocess.cpp 58-65](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/unified/multiprocess.cpp#L58-L65) Each thread or process is assigned a specific memory bucket to avoid data races while stressing the UMD's internal locking and TLB management [tests/unified/multiprocess.cpp 72-95](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/unified/multiprocess.cpp#L72-L95)

### Multi-Process Scenarios

*   **MultipleClusters**: Sequential creation and I/O of multiple `Cluster` instances in a single process [tests/unified/multiprocess.cpp 115-126](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/unified/multiprocess.cpp#L115-L126)
*   **MultipleThreadsSingleCluster**: Many threads sharing a single `Cluster` instance for concurrent I/O [tests/unified/multiprocess.cpp 129-143](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/unified/multiprocess.cpp#L129-L143)
*   **Inter-Process Locking**: (Implemented via `lock_virus` tool) tests the driver's ability to handle resource contention across OS process boundaries.

## CI/CD Testing Matrix

The testing infrastructure is integrated into GitHub Actions via a dynamic matrix that scales based on the available hardware runners.

### Test Card Definition

The workflow `build-and-run-all-tests.yml` defines a list of "cards" representing specific hardware configurations, including architecture and virtualization state (e.g., `n150-stable`, `p150b-viommu-stable`, `wh-glx-6u`) [.github/workflows/build-and-run-all-tests.yml 70-88](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/.github/workflows/build-and-run-all-tests.yml#L70-L88)

### Blacklist Logic

To optimize CI runtime, a blacklist filters the matrix based on the triggering event (e.g., skipping slow `viommu` runs on PRs but running them on `main`) [.github/workflows/build-and-run-all-tests.yml 116-126](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/.github/workflows/build-and-run-all-tests.yml#L116-L126)

## Microbenchmarks

Performance regressions are tracked using the `umd_microbenchmark` suite. These tests measure the overhead of UMD abstractions compared to raw hardware capabilities.

*   **PCIe DMA**: Measures throughput for large transfers between host and DRAM [tests/microbenchmark/benchmarks/pcie_dma/test_pcie_dma.cpp](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/microbenchmark/benchmarks/pcie_dma/test_pcie_dma.cpp)
*   **TLB Mapping Latency**: Benchmarks the time taken by `TLBManager` to reconfigure a dynamic TLB window [tests/microbenchmark/benchmarks/tlb/test_tlb.cpp](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/microbenchmark/benchmarks/tlb/test_tlb.cpp)
*   **Ethernet I/O**: Characterizes the latency of remote chip access over the Ethernet fabric [tests/microbenchmark/benchmarks/ethernet_io/test_ethernet_io.cpp](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/microbenchmark/benchmarks/ethernet_io/test_ethernet_io.cpp)
