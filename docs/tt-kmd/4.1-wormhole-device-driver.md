## Purpose and Scope

This document describes the Wormhole-specific device driver implementation, which provides hardware support for Tenstorrent's Wormhole generation accelerators (PCI Device ID: 0x401E). The Wormhole driver implements the generic `tenstorrent_device_class` interface defined in [device.h 67-93](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L67-L93) and provides hardware-specific functionality including 32-bit DMA addressing, three-tier TLB configuration, ARC firmware communication, and telemetry collection.

For the generic device abstraction layer, see [Device Abstraction Layer](https://deepwiki.com/tenstorrent/tt-kmd/3.2-device-abstraction-layer). For memory management concepts including TLB and IATU operation, see [Memory Management](https://deepwiki.com/tenstorrent/tt-kmd/6-memory-management). For ARC firmware protocol details, see [ARC Firmware Communication](https://deepwiki.com/tenstorrent/tt-kmd/4.3-arc-firmware-communication).

## Hardware Architecture Overview

The Wormhole device exposes three key PCI Base Address Registers (BARs) for memory-mapped I/O:

| BAR | Purpose | Size | Access Method |
| --- | --- | --- | --- |
| BAR0 | Device memory (TLB windows) | ~350MB | Direct mapping + TLB translation |
| BAR2 | PCIe configuration and IATU registers | Variable | `bar2_mapping` |
| BAR4 | System registers (via iATU) | 32MB | `bar4_mapping` |

**BAR0 TLB Window Layout:**[wormhole.c 20-33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L20-L33) defines the window offsets:

*   **Offset 0x00000000**: 156 × 1MB TLB windows (0-155)
*   **Offset 0x09C00000**: 10 × 2MB TLB windows (156-165)
*   **Offset 0x0AC00000**: 20 × 16MB TLB windows (166-185)

**BAR4 System Register Mapping** (via inbound iATU region 1): [wormhole.c 74-89](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L74-L89) defines the BAR4 layout targeting the SoC address space `0x1E000000`:

*   **Reset Unit**: `RESET_UNIT_START` (scratch registers, ARC control) [wormhole.c 78](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L78-L78)
*   **ARC CSM**: `ARC_CSM_START` (firmware memory) [wormhole.c 79](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L79-L79)
*   **TLB Registers**: `TLB_REGS_START` (186 TLB configuration registers) [wormhole.c 80](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L80-L80)
*   **NOC2AXI**: `NOC2AXI_START` (PCIe performance counters) [wormhole.c 81](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L81-L81)
*   **Kernel TLB**: `KERNEL_TLB_START` (last 16MB window reserved) [wormhole.c 88](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L88-L88)

## TLB Configuration System

Wormhole provides three types of Translation Lookaside Buffers (TLBs) for mapping host addresses to Network-on-Chip (NOC) coordinates.

### TLB Types and Counts

[wormhole.c 20-37](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L20-L37) defines the hierarchy:

| TLB Type | Count | Window Size | BAR0 Offset Range | Use Case |
| --- | --- | --- | --- | --- |
| 1MB TLBs | 156 | 1MB | 0x00000000-0x09C00000 | Small granular mappings |
| 2MB TLBs | 10 | 2MB | 0x09C00000-0x0AC00000 | Medium-sized regions |
| 16MB TLBs | 20 | 16MB | 0x0AC00000-0x14C00000 | Large contiguous areas |

**Total:** 186 TLBs (index 0-185), with index 185 (`KERNEL_TLB_INDEX`) reserved for kernel use [wormhole.c 87](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L87-L87)

### Kernel TLB for NOC Access

The driver uses the last 16MB TLB to perform internal NOC reads/writes. This is managed via `kernel_tlb_mutex`[wormhole.h 12](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.h#L12-L12)

**Sources:**[wormhole.c 86-90](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L86-L90)[wormhole.h 12](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.h#L12-L12)

## ARC Firmware Communication Protocol

Wormhole uses an on-chip ARC processor for firmware tasks. The driver communicates with ARC firmware through scratch registers in the Reset Unit.

### Message Protocol

[wormhole.c 108-112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L108-L112) defines the protocol where `WH_FW_MESSAGE_PRESENT` (0xAA00) is written to Scratch Register 5.

**Scratch Register Usage:**

*   **SR0**: `POST_CODE_REG` - Checks if ARC L2 is running (`0xC0DE0000`) [wormhole.c 101-103](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L101-L103)
*   **SR3**: Message arguments (arg0 and arg1) [wormhole.c 208](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L208-L208)
*   **SR5**: Message ID and handshake [wormhole.c 209](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L209-L209)
*   **SR6**: `PCIE_ARMISC_INFO_REG` - Hardware health check [wormhole.c 98](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L98-L98)

### Firmware Messaging Logic

The function `wormhole_send_arc_fw_message_with_args`[wormhole.c 205-212](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L205-L212) handles the transaction:

1.   Verify `arc_l2_is_running` via `POST_CODE_REG`[wormhole.c 199-203](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L199-L203)
2.   Write arguments to `SCRATCH_REG(3)`.
3.   Write `0xAA00 | message_id` to `SCRATCH_REG(5)`.
4.   Trigger an interrupt to ARC via `ARC_MISC_CNTL_IRQ0_MASK`[wormhole.c 106](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L106-L106)
5.   Poll for completion using `arc_msg_poll_completion`[wormhole.c 163-197](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L163-L197)

### Key ARC Messages

[wormhole.c 39-43](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L39-L43) and [pcie.c 16](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L16-L16) define IDs:

*   `FW_MSG_PCIE_RETRAIN` (0xB6): Triggers PCIe link retraining [pcie.c 112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L112-L112)
*   `WH_FW_MSG_TRIGGER_RESET` (0x56): Initiates a device-side reset [wormhole.c 42](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L42-L42)
*   `WH_FW_MSG_ASTATE0` (0xA0): Transitions device to active power state [wormhole.c 40](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L40-L40)

**Sources:**[wormhole.c 90-122](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L90-L122)[pcie.c 16-23](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L16-L23)

## PCIe Management and Initialization

### Link Retraining

The function `wormhole_complete_pcie_init`[pcie.c 92-131](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L92-L131) ensures the PCIe link reaches the target speed.

1.   It reads the `target_link_speed` from the upstream bridge [pcie.c 107](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L107-L107)
2.   Sends `FW_MSG_PCIE_RETRAIN` to ARC firmware [pcie.c 112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L112-L112)
3.   If retraining fails, it performs a `pcie_hot_reset_and_restore_state`[pcie.c 126](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L126-L126)

### Hot Reset Procedure

`pcie_hot_reset_and_restore_state`[pcie.c 61-90](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L61-L90) performs a secondary bus reset:

1.   Sets `PCI_BRIDGE_CTL_BUS_RESET` on the upstream bridge [pcie.c 74](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L74-L74)
2.   Sleeps for 2ms, then clears the reset bit [pcie.c 77](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L77-L77)
3.   Polls for the link to come back up using `poll_pcie_link_up`[pcie.c 80](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L80-L80)
4.   Restores PCI configuration state via `safe_pci_restore_state`[pcie.c 80](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L80-L80)

**Sources:**[pcie.c 24-131](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/pcie.c#L24-L131)

## IATU and DBI Access

The Wormhole driver uses the internal DesignWare iATU (Internal Address Translation Unit) to map BARs to SoC addresses.

### IATU Configuration

[wormhole.c 46-61](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L46-L61) defines the iATU register offsets relative to BAR2.

*   `IATU_LOWER_BASE_ADDR_OUTBOUND`: 0x08
*   `IATU_LIMIT_ADDR_OUTBOUND`: 0x10
*   `IATU_LOWER_TARGET_ADDR_OUTBOUND`: 0x14

### DBI Access

DBI (Data Bridge Interface) allows the driver to modify internal PCIe controller registers.

*   `PCIE_DBI_ADDR`: `0x800000000ULL`[wormhole.c 92](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L92-L92)
*   `DBI_ENABLE`: `0x00200000`[wormhole.c 97](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L97-L97)

**Sources:**[wormhole.c 45-72](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L45-L72)[wormhole.c 92-100](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L92-L100)

## Telemetry and Hardware Monitoring

Wormhole devices expose telemetry through the Reset Unit registers.

*   `ARC_TELEMETRY_PTR`: Points to the telemetry structure in CSM [wormhole.c 83](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L83-L83)
*   `ARC_TELEMETRY_DATA`: Contains the actual telemetry values [wormhole.c 84](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L84-L84)

The driver uses `delayed_work fw_ready_work`[wormhole.h 19](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.h#L19-L19) to wait for firmware to populate these structures before registering `hwmon` or `sysfs` groups.

**Sources:**[wormhole.c 83-85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L83-L85)[wormhole.h 19-24](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.h#L19-L24)
