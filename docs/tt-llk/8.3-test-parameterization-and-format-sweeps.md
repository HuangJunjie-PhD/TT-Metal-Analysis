This page documents the test parameterization system that enables comprehensive testing of LLK operations across different data format combinations, hardware configurations, and test variants. The system provides a declarative way to specify test parameters with automatic dependency resolution and generates thousands of test variants from compact test definitions.

For information about test execution and orchestration, see [TestConfig and Test Orchestration](https://deepwiki.com/tenstorrent/tt-llk/8.2-testconfig-and-test-orchestration). For details on the testing framework architecture, see [Test Architecture Overview](https://deepwiki.com/tenstorrent/tt-llk/8.1-test-architecture-overview).

## Purpose and Scope

The parameterization system solves the challenge of testing hardware operations that support multiple data formats (Float16, Float32, Bfp8_b, etc.) through different pipeline stages (unpack → math → pack). Testing all valid combinations manually would require thousands of individual test functions. Instead, this system:

*   Generates test variants from format lists and constraint functions
*   Resolves parameter dependencies automatically (e.g., destination accumulation mode depends on data formats)
*   Filters invalid combinations through declarative constraints
*   Creates readable test identifiers for debugging
*   Integrates seamlessly with pytest's parameterization infrastructure

## Dependency Resolution System

### Constraint Functions

Parameters can be callable functions that compute valid values based on previously resolved parameters. The system automatically determines resolution order using topological sorting.

**Diagram: Dependency Resolution Chain in test_eltwise_unary_datacopy**

**Sources:**[tests/python_tests/test_eltwise_unary_datacopy.py 80-96](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/test_eltwise_unary_datacopy.py#L80-L96)[tests/python_tests/helpers/param_config.py 235-289](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L235-L289)

### Dependency Extraction and Validation

The system uses Python introspection to extract parameter dependencies from callable signatures:

| Function | Purpose | Error Handling |
| --- | --- | --- |
| `_param_dependencies()` | Extract parameter names from callable using `inspect.signature()` | Returns empty list for non-callables |
| `_compute_dependency_map()` | Build dict mapping parameters to their dependencies | Calls `_verify_dependency_map()` |
| `_verify_dependency_map()` | Check all dependencies exist as known parameters | Raises `UnknownDependenciesError` if missing |
| `_compute_resolution_order()` | Topological sort to determine evaluation order | Raises `CircularDependencyError` if cycle detected |
| `_params_solve_dependencies()` | Recursive backtracking to generate all valid combinations | Raises `ResolutionError` if constraint function fails |

## Format Sweep Functions

### format_combination_sweep()

Generates all possible format combinations for the unpack-math-pack pipeline. Each `FormatConfig` specifies formats at five stages:

**Diagram: Format Configuration Pipeline Stages**

**Format Sweep Modes:**

| Mode | `all_same` Parameter | Generated Combinations | Use Case |
| --- | --- | --- | --- |
| Same-format sweep | `True` | All stages use same format | Smoke testing, basic validation |
| Full format flush | `False` | All possible stage combinations | Comprehensive format conversion testing |

Example for 3 formats with `all_same=True`:

*   Returns 3 FormatConfigs (Float16 everywhere, Float32 everywhere, Bfp8_b everywhere)

Example for 3 formats with `all_same=False`:

*   Returns 3^5 = 243 FormatConfigs (every combination of 5 stages × 3 formats)

### get_num_blocks_and_num_tiles_in_block()

Calculates how to partition input data into blocks that fit in destination register:

**Parameters:**

| Parameter | Type | Purpose |
| --- | --- | --- |
| `dest_sync` | `DestSync` | Half (8 tiles base) or Full (16 tiles base) |
| `dest_acc` | `DestAccumulation` | Whether using 32-bit accumulation (halves capacity) |
| `formats` | `InputOutputFormat` | Input/output formats (affects capacity for outliers) |
| `input_dimensions` | `List[int]` | Matrix dimensions `[rows, cols]` |
| `tile_dimensions` | `List[int]` | Tile size (default `[32, 32]`) |
| `blocks_algorithm` | `BlocksCalculationAlgorithm` | Standard, Tilize, or Untilize |

**Returns:**`(num_blocks, num_tiles_in_block)`

**Calculation Logic:**

1.   Calculate max tiles in dest: `base_capacity / divisor`
2.   Calculate total tiles in input: `(rows/32) * (cols/32)`
3.   For untilize: use only one tile-row for calculation
4.   Validate: total tiles must be evenly divisible by max tiles
5.   Return: `(ceil(total / max), ceil(total / num_blocks))`

**Sources:**[tests/python_tests/helpers/param_config.py 467-554](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L467-L554)

## Test ID Generation

The parameterization system automatically generates readable test identifiers:

**Example Generated IDs:**

```
test_reduce[input_dimensions:[32, 32]-formats:Float16_b->Float16_b-is_reduce_to_one:False-math_fidelity:HiFi2-reduce_dim:Row-pool_type:Max]
test_reduce[input_dimensions:[64, 64]-formats:Float32->Float32-is_reduce_to_one:True-math_fidelity:HiFi3-reduce_dim:Column-pool_type:Average]
```

These identifiers appear in pytest output and CI logs, making it easy to identify which specific combination failed.

## Integration with pytest

The `@parametrize` decorator ultimately delegates to pytest's built-in parameterization:

This ensures compatibility with pytest features like:

*   Test selection: `pytest -k "Float32"`
*   Parallel execution: `pytest -n auto`
*   Result reporting and JUnit XML generation
*   Coverage collection per test variant
