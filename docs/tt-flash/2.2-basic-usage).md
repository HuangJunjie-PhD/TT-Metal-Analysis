# Basic Usage

*   [README.md](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1)

## Purpose and Scope

This page demonstrates the common usage patterns for tt-flash: executing firmware flashing operations and verifying flash contents. It covers the two primary commands (`flash` and `verify`), their typical invocations, command-line options, and expected output.

For detailed command reference including all options and flags, see [Command Reference](https://deepwiki.com/tenstorrent/tt-flash/3-command-reference). For installation instructions, see [Installation](https://deepwiki.com/tenstorrent/tt-flash/2.1-installation). For in-depth explanations of the flashing process internals, see [Flashing Operations](https://deepwiki.com/tenstorrent/tt-flash/5-flashing-operations).

* * *

## Quick Reference

The two primary usage patterns are:

| Operation | Command Pattern | Purpose |
| --- | --- | --- |
| **Flash firmware** | `tt-flash flash <bundle.fwbundle>` | Update firmware on all detected Tenstorrent devices |
| **Verify firmware** | `tt-flash verify <bundle.fwbundle>` | Check SPI flash contents against a firmware bundle |

The simplified form `tt-flash <bundle.fwbundle>` defaults to the `flash` command for backward compatibility [tt_flash/main.py 139-149](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L139-L149)

### Execution Stages

The flash command executes in four distinct stages, each printed to the console:

| Stage | Purpose | Implementation |
| --- | --- | --- |
| **SETUP** | Load firmware bundle, extract manifest, search for system configuration | [tt_flash/main.py 213-219](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L213-L219) |
| **DETECT** | Enumerate Tenstorrent devices via PCI | [tt_flash/main.py 221-222](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L221-L222) |
| **FLASH** | Write firmware to SPI flash with verification | [tt_flash/main.py 224-234](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L224-L234) |
| **RESET** | Reset devices to load new firmware | (Handled in `flash.py`) |

The `load_manifest()` function [tt_flash/main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L201) extracts and validates the `manifest.json` from the tarfile, ensuring the version string is a valid three-component semantic version.

## Common Command Options

### Flash Command Options

| Option | Type | Default | Purpose |
| --- | --- | --- | --- |
| `fwbundle` | positional | required | Path to firmware bundle file (`.fwbundle` tarfile) |
| `--force` | flag | `False` | Force firmware update even if version is same/older |
| `--no-reset` | flag | `False` | Skip device reset after flashing |
| `--skip-missing-fw` | flag | `False` | Continue if bundle lacks firmware for detected board |
| `--allow-major-downgrades` | flag | `False` | Allow downgrading across major version boundaries |
| `--fw-tar` | path | `None` | Deprecated alias for positional `fwbundle` argument |

The deprecated `--fw-tar` option [tt_flash/main.py 91](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L91-L91) triggers a warning message [tt_flash/main.py 174-175](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L174-L175) but remains functional for backward compatibility.

**Sources**: [tt_flash/main.py 84-130](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L84-L130)[README.md 89-104](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1#L89-L104)

### Output Interpretation

| Output Element | Meaning | Related Code |
| --- | --- | --- |
| `Stage: SETUP/DETECT/FLASH` | Major execution phases | [tt_flash/main.py 213-224](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L213-L224) |
| `ROM version is: (18, 10, 0, 0)` | Current firmware version on device | Retrieved via chip-specific methods |
| `tt-flash version is: (18, 12, 0, 0)` | Version in the firmware bundle | Extracted from [manifest.json](https://deepwiki.com/tenstorrent/tt-flash/5.1-firmware-bundle-format) |
| `Writing new firmware... SUCCESS` | SPI write operation completed | Implemented in `flash.py` |
| `Firmware verification... SUCCESS` | Read-back verification passed | Write-verify cycle in `flash.py` |
| `FLASH SUCCESS` | All operations completed | Final status from `flash_chips()` |

## Verify Command Usage

### Basic Invocation

The verify command checks SPI flash contents against the provided firmware bundle without modifying the flash. It displays:

*   Currently running firmware version (from ARC processor)
*   Flashed firmware version (from SPI flash memory)
*   Checksum verification results

**Note**: The current implementation [tt_flash/main.py 235-236](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L235-L236) shows that the verify command handler is not yet implemented in the main execution flow, though the argument parser defines the command [tt_flash/main.py 109-127](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L109-L127)

### Verify Command Options

| Option | Type | Default | Purpose |
| --- | --- | --- | --- |
| `fwbundle` | positional | optional | Path to firmware bundle for checksum comparison |
| `--fw-tar` | path | `None` | Deprecated alias for positional `fwbundle` argument |
| `--skip-missing-fw` | flag | `False` | Continue if bundle lacks firmware for detected board |

## Global Options

The following options apply to all commands and must be specified before the subcommand:

| Option | Purpose | Default |
| --- | --- | --- |
| `-v, --version` | Display tt-flash version and exit | N/A |
| `-h, --help` | Display help text and exit | N/A |
| `--no-color` | Disable colored output | Color enabled |
| `--no-tty` | Force disable TTY command output | TTY auto-detect |

These are parsed by the top-level parser [tt_flash/main.py 63-80](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L63-L80) and stored in `CConfig`[tt_flash/main.py 207-208](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L207-L208) for use throughout the application.

**Version Display**: The version string is loaded from `.ignored/version.txt` if available in development builds [tt_flash/main.py 25-32](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L25-L32) otherwise falls back to `tt_flash.__version__` from package metadata.

**Sources**: [tt_flash/main.py 60-80](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L60-L80)[README.md 64-79](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1#L64-L79)

The argument parser implements a fallback mechanism for commands without an explicit subcommand:

This allows `tt-flash bundle.fwbundle` to work as shorthand for `tt-flash flash bundle.fwbundle`. The implementation uses a custom `NoExitArgumentParser` class [tt_flash/main.py 49-57](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L49-L57) that raises `ArgumentParseError` instead of calling `sys.exit()` during the initial parse attempts.

## Error Handling

Errors during execution are caught at the top level and displayed with color-coded output:

Common error scenarios include:

*   **Bundle not found**: Invalid path to firmware bundle [tt_flash/main.py 215-219](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L215-L219)
*   **Invalid manifest**: Corrupt or missing manifest.json in bundle [tt_flash/main.py 184-199](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L184-L199)
*   **No devices detected**: No Tenstorrent devices found on PCI bus
*   **Firmware mismatch**: Bundle lacks firmware for detected board type (unless `--skip-missing-fw` is used)
*   **Write/verify failure**: SPI flash operation or verification failed

**Sources**: [tt_flash/main.py 211-239](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L211-L239)
