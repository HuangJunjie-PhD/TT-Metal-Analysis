## Purpose and Scope

This document provides a detailed technical explanation of how tt-npe simulates Network-on-Chip traffic across multi-chip fabrics such as T3K and N150 configurations. It covers the topology representation, path-finding algorithms, route elaboration mechanisms, and how cross-device transfers are modeled during simulation.

For general device model information, see [Device Models](https://deepwiki.com/tenstorrent/tt-npe/5-device-models). For information about the workload data structure that holds these multi-chip transfers, see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For details on the preprocessing step that converts raw traces into multi-chip workloads, see [Trace Preprocessing and Fabric Routing](https://deepwiki.com/tenstorrent/tt-npe/2.2-trace-preprocessing-and-fabric-routing).

## Topology Representation

Multi-chip simulations require a precise description of how devices are physically connected in the fabric. This is captured in `topology.json` files.

### Topology JSON Structure

A topology file contains three main components:

| Component | Purpose |
| --- | --- |
| `forwarding_channel_pairs` | Defines which ethernet channel pairs on each device are used for forwarding traffic (not terminating locally) |
| `inter_device_channel_pairs` | Specifies the physical connections between devices via ethernet channels |
| `eth_chan_to_coord` | Maps ethernet channel numbers to physical (row, col) coordinates on the device grid |

**Sources:**[tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json 1-91](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json#L1-L91)[tt_npe/workload/T3K-fabric-traces/line-4/topology.json 1-91](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/line-4/topology.json#L1-L91)

### Forwarding Channel Pairs

The `forwarding_channel_pairs` array specifies which ethernet cores on a device are configured to forward packets between their two associated channels:

`{    "device_id": 0,    "channels": [1, 8]}`
This indicates that device 0 will forward traffic arriving on channel 1 to channel 8, and vice versa. This is essential for devices in the middle of a path that need to relay data.

**Sources:**[tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json 3-27](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json#L3-L27)

### Inter-device Channel Pairs

The `inter_device_channel_pairs` array defines the physical ethernet links connecting devices:

`{    "local_device_id": 0,    "local_eth_channel": 1,    "remote_device_id": 3,    "remote_eth_channel": 1}`
This represents a bidirectional link where device 0's channel 1 is connected to device 3's channel 1.

**Sources:**[tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json 28-71](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json#L28-L71)

### Ethernet Channel to Coordinate Mapping

The `eth_chan_to_coord` object maps ethernet channel numbers to their physical grid coordinates:

`{    "0": [9,0],    "1": [1,0],    ...}`
This mapping is critical for determining which cores on the device grid are ethernet cores and what their locations are for routing purposes.

**Sources:**[tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json 73-90](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/line-8dev/topology.json#L73-L90)

## Fabric Topology Graph

### TopologyGraph Class

The system constructs a graph representation of the fabric topology using the `fabric_post_process.TopologyGraph` class. This graph enables path finding between any two cores across multiple devices.

**Sources:**[tt_npe/scripts/generate_multichip_data.py 248](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L248-L248)

### Path Finding Algorithm

The `find_optimal_path()` method computes a sequence of waypoints that describe how to route data from a source coordinate on one device to a destination coordinate on another device.

**Algorithm Overview:**

1.   Determine if source and destination are on the same device

    *   If yes, return single waypoint (intra-device routing)
    *   If no, proceed to multi-hop routing

2.   For multi-device routing:

    *   Find path through topology graph from source device to destination device
    *   For each device in the path, determine the entry and exit ethernet cores
    *   Generate waypoints representing segments on each device

3.   Each waypoint contains:

    *   `device`: The device ID for this segment
    *   `segment_start_x`, `segment_start_y`: Entry point on this device
    *   `segment_end_x`, `segment_end_y`: Exit point on this device

**Sources:**[tt_npe/scripts/generate_multichip_data.py 279](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L279-L279)

## Route Elaboration

### From Simple Transfers to Multi-segment Routes

When a transfer crosses device boundaries, it must be broken down into multiple route segments, one per device traversed. Each segment describes the path through a single device's NoC.

**Sources:**[tt_npe/scripts/generate_multichip_data.py 267-316](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L267-L316)

### Route Segment Structure

Each route segment in a multi-chip transfer contains:

| Field | Type | Description |
| --- | --- | --- |
| `device_id` | int | Which device this segment executes on |
| `src` | [device, row, col] | Starting coordinate on this device |
| `dst` | [[device, row, col]] | Ending coordinate(s) on this device (may be ethernet gateway) |
| `noc_type` | string | "NOC0" or "NOC1" |
| `injection_rate` | float | Maximum injection rate in bytes/cycle |
| `start_cycle` | int | When this segment begins |
| `end_cycle` | int | When this segment completes |
| `links` | array | List of [device, row, col, link_type] tuples describing the path |

**Sources:**[tt_npe/scripts/timeline_schema_v1.json 63-80](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/timeline_schema_v1.json#L63-L80)

## Multi-chip Transfer Timing

### Link Latency

When a transfer crosses from one device to another, a link latency penalty is applied to model the delay of the inter-device ethernet connection:

`LINK_LATENCY = 1200  # cycles`
This latency is added between route segments. The start cycle of the next segment is calculated as:

```
next_segment.start_cycle = previous_segment.end_cycle + LINK_LATENCY
```

**Sources:**[tt_npe/scripts/generate_multichip_data.py 269](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L269-L269)[tt_npe/scripts/generate_multichip_data.py 313](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L313-L313)

### Transfer Time Estimation

For each segment, the transfer time is estimated based on:

`def estimate_transfer_time(total_bytes, num_links, injection_rate):    transfer_cycles = total_bytes / injection_rate    return max(40, int(transfer_cycles))`
The minimum transfer time is 40 cycles, regardless of payload size.

**Sources:**[tt_npe/scripts/generate_multichip_data.py 230-238](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L230-L238)

## Multi-chip Device Model

### WormholeMultichipDeviceModel

The `WormholeMultichipDeviceModel` class implements multi-chip simulation by composing multiple single-chip device models. It uses a composition pattern rather than inheritance.

**Key Characteristics:**

*   Maintains an array of single-chip device models (one per device)
*   Delegates routing calls to the appropriate device model based on device ID
*   Scales bandwidth and latency lookup tables across all devices
*   Handles cross-device coordinate translation

**Sources:** Referenced in high-level diagrams, implemented in device model subsystem (see [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.4-multi-chip-device-models))

## Timeline Output Format for Multi-chip Transfers

### Multi-device Transfer Structure

In the timeline JSON output, multi-chip transfers are represented with multiple route segments:

`{    "transfer_id": 1,    "noc_event_type": "WRITE",    "total_bytes": 2048,    "src": [0, 1, 1],    "dst": [[1, 1, 1]],    "start_cycle": 0,    "end_cycle": 250,    "route": [        {            "device_id": 0,            "src": [0, 1, 1],            "dst": [[0, 6, 1]],            "start_cycle": 0,            "end_cycle": 111,            "links": [...]        },        {            "device_id": 1,            "src": [1, 6, 1],            "dst": [[1, 1, 1]],            "start_cycle": 177,            "end_cycle": 250,            "links": [...]        }    ]}`
**Key Properties:**

1.   The top-level `src` and `dst` represent the logical endpoints
2.   The `route` array contains one or more segments
3.   Each segment has its own `device_id`, timing, and link list
4.   The gap between segments (111 → 177) represents link latency
5.   Links within each segment are prefixed with their device ID

**Sources:**[tt_npe/scripts/timeline_schema_v1.json 83-126](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/timeline_schema_v1.json#L83-L126)

### Chips Coordinate System

The timeline also includes a `chips` dictionary that maps device IDs to their physical coordinates in the fabric:

`{    "chips": {        "0": [1, 0, 0, 0],  // [x, y, rack, shelf]        "1": [1, 1, 0, 0],        ...    }}`
This enables visualization tools to display the physical layout of the multi-chip system.

**Sources:**[tt_npe/scripts/timeline_schema_v1.json 30-40](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/timeline_schema_v1.json#L30-L40)[tt_npe/scripts/generate_multichip_data.py 67-77](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/scripts/generate_multichip_data.py#L67-L77)

The T3K line configuration connects 8 devices in a linear chain:

Devices in the middle (0, 3, 7, 6, 2, 1) must forward traffic, so they have `forwarding_channel_pairs` configured. Endpoint devices (4, 5) do not forward.

## Summary

Multi-chip simulation in tt-npe involves:

1.   **Topology Definition:**`topology.json` files describe physical connectivity
2.   **Path Finding:**`TopologyGraph.find_optimal_path()` computes multi-hop routes
3.   **Route Elaboration:** Logical transfers are broken into per-device segments
4.   **Timing Model:** Link latency is added between segments
5.   **Output:** Timeline JSON contains detailed multi-segment routes with device IDs

This enables accurate modeling of NoC congestion and performance across complex multi-chip fabrics such as T3K and N150 configurations.
