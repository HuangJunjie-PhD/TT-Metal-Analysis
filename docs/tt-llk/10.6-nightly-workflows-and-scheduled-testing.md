This page documents scheduled test execution for extended validation beyond pull request workflows. The CI infrastructure supports nightly testing through the `pytest_markers` parameter in `setup-and-test.yml`, which controls test selection via pytest marker expressions. Nightly runs execute tests marked `@pytest.mark.nightly` and `@pytest.mark.perf`, excluded from PR workflows to maintain fast feedback cycles.

For PR-triggered workflows and change detection, see [on-pr.yml Workflow and Change Detection](https://deepwiki.com/tenstorrent/tt-llk/10.2-on-pr.yml-workflow-and-change-detection). For the reusable test execution workflow, see [setup-and-test.yml: Compilation and Execution](https://deepwiki.com/tenstorrent/tt-llk/10.3-setup-and-test.yml:-compilation-and-execution-phases).

## Test Marker System

Pytest markers control test selection through decorator annotations. Tests are marked with `@pytest.mark.<marker>` decorators, enabling selective execution via the `-m` flag.

### Marker Categories

| Marker | Decorator | PR Execution | Nightly Execution |
| --- | --- | --- | --- |
| `nightly` | `@pytest.mark.nightly` | Excluded via `"not nightly"` | Included via `"nightly"` |
| `perf` | `@pytest.mark.perf` | Excluded via `"not perf"` | Included via `"perf"` |
| `quasar` | `@pytest.mark.quasar` | Conditional via detect-changes | Always included |
| (none) | No decorator | Always included | Always included |

## Nightly vs PR Test Selection

Nightly workflows execute a superset of PR tests, including all standard tests plus extended validation suites. The primary differences are marker filtering and test execution time.

### PR Test Selection

PR workflows execute tests without `nightly` or `perf` markers. The [.github/workflows/on-pr.yml 148-151](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L148-L151) workflow configures test selection:

Conditional job execution via [.github/workflows/on-pr.yml 148-151](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L148-L151):

This excludes:

*   Tests decorated with `@pytest.mark.nightly` (extended format sweeps, large input dimensions)
*   Tests decorated with `@pytest.mark.perf` (profiler builds with `ProfilerBuild.Yes`)
*   Tests decorated with `@pytest.mark.quasar` (Quasar-specific tests run separately)

### Nightly Test Selection

Nightly workflows configure `pytest_markers` to include extended test categories. Example configuration:

This includes:

*   Tests decorated with `@pytest.mark.nightly` (comprehensive format sweeps via `format_combination_sweep`)
*   Tests decorated with `@pytest.mark.perf` (profiler builds with `-DLLK_PROFILER`)
*   Standard tests (no marker)

Nightly runs can increase `test_splits` from 2 to 10-20 for higher parallelization, as they execute thousands of test variants from exhaustive parameterization.

## Reusable Workflow Configuration

The `setup-and-test.yml` workflow is designed as a reusable workflow callable from both PR and nightly contexts. It accepts configuration inputs that control test selection, parallelization, and timeout behavior.

### Workflow Input Parameters

The [.github/workflows/setup-and-test.yml 4-47](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L4-L47) reusable workflow accepts configuration inputs:

| Parameter | Type | Default | PR Value | Nightly Example |
| --- | --- | --- | --- | --- |
| `docker_image` | string | (required) | `${{ needs.build-images.outputs.docker-image }}` | Same |
| `runs_on` | string | (required) | `tt-ubuntu-2204-n150-stable` (Wormhole) `tt-ubuntu-2204-p150b-stable` (Blackhole) | Same |
| `test_splits` | number | (required) | 2 | 10-20 |
| `pytest_markers` | string | `"not perf and not quasar"` | `"not perf and not nightly and not quasar"` | `"nightly or perf"` |
| `timeout_minutes` | number | 80 | 80 | 240-360 |
| `random_order` | boolean | false | false | Optional |
| `coverage` | boolean | false | false | true |
| `use_durations` | boolean | true | true | true |
| `log_level` | string | `"WARNING"` | `"WARNING"` | `"DEBUG"` or `"INFO"` |

### Marker Expression Syntax

The `pytest_markers` parameter accepts pytest marker boolean expressions:

*   `not perf` - Exclude `@pytest.mark.perf` decorated tests
*   `not nightly` - Exclude `@pytest.mark.nightly` decorated tests
*   `nightly or perf` - Include tests with either marker
*   `nightly and not perf` - Include `nightly` but exclude `perf`

## Test Parameterization for Extended Coverage

Nightly workflows benefit from the parameterization infrastructure that generates comprehensive test variants without manual enumeration.

### Format Combination Sweeps

The `format_combination_sweep` function generates all possible format combinations for exhaustive validation:

Nightly tests can use `all_same=False` to generate full cartesian products of format combinations, resulting in thousands of test variants that cover all format conversion paths.

### Constraint-Based Parameterization

The `parametrize` decorator supports dependency resolution for complex test parameterization:

This enables nightly tests to explore the full parameter space while respecting hardware constraints and format compatibility rules.

### Compilation Deduplication

The test infrastructure deduplicates compilation artifacts using variant hashing:

This ensures that nightly runs with thousands of test variants don't waste time recompiling identical code, and enables the two-phase PRODUCE/CONSUME model where compilation happens once and execution is parallelized across workers.
