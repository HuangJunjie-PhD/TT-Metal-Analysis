This page documents the GDDR memory controller subsystem managed by the CMFW (Chip Management Firmware). It covers the architecture, initialization sequence, runtime management, and testing of the 8 GDDR controller instances, each running dedicated MRISC (Memory RISC) firmware.

For information about MRISC firmware binaries and parameters, see [MRISC Firmware for GDDR](https://deepwiki.com/tenstorrent/tt-system-firmware/5.1-mrisc-firmware-for-gddr). For general hardware initialization flow, see [Boot and Initialization Sequence](https://deepwiki.com/tenstorrent/tt-system-firmware/3.1-boot-and-initialization-sequence).

* * *

## Architecture Overview

The Blackhole ASIC contains 8 independent GDDR memory controller instances, each managed by a dedicated MRISC processor. Each controller has 128KB of L1 memory and interfaces with the Network-on-Chip (NOC) through 3 NOC2AXI ports.

### GDDR Controller Instance Layout

**Sources:**[lib/tenstorrent/bh_arc/gddr.h 13-17](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.h#L13-L17)[lib/tenstorrent/bh_arc/gddr.c 41-49](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L41-L49)

### Memory Addressing

Each GDDR controller instance is accessed via NOC coordinates that vary by instance number. The CMFW uses TLB 13 (`MRISC_SETUP_TLB`) to access MRISC L1 memory and registers.

| Address Space | Base Address | Purpose |
| --- | --- | --- |
| `MRISC_L1_ADDR` | `0x20000000000` (1ULL << 37) | MRISC L1 memory access |
| `MRISC_REG_ADDR` | `0x10000000000` (1ULL << 40) | MRISC control registers |
| `GDDR_TELEMETRY_TABLE_ADDR` | `0x8000` (in L1) | Telemetry data structure |
| `GDDR_MSG_STRUCT_ADDR` | `0x6000` (in L1) | Message passing interface |
| `MRISC_FW_CFG_OFFSET` | `0x3C00` (in L1) | Firmware configuration |

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 44-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L44-L46)[lib/tenstorrent/bh_arc/gddr.h 19-21](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.h#L19-L21)

## Initialization Sequence

GDDR initialization follows a multi-stage process executed during boot, after PCIe and Tensix initialization. The sequence is orchestrated by two main functions: `InitMrisc()` and `gddr_training()`.

### Initialization Flow

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 351-441](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L351-L441)[lib/tenstorrent/bh_arc/gddr.c 509-544](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L509-L544)

The MRISC firmware is loaded using the `spi_arc_dma_transfer_to_tile()` function, which efficiently transfers data from SPI flash to NOC-accessible L1 memory.

## MRISC Communication Interface

The CMFW communicates with MRISC processors through a register-based message passing interface. Messages are written to control registers in the MRISC address space, and the MRISC firmware clears the register when processing is complete.

### Message Interface Registers

**Sources:**[lib/tenstorrent/bh_arc/gddr.h 23-49](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.h#L23-L49)

### Message Passing Flow

The `mrisc_message()` function implements the common pattern for sending messages to multiple GDDR instances:

1.   **Pre-check:** Verify `MRISC_MSG_REGISTER` is 0 (not busy)
2.   **Send:** Write message opcode to `MRISC_MSG_REGISTER`
3.   **Wait:** Poll until `MRISC_MSG_REGISTER` returns to 0
4.   **Timeout:** Return error if timeout expires

**Key Functions:**

*   `check_mrisc_busy()` - Validates message buffer is free ([gddr.c 211-221](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L211-L221))
*   `wait_mrisc_not_busy()` - Polls for message completion ([gddr.c 223-234](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L223-L234))
*   `mrisc_message()` - Generic message sender ([gddr.c 546-574](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L546-L574))
*   `set_mrisc_power_setting()` - Power control wrapper ([gddr.c 576-582](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/gddr.c#L576-L582))

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 211-234](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L211-L234)[lib/tenstorrent/bh_arc/gddr.c 546-582](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L546-L582)

## Training and BIST

### Training Process

After MRISC firmware is loaded and reset is released, the MRISC executes its training sequence autonomously. The CMFW polls `MRISC_INIT_STATUS` to detect completion.

**Training States:**

| Value | Constant | Meaning |
| --- | --- | --- |
| `0x11111111` | `MRISC_INIT_BEFORE` | Pre-training state |
| `0x0` | `MRISC_INIT_STARTED` | Training in progress |
| `0xdeadbeef` | `MRISC_INIT_FINISHED` | Training successful |
| `0xfa11` | `MRISC_INIT_FAILED` | Training failed |

The `CheckGddrTraining()` function polls with a 1000ms timeout, sleeping 1ms between iterations.

**Sources:**[lib/tenstorrent/bh_arc/gddr.h 30-34](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.h#L30-L34)[lib/tenstorrent/bh_arc/gddr.c 444-463](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L444-L463)

### Hardware BIST (Built-In Self-Test)

After successful training, the CMFW triggers a hardware memory test using the `MRISC_MSG_TYPE_RUN_MEMTEST` message. This test validates memory integrity before the system becomes operational.

**BIST Parameters:**

The test writes message arguments to L1 at `GDDR_MSG_STRUCT_ADDR` and reads the result from offset `+32` bytes.

## Runtime Management

### Power State Control

The MRISC PHY supports two power states controlled via messages:

*   **PHY Wakeup:**`MRISC_MSG_TYPE_PHY_WAKEUP` - Activates PHY for memory access
*   **PHY Powerdown:**`MRISC_MSG_TYPE_PHY_POWERDOWN` - Puts PHY in low-power state

The `set_mrisc_power_setting(bool on)` function broadcasts the appropriate message to all active MRISC instances with a 1000ms timeout.

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 576-582](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L576-L582)[lib/tenstorrent/bh_arc/gddr.h 44-47](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.h#L44-L47)

### GDDR Reset and Re-training

The `toggle_gddr_reset()` message handler (message ID `0xB6`) provides the ability to reset individual GDDR instances, re-train them, and re-run BIST. This is useful for diagnostic testing and error recovery.

### Telemetry Access

MRISC firmware maintains a telemetry table at L1 address `0x8000` (`GDDR_TELEMETRY_TABLE_ADDR`). The CMFW can read this table using DMA or direct NOC reads via `read_gddr_telemetry_table()`.

**Telemetry Structure Version Check:**

The telemetry includes MRISC firmware version information used to check for feature support (e.g., memtest requires version >= 2.7).

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 117-143](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L117-L143)[lib/tenstorrent/bh_arc/gddr.c 241-254](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L241-L254)

## Harvesting and Masking

Not all 8 GDDR instances may be available on a given board. The system supports two levels of masking:

1.   **Harvesting:** Determined by fuses, stored in `tile_enable.gddr_enabled` bitmask
2.   **FW Table Mask:** Optional override via `dram_table.dram_mask` if `dram_mask_en` is set

The `GetDramMask()` function computes the intersection of these masks to determine which instances should be initialized.

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 200-209](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L200-L209)

## Testing and Validation

### E2E Smoke Test: GDDR Reset

The `test_gddr_reset()` function validates the complete reset/retrain/BIST cycle for each GDDR instance:

**Test Flow:**

1.   Send `TT_SMC_MSG_TOGGLE_GDDR_RESET` message for each instance (0-7)
2.   Check response status (0 = success)
3.   Handle harvested instances (detail code 2)
4.   Read `TAG_GDDR_STATUS` telemetry to verify BIST results
5.   Validate BIST complete and pass/fail bits match message response

### Integration with System Init

GDDR initialization integrates into the Zephyr SYS_INIT framework with two priority levels:

*   `InitMrisc_PRIO` (105): Loads firmware and releases MRISC reset
*   `gddr_training_PRIO` (111): Waits for training and runs BIST

This ordering ensures MRISC firmware is loaded after PCIe/Tensix init but training completion is validated before the system proceeds to CAT init.

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 442](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L442-L442)[lib/tenstorrent/bh_arc/gddr.c 584](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L584-L584)[include/tenstorrent/sys_init_defines.h 28-34](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/sys_init_defines.h#L28-L34)
