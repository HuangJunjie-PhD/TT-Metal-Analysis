## Purpose and Scope

This page documents the EmitPy code generation path: how TTNN dialect IR is lowered to Python source code that calls into the `ttnn` Python library. The path covers the pipeline composition, pass sequence, op conversion patterns, the `EmitPy` intermediate dialect, and the final text emission step.

*   For the analogous C++ output path, see [Code Generation: EmitC](https://deepwiki.com/tenstorrent/tt-mlir/3.5-code-generation:-emitc).
*   For the TTNN dialect itself, see [TTNN Dialect](https://deepwiki.com/tenstorrent/tt-mlir/2.2-ttnn-dialect).
*   For the full TTIR-to-TTNN backend pipeline that precedes EmitPy, see [TTIR to TTNN Backend Pipeline](https://deepwiki.com/tenstorrent/tt-mlir/3.2-ttir-to-ttnn-backend-pipeline).
*   For the `ttrt emitpy` runtime tool that invokes this path, see [ttrt: Runtime Execution Tool](https://deepwiki.com/tenstorrent/tt-mlir/8.2-ttrt:-runtime-execution-tool).

## Pipeline Registration

Three named pipelines are registered for Python code generation:

| Pipeline Name | Function | Description |
| --- | --- | --- |
| `ttnn-to-emitpy-device-pipeline` | `createTTNNToEmitPyDevicePipeline` | Device module only: TTNN → EmitPy |
| `ttir-to-emitpy-pipeline` | `createTTIRToEmitPyPipeline` | Full: TTIR → TTNN → EmitPy (both modules) |
| `ttnn-to-emitpy-pipeline` | `createTTNNToEmitPyPipeline` | TTNN-only input: TTNN → EmitPy (both modules) |

Sources: `lib/Dialect/TTNN/Pipelines/TTNNPipelines.cpp`.

## Pipeline Options

The `TTNNToEmitPyDevicePipelineOptions` struct controls the EmitPy device pipeline behavior:

| Option | Type | Default | Effect |
| --- | --- | --- | --- |
| `targetModule` | `bool` | `false` | Generate a Python module with a `forward(inputs_tuple, device)` signature instead of a standalone script |
| `loadInputTensorsFromDisk` | `bool` | `false` | Use `ttnn.load_tensor` to read real inputs from disk instead of generating synthetic `ttnn.ones` tensors |
| `tensorLoadDirectory` | `string` | `""` | Directory where tensor `.tensorbin` files are found |
| `tensorLoadFilePrefix` | `string` | `"arg"` | File name prefix for loaded tensors |

The `ConvertTTNNToEmitPyOptions` struct, used by the conversion pass itself, carries flags like `enableGoldenMode` which is used for CPU module paths to generate TTNN calls as reference/golden implementations [lib/Conversion/TTNNToEmitPy/TTNNToEmitPyPass.cpp 168-172](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTNNToEmitPy/TTNNToEmitPyPass.cpp#L168-L172)

Sources: `lib/Conversion/TTNNToEmitPy/TTNNToEmitPyPass.cpp`.

## Const-Eval and CPU Hoisting

The compiler identifies subgraphs that can be evaluated on the host CPU during the initialization phase.

### CPU Hoisting Analysis

The `HoistCPUOps` pass identifies operations that can be moved from the device to the host CPU.

*   **`ConstEvalAnalyze`**: Specifically targets constant evaluation subgraphs for hoisting using a Disjoint Set Union (DSU) algorithm to group operations [lib/Transforms/ConstEvalHoist.cpp 108-114](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Transforms/ConstEvalHoist.cpp#L108-L114)
*   **`isSharedOp`**: Identifies ops with `TTCoreDuplicateConstEvalTrait` which are handled specifically during analysis [lib/Transforms/ConstEvalHoist.cpp 30-33](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Transforms/ConstEvalHoist.cpp#L30-L33)
*   **`EmitPyConstEvalCaching`**: Implements caching for constant evaluation results to avoid redundant host-side computation in the generated Python code [lib/Conversion/TTNNToEmitPy/EmitPyConstEvalCaching.cpp 25-40](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTNNToEmitPy/EmitPyConstEvalCaching.cpp#L25-L40)
*   **`TTNNPrepareConstEvalCaching`**: Prepares the IR for caching by identifying tensors that result from constant evaluation.

### TTNN Trace Hoisting

For performance, sequences of device operations can be captured into a "trace" and re-executed. The `TTNNTraceHoistTransform` pass identifies hoistable operations (excluding terminators, creation ops, and deallocs) and wraps them in `CaptureOrExecuteTraceOp`.

Sources: `lib/Transforms/ConstEvalHoist.cpp`, `lib/Conversion/TTNNToEmitPy/EmitPyConstEvalCaching.cpp`.

## Translation to Python Text: `PythonEmitter`

The final text emission is performed by the `PythonEmitter` class [lib/Target/Python/TranslateToPython.cpp 106-200](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/Python/TranslateToPython.cpp#L106-L200)

**Translation flow**

The emitter maintains proper Python indentation and manages value-to-name mapping. It also includes an `ExpressionBuilder` for recursively building inline Python expressions [lib/Target/Python/TranslateToPython.cpp 78-103](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/Python/TranslateToPython.cpp#L78-L103)

*   **`isValidPythonIdentifier`** (in `EmitPyOps.cpp`): Ensures generated variable and class names are legal in Python [lib/Dialect/EmitPy/IR/EmitPyOps.cpp 131-162](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/EmitPy/IR/EmitPyOps.cpp#L131-L162)
*   **`shouldEmitFile`**: Used during multi-file emission to filter ops based on the target `fileId`[lib/Target/Python/TranslateToPython.cpp 156-157](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/Python/TranslateToPython.cpp#L156-L157)

Sources: `lib/Target/Python/TranslateToPython.cpp`, `lib/Dialect/EmitPy/IR/EmitPyOps.cpp`.
