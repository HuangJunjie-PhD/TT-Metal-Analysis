# Installation Modes

Relevant source files
*   [install.sh](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh)

This page documents the different installation modes available in the TT-Installer script. These modes control how the installer interacts with the user and which components are installed. Understanding these modes is essential for configuring the installation process to meet your specific requirements, particularly for automated deployments or containerized environments.

For information about specific configuration options within each mode, see [Environment Variables](https://deepwiki.com/tenstorrent/tt-installer/4.1-environment-variables) and [Python Environment Options](https://deepwiki.com/tenstorrent/tt-installer/4.3-python-environment-options).

## Overview of Installation Modes

The TT-Installer provides three distinct installation modes:

1.   **Interactive Mode** - Default mode with full user prompting
2.   **Non-Interactive Mode** - For automated installations without user input
3.   **Container Mode** - Specialized mode for use within container environments

Each mode modifies how the installer operates, which components it installs, and how it handles user interaction.

Sources: [install.sh 95-113](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L95-L113)

## Mode Comparison

The following table summarizes the key differences between the installation modes:

| Feature | Interactive Mode | Non-Interactive Mode | Container Mode |
| --- | --- | --- | --- |
| User prompts | Yes | No | No |
| Reboot behavior | Ask user (default) | Never reboot | Never reboot |
| KMD installation | Yes (default) | Yes (default) | Skipped |
| HugePages setup | Yes (default) | Yes (default) | Skipped |
| Podman installation | Yes (default) | Yes (default) | Skipped |
| Environment variable | N/A (default) | `TT_MODE_NON_INTERACTIVE=0` | `TT_MODE_CONTAINER=0` |

Sources: [install.sh 95-113](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L95-L113)[install.sh 651-655](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L651-L655)

## Interactive Mode

Interactive mode is the default installation mode when no special mode flags are set. It provides a guided installation experience with user prompts at key decision points.

### Characteristics

*   Prompts the user for input at various decision points
*   Allows customization of the installation during the process
*   Provides informative messages and status updates
*   Asks for confirmation before proceeding with critical steps

### Decision Points

In Interactive Mode, the user is prompted to make decisions about:

1.   Whether to continue with the installation
2.   Python installation method (venv, system Python, pipx)
3.   Whether to install Podman and Metalium container
4.   Whether to force reinstall KMD if already present
5.   Whether to reboot after installation completion

## Non-Interactive Mode

Non-Interactive Mode is designed for automated installations where user input is not available or desired, such as in scripts or automated deployment pipelines.

### Characteristics

*   No user prompts or interactive questions
*   Uses default values or environment variables for all configuration options
*   Suitable for scripted or automated installations
*   Configurable via environment variables

### Enabling Non-Interactive Mode

To enable Non-Interactive Mode, set the `TT_MODE_NON_INTERACTIVE` environment variable to `0` before running the installer:

```
export TT_MODE_NON_INTERACTIVE=0
./install.sh
```

### Behavior Changes

When Non-Interactive Mode is enabled:

*   The `confirm()` function automatically returns true without prompting the user
*   If the reboot option is set to "Ask the user" (1), it's automatically changed to "Never reboot" (2)
*   All installation options use their default values unless overridden by environment variables

### Configuration in Non-Interactive Mode

In Non-Interactive Mode, you should configure all installation options via environment variables before running the installer. For example:

```
export TT_MODE_NON_INTERACTIVE=0
export TT_PYTHON_CHOICE=2
export TT_REBOOT_OPTION=2
export TT_SKIP_INSTALL_PODMAN=0
./install.sh
```

## Container Mode

Container Mode is a specialized installation mode designed for use when running the installer inside a container environment.

### Characteristics

*   Skips components that require host system privileges (KMD, HugePages)
*   Prevents reboot attempts
*   Skips Podman installation (containers within containers are problematic)
*   Designed for containerized deployment scenarios

### Enabling Container Mode

To enable Container Mode, set the `TT_MODE_CONTAINER` environment variable to `0` before running the installer:

```
export TT_MODE_CONTAINER=0
./install.sh
```

### Automatic Configuration

When Container Mode is enabled, the following settings are automatically applied:

*   `SKIP_INSTALL_KMD=0` (Skip kernel module driver installation)
*   `SKIP_INSTALL_HUGEPAGES=0` (Skip HugePages configuration)
*   `SKIP_INSTALL_PODMAN=0` (Skip Podman installation)
*   `REBOOT_OPTION=2` (Never reboot)

## Code Implementation

The installation modes are implemented in the main `install.sh` script through the following key sections:

### Mode Flags Definition

The mode-specific flags are defined at the beginning of the script:

```
# Container mode flag (set to 0 to enable, which skips KMD and HugePages and never reboots)
CONTAINER_MODE=${TT_MODE_CONTAINER:-1}

# Non-interactive mode flag (set to 0 to enable)
NON_INTERACTIVE_MODE=${TT_MODE_NON_INTERACTIVE:-1}
```

### Mode Behavior Configuration

The script then configures the behavior based on the selected mode:

```
# If container mode is enabled, skip KMD and HugePages
if [[ "${CONTAINER_MODE}" = "0" ]]; then
    SKIP_INSTALL_KMD=0
    SKIP_INSTALL_HUGEPAGES=0 # Both KMD and HugePages must live on the host kernel
    SKIP_INSTALL_PODMAN=0 # No podman in podman
    REBOOT_OPTION=2 # Do not reboot
fi

if [[ "${NON_INTERACTIVE_MODE}" = "0" ]]; then
    # In non-interactive mode, we can't ask the user for anything
    # So if they don't provide a reboot choice we will pick a default
    if [[ "${REBOOT_OPTION}" = "1" ]]; then
        REBOOT_OPTION=2 # Do not reboot
    fi
fi
```

### Confirmation Function

The `confirm()` function respects the Non-Interactive Mode setting:

```
# Function to prompt for yes/no
confirm() {
    # In non-interactive mode, always return true
    if [[ "${NON_INTERACTIVE_MODE}" = "0" ]]; then
        return 0
    fi

    while true; do
        read -rp "$1 [Y/n] " yn
        case ${yn} in
            [Nn]* ) echo && return 1;;
            [Yy]* | "" ) echo && return 0;;
            * ) echo "Please answer yes or no.";;
        esac
    done
}
```
