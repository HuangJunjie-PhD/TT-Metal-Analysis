This document describes the GitHub Actions workflow infrastructure that orchestrates benchmark and demo execution across Tenstorrent hardware architectures. This is the most critical component in the repository, providing automated performance validation and continuous testing.

For information about the specific model benchmarks and their implementations, see [torch-xla Backend Benchmarks](https://deepwiki.com/tenstorrent/tt-forge/3.3-torch-xla-backend-benchmarks). For information about metrics collection and result formatting, see [Performance Metrics and Reporting](https://deepwiki.com/tenstorrent/tt-forge/3.5-performance-metrics-and-reporting). For practical instructions on running benchmarks, see [Running Benchmarks](https://deepwiki.com/tenstorrent/tt-forge/6.1-running-benchmarks).

## Workflow Overview

The benchmark infrastructure consists of two primary GitHub Actions workflows that work in tandem to validate performance across different test scenarios:

| Workflow | Primary Purpose | Configuration File | Target |
| --- | --- | --- | --- |
| `perf-benchmark.yml` | Performance benchmarking | `perf-bench-matrix.json` | Model inference performance |
| `demo-tests.yml` | Demo validation | `models-matrix.json` | End-to-end demo applications |

Both workflows share a common architecture pattern: matrix-based test execution with dynamic filtering, hardware-specific job distribution, and automated failure notification.

**Sources:**[.github/workflows/demo-tests.yml 1-238](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L1-L238)

## Performance Benchmark Workflow

The `perf-benchmark.yml` workflow is the primary orchestrator for performance testing. It accepts both manual triggers via `workflow_dispatch` and automated calls via `workflow_call`, making it suitable for both ad-hoc testing and integration into CI/CD pipelines.

### Workflow Inputs

The workflow accepts several configuration inputs that control test execution:

| Input | Type | Description | Default |
| --- | --- | --- | --- |
| `docker-image` | string | Container image for test execution | `ghcr.io/tenstorrent/tt-forge-slim:latest` |
| `project-filter` | choice | Project to test (tt-forge-fe, tt-xla, tt-forge) | `tt-forge` |
| `test-filter` | string | Substring filter for test names | (empty) |
| `update-wheel` | boolean | Whether to update project wheel | false |
| `sh-runner` | boolean | Use shared runners instead of dedicated hardware | false |
| `parent_run_id` | string | Parent workflow ID for tracking | (empty) |
| `runs-on-filter` | choice | Hardware architecture filter (All, n150, p150) | `All` |

### Job Execution Flow

The workflow uses a multi-stage execution pattern:

1.   **Filter Stage**: The `filter-tests` job runs on a standard Ubuntu runner and dynamically generates test matrices by invoking the `filter-test-matrix.py` script. It processes the `perf-bench-matrix.json` configuration file and applies project and test filters.

2.   **Architecture Split**: The filtered matrix is split into `matrix_n150` (Wormhole) and `matrix_p150` (Blackhole) configurations using `jq` to filter by the `runs-on` field. Skip flags are set if matrices are empty or filtered out by `runs-on-filter`.

3.   **Hardware Execution**: Two separate jobs (`run-n150-perf-benchmarks` and `run-p150-perf-benchmarks`) execute in parallel, each calling the reusable `call-perf-test.yml` workflow with their respective matrices. These jobs only run if their skip flags are false.

4.   **Failure Tracking**: The `fail-notify` job aggregates results from both hardware-specific jobs using the `re-actors/alls-green@release/v1` action. It extracts metadata like branch name, draft status, and version tag from the Docker image.

5.   **Slack Notification**: The `fail-send-msg` job sends notifications to Slack channel `C088QN7E0R3` only when tests fail on the main branch in non-draft releases with a valid `parent_run_id`.

6.   **Data Production**: The final `produce-data` job triggers the `produce_data.yml` workflow to collect and process test metrics.

## Demo Tests Workflow

The `demo-tests.yml` workflow validates end-to-end demo applications across different frontends. It follows a similar architecture to `perf-benchmark.yml` but executes demos directly in the workflow jobs rather than delegating to a separate workflow.

### Demo Execution Architecture

**Sources:**[.github/workflows/demo-tests.yml 54-132](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L54-L132)

The demo workflow differs from performance benchmarks in key ways:

1.   **Single Matrix Execution**[[.github/workflows/demo-tests.yml 85-94](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L85-L94)]: Unlike performance benchmarks which split by architecture, demos use a single matrix strategy job that executes across all configurations in parallel using GitHub Actions' matrix feature.

2.   **Direct Hardware Access**[[.github/workflows/demo-tests.yml 94-104](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L94-L104)]: The job runs directly on hardware runners with pattern `tt-ubuntu-2204-{runs-on}-stable` and mounts the Tenstorrent device (`/dev/tenstorrent`) and hugepage volumes into the container.

3.   **Flexible Dependency Installation**: The workflow supports dependency installation via `requirements.txt` located in the demo folder [[.github/workflows/demo-tests.yml 128-130](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L128-L130)].

4.   **Environment Configuration**[[.github/workflows/demo-tests.yml 126-127](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L126-L127)]: The workflow sets up `PYTHONPATH` to include the repository root so demos can find models in the `third_party` folder.

## Test Matrix Configuration

The test matrix system uses JSON configuration files that define test parameters and project-specific defaults. These files are processed by `filter-test-matrix.py` to generate dynamic test matrices.

### Matrix File Structure

The `models-matrix.json` file contains project configurations for `tt-forge-onnx`, `tt-xla`, and `tt-torch`[[.github/workflows/models-matrix.json 1-45](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/models-matrix.json#L1-L45)].

| Project | Default Runners | Tests Count |
| --- | --- | --- |
| `tt-forge-onnx` | `["n150", "p150"]` | 6 |
| `tt-xla` | `n150` | 9 |
| `tt-torch` | (N/A) | 5 |

**Sources:**[.github/workflows/models-matrix.json 1-45](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/models-matrix.json#L1-L45)

### Test Parameter Inheritance

Test parameters follow an inheritance and override pattern within `filter-test-matrix.py`:

1.   **Flattening**: The `flatten_matrix` function merges `test-defaults` with specific `tests` entries [[.github/workflows/filter-test-matrix.py 10-24](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/filter-test-matrix.py#L10-L24)].
2.   **Runner Expansion**: If `runs-on` is a list, the script expands the matrix so each runner becomes a separate test entry [[.github/workflows/filter-test-matrix.py 18-20](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/filter-test-matrix.py#L18-L20)].
3.   **Project Filtering**: The `filter_matrix` function applies project and name-based filters [[.github/workflows/filter-test-matrix.py 27-48](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/filter-test-matrix.py#L27-L48)].
4.   **Runner Mapping**: The `update_runners` function maps generic runner names (e.g., `p150`) to specific infrastructure labels (e.g., `p150b` or `n150-perf`) based on the `--sh-runner` flag [[.github/workflows/filter-test-matrix.py 51-60](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/filter-test-matrix.py#L51-L60)].

**Sources:**[.github/workflows/filter-test-matrix.py 10-60](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L10-L60)

## Dynamic Test Filtering

The `filter-test-matrix.py` script provides dynamic test selection based on command-line filters.

### Filter Script Invocation

**Demo Tests Invocation**[[.github/workflows/demo-tests.yml 73-77](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L73-L77)]:

`result=$(python .github/workflows/filter-test-matrix.py \  .github/workflows/models-matrix.json \  ${{ inputs.project-filter || '' }} \  ${{ inputs.test-filter && format('--test-filter "{0}"', inputs.test-filter) || '' }} \  --sh-runner)`
The script returns JSON with the following structure:

`{  "matrix": [ { "project": "tt-xla", "name": "resnet50", "runs-on": "n150-perf", ... } ],  "matrix_skip": "false"}`
**Sources:**[.github/workflows/filter-test-matrix.py 63-90](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L63-L90)[.github/workflows/demo-tests.yml 72-83](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L72-L83)

## Basic Tests Workflow

The `basic-tests.yml` workflow provides a lightweight validation path for frontend smoke tests. It does not use the complex JSON matrix system, instead building a matrix directly in bash based on inputs.

### Matrix Construction

The `build_matrix` job constructs a JSON array of `frontend` and `runs_on` combinations [[.github/workflows/basic-tests.yml 59-98](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L59-L98)]:

1.   **Runner Filter**: Defaults to both `n150` and `p150` if "All" is selected [[.github/workflows/basic-tests.yml 72-76](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L72-L76)].
2.   **Frontend Filter**: Maps the `tt-forge` project filter to `tt-xla` for basic testing [[.github/workflows/basic-tests.yml 78-82](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L78-L82)].
3.   **Loop Generation**: Iterates through selected frontends and runners to populate the `json_matrix`[[.github/workflows/basic-tests.yml 84-92](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L84-L92)].

### Basic Test Execution

The `basic-test` job executes `basic_tests/${{ matrix.frontend }}/demo_test.py`[[.github/workflows/basic-tests.yml 153](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L153-L153)]. It includes a specific workaround for GitHub Actions container `HOME` directory overrides [[.github/workflows/basic-tests.yml 128-142](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/basic-tests.yml#L128-L142)].

**Sources:**[.github/workflows/basic-tests.yml 59-154](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/basic-tests.yml#L59-L154)

## Hardware Architecture Support

The benchmark infrastructure supports multiple Tenstorrent hardware architectures, each identified by specific runner labels:

| Architecture | Code | Runner Pattern |
| --- | --- | --- |
| Wormhole | `n150` | `tt-ubuntu-2204-n150-stable` |
| Blackhole | `p150` | `tt-ubuntu-2204-p150-stable` |
| Wormhole (Multi-chip) | `n300` | `tt-ubuntu-2204-n300-stable` |

**Sources:**[.github/workflows/demo-tests.yml 94](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L94-L94)[.github/workflows/models-matrix.json 38-42](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/models-matrix.json#L38-L42)

### Docker Container Execution

All tests run within Docker containers to ensure environment consistency. Key configuration includes:

*   **Device Access**: `--device /dev/tenstorrent` is passed to allow the container to communicate with Tenstorrent hardware [[.github/workflows/demo-tests.yml 97](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L97-L97)].
*   **Hugepages**: `/dev/hugepages` and `/dev/hugepages-1G` are mounted for high-performance memory management [[.github/workflows/demo-tests.yml 99-100](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L99-L100)].
*   **Virtual Environment**: The workflows check for and activate a virtual environment at `/opt/venv` if it exists [[.github/workflows/demo-tests.yml 119-121](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/[.github/workflows/demo-tests.yml#L119-L121)[.github/workflows/basic-tests.yml 150-152](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/basic-tests.yml#L150-L152)].

**Sources:**[.github/workflows/demo-tests.yml 95-104](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L95-L104)[.github/workflows/basic-tests.yml 109-117](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/basic-tests.yml#L109-L117)
