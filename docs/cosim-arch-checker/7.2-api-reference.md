# API Reference

*   [bridge/bridge.h](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h)
*   [bridge/whisper/whisper_client.h](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/whisper/whisper_client.h)

This document provides a comprehensive reference for all public APIs exposed by the cosim-arch-checker framework. It covers the SystemVerilog DPI functions for monitoring DUT state, the Bridge orchestration interface, and the Whisper ISS client API.

For configuration details of these APIs, see [Configuration](https://deepwiki.com/tenstorrent/cosim-arch-checker/4-configuration). For integration examples, see [Integration Guide](https://deepwiki.com/tenstorrent/cosim-arch-checker/6-integration-guide).

## SystemVerilog DPI Interface

The DPI interface provides functions callable from SystemVerilog testbenches to monitor architectural state changes in the DUT. These functions form the primary data ingestion layer for the co-simulation framework.

### DPI Function Overview

Sources: [mon/mon_instr/mon_instr_dpi.cc 1-53](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L1-L53)

### Core Monitoring Functions

| Function | Purpose | Parameters |
| --- | --- | --- |
| `monitor_instr` | Reports instruction retirement | `char *name, int hart, uint64_t cycle, uint64_t tag, uint64_t pc, uint64_t opcode, uint32_t trap` |
| `monitor_gpr` | Reports GPR state changes | `char *name, int hart, uint64_t cycle, uint32_t addr, uint64_t data` |
| `monitor_fpr` | Reports FPR state changes | `char *name, int hart, uint64_t cycle, uint32_t addr, uint64_t data` |
| `monitor_vr` | Reports vector register changes | `char *name, int hart, uint64_t cycle, uint32_t addr, uint64_t *data` |
| `monitor_csr` | Reports CSR state changes | `char *name, int hart, uint64_t cycle, uint32_t addr, uint64_t data` |

All monitoring functions use a name-based dispatch mechanism to route calls to registered `cMonInstr` interface implementations.

Sources: [mon/mon_instr/mon_instr_dpi.cc 18-51](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L18-L51)

### Interface Registration

`void instr_register_intf(std::string name, cMonInstr *intf);void instr_deregister_intf(std::string name);`
These functions manage the interface registry that maps string names to `cMonInstr` implementations. The `name` parameter in monitoring functions must match a registered interface name.

Sources: [mon/mon_instr/mon_instr_dpi.cc 12-16](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L12-L16)

## Bridge Interface

The `cBridge` class serves as the central orchestrator for co-simulation, implementing the `cBridgeBase` interface and coordinating between DUT monitoring and Whisper ISS execution.

### cBridge Class API

Sources: [bridge/bridge.h 1-94](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L1-L94)

### Constructor and Destructor

`cBridge(int numHarts, int vLen);~cBridge();`
The constructor initializes the bridge with the specified number of harts and vector length. The destructor handles cleanup including Whisper disconnection.

Sources: [bridge/bridge.h 15-16](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L15-L16)

### Main Processing Function

`virtual void processDutInstrRetire(int hart, sRvInstr &dutInstr) override;`
This is the primary entry point called when the DUT retires an instruction. It orchestrates the comparison between DUT and Whisper execution by:

1.   Stepping Whisper to the corresponding instruction
2.   Collecting architectural state changes from both DUT and Whisper
3.   Feeding both states to the Core Architecture Checker

Sources: [bridge/bridge.h 28](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L28-L28)

## Whisper Client API

The Whisper client provides a C interface for communicating with the Whisper RISC-V ISS over TCP sockets. These functions enable stepping the ISS and querying/modifying its architectural state.

### Whisper Client Function Mapping

Sources: [bridge/whisper/whisper_client.h 1-58](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/whisper/whisper_client.h#L1-L58)

### Connection Functions

| Function | Purpose | Return Type |
| --- | --- | --- |
| `whisperConnect` | Establish TCP connection to Whisper ISS | `int` |
| `whisperDisconnect` | Close connection to Whisper ISS | `void` |

Sources: [bridge/whisper/whisper_client.h 8-13](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/whisper/whisper_client.h#L8-L13)

### Execution Functions

| Function | Purpose | Key Parameters |
| --- | --- | --- |
| `whisperStep` | Step ISS execution and get full state | `hart, time, instrTag, pc, instruction, changeCount` |
| `whisperSimpleStep` | Step ISS execution with minimal state | `hart, pc, instruction, changeCount` |

The `whisperStep` function is the primary execution interface, providing comprehensive execution results including privilege mode, floating-point flags, and trap information.

Sources: [bridge/whisper/whisper_client.h 17-23](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/whisper/whisper_client.h#L17-L23)

### State Query and Modification

| Function | Purpose | Parameters |
| --- | --- | --- |
| `whisperChange` | Get architectural state changes | `hart, resource, addr, value, valid` |
| `whisperPeek` | Read ISS register/memory state | `hart, resource, addr, value, valid` |
| `whisperPoke` | Write ISS register/memory state | `hart, resource, addr, value, valid` |
| `whisperReset` | Reset hart state | `hart, valid` |

Sources: [bridge/whisper/whisper_client.h 27-57](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/whisper/whisper_client.h#L27-L57)

## Key Data Structures

### sWhisperState Structure

The `sWhisperState` structure encapsulates all state information exchanged with the Whisper ISS:

`typedef struct sWhisperState {    // whisperStep outputs    uint64_t tag;    uint64_t time;    uint64_t pc;    uint32_t opcode;    unsigned changeCount;    char buffer[128];    unsigned bufferSize = 128;        // whisperChange outputs      uint32_t resource;    uint64_t address;    uint64_t value;    bool valid;    uint32_t privMode;    uint32_t fpFlags;    bool trap;} sWhisperState;`
Sources: [bridge/bridge.h 32-50](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L32-L50)

### Resource Type Enumeration

`typedef enum {    k_IntReg = 0,    k_FpReg = 1,    k_VecReg = 4} eResource;`
This enumeration defines the resource types that can be monitored and compared between DUT and Whisper.

Sources: [bridge/bridge.h 52-56](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L52-L56)

### Source Type Enumeration

`typedef enum {    k_Dut,    k_Whisper} eSrc;`
Used internally to distinguish between DUT and Whisper data sources in state update functions.

Sources: [bridge/bridge.h 58-61](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/bridge/bridge.h#L58-L61)

## Interface Registration Pattern

The monitoring system uses a registry pattern where interfaces are registered by name and dispatched based on string matching:

`static std::map<std::string, cMonInstr *> intfs;`
This allows multiple monitoring implementations to coexist and be selected at runtime through the `name` parameter in DPI function calls.

Sources: [mon/mon_instr/mon_instr_dpi.cc 10](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L10-L10)
