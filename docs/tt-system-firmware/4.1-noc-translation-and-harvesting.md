This page documents the Network-on-Chip (NOC) translation mechanism that provides fault tolerance by routing around defective hardware tiles discovered during manufacturing testing (harvesting). NOC translation creates a logical coordinate space that remains consistent across different chip variants, while physical coordinates may vary based on which tiles are enabled.

For information about the TLB system used to access NOC nodes, see [NOC2AXI and TLB Configuration](https://deepwiki.com/tenstorrent/tt-system-firmware/4.2-noc2axi-and-tlb-configuration). For overall NOC architecture, see [Network-on-Chip (NOC)](https://deepwiki.com/tenstorrent/tt-system-firmware/4-network-on-chip-(noc)).

## Purpose and Scope

The NOC translation system serves two primary functions:

1.   **Harvesting Support**: Remaps NOC coordinates to route around defective tiles (Tensix columns, GDDR banks, Ethernet channels) identified during manufacturing
2.   **Coordinate Normalization**: Provides a predictable logical coordinate space for software, regardless of which physical tiles are disabled

Without translation, software would need to know exactly which tiles are defective on each chip variant. With translation enabled, software uses logical coordinates (0-N for N working tiles), and the hardware automatically maps these to whichever physical tiles are functional.

## NOC Coordinate System Overview

The Blackhole ASIC has a 17x12 physical NOC coordinate grid (NOC 0 perspective):

| X Range | Y Range | Contents |
| --- | --- | --- |
| 0 | 0-11 | GDDR 0-3 (west) |
| 1-7, 10-16 | 2-11 | Tensix cores |
| 1-7, 10-16 | 1 | Ethernet channels |
| 2, 11 | 0 | PCIe instances |
| 8 | 0-11 | ARC, L2CPU, Security |
| 9 | 0-11 | GDDR 4-7 (east) |

**NOC Translation States**:

*   **Disabled**: Physical coordinates are used directly
*   **Enabled**: Physical coordinates are translated to logical coordinates via lookup tables

The translation can be enabled/disabled via the `NIU_CFG_0.NOC_ID_TRANSLATE_EN` bit in each NOC node's configuration registers.

## Translation Table Structure

**Translation Structure (`struct NocTranslation`)**:

`struct NocTranslation {    bool translate_en;                          // Enable translation    uint8_t translate_table_x[32];              // Pre-X -> Post-X mapping    uint8_t translate_table_y[32];              // Pre-Y -> Post-Y mapping    uint32_t translate_col_mask[1];             // Bitmask: which rows use X translation    uint32_t translate_row_mask[1];             // Bitmask: which columns use Y translation    uint16_t logical_coords[17][12];            // Post -> Logical mapping for overlay};`
The masks control whether translation applies. For example, setting bit 0 and 1 in `translate_row_mask` means X translation does not apply to rows 0 and 1 (PCIe and Ethernet rows).

Sources: [lib/tenstorrent/bh_arc/noc_init.c 275-286](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L275-L286)[lib/tenstorrent/bh_arc/noc_init.c 288-310](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L288-L310)

## Translation Algorithm

### Tensix Column Remapping

The algorithm ensures good Tensix columns appear at logical coordinates 1-7 and 10-16 (the expected positions), while bad columns are pushed to the end:

**Example**: If physical columns 0, 3, 5 are bad (bits 0, 3, 5 set in `bad_tensix_cols`):

| Pre-X | Post-X | Logical Purpose |
| --- | --- | --- |
| 1-7 | Good columns in order (e.g., 1, 2, 4, 6, 7, 10, 11) | First 7 working columns |
| 10-16 | Remaining good + bad at end (e.g., 12, 13, 14, 15, 16, 0, 3) | Next 7 positions |

The remapping logic at [lib/tenstorrent/bh_arc/noc_init.c 476-522](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L476-L522) uses bitmask operations:

1.   Build `good_tensix_noc0_x` bitmask by clearing disabled column bits
2.   Fill pre-X entries 1-7 by popping lowest bits from good mask
3.   Fill pre-X entries 10-16 with remaining good bits
4.   Fill from end with bad column positions

Sources: [lib/tenstorrent/bh_arc/noc_init.c 458-586](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L458-L586)[lib/tenstorrent/bh_arc/noc_init.c 333-338](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L333-L338)

### GDDR Remapping

GDDR controllers are remapped to ensure good banks appear first:

`// Physical GDDR layout (NOC 0):static const uint8_t kGddrY[][3] = {    {0, 1, 11},   // GDDR[0] west at Y: 0,1,11    {2, 10, 3},   // GDDR[1] west at Y: 2,10,3    {9, 4, 8},    // GDDR[2] west at Y: 9,4,8    {5, 7, 6}     // GDDR[3] west at Y: 5,7,6};`
If `bad_gddr != NO_BAD_GDDR`, the algorithm:

1.   Reorders the GDDR indices to move the bad bank to position 3 (highest logical Y)
2.   Places GDDR columns at pre-X 17 (west) and 18 (east), with west first if bad GDDR is in east (≥4)

Sources: [lib/tenstorrent/bh_arc/noc_init.c 523-551](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L523-L551)[lib/tenstorrent/bh_arc/noc_init.c 337-338](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L337-L338)

### Logical Coordinate Assignment

The `ApplyLogicalCoords()` function assigns logical coordinates to each post-translation position:

`// Example: Apply logical coords for Tensix areaApplyLogicalCoords(&noc0,     1, 2, 7, 11,      // Post-translation box (X 1-7, Y 2-11)    1, 2, 16, 11);    // Pre-translation box (X 1-16, Y 2-11)`
This maps pre-translation coordinates within the specified range to their post-translation positions, recording the pre-translation coordinate as the "logical" coordinate for overlay routing.

Sources: [lib/tenstorrent/bh_arc/noc_init.c 359-383](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L359-L383)[lib/tenstorrent/bh_arc/noc_init.c 520-577](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L520-L577)

## Hardware Programming Sequence

**Critical Ordering**: The ARC node must be programmed last and have its translation enabled last. This is because:

1.   ARC uses NOC to program other nodes
2.   If ARC translation is enabled while programming other nodes, ARC's own NOC accesses would be translated, causing incorrect addressing
3.   After all nodes are programmed, ARC can safely enable its own translation

The programming sequence at [lib/tenstorrent/bh_arc/noc_init.c 386-452](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L386-L452) ensures this:

`const unsigned int arc_x = 8;const unsigned int arc_y = (noc_id == 0) ? 0 : NOC0_Y_TO_NOC1(0); for (x, y in all NOC nodes) {    // Program translation tables...        if (translate_en && (x != arc_x || y != arc_y)) {        // Enable for non-ARC nodes        WRITE_BIT(niu_cfg_0, NIU_CFG_0_NOC_ID_TRANSLATE_EN, 1);    }} // Finally enable ARC translation`
Sources: [lib/tenstorrent/bh_arc/noc_init.c 386-452](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L386-L452)[lib/tenstorrent/bh_arc/noc_init.c 649-663](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L649-L663)

## NOC Register Configuration

Each NOC node has configuration registers accessed via TLB window:

| Register Index | Name | Purpose |
| --- | --- | --- |
| 0x0 | `NIU_CFG_0` | Node control (translate_en, clock gating, etc.) |
| 0x1-0x5 | `ROUTER_CFG(0-4)` | Router configuration and broadcast masks |
| 0x6-0xB | `NOC_X_ID_TRANSLATE_TABLE(0-5)` | X coordinate translation tables (6x 32-bit) |
| 0xC-0x11 | `NOC_Y_ID_TRANSLATE_TABLE(0-5)` | Y coordinate translation tables (6x 32-bit) |
| 0x12 | `NOC_ID_LOGICAL` | Logical coordinate for this node |
| 0x14 | `NOC_ID_TRANSLATE_COL_MASK` | Bitmask: which rows use X translation |
| 0x15 | `NOC_ID_TRANSLATE_ROW_MASK` | Bitmask: which columns use Y translation |
| 0x16-0x1B | `DDR_COORD_TRANSLATE_TABLE(0-5)` | DDR-specific translation (not used) |

**NIU_CFG_0 Bit Fields**:

| Bit | Field | Purpose |
| --- | --- | --- |
| 0 | `NIU_CLOCK_GATING_EN` | Enable NIU clock gating |
| 12 | `TILE_CLK_OFF` | Disable tile clock (for harvested tiles) |
| 13 | `TILE_HEADER_STORE_OFF` | Disable NOC2AXI header double-write |
| 14 | `NOC_ID_TRANSLATE_EN` | Enable coordinate translation |
| 15 | `AXI_SLAVE_ENABLE` | Enable AXI slave interface |

The translation tables pack five 5-bit coordinate values into each 32-bit register (with 7 unused bits):

`#define NOC_TRANSLATE_ID_WIDTH 5#define NOC_TRANSLATE_TABLE_XY_SIZE 32  // Total entries across 6 registers for (i = 0; i < 32; i++) {    uint32_t index = i / 6;  // Which register (0-5)    uint32_t shift = (i % 6) * 5;  // Bit position within register    translate_table_x[index] |= (nt->translate_table_x[i] << shift);}`
Sources: [lib/tenstorrent/bh_arc/noc_init.c 29-47](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L29-L47)[lib/tenstorrent/bh_arc/noc_init.c 388-402](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L388-L402)[lib/tenstorrent/bh_arc/noc_init.c 221-273](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L221-L273)

## Broadcast Management

NOC supports broadcast operations to send a single packet to multiple nodes. The broadcast exclusion mechanism ensures broadcasts don't reach disabled tiles:

**Implementation**: `ProgramBroadcastExclusion()` at [lib/tenstorrent/bh_arc/noc_init.c 114-155](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L114-L155) programs each node's router configuration:

`// Always exclude non-Tensix columns (NOC 0)uint32_t router_cfg_1 = BIT(0) | BIT(8) | BIT(9);  // West GDDR, L2CPU, East GDDR // Add disabled Tensix columnsfor (i = 0; i < 14; i++) {    if (disabled_tensix_columns & BIT(i)) {        router_cfg_1 |= BIT(TensixPhysXToNoc(i, 0));    }} // Always exclude non-Tensix rowsuint32_t router_cfg_3 = BIT(0) | BIT(1);  // Ethernet, PCIe`
A node will not receive broadcasts if it's in a disabled row **or** column. This allows efficient broadcast to all working Tensix cores using a single multicast operation.

Sources: [lib/tenstorrent/bh_arc/noc_init.c 114-155](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L114-L155)[lib/tenstorrent/bh_arc/noc_init.c 267-269](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L267-L269)

## TLB Integration for Translation Access

The NOC2AXI TLB system is used to access NOC nodes for programming translation tables:

**TLB Setup Functions**:

*   `SetupNiuTlb(tlb_idx, nx, ny, noc_id)` - Setup TLB for logical coordinates (post-translation)
*   `SetupNiuTlbPhys(tlb_idx, px, py, noc_id)` - Setup TLB for physical coordinates (pre-translation)

During translation initialization, `SetupNiuTlbPhys()` is used because translation is not yet active. The function at [lib/tenstorrent/bh_arc/noc_init.c 60-76](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L60-L76) calculates the NIU register base address and programs the TLB to map a memory window to that address.

Example usage from [lib/tenstorrent/bh_arc/noc_init.c 145-148](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L145-L148):

## Tile Clock Disable for Harvesting

In addition to coordinate remapping, harvested tiles have their clocks disabled to save power:

`static bool GetTileClkDisable(uint8_t px, uint8_t py){    // Tensix columns    if (px >= 1 && px <= 14 && py >= 2) {        uint8_t tensix_x = px - 1;        return !IS_BIT_SET(tile_enable.tensix_col_enabled, tensix_x);    }    // Ethernet tiles    else if (px >= 1 && px <= 14 && py == 1) {        uint8_t eth_inst = px - 1;        return !IS_BIT_SET(tile_enable.eth_enabled, eth_inst);    }    // GDDR west (column 0)    else if (px == 0) {        uint8_t gddr_inst = py / 3;        return !IS_BIT_SET(tile_enable.gddr_enabled, gddr_inst);    }    // GDDR east (column 16)    else if (px == 16) {        uint8_t gddr_inst = 4 + py / 3;        return !IS_BIT_SET(tile_enable.gddr_enabled, gddr_inst);    }    return false;}`
This is programmed during `NocInit()` by setting `NIU_CFG_0.TILE_CLK_OFF` for each disabled tile at [lib/tenstorrent/bh_arc/noc_init.c 250-253](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L250-L253)

## NOC 0 to NOC 1 Mirroring

Blackhole has two NOC rings (NOC 0 and NOC 1) with mirrored coordinate systems:

`static void CopyNoc0ToNoc1(const struct NocTranslation *noc0,                            struct NocTranslation *noc1){    *noc1 = *noc0;        // Flip X coordinates    for (i = 0; i < 32; i++) {        noc1->translate_table_x[i] = NOC_X_SIZE - noc0->translate_table_x[i] - 1;        noc1->translate_table_y[i] = NOC_Y_SIZE - noc0->translate_table_y[i] - 1;    }        // Flip logical coordinates    for (x = 0; x < NOC_X_SIZE; x++) {        for (y = 0; y < NOC_Y_SIZE; y++) {            noc1->logical_coords[x][y] =                 noc0->logical_coords[NOC_X_SIZE-x-1][NOC_Y_SIZE-y-1];        }    }}`
**Coordinate Conversions**:

*   `NOC0_X_TO_NOC1(x) = 16 - x` (NOC_X_SIZE - 1 - x)
*   `NOC0_Y_TO_NOC1(y) = 11 - y` (NOC_Y_SIZE - 1 - y)

This mirroring ensures both NOC rings can access the same logical coordinate space, with NOC 1 providing redundant routing paths for reliability.

Sources: [lib/tenstorrent/bh_arc/noc_init.c 312-327](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L312-L327)[lib/tenstorrent/bh_arc/noc_init.c 594-601](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L594-L601)

## Runtime Translation Control

The global `noc_translation_enabled` flag tracks whether translation is active. This affects NOC operations:

*   **Tensix Broadcast**: When translation is enabled on NOC 1, multicast rectangles run backwards (higher to lower coordinates)
*   **Coordinate Interpretation**: TLB setup functions must use physical vs. logical coordinates appropriately

The `ClearNocTranslation()` function at [lib/tenstorrent/bh_arc/noc_init.c 649-663](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L649-L663) disables translation by:

1.   Directly writing to ARC's NOC registers (bypassing TLB to avoid translation)
2.   Programming an identity translation to all nodes
3.   Clearing the `noc_translation_enabled` flag

Sources: [lib/tenstorrent/bh_arc/noc_init.c 58](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L58-L58)[lib/tenstorrent/bh_arc/noc_init.c 603-606](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc_init.c#L603-L606)[lib/tenstorrent/bh_arc/noc2axi.c 146-177](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.c#L146-L177)
