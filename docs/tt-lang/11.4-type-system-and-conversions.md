This page documents the type conversion and materialization infrastructure in the `tt-lang` compiler, specifically focusing on the `convert-ttl-to-ttkernel` lowering. Type converters transform high-level TTL types (like circular buffers with shape and factor) to low-level TTKernel types (flattened hardware slots), while materialization creates the actual SSA values needed for hardware execution.

## Purpose and Scope

The `TTLToTTKernelTypeConverter` performs two key functions:

1.   **Type Conversion**: Maps TTL dialect types to TTKernel dialect types during pattern-based lowering [lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 157-160](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L157-L160)
2.   **Materialization**: Creates concrete SSA values for abstract type conversions, particularly materializing `CBType` handles from `CircularBufferType` during the lowering of tile operations [lib/Dialect/Utils/ConversionUtils.h 79-109](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/Utils/ConversionUtils.h#L79-L109)

This infrastructure enables the lowering of tensor-level operations in TTL to hardware-specific buffer operations in TTKernel.

## Type Converter Architecture

The conversion system relies on a standard MLIR `TypeConverter` configured with TTL-specific rules. It bridges the gap between the Python DSL's high-level abstractions and the low-level TTKernel IR.

## Type Conversion Rules

The converter implements rules to map the TTL programming model to hardware resources:

### CircularBufferType → CBType

| Source Type (TTL) | Destination Type (TTKernel) | Conversion Logic |
| --- | --- | --- |
| `!ttl.cb<[r, c], type, factor>` | `!ttkernel.cb<elements, type>` | `elements = r * c * factor` |

**Implementation**: The TTL CircularBuffer type carries 2D shape information plus a `block_count` (buffer factor) [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 31-37](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L31-L37) TTKernel's CB type flattens this to a single `total_elements` count. The conversion uses `getTotalElements()` to compute the flattened size [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 46-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L46-L49)[include/ttlang/Dialect/Utils/ConversionUtils.h 95-96](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/Utils/ConversionUtils.h#L95-L96)

### RankedTensorType → TensorAccessorType

Tensors with layout encodings are converted to `TensorAccessorType`, which provides a handle for runtime buffer addressing and metadata.

**Implementation**: During lowering, tensors attached to CBs or DRAM/L1 memory are often materialized as accessors to allow the kernel to calculate offsets for `noc_async_read` or `noc_async_write`[lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 168-175](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L168-L175) The `TTLLayoutTensor` constraint is used to identify ranked tensors with `LayoutAttr` encodings [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 18-25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L18-L25)

### TransferHandleType Preservation

Transfer handles (`!ttl.transfer_handle<read>` or `!ttl.transfer_handle<write>`) are preserved during lowering to track asynchronous state [include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td 52-58](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsTypes.td#L52-L58)

## Materialization Mechanism

Type conversion creates abstract type mappings, but MLIR requires concrete SSA values. Materialization callbacks create `UnrealizedConversionCastOp` operations that serve as placeholders until the conversion is finalized.

## Typecasting and SFPU Support

The `ttl.typecast` and `ttl.tile_typecast` operations handle conversions between floating-point formats (e.g., `bf16` to `f32`).

### Tensor and Tile Typecasting

*   `ttl.typecast`: A tensor-level operation that changes the element type of a tensor. It lowers to a `ttl.compute` block containing a `ttl.tile_typecast`[test/ttlang/Conversion/TTLToCompute/typecast.mlir 3-5](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToCompute/typecast.mlir#L3-L5)
*   `ttl.tile_typecast`: A tile-level operation that converts a tile in a DST register slot. It is lowered to the `ttkernel.typecast_tile` hardware instruction [test/ttlang/Conversion/TTLToTTKernel/typecast.mlir 2-3](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Conversion/TTLToTTKernel/typecast.mlir#L2-L3)

### Validation and Hardware Constraints

The SFPU typecast path only supports floating-point tile data types. Integer inputs or outputs will trigger a verification error [test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir 29-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir#L29-L49) Additionally, input and result tile shapes must match exactly [test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir 7-15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Dialect/TTL/IR/typecast_invalid.mlir#L7-L15)

## Index Conversions and Slicing

During lowering of `ttl.compute` or data movement to loops, the system must handle linearized indices and potential offsets from `tensor.extract_slice`.

### Slice Offset Materialization

The function `addSliceOffset` handles cases where a compute operand traces back to a `tensor.extract_slice`. It delinearizes the local index into coordinates using `affine::AffineDelinearizeIndexOp`, adds the slice's mixed offsets, and relinearizes against the source tensor's shape via `affine::AffineLinearizeIndexOp`[include/ttlang/Dialect/Utils/ConversionUtils.h 38-75](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/Utils/ConversionUtils.h#L38-L75)

### CB Type Conversion Helper

A utility `convertTTLCBToTTKernel` is provided to perform the conversion from `CircularBufferType` to `ttkernel::CBType`[include/ttlang/Dialect/Utils/ConversionUtils.h 79-109](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/Utils/ConversionUtils.h#L79-L109) It is used extensively in `ConvertTTLTileOpsToTTKernel.cpp` to resolve operands before emitting hardware-specific ops like `copy_tile` or SFPU instructions [lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp 157-175](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLTileOpsToTTKernel.cpp#L157-L175)

## Intermediate DFB Materialization

The compiler automatically materializes intermediate Dataflow Buffers (DFBs) when a producer-consumer relationship cannot be fused into a single compute operation.

### Materialization Rules:

*   **Fusion Failures**: If an operation (e.g., `add`) feeds a consumer that cannot fuse (e.g., a `reduce`), the pass inserts a compiler-allocated DFB [docs/development/AccumulatingComputeLowering.md 12-16](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/AccumulatingComputeLowering.md?plain=1#L12-L16)
*   **Accumulation Scopes**: `ttl.accumulation_scope` explicitly describes accumulation semantics, separating the policy from the storage mechanism (DST, L1, or DFB) [docs/development/AccumulatingComputeLowering.md 46-59](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/AccumulatingComputeLowering.md?plain=1#L46-L59)
*   **Shared Materialization**: If multiple consumers read the same non-CB-attached value, the pass creates only one compiler-allocated DFB and reuses it.
