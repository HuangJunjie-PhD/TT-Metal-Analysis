## Purpose and Scope

This document describes the pytest-based testing infrastructure used in tt-tvm. It covers test execution workflows, environment setup, test sharding, parallel execution, result reporting, and error handling mechanisms. For information about the CI pipeline that orchestrates these tests, see [Jenkins Pipeline System](https://deepwiki.com/tenstorrent/tt-tvm/10.1-jenkins-pipeline-system). For details on Docker environments used for testing, see [Docker CI Environments](https://deepwiki.com/tenstorrent/tt-tvm/10.2-docker-ci-environments).

## Testing Architecture Overview

The tt-tvm testing framework is built on pytest and provides a standardized execution environment with the following key features:

*   **Centralized test execution** through the `run_pytest()` function
*   **Test sharding** for parallel execution across multiple workers
*   **Automatic test reruns** for flaky test detection and mitigation
*   **JUnit XML reporting** for integration with Jenkins and other CI systems
*   **FFI testing** with multiple backend types (ctypes, cython)
*   **Error aggregation** to report all failures at the end of test runs

## Test Execution with `run_pytest()`

The `run_pytest()` function provides a standardized interface for executing pytest with proper configuration, result reporting, and error handling.

### Function Signature

`run_pytest <FFI_TYPE> <TEST_SUITE_NAME> [pytest args...]`
### Parameters

| Parameter | Description | Example Values |
| --- | --- | --- |
| `FFI_TYPE` | Foreign Function Interface backend | `ctypes`, `cython` |
| `TEST_SUITE_NAME` | Logical test group name | `unittest`, `frontend-pytorch`, `relay-ops` |
| `[pytest args...]` | Additional pytest arguments | `tests/python/relay`, `-k test_conv2d`, `-m gpu` |

### Execution Flow

**Diagram: run_pytest Function Flow**

### Suite Naming Convention

The function constructs a unique suite name by combining three components:

```
${test_suite_name}-${shard_id}-${ffi_type}
```

Examples:

*   `unittest-no-shard-ctypes`
*   `frontend-pytorch-shard-2-cython`
*   `relay-ops-shard-0-ctypes`

This naming scheme enables:

1.   **Hierarchical organization** in Jenkins test results
2.   **Parallel execution tracking** across shards
3.   **FFI-specific result isolation** for debugging backend issues

## Test Sharding

Test sharding distributes tests across multiple execution contexts to reduce overall test time. The framework uses the `TVM_SHARD_INDEX` environment variable to control which shard a test run belongs to.

### Sharding Configuration

**Diagram: Sharding Mechanism**

### Shard Distribution

When `TVM_SHARD_INDEX` is set, pytest's built-in sharding mechanism (via pytest-xdist) distributes tests across workers. The typical CI configuration uses:

*   Jenkins matrix builds with `TVM_SHARD_INDEX` ranging from 0 to N-1
*   Each shard executes pytest with the `-n` flag for intra-shard parallelism
*   Results are collected separately per shard with unique XML file names

### Example Shard Configuration

For a test suite with 3 shards:

| Shard | Environment Variable | Suite Name | Result File |
| --- | --- | --- | --- |
| 0 | `TVM_SHARD_INDEX=0` | `unittest-shard-0-ctypes` | `unittest-shard-0-ctypes.xml` |
| 1 | `TVM_SHARD_INDEX=1` | `unittest-shard-1-ctypes` | `unittest-shard-1-ctypes.xml` |
| 2 | `TVM_SHARD_INDEX=2` | `unittest-shard-2-ctypes` | `unittest-shard-2-ctypes.xml` |

## Test Reruns and Flaky Test Handling

The framework automatically enables test reruns when the `pytest-rerunfailures` plugin is available, helping to identify and mitigate flaky tests that occasionally fail due to timing issues or environmental factors.

### Rerun Detection and Configuration

**Diagram: Automatic Rerun Configuration**

When enabled, pytest will:

1.   Run failing tests up to 3 additional times
2.   Mark tests as passed if any retry succeeds
3.   Report the test as failed only if all retries fail
4.   Include retry information in the JUnit XML output

This mechanism helps distinguish transient failures from genuine bugs while maintaining test reliability.

Sources: [tests/scripts/setup-pytest-env.sh 71-74](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/scripts/setup-pytest-env.sh#L71-L74)

## Parallel Test Execution

The framework supports parallel test execution through pytest-xdist. By default, it sets parallelism to 1 (`DEFAULT_PARALLELISM=1`) but automatically respects user-specified parallelism options.

### Parallelism Detection

**Diagram: Parallel Execution Configuration**

### Parallelism Strategies

Test scripts can override default parallelism by passing pytest-xdist options:

| Option | Behavior | Use Case |
| --- | --- | --- |
| `-n=1` | Single worker (default) | Debugging, sequential tests |
| `-n=4` | 4 parallel workers | Fast multi-core execution |
| `-n=auto` | Workers = CPU count | Maximize hardware utilization |
| `-dist=loadscope` | Distribute by test scope | Balance test duration |
| `-dist=loadfile` | Distribute by file | Better locality |

## Error Collection and Reporting

The framework aggregates all test failures during execution and reports them collectively at the end, ensuring that all errors are visible even when multiple test suites fail.

### Error Collection Mechanism

**Diagram: Error Collection and Reporting Flow**

### Exit Code Handling

The framework distinguishes between different pytest exit codes:

| Exit Code | Meaning | Action |
| --- | --- | --- |
| 0 | All tests passed | No error recorded |
| 5 | No tests collected | No error recorded (valid scenario) |
| Other | Test failures or errors | Error recorded in `pytest_errors` array |

This handling ensures that empty test collections (common in sharded environments where a shard gets no tests) don't trigger false failures.

### Cleanup Trap Function

`function cleanup() {    set +x    if [ "${#pytest_errors[@]}" -gt 0 ]; then        echo "These pytest invocations failed..."        python3 ci/scripts/jenkins/pytest_wrapper.py "${pytest_errors[@]}"        exit 1    fi    set -x}trap cleanup 0`
The cleanup function:

1.   Executes automatically when the script exits (via `trap`)
2.   Checks if any errors were collected
3.   Invokes `pytest_wrapper.py` to format the error report
4.   Exits with code 1 to signal failure to CI systems

## Test Organization Patterns

Tests in tt-tvm follow consistent organizational patterns that align with the testing framework:

### Directory Structure

```
tests/
├── python/                      # Python test suites
│   ├── relay/                   # Relay IR tests
│   ├── tir/                     # TensorIR tests
│   ├── frontend/                # Frontend converter tests
│   ├── runtime/                 # Runtime system tests
│   └── unittest/                # General unit tests
├── scripts/                     # Test execution scripts
│   ├── setup-pytest-env.sh      # Environment setup
│   ├── task_python_*.sh         # Test suite runners
│   └── task_config_*.sh         # Build configurations
└── lint/                        # Code quality tests
```

### Common Test Invocation Pattern

Test scripts typically follow this pattern:

`#!/usr/bin/env bashsource tests/scripts/setup-pytest-env.sh # Run specific test suiterun_pytest ctypes unittest tests/python/unittest # Run with markersrun_pytest ctypes relay-gpu tests/python/relay -m gpu # Run with custom parallelismrun_pytest ctypes frontend tests/python/frontend -n=4`
This standardization ensures:

*   Consistent error reporting across all test suites
*   Uniform JUnit XML output for Jenkins integration
*   Predictable sharding behavior
*   Reliable result collection
