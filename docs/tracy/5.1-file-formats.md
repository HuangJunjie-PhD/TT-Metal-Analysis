# File Formats

This document covers the Tracy profiling system's native file format used for storing compressed trace data. The Tracy file format (`.tracy` files) is a binary format that efficiently stores profiling data using either LZ4 or Zstd compression algorithms. This format is used both for saving live profiling sessions and for offline analysis.

For information about import/export tools that work with these files, see [Import & Export Tools](https://deepwiki.com/tenstorrent/tracy/5.2-import-and-export-tools). For details about the data processing engine that reads these files, see [Data Processing Engine](https://deepwiki.com/tenstorrent/tracy/4.2-data-processing-engine).

## File Format Structure

The Tracy file format consists of a header followed by a series of compressed data blocks. The format is designed for efficient streaming decompression during loading.

### Header Format

The file begins with a 4-byte magic header that identifies the compression algorithm:

| Compression | Header Bytes | Hex Values |
| --- | --- | --- |
| LZ4 | `'t', 'l', 'Z', 4` | `0x74, 0x6C, 0x5A, 0x04` |
| Zstd | `'t', 'Z', 's', 't'` | `0x74, 0x5A, 0x73, 0x74` |

The header constants are defined in `TracyFileHeader.hpp` and used during file detection in `FileRead::FileRead()`.

Sources: [server/TracyFileHeader.hpp 11-12](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileHeader.hpp#L11-L12)[server/TracyFileRead.hpp 351-363](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileRead.hpp#L351-L363)

### Block Structure

After the header, the file contains a sequence of compressed blocks:

1.   **Block Size** (4 bytes): Size of the compressed data block in bytes
2.   **Compressed Data**: Variable-length compressed data (maximum 64KB when decompressed)

Each block is compressed independently, allowing for efficient random access and streaming decompression. The maximum uncompressed block size is defined as `BufSize = 64 * 1024` bytes.

## Compression Algorithms

Tracy supports multiple compression algorithms with different performance characteristics:

### LZ4 Compression

LZ4 provides fast compression and decompression with moderate compression ratios. Tracy supports two LZ4 variants:

*   **Fast**: Standard LZ4 compression using `LZ4_compress_fast_continue()`
*   **Slow**: LZ4 High Compression using `LZ4_compress_HC_continue()`
*   **Extreme**: LZ4 HC with maximum compression level (`LZ4HC_CLEVEL_MAX`)

### Zstd Compression

Zstd (Zstandard) provides better compression ratios than LZ4 at the cost of slower compression/decompression. Tracy allows configurable compression levels for Zstd.

## File I/O Classes

Tracy provides dedicated classes for reading and writing the file format with optimized buffering and asynchronous decompression.

### FileRead Class

The `FileRead` class handles reading and decompressing Tracy files:

Key features:

*   **Memory mapping**: Uses `mmap()` for efficient file access
*   **Double buffering**: Two 64KB buffers for continuous reading
*   **Background decompression**: Dedicated worker thread for decompression
*   **Template methods**: `Read2()` through `Read10()` for reading multiple values efficiently

Sources: [server/TracyFileRead.hpp 36-43](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileRead.hpp#L36-L43)[server/TracyFileRead.hpp 332-387](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileRead.hpp#L332-L387)[server/TracyFileRead.hpp 397-416](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileRead.hpp#L397-L416)

### FileWrite Class

The `FileWrite` class handles writing and compressing Tracy files:

Key features:

*   **Configurable compression**: Supports LZ4 fast/slow/extreme and Zstd
*   **Compression statistics**: Tracks source vs compressed byte counts
*   **Block-based writing**: Automatically flushes when buffer is full
*   **Stream compression**: Maintains compression dictionaries across blocks

## Version Handling

Tracy files include version information for compatibility checking:

`static constexpr tracy_force_inline int FileVersion( uint8_t h5, uint8_t h6, uint8_t h7 ){    return ( h5 << 16 ) | ( h6 << 8 ) | h7;}`
The version is encoded using three bytes, allowing for major.minor.patch version numbering. This enables the profiler to handle older file formats and provide appropriate compatibility warnings.

Sources: [server/TracyFileHeader.hpp 14-17](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileHeader.hpp#L14-L17)

## Error Handling

The file I/O system defines specific exceptions for error conditions:

| Exception | Description | Thrown When |
| --- | --- | --- |
| `NotTracyDump` | Invalid file header | Header doesn't match LZ4 or Zstd magic bytes |
| `FileReadError` | File system error | `stat()` fails or `mmap()` fails |

These exceptions allow the application to distinguish between format errors and system-level I/O errors.
