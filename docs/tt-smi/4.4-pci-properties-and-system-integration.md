# PCI Properties and System Integration

## Purpose and Scope

This page documents how TT-SMI detects and reports PCI Express (PCIe) properties for Tenstorrent devices through the Linux sysfs interface. This includes link speed, link width, and generation detection for PCIe Gen 1 through Gen 5. The system reads hardware characteristics from kernel-exposed interfaces and integrates them into device information reporting.

For information about broader device detection and enumeration, see [Supported Platforms and Board Types](https://deepwiki.com/tenstorrent/tt-smi/4.1-supported-platforms-and-board-types). For telemetry collection from devices, see [Telemetry Collection and Monitoring](https://deepwiki.com/tenstorrent/tt-smi/4.2-telemetry-collection-and-monitoring). For the GUI display of device information, see [Device Information Tab](https://deepwiki.com/tenstorrent/tt-smi/3.2-device-information-tab).

## PCI Properties Overview

TT-SMI collects four primary PCIe properties for each device through the `get_pci_properties` method:

| Property | Description | Source |
| --- | --- | --- |
| `current_link_speed` | Active PCIe link speed (GT/s) | `/sys/bus/pci/devices/{bdf}/current_link_speed` |
| `max_link_speed` | Maximum supported link speed (GT/s) | `/sys/bus/pci/devices/{bdf}/max_link_speed` |
| `current_link_width` | Active number of PCIe lanes | `/sys/bus/pci/devices/{bdf}/current_link_width` |
| `max_link_width` | Maximum supported lane count | `/sys/bus/pci/devices/{bdf}/max_link_width` |

These properties are defined in the `PCI_PROPERTIES` constant list and are collected during backend initialization for each non-remote device.

## Sysfs Integration

### PCI Device Path Resolution

The system locates PCIe device information by constructing the sysfs path from the device's Bus:Device.Function (BDF) identifier:

1.   **BDF Retrieval**: The `get_pci_bdf()` method retrieves the device's BDF identifier (format: `domain:bus:device.function`)
2.   **Path Construction**: The path `/sys/bus/pci/devices/{bdf}` is constructed and resolved to its real path using `os.path.realpath()`
3.   **Property Files**: Individual property files within this directory are read to extract PCIe characteristics

**Implementation Details:**

```
try:
    pcie_bdf = self.devices[board_num].get_pci_bdf()
    pci_bus_path = os.path.realpath(f"/sys/bus/pci/devices/{pcie_bdf}")
except:
    return {prop: "N/A" for prop in constants.PCI_PROPERTIES}
```

The implementation gracefully handles failures by returning "N/A" for all properties if the device path cannot be resolved.

**Sources:**[tt_smi/tt_smi_backend.py 316-319](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L316-L319)

### Property File Reading

Each PCIe property is read from a dedicated sysfs file within the device's directory:

The regular expression `r"\d+\.\d+|\d+"` extracts numeric values, supporting both integer (e.g., "16") and decimal (e.g., "16.0") formats from the sysfs output.

## PCIe Generation Detection

### Link Speed to Generation Mapping

The `get_pcie_gen` function converts link speed (in GT/s) to PCIe generation numbers. This mapping follows PCIe specification standards:

| Link Speed (GT/s) | PCIe Generation | Bandwidth per Lane |
| --- | --- | --- |
| 2.5 | Gen 1 | ~250 MB/s |
| 5.0 | Gen 2 | ~500 MB/s |
| 8.0 | Gen 3 | ~1 GB/s |
| 16.0 | Gen 4 | ~2 GB/s |
| 32.0 | Gen 5 | ~4 GB/s |

### Implementation

The function performs exact string matching for known link speeds and asserts on unrecognized values, ensuring that only valid PCIe generations are reported.

**Sources:**[tt_smi/tt_smi_backend.py 321-333](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L321-L333)

## Link Width Detection

### Supported Lane Configurations

TT-SMI detects both current and maximum PCIe lane widths from sysfs. Common configurations include:

*   **x1**: Single lane (typically low-bandwidth devices)
*   **x4**: Four lanes
*   **x8**: Eight lanes
*   **x16**: Sixteen lanes (maximum for standard PCIe slots)

The system defines `MAX_PCIE_WIDTH = 16` as the maximum expected width for validation purposes.

### Width Value Extraction

Width values are extracted using the same regex pattern as link speeds: `r"\d+\.\d+|\d+"`. The first matched numeric value from the sysfs file content is used:

```
value = re.findall(r"\d+\.\d+|\d+", output)[0]
```

If extraction fails or the file cannot be read, the value is set to "N/A".

## Integration with Device Information

### Device Info Structure

PCI properties are integrated into the broader device information structure through the `get_device_info` method. Specifically, `pcie_speed` and `pcie_width` fields are populated from the collected PCI properties:

The mapping is straightforward:

*   `dev_info["pcie_speed"]` = `self.pci_properties[board_num]["current_link_speed"]`
*   `dev_info["pcie_width"]` = `self.pci_properties[board_num]["current_link_width"]`

## Constants and Configuration

### System-Wide PCI Constants

The system defines several constants related to PCIe properties:

| Constant | Value | Purpose |
| --- | --- | --- |
| `PCI_PROPERTIES` | `["current_link_speed", "max_link_speed", "current_link_width", "max_link_width"]` | List of properties to collect from sysfs |
| `MAX_PCIE_WIDTH` | `16` | Maximum expected PCIe lane width |
| `MAX_PCIE_SPEED` | `4` | Maximum expected PCIe generation (Gen 4 at time of definition) |

Note: While `MAX_PCIE_SPEED` is defined as 4, the `get_pcie_gen` function supports Gen 5 (value 5), indicating the constant may be outdated or used for different purposes than validation.

## Error Handling and Robustness

### Exception Management

The PCI property collection system implements multiple layers of error handling:

1.   **Remote Device Check**: Early return with "N/A" values for remote devices
2.   **BDF Retrieval**: Try-except block around `get_pci_bdf()` call
3.   **Per-Property Reading**: Individual try-except blocks for each property file read
4.   **Regex Extraction**: Safe indexing with default "N/A" on extraction failure

This defensive approach ensures that PCI property collection failures do not prevent the rest of the system from functioning, particularly important when dealing with diverse hardware configurations or partial system states.
