# Container Configuration

 Relevant source files

* [docs/container.md](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1)
* [infra/singularity/container.py](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py)

This document describes the `.container_config` JSON file used to configure the Singularity container environment in RiESCUE. This file controls registry settings for remote container storage, bind mount configurations, and authentication credentials.

For information about building containers, see [Container Definition and Build](/tenstorrent/riescue/3.1.1-container-definition-and-build). For information about running container management commands, see [Container Management Commands](/tenstorrent/riescue/3.1.2-container-management-commands).

## Overview

The `.container_config` file is an optional JSON configuration file that provides runtime settings for the `Container` class. When present, it enables:

* Remote container registry integration for pulling/pushing container images
* Custom bind mount directories beyond the default `/usr/lib64:/shared`
* Authentication tokens for private registries

If no `.container_config` file is found, the container system operates in local-only mode, using only the local `.sif` file without remote capabilities.

**Sources:** [infra/singularity/container.py13-37](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L13-L37) [docs/container.md16-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L16-L26)

## Configuration File Location

The `Container` class searches for `.container_config` in the following order, using the first file found:

**Configuration File Search Order:**

| Priority | Location | Description |
| --- | --- | --- |
| 1 | `$PWD/.container_config` | Current working directory |
| 2 | `{repo_path}/.container_config` | Repository root |
| 3 | `{repo_path}/infra/.container_config` | Infra directory |

This search order allows per-project configurations to override repository-wide defaults.

## Configuration Options

### Registry Configuration

Registry settings enable remote container storage and distribution, allowing teams to share pre-built containers without rebuilding locally.

#### registry\_uri

The OCI/ORAS registry URI where container images are stored. This is used as the target for `pull` and `push` operations.

**Format:** `oras://registry.example.com/path/to/repository`

**Usage in Code:**

* [infra/singularity/container.py55](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L55-L55) - Loaded from config

#### registry\_remote

The name of the Singularity remote endpoint. This must match a remote configured in Singularity's remote list or will be added automatically during operations.

### Bind Mounts

The `binds` field specifies additional directory bind mounts beyond the default `/usr/lib64:/shared` binding.

**Format:** Comma-separated list of bind mount specifications. Each specification can be:

* `/host/path` - Bind to same path in container
* `/host/path:/container/path` - Bind to different path in container

**Example:** `"/data:/data,/scratch:/scratch,/home/user/projects"`

**Bind Mount Processing:**

1. Default bind: `/usr/lib64:/shared` is always included [infra/singularity/container.py41](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L41-L41)
2. Config binds: Added from `.container_config` if present [infra/singularity/container.py57-59](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L57-L59)
3. CLI binds: Added via `--singularity-binds` argument [infra/singularity/container.py127-128](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L127-L128)
4. Path validation: Only binds where the host path exists are included [infra/singularity/container.py148-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L148-L156)

**Sources:** [infra/singularity/container.py41](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L41-L41) [infra/singularity/container.py57-59](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L57-L59) [infra/singularity/container.py148-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L148-L156) [docs/container.md25-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L25-L26)

## Configuration Examples

### Minimal Configuration (Local Only)

If no `.container_config` file exists, the container operates in local mode:

* Uses only local `.sif` file at `infra/riescue.sif`
* No remote pull/push capabilities
* Default bind: `/usr/lib64:/shared`

### Full Configuration with Remote Registry

This configuration enables:

* Remote container pull/push to GitHub Container Registry
* Custom bind mounts for `/data` and `/mnt/scratch`
* Token-based authentication

### Configuration with Custom Binds Only

This configuration:

* Operates in local mode (no remote capabilities)
* Adds custom bind mounts to the default `/usr/lib64:/shared`
* Final bind string: `/usr/lib64:/shared,/home/user/projects:/projects,/datasets:/datasets`

**Sources:** [docs/container.md16-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L16-L26)

## Configuration Usage by Container Operations

Different `Container` class methods interact with configuration fields in specific ways:

### build() Method

When `registry_uri` and `registry_remote` are configured:

* After building, prompts user to push to registry [infra/singularity/container.py101-112](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L101-L112)
* Calls `push()` if user confirms

When not configured:

* Skips push step [infra/singularity/container.py101-103](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L101-L103)

### run() Method

When `registry_uri` and `registry_remote` are configured:

* If local `.sif` doesn't exist, automatically calls `pull()` [infra/singularity/container.py134-139](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L134-L139)
* If local version mismatches container version, pulls latest [infra/singularity/container.py140-146](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L140-L146)

When not configured:

* If local `.sif` doesn't exist, raises `RuntimeError` and instructs user to build [infra/singularity/container.py135-137](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L135-L137)
* If version mismatch, rebuilds locally [infra/singularity/container.py141-143](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L141-L143)

The `binds` configuration affects all runs:

* Applied via `--bind` flag to `singularity exec` [infra/singularity/container.py158](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L158-L158)

### pull() Method

Requires both `registry_uri` and `registry_remote`:

* Checks remotes and adds if necessary via `_check_singularity_remotes()` [infra/singularity/container.py185](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L185-L185)
* Pulls from `{registry_uri}:{container_string}` [infra/singularity/container.py186](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L186-L186)

### push() Method

Requires `registry_uri`:

* Pushes to `{registry_uri}:{version_string}` [infra/singularity/container.py176](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L176-L176)

**Sources:** [infra/singularity/container.py85-112](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L85-L112) [infra/singularity/container.py114-170](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L114-L170) [infra/singularity/container.py172-176](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L172-L176) [infra/singularity/container.py178-187](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L178-L187)

## Environment Variable Integration

After loading `.container_config`, the `Container` class also loads environment variables from `.env` file:

The `_load_dotenv()` method:

* Reads `.env` file from repository root [infra/singularity/container.py73](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L73-L73)
* Sets environment variables for current process [infra/singularity/container.py291-303](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L291-L303)
* Special handling for `RV_TOOLCHAIN` to append to `PATH` [infra/singularity/container.py75-82](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L75-L82)

This allows `.env` to configure the RISC-V toolchain location, which affects how the `Toolchain` class (see [RISC-V Toolchain Integration](/tenstorrent/riescue/4.1-risc-v-toolchain-integration)) discovers executables.

**Sources:** [infra/singularity/container.py73-82](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L73-L82) [infra/singularity/container.py291-303](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L291-L303)

## Configuration Validation

The `Container` class performs minimal validation on configuration:

| Validation | Location | Error Handling |
| --- | --- | --- |
| JSON parsing | [infra/singularity/container.py54](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L54-L54) | Implicit Python JSON error |
| Token file existence | [infra/singularity/container.py66](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L66-L66) | File I/O error if missing |
| Bind path existence | [infra/singularity/container.py154](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L154-L154) | Silent skip of non-existent paths |
| Registry fields for operations | [infra/singularity/container.py101-103](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L101-L103) [infra/singularity/container.py135-137](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L135-L137) [infra/singularity/container.py180-181](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L180-L181) | Method-specific handling |

The system is designed to degrade gracefully:

* Missing config file → local-only mode
* Missing bind paths → skipped from bind list
* Missing registry fields → operations requiring them are disabled or error
