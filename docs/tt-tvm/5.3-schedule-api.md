## Purpose and Scope

The Schedule API provides the primary user-facing interface for transforming TensorIR (TIR) programs through a rich set of transformation primitives. It enables optimization of tensor computations by manipulating loop structures, memory hierarchies, and computation locations while preserving program semantics. This interface is used both for manual optimization and as the foundation for automated tuning systems.

For information about the internal state representation used by the Schedule API, see [Schedule State Management](https://deepwiki.com/tenstorrent/tt-tvm/5.4-schedule-state-management). For details on the two schedule implementation modes, see [ConcreteSchedule vs TracedSchedule](https://deepwiki.com/tenstorrent/tt-tvm/5.5-concreteschedule-vs-tracedschedule). For the DSL used to define TIR programs, see [TVMScript DSL](https://deepwiki.com/tenstorrent/tt-tvm/5.6-tvmscript-dsl).

## Schedule Class Architecture

The Schedule system is built around an abstract base class with two primary implementations that serve different use cases:

The `ScheduleNode` abstract class defines the interface for all transformation operations, while the two implementations provide different execution semantics:

*   **ConcreteSchedule**: Applies transformations immediately to the IR, optimized for direct manipulation
*   **TracedSchedule**: Records all operations into a trace for serialization, replay, and debugging

## Random Variables

The Schedule API uses **random variables** (RVs) as opaque handles to reference IR elements. These provide stability across transformations since the underlying IR structure changes:

| Type | Purpose | Maps To |
| --- | --- | --- |
| `BlockRV` | References a TIR block | `Block` (via `StmtSRef`) |
| `LoopRV` | References a for loop | `For` (via `StmtSRef`) |
| `ExprRV` | References an integer expression | `PrimExpr` (integer value) |

Random variables enable the Schedule API to maintain stable references even as the underlying IR is transformed. Users interact only with RVs, never directly with IR nodes.

**Example lookup operations:**

*   `sch.get(block_rv)` → returns the `Block` object
*   `sch.get_sref(block_rv)` → returns the `StmtSRef` for internal operations
*   `sch.remove_rv(block_rv)` → removes the mapping from the symbol table

## Schedule Initialization and Configuration

Creating a schedule requires an IRModule (or PrimFunc) and accepts several configuration options:

**Key parameters:**

| Parameter | Purpose | Values |
| --- | --- | --- |
| `debug_mask` | Controls verification after transformations | `"none"`, `"all"`, or bitmask |
| `error_render_level` | Detail level for error messages | `"detail"`, `"fast"`, `"none"` |
| `enable_check` | Enable/disable prerequisite checks | `True` (safe), `False` (fast) |

The schedule automatically identifies the working function (defaults to `"main"` or the only function in the module). Use `work_on(func_name)` to switch between functions in multi-function modules.

## Transformation Primitive Categories

The Schedule API provides over 30 transformation primitives organized into functional categories:

### Loop Transformation Primitives

**Loop transformation primitives:**

| Primitive | Signature | Purpose |
| --- | --- | --- |
| `split` | `split(loop, factors) → [loops]` | Split loop into nested loops by factors |
| `fuse` | `fuse(*loops) → loop` | Fuse consecutive loops into one |
| `merge` | `merge(*loops) → loop` | Merge loops at same level (stricter than fuse) |
| `reorder` | `reorder(*loops)` | Change loop nesting order |
| `add_unit_loop` | `add_unit_loop(block/loop) → loop` | Add a loop with extent 1 |

**Sources:**[python/tvm/tir/schedule/schedule.py 600-877](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/tir/schedule/schedule.py#L600-L877)[include/tvm/tir/schedule/schedule.h 326-390](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/schedule/schedule.h#L326-L390)[src/tir/schedule/primitive.h 201-272](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/primitive.h#L201-L272)

### Loop Annotation Primitives

These primitives change loop execution semantics without altering the computation:

| Primitive | Effect | Requirements |
| --- | --- | --- |
| `parallel` | Mark loop for parallel execution | Block must be complete/reduction, affine binding |
| `vectorize` | Mark loop for SIMD vectorization | Block must be complete/reduction, affine binding |
| `unroll` | Fully unroll loop at compile time | None |
| `bind` | Bind loop to GPU thread axis | Block must be complete/reduction |

### Cache Insertion Primitives

Cache primitives create intermediate buffers in different memory scopes:

**Cache primitives:**

| Primitive | Signature | Purpose |
| --- | --- | --- |
| `cache_read` | `cache_read(block, buffer_index, scope) → block` | Insert read cache stage |
| `cache_write` | `cache_write(block, buffer_index, scope) → block` | Insert write cache stage |
| `cache_inplace` | `cache_inplace(block, buffer_index, scope) → [blocks]` | In-place cache transformation |
| `reindex_cache_read` | `reindex_cache_read(block, index, scope, index_map) → block` | Cache with layout transformation |
| `reindex_cache_write` | `reindex_cache_write(block, index, scope, index_map) → block` | Write cache with layout transformation |

**Common storage scopes:**

*   `"global"` - Global memory (default)
*   `"shared"` - GPU shared memory
*   `"local"` - Thread-local/register memory
*   `"global.texture"` - Texture memory (GPU)

### Compute Location Primitives

These primitives control where and when computations occur:

| Primitive | Signature | Purpose |
| --- | --- | --- |
| `compute_at` | `compute_at(block, loop, preserve_unit=True)` | Move block under specified loop |
| `reverse_compute_at` | `reverse_compute_at(block, loop, preserve_unit=True)` | Move consumer block |
| `compute_inline` | `compute_inline(block)` | Inline block into its consumers |
| `reverse_compute_inline` | `reverse_compute_inline(block)` | Inline block into its producers |

**Special loop values for `compute_at`:**

*   Normal `LoopRV`: Move block under that loop
*   `InlineMark`: Equivalent to `compute_inline(block)`
*   `RootMark`: Move to root scope (no-op)

## Query and Navigation Methods

The Schedule API provides methods to query and navigate the IR structure:

**Key query methods:**

| Method | Returns | Purpose |
| --- | --- | --- |
| `get_block(name, func_name=None)` | `BlockRV` | Find block by name in current/specified function |
| `get_loops(block)` | `[LoopRV]` | Get parent loops from outer to inner |
| `get_child_blocks(block/loop)` | `[BlockRV]` | Get all leaf blocks under block/loop |
| `get_producers(block)` | `[BlockRV]` | Get blocks that produce consumed buffers |
| `get_consumers(block)` | `[BlockRV]` | Get blocks that consume produced buffers |
| `get_output_blocks(scope_block)` | `[BlockRV]` | Get blocks writing to non-allocated buffers |

## Sampling Methods for Auto-Tuning

The Schedule API includes sampling primitives used by auto-tuning systems to explore the optimization space:

| Method | Purpose | Use Case |
| --- | --- | --- |
| `sample_categorical(candidates, probs)` | Sample from discrete distribution | Choose optimization strategy |
| `sample_perfect_tile(loop, n, max_innermost)` | Sample tile factors for loop | Tiling parameters |
| `sample_partitioned_tile(loop, n, pos, factor)` | Sample tiles with partition constraint | GPU thread block sizing |
| `sample_compute_location(block)` | Sample where to compute block | Fusion decisions |

These methods support both random sampling (when `decision=None`) and deterministic replay (when `decision` is provided), enabling reproducibility in auto-tuning workflows.

## Error Handling and Validation

The Schedule API performs extensive validation to prevent invalid transformations:

**Error rendering levels:**

*   **detail**: Full error message with TIR code and highlighted locations
*   **fast**: Simple error message without IR rendering
*   **none**: Minimal error (exception only)

**Common error types:**

*   `NotSingleResult`: Block name resolves to zero or multiple blocks
*   `IncompleteBlockError`: Block violates complete block definition
*   `NotAffineBindingError`: Block bindings are not affine expressions
*   `NotStagePipelineError`: Scope does not have stage pipeline property

The `enable_check` parameter can disable some prerequisite checks for performance, but users must ensure transformation correctness manually.
