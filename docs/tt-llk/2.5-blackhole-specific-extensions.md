This page documents hardware features and API extensions that are exclusive to the Blackhole architecture and not available in Wormhole B0. These extensions provide enhanced synchronization control, instruction optimization capabilities, and hardware introspection mechanisms.

For core hardware abstraction functions shared across all architectures, see [Hardware Abstraction Layer](https://deepwiki.com/tenstorrent/tt-llk/2.1-hardware-abstraction-layer). For synchronization primitives available on all architectures, see [Synchronization Primitives](https://deepwiki.com/tenstorrent/tt-llk/2.2-synchronization-primitives).

## TTSYNC Hardware Synchronization

Blackhole introduces TTSYNC, a hardware-assisted synchronization mechanism that automatically manages memory ordering between TRISC operations and Tensix instructions. When enabled, TTSYNC eliminates the need for explicit `tensix_sync()` or `sync_regfile_write()` calls by automatically stalling TRISC memory accesses and Tensix instructions when there are potential ordering conflicts with tracked resources.

On Wormhole B0, kernels must manually insert synchronization primitives whenever TRISC code modifies configuration registers or GPRs that are subsequently accessed by Tensix instructions. TTSYNC automates this synchronization, improving code clarity and reducing the risk of synchronization bugs.

### Synchronization Control Flags

The TTSYNC system exposes five control flags that determine which hardware resources are tracked for automatic synchronization:

| Flag | Bit Position | Purpose |
| --- | --- | --- |
| `TRACK_GLOBAL_CFG` | 0 | Track global configuration register accesses |
| `EN_SUBDIVIDED_CFG_FOR_UNPACR` | 1 | Enable fine-grained CFG tracking (unpacker 0, unpacker 1, others) |
| `TRACK_GPR` | 2 | Track general-purpose register accesses |
| `TRACK_TDMA` | 3 | Track TDMA (Tensor DMA) operations |
| `TRACK_TENSIX_INSTRUCTIONS` | 4 | Track all Tensix instruction execution |
| `TRACK_ALL` | 0x1F | Enable all tracking modes |

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 474-479](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L474-L479)

### Configuration Interface

**Diagram: TTSYNC Configuration Flow**

The `set_ttsync_enables<bitmask>()` template function accepts a compile-time bitmask specifying which resources to track. Static assertion validates that only defined flag bits are set. When enabled, the hardware automatically stalls TRISC memory accesses and Tensix instructions to guarantee correct ordering.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 483-488](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L483-L488)

### Subdivided CFG Tracking

When `EN_SUBDIVIDED_CFG_FOR_UNPACR` is enabled, configuration register tracking is subdivided into three categories:

*   Unpacker 0 configuration registers
*   Unpacker 1 configuration registers
*   All other configuration registers

This granularity prevents unnecessary synchronization between independent unpacker operations, allowing unpacker 0 Tensix instructions to proceed while TRISC modifies unpacker 1 configuration.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 465-472](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L465-L472)

## Instruction Gathering Control

Blackhole provides explicit control over instruction gathering, a pipeline optimization that combines multiple instructions for more efficient execution. Disabling gathering is required when loading replay buffers to ensure instructions are captured individually.

### Gathering Control Functions

**Diagram: Instruction Gathering Control Flow**

The `disable_gathering<add_nops>()` function performs a careful sequence of CSR operations with memory fences to ensure gathering is disabled before subsequent instructions execute. The optional `add_nops` template parameter (default `true`) inserts three NOPs to account for pipeline depth, ensuring the CSR change propagates before replay buffer loading instructions.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 490-514](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L490-L514)

### Implementation Details

The disable sequence uses inline assembly to manipulate CSR 0x7c0:

1.   Set bit 1 as a memory fence preparation
2.   Execute fence instruction
3.   Set bit 18 to disable gathering
4.   Clear bit 1 to complete fence
5.   Execute fence instruction
6.   Optionally insert three NOPs for pipeline flushing

The enable sequence simply clears bit 18 via inline assembly.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 493-513](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L493-L513)

## Replay Buffer Loading System

Blackhole extends the replay buffer system with a unified interface for recording instruction sequences, integrating with the `lltt` (Low-Level Tensix Tools) library. This system allows runtime-programmable instruction sequences with optional immediate execution.

### Replay Buffer API

**Diagram: Replay Buffer Loading Workflow**

The `load_replay_buf<Exec, Callable, Args>()` template function provides a type-safe interface for loading replay buffers. It accepts:

*   `Exec`: `ExecBool::NoExec` (record only) or `ExecBool::Exec` (record and execute immediately)
*   `start`: Starting offset in replay buffer
*   `len`: Number of instructions to record
*   `callable`: Function/lambda containing instructions to record
*   `args...`: Arguments forwarded to callable

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 516-548](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L516-L548)

### Execution Modes

The `ExecBool` enum provides two modes:

| Mode | Value | Behavior |
| --- | --- | --- |
| `NoExec` | `false` | Record instructions without executing |
| `Exec` | `true` | Record instructions and execute immediately |

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 523-527](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L523-L527)

### Gathering Integration

When the `ENABLE_GATHERING` build flag is defined, the function automatically:

1.   Disables gathering before recording (`disable_gathering()`)
2.   Records instructions via `lltt::record<>`
3.   Re-enables gathering after recording (`enable_gathering()`)

This integration is disabled by default due to issue tt-metal#16439, but can be enabled at build time.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 533-546](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L533-L546)

### Usage Constraints

The replay buffer loading interface is only available when compiling for TRISC, enforced by the `#if defined(COMPILE_FOR_TRISC)` guard. The function uses `[[gnu::always_inline, gnu::flatten]]` attributes to ensure the callable's instructions are properly inlined into the recording context.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 516-548](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L516-L548)

## CSR Access Interface

Blackhole exposes Control and Status Registers (CSRs) for hardware introspection, providing read-only access to queue status, busy status, and stream state information.

### CSR Register Map

**Diagram: CSR Register Map and Access Patterns**

The `CSR` enum class provides strongly-typed identifiers for available CSRs. Two `csr_read<>()` template overloads support both type-safe access (using `CSR` enum) and raw access (using `uint16_t` CSR number).

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 550-562](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L550-L562)

### CSR Read Interface

Both overloads support an optional `fence` template parameter (default `true`) that controls whether a memory fence instruction precedes the CSR read. The fence ensures all prior memory operations complete before the CSR read executes.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 564-592](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L564-L592)

## Hardware Status Monitoring

Blackhole provides structured access to hardware queue and busy status through union types that decode CSR bitfields.

### Queue Status Structure

**Diagram: Queue Status Union Bitfield Layout**

The `qstatus_u` union provides structured access to the `tensix_queue_status` CSR. It contains two sets of flags:

*   **Local queue status** (bits 0-12): Status of current thread's hardware queues
*   **Global queue status** (bits 13-24): Status of all threads' hardware queues

Each set tracks 11 distinct queue types (replay, MOP, THCON, XMOV, unpack, pack, cfg, sync, TDMA, SFPU, FPU) plus a 2-bit SFPU condition code field.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 594-626](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L594-L626)

### Busy Status Structure

**Diagram: Busy Status Union Bitfield Layout**

The `bstatus_u` union provides structured access to the `tensix_busy_status` CSR. Similar to queue status, it distinguishes between:

*   **Local busy status** (bits 0-10): Whether current thread's hardware units are busy
*   **Global busy status** (bits 11-21): Whether any thread's hardware units are busy

Each set tracks 11 hardware units without the SFPUCC field present in queue status.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 628-658](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L628-L658)

## Sign Magnitude Conversion Workaround

Blackhole includes a specialized function to work around an RTL bug when performing sign magnitude conversions for certain data formats.

### Conversion Function

**Diagram: Sign Magnitude Conversion Sequence**

The `apply_sign_magnitude_conversion()` function performs a two-step operation:

1.   `TTI_SFPCAST`: Convert data from source to destination register using specified cast mode
2.   `TTI_SFPSETSGN`: Correct the sign bit in the destination using the original source

The second instruction is required due to a Blackhole RTL bug that can corrupt the sign bit during casting operations. The immediate value (`0`) and modifier (`0`) parameters to `TTI_SFPSETSGN` configure it to simply copy the sign from the source register.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 678-683](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L678-L683)

### Function Signature

This function should be used instead of raw `TTI_SFPCAST` instructions when performing sign magnitude conversions on Blackhole to ensure correct results.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 678-683](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L678-L683)

## Architecture Differences Summary

The following table summarizes key API differences between Blackhole and Wormhole B0:

| Feature | Blackhole | Wormhole B0 |
| --- | --- | --- |
| TTSYNC control | `set_ttsync_enables<>()` with 5 tracking flags | Not available |
| Instruction gathering | `disable_gathering<>()`, `enable_gathering()` | Not available |
| Replay buffer loading | `load_replay_buf<>()` with `lltt` integration | Manual implementation required |
| CSR access | `csr_read<>()` with `CSR` enum | Not available |
| Status monitoring | `qstatus_u`, `bstatus_u` unions | Not available |
| Sign conversion | `apply_sign_magnitude_conversion()` workaround | Not needed |
| `lltt.h` header | Included for TRISC builds | Not available |
| `ZEROACC` signature | 5 parameters: `TT_ZEROACC(p_zeroacc::CLR_ALL, 0, 0, ADDR_MOD_1, 0)` | 3 parameters: `TT_ZEROACC(p_zeroacc::CLR_ALL, ADDR_MOD_1, 0)` |
