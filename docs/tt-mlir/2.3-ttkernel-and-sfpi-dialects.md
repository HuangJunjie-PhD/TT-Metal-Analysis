## Purpose and Scope

The **TTKernel** dialect represents low-level hardware kernel operations that execute directly on Tenstorrent Tensix or Blackhole cores. It provides an MLIR abstraction for hardware resources including **Circular Buffers (CB)**, **DST registers** (a 16-tile scratchpad), the **FPU** (Floating Point Unit), and the **SFPU** (Special Function Processing Unit).

The **SFPI (SFPU Programming Interface)** dialect provides direct access to SFPU hardware via GCC builtins, allowing for high-performance element-wise math and logic within kernels.

**Position in Compilation Stack:**

*   **Input**: TTKernel receives lowered operations from the **D2M (Data-to-Metal)** dialect via the `ConvertD2MToTTKernel` pass [lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp 5-34](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp#L5-L34)
*   **Output**: TTKernel operations are converted to the **EmitC** dialect via `ConvertTTKernelToEmitC`[lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp 4-41](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp#L4-L41) which are then translated to C++ kernel source code using `translateKernelFuncToCpp`[lib/Target/TTKernel/TTKernelToCpp.cpp 5-34](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/TTKernel/TTKernelToCpp.cpp#L5-L34)

**Sources:**[include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 1-17](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L1-L17)[lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp 5-34](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp#L5-L34)[lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp 4-41](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp#L4-L41)[lib/Target/TTKernel/TTKernelToCpp.cpp 5-34](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/TTKernel/TTKernelToCpp.cpp#L5-L34)

## Overview and Compilation Flow

TTKernel serves as the final MLIR layer before source code generation. It maps high-level tensor operations into the specific execution phases required by the hardware (Init, Unpack, Math, Pack).

**Diagram: D2M to TTKernel to C++ Pipeline**

The conversion from D2M to TTKernel uses specialized rewriters to map D2M generic region ops into TTKernel **Init** and **Compute** pairs. Init operations are often hoisted to the start of the function using `setInsertionPointToFuncStart`.

**Sources:**[lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp 5-166](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/D2MToTTKernel/D2MToTTKernel.cpp#L5-L166)[lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp 157-166](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTKernelToEmitC/TTKernelToEmitC.cpp#L157-L166)[include/ttmlir/Dialect/D2M/IR/D2MGenericRegionOps.td 151-162](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/IR/D2MGenericRegionOps.td#L151-L162)

## Circular Buffer (CB) Operations

Circular Buffers are the primary data interface for kernels. They are managed via a producer-consumer synchronization protocol. Verification ensures these operations reside within appropriate execution regions [lib/Dialect/TTKernel/IR/TTKernelOps.cpp 42-72](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTKernel/IR/TTKernelOps.cpp#L42-L72)

**Diagram: Circular Buffer Control Flow**

Key operations include:

*   `CBPushBackOp`: Increments the write pointer and signals the consumer [lib/Dialect/TTKernel/IR/TTKernelOps.cpp 42-48](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTKernel/IR/TTKernelOps.cpp#L42-L48)
*   `CBPopFrontOp`: Increments the read pointer and signals the producer [lib/Dialect/TTKernel/IR/TTKernelOps.cpp 50-56](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTKernel/IR/TTKernelOps.cpp#L50-L56)
*   `CBReserveBackOp`: Blocks until space for the specified number of tiles is available for writing [lib/Dialect/TTKernel/IR/TTKernelOps.cpp 58-64](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTKernel/IR/TTKernelOps.cpp#L58-L64)

## Register and DST Management

The hardware contains a **DST register** file capable of holding 16 tiles. Access to this register must be synchronized between the **Math** (computation) and **Pack** (data movement) threads.

### Register Lock Protocol

1.   **Math Thread**: Calls `TTKernel_TileRegsAcquireOp` to lock DST for computation [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 58-69](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L58-L69)
2.   **Math Thread**: Performs compute (e.g., `add_tiles`) writing to DST.
3.   **Math Thread**: Calls `TTKernel_TileRegsCommitOp` to release the lock for the math side [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 71-80](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L71-L80)
4.   **Pack Thread**: Calls `TTKernel_TileRegsWaitOp` to acquire the lock for packing [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 82-93](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L82-L93)
5.   **Pack Thread**: Calls `TTKernel_PackTileOp` or `TTKernel_PackTileBlockOp` to move data from DST to an output CB [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 106-165](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L106-L165)
6.   **Pack Thread**: Calls `TTKernel_TileRegsReleaseOp` to release the lock for the pack side [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 95-104](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L95-L104)

**Sources:**[include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 58-165](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L58-L165)[test/ttmlir/Conversion/TTKernelToEmitC/ttkernel.mlir 52-104](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/ttmlir/Conversion/TTKernelToEmitC/ttkernel.mlir#L52-L104)

## FPU and SFPU Operations

The dialect distinguishes between standard FPU operations and Special Function (SFPU) operations.

### FPU Operations (Matrix/Binary)

FPU operations typically read from input CBs and write to a DST register index.

*   `add_binary_tile`: Performs element-wise addition [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 322-332](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L322-L332)

### SFPU Operations (Unary/Element-wise)

SFPU operations perform complex math (Exp, Log, Sqrt) on tiles already residing in the DST register.

*   `exp_tile`: Calculates $e^x$ for a tile at a specific DST index [include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td 482-491](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTKernel/IR/TTKernelOps.td#L482-L491)

## SFPI Dialect (SFPU Programming Interface)

The SFPI dialect allows for lower-level control, mapping directly to Tenstorrent's SFPU C++ headers and GCC builtins. It is used for authoring custom kernels where high-level abstractions are insufficient.

### Key Features

*   **Direct Hardware Access**: Maps to `sfpi` namespace functions in the tt-metal backend.
*   **Custom Math**: Enables implementation of activation functions or logic not covered by standard TTKernel ops.
*   **Integration**: SFPI operations are typically emitted within a `ttkernel.compute` thread context, triggering the inclusion of specialized SFPU headers like `api/compute/eltwise_binary_sfpu.h`[lib/Target/TTKernel/TTKernelToCpp.cpp 60-62](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/TTKernel/TTKernelToCpp.cpp#L60-L62)

The `ScopedModuleHelper` ensures that headers for specific SFPU operations (e.g., `experimental_reg_api_generated.h`, `experimental_semaphore_generated.h`) are included based on the thread type and the specific callees found in the IR [lib/Target/TTKernel/TTKernelToCpp.cpp 106-164](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Target/TTKernel/TTKernelToCpp.cpp#L106-L164)

## Code Generation: EmitC and C++

The final stage of the TTKernel pipeline is lowering to C++.

### Type Conversion

The `TTKernelToEmitCTypeConverter` handles the mapping of MLIR types to C++ types:

*   `BFloat16Type` → `float`
*   `CBType` → `::tt::CB`
*   `L1AddrType` → `uint32_t`
*   `NocAddrType` → `uint64_t`
*   `FabricConnectionManagerType` → `experimental::FabricConnectionManager`

### Opaque Call Generation

Most TTKernel operations are lowered to `emitc.call_opaque` which maps directly to hardware LLK (Low Level Kernel) function calls. For example, `pack_tile` in MLIR becomes `pack_tile(dst_index, cb_id, cb_index)` in C++ [test/ttmlir/Conversion/TTKernelToEmitC/ttkernel.mlir 80-91](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/ttmlir/Conversion/TTKernelToEmitC/ttkernel.mlir#L80-L91)
