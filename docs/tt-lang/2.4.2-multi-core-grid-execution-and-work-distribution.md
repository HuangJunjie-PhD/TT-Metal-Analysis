This page describes how to write tt-lang kernels that execute across multiple cores in parallel, distributing work across a 2D grid of cores. It covers grid specification, core coordinate queries, work partitioning, and validation rules for multi-core execution.

For information about communication between cores using pipes, see **2.2.4 Pipes and Inter-core Communication**. For single-core kernel execution fundamentals, see **2.4.1 Single-core Execution**.

## Overview

Multi-core kernels in tt-lang execute the same kernel code on multiple cores simultaneously (SPMD model), with each core processing a different portion of the input tensors. The key mechanisms are:

*   **Grid specification**: Define the number of cores using `grid=(cols, rows)` or `grid="auto"`[docs/sphinx/specs/TTLangSpecification.md 97-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L97-L102)
*   **Core coordinates**: Query the current core's position using `ttl.node()`[docs/sphinx/specs/TTLangSpecification.md 127-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L127-L130)
*   **Grid dimensions**: Query the total grid size using `ttl.grid_size()`[docs/sphinx/specs/TTLangSpecification.md 105-115](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L105-L115)
*   **Work distribution**: Divide tensor tiles across cores using linearized indexing patterns or coordinate-based slicing [docs/sphinx/specs/TTLangSpecification.md 55-57](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L55-L57)

Each core runs three thread types (`MATH`, `BRISC`, `NCRISC`) independently with their own Dataflow Buffer (DFB) instances, synchronized through the DFB producer-consumer protocol [docs/sphinx/specs/TTLangSpecification.md 55-57](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L55-L57)

**Sources:**[docs/sphinx/specs/TTLangSpecification.md 53-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L53-L130)

## Grid Specification

### Grid Syntax

Grids are specified using the `grid` parameter in the `@ttl.operation` decorator [python/ttl/ttl_api.py 344-348](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L344-L348) The grid defines a 2D array of nodes where `(cols, rows)` maps to `(x, y)` coordinates [docs/sphinx/specs/TTLangSpecification.md 100-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L102) Currently, the compiler and AST visitor primarily support 2D grids [python/ttl/_src/ttl_ast.py 75-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L75-L76)

The coordinate system:

*   **x-axis (columns)**: increases left-to-right [docs/sphinx/specs/TTLangSpecification.md 113-124](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L113-L124)
*   **y-axis (rows)**: increases top-to-bottom [docs/sphinx/specs/TTLangSpecification.md 113-124](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L113-L124)

**Automatic grid selection** is supported via `grid="auto"`, allowing the compiler to optimize the grid size based on the target hardware's available compute resources and tensor shapes.

**Multi-chip and SPMD**:

*   In single-chip cases, the grid is two-dimensional [docs/sphinx/specs/TTLangSpecification.md 100-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L102)
*   In multi-chip cases using TT-Fabric, the grid has additional mesh dimensions representing card/host/rack connectivity [docs/sphinx/specs/TTLangSpecification.md 100-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L100-L102)
*   In SPMD mode, the grid remains 2D while the operation is submitted to multiple chips via TT-NN Mesh Devices [docs/sphinx/specs/TTLangSpecification.md 102-103](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L102-L103) This is demonstrated in SPMD tests where tensors are sharded across devices [test/python/test_matmul_with_bias_spmd.py 8-10](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_matmul_with_bias_spmd.py#L8-L10)

## Core Coordinate Functions

### ttl.node() and ttl.grid_size()

The `ttl.node()` function returns the current core's coordinates, while `ttl.grid_size()` returns the total dimensions of the allocated grid [docs/sphinx/specs/TTLangSpecification.md 105-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L105-L130) Both take a `dims` parameter to specify dimensionality.

| Syntax | Returns | Description |
| --- | --- | --- |
| `ttl.grid_size(dims=1)` | `int` | Returns total cores (cols * rows). If dims < grid dims, highest rank is flattened [docs/sphinx/specs/TTLangSpecification.md 117-119](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L117-L119) |
| `ttl.grid_size(dims=2)` | `(int, int)` | Returns `(cols, rows)`[docs/sphinx/specs/TTLangSpecification.md 120](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L120-L120) |
| `ttl.node(dims=2)` | `(int, int)` | Returns current `(node_col, node_row)` coordinates [docs/sphinx/specs/TTLangSpecification.md 127-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L127-L130) |

## Work Distribution Patterns

### Work Partitioning and Slicing

A common pattern involves defining a `block_count` (formerly `buffer_factor`) to balance L1 memory usage and transfer efficiency [docs/sphinx/specs/TTLangSpecification.md 47](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L47-L47)

1.   **Calculate blocks per node**: The user divides total tensor blocks by the grid dimensions [examples/elementwise-tutorial/step_3_multinode.py 59-60](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/elementwise-tutorial/step_3_multinode.py#L59-L60)
2.   **Coordinate-based access**: Use `node_col` and `node_row` to calculate global offsets into the tensor [examples/elementwise-tutorial/step_3_multinode.py 96-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/elementwise-tutorial/step_3_multinode.py#L96-L102)
3.   **Slice Transfers**: Use `ttl.copy` with tensor slices (e.g., `a[start:end, start:end]`) to fetch data for the specific node [examples/elementwise-tutorial/step_3_multinode.py 116-122](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/elementwise-tutorial/step_3_multinode.py#L116-L122)

### Validation and Constraints

*   **Grid Dimensionality**: Only 2D grids are currently supported by the MLIR lowering logic [python/ttl/_src/ttl_ast.py 75-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L75-L76)
*   **Tensor Layout**: Only tiled tensors are supported for multi-core operations [python/ttl/_src/ttl_ast.py 71-72](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L71-L72)
*   **Data Movement Synchronization**: `ttl.copy()` operations return a handle that must be explicitly synchronized with `.wait()`[docs/sphinx/specs/TTLangSpecification.md 15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L15-L15)

## Compilation and Lowering

The compilation pipeline transforms multi-core Python DSL into specialized MLIR and eventually C++ code.

### Pipeline Stages

The `TTLGenericCompiler` in `python/ttl/_src/ttl_ast.py` handles the initial AST traversal, converting Python control flow into MLIR [python/ttl/_src/ttl_ast.py 128-151](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L128-L151)

Key passes in the multi-core pipeline include:

1.   **TTLInsertIntermediateDFBs**: Materializes fused expressions into L1 buffers so operands are CB-attached for compute kernels [lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp 9-13](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp#L9-L13)
2.   **TTLAssignDST**: Allocates hardware DST registers for compute operations [lib/Dialect/TTL/Transforms/CMakeLists.txt 26](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/CMakeLists.txt#L26-L26)
3.   **ConvertTTLToTTKernel**: Lowers high-level TTL DMA and compute ops to hardware-specific `ttkernel` operations [lib/Dialect/TTL/Transforms/CMakeLists.txt 2](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/CMakeLists.txt#L2-L2)
