## Purpose and Scope

This document describes the hierarchical data structures that represent a NoC workload in tt-npe. The workload data model defines the fundamental abstractions for describing data transfers across the Network-on-Chip: `npeWorkload`, `npeWorkloadPhase`, and `npeWorkloadTransfer`. These structures form the input to the simulation engine and can be constructed either by parsing JSON files or programmatically.

For information about loading workloads from JSON files, see [Creating Workloads from JSON](https://deepwiki.com/tenstorrent/tt-npe/3.2-creating-workloads-from-json). For programmatic construction, see [Programmatic Workload Construction](https://deepwiki.com/tenstorrent/tt-npe/3.3-programmatic-workload-construction). For converting NoC traces to workloads, see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads).

## Hierarchical Structure

The workload data model follows a three-level hierarchy:

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 97-148](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L97-L148)

Each level serves a distinct purpose:

*   **`npeWorkload`**: Top-level container managing the entire workload, including metadata like golden cycle counts and zone information
*   **`npeWorkloadPhase`**: Logical grouping of transfers that execute together; phases execute sequentially
*   **`npeWorkloadTransfer`**: Individual data movement operation specifying source, destination, size, timing, and routing

* * *

## npeWorkload Class

The `npeWorkload` class is the top-level container that aggregates all phases and provides global workload metadata.

### Class Definition

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 97-148](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L97-L148)

### Key Fields

| Field | Type | Purpose |
| --- | --- | --- |
| `phases` | `std::vector<npeWorkloadPhase>` | Ordered list of workload phases |
| `gbl_transfer_id` | `npeWorkloadTransferID` | Counter for assigning unique transfer IDs |
| `num_transfer_groups` | `npeWorkloadTransferGroupID` | Counter for multi-hop transfer groups |

## npeWorkloadPhase Structure

A phase represents a logical grouping of transfers that conceptually execute together. Phases are processed sequentially by the simulation engine.

### Structure Definition

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 85-95](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L85-L95)

### Fields

| Field | Type | Access | Purpose |
| --- | --- | --- | --- |
| `transfers` | `std::vector<npeWorkloadTransfer>` | Public | List of all transfers in this phase |
| `id` | `npeWorkloadPhaseID` | Private | Unique phase identifier assigned by `npeWorkload::addPhase()` |

The phase structure is intentionally minimal. Its primary purpose is to group transfers and establish phase boundaries. The simulation engine does not enforce strict phase semantics—transfers from different phases can overlap in execution based on their `phase_cycle_offset` values.

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 85-95](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L85-L95)

## npeWorkloadTransfer Structure

The `npeWorkloadTransfer` structure defines a single data movement operation. It is the most detailed level of the workload hierarchy.

### Structure Definition

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 25-83](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L25-L83)

### Core Transfer Fields

| Field | Type | Purpose |
| --- | --- | --- |
| `packet_size` | `uint32_t` | Size of each packet in bytes |
| `num_packets` | `uint32_t` | Number of packets to send |
| `total_bytes` | `uint32_t` | Computed as `packet_size * num_packets` |
| `src` | `Coord` | Source coordinate (device + row + col) |
| `dst` | `NocDestination` | Destination: either `Coord` (unicast) or `MulticastCoordSet` (multicast) |
| `injection_rate` | `float` | GB/cycle the source can inject (default: 28.1) |
| `phase_cycle_offset` | `CycleCount` | Cycle offset relative to phase start when this transfer begins |
| `noc_type` | `nocType` | Either `NOC0` or `NOC1` |
| `noc_event_type` | `std::string` | Event type string from trace (e.g., "READ", "WRITE_", "WRITE_MULTICAST") |

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 58-66](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L58-L66)

### Metadata Fields

| Field | Type | Purpose |
| --- | --- | --- |
| `enclosing_zone_path` | `std::string` | Hierarchical path of zones enclosing this transfer (e.g., "prefetch[0]/read[2]") |
| `phase_id` | `npeWorkloadPhaseID` | ID of containing phase (assigned by `npeWorkload::addPhase()`) |
| `id` | `npeWorkloadTransferID` | Globally unique transfer ID |

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 68-82](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L68-L82)

### Transfer Group Fields

For multi-hop fabric transfers that span devices, individual single-device hops are linked via transfer groups:

| Field | Type | Purpose |
| --- | --- | --- |
| `transfer_group_id` | `npeWorkloadTransferGroupID` | Shared ID for all hops in a multi-device transfer (-1 if not part of group) |
| `transfer_group_index` | `npeWorkloadTransferGroupIndex` | Index of this hop within the group |
| `transfer_group_parent` | `npeWorkloadTransferGroupParent` | Index of parent hop that must complete before this hop starts |

See [Zone Tracking and Transfer Groups](https://deepwiki.com/tenstorrent/tt-npe/3.5-zone-tracking-and-transfer-groups) for detailed information on transfer groups.

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 70-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L70-L72)

## Supporting Data Types

### Coord Structure

The `Coord` structure identifies a specific core location within a device:

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 162-170](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L162-L170)

| Field | Type | Purpose |
| --- | --- | --- |
| `device_id` | `DeviceID` | Device identifier (typically 0 for single-chip, 0-7 for 8-chip T3K) |
| `row` | `int16_t` | Row coordinate (Y-axis) |
| `col` | `int16_t` | Column coordinate (X-axis) |

### NocDestination Type

`NocDestination` is a `std::variant` that represents either unicast or multicast destinations:

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 303](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L303-L303)

### MulticastCoordSet Structure

For multicast transfers, `MulticastCoordSet` defines one or more rectangular grids of destination cores:

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 172-300](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L172-L300)

Each `CoordGrid` defines a rectangular region with inclusive start and end coordinates. The iterator interface allows traversing all destination cores in row-major order.

### npeZone Structure

Zones are temporal markers used to track execution phases within application code:

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 313-325](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L313-L325)

Zones are stored hierarchically and flattened into paths like `"outer_zone[0]/inner_zone[2]"` that are attached to transfers.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 291-301](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L291-L301)

## Validation Rules

The `npeWorkload::validate()` method verifies workload integrity against a device model. Validation occurs at three levels:

### Workload-Level Validation

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 89-131](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L89-L131)

### Transfer-Level Validation

Each transfer must satisfy:

| Rule | Validation Check | Error Message |
| --- | --- | --- |
| Positive packet count | `num_packets > 0` | `INVALID_NUM_PACKETS` |
| Positive packet size | `packet_size > 0` | `INVALID_PACKET_SIZE` |
| Valid source coordinate | `0 <= src.row < device.rows && 0 <= src.col < device.cols` | `INVALID_SRC` |
| Valid destination coordinates | Same check for all destination coords | `INVALID_DST` |
| Non-negative start time | `phase_cycle_offset >= 0` | `INVALID_REL_START_TIME` |
| Matching device IDs | `src.device_id == dst.device_id` (for single-device transfers) | `SRC_AND_DST_DEVICE_IDS_MISMATCH` |
| Valid device IDs | `device_model.isValidDeviceID(src.device_id)` | `INVALID_DEVICE_IDS` |

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 15-76](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L15-L76)

Validation errors are logged with transfer IDs and source file names. After 5 errors (or 50 in verbose mode), additional errors are suppressed to prevent log spam.

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 48-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L48-L72)

### Multicast Validation

For multicast destinations, all coordinate grids within the `MulticastCoordSet` must have valid start and end coordinates within the device grid:

`for (const auto &[start, end] : dst_mcast.coord_grids) {    valid_dst = valid_dst || (is_valid_coord(start, ...) &&                              is_valid_coord(end, ...));}`
**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 36-40](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L36-L40)

## Workload Operations

### Injection Rate Inference

The `inferInjectionRates()` method queries the device model to set appropriate injection rates based on core types:

Different core types have different injection rates (e.g., DRAM cores: 23.2 GB/cycle, Worker cores: 28.1 GB/cycle).

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 145-154](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L145-L154)

### Schedule Scaling

The `scaleWorkloadSchedule()` method applies a linear scaling factor to all `phase_cycle_offset` values:

`tr.phase_cycle_offset = static_cast<CycleCount>(tr.phase_cycle_offset * scale_factor);`
This allows compressing or expanding the workload timeline for analysis purposes.

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 133-143](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L133-L143)

### Local Transfer Removal

The `removeLocalUnicastTransfers()` method filters out transfers where source and destination are in the same 2x2 core tile:

`if ((tr.src.row / 2 == dst_coord.row / 2) &&    (tr.src.col / 2 == dst_coord.col / 2)) {    // Remove this transfer}`
This is useful for focusing analysis on cross-tile or cross-chip communication.

**Sources:**[tt_npe/cpp/src/npeWorkload.cpp 156-185](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkload.cpp#L156-L185)

## Transfer Groups for Multi-chip Transfers

Multi-hop fabric transfers that span devices are represented as multiple single-device transfers linked by transfer group IDs. Each hop has:

*   **`transfer_group_id`**: Shared across all hops in the same multi-device transfer
*   **`transfer_group_index`**: Sequential index of this hop
*   **`transfer_group_parent`**: Index of the parent hop that must complete first

The simulation engine enforces these dependencies to ensure correct ordering of multi-hop paths. See [Zone Tracking and Transfer Groups](https://deepwiki.com/tenstorrent/tt-npe/3.5-zone-tracking-and-transfer-groups) for more details.

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 70-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L70-L72)[tt_npe/cpp/src/npeWorkloadIngest.cpp 519-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L519-L639)
