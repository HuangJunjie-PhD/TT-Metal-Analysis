## Purpose and Scope

This document describes the character device layer that exposes Tenstorrent devices to user-space through `/dev/tenstorrent/{ordinal}` nodes. It covers device registration (`tenstorrent_register_device`), file operation handlers (`chardev_fops`), per-file descriptor state management (`chardev_private`), resource tracking, and automatic cleanup on file descriptor release.

For IOCTL command specifications, see [IOCTL Command Reference](https://deepwiki.com/tenstorrent/tt-kmd/5.2-ioctl-command-reference). For memory mapping operations, see [Memory Mapping Interface](https://deepwiki.com/tenstorrent/tt-kmd/5.3-memory-mapping-interface). For synchronization primitives, see [Synchronization Primitives](https://deepwiki.com/tenstorrent/tt-kmd/9.1-synchronization-primitives).

## Release Operation

### Automatic Resource Cleanup

The `tt_cdev_release` function performs comprehensive cleanup of all resources allocated through a file descriptor. This guarantees no resource leaks, even when applications terminate abnormally.

**Release Sequence**

**Sources:**[chardev.c 2543-2598](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2543-L2598)

### Cleanup Operations Detail

The `tt_cdev_release` function performs cleanup in a specific order:

1.   **NOC Cleanup**: Performs a registered 32-bit NOC write if configured via `SET_NOC_CLEANUP`[chardev.c 2556-2563](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2556-L2563)
2.   **Memory Cleanup**: Calls `tenstorrent_memory_cleanup` to free DMA buffers, unpin pages, and clean VMAs [chardev.c 2568](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2568-L2568)
3.   **Resource Locks**: Releases all bitmask-based resource locks and wakes up waiting processes [chardev.c 2570-2576](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2570-L2576)
4.   **TLB Cleanup**: Releases all TLBs associated with the file descriptor [chardev.c 2578-2580](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2578-L2580)
5.   **Power Aggregation**: Recomputes the aggregated power state if the FD contributed to power policy [chardev.c 2590-2591](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2590-L2591)

**Sources:**[chardev.c 2543-2598](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2543-L2598)

### Reset Generation Tracking

The `reset_gen` counter implements generation-based file descriptor invalidation. When a device is reset, the counter increments [chardev.c 223](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L223-L223) permanently invalidating all file descriptors opened before the reset.

**Validation Checks** In `tt_cdev_ioctl`, the following checks are performed while holding the semaphore:

*   **Detached**: If `tt_dev->detached` is true, return `-ENODEV`[chardev.c 2321](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2321-L2321)
*   **Stale FD**: If `tt_dev->reset_gen != priv->open_reset_gen`, the FD is stale; return `-ENODEV`[chardev.c 2327](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2327-L2327)
*   **Hardware Init**: If `tt_dev->needs_hw_init` is true, only allow specific info or reset commands [chardev.c 2331-2334](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2331-L2334)

**Sources:**[chardev.c 2321-2334](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L2321-L2334)

## Power Aggregation Lifecycle

The driver aggregates power preferences from all open file descriptors. Each `chardev_private` holds a `power_state` structure [chardev_private.h 75](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L75-L75)

**Aggregation Logic**

1.   **State Capture**: Clients set preferences via `SET_POWER_STATE` IOCTL [chardev.c 384](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L384)
2.   **Aggregation**: The driver iterates through `open_fds_list`[chardev.c 415](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L415-L415)
3.   **Policy**: It calculates the maximum requested clock frequencies and performs a bitwise OR of all requested power flags [chardev.c 417-428](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L417-L428)
4.   **Application**: The final aggregated state is sent to the ARC firmware [chardev.c 433](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L433-L433)

**Sources:**[chardev.c 384-436](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L436)[chardev.h 15](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.h#L15-L15)

## Diagnostic Interfaces

### procfs: Active PIDs

The `/proc/tenstorrent/{ordinal}/pids` file lists the process IDs of all processes with open file descriptors to the device.

**Sources:**[chardev.c 112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L112-L112)[enumerate.h 27](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.h#L27-L27)

### debugfs: Memory Mappings

The driver creates a `mappings` file in debugfs for each device to dump the state of all active `chardev_private` structures, including their DMA buffers and TLB assignments.

**Sources:**[chardev.c 109](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L109-L109)[chardev.c 289](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L289-L289)
