# Grayskull Tensix Reset

Relevant source files
*   [CHANGELOG.md](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1)
*   [tt_tools_common/reset_common/gs_tensix_reset.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/gs_tensix_reset.py)

## Purpose and Scope

This document describes the Grayskull-specific Tensix core reset implementation provided by the `GSTensixReset` class. Unlike Blackhole and Wormhole chips which use driver-based reset mechanisms (see [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation) and [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation)), Grayskull requires a software-controlled reset sequence that directly manipulates Tensix cores and Network-on-Chip (NOC) routers through register writes and ARC firmware messages.

The Grayskull reset procedure handles:

*   Detection of harvested (disabled) Tensix rows via efuse reading
*   Safe clock frequency management during reset operations
*   NOC router configuration including broadcast disables for harvested cores
*   Sequential hard and soft reset assertion/deassertion of RISC-V processors
*   Differential handling of harvested vs. non-harvested cores

For the unified reset interface that routes to chip-specific implementations, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns). For reset configuration and logging, see [Reset Configuration and Logging](https://deepwiki.com/tenstorrent/tt-tools-common/2.6-reset-configuration-and-logging).

## Grid Architecture and Coordinate Systems

The Grayskull chip contains a 13×12 grid of nodes, including Tensix cores, DRAM controllers, ARC processor, and PCIe interface. The grid uses two coordinate systems that must be mapped between:

### Physical and Routing Coordinates

**Physical coordinates** represent the actual hardware layout, while **routing coordinates** represent NOC addressing. The conversion uses a mirroring algorithm where even physical coordinates map to the lower half and odd coordinates map to the mirrored upper half of routing space.

**Diagram: Grayskull Grid Architecture and Special Component Locations**

The conversion function `_physical_to_routing()` implements the mirroring:

```
if (phys % 2) == 0:
    rout = phys // 2
else:
    rout = limit - 1 - (phys // 2)
```

During initialization, the `ROUTING_TO_PHYSICAL_TABLE` dictionary is populated for bidirectional lookups between coordinate systems.

| Component | Grid Region | Count | Purpose |
| --- | --- | --- | --- |
| Tensix Cores | Columns 1-12, Rows 1-5,7-11 | Up to 120 | Compute cores with RISC-V processors |
| DRAM Controllers | Rows 0,6 at columns 1,4,7,10 | 8 | Memory interface |
| ARC Processor | (0,2) | 1 | Chip management firmware |
| PCIe Interface | (0,4) | 1 | Host communication |
| NOC Routers | Row 0, Column 0, Row 6 | Variable | Routing fabric nodes |

## Harvesting Detection and Core List Generation

Grayskull chips may have rows of Tensix cores disabled during manufacturing due to defects. These "harvested" rows are identified through efuse values read via ARC firmware message.

**Diagram: Harvesting Detection and Core List Generation Process**

The harvesting efuse contains two 10-bit fields:

*   **Bits [9:0]**: Memory defect rows
*   **Bits [19:10]**: Logic defect rows

These are OR'd together and shifted left by 1 to produce a bitmap of bad physical rows. The physical row indices are then mapped to NOC0 Y coordinates using `PHYS_Y_TO_NOC_0_Y` lookup table, which implements the physical-to-routing transformation specific to the Y axis.

The resulting `self.harvested_rows` frozenset is used throughout the reset process to differentiate handling of harvested vs. active cores.

## Safe Clock Management

Before manipulating Tensix reset signals, the ARC firmware must lower clock frequencies to safe levels. This prevents timing violations during reset transitions.

**Diagram: Safe Clock Entry and Exit Sequence**

The `set_safe_clks()` method sends the `MSG_TYPE_RESET_SAFE_CLKS` ARC message with `arg0=1` to enter safe mode and `arg0=0` to exit. The main reset sequence uses a try/finally block to ensure clocks are always restored, even if the reset sequence encounters an error:

`self.set_safe_clks(True)try:    # Reset operationsfinally:    self.set_safe_clks(False)`

## NOC Router Configuration

The Network-on-Chip routers must be configured to handle broadcast operations correctly, accounting for harvested rows and special components. This configuration involves setting broadcast disable masks and enabling clock gating.

### NOC Router Register Layout

| Register | Offset | Purpose |
| --- | --- | --- |
| `NIU_CFG_0` | 0x100 | Bit 0: CG enable, Bit 12: Tile clock disable |
| `ROUTER_CFG_0` | 0x104 | Bit 0: CG enable, Bits [11:8]: Max backoff exp |
| `ROUTER_CFG_1` | 0x108 | Column broadcast disable mask |
| `ROUTER_CFG_3` | 0x110 | Row broadcast disable mask |

### Broadcast Disable Calculation

**Diagram: NOC Router Broadcast Disable Computation**

The broadcast disable masks ensure that:

1.   **Column 0** (NOC0) and **Column 12** (NOC1) are always disabled as they contain special components
2.   **Row 0** and **Row 6** are always disabled as they contain DRAM controllers
3.   **Harvested rows** are disabled to prevent routing to non-functional cores
4.   **NOC1 uses mirrored coordinates**, requiring row remapping: `GRID_SIZE_Y - y - 1`

### NOC Setup Procedure

The `setup_noc()` method iterates through all grid locations and configures each node based on its type:

**Diagram: NOC Setup Flow for All Grid Nodes**

For each node, the configuration writes:

*   **ROUTER_CFG_1**: Column broadcast disable mask
*   **ROUTER_CFG_3**: Row broadcast disable mask
*   **NIU_CFG_0 bit 0**: Clock gating enable
*   **ROUTER_CFG_0 bit 0**: Clock gating enable
*   **ROUTER_CFG_0 bits [11:8]**: Maximum exponential backoff (`NOC_MAX_BACKOFF_EXP = 0xF`)

For Tensix cores specifically, **NIU_CFG_0 bit 12** controls tile clock enable/disable based on harvesting status. Harvested cores have their tile clocks disabled to save power.

## Reset Sequence

The complete Tensix reset sequence involves five distinct phases, executed while in safe clock mode:

**Diagram: Complete Tensix Reset Sequence**

### Phase 1: Assert Hard Reset on All RISCs

The `all_riscs_assert_hard_reset()` method writes 0x0 to all 8 `ARC_RESET.RISCV_RESET[i]` registers, placing every RISC-V processor (across all Tensix cores, harvested or not) into hard reset state.

### Phase 2: Toggle Tensix Core Reset

The `tensix_toggle_reset()` method:

1.   Writes 0x0 to all 8 `ARC_RESET.TENSIX_RESET[i]` registers (assert reset)
2.   Writes 0xFFFFFFFF to all 8 `ARC_RESET.TENSIX_RESET[i]` registers (deassert reset)

This resets the entire Tensix core fabric, including NOC routers, regardless of harvesting status. All cores' NOC routers must be functional for the network to operate correctly.

### Phase 3: Setup NOC Routers

The `setup_noc()` method configures all NOC nodes as described in the previous section, including broadcast disables, clock gating, and tile clock management.

### Phase 4: Assert Soft Reset via NOC Broadcast

The `assert_all_riscv_soft_reset()` method uses NOC broadcast to write soft reset bits to all **non-harvested** Tensix cores in a single operation:

```
BRISC_SOFT_RESET = 1 << 11
TRISC_SOFT_RESETS = (1 << 12) | (1 << 13) | (1 << 14)
NCRISC_SOFT_RESET = 1 << 18
noc_broadcast32(0, 0xFFB121B0, combined_bits)
```

This targets address 0xFFB121B0 on NOC0, which reaches only active cores due to the broadcast disable configuration set in Phase 3.

### Phase 5: Deassert Hard Reset on Active Cores

The `all_riscs_deassert_hard_reset()` method uses the reset mask computed by `all_tensix_reset_mask()` to selectively deassert hard reset only for non-harvested cores:

**Diagram: Selective Hard Reset Deassertion for Active Cores**

The bit position calculation converts NOC coordinates to "blue coordinates" (Tensix indexing) using the `ROUTING_TO_PHYSICAL_TABLE`:

```
phys_x, phys_y = ROUTING_TO_PHYSICAL_TABLE[(noc_x, noc_y)]
blue_x, blue_y = phys_x - 1, phys_y - 1
reset_bit_index = blue_x * NUM_TENSIX_X + blue_y
reset_reg_index = reset_bit_index // 32
reset_bit_mask = 1 << (reset_bit_index % 32)
```

After Phase 5, active Tensix cores are:

*   **Out of hard reset** (RISCs can execute)
*   **In soft reset** (RISCs held at reset vector, awaiting firmware load)

Harvested cores remain in hard reset permanently.

## GSTensixReset Class Implementation

### Class Structure and Dependencies

**Diagram: GSTensixReset Class Structure and Dependencies**

### Initialization Process

`def __init__(self, device: PciChip):    # Build coordinate mapping table    self.ROUTING_TO_PHYSICAL_TABLE = dict()    for phys_y in range(self.GRID_SIZE_Y):        for phys_x in range(self.GRID_SIZE_X):            rout_x = self._physical_to_routing(self.GRID_SIZE_X, phys_x)            rout_y = self._physical_to_routing(self.GRID_SIZE_Y, phys_y)            self.ROUTING_TO_PHYSICAL_TABLE[(rout_x, rout_y)] = (phys_x, phys_y)        self.device = device    self.axi_registers = get_register_object(device)    self.fw_defines = init_fw_defines("grayskull", "tt_tools_common.data")    self.harvested_rows = self.get_harvested_rows()    self.core_list = self.get_core_list()`
The constructor performs five initialization steps:

1.   **Populate coordinate mapping table** for bidirectional NOC↔physical conversion
2.   **Store device reference** for hardware communication
3.   **Initialize register abstraction** via `get_register_object()` for AXI writes
4.   **Load firmware definitions** from `fw_defines.yaml` for ARC message types
5.   **Detect harvesting** and build list of active cores

### Public API: tensix_reset()

The primary entry point is `tensix_reset(silent=False)`, which orchestrates the complete reset sequence:

`def tensix_reset(self, silent=False) -> None:    if not silent:        print(CMD_LINE_COLOR.YELLOW, "Lowering clks to safe value...", CMD_LINE_COLOR.ENDC)    self.set_safe_clks(True)    try:        if not silent:            print(CMD_LINE_COLOR.YELLOW, "Beginning reset sequence...", CMD_LINE_COLOR.ENDC)        self.all_riscs_assert_hard_reset()        self.tensix_toggle_reset()        self.setup_noc()        self.assert_all_riscv_soft_reset()        self.all_riscs_deassert_hard_reset()        if not silent:            print(CMD_LINE_COLOR.YELLOW, "Finishing reset sequence...", CMD_LINE_COLOR.ENDC)    finally:        if not silent:            print(CMD_LINE_COLOR.YELLOW, "Returning clks to original values...", CMD_LINE_COLOR.ENDC)        self.set_safe_clks(False)`
The `silent` parameter suppresses console output for use in automated tools. The try/finally ensures safe clock restoration even on error.

## Usage Example

The following code demonstrates typical usage of `GSTensixReset`:

`from pyluwen import PciChipfrom tt_tools_common.reset_common.gs_tensix_reset import GSTensixReset # Open chip connectiondevice = PciChip(pci_interface=0) # Create reset instance (automatically detects harvesting)reset = GSTensixReset(device) # Examine harvested rowsprint(f"Harvested rows (NOC0 Y): {reset.harvested_rows}")print(f"Active Tensix cores: {len(reset.core_list)}") # Perform resetreset.tensix_reset(silent=False) # For silent operation in automated scriptsreset.tensix_reset(silent=True)`
The reset leaves all active Tensix cores in soft reset state, ready for firmware loading via subsequent operations (typically performed by higher-level tools like tt-smi or tt-flash).

## Integration with Reset System

`GSTensixReset` is integrated into the broader reset system through the `parse_reset_input()` function and chip detection logic. The class is instantiated and used for Grayskull devices when a reset is requested:

`# Typical integration pattern (simplified)from tt_tools_common.utils_common.tools_utils import detect_chips_with_callbackfrom tt_tools_common.reset_common.gs_tensix_reset import GSTensixReset # Detect Grayskull chipschips = detect_chips_with_callback(pci_interface_ids=[...])for chip in chips:    if chip.architecture == "grayskull":        reset = GSTensixReset(chip)        reset.tensix_reset()`
Unlike Blackhole and Wormhole which route through `ChipReset` based on driver version (see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns)), Grayskull always uses this direct implementation as it predates the unified reset interface.

## Migration History

The Grayskull Tensix reset implementation was migrated from tt-smi to tt-tools-common in version 1.3.0 (released 2024-03-06) along with related data files. This migration made the reset functionality available to all Tenstorrent tools, promoting code reuse and consistent behavior across the ecosystem.

The implementation includes a `silent` flag (added in version 1.4.2) to make it more versatile for integration into tools that prefer to manage their own output formatting.
