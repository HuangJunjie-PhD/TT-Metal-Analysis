This page documents the internal data structures used throughout tt-npe's simulation engine and workload processing pipeline. These types are critical for developers who need to understand the simulation implementation, add new device models, or extend the statistics collection system.

For information about the public-facing API types exposed to Python, see [Python API Reference](https://deepwiki.com/tenstorrent/tt-npe/6.2-python-api-reference). For device model interface details, see [Device Model Interface](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface).

## Overview of Data Structure Categories

The tt-npe codebase organizes data structures into several functional categories:

| Category | Purpose | Key Types |
| --- | --- | --- |
| **Workload Representation** | Define NoC traffic patterns before simulation | `npeWorkloadTransfer`, `npeWorkloadPhase`, `npeWorkload` |
| **Simulation State** | Track active transfers during execution | `PETransferState`, `TransferQueuePair` |
| **Coordinate System** | Specify source/destination locations | `Coord`, `MulticastCoordSet`, `NocDestination` |
| **Routing** | Represent NoC paths and resources | `nocRoute`, `nocLinkID`, `nocNIUID` |
| **Device State** | Track runtime resource contention | `npeDeviceState`, `LinkDemandGrid`, `NIUDemandGrid` |
| **Statistics** | Collect and report performance metrics | `TimestepStats`, `npeStats::deviceStats` |
| **Dependencies** | Enforce transfer ordering constraints | `npeTransferDependencyTracker`, `npeCheckpointID` |
| **Profiling Context** | Track execution zones from traces | `npeZone`, `ZoneIterator` |

## Transfer Representation

Transfers are represented at two levels: **workload-level** (before simulation) and **simulation-level** (during execution).

### Workload-Level: npeWorkloadTransfer

**npeWorkloadTransfer** defines a single NoC transfer at the workload level. Key fields:

*   **packet_size** / **num_packets**: Transfer size parameters
*   **src** / **dst**: Source coordinate and destination (unicast or multicast)
*   **injection_rate**: Bytes/cycle the source can inject (0.0 = infer from device model)
*   **phase_cycle_offset**: When this transfer starts relative to phase beginning
*   **noc_type**: NOC0 or NOC1
*   **enclosing_zone_path**: Profiling context string (e.g., `"kernel_main[0]/matmul_loop[5]"`)
*   **transfer_group_id/index/parent**: Multi-chip routing dependency information (see [Multi-chip Simulation](https://deepwiki.com/tenstorrent/tt-npe/7.2-multi-chip-simulation-details))

Each transfer is assigned unique IDs when added to a workload: a global `id` and a `phase_id` identifying which phase it belongs to.

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 25-83](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L25-L83)

### Simulation-Level: PETransferState

During simulation, each `npeWorkloadTransfer` is converted to a `PETransferState` that tracks runtime progress:

**PETransferState fields:**

*   **params**: Original workload transfer definition
*   **route**: Computed sequence of NoC links from source to destination (see [Routing Algorithms](https://deepwiki.com/tenstorrent/tt-npe/5.5-routing-algorithms-and-link-lookup))
*   **start_cycle**: When transfer actually begins (after dependencies resolve)
*   **end_cycle**: When transfer completes
*   **total_bytes_transferred**: Progress tracker
*   **curr_bandwidth**: Current transfer rate in bytes/cycle (computed each timestep)
*   **depends_on**: Checkpoint this transfer waits for (see [Dependency Tracking System](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system))
*   **required_by**: Checkpoints that depend on this transfer completing

The simulation engine creates a `std::vector<PETransferState>` containing all transfers, indexed by their workload ID.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 25-47](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L25-L47)[tt_npe/cpp/src/npeEngine.cpp 212-218](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L212-L218)

## Coordinate System

### Coord Structure

**Coord** represents a single point in the device grid with device ID, row, and column. Note that:

*   **row** corresponds to Y coordinate
*   **col** corresponds to X coordinate
*   Device IDs enable multi-chip workload representation

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 162-170](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L162-L170)

### MulticastCoordSet

For multicast transfers, destinations are specified as one or more rectangular grids:

**MulticastCoordSet** can contain multiple `CoordGrid` rectangles, each defined by start/end coordinates. The iterator allows traversing all destination cores in row-major order across all grids.

Example usage:

### NocDestination Variant

**NocDestination** is a `std::variant<Coord, MulticastCoordSet>` that allows a transfer to specify either a unicast or multicast destination. Use `std::visit` with the `overloaded` helper to handle both cases.

## Routing Data Structures

### nocRoute

A **nocRoute** is a `std::vector<nocLinkID>` representing the ordered sequence of NoC links traversed by a transfer from source to destination. Each element identifies one hop.

For local transfers (source == destination), the route is empty. For unicast transfers, the route length equals the Manhattan distance. For multicast, device models may generate tree-based routes.

**Sources:** Inferred from usage in [tt_npe/cpp/src/npeEngine.cpp 42](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L42-L42)

### nocLinkID and nocNIUID

These types identify NoC resources:

| Type | Purpose | Structure |
| --- | --- | --- |
| **nocLinkID** | Identifies a unidirectional link between two cores | `std::tuple<nocType, Coord, Coord>` (noc_type, src_core, dst_core) |
| **nocNIUID** | Identifies a Network Interface Unit on a core | `std::tuple<nocType, int, int>` (noc_type, row, col) |

Both types use tuple-based representation for efficient hashing in unordered maps. Device models use bidirectional link lookups to find link attributes in both directions.

**Sources:** Inferred from [tt_npe/cpp/src/npeEngine.cpp 71-87](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L71-L87) device model implementations

## Device State Tracking

### npeDeviceState

**npeDeviceState** tracks runtime resource utilization during simulation:

*   **LinkDemandGrid**: Maps each `nocLinkID` to aggregate bandwidth demand (bytes/cycle). Multiple transfers using the same link have their demands summed.
*   **NIUDemandGrid**: Maps each `nocNIUID` to aggregate injection/absorption demand.
*   **MulticastWriteLinkDemandGrid**: Separately tracks multicast write demand for statistics purposes.

These grids are populated by `npeDeviceModel::computeCurrentTransferRate()` each timestep and reset between timesteps. When congestion modeling is enabled, high demand values trigger bandwidth derating.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 207-209](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L207-L209)[tt_npe/cpp/src/npeEngine.cpp 258-274](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L258-L274)

## Statistics Collection

### TimestepStats

**TimestepStats** captures per-timestep simulation metrics:

*   **Cycle bounds**: `start_cycle`, `end_cycle` define the timestep window
*   **Demand metrics**: Aggregate and maximum utilization across all NoC resources
*   **Per-NoC breakdowns**: Separate statistics for NOC0 and NOC1
*   **Demand grids**: Snapshot of resource utilization for visualization
*   **live_transfer_ids**: Active transfers during this timestep

Important distinction: **link_demand** can exceed 100% (multiple overlapping transfers), while **link_util** represents actual cycle utilization (capped at 100%).

**Sources:**[tt_npe/cpp/include/npeStats.hpp 20-47](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeStats.hpp#L20-L47)

### npeStats and deviceStats

**npeStats** is the top-level statistics container with per-device breakdown. For multi-chip simulations, `per_device_stats` contains entries for each device ID plus a special `MESH_DEVICE` entry for aggregate statistics.

**deviceStats** accumulates both:

*   **Overall metrics**: Computed from `per_timestep_stats` during `computeSummaryStats()`
*   **Per-timestep history**: Stored in `per_timestep_stats` vector

Key metrics:

*   **estimated_cycles**: Latest transfer end cycle (worst-case)
*   **estimated_cong_free_cycles**: Cycle count from congestion-free simulation (if enabled)
*   **golden_cycles**: Expected cycles from trace or user-provided value
*   **cycle_prediction_error**: `(estimated - golden) / golden * 100`
*   **eth_bw_util_per_core**: Ethernet bandwidth utilization per core (multi-chip only)

**Sources:**[tt_npe/cpp/include/npeStats.hpp 50-125](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeStats.hpp#L50-L125)[tt_npe/cpp/src/npeEngine.cpp 203-204](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L203-L204)[tt_npe/cpp/src/npeEngine.cpp 340](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L340-L340)

## Dependency Management

### npeTransferDependencyTracker

**npeTransferDependencyTracker** manages transfer dependencies using a checkpoint-based system:

1.   **Checkpoint creation**: A checkpoint is created with a `remaining_count` (how many events must satisfy it) and optional `delay` (additional cycles after satisfaction).

2.   **Dependency assignment**: Transfers store `npeCheckpointID depends_on` (which checkpoint they wait for) and `std::vector<npeCheckpointID> required_by` (which checkpoints depend on them).

3.   **Runtime updates**: When a transfer completes, `updateCheckpoint()` is called for each checkpoint it contributes to, decrementing `remaining_count` and updating `end_cycle`.

4.   **Activation checking**: Before activating a transfer, `done(depends_on, curr_cycle)` checks if its dependency is satisfied.

Special value: `npeTransferDependencyTracker::UNDEFINED_CHECKPOINT` indicates no dependency.

### Dependency Types Generated

The `npeEngine::genDependencies()` function creates two types of dependencies:

**NIU Ordering Dependencies** ([tt_npe/cpp/src/npeEngine.cpp 71-110](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L71-L110)):

*   Groups transfers by source NIU and link type
*   Each transfer depends on the Nth-previous transfer from same NIU (stride=2, approximating 2-VC effects)
*   Ensures realistic serialization of transfers from same source

**Transfer Group Dependencies** ([tt_npe/cpp/src/npeEngine.cpp 112-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L112-L153)):

*   For multi-chip fabric routes split into segments
*   Each segment depends on parent segment completing
*   Includes latency and ethernet hop delays
*   See [Multi-chip Simulation Details](https://deepwiki.com/tenstorrent/tt-npe/7.2-multi-chip-simulation-details) for more information

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 67-170](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L67-L170)[tt_npe/cpp/src/npeEngine.cpp 239-253](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L239-L253)

## Profiling Context Tracking

### npeZone

**npeZone** represents a single zone event from tt-metal trace profiling. Zones are nested scopes (e.g., kernel invocations, loop iterations) that provide execution context. Each zone has a start and end event at specific timestamps.

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 316-325](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L316-L325)

### ZoneIterator

**ZoneIterator** maintains a stack of currently-open zones while traversing a zone event list:

*   **operator++()**: Advances to next zone event, updating `enclosing_zones` stack 
    *   `ZONE_START`: Push zone onto stack with iteration count
    *   `ZONE_END`: Pop matching zone from stack

*   **enclosing_zones**: Stack of `(zone, iteration_count)` pairs representing current nesting
*   **zone_counts**: Tracks how many times each zone name has been entered (for iteration numbering)

This allows transfer ingestion to tag each transfer with its full context path (e.g., `"matmul_kernel[0]/inner_loop[5]"`).

**Usage during trace ingestion** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 384-401](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L384-L401)):

## Data Flow Through Simulation Pipeline

**Initialization Phase:**

1.   `npeWorkload` → `initTransferState()` → `vector<PETransferState>` with routes computed
2.   `PETransferState` → `createTransferQueue()` → sorted activation schedule
3.   `PETransferState` → `genDependencies()` → dependency tracker with checkpoints

**Simulation Loop (each timestep):**

1.   Activate transfers: Check dependencies, move from queue to live list
2.   Compute bandwidth: `computeCurrentTransferRate()` updates each transfer's `curr_bandwidth` based on device state
3.   Update statistics: Record demand grids and metrics in `TimestepStats`
4.   Transfer progress: Update `total_bytes_transferred`, check for completion
5.   Checkpoint updates: Completed transfers update their `required_by` checkpoints

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 202-361](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L202-L361)

## Memory Layout Considerations

Key performance characteristics of the core data structures:

| Structure | Size Estimation | Access Pattern | Optimization |
| --- | --- | --- | --- |
| `vector<PETransferState>` | ~200 bytes × num_transfers | Random access by ID | Flat vector for cache locality |
| `LinkDemandGrid` | ~32 bytes × num_active_links | Dense iteration | `boost::unordered_flat_map` for speed |
| `NIUDemandGrid` | ~32 bytes × num_active_nius | Dense iteration | `boost::unordered_flat_map` for speed |
| `per_timestep_stats` | ~500 bytes × num_timesteps | Sequential append | `std::vector` pre-allocated |
| `nocRoute` | 8 bytes × route_length | Sequential iteration | Small vector optimization possible |

For large workloads (>100K transfers), the `vector<PETransferState>` dominates memory usage. The simulation engine uses flat indexing (`PETransferID` as array index) for O(1) lookup without pointer indirection.
