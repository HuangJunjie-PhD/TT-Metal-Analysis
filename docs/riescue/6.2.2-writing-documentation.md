This page provides comprehensive guidance for contributing documentation to the RiESCUE project. It covers reStructuredText syntax, Python docstring conventions, custom Sphinx directives, and documentation best practices.

For information about building documentation locally, see [Building Documentation](https://deepwiki.com/tenstorrent/riescue/6.1-building-documentation). For details about Sphinx configuration and extensions, see [Sphinx Configuration](https://deepwiki.com/tenstorrent/riescue/6.2.1-sphinx-configuration).

## Purpose and Scope

This guide is intended for developers who want to:

*   Add or update narrative documentation (tutorials, user guides)
*   Document Python code using proper docstring conventions
*   Use Sphinx extensions and custom directives effectively
*   Follow RiESCUE documentation standards

**Sources:**[.github/CONTRIBUTING.md 51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L51-L52)

The RiESCUE documentation is organized into three main categories, each serving a distinct purpose and audience.

## reStructuredText Syntax

RiESCUE documentation uses reStructuredText (reST), which is the standard markup language for Sphinx. This section covers commonly used syntax patterns.

### Basic Formatting

`**Bold text** for emphasis*Italic text* for subtle emphasis``code literals`` for inline code, variable names, or commandsLinks:- Internal doc reference: :doc:`/path/to/doc`- External URL: `Link text <https://example.com>`_- Section reference: :ref:`section-label``
### Section Headers

`Page Title (H1)===============Major Section (H2)------------------Subsection (H3)~~~~~~~~~~~~~~~Sub-subsection (H4)^^^^^^^^^^^^^^^^^^^`
### Lists and Code Blocks

`Bulleted list:- Item one- Item two  - Nested itemNumbered list:1. First step2. Second stepCode block with syntax highlighting:.. code-block:: python   def example_function():       return "Hello"Shell command example:.. code-block:: bash   $ riescued example.rasm --output-dir ./output`
### Directives and Admonitions

`.. note::   Important information for readers... warning::   Critical warnings about potential issues... code-block:: python   :caption: Optional caption for code   :linenos:   # Code with line numbers`

## Python Docstring Format

All Python code in RiESCUE must include docstrings following the Sphinx docstring format. This format enables automatic API documentation generation via the `autodoc` extension.

### Docstring Structure

**Sources:**[docs/source/conf.py 88-94](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L88-L94)[.github/CONTRIBUTING.md 51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L51-L52)

### Module and Class Docstrings

Can span multiple    paragraphs.     :param arg1: Description of the first argument    :type arg1: str    :param arg2: Description of the second argument, defaults to None    :type arg2: int, optional     Example:        Creating and using an instance::             obj = MyClass("value", arg2=42)            result = obj.process()    """     def __init__(self, arg1: str, arg2: int = None):        """Initialize MyClass with configuration parameters."""        self.arg1 = arg1        self.arg2 = arg2`
**Sources:**[.github/CONTRIBUTING.md 51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L51-L52)

### Function and Method Docstrings

`def function_name(param1: str, param2: int, optional_param: bool = False) -> dict:    """Short description of what the function does.     Longer description providing more context about the function's    behavior, side effects, or important implementation details.     :param param1: Description of the first parameter    :type param1: str    :param param2: Description of the second parameter    :type param2: int    :param optional_param: Description of optional parameter, defaults to False    :type optional_param: bool, optional     :return: Description of the return value    :rtype: dict     :raises ValueError: When param2 is negative    :raises IOError: When file operations fail     Example:        Basic usage::             result = function_name("test", 42)            print(result['key'])     Note:        Important notes about behavior or limitations.    """    if param2 < 0:        raise ValueError("param2 must be non-negative")        return {"key": "value"}`
### Field List Tags

| Tag | Purpose | Example |
| --- | --- | --- |
| `:param name:` | Parameter description | `:param path: Path to input file` |
| `:type name:` | Parameter type | `:type path: str` |
| `:return:` | Return value description | `:return: Parsed configuration` |
| `:rtype:` | Return type | `:rtype: dict` |
| `:raises Exception:` | Exception that can be raised | `:raises FileNotFoundError: When file missing` |

## Custom Sphinx Directives

RiESCUE extends Sphinx with custom directives for specialized documentation needs.

### ArgparseDirective

The `ArgparseDirective` automatically generates documentation for command-line arguments from `argparse.ArgumentParser` definitions.

**Sources:**[docs/source/conf.py 21-56](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L21-L56)[docs/source/conf.py 129-131](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L129-L131)

#### Implementation

The directive is defined in [docs/source/conf.py 21-56](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L21-L56) and registered in the `setup()` function at [docs/source/conf.py 129-131](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L129-L131):

`class ArgparseDirective(Directive):    required_arguments = 1  # module.function     def run(self):        # Import the class/module        module_path, item_name = self.arguments[0].rsplit(".", 1)        module = __import__(module_path, fromlist=[item_name])        item = getattr(module, item_name)         # Create parser and call add_arguments        parser = argparse.ArgumentParser()        item.add_arguments(parser)                # Generate documentation nodes from parser actions        # ...`
#### Usage in Documentation

`RiescueD Command-Line Arguments================================.. argparse:: riescue.dtest_framework.RiescueDThis will automatically generate formatted documentation for allcommand-line arguments defined in RiescueD.add_arguments().`
**Sources:**[docs/source/conf.py 21-56](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L21-L56)[docs/source/conf.py 129-131](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L129-L131)

### Writing Style Guidelines

| Guideline | Explanation | Example |
| --- | --- | --- |
| **Be concise** | Technical audience prefers direct information | Use "Returns a list of tests" not "This method will return a list containing all the tests" |
| **Use present tense** | Describe what code does, not what it will do | "Parses directives" not "Will parse directives" |
| **Use active voice** | Makes documentation clearer | "The parser extracts data" not "Data is extracted by the parser" |
| **Include examples** | Code examples clarify usage | Always provide working code snippets |
| **Link related docs** | Help readers navigate | Use `:doc:` and `:ref:` directives |

### Code Examples

All code examples should be:

*   **Executable**: Use realistic, working code that users can run
*   **Complete**: Include necessary imports and setup
*   **Commented**: Explain non-obvious aspects
*   **Tested**: Verify examples work before committing

`# Good example - complete and executablefrom riescue.dtest_framework import RiescueD # Create RiescueD instance with configurationriescued = RiescueD()riescued.parse_args(['test.rasm', '--output-dir', './out'])riescued.run() # Bad example - incomplete and vagueriescued = RiescueD()riescued.run()  # What args? What configuration?`

## Documentation Build Integration

Understanding how documentation integrates with the CI/CD pipeline helps ensure your contributions build correctly.

**Sources:**[docs/source/conf.py 1-132](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/source/conf.py#L1-L132)[.github/CONTRIBUTING.md 51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L51-L52)

### Testing Documentation Locally

Before submitting documentation changes:

1.   **Build and serve locally**:

`python docs/build.py --local_host`
This builds documentation and serves it at `http://localhost:8888`.

2.   **Check for errors**:

`python docs/build.py --check`
This runs the same validation as CI.

3.   **Verify**:

    *   All links work
    *   Code examples are formatted correctly
    *   API documentation generates from docstrings
    *   Custom directives render properly

**Sources:**[.github/CONTRIBUTING.md 51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L51-L52)

## Python Code Requirements for Documentation

To ensure documentation can be generated correctly, follow these coding standards:

| Requirement | Rationale | Example |
| --- | --- | --- |
| **Type hints** | Enables better API documentation | `def func(arg: str) -> int:` |
| **Absolute imports** | Autodoc can resolve imports correctly | `from riescue.lib.tool import Tool` |
| **Docstrings on all public APIs** | Users need reference documentation | Every public class, function, method |
| **Avoid global variables** | Harder to document and understand | Use class attributes or constants |
| **Handle exceptions gracefully** | Document with `:raises:` tags | Catch and re-raise with context |

**Sources:**[.github/CONTRIBUTING.md 54-58](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L54-L58)

## Contributing Documentation Checklist

Before submitting documentation changes, verify:

*   - [x]  All `.rst` files use proper reStructuredText syntax
*   - [x]  Python docstrings follow Sphinx format with `:param:`, `:type:`, `:return:`, `:rtype:` tags
*   - [x]  Code examples are complete, tested, and include necessary imports
*   - [x]  New public APIs have complete docstrings
*   - [x]  Documentation builds without errors: `python docs/build.py --check`
*   - [x]  Links to other documentation pages work correctly
*   - [x]  Custom directives (like `.. argparse::`) are used where appropriate
*   - [x]  Changes are mentioned in pull request description
