This page documents key optimization techniques for SFPI (Single Floating-Point Interface) programming, focusing on register management, load-store operations, latency hiding, and hardware-specific optimizations. For information about common usage patterns, see [Common Usage Patterns](https://deepwiki.com/tenstorrent/sfpi/7.1-integration-with-tt-metal), and for debugging optimization issues, see [Debugging and Troubleshooting](https://deepwiki.com/tenstorrent/sfpi/7.3-optimization-techniques).

## Register Movement Optimization

The SFPI compiler implements several optimization passes to minimize unnecessary register moves. Understanding these optimizations can help you write more efficient code with fewer instructions.

### Register Swap Optimization

The compiler can automatically swap operands for commutative operations to eliminate unnecessary register moves. This optimization applies to operations like addition, AND, OR, and XOR.

Example scenarios where swap optimization applies:

1.   **Addition**: `z = x + y` can use `y` as the target register

2.   **Logical Operations**: `z = x & y` or `z = x | y` can reuse registers

The compiler analyzes subsequent register usage and will only apply swap optimization when it doesn't affect other operations.

Sources: [tests/wormhole/sfpi/gold/regmov.S 10-77](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/regmov.S#L10-L77)[tests/wormhole/sfpi/regmov.cc 19-77](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc#L19-L77)

### Move Injection

When register swap isn't possible (e.g., non-commutative operations, subsequent register use), the compiler will inject an explicit move instruction to avoid compiler-generated less efficient moves.

Key cases where move injection is used:

1.   **Subtraction**: Non-commutative operations like `z = y - x`

2.   **Subsequent Use**: When both source operands are used later

3.   **Special Operations**: `setexp`, `setman`, and `setsgn` can't swap operands

Sources: [tests/wormhole/sfpi/gold/regmov.S 82-153](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/regmov.S#L82-L153)[tests/wormhole/sfpi/regmov.cc 82-188](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc#L82-L188)

### Predicated Move Optimization

For operations in conditional blocks, the compiler carefully places register moves to ensure correct results while minimizing instruction count.

In predicated code, the compiler:

1.   Places moves outside predicated blocks when possible
2.   Preserves register liveness across predication boundaries
3.   Manages complex register dependencies in conditionals

## Load Latency Optimization

SFPI operations have associated latencies that can impact performance. Understanding how to organize your code to hide these latencies is critical for optimization.

### Load-to-Use Latency

Loads from memory have an associated latency before the loaded value can be used. This creates dependency chains that can reduce performance.

Key techniques for load latency optimization:

1.   **Break dependency chains**: Split operations that don't depend on each other
2.   **Parallelize operations**: Group independent operations together
3.   **Preload values**: Load values before they're needed when possible

Sources: [tests/riscv/load.S 10-53](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.S#L10-L53)[tests/riscv/load.cc 43-71](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.cc#L43-L71)

### L1 Pointer Attribute Usage

The `l1p` attribute (`__attribute__((rvtt_l1_ptr))`) is used to denote pointers to L1 memory. This allows the compiler to generate optimal code for L1 memory accesses.

When using L1 pointers:

1.   **Consistent decoration**: Use the `l1p` attribute for all pointers to L1 memory
2.   **Avoid mixing**: Don't use the same register for both L1 and local memory accesses
3.   **Consider WAR hazards**: On Grayskull, handle write-after-read hazards (see below)

Sources: [tests/riscv/load.S 53-92](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.S#L53-L92)[tests/riscv/load.cc 59-90](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.cc#L59-L90)

### Wormhole Register Allocation

Wormhole has specific register allocation patterns that the compiler optimizes for. Understanding these patterns can help you write more efficient code.

Key techniques for Wormhole register optimization:

1.   **Short-lived variables**: Use for operations with limited scope
2.   **Long-lived variables**: Prefer higher-numbered registers (L0-L3) for these
3.   **Constant registers**: L13-L15 are often used for constants, be aware of their usage

## Immediate Value and Constant Optimization

### Immediate Value Limits

SFPI instructions have limits on the immediate values they can use. Understanding these limits helps in writing efficient code.

| Instruction | Immediate Range | Bits | Notes |
| --- | --- | --- | --- |
| SFPLOADI | 0 to 65535 (unsigned) -32768 to 32767 (signed) | 16 | Mode determines interpretation |
| SFPIADD | -2048 to 2047 | 12 | Signed immediate |
| SFPSHFT | -2048 to 2047 | 12 | Signed shift amount |
| SFPDIVP2 | -2048 to 2047 | 12 | Signed power of 2 |
| SFPSETEXP | 0 to 4095 | 12 | Unsigned immediate |
| SFPSETMAN | 0 to 4095 | 12 | Unsigned immediate |
| SFPSETSGN | 0 to 4095 | 12 | Unsigned immediate |

When immediate values exceed these limits, the compiler generates additional instructions to load the values.

Sources: [tests/wormhole/gcc/gold/limits.S 18-109](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/gcc/gold/limits.S#L18-L109)

### Constant Register Usage

The constant registers (L13-L15) provide special values that can be used without loading them first.

Efficient constant register usage:

1.   **Use directly**: Reference `vConstTileId` directly when possible
2.   **Avoid copying**: Don't unnecessarily copy from constant registers
3.   **Know the values**: Understand what values are in the constant registers

Sources: [tests/wormhole/sfpi/gold/regmov.S 426-439](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/regmov.S#L426-L439)[tests/wormhole/sfpi/regmov.cc 426-451](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc#L426-L451)

## Code Structure Optimization

### Loop Optimization Techniques

Loops are critical areas for optimization. The compiler applies several techniques to optimize loop execution.

Key loop optimization techniques:

1.   **Hoisting**: Move invariant operations outside the loop
2.   **Register allocation**: Keep frequently used values in registers
3.   **WAR hazard handling**: Properly place WAR NOPs before loops
4.   **Unrolling**: Use `#pragma GCC unroll` to control loop unrolling

Sources: [tests/riscv/gold/gs_arbiter_war.S 299-345](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/gold/gs_arbiter_war.S#L299-L345)[tests/riscv/gs_arbiter_war.cc 141-166](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/gs_arbiter_war.cc#L141-L166)

### Predication and Conditional Execution

SFPI provides vector condition codes that allow efficient conditional execution. The compiler optimizes these to minimize branch overhead.

Effective predication usage:

1.   **Minimize branches**: Use vector condition codes instead of branches when possible
2.   **Structure code**: Group similar operations in conditional blocks
3.   **Consider register pressure**: Predication may increase register usage

Sources: [tests/wormhole/sfpi/gold/regmov.S 154-178](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/regmov.S#L154-L178)[tests/wormhole/sfpi/regmov.cc 142-157](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc#L142-L157)

### Basic Block Organization

The compiler optimizes basic blocks (straight-line code sequences) to maximize instruction-level parallelism.

Key basic block optimization techniques:

1.   **Instruction scheduling**: Reorder instructions to hide latencies
2.   **Register allocation**: Optimize register usage across blocks
3.   **Branch prediction**: Structure code to make branches predictable
4.   **Cross-block optimization**: Allow optimizations across block boundaries

Sources: [tests/wormhole/sfpi/gold/regmov.S 248-306](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/regmov.S#L248-L306)[tests/wormhole/sfpi/regmov.cc 248-320](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/regmov.cc#L248-L320)

## Advanced Techniques

### Non-Immediate Instruction Handling

When immediate values exceed allowed ranges, the compiler generates instructions to handle them. Understanding these patterns helps in optimizing code.

The compiler uses several techniques:

1.   **Register loading**: Load large immediates into registers first
2.   **Instruction sequences**: Generate multiple instructions for large values
3.   **Constant pooling**: Share constants to reduce loading overhead

Sources: [tests/wormhole/gcc/gold/nonimm.S 10-214](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/gcc/gold/nonimm.S#L10-L214)

### Memory Access Patterns

Efficient memory access patterns can significantly improve performance.

Optimizing memory access:

1.   **Sequential access**: Preferred for performance
2.   **Minimize pointer arithmetic**: Calculate addresses efficiently
3.   **Align data**: Ensure proper alignment for data structures
4.   **Batch operations**: Group similar memory operations

Sources: [tests/riscv/load.S 121-142](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.S#L121-L142)[tests/riscv/load.cc 135-162](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/riscv/load.cc#L135-L162)
