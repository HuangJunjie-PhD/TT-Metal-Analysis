## Purpose and Scope

This page documents the device lifecycle and mesh management APIs in the `tt-mlir` runtime: how physical Tenstorrent hardware is opened, partitioned into submeshes, inspected, and closed. It covers the `MeshDevice` abstraction used by both the TTNN and TTMetal runtime backends, the dispatch layer that routes calls between backends, and the configuration options that control device initialization.

For general runtime architecture and backend selection, see [4.1](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/4.1) For tensor creation and memory operations that operate on an open device, see [4.3](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/4.3) For the `SystemDesc` generation used during compilation for hardware discovery, see [8.4](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/8.4)

## Core Concepts

### MeshDevice

All Tenstorrent hardware is accessed through a `MeshDevice`, representing a 2D grid of physical chips. It is characterized by:

*   **MeshShape**: A `[rows, cols]` tuple describing how many chips span each dimension [runtime/include/tt/runtime/runtime.h 168](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L168-L168)
*   **MeshOffset**: A `[row, col]` coordinate pinning the top-left corner of this mesh within the full system chip topology [runtime/include/tt/runtime/types.h 209](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L209-L209)
*   **Device IDs**: An optional explicit list of physical chip IDs to include [runtime/include/tt/runtime/types.h 210](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L210-L210)

A **parent mesh** owns the underlying hardware resources. A **submesh** is a logical view into a rectangular sub-region of a parent mesh. Submeshes share the parent's hardware but carry their own handle and trace cache. Only parent meshes are opened and closed; submeshes are created and released against their parent [runtime/include/tt/runtime/runtime.h 154-163](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L154-L163)

### Device Handle

The `Device` type is defined in [runtime/include/tt/runtime/types.h](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h) and wraps:

*   A `shared_ptr<void>` pointing to the backend-specific mesh device (`::ttnn::MeshDevice` or `::tt::tt_metal::distributed::MeshDevice`) [runtime/include/tt/runtime/types.h 102](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L102-L102)
*   A `DeviceRuntime` tag (`TTNN` or `TTMetal`) enabling safe type casting [runtime/include/tt/runtime/types.h 103](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L103-L103)
*   An optional `shared_ptr<TraceCache>` for trace execution caching (populated by the TTNN backend only) [runtime/include/tt/runtime/types.h 102](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L102-L102)

### Singleton Device Context (OpModel)

For compiler-side analysis (OpModel), a `SingletonDeviceContext` is used to manage a global device instance required for querying operation constraints and runtimes. This context can operate in "Mock Device" mode for environments without physical hardware.

Sources: [runtime/include/tt/runtime/runtime.h 154-168](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L154-L168)[runtime/include/tt/runtime/types.h 102-210](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L102-L210)
