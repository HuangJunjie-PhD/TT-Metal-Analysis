vLLM provides an OpenAI-compatible HTTP API server implemented with FastAPI. The server exposes endpoints that match OpenAI's API specification, enabling drop-in replacement for applications using the OpenAI Python client or any OpenAI-compatible client.

## Overview

The server is implemented in [vllm/entrypoints/openai/api_server.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/openai/api_server.py) It creates a FastAPI application with route handlers organized by endpoint type. Each endpoint is handled by a specialized serving class that inherits from `OpenAIServing`.

## Dependency Injection Pattern

FastAPI's dependency injection is used to provide serving instances to route handlers. This is implemented via getter functions that retrieve instances from `app.state`:

Example from [vllm/entrypoints/openai/api_server.py 288-342](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/openai/api_server.py#L288-L342):

*   `chat(request: Request) -> OpenAIServingChat` returns `request.app.state.openai_serving_chat`
*   `completion(request: Request) -> OpenAIServingCompletion` returns `request.app.state.openai_serving_completion`
*   `embedding(request: Request) -> OpenAIServingEmbedding` returns `request.app.state.openai_serving_embedding`

Route handlers use these as dependencies:

`@router.post("/v1/chat/completions")async def create_chat_completion(    request: ChatCompletionRequest,    raw_request: Request):    handler = chat(raw_request)  # Get serving instance    generator = await handler.create_chat_completion(request, raw_request)    # ...`

## Completion Implementation

`OpenAIServingCompletion` handles `/v1/completions` for text completion without chat formatting.

The implementation at [vllm/entrypoints/openai/serving_completion.py 80-314](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/openai/serving_completion.py#L80-L314) follows a similar pattern to chat completions:

1.   Validate model with `_check_model()`
2.   Get tokenizer and render prompts
3.   Convert to `SamplingParams` or `BeamSearchParams`
4.   Call `engine_client.generate()` or `engine_client.beam_search()`
5.   Stream or return full response

Key difference: accepts raw prompts (strings or token IDs) without chat template formatting.

## Embedding Implementation

`OpenAIServingEmbedding` handles `/v1/embeddings` via pooling operations.

The `EmbeddingMixin` class at [vllm/entrypoints/openai/serving_embedding.py 57-274](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/openai/serving_embedding.py#L57-L274) provides:

*   Chunked processing support for long inputs (splits into multiple requests)
*   Mean/max/first pooling aggregation
*   Base64 or float encoding
*   Chat-style embedding requests via `EmbeddingChatRequest`

The flow:

1.   Preprocess inputs (tokenize or apply chat template)
2.   Create `PoolingParams` from request
3.   Call `engine_client.encode()` for each prompt/chunk
4.   Aggregate results if chunked
5.   Return `EmbeddingResponse` with embeddings

## Differences from Official OpenAI API

While vLLM strives for high compatibility with the OpenAI API, there are some differences and extensions:

| Feature | Difference |
| --- | --- |
| Models | Only locally loaded models are available |
| Additional Endpoints | vLLM adds non-standard endpoints like `/pooling`, `/score`, and `/rerank` |
| Parameters | vLLM adds additional parameters like `guided_json`, `guided_regex`, and `logits_processors` |
| Performance | Performance characteristics depend on local hardware rather than OpenAI's infrastructure |
| Streaming | Implementation is identical, using Server-Sent Events (SSE) |
| Authentication | Simple API key authentication without user/organization management |
| Rate Limiting | Not implemented by default |

## Conclusion

vLLM's OpenAI API compatibility layer provides a seamless experience for users transitioning from OpenAI's hosted services to self-hosted vLLM deployments. The implementation closely follows OpenAI's API design, making it easy to use existing client libraries and code without modification. Additionally, vLLM extends the API with custom endpoints for advanced use cases like pooling and reranking.

For information about model support in vLLM, see [Supported Models](https://deepwiki.com/tenstorrent/vllm/1.2-supported-models).
