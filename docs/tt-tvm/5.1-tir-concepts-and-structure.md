## Purpose and Scope

This document explains the core concepts and structural components of TensorIR (TIR), the low-level tensor program intermediate representation in TVM. TIR is the layer where kernel-level optimizations are expressed through loop transformations, memory management, and compute scheduling.

**Related pages:**

*   For information about applying transformations to TIR programs, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api)
*   For the TVMScript syntax used to write TIR programs, see [TVMScript DSL](https://deepwiki.com/tenstorrent/tt-tvm/5.6-tvmscript-dsl)
*   For TIR-level operators and intrinsics, see [TIR Operators](https://deepwiki.com/tenstorrent/tt-tvm/5.2-tir-operators)

## TensorIR in the Compilation Pipeline

TensorIR sits between high-level graph IR (Relay) and low-level code generation. After Relay performs graph-level optimizations, it lowers operations to TIR, which then undergoes loop-level transformations before code generation.

## PrimFunc: The Top-Level Container

A `PrimFunc` (primitive function) is the top-level construct in TensorIR representing a single kernel or computational stage. It contains:

| Component | Description | Example |
| --- | --- | --- |
| **Parameters** | Input/output buffer handles | `a: T.handle, b: T.handle` |
| **Buffer Bindings** | Match parameters to shaped buffers | `A = T.match_buffer(a, [128, 128])` |
| **Body** | The computation (loops and blocks) | `for i in range(128): ...` |
| **Attributes** | Metadata like `tir.noalias` | `T.func_attr({"tir.noalias": True})` |

## Buffer: Memory Abstraction

A `Buffer` in TIR represents a multi-dimensional memory allocation with specific properties:

### Buffer Properties

| Property | Description | Example |
| --- | --- | --- |
| **data** | Pointer to the memory | `T.handle` type |
| **shape** | Dimensions of the buffer | `[1024, 1024]` |
| **dtype** | Element data type | `"float32"`, `"int32"` |
| **elem_offset** | Offset in elements | `0` |
| **scope** | Memory scope/location | `"global"`, `"shared"`, `"local"` |
| **align** | Alignment requirement | `64` bytes |
| **offset_factor** | Offset alignment factor | `1` |

### BufferRegion: Access Patterns

`BufferRegion` specifies which part of a buffer is accessed, consisting of:

*   A `Buffer` reference
*   A `Region` (list of `Range` objects) defining the accessed indices

## For Loops: Iteration Control

TIR provides several loop types to express different execution strategies:

### Loop Types and Attributes

| Loop Type | TVMScript | Purpose | Hardware Mapping |
| --- | --- | --- | --- |
| **Serial** | `T.serial(0, n)` | Sequential execution | Standard CPU loop |
| **Parallel** | `T.parallel(0, n)` | Parallel execution | OpenMP/pthread |
| **Vectorized** | `T.vectorized(0, n)` | SIMD vectorization | AVX, NEON |
| **Unroll** | `T.unroll(0, n)` | Loop unrolling | Instruction-level parallelism |
| **Thread Binding** | `T.thread_binding(0, n, "threadIdx.x")` | GPU thread mapping | CUDA/OpenCL threads |

## TIR Program Structure

A complete TIR program follows a hierarchical structure:

### Concrete Example Structure

From a matrix multiplication kernel:

```
IRModule
└── PrimFunc "mmult"
    ├── Parameters: A, B, C (handles)
    ├── Buffer bindings: A[1024,1024], B[1024,1024], C[1024,1024]
    ├── Attributes: {"tir.noalias": True}
    └── Body:
        └── For i (parallel, 0 to 32)
            └── For j (serial, 0 to 32)
                └── For k (serial, 0 to 256)
                    └── BlockRealize
                        └── Block "update"
                            ├── iter_vars: [vi, vj, vk]
                            ├── reads: [A[vi,vk], B[vj,vk], C[vi,vj]]
                            ├── writes: [C[vi,vj]]
                            ├── init: C[vi,vj] = 0
                            └── body: C[vi,vj] += A[vi,vk] * B[vj,vk]
```

## Random Variables: Schedule API Bridge

The Schedule API uses "random variables" (RVs) to refer to TIR constructs:

| RV Type | C++ Class | Purpose | Example |
| --- | --- | --- | --- |
| **BlockRV** | `BlockRVNode` | References a Block | `block_rv = sch.get_block("matmul")` |
| **LoopRV** | `LoopRVNode` | References a For loop | `loops = sch.get_loops(block_rv)` |
| **ExprRV** | `PrimExpr` (alias) | References an expression/integer | `factors = sch.split(loop, [4, 8])` |

These RVs are maintained in a symbol table by the Schedule, mapping to actual `StmtSRef` (statement references) in the IR.

**Sources:**[include/tvm/tir/schedule/schedule.h 51-99](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/schedule/schedule.h#L51-L99)[src/tir/schedule/concrete_schedule.h 38-49](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.h#L38-L49)[python/tvm/tir/schedule/schedule.py 42-69](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/tir/schedule/schedule.py#L42-L69)

## StmtSRef: The IR Navigation System

`StmtSRef` (Statement Scheduled Reference) is an internal structure used to navigate the TIR tree during scheduling. It maintains:

*   A pointer to a statement node (`stmt`)
*   A pointer to the parent `StmtSRef` (`parent`)
*   A sequence index within the parent (`seq_index`)

This creates a "sref tree" parallel to the IR tree, enabling efficient navigation and tracking during transformations. The Schedule maintains a `stmt2ref` map from IR nodes to their SRefs.

**Sources:**[src/tir/schedule/concrete_schedule.cc 49-102](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/concrete_schedule.cc#L49-L102)[src/tir/schedule/state.h (implied)](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/schedule/state.h%20(implied))

## Affine Binding and Complete Blocks

TIR enforces certain properties for correctness and optimization:

### Affine Block Binding

A block has **affine binding** if all `iter_values` in its `BlockRealize` are affine expressions of enclosing loop variables. This ensures predictable data access patterns.

### Complete Block

A block is **complete** if it satisfies:

1.   All block vars are data parallel
2.   The block is the only writer of its outputs (dominance)
3.   No overlap between read and write buffers

### Reduction Block

A block is a **reduction block** if:

1.   It has an `init` statement
2.   All block bindings are affine
3.   Block vars are data parallel or reduction
4.   The block is dominant
5.   Reduction vars don't index output buffers

## Summary: Key TIR Concepts

| Concept | Role | Key Properties |
| --- | --- | --- |
| **PrimFunc** | Top-level function | Parameters, buffer map, body, attributes |
| **Block** | Computation unit | iter_vars, reads, writes, body, init |
| **BlockRealize** | Block instantiation | iter_values, predicate, block |
| **Buffer** | Memory abstraction | shape, dtype, scope, alignment |
| **BufferRegion** | Access pattern | buffer + region (ranges) |
| **IterVar** | Iterator variable | type (data parallel/reduction), domain |
| **For** | Loop construct | type (serial/parallel/vectorized/unroll) |
| **StmtSRef** | IR navigation | Parent-child tree structure |
| **BlockRV/LoopRV** | Schedule API handles | Maps to StmtSRef via symbol table |

TensorIR provides a structured, analyzable representation of tensor programs where computation (blocks), iteration (loops), and memory (buffers) are explicitly modeled. This enables powerful transformations through the Schedule API while maintaining correctness guarantees.
