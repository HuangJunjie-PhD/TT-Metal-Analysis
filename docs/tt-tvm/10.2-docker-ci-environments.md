This document describes the specialized Docker container images used in TVM's continuous integration system. These images provide reproducible testing environments across diverse hardware targets, frameworks, and build configurations.

For information about the Jenkins pipeline that orchestrates these Docker images, see [Jenkins Pipeline System](https://deepwiki.com/tenstorrent/tt-tvm/10.1-jenkins-pipeline-system). For details on the Python dependency management used within these environments, see [Python Environment Setup](https://deepwiki.com/tenstorrent/tt-tvm/10.3-python-environment-setup).

## Overview

TVM's CI infrastructure maintains multiple Docker images, each tailored to test specific hardware targets, architectures, or build configurations. All images are defined in `docker/` directory with names following the pattern `Dockerfile.ci_<target>`. Each image is built on Ubuntu (primarily 22.04) and incorporates modular installation scripts to configure dependencies.

## Common Infrastructure Components

All Docker images share a standardized set of utilities and setup procedures to ensure consistency across environments.

### Core Utilities

Every image begins by copying the `apt-install-and-clear` utility, which wraps package installation with automatic cleanup to minimize image size:

`COPY utils/apt-install-and-clear.sh /usr/local/bin/apt-install-and-clear`
### Standard Setup Sequence

All images follow this initialization sequence:

1.   **Timezone Configuration**: [docker/install/ubuntu_setup_tz.sh 1-26](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/install/ubuntu_setup_tz.sh#L1-L26) - Sets `TZ=America/Los_Angeles` to avoid interactive prompts

## Python Environment Configuration

### Virtual Environment Structure

All images use a standardized Python virtual environment:

```
Environment Variable: TVM_VENV=/venv/apache-tvm-py3.8
Python Version: 3.8 (with support for 3.7)
Activation: PATH includes ${TVM_VENV}/bin
Isolation: PYTHONNOUSERSITE=1 (disables ~/.local)
```

The Python installation process [docker/install/ubuntu_install_python.sh 84-94](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/install/ubuntu_install_python.sh#L84-L94) creates the venv and ensures compatibility by symlinking `python3.X` executables for CMake's `find_package(PythonInterp)`.

### Package Constraints

Python packages are installed with hash verification from lockfiles in `python/bootstrap/lockfiles/constraints-3.8.txt`. The pip version is pinned and upgraded first [docker/install/ubuntu_install_python.sh 98-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/install/ubuntu_install_python.sh#L98-L100) before installing other dependencies with `--require-hashes` flag [docker/install/ubuntu_install_python.sh 106-108](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/install/ubuntu_install_python.sh#L106-L108)

### Base Python Dependencies

The `ubuntu_install_python_package.sh` script installs core testing dependencies:

*   **Testing**: pytest, pytest-xdist, pytest-profiling, pytest-rerunfailures, pytest-lazy-fixture
*   **Type Checking**: mypy
*   **Scientific**: numpy==1.21.*, scipy, Pillow==9.1.0
*   **Infrastructure**: Jinja2, requests, cloudpickle, tornado
*   **Build**: cython, packaging
*   **Utilities**: psutil, decorator, orderedset, junitparser

### Environment Variables

`PATH=/usr/local/nvidia/bin:/usr/local/cuda/bin:${PATH}CPLUS_INCLUDE_PATH=/usr/local/cuda/include:${CPLUS_INCLUDE_PATH}C_INCLUDE_PATH=/usr/local/cuda/include:${C_INCLUDE_PATH}LIBRARY_PATH=/usr/local/cuda/lib64:/usr/local/cuda/compat:${LIBRARY_PATH}LD_LIBRARY_PATH=/usr/local/cuda/lib64:/usr/local/cuda/compat:${LD_LIBRARY_PATH}LD_LIBRARY_PATH=/usr/lib/x86_64-linux-gnu/:${LD_LIBRARY_PATH}  # Priority for local libcudaLD_LIBRARY_PATH=/opt/rocm/lib:${LD_LIBRARY_PATH}VULKAN_SDK=/usrNVIDIA_DRIVER_CAPABILITIES=compute,graphics,utility`

### TensorFlow and TFLite

TensorFlow installation varies by architecture:

## Docker Image Build Patterns

### Layer Ordering Strategy

Dockerfiles follow this optimization pattern to minimize rebuild time:

1.   **Fast-changing utilities first:**`apt-install-and-clear.sh`
2.   **System configuration:** Timezone, core packages, CMake
3.   **Python environment:** Virtual environment creation, base packages
4.   **Large, slow-building dependencies:** Rust (marked with comment "build early; takes a while")
5.   **Compiler toolchains:** LLVM, target-specific compilers
6.   **Framework installations:** TensorFlow, PyTorch, etc.
7.   **Testing infrastructure:** Redis, additional tools

This ordering ensures that changes to TVM code don't invalidate slow-building layers like Rust compilation.

### Common Installation Script Pattern

All installation scripts follow this structure:

`#!/bin/bashset -e          # Exit on errorset -u          # Exit on undefined variableset -o pipefail # Fail on pipe errors # Script logicapt-install-and-clear -y <packages># orpip3 install <packages>`
**Sources:**[docker/Dockerfile.ci_cpu 60-65](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/Dockerfile.ci_cpu#L60-L65)[docker/install/ubuntu_install_rust.sh](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/docker/install/ubuntu_install_rust.sh)

## Relationship to CI Pipeline

These Docker images are referenced by name in Jenkins pipeline configurations. The Jenkins system (see [Jenkins Pipeline System](https://deepwiki.com/tenstorrent/tt-tvm/10.1-jenkins-pipeline-system)) pulls these images and executes build/test commands within them.

**Image Tagging:** Most images include version tags in comments (e.g., `# tag: v0.60`) to track breaking changes and coordinate updates across the CI infrastructure.

**Build Matrix:** The Jenkins pipeline uses these images to create a test matrix covering:

*   Multiple Python versions (via separate base images)
*   Different hardware targets (CPU, GPU, ARM, embedded)
*   Various ML frameworks (TensorFlow, PyTorch, ONNX, etc.)
*   Cross-compilation scenarios (x86 → ARM, x86 → RISC-V)
