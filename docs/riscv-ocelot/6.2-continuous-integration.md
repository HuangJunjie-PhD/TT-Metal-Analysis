This document explains the Continuous Integration (CI) system used by the RISC-V Ocelot repository. It covers the CI pipeline setup, the testing framework, configurations tested, and how to interpret and debug CI results. For information about building and configuring the system without using CI, see [Building and Configuration](https://deepwiki.com/tenstorrent/riscv-ocelot/6-building-and-configuration).

## Overview

The RISC-V Ocelot repository uses CircleCI for continuous integration, which automatically tests code changes to ensure they don't break existing functionality. The CI system performs the following key functions:

1.   Builds multiple BOOM configurations (small, medium, large, mega, etc.)
2.   Runs RISC-V ISA tests on all configurations
3.   Runs Csmith random tests to stress test the processor
4.   Performs FireSim-based hardware simulation tests
5.   Ensures code style consistency

The CI system helps maintain code quality by detecting regressions early and validating that changes work across all supported configurations.

## CI Infrastructure

### Build Environment

The CI system runs inside a Docker container with all necessary dependencies pre-installed. This ensures a consistent build environment across all CI runs.

The Docker container (`ucbbar/chipyard-image:1.0.1`) contains:

*   RISC-V and ESP toolchains
*   Verilator for RTL simulation
*   SBT for Scala building
*   Other dependencies for building and testing

The CI executes some work directly in the Docker container and offloads heavier computation to remote build servers via SSH.

Sources: [.circleci/config.yml 10-16](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/config.yml#L10-L16)[.circleci/defaults.sh 93-148](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/defaults.sh#L93-L148)

### Remote Execution

To speed up builds and tests, the CI system uses remote execution on more powerful machines:

1.   A build server for RTL generation and Verilator simulation
2.   AWS instances for FireSim FPGA-based testing

This distributed approach allows parallelization of different configurations and tests.

Sources: [.circleci/defaults.sh 10-92](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/defaults.sh#L10-L92)[.circleci/README.md 1-54](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/README.md?plain=1#L1-L54)

## CI Workflow

The main CI workflow builds and tests multiple BOOM configurations in parallel:

This workflow runs automatically for every pull request and commit to the master branch.

Sources: [.circleci/config.yml 489-580](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/config.yml#L489-L580)

### Toolchain Installation

The CI first builds or restores from cache the necessary toolchains:

1.   RISC-V toolchain - for standard RISC-V configurations
2.   ESP toolchain - for Hwacha vector extension support

These toolchains are built using scripts from the Chipyard repository and cached to speed up future CI runs.

Sources: [.circleci/build-toolchains.sh 1-33](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/build-toolchains.sh#L1-L33)[.circleci/create-hash.sh 1-38](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/create-hash.sh#L1-L38)

### Environment Preparation

Next, the CI prepares the build environment by:

1.   Cloning the Chipyard repository
2.   Checking out the specific Chipyard version specified in CHIPYARD.hash
3.   Initializing submodules
4.   Moving the BOOM repository code into the Chipyard repository structure
5.   Installing Verilator for RTL simulation

Sources: [.circleci/prepare-for-rtl-build.sh 1-44](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/prepare-for-rtl-build.sh#L1-L44)[CHIPYARD.hash 1-2](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/CHIPYARD.hash#L1-L2)

### Configuration Builds

The CI system then builds multiple BOOM configurations in parallel, each with different parameters:

| Configuration | Description | Parameters |
| --- | --- | --- |
| SmallBoomConfig | Small, simple configuration | Fewer execution units, smaller issue width |
| MediumBoomConfig | Medium-sized configuration | Balanced resources |
| LargeBoomConfig | Large, high-performance configuration | More execution units, wider issue |
| MegaBoomConfig | Very large configuration | Maximum resources, widest issue |
| HwachaBoomConfig | Configuration with vector support | Includes Hwacha vector extension |
| SmallRV32BoomConfig | 32-bit configuration | RV32 instead of RV64 |

Each configuration is built using a remote execution model to generate Verilog from the Chisel RTL.

Sources: [.circleci/do-rtl-build.sh 1-62](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/do-rtl-build.sh#L1-L62)[.circleci/defaults.sh 140-148](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/defaults.sh#L140-L148)

### Test Execution

After building each configuration, the CI runs multiple test suites:

1.   RISC-V ISA tests - Verify compliance with the RISC-V specification

    *   Assembly tests (`asm-tests`)
    *   Benchmark tests (`bmark-tests`)

2.   Csmith random tests - Generate random C programs to test the processor under stress

    *   Multiple random programs are generated and executed for each configuration

Sources: [.circleci/run-tests.sh 1-61](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/run-tests.sh#L1-L61)[.circleci/build-run-csmith-tests.sh 1-28](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/build-run-csmith-tests.sh#L1-L28)

## FireSim Integration

In addition to Verilator-based simulation, the CI system also supports FireSim testing on AWS FPGA instances for faster and more thorough testing.

The FireSim workflow:

1.   Initializes an AWS manager instance
2.   Builds FPGA images (AFIs) for the LargeFireBoom configuration
3.   Builds workloads (Buildroot, Fedora, CoreMark, SPEC17)
4.   Launches FireSim simulations with these workloads
5.   Collects and reports results

This workflow runs on a schedule (weekly) to test larger workloads that would be impractical to run in Verilator simulation.

Sources: [.circleci/config.yml 582-641](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/config.yml#L582-L641)[.circleci/README.md 20-54](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/README.md?plain=1#L20-L54)

## Testing Matrix

The following table shows which tests are run on each configuration:

| Configuration | RISC-V ASM Tests | RISC-V Benchmark Tests | Csmith Tests | FireSim |
| --- | --- | --- | --- | --- |
| SmallBoomConfig | ✓ | ✓ | ✓ (50 tests) | ✗ |
| MediumBoomConfig | ✓ | ✓ | ✓ (50 tests) | ✗ |
| LargeBoomConfig | ✗ | ✓ | ✓ (40 tests) | ✗ |
| MegaBoomConfig | ✗ | ✓ | ✓ (40 tests) | ✗ |
| HwachaBoomConfig | ✓ (vector) | ✗ | ✗ | ✗ |
| LargeFireBoom | ✗ | ✗ | ✗ | ✓ |

The testing strategy focuses on:

*   Running more thorough tests on smaller configurations (faster to simulate)
*   Running specialized tests for specific features (e.g., vector tests for Hwacha)
*   Using FireSim for large workloads on the most representative configuration

## Running Tests Locally

To run the same tests locally as in the CI system:

1.   Clone the Chipyard repository and follow its setup instructions
2.   Place the BOOM repository in the correct location within Chipyard
3.   Use the standard Chipyard test commands:

`# RISC-V testsmake CONFIG=SmallBoomConfig run-asm-tests-fast -C sims/verilatormake CONFIG=SmallBoomConfig run-bmark-tests-fast -C sims/verilator # Csmith testscd generators/boom/util/csmith./install-csmith.sh./run-csmith.sh --sim ../../sims/verilator/simulator-chipyard-SmallBoomConfig --run 10 --nodebug`
These commands follow the same pattern used in the CI scripts but simplified for local execution.

Sources: [.circleci/run-tests.sh 18-29](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/.circleci/run-tests.sh#L18-L29)[docs/sections/boom-ecosystem.rst 1-117](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/boom-ecosystem.rst#L1-L117)

## Debugging CI Issues

When CI failures occur, the following debugging approaches can be helpful:

1.   Check the CircleCI logs for specific error messages

2.   Look for failed tests in the appropriate test category:

    *   ASM or benchmark test failures - Check for ISA compliance issues
    *   Csmith failures - Often indicate corner-case bugs in the pipeline
    *   FireSim failures - Check AWS logs for timing or hardware-specific issues

3.   Reproduce locally using the commands shown in the previous section

A chicken bit is also provided for debugging that can disable out-of-order execution by writing to CSR `0x7c1` with value `0x8`, allowing for simpler debugging.

Sources: [docs/sections/debugging.rst 1-20](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/docs/sections/debugging.rst#L1-L20)

## CI Configuration Management

The CI system configuration is managed through several key files:

1.   `.circleci/config.yml` - Main CircleCI workflow definition
2.   `.circleci/defaults.sh` - Shared variables and functions
3.   CHIPYARD.hash - Specifies the version of Chipyard to use

To modify the CI configuration:

1.   Update the appropriate configuration files
2.   Test changes locally when possible
3.   Submit a pull request with the changes

The CI infrastructure is designed to be portable and consistent, regardless of where it runs.
