## Purpose and Scope

The Performance Modeling API (`PerfApi`) provides a fine-grained interface for cycle-accurate performance modeling of RISC-V processors. This API enables external performance models (typically written in Verilog or SystemVerilog) to drive Whisper as a functional golden model while maintaining detailed control over instruction pipeline stages, register renaming, speculative execution, and memory ordering.

The API supports:

*   **Out-of-order execution simulation** with register renaming
*   **Store-to-load forwarding** and memory consistency modeling
*   **Speculative execution** with branch prediction and pipeline flushing
*   **Pipeline stage control** (fetch, decode, execute, retire, drain)
*   **Lock-step verification** between hardware and ISS

For basic command-based interaction with Whisper, see [Server API and Message Protocol](https://deepwiki.com/tenstorrent/whisper/6.1-server-api-and-message-protocol). For interactive debugging, see [Interactive Command-Line Interface](https://deepwiki.com/tenstorrent/whisper/6.2-interactive-command-line-interface).

## Architecture Overview

The Performance Modeling API sits at the boundary between external performance models and Whisper's core execution engine. It maintains shadow state representing in-flight instructions and provides mechanisms to execute instructions speculatively without modifying the hart's architectural state until retirement.

**Sources:**[PerfApi.hpp 444-798](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L444-L798)[PerfApi.cpp 24-38](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L24-L38)

## Core Concepts

### Instruction Packets

Each in-flight instruction is represented by an `InstrPac` object, identified by a unique monotonically increasing tag. The packet tracks the instruction's state through all pipeline stages and maintains information needed for speculative execution and register renaming.

**Sources:**[PerfApi.hpp 66-441](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L66-L441)[DecodedInst.hpp 37-580](https://github.com/tenstorrent/whisper/blob/733fd921/DecodedInst.hpp#L37-L580)

### Register Renaming

The API maintains a register renaming table (`hartRegProducers_`) that maps each architectural register to the most recent in-flight instruction that will produce its value. This enables out-of-order execution by allowing instructions to read operand values from producer instructions rather than from the register file.

**Global Register Indexing:**

| Register Type | Global Index Range | Count |
| --- | --- | --- |
| Integer (`x0-x31`) | 0-31 | 32 |
| Floating-Point (`f0-f31`) | 32-63 | 32 |
| Vector (`v0-v31` + groups) | 64-575 | 512 |
| CSR | 576-4671 | 4096 |

## Pipeline Stage API

### Fetch Stage

The `fetch()` method simulates instruction fetch, performing address translation and checking for fetch-time traps. It automatically calls `decode()` internally.

**Behavior:**

1.   Translates virtual PC (`vpc`) to physical address
2.   Fetches instruction bytes from memory
3.   Creates new `InstrPac` with the tag
4.   Automatically decodes the instruction
5.   Detects fetch-time exceptions (page faults, access faults)
6.   Returns trap information if exception occurs

**Sources:**[PerfApi.hpp 450-461](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L450-L461)[PerfApi.cpp 82-168](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L82-L168)

### Decode Stage

The `decode()` method decodes the instruction and determines operands and their producers (for register renaming).

**Behavior:**

1.   Decodes the instruction into `DecodedInst` structure
2.   Determines explicit operands (from instruction encoding)
3.   Determines implicit operands (e.g., VTYPE, VSTART for vector instructions)
4.   Identifies producer instructions for each source operand
5.   Updates register renaming table for destination operands
6.   Computes effective LMUL for vector operands

**Sources:**[PerfApi.hpp 463-467](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L463-L467)[PerfApi.cpp 172-270](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L172-L270)

### Execute Stage

The `execute()` method speculatively executes the instruction without committing architectural state changes.

**Behavior:**

1.   Collects source operand values (from register file or producer instructions)
2.   Temporarily pokes operand values into hart registers
3.   Executes instruction via `hart.singleStep()`
4.   Captures destination register values
5.   Restores hart state to pre-execution values
6.   Records execution results in the packet
7.   Captures page table walks for address translations

**Key Implementation Details:**

*   Instructions can be re-executed if they depend on a replayed load/store
*   Time adjustments ensure CSRs like `MCYCLE` remain consistent
*   Special handling for IMSIC TOPEI CSRs to preserve interrupt state
*   Load from I/O space is skipped during speculative execute (happens at retire)

**Sources:**[PerfApi.hpp 479-487](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L479-L487)[PerfApi.cpp 274-494](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L274-L494)

### Retire Stage

The `retire()` method commits the instruction's architectural state changes.

**Behavior:**

1.   Verifies instruction was executed and PC matches
2.   Executes instruction again (non-speculatively) via `hart.singleStep()`
3.   Compares execute-stage and retire-stage results (sanity check)
4.   Undoes register renaming for destination registers
5.   Generates instruction trace if enabled
6.   For multi-hart systems: drains AMO/SC instructions immediately

**Sources:**[PerfApi.hpp 489-493](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L489-L493)[PerfApi.cpp 498-626](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L498-L626)

### Drain Stage

The `drainStore()` method commits store data to memory.

**Behavior:**

1.   Verifies instruction is a store/AMO/cbo_zero that has been retired
2.   Commits store data to physical memory
3.   For SC instructions: cancels any active LR reservation
4.   Removes packet from packet map and store map

**Notes:**

*   In single-hart systems, stores drain via this method
*   In multi-hart systems, AMO/SC drain at retire time
*   Regular stores can drain out-of-order (controlled by performance model)

**Sources:**[PerfApi.hpp 549-552](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.hpp#L549-L552)[PerfApi.cpp 811-874](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L811-L874)

## Register Renaming and Data Forwarding

### Register Renaming Mechanism

During decode, the API builds a dependency graph by recording which in-flight instruction produces each register. When an instruction is executed, source operand values are obtained from producer instructions rather than the register file.

**Implementation:**

*   `hartRegProducers_[hartIx][globalRegIx]` points to producer packet
*   Updated at decode time for destination operands
*   Queried at execute time for source operands
*   Undone at retire time (registers point back to register file)
*   Restored at flush time (for flushed destination registers)

**Sources:**[PerfApi.cpp 210-267](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L210-L267)[PerfApi.cpp 1629-1678](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L1629-L1678)

### Store-to-Load Forwarding

The API supports store-to-load forwarding, allowing loads to read data from earlier in-flight stores that have executed but not yet drained.

**Sources:**[PerfApi.cpp 878-976](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L878-L976)[PerfApi.cpp 980-1028](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L980-L1028)

## Memory Operations

### Load Data Retrieval

**Behavior:**

1.   If instruction already executed, return cached value
2.   Check for I/O device addresses (skip if speculative)
3.   Check for `toHost` address (return 0)
4.   Scan `hartStoreMaps_` for overlapping earlier stores
5.   Forward bytes from matching stores
6.   Read non-forwarded bytes from memory
7.   Return merged data

**Page-Crossing Loads:**

*   `pa1` is the physical address for first bytes
*   `pa2` is the physical address for remaining bytes (if crossing page)
*   Forwarding checks each byte individually

**Sources:**[PerfApi.cpp 878-976](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L878-L976)

### Store Data Specification

**Behavior:**

1.   Verifies packet is a store/AMO/vector-store/cbo_zero
2.   Stores physical addresses and data value in packet
3.   Builds byte-level map (`stDataMap_`) for forwarding
4.   Handles page-crossing stores

**Vector Stores:**

*   Multiple calls to `setStoreData()` for each element
*   `stDataMap_` accumulates all element bytes
*   Committed atomically at drain time

**Sources:**[PerfApi.cpp 980-1028](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L980-L1028)

## Branch Prediction Support

### Recording Predictions

Allows the performance model to record its branch prediction. The API automatically detects mispredictions when the instruction executes.

### Automatic Misprediction Detection

During `execute()`, the API compares predicted target (`prTarget_`) with actual target (`nextIva_`) and sets the `mispredicted_` flag.

**Sequential Fetch Detection:**

If no prediction is made for a branch, the API checks if the next fetched PC is sequential. If not, it automatically records a prediction.

## Flushing and Exception Handling

### Flush Operation

Flushes the instruction with the given tag and all younger (higher tag) instructions. This is used for:

*   Branch mispredictions
*   Exception handling
*   Pipeline squashes

**Behavior:**

1.   Iterate through packets in reverse order (newest to oldest)
2.   For each flushed packet's destination registers: 
    *   Restore producer pointer to previous producer
    *   Handle vector registers with LMUL > 1

3.   Remove packets from `hartPacketMaps_` and `hartStoreMaps_`
4.   Clear `prevFetch_` if it was flushed

**Sources:**[PerfApi.cpp 1089-1175](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L1089-L1175)

### Trap Handling

When `fetch()` or `execute()` encounters a trap:

1.   The `trap_` flag is set in the packet
2.   `trapCause_` contains the exception/interrupt code
3.   For fetch traps: instruction is marked as trapped immediately
4.   For execute traps: architectural state is updated at retire time
5.   The PC after trap handling is recorded in `nextIva_`

**Sources:**[PerfApi.cpp 140-168](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L140-L168)[PerfApi.cpp 414-418](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L414-L418)

## Advanced Features

### Vector Instruction Support

Vector instructions have special handling for:

*   **Effective LMUL:** Computed per operand based on EEW/SEW ratio
*   **Element-wise execution:** Each element can have separate address
*   **Segmented loads/stores:** Multiple fields per element
*   **Masked operations:**`vecAddrs_` tracks skipped elements

**Vector Register Renaming:**

Vector registers with LMUL > 1 occupy multiple register slots. The renaming table tracks each component:

```
v4 with LMUL=4 occupies:
  vecRegProducers_[globalRegIx(VecReg, 4) + 0] → packet
  vecRegProducers_[globalRegIx(VecReg, 4) + 1] → packet  
  vecRegProducers_[globalRegIx(VecReg, 4) + 2] → packet
  vecRegProducers_[globalRegIx(VecReg, 4) + 3] → packet
```

**Sources:**[PerfApi.cpp 204-207](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L204-L207)[PerfApi.cpp 228-265](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L228-L265)

### Hypervisor Support

For two-stage address translation:

*   `InstrPac` stores both guest physical (`gpc`) and physical addresses
*   Page table walks include both VS-stage and G-stage translations
*   `fetchWalks_` and `dataWalks_` contain nested walk vectors

**Sources:**[PerfApi.cpp 136-140](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L136-L140)

### Multi-Hart Coordination

Each hart maintains separate:

*   `hartPacketMaps_[hartIx]` - in-flight instruction packets
*   `hartStoreMaps_[hartIx]` - in-flight store packets
*   `hartLastRetired_[hartIx]` - last retired tag
*   `hartRegProducers_[hartIx]` - register renaming table

AMO and SC instructions drain at retire time in multi-hart systems to maintain memory ordering guarantees.

**Sources:**[PerfApi.cpp 27-37](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L27-L37)[PerfApi.cpp 591-607](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L591-L607)

## Performance Considerations

### Memory Overhead

Each `InstrPac` consumes approximately:

*   Base structure: ~400 bytes
*   Store data map: variable (per byte stored)
*   Vector addresses: variable (per element)
*   Page table walks: variable (per translation level)

The `getPacketSize()` method provides actual size calculation.

**Sources:**[PerfApi.cpp 1429-1454](https://github.com/tenstorrent/whisper/blob/733fd921/PerfApi.cpp#L1429-L1454)

### Optimization Strategies

1.   **Flush Early:** Remove mispredicted instructions as soon as detected
2.   **Drain Promptly:** Free store packets after draining
3.   **Limit In-Flight:** Control maximum in-flight instruction count
4.   **Clear Dependencies:** Packets release producer references at retire/drain

## Error Handling

The API uses assertions for programming errors (invalid arguments) and returns `false` for recoverable errors (traps, missing packets). Key checks:

*   **Tag uniqueness:** Tags must be monotonically increasing per hart
*   **Time monotonicity:** Time must not decrease
*   **Packet lifecycle:** Operations must follow fetch→decode→execute→retire→drain order
*   **Hart validity:** Hart index must be valid
*   **Execute vs Retire consistency:** Results must match (sanity check)
