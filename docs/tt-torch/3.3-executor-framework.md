## Purpose and Scope

The Executor Framework provides the execution layer that runs compiled model binaries on Tenstorrent hardware. Executors manage the entire runtime lifecycle including device initialization, input preprocessing, binary execution, output collection, and resource cleanup. The framework supports multiple execution strategies: full end-to-end execution, op-by-op compilation for debugging, and op-by-op execution with verification against golden PyTorch outputs.

For information about the compilation pipeline that produces the binaries executed by this framework, see [Compilation Pipeline](https://deepwiki.com/tenstorrent/tt-torch/3.1-compilation-pipeline). For details on device management and multi-device execution, see [Device Management and Parallelism](https://deepwiki.com/tenstorrent/tt-torch/3.5-device-management-and-parallelism).

## Base Executor Class

### Initialization and Configuration

The `Executor` class is initialized with a `MultiChipGraph` (mcg) that encapsulates the compiled binaries, graph structure, and input/output mappings. The `CompilerConfig` controls execution behavior including compile depth, verification settings, and mesh shape.

**Key initialization parameters:**

| Parameter | Type | Purpose |
| --- | --- | --- |
| `mcg` | `MultiChipGraph` | Contains compiled binaries and graph topology for single or multi-device execution |
| `compiler_config` | `CompilerConfig` | Controls compilation depth, verification, and execution settings |
| `required_pcc` | `float` | Minimum Pearson correlation coefficient for op-by-op verification (default: 0.99) |
| `required_atol` | `float` | Maximum absolute tolerance for op-by-op verification (default: 1e-2) |
| `devices` | `list` | Pre-allocated device handles; if None, executor creates default devices |
| `async_mode` | `bool` | Enables asynchronous execution returning on-device tensors without host transfer |
| `buffer_cache` | `dict` | Optional cache for reusing preprocessed buffer tensors across calls |
| `constant_cache` | `dict` | Optional cache for reusing preprocessed constant tensors across calls |

### Type Conversion for Hardware Compatibility

The executor maintains a type conversion dictionary to automatically cast unsupported PyTorch types to hardware-compatible types during execution:

The `typecast_inputs()` method applies these conversions before preprocessing inputs for device execution. This ensures compatibility with Tenstorrent hardware constraints while maintaining functional correctness.

### Device Management

Executors manage device lifecycle through system descriptors and mesh device handles:

The `_create_system_descriptors()` method generates `.ttsys` system descriptor files for each device, which contain hardware configuration metadata required by the tt-mlir runtime. For user-provided devices, descriptors are created but devices remain open. For executor-owned devices (created when `devices[i] is None`), devices may be closed after descriptor creation if not needed for subsequent operations.

Sources: [tt_torch/dynamo/executor.py 204-226](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L204-L226)[tt_torch/dynamo/executor.py 264-278](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L264-L278)

### Three-Level Caching Architecture

The executor implements a sophisticated caching system to avoid redundant tensor preprocessing across multiple invocations:

```
Cache Structure:
  buffer_cache[device][torch_tensor] -> runtime_tensor
  constant_cache[device][torch_tensor] -> runtime_tensor
```

**Buffer Cache:** Stores preprocessed activations and intermediate tensors that change between invocations but may be reused within a session.

**Constant Cache:** Stores preprocessed model weights and constants that remain static across all invocations. The cache is validated and potentially invalidated if the set of constants changes due to graph decompositions or different execution paths.

The `_try_initialize_or_invalidate_constant_cache()` method handles cache invalidation scenarios, particularly important when the same cache is shared between multiple executors (e.g., prefill and decode in LLM inference) that may have different constant sets due to FX-level decompositions.

Sources: [tt_torch/dynamo/executor.py 176-203](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L176-L203)[tt_torch/dynamo/executor.py 284-323](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L284-L323)

### Input Preprocessing Flow

The `get_inputs()` method orchestrates the transformation of PyTorch tensors to runtime tensors compatible with tt-mlir execution:

**Preprocessing Steps:**

1.   **Cache Lookup:** Check `constant_cache` and `buffer_cache` for existing runtime tensors matching input torch tensors by Python `id()`
2.   **Type Conversion:** Apply `typecast_inputs()` to convert unsupported types
3.   **Runtime Conversion:** Call `tt_mlir.preprocess_inputs()` to perform tiling, layout transformation, and device placement
4.   **Cache Population:** Store newly created runtime tensors in appropriate cache for future reuse

The method separates constants, buffers, and user inputs, preprocessing only those not already cached. This optimization significantly reduces latency for repeated invocations with the same model weights.

Sources: [tt_torch/dynamo/executor.py 325-456](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L325-L456)

### Execution and Output Collection

The `__call__()` method implements the main execution path, supporting both single-device and multi-device execution:

**Multi-Device Execution:** When `mcg.binaries` contains multiple entries, the executor orchestrates pipeline-parallel execution by routing intermediate results between devices according to `IOType` annotations. Outputs marked `IOType.INTER_DEVICE` are fed as inputs to subsequent device programs, while `IOType.USER` outputs are collected for final return.

**Asynchronous Mode:** When `async_mode=True`, `tt_mlir.run_async()` returns immediately with on-device runtime tensors without transferring to host. This is useful for chaining multiple model invocations or implementing custom post-processing on device.

Sources: [tt_torch/dynamo/executor.py 458-578](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L458-L578)

## OpByOpExecutor Class

The `OpByOpExecutor` extends the base `Executor` to support granular operation-level compilation and execution, critical for debugging compilation failures and verifying correctness against PyTorch reference implementations.

### Compilation and Execution Modes

OpByOpExecutor operates in two primary modes controlled by `CompileDepth`:

| Mode | CompileDepth | Behavior |
| --- | --- | --- |
| Compile-only | `COMPILE_OP_BY_OP` | Extracts and compiles each operation to TTNN binary without execution |
| Execute and verify | `EXECUTE_OP_BY_OP` | Compiles each operation, executes on device, and compares against PyTorch golden output |

The class maintains global state across graph breaks to track all operations in a model:

Sources: [tt_torch/dynamo/executor.py 617-667](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L617-L667)

### Process-Based Isolation

To isolate compilation and execution from the main process and enable timeout handling, OpByOpExecutor uses multiprocessing:

**Compilation Process (`compile_process`):**

*   Runs in isolated process to prevent crashes from blocking main execution
*   Performs StableHLO → TTIR → TTNN → Binary transformation
*   Reports progress via multiprocessing queues with events for synchronization
*   Subject to `single_op_timeout` (default: 120s) to prevent indefinite hangs

**Execution Process (`execute_process`):**

*   Long-lived process reused across operations to amortize device initialization cost
*   Handles large inputs by serializing to disk when size exceeds `op_memory_limit` (512MB default)
*   Redirects stderr to temporary file for capturing runtime stack dumps
*   Terminates and recreates on timeout or crash

Sources: [tt_torch/dynamo/executor.py 67-81](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L67-L81)[tt_torch/dynamo/executor.py 99-138](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L99-L138)[tt_torch/dynamo/executor.py 753-816](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L753-L816)[tt_torch/dynamo/executor.py 818-911](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L818-L911)

### Operation Compilation Flow

The compilation process communicates through queues and events to allow incremental progress reporting. The main process can inspect intermediate IR representations (TTIR, TTNN) even if final binary compilation fails, aiding in debugging.

Sources: [tt_torch/dynamo/executor.py 753-816](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L753-L816)

### Operation Execution and Verification

When `compile_depth=EXECUTE_OP_BY_OP`, the executor runs each operation and compares against PyTorch golden output:

**Execution Steps:**

1.   **Golden Reference:** Execute operation using PyTorch with `cast_ios_and_run()` to handle dtype conversions
2.   **Binary Execution:** Call `run_op(binary, *inputs)` which invokes `tt_mlir.run_end_to_end()` in execution process
3.   **Metric Calculation:** Compute PCC (Pearson correlation coefficient) and ATOL (absolute tolerance) between calculated and golden outputs
4.   **Verification:** Compare metrics against `required_pcc` and `required_atol` thresholds

The `print_marker()` method logs progress with timing information:

```
Processing  global_op_idx: 42 (15/120): aten.linear.default | time: 0.0000 s |
Compiling   global_op_idx: 42 (15/120): aten.linear.default | time: 2.3451 s |
Golden      global_op_idx: 42 (15/120): aten.linear.default | time: 0.0123 s |
Running     global_op_idx: 42 (15/120): aten.linear.default | time: 0.4567 s |
```

## Specialized Executor Classes

### TorchExecutor

`TorchExecutor` handles execution of PyTorch FX graphs with op-by-op support. It traverses the FX graph, extracting individual operations, compiling them to StableHLO, and optionally executing with verification.

**Key Responsibilities:**

*   Extract single operations from FX graph into isolated torch-mlir modules
*   Handle multi-output operations using `operator.getitem` nodes
*   Manage placeholder metadata propagation for type inference
*   Implement `run_gm_op_by_op()` to execute graph with reference tracking and memory management

The `get_stable_hlo_graph()` method constructs a minimal FX graph containing only the target operation, imports it to torch-mlir, and lowers to StableHLO:

### StablehloExecutor

`StablehloExecutor` operates on pre-compiled StableHLO IR, extracting individual StableHLO operations for compilation to TTNN. This executor is used when starting from ONNX models or when debugging at the StableHLO IR level.

**Unique Features:**

*   Parses StableHLO module string to extract individual operations
*   Creates `StablehloOp` objects with input/output shape inference from IR signatures
*   Generates random inputs matching StableHLO tensor types for testing
*   Supports fallback to ONNX inference session or PyTorch graph module if binary execution fails

### OnnxExecutor

`OnnxExecutor` provides a lightweight execution path for ONNX models, supporting both ONNX Runtime inference and compiled TTNN binary execution.

**Execution Modes:**

*   **ONNX Runtime:** When `binary=None` or `compile_depth=TTNN_IR`, uses ONNXRuntime inference session
*   **TTNN Binary:** When binary is available, executes via `tt_mlir.run_end_to_end()`

Unlike other executors, OnnxExecutor does not support op-by-op modes. Users should convert ONNX to StableHLO and use `StablehloExecutor` for granular debugging.

## Complete Execution Data Flow

The following diagram illustrates the end-to-end data flow through the Executor framework, from PyTorch tensors to device execution and back:

**Key Insights:**

1.   **Cache-First Strategy:** Runtime tensors are reused aggressively through buffer and constant caches, avoiding redundant preprocessing
2.   **Type Safety:** Type conversion happens early to ensure hardware compatibility before expensive preprocessing
3.   **Multi-Device Orchestration:** IOType annotations control data routing between devices in pipeline-parallel execution
4.   **Resource Management:** Activation runtime tensors are explicitly deallocated after use; cached constant/buffer tensors persist

## Integration with Runtime and Backend

Executors interact with two primary external systems:

**tt_mlir Runtime Module:** Provides low-level device operations through pybind11 bindings:

*   `tt_mlir.open_mesh_device()` / `tt_mlir.close_mesh_device()` - Device lifecycle management
*   `tt_mlir.preprocess_inputs()` - Convert torch tensors to runtime tensors with tiling/layout
*   `tt_mlir.run()` / `tt_mlir.run_async()` - Execute binary on device
*   `tt_mlir.compile_stable_hlo_to_ttir()` / `tt_mlir.compile_ttir_to_bytestream()` - Compilation functions used by OpByOpExecutor

**MultiChipGraph (mcg):** Contains the compiled artifacts and execution metadata:

*   `mcg.binaries` - Dictionary mapping device_idx to flatbuffer binaries
*   `mcg.constant_inputs` - Model weights per device
*   `mcg.buffers` - Persistent state tensors per device
*   `mcg.graph_inputs` / `mcg.graph_outputs` - Input/output specifications with IOType annotations
*   `mcg.programs` - Optional torch.fx.GraphModule for non-EXECUTE depths

For details on MultiChipGraph construction and device partitioning, see [Device Management and Parallelism](https://deepwiki.com/tenstorrent/tt-torch/3.5-device-management-and-parallelism).
