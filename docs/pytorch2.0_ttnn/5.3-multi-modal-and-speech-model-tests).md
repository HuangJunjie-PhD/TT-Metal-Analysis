## Purpose and Scope

This document covers the test suite for multi-modal and speech models in the pytorch2.0_ttnn repository. Multi-modal models process multiple input modalities (e.g., vision and language), while speech models handle audio processing tasks. These models require specialized input preprocessing and handling compared to single-modality models.

For single-modality vision models, see [Computer Vision Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.1-computer-vision-model-tests). For single-modality language models, see [Natural Language Processing Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.2-natural-language-processing-model-tests). This document focuses on models that combine multiple modalities or handle speech/audio data.

**Models Covered:**

*   **Vision-Language Models**: CLIP (Contrastive Language-Image Pre-training), ViLT (Vision-and-Language Transformer)
*   **Vision Transformer Variants**: BEiT (BERT Pre-Training of Image Transformers), DeiT (Data-efficient Image Transformers)
*   **Speech Models**: SpeechT5 (Unified-Modal Encoder-Decoder Pre-training for Spoken Language Processing)

## Multi-Modal Model Testing Architecture

Multi-modal models differ from single-modality models in their input requirements and preprocessing pipelines. The `ModelTester` framework [tests/utils.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py) accommodates these differences through flexible input handling.

**Key Differences from Single-Modality Tests:**

| Aspect | Single-Modality | Multi-Modal |
| --- | --- | --- |
| Input Structure | Single tensor | Dictionary of tensors or multiple tensors |
| Preprocessing | Single transform pipeline | Multiple processors (vision, text, audio) |
| Processor Objects | Optional (transforms.Compose) | Required (HuggingFace Processors) |
| Batch Handling | Simple tensor repetition | Dictionary-aware repetition with `repeat_inputs()` |

Sources: [tests/models/vilt/test_vilt.py 1-50](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L1-L50)

## Vision-Language Models

Vision-language models combine visual and textual understanding, requiring synchronized preprocessing of both image and text inputs.

### ViLT (Vision-and-Language Transformer)

ViLT is a visual question answering model that processes both images and text questions to produce answers.

**Implementation Details:**

The ViLT test follows the standard `ModelTester` pattern with multi-modal adaptations:

1.   **Model Initialization**[tests/models/vilt/test_vilt.py 15-18](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L15-L18):

    *   Uses `ViltProcessor` for input preprocessing
    *   Loads `ViltForQuestionAnswering` model
    *   Converts to `torch.bfloat16` for Tenstorrent hardware compatibility

2.   **Input Preparation**[tests/models/vilt/test_vilt.py 20-29](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L20-L29):

    *   Downloads sample image from COCO dataset
    *   Defines text question ("How many cats are there?")
    *   Uses processor to create unified encoding with `pixel_values`, `input_ids`, and `attention_mask`
    *   Converts pixel values to `torch.bfloat16`
    *   Applies `repeat_inputs()` for batch size scaling

3.   **Test Execution**[tests/models/vilt/test_vilt.py 36-49](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L36-L49):

    *   Marked with `@pytest.mark.compilation_xfail` due to sampling issue
    *   Outputs predicted answer by taking argmax of logits and mapping to label

**Key Pattern:**

`# Multi-modal input encoding structureencoding = {    'pixel_values': tensor,  # Visual input    'input_ids': tensor,     # Text tokens    'attention_mask': tensor # Attention mask}`
Sources: [tests/models/vilt/test_vilt.py 1-50](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L1-L50)

### CLIP (Contrastive Language-Image Pre-training)

While not present in the provided files, CLIP follows a similar pattern to ViLT but focuses on image-text similarity rather than question answering. The expected structure would include:

*   **Processor**: `CLIPProcessor.from_pretrained()`
*   **Model**: `CLIPModel.from_pretrained()` or `CLIPVisionModel` / `CLIPTextModel`
*   **Input Encoding**: Dictionary with `pixel_values` and `input_ids`
*   **Output**: Image and text embeddings for similarity computation

## Vision Transformer Variants

Vision transformers like BEiT and DeiT adapt the transformer architecture to pure vision tasks. While they process only images (not multi-modal in the strictest sense), they're often grouped with multi-modal models due to their transformer-based architecture and pre-training methodologies.

### BEiT (BERT Pre-Training of Image Transformers)

BEiT applies masked image modeling (similar to BERT's masked language modeling) to vision transformers:

**Expected Test Structure:**

**Characteristics:**

*   Uses `BeitImageProcessor` for image preprocessing
*   Requires specific image normalization (ImageNet statistics)
*   Supports patch-based image encoding
*   Output is classification logits similar to standard vision models

### DeiT (Data-efficient Image Transformers)

DeiT focuses on efficient training of vision transformers with less data:

**Expected Test Structure:**

*   **Processor**: `DeiTImageProcessor` or `AutoImageProcessor`
*   **Model**: `DeiTForImageClassification` or `DeiTForImageClassificationWithTeacher`
*   **Input**: Single `pixel_values` tensor after preprocessing
*   **Distillation Token**: May include distillation head for teacher-student learning

## Speech Models

Speech models process audio inputs, requiring specialized feature extraction pipelines distinct from vision or text processing.

### SpeechT5

SpeechT5 is a unified-modal model that can handle speech recognition, speech synthesis, and other speech tasks:

**Key Characteristics:**

1.   **Audio Feature Extraction**:

    *   Uses `SpeechT5Processor` which wraps audio feature extractor and tokenizer
    *   Converts raw audio waveforms to model-compatible features
    *   May require specific sampling rates (typically 16kHz)

2.   **Dual-Modal Nature**:

    *   Speech-to-text: Audio input → Text output
    *   Text-to-speech: Text input → Audio output
    *   Requires different input/output handling based on task

3.   **Expected Input Structure**:

`# Speech-to-textinputs = {    'input_values': audio_features,  # Processed audio    'attention_mask': mask          # Optional} # Text-to-speechinputs = {    'input_ids': text_tokens,       # Text input    'speaker_embeddings': embeddings # Optional speaker info}`

Sources: Inferred from multi-modal model patterns and HuggingFace architecture

## Common Patterns in Multi-Modal Tests

### Input Dictionary Handling

Multi-modal models typically return dictionary inputs that must be properly batched:

The `repeat_inputs()` utility function [tests/utils.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py) handles recursive batching for complex input structures, ensuring that nested dictionaries and tensors are properly repeated to achieve the desired batch size.

### Processor-Based Preprocessing

Multi-modal models rely heavily on HuggingFace processors:

| Model Type | Processor Class | Key Methods | Output Keys |
| --- | --- | --- | --- |
| ViLT | `ViltProcessor` | `__call__(image, text)` | `pixel_values`, `input_ids`, `attention_mask` |
| CLIP | `CLIPProcessor` | `__call__(text, images)` | `pixel_values`, `input_ids`, `attention_mask` |
| BEiT | `BeitImageProcessor` | `__call__(images)` | `pixel_values` |
| DeiT | `DeiTImageProcessor` | `__call__(images)` | `pixel_values` |
| SpeechT5 | `SpeechT5Processor` | `__call__(audio, text)` | `input_values`, `input_ids` |

### Data Type Conversion

All multi-modal models convert inputs to `torch.bfloat16` for Tenstorrent hardware:

`# Vision-language exampleencoding = processor(image, text, return_tensors="pt")encoding["pixel_values"] = encoding["pixel_values"].to(torch.bfloat16) # Pure vision example  inputs = processor(images, return_tensors="pt")inputs["pixel_values"] = inputs["pixel_values"].to(torch.bfloat16)`
Sources: [tests/models/vilt/test_vilt.py 26-28](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L26-L28)

Multi-modal tests follow standard pytest patterns:

## Compilation Challenges for Multi-Modal Models

Multi-modal models face unique challenges during TTNN lowering:

**Specific Issues:**

1.   **ViLT Compilation Failure**[tests/models/vilt/test_vilt.py 36](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L36-L36):

    *   Reason: "cannot sample n_sample <= 0"
    *   Related to sampling operations in attention or pooling layers
    *   Blocked by guard system from compilation

2.   **Attention Mechanism Complexity**:

    *   Cross-attention between modalities requires complex tensor operations
    *   Multi-head attention with multiple input sources
    *   May require fallback for certain attention patterns

3.   **Dynamic Shape Handling**:

    *   Text inputs can vary in sequence length
    *   Vision inputs may have different resolutions
    *   Requires careful shape management through `AddDataMovePass`[torch_ttnn/passes/lowering/add_data_move_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py)

Sources: [tests/models/vilt/test_vilt.py 36](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/models/vilt/test_vilt.py#L36-L36) Architecture Overview from Diagram 2

## Integration with Compilation Pipeline

Multi-modal models flow through the standard compilation pipeline with modality-specific considerations:

**Pipeline Stages:**

1.   **Input Analysis**[torch_ttnn/passes/analysis/input_analysis_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py):

    *   Identifies multiple input tensors
    *   Tags each input appropriately
    *   Enables proper data distribution for multi-device scenarios

2.   **Operation Lowering**[torch_ttnn/passes/lowering/to_tt_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py):

    *   Attempts to convert attention, embedding, and normalization operations
    *   Uses `ReplaceMoreTt` transformer for systematic conversion
    *   Falls back to PyTorch for unsupported patterns

3.   **Guard System**[torch_ttnn/passes/lowering/to_tt_guard.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py):

    *   Validates each operation before lowering
    *   Blocks known problematic patterns (e.g., certain sampling operations)
    *   Maintains blocklists specific to multi-modal operations

4.   **Data Movement**[torch_ttnn/passes/lowering/add_data_move_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/add_data_move_pass.py):

    *   Aligns multiple input tensors for device execution
    *   Inserts `ttnn.from_torch` for each input tensor
    *   Ensures correct memory layouts for multi-modal fusion

Sources: Architecture Overview from Diagram 2, [torch_ttnn/passes/analysis/input_analysis_pass.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/input_analysis_pass.py)
