## Purpose and Scope

This page documents the mechanisms by which the five Baby RISC-V cores within a Tensix tile communicate and synchronize with each other. These mechanisms include:

*   **Mailboxes**: Bidirectional FIFO queues for passing 32-bit values between cores B, T0, T1, and T2
*   **PCBufs**: Unidirectional FIFOs from RISCV B to T0/T1/T2, specialized for pushing Tensix instructions
*   **TTSync**: Lightweight synchronization primitives between T0, T1, and T2
*   **Tensix Semaphores**: Coprocessor semaphores accessible from RISC-V cores

For information about the five RISC-V cores themselves (their pipelines, instruction sets, and local memories), see [Baby RISC-V Cores](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores). For information about how these cores control the Tensix coprocessor, see [Tensix Coprocessor](https://deepwiki.com/tenstorrent/tt-isa-documentation/4-tensix-coprocessor-architecture).

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 1-119](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L1-L119)

## Inter-Core Communication Overview

The five Baby RISC-V cores in a Tensix tile have distinct roles and different communication needs. Four mechanisms enable coordination:

**Inter-Core Communication Mechanisms and Memory Map**

**Key Characteristics:**

*   **RISCV NC is isolated**: No access to any inter-core communication mechanisms
*   **Mailboxes** (`0xFFEC_0000` - `0xFFEC_3FFF`): 16 bidirectional FIFOs (4×4 grid), 32-bit values
*   **PCBufs** (`PC_BUF_BASE`, `PC1_BUF_BASE`, `PC2_BUF_BASE`): 3 unidirectional FIFOs for Tensix instruction pushing
*   **TTSync** (`0xFFE8_0004` - `0xFFE8_001F`): Synchronization registers for T0, T1, T2
*   **Tensix Semaphores** (`0xFFE8_0020` - `0xFFE8_FFFF`): Shared with Tensix coprocessor

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 80-105](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L80-L105)[WormholeB0/TensixTile/BabyRISCV/Mailboxes.md 1-3](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/Mailboxes.md?plain=1#L1-L3)

## Mailboxes

### Architecture

There are **16 mailboxes** within each Tensix tile, forming a 4×4 grid: one mailbox from each of (B, T0, T1, T2) to each of (B, T0, T1, T2). RISCV NC has no access to mailboxes.

**FIFO Characteristics:**

*   Each mailbox holds up to **4 values** of 32 bits each
*   **Aggregate constraint**: A RISC-V core can have at most 4 values across all four of its destination mailboxes 
    *   Example: B can push 4 values to B→T0, or 2 to B→T0 and 2 to B→T1, etc.
    *   Exceeding this limit causes writes to buffer in the memory subsystem and potentially stall the issuing core

*   Reads from an empty mailbox will block until a value is written
*   Mailboxes are **not accessible** from NoC or Tensix coprocessor

Sources: [WormholeB0/TensixTile/BabyRISCV/Mailboxes.md 1-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/Mailboxes.md?plain=1#L1-L46)

### Memory Map and Addressing

The mailbox accessed by a particular address depends on **three factors**:

1.   The address range (which of the 4 `TENSIX_MAILBOX{i}_BASE` regions)
2.   The issuing RISC-V core
3.   Whether the access is a **read** or **write**

**Mailbox Address Mapping Table**

| Constant | Address Range | RISCV B | RISCV T0 | RISCV T1 | RISCV T2 |
| --- | --- | --- | --- | --- | --- |
| `TENSIX_MAILBOX0_BASE` | `0xFFEC_0000` - `0xFFEC_0FFF` | B↔B | Write: T0→B Read: B→T0 | Write: T1→B Read: B→T1 | Write: T2→B Read: B→T2 |
| `TENSIX_MAILBOX1_BASE` | `0xFFEC_1000` - `0xFFEC_1FFF` | Write: B→T0 Read: T0→B | T0↔T0 | Write: T1→T0 Read: T0→T1 | Write: T2→T0 Read: T0→T2 |
| `TENSIX_MAILBOX2_BASE` | `0xFFEC_2000` - `0xFFEC_2FFF` | Write: B→T1 Read: T1→B | Write: T0→T1 Read: T1→T0 | T1↔T1 | Write: T2→T1 Read: T1→T2 |
| `TENSIX_MAILBOX3_BASE` | `0xFFEC_3000` - `0xFFEC_3FFF` | Write: B→T2 Read: T2→B | Write: T0→T2 Read: T2→T0 | Write: T1→T2 Read: T2→T1 | T2↔T2 |

Sources: [WormholeB0/TensixTile/BabyRISCV/Mailboxes.md 12-20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/Mailboxes.md?plain=1#L12-L20)[WormholeB0/TensixTile/BabyRISCV/README.md 100-103](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L100-L103)

### Access Latency

Mailbox accesses have variable latency depending on FIFO state:

| Operation | Latency | Notes |
| --- | --- | --- |
| Read from non-empty mailbox | ≥ 3 cycles | Minimum 3 cycles for FIFO pop |
| Read from empty mailbox | Indefinite | Blocks in Load/Store Unit until write occurs |
| Write to non-full mailbox | Variable | Depends on shared buffer availability |
| Write when aggregate limit reached | Indefinite | Stalls issuing core until space available |

**Load Latency Details:**

*   Base latency: ≥ 3 cycles (more if waiting on empty FIFO or TTSync event)
*   Maximum loads in flight: 4 (aggregate across Mailboxes, PCBufs, TTSync, Tensix semaphores, Tensix GPRs, and Tensix backend configuration)
*   Hidden latency requires 2+ independent instructions after load

### Memory Ordering Considerations

Mailboxes do **not** provide automatic memory ordering guarantees. The read and write sides of each mailbox count as **separate memory regions** for ordering purposes, even though their addresses are identical.

**Common Requirements:**

*   **Write ordering**: Programmer usually wants mailbox write to be processed _after_ all earlier writes
*   **Read ordering**: Programmer usually wants mailbox read-response obtained _before_ any later reads are emitted

Neither of these happens by default. Software must use explicit memory ordering techniques (see [Memory Ordering](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores)) to ensure correct synchronization.

## PCBufs (Program Counter Buffers)

### Purpose and Architecture

PCBufs are specialized **unidirectional FIFO queues** from RISCV B to each of T0, T1, and T2. They are used exclusively for **pushing Tensix instructions** from the master controller (RISCV B) to the three compute control cores (T0, T1, T2).

**Key Differences from Mailboxes:**

*   **Unidirectional only**: RISCV B can write, T0/T1/T2 can read (no reverse direction)
*   **Purpose-built for instructions**: Used to push Tensix instructions to the coprocessor
*   **MOP expansion context**: Instructions pushed via PCBufs bypass the MOP expander (see below)

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 95-99](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L95-L99)

### Memory Map

Each PCBuf has two address ranges: one for the **write side** (RISCV B) and one for the **read side** (T0, T1, or T2).

| PCBuf | Write Side (RISCV B) | Read Side (Target Core) | Target Core |
| --- | --- | --- | --- |
| PCBuf B→T0 | `PC_BUF_BASE` `0xFFE8_0000` - `0xFFE8_0003` | `PC_BUF_BASE` `0xFFE8_0000` - `0xFFE8_0003` | RISCV T0 |
| PCBuf B→T1 | `PC1_BUF_BASE` `0xFFE9_0000` - `0xFFE9_FFFF` | `PC_BUF_BASE` `0xFFE8_0000` - `0xFFE8_0003` | RISCV T1 |
| PCBuf B→T2 | `PC2_BUF_BASE` `0xFFEA_0000` - `0xFFEA_FFFF` | `PC_BUF_BASE` `0xFFE8_0000` - `0xFFE8_0003` | RISCV T2 |

**Note:** From the perspective of T0, T1, T2, all three PCBufs share the same read address (`PC_BUF_BASE`). The hardware routes the read to the appropriate PCBuf based on which core is issuing the read.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 95-99](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L95-L99)

### Instruction Push Mechanism

RISCV cores push Tensix instructions through memory-mapped interfaces. The instruction path differs based on which core is pushing and which address is used:

**Tensix Instruction Push Paths**

**Address-Based Routing:**

| Constant | Address | Accessible From | Destination | Goes Through MOP? |
| --- | --- | --- | --- | --- |
| `INSTRN_BUF_BASE` | `0xFFE4_0000` | B, T0, T1, T2 | T0 thread pipeline | Yes (T0 MOP expander) |
| `INSTRN1_BUF_BASE` | `0xFFE5_0000` | B only | T1 thread pipeline | Yes (T1 MOP expander) |
| `INSTRN2_BUF_BASE` | `0xFFE6_0000` | B only | T2 thread pipeline | Yes (T2 MOP expander) |
| `PC_BUF_BASE` | `0xFFE8_0000` | B (write), T0 (read) | T0 thread pipeline | No (post-MOP) |
| `PC1_BUF_BASE` | `0xFFE9_0000` | B (write), T1 (read) | T1 thread pipeline | No (post-MOP) |
| `PC2_BUF_BASE` | `0xFFEA_0000` | B (write), T2 (read) | T2 thread pipeline | No (post-MOP) |

**Typical Usage Pattern:**

1.   Each T-core pushes its own instructions via `INSTRN_BUF_BASE` (enters before MOP expansion)
2.   RISCV B can inject pre-expanded instructions via PCBufs (bypasses MOP expansion)
3.   PCBufs enable RISCV B to override or coordinate T-core execution with precise control

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 92-94](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L92-L94)

## TTSync

### Purpose and Architecture

TTSync provides lightweight **synchronization primitives** specifically for T0, T1, and T2 to coordinate their execution. These are distinct from the Tensix coprocessor's internal TTSync mechanism used for thread synchronization.

**Characteristics:**

*   Located at address range `0xFFE8_0004` - `0xFFE8_001F` (28 bytes)
*   Accessible from RISCV B, T0, T1, T2
*   **Not accessible** from RISCV NC or via NoC

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 96](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L96-L96)

### Memory Map

| Address Range | Content |
| --- | --- |
| `0xFFE8_0000` - `0xFFE8_0003` | PCBuf (see above) |
| `0xFFE8_0004` - `0xFFE8_001F` | **TTSync registers** (T0, T1, T2) |
| `0xFFE8_0020` - `0xFFE8_FFFF` | Tensix Semaphores (see below) |

The TTSync registers provide synchronization events that T0, T1, and T2 can wait on or signal. The exact register layout within this 28-byte range is hardware-specific.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 96](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L96-L96)

## Tensix Semaphores

### Purpose and Access

Tensix semaphores are part of the **Tensix coprocessor's Sync Unit** but are memory-mapped and accessible from RISC-V cores. They provide synchronization between RISC-V cores and the Tensix coprocessor backend.

**Access Methods:**

*   **From RISC-V cores**: Memory-mapped loads/stores to `0xFFE8_0020` - `0xFFE8_FFFF`
*   **From Tensix backend**: Via Sync Unit instructions

**Typical Usage:**

*   RISC-V cores wait on semaphores signaled by Tensix backend operations
*   RISC-V cores set semaphores to gate Tensix backend execution
*   Coordination between data movement (RISC-V) and computation (Tensix)

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 97](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L97-L97)

### Memory Map

| Address Range | RISCV B | RISCV T0 | RISCV T1 | RISCV T2 | NoC |
| --- | --- | --- | --- | --- | --- |
| `0xFFE8_0020` - `0xFFE8_FFFF` | Tensix Semaphores | Tensix Semaphores | Tensix Semaphores | Tensix Semaphores | Unmapped |

The semaphore array provides multiple semaphore slots. For details on semaphore operations and semantics, see [Tensix Sync Unit](https://deepwiki.com/tenstorrent/tt-isa-documentation/4-tensix-coprocessor-architecture).

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 97](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L97-L97)

## Summary: Communication Mechanism Comparison

| Mechanism | Participants | Direction | Size | Purpose |
| --- | --- | --- | --- | --- |
| **Mailboxes** | B, T0, T1, T2 | Bidirectional (16 FIFOs: 4×4 grid) | 4 × 32-bit values per FIFO 4 values aggregate per sender | General-purpose value passing |
| **PCBufs** | B → T0/T1/T2 | Unidirectional (3 FIFOs) | FIFO depth (unspecified) | Pushing Tensix instructions (post-MOP) |
| **TTSync** | T0, T1, T2 | Synchronization registers | 28 bytes (register layout) | Lightweight sync between T0, T1, T2 |
| **Tensix Semaphores** | B, T0, T1, T2 ↔ Tensix | Shared memory-mapped | Array of semaphores | RISC-V ↔ Tensix coprocessor sync |

**Key Exclusions:**

*   **RISCV NC** has access to **none** of these mechanisms
*   **NoC** cannot access mailboxes, PCBufs, TTSync, or semaphores
*   **Tensix coprocessor** can only access semaphores (via Sync Unit), not mailboxes/PCBufs/TTSync
