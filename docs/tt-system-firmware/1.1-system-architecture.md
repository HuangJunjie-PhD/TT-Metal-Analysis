## Purpose and Scope

This document describes the multi-layered firmware architecture of tt-system-firmware, which provides system management and chip-level control for Tenstorrent AI accelerator hardware. The architecture consists of three cooperating firmware layers (SMC, DMC, and CMFW) that manage hardware initialization, telemetry collection, power management, and runtime control.

For information about specific firmware applications, see [Firmware Applications](https://deepwiki.com/tenstorrent/tt-system-firmware/2-firmware-applications). For hardware subsystem details, see [Hardware Subsystems](https://deepwiki.com/tenstorrent/tt-system-firmware/3-hardware-subsystems). For build system information, see [Build System](https://deepwiki.com/tenstorrent/tt-system-firmware/6-build-system).

## Three-Layer Firmware Architecture

The tt-system-firmware system implements a hierarchical three-layer architecture where each layer runs on different hardware and has distinct responsibilities:

**Sources:**[include/tenstorrent/bh_chip.h 1-184](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_chip.h#L1-L184)[include/tenstorrent/bh_arc.h 1-95](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_arc.h#L1-L95)[app/dmc/src/main.c 1-714](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L1-L714)

### Layer Responsibilities

| Layer | Processor | RTOS | Primary Responsibilities |
| --- | --- | --- | --- |
| **SMC** | ARC (on board) | Zephyr | Host message handling, telemetry aggregation, DVFS control, fan management |
| **DMC** | RISC-V (on board) | Zephyr | Chip reset management, JTAG operations, board peripherals, CM2DM messaging |
| **CMFW** | ARC (in ASIC) | None (bare-metal) | Hardware initialization, NOC management, telemetry collection, subsystem control |

**Sources:**[app/dmc/src/main.c 1-714](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L1-L714)[lib/tenstorrent/bh_arc/cm2dm_msg.c 1-442](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/cm2dm_msg.c#L1-L442)

## Communication Architecture

The three firmware layers communicate using distinct protocols optimized for their interaction patterns:

### Message Queue System (Host ↔ SMC)

The SMC exposes a message queue interface to the host via PCIe BAR space. The system supports 256 distinct message types defined in `enum tt_smc_msg`.

**Key structures:**

*   `struct message_queue_header`: Queue pointers and metadata
*   `union request`: Host request with command-specific payloads
*   `struct response`: SMC response (8×32-bit words)
*   `msgqueue_request_handler_t`: Handler function pointer type

**Registration mechanism:**

`REGISTER_MESSAGE(TT_SMC_MSG_FORCE_FAN_SPEED, force_fan_speed);`
**Sources:**[include/tenstorrent/msgqueue.h 14-481](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L14-L481)[lib/tenstorrent/bh_arc/msgqueue.c 1-346](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L1-L346)[include/tenstorrent/smc_msg.h 1-149](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/smc_msg.h#L1-L149)

### CM2DM Protocol (SMC ↔ DMC)

The CM2DM protocol enables CMFW to send requests to the DMC over SMBus. The protocol uses sequence numbers for message acknowledgment.

**Message types (Cm2DmMsgId enum):**

*   `kCm2DmMsgIdResetReq`: Trigger ASIC or system reset
*   `kCm2DmMsgIdPing`: Connectivity test
*   `kCm2DmMsgIdFanSpeedUpdate`: Fan control update
*   `kCm2DmMsgIdAutoResetTimeoutUpdate`: Watchdog configuration
*   `kCm2DmMsgTelemHeartbeatUpdate`: Telemetry heartbeat

**Message structure:**

`typedef struct cm2dmMessage {    uint8_t msg_id;    uint8_t seq_num;    uint32_t data;} __packed cm2dmMessage;`
**Sources:**[include/tenstorrent/bh_arc.h 15-62](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_arc.h#L15-L62)[lib/tenstorrent/bh_arc/cm2dm_msg.c 54-173](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/cm2dm_msg.c#L54-L173)

### SMBus Target Interface (DMC ↔ CMFW)

The CMFW implements an SMBus target (slave) interface for DMC communication. Register addresses are defined in `enum CMFWSMBusReg`.

**Key SMBus registers:**

*   `CMFW_SMBUS_REQ` (0x10): Read CM2DM message request
*   `CMFW_SMBUS_ACK` (0x11): Write acknowledgment
*   `CMFW_SMBUS_DM_STATIC_INFO` (0x20): Static info including versions
*   `CMFW_SMBUS_POWER_INSTANT` (0x25): Current board power
*   `CMFW_SMBUS_FAN_SPEED` (0x22): Fan speed broadcast

**Sources:**[include/tenstorrent/tt_smbus_regs.h 16-66](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/tt_smbus_regs.h#L16-L66)[lib/tenstorrent/bh_arc/smbus_target.c 1-271](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/smbus_target.c#L1-L271)

### NOC (Network-on-Chip) Translation

The CMFW configures NOC translation to enable fault tolerance by routing around harvested (defective) tiles. The translation layer remaps logical coordinates to physical coordinates.

**Key components:**

*   `NOC2AXITlbSetup`: Configure translation lookaside buffer entries
*   `GetEnabledTensix`: Obtain coordinates of working Tensix core
*   `GetGddrNocCoords`: Map GDDR instance to NOC coordinates
*   `GetEthNocCoords`: Map Ethernet instance to NOC coordinates

**Translation TLB indices:**

*   `MRISC_SETUP_TLB` (13): MRISC firmware access
*   `ETH_SETUP_TLB` (0): Ethernet firmware access
*   `PCIE_DBI_REG_TLB` (14): PCIe DBI register access

**Sources:**[lib/tenstorrent/bh_arc/gddr.c 72-115](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/gddr.c#L72-L115)[lib/tenstorrent/bh_arc/eth.c 78-85](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L78-L85)[lib/tenstorrent/bh_arc/pcie.c 23-41](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L23-L41)

### Chip Reset Management

The DMC manages chip resets through multiple mechanisms:

**Reset types (Cm2DmResetLevel):**

*   `kCm2DmResetLevelAsic` (0): ASIC-only reset via JTAG
*   `kCm2DmResetLevelDmc` (3): Full system reset including DMC

**Reset sequence:**

1.   `bharc_disable_i2cbus`: Disable I2C bus
2.   `jtag_bootrom_reset_asic`: Assert/deassert ASIC reset via GPIO
3.   `jtag_bootrom_soft_reset_arc`: Load bootrom and release ARC from reset
4.   `bharc_enable_i2cbus`: Re-enable I2C bus

**Sources:**[lib/tenstorrent/bh_chip/bh_chip.c 172-190](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L172-L190)[lib/tenstorrent/jtag_bootrom/jtag_bootrom.c 86-134](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/jtag_bootrom/jtag_bootrom.c#L86-L134)[app/dmc/src/main.c 101-120](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L101-L120)

## State Management

### ASIC Power States

The CMFW implements a simple state machine for power management:

| State | Value | Description | Entry Function |
| --- | --- | --- | --- |
| `A0State` | 0 | Normal operation | `enter_state0` |
| `A3State` | 3 | Safe state: no I2C, safe voltage/frequency | `enter_state3` |

**State transitions:**

*   Host sends `TT_SMC_MSG_ASIC_STATE0` or `TT_SMC_MSG_ASIC_STATE3`
*   `asic_state_handler` processes request
*   `lock_down_for_reset` forces A3 state during reset

**Sources:**[lib/tenstorrent/bh_arc/asic_state.c 1-74](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/asic_state.c#L1-L74)[lib/tenstorrent/bh_arc/asic_state.h 9-12](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/asic_state.h#L9-L12)

### Chip Data State

Each chip tracks runtime state in `struct bh_chip_data`:

**Key state flags:**

*   `workaround_applied`: Bootrom loaded successfully
*   `arc_needs_init_msg`: Requires one-time initialization data
*   `trigger_reset`: Atomic flag to request reset
*   `therm_trip_triggered`: Thermal protection triggered
*   `pgood_fall_triggered` / `pgood_rise_triggered`: Power good events
*   `arc_wdog_triggered`: Watchdog timeout occurred

**Sources:**[include/tenstorrent/bh_chip.h 41-90](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/bh_chip.h#L41-L90)
