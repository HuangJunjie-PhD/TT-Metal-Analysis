This page describes the Device Management Controller (DMC) firmware's event-driven architecture and main control loop. The DMC runs on a RISC-V processor and orchestrates chip management operations, responding to hardware events and periodic timers while coordinating with the System Management Controller (SMC) and Chip Management Firmware (CMFW).

For information about the DMC-to-CMFW communication protocol, see [CM2DM Communication Protocol](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.2-cm2dm-communication-protocol). For chip reset operations, see [JTAG and Chip Reset Management](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.3-jtag-and-chip-reset-management).

## Overview

The DMC firmware operates as an event-driven system that responds to:

*   **Hardware interrupts** from GPIO pins (thermal trip, power good changes)
*   **Periodic timers** for telemetry updates (20ms and 1ms intervals)
*   **Watchdog timers** for ARC hang detection
*   **Inter-chip messages** from CMFW via CM2DM protocol

The main loop blocks on events and dispatches handlers in priority order, with urgent hardware events processed immediately before periodic tasks.

**Sources:**[app/dmc/src/main.c 581-713](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L581-L713)

### Event Loop Structure

The main loop structure:

`while (true) {    uint32_t events = tt_event_wait(TT_EVENT_ANY, K_FOREVER);        // Always check urgent events (gated by internal flags)    handle_therm_trip();    handle_watchdog_reset();    handle_perst();    handle_pgood_change();        // One-time initialization data transfer    send_init_data();        // Periodic tasks (gated by event flags)    if (events & (TT_EVENT_BOARD_POWER_TO_SMC | TT_EVENT_WAKE)) {        board_power_update();    }    if (events & (TT_EVENT_FAN_RPM_TO_SMC | TT_EVENT_WAKE)) {        fan_rpm_feedback();    }    if (events & (TT_EVENT_CM2DM_POLL | TT_EVENT_WAKE)) {        handle_cm2dm_messages();    }    if (events & (TT_EVENT_LOGS_TO_SMC | TT_EVENT_WAKE)) {        send_logs_to_smc();    }}`
**Sources:**[app/dmc/src/main.c 680-710](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L680-L710)

### Per-Chip Auto-Reset Timer

Each `bh_chip` instance has an `auto_reset_timer` that monitors for ARC firmware hangs:

*   Timer duration configured by CMFW via `kCm2DmMsgIdAutoResetTimeoutUpdate` message
*   Restarted when telemetry heartbeat updates (indicates ARC is alive)
*   On expiration, triggers `TT_EVENT_WATCHDOG_EXPIRED` and marks chip for watchdog reset

## Urgent Event Handlers

### Thermal Trip Handling

When a thermal trip interrupt occurs:

Key behaviors:

*   **Immediate actions**: Sets board fault LED, forces fan to 100%
*   **Bus cancellation**: Prevents SMBus transfers to potentially hung chip
*   **Deferred reset**: Waits if another reset is already in progress
*   **Counter tracking**: Increments `therm_trip_count` for telemetry

**Sources:**[app/dmc/src/main.c 375-417](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L375-L417)[lib/tenstorrent/bh_chip/bh_chip.c 192-199](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L192-L199)

### Watchdog Reset Handling

When the auto-reset timer expires:

Unique behaviors:

*   **PC capture**: Reads and stores ARC program counter via JTAG before reset (useful for debugging hangs)
*   **Timeout disablement**: Clears `auto_reset_timeout` to prevent repeated resets
*   **Address preserved**: `RESET_UNIT_ARC_PC_CORE_0` at 0x80030C00

**Sources:**[app/dmc/src/main.c 419-450](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L419-L450)[lib/tenstorrent/bh_chip/bh_chip.c 128-136](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L128-L136)

### PCIe Reset (PERST) Handling

PERST triggers full chip reinitialization:

`if (atomic_set(&chip->data.trigger_reset, false)) {    chip->data.performing_reset = true;    chip->data.last_cm2dm_seq_num_valid = false;        bh_chip_cancel_bus_transfer_clear(chip);    bharc_disable_i2cbus(&chip->config.arc);        jtag_bootrom_reset_asic(chip);    jtag_bootrom_soft_reset_arc(chip);    jtag_bootrom_teardown(chip);        bharc_enable_i2cbus(&chip->config.arc);        if (atomic_get(&chip->data.trigger_reset)) {        bh_chip_cancel_bus_transfer_set(chip);    }        chip->data.therm_trip_count = 0;    chip->data.arc_hang_pc = 0;    chip->data.performing_reset = false;}`
Reset sequence includes:

*   **State invalidation**: Clears CM2DM sequence number tracking
*   **JTAG operations**: Full bootrom reload and ARC soft reset
*   **Counter reset**: Clears thermal trip and hang counters

**Sources:**[app/dmc/src/main.c 452-482](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L452-L482)

### PGOOD Change Handling

Power good state changes are delegated to per-chip handler:

`void handle_pgood_event(struct bh_chip *chip, struct gpio_dt_spec board_fault_led){    if (chip->data.pgood_fall_triggered && !chip->data.pgood_severe_fault) {        // Assert board fault LED        gpio_pin_set_dt(&board_fault_led, 1);        // Assert ASIC reset        bh_chip_assert_asic_reset(chip);        // Check for repeated failures within 1 second        if (chip->data.pgood_last_trip_ms != 0 &&            current_uptime_ms - chip->data.pgood_last_trip_ms < 1000) {            chip->data.pgood_severe_fault = true;        }        chip->data.pgood_last_trip_ms = current_uptime_ms;    }        if (chip->data.pgood_rise_triggered && !chip->data.pgood_severe_fault) {        // Follow out-of-reset procedure        bh_chip_reset_chip(chip, true);        // Clear board fault LED (unless assembly test mode)        if (!IS_ENABLED(CONFIG_TT_ASSEMBLY_TEST)) {            gpio_pin_set_dt(&board_fault_led, 0);        }    }}`
**Sources:**[lib/tenstorrent/bh_chip/bh_chip.c 264-295](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L264-L295)

### Fan Speed Control

Fan speed is determined by taking the maximum requested speed across all chips:

`void update_fan_speed(bool notify_smcs){    uint8_t fan_speed = 0;    uint8_t forced_fan_speed = 0;        ARRAY_FOR_EACH_PTR(BH_CHIPS, chip) {        fan_speed = MAX(fan_speed, chip->data.fan_speed);        forced_fan_speed = MAX(forced_fan_speed,            chip->data.fan_speed_forced ? chip->data.fan_speed : 0);    }        if (forced_fan_speed != 0) {        fan_speed = forced_fan_speed;    }        uint32_t fan_speed_pwm = DIV_ROUND_UP(fan_speed * UINT8_MAX, 100);    pwm_set_cycles(max6639_pwm_dev, 0, UINT8_MAX, fan_speed_pwm, 0);        if (notify_smcs) {        ARRAY_FOR_EACH_PTR(BH_CHIPS, chip) {            bharc_smbus_word_data_write(&chip->config.arc,                 CMFW_SMBUS_FAN_SPEED, fan_speed);        }    }}`
Priority order:

1.   **Forced speed** (from any chip) takes precedence
2.   **Maximum automatic** speed across all chips
3.   Broadcasts final speed back to all chips for telemetry consistency

**Sources:**[app/dmc/src/main.c 70-99](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L70-L99)

### Log Forwarding to SMC

DMC logs are buffered in a ring buffer and forwarded to the primary chip's SMC via SMBus:

`static void send_logs_to_smc(void){    uint8_t *log_data;    int ret;        // Pull up to 32 bytes from ringbuf log backend    ret = log_backend_ringbuf_get_claim(&log_data, 32);    if (ret > 0) {        // Write log data to the first BH chip        if (bh_chip_write_logs(&BH_CHIPS[BH_CHIP_PRIMARY_INDEX],                                log_data, ret) == 0) {            log_backend_ringbuf_finish_claim(ret);        } else {            log_backend_ringbuf_finish_claim(0); // Retry later        }    }}`
Uses `bharc_smbus_block_write(CMFW_SMBUS_DMC_LOG, ...)` for up to 32 bytes per cycle.

## State Management and Synchronization

### Bus Transfer Cancellation

SMBus operations can be cancelled when a chip is known to be in a bad state:

Functions:

*   `bh_chip_cancel_bus_transfer_set(chip)` - Calls `smbus_cancel()` to prevent future transfers
*   `bh_chip_cancel_bus_transfer_clear(chip)` - Calls `smbus_uncancel()` to resume transfers

**Sources:**[lib/tenstorrent/bh_chip/bh_chip.c 21-29](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L21-L29)

### Reset State Machine

Synchronization mechanism:

*   **`trigger_reset` (atomic)**: Signal that reset is needed
*   **`performing_reset` (bool)**: Indicates reset in progress
*   Bus cancellation state follows: `(trigger_reset && !performing_reset)`

This prevents nested resets and coordinates between thermal trip, watchdog, and PERST handlers.
