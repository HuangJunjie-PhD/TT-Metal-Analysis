This page provides specific instructions for integrating the cosim-arch-checker framework with the Boom core and Ocelot vector unit in the Chipyard environment. It covers the build configuration, runtime setup, and integration points required for lockstep architectural verification within the Chipyard ecosystem.

For general build system information, see [Build System](https://deepwiki.com/tenstorrent/cosim-arch-checker/5-build-system). For DUT parameter configuration details, see [DUT Parameters](https://deepwiki.com/tenstorrent/cosim-arch-checker/4.2-dut-parameters).

## Overview

The cosim-arch-checker framework has been verified and optimized for use with the Boom core and Ocelot vector unit within the Chipyard environment. This integration leverages Chipyard's VCS simulation infrastructure and requires specific configuration parameters to match the target processor configuration.

Sources: [README.md 13](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/README.md?plain=1#L13-L13)[README.md 103-122](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/README.md?plain=1#L103-L122)

## Build Configuration

### Bazel Build Command

The framework uses Bazel to build the DPI interface with Chipyard-specific configuration flags:

`bazel build :dpi --copt=-DCONFIG=MediumBoomVecConfig`
The `CONFIG` parameter automatically sets the DUT parameters in `env/params.h` based on the specified Chipyard configuration.

### Configuration Parameters

The `MediumBoomVecConfig` setting configures the following DUT parameters:

| Parameter | Value | Description |
| --- | --- | --- |
| `k_NumHarts` | 1 | Number of hardware threads |
| `k_XLen` | 64 | Register width in bits |
| `k_VLen` | 256 | Vector register width in bits |

## Runtime Execution

### VCS Simulation Command

The framework integrates with Chipyard's VCS simulation infrastructure using the following command pattern:

`make -C sims/vcs/ run-binary-debug-hex CONFIG=MediumBoomVecConfig BINARY=/path/to/elf SIM_FLAGS="+cosim"`
### Runtime Flags

The following optional runtime flags control debugging and tracing:

| Flag | Default | Description |
| --- | --- | --- |
| `+cosim` | Required | Enables cosimulation mode |
| `+harness_tracer` | OFF | SystemVerilog harness tracing |
| `+dutmon_tracer` | OFF | DUT monitor interface tracing |
| `+bridge_tracer` | ON | Bridge orchestrator tracing |

## Whisper Configuration

### Ocelot Configuration File

The framework uses a specific Whisper configuration file for the Boom+Ocelot setup:

**Location**: `bridge/whisper/config/ocelot.json`

This configuration file contains:

*   Processor architectural features
*   CSR initialization values
*   Vector extension parameters
*   Memory map configuration

### Whisper Server Mode

The Whisper ISS runs in server mode for the Chipyard integration:

`/path/to/whisper /path/to/elf /path/to/bootrom \  --harts 1 \  --raw \  --configfile /path/to/ocelot.json \  --logfile iss_cosim.log \  --traceload \  --server whisper_connect &`

## Directory Structure

The Chipyard integration uses the following key files and directories:

```
cosim-arch-checker/
├── env/params.h                    # DUT configuration parameters
├── mon/mon_instr/
│   └── mon_instr_dpi.cc           # DPI interface implementation
├── bridge/whisper/
│   ├── config/ocelot.json         # Whisper configuration
│   ├── whisper_client.cpp         # Whisper TCP client
│   └── whisper_client.h
├── BUILD.bazel                     # Bazel build configuration
└── bootrom/                       # Boot ROM binaries
```
