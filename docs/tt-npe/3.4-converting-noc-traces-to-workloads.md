This page documents the process of converting raw Network-on-Chip (NoC) traces captured from actual hardware into the `npeWorkload` data structure used by the simulation engine. This conversion is performed by the `convertNocTracesToNpeWorkload()` function and supporting utilities.

**Scope**: This page covers NoC trace ingestion, event parsing, state management, latency computation, multi-chip fabric path elaboration, and zone extraction. For details on the workload data model itself, see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For information on creating workloads from simplified JSON format (not traces), see [Creating Workloads from JSON](https://deepwiki.com/tenstorrent/tt-npe/3.2-creating-workloads-from-json).

## NoC Trace Event Format

NoC trace events are JSON objects captured by the tt-metal profiler during actual hardware execution. Each event represents a NoC transaction:

| Field | Type | Description |
| --- | --- | --- |
| `timestamp` | int64 | Cycle count when event was issued |
| `type` | string | Event type (READ, WRITE_, WRITE_MULTICAST, FABRIC_UNICAST_WRITE, etc.) |
| `proc` | string | Processor type (BRISC, NCRISC, TRISC_0, etc.) |
| `sx`, `sy` | int64 | Source core coordinates |
| `dx`, `dy` | int64 | Destination core coordinates (unicast) |
| `src_device_id` | int64 | Source device ID |
| `dst_device_id` | int64 | Destination device ID |
| `num_bytes` | int64 | Transfer size in bytes |
| `noc` | string | NoC type (NOC_0 or NOC_1) |
| `mcast_start_x/y`, `mcast_end_x/y` | int64 | Multicast rectangle (if applicable) |
| `fabric_send` | object | Multi-chip routing metadata (if applicable) |
| `zone` | string | Zone name for temporal grouping |
| `zone_phase` | string | ZONE_START or ZONE_END |

**Supported Event Types** (filtered at ingestion):

[tt_npe/cpp/src/npeWorkloadIngest.cpp 314-330](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L314-L330) defines the `SUPPORTED_NOC_EVENTS` set:

*   `READ`, `READ_SET_STATE`, `READ_WITH_STATE`, `READ_WITH_STATE_AND_TRID`
*   `READ_DRAM_SHARDED_SET_STATE`, `READ_DRAM_SHARDED_WITH_STATE`
*   `WRITE_`, `WRITE_MULTICAST`, `WRITE_SET_STATE`, `WRITE_WITH_STATE`
*   `FABRIC_UNICAST_WRITE`, `FABRIC_UNICAST_INLINE_WRITE`, `FABRIC_UNICAST_ATOMIC_INC`, `FABRIC_FUSED_UNICAST_ATOMIC_INC`, `FABRIC_UNICAST_SCATTER_WRITE`

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 314-330](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L314-L330)[tt_npe/py/util/fabric_post_process.py 554-591](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L554-L591)

## Computing Golden Cycles and T0 Timestamp

The `computeGoldenCyclesAndT0()` function extracts reference timing information from the trace:

**Golden Cycles**: The actual hardware execution time range for each device, computed as:

*   **min_kernel_cycles**: Earliest event timestamp on device (relative to t0)
*   **max_kernel_cycles**: Latest event timestamp on device minus 20 cycles (overhead estimation)

[tt_npe/cpp/src/npeWorkloadIngest.cpp 221-269](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L221-L269) computes these values:

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 221-269](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L221-L269)

## SET_STATE and WITH_STATE Event Pattern

NoC traces use a state machine pattern to reduce redundancy. Instead of repeating full event metadata, a `SET_STATE` event stores parameters, and subsequent `WITH_STATE` events reuse them:

**Implementation** at [tt_npe/cpp/src/npeWorkloadIngest.cpp 355-446](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L355-L446):

Separate state is maintained for READ and WRITE operations since they can interleave.

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 355-446](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L355-L446)

## READ Event Source/Destination Swap

NoC READ operations are logically pulls: the source core initiates a read from a destination. However, for simulation purposes, we model all transfers as pushes from sender to receiver. Therefore, READ events have their source and destination swapped:

[tt_npe/cpp/src/npeWorkloadIngest.cpp 448-452](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L448-L452):

After this swap:

*   `sx, sy` → actual data source (where data comes from)
*   `dx, dy` → actual data destination (requester core)

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 448-452](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L448-L452)

## Multi-chip Fabric Path Elaboration

For multi-chip transfers, the `fabric_send` metadata contains routing information that must be elaborated into individual transfer segments:

**fabric_post_process.py Integration**:

[tt_npe/py/util/fabric_post_process.py 553-591](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L553-L591) elaborates the path:

**C++ Path Processing** at [tt_npe/cpp/src/npeWorkloadIngest.cpp 519-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L519-L639):

Each route segment has:

*   `device`: Current device ID
*   `segment_start_x/y`: Ethernet core coordinates
*   `forward_x/y`: Next hop ethernet core (if not terminating)
*   `local_writes`: Array of local worker cores to write to (optional)
*   `parent_id`: Index of parent transfer in group (-1 for first)

All transfers in a fabric path share the same `transfer_group_id` for dependency tracking. The `transfer_group_index` increments sequentially. Dependencies are resolved in `npeEngine::genDependencies()` (see [Dependency Tracking System](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system)).

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 519-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L519-L639)[tt_npe/py/util/fabric_post_process.py 553-591](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L553-L591)

## Transfer Group Construction

Transfer groups link multi-hop fabric transfers with dependencies:

| Field | Type | Description |
| --- | --- | --- |
| `transfer_group_id` | int | Unique ID shared by all transfers in a multi-hop path |
| `transfer_group_index` | int | Sequential index within the group (0, 1, 2, ...) |
| `transfer_group_parent` | int | Index of the parent transfer this depends on |

**Dependency Chain Example**:

This represents a fabric path where:

*   Transfer 0: Worker → local ethernet core
*   Transfer 1: Ethernet → remote device ethernet core
*   Transfers 2A, 2B: Remote ethernet → local workers (parallel)
*   Transfer 3: Continue forwarding to next device

**Registration** at [tt_npe/cpp/include/npeWorkload.hpp 129-132](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L129-L132):

**Usage** at [tt_npe/cpp/src/npeWorkloadIngest.cpp 522-637](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L522-L637):

Dependencies are enforced in `npeEngine::genDependencies()` to ensure proper ordering of multi-hop transfers.

**Sources**: [tt_npe/cpp/src/npeWorkloadIngest.cpp 519-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L519-L639)[tt_npe/cpp/include/npeWorkload.hpp 129-132](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L129-L132)

## Workload Output Structure

The final `npeWorkload` produced by `convertNocTracesToNpeWorkload()` contains:

**Key Properties**:

*   **Single Phase**: Trace ingestion creates one phase containing all transfers
*   **Golden Cycles**: Per-device min/max execution times from hardware
*   **Zones**: Hierarchical temporal markers indexed by (Coord, RiscType)
*   **Transfer Groups**: Multi-hop fabric paths with dependency information
*   **Source Filepath**: Original trace file path for debugging

The workload is validated using `npeWorkload::validate()` against the device model before simulation (see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model)).

**Sources**: [tt_npe/cpp/include/npeWorkload.hpp 97-148](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L97-L148)[tt_npe/cpp/src/npeWorkloadIngest.cpp 655-659](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L655-L659)
