This page documents the command-line scripts used to manage the Singularity container lifecycle in RiESCUE. These scripts are thin wrappers around the `Container` class that provide a user-friendly interface for building, running, pushing, and pulling container images. For information about the container definition and build process, see [Container Definition and Build](/tenstorrent/riescue/3.1.1-container-definition-and-build). For configuration options, see [Container Configuration](/tenstorrent/riescue/3.1.3-container-configuration).

## Overview

RiESCUE provides four primary container management commands located in the `infra/` directory:

| Command | Purpose | Primary Method |
| --- | --- | --- |
| `container-build` | Build a new container image from `Container.def` | `Container.build()` |
| `container-run` | Execute commands inside the container | `Container.run()` |
| `container-push` | Upload container image to remote registry | `Container.push()` |
| `container-pull` | Download container image from remote registry | `Container.pull()` |

All commands are implemented as Python scripts that instantiate the `Container` class from [infra/singularity/container.py](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py) and invoke the corresponding method.

## container-build

The `container-build` script builds a new Singularity container image from the `Container.def` definition file.

### Usage

### Options

| Option | Description |
| --- | --- |
| `--no_push` | Skip the push-to-registry prompt after building |

### Build Process

The build process follows these steps:

1. **Version Check**: Compares `def_version` from [Container.def](https://github.com/tenstorrent/riescue/blob/07cefde3/Container.def) with `container_version` from [infra/container/singularity-id](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/container/singularity-id)
2. **Version Increment**: If versions match, increments the definition version by 1
3. **Build**: Executes `singularity build --fakeroot --force` to create the `.sif` image
4. **SHA Computation**: Calculates SHA256 payload of the filesystem layer
5. **Version File Update**: Writes `{version}-{sha}` format to [singularity-id](https://github.com/tenstorrent/riescue/blob/07cefde3/singularity-id)
6. **Push Prompt**: If registry configured and `--no_push` not set, prompts user to push

## container-run

The `container-run` script launches the container and executes commands inside it. This is the primary interface for using RiESCUE tools within the containerized environment.

### Usage

### Arguments

#### Singularity-Specific Arguments

These arguments control container behavior and are stripped before passing to the command:

| Argument | Type | Description |
| --- | --- | --- |
| `--singularity-binds` | String | Comma-separated bind mounts (e.g., `/host:/container,/host2:/container2`) |
| `--singularity-sif` | Path | Path to custom `.sif` file (skips version checks and pulling) |

**Important**: All `--singularity-*` arguments must use `key=value` format. The parser enforces this at [infra/singularity/container.py311-317](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L311-L317)

#### Command Arguments

Any arguments not starting with `--singularity` are passed to the command executed inside the container.

### Bind Mount Resolution

The `run()` method performs bind mount resolution to avoid errors when paths don't exist on the current host:

1. Splits the bind string on commas
2. For each bind, checks if source path exists using `Path(b).exists()`
3. Only includes binds where the source path is present
4. Joins remaining binds back into comma-separated string

This allows the same `.container_config` to work across different machines with different mounted filesystems.

Sources: [infra/singularity/container.py148-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L148-L156)

### Command Execution Modes

The script handles three command execution modes:

| Condition | Command Built | Use Case |
| --- | --- | --- |
| `len(args) == 0` | `singularity exec --bind  riescue.sif bash` | Interactive session |
| `Path(args[0]).exists()` | `singularity exec ... bash -c ""` | Execute script file |
| Otherwise | `singularity exec ... bash -c ""` | Execute command string |

## container-push

The `container-push` script uploads the local container image to a remote registry.

### Usage

### Prerequisites

* Local `riescue.sif` file must exist
* `.container_config` must specify `registry_uri` and `registry_remote`
* User must have authentication credentials (see [Container Configuration](/tenstorrent/riescue/3.1.3-container-configuration))

### Push Process

1. **Version String Generation**: Creates `{def_version}-{sha_payload}` identifier
2. **Registry Authentication**: Calls `_check_singularity_remotes()` to ensure remote is configured
3. **Upload**: Executes `singularity push riescue.sif {registry_uri}:{version_string}`

The push operation tags the image with both the version number and SHA payload, ensuring exact reproducibility.

## container-pull

The `container-pull` script downloads the latest container image from the remote registry.

### Usage

### Prerequisites

* `.container_config` must specify `registry_uri` and `registry_remote`
* User must have authentication credentials

### Pull Process

1. **Check Configuration**: Validates that `registry_uri` and `registry_remote` are set
2. **Remove Local SIF**: If `riescue.sif` exists locally, deletes it
3. **Remote Authentication**: Ensures registry remote is configured
4. **Download**: Executes `singularity pull riescue.sif {registry_uri}:{container_string}`

The `container_string` comes from [infra/container/singularity-id](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/container/singularity-id) ensuring the exact version specified in the repository is downloaded.

## Integration with .env File

The `Container` class automatically loads environment variables from `.env` in the repository root. This is particularly important for the `RV_TOOLCHAIN` variable:

1. Loads `.env` file via `_load_dotenv()` at [infra/singularity/container.py73](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L73-L73)
2. If `RV_TOOLCHAIN` is set, appends it to `PATH` at [infra/singularity/container.py75-82](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L75-L82)
3. This ensures RISC-V toolchain is available to commands run via `container-run`

## Error Handling

### Common Error Scenarios

| Error | Cause | Solution |
| --- | --- | --- |
| `FileNotFoundError` | `Container.def` not found | Ensure running from repository root |
| `RuntimeError: No registry URI` | No `.container_config` when pulling | Build locally or add configuration |
| `RuntimeError: must be in key=value format` | `--singularity-*` arg not formatted correctly | Use `--singularity-binds=/path` not `--singularity-binds /path` |
| `ValueError: Couldn't find VERSION` | Malformed `Container.def` or `.sif` | Verify VERSION line exists in definition |
