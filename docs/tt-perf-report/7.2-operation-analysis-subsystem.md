# Operation Analysis Subsystem

Relevant source files
*   [src/tt_perf_report/perf_report.py](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/src/tt_perf_report/perf_report.py)

## Purpose and Scope

The Operation Analysis Subsystem is a critical component of the tt-perf-report tool that examines and extracts metrics from different operation types in machine learning workloads. It specializes in analyzing matrix multiplications, convolutions, and halo operations to calculate performance metrics such as DRAM bandwidth utilization, FLOPs (floating-point operations per second), and other operation-specific characteristics. These metrics serve as the foundation for classifying operations and generating optimization advice.

For information about how these metrics are used to classify performance bottlenecks, see [Performance Classification System](https://deepwiki.com/tenstorrent/tt-perf-report/5-performance-classification-system). For details on how the entire data processing pipeline works, see [Data Processing Pipeline](https://deepwiki.com/tenstorrent/tt-perf-report/7.1-data-processing-pipeline).

## Operation Detection and Dispatch

The Operation Analysis Subsystem first identifies the type of operation from the trace data, then dispatches the analysis to specialized functions for each operation type. This is handled primarily by the `analyze_op` function.

Sources: [src/tt_perf_report/perf_report.py 375-512](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/src/tt_perf_report/perf_report.py#L375-L512)

## Matrix Multiplication Analysis

The `analyze_matmul` function calculates key performance metrics for matrix multiplication operations:

1.   **DRAM Bandwidth**: Calculates the amount of data transferred to/from DRAM per second
2.   **DRAM Utilization**: Expresses DRAM bandwidth as a percentage of maximum bandwidth (288 GB/s)
3.   **FLOPS**: Calculates floating-point operations per second
4.   **FLOPS Utilization**: Expresses FLOPS as a percentage of peak theoretical performance
5.   **Operation Size**: Extracts matrix dimensions (M×K×N)
6.   **Memory Configuration**: Determines memory locations for inputs and outputs
7.   **DRAM Sharding Detection**: Identifies if the operation uses a DRAM-sharded configuration

### Key Calculations

*   **DRAM Bandwidth**: `total_data_size_bytes / duration_s / 1e9` (in GB/s)
*   **DRAM Utilization**: `(dram_speed_gb_s / 288) * 100` (as percentage)
*   **FLOPS**: `(M * K * N * W * Z * 2) / duration_s` (operations per second)
*   **FLOPS Utilization**: `(flops / peak_flops_value) * 100` (as percentage)

Where:

*   `M`, `K`, `N` are matrix dimensions
*   `W`, `Z` are additional dimensions for batched operations
*   `peak_flops_value` is calculated based on math fidelity and core count

Sources: [src/tt_perf_report/perf_report.py 208-277](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/src/tt_perf_report/perf_report.py#L208-L277)

## Convolution Analysis

The `analyze_conv` function extracts and calculates metrics for convolution operations:

1.   **FLOPS**: Calculates floating-point operations per second for convolutions
2.   **FLOPS Utilization**: Expresses FLOPS as a percentage of theoretical peak
3.   **Operation Size**: Extracts dimensions relevant to convolutions
4.   **Configuration Details**: Extracts specialized convolution parameters: 
    *   Activation block height and tiles
    *   Double buffering settings
    *   Split reader configuration
    *   Output matrix height per tile

### Convolution Dimensions and FLOPS Calculation

For convolutions, the dimensions are calculated as:

*   `M = NHW` (output size)
*   `K = CH_IN * W[0] * W[1]` (input channels × filter width × filter height)
*   `N = CH_OUT` (output channels)

FLOPS are then calculated as: `(M * K * N * 2) / duration_s`

## Halo Operation Analysis

The `analyze_halo` function extracts configuration details for halo operations, which are used in operations like convolutions to handle edge padding:

1.   **Window Size**: Extracts height and width of the window
2.   **Stride**: Extracts stride values
3.   **Padding**: Extracts padding configuration
4.   **Dilation**: Extracts dilation values
5.   **Memory Layout**: Extracts memory layout information

The extracted information is formatted as a configuration string: `w=[window_hw] s=[stride_hw] p=[pad_hw] d=[dilation_hw] | [memory_layout]`

## Integration with the Cell System

All analysis results are formatted using the `Cell` class, which provides a consistent formatting approach for all metrics:

1.   **Cell Creation**: Each metric is wrapped in a `Cell` object with:

    *   Raw value (numeric or string)
    *   Unit (e.g., "GB/s", "TFLOPs", "%")
    *   Decimal precision
    *   Color information (for console display)

2.   **Cell Formatting**: The `Cell` class provides formatting methods to display the value consistently:

    *   Numeric values are formatted with thousands separators and specified decimal places
    *   Units are appended
    *   Colors are applied if console output supports it

## Performance Metrics and Derived Columns

After operation analysis, the `add_derived_columns` function processes the analyzed operations to:

1.   Calculate the percentage of total execution time for each operation
2.   Classify operations as DRAM-bound, FLOP-bound, both, or slow based on utilization percentages: 
    *   DRAM-bound: DRAM utilization ≥ 65%
    *   FLOP-bound: FLOPS utilization ≥ 65%
    *   BOTH-bound: Both DRAM and FLOPS utilization ≥ 65%
    *   SLOW: Neither utilization reaches 65%
    *   HOST: Operations running on the host (CPU) instead of the device

## Operation Coloring Logic

After analysis and classification, operations are colored based on their characteristics using the `color_row` function:

1.   **Operation Type Coloring**:

    *   Matrix multiplications: magenta
    *   Convolutions: orange
    *   Host operations: red
    *   Other specialized operations: various colors

2.   **Performance Coloring**:

    *   DRAM-bound: green DRAM metrics
    *   FLOP-bound: green FLOPS metrics
    *   SLOW: yellow metrics for the limiting factor
    *   Large operation gaps: red for gaps over 6.5μs

3.   **Math Fidelity Coloring**:

    *   Sufficient: green
    *   Too high: red (suboptimal use of compute resources)
    *   Too low: cyan (potential accuracy issues)

## Relationship to Optimization Advice System

The metrics and classifications from the Operation Analysis Subsystem feed directly into the Optimization Advice System ([Optimization Advice System](https://deepwiki.com/tenstorrent/tt-perf-report/6-optimization-advice-system)), which generates targeted suggestions for improving performance:

1.   For DRAM-bound matrix multiplications:

    *   Suggests DRAM-sharded program configurations
    *   Evaluates if math fidelity is appropriate

2.   For FLOP-bound operations:

    *   Suggests increasing core count
    *   Evaluates if lower-precision math fidelity could be used

3.   For SLOW operations:

    *   Suggests memory placement optimizations (e.g., placing inputs in L1)
    *   Suggests modifying block sizes and program configurations

This tight integration between analysis and advice creates a comprehensive performance optimization workflow.

## Summary

The Operation Analysis Subsystem is the analytical core of the tt-perf-report tool, responsible for extracting and calculating performance metrics for different operation types. By providing detailed insights into DRAM bandwidth, computational throughput, and operation-specific configurations, it enables the tool to identify performance bottlenecks and generate targeted optimization advice.

Key components include:

*   Specialized analyzers for matrix multiplications, convolutions, and halo operations
*   Detailed performance metric calculations
*   Integration with the classification system
*   Formatted output using the Cell system
*   Direct connection to the optimization advice generation

Together, these components create a powerful system for understanding and optimizing the performance of machine learning workloads on Tenstorrent hardware.
