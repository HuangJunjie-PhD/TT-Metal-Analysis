## Purpose and Scope

This document explains the fundamental configuration, initialization, and execution patterns for the unpacker hardware unit in the Tensix core. The unpacker is the first stage of the three-stage processing pipeline (unpack-math-pack) and is responsible for reading data from L1 memory and loading it into source registers (SrcA and SrcB) for consumption by the math unit.

This page covers the core unpacker mechanics including hardware architecture, configuration structures, initialization sequences, and basic execution patterns. For specialized unpacker operations, see:

*   Tilize/untilize operations: [Tilize and Untilize Operations](https://deepwiki.com/tenstorrent/tt-llk/4.2-tilize-and-untilize-operations)
*   Broadcast and reduce patterns: [Broadcast and Reduce Unpacking](https://deepwiki.com/tenstorrent/tt-llk/4.3-broadcast-and-reduce-unpacking)
*   Architecture-specific implementations: [Architecture-Specific Unpacker Implementations](https://deepwiki.com/tenstorrent/tt-llk/4.4-architecture-specific-unpacker-implementations)
*   32-bit direct-to-dest mode: [Unpack-to-Dest for 32-bit Formats](https://deepwiki.com/tenstorrent/tt-llk/4.5-unpack-to-dest-for-32-bit-formats)

## Unpacker Hardware Architecture

The Tensix core contains two independent unpacker units that operate in parallel, each capable of loading data from L1 memory into separate source register banks.

**Unpacker Hardware Organization**

### Key Hardware Components

| Component | Description | Register Prefix |
| --- | --- | --- |
| Unpacker 0 (SEC0) | First unpacker unit, typically loads into SrcA or SrcB | `THCON_SEC0_*` |
| Unpacker 1 (SEC1) | Second unpacker unit, typically loads into SrcB or SrcA | `THCON_SEC1_*` |
| Context 0/1 | Double-buffering contexts for ping-pong operation | `*_cntx0_*`, `*_cntx1_*` |
| Address Counters | X, Y, Z, W counters for multi-dimensional addressing | `UNP0_ADDR_CTRL_*`, `UNP1_ADDR_CTRL_*` |
| Source Registers | 16x16 element banks for SrcA and SrcB | Addressed by channel and counter |

## Configuration Data Structures

The unpacker is configured through several key structures that define tile format, data layout, and conversion parameters.

### Tile Descriptor Structure

The `unpack_tile_descriptor_t` structure describes the input tile format and dimensions:

**Key Fields:**

*   `in_data_format`: Input data format enumeration (Float16, Bfp8, etc.)
*   `x_dim`: Tile width in elements (typically `FACE_R_DIM * FACE_C_DIM = 256`)
*   `y_dim`: Number of rows (typically 1 for standard tiles)
*   `z_dim`: Number of faces (1, 2, or 4 for 16x16, 16x32/32x16, or 32x32 tiles)
*   `uncompressed`: Always 1 for uncompressed tiles

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 22-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L22-L43)

### Unpack Configuration Structure

The `unpack_config_t` structure controls unpacker behavior and format conversion:

**Key Fields:**

*   `out_data_format`: Target format in source registers (may differ from L1 format)
*   `haloize_mode`: Enables 16x16 transpose within each face
*   `throttle_mode`: Typically set to 2 for standard operation
*   `uncompress_cntx0_3`: Enables uncompression for contexts 0-3

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 52-80](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L52-L80)

### ALU Configuration Structure

The `alu_config_t` structure affects how unpacked data is interpreted by the math unit:

| Field | Purpose | Values |
| --- | --- | --- |
| `ALU_FORMAT_SPEC_REG0_SrcA` | Source A data format | 4-bit format code |
| `ALU_FORMAT_SPEC_REG1_SrcB` | Source B data format | 4-bit format code |
| `ALU_FORMAT_SPEC_REG0_SrcAUnsigned` | Treat SrcA as unsigned | 0 or 1 |
| `ALU_FORMAT_SPEC_REG0_SrcBUnsigned` | Treat SrcB as unsigned | 0 or 1 |
| `ALU_ACC_CTRL_Fp32_enabled` | Enable FP32 accumulation mode | 0 or 1 |
| `ALU_ROUNDING_MODE_Fpu_srnd_en` | Enable stochastic rounding | 0 or 1 |

## Initialization Sequence

The standard unpacker initialization follows a well-defined sequence that sets up address counters, tile descriptors, unpacker configuration, and ALU format settings.

**Unpacker Initialization Flow**

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 208-377](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L208-L377)

### Address Counter Initialization

The `unpacker_addr_counter_init()` function resets all address counters to their default state:

```
TTI_SETADCXY(0b011, 0, 0, 0, 0, 0b1011);  // Reset X, Y counters for both unpackers
TTI_SETADCZW(0b011, 0, 0, 0, 0, 0b1111);  // Reset Z, W counters for both unpackers
```

The address counters track position within the tile during unpacking:

*   **X counter**: Column position within a face (0 to FACE_C_DIM-1)
*   **Y counter**: Row position within a face (0 to FACE_R_DIM-1)
*   **Z counter**: Face index (0 to num_faces-1)
*   **W counter**: Tile index (typically not used in basic operations)

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 118-122](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L118-L122)

### Stride Configuration

Strides determine how address counters increment when moving through data:

The stride is set based on the output data format element size:

*   Float32/Int32: 4 bytes per element
*   Float16/Bfp16: 2 bytes per element
*   Int8/Bfp8: 1 byte per element

## Context Management and Double-Buffering

The unpacker uses a double-buffering scheme with two contexts to enable overlapping of configuration and execution.

**Context Switching Mechanism**

### Context Switching Functions

**switch_config_context()**

Toggles the active configuration context:

The context offset register controls which set of base address registers the unpacker hardware reads from.

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 171-183](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L171-L183)

**wait_for_next_context()**

Blocks until a context becomes available:

The `UNPACK_SYNC` semaphore tracks how many contexts are currently in use (0, 1, or 2). When both contexts are busy, the code must wait before configuring the next operation.

Sources: [tt_llk_wormhole_b0/common/inc/cunpack_common.h 164-169](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/cunpack_common.h#L164-L169)

**wait_for_idle()**

Waits for all unpacker activity to complete:

This is used during initialization and reconfiguration to ensure the unpacker is completely idle.
