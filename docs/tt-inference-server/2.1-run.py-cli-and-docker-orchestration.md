The `run.py` script serves as the primary entry point for the Tenstorrent inference server ecosystem. It orchestrates the lifecycle of inference services and workflows (benchmarks, evals, tests) by resolving hardware-specific configurations, managing host environments, and generating complex Docker execution commands.

## Architecture Overview

The orchestration logic is divided into several layers:

1.   **Argument Parsing & Validation**: CLI input is parsed and validated against the `MODEL_SPECS` registry and system capabilities.
2.   **Runtime Configuration**: Creation of a `RuntimeConfig` object that encapsulates the desired state.
3.   **Host Setup Management**: Pre-flight checks and environment preparation via `SetupConfig`.
4.   **Container/Local Execution**: Generation of execution commands for either Docker containers or local Python environments.

## SetupConfig and Pre-flight Checks

Before execution, the `SetupConfig` class in `workflows/setup_host.py` prepares the host environment. This includes resolving paths for model weights and ensuring persistent storage is available.

### Volume Strategies

The server supports three primary methods for providing weights to the container via `SetupConfig`:

1.   **Docker Volumes**: Default behavior. Uses `generate_docker_volume_name` to create a persistent volume named `volume_id_<impl_id>-<model_name>``<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L91-L96" min=91 max=96 file-path="workflows/run_docker_server.py">Hii</FileRef>`.
2.   **Host Bind Mounts**: Triggered by `--host-volume`. Maps a specific host directory to the container's `cache_root``<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L83-L91" min=83 max=91 file-path="workflows/setup_host.py">Hii</FileRef>`.
3.   **Read-only Weights**: Triggered by `--host-hf-cache` or `--host-weights-dir`. Maps existing weights into the container as read-only volumes `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L114-L138" min=114 max=138 file-path="workflows/setup_host.py">Hii</FileRef>`.

### Permission Validation

The system performs critical checks on bind mounts to prevent container runtime failures. `ensure_readwriteable_dir` verifies that the target paths are accessible `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/utils.py#L124-L180" min=124 max=180 file-path="workflows/utils.py">Hii</FileRef>`. For local server execution, `validate_local_setup` ensures the environment is ready and optionally triggers `run_system_software_validation.py` to check firmware and KMD versions `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/validate_setup.py#L157-L185" min=157 max=185 file-path="workflows/validate_setup.py">Hii</FileRef>`.

## Docker Orchestration

The core of the Docker logic resides in `generate_docker_run_command` within `workflows/run_docker_server.py`.

### Device Mapping

Tenstorrent devices are mapped into the container using the `--device` flag. If no specific `--device-id` is provided, the entire `/dev/tenstorrent` character device range is mapped `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L155-L156" min=155 max=156 file-path="workflows/run_docker_server.py">Hii</FileRef>`. Otherwise, specific indices are mapped (e.g., `/dev/tenstorrent/0:/dev/tenstorrent/0`) `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L158-L161" min=158 max=161 file-path="workflows/run_docker_server.py">Hii</FileRef>`.

## Local Server Execution

When `--local-server` is used, the system bypasses Docker and runs the entrypoint directly on the host using a pre-configured `tt-metal` virtual environment.

1.   **Validation**: `validate_local_setup` checks if the `tt-metal` home and Python environments are correctly configured `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/validate_setup.py#L157-L185" min=157 max=185 file-path="workflows/validate_setup.py">Hii</FileRef>`.
2.   **vLLM Check**: If the inference engine is vLLM, `_validate_local_vllm_installation` verifies that the `vllm` package can be imported within the target venv `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/validate_setup.py#L142-L154" min=142 max=154 file-path="workflows/validate_setup.py">Hii</FileRef>`.
3.   **Process Launch**: The server is started as a local process via `run_local_server` (orchestrated from `run.py`) `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L36-L36" min=36  file-path="run.py">Hii</FileRef>`.

## Multi-Host Orchestration

For large-scale deployments (e.g., Dual or Quad Galaxy), `run.py` utilizes the `MultiHostOrchestrator`.

1.   **Validation**: `run_multihost_validation_subprocess` ensures connectivity and system readiness across all participating nodes `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L24-L24" min=24  file-path="workflows/run_docker_server.py">Hii</FileRef>`.
2.   **Configuration**: `setup_multihost_config` prepares the environment for distributed execution, including rankfiles if necessary `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L22-L22" min=22  file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`.
3.   **Deployment Logic**: The system determines the required number of hosts using `get_expected_num_hosts` based on the requested `DeviceTypes``<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L20-L21" min=20 max=21 file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`.
