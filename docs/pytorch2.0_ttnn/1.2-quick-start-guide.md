## Eager Mode Execution

Eager mode allows running models directly on Tenstorrent hardware by switching the device. This mode bypasses compilation and is useful for initial testing.

### Basic Eager Mode Example

`import torchimport torch_ttnn # Create your PyTorch modelmodel = YourModel() # Open a single Tenstorrent devicedevice = ttnn.open_device(device_id=0) # Move model to TT devicemodel.to(torch_ttnn.ttnn_device_as_torch_device(device)) # Run inferenceoutput = model(input_data)`
In eager mode, operations are executed immediately on the device without graph-level optimization. This provides a quick way to verify hardware compatibility but does not leverage the full compilation pipeline.

## Compilation Mode (Recommended)

Compilation mode uses `torch.compile` to optimize the model graph for Tenstorrent hardware. This is the recommended approach for production workloads as it enables graph-level transformations and optimizations.

### Single Device Compilation

`import torchimport torch_ttnn # Create modelmodel = YourModel() # Open single devicedevice = ttnn.open_device(device_id=0) # Configure backendoption = torch_ttnn.TorchTtnnOption(device=device) # Compile modelmodel = torch.compile(model, backend=torch_ttnn.backend, options=option) # Run inferenceoutput = model(input_data)`

### Multi-Device Compilation (Data Parallel)

For multi-device execution with data parallelism, use `ttnn.open_mesh_device()` to create a device mesh:

`import torchimport torch_ttnn # Create modelmodel = YourModel() # Open mesh device with 1x2 grid (2 devices)device = ttnn.open_mesh_device(ttnn.MeshShape(1, 2)) # Configure for data parallel executionoption = torch_ttnn.TorchTtnnOption(    device=device,    data_parallel=True) # Compile modelmodel = torch.compile(model, backend=torch_ttnn.backend, options=option) # Run inference with automatic data distributionoutput = model(input_data)`
The `data_parallel=True` flag enables `MultiDevicePass` which automatically shards input arguments across devices and replicates parameters/buffers. See [Multi-Device Support and Sharding](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/3.4-multi-device-support-and-sharding) for details.

## TorchTtnnOption Configuration

The `TorchTtnnOption` class provides configuration parameters for the compilation backend. The following table describes commonly used options:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `device` | `ttnn.Device` or `ttnn.MeshDevice` | **Required** | Target device for execution |
| `data_parallel` | `bool` | `False` | Enable data parallel execution across multiple devices |
| `use_less_ttnn_op_types` | `bool` | `True` | Prefer generic TTNN ops over specialized variants |
| `load_params_once` | `bool` | `True` | Preprocess model parameters once and cache on device |
| `metrics_path` | `str` | `""` | Path to save compilation metrics |
| `gen_graphviz` | `bool` | `False` | Generate GraphViz visualizations at each pass |
| `export_code` | `str` or `None` | `None` | Export standalone Python script for debugging |

## Common Usage Patterns

### Pattern 1: Basic Inference

`import torchimport torch_ttnn # Initializedevice = ttnn.open_device(device_id=0)option = torch_ttnn.TorchTtnnOption(device=device) # Compilemodel = torch.compile(model, backend=torch_ttnn.backend, options=option) # Single inferenceoutput = model(input_tensor)`
### Pattern 2: Batched Inference with Warm-up

`# First run includes compilation timeoutput = model(first_batch)  # Slow: includes compilation # Subsequent runs use cached compiled graphfor batch in data_loader:    output = model(batch)  # Fast: uses cached graph`
The first execution triggers compilation via `ttnn_backend()` → `aten_backend()` which runs the full pass pipeline. Subsequent executions reuse the cached `GraphModule`.

### Pattern 3: Multi-Device Training Forward Pass

`# Setup multi-devicedevice = ttnn.open_mesh_device(ttnn.MeshShape(1, 2))option = torch_ttnn.TorchTtnnOption(    device=device,    data_parallel=True,    load_params_once=True  # Preprocess parameters once) # Compilemodel = torch.compile(model, backend=torch_ttnn.backend, options=option) # Training loopfor inputs, labels in train_loader:    outputs = model(inputs)  # Inputs auto-sharded across devices    loss = criterion(outputs, labels)    loss.backward()`
The `MultiDevicePass` automatically inserts sharding operations for input arguments and replication operations for parameters, as analyzed by `MultiDeviceShardAnalysisPass`.

## Compilation Pipeline Overview

When `torch.compile()` is invoked with the `torch_ttnn.backend`, the following high-level flow occurs:

**Key Steps:**

1.   **`ttnn_backend()`**[torch_ttnn/backend.py 271-323](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L271-L323) wraps the compilation process
2.   **`insert_clones_for_input_aliasing()`**[torch_ttnn/preprocessing/handle_input_aliasing.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/preprocessing/handle_input_aliasing.py) handles tensor aliasing
3.   **`aot_autograd()`** decomposes to low-level ATen operations
4.   **`aten_backend()`**[torch_ttnn/backend.py 101-259](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/backend.py#L101-L259) orchestrates the pass pipeline: 
    *   **`ConstantFoldingPass`** evaluates constant expressions
    *   **`ToTtPass`** converts ATen operations to TTNN operations
    *   **`AddDataMovePass`** inserts data movement between host and device
    *   **Optimization passes** perform graph cleanup and optimization

## Debugging and Metrics

### Enabling Metrics Collection

`option = torch_ttnn.TorchTtnnOption(    device=device,    metrics_path="my_model_metrics")`
This saves compilation metrics to `metrics/my_model_metrics/` including:

*   `original-schema_list.pickle`: Input variations before compilation
*   `compiled-schema_list.pickle`: Operations after compilation
*   Runtime metrics and accuracy measurements

### Exporting Standalone Code

`option = torch_ttnn.TorchTtnnOption(    device=device,    export_code="debug_script.py")`
This generates a standalone Python script with serialized inputs for debugging. See [Code Export and Standalone Script Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/8.1-code-export-and-standalone-script-generation) for details.

### Visualizing Compilation Passes

`option = torch_ttnn.TorchTtnnOption(    device=device,    gen_graphviz=True,    metrics_path="my_model")`
Generates `.dot.svg` files showing the graph structure after each compilation pass in `metrics/my_model/`.
