## Purpose and Scope

This document provides a detailed technical comparison between Wormhole B0 and Blackhole P100 hardware platforms, focusing on architectural differences that impact data movement operations. This comparison covers core counts, NOC interface specifications, register availability, performance characteristics, and architectural evolution considerations.

For information about how register state is preserved across API calls, see [Register State Preservation](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.1-register-state-preservation). For baseline performance metrics of specific data movement primitives, see [Baseline Performance Metrics](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.1-baseline-performance-metrics).

* * *

## Architectural Specifications Overview

The following table summarizes the key architectural differences between the two platforms:

| Specification | Wormhole B0 | Blackhole P100 | Impact |
| --- | --- | --- | --- |
| Tensix Core Count | 64 | 110 | +72% compute capacity |
| NOC Interface Width | 256 bits/cycle (32 bytes/cycle) | 512 bits/cycle (64 bytes/cycle) | 2x theoretical bandwidth |
| Max Packet Size | 8 KB | 16 KB | Reduced header overhead |
| Address Space | 48-bit addressing | 64-bit addressing | Extended memory addressability |
| NOC Instances | 2 (NOC0, NOC1) | 2 (NOC0, NOC1) | Consistent dual-NOC architecture |
| Virtual Channels | 16 per NOC | 16 per NOC | Consistent VC architecture |
| Topology | 2D Torus | 2D Torus | Consistent routing substrate |

## NOC Interface and Bandwidth Evolution

**Diagram: NOC Interface Width and Core Count Evolution**

The doubling of interface width from Wormhole B0 to Blackhole P100 represents a fundamental bandwidth increase. Combined with larger packet sizes, Blackhole reduces the relative overhead of packet headers, improving efficiency for large data transfers.

**Sources:**[data_movement_doc/general/intro_to_dm.md 18-24](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/intro_to_dm.md?plain=1#L18-L24)[data_movement_doc/general/ideal_performance.md 6](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/ideal_performance.md?plain=1#L6-L6)

## Performance Characteristics Comparison

### Actual Performance Metrics

The following table shows measured performance across all data movement primitives, measured in bytes per cycle:

| Data Movement Primitive | Wormhole B0 (bytes/cycle) | Blackhole P100 (bytes/cycle) | Speedup |
| --- | --- | --- | --- |
| DRAM Read | 22 | 33 | 1.50x |
| DRAM Write | 21 | 34 | 1.62x |
| One To One | 29 | 60 | 2.07x |
| One From One | 28 | 60 | 2.14x |
| One To All (Unicast) | 31 | 62 | 2.00x |
| One To All (Multicast) | 15 | 24 | 1.60x |
| One To All (Multicast + Linked) | 22 | 41 | 1.86x |
| One From All | 30 | 60 | 2.00x |

**Sources:**[data_movement_doc/general/ideal_performance.md 8-17](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/ideal_performance.md?plain=1#L8-L17)

### Performance Analysis

**Diagram: Theoretical vs Actual Performance Scaling**

Core-to-core operations achieve near-theoretical 2x speedup on Blackhole, reflecting the doubled interface width. DRAM operations show lower speedup (1.5-1.6x) due to DRAM controller bandwidth limitations. Multicast operations show intermediate speedup (1.6x) as they are limited by network congestion patterns rather than pure interface bandwidth.

**Sources:**[data_movement_doc/general/ideal_performance.md 8-17](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/ideal_performance.md?plain=1#L8-L17)[data_movement_doc/general/intro_to_dm.md 18-24](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/intro_to_dm.md?plain=1#L18-L24)

## Register Architecture Differences

### Register Availability by Platform

**Diagram: Register Availability Across Platforms**

### 64-bit Addressing Support

The most significant register difference is Blackhole's addition of `NOC_TARG_ADDR_HI` and `NOC_RET_ADDR_HI` registers to support 64-bit addressing:

| Address Component | Wormhole B0 | Blackhole P100 |
| --- | --- | --- |
| Target Address Low | `NOC_TARG_ADDR_LO` (stateful) | `NOC_TARG_ADDR_LO` (stateful) |
| Target Address Mid | `NOC_TARG_ADDR_MID` (stateful) | `NOC_TARG_ADDR_MID` (stateful) |
| Target Address High | Unused | `NOC_TARG_ADDR_HI` (stateful) |
| Return Address Low | `NOC_RET_ADDR_LO` (stateful) | `NOC_RET_ADDR_LO` (stateful) |
| Return Address Mid | `NOC_RET_ADDR_MID` (stateful) | `NOC_RET_ADDR_MID` (stateful) |
| Return Address High | Unused | `NOC_RET_ADDR_HI` (stateful) |
| Address Space | 48-bit | 64-bit |

**Sources:**[data_movement_doc/general/state_preservation.md 8-13](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/state_preservation.md?plain=1#L8-L13)

### Blackhole-Specific Control Registers

Blackhole introduces several new control and status registers:

| Register | Purpose | Statefulness |
| --- | --- | --- |
| `NOC_AT_LEN_BE_1` | Extended atomic length and byte enable | Stateful |
| `NOC_BRCST_EXCLUDE` | Broadcast exclusion mask | Stateful |
| `NOC_L1_ACC_AT_INSTRN` | L1 access atomic instruction | Stateful |
| `NOC_SEC_CTRL` | Security control | Stateful |
| `CMD_BUF_AVAIL` | Command buffer availability status | Non-stateful (control flag) |
| `CMD_BUF_OVFL` | Command buffer overflow status | Non-stateful (control flag) |

**Sources:**[data_movement_doc/general/state_preservation.md 17-32](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/state_preservation.md?plain=1#L17-L32)

### Wormhole-Specific Registers

Wormhole B0 has one platform-specific register that was removed in Blackhole:

| Register | Purpose | Statefulness | Blackhole Equivalent |
| --- | --- | --- | --- |
| `RISC_IF_STATUS_OFFSET` | RISC interface status offset | Non-stateful (control flag) | Removed |

## State Preservation Consistency

Despite register availability differences, state preservation behavior is consistent across platforms for shared registers:

**Diagram: State Preservation Categories**

All stateful registers preserve their values across API calls on both platforms unless explicitly modified. This allows optimization patterns where address configuration, packet tags, and atomic operation parameters can be set once and reused for multiple operations.

## Architectural Evolution and Compatibility

### Design Continuity

**Diagram: Architectural Continuity from Wormhole to Blackhole**

The NOC architecture maintains fundamental design consistency between Wormhole B0 and Blackhole P100, with enhancements focused on performance and scalability rather than architectural redesign. This continuity enables high software compatibility while delivering significant performance improvements.

**Sources:**[data_movement_doc/general/intro_to_dm.md 32-34](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/intro_to_dm.md?plain=1#L32-L34)

### Migration Considerations

When migrating code from Wormhole B0 to Blackhole P100:

1.   **Register Usage**: Code using platform-specific registers (e.g., `RISC_IF_STATUS_OFFSET` on Wormhole, `CMD_BUF_AVAIL` on Blackhole) requires platform-specific compilation paths.

2.   **Address Space**: Operations using full 64-bit addressing require Blackhole. Wormhole code using only 48-bit addressing is forward-compatible.

3.   **Packet Sizes**: Code may need tuning for optimal packet sizes. Wormhole's 8KB limit vs Blackhole's 16KB limit affects batching strategies.

4.   **Performance Expectations**: Performance characteristics scale differently:

    *   Core-to-core: ~2x speedup (matches interface bandwidth)
    *   DRAM: ~1.5x speedup (DRAM controller limited)
    *   Multicast: ~1.6x speedup (congestion pattern limited)
