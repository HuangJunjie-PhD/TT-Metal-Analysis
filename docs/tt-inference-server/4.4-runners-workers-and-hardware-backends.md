The tt-media-server C++ backend utilizes a multi-process architecture to isolate high-performance Tenstorrent hardware runners from the main API service. This design ensures that heavy compute tasks, Python interpreter GIL constraints, and hardware-specific memory management do not block the primary request handling and scheduling logic.

## Worker Lifecycle and Fault Tolerance

The `WorkerManager` is responsible for spawning and monitoring worker processes. Each worker runs as a separate process to provide fault isolation; if a hardware kernel causes a segmentation fault or the Python interpreter crashes, only that specific worker is affected.

### Key Classes and Functions

*   **`WorkerManager`**: Manages the collection of `SingleProcessWorker` instances. It handles the initial `fork()` and `execv()` calls to transition from the main server process into a specialized worker process via `execWorkerProcessHelper`[tt-media-server/cpp_server/src/worker/worker_manager.cpp 29-53](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/worker_manager.cpp#L29-L53)[tt-media-server/cpp_server/src/worker/worker_manager.cpp 193-206](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/worker_manager.cpp#L193-L206)
*   **`SingleProcessWorker`**: Represents the lifecycle of a single worker. It installs fatal signal handlers (SIGSEGV, SIGABRT, etc.) via `installFatalSignalHandlers` to log crashes before the process terminates [tt-media-server/cpp_server/src/worker/single_process_worker.cpp 59-70](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/single_process_worker.cpp#L59-L70)[tt-media-server/cpp_server/src/worker/single_process_worker.cpp 85](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/single_process_worker.cpp#L85-L85)
*   **`startWarmupListener`**: A thread in the manager that waits for workers to signal they have finished hardware initialization and operator compilation using the `BoostIpcWarmupSignalQueue`[tt-media-server/cpp_server/src/worker/worker_manager.cpp 217-240](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/worker_manager.cpp#L217-L240)
*   **`checkWorkerAlive`**: Periodically called by the manager to check if worker PIDs are still active via `waitpid(WNOHANG)`. If a worker is found dead, it logs the exit status or signal and triggers `restartWorker`[tt-media-server/cpp_server/src/worker/worker_manager.cpp 128-166](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/worker/worker_manager.cpp#L128-L166)

## LLMRunner and Model Backends

The `LLMRunner` is the primary orchestrator within a worker process. It bridges the C++ `Scheduler` and the actual hardware-specific `IModelRunner` implementation.

### Implementation Details

*   **Scheduling**: It calls `scheduler->schedule()` to retrieve a batch of sequences ready for either prefill or decode [tt-media-server/cpp_server/src/runners/llm_runner.cpp 166](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/llm_runner.cpp#L166-L166)
*   **Guided Decoding**: Integrates `GuidedDecoderManager` to apply bitmasks to logits during `applyGuidedDecodingMasks`, ensuring generated text follows specific grammars or JSON schemas [tt-media-server/cpp_server/src/runners/llm_runner.cpp 135-154](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/llm_runner.cpp#L135-L154)
*   **Result Dispatch**: Once a model generates a token, `LLMRunner` processes it in a callback and pushes it to the `ipc::IResultQueue` via `ipc::pushToken`, where the main process picks it up for streaming [tt-media-server/cpp_server/src/runners/llm_runner.cpp 49-97](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/llm_runner.cpp#L49-L97)
*   **Cancellation**: Monitors the `cancelQueue` during every `step()` to abort requests as requested by the main service [tt-media-server/cpp_server/src/runners/llm_runner.cpp 157-164](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/llm_runner.cpp#L157-L164)

## Shared Memory and Multi-Mesh Runners

For extremely large models like DeepSeek V3 B1 that span multiple meshes (e.g., Galaxy systems), the server uses a specialized `SharedMemory` bridge.

*   **`SharedMemory` Bridge**: A Python mirror of the C++ shared memory ring buffer used for cross-process IPC. It implements a ring buffer with 64 slots [tt-media-server/cpp_server/src/runners/shared_memory.py 53-61](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/shared_memory.py#L53-L61)
*   **Inference Loop**: The `runner.py` script opens `c2p` (Controller-to-Processor) and `p2c` (Processor-to-Controller) shared memory buffers to receive `PrefillMessage` tasks and write back tokens [tt-media-server/cpp_server/src/runners/runner.py 82-115](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/runner.py#L82-L115)
*   **Multi-Mesh Setup**: Uses `ttnn.open_mesh_device` with a `MeshShape(4, 2)` for distributed execution across T3K/Galaxy hardware [tt-media-server/cpp_server/src/runners/runner.py 152-156](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/runner.py#L152-L156)
*   **Lifecycle Management**: The Python peer detaches from the `multiprocessing.resource_tracker` to ensure the C++ process remains the owner of the shared memory segment lifecycle [tt-media-server/cpp_server/src/runners/shared_memory.py 21-34](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/cpp_server/src/runners/shared_memory.py#L21-L34)
