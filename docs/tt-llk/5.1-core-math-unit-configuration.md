## Purpose and Scope

This document describes the low-level configuration of the math unit (TRISC1) in the tt-llk architecture, including ALU register configuration, destination register management, semaphore-based synchronization, and accumulation modes. The math unit receives data from the unpacker through SrcA and SrcB registers, performs computations, and writes results to destination registers for the packer.

For specific math operations (matrix multiplication, element-wise, SFPU), see [Element-wise Operations](https://deepwiki.com/tenstorrent/tt-llk/5.2-element-wise-operations), [SFPU Operations Library](https://deepwiki.com/tenstorrent/tt-llk/5.3-sfpu-operations-library), [Matrix Multiplication Operations](https://deepwiki.com/tenstorrent/tt-llk/5.4-matrix-multiplication-operations), and [Reduction and Pooling Operations](https://deepwiki.com/tenstorrent/tt-llk/5.5-reduction-and-pooling-operations). For fidelity and precision control mechanisms, see [Math Fidelity and Precision Control](https://deepwiki.com/tenstorrent/tt-llk/5.6-math-fidelity-and-precision-control).

## ALU Configuration Structure

The ALU configuration is controlled through the `alu_config_t` structure and three primary register sets: `ALU_FORMAT_SPEC_REG0`, `ALU_FORMAT_SPEC_REG1`, and `ALU_FORMAT_SPEC_REG2`.

### ALU Configuration Register Layout

**Diagram: ALU Configuration Register Structure**

The `alu_config_t` structure packs all configuration fields into a single 32-bit value for efficient read-modify-write operations:

| Field | Bits | Purpose |
| --- | --- | --- |
| `Fpu_srnd_en` | 1 | Enable stochastic rounding in FPU |
| `Gasket_srnd_en` | 1 | Enable stochastic rounding in gasket |
| `Packer_srnd_en` | 1 | Enable stochastic rounding in packer |
| `SrcAUnsigned` | 1 | Treat SrcA as unsigned integer |
| `SrcBUnsigned` | 1 | Treat SrcB as unsigned integer |
| `SrcA` | 4 | SrcA data format |
| `SrcB` | 4 | SrcB data format |
| `Dstacc` | 4 | Destination accumulator format |
| `Fp32_enabled` | 1 | Enable FP32 accumulation mode |
| `SFPU_Fp32_enabled` | 1 | Enable SFPU FP32 reading |
| `INT8_math_enabled` | 1 | Enable INT8 math operations |

## Math Hardware Configuration

The `_llk_math_hw_configure_` function performs initial hardware configuration for math operations, setting up data formats and accumulation modes.

### Configuration Flow

**Diagram: Math Hardware Configuration Sequence**

The configuration process:

1.   **Stall on pending configurations**: Wait for any pending configuration writes to complete
2.   **Detect INT8 mode**: Check if either source uses INT8 or INT32 formats
3.   **Build configuration word**: Pack source formats and FP32 flags into register value
4.   **Write ALU format register**: Update `ALU_FORMAT_SPEC_REG0` with new formats
5.   **Enable INT8 if needed**: Set `ALU_ACC_CTRL_INT8_math_enabled` flag

Key configuration parameters:

| Parameter | Type | Purpose |
| --- | --- | --- |
| `srca_data_format` | `uint32_t` | Source A data format from `DataFormat` enum |
| `srcb_data_format` | `uint32_t` | Source B data format from `DataFormat` enum |
| `is_fp32_dest_acc_en` | `bool` (template) | Enable 32-bit destination accumulation |

## FP32 Destination Accumulation Mode

FP32 destination accumulation mode enables 32-bit precision in destination registers at the cost of halved capacity. This mode is controlled by the `is_fp32_dest_acc_en` template parameter throughout the math pipeline.

### Capacity and Precision Trade-off

**Diagram: FP32 Mode Impact on Destination Register Organization**

### Configuration Requirements

When FP32 mode is enabled, multiple configuration registers must be updated consistently:

1.   **ALU Accumulator Control**: Set `ALU_ACC_CTRL_Fp32_enabled` and `ALU_ACC_CTRL_SFPU_Fp32_enabled`
2.   **Packer Read Control**: Configure `PCK_DEST_RD_CTRL_Read_32b_data` in packer
3.   **Unpacker Destination Format**: May need adjustment for proper format conversion
4.   **ZEROACC Instructions**: Use 32B variants (`CLR_HALF_32B`, `CLR_ALL_32B`)

The mode affects destination register addressing calculations throughout the pipeline:

```
Standard mode:  DEST_REGISTER_HALF_SIZE = 256 rows (8 tiles)
FP32 mode:      DEST_REGISTER_HALF_SIZE = 128 rows (4 tiles)
```

## Address Modifiers for Math Operations

Address modifiers control how source and destination address counters increment during math operations. The math unit uses 8 address modifier slots (ADDR_MOD_0 through ADDR_MOD_7) configured through the `addr_mod_t` structure.

### Address Modifier Structure

**Diagram: Address Modifier Field Organization and Counter Hierarchy**

### Common Address Modifier Patterns

Different math operations use standard address modifier configurations:

**Element-wise Binary Operations** ([tt_llk_wormhole_b0/llk_lib/llk_math_eltwise_binary.h 19-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_eltwise_binary.h#L19-L43)):

*   `ADDR_MOD_0`: Increment SrcA, SrcB (if not broadcast), Dest by 8 rows
*   `ADDR_MOD_1`: No increments (fixed address)
*   `ADDR_MOD_2`: Clear SrcA/SrcB, wrap Dest to next face, increment fidelity
*   `ADDR_MOD_3`: Clear all, wrap Dest, clear fidelity

**Matrix Multiplication** ([tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h 23-297](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_matmul.h#L23-L297)):

*   `ADDR_MOD_0`: Increment SrcB and Dest by 8 rows (inner loop)
*   `ADDR_MOD_1`: Increment SrcA by 16, wrap SrcB to next face
*   `ADDR_MOD_2`: Clear SrcA, increment SrcB by 32, increment Dest by 8
*   `ADDR_MOD_3`: Copy of ADDR_MOD_0 with bias increment
*   `ADDR_MOD_4`: Complex wrapping for next tile
*   `ADDR_MOD_5`: Reset with fidelity increment (HiFi modes)
*   `ADDR_MOD_6`: Full reset including fidelity (throttling)

**Reduction Operations** ([tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 233-250](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L233-L250)):

*   `ADDR_MOD_0`: Simple 8-row increment for pooling instructions
*   `ADDR_MOD_3`: No increments for fixed-address SFPU operations

### Fidelity Counter Integration

High-fidelity math operations use the fidelity field to iterate through multiple computation phases:

```
LoFi:  fidelity.incr = 0 (no iterations)
HiFi2: fidelity.incr = 1 (1 additional iteration)
HiFi3: fidelity.incr = 1 (2 additional iterations)
HiFi4: fidelity.incr = 1 (3 additional iterations)
```

Address modifier slots 5 and 6 typically handle fidelity management:

*   `ADDR_MOD_5`: Increment fidelity, reset counters (continue iterating)
*   `ADDR_MOD_6`: Clear fidelity, reset counters (complete operation)
