## Purpose and Scope

This page documents the command-line interface for `riescued`, the executable entry point for the RiescueD Directed Test Framework. It covers all command-line arguments, options, and their usage patterns.

For information about RiescueD's internal architecture and how it processes tests, see [RiescueD Internal Architecture](https://deepwiki.com/tenstorrent/riescue/2.1.3-riescued-internal-architecture). For documentation about test directives and syntax used in `.rasm` files, see [Test Directives and Syntax](https://deepwiki.com/tenstorrent/riescue/2.1.2-test-directives-and-syntax). For broader context about RiescueD's role in the testing framework, see [RiescueD: Directed Test Framework](https://deepwiki.com/tenstorrent/riescue/2.1-riescued:-directed-test-framework).

## Overview

The `riescued` command-line tool is installed as a Python entry point that invokes the `RiescueD` class. It takes a RISC-V assembly test file (with RiescueD directives), processes it through parsing, generation, compilation, and optionally simulation stages.

The CLI supports three primary workflows:

1.   **Full Run**: Parse, generate, compile, and optionally simulate a test
2.   **Elaborate Only**: Parse and generate assembly/linker scripts without compilation
3.   **Build and Simulate**: Complete workflow including ISS execution

## Basic Usage Syntax

`riescued --testfile <path/to/test.rasm> [options]`
Minimal example:

`riescued --testfile my_test.s`
With simulation:

`riescued --testfile my_test.s --run_iss`
With specific configuration:

`riescued --testfile my_test.s \  --cpuconfig cpu_config.json \  --run_dir ./output \  --seed 12345 \  --run_iss`
**Sources:**[riescue/riescued.py 141-157](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L141-L157)[README.md 75-96](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L75-L96)

## Feature Manager Arguments

The `FeatMgrBuilder` class adds a comprehensive set of arguments for controlling test generation features. These arguments override test header directives and CPU configuration:

### Privilege and Environment Arguments

| Argument | Values | Description |
| --- | --- | --- |
| `--privilege_mode` | `machine`, `super`, `user`, `any` | Override privilege mode for test execution |
| `--test_env` | `bare_metal`, `virtualized` | Override test environment |
| `--paging_mode` | `sv39`, `sv48`, `sv57`, `bare`, `any`, `disable` | Override virtual memory paging mode |
| `--paging_g_mode` | `sv39x4`, `sv48x4`, `sv57x4`, `bare`, `any`, `disable` | Override guest-stage paging mode for virtualized tests |

### Architecture Arguments

| Argument | Description |
| --- | --- |
| `--march` | Override ISA string for compilation (e.g., `rv64imafdc`) |
| `--mabi` | Override ABI for compilation (e.g., `lp64d`) |

### Extension and Feature Overrides

The feature manager supports enabling/disabling specific RISC-V extensions via command-line arguments. These are dynamically generated based on the CPU configuration.

## Usage Examples

### Example 1: Basic Test Generation

`riescued --testfile tests/my_test.s`
This generates:

*   `my_test.S` - Elaborated assembly file
*   `my_test.ld` - Linker script
*   `my_test.elf` - Compiled ELF executable
*   `my_test.dis` - Disassembly

**Sources:**[riescue/riescued.py 188-220](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L188-L220)

### Example 2: Elaborate Only (No Compilation)

`riescued --testfile tests/my_test.s --elaborate_only`
Generates only `.S` and `.ld` files without invoking the compiler. Useful for debugging test generation or when external compilation is handled separately.

**Sources:**[riescue/riescued.py 206-208](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L206-L208)

### Example 3: Full Simulation Run

`riescued --testfile tests/my_test.s \  --run_iss \  --iss whisper \  --seed 42 \  --log_level DEBUG`
Executes the complete workflow including simulation on Whisper ISS with a specific seed and debug logging enabled.

**Sources:**[riescue/riescued.py 211-218](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L211-L218)

### Example 4: Override Test Configuration

`riescued --testfile tests/my_test.s \  --privilege_mode machine \  --paging_mode sv39 \  --test_env bare_metal \  --march rv64imafdc \  --run_iss`
Overrides test header directives to force specific privilege mode, paging mode, and environment settings.

**Sources:**[README.md 94-104](https://github.com/tenstorrent/riescue/blob/07cefde3/README.md?plain=1#L94-L104)[riescue/dtest_framework/README.md 97-104](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/README.md?plain=1#L97-L104)

### Example 5: Custom CPU Configuration

`riescued --testfile tests/my_test.s \  --cpuconfig configs/custom_cpu.json \  --run_dir output/test_run`
Uses a custom CPU configuration file and writes all output to a specific directory.

**Sources:**[riescue/riescued.py 47-60](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L47-L60)

### Example 6: Virtualized Environment Test

`riescued --testfile tests/hypervisor_test.s \  --test_env virtualized \  --paging_mode sv39 \  --paging_g_mode sv39x4 \  --run_iss`
Tests hypervisor functionality with two-stage address translation.

**Sources:**[tests/cli_tests/riescued/hypervisor_test.py 11-24](https://github.com/tenstorrent/riescue/blob/07cefde3/tests/cli_tests/riescued/hypervisor_test.py#L11-L24)

## Argument Resolution and Precedence

Configuration values are resolved in the following precedence order (highest to lowest):

1.   **Command-line arguments** - Explicit CLI flags override all other sources
2.   **Test header directives** - Directives in `.rasm` file (e.g., `;#test.priv machine`)
3.   **CPU configuration file** - Default features from `cpuconfig` JSON
4.   **Built-in defaults** - Hardcoded defaults in `FeatMgr`

This precedence is implemented in the `FeatMgrBuilder` class:

## Path Resolution

The `RiescueD._resolve_path()` method resolves file paths in the following search order:

1.   Absolute path or relative to current working directory
2.   Relative to package installation directory
3.   Relative to `run_dir`
4.   Relative to parent of package installation directory (legacy compatibility)

This allows flexible file references without requiring absolute paths:

`# All of these can work depending on file location:riescued --testfile my_test.sriescued --testfile ./tests/my_test.sriescued --testfile /absolute/path/to/my_test.sriescued --cpuconfig dtest_framework/lib/config.json  # Found in package`
**Sources:**[riescue/riescued.py 413-432](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L413-L432)

## Exit Behavior

The CLI exits with the following behavior:

*   **Success (exit code 0)**: Test generation/compilation/simulation completed without errors
*   **Exception raised**: Python exception with traceback if any stage fails 
    *   `FileNotFoundError`: Test file or config file not found
    *   `ValueError`: Invalid argument combinations or missing required arguments
    *   `RunnerError`: ISS simulation detected incorrect test behavior
    *   Tool execution errors: Compiler or ISS invocation failures

**Sources:**[riescue/riescued.py 435-440](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L435-L440)

## Integration with Python API

The CLI is a thin wrapper around the `RiescueD` class API. For programmatic usage, the class can be instantiated directly:

`from pathlib import Pathfrom riescue.riescued import RiescueDfrom riescue.lib.toolchain import Toolchain rd = RiescueD(    testfile=Path("tests/my_test.s"),    cpuconfig=Path("configs/cpu_config.json"),    run_dir=Path("./output"),    seed=42,    toolchain=Toolchain()) # Manual control over each stagefeatmgr = rd.configure()rd.generate(featmgr)rd.build(featmgr)rd.simulate(featmgr, iss=rd.toolchain.simulator)`
This provides more flexibility than the CLI for test automation and integration into larger systems.
