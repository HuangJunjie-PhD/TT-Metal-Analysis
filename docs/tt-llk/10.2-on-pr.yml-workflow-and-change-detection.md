## Purpose and Scope

This document describes the main PR pipeline workflow defined in `.github/workflows/on-pr.yml`, which orchestrates all validation activities for pull requests. The workflow implements intelligent change detection to conditionally execute architecture-specific tests, automatic PR labeling, and comprehensive status verification.

For details on test execution, see [setup-and-test.yml: Compilation and Execution](/tenstorrent/tt-llk/10.3-setup-and-test.yml:-compilation-and-execution-phases). For tt-metal integration testing, see [tt-metal Integration Testing](/tenstorrent/tt-llk/10.4-architecture-specific-test-execution).

## Workflow Overview

The `on-pr.yml` workflow is the primary CI/CD orchestrator for the tt-llk repository. It detects which files have changed, automatically labels PRs, builds Docker images, and conditionally triggers test suites for Wormhole, Blackhole, and Quasar architectures based on the modified files.

## Trigger Events and Concurrency Control

### Trigger Events

The workflow responds to multiple trigger types to validate different stages of the development lifecycle:

| Event Type | Trigger Conditions | Purpose |
| --- | --- | --- |
| `push` | Branch: `main` | Validate commits merged to main branch |
| `pull_request` | Types: `opened`, `synchronize`, `reopened`, `ready_for_review` | Validate PR changes on creation, updates, and reopening |
| `workflow_dispatch` | Manual trigger | Enable manual workflow execution for testing |
| `merge_group` | Merge queue events | Validate changes in GitHub merge queue |

**Sources:** [.github/workflows/on-pr.yml2-9](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L2-L9)

### Concurrency Strategy

The workflow implements context-aware concurrency control to prevent redundant executions:

**Concurrency Logic:**

1. **Protected branches (main):** Uses `github.run_id` to ensure no runs are cancelled
2. **Pull requests:** Uses `github.event.pull_request.number` to cancel obsolete runs when PR is updated
3. **Other branches:** Uses `github.ref` for unique grouping per branch

**Cancellation behavior:**

* **PR updates:** Superseded runs are automatically cancelled
* **Main branch:** No cancellation occurs to preserve historical build data

**Sources:** [.github/workflows/on-pr.yml17-25](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L17-L25)

## Change Detection System

The `detect-changes` job uses `tj-actions/changed-files` to analyze file path patterns and determine which test suites should execute. This pattern-based detection enables conditional test execution, reducing CI costs by skipping irrelevant test suites.

### Change Detection Implementation

**File Pattern Analysis Flow**

**Sources:** [.github/workflows/on-pr.yml32-76](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L32-L76)

### File Pattern Configuration

The workflow defines seven pattern groups using YAML configuration:

**Pattern Features:**

* **Negative patterns:** The `tests` group uses `!` prefix to exclude Quasar-specific test files
* **Wildcard matching:** `**` matches any directory depth
* **Multiple patterns:** Each group can have multiple file patterns
* **Cross-directory patterns:** Performance pattern matches across all directories

**Sources:** [.github/workflows/on-pr.yml54-75](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L54-L75)

### Change Detection Outputs

The `detect-changes` job produces boolean outputs for each pattern group:

| Output | Pattern Match | Used By |
| --- | --- | --- |
| `wormhole` | `tt_llk_wormhole_b0/**` | `setup-and-test-wormhole` job |
| `blackhole` | `tt_llk_blackhole/**` | `setup-and-test-blackhole` job |
| `quasar` | `tt_llk_quasar/**`, quasar tests | `build-quasar` job |
| `tests` | `tests/**` (excluding quasar) | All test jobs |
| `github` | `.github/**` | All test jobs (CI changes) |
| `documentation` | `docs/**`, `*.md`, `*.rst` | `label-pr` job only |
| `performance` | `**/*perf**`, `**/*profiler**` | `perf-tests` job |

**Sources:** [.github/workflows/on-pr.yml36-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L36-L43)

## Conditional Test Execution

The workflow conditionally triggers test suites based on change detection outputs, implementing an architecture-aware testing strategy that minimizes unnecessary test execution.

### Test Job Matrix

**Test Execution Decision Matrix**

| Job | Condition | Runners | Test Splits | pytest Markers | Timeout |
| --- | --- | --- | --- | --- | --- |
| `setup-and-test-wormhole` | `wormhole` OR `tests` OR `github` | `n150-stable` | 2 | `not perf and not nightly and not quasar` | 80 min |
| `setup-and-test-blackhole` | `blackhole` OR `tests` OR `github` | `p150b-stable` | 2 | `not perf and not nightly and not quasar` | 80 min |
| `build-quasar` | `quasar` | `xlarge-stable` | 1 | `quasar` | 80 min |
| `perf-tests` (Wormhole) | `performance` | `n150-stable` | 5 | `perf` | 240 min |
| `perf-tests` (Blackhole) | `performance` | `p150b-stable` | 5 | `perf` | 240 min |

**Sources:** [.github/workflows/on-pr.yml144-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L206)

### Wormhole Test Configuration

**Conditional logic breakdown:**

* **`github.event_name != 'push'`:** Skips tests on direct pushes to main (already validated in PR)
* **`wormhole == 'true'`:** Runs if Wormhole-specific files changed
* **`tests == 'true'`:** Runs if test infrastructure files changed
* **`github == 'true'`:** Runs if CI workflow files changed

**Sources:** [.github/workflows/on-pr.yml144-157](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L157)

### Blackhole Test Configuration

**Key differences from Wormhole:**

* **Conditional:** Checks `blackhole` output instead of `wormhole`
* **Runners:** Uses `p150b-stable` (Blackhole hardware) instead of `n150-stable`

**Sources:** [.github/workflows/on-pr.yml159-172](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L159-L172)

### Quasar Build Configuration

**Quasar-specific behavior:**

* **Compile-only:** Uses `build-quasar.yml` which only compiles tests (no hardware execution)
* **Single-condition:** Only runs when `quasar` files change (not for `tests` or `github` changes)
* **Large runners:** Uses `xlarge-stable` for faster parallel compilation
* **No duration-based splitting:** Quasar tests not included in duration collection

**Sources:** [.github/workflows/on-pr.yml174-184](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L174-L184)

### Performance Test Configuration

**Performance test characteristics:**

* **Matrix strategy:** Runs on both architectures using `fail-fast: false`
* **Higher split count:** 5 splits vs 2 for regular tests (more parallelism for long-running tests)
* **Extended timeout:** 240 minutes vs 80 minutes for regular tests
* **Selective execution:** Only runs when performance-related files change

**Sources:** [.github/workflows/on-pr.yml186-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L186-L206)

## All-Green Verification

The `check-all-green` job serves as the final status check, verifying that all required jobs completed successfully. This job is configured as a required status check in GitHub branch protection rules.

### Verification Job Configuration

**Verification logic:**

* **`if: always()`:** Runs even if upstream jobs fail or are skipped
* **`needs` dependencies:** Waits for all test jobs to complete
* **`re-actors/alls-green` action:** Evaluates job statuses and allows specified jobs to be skipped

**Sources:** [.github/workflows/on-pr.yml208-224](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L208-L224)

### Allowed Skips Configuration

The `allowed-skips` parameter defines which jobs can be skipped without failing the verification:

**Skip scenarios:**

| Job | Can Skip | Reason |
| --- | --- | --- |
| `pre-commit` | No | Always required |
| `build-images` | No | Always required |
| `setup-and-test-wormhole` | Yes | Skipped if no Wormhole changes |
| `setup-and-test-blackhole` | Yes | Skipped if no Blackhole changes |
| `build-quasar` | Yes | Skipped if no Quasar changes |
| `perf-tests` | Yes | Skipped if no performance changes |

**Status evaluation:**

* **Success:** All required jobs pass; allowed jobs may be skipped
* **Failure:** Any required job fails, or any non-skipped job fails

This design ensures the check-all-green job always provides a reliable pass/fail signal for branch protection, regardless of which conditional test suites actually executed.

**Sources:** [.github/workflows/on-pr.yml208-224](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L208-L224)

## Integration with Reusable Workflows

The `on-pr.yml` workflow delegates execution to specialized reusable workflows, promoting code reuse and separation of concerns.

### Reusable Workflow Invocation Pattern

**Sources:** [.github/workflows/on-pr.yml30-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L30-L206)

### Workflow Call Syntax

The workflow uses `uses` keyword to invoke reusable workflows:

**Invocation features:**

* **`uses` path:** References workflow file in same repository
* **`needs` dependencies:** Ensures docker image is built and changes are detected before execution
* **`if` conditionals:** Controls whether workflow executes based on change detection
* **`with` inputs:** Passes parameters to reusable workflow
* **Output propagation:** Docker image tag flows from `build-images` to test jobs

**Sources:** [.github/workflows/on-pr.yml144-157](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L157)

### Parameter Variations Across Invocations

The same `setup-and-test.yml` workflow is invoked with different parameters:

**Regular Tests:**

* **Test splits:** 2 (lower parallelism for faster feedback)
* **pytest markers:** `not perf and not nightly and not quasar` (regular test suite)
* **Timeout:** 80 minutes (default from reusable workflow)
* **Duration-based splitting:** Enabled (`use_durations: true`)

**Performance Tests:**

* **Test splits:** 5 (higher parallelism for long-running tests)
* **pytest markers:** `perf` (performance benchmark suite)
* **Timeout:** 240 minutes (4-hour timeout for extensive benchmarks)
* **Duration-based splitting:** Enabled

**Quasar Tests:**

* **Uses different workflow:** `build-quasar.yml` (compile-only, no execution)
* **Test splits:** 1 (single compilation job)
* **pytest markers:** `quasar` (Quasar-specific tests)

This parameterization enables code reuse while supporting diverse testing requirements across architectures and test types.

**Sources:** [.github/workflows/on-pr.yml144-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L206)

## Workflow Dependency Graph

The following diagram illustrates the complete dependency structure of the on-pr.yml workflow:

**Complete Job Dependency Flow**

**Dependency characteristics:**

* **Sequential foundation:** `pre-commit` must complete before `detect-changes` and `build-images`
* **Parallel execution:** Once pre-commit passes, change detection and image building run in parallel
* **Test parallelism:** All test jobs (`WH`, `BH`, `QS`, `PERF`) can run simultaneously once dependencies are met
* **Final gate:** `check-all-green` waits for all jobs but uses `if: always()` to evaluate even if some jobs are skipped

## Permissions and Security Model

The workflow declares specific GitHub token permissions to follow the principle of least privilege:

**Permission usage by job:**

| Permission | Used By | Purpose |
| --- | --- | --- |
| `checks: write` | `setup-and-test-*`, `check-all-green` | Publish JUnit reports and status checks |
| `contents: read` | All jobs | Clone repository and read source files |
| `packages: write` | `build-images` | Push Docker images to Harbor registry |
| `pull-requests: write` | `label-pr` | Modify PR labels via GitHub API |
