# Generating Traces

Relevant source files
*   [README.md](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1)
*   [images/example_perf_report.png](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/images/example_perf_report.png)

This document explains how to generate performance traces for analysis with the tt-perf-report tool. Performance traces capture timing data from TT-Metal operations, which the tool uses to provide insights about throughput, bottlenecks, and optimization opportunities. For information about analyzing these traces, see [Command Line Options](https://deepwiki.com/tenstorrent/tt-perf-report/3.1-command-line-options) and [Performance Report Format](https://deepwiki.com/tenstorrent/tt-perf-report/4-performance-report-format).

## Overview of Trace Generation Process

Before analyzing performance with tt-perf-report, you need to capture performance data during model execution. This involves enabling performance tracing in your TT-Metal build and using the tracy module to collect timing data.

Sources: [README.md 17-28](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1#L17-L28)

## Building TT-Metal with Performance Tracing

The first step is to build TT-Metal with performance tracing enabled:

1.   Use the `-p` flag with the build_metal script:

```
./build_metal -p
```

This flag enables the necessary instrumentation in the TT-Metal framework to capture performance data during execution.

Sources: [README.md 19-22](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1#L19-L22)

## Capturing Traces with Tracy Module

Once you have built TT-Metal with tracing enabled, you can use the tracy module to capture traces when running your tests:

```
python -m tracy -r -p -v -m pytest path/to/test.py
```

### Command-line Parameters

| Flag | Description |
| --- | --- |
| `-r` | Record trace data |
| `-p` | Profile operations |
| `-v` | Verbose output |
| `-m` | Module (e.g., pytest) |

The tracy module will generate a CSV file containing detailed timing information about all operations executed by TT-Metal.

Sources: [README.md 24-28](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1#L24-L28)

## Using Tracy Signposts

Tracy signposts are markers in your code that help identify specific sections for analysis. They're particularly useful when your test contains multiple phases such as initialization, compilation, and actual inference.

### Adding Signposts to Your Code

To add signposts to your Python code:

`import tracy # Mark different sections of your codetracy.signpost("Compilation pass")model(input_data) tracy.signpost("Performance pass")for _ in range(10):    model(input_data)`
By default, tt-perf-report analyzes operations after the last signpost, which typically represents the performance-critical section (e.g., after compilation and warmup).

Sources: [README.md 30-46](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1#L30-L46)

## Signpost Options in tt-perf-report

When analyzing traces with tt-perf-report, you can control which section to analyze using these options:

| Option | Description |
| --- | --- |
| `--signpost name` | Analyze operations after the specified signpost |
| `--ignore-signposts` | Analyze the entire trace file |

Example:

```
tt-perf-report trace.csv --signpost "Performance pass"
```

Sources: [README.md 47-51](https://github.com/tenstorrent/tt-perf-report/blob/076a1b39/README.md?plain=1#L47-L51)

## Trace File Structure

The generated trace CSV file contains detailed timing information for each operation executed by TT-Metal. This data includes:

1.   Execution timestamps
2.   Operation types
3.   Core utilization
4.   Memory bandwidth consumption
5.   Compute throughput
6.   Host-to-device communication overhead

While the specific columns can vary based on the traced operations, tt-perf-report is designed to parse and analyze all relevant metrics.

## Best Practices for Trace Generation

### 1. Separate Compilation and Performance Passes

`import tracy # First pass - compilation (will be ignored by default)tracy.signpost("Compilation")output = model(input_data) # Second pass - actual performance measurementtracy.signpost("Performance")for _ in range(10):  # Run multiple iterations for stable measurements    output = model(input_data)`
### 2. Use Multiple Runs for Stable Measurements

Performance can vary between runs. For reliable metrics, run your model multiple times after warmup:

`# Warmup (will be ignored by default if you add a signpost after)for _ in range(2):    model(input_data) # Performance measurementtracy.signpost("Performance measurement")for _ in range(10):    model(input_data)`
### 3. Isolate Specific Components

If you're interested in the performance of specific layers or operations, use signposts to mark those sections:

`# Run model up to a specific pointtracy.signpost("Before transformer block")model.embedding(input_data) # Measure specific componenttracy.signpost("Transformer block")model.transformer_block(embedded_data) # Measure another componenttracy.signpost("Output projection")model.projection(transformed_data)`

## Troubleshooting

### Common Issues

1.   **Empty trace files**: Ensure you've built TT-Metal with the `-p` flag
2.   **Missing operations**: Check if operations occurred after your signpost
3.   **Large trace files**: Use signposts to limit capture to relevant sections

### Verifying Trace Generation

To verify that your trace file contains the expected data:

```
head -n 10 trace.csv    # View the first few lines
wc -l trace.csv         # Check total number of entries
```

A properly generated trace file should contain numerous entries with timestamps, operation names, and performance metrics.

## Integration with Analysis Workflow

The trace generation process is typically part of an iterative development cycle:

1.   Implement or modify your model
2.   Generate a performance trace with tracy
3.   Analyze the trace with tt-perf-report
4.   Identify performance bottlenecks
5.   Optimize your code
6.   Repeat the process to verify improvements
