This page documents the callstack unwinding API in TTExaLens: the `callstack` and `top_callstack` library functions, the `CallstackEntry` data structure, how ELF files and load offsets are used for symbol resolution, and how DWARF Call Frame Information (CFI) drives the frame-unwinding logic.

For the lower-level RISC-V core control operations that callstack unwinding depends on (halting, reading GPRs, reading memory), see [RISC-V Core Control](https://deepwiki.com/tenstorrent/tt-exalens/3.6-risc-v-core-control). For ELF loading and execution, see [ELF Management](https://deepwiki.com/tenstorrent/tt-exalens/3.5-elf-management). For the deeper implementation of `FrameInfoProvider` and DWARF CFI parsing, see [Callstack Unwinding](https://deepwiki.com/tenstorrent/tt-exalens/7.5-callstack-unwinding).

* * *

## Overview

When a RISC-V firmware core is running on Tenstorrent hardware, it is often necessary to determine _where_ in the code the core is executing and how it arrived there. TTExaLens does this by:

1.   Reading the current program counter (PC) and stack pointer (SP) from the halted core via the debug hardware.
2.   Using the DWARF `.debug_frame` / `.eh_frame` sections embedded in the firmware ELF to iteratively unwind stack frames.
3.   Resolving each return address to a source location using the ELF's symbol and line-number tables.

Multiple ELF files can be supplied in one call (e.g., to cover code that spans a boot loader and an application), and an optional per-ELF load offset adjusts symbol addresses when the firmware is position-independently loaded.

## Key Data Types

### `CallstackEntry`

`CallstackEntry` is the result type returned for each frame in a callstack. It is defined in `ttexalens/hardware/risc_debug.py` and imported by the library layer.

| Field | Type | Description |
| --- | --- | --- |
| `pc` | `int | None` | Absolute program counter for this frame. |
| `function_name` | `str | None` | Symbol name resolved from the ELF. |
| `file` | `str | None` | Source file path from DWARF line info. |
| `line` | `int | None` | Source line number. |
| `column` | `int | None` | Source column number. |

A `None` value in any field means the information was not resolvable—either no ELF was provided, the address falls outside all known symbol ranges, or the DWARF info is incomplete.

## Library API

Both functions live in `ttexalens/tt_exalens_lib.py` and follow the same decorator and context-resolution conventions used by the rest of the library (see [Context and Initialization](https://deepwiki.com/tenstorrent/tt-exalens/3.1-context-and-initialization)).

### `callstack`

```
callstack(
    location,
    elfs,
    offsets,
    risc_name,
    max_depth=100,
    stop_on_main=True,
    neo_id=None,
    device_id=0,
    context=None,
) -> list[CallstackEntry]
```

| Parameter | Type | Description |
| --- | --- | --- |
| `location` | `str | OnChipCoordinate` | Core location on chip. |
| `elfs` | `list[ParsedElfFile]` | Parsed ELF files for symbol resolution. |
| `offsets` | `list[int | None]` | Per-ELF load offsets; `None` means no adjustment. |
| `risc_name` | `str` | Core name: `brisc`, `trisc0`, `trisc1`, `trisc2`, `ncrisc`, `erisc`, etc. |
| `max_depth` | `int` | Maximum number of frames to unwind. |
| `stop_on_main` | `bool` | Stop unwinding when `main` is reached. |
| `neo_id` | `int | None` | NEO ID for Quasar devices. |
| `device_id` | `int` | Device index. |
| `context` | `Context | None` | TTExaLens context; auto-initialized if `None`. |

Returns a `list[CallstackEntry]`, ordered from the innermost (most recent) frame to the outermost.

### `top_callstack`

```
top_callstack(
    location,
    elfs,
    offsets,
    risc_name,
    neo_id=None,
    device_id=0,
    context=None,
) -> CallstackEntry
```

A convenience wrapper that returns only the single topmost `CallstackEntry`—i.e., where the core is currently executing. Useful when you only need the current function and source line without full unwinding.

### `parse_elf`

```
parse_elf(elf_path, context=None) -> ParsedElfFile
```

Parses an ELF file and returns a `ParsedElfFile` object. The result should be passed to `callstack` or `top_callstack`. `ParsedElfFile` caches DWARF symbol tables, type information, and frame CFI data as lazy properties.

## ELF Files and Load Offsets

ELF files serve two purposes during callstack unwinding:

1.   **Symbol resolution** — mapping a return-address PC to a function name and source location via DWARF `.debug_info` and `.debug_line`.
2.   **Frame unwinding** — reading the DWARF `.debug_frame` or `.eh_frame` sections to determine how to recover the caller's SP and return address from each frame.

When firmware is loaded at a base address that differs from the addresses in the ELF (for example, when a loader places code at a runtime-determined location), each ELF entry in `elfs` can be paired with an `offset` value. The offset is added to all ELF-internal addresses before comparison with the live PC values read from hardware.

If `offset` is `None` for a given ELF, symbol addresses are used as-is.

## DWARF CFI and Frame Unwinding

DWARF Call Frame Information (CFI) encodes, for every instruction address in a compiled program, how to compute the caller's canonical frame address (CFA) and recover callee-saved registers (including the return address, `ra` on RISC-V).

`FrameInfoProvider` (in the ELF/DWARF layer, see [Callstack Unwinding](https://deepwiki.com/tenstorrent/tt-exalens/7.5-callstack-unwinding)) uses the `frame_info` cached property of `ParsedElfFile` to:

1.   Find the CFI row matching the current PC.
2.   Apply the CFA rule (usually `SP + constant_offset`) to compute the caller's frame base.
3.   Recover the return address register from the CFI column rule.
4.   Advance to the next frame by setting `PC = RA` and repeating.

This process requires no frame pointer (`s0`/`fp`) and is robust to compiler optimizations such as frame-pointer omission (`-fomit-frame-pointer`).

## CLI Command: `callstack` / `bt`

The `callstack` command (short alias `bt`) is the interactive equivalent of the library functions. It is implemented in `ttexalens/cli_commands/callstack.py`.

**Usage:**

```
callstack <elf-files> [-o <offsets>] [-r <risc>] [-m <max-depth>] [-d <device>] [-l <loc>]
```

| Argument / Option | Description |
| --- | --- |
| `<elf-files>` | Comma-separated paths to one or more ELF files. |
| `-o <offsets>` | Comma-separated integer offsets (hex or decimal) matching `<elf-files>`. |
| `-r <risc>` | RISC-V core name. Defaults to the first available core at the location. |
| `-m <max-depth>` | Maximum frames to print. Default: `100`. |
| `-d <device>` | Device ID. |
| `-l <loc>` | Core location (e.g., `1,0` for logical or `1-0` for noc0). |

**Example output:**

```
Location: 1,0, core: brisc
  #0  0x00008042 in compute_kernel () at kernel.cc 87:8
  #1  0x00007F10 in dispatch_loop () at dispatch.cc 34:4
  #2  0x00000010 in main () at main.cc 12:0
```

When `-r` is omitted, the command picks the first RISC reported by `NocBlock.all_riscs`.

## Prerequisites and Constraints

*   **Core state**: The RISC-V core does not need to be formally halted before calling `callstack`, but the PC and SP values read must be stable for the result to be meaningful. In practice, the core is typically halted (via `RiscDebug.halt()` or an `ebreak` instruction) before requesting a callstack.
*   **ELF availability**: Without at least one ELF, `callstack` will still return a list of `CallstackEntry` objects, but `function_name`, `file`, `line`, and `column` will all be `None`. Only `pc` will be populated.
*   **Debug symbols**: ELF files must be compiled with debug info (`-g`) for DWARF line-number and CFI data to be present. Release builds compiled with `-O3` and stripped of debug info will not produce useful callstacks.
*   **Multiple ELFs**: When firmware is split across multiple linked ELF files (e.g., a boot stub and an application), all of them should be passed to `elfs`. The unwinding logic searches all provided ELF files to resolve each PC.
