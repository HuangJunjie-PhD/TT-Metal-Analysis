Model Runners are the hardware-abstraction layer of the `tt-inference-server`. They encapsulate the logic for model loading, weight distribution, hardware warmup, and inference execution on Tenstorrent devices. Each runner is responsible for a specific model architecture or inference engine (e.g., `ttnn`, `vLLM`, or `Forge`).

## Base Runner Hierarchy

The runner system is built on a hierarchical class structure that separates general runner interfaces from hardware-specific implementations.

### BaseDeviceRunner

The root abstract class [tt-media-server/tt_model_runners/base_device_runner.py 10-15](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_device_runner.py#L10-L15) defining the core interface:

*   `warmup()`: Asynchronous method for model loading and initial hardware execution [tt-media-server/tt_model_runners/base_device_runner.py 25-27](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_device_runner.py#L25-L27)
*   `run(requests)`: Synchronous method for executing inference [tt-media-server/tt_model_runners/base_device_runner.py 34-36](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_device_runner.py#L34-L36)
*   `is_request_batchable(request, batch)`: Logic to determine if a new request can be appended to an existing batch [tt-media-server/tt_model_runners/base_device_runner.py 43-45](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_device_runner.py#L43-L45)

### BaseMetalDeviceRunner

Inherits from `BaseDeviceRunner` and provides `ttnn`-specific device management [tt-media-server/tt_model_runners/base_metal_device_runner.py 15-20](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_metal_device_runner.py#L15-L20) It handles:

*   **Device Initialization**: Opening `ttnn` devices or meshes based on `RuntimeConfig`[tt-media-server/tt_model_runners/base_metal_device_runner.py 45-55](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_metal_device_runner.py#L45-L55)
*   **Fabric Configuration**: Setting up Tenstorrent Fabric for multi-chip communication (e.g., `FABRIC_1D`) [tt-media-server/tt_model_runners/dit_runners.py 50-73](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/dit_runners.py#L50-L73)
*   **Trace Management**: Managing hardware traces for optimized execution loops.

### Runner Fabric Factory

The `runner_fabric.py` module acts as a factory, mapping `ModelRunners` enum values to their concrete implementations [tt-media-server/tt_model_runners/runner_fabric.py 10-133](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L10-L133)

**Runner Resolution Flow:**

1.   `get_device_runner(worker_id)` is called by the `device_worker`[tt-media-server/tt_model_runners/runner_fabric.py 136-138](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L136-L138)
2.   The factory looks up the runner class in `AVAILABLE_RUNNERS` using the `settings.model_runner` configuration [tt-media-server/tt_model_runners/runner_fabric.py 138-144](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L138-L144)
3.   The runner is dynamically imported and instantiated, ensuring only required dependencies are loaded into the worker process [tt-media-server/tt_model_runners/runner_fabric.py 10-132](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L10-L132)

## Concrete Runner Implementations

### Diffusion and DiT Runners (SDXL, SD3.5, FLUX, Wan2.2)

These runners handle complex multi-stage pipelines including text encoding, Transformer denoising, and VAE decoding.

*   **SDXL Runners**: Implementations for Text-to-Image (`TTSDXLGenerateRunnerTrace`), Image-to-Image, and Inpainting [tt-media-server/tt_model_runners/runner_fabric.py 11-21](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L11-L21) They utilize `TtSDXLPipeline`[tt-media-server/tt_model_runners/sdxl_generate_runner_trace.py 33-44](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/sdxl_generate_runner_trace.py#L33-L44) and support LoRA injection via prompt triggers [tt-media-server/tt_model_runners/base_sdxl_runner.py 188-194](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_sdxl_runner.py#L188-L194)
*   **DiT Runners**: A specialized hierarchy (`TTDiTRunner`) for Diffusion Transformers like FLUX.1, SD3.5, Mochi-1, and Wan2.2 [tt-media-server/tt_model_runners/dit_runners.py 46-49](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/dit_runners.py#L46-L49) These models often use large device meshes and require significant `ttnn.distribute` timeouts [tt-media-server/tt_model_runners/dit_runners.py 106-114](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/dit_runners.py#L106-L114)
*   **Fabric Configuration**: DiT runners configure `ttnn.set_fabric_config` with reliability modes and router configurations [tt-media-server/tt_model_runners/dit_runners.py 51-73](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/dit_runners.py#L51-L73)

### Audio Runners (Whisper and SpeechT5)

The audio suite includes transcription and text-to-speech runners.

*   **TTWhisperRunner**: Manages the Whisper speech-to-text pipeline [tt-media-server/tt_model_runners/whisper_runner.py 46-50](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/whisper_runner.py#L46-L50) It performs warmup with zero-initialized audio to capture hardware traces [tt-media-server/tt_model_runners/whisper_runner.py 87-100](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/whisper_runner.py#L87-L100) and uses a 2-Command-Queue (2CQ) path for optimized performance [tt-media-server/tt_model_runners/whisper_runner.py 43](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/whisper_runner.py#L43-L43)
*   **TTSpeechT5Runner**: Implements text-to-speech capabilities, registered in the runner fabric [tt-media-server/tt_model_runners/runner_fabric.py 122-124](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/runner_fabric.py#L122-L124)

### Forge Runners (CNNs and Training)

Runners using the `Forge` (torch-xla) backend for CNN architectures and LoRA training.

*   **ForgeRunner**: Base class for `tt-xla` models like ResNet, VoVNet, and MobileNetV2 [tt-media-server/tt_model_runners/forge_runners/forge_runner.py 30-36](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_runners/forge_runner.py#L30-L36) It handles `torch_xla` device setup and model compilation [tt-media-server/tt_model_runners/forge_runners/forge_runner.py 60-73](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_runners/forge_runner.py#L60-L73)

## Key Features

### Warmup and Trace Management

To avoid latency spikes during first-run, runners implement a `warmup` phase [tt-media-server/tt_model_runners/base_device_runner.py 25-27](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_device_runner.py#L25-L27)

1.   **Weights Loading**: Models are loaded into host memory and then distributed to the TT device mesh [tt-media-server/tt_model_runners/dit_runners.py 103-114](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/dit_runners.py#L103-L114)
2.   **Trace Capture**: For models like SDXL or Whisper, a dummy inference is run to record the command sequence into a hardware trace [tt-media-server/tt_model_runners/whisper_runner.py 87-100](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/whisper_runner.py#L87-L100)
3.   **Memory Configuration**: L1 small size and trace region sizes are configured per model requirements (e.g., `WHISPER_L1_SMALL_SIZE`) [tt-media-server/tt_model_runners/whisper_runner.py 51-57](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/whisper_runner.py#L51-L57)

### Dynamic LoRA Management

The `BaseSDXLRunner` supports dynamic LoRA switching and state tracking [tt-media-server/tt_model_runners/base_sdxl_runner.py 212-214](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_sdxl_runner.py#L212-L214)

*   **State Tracking**: The runner tracks `_current_lora_path` and `_current_lora_scale`[tt-media-server/tt_model_runners/base_sdxl_runner.py 31-32](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/base_sdxl_runner.py#L31-L32)
