## Purpose and Scope

This page describes the preprocessing step required for NoC traces that involve multi-chip (fabric) communication. When tt-metal captures NoC trace events on multi-device systems, fabric events contain only high-level routing metadata (ethernet channel, hop counts, etc.) rather than complete device-by-device routing paths. The `fabric_post_process.py` utility elaborates these compact fabric descriptors into explicit multi-hop transfer sequences before the trace can be ingested by the simulation engine.

This preprocessing occurs **after** trace collection ([2.1](https://deepwiki.com/tenstorrent/tt-npe/2.1-trace-collection-with-tt-metal-profiler)) but **before** workload ingestion and simulation. For single-chip traces or programmatically-generated workloads, this step is not required. For information about the overall trace analysis workflow, see [2](https://deepwiki.com/tenstorrent/tt-npe/2-noc-trace-analysis-pipeline). For details on how the elaborated traces are converted to `npeWorkload` objects, see [3.4](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads).

## Multi-chip Communication Overview

In multi-chip Tenstorrent systems, NoC traffic can span device boundaries through Ethernet-connected routing cores. When tt-metal profiler captures these fabric events, it records:

*   Source worker core coordinates
*   Destination worker core coordinates
*   Ethernet channel used for the first hop
*   Routing metadata: hop counts, distances, or direction flags depending on fabric topology

The fabric preprocessing step expands this compact representation into an explicit sequence of transfers, where each hop represents:

*   A NoC transfer on a specific device
*   From a source Ethernet core to either a destination worker core (local write) or another Ethernet core (forward)
*   Organized as a **transfer group** with dependency relationships

## The fabric_post_process.py Utility

The `fabric_post_process.py` script is the primary tool for elaborating fabric routing paths. It can be invoked standalone or is automatically called by the trace analysis pipeline.

### Command Line Usage

The script:

1.   Reads all specified trace files
2.   Combines and sorts events by timestamp
3.   Optionally filters desynchronized events (large timestamp jumps)
4.   For each event with a `fabric_send` field, elaborates the complete routing path
5.   Writes combined trace with `fabric_send.path` field populated
6.   Re-sorts by `(device_id, sx, sy, proc, timestamp)` for deterministic ingestion

## Path Elaboration Algorithm

The path elaboration transforms compact fabric routing metadata into explicit per-device transfer sequences. The algorithm differs based on fabric topology configuration.

### 1D Fabric Path Elaboration

For linear and ring topologies (`FABRIC_1D`, `FABRIC_1D_RING`), the trace contains:

*   `eth_chan`: Starting ethernet channel on source device
*   `start_distance`: Number of hops before first local write
*   `range`: Number of consecutive devices to write to

The `find_path_and_destination_1d` method constructs the path:

**Key steps in `add_hops`:**

1.   Iterate from hop 1 to `(start_distance + range_devices - 1)`
2.   For each hop: 
    *   Get next device in current direction via `get_next_device_in_dir`
    *   Look up forward ethernet channel from routing plane
    *   Look up receiver channel on next device (inverse direction)
    *   Create path segment with `segment_start_x/y` and `forward_x/y`
    *   If hop ≥ start_distance, add `local_writes` array with destination worker cores

3.   Update current device and receiver coordinates for next iteration

**Sources:**[tt_npe/py/util/fabric_post_process.py 353-394](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L353-L394)[tt_npe/py/util/fabric_post_process.py 300-351](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L300-L351)

### 2D Fabric Path Elaboration

For mesh and torus topologies (`FABRIC_2D`, `FABRIC_2D_TORUS`), the trace contains:

*   `eth_chan`: Starting ethernet channel
*   `ns_hops`: Number of north/south hops
*   `e_hops`: Number of east hops
*   `w_hops`: Number of west hops
*   `is_mcast`: Boolean indicating multicast routing

The `find_path_and_destination_2d` method handles three cases:

| Routing Type | Hop Pattern | Description |
| --- | --- | --- |
| **Multicast Tree** | N/S trunk + E/W branches | Traverse N/S hops first, at each hop branch E or W to local destinations |
| **Multicast Line** | E-only or W-only | Linear multicast along single axis |
| **Unicast** | Dimension-ordered | N/S hops followed by E/W hops (or vice versa if initial direction is E/W) |

**Sources:**[tt_npe/py/util/fabric_post_process.py 396-482](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L396-L482)

### Fabric Mux Handling

When a source worker core is not directly adjacent to an ethernet router, the trace may include a `fabric_mux` field specifying an intermediate core. The path elaboration inserts an additional hop:

```
Worker Core → Fabric Mux Core → Ethernet Router → [Remote Devices...]
```

This is handled in `add_first_hop` by creating two path segments with appropriate `parent_id` relationships.

**Sources:**[tt_npe/py/util/fabric_post_process.py 249-298](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L249-L298)

### Elaborated Path Format

Each elaborated path is an array of objects with this structure:

| Field | Description |
| --- | --- |
| `device` | Device ID for this segment |
| `noc` | NoC type used (typically `NOC_1` for fabric) |
| `segment_start_x/y` | Source core coordinates |
| `forward_x/y` | Destination core coordinates for forward (optional) |
| `parent_id` | Index of parent transfer in group (-1 for root) |
| `local_writes` | Array of local destination writes at this hop (optional) |

**Sources:**[tt_npe/py/util/fabric_post_process.py 308-347](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L308-L347)[tt_npe/cpp/src/npeWorkloadIngest.cpp 536-638](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L536-L638)

## Integration with Trace Ingestion

After fabric preprocessing, the elaborated trace is ingested by `convertNocTracesToNpeWorkload` in C++. When the function encounters a `fabric_send` field with an elaborated `path`, it creates a **transfer group**.

### Transfer Group Construction

**Key data structures:**

*   `transfer_group_id`: Unique identifier for the entire multi-hop fabric transfer
*   `transfer_group_index`: Sequential index within the group (0, 1, 2, ...)
*   `transfer_group_parent`: Index of the parent transfer this depends on

These fields enable the dependency tracker to correctly sequence transfers across device boundaries.

## Transfer Groups and Dependency Resolution

Transfer groups enable the simulation engine to correctly model multi-hop fabric transfers with proper sequencing and latency accounting.

### Dependency Relationships

Within a transfer group, dependencies are established via the `transfer_group_parent` field:

**Dependency semantics:**

*   A transfer with `parent_id = N` cannot start until transfer N in the same group has completed
*   Transfers with the same parent can execute concurrently (e.g., T2, T3, T4 above)
*   The root transfer (parent_id = -1) can start immediately based on `phase_cycle_offset`

The `npeTransferDependencyTracker` class (see [4.3](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system)) processes these relationships during simulation initialization to construct the dependency graph.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 614-636](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L614-L636)[tt_npe/cpp/include/npeWorkload.hpp 70-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L70-L72)

### Latency Propagation

Latency is added at two points:

1.   **First segment latency**[npeWorkloadIngest.cpp 557-570](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npeWorkloadIngest.cpp#L557-L570):

    *   Added during trace ingestion for the initial worker-to-eth transfer
    *   Based on device model's write latency calculation

2.   **Subsequent segment latencies** (added by simulation engine):

    *   The dependency tracker adds latencies for subsequent hops during dependency generation
    *   This ensures each hop's start time accounts for the previous hop's latency

This staged approach avoids double-counting latencies while ensuring correct temporal sequencing.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 555-570](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L555-L570)

## Validation and Error Handling

The preprocessing pipeline includes multiple validation checks:

| Validation | Location | Error Handling |
| --- | --- | --- |
| Topology file existence | `fabric_post_process.py` | Immediate exit with error |
| Missing required fields | `TopologyGraph.__init__` | Raises `ProcessingError` exception |
| Invalid device IDs | `get_next_device_in_dir` | Raises `ProcessingError` with fabric node context |
| Missing routing planes | `add_hops` | Raises `ProcessingError` with device/direction info |
| Timestamp desynchronization | `process_traces` | Optional filtering with `--remove_desynchronized_events` |
| Transfer group validation | Workload validation | Checks device ID consistency across group |

**Sources:**[tt_npe/py/util/fabric_post_process.py 24-30](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L24-L30)[tt_npe/py/util/fabric_post_process.py 189-247](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L189-L247)[tt_npe/py/util/fabric_post_process.py 324-339](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L324-L339)[tt_npe/cpp/test/test_npe_workload.cpp 160-169](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/test/test_npe_workload.cpp#L160-L169)

## Summary

The fabric preprocessing step is essential for multi-chip trace analysis, transforming compact routing descriptors into explicit transfer sequences. The `fabric_post_process.py` utility, guided by `topology.json` configuration, elaborates routing paths that are then ingested as transfer groups with proper dependency relationships. This enables the simulation engine to accurately model multi-hop latencies and congestion patterns across device boundaries.

For details on how these transfer groups are processed during simulation, see [4.3](https://deepwiki.com/tenstorrent/tt-npe/4.3-dependency-tracking-system).
