## Purpose and Scope

This document covers the Python wheel packaging system for `tt-tools-common`, including the configuration in `pyproject.toml`, setuptools integration, version management, and package data inclusion rules. The build process creates standard Python wheels (`.whl` files) that can be installed via `pip` and published to PyPI.

For Debian package creation, see [Debian Package Build System](https://deepwiki.com/tenstorrent/tt-tools-common/5.1-debian-package-build-system). For the automated release workflow that triggers builds, see [Automated Release Workflow](https://deepwiki.com/tenstorrent/tt-tools-common/5.3-automated-release-workflow). For PyPI publication procedures, see [Publishing to PyPI and TestPyPI](https://deepwiki.com/tenstorrent/tt-tools-common/5.4-publishing-to-pypi-and-testpypi).

## Build System Architecture

The Python package build uses a **PEP 517/518 compliant** build system with `setuptools` as the build backend. The configuration is centralized in `pyproject.toml` with a compatibility shim in `setup.py` for Ubuntu 22.04 packaging tools.

**Sources:**[pyproject.toml 1-84](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L1-L84)[setup.py 1-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/setup.py#L1-L21)

## pyproject.toml Structure

The `pyproject.toml` file contains all package metadata and build configuration. It follows the PEP 621 standard for project metadata.

### Project Metadata

| Field | Value | Purpose |
| --- | --- | --- |
| `name` | `"tt-tools-common"` | Package name on PyPI |
| `version` | `"1.6.0"` | Single source of truth for version |
| `requires-python` | `">=3.10"` | Minimum Python version requirement |
| `description` | `"Common library for Tenstorrent tooling"` | Short package description |
| `readme` | `"README.md"` | Path to long description |
| `license` | `{file = "LICENSE"}` | Apache 2.0 license file |
| `authors` | Sam Bansal | Primary contact |

**Sources:**[pyproject.toml 1-22](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L1-L22)

### Python Version Compatibility

The package requires **Python 3.10 or higher** to leverage modern language features:

*   Pattern matching (match/case statements)
*   Improved type hints (e.g., `X | Y` union syntax)
*   Structural pattern matching in error handling

The classifiers explicitly list supported versions:

*   `Programming Language :: Python :: 3.10`
*   `Programming Language :: Python :: 3.11`
*   `Programming Language :: Python :: 3 :: Only`

**Sources:**[pyproject.toml 3-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L3-L21)

## Version Management

Version management uses a **single source of truth** approach with manual versioning in `pyproject.toml`.

### Version Format

The version follows **semantic versioning** (MAJOR.MINOR.PATCH):

*   `version = "1.6.0"` in [pyproject.toml 4](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L4-L4)

### Version Extraction in setup.py

The `setup.py` compatibility layer reads the version from `pyproject.toml`:

This ensures version consistency when using Ubuntu 22.04 packaging tools that require `setup.py`.

**Sources:**[setup.py 11-17](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/setup.py#L11-L17)

### Version Extraction in CI

The Debian build workflow extracts version components from Git tags using `damienaicheh/extract-version-from-tag-action`:

This populates environment variables:

*   `$MAJOR` - Major version number
*   `$MINOR` - Minor version number
*   `$PATCH` - Patch version number
*   `$NUMBER_OF_COMMITS_SINCE_TAG` - Distance from last tag

These are used by `gbp dch` to generate Debian changelogs with proper versioning.

**Sources:**[.github/workflows/build-debian.yml 80-107](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L80-L107)

## Compatibility Layer: setup.py

The `setup.py` file exists solely for compatibility with Ubuntu 22.04 Debian packaging tools that don't fully support PEP 517.

### Implementation Details

The `setup.py` file is minimal and delegates to `pyproject.toml`:

**Key points:**

*   **Reads version from `pyproject.toml`** to maintain single source of truth
*   **Uses `tomli`** for TOML parsing (Python <3.11 compatibility)
*   **Minimal configuration** - only what's required for legacy tools
*   **Delegates package discovery** to `find_packages()`

**When this is used:**

*   Debian package builds on Ubuntu 22.04 via `gbp buildpackage`
*   Systems with older packaging tools that don't support PEP 517

**When this is NOT used:**

*   Modern `pip install` operations (uses `pyproject.toml` directly)
*   GitHub Actions Python package builds (uses `setuptools.build_meta`)
*   Manual `python -m build` commands (PEP 517 compliant)

**Sources:**[setup.py 1-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/setup.py#L1-L21)

## Build Artifacts

The Python build process creates two types of distribution artifacts:

| Artifact Type | Filename Pattern | Contents | Use Case |
| --- | --- | --- | --- |
| **Wheel** | `tt_tools_common-1.6.0-py3-none-any.whl` | Pre-built package with all files | Fast installation via `pip install` |
| **Source Distribution** | `tt-tools-common-1.6.0.tar.gz` | Raw source code + metadata | Building from source, auditing |

### Wheel Structure

The wheel contains:

```
tt_tools_common-1.6.0.dist-info/
  - METADATA (package metadata)
  - WHEEL (wheel format metadata)
  - RECORD (file checksums)
  - LICENSE
  - top_level.txt

reset_common/
  - __init__.py
  - *.py files

utils_common/
  - __init__.py
  - *.py files

ui_common/
  - __init__.py
  - *.py files
  - common_style.css

data/
  - blackhole/fw_defines.yaml
  - wormhole/fw_defines.yaml
  - grayskull/*.yaml
```

## Summary

The Python package build system for `tt-tools-common` uses:

1.   **`pyproject.toml`** as the single configuration source (PEP 621 compliant)
2.   **`setup.py`** as a compatibility shim for Ubuntu 22.04 tools
3.   **`setuptools.build_meta`** as the PEP 517 build backend
4.   **Manual version management** in `pyproject.toml` (`version = "1.6.0"`)
5.   **Explicit package data rules** to include CSS and YAML files
6.   **Python 3.10+ requirement** for modern language features
7.   **Universal wheel format** (`py3-none-any`) for platform independence

The build outputs are consumed by:

*   **PyPI** for pip installation
*   **GitHub Releases** for artifact storage
*   **TestPyPI** for pre-release validation

For release automation, see [Automated Release Workflow](https://deepwiki.com/tenstorrent/tt-tools-common/5.3-automated-release-workflow). For Debian packaging, see [Debian Package Build System](https://deepwiki.com/tenstorrent/tt-tools-common/5.1-debian-package-build-system).
