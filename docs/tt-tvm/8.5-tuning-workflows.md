## Purpose and Scope

This page documents the end-to-end workflows for tuning neural networks using TVM's auto-tuning systems. It covers task extraction from networks, configuring tuning options, running the tuning process, managing measurement logs, and applying the best schedules for compilation.

For details on the specific auto-tuning systems themselves, see:

*   [AutoTVM](https://deepwiki.com/tenstorrent/tt-tvm/8.1-autotvm) - Template-based auto-tuning
*   [AutoScheduler](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler) - Template-free auto-scheduling
*   [Meta Schedule](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule) - Advanced meta-scheduling infrastructure

For information about the RPC infrastructure used for distributed measurement, see [RPC Infrastructure](https://deepwiki.com/tenstorrent/tt-tvm/8.4-rpc-infrastructure).

## Task Extraction

### Extracting Tasks from Networks

The `auto_scheduler.extract_tasks` function partitions a neural network into small subgraphs, where each subgraph becomes a search task. This function takes a Relay function, parameters, and target as input:

`tasks, task_weights = auto_scheduler.extract_tasks(    mod["main"],  # Relay function    params,        # Model parameters    target         # Compilation target)`
Each `SearchTask` object represents a computational subgraph with:

*   A `workload_key` - unique identifier for the computation pattern
*   A `compute_dag` - computational DAG in tensor expression form
*   Hardware parameters for the target device

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 268-278](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L268-L278)[gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 173-180](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L173-L180)

### Task Weights

Task weights represent the number of times each task's subgraph appears in the network. These weights are critical for the `TaskScheduler` to prioritize tasks:

The task scheduler optimizes the weighted sum of latencies: `sum(latency[t] * weight[t])`, which approximates the end-to-end network execution time.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 258-266](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L258-L266)[gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 165-171](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L165-L171)

### Builders

Builders compile generated code for measurement. Key implementations:

| Builder Class | Purpose | Use Case |
| --- | --- | --- |
| `LocalBuilder` | Compile on local machine | Same architecture as host |
| `LocalBuilder(build_func="ndk")` | Cross-compile using NDK | Android ARM targets |
| `LocalBuilder(build_func="default")` | Default compiler | Linux ARM targets |

### Runners

Runners execute compiled code and measure performance:

| Runner Class | Connection | Use Case |
| --- | --- | --- |
| `auto_scheduler.RPCRunner` | Remote device via RPC | Cross-platform tuning, distributed measurement |
| `auto_scheduler.LocalRunner` | Local execution | Same-platform tuning |

**RPCRunner Configuration:**

`runner = auto_scheduler.RPCRunner(    device_key,              # Device identifier in RPC tracker    host=rpc_host,           # RPC tracker host    port=rpc_port,           # RPC tracker port (default 9190)    timeout=30,              # Maximum execution time per measurement    repeat=1,                # Number of measurement repeats    min_repeat_ms=200,       # Minimum duration for measurements    enable_cpu_cache_flush=True  # Clear cache for accurate timing)`

### Measurement Callbacks

The `measure_callbacks` parameter accepts a list of callback objects that process measurement results. The most important is `RecordToFile`:

`measure_callbacks=[auto_scheduler.RecordToFile(log_file)]`
This callback saves all measurement records to a JSON file for later analysis and reuse.

## Running the Tuning Process

### TaskScheduler

The `TaskScheduler` manages the tuning of multiple tasks, dynamically allocating measurement trials based on predicted impact on end-to-end performance:

**Usage:**

`tuner = auto_scheduler.TaskScheduler(tasks, task_weights)tuner.tune(tune_option)`
**Resuming from Previous Logs:**

`tuner = auto_scheduler.TaskScheduler(    tasks,     task_weights,     load_log_file=log_file  # Resume from existing log)`
Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 303-321](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L303-L321)[gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 229-240](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L229-L240)

### Understanding Tuning Output

During tuning, the `TaskScheduler` prints a status table:

```
----------------------------------------------------------------------
------------------------------  [ Task Scheduler ]
----------------------------------------------------------------------
|  ID  | Latency (ms) | Speed (GFLOPS) | Trials |
-------------------------------------------------
|    0 |        0.013 |           0.31 |     64 |
|    1 |        0.845 |           2.43 |    448 |
|    2 |        0.046 |          -0.00 |     64 |
|    3 |        4.194 |          24.53 |   2112 |
...
-------------------------------------------------
Estimated total latency: 38.347 ms    Trials: 19992   Used time: 19260 s    Next ID: 3
```

**Key Information:**

| Column | Meaning |
| --- | --- |
| ID | Task index |
| Latency (ms) | Best measured latency for this task |
| Speed (GFLOPS) | Computational throughput |
| Trials | Number of measurement trials allocated to this task |
| Estimated total latency | Weighted sum of task latencies (approximates network latency) |
| Next ID | Which task will be tuned next |

## Log Management

### Log File Format

Measurement records are saved in JSON format by `RecordToFile`. Each record contains:

*   Input configuration (schedule decisions)
*   Measured performance
*   Task workload key
*   Target hardware information

**Log File Location:**

`log_file = "%s-%s-B%d-%s.json" % (    network,           # e.g., "mobilenet"    layout,            # e.g., "NHWC"    batch_size,        # e.g., 1    target.kind.name   # e.g., "llvm")`
Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 255](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L255-L255)[gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 148](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L148-L148)

### Distilling Logs

Large log files can be reduced to only the best records:

`python3 -m tvm.auto_scheduler.measure_record --mode distill -i log.json`
This command filters the log file to retain only the best-performing schedule for each task, significantly reducing file size while preserving all useful information.

## Applying Tuned Schedules

### ApplyHistoryBest

After tuning, the best schedules are applied during compilation using `ApplyHistoryBest`:

`with auto_scheduler.ApplyHistoryBest(log_file):    with tvm.transform.PassContext(        opt_level=3,         config={"relay.backend.use_auto_scheduler": True}    ):        lib = relay.build(mod, target=target, params=params)`
**Key Points:**

*   `ApplyHistoryBest` is a context manager that loads the best schedule for each task from the log file
*   The config `"relay.backend.use_auto_scheduler": True` enables auto-scheduler integration in the compilation pipeline
*   If no schedule is found for a task, the auto-scheduler falls back to default schedules

## Complete Workflow Examples

### ARM CPU Tuning Workflow

The following table shows a complete tuning workflow for ARM CPU:

| Step | Code Entity | Purpose |
| --- | --- | --- |
| 1. Load Network | `relay.testing.mobilenet.get_workload()` | Get pre-defined network or load from framework |
| 2. Set Target | `tvm.target.Target("llvm -mtriple=aarch64-linux-gnu")` | Define compilation target |
| 3. Extract Tasks | `auto_scheduler.extract_tasks(mod["main"], params, target)` | Partition network into tasks |
| 4. Configure Builder | `LocalBuilder(build_func="ndk")` | Cross-compilation for Android |
| 5. Configure Runner | `RPCRunner(device_key, host, port, ...)` | Remote execution on ARM device |
| 6. Create Scheduler | `TaskScheduler(tasks, task_weights)` | Initialize task scheduler |
| 7. Run Tuning | `tuner.tune(tune_option)` | Execute tuning process |
| 8. Apply Results | `ApplyHistoryBest(log_file)` | Load best schedules |
| 9. Compile | `relay.build(mod, target, params)` | Build final library |
| 10. Deploy | `remote.load_module(filename)` | Load on target device |

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 269-357](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L269-L357)

### GPU Tuning Workflow

For GPU targets like Mali, the workflow is similar with target-specific configuration:

**Target Configuration:**

`target = tvm.target.Target(    "opencl -device=mali",           # GPU backend    host="llvm -mtriple=aarch64-linux-gnu"  # CPU host)`
**Hardware Parameters:**

GPU-specific hardware parameters can be queried and passed to task extraction:

`from tvm.auto_scheduler.utils import request_remote remote = request_remote(device_key, host, port)dev = remote.cl() hardware_params = auto_scheduler.HardwareParams(    -1,  # Not used    16,  # Vector width    64,  # Cache line size    dev.max_shared_memory_per_block,    2147483647,  # INT32_MAX for local memory (no explicit limit)    dev.max_threads_per_block,    int(dev.warp_size / 4) if int(dev.warp_size / 4) > 1 else dev.warp_size,    dev.warp_size) tasks, task_weights = auto_scheduler.extract_tasks(    mod["main"], params, target, hardware_params=hardware_params)`

## Best Practices

### Determining Number of Trials

The recommended number of measurement trials is approximately:

```
num_measure_trials ≈ 800 × len(tasks)
```

This is typically sufficient for the search to converge. For example:

*   ResNet-50 has ~29 tasks → 20,000 trials recommended
*   MobileNet has fewer tasks → fewer trials needed

For quick demonstration runs, 200 trials can show basic functionality.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 286-291](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L286-L291)[gallery/how_to/tune_with_autoscheduler/tune_network_mali.py 214-219](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_mali.py#L214-L219)

### Parallel Measurement

To accelerate tuning, register multiple devices to the RPC tracker:

All devices with the same `device_key` form a pool for parallel measurements.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 439-443](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L439-L443)

### Early Termination

Tuning can be terminated early by killing the process. As long as at least one valid schedule exists for each task in the log file, compilation can proceed. The `ApplyHistoryBest` context manager will use the best available schedule for each task.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 420-425](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L420-L425)

### CPU-Intensive Operations

The auto-scheduler performs CPU-intensive operations during tuning:

*   Compiling many candidate programs
*   Extracting features from programs
*   Search space exploration

A high-performance CPU with many cores is recommended for faster search, especially when using `LocalBuilder`.

Sources: [gallery/how_to/tune_with_autoscheduler/tune_network_arm.py 430-432](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/gallery/how_to/tune_with_autoscheduler/tune_network_arm.py#L430-L432)

### Error Handling

During tuning, you may see error messages (e.g., "dmlc::Error" or "tvm::Error") because the auto-scheduler tries invalid schedules. These errors can be safely ignored as long as tuning continues - they are isolated from the main process.
