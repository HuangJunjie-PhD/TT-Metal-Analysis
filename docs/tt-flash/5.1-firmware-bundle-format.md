# Firmware Bundle Format

Relevant source files
*   [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py)
*   [tt_flash/main.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py)

This document describes the structure and format of `.fwbundle` files used by tt-flash to package firmware images for Tenstorrent AI chips. The firmware bundle is a tarfile containing a manifest, board-specific firmware images, and parameter replacement specifications.

For information about how these bundles are processed during flashing operations, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages). For details on parameter handling and tag functions, see [Parameter Handling and Masks](https://deepwiki.com/tenstorrent/tt-flash/5.4-parameter-handling-and-masks).

* * *

## Bundle Structure Overview

A `.fwbundle` file is a standard tarfile (`.tar` archive) containing firmware images and metadata organized in a specific directory structure. The bundle supports multiple board types in a single package, with each board type having its own subdirectory.

**Sources:**[tt_flash/flash.py 336-366](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L336-L366)[tt_flash/main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L201)

* * *

## Manifest Schema

The `manifest.json` file at the root of the tarfile contains metadata about the bundle format version and the firmware version being distributed.

### Manifest Fields

| Field | Type | Description |
| --- | --- | --- |
| `version` | String | Manifest format version (e.g., "2.0.0") |
| `bundle_version` | Object | Semantic firmware version information |
| `bundle_version.fwId` | Integer | Major firmware identifier |
| `bundle_version.releaseId` | Integer | Release identifier |
| `bundle_version.patch` | Integer | Patch level |
| `bundle_version.debug` | Integer | Debug identifier |

### Example Manifest

`{  "version": "2.0.0",  "bundle_version": {    "fwId": 19,    "releaseId": 2,    "patch": 1,    "debug": 0  }}`
### Version Parsing

The manifest version is parsed as a 3-tuple of integers at [tt_flash/main.py 187-199](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L187-L199) The bundle version is converted to a 4-tuple at [tt_flash/flash.py 676-681](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L676-L681):

`manifest_bundle_version = manifest.get("bundle_version", {})new_bundle_version = (    manifest_bundle_version.get("fwId", 0),    manifest_bundle_version.get("releaseId", 0),    manifest_bundle_version.get("patch", 0),    manifest_bundle_version.get("debug", 0),)`
**Sources:**[tt_flash/flash.py 663-698](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L663-L698)[tt_flash/main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L201)

## Bundle Version Validation

Bundle versions must meet specific compatibility requirements based on the manifest format version. Validation occurs in the `verify_package` function.

The `BUNDLE_VALIDATION_FUNCS` dictionary at [tt_flash/flash.py 658-661](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L658-L661) maps manifest versions to validation functions:

`BUNDLE_VALIDATION_FUNCS = {    (2, 0, 0): lambda bundle_version: bundle_version[0] >= 19,}`
For manifest version 2.0.0, the firmware `fwId` (major version) must be 19 or newer.

**Sources:**[tt_flash/flash.py 658-698](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L658-L698)

## Board-Specific Directory Structure

Each supported board type has its own subdirectory within the tarfile, named using the board's internal codename (not the public display name).

### Board Directory Naming

Board directories use the internal codename format from `utility.get_board_type()`:

*   `NEBULA_X1` (not "n150")
*   `NEBULA_X2` (not "n300")
*   `WH_UBB` (not "Galaxy Wormhole")
*   `E300_R2`, `E300_R3` (not "e150")
*   `P300a_left`, `P300a_right` (for dual-chip P300 boards)

The P300 series requires special handling at [tt_flash/flash.py 831-836](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L831-L836) to distinguish left and right chips:

`if "P300" in boardname:    # 0 = Right, 1 = Left    if dev.get_asic_location() == 0:        boardname = f"{boardname}_right"    elif dev.get_asic_location() == 1:        boardname = f"{boardname}_left"`
### Required Files

Each board directory must contain:

*   `image.bin` - Firmware image data in hex format
*   `mask.json` - Parameter replacement specifications

**Sources:**[tt_flash/flash.py 336-366](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L336-L366)[tt_flash/flash.py 823-838](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L823-L838)[tt_flash/utility.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py)

## image.bin Format

The `image.bin` file contains firmware binary data encoded in a text-based hexadecimal format with address markers.

### Format Specification

The image format consists of:

1.   **Address lines** - Start with `@` followed by a hexadecimal address
2.   **Data lines** - Base16 (hexadecimal) encoded binary data

### Example Format

```
@0
466C617368496D616765446174610A
@100
48656C6C6F576F726C64
@200
4669726D7761726542696E617279
```

### Parsing Logic

The image is parsed line-by-line at [tt_flash/flash.py 428-454](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L428-L454):

`curr_addr = 0for line in image.decode("utf-8").splitlines():    line = line.strip()    if line.startswith("@"):        curr_addr = int(line.lstrip("@").strip())    else:        data = b16decode(line)        curr_stop = curr_addr + len(data)        # ... parameter handling ...        writes.append(FlashWrite(curr_addr, data))        curr_addr = curr_stop`
The parser:

1.   Maintains current address in `curr_addr`
2.   When encountering `@`, updates `curr_addr` to the new address
3.   When encountering data, decodes hex and creates a `FlashWrite` object
4.   Advances `curr_addr` by the length of decoded data

### Blackhole Image Format

Blackhole chips use the same format but follow a different processing path at [tt_flash/flash.py 374-393](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L374-L393) that creates `FlashWrite` objects before parameter handling via `boot_fs_write()`.

**Sources:**[tt_flash/flash.py 372-454](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L372-L454)

## mask.json Format

The `mask.json` file specifies parameter replacement regions within the firmware image. These parameters are dynamically calculated or read from the chip during the flashing process.

### Mask Entry Schema

Each entry in the mask array is a JSON object with three fields:

| Field | Type | Description |
| --- | --- | --- |
| `start` | Integer | Starting byte offset in SPI flash |
| `end` | Integer | Ending byte offset (exclusive) |
| `tag` | String | Parameter handler identifier |

### Example Mask

`[  {    "start": 256,    "end": 260,    "tag": "rmw"  },  {    "start": 512,    "end": 516,    "tag": "incr"  },  {    "start": 1024,    "end": 1028,    "tag": "date"  },  {    "start": 2048,    "end": 2052,    "tag": "flash_version"  },  {    "start": 4096,    "end": 4100,    "tag": "bundle_version"  }]`
### Validation

The mask format is validated at [tt_flash/flash.py 397-424](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L397-L424):

`for v in mask:    start = v.get("start", None)    end = v.get("end", None)    tag = v.get("tag", None)        if ((start is None or not isinstance(start, int))        or (end is None or not isinstance(end, int))        or (tag is None or not isinstance(tag, str))):        raise TTError(            f"Invalid mask format for {boardname_to_display}; "            "expected to see a list of dicts with keys 'start', 'end', 'tag'"        )`
**Sources:**[tt_flash/flash.py 368-424](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L368-L424)

## Parameter Handler Tags

The `tag` field in mask entries maps to handler functions defined in the `TAG_HANDLERS` dictionary at [tt_flash/flash.py 142-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L142-L148)

### Available Handlers

| Tag | Function | Description |
| --- | --- | --- |
| `rmw` | `rmw_param()` | Read-Modify-Write: preserves existing SPI data |
| `incr` | `incr_param()` | Increments existing SPI value by 1 |
| `date` | `date_param()` | Writes current date in `0xYYYYMMDD` format |
| `flash_version` | `flash_version()` | Writes tt-flash tool version (4 bytes) |
| `bundle_version` | `bundle_version()` | Writes firmware bundle version (4 bytes) |

### Handler Function Signature

All handlers follow the same signature defined at [tt_flash/flash.py 33-43](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L43):

`def handler_function(    chip: TTChip,    data: bytearray,    spi_addr: int,    data_addr: int,    len: int) -> bytearray:    # Modify data[data_addr:data_addr+len]    # May read from chip.spi_read(spi_addr, len)    return data`
Parameters:

*   `chip` - The `TTChip` instance for reading existing SPI data
*   `data` - The firmware image data being prepared
*   `spi_addr` - Absolute address in SPI flash
*   `data_addr` - Offset within the current `data` chunk
*   `len` - Number of bytes to process

**Sources:**[tt_flash/flash.py 33-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L148)

## Bundle Loading Process

The firmware bundle is loaded and validated through a multi-step process before flashing begins.

### Loading Sequence

1.   **Initial Load** at [tt_flash/main.py 215](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L215-L215):

    *   Opens tarfile in read mode
    *   Extracts and parses `manifest.json`
    *   Returns tar handle and manifest version tuple

2.   **Package Verification** at [tt_flash/flash.py 806](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L806-L806):

    *   Re-reads `manifest.json` from tar
    *   Parses `bundle_version` object
    *   Validates against version requirements
    *   Sets global `__SEMANTIC_BUNDLE_VERSION` for later use by `bundle_version()` handler

3.   **Board File Extraction** at [tt_flash/flash.py 336-366](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L336-L366):

    *   For each detected chip, extracts `{boardname}/image.bin`
    *   Extracts corresponding `{boardname}/mask.json`
    *   Raises `TTError` if files are missing (unless `--skip-missing-fw` is used)

### Error Handling

Missing manifest:

`if manifest_data is None:    raise TTError(        "Could not find manifest in fw package, "        "please check that the correct one was used."    )`
Missing board files at [tt_flash/flash.py 347-358](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L347-L358):

`if image is None and mask is None:    if skip_missing_fw:        print(f"Could not find flash data for {boardname_to_display} in tarfile")        return FlashStageResult(state=FlashStageResultState.NoFlash, ...)    else:        raise TTError(            f"Could not find flash data for {boardname_to_display} in tarfile"        )`
**Sources:**[tt_flash/main.py 179-234](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L234)[tt_flash/flash.py 663-698](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L663-L698)[tt_flash/flash.py 336-366](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L336-L366)

## Global State: __SEMANTIC_BUNDLE_VERSION

The firmware bundle version is stored in a global variable at [tt_flash/flash.py 102](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L102-L102):

`__SEMANTIC_BUNDLE_VERSION = [0xFF, 0xFF, 0xFF, 0xFF]`
This global is set during `verify_package()` at [tt_flash/flash.py 694-695](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L694-L695) and consumed by the `bundle_version()` handler at [tt_flash/flash.py 105-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L105-L124):

`def bundle_version(    chip: TTChip, data: bytearray, spi_addr: int, data_addr: int, len: int) -> bytearray:    global __SEMANTIC_BUNDLE_VERSION        for _ in range(__SEMANTIC_BUNDLE_VERSION.__len__(), 4):        __SEMANTIC_BUNDLE_VERSION.append(0)    version_parts = __SEMANTIC_BUNDLE_VERSION[:4]        version = [        int(version_parts[3]),        int(version_parts[2]),        int(version_parts[1]),        int(version_parts[0]),    ]        data[data_addr : data_addr + len] = bytes(version)    return data`
The version is stored in little-endian byte order when written to the firmware image.

**Sources:**[tt_flash/flash.py 100-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L100-L124)[tt_flash/flash.py 694-695](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L694-L695)

## Architecture-Specific Processing

### Wormhole Processing

Wormhole chips process mask parameters during image parsing at [tt_flash/flash.py 396-456](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L396-L456) For each writeable region:

1.   Scan all mask entries to find overlapping regions
2.   Call corresponding handler function for each overlap
3.   Validate that parameters don't split writeable regions
4.   Create `FlashWrite` objects with processed data

### Blackhole Processing

Blackhole chips follow a different path at [tt_flash/flash.py 374-393](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L374-L393):

1.   Parse entire image into `FlashWrite` objects first
2.   Pass to `boot_fs_write()` function from `tt_flash.blackhole`
3.   `boot_fs_write()` handles mask parameter processing internally

The separation exists because Blackhole chips require special boot filesystem handling.

**Sources:**[tt_flash/flash.py 374-456](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L374-L456)[tt_flash/blackhole.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py)
