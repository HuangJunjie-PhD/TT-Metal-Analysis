The `tt-emule` Python pytest suite provides a curated regression path for the `ttnn` high-level API. It targets single-device Wormhole (N150) and Blackhole (P100) architectures, executing a subset of the upstream `tt-metal` unit tests that have been verified to pass under emulation. The suite is designed for high-throughput CI execution via round-robin sharding and optimized process management.

## Test Runner Architecture

The suite is driven by two primary architecture-specific scripts: `scripts/run_ttnn_pytests_wormhole.sh` and `scripts/run_ttnn_pytests_blackhole.sh`. These scripts wrap the standard `pytest` binary and inject the necessary environment variables to redirect `tt-metal` to the emulation backend.

### Environment Configuration

The runners establish a specific execution environment to ensure `ttnn` resolves to the emulated build:

*   `TT_METAL_EMULE_MODE=1`: Activates the emulation backend in the `tt-metal` library [scripts/run_ttnn_pytests_blackhole.sh 87](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L87-L87)
*   `TT_METAL_SLOW_DISPATCH_MODE=1`: Forces slow dispatch, as the emulation layer does not currently support Fast Dispatch Command Queues [scripts/run_ttnn_pytests_blackhole.sh 88](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L88-L88)
*   `TT_METAL_MOCK_CLUSTER_DESC_PATH`: Points to the specific YAML cluster descriptor (e.g., `wormhole_N150.yaml` or `blackhole_P100.yaml`) [scripts/run_ttnn_pytests_wormhole.sh 86](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_wormhole.sh#L86-L86)
*   `PYTHONPATH`: Prepended with the build directory to ensure `import ttnn` loads the emulated `_ttnn.so`[scripts/run_ttnn_pytests_blackhole.sh 82](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L82-L82)

### Sharding Strategy

To minimize total execution time in CI, the suite implements a **round-robin sharding strategy**. Unlike contiguous block sharding, round-robin ensures that computationally expensive tests (e.g., large `reduce` or `matmul` operations) are distributed evenly across all CI workers.

*   **SHARD_INDEX**: 1-based index of the current worker [scripts/run_ttnn_pytests_blackhole.sh 45](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L45-L45)
*   **SHARD_COUNT**: Total number of parallel workers [scripts/run_ttnn_pytests_blackhole.sh 46](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L46-L46)
*   **Logic**: An internal `ENTRY_NUM` counter is incremented for every `run_pytest` call. An entry is only executed if `((ENTRY_NUM - 1) % SHARD_COUNT + 1) == SHARD_INDEX`[scripts/run_ttnn_pytests_blackhole.sh 70](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L70-L70)

### Process Isolation and Performance

The runner balances isolation against the high overhead of initializing `ttnn` (which can take tens of seconds per process).

*   **Default (FORKED=0)**: Multiple test nodes from the same file are grouped into a single `pytest` invocation. This amortizes the cost of `import ttnn` and device initialization [scripts/run_ttnn_pytests_blackhole.sh 64-67](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L64-L67)
*   **Debug Mode (FORKED=1)**: Uses the `--forked` pytest plugin to isolate every single test case in its own process. This is used primarily for debugging memory corruption or state bleed [scripts/run_ttnn_pytests_blackhole.sh 101](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L101-L101)

## Filter Patterns and Regression Guards

The suite uses a "curated list" approach. Instead of running all tests and filtering failures, it explicitly lists passing targets. This prevents new, unverified tests from breaking the CI signal.

### Filtering Techniques

1.   **Whole Files**: Used when every test in a file passes [scripts/run_ttnn_pytests_blackhole.sh 123](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L123-L123)
2.   **Specific Nodes**: Targets individual functions within a file to skip failing siblings [scripts/run_ttnn_pytests_blackhole.sh 132](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L132-L132)
3.   **Deselection**: Uses `--deselect` to exclude specific parameterizations of a test that are known to fail due to hardware limits or emulation constraints [scripts/run_ttnn_pytests_blackhole.sh 128-129](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L128-L129)

### Regression Guard Example: `MEM_ZEROS`

A critical hardware contract emulated is the `MEM_ZEROS` region—a 512-byte L1 area that must remain zero. Kernels often use NOC reads from this region for cheap zero-filling [docs/mem-zeros-handling.md 3-5](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L3-L5) The JIT shim `include/jit_hw/dev_mem_map.h` ensures that `MEM_ZEROS_BASE` is correctly defined per architecture:

*   **Wormhole**: `0x3280`[include/jit_hw/dev_mem_map.h 42](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L42-L42)
*   **Blackhole**: `0x32E0`[include/jit_hw/dev_mem_map.h 38](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L38-L38)

The suite includes tests to ensure that no emulation logic (like the DFB bump allocator) clobbers this reserved memory [docs/mem-zeros-handling.md 147-150](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L147-L150)

**Sources:**[scripts/run_ttnn_pytests_blackhole.sh 122-136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/run_ttnn_pytests_blackhole.sh#L122-L136)[include/jit_hw/dev_mem_map.h 36-44](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/dev_mem_map.h#L36-L44)[docs/mem-zeros-handling.md 1-51](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/mem-zeros-handling.md?plain=1#L1-L51)

## CI Wrapper: `ci-ttnn-pytests.sh`

The `.github/scripts/ci-ttnn-pytests.sh` script serves as the entry point for GitHub Actions. It handles environment setup that is typically manual for developers.

### Key Responsibilities

*   **Architecture Selection**: Validates `TT_EMULE_ARCH` is either `wormhole` or `blackhole`[ci-ttnn-pytests.sh 24-27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/ci-ttnn-pytests.sh#L24-L27)
*   **Symlink Management**: Automatically creates the `_ttnn.so` symlink in the `tt-metal` source tree so that Python can find the emulated binary [ci-ttnn-pytests.sh 44-46](https://github.com/tenstorrent/tt-emule/blob/a57cf072/ci-ttnn-pytests.sh#L44-L46)
*   **Result Aggregation**: Captures stdout/stderr to a log file and ensures JUnit XMLs are generated in the `GTEST_XML_DIR` for CI reporting [ci-ttnn-pytests.sh 59-63](https://github.com/tenstorrent/tt-emule/blob/a57cf072/ci-ttnn-pytests.sh#L59-L63)
