The C++ regression suite provides a multi-tiered validation framework for `tt-emule`. It spans from host-only utility tests to full-chip emulation of Wormhole, Blackhole, and Quasar architectures. The suite is designed to catch regressions in memory safety, JIT kernel compilation, and numerical accuracy (PCC) across different hardware targets.

## Regression Architecture

The regression suite is organized into **Tiers**, where each tier increases in complexity and hardware dependency.

### Tiered Execution Model

| Tier | Focus | Scope |
| --- | --- | --- |
| **Tier 1** | **Host-only** | Tests bit utilities, `CoreRange` logic, and tile-layout math. No device initialization. |
| **Tier 2** | **Buffer I/O** | Validates `L1` and `DRAM` buffer allocation and host R/W via `SWEmuleChip`. |
| **Tier 3** | **JIT Kernels** | Full execution of Tensix kernels on Wormhole (N150) mock device. |
| **Tier 3k/l** | **Data Movement** | Stress tests NOC packet sizes, direct writes, and DRAM unary ops. |
| **Tier 4** | **Blackhole** | Architecture-specific tests for Blackhole (P100), including INT32 support. |
| **Tier 5/6** | **ttnn / PCC** | High-level `ttnn` operations (Matmul, Reduction) with Bfloat16/FP32 precision checks. |

## Implementation Details

### Circular Buffer Pointer Validation (`test_cb_ptr`)

A critical regression test `test_cb_ptr.cpp` exists to prevent race conditions in the per-RISC circular buffer model (Issue #139). It exercises the `jit_hw` headers directly by mocking the `thread_local` state.

*   **Per-RISC Isolation**: Ensures that if a writer RISC advances a pointer, the reader RISC's local view remains consistent [tests/cb_ptr/test_cb_ptr.cpp 77-107](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tests/cb_ptr/test_cb_ptr.cpp#L77-L107)
*   **Write-Back**: Verifies that manual overrides to `fifo_rd_ptr` (used in silicon-style kernel optimizations) are correctly reflected in subsequent `__emule_cb_rd_addr` calls [tests/cb_ptr/test_cb_ptr.cpp 109-118](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tests/cb_ptr/test_cb_ptr.cpp#L109-L118)
*   **Pointer Encoding**: Uses `MAP_32BIT` to ensure 16-byte aligned pointers can be safely shifted/encoded into `uint32_t` for kernel consumption [tests/cb_ptr/test_cb_ptr.cpp 60-69](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tests/cb_ptr/test_cb_ptr.cpp#L60-L69)

### Emulation Toggle Verification

The `run_toggle_test.sh` script proves that a single binary can switch between Silicon and Emulation at runtime using the `TT_METAL_EMULE_MODE` environment variable.

*   **Phase 1 (Silicon)**: Unsets emulation variables; tests confirm `TargetDevice::Silicon`[tests/integration/test_emulation_toggle.cpp 72-80](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tests/integration/test_emulation_toggle.cpp#L72-L80)
*   **Phase 2 (Emulated)**: Sets `TT_METAL_EMULE_MODE=1`; tests confirm `TargetDevice::Emule` and verify that `SlowDispatch` is forced [tests/integration/test_emulation_toggle.cpp 94-109](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tests/integration/test_emulation_toggle.cpp#L94-L109)

### Result Classification Logic

The `classify-results.py` script manages the "Known Failures" allowlist. It performs three distinct checks:

1.   **New Failures**: Tests that failed but are not in the allowlist (Strict Error).
2.   **Newly Passing**: Tests in the allowlist that now pass (Warning/Error to keep allowlist clean) [.github/scripts/classify-results.py 173-183](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/scripts/classify-results.py#L173-L183)
3.   **Stale Entries**: Patterns in the allowlist that matched no tests in the current run [.github/scripts/classify-results.py 81-91](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/scripts/classify-results.py#L81-L91)
