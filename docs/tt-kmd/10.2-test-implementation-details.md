## Purpose and Scope

This document provides detailed technical information about implementing tests in the `ttkmd_test` suite. It covers test framework architecture, common utilities, validation strategies, and step-by-step guidance for adding new tests.

For an overview of the test suite and how to run tests, see [User-Space Test Suite](https://deepwiki.com/tenstorrent/tt-kmd/10.1-user-space-test-suite). For information about the driver's IOCTL interface being tested, see [IOCTL Command Reference](https://deepwiki.com/tenstorrent/tt-kmd/5.2-ioctl-command-reference).

## Test Framework Architecture

The test suite is structured as a single executable that enumerates devices and executes a sequence of validation tests on each device.

### Test Execution Flow

**Sources:**[test/main.cpp 28-73](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/main.cpp#L28-L73)

### Test Function Signature

All test functions follow a consistent signature pattern:

`void TestFunctionName(const EnumeratedDevice &dev);`
For peer-to-peer tests that require two devices:

`void TestMapPeerBar(const EnumeratedDevice &dev1, const EnumeratedDevice &dev2);`
The `EnumeratedDevice` structure ([enumeration.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumeration.h)) contains:

*   `path`: Device path (e.g., `/dev/tenstorrent/0`)
*   `location`: PCI bus/device/function coordinates
*   `type`: Device type (`Wormhole` or `Blackhole`)

**Sources:**[test/main.cpp 11-26](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/main.cpp#L11-L26)[test/enumeration.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/enumeration.h)

## Core Test Utilities

The test suite provides several RAII wrapper classes and utility functions to simplify test implementation.

### Test Infrastructure Classes

**Sources:**[test/enumeration.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/enumeration.h)[test/devfd.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/devfd.h)[test/tlbs.h 19-66](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L19-L66)[test/tlbs.h 68-107](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L68-L107)

### DevFd: Device File Descriptor Wrapper

The `DevFd` class provides RAII management of device file descriptors:

`DevFd dev_fd(dev.path);  // Opens /dev/tenstorrent/N// File descriptor automatically closed when dev_fd goes out of scopeint fd = dev_fd.get();   // Access underlying fd for ioctl() calls`
**Key Features:**

*   Automatic `open()` on construction [test/devfd.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/devfd.cpp)
*   Automatic `close()` on destruction [test/devfd.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/devfd.cpp)
*   Throws `test_failure` if open fails
*   Prevents fd leaks in test code

**Sources:**[test/devfd.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/devfd.cpp)[test/devfd.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/devfd.h)

### TLB Window Helpers

The `TlbHandle` and `TlbWindow` template classes encapsulate TLB allocation, configuration, and mapping. `TlbHandle` manages the lifecycle of a specific TLB ID [test/tlbs.h 19-66](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L19-L66) while `TlbWindow` provides a higher-level interface for offset-based NOC access [test/tlbs.h 68-107](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L68-L107)

`// Allocate, configure, and map a 2MB TLB windowTlbWindow2M tlb(dev_fd.get(), x_coord, y_coord, noc_address); // Write to NOC via TLBtlb.write32(offset, 0xDEADBEEF); // Read from NOC via TLBuint32_t value = tlb.read32(offset); // TLB automatically freed and unmapped when tlb goes out of scope`
**Implementation Details:**

*   `TlbHandle` issues `TENSTORRENT_IOCTL_ALLOCATE_TLB`[test/tlbs.h 33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L33-L33)
*   `TlbHandle` issues `TENSTORRENT_IOCTL_CONFIGURE_TLB`[test/tlbs.h 41](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L41-L41)
*   `TlbHandle` calls `mmap()` to map the TLB using the returned `mmap_offset_uc`[test/tlbs.h 44](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L44-L44)
*   Destructor issues `TENSTORRENT_IOCTL_FREE_TLB` and `munmap()`[test/tlbs.h 58-65](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h#L58-L65)

**Sources:**[test/tlbs.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.h)

## Specific Test Implementations

### DMA Buffer Tests

The `TestDmaBuf` function validates the allocation and mapping of DMA buffers [test/dma_buf.cpp 133-169](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L133-L169)

*   **Max Buffer Size**: Queries device info to determine `max_dma_buf_size_log2`[test/dma_buf.cpp 31-34](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L31-L34)
*   **Allocation Limits**: Verifies that the driver rejects more than `TENSTORRENT_MAX_DMA_BUFS`[test/dma_buf.cpp 156-164](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L156-L164)
*   **Mapping Verification**: `VerifyBufferMapping` maps the buffer, writes a pattern, and reads it back to ensure data integrity [test/dma_buf.cpp 91-116](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L91-L116)
*   **Duplicate Indices**: Ensures that requesting the same `buf_index` twice fails with `EINVAL`[test/dma_buf.cpp 145-150](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L145-L150)

**Sources:**[test/dma_buf.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp)

### TLB Quantity and Access Tests

TLB tests verify the allocation limits and data path for different window sizes.

*   **Quantity Validation**: `VerifyTlbQuantitiesWormhole` attempts to exhaust the TLB pool, ensuring 156x 1M, 10x 2M, and 19x 16M windows can be allocated [test/tlbs.cpp 100-151](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.cpp#L100-L151)
*   **Off-limits Region**: Verifies that the final 16M window (reserved for kernel) cannot be allocated by userspace [test/tlbs.cpp 135-142](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.cpp#L135-L142)
*   **Data Integrity**: `VerifyManyWindowsWormhole` writes data through a 16M window and reads it back through 1M and 2M windows to ensure all mapping types point to the same physical memory [test/tlbs.cpp 188-230](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.cpp#L188-L230)

**Sources:**[test/tlbs.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/tlbs.cpp)

### Page Pinning Tests

The `TestPinPages` suite validates the `TENSTORRENT_IOCTL_PIN_PAGES` interface [test/pin_pages.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp)

*   **Simple Pin**: Verifies pinning a single page with both flags=0 and `TENSTORRENT_PIN_PAGES_CONTIGUOUS`[test/pin_pages.cpp 48-72](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp#L48-L72)
*   **Error Handling**: Validates rejection of `size == 0`, misaligned sizes, and invalid flags [test/pin_pages.cpp 74-134](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp#L74-L134)
*   **Unmapped Ranges**: Ensures the driver fails if requested virtual addresses are not mapped in the process [test/pin_pages.cpp 136-178](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp#L136-L178)
*   **Contiguity**: `VerifyPinPagesContiguous` uses hugepages to verify the driver correctly identifies and accepts contiguous physical memory [test/pin_pages.cpp 207-225](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp#L207-L225)

**Sources:**[test/pin_pages.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/pin_pages.cpp)

### Resource Lock Tests

`TestLock` validates the `TENSTORRENT_IOCTL_LOCK_CTL` interface for inter-process coordination [test/lock.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/lock.cpp)

*   **Exclusivity**: Verifies that once `fd0` acquires a lock index, `fd1` cannot acquire it until released [test/lock.cpp 128-135](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/lock.cpp#L128-L135)
*   **Auto-Release**: Confirms that closing a file descriptor automatically releases all locks held by that descriptor [test/lock.cpp 161-185](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/lock.cpp#L161-L185)
*   **Blocking Acquire**: Tests `TENSTORRENT_LOCK_CTL_ACQUIRE_BLOCKING` by spawning a child process to hold the lock and then release it, ensuring the parent blocks and then proceeds [test/lock.cpp 218-270](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/lock.cpp#L218-L270)

**Sources:**[test/lock.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/lock.cpp)

### Hardware Monitoring (hwmon)

`TestHwmon` checks the standard Linux `hwmon` interface exposed by the driver [test/hwmon.cpp 73-93](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/hwmon.cpp#L73-L93)

*   **Label Verification**: Ensures sensor labels match expected regex patterns (e.g., `vcore`, `asic_temp`) [test/hwmon.cpp 12-32](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/hwmon.cpp#L12-L32)
*   **Range Checking**: Verifies that current inputs are within the limits defined by `*_max` files [test/hwmon.cpp 34-68](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/hwmon.cpp#L34-L68)

**Sources:**[test/hwmon.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/hwmon.cpp)

### Peer BAR Mapping

`TestMapPeerBar` validates the ability to map one Tenstorrent device's BAR into the address space of another Tenstorrent device [test/map_peer_bar.cpp 151-167](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/map_peer_bar.cpp#L151-L167)

*   **Same Device**: Rejects attempts to map a device to itself [test/map_peer_bar.cpp 110-125](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/map_peer_bar.cpp#L110-L125)
*   **Chip Matching**: Ensures peer mapping only works between identical chip types [test/map_peer_bar.cpp 127-142](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/map_peer_bar.cpp#L127-L142)
*   **BAR Discovery**: Uses sysfs `/resource` to find valid memory BARs for mapping [test/map_peer_bar.cpp 35-81](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/map_peer_bar.cpp#L35-L81)

**Sources:**[test/map_peer_bar.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/map_peer_bar.cpp)

### IOCTL Zeroing and Overrun

*   **Zeroing**: `TestIoctlZeroing` provides a buffer larger than the output structure, fills it with `0xFF`, and verifies the driver zeros the remainder of the page after the valid output data [test/ioctl_zeroing.cpp 28-42](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/ioctl_zeroing.cpp#L28-L42)
*   **Overrun**: Ensures the driver respects the `output_size_bytes` limit and does not write past the user-provided buffer.

**Sources:**[test/ioctl_zeroing.cpp](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/ioctl_zeroing.cpp)

## Validation Strategies

### Strategy 1: Output Size Validation

IOCTL structures use the `output_size_bytes` field for forward/backward compatibility:

`// Set expected output sizeget_info.in.output_size_bytes = sizeof(get_info.out); // Call IOCTLioctl(fd, IOCTL_CMD, &get_info); // Validate kernel wrote expected sizeif (get_info.out.output_size_bytes < minimum_required_size)    THROW_TEST_FAILURE("Output too small");`
**Sources:**[test/ioctl_zeroing.cpp 44-50](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/ioctl_zeroing.cpp#L44-L50)[test/dma_buf.cpp 19-29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/dma_buf.cpp#L19-L29)

### Strategy 2: Cross-Verification with System Files

Many tests verify driver behavior by cross-referencing with sysfs or debugfs. For example, `TestHwmon` reads `/sys/bus/pci/devices/.../hwmon/hwmonX/`[test/hwmon.cpp 75-76](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/test/hwmon.cpp#L75-L76)

## Adding New Tests

### Step-by-Step Guide

### 1. Create Test File

Create a new `.cpp` file in the `test/` directory. Use `DevFd` for management and `THROW_TEST_FAILURE` for errors.

### 2. Update Makefile

Add the new test file to `TEST_SOURCES` in the test Makefile.

### 3. Update main.cpp

Add the forward declaration and call the test function within the device loop in `main()`.
