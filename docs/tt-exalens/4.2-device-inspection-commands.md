This page documents the `device` command (alias `d`) in the TTExaLens CLI. This command renders a 2D grid of the chip's on-chip blocks, allowing you to inspect the spatial layout of the chip, the status of RISC-V cores, coordinate assignments, and block types — all with configurable axis coordinate systems and cell content modes.

For the underlying coordinate system concepts (noc0, noc1, logical, translated, die), see [1.2](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/1.2) For the Python library equivalents (`get_block`, `get_block_type`, `get_block_locations`), see [3.2](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/3.2) For memory and register commands used after locating a core, see [4.3](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/4.3)

* * *

## Command Syntax

```
device [-d <device-id>] [<axis-coordinate> [<cell-contents>]] [--no-legend]
```

**Short name:**`d`

**Implemented in:**[ttexalens/cli_commands/device_summary.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py)

| Argument / Option | Default | Description |
| --- | --- | --- |
| `-d <device-id>` | all | Restrict output to a single device by its integer ID |
| `<axis-coordinate>` | `logical-tensix` | Coordinate system used for the row/column axes of the grid |
| `<cell-contents>` | `riscv` | Comma-separated list of what to display in each cell |
| `--no-legend` | off | Suppress the legend printed above the grid |

* * *

## Axis Coordinate Systems

The `<axis-coordinate>` argument controls which coordinate system is used to label the rows and columns of the displayed grid. Only locations that exist in the chosen coordinate space are shown.

| Axis Value | What is shown | Includes harvested? |
| --- | --- | --- |
| `logical-tensix` | Only functional Tensix worker cores | No |
| `logical-eth` | Only Ethernet cores | No |
| `logical-dram` | Only DRAM channel cores | No |
| `noc0` | All blocks using NOC0 physical coordinates | Yes |
| `noc1` | All blocks using NOC1 physical coordinates | Yes |
| `translated` | All blocks using translated NOC coordinates | Yes |
| `die` | All blocks using die-level coordinates | Yes |

The set of valid axis values is defined by `VALID_COORDINATE_TYPES` in [ttexalens/coordinate.py 53-62](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/coordinate.py#L53-L62) minus the bare `logical` entry (which is excluded at [ttexalens/cli_commands/device_summary.py 83-86](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py#L83-L86)).

## Cell Contents Options

The `<cell-contents>` argument is a comma-separated list of tokens. Each token selects what is rendered inside each grid cell. Multiple tokens produce comma-separated values within the same cell.

| Token | Description |
| --- | --- |
| `riscv` | Per-core status string: `R` (running) or `-` (in reset) for each RISC-V core. Falls back to block type name if no RISC-V cores exist on that block. |
| `block` | The block type name (e.g., `functional_workers`, `dram`, `eth`) |
| `noc0` | NOC0 coordinate of this block, formatted as `X-Y` |
| `noc1` | NOC1 coordinate of this block |
| `translated` | Translated NOC coordinate |
| `die` | Die coordinate |
| `logical` | Logical coordinate (with core type prefix, e.g., `t0,0` for tensix) |
| `noc0_id` | NOC0 node ID read from the `NOC_NODE_ID` hardware register |
| `noc1_id` | NOC1 node ID read from the `NOC_NODE_ID` register via NOC1 |
| `noc0_logical_id` | NOC0 logical node ID from the `NOC_ID_LOGICAL` register |
| `noc1_logical_id` | NOC1 logical node ID from the `NOC_ID_LOGICAL` register via NOC1 |

The coordinate tokens (`noc0`, `noc1`, etc.) call `loc.to_str(coord_type)` on the `OnChipCoordinate` object for each cell. The `noc0_id` / `noc1_id` tokens read live hardware registers via `register_store.read_register()`, making them suitable for verifying NOC routing table configuration.

## RISC-V Status Display

**Diagram: RISC-V status string construction**

Sources: [ttexalens/cli_commands/device_summary.py 57-76](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py#L57-L76)

The function `get_riscv_run_status` produces a string of characters, one per RISC-V core on the block:

*   `R` — core is running (not in reset)
*   `-` — core is in reset

For blocks with no RISC-V cores (DRAM, ARC, PCIe, router-only, etc.), the function falls back to printing the block type name. For `harvested_workers` blocks, it unconditionally returns `"-----"` because those cores have no functional hardware.

For a typical functional Tensix worker with BRISC, NCRISC, TRISC0, TRISC1, TRISC2, the status string is five characters wide (e.g., `RRRRR` or `-----`).

## Block Types and Colors

When the legend is enabled, each block type is printed in a distinct terminal color. The set of block types shown depends on the chosen axis:

| Axis | Block types shown in legend |
| --- | --- |
| `logical-tensix` | `functional_workers` only |
| `logical-eth` | `eth` only |
| `logical-dram` | `dram` only |
| Any raw coordinate (`noc0`, `noc1`, `die`, `translated`) | All block types |

Block type colors come from `Device.block_types[block_type].color` accessed by `color_block()` at [ttexalens/cli_commands/device_summary.py 52-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py#L52-L54)

Known block type names (as seen in grid output):

*   `functional_workers`
*   `harvested_workers`
*   `eth`
*   `harvested_eth`
*   `dram`
*   `harvested_dram`
*   `arc`
*   `pcie`
*   `router_only`
*   `security`
*   `l2cpu`

## Examples

### Show RISC-V status for all devices (default)

```
device
```

Output uses `logical-tensix` axis. Each cell shows a 5-character status string, one character per RISC-V core. Cells are color-coded by block type.

```
Legend:
  Axis coordinates: logical-tensix
  Cell contents: riscv
    riscv - show the status of the RISC-V ('R': running, '-': in reset)
  Colors:
    functional_workers

==== Device 0 [p100 / local] (Unique ID: 0x40500)
    00     01     02     ...
00  -----  -----  -----  ...
01  -----  -----  -----  ...
```

### Show all block types on NOC0 grid

```
device noc0
```

Displays all blocks including DRAM, Ethernet, ARC, PCIe etc. using NOC0 X-Y coordinates for the axis labels.

### Show NOC0 coordinates on a logical-tensix grid

```
device logical-tensix noc0
```

Each cell shows the NOC0 `X-Y` address of the functional Tensix worker at that logical position. Useful for building mental maps between logical and physical addresses.

```
==== Device 0 [p100 / local] (Unique ID: 0x40500)
    00    01    02    ...
00  1-2   2-2   3-2   ...
01  1-3   2-3   3-3   ...
```

### Show block types without legend, on NOC0 grid

```
device noc0 block --no-legend
```

```
00              01                  02
00  dram            router_only         pcie
01  dram            harvested_eth       harvested_eth
02  dram            functional_workers  functional_workers
```

### Show RISC-V status on die axis for a specific device

```
device -d 0 die
```

Restricts output to device 0 and uses die-level coordinates for axis labels.

### Show NOC0 coordinates on logical-dram grid for device 0

```
device -d 0 logical-dram noc0
```

Lists DRAM channel blocks only, showing their NOC0 addresses:

```
==== Device 0 [p100 / local] (Unique ID: 0x40500)
    00    01    02   ...
00  0-0   0-2   0-5  ...
01  0-1   0-10  0-7  ...
```

### Show multiple cell contents

```
device noc0 riscv,noc0
```

Each cell in the grid displays both the RISC-V status and the NOC0 coordinate for that location, separated by a comma.

Sources: [docs/ttexalens-app-docs.md 403-595](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/docs/ttexalens-app-docs.md?plain=1#L403-L595)[ttexalens/cli_commands/device_summary.py 79-182](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py#L79-L182)

## Device Header Format

For each device, the header line is printed using `util.INFO` with this format:

```
==== Device <id> [<board_type> / <local|remote>] (Unique ID: 0x<unique_id>)
```

*   `board_type` comes from `device.board_type`
*   Local vs. remote is determined by `device.is_local`; remote devices show `remote(<local_device_id>)`
*   `unique_id` from `device.unique_id` is shown as hex

Sources: [ttexalens/cli_commands/device_summary.py 125-133](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/cli_commands/device_summary.py#L125-L133)
