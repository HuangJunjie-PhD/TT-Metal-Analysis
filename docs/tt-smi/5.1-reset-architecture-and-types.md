## Purpose and Overview

This document describes the reset architecture of TT-SMI across different Tenstorrent hardware platforms. TT-SMI implements four distinct reset types, each tailored to specific hardware characteristics and operational requirements:

1.   **Board-Level Reset (Wormhole)** - Full power cycle with Ethernet retraining
2.   **ASIC-Level Reset (Blackhole)** - ASIC-focused reset without full board power cycling
3.   **IPMI-Based Reset (Galaxy Systems)** - Tray-based reset via IPMI commands for large-scale deployments
4.   **Tensix-Level Reset (Grayskull, deprecated v3.0.35)** - Core-level reset without power cycling

Each reset type is implemented through specialized reset handlers that abstract hardware-specific operations while maintaining a consistent interface for the user.

## Reset Architecture Overview

TT-SMI routes reset requests through the `pci_board_reset` function, which dispatches to platform-specific reset handlers based on the detected chip type. The architecture separates reset logic by hardware platform while providing a unified interface.

**Reset Dispatch Architecture**

The reset dispatcher determines which hardware is present, groups devices by type, and invokes the appropriate reset handler for each platform.

## Board-Level Reset (Wormhole)

Wormhole devices implement a **board-level reset** that performs a full power cycle of the board, cutting power and then restoring it. This reset type is the most comprehensive for Wormhole hardware, ensuring a clean hardware state.

**Wormhole Reset Implementation**

The reset handler is invoked through `WHChipReset().full_lds_reset()`, which is imported from `tt_tools_common.reset_common.wh_reset`. For Galaxy systems (board type `0x35`), the reset disables secondary bus reset to avoid conflicts with the Galaxy's specialized reset mechanism.

**Key Characteristics:**

*   Full board power cycle
*   Ethernet link retraining after power restoration
*   Optional secondary bus reset (disabled for Galaxy)
*   Requires driver reload to re-enumerate devices

## ASIC-Level Reset (Blackhole)

Blackhole devices implement an **ASIC-level reset** that resets the ASIC without performing a full board power cycle. This approach is more targeted than Wormhole's board-level reset and allows for faster reset times.

**Blackhole Reset Implementation**

The `BHChipReset` handler is invoked through `BHChipReset().full_lds_reset()`, imported from `tt_tools_common.reset_common.bh_reset`. The ASIC-level reset provides a middle ground between Grayskull's core-level reset and Wormhole's full board reset.

**Key Characteristics:**

*   ASIC logic reset without full power cycle
*   Faster than full board reset
*   Maintains board-level power
*   Still requires driver re-enumeration

## IPMI-Based Reset (Galaxy Systems)

Galaxy 6U systems implement an **IPMI-based reset** mechanism that uses Intelligent Platform Management Interface commands to reset entire trays of devices. This architecture is designed for large-scale deployments with 32 devices across 4 trays.

**Galaxy Reset Architecture**

The Galaxy reset flow begins with `glx_6u_trays_reset()`, which issues IPMI commands via `run_wh_ubb_ipmi_reset()` from pyluwen. The reset parameters control which trays and devices are affected:

**IPMI Reset Parameters**

| Parameter | Type | Description |
| --- | --- | --- |
| `ubb_num` | str | UBB number bitmap (0x0-0xF), 0xF = all trays |
| `dev_num` | str | Device number bitmap (0x0-0xFF), 0xFF = all devices |
| `op_mode` | str | 0x0=Assert/Deassert, 0x1=Assert only, 0x2=Deassert only |
| `reset_time` | str | Reset duration in 10ms units (0xF = 150ms) |

After the IPMI reset completes, the system waits 30 seconds for the driver to reload, then performs Ethernet link status validation via `check_wh_galaxy_eth_link_status()`. This validation checks all 16 Ethernet ports on each of the 32 devices for the `LINK_INACTIVE_FAIL_DUMMY_PACKET` error condition.

**Key Characteristics:**

*   IPMI-based tray reset for large deployments
*   30-second wait for driver reload
*   Mandatory Ethernet link validation (for full galaxy resets)
*   Support for single-tray resets
*   Auto-retry capability (up to 3 attempts)

## Deprecated Tensix-Level Reset (Grayskull)

Grayskull devices formerly supported a **Tensix-level reset** that reset individual Tensix cores without power cycling the entire board. This reset type was removed in TT-SMI v3.0.35 when Grayskull support was deprecated.

**Historical Tensix Reset Architecture**

The Tensix reset operated at the core level, resetting computational units while maintaining board power and PCIe connectivity. This approach allowed for rapid reset cycles during development but required careful state management to ensure consistent behavior.

**Deprecation Notice:**

*   Grayskull support removed in v3.0.35
*   Tensix-level reset no longer available
*   Users with Grayskull devices should use older TT-SMI versions (< v3.0.35)

## Troubleshooting

### Common Issues

| Issue | Possible Cause | Solution |
| --- | --- | --- |
| "Error accessing board at PCI index X" | Invalid PCI index or device not present | Use `tt-smi -ls` to list available devices |
| Reset fails on ARM systems | Different PCIe configuration | System reboot recommended for ARM systems |
| Reset doesn't affect all devices | Incorrect configuration | Verify PCI indices in configuration file |
| No devices re-initialized after reset | `re_init_devices` set to false | Set `re_init_devices: true` in configuration |

## Reset Configuration Processing Internals

Internally, TT-SMI processes reset configurations through several key functions:

1.   `pci_indices_from_json` - Extracts PCI indices from the configuration
2.   `mobo_reset_from_json` - Processes motherboard reset configurations
3.   `pci_board_reset` - Performs the actual resets based on device type

The code ensures that motherboard resets are performed before individual device resets and properly handles device re-initialization if configured.
