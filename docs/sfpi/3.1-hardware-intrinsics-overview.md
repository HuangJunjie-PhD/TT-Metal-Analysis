# Hardware Intrinsics Overview

## Purpose and Scope

This document describes the hardware intrinsics layer that provides direct access to SFPU (Special Function Processing Unit) instructions across Tenstorrent architectures. Hardware intrinsics are compiler built-in functions prefixed with `__builtin_rvtt_sfp*` that map directly to SFPU machine instructions. These intrinsics form the lowest-level programmable interface to the SFPU hardware.

For higher-level C++ wrapper types and operators that use these intrinsics, see [Vector Types and Operations](https://deepwiki.com/tenstorrent/sfpi/2.1-vector-types-and-operations). For platform-specific implementations and differences, see [Wormhole Architecture](https://deepwiki.com/tenstorrent/sfpi/3.2-wormhole-architecture), [Blackhole Architecture](https://deepwiki.com/tenstorrent/sfpi/3.3-blackhole-architecture), and [Grayskull Architecture](https://deepwiki.com/tenstorrent/sfpi/3.4-grayskull-architecture). For math library functions built on these intrinsics, see [Math and Vector Libraries](https://deepwiki.com/tenstorrent/sfpi/4-math-and-vector-libraries).

## Intrinsics Architecture

Hardware intrinsics are defined as preprocessor macros that wrap GCC built-in functions. Each architecture (Wormhole, Blackhole, Grayskull) provides its own set of intrinsics in `<arch>/sfpi_hw.h` files. The macros follow a consistent naming pattern and expand to architecture-specific built-in functions with platform suffixes (`_wh_`, `_bh_`, `_gs_`).

## Intrinsic Naming Conventions

All hardware intrinsics follow consistent naming patterns:

| Pattern | Description | Example |
| --- | --- | --- |
| `__builtin_rvtt_sfp<op>` | Base intrinsic name | `__builtin_rvtt_sfpadd` |
| `__builtin_rvtt_sfpx<op>` | Extended/experimental operation | `__builtin_rvtt_sfpxloadi` |
| `__builtin_rvtt_sfp<op>_i` | Immediate operand variant | `__builtin_rvtt_sfpsetexp_i` |
| `__builtin_rvtt_sfp<op>_v` | Vector operand variant | `__builtin_rvtt_sfpsetexp_v` |
| `__builtin_rvtt_<arch>_sfp<op>` | Architecture-specific built-in | `__builtin_rvtt_wh_sfpadd` |

The macros in `sfpi_hw.h` hide the architecture suffix by conditionally compiling the appropriate variant based on preprocessor symbols (`__riscv_xtttensixwh`, `__riscv_xtttensixbh`, `__riscv_xtttensixgs`).

## Intrinsic Parameters and Modifiers

Most intrinsics accept modifier parameters (`mod0`, `mod1`) that control operation behavior. These modifiers are defined as constants in `sfpi_hw.h`.

### Common Parameter Types

| Parameter | Type | Purpose | Example |
| --- | --- | --- | --- |
| `va`, `vb`, `vc` | `__rvtt_vec_t` | Vector operands | `sfpadd(va, vb, mod1)` |
| `src`, `dst` | `__rvtt_vec_t` | Source/destination vectors | `sfpabs(src, mod1)` |
| `mod0`, `mod1` | `unsigned int` | Operation modifiers | Format, rounding, sign handling |
| `imm8`, `imm12`, `imm16` | Immediate | Constant operands | Shift amounts, immediate values |
| `addr` | Address | Memory address | DEST register index |

## Architecture-Specific Variations

### Wormhole (51 Operations)

Wormhole provides the baseline set of SFPU operations defined in [include/wormhole/sfpi_hw.h 1-282](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/wormhole/sfpi_hw.h#L1-L282)

**Key characteristics:**

*   `SFPLOAD_ADDR_MODE_NOINC = 3`
*   `SFPSTORE_ADDR_MODE_NOINC = 3`
*   No `sfparecip`, `sfpmul24`, or `sfpmov_config` intrinsics

### Blackhole (54 Operations)

Blackhole extends Wormhole with additional operations defined in [include/blackhole/sfpi_hw.h 1-306](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/blackhole/sfpi_hw.h#L1-L306)

**Key characteristics:**

*   `SFPLOAD_ADDR_MODE_NOINC = 7`
*   `SFPSTORE_ADDR_MODE_NOINC = 7`
*   Additional intrinsics: 
    *   `__builtin_rvtt_sfpmov_config` - Move to configuration register
    *   `__builtin_rvtt_sfparecip` - Approximate reciprocal
    *   `__builtin_rvtt_sfpmul24` - 24-bit multiply

*   Enhanced `sfpshft` with logical/arithmetic modes
*   Additional `sfpcast` modes for sign-magnitude conversion

## Hardware Configuration Constants

Both architectures define common hardware configuration constants:

### Register File Configuration

`// LREG (Local Registers)constexpr unsigned int SFP_LREG_COUNT = 8;Â // DREG (Destination Registers)constexpr unsigned int SFP_DESTREG_STRIDE = 2;constexpr unsigned int SFP_DESTREG_MAX_SIZE = 1024;constexpr unsigned int SFP_DESTREG_MAX_ADDR = 512;  // MAX_SIZE / STRIDE`
The SFPU has:

*   **8 LREGs** (LREG0-LREG7): Local registers for intermediate values
*   **512 DREGs**: Destination register file (1024 rows Ă— 16 cols Ă— 16-bit, or 512 rows Ă— 16 cols Ă— 32-bit)
*   **64-element vectors**: All operations work on 64-element SIMD vectors

## Vector Type Definition

All intrinsics operate on the `__rvtt_vec_t` type, which represents a 64-element vector:

`typedef float __rvtt_vec_t __attribute__((vector_size(64*4)));`
This type is:

*   256 bytes (64 floats Ă— 4 bytes)
*   Treated as a GCC vector type with 64 elements
*   Used for all SFPU vector operations
*   Automatically handled by the compiler for register allocation

The `sfpi_inline` macro ensures intrinsics are always inlined:

`#define sfpi_inline __attribute__((always_inline)) inline`

## Summary

Hardware intrinsics provide direct, low-level access to SFPU instructions through GCC built-in functions. The intrinsics are organized into seven functional categories (memory, arithmetic, bitwise, conditional, float manipulation, lookup/conversion, vector manipulation) and use modifier constants to control operation behavior. Wormhole defines 51 operations as the baseline, while Blackhole extends this to 54 with additional reciprocal, multiply, and configuration operations. All intrinsics operate on 64-element vectors stored in an 8-LREG, 512-DREG register file architecture.

For implementation details of how these intrinsics are used in C++ wrapper types, see [Vector Types and Operations](https://deepwiki.com/tenstorrent/sfpi/2.1-vector-types-and-operations). For architecture-specific details and differences, see [Wormhole Architecture](https://deepwiki.com/tenstorrent/sfpi/3.2-wormhole-architecture), [Blackhole Architecture](https://deepwiki.com/tenstorrent/sfpi/3.3-blackhole-architecture), and [Grayskull Architecture](https://deepwiki.com/tenstorrent/sfpi/3.4-grayskull-architecture).
