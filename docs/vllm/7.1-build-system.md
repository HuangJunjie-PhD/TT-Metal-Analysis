This document explains the vLLM build system, which is responsible for compiling C++/CUDA extensions and creating Python packages. If you're looking for general installation instructions, see [Installation Guide](https://deepwiki.com/tenstorrent/vllm/1.1-installation-guide).

## Overview

vLLM uses a CMake-based build system to compile native extensions for optimal performance across multiple hardware platforms. The build system supports CUDA, ROCm, CPU, TPU, HPU, and Neuron architectures, with optimized builds for each platform. This document explains the build architecture, configuration options, and provides guidance for developers working with the codebase.

Sources: [setup.py 60-675](https://github.com/tenstorrent/vllm/blob/cd9e5b83/setup.py#L60-L675)[pyproject.toml 1-51](https://github.com/tenstorrent/vllm/blob/cd9e5b83/pyproject.toml#L1-L51)

## Build System Components

### Main Components

The build system consists of the following components:

1.   **setup.py**: The main entry point for building vLLM, which manages the configuration and compilation process
2.   **pyproject.toml**: Defines build dependencies and project metadata
3.   **CMake**: Used to configure and build the C++/CUDA extensions
4.   **Extension modules**: Native code compiled into Python-loadable libraries

## Extension Modules

vLLM compiles different extension modules based on the target platform:

| Extension Module | Description | Platforms | Requirements |
| --- | --- | --- | --- |
| `vllm._C` | Core kernels (PagedAttention, activations, layernorm, quantization) | CUDA, ROCm, CPU | Always built for supported devices |
| `vllm._moe_C` | Mixture of Experts (FusedMoE) kernels | CUDA, ROCm | Built when `_build_custom_ops()` returns True |
| `vllm._rocm_C` | ROCm-specific kernels (AIter attention) | ROCm | Built only on ROCm platforms |
| `vllm.vllm_flash_attn._vllm_fa2_C` | FlashAttention 2 kernels | CUDA | CUDA only |
| `vllm.vllm_flash_attn._vllm_fa3_C` | FlashAttention 3 kernels | CUDA | CUDA 12.3+, built conditionally |
| `vllm._flashmla_C` | FlashMLA kernels for DeepSeek | CUDA | CUDA 12.3+, Hopper+ (optional) |
| `vllm._flashmla_extension_C` | FlashMLA extension kernels | CUDA | CUDA 12.3+, Hopper+ (optional) |
| `vllm._sparse_flashmla_C` | Sparse FlashMLA kernels | CUDA | CUDA 12.3+, Hopper+ (optional) |
| `vllm.cumem_allocator` | CUDA memory allocator wrapper | CUDA | Links against `libcuda.so` |

The extension build is controlled by the `ext_modules` list in `setup.py`, which is conditionally populated based on device detection:

`# From setup.pyif _is_cuda() or _is_hip():    ext_modules.append(CMakeExtension(name="vllm._moe_C")) if _is_hip():    ext_modules.append(CMakeExtension(name="vllm._rocm_C")) if _is_cuda():    ext_modules.append(CMakeExtension(name="vllm.vllm_flash_attn._vllm_fa2_C"))    if envs.VLLM_USE_PRECOMPILED or get_nvcc_cuda_version() >= Version("12.3"):        ext_modules.append(CMakeExtension(name="vllm.vllm_flash_attn._vllm_fa3_C"))        # Optional extensions for Hopper+        ext_modules.append(CMakeExtension(name="vllm._flashmla_C", optional=True))    ext_modules.append(CMakeExtension(name="vllm.cumem_allocator")) if _build_custom_ops():    ext_modules.append(CMakeExtension(name="vllm._C"))`

## Advanced Build Topics

### Kernel Generation and Caching

Some kernels are auto-generated to reduce binary size and compilation time:

**Marlin Kernel Generation** ([CMakeLists.txt 339-370](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L339-L370)):

*   Script: `csrc/quantization/gptq_marlin/generate_kernels.py`
*   Generates specialized kernels for different data types and schedules
*   Uses MD5 hash to detect script changes and avoid regeneration
*   Configured via `MARLIN_GEN_SCRIPT_HASH` CMake cache variable

**Machete Kernel Generation** ([CMakeLists.txt 752-783](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L752-L783)):

*   Script: `csrc/quantization/machete/generate.py`
*   Generates mixed-precision GEMM kernels for Hopper
*   Requires CUTLASS Python bindings in `VLLM_PYTHON_PATH`
*   Hash-based caching similar to Marlin

### Precompiled Wheel Usage

For rapid development without C++ changes, use precompiled wheels ([setup.py 296-358](https://github.com/tenstorrent/vllm/blob/cd9e5b83/setup.py#L296-L358)):

`VLLM_USE_PRECOMPILED=1 pip install -e .`
The precompiled wheel extraction process:

1.   Determines base commit via `git merge-base` with upstream main
2.   Downloads wheel from `https://wheels.vllm.ai/{commit}/vllm-*.whl`
3.   Extracts `.so` files and Python files from wheel
4.   Patches `package_data` to include extracted files

Falls back to nightly wheel if commit-specific wheel is unavailable.

### CUTLASS Integration

vLLM uses CUTLASS for optimized GEMM kernels ([CMakeLists.txt 282-312](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L282-L312)):

**FetchContent Configuration**:

`FetchContent_Declare(    cutlass    GIT_REPOSITORY https://github.com/nvidia/cutlass.git    GIT_TAG v4.2.1    GIT_SHALLOW TRUE)`
**Local CUTLASS Development**:

`export VLLM_CUTLASS_SRC_DIR=/path/to/cutlasspip install -e .`
This allows testing CUTLASS changes without modifying vLLM's fetch configuration.

### Sparse Compilation Support

**FP8 Sparse Kernels** ([CMakeLists.txt 547-568](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L547-L568)):

*   2:4 structured sparsity via `cutlass_scaled_sparse_mm_c3x`
*   Requires CUDA 12.2+ and SM 9.0 (Hopper)
*   Includes sparse compressor for runtime weight conversion

**Sparse FlashMLA** ([setup.py 326](https://github.com/tenstorrent/vllm/blob/cd9e5b83/setup.py#L326-L326)):

*   Optional sparse attention for DeepSeek models
*   Built conditionally on Hopper systems

### Cross-Compilation Notes

**ROCm/HIP Compilation** ([CMakeLists.txt 122-165](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L122-L165)):

*   Requires explicit `enable_language(HIP)` after Torch import
*   Uses hipify for CUDA→HIP source translation
*   Special debug flags: `-O0 -ggdb3` for verbose debugging
*   Suppresses `-Wno-unused-result` warnings

**CPU Compilation** ([cmake/cpu_extension.cmake](https://github.com/tenstorrent/vllm/blob/cd9e5b83/cmake/cpu_extension.cmake)):

*   OpenMP-based kernels for CPU inference
*   Separate CMake script for CPU-specific configuration
*   No GPU architecture flags needed

## Troubleshooting Build Issues

### Compilation Errors

**Out of Memory During Compilation**:

`# Reduce parallel jobsexport MAX_JOBS=1pip install -e . # Or reduce NVCC threads (CUDA 11.2+)export MAX_JOBS=4export NVCC_THREADS=2pip install -e .`
**CUDA Toolkit Not Found**:

`export CUDA_HOME=/usr/local/cuda-12.8export PATH="${CUDA_HOME}/bin:$PATH"export LD_LIBRARY_PATH="${CUDA_HOME}/lib64:$LD_LIBRARY_PATH"nvcc --version  # Verify`
**CMake Version Too Old**:

`# vLLM requires CMake 3.26+pip install "cmake>=3.26"`

## Internal Build Process Details

The CMake build process internally follows these steps:

1.   Configure build type (Debug/Release/RelWithDebInfo)
2.   Set target device-specific parameters
3.   Configure compiler and build tool settings
4.   Configure parallelism settings
5.   Build extension modules with specified options
6.   Install the compiled extensions to the package directory

For C++ and CUDA extensions, the build process:

1.   Creates or updates CMake configuration
2.   Builds all extension targets in parallel
3.   Installs the resulting libraries to the correct Python package location
