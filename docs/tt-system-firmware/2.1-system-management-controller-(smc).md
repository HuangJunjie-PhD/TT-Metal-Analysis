## Purpose and Scope

The System Management Controller (SMC) is firmware that runs on the ARC processor within Blackhole ASIC boards, providing comprehensive system management, monitoring, and control capabilities. The SMC operates on Zephyr RTOS and serves as the primary interface between the host system and the ASIC hardware.

This document covers the SMC firmware architecture, message handling, telemetry collection, dynamic voltage/frequency scaling (DVFS), and power management mechanisms. For information about the Device Management Controller that works alongside SMC, see [DMC](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2-device-management-controller-(dmc)). For details on specific hardware subsystems managed by SMC, see [Hardware Subsystems](https://deepwiki.com/tenstorrent/tt-system-firmware/3-hardware-subsystems).

**Sources:**[app/smc/sample.yaml 1-93](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/sample.yaml#L1-L93)[lib/tenstorrent/bh_arc/telemetry.c 1-532](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L1-L532)

## Architecture Overview

### Runtime Environment

The SMC firmware executes on an ARC processor embedded within the Blackhole ASIC. It runs Zephyr RTOS v0.29.99.0 and provides a structured event-driven architecture for managing hardware and responding to host requests.

**Key Components:**

*   **Message Queue System**: 4 request/response queues for host communication
*   **Telemetry Engine**: Periodic (100ms) collection and publication of system metrics
*   **DVFS Engine**: 1ms periodic dynamic voltage/frequency adjustment
*   **Throttlers**: PID controllers limiting AICLK based on power/thermal constraints
*   **AICLK PPM**: Power/Performance Management arbitration for AI clock frequency

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 1-475](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L1-L475)[lib/tenstorrent/bh_arc/dvfs.c 1-89](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L1-L89)[lib/tenstorrent/bh_arc/telemetry.c 1-532](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/telemetry.c#L1-L532)

## Message Handling System

### Message Queue Architecture

The SMC implements a lock-free message queue system with 4 independent queues, each containing request and response FIFOs. The host writes requests and reads responses, while the SMC processes requests and writes responses.

**Key Data Structures:**

```
struct message_queue_header {
    uint32_t request_queue_wptr;   // Host writes
    uint32_t response_queue_rptr;  // Host writes
    uint32_t request_queue_rptr;   // ARC writes
    uint32_t response_queue_wptr;  // ARC writes
    uint32_t last_serial;          // Message sequence tracking
}

union request {
    uint32_t data[8];              // Generic access
    uint8_t command_code;          // Message type from tt_smc_msg
    struct force_fan_speed_rqst force_fan_speed;
    struct aiclk_set_speed_rqst aiclk_set_speed;
    // ... 15+ specialized request types
}
```

## DVFS and Power Management

### DVFS Overview

The SMC implements a comprehensive Dynamic Voltage and Frequency Scaling (DVFS) system that runs every 1ms, coordinating voltage changes with frequency changes to maintain stability while optimizing power consumption.

**Critical Ordering**: The sequence decreases frequency before increasing voltage, and increases frequency after decreasing voltage, to prevent instability.

**Sources:**[lib/tenstorrent/bh_arc/dvfs.c 14-30](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L14-L30)[lib/tenstorrent/bh_arc/dvfs.c 32-58](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/dvfs.c#L32-L58)

### AICLK Arbitration System

The AICLK Power/Performance Management (PPM) system uses a two-tier arbitration scheme with minimum and maximum arbiters:

**Arbiter Priority Rules:**

1.   Calculate highest enabled minimum arbiter → `effective_min`
2.   Calculate lowest enabled maximum arbiter → `effective_max`
3.   `targ_freq = min(max(effective_min, fmin), effective_max)`
4.   If sweep enabled, randomize in range
5.   If forced frequency set, override all

**Sources:**[lib/tenstorrent/bh_arc/aiclk_ppm.c 91-140](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.c#L91-L140)[lib/tenstorrent/bh_arc/aiclk_ppm.h 19-94](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/aiclk_ppm.h#L19-L94)

### Throttler System

Seven throttlers implement PID controllers that adjust AICLK maximum arbiters based on power and thermal measurements:

### Doppler Throttling

Doppler throttling is an advanced power limiting mode that uses moving average power measurements and kernel-level throttling:

**Doppler Tiers:**

1.   **Slow Throttle**: Moving average (1s window) exceeds board power limit → Reduce AICLK via PID
2.   **T2 Throttle**: Instantaneous power > 2x limit for 10 consecutive samples (1s) → Enable kernel NOPs
3.   **T3 Throttle**: Instantaneous power > 2.5x limit for 2 consecutive samples (200ms) → Enable kernel NOPs + critical AICLK=fmin

**Kernel NOP Injection**: When throttling is active, the SMC broadcasts a throttle counter to all Tensix cores via NOC:

`// From throttler.c:156-188static void BroadcastKernelThrottleState(void) {    const uint8_t kNocRing = 0;    const uint8_t kNocTlb = 1;        if (tensixes_enabled) {        NOC2AXITensixBroadcastTlbSetup(kNocRing, kNocTlb,                                        kKernelThrottleAddress,  // 0x10                                       kNoc2AxiOrderingStrict);        NOC2AXIWrite32(kNocRing, kNocTlb, kKernelThrottleAddress, throttle_counter);    }} static void SendKernelThrottlingMessage(bool throttle) {    throttle_counter++;    if ((throttle_counter & 1) != throttle) {        throttle_counter++;  // Ensure parity: even=fast, odd=slow    }    BroadcastKernelThrottleState();}`

### Fan Control

The SMC controls fan speed based on ASIC and GDDR temperatures using a quadratic curve:

`// From fan_ctrl.c:47-74static uint32_t fan_curve(float max_asic_temp, float max_gddr_temp) {    uint32_t fan_speed1, fan_speed2;        // ASIC curve    if (max_asic_temp < 49)        fan_speed1 = 35;    else if (max_asic_temp < 90)        fan_speed1 = (uint32_t)(0.03867f * (max_asic_temp - 49.0f) *                                 (max_asic_temp - 49.0f)) + 35;    else        fan_speed1 = 100;        // GDDR curve    if (max_gddr_temp < 43)        fan_speed2 = 35;    else if (max_gddr_temp < 82)        fan_speed2 = (uint32_t)(0.04274f * (max_gddr_temp - 43.0f) *                                 (max_gddr_temp - 43.0f)) + 35;    else        fan_speed2 = 100;        return MAX(fan_speed1, fan_speed2);}`
Fan speed is updated every 1 second with alpha filtering (default α=0.5) to prevent rapid oscillations.

**Sources:**[lib/tenstorrent/bh_arc/fan_ctrl.c 1-170](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/fan_ctrl.c#L1-L170)[lib/tenstorrent/bh_arc/fan_ctrl.c 47-97](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/fan_ctrl.c#L47-L97)
