This page provides complete API documentation for all reset-related classes, functions, and data models in tt-tools-common. The reset system provides both modern unified reset implementations and chip-specific legacy implementations for Blackhole, Wormhole, and Grayskull devices.

For architectural information about how these components interact and the migration strategy from legacy to modern implementations, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns). For detailed implementation guides for specific chip types, see [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation), [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation), and [Grayskull Tensix Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.5-grayskull-tensix-reset).

### Methods

#### full_lds_reset()

Performs a full LDS (Link Down State) reset of a list of chips using the modern unified reset sequence.

**Signature:**

**Parameters:**

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `pci_interfaces` | `List[int]` | Required | List of PCI interface IDs to reset |
| `reset_m3` | `bool` | `False` | Whether M3/DMC should be reset (full board reset) |
| `silent` | `bool` | `False` | Whether to suppress console output |
| `m3_delay` | `int` | `20` | Time in seconds to wait after M3 reset |
| `secondary_bus_reset` | `bool` | `True` | Whether to perform secondary bus reset before ASIC reset |

**Returns:**`List[PciChip]` - List of `PciChip` objects for the reset devices.

**Raises:**

*   `SystemExit` - If driver version < 2.4.1, platform is ARM, or reset operations fail
*   `FileNotFoundError` - If device file `/dev/tenstorrent/{interface_id}` doesn't exist

**Implementation Details:**

1.   Validates driver version >= 2.4.1 and platform is not ARM [chip_reset.py 137-148](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L137-L148)
2.   Collects BDF (Bus:Device.Function) for all devices [chip_reset.py 159-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L159-L165)
3.   If Xen HVM detected, writes to xenstore to coordinate with host [chip_reset.py 169-193](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L169-L193)
4.   Performs optional secondary bus reset using `RESET_PCIE_LINK` flag [chip_reset.py 196-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L196-L203)
5.   Issues ASIC reset or ASIC+DMC reset based on `reset_m3` parameter [chip_reset.py 206-212](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L206-L212)
6.   Waits for xenstore files to be removed by host if in Xen HVM mode [chip_reset.py 214-256](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L214-L256)
7.   Waits for devices to reappear on PCI bus [chip_reset.py 267-268](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L267-L268)
8.   Executes post-reset cleanup using `POST_RESET` flag [chip_reset.py 270-278](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/chip_reset.py#L270-L278)

**Sources:**[tt_tools_common/reset_common/chip_reset.py 114-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L114-L288)

#### reset_device_ioctl()

Low-level method to issue IOCTL commands to the kernel driver for device reset operations.

**Signature:**

**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `interface_id` | `int` | PCI interface ID of the device |
| `flags` | `int` | Reset operation flags (use `IoctlResetFlags` enum) |

**Returns:**`bool` - `True` if operation succeeded, `False` otherwise.

**Implementation:** Opens `/dev/tenstorrent/{interface_id}`, packs reset request struct with output size and flags, executes `fcntl.ioctl()`, and unpacks result.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 60-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L85)

#### wait_for_device_to_reappear()

Waits for a device to reappear on the PCI bus after reset, typically after PCIe hot-plug removal/insertion.

**Signature:**

**Parameters:**

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `bdf` | `str` | Required | PCI Bus:Device.Function string (e.g., "0000:01:00.0") |
| `timeout` | `int` | `10` | Maximum wait time in seconds |

**Returns:**`int` - New PCI interface ID for the device.

**Raises:**`SystemExit` - If device doesn't reappear within timeout.

**Implementation:** Polls `/sys/bus/pci/devices/{bdf}/tenstorrent/tenstorrent!*` and `/dev/tenstorrent/{interface_id}` until both exist or timeout expires.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111)

#### full_lds_reset()

Performs full LDS reset for Blackhole devices. Automatically delegates to modern `ChipReset` implementation when driver version >= 2.4.1 or running under Xen HVM.

**Signature:**

**Parameters:** Same as `ChipReset.full_lds_reset()` (see above).

**Returns:**`List[PciChip]` - List of reset devices.

**Delegation Logic:**

1.   Checks driver version using `is_driver_version_at_least(get_driver_version(), "2.4.1")`[bh_reset.py 92](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L92-L92)
2.   Checks for Xen HVM using `check_xen_hvm()`[bh_reset.py 92](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L92-L92)
3.   If either condition is true, delegates to `ChipReset().full_lds_reset()`[bh_reset.py 93-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L93-L99)
4.   Otherwise, executes legacy implementation [bh_reset.py 101-223](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L101-L223)

**Legacy Implementation Flow:**

1.   Validates platform is not ARM [bh_reset.py 111-118](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L111-L118)
2.   Collects PCI BDF for all devices [bh_reset.py 127-136](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L127-L136)
3.   Issues reset IOCTLs or ARC messages based on `reset_m3`[bh_reset.py 137-144](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L137-L144)
4.   Polls config space reset bit until completion [bh_reset.py 150-192](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L150-L192)
5.   Restores device state [bh_reset.py 207-210](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/bh_reset.py#L207-L210)

**Sources:**[tt_tools_common/reset_common/bh_reset.py 68-223](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L68-L223)

## Helper Functions

### parse_reset_input()

Parses command-line arguments into structured reset input.

**Signature:**

**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `value` | `list` | List of string arguments from command line |

**Returns:**`ResetInput` - Dataclass containing:

*   `type`: One of `ResetType.ALL`, `ResetType.CONFIG_JSON`, or `ResetType.ID_LIST`
*   `value`: `None` for ALL, `Dict` for CONFIG_JSON, `List[int]` for ID_LIST

**Parsing Rules:**

1.   Empty list or `["all"]` → `ResetType.ALL` with `value=None`
2.   Single argument that is valid file path → `ResetType.CONFIG_JSON` with parsed JSON dict
3.   Single argument with commas (e.g., "0,1,2") → `ResetType.ID_LIST` with list of ints
4.   Multiple arguments → `ResetType.ID_LIST` with list of ints

**Raises:**`SystemExit` if input cannot be parsed.

**Sources:**[tt_tools_common/reset_common/reset_utils.py 33-87](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L33-L87)

### ARC Message Types

ARC firmware messages used in reset operations:

| Constant | Value | Description |
| --- | --- | --- |
| `MSG_TRIGGER_SPI_COPY_LtoR` | `0x50` | Trigger SPI copy from left to right |
| `MSG_TYPE_ARC_STATE3` | `0xA3` | Enter A3 safe state (no pending regulator requests) |
| `MSG_TYPE_TRIGGER_RESET` | `0x56` | Trigger device reset |

**Usage:**

*   `arg0=3` with `MSG_TYPE_TRIGGER_RESET` triggers full M3/DMC reset
*   `arg0` omitted or other values trigger ASIC-only reset
