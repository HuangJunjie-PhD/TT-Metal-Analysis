This page documents tt-flash's support for Blackhole architecture chips, including the `BhChip` class implementation, supported board variants (GALAXY, NEBULA, P-series), boot filesystem handling, and Blackhole-specific firmware operations. For general chip abstraction concepts, see [TTChip Class Hierarchy](https://deepwiki.com/tenstorrent/tt-flash/6.1-ttchip-class-hierarchy). For flashing procedures common to all platforms, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages).

## Overview

The Blackhole platform represents the second generation of Tenstorrent AI accelerator chips supported by tt-flash. Blackhole chips use a significantly different firmware architecture compared to Wormhole, including a boot filesystem structure for firmware storage, telemetry-based version reporting, and cryptographic post-flash validation. The `BhChip` class ([tt_flash/chip.py 241-295](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L241-L295)) implements Blackhole-specific hardware communication and firmware management operations.

## BhChip Class Architecture

The `BhChip` class inherits from `TTChip` and provides Blackhole-specific implementations for firmware version retrieval and hardware identification. Unlike Wormhole chips, Blackhole always returns `0x0` for minimum firmware version ([tt_flash/chip.py 242-243](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L242-L243)) because all Blackhole firmware versions support version reporting through telemetry.

**Key Methods:**

*   `min_fw_version()` - Returns `0x0` (all BH firmware supports version retrieval)
*   `get_bundle_version()` - Dual-source version reading from telemetry (running) and boot_fs_table (SPI)
*   `get_asic_location()` - Determines left/right ASIC position for P300 boards
*   `__repr__()` - Returns string representation: `Blackhole[{interface_id}]`

**Sources:**[tt_flash/chip.py 241-295](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L241-L295)

## Supported Board Variants

Blackhole supports three major board families:

### GALAXY-1 Boards (UPI: 0xB)

Galaxy boards are deployed in large-scale systems with 32 Blackhole chips connected via Ethernet. These boards require special reset procedures using IPMI commands and include Ethernet link validation after reset ([tt_flash/flash.py 700-736](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L736)). The reset procedure calls `glx_6u_trays_reset()` which differs from the Galaxy WH reset.

### NEBULA Series

| Board Type | UPI | Public Name | Special Features |
| --- | --- | --- | --- |
| NEBULA_X1 | 0x18 | n150 | Single-chip configuration |
| NEBULA_X2 | 0x14 | n300 | Dual-chip with local-to-remote SPI copy |
| NEBULA_CB | 0x8 | - | Carrier board configuration |

NEBULA_X2 boards require triggering a firmware-controlled local-to-remote SPI copy operation after flashing ([tt_flash/flash.py 619-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L619-L650)). This uses the `MSG_TRIGGER_SPI_COPY_LtoR` firmware message and includes a 15-second wait for completion.

### P-Series Boards

The P-series includes P100, P150, and P300 families with UPI codes ranging from 0x36 to 0x47. P300 boards contain two ASICs (left and right) that must be identified separately during flashing ([tt_flash/flash.py 830-837](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L830-L837)). The board name is suffixed with `_left` or `_right` based on ASIC location.

## Firmware Version Retrieval

Blackhole firmware version retrieval uses two distinct sources ([tt_flash/chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L248-L275)):

### Running Version (Telemetry-based)

The currently executing firmware version is read from telemetry using `telem.fw_bundle_version`. This 32-bit value is parsed as:

*   **Bits 24-31:** Component/fwId
*   **Bits 16-23:** Major version
*   **Bits 8-15:** Minor version
*   **Bits 0-7:** Patch version

### SPI Version (Boot Filesystem)

The firmware version stored in SPI flash is read using `luwen_chip.decode_boot_fs_table("cmfwcfg")`, which parses the boot filesystem to locate the `cmfwcfg` (Common Firmware Configuration) entry. The version is extracted from the `fw_bundle_version` field using the same bit layout.

### Exception Handling

Unlike Wormhole, Blackhole chips set `allow_exception=True` but do not permit version retrieval failures. If an exception occurs during version reading, it is captured in the `FwVersion.exception` field and will cause the flash operation to fail ([tt_flash/flash.py 243-246](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L243-L246)).

**Sources:**[tt_flash/chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L248-L275)[tt_flash/flash.py 223-246](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L223-L246)

## Boot Filesystem Structure

Blackhole uses a boot filesystem structure defined in [tt_flash/boot_fs.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py) to organize firmware components in SPI flash. The filesystem consists of a linked list of file descriptors (`tt_boot_fs_fd`), each describing a firmware component or configuration blob.

### File Descriptor Structure

Each `tt_boot_fs_fd` ([tt_flash/boot_fs.py 110-128](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py#L110-L128)) contains:

*   **spi_addr:** Offset in SPI flash where data is stored
*   **copy_dest:** Memory address to copy data to during boot
*   **flags:** 32-bit field containing image size (24 bits), invalid flag (1 bit), executable flag (1 bit)
*   **data_crc:** CRC32 checksum of the data
*   **security_flags:** Signature size and secure boot phase information
*   **image_tag:** 8-byte ASCII tag identifying the component (e.g., "cmfwcfg", "boardcfg")
*   **fd_crc:** CRC32 checksum of the file descriptor itself

### Tag Reading

The `read_tag()` function ([tt_flash/boot_fs.py 135-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py#L135-L148)) iterates through file descriptors until it finds a matching tag or encounters an invalid descriptor. This is used to locate specific firmware components like `cmfwcfg` for version information or `boardcfg` for board configuration.

**Sources:**[tt_flash/boot_fs.py 1-149](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py#L1-L149)[tt_flash/chip.py 263-269](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L263-L269)

## Blackhole Firmware Flashing Process

Blackhole firmware flashing differs from Wormhole in several key ways:

### Image Format Processing

Blackhole firmware images use hexadecimal format ([tt_flash/flash.py 378-393](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L378-L393)):

*   Lines starting with `@` specify the base address
*   Subsequent lines contain hex-encoded data (base16)
*   Multiple address/data sections are assembled into `FlashWrite` objects

### Boot Filesystem Handling

The `boot_fs_write()` function ([tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)) processes Blackhole-specific mask operations. Unlike Wormhole's parameter handlers that operate on byte ranges, Blackhole uses tag-based handlers that operate on entire boot filesystem entries.

### FlashWrite Data Structure

Blackhole uses a dedicated `FlashWrite` dataclass ([tt_flash/blackhole.py 13-16](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L13-L16)):

This structure is used consistently throughout the BH flash pipeline, simplifying the write and verify operations.

**Sources:**[tt_flash/flash.py 374-393](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L374-L393)[tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)

## Board Configuration Preservation

The `writeback_boardcfg()` handler ([tt_flash/blackhole.py 18-46](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L18-L46)) preserves board-specific configuration during firmware updates. This is critical because the `boardcfg` file descriptor contains calibration data, MAC addresses, and other board-unique settings that must not be overwritten.

### Preservation Process

1.   **Read Current Config:** Uses `boot_fs.read_tag()` to locate and read the `boardcfg` entry from the chip's current SPI flash
2.   **Locate New Config:** Searches through the `FlashWrite` objects to find the `boardcfg` entry in the firmware bundle
3.   **Replace Data:** Overwrites the new `boardcfg` data with the current board-specific data at the file descriptor offset
4.   **Verify:** Confirms the replacement was successful by reading the tag from the modified write buffer

## P300 ASIC Location Detection

| Detection Method | Priority | Implementation |
| --- | --- | --- |
| Telemetry | Primary | `telem.asic_location` from `get_telemetry()` |
| GPIO Strap | Fallback | Read `GPIO_STRAP_REG_L` (0x80030D20), check GPIO6 |

P300 boards contain two Blackhole ASICs mounted on a single carrier board. The flashing process must distinguish between the left and right ASICs because they may have different firmware requirements. The `get_asic_location()` method ([tt_flash/chip.py 277-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L277-L294)) implements a two-tier detection strategy:

### Primary: Telemetry-based Detection

The preferred method reads `telem.asic_location` from the chip's telemetry data. This value is:

*   **0:** Right ASIC
*   **1:** Left ASIC

### Fallback: GPIO Strap Register

If telemetry retrieval fails (due to firmware issues or communication errors), the code falls back to reading the GPIO strap register at `0x80030D20`. Bit 6 of this register indicates the ASIC position:

*   **GPIO6 = 1:** Left ASIC
*   **GPIO6 = 0:** Right ASIC

### Board Name Suffix

During the flash verification stage, P300 boards have their board name modified ([tt_flash/flash.py 830-837](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L830-L837)):

This allows the firmware bundle to contain separate images for left and right ASICs if necessary.

**Sources:**[tt_flash/chip.py 277-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L277-L294)[tt_flash/flash.py 830-837](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L830-L837)

## Reset Procedures

Blackhole chips support two reset methods depending on board configuration:

### Standard Blackhole Reset

Most Blackhole boards use `BHChipReset().full_lds_reset()` ([tt_flash/flash.py 932-936](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L932-L936)) with the following parameters:

*   **pci_interfaces:** List of PCI interface IDs to reset
*   **reset_m3:** Always `True` to reset the M3 management processor
*   **m3_delay:** Boot timeout period 
    *   **20 seconds:** Standard delay for same major version updates
    *   **60 seconds:** Extended delay when crossing major version boundaries ([tt_flash/flash.py 893-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L893-L900))

### Galaxy Blackhole Reset

GALAXY-1 boards in 32-device configurations use the same `glx_6u_trays_reset()` procedure as Galaxy Wormhole systems, but skip conventional BH reset afterward ([tt_flash/flash.py 915-920](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L915-L920)).

### Adaptive M3 Delay

The M3 boot delay is adaptive based on firmware version transitions ([tt_flash/flash.py 893-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L893-L900)):

## Post-Flash Validation

Blackhole firmware version 19 and later includes cryptographic post-flash validation using the `MSG_CONFIRM_FLASHED_SPI` message ([tt_flash/flash.py 941-952](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L941-L952)[tt_flash/data/blackhole/fw_defines.yaml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L6-L6)):

### Challenge-Response Protocol

1.   **Generate Challenge:** tt-flash generates a random 16-bit value (1 to 0xFFFF)
2.   **Send Message:** Calls `arc_msg(MSG_CONFIRM_FLASHED_SPI, arg0=check_val)`
3.   **Verify Response:** Checks that `(response[0] & 0xFFFF) == check_val`

### Validation Behavior

The validation confirms that:

*   The new firmware is loaded into memory
*   The ARC processor is responsive
*   The firmware matches what was written to SPI

### Error Handling

If validation fails, tt-flash prints a yellow warning message but does not halt the process:

```
WARNING: Post flash check failed for chip {idx}
Try resetting the board to ensure the new firmware is loaded correctly.
```

This non-blocking behavior is intentional because the firmware may still be functional even if the validation message fails (e.g., due to timing issues during the transition period).

### Version Requirement

The validation is only performed when `manifest.bundle_version[0] >= 19`, ensuring compatibility with older firmware that doesn't support this message type.

**Sources:**[tt_flash/flash.py 941-952](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L941-L952)[tt_flash/data/blackhole/fw_defines.yaml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L6-L6)

## Blackhole Firmware Messages

| Message Type | Hex Code | Purpose | Usage |
| --- | --- | --- | --- |
| MSG_TYPE_ARC_STATE0 | 0xa0 | Query ARC processor state | Pre-flash status check |
| MSG_TYPE_ARC_STATE3 | 0xa3 | Extended ARC state query | Used in flash_chip_stage1 |
| MSG_CONFIRM_FLASHED_SPI | 0xc4 | Post-flash validation | Challenge-response verification (v19+) |

Blackhole firmware messages are defined in [tt_flash/data/blackhole/fw_defines.yaml 1-7](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/data/blackhole/fw_defines.yaml#L1-L7) These messages are significantly different from Wormhole's message set:

### MSG_TYPE_ARC_STATE3 (0xa3)

Called at the start of `flash_chip_stage1()` ([tt_flash/flash.py 216-221](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L216-L221)) to ensure the ARC processor is responsive. The function tolerates timeouts and continues the flash process.

### MSG_CONFIRM_FLASHED_SPI (0xc4)

Introduced in firmware v19, this message implements a cryptographic challenge-response protocol to verify that newly flashed firmware is loaded correctly. See [Post-Flash Validation](https://github.com/tenstorrent/tt-flash/blob/60c06032/Post-Flash%20Validation) for details.

### Version Message Differences

Unlike Wormhole which uses `MSG_TYPE_FW_VERSION` with arg0/arg1 parameters to select running vs. SPI versions, Blackhole retrieves versions through telemetry and boot filesystem parsing, eliminating the need for version-specific ARC messages.
