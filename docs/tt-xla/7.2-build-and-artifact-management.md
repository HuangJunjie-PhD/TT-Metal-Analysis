This page covers the `call-build.yml` reusable workflow: how it produces Python wheels, how artifact names are determined, how existing artifacts are reused across runs, and how consuming workflows (test jobs) download and install the outputs.

For information about the CMake build system and Python packaging internals, see page 3.1 and 3.2. For Docker image construction, see page 3.3. For test job execution using these artifacts, see page 7.3.

## Artifact Types and Organization

The CI/CD system produces and manages four primary artifact types:

| Artifact Type | GitHub Artifact Name Pattern | Produced By | Used By |
| --- | --- | --- | --- |
| PJRT plugin wheel | `xla-whl-{build_type}[-{short_sha}]` | `build-ttxla` job | All test jobs |
| vLLM plugin wheel | `vllm-tt-whl-{build_type}[-{short_sha}]` | `build-ttxla` job | vLLM test jobs only |
| Alchemist library | `alchemist-lib-{build_type}[-{short_sha}]` | `build-ttxla` job | Codegen test jobs |
| Test reports | `test-reports-{matrix_hash}-{attempt}.{index}-*` | `run-tests` jobs | `test-summary` job |

The `{short_sha}` suffix is only appended when the artifact can be safely reused across workflow runs (see [Artifact Caching Logic](https://github.com/tenstorrent/tt-xla/blob/c77995f6/Artifact%20Caching%20Logic) below).

### Wheel Package Layout

The PJRT plugin wheel (`pjrt-plugin-tt-*.whl`) bundles the plugin binary together with all runtime dependencies. Its internal structure, as described in `SetupConfig`, is:

```
pjrt_plugin_tt/
    __init__.py
    pjrt_plugin_tt.so         # PJRT C API plugin binary
    tt-metal/                 # Kernels, RISC-V compiler/linker, etc.
    lib/                      # Shared library dependencies (tt-mlir, tt-metal)
jax_plugin_tt/
    __init__.py               # Registers pjrt_plugin_tt with JAX
torch_plugin_tt/
    __init__.py               # Registers pjrt_plugin_tt with PyTorch/XLA
tracy/                        # Wrapped Tracy profiler from tt-metal
ttnn/                         # Wrapped TTNN module from tt-metal
```

The alchemist artifact contains two paths needed for codegen-backend tests:

*   `lib/libtt-alchemist-lib.so`
*   `tools/tt-alchemist/templates`

## The `call-build.yml` Workflow

`call-build.yml` is the central reusable workflow responsible for compiling tt-xla and producing all distribution artifacts. It is called by orchestrating workflows such as `schedule-nightly.yml`, `manual-test.yml`, and the PR workflow.

### Inputs and Outputs

| Parameter | Direction | Type | Description |
| --- | --- | --- | --- |
| `mlir_override` | input | string | Optional git SHA of a `tenstorrent/tt-mlir` commit to substitute into `third_party/CMakeLists.txt` before building |
| `docker_image` | input | string | Docker image to run the build container in |
| `build_type` | input | string | `release` (default) or `explorer` |
| `wheel_artifact_name` | output | string | Name of the uploaded PJRT wheel artifact |
| `wheel_release_vllm_tt_artifact_name` | output | string | Name of the uploaded vLLM wheel artifact |
| `alchemist_artifact_name` | output | string | Name of the uploaded alchemist library artifact |
| `artifacts_run_id` | output | string | GitHub run ID where artifacts were uploaded (may differ from the current run if a cached artifact was reused) |

Sources: [.github/workflows/call-build.yml 7-35](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L7-L35)

### Jobs in `call-build.yml`

The workflow has two jobs:

1.   **`check-existing-artifact`** — Determines artifact names and checks whether a matching artifact already exists in the repository. If it finds one, it short-circuits the build by outputting its `run_id`.
2.   **`build-ttxla`** — Runs only when `check-existing-artifact` found no cached artifact. It builds the wheel using `CMakeBuildPy` and uploads all artifacts.

**`call-build.yml` job dependency and artifact flow**

Sources: [.github/workflows/call-build.yml 37-197](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L37-L197)

### `mlir_override` Mechanism

When `mlir_override` is set, a `sed` command replaces the `TT_MLIR_VERSION` line in `third_party/CMakeLists.txt` before the CMake build runs:

```
sed -i "s/set(TT_MLIR_VERSION \".*\")/set(TT_MLIR_VERSION \"{mlir_override}\")/" third_party/CMakeLists.txt
```

This causes the `ExternalProject_Add` in the CMake build to fetch the overridden commit of `tt-mlir` instead of the version pinned in the repository. The same mechanism is forwarded to `call-build-docker.yml` so that the Docker image uses a matching base.

Sources: [.github/workflows/call-build.yml 124-129](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L124-L129)[.github/workflows/manual-test.yml 36-59](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/manual-test.yml#L36-L59)

### Wheel Build Internals

The wheel is built by running `python setup.py bdist_wheel --build-type {build_type}` inside `python_package/`. Two custom setuptools commands drive the process:

| Class | Base Class | Role |
| --- | --- | --- |
| `BdistWheel` | `bdist_wheel` | Sets `root_is_pure = False`, forces `cp312-cp312` ABI tag, accepts `--build-type` option |
| `CMakeBuildPy` | `build_py` | Invokes CMake configure/build/install, then calls `_prune_install_tree` |

After the CMake install, `_prune_install_tree` runs several cleanup passes on the install directory:

| Function | When | Effect |
| --- | --- | --- |
| `_remove_broken_symlinks` | Always | Removes broken symlinks from the tt-umd/tt-metal dependency chain |
| `_remove_static_archives` | Always | Removes `lib/*.a` archives to reduce wheel size |
| `_strip_shared_objects` | Release builds only | Runs `strip --strip-unneeded` on all `.so` files |
| `_deduplicate_shared_objects` | Always | Replaces duplicate `.so` files (by SHA-256) with symlinks |

Sources: [python_package/setup.py 197-408](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/setup.py#L197-L408)

## Artifact Naming and Versioning

### Build Artifact Names

Build artifact names are computed in the `generate-name` step of `check-existing-artifact`:

| Artifact | Name when reusable | Name for PR / mlir_override |
| --- | --- | --- |
| PJRT wheel | `xla-whl-{build_type}-{short_sha}` | `xla-whl-{build_type}` |
| vLLM wheel | `vllm-tt-whl-{build_type}-{short_sha}` | `vllm-tt-whl-{build_type}` |
| Alchemist lib | `alchemist-lib-{build_type}-{short_sha}` | `alchemist-lib-{build_type}` |

The `{short_sha}` suffix is included only when all three conditions hold:

*   The triggering event is **not** a pull request
*   The commit has **exactly one parent** (i.e., is not a merge commit)
*   `mlir_override` is **not** set

Sources: [.github/workflows/call-build.yml 52-74](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L52-L74)

### Wheel Package Version String

The wheel version is computed by `SetupConfig.version` in `python_package/setup.py`:

```
0.1.{YYMMDD}+dev.{short_sha}
```

The wheel's `metadata.summary` field is populated by `SetupConfig.description_with_versions` and encodes the full build provenance:

```
commit={tt_xla_sha}, tt-mlir-commit={mlir_sha}, tt-metal-commit={metal_sha}, built-date={date}, build-type={build_type}
```

The `build-ttxla` job validates this field after packaging to verify the `build_type` matches the workflow input.

Sources: [python_package/setup.py 51-67](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/setup.py#L51-L67)[python_package/setup.py 95-144](https://github.com/tenstorrent/tt-xla/blob/c77995f6/python_package/setup.py#L95-L144)[.github/workflows/call-build.yml 152-164](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L152-L164)

### Test Report Artifact Names

Test artifacts use component-rich names to uniquely identify each parallel job and run attempt:

```
test-reports-{test_matrix_hash}-{run_attempt}.{job_index}-{runs-on}-{sanitized_test_mark}-{job_id}
test-log-{runs-on}-{sanitized_test_mark}-{job_id}
perf-reports-{test_matrix_hash}-{run_attempt}.{job_index}-{runs-on}-{sanitized_test_mark}-{job_id}
```

| Component | Source |
| --- | --- |
| `test_matrix_hash` | MD5 of the expanded test matrix JSON (from `call-generate-matrix.yml`) |
| `run_attempt` | `github.run_attempt` |
| `job_index` | `strategy.job-index` |
| `runs-on` | Runner label (e.g., `n150`, `n300`) |
| `sanitized_test_mark` | `matrix.build.test-mark` with spaces replaced by `_` |
| `job_id` | Fetched via `tenstorrent/tt-github-actions/.github/actions/job_id@main` |

The sanitization step (`echo "${TEST_MARK// /_}"`) prevents artifact name failures caused by spaces.

## Artifact Caching Logic

The `check-existing-artifact` job in `call-build.yml` avoids redundant builds for commits that have already been compiled. The logic is:

**Conditions for a reusable artifact (SHA suffix added to name):**

*   `github.event_name != 'pull_request'`
*   The commit has exactly one parent (not a merge commit, checked via `git cat-file -p HEAD | grep -c "^parent "`)
*   `inputs.mlir_override == ''`

When all three conditions hold, `reusable_artifact=true` is set and the GitHub REST API is queried for an artifact with the computed name. If found, `artifacts_run_id` is set to the existing run's ID and the `build-ttxla` job is skipped entirely via its `if: ${{ !needs.check-existing-artifact.outputs.artifacts_run_id }}` condition.

This means PR builds and builds with `mlir_override` always rebuild from scratch, while push-to-main and nightly builds on individual commits can reuse a previously uploaded artifact.

Sources: [.github/workflows/call-build.yml 37-101](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L37-L101)[.github/workflows/call-build.yml 102-104](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L102-L104)

## Compiler Cache (`ccache`)

The `build-ttxla` job uses `hendrikmuhs/ccache-action@v1.2` to cache compiled C++ objects between workflow runs. The cache key is `{build_type}-build`, so `release-build` and `explorer-build` maintain separate caches.

Sources: [.github/workflows/call-build.yml 139-143](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L139-L143)

## Runner-Side Model and Framework Caches

Test jobs use two cache strategies depending on runner type:

| Runner Type | Cache Variable | Cache Root |
| --- | --- | --- |
| CIv1 (dedicated) | `DOCKER_CACHE_ROOT` | `/mnt/dockercache` |
| CIv2 (shared) | `IRD_LF_CACHE` | `vars.IRD_LF_CACHE` |

Model weights are cached at `HF_HOME=/mnt/dockercache/huggingface` and `TORCH_HOME=/mnt/dockercache/torchhub` on dedicated runners.

On CIv2 shared runners, the `cleanup_cache_fixture` autouse fixture in `tests/conftest.py` removes a fixed set of cache directories before and after each test to prevent disk exhaustion:

```
CACHE_DIRECTORIES = [
    ~/.cache/lfcache, ~/.cache/url_cache,
    /mnt/dockercache/huggingface, ~/.cache/huggingface, /tmp/huggingface,
    ~/.cache/jax, ~/.cache/jaxlib,
    /tmp/torchinductor_{USER}
]
```

The `_is_on_CIv2()` helper checks `TT_XLA_CI=1` and the absence of `DOCKER_CACHE_ROOT` to detect CIv2 and enable cleanup.

When a workflow run is re-triggered (i.e., `github.run_attempt > 1`), the `check-rerun-attempt` step downloads JUnit XML reports from the previous attempt to determine which tests failed. This avoids re-running tests that already passed.

The `pytest_collection_modifyitems` hook in `tests/conftest.py` applies the filter: if `.pytest_tests_to_run` exists, only tests whose `nodeid` appears in that file are collected.

## Wheel Validation

After packaging, the `build-ttxla` job runs a validation step to confirm the `build-type` metadata is correct before uploading the artifact. It uses `wheel-inspect` (`wheel2json`) to parse the wheel metadata without installing it:

```
wheel2json pjrt_plugin_tt*.whl | jq -r '.dist_info.metadata.summary'
```

The step asserts that the summary string contains `build-type={build_type}`. A mismatch causes the job to fail before any upload occurs.

Sources: [.github/workflows/call-build.yml 152-164](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-build.yml#L152-L164)

## Test Report Aggregation

The `test-summary` job in `call-test.yml` collects JUnit XML reports from all parallel `run-tests` jobs and renders a consolidated GitHub check.

**Test report aggregation flow**

The `--pattern test-reports-{test_matrix_hash}-*` glob ensures only reports from the current matrix configuration are included, even if previous runs' artifacts are still present. The `include_passed` input to `call-test.yml` controls whether the JUnit report shows passed tests.

Sources: [.github/workflows/call-test.yml 382-408](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/call-test.yml#L382-L408)

## Artifact Lifecycle

GitHub Actions artifacts have a default retention period of 90 days, after which they are automatically deleted. Build artifacts can be reused across multiple test runs within this period.
