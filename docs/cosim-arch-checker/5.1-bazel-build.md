This document covers the Bazel build system for the cosim-arch-checker framework, specifically focusing on the `dpi` target that creates the shared library used for DUT-ISS lockstep verification. The Bazel build handles external dependencies, compiler configuration, and integration with the Chipyard/Boom environment.

For the alternative Makefile-based build approach, see [Makefile Build](https://deepwiki.com/tenstorrent/cosim-arch-checker/5.2-makefile-build). For DUT-specific configuration parameters, see [DUT Parameters](https://deepwiki.com/tenstorrent/cosim-arch-checker/4.2-dut-parameters).

## Build Target Overview

The primary Bazel target is `dpi`, which creates a shared library containing all components needed for architectural verification. This target compiles the Bridge system, DUT monitoring interface, Whisper client, and integrates with the external Core Arch Checker.

## Integration with Chipyard Environment

The Bazel build integrates with the Chipyard simulation environment through the generated shared library. The build output is consumed by VCS or Verilator testbenches.

### Chipyard Integration Flow

### Required Runtime Environment

The Bazel-built shared library requires these runtime components:

*   **Whisper executable**: Referenced by `WHISPER_PATH` environment variable
*   **Whisper configuration**: JSON files in `bridge/whisper/config/`
*   **Test binary**: ELF file to execute on DUT
*   **Boot ROM**: Initialization code for system startup

## Build Performance and Caching

Bazel provides incremental compilation and caching benefits for the cosim-arch-checker build:

*   **Incremental builds**: Only modified source files are recompiled
*   **External dependency caching**: Whisper and googletest repositories are cached
*   **Build artifact reuse**: Shared library is only regenerated when dependencies change

This is particularly beneficial during development when making frequent changes to the Bridge system or DUT monitoring interface while keeping external dependencies stable.
