# Tool Base Class

Relevant source files
*   [riescue/lib/toolchain/tool.py](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py)

This document describes the `Tool` abstract base class that provides a unified interface for executing external RISC-V toolchain executables. The `Tool` class handles executable discovery, subprocess management, and error classification for all external tools used by RiESCUE.

For detailed documentation of specific tool implementations (Compiler, Disassembler, Objcopy, Spike), see [Tool Subclasses](https://deepwiki.com/tenstorrent/riescue/7.3.2-tool-subclasses). For broader context on toolchain integration, see [Toolchain and External Dependencies](https://deepwiki.com/tenstorrent/riescue/4-toolchain-and-external-dependencies).

## Overview

The `Tool` class is an abstract base class (ABC) defined in [riescue/lib/toolchain/tool.py 18-126](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L18-L126) that encapsulates the logic for locating, configuring, and executing external RISC-V toolchain binaries. It provides three core capabilities:

1.   **Executable Discovery**: Multi-stage search for executables using explicit paths, environment variables, or PATH lookup
2.   **Subprocess Execution**: Standardized execution with output redirection, timeout handling, and working directory control
3.   **Error Classification**: Abstract error handling interface that subclasses implement to provide tool-specific failure diagnostics

The class serves as the foundation for all toolchain tools including the compiler (GCC), disassembler (objdump), objcopy, and instruction set simulators (Spike).

**Sources:**[riescue/lib/toolchain/tool.py 18-126](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L18-L126)

## Tool Base Class Architecture

### Initialization and Configuration

The `Tool` constructor accepts four parameters that control executable location and default arguments:

| Parameter | Type | Description |
| --- | --- | --- |
| `path` | `Optional[Path]` | Explicit path to executable (highest priority) |
| `env_name` | `str` | Environment variable name for executable path |
| `tool_name` | `str` | Executable name to search in PATH (lowest priority) |
| `args` | `Optional[list[str]]` | Default arguments always passed to executable |

During initialization at [riescue/lib/toolchain/tool.py 31-34](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L31-L34) the class:

1.   Stores default arguments in `self.args`
2.   Locates the executable via `find_executable()` and resolves it to an absolute path
3.   Stores the resolved path in `self.executable`
4.   Logs the discovered executable path for debugging

The class also defines a `package_path` class variable at [riescue/lib/toolchain/tool.py 29](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L29-L29) that points to the `riescue/` directory, used for locating package-relative data files.

**Sources:**[riescue/lib/toolchain/tool.py 29-34](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L29-L34)

## Command Execution

### Run Method

The `run()` method at [riescue/lib/toolchain/tool.py 74-108](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L74-L108) executes the tool with standardized subprocess handling:

**Method Signature:**

`def run(    self,     output_file: Optional[Path] = None,    cwd: Optional[Path] = None,    timeout: int = 90,    args: Optional[list[str]] = None) -> subprocess.CompletedProcess`
**Parameters:**

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `output_file` | `Optional[Path]` | `None` | If provided, stdout is redirected to this file; stderr still goes to stdout |
| `cwd` | `Optional[Path]` | `None` | Working directory for command execution |
| `timeout` | `int` | `90` | Command timeout in seconds |
| `args` | `Optional[list[str]]` | `None` | Additional arguments appended to command |

**Execution Flow:**

**Key Behaviors:**

1.   **Command Construction**[riescue/lib/toolchain/tool.py 87](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L87-L87): Combines executable path, instance default args, and runtime args
2.   **Output Redirection**[riescue/lib/toolchain/tool.py 90-98](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L90-L98): When `output_file` is provided, both stdout and stderr are written to the file
3.   **Segfault Detection**[riescue/lib/toolchain/tool.py 100-101](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L100-L101): Return code 139 indicates segmentation fault and raises `ToolchainError` with `SEGFAULT` type
4.   **Error Classification**[riescue/lib/toolchain/tool.py 103](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L103-L103): Delegates tool-specific error handling to abstract `_classify()` method
5.   **Stderr Logging**[riescue/lib/toolchain/tool.py 104-107](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L104-L107): Non-fatal stderr output is logged as warnings

## Error Handling

### Error Classification Pattern

The `Tool` class implements a Template Method pattern for error handling. The base class handles common errors (segfaults) while delegating tool-specific error interpretation to subclasses:

**Abstract Method:**

`@abstractmethoddef _classify(self, process: subprocess.CompletedProcess, output_file: Optional[Path]):    """    Check the return code of the subprocess run, raises meaningful ToolchainError if tool failed to run.    """    pass`
This method is called after every subprocess execution at [riescue/lib/toolchain/tool.py 103](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L103-L103) and must be implemented by all concrete tool subclasses.

**Sources:**[riescue/lib/toolchain/tool.py 116-125](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L116-L125)

### ToolchainError Construction

The `_raise_toolchain_error()` helper at [riescue/lib/toolchain/tool.py 110-114](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L110-L114) provides a standardized way to raise `ToolchainError` exceptions:

`def _raise_toolchain_error(    self,     process: subprocess.CompletedProcess,     kind: ToolFailureType,     error_text: Optional[str] = None,     **kwargs):`
This method automatically populates:

*   `tool_name`: Class name of the tool (e.g., "Compiler", "Spike")
*   `cmd`: Full command that was executed (from `process.args`)
*   `kind`: Type of failure from `ToolFailureType` enum
*   `returncode`: Process exit code
*   `error_text`: Optional error message
*   Additional keyword arguments passed through to `ToolchainError`

The `ToolFailureType` enum (defined in `riescue/lib/toolchain/exceptions.py`) includes values like:

*   `SEGFAULT`: Segmentation fault (handled by base `Tool` class)
*   `COMPILE_FAILURE`: Compilation or assembly error
*   `NONZERO_EXIT`: Generic non-zero exit code
*   Others as needed by specific tools

**Sources:**[riescue/lib/toolchain/tool.py 110-114](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L110-L114)[riescue/lib/toolchain/exceptions.py](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/exceptions.py)

### Common Patterns in Subclasses

All concrete tool implementations follow these patterns:

1.   **Constructor**: Builds tool-specific default arguments and calls `super().__init__()`
2.   **Static `add_arguments()` method**: Defines argparse arguments for CLI integration
3.   **Class method `from_clargs()`**: Factory method to construct tool from parsed CLI arguments
4.   **`_classify()` implementation**: Tool-specific error detection and `ToolchainError` raising

## Integration with RiESCUE Framework

### Usage in Test Generation

The `Tool` classes are instantiated by higher-level components in the RiESCUE framework:

The typical flow:

1.   Command-line interface calls `Tool.add_arguments()` to define CLI options
2.   After parsing, calls `Tool.from_clargs()` to construct tool instance
3.   Test generator invokes `tool.run()` with test-specific arguments
4.   Tool handles execution and error classification automatically

**Sources:**[riescue/lib/toolchain/tool.py 169-230](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/lib/toolchain/tool.py#L169-L230)

### Relationship to Toolchain Class

The `Tool` implementations are typically managed by a `Toolchain` class (documented in [Toolchain Abstraction Layer](https://deepwiki.com/tenstorrent/riescue/7.3-toolchain-abstraction-layer)) that provides:

*   Coordinated initialization of all tools
*   Shared configuration (march, ISA strings)
*   High-level operations that combine multiple tools

The `Tool` base class focuses on individual executable invocation while `Toolchain` orchestrates multi-tool workflows.
