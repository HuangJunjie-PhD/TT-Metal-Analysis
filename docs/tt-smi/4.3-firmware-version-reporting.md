# Firmware Version Reporting

Relevant source files
*   [tt_smi/constants.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py)
*   [tt_smi/tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py)

## Purpose and Scope

This document describes how TT-SMI collects, processes, and displays firmware version information from Tenstorrent devices. It covers the firmware components tracked on each device, the data collection pipeline from hardware telemetry to UI display, version conversion utilities, and platform-specific considerations.

For information about the interactive TUI tab that displays firmware versions, see [TUI Firmware Version Tab](https://deepwiki.com/tenstorrent/tt-smi/3.4-firmware-version-tab). For details on telemetry collection mechanisms, see [Telemetry Collection and Monitoring](https://deepwiki.com/tenstorrent/tt-smi/4.2-telemetry-collection-and-monitoring).

* * *

## Firmware Components Overview

TT-SMI tracks seven distinct firmware components across Tenstorrent devices. Each component serves a specific role in the hardware stack:

| Firmware Component | Field Name | Description | Telemetry Source |
| --- | --- | --- | --- |

| **TT-Flash Version** | `tt_flash_version` | Flash programming utility version | `TT_FLASH_VERSION` |
| **CM FW Version** | `cm_fw` | Chip Management (ARC0) firmware version | `ARC0_FW_VERSION` |
| **CM FW Date** | `cm_fw_date` | Chip Management firmware build date | `WH_FW_DATE` |
| **ETH FW Version** | `eth_fw` | Ethernet firmware version | `ETH_FW_VERSION` |
| **BM BL Version** | `bm_bl_fw` | Board Management bootloader (M3) version | `M3_BL_FW_VERSION` |
| **BM App Version** | `bm_app_fw` | Board Management application (M3) version | `M3_APP_FW_VERSION` |

## Firmware Version Retrieval Implementation

### Primary Collection Method

The `get_firmware_versions()` method in `TTSMIBackend` is responsible for extracting and converting all firmware version information for a specific board.

**Method signature:**

`def get_firmware_versions(self, board_num) -> dict`
**Location:**[tt_smi/tt_smi_backend.py 588-649](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L588-L649)

The method performs the following operations:

1.   **Reads raw telemetry data** from the pre-collected `smbus_telem_info[board_num]` dictionary
2.   **Applies appropriate conversion** based on firmware type: 
    *   M3 firmware versions use `hex_to_semver_m3_fw()`
    *   Ethernet firmware versions use `hex_to_semver_eth()`
    *   Firmware dates use `hex_to_date()`

3.   **Handles missing data** by setting fields to `"N/A"`
4.   **Returns a dictionary** with keys matching `constants.FW_LIST`

### Telemetry Source Collection

Before firmware versions can be retrieved, the raw telemetry structure must be collected via `get_smbus_board_info()`:

**Location:**[tt_smi/tt_smi_backend.py 241-256](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L241-L256)

This method:

*   Calls `pyluwen_chip.as_wh().get_telemetry()` or `pyluwen_chip.as_bh().get_telemetry()`
*   Converts the telemetry structure to a dictionary using `dict_from_public_attrs()`
*   Populates `smbus_telem_info[board_num]` with hex-encoded values

**Sources:**[tt_smi/tt_smi_backend.py 241-256](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L241-L256)[tt_smi/tt_smi_backend.py 588-649](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L588-L649)[tt_smi/tt_smi_backend.py 713-721](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L713-L721)

## Platform-Specific Firmware Handling

### Galaxy System Firmware Bundle Workaround

Galaxy 6U systems (board type `"wh_4u"`) have a known firmware flashing bug that requires special handling for the `fw_bundle_version` field.

**Issue:** On Galaxy systems, the `FW_BUNDLE_VERSION` telemetry field may contain `0xffffffff` (invalid value) due to a firmware flashing bug.

**Workaround:** When this condition is detected, TT-SMI reads the firmware bundle version from the `TT_FLASH_VERSION` field instead:

`# Location: tt_smi/tt_smi_backend.py:633-644if (    get_board_type(self.get_board_id(board_num)) == "wh_4u"    and val == "0xffffffff"):    # Workaround: Read from TT_FLASH_VERSION instead    val = self.smbus_telem_info[board_num]["TT_FLASH_VERSION"]    fw_versions[field] = hex_to_semver_m3_fw(int(val, 16))`
**Consequence:** On Galaxy systems, the `tt_flash_version` field is manually set to `"N/A"` since its value has been repurposed for the bundle version:

`# Location: tt_smi/tt_smi_backend.py:629-630elif get_board_type(self.get_board_id(board_num)) == "wh_4u":    fw_versions[field] = "N/A"`
**Related Issue:**[GitHub Issue #72](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/GitHub%20Issue%20#72)

**Sources:**[tt_smi/tt_smi_backend.py 633-648](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L633-L648)[tt_smi/tt_smi_backend.py 625-632](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L625-L632)

## Firmware Data Initialization

During `TTSMIBackend` initialization, firmware versions are collected for all detected devices:

**Initialization sequence:**

1.   For each device index `i` in `self.devices`: 
    *   Call `get_smbus_board_info(i)` to populate raw telemetry → stored in `smbus_telem_info[i]`
    *   Call `get_firmware_versions(i)` to extract and convert firmware data → stored in `firmware_infos[i]`
    *   Progress is displayed using `rich.progress.track()` (disabled in non-pretty mode)

**Sources:**[tt_smi/tt_smi_backend.py 53-96](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L53-L96)[tt_smi/tt_smi_backend.py 84-96](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L84-L96)

## Firmware Display in TUI

The firmware version information is displayed in **Tab 3** of the TT-SMI TUI interface. The table uses the header defined in `constants.FIRMWARES_TABLE_HEADER`:

| Column Header | Source Field | Description |
| --- | --- | --- |
| # | Board index | Sequential device number |
| FW Bundle Version | `fw_bundle_version` | Overall firmware package |
| TT-Flash Version | `tt_flash_version` | Flash utility version |
| CM FW Version | `cm_fw` | ARC0 (Chip Management) firmware |
| CM FW Date | `cm_fw_date` | Build date of CM firmware |
| ETH FW Version | `eth_fw` | Ethernet core firmware |
| BM BL Version | `bm_bl_fw` | M3 bootloader version |
| BM App Version | `bm_app_fw` | M3 application version |

Unlike the telemetry tab, the firmware tab is **static** and does not auto-refresh, as firmware versions do not change during runtime.

**Sources:**[tt_smi/constants.py 189-198](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L189-L198)

## Missing or Invalid Firmware Data

TT-SMI handles missing or invalid firmware data gracefully:

### Null/None Values

If a telemetry field is `None`, the corresponding firmware field is set to `"N/A"`:

`# Example from tt_smi/tt_smi_backend.py:593-597if val is None:    fw_versions[field] = "N/A"else:    fw_versions[field] = hex_to_semver_m3_fw(int(val, 16))`
### Firmware Not Loaded

If the ARC firmware has not been loaded on the device, the telemetry structure will contain default or zero values. These are converted to `"N/A"` strings, allowing the UI to display a consistent state rather than showing malformed version numbers.

**Sources:**[tt_smi/tt_smi_backend.py 593-597](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L593-L597)[tt_smi/tt_smi_backend.py 607-611](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L607-L611)
