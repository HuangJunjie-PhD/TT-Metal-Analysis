This document provides a detailed overview of the backend architecture of the TTNN Visualizer, focusing on the Flask application structure, API routes, database query systems, session management, and remote data access capabilities.

The backend serves as the data processing and API layer of the TTNN Visualizer, handling profiler data access, query execution, and communication between the frontend and various data sources.

## Overview

The TTNN Visualizer backend is a Flask application that provides RESTful API endpoints for accessing and analyzing TT-NN model profiling data. The backend abstracts the complexities of data access, supporting both local SQLite database querying and remote data access via SSH/SFTP.

## Database Query System

The database query system abstracts access to SQLite databases, supporting both local and remote access. The `DatabaseQueries` class provides a consistent interface regardless of whether the database is local or remote.

**Database Query System Architecture**

Sources:

*   [backend/ttnn_visualizer/queries.py 166-189](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/backend/ttnn_visualizer/queries.py#L166-L189)
*   [backend/ttnn_visualizer/queries.py 33-65](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/backend/ttnn_visualizer/queries.py#L33-L65)
*   [backend/ttnn_visualizer/queries.py 66-164](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/backend/ttnn_visualizer/queries.py#L66-L164)
*   [backend/ttnn_visualizer/models.py 30-157](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/backend/ttnn_visualizer/models.py#L30-L157)

The query system consists of three main components:

1.   **DatabaseQueries**: High-level class that provides methods for querying specific entities and abstracts the underlying query execution.

2.   **Query Runners**: Implement the actual query execution, with specialized versions for local and remote databases:

    *   `LocalQueryRunner`: Executes SQLite queries directly on a local database file
    *   `RemoteQueryRunner`: Executes SQLite queries through SSH commands on a remote database

3.   **Entity Models**: Data classes that represent the query results:

    *   `Operation`: Represents a TT-NN operation with properties like ID, name, and duration
    *   `Tensor`: Represents a tensor with properties like shape, dtype, and memory layout
    *   `Buffer`: Represents a memory buffer with properties like address and size
    *   `Device`: Represents a hardware device with properties for cores and memory

## Error Handling

The backend implements several error handling strategies:

1.   **Custom Exceptions**: Defined in `exceptions.py` for specific error scenarios
2.   **Route Error Handlers**: Registered in Flask to handle specific exceptions
3.   **Remote Exception Handler**: Specialized decorator for handling remote connectivity issues

## Performance Considerations

The backend addresses performance considerations through:

1.   **Lazy Loading**: Query generators yield results rather than loading everything in memory
2.   **Pagination**: Many endpoints support limiting result sets
3.   **Caching**: Session information caches paths and connection details
4.   **Connection Pooling**: SQLAlchemy engine uses connection pooling

## Integration with Frontend

The backend primarily communicates with the frontend through REST API endpoints, but also supports:

1.   **WebSockets**: Optional real-time updates (configurable)
2.   **Static File Serving**: In production mode, serves the React frontend
3.   **Session Tracking**: Maintains state across browser tabs through the instance ID

For more information about the frontend architecture, see [Frontend Architecture](https://deepwiki.com/tenstorrent/ttnn-visualizer/2.2-frontend-architecture).
