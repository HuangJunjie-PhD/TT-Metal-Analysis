## Core Workflow

Every usage of tt-forge-onnx follows the same two-phase pattern:

1.   **Compile** — `forge.compile` ingests a framework model, runs it through TVM/Relay and MLIR lowering, and returns a `CompiledModel` ready for hardware execution.
2.   **Verify** — `verify` runs both the original framework model and the compiled model on the same inputs and compares outputs numerically.

## Configuring the Compiler

`CompilerConfig` from `forge.config` controls compilation behaviour. Pass it as the `compiler_cfg` keyword argument to `forge.compile`.

| `CompileDepth` value | Pipeline stops after |
| --- | --- |
| `INIT_COMPILE` | Initialization |
| `SPLIT_GRAPH` | Graph splitting (before MLIR) |
| `FULL` | Full compilation to hardware binary |

## Standard Model Test Pattern

Model integration tests (under `forge/test/models/`) follow an extended pattern that adds property recording, a `ModelLoader`, and a `CompilerConfig`. This pattern feeds test metadata into the CI reporting system via `record_model_properties`.

`record_model_properties` is a context manager that attaches structured metadata (framework, architecture, task, source) to the test for tracking. If the test fails at any stage, the `ExecutionStage` enum records exactly where it failed — from TVM graph generation through MLIR compilation to hardware execution.

Sources: [forge/forge/forge_property_utils.py 38-238](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L38-L238)[forge/forge/forge_property_utils.py 279-307](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L279-L307)[forge/test/models/onnx/vision/ssdlite320_mobilenetv3/test_ssdlite320_mobilenetv3_onnx.py 1-40](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/models/onnx/vision/ssdlite320_mobilenetv3/test_ssdlite320_mobilenetv3_onnx.py#L1-L40)

## Verification Details

`verify` compares compiled model outputs to framework (CPU) outputs using `compare_tensor_to_golden`. The default metric is **PCC (Pearson Correlation Coefficient)** via `AutomaticValueChecker`. A PCC close to 1.0 indicates numerically equivalent outputs.

| Checker class | Metric |
| --- | --- |
| `AutomaticValueChecker` | PCC threshold (default) |
| `AllCloseValueChecker` | `torch.allclose` with atol/rtol |
| `FullValueChecker` | Exact equality |

## Execution Stages for Diagnostics

If compilation or verification fails, the `ExecutionStage` enum in `forge_property_utils.py` records where the failure occurred. This is primarily used in CI, but is useful to understand error messages.
