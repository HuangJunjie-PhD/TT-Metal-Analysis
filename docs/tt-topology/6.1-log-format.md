# Log Format

Relevant source files
*   [tt_topology/log.py](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py)

This document describes the structured logging format used by tt-topology for recording hardware configuration operations and diagnostic information. The logging system generates JSON-formatted logs that are compatible with Elasticsearch for storage and analysis.

For information about Elasticsearch integration and configuration, see [Elasticsearch Integration](https://deepwiki.com/tenstorrent/tt-topology/6.2-elasticsearch-integration).

## Purpose and Structure

The tt-topology logging system captures comprehensive information about each topology configuration operation, including system state before and after configuration, connection mappings, coordinate assignments, and any errors encountered. All logs are structured using Pydantic models to ensure consistency and type safety.

The logging system is implemented in [tt_topology/log.py 1-200](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L1-L200) and provides Elasticsearch-compatible data models for storing operational data.

## Core Log Entry Structure

The main log entry is represented by the `TTToplogyLog` class, which captures the complete state of a topology configuration operation:

| Field | Type | Description |
| --- | --- | --- |
| `time` | `datetime.datetime` | Timestamp of the operation |
| `host_info` | `HostInfo` | System information where operation was performed |
| `chip_layout` | `str` | Layout type (mesh, linear, torus, isolated) |
| `png_filename` | `str` | Path to generated visualization file |
| `starting_configs` | `List[ChipConfig]` | Initial chip configurations |
| `post_default_flashing_configs` | `List[ChipConfig]` | Configurations after default flashing |
| `connection_map` | `List[ConnectionMap]` | Physical connection topology |
| `coordinate_map` | `CoordinateMap` | Logical coordinate assignments |
| `final_coords_flash_config` | `List[ChipConfig]` | Final chip configurations |
| `errors` | `str` | Error messages encountered during operation |

**Sources**: [tt_topology/log.py 183-194](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L183-L194)

## Host Information Model

The `HostInfo` model captures system environment details:

| Field | Type | Description |
| --- | --- | --- |
| `OS` | `str` | Operating system name |
| `Distro` | `str` | Linux distribution |
| `Kernel` | `str` | Kernel version |
| `Hostname` | `str` | System hostname |
| `Platform` | `str` | Hardware platform |
| `Python` | `str` | Python version |
| `Memory` | `str` | System memory information |
| `Driver` | `str` | Tenstorrent driver version |

**Sources**: [tt_topology/log.py 143-152](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L143-L152)

## Chip Configuration Model

The `ChipConfig` model represents the state of individual chips with optional fields:

| Field | Type | Description |
| --- | --- | --- |
| `board_id` | `str` | Unique board identifier |
| `fw_version` | `str` | Firmware version |
| `chip_coord_l` | `str` | Left chip coordinates |
| `port_disable_l` | `str` | Left port disable status |
| `rack_shelf_l` | `str` | Left rack/shelf information |
| `chip_coord_r` | `str` | Right chip coordinates |
| `port_disable_r` | `str` | Right port disable status |
| `rack_shelf_r` | `str` | Right rack/shelf information |

All fields in `ChipConfig` are optional and can be `None` when not applicable.

**Sources**: [tt_topology/log.py 154-164](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L154-L164)

## Connection and Coordinate Models

### ConnectionMap Model

Represents physical connections between chips:

| Field | Type | Description |
| --- | --- | --- |
| `id` | `int` | Connection identifier |
| `board_type` | `str` | Type of board |
| `board_id` | `str` | Board identifier |
| `eth_board_info` | `str` | Ethernet board information |
| `connections` | `List[Tuple[int, str]]` | List of connections as (id, description) pairs |

### CoordinateMap Model

Represents logical coordinate assignments:

| Field | Type | Description |
| --- | --- | --- |
| `chip_id` | `int` | Chip identifier |
| `x_coord` | `int` | X coordinate in topology grid |
| `y_coord` | `int` | Y coordinate in topology grid |

**Sources**: [tt_topology/log.py 166-180](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L166-L180)

## JSON Output Format

The `TTToplogyLog` class provides a `save_as_json()` method that serializes the log entry to a formatted JSON file:

The JSON output excludes `None` values and is formatted with 4-space indentation for readability. Log files are typically saved to the `~/tt_topology_logs/` directory.

**Sources**: [tt_topology/log.py 195-199](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L195-L199)

## Optional Field Decorator

The logging system uses an `@optional` decorator to make all fields in a model optional, allowing for partial data capture when full information is not available:

`@optionalclass ChipConfig(ElasticModel):    # All fields become optional and default to None`
This decorator can be applied to entire classes or specific field names, providing flexibility in log data completeness.

**Sources**: [tt_topology/log.py 46-64](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/log.py#L46-L64)
