## Purpose and Scope

This document describes the kernel module initialization and shutdown sequences for the Tenstorrent driver (`tt-kmd`). It covers the driver's entry points (`module_init` and `module_exit`), the initialization order of subsystems, module parameters that control driver behavior, and cleanup procedures.

The module serves as the primary entry point for the Tenstorrent driver stack, coordinating the registration of the character device subsystem, the PCI driver, and the setup of diagnostic filesystems (debugfs and procfs).

**Sources**: [module.c 1-121](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L1-L121)[module.h 1-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L1-L38)

## Module Entry and Exit Points

The Tenstorrent kernel driver defines two primary entry points using the standard Linux kernel module macros. These are declared at the end of the module source file.

*   **`module_init(ttdriver_init)`**: Invoked when the module is loaded via `insmod`, `modprobe`, or during system boot. It handles global subsystem initialization. [module.c 120](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L120-L120)
*   **`module_exit(ttdriver_cleanup)`**: Invoked when the module is unloaded via `rmmod` or during system shutdown. It ensures all resources are released and devices are detached. [module.c 121](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L121-L121)

## Driver Initialization Sequence

The `ttdriver_init()` function [module.c 76-108](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L76-L108) performs initialization in a specific order. It uses a "goto-fail" pattern to ensure that if any stage fails, all previously successful steps are unwound in reverse order.

### Initialization Flow

**Sources**: [module.c 76-108](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L76-L108)

### Initialization Details

1.   **Version Announcement**: The driver prints its version string (e.g., `2.9.1-pre`) to the kernel log [module.c 80](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L80-L80)
2.   **debugfs Setup**: Creates `/sys/kernel/debug/tenstorrent/`[module.c 82](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L82-L82) This directory acts as the parent for per-device debug files.
3.   **procfs Setup**: Creates `/proc/driver/tenstorrent/`[module.c 84](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L84-L84) This is primarily used for process-specific information like `pids`.
4.   **Character Driver**: Calls `init_char_driver()`[module.c 90](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L90-L90) This allocates the character device major/minor numbers and sets up the `/dev/tenstorrent/` device nodes.
5.   **PCI Registration**: Calls `tenstorrent_pci_register_driver()`[module.c 94](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L94-L94) This is the final step where the driver informs the kernel it is ready to handle Tenstorrent PCI devices. This triggers the `probe` sequence for any hardware found in the system.

**Sources**: [module.c 80-94](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L80-L94)[chardev.h 10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.h#L10-L10)[enumerate.h 23](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.h#L23-L23)

## Module Parameters

Module parameters allow users to configure driver behavior at load time. Most parameters are read-only (`0444`) after the module is loaded.

### Parameter Reference

| Parameter | Type | Default | Description | Defined At |
| --- | --- | --- | --- | --- |
| `max_devices` | `uint` | 32 | Maximum number of Tenstorrent devices to support. | [module.c 36-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L36-L38) |
| `dma_address_bits` | `uint` | 0 | DMA address bits (e.g., 32 or 40). 0 for automatic. | [module.c 40-42](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L40-L42) |
| `reset_limit` | `uint` | 10 | Max number of reset attempts during boot. | [module.c 44-46](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L44-L46) |
| `auto_reset_timeout` | `byte` | 10 | Seconds for M3 auto-reset to occur. | [module.c 48-50](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L48-L50) |
| `power_policy` | `bool` | true | Enable low power at probe and aggregation on close. | [module.c 52-54](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L52-L54) |
| `idle_power_down_grace_ms` | `uint` | 5000 | Delay before idle power-down after last FD close. | [module.c 56-62](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L56-L62) |

The `idle_power_down_grace_ms` parameter is unique as it has `0644` permissions [module.c 57](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L57-L57) allowing it to be modified at runtime via `/sys/module/tenstorrent/parameters/idle_power_down_grace_ms`.

## PCI Device ID Table

The driver uses a `pci_device_id` table to determine which hardware it can manage. This table maps PCI IDs to specific hardware "classes" (Wormhole or Blackhole).

The `driver_data` field in the table carries a pointer to the `tenstorrent_device_class` structure for that specific hardware generation [module.c 68-70](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L68-L70) This pointer is used during the PCI `probe` phase to instantiate the correct device logic.

**Sources**: [module.c 64-74](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L64-L74)[enumerate.h 15-18](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.h#L15-L18)

## Driver Cleanup and Unload

Unloading the driver follows the exact reverse order of initialization to ensure that hardware is properly released before the software subsystems are torn down.

1.   **PCI Unregistration**: `tenstorrent_pci_unregister_driver()`[module.c 114](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L114-L114) is called first. This triggers the `remove` callback for every active device, ensuring hardware interrupts are disabled and BARs are unmapped.
2.   **Character Driver Cleanup**: `cleanup_char_driver()`[module.c 115](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L115-L115) destroys the character device nodes and releases major/minor numbers.
3.   **Filesystem Removal**: `debugfs_remove()` and `proc_remove()`[module.c 116-117](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L116-L117) delete the diagnostic directories.

**Sources**: [module.c 110-118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L110-L118)

## Version Management

The driver version is managed through several macros in `module.h`. This version is used for the kernel's `MODULE_VERSION` and is reported to user-space via IOCTLs.

*   **Major Version**: 2 [module.h 19](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L19-L19)
*   **Minor Version**: 9 [module.h 20](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L20-L20)
*   **Patch Version**: 1 [module.h 21](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L21-L21)
*   **Suffix**: "-pre" [module.h 22](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L22-L22)

The final string is constructed as `TENSTORRENT_DRIVER_VERSION_STRING`[module.c 23-27](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L23-L27)

## External Configuration Integration

The driver supports standard Linux packaging and build systems.

*   **DKMS**: The `dkms.conf` file specifies the package name (`tenstorrent`) and version, and defines a post-install script `dkms-post-install`[dkms.conf 1-10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/dkms.conf#L1-L10)
*   **AKMS**: For Alpine Linux, the `AKMBUILD` file defines the module name and version for the Alpine Kernel Module Support system [AKMBUILD 1-4](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/AKMBUILD#L1-L4)
