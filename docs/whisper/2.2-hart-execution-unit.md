# Hart Execution Unit

Relevant source files
*   [Hart.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp)
*   [Hart.hpp](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.hpp)

## Purpose and Scope

The Hart Execution Unit is the core instruction execution component of the Whisper RISC-V simulator. This document describes the `Hart<URV>` template class, which models a single RISC-V hardware thread (hart) capable of fetching, decoding, and executing RISC-V instructions. The Hart manages all architectural state including registers, privilege modes, and integrates with the memory subsystem, CSR management, and debug facilities.

For information about register files and CSRs, see [Control and Status Registers](https://deepwiki.com/tenstorrent/whisper/3.1-control-and-status-registers-(csrs)), [Vector Register File](https://deepwiki.com/tenstorrent/whisper/3.2-vector-register-file), and [Performance Counters](https://deepwiki.com/tenstorrent/whisper/3.3-performance-counters). For memory management details, see [Memory Subsystem](https://deepwiki.com/tenstorrent/whisper/4-memory-subsystem). For extension-specific execution details, see [RISC-V Extensions](https://deepwiki.com/tenstorrent/whisper/5-risc-v-extensions).

## Hart Class Structure

The `Hart<URV>` template class is parameterized by the unsigned register type `URV`, which is `uint32_t` for RV32 and `uint64_t` for RV64. The class encapsulates all architectural state and execution logic for a single hardware thread.

### Reset Behavior

The `reset()` method restores the hart to its initial state:

| Component | Reset Action |
| --- | --- |
| Privilege Mode | Set to `PrivilegeMode::Machine` |
| Virtual Mode | Clear to `false` |
| Program Counter | Set to `resetPc_` (default 0) |
| Integer Registers | Reset all to 0 (x0 hardwired) |
| FP/Vector Registers | Reset to initial state |
| CSRs | Reset to configured values |
| Debug Mode | Clear to `false` |
| Decode Cache | Invalidated |
| Memory Reservations | Cancelled via `cancelLr()` |

### Fetch Stage

### Decode Stage

The decode stage uses the `Decoder` class to look up instruction information in the `InstTable` and populate a `DecodedInst` structure.

| Field | Purpose |
| --- | --- |
| `instEntry_` | Pointer to `InstEntry` in instruction table |

| `physAddr_` | Physical address of instruction |
| `instId_` | Enumerated instruction identifier |

### Execute Stage

The execute stage dispatches to instruction-specific execution functions. Each instruction type has a corresponding `exec*()` method.

#### Sample Execution Methods

| Method | Instruction Type | Key Operations |
| --- | --- | --- |
| `execAdd()` | Integer arithmetic | Read rs1, rs2; write rd |
| `execAddi()` | Immediate arithmetic | Read rs1, imm; write rd; check HINT ops |
| `execLw()` | Load word | Compute address; call `load<int32_t>()` |
| `execSw()` | Store word | Compute address; call `store<uint32_t>()` |
| `execBeq()` | Branch equal | Compare rs1, rs2; conditionally update PC |

**Sources:**[Hart.cpp 1334-2752](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L1334-L2752)

#### Load Operation Detail

**Sources:**[Hart.cpp 2088-2339](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L2088-L2339)

#### Store Operation Detail

Store operations follow a similar pattern but write to memory and handle write permissions:

| Stage | Operation |
| --- | --- |
| Address Calculation | `base + offset` |
| Trigger Check (Before) | Check address and data triggers |
| Exception Check | `determineStoreException()` - translation, PMP, PMA |
| Write Memory | `writeForStore()` - handle devices, MCM, or direct write |
| Trigger Check (After) | Check post-store triggers |
| State Update | Mark `ldStWrite_ = true`, record address/data |

## State Management

### Privilege Modes

The Hart supports three standard privilege modes plus debug mode:

**Sources:**[Hart.cpp 3043-3167](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L3043-L3167) (exception initiation), [Hart.cpp 3421-3489](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L3421-L3489) (debug mode entry/exit)

### Virtual Mode (Hypervisor Extension)

When the Hypervisor extension is enabled, the `virtMode_` flag indicates whether the hart is executing virtualized supervisor or user code:

| `privMode_` | `virtMode_` | Effective Mode | Description |
| --- | --- | --- | --- |
| Machine | false | M-mode | Hypervisor/firmware |
| Supervisor | false | HS-mode | Host supervisor |
| Supervisor | true | VS-mode | Virtual supervisor (guest OS) |
| User | false | U-mode | Host user |
| User | true | VU-mode | Virtual user (guest app) |

**Sources:**[Hart.hpp 2068-2073](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.hpp#L2068-L2073)

### Debug Mode

Debug mode is entered via:

*   Debug request from external debugger
*   `ebreak` instruction when `dcsr.ebreakm/s/u` is set
*   Trigger match with action=1 (enter debug mode)

**Sources:**[Hart.cpp 3421-3489](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L3421-L3489)

## Exception and Interrupt Handling

### Exception Initiation

Exceptions are synchronous traps caused by instruction execution:

**Sources:**[Hart.cpp 3079-3167](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L3079-L3167)

### Interrupt Handling

Interrupts are asynchronous traps:

| Interrupt Type | Source | CSR Bits |
| --- | --- | --- |
| Machine Timer | ACLINT mtime compare | MIP.MTIP |
| Machine Software | ACLINT software interrupt | MIP.MSIP |
| Machine External | External interrupt controller | MIP.MEIP |
| Supervisor Timer | Delegated or VS-mode | MIP.STIP / MIP.VSTIP |
| Supervisor Software | Delegated or VS-mode | MIP.SSIP / MIP.VSSIP |
| Supervisor External | Delegated or VS-mode | MIP.SEIP / MIP.VSEIP |

Interrupt priority is determined by the order in `mInterrupts_`, `sInterrupts_`, and `vsInterrupts_` vectors, which can be configured via `setMachineInterrupts()` and `setSupervisorInterrupts()`.

## Memory Access

### Virtual Memory Integration

The Hart integrates with the `VirtMem` class for address translation:

Memory access callbacks are configured in `setupVirtMemCallbacks()` to handle special cases:

*   STEE (secure memory extension)
*   MCM (memory consistency model)
*   Data cache simulation

**Sources:**[Hart.cpp 313-398](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L313-L398)

### Load/Store Address Translation

For each memory access, the Hart performs:

1.   **Address Masking**: Apply pointer masking based on privilege mode and PMM settings
2.   **Translation**: Call `virtMem_.translateForLoad/Store()` if paging is enabled
3.   **PMP Check**: Verify physical memory protection permissions
4.   **PMA Check**: Verify physical memory attributes (cacheable, readable, writable, etc.)
5.   **Memory Access**: Read or write physical memory

**Sources:**[Hart.cpp 1793-1940](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L1793-L1940) (load exception determination), similar for stores

## Integration with Subsystems

### CSR Interface

The Hart provides methods to access CSRs:

| Method | Purpose |
| --- | --- |
| `peekCsr(CsrNumber, val)` | Read CSR value |
| `pokeCsr(CsrNumber, val)` | Write CSR value |
| `findCsr(CsrNumber)` | Get CSR object pointer |
| `configCsr(...)` | Configure CSR properties |

CSR accesses are subject to privilege checks and side effects managed by the `CsRegs` class.

**Sources:**[Hart.hpp 247-350](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.hpp#L247-L350)

### Trigger System

Debug triggers are managed through the CSR interface but have special handling in the Hart:

**Sources:**[Hart.cpp 2101-2467](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L2101-L2467) (trigger checks in load/store)

### MCM and Performance API

Optional integration with out-of-order memory models:

| Component | Interface | Purpose |
| --- | --- | --- |
| `Mcm<URV>* mcm_` | Memory Consistency Model | Simulates out-of-order memory operations |
| `PerfApi* perfApi_` | Performance Model API | Provides pipeline-accurate simulation |
| `fetchCache_` | Instruction cache model | Tracks fetch cache state |
| `dataCache_` | Data cache model | Tracks data cache state |

When MCM is enabled, load/store operations interact with cache models and merge buffers instead of directly accessing memory.

## Instruction Tracing and Logging

The Hart provides detailed instruction tracing capabilities:

### Trace Data Collection

After each instruction, the following data is available:

| Category | Accessor Methods |
| --- | --- |
| PC | `lastPc()`, `peekPc()` |
| Integer Register | `lastIntReg()` - returns register number and previous value |
| FP Register | `lastFpReg()` - returns register number and previous value |
| Vector Register | `lastVecReg()` - returns register number and group |
| CSR Changes | `lastCsr()` - fills vector with changed CSRs |
| Memory Access | `lastStore()`, `lastLdStAddress()` |
| Branch Taken | `lastBranchTaken()` |
| Trap | `lastInstructionTrapped()`, `lastTrapCause()` |

**Sources:**[Hart.hpp 1032-1157](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.hpp#L1032-L1157)

## Performance Counters

The Hart integrates with the performance counter system through `CsRegs`:

| Counter | Description |
| --- | --- |
| `instCounter_` | Total instructions executed (including exceptions) |
| `retiredInsts_` | Successfully retired instructions (tied to MINSTRET) |
| `cycleCount_` | Cycle count (tied to MCYCLE) |
| `exceptionCount_` | Total exceptions |
| `interruptCount_` | Total interrupts |

Performance event counters (MHPMCOUNTER3-31) are updated based on the `perfControl_` mask and configured event mappings.

**Sources:**[Hart.cpp 252-308](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L252-L308) (tieCsrs), [Hart.cpp 789-806](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L789-L806) (reset performance state)
