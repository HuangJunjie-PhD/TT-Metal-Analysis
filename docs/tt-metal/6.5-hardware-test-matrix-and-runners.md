This page documents the hardware test infrastructure for `tt-metal`, including the test matrix that maps test categories to hardware platforms, GitHub runner configurations, and SKU-to-runner selection mechanisms. This infrastructure ensures comprehensive validation across Tenstorrent's hardware lineup, from single-card devices (N150, N300, P150) to multi-chip systems (T3000, Galaxy).

* * *

## Hardware Platform Hierarchy

The `tt-metal` test infrastructure supports multiple hardware platforms, organized by chip count and architecture. The selection of a specific platform is driven by the `ARCH_NAME` environment variable and `runner-label` inputs in GitHub Actions.

### Platform Architecture Mapping

| --- | --- | --- | --- |

| **Single-Card** | Wormhole B0 | N300 | 2x Wormhole |
| **Single-Card** | Blackhole | P100 / P150 | 1x Blackhole |
| **Multi-Chip** | Wormhole B0 | T3000 (QuietBox) | 8x Wormhole |
| **Multi-Chip** | Wormhole B0 | Galaxy (TG) | 32x Wormhole |

## Runner Infrastructure Types

The CI infrastructure uses several runner types defined by their labels and execution environments. These are mapped in `.github/sku_config.yaml` to ensure consistent environment variables across different hardware targets.

| Runner Type | Label Example | Description | Typical Use Case |
| --- | --- | --- | --- |
| **CIv2 (Current)** | `tt-ubuntu-2204-n300-stable` | Standardized containerized environment with stable versioning. | Functional regressions, unit tests. |
| **CIv2 (vIOMMU)** | `tt-ubuntu-2204-P150-viommu-stable` | Runners with vIOMMU enabled for specific hardware validation. | IOMMU-specific unit tests. |
| **Cloud VM** | `["cloud-virtual-machine", "P100"]` | Virtualized runners for specific cards. | Nightly model regressions. |
| **Bare Metal** | `["in-service", "P100"]` | High-performance dedicated hardware for specific SKUs. | Blackhole P100 nightly validation. |
