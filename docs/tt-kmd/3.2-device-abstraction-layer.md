# Device Abstraction Layer

Relevant source files
*   [chardev_private.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h)
*   [device.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h)

## Purpose and Scope

The Device Abstraction Layer provides a polymorphic interface that enables the Tenstorrent kernel driver to support multiple hardware generations through a common API. This abstraction is implemented via the `tenstorrent_device_class` structure, which defines both static hardware properties and function pointers for hardware-specific operations. The driver currently supports hardware generations such as Wormhole and Blackhole, each with distinct DMA addressing capabilities, TLB configurations, and firmware communication protocols.

This page documents the abstraction interface itself. For details on specific hardware implementations, see [Wormhole Device Driver](https://deepwiki.com/tenstorrent/tt-kmd/4.1-wormhole-device-driver) and [Blackhole Device Driver](https://deepwiki.com/tenstorrent/tt-kmd/4.2-blackhole-device-driver).

## Core Abstraction Structure

### The tenstorrent_device_class Structure

The `tenstorrent_device_class` structure [device.h 81-119](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L81-L119) defines the interface that each hardware generation must implement. This structure contains two categories of members:

1.   **Static Properties**: Hardware characteristics that are fixed for a given device generation (e.g., DMA address width, TLB counts, NOC parameters).
2.   **Function Pointers**: Hardware-specific operations that vary in implementation across device generations (e.g., reset, TLB configuration, power management).

**Relationship Diagram: Device Instance and Class Structure**

The `tenstorrent_device` structure [device.h 25-77](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L25-L77) contains a `const struct tenstorrent_device_class *dev_class` pointer [device.h 31](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L31-L31) that references the hardware-specific class implementation. This pointer is assigned during device probe based on the PCI device ID.

Sources: [device.h 25-119](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L25-L119)

### Instance Data vs. Class Data

The separation between `tenstorrent_device` (instance) and `tenstorrent_device_class` (class) follows object-oriented design principles:

| Structure | Purpose | Lifetime | Multiplicity |
| --- | --- | --- | --- |
| `tenstorrent_device_class` | Defines hardware generation capabilities | Static, exists for driver lifetime | One per hardware generation |
| `tenstorrent_device` | Represents a specific physical device | Dynamic, created at probe, destroyed at remove | One per physical device |

The class defines what a hardware generation _can do_, while the instance tracks what a specific device _is doing_. For example, `tlb_counts` appears in both structures: `dev_class->tlb_counts` defines the default TLB configuration for that hardware generation [device.h 88](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L88-L88) while `ttdev->tlb_counts` stores the actual TLB allocation for a specific device instance [device.h 64](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L64-L64)

Sources: [device.h 25-119](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L25-L119)

## Static Properties

Static properties define hardware characteristics that are intrinsic to a device generation and do not change at runtime.

### Hardware Identification and Sizing

| Property | Type | Purpose |
| --- | --- | --- |
| `name` | `const char *` | Human-readable hardware generation name ("wormhole", "blackhole") [device.h 82](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L82-L82) |
| `instance_size` | `u32` | Size of the device-specific extension structure (e.g., `struct wormhole_device`) [device.h 83](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L83-L83) |

The `instance_size` field enables device-specific implementations to extend the base `tenstorrent_device` structure with additional private data. The allocation logic uses this size to allocate sufficient memory for the extended structure.

Sources: [device.h 82-83](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L82-L83)

### DMA Addressing Capabilities

**DMA Addressing Differences Between Hardware Generations**

| Property | Type | Purpose |
| --- | --- | --- |
| `dma_address_bits` | `u32` | Width of DMA addresses supported by hardware [device.h 84](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L84-L84) |
| `noc_dma_limit` | `u64` | Maximum address that can be accessed via NOC DMA [device.h 85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L85-L85) |
| `noc_pcie_offset` | `u64` | Base offset for NOC addresses when accessing PCIe space [device.h 86](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L86-L86) |

These properties are critical for address translation and memory management. The memory management subsystem uses `dma_address_bits` to validate physical addresses and configure address translation units.

Sources: [device.h 84-86](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L84-L86)

### TLB Configuration

Translation Lookaside Buffers (TLBs) provide inbound address translation, allowing the device's Network-on-Chip (NOC) to access host memory. Different hardware generations support different TLB configurations.

| Property | Type | Purpose |
| --- | --- | --- |
| `tlb_kinds` | `u32` | Number of distinct TLB window sizes supported [device.h 87](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L87-L87) |
| `tlb_counts[MAX_TLB_KINDS]` | `u32[4]` | Number of TLBs available for each window size [device.h 88](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L88-L88) |
| `tlb_sizes[MAX_TLB_KINDS]` | `u64[4]` | Size of each TLB window type [device.h 89](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L89-L89) |

For example, Wormhole supports three TLB kinds with different window sizes (1MB, 2MB, 16MB), while Blackhole supports two kinds (2MB, 4GB). These configurations are defined in the hardware-specific class implementations.

Sources: [device.h 21-89](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L21-L89)

## Virtual Functions

Virtual functions are implemented as function pointers in the `tenstorrent_device_class` structure. These enable hardware-specific behavior while maintaining a uniform interface.

### File Descriptor Lifecycle Callbacks

| Function Pointer | Signature | Purpose |
| --- | --- | --- |
| `first_open_cb` | `void (*)(struct tenstorrent_device *)` | Called when first file descriptor is opened [device.h 97](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L97-L97) |
| `last_release_cb` | `void (*)(struct tenstorrent_device *)` | Called when last file descriptor is closed [device.h 98](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L98-L98) |

These callbacks allow hardware-specific implementations to perform actions when the device transitions between idle and in-use states. The driver also supports `defer_idle_powerdown`[device.h 118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L118-L118) which allows the idle power-down message to be sent from a `delayed_work` item [device.h 61](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L61-L61) armed when the last FD closes, rather than synchronously.

Sources: [device.h 61-118](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L61-L118)

### Per-File-Descriptor State

When a user-space process opens the character device, a `chardev_private` structure [chardev_private.h 55-78](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L55-L78) is created. This structure tracks resources owned by that specific file descriptor:

*   **Reset Tracking**: `open_reset_gen` stores the `reset_gen` value at the time of open [chardev_private.h 77](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L77-L77)
*   **Resource Ownership**: Bitmaps for `resource_lock`[chardev_private.h 68](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L68-L68) and `tlbs`[chardev_private.h 72](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L72-L72)
*   **Memory Mappings**: A list of `tenstorrent_mmap_vma` structures [chardev_private.h 62](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L62-L62) to track active mmaps.
*   **DMA Buffers**: A hashtable of `dmabuf` structures [chardev_private.h 58](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L58-L58)

Sources: [device.h 34-76](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L34-L76)[chardev_private.h 55-78](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L55-L78)
