This page documents the unit testing infrastructure for tt-system-firmware, which uses Zephyr's Twister test framework to validate firmware components in simulated environments. Unit tests run on native platforms without requiring physical hardware, enabling fast feedback during development and continuous integration.

For testing on physical hardware, see [Hardware Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests). For integration testing across firmware layers, see [End-to-End Testing with pyluwen](https://deepwiki.com/tenstorrent/tt-system-firmware/7.3-end-to-end-testing-with-pyluwen).

* * *

## Overview of Twister

Twister is Zephyr RTOS's native test runner and build system integration tool. It discovers, builds, and executes test cases across multiple platforms and configurations. In tt-system-firmware, Twister serves two primary functions:

1.   **Native Simulation Testing**: Executes unit tests on `native_sim` platforms that emulate embedded targets on the host system
2.   **Build Verification**: Validates that firmware builds successfully for all supported hardware configurations without execution

Twister automatically discovers tests by scanning for `testcase.yaml` files, builds them with appropriate configurations, and collects results in machine-readable formats.

## Test Execution Platforms

The codebase uses three platform configurations for Twister-based testing:

| Platform | Architecture | Purpose | Execution Mode |
| --- | --- | --- | --- |
| `unit_testing` | Host native | Fast unit tests with minimal dependencies | Run |
| `native_sim` | Host native (POSIX) | Full Zephyr simulation with device drivers | Run |
| `native_sim/native/64` | Host native 64-bit | Extended tests requiring 64-bit addressing | Run |
| `tt_blackhole@*/tt_blackhole/smc` | ARC | SMC firmware build validation | Build-only |
| `tt_blackhole@*/tt_blackhole/dmc` | RISC-V | DMC firmware build validation | Build-only |
| `tt_grendel_smc` | ARC | Grendel board smoke tests | Build-only |

## CI Workflow Integration

The unit testing infrastructure integrates with GitHub Actions through two primary workflows:

### Unit Test Workflow (`run-unit-tests.yml`)

This workflow executes on every push, pull request, and merge queue event to validate code changes.

**Key workflow features:**

*   **Two-stage execution**: Separates build (`--build-only`) from run (`--run-only`) for better failure diagnostics
*   **Incremental builds**: Uses `-i` flag to reuse previous build artifacts when possible
*   **Artifact preservation**: Uploads test results, logs, and build artifacts for debugging failures
*   **Parallel execution**: `unit-test` and `tt-sim-test` jobs run concurrently

**Sources:**

*   [.github/workflows/run-unit-tests.yml 26-67](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/run-unit-tests.yml#L26-L67)

### Build-Only Tests (`build-fw.yml`)

The main firmware build workflow includes additional Twister-based build verification:

**Build-only test characteristics:**

*   **Multi-board matrix**: Tests all three Blackhole board variants (P100A, P150A, P300A)
*   **Resource optimization**: `--runtime-artifact-cleanup pass` removes intermediate build files to conserve disk space
*   **Parallel builds**: `-j 2` limits concurrent jobs to avoid overwhelming runners
*   **Tag filtering**: Grendel tests use `--tag smoke` to select smoke test subset

## Twister Command Reference

### Core Command Patterns

`# Build and run unit tests on simulation platformswest twister -i \  -p unit_testing -p native_sim \  -T tt-system-firmware/tests \  -T tt-system-firmware/samples # Build-only for hardware platform validationwest twister \  -p tt_blackhole@p150a/tt_blackhole/smc \  --build-only \  -T tt-system-firmware/app/smc \  --outdir twister-smc-tests-p150a \  --runtime-artifact-cleanup pass \  -j 2 # Run tests with specific tags on 64-bit simulationwest twister -i \  -p native_sim/native/64 \  --tag tt_sim \  --alt-config-root tt-system-firmware/test-conf \  -T zephyr`
### Common Flags and Options

| Flag | Purpose | Usage Pattern |
| --- | --- | --- |
| `-i, --inline-logs` | Incremental builds, inline test output | Always use for CI |
| `-p, --platform` | Target platform(s) | `-p native_sim -p unit_testing` |
| `-T, --testsuite-root` | Test search directory | `-T tt-system-firmware/tests` |
| `--build-only` | Compile without execution | Hardware platform validation |
| `--run-only` | Execute pre-built tests | Two-stage CI workflow |
| `--tag` | Filter by test tags | `--tag smoke` for quick tests |
| `--outdir` | Output directory | `--outdir twister-out` |
| `--runtime-artifact-cleanup` | Delete build intermediates | `pass` to clean after pass |
| `-j, --jobs` | Parallel build jobs | `-j 2` for resource-limited runners |
| `-v, --verbose` | Detailed output | Debugging test failures |
| `--alt-config-root` | Additional config search paths | Board-specific overlays |

## Test Artifacts and Output

Twister generates structured output for result analysis and debugging:

### Artifact Structure

```
twister-out/                           # Default output directory
├── twister.log                        # Detailed execution log
├── twister.json                       # Machine-readable test results
├── <platform>/
│   └── <test-name>/
│       ├── handler.log                # Test execution output
│       ├── twister_harness.log        # Harness communication log
│       ├── zephyr.dts                 # Generated device tree
│       ├── .config                    # Final Kconfig configuration
│       ├── zephyr.elf                 # Executable binary (ELF)
│       ├── zephyr.bin                 # Raw binary image
│       ├── zephyr.map                 # Symbol map
│       └── zephyr.stat                # Binary statistics
└── device.log                         # Device output (if applicable)
```

### CI Artifact Upload

The workflows upload specific files for post-run analysis:

`# From run-unit-tests.ymlpath: |  twister-out/**/.config  twister-out/**/devicetree_generated.h  twister-out/**/zephyr.bin  twister-out/**/zephyr.dts  twister-out/**/zephyr.elf  twister-out/**/zephyr.map  twister-out/**/zephyr.stat  twister-out/**/handler.log  twister-out/twister.log  twister-out/twister.json`
**Key artifact files:**

*   **`twister.json`**: Structured test results with pass/fail status, execution time, and failure messages
*   **`handler.log`**: Raw test output including assertion failures and debug prints
*   **`zephyr.map`**: Memory layout and symbol addresses for debugging crashes
*   **`.config`**: Resolved Kconfig showing which features were enabled

### Prerequisites

1.   **Zephyr SDK Installation**: Follow [Getting Started](https://deepwiki.com/tenstorrent/tt-system-firmware/9.1-getting-started) for environment setup
2.   **West Workspace**: Initialize with `west init -l tt-system-firmware && west update`

### Basic Test Execution

`# From workspace root, run all unit testswest twister -i \  -p unit_testing -p native_sim \  -T tt-system-firmware/tests \  -T tt-system-firmware/samples # Run specific test suitewest twister -i \  -p native_sim \  -T tt-system-firmware/tests/specific_component # Run tests matching tagwest twister -i \  -p native_sim \  --tag smoke \  -T tt-system-firmware/tests # Verbose output for debuggingwest twister -i -v \  -p native_sim \  -T tt-system-firmware/tests/failing_test`

`# Verify SMC builds for P150A boardwest twister \  -p tt_blackhole@p150a/tt_blackhole/smc \  --build-only \  -T tt-system-firmware/app/smc \  --outdir build-smc-p150a # Build tests for all board revisions (manual matrix)for board in p100a p150a p300a; do  BOARD=$(./tt-system-firmware/scripts/rev2board.sh "$board" smc)  west twister \    -p "$BOARD" \    --build-only \    -T tt-system-firmware/app/smc \    --outdir "twister-smc-$board"done`

`# View summarycat twister-out/twister.log | grep -E "PASSED|FAILED|SKIPPED" # Check specific test outputcat twister-out/native_sim/path/to/test/handler.log # Parse JSON results programmaticallypython -m json.tool twister-out/twister.json`

## Troubleshooting Common Issues

### Build Failures

**Symptom**: Twister reports compilation errors

**Debug steps:**

1.   Check `.config` file in output directory for unexpected Kconfig settings
2.   Verify device tree overlay syntax in `zephyr.dts`
3.   Review `handler.log` for compiler error messages
4.   Build manually with verbose output: `west build -p -b <platform> -d build-debug <test-path> -- -DCMAKE_VERBOSE_MAKEFILE=ON`

### Test Execution Timeouts

**Symptom**: Tests hang or timeout

**Common causes:**

*   Infinite loops in test code
*   Missing harness output markers
*   Simulation platform limitations

**Solutions:**

*   Add timeout constraints in `testcase.yaml`: `timeout: 60`
*   Verify test uses proper ztest framework macros
*   Check `device.log` for harness communication

### Platform Not Found

**Symptom**: `Platform <name> not found`

**Resolution:**

1.   Verify platform name matches board definitions in `boards/tenstorrent/`
2.   Check board revision syntax: `tt_blackhole@p150a/tt_blackhole/smc`
3.   Ensure Zephyr workspace is initialized: `west update`
