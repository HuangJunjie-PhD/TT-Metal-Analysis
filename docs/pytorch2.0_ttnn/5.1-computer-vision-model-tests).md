## Purpose and Scope

This document describes the comprehensive test suite for computer vision models in the pytorch2.0_ttnn repository. These tests validate that image classification, object detection, segmentation, and depth estimation models can be successfully compiled and executed through the torch_ttnn backend. The tests cover models from multiple sources including timm, torchvision, Hugging Face transformers, and PyTorch Hub.

For NLP model tests, see [Natural Language Processing Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.2-natural-language-processing-model-tests). For multi-modal and speech model tests, see [Multi-Modal and Speech Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.3-multi-modal-and-speech-model-tests). For information about the testing framework infrastructure and fixtures, see [pytest Infrastructure and Fixtures](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.1-pytest-infrastructure-and-fixtures) and [ModelTester Framework](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.2-modeltester-framework).

## Test Organization and Structure

All computer vision model tests follow a consistent pattern using the `ModelTester` base class defined in [tests/utils.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/utils.py) Each test file defines a `ThisTester` class that inherits from `ModelTester` and implements two abstract methods:

*   **`_load_model()`**: Instantiates and returns the model, typically converting to `torch.bfloat16` precision
*   **`_load_inputs(batch_size)`**: Loads and preprocesses input images, applying model-specific transformations

The test functions are parameterized using pytest to test multiple model variants and execution modes (train/eval). Tests use the `compile_and_run` fixture which automatically compiles models with the torch_ttnn backend, executes them, collects metrics, and validates accuracy.

## Test Execution Patterns and Markers

Computer vision tests use several pytest markers to indicate test status and requirements:

### Marker Types

| Marker | Purpose | Example |
| --- | --- | --- |
| `converted_end_to_end` | Model compiles completely without fallback | Many ResNet/VGG/MobileNet models |
| `compilation_xfail` | Known compilation failure with reason | xception71, dla34, densenet201 |
| `skip` | Test skipped entirely with reason | BEiT (issue #865), RetinaNet (issue #866) |
| `xfail` | Expected failure (usually for unsupported ops) | Some masked_fill test cases |

## Input Loading and Preprocessing Strategies

Computer vision models require diverse preprocessing pipelines. The test suite demonstrates several strategies:

### Strategy Comparison

| Strategy | Models Using It | Key Characteristics |
| --- | --- | --- |
| **Library-specific transforms** | timm models | Uses `timm.data.resolve_model_data_config()` and `create_transform()` |
| **Weights-based transforms** | torchvision models | Uses `weights.transforms()` from model weights |
| **Processor-based** | Hugging Face models (BEiT, DeiT, GLPN) | Uses model-specific processor classes |
| **Manual transforms** | HardNet, YOLOv3, U-Net | Explicit `transforms.Compose()` with standard operations |
| **Synthetic inputs** | U-Net Carvana | `torch.rand()` for testing without real data |
