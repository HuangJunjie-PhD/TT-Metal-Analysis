This page documents the GitHub Actions-based CI/CD infrastructure for tt-system-firmware. It describes the workflow files, their trigger conditions, job matrices, and how they coordinate to provide continuous integration, testing, and deployment capabilities.

For information about the physical test infrastructure and board configurations, see [Hardware Test Infrastructure](/tenstorrent/tt-system-firmware/8.2-hardware-test-infrastructure). For details on quality assurance checks, see [Quality Assurance Checks](/tenstorrent/tt-system-firmware/8.4-quality-assurance-checks). For the board configuration schema, see [Board Configuration Matrix](/tenstorrent/tt-system-firmware/8.3-board-configuration-matrix).

## Overview

The tt-system-firmware repository uses GitHub Actions to automate building, testing, compliance checking, and documentation generation. The CI/CD system consists of 12+ workflow files organized into five categories:

1. **Build Workflows** - Compile firmware, tooling, and recovery images
2. **Test Workflows** - Execute unit, smoke, stress, and integration tests
3. **Quality Workflows** - Enforce code style, licensing, and compliance standards
4. **Documentation Workflows** - Generate and deploy Sphinx/Doxygen documentation
5. **Maintenance Workflows** - Automated cleanup and housekeeping tasks

All workflows use reusable composite actions for environment setup to maintain consistency across jobs. Hardware tests execute on self-hosted runners with direct access to Blackhole and Wormhole boards via JTAG.

Sources: [.github/workflows/hardware-smoke.yml](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml) [.github/workflows/build-fw.yml](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml) [.github/workflows/compliance.yml](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/compliance.yml) [README.md3-7](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/README.md?plain=1#L3-L7)

## Build Workflows

### build-fw.yml

The primary firmware build workflow that produces signed firmware bundles for all supported board revisions. This workflow is reusable and called by other workflows that need firmware artifacts.

**Key Jobs:**

* `genboards` - Generates board matrix from [.github/boards.json](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/boards.json)
* `build` - Matrix build for each board revision (p100a, p150a, p300a, galaxy)
* `build-recovery` - Builds standalone recovery firmware for p100a
* `build-assembly-test` - Builds assembly test firmware for manufacturing
* `build-combined-fwbundle` - Packages all firmwares into a single `.fwbundle` file
* `smc-build-only-tests` - Twister build-only tests for SMC application
* `grendel-build-test` - Builds for Grendel SMC platform
* `build-bh-flm` - Builds Blackhole Flash Loader Module
