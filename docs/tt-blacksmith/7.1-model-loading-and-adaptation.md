## Purpose and Scope

This document explains the model loading and adaptation system in TT-Blacksmith, focusing on how pre-trained HuggingFace models are loaded, modified for parameter-efficient fine-tuning, and prepared for training on Tenstorrent hardware. The `get_model` function serves as the central entry point for model initialization, supporting both LoRA-based fine-tuning (via the `peft` library) and custom adapter injection.

For information about training execution and integration with the training loop, see [Training Infrastructure](https://deepwiki.com/tenstorrent/tt-blacksmith/7.1-model-loading-and-adaptation#3.5). For details about specific model architectures and dataset implementations, see [Dataset Implementations](https://deepwiki.com/tenstorrent/tt-blacksmith/5.2-device-management-and-hardware-abstraction).

## The `get_model` Function

The `get_model` function is the primary interface for model loading across all PyTorch experiments. It accepts a `TrainingConfig` object and a device, returning a fully prepared model ready for training.

### Function Signature

### Core Responsibilities

| Responsibility | Implementation | Configuration Field |
| --- | --- | --- |
| Load pre-trained model | `AutoModelForCausalLM.from_pretrained()` | `config.model_name` |
| Select adaptation method | Branch on `config.training_type` | `config.training_type` |
| Apply LoRA | `_apply_lora()` helper | `config.lora_*` fields |
| Apply adapters | `_apply_adapters()` helper | `config.adapter_*` fields |
| Convert dtype | `model.to(eval(config.dtype))` | `config.dtype` |
| Move to device | `model.to(device)` | Passed as parameter |

The function validates that `training_type` is either `"lora"` or `"adapters"`, raising a `ValueError` for any other value.

**Sources:**[blacksmith/models/torch/huggingface/hf_models.py 12-30](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/models/torch/huggingface/hf_models.py#L12-L30)

## LoRA Implementation

The LoRA (Low-Rank Adaptation) implementation leverages the `peft` library, which provides a production-ready interface for injecting low-rank matrices into transformer layers.

### Implementation Flow

### `_apply_lora` Helper Function

The helper function creates a `LoraConfig` object and passes it to the `peft` library:

### LoRA Configuration Parameters

| Parameter | Description | Typical Value |
| --- | --- | --- |
| `lora_r` | Rank of low-rank matrices | 8 |
| `lora_alpha` | Scaling factor for LoRA updates | 32 |
| `lora_target_modules` | List of module names to adapt | `["q_proj", "k_proj", "v_proj", "o_proj"]` |
| `lora_task_type` | Task type for PEFT | `"CAUSAL_LM"` |

The `peft` library automatically:

*   Freezes all original model parameters
*   Injects low-rank decomposition matrices (A and B) into specified modules
*   Marks only the injected LoRA parameters as trainable
*   Implements the forward pass computation: `output = original_output + (B @ A @ input) * (alpha / r)`

**Sources:**[blacksmith/models/torch/huggingface/hf_models.py 33-41](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/models/torch/huggingface/hf_models.py#L33-L41)[env/requirements.txt 4](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/env/requirements.txt#L4-L4)

## Custom Adapter Implementation

The custom adapter implementation provides an alternative to LoRA by inserting residual bottleneck layers at strategic points in the transformer architecture. This approach is based on the original Adapter Layers paper (Houlsby et al., 2019).

### ResidualAdapter Architecture

### ResidualAdapter Class

The `ResidualAdapter` is implemented as a nested class within the `make_adapted_layer` helper function:

### Key Design Features

| Feature | Implementation | Purpose |
| --- | --- | --- |
| Bottleneck architecture | `d → bottleneck_dim → d` | Reduce parameter count |
| Zero initialization | Final layer weights/bias = 0 | Start as identity function |
| Residual connection | `output = original + adapter` | Preserve pre-trained behavior |
| Frozen base | Original linear frozen | Only adapter trainable |

**Sources:**[blacksmith/models/torch/huggingface/hf_models.py 69-91](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/models/torch/huggingface/hf_models.py#L69-L91)

## Adapter Insertion Strategy

The `_apply_adapters` function implements a systematic insertion strategy that targets specific projection layers within each transformer block.

### Insertion Algorithm

### Target Modules

Adapters are inserted at two locations per transformer block:

| Location | Module Path | Purpose |
| --- | --- | --- |
| Attention output | `model.model.layers[i].self_attn.o_proj` | Adapt attention outputs |
| MLP output | `model.model.layers[i].mlp.down_proj` | Adapt feed-forward outputs |

### Layer Selection

An empty `adapter_layers` list applies adapters to all transformer blocks. Specifying indices (e.g., `[0, 1, 2]`) restricts adaptation to those specific layers.

**Sources:**[blacksmith/models/torch/huggingface/hf_models.py 44-66](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/models/torch/huggingface/hf_models.py#L44-L66)

## Model Preparation and Device Placement

After adaptation (LoRA or adapters), the model undergoes final preparation steps before being returned to the training loop.

### Preparation Steps

### Dtype Conversion

The configuration specifies the training precision as a string that is evaluated at runtime:

Typical values:

*   `"torch.bfloat16"` - BFloat16 for efficient training
*   `"torch.float32"` - Full precision (rarely used)
*   `"torch.float16"` - Half precision

### Device Placement

The device is determined by the calling code and passed to `get_model`:

Device types:

*   **XLA device**: `torch_xla.device()` for Tenstorrent hardware
*   **CUDA device**: `torch.device("cuda")` for GPU
*   **CPU device**: `torch.device("cpu")` for debugging

### Usage Example

The logging statements verify that parameter-efficient fine-tuning has correctly frozen base parameters while keeping adaptation parameters trainable.

## Configuration Reference

The following table summarizes all configuration fields relevant to model loading and adaptation:

| Field | Type | Purpose | Example Value |
| --- | --- | --- | --- |
| `model_name` | `str` | HuggingFace model identifier | `"meta-llama/Llama-3.2-1B"` |
| `dtype` | `str` | Training precision (as string) | `"torch.bfloat16"` |
| `training_type` | `str` | Adaptation method | `"lora"` or `"adapters"` |
| `gradient_checkpointing` | `bool` | Enable gradient checkpointing | `false` |
| `lora_r` | `int` | LoRA rank | `8` |
| `lora_alpha` | `int` | LoRA scaling factor | `32` |
| `lora_target_modules` | `list[str]` | Modules to adapt | `["q_proj", "k_proj", "v_proj", "o_proj"]` |
| `lora_task_type` | `str` | PEFT task type | `"CAUSAL_LM"` |
| `adapter_bottleneck_dim` | `int` | Adapter hidden dimension | `24` |
| `adapter_non_linearity` | `str` | Activation function | `"torch.nn.GELU"` |
| `adapter_layers` | `list[int]` | Layer indices (empty = all) | `[]` or `[0, 1, 2]` |

All these fields are defined in the `TrainingConfig` Pydantic model and can be specified in YAML configuration files.
