## Purpose and Scope

This document describes the types of transactions supported by the NoC (Network-on-Chip), how packets are addressed, and how they are routed through the 2D torus network. For details about the NIU registers used to initiate transactions, see [NoC Memory Map and NIU Interface](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface). For information about virtual channel assignment and ordering guarantees, see [Virtual Channels and Ordering Guarantees](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.3-virtual-channels-and-ordering-guarantees). For address alignment requirements, see [NoC Counters and Alignment Requirements](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements).

## Transaction Types

The NoC supports three fundamental transaction types: read requests, write requests, and atomic requests. Each transaction consists of one or more packets, where each packet contains a header flit followed by zero or more data flits.

### Read Requests

A read request retrieves data from a target tile's address space and delivers it to a return address (typically in the initiating tile).

**Packet Flow:**

1.   Read request packet (single header flit) travels from initiating NIU to `NOC_TARG_ADDR`
2.   Read response packet (header + data flits) travels from `NOC_TARG_ADDR` back to `NOC_RET_ADDR`

**Key Characteristics:**

*   Request type: `NOC_CMD_RD` (`0`)
*   Maximum payload: 8192 bytes per request (256 data flits × 32 bytes/flit)
*   Always generates a response packet
*   Cannot be broadcast (unicast only)

**Register Configuration:**

*   `NOC_TARG_ADDR`: Source location for data (coordinates + memory address)
*   `NOC_RET_ADDR`: Destination for read data (coordinates + memory address)
*   `NOC_AT_LEN_BE`: Number of bytes to read (1 to 8192)

### Write Requests

Write requests transfer data from a source to a destination. The NoC supports two variants based on the source of the data.

#### Write with Inline Data (`NOC_CMD_WR_INLINE=true`)

Data comes from the 32-bit `NOC_AT_DATA` register.

**Packet Flow:**

1.   Write request packet (single header flit containing immediate data) travels to `NOC_TARG_ADDR`
2.   Optional write acknowledgement packet (single header flit) returns to initiating NIU if `NOC_CMD_RESP_MARKED` is set

**Key Characteristics:**

*   Data source: `NOC_AT_DATA` immediate value
*   Maximum write size: 16 bytes (immediate value is broadcasted and masked)
*   Can be broadcast to multiple tiles
*   Requires destination to be Tensix or Ethernet tile

Sources: [WormholeB0/NoC/MemoryMap.md 81](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L81-L81)[WormholeB0/NoC/MemoryMap.md 126](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L126-L126)

#### Write with Memory Data (`NOC_CMD_WR_INLINE=false`)

Data comes from the initiating tile's address space.

**Packet Flow:**

1.   Initiating NIU reads data from address within `NOC_TARG_ADDR`
2.   Write request packet (header + data flits) travels to `NOC_RET_ADDR`
3.   Optional write acknowledgement packet returns to `NOC_TARG_ADDR` coordinates if `NOC_CMD_RESP_MARKED` is set

**Key Characteristics:**

*   Data source: Memory address at initiating tile (from `NOC_TARG_ADDR` memory address field)
*   Maximum payload: 8192 bytes per request (can be split automatically)
*   Can be broadcast to multiple tiles
*   Posted writes (no acknowledgement) supported via `NOC_CMD_RESP_MARKED=false`

**Address Semantics:**

*   `NOC_TARG_ADDR` memory address: Source address in initiating tile
*   `NOC_TARG_ADDR` coordinates: Initiating tile (for acknowledgement routing)
*   `NOC_RET_ADDR` memory address: Destination address
*   `NOC_RET_ADDR` coordinates: Destination tile

Sources: [WormholeB0/NoC/MemoryMap.md 82](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L82-L82)[WormholeB0/NoC/MemoryMap.md 124-125](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L124-L125)[WormholeB0/NoC/README.md 25-34](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L25-L34)

### Atomic Requests

Atomic requests perform read-modify-write operations on 128-bit aligned regions in L1 memory.

**Packet Flow:**

1.   Atomic request packet (single header flit) travels to `NOC_TARG_ADDR`
2.   Optional atomic response packet (single header flit with result data) returns to `NOC_RET_ADDR` if `NOC_CMD_RESP_MARKED` is set

**Key Characteristics:**

*   Request type: `NOC_CMD_AT` (`1`)
*   Operates on 128-bit (16-byte) aligned L1 regions only
*   Can be broadcast to multiple tiles
*   Destination must be Tensix or Ethernet tile (L1 required)
*   Opcode and operands specified in `NOC_AT_LEN_BE`

**Register Configuration:**

*   `NOC_TARG_ADDR`: Target L1 address (coordinates + memory address)
*   `NOC_RET_ADDR`: Return address for result (coordinates + memory address)
*   `NOC_AT_LEN_BE`: Atomic opcode and operands
*   `NOC_AT_DATA`: Immediate operand data (for most atomic operations)

## Addressing Modes

The NoC supports two addressing modes: unicast (one destination) and broadcast (multiple destinations in a rectangular region).

### Unicast Addressing

Unicast packets are addressed to a single destination tile specified by X/Y coordinates.

**Address Encoding (`NOC_TARG_ADDR` and `NOC_RET_ADDR`):**

| Bits | Field | Description |
| --- | --- | --- |
| 0-35 | Memory Address | 36-bit address within the tile (typically high 4 bits are zero) |
| 36-41 | X Coordinate | Target tile X coordinate (0-9) |
| 42-47 | Y Coordinate | Target tile Y coordinate (0-11) |
| 48-63 | Reserved | Ignored by hardware |

When coordinate translation is enabled (bit 14 of `NIU_CFG_0`), the X/Y coordinates are translated from logical space to NoC coordinate space using lookup tables at `NOC_X_ID_TRANSLATE_TABLE_0` and `NOC_Y_ID_TRANSLATE_TABLE_0`.

Sources: [WormholeB0/NoC/MemoryMap.md 93-102](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L93-L102)

### Broadcast Addressing

Broadcast packets are addressed to a rectangular region of tiles. The `NOC_CMD_BRCST_PACKET` flag in `NOC_CTRL` enables broadcast mode.

**Address Encoding (for `NOC_TARG_ADDR` in broadcast mode):**

| Bits | Field | Description |
| --- | --- | --- |
| 0-35 | Memory Address | Same address written to all recipient tiles |
| 36-41 | EndX | End X coordinate of broadcast rectangle |
| 42-47 | EndY | End Y coordinate of broadcast rectangle |
| 48-53 | StartX | Start X coordinate of broadcast rectangle |
| 54-59 | StartY | Start Y coordinate of broadcast rectangle |
| 60-63 | Reserved | Ignored by hardware |

**Rectangle Definition:**

*   X span: All `x` where `StartX ≤ x ≤ EndX` (or wraps around if `StartX > EndX`)
*   Y span: All `y` where `StartY ≤ y ≤ EndY` (or wraps around if `StartY > EndY`)
*   Broadcast rectangle = X span × Y span

**Restrictions:**

*   Recipients can only be Tensix tiles (other tile types are opted out via `ROUTER_CFG_1` and `ROUTER_CFG_3`)
*   Individual fused-off Tensix tiles are also opted out by firmware
*   The `NOC_CMD_BRCST_SRC_INCLUDE` flag controls whether the initiating tile receives the broadcast

**Use Cases:**

*   Write requests: Broadcast data to multiple tiles
*   Atomic requests: Perform same atomic operation on multiple tiles
*   Read requests: Not supported (unicast only)

Sources: [WormholeB0/NoC/MemoryMap.md 103-114](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L103-L114)[WormholeB0/NoC/MemoryMap.md 59](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L59-L59)[WormholeB0/NoC/README.md 33-34](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L33-L34)

## Routing Mechanisms

The NoC uses dimension-order routing to move packets through the 2D torus topology. Each NoC uses a different axis ordering.

### Dimension-Order Routing

**NoC #0 (X-major routing):**

*   All X-axis movement happens first
*   Then all Y-axis movement happens
*   Bit 28 of `NOC_NODE_ID` is always `true` for NoC #0

**NoC #1 (Y-major routing):**

*   All Y-axis movement happens first
*   Then all X-axis movement happens
*   Bit 28 of `NOC_NODE_ID` is always `false` for NoC #1

This opposite routing pattern between the two NoCs provides:

1.   Redundancy for reliability
2.   Higher aggregate bandwidth
3.   Deadlock prevention through natural traffic separation

Sources: [WormholeB0/NoC/MemoryMap.md 189](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L189-L189)[WormholeB0/NoC/README.md 5-11](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L5-L11)

### Dateline Bit and Wraparound

The 2D torus topology means packets can wrap around the edges. The dateline bit in the virtual channel number flips when packets cross certain boundaries to prevent routing loops.

**Dateline Flip Locations:**

*   X-axis: Dateline bit flips when crossing from X=9 to X=0 via the wraparound edge (if enabled for router)
*   Y-axis: Dateline bit flips when crossing from Y=11 to Y=0 via the wraparound edge (if enabled for router)

The router capabilities are encoded in `NOC_NODE_ID`:

*   Bit 26: X-axis dateline flip capability
*   Bit 27: Y-axis dateline flip capability

### Broadcast Routing Trees

Broadcast packets use tree-based routing determined by the `NOC_CMD_BRCST_XY` flag.

**X-as-Major-Axis (`NOC_CMD_BRCST_XY=0`):**

*   Packet first splits along X-axis to reach all columns in the rectangle
*   Then each copy splits along Y-axis to reach all rows

**Y-as-Major-Axis (`NOC_CMD_BRCST_XY=1`):**

*   Packet first splits along Y-axis to reach all rows in the rectangle
*   Then each copy splits along X-axis to reach all columns

The choice of axis ordering can significantly impact broadcast latency depending on the rectangle shape and network congestion patterns.

## Control Flags Summary

Key control flags in `NOC_CTRL` that determine transaction type and routing behavior:

| Flag | Bits | Values | Purpose |
| --- | --- | --- | --- |
| Request Type | 0-1 | `0`=Read, `1`=Atomic, `2`=Write | Selects transaction type |
| `NOC_CMD_WR_BE` | 2 | `0`/`1` | Write uses length vs byte-enable mask |
| `NOC_CMD_WR_INLINE` | 3 | `0`/`1` | Write data from memory vs immediate |
| `NOC_CMD_RESP_MARKED` | 4 | `0`/`1` | Posted vs non-posted (response required) |
| `NOC_CMD_BRCST_PACKET` | 5 | `0`/`1` | Unicast vs broadcast |
| `NOC_CMD_VC_LINKED` | 6 | `0`/`1` | Multi-request transaction linking |
| `NOC_CMD_VC_STATIC` | 7 | `0`/`1` | Static vs dynamic virtual channel assignment |
| `NOC_CMD_PATH_RESERVE` | 8 | `0`/`1` | Broadcast path reservation |
| `NOC_CMD_BRCST_XY` | 16 | `0`/`1` | Broadcast X-major vs Y-major |
| `NOC_CMD_BRCST_SRC_INCLUDE` | 17 | `0`/`1` | Include/exclude source in broadcast |
| `NOC_CMD_ARB_PRIORITY` | 27-30 | `0`-`15` | Arbitration priority level |
