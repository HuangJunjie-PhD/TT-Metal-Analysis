# Vector Extension (V)

Relevant source files
*   [VecRegs.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.cpp)
*   [VecRegs.hpp](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp)
*   [vector.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp)

This document provides comprehensive coverage of the RISC-V Vector Extension (V) implementation in Whisper, including vector register file organization, configuration management, instruction execution, masking policies, and operation patterns. The vector extension enables SIMD (Single Instruction Multiple Data) processing on variable-length vectors with configurable element widths and register groupings.

For hypervisor-related vector support (VS-mode), see [5.2](https://deepwiki.com/tenstorrent/whisper/5.2-hypervisor-extension-(h)). For vector load/store memory operations, many details are covered in [4.1](https://deepwiki.com/tenstorrent/whisper/4.1-physical-memory-and-protection).

## Vector Register File Architecture

The vector register file is implemented in the `VecRegs` class and provides 32 vector registers (v0-v31), each with a configurable width called VLEN (Vector Length). The register file supports dynamic configuration of element width (SEW) and register grouping (LMUL).

### VecRegs Class Structure

**Sources:**[VecRegs.hpp 124-945](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L124-L945)[VecRegs.cpp 25-56](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.cpp#L25-L56)

### Register Organization and Access

The `VecRegs` class maintains a flat byte array (`data_`) containing all 32 vector registers. Individual elements are accessed using the templated `read()` and `write()` methods:

| Method | Purpose | Parameters |
| --- | --- | --- |
| `read<T>(regNum, elemIx, groupX8, value)` | Read element of type T | Register number, element index, group multiplier×8, output value |
| `write<T>(regNum, elemIx, groupX8, value)` | Write element of type T | Register number, element index, group multiplier×8, input value |
| `readIndexReg(vecReg, elemIx, eew, groupX8)` | Read index for indexed ops | Register, element index, element width, group multiplier×8 |
| `writeMaskRegister(maskReg, i, value)` | Write single mask bit | Mask register, bit index, boolean value |
| `readMaskRegister(maskReg, i, value)` | Read single mask bit | Mask register, bit index, output boolean |

**Sources:**[VecRegs.hpp 169-211](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L169-L211)[VecRegs.cpp 251-285](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.cpp#L251-L285)

### Vector Length (VLEN) Configuration

VLEN is the width of each vector register in bits. The simulator configures VLEN through the `config()` method:

The configuration enforces several constraints:

*   VLEN must be a power of 2 between 32 and 4096 bytes
*   Minimum element size (SEW) must be between 1 byte and VLEN
*   Maximum element size must not exceed VLEN
*   Illegal SEW/LMUL combinations are marked in `legalConfigs_`

**Sources:**[VecRegs.cpp 59-194](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.cpp#L59-L194)

## Vector Configuration (vtype and vl)

Vector operations are configured through the `vtype` CSR, which specifies SEW (Selected Element Width), LMUL (Group Multiplier), and agnostic policies. The `vl` CSR specifies how many elements to process.

### Element Width (SEW) and Group Multiplier (LMUL)

The `groupX8_` field stores the group multiplier scaled by 8 to avoid fractions:

*   LMUL=8 → `groupX8_`=64
*   LMUL=1 → `groupX8_`=8
*   LMUL=1/8 → `groupX8_`=1

**Sources:**[VecRegs.hpp 37-79](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L37-L79)[VecRegs.hpp 354-361](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L354-L361)

### Configuration Instructions (vsetvl, vsetvli, vsetivli)

These instructions configure the vector unit by updating `vtype` and `vl`:

| Instruction | Encoding | AVL Source | vtype Source |
| --- | --- | --- | --- |
| `vsetvl rd, rs1, rs2` | R-type | x[rs1] | x[rs2] |
| `vsetvli rd, rs1, imm` | I-type | x[rs1] | immediate |
| `vsetivli rd, uimm, imm` | I-type | uimm (0-31) | immediate |

## Element Operations and Masking

Vector instructions process elements from `vstart` to `vl-1`. Each element may be active (processed) or inactive (preserved/set to 1s) based on the mask register and agnostic policies.

### Active Element Determination

The `isDestActive()` method implements this logic:

**Sources:**[VecRegs.hpp 704-723](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L704-L723)

### Tail and Mask Agnostic Policies

Agnostic policies are configured via `vtype`:

*   **Tail Agnostic (TA)**: Elements at indices `ix >= vl`
*   **Mask Agnostic (MA)**: Masked-off elements

| Policy Bit | Value 0 | Value 1 |
| --- | --- | --- |
| TA bit (vtype[6]) | Preserve tail elements | Tail agnostic |
| MA bit (vtype[7]) | Preserve masked elements | Mask agnostic |
| `tailAgnOnes_` config | Preserve undisturbed | Write all 1s |
| `maskAgnOnes_` config | Preserve undisturbed | Write all 1s |

Configuration methods:

*   `configTailAgnosticAllOnes(flag)` - Sets policy for tail elements
*   `configMaskAgnosticAllOnes(flag)` - Sets policy for masked elements
*   `configUpdateWholeMask(flag)` - Controls whether mask instructions update entire register

## Vector Operation Patterns

Vector instructions follow standard naming patterns based on operand types. The `Hart` class implements type-dispatching wrappers that call templated worker functions.

### Operation Type Dispatch

**Sources:**[vector.cpp 1196-1530](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1196-L1530)

### Vector-Vector Operations (vv)

Template: `vop_vv<ELEM_TYPE>(vd, vs1, vs2, group, start, elems, masked, op)`

Example: `vadd.vv vd, vs1, vs2, vm`

```
for (ix = start; ix < elems; ++ix) {
    if (isDestActive(vd, ix, group, masked, dest)) {
        read(vs1, ix, group, e1);
        read(vs2, ix, group, e2);
        dest = op(e1, e2);
    }
    write(vd, ix, group, dest);
}
```

Public wrappers:

*   `execVop_vv(di, op)` - Signed integer ops
*   `execVopu_vv(di, op)` - Unsigned integer ops

**Sources:**[vector.cpp 1196-1360](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1196-L1360)

### Vector-Scalar Operations (vx)

Template: `vop_vx<ELEM_TYPE>(vd, vs1, e2, group, start, elems, masked, op)`

Example: `vadd.vx vd, vs1, rs2, vm`

```
SRV e2 = intRegs_.read(rs2);
for (ix = start; ix < elems; ++ix) {
    if (isDestActive(vd, ix, group, masked, dest)) {
        read(vs1, ix, group, e1);
        dest = op(e1, e2);
    }
    write(vd, ix, group, dest);
}
```

The scalar operand `e2` is read once from integer register `rs2` and used for all elements.

**Sources:**[vector.cpp 1254-1446](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1254-L1446)

### Vector-Immediate Operations (vi)

Template: Same as vx, but immediate is sign-extended

Example: `vadd.vi vd, vs1, imm, vm`

```
int32_t imm = di->op2As<int32_t>();
for (ix = start; ix < elems; ++ix) {
    if (isDestActive(vd, ix, group, masked, dest)) {
        read(vs1, ix, group, e1);
        dest = op(e1, imm);
    }
    write(vd, ix, group, dest);
}
```

**Sources:**[vector.cpp 1452-1530](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1452-L1530)

### Floating-Point Vector Operations

FP operations use `vfop_vv()` which tracks per-element FP exception flags:

```
for (ix = start; ix < elems; ++ix) {
    if (isDestActive(vd, ix, group, masked, dest)) {
        read(vs1, ix, group, e1);
        read(vs2, ix, group, e2);
        dest = fop(e1, e2);
        URV flags = activeSimulatorFpFlags();  // FP exceptions
        fpFlags_.push_back(flags);
    } else {
        fpFlags_.push_back(0);
    }
    write(vd, ix, group, dest);
}
```

The `fpFlags_` vector accumulates exception flags for each element, later merged into the `fflags` CSR.

**Sources:**[vector.cpp 1224-1250](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1224-L1250)

## Widening and Narrowing Operations

Widening operations produce results with 2×SEW and 2×LMUL. Narrowing operations consume 2×SEW operands and produce SEW results.

### Widening Operations

Widening instruction families:

*   **vw*add/sub.vv/vx** - Both operands narrow (widened to dest width)
*   **vw*add/sub.wv/wx** - First operand wide, second narrow
*   **vw*mul/mulsu.vv/vx** - Multiply widening

**Sources:**[vector.cpp 1590-2195](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1590-L2195)

### Double-Wide Type Mapping

The `makeDoubleWide_t<T>` template provides type promotion:

| Source Type | Double-Wide Type |
| --- | --- |
| `int8_t` | `int16_t` |
| `int16_t` | `int32_t` |
| `int32_t` | `int64_t` |
| `int64_t` | `Int128` |
| `Float16` | `float` |

**Sources:**[vector.cpp 42-44](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L42-L44)

### Overlap Rules for Widening

Widening operations have specific overlap constraints:

| Operation Type | Overlap Rule |
| --- | --- |
| `checkVecOpsVsEmulW0(vd, vs1, vs2, group)` | Dest (2×EMUL) can overlap sources (EMUL) only at highest dest registers if EMUL≥1 |
| `checkVecOpsVsEmulW0W1(vd, vs1, vs2, group)` | Dest and vs1 both 2×EMUL, vs2 is EMUL |
| `checkVecOpsVsEmulW1(vd, vs1, vs2, group)` | vs1 is 2×EMUL (wide), vd and vs2 are EMUL |

## Vector Load/Store Operations

Vector load/store instructions transfer data between memory and vector registers. They support unit-stride, strided, and indexed addressing modes with optional segment (field) access.

### Load/Store Element Tracking

The `VecLdStInfo` structure tracks memory operations:

| Field | Type | Purpose |
| --- | --- | --- |
| `vec_` | `uint32_t` | Base vector register |
| `elemSize_` | `uint8_t` | Element size in bytes |
| `eew_` | `ElementWidth` | Effective element width |
| `fieldCount_` | `uint8_t` | Number of fields (1-8) |
| `elems_` | `std::vector<VecLdStElem>` | Per-element info |

Each `VecLdStElem` records:

*   `pa_` - Physical address
*   `va_` - Virtual address
*   `ix_` - Element index
*   `field_` - Field number (for segments)
*   `stData_` - Store data value

**Sources:**[VecLdStInfo.hpp 18-62](https://github.com/tenstorrent/whisper/blob/733fd921/VecLdStInfo.hpp#L18-L62)

### Indexed Load/Store Validation

For indexed operations (e.g., `vluxei32.v`), the simulator validates:

```
bool checkVecLdStIndexedInst(di, vd, vi, offsetWidth, offsetGroupX8, fieldCount) {
    if (!isLegalVecLdSt(di, eew, emul))
        return false;
    
    // Prevent vd/vi overlap for indexed segment loads
    if (fieldCount > 1) {
        if (overlap(vd, segGroup, vi, offsetGroup))
            return false;
    } else {
        if (!checkDestSourceOverlap(vd, sew, groupX8, vi, offsetWidth, offsetGroupX8))
            return false;
    }
    
    return true;
}
```

**Sources:**[vector.cpp 902-944](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L902-L944)

## Mask Operations

Mask operations produce single-bit results per element, stored in mask register format (v0 by default).

### Mask Register Format

Mask registers store one bit per element:

*   Bit 0 of byte 0 = element 0
*   Bit 1 of byte 0 = element 1
*   ...
*   Bit 0 of byte 1 = element 8

### Mask Operation Validation

Key constraint: Mask destination (EEW=1) can only overlap source (EEW=SEW) if overlap is at first source register (`dest == src`).

**Sources:**[vector.cpp 563-652](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L563-L652)

### Mask Write Methods

The `VecRegs` class provides mask-specific accessors:

`void writeMaskRegister(uint32_t maskReg, uint32_t i, bool value);void readMaskRegister(uint32_t maskReg, uint32_t i, bool& value) const;bool isActive(uint32_t maskReg, uint32_t ix) const;`
The `updateWholeMask_` configuration flag controls whether mask instructions write the entire destination register or only up to `vl`.

## Reduction Operations

Reduction operations accumulate vector elements into a scalar result in element 0 of the destination register.

### Reduction Validation

```
bool checkRedOpVsEmul(di) {
    // Reductions require vstart=0
    if (vstart > 0)
        return false;
    
    // V0 cannot be used with different EEWs
    if (masked && (vs1 == 0 || vs2 == 0))
        return false;
    
    // vs1 must be multiple of LMUL
    if ((vs1 & (lmul-1)) != 0)
        return false;
    
    // Track: vd EMUL=1, vs1 EMUL=lmul, vs2 EMUL=1
    setOpEmul(1, lmul, 1);
    return true;
}
```

Wide reductions (e.g., `vwredsumu`) have additional checks via `checkWideRedOpVsEmul()`:

*   vs2 and vs1 cannot overlap with different EEWs
*   vs2[0] (scalar) cannot overlap any part of vs1 group

**Sources:**[vector.cpp 403-476](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L403-L476)

### Reduction Patterns

Reductions are tracked using the `Step` structure to record intermediate operations:

| Operation | Step Type |
| --- | --- |
| Cross-register reduction | `CrossRegRed` |
| Adjacent element reduction | `AdjacRed` |
| Strided reduction | `StrideRed` |
| Scalar reduction | `ScalarRed` |

The `fpUnorderedSumTreeRed_` configuration flag controls whether unordered FP reductions use tree-based or sequential accumulation.

**Sources:**[VecRegs.hpp 549-580](https://github.com/tenstorrent/whisper/blob/733fd921/VecRegs.hpp#L549-L580)

## vstart Handling

The `vstart` CSR indicates where to begin/resume execution of a vector instruction. It is used for:

1.   **Traps**: If an exception occurs at element N, `vstart` is set to N
2.   **Resume**: Next vector instruction continues from element `vstart`
3.   **Completion**: `vstart` is cleared to 0 after successful completion

### vstart Constraints

Configuration flags:

*   `trapNonZeroVstart_` - Trap on non-zero vstart for arithmetic ops
*   `trapOobVstart_` - Trap if vstart ≥ VLMAX

**Sources:**[vector.cpp 193-197](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L193-L197)[vector.cpp 240-242](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L240-L242)[vector.cpp 405-411](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L405-L411)

## Vector Status Tracking

The VS field in `mstatus` (and `vsstatus` in VS-mode) tracks the state of vector registers:

| VS Value | VecStatus | Meaning |
| --- | --- | --- |
| 0 | Off | Vector unit disabled |
| 1 | Initial | Vector registers in initial state |
| 2 | Clean | Vector registers modified but saved |
| 3 | Dirty | Vector registers modified |

The `markVsDirty()` method sets VS=Dirty when:

*   Vector registers are written
*   Vector CSRs (VL, VTYPE, VSTART) are modified

The `alwaysMarkDirty_` flag controls whether VS is marked dirty even when no register is written (e.g., all elements masked).

**Sources:**[vector.cpp 138-155](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L138-L155)[vector.cpp 1051-1066](https://github.com/tenstorrent/whisper/blob/733fd921/vector.cpp#L1051-L1066)

| Category | Instructions | Handler Pattern |
| --- | --- | --- |
| **Arithmetic** | vadd, vsub, vand, vor, vxor, etc. | `execVop_vv/vx/vi` |
| **Compare** | vmseq, vmsne, vmslt, etc. | `checkVecMaskInst` → write mask |
