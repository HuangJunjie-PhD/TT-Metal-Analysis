This page provides comprehensive guidance for diagnosing and resolving common issues that may arise when setting up, configuring, and using TT-Studio. It covers deployment failures, hardware detection issues, and service-specific errors.

## Service Architecture Overview

TT-Studio is a multi-container application where services communicate over a dedicated Docker network. Understanding these connections is the first step in troubleshooting.

Sources: [app/docker-compose.yml 5-141](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L5-L141)[app/backend/docker_control/docker_control_client.py 32-38](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L32-L38)

## System Startup Issues

### Common Startup Problems

| Issue | Symptoms | Solution |
| --- | --- | --- |
| **Docker Permission Denied** | Error accessing Docker API or socket. | Add user to docker group: `sudo usermod -aG docker $USER` and log out/in. [README.md 31-46](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L31-L46) |
| **Trailing Slash Error** | `RuntimeError` during POST requests to backend. | Ensure API calls end with `/` (e.g., `/docker/deploy/`). [app/backend/README.md 118-135](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/README.md?plain=1#L118-L135) |
| **Missing JWT Secret** | `ValueError: DOCKER_CONTROL_JWT_SECRET... is required` | Ensure `.env` is populated from `.env.default`. [app/backend/docker_control/docker_control_client.py 35-36](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L35-L36) |
| **Port Conflicts** | `Bind for 0.0.0.0:8000 failed`. | Check for existing processes: `netstat -tuln | grep 8000`. |

### Docker Control Service (DCS) Failures

The Docker Control Service is a FastAPI-based security layer [app/backend/docker_control/docker_control_client.py 8-10](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L8-L10) If the backend cannot communicate with it, model deployment will fail.

**Diagnostic Steps:**

1.   Verify DCS is running on port 8002.
2.   Check `DOCKER_CONTROL_JWT_SECRET` in `.env`. The backend uses this to sign tokens for DCS [app/backend/docker_control/docker_control_client.py 41-50](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L41-L50)
3.   Check logs for "Docker Control Service request failed" [app/backend/docker_control/docker_control_client.py 62-63](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L62-L63)

## Hardware Integration Issues

### Tenstorrent Hardware Detection

TT-Studio automatically detects hardware by checking for `/dev/tenstorrent`[startup.sh 112-155](https://github.com/tenstorrent/tt-studio/blob/a6d347af/startup.sh#L112-L155)

Sources: [startup.sh 112-155](https://github.com/tenstorrent/tt-studio/blob/a6d347af/startup.sh#L112-L155)[app/backend/docker_control/urls.py 42](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/urls.py#L42-L42)[README.md 159-161](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L159-L161)

**Common Hardware Errors:**

*   **Missing Drivers:** If `lspci | grep -i tenstorrent` shows the card but `/dev/tenstorrent` is missing, the drivers are not installed.
*   **Chip Allocation Blocked:** If a previous deployment crashed, it might leave a "starting" record that blocks a chip slot. The system attempts to clean these after 10 minutes [app/backend/docker_control/health_monitor.py 19-44](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/health_monitor.py#L19-L44)

## Model Deployment Troubleshooting

### Deployment Workflow Failures

Deployments are managed by the `DeployView` and tracked via `job_id`[app/backend/docker_control/urls.py 35-38](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/urls.py#L35-L38)

| Stage | Common Failure | Fix |
| --- | --- | --- |
| **Image Pull** | `Image not found` or `Auth failed`. | Ensure `HF_TOKEN` is set in `.env` for gated models. |
| **Container Start** | Container exits immediately. | Check model logs: `docker logs <container_name>`. Often due to OOM or missing weights. |
| **Health Check** | Status stuck in "starting". | The `health_monitor` thread checks container status every 60s [app/backend/docker_control/health_monitor.py 101-114](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/health_monitor.py#L101-L114) |

### Manual Cleanup

If the UI becomes unresponsive or containers are stuck:

1.   Stop all containers: `python3 run.py --cleanup`[README.md 81-86](https://github.com/tenstorrent/tt-studio/blob/a6d347af/README.md?plain=1#L81-L86)
2.   Force remove specific model containers: `docker rm -f <container_id>`.
3.   Prune networks: `docker network rm tt_studio_network`.

## Persistent Storage & Permissions

TT-Studio uses a persistent volume for model weights and configuration [app/README.md 7-11](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/README.md?plain=1#L7-L11)

**File Permission Issues:** Model containers run as a non-root user (UID 1000). The backend explicitly attempts to fix permissions during initialization [app/backend/README.md 187-205](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/README.md?plain=1#L187-L205) If you manually copy weights, ensure they are owned by the correct UID:

`sudo chown -R 1000:1000 ./tt_studio_persistent_volume/`
Sources: [app/backend/README.md 187-205](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/README.md?plain=1#L187-L205)
