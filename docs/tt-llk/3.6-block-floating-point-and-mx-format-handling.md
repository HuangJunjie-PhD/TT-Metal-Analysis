## Block Floating-Point Format Overview

Block Floating-Point (BFP) formats reduce memory bandwidth by storing a shared exponent for groups of mantissas. The tt-llk library primarily supports Bfp8_b format:

| Format | Mantissa Bits | Elements per Block | Block Size |
| --- | --- | --- | --- |
| Bfp8_b | 8 bits (sign + 7-bit magnitude) | 16 elements | 1 byte exponent + 16 bytes mantissas |

Each BFP8_b block contains:

*   1 shared 8-bit exponent (max value from all elements in block)
*   16 mantissas with 1 sign bit + 7 magnitude bits

A face (16×16 = 256 elements) contains 16 blocks. The hardware requires a minimum of 16 exponents per tile (`MIN_BFP_EXPONENTS`), padding with zeros if fewer blocks are present.

## BFP8_b Unpacking Implementation

The `unpack_bfp8_b` function reverses the packing process, converting BFP8_b data from L1 memory back to floating-point tensors:

**Unpacking Algorithm:**

**Unpacking Details:**

| Aspect | Implementation |
| --- | --- |
| Exponent section | Read at least `MIN_BFP_EXPONENTS` bytes, use only `actual_exponents` |
| Bias removal | Subtract 127 from exponent before scaling |
| Mantissa decode | Sign-magnitude: `(-1)^sign * fractional_value` |
| Caching | Reuses decoded (exponent, mantissa) pairs for efficiency |
| Output format | `torch.bfloat16` tensor |

## MX Format Overview

MX (Microscaling) formats follow the OCP MX specification and provide 8-bit floating-point representations with two variants:

| Format | Element Type | Exponent Bits | Mantissa Bits | Max Normal Value | Description |
| --- | --- | --- | --- | --- | --- |
| MxFp8R | E5M2 | 5 | 2 | 57,344 | Wider exponent range |
| MxFp8P | E4M3 | 4 | 3 | 448 | More mantissa precision |

Both formats use:

*   **Block size**: 32 elements per block (per OCP MX specification)
*   **Scale format**: E8M0 (8-bit exponent only, no mantissa, bias=127)
*   **Memory layout**: Fully separated `[all_scales][all_elements]`
*   **Blocks per face**: 8 blocks (256 elements / 32 elements per block)

The hardware unpacks MX formats from L1 memory to `Float16_b` (bfloat16) for computation in source registers.

## MX Format Packing Implementation

The `pack_mxfp8r` and `pack_mxfp8p` functions use a shared `_pack_mxfp8` helper that implements the OCP MX specification:

**MX Packing Algorithm:**

**Key Packing Details:**

| Aspect | Implementation |
| --- | --- |
| Block size | 32 elements (per OCP spec) |
| Scale calculation | `ceil(log2(max_abs_value / format_max))` |
| E8M0 encoding | `clamp(exponent, -127, 127) + 127` |
| Special values | Zero/NaN→127, Inf→254 |
| FP8 conversion | Uses `ml_dtypes.float8_e5m2` or `float8_e4m3fn` |
| Layout | `[all 32 scales][all 1024 elements]` |

## MX Format Unpacking Implementation

The `unpack_mxfp8r` and `unpack_mxfp8p` functions use a shared `_unpack_mxfp8` helper that reverses the packing process:

**MX Unpacking Algorithm:**

**Key Unpacking Details:**

| Aspect | Implementation |
| --- | --- |
| Scale extraction | First `num_faces * 8` bytes (E8M0 format) |
| Scale decoding | `2^(e8m0_value - 127)` or 0 if NaN/zero scale |
| FP8 conversion | Uses `ml_dtypes.float8_e5m2` or `float8_e4m3fn` |
| Vectorized scaling | Multiply all elements by scale factors at once |
| Output format | `torch.bfloat16` tensor |

## Data Format Inference for BFP and MX Formats

The format inference system handles BFP and MX formats with special rules:

**Format Inference Rules:**

**Tolerance Configuration:**

| Format | Absolute Tolerance | Relative Tolerance | Notes |
| --- | --- | --- | --- |
| Bfp8_b | 0.1 | 0.2 | Higher tolerance due to shared exponent |
| MxFp8R | 0.2 | 0.3 | E5M2: wider exponent range |
| MxFp8P | 0.2 | 0.3 | E4M3: more mantissa precision |
| Float16_b | 0.05 | 0.05 | Standard bfloat16 |

**Key Inference Behaviors:**

1.   **MX Format Unpacking**: Hardware always unpacks MX→`Float16_b` for source registers
2.   **BFP Tolerance Adjustment**: PCC threshold decreases with multiple L1-to-L1 iterations
3.   **Format Compatibility**: Checked by `infer_math_format` - prevents mixing exponent families
