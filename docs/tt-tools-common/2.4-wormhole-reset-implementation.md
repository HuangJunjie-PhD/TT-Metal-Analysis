# Wormhole Reset Implementation

Relevant source files
*   [CHANGELOG.md](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1)
*   [tt_tools_common/reset_common/wh_reset.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py)

## Purpose and Scope

This document details the Wormhole-specific reset implementation in `tt-tools-common`. It covers the `WHChipReset` class, which orchestrates PCIe link resets and ARC message sequences for Wormhole chips. This page focuses on the legacy implementation path and delegation logic specific to Wormhole.

For the modern unified reset implementation that supersedes the legacy path, see [ChipReset: Modern Unified Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset). For architecture and design patterns common to all chip types, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns).

**Sources:**[tt_tools_common/reset_common/wh_reset.py 1-181](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L1-L181)

## Architecture Overview

The Wormhole reset system implements a conditional delegation pattern where the `WHChipReset` class routes to either the modern `ChipReset` implementation or a legacy reset sequence based on driver version and virtualization context.

### Reset Path Selection Logic

**Diagram: Wormhole Reset Delegation Logic**

The decision to use modern or legacy reset is determined at [wh_reset.py 89-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L89-L90) where driver version is checked using `is_driver_version_at_least()` or Xen HVM detection via `check_xen_hvm()`. The legacy path requires minimum driver 1.26.0 (checked at [wh_reset.py 101](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L101-L101)) and explicitly blocks ARM platforms (checked at [wh_reset.py 105-112](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L105-L112)).

**Sources:**[tt_tools_common/reset_common/wh_reset.py 68-112](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L68-L112)

## WHChipReset Class

The `WHChipReset` class provides the public interface for Wormhole chip resets. It is defined at [wh_reset.py 28-29](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L28-L29) and contains constants, IOCTL wrappers, and the main reset orchestration logic.

### Class Constants

| Constant | Value | Purpose |
| --- | --- | --- |
| `TENSTORRENT_IOCTL_MAGIC` | `0xFA` | Magic number for ioctl commands |
| `TENSTORRENT_IOCTL_RESET_DEVICE` | `(0xFA << 8) | 6` | IOCTL command code for device reset |
| `TENSTORRENT_RESET_DEVICE_RESTORE_STATE` | `0` | Flag to restore PCIe state after reset |
| `TENSTORRENT_RESET_DEVICE_RESET_PCIE_LINK` | `1` | Flag to reset PCIe link |
| `A3_STATE_PROP_TIME` | `0.03` | Wait time (seconds) for A3 state propagation |
| `POST_RESET_MSG_WAIT_TIME` | `2` | Wait time (seconds) after reset messages |
| `MSG_TRIGGER_SPI_COPY_LtoR` | `0x50` | ARC message: trigger SPI copy left to right |
| `MSG_TYPE_ARC_STATE3` | `0xA3` | ARC message: enter A3 safe state |
| `MSG_TYPE_TRIGGER_RESET` | `0x56` | ARC message: trigger reset sequence |

These constants are defined at [wh_reset.py 31-40](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L31-L40) and specify the low-level protocol for communicating with the kernel driver and ARC firmware.

### Primary Method

The `full_lds_reset()` method is the main entry point:

`def full_lds_reset(    self,    pci_interfaces: List[int],    reset_m3: bool = False,    silent: bool = False,    secondary_bus_reset: bool = True,) -> List[PciChip]`
**Parameters:**

*   `pci_interfaces`: List of PCI interface IDs to reset
*   `reset_m3`: Whether to perform M3 board-level reset (default: `False`)
*   `silent`: Whether to suppress console output (default: `False`)
*   `secondary_bus_reset`: Reserved for compatibility with `ChipReset` interface (not used in legacy path)

**Returns:** List of `PciChip` objects representing the reset devices

**Sources:**[tt_tools_common/reset_common/wh_reset.py 28-86](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L28-L86)

## Legacy Reset Sequence

When the legacy path is taken (driver < 2.4.1 and not in Xen HVM), the reset sequence follows a three-phase protocol involving PCIe link manipulation and ARC firmware commands.

### Reset Sequence Phases

**Diagram: Legacy Wormhole Reset Sequence**

**Phase 1 (PCIe Link Reset):** Calls `reset_device_ioctl()` with flag `TENSTORRENT_RESET_DEVICE_RESET_PCIE_LINK` for each interface ([wh_reset.py 122-124](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L122-L124)). This puts the PCIe link into reset state through the kernel driver.

**Phase 2 (ARC Firmware Reset):** For each chip, the sequence:

1.   Reads the reference clock counter before reset ([wh_reset.py 131-132](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L131-L132))
2.   Sends `MSG_TYPE_ARC_STATE3` (0xA3) to enter A3 safe state ([wh_reset.py 143](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L143-L143))
3.   Waits 0.03 seconds for state propagation ([wh_reset.py 144](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L144-L144))
4.   Sends `MSG_TYPE_TRIGGER_RESET` (0x56) with optional `arg0=3` for M3 reset ([wh_reset.py 146-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L146-L149))

**Phase 3 (PCIe Restore):** Calls `reset_device_ioctl()` with flag `TENSTORRENT_RESET_DEVICE_RESTORE_STATE` for each interface ([wh_reset.py 154-156](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L154-L156)) and validates the reset by checking that reference clock counter decreased ([wh_reset.py 157-164](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L157-L164)).

**Sources:**[tt_tools_common/reset_common/wh_reset.py 121-180](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L121-L180)

## IOCTL Interface

The `reset_device_ioctl()` method provides a Python wrapper around the kernel driver's reset ioctl interface.

### IOCTL Structure

**Diagram: IOCTL Communication Flow**

The implementation at [wh_reset.py 42-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L42-L66) follows this protocol:

1.   **Open device file:** Opens `/dev/tenstorrent/{interface_id}` with flags `O_RDWR | O_CLOEXEC` ([wh_reset.py 43-46](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L43-L46))
2.   **Pack input structure:** Creates a buffer with format `"IIII"` containing output size, flags, and two zero fields ([wh_reset.py 48-56](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L48-L56))
3.   **Execute ioctl:** Calls `fcntl.ioctl()` with command `TENSTORRENT_IOCTL_RESET_DEVICE` ([wh_reset.py 57-59](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L57-L59))
4.   **Unpack output:** Extracts the result code from the output portion of the buffer ([wh_reset.py 61-62](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L61-L62))
5.   **Return status:** Returns `True` if result is 0, `False` otherwise ([wh_reset.py 64](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L64-L64))

The file descriptor is guaranteed to be closed via a `finally` block ([wh_reset.py 65-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L65-L66)).

**Sources:**[tt_tools_common/reset_common/wh_reset.py 42-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L42-L66)

## ARC Message Sequence

The ARC (Accelerated RISC Core) firmware running on the Wormhole chip must be instructed to perform reset operations through a specific message protocol.

### A3 State and Reset Messages

The reset sequence uses two critical ARC messages:

#### MSG_TYPE_ARC_STATE3 (0xA3)

This message instructs the ARC firmware to enter A3 safe state, where no pending regulator requests exist. The call is synchronous:

`chip.arc_msg(self.MSG_TYPE_ARC_STATE3, wait_for_done=True)`
After sending, the implementation waits `A3_STATE_PROP_TIME` (0.03 seconds) for the state to propagate ([wh_reset.py 143-144](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L143-L144)). This ensures the chip is in a stable state before triggering the actual reset.

#### MSG_TYPE_TRIGGER_RESET (0x56)

This message triggers the reset sequence. The call is asynchronous (`wait_for_done=False`) because the chip will be resetting and cannot acknowledge:

`# Standard resetchip.arc_msg(self.MSG_TYPE_TRIGGER_RESET, wait_for_done=False) # M3 board-level resetchip.arc_msg(self.MSG_TYPE_TRIGGER_RESET, wait_for_done=False, arg0=3)`
When `reset_m3=True`, the implementation passes `arg0=3` to request an M3 board-level reset rather than just a chip reset ([wh_reset.py 146-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L146-L149)).

After all chips have been sent reset messages, the implementation waits `POST_RESET_MSG_WAIT_TIME` (2 seconds) for the reset operations to complete ([wh_reset.py 151](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L151-L151)).

**Sources:**[tt_tools_common/reset_common/wh_reset.py 143-151](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L143-L151)

## Reference Clock Validation

The Wormhole reset implementation validates that a reset actually occurred by comparing the ARC reference clock counter values before and after the reset sequence.

### Validation Logic

**Diagram: Reference Clock Reset Validation**

The validation process:

1.   **Pre-reset capture:** Before sending ARC messages, read and store each chip's reference clock counter using `read_refclk_counter(chip)` ([wh_reset.py 131-132](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L131-L132))
2.   **Post-reset read:** After PCIe state restoration, read the current reference clock value ([wh_reset.py 157](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L157-L157))
3.   **Comparison:** Verify that `refclk_list[i] < current_refclk` - if the clock did NOT decrease, the reset failed ([wh_reset.py 158-164](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L158-L164))
4.   **Error handling:** If any chip fails validation, print an error and set the `fail` flag ([wh_reset.py 159-164](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L159-L164))
5.   **Exit on failure:** If `fail` is set, exit with status code 1 ([wh_reset.py 166-172](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L166-L172))

The reference clock counter is expected to be **smaller** after a successful reset because the counter resets to zero and begins counting from the start. If the counter is larger or equal, it indicates the reset did not occur.

**Sources:**[tt_tools_common/reset_common/wh_reset.py 131-180](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L131-L180)[tt_tools_common/utils_common/tools_utils.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py) (for `read_refclk_counter`)

## Platform Restrictions

The legacy Wormhole reset implementation has specific platform restrictions that prevent operation on certain architectures.

### ARM Platform Block

ARM and AArch64 platforms handle PCIe device rescans differently than x86, causing Wormhole device resets to fail. The implementation explicitly checks for ARM platforms and exits:

`platform = get_host_info()["Platform"]if platform.startswith("arm") or platform.startswith("aarch"):    print(        CMD_LINE_COLOR.RED,        "Cannot perform WH board reset on Arm systems, please reboot the system to reset the boards. Exiting...",        CMD_LINE_COLOR.ENDC,    )    sys.exit(1)`
This check occurs at [wh_reset.py 105-112](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L105-L112) immediately after driver version validation. The restriction applies **only to the legacy path**; the modern `ChipReset` implementation may handle ARM differently (see [ChipReset: Modern Unified Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset)).

### Driver Version Requirements

The legacy reset requires kernel driver version >= 1.26.0, enforced by calling `check_driver_version(operation="board reset")` at [wh_reset.py 101](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L101-L101) This minimum version is required to support the IOCTL commands for PCIe link reset and state restoration.

**Sources:**[tt_tools_common/reset_common/wh_reset.py 101-112](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L101-L112)[CHANGELOG.md 102-106](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L102-L106)

## Migration to ChipReset

The Wormhole reset implementation migrates to the modern `ChipReset` implementation when driver version >= 2.4.1 or when running in Xen HVM mode.

### Delegation Implementation

The delegation occurs at the start of `full_lds_reset()`:

`# Use new reset for driver version >= 2.4.1 or if in Xen HVM mode. Xen mode also requires newer driver.if is_driver_version_at_least(get_driver_version(), "2.4.1") or check_xen_hvm():    return ChipReset().full_lds_reset(pci_interfaces, reset_m3=reset_m3, silent=silent, secondary_bus_reset=secondary_bus_reset)`
This check at [wh_reset.py 89-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L89-L90) immediately delegates to `ChipReset` if conditions are met, bypassing all legacy logic.

### Migration Timeline

| Version | Change | Reference |
| --- | --- | --- |
| 1.4.3 (May 2024) | Added ARM platform check and Xen HVM detection | [CHANGELOG.md 80-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L80-L85) |
| 1.4.2 (April 2024) | Added `silent` flag for versatility in other tools | [CHANGELOG.md 87-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L87-L90) |
| 1.4.0 (March 2024) | Updated minimum driver version to 1.26 for LDS reset | [CHANGELOG.md 97-106](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L97-L106) |
| 1.1.0 (February 2024) | Added refclk check to WH link reset | [CHANGELOG.md 126-131](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L126-L131) |

The migration strategy allows gradual adoption of the modern reset implementation while maintaining backward compatibility for systems with older kernel drivers. When the legacy path is used, a warning is printed encouraging users to upgrade ([wh_reset.py 92-98](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L92-L98)).

### Implications for Consumers

Tools consuming `WHChipReset` should:

*   **Update drivers:** Upgrade to tt-kmd >= 2.4.1 to use the modern implementation
*   **Handle warnings:** The legacy path warning is informational, not fatal
*   **Test both paths:** If supporting multiple driver versions, test both legacy and modern reset behavior
*   **Plan for deprecation:** The legacy implementation will be removed in a future version

## Error Handling

The Wormhole reset implementation includes several failure modes and error handling strategies.

### Fatal Errors

| Error Condition | Location | Action |
| --- | --- | --- |
| ARM platform detected | [wh_reset.py 107-112](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L107-L112) | Print error, `sys.exit(1)` |
| Driver version < 1.26.0 | `check_driver_version()` call | Print error, `sys.exit(1)` |
| Cannot read refclk before reset | [wh_reset.py 131-141](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L131-L141) | Print "failed to recover" message, `sys.exit(1)` |
| Refclk validation failed | [wh_reset.py 166-172](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L166-L172) | Print "reset failed" warning, `sys.exit(1)` |

### Non-Fatal Warnings

*   **Legacy path usage:** Prints yellow warning when driver < 2.4.1 ([wh_reset.py 92-98](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L92-L98))
*   **Individual chip failures:** Prints red error for each chip with failed refclk validation ([wh_reset.py 159-164](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/wh_reset.py#L159-L164))

The implementation treats refclk validation failures as fatal, exiting with code 1 if any chip fails validation. This ensures that consumers do not proceed with partially-reset hardware.

**Sources:**[tt_tools_common/reset_common/wh_reset.py 107-172](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L107-L172)
