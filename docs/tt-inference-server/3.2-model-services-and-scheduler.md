The Model Services and Scheduler layer forms the core orchestration engine of the Python-based `tt-media-server`. It is responsible for managing the lifecycle of hardware-bound worker processes, scheduling inference tasks across multiple Tenstorrent devices, and providing specialized service logic for different modalities (Audio, Image, Video).

## System Architecture

The architecture follows a multi-process producer-consumer pattern where the FastAPI application (Producer) submits requests to a centralized `Scheduler`, which then dispatches them to `device_worker` processes (Consumers) managing individual Tenstorrent devices.

## Scheduler and Worker Lifecycle

The `Scheduler` class [tt-media-server/model_services/scheduler.py 24](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L24-L24) manages the pool of `device_worker` processes. It calculates the required worker count based on the available `device_ids` provided in the system settings [tt-media-server/model_services/scheduler.py 55-58](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L55-L58)

### Worker Initialization

1.   **Queue Setup**: The scheduler initializes a `task_queue` for incoming requests and individual `result_queues_by_worker`[tt-media-server/model_services/scheduler.py 32-53](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L32-L53)
2.   **Process Spawning**: Workers are started via `_start_worker` using the `multiprocessing.Process` API [tt-media-server/model_services/scheduler.py 150-194](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L150-L194) Depending on configuration, it uses either `device_worker`[tt-media-server/device_workers/device_worker.py 14](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/device_workers/device_worker.py#L14-L14) or `device_worker_dynamic_batch`[tt-media-server/device_workers/device_worker_dynamic_batch.py 71](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/device_workers/device_worker_dynamic_batch.py#L71-L71)
3.   **Warmup**: Each `device_worker` initializes its specific `ModelRunner` and performs a warmup inference. A `warmup_signals_queue` is used to signal the scheduler when a worker is ready [tt-media-server/model_services/scheduler.py 214-230](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L214-L230)
4.   **Monitoring**: An `error_listener` and `result_listener` run as background `asyncio` tasks to monitor worker health and route inference results [tt-media-server/model_services/scheduler.py 121-130](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L121-L130)

### Fault Tolerance and Reset

The scheduler implements several mechanisms for system stability:

*   **Worker Restart**: If a worker process dies, the `monitor_workers` loop attempts to restart it up to `max_worker_restart_count` times [tt-media-server/model_services/scheduler.py 284-315](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L284-L315)
*   **Deep Reset**: Triggered via `deep_restart_workers`, this performs a hardware-level reset using the command defined in `settings.reset_device_command` (typically `tt-smi -r`) and re-initializes all workers [tt-media-server/model_services/scheduler.py 237-261](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L237-L261)

## Model Services

Model services inherit from `BaseService`[tt-media-server/model_services/base_service.py 18](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/base_service.py#L18-L18) or `BaseJobService`[tt-media-server/model_services/video_service.py 48](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/video_service.py#L48-L48) and provide the bridge between the API layer and the Scheduler.

### BaseService Logic

The `BaseService` provides the standard `process_request` and `process_streaming_request` methods. It handles:

*   **Segmentation**: If a request contains multiple segments (e.g., multi-image generation), it dispatches them in parallel using `asyncio.gather` and merges them via `combine_results`[tt-media-server/model_services/base_service.py 52-63](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/base_service.py#L52-L63)
*   **Streaming**: `process_streaming` manages model-level streaming by listening to a local `asyncio.Queue` and yielding chunks to the client [tt-media-server/model_services/base_service.py 172-205](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/base_service.py#L172-L205)

### Image and Video Services (CPU Workload Handler)

Modality-specific services use `CpuWorkloadHandler`[tt-media-server/model_services/cpu_workload_handler.py 64](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/cpu_workload_handler.py#L64-L64) to offload non-TT tasks (like MP4 encoding or Base64 conversion) to a separate pool of CPU-bound processes.

## Implementation Details: Queues and IPC

The system supports multiple Inter-Process Communication (IPC) queue types via `get_queue` and `get_task_queue` in `simple_queue_factory.py` to handle different performance requirements.

| Queue Type | Implementation Class | Characteristics |
| --- | --- | --- |
| `MemoryQueue` | `MemoryQueue` | Standard `multiprocessing.Queue` behavior [tt-media-server/model_services/queues/memory_queue.py](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/queues/memory_queue.py) |
| `FasterFifo` | `TTFasterFifoQueue` | Optimized for high-throughput cross-process communication [tt-media-server/model_services/queues/tt_faster_fifo_queue.py](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/queues/tt_faster_fifo_queue.py) |
| `TTQueue` | `TTQueue` | Tenstorrent-optimized queue implementation [tt-media-server/model_services/queues/tt_queue.py](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/queues/tt_queue.py) |
| `BatchFifo` | `TTBatchFasterFifoQueue` | Optimized for dynamic batching workloads. |

## Health Monitoring and Telemetry

The server provides a detailed health status via `check_is_model_ready`[tt-media-server/model_services/base_service.py 87-99](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/base_service.py#L87-L99) This returns:

*   `model_ready`: Boolean status from the scheduler.
*   `queue_size`: Current number of pending tasks in the `task_queue`.
*   `worker_info`: Detailed status of each `device_worker` (PID, readiness, restart count, etc.) [tt-media-server/model_services/scheduler.py 403-408](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/scheduler.py#L403-L408)
*   **Prometheus Metrics**: The `TelemetryClient` records events like `MODEL_INFERENCE`, `DEVICE_WARMUP`, and `TOTAL_PROCESSING` to track latency and throughput [tt-media-server/telemetry/telemetry_client.py 182-209](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/telemetry/telemetry_client.py#L182-L209)
