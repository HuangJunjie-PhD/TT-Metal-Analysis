This document explains how tt-torch integrates with PyTorch's `torch.compile` system as a compilation backend. tt-torch provides two backend implementations:

*   **tt-legacy**: Dynamo-based backend using direct FX graph capture and transformation
*   **tt-experimental**: XLA-based backend leveraging torch-xla's PJRT infrastructure

This page covers the backend registration mechanism, configuration options, pass pipeline architecture, and the main compilation flows that bridge PyTorch models to Tenstorrent hardware execution.

For information about the compilation pipeline stages and MLIR transformations, see [Compilation Pipeline](https://deepwiki.com/tenstorrent/tt-torch/3.1-compilation-pipeline). For details about executor implementations and caching, see [Executor Framework](https://deepwiki.com/tenstorrent/tt-torch/3.3-executor-framework).

## Backend Registration and Discovery

tt-torch registers two compilation backends with PyTorch:

1.   **tt-legacy**: Registered via `torch_dynamo_backends` entry point in `setup.py`
2.   **tt-experimental**: Registered programmatically in `tt_torch.dynamo.experimental.xla_backend`

The registration process allows PyTorch to discover and invoke tt-torch backends when `torch.compile` is called with the appropriate backend name.

**Backend Registration and Discovery Flow**

Sources: [setup.py 225-228](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/setup.py#L225-L228)[tt_torch/dynamo/backend.py 312-313](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L312-L313)[tt_torch/__init__.py 26-28](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/__init__.py#L26-L28)

### tt-legacy Backend Registration

The `tt-legacy` backend is registered with the name `"tt-legacy"` and can be invoked using `torch.compile(model, backend="tt-legacy")`. The registration uses the `@register_backend` decorator from PyTorch's dynamo system and the `torch_dynamo_backends` entry point mechanism.

### tt-experimental Backend Registration

The `tt-experimental` backend integrates with torch-xla's infrastructure and is registered programmatically when the `tt_torch` package is imported. It leverages the PJRT plugin system for device communication.

## BackendOptions Configuration

The `BackendOptions` class serves as the primary configuration interface between user code and the tt-torch backend. It encapsulates compilation settings, device management, and caching options.

**BackendOptions Structure**

Sources: [tt_torch/dynamo/backend.py 21-54](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L21-L54)[tt_torch/tools/utils.py 300-340](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L300-L340)

The `BackendOptions` class manages several key aspects:

| Component | Purpose | Default |
| --- | --- | --- |
| `compiler_config` | Controls compilation behavior and optimization settings | `CompilerConfig()` |
| `devices` | Specifies target devices for execution | `[None]` |
| `async_mode` | Enables asynchronous execution across devices | `False` |
| `buffer_cache` | Caches runtime tensor buffers for reuse | `{}` |
| `constant_cache` | Caches graph constants between executors | `{}` |

The caching system in `BackendOptions` provides memory management for runtime tensors, automatically deallocating cached tensors when the options object is destroyed.

## tt-experimental Backend Flow

The `tt-experimental` backend takes a different approach by leveraging torch-xla's infrastructure. Instead of directly processing FX graphs, it uses the XLA compiler stack and PJRT runtime.

**tt-experimental Backend Architecture**

Sources: [tt_torch/__init__.py 30-74](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/__init__.py#L30-L74)[requirements.txt 4](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/requirements.txt#L4-L4)

The XLA backend flow:

1.   User calls `torch.compile` with `backend='tt-experimental'`
2.   torch-xla captures the model using its tracing infrastructure
3.   The PJRT plugin (`pjrt_plugin_tt.so`) routes compilation to tt-mlir
4.   If mesh configuration is enabled, automatic parallelization is applied
5.   Compilation proceeds through TTIR and TTNN stages
6.   PJRT runtime manages execution on Tenstorrent devices

The PJRT (Pretty Jax Runtime) plugin provides the bridge between torch-xla and Tenstorrent hardware:

The plugin is registered when `tt_torch` is imported and enables XLA to discover and use Tenstorrent devices.

## Compilation Modes and Execution Paths

Both backends support multiple compilation modes to accommodate different development and deployment scenarios. Each mode provides different levels of compilation depth and execution strategies.

**Compilation Mode Architecture**

Sources: [tt_torch/tools/utils.py 48-55](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L48-L55)[tt_torch/dynamo/backend.py 337-360](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L337-L360)

### Op-by-Op Compilation (tt-legacy only)

Op-by-op compilation modes execute operations individually, providing fine-grained control and debugging capabilities:

*   `COMPILE_OP_BY_OP`: Compiles each operation separately but executes the full graph
*   `EXECUTE_OP_BY_OP`: Executes each operation individually with intermediate verification

The backend selection within op-by-op mode determines the IR level:

*   `OpByOpBackend.TORCH` (line 69 in utils.py): Uses Torch IR representation via `_torch_backend`
*   `OpByOpBackend.STABLEHLO` (line 70 in utils.py): Converts to StableHLO IR via `torch_to_shlo` then uses `_shlo_backend`

### Full Compilation Pipeline

Full compilation modes process the entire graph through the complete MLIR pipeline:

*   `TORCH_FX` (line 49 in utils.py): Stops after FX graph capture and initial transformations
*   `STABLEHLO` (line 50): Compiles through StableHLO IR generation
*   `TTNN_IR` (line 51): Completes MLIR lowering to TTNN dialect
*   `EXECUTE` (line 54): Performs full compilation and hardware execution

Sources: [tt_torch/tools/utils.py 48-55](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L48-L55)

## Integration with Executor Framework

The backend system creates and configures executor objects that handle the actual model execution. The relationship between backend configuration and executor creation depends on the compilation mode selected.

**Backend to Executor Flow**

Sources: [tt_torch/dynamo/backend.py 183-194](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L183-L194)[tt_torch/dynamo/backend.py 273-309](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L273-L309)[tt_torch/dynamo/backend.py 165-179](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L165-L179)

The backend function creates different executor types based on the compilation path:

1.   **TorchExecutor**: Created by `_torch_backend` for op-by-op Torch IR execution
2.   **StablehloExecutor**: Created by `_shlo_backend` for op-by-op StableHLO execution
3.   **Executor**: Created by `_base_backend` for full pipeline compilation

Each executor type receives:

*   A `MultiChipGraph` containing the appropriate IR representation
*   Compilation configuration from `CompilerConfig`
*   Device specifications and async mode settings
*   Shared cache objects for runtime optimization

The caching system in `BackendOptions` enables executor reuse across multiple compilation calls, improving performance for repeated model execution with the same configuration.
