## Purpose and Scope

`ttrt` (TT Runtime Tool) is the primary command-line tool and Python package for interacting with compiled flatbuffer binaries produced by the `tt-mlir` compiler. Its primary role is to inspect and run flatbuffer files, enabling execution on Tenstorrent hardware without requiring a frontend runtime framework. It provides a unified interface for metadata inspection, program execution, performance profiling, and system compatibility validation. [docs/src/ttrt.md 1-3](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L1-L3)

This tool bridges the gap between the compiler output (Flatbuffers) and the hardware, serving as both a developer utility for debugging and the standard execution driver for silicon validation tests. [docs/src/ttrt.md 81-84](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L81-L84)

## Architecture Overview

The `ttrt` tool provides both command-line and Python interfaces to the underlying `tt::runtime` C++ API. It dispatches execution to either the TTNN or TTMetal runtime backends based on the binary's target runtime type. [tools/ttrt/common/perf.py 214-215](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/common/perf.py#L214-L215)

### System Integration

**ttrt System Integration**: This diagram shows ttrt's position as the primary interface to the tt-mlir runtime system. The Python-based tool logic wraps the `TTMLIRRuntime` shared library. Execution flow moves from the Python `Run` or `Perf` classes into the C++ `ProgramExecutor`, which manages operation dispatch to the hardware. ## Build and Installation

The `ttrt` tool is built as part of the standard `tt-mlir` build process. It generates a Python wheel that packages the necessary shared libraries. [docs/src/ttrt.md 44-45](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L44-L45)

### Build Configuration

The build is controlled via CMake, with several key options:

*   `TTMLIR_ENABLE_RUNTIME`: Enables the runtime components required for `run` and `perf` subcommands. [docs/src/ttrt.md 14-18](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L14-L18)[tools/ttrt/setup.py 54](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/setup.py#L54-L54)
*   `TT_RUNTIME_ENABLE_PERF_TRACE`: Enables Tracy-based performance tracing for perf mode. [docs/src/ttrt.md 20-25](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L20-L25)[tools/ttrt/setup.py 58](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/setup.py#L58-L58)

### Python Wheel Distribution

Every time `ttrt` is built, it creates a `.whl` file in `build/tools/ttrt/build`. This wheel can be installed in any Python virtual environment to provide the `ttrt` command and library. [docs/src/ttrt.md 44-55](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttrt.md?plain=1#L44-L55)[tools/ttrt/setup.py 11-35](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/setup.py#L11-L35)[tools/ttrt/CMakeLists.txt 34-35](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/CMakeLists.txt#L34-L35)

## Execution Flow and Callbacks

When executing a program, `ttrt` can register callbacks that trigger before or after every operation. This is used for intermediate tensor validation and golden PCC checks. [tools/ttrt/common/callback.py 13-17](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttrt/common/callback.py#L13-L17)

**Operation Callback Flow**: This sequence shows how `ttrt` manages execution. While the C++ `ProgramExecutor` handles hardware dispatch, the Python wrapper manages pre/post operation logic. Performance data is logged via the `perf::Env` singleton which tracks metadata and operation locations for Tracy. ## Versioning and Compatibility
