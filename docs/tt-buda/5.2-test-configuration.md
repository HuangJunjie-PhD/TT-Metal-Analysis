The Test Configuration system in PyBuda provides a framework for configuring and controlling test execution across different device types, test scenarios, and verification requirements. This page documents the key components that allow users to customize how models are tested and verified on Tenstorrent hardware.

For compiler-specific configuration options, see [Compiler Configuration](https://deepwiki.com/tenstorrent/tt-buda/5.1-compiler-configuration).

## VerifyConfig

The `VerifyConfig` class is the primary mechanism for configuring verification behavior in PyBuda tests. It controls aspects like tolerance thresholds, whether to run golden reference comparisons, and device-specific verification settings.

### Key Configuration Properties

| Property | Description | Default |
| --- | --- | --- |
| `enabled` | Whether verification is enabled | `True` |
| `intermediates` | Verify intermediate results | `True` |
| `rtol` | Relative tolerance for comparisons | Data format dependent |
| `atol` | Absolute tolerance for comparisons | Data format dependent |
| `pcc` | Pearson correlation coefficient threshold | `0.99` |
| `run_golden` | Run backend golden reference | Based on `PYBUDA_VERIFY_GOLDEN` |
| `verify_all` | Verify after every compilation stage | Based on `PYBUDA_FORCE_VERIFY_ALL` |
| `verify_last` | Verify after final stage | `True` |

## TestKind

The `TestKind` enum defines the kind of test to be performed:

| Value | Description |
| --- | --- |
| `INFERENCE` | Tests model inference only |
| `TRAINING` | Tests model training with backward pass |
| `TRAINING_RECOMPUTE` | Tests training with activation recomputation |

Sources: [pybuda/pybuda/verify/config.py 18-47](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/pybuda/verify/config.py#L18-L47)

## TestDevice

The `TestDevice` class represents the device configuration for testing. It encapsulates device type, architecture, and execution mode.

### Supported Device Types

| Device Type | Description |
| --- | --- |
| `Golden` | CPU reference implementation |
| `Model` | Simplified device model |
| `Versim` | Cycle-accurate simulator |
| `Emulation` | Hardware emulation |
| `Silicon` | Physical Tenstorrent hardware |

### Supported Device Architectures

| Architecture | Description |
| --- | --- |
| `Grayskull` | First-generation Tenstorrent architecture |
| `Wormhole_B0` | Second-generation architecture |
| `Blackhole` | Latest-generation architecture |

### Device Modes

| Mode | Description |
| --- | --- |
| `CompileAndRun` | Compile model and run it |
| `CompileOnly` | Only compile the model, don't run |
| `RunOnly` | Load pre-compiled model and run it |

## PyTest Configuration

PyBuda extends pytest with custom fixtures and command-line options to control test execution.

### Key Fixtures

*   `test_device`: Parameterized fixture that creates available device configurations
*   `test_kind`: Parameterized fixture that runs tests with different `TestKind` values
*   `clear_pybuda`: Fixture that resets PyBuda state between tests

### Command Line Options

| Option | Description |
| --- | --- |
| `--silicon-only` | Run only on silicon devices, skip golden/model |
| `--no-silicon` | Skip silicon tests |
| `--compile-only` | Only compile the model, don't run it |
| `--run-only` | Load pre-compiled model and run it |
| `--tti-path` | Path to save/load TTI (compiled model) |
| `--device-config` | Configure runtime based on a predefined config |
| `--devtype` | Set backend device type (golden/silicon) |
| `--no-skips` | Ignore `pytest.skip()` calls and continue tests |

## Device Config System

The device configuration system allows configuring runtime parameters for specific hardware targets. It supports the following configurations:

| Config Name | Description | Architecture |
| --- | --- | --- |
| `gs_e150` | Grayskull E-150 | Grayskull |
| `gs_e300` | Grayskull E-300 | Grayskull |
| `wh_n150` | Wormhole N-150 | Wormhole B0 |
| `wh_n300` | Wormhole N-300 | Wormhole B0 |
| `galaxy` | Galaxy multi-chip | Wormhole B0 |
| `bh_test` | Blackhole test | Blackhole |

Each configuration defines system-level parameters including chip locations, MMIO configuration, and harvesting masks.

## Best Practices

When configuring tests in PyBuda:

1.   Use `VerifyConfig` to control verification behavior and tolerances
2.   Select appropriate `TestKind` (INFERENCE, TRAINING, TRAINING_RECOMPUTE) based on test goals
3.   Configure device selection with proper architecture (Grayskull, Wormhole, etc.)
4.   Use environment variables for global settings
5.   Use pytest command-line options for run-specific settings
6.   Use `--compile-only` and `--tti-path` for model caching between test runs
7.   Set appropriate tolerance values based on the precision requirements of your model
