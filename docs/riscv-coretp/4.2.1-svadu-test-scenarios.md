# SVADU Test Scenarios

Relevant source files
*   [coretp/plans/svadu/svadu_scenarios.py](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py)

## Purpose and Scope

This document details the test scenarios for the SVADU (Supervisor Virtual Address Update) RISC-V extension. SVADU introduces hardware-managed updates to the Accessed (A) and Dirty (D) bits in page table entries (PTEs), controlled via the `menvcfg.adue` CSR bit. These scenarios verify correct behavior across three modes: SVADU disabled (software-managed A/D bits), SVADU enabled (hardware-managed A/D bits), and various fault conditions.

For general virtual memory test coverage including page table walks and fault conditions, see [Page Table Walk Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.2-page-table-walk-scenarios). For environment configuration details including paging mode setup, see [Test Environment Configuration](https://deepwiki.com/tenstorrent/riscv-coretp/3.2-test-environment-configuration).

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 1-864](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L1-L864)

## SVADU Extension Overview

The SVADU extension modifies how the Accessed and Dirty bits in PTEs are managed:

*   **Accessed (A) bit**: Set on any memory access (load, store, instruction fetch) to indicate the page has been referenced
*   **Dirty (D) bit**: Set on store or AMO operations to indicate the page has been modified
*   **Control**: The `menvcfg.adue` CSR bit (bit 61) enables/disables hardware A/D bit management
*   **Behavior**: When disabled (`adue=0`), memory accesses with `pte.a=0` or stores with `pte.d=0` raise page faults. When enabled (`adue=1`), hardware automatically sets these bits on cacheable memory

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 33-209](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L33-L209)

## Active Test Scenarios

### SID_SVADU_01: Fault on Cleared A/D Bits

These scenarios verify that when SVADU is disabled (`menvcfg.adue=0`), memory accesses correctly fault when the corresponding A or D bit is not set.

#### Scenario 1: Fault on A Bit Cleared

**Test Logic:**

1.   Allocate memory with `pte.a=0` (excluded via `exclude_flags=PageFlags.ACCESSED`)
2.   Disable SVADU by clearing `menvcfg.adue` (bit 61)
3.   Attempt load operation
4.   Verify `LOAD_PAGE_FAULT` exception occurs

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 32-62](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L32-L62)

#### Scenario 2: Fault on D Bit Cleared

**Test Logic:**

1.   Allocate memory with `pte.d=0`, `pte.a=1` (A bit set to avoid A-bit fault)
2.   Disable SVADU
3.   Attempt store operation
4.   Verify `STORE_AMO_PAGE_FAULT` exception occurs

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 65-97](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L65-L97)

### SID_SVADU_02: Hardware A/D Bit Updates

These scenarios verify that when SVADU is enabled (`menvcfg.adue=1`), hardware automatically sets the A and D bits on memory accesses.

#### Scenario 3: Hardware Update A Bit

**Test Logic:**

1.   Allocate memory with `pte.a=0`
2.   Enable SVADU by setting `menvcfg.adue` (bit 61)
3.   Read leaf PTE and verify A bit (bit 6) is initially 0
4.   Perform load operation
5.   Read leaf PTE again and verify A bit is now 1

**Key Code Entities:**

*   `ReadLeafPTE`: Reads the leaf-level PTE for a memory region
*   `Arithmetic(op="and")`: Performs bitwise AND to check specific bit
*   `LoadImmediateStep`: Loads constant values for bit masks

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 100-152](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L100-L152)

#### Scenario 4: Hardware Update D Bit

**Test Logic:**

1.   Allocate memory with `pte.d=0`
2.   Enable SVADU
3.   Verify D bit (bit 7) is initially 0
4.   Perform store operation
5.   Verify D bit is now 1

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 155-209](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L155-L209)

## Test Step Composition Pattern

All SVADU scenarios follow a consistent pattern of composing `TestStep` primitives:

**Common Patterns:**

| Pattern | Purpose | Code Example |
| --- | --- | --- |
| CSR Configuration | Enable/disable SVADU | `CsrWrite(csr_name="menvcfg", set_mask=1<<61)` |
| Memory Allocation | Create test memory region | `Memory(size=0x1000, flags=..., exclude_flags=...)` |
| Bit Checking | Verify PTE bit states | `Arithmetic(op="and", src1=pte, src2=mask)` |
| Fault Verification | Assert expected exceptions | `AssertException(cause=ExceptionCause.LOAD_PAGE_FAULT, code=[...])` |

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 6-27](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L6-L27)

## Environment Configuration

All SVADU scenarios use the same `TestEnvCfg` configuration to test across multiple paging modes:

`TestEnvCfg(paging_modes=[PagingMode.SV39, PagingMode.SV48, PagingMode.SV57])`
This configuration ensures each scenario is validated across:

*   **SV39**: 39-bit virtual addresses (3-level page table)
*   **SV48**: 48-bit virtual addresses (4-level page table)
*   **SV57**: 57-bit virtual addresses (5-level page table)

The `TestEnvSolver` generates concrete `TestEnv` instances for each paging mode, creating 3 test variations per scenario.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 56-57](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L56-L57)[coretp/plans/svadu/svadu_scenarios.py 90-91](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L90-L91)[coretp/plans/svadu/svadu_scenarios.py 137-138](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L137-L138)[coretp/plans/svadu/svadu_scenarios.py 193-194](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L193-L194)

## Planned Scenarios (Commented Out)

The following scenarios are defined but currently commented out, indicating future test coverage:

### NC/IO Memory Scenarios (ID 5-6)

Test A/D bit behavior when the page's physical address is mapped to non-cacheable (NC) or I/O memory regions. According to the RISC-V specification, SVADU should not perform hardware updates on NC/IO memory.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 212-281](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L212-L281)

### Speculative Update Scenarios (ID 7-8)

Test speculative A bit updates (which may occur even on faults) versus non-speculative D bit updates (which must not occur on faults). These scenarios require PMA/PMP support for generating access faults.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 284-373](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L284-L373)

### PTE Ordering Scenarios (ID 9-10)

Test that A/D PTE updates occur in global memory order, including:

*   Recursive PTE mappings where a page's PTE points to itself
*   Store operations followed by PTE modifications

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 376-469](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L376-L469)

### Dynamic Enable/Disable Scenarios (ID 13-14)

Test sequences of enabling and disabling SVADU with intervening memory accesses to verify correct state transitions.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 479-552](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L479-L552)

### Vector Operation Scenarios (ID 18)

Test hardware A/D bit updates during vector load/store operations. Requires vector extension support.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 639-690](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L639-L690)

### AMO and TLB Scenarios (ID 22)

Test atomic memory operations (AMOs) combined with TLB invalidation (`sfence.vma`) to verify correct A/D bit management across TLB flushes.

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 704-750](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L704-L750)

### Multi-Processor Scenarios (ID 23-25)

Test PTE visibility and synchronization across multiple harts, including:

*   Fence instruction ordering of PTE updates
*   Hart coordination with A/D bit updates
*   Cross-hart observation of hardware updates

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 753-857](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L753-L857)

## Exception Causes

SVADU scenarios verify these exception types:

| Exception | Constant | When Raised |
| --- | --- | --- |
| Load Page Fault | `ExceptionCause.LOAD_PAGE_FAULT` | Load with `pte.a=0` and SVADU disabled |
| Store/AMO Page Fault | `ExceptionCause.STORE_AMO_PAGE_FAULT` | Store with `pte.d=0` and SVADU disabled |
| Load Access Fault | `ExceptionCause.LOAD_ACCESS_FAULT` | Load with insufficient permissions (planned) |
| Store/AMO Access Fault | `ExceptionCause.STORE_AMO_ACCESS_FAULT` | Store with insufficient permissions (planned) |

**Sources:**[coretp/plans/svadu/svadu_scenarios.py 5](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L5-L5)
