## Purpose and Scope

This page documents the three types of edge tiles that handle chip I/O and management functions:

*   **PCIe tiles**: Provide host interface via PCIe 4.0 x16
*   **ARC tile**: Provides chip management via 4-core ARC CPU with 512 KiB CSM
*   **L2CPU tiles**: Provide application-level processing via x280 RISC-V cores (Blackhole A0 only)

For network I/O, see Ethernet tiles in [9.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/9.1-ethernet-tile-architecture). For off-chip memory, see DRAM tiles in [9.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/9.3-dram-tiles). For NoC architecture, see [2](https://deepwiki.com/tenstorrent/tt-isa-documentation/2-network-on-chip-(noc)).

## PCIe Tile Architecture

### Overview

The PCIe tile provides the interface between the host system and the Tenstorrent chip. Wormhole B0 contains one PCIe tile, while Blackhole A0 contains two (PCIe 0 and PCIe 1).

**Key Components:**

*   PCI Express Controller and PHY (PCIe 4.0 x16)
*   Inbound and Outbound iATU (internal Address Translation Unit)
*   Configurable TLBs
*   DMA engines
*   2 NIUs (one per NoC)
*   AXI/APB interfaces

### Address Translation

#### Inbound iATU

Translates host physical addresses to chip-local addresses for host-initiated reads and writes. Configuration is accessible from ARC via APB.

#### Outbound iATU

Translates chip-local addresses to host physical addresses for device-initiated transfers (DMA, ARC-to-host writes). Configuration is accessible from ARC via APB.

#### Configurable TLBs

*   **Wormhole B0**: Single TLB set for host-to-NoC direction
*   **Blackhole A0 H2D**: Separate TLB set for host-to-NoC direction
*   **Blackhole A0 D2H**: Separate TLB sets for NoC-to-host and ARC-to-host directions

Sources: [Diagrams/Src/EdgeTile.lua 168-169](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L168-L169)[Diagrams/Src/EdgeTile.lua 260-276](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L260-L276)

### DMA Engines

Hardware DMA engines enable bulk data transfers without CPU intervention. DMA engines read from one address space and write to another, automatically handling:

*   Scatter-gather operations
*   Interrupt generation on completion
*   Error reporting

Sources: [Diagrams/Src/EdgeTile.lua 210-242](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L210-L242)

### Blackhole A0 Enhancements

Blackhole A0 includes several PCIe improvements over Wormhole B0:

| Feature | Wormhole B0 | Blackhole A0 |
| --- | --- | --- |
| Number of PCIe tiles | 1 | 2 (PCIe 0 and PCIe 1) |
| Interrupt Receiver | Not present | MSI-X receiver on PCIe 0 (Root Port mode) |
| TLB Architecture | Single TLB set | Separate TLBs for H←N and H←A paths |
| Serdes Configuration | Via ARC | Direct APB path |

## ARC Tile Architecture

### Overview

The ARC tile provides chip-level management via a 4-core ARC processor with dedicated memory and peripheral interfaces. There is one ARC tile per chip.

**Key Components:**

*   ARC CPU (4 cores)
*   ARC CSM (512 KiB scratchpad memory)
*   ARC XBAR (crossbar interconnect)
*   Configurable and Fixed TLBs
*   2 NIUs (one per NoC)
*   Reset Unit
*   Scratch registers (8× 32-bit)
*   Reference cycle counter
*   APB peripherals (SPI, I2C, eFuses)

### ARC Tile Block Diagram

Sources: [Diagrams/Src/EdgeTile.lua 292-436](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L292-L436)

### ARC CPU

The ARC CPU is a 4-core processor responsible for:

*   Chip initialization sequences
*   Runtime configuration management
*   Error monitoring and recovery
*   Power management
*   Host communication coordination
*   NoC routing table setup

Sources: [Diagrams/Src/EdgeTile.lua 399-400](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L399-L400)

### ARC CSM (Close-to-CPU Scratchpad Memory)

512 KiB of fast local memory with direct CPU access. Not accessible via NoC. Stores:

*   ARC firmware code
*   Runtime data structures
*   Configuration state

Sources: [Diagrams/Src/EdgeTile.lua 397-398](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L397-L398)

### ARC XBAR (Crossbar)

Multi-port interconnect fabric connecting:

*   ARC CPU to peripherals
*   TLBs to NIUs
*   APB interfaces to chip resources
*   Reset unit to tile control

Sources: [Diagrams/Src/EdgeTile.lua 315-316](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L315-L316)

### Translation Lookaside Buffers

The ARC tile contains multiple TLB sets depending on architecture and direction:

**Wormhole B0:**

*   H2D: Configurable TLBs map host addresses to NoC destinations
*   H2D: Fixed TLBs provide pre-configured address ranges for PCIe traffic
*   D2H: Configurable TLBs handle ARC-to-host and NoC-to-host traffic
*   D2H: Direct path to DRAM D0 tiles

**Blackhole A0:**

*   H2D: Configurable TLBs map host addresses to NoC destinations
*   D2H: Separate configurable TLB sets for ARC-to-host and NoC-to-host paths

Sources: [Diagrams/Src/EdgeTile.lua 356-376](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L356-L376)

### Reset Unit

Manages chip-level reset sequences:

*   Controls soft reset of individual tiles (see [SoftReset.md](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/SoftReset.md?plain=1))
*   Manages power-on reset sequences
*   Monitors reset status
*   Provides selective tile reset capability

Sources: [Diagrams/Src/EdgeTile.lua 381-405](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L381-L405)

### Scratch Registers

Eight 32-bit registers (`Scratch[0]` through `Scratch[7]`) provide:

*   Persistent storage across soft resets
*   Boot status flags
*   Inter-processor communication
*   Debug information

Sources: [Diagrams/Src/EdgeTile.lua 394](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L394-L394)

### Cycle Counter

Reference cycle counter driven by `REF Clock` provides:

*   Monotonic timestamp generation
*   Performance measurement baseline
*   Timeout detection

Sources: [Diagrams/Src/EdgeTile.lua 406-407](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L406-L407)

### APB Peripherals

ARC has APB access to:

*   **SPI**: Serial flash and external devices
*   **I2C**: Board-level device communication
*   **eFuses**: One-time programmable configuration
*   **NoC Configuration**: Direct access to NoC routing tables and stream registers

Sources: [Diagrams/Src/EdgeTile.lua 389-392](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L389-L392)

### ARC Clock Domains

Four clock domains with CDC (Clock Domain Crossing) logic:

*   **REF Clock**: External reference oscillator
*   **ARC Clock**: ARC CPU and local logic
*   **AXI Clock**: ARC XBAR and APB fabric
*   **AI Clock**: NoC and tile fabric

Sources: [Diagrams/Src/EdgeTile.lua 409-431](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/EdgeTile.lua#L409-L431)

## L2CPU Tile Architecture

### Overview

L2CPU tiles provide general-purpose application processing using x280 RISC-V cores. These tiles exist only in Blackhole A0.

**Configuration per tile:**

*   4× x280 RISC-V cores
*   Per-core L1 instruction and data caches
*   Shared L2 cache
*   2 MiB L3 cache
*   2 NIUs (one per NoC)

**Total Blackhole A0:**

*   4 L2CPU tiles
*   16 x280 cores total
*   8 MiB L3 cache total

### Tile Locations in Blackhole A0

| Grid Position (x, y) | Cores | Label |
| --- | --- | --- |
| (8, 3) | 0-3 | CPU 0-3 |
| (8, 5) | 8-11 | CPU 8-11 |
| (8, 7) | 12-15 | CPU 12-15 |
| (8, 9) | 4-7 | CPU 4-7 |

Sources: [Diagrams/Src/NoC.lua 28-43](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Diagrams/Src/NoC.lua#L28-L43)

### Cache Hierarchy

**L1 Cache:** Per-core, split instruction and data caches

**L2 Cache:** Shared across all 4 cores within the tile

**L3 Cache:** 2 MiB unified cache providing high-bandwidth access to NoC

Sources: High-level architecture overview (Diagram 1 in provided context)

### NoC Connectivity

L2CPU tiles connect to both NoC #0 and NoC #1 via NIUs, enabling:

*   Access to DRAM tiles (see [9.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/9.3-dram-tiles))
*   Communication with Tensix tiles (see [3](https://deepwiki.com/tenstorrent/tt-isa-documentation/3-tensix-tile-architecture))
*   Data exchange with other L2CPU tiles
*   PCIe access for host communication (see PCIe section above)

For NoC routing and addressing, see [2.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing).

### Use Cases

L2CPU tiles are designed for:

*   **Control Flow**: Complex application logic exceeding Baby RISC-V capabilities (see [3.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores))
*   **Data Preprocessing**: Preparing data for Tensix compute tiles
*   **Post-Processing**: Aggregating and formatting computation results
*   **Host Communication**: Managing PCIe transactions and data transfers
*   **Runtime Coordination**: Orchestrating multi-tile operations and synchronization

## Edge Tile Address Mapping

### NIU Register Access

All edge tiles have NIU registers accessible via NoC at tile-specific coordinates. For NIU memory map, see [2.1](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface).

### Configuration Access Patterns

| Component | Access Method | Accessible From |
| --- | --- | --- |
| PCIe NIU registers | NoC memory-mapped | Any tile via NoC |
| PCIe iATU configuration | APB (via ARC XBAR) | ARC CPU |
| PCIe DMA engines | APB (via ARC XBAR) | ARC CPU |
| ARC NIU registers | NoC memory-mapped | Any tile via NoC |
| ARC CSM | Direct CPU access | ARC CPU only |
| ARC scratch registers | Direct CPU access | ARC CPU only |
| ARC peripherals | APB (via ARC XBAR) | ARC CPU |
| L2CPU NIU registers | NoC memory-mapped | Any tile via NoC |
| L2CPU caches | Direct CPU access | x280 cores |

## Architecture Comparison

### Wormhole B0 Edge Tiles

**Wormhole B0:**

*   1 PCIe tile at grid position (0, 3)
*   1 ARC tile at grid position (0, 10)
*   No L2CPU tiles

### Blackhole A0 Edge Tiles

**Blackhole A0:**

*   2 PCIe tiles: PCIe 0 at (2, 0), PCIe 1 at (11, 0)
*   1 ARC tile at (8, 0)
*   4 L2CPU tiles at (8, 3), (8, 5), (8, 7), (8, 9) 
    *   16 x280 cores total (4 per tile)
    *   8 MiB L3 cache total (2 MiB per tile)

*   Direct path from PCIe 1 to ARC (management traffic)

## Integration with NoC

All edge tiles connect to the NoC fabric through NIUs (Network Interface Units). Each edge tile has 2 NIUs (one per NoC), providing:

*   Redundant data paths
*   Higher aggregate bandwidth
*   Independent routing on NoC #0 and NoC #1

### NoC Capabilities for Edge Tiles

*   **Unicast**: Point-to-point transfers between tiles
*   **Broadcast**: One-to-many configuration updates
*   **Atomic operations**: Synchronization primitives (see [2.4](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements))
*   **Virtual channels**: Deadlock-free routing (see [2.3](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.3-virtual-channels-and-ordering-guarantees))

For detailed NoC architecture, see [2](https://deepwiki.com/tenstorrent/tt-isa-documentation/2-network-on-chip-(noc)).
