The `tt-emule` execution engine models the multi-processor architecture of Tenstorrent silicon by mapping hardware RISC-V cores to host CPU threads. This page details the lifecycle of a kernel from its original C++ source through JIT compilation to parallel execution against emulated L1 and DRAM memory.

## Per-Core Thread Model

The emulator supports two distinct threading models based on the target architecture. These models mirror the hardware's RISC-V processor topology, ensuring that synchronization primitives like Circular Buffers (CBs) and Dataflow Buffers (DFBs) exercise real concurrency on the host.

### Wormhole & Blackhole (3-Thread Model)

For Wormhole and Blackhole architectures, each emulated core spawns three threads representing the Tensix processors:

1.   **NCRISC (NOC Reader):** Responsible for data movement from DRAM/L1 to local CBs.
2.   **TRISC (Compute):** Executes the UNPACK-MATH-PACK pipeline.
3.   **BRISC (NOC Writer):** Responsible for data movement from local CBs to DRAM/L1.

### Quasar / Tensix Neo (12-Thread Model)

The Quasar architecture utilizes the "Neo" core, which expands the thread count significantly:

*   **DM0–DM7 (8 Data Movement Threads):** Handle complex multi-directional data movement.
*   **E0–E3 (4 Compute Engines):** Parallel math engines sharing a larger 4MB L1.

### Thread Execution Lifecycle

The `execute_program_emulated` function in `tt-metal` (the primary entry point) orchestrates the launch. For every active core in a `Program`, it:

1.   Retrieves the `KernelDescriptor` for each processor [include/tt_emule/program.hpp 143-156](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/program.hpp#L143-L156)
2.   Triggers the JIT pipeline to obtain a shared object (`.so`) handle.
3.   Spawns host threads (using `std::thread`) that initialize thread-local state and call the kernel entry point.

## JIT Compilation Pipeline

Kernels are compiled on-demand using the host's `clang++-20` compiler. This ensures that the exact same source code running on silicon is used for emulation, with minor shims provided by the `jit_hw` headers.

### Source Rewriting & Preprocessing

Before compilation, the `emulated_program_runner` performs regex-based source rewriting [docs/api-injection-points.md 25-31](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L25-L31) Key transformations include:

*   **CSR Patching:** For Quasar, reads to Control and Status Registers (CSRs) like `mhartid` or `NEO_ID` are patched to reference thread-local variables [docs/QUASAR_MATMUL_REPORT.md 58-60](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_MATMUL_REPORT.md?plain=1#L58-L60)
*   **Include Injection:** The compiler is invoked with a high-priority `-I` path pointing to `tt-emule/include/jit_hw/`, which contains mocks for hardware-specific headers [docs/api-injection-points.md 36-41](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L36-L41)

### Compilation Flow

1.   **Source Preparation:** A `wrapper.cpp` is generated that includes the kernel source and sets up the `extern "C" void __emule_kernel_entry()` symbol.
2.   **Clang Invocation:**`clang++` compiles the source into a position-independent shared library (`.so`).
3.   **DLOpen:** The host process loads the `.so` using `dlopen` with `RTLD_GLOBAL` so that the kernel can resolve bridge symbols defined in `libtt_metal.so`[docs/api-injection-points.md 126-128](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L126-L128)
4.   **JIT Cache:** Compiled kernels are stored in `/tmp/tt_emule_jit_<hash>` to avoid redundant compilations [docs/api-injection-points.md 101](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L101-L101)

## Thread-Local State Injection

Since multiple threads (representing different cores and RISC processors) run within the same host process, `tt-emule` uses `thread_local` storage to isolate the state of each virtual processor.

### Key Thread-Local Variables

When a thread starts, the `emulated_program_runner` injects the following state:

| Variable | Type | Description |
| --- | --- | --- |
| `__processor_id` | `uint32_t` | The ID of the RISC processor (e.g., BRISC=0, NCRISC=1). |
| `__rt_args` | `std::vector<uint32_t>` | Runtime arguments passed from the host via `SetRuntimeArgs`. |
| `__core` | `tt_emule::Core*` | Pointer to the current core's L1 memory and metadata. |
| `__device` | `tt_emule::Device*` | Pointer to the parent emulated device object. |
| `__emule_cbs` | `CBSyncState*` | Array of synchronization states for Circular Buffers. |
| `__emule_dfbs` | `DFBSyncState*` | (Quasar only) Array of Dataflow Buffer interfaces. |

## Execution Lifecycle: `emulated_program_runner`

The `emulated_program_runner` manages the transition from host-side `EnqueueProgram` to kernel execution.

### Data Flow and Lifecycle

1.   **Host API Call:** The user calls `EnqueueProgram`[src/host_api.cpp 29](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/host_api.cpp#L29-L29)
2.   **Kernel Resolution:** The runner identifies all kernels required by the program and checks the JIT cache.
3.   **Compilation (if needed):**`jit_compile_kernel` is called to produce the `.so`[docs/api-injection-points.md 27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L27-L27)
4.   **Memory Mapping:** The runner ensures the `tt_emule::Core` L1 mmaps are ready [IMPLEMENTATION_REPORT.md 79-83](https://github.com/tenstorrent/tt-emule/blob/a57cf072/IMPLEMENTATION_REPORT.md?plain=1#L79-L83)
5.   **Thread Launch:** For each core and processor, a thread is spawned. 
    *   **Setup:** TLS variables are assigned.
    *   **Execution:**`__emule_kernel_entry()` is called.
    *   **Synchronization:** The host thread waits for all kernel threads to finish using `thread.join()`.

6.   **Finish:**`Finish()` blocks until the `CommandQueue` is empty and all launched programs have completed [src/host_api.cpp 27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/src/host_api.cpp#L27-L27)

### Kernel-to-Host Bridge (Bridge Functions)

Because the JIT kernel is a separate shared object, it cannot directly access C++ member functions of the host-side `Device` or `Core` classes. Instead, it uses `extern "C"` bridge functions:

| Bridge Function | Purpose |
| --- | --- |
| `__emule_dram_ptr(offset)` | Resolves a DRAM offset to a host pointer [docs/api-injection-points.md 119](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L119-L119) |
| `__emule_resolve_noc_addr(addr)` | Translates a 64-bit NOC address to a host pointer [docs/api-injection-points.md 123](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L123-L123) |
| `__emule_multicast_write(...)` | Performs a multicast write across multiple emulated cores [docs/api-injection-points.md 124](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/api-injection-points.md?plain=1#L124-L124) |
