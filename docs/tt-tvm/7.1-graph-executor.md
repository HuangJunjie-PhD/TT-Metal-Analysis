## Purpose and Scope

The Graph Executor is TVM's static graph-based runtime system designed for executing compiled neural network models with fixed computation graphs. It provides high-performance inference execution using a JSON-based graph representation and pre-allocated memory management. This executor is optimized for production inference scenarios where the model structure is known ahead of time and does not change during execution.

For dynamic execution with control flow and variable shapes, see the [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor). For embedded deployment with minimal runtime overhead, see the [AOT Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.3-aot-executor).

## Overview

The Graph Executor operates on a static graph representation where:

*   The computation graph structure is defined at compile time and encoded in JSON format
*   Memory allocation is determined during compilation and remains fixed during execution
*   Operators are executed in a predetermined topological order
*   All tensors have statically-known shapes and types

This design makes the Graph Executor particularly suitable for scenarios requiring predictable performance and minimal runtime overhead, such as production inference servers and edge deployment where models have fixed architectures.

## Graph JSON Format

The Graph Executor uses a JSON-based representation that encodes the entire computation graph structure. This format includes:

| Component | Description | Example Usage |
| --- | --- | --- |
| **nodes** | Array of computation nodes, each with `op_type`, `name`, `inputs`, and `attrs` | Defines operators like "tvm_op", "null" (inputs) |
| **arg_nodes** | Indices of input parameter nodes | Used by `set_input()` to locate input tensors |
| **heads** | Output indices that specify which node outputs are final results | Retrieved by `get_output()` |
| **node_row_ptr** | Cumulative count of outputs per node | Used for indexing into data entries |
| **attrs** | Graph-level attributes including shapes, dtypes, and storage IDs | Determines memory allocation |

The JSON format enables serialization and deserialization of compiled models, allowing deployment without recompilation.

## Execution Flow

### Initialization and Setup

The initialization process involves:

1.   **Graph Parsing**: JSON structure is parsed to build internal node and edge representations
2.   **Memory Allocation**: Storage is pre-allocated based on shape and dtype information from the graph
3.   **Operator Loading**: Compiled kernel functions are loaded from the runtime module
4.   **Device Setup**: Device-specific allocators and contexts are initialized

Sources: [python/tvm/contrib/debugger/debug_executor.py 40-90](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/contrib/debugger/debug_executor.py#L40-L90)

### Runtime Execution

During execution, the Graph Executor follows a simple sequential execution model:

1.   **Input Setting**: User provides input tensors via `set_input(name, data)` or `set_input(index, data)`
2.   **Node Execution**: Each operator node is executed in topological order
3.   **Data Flow**: Output tensors from one node become inputs to downstream nodes
4.   **Output Retrieval**: Final results are accessed via `get_output(index)`

The execution is deterministic and follows the pre-computed topological order, ensuring all dependencies are satisfied before each node executes.

## Debug and Profiling Support

### GraphExecutorDebug

The Graph Executor provides an extended debug variant with profiling and introspection capabilities:

The debug executor provides:

*   **Per-operator profiling**: Time each operator independently via `run_individual()`
*   **Intermediate outputs**: Access outputs of any node in the graph
*   **Performance reports**: Detailed timing and performance metrics in CSV/JSON/table formats
*   **Shape information**: Track tensor shapes through execution

The profiling system captures:

*   **Duration (us)**: Microseconds per operator execution
*   **Count**: Number of times each operator is called
*   **Hash**: Unique identifier for operator implementation
*   **Name**: Operator name (e.g., "fused_nn_softmax")
*   **Argument Shapes**: Input/output tensor shapes

## Comparison with Other Executors

| Feature | Graph Executor | VM Executor | AOT Executor |
| --- | --- | --- | --- |
| **Graph Structure** | Static, JSON-based | Bytecode-based | Static, C code |
| **Dynamic Shapes** | No | Yes | No |
| **Control Flow** | No | Yes (if/while) | Limited |
| **Memory Model** | Pre-allocated pools | Dynamic allocation | Static allocation |
| **Startup Time** | Fast | Medium | Fastest |
| **Runtime Overhead** | Low | Medium | Minimal |
| **Debugging** | Good (debug variant) | Good (profiler VM) | Limited |
| **Serialization** | JSON + lib | Bytecode + lib | Compiled C |
| **Use Case** | Production inference | Dynamic models | Embedded/MCU |

The Graph Executor is the recommended choice for:

*   Production deployment of static models
*   High-throughput inference servers
*   Scenarios requiring predictable latency
*   Models without dynamic control flow

## Memory Management

The Graph Executor uses a pool-based memory allocation strategy:

1.   **Static Allocation**: Memory requirements are computed during compilation based on tensor shapes
2.   **Storage Pooling**: Multiple tensors can share the same storage if their lifetimes don't overlap
3.   **Device-Specific**: Separate allocators for each device (CPU, GPU, etc.)
4.   **Pre-allocation**: All storage is allocated before execution begins

This approach minimizes allocation overhead during execution and enables predictable memory usage.

## Performance Considerations

### Advantages

*   **Minimal overhead**: No bytecode interpretation or dynamic dispatch
*   **Pre-allocated memory**: Zero allocation overhead during execution
*   **Optimized scheduling**: Operator execution order optimized at compile time
*   **Device placement**: Explicit device placement with minimal data movement

### Limitations

*   **No dynamic shapes**: All tensor shapes must be known at compile time
*   **No control flow**: Cannot handle if/while/for constructs
*   **Limited flexibility**: Graph structure cannot change at runtime
*   **Memory overhead**: Must allocate for peak memory usage

For workloads with dynamic requirements, consider the [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor) instead.
