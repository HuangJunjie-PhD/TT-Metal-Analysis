# Hardware Communication Interfaces

Relevant source files
*   [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py)
*   [tt_flash/error.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/error.py)

## Purpose and Scope

This document describes the hardware communication interfaces provided by the `TTChip` class hierarchy for interacting with Tenstorrent AI accelerators. These interfaces abstract low-level hardware operations through pyluwen's `PciChip` class, providing AXI bus access, SPI flash operations, ARC processor messaging, and telemetry retrieval.

For information about the `TTChip` class hierarchy and its concrete implementations (`WhChip`, `BhChip`), see [TTChip Class Hierarchy](https://deepwiki.com/tenstorrent/tt-flash/6.1-ttchip-class-hierarchy). For chip detection and enumeration, see [Chip Detection](https://deepwiki.com/tenstorrent/tt-flash/6.2-chip-detection). For specific firmware message types and protocols, see [Firmware Message Types](https://deepwiki.com/tenstorrent/tt-flash/6.4-firmware-message-types).

## Communication Stack Architecture

The hardware communication layer consists of three levels: the `TTChip` abstraction layer, the pyluwen `PciChip` wrapper, and the underlying hardware interfaces (AXI bus, SPI flash controller, ARC processor mailboxes).

## AXI Bus Interface

The AXI bus interface provides memory-mapped access to chip resources including configuration registers, GPIO pins, and internal memory. The `TTChip` class exposes four AXI methods that delegate to the underlying `PciChip` object.

### AXI Method Summary

| Method | Address | Data | Return | Purpose |
| --- | --- | --- | --- | --- |
| `axi_write32` | `int` | `int` (32-bit) | None | Single 32-bit register write |
| `axi_write` | `int` | `bytes` | None | Burst write of arbitrary length |
| `axi_read32` | `int` | N/A | `int` | Single 32-bit register read |
| `axi_read` | `int`, `int` (size) | N/A | `bytes` | Burst read of specified length |

**Sources**: [tt_flash/chip.py 205-218](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L205-L218)

### AXI Operation Flow

**Sources**: [tt_flash/chip.py 205-218](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L205-L218)

### AXI Implementation Details

The `TTChip` class provides thin wrapper methods that delegate to `pyluwen.PciChip`:

```
axi_write32(addr: int, value: int)
    → self.luwen_chip.axi_write32(addr, value)
    
axi_write(addr: int, data: bytes)
    → self.luwen_chip.axi_write(addr, data)
    
axi_read32(addr: int) → int
    → return self.luwen_chip.axi_read32(addr)
    
axi_read(addr: int, size: int) → bytes
    → data = bytearray(size)
    → self.luwen_chip.axi_read(addr, data)
    → return bytes(data)
```

The read methods use a mutable `bytearray` buffer that pyluwen fills in-place, then convert to immutable `bytes` for return. This pattern avoids unnecessary memory copies while maintaining Python's immutability conventions.

**Sources**: [tt_flash/chip.py 205-218](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L205-L218)

### Example: BhChip ASIC Location Read

The `BhChip.get_asic_location()` method demonstrates AXI usage for reading hardware straps on P300 boards:

```
GPIO_STRAP_REG_L = 0x80030D20

try:
    location = super().get_asic_location()  # Telemetry path
except Exception:
    # Fallback: Read GPIO strap register via AXI
    gpio_strap = self.luwen_chip.axi_read32(GPIO_STRAP_REG_L)
    location = (gpio_strap >> 6) & 0x1  # Extract GPIO6 bit
```

This reads a 32-bit register at address `0x80030D20` to determine if the chip is the left (0) or right (1) ASIC on a dual-ASIC P300 board.

**Sources**: [tt_flash/chip.py 284-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L284-L294)

## SPI Flash Interface

The SPI flash interface provides read/write access to the on-board SPI flash memory that stores firmware images, configuration data, and boot parameters. These operations are critical for firmware flashing and verification.

### SPI Method Summary

| Method | Address | Data/Size | Return | Purpose |
| --- | --- | --- | --- | --- |
| `spi_write` | `int` | `bytes` | None | Write data to SPI flash at address |
| `spi_read` | `int` | `int` (size) | `bytes` | Read data from SPI flash at address |

**Sources**: [tt_flash/chip.py 220-227](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L220-L227)

### SPI Operation Flow

**Sources**: [tt_flash/chip.py 220-227](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L220-L227)

### SPI Implementation Details

The SPI methods follow the same delegation pattern as AXI operations:

```
spi_write(addr: int, data: bytes)
    → self.luwen_chip.spi_write(addr, data)
    
spi_read(addr: int, size: int) → bytes
    → data = bytearray(size)
    → self.luwen_chip.spi_read(addr, data)
    → return bytes(data)
```

The `spi_read` method uses a mutable `bytearray` that pyluwen fills with data from the SPI flash, then converts to immutable `bytes` for safe return to the caller.

**Sources**: [tt_flash/chip.py 220-227](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L220-L227)

### SPI Usage in Flash Operations

The flash workflow uses SPI operations extensively:

1.   **Firmware Write**: `spi_write(offset, firmware_image)` writes each firmware component to its designated SPI offset
2.   **Immediate Verification**: `spi_read(offset, size)` reads back the written data for comparison
3.   **Version Retrieval**: BhChip uses `decode_boot_fs_table("cmfwcfg")` which internally reads SPI to parse the boot filesystem
4.   **Parameter Reads**: Configuration parameters stored in SPI are read during verification operations

**Sources**: [tt_flash/chip.py 220-227](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L220-L227)[tt_flash/chip.py 263-269](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L263-L269)

## ARC Messaging Interface

The ARC (Auxiliary RISC Core) messaging interface provides a request/response protocol for communicating with the on-chip ARC processor. This interface is used for firmware version queries, state checking, and triggering firmware operations.

### ARC Message Method

```
arc_msg(*args, **kwargs) → tuple
    → return self.luwen_chip.arc_msg(*args, **kwargs)
```

The `arc_msg` method is a pass-through to pyluwen's implementation, accepting variable arguments to support different message types and parameters defined in `fw_defines.yaml`.

**Sources**: [tt_flash/chip.py 229-230](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L229-L230)

### ARC Message Flow

**Sources**: [tt_flash/chip.py 229-230](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L229-L230)

### Message Types and Parameters

ARC messages are defined in `fw_defines.yaml` and loaded during `TTChip` initialization:

```
self.fw_defines = init_fw_defines(self)
```

Common message types accessed via `chip.fw_defines`:

*   `MSG_TYPE_FW_VERSION`: Query firmware version with `arg0` selecting running (0), bundle running (1), or bundle SPI (2)
*   `MSG_TYPE_ARC_STATE`: Check ARC processor state
*   `MSG_TRIGGER_SPI_COPY_LtoR`: Trigger local-to-remote SPI copy on NEBULA_X2 boards
*   `MSG_CONFIRM_FLASHED_SPI`: Post-flash validation challenge-response (BH firmware v19+)

**Sources**: [tt_flash/chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L109-L110)[tt_flash/chip.py 118](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L118-L118)

### Example: WhChip Bundle Version Retrieval

The `get_bundle_version_v1()` function demonstrates complex ARC messaging:

```
# Query firmware version to check minimum version
fw_version = chip.arc_msg(
    chip.fw_defines["MSG_TYPE_FW_VERSION"], 
    wait_for_done=True, 
    arg0=0, 
    arg1=0
)[0]

# Query running bundle version (arg0=1)
temp = chip.arc_msg(
    chip.fw_defines["MSG_TYPE_FW_VERSION"],
    wait_for_done=True,
    arg0=1,
    arg1=0
)[0]

# Parse version from 32-bit response
patch = temp & 0xFF
minor = (temp >> 8) & 0xFF
major = (temp >> 16) & 0xFF
component = (temp >> 24) & 0xFF
running_bundle_version = (component, major, minor, patch)

# Query SPI bundle version (arg0=2)
temp = chip.arc_msg(
    chip.fw_defines["MSG_TYPE_FW_VERSION"],
    wait_for_done=True,
    arg0=2,
    arg1=0
)[0]
```

The response is a 32-bit integer where each byte encodes a version component: `[component][major][minor][patch]`.

## Telemetry Interface

The telemetry interface retrieves structured hardware status information including firmware versions, boot state, thermal data, and board-specific configuration. Telemetry is accessed through pyluwen's `Telemetry` object.

### Telemetry Method Summary

| Method | Return Type | Purpose |
| --- | --- | --- |
| `get_telemetry()` | `Telemetry` | Fetch fresh telemetry, update cache |
| `get_telemetry_unchanged()` | `Telemetry` | Return cached telemetry or fetch if none |
| `m3_fw_app_version()` | `tuple[int,int,int,int]` | M3 firmware version from telemetry |
| `smbus_fw_version()` | `tuple[int,int,int,int]` | SMBus (ARC1) firmware version |
| `arc_l2_fw_version()` | `tuple[int,int,int,int]` | ARC L2 (ARC0) firmware version |
| `get_asic_location()` | `int` | ASIC location for P300 boards (0=L, 1=R) |

**Sources**: [tt_flash/chip.py 163-200](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L163-L200)

### Telemetry Retrieval Flow

**Sources**: [tt_flash/chip.py 163-171](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L163-L171)

### Telemetry Caching Strategy

The `TTChip` class maintains a `telmetry_cache` instance variable initialized to `None`. Two access patterns exist:

1.   **`get_telemetry()`**: Always fetches fresh data from hardware, updates cache
2.   **`get_telemetry_unchanged()`**: Returns cached data if available, otherwise fetches and caches

This pattern optimizes for cases where multiple telemetry fields are needed without hardware polling overhead.

**Sources**: [tt_flash/chip.py 120](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L120-L120)[tt_flash/chip.py 163-171](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L163-L171)

### Telemetry Version Extraction

Helper methods extract version information from telemetry registers:

```
def __vnum_to_version(self, version: int) -> tuple[int, int, int, int]:
    return (
        (version >> 24) & 0xFF,  # component
        (version >> 16) & 0xFF,  # major
        (version >> 8) & 0xFF,   # minor
        version & 0xFF,          # patch
    )

def m3_fw_app_version(self):
    telem = self.get_telemetry_unchanged()
    return self.__vnum_to_version(telem.m3_app_fw_version)

def smbus_fw_version(self):
    telem = self.get_telemetry_unchanged()
    return self.__vnum_to_version(telem.arc1_fw_version)

def arc_l2_fw_version(self):
    telem = self.get_telemetry_unchanged()
    return self.__vnum_to_version(telem.arc0_fw_version)
```

These methods use cached telemetry and parse 32-bit version fields into `(component, major, minor, patch)` tuples.

**Sources**: [tt_flash/chip.py 173-191](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L173-L191)

### BhChip: Telemetry-Based Bundle Version

`BhChip` uses telemetry to retrieve the running firmware bundle version:

```
def get_bundle_version(self) -> FwVersion:
    telem = self.get_telemetry_unchanged()
    temp = telem.fw_bundle_version
    
    patch = temp & 0xFF
    minor = (temp >> 8) & 0xFF
    major = (temp >> 16) & 0xFF
    component = (temp >> 24) & 0xFF
    running = (component, major, minor, patch)
```

This is more efficient than the ARC message-based approach used by `WhChip`, directly reading the bundle version from a dedicated telemetry field.

**Sources**: [tt_flash/chip.py 253-260](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L253-L260)

### BhChip: ASIC Location from Telemetry

The `get_asic_location()` method reads ASIC position for P300 dual-ASIC boards:

```
def get_asic_location(self) -> int:
    try:
        location = super().get_asic_location()  # TTChip reads from telemetry
    except Exception:
        # Fallback to AXI GPIO read if telemetry fails
        GPIO_STRAP_REG_L = 0x80030D20
        gpio_strap = self.luwen_chip.axi_read32(GPIO_STRAP_REG_L)
        location = (gpio_strap >> 6) & 0x1
    
    return location  # 0=Left, 1=Right
```

This demonstrates a graceful degradation pattern: try telemetry first (fast, high-level), fall back to AXI register read if necessary.

**Sources**: [tt_flash/chip.py 277-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L277-L294)

## PyLuwen Integration

All hardware communication ultimately flows through the `pyluwen` library's `PciChip` class. The `TTChip` class maintains a reference to the underlying pyluwen chip and delegates all hardware operations to it.

### PyLuwen Chip Lifecycle

**Sources**: [tt_flash/chip.py 114-161](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L114-L161)

### Initialization Process

The `TTChip.__init__()` method receives a `PciChip` instance from pyluwen's detection functions:

```
def __init__(self, chip: PciChip):
    self.luwen_chip = chip
    self.interface_id = chip.pci_interface_id()
    self.fw_defines = init_fw_defines(self)
    self.telmetry_cache = None
```

The interface ID is cached to enable chip reinitialization after resets or driver reloads.

**Sources**: [tt_flash/chip.py 114-120](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L114-L120)

### Reinitialization After Reset

The `reinit()` method recreates the pyluwen connection after chip resets:

```
def reinit(self, callback=None):
    self.luwen_chip = PciChip(self.interface_id)
    self.telmetry_cache = None
    
    # Define chip_detect_callback for progress reporting...
    
    self.luwen_chip.init(
        callback=chip_detect_callback if callback is None else callback
    )
```

This is called after firmware flashing to re-establish communication with the newly flashed firmware. The callback parameter enables progress reporting during chip detection.

**Sources**: [tt_flash/chip.py 122-161](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L122-L161)

## Error Handling and Exceptions

Hardware communication operations can fail due to communication errors, firmware hangs, or hardware faults. The interface methods propagate exceptions from pyluwen, with callers responsible for exception handling.

### Exception Propagation Pattern

### FwVersion Exception Handling

The `get_bundle_version()` methods use `FwVersion` dataclass to encapsulate version data with exception information:

```
@dataclass
class FwVersion:
    allow_exception: bool
    exception: Optional[Exception]
    running: Optional[tuple[int, int, int, int]]
    spi: Optional[tuple[int, int, int, int]]
```

This allows version retrieval to partially succeed even if some operations fail:

```
try:
    # Attempt to read running version via telemetry
    running = (component, major, minor, patch)
    
    # Attempt to read SPI version
    spi = (component, major, minor, patch)
except Exception as e:
    exception = e

return FwVersion(
    allow_exception=True,
    exception=exception,
    running=running,
    spi=spi
)
```

The caller can check `exception` field and decide whether to proceed with partial information or abort.

**Sources**: [tt_flash/chip.py 21-27](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L21-L27)[tt_flash/chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L248-L275)[tt_flash/chip.py 83-91](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L83-L91)

### ASIC Location Fallback

The `BhChip.get_asic_location()` method demonstrates exception-based fallback:

```
try:
    location = super().get_asic_location()  # Try telemetry
except Exception:
    print(f"\rWarning: Unable to retrieve telemetry, reading ASIC location "
        "via fallback\n", end="", flush=True)
    gpio_strap = self.luwen_chip.axi_read32(GPIO_STRAP_REG_L)
    location = (gpio_strap >> 6) & 0x1
```

If telemetry is unavailable, the code falls back to direct AXI register access, printing a warning but continuing execution.

**Sources**: [tt_flash/chip.py 285-293](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L285-L293)

## Summary

The hardware communication interfaces in `TTChip` provide a clean abstraction over pyluwen's `PciChip` class, enabling firmware flashing and verification operations. The key interfaces are:

*   **AXI Bus**: Memory-mapped register access for configuration and status
*   **SPI Flash**: Read/write operations for firmware storage and verification
*   **ARC Messaging**: Request/response protocol for firmware commands and queries
*   **Telemetry**: Structured hardware status including versions, boot state, and configuration

All interfaces follow a delegation pattern where `TTChip` methods wrap pyluwen operations with minimal transformation, typically converting mutable buffers to immutable return values. Exception handling is delegated to callers, with some methods using structured error types (`FwVersion`) to support partial success scenarios.

**Sources**: [tt_flash/chip.py 113-305](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L113-L305)
