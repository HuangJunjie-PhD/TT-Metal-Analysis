The tt-torch testing framework provides comprehensive validation of PyTorch models through the compilation pipeline across multiple hardware configurations and execution modes. The framework implements two complementary testing strategies: basic unit tests for operation-level validation and full model tests for end-to-end validation with automated test discovery, configuration management, and CI/CD integration.

## Testing Architecture Overview

The testing framework consists of multiple layers, from basic operation tests to full model validation with hardware-specific configurations:

**Testing Framework Architecture**

**Key Testing Components:**

*   **Unit Tests**[tests/torch/test_basic.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py): Operation-level tests using `verify_module()` for basic PyTorch operations
*   **Model Tests**[tests/runner/test_models.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_models.py): Full model execution tests with dynamic discovery and hardware matrix testing
*   **Async Tests**[tests/torch/test_basic_async.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic_async.py): Device tensor tests using `verify_torch_module_async()` for asynchronous execution
*   **Configuration System**[tests/runner/test_config.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_config.py): Centralized configuration with per-model and per-architecture settings
*   **Verification System**: PCC and ATOL validation with configurable thresholds [tt_torch/tools/verify.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py)

## Two-Tier Testing Strategy

The framework implements two complementary testing approaches:

### Unit Tests (Operation-Level)

Basic tests validate individual PyTorch operations through the compilation pipeline using the `verify_module()` helper function [tests/torch/test_basic.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py) These tests:

*   Focus on single operations (add, multiply, concat, etc.)
*   Use simple input shapes and data types
*   Validate correctness with PCC and ATOL thresholds
*   Support parameterized testing with pytest
*   Can test multi-device execution [tests/torch/test_basic.py 66-106](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py#L66-L106)

Example test structure:

`def test_add():    class Basic(nn.Module):        def forward(self, x, y):            return torch.add(x, y)        verify_module(Basic(), input_shapes=[(256, 256)] * 2)`
### Model Tests (End-to-End)

Full model tests validate complete neural networks with automatic test discovery and hardware matrix testing [tests/runner/test_models.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_models.py) These tests:

*   Discover models dynamically from `third_party/tt_forge_models`
*   Use `DynamicTester` for model loading and execution [tests/runner/test_utils.py 152-168](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L152-L168)
*   Apply hardware-specific configurations via `arch_overrides`
*   Support multiple execution modes (full compilation, op-by-op)
*   Integrate with CI/CD for parallel test execution

## Verification System

The testing framework provides a comprehensive verification system for validating model outputs against golden references using Pearson Correlation Coefficient (PCC) and Absolute Tolerance (ATOL) metrics.

### Verification Architecture

**Verification Flow**

**Key Verification Functions:**

*   `verify_module()`[tests/torch/test_basic.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py): Synchronous verification for standard tests
*   `verify_torch_module_async()`[tests/torch/test_basic_async.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic_async.py): Asynchronous verification with device tensors
*   `verify_against_golden()`[tests/torch/test_basic.py 726-862](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py#L726-L862): Core comparison function with PCC/ATOL validation

**Verification Parameters:**

| Parameter | Type | Default | Purpose |
| --- | --- | --- | --- |
| `required_pcc` | float | 0.99 | Minimum PCC threshold for passing |
| `required_atol` | float | 0.01 | Maximum ATOL threshold for passing |
| `assert_pcc` | bool | True | Whether to fail test on PCC violation |
| `assert_atol` | bool | True | Whether to fail test on ATOL violation |
| `input_range` | tuple | (-1, 1) | Range for random input generation |

**Verification Test Examples:**

The framework includes comprehensive tests for the verification system itself [tests/torch/test_basic.py 724-862](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/torch/test_basic.py#L724-L862) validating behavior in four scenarios:

1.   **Low ATOL, High PCC**: Perfect numeric match (both pass)
2.   **High ATOL, High PCC**: Correlated but different (PCC passes, ATOL may fail)
3.   **Low ATOL, Low PCC**: Numerically close but uncorrelated (ATOL passes, PCC may fail)
4.   **High ATOL, Low PCC**: Completely different tensors (both fail)

## CI/CD Integration

The testing framework integrates with GitHub Actions to provide automated testing across hardware configurations with comprehensive reporting and artifact management.

### CI/CD Workflow Architecture

**Workflow Configuration**

The main test workflow [.github/workflows/tt-forge-models-tests.yml](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/tt-forge-models-tests.yml) accepts configurable parameters:

*   `tests-filter-passing`: Filter for expected passing tests (default: "expected_passing")
*   `num-splits-passing`: Number of test splits for parallel execution (default: 8)
*   `runners`: JSON array of target hardware (default: ["wormhole","blackhole"])
*   `reuse-build-run-id`: Option to reuse build artifacts from previous runs

**Dynamic Matrix Generation**

The CI system generates test execution matrices dynamically based on the number of splits, creating parallel jobs for each hardware target:

`{  "wh-runner": "wormhole_b0",  "bh-runner": "p150",   "name": "grp_1",  "group": 1}`
**Test Execution Environment**

Each test job runs in Docker containers with:

*   Hardware device access (`/dev/tenstorrent/0`)
*   Shared memory configuration (`--shm-size=2gb`)
*   Cached model files (`HF_HOME`, `TORCH_HOME`)
*   Memory usage monitoring

## Performance Benchmarking and Analysis

The testing framework includes comprehensive performance monitoring and cross-repository integration for tracking model execution performance across hardware configurations.

### Test Result Analysis

The framework automatically generates test reports in multiple formats and tracks execution metrics:

*   **JUnit XML Reports**: Standard test result format for CI integration
*   **Memory Usage Tracking**: CSV logs of memory consumption during test execution
*   **Test Duration Database**: Historical timing data for test splitting optimization
*   **PCC/ATOL Validation**: Numerical accuracy validation with configurable thresholds

**Cross-Repository Integration**

For performance benchmarking workflows and cross-repository model testing integration, see [Performance Benchmarking](https://deepwiki.com/tenstorrent/tt-torch/4.2.4-performance-benchmarking).
