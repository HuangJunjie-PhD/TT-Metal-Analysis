This page documents vLLM's memory management system in the v1 architecture, focusing on KV cache block allocation, prefix caching, and memory optimization strategies. The key-value (KV) cache is the most memory-intensive component of LLM inference, and vLLM's block-based management approach enables efficient memory utilization and high throughput.

## Overview of Memory Management in vLLM

vLLM's v1 memory management architecture revolves around three core components:

1.   **KVCacheManager**: Coordinates block allocation, prefix caching, and memory optimization
2.   **BlockPool**: Manages the physical pool of KV cache blocks
3.   **KVCacheBlock**: Represents individual memory blocks with metadata for caching and reference counting

The system uses a block-based allocation strategy where memory is divided into fixed-size blocks (typically 16 tokens per block). Blocks are allocated on-demand and can be shared across requests through prefix caching.

## KV Cache Architecture

### KVCacheBlock and Block Metadata

The `KVCacheBlock` class represents a single block of KV cache memory with associated metadata for memory management and prefix caching.

**KVCacheBlock Structure**

Key fields:

*   `block_id`: Physical block index (0 to `num_blocks - 1`)
*   `ref_cnt`: Reference count for the block (number of requests using it)
*   `_block_hash`: Hash of block contents for prefix caching (only set when block is full)
*   `prev_free_block` / `next_free_block`: Doubly-linked list pointers for free block management
*   `is_null`: Marks blocks that should never be cached (e.g., for sliding window attention)

### BlockPool: Physical Memory Management

The `BlockPool` manages the physical pool of KV cache blocks and maintains the mapping between block hashes and blocks for prefix caching.

**BlockPool Components**

The `BlockPool` provides:

*   **Allocation**: `allocate_block()` pops blocks from `free_block_queue`
*   **Deallocation**: `free_block()` returns blocks to the queue (with LRU ordering)
*   **Caching**: `cache_block()` associates block hash with physical block
*   **Lookup**: `find_cached_block()` retrieves cached blocks by hash
*   **Touch**: `touch()` updates LRU status to prevent eviction

## Block Allocation

### KVCacheManager: High-Level Allocation Interface

The `KVCacheManager` provides the high-level API for block allocation and manages interactions with the `BlockPool`. It tracks which blocks are allocated to which requests and implements prefix caching logic.

**Block Allocation Flow**

The allocation process involves:

1.   **Prefix cache lookup**: Check if any blocks can be reused from previously computed requests
2.   **Block count calculation**: Determine how many new blocks are needed
3.   **Free block check**: Verify sufficient free blocks are available
4.   **Allocation**: Pop blocks from the free queue and associate them with the request
5.   **Failure handling**: Return `None` if allocation fails, triggering preemption

## Prefix Caching

Prefix caching allows vLLM to reuse KV cache blocks from previously computed requests, significantly improving throughput when requests share common prefixes.

### Block Hashing for Prefix Caching

Each full KV cache block is assigned a hash based on its token contents. The hash is computed incrementally as blocks fill up:

**Block Hash Computation**

Hash computation properties:

*   Hashes are computed using the token contents and parent block hash
*   Hash includes block position in sequence (via parent hash chaining)
*   Multimodal features and LoRA IDs are included in hash when present
*   Hashes are only computed for full blocks (partial blocks remain unhashed)

### Cache Hit Detection

When a new request arrives, `KVCacheManager.get_computed_blocks()` checks for prefix cache hits:

**Prefix Cache Lookup Process**

Cache hit benefits:

*   Skips recomputation of cached tokens
*   Reduces scheduling latency for prefilled tokens
*   Shares blocks across multiple requests with same prefix
*   Particularly effective for prompt templates with varying suffixes

### Cache Block Lifecycle

**Block States and Transitions**

Block lifecycle:

1.   **Free**: Block is in `FreeKVCacheBlockQueue`, available for allocation
2.   **In Use**: Block is allocated to request(s), `ref_cnt > 0`
3.   **Cached**: Full block with hash stored in `BlockHashToBlockMap` for prefix caching
4.   **Eviction**: When free blocks are exhausted, cached blocks with `ref_cnt == 0` are evicted (LRU)

## Memory Optimization Strategies

### Reference Counting and Block Sharing

The `ref_cnt` field in `KVCacheBlock` enables efficient block sharing:

*   **Shared prefix blocks**: Multiple requests can reference the same cached blocks
*   **Reference counting**: `ref_cnt` tracks how many requests use each block
*   **Delayed deallocation**: Blocks are only returned to free pool when `ref_cnt` reaches 0

### LRU Eviction Policy

When free blocks are exhausted, the `BlockPool` evicts cached blocks using an LRU (Least Recently Used) policy:

**Free Block Queue Ordering**

The `FreeKVCacheBlockQueue` maintains blocks in LRU order:

1.   Newly freed blocks are appended to the tail (most recently used)
2.   Blocks are allocated from the head (least recently used)
3.   `touch()` operation moves blocks to tail when accessed via cache hit

Eviction ensures:

*   Frequently accessed cached blocks remain available
*   Rarely used cached blocks are evicted first
*   Cache hit rate improves over time as hot blocks stabilize

## Memory Swapping

When GPU memory is exhausted, vLLM can swap KV cache blocks between GPU and CPU memory.

### Swapping Mechanism

1.   The scheduler identifies which blocks to swap in and out based on the current state and requested operations
2.   Blocks that belong to actively processing sequences stay in GPU memory
3.   Blocks from waiting or low-priority sequences may be swapped out to CPU
4.   When a swapped-out sequence becomes active again, its blocks are swapped back in

The swapping mechanism enables vLLM to handle more concurrent requests than would fit in GPU memory alone, at the cost of some performance overhead for the CPU-GPU data transfer.

## Memory Efficiency and Throughput

The combination of block-based KV cache and PagedAttention enables vLLM to achieve high memory efficiency and throughput:

1.   **Reduced Fragmentation**: Block-based allocation minimizes memory fragmentation
2.   **Continuous Batching**: Allows sequences of different lengths to be batched efficiently
3.   **Efficient Preemption**: Low-priority sequences can be paused by swapping out their blocks
4.   **Memory Oversubscription**: More sequences can be managed than would fit in GPU memory simultaneously

This memory management approach is one of the key factors enabling vLLM to achieve high throughput for LLM inference.

## Advanced Memory Features

### Prefix Caching

Prefix caching is an optimization that allows vLLM to reuse computation for common prompt prefixes. When enabled:

1.   The system caches the KV states for prompt prefixes that have been processed before
2.   When a prompt with the same prefix arrives, computation is reused instead of being redone
3.   This greatly accelerates processing when the same prompt template is used with minor variations

### CPU Offloading for Model Weights

In addition to the KV cache, vLLM can also offload model weights to CPU memory:

1.   The `cpu_offload_gb` parameter controls how much model weight data is stored in CPU
2.   During inference, weights are loaded to GPU as needed
3.   This effectively increases the available GPU memory at the cost of some computation speed
4.   Useful for running very large models on GPUs with limited memory

### Sleep Mode

vLLM includes a sleep mode to free up GPU memory when the system is idle:

1.   When enabled with `enable_sleep_mode`, the system can offload weights to CPU when not in use
2.   Activates automatically when no requests are being processed
3.   Weights are quickly restored when new requests arrive
4.   Allows other applications to use GPU memory when vLLM is idle

## Memory Management in Distributed Settings

When running with multiple GPUs, vLLM extends its memory management to handle distributed execution:

1.   **Tensor Parallelism**: Model layers are split across GPUs, each handling a portion of the computation
2.   **Pipeline Parallelism**: Different transformer layers run on different GPUs
3.   **KV Cache Distribution**: Each GPU maintains its own KV cache blocks for its part of the model
4.   **Synchronization**: The distributed executor ensures proper synchronization of memory operations

## Conclusion

vLLM's memory management system is a key component of its high-performance inference capabilities. By using block-based allocation, PagedAttention, and efficient swapping mechanisms, it maximizes GPU memory utilization while supporting features like continuous batching and sequence preemption.

Understanding these memory management concepts is essential for effectively configuring and optimizing vLLM deployments, especially when working with large models or handling high request volumes.
