## Purpose and Scope

This page documents the TVM frontend converters that translate models from PyTorch, ONNX, TensorFlow, JAX, and PaddlePaddle into Apache TVM's Relay intermediate representation (IR). It covers the Relay IR transformation passes implemented via the `DFPatternCallback` system, graph partitioning logic that determines CPU vs. device execution, and the bridge to Forge operations.

Key topics include:

*   TVM frontend converters (`from_pytorch`, `from_onnx`, `from_tensorflow`, etc.)
*   Relay IR graph transformations via `DFPatternCallback` passes
*   Graph partitioning and CPU fallback mechanisms
*   Op name mapping and argument translation to Forge operations

For the overall compilation flow, see [Compilation Pipeline Overview](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.1-compilation-pipeline-overview). For framework-specific module wrappers, see [Multi-Framework Support](https://deepwiki.com/tenstorrent/tt-forge-onnx/4.2-multi-framework-support). For the TVM-to-Python code generation, see [Operation Translation](https://deepwiki.com/tenstorrent/tt-forge-onnx/4.3-operation-translation-(tvm-to-python)).

## TVM Frontend Converters

Apache TVM provides frontend converters that translate models from different frameworks into the unified Relay IR. Each framework has a dedicated import function:

| Framework | TVM Converter Function | Input Type |
| --- | --- | --- |
| PyTorch | `tvm.relay.frontend.from_pytorch()` | `torch.jit.ScriptModule` (traced model) |
| ONNX | `tvm.relay.frontend.from_onnx()` | `onnx.ModelProto` |
| TensorFlow | `tvm.relay.frontend.from_tensorflow()` | `tf.GraphDef` or concrete function |
| JAX | via TensorFlow | JAX → `jax2tf` → TensorFlow converter |
| PaddlePaddle | `tvm.relay.frontend.from_paddle()` | `paddle.nn.Layer` (via `jit.to_static`) |

## Relay IR in the Compilation Pipeline

Once converted to Relay IR, the graph undergoes several transformation stages:

1.   **Relay Passes**: Standard TVM optimization passes (constant folding, dead code elimination, etc.)
2.   **Forge DFPatternCallback Passes**: Custom graph rewrites for Forge compatibility
3.   **Graph Partitioning**: Determines which subgraphs execute on CPU vs. device
4.   **Op Name Mapping**: Converts Relay ops to Forge operation names
5.   **Argument Translation**: Extracts attributes and maps them to Forge op arguments

## Relay Graph Transformation Passes

The file `forge/forge/tvm_calls/relay/op/forge_passes.py` contains all custom TVM `DFPatternCallback` subclasses. Each callback matches a specific subgraph pattern in Relay and rewrites it to an equivalent form that either simplifies the graph or maps cleanly to a Forge op.

TVM applies these passes using `tvm.relay.dataflow_pattern.rewrite(callback, relay_module)` internally. Each `DFPatternCallback` defines:

*   A **pattern** (using `wildcard()`, `is_op()`, `is_constant()` combinators) that matches the target subgraph.
*   A **`callback(pre, post, node_map)`** method that returns the replacement subgraph.

### ConvertLayout

`ConvertLayout` matches `nn.conv2d`, `nn.max_pool2d`, `nn.avg_pool2d`, `nn.conv2d_transpose`, and pooling variants. When a `conv2d` uses `data_layout="NHWC"` with `kernel_layout="HWIO"` or `"HWOI"`, the callback transposes the weight tensor to `OIHW` and rewrites the op. Other layouts are passed through unchanged.

[forge/forge/tvm_calls/relay/op/forge_passes.py 19-96](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L19-L96)

### ResolveConvChannels

`ResolveConvChannels` matches the pattern produced when a channel-last model wraps a convolution in four transpose ops to move between `NCHW` and `NHWC`. It detects the double-transpose sandwich and replaces the entire sequence with a single conv/pool op whose `data_layout` has been toggled between `NCHW` and `NHWC`.

[forge/forge/tvm_calls/relay/op/forge_passes.py 99-157](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L99-L157)

### FuseConvAndPoolPadding

`FuseConvAndPoolPadding` matches a zero-constant `nn.pad` followed immediately by a `nn.conv2d` or `nn.max_pool2d` that has no existing padding. It adds the pad amounts into the op's `padding` attribute and eliminates the separate pad node. Asymmetric padding on `max_pool2d` and non-constant pad modes (`edge`, `reflect`) are excluded from fusion.

[forge/forge/tvm_calls/relay/op/forge_passes.py 160-214](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L160-L214)

### DecomposeStack

`DecomposeStack` detects the pattern TVM emits when lowering `tf.stack`: two `expand_dims` operations whose outputs are concatenated and then reshaped. It replaces this with `relay.stack` followed by `relay.reshape`, which maps cleanly onto Forge's `stack` op.

[forge/forge/tvm_calls/relay/op/forge_passes.py 383-414](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L383-L414)

## Argument Population Functions

Each `populate_torch_*_args` function extracts attribute values from the TVM graph node dict and returns a standardized `args` dict with `"inp"`, `"attr"`, and `"inplace"` keys. The `"attr"` sub-dict maps each argument name to its value and positional index (`inp_pos`) in the target PyTorch call.

The internal helper `_populate_torch_init_args` initializes the dict structure and collects the input node names from `curr_node["inputs"]`.

[forge/forge/tvm_to_python.py 406-413](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_to_python.py#L406-L413)

| Function | Key behavior |
| --- | --- |
| `populate_torch_transpose_args` | Converts TVM's full axes permutation list to a pair of swapped dimension indices for `torch.transpose` |

| `populate_avgpool2d_args` | Reorders TVM padding from `[top, left, bottom, right]` to `[left, right, top, bottom]` |

| `forge_cpudevice.concatenate` | `concatenate` |
