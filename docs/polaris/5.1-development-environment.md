This document details the development environment setup for the Polaris repository, including conda environment configuration, code quality tools, and development workflow infrastructure. The development environment emphasizes code quality, license compliance, and automated testing to ensure consistent development practices across the project.

For information about test automation and CI/CD workflows, see [Test Automation](https://deepwiki.com/tenstorrent/polaris/5.2-test-automation). For details about specific code quality tools and compliance checking, see [Code Quality and Compliance](https://deepwiki.com/tenstorrent/polaris/5.3-code-quality-and-compliance).

## Environment Setup

The Polaris development environment is configured using conda with Python 3.13.2 as the base runtime. The environment specification is defined in `envdev.yaml` with the environment name `polarisdev`.

### Core Dependencies

The development environment includes several categories of dependencies:

| Category | Dependencies | Purpose |
| --- | --- | --- |
| **Core Runtime** | `python=3.13.2` | Base Python interpreter |
| **Configuration** | `hydra-core=1.3.2`, `pyyaml=6.0.2`, `yaml=0.2.5` | Configuration management |
| **Data Processing** | `numpy=2.2.3`, `networkx=3.4.2`, `matplotlib=3.10.0` | Scientific computing and visualization |
| **ML/AI** | `onnx=1.17.0` | Neural network model support |
| **Validation** | `pydantic=2.10.6`, `deepdiff` | Data validation and comparison |
| **Development Tools** | `mypy=1.15.0`, `pytest==8.3.4`, `coverage=7.6.12`, `pre-commit` | Code quality and testing |
| **Utilities** | `loguru`, `gitpython`, `openpyxl=3.1.5`, `lxml` | Logging, git integration, file processing |

### SPDX License Compliance System

The `tools/spdxchecker.py` implements a comprehensive license and copyright validation system supporting multiple programming languages.

#### Language Support and Syntax

The SPDX checker supports multiple file types with language-specific comment syntax:

| Language | Extensions | Comment Syntax |
| --- | --- | --- |
| **Python** | `.py` | `#` |
| **JavaScript** | `.js`, `.mjs`, `.cjs` | `//` |
| **HTML** | `.html`, `.htm` | `<!-- -->` |
| **CSS** | `.css` | `/* */` |
| **YAML** | `.yaml`, `.yml` | `#` |

#### SPDX Validation Components

The system validates two required SPDX headers:

*   `SPDX-License-Identifier: Apache-2.0`
*   `SPDX-FileCopyrightText: (C) 2025 Tenstorrent AI ULC`

## Development Workflow Integration

The development environment integrates with git workflows through automated file classification and ignore pattern handling.

### File Processing Pipeline

The system processes files through several stages:

1.   **File Collection**: `collect_all_files()` recursively scans directories [tools/spdxchecker.py 137-152](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L137-L152)
2.   **Git Integration**: `collect_git_status_files()` identifies ignored and untracked files [tools/spdxchecker.py 111-134](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L111-L134)
3.   **Language Classification**: `ext_2_lang()` maps file extensions to programming languages [tools/spdxchecker.py 69-77](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L69-L77)
4.   **Header Analysis**: `analyze_file()` validates SPDX headers per language [tools/spdxchecker.py 165-187](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L165-L187)

### Ignore Pattern Configuration

The system supports flexible ignore patterns through `IgnoreFileModel` which allows:

*   **Ignore patterns**: Files matching these patterns are completely skipped
*   **Warning patterns**: Files matching these patterns generate warnings instead of errors

## Tool Configuration

The development environment uses several configuration files to coordinate tool behavior:

### Environment Dependencies

The conda environment pulls dependencies from both conda-forge and pip:

**Conda Dependencies** (from conda-forge):

*   Core libraries: `python=3.13.2`, `numpy=2.2.3`, `pydantic=2.10.6`
*   Development tools: `mypy=1.15.0`, `coverage=7.6.12`, `pre-commit`

**Pip Dependencies**:

*   Testing: `pytest==8.3.4`, `pytest-mock`
*   Type annotations: `types-PYYAML==6.0.12.20241230`, `types-networkx==3.4.2.20241227`

### Pre-commit Hook Arguments

The SPDX checker is configured with specific arguments:

*   `--loglevel debug` and `--loglevel error` for different verbosity levels
*   `--ignore .github/spdxchecker-ignore.yml` for custom ignore patterns
*   `--allowed-licenses Apache-2.0` for license validation
*   `--allowed-copyright "Tenstorrent AI ULC"` for copyright validation

## Development Environment Features

### Static Analysis Integration

The development environment integrates `mypy` for static type checking with the entire codebase:

*   Executed via pre-commit hook: `mypy .`
*   Configured with type stub packages for external dependencies
*   Ensures type safety across the entire project

### Unit Testing Framework

The testing infrastructure uses `pytest` with specific markers:

*   Unit tests executed with `pytest -m unit`
*   Mock support through `pytest-mock`
*   Coverage analysis with `coverage=7.6.12`

### License Compliance Automation

The SPDX checker provides comprehensive license compliance:

*   Supports multiple programming languages
*   Validates both license identifiers and copyright notices
*   Integrates with git to respect `.gitignore` patterns
*   Provides detailed error reporting and diagnostics
