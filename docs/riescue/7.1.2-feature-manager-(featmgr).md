# Feature Manager (FeatMgr)

The Feature Manager (`FeatMgr`) is a central configuration consolidation system in RiescueD that merges configuration parameters from multiple sources into a single unified configuration object used throughout test generation, compilation, and simulation.

This page covers the internal architecture and usage of `FeatMgr` and `FeatMgrBuilder`. For information about specific test directives that populate the test header, see [Test Directives and Syntax](https://deepwiki.com/tenstorrent/riescue/2.1.2-test-directives-and-syntax). For CPU configuration file format, see [CPU Configuration](https://deepwiki.com/tenstorrent/riescue/7.2.1-cpu-configuration).

## Overview

`FeatMgr` serves as the single source of truth for all feature and environment configuration during test execution. It consolidates parameters from three distinct sources with different precedence levels, resolving conflicts and providing a unified interface for downstream components like the Generator, Compiler, and ISS tools.

The configuration system uses the Builder pattern via `FeatMgrBuilder` to construct immutable `FeatMgr` instances with proper validation and precedence handling.

### Test Header

The test header is parsed from directives at the beginning of the `.rasm` test file. These directives define test-specific configuration like privilege mode, paging mode, and environment settings. The `Parser` class extracts these directives into a `test_header` dataclass that is passed to `FeatMgrBuilder`.

### CPU Configuration JSON

The CPU configuration file (`cpu_config.json` by default) defines hardware-level features including:

*   Memory map (RAM, ROM, device memory regions)
*   Supported ISA extensions
*   Physical memory attributes
*   Core-specific features

This configuration represents the target CPU's capabilities and is loaded from a JSON file path.

**Sources:**[riescue/riescued.py 60](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L60-L60)[riescue/riescued.py 234](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L234-L234)

### Command-Line Arguments

Command-line arguments provide runtime overrides for configuration. These have the highest precedence and can override both test header and CPU config settings. Arguments are added to the parser via `FeatMgrBuilder.add_arguments()`.

**Sources:**[riescue/riescued.py 136](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L136-L136)[riescue/riescued.py 235-236](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L235-L236)

## FeatMgrBuilder Pattern

The `FeatMgrBuilder` class implements the Builder pattern to construct `FeatMgr` instances. This pattern allows for flexible, step-by-step construction with clear separation between configuration sources.

### Builder Methods

| Method | Purpose | Required |
| --- | --- | --- |
| `with_test_header(test_header)` | Add parsed test header directives | Yes |
| `with_cpu_json(cpu_config_path)` | Load CPU configuration from JSON file | Yes |
| `with_args(args)` | Add command-line argument overrides | No |
| `build(rng)` | Construct final `FeatMgr` instance | Yes |

The `build()` method takes a `RandNum` instance for any randomization needed during configuration resolution.

**Sources:**[riescue/riescued.py 232-237](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L232-L237)

## Configuration Precedence

Configuration sources are applied in order of increasing precedence:

1.   **CPU Config JSON** (lowest precedence) - Hardware capabilities
2.   **Test Header** (middle precedence) - Test-specific defaults
3.   **Command-Line Arguments** (highest precedence) - Runtime overrides

When the same parameter is specified in multiple sources, the higher precedence source takes effect. This allows tests to specify defaults while still permitting runtime customization.

**Sources:**[riescue/riescued.py 232-237](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L232-L237)

## WYSIWYG Mode

The "What You See Is What You Get" (wysiwyg) mode requires special handling in `FeatMgr` and simulation:

*   Disables OS code injection and uses raw assembly
*   Requires custom end-of-test detection via x31 register value (`0xc001c0de`)
*   Uses `--endpc` / `--end-pc` in ISS to stop at `<failed>` label

The simulation code parses the disassembly to find the `<failed>` label PC and uses it as the end condition.
