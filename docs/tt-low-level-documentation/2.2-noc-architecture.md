## Purpose and Scope

This document provides a detailed technical specification of the Network-on-Chip (NOC) architecture that forms the core data movement infrastructure in TT-Metal. The NOC is a packet-switched interconnect that enables communication between Tensix cores, DRAM, Ethernet, and other system components. This page covers the NOC's physical topology, dual-instance architecture, virtual channel system, packet structure, and control mechanisms.

For an overview of data movement concepts and the testing framework, see [Introduction to Data Movement](https://deepwiki.com/tenstorrent/tt-low-level-documentation/2.1-introduction-to-data-movement). For platform-specific details about Wormhole B0 and Blackhole P100, see [Hardware Platforms](https://deepwiki.com/tenstorrent/tt-low-level-documentation/2.3-hardware-platforms).

## Dual NOC Instance Architecture

The data movement system implements **two independent NOC instances** that operate in parallel:

*   **NOC0**: Controlled by the BRISC processor
*   **NOC1**: Controlled by the NCRISC processor

Each NOC instance provides a complete, independent communication fabric with its own routing logic, virtual channels, and traffic management. This dual-instance design enables:

1.   **Increased aggregate bandwidth** through parallel operation
2.   **Traffic separation** between different data movement classes
3.   **Routing diversity** with complementary routing behaviors (NOC0: column-emphasis, NOC1: row-emphasis)

**Dual NOC Instance Control Architecture**

Each RISC-V processor has exclusive control over its associated NOC instance, with dedicated register interfaces for issuing operations. The BRISC typically handles compute-related data movement on NOC0, while NCRISC manages I/O and system communication on NOC1.

## 2D Torus Topology

The NOC routers are arranged in a **2D torus topology**, also known as a ring or donut topology. In this configuration:

*   Each router connects to its four neighbors: north, south, east, west
*   Wraparound connections exist at the grid edges (creating the "torus" shape)
*   Folding is applied to ensure **uniform latency** between any two nodes

**2D Torus Topology with Wraparound Connections**

The torus topology provides:

*   **Path diversity**: Multiple routes between any source-destination pair
*   **Uniform distance**: Folding ensures consistent hop counts
*   **Deadlock avoidance**: Enabled through virtual channel allocation
*   **Scalability**: Extends naturally to larger grid sizes

The actual physical layout includes additional optimizations for Wormhole (8x8 grid) and Blackhole (12x10 grid) configurations.

## Packet-Switched Network

The NOC operates as a **packet-switched network** where data is transmitted in discrete packets:

### Packet Structure

| Parameter | Wormhole B0 | Blackhole P100 |
| --- | --- | --- |
| **Maximum Packet Size** | 8 KB | 16 KB |
| **Interface Width** | 256 bits/cycle (32 bytes/cycle) | 512 bits/cycle (64 bytes/cycle) |
| **Flit Size** | Variable (multiple cycles) | Variable (multiple cycles) |

Each packet is subdivided into **flits** (flow control units) for transmission across router links. A packet header contains routing information, while the payload carries the actual data.

**Packet Format and Transmission Flow**

### Operation Types

The NOC supports three primary operation types:

1.   **Read Operations**: Request data from a remote location (DRAM, L1, or remote core)
2.   **Write Operations**: Send data to a remote location 
    *   **Unicast**: Single destination
    *   **Multicast**: Multiple destinations simultaneously
    *   **Posted**: Fire-and-forget (no acknowledgment)
    *   **Non-Posted**: Requires acknowledgment

3.   **Atomic Operations**: Read-modify-write operations with hardware atomicity guarantees

## Virtual Channel Architecture

Each NOC instance provides **16 virtual channels** (VCs) per physical channel. Virtual channels enable multiple logical data streams to share the same physical links without head-of-line blocking.

**Virtual Channel Organization**

### Virtual Channel Categories

| Category | Count | Purpose |
| --- | --- | --- |
| **Non-Dateline VCs** | 8 | Normal routing without crossing the torus wraparound |
| **Dateline VCs** | 8 | Routing that crosses the torus wraparound (deadlock prevention) |
| **Unicast VCs** | 4 (subset) | Software-controlled channels for separating traffic classes |

### Traffic Separation

Virtual channels are primarily used to separate different **traffic classes** rather than parallelizing identical traffic. Recommended VC usage:

*   **DRAM traffic**: Dedicated VC(s)
*   **L1 traffic**: Dedicated VC(s)
*   **Ethernet traffic**: Dedicated VC(s)
*   **Credit/control traffic**: Dedicated VC(s)

Attempting to split the same traffic class across multiple VCs typically **degrades performance** due to VC arbitration overhead and shared input buffer limitations.

## Register Interface and Control

Each RISC-V processor (BRISC, NCRISC) controls its NOC instance through memory-mapped registers. The NOC API provides functions that abstract register access:

### Key NOC API Functions

While the actual API implementation details are in the tt-metal repository, the NOC operations generally follow these patterns:

*   **`noc_async_read`**: Initiate asynchronous read from remote location
*   **`noc_async_write`**: Initiate asynchronous write to remote location
*   **`noc_async_read_barrier`**: Wait for pending reads to complete
*   **`noc_async_write_barrier`**: Wait for pending writes to complete
*   **`noc_semaphore_*`**: Atomic semaphore operations

### Stateful vs. Stateless Operations

The NOC supports both:

1.   **Stateful operations**: Configure registers once, reuse for multiple transactions (e.g., `noc_async_read_one_packet_set_state`)
2.   **Stateless operations**: Each transaction programs all necessary registers (e.g., `noc_async_read_one_packet`)

Stateful operations reduce programming overhead for repeated data movement patterns. See [Stateful vs Stateless APIs](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.5-stateful-vs-stateless-apis) for detailed usage patterns.

**Sources:**[data_movement_doc/general/intro_to_dm.md 1-107](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/intro_to_dm.md?plain=1#L1-L107)

## Routing Behavior

The two NOC instances exhibit complementary routing behaviors:

**Complementary Routing Strategies**

### Routing Implications

*   **NOC0 (Column-Emphasis)**: Prioritizes vertical movement first, then horizontal

    *   Shares columns with cores in the same vertical line
    *   Better for vertical data distribution patterns

*   **NOC1 (Row-Emphasis)**: Prioritizes horizontal movement first, then vertical

    *   Shares rows with cores in the same horizontal line
    *   Better for horizontal data distribution patterns

This complementary behavior becomes critical in multicast scenarios where choosing the wrong NOC can lead to **self-interference** and significant performance degradation. See [Multicast Schemes Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study) for detailed analysis.

## Address Space and Addressing

The NOC supports addressing for multiple memory types:

| Memory Type | Description | Access Pattern |
| --- | --- | --- |
| **L1 Memory** | Local SRAM within each Tensix core | NOC read/write |
| **DRAM** | Off-chip memory banks | NOC read/write via DRAM controllers |
| **Ethernet** | Chip-to-chip communication ports | NOC read/write via Ethernet cores |
| **PCIe** | Host communication interface | NOC read/write via PCIe controller |

### Addressing Modes

*   **48-bit addressing** (Wormhole B0): Supports core coordinates + local address
*   **64-bit addressing** (Blackhole P100): Extended address space for larger configurations

The NOC API provides helper functions to construct NOC addresses from logical coordinates and memory offsets.

## Performance Characteristics

### Theoretical Bandwidth

| Platform | NOC0 Bandwidth | NOC1 Bandwidth | Combined Bandwidth |
| --- | --- | --- | --- |
| **Wormhole B0** | 256 bits/cycle (32 bytes/cycle) | 256 bits/cycle (32 bytes/cycle) | 512 bits/cycle (64 bytes/cycle) |
| **Blackhole P100** | 512 bits/cycle (64 bytes/cycle) | 512 bits/cycle (64 bytes/cycle) | 1024 bits/cycle (128 bytes/cycle) |

### Latency Considerations

*   **Hop latency**: Fixed per-router delay
*   **Contention**: Variable delay when multiple packets compete for the same link
*   **Virtual channel arbitration**: Additional cycles when multiple VCs are active
*   **Acknowledgment path**: Return path for non-posted writes adds latency

For detailed performance measurements and analysis, see [Baseline Performance Metrics](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.1-baseline-performance-metrics).

## Summary

The NOC architecture provides a sophisticated, high-bandwidth interconnect for data movement in TT-Metal:

*   **Dual independent instances** (NOC0, NOC1) with RISC-V control
*   **2D torus topology** with uniform latency and deadlock-free routing
*   **Packet-switched network** supporting 8KB (Wormhole) or 16KB (Blackhole) packets
*   **16 virtual channels** per instance for traffic class separation
*   **Complementary routing behaviors** enabling optimized data distribution patterns
*   **Multiple address spaces** covering L1, DRAM, Ethernet, and PCIe

Understanding the NOC architecture is essential for writing efficient data movement kernels and achieving optimal performance in TT-Metal applications.
