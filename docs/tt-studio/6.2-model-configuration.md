This document covers the configuration system for AI models deployed in TT-Studio, including environment variables, runtime parameters, and model-specific settings. It focuses on how models are configured during deployment and execution, including weights selection and device assignment.

For information about the step-by-step model deployment process, see [6.1 Deployment Workflow](https://github.com/tenstorrent/tt-studio/blob/a6d347af/6.1%20Deployment%20Workflow) For details about the inference API and request processing, see [7.1 Inference API](https://github.com/tenstorrent/tt-studio/blob/a6d347af/7.1%20Inference%20API)

## Deployment Parameters

When a model is deployed via the frontend, several key parameters are passed to the backend to configure the container instance.

### Device and Port Assignment

In the `DeployModelStep.tsx` and `SelectionSteps.tsx` components, the system manages how models are bound to specific hardware and network resources:

*   **Device ID Assignment**: The `selectedDeviceId` determines which Tenstorrent chip the model will occupy [app/frontend/src/components/DeployModelStep.tsx 19-26](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L19-L26)
*   **Port Binding**: The `host_port` is used to map the internal model service port (defaulting to 7000) to a host-accessible port [app/frontend/src/components/DeployModelStep.tsx 24-25](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L24-L25)
*   **Chip Allocation**: The system monitors available slots via the `/docker-api/chip-status/` endpoint to prevent over-subscription of hardware [app/frontend/src/components/DeployModelStep.tsx 174-190](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/DeployModelStep.tsx#L174-L190)

### Weights Selection

The `model_weights_handler.py` manages how model weights and compiled TT-Metal caches are located:

*   **Default Weights**: If no specific weights are provided, the system defaults to an `id_default` directory within the cache root [models/dummy_echo_model/src/model_weights_handler.py 37-48](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/model_weights_handler.py#L37-L48)
*   **Custom Weights**: When `MODEL_WEIGHTS_ID` and `MODEL_WEIGHTS_PATH` are provided, the system validates the path and creates a corresponding TT-Metal cache directory [models/dummy_echo_model/src/model_weights_handler.py 49-61](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/model_weights_handler.py#L49-L61)

## Dummy Echo Model Reference Implementation

The `dummy_echo_model` serves as the reference implementation for how a model container should be structured and configured.

### Container Structure

The model is containerized using a multi-stage Dockerfile that installs dependencies and runs the application via `gunicorn`[models/dummy_echo_model/Dockerfile 5-65](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/Dockerfile#L5-L65) It exposes a health check on the `SERVICE_PORT`[models/dummy_echo_model/Dockerfile 64](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/Dockerfile#L64-L64)

### Backend Execution Logic

The `DummyEchoBackend` simulates the behavior of a Large Language Model (LLM):

1.   **Queue Management**: It picks prompts from an `input_queue`[models/dummy_echo_model/src/dummy_echo_backend.py 124-138](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/dummy_echo_backend.py#L124-L138)
2.   **Simulation**: It simulates a specific Tokens Per Second (TPS) rate (default 12.0) to mimic hardware performance [models/dummy_echo_model/src/dummy_echo_backend.py 145-148](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/dummy_echo_backend.py#L145-L148)
3.   **Output Generation**: It echoes the input text back to the `output_queue` chunk by chunk [models/dummy_echo_model/src/dummy_echo_backend.py 149-168](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/dummy_echo_backend.py#L149-L168)

### Process Affinity and Performance

To simulate real-world Tenstorrent deployment performance, the `inference_api_server.py` performs CPU pinning:

*   The **Backend Process** is pinned to NUMA node 0 CPUs to minimize latency [models/dummy_echo_model/src/inference_api_server.py 125-130](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/inference_api_server.py#L125-L130)
*   The **Flask API Server** is moved to non-NUMA node 0 CPUs and assigned a higher "nice" value (lower priority) to ensure inference is not interrupted by API overhead [models/dummy_echo_model/src/inference_api_server.py 131-140](https://github.com/tenstorrent/tt-studio/blob/a6d347af/models/dummy_echo_model/src/inference_api_server.py#L131-L140)
