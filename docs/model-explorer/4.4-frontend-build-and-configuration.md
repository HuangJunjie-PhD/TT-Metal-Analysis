## Build System Architecture

The frontend uses Angular CLI with a multi-project workspace configuration supporting both the main application and a standalone custom element build.

## Angular Build Configuration

The Angular workspace is configured with two distinct projects in `angular.json`:

### Project Structure

Sources: [src/ui/angular.json 5-155](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/angular.json#L5-L155)

### Main Application Build

The primary `model_explorer` project configuration:

*   **Output Path**: `dist/model_explorer`
*   **Entry Point**: `src/main.ts`
*   **Assets**: Copied from `public` directory
*   **Styles**: `src/styles.scss` as main stylesheet
*   **Web Workers**: Configured via `tsconfig.worker.json`

### Custom Element Build

The `custom_element` project creates a standalone web component:

*   **Output Path**: `dist/custom_element`
*   **Entry Point**: `src/custom_element/custom_element.ts`
*   **Assets**: Limited to fonts and icons only
*   **Styles**: `src/custom_element/styles.scss`
*   **File Replacements**: Different worker service in production

Sources: [src/ui/angular.json 98-154](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/angular.json#L98-L154)

## Build Configurations

### Production Configuration

`{  "outputHashing": "all",  "optimization": {    "scripts": true,    "styles": true,    "fonts": false  }}`
*   Enables content hashing for cache busting
*   Optimizes JavaScript and CSS
*   Preserves font files without optimization

### Development Configuration

`{  "optimization": false,  "extractLicenses": false,  "sourceMap": true}`
*   Disables optimization for faster builds
*   Includes source maps for debugging
*   Skips license extraction

Sources: [src/ui/angular.json 41-53](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/angular.json#L41-L53)

## NPM Package Build Process

The `build_npm.sh` script creates a distributable NPM package for the custom element:

### Build Process Steps

1.   **Compile Projects**: Builds both `custom_element` and `model_explorer`
2.   **Bundle Browser Script**: Creates browser-loadable version with esbuild
3.   **Prepare Directories**: Clears and creates output structure
4.   **Copy Assets**: Moves static files, CSS, and compiled JavaScript
5.   **Copy Sources**: Includes TypeScript definitions and component sources

Sources: [src/ui/scripts/build_npm.sh 21-48](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/scripts/build_npm.sh#L21-L48)

`// proxy.conf.json structure{  "/api/*": {    "target": "http://localhost:8080",    "secure": true,    "changeOrigin": true  }}`
This allows the frontend development server to proxy backend requests during development.

## Asset Management

### Static File Handling

Assets are managed through the Angular build system:

*   **Public Directory**: Static files copied as-is from `public/`
*   **Font Files**: Included in custom element builds
*   **Icons**: Specific icon files for custom element
*   **Worker Dependencies**: Special handling for web worker files

### Worker File Configuration

The visualization engine uses web workers with special asset handling:

*   **Worker Scripts**: Built separately and included in final distribution
*   **Dependencies**: External libraries like Dagre bundled in `worker_deps.js`
*   **Custom Element**: Different worker service implementation for standalone use

Sources: [src/ui/public/static_files/worker_deps.js 1-35](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/public/static_files/worker_deps.js#L1-L35)[src/ui/angular.json 137-141](https://github.com/tenstorrent/model-explorer/blob/70d3e3e4/src/ui/angular.json#L137-L141)

## TypeScript Configuration

Multiple TypeScript configurations support different build targets:

*   **`tsconfig.app.json`**: Main application compilation
*   **`tsconfig.worker.json`**: Web worker compilation
*   **`tsconfig.spec.json`**: Test file compilation

Each configuration extends base settings while targeting specific environments and module systems.

## Node.js Version Management

The project uses Volta for Node.js version management:

`{  "volta": {    "node": "22.14.0"  }}`
This ensures consistent Node.js versions across development environments.
