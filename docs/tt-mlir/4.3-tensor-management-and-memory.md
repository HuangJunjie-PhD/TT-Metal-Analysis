The Tensor Management and Memory subsystem provides facilities for creating, manipulating, and transferring tensor data between host and device memory. It abstracts the underlying runtime implementations (**TTNN** and **TTMetal**) while providing a unified API for tensor lifecycle management, data movement, and layout conversions.

## Overview

The tensor management system operates across three primary domains:

1.   **Tensor Creation**: Multiple creation modes including borrowed (external memory), owned (internal allocation), multi-device (distributed tensors), and empty (pre-allocated device buffers). [runtime/include/tt/runtime/runtime.h 55-102](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L55-L102)
2.   **Data Movement**: Host-to-device, device-to-host, and device-to-device transfers with synchronization and layout transformation support. [runtime/lib/ttnn/runtime.cpp 100-170](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L100-L170)
3.   **Memory Management**: Lifecycle control through reference counting, explicit deallocation, and retention policies. [runtime/include/tt/runtime/runtime.h 138-141](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L138-L141)

The system supports two runtime backends (TTNN and TTMetal) with a macro-based `DISPATCH_TO_CURRENT_RUNTIME` selecting the appropriate implementation based on the active `DeviceRuntime`. [runtime/lib/runtime.cpp 63-90](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/runtime.cpp#L63-L90)

## Tensor Creation and Allocation

Tensors are instantiated through various modes, supporting both local and distributed execution.

### Host Tensor Creation

*   **Borrowed**: `createBorrowedHostTensor` creates a tensor that wraps an existing external buffer. The caller is responsible for the buffer's lifecycle. [runtime/include/tt/runtime/runtime.h 55-59](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L55-L59)
*   **Owned**: `createOwnedHostTensor` allocates internal memory and copies the provided data into it. [runtime/lib/ttnn/runtime.cpp 43-96](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L43-L96)
*   **Multi-Device**: Distributed host tensors can be created from shards using `createMultiDeviceHostTensor`, which supports various sharding strategies across a mesh of devices. [runtime/lib/ttmetal/runtime.cpp 149-179](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L149-L179)
*   **Unsafe Borrowed**: `createUnsafeBorrowedHostTensor` creates a host tensor with borrowed storage that aliases an existing owned tensor without a copy. [runtime/include/tt/runtime/runtime.h 70-75](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L70-L75)

### Device Allocation

Device tensors are typically created as "Empty" buffers with a specific `Layout` and `MemoryConfig`.

*   **TTNN Path**: Uses `ttnn::Tensor` with device storage, created via `createEmptyTensor`. [runtime/include/tt/runtime/detail/ttnn/ttnn.h 134-136](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/detail/ttnn/ttnn.h#L134-L136)
*   **TTMetal Path**: Uses `MetalTensor` which is a variant wrapping `HostBuffer`, `DistributedHostBuffer`, or `MeshBuffer`. [runtime/lib/ttmetal/runtime.cpp 77-86](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L77-L86)

## Layout Conversion and Data Movement

The runtime manages transitions between host and device memory, as well as layout transformations (e.g., Row-Major to Tile).

### Layout Conversion Logic

The runtime provides explicit operations for layout and memory configuration changes:

*   **to_layout**: Converts a tensor to a specified layout (e.g., `ROW_MAJOR`). [runtime/lib/ttnn/program_executor.cpp 76](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L76-L76)
*   **to_memory_config**: Changes the memory configuration (e.g., `DRAM`, `L1`). [runtime/lib/ttnn/program_executor.cpp 77](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L77-L77)

**Layout Conversion Flow**

### Conversion Implementation

1.   **Tilize/Untilize**: Converting between `RowMajor` and `Tile` layouts. On-device tilization is supported for various data types including `BFloat16`, `Float32`, and `Int32`. [runtime/lib/ttnn/utils/utils.cpp 79-85](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/utils/utils.cpp#L79-L85)
2.   **Device Transfers**: `to_device` moves a tensor from host to a `MeshDevice`. `from_device` pulls it back to system memory. [runtime/lib/ttnn/runtime.cpp 100-170](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L100-L170)
3.   **Host-Side Casting**: If the runtime receives a data type not supported by the hardware, `createOwnedTTNNTensor` performs a buffer cast using `handleBufferCast` to a supported type before transfer. [runtime/lib/ttnn/runtime.cpp 43-73](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L43-L73)

## Memory Management and Synchronization

### Buffer Management

*   **Deallocation**: Explicit buffer deallocation on a device can be triggered via `deallocateBuffers`. [runtime/include/tt/runtime/runtime.h 177](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/runtime.h#L177-L177)
*   **Memory Reports**: The system can dump memory usage reports via `dumpMemoryReport`, which provides a `MemoryView` containing bank-level allocation details. [runtime/include/tt/runtime/types.h 197-206](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L197-L206)
*   **Memory Buffer Types**: Supported types include `DRAM`, `L1`, `L1_SMALL`, and `TRACE`. [runtime/include/tt/runtime/types.h 41-46](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L41-L46)

### Synchronization

Data transfers from device to host (`toHostSingleTensor`) can be `blocking` or non-blocking.

*   **Blocking**: The call waits for the device to finish the transfer before returning. [runtime/lib/ttnn/runtime.cpp 152-154](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L152-L154)
*   **Non-Blocking**: The runtime records a `MeshEvent` using `::ttnn::events::record_mesh_event` and returns a tensor handle associated with that event. [runtime/lib/ttnn/runtime.cpp 163-166](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L163-L166)

## Layout Converter Utility

The `LayoutConverter` (and the `TensorDesc` struct) is responsible for reconciling host-side tensor layouts with the requirements of the compiled program. It handles:

*   **Padding**: Adjusting tensor shapes to meet hardware alignment requirements (e.g., tile boundaries), as indicated by `physicalVolume` exceeding logical `volume`. [runtime/include/tt/runtime/types.h 192-194](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L192-L194)
*   **Striding**: Re-calculating strides for row-major data to match the expected memory footprint. [runtime/include/tt/runtime/types.h 176](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L176-L176)
