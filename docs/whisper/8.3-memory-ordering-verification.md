## Purpose and Scope

This document describes Whisper's memory consistency model (MCM) verification system, which validates that memory operations in a simulated RISC-V system adhere to the RISC-V Weak Memory Ordering (RVWMO) or Total Store Order (TSO) specifications. The MCM system checks Preserved Program Order (PPO) rules as defined in Chapter 17 of the RISC-V unprivileged specification.

For information about general memory subsystem architecture including physical memory and virtual memory translation, see [Physical Memory and Protection](https://deepwiki.com/tenstorrent/whisper/4.1-physical-memory-and-protection) and [Virtual Memory and Address Translation](https://deepwiki.com/tenstorrent/whisper/4.2-virtual-memory-and-address-translation). For details on the out-of-order memory simulation capabilities, see [Memory Consistency Model](https://deepwiki.com/tenstorrent/whisper/4.3-memory-consistency-model).

## Memory Operation Tracking

### MemoryOp Structure

The `MemoryOp` structure represents a single memory read or write operation associated with an instruction. Each operation is timestamped and tracked throughout the instruction lifecycle.

**Key Fields:**

*   `time_`: Timestamp when the operation occurred in global memory order
*   `pa_`: Physical address of the operation
*   `data_`: Whisper (reference model) data value
*   `rtlData_`: RTL (implementation) data value for comparison
*   `tag_`: Instruction tag associating operation with an instruction
*   `fwOffset_`: Per-byte forward time offsets for store-to-load forwarding
*   `canceled_`: Whether the operation was speculatively executed and later canceled

**Sources:**[Mcm.hpp 23-108](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L23-L108)[Mcm.cpp 165-189](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L165-L189)

### McmInstr Structure

The `McmInstr` structure represents a memory instruction (load, store, AMO) and aggregates all memory operations associated with it.

**Key Fields:**

*   `memOps_`: Indices into the global `sysMemOps_` vector for this instruction's operations
*   `addrTime_`/`dataTime_`: Time when address/data registers were produced (for dependency tracking)
*   `addrProducer_`/`dataProducer_`: Tags of instructions that produced address/data values
*   `retired_`: Whether the instruction has been retired
*   `complete_`: Whether all memory operations for this instruction have completed
*   `hasOverlap_`: For vector instructions, whether elements may have overlapping addresses

**Sources:**[Mcm.hpp 112-207](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L112-L207)[Mcm.cpp 266-309](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L266-L309)

## Preserved Program Order (PPO) Rules

The MCM system verifies 13 PPO rules plus I/O-specific rules defined by the RISC-V memory model. Each rule specifies a constraint on when memory operations can become globally visible.

### PPO Rule Configuration

Rules can be individually enabled or disabled:

**Sources:**[Mcm.cpp 26-31](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L26-L31)[Mcm.hpp 332-341](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L332-L341)

### Rule 1: Overlapping Memory Operations

**Rule:** Memory operations from two different instructions to overlapping addresses must complete in program order if they are in the same hart and at least one is a store.

**Implementation:** The system maintains an `undrainedStores_` set per hart. When an instruction B retires, all preceding undrained stores A that overlap with B are checked to ensure their memory operations complete before B's operations.

**Sources:**[Mcm.cpp 3586-3692](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L3586-L3692)[README.md 972-986](https://github.com/tenstorrent/whisper/blob/733fd921/README.md?plain=1#L972-L986)

### Rule 3: Fence.rw.rw Semantics

**Rule:** If a `fence.rw.rw` instruction appears between two memory operations in program order, those operations must complete in program order.

**Implementation:** When a store completes, the system checks if any fence instruction with `predecessor=w` and `successor=w` exists between it and any subsequent overlapping operations.

**Sources:**[Mcm.cpp 3765-3848](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L3765-L3848)

### Rule 5: Address Dependencies

**Rule:** A load B that depends on the address computed by a prior load/AMO A must respect the ordering constraint.

**Sources:**[Mcm.cpp 3870-3955](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L3870-L3955)

### Rule 6: Data Dependencies

**Rule:** A store B that depends on data from a prior load/AMO A must respect ordering if there's an overlapping store between them.

**Sources:**[Mcm.cpp 3958-4039](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L3958-L4039)

### Rule 11: Control Dependencies

**Rule:** Memory operations after a branch must not execute before the branch condition is resolved.

**Sources:**[Mcm.cpp 4272-4342](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L4272-L4342)

## Memory Models

### RVWMO vs TSO

Whisper supports two memory models:

| Model | Characteristic | Configuration |
| --- | --- | --- |
| **RVWMO** (RISC-V Weak Memory Ordering) | Allows most reorderings subject to PPO rules | Default |
| **TSO** (Total Store Order) | More restrictive, stores must drain in order | `enableTso(true)` |

**Configuration:**`Mcm::enableTso(bool flag)` in [Mcm.hpp 300-302](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L300-L302)

## Vector Memory Operations

Vector load/store instructions require special handling due to their element-based nature.

### Vector Instruction Tracking

**Per-Hart Data:** Each hart maintains a `vecRefMap_` mapping instruction tags to their vector element references.

**Overlap Detection:** Vector instructions set `hasOverlap_` if elements may overlap (e.g., non-unit-stride accesses), requiring conservative PPO checking.

**Sources:**[Mcm.cpp 1054-1131](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1054-L1131)[Mcm.cpp 463-541](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L463-L541)[Mcm.hpp 699-748](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L699-L748)

## Interactive Commands

The MCM system can be controlled via interactive mode or server API commands.

### Command Reference

| Command | Purpose | Format |
| --- | --- | --- |
| `mread` | Simulate out-of-order read | `time=T hart=H mread TAG ADDR SIZE DATA i|e` |
| `mbinsert` | Insert store into merge buffer | `time=T hart=H mbinsert TAG ADDR SIZE DATA` |
| `mbwrite` | Drain merge buffer line to memory | `time=T hart=H mbwrite ADDR DATA` |
| `mbbypass` | Bypass merge buffer, write directly | `time=T hart=H mbbypass TAG ADDR SIZE DATA` |

**Example Session:**

```
time=100 hart=0 mread 5 0x1000 4 0xdeadbeef i
time=200 hart=0 mbinsert 3 0x1000 4 0xcafebabe
time=300 hart=0 mbwrite 0x1000 cafebabe...
```

## Error Reporting

When PPO violations or data mismatches occur, the MCM system produces detailed error messages.

### Error Message Format

**PPO Rule Violation:**

```
Error: PPO rule X failed: hart-id=H tag1=A tag2=B addr=0xADDR
  instrA: time=T1 pa=0xPA1
  instrB: time=T2 pa=0xPA2
```

**Data Mismatch:**

```
Error: hart-id=H time=T tag=TAG addr=0xADDR size=S
  RTL data:  0xRTL_DATA
  Whisper:   0xREF_DATA
```
