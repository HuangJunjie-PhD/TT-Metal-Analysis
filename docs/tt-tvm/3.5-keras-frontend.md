## Purpose and Scope

The Keras Frontend provides conversion from Keras models to TVM's Relay intermediate representation (IR). It supports both the standalone Keras library and TensorFlow's integrated Keras API (`tf.keras`), enabling deployment of Keras-trained models on TVM's compilation and execution infrastructure.

This page covers the layer-by-layer conversion process, weight extraction, activation handling, and support for Sequential, Functional, and recurrent (RNN/LSTM) model architectures. For information about the Relay IR output format, see [Relay IR System](https://deepwiki.com/tenstorrent/tt-tvm/4-relay-ir-system). For other ML framework frontends, see [Frontend Converters](https://deepwiki.com/tenstorrent/tt-tvm/3-frontend-converters).

## System Architecture

The Keras frontend operates as a converter that transforms Keras model definitions into Relay expressions. It processes models layer-by-layer, extracting weights and converting operations to their Relay equivalents while maintaining computational semantics.

**Sources:**[python/tvm/relay/frontend/keras.py 1-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L1-L100)[python/tvm/relay/frontend/common.py 1-50](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/common.py#L1-L50)

## Entry Point and Main Conversion Flow

The `from_keras` function in [python/tvm/relay/frontend/keras.py 500-700](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L500-L700) serves as the primary interface. It accepts a Keras model and shape dictionary, then orchestrates the conversion process.

The function processes layers in topological order, maintaining an expression table to track intermediate results. Each layer's inputs are retrieved from the expression table, converted to Relay operations, and the output is stored back for subsequent layers.

**Sources:**[python/tvm/relay/frontend/keras.py 500-700](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L500-L700)[python/tvm/relay/frontend/common.py 100-200](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/common.py#L100-L200)

## RNN and LSTM Support

The Keras frontend provides specialized support for recurrent layers including SimpleRNN, LSTM, and GRU. These layers require different handling due to their stateful nature and sequential computation patterns.

### LSTM Implementation Details

The LSTM conversion at [python/tvm/relay/frontend/keras.py 400-600](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L400-L600) handles:

1.   **Weight Matrix Splitting**: Keras concatenates input/forget/cell/output gate weights; these must be separated
2.   **Cell State Management**: Maintains hidden state and cell state through time steps
3.   **Peephole Connections**: Optional peephole weights for direct cell state access to gates
4.   **Bidirectional Support**: Processes sequences in forward and backward directions

The recurrent activation (typically sigmoid for gates) is converted via `_convert_recurrent_activation` at [python/tvm/relay/frontend/keras.py 66-68](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L66-L68)

## Data Layout Considerations

Keras uses `channels_last` (NHWC) data layout by default, while many TVM optimizations target `channels_first` (NCHW). The frontend provides layout specification through the `layout` parameter.

### Layout Handling

| Keras Layout | TVM Layout | Conversion Required |
| --- | --- | --- |
| `channels_last` | NHWC | None (pass-through) |
| `channels_last` | NCHW | Transpose: (0,3,1,2) |
| Default | NHWC | None |

The `_check_data_format` function at [python/tvm/relay/frontend/keras.py 38-42](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L38-L42) validates that layers use `channels_last`:

## Model Type Support

The Keras frontend supports three primary model types:

1.   **Sequential Models**: Linear stack of layers processed in order
2.   **Functional Models**: DAG of layers with arbitrary connectivity
3.   **Subclassed Models**: Custom model classes (limited support)

### Functional Model Processing

Functional models define explicit input/output connections. The frontend traverses the model graph, respecting layer dependencies:

The conversion respects data dependencies, processing each layer only after its inputs are available in the expression table.

**Sources:**[python/tvm/relay/frontend/keras.py 500-700](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/frontend/keras.py#L500-L700)[tests/python/frontend/keras/test_forward.py 142-200](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/python/frontend/keras/test_forward.py#L142-L200)
