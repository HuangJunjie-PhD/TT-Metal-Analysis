## Purpose and Scope

This document provides a step-by-step guide for creating new test plans in the coretp framework. A test plan is a collection of related test scenarios organized around a specific RISC-V extension or functional area (e.g., cache management, virtual memory, supervisor extensions).

This page covers the test plan registration mechanism, package structure, and decorator usage. For details on writing individual test scenario functions, see [Writing Test Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/6.2-writing-test-scenarios). For architectural details about the registry system, see [Test Plan Registry System](https://deepwiki.com/tenstorrent/riscv-coretp/3.1-test-plan-registry-system).

## Step 1: Create Test Plan Package

Each test plan resides in its own package under `coretp/plans/`. The package structure follows this convention:

| File | Purpose |
| --- | --- |
| `__init__.py` | Defines the test plan decorator using `new_test_plan()` |
| `*_scenarios.py` | Contains decorated scenario functions |
| `README.md` (optional) | Documentation for the test plan |

### Example Package Structure

```
coretp/plans/
├── my_extension/
│   ├── __init__.py
│   └── my_extension_scenarios.py
```

The package name should match the RISC-V extension name (lowercase) or describe the functional area being tested.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/__init__.py 1-14](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/__init__.py#L1-L14)

## Step 2: Define the Test Plan Decorator

In `__init__.py`, import `new_test_plan` and create a decorator instance with appropriate metadata:

### Decorator Definition Pattern

The decorator is defined by calling `new_test_plan()` with the following parameters:

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `name` | `str` | Yes | Unique identifier for the test plan |
| `description` | `str` | No | Human-readable description |
| `tags` | `list[str]` | No | Searchable tags (e.g., `["cache", "cmo"]`) |
| `features` | `list[str]` | No | Feature dependencies (reserved for future use) |

### Scenario Function Structure

Each scenario function must:

1.   Be decorated with the test plan's decorator instance
2.   Return a `TestScenario` object

### Retrieval Process

The registry enforces a build-once pattern: once `get_plan()` is called, the `_TestPlanInfo` object builds the `TestPlan` by executing all registered scenario functions, then caches the result. Subsequent calls to `add_scenario()` raise a `RuntimeError` at [coretp/plans/test_plan_registry.py 67-68](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/test_plan_registry.py#L67-L68)

## Best Practices and Conventions

### Naming Conventions

| Element | Convention | Example |
| --- | --- | --- |
| Package name | Lowercase extension name | `zicbom_zicboz_zicbop/` |
| Test plan name | Same as package name | `"zicbom_zicboz_zicbop"` |
| Decorator variable | `<extension>_scenario` | `zicbom_zicboz_zicbop_scenario` |
| Scenario function | `SID_<EXT>_<NUM>[_<VARIANT>]` | `SID_XCOUNTEREN_01_U` |

### Metadata Guidelines

**Tags:** Use multiple specific tags for discoverability. Include:

*   Extension names (e.g., `"zicbom"`, `"zicboz"`)
*   Functional areas (e.g., `"cache"`, `"memory"`, `"supervisor"`)
*   Operation types (e.g., `"cmo"`, `"paging"`, `"csr"`)

**Description:** Provide a concise summary that appears in test plan listings and exports.

### Scenario Organization

Group related scenarios in the same file. For large test plans, consider multiple scenario files:

*   `basic_scenarios.py` - Core functionality tests
*   `edge_cases.py` - Boundary and error conditions
*   `multiprocessor_scenarios.py` - Multi-hart synchronization tests

Import all scenario modules in `__init__.py` to ensure registration on package import.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/__init__.py 1-14](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/__init__.py#L1-L14)[coretp/plans/zicntr_zihpm_sscounterenw/zicntr_zihpm_sscounterenw_scenarios.py 1-567](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicntr_zihpm_sscounterenw/zicntr_zihpm_sscounterenw_scenarios.py#L1-L567)

## Registration Lifecycle

### Complete Registration Flow

Key points:

1.   Registration occurs during module import via `new_test_plan()` at [coretp/plans/test_plan_registry.py 144-158](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/test_plan_registry.py#L144-L158)
2.   Scenario functions are added via decorator at [coretp/plans/test_plan_registry.py 160-163](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/test_plan_registry.py#L160-L163)
3.   Lazy building happens on first `get_plan()` call at [coretp/plans/test_plan_registry.py 71-79](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/test_plan_registry.py#L71-L79)
4.   Subsequent retrievals return the cached `TestPlan` object
