## Purpose and Scope

This document describes the build system architecture used to compile, test, and package the SFPI toolchain. The build system follows the traditional GNU Autotools pattern (Autoconf + Make) with custom orchestration scripts to coordinate the multi-stage GCC/Binutils toolchain build process.

For information about the release and packaging process, see [Packaging and Release](https://deepwiki.com/tenstorrent/sfpi/5.3-packaging-and-release). For CI/CD automation details, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/sfpi/5.4-cicd-pipeline). For testing infrastructure, see [Test Infrastructure Overview](https://deepwiki.com/tenstorrent/sfpi/6.1-test-infrastructure-overview).

* * *

## Build System Overview

The SFPI build system consists of three interconnected layers: configuration (Autoconf), orchestration (shell scripts), and execution (Make). The system builds a complete RISC-V cross-compilation toolchain with Tenstorrent SFPU extensions.

## Configuration Phase Architecture

The configuration phase uses GNU Autoconf to detect system capabilities and generate a customized Makefile from templates. This phase translates user-provided options into concrete build parameters.

### Autoconf Flow

**Key Configuration Options:**

| Option | Variable | Default | Purpose |
| --- | --- | --- | --- |
| `--with-arch` | `WITH_ARCH` | `rv32i` | Base RISC-V ISA |
| `--with-abi` | `WITH_ABI` | `ilp32` | Base RISC-V ABI |
| `--with-mfc` | `MFC` | `tt` | Manufacturer (target triple) |
| `--enable-multilib` | `multilib_flags` | enabled | Build multiple library variants |
| `--enable-gdb` | `enable_gdb` | `--disable-gdb` | Include GDB debugger |
| `--enable-gcc-checking` | `gcc_checking` | empty | GCC internal checks |

## Component Build Process

Each toolchain component follows a standard configure-make-install pattern. The Makefile encodes the precise dependencies and configuration flags for each stage.

### Binutils Build

**Key Configuration Flags:**

*   `--target=$(NEWLIB_TUPLE)` - Set to `riscv-tt-elf` (MFC=tt)
*   `--disable-gdb/--enable-gdb` - Based on `enable_gdb` setting
*   `--disable-werror` - Allow compilation with warnings
*   `$(IDENT_OPTIONS)` - Bug URL and package version strings

**Sources:**[Makefile.in 229-251](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L229-L251)[Makefile.in 58-64](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L58-L64)

### GCC Bootstrap (Stage 1)

GCC stage 1 is a minimal C-only compiler used to build the C library. It cannot link executables because no C runtime exists yet.

**Critical Flags:**

*   `--disable-shared` - Static linking only
*   `--disable-threads` - No thread support in stage 1
*   `--with-newlib` - Indicates we're building for newlib (even though it doesn't exist yet)
*   `--disable-libmudflap/libssp/libquadmath/libgomp` - Disable unnecessary libraries
*   `CFLAGS_FOR_TARGET="-Os"` - Optimize for size

**Sources:**[Makefile.in 274-308](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L274-L308)

### Newlib Build

Newlib is built in two configurations: full-featured and "nano" (minimal). Both are installed, allowing users to choose at link time via `-specs=nano.specs`.

**Full Newlib Options:**

*   `--enable-newlib-io-long-double` - Support for long double I/O
*   `--enable-newlib-io-c99-formats` - C99 printf formats
*   `CFLAGS_FOR_TARGET="-O2 -D_POSIX_MODE"`

**Nano Newlib Options:**

*   `--enable-newlib-reent-small` - Minimal reentrancy structure
*   `--enable-newlib-nano-malloc` - Tiny malloc implementation
*   `--enable-newlib-nano-formatted-io` - Minimal printf/scanf
*   `CFLAGS_FOR_TARGET="-Os"` - Aggressive size optimization

**Sources:**[Makefile.in 310-372](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L310-L372)

### GCC Final Build (Stage 2)

Stage 2 builds the complete GCC with full C++ support and target libraries, now that newlib provides a functional C runtime.

**Key Differences from Stage 1:**

*   `--enable-tls` - Thread-local storage support
*   `--with-sysroot` - Point to installed newlib headers/libraries
*   Full `make` (not just `make all-gcc`) - Builds libgcc, libstdc++, etc.
*   `CFLAGS_FOR_TARGET="-Os"` - Size-optimized target libraries

**Sources:**[Makefile.in 374-410](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L374-L410)

## Environment and Variables

The build system carefully controls the environment to ensure reproducible builds.

### Environment Sanitization

**Rationale:** Locale settings can affect string sorting, number formatting, and other behaviors. Setting `LC_ALL=C` ensures consistent behavior across different systems.

## Stamp File System

The build system uses "stamp files" to track build completion. This is a common pattern in GNU build systems to avoid rebuilding already-complete stages.

**Stamp File Locations:**

*   `stamps/build-binutils-newlib` - Binutils complete
*   `stamps/build-gcc-newlib-stage1` - GCC stage 1 complete
*   `stamps/build-newlib` - Full newlib complete
*   `stamps/build-newlib-nano` - Nano newlib complete
*   `stamps/merge-newlib-nano` - Nano libraries merged
*   `stamps/build-gcc-newlib-stage2` - Final GCC complete
*   `stamps/check-write-permission` - Installation directory verified

**Behavior:** Make checks the timestamp of the stamp file against dependencies. If dependencies are newer, the target is rebuilt and the stamp updated.

**Sources:**[Makefile.in 159-166](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L159-L166)[Makefile.in 235-410](https://github.com/tenstorrent/sfpi/blob/ed989b9d/Makefile.in#L235-L410)

## Multilib Support

Multilib allows the toolchain to build libraries for multiple architecture/ABI combinations, selectable via compiler flags.

**Multilib in configure.ac:**

*   Default: 7 configurations for newlib when multilib enabled
*   Generator string allows custom combinations
*   Single-lib builds use `--with-arch` and `--with-abi` values

## Version Management System

The `scripts/sfpi-info.sh` script provides a centralized version management system used by both build and release processes.

**sfpi-info.sh Modes:**

| Mode | Usage | Purpose |
| --- | --- | --- |
| `VERSION $ver` | Release scripts | Generate release variables from version string |
| `SHELL [$pkg]` | Shell scripts | Emit `eval`-able shell assignments |
| `CMAKE [$pkg]` | CMake integration | Emit CMake `set()` commands |
| `BUILD [$dir]` | Build automation | Clone and build from version |
| `CREATE [$dirs]` | Release creation | Generate `sfpi-version` file from artifacts |

**Generated Variables:**

*   `sfpi_version` - Version string (e.g., `1.0.0`)
*   `sfpi_arch` - Host architecture (`x86_64`, `aarch64`)
*   `sfpi_dist` - Distribution type (`debian`, `fedora`, `unknown`)
*   `sfpi_pkg` - Package format (`deb`, `rpm`, `txz`)
*   `sfpi_filename` - Full filename: `sfpi_${version}_${arch}_${dist}`
*   `sfpi_hashtype` - Hash algorithm (`sha256`)
*   `sfpi_hash` - Package checksum
*   `sfpi_url` - Download URL

**Sources:**[scripts/sfpi-info.sh 1-211](https://github.com/tenstorrent/sfpi/blob/ed989b9d/scripts/sfpi-info.sh#L1-L211)
