## Purpose and Scope

This document explains how RiescueD processes assembly test files internally. It covers:

*   Core class architecture and responsibilities
*   Data flow from parsing through generation
*   Component interactions and dependencies
*   Key data structures used throughout the framework

This page is intended for developers who need to understand or modify RiescueD's implementation, or who want to build tools that integrate with the framework.

## Architecture Overview

RiescueD implements a pipeline architecture with four distinct phases:

1.   **Parse Phase**: Extract directives from `.rasm` files into structured data
2.   **Configure Phase**: Consolidate configuration from multiple sources
3.   **Generate Phase**: Produce assembly files, linker scripts, and page tables
4.   **Build Phase**: Compile and disassemble using RISC-V toolchain
5.   **Simulate Phase** (optional): Execute on instruction set simulator

The framework separates concerns through specialized classes, each responsible for a specific aspect of test generation.

## Core Components

### RiescueD Class

The `RiescueD` class serves as the main orchestrator for the test generation pipeline. It coordinates all phases and maintains references to key objects.

**Key Responsibilities:**

*   Initialize the parsing infrastructure (`Parser`, `Pool`)
*   Coordinate configuration through `FeatMgrBuilder`
*   Invoke generation through `Generator`
*   Execute compilation and disassembly
*   Optionally run simulation

**Key Attributes:**

| Attribute | Type | Description |
| --- | --- | --- |
| `testfile` | `Path` | Input `.rasm` test file |
| `testname` | `str` | Test name derived from filename |
| `run_dir` | `Path` | Working directory for outputs |
| `cpuconfig` | `Path` | CPU configuration JSON file |
| `rng` | `RandNum` | Random number generator |
| `pool` | `Pool` | Repository of parsed data |
| `parsed_data` | `Parser` | Parser instance containing parsed structures |
| `toolchain` | `Toolchain` | Compiler and simulator tools |
| `generated_files` | `GeneratedFiles` | Output file tracker |

**Key Methods:**

| Method | Description |
| --- | --- |
| `__init__()` | Initialize parser, pool, and RNG |
| `configure()` | Build `FeatMgr` from sources |
| `generate()` | Generate assembly and linker scripts |
| `build()` | Compile to ELF and disassemble |
| `simulate()` | Run on ISS (Whisper or Spike) |
| `run()` | Execute complete pipeline |

**Sources:**[riescue/riescued.py 31-441](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L31-L441)

### Parser Class

The `Parser` class extracts RiescueD directives from assembly test files and populates data structures.

**Supported Directives:**

*   `;#test.<header>` - Test metadata (priv, env, paging, etc.)
*   `;#random_data()` - Random data generation requests
*   `;#random_addr()` - Random address allocation requests
*   `;#page_mapping()` - Page table mapping specifications
*   `;#page_map()` - Page map declarations
*   `;#reserve_memory()` - Memory reservation requests
*   `;#discrete_test()` - Multi-test markers
*   `;#init_memory` - Memory initialization markers
*   `;#vectored_interrupt()` - Interrupt vector table entries
*   `;#csr_rw()` - CSR access instrumentation
*   `;#read_leaf_pte()` - PTE read instrumentation

**Key Methods:**

| Method | Description |
| --- | --- |
| `parse()` | Main entry point: parse entire file |
| `parse_test_header()` | Extract test header fields |
| `parse_random_data()` | Parse random data directives |
| `parse_random_addr()` | Parse random address directives |
| `parse_page_mappings()` | Parse page mapping directives |
| `parse_page_maps()` | Parse page map declarations |

*   Discrete test markers
*   Section names

### Generator Class

The `Generator` class produces final assembly code, linker scripts, and auxiliary files based on parsed data and configuration.

**Key Responsibilities:**

*   Resolve random addresses with constraints
*   Generate page table structures
*   Create memory layout and linker scripts
*   Generate OS boot code
*   Produce final assembly file

**Generation Products:**

| File | Description |
| --- | --- |
| `<test>.S` | Final assembly with resolved addresses and generated code |
| `<test>.ld` | Linker script with memory regions |
| `<test>_pt.h` | Page table definitions (C header) |
| `<test>.elf` | Compiled executable (after build) |
| `<test>.dis` | Disassembly (after build) |

**Address Resolution:**

The `Generator` resolves random addresses by:

1.   Querying `Pool` for `ParsedRandomAddress` entries
2.   Applying constraints (alignment, masks, size)
3.   Allocating from available memory regions
4.   Checking PMA (Physical Memory Attribute) requirements
5.   Recording allocations to prevent conflicts

**Page Table Generation:**

For tests with paging enabled, `Generator`:

1.   Reads `ParsedPageMapping` entries from `Pool`
2.   Determines page table hierarchy (Sv39/Sv48/Sv57)
3.   Generates multi-level page tables
4.   Applies PTE bit configurations (V, R, W, X, U, A, D, G)
5.   Outputs page table data structures

### ParsedPageMapping

Represents a `;#page_mapping()` directive with extensive configuration options.

Key fields include:

*   `lin_name`, `phys_name`: Address references
*   `v`, `r`, `w`, `x`, `u`, `a`, `d`, `g`: PTE permission bits
*   `v_leaf`, `v_nonleaf`: Validity bits for different PTE levels
*   `pagesizes`: Allowed page sizes (4KB, 2MB, 1GB, etc.)
*   `page_maps`: Associated page map names
*   `secure`: Security flag for secure memory

### WYSIWYG Mode Handling

In "What You See Is What You Get" (WYSIWYG) mode, RiescueD uses a special exit mechanism:

1.   Test writes `0xc001c0de` to `x31` register to signal pass
2.   `simulate()` parses disassembly to find `<failed>` label PC
3.   Passes `--endpc` to ISS to stop at failure address
4.   Parses ISS log to verify `x31` contains correct exit code

This allows tests to control their own pass/fail criteria without relying on tohost mechanisms.

**Sources:**[riescue/riescued.py 326-408](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L326-L408)

## Summary

RiescueD implements a clean separation of concerns through specialized classes:

*   **RiescueD**: Pipeline orchestration
*   **Parser**: Directive extraction
*   **Pool**: Data storage and access
*   **FeatMgrBuilder/FeatMgr**: Configuration management
*   **Generator**: Code generation
*   **Toolchain**: External tool integration

Data flows unidirectionally through the pipeline: parse → configure → generate → build → simulate. Each phase produces well-defined outputs consumed by the next phase, enabling modular testing and flexible customization.
