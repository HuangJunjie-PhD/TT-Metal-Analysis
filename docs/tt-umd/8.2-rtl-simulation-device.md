## Purpose and Scope

This page documents the RTL simulation infrastructure in `tt-umd`, specifically the `RtlSimulationTTDevice`, `RtlSimCommunicator`, and the FlatBuffer-based communication protocol used to interface with external RTL simulators. These components provide a hardware abstraction layer for Tenstorrent architectures (Wormhole, Blackhole, and Quasar/Grendel) within a cycle-accurate RTL simulation environment.

The RTL simulation device differs from the standard `TTSimTTDevice` (which uses a shared library `.so`) by spawning an external simulator process and communicating over a network socket using the NNG (nanomsg-next-generation) library.

## RtlSimulationTTDevice

`RtlSimulationTTDevice` is the primary implementation of the `TTDevice` interface for RTL simulation. It manages the lifecycle of the simulator and handles coordinate translation, TLB allocation, and memory management.

### Key Responsibilities

*   **Initialization**: Spawns the simulator via `RtlSimCommunicator::initialize()` and sets up system memory callbacks [device/tt_device/rtl_simulation_tt_device.cpp 79-105](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L79-L105)
*   **TLB Management**: Uses `SimulationTlbAllocator` to manage virtual TLB windows. For Quasar (Grendel), it uses a dummy 4GB window since the architecture bypasses traditional TLBs [device/tt_device/rtl_simulation_tt_device.cpp 107-115](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L107-L115)
*   **Reset Sequences**: Implements architecture-specific reset sequences, including Tensix and NEO DM resets [device/tt_device/rtl_simulation_tt_device.cpp 197-235](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L197-L235)

### Architecture Support: Quasar (Grendel)

The RTL simulation device provides specific support for the Quasar architecture:

*   **System NOC**: Supported only on Quasar [device/tt_device/rtl_simulation_tt_device.cpp 48-50](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L48-L50)
*   **NOC1**: Explicitly unsupported on Quasar [device/tt_device/rtl_simulation_tt_device.cpp 51-53](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L51-L53)
*   **NEO DM Uncore Reset**: Quasar utilizes "Uncore" reset sequences for NEO DM cores, handled via `neo_dm_uncore_reset_assert/deassert`[device/tt_device/rtl_simulation_tt_device.cpp 218-235](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L218-L235)

## RtlSimCommunicator & FlatBuffer Protocol

`RtlSimCommunicator` handles the low-level serialization of device commands into FlatBuffer messages and manages the asynchronous response handling.

### FlatBuffer Schema: simulation_device.fbs

The communication protocol uses the `DeviceRequestResponse` table to wrap commands and data.

| Command Tag | Enum Value | Description |
| --- | --- | --- |
| `WRITE` | 0 | Write data to a specific core and address |
| `READ` | 1 | Read data from a specific core and address |
| `START` | 4 | Signal simulator to begin execution |
| `EXIT` | 5 | Shutdown signal for simulator |
| `NEO_DM_RESET_ASSERT` | 8 | Assert reset for a specific NEO DM index |
| `AXI_RAM_WRITE_NOTIFICATION` | 10 | Simulator-initiated write to host memory |
| `AXI_RAM_READ_NOTIFICATION` | 11 | Simulator-initiated read from host memory |

**Sources:**[device/simulation/simulation_device.fbs 3-26](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_device.fbs#L3-L26)[device/simulation/simulation_device.fbs 33-39](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_device.fbs#L33-L39)

### Asynchronous Notification Handling

The communicator runs a `notification_handler_thread` to process messages arriving from the simulator. This is critical for handling **AXI RAM Notifications**, where the simulated device initiates a DMA transfer to or from the host system memory.

**Notification Flow:**

1.   Simulator sends an `AXI_RAM_READ_NOTIFICATION` message.
2.   `notification_handler_thread` receives the message [device/simulation/rtl_sim_communicator.cpp 220-224](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L220-L224)
3.   It identifies the message as a notification and calls the registered `RamReadCallback`[device/simulation/rtl_sim_communicator.cpp 250-255](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L250-L255)
4.   The result is sent back to the simulator via `host_.send_to_device()` so the simulation can proceed [device/simulation/rtl_sim_communicator.cpp 257](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L257-L257)
5.   If the message is a response to a UMD-initiated `READ`, it is pushed to the `command_queue_` for the calling thread to consume [device/simulation/rtl_sim_communicator.cpp 233-238](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L233-L238)

**Sources:**[device/api/umd/device/simulation/rtl_sim_communicator.hpp 203-210](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/simulation/rtl_sim_communicator.hpp#L203-L210)[device/simulation/rtl_sim_communicator.cpp 219-265](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L219-L265)

## SimulationHost

The `SimulationHost` class provides the network transport layer using NNG's `pair1` protocol.

**Network Initialization Sequence:**

1.   **Port Selection**: Checks `NNG_SOCKET_LOCAL_PORT` or selects a random free port between 50000-59999 [device/simulation/simulation_host.cpp 61-73](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_host.cpp#L61-L73)
2.   **Address Export**: Sets the `NNG_SOCKET_ADDR` environment variable (e.g., `tcp://localhost:55555`) so the spawned simulator process knows where to connect [device/simulation/simulation_host.cpp 81-84](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/simulation_host.cpp#L81-L84)
3.   **Connection**: Spawns `run.sh` from the simulator directory [device/simulation/rtl_sim_communicator.cpp 94-118](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L94-L118) and waits for a handshake `EXIT` command (used as an ACK) [device/simulation/rtl_sim_communicator.cpp 127-133](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/simulation/rtl_sim_communicator.cpp#L127-L133)

## Reset and Staggered Start

The RTL simulation device supports complex reset sequences required by modern Tenstorrent architectures.

### NEO DM Reset

The simulator can reset individual NEO DM cores (0-7) within a Tensix core. This is mapped through the `deassert_risc_reset` function.

`// Mapping RiscType to NEO DM indicesstatic constexpr std::array<RiscType, 8> RISC_TYPES_DMS = {    RiscType::DM0, RiscType::DM1, RiscType::DM2, RiscType::DM3,    RiscType::DM4, RiscType::DM5, RiscType::DM6, RiscType::DM7};`
**Sources:**[device/tt_device/rtl_simulation_tt_device.cpp 34-42](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L34-L42)

### Staggered Start

When `staggered_start` is requested in `deassert_risc_reset`, the driver iterates through the requested RISC types and issues individual reset deassertion commands to the simulator [device/tt_device/rtl_simulation_tt_device.cpp 211-216](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L211-L216) If not staggered, it issues an "all" deassert command for the core [device/tt_device/rtl_simulation_tt_device.cpp 208](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L208-L208)

## System Memory Callbacks

To support DMA operations where the simulated device accesses host memory, `RtlSimulationTTDevice` registers callbacks with the `RtlSimCommunicator`. These callbacks interface with the `SimulationSysmemManager`.

**Callback Implementation:**

*   **Write Callback**: When the simulator writes to host RAM, the callback calculates the correct host memory channel and offset, then performs the write to the UMD-managed sysmem buffer [device/tt_device/rtl_simulation_tt_device.cpp 82-92](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L82-L92)
*   **Read Callback**: When the simulator requests data from host RAM, the callback reads from the UMD sysmem buffer and provides the data back to the simulator [device/tt_device/rtl_simulation_tt_device.cpp 94-102](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L94-L102)
