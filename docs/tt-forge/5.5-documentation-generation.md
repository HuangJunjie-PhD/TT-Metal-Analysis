## Purpose and Scope

This document explains the automated documentation generation system that produces comprehensive release documentation and static project documentation for TT-Forge. The system combines changelogs, hardware compatibility tables, and installation instructions into unified release documents, while also maintaining a web-based manual using `mdBook`.

For information about the overall release lifecycle that triggers documentation generation, see [Release Lifecycle and Versioning](https://deepwiki.com/tenstorrent/tt-forge/5.1-release-lifecycle-and-versioning). For details on the build artifacts included in releases, see [Python Wheel Building](https://deepwiki.com/tenstorrent/tt-forge/5.4.1-python-wheel-building) and [Artifact Management](https://deepwiki.com/tenstorrent/tt-forge/5.4.2-artifact-management).

## Documentation Generation Overview

The documentation generation system is split into two primary paths: release-specific documentation (changelogs, installation guides) and static project documentation (guides, summaries).

Release documentation is implemented as a composite GitHub Action (`docs-generator`) that orchestrates multiple components into a single cohesive release document. Static documentation is managed via `mdBook` and deployed to GitHub Pages.

## Release Documentation Components

### Tag Resolution and Version Comparison

The `docs-generator` action determines the comparison baseline for changelog generation based on the `release_type` and `new_version_tag`.

The logic uses `gh release list` with JSON queries to find appropriate comparison points [[.github/actions/docs-generator/action.yml:66-126]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation):

*   **Stable Releases**: Compares against the previous stable release. If none exist, it falls back to the latest nightly.
*   **RC Releases**: Initial `rc1` compares against the previous stable; subsequent `rc2+` compare against the previous RC.
*   **Nightly Releases**: Always compare against the previous nightly (`.dev` tag).

### Changelog Generation

Changelogs are generated using the `mikepenz/release-changelog-builder-action@v5`. It categorizes commits based on labels extracted via a regex pattern following the Conventional Commits format [[.github/actions/docs-generator/action.yml:146-212]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

| Category | Labels | Purpose |
| --- | --- | --- |
| 🚀 Features | feature, enhancement | New functionality |
| 🛠 Fixes | fix, bug | Bug fixes |
| 🧹 Maintenance | chore, refactor, cleanup | Code maintenance |
| ⚡️ Performance | perf | Performance optimizations |
| 🧪 Tests | test | Test modifications |

### Hardware Compatibility Table

The compatibility table is generated by a script that processes JUnit XML test results. It extracts test names and statuses (passed/failed) to create a markdown matrix of model support across hardware platforms [[.github/actions/docs-generator/action.yml:128-144]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation). After processing, the script deletes the XML files to prevent artifact bloat.

### Installation Instructions

Installation instructions are dynamically generated based on the `pip_wheel_names` and `new_version_tag` inputs [[.github/actions/docs-generator/action.yml:214-267]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

*   **Pip**: Generates `pip install {wheel}=={version} --extra-index-url https://pypi.eng.aws.tenstorrent.com/`.
*   **Docker**: Generates `docker pull ghcr.io/tenstorrent/{repo}-slim:{tag}` unless `skip_docker_build` is true.

**Sources**: [.github/actions/docs-generator/action.yml 4-61](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/docs-generator/action.yml#L4-L61)[.github/actions/docs-generator/action.yml 214-267](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/docs-generator/action.yml#L214-L267)

## Static Documentation (mdBook)

The project maintains a structured manual in the `docs/` directory, managed by `mdBook`.

### Structure and Content

The documentation structure is defined in `docs/src/SUMMARY.md`[[docs/src/SUMMARY.md:1-5]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation):

*   **Getting Started**: Initial setup and environment configuration.
*   **Model Bring-Up Guide**: Technical walkthrough for porting models to Tenstorrent hardware [[docs/src/model-bring-up-guide.md:1-23]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

### Deployment Workflow

The `pages.yml` workflow automates the deployment of the documentation to GitHub Pages on every push to the `main` branch [[.github/workflows/pages.yml:4-5]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

1.   **Setup**: Configures GitHub Pages and installs `mdBook` version `0.4.47`[[.github/workflows/pages.yml:24-32]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).
2.   **Build**: Runs `mdbook build docs`, which outputs the static site to `docs/book`[[.github/workflows/pages.yml:36-41]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).
3.   **Deploy**: Uploads the `docs/book` artifact and deploys it to the `github-pages` environment [[.github/workflows/pages.yml:43-53]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

## Submodule Documentation Integration

The repository tracks `tt-forge-models` as a submodule [[.github/workflows/schedule-uplift.yml:20]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation). A weekly "Uplift" workflow automates the synchronization of this submodule and generates documentation regarding the changes.

### Uplift Documentation Flow

When the `Weekly Forge Models Uplift` workflow runs, it fetches the commit history between the current submodule SHA and the latest available SHA in the upstream repository [[.github/workflows/schedule-uplift.yml:33-38]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).

The workflow formats the commit list using the pattern `%cd (%h) by %cn (Author %ae) : %s` and includes it in the Pull Request body and comments to provide context for the submodule version jump [[.github/workflows/schedule-uplift.yml:54-89]](https://deepwiki.com/tenstorrent/tt-forge/5.5-documentation-generation).
