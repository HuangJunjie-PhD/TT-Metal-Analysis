## Purpose and Scope

This page describes the posted write optimization technique for NOC data movement operations. Posted writes eliminate acknowledgment traffic to achieve performance gains up to 75% in NOC-bandwidth-saturating workloads, at the cost of requiring more careful synchronization to avoid data race conditions. This page covers the performance benefits, API usage patterns, synchronization requirements, and critical safety considerations for using posted writes.

For general NOC API usage patterns and best practices, see [Core API Concepts](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.1-core-api-concepts). For understanding how to batch operations and use barriers effectively, see [Asynchronous Operations and Batching](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.2-asynchronous-operations-and-batching).

## Transaction Semantics: Posted vs Non-Posted Writes

The NOC supports two write transaction modes that differ fundamentally in their completion semantics and network traffic patterns.

### Transaction Flow Comparison

Sources: [data_movement_doc/general/posted_writes.md 9-21](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L9-L21)

### Semantic Differences

| Aspect | Non-Posted Write | Posted Write |
| --- | --- | --- |
| **Command Flag** | `NOC_CMD_RESP_MARKED` set | `NOC_CMD_RESP_MARKED` not set |
| **ACK Traffic** | Required for every write | None |
| **Completion Guarantee** | Data written to destination | Data left source core |
| **Latency** | Round-trip (write + ACK) | One-way (write only) |
| **Safe for Reuse** | Source buffer after ACK | Source buffer immediately |
| **Safe for Read** | Destination after ACK | No guarantee |
| **Default Behavior** | Yes (all NOC writes) | No (opt-in via template parameter) |

Sources: [data_movement_doc/general/posted_writes.md 9-21](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L9-L21)

## Performance Impact Analysis

Posted writes eliminate ACK traffic congestion and round-trip latency penalties. The performance benefit is most significant when NOC bandwidth is saturated with write operations, as ACK messages compete for the same physical links.

### Measured Performance Gains by Operation Class

#### NOC-Bandwidth-Saturating Operations

Operations that saturate NOC bandwidth see the largest gains because ACK traffic creates significant congestion:

| Operation | Performance Gain | Explanation |
| --- | --- | --- |
| Interleaved to sharded | Up to 50% | High NOC write volume, ACKs congest return path |
| Reshard (saturated) | Up to 60% | Extremely high write traffic, ACK bottleneck severe |
| Untilize | Up to 75% | Maximum write saturation, eliminating ACKs doubles effective bandwidth |
| Tilize | ~2% | Not NOC-limited, compute-bound operation |
| Transpose | ~2% | Not NOC-limited, compute-bound operation |
| Concat | ~5% | Low NOC pressure, minimal ACK congestion |

Sources: [data_movement_doc/general/posted_writes.md 38-48](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L38-L48)

#### Data Movement Primitives

Transfer size determines whether operations are NOC-saturating:

| Primitive | Small Transfers | Large Transfers |
| --- | --- | --- |
| **One-to-one** | Significant gains (10-30%) | Minimal improvement (2-5%) |
| **One-to-all** | Significant gains (10-30%) | Minimal improvement (2-5%) |

**Explanation**: Small transactions have proportionally higher ACK overhead. Large transactions that already saturate the NOC see less relative benefit.

## API Implementation

### Template Parameter Control

Most NOC write functions support posted writes via a template parameter. The default is always non-posted for safety.

The final template parameter controls posted vs non-posted behavior:

*   `posted = true`: Skips `NOC_CMD_RESP_MARKED` flag, no ACK required
*   `posted = false` (default): Sets `NOC_CMD_RESP_MARKED`, requires ACK

Sources: [data_movement_doc/general/posted_writes.md 56-66](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L56-L66)

## Synchronization Mechanisms

Posted writes require different synchronization primitives because there are no ACK messages to wait for. The NOC hardware maintains separate counters for tracking posted and non-posted write progress.

### Counter Types and Semantics

Sources: [data_movement_doc/general/posted_writes.md 83-88](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L83-L88)

### Barrier Function Selection

| Function | Use Case | What It Waits For | Posted/Non-Posted |
| --- | --- | --- | --- |
| `noc_async_write_barrier()` | Non-posted writes, need completion guarantee | `NONPOSTED_WRITES_ACKED == ISSUED` | Non-posted only |
| `noc_async_writes_flushed()` | Non-posted writes, only need departure | Write sent (not ACKed) | Non-posted only |
| `noc_async_posted_writes_flushed()` | Posted writes | `POSTED_WRITES_NUM_ISSUED == SENT` | Posted only |

**Key Distinction**:

*   For **non-posted** writes: Can choose between waiting for ACK (`barrier`) or just send (`flushed`)
*   For **posted** writes: Only `posted_writes_flushed` is valid, waits for send only

## Data Race Hazards and Mitigation

**Critical Warning**: Posted writes provide **no guarantee** about when data arrives at the destination. This creates data race hazards in multi-core synchronization scenarios.

### Classic Data Race Scenario

Sources: [data_movement_doc/general/posted_writes.md 94-113](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L94-L113)

### Root Cause Analysis

The race condition occurs because:

1.   **Posted write completes** when data leaves Core A (not when it arrives at Core B)
2.   **Semaphore/signal** can travel faster than data (different NOC priority, routing)
3.   **Core C receives signal** before data has been committed to Core B's memory
4.   **Core C reads garbage** because the data hasn't arrived yet

This is a **timing-dependent** bug that may not appear in testing with low NOC congestion but manifests under production workloads.

## Decision Framework

### When to Use Posted Writes

Sources: [data_movement_doc/general/posted_writes.md 23-28](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L23-L28)[data_movement_doc/general/posted_writes.md 115-121](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L115-L121)

### Best Practices Summary

| Practice | Description | Rationale |
| --- | --- | --- |
| **Profile first** | Measure actual performance gain in your workload | Posted writes add complexity; ensure benefit justifies risk |
| **Use for bulk operations** | Apply to high-volume, NOC-saturating writes | Maximum benefit where ACK traffic causes congestion |
| **Non-posted for sync** | Use non-posted writes for synchronization/signaling | Guarantees ordering for cross-core dependencies |
| **Hybrid approach** | Posted for data, non-posted for completion signal | Combines performance with safety |
| **Explicit barriers** | Always call `noc_async_posted_writes_flushed()` | Ensure posted writes departed before proceeding |
| **Test under load** | Validate with congested NOC conditions | Data races are timing-dependent, may not appear in light testing |

Sources: [data_movement_doc/general/posted_writes.md 115-121](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/posted_writes.md?plain=1#L115-L121)

### Operations Where Posted Writes Are Beneficial

**High Benefit (>30% gain):**

*   Interleaved to sharded conversion
*   Reshard operations (NOC-saturated)
*   Untilize operations
*   High-volume one-to-one writes (small transfers)
*   High-volume one-to-all writes (small transfers)

**Low Benefit (<5% gain):**

*   Compute-bound operations (tilize, transpose, concat)
*   Large transfers that already saturate NOC
*   Low write volume operations
