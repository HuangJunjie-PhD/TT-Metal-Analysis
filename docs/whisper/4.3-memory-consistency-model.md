# Memory Consistency Model

Relevant source files
*   [Mcm.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp)

## Purpose and Scope

The Memory Consistency Model (MCM) subsystem in Whisper enables simulation and verification of out-of-order memory operations in RISC-V systems. It implements the RISC-V Weak Memory Ordering (RVWMO) specification and optionally Total Store Order (TSO), tracking memory operations across time, validating Preserved Program Order (PPO) rules, and simulating store-to-load forwarding through merge buffers.

This page documents the MCM implementation, including memory operation tracking, merge buffer simulation, PPO rule enforcement, and lock-step verification with RTL implementations. For virtual memory address translation, see [4.2](https://deepwiki.com/tenstorrent/whisper/4.2-virtual-memory-and-address-translation). For physical memory protection and attributes, see [4.1](https://deepwiki.com/tenstorrent/whisper/4.1-physical-memory-and-protection). For performance modeling APIs that interact with the MCM, see [6.3](https://deepwiki.com/tenstorrent/whisper/6.3-performance-modeling-api).

## Core Concepts

### Memory Operations

Each memory access (read or write) is represented by a `MemoryOp` structure that tracks timing, physical addresses, data values (both RTL and reference model), and state information.

**Key Fields of MemoryOp:**

| Field | Type | Purpose |
| --- | --- | --- |
| `time_` | `uint64_t` | Global memory time when operation executes |
| `pa_` | `uint64_t` | Physical address |
| `data_` | `uint64_t` | Reference (Whisper) data value |
| `rtlData_` | `uint64_t` | RTL data value (for verification) |
| `tag_` | `McmInstrIx` | Instruction tag this operation belongs to |
| `size_` | `uint8_t` | Operation size in bytes (1-8) |
| `isRead_` | `bool` | True for loads, false for stores |
| `bypass_` | `bool` | True if operation bypasses merge buffer |
| `fwOffset_` | `array<uint16_t, 8>` | Per-byte forwarding time offset |
| `elemIx_` | `uint16_t` | Vector element index |
| `canceled_` | `bool` | True if operation was canceled/speculative |

**Sources:**[Mcm.hpp 23-108](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L23-L108)

### MCM Instructions

The `McmInstr` structure represents an instruction in the MCM system, tracking its memory operations, dependencies, and completion state.

### Merge Buffer Simulation

The MCM simulates a merge/store buffer where stores are held before draining to memory. This enables:

*   **Store coalescing**: Multiple stores to the same cache line
*   **Out-of-order draining**: Stores can drain out of program order
*   **Forwarding**: Pending stores can forward to subsequent loads

**Diagram: Merge Buffer Operation Flow**

**Sources:**[Mcm.cpp 930-969](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L930-L969)[Mcm.cpp 1532-1662](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1532-L1662)

### Store-to-Load Forwarding

When a load instruction reads from an address that has pending (undrained) stores, the MCM forwards data from the most recent overlapping store in program order.

**Forwarding Rules:**

1.   Only undrained stores (in merge buffer) can forward
2.   Only stores preceding the load in program order can forward
3.   Most recent overlapping store forwards to each byte
4.   Forwarded bytes track time offset for ordering checks

## Memory Ordering Models

### RVWMO (RISC-V Weak Memory Ordering)

Whisper's MCM implements the RISC-V weak memory ordering model, which allows extensive reordering of memory operations for performance, subject to specific ordering constraints defined by the PPO rules.

**Key RVWMO Properties:**

*   Loads may execute out of program order
*   Stores may execute out of program order
*   Stores may drain to memory out of program order
*   Dependencies and fences create ordering constraints

**Sources:**[Mcm.hpp 215-217](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L215-L217)

### TSO (Total Store Order) Mode

TSO is a stricter ordering model where stores appear to drain to memory in program order. This is enabled via `enableTso()`.

`// Enable TSO modemcm.enableTso(true);`
**Sources:**[Mcm.hpp 300-302](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L300-L302)

### PPO (Preserved Program Order) Rules

The MCM enforces 13 PPO rules plus I/O ordering rules as defined in the RISC-V memory model specification. These rules define when memory operations must appear to execute in program order.

**PPO Rules Overview:**

| Rule | Description | Validation |
| --- | --- | --- |
| R1 | Overlapping store-store must drain in order | `ppoRule1()` |
| R2 | Load-load with address dependency | `ppoRule2()` |
| R3 | Load-store with address/data dependency | `ppoRule3()` |
| R4 | Store/load before fence.w before store/AMO | `ppoRule4()` |
| R5 | Load/store before fence.r before load | `ppoRule5()` |
| R6 | AMO/SC operations ordering | `ppoRule6()` |
| R7 | Load-load with control dependency | `ppoRule7()` |
| R8 | Store-store with control dependency | `ppoRule8()` |
| R9 | Load-load/store within same instr | `ppoRule9()` |
| R10 | AMO address dependency | `ppoRule10()` |
| R11 | Load with dependency before store | `ppoRule11()` |
| R12 | Load-load via dependency chain | `ppoRule12()` |
| R13 | Store-load/store via dependency chain | `ppoRule13()` |

**Sources:**[Mcm.hpp 215-216](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L215-L216)[Mcm.cpp 1330-1380](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1330-L1380)

## MCM Operation Flow

### Read Operation (readOp)

When the RTL issues a read operation, the MCM processes it to enable out-of-order execution and forwarding.

**Diagram: Read Operation Flow**

**Sources:**[Mcm.cpp 117-261](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L117-L261)

### Merge Buffer Insert

Store operations are inserted into the merge buffer (per-hart pending writes) when the store instruction executes but before draining to memory.

`bool mergeBufferInsert(Hart<URV>& hart, uint64_t time, uint64_t tag,                        uint64_t pa, unsigned size, uint64_t rtlData,                       unsigned elemIx, unsigned field);`
**Insert Operation Steps:**

1.   Create `MemoryOp` with write data
2.   Add to `hartData_.pendingWrites_`
3.   Mark instruction tag in `undrainedStores_`
4.   Store remains in buffer until `mergeBufferWrite()` drains it

**Sources:**[Mcm.cpp 930-969](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L930-L969)

### Merge Buffer Write (Drain)

The merge buffer write operation drains pending stores to memory, validating that RTL data matches reference data.

**Diagram: Merge Buffer Write Flow**

**Sources:**[Mcm.cpp 1532-1662](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1532-L1662)

### Bypass Operation

Some operations bypass the merge buffer and write directly to memory (e.g., AMOs, cache management operations with cache parameter set).

`bool bypassOp(Hart<URV>& hart, uint64_t time, uint64_t tag, uint64_t pa,              unsigned size, uint64_t rtlData, unsigned elemIx,               unsigned field, bool cache);`
**Bypass vs Insert:**

*   **Bypass**: Writes immediately to memory, checks immediately if retired
*   **Insert**: Writes to merge buffer, drains later with `mergeBufferWrite()`
*   Cache parameter determines data vs instruction cache

**Sources:**[Mcm.cpp 973-1049](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L973-L1049)

### Instruction Retirement

When an instruction retires, the MCM validates its memory operations and checks all applicable PPO rules.

**Retire Steps:**

1.   Cancel non-retired instructions before this tag
2.   Mark instruction as retired
3.   For loads: Commit read ops (validate, trim, forward)
4.   For stores: Capture store data and addresses
5.   Set producer times for dependencies
6.   Check all enabled PPO rules (R1-R13)
7.   Update register dependency times

**Sources:**[Mcm.cpp 1236-1380](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1236-L1380)

## Data Dependency Tracking

The MCM tracks data and address dependencies between instructions to validate PPO rules that involve dependencies.

### Register Dependency Times

Each hart maintains per-register timing information:

*   `regTime_`: Time when register value was produced
*   `regProducer_`: Tag of instruction that produced value

**Register Mapping:**

| Register Type | Offset | Range |
| --- | --- | --- |
| Integer (x0-x31) | 0 | 0-31 |
| FP (f0-f31) | 32 | 32-63 |
| Vector (v0-v31) | 64 | 64-95 |
| CSR | 96 | 96-4191 |

**Sources:**[Mcm.hpp 915-919](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L915-L919)[Mcm.cpp 546-663](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L546-L663)

### Dependency Propagation

Dependencies propagate through the register file during `updateDependencies()`:

**Key Dependency Rules:**

*   Loads/AMOs break dependency chains (dest time = load time)
*   Other instructions propagate maximum source time to dest
*   CSR operations have separate timing
*   Vector operations track per-register group

**Sources:**[Mcm.cpp 546-663](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L546-L663)

## Vector Memory Operations

Vector load/store instructions require special handling due to multiple element accesses, masking, and segmented/indexed addressing.

### Vector Reference Tracking

The MCM tracks reference (Whisper) addresses and data for each active vector element in `VecRefs`:

`struct VecRef {    uint64_t pa_;      // Physical address    uint64_t data_;    // Reference value    uint8_t ix_;       // Element index    uint8_t size_;     // Bytes covered    uint8_t reg_;      // Data register number    uint8_t ixReg_;    // Index register (if indexed)    uint8_t field_;    // Segment field number};`
**Sources:**[Mcm.hpp 928-946](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L928-L946)

### Vector Load Commit

Vector loads require matching RTL read operations with reference elements considering stride, masking, and element indices.

**Vector Load Cases:**

| Case | Strategy |
| --- | --- |
| Unit stride | Match by address only (indices don't matter) |
| Strided (stride=0) | All elements same address, verify one element covered |
| Strided (stride≠0) | Match by address AND element index |
| Indexed | Match by address AND element index |

**Sources:**[Mcm.cpp 2547-2681](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L2547-L2681)

## Configuration and Control

### Enabling/Disabling PPO Rules

Individual PPO rules can be enabled or disabled:

`// Disable all PPO checksmcm.enablePpo(false); // Enable only specific rulemcm.enablePpo(PpoRule::R1, true); // Check if rule enabledif (mcm.isEnabled(PpoRule::R3)) { ... }`
**Sources:**[Mcm.hpp 330-340](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L330-L340)

### TSO Mode

Enable Total Store Order mode for stricter ordering:

`mcm.enableTso(true);`
This affects how stores are checked for ordering violations.

**Sources:**[Mcm.hpp 300-302](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L300-L302)

### Merge Buffer Configuration

The merge buffer size (line size) is set at construction:

`Mcm<uint64_t> mcm(hartCount, pageSize, mergeBufferSize);`
Typical merge buffer size matches cache line size (64 bytes).

**Sources:**[Mcm.cpp 12-31](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L12-L31)

## Time Ordering and Global Memory Time

All memory operations are ordered by a global `time_` value that represents the memory system time. This enables:

1.   **Out-of-order detection**: Operations with time < preceding op time
2.   **Forwarding decisions**: Stores forward to loads at later times
3.   **PPO validation**: Checking time relationships between dependent ops

**Time-ordered Operations Vector:**

`std::vector<MemoryOp> sysMemOps_;  // All operations sorted by time_`
Operations are inserted maintaining time order, with special handling for out-of-order insertions.

**Sources:**[Mcm.hpp 1047-1049](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.hpp#L1047-L1049)[Mcm.cpp 191-255](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L191-L255)

## Verification and Checking

### Read Data Verification

For each read operation, the MCM compares RTL data with reference (Whisper) data:

`if (op.rtlData_ != op.data_) {    printReadMismatch(hart, op.time_, op.tag_, addr,                       op.size_, op.rtlData_, op.data_);    return false;}`
Device addresses and memory-mapped I/O may skip checking.

**Sources:**[Mcm.cpp 1792-1825](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1792-L1825)

### Store Data Verification

When stores drain via `mergeBufferWrite()`, the MCM verifies:

1.   All store bytes have corresponding merge buffer inserts
2.   Inserted data matches RTL drained data
3.   Store completes when all bytes drained

**Sources:**[Mcm.cpp 1883-1979](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L1883-L1979)

### PPO Violation Reporting

PPO violations are reported with detailed information:

`cerr << "Error: PPO rule " << ruleNum << " failed: hart-id=" << hartId      << " tag1=" << tagA << " tag2=" << tagB      << " time1=" << time1 << " time2=" << time2 << '\n';`
**Sources:**[Mcm.cpp 3513-3558](https://github.com/tenstorrent/whisper/blob/733fd921/Mcm.cpp#L3513-L3558)

## Summary

The Memory Consistency Model subsystem provides comprehensive out-of-order memory simulation and verification:

*   **Memory Operation Tracking**: All reads/writes tracked with timing and data
*   **Merge Buffer**: Simulates store buffer with coalescing and delayed draining
*   **Store-to-Load Forwarding**: Enables reading from pending stores
*   **PPO Rules**: Enforces 13 RISC-V ordering rules plus I/O constraints
*   **Dependency Tracking**: Maintains register producer/time information
*   **Vector Support**: Handles complex vector addressing modes
*   **Verification**: Compares RTL vs reference data at every operation

The MCM integrates with the Hart execution unit for reference operations, the Memory subsystem for physical storage, and the Performance API for detailed pipeline modeling. It enables lock-step verification of out-of-order memory implementations against the RISC-V memory model specification.
