## Purpose and Scope

This page documents the FP32 Fused Multiply-Add (FMA) implementations in Tenstorrent ASICs, their divergence from IEEE754, and bit-precise software models for verification.

**Covered FMA implementations:**

*   Vector Unit (SFPU): `SFPMAD` instruction ([BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1)[WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1))
*   Baby RISC-V: `fmadd.s`, `fmsub.s`, `fnmsub.s`, `fnmadd.s` ([BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 23](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L23-L23))
*   Software models: `fma_model_ieee`, `fma_model_bh`, `fma_model_wh` ([Miscellaneous/FMA/fma.c 14-288](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L14-L288))

### Partial vs. Full Fusion

Tenstorrent FMA implementations use **partial fusion**, not IEEE754-compliant full fusion.

**Diagram: FMA Precision Pipeline Comparison**

**Key difference:** IEEE754 adds precision to the addend (`z_m << 23`), Tenstorrent removes precision from the product (`p_m >> 23`). This reduces product mantissa from 49-51 bits to 26-28 bits before the add, limiting the final precision.

**Sources:**[Miscellaneous/FMA/fma.c 32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L32-L32)[Miscellaneous/FMA/fma.c 52-54](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L52-L54)[Miscellaneous/FMA/fma.c 70-77](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L70-L77)[Miscellaneous/FMA/fma.c 118](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L118-L118)[Miscellaneous/FMA/fma.c 125](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L125-L125)[Miscellaneous/FMA/fma.c 156-163](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L156-L163)[Miscellaneous/FMA/fma.c 213-214](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L213-L214)[Miscellaneous/FMA/README.md 12-22](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/README.md?plain=1#L12-L22)

### Mantissa Width Evolution

**Diagram: Mantissa Width Through Pipeline Stages**

After adding 3 guard bits, IEEE754 model expands addend to 50 bits to match product. Tenstorrent models reduce product to 27-28 bits to match addend. The product retains only 4 extra bits beyond FP32 (3 G/R/S guard bits + 1 overflow bit).

## Baby RISC-V FMA Instructions

Blackhole Baby RISC-V cores ([BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 1](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L1-L1)) implement four FP32 FMA instruction variants from RISC-V `F` Extension:

**Diagram: Baby RISC-V FMA Instruction Mapping**

**Semantics:**

*   `fmadd.s`: `rd = rs1 * rs2 + rs3`
*   `fmsub.s`: `rd = rs1 * rs2 + (-rs3)` (negate addend)
*   `fnmsub.s`: `rd = -(rs1 * rs2 + (-rs3))` (negate addend and result)
*   `fnmadd.s`: `rd = -(rs1 * rs2 + rs3)` (negate result)

**Conformance:** These instructions use `fma_model_bh`, NOT IEEE754 FMA. From [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 23](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L23-L23):

> semantics somewhere between separate multiply/add and fused multiply/add (i.e. even aside from denormals, they do _not_ conform to the IEEE754 definition of fused multiply/add)

**Conservative stalling:** When using `SFPMAD_MOD1_INDIRECT_VA` or `SFPMAD_MOD1_INDIRECT_VD`, hardware conservatively assumes all `LReg` are accessed [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 70-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L70-L71)

*   `SFPOR` with `SFPOR_MOD1_USE_VB`

*   `SFPCONFIG` reading `LReg[0]`
*   `SFPSWAP` 1st-cycle reads

## Rounding and Guard Bits

All three FMA models use **round-to-nearest-even (RNE)**[Miscellaneous/FMA/fma.c 96-282](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L96-L282):

**Guard/Round/Sticky (GRS) bits:** The low 3 bits of `r_m` encode:

*   **G (guard)**: Bit 2 - first discarded bit
*   **R (round)**: Bit 1 - second discarded bit
*   **S (sticky)**: Bit 0 - OR of all remaining discarded bits

**Rounding truth table:**

| `r_m & 7` | G | R | S | Discarded Value | Decision | Action |
| --- | --- | --- | --- | --- | --- | --- |
| `7` | 1 | 1 | 1 | (6, 8) open interval | Round up | Nearest |
| `6` | 1 | 1 | 0 | Exactly 6 | Round up | Nearest |
| `5` | 1 | 0 | 1 | (4, 6) open interval | Round up | Nearest |
| `4` | 1 | 0 | 0 | Exactly 4 (tie) | Check `r & 1` | Ties to even |
| `3` | 0 | 1 | 1 | (2, 4) open interval | Round down | Nearest |
| `2` | 0 | 1 | 0 | Exactly 2 | Round down | Nearest |
| `1` | 0 | 0 | 1 | (0, 2) open interval | Round down | Nearest |
| `0` | 0 | 0 | 0 | Exactly 0 | Round down | Exact value |

**Ties to even:** When `r_m & 7 == 4`, the expression `((4 + (r & 1)) > 4)` evaluates to:

*   `false` if `r & 1 == 0` (already even) → round down
*   `true` if `r & 1 == 1` (currently odd) → round up to even

The sticky bit enables correct rounding even though precision was reduced earlier in the pipeline.

## Sticky Shift Implementation

Sticky shifts right-shift a value while preserving information about discarded bits in the LSB (sticky bit).

### IEEE754 Sticky Shift: `sticky_shift`

[Miscellaneous/FMA/fma.c 59-69](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L59-L69):

**Behavior:**

*   Shift ≥ word width: Result is `1` if input was non-zero, else `0`
*   Shift < word width: Set LSB if any bits were discarded

**Purpose:** Preserves enough information for correct rounding in fully fused FMA.

### Tenstorrent Semi-Sticky Shift: `semi_sticky_shift`

[Miscellaneous/FMA/fma.c 145-155](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L145-L155):

**Behavioral differences:**

1.   Shift ≥ word width: Always `0` (not `1`)
2.   Sticky bit only set if `var` is non-zero after shift (via `if (var)` guard)

**Purpose:** Reflects partial fusion model where extreme underflows are treated as zero.

**Example comparison:**

```
Input: 0x1 (1), Shift: 64 bits
  sticky_shift:       Result = 0x1 (preserves non-zero)
  semi_sticky_shift:  Result = 0x0 (flushes to zero)

Input: 0x100, Shift: 7 bits
  Both:               Result = 0x3 (0x2 shifted + 0x1 sticky bit)
```

## Performance Characteristics

Each `SFPMAD` performs 64 FP32 operations per cycle (32 lanes × (1 multiply + 1 add)):

| Architecture | Clock Rate | Peak SFPMAD Throughput | Calculation |
| --- | --- | --- | --- |
| Wormhole B0 | 1.0 GHz | 0.064 TFLOP/s | 32 lanes × 2 ops × 1.0 GHz |
| Blackhole A0 | 1.35 GHz | 0.0864 TFLOP/s | 32 lanes × 2 ops × 1.35 GHz |

**Achieving peak throughput:** Use `SFPLOADMACRO`[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 90](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L90-L90) to overlap load/compute/store:

```
SFPLOADMACRO enables:
  - Load from Dst to LReg (cycle N)
  - Compute SFPMAD (cycle N+1)
  - Store from LReg to Dst (cycle N+2)
All overlapped for maximum throughput.
```

See page 5.4 for `SFPLOADMACRO` details.

## Architectural Evolution Summary

**Wormhole → Blackhole improvements:**

1.   Canonical NaN (`0x7fc00000`) instead of variable mantissa
2.   Preserved sign on negative zero (vs. always positive)
3.   Hardware automatic instruction scheduling for most cases
4.   `SFPMAD_MOD1_NEGATE_VA` and `SFPMAD_MOD1_NEGATE_VC` modifiers
5.   Improved edge case handling in corner cases
