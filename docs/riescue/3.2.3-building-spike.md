# Building Spike

 Relevant source files

* [infra/build\_spike.sh](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh)
* [infra/build\_whisper.sh](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh)

## Purpose and Scope

This document describes how the `build_spike.sh` script compiles and installs two versions of the Spike RISC-V instruction set simulator: the standard reference implementation and a Tenstorrent-specific fork (`tt_spike`). Both simulators are essential for test execution in RiESCUE's testing framework.

For information about the Whisper simulator build process, see [Building Whisper](/tenstorrent/riescue/3.2.2-building-whisper). For details on using these simulators in tests, see [Instruction Set Simulators](/tenstorrent/riescue/4.2-instruction-set-simulators).

Sources: [infra/build\_spike.sh1-69](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L1-L69)

## Overview of Spike Simulators

RiESCUE builds and installs two distinct versions of Spike:

| Simulator | Repository | Commit SHA | Binary Path | Purpose |
| --- | --- | --- | --- | --- |
| `spike` | `riscv-software-src/riscv-isa-sim` | `4703ad98bf4c247a0841a6d7254357b14a97ff29` | `/usr/local/bin/spike` | Standard RISC-V reference simulator |
| `tt_spike` | `tenstorrent/spike` | `344b9ef3951fc9318caae4ce41da8ede5d6085fc` | `/usr/local/bin/tt_spike` | Tenstorrent fork with custom extensions |

Both simulators are compiled from source using pinned commit SHAs to ensure reproducible builds across different environments. The `tt_spike` variant includes Tenstorrent-specific features for test infrastructure and debugging.

Sources: [infra/build\_spike.sh9-14](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L9-L14)

## Build Script Architecture

### Build Flow

The script implements a reusable `build_spike()` function that handles the complete build lifecycle for any spike variant. This function is called twice: once for the standard reference simulator and once for the TT-specific fork.

Sources: [infra/build\_spike.sh16-32](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L16-L32) [infra/build\_spike.sh37-69](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L37-L69)

### The build\_spike Function

The `build_spike()` function at [infra/build\_spike.sh16-32](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L16-L32) abstracts the build process into a reusable component. It performs a shallow clone of the specified commit using `git fetch --depth 1`, which minimizes download time and disk usage. The `shift 3` command allows all additional arguments to be forwarded as configure flags via `$@`.

Sources: [infra/build\_spike.sh16-32](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L16-L32)

## Standard Spike Build Configuration

### Standard Spike Build Invocation

The standard spike build is invoked at [infra/build\_spike.sh43](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L43-L43):

### Configuration Flags

| Flag | Value | Purpose |
| --- | --- | --- |
| `--enable-dual-endian` | N/A | Enables both big-endian and little-endian instruction fetch and memory access |
| `--with-isa` | `RV64IMAFDCV_ZBA_ZBB_ZBC_ZBS` | Specifies supported ISA extensions |
| `--with-priv` | `MSU` | Enables Machine, Supervisor, and User privilege modes |

#### ISA Extension Breakdown

The `--with-isa=RV64IMAFDCV_ZBA_ZBB_ZBC_ZBS` flag enables:

* **RV64I**: 64-bit base integer instruction set
* **M**: Integer multiplication and division
* **A**: Atomic instructions
* **F**: Single-precision floating-point
* **D**: Double-precision floating-point
* **C**: Compressed instructions (16-bit encodings)
* **V**: Vector extension
* **Zba**: Address generation instructions (bit manipulation)
* **Zbb**: Basic bit manipulation
* **Zbc**: Carry-less multiplication
* **Zbs**: Single-bit instructions

### Installation

Standard spike installs to the default prefix `/usr/local`, resulting in the binary at `/usr/local/bin/spike`. The build directory is removed after installation to conserve space: [infra/build\_spike.sh44](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L44-L44)

Sources: [infra/build\_spike.sh41-44](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L41-L44)

## TT-Specific Spike Build Configuration

### TT Spike Build Invocation

The Tenstorrent fork is built with additional configuration flags at [infra/build\_spike.sh48-49](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L48-L49):

### Tenstorrent-Specific Configuration Flags

| TT-Specific Flag | Purpose |
| --- | --- |
| `--enable-tt-stop-if-tohost-nonzero` | Provides automated test termination when the test writes to the `tohost` CSR, enabling self-terminating tests |
| `--enable-tt-table-walk-debug` | Adds debug output for page table walk operations, useful for diagnosing virtual memory issues |
| `--enable-tt-expanded-dram-address-range` | Extends supported DRAM addressing to accommodate larger memory configurations in Tenstorrent hardware |

### Installation and Binary Relocation

Unlike standard spike, `tt_spike` uses a custom installation prefix and requires manual binary relocation:

1. Configure with `--prefix=/usr/local/tt_spike` [infra/build\_spike.sh48-49](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L48-L49)
2. Execute `make install` to install to the prefix directory
3. Copy the binary from `/usr/local/tt_spike/bin/spike` to `/usr/local/bin/tt_spike` [infra/build\_spike.sh50](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L50-L50)
4. Remove temporary installation directory and build directory [infra/build\_spike.sh51-52](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L51-L52)

This ensures both `spike` and `tt_spike` binaries coexist in `/usr/local/bin/` with distinct names.

Sources: [infra/build\_spike.sh46-52](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L46-L52)

## Repository and Commit Pinning

### Pinned Commits

Both spike versions use hardcoded commit SHAs to ensure reproducibility:

* **Standard Spike**: `spike_remote` and `spike_sha` variables at [infra/build\_spike.sh10-11](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L10-L11)
* **TT Spike**: `tt_spike_remote` and `tt_spike_sha` variables at [infra/build\_spike.sh13-14](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L13-L14)

The shallow clone strategy (`git fetch --depth 1`) fetches only the specific commit without full repository history, significantly reducing download time and storage requirements.

Sources: [infra/build\_spike.sh9-14](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L9-L14)

## Container vs Direct Execution

### Execution Context Detection

The script detects execution context at [infra/build\_spike.sh38-39](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L38-L39):

**Direct Execution Mode** (lines 41-52): Used when:

* Singularity is not available
* `NO_SINGULARITY` environment variable is set
* Already running inside a container

**Container Execution Mode** (lines 56-68): Used when:

* Singularity is available
* `NO_SINGULARITY` is not set
* Running on host system

In container mode, the entire build sequence is passed as a bash command string to `container-run`, which executes it within the containerized environment.

Sources: [infra/build\_spike.sh37-69](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L37-L69)

## Build Dependencies

The `build_spike.sh` script assumes that base system dependencies have been installed via `build_base_deps.sh` (see [Base Dependencies](/tenstorrent/riescue/3.2.1-base-dependencies)). Required dependencies include:

* **Build tools**: `gcc`, `g++`, `make`, `autoconf`, `automake`
* **Version control**: `git`
* **Build system**: Standard Unix utilities

The spike build process uses the standard autotools workflow:

1. `./configure` generates Makefiles with specified feature flags
2. `make -j 12` compiles with 12 parallel jobs
3. `make install` installs binaries and libraries

Sources: [infra/build\_spike.sh28-30](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh#L28-L30)

## Summary

The `build_spike.sh` script provides a unified build process for both standard and Tenstorrent-specific spike simulators. Key design features include:

* **Commit pinning**: Hardcoded SHAs ensure reproducible builds
* **Shallow clones**: `--depth 1` minimizes clone time and storage
* **Reusable function**: `build_spike()` abstracts common build operations
* **Parallel builds**: `-j 12` flag accelerates compilation
* **Cleanup**: Temporary build directories removed after installation
* **Dual-mode execution**: Supports both direct and containerized builds

Both simulators are installed to `/usr/local/bin/` with distinct names (`spike` and `tt_spike`), allowing test frameworks to select the appropriate simulator based on test requirements.
