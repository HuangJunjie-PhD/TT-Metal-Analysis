# Chip Detection and Communication

## Purpose and Scope

This page documents the chip detection and communication verification system in tt-tools-common. The system discovers Tenstorrent hardware devices on the host, validates that communication channels are functional, and prepares chips for use by performing firmware upgrades when necessary.

This page focuses on the detection and communication aspects. For chip-specific data loading and version conversions, see [Version Conversions and Data Loading](https://deepwiki.com/tenstorrent/tt-tools-common/3.4-version-conversions-and-data-loading). For reference clock monitoring performed after detection, see [Reference Clock Monitoring](https://deepwiki.com/tenstorrent/tt-tools-common/3.5-reference-clock-monitoring).

## Overview

Chip detection is implemented through the `detect_chips_with_callback()` function, which wraps pyluwen's `detect_chips_fallible()` with a rich callback-based progress reporting system. The detection process:

1.   Enumerates PCI devices using pyluwen's hardware abstraction layer
2.   Validates communication with each discovered device
3.   Reports detailed progress through callback functions
4.   Automatically upgrades firmware when needed
5.   Returns a list of communicable `PciChip` objects

**Sources:**[tt_tools_common/utils_common/tools_utils.py 255-354](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L255-L354)

### Detection Parameters

The `detect_chips_with_callback()` function accepts three configuration parameters:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `local_only` | `bool` | `False` | When `True`, only detects chips directly connected via PCIe. When `False`, also discovers remote chips via Ethernet fabric. |
| `ignore_ethernet` | `bool` | `False` | Passed as `noc_safe` to pyluwen. Skips Ethernet core detection for faster operation. |
| `print_status` | `bool` | `True` | Controls whether detection progress is printed to terminal. Set to `False` for silent operation. |

## Communication Verification

After detection completes, each device undergoes communication verification before being added to the output list.

### have_comms() Check

The `device.have_comms()` method (provided by pyluwen) validates that:

*   The device responds to PCIe transactions
*   ARC firmware is running and responsive
*   Basic register reads/writes succeed

If communication cannot be established, the function raises an exception with a diagnostic message:

**Sources:**[tt_tools_common/utils_common/tools_utils.py 346-349](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L346-L349)

### Communication Failure Scenarios

| Scenario | Likely Cause | Recommended Action |
| --- | --- | --- |
| PCIe enumeration succeeds but `have_comms()` fails | ARC firmware hang, hardware fault | Perform chip reset (see [ChipReset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset)) |
| Device not enumerated at all | Power issue, PCIe link failure | Check hardware connections |
| Communication intermittent | PCIe signal integrity, thermal throttling | Check host system compatibility (see [Host Information and Compatibility](https://deepwiki.com/tenstorrent/tt-tools-common/3.2-host-information-and-compatibility)) |

## Firmware Upgrade Process

After communication is verified, the detection system automatically performs firmware upgrades through the `force_upgrade()` method.

### Upgrade Flow

**Sources:**[tt_tools_common/utils_common/tools_utils.py 351-352](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L351-L352)

### When Upgrades Occur

Firmware upgrades happen automatically when:

1.   The installed firmware version is older than the version bundled with pyluwen
2.   The firmware is corrupted or reports an invalid version
3.   The firmware is incompatible with the current driver version

The upgrade process is transparent to the caller and ensures all returned `PciChip` objects have compatible firmware.

### Upgrade Timing Considerations

From CHANGELOG.md, version 1.4.8 notes:

> "Updated reset completion logic to handle the case where the bmfw needs to upgrade itself"

This indicates that firmware upgrades can occur during reset operations as well, not just during detection. The system handles these asynchronous upgrades gracefully.

## Detection Flags and Behavior

The function's behavior varies based on configuration flags:

### Local-Only Mode

When `local_only=True`:

*   Only detects chips directly connected to the host via PCIe
*   Skips Ethernet fabric traversal
*   Faster detection for systems without multi-chip Ethernet connections
*   Callback status strings are minimal

**Sources:**[tt_tools_common/utils_common/tools_utils.py 256-341](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L256-L341)

### Ethernet-Ignored Mode

When `ignore_ethernet=True`:

*   Detection still occurs for all chips (local and remote)
*   Ethernet core validation is skipped via `noc_safe=True`
*   Reduces detection time when Ethernet functionality is not required
*   Used when only Tensix/DRAM communication is needed

**Sources:**[tt_tools_common/utils_common/tools_utils.py 257-344](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L257-L344)

### Silent Mode

When `print_status=False`:

*   All terminal output is suppressed
*   Callback logic still executes for counter tracking
*   Useful for scripting and automated tools
*   Added in version 1.4.10 (2024-12-16)

**Sources:**[tt_tools_common/utils_common/tools_utils.py 258-296](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L258-L296)[CHANGELOG.md 37-40](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L37-L40)

## Error Handling

The detection system implements fail-fast error handling:

The system uses `continue_on_failure=False` when calling `detect_chips_fallible()`, meaning detection stops at the first non-communicable device rather than attempting to continue.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 340-349](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L340-L349)

## Return Value and Usage

The function returns `List[PciChip]`, where each `PciChip` object:

*   Has verified communication (`have_comms()` returned `True`)
*   Has up-to-date firmware (`force_upgrade()` completed)
*   Is ready for immediate use in reset operations, telemetry queries, etc.

## Summary

The chip detection and communication system provides a robust, user-friendly interface for discovering and validating Tenstorrent hardware. Key features include:

*   **Progressive feedback** through callback-based status updates
*   **Automatic firmware management** via force_upgrade()
*   **Flexible detection modes** for different use cases
*   **Fail-fast validation** ensuring all returned chips are communicable
*   **Integration with pyluwen** for hardware abstraction

Consumer tools like tt-smi and tt-flash rely on this system as their primary hardware discovery mechanism, typically calling `detect_chips_with_callback()` during initialization to build their device inventory.
