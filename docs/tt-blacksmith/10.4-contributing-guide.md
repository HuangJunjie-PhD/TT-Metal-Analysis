## Testing Requirements

### Running Tests Locally

Before submitting a pull request, contributors should run relevant tests locally:

**Sources:** Based on testing infrastructure described in architecture diagrams

### Test Markers

Tests use pytest markers to categorize and filter test execution:

| Marker Category | Values | Purpose |
| --- | --- | --- |
| Hardware | `n150`, `n300`, `galaxy` | Target hardware configuration |
| Framework | `torch`, `jax` | ML framework being tested |
| Strategy | `single_chip`, `data_parallel`, `tensor_parallel` | Parallelization strategy |
| Frequency | `push`, `uplift` | When test should run in CI |

**Sources:** Based on architecture diagram "Diagram 4: Multi-Layer Testing and Validation System"

### Adding New Tests

When adding new experiments or features, contributors should:

1.   **Add test case to `TRAINING_TEST_CASES`**: Define declarative test parameters
2.   **Apply appropriate markers**: Use pytest markers for categorization
3.   **Create golden files**: If adding a new model, include reference loss values for validation
4.   **Document test requirements**: Update test documentation with any special requirements

**Sources:** Based on testing architecture and patterns

## Tenstorrent Bounty Program

### Overview

TT-Blacksmith participates in the Tenstorrent Bounty Program, which rewards community contributions for specific issues. The bounty program offers financial incentives for successfully completing tagged issues.

**Sources:**[README.md 50-51](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/README.md?plain=1#L50-L51)

### Finding Bounty Issues

Bounty issues are tagged with both:

*   **`bounty`** label indicating financial reward
*   **Difficulty level** label (e.g., easy, medium, hard)

To find bounty issues:

1.   Visit the [TT-Forge issues page](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/TT-Forge%20issues%20page)
2.   Filter by difficulty level that matches your expertise
3.   Read the issue description and requirements carefully
4.   Check if the issue is already assigned before starting work

**Note:** TT-Blacksmith bounty issues may be tracked in the TT-Forge repository as part of the broader Tenstorrent ecosystem.

**Sources:**[README.md 7](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/README.md?plain=1#L7-L7)[README.md 50-51](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/README.md?plain=1#L50-L51)

### Terms and Conditions

Before participating in the bounty program, contributors must:

*   Read the [Tenstorrent Bounty Program Terms and Conditions](https://docs.tenstorrent.com/bounty_terms.html)
*   Understand payment requirements and timelines
*   Agree to contribution licensing terms

**Sources:**[README.md 50-51](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/README.md?plain=1#L50-L51)

## Pull Request Process

### Submission Workflow

**Sources:** Based on CI/CD architecture and standard Git workflow

### PR Requirements

A pull request should include:

1.   **Clear description**: Explain what the PR does and why
2.   **Tests**: Include tests for new functionality
3.   **Documentation**: Update relevant documentation
4.   **Clean commit history**: Use meaningful commit messages
5.   **No merge conflicts**: Rebase on latest main if needed

### CI/CD Validation

When you submit a pull request, the following automated checks run:

| Check Type | Purpose | Required |
| --- | --- | --- |
| Test Matrix | Run tests on multiple hardware configs | Yes |
| Golden File Validation | Compare training outputs to references | Yes |
| Environment Check | Verify dependencies install correctly | Yes |
| Lint Checks | Code style validation | Recommended |

**Sources:** Based on "Diagram 4: Multi-Layer Testing and Validation System" and CI/CD architecture

### Review Process

1.   **Automated checks**: CI/CD pipeline runs tests automatically
2.   **Maintainer review**: Core maintainers review code quality, design, and test coverage
3.   **Feedback iteration**: Address review comments through additional commits
4.   **Approval and merge**: Once approved, maintainers merge the PR

## Development Patterns

### Adding New Experiments

When contributing a new experiment, follow this pattern:

**Key steps:**

1.   Create experiment directory following naming convention
2.   Implement training script using standard manager pattern
3.   Create default configuration YAML
4.   Write comprehensive README
5.   Add test case with appropriate markers
6.   Generate golden files by running experiment
7.   Submit PR with all components

### Using Manager Classes

All experiments should use the standardized manager classes:

See [Manager Classes](https://deepwiki.com/tenstorrent/tt-blacksmith/5.1-manager-classes) for detailed documentation on each manager's responsibilities.

**Sources:** Based on architecture patterns and core infrastructure

### Configuration-Driven Development

Follow the configuration-driven pattern where all experiment parameters come from `TrainingConfig`:

**Benefits of this pattern:**

*   Complete reproducibility through YAML files
*   Easy experiment configuration without code changes
*   Consistent validation via Pydantic
*   Centralized configuration management

**Sources:** Based on "Diagram 3: Configuration-Driven System Architecture"

## Common Contribution Types

### 1. New Model Support

To add support for a new model:

*   Create experiment directory under `experiments/torch_experiments/` or `experiments/jax_experiments/`
*   Implement `get_model` function with proper device placement
*   Add LoRA/adapter support if applicable
*   Create test case with appropriate hardware markers
*   Document model-specific requirements

### 2. New Dataset Integration

To add a new dataset:

*   Implement `BaseDataset` pattern in dataset factory
*   Add proper preprocessing and tokenization
*   Support both single-chip and distributed data loading
*   Add dataset to available options in documentation

### 3. Infrastructure Improvements

For core infrastructure changes:

*   Ensure backward compatibility with existing experiments
*   Update all affected experiments
*   Add comprehensive tests
*   Update relevant documentation sections

### 4. Bug Fixes

When fixing bugs:

*   Include test that reproduces the bug
*   Verify fix doesn't break existing functionality
*   Run full test suite locally before submitting
*   Reference issue number in PR description

## Summary

Contributing to TT-Blacksmith involves:

1.   **Setup**: Clone repository and activate appropriate environment
2.   **Development**: Follow established patterns for experiments, tests, and documentation
3.   **Testing**: Run tests locally and ensure CI/CD checks pass
4.   **Submission**: Create well-documented PRs with complete test coverage
5.   **Review**: Engage with maintainers during code review process

The repository maintains high standards for code quality, test coverage, and documentation. Following these guidelines ensures smooth contribution integration and helps maintain the project's quality.
