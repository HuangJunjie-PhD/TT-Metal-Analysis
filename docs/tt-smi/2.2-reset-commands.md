## Purpose and Scope

This page documents all reset-related command-line flags available in TT-SMI. Reset commands allow users to perform hardware resets on Tenstorrent devices through various CLI options, each designed for specific reset scenarios and hardware configurations.

For implementation details of the reset architecture, platform-specific reset handlers, and the underlying reset mechanisms, see [Reset Operations](https://deepwiki.com/tenstorrent/tt-smi/5-reset-operations). For basic CLI operations like device listing and version display, see [Basic CLI Operations](https://deepwiki.com/tenstorrent/tt-smi/2.1-basic-cli-operations).

## Reset Command Overview

TT-SMI provides five distinct reset command options, each serving different use cases:

| CLI Flag | Long Form | Purpose | Target Hardware |
| --- | --- | --- | --- |
| `-r [indices]` | `--reset [indices]` | Standard PCI device reset | All non-Galaxy boards |
| `-glx_reset` | `--galaxy_6u_trays_reset` | Full Galaxy system reset | Galaxy 6U systems only |
| `-glx_reset_auto` | `--galaxy_6u_trays_reset_auto` | Galaxy reset with auto-retry (up to 3x) | Galaxy 6U systems only |
| `-glx_reset_tray N` | `--galaxy_6u_reset_tray N` | Single tray reset | Galaxy 6U systems only |
| `--no_reinit` | N/A | Skip device re-detection after reset | All reset commands |

**Sources:**[tt_smi/tt_smi.py 624-680](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L624-L680)[tt_smi/tt_smi.py 789-871](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L789-L871)

## Standard PCI Reset (`-r`)

### Command Syntax

### Behavior

The `-r` / `--reset` flag performs standard PCI-based resets on specified devices. The command accepts an optional comma-separated list of PCI indices. If no indices are provided, all detected devices will be reset.

**Device Selection Logic:**

*   **With indices**: Resets only specified PCI devices (e.g., `-r 0,2,5`)
*   **Without indices**: Resets all devices on the system (e.g., `-r`)

### Platform-Specific Reset Types

The `pci_board_reset()` function dispatches to platform-specific reset handlers based on the detected chip type:

| Platform | Reset Handler | Reset Type |
| --- | --- | --- |
| Wormhole | `WHChipReset().full_lds_reset()` | Board-level reset (power cycle + ETH retrain) |

| Galaxy WH/BH | Special handling (no secondary bus reset) | Board-level with constraints |

## Galaxy Reset Commands

Galaxy 6U systems require specialized reset commands that use IPMI-based reset mechanisms rather than standard PCI resets. These commands are only valid on Galaxy hardware configurations.

**Sources:**[tt_smi/tt_smi.py 805-871](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L805-L871)[tt_smi/tt_smi_backend.py 860-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L928)[tt_smi/tt_smi.py 733-766](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L733-L766)

### Full Galaxy Reset (`-glx_reset`)

**Command Syntax:**

**Behavior:** Performs a full reset of all ASICs on all trays in the Galaxy 6U system. This command:

1.   Executes IPMI reset command on all trays (UBB bitmask `0xF`)
2.   Waits 30 seconds for hardware reset completion
3.   Waits for driver to load
4.   Re-initializes all chips (local only to avoid ETH status issues)
5.   Validates Ethernet link status for all 32 devices
6.   Exits with error if any Ethernet links fail validation

**Sources:**[tt_smi/tt_smi.py 805-821](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L805-L821)[tt_smi/tt_smi_backend.py 860-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L928)

### Auto-Retry Galaxy Reset (`-glx_reset_auto`)

**Command Syntax:**

**Behavior:** Same as `-glx_reset`, but automatically retries up to 3 times if Ethernet link validation fails. This command is useful for handling transient reset failures.

**Retry Logic:**

*   Attempt 1: Initial reset attempt
*   Attempt 2-3: Retry if Exception raised (typically from Ethernet validation failure)
*   After 3 failures: Exit with error code 1

**Sources:**[tt_smi/tt_smi.py 822-857](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L822-L857)

### Single Tray Reset (`-glx_reset_tray`)

**Command Syntax:**

Where `N` is the tray number (1, 2, 3, or 4).

**Behavior:** Resets a specific tray on the Galaxy 6U system. Uses a bitmask to target only the specified tray:

**Tray Bitmask Encoding:**

| Tray Number | Bitmask (hex) | Binary |
| --- | --- | --- |

### IPMI Reset Parameters

The `glx_6u_trays_reset()` function accepts several parameters that map to raw IPMI command arguments:

**Parameter Details:**

| Parameter | Default | Description |
| --- | --- | --- |
| `ubb_num` | `"0xF"` | UBB number bitmask. `0xF` = all 4 trays |
| `dev_num` | `"0xFF"` | Device number bitmask. `0xFF` = all 8 devices per tray |
| `op_mode` | `"0x0"` | `0x0` = Assert/deassert with period `0x1` = Assert only `0x2` = Deassert only |
| `reset_time` | `"0xF"` | Reset duration in 10ms units (`0xF` = 15 × 10ms = 150ms) |

These parameters are passed directly to the IPMI command:

**Sources:**[tt_smi/tt_smi_backend.py 860-889](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L889)[pyluwen](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyluwen#LNaN-LNaN)

### Ethernet Link Validation

After Galaxy resets, TT-SMI validates Ethernet connectivity by checking for link errors:

This validation:

*   Expects exactly 32 devices (full Galaxy system)
*   Checks all 16 Ethernet ports on each device
*   Reads error status from firmware debug buffer at offset `0x12c0 + 0x4*96`
*   Raises exception if any link has `LINK_INACTIVE_FAIL_DUMMY_PACKET` error

**Sources:**[tt_smi/tt_smi_backend.py 819-858](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L819-L858)

## Device Re-initialization Control (`--no_reinit`)

### Command Syntax

### Behavior

The `--no_reinit` flag controls whether TT-SMI re-detects and re-initializes devices after performing a reset. By default, all reset commands re-initialize devices to verify successful reset completion.

### Application in Standard Resets

### Application in Galaxy Resets

### Use Cases

| Scenario | Use `--no_reinit`? | Reason |
| --- | --- | --- |
| Production reset | No (default) | Verify reset succeeded by detecting devices |
| Fast iteration testing | Yes | Skip 30s+ detection time |
| Custom initialization | Yes | Run custom scripts post-reset |
| Debugging reset only | Yes | Isolate reset behavior from detection |

**Important:** For Galaxy resets, `--no_reinit` is **not recommended** with `-glx_reset_auto` because the auto-retry mechanism requires device detection to determine if Ethernet links are healthy.

## Integration with Device Listing

To identify correct PCI indices for reset commands, users can leverage the `-ls` flag to list all available devices and their reset eligibility.

### List Command Syntax

### Output Structure

The listing command generates two tables:

1.   **All available boards on host**: Shows all detected devices
2.   **Boards that can be reset**: Shows only devices eligible for `-r` reset

### Reset Eligibility Rules

A device appears in "Boards that can be reset" if:

1.   **Not a remote device** (`not device.is_remote()`)
2.   **Not a Galaxy 4U board** (`board_type != "wh_4u"`)

Remote devices on n300 boards are automatically reset when their local counterpart is reset, so they are excluded from direct reset targeting.

**Sources:**[tt_smi/tt_smi_backend.py 154-201](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L154-L201)[tt_smi/tt_smi.py 700-702](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L700-L702)

### Common Error Messages

| Error | Cause | Solution |
| --- | --- | --- |
| "No Tenstorrent driver detected!" | Driver not installed or not loaded | Install tt-kmd driver |
| "This is not a Galaxy board" | Galaxy command used on non-Galaxy hardware | Use `-r` for standard boards |
| "Error accessing board at PCI index N" | Invalid PCI index or inaccessible device | Run `tt-smi -ls` to verify indices |
| "Unknown chip type detected" | Unsupported hardware | Update TT-SMI or use supported hardware |
| "Expected 32 devices for WH Galaxy" | Incomplete device detection | Retry reset or cold boot system |
| "Board X has link error on eth port Y" | Ethernet link training failure | Use `-glx_reset_auto` for auto-retry |
