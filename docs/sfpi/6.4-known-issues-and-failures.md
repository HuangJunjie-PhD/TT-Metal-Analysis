This document catalogs the known test failures, compiler diagnostics, and limitations tracked in the SFPI toolchain testing infrastructure. These failures are systematically recorded in `.xfail` files and used to distinguish expected failures from new regressions during continuous integration.

For information about the test infrastructure and execution, see [Test Infrastructure Overview](https://deepwiki.com/tenstorrent/sfpi/6.1-test-infrastructure-overview). For details on the test suites themselves, see [SFPU Kernel Tests](https://deepwiki.com/tenstorrent/sfpi/6.2-sfpu-kernel-tests) and [Compiler Optimization Tests](https://deepwiki.com/tenstorrent/sfpi/6.3-compiler-optimization-tests).

## Xfail System Architecture

The SFPI testing infrastructure uses a systematic approach to track and filter known failures across multiple test suites. This prevents expected failures from masking new regressions.

**Xfail File Format**

Each xfail file uses a simple format to specify which test failures are expected:

| Element | Purpose | Example |
| --- | --- | --- |
| `multilibs:` | Specifies which multilib configurations the failures apply to | `multilibs: riscv-sim/*` |
| `FAIL:` | Identifies a specific test failure pattern | `FAIL: gcc.c-torture/execute/20000822-1.c -O0 execution test` |
| Comments | Document the source or category of failures | `# these fails occur with stock gcc 12.4.0` |

**Sources:**[xfails/gcc.xfail 1-134](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gcc.xfail#L1-L134)[xfails/g++.xfail 1-61](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/g++.xfail#L1-L61)[xfails/ld.xfail 1-30](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/ld.xfail#L1-L30)[xfails/binutils.xfail 1-2](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/binutils.xfail#L1-L2)[xfails/gas.xfail 1-2](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gas.xfail#L1-L2)

## GCC Test Suite Known Failures

The GCC test suite contains 126 documented expected failures that fall into three categories: upstream stock failures, architecture-specific issues, and issues under investigation.

### Stock GCC Failures

These failures are inherited from the upstream GCC 12.4.0 toolchain and affect all multilib configurations:

| Test Pattern | Failure Count | Issue Category |
| --- | --- | --- |
| `gcc.c-torture/execute/20000822-1.c` | 7 (all optimization levels) | Execution failure |
| `gcc.c-torture/execute/nestfunc-*.c` | 35 (nestfunc-1,2,3,5,6) | Nested function trampolines |
| `gcc.c-torture/execute/fprintf-2.c` | 7 | I/O formatting |
| `gcc.c-torture/execute/printf-2.c` | 7 | I/O formatting |
| `gcc.dg/torture/fp-*-convert-*.c` | 21 | Floating-point conversions |
| `gcc.dg/torture/stackalign/nested-*.c` | 24 | Stack alignment with nested functions |

**Key Patterns:**

*   **Nested Function Failures**: Tests `nestfunc-1.c` through `nestfunc-6.c` fail across all optimization levels [xfails/gcc.xfail 19-42](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gcc.xfail#L19-L42) These involve GCC's trampoline mechanism for nested function closures.
*   **Floating-Point Conversion**: Tests like `fp-double-convert-float-1.c` and `fp-uint64-convert-double-*.c` fail consistently [xfails/gcc.xfail 70-90](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gcc.xfail#L70-L90)
*   **Debug Information**: BTF (BPF Type Format) datasec tests fail due to section naming differences [xfails/gcc.xfail 60-63](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gcc.xfail#L60-L63)

### Architecture-Specific Failures

These failures only occur on specific Tenstorrent architectures:

```
multilibs: riscv-sim/cpu=tt-gs riscv-sim/cpu=tt-wh
FAIL: c-c++-common/pr69669.c -Wc++-compat scan-rtl-dump-times final "mem:QI" 1
FAIL: gcc.dg/20141029-1.c scan-rtl-dump-times final "mem/v(/.)*:HI" 4
FAIL: gcc.dg/loop-unswitch-6.c (test for excess errors)
FAIL: gcc.dg/pr86617.c scan-rtl-dump-times final "mem/v" 6
```

These failures affect Grayskull (`tt-gs`) and Wormhole (`tt-wh`) but not Blackhole, suggesting differences in memory operation handling or RTL generation patterns.

**Sources:**[xfails/gcc.xfail 1-134](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/gcc.xfail#L1-L134)

## G++ Test Suite Known Failures

The G++ test suite has 60 documented failures, with distinct patterns for upstream issues, architecture-specific problems, and timeout scenarios.

### Stock G++ Failures

Inherited from upstream GCC 12.4.0, affecting all C++ standard versions:

| Test Pattern | Standard Versions | Issue |
| --- | --- | --- |
| `c-c++-common/patchable_function_entry-*.c` | gnu++98/14/17/20 | NOP instruction scanning |
| `c-c++-common/spec-barrier-1.c` | gnu++98/14/17/20 | Excess errors |
| `g++.dg/abi/pure-virtual1.C` | c++98/14/17/20 | Excess errors |
| `g++.dg/cpp2a/decomp2.C` | gnu++14/17/20 | Execution failure |
| `g++.dg/warn/Wstringop-overflow-6.C` | gnu++14/17/20 | Excess errors |

### TT-Specific Failures

Failures unique to Tenstorrent architecture support:

```
multilibs: riscv-sim riscv-sim/cpu=tt-bh riscv-sim/cpu=tt-gs
FAIL: g++.target/tt/pr13.C (test for excess errors)
```

This failure affects all three architectures, indicating a common issue in TT-specific test infrastructure [xfails/g++.xfail 34-35](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/g++.xfail#L34-L35)

### RAW Hazard Test Failures

Blackhole and base RISC-V configurations fail specific RAW (Read-After-Write) hazard tests:

```
multilibs: riscv-sim/ riscv-sim/cpu=tt-bh
FAIL: g++.target/tt/raw-race-1.C scan-assembler _Z2okIcEjjj:...
FAIL: g++.target/tt/raw-race-1.C scan-assembler _Z2okIsEjjj:...
```

These tests verify proper assembly generation for preventing RAW hazards in byte and short operations [xfails/g++.xfail 37-41](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/g++.xfail#L37-L41)

### Timeout Failures

Several `pr31863.C` tests timeout in checking builds across all optimization levels:

```
multilibs: riscv-sim/*
FAIL: g++.dg/torture/pr31863.C -O0 (test for excess errors)
FAIL: g++.dg/torture/pr31863.C -O1 (test for excess errors)
...
```

These represent compilation performance issues rather than correctness problems [xfails/g++.xfail 52-60](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/g++.xfail#L52-L60)

**Sources:**[xfails/g++.xfail 1-61](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/g++.xfail#L1-L61)

## SFPI-Specific Compiler Diagnostics

Beyond upstream test failures, the SFPI compiler enforces specific constraints on SFPU programming that generate intentional errors and warnings. These are captured in gold standard warning files.

### SFPU Vector Memory Constraints

The SFPI compiler prohibits certain operations that would violate SFPU hardware constraints:

### Error Messages and Line Numbers

**Cannot Write SFPU Vector to Memory:**

```
../../../include/sfpi.h:798:7: error: cannot write sfpu vector to memory
  798 |     v = (initialized) ? __builtin_rvtt_sfpassign_lv(v, in) : in;
```

This error occurs when attempting to:

*   Assign to global `vFloat` variables [tests/wormhole/sfpi/gold/warn.txt 6-8](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L6-L8)
*   Use pointer dereferencing [tests/wormhole/sfpi/gold/warn.txt 12-14](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L12-L14)
*   Take the address of `vFloat`[tests/wormhole/sfpi/gold/warn.txt 19-21](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L19-L21)
*   Return `vFloat` from non-inlined functions [tests/wormhole/sfpi/gold/warn.txt 38-40](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L38-L40)

**Stack Passing Restriction:**

```
cc1plus: error: invalid declaration for function 'argcall', 
         sfpu types cannot be passed on the stack (missing sfpi_inline?)
```

Functions accepting `vFloat`, `vInt`, or `vUInt` parameters must be marked `sfpi_inline` to ensure register passing [tests/wormhole/sfpi/gold/warn.txt 32-33](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L32-L33)

**Malformed Conditional:**

```
warn.cc:76:33: error: malformed program, sfpsetcc_v outside of pushc/popc
   76 |     __builtin_rvtt_wh_sfpsetcc_v(a, 12);
```

The `sfpsetcc_v` instruction must be used within conditional blocks managed by `pushc`/`popc` intrinsics [tests/wormhole/sfpi/gold/warn.txt 41-44](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L41-L44)

**Register Spill Error:**

```
../../../include/wormhole/sfpi_imp.h:105:11: fatal error: 
    cannot store sfpu register (register spill)
  105 |     assign(__builtin_rvtt_sfpxloadi(SFPXLOADI_MOD0_FLOAT, __f32asui(val)));
```

This fatal error occurs when the compiler runs out of LREG registers and attempts to spill to memory, which is prohibited [tests/wormhole/sfpi/gold/warn_spill.txt 7-9](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn_spill.txt#L7-L9)

### Uninitialized Variable Warnings

## Using Xfail Files in Testing

The xfail system integrates with the build and test workflow to provide clean regression detection:

**Build Script Integration:**

The `build.sh` script runs tests with xfail filtering enabled:

`# Test execution with xfail filtering./build.sh --test-tt         # Run TT-specific tests./build.sh --test-gcc        # Run GCC test suite./build.sh --test-binutils   # Run binutils test suite`
**Local Xfails Filtering:**

The `local-xfails.py` script (referenced but not provided in files) reads the xfail files and filters test results to show only unexpected failures. This enables:

*   Clean CI builds despite known upstream issues
*   Clear identification of new regressions
*   Architecture-specific failure tracking
*   Multilib-aware failure patterns

**Adding New Expected Failures:**

When a new expected failure is identified:

1.   Verify it occurs in stock GCC/binutils (if upstream)
2.   Add entry to appropriate xfail file with multilib specification
3.   Add comment documenting source/reason
4.   Re-run tests to verify filtering works correctly

**Sources:** All xfail files in [xfails/](https://github.com/tenstorrent/sfpi/blob/ed989b9d/xfails/)
