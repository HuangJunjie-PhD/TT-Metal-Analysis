The Telemetry system in Luwen provides monitoring and diagnostic capabilities for Tenstorrent AI accelerator chips. It enables users to access hardware metrics, component statuses, firmware versions, and operational parameters. This document describes the telemetry data structure, retrieval mechanisms, and common usage patterns.

Sources: [crates/luwen-if/src/chip/mod.rs 67-131](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L67-L131)

## System Overview

The telemetry system forms a critical part of chip monitoring in Luwen, allowing applications to retrieve performance data and status information from various chip architectures (Wormhole, Grayskull, and Blackhole). Telemetry data is accessed through the chip interface abstraction and is collected at runtime from the hardware.

## Telemetry Retrieval

The telemetry data is accessed through the `ChipImpl` trait's `get_telemetry()` method, which must be implemented by all chip types. The implementation is responsible for collecting the relevant data from the hardware and populating the `Telemetry` structure.

Sources: [crates/luwen-if/src/chip/mod.rs 301](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L301-L301)[crates/luwen-if/src/chip/mod.rs 378-380](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/mod.rs#L378-L380)

### Implementation in Different Chip Architectures

Each chip architecture implements the telemetry collection differently based on its hardware specifics:

1.   **Wormhole** - Retrieves telemetry through ARC messaging system
2.   **Grayskull** - Uses memory-mapped registers for telemetry access
3.   **Blackhole** - Employs a tag-based system with structured telemetry data

Blackhole's implementation uses a tagged approach defined in the `TelemetryTags` enum to identify different telemetry values.

Sources: [crates/luwen-if/src/chip/blackhole/telemetry_tags.rs 1-39](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/chip/blackhole/telemetry_tags.rs#L1-L39)[tests/telem_test.rs 29-72](https://github.com/tenstorrent/luwen/blob/55e35616/tests/telem_test.rs#L29-L72)[tests/telem_test.rs 118-171](https://github.com/tenstorrent/luwen/blob/55e35616/tests/telem_test.rs#L118-L171)

## Using Telemetry Data

The `Telemetry` struct provides several helper methods to convert raw values into more useful representations:

1.   **Firmware Information**:

    *   `firmware_date()` - Returns firmware date in YYYY-MM-DD format
    *   `arc_fw_version()` - Returns ARC firmware version in MAJOR.MINOR.PATCH format
    *   `eth_fw_version()` - Returns Ethernet firmware version in MAJOR.MINOR.PATCH format

2.   **Board Identification**:

    *   `board_serial_number()` - Returns the board serial number as an integer
    *   `board_serial_number_hex()` - Returns the board serial number as a hex-formatted string
    *   `board_type()` - Returns the board type based on the serial number

3.   **Performance Metrics**:

    *   `ai_clk()` - Returns the AI clock speed in MHz
    *   `axi_clk()` - Returns the AXI clock speed in MHz
    *   `arc_clk()` - Returns the ARC clock speed in MHz

4.   **Environmental Monitoring**:

    *   `voltage()` - Returns the core voltage in volts
    *   `asic_temperature()` - Returns the ASIC temperature in degrees celsius
    *   `vreg_temperature()` - Returns the voltage regulator temperature in degrees celsius
    *   `inlet_temperature()` - Returns the inlet temperature in degrees celsius
    *   `outlet_temperature1()` - Returns the first outlet temperature in degrees celsius
    *   `outlet_temperature2()` - Returns the second outlet temperature in degrees celsius

5.   **Power Management**:

    *   `power()` - Returns the power consumption in watts
    *   `current()` - Returns the current consumption in amperes

## Example Usage

Below is a typical example of retrieving and using telemetry data from a chip:

`// Retrieve a chip (assuming you've already detected chips)let chips = luwen_ref::detect_chips().unwrap();let chip = &chips[0]; // Get telemetry datalet telemetry = chip.get_telemetry().unwrap(); // Print basic chip informationprintln!("Chip type: {}", telemetry.board_type());println!("Serial number: {}", telemetry.board_serial_number_hex());println!("Firmware date: {}", telemetry.firmware_date());println!("ARC firmware: {}", telemetry.arc_fw_version()); // Print performance metricsprintln!("AI clock: {} MHz", telemetry.ai_clk());println!("ASIC temperature: {:.1}°C", telemetry.asic_temperature());println!("Power consumption: {:.1} W", telemetry.power());println!("Core voltage: {:.2} V", telemetry.voltage());`

## Telemetry in Testing

Telemetry data is frequently used in testing to verify hardware functionality and chip status. The test suite includes specific tests for each chip architecture that retrieve and validate telemetry information:

*   `wormhole_test_chip_telemetry` - Tests telemetry collection on Wormhole chips
*   `grayskull_test_chip_telemetry` - Tests telemetry collection on Grayskull chips
*   `blackhole_test_chip_telemetry` - Tests telemetry collection on Blackhole chips

These tests verify that the telemetry values are consistent and correctly represent the hardware state.

Sources: [tests/telem_test.rs 1-172](https://github.com/tenstorrent/luwen/blob/55e35616/tests/telem_test.rs#L1-L172)[tests/test_utils.rs 1-68](https://github.com/tenstorrent/luwen/blob/55e35616/tests/test_utils.rs#L1-L68)

## Best Practices

1.   **Minimize Telemetry Calls**: As noted in the source code, telemetry information is not cached and should not be called repeatedly. Consider caching the results if needed for multiple uses.

2.   **Error Handling**: Always handle potential errors when retrieving telemetry, as hardware communication can fail.

3.   **Type-Specific Telemetry**: Some telemetry fields may be specific to certain chip architectures. Check the architecture type before accessing architecture-specific fields.

4.   **Board Type Verification**: Use the `try_board_type()` method when you need to know if a board type was successfully determined, rather than defaulting to "UNSUPPORTED".
