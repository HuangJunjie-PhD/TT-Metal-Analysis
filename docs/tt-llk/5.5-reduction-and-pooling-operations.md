## Purpose and Scope

This page documents the reduction and pooling operations in the math unit stage of the LLK pipeline. Reduction operations aggregate tile data across specified dimensions (row, column, or scalar), while pooling operations perform MAX or AVG (average) pooling on the aggregated data. These operations are critical for neural network layers like global pooling, attention mechanisms, and embedding gradients.

Additionally, this page covers specialized reduction operations such as row reshuffling for embedding backward passes (`_calculate_reshuffle_rows_`), which performs scatter-add operations for gradient accumulation.

For element-wise operations without reduction, see page 5.2. For matrix multiplication operations, see page 5.4. For SFPU operations, see page 5.3. For unpacker configuration details, see page 4.

## Reduction Overview

Reduction operations compress tile data along one or more dimensions:

**Reduction Dimensions:**

| Dimension | Description | Input Shape | Output Shape | Use Case |
| --- | --- | --- | --- | --- |
| `REDUCE_ROW` | Reduce across columns within each row | 32×32 | 32×1 | Row-wise max/avg, produces 32 scalar values |
| `REDUCE_COL` | Reduce across rows within each column | 32×32 | 1×32 | Column-wise max/avg, produces 32 scalar values |
| `REDUCE_SCALAR` | Reduce entire tile to single value | 32×32 | 1×1 | Global max/avg, produces 1 scalar value |

**Pooling Types:**

*   **MAX Pooling** (`PoolType::MAX`): Uses `GMPOOL` instruction to find maximum value across the specified dimension
*   **AVG/SUM Pooling** (`PoolType::SUM`): Uses `GAPOOL` instruction to compute sum (average computed as sum/count in higher layers)

**Tile Size Support:**

The `narrow_tile` parameter controls whether to process partial tiles:

*   `narrow_tile = false` (default): Process full 4-face tiles (32×32)
*   `narrow_tile = true`: Process 2-face tiles (16×32 or 32×16)

The `num_faces` parameter specifies the number of 16×16 faces to process (1, 2, or 4).

## Core Math Reduce Function

The primary API for reduction operations is `_llk_math_reduce_()`:

**Template Parameters:**

*   `type`: `PoolType::MAX` or `PoolType::SUM` (average)
*   `dim`: `ReduceDim::REDUCE_ROW`, `REDUCE_COL`, or `REDUCE_SCALAR`
*   `is_fp32_dest_acc_en`: Enable 32-bit accumulator mode
*   `MATH_FIDELITY_DESC`: Fidelity descriptor for high-precision operations
*   `is_int_fpu_en`: Enable integer FPU operations (for int8/int32)
*   `enforce_fp32_accumulation`: Force 32-bit hi/lo accumulation path

**Function Parameters:**

*   `dst_index`: Destination register tile index
*   `narrow_tile`: Process narrow tiles (fewer than 4 faces)
*   `num_faces`: Number of 16×16 faces (1, 2, or 4)

## Row Reduction with Transpose

Row reduction (`REDUCE_ROW`) requires special handling due to hardware constraints. The reduction aggregates values across each row's columns, producing a 32×1 output. However, the hardware pooling instructions operate on transposed data, requiring a transpose-reduce-transpose sequence:

The transpose is performed at the unpacker level using the `within_face_16x16_transpose` flag, which rearranges data within each 16×16 face. After pooling, the result must be transposed back using `MOVD2B`, `TRNSPSRCB`, and `MOVB2D` instructions.

Sources: [tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 39-146](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L39-L146)

## FP32 Accumulation for Row Reduction

When `enforce_fp32_accumulation` is enabled, row reduction uses separate hi/lo 16-bit paths to maintain 32-bit precision:

This approach processes hi and lo 16-bit portions separately, transposing each independently before recombining in the destination register. Each `MOVB2D` writes 4 rows at a time across 4 groups (16 total rows).

## Address Modifier Configuration

Reduction operations use specific address modifier configurations to control counter advancement:

**Address Modifier Usage:**

| Modifier | Purpose | SrcA | SrcB | Dest | Used With |
| --- | --- | --- | --- | --- | --- |
| ADDR_MOD_0 | Face-by-face processing | +8 | 0 | +8 | GMPOOL/GAPOOL, MOVB2D |
| ADDR_MOD_1 | Reset after face, advance dest | clr+cr | clr+cr | +8 | Between faces |
| ADDR_MOD_2 | Transpose reconstruction | 0 | +8 | +8 | ELWADD (transpose) |
| ADDR_MOD_3 | No increment (SFPU) | 0 | 0 | 0 | SFPU loads |

## MOP Configuration for Reduction

The Macro Operation (MOP) programming for reduction depends on the dimension and fidelity settings:

**High Fidelity Mode:** When `num_fidelity_phases > 0` (derived from `MATH_FIDELITY_DESC`), the reduction operation runs multiple passes with incrementing fidelity phases. Each phase processes the input with progressively refined precision, accumulating results with different mantissa bit widths:

*   Phase 0: Initial computation with base precision
*   Phase 1: Additional precision bits added (mantissa masking adjusted)
*   Phase N: Final pass with full precision

The fidelity increment (1 or 2) is encoded in bits [3] of `MATH_FIDELITY_DESC`, while the number of phases is in bits [2:0]. This multi-pass approach improves numerical accuracy for operations requiring high precision.

Sources: [tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 35-36](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L35-L36)[tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 45-63](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L45-L63)[tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 299-417](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L299-L417)

## Column and Scalar Reduction

Column and scalar reduction follow a simpler path than row reduction since they don't require the transpose sequence:

**Column Reduction Output Layout:**

*   Input: 32 rows × 32 columns
*   Output: 1 row × 32 columns (stored in face 0, rows 0-15 and face 2, rows 0-15)

**Scalar Reduction Output Layout:**

*   Input: 32 rows × 32 columns
*   Output: Single value (typically in face 0, row 0, column 0)

## Integer Format Support

Reduction operations support integer data formats (Int8, Int32) through SFPU casting:

This approach uses SFPU load/store instructions with `InstrModLoadStore` enum to convert between Int32 (dest register native format) and Int8 (SrcB format). The `FP16A_FORCE_Enable` chicken bit forces the hardware to interpret Int8 datums as FP16A format in source registers, enabling integer reductions without separate hardware paths.

The same approach is used for the second row reduction when processing 4-face tiles (rows 16-31).

Sources: [tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 105-143](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L105-L143)[tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 230-264](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L230-L264)

## SFPU-Based Reduction Operations

The SFPU subsystem provides additional reduction primitives accessible through dedicated SFPU operations. These are referenced in the main SFPU header but implemented in specialized SFPU reduce files:

SFPU reductions complement the main FPU-based reductions by providing:

*   Custom reduction operations not supported by GMPOOL/GAPOOL
*   Fine-grained control over reduction logic
*   Integration with other SFPU operations in fused kernels

For details on SFPU operations, see [SFPU Operations Library](https://deepwiki.com/tenstorrent/tt-llk/5.3-sfpu-operations-library).
