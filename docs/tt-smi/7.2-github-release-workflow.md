## Purpose and Scope

This document describes the automated GitHub Actions release workflow defined in [.github/workflows/release.yml 1-454](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L1-L454) This workflow orchestrates the complete release process for TT-SMI, including version bumping, changelog generation, package building, Git tagging, GitHub release creation, and publication to PyPI. The workflow is triggered manually via `workflow_dispatch` and requires a version bump type selection (patch, minor, or major).

For package metadata configuration, see [Package Configuration (pyproject.toml)](https://deepwiki.com/tenstorrent/tt-smi/7.1-package-configuration-(pyproject.toml)). For details on Debian package building, see [Debian Package Building](https://deepwiki.com/tenstorrent/tt-smi/7.3-debian-package-building). For information about distribution channels and publishing, see [Distribution Channels](https://deepwiki.com/tenstorrent/tt-smi/7.4-distribution-channels).

**Sources:**[.github/workflows/release.yml 1-454](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L1-L454)

## Workflow Trigger and Inputs

The release workflow is triggered manually through GitHub's UI or API using the `workflow_dispatch` event. The workflow accepts a single input parameter that determines the type of version increment:

| Input Parameter | Type | Options | Default | Description |
| --- | --- | --- | --- | --- |
| `version_bump_type` | choice | patch, minor, major | patch | Semantic version component to increment |

The workflow uses the `bump2version` tool to apply the version increment to [pyproject.toml 3](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L3-L3)

**Sources:**[.github/workflows/release.yml 4-16](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L4-L16)

## Release Job Architecture

The release workflow consists of nine sequential and parallel jobs that orchestrate the complete release process. The following diagram shows the job dependency graph:

**Job Execution Flow:**

1.   **create-temp-branch** creates a temporary release candidate branch with timestamp
2.   **versionchange** bumps the version and commits to temp branch
3.   **changelogs** generates Debian changelog using `gbp dch`
4.   **build_all_depends** orchestrates parallel building of Debian and PyPI packages
5.   **tagrelease** creates a Git tag `vX.Y.Z` pointing to the final commit
6.   **generate-release** creates GitHub release with attached artifacts
7.   **mergeback** merges the temp branch back to main and deletes it
8.   **publish-to-testpypi** publishes to TestPyPI for validation
9.   **publish-to-pypi** publishes to production PyPI

**Sources:**[.github/workflows/release.yml 21-453](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L21-L453)

## Temporary Branch Creation

The workflow begins by creating a temporary release candidate branch to isolate release changes from the main branch during the multi-stage process.

### Branch Naming Convention

The temporary branch name follows this format:

```
rc-temp-{SHORT_COMMIT_HASH}-{TIMESTAMP}
```

Example: `rc-temp-a3f82b1-2025.01.15-14.30.45`

### Implementation

The `create-temp-branch` job generates the branch name and pushes it to the repository:

[.github/workflows/release.yml 41-62](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L41-L62)

The branch name is output as `temp_branch_ref` and used by all subsequent jobs via the `needs` context.

## Version Bumping Process

The `versionchange` job is responsible for incrementing the package version in [pyproject.toml 3](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L3-L3) and extracting version metadata for downstream jobs.

### Version Bumping Workflow

### Job Outputs

The `versionchange` job produces multiple outputs used by downstream jobs:

| Output | Source | Description |
| --- | --- | --- |
| `git_hash` | `git log --format="%H" -n 1` | Commit hash after version bump |
| `package_name` | `toml get project.name` | Package name (tt-smi) |
| `package_version` | `toml get project.version` | Original version |
| `package_version_new` | `toml get project.version` | New bumped version |
| `version_major` | `release-kit/semver@v2` | Major version number |
| `version_minor` | `release-kit/semver@v2` | Minor version number |
| `version_patch` | `release-kit/semver@v2` | Patch version number |
| `number_of_commits_since_tag` | `git rev-list` count | Commits since last tag |

### Version Bump Implementation

[.github/workflows/release.yml 64-160](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L64-L160)

The job uses:

*   **toml-cli**: Python package for reading/writing TOML files
*   **bump2version**: Automated version incrementing tool
*   **release-kit/semver@v2**: GitHub Action for semantic version parsing

## Git Tagging

The `tagrelease` job creates an annotated Git tag pointing to the final commit on the temporary branch after all version and changelog updates.

### Tagging Process

### Tag Implementation

[.github/workflows/release.yml 257-283](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L257-L283)

1.   Checks out the commit hash from the `changelogs` job output
2.   Creates a lightweight tag with prefix `v` (e.g., `v3.1.1`)
3.   Pushes the tag to the remote repository

### Tag Naming Convention

Tags follow the format: `vMAJOR.MINOR.PATCH`

## Merge Back to Main

The `mergeback` job completes the release process by merging the temporary release branch back to the main branch and cleaning up the temporary branch.

### Merge Process

### Merge Implementation

[.github/workflows/release.yml 362-386](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L362-L386)

The job performs:

1.   Checkout of the **original main branch** (not the temp branch)
2.   Rebase of main onto the temp branch to incorporate all release commits
3.   Pull with rebase to ensure up-to-date state
4.   Push to update the remote main branch
5.   Deletion of the remote temporary branch

## Permissions Model

Each job declares specific GitHub Actions permissions to follow the principle of least privilege:

### Permission Requirements by Job

| Job | contents | packages | id-token | Purpose |
| --- | --- | --- | --- | --- |
| create-temp-branch | write | - | - | Create and push branch |
| versionchange | write | - | - | Commit version changes |
| changelogs | write | write | write | Commit changelog, potential package access |
| tagrelease | write | write | write | Create and push tags |
| generate-release | write | write | write | Create GitHub release |
| mergeback | write | write | write | Merge and delete branches |
| publish-to-testpypi | - | - | write | OIDC authentication for PyPI |
| publish-to-pypi | - | - | write | OIDC authentication for PyPI |

### Permissions Explanation

*   **contents: write**: Allows pushing commits, tags, and creating releases
*   **packages: write**: Allows publishing to GitHub Packages (not used but declared)
*   **id-token: write**: Enables OIDC token generation for trusted publishing to PyPI

*   No Git tags are created
*   No GitHub release is published
*   No PyPI publication occurs

## Version String Format

The release workflow constructs version strings that differ between PyPI (Python) and Debian packaging:

### PyPI Version Format

Python packages use simple semantic versioning:

```
{MAJOR}.{MINOR}.{PATCH}
```

Example: `3.1.1`

This is stored in [pyproject.toml 3](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L3-L3) and used for:

*   Python wheel filename
*   PyPI package version
*   Git tag name (with `v` prefix)

### Debian Version Format

Debian packages use an extended version format that includes the commit count:

```
{MAJOR}.{MINOR}.{PATCH}-{NUMBER_OF_COMMITS_SINCE_TAG}
```

Example: `3.1.1-15`

This is generated by `gbp dch` in the changelog job and appears in:

*   `debian/changelog` entry
*   Debian package filename
*   Debian package metadata

### Version Extraction Process

[.github/workflows/release.yml 101-107](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L101-L107) - Extract current version from pyproject.toml using `toml get`

[.github/workflows/release.yml 115-121](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/release.yml#L115-L121) - Bump version using `bump2version`
