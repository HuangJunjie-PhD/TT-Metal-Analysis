The NOC (Network on Chip) emulation layer in `tt-emule` provides a bit-accurate functional model of the Tenstorrent NOC. It supports unicast, multicast, and bank-aware addressing (L1/DRAM) by translating 64-bit NOC addresses into host-accessible pointers.

## Emulation Model

`tt-emule` operates in a **slow-dispatch mode** where every NOC operation executes synchronously. This design choice collapses several hardware-specific behaviors into simplified software equivalents:

| Hardware Concept | Emulation Implementation |
| --- | --- |
| **Async / Posted Transfers** | Synchronous `std::memcpy`. All transactions are complete before the function returns [docs/noc-emulation.md 24-27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L24-L27) |
| **NOC Barriers** | No-ops. Since transfers are synchronous, `noc_async_read_barrier` and similar calls return immediately [docs/noc-emulation.md 25-28](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L25-L28) |
| **Transaction IDs (TRID)** | No-ops. State is accepted for API compatibility but has no side effects [docs/noc-emulation.md 29-31](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L29-L31) |
| **Dual-NOC (NOC0/NOC1)** | State (command buffers) is modeled per-NOC, but the transfer path is a single unified `memcpy`[docs/noc-emulation.md 32-37](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L32-L37) |

## NOC Address Encoding

A 64-bit NOC address packs destination coordinates and a local offset. The bitfields vary by architecture (e.g., Wormhole vs. Blackhole) and are defined in `noc_parameters.h`[include/jit_hw/api/dataflow/dataflow_api.h 54-64](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/dataflow_api.h#L54-L64)

*   **Unicast:**`x_end == x_start`, `y_end == y_start`.
*   **Multicast:** Start and end coordinates define an inclusive rectangle [docs/noc-emulation.md 47-64](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L47-L64)

The encoding generally follows: `[ y_start | x_start | y_end | x_end | L1 offset ]`[docs/noc-emulation.md 53](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/noc-emulation.md?plain=1#L53-L53)

### Address Helpers

*   `__emule_addr_to_offset(addr)`: Extracts the L1 offset from a host pointer [include/jit_hw/api/dataflow/dataflow_api.h 92-100](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/dataflow_api.h#L92-L100)
*   `__emule_local_l1_to_ptr(l1_addr)`: Converts a 32-bit firmware address or truncated host pointer back to a dereferenceable host pointer [include/jit_hw/api/dataflow/dataflow_api.h 111-120](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/dataflow_api.h#L111-L120)

## Primary API Surface

### Async Read/Write

The primary data movement functions are `noc_async_read` and `noc_async_write`. In emulation, these resolve the destination/source NOC address via `__emule_resolve_noc_addr` and perform a `memcpy`[include/jit_hw/api/dataflow/dataflow_api.h 207-220](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/dataflow_api.h#L207-L220)

`template <uint32_t max_page_size = NOC_MAX_BURST_SIZE>void noc_async_read(uint64_t src_noc_addr, uint32_t dst_local_l1_addr, uint32_t size, uint8_t noc = 0); template <uint32_t max_page_size = NOC_MAX_BURST_SIZE>void noc_async_write(uint32_t src_local_l1_addr, uint64_t dst_noc_addr, uint32_t size, uint8_t noc = 0);`
### Multicast

`__emule_multicast_write` emulates the hardware's multicast engine. It decodes the coordinate rectangle and iterates through the `core_map` to perform a `memcpy` to every target core [include/jit_hw/api/dataflow/dataflow_api.h 73-74](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/dataflow_api.h#L73-L74)

### Stateful 'with_state' Variants

Silicon uses command-buffer registers to cache NOC coordinates for "one-packet" or "with-state" transfers to reduce overhead. `tt-emule` mirrors this using `thread_local` arrays indexed by NOC ID [include/jit_hw/api/dataflow/noc.h 78-79](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc.h#L78-L79)

*   `set_async_read_state` / `async_read_with_state`[include/jit_hw/api/dataflow/noc.h 168-185](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc.h#L168-L185)
*   `set_async_write_state` / `async_write_with_state`[include/jit_hw/api/dataflow/noc.h 234-245](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc.h#L234-L245)

## Semaphore Operations

Semaphores are implemented using C++ atomics (`std::atomic<uint32_t>`) to ensure cross-thread visibility between emulated cores [include/jit_hw/api/dataflow/noc_semaphore.h 16-19](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc_semaphore.h#L16-L19)

| Function | Behavior |
| --- | --- |
| `noc_semaphore_inc` | Atomically increments a remote semaphore via its NOC address [include/jit_hw/api/dataflow/noc_semaphore.h 69](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc_semaphore.h#L69-L69) |
| `noc_semaphore_wait` | Busy-spins with hang detection until the semaphore matches a target value [include/jit_hw/api/dataflow/noc_semaphore.h 94-113](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc_semaphore.h#L94-L113) |
| `noc_semaphore_set` | Directly sets the semaphore value [include/jit_hw/api/dataflow/noc_semaphore.h 136-138](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc_semaphore.h#L136-L138) |

**Hang Detection:** To prevent infinite loops in JIT kernels, semaphore waits include a counter that aborts execution after 10,000,000 spins [include/jit_hw/api/dataflow/noc_semaphore.h 83-89](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc_semaphore.h#L83-L89)

## Type-Erasure System: Noc & noc_traits_t

The `Noc` class provides a modern C++ interface to the NOC, utilizing a trait-based system (`noc_traits_t`) to handle different endpoint types (Unicast, Multicast, or Allocator Banks) [include/jit_hw/api/dataflow/noc.h 83-115](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/noc.h#L83-L115)

### Supported Endpoints

1.   **UnicastEndpoint:** Resolves to a single core's L1 [include/jit_hw/api/dataflow/endpoints.h 26-31](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/endpoints.h#L26-L31)
2.   **MulticastEndpoint:** Encodes a range of cores [include/jit_hw/api/dataflow/endpoints.h 35-43](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/endpoints.h#L35-L43)
3.   **AllocatorBank:** Resolves a `bank_id` to the owning core's coordinates using the device's bank-to-NOC mapping tables (`dram_bank_to_noc_xy` or `l1_bank_to_noc_xy`) [include/jit_hw/api/dataflow/endpoints.h 47-57](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dataflow/endpoints.h#L47-L57)
