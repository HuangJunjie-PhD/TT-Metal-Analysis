## Purpose and Scope

This page documents the GitHub Actions workflow orchestration layer in `tt-metal`, focusing on workflow triggers, the `workflow_call` pattern, and how orchestration workflows coordinate test execution across diverse hardware. It covers:

*   **Trigger types**: `workflow_call`, `workflow_dispatch`, `schedule`, and `push`.
*   **Workflow orchestration**: Coordination by top-level workflows and implementation workflows (`*-impl.yaml`).
*   **Pipeline-select pattern**: Manual trigger mechanisms for targeted model and hardware validation.
*   **Dynamic matrix generation**: How test definitions are parsed from YAML files and filtered by SKUs or models.

## GitHub Actions Triggers

Workflows in the `tt-metal` repository utilize several primary trigger types to manage the lifecycle of a code change and hardware validation.

### Trigger Types

| Trigger | Usage Pattern | Examples in Codebase |
| --- | --- | --- |
| `workflow_call` | Reusable workflows called by other pipelines to modularize logic. | [.github/workflows/single-card-demo-tests-impl.yaml 4-38](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/single-card-demo-tests-impl.yaml#L4-L38) |

### Manual Dispatch and Inputs

Manual triggers allow developers to specify architectures, build types, and specific models. The `pipeline-select-galaxy.yaml` workflow allows selecting specific models for unit, demo, and perf tests (e.g., `llama3`, `deepseek_v3`, `flux1`) and choosing the `build-type` (e.g., `Release`, `Debug`, `ASan`, `TSan`) [[.github/workflows/pipeline-select-galaxy.yaml 138-147](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/pipeline-select-galaxy.yaml#L138-L147)]. It also provides toggles for specific hardware targets like `galaxy-e2e-wormhole` and `galaxy-e2e-blackhole`[[.github/workflows/pipeline-select-galaxy.yaml 120-129](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/pipeline-select-galaxy.yaml#L120-L129)].

## Reusable Workflow Pattern (`workflow_call`)

`tt-metal` uses the `workflow_call` trigger to modularize logic. Implementation workflows (suffixed with `-impl.yaml`) provide a standardized interface for orchestrators.

### Standardized Inputs

Most implementation workflows require:

*   `docker-image`: The pre-built container tag [[.github/workflows/single-card-demo-tests-impl.yaml 9-11](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/single-card-demo-tests-impl.yaml#L9-L11)].
*   `build-artifact-name` / `wheel-artifact-name`: Names of the pre-built artifacts to download [[.github/workflows/perf-device-models-impl.yaml 13-18](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/perf-device-models-impl.yaml#L13-L18)].
*   `arch`: Target architecture (e.g., `blackhole`, `wormhole_b0`) [[.github/workflows/single-card-demo-tests-impl.yaml 15-17](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/single-card-demo-tests-impl.yaml#L15-L17)].

### Orchestration Logic

Implementation workflows often include a `load-test-matrix` job that dynamically generates the execution strategy. For example, `t3000-demo-tests-impl.yaml` builds a matrix based on enabled SKUs and filters them by model [[.github/workflows/t3000-demo-tests-impl.yaml 29-70](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/t3000-demo-tests-impl.yaml#L29-L70)].

## Dynamic Matrix Generation and Filtering

Workflows generate test matrices at runtime by parsing YAML configuration files and applying filters.

### Matrix Generation Flow

1.   **Load Configuration**: Workflows read from centralized YAML files such as `t3k_demo_tests.yaml`[[.github/workflows/t3000-demo-tests-impl.yaml 34](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/t3000-demo-tests-impl.yaml#L34-L34)].
2.   **Filter by Inputs**: Scripts like `prepare_test_matrix.py` process these files against provided SKU labels [[.github/workflows/t3000-demo-tests-impl.yaml 57-60](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/t3000-demo-tests-impl.yaml#L57-L60)].
3.   **Verify Budget**: The `verify_time_budget.py` script ensures that defined test timeouts do not exceed the allowed limits in `time_budget.yaml`[[.github/workflows/t3000-demo-tests-impl.yaml 50-53](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[.github/workflows/t3000-demo-tests-impl.yaml#L50-L53)[.github/time_budget.yaml 1-262](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/time_budget.yaml#L1-L262)].
