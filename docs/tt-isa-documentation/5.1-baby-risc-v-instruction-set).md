## Overview

The Baby RISC-V cores implement a 32-bit RISC-V instruction set architecture optimized for control plane operations and orchestration of specialized accelerators, rather than general-purpose compute throughput. Each Tensix tile contains five Baby RISC-V cores (B, T0, T1, T2, NC), while each Ethernet tile contains one (E).

The cores execute one instruction per cycle at 1 GHz (Wormhole B0) or 1.35 GHz (Blackhole A0), using an in-order pipeline. Their primary role is to coordinate data movement via NoC and orchestrate computation on the Tensix Coprocessor.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 1-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L1-L13)[WormholeB0/TensixTile/README.md 1-16](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/README.md?plain=1#L1-L16)

## RV32IM Base Specification

### Integer Base Instruction Set (RV32I)

The Baby RISC-V cores implement the **RV32I Base Integer Instruction Set, Version 2.1**, which includes:

*   **Arithmetic**: `add`, `sub`, `addi`, `lui`, `auipc`
*   **Logical**: `and`, `or`, `xor`, `andi`, `ori`, `xori`
*   **Shifts**: `sll`, `srl`, `sra`, `slli`, `srli`, `srai`
*   **Comparisons**: `slt`, `sltu`, `slti`, `sltiu`
*   **Loads**: `lb`, `lh`, `lw`, `lbu`, `lhu`
*   **Stores**: `sb`, `sh`, `sw`
*   **Branches**: `beq`, `bne`, `blt`, `bge`, `bltu`, `bgeu`
*   **Jumps**: `jal`, `jalr`
*   **System**: `ecall`, `ebreak`, `fence` (with caveats, see below)

### Integer Multiply/Divide Extension (M)

The **M Extension for Integer Multiplication and Division, Version 2.0** provides:

*   **Multiply**: `mul`, `mulh`, `mulhsu`, `mulhu`
*   **Divide**: `div`, `divu`, `rem`, `remu`

**Latencies**:

*   Multiply: 2 cycles
*   Divide: 2-33 cycles (depends on operand values) 
    *   Division by 0 or 1: 2 cycles
    *   Signed division of -2³¹ by -1: 2 cycles
    *   Other cases: 6-33 cycles

### Invalid Instruction Handling

**Unsupported or invalid RISC-V instructions result in UndefinedBehavior.** In practice, they are silently executed as if they were some other instruction (usually, but not always, `nop`), but software must not depend on this behavior.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 15-26](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L15-L26)[WormholeB0/TensixTile/BabyRISCV/README.md 44-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L44-L51)

## Critical Instruction Set Caveats

### FENCE Instruction Caveat

**The `fence` instruction is executed as if it were `nop`** and **cannot be used to enforce memory ordering** between Baby RISC-V cores or between a Baby RISC-V core and external agents (NoC, Tensix coprocessor).

**Rationale**: The Baby RISC-V Load/Store Unit can significantly reorder memory operations internally. While `fence` is defined in RV32I, the hardware does not provide the necessary synchronization primitives for full fence semantics.

**Workarounds** for synchronization:

| Mechanism | Use Case | Location |
| --- | --- | --- |
| **Mailboxes** | Inter-core communication (4×4 FIFO matrix between B/T0/T1/T2) | See [Inter-Core Communication](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.3-inter-core-communication) |
| **Tensix Semaphores** | Synchronization with Tensix Coprocessor execution | See [Sync Unit Semaphores](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Sync%20Unit%20Semaphores) |
| **TTSync** | Thread-to-thread synchronization within a tile | See [Inter-Core Communication](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.3-inter-core-communication) |
| **PCBufs** | RISCV B controlling T0/T1/T2 program counters | See [Inter-Core Communication](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.3-inter-core-communication) |

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 18](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L18-L18)

### EBREAK and ECALL Instructions

Both `ebreak` and `ecall` instructions **trigger a debug pause**, identical to a hardware breakpoint. The paused core will not resume execution until an external debugger explicitly resumes it.

**Execution Behavior**:

1.   Core completes execution of `ebreak`/`ecall` instruction
2.   Core pauses before fetching next instruction
3.   Core enters halted state visible to debuggers via GDB/Debug interface
4.   **No trap handler is invoked** (Baby RISC-V cores do not support traps/exceptions)
5.   Core remains paused indefinitely until debugger issues resume command

**Use Cases**:

*   Explicit breakpoints in debug builds
*   Triggering debugger attachment on error conditions
*   **Not recommended for production code** (core will hang without attached debugger)

**Alternative for Production**: Use polling on status registers or interrupts (via PIC) instead of relying on `ebreak`/`ecall`.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 19](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L19-L19)[WormholeB0/TensixTile/BabyRISCV/README.md 114](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L114-L114)

## .ttinsn Extension (Tensix Instruction Push)

The `.ttinsn` instruction set extension provides a mechanism for Baby RISC-V cores to efficiently push instructions to the Tensix Coprocessor. This is implemented via **memory-mapped store operations** rather than as traditional executable instructions.

### Memory-Mapped Instruction Push Architecture

### Core-Specific Push Mapping

| Core | `INSTRN_BUF_BASE` (0xFFE4_0000) | `INSTRN1_BUF_BASE` (0xFFE5_0000) | `INSTRN2_BUF_BASE` (0xFFE6_0000) |
| --- | --- | --- | --- |
| **RISCV B** | → T0 (after MOP Expander) | → T1 (after MOP Expander) | → T2 (after MOP Expander) |
| **RISCV T0** | → T0 (before MOP Expander) | Unmapped | Unmapped |
| **RISCV T1** | → T1 (before MOP Expander) | Unmapped | Unmapped |
| **RISCV T2** | → T2 (before MOP Expander) | Unmapped | Unmapped |
| **RISCV NC** | Push via debug bus only | Unmapped | Unmapped |

**MOP Expander Routing**:

*   **Before MOP**: Instructions pushed by T0/T1/T2 to their own threads enter the instruction FIFO before the [MOP Expander](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.1-instruction-pipeline-and-frontend), allowing use of macro-operation expansion
*   **After MOP**: Instructions pushed by RISCV B enter after MOP Expander, requiring pre-expanded instruction sequences

### Usage Pattern

**Important Notes**:

*   Store size does not matter (byte, halfword, word stores all work)
*   Store address offset within the 64 KiB region does not matter
*   Only the base address determines which thread receives the instruction
*   Instructions must be properly encoded Tensix instructions (see sections [5.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.2-matrix-unit-instructions) through [5.8](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.8-configuration-instructions))

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 23](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L23-L23)[WormholeB0/TensixTile/BabyRISCV/README.md 93-95](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L93-L95)

## Blackhole A0 Floating-Point Extensions

Blackhole A0 introduces partial support for RISC-V floating-point extensions. **These instructions are not available in Wormhole B0.**

### F Extension (Single-Precision Floating-Point)

Blackhole implements **partial support** for the F Extension for Single-Precision Floating-Point, Version 2.2.

#### Implemented Instructions

| Category | Instructions |
| --- | --- |
| **Arithmetic** | `fadd.s`, `fsub.s`, `fmul.s` |
| **Fused Multiply-Add** | `fmadd.s`, `fmsub.s`, `fnmsub.s`, `fnmadd.s` |
| **Comparison** | `feq.s`, `flt.s`, `fle.s` |
| **Move/Convert** | `fmv.w.x`, `fmv.x.w`, `fcvt.w.s`, `fcvt.wu.s`, `fcvt.s.w`, `fcvt.s.wu` |
| **Sign Injection** | `fsgnj.s`, `fsgnjn.s`, `fsgnjx.s` |
| **Min/Max** | `fmin.s`, `fmax.s` |
| **Classification** | `fclass.s` |

#### Not Implemented

*   `fdiv.s` (division)
*   `fsqrt.s` (square root)
*   Rounding mode control via `frm` CSR
*   `fflags` CSR (though `fcsr` exists and contains relevant bits)

### IEEE754 Conformance and Divergence

#### Major Differences from IEEE754

| Aspect | IEEE754 | Blackhole Behavior |
| --- | --- | --- |
| **Denormal inputs** | Normal handling | **Flushed to signed zero** |
| **Denormal outputs** | Normal handling | **Flushed to signed zero (after rounding)** |
| **Product precision** | Infinite | **48-bit mantissa (4 extra precision bits)** |
| **Addition alignment** | Add precision to `z` | **Remove precision from product (shift right 23)** |
| **Rounding mode** | Configurable | **Always round-to-nearest, ties-to-even (RNE)** |
| **Overflow handling** | Correct | **If `x*y` alone overflows → infinity** |
| **Underflow handling** | Correct | **If `x*y` alone underflows → treated as exact zero** |
| **Output NaN** | Implementation-defined | **Always canonical `0x7fc00000`** |

#### Partial vs Complete Fusion

The Blackhole FMA is **partially fused**:

*   ✓ Product has higher precision than FP32 (48-bit mantissa)
*   ✗ Product does not have infinite precision
*   ✓ Single rounding step
*   ✗ But not at infinite precision

**Practical Impact**:

*   More accurate than separate `fmul.s` followed by `fadd.s`
*   Less accurate than fully compliant IEEE754 FMA
*   When multiplying by 1 or adding 0, equivalent to standalone operation (modulo denormal handling)

### Bit-Precise Software Model

A bit-precise software model is provided at [Miscellaneous/FMA/fma.c](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c) for validation and testing:

The model exactly reproduces:

*   Denormal flushing (input and output)
*   Partial fusion (23-bit precision reduction on product)
*   Rounding behavior
*   NaN canonicalization
*   Overflow/underflow edge cases

Sources: [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 17-23](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L17-L23)[Miscellaneous/FMA/README.md 3-31](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/README.md?plain=1#L3-L31)[Miscellaneous/FMA/fma.c 102-189](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c#L102-L189)

### Zfh Extension (Half-Precision Floating-Point)

Blackhole implements **partial support** for the Zfh Extension for Half-Precision Floating-Point, Version 1.0:

*   All half-precision equivalents of single-precision instructions (`.h` suffix instead of `.s`)
*   Same IEEE754 conformance caveats as F Extension
*   **BF16 Mode**: Custom CSR bit switches `.h` instructions to operate on BF16 (BFloat16) instead of FP16

**BF16 vs FP16**:

| Format | Sign | Exponent | Mantissa | Range | Precision |
| --- | --- | --- | --- | --- | --- |
| **FP32** | 1 bit | 8 bits | 23 bits | ±3.4×10³⁸ | ~7 decimal digits |
| **FP16** | 1 bit | 5 bits | 10 bits | ±6.5×10⁴ | ~3 decimal digits |
| **BF16** | 1 bit | 8 bits | 7 bits | ±3.4×10³⁸ | ~2 decimal digits |

**BF16 Advantages**:

*   Same exponent range as FP32 (reduces overflow/underflow issues)
*   Direct truncation from FP32 (simple conversion)
*   Better training stability for neural networks

Sources: [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 24-26](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L24-L26)

## Additional Blackhole A0 Extensions

### Zicsr (Control and Status Registers)

Blackhole implements the Zicsr Extension (Version 2.0), providing CSR manipulation instructions: `csrrw`, `csrrs`, `csrrc`, `csrrwi`, `csrrsi`, `csrrci`.

See separate CSR documentation for details on available CSRs (including `fcsr`, `vtype`, `vl`, etc.).

### Zaamo (Atomic Memory Operations)

Blackhole implements the Zaamo Extension (part of A Extension, Version 2.1), providing atomic read-modify-write instructions:

**Supported**: `amoadd.w`, `amoswap.w`, `amoxor.w`, `amoor.w`, `amoand.w`, `amomin.w`, `amomax.w`, `amominu.w`, `amomaxu.w`

**Restrictions**:

*   Can only target local L1 scratchpad memory
*   Cannot target MMIO regions or remote tiles
*   `aq` and `rl` bits are always treated as set (conformant but slight performance cost)

**Not Supported**: Load-reserved/store-conditional (`lr.w`, `sc.w`) from Zalrsc extension

Sources: [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 9-10](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L9-L10)[BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 55-59](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L55-L59)

### Bit Manipulation Extensions (Zba, Zbb, partial Zbkb)

**Zba (Address Generation)**: `sh1add`, `sh2add`, `sh3add`

**Zbb (Basic Bit Manipulation)**: `andn`, `clz`, `cpop`, `ctz`, `max`, `maxu`, `min`, `minu`, `orc.b`, `orn`, `rev8`, `rol`, `ror`, `rori`, `sext.b`, `sext.h`, `xnor`, `zext.h`

**Zbkb (Cryptography, partial)**: `pack`, `brev8` (but **not**`packh`, `zip`, `unzip`)

**Non-standard**: `grevi` (from Bitmanip 0.94-draft, provides superset of `rev8` and `brev8` functionality)

Sources: [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 12-15](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L12-L15)[BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 61-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L61-L74)

### V Extension (Vector Operations, RISCV T2 only)

**RISCV T2** in Blackhole implements partial support for V Extension (Version 1.0):

**Configuration**:

*   `VLEN = 128` (16-byte vector registers)
*   Maximum element width: 32 bits
*   Supports `SEW = 8, 16, 32`
*   Supports `LMUL = 1/8, 1/4, 1/2, 1, 2, 4, 8`

**Implemented**: Most vector instructions (integer arithmetic, FP arithmetic, loads/stores, permutes, reductions)

**Not Implemented**:

*   Integer division/remainder: `vdiv`, `vdivu`, `vrem`, `vremu`
*   FP division/square root: `vfdiv`, `vfrdiv`, `vfsqrt`, `vfrsqrt7`, `vfrec7`

**Floating-Point Caveats**: Vector FP instructions have same denormal flushing and FMA semantics as scalar FP instructions.

**Performance Characteristics**:

*   Most instructions have false dependency on destination register
*   Fractional LMUL has false dependency bugs (requires careful register assignment)
*   `LMUL > 1` splits into multiple micro-ops
*   Some instructions split further into one micro-op per element

Sources: [BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 28-35](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L28-L35)[BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md 75-82](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/BabyRISCV/InstructionSet.md?plain=1#L75-L82)

## Instruction Latency Summary

### Integer and Control Flow

| Instruction Type | Latency (cycles) | Throughput | Notes |
| --- | --- | --- | --- |
| Integer ALU | 1 | 1/cycle | Forwarding allows back-to-back dependent ops |
| `mul`, `mulh`, etc. | 2 | 1/2 cycles | Blocks Integer Unit for 2 cycles |
| `div`, `divu`, `rem`, `remu` | 2-33 | 1/2-33 cycles | Depends on operand values |
| Branch (predicted) | 1 | 1/cycle | Prediction in Frontend |
| Branch (mispredicted) | 3 effective | — | Includes 2-cycle bubble |
| `jal`, `jalr` | 1 | 1/cycle | Unconditional jumps |

### Memory Operations

| Operation | Target | Latency (cycles) | Max In-Flight | Throughput |
| --- | --- | --- | --- | --- |
| **Load** | Local data RAM | 2 | 8 | 1/cycle (if latency hidden) |
| **Load** | Mailboxes, PCBufs, TTSync, Semaphores | ≥3 | 4 (aggregate) | 1/cycle or 4/(N-1) |
| **Load** | Tensix GPRs, Backend Config | ≥4 | 4 (aggregate) | 4/(N-1) cycles |
| **Load** | TDMA, NoC, PIC, Tile Control | ≥7 | 4 (aggregate) | 4/(N-1) cycles |
| **Load** | L1 scratchpad | ≥8 | 4 (aggregate) | 4/(N-1) cycles |
| **Store** | L1 scratchpad | ~5 | — | 1/5 cycles max |
| **Store** | Other regions | ~1 | — | 1/cycle |

**Note**: Throughput of `4/(N-1)` means 4 loads complete every `N-1` cycles when latency is `N` cycles.

## Exception and Fault Handling

The Baby RISC-V cores **cannot fault** in the traditional sense. There are no trap handlers, exception vectors, or interrupt support.

### Non-Faulting Behavior Table

| Condition | Standard RISC-V Behavior | Baby RISC-V Behavior |
| --- | --- | --- |
| **Unaligned load/store** | Trap (misaligned address exception) | **Silently round down to aligned address** (NonContractualBehavior) |
| **Load from unmapped address** | Trap (load access fault) | **Stall forever** (waiting for response) |
| **Store to unmapped address** | Trap (store access fault) | **Might stall forever or silently discard** |
| **`ecall`** | Trap (environment call) | **Pause core** (requires debugger) |
| **`ebreak`** | Trap (breakpoint) | **Pause core** (requires debugger) |
| **Invalid instruction** | Trap (illegal instruction) | **UndefinedBehavior** (usually executes as `nop`) |
| **Integer overflow** | No trap (wraps) | **Wraps** (standard behavior) |
| **Division by zero** | No trap (defined result) | **Returns defined result** (per M Extension) |

### Interrupt Handling

**Wormhole and Blackhole do not support interrupts**. The [Programmable Interrupt Controller (PIC)](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.4-programmable-interrupt-controller-(pic)) provides:

*   IRQ status tracking (11 IRQ sources)
*   Atomic set/clear operations
*   Optional clock gating for RISCV NC/E

However, **IRQs cannot interrupt execution**. Software must poll `PIC_STATUS` to check for pending IRQs.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 108-116](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L108-L116)[WormholeB0/TensixTile/PIC.md 1-8](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/PIC.md?plain=1#L1-L8)

### Tensix Vector Unit (SFPU) FMA Comparison

While Baby RISC-V FMA instructions are only in Blackhole, both architectures provide FMA in the Tensix Vector Unit:

| Aspect | Wormhole SFPU `SFPMAD` | Blackhole Baby RISC-V `fmadd.s` | Blackhole SFPU `SFPMAD` |
| --- | --- | --- | --- |
| **Availability** | Tensix Vector Unit only | All Baby RISC-V cores | Tensix Vector Unit only |
| **Width** | 32 lanes (SIMD) | Scalar | 32 lanes (SIMD) |
| **FMA semantics** | Partially fused (buggy) | Partially fused | Partially fused (improved) |

## Core Differences: Tensix vs Ethernet Tiles

### Tensix Tile Baby RISC-V Cores

| Core | Role | Local DRAM | Local IRAM | Tensix Access |
| --- | --- | --- | --- | --- |
| **RISCV B** (Brisc) | Data movement, orchestration | 4 KiB | 2 KiB cache | Push after MOP |
| **RISCV T0** (Trisc0) | UNPACK kernel phase | 2 KiB | 2 KiB cache | Push before MOP (T0) |
| **RISCV T1** (Trisc1) | MATH kernel phase | 2 KiB | ½ KiB cache | Push before MOP (T1) |
| **RISCV T2** (Trisc2) | PACK kernel phase | 2 KiB | 2 KiB cache | Push before MOP (T2) |
| **RISCV NC** (NCrisc) | Data movement | 4 KiB | ½ KiB cache + 16 KiB RAM | Debug bus only |

**Shared Resources**: 1464 KiB L1 scratchpad, NoC interfaces, Tensix Coprocessor

### Ethernet Tile Baby RISC-V Core

| Core | Role | Local DRAM | Local IRAM | Special Access |
| --- | --- | --- | --- | --- |
| **RISCV E** (Erisc) | Data movement, Ethernet control | 4 KiB (first 1.5 KiB reserved) | 16 KiB RAM | Ethernet TX/RX queues |

**Shared Resources**: 256 KiB L1 scratchpad, NoC interfaces, Ethernet MAC/PCS

**Key Difference**: Ethernet tile's RISCV E is cooperatively shared between Tenstorrent firmware (link training, baseline data movement service) and customer code, whereas Tensix cores run exclusively customer code.

## Summary

The Baby RISC-V instruction set provides a control-oriented ISA for orchestrating high-throughput accelerators. Key takeaways:

1.   **RV32IM base** is universally implemented with two caveats (`fence` as `nop`, `ebreak`/`ecall` pause core)
2.   **`.ttinsn` extension** enables efficient Tensix instruction push via memory-mapped stores
3.   **Blackhole adds floating-point** with partial IEEE754 conformance (denormal flushing, partially-fused FMA)
4.   **No exceptions/interrupts** – cores handle errors by stalling or undefined behavior
5.   **Architecture differences** between Wormhole and Blackhole are significant (FP support, bit manipulation, vector ops)

For compute-intensive operations, use the [Tensix Coprocessor](https://deepwiki.com/tenstorrent/tt-isa-documentation/4-tensix-coprocessor-architecture) instructions documented in sections [5.2](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.2-matrix-unit-instructions) through [5.8](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.8-configuration-instructions), not Baby RISC-V scalar operations.
