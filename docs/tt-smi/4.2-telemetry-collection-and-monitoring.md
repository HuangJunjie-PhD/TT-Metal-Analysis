## Overview

This page documents the telemetry collection and monitoring system in TT-SMI. Telemetry refers to real-time performance and operational metrics collected from Tenstorrent hardware, including voltage, current, power, temperature, clock frequencies, and fan speeds. TT-SMI collects telemetry through hardware-specific interfaces and provides both snapshot and continuous monitoring capabilities.

Telemetry collection differs significantly between hardware platforms, with Wormhole using SMBus-based telemetry and Blackhole using TAG-based telemetry formats. This page covers the telemetry data sources, collection mechanisms, platform-specific differences, and update mechanisms.

For related topics, see page 4.1 for platform support details, page 4.3 for firmware version reporting, and page 3.3 for the TUI telemetry display.

## Telemetry Data Sources

TT-SMI collects telemetry through platform-specific hardware interfaces provided by the pyluwen library.

### SMBus Telemetry Interface (Wormhole)

**SMBus Telemetry Register List**

Wormhole devices expose telemetry through SMBus registers defined in `SMBUS_TELEMETRY_LIST`. The list contains 54 telemetry fields covering firmware versions, operational metrics, status registers, and hardware health indicators.

Key telemetry registers include:

*   Power and voltage: `VCORE`, `TDP`, `TDC`, `VDD_LIMITS`, `INPUT_POWER`
*   Temperature: `ASIC_TEMPERATURE`, `VREG_TEMPERATURE`, `BOARD_TEMPERATURE`, `ASIC_TMON0`, `ASIC_TMON1`
*   Clock frequencies: `AICLK`, `AXICLK`, `ARCCLK`
*   Fan control: `FAN_SPEED`, `FAN_RPM`
*   Status and health: `ARC0_HEALTH`, `ARC1_HEALTH`, `ARC2_HEALTH`, `ARC3_HEALTH`, `FAULTS`, `THROTTLER`
*   Memory: `DDR_SPEED`, `DDR_STATUS`
*   Limits and thresholds: `THM_LIMITS`, `BOARD_POWER_LIMIT`

Sources: [tt_smi/constants.py 8-64](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L64)

### TAG-Based Telemetry Interface (Blackhole)

**Blackhole Telemetry Register List**

Blackhole devices use a TAG-based telemetry format defined in `BH_TELEMETRY_LIST`. The list contains 33 TAG-prefixed fields with a subset of Wormhole's telemetry capabilities.

Key TAG telemetry registers include:

*   Power and voltage: `TAG_VCORE`, `TAG_TDP`, `TAG_TDC`, `TAG_VDD_LIMITS`
*   Temperature: `TAG_ASIC_TEMPERATURE`, `TAG_VREG_TEMPERATURE`, `TAG_BOARD_TEMPERATURE`, `TAG_THM_LIMITS`
*   Clock frequencies: `TAG_AICLK`, `TAG_AXICLK`, `TAG_ARCCLK`, `TAG_L2CPUCLK0-3`
*   Fan control: `TAG_FAN_SPEED`, `TAG_FAN_RPM`
*   Memory: `TAG_DDR_STATUS`, `TAG_DDR_SPEED`
*   Firmware versions: `TAG_ETH_FW_VERSION`, `TAG_CM_FW_VERSION`, `TAG_BM_APP_FW_VERSION`, `TAG_BM_BL_FW_VERSION`
*   Health indicators: `TAG_TIMER_HEARTBEAT`

## Platform-Specific Telemetry Processing

TT-SMI implements separate telemetry processing functions for Wormhole and Blackhole due to differences in register formats and value encoding.

### Wormhole Telemetry Processing

**get_wh_chip_telemetry() Implementation**

Wormhole processing details:

*   Temperature: 16-bit unsigned value divided by 16 for °C
*   Voltage: Millivolts divided by 1000 for V
*   Current and Power: Lower 16 bits of TDC/TDP registers
*   Heartbeat: ARC3_HEALTH divided by 5 (~2 updates per second)

Sources: [tt_smi/tt_smi_backend.py 471-499](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L471-L499)

### Blackhole Telemetry Processing

**get_bh_chip_telemetry() Implementation**

Blackhole processing differences:

*   Temperature: Signed 16.16 fixed-point format requiring `convert_signed_16_16_to_float()`
*   Fan Speed: Converted from RPM (capped at 5000) to percentage by dividing by 50
*   Heartbeat: TIMER_HEARTBEAT divided by 6 (~2 updates per second)

**Signed 16.16 Fixed-Point Conversion**

The `convert_signed_16_16_to_float()` method handles Blackhole's temperature format:

## Telemetry Update Mechanism

TT-SMI provides real-time telemetry monitoring through a background update thread in the TUI application.

### Continuous Update Flow

**TUI Telemetry Update Thread**

### Update Implementation Details

The `update_telem()` method refreshes telemetry for all devices:

```
def update_telem(self):
    """Update telemetry in a given interval"""
    for i, _ in enumerate(self.devices):
        self.smbus_telem_info[i] = self.get_smbus_board_info(i)
        self.device_telemetrys[i] = self.get_chip_telemetry(i)
```

Update timing:

*   Interval: `GUI_INTERVAL_TIME = 0.1` seconds (100ms)
*   Thread managed by TTSMI application class
*   Runs continuously while TUI is active
*   Each update processes all detected devices sequentially

## Chip Limits and Thresholds

TT-SMI reads operational limits from hardware telemetry registers to provide context for telemetry values. These limits are displayed alongside metrics in the TUI.

### Limits Data Structure

Limits are defined in the `LIMITS` constant list:

| Limit Key | Description | WH Source | BH Source |
| --- | --- | --- | --- |
| `vdd_min` | Minimum core voltage (V) | `VDD_LIMITS`& 0xFFFF / 1000 | `VDD_LIMITS`& 0xFFFF / 1000 |
| `vdd_max` | Maximum core voltage (V) | `VDD_LIMITS`>> 16 / 1000 | `VDD_LIMITS`>> 16 / 1000 |
| `tdp_limit` | Thermal Design Power limit (W) | `TDP`>> 16 | `TDP_LIMIT_MAX` |
| `tdc_limit` | Thermal Design Current limit (A) | `TDC`>> 16 | `TDC_LIMIT_MAX` |
| `asic_fmax` | Maximum ASIC frequency (MHz) | `AICLK`>> 16 | `AICLK_LIMIT_MAX` |
| `therm_trip_l1_limit` | Thermal throttle trigger (°C) | `THM_LIMITS`>> 16 | `THM_LIMIT_THROTTLE` |
| `thm_limit` | Thermal shutdown limit (°C) | `THM_LIMITS`& 0xFFFF | `THM_LIMITS` |
| `bus_peak_limit` | PCIe bus power limit | Not implemented | Not implemented |

Sources: [tt_smi/constants.py 112-121](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L112-L121)

### Platform-Specific Limit Extraction

**Wormhole Limits - get_wh_chip_limits()**

Wormhole packs multiple limits into 32-bit registers with upper/lower 16-bit fields:

*   `VDD_LIMITS`: [31:16] = vdd_max, [15:0] = vdd_min
*   `TDP`: [31:16] = tdp_limit, [15:0] = current TDP
*   `TDC`: [31:16] = tdc_limit, [15:0] = current TDC
*   `AICLK`: [31:16] = asic_fmax, [15:0] = current AICLK
*   `THM_LIMITS`: [31:16] = therm_trip_l1_limit, [15:0] = thm_limit

Sources: [tt_smi/tt_smi_backend.py 528-556](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L528-L556)

**Blackhole Limits - get_bh_chip_limits()**

Blackhole uses separate dedicated limit registers:

*   `VDD_LIMITS`: Same 16-bit packing as Wormhole
*   `TDP_LIMIT_MAX`: Dedicated TDP limit register
*   `TDC_LIMIT_MAX`: Dedicated TDC limit register
*   `AICLK_LIMIT_MAX`: Dedicated frequency limit register
*   `THM_LIMIT_THROTTLE`: Dedicated throttle threshold register
*   `THM_LIMITS`: Thermal shutdown limit

Sources: [tt_smi/tt_smi_backend.py 558-586](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L558-L586)

## Using the Device Information and Telemetry Features

TT-SMI provides two interfaces for accessing device information and telemetry:

### Command Line Interface (CLI)

Use the `--snapshot` option to capture a point-in-time snapshot of device information and telemetry:

```
tt-smi --snapshot
```

To save the snapshot to a file:

```
tt-smi --filename output.json
```

Sources: [tt_smi/tt_smi.py 639-651](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L639-L651)[tt_smi/tt_smi.py 739-750](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L739-L750)

### Text User Interface (TUI)

The TUI provides three main tabs for monitoring:

1.   **Device Information (Tab 1)**: Static device details
2.   **Telemetry (Tab 2)**: Real-time performance metrics updated every 100ms
3.   **Firmware Version (Tab 3)**: Firmware details

Navigation:

*   Press `1`, `2`, or `3` to switch between tabs
*   Press `q` to quit
*   Press `h` for help

The telemetry tab automatically polls hardware at regular intervals to provide up-to-date metrics and visual formatting that highlights values exceeding defined limits.

## Data Visualization and Thresholds

TT-SMI's TUI employs color-coding to visually indicate the status of telemetry metrics in relation to their thresholds:

*   **Green**: Values within safe operating range
*   **Red/Attention**: Values exceeding recommended limits
*   **Yellow**: Threshold values for reference
*   **Gray**: Unavailable or not applicable data

Each telemetry metric is displayed with both its current value and the threshold value in the format "current / threshold".

Sources: [tt_smi/tt_smi.py 212-366](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi.py#L212-L366)

## Remote vs. Local Device Handling

For Wormhole devices, TT-SMI distinguishes between local and remote ASICs on boards like n300:

*   Local devices show complete telemetry and PCIe information
*   Remote devices show limited information with some fields marked as "N/A"
*   Board type displays with an "L" (Local) or "R" (Remote) suffix
