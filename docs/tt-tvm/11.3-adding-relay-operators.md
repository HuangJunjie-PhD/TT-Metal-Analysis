This guide explains how to add new operators to the Relay intermediate representation in TVM. It covers the complete workflow from defining operator attributes and type relations in C++ to exposing user-facing Python APIs and implementing backend-specific compute strategies.

For information about adding backend-specific code generation, see [Adding Backend Support](https://deepwiki.com/tenstorrent/tt-tvm/11.4-adding-backend-support). For details on Relay's operator library and optimization passes, see [Relay IR System](https://deepwiki.com/tenstorrent/tt-tvm/4-relay-ir-system).

## Core Components

### Operator Attributes

Attributes define the configurable parameters for an operator. They are declared as C++ structs inheriting from `tvm::AttrsNode`.

| Component | Location | Purpose |
| --- | --- | --- |
| Attribute Struct | `include/tvm/relay/attrs/*.h` | Define operator parameters |
| Registration Macro | `TVM_DECLARE_ATTRS` | Register attribute fields |
| Field Declarations | `TVM_ATTR_FIELD` | Document and set defaults |

**Example: Cast Operator Attributes**

**Sources:**[include/tvm/relay/attrs/transform.h 63-70](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/relay/attrs/transform.h#L63-L70)[src/relay/op/tensor/transform.cc 131-132](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L131-L132)

### Type Relations

Type relations perform shape and type inference for operators. They implement the `TypeRelation` interface to deduce output types from input types and attributes.

**Sources:**[src/relay/op/tensor/transform.cc 133-145](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L133-L145)[src/relay/op/tensor/transform.cc 253-286](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L253-L286)

### Operator Registration

The `RELAY_REGISTER_OP` macro registers an operator with the Relay system, associating it with attributes, type relations, compute functions, and optimization patterns.

| Attribute | Purpose |
| --- | --- |
| `.describe()` | Documentation string |
| `.set_num_inputs()` | Number of input tensors |
| `.add_argument()` | Input parameter documentation |
| `.set_support_level()` | Operator support priority (1-10) |
| `.add_type_rel()` | Type relation function |
| `.set_attr<FTVMCompute>()` | Compute implementation |
| `.set_attr<TOpPattern>()` | Operator pattern classification |
| `.set_attr<FInferCorrectLayout>()` | Layout transformation rules |

**Sources:**[src/relay/op/tensor/transform.cc 164-176](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L164-L176)[src/relay/op/nn/nn.cc 91-106](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/nn/nn.cc#L91-L106)

## Advanced Topics

### Layout Transformations

Operators that are layout-sensitive (e.g., convolution, pooling) should implement `FInferCorrectLayout` to handle automatic layout transformations.

**Key functions:**

*   `InferCorrectLayoutOutput`: Return type for layout inference
*   `BijectiveLayout`: Bidirectional layout mappings
*   `AlterOpLayout`: Suggest better layouts for performance

**Sources:**[src/relay/op/tensor/transform.cc 551-624](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L551-L624)[python/tvm/relay/op/nn/_nn.py 215-316](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L215-L316)

### Legalization

The `FLegalize` attribute allows operators to be lowered into simpler operations during compilation.

**Example use cases:**

*   Lower `batch_matmul` to `matmul` with reshapes
*   Lower high-level ops to canonical forms
*   Apply target-specific transformations

**Sources:**[python/tvm/relay/op/nn/_nn.py 56-74](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L56-L74)[python/tvm/relay/op/nn/_nn.py 221-240](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L221-L240)

### Operator Strategies with Multiple Implementations

A single operator can have multiple implementations with different performance characteristics. The strategy system selects the best implementation based on:

*   Input shapes and dtypes
*   Target hardware capabilities
*   Auto-scheduler/AutoTVM availability

**Priority levels (plevel):**

*   Higher values = higher priority
*   Range: 0-99 (typically 1-25)
*   Used when multiple implementations are available

**Sources:**[python/tvm/relay/op/strategy/cuda.py 138-460](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L138-L460)[python/tvm/relay/op/strategy/generic.py 173-320](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/generic.py#L173-L320)

## Common Patterns and Best Practices

### Attribute Definition Checklist

*   - [x]  Inherit from `tvm::AttrsNode<YourAttrs>`
*   - [x]  Use `TVM_DECLARE_ATTRS` macro
*   - [x]  Document each field with `.describe()`
*   - [x]  Set sensible defaults with `.set_default()`
*   - [x]  Register in `TVM_REGISTER_NODE_TYPE`

### Type Relation Checklist

*   - [x]  Check input count: `ICHECK_EQ(types.size(), expected)`
*   - [x]  Validate input types are not `IncompleteType`
*   - [x]  Extract attributes: `attrs.as<YourAttrs>()`
*   - [x]  Compute output shape from inputs and attributes
*   - [x]  Use `reporter->Assign(types[output_idx], output_type)`
*   - [x]  Return `true` on success, `false` if inputs incomplete

### Registration Checklist

*   - [x]  Descriptive `.describe()` string
*   - [x]  Correct `.set_num_inputs()`
*   - [x]  Document each input with `.add_argument()`
*   - [x]  Set appropriate `.set_support_level()` (1-10)
*   - [x]  Add type relation with `.add_type_rel()`
*   - [x]  Set compute with `.set_attr<FTVMCompute>()`
*   - [x]  Set pattern with `.set_attr<TOpPattern>()`
*   - [x]  Add layout inference if layout-sensitive

### Common Pitfalls

| Issue | Solution |
| --- | --- |
| Segfault in type relation | Always check for `nullptr` after `as<T>()` casts |
| Missing operator in Python | Ensure FFI registration with `TVM_REGISTER_GLOBAL` |
| Operator not found | Check operator name matches between C++ and Python |
| Incorrect output shape | Verify type relation logic with unit tests |
| Poor performance | Implement target-specific strategies and schedules |

**Sources:**[src/relay/op/tensor/transform.cc 1-1176](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/tensor/transform.cc#L1-L1176)[python/tvm/relay/op/strategy/cuda.py 1-2500](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L1-L2500)
