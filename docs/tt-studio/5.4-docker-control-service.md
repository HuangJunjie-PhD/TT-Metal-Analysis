The Docker Control Service is a secure, host-resident FastAPI application (running on port 8002) that serves as the exclusive gateway for Docker operations in TT-Studio. It replaces the insecure practice of mounting `/var/run/docker.sock` directly into containerized services, providing a layer of authentication, resource enforcement, and operation whitelisting.

## System Architecture

The service runs directly on the host system to maintain access to the Docker daemon while exposing a hardened REST API to the `tt_studio_backend` container.

## Security Implementation

The service implements several critical security layers to prevent privilege escalation and resource exhaustion.

### 1. JWT Authentication

All endpoints, except for the health check, are protected by the `authenticate_request` middleware. This middleware validates HS256-signed JWT tokens using a secret shared between the host service and the backend container.

*   **Middleware:**`authenticate_request` in [docker-control-service/middleware/auth.py 18-55](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/middleware/auth.py#L18-L55)
*   **Secret Configuration:**`JWT_SECRET` in [docker-control-service/config.py 21](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/config.py#L21-L21)

### 2. Operation Whitelisting

The service enforces strict policies defined in `config.py` to limit the scope of Docker operations:

*   **Image Whitelist:** Only images from approved registries (e.g., `ghcr.io/tenstorrent/`, `alpine:`) can be pulled or run [docker-control-service/config.py 24-31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/config.py#L24-L31)
*   **Network Restrictions:** Containers can only be attached to a specific set of networks (e.g., `tt_studio_network`, `bridge`) [docker-control-service/config.py 33-37](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/config.py#L33-L37)
*   **Privileged Mode Blocked:** The service explicitly prevents running containers in privileged mode to protect host integrity [docker-control-service/README.md 21](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L21-L21)

### 3. Resource Limits

Global maximums are enforced for every container created through the service:

*   **Max Memory:** 16GB [docker-control-service/config.py 40](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/config.py#L40-L40)
*   **Max CPUs:** 8 [docker-control-service/config.py 41](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/config.py#L41-L41)

## API Endpoints

The service organizes functionality into four specialized routers.

### Container Management (`/api/v1/containers`)

| Endpoint | Method | Role |
| --- | --- | --- |
| `/run` | POST | Starts a new container after validating images and networks [docker-control-service/routers/containers.py 33-58](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L33-L58) |
| `/{id}/stop` | POST | Stops a running container with configurable timeout [docker-control-service/routers/containers.py 61-84](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L61-L84) |
| `/{id}/remove` | POST | Deletes a container [docker-control-service/routers/containers.py 87-110](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L87-L110) |
| `/{id}/logs` | GET | Streams real-time logs via Server-Sent Events (SSE) [docker-control-service/routers/containers.py 192-202](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L192-L202) |

### Image Management (`/api/v1/images`)

| Endpoint | Method | Role |
| --- | --- | --- |
| `/pull` | POST | Pulls an image from a registry [docker-control-service/routers/images.py 19-45](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/images.py#L19-L45) |
| `/remove` | DELETE | Removes a local image [docker-control-service/routers/images.py 62-84](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/images.py#L62-L84) |
| `/exists` | GET | Checks if an image is available locally [docker-control-service/routers/images.py 140-169](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/images.py#L140-L169) |

### Network Management (`/api/v1/networks`)

Provides endpoints to `create`, `remove`, `list`, `connect`, and `disconnect` containers from Docker networks [docker-control-service/routers/networks.py 28-213](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/networks.py#L28-L213)

## Implementation Details

### ContainerService Class

The `ContainerService` in `container_service.py` encapsulates the business logic for Docker interactions. It uses lazy initialization for the Docker SDK client [docker-control-service/services/container_service.py 23-36](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/services/container_service.py#L23-L36) and handles the translation of API requests into Docker SDK `run` arguments, including volume mapping and device pass-through [docker-control-service/services/container_service.py 52-100](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/services/container_service.py#L52-L100)

### Error Handling

The service maps Docker SDK exceptions to standard HTTP status codes:

*   `docker.errors.NotFound` → `404 Not Found`[docker-control-service/routers/containers.py 177-180](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L177-L180)
*   `docker.errors.APIError` → `500 Internal Server Error`[docker-control-service/routers/containers.py 182-185](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L182-L185)
*   Validation failures → `400 Bad Request`[docker-control-service/routers/containers.py 46-47](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/routers/containers.py#L46-L47)

## Integration with TT-Studio

The service is managed by the main `run.py` script. It starts alongside other components unless `--skip-docker-control` is specified [docker-control-service/README.md 176-187](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L176-L187) The backend communicates with it using the `DOCKER_CONTROL_SERVICE_URL` environment variable [docs/DOCKER_SOCKET_MIGRATION.md 91-96](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docs/DOCKER_SOCKET_MIGRATION.md?plain=1#L91-L96)
