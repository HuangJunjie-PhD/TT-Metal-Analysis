The Register Rename Stage is a critical component in BOOM's out-of-order execution pipeline. It maps logical (ISA) register specifiers to physical register specifiers, which enables out-of-order execution by eliminating false dependencies between instructions. This page describes the implementation, key components, and operation of the register rename stage in RISC-V Ocelot.

For information about the Reorder Buffer which works closely with the rename stage, see [Reorder Buffer](https://deepwiki.com/tenstorrent/riscv-ocelot/2.2-reorder-buffer).

## Overview and Purpose

The primary purpose of the register rename stage is to eliminate Write-After-Write (WAW) and Write-After-Read (WAR) hazards by mapping architectural (logical) registers to a larger pool of physical registers. This leaves only the true Read-After-Write (RAW) dependencies, allowing for more instruction-level parallelism.

The rename stage receives instructions from the decode stage, assigns physical registers to each instruction's destination register, looks up the most recent physical register mappings for the source registers, and forwards the renamed instructions to the dispatch stage.

### 1. Map Table (RenameMapTable)

The map table maintains the current mapping from logical registers to physical registers. For each logical register, it stores which physical register currently holds its value. It contains:

*   Current register mappings
*   Snapshots for speculative execution (branch shadow)
*   Handling of rollback on branch misprediction

Sources: [src/main/scala/exu/rename/rename-maptable.scala 60-108](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rename/rename-maptable.scala#L60-L108)

### 2. Free List (RenameFreeList)

The free list tracks which physical registers are available for allocation. When an instruction with a destination register is renamed, a physical register is allocated from the free list. The free list:

*   Manages a pool of available physical registers
*   Allocates new physical registers for destination registers
*   Returns physical registers to the pool when they're no longer needed
*   Tracks speculative allocations for recovery

Sources: [src/main/scala/exu/rename/rename-freelist.scala 50-76](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rename/rename-freelist.scala#L50-L76)

### 3. Busy Table (RenameBusyTable)

The busy table tracks which physical registers are currently being computed (busy) or are ready (not busy). When a destination register is renamed, its physical register is marked as busy. When an instruction completes execution and writes its result, the corresponding physical register is marked as not busy.

Sources: [src/main/scala/exu/rename/rename-busytable.scala 48-56](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/rename/rename-busytable.scala#L48-L56)

## Rename Process Flow

The rename process happens across two pipeline stages - Rename1 and Rename2/Dispatch.

### Rename1 Stage

In this stage, the processor:

1.   Looks up the current physical register mapping for each source register
2.   Allocates a new physical register from the free list for the destination register
3.   Remembers the old physical register mapping for the destination (stale_pdst) for later deallocation

### Rename2 Stage

In this stage, the processor:

1.   Checks the busy status of the physical source registers
2.   Updates the map table with the new destination register mapping
3.   Handles bypassing of register mappings from older instructions in the same rename group
4.   Forwards the renamed instruction to the dispatch stage

### Stall Logic

The rename stage will stall if:

1.   The free list is empty (no physical registers available)
2.   The dispatch stage is not ready
3.   The ROB is full
4.   A branch mispredict is detected

## Specialized Rename Stages

BOOM implements two specialized rename stages:

1.   **Integer/FP Rename Stage**: Handles renaming of integer and floating-point registers
2.   **Predicate Rename Stage**: Specializes in handling branch predicate registers for Single-Fetch-Multiple-Execute (SFME) optimization

These stages share the same abstract structure but differ in their implementation details to handle the specific register types.

## Branch Handling and Recovery

Register renaming is speculative and must support recovery when a branch misprediction occurs. The rename stage implements this through:

1.   **Branch Tags**: Each instruction receives a branch tag that identifies which speculative path it belongs to
2.   **Branch Masks**: Each instruction has a mask of branch tags it depends on
3.   **Snapshots**: The map table takes snapshots for each branch to enable recovery
4.   **Allocation Lists**: The free list tracks which registers were allocated under each branch

When a branch misprediction is detected:

1.   The map table reverts to the snapshot corresponding to the mispredicted branch
2.   The free list reclaims any physical registers allocated after the branch
3.   Later instructions with matching branch tags are killed

## Implementation in RISC-V Ocelot

In RISC-V Ocelot, the rename stage is implemented through several key classes:

1.   `RenameStage`: The main class that orchestrates the rename process
2.   `RenameMapTable`: Maps logical registers to physical registers
3.   `RenameFreeList`: Manages the pool of physical registers
4.   `RenameBusyTable`: Tracks which physical registers are busy

The rename stage is parameterized to support different configurations:

*   Width (number of instructions renamed per cycle)
*   Number of physical registers
*   Number of wakeup ports
*   Register type (integer/floating-point)

## Summary

The register rename stage is a critical component that enables out-of-order execution by breaking false dependencies between instructions. It maintains the illusion of a small architectural register file while using a larger physical register file to hold intermediate values. Through the map table, free list, and busy table, it provides the mechanisms needed for speculative execution, branch recovery, and precise exceptions.
