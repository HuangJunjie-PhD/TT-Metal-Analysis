## Purpose and Scope

This document explains the multiple coordinate systems used throughout `tt-umd` to address cores on Tenstorrent chips and how to translate between them. Different coordinate systems exist to handle harvesting (disabled cores), NOC routing differences, and the distinction between physical hardware layout and logical software addressing.

The coordinate translation logic is primarily managed by the `CoordinateManager` hierarchy and exposed via the `SocDescriptor` class.

* * *

## Overview of Coordinate Systems

The `tt-umd` codebase uses multiple coordinate systems to reference cores on a chip. Each system serves a different purpose:

| Coordinate System | Purpose | Used For |
| --- | --- | --- |
| **NOC0** | Hardware NOC0 router coordinates | Direct hardware addressing, low-level routing. |
| **NOC1** | Hardware NOC1 router coordinates | Alternative NOC routing path. |
| **TRANSLATED** | Harvesting-aware virtual coordinates | Software operations that must account for disabled cores while maintaining hardware-like grid spacing. |
| **LOGICAL** | Software logical coordinates | High-level programming interface; provides a dense 0-indexed grid (e.g., Worker (0,0) to (N,M)). |
| **LITERAL** | Raw coordinates | Used when no translation or system-specific logic should be applied. |

## CoreCoord and CoordSystem

Cores are referenced using the `CoreCoord` struct, which encapsulates spatial coordinates (`x`, `y`), a `CoreType`, and a `CoordSystem`.

### CoreType Enumeration

The driver distinguishes between various functional units on the chip:

*   `TENSIX` / `WORKER`: General purpose compute cores.
*   `DRAM`: Memory controllers.
*   `ETH` / `ACTIVE_ETH` / `IDLE_ETH`: Ethernet communication cores.
*   `PCIE`, `ARC`, `ROUTER_ONLY`, `SECURITY`, `L2CPU`, `DISPATCH`.

## Coordinate Manager Hierarchy

The `CoordinateManager` is the engine behind all translations. It is an abstract base class with architecture-specific implementations that handle the unique harvesting and NOC layout patterns of each chip.

### Implementation Details

*   **Wormhole**: Harvesting typically affects Tensix rows. `WormholeCoordinateManager` maps `LOGICAL` to `NOC0` by skipping harvested rows and applies specific offsets for `TRANSLATED` coordinates [device/coordinates/wormhole_coordinate_manager.cpp 56-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/wormhole_coordinate_manager.cpp#L56-L87)
*   **Blackhole**: Harvesting typically affects Tensix columns. `BlackholeCoordinateManager` handles complex `TRANSLATED` mappings where harvested columns are moved to the end of the grid to maintain a contiguous active area [device/coordinates/blackhole_coordinate_manager.cpp 103-149](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/blackhole_coordinate_manager.cpp#L103-L149)
*   **NOC Translation**: When `noc_translation_enabled` is true, the manager populates mappings that account for hardware-level coordinate swizzling [device/coordinates/coordinate_manager.cpp 46-63](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/coordinate_manager.cpp#L46-L63)

## Coordinate Translation Flow

Translation is performed by converting the input coordinate to a common `NOC0` base and then looking up the target system representation.

### Key Functions in SocDescriptor

*   `translate_coord_to`: The primary API for converting between any two systems [device/api/umd/device/soc_descriptor.hpp 49-58](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/soc_descriptor.hpp#L49-L58)

## Harvesting Effects

Harvesting removes cores from the `LOGICAL` and `TRANSLATED` active grids.

1.   **Logical Grid**: A dense grid of size `(Physical_Width - Harvested_Cols) x (Physical_Height - Harvested_Rows)`.
2.   **Translated Grid**: Often maintains the physical grid size but reorders or offsets coordinates so that active cores appear in a specific range, often to simplify broadcast or multicast logic.
3.   **Harvested Cores**: These cores exist in `NOC0/NOC1` space but have no mapping in `LOGICAL` space. Attempting to translate a harvested `NOC0` coordinate to `LOGICAL` will result in an error [device/coordinates/coordinate_manager.cpp 154-162](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/coordinates/coordinate_manager.cpp#L154-L162)

### Example: Blackhole Column Harvesting

In Blackhole, if column 0 is harvested, the `CoordinateManager` ensures that `LOGICAL(0,0)` maps to the first available physical column (e.g., `NOC0(2,2)`) [tests/baremetal/test_core_coord_translation_bh.cpp 57-70](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/baremetal/test_core_coord_translation_bh.cpp#L57-L70)

## Verification and Testing

The `tt-umd` repository includes extensive tests to ensure coordinate systems are consistent with hardware registers.

### NOC ID Verification

Tests read the `NODE_ID` register from hardware cores using different NOCs. The value returned by the hardware depends on which NOC was used for the transaction, effectively verifying that the host's `translate_coord_to` matches the chip's internal routing logic.
