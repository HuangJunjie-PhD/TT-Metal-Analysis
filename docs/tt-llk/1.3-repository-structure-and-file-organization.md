## Purpose and Scope

This document describes the physical organization of the tt-llk repository, explaining the purpose of each major directory, the architecture-specific code layout, and the location of key components. For information about the three-stage pipeline architecture itself, see [Three-Stage Pipeline Architecture](https://deepwiki.com/tenstorrent/tt-llk/1.1-three-stage-pipeline-architecture). For details on multi-architecture support and feature differences, see [Multi-Architecture Support](https://deepwiki.com/tenstorrent/tt-llk/1.2-multi-architecture-support).

## Top-Level Directory Structure

The repository follows a clear separation between architecture-specific implementations, shared components, testing infrastructure, and CI/CD configuration:

**Sources:**[.gitignore 1-28](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.gitignore#L1-L28)[tests/requirements.txt 1-48](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/requirements.txt#L1-L48)[.github/workflows/setup-and-test.yml 1-237](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L1-L237)

The three primary architecture directories (`tt_llk_wormhole_b0/`, `tt_llk_blackhole/`, `tt_llk_quasar/`) contain complete implementations of the Low-Level Kernel library for each hardware target. Each directory is self-contained with its own headers and library implementations.

### C++ Test Helpers: `tests/helpers/`

The C++ helpers provide the runtime framework for executing LLK operations on hardware:

**Sources:**[tests/python_tests/helpers/test_config.py 559-658](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L559-L658)[tests/python_tests/helpers/test_config.py 792-866](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L792-L866)

The compilation process generates three key artifacts per test variant:

*   `build.h` - Auto-generated configuration header with format constants and runtime parameter structures
*   `*.o` object files - Compiled kernel components (unpack, math, pack)
*   `*.elf` executables - Linked binaries ready for hardware execution

Architecture-specific hardware definitions are downloaded separately during test environment setup:

**Sources:**[tests/python_tests/conftest.py 30-84](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L30-L84)[.gitignore 18](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.gitignore#L18-L18)

## Build Artifacts and Temporary Directories

The build system generates artifacts in `/tmp/tt-llk-build/` to separate build outputs from the source tree:

**Sources:**[tests/python_tests/helpers/test_config.py 86-246](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L86-L246)[tests/python_tests/helpers/test_config.py 560-657](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L560-L657)[tests/python_tests/helpers/test_config.py 868-897](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L868-L897)

### Shared vs Variant Artifacts

The build system employs a two-tier artifact strategy to minimize redundant compilation:

**Shared Artifacts** (`/tmp/tt-llk-build/shared/`):

*   Built once per test session using `FileLock` for coordination
*   Contains architecture-independent object files: `tmu-crt0.o`, `brisc.o`, `main_unpack.o`, `main_math.o`, `main_pack.o`
*   Marked complete with `.shared_complete` file ([tests/python_tests/helpers/test_config.py 578-653](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L578-L653))
*   Profiler builds use separate `shared-profiler/` directory with `-DLLK_PROFILER` flag

**Variant Artifacts** (`/tmp/tt-llk-build/{test_name}/{variant_id}/`):

*   Each test variant gets a unique directory identified by SHA256 hash of parameters
*   Contains test-specific files: `build.h`, `kernel_*.o`, `unpack.elf`, `math.elf`, `pack.elf`
*   Built using `ThreadPoolExecutor` with 3 parallel workers for kernel components ([tests/python_tests/helpers/test_config.py 637-643](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L637-L643))
*   Protected by per-variant `FileLock` to prevent race conditions ([tests/python_tests/helpers/test_config.py 886-887](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L886-L887))

### Address Spaces and Memory Layout

Test execution uses specific L1 memory addresses for runtime parameters and performance data:

| Purpose | Address | Size | Notes |
| --- | --- | --- | --- |
| Runtime params (normal) | `0x20000` | Variable | Test-specific parameters |
| Runtime params (coverage) | `0x64000` | Variable | Higher address for coverage builds |
| Performance counters | `0x16A000` | `0xC18` (3096 bytes) | 86 config + 172 data words per thread |
| Profiler barrier | `0x16AFF4` | 4 bytes | Synchronization point |
| TRISC start addresses | `0x16DFF0-0x16DFF8` | 12 bytes | 3 TRISC threads |
| Unpack perf buffer | `0x16B000` | `0x400` (1024 bytes) | Thread 0 performance data |
| Math perf buffer | `0x16C000` | `0x400` (1024 bytes) | Thread 1 performance data |
| Pack perf buffer | `0x16D000` | `0x400` (1024 bytes) | Thread 2 performance data |

**Sources:**[tests/python_tests/helpers/test_config.py 138-180](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L138-L180)[tests/python_tests/helpers/test_config.py 418-486](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L418-L486)

## CI/CD Workflow Files

The `.github/workflows/` directory contains GitHub Actions workflows for automated testing and building:

**Sources:**[.github/workflows/setup-and-test.yml 1-237](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L1-L237)

The `setup-and-test.yml` workflow implements a producer-consumer pattern for efficient parallel testing:

1.   **Producer phase** (`--compile-producer`): Compiles all test variants in parallel with `-n 10` workers ([.github/workflows/setup-and-test.yml 137-142](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L137-L142))
2.   **Consumer phase** (`--compile-consumer`): Executes pre-compiled ELFs on hardware ([.github/workflows/setup-and-test.yml 145-150](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L145-L150))
3.   **Splitting strategy**: Tests are divided into N groups using duration-based balancing when `.test_durations` file exists ([.github/workflows/setup-and-test.yml 128-134](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L128-L134))

The workflow supports multiple pytest markers for selective execution:

*   Default: `"not perf and not quasar"` - Regular tests excluding performance and Quasar-specific
*   Coverage: `--coverage` flag enables GCOV instrumentation
*   Performance: `-m "perf"` runs performance benchmarking tests
*   Architecture-specific: Determined by runner label (`n150` for Wormhole, `p150` for Blackhole)

**Sources:**[.github/workflows/setup-and-test.yml 18-47](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L18-L47)[.github/workflows/setup-and-test.yml 106-165](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L106-L165)
