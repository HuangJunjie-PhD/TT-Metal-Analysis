## Purpose and Scope

This document describes how the TT-NN Visualizer manages local data, focusing on uploading, validating, and accessing profile and performance data from the local file system. The local data management system provides a way for users to analyze TT-NN model execution data without requiring remote connections. For information about handling remote data sources, see [Remote Data Management](https://deepwiki.com/tenstorrent/ttnn-visualizer/3.2-remote-data-management).

## Local Data Types

The TT-NN Visualizer handles two main types of local data:

1.   **Memory Reports** - Profile data containing operation details, memory allocations, and tensor information
2.   **Performance Reports** - Timing and performance metrics for operations executed on the device

### Memory Report Requirements

Memory reports require the following files:

*   `db.sqlite` - Core database containing operation and tensor details
*   `config.json` - Configuration information for the report

### Performance Report Requirements

Performance reports require the following files:

*   `profile_log_device.csv` - Device-level profiling information
*   `tracy_profile_log_host.tracy` - Host-side profiling information
*   An `ops_perf_results_*.csv` file - Operation-specific performance metrics

## Uploading Local Data

### Upload Process Flow

The following diagram details the steps involved in uploading a local folder:

The upload process follows these steps:

1.   User selects a local directory through the file input in `LocalFolderSelector`
2.   Selected files are filtered to exclude unnecessary files (like tensors)
3.   The directory is validated to ensure it contains all required files
4.   If valid, files are uploaded to the server using FormData with progress tracking
5.   The server processes and stores the files in the appropriate directory
6.   On success, the UI updates to show the newly uploaded report is active

### File Validation

Before uploading, files are validated to ensure they contain all required components:

**Memory Report Validation:**

`// Check for required files in a memory reportconst checkRequiredReportFiles = (files: FileList): boolean => {    const requiredFiles = ['db.sqlite', 'config.json'];    const fileSet = new Set<string>();     Array.from(files).forEach((file) => {        const pathParts = file.webkitRelativePath.split('/');        if (pathParts.length === 2) {            fileSet.add(pathParts[1]);        }    });     return requiredFiles.every((file) => fileSet.has(file));};`

## Working with Local Data

After reports are uploaded, they can be selected for analysis. The system maintains separate active reports for memory and performance data.

### Active Report Selection

The `LocalFolderPicker` component displays available reports and allows users to:

1.   Select a report to make it active
2.   Delete a report that is no longer needed

When a report is made active, the system:

1.   Updates the session state via the `updateTabSession` API call
2.   Sets the active report atoms (`activeProfilerReportAtom` or `activePerformanceReportAtom`)
3.   Sets the report location atom to 'local'
4.   Notifies the user with a toast notification

## Data Querying and Access

### Local CSV Query Runner

The backend provides a `LocalCSVQueryRunner` class that handles data querying from local CSV files:

The `LocalCSVQueryRunner` provides:

1.   Context management for file access (`__enter__` and `__exit__`)
2.   Header retrieval with `get_csv_header()`
3.   Flexible query execution with `execute_query()` that supports: 
    *   Column selection
    *   Filtering by column values
    *   Result format (list or dictionary)
    *   Result limiting

Higher-level query classes like `DeviceLogProfilerQueries` and `OpsPerformanceQueries` provide domain-specific query methods for different file types.

## Limitations and Considerations

1.   **Safari Support**: The local file directory upload feature is not supported in Safari browsers due to browser limitations. Users are advised to use Chrome or Firefox instead.

2.   **Performance**: Large reports with many files can take significant time to upload. The system provides upload progress tracking to keep users informed.

3.   **File Size**: The system is designed to handle typical TT-NN model reports. Extremely large files may cause performance issues or timeouts.

4.   **Validation**: The system performs basic validation by checking for required files but does not validate the content of these files. Invalid file contents may cause unexpected behavior.
