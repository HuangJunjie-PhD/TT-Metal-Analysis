This page documents the message queue system that enables host-to-SMC communication. The message queue provides a standardized request-response mechanism for the host to control hardware operations, adjust voltage/frequency settings, manage fan speed, trigger PCIe operations, and query telemetry data.

For information about the SMC telemetry system that exposes data to the host, see [SMC Telemetry System](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.2-smc-telemetry-system). For information about DVFS and throttling operations invoked by some messages, see [DVFS and Throttling](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.3-dvfs-and-throttling).

* * *

## Message Queue Architecture

The SMC firmware implements 4 independent message queues, each containing 4 slots for requests and 4 slots for responses. This ring buffer architecture allows the host to enqueue multiple commands without blocking, while the SMC processes them asynchronously.

### Queue Structure

The message queue system uses a shared memory region accessible by both the host CPU and the SMC ARC processor. Each queue contains a header with read/write pointers and separate arrays for request and response messages.

**Sources:**[include/tenstorrent/msgqueue.h 31-43](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L31-L43)[lib/tenstorrent/bh_arc/msgqueue.c 85-89](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L85-L89)

### Memory Layout

The message queue information structure is exposed to the host at address `STATUS_MSG_Q_INFO_REG_ADDR`:

| Field | Description |
| --- | --- |
| `message_queues` | Base address of the queue array |
| `MSG_QUEUE_SIZE | (NUM_MSG_QUEUES << 8)` | Encoded size information |
| Reserved | Unused fields |

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 97-98](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L97-L98)[lib/tenstorrent/bh_arc/msgqueue.c 359-368](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L359-L368)

## Message Structure

### Request Format

All requests are 32 bytes (8 uint32_t entries). The first byte contains the command code, with the remaining bytes interpreted based on the specific message type.

The `union request` provides structured access to different message formats:

**Sources:**[include/tenstorrent/msgqueue.h 390-452](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L390-L452)

### Response Format

Responses are also 32 bytes (8 uint32_t entries). The first entry typically contains the status code, with `data[0] = 0` indicating success.

| Response Field | Description |
| --- | --- |
| `data[0]` | Status code (0 = success, 0xFF = error, 0xFE = scratch-only) |
| `data[1-7]` | Message-specific response data |

**Sources:**[include/tenstorrent/msgqueue.h 456-458](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L456-L458)[lib/tenstorrent/bh_arc/msgqueue.c 24](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L24-L24)

## Message Handler Registration

The SMC firmware uses a static registration mechanism to associate message command codes with handler functions. Handlers are registered using the `REGISTER_MESSAGE` macro, which places handler entries into a special linker section.

### Handler Registration Pattern

**Sources:**[include/tenstorrent/msgqueue.h 467-471](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L467-L471)[lib/tenstorrent/bh_arc/msgqueue.c 350-357](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L350-L357)[lib/tenstorrent/bh_arc/msgqueue.c 371-379](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L371-L379)

### Handler Function Signature

All message handlers must conform to the `msgqueue_request_handler_t` signature:

`typedef uint8_t (*msgqueue_request_handler_t)(const union request *req, struct response *rsp);`
The handler returns 0 for success or a non-zero error code, which is OR'd into `response->data[0]`.

**Sources:**[include/tenstorrent/msgqueue.h 460](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L460-L460)

## Message Processing Flow

The message queue system is interrupt-driven, with three interrupt sources triggering message processing:

### Interrupt Sources

| Interrupt | Description |
| --- | --- |
| `IRQNUM_ARC_MISC_CNTL_IRQ0` | Direct message queue interrupt from host |
| `IRQNUM_MSI_CATCHER_NONEMPTY` | PCIe MSI with vector 0 indicates message available |
| `IRQNUM_MSI_CATCHER_OVERFLOW` | MSI catcher FIFO overflow (flushes and processes) |

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 449-456](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L449-L456)

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 383-395](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L383-L395)[lib/tenstorrent/bh_arc/msgqueue.c 340-348](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L340-L348)[lib/tenstorrent/bh_arc/msgqueue.c 310-327](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L310-L327)[lib/tenstorrent/bh_arc/msgqueue.c 290-308](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L290-L308)

### Queue State Validation

The `start_next_message()` function validates queue state before processing:

1.   Checks that request queue is not empty (`request_wptr != request_rptr`)
2.   Checks that response queue has available space
3.   Validates all pointers are within valid range
4.   Uses memory fences to ensure proper ordering

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 204-236](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L204-L236)

## Special Message Handling

Three message types receive special handling before general dispatch:

### TT_SMC_MSG_SET_LAST_SERIAL (0xBE)

Updates the `last_serial` field in the message queue header without sending a response. Used for synchronization between host and SMC.

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 266-269](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L266-L269)

### TT_SMC_MSG_TEST (0x90)

A test/diagnostic message that increments the provided test value and returns both the incremented value and the next serial number in the response.

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 276-283](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L276-L283)

### TT_SMC_MSG_REPORT_SCRATCH_ONLY (0x16)

Sets the response status to `MESSAGE_QUEUE_STATUS_SCRATCH_ONLY` (0xFE) to indicate the message was processed but only scratch/telemetry data should be read.

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 286-288](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L286-L288)

## Error Handling

Message handlers return error codes that are OR'd into `response->data[0]`:

| Error Code | Meaning |
| --- | --- |
| 0 | Success |
| 1 | Generic failure |
| 0xFE | Scratch-only message |
| 0xFF | Unrecognized message type |
| Other | Handler-specific error codes |

For example, the `TT_SMC_MSG_TOGGLE_GDDR_RESET` message defines specific error codes in `enum gddr_reset_err`:

| Error Code | Name | Meaning |
| --- | --- | --- |
| 1 | `GDDR_RESET_ERR_INVALID_INST` | Invalid GDDR instance number |
| 2 | `GDDR_RESET_ERR_HARVESTED` | GDDR controller is harvested |
| 3 | `GDDR_RESET_ERR_MASKED` | GDDR controller is masked |
| 4 | `GDDR_RESET_ERR_TRAINING` | Memory training failed |
| 5 | `GDDR_RESET_ERR_BIST` | BIST failed |
| 6 | `GDDR_RESET_ERR_POWERDOWN` | Controller in powerdown |

**Sources:**[include/tenstorrent/msgqueue.h 350-358](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L350-L358)[lib/tenstorrent/bh_arc/msgqueue.c 251-264](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L251-L264)

## Message Queue Initialization

The message queue system is initialized during SMC boot through the `init_msgqueue()` function:

1.   Clears all message queue headers
2.   Writes the `message_queue_info` pointer to `STATUS_MSG_Q_INFO_REG_ADDR`
3.   Connects interrupt handlers for message processing
4.   Enables the three interrupt sources
5.   Sets the `msg_queue_ready` flag in `STATUS_BOOT_STATUS0_REG`

**Sources:**[lib/tenstorrent/bh_arc/msgqueue.c 444-462](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L444-L462)
