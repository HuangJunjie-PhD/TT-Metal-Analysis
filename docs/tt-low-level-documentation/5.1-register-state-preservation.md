# Register State Preservation

Relevant source files
*   [data_movement_doc/general/state_preservation.md](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/state_preservation.md?plain=1)

## Purpose and Scope

This document provides a technical reference for NOC (Network-on-Chip) register state preservation behavior across API calls. It documents which registers maintain their values (stateful) versus which are rewritten on each operation (non-stateful), and highlights platform-specific differences between Wormhole B0 and Blackhole.

For guidance on using stateful versus stateless APIs in practice, including performance optimization patterns, see [Stateful vs Stateless APIs](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.5-stateful-vs-stateless-apis). For broader platform comparisons beyond register behavior, see [Platform Comparison](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.2-platform-comparison).

## Overview

NOC APIs issue instructions by writing to command buffer registers. Understanding register statefulness is critical for:

1.   **Performance Optimization**: Stateful registers can be configured once and reused across multiple operations, reducing command buffer overhead
2.   **API Selection**: Choosing between stateful and stateless API variants based on operation patterns
3.   **Correctness**: Avoiding unintended register reuse when switching operation types
4.   **Platform Portability**: Accounting for architectural differences between Wormhole B0 and Blackhole

**Stateful Register**: Preserves its value across multiple NOC API calls unless explicitly modified by the API or programmer.

**Non-Stateful Register**: Re-programmed on every NOC API call or serves as a control flag that is written per-operation.

## Register Categories by Function

### Addressing Registers

These registers control target and return addresses for NOC operations:

| Register | Function | Wormhole B0 | Blackhole | Notes |
| --- | --- | --- | --- | --- |
| `NOC_TARG_ADDR_LO` | Target address low bits | Stateful | Stateful | Destination for writes/reads |
| `NOC_TARG_ADDR_MID` | Target address middle bits | Stateful | Stateful | Extended addressing |
| `NOC_TARG_ADDR_HI` | Target address high bits | Unused | Stateful | Blackhole uses 64-bit addressing |
| `NOC_RET_ADDR_LO` | Return address low bits | Stateful | Stateful | Source for reads |
| `NOC_RET_ADDR_MID` | Return address middle bits | Stateful | Stateful | Extended addressing |
| `NOC_RET_ADDR_HI` | Return address high bits | Unused | Stateful | Blackhole uses 64-bit addressing |

**Key Difference**: Blackhole introduces `NOC_TARG_ADDR_HI` and `NOC_RET_ADDR_HI` for 64-bit addressing, expanding from Wormhole B0's 48-bit address space.

### Control and Configuration Registers

| Register | Function | Wormhole B0 | Blackhole | Notes |
| --- | --- | --- | --- | --- |
| `NOC_CTRL` | General control settings | Stateful | Stateful | Routing, VC selection |
| `NOC_PACKET_TAG` | Packet identification tag | Stateful | Stateful | Ordering and matching |
| `NOC_AT_LEN_BE` | Transaction length/byte-enable | Stateful | Stateful | Transfer size control |
| `NOC_AT_LEN_BE_1` | Additional length control | N/A | Stateful | Blackhole-only |
| `NOC_AT_DATA` | Atomic transaction data | Stateful | Stateful | For atomic operations |
| `NOC_BRCST_EXCLUDE` | Multicast exclusion mask | N/A | Stateful | Blackhole-only |
| `NOC_L1_ACC_AT_INSTRN` | L1 access atomic instruction | N/A | Stateful | Blackhole-only |
| `NOC_SEC_CTRL` | Security control | N/A | Stateful | Blackhole-only |

**Key Additions in Blackhole**: Four new stateful registers (`NOC_AT_LEN_BE_1`, `NOC_BRCST_EXCLUDE`, `NOC_L1_ACC_AT_INSTRN`, `NOC_SEC_CTRL`) provide enhanced functionality not present in Wormhole B0.

### Command and Control Flags (Non-Stateful)

These registers are rewritten on every operation or serve as control flags:

| Register | Function | Wormhole B0 | Blackhole | Purpose |
| --- | --- | --- | --- | --- |
| `NOC_CMD_CTRL` | Command control | Non-stateful | Non-stateful | Triggers operations, controls other registers |
| `NOC_NODE_ID` | Node identifier | Non-stateful | Non-stateful | Specifies target node per operation |
| `NOC_ENDPOINT_ID` | Endpoint identifier | Non-stateful | Non-stateful | Specifies endpoint per operation |
| `ECC_CTRL` | Error correction control | Non-stateful | Non-stateful | Controls error detection/correction |
| `NOC_CLEAR_OUTSTANDING_REQ_CNT` | Clear request counter | Non-stateful | Non-stateful | Resets outstanding request tracking |
| `RISC_IF_STATUS_OFFSET` | RISC interface status | Non-stateful | N/A | Wormhole B0 only |
| `CMD_BUF_AVAIL` | Command buffer available | N/A | Non-stateful | Blackhole-only status flag |
| `CMD_BUF_OVFL` | Command buffer overflow | N/A | Non-stateful | Blackhole-only overflow indicator |

### Error Tracking Registers (Stateful)

| Register | Function | State | Notes |
| --- | --- | --- | --- |
| `NUM_MEM_PARITY_ERR` | Memory parity error count | Stateful | Accumulates across operations |
| `NUM_HEADER_1B_ERR` | 1-bit header error count | Stateful | Accumulates across operations |
| `NUM_HEADER_2B_ERR` | 2-bit header error count | Stateful | Accumulates across operations |

These error counters are stateful because they accumulate error statistics across multiple operations, rather than reflecting single-operation state.

## Stateful API Implementation Pattern

This diagram illustrates how stateful APIs exploit register preservation to reduce command buffer writes on repeated operations. The first call (`noc_async_write_init()` or equivalent) configures all stateful registers. Subsequent calls only write `NOC_CMD_CTRL` and other non-stateful registers, significantly reducing overhead.

## Platform Evolution: Wormhole B0 → Blackhole

### Key Architectural Changes

| Feature | Wormhole B0 | Blackhole | Impact |
| --- | --- | --- | --- |
| Address Space | 48-bit (LO, MID registers) | 64-bit (LO, MID, HI registers) | Larger addressable memory, requires HI register programming |
| Multicast Control | Basic | `NOC_BRCST_EXCLUDE` added | Fine-grained exclusion in multicast operations |
| Atomic Operations | `NOC_AT_DATA` | `NOC_AT_DATA` + `NOC_AT_LEN_BE_1` | Enhanced atomic transaction support |
| Security | None | `NOC_SEC_CTRL` | Security features in Blackhole |
| L1 Access | Standard | `NOC_L1_ACC_AT_INSTRN` | Specialized L1 atomic instructions |
| Buffer Status | `RISC_IF_STATUS_OFFSET` | `CMD_BUF_AVAIL`, `CMD_BUF_OVFL` | More detailed buffer monitoring |

## Usage Implications

### Performance Optimization

When the same operation is repeated with identical configuration (e.g., writing to the same destination address repeatedly):

1.   **Use stateful APIs**: Configure registers once with initialization call, then reuse configuration
2.   **Reduced overhead**: Avoid rewriting `NOC_TARG_ADDR_*`, `NOC_RET_ADDR_*`, `NOC_CTRL`, etc. on every operation
3.   **Command buffer efficiency**: Fewer register writes per operation means more operations can be queued

### Context Switching Risks

When switching between different operation types or destinations:

1.   **Explicit reconfiguration required**: Stateful registers retain their values, which may be inappropriate for new operation
2.   **Use stateless APIs**: Ensure all registers are written explicitly when operation parameters change
3.   **Barrier before state change**: Place synchronization barriers when transitioning between operation modes to avoid unintended register reuse

### Platform Portability Considerations

Code targeting both Wormhole B0 and Blackhole must:

1.   **Check register availability**: `NOC_TARG_ADDR_HI`, `NOC_RET_ADDR_HI` are unused on Wormhole B0
2.   **Handle new registers**: Blackhole-specific registers (`NOC_BRCST_EXCLUDE`, `NOC_L1_ACC_AT_INSTRN`, etc.) require conditional initialization
3.   **Address space awareness**: Wormhole B0 uses 48-bit addresses, Blackhole uses 64-bit addresses

## Summary

Register state preservation is a fundamental architectural feature that enables performance optimization through reduced command buffer overhead. Stateful registers maintain their values across API calls, allowing repeated operations to reuse configuration. Understanding which registers are stateful versus non-stateful, and the differences between Wormhole B0 and Blackhole platforms, is essential for writing efficient and portable NOC code.

Key takeaways:

*   **Stateful registers** (addressing, control, error counters) preserve values across calls
*   **Non-stateful registers** (command flags, node IDs) are rewritten per operation
*   **Blackhole extends** register set with 64-bit addressing and additional control registers
*   **Stateful APIs** exploit register preservation for performance optimization
*   **Context switching** requires explicit reconfiguration or use of stateless APIs
