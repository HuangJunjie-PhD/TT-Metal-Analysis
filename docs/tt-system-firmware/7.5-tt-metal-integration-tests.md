This page documents the TT-Metal integration tests that validate firmware compatibility with the TT-Metal ML runtime by running actual machine learning workloads on physical hardware. These tests represent the highest level of integration testing in the tt-system-firmware validation pipeline.

For information about other testing strategies, see:

*   Basic firmware functionality validation: [Hardware Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests)
*   Firmware communication chain testing: [End-to-End Testing with pyluwen](https://deepwiki.com/tenstorrent/tt-system-firmware/7.3-end-to-end-testing-with-pyluwen)
*   Reliability and reset mechanism testing: [Stress and Long-Duration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.4-stress-and-long-duration-tests)

## Overview

The Metal integration tests verify that the system firmware (SMC, DMC, and CMFW) correctly enables TT-Metal workloads by:

1.   **Flashing the latest firmware** onto the target hardware
2.   **Running TT-Metal upstream tests** that exercise the complete stack from ML frameworks down to hardware
3.   **Validating performance and correctness** of ML workloads including language models like Llama

These tests run nightly or can be triggered manually with the `metal-tests` label. They use pre-built TT-Metal container images containing the ML runtime, test suites, and model weights.

**Sources:**[.github/workflows/metal.yml 1-135](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/metal.yml#L1-L135)

### Test Patches and Workarounds

The workflow applies several patches to the TT-Metal test suite to accommodate CI environment differences:

#### Whisper Performance Check Bypass

This disables performance validation for the Whisper model test, as CI runners have different hardware characteristics than the standard Metal test systems.

#### ResNet Determinism Warmup

This warmup run addresses a known issue (SYS-2705) where P300A cards produce different outputs on the first inference after a cold boot. The test is run once without determinism checks to warm up the system before the actual validation runs.

**Sources:**[.github/workflows/metal.yml 111-126](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/metal.yml#L111-L126)

## Test Execution

### Upstream Test Script

The actual ML workload tests are executed via the TT-Metal upstream test script:

The script receives a target parameter (`blackhole`, `blackhole_p300`, or `blackhole_glx`) that determines which test suite to run. This script is maintained in the TT-Metal repository and contains:

*   **Unit tests** for Metal runtime components
*   **Integration tests** for ML operators and kernels
*   **End-to-end model tests** including language models and vision models

*   **Determinism tests** to ensure reproducible results

## Validation Coverage

The Metal integration tests provide comprehensive validation of the firmware stack by exercising:

### Firmware Features Validated

By running actual ML workloads, the Metal tests indirectly validate:

1.   **Power Management (DVFS)**: Workloads trigger AICLK throttling and thermal management
2.   **Memory Training**: GDDR initialization must succeed for workloads to load weights
3.   **PCIe Stability**: Large model transfers exercise DMA and PCIe link stability
4.   **NOC Translation**: Multi-chip workloads validate NOC routing and harvesting support
5.   **Telemetry Collection**: Runtime monitoring validates SMC telemetry updates
6.   **Reset Recovery**: Failed tests may trigger resets, validating watchdog functionality

**Sources:**[.github/workflows/metal.yml 1-135](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/metal.yml#L1-L135)

## Comparison with Other Test Levels

| Test Level | Duration | Hardware Required | Validation Scope | Trigger Frequency |
| --- | --- | --- | --- | --- |
| [Unit Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.1-unit-testing-with-twister) | Seconds | No (native_sim) | Individual functions | Every PR |
| [Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests) | Minutes | Yes (JTAG access) | Basic firmware boot | Every PR |
| [E2E Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.3-end-to-end-testing-with-pyluwen) | Minutes | Yes (JTAG access) | Firmware communication | Every PR |
| [Stress Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.4-stress-and-long-duration-tests) | Hours | Yes (JTAG access) | Reset reliability (1000+ iterations) | Nightly |
| **Metal Tests** | **Hours** | **Yes (full system)** | **ML workload integration** | **Nightly** |

Metal tests are the most comprehensive but also the slowest, making them suitable for nightly validation rather than per-commit testing.
