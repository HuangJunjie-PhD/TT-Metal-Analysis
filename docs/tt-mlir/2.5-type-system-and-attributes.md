This page documents the type system and attribute infrastructure in tt-mlir, focusing on MLIR attributes that represent tensor layouts, memory configurations, sharding specifications, and hardware characteristics. These attributes enable precise control over how tensors are laid out and computed on Tenstorrent hardware.

For information about the MLIR type system itself (tensor types, scalar types), see the individual dialect documentation in [2.1](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/2.1)[2.2](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/2.2)[2.3](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/2.3) and [2.4](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/2.4) For details on how these attributes are used in optimization and code generation, see [5.2](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/5.2) and chapter [3](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/3)

## Attribute Type Hierarchy

The tt-mlir type system is organized into several attribute categories that span multiple dialects. These attributes serve as metadata for operations and types throughout the compilation pipeline. All attributes are defined using MLIR TableGen (`.td` files) and compiled into C++ classes.

### TableGen to C++ Class Generation

All attributes are defined in TableGen (`.td`) files and automatically generate C++ classes. The generation process follows this pattern:

**Example TableGen definition:**

`// From TTNNOpsAttrs.tddef TTNN_LayoutAttr : EnumAttr<TTNN_Dialect, TTNN_Layout, "layout"> {  let assemblyFormat = "`<` $value `>`";}`
**Generated C++ class:** The C++ implementation for these attributes provides logic for verification and property access. ### Layout Verification Rules

The layout system enforces several constraints through verification functions:

1.   **Grid Shape Verification**[lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp 90-110](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp#L90-L110):

    *   L1 buffer type allows any grid shape.
    *   Non-L1 buffer types (e.g., DRAM) require a unit grid `(1, 1)` for interleaved layouts.

2.   **Buffer and Memory Layout Verification**[lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp 118-138](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp#L118-L138):

    *   `SystemMemory` buffer type cannot have a memory layout.
    *   `DRAM` buffer type does not support `BlockSharded` layout.

3.   **Sharding Verification**[lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp 146-173](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp#L146-L173):

    *   Sharding is only valid for `L1` or `DRAM` buffer types.
    *   Sharding is only valid for `BlockSharded`, `HeightSharded`, or `WidthSharded` layouts.

Sources: [lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp 90-173](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/IR/TTNNOpsAttrs.cpp#L90-L173)

## Sharding Attributes

Sharding attributes specify how tensors are distributed across multiple compute cores.

### Core Placement Attributes

These attributes define core placement at increasing levels of granularity:

*   `CoreCoordAttr`: A single 2D coordinate `(x, y)` in the core grid [include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td 33-51](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td#L33-L51)
*   `CoreRangeAttr`: A rectangular range of cores defined by a `start_coord` and `end_coord`[include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td 53-82](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td#L53-L82)
*   `CoreRangeSetAttr`: A collection of non-intersecting `CoreRangeAttr` objects [include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td 84-122](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/IR/TTNNOpsAttrs.td#L84-L122)

### Conv2d Configuration

The `Conv2dConfig` attribute controls various convolution-specific optimizations:

*   `dtype`: Data type for weights/activations.
*   `weights_dtype`: Specific data type for weights.
*   `activation`: Fused activation.
*   `input_layer_norm`: Whether to apply layer norm to input.
