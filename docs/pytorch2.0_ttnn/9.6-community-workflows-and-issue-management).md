## Purpose and Scope

This document covers the automated community engagement workflows that handle issue and pull request labeling when community members open issues or PRs. This system integrates with centralized Tenstorrent GitHub Actions to maintain consistent labeling and triage across Tenstorrent repositories.

For information about the main testing and deployment workflows, see [GitHub Actions Workflows Overview](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.1-github-actions-workflows-overview). For PR validation and merge queue handling, see [Testing Pipeline and Parallelization](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.2-testing-pipeline-and-parallelization).

## Overview

The pytorch2.0_ttnn repository implements a lightweight community engagement system that delegates issue and PR management to a centralized workflow maintained in the `tenstorrent/tt-github-actions` repository. This approach ensures consistent community interaction patterns across all Tenstorrent projects while minimizing duplication of workflow logic.

The system automatically triggers on two events:

*   New issues are opened
*   New pull requests are opened

When triggered, the workflow applies appropriate labels to help maintainers triage and prioritize community contributions.

**Sources:**[.github/workflows/community-issue-tagging.yml 1-15](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L1-L15)

## Workflow Execution Flow

### Trigger Conditions and Job Delegation

**Sources:**[.github/workflows/community-issue-tagging.yml 3-14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L3-L14)

### Job Definition

The workflow contains a single job named `call-central-workflow` with minimal configuration:

`jobs:  call-central-workflow:    permissions:      issues: write      pull-requests: write    uses: tenstorrent/tt-github-actions/.github/workflows/on-community-issue.yml@main`
The job structure follows GitHub Actions' reusable workflow pattern where:

*   **`permissions`**: Grants the external workflow necessary API access
*   **`uses`**: References the external workflow using repository/path@ref syntax
*   **`@main`**: Pins to the main branch of tt-github-actions (always uses latest version)

**Sources:**[.github/workflows/community-issue-tagging.yml 9-14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L9-L14)

## Integration with External Workflow System

### Centralized Workflow Architecture

The pytorch2.0_ttnn repository does not implement labeling logic internally. Instead, it delegates to a centralized workflow maintained by the Tenstorrent organization:

**External Workflow Reference:**`tenstorrent/tt-github-actions/.github/workflows/on-community-issue.yml`

This architecture provides several benefits:

| Benefit | Description |
| --- | --- |
| **Consistency** | All Tenstorrent repositories use identical community engagement patterns |
| **Maintainability** | Labeling logic updates propagate automatically to all repositories |
| **Reduced Duplication** | Single source of truth for community management workflows |
| **Centralized Monitoring** | Organization-wide analytics on community engagement |

**Sources:**[.github/workflows/community-issue-tagging.yml 14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L14-L14)

## Permission Model

### Required API Permissions

The workflow requires specific GitHub API permissions to modify issue and PR metadata:

`permissions:  issues: write  pull-requests: write`
These permissions enable the workflow to:

| Permission | Scope | Operations Enabled |
| --- | --- | --- |
| `issues: write` | Repository issues | Add/remove labels Modify issue metadata Add comments (if needed) |
| `pull-requests: write` | Repository pull requests | Add/remove labels Modify PR metadata Add comments (if needed) |

The permission model follows GitHub's principle of least privilege - the workflow can only modify labels and metadata, not merge PRs or modify repository settings.

## Operational Considerations

### Workflow Maintenance

Since the workflow delegates to an external repository, updates to labeling logic do not require changes to pytorch2.0_ttnn. However, the following aspects require local maintenance:

| Maintenance Task | Location | When Required |
| --- | --- | --- |
| **Workflow triggers** | [.github/workflows/community-issue-tagging.yml 3-7](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L3-L7) | When changing which events activate labeling |
| **Permission grants** | [.github/workflows/community-issue-tagging.yml 11-13](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L11-L13) | When external workflow requires additional API access |
| **External workflow reference** | [.github/workflows/community-issue-tagging.yml 14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L14-L14) | When changing branch/version pinning strategy |

### Version Pinning Strategy

The workflow uses `@main` branch pinning rather than SHA or tag pinning:

`uses: tenstorrent/tt-github-actions/.github/workflows/on-community-issue.yml@main`
This approach means:

*   The workflow always uses the latest version of centralized logic
*   No manual updates required when tt-github-actions improves labeling
*   Changes propagate immediately across all Tenstorrent repositories
*   Potential risk: breaking changes in tt-github-actions affect all repositories simultaneously

**Sources:**[.github/workflows/community-issue-tagging.yml 14](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/community-issue-tagging.yml#L14-L14)

## Error Handling and Monitoring

### Workflow Failure Scenarios

The workflow can fail under several conditions:

| Failure Scenario | Impact | Resolution |
| --- | --- | --- |
| **tt-github-actions unavailable** | Labels not applied | GitHub automatically retries; manual label application if persistent |
| **Insufficient permissions** | API calls rejected | Update permissions block in workflow file |
| **External workflow syntax error** | Workflow fails to execute | Tenstorrent team must fix tt-github-actions repository |
| **Network/API timeouts** | Temporary failure | GitHub automatically retries failed workflows |

### Monitoring and Debugging

To monitor workflow execution:

1.   **Navigate to Actions tab** in the GitHub repository
2.   **Filter by workflow name**: "Community Issue / PR Labeling Workflow"
3.   **Check run status** for each triggered event
4.   **Review logs** in the `call-central-workflow` job

The workflow logs will show:

*   External workflow invocation
*   Permissions granted
*   Any errors from the centralized labeling system

## Relationship to Repository Workflow Ecosystem

### Position in CI/CD Pipeline

The community workflow operates independently from the testing pipeline. It does not block PRs from entering the merge queue or affect test execution. Its sole purpose is to enhance issue/PR organization for maintainers.
