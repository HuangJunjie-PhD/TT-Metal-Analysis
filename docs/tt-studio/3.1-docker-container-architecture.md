This document describes TT-Studio's Docker container architecture, including service composition, inter-service communication, volume management, and the security layer provided by the Docker Control Service.

## Overview

TT-Studio uses Docker Compose to orchestrate a multi-container application consisting of four primary services running in an external Docker network. A critical architectural feature is the **Docker Control Service**, which runs on the host to provide a secure API for container orchestration, replacing the insecure practice of mounting the Docker socket directly into containers.

## Docker Control Service (Security Layer)

The `docker-control-service` is a FastAPI application running on the host (port 8002) that acts as a secure proxy for the Docker Daemon. It eliminates the need for the `tt_studio_backend` to run as `root` or have direct access to `/var/run/docker.sock`.

### Security Features

*   **JWT Authentication**: All operations (except health checks) require a valid JWT token signed with `DOCKER_CONTROL_JWT_SECRET`[docker-control-service/README.md 18](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L18-L18)
*   **Registry Whitelisting**: Only images from approved registries (e.g., `ghcr.io/tenstorrent/`, `tenstorrent/`) can be pulled or run [docker-control-service/README.md 58-65](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L58-L65)
*   **Resource Constraints**: Enforces `MAX_MEMORY` (16g) and `MAX_CPUS` (8) on all dynamic containers [docker-control-service/README.md 73-74](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L73-L74)
*   **Privilege Blocking**: Explicitly prevents running containers in privileged mode [docker-control-service/README.md 21](https://github.com/tenstorrent/tt-studio/blob/a6d347af/docker-control-service/README.md?plain=1#L21-L21)

### API Implementation

The service uses a `ContainerService` class to interface with the Python Docker SDK, exposing endpoints for the backend to manage model lifecycles.

| Category | Key Endpoints | Code Entity |
| --- | --- | --- |
| **Containers** | `POST /api/v1/containers/run` | `routers.containers.run_container` |

| **Images** | `POST /api/v1/images/pull` | `routers.images.pull_image` |
| **Networks** | `POST /api/v1/networks/create` | `routers.networks.create_network` |

## Core Docker Services

### tt_studio_backend Service

The backend runs the Django REST API. It communicates with the host-level Docker Control Service via `DOCKER_CONTROL_SERVICE_URL`.

*   **Image**: `ghcr.io/tenstorrent/tt-studio/backend:v0.0.1`[app/docker-compose.yml 9](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L9-L9)
*   **Command**: `uvicorn api.asgi:application`[app/docker-compose.yml 24](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L24-L24)
*   **Health Check**: Probes `http://localhost:8000/up/` with a 120s timeout to allow for model resource downloads [app/docker-compose.yml 65-73](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L65-L73)
*   **Networking**: Uses `extra_hosts` to resolve `host.docker.internal` for communicating with the host-bound Docker Control Service [app/docker-compose.yml 75-76](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L75-L76)

### tt_studio_frontend Service

Serves the React application. In development, it uses a Vite dev server; in production, it serves static files via Nginx.

*   **Port Mapping**: `3000:3000` (Dev) [app/docker-compose.dev-mode.yml 31](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.dev-mode.yml#L31-L31) or `3000:80` (Prod) [app/docker-compose.prod.yml 19](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.prod.yml#L19-L19)
*   **Dependency**: Waits for `tt_studio_backend` to be `service_healthy`[app/docker-compose.yml 82-84](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L82-L84)

### tt_studio_agent Service

Handles multi-LLM orchestration and agentic workflows.

*   **Port**: `8080`[app/docker-compose.yml 99](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L99-L99)
*   **Environment**: Includes circuit breaker thresholds for LLM polling (`AGENT_LLM_POLLING_CIRCUIT_BREAKER_THRESHOLD`) [app/docker-compose.yml 110](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L110-L110)

### tt_studio_chroma Service

Vector database for RAG operations.

*   **Persistence**: Mounts `${HOST_PERSISTENT_STORAGE_VOLUME}/chroma` to `/chroma/chroma`[app/docker-compose.yml 127](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L127-L127)
*   **Health Check**: Probes `http://localhost:8111/api/v1/heartbeat`[app/docker-compose.yml 139](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L139-L139)

## Network Architecture

All services communicate through a bridge network named `tt_studio_network`.

*   **External Flag**: The network is marked as `external: true`[app/docker-compose.yml 153](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L153-L153) This prevents Docker Compose from namespacing the network (e.g., `app_tt_studio_network`), allowing the Backend (via the Docker Control Service) to attach dynamically created model containers to the same network by name.
*   **Host Resolution**: Containers use `host.docker.internal` to reach the `docker-control-service` on the host [app/docker-compose.yml 75-76](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L75-L76)
