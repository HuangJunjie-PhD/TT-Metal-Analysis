# DUT Parameters

Relevant source files
*   [BUILD.bazel](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/BUILD.bazel)
*   [README.md](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/README.md?plain=1)

This page documents the configuration of Device Under Test (DUT) specific parameters that control the architectural characteristics of the processor being verified. These parameters define hart count, register widths, and other architectural features that the cosim-arch-checker framework needs to properly interface with the DUT.

For information about configuring the Whisper ISS reference model, see [Whisper Configuration](https://deepwiki.com/tenstorrent/cosim-arch-checker/4.1-whisper-configuration). For build system configuration, see [Build System](https://deepwiki.com/tenstorrent/cosim-arch-checker/5-build-system).

## Purpose and Scope

DUT parameters configure the cosim-arch-checker framework to match the architectural specifications of the target RISC-V processor. These parameters are essential for:

*   Establishing the correct number of hardware threads (harts)
*   Setting register width configurations (XLEN, VLEN)
*   Defining architectural feature support
*   Ensuring proper DPI interface sizing for state extraction

The parameters are primarily defined in compile-time constants and propagated through the build system to ensure type-safe interfaces between the DUT monitoring layer and the verification framework.

## Parameter Overview

The DUT parameters control the fundamental architectural characteristics that the verification framework must understand to properly monitor and compare processor state.

## Core Parameter Definitions

### Hardware Thread Configuration

| Parameter | Type | Purpose | Example Values |
| --- | --- | --- | --- |
| `k_NumHarts` | `const int` | Number of hardware threads (harts) in the DUT | `1` (single-core), `4` (quad-core) |

The `k_NumHarts` parameter determines how many independent execution contexts the verification framework must track. This directly impacts the DPI interface functions which accept hart IDs for state monitoring.

### Register Width Parameters

| Parameter | Type | Purpose | RISC-V Specification |
| --- | --- | --- | --- |
| `k_XLen` | `const int` | Integer register width in bits | `32` (RV32), `64` (RV64) |
| `k_VLen` | `const int` | Vector register width in bits | `128`, `256`, `512`, `1024` |

These parameters control the data type sizes used throughout the verification framework for register state comparison.

## Configuration Methods

### Static Configuration via env/params.h

The primary method for configuring DUT parameters is through compile-time constants defined in the parameter header file:

`// Example DUT configurationconst int k_NumHarts = 1;const int k_XLen = 64;const int k_VLen = 256;`
This approach ensures type safety and allows the compiler to optimize based on known architectural constraints.

### Build-Time Configuration Selection

The framework supports multiple processor configurations through build-time flags:

`bazel build :dpi --copt=-DCONFIG=MediumBoomVecConfig`
The `CONFIG` preprocessor definition allows conditional compilation of different parameter sets for various processor implementations.

## Parameter Validation and Constraints

### Architectural Compatibility

DUT parameters must be compatible with the target processor architecture:

*   `k_XLen` must match the processor's XLEN (32 or 64 bits)
*   `k_VLen` must match the implemented vector register width
*   `k_NumHarts` must correspond to the actual number of hardware threads

### Framework Limitations

The current framework implementation has specific constraints:

*   Vector register monitoring assumes `k_VLen` is a multiple of 64 bits for `uint64_t` array sizing
*   Hart indexing assumes zero-based numbering from 0 to `k_NumHarts-1`
*   CSR address space is assumed to be consistent across all harts

## Build System Integration

The DUT parameters integrate with both Bazel and Makefile build systems:

| Build System | Configuration Method | Parameter Propagation |
| --- | --- | --- |
| Bazel | `--copt=-DCONFIG=<config>` | Preprocessor definitions |
| Makefile | Environment variables | Compiler flags |

The build systems ensure that the DUT parameters are consistently applied across all framework components during compilation.
