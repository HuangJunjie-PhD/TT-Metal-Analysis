## Purpose and Scope

This page documents the virtual memory extension test plans implemented in the coretp framework. It covers test scenarios for two main areas:

1.   **SVADU Extension** - Tests for the Supervisor-mode Virtual Address Update extension, which enables hardware-managed Accessed (A) and Dirty (D) bit updates in Page Table Entries (PTEs)
2.   **Page Table Walks** - Comprehensive tests for page table walk behavior across all supported paging modes (SV39, SV48, SV57)

For detailed scenario descriptions, see [SVADU Test Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.1-svadu-test-scenarios) and [Page Table Walk Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.2-page-table-walk-scenarios). For information about cache management extensions, see [Cache Management Extensions](https://deepwiki.com/tenstorrent/riscv-coretp/4.1-cache-management-extensions-(zicbomzicbozzicbop)). For supervisor mode timer and counter extensions, see [Supervisor Mode Extensions](https://deepwiki.com/tenstorrent/riscv-coretp/4.3-supervisor-mode-extensions).

## SVADU Extension Overview

The SVADU (Supervisor-mode Virtual Address Update) extension enables hardware-managed updates to the Accessed (A) and Dirty (D) bits in Page Table Entries. When enabled via the `menvcfg.adue` bit (bit 61), the hardware automatically sets these bits during memory accesses, eliminating the need for software page fault handlers to manage them.

### SVADU Control and Behavior

The SVADU extension behavior is controlled by the `menvcfg.adue` CSR bit:

| menvcfg.adue | PTE A=0 Load | PTE D=0 Store | Hardware Updates |
| --- | --- | --- | --- |
| 0 (disabled) | Page fault | Page fault | No |
| 1 (enabled) | Success | Success | Yes (for cacheable memory) |

**Key Testing Points:**

*   Fault behavior when SVADU is disabled and A/D bits are clear
*   Hardware update behavior when SVADU is enabled
*   Bit 6 (A bit) and bit 7 (D bit) manipulation and verification
*   Support across all paging modes (SV39, SV48, SV57)

## Page Table Walk Testing

The page table walk test scenarios provide comprehensive coverage of the virtual memory translation process across all supported paging modes. These tests verify correct behavior for both successful translations and fault conditions.

### Paging Modes and Page Sizes

The coretp framework tests three paging modes, each supporting different virtual address widths and page sizes:

| Paging Mode | VA Width | Page Sizes | Test Coverage |
| --- | --- | --- | --- |
| SV39 | 39 bits | 4K, 2M, 1G | VPN[2:0] |
| SV48 | 48 bits | 4K, 2M, 1G, 512G | VPN[3:0] |
| SV57 | 57 bits | 4K, 2M, 1G, 512G, 256T | VPN[4:0] |

## Environment Configuration

Virtual memory tests use `TestEnvCfg` to specify the environment parameters under which tests should execute. The most important configuration parameter for these tests is `paging_modes`.

### Common Configuration Patterns

**SVADU Tests:**

`TestEnvCfg(paging_modes=[PagingMode.SV39, PagingMode.SV48, PagingMode.SV57])`
**Page Table Walk Tests with Privilege Modes:**

`TestEnvCfg(    priv_modes=[PrivilegeMode.U, PrivilegeMode.S, PrivilegeMode.M],    paging_modes=[PagingMode.SV39, PagingMode.SV48, PagingMode.SV57])`
The `TestEnvSolver` (see [Test Environment Configuration](https://deepwiki.com/tenstorrent/riscv-coretp/3.2-test-environment-configuration)) resolves these configurations into concrete `TestEnv` instances, one for each valid combination of parameters.

## Test Steps for Virtual Memory

Virtual memory tests leverage several specialized TestStep types beyond the basic memory operations. The following table summarizes the key steps:

| TestStep Class | Purpose | Usage in VM Tests |
| --- | --- | --- |
| `Memory` | Define memory region with page properties | Every test; specifies `page_size`, `flags`, `base_va`, `base_pa` |
| `Load` | Read from memory | Trigger page table walk, test A bit updates |
| `Store` | Write to memory | Test D bit updates, permission checks |
| `CodePage` | Executable memory region | Test execute permissions, instruction fetch PTW |
| `CsrWrite` | Write CSR | Configure `menvcfg.adue`, `satp` |
| `CsrRead` | Read CSR | Read `satp` for ASID manipulation |
| `ReadLeafPTE` | Read leaf PTE entry | Verify A/D bit state |
| `ModifyPte` | Modify PTE structure | Create fault conditions, recursive PTEs |
| `AssertException` | Verify exception occurs | Test fault behavior |
| `AssertEqual` | Verify values equal | Verify A/D bit state after operations |

### Memory Step Configuration

The `Memory` step is the foundation of virtual memory tests. Key parameters for VM testing:

**Common PageFlags Combinations:**

| Test Type | Flags Used |
| --- | --- |
| Read-only | `VALID | READ` |
| Read-write | `VALID | READ | WRITE` |
| Executable | `VALID | READ | EXECUTE` |
| SVADU A=0 test | `VALID | READ | WRITE` with `exclude_flags=ACCESSED` |
| SVADU D=0 test | `VALID | READ | WRITE | ACCESSED` with `exclude_flags=DIRTY` |
| Global page | `VALID | READ | WRITE | GLOBAL` |

### ReadLeafPTE and Bit Verification

The `ReadLeafPTE` step reads the leaf PTE for a given memory region, returning the PTE value. This is used in SVADU tests to verify A/D bit state:

`# Check if A bit (bit 6) is setread_leaf_pte = ReadLeafPTE(memory=mem)load_immediate_mask_check = LoadImmediateStep(imm=1 << 6)and_op = Arithmetic(op="and", src1=read_leaf_pte, src2=load_immediate_mask_check)assert_equal = AssertEqual(src1=and_op, src2=load_immediate_mask_check)`
**PTE Bit Positions:**

*   Bit 6: Accessed (A) bit
*   Bit 7: Dirty (D) bit

*   [coretp/plans/svadu/svadu_scenarios.py 118-131](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/svadu/svadu_scenarios.py#L118-L131)

### ModifyPte Step

The `ModifyPte` step allows modification of PTE structures to create specific test conditions. It's used to:

*   Create recursive PTE mappings (PTE points to itself)
*   Invalidate PTEs
*   Create reserved PTE encodings

Example from recursive walk test:

`mem = Memory(size=0x1000, page_size=PageSize.SIZE_4K,              flags=PageFlags.VALID | PageFlags.READ, modify=True)modify_pte = ModifyPte(memory=mem, level=0, make_recursive=True)`
The `modify=True` parameter on `Memory` is required to enable PTE modification.

## Summary

The virtual memory extension tests provide comprehensive coverage of RISC-V virtual memory features:

1.   **SVADU Extension**: 4 active test scenarios covering fault behavior and hardware A/D bit updates across all paging modes
2.   **Page Table Walks**: 12 test scenarios covering basic walks, boundary conditions, fault scenarios, and special cases

These tests leverage the coretp framework's compositional TestStep architecture, particularly the specialized `ReadLeafPTE` and `ModifyPte` steps for low-level PTE manipulation and verification.

For implementation details of specific test scenarios, see [SVADU Test Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.1-svadu-test-scenarios) and [Page Table Walk Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.2.2-page-table-walk-scenarios).
