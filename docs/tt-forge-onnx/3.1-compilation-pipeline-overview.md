This page describes the end-to-end compilation pipeline in tt-forge-onnx, detailing how models from various ML frameworks (PyTorch, ONNX, TensorFlow, JAX, PaddlePaddle) are transformed into executable binaries for Tenstorrent hardware. The pipeline consists of six major stages: framework ingestion, TVM conversion, Forge graph construction, optimization passes, MLIR lowering, and backend compilation.

For information about specific optimization passes, see [Graph Optimization Passes](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.2-tvm-frontend-and-relay-ir). For details on MLIR generation, see [MLIR Lowering](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.3-mlir-backend-and-code-generation). For runtime execution, see [Runtime Execution](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.6-runtime-execution-model).

## Stage 2: TVM Conversion

Framework models are converted to TVM Relay IR, which serves as a common intermediate representation.

**TVM Conversion Flow**

Key functions for TVM conversion:

*   **`load_tvm_graph()`**: Main entry point for TVM graph loading ([forge/forge/tvm_calls/forge_compile.py 79-135](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L79-L135))
*   **`compile_tvm_graph()`**: Handles TVM compilation process ([forge/forge/tvm_calls/forge_compile.py 137-232](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L137-L232))
*   **Framework-specific compilers**: 
    *   `compile_pytorch_for_forge()` ([forge/forge/tvm_calls/forge_compile.py 235-320](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L235-L320))
    *   `compile_onnx_for_forge()` ([forge/forge/tvm_calls/forge_compile.py 527-619](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L527-L619))
    *   `compile_tensorflow_for_forge()` ([forge/forge/tvm_calls/forge_compile.py 426-525](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L426-L525))

After conversion, the Relay IR undergoes several transformation passes implemented in [forge/forge/tvm_calls/relay/op/forge_passes.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py):

*   **`ConvertLayout`**: Converts NHWC to NCHW layout ([forge/forge/tvm_calls/relay/op/forge_passes.py 19-97](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L19-L97))
*   **`FuseConvAndPoolPadding`**: Fuses padding into convolution/pooling operations ([forge/forge/tvm_calls/relay/op/forge_passes.py 160-215](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L160-L215))
*   **`DecomposeRoll`**: Decomposes roll operations into slices and concatenation ([forge/forge/tvm_calls/relay/op/forge_passes.py 217-322](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge_passes.py#L217-L322))

## Stage 3: Forge Graph Construction

The Relay IR is partitioned and converted to Forge's internal graph representation.

**Partitioning and Graph Construction**

Key components:

*   **`partition_for_forge()`**: Partitions the Relay graph into CPU and device subgraphs based on operation support ([forge/forge/tvm_calls/relay/op/forge.py 800-900](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L800-L900))
*   **`compile_for_forge()`**: Main compilation function that orchestrates the entire TVM-to-Forge conversion ([forge/forge/tvm_calls/relay/op/forge.py 400-700](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L400-L700))
*   **`ForgeGraphModule`**: C++ class representing the compiled graph module ([forge/forge/_C/__init__.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/_C/__init__.py))

The partitioning process identifies which operations can run on Tenstorrent hardware versus CPU:

*   Operations with native hardware support go to the device graph
*   Unsupported operations remain in CPU pre/post graphs

Sources: [forge/forge/tvm_calls/relay/op/forge.py 1-1000](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/relay/op/forge.py#L1-L1000)[forge/forge/compile.py 800-1000](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L800-L1000)

## Stage 4: Graph Optimization Passes

The Forge graph undergoes multiple optimization passes to improve performance and prepare for lowering.

**Graph Pass Pipeline**

The passes are controlled by the `CompileDepth` enum, which tracks compilation progress:

`enum class CompileDepth {    INIT_COMPILE,    POST_INITIAL_GRAPH_PASS,    CONSTEVAL_GRAPH,    POST_PATTERN_MATCHER,    POST_OPTIMIZE_PASS,    POST_AUTOGRAD_PASS,    PRE_LOWERING,    SPLIT_GRAPH,    RUN_MLIR_COMPILER,    EXECUTE_BINARY}`
Key pass functions in [forge/forge/compile.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py):

*   **`run_post_initial_graph_passes()`**: Initial transformations and shape inference ([forge/forge/compile.py 900-950](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L900-L950))
*   **`run_consteval_graph_pass()`**: Constant folding and evaluation ([forge/forge/compile.py 950-1000](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L950-L1000))
*   **`run_optimization_graph_passes()`**: Performance optimizations ([forge/forge/compile.py 1000-1050](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1000-L1050))
*   **`run_autograd_pass()`**: Generates backward pass for training ([forge/forge/compile.py 1050-1100](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1050-L1100))
*   **`run_pre_lowering_passes()`**: Final transformations before MLIR lowering ([forge/forge/compile.py 1100-1150](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1100-L1150))

For detailed information about each pass, see [Graph Optimization Passes](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.2-tvm-frontend-and-relay-ir).

Sources: [forge/forge/compile.py 800-1200](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L800-L1200)[forge/forge/config.py 1-200](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/config.py#L1-L200)

## Stage 5: MLIR Lowering

The Forge graph is lowered to MLIR (Multi-Level Intermediate Representation) using the TTIR dialect.

**MLIR Generation Architecture**

The core MLIR generation is implemented in the `MLIRGenerator` class in [forge/csrc/passes/lower_to_mlir.cpp 178-1440](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L178-L1440):

**MLIRGenerator Key Methods**

| Method | Purpose | Lines |
| --- | --- | --- |
| `emit_mlir()` | Main entry point, generates MLIR module | [184-259](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/184-259) |
| `emit_mlir_function()` | Generates MLIR function for each graph | [388-509](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/388-509) |
| `emit_mlir_tt_forge_operation()` | Generates MLIR for individual operations | [512-530](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/512-530) |
| `emit_mlir_ttforge_op<>()` | Template for operation lowering | [534-559](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/534-559) |

**Operation Support Check**

Before lowering, the pipeline verifies all operations are supported using `extract_unique_op_configuration()`:

`auto unsupported_op_shapes_attrs =     tt::passes::extract_unique_op_configuration(graph, supported_ops); if (!unsupported_op_shapes_attrs.empty()) {    log_error("Found Unsupported operations...");    throw std::runtime_error("...");}`
The `lowering_handler_map` maps operation names to handlers:

`lowering_handler_map["abs"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::AbsOp>;lowering_handler_map["add"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::AddOp>;lowering_handler_map["matmul"] = &MLIRGenerator::emit_mlir_ttforge_op<mlir::tt::ttir::MatmulOp>;// ... 80+ more operations`
**Attribute Mapping**

The `AttributeMapper` class handles conversion of Forge attributes to MLIR attributes:

`class AttributeMapper {    void initialize_default_mappings() {        // Example: softmax dim -> dimension        add_op_mapping("softmax", "dim",             AttributeRemap("dimension"));                // Example: index parameters with renaming        add_op_mapping("index", "start",             AttributeRemap("begin", TargetType::I32Attr));    }}`
For more details on MLIR generation, see [MLIR Lowering](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.3-mlir-backend-and-code-generation).

Sources: [forge/csrc/passes/lower_to_mlir.cpp 1-1440](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/passes/lower_to_mlir.cpp#L1-L1440)[forge/forge/compile.py 1200-1300](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1200-L1300)

## Stage 6: Backend Compilation

The MLIR module is compiled to a device binary through the TT-MLIR backend compiler.

**Backend Compilation Flow**

The backend compilation is triggered by `run_mlir_compiler()` functions:

**Backend Compilation Options**

| Function | Output | Purpose | File Reference |
| --- | --- | --- | --- |
| `run_mlir_compiler_to_cpp()` | C++ source | Debugging/inspection | [forge/forge/_C/__init__.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/_C/__init__.py) |
| `run_mlir_compiler_to_shared_object()` | Shared library | Alternative runtime | [forge/forge/_C/__init__.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/_C/__init__.py) |
| `run_mlir_compiler()` (default) | Binary flatbuffer | Hardware execution | [forge/forge/compile.py 1300-1400](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1300-L1400) |

The compilation produces a `Binary` object containing the executable code and a `ModelState` for runtime execution:

`binary = run_mlir_compiler(    mlir_module,    chip_ids=[0],  # Target device IDs    compiler_cfg=compiler_cfg) model_state = create_program_state(    binary=binary,    tensor_pool=tensor_pool,    program_type=ProgramType.forward)`
The resulting `ModelState` encapsulates all runtime execution state including:

*   Compiled binary code
*   Memory allocation information
*   Program execution parameters

Sources: [forge/forge/compile.py 1200-1500](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L1200-L1500)[forge/forge/compiled_graph_state.py 1-500](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compiled_graph_state.py#L1-L500)

## Compilation Control Flow

The main compilation orchestration happens in `_compile_module_single_device()`:

**Compilation State Machine**

The `CompileContext` dataclass tracks compilation state:

`@dataclassclass CompileContext:    modules: List[Module]    graph_name: str    compiler_cfg: CompilerConfig    verify_cfg: DeprecatedVerifyConfig    inputs: List[torch.Tensor]    training: bool    graph: Optional[Graph]    stage: CompileDepth  # Current compilation stage    # ... more fields`
**Execution Stage Tracking**

Throughout compilation, execution stages are recorded for debugging and error reporting using the `ExecutionStage` enum:

`class ExecutionStage(Enum):    FAILED_TVM_RELAY_IRMODULE_GENERATION    FAILED_TVM_RELAY_IO_FLATTENING    FAILED_TVM_PATTERN_CALLBACKS    FAILED_FORGE_MODULE_GENERATION    FAILED_FORGE_INITIAL_GRAPH_PASS    FAILED_FORGE_CONSTEVAL    FAILED_FORGE_OPTIMIZATION_GRAPH_PASS    FAILED_FORGE_AUTOGRAD_PASS    FAILED_FORGE_PRE_LOWERING    FAILED_FORGE_MLIR_COMPILATION    FAILED_TTNN_BINARY_EXECUTION    PASSED`
Stages are recorded using `record_execution()` from [forge/forge/forge_property_utils.py 600-700](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L600-L700)

**Compilation Output**

The final output is a `CompiledModel` object that encapsulates:

`class CompiledModel:    compiled_graph_state: CompiledGraphState  # Graph metadata    device_binaries: Dict[int, Binary]        # Compiled binaries per device    model_state: ModelState                    # Runtime execution state        def __call__(self, *inputs):        # Forward pass execution        pass        def backward(self, *losses):        # Backward pass for training        pass`
The `CompiledModel` provides the runtime interface for executing the compiled graph. For details on runtime execution, see [Runtime Execution](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.6-runtime-execution-model).
