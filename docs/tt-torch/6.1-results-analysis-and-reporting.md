This page documents the test result collection, analysis, and reporting infrastructure in tt-torch. The system generates structured test reports, tracks performance metrics, and aggregates results across multiple test runners and hardware configurations.

For information about the actual test execution framework, see [Model Testing Framework](https://deepwiki.com/tenstorrent/tt-torch/4.2.1-model-testing-framework). For test configuration management, see [Test Configuration and Management](https://deepwiki.com/tenstorrent/tt-torch/4.2.2-test-configuration-and-management). For the broader CI/CD pipeline that orchestrates these workflows, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/tt-torch/4.3-cicd-pipeline).

## Overview

The tt-torch project employs a comprehensive results tracking system that:

*   Captures test outcomes in JUnit XML format for CI/CD integration
*   Records performance metrics including PCC (Pearson Correlation Coefficient) and execution time
*   Aggregates results from distributed test execution across hardware architectures
*   Generates summary reports and documentation from test runs
*   Tracks test durations to optimize test splitting strategies

The reporting infrastructure supports both immediate test result analysis and historical trending through standardized artifact formats.

### Duration Tracking

The `.test_durations` file maintains a JSON mapping of test IDs to execution times, enabling intelligent test splitting based on historical runtime data.

**Structure:**

`{  "tests/runner/test_models.py::test_all_models[model/variant]": 45.123,  ...}`
This file is used by pytest-split's `--splitting-algorithm least_duration` strategy to balance test execution time across parallel jobs:

`pytest --splits 8 --group 1 \  --splitting-algorithm least_duration \  --durations-path .test_durations`
Sources: [.test_durations 1-5](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.test_durations#L1-L5)[.github/workflows/run-tt-forge-models-tests.yml 159-162](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-tt-forge-models-tests.yml#L159-L162)

## Test Metadata and Status Tracking

### ModelStatus Enum

The `ModelStatus` enum defines four test classification states that drive pytest markers and test behavior:

Sources: [tests/runner/test_utils.py 15-24](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/test_utils.py#L15-L24)[tests/runner/conftest.py 58-66](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/runner/conftest.py#L58-L66)

**Status Semantics:**

*   **EXPECTED_PASSING**: Production-ready models that should pass on all hardware
*   **KNOWN_FAILURE_XFAIL**: Models with documented issues, marked as xfail with `strict=True` to detect unexpected passes
*   **NOT_SUPPORTED_SKIP**: Models unsupported on specific architectures or too large for single-chip execution
*   **UNSPECIFIED**: New models awaiting triage and classification

## Result Aggregation

### Multi-Runner Aggregation

Test results are aggregated across multiple hardware architectures and test splits using artifact naming conventions:

**Artifact Naming Pattern:**

```
test-reports-models-{runner}-{group}-{job_id}-{filter}
```

Where:

*   `{runner}`: `wormhole` or `blackhole`
*   `{group}`: `grp_1`, `grp_2`, ..., `grp_N` (from matrix splitting)
*   `{job_id}`: Unique GitHub Actions job identifier
*   `{filter}`: Test filter applied (e.g., `expected_passing`)

**Matrix Strategy:**

Sources: [.github/workflows/run-tt-forge-models-tests.yml 44-72](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-tt-forge-models-tests.yml#L44-L72)

### Test Splitting Algorithm

Tests are distributed across groups using pytest-split's `least_duration` algorithm, which balances execution time based on historical data from `.test_durations`:

**Splitting Parameters:**

| Parameter | CLI Flag | Purpose |
| --- | --- | --- |
| Total Groups | `--splits 8` | Number of parallel groups |
| Current Group | `--group 1` | Which group to execute (1-indexed) |
| Algorithm | `--splitting-algorithm least_duration` | Balancing strategy |
| Duration Data | `--durations-path .test_durations` | Historical timing data |

Sources: [.github/workflows/run-tt-forge-models-tests.yml 154-163](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-tt-forge-models-tests.yml#L154-L163)

**Algorithm Behavior:**

This approach minimizes total execution time by preventing "long pole" groups where one group takes significantly longer than others.

## Memory Usage Tracking

The `--log-memory-usage` pytest plugin generates a CSV file tracking memory consumption per test:

**CSV Format:**

`test_name,peak_memory_mb,timestamptests/runner/test_models.py::test_all_models[model1],1234.5,2025-01-15T10:30:00tests/runner/test_models.py::test_all_models[model2],2345.6,2025-01-15T10:35:00`
**Artifact Upload:**

`- name: Upload Memory Usage Log  uses: actions/upload-artifact@v4  with:    name: memory-usage-${{ matrix.runner }}-${{ matrix.build.name }}-...    path: pytest-memory-usage.csv`
Sources: [.github/workflows/run-tt-forge-models-tests.yml 157](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-tt-forge-models-tests.yml#L157-L157)[.github/workflows/run-tt-forge-models-tests.yml 191-196](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/.github/workflows/run-tt-forge-models-tests.yml#L191-L196)
