This document describes the GitHub Actions-based continuous integration and continuous delivery (CI/CD) system for tt-forge-onnx. The pipeline automates building Docker images, compiling wheels, and running comprehensive test suites across multiple hardware platforms.

For information about the test framework and test planning system, see [Test Planning Framework](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.1-test-planning-framework). For details on operator and model testing, see [Operator Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.2-operator-testing) and [Model Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.3-model-testing). For failure classification mechanisms, see [Failure Classification](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.5-failure-classification-system).

## Workflow Architecture

The CI/CD system consists of six primary workflows that coordinate through GitHub Actions' reusable workflow mechanism. The architecture follows a three-tier pattern: entry point workflows that trigger on events, reusable workflows that perform core tasks, and supporting workflows that handle auxiliary functions.

**Workflow Dependency Graph**

**Sources:**[.github/workflows/on-pr.yml 1-149](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/on-pr.yml#L1-L149)[.github/workflows/on-nightly.yml 1-39](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/on-nightly.yml#L1-L39)[.github/workflows/test.yml 1-167](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test.yml#L1-L167)[.github/workflows/produce_data.yml 1-32](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/produce_data.yml#L1-L32)

### Workflow Communication

Workflows communicate through three mechanisms:

1.   **workflow_call**: Reusable workflows accept inputs and produce outputs. For example, `build-image.yml` outputs `docker-image` which is consumed by `build.yml` via `${{ needs.docker-build.outputs.docker-image }}`.

2.   **Artifacts**: The `build.yml` workflow uploads `forge-wheel` artifacts containing `tt_forge_onnx*.whl` and `tt_tvm*.whl` files. The `test-sub.yml` workflow downloads these using `actions/download-artifact@v4`.

3.   **workflow_run**: The `produce_data.yml` workflow triggers automatically when entry point workflows complete, using `github.event.workflow_run.id` to collect data.

**Sources:**[.github/workflows/build-image.yml 17-19](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/build-image.yml#L17-L19)[.github/workflows/build.yml 36-38](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/build.yml#L36-L38)[.github/workflows/test-sub.yml 149-176](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L149-L176)[.github/workflows/produce_data.yml 5-12](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/produce_data.yml#L5-L12)

### Test Splitting Algorithm

The `--splitting-algorithm least_duration` option uses the `.test_durations` file to distribute tests:

1.   **Duration File**: pytest-split plugin reads `.test_durations` containing test execution times from previous runs
2.   **Group Assignment**: Tests are assigned to groups to balance total execution time across groups
3.   **Group Selection**: `--group $test_group_id` selects tests assigned to the current group

This ensures parallel test groups complete in approximately equal time, maximizing throughput.

**Sources:**[.github/workflows/test-sub.yml 212-216](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/.github/workflows/test-sub.yml#L212-L216)

### Artifact Naming Convention

Artifact names use a structured pattern for unique identification:

```
<artifact-type>-<runs-on>-<test_group_id>-[<test_mark>]-<job_id>
```

**Components:**

*   `<artifact-type>`: Type of artifact (collected-test-list, test-log, test-crash-log, memory-usage, test-reports)
*   `<runs-on>`: Hardware runner (n150, n300, p150)
*   `<test_group_id>`: Test group number (1-N)
*   `<test_mark>`: Pytest marker (push, nightly, pr_models_regression) - omitted for some artifacts
*   `<job_id>`: Unique GitHub Actions job ID from `steps.fetch-job-id.outputs.job_id`

Example: `test-log-n150-3-push-12345678` represents the test log from n150 runner, group 3, with push mark, job ID 12345678.

## Best Practices Summary

### Workflow Design Patterns

1.   **Reusable Workflows**: The `test-sub.yml`, `build.yml`, and `build-image.yml` workflows are designed for reuse via `workflow_call`, reducing duplication across entry point workflows.

2.   **Matrix Parallelization**: Test execution uses matrix strategy to parallelize across hardware platforms and test groups, maximizing throughput.

3.   **Artifact Management**: All workflows upload comprehensive artifacts (logs, reports, collected tests) for debugging and analysis.

4.   **Fail-Fast Disabled**: Test matrices use `fail-fast: false` to ensure all test groups complete even if some fail, providing complete coverage data.

5.   **Conditional Builds**: The test workflow supports skipping builds when using pre-built wheels via `workflow_runners` input.

6.   **Crash Handling**: Tests can continue or restart after crashes via the `continue-on-crash` parameter, supporting different debugging workflows.

7.   **Verbose Output**: The `enable-verbose-output` flag provides detailed test execution information when troubleshooting failures.
