## Purpose and Scope

This document describes platform-specific implementations of the RISC-V debug system for different Tenstorrent chip architectures. Each architecture (Blackhole, Wormhole, Quasar) extends the base `BabyRiscDebug` class with hardware-specific workarounds, optimizations, and bug fixes required for reliable debugging on that platform.

For the general RISC-V debugging architecture and base implementation, see [Debug Architecture Overview](https://deepwiki.com/tenstorrent/tt-exalens/6.1-debug-architecture-overview). For execution control operations that work across all platforms, see [Execution Control](https://deepwiki.com/tenstorrent/tt-exalens/6.2-execution-control).

## Platform-Specific Method Overrides

| Method | Base Behavior | Blackhole Override | Wormhole Override | Quasar Override |
| --- | --- | --- | --- | --- |
| `step()` | Execute one instruction via `COMMAND_STEP` | **Executes twice** due to HW bug | Disables branch prediction before step | Inherited from base |
| `cont()` | Resume via `COMMAND_CONTINUE` with debug mode | Inherited from base | Disables branch prediction for FW cores; uses `continue_without_debug()` for erisc | Inherited from base |
| `read_gpr(32)` | Read PC via debug command | Read from `{risc}_pc` debug bus signal | Read from `{risc}_pc` debug bus signal | Read from neo_block debug bus |
| `read_memory_bytes()` | Via `debug_hardware.read_memory()` | **Optimized:** Use NOC access when core not in reset | Inherited from base | Inherited from base |
| `write_memory_bytes()` | Via `debug_hardware.write_memory()` | **Optimized:** Use NOC access when core not in reset | Inherited from base | Inherited from base |
| `invalidate_instruction_cache()` | Write to `RISCV_IC_INVALIDATE_InvalidateAll` | Inherited from base | Inherited from base | Custom: Invalidate + flush PC |

## Blackhole-Specific Implementation

### Double-Step Hardware Bug

Blackhole requires executing the step command twice to advance by a single instruction.

`def step(self):    # There is a bug in hardware and for blackhole step should be executed twice    super().step()    super().step()`
**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 15-18](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L15-L18)

### PC Read Workaround

Reading the program counter (register 32) through the debug interface is unreliable on Blackhole. The implementation reads from the debug bus instead:

`def read_gpr(self, register_index: int) -> int:    if register_index != 32:        return super().read_gpr(register_index)    else:        assert self.noc_block.debug_bus is not None, "Debug bus is not initialized."        return int(self.noc_block.debug_bus.read_signal(self.risc_info.risc_name + "_pc"))`
**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 20-25](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L20-L25)[test/ttexalens/unit_tests/test_risc_debug.py 728-729](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_risc_debug.py#L728-L729)

### Optimized Memory Access

Blackhole can access L1 memory directly via NOC when the core is not in reset, avoiding the slower debug interface:

`def read_memory_bytes(self, address: int, size_bytes: int):    if self.enable_asserts:        self.assert_not_in_reset()     noc_address = self.baby_risc_info.translate_to_noc_address(address)    if noc_address is not None and not self.is_in_reset():        return self.risc_info.noc_block.location.noc_read(noc_address, size_bytes, use_4B_mode=True)    else:        self.assert_trisc2_address(address)        return super().read_memory_bytes(address, size_bytes)`
**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 27-36](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L27-L36)

### TRISC2 Private Memory Bug

Accessing TRISC2 private memory at certain addresses is broken on Blackhole:

`def assert_trisc2_address(self, address: int):    if self.risc_info.risc_name == "trisc2" and address % 16 > 4:        raise TTException(            f"Accessing trisc2 private memory address 0x{address:08x} does not work due to blackhole bug. "            f"For more information see issue #528 in tt-exalens repo."        )`
**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 49-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L49-L54)

## Wormhole-Specific Implementation

### Branch Prediction Workaround

Wormhole requires disabling branch prediction during step and continue operations to avoid core lockups:

`def step(self):    # We need to disable branch prediction as a hardware workaround    if self.baby_risc_info.branch_prediction_register is not None:        self.set_branch_prediction(False)    return super().step() def cont(self):    if self.baby_risc_info.branch_prediction_register is not None:        self.set_branch_prediction(False)        super().cont()    else:        # For erisc, we don't have option to disable branch prediction        self.debug_hardware.continue_without_debug()`
The test suite verifies this workaround prevents core lockups:

**Sources:**[ttexalens/hardware/wormhole/baby_risc_debug.py 13-30](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L13-L30)[test/ttexalens/unit_tests/test_risc_debug.py 535-571](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_risc_debug.py#L535-L571)

### ERISC Continue Without Debug

Ethernet RISC cores on Wormhole do not support branch prediction control, so they use `continue_without_debug()` which omits the `COMMAND_DEBUG_MODE` flag:

`def continue_without_debug(self):    """    Continues RISC-V core execution without debugging functionality.    Note: This function might trigger core lockup - use cont() method instead.    """    if not self.is_halted():        util.WARN(f"Continue: {self.risc_info.risc_name} at {self.risc_info.noc_block.location} is already running")        return    self.__riscv_write(REG_COMMAND, COMMAND_CONTINUE)`
**Sources:**[ttexalens/hardware/wormhole/baby_risc_debug.py 13-24](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L13-L24)[ttexalens/hardware/baby_risc_debug.py 350-368](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L350-L368)

### PC Read via Debug Bus

Like Blackhole, Wormhole reads the PC from debug bus signals:

`def read_gpr(self, register_index: int) -> int:    if register_index != 32:        return super().read_gpr(register_index)    else:        assert self.noc_block.debug_bus is not None, "Debug bus is not initialized."        return int(self.noc_block.debug_bus.read_signal(self.risc_info.risc_name + "_pc"))`
**Sources:**[ttexalens/hardware/wormhole/baby_risc_debug.py 32-37](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L32-L37)

## Quasar-Specific Implementation

### NEO Block Architecture

Quasar's functional workers contain multiple NEO (Neural Engine Operator) blocks, each with its own RISC cores. The debug implementation requires a reference to the NEO block:

`class QuasarBabyRiscDebug(BabyRiscDebug):    def __init__(self, risc_info: BabyRiscInfo, neo_block, enable_asserts: bool = True):        self.neo_block = neo_block        super().__init__(risc_info, enable_asserts)`
**Sources:**[ttexalens/hardware/quasar/baby_risc_debug.py 9-12](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L9-L12)

### Instruction Cache Invalidation

Quasar requires an additional flush operation after invalidating the instruction cache:

`def invalidate_instruction_cache(self):    pc = self.get_pc()        # Execute invalidate instruction cache    register = self.register_store.get_register_description("RISCV_IC_INVALIDATE_InvalidateAll")    self._write_register(register, 0)    self._write_register(register, 1 << (3 - self.risc_info.risc_id))        # Flush PC address to activate instruction cache (needed on Quasar)    assert self.debug_hardware is not None    self.debug_hardware.flush(pc)`
The flush operation uses the `COMMAND_FLUSH` debug command to reset the PC and clear the pipeline:

**Sources:**[ttexalens/hardware/quasar/baby_risc_debug.py 14-27](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L14-L27)[ttexalens/hardware/baby_risc_debug.py 308-321](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L308-L321)

### PC Read from NEO Block

Quasar reads the PC from the NEO block's debug bus rather than the NocBlock's debug bus:

`def read_gpr(self, register_index: int) -> int:    if register_index != 32:        return super().read_gpr(register_index)    else:        assert self.neo_block.debug_bus is not None, "Debug bus is not initialized."        return int(self.neo_block.debug_bus.read_signal(self.risc_info.risc_name + "_pc"))`
**Sources:**[ttexalens/hardware/quasar/baby_risc_debug.py 29-34](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L29-L34)

## Hardware Workarounds Summary

| Platform | Issue | Workaround | Affected Components |
| --- | --- | --- | --- |
| Blackhole | Step advances by 2 instructions | Execute `step()` twice | All cores |
| Blackhole | PC read unreliable | Read from debug bus signal `{risc}_pc` | All cores |
| Blackhole | TRISC2 private memory inaccessible at offsets > 4 in 16-byte blocks | Validate addresses, raise exception | TRISC2 only |
| Wormhole | Core lockup during step/continue | Disable branch prediction before operation | Functional worker cores |
| Wormhole | ERISC cannot disable branch prediction | Use `continue_without_debug()` instead of `cont()` | ERISC only |
| Wormhole | PC read unreliable | Read from debug bus signal `{risc}_pc` | All cores |
| Quasar | Instruction cache not properly invalidated | Flush PC after invalidation | All cores |

**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 15-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L15-L54)[ttexalens/hardware/wormhole/baby_risc_debug.py 13-37](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L13-L37)[ttexalens/hardware/quasar/baby_risc_debug.py 14-27](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L14-L27)[test/ttexalens/unit_tests/test_risc_debug.py 483-571](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_risc_debug.py#L483-L571)
