The `DeviceManager` class provides hardware abstraction and parallelization strategies for training models on CPU, GPU, or Tenstorrent hardware. This system enables experiments to run on single-chip or multi-chip configurations without code changes, using configuration-driven mesh topology and sharding patterns. For information about the broader training infrastructure that consumes `DeviceManager`, see [Manager Classes](https://deepwiki.com/tenstorrent/tt-blacksmith/5.1-manager-classes). For details on specific parallelization strategies, see [Distributed Training](https://deepwiki.com/tenstorrent/tt-blacksmith/8-distributed-training).

## Overview

The device management system abstracts hardware differences and parallelization strategies through a single unified interface. The `DeviceManager` class is instantiated with a `TrainingConfig` object and handles:

*   Hardware detection and initialization (CPU, CUDA, or Tenstorrent)
*   Environment variable configuration for XLA compilation
*   Mesh topology creation for multi-chip parallelism
*   Automatic batch sharding for data parallelism
*   Regex-based model weight sharding for tensor parallelism
*   Synchronized optimizer stepping across devices

Sources: [blacksmith/tools/device_manager.py 1-137](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L1-L137)

## Hardware Initialization Flow

The initialization process branches based on the `use_tt` configuration flag. For Tenstorrent hardware, additional environment setup and mesh creation occurs:

Sources: [blacksmith/tools/device_manager.py 22-49](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L22-L49)

### Single-Chip Initialization

For single-chip training, the `_setup_tt_environment` method configures minimal environment variables:

| Environment Variable | Value | Purpose |
| --- | --- | --- |
| `PJRT_DEVICE` | `"TT"` | Direct PJRT plugin to Tenstorrent device |
| `XLA_STABLEHLO_COMPILE` | `"1"` | Enable StableHLO compilation path |

The device type is set via `xr.set_device_type("TT")`, and the PyTorch device is obtained from `torch_xla.device()`.

Sources: [blacksmith/tools/device_manager.py 38-42](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L38-L42)

### Multi-Chip Initialization

When `mesh_shape` is configured, additional environment variables enable SPMD (Single Program Multiple Data) execution:

| Environment Variable | Value | Purpose |
| --- | --- | --- |
| `XLA_ALWAYS_ALLREDUCE` | `"1"` | Force all-reduce operations for gradient synchronization |
| `CONVERT_SHLO_TO_SHARDY` | `"1"` | Enable Shardy sharding dialect conversion |
| `DISABLE_NUMERIC_CC_TOKEN` | `"1"` | Disable numeric collective-communication tokens |

The `xr.use_spmd()` call activates SPMD mode in the XLA runtime.

Sources: [blacksmith/tools/device_manager.py 44-49](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L44-L49)

## Mesh Configuration and Creation

The mesh abstraction defines the logical topology of devices for distributed training. A mesh is characterized by:

*   **Shape**: A tuple defining the number of devices along each dimension (e.g., `[2, 4]` for 2×4 grid)
*   **Axis names**: Semantic labels for each dimension (e.g., `["data", "model"]`)
*   **Device IDs**: Physical device identifiers assigned to mesh positions

Sources: [blacksmith/tools/device_manager.py 51-69](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L51-L69)

### Mesh Shape Examples

| Configuration | mesh_shape | mesh_axis_names | Topology | Use Case |
| --- | --- | --- | --- | --- |
| Single chip | `null` | `null` | N/A | Single N150 device |
| Data parallel (2-chip) | `[1, 2]` | `["model", "data"]` | 1×2 grid | Replicate model, shard data |
| Tensor parallel (2-chip) | `[1, 2]` | `["data", "model"]` | 1×2 grid | Shard model, replicate data |
| Hybrid (8-chip) | `[2, 4]` | `["data", "model"]` | 2×4 grid | Shard both data and model |

Sources: [blacksmith/experiments/torch/mnist/test_mnist_training.yaml 54-55](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/test_mnist_training.yaml#L54-L55)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training_dp.yaml 54-55](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training_dp.yaml#L54-L55)[blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml 54-55](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml#L54-L55)

## Parallelization Strategy Detection

The `DeviceManager` provides two boolean methods to determine which parallelization strategies are active:

Sources: [blacksmith/tools/device_manager.py 71-77](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L71-L77)

These methods are used throughout the training loop to conditionally apply sharding:

*   `is_data_parallel()`: Returns `True` if a mesh exists with a `"data"` axis of size > 1
*   `is_tensor_parallel()`: Returns `True` if a mesh exists with a `"model"` axis of size > 1

## Batch Preparation and Data Sharding

The `prepare_batch` method handles data transfer to device and optional data-parallel sharding:

Sources: [blacksmith/tools/device_manager.py 115-125](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L115-L125)

### Partition Spec Construction

For a 3D tensor with shape `[batch_size, seq_length, hidden_dim]`, the partition spec would be:

```
partition_spec = ("data", None, None)
```

This specification shards only the first dimension (batch) across the `"data"` mesh axis, replicating the other dimensions. The tensor is marked for sharding using `xs.mark_sharding(tensor, self.mesh, partition_spec)`.

### Usage in Training Scripts

Training scripts call `prepare_batch` before each forward pass:

`# Example from data_parallel trainingbatch = {"inputs": inputs, "targets": targets}batch = device_manager.prepare_batch(batch) outputs = model(batch["inputs"])loss = loss_fn(outputs, batch["targets"])`
Sources: [blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 107-117](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L107-L117)

## Model Sharding for Tensor Parallelism

The `shard_model` method applies tensor parallelism by sharding model weights according to regex-based patterns:

Sources: [blacksmith/tools/device_manager.py 82-110](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L82-L110)

### Sharding Pattern Configuration

The `model_sharding_patterns` field in `TrainingConfig` is a list of `(regex_pattern, sharding_spec)` tuples:

`model_sharding_patterns:    - ["linear_relu_stack\\.0$", [null, "model"]]  # Shard columns    - ["linear_relu_stack\\.2$", ["model", null]]  # Shard rows    - ["linear_relu_stack\\.4$", [null, "model"]]  # Shard columns`
Sources: [blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml 58-61](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml#L58-L61)

### Regex Matching Logic

The method iterates through `model.named_modules()` and applies the first matching pattern:

1.   Check if module has a `weight` attribute
2.   For each pattern in `model_sharding_patterns`: 
    *   Use `re.search(pattern, name)` to match module name
    *   If match found, apply `xs.mark_sharding(module.weight, self.mesh, shard_spec)`
    *   Break after first match (patterns are evaluated in order)

### Sharding Spec Semantics

| Sharding Spec | Meaning | Example Layer |
| --- | --- | --- |
| `[None, "model"]` | Shard columns (dimension 1) | Input layer: split output features |
| `["model", None]` | Shard rows (dimension 0) | Hidden layer: split input features |
| `[None, "model"]` | Shard columns (dimension 1) | Output layer: split output features |

This implements a "column-row-column" sharding strategy common in tensor parallel training.

Sources: [blacksmith/tools/device_manager.py 89-110](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L89-L110)

## Optimizer Step Synchronization

The `optimizer_step` method handles optimizer updates with appropriate synchronization for single-chip and multi-chip configurations:

Sources: [blacksmith/tools/device_manager.py 127-136](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L127-L136)

### Single-Chip Path

For single-chip training (`self.mesh is None`):

1.   Call `optimizer.step()` directly
2.   If using Tenstorrent (`config.use_tt`), call `torch_xla.sync(wait=True)` to ensure XLA execution completes

This path is used for both CPU/GPU and single-chip TT training.

Sources: [blacksmith/tools/device_manager.py 129-133](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L129-L133)

### Multi-Chip Path

For multi-chip training (`self.mesh is not None`):

1.   Call `xm.optimizer_step(optimizer, barrier=True)`
2.   This function forces execution and ensures correct all-reduce operations for gradient synchronization

The `barrier=True` argument ensures all devices complete the optimizer step before proceeding, critical for maintaining consistency in distributed training.

Sources: [blacksmith/tools/device_manager.py 135-136](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L135-L136)

### Usage in Training Loops

All training scripts use the same interface regardless of parallelization strategy:

`# Common pattern across all experimentsloss.backward()device_manager.optimizer_step(optimizer)`
Sources: [blacksmith/experiments/torch/mnist/test_mnist_training.py 129](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/test_mnist_training.py#L129-L129)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 124](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L124-L124)[blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py 132](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py#L132-L132)

## Integration with Training Scripts

The device manager integrates into training scripts through a standard initialization pattern:

Sources: [blacksmith/experiments/torch/mnist/test_mnist_training.py 177-196](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/test_mnist_training.py#L177-L196)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 165-194](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L165-L194)[blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py 174-204](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py#L174-L204)

Within the training loop, the device manager is used at specific points:

1.   **Model initialization**: Move model to device using `device_manager.device`
2.   **Batch preparation**: Call `device_manager.prepare_batch(batch)` for data parallelism
3.   **Model sharding**: Call `device_manager.shard_model(model)` for tensor parallelism
4.   **Optimizer step**: Call `device_manager.optimizer_step(optimizer)` for synchronized updates

This unified interface allows the same training loop code to run across single-chip, data parallel, and tensor parallel configurations by only changing the YAML configuration.

Sources: [blacksmith/experiments/torch/mnist/test_mnist_training.py 61-196](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/test_mnist_training.py#L61-L196)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 56-194](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L56-L194)[blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py 63-204](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py#L63-L204)
