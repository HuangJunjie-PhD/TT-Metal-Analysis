The Bounty Program Workflows automate the lifecycle of community-driven contributions, from initial assignment and terms acceptance to pull request (PR) tracking and final payment processing. These workflows ensure that both contributors and internal maintainers adhere to Service Level Agreements (SLAs) and program requirements.

## Workflow Suite Overview

The bounty system is composed of five specialized workflows, each triggered by specific events in the GitHub lifecycle of an issue or pull request.

| Workflow | Purpose | Trigger |
| --- | --- | --- |
| `bounty-terms.yml` | Informs contributors of legal terms upon assignment. | `workflow_call` (typically on issue assignment) |
| `bounty-pr.yml` | Acknowledges PRs and alerts the team via Slack. | `workflow_call` (typically on PR open/sync) |
| `bounty-pr-review-reminder.yml` | Escalates unassigned bounty PRs to Slack after 24h. | `workflow_call` (typically scheduled) |
| `bounty-stale.yml` | Reminds assigned contributors to provide updates. | `workflow_call` (typically scheduled) |
| `bounty-complete.yml` | Provides payment and feedback instructions on close. | `workflow_call` (typically on issue close) |

## Detailed Workflow Implementation

### PR Acknowledgment and Slack Notification (`bounty-pr.yml`)

This workflow identifies if a new Pull Request is intended to resolve a bounty. It uses the GitHub GraphQL API to traverse the relationship between the PR and its linked issues.

1.   **Linked Issue Resolution**: It fetches `closingIssuesReferences` via the GitHub CLI `gh pr view` command [[.github/workflows/bounty-pr.yml:37-37]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
2.   **Label Verification**: For each linked issue, it executes a GraphQL query to check for the presence of the `bounty` label [[.github/workflows/bounty-pr.yml:45-56]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
3.   **Notification**: If a bounty label is found, it sets the `is_bounty` output to `true`[[.github/workflows/bounty-pr.yml:65-65]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows), posts a comment to the PR [[.github/workflows/bounty-pr.yml:66-66]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows), and sends a message to the internal Slack channel using the `slackapi/slack-github-action`[[.github/workflows/bounty-pr.yml:84-97]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).

### Review Reminders (`bounty-pr-review-reminder.yml`)

To maintain the 24-hour response SLA, this workflow monitors open bounty PRs that lack assigned reviewers.

*   **Age Check**: Calculates the difference between `PR_CREATED_AT` and the current time [[.github/workflows/bounty-pr-review-reminder.yml:74-87]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
*   **Reviewer Check**: Queries `reviewRequests` via the GitHub CLI [[.github/workflows/bounty-pr-review-reminder.yml:97-97]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
*   **Condition**: If the PR is >24 hours old, is linked to a bounty, and has no reviewers, a Slack notification is triggered [[.github/workflows/bounty-pr-review-reminder.yml:119-130]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).

### Stale Bounty Management (`bounty-stale.yml`)

This workflow utilizes the `actions/stale@v9` action to manage contributor inactivity [[.github/workflows/bounty-stale.yml:16-16]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).

*   **Targeting**: It only acts on issues that are assigned and have the `bounty` label [[.github/workflows/bounty-stale.yml:21-22]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
*   **Threshold**: Issues with no activity for 14 days receive a warning message [[.github/workflows/bounty-stale.yml:20-28]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).

### Completion and Payment (`bounty-complete.yml`)

When a bounty issue is closed, this workflow ensures the contributor is provided with the necessary steps to receive payment.

*   **Validation**: Re-verifies the `bounty` label on the closed issue [[.github/workflows/bounty-complete.yml:20-20]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
*   **Instruction**: Posts a comment tagging the assignee with the payment contact email and a feedback survey link [[.github/workflows/bounty-complete.yml:23-27]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).

## Key Security and Permissions

*   **Secrets**: Workflows requiring Slack integration expect a `SLACK_WEBHOOK_URL` secret [[.github/workflows/bounty-pr.yml:6-7]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
*   **Permissions**: 
    *   `pull-requests: write` is required to post comments on PRs [[.github/workflows/bounty-pr.yml:15-15]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
    *   `issues: write` is required to post comments on issues and manage stale status [[.github/workflows/bounty-complete.yml:11-11]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows), [[.github/workflows/bounty-stale.yml:13-13]](https://deepwiki.com/tenstorrent/tt-github-actions/6.2-bounty-program-workflows).
