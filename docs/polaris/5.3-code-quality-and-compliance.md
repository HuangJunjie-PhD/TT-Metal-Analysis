## SPDX License Compliance System

The repository enforces SPDX license compliance through the `spdxchecker.py` tool, which validates that all source files contain proper license and copyright headers.

**SPDX Header Requirements**

The system validates two required SPDX headers in all source files:

| Header Type | Format | Example |
| --- | --- | --- |
| License | `SPDX-License-Identifier: <license>` | `SPDX-License-Identifier: Apache-2.0` |
| Copyright | `SPDX-FileCopyrightText: <copyright>` | `SPDX-FileCopyrightText: (C) 2025 Tenstorrent AI ULC` |

**Status Enumeration**

The `SPDXHeaderStatus` enum defines validation states:

`class SPDXHeaderStatus(Enum):    ST_OK = 'ok'              # Valid header found    ST_MISSING = 'missing'    # Header not found    ST_INCORRECT = 'incorrect'# Header content invalid    ST_ILLFORMED = 'illformed'# Header format invalid`
**Language Parser Architecture**

The `LanguageParser` class [tools/spdxchecker.py 190-277](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/tools/spdxchecker.py#L190-L277) handles file parsing for different programming languages:

*   **Initialization**: Sets up comment syntax patterns and regular expressions
*   **Parsing**: Extracts and validates SPDX headers from source files
*   **Validation**: Checks license and copyright content against allowed values

## Static Analysis Configuration

The repository uses `mypy` for static type checking with specific configuration options to ensure code quality.

**MyPy Configuration Details**

| Setting | Value | Purpose |
| --- | --- | --- |
| `check_untyped_defs` | `true` | Enable checking of untyped function definitions |
| `explicit_package_bases` | `true` | Require explicit package structure |
| `exclude` | `"^__.*/"` | Exclude directories starting with double underscore |
| `warn_unreachable` | `true` | Warn about unreachable code |
| `allow_untyped_globals` | `true` | Allow global variables without type annotations |

**Coverage Exclusion Patterns**

The coverage configuration [pyproject.toml 24-42](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/pyproject.toml#L24-L42) excludes common patterns:

*   Debug code: `"def __repr__"`, `"if self\\.debug"`, `"if DEBUG"`
*   Type checking imports: `"if TYPE_CHECKING"`
*   Defensive assertions: `"raise AssertionError"`, `"raise NotImplementedError"`
*   Non-runnable code: `"if 0:"`, `"if __name__ == .__main__.:""`

Sources: [pyproject.toml 9-58](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/pyproject.toml#L9-L58)[envdev.yaml 23-24](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/envdev.yaml#L23-L24)

## Pre-commit Hooks

The repository uses pre-commit hooks to enforce code quality standards before commits are accepted.

**Hook Configuration**

Each pre-commit hook is configured with specific parameters:

| Hook | Entry Point | Arguments | Dependencies |
| --- | --- | --- | --- |
| `spdx-checker` | `tools/spdxchecker.py` | `--loglevel debug`, `--ignore .github/spdxchecker-ignore.yml` | `pyyaml`, `loguru`, `pydantic` |
| `static checker` | `mypy .` | None | System mypy |
| `unit tests` | `pytest -m unit` | None | System pytest |

**Execution Flow**

1.   **SPDX Validation**: Checks all files for proper license and copyright headers
2.   **Static Analysis**: Runs mypy type checking on the entire codebase
3.   **Unit Testing**: Executes unit tests to ensure code functionality

The `fail_fast: true` setting [.pre-commit-config.yaml 6](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/.pre-commit-config.yaml#L6-L6) ensures that if any hook fails, the commit process stops immediately.

## Testing Framework Configuration

The repository uses pytest with specific markers and configuration options to organize and execute tests effectively.

**Test Markers**

The pytest configuration defines three test markers:

| Marker | Purpose | Usage |
| --- | --- | --- |
| `unit` | General unit tests | Core functionality testing |
| `opunit` | Operator unit tests | Specific operator testing |
| `tools_secondary` | Secondary tool tests | Utility and helper tool testing |

**Coverage Configuration**

The coverage system [pyproject.toml 19-45](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/pyproject.toml#L19-L45) provides:

*   **Branch Coverage**: Tracks both line and branch coverage
*   **Parallel Processing**: Supports data collection from subprocess tests
*   **Test Exclusion**: Omits test files from coverage analysis
*   **HTML Reports**: Generates coverage reports in `__htmlcov` directory

**Test Discovery**

The `pythonpath = "."` setting [pyproject.toml 7](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/pyproject.toml#L7-L7) ensures tests can import modules from the project root directory without installation.

The development environment is configured through `envdev.yaml` with specific tool versions and dependencies to ensure consistent development experiences.

**Development Tool Versions**

| Tool | Version | Purpose |
| --- | --- | --- |
| `mypy` | 1.15.0 | Static type checking |

| `pytest` | 8.3.4 | Testing framework |
| `pydantic` | 2.10.6 | Data validation and settings |

**Type Checking Support**

The environment includes comprehensive type stubs for external libraries:

*   `types-PYYAML`: YAML parsing type support
*   `types-networkx`: Graph library type support
*   `types-requests`: HTTP library type support
*   `lxml-stubs`: XML processing type support

**Environment Creation**

To create the development environment:

`conda env create -f envdev.yamlconda activate polarisdev`
This sets up a complete development environment with all necessary tools for code quality enforcement, testing, and compliance checking.
