This document describes the code ownership structure and review process for the `tt-metal` repository. It explains how the `.github/CODEOWNERS` file assigns review responsibilities to teams and individuals, how the review process integrates with CI/CD, and how test failures are routed to the appropriate owners.

For information about the CI/CD pipeline architecture and test execution, see [CI/CD and Testing Infrastructure](https://deepwiki.com/tenstorrent/tt-metal/6-cicd-and-testing-infrastructure). For details on contributing code and pull request requirements, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/tt-metal/9.5-contributing-guidelines).

* * *

## Purpose and Scope

The CODEOWNERS system in `tt-metal` serves several critical functions:

1.   **Automatic Review Assignment**

 GitHub automatically requests reviews from code owners when pull requests modify the files they own [[.github/CODEOWNERS:4-4]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

2.   **Accountability and Ownership Clarity**

 Defines clear ownership boundaries per subsystem and directory to distribute responsibility [[.github/CODEOWNERS:53-249]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

3.   **Quality and Expertise Enforcement**

 Ensures critical components (e.g., kernel code, runtime system, operation implementations) are reviewed by domain experts [[.github/CODEOWNERS:79-81]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

4.   **Failure Notification Routing**

 Integrates with CI/CD workflows to notify specific reviewers using Slack user IDs when tests fail for their owned components [[.github/workflows/single-card-demo-tests-impl.yaml:48-60]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

5.   **Decentralized Maintenance**

 Enables specialized teams to maintain their respective areas, enhancing scalability and reducing bottlenecks [[.github/CODEOWNERS:4-249]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

## CODEOWNERS File Structure

### Pattern Matching Rules

The `.github/CODEOWNERS` file uses gitignore-style wildcard patterns to assign ownership. Important points:

*   **Order matters**: The _last_ matching pattern in the file takes precedence for a given file or directory [[.github/CODEOWNERS:1-1]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).
*   Patterns can match files or directories, and are hierarchical to override more general ownership.
*   Each matching pattern defines one or more owners — individual GitHub IDs or GitHub teams.
*   The special team `@tenstorrent/codeowner-bypass` is always included for override and emergency fixes.

**Diagram: CODEOWNERS Pattern Matching Flow**

This mechanism guarantees the correct, fine-grained assignment of reviewers triggered automatically for any PR based on changed files.

## Ownership Patterns

### Pattern Hierarchy and Examples

Ownership patterns demonstrate hierarchical specificity:

*   General ownership of `tt_metal/` by infrastructure team ensures broad coverage [[.github/CODEOWNERS:75]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).
*   More specific subdirectories like `tt_metal/api/` override with API owners [[.github/CODEOWNERS:77]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).
*   Extremely specific folders such as `tt_metal/api/tt-metalium/experimental/fabric` have designated individual maintainers [[.github/CODEOWNERS:78]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

This hierarchy supports precise targeting of owners per file.

### Pattern Types and Purposes

| Pattern Type | Example | Purpose |
| --- | --- | --- |
| Directory wildcard | `tt_metal/` | Owns directory and all files within |
| Recursive glob | `tt_metal/**/CMakeLists.txt` | Owns all `CMakeLists.txt` recursively |
| Specific file | `METALIUM_GUIDE.md` | Owns individual file |
| Hardware-specific | `tt_metal/hw` | Owns low-level hardware abstraction code |

This flexibility in patterns allows clear assignment in large codebases.

**Sources**: [[.github/CODEOWNERS:5-90]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process)

## Test Script Organization

### Hardware-Specific Test Scripts

The test suite is organized by hardware platforms and use cases:

*   **Single card tests (N150, N300)** use scripts such as `tests/scripts/single_card/run_single_card_demo_tests.sh` defining functions like `run_resnet_func`, `run_vgg_func`[[tests/scripts/single_card/run_single_card_demo_tests.sh:79-84]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process), [[tests/scripts/single_card/run_single_card_demo_tests.sh:110-114]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).
*   **T3000 multi-device tests** use `tests/scripts/t3000/run_t3000_demo_tests.sh` with test functions for multi-chip demos including `run_t3000_resnet50_tests`, `run_t3000_sd35large_tests`[[tests/scripts/t3000/run_t3000_demo_tests.sh:8-25]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process), [[tests/scripts/t3000/run_t3000_demo_tests.sh:67-69]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).
*   **Performance regression groups** are managed via gated matrices in `.github/workflows/perf-device-models-impl.yaml` which define test sets per hardware cluster, enforcing resource budgets [[.github/workflows/perf-device-models-impl.yaml:134-147]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

### Nightly Regression and Model Validation

Nightly CI runs maintain stable sets of models across device types (e.g., N150, N300) and validate correctness and performance. This is defined with JSON catalogs embedded in workflows such as `[fast-dispatch-full-regressions-and-models-impl.yaml]`[[.github/workflows/fast-dispatch-full-regressions-and-models-impl.yaml:47-70]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

### Test Time Budget Enforcement

Timeouts and maximum runtimes are enforced by verifying `.github/time_budget.yaml` during test matrix generation (e.g., in T3000 demo tests) to ensure CI pipeline throughput [[.github/workflows/t3000-demo-tests-impl.yaml:46-52]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process).

## Bypass Mechanism

The team `@tenstorrent/codeowner-bypass` appears on nearly every ownership line [[.github/CODEOWNERS:4-10]](https://deepwiki.com/tenstorrent/tt-metal/6.12-codeowners-and-review-process). This provides:

*   **Emergency overrides** for urgent fixes blocking the pipeline.
*   **Cross-cutting infrastructure code** changes handled safely with broader consent.
*   Ensures critical maintenance can proceed when primary owners are unavailable.
