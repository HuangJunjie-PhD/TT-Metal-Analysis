## Purpose and Scope

This document describes GCC's pass management system and optimization pipeline, which orchestrates the sequence of compilation passes that transform source code through various intermediate representations (GIMPLE, RTL) to final assembly output. The system is responsible for defining pass execution order, managing pass properties and dependencies, and controlling optimization levels.

For information about specific SFPU custom passes, see [2.2](https://deepwiki.com/tenstorrent/sfpi-gcc/2.2-tenstorrent-sfpu-extensions). For details on vectorization passes specifically, see [5.1](https://deepwiki.com/tenstorrent/sfpi-gcc/5.1-vectorization-system). For middle-end transformation passes in detail, see [5.2](https://deepwiki.com/tenstorrent/sfpi-gcc/5.2-middle-end-transformation-passes).

## Pass Infrastructure Overview

GCC's pass system is built on an object-oriented hierarchy defined in [gcc/tree-pass.h](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h) Each pass is an instance of a pass class that inherits from base classes defining the pass type and behavior.

### Pass Class Hierarchy

Sources: [gcc/tree-pass.h 22-344](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L22-L344)

The base `opt_pass` class ([gcc/tree-pass.h 73-113](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L73-L113)) defines the core pass interface:

*   `gate()` - Returns true if pass should execute for current function
*   `execute()` - Performs the actual transformation
*   `clone()` - Creates a new instance for multiple instantiation
*   `sub` - Pointer to sub-passes (nested passes)
*   `next` - Pointer to next pass in sequence
*   `static_pass_number` - Unique identifier for dump files

### Pass Types

The `opt_pass_type` enum ([gcc/tree-pass.h 31-37](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L31-L37)) defines four pass categories:

| Type | Description | Base Class | Scope |
| --- | --- | --- | --- |
| `GIMPLE_PASS` | GIMPLE tree transformation | `gimple_opt_pass` | Per-function |
| `RTL_PASS` | RTL instruction transformation | `rtl_opt_pass` | Per-function |
| `SIMPLE_IPA_PASS` | Simple inter-procedural | `simple_ipa_opt_pass` | Whole program |
| `IPA_PASS` | Full IPA with summaries | `ipa_opt_pass_d` | Whole program |

Sources: [gcc/tree-pass.h 31-37](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L31-L37)[gcc/tree-pass.h 116-204](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L116-L204)

### Pass Properties

Pass properties ([gcc/tree-pass.h 206-234](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L206-L234)) define the state of the IR and are used to track transformations:

Key properties defined in [gcc/tree-pass.h 206-234](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L206-L234):

*   `PROP_gimple_any` (0x1) - GIMPLE grammar available
*   `PROP_gimple_lcf` (0x2) - Lowered control flow
*   `PROP_gimple_leh` (0x4) - Lowered exception handling
*   `PROP_cfg` (0x8) - Control flow graph built
*   `PROP_ssa` (0x20) - SSA form
*   `PROP_rtl` (0x80) - RTL form
*   `PROP_loops` (0x800) - Loop structures preserved

Sources: [gcc/tree-pass.h 206-234](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L206-L234)

### TODO Flags

TODO flags ([gcc/tree-pass.h 238-324](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L238-L324)) specify cleanup and verification actions to perform after pass execution:

| TODO Flag | Value | Purpose |
| --- | --- | --- |
| `TODO_cleanup_cfg` | 1 << 5 | Clean up control flow graph |
| `TODO_verify_il` | 1 << 6 | Verify intermediate language |
| `TODO_update_ssa` | 1 << 11 | Update SSA form with new PHI nodes |
| `TODO_update_ssa_no_phi` | 1 << 12 | Update SSA without new PHIs |
| `TODO_update_ssa_full_phi` | 1 << 13 | Insert PHI nodes everywhere |
| `TODO_remove_unused_locals` | 1 << 15 | Remove unused local variables |
| `TODO_rebuild_alias` | 1 << 20 | Rebuild alias information |
| `TODO_update_address_taken` | 1 << 21 | Update addressable-vars bitmap |
| `TODO_rebuild_cgraph_edges` | 1 << 22 | Rebuild callgraph edges |

Sources: [gcc/tree-pass.h 238-324](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L238-L324)

## Pass Pipeline Definition

The compilation pass pipeline is defined in [gcc/passes.def](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def) which uses a simple domain-specific language parsed by [gcc/gen-pass-instances.awk](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/gen-pass-instances.awk)

### Pipeline Structure

Sources: [gcc/passes.def 1-469](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L1-L469)

### Pass Definition Syntax

The [gcc/passes.def](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def) file uses these macros ([gcc/passes.def 20-23](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L20-L23)):

*   `INSERT_PASSES_AFTER(group_name)` - Begin a pass group
*   `NEXT_PASS(pass_name)` - Add a pass to current group
*   `PUSH_INSERT_PASSES_WITHIN(container_pass)` - Begin nested sub-passes
*   `POP_INSERT_PASSES()` - End nested sub-passes
*   `TERMINATE_PASS_LIST(group_name)` - End pass group

Example from [gcc/passes.def 30-46](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L30-L46):

```
INSERT_PASSES_AFTER (all_lowering_passes)
  NEXT_PASS (pass_warn_unused_result);
  NEXT_PASS (pass_diagnose_omp_blocks);
  NEXT_PASS (pass_lower_omp);
  NEXT_PASS (pass_lower_cf);
  NEXT_PASS (pass_lower_eh);
  NEXT_PASS (pass_build_cfg);
  ...
TERMINATE_PASS_LIST (all_lowering_passes)
```

Sources: [gcc/passes.def 20-46](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L20-L46)

### Major Pass Groups

#### 1. Lowering Passes (`all_lowering_passes`)

Transforms high-level constructs into GIMPLE form ([gcc/passes.def 30-46](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L30-L46)):

*   `pass_lower_omp` - Lower OpenMP constructs
*   `pass_lower_cf` - Lower control flow (switch, goto)
*   `pass_lower_eh` - Lower exception handling
*   `pass_build_cfg` - Build control flow graph
*   `pass_build_cgraph_edges` - Build callgraph edges

#### 2. Small IPA Passes (`all_small_ipa_passes`)

Early inter-procedural optimizations ([gcc/passes.def 49-151](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L49-L151)):

*   `pass_ipa_free_lang_data` - Free language-specific data
*   `pass_build_ssa_passes` - Build SSA form
*   `pass_local_optimization_passes` - Per-function early optimizations
*   `pass_early_inline` - Inline small functions early

#### 3. Regular IPA Passes (`all_regular_ipa_passes`)

Main inter-procedural optimizations ([gcc/passes.def 153-176](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L153-L176)):

*   `pass_ipa_cp` - Inter-procedural constant propagation
*   `pass_ipa_sra` - Inter-procedural scalar replacement
*   `pass_ipa_inline` - Function inlining
*   `pass_ipa_devirt` - Virtual function devirtualization
*   `pass_ipa_pure_const` - Pure/const function detection
*   `pass_ipa_reference` - Reference analysis

## Pass Registration and Execution

### Pass Registration

Passes are registered through factory functions defined in [gcc/tree-pass.h 354-573](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L354-L573) Each pass has a `make_pass_*` function that creates an instance:

Custom pass registration ([gcc/tree-pass.h 346-352](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L346-L352)):

*   `register_pass()` - Register new pass at specified position
*   Pass positioning: `PASS_POS_INSERT_AFTER`, `PASS_POS_INSERT_BEFORE`, `PASS_POS_REPLACE`

Sources: [gcc/tree-pass.h 328-573](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L328-L573)

### Pass Execution Flow

The main driver in [gcc/toplev.h 30-50](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/toplev.h#L30-L50) creates a `toplev` object that:

1.   Initializes optimization passes via `init_optimization_passes()` ([gcc/toplev.h 54](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/toplev.h#L54-L54))
2.   Processes each compilation unit
3.   Executes pass pipeline for each function

Sources: [gcc/toplev.h 23-64](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/toplev.h#L23-L64)

## Optimization Levels and Pass Selection

### Optimization Level Flags

Optimization levels are defined in [gcc/common.opt 470-488](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L470-L488):

| Option | Level | Description |
| --- | --- | --- |
| `-O0` | None | No optimization (default) |
| `-O1` | Basic | Basic optimizations, fast compilation |
| `-O2` | Moderate | Most optimizations, balanced |
| `-O3` | Aggressive | All optimizations, slower compilation |
| `-Os` | Size | Optimize for code size |
| `-Ofast` | Fastest | `-O3` + fast-math optimizations |
| `-Og` | Debug | Optimize for debugging experience |
| `-Oz` | Smallest | Aggressive size optimization |

Sources: [gcc/common.opt 470-488](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L470-L488)

### Pass Conditional Execution

Many passes check optimization level in their `gate()` function:

Example gates:

*   `pass_vectorize`: Requires `flag_tree_loop_vectorize` and `optimize >= 2`
*   `pass_inline`: Controlled by `optimize` and `flag_inline_functions`
*   `pass_vrp`: Controlled by `flag_tree_vrp`

## Pass Dependencies and TODO Execution

### Pass Property Flow

When a pass declares `properties_required`, the pass manager verifies that previous passes have provided those properties. After execution:

1.   `properties_provided` bits are added to current state
2.   `properties_destroyed` bits are removed from current state
3.   TODO flags are executed to perform cleanup

Sources: [gcc/tree-pass.h 56-64](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L56-L64)

### TODO Flag Execution

Common TODO flag combinations ([gcc/tree-pass.h 312-318](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L312-L318)):

*   `TODO_update_ssa_any` - Any SSA update operation
*   `TODO_verify_all` - All verification checks
*   SSA update variants: 
    *   `TODO_update_ssa` - Standard SSA update with pruned PHI insertion
    *   `TODO_update_ssa_full_phi` - Insert PHI nodes everywhere (expensive)
    *   `TODO_update_ssa_only_virtuals` - Update only virtual operands

Sources: [gcc/tree-pass.h 238-318](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L238-L318)

## Pass Debugging and Dumping

### Dump File Generation

Each pass can generate dump files controlled by:

*   `-fdump-tree-PASSNAME` - Dump GIMPLE pass output
*   `-fdump-rtl-PASSNAME` - Dump RTL pass output
*   `-fdump-ipa-PASSNAME` - Dump IPA pass output

The pass name comes from `pass_data::name` ([gcc/tree-pass.h 47](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L47-L47)):

*   Names starting with `*` suppress dumping
*   `static_pass_number` distinguishes multiple instances

**Developer debug flags** ([gcc/common.opt 768-809](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L768-L809)):

*   `-fdump-passes` - List all passes
*   `-fenable-KIND-PASS` - Enable specific pass
*   `-fdisable-KIND-PASS` - Disable specific pass
*   `-fopt-info` - Report optimization decisions

## Summary

The GCC pass management system provides:

1.   **Object-oriented infrastructure** - `opt_pass` hierarchy for different pass types
2.   **Declarative pipeline definition** - `passes.def` specifies pass ordering
3.   **Property tracking** - `PROP_*` bits track IR state transformations
4.   **TODO system** - Automated cleanup and verification after passes
5.   **Conditional execution** - `gate()` functions enable/disable passes based on flags and targets
6.   **Extensibility** - Custom passes can be registered via `register_pass()`

The pipeline flows from high-level source through multiple lowering and optimization stages:

*   **Lowering** - Convert to GIMPLE
*   **Early IPA** - Build SSA, early inlining
*   **Regular IPA** - Inter-procedural optimizations
*   **Per-function** - Main optimization loop including vectorization
*   **RTL** - Expand to RTL, register allocation, scheduling
*   **Assembly** - Final code emission

This architecture allows GCC to support multiple languages, optimization levels, and target architectures while maintaining a clean separation between pass types and enabling extensive customization.
