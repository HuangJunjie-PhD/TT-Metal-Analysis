## Purpose and Scope

This document explains the architectural design and key patterns used in the chip reset subsystem of tt-tools-common. It covers the delegation pattern that enables gradual migration from legacy to modern reset implementations, driver version requirements that trigger different code paths, pre-flight validation systems, and platform-specific considerations.

For implementation details of specific chip types, see [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation), [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation), [Grayskull Tensix Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.5-grayskull-tensix-reset), and [ChipReset: Modern Unified Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset). For configuration and logging, see [Reset Configuration and Logging](https://deepwiki.com/tenstorrent/tt-tools-common/2.6-reset-configuration-and-logging). For virtualization details, see [Platform Support and Virtualization](https://deepwiki.com/tenstorrent/tt-tools-common/2.7-platform-support-and-virtualization).

## Delegation Pattern for Legacy-to-Modern Migration

The reset system implements a **conditional delegation pattern** that allows gradual migration from legacy reset sequences to the modern unified implementation. This pattern appears in both `BHChipReset` and `WHChipReset`.

### Delegation Decision Logic

**Sources:**[tt_tools_common/reset_common/bh_reset.py 68-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L68-L99)[tt_tools_common/reset_common/wh_reset.py 68-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L68-L90)

### Blackhole Delegation Implementation

The delegation logic in `BHChipReset.full_lds_reset()` evaluates two conditions before choosing a code path:

[tt_tools_common/reset_common/bh_reset.py 90-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L90-L99)

`# Use new reset for driver version >= 2.4.1# or perform reset using ChipReset class which handles Xen HVM caseif is_driver_version_at_least(get_driver_version(), "2.4.1") or check_xen_hvm():    return ChipReset().full_lds_reset(        pci_interfaces,        reset_m3=reset_m3,        silent=silent,        m3_delay=m3_delay,        secondary_bus_reset=secondary_bus_reset    )`
When neither condition is met, the legacy path executes with a deprecation warning:

[tt_tools_common/reset_common/bh_reset.py 101-107](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L101-L107)

**Sources:**[tt_tools_common/reset_common/bh_reset.py 68-224](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L68-L224)

### Wormhole Delegation Implementation

`WHChipReset` follows an identical delegation pattern:

[tt_tools_common/reset_common/wh_reset.py 88-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L88-L90)

**Sources:**[tt_tools_common/reset_common/wh_reset.py 68-90](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L68-L90)

### Migration Timeline

The delegation pattern enables the following migration strategy:

| Phase | Driver Version | Code Path | Status |
| --- | --- | --- | --- |
| Legacy Only | < 1.26.0 | Not supported | Deprecated |
| Legacy Supported | 1.26.0 - 2.4.0 | Legacy reset sequences | Maintained |
| Transition | 2.4.1+ | Modern `ChipReset` | Current |
| Modern Only | Future release | Legacy code removed | Planned |

**Sources:**[CHANGELOG.md 103-107](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L103-L107)[tt_tools_common/reset_common/chip_reset.py 137](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L137-L137)

## IOCTL Reset Flags Enumeration

The modern reset system uses a standardized set of IOCTL operation codes defined in the `IoctlResetFlags` enum:

[tt_tools_common/reset_common/chip_reset.py 25-32](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L25-L32)

`class IoctlResetFlags(IntEnum):    RESTORE_STATE = 0    RESET_PCIE_LINK = 1    CONFIG_WRITE = 2    USER_RESET = 3    ASIC_RESET = 4    ASIC_DMC_RESET = 5    POST_RESET = 6`
### Reset Flag Usage

| Flag | Value | Purpose | When Used |
| --- | --- | --- | --- |
| `RESTORE_STATE` | 0 | Legacy state restoration | Legacy BH/WH only |
| `RESET_PCIE_LINK` | 1 | Secondary bus reset (SBR) | Optional pre-reset phase |
| `CONFIG_WRITE` | 2 | Legacy config space reset | Legacy BH only |
| `USER_RESET` | 3 | Reserved for future use | Not currently used |
| `ASIC_RESET` | 4 | ASIC-only reset | Default reset operation |
| `ASIC_DMC_RESET` | 5 | ASIC + M3/DMC reset | When `reset_m3=True` |
| `POST_RESET` | 6 | Post-reset state restoration | Always after device reappears |

## Three-Phase Reset Sequence

The modern reset implementation follows a three-phase execution model with distinct pre-reset, reset, and post-reset stages.

### Phase Overview

**Sources:**[tt_tools_common/reset_common/chip_reset.py 114-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L114-L288)

### Phase 1: Pre-Reset Setup

Pre-reset activities prepare the system for the reset operation:

1.   **BDF Collection**: Store Bus:Device.Function addresses for each PCI interface ([tt_tools_common/reset_common/chip_reset.py 159-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L159-L165))
2.   **File Descriptor Cleanup**: Force garbage collection to close pyluwen handles ([tt_tools_common/reset_common/chip_reset.py 163-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L163-L165))
3.   **Xenstore Notification**: If Xen HVM, write `1` to xenstore path ([tt_tools_common/reset_common/chip_reset.py 167-193](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L167-L193))
4.   **Secondary Bus Reset**: Optionally perform SBR to reset PCIe link ([tt_tools_common/reset_common/chip_reset.py 195-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L195-L203))

**Sources:**[tt_tools_common/reset_common/chip_reset.py 159-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L159-L203)

### Phase 2: Reset Execution

The actual reset is triggered via IOCTL with one of two flags:

[tt_tools_common/reset_common/chip_reset.py 206-212](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L206-L212)

`for pci_interface in pci_interfaces:    if reset_m3:        reset_flag = IoctlResetFlags.ASIC_DMC_RESET    else:        reset_flag = IoctlResetFlags.ASIC_RESET        self.reset_device_ioctl(pci_interface, reset_flag)`
For Xen HVM systems, the process waits for the host to remove xenstore entries, indicating PCIe hot-unplug completion ([tt_tools_common/reset_common/chip_reset.py 214-256](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L214-L256)).

After issuing resets, the system waits for devices to become available:

[tt_tools_common/reset_common/chip_reset.py 259-265](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L259-L265)

**Sources:**[tt_tools_common/reset_common/chip_reset.py 206-265](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L206-L265)

### Phase 3: Post-Reset Validation

After the wait period, the system validates that devices have returned:

1.   **Device Reappearance**: Poll sysfs for device node ([tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111))
2.   **Post-Reset IOCTL**: Restore device state ([tt_tools_common/reset_common/chip_reset.py 270-278](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L270-L278))
3.   **Success Verification**: Check IOCTL return value

[tt_tools_common/reset_common/chip_reset.py 267-278](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L267-L278)

If post-reset operations fail, the system exits with a non-zero status ([tt_tools_common/reset_common/chip_reset.py 275-278](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L275-L278)).

**Sources:**[tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111)[tt_tools_common/reset_common/chip_reset.py 267-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L267-L288)

## Communication Mechanisms

The reset system uses three distinct hardware communication methods depending on the chip type and code path.

### Communication Method Selection

**Sources:**[tt_tools_common/reset_common/chip_reset.py 60-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L85)[tt_tools_common/reset_common/bh_reset.py 142-144](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L142-L144)[tt_tools_common/reset_common/wh_reset.py 143-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L143-L149)

### IOCTL Communication

The primary mechanism for modern and legacy BH resets. Direct system calls to the kernel driver:

[tt_tools_common/reset_common/chip_reset.py 60-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L85)

**IOCTL Structure:**

*   Input: `output_size_bytes, flags` (8 bytes)
*   Output: `output_size_bytes, result` (8 bytes)
*   Device path: `/dev/tenstorrent/{interface_id}`
*   IOCTL command: `0xFA06` (magic: 0xFA, number: 6)

**Sources:**[tt_tools_common/reset_common/chip_reset.py 56-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L56-L85)

### ARC Message Communication

Legacy Wormhole resets use ARC (Autonomic Reset Controller) messages sent through pyluwen:

[tt_tools_common/reset_common/wh_reset.py 143-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L143-L149)

**ARC Message Types:**

| Message Type | Value | Purpose |
| --- | --- | --- |
| `MSG_TYPE_ARC_STATE3` | 0xA3 | Enter safe state (no pending regulator requests) |
| `MSG_TYPE_TRIGGER_RESET` | 0x56 | Trigger reset sequence |
| `MSG_TRIGGER_SPI_COPY_LtoR` | 0x50 | SPI flash copy (not used in reset) |

The legacy WH sequence sends messages in order:

1.   `MSG_TYPE_ARC_STATE3` with `wait_for_done=True`
2.   Sleep for `A3_STATE_PROP_TIME` (0.03s)
3.   `MSG_TYPE_TRIGGER_RESET` with `wait_for_done=False`

**Sources:**[tt_tools_common/reset_common/wh_reset.py 36-40](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L36-L40)[tt_tools_common/reset_common/wh_reset.py 143-149](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/wh_reset.py#L143-L149)

### NOC Communication

Grayskull Tensix reset uses Network-on-Chip read/write operations directly through pyluwen's NOC interface. This is documented in detail in [Grayskull Tensix Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.5-grayskull-tensix-reset).

**Sources:** Referenced in architecture diagrams, detailed implementation in `gs_reset.py`

## Legacy vs Modern Implementation Comparison

### Architectural Differences

| Aspect | Legacy (BH/WH) | Modern (ChipReset) |
| --- | --- | --- |
| **Driver Version** | 1.26.0 - 2.4.0 | 2.4.1+ |
| **Reset Mechanism** | CONFIG_WRITE (BH), ARC messages (WH) | ASIC_RESET / ASIC_DMC_RESET |
| **Config Space Polling** | Manual byte-by-byte polling | Handled by driver |
| **Device Reappearance** | Immediate (no hotplug) | Explicit wait with BDF matching |
| **Xen Support** | None | Xenstore coordination |
| **Error Handling** | Reset bit polling with timeout | IOCTL return codes |
| **State Restoration** | `RESTORE_STATE` flag | `POST_RESET` flag |

### Migration Benefits

The modern `ChipReset` implementation addresses several limitations of the legacy approach:

1.   **Unified Code Path**: Single implementation for both BH and WH
2.   **Kernel-Managed Hotplug**: Driver handles PCIe device removal/reappearance
3.   **Proper Xen Support**: Xenstore-based coordination for virtualized environments
4.   **Cleaner Error Handling**: IOCTL return codes instead of timeout-based polling
5.   **Secondary Bus Reset**: Optional PCIe link reset before ASIC reset

## Summary of Design Patterns

### 1. Conditional Delegation Pattern

Chip-specific classes delegate to the modern implementation when conditions are met, enabling gradual migration without breaking existing tools.

### 2. Fail-Fast Validation Pattern

Pre-flight checks (driver version, platform, Xen detection) fail immediately with clear error messages, preventing partial reset attempts.

### 3. Three-Phase Reset Pattern

Separation of pre-reset setup, reset execution, and post-reset validation provides clear structure and error isolation.

### 4. Communication Abstraction Pattern

Different hardware communication methods (IOCTL, ARC, NOC) are encapsulated within reset classes, hiding complexity from callers.

### 5. Version-Gated Migration Pattern

Driver version checks gate access to new features while maintaining backward compatibility through legacy code paths.
