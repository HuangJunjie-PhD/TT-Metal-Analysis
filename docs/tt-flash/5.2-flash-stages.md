## Purpose and Scope

This document describes the four-stage firmware flashing pipeline that tt-flash executes when updating Tenstorrent device firmware. Each stage has distinct responsibilities and outputs that feed into the next stage. For details on the firmware bundle structure consumed during SETUP, see [Firmware Bundle Format](https://deepwiki.com/tenstorrent/tt-flash/5.1-firmware-bundle-format). For specifics on version comparison logic executed during FLASH, see [Version Checking and Compatibility](https://deepwiki.com/tenstorrent/tt-flash/5.3-version-checking-and-compatibility). For write/verify mechanics, see [Write and Verify Cycle](https://deepwiki.com/tenstorrent/tt-flash/5.5-write-and-verify-cycle). For platform-specific reset mechanisms, see [Reset Procedures](https://deepwiki.com/tenstorrent/tt-flash/5.6-reset-procedures).

The flashing pipeline is implemented across [tt_flash/main.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py) and [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py) with the orchestration split between user-facing CLI code and core flashing logic.

## Stage Overview

The flash operation proceeds through four sequential stages, each logged with colored output markers. These stages are executed by `main()` in [tt_flash/main.py 204-242](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L204-L242)

**Sources:**[tt_flash/main.py 204-242](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L204-L242)

* * *

## Stage 1: SETUP

The SETUP stage loads and validates the firmware bundle file before any hardware interaction occurs. This stage is implemented in [tt_flash/main.py 213-219](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L213-L219)

### Operations

1.   **Bundle Loading** - Opens the `.fwbundle` tarfile
2.   **Manifest Extraction** - Reads `manifest.json` from archive root
3.   **Version Parsing** - Extracts semantic version tuple `(major, minor, patch)`
4.   **Format Validation** - Ensures version string is valid and manifest contains required fields

### Key Functions

| Function | Location | Purpose |
| --- | --- | --- |
| `load_manifest()` | [tt_flash/main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L201) | Opens tarfile and extracts version tuple |
| `verify_package()` | [tt_flash/flash.py 663-697](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L663-L697) | Full manifest validation including bundle version extraction |

### Version Tuple Format

The manifest version follows a three-part semantic version scheme:

*   Format: `(major, minor, patch)` as integers
*   Example: version `"2.0.0"` becomes `(2, 0, 0)`
*   Used for manifest format versioning, distinct from firmware version

**Sources:**[tt_flash/main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L179-L201)[tt_flash/main.py 213-219](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L213-L219)

## Stage 2: DETECT

The DETECT stage enumerates all Tenstorrent devices on the system via PCI bus scanning. This stage is implemented by a single function call in [tt_flash/main.py 222](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L222-L222)

### Operations

1.   **PCI Enumeration** - Calls `pyluwen` to scan for Tenstorrent devices
2.   **Chip Wrapping** - Creates `WhChip` or `BhChip` objects for each device
3.   **Ethernet Exclusion** - Ignores remote chips connected via Ethernet (uses `ignore_ethernet=True`)
4.   **Device List** - Returns list of local chip objects

### Chip Detection Details

The detection process wraps raw `pyluwen` chip handles into architecture-specific Python objects:

| Chip Type | Wrapper Class | File Location |
| --- | --- | --- |
| Wormhole | `WhChip` | [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py) |
| Blackhole | `BhChip` | [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py) |

The `ignore_ethernet=True` parameter filters out remotely-connected chips that would be accessed over Ethernet links rather than local PCI, as these should not be flashed directly.

**Sources:**[tt_flash/main.py 222](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/main.py#L222-L222)[tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py)

## Stage 3: FLASH

The FLASH stage is the most complex, consisting of multiple substages and per-chip processing. The orchestration occurs in `flash_chips()` at [tt_flash/flash.py 792-959](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L792-L959)

### FLASH Substages

### VERIFY Substage

Located at [tt_flash/flash.py 801-815](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L801-L815) this substage validates the bundle before any hardware writes.

**Key operations:**

*   Call `verify_package(fw_package, version)` at [tt_flash/flash.py 663-697](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L663-L697)
*   Extract `bundle_version` from `manifest.json`
*   Validate bundle version against manifest version requirements
*   Set global `__SEMANTIC_BUNDLE_VERSION` for parameter handlers

### FLASH Step 1: Preparation

Implemented in `flash_chip_stage1()` at [tt_flash/flash.py 194-492](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L194-L492) this function executes per-chip preparation.

**Version normalization:**[tt_flash/flash.py 127-139](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L127-L139) handles legacy version format where `(80, major, minor, patch)` is converted to `(major, minor, patch, 0)`.

**Board-specific handling:**

*   **Blackhole:** Uses `boot_fs_write()` from [tt_flash/blackhole.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py) for boot filesystem layout
*   **Wormhole:** Uses `TAG_HANDLERS` dict at [tt_flash/flash.py 142-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L142-L148) to process parameters

**Return values:**

*   `FlashStageResult.Ok` with `FlashData` containing `write` operations
*   `FlashStageResult.NoFlash` when firmware is already current
*   `FlashStageResult.Err` on validation failure

**Sources:**[tt_flash/flash.py 194-492](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L194-L492)

### FLASH Step 2: Write and Verify

Implemented in `flash_chip_stage2()` at [tt_flash/flash.py 495-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L495-L650) this function performs the actual SPI flash operations.

**SIGINT Protection:** Functions `perform_write()` and `perform_verify()` at [tt_flash/flash.py 503-533](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L503-L533) install a signal handler that blocks Ctrl-C during flash operations to prevent corruption.

**Retry Logic:** If initial verification fails:

1.   Print mismatch details (first mismatch offset, total mismatch count)
2.   Retry write operation
3.   Verify second attempt
4.   Return `None` if second verification fails (requires support contact)

**NEBULA_X2 Special Handling:**[tt_flash/flash.py 620-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L620-L650) triggers SPI copy from local to remote chip using `MSG_TRIGGER_SPI_COPY_LtoR` ARC message. This includes mitigation for M3 app version 5.8.0.1 bootloop bug by disabling auto-reset.

**Sources:**[tt_flash/flash.py 495-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L495-L650)

### Per-Chip Processing Loop

The main loop at [tt_flash/flash.py 846-882](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L846-L882) processes each detected chip:

| Loop Iteration | Function Called | Produces |
| --- | --- | --- |
| First loop (lines 846-869) | `flash_chip_stage1()` | `FlashStageResult` with preparation data |
| Second loop (lines 873-882) | `flash_chip_stage2()` | Write/verify result, return code accumulation |

Chips requiring reset are accumulated in `needs_reset_wh` and `needs_reset_bh` lists by PCI interface ID.

**Sources:**[tt_flash/flash.py 792-959](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L792-L959)

## Stage 4: RESET

The RESET stage applies platform-specific reset procedures to load the newly flashed firmware. This stage is implemented at [tt_flash/flash.py 890-952](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L890-L952)

### Reset Decision Logic

### Platform-Specific Reset Methods

| Platform | Function | Location | Mechanism |
| --- | --- | --- | --- |
| Galaxy (WH_UBB) | `glx_6u_trays_reset()` | [tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789) | IPMI reset commands via `pyluwen` |
| Galaxy (GALAXY-1) | `glx_6u_trays_reset()` | [tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789) | IPMI reset commands via `pyluwen` |
| Standalone Wormhole | `WHChipReset.full_lds_reset()` | `tt_tools_common.reset_common.wh_reset` | PCI link reset with M3 recovery |
| Standalone Blackhole | `BHChipReset.full_lds_reset()` | `tt_tools_common.reset_common.bh_reset` | PCI link reset with M3 recovery |

### Galaxy Reset Sequence

For Galaxy systems (all chips are `WH_UBB` or `GALAXY-1`), the reset procedure at [tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789):

1.   Call `run_wh_ubb_ipmi_reset()` from `pyluwen` with parameters
2.   Wait 30 seconds for trays to complete reset (`live_countdown`)
3.   Call `run_ubb_wait_for_driver_load()` to wait for kernel driver
4.   Re-enumerate chips with `detect_chips_with_callback(local_only=True, ignore_ethernet=True)`
5.   For full Galaxy reset (`ubb_num == 0xF`), validate Ethernet link status via `check_wh_galaxy_eth_link_status()`

**Ethernet Link Validation:**[tt_flash/flash.py 700-735](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L735) checks for `LINK_INACTIVE_FAIL_DUMMY_PACKET` errors on all 16 Ethernet links across 32 devices.

### Adaptive M3 Boot Timeout

At [tt_flash/flash.py 893-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L893-L900) the system determines M3 boot timeout:

`m3_delay = 20  # Default: 20 secondsrunning_version = chip.get_bundle_version().runningif (running_version is None) or (running_version[0] != manifest.bundle_version[0]):    # Crossed major version boundary    print("Detected update across major version, will wait 60 seconds...")    m3_delay = 60`
Major version upgrades receive 3x timeout to handle longer initialization sequences.

### Post-Flash Validation (Blackhole ≥ v19)

For Blackhole chips with firmware version 19+, cryptographic validation occurs at [tt_flash/flash.py 942-951](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L942-L951):

1.   Generate random check value (16-bit)
2.   Send `MSG_CONFIRM_FLASHED_SPI` with check value as `arg0`
3.   Verify response matches check value
4.   Warn if validation fails (suggests reset needed)

This confirms the new firmware is loaded and responding correctly via ARC message challenge-response.

**Sources:**[tt_flash/flash.py 890-952](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L890-L952)[tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789)[tt_flash/flash.py 700-735](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L735)

## Stage Transition and State Management

### Return Code Accumulation

The flash operation accumulates errors across all chip operations:

`rc = 0  # Initialize at flash.py:870for chip, data in flash_data:    result = flash_chip_stage2(chip, data)    if result is None:        rc += 1  # Increment on write/verify failure`
Final status determination at [tt_flash/flash.py 953-956](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L953-L956):

`if rc == 0:    print(f"FLASH {CConfig.COLOR.GREEN}SUCCESS{CConfig.COLOR.ENDC}")else:    print(f"FLASH {CConfig.COLOR.RED}FAILED{CConfig.COLOR.ENDC}")return rc`
### Global State: Bundle Version

The global variable `__SEMANTIC_BUNDLE_VERSION` at [tt_flash/flash.py 102](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L102-L102) is set during VERIFY substage and consumed by the `bundle_version` TAG_HANDLER at [tt_flash/flash.py 105-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L105-L124) to inject bundle version into parameter regions during image processing.

**Sources:**[tt_flash/flash.py 870-959](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L870-L959)[tt_flash/flash.py 102-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L102-L124)
