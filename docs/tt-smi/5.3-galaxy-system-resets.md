## Purpose and Scope

This page documents reset operations specific to Galaxy 6U systems (Wormhole and Blackhole variants). Galaxy systems use IPMI-based reset commands instead of standard PCI resets due to their unique tray-based architecture. This page covers the reset process flow, tray management, Ethernet validation, and auto-retry mechanisms.

For standard PCI device resets (Wormhole and Blackhole boards), see [Standard PCI Device Resets](https://deepwiki.com/tenstorrent/tt-smi/5.2-standard-pci-device-resets). For general reset architecture concepts, see [Reset Architecture and Types](https://deepwiki.com/tenstorrent/tt-smi/5.1-reset-architecture-and-types).

* * *

## Galaxy System Architecture

Galaxy 6U systems consist of **4 trays** with **8 devices per tray**, totaling **32 devices** per system. Each tray is identified by a unique UBB (Universal Baseboard) bus ID, which differs between Wormhole and Blackhole variants.

### UBB Bus ID Mappings

The bus ID mappings are platform-specific and defined in `constants.py`:

**Wormhole Galaxy (`WH_UBB_BUS_IDS`):**

| Tray Number | UBB Bus ID | Devices |
| --- | --- | --- |
| 1 | 0xC0 | 0-7 |
| 2 | 0x80 | 8-15 |
| 3 | 0x00 | 16-23 |
| 4 | 0x40 | 24-31 |

**Blackhole Galaxy (`BH_UBB_BUS_IDS`):**

| Tray Number | UBB Bus ID | Devices |
| --- | --- | --- |
| 1 | 0x00 | 0-7 |
| 2 | 0x40 | 8-15 |
| 3 | 0xC0 | 16-23 |
| 4 | 0x80 | 24-31 |

Note that the bus ID assignments differ between architectures, reflecting hardware addressing differences.

## Reset Process Flow

Galaxy resets use IPMI commands to control trays and require significantly longer wait times than standard PCI resets. The process involves issuing IPMI commands, waiting for hardware stabilization, waiting for driver reload, and validating Ethernet connectivity.

**Sources:**[tt_smi/tt_smi_backend.py 860-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L928)

## IPMI Reset Command Parameters

The `glx_6u_trays_reset()` function accepts several parameters that control the IPMI reset behavior:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `reinit` | bool | `True` | Whether to re-initialize devices after reset |
| `ubb_num` | str | `"0xF"` | UBB bitmap (0x0-0xF). 0xF = all 4 trays |
| `dev_num` | str | `"0xFF"` | Device bitmap per tray (0x0-0xFF). 0xFF = all 8 devices |
| `op_mode` | str | `"0x0"` | Operation mode: 0x0 = Assert/Deassert with period 0x1 = Assert only 0x2 = Deassert only |
| `reset_time` | str | `"0xF"` | Reset duration in 10ms units (0xF = 150ms) |
| `print_status` | bool | `True` | Whether to print status animations |

The IPMI command is executed as:

```
sudo ipmitool raw 0x30 0x8B <ubb_num> <dev_num> <op_mode> <reset_time>
```

**Sources:**[tt_smi/tt_smi_backend.py 860-883](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L883)[README.md 206-232](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/README.md?plain=1#L206-L232)

## CLI Commands for Galaxy Resets

TT-SMI provides three CLI options for Galaxy system resets:

### Full Galaxy Reset

Resets all 4 trays (32 devices). Performs Ethernet link validation after reset. Exits with error if any link failures are detected.

### Auto-Retry Reset

Same as `-glx_reset` but automatically retries up to **3 times** if Ethernet link failures are detected.

### Single Tray Reset

Resets a specific tray. **Skips Ethernet validation** because partial resets break Ethernet training synchronization across the system.

### Skip Re-initialization

The `--no_reinit` flag can be combined with any reset command to skip device re-detection after reset, allowing for faster operation when re-initialization is not needed.

## Post-Reset Ethernet Validation

For Wormhole Galaxy systems, the `check_wh_galaxy_eth_link_status()` function validates that all Ethernet links are operational after a full galaxy reset.

### Validation Process

1.   **Device Count Check**: Verifies exactly 32 devices are present
2.   **Link Error Detection**: For each device, checks all 16 Ethernet ports by reading NOC memory at address `0x12c0 + 0x4*96`
3.   **Error Condition**: Link error value of `10` (LINK_INACTIVE_FAIL_DUMMY_PACKET) indicates a failed link
4.   **Reporting**: Collects all link errors and reports them with device and port numbers

### Ethernet Port Locations

The 16 Ethernet ports per device are located at specific NOC coordinates on NOC 0:

| Port | (x, y) | Port | (x, y) |
| --- | --- | --- | --- |
| 0 | (9, 0) | 8 | (9, 6) |
| 1 | (1, 0) | 9 | (1, 6) |
| 2 | (8, 0) | 10 | (8, 6) |
| 3 | (2, 0) | 11 | (2, 6) |
| 4 | (7, 0) | 12 | (7, 6) |
| 5 | (3, 0) | 13 | (3, 6) |
| 6 | (6, 0) | 14 | (6, 6) |
| 7 | (4, 0) | 15 | (4, 6) |

If any link errors are detected, the function raises an exception with details about failed links, causing the reset to be reported as failed.

## Implementation Details

### Key Functions and Data Flow

### Core Reset Function

The `glx_6u_trays_reset()` function in [tt_smi/tt_smi_backend.py 860-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L928) orchestrates the entire Galaxy reset sequence:

1.   **Issue IPMI Reset**[line 889](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/line%20889): Calls `run_wh_ubb_ipmi_reset()` with parameters
2.   **Wait 30 Seconds**[line 890](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/line%20890): Hardware stabilization period using `timed_wait(30)`
3.   **Wait for Driver**[line 891](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/line%20891): Calls `run_ubb_wait_for_driver_load()` to ensure driver is ready
4.   **Re-detect Chips**[lines 904-907](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/lines%20904-907): Calls `detect_chips_with_callback(local_only=True, ignore_ethernet=True)` to discover devices
5.   **Validate Ethernet**[lines 918-921](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/lines%20918-921): If full reset (ubb_num == 0xF), calls `check_wh_galaxy_eth_link_status()`
6.   **Exit**[lines 923-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/lines%20923-928): Exits with success (code 0) or error (code 1)

### Wait Time Justification

The 30-second wait period is necessary for:

*   ASIC power-on sequences to complete
*   Board-level firmware initialization
*   PCIe enumeration and link training
*   System-level stabilization before driver interaction

**Sources:**[tt_smi/tt_smi_backend.py 860-928](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L860-L928)[tt_smi/tt_smi_backend.py 819-858](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L819-L858)[tt_smi/tt_smi_backend.py 808-817](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L808-L817)

## Auto-Retry Logic

The `-glx_reset_auto` option implements automatic retry logic up to 3 times if Ethernet link failures are detected. This is handled at the CLI level by catching exceptions from `glx_6u_trays_reset()` and retrying the operation.

The retry mechanism is particularly useful for Galaxy systems where Ethernet training can occasionally fail due to timing variations during concurrent resets across multiple devices.

**Note:** The actual retry implementation is in the main CLI entry point, not in the backend function. The backend function always exits with `sys.exit()` on completion or failure.

**Sources:**[README.md 120-121](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/README.md?plain=1#L120-L121)

## Limitations and Considerations

### Partial Reset Ethernet Skipping

When resetting a single tray (`-glx_reset_tray`), Ethernet validation is **skipped**[tt_smi/tt_smi_backend.py 918-921](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L918-L921) This is because:

*   Partial resets break Ethernet training synchronization across the Galaxy mesh
*   Devices on unreset trays maintain their Ethernet state
*   Re-training would require a full system reset

### CPLD Firmware Requirements

Galaxy systems require **CPLD firmware version 1.16 or higher** for `tt-smi -r` (standard PCI reset) to work correctly [tt_smi/tt_smi_backend.py 770-777](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L770-L777) Systems with older CPLD versions should continue using `-glx_reset` IPMI-based resets.

### VM Environment Restriction

The `-glx_list_tray_to_device` command requires **direct hardware access** and should not be run in virtual machine environments [README.md 233](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/README.md?plain=1#L233-L233)

### Driver Load Timeout

The `run_ubb_wait_for_driver_load()` function polls for driver readiness but has an internal timeout. If the driver fails to load within the timeout period, the reset operation will fail.
