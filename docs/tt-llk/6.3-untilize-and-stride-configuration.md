This page documents the untilize operation in the packer system and its associated stride configuration. Untilize is the process of converting data from tiled format (organized as 16×16 faces) back to row-major format in L1 memory. This is the inverse of the tilize operation performed by the unpacker (see [Tilize and Untilize Operations](https://deepwiki.com/tenstorrent/tt-llk/4.2-tilize-and-untilize-operations)).

For general packer operations and configuration, see [Core Packer Operations and Configuration](https://deepwiki.com/tenstorrent/tt-llk/6.1-core-packer-operations-and-configuration). For destination register management during packing, see [Destination Register Management](https://deepwiki.com/tenstorrent/tt-llk/6.2-destination-register-management).

* * *

## Untilize Operation Overview

The untilize operation reads tiles from destination registers and writes them to L1 memory in row-major format. Each tile is organized as four 16×16 faces (Face 0, Face 1, Face 2, Face 3), and untilize must reorganize these faces into contiguous rows.

### Data Layout Transformation

**Diagram**: Untilize data transformation from tiled to row-major format

During untilize:

*   **Face 0** (rows 0-15, columns 0-15) and **Face 1** (rows 0-15, columns 16-31) are interleaved to form output rows 0-15
*   **Face 2** (rows 16-31, columns 0-15) and **Face 3** (rows 16-31, columns 16-31) are interleaved to form output rows 16-31
*   Each PACR instruction packs 2×16 datums when `num_faces > 1`, writing one complete row across multiple tiles in a block

## Stride Configuration

Packer strides control how the packer hardware increments through source (destination register) and destination (L1 memory) addresses. The `set_packer_strides` function configures these strides based on the operation mode.

### Stride Parameters

| Stride | Purpose | Untilize Configuration | Normal Pack Configuration |
| --- | --- | --- | --- |
| **X Stride** | Column increment | Not used (Y stride controls row traversal) | `x_stride` datums (2, 4, or 1 bytes) |
| **Y Stride** | Row increment | `FACE_C_DIM * x_stride` bytes | `FACE_C_DIM * x_stride` bytes |
| **Z Stride** | Face increment | `2 * FACE_R_DIM * y_stride` (for full tiles) | `FACE_R_DIM * y_stride` |
| **W Stride** | Tile increment | `TILE_NUM_FACES * FACE_C_DIM * FACE_R_DIM * x_stride` | Same as normal |

**Diagram**: Packer stride calculation flow

The Z stride for untilize mode is doubled (`2 * FACE_R_DIM * y_stride`) because the packer has two interfaces active, each handling alternate faces. The Z counter increments by 1 but effectively jumps by 2 faces.

**Sources**: [tt_llk_blackhole/common/inc/cpack_common.h 166-199](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/cpack_common.h#L166-L199)[tt_llk_blackhole/llk_lib/llk_pack_untilize.h 162-168](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L162-L168)

## Block and Full Dimension Parameters

Untilize supports packing multiple tiles horizontally in a block. The `block_ct_dim` and `full_ct_dim` template parameters control this behavior.

### Block Configuration

**Diagram**: Block-based untilize processing for wide tensors

| Parameter | Description | Constraints |
| --- | --- | --- |
| `block_ct_dim` | Number of tiles processed in a single untilize call | ≤ 8 tiles (destination register capacity) |
| `full_ct_dim` | Total tensor width in tiles | Must be divisible by `block_ct_dim` |
| Row offset | Address increment per row | `SCALE_DATUM_SIZE(format, full_ct_dim * num_datums)` |

For tensors wider than 8 tiles, the operation is split into multiple blocks. Each `_llk_pack_untilize_` call processes `block_ct_dim` tiles, and the L1 address increments by `full_ct_dim` worth of datums to move to the next row.

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 36-44](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L36-L44)[tt_llk_blackhole/llk_lib/llk_pack_untilize.h 169-178](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L169-L178)

## MOP Configuration for Untilize

The Micro-Operation (MOP) template for untilize uses a two-level loop structure with specialized address modifiers.

### MOP Loop Structure

**Diagram**: Untilize MOP execution flow with nested loops

### Key MOP Components

**Outer Loop** (iterates over rows):

*   Loop count = `face_r_dim` (typically 16)
*   Each iteration processes one complete row across all tiles in the block
*   After each row, Y counter increments and L1 address updates

**Inner Loop** (iterates over tiles):

*   Loop count = `block_ct_dim` (number of tiles in block)
*   Each iteration packs one row segment from one tile
*   W counter increments to move to the next tile in destination

**Start Operation**:

Sets W counter to prepare for first tile access with proper offset.

**End Operations**:

*   `TT_OP_INCADCXY`: Increments Y counter to next row
*   Replay buffer: Updates L1 address for next row using `OUTPUT_ADDR_OFFSET`

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 36-133](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L36-L133)

## Address Modifier Configuration

The untilize operation uses a single address modifier (ADDR_MOD_0) configured for strided access.

**Diagram**: Address modifier configuration for untilize

Unlike normal packing which uses Y increments for row traversal, untilize relies on:

1.   **W counter increments** to move between tiles in a block
2.   **Y counter increments** (via `TT_OP_INCADCXY`) to move to the next row within a face
3.   **L1 address updates** (via replay buffer) to move to the next output row in memory

The DST_ACCESS_STRIDED_MODE in PACR instructions causes each packer interface to have a stride of `16 * block_size`, where block_size is the size of one row within a face.

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 22-30](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L22-L30)

## Narrow Row Support

Narrow row mode supports untilizing tiles where the row width is less than the full tile width (32 datums). This is useful for tensors with widths that don't align to full tile boundaries.

### Narrow Row Configuration

When `narrow_row = true` and `row_num_datums < TILE_C_DIM`:

*   Only a **single packer interface is active** (`PACK_INTF_SEL = p_pacr::SINGLE_INTF_ACTIVE`)
*   X counter end is set to `row_num_datums - 1` instead of `FACE_C_DIM - 1`
*   Output address offset is calculated as `SCALE_DATUM_SIZE(format, full_ct_dim * row_num_datums)`
*   Faces are stored in the first column of the tile

| Mode | Packer Interfaces | X Counter End | Row Width |
| --- | --- | --- | --- |
| Normal (full tile) | 2 active (if num_faces > 1) | `FACE_C_DIM - 1` (15) | 32 datums |
| Narrow row | 1 active | `row_num_datums - 1` | < 32 datums |

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 143-193](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L143-L193)

## Initialization and Execution Flow

### Complete Untilize Workflow

**Diagram**: Untilize operation execution sequence

### Key Functions

**`_llk_pack_untilize_init_<block_ct_dim, full_ct_dim, diagonal, narrow_row, row_num_datums>()`**

*   Sets up address modifiers via `_llk_pack_untilize_configure_addrmod_`
*   Configures MOP template via `_llk_pack_untilize_mop_config_`
*   Sets Z stride to `2 * face_r_dim * y_stride` for untilize mode
*   Calculates and stores `OUTPUT_ADDR_OFFSET` for row-to-row L1 address increments
*   Configures X counter end based on `narrow_row` and `row_num_datums`

**`_llk_pack_untilize_<block_ct_dim, full_ct_dim, diagonal, narrow_row, row_num_datums, tile_dst_ct_offset>()`**

*   Programs packer destination address
*   Resets ZW and XY counters
*   Reconfigures MOP if `tile_dst_offset` changed (performance optimization)
*   Executes MOP for each face group (top faces, then bottom faces if `num_faces > 2`)
*   Increments Z counter and resets Y counter between face groups
*   Resets Z counter and destination write address at completion

**`_llk_pack_untilize_uninit_()`**

*   Restores Z stride to normal packing mode (`FACE_R_DIM * FACE_C_DIM`)
*   Must be called before switching back to normal packing operations

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 143-251](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L143-L251)

## Wormhole B0 Implementation Differences

Wormhole B0 does not have dedicated untilize pack functions. The untilize functionality on Wormhole B0 is handled differently through the standard packer configuration with specific address modifier and MOP setups in user code. Blackhole provides specialized, optimized untilize support with:

*   Dedicated `_llk_pack_untilize_*` functions
*   Optimized MOP templates with replay buffer usage
*   Block-based processing up to 8 tiles
*   Narrow row support
*   Simplified configuration interface

**Sources**: [tt_llk_blackhole/llk_lib/llk_pack_untilize.h 1-252](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack_untilize.h#L1-L252)
