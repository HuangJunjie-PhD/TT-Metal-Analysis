# Data Models and Logging

Relevant source files
*   [tt_smi/log.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py)

This document describes the data models and logging system used in the TT-SMI tool. It covers the structured data representations for device information, telemetry, and reset operations, as well as the logging functionality that enables monitoring and troubleshooting of Tenstorrent devices. For information about hardware interaction and backend implementation details, see [Backend System](https://deepwiki.com/tenstorrent/tt-smi/5.1-reset-architecture-and-types).

## 1. Data Model Architecture

The TT-SMI data models follow a structured, hierarchical design based on Pydantic, with support for Elasticsearch integration. These models provide type safety, validation, and a consistent way to represent data throughout the application.

Sources: [tt_smi/log.py 120-304](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L120-L304)

### 1.1 Base Model Classes

The foundation of the data model system lies in two key classes:

*   **Pydantic BaseModel**: Provides data validation, serialization, and object-document mapping functionality
*   **ElasticModel**: Extends BaseModel with Elasticsearch-specific functionality, particularly mapping Python types to Elasticsearch field types

Sources: [tt_smi/log.py 25-138](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L25-L138)

The `ElasticModel` class contains the `get_mapping()` method that returns the Elasticsearch mapping for model fields. Each field is processed by the `field_to_mapping()` function, which determines the appropriate Elasticsearch field type based on the Python type.

## 2. Device Information and Telemetry Models

TT-SMI uses multiple interconnected models to represent device information, telemetry data, and firmware versions.

### 2.1 Main Device Models

The key device information models include:

| Model | Purpose | Key Fields |
| --- | --- | --- |
| `SmbusTelem` | Raw SMBus telemetry data | BOARD_ID, DEVICE_ID, ASIC_RO, VCORE, ASIC_TEMPERATURE, etc. |
| `BoardInfo` | Basic board information | bus_id, board_type, board_id, coords, etc. |
| `Telemetry` | Processed telemetry metrics | voltage, current, aiclk, power, asic_temperature |
| `Firmwares` | Firmware version information | arc_fw, eth_fw, m3_bl_fw, m3_app_fw, etc. |
| `Limits` | Device operational limits | vdd_min, vdd_max, tdp_limit, tdc_limit, etc. |
| `TTSMIDeviceLog` | Combined device log | Contains instances of all the above models |

Sources: [tt_smi/log.py 140-259](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L140-L259)[tt_smi/constants.py 8-135](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L135)

Sources: [tt_smi/log.py 157-259](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L157-L259)

### 2.2 Host Information Models

Host information is captured in two primary models:

*   **HostInfo**: Contains operating system details, kernel version, hostname, memory information, and driver information
*   **HostSWVersions**: Contains TT-SMI and pyluwen version information

These models provide context for the device information and telemetry data, making it possible to correlate device behavior with the host environment.

Sources: [tt_smi/log.py 140-154](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L140-L154)

## 3. Reset Operation Models

TT-SMI includes specific models for device reset operations, which capture the configuration and execution of reset commands.

Sources: [tt_smi/log.py 278-304](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L278-L304)

The reset models enable:

1.   Configuration of device-specific reset operations through `PciResetDeviceInfo`
2.   Configuration of motherboard-level resets including credo chip resets through `MoboReset`
3.   Logging of reset operations for auditing and troubleshooting through `TTSMIResetLog`

## 4. Elasticsearch Integration

The TT-SMI logging system is designed to be compatible with Elasticsearch, allowing for centralized logging and advanced search capabilities.

### 4.1 Type Mapping System

The Elasticsearch integration is facilitated by a comprehensive type mapping system that converts Python types to Elasticsearch field types:

Sources: [tt_smi/log.py 65-110](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L65-L110)

The mapping system includes:

*   `type_to_mapping()`: Maps Python types to Elasticsearch field types
*   `field_to_mapping()`: Maps Pydantic fields to Elasticsearch field definitions
*   Special type classes (`Long`, `Keyword`, `Text`, `Date`): Provide explicit mapping to specific Elasticsearch field types
*   Support for nested fields through the `Nested` generic class

### 4.2 Model Serialization

The data models provide methods for serialization to JSON for storage or transmission:

*   `TTSMILog.get_clean_json_string()`: Returns a cleaned JSON string with nulls excluded
*   `TTSMIResetLog.save_as_json()`: Saves the reset log as a formatted JSON file

Sources: [tt_smi/log.py 269-304](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L269-L304)

## 6. Optional Fields and Model Flexibility

The TT-SMI data models are designed to be flexible, allowing for different hardware generations and varying levels of available information. This is achieved through the `@optional` decorator that modifies Pydantic models to make all fields optional.

Sources: [tt_smi/log.py 44-62](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L44-L62)[tt_smi/log.py 156-304](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/log.py#L156-L304)

This flexibility ensures that:

1.   Fields that are not available for certain device types won't cause validation errors
2.   New fields can be added over time without breaking existing code
3.   Different hardware generations with different telemetry capabilities can use the same model structures

## Summary

The Data Models and Logging system in TT-SMI provides a structured approach to representing, storing, and retrieving device information and telemetry. The use of Pydantic models ensures type safety and validation, while the Elasticsearch integration enables advanced search and analysis capabilities. The system's flexibility accommodates different hardware generations and varying levels of available information, making it a robust foundation for the TT-SMI tool's monitoring and management functionality.
