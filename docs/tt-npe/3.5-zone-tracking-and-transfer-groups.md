## Purpose and Scope

This document explains two systems in tt-npe that provide execution context and multi-chip routing organization:

1.   **Zone Tracking**: A profiling instrumentation system that associates transfers with hierarchical execution zones (profiling regions) extracted from tt-metal traces. Zones provide debugging context by identifying which high-level operation or kernel phase each NoC transfer belongs to.

2.   **Transfer Groups**: A data structure for organizing multi-hop fabric paths across ethernet links. When a transfer crosses device boundaries, it is decomposed into a sequence of component transfers organized into a group with parent-child relationships that enforce correct execution order.

For information about the general dependency tracking system that enforces all types of dependencies, see page 4.3. For details on fabric path elaboration that creates the routing paths, see page 2.2.

## Zone Tracking System

The zone tracking system extracts hierarchical profiling regions from tt-metal traces and associates each NoC transfer with its enclosing zone path. This provides attribution and debugging context for simulation results.

### Zone Data Model

Diagram: Zone Data Structures and Relationships

**Core Types:**

| Type | Definition | Purpose |
| --- | --- | --- |
| `npeZone` | `{double timestamp, string zone, ZonePhase zone_phase}` | Represents a single zone event (start or end) |
| `ZonePhase` | `enum {ZONE_START, ZONE_END}` | Indicates whether zone is opening or closing |
| `RiscType` | `enum {BRISC, NCRISC, TRISC_0, TRISC_1, TRISC_2, ERISC, CORE_AGG}` | Processor type on the core |

**Storage:** Zones are stored per `(Coord, RiscType)` pair in `npeWorkload`, allowing different processors on the same core to have independent zone hierarchies.

### Zone Extraction from Traces

During trace ingestion, zone events are extracted from the JSON trace file and organized by core and processor.

Diagram: Zone Extraction Pipeline

**Extraction Algorithm** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 271-289](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L271-L289)):

1.   **Compute Time Origin:** Calculate `t0_timestamp` as the minimum timestamp across all events to normalize all times to zero-based cycles

2.   **Filter Zone Events:** Iterate through all trace events and extract those with both `zone` and `zone_phase` fields

3.   **Normalize and Group:** For each zone event:

    *   Subtract `t0_timestamp` from timestamp
    *   Create key from `(Coord(device_id, sy, sx), RiscType(proc))`
    *   Append `npeZone(normalized_timestamp, zone, zone_phase)` to the vector for that key

4.   **Store in Workload:** The resulting map is stored in `npeWorkload::zones`

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 221-353](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L221-L353)

### Zone Iterator and Enclosing Zone Stack

The `ZoneIterator` class maintains a stack of currently-active zones, allowing transfers to be annotated with their full hierarchical context.

Diagram: ZoneIterator State Machine

**ZoneIterator State:**

| Field | Type | Purpose |
| --- | --- | --- |
| `zone_list` | `const vector<npeZone>&` | Chronologically-ordered zones for this core/processor |
| `zone_index` | `int` | Current position in zone_list |
| `enclosing_zones` | `vector<pair<npeZone, int>>` | Stack of currently-active zones with iteration counts |
| `zone_counts` | `map<string, int>` | Tracks how many times each zone has been entered (for iteration numbering) |

**Key Operations** ([tt_npe/cpp/include/npeUtil.hpp 338-373](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L338-L373)):

1.   **Increment (`operator++`)**: Processes the next zone event

    *   If `ZONE_START`: Push `(zone, current_count)` onto stack, increment count
    *   If `ZONE_END`: Validate against stack top and pop
    *   Advance `zone_index`

2.   **Get Enclosing Zones (`getEnclosingZones()`)**: Returns current stack of active zones with their iteration numbers

3.   **Get Next Zone (`getNextZone()`)**: Returns the zone event at `zone_index` without advancing

4.   **Check End (`isEnd()`)**: Returns true if `zone_index == zone_list.size()`

**Sources:**[tt_npe/cpp/include/npeUtil.hpp 329-373](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeUtil.hpp#L329-L373)

### Transfer Annotation with Zone Context

During trace-to-workload conversion, each transfer is annotated with a flattened zone path string describing its hierarchical profiling context.

Diagram: Transfer Zone Annotation Process

**Annotation Algorithm** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 384-400](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L384-L400)):

1.   **Initialize Iterator Per Core/Proc:** When processing an event from a new `(Coord, RiscType)` pair, create a fresh `ZoneIterator` for that core/processor's zone list

2.   **Advance to Current Time:** Increment the iterator while `zone_iterator.getNextZone().timestamp <= event_timestamp`, building the stack of enclosing zones

3.   **Flatten Zone Path** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 291-301](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L291-L301)): Convert the stack to a string:

```
For each (zone, iteration) in enclosing_zones:
    path += zone.zone + "[" + iteration + "]"
    if not last: path += "/"
```
4.   **Annotate Transfer:** Pass the flattened path string to the `npeWorkloadTransfer` constructor via the `enclosing_zone_path` parameter

**Example Zone Path:** A transfer occurring during the third iteration of a nested loop might have:

```
enclosing_zone_path = "MatmulOp[0]/ComputeKernel[2]/InnerLoop[5]"
```

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 291-400](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L291-L400)[tt_npe/cpp/include/npeWorkload.hpp 38-68](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L38-L68)

### Zone Usage in Timeline Output

Zone paths are included in the timeline JSON output to enable zone-based filtering and analysis in visualization tools.

**Timeline Schema:** Each transfer in the timeline includes an `enclosing_zone_path` field with the annotated zone string. This allows visualizers to:

*   Filter transfers by zone (e.g., show only "MatmulOp" transfers)
*   Color-code transfers by their outermost zone
*   Compute per-zone statistics (bandwidth, latency, congestion)

**Sources:**[tt_npe/cpp/src/npeStats.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp)[tt_npe/cpp/include/npeWorkload.hpp 68](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L68-L68)

## Transfer Groups for Multi-Hop Fabric Paths

When a transfer crosses ethernet links between devices in a multi-chip system, it cannot be modeled as a single atomic operation. Instead, the transfer is decomposed into a sequence of per-device hop transfers organized as a **transfer group**. Each component transfer has a parent-child relationship that enforces the correct execution order.

### Transfer Group Data Structure

Diagram: Transfer Group Structure and Relationships

**Transfer Group Fields** ([tt_npe/cpp/include/npeWorkload.hpp 70-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L70-L72)):

| Field | Type | Purpose |
| --- | --- | --- |
| `transfer_group_id` | `npeWorkloadTransferGroupID` | Unique ID for the logical transfer group; all component transfers share this ID. Value of `-1` indicates no group (single-device transfer) |
| `transfer_group_index` | `npeWorkloadTransferGroupIndex` | Sequential index within the group (0, 1, 2, ...). Determines order of execution |
| `transfer_group_parent` | `npeWorkloadTransferGroupParent` | Index of the prerequisite transfer within the same group. Value of `-1` indicates root transfer (no prerequisite) |

**Group ID Registration** ([tt_npe/cpp/include/npeWorkload.hpp 129-132](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L129-L132)): The `npeWorkload` class maintains a counter `num_transfer_groups` and provides `registerTransferGroupID()` to allocate unique group IDs during workload construction.

**Sources:**[tt_npe/cpp/include/npeWorkload.hpp 16-132](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L16-L132)

### Creation of Transfer Groups from Fabric Paths

The fabric post-processing pipeline elaborates high-level `fabric_send` events into detailed multi-hop paths with explicit parent-child relationships. During workload ingestion, these paths are converted into transfer groups.

Diagram: Fabric Path to Transfer Group Conversion

**Elaboration Steps:**

1.   **Topology-Based Path Finding** ([tt_npe/py/util/fabric_post_process.py 353-482](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L353-L482)):

    *   `find_path_and_destination_1d()`: For linear fabric topologies (FABRIC_1D, FABRIC_1D_RING)
    *   `find_path_and_destination_2d()`: For 2D mesh/torus topologies (FABRIC_2D, FABRIC_2D_TORUS)
    *   Each path segment contains: `device`, `segment_start_x/y`, `forward_x/y`, `parent_id`, optional `local_writes`

2.   **Parent ID Assignment** ([tt_npe/py/util/fabric_post_process.py 300-351](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L300-L351)):

    *   First hop has `parent_id=-1` (root)
    *   Each subsequent hop references its parent's index
    *   Local writes branch from the same parent as the forward hop

3.   **Transfer Group Construction** ([tt_npe/cpp/src/npeWorkloadIngest.cpp 520-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L520-L639)):

    *   Register unique `transfer_group_id` via `wl.registerTransferGroupID()`
    *   Initialize `transfer_group_index=0`
    *   For each route segment: 
        *   Create local write transfers (if `local_writes` present)
        *   Create forward transfer (if `forward_x/y` present)
        *   Increment `transfer_group_index` after each transfer
        *   Set `transfer_group_parent` from route's `parent_id`

**Example Route Structure:**

This creates a transfer group with 4 transfers:

*   Transfer 0: Device 0, (2,3)→(1,9), parent=-1 (root)
*   Transfer 1: Device 1, (1,9)→(5,5), parent=0 (local write)
*   Transfer 2: Device 1, (1,9)→(1,8), parent=0 (forward)
*   Transfer 3: Device 2, (1,8)→(7,3), parent=2 (forward)

**Sources:**[tt_npe/py/util/fabric_post_process.py 300-482](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py#L300-L482)[tt_npe/cpp/src/npeWorkloadIngest.cpp 520-639](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L520-L639)

## Transfer Group Dependency Generation

During simulation initialization in `npeEngine::genDependencies()`, the parent-child relationships encoded in transfer groups are converted to checkpoint-based dependencies with cycle delays that model hardware latencies.

### Dependency Mapping Algorithm

Diagram: Transfer Group Dependency Creation

**Implementation Steps** ([tt_npe/cpp/src/npeEngine.cpp 112-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L112-L153)):

1.   **Build Index Map** ([tt_npe/cpp/src/npeEngine.cpp 117-123](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L117-L123)):

```
For each transfer in all_transfers:
    If transfer.group_id != -1:
        map[(transfer.group_id, transfer.group_index)] = transfer.id
```
2.   **Create Dependencies** ([tt_npe/cpp/src/npeEngine.cpp 125-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L125-L153)):

```
For each transfer where transfer.group_parent != -1:
    parent_id = map[(transfer.group_id, transfer.group_parent)]
    parent_transfer = transfer_state[parent_id]
    
    delay = calculate_delay(parent_transfer, transfer)
    
    checkpoint_id = dep_tracker.createCheckpoint(1, delay)
    transfer.depends_on = checkpoint_id
    parent_transfer.required_by.push_back(checkpoint_id)
```

### Cycle Delay Components

The checkpoint delay models the physical time between parent completion and child start, accounting for NoC write time and optional ethernet link traversal.

**Delay Formula:**

| Component | When Applied | Formula | Purpose |
| --- | --- | --- | --- |
| Write Latency | Always | `get_write_latency(parent.src, parent.dst, noc_type)` | Time for parent's NoC write to complete |
| Ethernet Hop | Cross-device only | `600 + 0.1055 * packet_size` | Latency for ethernet link traversal |

**Write Latency Calculation** ([tt_npe/cpp/src/npeEngine.cpp 131-142](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L131-L142)):

*   Architecture-specific: `WormholeB0DeviceModel::get_write_latency()` or `BlackholeDeviceModel::get_write_latency()`
*   Depends on Manhattan distance and NoC type
*   Accounts for packet header overhead

**Ethernet Hop Delay** ([tt_npe/cpp/src/npeEngine.cpp 144-147](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L144-L147)):

*   Base latency: 600 cycles (link setup and header processing)
*   Size-dependent component: 0.1055 cycles per byte (link bandwidth limitation)
*   Only applied when `parent.src.device_id != child.src.device_id`

**Example Calculation:**

```
Parent: Device 0, (2,3)→(1,9), packet_size=2048
Child:  Device 1, (1,9)→(5,5)

write_latency = get_write_latency(2, 3, 1, 9, "NOC_0") = ~45 cycles
eth_delay = 600 + 0.1055 * 2048 = 816 cycles
checkpoint_delay = 45 + 816 = 861 cycles
```

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 112-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L112-L153)

### Dependency Enforcement in Simulation

Transfer group dependencies are enforced by the general checkpoint mechanism during simulation. The simulation loop checks whether each transfer's parent checkpoint is satisfied before activating the transfer.

**Key Enforcement Points:**

1.   **Activation Check** ([tt_npe/cpp/src/npeEngine.cpp 234-256](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L234-L256)): Before activating a transfer, verify `dep_tracker.done(transfer.depends_on, current_cycle)` returns true

2.   **Completion Notification** ([tt_npe/cpp/src/npeEngine.cpp 310-312](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L310-L312)): When a parent transfer completes, notify all child checkpoints via `dep_tracker.updateCheckpoint()`

3.   **Cycle Adjustment**: If a parent completes late, the child's effective start cycle is adjusted to `max(nominal_start, parent_end + delay)`

For complete details on the dependency tracking system and enforcement logic, see page 4.3.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 234-312](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L234-L312)

## Dependency Sanity Checks

The system performs validation to ensure dependency graphs are correctly constructed.

### Validation Steps

| Check | Location | Purpose |
| --- | --- | --- |
| Initial checkpoint consistency | [tt_npe/cpp/src/npeEngine.cpp 155-164](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L155-L164) | After dependency generation, simulate updating all checkpoints to verify they're exactly satisfied |
| Post-simulation completion | [tt_npe/cpp/src/npeEngine.cpp 333-335](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L333-L335) | Verify all checkpoints are satisfied and all transfers completed |
| Transfer group mapping consistency | [tt_npe/cpp/src/npeStats.cpp 438-477](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L438-L477) | During timeline serialization, check for duplicate IDs and validate group-to-transfer mappings |

**Consistency Check Algorithm** ([tt_npe/cpp/src/npeEngine.cpp 155-164](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L155-L164)):

```
// Verify all required checkpoints are defined
For each transfer in transfer_state:
    For each checkpoint_id in transfer.required_by:
        dep_tracker.updateCheckpoint(checkpoint_id, 0)  // Dummy cycle

If NOT dep_tracker.sanityCheck():
    ERROR: Inconsistency in dependencies
    
If NOT dep_tracker.allComplete():
    ERROR: Inconsistency in dependencies

// Reset tracker for actual simulation
dep_tracker.reset()
```

This validation ensures that every checkpoint has exactly the number of prerequisite transfers it expects, preventing deadlocks or unsatisfied dependencies.

**Sources:**[tt_npe/cpp/src/npeEngine.cpp 155-170](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L155-L170)[tt_npe/cpp/src/npeStats.cpp 438-477](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L438-L477)
