This page covers debugging tools and techniques for tt-lang kernels, including simulator debugging, environment variables for MLIR inspection, runtime print debugging (DPrint), auto-profiling, and performance analysis.

tt-lang provides multiple debugging approaches depending on the issue:

*   **Simulator debugging** - Use Python debuggers (VSCode, pdb) to step through kernel logic as pure Python.
*   **DPrint debugging** - Use `print()` inside kernels to emit values to the console during hardware or simulator execution.
*   **MLIR debugging** - Inspect the compilation pipeline stages using environment variables.
*   **Auto-profiling** - Obtain per-line cycle counts and hotspot analysis on hardware.

## Simulator Debugging

The tt-lang simulator runs as standard Python code, enabling any Python debugger to work with it. This is the recommended approach for debugging kernel logic without requiring hardware or a full compiler build. If hardware libraries are missing, `TTLANG_SIM_ONLY=1` forces the use of the simulator [python/ttl/__init__.py 21-23](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/__init__.py#L21-L23)

### VSCode Debugger Configuration

To debug kernels in VSCode, create a `.vscode/launch.json` configuration:

**Usage:**

1.   Open a kernel file.
2.   Set breakpoints directly in the Python kernel code.
3.   Press **F5** to start the simulator under the debugger.
4.   Inspect variables and step through the cooperative execution of different threads/cores.

## Runtime Print Debugging (DPrint)

The tt-lang DSL supports standard Python `print()` calls inside kernels [test/python/dprint/codegen.py 10-18](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L10-L18) These are lowered to hardware `DPRINT` macros via the `LowerDPrintToEmitC` pass [lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp 17-19](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp#L17-L19)

### Supported Print Modes

The compiler detects the type of the argument and generates specialized hardware calls:

*   **Scalar/String**: Prints string constants, integers, or floats [lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp 74-95](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp#L74-L95)
*   **CB/DFB State**: Prints Circular Buffer metadata via `ttmlir::CBPrinter`[test/python/dprint/codegen.py 87-89](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L87-L89)
*   **Tile Data**: Prints tile contents using `TSLICE` macros [test/python/dprint/codegen.py 97-99](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L97-L99)
*   **DST Regs**: Dumps DST register file contents with live slot info [lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp 204-207](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp#L204-L207)

### Thread Conditioning

Compute kernels run on three TRISC threads. Prints can be explicitly conditioned using the `thread` parameter [test/python/dprint/runtime.py 52-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/runtime.py#L52-L66):

*   `print(..., thread="math")` -> Wrapped in `MATH({})`[test/python/dprint/codegen.py 80-82](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L80-L82)
*   `print(..., thread="pack")` -> Wrapped in `PACK({})`[test/python/dprint/codegen.py 97-99](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L97-L99)
*   `print(..., thread="unpack")` -> Wrapped in `UNPACK({})`[test/python/dprint/runtime.py 66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/runtime.py#L66-L66)

**Sources:**[lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp 74-153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/LowerDPrintToEmitC.cpp#L74-L153)[test/python/dprint/codegen.py 33-68](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/codegen.py#L33-L68)[test/python/dprint/runtime.py 42-89](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/dprint/runtime.py#L42-L89)

### Pass Analysis Workflow

When compilation fails, use `TTLANG_VERBOSE_PASSES=1` to trace the IR.

1.   **Identify Failure**: Locate which pass in `createTTLToTTKernelPipeline` failed [include/ttlang/Dialect/TTL/Pipelines/TTLPipelines.h 59-60](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Pipelines/TTLPipelines.h#L59-L60)
2.   **Inspect Options**: Verify if flags like `--no-ttl-maximize-dst` or `--no-ttl-compiler-dfbs` resolve the issue [docs/sphinx/reference/compiler-options.md 13-21](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/reference/compiler-options.md?plain=1#L13-L21)
3.   **Check DST Allocation**: If math is corrupted, verify subblock partitioning and DST utilization enabled by `maximize-dst`[docs/sphinx/reference/compiler-options.md 15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/reference/compiler-options.md?plain=1#L15-L15)
4.   **F32 Capacity**: Use `--ttl-strict-f32-acc` to catch DST overflows early during accumulation loops [docs/sphinx/reference/compiler-options.md 20](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/reference/compiler-options.md?plain=1#L20-L20)

## Compiler Options and Flags

Compiler behavior can be tuned via `CompilerOptions`[python/ttl/compiler_options.py 123-148](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/compiler_options.py#L123-L148)

| Flag | Default | Description |
| --- | --- | --- |
| `--ttl-maximize-dst` | `True` | Maximize DST utilization via subblocking [python/ttl/compiler_options.py 139](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/compiler_options.py#L139-L139) |
| `--ttl-fpu-binary-ops` | `True` | Use FPU for binary ops (add/sub/mul) [python/ttl/compiler_options.py 140](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/compiler_options.py#L140-L140) |
| `--ttl-subblock-sync` | `False` | Refine DFB sync to subblock granularity [python/ttl/compiler_options.py 142](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/compiler_options.py#L142-L142) |
| `--ttl-strict-f32-acc` | `False` | Error if accumulation exceeds f32 DST capacity [python/ttl/compiler_options.py 146](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/compiler_options.py#L146-L146) |

## Performance Analysis Tools

### Auto-Profiling

Profiling flags are managed separately from code generation flags [docs/sphinx/reference/compiler-options.md 73-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/reference/compiler-options.md?plain=1#L73-L76)

*   `TTLANG_AUTO_PROFILE`: Enable line-level cycle counts.
*   `TTLANG_SIGNPOST_PROFILE`: Enable custom signpost insertion.
*   `TTLANG_PROFILE_CSV`: Output profile data to a CSV file.

### Matmul Benchmarking

For performance debugging of matrix multiplications, the `benchmarks/matmul/` suite provides tools for:

*   **K-Split Analysis**: Evaluating performance with different K-dimension partitioning [test/python/simple_matmul_subblock.py 11-15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/simple_matmul_subblock.py#L11-L15)
*   **Sweep Testing**: Running performance sweeps across different grid sizes and subblock configurations.
