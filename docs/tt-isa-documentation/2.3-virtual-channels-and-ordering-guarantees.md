## Purpose and Scope

This page describes the NoC virtual channel system, including the structure of virtual channel numbers, the meaning of class bits and other bit fields, and how virtual channels affect packet ordering guarantees. Virtual channels enable concurrent transactions through the same physical links and provide ordering semantics that software can rely on or control.

For information about initiating NoC transactions and configuring request flags, see [NoC Memory Map and NIU Interface](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface). For details on transaction types and routing paths, see [Transaction Types and Routing](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.2-transaction-types-and-routing). For monitoring transaction completion, see [NoC Counters and Alignment Requirements](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements).

## Virtual Channel Number Structure

When a packet traverses a link between two routers (or between a router and an NIU), it is assigned a **4-bit virtual channel number**. This number enables multiple packets to share the same physical link concurrently without head-of-line blocking. Once a packet has fully traversed a link, the virtual channel number is released for use by other packets.

**Figure: Virtual Channel Number Structure**

The 4-bit virtual channel number consists of:

| Bit Field | Width | Purpose |
| --- | --- | --- |
| **Class bits** | 2 bits | Statically determined packet type, enforced by hardware |
| **Buddy bit** | 1 bit | Congestion management, can flip dynamically or remain static |
| **Dateline bit** | 1 bit | Torus wrap detection, flips at predetermined points |

As a packet traverses multiple hops from initiating NIU to receiving NIU, the virtual channel number can change at each hop according to specific rules described in subsequent sections.

Sources: [WormholeB0/NoC/README.md 44-52](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L44-L52)[WormholeB0/NoC/MemoryMap.md 61-66](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L61-L66)

## Class Bits and Packet Types

The two **class bits** within the virtual channel number serve to classify packets by type. This classification is statically determined and enforced by hardware to prevent deadlock:

| Class Bits | Packet Type | Description |
| --- | --- | --- |
| `0b00` | Unicast request | Standard request packet to a single destination |
| `0b01` | Unicast request | Alternative class for unicast requests (enables concurrent unicast flows) |
| `0b10` | Broadcast request | Request packet to a rectangular region of tiles |
| `0b11` | Response packet | Response/acknowledgement packet (always unicast, even when responding to broadcast) |

**Key Properties:**

*   **Request packets** (unicast) can use either `0b00` or `0b01` class bits. Software can use both classes to enable two independent streams of unicast requests without head-of-line blocking.
*   **Broadcast request packets** must use `0b10` class bits.
*   **Response packets** (read responses, write acknowledgements, atomic responses) always use `0b11` class bits, regardless of whether they are responding to unicast or broadcast requests.
*   Class bits **never change** as a packet traverses from source to destination.

The separation of response packets into their own class prevents a common deadlock scenario where requests and responses would otherwise contend for the same virtual channel resources.

Sources: [WormholeB0/NoC/README.md 47-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L47-L51)[WormholeB0/NoC/MemoryMap.md 66](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L66-L66)

## Buddy Bit and Congestion Management

The **buddy bit** provides a mechanism for dynamic congestion management. By default, routers can flip the buddy bit at any hop in response to network congestion, allowing packets to take alternate virtual channel resources when their preferred channel is busy.

**Figure: Dynamic Buddy Bit Flipping for Congestion Management**

**Dynamic Mode (default):**

*   When `NOC_CMD_VC_STATIC` flag is `false` (see [WormholeB0/NoC/MemoryMap.md 61](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L61-L61)), the initiating NIU dynamically chooses the class bits and buddy bit for the initial hop
*   Any router along the path can dynamically flip the buddy bit in response to congestion
*   Packets with the same class bits but different buddy bits can proceed independently without blocking each other

**Static Mode:**

*   When `NOC_CMD_VC_STATIC` flag is `true`, software dictates the buddy bit value for the first hop using bits 13-14 of `NOC_CTRL`
*   Routers are **not allowed** to flip the buddy bit
*   Only the dateline bit can change between hops
*   Provides stronger ordering guarantees at the cost of potentially increased latency

Sources: [WormholeB0/NoC/README.md 46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/README.md?plain=1#L46-L46)[WormholeB0/NoC/MemoryMap.md 61-66](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L61-L66)[WormholeB0/NoC/Ordering.md 14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L14-L14)

## Dateline Bit and Torus Wrap Detection

The **dateline bit** handles the topology of the 2D torus network. In a torus, packets can wrap around edges (e.g., traveling from X=9 to X=0, or Y=11 to Y=0). The dateline bit automatically flips at statically predetermined wrap boundaries to prevent routing loops.

**Figure: Dateline Bit Flipping at Torus Wrap Boundaries**

The dateline bit flip locations are statically determined per router and reported in the `NOC_NODE_ID` register:

| Bit | Purpose |
| --- | --- |
| 26 | If `true`, dateline bit flips when packets transit through the outbound X port of this router |
| 27 | If `true`, dateline bit flips when packets transit through the outbound Y port of this router |

See [WormholeB0/NoC/MemoryMap.md 187-188](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L187-L188) for the `NOC_NODE_ID` register structure.

**Key Properties:**

*   Dateline bit flips are **automatic** and happen at statically predetermined locations based on the packet's route
*   The flip locations depend on the physical topology and routing direction (X-major for NoC #0, Y-major for NoC #1)
*   Software does not need to manage dateline bit flips; they occur transparently
*   The dateline bit **always** flips at wrap boundaries, regardless of whether the buddy bit is static or dynamic

Sources: [WormholeB0/NoC/MemoryMap.md 187-188](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L187-L188)[WormholeB0/NoC/Ordering.md 14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L14-L14)

## Virtual Channel Assignment Process

When software initiates a NoC request by writing to `NOC_CMD_CTRL` (see [WormholeB0/NoC/MemoryMap.md 148-157](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L148-L157)), the NIU appends the request to an internal queue. Hardware continuously attempts to assign a virtual channel number to the request at the front of the queue for its first hop (NIU to router).

**Figure: Virtual Channel Assignment Flow**

**Assignment Rules:**

1.   **Order preservation:** Requests are assigned virtual channel numbers in the order software writes to `NOC_CMD_CTRL`
2.   **Blocking behavior:** If two requests require the same virtual channel number (e.g., due to `NOC_CMD_VC_STATIC` or `NOC_CMD_VC_LINKED`), they are assigned in order, with later requests blocking until earlier ones depart
3.   **Initiator reuse:** Software must not write to any fields of an initiator until `NOC_CMD_CTRL` bit 0 reverts to `0`, indicating the virtual channel has been assigned

The number of simultaneously active virtual channels is limited by hardware. At the final hop (router to destination NIU), each NIU can handle up to **12 concurrent transactions** (one per distinct virtual channel number used at that hop).

Sources: [WormholeB0/NoC/MemoryMap.md 148-157](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L148-L157)[WormholeB0/NoC/Ordering.md 16-17](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L16-L17)

## Ordering Guarantees: Packet-Level

### Single Packet Ordering

A single write request or read response packet involves reading bytes from one tile's address space and writing them to another tile's address space.

**For writes to L1:**

*   The receiving NIU decomposes writes into **two separate streams**: 
    *   **Stream 0:** Byte addresses `0 mod 32` through `15 mod 32` (inclusive)
    *   **Stream 1:** Byte addresses `16 mod 32` through `31 mod 32` (inclusive)

*   **Within each stream**, writes occur in **ascending address order**
*   **Between streams**, no ordering is guaranteed

**For all memory operations:**

*   Individual reads and writes are broken into **aligned 16-byte atomic units**
*   The order of these 16-byte operations can be arbitrary
*   Operations can interleave with other agents accessing the same memory

Software should use [NIU counters](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements) to determine when all reads or writes from a packet have completed, but should not assume any particular order for individual 16-byte chunks except as described above.

Sources: [WormholeB0/NoC/Ordering.md 3-7](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L3-L7)

## Ordering Guarantees: Multi-Packet

### When Packets Share Virtual Channel Numbers

If two packets coincide for part or all of their route, and both are assigned the **same virtual channel number at every hop along that route**, then strong ordering guarantees apply:

**Figure: Ordering Guarantee for Packets with Identical Virtual Channel Numbers**

**Ordering Rule:** Whichever packet enters the common route first will:

1.   Enter **entirely** before any of the second packet can enter
2.   Leave **entirely** before any of the second packet can leave

This applies to all hop types:

*   Router-to-router hops (most common)
*   Initial hop from NIU to router
*   Final hop from router to NIU

### Controlling Virtual Channel Assignment

Software has three mechanisms to control virtual channel assignment and thereby control ordering:

#### 1. `NOC_CMD_VC_STATIC` Flag

When this flag is set to `true` in `NOC_CTRL` bits 7 and 13-14 (see [WormholeB0/NoC/MemoryMap.md 61-66](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L61-L66)):

*   Software dictates the buddy bit and class bits for the first hop using bits 13-14 of `NOC_CTRL`
*   Routers may **only** flip the dateline bit; buddy bit remains static
*   Dateline bit flips at statically predetermined points based on the packet's route
*   Provides **strongest ordering guarantees** when multiple packets use the same static VC

**Example configuration:**

#### 2. `NOC_CMD_VC_LINKED` Flag

When this flag is set to `true` in `NOC_CTRL` bit 6 (see [WormholeB0/NoC/MemoryMap.md 60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L60-L60)):

*   Creates a **multi-request transaction** where multiple request packets share the same virtual channel treatment
*   Hardware chooses the VC for the first request, but subsequent requests in the transaction use the same VC choices
*   The buddy bit and dateline bit flip at **exactly the same places** for all requests in the transaction
*   All requests must have the same destination (or destination rectangle for broadcasts)

**Important side effects:**

*   The NIU cannot initiate requests on any other virtual channel until the transaction completes
*   All routers along the path reserve the chosen VC numbers exclusively for this transaction
*   Software must promptly complete the transaction by issuing a request with `NOC_CMD_VC_LINKED=false`
*   Failure to complete the transaction will cause the NIU to become unable to initiate any requests

See the detailed description at [WormholeB0/NoC/MemoryMap.md 60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L60-L60)

#### 3. Write Order to `NOC_CMD_CTRL`

The order in which software writes to `NOC_CMD_CTRL` across different request initiators determines:

*   The order in which requests are assigned virtual channel numbers
*   When two requests require the same VC (due to `NOC_CMD_VC_STATIC` or `NOC_CMD_VC_LINKED`), their departure order from the NIU

### Response Packet Ordering

**Response and acknowledgement packets** (read responses, write acknowledgements, atomic responses) have **no software-controllable ordering**:

*   Class bits are always `0b11`
*   Buddy bit can flip at any router
*   Multiple response packets can arrive in any order, even if the corresponding requests were ordered
*   Software must use [NIU counters](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements) to detect when **all** responses have arrived, but should not assume any particular arrival order

**Exception:** Software can detect completion of subsets of responses only when different subsets manipulate different counters (e.g., different values of `NOC_PACKET_TRANSACTION_ID`).

Sources: [WormholeB0/NoC/Ordering.md 9-18](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L9-L18)

## Ordering Guarantees: Recipient NIU

The recipient NIU receives packets, acts on them, and optionally sends response packets. The NIU can process up to **12 different request packets** (and 4 different response packets) simultaneously, based on the virtual channel number used for the final hop.

**Figure: Recipient NIU Concurrent Processing**

### Ordering Rules for Same Virtual Channel

When two packets arrive on the **same virtual channel**, one after the other, the following ordering guarantees apply:

| First Packet Type | Second Packet Type | Ordering Guarantee |
| --- | --- | --- |
| Write request | Read request | All writes from 1st packet committed before reads for 2nd packet start |
| Read request with `NOC_CMD_VC_LINKED` | Any request | All reads for 1st packet completed before reads/writes from 2nd packet |
| Atomic with `NOC_CMD_RESP_MARKED` | Atomic request | 1st atomic operation completes before 2nd atomic operation starts |
| Write to L1 | Write to L1 (overlapping addresses) | Every byte in overlap receives write from 1st before write from 2nd |
| Write to MMIO | Write to MMIO (same device) | 1st write processed by device before 2nd write |
| Read from MMIO | Read request | 1st read completes before 2nd read starts |
| Write to L1 | Write to MMIO | **No ordering guarantee** - MMIO write can race ahead |

See [WormholeB0/NoC/Ordering.md 20-28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L20-L28) for complete details.

### No Ordering for Different Virtual Channels

If two packets arrive on **different virtual channels**, arbitrary reordering and interleaving can occur. The NIU processes up to 12 requests concurrently, with no ordering guarantees between them.

Sources: [WormholeB0/NoC/Ordering.md 20-28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L20-L28)

## Multi-Request Transactions

The `NOC_CMD_VC_LINKED` flag enables **multi-request transactions** where multiple request packets are treated as a single atomic unit with respect to virtual channel reservation:

**Figure: Multi-Request Transaction with NOC_CMD_VC_LINKED**

### Requirements

1.   **All requests** in the transaction must have the **same destination** (or same destination rectangle for broadcasts)
2.   **Software must** issue a final request with `NOC_CMD_VC_LINKED=false` to complete the transaction
3.   While a transaction is in progress: 
    *   The NIU **cannot** initiate requests on any other virtual channel
    *   All routers along the path **reserve** the virtual channel exclusively for this transaction
    *   Other NIUs and the NoC Overlay at the same tile **cannot** initiate requests

### Consequences of Non-Compliance

**Failure to meet requirements will cause:**

*   **Complete NoC failure** if requests in the transaction go to different destinations
*   **NIU lockup** if software attempts to initiate a request with an incompatible VC (the NIU continuously fails VC assignment and cannot proceed)
*   **Deadlock** if software does not promptly issue the final request

### Use Cases

Multi-request transactions are primarily useful when:

*   Software needs stronger ordering guarantees than default dynamic VC assignment provides
*   A sequence of requests must be guaranteed to use the same path through the network
*   Atomicity of a request sequence is required (all-or-nothing completion semantics)

**Caution:** This feature severely restricts concurrency. Software is responsible for ensuring deadlock freedom when using multi-request transactions.

Sources: [WormholeB0/NoC/MemoryMap.md 60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/MemoryMap.md?plain=1#L60-L60)[WormholeB0/NoC/Ordering.md 15-16](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L15-L16)

## Stronger Ordering Approaches

If software requires ordering stronger than the default NoC semantics provide, two approaches are available:

### 1. Issue and Wait Pattern

Software can issue a request, wait for the response to arrive (by polling [NIU counters](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.4-noc-counters-and-alignment-requirements)), and only then issue the next request:

**Tradeoffs:**

*   **Guarantees:** Provides strongest possible ordering
*   **Latency:** High - each request incurs full round-trip latency before next request starts
*   **Throughput:** Low - no pipeline parallelism
*   **Cross-NoC:** Only approach that provides ordering between requests on different NoCs

### 2. Static Virtual Channels

Software can use `NOC_CMD_VC_STATIC` to dictate virtual channel numbers, ensuring requests share the same VC and thus benefit from hardware ordering guarantees:

**Tradeoffs:**

*   **Guarantees:** Strong ordering at each hop where VCs match
*   **Latency:** Moderate - second request may wait for first at each router, but no software polling overhead
*   **Throughput:** Reduced - head-of-line blocking occurs
*   **Cross-NoC:** Does not provide ordering between NoCs

Sources: [WormholeB0/NoC/Ordering.md 30-32](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/NoC/Ordering.md?plain=1#L30-L32)

## Summary Table: Ordering Control Mechanisms

| Mechanism | Ordering Strength | Latency Impact | Throughput Impact | Complexity |
| --- | --- | --- | --- | --- |
| **Default (dynamic VC)** | Weak - packets can reorder | Low | High | Low |
| **`NOC_CMD_VC_STATIC`** | Medium - packets with same VC ordered | Medium | Medium | Medium |
| **`NOC_CMD_VC_LINKED`** | Strong - all packets in transaction ordered | Medium-High | Low | High |
| **Issue-and-wait** | Strongest - full sequential ordering | High | Lowest | Low |

**Recommendation:** Use default dynamic VCs unless specific ordering is required. When ordering is needed, prefer `NOC_CMD_VC_STATIC` over `NOC_CMD_VC_LINKED` when possible, as the latter severely restricts concurrency and requires careful deadlock avoidance.
