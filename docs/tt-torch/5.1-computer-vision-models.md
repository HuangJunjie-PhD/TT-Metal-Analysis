## Supported Model Libraries

tt-torch provides comprehensive support for computer vision models from multiple popular libraries, each with dedicated test suites and optimized configurations.

### TorchVision Models

The TorchVision test suite covers the most widely used computer vision architectures for both image classification and object detection tasks.

#### Image Classification Models

The image classification test suite supports over 50 different architectures spanning multiple model families:

*   **ResNet family**: ResNet18, ResNet34, ResNet50, ResNet101, ResNet152, Wide ResNet variants
*   **DenseNet family**: DenseNet121, DenseNet161, DenseNet169, DenseNet201
*   **MobileNet family**: MobileNet V2, MobileNet V3 Small/Large
*   **VGG family**: VGG11/13/16/19 with and without batch normalization
*   **Vision Transformers**: ViT-B/16, ViT-B/32, ViT-L/16, ViT-L/32, ViT-H/14
*   **RegNet family**: RegNet-X and RegNet-Y variants from 400MF to 128GF
*   **Swin Transformers**: Swin-T/S/B and Swin-V2-T/S/B
*   **Specialized architectures**: GoogLeNet, ResNeXt variants

#### Object Detection Models

Object detection support includes modern detection architectures:

*   **SSD variants**: SSD300 with VGG16, SSDLite320 with MobileNet V3
*   **RetinaNet**: ResNet50 FPN backbone variants

### TIMM (PyTorch Image Models)

The TIMM integration provides access to state-of-the-art vision models with standardized preprocessing:

*   **EfficientNet variants**: tf_efficientnet_lite0 through lite4
*   **GhostNet**: GhostNet 100 and GhostNetV2 100
*   **Inception**: Inception V4
*   **MLP-Mixer**: mixer_b16_224
*   **Specialized architectures**: Xception71, DLA34, HRNet, VovNet variants

### Specialized Models

Additional computer vision models are supported through dedicated test suites:

*   **HarDNet**: Hardware-aware efficient CNN architectures
*   **YOLOS**: You Only Look One Sequence object detection with Vision Transformers

## Compilation and Execution Modes

Computer vision models support multiple compilation depths and execution backends as detailed in [Op-by-Op Compilation](https://deepwiki.com/tenstorrent/tt-torch/3.1.2-op-by-op-compilation).

### Compilation Configuration

| Configuration | Purpose | Supported Backends |
| --- | --- | --- |
| `CompileDepth.EXECUTE` | Full compilation to TTNN | Native execution |
| `CompileDepth.EXECUTE_OP_BY_OP` | Operation-by-operation execution | `OpByOpBackend.TORCH`, `OpByOpBackend.STABLEHLO` |
| Data Parallel Mode | Multi-device execution | Single device, Data parallel |

### Backend Selection

Different models may require specific backend configurations:

*   **Default**: `tt-experimental` backend for most models
*   **Legacy fallback**: `tt-legacy` backend for models with known issues (e.g., Inception V4)
*   **Op-by-op debugging**: TORCH or STABLEHLO backends for development

## Model-Specific Configurations

### Accuracy Thresholds

Different model architectures require different accuracy validation thresholds:

| Model Category | Required PCC | Assertion Mode |
| --- | --- | --- |
| Most TorchVision models | 0.96 | `assert_pcc=True` |
| TIMM models (general) | 0.99 | `assert_pcc=True` |
| TIMM EfficientNet | 0.98 | `assert_pcc=True` |
| TIMM MobileNet | 0.95 | `assert_pcc=True` |
| HarDNet | 0.98 (0.97 data parallel) | `assert_pcc=True` |
| Object detection | Varies by model | Some `assert_pcc=False` |

### Hardware-Specific Configurations

The N300 test suite demonstrates hardware-specific optimizations:

*   **Automatic parallelization**: `cc.automatic_parallelization = True`
*   **Mesh configuration**: `cc.mesh_shape = [1, 2]`
*   **Batch size adjustments**: Increased batch sizes for better utilization
*   **Memory considerations**: Some large models (e.g., ViT-H/14) may encounter memory limitations

## Performance and Accuracy Validation

### Output Interpretation

Computer vision models provide different output formats requiring specialized interpretation:

*   **Image classification**: Top-K predictions with confidence scores
*   **Object detection**: Bounding boxes, confidence scores, and class labels
*   **Specialized models**: Model-specific output formats (e.g., YOLOS post-processing)

### Validation Metrics

The testing framework validates models using:

*   **Pearson Correlation Coefficient (PCC)**: Primary accuracy metric
*   **Absolute tolerance (ATOL)**: Secondary validation for numerical precision
*   **Top-K accuracy**: Task-specific validation for classification models
