# Data Models and Configuration

Relevant source files
*   [tt_tools_common/reset_common/host_reset_log.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py)
*   [tt_tools_common/reset_common/reset_utils.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py)

This page documents the Pydantic data models and configuration file formats used throughout tt-tools-common. These models provide structured data validation, Elasticsearch integration, and JSON serialization for reset operations and system logging.

For information about using these models in reset operations, see [Reset Configuration and Logging](https://deepwiki.com/tenstorrent/tt-tools-common/2.6-reset-configuration-and-logging). For API usage examples, see [Reset API Reference](https://deepwiki.com/tenstorrent/tt-tools-common/7.1-reset-api-reference).

## Overview

The configuration system provides three primary capabilities:

1.   **Structured Data Validation**: Pydantic models ensure type safety and required field validation
2.   **Elasticsearch Integration**: Models automatically map to Elasticsearch field types for centralized logging
3.   **JSON Configuration**: Human-readable and machine-parseable configuration files for complex reset operations

## Pydantic Model Architecture

The model hierarchy is built on `ElasticModel`, which extends Pydantic's `BaseModel` with Elasticsearch mapping capabilities.

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 1-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L1-L185)

### ElasticModel Base Class

`ElasticModel` provides the foundation for all data models with Elasticsearch integration:

| Method | Return Type | Description |
| --- | --- | --- |
| `get_mapping()` | `dict` | Generates Elasticsearch field mappings from Pydantic field types |
| `save_as_json()` | `None` | Serializes model to JSON file with proper formatting |

The class implements automatic type mapping through helper functions:

*   `type_to_mapping()`: Converts Python types to Elasticsearch types
*   `field_to_mapping()`: Converts Pydantic fields to ES mappings
*   Supports custom types: `Long`, `Keyword`, `Text`, `Date`, `Nested[T]`

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 127-146](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L127-L146)

### Type Mapping System

The following table shows how Python types map to Elasticsearch types:

| Python Type | Elasticsearch Type | Notes |
| --- | --- | --- |
| `int` | `integer` | Standard integer |
| `Long` | `long` | 64-bit integer |
| `float` | `float` | Floating point |
| `bool` | `boolean` | Boolean |
| `str` | `text` + `keyword` | Text with keyword subfield |
| `Keyword` | `keyword` | Exact match only |
| `Text` | `text` | Full-text search |
| `bytes` | `binary` | Base64 encoded |
| `datetime.datetime` | `date` | ISO 8601 format |
| `Date` | `date` | Custom format support |
| `ElasticModel` | `object` | Nested object |
| `Nested[T]` | `nested` | Elasticsearch nested type |

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 68-116](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L68-L116)

## Reset Log Models

### HostResetLog

The top-level model for reset operation logs, capturing all information about a reset event:

**Field Descriptions**:

| Field | Type | Description |
| --- | --- | --- |
| `time` | `datetime.datetime` | Timestamp of reset operation |
| `host_name` | `str` | Hostname from `get_host_info()` |
| `gs_tensix_reset` | `PciResetDeviceInfo` | Grayskull devices to reset |
| `wh_link_reset` | `PciResetDeviceInfo` | Wormhole devices to reset |
| `re_init_devices` | `bool` | Whether to reinitialize devices post-reset |
| `wh_mobo_reset` | `List[MoboReset]` | Motherboard-level reset configurations |
| `disable_serial_report` | `bool` | Skip serial number reporting |
| `disable_sw_version_report` | `bool` | Skip software version reporting |

All fields are decorated with `@optional`, making them optional in the Pydantic schema.

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 167-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L167-L185)

### PciResetDeviceInfo

Represents a list of PCIe devices to be reset, identified by their PCI interface indices:

| Field | Type | Description |
| --- | --- | --- |
| `pci_index` | `List[int]` | List of PCI interface IDs (e.g., `[0, 1, 2]`) |

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 149-154](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L149-L154)

### MoboReset

Configuration for motherboard-level reset operations, typically used in multi-chip systems with custom PCIe switching:

| Field | Type | Description |
| --- | --- | --- |
| `nb_host_pci_idx` | `Union[List[int], None]` | Northbridge host PCIe indices |
| `mobo` | `str` | Motherboard identifier string |
| `credo` | `List[str]` | Credo retimer identifiers (format: `"group_id:credo_id"`) |
| `disabled_ports` | `List[str]` | Ports to keep disabled (format: `"group_id:credo_id"`) |

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 156-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L156-L165)

## Configuration Input System

The input system parses various reset specification formats into a unified `ResetInput` structure.

**Sources**: [tt_tools_common/reset_common/reset_utils.py 23-87](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L23-L87)

### ResetType Enum

Defines three modes of specifying which devices to reset:

`class ResetType(Enum):    ALL = 1           # Reset all detected devices    CONFIG_JSON = 2   # Use JSON configuration file    ID_LIST = 3       # List of PCI indices`
**Sources**: [tt_tools_common/reset_common/reset_utils.py 23-26](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L23-L26)

### ResetInput Dataclass

Container for parsed reset specifications:

| Field | Type | Description |
| --- | --- | --- |
| `type` | `ResetType` | The reset mode |
| `value` | `Union[Dict, int, None]` | Parsed value based on type |

**Sources**: [tt_tools_common/reset_common/reset_utils.py 28-31](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L28-L31)

### parse_reset_input Function

Parses various input formats into `ResetInput`:

**Input Formats**:

| Format | Example | Result |
| --- | --- | --- |
| Empty or "all" | `[]` or `["all"]` | `ResetInput(ALL, None)` |
| JSON file | `["config.json"]` | `ResetInput(CONFIG_JSON, {...})` |
| Comma-separated | `["0,1,2"]` | `ResetInput(ID_LIST, [0,1,2])` |
| Space-separated | `["0", "1", "2"]` | `ResetInput(ID_LIST, [0,1,2])` |

**Parsing Logic Flow**:

The function automatically:

*   Filters duplicate indices with `sorted(set(...))`
*   Validates JSON structure with `json.load()`
*   Provides helpful error messages with suggestions

**Sources**: [tt_tools_common/reset_common/reset_utils.py 33-87](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L33-L87)

## Configuration Generation

### generate_reset_logs Function

Creates a sample reset configuration file with placeholders for motherboard resets:

**Function Signature**:

`def generate_reset_logs(devices, result_filename: str = None) -> str`
**Parameters**:

| Parameter | Type | Description |
| --- | --- | --- |
| `devices` | `List[PciChip]` | List of pyluwen device objects |
| `result_filename` | `str` | Optional custom output path |

**Returns**: Path to generated JSON file (default: `~/.config/tenstorrent/reset_config.json`)

**Generation Logic**:

**Generated Template Structure**:

The function creates a `HostResetLog` with:

*   Current timestamp
*   Host information from `get_host_info()`
*   Detected device indices for Grayskull and Wormhole
*   Two template `MoboReset` entries with placeholder strings: 
    *   `"<MOBO NAME>"` for motherboard identifier
    *   `"<group id>:<credo id>"` for Credo retimer specifications

**Sources**: [tt_tools_common/reset_common/reset_utils.py 89-134](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L89-L134)

## JSON Configuration Format

### Complete Example

A fully configured reset log JSON file:

`{    "time": "2024-01-15T10:30:00.123456",    "host_name": "tt-host-01",    "gs_tensix_reset": {        "pci_index": []    },    "wh_link_reset": {        "pci_index": [0, 1, 2, 3]    },    "re_init_devices": true,    "disable_serial_report": false,    "disable_sw_version_report": false,    "wh_mobo_reset": [        {            "nb_host_pci_idx": [0, 1],            "mobo": "custom_galaxy_board",            "credo": ["0:0", "0:1", "1:0", "1:1"],            "disabled_ports": ["2:0", "2:1"]        },        {            "nb_host_pci_idx": [2, 3],            "mobo": "custom_galaxy_board",            "credo": ["0:2", "0:3", "1:2", "1:3"],            "disabled_ports": []        }    ]}`
### Field Reference

**Top-Level Fields**:

| Field | Required | Type | Example |
| --- | --- | --- | --- |
| `time` | Yes | ISO 8601 datetime | `"2024-01-15T10:30:00"` |
| `host_name` | Yes | String | `"tt-host-01"` |
| `gs_tensix_reset` | Yes | Object | `{"pci_index": []}` |
| `wh_link_reset` | Yes | Object | `{"pci_index": [0,1,2]}` |
| `re_init_devices` | Yes | Boolean | `true` |
| `wh_mobo_reset` | Yes | Array | `[{...}, {...}]` |
| `disable_serial_report` | Yes | Boolean | `false` |
| `disable_sw_version_report` | Yes | Boolean | `false` |

**PciResetDeviceInfo Structure**:

`{    "pci_index": [0, 1, 2]}`
**MoboReset Structure**:

`{    "nb_host_pci_idx": [0, 1],    "mobo": "board_identifier",    "credo": ["group_id:credo_id", ...],    "disabled_ports": ["group_id:port_id", ...]}`
**Credo Identifier Format**: `"<group_id>:<credo_id>"` where both IDs are integers encoded as strings

**Sources**: [tt_tools_common/reset_common/reset_utils.py 89-134](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L89-L134)[tt_tools_common/reset_common/host_reset_log.py 167-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L167-L185)

## Validation and Serialization

### Pydantic Compatibility

The models support both Pydantic v1 and v2 through conditional imports:

`try:    # Pydantic v2 compatibility    from pydantic.v1 import BaseModel    from pydantic.v1.fields import Fieldexcept:    # Pydantic v1 fallback    from pydantic import BaseModel    from pydantic.fields import Field`
This ensures compatibility across different Python environments and dependency versions.

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 15-22](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L15-L22)

### Optional Fields Decorator

The `@optional` decorator makes all fields in a model optional:

**Usage**:

`@optionalclass MyModel(ElasticModel):    field1: str    field2: int`
**Result**: All fields become `Optional[T]` with `default=None`

The decorator can also target specific fields:

`@optional("field1", "field2")class MyModel(ElasticModel):    field1: str  # Optional    field2: int  # Optional    field3: str  # Required`
**Sources**: [tt_tools_common/reset_common/host_reset_log.py 47-65](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L47-L65)

### JSON Serialization

The `save_as_json()` method provides formatted JSON output:

**Behavior**:

*   Excludes `None` values with `exclude_none=False`
*   Pretty-prints with 4-space indentation
*   Maintains field order from model definition
*   Writes to specified file path (creates directories if needed)

**Sources**: [tt_tools_common/reset_common/host_reset_log.py 180-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L180-L185)

## Integration with Reset System

The models integrate with the reset system through this flow:

**Key Integration Points**:

1.   **Input Parsing**: `parse_reset_input()` converts various formats to `ResetInput`
2.   **Configuration Loading**: JSON files are loaded and validated against `HostResetLog` schema
3.   **Reset Execution**: Reset classes consume validated configuration data
4.   **Configuration Generation**: `generate_reset_logs()` creates templates for future use

**Sources**: [tt_tools_common/reset_common/reset_utils.py 1-135](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L1-L135)[tt_tools_common/reset_common/host_reset_log.py 1-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L1-L185)
