## Purpose and Scope

This document describes the configuration parameters available for the Tenstorrent kernel driver module (`tenstorrent.ko`). Module parameters allow administrators to customize driver behavior at module load time, controlling resource limits, hardware behavior, and power management policies. These parameters are essential for tuning the driver to specific hardware topologies (like Galaxy systems) or performance requirements.

**Sources:**[module.c 1-121](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L1-L121)[module.h 1-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L1-L38)

## Overview

The Tenstorrent kernel driver exposes six module parameters that control driver initialization and runtime behavior. Most parameters are declared as read-only (`0444` permissions), meaning they can only be set at module load time. One parameter, `idle_power_down_grace_ms`, is writable (`0644`), allowing runtime adjustment of power management timing.

## Parameter Reference

| Parameter | Type | Default | Permissions | Description |
| --- | --- | --- | --- | --- |
| `max_devices` | uint | 32 | 0444 | Maximum number of Tenstorrent devices (chips) to support. |
| `dma_address_bits` | uint | 0 | 0444 | DMA address width in bits; 0 enables automatic detection. |
| `reset_limit` | uint | 10 | 0444 | Maximum reset attempts during device boot/probe. |
| `auto_reset_timeout` | byte | 10 | 0444 | Timeout (seconds) for M3 firmware auto-reset. |
| `power_policy` | bool | true | 0444 | Enables low-power mode at probe and aggregation on close. |
| `idle_power_down_grace_ms` | uint | 5000 | 0644 | Delay before idle power-down after last FD close. |

**Sources:**[module.c 36-62](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L36-L62)

## Detailed Parameter Descriptions

### max_devices

**Definition:**[module.c 36-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L36-L38) Limits the number of Tenstorrent devices the driver will manage. This value is passed directly to `init_char_driver()`[module.c 90](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L90-L90) to allocate the necessary character device major/minor numbers and internal tracking structures.

### dma_address_bits

**Definition:**[module.c 40-42](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L40-L42) Controls the DMA mask used by the driver.

*   **Value 0 (Default):** The driver automatically selects the mask based on the device class. Wormhole typically uses 32-bit, while Blackhole supports larger address spaces.
*   **Manual Override:** Can be set to specific values (e.g., 32, 48) if the system IOMMU or platform requires a restricted DMA range.

### reset_limit

**Definition:**[module.c 44-46](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L44-L46) Specifies the maximum number of times the driver will attempt to reset the hardware during the `probe` sequence if the device fails to initialize or the ARC/M3 firmware does not reach a ready state.

### auto_reset_timeout

**Definition:**[module.c 48-50](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L48-L50) Sets the wait duration for the M3 (Management Processor) to perform an automatic reset. This is particularly relevant for complex multi-chip systems where link training and internal resets take significant time.

### power_policy

**Definition:**[module.c 52-54](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L52-L54) When `true`, the driver places the device into a low-power state immediately after a successful probe. Power is only ramped up when a user-space process opens the character device and requests specific power states via IOCTL. On the last close, the driver re-aggregates power requirements and returns to the idle state.

### idle_power_down_grace_ms

**Definition:**[module.c 56-62](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L56-L62) This parameter defines the delay between the last user-space file descriptor closing and the driver sending the idle power-down message to the firmware.

*   **Value 0:** The message is sent synchronously during the `close()` operation.
*   **Value > 0:** The driver defers the power-down, which prevents rapid power-cycling if a new process opens the device shortly after another closes it.
*   **Note:** This is only honored by device classes that opt-in via the `defer_idle_powerdown` flag in their `tenstorrent_device_class` definition.

**Sources:**[module.c 36-62](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L36-L62)[module.h 25-29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.h#L25-L29)

## Implementation and Lifecycle

The parameters are initialized during the `ttdriver_init` sequence.

**Sources:**[module.c 76-108](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L76-L108)

## Runtime Inspection

While most parameters are read-only at the filesystem level (`0444`), their values can be verified via `sysfs`:

`# View the current value of power_policycat /sys/module/tenstorrent/parameters/power_policy # Adjust the idle grace period (writable 0644)echo 10000 | sudo tee /sys/module/tenstorrent/parameters/idle_power_down_grace_ms`
**Sources:**[module.c 37-57](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/module.c#L37-L57)
