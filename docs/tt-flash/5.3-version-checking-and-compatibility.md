# Version Checking and Compatibility

Relevant source files
*   [tt_flash/__init__.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/__init__.py)
*   [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py)
*   [tt_flash/utility.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py)

This document describes the firmware version checking and compatibility enforcement mechanisms in tt-flash. Version checking occurs during the FLASH stage (specifically in `flash_chip_stage1`) to determine whether a chip requires firmware updates and to prevent incompatible downgrades.

For information about the overall flashing workflow, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages). For details on firmware bundle structure and versioning metadata, see [Firmware Bundle Format](https://deepwiki.com/tenstorrent/tt-flash/5.1-firmware-bundle-format).

## Purpose and Scope

The version checking system ensures:

*   Chips are not flashed unnecessarily when already running current or newer firmware
*   Major version downgrades are blocked unless explicitly permitted
*   Legacy firmware format transitions (80.X.X.X → X.X.X) are handled correctly
*   Both SPI flash content and running firmware versions are considered
*   Appropriate warnings and errors guide users through version boundaries

## Version Format and Normalization

### Version Tuple Structure

Firmware versions are represented as 4-tuples: `(fwId, releaseId, patch, debug)`. The version format changed at major version 18 from a prefixed format to a standard semantic versioning approach.

**Diagram: Version Format Evolution**

**Sources:**[tt_flash/flash.py 127-139](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L127-L139)

### Normalization Implementation

The `normalize_fw_version()` function handles the legacy format transition:

`# Key logic from normalize_fw_versionif version[0] == 80:    return (version[1], version[2], version[3], 0)return version`
This normalization is applied to three version sources:

*   **SPI version**: Firmware currently stored in SPI flash
*   **Running version**: Firmware currently executing on the chip
*   **Manifest version**: Firmware version in the bundle being flashed

**Sources:**[tt_flash/flash.py 127-139](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L127-L139)[tt_flash/flash.py 226-228](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L226-L228)

## Version Source Hierarchy

### Version Retrieval

The system retrieves versions from multiple sources through `chip.get_bundle_version()`:

| Version Source | Description | Availability |
| --- | --- | --- |
| **SPI Version** | Version stored in SPI flash memory | Available on most firmware ≥v6.0 |
| **Running Version** | Version currently executing on M3/ARC | Available on firmware ≥v6.0 |
| **Manifest Version** | Version in the `.fwbundle` being flashed | Always available |

**Diagram: Version Source Retrieval Flow**

**Sources:**[tt_flash/flash.py 223-228](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L223-L228)[tt_flash/flash.py 249-334](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L249-L334)

### Exception Handling

When version retrieval fails, the system has different policies based on chip architecture:

**Wormhole (WhChip)**:

*   Old firmware may not support version queries
*   Exception allowed if `force` flag is used
*   Falls back to assuming update is needed

**Blackhole (BhChip)**:

*   Must always successfully return firmware version
*   Exception causes immediate failure
*   No fallback permitted

**Sources:**[tt_flash/flash.py 230-246](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L230-L246)

## Version Comparison Logic

### Decision Tree

**Diagram: Version Checking Decision Flow**

**Sources:**[tt_flash/flash.py 194-492](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L194-L492)

### Major Version Boundaries

The system enforces strict rules around major version changes (changes to `version[0]`):

| Scenario | Default Behavior | Override Flag |
| --- | --- | --- |
| **Major downgrade** (e.g., v19 → v18) | Blocked with `TTError` | `--allow-major-downgrades` |
| **One major upgrade** (e.g., v18 → v19) | Permitted with yellow warning | None needed |
| **Multi-major jump** (e.g., v17 → v19) | Blocked with `TTError` | `--force` |
| **Same major, any minor/patch** | Permitted silently | None needed |

**Code Implementation:**

`# Major version downgrade preventionif running_version[0] > manifest_version[0]:    if allow_major_downgrades:        print("Major version downgrade, but allowed")    else:        raise TTError("Major version downgrade not supported") # Single major version upgrade permittedif running_version[0] == manifest_version[0] - 1:    print("Detected major version upgrade from X to X+1") # Multi-major version changes require forceelif running_version[0] != manifest_version[0]:    if force:        print("Unexpected bundle version, forcing anyway")    else:        raise TTError("Bundle fwId mismatch; bypass with --force")`
**Sources:**[tt_flash/flash.py 263-289](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L263-L289)

### SPI vs Running Version Priority

When both SPI and running versions are available, the system prioritizes SPI version for determining update necessity:

**Diagram: SPI Priority Logic**

**Sources:**[tt_flash/flash.py 299-330](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L299-L330)

This priority system ensures:

1.   **SPI flash content is authoritative** - If SPI is up to date, no flash occurs even if running version is old
2.   **Reboot instructions given** - User informed they can load new firmware via reboot/reset
3.   **Force flag bypasses** - `--force` overrides all version checks

## Override Flags

### Force Flag (`--force`)

The `force` flag bypasses most version checks and forces a flash operation:

| Condition | Without `--force` | With `--force` |
| --- | --- | --- |
| Cannot retrieve version | Error | Proceed with warning |
| Running very old firmware | Error | Proceed with assumption |
| Multi-major version jump | Error | Proceed with warning |
| SPI already up to date | Skip flash | Force flash anyway |

**Sources:**[tt_flash/flash.py 234-241](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L234-L241)[tt_flash/flash.py 253-260](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L253-L260)[tt_flash/flash.py 281-289](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L281-L289)[tt_flash/flash.py 296-298](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L296-L298)

### Allow Major Downgrades Flag (`--allow-major-downgrades`)

This flag specifically permits major version downgrades that are normally blocked:

`if running_version[0] > manifest_version[0]:    if allow_major_downgrades:        print(f"Detected major version downgrade from {running_version} "              f"to {manifest_version}, but major downgrades are allowed")    else:        raise TTError(            f"Detected major version downgrade, not supported. "            "Re-run with --allow-major-downgrades"        )`
**Sources:**[tt_flash/flash.py 263-273](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L263-L273)

## Version Comparison Examples

### Scenario 1: Standard Update

```
Current Running: (18, 5, 2, 0)
Current SPI:     (18, 5, 2, 0)
Bundle:          (18, 6, 0, 0)

Result: Flash proceeds (minor version increment)
```

### Scenario 2: Major Version Upgrade

```
Current Running: (18, 12, 5, 0)
Current SPI:     (18, 12, 5, 0)
Bundle:          (19, 0, 0, 0)

Result: Flash proceeds with yellow warning about major upgrade
```

### Scenario 3: Already Updated SPI

```
Current Running: (18, 5, 2, 0)
Current SPI:     (18, 6, 0, 0)
Bundle:          (18, 6, 0, 0)

Result: No flash - SPI already contains target version, reboot recommended
```

### Scenario 4: Major Downgrade (Blocked)

```
Current Running: (19, 2, 1, 0)
Current SPI:     (19, 2, 1, 0)
Bundle:          (18, 12, 5, 0)

Result: TTError unless --allow-major-downgrades is used
```

### Scenario 5: Multi-Major Jump (Blocked)

```
Current Running: (17, 8, 0, 0)
Current SPI:     (17, 8, 0, 0)
Bundle:          (19, 0, 0, 0)

Result: TTError - cannot skip major version 18. Use --force to override.
```

**Sources:**[tt_flash/flash.py 263-334](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L263-L334)

## Post-Flash Version Validation

For Blackhole chips running firmware v19 or newer, an additional cryptographic validation occurs after reset:

`if manifest.bundle_version[0] >= 19 and isinstance(chip, BhChip):    check_val = random.randint(1, 0xFFFF)    response = chip.arc_msg(chip.fw_defines["MSG_CONFIRM_FLASHED_SPI"], arg0=check_val)    if (response[0] & 0xFFFF) != check_val:        print("WARNING: Post flash check failed")`
This challenge-response confirms the new firmware is actually running from SPI flash.

**Sources:**[tt_flash/flash.py 942-951](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L942-L951)
