## Purpose and Scope

The Test Plan Registry System provides a centralized mechanism for registering, organizing, and discovering test plans within the coretp framework. It implements a decorator-based pattern that allows test scenarios to self-register during module import, creating a discoverable catalog of all available test plans. The registry supports metadata-based queries through tags and features, enabling test plan selection based on functional area, extension type, or other criteria.

For information about how test scenarios are structured and executed, see [Test Step System](https://deepwiki.com/tenstorrent/riscv-coretp/3.3-test-step-system). For details on environment configuration resolved per test plan, see [Test Environment Configuration](https://deepwiki.com/tenstorrent/riscv-coretp/3.2-test-environment-configuration). For comprehensive coverage of individual test plans, see [Test Plans Overview](https://deepwiki.com/tenstorrent/riscv-coretp/4-test-plans-overview).

## _TestPlanInfo Data Structure

The `_TestPlanInfo` dataclass serves as the registry entry for each test plan, managing metadata and scenario accumulation.

### Fields and Methods

| Field | Type | Purpose |
| --- | --- | --- |
| `name` | `str` | Unique identifier for the test plan |
| `description` | `str` | Human-readable description |
| `tags` | `list[str]` | Metadata tags (e.g., `"cache"`, `"paging"`) |

| `_scenarios` | `list[Callable[[], TestScenario]]` | Scenario factory functions |

## Lifecycle Management

The registry implements a two-phase lifecycle to ensure immutability and consistent behavior.

### Phase 1: Accumulation (Module Import)

During Python module import, test scenario functions register themselves:

**Sources:**[coretp/plans/__init__.py 8-18](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/__init__.py#L8-L18)

### Phase 2: Frozen (After First Retrieval)

Once a plan is built, no more scenarios can be added:

`# First retrieval builds and caches the planplan = get_plan("svadu")  # Executes all scenario functions # Subsequent calls return cached instanceplan2 = get_plan("svadu")  # Same objectassert plan is plan2  # True # Attempting to add scenarios now raises RuntimeError# (This would only happen during development if modules are imported after retrieval)`
**Error Handling:**

`# coretp/plans/test_plan_registry.py:65-69def add_scenario(self, scenario_func: Callable[[], TestScenario]) -> None:    """Add a scenario function to this plan"""    if self._built_plan is not None:        raise RuntimeError(f"Cannot add scenarios to already-built plan '{self.name}'")    self._scenarios.append(scenario_func)`

## Module Import Order

The registry requires all test plan modules to be imported for registration to occur. This is enforced at the package level:

`# coretp/plans/__init__.py:8-18# Have to import all plans here to ensure they are registered. # Need to include the module itself to ensure it's registered.from .paging import page_table_walksfrom .sstc import sstc_scenariosfrom .svadu import svadu_scenariosfrom .zicond import zicond_scenariosfrom .zkt import zkt_scenariosfrom .zimop_zcmop import zimop_zcmop_scenariosfrom .zifencei import zifencei_scenariosfrom .zicbom_zicboz_zicbop import zicbom_zicboz_zicbop_scenariosfrom .zicntr_zihpm_sscounterenw import zicntr_zihpm_sscounterenw_scenariosfrom .sscofpmf import sscofpmf_scenarios`
This ensures that when `coretp.plans` is imported, all decorators execute and scenarios register themselves with the global `_registry` instance.

## Summary

The Test Plan Registry System provides:

1.   **Centralized Registration**: Single global registry (`_registry`) manages all test plans
2.   **Decorator Pattern**: `@new_test_plan()` enables declarative scenario registration
3.   **Lazy Evaluation**: Scenario functions stored but not executed until plan retrieval
4.   **Immutability**: Built plans are cached and protected from modification
5.   **Metadata Queries**: Filter plans by name, tags, or features without building them
6.   **Import-Time Registration**: Decorators execute during module import, ensuring all plans are discoverable

The registry serves as the entry point for test discovery in the coretp framework, enabling tools like `coretp-export` to enumerate and document all available test scenarios.
