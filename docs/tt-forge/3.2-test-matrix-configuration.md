## Purpose and Scope

This document details the test matrix configuration system that defines how benchmark and demo tests are organized and filtered in the TT-Forge repository. The system utilizes JSON matrix files to specify test configurations and a Python filtering script to dynamically select tests based on project, test name, and hardware architecture.

The matrix configuration system is used by several primary workflows:

*   **perf-benchmark.yml** - Performance benchmarks using `perf-bench-matrix.json`
*   **demo-tests.yml** - Demo tests using `models-matrix.json`[.github/workflows/demo-tests.yml 74](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/demo-tests.yml#L74-L74)
*   **basic-tests.yml** - Quick validation tests using a dynamically generated matrix [.github/workflows/basic-tests.yml 59-98](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/basic-tests.yml#L59-L98)

## Filter Script Implementation

### filter-test-matrix.py Functions

The `filter-test-matrix.py` script performs three primary operations on matrix configuration files:

#### 1. flatten_matrix()

Expands test configurations where `runs-on` is an array into individual entries, ensuring each job in the GitHub Actions matrix has a single target architecture.

`def flatten_matrix(data):    matrix = []    for proj in data:        test_defaults = proj.get("test-defaults", {})        for test in proj.get("tests", []):            merged_test = {**test_defaults, **test, "project": proj["project"]}            runs_on = merged_test.get("runs-on", [])            if isinstance(runs_on, list):                matrix.extend({**merged_test, "runs-on": runner} for runner in runs_on)            else:                matrix.append(merged_test)    return matrix`
**Sources:**[.github/workflows/filter-test-matrix.py 10-24](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L10-L24)

#### 2. filter_matrix()

Filters the flattened matrix based on the `project_filter` and an optional `name_filter`. The `name_filter` supports comma-separated substrings for partial matches.

`def filter_matrix(matrix, project_filter, name_filter=None):    name_filters = name_filter.split(",") if name_filter else []    def should_include(item):        if project_filter == "tt-forge" and item.get("project") not in ["tt-xla"]:            return False        if project_filter != "tt-forge" and item.get("project") != project_filter:            return False        # ... logic for name_filters ...        return True    return [item for item in matrix if should_include(item)]`
**Sources:**[.github/workflows/filter-test-matrix.py 27-48](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L27-L48)

#### 3. update_runners()

Maps logical runner names (e.g., `n150`) to specific labels used by the Tenstorrent CI infrastructure.

| Logical Name | With `--sh-runner` | Without `--sh-runner` |
| --- | --- | --- |
| `n150` | `n150` | `n150-perf` |
| `p150` | `p150b` | `p150` |

**Sources:**[.github/workflows/filter-test-matrix.py 51-60](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/filter-test-matrix.py#L51-L60)

### Basic Tests (basic-tests.yml)

Unlike the benchmark and demo workflows, `basic-tests.yml` generates its matrix dynamically in bash using `jq` instead of the Python filter script. This is used for high-level smoke tests of the frontends.

`# From basic-tests.yml build_matrix jobfor frontend in $frontends; do  for run_on in $runs_on; do    json_matrix=$(echo $json_matrix | jq -r -c --arg frontend "$frontend" --arg run_on "$run_on" '. += [{    "frontend": $frontend,    "runs_on": $run_on    }]')  donedone`
