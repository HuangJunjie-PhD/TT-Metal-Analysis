# TT-Metalium Integration

Relevant source files
*   [README.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1)

## Purpose and Scope

This page describes how ttsim integrates with the TT-Metalium software stack. It covers the shared library interface, environment variable configuration, SOC descriptor requirements, and dispatch mode settings that enable TT-Metalium to discover and communicate with the simulator. For overall system architecture, see [System Architecture](https://deepwiki.com/tenstorrent/ttsim/3.1-system-architecture). For configuration details and troubleshooting, see [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration).

## Integration Architecture

TT-Metalium interfaces with ttsim through a shared library API. The simulator is packaged as architecture-specific shared objects (`libttsim_wh.so` for Wormhole, `libttsim_bh.so` for Blackhole) that export an API TT-Metalium uses to drive simulation. Discovery and configuration occur through environment variables and file system conventions.

**Integration Flow Diagram**

The integration follows this sequence:

1.   TT-Metalium runtime checks environment variables to determine simulator mode
2.   If `TT_METAL_SIMULATOR` is set, it loads the specified shared library
3.   SOC descriptor is loaded from the same directory as the simulator library
4.   Dispatch engine configuration is validated (slow dispatch required)
5.   API calls are routed through the simulator instead of hardware drivers

## Environment Variable Interface

TT-Metalium discovers and configures ttsim through three mandatory environment variables:

| Variable | Purpose | Valid Values | Required |
| --- | --- | --- | --- |
| `TT_METAL_HOME` | Locates TT-Metalium installation | Absolute path to tt-metal directory | Yes |
| `TT_METAL_SIMULATOR` | Specifies simulator library path | Absolute path to `libttsim_*.so` | Yes (for simulator mode) |
| `TT_METAL_SLOW_DISPATCH_MODE` | Forces slow dispatch | `1` | Yes (currently) |

### TT_METAL_HOME

This variable must point to the root directory of the TT-Metalium installation. TT-Metalium uses this path to:

*   Locate SOC descriptor source files in `$TT_METAL_HOME/tt_metal/soc_descriptors/`
*   Resolve internal dependencies and configurations
*   Determine build artifacts location

Sources: [README.md 25](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L25-L25)

### TT_METAL_SIMULATOR

When set, this variable switches TT-Metalium from hardware mode to simulator mode. The value must be an absolute path to the simulator shared library. The choice of library determines the target architecture:

TT-Metalium treats this as a simple presence check: if the variable exists and points to a valid shared library, simulator mode activates. If unset, TT-Metalium attempts to use physical hardware.

### TT_METAL_SLOW_DISPATCH_MODE

This variable must be set to `1` when using ttsim. Fast dispatch mode is not currently supported by the simulator.

The dispatch mode affects how TT-Metalium queues and submits work to the device. Slow dispatch provides simpler synchronization semantics that the simulator currently implements.

## Shared Library API

The simulator exports a C/C++ API that TT-Metalium dynamically loads and invokes. While the source code is not yet publicly available, the integration pattern follows standard shared library conventions.

**Simulator API Functional Decomposition**

Sources: [README.md 9-10](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L9-L10)

### API Characteristics

The API exported by `libttsim.so` enables:

*   Device initialization and configuration
*   Memory buffer allocation and management
*   Kernel binary loading and execution
*   Work queue submission and synchronization
*   Data transfer between host and simulated device memory
*   Device state queries and control

TT-Metalium uses the same API abstraction layer for both hardware and simulator, selecting the implementation at runtime based on the `TT_METAL_SIMULATOR` environment variable.

Sources: [README.md 9-10](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L9-L10)

## Dispatch Mode Configuration

The simulator currently requires slow dispatch mode, which affects how TT-Metalium batches and submits work to the device.

### Slow Dispatch vs Fast Dispatch

| Aspect | Slow Dispatch (Required) | Fast Dispatch (Not Supported) |
| --- | --- | --- |
| Work Submission | Synchronous, kernel-at-a-time | Asynchronous, batched queuing |
| Synchronization | Explicit after each kernel | Implicit barrier management |
| Host Overhead | Higher per-kernel cost | Amortized across batch |
| Simulator Support | ✓ Fully implemented | ✗ Not yet available |

### Configuration

Slow dispatch mode must be explicitly enabled when running any TT-Metalium program with ttsim:

Attempting to use the simulator without this setting will result in errors. The restriction exists because fast dispatch requires additional simulator infrastructure that is under development.

## Integration Validation

After configuration, TT-Metalium's runtime performs validation checks:

1.   **Environment Variable Presence**: Verifies `TT_METAL_HOME` and `TT_METAL_SIMULATOR` are set
2.   **Library Loading**: Attempts to dlopen the specified `.so` file
3.   **SOC Descriptor Discovery**: Searches for `soc_descriptor.yaml` adjacent to the library
4.   **Dispatch Mode Validation**: Ensures slow dispatch is enabled
5.   **API Symbol Resolution**: Resolves expected function symbols from the shared library

Configuration errors at any stage produce error messages categorized as `ConfigurationError` or `SystemError`.
