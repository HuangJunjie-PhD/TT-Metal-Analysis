## Purpose and Scope

The Layout Transformation System manages data layout conversions in Relay IR to optimize operator execution on different hardware targets. This system enables TVM to transform tensor layouts (e.g., NCHW ↔ NHWC) throughout the computation graph to match target-specific performance requirements.

For information about the Relay IR operators themselves, see [Neural Network Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.1-neural-network-operators) and [Tensor Transformation Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.2-tensor-transformation-operators). For overall optimization pass infrastructure, see [Optimization Passes](https://deepwiki.com/tenstorrent/tt-tvm/4.5-optimization-passes).

* * *

## Overview

Tensor data layout refers to the ordering of dimensions in memory. For example, a 4D image tensor can be stored as:

*   **NCHW**: Batch, Channels, Height, Width (common on CUDA GPUs)
*   **NHWC**: Batch, Height, Width, Channels (common on mobile/CPU)
*   **NCHW16c**: Batch, Channels (outer), Height, Width, Channels (inner, 16-element blocks)

The Layout Transformation System handles three key tasks:

1.   **Layout Suggestion**: Analyze operators and suggest optimal layouts for the target hardware
2.   **Layout Conversion**: Force specific layouts by inserting `layout_transform` operations
3.   **Layout Propagation**: Minimize redundant transformations by propagating layout requirements through the graph

Sources: [src/relay/transforms/alter_op_layout.cc 1-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/transforms/alter_op_layout.cc#L1-L100)[src/relay/transforms/convert_layout.cc 1-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/transforms/convert_layout.cc#L1-L100)

## Core Layout Infrastructure

### BijectiveLayout Class

**BijectiveLayout Class Functionality**

`tir::BijectiveLayout` represents a bijective (one-to-one) mapping between two layouts. Key operations:

| Method | Purpose | Example |
| --- | --- | --- |
| `ForwardShape()` | Transform shape from source to target | NCHW shape (1,64,56,56) → NHWC shape (1,56,56,64) |
| `BackwardShape()` | Transform shape from target to source | NHWC shape (1,56,56,64) → NCHW shape (1,64,56,56) |
| `defined()` | Check if transformation is valid | Returns true if layouts are compatible |

The system validates that transformations are bijective (no information loss) before applying them. Non-bijective transformations (e.g., NCHW → NC) are rejected.

Sources: [src/relay/op/nn/convolution.cc 74-88](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/op/nn/convolution.cc#L74-L88)[src/relay/transforms/transform_layout.h 1-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/transforms/transform_layout.h#L1-L100)

### layout_transform Operator

The `layout_transform` operator is inserted into the IR to perform actual data reordering:

`# Python API exampley = relay.layout_transform(x, "NCHW", "NHWC")`
This operator physically reorders tensor data in memory to match the target layout.

## Layout Transformation Passes

### AlterOpLayout Pass

**AlterOpLayout Pass Operation**

The `AlterOpLayout` pass allows operators to suggest alternative layouts that may be more efficient for the target hardware. It works by:

1.   **Querying Operators**: For each operator with `FTVMAlterOpLayout` registered, call the handler function
2.   **Receiving Suggestions**: Handler returns modified operator with suggested layouts
3.   **Inserting Transformations**: Insert `layout_transform` operations where layout changes occur
4.   **Propagating Changes**: Update downstream operators to use new layouts

**Registration Pattern**

`# From python/tvm/relay/op/nn/_nn.py:106-110@reg.register_alter_op_layout("nn.dense")def alter_op_layout_dense(attrs, inputs, tinfos, out_type):    """Alternate the layout of dense"""    return topi.nn.dense_alter_layout(attrs, inputs, tinfos, out_type)`
The handler function receives:

*   `attrs`: Current operator attributes
*   `inputs`: Input expressions
*   `tinfos`: Input type information
*   `out_type`: Expected output type

And returns either:

*   Modified `Call` expression with new layout
*   `None` to indicate no change

Sources: [src/relay/transforms/alter_op_layout.cc 1-200](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/transforms/alter_op_layout.cc#L1-L200)[python/tvm/relay/op/nn/_nn.py 106-110](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L106-L110)[python/tvm/relay/op/nn/_nn.py 215-219](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L215-L219)

### ConvertLayout Pass

**ConvertLayout Pass Operation**

Unlike `AlterOpLayout` which suggests layouts, `ConvertLayout`**forces** specific layouts based on user configuration:

1.   **Accept Configuration**: User specifies desired layouts per operator type
2.   **Match Operators**: Find all instances of configured operator types
3.   **Call Converter**: Invoke registered `FTVMConvertOpLayout` function
4.   **Insert Transforms**: Add `layout_transform` operations as needed

**Key Differences from AlterOpLayout**

| Aspect | AlterOpLayout | ConvertLayout |
| --- | --- | --- |
| Decision Making | Target-driven suggestions | User-specified requirements |
| Flexibility | Operators propose changes | Fixed configuration |
| Use Case | Hardware optimization | API standardization, testing |

**Registration Pattern**

`# From python/tvm/relay/op/nn/_nn.py:242-316@reg.register_convert_op_layout("nn.conv2d")def convert_conv2d(attrs, inputs, tinfos, desired_layouts):    """Convert Layout pass registration for conv2d op."""    data, weight = inputs    # ... process desired_layouts    new_attrs = dict(attrs)    new_attrs["data_layout"] = desired_data_layout        if desired_kernel_layout != "default":        new_attrs["kernel_layout"] = desired_kernel_layout        return relay.nn.conv2d(data, weight, **new_attrs)        # Handle default kernel layouts based on data layout    if desired_data_layout == "NCHW":        new_attrs["kernel_layout"] = "OIHW"    elif desired_data_layout == "NHWC":        new_attrs["kernel_layout"] = "HWIO"        return relay.nn.conv2d(data, weight, **new_attrs)`

### Example: conv2d Layout Handlers

**AlterOpLayout Handler for conv2d**

The handler in [python/tvm/relay/op/nn/_nn.py 215-219](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L215-L219) delegates to TOPI which analyzes:

*   Target hardware capabilities (AVX512, NEON, Tensor Cores)
*   Input tensor shapes and current layouts
*   Computational intensity

And may suggest:

*   `NCHW16c` on x86 with AVX-512 for vectorization
*   `NHWC` on ARM for better cache locality
*   No change if current layout is already optimal

**ConvertOpLayout Handler for conv2d**

The handler in [python/tvm/relay/op/nn/_nn.py 242-316](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L242-L316) performs:

1.   **LayoutConfig Check**: Respects `LayoutConfig.current` to skip specific layers if needed
2.   **Layout Parsing**: Extracts `desired_data_layout` and `desired_kernel_layout` from configuration
3.   **Kernel Inference**: If kernel layout is "default", infers appropriate layout: 
    *   `NCHW` data → `OIHW` kernel
    *   `NHWC` data → `HWIO` kernel (or `HWOI` for depthwise)
    *   `HWNC` data → `HWOI` kernel

4.   **Tiled Layout Handling**: For layouts like `NCHW16c`, validates tile alignment and calls `contrib_conv2d_nchwc`

## Layout Propagation and Optimization

### Propagation Strategy

**Layout Propagation Mechanism**

The system propagates layout requirements through the computation graph:

1.   **Forward Analysis**: Starting from graph inputs, track current layouts
2.   **Operator Consultation**: At each operator, check if layout change is needed
3.   **Transform Insertion**: Insert `layout_transform` where layouts mismatch
4.   **Backward Propagation**: After final operators, insert transforms back to expected output layouts

**Common Optimization Patterns**

| Pattern | Before | After | Benefit |
| --- | --- | --- | --- |
| Transform Cancellation | `transform(L1→L2)` → `transform(L2→L1)` | Identity | Eliminates redundant work |
| Transform Fusion | `transform(L1→L2)` → `transform(L2→L3)` | `transform(L1→L3)` | Reduces operations |
| Broadcast Compatible | Multiple consumers with same layout | Single transform, shared result | Memory reuse |

Sources: [src/relay/transforms/alter_op_layout.cc 1-200](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/transforms/alter_op_layout.cc#L1-L200)[tests/python/relay/test_pass_alter_op_layout.py 105-167](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/python/relay/test_pass_alter_op_layout.py#L105-L167)

### Example: Dual-Path Layout Handling

From [tests/python/relay/test_pass_alter_op_layout.py 267-330](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/tests/python/relay/test_pass_alter_op_layout.py#L267-L330):

```
Input (NCHW)
    ↓
conv2d → suggest NCHW16c
    ↓ [transform NCHW → NCHW16c]
relu (accepts NCHW16c)
    ├─→ Path 1: conv2d (continues NCHW16c)
    │       ↓
    │   relu → [transform NCHW16c → NCHW]
    │
    └─→ Path 2: batch_flatten (requires NCHW)
            ↓ [transform NCHW16c → NCHW]
        flatten output
```

This demonstrates:

*   **Shared Transformation**: First `NCHW → NCHW16c` benefits both paths initially
*   **Divergent Requirements**: Path 1 continues optimized, Path 2 needs original layout
*   **Minimal Transforms**: Only insert transforms where absolutely necessary

### Special Layout Handling

**Depthwise Convolution**

Depthwise convolutions require different kernel layouts. The system detects depthwise convolutions using [python/tvm/relay/op/strategy/generic.py](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/generic.py) and adjusts kernel layout accordingly:

*   Standard: `HWIO`
*   Depthwise: `HWOI`

**Tiled Layouts (NCHW16c, OIHW16i, etc.)**

Tiled layouts pack data into blocks for vectorization:

*   Validate tile size divides dimension evenly
*   May call specialized operators like `contrib_conv2d_nchwc`
*   Require both input and weight to be tiled consistently

Example from [python/tvm/relay/op/nn/_nn.py 306-314](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/nn/_nn.py#L306-L314):

`need_tile = re.match(r"NCHW(\d*)c", desired_data_layout)if need_tile:    tile = int(need_tile.group(1))    if isinstance(data, relay.expr.Var) and data.checked_type.shape[1] % tile != 0:        return relay.nn.conv2d(data, weight, **attrs)  # Keep original    else:        new_attrs["kernel_layout"] = desired_kernel_layout        return relay.nn.contrib_conv2d_nchwc(data, weight, **new_attrs)`

## Usage Patterns

### Running AlterOpLayout

`# Typical usage in optimization pipelineimport tvmfrom tvm import relay mod = tvm.IRModule.from_expr(func) # Run AlterOpLayout passwith tvm.transform.PassContext(opt_level=3):    mod = relay.transform.AlterOpLayout()(mod)`
### Running ConvertLayout

`# Force specific layoutslayout_config = {    "nn.conv2d": ["NHWC", "HWIO"],    "nn.dense": ["NC", "NC"]} with tvm.transform.PassContext(opt_level=3):    mod = relay.transform.ConvertLayout(layout_config)(mod)`
### Combined Usage

The passes are often used together in the compilation pipeline:

`seq = tvm.transform.Sequential([    relay.transform.InferType(),    relay.transform.CanonicalizeOps(),    relay.transform.AlterOpLayout(),      # Suggest optimal layouts    relay.transform.FoldConstant(),    relay.transform.ConvertLayout(config) # Force final layouts if needed])`
