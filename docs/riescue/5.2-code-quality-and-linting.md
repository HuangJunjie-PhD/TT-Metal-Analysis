## Purpose and Scope

This document covers the code quality tools and linting infrastructure used in RiESCUE to maintain consistent code style and catch errors before runtime. It describes three primary tools (`black`, `flake8`, and `pyright`), their configurations, local usage, CI/CD integration, and automated pre-commit hooks.

For information about the broader CI/CD pipeline including testing and deployment, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline). For general contributing guidelines and code standards, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/riescue/5.1-contributing-guidelines).

## Overview of Linting Tools

RiESCUE enforces code quality through three complementary tools that run both locally and in CI:

| Tool | Purpose | Type | Configuration File |
| --- | --- | --- | --- |
| `black` | Code formatting - enforces whitespace and style rules | Formatter | `.black` |
| `flake8` | Style guide enforcement - checks PEP 8 compliance | Linter | `.flake8` |
| `pyright` | Static type checking - validates type hints | Type Checker | `pyproject.toml` |

These tools are designed to work together: `black` automatically fixes formatting issues, `flake8` catches style violations that `black` doesn't handle, and `pyright` validates type correctness to prevent runtime errors.

## Flake8 Linter

### Configuration

Flake8 configuration is stored in `.flake8`. Key settings include:

| Setting | Value | Purpose |
| --- | --- | --- |
| `max-line-length` | 200 | Maximum line length (more lenient than PEP 8's 79) |
| `statistics` | True | Show error statistics |
| `show-source` | True | Display source code for errors |
| `output_file` | flake8.log | Log file for results |

### Excluded Directories

The following directories are excluded from flake8 checks:

```
riscv-dv, riscv-isa-sim, whisper, fpgen, riescue_config
*_clone, *-scripts, infra, .git, opensrc*, .venv
```

These exclusions prevent linting external dependencies and generated code.

**Sources:**[.flake8 1-26](https://github.com/tenstorrent/riescue/blob/07cefde3/.flake8#L1-L26)

### Ignored Error Codes

The `extend-ignore` setting suppresses specific error codes:

| Code | Description | Reason |
| --- | --- | --- |
| E203 | Whitespace before ':' | Conflicts with Black formatting |
| F401 | Module imported but unused | Allows intentional imports |
| F841 | Local variable assigned but never used | Allows intentional assignments |
| E731 | Do not assign a lambda expression | Allows lambda assignments |

**Sources:**[.flake8 21-25](https://github.com/tenstorrent/riescue/blob/07cefde3/.flake8#L21-L25)

### Local Usage

This command:

2.   Checks all Python files in the current directory
3.   Respects exclusion rules
4.   Outputs results to console and `flake8.log`

## Pyright Type Checker

### Purpose

Pyright performs static type checking on Python code to:

*   Catch undefined variables and incorrect type usage before runtime
*   Enforce statically-typed style for better code readability
*   Enable IDE features like "jump to definition" and autocomplete
*   Prevent type-related runtime errors during longer test runs

Pyright was selected for RiESCUE because it powers the [Pylance](https://marketplace.visualstudio.com/items?itemName=ms-python.vscode-pylance) VS Code extension, providing consistent type checking between local development and CI.

**Sources:**[docs/lint.md 50-53](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L50-L53)

### Configuration

Pyright configuration is stored in `pyproject.toml` under the `[tool.pyright]` section (not in `pyrightconfig.json`, which is the default). This allows version control of type checking settings alongside other project metadata.

The configuration specifies which directories should pass static type checking. To view current settings, see [pyproject.toml](https://github.com/tenstorrent/riescue/blob/07cefde3/pyproject.toml) (search for `[tool.pyright]`).

**Sources:**[docs/lint.md 55-64](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L55-L64)

### Dependency

RiESCUE uses the [pyright-python](https://github.com/tenstorrent/riescue/blob/07cefde3/pyright-python) package, which allows calling pyright from Python:

The `-p pyproject.toml` argument explicitly points to the configuration file.

**Sources:**[docs/lint.md 55-63](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L55-L63)

### Python Code Requirements

To pass pyright checks, Python code must:

*   Use Python 3.9 syntax and features
*   Include type hints where possible
*   Avoid global variables
*   Use absolute imports in `riescue/` source code
*   Handle errors gracefully with appropriate exceptions

**Sources:**[.github/CONTRIBUTING.md 54-58](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L54-L58)

## Pre-commit Hook Integration

### Overview

Pre-commit hooks automate code quality checks before commits are created, catching issues early in the development process. RiESCUE uses the [pre-commit framework](https://git-scm.com/book/ms/v2/Customizing-Git-Git-Hooks) to manage git hooks.

**Sources:**[docs/lint.md 66-86](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L66-L86)

### Installation

Install pre-commit as a tool and activate it for the repository:

This reads `.pre-commit-config.yaml` and sets up hooks to automatically run `black` on Python files before each commit.

**Sources:**[docs/lint.md 79-85](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L79-L85)

### Local vs. Repository Hooks

The default pre-commit configuration pulls Black from its GitHub repository for reproducibility. However, you can configure it to use a locally installed version of Black (installed via pip) if preferred.

**Note:** Using local hooks relies on your local environment and won't offer the same isolation or reproducibility as the default configuration.

**Sources:**[docs/lint.md 72-77](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L72-L77)

### Black in CI: Check Mode

In CI, Black runs with the `--check` flag:

This verifies that code is properly formatted but does **not** modify files. If formatting issues are found:

*   The command exits with a non-zero status
*   The workflow fails
*   The PR is blocked from merging

Developers must run `black .` locally to fix formatting, then commit and push the changes.

**Sources:**[docs/lint.md 30-35](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L30-L35)

Any style violations cause the workflow to fail, blocking the PR.

## Code Quality Requirements for PRs

Pull requests must satisfy all linting requirements to be merged. According to the contribution standards:

*   PRs must pass all CI checks (lint and tests)
*   Code must follow Sphinx docstring format
*   Type hints should be used where possible
*   Absolute imports must be used in `riescue/` source code
*   Errors must be handled gracefully with appropriate exceptions

**Sources:**[.github/CONTRIBUTING.md 36-58](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L36-L58)

To avoid CI failures, always run linting locally before pushing:

Alternatively, set up pre-commit hooks to automate Black formatting on every commit.

## Documentation Standards

When adding or modifying Python code, follow these documentation requirements:

*   Use [Sphinx Docstring Format](https://sphinx-rtd-tutorial.readthedocs.io/en/latest/docstrings.html)
*   Add docstrings to all new Python code
*   Include clear descriptions in docstrings
*   Document parameters, return values, and exceptions

These docstrings are automatically processed by Sphinx to generate API documentation.
