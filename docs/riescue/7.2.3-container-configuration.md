# Container Configuration

Relevant source files
*   [docs/container.md](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1)
*   [infra/singularity/container.py](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py)

This page documents the `.container_config` JSON file format used to configure the Singularity container system in RiESCUE. This configuration file controls registry settings, bind mounts, and authentication for container operations.

For information about building and managing containers, see [Container Definition and Build](https://deepwiki.com/tenstorrent/riescue/3.1.1-container-definition-and-build) and [Container Management Commands](https://deepwiki.com/tenstorrent/riescue/3.1.2-container-management-commands). For details about the Container.def file itself, see [Container Definition and Build](https://deepwiki.com/tenstorrent/riescue/3.1.1-container-definition-and-build).

## Purpose and Scope

The `.container_config` file is an optional JSON configuration file that provides persistent settings for container operations. It allows users to:

*   Configure remote registry locations for pulling pre-built containers
*   Specify authentication credentials for private registries
*   Define custom bind mounts that persist across container invocations
*   Avoid repeatedly specifying these settings via command-line arguments

The configuration is loaded by the `Container` class in [infra/singularity/container.py 40-83](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L40-L83) during initialization and affects the behavior of `container-run`, `container-pull`, and `container-push` commands.

**Sources:**[infra/singularity/container.py 13-37](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L13-L37)[docs/container.md 1-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L1-L26)

## Configuration File Search Order

The `Container` class searches for `.container_config` in three locations, in the following priority order:

**Configuration File Search Algorithm**

| Priority | Location | Description |
| --- | --- | --- |
| 1 | `pwd/.container_config` | Current working directory |
| 2 | `{repo_root}/.container_config` | Repository root |
| 3 | `{repo_root}/infra/.container_config` | Infra directory |

If no configuration file is found, the `Container` class operates with default settings and assumes a local `.sif` file is present.

**Sources:**[infra/singularity/container.py 273-289](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L273-L289)[infra/singularity/container.py 48](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L48-L48)

## Configuration Schema

The `.container_config` file uses JSON format with the following structure:

### Field Definitions

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `registry_uri` | string | No | Full URI to the container registry (e.g., `oras://registry.example.com/path/riescue`) |
| `registry_remote` | string | No | Name of the Singularity remote endpoint |
| `remote_token_path` | string | No | Path to authentication token file for registry access |
| `binds` | string | No | Comma-separated list of directory binds in format `source:target` |

**Sources:**[infra/singularity/container.py 52-68](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L52-L68)[docs/container.md 16-26](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/container.md?plain=1#L16-L26)

### Registry Configuration

Registry configuration enables pulling pre-built containers from remote repositories instead of building locally.

#### registry_uri

The `registry_uri` specifies the complete URI to the container image in the registry. It typically uses the `oras://` protocol for OCI-compatible registries:

This URI is used in pull and push operations: [infra/singularity/container.py 186](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L186-L186) and [infra/singularity/container.py 176](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L176-L176)

#### registry_remote

The `registry_remote` specifies the name of the Singularity remote endpoint. This must match a remote name configured via `singularity remote login`:

The `Container` class checks if this remote exists and attempts to add it if missing: [infra/singularity/container.py 259-271](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L259-L271)

#### Authentication with remote_token_path

For private registries, authentication can be configured by specifying a path to a token file:

## Bind Mount Configuration

Bind mounts allow directories from the host system to be accessible inside the container. The configuration supports specifying persistent bind mounts.

### Default Binds

The `Container` class has a default bind that is always included:

This default is defined in [infra/singularity/container.py 41](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L41-L41)

### Custom Binds Format

Custom binds are specified as a comma-separated string in the `binds` field:

**Bind Format:**

*   **Direct mirror:**`/path` - Binds host `/path` to container `/path`
*   **Custom target:**`/host/path:/container/path` - Binds host path to different container path

Custom binds are appended to the default binds: [infra/singularity/container.py 57-59](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L57-L59)

### Bind Resolution at Runtime

When the container runs, binds are validated to ensure the source path exists on the host system:

This validation occurs in [infra/singularity/container.py 148-156](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L148-L156) to prevent errors from non-existent paths.

## Configuration Absence Behavior

When no `.container_config` file is found, the `Container` class operates in local-only mode:

1.   **Build operations:** Always available, builds locally using `Container.def`
2.   **Run operations:** Uses local `riescue.sif` if it exists
3.   **Pull/Push operations:** Fail with an error message indicating missing registry configuration

This behavior is implemented in:

*   Pull: [infra/singularity/container.py 180-181](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L180-L181)
*   Run: [infra/singularity/container.py 135-137](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L135-L137)
*   Remote check: [infra/singularity/container.py 101-102](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/singularity/container.py#L101-L102)
