## Purpose and Scope

This page documents the architecture of Ethernet tiles in Tenstorrent ASICs (Wormhole B0 and Blackhole A0). Ethernet tiles provide high-speed inter-chip networking capabilities, enabling multi-chip systems to communicate via 100 Gigabit Ethernet links. Each chip contains 16 Ethernet tiles positioned at the chip edges.

For details on Tensix tiles (the primary compute tiles), see [Tensix Tile Architecture](https://deepwiki.com/tenstorrent/tt-isa-documentation/3-tensix-tile-architecture). For information on other edge tiles (PCIe, ARC, L2CPU, DRAM), see [Edge Tiles](https://deepwiki.com/tenstorrent/tt-isa-documentation/9.2-edge-tiles:-pcie-arc-and-l2cpu).

## Tile Overview

Each Ethernet tile is a specialized tile type containing:

*   **1× Baby RISC-V E core**: Single RV32IM processor for control and data movement
*   **256 KiB L1 scratchpad RAM**: Significantly smaller than Tensix tiles (1464 KiB)
*   **Ethernet MAC/PCS subsystems**: 100 Gigabit Ethernet physical layer
*   **TX/RX queue hardware**: Hardware-assisted packet transmission and reception
*   **2× NoC connections**: Integration with both NoC #0 and NoC #1
*   **PIC (Programmable Interrupt Controller)**: IRQ management and clock gating

Unlike Tensix tiles, Ethernet tiles do **not** contain:

*   Tensix Coprocessor (Matrix Unit, Vector Unit, Unpackers, Packers)
*   Multiple RISC-V cores (only one core)
*   TDMA-RISC or Mover units
*   Mailboxes or PCBufs (inter-core communication mechanisms)

## RISC-V E Core Architecture

### Core Characteristics

The RISC-V E core in each Ethernet tile is a "Baby" RISC-V core—a 32-bit in-order single-issue processor optimized for area and power efficiency rather than high performance. It implements the RV32IM instruction set (base integer + multiplication/division extensions) and targets 1 instruction per cycle execution at 1 GHz clock speed.

Key characteristics:

*   **ISA**: RV32IM with two caveats: 
    *   `fence` instruction executes as `nop` (cannot enforce memory ordering)
    *   `ebreak` and `ecall` trigger debug pause instead of trap

*   **Pipeline**: 4-stage in-order pipeline (Frontend → Integer Unit → Load/Store Unit → Retire Unit)
*   **No exceptions/faults**: Unaligned accesses round down, invalid instructions cause undefined behavior
*   **Local storage**: 4 KiB local data RAM (2-cycle latency) + 16 KiB instruction RAM

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 1-26](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L1-L26)[WormholeB0/TensixTile/BabyRISCV/README.md 15-27](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L15-L27)

### Pipeline Organization

All instructions flow through all four stages in order, spending at least one cycle in each stage. The Load/Store Unit is the exception: instructions enter and exit in order, but can reorder internally. The unit can hold up to 8 instructions simultaneously (the oldest non-retired load plus up to 7 subsequent instructions).

**Integer Unit latencies:**

*   Most instructions: 1 cycle
*   Multiply (`mul`, `mulh`, `mulhsu`, `mulhu`): 2 cycles
*   Divide (`div`, `divu`, `rem`, `remu`): 2-33 cycles (data-dependent)
*   Mispredicted branches: 3 cycles effective (1 cycle + 2 bubble cycles)

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 26-60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L26-L60)[WormholeB0/TensixTile/BabyRISCV/README.md 28-75](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L28-L75)

### Memory Access Performance

| Memory Region | Load Latency | Max Loads in Flight | Store Throughput |
| --- | --- | --- | --- |
| [Local data RAM](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Local%20data%20RAM) | 2 cycles | 8 | 1/cycle |
| [L1 scratchpad](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/L1%20scratchpad) | ≥7 cycles | 4 | 1/5 cycles |
| Other registers/devices | ≥7 cycles | 4 | 1/cycle (typically) |

A load latency of N cycles means N-1 independent instructions are needed to fully hide the latency. Local RAM's 2-cycle latency means a single independent instruction is sufficient, making it ideal for the call stack and frequently-accessed variables.

Load throughput:

*   For latency < 5 cycles: 1 load/cycle sustained
*   For latency ≥ N cycles: 4 loads every (N-1) cycles sustained

## Memory Hierarchy

### L1 Scratchpad RAM

Each Ethernet tile contains **256 KiB** of L1 scratchpad RAM, organized as a shared memory resource accessible by:

*   RISC-V E core via load/store instructions
*   NoC #0 and NoC #1 for inter-tile communication
*   Ethernet subsystem for direct packet writes

The L1 address range is `MEM_ETH_BASE` (`0x0000_0000` to `0x0003_FFFF`). All accesses must be 16-byte aligned for optimal performance. Bank conflicts and port conflicts can increase access latency beyond the base 7 cycles.

**Ethernet packet access:**

*   **Raw writes**: Direct packet payload to L1
*   **TT-link L1 writes**: Structured writes via TT-link protocol

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 68-69](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L68-L69)

### Local Data RAM

The RISC-V E core has **4 KiB** of fast local data RAM at address `MEM_LOCAL_BASE` (`0xFFB0_0000` to `0xFFB0_0FFF`). This RAM is:

*   **Private**: Only accessible by the local RISC-V E core (not visible to NoC or Ethernet)
*   **Fast**: 2-cycle load latency with no contention
*   **Partitioned**: 
    *   First 1.5 KiB: Reserved for Tenstorrent firmware
    *   Remaining 2.5 KiB: Available for customer code

The customer code call stack is initially configured to use the top of the 2.5 KiB customer region. Remaining space can be used for thread-local variables and frequently-accessed data structures.

**Performance advantage:** Local RAM requires only 1 independent instruction to hide load latency, compared to 6 independent instructions for L1 access.

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 88-90](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L88-L90)

### Instruction Memory

The RISC-V E core fetches instructions from two possible sources:

1.   **Instruction Cache**: Small cache (details not specified) between core and L1
2.   **Instruction RAM (IRAM)**: 16 KiB dedicated instruction memory at `ERISC_IRAM_BASE` (`0xFFC0_0000` to `0xFFC0_FFFF`)

The IRAM is **not accessible** via load/store instructions; it must be populated via external means (e.g., NoC writes during initialization). Instructions cannot execute from any other memory regions (e.g., not from L1 data space or device registers).

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 32-37](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L32-L37)[WormholeB0/EthernetTile/BabyRISCV/README.md 81](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L81-L81)

## Ethernet Transmit and Receive Subsystems

The Ethernet tile provides hardware-accelerated packet transmission and reception through dedicated TX and RX queues, MAC (Media Access Control), and PCS (Physical Coding Sublayer) components.

### Hardware Components

### Packet Types

The Ethernet subsystem supports multiple packet types with different behaviors:

| Packet Type | Description | Destination |
| --- | --- | --- |
| **Raw writes** | Direct payload writes | L1 RAM at packet-specified address |
| **TT-link L1 writes** | Structured L1 writes | L1 RAM via TT-link protocol |
| **TT-link MMIO writes** | Register writes | Control registers (NoC, PIC, etc.) |

TT-link MMIO writes can target various control registers, enabling remote configuration of:

*   Tile control/debug/status registers (`0xFFB1_2000`)
*   PIC configuration (`0xFFB1_3000`)
*   NoC interface registers (`0xFFB2_0000`, `0xFFB3_0000`)
*   NoC overlay configuration (`0xFFB4_0000`)
*   Ethernet TX/RX queue registers (`0xFFB9_0000`, `0xFFB9_2000`)
*   MAC/PCS registers (`0xFFBA_0000`, `0xFFBB_0000`)

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 66-80](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L66-L80)

### Link Training and Management

The Tenstorrent firmware running on RISC-V E is responsible for:

*   **Initial configuration**: Setting up TX/RX queues and associating them appropriately
*   **Link training**: Establishing and calibrating the Ethernet physical link
*   **Link monitoring**: Detecting link drops and errors
*   **Retraining**: Automatically retraining the link in response to failures

This firmware layer operates cooperatively with customer code, handling the low-level physical layer management while allowing customer code to perform application-specific data movement.

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 9-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L9-L13)

## Firmware Cooperation Model

The RISC-V E core is **cooperatively shared** between Tenstorrent firmware and customer code. This is a key difference from Tensix tiles, where all five RISC-V cores run exclusively customer code.

### Responsibility Division

### Tenstorrent Firmware Responsibilities

The resident firmware handles:

1.   **Link initialization and training**

    *   Configuring TX/RX queue associations
    *   Physical layer training and calibration
    *   Link establishment protocols

2.   **Link maintenance**

    *   Monitoring link health
    *   Detecting link drops or errors
    *   Automatic retraining on failure

3.   **Baseline data movement service**

    *   Host-initiated data transfers
    *   System-level communication primitives
    *   NoC coordination for Ethernet-to-tile transfers

4.   **Customer code invocation**

    *   Calling into customer-provided entry points
    *   Providing context and state information
    *   Managing execution time allocation

### Customer Code Responsibilities

Customer code runs within the context established by Tenstorrent firmware and is responsible for:

*   Application-specific Ethernet operations
*   High-level protocol implementation
*   Custom packet processing
*   Coordination with customer code on other tiles

Customer code must be designed to:

*   **Return control**: Periodically return to firmware to allow link monitoring
*   **Respect memory partitioning**: Use only the 2.5 KiB customer region of local RAM
*   **Coordinate with firmware**: Not interfere with firmware's link management duties

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 5-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L5-L13)

## Memory Map

The RISC-V E core's address space is organized as follows:

### Detailed Memory Map Table

| Name and Address Range | RISC-V E Access | NoC Access | Ethernet Access |
| --- | --- | --- | --- |
| `MEM_ETH_BASE` `0x0000_0000` to `0x0003_FFFF` | L1 scratchpad RAM (256 KiB) | Yes | Raw writes TT-link L1 writes |
| `MEM_LOCAL_BASE` `0xFFB0_0000` to `0xFFB0_0FFF` | Local data RAM (4 KiB) | Unmapped | Unmapped |
| `RISCV_DEBUG_REGS_START_ADDR` `0xFFB1_2000` to `0xFFB1_2FFF` | Tile control/debug/status | Yes | TT-link MMIO writes |
| `0xFFB1_3000` to `0xFFB1_3FFF` | PIC configuration and status | Yes | TT-link MMIO writes |
| `NOC0_REGS_START_ADDR` `0xFFB2_0000` to `0xFFB2_FFFF` | NoC #0 configuration and command | Yes | TT-link MMIO writes |
| `NOC1_REGS_START_ADDR` `0xFFB3_0000` to `0xFFB3_FFFF` | NoC #1 configuration and command | Yes | TT-link MMIO writes |
| `NOC_OVERLAY_START_ADDR` `0xFFB4_0000` to `0xFFB7_FFFF` | NoC overlay configuration and command | Yes | TT-link MMIO writes |
| `ETH_TXQ0_REGS_START` `0xFFB9_0000` to `0xFFB9_1FFF` | Ethernet TX queues config and command | Yes | TT-link MMIO writes |
| `ETH_RXQ0_REGS_START` `0xFFB9_2000` to `0xFFB9_3FFF` | Ethernet RX queues configuration | Yes | TT-link MMIO writes |
| `ETH_CTRL_REGS_START` `0xFFB9_4000` to `0xFFB9_4053` | Additional tile control/status | Yes | TT-link MMIO writes |
| `0xFFBA_0000` to `0xFFBA_FFFF` | Ethernet MAC registers | Yes | TT-link MMIO writes |
| `0xFFBB_0000` to `0xFFBB_FFFF` | Ethernet PCS registers | Yes | TT-link MMIO writes |
| `ERISC_IRAM_BASE` `0xFFC0_0000` to `0xFFC0_FFFF` | Local IRAM (not LSU accessible) | Unmapped | Unmapped |

**Key observations:**

*   Most control registers are accessible via NoC and Ethernet TT-link MMIO writes, enabling remote configuration
*   Only L1 RAM and local data RAM store actual data
*   IRAM can only be written externally (e.g., during boot via NoC) and read by instruction fetch

## NoC Integration

Ethernet tiles connect to both NoC #0 and NoC #1 via Network Interface Units (NIUs), enabling:

*   Inter-tile communication
*   Access to DRAM tiles
*   Host communication via PCIe tile routing
*   Coordination with Tensix tiles for compute workloads

### NIU Configuration

Each Ethernet tile has two NIUs:

*   **NIU #0** at `NOC0_REGS_START_ADDR` (`0xFFB2_0000` to `0xFFB2_FFFF`)
*   **NIU #1** at `NOC1_REGS_START_ADDR` (`0xFFB3_0000` to `0xFFB3_FFFF`)

Each NIU provides:

*   4 request initiators for issuing NoC transactions
*   Memory-mapped command interface for programming transactions
*   Direct connection to L1 RAM for DMA operations

See [NoC Memory Map and NIU Interface](https://deepwiki.com/tenstorrent/tt-isa-documentation/2.1-noc-memory-map-and-niu-interface) for detailed NIU register descriptions.

**Sources:**[WormholeB0/EthernetTile/BabyRISCV/README.md 73-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L73-L74)

## Programmable Interrupt Controller (PIC)

The PIC in Ethernet tiles provides IRQ management and clock gating functionality. It is identical to the PIC in Tensix tiles except for:

*   Absence of debug timestamper IRQs (IRQ 17-18)
*   Clock gating applies to RISC-V E instead of RISC-V NC

### IRQ Sources

The 11 IRQ sources include:

*   **IRQ 0-3**: NoC Overlay streams 0-3 (phase start/end)
*   **IRQ 4-7**: NoC Overlay streams 8-11 (phase start/end)
*   **IRQ 16**: NoC Overlay blob auto-configuration complete

### Clock Gating

When enabled via `PIC_CLKGT_EN.GATING_ENABLED`, the RISC-V E core's clock can be automatically stopped when no IRQs are pending, reducing power consumption during idle periods. The clock will restart when:

*   Any IRQ becomes pending
*   Any agent accesses PIC registers
*   `PIC_CLKGT_EN.FORCE_ACTIVE` is set to `true`

The `PIC_CLKGT_EN.HYST` register controls how many cycles the clock remains active after an event.

**Note:** There is no actual interrupt delivery mechanism; software must poll `PIC_STATUS` to check for pending IRQs.

See [Programmable Interrupt Controller (PIC)](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.4-programmable-interrupt-controller-(pic)) for complete PIC documentation.

## Comparison with Tensix Tiles

| Feature | Tensix Tile | Ethernet Tile |
| --- | --- | --- |
| **RISC-V cores** | 5 (B, T0, T1, T2, NC) | 1 (E) |
| **L1 RAM** | 1464 KiB | 256 KiB |
| **Local data RAM** | 4 KiB (B, NC) or 2 KiB (T0-T2) per core | 4 KiB (1.5 KiB reserved) |
| **Instruction RAM** | 16 KiB (NC only) | 16 KiB (E) |
| **Coprocessor** | Tensix (Matrix, Vector, Scalar Units) | None |
| **Data movement** | Unpackers, Packers, Mover, TDMA-RISC | TX/RX queues, MAC/PCS |
| **Inter-core comm** | Mailboxes, PCBufs, TTSync | None (single core) |
| **Code ownership** | Customer exclusive | Tenstorrent firmware + customer |
| **External I/O** | NoC only | NoC + 100 GbE |
| **Primary purpose** | AI/ML compute | Inter-chip networking |
