This document provides guidance on debugging and troubleshooting SFPI (Single Floating-Point Unit Interface) code in the Tenstorrent ecosystem. It covers common compilation errors, runtime issues, and best practices for writing maintainable, debuggable SFPI code. For information about optimizing SFPI code, see [Optimization Techniques](https://deepwiki.com/tenstorrent/sfpi/7.2-writing-sfpu-kernels).

## Common Compilation Errors

SFPI code has several unique constraints that can lead to compilation errors not typically seen in standard C/C++ code. Understanding these errors is essential for effective development.

### Register Spilling Errors

One of the most common and critical errors in SFPI programming is register spilling. The SFPU has a limited number of registers, and when you use more variables than available registers, the compiler attempts to "spill" registers to memory, which is not supported for SFPU vectors.

**Example Error:**

```
fatal error: cannot store sfpu register (register spill)
```

**Common Causes:**

1.   Too many active variables in a function
2.   Deeply nested expressions
3.   Complex control flow with many live variables

**Solutions:**

1.   Break complex functions into smaller ones
2.   Reuse variables when possible
3.   Minimize the number of live variables at any point

### SFPU Vector Memory Errors

SFPU vectors cannot be stored to or loaded from memory directly, leading to errors when code attempts to do so.

**Example Error:**

```
error: cannot write sfpu vector to memory
```

**Common Operations Causing This Error:**

1.   Attempting to use SFPU vector types in arrays
2.   Passing SFPU vectors by value to functions
3.   Returning SFPU vectors from functions without proper annotations

Sources: [tests/grayskull/sfpi/gold/warn.txt 1-14](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/gold/warn.txt#L1-L14)[tests/wormhole/sfpi/gold/warn.txt 1-14](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L1-L14)

### Function Parameter and Return Value Errors

SFPU vector types cannot be passed on the stack, requiring specific function annotations.

**Example Error:**

```
error: invalid declaration for function 'argcall', sfpu types cannot be passed on the stack (missing sfpi_inline?)
```

**Solution:** Always use the `sfpi_inline` attribute for functions that:

1.   Take SFPU vector types as parameters
2.   Return SFPU vector types
3.   Have SFPU vector types as local variables that might require spilling

Sources: [tests/grayskull/sfpi/gold/warn.txt 32-34](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/gold/warn.txt#L32-L34)[tests/wormhole/sfpi/gold/warn.txt 32-34](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L32-L34)

## Control Flow and Predication Issues

### Conditional Code Structure Errors

SFPI uses a specific mechanism for conditional execution that requires proper structuring.

**Example Error:**

```
error: malformed program, sfpsetcc_v outside of pushc/popc
```

**Best Practices:**

1.   Always use the v_if/v_elseif/v_else/v_endif macros for conditional execution
2.   Ensure proper nesting of conditional blocks
3.   Avoid direct use of low-level predication primitives unless absolutely necessary

Sources: [tests/grayskull/sfpi/gold/warn.txt 41-46](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/gold/warn.txt#L41-L46)[tests/wormhole/sfpi/gold/warn.txt 41-46](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L41-L46)

## Uninitialized Variable Warnings

The SFPI compiler can detect uninitialized variables, which is particularly important since using uninitialized SFPU registers can lead to unpredictable results.

**Example Warning:**

```
warning: 'x' is used uninitialized in this function [-Wuninitialized]
```

**Prevention Strategies:**

1.   Initialize all variables at declaration
2.   Use conditional code to ensure all execution paths initialize variables before use
3.   Enable and pay attention to compiler warnings

Sources: [tests/wormhole/sfpi/gold/warn.txt 22-32](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L22-L32)[tests/wormhole/sfpi/gold/warn.txt 45-72](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt#L45-L72)

## Common Error Patterns and Solutions

The following table summarizes common errors and their solutions:

| Error/Warning | Cause | Solution |
| --- | --- | --- |
| Register spill | Too many live variables | Reduce variable count, break function into smaller parts |
| Cannot write vector to memory | Storing SFPU vector to memory | Use `dst_reg` for storage or redesign algorithm |
| Types cannot be passed on stack | Missing function attribute | Add `sfpi_inline` attribute to function |
| Malformed program | Incorrect predication structure | Use v_if/v_endif macros correctly |
| Uninitialized variable | Missing initialization | Initialize variables at declaration |
| Illegal builtin in conditional | Unsupported operation in condition | Simplify condition or use temporary variables |

### Register Pressure Management

Register pressure is one of the most critical aspects of SFPI programming. Understanding how to manage it can prevent many errors.

**Effective Strategies:**

1.   Limit the scope of variables to minimum required
2.   Reuse variables for intermediate calculations
3.   Break complex expressions into simpler steps with explicit temporary variables
4.   Consider the register allocation impact of nested conditions
5.   Use conditional execution to reduce branch complexity

## Architecture-Specific Considerations

Different Tenstorrent architectures (Grayskull, Wormhole, Blackhole) have slightly different constraints and error behaviors.

**Key Differences:**

1.   Register count and allocation strategies may vary
2.   Certain operations may have different performance characteristics
3.   Error messages may differ slightly in format and detail
4.   Some instructions may be available on some architectures but not others

Sources: [tests/wormhole/sfpi/gold/warn.txt](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/wormhole/sfpi/gold/warn.txt)[tests/grayskull/sfpi/gold/warn.txt](https://github.com/tenstorrent/sfpi/blob/ed989b9d/tests/grayskull/sfpi/gold/warn.txt)

## Best Practices Summary

To minimize debugging issues with SFPI code, follow these best practices:

1.   **Initialization**: Always initialize variables before use
2.   **Function Structure**: Use `sfpi_inline` for functions with SFPU types
3.   **Control Flow**: Use v_if/v_endif macros correctly for conditional execution
4.   **Register Management**: Minimize live variable count to avoid register spills
5.   **Debugging**: Use dst_reg strategically to observe intermediate values
6.   **Testing**: Create isolated test cases for specific functionality
7.   **Compiler Warnings**: Pay attention to all compiler warnings, especially uninitialized variable warnings

By understanding common errors and following these best practices, you can develop SFPI code more efficiently and with fewer debugging challenges.
