## Purpose and Scope

The tt-llk repository implements a three-stage pipeline architecture for processing tensor operations on Tenstorrent Tensix cores. Each stage executes on a dedicated RISC-V thread (TRISC) and operates on separate register banks, enabling concurrent execution:

| Stage | TRISC Thread | Function | Register Banks |
| --- | --- | --- | --- |
| **Unpacker** | TRISC0 | Load tiles from L1 → SrcA/SrcB registers | Reads L1, writes SrcA/SrcB |
| **Math** | TRISC1 | Execute operations: SrcA/SrcB → Dest | Reads SrcA/SrcB, writes Dest |
| **Packer** | TRISC2 | Write tiles from Dest → L1 | Reads Dest, writes L1 |

The pipeline uses hardware semaphores (`semaphore::UNPACK_SYNC`, `semaphore::MATH_PACK`, `semaphore::UNPACK_TO_DEST`) and double-buffered configurations (`unp_cfg_context`, `dest_offset_id`) to overlap computation with data movement.

For detailed operation implementations:

*   Unpacker: [Unpacker System](https://deepwiki.com/tenstorrent/tt-llk/4-unpacker-system) - Functions like `_llk_unpack_A_()`, `_llk_unpack_AB_()`
*   Math: [Math Unit Operations](https://deepwiki.com/tenstorrent/tt-llk/5-math-unit-operations) - Functions like `_llk_math_matmul_()`, `_llk_math_eltwise_binary_()`
*   Packer: [Packer System](https://deepwiki.com/tenstorrent/tt-llk/6-packer-system) - Functions like `_llk_pack_()`, `program_packer_destination()`

### Unpacker Execution Flow

The unpacker operates through a context-switching mechanism that enables pipelined operation:

1.   **Wait for available context**: `wait_for_next_context(2)` ensures a configuration slot is available
2.   **Configure base address**: Write tile address to `THCON_SEC0_REG3_Base_address_ADDR32` or `THCON_SEC1_REG3_Base_address_ADDR32`
3.   **Post semaphore**: `semaphore_post(semaphore::UNPACK_SYNC)` signals configuration is ready
4.   **Execute MOP**: `ckernel_template::run()` executes the micro-operation program
5.   **Release context**: `t6_semaphore_get(semaphore::UNPACK_SYNC)` frees the context
6.   **Switch context**: `switch_config_context(unp_cfg_context)` alternates between contexts 0 and 1

### Address Modifiers

Address modifiers control how register counters increment during math operations. Each address modifier specifies increments for SrcA, SrcB, Dest, and fidelity counters:

| Address Modifier | Typical Usage | Key Increments |
| --- | --- | --- |
| `ADDR_MOD_0` | Inner loop processing | `srca.incr`, `srcb.incr`, `dest.incr` |
| `ADDR_MOD_1` | Outer loop boundaries | `srca.cr`, `srcb.cr` (carry/reset) |
| `ADDR_MOD_2` | Face/tile boundaries | `dest.cr`, larger increments |
| `ADDR_MOD_3` | Fidelity phase 1 | `fidelity.incr = 1` |
| `ADDR_MOD_5` | Fidelity phase reset | `fidelity.incr`, clears counters |
| `ADDR_MOD_6` | Full reset | `fidelity.clr = 1` |

### Packer L1 Address Calculation

Each packer handles a specific face of the tile, requiring careful address offset calculation:

| Packer | Register Base | Face Offset | Calculation |
| --- | --- | --- | --- |
| Packer 0 | `THCON_SEC0_REG1` | 0 | Base address |
| Packer 1 | `THCON_SEC0_REG8` | `l1_offset_1` | `(face_dim/16) * datum_size - PACK_TILE_HEADER_OFFSET` |
| Packer 2 | `THCON_SEC1_REG1` | `l1_offset_2` | `2 * l1_offset_1` |
| Packer 3 | `THCON_SEC1_REG8` | `l1_offset_3` | `3 * l1_offset_1` |

The `PACK_TILE_HEADER_OFFSET = 1` (in 16B units) must be subtracted because hardware automatically adds the tile header size.

## Synchronization Mechanisms

The three-stage pipeline uses hardware semaphores to coordinate data flow between stages. Each semaphore is accessed through specific functions:

### Semaphore Usage Patterns

| Semaphore | Producer | Consumer | Producer Functions | Consumer Functions | Purpose |
| --- | --- | --- | --- | --- | --- |
| `UNPACK_SYNC` | Unpacker | Unpacker | `semaphore_post()` | `t6_semaphore_get()`, `wait_for_next_context()` | Configuration context availability (2 contexts) |
| `MATH_PACK` | Math | Packer | `TTI_SEMINIT` in `set_math_semaphores()` | `TTI_SEMWAIT` in `_llk_packer_wait_for_math_done_()` | Dest register data ready signal |
| `UNPACK_TO_DEST` | Unpacker | Math | `TTI_SEMPOST` | `TTI_SEMWAIT` in `math_unpack_to_dest_math_ready()` | 32-bit unpack-to-dest coordination |

## Register Banks and Capacity

The three register banks have different capacities depending on the data format and accumulation mode:

### Register Bank Capacities

| Register | 16-bit Mode | 32-bit Mode | FP32 Dest Accumulation |
| --- | --- | --- | --- |
| SrcA | 16 tiles | 8 tiles | N/A |
| SrcB | 16 tiles | 8 tiles | N/A |
| Dest | 16 tiles | 8 tiles | 8 tiles (16-bit format) 4 tiles (32-bit format) |

The `DEST_REGISTER_HALF_SIZE` constant defines the offset between halves:

*   16-bit mode: 256 rows (8 tiles × 32 rows)
*   32-bit mode: 128 rows (4 tiles × 32 rows)

## Pipelined Execution Example

**Timeline: Three Stages Operating on Multiple Tiles**

**Key Observations:**

1.   **Configuration Overlap**: Unpacker configures Tile 1 (context 1) at cycle 4 while Tile 0 (context 0) executes at cycles 2-10
2.   **Semaphore Coordination**: Math waits for `UNPACK_SYNC` semaphore before starting Tile 0 at cycle 10
3.   **Pipeline Fill**: By cycle 22, all three stages operate concurrently on different tiles (Unpacker on Tile 2, Math on Tile 1, Packer on Tile 0)
4.   **Double-Buffering**: Context switching (`switch_config_context()`) enables zero-overhead tile configuration
5.   **Dest Register Management**: `DstSync::SyncHalf` mode allows Math to write new results while Packer reads previous results from different half
