This page documents the TTIR to TTMetal backend compilation pipeline, which transforms hardware-agnostic TTIR operations into explicit device kernels through the D2M (Data-to-Metal) dialect. The D2M dialect represents operations with explicit grid assignments, memory layouts, and hardware thread mappings, providing fine-grained control over kernel execution.

The TTMetal pipeline is an alternative to the TTNN backend that offers explicit control over:

*   Memory allocation and layout (L1/DRAM placement, streaming)
*   Grid topology and sharding strategies
*   DST register management and operation scheduling
*   Hardware thread assignment (Compute, NOC0, NOC1)

For the higher-level TTNN backend with automatic memory management, see TTIR to TTNN Backend Pipeline (page 3.2). For the common frontend stages, see Frontend: StableHLO to TTIR (page 3.1).

**Key Pipeline Stages:**

1.   **Frontend**: TTIR → D2M conversion with grid selection and layout optimization.
2.   **Middleend**: Memory allocation, loop tiling, and operation scheduling.
3.   **Backend**: TTKernel code generation and EmitC translation.

### TTIR to D2M Conversion Process

The `createTTIRToD2MPass()` performs the initial lowering from TTIR to D2M operations. This conversion handles the mapping of high-level reduction and elementwise operations to device-aware D2M forms.

**Key Conversion Patterns:**

1.   **Reduction Handling**: [lib/Conversion/TTIRToD2M/TTIRToD2M.cpp 44-61](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTIRToD2M/TTIRToD2M.cpp#L44-L61)

    *   Distinguishes between "outer reductions" (touching dimensions before the last two tile dimensions) and standard tile-based reductions via `isOuterReduction`.
    *   Enforced by `checkTTIRReductionPreconditions` which requires `dim_arg` and `keep_dim=true`[lib/Conversion/TTIRToD2M/TTIRToD2M.cpp 82-87](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Conversion/TTIRToD2M/TTIRToD2M.cpp#L82-L87)

## Middleend Stage: Optimization and Scheduling

The middleend stage applies a series of optimization passes to improve performance and manage hardware resources.

### Elementwise Fusion and Scratch Buffers

1.   **D2MElementwiseFusion**: Fuses chains of compute-only generic operations to reduce intermediate tensors. It targets elementwise and elementwise->reduction fusion [include/ttmlir/Dialect/D2M/Transforms/Passes.td 57-67](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L57-L67)
2.   **D2MInsertScratchBuffers**: Post-bufferization, this pass adds scratch L1 buffers (typically 128KB) to fused generics containing multiple `linalg.generic` ops to handle intermediate spills [include/ttmlir/Dialect/D2M/Transforms/Passes.td 11-26](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L11-L26) It inserts a `memref.alloc` with `CBLayoutAttr`[include/ttmlir/Dialect/D2M/Transforms/Passes.td 42-43](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L42-L43)

The `D2MAllocate` pass performs comprehensive memory planning using a `Planner` that manages `Scratch` (L1) and `Spill` (DRAM) spaces [lib/Dialect/D2M/Transforms/Allocate.cpp 92-118](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/Transforms/Allocate.cpp#L92-L118)

**Key Allocation Algorithm Features:**

*   **Memref Value Context**: Tracks `allocSize`, `live` ranges, and `remappedMemSpace` for every allocation [lib/Dialect/D2M/Transforms/Allocate.cpp 130-162](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/Transforms/Allocate.cpp#L130-L162)
*   **Operand Context**: Links `OperandContext` with originating `Value`s through `chainRoots`[lib/Dialect/D2M/Transforms/Allocate.cpp 174-193](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/D2M/Transforms/Allocate.cpp#L174-L193)
*   **Spilling**: Generic outputs in L1 can be marked eligible for spilling to DRAM via `allowL1OutputSpilling`.
*   **Stream Insertion**: Uses policies like `always` or `infer` to decide when to insert operand streams to pipeline data movement.

### Loop Tiling and Scheduling

After bufferization, the pipeline applies a series of loop transformations to optimize hardware utilization.

1.   **D2MGenericTileComputeLoops**: Tiles compute loops according to block factors and hardware constraints. It uses `maxDstPhysicalSizeTiles` to bound `numTilesPerFlip`[include/ttmlir/Dialect/D2M/Transforms/Passes.td 119-128](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L119-L128)
2.   **D2MOpScheduler**: Uses the Sethi-Ullman algorithm to reorder operations within fused compute loops to minimize DST register pressure [include/ttmlir/Dialect/D2M/Transforms/Passes.td 80-90](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L80-L90)
3.   **D2MSFPUTileLoopFission**: Performs loop distribution by extracting `load -> compute -> store` triplets into separate loop nests to optimize SFPU utilization [include/ttmlir/Dialect/D2M/Transforms/Passes.td 104-111](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L104-L111)
4.   **Insert Spill and Scratch**: Replaces intermediate allocations with scratch buffers and updates `affine.store`/`affine.load` to use scratch indices [include/ttmlir/Dialect/D2M/Transforms/Passes.td 137-165](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/D2M/Transforms/Passes.td#L137-L165)
