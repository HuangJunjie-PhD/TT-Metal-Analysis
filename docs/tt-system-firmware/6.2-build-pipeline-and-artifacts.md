This document describes the GitHub Actions-based continuous integration pipeline that builds firmware for all supported hardware platforms, generates signed binaries, and packages them into distributable artifacts. The pipeline implements a matrix build strategy to efficiently build multiple board variants in parallel.

For information about how the firmware is signed and integrated with MCUBoot, see [Firmware Signing and MCUBoot](https://deepwiki.com/tenstorrent/tt-system-firmware/6.3-firmware-signing-and-mcuboot). For details on the final combined bundle format, see [Combined Firmware Bundles](https://deepwiki.com/tenstorrent/tt-system-firmware/6.4-combined-firmware-bundles). For the underlying Zephyr integration, see [Zephyr Integration](https://deepwiki.com/tenstorrent/tt-system-firmware/6.1-zephyr-integration).

**Sources:**[.github/workflows/build-fw.yml 1-523](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L1-L523)

## Pipeline Architecture

The build pipeline consists of multiple parallel and sequential jobs triggered by code changes, pull requests, or manual dispatch. The primary workflow is defined in `build-fw.yml` and coordinates several specialized build jobs.

### Workflow Triggers

The pipeline activates on:

*   Pushes to `main`, `v*-branch`, or `collab*` branches
*   Pull request events (opened, reopened, synchronize)
*   Merge group check requests
*   Manual workflow dispatch

Concurrency control ensures only one pipeline runs per PR or commit SHA, canceling in-progress runs when new commits arrive.

## Board Matrix Generation

The `genboards` job reads the board configuration matrix from `.github/boards.json` and exposes it as a JSON output that downstream jobs consume.

The job uses `jq` to compact the JSON configuration and pipes it to `GITHUB_OUTPUT` for consumption by matrix strategies.

## Main Firmware Build Job

The `build` job is the primary firmware builder, executing in parallel for each board variant defined in the matrix. Each instance builds both SMC and DMC firmware with optional signature verification.

### Build Process Flow

**Sources:**[.github/workflows/build-fw.yml 46-139](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L46-L139)

### Board Identifier Mapping

The build process converts board revision identifiers from `boards.json` into Zephyr-compatible board strings using `scripts/rev2board.sh`:

```
p100a → tt_blackhole@p100a/tt_blackhole/smc
p150a → tt_blackhole@p150a/tt_blackhole/smc
p300a → tt_blackhole@p300a/tt_blackhole/smc
```

**Sources:**[.github/workflows/build-fw.yml 70-75](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L70-L75)

### Build Configuration

| Parameter | Value | Description |
| --- | --- | --- |
| Build Directory | `build-${{matrix.board}}` | Isolated build output per board |
| Build System | `--sysbuild` | Enables multi-application builds (SMC+DMC+recovery) |
| Pristine Build | `-p` | Clean build from scratch |
| Board Target | `-b $BOARD` | Zephyr board identifier |
| App Path | `tt-system-firmware/app/smc` | SMC app as primary (pulls in DMC via sysbuild) |
| Warnings | `-DCONFIG_COMPILER_WARNINGS_AS_ERRORS=y` | Treat warnings as errors |

**Sources:**[.github/workflows/build-fw.yml 86-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L86-L106)

## Build Artifacts Structure

Each build job uploads artifacts containing firmware binaries and build metadata. The artifact structure follows a consistent pattern across all jobs.

### Main Firmware Artifacts

For each board variant, the `build` job uploads artifacts named `firmware-${{matrix.board}}`:

| File Pattern | Description |
| --- | --- |
| `build-*/update.fwbundle` | Combined firmware bundle for this board |
| `build-**/.config` | Kconfig configuration files |
| `build-**/devicetree_generated.h` | Generated device tree headers |
| `build-**/zephyr.bin` | Raw binary images |
| `build-**/zephyr.dts` | Device tree source files |
| `build-**/zephyr.elf` | ELF executables with debug symbols |
| `build-**/zephyr.hex` | Intel HEX format images |
| `build-**/zephyr.map` | Linker map files |
| `build-**/zephyr.signed.bin` | Signed binary images for MCUBoot |
| `build-**/zephyr.signed.hex` | Signed images in HEX format |
| `build-**/zephyr.stat` | Binary statistics |

**Sources:**[.github/workflows/build-fw.yml 123-138](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L123-L138)

### Artifact Path Resolution

The wildcard pattern `build-${{matrix.board}}/**/` captures outputs from multiple sub-applications:

```
build-p100a/
├── dmc/zephyr/          # DMC firmware outputs
├── smc/zephyr/          # SMC firmware outputs
├── recovery/zephyr/     # Recovery firmware outputs
└── update.fwbundle      # Combined bundle
```

**Sources:**[.github/workflows/build-fw.yml 127-138](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L127-L138)

## Recovery Firmware Build

The `build-recovery` job creates a standalone recovery firmware image used for emergency system recovery when normal firmware cannot boot.

### Recovery Build Configuration

The recovery build uses a special configuration overlay:

| Parameter | Value |
| --- | --- |
| Board | `tt_blackhole@p100a/tt_blackhole/smc` |
| Extra Config | `-DEXTRA_CONF_FILE=sysbuild/recovery.conf` |
| Unsigned Image | `-DCONFIG_MCUBOOT_GENERATE_UNSIGNED_IMAGE=n` |
| Signature Key | `-DCONFIG_MCUBOOT_SIGNATURE_KEY_FILE` |

The build targets only the P100A variant as recovery firmware is board-agnostic at the hardware level.

**Sources:**[.github/workflows/build-fw.yml 140-203](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L140-L203)

## Assembly Test Firmware

The `build-assembly-test` job builds specialized firmware for hardware assembly and validation testing. These builds run on P150 and P300 boards with manufacturing test configurations.

### Assembly Test Configuration

The assembly test builds:

*   Enable `CONFIG_TT_ASSEMBLY_TEST=y` for manufacturing test modes
*   Use RTT console (`-S rtt-console`) for real-time debugging
*   Build only DMC firmware (`app/dmc`)
*   Generate separate MCUBoot bootloader artifacts
*   Use sanitized branch names in artifact naming

**Sources:**[.github/workflows/build-fw.yml 204-326](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L204-L326)

## Combined Firmware Bundle

The `build-combined-fwbundle` job depends on the completion of all matrix `build` jobs and combines their artifacts into a single distributable bundle.

The job downloads all board-specific firmware artifacts, validates them using `pykwalify` schema validation, and packages them using `build-combined-fwbundle.sh`. The resulting bundle is uploaded with a predictable artifact name that downstream workflows can reference.

**Sources:**[.github/workflows/build-fw.yml 349-381](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L349-L381)

## Version Management

Firmware versions are managed through three separate `VERSION` files that define a hierarchical versioning scheme.

### Version Files

| File | Component | Current Version | Description |
| --- | --- | --- | --- |
| `VERSION` | Product | `19.7.99.0` | Overall firmware product version |
| `app/smc/VERSION` | SMC | `0.29.99.0` | System Management Controller version |
| `app/dmc/VERSION` | DMC | `0.23.99.0` | Device Management Controller version |

Each VERSION file contains:

```
VERSION_MAJOR = X
VERSION_MINOR = Y
PATCHLEVEL = Z
VERSION_TWEAK = W
EXTRAVERSION =
```

The build system reads these files during CMake configuration to embed version information in the compiled firmware binaries. Git describe tags are also incorporated when available to track exact commit information.

**Sources:**[VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/VERSION#L1-L6)[app/smc/VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/VERSION#L1-L6)[app/dmc/VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/VERSION#L1-L6)

## Artifact Naming Conventions

The pipeline uses consistent naming patterns for artifacts:

| Pattern | Example | Description |
| --- | --- | --- |
| `firmware-{board}` | `firmware-p100a` | Per-board firmware outputs |
| `recovery` | `recovery` | Recovery firmware (board-agnostic) |
| `assembly-fw-{board}` | `assembly-fw-p150` | Assembly test firmware |
| `assembly-mcuboot-{board}` | `assembly-mcuboot-p150` | Assembly test bootloader |
| `assembly-test-{board}-{branch}` | `assembly-test-p300-main` | Pre-flash test collateral |
| `combined-fwbundle` | `combined-fwbundle` | Combined multi-board bundle |
| `preflash-fw` | `preflash-fw` | Pre-flash firmware images |
| `build-smc-tests-{board}` | `build-smc-tests-p100a` | SMC test build outputs |

Sanitized branch names replace spaces and slashes with underscores for filesystem compatibility.

## Build Environment

All build jobs execute on `ubuntu-24.04` runners with the following environment setup:

1.   **Repository Checkout**: Uses `actions/checkout@v4` with configurable paths
2.   **Zephyr Preparation**: Custom action at `.github/workflows/prepare-zephyr` initializes west workspace
3.   **Tag Fetching**: Unshallows git repositories to enable `git describe` version tagging
4.   **Python Dependencies**: Installs `pykwalify`, `intelhex`, `imgtool` as needed
5.   **West Configuration**: Configured through `west.yml` manifest (see [Zephyr Integration](https://deepwiki.com/tenstorrent/tt-system-firmware/6.1-zephyr-integration))

The build environment is reproducible and containerizable for local development.
