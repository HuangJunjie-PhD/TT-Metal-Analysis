The `tt-toplike` codebase utilizes a structured hierarchy of data models to represent Tenstorrent hardware state, telemetry metrics, and architectural constraints. These models are designed to be backend-agnostic, supporting data ingestion from direct hardware access (Luwen), Linux `sysfs`, or `tt-smi` JSON streams.

## Hardware Representation

The core of the hardware model is the `Device` struct and the `Architecture` enum. These entities define the physical properties and capabilities of a chip before any telemetry is applied.

### Architecture Enum

The `Architecture` enum categorizes Tenstorrent hardware into three primary generations: **Grayskull**, **Wormhole**, and **Blackhole**. Each variant defines specific grid dimensions and memory configurations used by the UI layer to render chip portraits.

| Architecture | Abbreviation | Tensix Grid (R x C) | DDR Channels | Example Boards |
| --- | --- | --- | --- | --- |
| `Grayskull` | GS | 10 ֳ— 12 | 4 | e75, e150 |
| `Wormhole` | WH | 8 ֳ— 10 | 8 | n150, n300 |
| `Blackhole` | BH | 14 ֳ— 16 | 12 | p150, p300 |

### Device Struct

The `Device` struct acts as a lightweight proxy for a physical chip. It stores identifying information such as the PCI `bus_id`, `board_type`, and `index`. While it contains static metadata like `firmwares` and `limits`, it does not hold volatile telemetry data (which is handled by the `Telemetry` struct).

## Telemetry Models

Telemetry is split into high-level metrics (real-time power/temp) and low-level SMBUS data (firmware versions/detailed health).

### Telemetry Struct

The `Telemetry` struct captures the primary "heartbeat" of the device. All fields are wrapped in `Option<T>` to ensure the application remains resilient if a specific sensor (like voltage or current) fails to report.

*   **`voltage`**: Supply voltage in Volts.
*   **`current`**: Current draw in Amperes.
*   **`power`**: Power consumption in Watts.
*   **`asic_temperature`**: Die temperature in Celsius.
*   **`aiclk`**: AI clock frequency in MHz.
*   **`heartbeat`**: ARC firmware status (0 = stalled, 1 = healthy).

Sources: `<FileRef file-url="https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/models/telemetry.rs#L18-L40" min=18 max=40 file-path="src/models/telemetry.rs">Hii</FileRef>`

### SmbusTelemetry Struct

The `SmbusTelemetry` struct is a comprehensive model of the System Management Bus data. It includes detailed firmware strings, clock frequencies (`axiclk`, `arcclk`), and specialized thermal monitors.

A critical component of this struct is the `gddr_temps` array, which stores `GddrTempPair` objects. These represent the temperatures of GDDR memory modules.

## Constants and Parsing Helpers

The library provides utility functions to handle architecture-specific bitmasks and data unpacking.

### GDDR Temperature Unpacking

GDDR temperatures are often stored as pairs in 32-bit registers. The `unpack_gddr_temps` helper extracts these into a flat array of floats for easier consumption by the UI.

Sources: `<FileRef file-url="https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/models/telemetry.rs#L567-L582" min=567 max=582 file-path="src/models/telemetry.rs">Hii</FileRef>`

### Harvesting and Grid Constants

To correctly render the "Chip Portrait," the models must account for "harvesting" (where specific Tensix columns are disabled during manufacturing).

*   **`BH_TENSIX_COL_COUNT`**: Defined as 16 for Blackhole.
*   **`tensix_col_harvested`**: A helper function that takes a `harvesting_mask` (u32) and a column index to determine if a specific core should be rendered as "disabled."

Sources: `<FileRef file-url="https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/models/telemetry.rs#L596-L601" min=596 max=601 file-path="src/models/telemetry.rs">Hii</FileRef>`, `<FileRef file-url="https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/models/telemetry.rs#L604-L610" min=604 max=610 file-path="src/models/telemetry.rs">Hii</FileRef>`

### Firmware and Limits

Newer versions of Tenstorrent hardware provide structured information about firmware versions and operational limits:

*   **`FirmwaresInfo`**: Contains versions for `arc`, `ethernet`, `m3_bl`, and `m3_app`.
*   **`DeviceLimits`**: Contains `tdp` (power limit) and `tdc` (current limit).
