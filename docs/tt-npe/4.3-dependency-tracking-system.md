## Purpose and Scope

The Dependency Tracking System ensures correct temporal ordering of NoC transfers during simulation. This system implements a checkpoint-based mechanism that models hardware constraints such as network interface unit (NIU) queueing behavior and multi-chip routing dependencies. The dependency tracker prevents transfers from executing before their prerequisites complete, enabling accurate performance estimation of complex workload patterns.

For information about how transfers are defined and organized, see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For details on how dependencies affect the simulation loop, see [Transfer State Lifecycle](https://deepwiki.com/tenstorrent/tt-npe/4.2-transfer-state-lifecycle).

## Checkpoint-Based Dependency Model

The dependency system uses **checkpoints** as synchronization points between transfers. Instead of creating direct transfer-to-transfer dependencies, transfers declare that they depend on checkpoints and contribute to satisfying checkpoints. This indirection enables multiple transfers to depend on the same checkpoint and a single transfer to satisfy multiple checkpoints.

### Core Components

| Component | Type | Purpose |
| --- | --- | --- |
| `npeTransferDependencyTracker` | Class | Central registry managing all checkpoints and their states |
| `npeCheckpointID` | Type alias (int) | Unique identifier for each checkpoint |
| `PETransferState.depends_on` | Field | Checkpoint this transfer waits for before activation |
| `PETransferState.required_by` | Field (vector) | Checkpoints this transfer contributes to when complete |

### Checkpoint Properties

Each checkpoint maintains:

*   **Counter**: Number of transfers that must complete before the checkpoint is satisfied (decremented as transfers complete)
*   **Delay**: Additional cycle delay after the checkpoint counter reaches zero
*   **End Cycle**: The actual cycle when the checkpoint was satisfied (set when counter reaches zero)

## NIU Ordering Dependencies

To approximate the effects of 2-virtual-channel (2-VC) hardware behavior, the system creates dependencies between transfers originating from the same network interface unit (NIU). This prevents unrealistic parallelism where a single NIU would appear to inject unlimited concurrent transfers.

### Bucketing Strategy

Transfers are grouped into buckets based on:

| Dimension | Value | Purpose |
| --- | --- | --- |
| NoC Type | `NOC0` or `NOC1` | Separate dependency chains per NoC |
| Source Row | Integer | Core location |
| Source Column | Integer | Core location |
| Link Type | `nocLinkType` or special constant | Distinguishes first-hop direction or local transfers |

Special link type constants for purely local transfers (no route):

*   `LOCAL_NOC0_TRANSFER_TYPE = 1000`
*   `LOCAL_NOC1_TRANSFER_TYPE = 2000`

**Diagram:** NIU ordering dependency buckets and stride-2 chaining

### Stride-2 Dependency Chain

Within each bucket, transfers are sorted by `start_cycle`. Dependencies are created with **stride=2**, meaning each transfer depends on the transfer that started 2 positions earlier in the sorted list. This approximates 2-VC behavior where two transfers can overlap in the network simultaneously.

**Implementation:**

```
For bucket with transfers [T0, T1, T2, T3, T4, ...] sorted by start_cycle:
  T2 depends on T0
  T3 depends on T1
  T4 depends on T2
  T5 depends on T3
  ...
```

## Transfer Group Dependencies

For multi-chip workloads, transfers are organized into **transfer groups** that model sequential routing through multiple devices. Each transfer in a group has:

*   `transfer_group_id`: Identifies the group
*   `transfer_group_index`: Position within the group
*   `transfer_group_parent`: Index of the preceding transfer in the group

These dependencies ensure that a transfer on device N+1 does not begin until the corresponding transfer on device N has completed, modeling the cascading nature of fabric routing.

### Checkpoint Delay Calculation

When a transfer depends on a parent in the same group, the checkpoint includes two delay components:

| Delay Component | Purpose | Calculation |
| --- | --- | --- |
| Write Latency | On-chip NoC propagation | Device-specific function: `get_write_latency(src, dst, noc_type)` |
| Ethernet Hop Delay | Inter-chip propagation | `600 + 0.1055 * packet_size` cycles (only if different devices) |

### Validation Step

After creating all dependencies, the system validates consistency by:

1.   Iterating all transfers and updating their required checkpoints
2.   Calling `dep_tracker.sanityCheck()` to verify all checkpoints have valid states
3.   Calling `dep_tracker.allComplete()` to ensure all counters are satisfied
4.   Resetting the tracker to initial state for simulation

## Dependency Checking During Simulation

### Transfer Activation

During each simulation timestep, the engine checks if queued transfers can activate by calling `dep_tracker.done(transfer.depends_on, curr_cycle)`. A transfer is ready if:

*   Its `depends_on` checkpoint is `UNDEFINED_CHECKPOINT`, OR
*   The checkpoint's end cycle plus delay is less than or equal to `curr_cycle`

**Diagram:** Transfer activation decision flow

### Adjusting Start Cycles

When a dependency completes after a transfer's nominal `start_cycle`, the system adjusts the transfer's start to the dependency end cycle plus delay:

```
adjusted_start = max(original_start_cycle, dep_end_cycle + dep_delay)
```

This ensures transfers do not appear to start before their dependencies complete, maintaining temporal correctness.

## Integration with Transfer State

### PETransferState Fields

The `PETransferState` structure includes dependency-related fields:

| Field | Type | Purpose |
| --- | --- | --- |
| `depends_on` | `npeCheckpointID` | Checkpoint this transfer waits for |
| `required_by` | `vector<npeCheckpointID>` | Checkpoints satisfied by this transfer |
| `start_cycle` | `CycleCount` | May be adjusted by dependency completion |
| `end_cycle` | `CycleCount` | Used to update dependent checkpoints |

### Dependency Impact on Timeline

Dependencies affect the simulation timeline by:

1.   **Delaying activation**: Transfers wait in the queue until dependencies are satisfied
2.   **Adjusting start times**: Transfers that would nominally start early are delayed
3.   **Creating sequential chains**: Related transfers execute in correct order
4.   **Modeling hardware limitations**: NIU queueing and fabric latency are represented

### Ethernet Hop Delay Model

Multi-chip transfers crossing ethernet links incur delay:

This delay is only added when `src.device_id != parent_src.device_id`, distinguishing true cross-chip hops from on-chip fabric multiplexing.

### NIU Dependency Stride

The stride parameter controls how many concurrent transfers from a single NIU can overlap in the network. Increasing this value would allow more parallelism but may reduce accuracy for hardware with limited virtual channels.
