## Purpose and Scope

This page documents the GPU code generation backends in TVM, specifically for NVIDIA GPUs (via NVPTX/PTX) and AMD GPUs (via AMDGPU/ROCm). These backends compile TensorIR (TIR) programs into GPU-executable code using LLVM's GPU code generation infrastructure.

For the general LLVM backend architecture, see [LLVM Backend Architecture](https://deepwiki.com/tenstorrent/tt-tvm/6.1-llvm-backend-architecture). For CPU-specific code generation, see [CPU Code Generation](https://deepwiki.com/tenstorrent/tt-tvm/6.2-cpu-code-generation). For other specialized targets like Hexagon DSP, see [Specialized Target Code Generation](https://deepwiki.com/tenstorrent/tt-tvm/6.4-specialized-target-code-generation).

**Scope:**

*   NVPTX code generator for CUDA-capable NVIDIA GPUs
*   AMDGPU code generator for ROCm-capable AMD GPUs
*   GPU-specific memory management (shared memory, address spaces)
*   Thread indexing and synchronization primitives
*   GPU-specific intrinsics and optimizations

## NVPTX Code Generator (CUDA)

The `CodeGenNVPTX` class generates PTX (Parallel Thread Execution) code for NVIDIA GPUs. It inherits from `CodeGenLLVM` and specializes GPU-specific operations.

### Kernel Function Declaration

NVPTX kernels are declared with specific metadata to mark them as device entry points:

**Function Declaration Process:**

| Step | Method | Purpose |
| --- | --- | --- |
| 1. Base declaration | `DeclareFunction()` | Creates LLVM function with proper signature |
| 2. Function body | `AddFunctionInternal()` | Adds TIR body compilation |
| 3. Metadata annotation | `AddFunction()` | Adds `nvvm.annotations` metadata marking kernel |

Sources: [src/target/llvm/codegen_nvptx.cc 72-85](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L72-L85)

### Thread Indexing

NVPTX provides intrinsics to access thread and block indices within the CUDA execution model:

**Thread Index Intrinsics:**

### NVPTX-Specific Optimizations

The NVPTX backend applies optimizations to remove unused NVIDIA device functions:

**Optimization Strategy:**

1.   Iterate through all functions with `__nv` prefix
2.   Mark non-inlined declarations as `AvailableExternallyLinkage`
3.   Allow LLVM to inline at call sites and eliminate unused copies
4.   Call `CodeGenLLVM::Optimize()` for standard LLVM optimization pipeline

Sources: [src/target/llvm/codegen_nvptx.cc 201-213](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L201-L213)

## Common GPU Patterns

Both backends use the `AllocateSharedMemory()` helper to create global variables for shared memory:

**AllocateSharedMemory Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `dtype` | `DataType` | Element data type |
| `size` | `size_t` | Number of elements (0 for dynamic) |

## GPU-Specific Optimizations

### Native Vector Bits

Both GPU backends set `native_vector_bits_` to optimize vectorization:

**Vector Configuration:**

| Backend | Native Vector Bits | Rationale |
| --- | --- | --- |
| NVPTX | 128 (4 × 32) | `float4` is maximum efficient vector width |
| AMDGPU | Default (inherited) | Uses base class defaults |

Sources: [src/target/llvm/codegen_nvptx.cc 218-222](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L218-L222)

### Alignment Constraints

GPU memory has specific alignment requirements:

**Alignment Rules:**

| Backend | Maximum Alignment | Reason |
| --- | --- | --- |
| NVPTX | 16 bytes | Hardware alignment limit |
| AMDGPU | 16 bytes | LDS alignment constraint |

Both backends clamp alignment values to ensure they don't exceed hardware limits.

Sources: [src/target/llvm/codegen_nvptx.cc 92-94](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L92-L94)[src/target/llvm/codegen_amdgpu.cc 119-122](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L119-L122)
