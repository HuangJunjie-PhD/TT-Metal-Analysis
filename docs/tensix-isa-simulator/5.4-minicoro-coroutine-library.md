# Minicoro Coroutine Library

Relevant source files
*   [tensix/whb0/src/minicoro/minicoro.h](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h)

## Overview

The Minicoro Coroutine Library provides a minimal, cross-platform implementation of stackful asymmetric coroutines in pure C. It serves as a supporting library in the Tensix ISA Simulator, enabling cooperative multitasking through coroutines. This document describes the library's architecture, API, and usage patterns.

Sources: [tensix/whb0/src/minicoro/minicoro.h 9-57](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L9-L57)

## Coroutine Fundamentals

Coroutines represent independent "green" threads of execution that, unlike system threads, only suspend their execution by explicitly calling a yield function. This cooperative multitasking model allows for simplified concurrency without the complexities of locks and synchronization primitives.

Key characteristics of coroutines in Minicoro:

*   **Stackful**: Each coroutine has its own stack, allowing it to yield from any function call depth
*   **Asymmetric**: Control flow always returns to the caller when a coroutine yields
*   **Cooperative**: Coroutines voluntarily yield control, not preemptively interrupted

The lifecycle of a coroutine involves the following states:

| State | Description |
| --- | --- |
| `MCO_DEAD` | The coroutine has finished or was uninitialized before finishing |
| `MCO_NORMAL` | The coroutine is active but not running (it has resumed another coroutine) |
| `MCO_RUNNING` | The coroutine is active and running |
| `MCO_SUSPENDED` | The coroutine is suspended (in a call to yield, or not started yet) |

## Library Architecture

The Minicoro library is designed to be minimal and self-contained, with a focus on portability across different platforms and architectures. It implements coroutines through different context-switching mechanisms based on the target platform.

### Implementation Methods

Minicoro uses different methods for context switching based on the platform and architecture:

1.   **Assembly Method**: Direct assembly implementation for specific architectures (x86_64, i686, ARM, ARM64, RISC-V)
2.   **ucontext Method**: Uses the POSIX ucontext API as a fallback on UNIX-like systems
3.   **Fibers Method**: Uses Windows fibers or Emscripten fibers on those platforms

Sources: [tensix/whb0/src/minicoro/minicoro.h 38-57](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L38-L57)[tensix/whb0/src/minicoro/minicoro.h 332-357](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L332-L357)

### Core Data Structures

The library is built around two main structures:

1.   `mco_coro`: The coroutine structure holding the execution context and state
2.   `mco_desc`: Structure used to initialize a coroutine

## API Reference

The Minicoro API consists of several functions organized into the following categories:

### Coroutine Lifecycle Functions

| Function | Description |
| --- | --- |
| `mco_desc_init` | Initialize description of a coroutine |
| `mco_init` | Initialize a coroutine |
| `mco_uninit` | Uninitialize a coroutine |
| `mco_create` | Allocate and initialize a new coroutine |
| `mco_destroy` | Uninitialize and deallocate a coroutine |
| `mco_status` | Get the status of a coroutine |
| `mco_get_user_data` | Get coroutine user data |

### Coroutine Execution Functions

| Function | Description |
| --- | --- |
| `mco_resume` | Start or continue execution of a coroutine |
| `mco_yield` | Suspend execution of a coroutine |
| `mco_running` | Get the currently running coroutine |

### Storage Interface Functions

| Function | Description |
| --- | --- |
| `mco_push` | Push bytes to coroutine storage |
| `mco_pop` | Pop bytes from coroutine storage |
| `mco_peek` | Peek bytes without consuming them |
| `mco_get_bytes_stored` | Get available bytes in storage |
| `mco_get_storage_size` | Get total storage size |

## Data Exchange Between Coroutines

Minicoro provides a storage system to pass data between yield and resume operations. This works in a LIFO (Last In, First Out) manner through the storage buffer associated with each coroutine.

Sources: [tensix/whb0/src/minicoro/minicoro.h 293-298](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L293-L298)[tensix/whb0/src/minicoro/minicoro.h 162-168](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L162-L168)

## Usage Patterns

### Basic Coroutine Usage

To use a coroutine, follow these steps:

1.   Initialize a `mco_desc` structure with `mco_desc_init`
2.   Configure any custom settings in the `mco_desc` structure
3.   Create the coroutine with `mco_create`
4.   Resume the coroutine with `mco_resume`
5.   The coroutine can yield with `mco_yield`
6.   When finished, destroy the coroutine with `mco_destroy`

### Passing Data Between Yield and Resume

The storage interface allows coroutines to pass data to and from their caller:

1.   Before calling `mco_resume`, the caller can push data with `mco_push`
2.   After resuming, the coroutine can retrieve this data with `mco_pop`
3.   Before yielding, the coroutine can push its own data with `mco_push`
4.   After the yield returns, the caller can retrieve this data with `mco_pop`

## Customization Options

The library behavior can be customized through several preprocessor definitions:

| Definition | Description | Default |
| --- | --- | --- |
| `MCO_API` | Public API qualifier | `extern` |
| `MCO_MIN_STACK_SIZE` | Minimum stack size | 32768 |
| `MCO_DEFAULT_STORAGE_SIZE` | Storage buffer size | 1024 |
| `MCO_DEFAULT_STACK_SIZE` | Default stack size | 57344 |
| `MCO_MALLOC` | Allocation function | `malloc` |
| `MCO_FREE` | Deallocation function | `free` |
| `MCO_DEBUG` | Enable debug mode | Defined unless `NDEBUG` or `MCO_NO_DEBUG` is defined |

## Limitations and Caveats

When using Minicoro, be aware of these limitations:

1.   Coroutines are not compatible with C++ exceptions
2.   When using C++ RAII, resume the coroutine until it dies to execute all destructors
3.   The `mco_coro` object is not thread-safe; lock each coroutine to a thread
4.   Stack space is fixed and cannot grow
5.   Stack overflows must be avoided through proper sizing

Sources: [tensix/whb0/src/minicoro/minicoro.h 58-68](https://github.com/tenstorrent/tensix-isa-simulator/blob/afb83529/tensix/whb0/src/minicoro/minicoro.h#L58-L68)

## Integration with Tensix ISA Simulator

In the Tensix ISA Simulator, Minicoro serves as a supporting library that enables cooperative multitasking within the simulation environment. It can be used to:

1.   Implement task scheduling without thread synchronization complexity
2.   Allow long-running operations to yield control periodically
3.   Simplify asynchronous operations by maintaining sequential code flow

The library can be found in the supporting libraries section of the codebase, along with other foundational components like the Core Library, Schedule Library, and Serial Flash Programming Interface.
