This page documents the custom JSON workload format used to define synthetic NoC traffic patterns for performance estimation. This format allows users to manually specify transfers without requiring hardware execution or trace collection.

For information about converting tt-metal NoC traces to workloads, see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads). For programmatic workload construction using the Python API, see [Programmatic Workload Construction](https://deepwiki.com/tenstorrent/tt-npe/3.3-programmatic-workload-construction).

* * *

## Purpose and Scope

The JSON workload format provides a human-readable, declarative specification for NoC traffic patterns. This page documents the **custom JSON format** for creating synthetic workloads, distinct from the tt-metal trace format covered in [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads).

The custom JSON format is primarily used for:

*   **Exploratory design** - Testing hypothetical traffic scenarios before implementing them
*   **Benchmarking** - Creating standardized synthetic workloads for device comparison
*   **Regression testing** - Defining fixed workload patterns to detect performance regressions
*   **Educational purposes** - Understanding NoC behavior with simple, controlled examples

The primary API entry point is `createWorkloadFromJSON`[tt_npe/cpp/src/npeWorkloadIngest.cpp 662-681](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L662-L681) which dispatches to either:

*   `loadJSONWorkloadFormat`[tt_npe/cpp/src/npeWorkloadIngest.cpp 31-219](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L31-L219) for custom JSON workloads (this page's focus)
*   `convertNocTracesToNpeWorkload`[tt_npe/cpp/src/npeWorkloadIngest.cpp 303-660](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L303-L660) for tt-metal trace files (see page 3.4)

## Transfer Field Specification

Each transfer object within a phase describes a single NoC data transfer. The following tables document all supported fields.

### Required Fields

| Field | Type | Description | Validation |
| --- | --- | --- | --- |
| `packet_size` | integer | Size of each packet in bytes | Must be > 0 |
| `num_packets` | integer | Number of packets to transfer | Must be > 0 |
| `src_x` | integer | Source core X coordinate | Must be valid for device grid |
| `src_y` | integer | Source core Y coordinate | Must be valid for device grid |
| `noc_type` | string | NoC to use: `"NOC_0"` or `"NOC_1"` | Must be valid NoC type |

For **unicast** transfers, provide:

*   `dst_x` (integer) - Destination core X coordinate
*   `dst_y` (integer) - Destination core Y coordinate

For **multicast** transfers, provide:

*   `mcast_start_x` (integer) - Multicast grid start X coordinate
*   `mcast_start_y` (integer) - Multicast grid start Y coordinate
*   `mcast_end_x` (integer) - Multicast grid end X coordinate
*   `mcast_end_y` (integer) - Multicast grid end Y coordinate

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 91-203](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L91-L203)

### Optional Fields

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `injection_rate` | float | 0.0 (inferred) | Injection rate in bytes/cycle at source. If 0, will be inferred from source core type |
| `phase_cycle_offset` | integer | 0 | Cycle offset from phase start when transfer can begin |
| `device_id` | integer | 0 | Device ID for multi-chip workloads |
| `noc_event_type` | string | "" | Event type string for metadata (e.g., "READ", "WRITE") |

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 166-191](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L166-L191)

## Unicast vs Multicast Transfer Detection

The parser automatically determines whether a transfer is unicast or multicast based on the presence of destination coordinate fields:

**Unicast Example:**

`{  "packet_size": 2048,  "num_packets": 16,  "src_x": 1, "src_y": 1,  "dst_x": 5, "dst_y": 3,  "noc_type": "NOC_0"}`
**Multicast Example:**

`{  "packet_size": 4096,  "num_packets": 8,  "src_x": 1, "src_y": 1,  "mcast_start_x": 2, "mcast_start_y": 2,  "mcast_end_x": 4, "mcast_end_y": 4,  "noc_type": "NOC_1"}`

## Multi-chip Workload Support

For multi-chip simulations, the `device_id` field specifies which device the source and destination coordinates refer to. The custom JSON parser defaults both source and destination to device ID 0:

`{  "packet_size": 8192,  "num_packets": 1,  "src_x": 1, "src_y": 1,  "dst_x": 1, "dst_y": 1,  "device_id": 0,  "noc_type": "NOC_0"}`
**Important Limitation:** The custom JSON workload format **does not support** multi-chip transfers or explicit fabric routing path specification. All transfers must occur within a single device. The validation logic in `npeWorkloadTransfer::validate` will reject transfers where source and destination device IDs differ [tt_npe/cpp/test/test_npe_workload.cpp 76-94](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/test/test_npe_workload.cpp#L76-L94)

For multi-chip workloads with fabric routing, you must:

1.   Use the tt-metal trace format (see [Converting NoC Traces to Workloads](https://deepwiki.com/tenstorrent/tt-npe/3.4-converting-noc-traces-to-workloads)) which includes `fabric_send` metadata with elaborated routing paths
2.   Process traces with `fabric_post_process.py` (see [Trace Preprocessing and Fabric Routing](https://deepwiki.com/tenstorrent/tt-npe/2.2-trace-preprocessing-and-fabric-routing)) to elaborate fabric paths
3.   Load the resulting trace file using `createWorkloadFromJSON` with `is_tt_metal_trace_format=true`

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 110-125](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L110-L125)[tt_npe/cpp/test/test_npe_workload.cpp 76-94](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/test/test_npe_workload.cpp#L76-L94)

## Golden Result Cycles

The optional `golden_result` object allows specifying expected execution cycles for validation:

`{  "golden_result": {    "cycles": 50000  },  "phases": [...]}`
This is extracted and stored in the workload for comparison with simulation results. The golden cycles represent the expected total execution time.

**Sources:**[tt_npe/cpp/src/npeWorkloadIngest.cpp 70-76](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L70-L76)

## Complete Workload Example

Here is a complete example of a valid JSON workload file:

`{  "golden_result": {    "cycles": 1000  },  "phases": [    {      "transfers": [        {          "packet_size": 2048,          "num_packets": 10,          "src_x": 1,          "src_y": 1,          "dst_x": 5,          "dst_y": 5,          "noc_type": "NOC_0",          "phase_cycle_offset": 0,          "injection_rate": 28.1,          "noc_event_type": "READ"        },        {          "packet_size": 4096,          "num_packets": 5,          "src_x": 2,          "src_y": 2,          "mcast_start_x": 3,          "mcast_start_y": 3,          "mcast_end_x": 5,          "mcast_end_y": 5,          "noc_type": "NOC_1",          "phase_cycle_offset": 100,          "noc_event_type": "WRITE_MULTICAST"        }      ]    },    {      "transfers": [        {          "packet_size": 1024,          "num_packets": 20,          "src_x": 3,          "src_y": 3,          "dst_x": 1,          "dst_y": 1,          "noc_type": "NOC_0",          "phase_cycle_offset": 0        }      ]    }  ]}`
This workload defines:

*   Phase 0: A unicast read and a multicast write, with the multicast starting 100 cycles after phase start
*   Phase 1: A unicast transfer from (3,3) to (1,1)
*   Total expected execution time: 1000 cycles

## Coordinate System and NoC Selection

Transfer coordinates use the logical NoC coordinate system:

*   **src_x / dst_x** = column position (X dimension)
*   **src_y / dst_y** = row position (Y dimension)
*   Coordinate (0, 0) is typically the top-left corner of the grid

The `noc_type` field determines routing behavior:

*   **`"NOC_0"`** - Uses NoC 0 routing (east-then-south dimension-ordered routing)
*   **`"NOC_1"`** - Uses NoC 1 routing (west-then-north dimension-ordered routing)

These routing directions are device-specific and defined by the device model. See [Wormhole B0 Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.2-wormhole-b0-implementation) and [Blackhole Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.3-blackhole-implementation) for architecture-specific details.

## Workload Construction and Validation Flow

The complete flow from JSON file to validated workload object follows this pipeline:

### Validation Checks

The `npeWorkload::validate` method performs comprehensive checks on the loaded workload:

| Check Category | Validation Logic | Rejection Condition |
| --- | --- | --- |
| **Coordinate validity** | All source/destination coordinates within device grid bounds | Out-of-bounds coordinates |
| **Core type validity** | Source cores can inject data (not DRAM-only cores) | Invalid source core type |
| **Packet parameters** | Non-zero packet sizes and counts | `packet_size == 0` or `num_packets == 0` |
| **Device ID consistency** | Source and destination devices match | Mismatched device IDs (custom JSON format) |
| **Device ID existence** | Device IDs are valid for the device model | Non-existent device ID |
| **NoC type validity** | Valid NoC identifier (NOC0/NOC1) | Invalid NoC type |

### C++ API

For direct access to the custom JSON parser (advanced usage):

`std::optional<npeWorkload> workload =     tt_npe::loadJSONWorkloadFormat("path/to/workload.json", true);`

## Best Practices

1.   **Use meaningful phase_cycle_offsets** - Stagger transfers within a phase to model realistic execution patterns
2.   **Specify injection_rate = 0.0 for automatic inference** - Let the device model determine appropriate rates based on core types
3.   **Validate early** - Call `validate()` immediately after loading to catch errors before simulation
4.   **Keep phases logically grouped** - Use phases to represent distinct execution stages (e.g., setup, compute, output)
5.   **Document golden_result** - Include expected cycle counts for regression testing
6.   **Test with small workloads first** - Verify correct behavior with simple transfers before scaling up
