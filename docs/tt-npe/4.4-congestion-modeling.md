## Purpose and Scope

This page describes the congestion modeling algorithm used by tt-npe to simulate resource contention on the Network-on-Chip (NoC). Congestion occurs when multiple concurrent transfers compete for shared resources (links and Network Interface Units), reducing the effective bandwidth available to each transfer. The congestion model estimates this bandwidth degradation to provide accurate performance predictions.

For information about the overall simulation algorithm, see [Performance Estimation Algorithm](https://deepwiki.com/tenstorrent/tt-npe/4.1-performance-estimation-algorithm). For details on device-specific characteristics that influence congestion behavior, see [Device Models](https://deepwiki.com/tenstorrent/tt-npe/5-device-models).

* * *

## Congestion Modeling Overview

The congestion model operates as part of each timestep in the simulation loop. When enabled, it tracks demand on shared NoC resources and derates transfer bandwidth when demand exceeds resource capacity. This modeling is critical for accurate performance estimation in workloads with high NoC utilization.

### Key Characteristics

| Aspect | Description |
| --- | --- |
| **Resource Types** | NoC links (directional), NIU sources, NIU sinks |
| **Granularity** | Per-timestep demand calculation |
| **Algorithm** | Demand accumulation → bottleneck identification → bandwidth derating |
| **Iteration Count** | Single iteration (first-order congestion approximation) |
| **Configuration** | Controlled via `npe.Config.congestion_model_name` ("none" or "default") |

## Demand Calculation

Demand represents the aggregate bandwidth requirement on a resource during a timestep. For each active transfer, the effective demand is computed based on its current bandwidth and the fraction of the timestep it occupies.

### Effective Demand Formula

For a transfer starting at cycle `start_cycle` within a timestep `[start_timestep, end_timestep)`:

```
effective_demand = (end_timestep - max(start_timestep, start_cycle)) / cycles_per_timestep * curr_bandwidth
```

This accounts for transfers that activate mid-timestep, pro-rating their contribution to resource demand.

### Demand Accumulation

Demand is accumulated across three resource grids:

| Grid | Type | Purpose |
| --- | --- | --- |
| `link_demand_grid` | `LinkDemandGrid` | Tracks demand on each directional NoC link |
| `niu_demand_grid` | `NIUDemandGrid` | Tracks demand on each NIU (source and sink) |
| `multicast_write_link_demand_grid` | `LinkDemandGrid` | Separately tracks multicast write demand for statistics |

## Bottleneck Identification and Bandwidth Derating

After accumulating demand, the algorithm identifies the bottleneck resource for each transfer and computes a bandwidth derate factor.

### Bottleneck Detection

For each transfer, the algorithm computes:

1.   **Link bottleneck**: Maximum demand across all links in the route
2.   **Source NIU bottleneck**: Demand at the source NIU divided by injection rate
3.   **Sink NIU bottleneck**: Demand at the sink NIU divided by absorption rate

### Derate Factor Calculation

The derate factor represents the fraction of ideal bandwidth available:

```
derate_factor = resource_capacity / resource_demand
```

When demand exceeds capacity (`derate_factor < 1.0`), the transfer bandwidth is reduced proportionally:

```
new_bandwidth = current_bandwidth * derate_factor
```

**Note:** The implementation uses `1.0 - (grad_fac * (1.0f - overall_bw_derate))` with `grad_fac=1.0`, which simplifies to `overall_bw_derate` for single-iteration modeling.

## Multicast Transfer Handling

Multicast transfers (WRITE_MULTICAST events) require special handling because they fan out to multiple destination cores but share common links.

### Multicast Demand Calculation

For multicast destinations:

1.   **Link demand**: Accumulated on all unique links in the multicast route (union of paths to all destinations)
2.   **Sink NIU demand**: Accumulated on each destination core's sink NIU
3.   **Bottleneck**: Determined by the slowest sink NIU among all destinations

**Note:** Only WORKER cores participate in multicast congestion modeling; other core types ignore multicast traffic.

**Sources:**[tt_npe/cpp/include/device_models/wormhole_b0.hpp 118-128](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L118-L128)[tt_npe/cpp/include/device_models/wormhole_b0.hpp 167-178](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L167-L178)

## Two-Pass Simulation for Congestion Impact

When `npe.Config.estimate_cong_impact` is enabled, the engine runs two simulations:

1.   **Congestion-enabled pass**: Models resource contention (default behavior)
2.   **Congestion-free pass**: Assumes infinite resource capacity

The difference quantifies congestion overhead.

**Diagram: Two-Pass Simulation for Congestion Impact Analysis**

### Congestion Impact Metric

This metric represents the percentage of runtime attributable to congestion, useful for workload optimization.

*   `overall_avg_link_demand`: Time-averaged link demand

*   `overall_avg_link_util`: Time-averaged link utilization
*   `dram_bw_util`: DRAM bandwidth utilization based on transfer activity

## Algorithm Limitations and Trade-offs

The current congestion model makes several simplifying assumptions:

1.   **Single iteration**: Uses first-order approximation (NUM_ITERS=1) rather than iterative convergence
2.   **Timestep granularity**: Finer timesteps increase accuracy but slow simulation
3.   **Instantaneous contention**: Assumes all active transfers contend uniformly within a timestep
4.   **Static routing**: Routes are pre-computed and do not adapt to congestion

These trade-offs prioritize simulation speed while maintaining sufficient accuracy for performance estimation. The single-iteration approach was empirically validated to provide reasonable accuracy (within ~10% of hardware measurements for typical workloads).
