## Overview

The PCIe Subsystem provides high-speed host communication for Blackhole ASIC devices. This document describes the PCIe initialization, configuration, link training, MSI interrupt handling, and DMA operations managed by the Chip Management Firmware (CMFW) running on the ARC processor.

The subsystem supports two independent PCIe instances (PCIe0 and PCIe1), each capable of operating as either an Endpoint (EP) or Root Complex (RC). Configuration is controlled through firmware tables that specify SerDes count, lane configuration, BAR sizes, and device type per board variant.

For information about the boot sequence context where PCIe initialization occurs, see [Boot and Initialization Sequence](https://deepwiki.com/tenstorrent/tt-system-firmware/3.1-boot-and-initialization-sequence). For details on using the message queue to trigger MSI operations, see [SMC Message Handling](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.1-smc-message-handling).

## Architecture Overview

The PCIe subsystem consists of multiple hardware and firmware components that work together to establish and maintain PCIe connectivity:

**Sources:**[lib/tenstorrent/bh_arc/pcie.c 1-506](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L1-L506)[lib/tenstorrent/bh_arc/pcie_msi.c 1-97](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_msi.c#L1-L97)[lib/tenstorrent/bh_arc/pcie_dma.c 1-210](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L1-L210)[lib/tenstorrent/bh_arc/noc2axi.h](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/noc2axi.h)

The PCIe subsystem uses a layered architecture:

*   **Hardware Layer**: SerDes PHY, DBI configuration registers, HDMA engine, MSI catcher
*   **NOC Access Layer**: TLB-based register mapping through the Network-on-Chip
*   **Firmware Control Layer**: Initialization, MSI, and DMA handlers in CMFW

Each PCIe instance has distinct NOC coordinates:

*   PCIe0: logical coordinates (2, 0) - `PCIE_INST0_LOGICAL_X`[lib/tenstorrent/bh_arc/pcie.h 24](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L24-L24)
*   PCIe1: logical coordinates (11, 0) - `PCIE_INST1_LOGICAL_X`[lib/tenstorrent/bh_arc/pcie.h 25](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L25-L25)

## Initialization Sequence

PCIe initialization occurs during the boot sequence at priority 103 (`pcie_init_PRIO`) via `SYS_INIT_APP(pcie_init)`[lib/tenstorrent/bh_arc/pcie.c 505](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L505-L505)

**Sources:**[lib/tenstorrent/bh_arc/pcie.c 450-505](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L450-L505)[lib/tenstorrent/bh_arc/pcie.c 423-448](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L423-L448)[lib/tenstorrent/bh_arc/pcie.c 364-381](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L364-L381)

### Initialization Steps

The initialization process consists of several phases executed for each enabled PCIe instance:

**1. Configuration Parameter Setup**[lib/tenstorrent/bh_arc/pcie.c 195-260](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L195-L260)

`CntlInitV2ParamInit()` reads the firmware table to populate a `struct CntlInitV2Param` with:

*   Board and vendor IDs from `ReadOnly` table
*   SerDes instance count (`num_serdes`: 1 or 2)
*   Maximum PCIe speed (Gen1-4)
*   Device type: `EndPoint` or `RootComplex`[lib/tenstorrent/bh_arc/pcie.h 13-16](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L13-L16)
*   BAR size masks (BAR0, BAR2, BAR4) with power-of-2 rounding

**2. TLB Configuration**[lib/tenstorrent/bh_arc/pcie.c 319-339](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L319-L339)

`ConfigurePCIeTlbs()` sets up NOC2AXI translation lookaside buffers:

| TLB ID | Purpose | Base Address | Description |
| --- | --- | --- | --- |
| 0 | SERDES0_ALPHACORE | `CMN_A_REG_MAP_BASE_ADDR` | SerDes 0 registers |
| 1 | SERDES1_ALPHACORE | `CMN_A_REG_MAP_BASE_ADDR + 0x04000000` | SerDes 1 registers |
| 2 | SERDES0_CTRL | `SERDES_SS_0_A_REG_MAP + 0x03000000` | SerDes 0 control |
| 3 | SERDES1_CTRL | `SERDES_SS_0_A_REG_MAP + 0x07000000` | SerDes 1 control |
| 4 | SII_REG | `PCIE_SII_A_REG_MAP_BASE_ADDR` | SII configuration |
| 14 | DBI_REG | `DBI_ADDR` (TLB 62 << 58) | Config space |
| 5 | TLB_CONFIG | `PCIE_TLB_CONFIG_ADDR` | TLB configuration |

**3. SerDes Initialization**[lib/tenstorrent/bh_arc/pciesd.h 38-39](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L38-L39)

`SerdesInit()` from `libpciesd.a` loads and configures the SerDes firmware. This is a pre-compiled binary blob [lib/tenstorrent/bh_arc/CMakeLists.txt 69](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/CMakeLists.txt#L69-L69)

**4. DBI Access Setup**[lib/tenstorrent/bh_arc/pcie.c 182-193](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L182-L193)

`SetupDbiAccess()` configures the NOC_TLB_DATA register to enable DBI (Data Bus Interface) access by setting the `dbi` bit in TLB 62.

**5. Controller Initialization**[lib/tenstorrent/bh_arc/pciesd.h 45](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L45-L45)

`CntlInitV2()` from `libpciesd.a` programs the PCIe controller with BAR sizes, device type, and other configuration parameters from the `struct CntlInitV2Param`.

**6. Outbound TLB Setup**[lib/tenstorrent/bh_arc/pcie.c 277-317](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L277-L317)

`SetupOutboundTlbs()` configures 8 TLBs (0-7) for different memory access patterns:

*   TLBs with `atu_bypass=1` skip address translation
*   `ro` (relaxed ordering) and `ns` (no snoop) bits control cache coherency
*   Various combinations support different transaction types

## SerDes Configuration

The SerDes (Serializer/Deserializer) PHY provides the physical layer for PCIe connectivity. Blackhole supports up to 2 SerDes instances per PCIe controller, enabling x8 or x16 configurations.

**SerDes Initialization Flow**

  └─> NOC2AXIWrite32(noc_id=0, tlb=(inst==0 ? 0 : 1), addr, data)

## BAR Configuration

Base Address Registers (BARs) define the memory windows through which the host accesses chip resources. Blackhole supports three 64-bit BARs (BAR0/1, BAR2/3, BAR4/5).

## Link Training

Link training establishes the PCIe link and negotiates speed/width. The process differs between Endpoint and Root Complex modes.

### Endpoint Mode

In Endpoint mode, the ASIC waits for the host Root Complex to initiate link training. No firmware intervention is required after initialization.

### Root Complex Mode

When configured as a Root Complex, the ASIC must control the PERST# signal and poll for link-up [lib/tenstorrent/bh_arc/pcie.c 383-399](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L383-L399):

**Sources:**[lib/tenstorrent/bh_arc/pcie.c 383-399](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L383-L399)[lib/tenstorrent/bh_arc/pcie.c 401-421](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L401-L421)[lib/tenstorrent/bh_arc/pcie.c 423-448](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L423-L448)

### LTSSM State Polling

The Link Training and Status State Machine (LTSSM) state is monitored via the `PCIE_SII_A_LTSSM_STATE_REG_OFFSET` register [lib/tenstorrent/bh_arc/pcie.c 401-421](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L401-L421):

Link is considered up when both `rdlh_link_up_sync` and `smlh_link_up_sync` are set [lib/tenstorrent/bh_arc/pcie.c 105-116](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L105-L116)

## Message Signaled Interrupts (MSI)

MSI provides a mechanism for the ASIC to signal interrupts to the host without legacy interrupt pins.

### MSI from ASIC to Host

The firmware can send MSI to the host via the `TT_SMC_MSG_SEND_PCIE_MSI` message [include/tenstorrent/msgqueue.h 263-278](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/include/tenstorrent/msgqueue.h#L263-L278):

**Sources:**[lib/tenstorrent/bh_arc/pcie_msi.c 87-96](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_msi.c#L87-L96)[lib/tenstorrent/bh_arc/pcie_msi.c 45-71](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_msi.c#L45-L71)

| Offset | Register | Purpose |
| --- | --- | --- |
| 0x50 | PCI_MSI_CAP_ID_NEXT_CTRL_REG | Enable, message count |

When the host sends an MSI to the ASIC, the MSI Catcher hardware captures it and triggers an interrupt [lib/tenstorrent/bh_arc/msgqueue.c 415-433](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/msgqueue.c#L415-L433):

## DMA Operations

The Hardware DMA (HDMA) controller enables efficient data transfers between host memory and chip memory without CPU intervention.

### DMA Architecture

**Sources:**[lib/tenstorrent/bh_arc/pcie_dma.c 110-146](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L110-L146)[lib/tenstorrent/bh_arc/pcie_dma.c 149-185](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L149-L185)

### DMA Transfer Sequence

#### Chip-to-Host Transfer (Write Channel)

`PcieDmaWriteTransfer()`[lib/tenstorrent/bh_arc/pcie_dma.c 110-146](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L110-L146) performs the following steps:

1.   **Check Channel Status**: Read `HDMA_STATUS_OFF_WRCH_0` to verify channel is idle (not `DMARunning`)
2.   **Configure Completion Interrupt**: 
    *   Set `HDMA_INT_SETUP_OFF_WRCH_0` with `rsie=1`, `raie=1` (remote stop/abort interrupts)
    *   Program MSI address in `HDMA_MSI_STOP_LOW/HIGH_OFF_WRCH_0`
    *   Program abort MSI address (stop address + 4 bytes)
    *   Set completion data in `HDMA_MSI_MSGD_OFF_WRCH_0`

3.   **Enable Channel**: Write 0x1 to `HDMA_EN_OFF_WRCH_0`
4.   **Configure Transfer**: 
    *   Source address (chip): `HDMA_SAR_LOW/HIGH_OFF_WRCH_0`
    *   Destination address (host): `HDMA_DAR_LOW/HIGH_OFF_WRCH_0`
    *   Transfer size: `HDMA_XFERSIZE_OFF_WRCH_0`

5.   **Trigger Transfer**: Write 0x1 to `HDMA_DOORBELL_OFF_WRCH_0`

#### Host-to-Chip Transfer (Read Channel)

`PcieDmaReadTransfer()`[lib/tenstorrent/bh_arc/pcie_dma.c 149-185](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L149-L185) follows the same pattern but uses read channel registers at offset 0x380100 instead of 0x380000.

### DMA Register Layout

| Register | Write Offset | Read Offset | Description |
| --- | --- | --- | --- |
| STATUS | 0x380080 | 0x380180 | Channel status (idle/running/stopped/aborted) |
| INT_SETUP | 0x380088 | 0x380188 | Interrupt enable flags |
| MSI_STOP_LOW | 0x380090 | 0x380190 | Completion MSI address (low) |
| MSI_STOP_HIGH | 0x380094 | 0x380194 | Completion MSI address (high) |
| MSI_ABORT_LOW | 0x3800A0 | 0x3801A0 | Abort MSI address (low) |
| MSI_ABORT_HIGH | 0x3800A4 | 0x3801A4 | Abort MSI address (high) |
| MSI_MSGD | 0x3800A8 | 0x3801A8 | MSI data payload |
| EN | 0x380000 | 0x380100 | Channel enable |
| SAR_LOW | 0x380020 | 0x380120 | Source address (low) |
| SAR_HIGH | 0x380024 | 0x380124 | Source address (high) |
| DAR_LOW | 0x380028 | 0x380128 | Destination address (low) |
| DAR_HIGH | 0x38002C | 0x38012C | Destination address (high) |
| XFERSIZE | 0x38001C | 0x38011C | Transfer size in bytes |
| DOORBELL | 0x380004 | 0x380104 | Start DMA (write 1) |

**Sources:**[lib/tenstorrent/bh_arc/pcie_dma.c 16-63](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L16-L63)

### Message Handler Integration

The DMA functionality is exposed to the host via message queue commands [lib/tenstorrent/bh_arc/pcie_dma.c 187-209](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L187-L209):

The handler extracts transfer parameters from the request:

*   `data[0]`: command code and completion_data (8-bit)
*   `data[1]`: transfer_size_bytes
*   `data[2-3]`: chip_addr (64-bit)
*   `data[4-5]`: host_addr (64-bit)
*   `data[6-7]`: msi_completion_addr (64-bit)

## TLB and Register Access

The firmware accesses PCIe hardware registers through the NOC2AXI TLB (Translation Lookaside Buffer) system. This indirection layer maps NOC addresses to AXI addresses.

### TLB Address Mapping

**Sources:**[lib/tenstorrent/bh_arc/pcie.h 29-41](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L29-L41)[lib/tenstorrent/bh_arc/pcie.c 120-146](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L120-L146)

### TLB Setup Process

Before accessing PCIe registers, the firmware must configure the TLBs via `NOC2AXITlbSetup()`[lib/tenstorrent/bh_arc/pcie.c 319-339](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L319-L339):

This maps TLB 14 to the DBI register space at NOC coordinates (2, 0) for PCIe0 or (11, 0) for PCIe1.

### DBI Register Access Pattern

The DBI (Data Bus Interface) provides access to the PCIe configuration space. Before any DBI access, the firmware must configure the NOC_TLB_DATA register [lib/tenstorrent/bh_arc/pcie.c 182-193](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L182-L193):

After this, any access through TLB 62 (mapped as TLB 14 in firmware) targets the DBI configuration space.

### Outbound TLB Configuration

`SetupOutboundTlbs()`[lib/tenstorrent/bh_arc/pcie.c 277-317](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L277-L317) configures 8 TLBs (0-7) for different transaction attributes:

| TLB | atu_bypass | ro | ns | Use Case |
| --- | --- | --- | --- | --- |
| 0 | 1 | 0 | 0 | Standard bypass |
| 1 | 1 | 1 | 0 | Relaxed ordering bypass |
| 2 | 1 | 0 | 1 | No-snoop bypass |
| 3 | 1 | 1 | 1 | RO + NS bypass |
| 4 | 0 | 0 | 0 | ATU translated |
| 5 | 0 | 1 | 0 | RO + ATU translated |
| 6 | 0 | 0 | 1 | NS + ATU translated |
| 7 | 0 | 1 | 1 | RO + NS + ATU |

These settings allow the firmware to select appropriate ordering and caching behavior for different memory regions.

## Recovery and Error Handling

The PCIe subsystem implements several error handling mechanisms to maintain reliability.

### Reset Interrupt Handling

Each PCIe instance has an error interrupt registered during initialization [lib/tenstorrent/bh_arc/pcie.c 262-275](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L262-L275):

When a PCIe error interrupt fires, `ChipResetRequest()` is invoked to handle the error condition and potentially initiate a chip reset.

### DMA Busy State Handling

Before starting a DMA transfer, the firmware checks if the channel is already busy [lib/tenstorrent/bh_arc/pcie_dma.c 114-118](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L114-L118)[lib/tenstorrent/bh_arc/pcie_dma.c 153-157](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie_dma.c#L153-L157):

This prevents concurrent DMA operations that could corrupt data or cause hardware errors.

### Link Training Timeout

If link training doesn't complete within 500ms in Root Complex mode, the function returns `PCIeLinkTrainTimeout`[lib/tenstorrent/bh_arc/pcie.c 416-418](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L416-L418):

The calling code can handle this error and potentially retry initialization or report the failure to the DMC.
