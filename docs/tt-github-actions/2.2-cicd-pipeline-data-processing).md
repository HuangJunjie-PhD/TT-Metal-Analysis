The CICD data processing logic is responsible for transforming raw JSON metadata from the GitHub Actions REST API into structured, validated Pydantic models. This pipeline aggregates workflow run information, individual job details, and associated test reports (e.g., JUnit XML) into a unified `Pipeline` object suitable for long-term telemetry storage and analysis.

## Data Flow Overview

The processing is orchestrated by `create_cicd_json_for_data_analysis` in `cicd.py`[[.github/actions/collect_data/src/cicd.py:33-84]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)). It follows a hierarchical construction:

1.   **Pipeline Level**: Metadata about the workflow run (trigger, branch, commit, author).
2.   **Job Level**: Execution details for each job in the workflow (start/end times, status, runner environment).
3.   **Test Level**: Granular results from test reports associated with specific jobs.

## Pipeline Metadata Extraction

The function `get_pipeline_row_from_github_info`[[.github/actions/collect_data/src/utils.py:94-155]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)) extracts top-level metadata. A critical step is determining the actual `pipeline_start_ts`. Because GitHub's `created_at` timestamp reflects the submission time, the pipeline start time is derived by finding the earliest start time among all eligible jobs that started after the pipeline submission [[.github/actions/collect_data/src/utils.py:105-117]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).

| Field | Source Logic |
| --- | --- |
| `github_pipeline_id` | `github_pipeline_json["id"]` |
| `git_author` | `github_pipeline_json["head_commit"]["author"]["name"]` |
| `pipeline_start_ts` | Min(`job["started_at"]`) where `started_at >= created_at` |
| `trigger` | `github_runner_environment["github_event_name"]` |

**Sources:**[[.github/actions/collect_data/src/utils.py:94-155]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing))

## Job Processing and Failure Extraction

Jobs are processed individually to capture execution status and failure context.

### Timestamp Normalization

Timestamps from GitHub are normalized using `parse_timestamp`[[.github/actions/collect_data/src/utils.py:22-67]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)), which handles ISO-8601 variations, including "Z" suffixes and fractional seconds (truncated to microseconds). To ensure uniqueness in databases where job start times might overlap, `alter_time` adds a random millisecond offset to job start timestamps [[.github/actions/collect_data/src/cicd.py:129-135]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).

### Failure Signature and Log Scraping

When a job fails, the system attempts to extract a "signature" to categorize the failure:

1.   **Step Extraction**: `get_failed_steps` identifies all steps with a `failure` conclusion [[.github/actions/collect_data/src/utils.py:179-187]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).
2.   **Signature Generation**: `get_job_failure_signature` uses the name of the first failed step. If logs contain strings like "device timeout" or "potential hang", it appends a `[tt-triage]` tag [[.github/actions/collect_data/src/utils.py:158-176]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).
3.   **Description**: `get_failure_description` scrapes the first 5 lines of error messages from the logs to provide concise context [[.github/actions/collect_data/src/utils.py:190-211]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).

## Artifact Correlation

Test reports (JUnit XML or JSON) are correlated to specific jobs using a naming convention: `<report_name>_<job_id>.xml`.

The function `get_github_job_id_to_test_reports`[[.github/actions/collect_data/src/cicd.py:87-126]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)) walks the artifacts directory and parses the `job_id` from the filename [[.github/actions/collect_data/src/cicd.py:116]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)). It specifically excludes performance reports (`report_perf_`), which are handled by the benchmark pipeline [[.github/actions/collect_data/src/cicd.py:109-110]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).

## Hardware and Environment Detection

The pipeline detects hardware configurations primarily through the `BuilderPytestParser` (used for OpTests). It extracts properties from the JUnit XML:

*   **Card Type**: Extracted from the `<property name="card" ...>` tag in the testsuite [[.github/actions/collect_data/src/parsers/builder_pytest_parser.py:85-94]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).
*   **Backend/Frontend**: Extracted from testcase-level properties (e.g., `ttnn`, `ttmetal`, `shlo`) [[.github/actions/collect_data/src/parsers/builder_pytest_parser.py:171-173]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).
*   **Failure Stage**: Maps stages like `compile`, `golden`, or `runtime` to a `TestStatus` enum [[.github/actions/collect_data/src/parsers/builder_pytest_parser.py:17-22]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).

In general CI jobs, the `card_type` is validated against a known list (e.g., `N300`, `N150`, `E150`) during processing [[.github/actions/collect_data/test/test_generate_data.py:41-43]](https://deepwiki.com/tenstorrent/tt-github-actions/2.2-cicd-pipeline-data-processing)).
