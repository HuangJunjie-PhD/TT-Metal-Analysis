### Hardware Requirements

The compilation backend requires a Tenstorrent AI accelerator card for execution. The system supports:

*   **Single device**: Opened via `ttnn.open_device(device_id=0)` for Wormhole or Blackhole cards
*   **Mesh device**: Opened via `ttnn.open_mesh_device(ttnn.MeshShape(1, 2))` for data parallelism on N300

Multi-device configurations enable data parallel execution through [torch_ttnn/passes/multi_device_pass.py 16-122](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L16-L122) where:

*   Input arguments are sharded across devices via `shard_to_mesh()`[torch_ttnn/passes/multi_device_pass.py 36-42](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L36-L42)
*   Parameters and buffers are replicated via `replicate_to_mesh()`[torch_ttnn/passes/multi_device_pass.py 31-34](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L31-L34)
*   Output tensors are concatenated via `concat_to_tensor()`[torch_ttnn/passes/multi_device_pass.py 44-60](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L44-L60)

## Installation Procedures

### Method 1: End-User Installation from GitHub

For users who want to compile PyTorch models on Tenstorrent hardware without modifying the backend:

`pip install git+https://github.com/tenstorrent/pytorch2.0_ttnn`
This method:

*   Downloads the latest release from the repository
*   Installs the `torch_ttnn` package
*   Pulls all dependencies from `requirements.txt`
*   Makes the backend available as `torch_ttnn.backend`

**Sources:**[docs/README.md.in 16-18](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/README.md.in#L16-L18)

### Method 2: Developer Installation from Source

For contributors who need to modify passes, add operation lowerings, or extend functionality:

`# Clone the repositorygit clone https://github.com/tenstorrent/pytorch2.0_ttnn.gitcd pytorch2.0_ttnn # Install in editable modepip install -e .`
The `-e` flag creates a symlink installation, allowing code changes to take effect immediately without reinstalling.

**Building wheel packages:**

`python -m build`
This generates wheel files in the `dist/` directory.

**Sources:**[docs/README.md.in 20-25](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/README.md.in#L20-L25)[docs/README.md.in 96-99](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/docs/README.md.in#L96-L99)

### Method 3: Docker Container Deployment

The project uses content-addressable Docker containers for reproducible CI/CD environments:

**Content-addressable tagging strategy:**

The `DEV_TAG` is computed as the SHA1 hash of the concatenated contents of:

*   `requirements.txt`
*   `dockerfile/Dockerfile`
*   `requirements-dev.txt`

This ensures:

1.   **Reproducibility**: Same dependencies always produce the same container
2.   **Caching**: Pre-built containers are reused when dependencies don't change
3.   **Traceability**: Container tag directly maps to dependency versions

**Key workflow steps:**

1.   **Check if container exists** ([.github/workflows/update-docker-container.yaml 15-52](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L15-L52)):

    *   Compute `DEV_TAG` from file contents
    *   Query GitHub Container Registry for existence
    *   Skip build if container already exists

2.   **Build and push new container** ([.github/workflows/update-docker-container.yaml 54-82](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L54-L82)):

    *   Build from `dockerfile/Dockerfile`
    *   Push to registry with `DEV_TAG`
    *   Only runs if container doesn't exist

3.   **Run tests** ([.github/workflows/update-docker-container.yaml 83-97](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L83-L97)):

    *   Execute full test suite in the new container
    *   Validate functionality before promoting

4.   **Update `latest` tag** ([.github/workflows/update-docker-container.yaml 98-132](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L98-L132)):

    *   Only if tests pass
    *   Makes the container available for production use

**Base image dependencies:**

The Dockerfile starts from `ghcr.io/tenstorrent/tt-metal/tt-metalium/ubuntu-22.04-dev-amd64:latest`, which provides:

*   Pre-installed tt-metal libraries
*   System-level dependencies for Tenstorrent hardware
*   Development toolchain

Additional system packages installed:

*   `sudo`: For privileged operations
*   `libgl1-mesa-glx`: OpenGL libraries
*   `git-lfs`: Large file support for models
*   `libsndfile1`: Audio processing (for speech models)
*   `docker.io`: Nested Docker support

**Sources:**[.github/workflows/update-docker-container.yaml 26-34](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L26-L34)[.github/workflows/update-docker-container.yaml 113-126](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-docker-container.yaml#L113-L126)[dockerfile/Dockerfile 1-15](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/dockerfile/Dockerfile#L1-L15)

## Dependency Installation Process

### Pip Configuration for Extra Index URLs

The installation process configures pip to access both the standard PyPI and specialized PyTorch/Tenstorrent package indexes:

The configuration command used in CI/CD:

`python3 -m pip config set global.extra-index-url https://download.pytorch.org/whl/cpu`
This allows pip to find the CPU-only PyTorch builds and the proprietary ttnn wheel without requiring manual URL specification.

**Sources:**[.github/actions/common_repo_setup/action.yaml 28](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/actions/common_repo_setup/action.yaml#L28-L28)[requirements.txt 9](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/requirements.txt#L9-L9)

### ttnn Wheel Specification

The ttnn dependency is specified with explicit version pinning and checksum verification:

`ttnn @ https://pypi.eng.aws.tenstorrent.com/ttnn/ttnn-0.62.0rc36.dev247%2Bg627c4eed5b-cp310-cp310-manylinux_2_34_x86_64.whl#sha256=c219ab8b8179d160a5262a802706710c0c4ae135bfbd5d203d8107fa3b6d0d07 ; python_version=="3.10"`
This specification includes:

*   **Version**: `0.62.0rc36.dev247+g627c4eed5b`
*   **Python compatibility**: `cp310` (CPython 3.10)
*   **Platform**: `manylinux_2_34_x86_64` (glibc 2.34+, x86-64)
*   **SHA256 checksum**: Ensures wheel integrity
*   **Python version constraint**: Only installs on Python 3.10

**Sources:**[requirements.txt 9](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/requirements.txt#L9-L9)

## Verification and Testing

### Verifying Installation

After installation, verify the backend is available:

`import torchimport torch_ttnn # Check versionprint(torch.__version__)  # Should be 2.7.1+cpuprint(torch_ttnn.__version__) # Verify backend registrationprint(torch_ttnn.backend)  # Should show <function backend at 0x...>`
For hardware access verification, open a device:

`import ttnn # Single device (Wormhole/Blackhole)device = ttnn.open_device(device_id=0)print(f"Device opened: {device}")print(f"Number of devices: {device.get_num_devices()}")  # Returns 1ttnn.close_device(device) # Multi-device mesh (N300)mesh = ttnn.open_mesh_device(ttnn.MeshShape(1, 2))print(f"Mesh device opened: {mesh}")print(f"Number of devices: {mesh.get_num_devices()}")  # Returns 2ttnn.close_mesh_device(mesh)`
The `device.get_num_devices()` method is checked by [torch_ttnn/passes/multi_device_pass.py 82](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/multi_device_pass.py#L82-L82) and [torch_ttnn/passes/analysis/multi_device_shard_analysis_pass.py 46](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/analysis/multi_device_shard_analysis_pass.py#L46-L46) to determine if multi-device transformations should be applied.

### Running Tests

Development installations should run the test suite to verify functionality:

`# Run all testspytest tests/ # Run specific test categoriespytest tests/models/          # Full model tests (e.g., ResNet, BERT)pytest tests/lowering/        # Operation lowering tests (e.g., test_add.py)pytest tests/multi_device/    # Multi-device data parallel tests`
Example test execution for a specific operation:

`# Test scaled dot product attention loweringpytest tests/lowering/misc/test_scaled_dot_product_attention.py -v`
This test verifies that `torch.nn.functional.scaled_dot_product_attention` is correctly lowered to `ttnn.transformer.scaled_dot_product_attention`[tests/lowering/misc/test_scaled_dot_product_attention.py 72-74](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/misc/test_scaled_dot_product_attention.py#L72-L74)

## Summary

The `pytorch2.0_ttnn` backend requires:

*   **Python 3.10** on Ubuntu 22.04 (recommended)
*   **PyTorch 2.7.1** (CPU-only build)
*   **ttnn wheel** from Tenstorrent's internal PyPI
*   **Tenstorrent hardware** (Wormhole/Blackhole/N300) for execution

Three installation methods are available:

1.   **Pip install** for end-users
2.   **Editable install** for developers
3.   **Docker containers** for CI/CD with content-addressable tagging

The installation process automatically handles PyTorch CPU builds and ttnn wheel resolution through configured pip index URLs. Docker containers provide reproducible environments with SHA1-based versioning of all dependencies.
