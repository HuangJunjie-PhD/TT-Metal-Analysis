This page explains how tt-system-firmware integrates with the Zephyr RTOS ecosystem as a Zephyr module. It covers the west workspace configuration, module definition, binary blob management, custom board support, and integration with the Zephyr build system. For information about the build pipeline and CI/CD workflows, see [Build Pipeline and Artifacts](https://deepwiki.com/tenstorrent/tt-system-firmware/6.2-build-pipeline-and-artifacts). For details on firmware signing, see [Firmware Signing and MCUBoot](https://deepwiki.com/tenstorrent/tt-system-firmware/6.3-firmware-signing-and-mcuboot).

## West Workspace Configuration

The tt-system-firmware repository uses west as a meta-tool to manage multiple Git repositories. The workspace is defined in `west.yml`, which specifies the Zephyr fork, dependencies, and import filters.

### Repository Manifest

**West Manifest Configuration**

The manifest defines two Git repository remotes and imports selected Zephyr modules. The Zephyr project uses a custom fork at `tenstorrent/zephyr-fork` rather than the upstream Zephyr repository, allowing for board-specific patches and customizations.

**Sources:**[west.yml 1-33](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/west.yml#L1-L33)

### Project Dependencies

| Project | Remote | Path | Purpose |
| --- | --- | --- | --- |
| librpmi | tenstorrent-forks | modules/lib/librpmi | RPMI library for power management |
| zephyr | tenstorrent | (zephyr-fork) | Zephyr RTOS with custom patches |
| cmsis_6 | (imported) | modules/hal/cmsis | ARM CMSIS headers |
| hal_stm32 | (imported) | modules/hal/stm32 | STM32 hardware abstraction |
| mbedtls | (imported) | modules/crypto/mbedtls | TLS/crypto library |
| mcuboot | (imported) | bootloader/mcuboot | Secure bootloader |
| nanopb | (imported) | modules/lib/nanopb | Protocol buffers |
| segger | (imported) | modules/debug/segger | Segger RTT for debugging |

The `import` section uses `name-allowlist` to selectively include only required Zephyr modules, reducing workspace size and build time.

**Sources:**[west.yml 11-29](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/west.yml#L11-L29)

## Zephyr Module Definition

The `zephyr/module.yml` file declares tt-system-firmware as a Zephyr module, specifying build integration, custom directories, test configurations, and binary blob management.

### Module Structure

**Module Configuration**

The module definition integrates tt-system-firmware into the Zephyr build system by specifying:

*   Build system files (`CMakeLists.txt` and `Kconfig` at repository root)
*   Custom directory roots for boards, device trees, snippets, and SoCs
*   Sample and test locations for Zephyr's Twister test framework
*   Custom runners for firmware flashing and debugging
*   Python dependencies via pip

**Sources:**[zephyr/module.yml 1-34](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L1-L34)

### Build Integration Settings

The build integration settings allow Zephyr to discover custom components:

| Setting | Value | Purpose |
| --- | --- | --- |
| `cmake` | `.` | Top-level CMakeLists.txt location |
| `kconfig` | `Kconfig` | Top-level Kconfig file location |
| `board_root` | `.` | Search path for custom board definitions |
| `dts_root` | `.` | Search path for device tree files |
| `snippet_root` | `.` | Search path for device tree snippets |
| `soc_root` | `.` | Search path for SoC definitions |

These settings enable Zephyr's build system to locate board definitions under `boards/tenstorrent/`, device tree files, and other custom components without requiring them to be in the Zephyr tree itself.

**Sources:**[zephyr/module.yml 6-14](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L6-L14)

### Twister Integration

**Test Discovery Configuration**

The module.yml declares paths that Zephyr's Twister test framework uses to discover buildable applications and test cases. This enables `west twister` to automatically find and build tests without manual configuration.

**Sources:**[zephyr/module.yml 16-23](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L16-L23)

## Binary Firmware Blobs

The module.yml declares 20 pre-compiled binary firmware blobs that are packaged with the SMC/DMC firmware. These blobs are for hardware subsystems (MRISC, ERISC, SerDes) that require specialized firmware.

### Blob Categories

**Blob Organization**

Binary blobs are categorized by hardware subsystem. Each blob includes version information, SHA256 checksum for integrity verification, license information, and a download URL. The blobs are stored in `zephyr/blobs/` and referenced during the build process.

**Sources:**[zephyr/module.yml 36-235](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L36-L235)

### Blob Declaration Format

Each blob entry in module.yml follows a consistent structure:

`- path: tt_blackhole_erisc.bin  sha256: c797a2f75823c5499791c4c7d02a9196ede520d11e03bd4fc1ffd583767ad28f  type: img  version: '1.9.0'  license-path: zephyr/blobs/license.txt  url: "https://github.com/tenstorrent/tt-system-firmware/..."  description: "Ethernet FW"  doc-url: https://tenstorrent.com`
The SHA256 checksum enables automated verification of blob integrity during builds and CI/CD pipelines.

**Sources:**[zephyr/module.yml 37-44](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L37-L44)

### GDDR Parameter Variants

The GDDR subsystem requires board-specific parameter files to configure memory training:

| Board Variant | Blob File | Purpose |
| --- | --- | --- |
| ORION | `tt_blackhole_gddr_params_ORION.bin` | ORION board memory configuration |
| P100A | `tt_blackhole_gddr_params_P100A.bin` | P100A board memory configuration |
| P100B | `tt_blackhole_gddr_params_P100B.bin` | P100B board memory configuration |
| P100 | `tt_blackhole_gddr_params_P100.bin` | P100 generic board configuration |
| P150A/B/C | `tt_blackhole_gddr_params_P150*.bin` | P150 board variants |
| P300A/B/C | `tt_blackhole_gddr_params_P300*.bin` | P300 board variants |
| GALAXY | `tt_blackhole_gddr_params_GALAXY.bin` | Galaxy board configuration |
| GALAXY_REVC | `tt_blackhole_gddr_params_GALAXY_REVC.bin` | Galaxy Rev C configuration |

All GDDR parameter files share the same base firmware (`tt_blackhole_gddr_init.bin` v2.13) but use board-specific parameter blobs for memory initialization.

**Sources:**[zephyr/module.yml 91-198](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L91-L198)

## Custom Board Definitions

The tt-system-firmware repository provides custom board definitions for Tenstorrent hardware platforms. Board definitions are located under `boards/tenstorrent/` and follow Zephyr's board definition structure.

### Board Definition Structure

**Board Definition Organization**

Zephyr discovers board definitions through the `board_root` setting. Each board has a directory containing device tree files, default configuration, and build scripts. Applications and tests can provide board-specific overlays to customize device tree and configuration settings.

**Sources:**[zephyr/module.yml 11](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L11-L11)[app/dmc/boards/tt_blackhole_tt_blackhole_dmc.overlay 1-10](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/boards/tt_blackhole_tt_blackhole_dmc.overlay#L1-L10)

### Board-Specific Overlays

Applications provide board-specific device tree overlays to enable or configure peripherals:

`// app/dmc/boards/tt_blackhole_tt_blackhole_dmc.overlay&jtag0 {    status = "okay";};`
This overlay enables the JTAG peripheral for the DMC application on Blackhole boards. Test configurations similarly use overlays to configure test-specific peripherals.

**Sources:**[app/dmc/boards/tt_blackhole_tt_blackhole_dmc.overlay 7-9](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/boards/tt_blackhole_tt_blackhole_dmc.overlay#L7-L9)

### Test Board Configuration

**Test Configuration Mechanism**

Test cases use `testcase.yaml` to specify board-specific overlays and configuration files. The overlay file modifies the device tree (e.g., enabling PWM test alias), while the configuration file enables required Kconfig options (e.g., `CONFIG_MFD_MAX6639=y`).

**Sources:**[test-conf/tests/drivers/pwm/pwm_api/testcase.yaml 1-18](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/test-conf/tests/drivers/pwm/pwm_api/testcase.yaml#L1-L18)[test-conf/tests/drivers/pwm/pwm_api/boards/tt_blackhole_tt_blackhole_dmc.overlay 1-10](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/test-conf/tests/drivers/pwm/pwm_api/boards/tt_blackhole_tt_blackhole_dmc.overlay#L1-L10)[test-conf/tests/drivers/pwm/pwm_api/boards/tt_blackhole_tt_blackhole_dmc.conf 1-22](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/test-conf/tests/drivers/pwm/pwm_api/boards/tt_blackhole_tt_blackhole_dmc.conf#L1-L22)

## Custom Flash Runners

The module.yml declares three custom flash runners that integrate with Zephyr's `west flash` and `west debug` commands:

### Flash Runner Architecture

**Runner Integration**

Custom runners extend Zephyr's flash/debug infrastructure to support Tenstorrent hardware. The `tt_flash_runner.py` uses the tt-flash library to update firmware on physical boards via PCIe or SMBus, while `tt_pyluwen_runner.py` provides debug capabilities through the pyluwen hardware abstraction library.

**Sources:**[zephyr/module.yml 25-29](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L25-L29)

### Runner Declaration

The runners section in module.yml registers Python scripts as Zephyr flash runners:

`runners:  - file: scripts/tt_bootstrap.py  - file: scripts/tt_flash_runner.py  - file: scripts/tt_pyluwen_runner.py`
These runners are automatically discovered by Zephyr's build system and can be selected via the `--runner` argument to `west flash` or configured as the default runner for a board.

**Sources:**[zephyr/module.yml 25-29](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L25-L29)

## Zephyr Fork and Patches

The tt-system-firmware project uses a custom Zephyr fork (`tenstorrent/zephyr-fork`) rather than the upstream Zephyr repository. This allows for board-specific patches, driver additions, and custom features.

### Fork Management

**Fork Strategy**

The Zephyr fork is maintained at a specific revision (01cee78c8305362729c1293714d1ac449203152f) and includes custom drivers for ARC and RISC-V processors, board support for Tenstorrent platforms, and bug fixes that may not yet be in upstream Zephyr.

**Sources:**[west.yml 17-20](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/west.yml#L17-L20)

### Module Import Filtering

The Zephyr import uses `name-allowlist` to include only required modules:

`import:  name-allowlist:    - cmsis_6    - hal_stm32    - mbedtls    - mcuboot    - nanopb    - segger`
This selective import reduces the workspace size and build time by excluding unnecessary Zephyr modules (e.g., hardware abstraction layers for unsupported platforms, sample applications, documentation).

**Sources:**[west.yml 21-28](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/west.yml#L21-L28)

## Version Management

The tt-system-firmware repository uses a hierarchical versioning scheme where the product version contains independently versioned firmware applications.

### Version Hierarchy

**Version Structure**

Each VERSION file follows the format:

```
VERSION_MAJOR = X
VERSION_MINOR = Y
PATCHLEVEL = Z
VERSION_TWEAK = W
EXTRAVERSION =
```

The product version (19.7.99.0) represents the overall firmware bundle, while SMC (0.29.99.0) and DMC (0.23.99.0) have independent versions. Binary blobs also maintain independent versions declared in module.yml.

**Sources:**[VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/VERSION#L1-L6)[app/smc/VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/VERSION#L1-L6)[app/dmc/VERSION 1-6](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/VERSION#L1-L6)[zephyr/module.yml 40-85](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L40-L85)

The module.yml declares Python dependencies that are automatically installed by west:

**Dependency Installation**

`package-managers:  pip:    requirement-files:      - scripts/requirements.txt`

## Summary

The tt-system-firmware integration with Zephyr provides:

1.   **West Workspace**: Managed multi-repository workspace with Zephyr fork and dependencies
2.   **Module Definition**: Comprehensive module.yml declaring build integration, custom roots, and binary blobs
3.   **Binary Blobs**: 20 pre-compiled firmware images for hardware subsystems with SHA256 verification
4.   **Custom Boards**: Board definitions under `boards/tenstorrent/` with application-specific overlays
5.   **Flash Runners**: Custom `west flash` runners for Tenstorrent hardware via tt-flash and pyluwen
6.   **Zephyr Fork**: Customized Zephyr with board support, drivers, and patches
7.   **Version Management**: Hierarchical versioning for product, applications, and binary blobs
8.   **Dependency Management**: Automated Python dependency installation via pip

This integration allows tt-system-firmware to leverage Zephyr's build system, testing framework (Twister), and tooling infrastructure while maintaining full control over board definitions, firmware blobs, and flash mechanisms.
