The GitHub Metadata Fetcher is a Python-based automation component responsible for augmenting static entry data with live signals from the GitHub REST API. It populates star counts, identifies repository forks, retrieves release histories, and extracts preview images from README files. This dynamic data is stored in a central JSON cache and merged into the project's data layer at build time.

## Overview and Purpose

The primary goal of `scripts/fetch_github_meta.py` is to provide a "live" feel to the tt-awesome directory without requiring manual updates for every repository metric [scripts/fetch_github_meta.py 5-6](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L5-L6) It operates by scanning all project entries in the `entries/` directory, identifying those with a GitHub repository link, and querying the GitHub API for the latest metadata [scripts/fetch_github_meta.py 120-125](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L120-L125)

### Key Capabilities

*   **Star Tracking**: Fetches `stargazers_count` for community traction metrics [scripts/fetch_github_meta.py 49](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L49-L49)
*   **Fork Detection**: Identifies if a project is a fork and preserves the upstream parent name and URL [scripts/fetch_github_meta.py 54-56](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L54-L56)
*   **Release History**: Retrieves the most recent 5 non-draft releases for the "Recent Releases" sidebar [scripts/fetch_github_meta.py 88-111](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L88-L111)
*   **Image Extraction**: Heuristically finds the first "meaningful" image in a repository's README to use as a card preview [scripts/fetch_github_meta.py 65-85](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L65-L85)

## Implementation Detail

### Data Flow: Fetching to Merging

The fetcher acts as the first stage in a nightly pipeline. It produces `src/_data/github_meta.json`, which is then consumed by the Eleventy static site generator.

### Fork and Image Heuristics

The fetcher uses a specific regex, `SKIP_IMG_RE`, to avoid using CI badges (Shields.io, Travis, etc.) or small icons as project previews [scripts/fetch_github_meta.py 23-28](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L23-L28) For relative image paths found in READMEs, the script normalizes them into absolute `raw.githubusercontent.com` URLs using the repository's default branch [scripts/fetch_github_meta.py 82-84](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L82-L84)

## Data Schema: github_meta.json

The output file is a flat dictionary where keys are the full GitHub repository URLs. This allows the frontend to perform an O(1) lookup during the build phase.

## Automation Schedule

The fetcher is executed via the `nightly.yml` GitHub Actions workflow.

### Nightly Workflow Execution

Sources: [.github/workflows/nightly.yml 1-50](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/.github/workflows/nightly.yml#L1-L50)[scripts/fetch_github_meta.py 1-18](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/scripts/fetch_github_meta.py#L1-L18)

### Schedule Details

*   **Frequency**: Twice daily at 06:17 and 18:17 UTC [`.github/workflows/nightly.yml 4](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/%60.github/workflows/nightly.yml#L4-L4)
*   **PR Mechanism**: Instead of pushing directly to `main`, the workflow creates a branch named `chore/nightly-github-metadata` and opens a Pull Request [`.github/workflows/nightly.yml 45-48](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/%60.github/workflows/nightly.yml#L45-L48) This allows maintainers to verify that no breaking changes occurred in the metadata format before merging.
*   **Token Usage**: The script uses the standard `GITHUB_TOKEN` provided by the action to avoid rate-limiting [`.github/workflows/nightly.yml 28](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/%60.github/workflows/nightly.yml#L28-L28)

## Testing

The fetcher logic is validated by `tests/test_fetch_github_meta.py`, which mocks GitHub API responses to ensure:

1.   **Fork Logic**: Correctly handles repositories that are forks vs. those that are not [tests/test_fetch_github_meta.py 25-56](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_fetch_github_meta.py#L25-L56)
2.   **Release Filtering**: Verifies that draft releases are excluded while pre-releases are kept [tests/test_fetch_github_meta.py 74-102](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_fetch_github_meta.py#L74-L102)
3.   **Resilience**: Ensures the script does not crash if the `parent` key is missing in a fork response [tests/test_fetch_github_meta.py 58-70](https://github.com/tenstorrent/tt-awesome/blob/1d86dbe5/tests/test_fetch_github_meta.py#L58-L70)
