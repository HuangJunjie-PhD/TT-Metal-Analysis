# JSON Backend

Relevant source files
*   [src/backend/json.rs](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/json.rs)

The **JSON Backend** is a subprocess-based telemetry provider that integrates with the `tt-smi` utility. It operates by spawning `tt-smi -s` (snapshot mode), capturing the resulting JSON output, and parsing it into the internal `tt-toplike` data models. This backend is particularly useful in environments where direct sysfs access is restricted or when the user prefers a high-level hardware abstraction provided by official tooling.

### Architecture and Data Flow

The backend follows a request-response cycle. Each call to `update()` triggers a new subprocess execution. The output is deserialized from JSON, which often contains numeric values encoded as strings (e.g., `"power": " 16.0"`), requiring specialized handling.

### Implementation Details

The core logic resides in the `JSONBackend` struct, which manages the command path and the cached telemetry state.

#### Key Structures

*   **`TTSMIDeviceRaw`**: Represents the raw structure returned by `tt-smi`. It includes optional fields for `board_info`, `telemetry`, `smbus_telem`, `firmwares`, and `limits`[src/backend/json.rs 71-79](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/json.rs#L71-L79)
*   **`TelemetryJSON`**: Handles core telemetry. It uses custom deserializers `de_opt_f32_str` and `de_opt_u32_str` to handle the whitespace-padded strings produced by `tt-smi`[src/backend/json.rs 123-136](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/json.rs#L123-L136)
*   **`SmbusTelemetryJSON`**: Maps the `SCREAMING_SNAKE_CASE` keys from `tt-smi`'s SMBUS output to Rust fields [src/backend/json.rs 143-181](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/json.rs#L143-L181)

#### Handling String-Encoded Numbers

Because `tt-smi` often outputs numbers as strings (e.g., `" 1.23"`), the backend implements `de_opt_f32_str`. This function trims leading/trailing whitespace before attempting to parse the value as a float, returning `None` if the field is null or unparseable [src/backend/json.rs 47-56](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/backend/json.rs#L47-L56)

### Configuration and Use Cases

The `JSONBackend` is initialized with the path to the `tt-smi` binary.

`let mut backend = JSONBackend::new("tt-smi");backend.init()?;`

#### When to Prefer JSON Backend

1.   **Compatibility**: When running on systems where `tt-smi` is the only validated source of truth for telemetry.
2.   **SMBUS Data**: When rich SMBUS data (firmware versions, board IDs) is required without manual bit-banging of sysfs entries.
3.   **Restricted Environments**: When the user has permissions to run `tt-smi` but lacks read access to `/sys/class/hwmon/`.

#### Limitations

*   **Overhead**: Spawning a process for every update is more resource-intensive than reading sysfs files directly.
*   **Latency**: The sampling rate is limited by the execution time of `tt-smi`.
*   **Dependency**: Requires the `tt-smi` utility to be installed and present in the system `PATH`.
