## Purpose and Scope

The C Runtime (CRT) is TVM's minimal, standalone runtime implementation written in pure C for executing compiled models on resource-constrained embedded systems and microcontrollers. Unlike the full C++ TVM runtime, CRT is designed for bare-metal and RTOS environments where dynamic memory allocation, operating system services, and C++ features are limited or unavailable.

This page documents the architecture, components, and usage of CRT. For information about the AOT (Ahead-of-Time) executor that compiles models for CRT, see [AOT Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.3-aot-executor). For the broader MicroTVM ecosystem that uses CRT for embedded deployment, refer to the MicroTVM sections mentioned in the project documentation.

## Overview

### What is CRT?

CRT is a lightweight, portable runtime that enables TVM-compiled models to execute on embedded devices with minimal resource overhead. It provides the essential runtime services needed to:

*   Initialize the TVM runtime environment
*   Load and execute AOT-compiled models
*   Manage memory through platform-specific abstractions
*   Handle errors without exception mechanisms
*   Provide a C API compatible with embedded toolchains

### Design Principles

| Principle | Implementation |
| --- | --- |
| **Pure C** | No C++ features, compatible with C99 compilers |
| **Minimal Dependencies** | No standard library dependencies beyond basic types |
| **Platform Abstraction** | Memory and system functions provided by platform |
| **Static Allocation** | Primarily stack and statically-allocated memory |
| **Error Codes** | Integer error codes instead of exceptions |
| **AOT-Focused** | Designed for ahead-of-time compiled models |

### Runtime Initialization

**CRT Initialization Sequence**

The `TVMInitializeRuntime()` function is the entry point for CRT initialization:

*   **Preconditions**: Platform memory functions (`TVMPlatformMemory*`) must be available

## Memory Management

### Platform Memory Abstraction

CRT delegates all memory allocation to platform-specific implementations through the `TVMPlatformMemory*` function family:

**Platform Memory Abstraction Layer**

### Memory Management Requirements

Platform implementations must provide the following memory functions:

| Function | Purpose | Constraints |
| --- | --- | --- |
| `TVMPlatformMemoryAllocate` | Allocate memory block | Must return aligned pointer or NULL |
| `TVMPlatformMemoryFree` | Free memory block | Must handle NULL pointer safely |
| `TVMPlatformMemoryInit` | Initialize memory system | Called before `TVMInitializeRuntime()` |

**Key Characteristics:**

*   **No Standard Library**: CRT does not use `malloc`/`free` directly
*   **Alignment**: Platform must ensure proper alignment for data structures
*   **Failure Handling**: Allocation failure returns NULL, not exception
*   **Determinism**: Many embedded platforms use static allocation for deterministic behavior

## Integration with AOT Executor

### AOT Compilation for CRT

The AOT executor generates C code specifically designed to work with CRT:

**AOT Compilation Flow for CRT**

### Generated Code Structure

AOT-generated code for CRT has the following characteristics:

*   **Single Entry Point**: `__tvm_main__()` function serves as the model entry
*   **Statically Linked**: All operators compiled into the binary
*   **No Dynamic Dispatch**: Function calls resolved at compile time
*   **Workspace Calculation**: Static analysis determines scratch buffer sizes
*   **Parameter Embedding**: Model weights can be embedded in code or loaded separately

### Memory Budget Planning

CRT applications require careful memory planning:

| Component | Typical Size | Considerations |
| --- | --- | --- |
| **Code Section** | 10KB - 1MB | Model complexity, operator fusion |
| **Model Parameters** | 100KB - 10MB | Quantization, weight sharing |
| **Activation Buffers** | 1KB - 100KB | Input/output tensor sizes |
| **Workspace** | 1KB - 50KB | Largest operator scratch space |
| **Stack** | 4KB - 32KB | Function call depth, local variables |

**Optimization Strategies:**

*   **Quantization**: Use INT8/INT16 instead of FP32
*   **Operator Fusion**: Reduce intermediate buffer allocation
*   **In-Place Operations**: Reuse buffers when possible
*   **Workspace Reuse**: Share scratch space across operators

## Performance Considerations

### Code Size Optimization

CRT enables minimal code size through:

**Code Size Optimization Strategy**

**Measured Impact:**

*   Operator fusion: 20-40% code size reduction
*   Quantization: 4x model size reduction (FP32→INT8)
*   Link-time optimization: 10-20% additional reduction

### Memory Optimization

| Technique | Memory Savings | Trade-offs |
| --- | --- | --- |
| **Weight Quantization** | 75% (FP32→INT8) | Potential accuracy loss |
| **Activation Quantization** | 50-75% runtime | Requires calibration |
| **Buffer Reuse** | 30-50% workspace | AOT compiler optimization |
| **In-Place Operations** | Variable | Limited operator support |
| **Workspace Pooling** | 20-40% scratch | Static analysis required |

### Latency Optimization

CRT achieves low latency through:

*   **Static Scheduling**: No runtime overhead for graph traversal
*   **Operator Fusion**: Reduced memory bandwidth
*   **Hardware Intrinsics**: Platform-specific SIMD when available
*   **Cache-Friendly Access**: Optimized memory access patterns

**Auto-Tuning for Embedded Targets:** Auto-scheduler can optimize specifically for CRT constraints:

*   Limited memory bandwidth
*   Small cache sizes
*   No branch prediction (simple cores)
*   Instruction-level parallelism opportunities

## Limitations and Constraints

### Current Limitations

| Area | Limitation | Workaround |
| --- | --- | --- |
| **Dynamic Shapes** | Not supported in AOT+CRT | Use fixed shapes or multiple compiled variants |
| **Control Flow** | Limited support | Unroll loops at compile time |
| **Dynamic Memory** | Platform-dependent | Use static allocation where possible |
| **Debugging** | Limited tooling | Platform-specific debuggers, logging |
| **Model Updates** | Requires recompilation | Over-the-air update infrastructure |

### Resource Requirements

**Minimum Requirements:**

*   **Flash**: 32KB+ (small models), typical 128KB-1MB
*   **RAM**: 8KB+ (simple models), typical 32KB-256KB
*   **CPU**: ARM Cortex-M3+ or equivalent
*   **Compiler**: C99-compatible toolchain

**Recommended Requirements:**

*   **Flash**: 512KB+ for practical models
*   **RAM**: 128KB+ for intermediate complexity
*   **CPU**: ARM Cortex-M4+ with DSP/FPU
*   **Debug**: JTAG/SWD interface for development

## Summary

The C Runtime (CRT) provides a minimal, portable foundation for deploying TVM-compiled models on embedded systems. Its key strengths are:

*   **Minimal Footprint**: Pure C implementation with no standard library dependencies
*   **Platform Portability**: Abstracted platform functions enable wide hardware support
*   **AOT Integration**: Seamless integration with AOT executor for static compilation
*   **MicroTVM Ecosystem**: Part of a complete embedded ML deployment solution
*   **Production Ready**: Used in commercial embedded deployments across multiple platforms

CRT enables TVM to target the full spectrum of embedded devices from high-end microcontrollers to bare-metal systems, making advanced ML models accessible in resource-constrained environments.
