## Purpose and Scope

This document describes the tensor parallelism implementation in TT-Blacksmith, which enables distributing model weights across multiple devices to train models that exceed single-chip memory capacity. Tensor parallelism shards the model parameters along specified dimensions while replicating input data across all devices. For information about sharding data across devices while replicating the model, see [Data Parallelism](https://deepwiki.com/tenstorrent/tt-blacksmith/8.1-data-parallelism). For strategies that combine both approaches, see [Hybrid Parallelism Strategies](https://deepwiki.com/tenstorrent/tt-blacksmith/8.3-hybrid-parallelism-strategies).

## Overview

Tensor parallelism partitions model layers across multiple devices by splitting weight matrices along one or more dimensions. This contrasts with data parallelism, where each device holds a complete copy of the model but processes different data shards. The TT-Blacksmith implementation uses a mesh-based abstraction where devices are arranged in a logical grid with named axes, and model parameters are assigned partition specifications that determine how they are distributed across the mesh.

## Configuration

Tensor parallelism is configured through three key fields in `TrainingConfig`:

| Field | Type | Description |
| --- | --- | --- |
| `mesh_shape` | `Optional[list[int]]` | Shape of device mesh, e.g., `[1, 2]` for 1 data axis, 2 model axis |
| `mesh_axis_names` | `Optional[list[str]]` | Semantic names for axes, e.g., `["data", "model"]` |
| `model_sharding_patterns` | `Optional[List[Tuple[str, Tuple[Optional[str], ...]]]]` | Regex patterns mapping module names to partition specs |

The `mesh_shape` defines the physical layout of devices. For pure tensor parallelism, the data axis dimension should be 1. The `mesh_axis_names` assign semantic meaning: the "model" axis indicates tensor parallel distribution. When `mesh_shape` is `null`, the system operates in single-device mode.

## Sharding Strategies

### Column-Row-Column Pattern

The most common tensor parallelism strategy in TT-Blacksmith is column-row-column sharding, used in the MNIST experiments. This pattern minimizes inter-device communication by strategically placing all-reduce operations.

**Column-Row-Column Sharding Strategy**

In this pattern:

*   **Layer 1 (Column Parallel)**: Weight columns are distributed, producing sharded activations
*   **Layer 2 (Row Parallel)**: Weight rows are distributed, requiring all-reduce to combine partial sums
*   **Layer 3 (Column Parallel)**: Weight columns are distributed again

This strategy places communication overhead at layer 2, avoiding all-reduce on the final layer which may have non-divisible output dimensions.

Sources: [blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml 57-61](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml#L57-L61)[blacksmith/experiments/jax/mnist/multi_chip/tensor_parallel/test_pure_jax_mnist.py 47-66](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/jax/mnist/multi_chip/tensor_parallel/test_pure_jax_mnist.py#L47-L66)

### Shard Map Operations

JAX tensor parallelism uses `shard_map.shard_map` to explicitly define how computations are distributed across the mesh. This functional approach maps a single-device computation across multiple devices:

The `in_specs` and `out_specs` arguments specify how each input and output tensor is distributed across the mesh. The lambda function represents the single-device computation that gets mapped across all devices.

## Optimizer Step Synchronization

The `DeviceManager.optimizer_step` method handles synchronization differently for single-device and multi-device configurations:

For tensor parallelism (multi-chip), `xm.optimizer_step` with `barrier=True` ensures:

*   All gradient computations complete before parameter updates
*   All-reduce operations synchronize gradients across devices
*   Parameter updates occur atomically across the mesh

## Key Implementation Differences

| Aspect | PyTorch | JAX |
| --- | --- | --- |
| **Sharding Definition** | Regex patterns in config + `xs.mark_sharding` | Explicit `PartitionSpec` in code |
| **Communication** | Implicit via XLA compiler | Explicit `lax.psum`, `lax.all_gather` |
| **Shard Application** | Called per training step via `shard_model()` | Set once during initialization |
| **Device Placement** | Automatic via `to(device)` | Explicit via `jax.device_put` with specs |
| **Loss Computation** | Custom loss with keepdim for shape compatibility | All-gather fragments then compute global loss |
