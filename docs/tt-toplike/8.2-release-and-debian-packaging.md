The `tt-toplike` release pipeline automates the generation of Debian packages for multiple Ubuntu distributions, manages GitHub Releases with race-condition-safe asset uploads, and synchronizes project metadata with the GitHub Pages site. The system is designed for reproducible, offline-compliant builds suitable for official distribution.

## Packaging Strategy

The project produces two primary Debian packages:

1.   **`tt-toplike`**: The core TUI (Terminal User Interface) binary and documentation [[debian/control:10-24]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
2.   **`tt-toplike-app`**: The native window host that wraps the TUI inside a PTY using `egui`[[debian/control:36-44]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### Offline Build and Vendoring

To comply with Debian build daemon requirements, the project uses a vendoring strategy to ensure network-free builds. The `build-deb.sh` script automates this by:

*   Running `cargo vendor vendor/` to download all dependencies locally [[build-deb.sh:73]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
*   Generating a `.cargo/config.toml` that redirects all crate lookups to the `vendor/` directory [[build-deb.sh:84-93]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
*   Enforcing the `--frozen` flag during compilation to prevent any network access [[debian/rules:49-56]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### Build Environment

The `debian/rules` file manages the build process via `debhelper`. It isolates the build environment by setting `CARGO_HOME` to a local directory within the `debian/` tree, preventing pollution of the user's home directory [[debian/rules:25-26]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

## Matrix Build and Suite Patching

The release workflow uses a GitHub Actions matrix to build binaries against specific `glibc` versions for different Ubuntu releases:

*   **Noble (24.04)**: Built on `ubuntu-24.04` runners [[.github/workflows/release.yml:32-33]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
*   **Jammy (22.04)**: Built on `ubuntu-22.04` runners to ensure compatibility with `libc6 >= 2.35`[[.github/workflows/release.yml:18-35]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### Transient Changelog Patching

Since `dpkg-buildpackage` uses the first entry in `debian/changelog` to determine the target suite, the workflow performs an in-place `sed` edit for non-noble builds. This changes the suite name (e.g., from `noble` to `jammy`) before the build starts [[.github/workflows/release.yml:52-54]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### Package Naming

To allow multiple suites to coexist in a single GitHub Release, packages are renamed after the build to include a suite suffix (e.g., `tt-toplike_0.6.2_amd64_noble.deb`) [[.github/workflows/release.yml:62-71]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

## Release Creation and Asset Upload

The release process is triggered by tags matching the `v*` pattern [[.github/workflows/release.yml:5-6]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### Race-Condition Handling

Because the matrix jobs run in parallel, they "race" to create the GitHub Release. The workflow uses `gh release view` to check for existence before attempting `gh release create`[[.github/workflows/release.yml:83-91]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging). If a job loses the race, it simply proceeds to upload its specific suite assets using the `--clobber` flag [[.github/workflows/release.yml:93-97]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

## Automated Project Site Updates

After the Debian packages are successfully built and uploaded, the `update-site` job synchronizes the project's landing page on the `gh-pages` branch [[.github/workflows/release.yml:100-104]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).

### gh-pages Synchronization

The job performs the following steps:

1.   **Checkout**: Checks out the `gh-pages` branch into a sibling directory [[.github/workflows/release.yml:111-115]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
2.   **Version Bump**: Uses `sed` to update the version badge in the `index.html` hero section [[.github/workflows/release.yml:124-126]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
3.   **Command Update**: Updates the `gh release download` example command to point to the correct repository and version [[.github/workflows/release.yml:130-131]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
4.   **Commit and Push**: If changes are detected, it commits as `github-actions[bot]` and pushes back to `gh-pages`[[.github/workflows/release.yml:133-143]](https://deepwiki.com/tenstorrent/tt-toplike/8.2-release-and-debian-packaging).
