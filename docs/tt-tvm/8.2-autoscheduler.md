## Purpose and Scope

AutoScheduler is TVM's **template-free automatic scheduling system** that generates optimized schedules for tensor programs without requiring manually-defined schedule templates. Unlike AutoTVM (see [AutoTVM](https://deepwiki.com/tenstorrent/tt-tvm/8.1-autotvm)), which relies on pre-written schedule templates with tunable knobs, AutoScheduler automatically constructs a search space from the compute declaration (ComputeDAG) and explores it to find high-performance schedules.

This page covers:

*   The AutoScheduler architecture and its key components
*   How search tasks are extracted from neural networks
*   The search and tuning process
*   How discovered schedules are stored and applied

For the advanced successor to AutoScheduler, see [Meta Schedule](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule). For the RPC infrastructure used during tuning, see [RPC Infrastructure](https://deepwiki.com/tenstorrent/tt-tvm/8.4-rpc-infrastructure).

## System Architecture

AutoScheduler consists of several interconnected components that work together to automate the scheduling process:

**Architecture Overview**

The AutoScheduler system follows a search-measure-update loop:

1.   **Task Extraction**: Neural network operators are partitioned into search tasks
2.   **Task Scheduling**: A scheduler allocates measurement trials across tasks
3.   **Search**: A search policy generates candidate schedules from the search space
4.   **Measurement**: Candidates are built and executed on target hardware
5.   **Update**: Results update the cost model and inform future searches
6.   **Application**: Best schedules are loaded during compilation

## Task Extraction

AutoScheduler begins by extracting **search tasks** from a Relay module. Each task represents a distinct computational pattern (e.g., a specific convolution configuration).

### Task Extraction Process

**Extraction Example**

`# Extract tasks from networktasks, task_weights = auto_scheduler.extract_tasks(    mod["main"],    # Relay function    params,         # Model parameters    target          # Compilation target) # Each task has a unique workload_keyfor idx, task in enumerate(tasks):    print(f"Task {idx}: {task.workload_key}")    print(task.compute_dag)  # ComputeDAG representation`
**Task Weight Calculation**

The weight of a task represents how many times its computational pattern appears in the network. This allows the scheduler to approximate end-to-end latency as:

```
end_to_end_latency ≈ Σ(latency[task] × weight[task])
```

Tasks with higher weights have greater impact on overall performance and receive proportionally more tuning trials.

## SearchTask and ComputeDAG

Each `SearchTask` encapsulates a computational workload with its target hardware configuration.

### SearchTask Components

| Component | Description |
| --- | --- |
| `workload_key` | Unique identifier derived from compute declaration |
| `compute_dag` | `ComputeDAG` representing the tensor computation |
| `target` | Target hardware (CPU, GPU, etc.) |
| `target_host` | Host target for heterogeneous execution |
| `hardware_params` | Hardware constraints (memory, threads, etc.) |

### ComputeDAG Structure

The `ComputeDAG` is the core representation of a computation in AutoScheduler. It captures:

*   **Tensor operations**: Compute, placeholder, reduction operations
*   **Data dependencies**: Producer-consumer relationships between operations
*   **Access patterns**: How data is read and written
*   **Reduction structure**: Information about reduction axes

**Example ComputeDAG**

For a simple matrix multiplication:

`n = te.var("n")A = te.placeholder((n, n), name="A")B = te.placeholder((n, n), name="B")k = te.reduce_axis((0, n), name="k")C = te.compute((n, n), lambda i, j: te.sum(A[i, k] * B[k, j], axis=k), name="C") # This creates a ComputeDAG with:# - 2 placeholder operations (A, B)# - 1 compute operation (C) with a reduction axis (k)`

## TaskScheduler

The `TaskScheduler` is responsible for allocating measurement trials across multiple tasks to optimize overall network performance.

### Scheduling Strategy

**Scheduling Algorithm**

The scheduler uses a gradient-based approach:

1.   **Predict Impact**: Estimate how tuning each task would reduce overall latency
2.   **Allocate Trials**: Prioritize tasks with highest predicted impact
3.   **Round-Robin**: Cycle through tasks, allocating small batches of trials
4.   **Early Stopping**: Stop tuning tasks that show no improvement

**TaskScheduler Usage**

`tuner = auto_scheduler.TaskScheduler(tasks, task_weights) # Optional: resume from previous logtuner = auto_scheduler.TaskScheduler(    tasks,     task_weights,    load_log_file="previous_log.json"  # Resume search) tuner.tune(tune_option)`

## Search Policy and Search Space

The **SearchPolicy** defines how the search space is explored to find optimal schedules.

### SketchPolicy

The default `SketchPolicy` uses a two-level hierarchical search:

**Search Space Components**

| Component | Description |
| --- | --- |
| **Sketches** | High-level schedule structures (e.g., loop tiling strategy) |
| **Tile Sizes** | Dimensions for multi-level loop tiling |
| **Parallelization** | Thread/block binding for parallel execution |
| **Vectorization** | SIMD vector width configuration |
| **Unrolling** | Loop unroll factors |
| **Cache Locality** | Data reuse through cache stages |

**Evolution Process**

1.   **Random Sampling**: Generate initial population of sketches
2.   **Evolutionary Search**: Mutate top-performing candidates
3.   **Cost Model Guidance**: Use XGBoost to predict costs without measurement
4.   **Measurement**: Evaluate best candidates on real hardware
5.   **Model Update**: Retrain cost model with new measurements

## Tuning Options and Measurement

The tuning process is configured through `TuningOptions` which specifies builders, runners, and callbacks.

### TuningOptions Configuration

**Tuning Configuration Example**

`tune_option = auto_scheduler.TuningOptions(    num_measure_trials=20000,  # Total measurements to perform        # Builder: compiles generated code    builder=auto_scheduler.LocalBuilder(        build_func="ndk"  # Use Android NDK for cross-compilation    ),        # Runner: executes code and measures performance    runner=auto_scheduler.RPCRunner(        device_key,              # RPC device identifier        host=rpc_host,           # RPC tracker host        port=rpc_port,           # RPC tracker port        timeout=30,              # Timeout per measurement        repeat=1,                # Number of times to run        min_repeat_ms=200,       # Minimum repeat time        enable_cpu_cache_flush=True  # Flush CPU cache between runs    ),        # Callbacks: called after each batch of measurements    measure_callbacks=[        auto_scheduler.RecordToFile(log_file)  # Save results    ])`
### Builder System

The **Builder** compiles generated schedules into executable code:

| Builder | Description | Use Case |
| --- | --- | --- |
| `LocalBuilder` | Compiles on local machine | Same-architecture tuning |
| `build_func='default'` | Standard C++ compiler | Native compilation |
| `build_func='ndk'` | Android NDK toolchain | Cross-compilation for Android |

### Runner System

The **Runner** executes compiled code and measures performance:

| Runner | Description | Use Case |
| --- | --- | --- |
| `LocalRunner` | Runs on local machine | CPU/GPU on same machine |
| `RPCRunner` | Runs on remote device via RPC | Mobile devices, embedded systems |

**RPCRunner Parameters**

*   `device_key`: Identifier registered with RPC tracker
*   `timeout`: Maximum time allowed per measurement
*   `repeat`: Number of execution repeats for averaging
*   `min_repeat_ms`: Minimum total execution time for stable measurements
*   `enable_cpu_cache_flush`: Flush caches for accurate CPU measurements

## Measurement Workflow

The measurement process transforms candidate schedules into performance data:

**Error Handling**

The AutoScheduler gracefully handles measurement failures:

*   **Compilation errors**: Invalid schedules that fail to compile
*   **Runtime errors**: Segfaults, assertion failures
*   **Timeouts**: Schedules that take too long to execute
*   **Wrong results**: Schedules that produce incorrect output (detected by verification)

Invalid results are recorded with error codes and excluded from cost model training.

## Log File Format and Management

AutoScheduler stores measurement records in JSON format for persistence and analysis.

### Log File Structure

Each line in the log file is a JSON record containing:

`{  "input": {    "task": {      "workload_key": "conv2d_nchw.cuda",      "target": "cuda -keys=cuda,gpu -max_num_threads=1024"    },    "state": {      "stages": [...],      "transform_steps": [...]    }  },  "result": {    "costs": [0.001234, 0.001245, 0.001230],    "error_no": 0,    "all_cost": 0.15,    "timestamp": 1634567890.123  }}`
### Log File Operations

**Distilling Log Files**

Large log files can be compressed by keeping only the best record for each task:

`python3 -m tvm.auto_scheduler.measure_record --mode distill -i log.json`
This removes inferior results, reducing file size while preserving the best schedules.

**Resuming Tuning**

AutoScheduler supports resuming from a previous log file:

`tuner = auto_scheduler.TaskScheduler(    tasks,     task_weights,    load_log_file="previous_log.json"  # Resume from here)`
The scheduler skips tasks that already have sufficient measurements and continues from where it left off.

## Integration with Relay Compilation

AutoScheduler integrates with Relay's compilation pipeline through the `ApplyHistoryBest` context manager:

### Compilation with AutoScheduler

**Compilation Example**

`# Load best schedules from tuning logwith auto_scheduler.ApplyHistoryBest(log_file):    # Configure compilation to use AutoScheduler schedules    with tvm.transform.PassContext(        opt_level=3,        config={"relay.backend.use_auto_scheduler": True}    ):        # Build the module with optimal schedules        lib = relay.build(mod, target=target, params=params)`
**How It Works**

1.   `ApplyHistoryBest` loads the log file and builds a workload_key → schedule mapping
2.   During `relay.build()`, when the TE compiler encounters an operator: 
    *   It computes the workload_key for that operator
    *   It looks up the best schedule in the loaded mapping
    *   If found, it applies that schedule instead of using a default template

3.   The resulting library contains code generated from the tuned schedules

**Configuration Flag**

The `relay.backend.use_auto_scheduler` flag tells Relay to:

*   Disable AutoTVM-style template lookup
*   Enable AutoScheduler schedule application
*   Use auto-scheduler-compatible layout conversion

## Task Scheduler Output

During tuning, the TaskScheduler periodically prints status tables showing progress:

### Status Table Format

```
----------------------------------------------------------------------
------------------------------  [ Task Scheduler ]
----------------------------------------------------------------------
|  ID  | Latency (ms) | Speed (GFLOPS) | Trials |
-------------------------------------------------
|    0 |        0.013 |           0.31 |     64 |
|    1 |        0.845 |           2.43 |    448 |
|    2 |        0.046 |          -0.00 |     64 |
|    3 |        4.194 |          24.53 |   2112 |
...
-------------------------------------------------
Estimated total latency: 38.347 ms    Trials: 19992    Used time: 19260 s    Next ID: 3
```

**Table Columns**

| Column | Description |
| --- | --- |
| `ID` | Task index in the task list |
| `Latency (ms)` | Best measured latency for this task |
| `Speed (GFLOPS)` | Computed GFLOPS based on FLOPs and latency |
| `Trials` | Number of measurements allocated to this task |

**Summary Line**

*   **Estimated total latency**: Weighted sum of task latencies (approximates end-to-end time)
*   **Trials**: Total measurements performed so far
*   **Used time**: Wall-clock time spent tuning
*   **Next ID**: The task that will be tuned next

This output helps monitor tuning progress and identify which tasks need more optimization.

## Practical Usage Tips

### Recommended Trial Budget

For production tuning, set `num_measure_trials` based on the number of tasks:

`num_measure_trials = 800 * len(tasks)`
For a typical ResNet-50 (29 tasks), this gives approximately 23,200 trials, which is usually sufficient for convergence.

### Layout Recommendations

AutoScheduler performs best with **NHWC layout** for CNNs on CPU/GPU:

`# Convert model to NHWC layoutfrom tvm.relay.transform import ConvertLayout with tvm.transform.PassContext(opt_level=3):    mod = ConvertLayout({"nn.conv2d": ["NHWC", "default"]})(mod)`
NHWC layout enables better optimization because:

*   Vectorization along channel dimension is more efficient
*   Cache locality is improved for common CNN patterns
*   AutoScheduler has more specialized rules for NHWC

### Distributed Tuning

For faster tuning, use multiple devices with RPC:

`# Start RPC tracker# python3 -m tvm.exec.rpc_tracker --host=0.0.0.0 --port=9190 # Register multiple devices with same key# Device 1: python3 -m tvm.exec.rpc_server --tracker=HOST:9190 --key=device# Device 2: python3 -m tvm.exec.rpc_server --tracker=HOST:9190 --key=device# ... runner = auto_scheduler.RPCRunner(    "device",  # All devices with this key will be used    host="127.0.0.1",    port=9190,    # Measurements will be distributed across all registered devices)`
This parallelizes measurements across multiple devices, significantly reducing tuning time.

### Early Termination

AutoScheduler can be safely terminated early (Ctrl+C) as long as:

*   At least one valid schedule exists for each task in the log file
*   The log file is not corrupted

You can then compile with the partial results:

`with auto_scheduler.ApplyHistoryBest(log_file):    lib = relay.build(mod, target=target, params=params)`
Tasks without measurements will fall back to default schedules.

## Comparison with AutoTVM

| Aspect | AutoTVM | AutoScheduler |
| --- | --- | --- |
| **Search Space** | Template-based with tunable knobs | Generated from ComputeDAG automatically |
| **Template Requirement** | Requires manual schedule templates | No templates needed, uses only compute declarations |
| **Extensibility** | Adding new ops requires writing templates | New ops work automatically if TOPI compute exists |
| **Search Strategy** | XGBoost cost model + genetic algorithm | Sketch-based hierarchical search + cost model |
| **Hardware Modeling** | Template author encodes hardware knowledge | Hardware params + automatic feature extraction |
| **Typical Performance** | Depends on template quality | Often matches or exceeds hand-tuned templates |
| **Maintenance Burden** | High (templates need updating for new hardware) | Low (generic search works across hardware) |

AutoScheduler represents a significant step toward fully automated optimization, reducing the need for domain expertise in schedule writing.

For the template-based AutoTVM system, see [AutoTVM](https://deepwiki.com/tenstorrent/tt-tvm/8.1-autotvm). For the more advanced Meta Schedule system, see [Meta Schedule](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule).
