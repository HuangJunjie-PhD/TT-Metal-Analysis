# Wormhole Platform

This page documents tt-flash support for the Wormhole chip architecture, including the `WhChip` implementation class, supported board variants, firmware version retrieval mechanisms, and platform-specific reset procedures. For Blackhole platform support, see [4.4](https://deepwiki.com/tenstorrent/tt-flash/4.4-blackhole-platform). For board identification and UPI mapping details, see [4.2](https://deepwiki.com/tenstorrent/tt-flash/4.2-board-identification).

## Overview

Wormhole is one of two currently supported Tenstorrent AI accelerator architectures in tt-flash. The platform includes several board variants designed for different use cases, from development boards to large-scale Galaxy systems. All Wormhole chips share a common firmware interface based on ARC messaging but have platform-specific requirements for reset procedures and version detection.

**Sources**: [README.md](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1)[tt_flash/chip.py 297-305](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L297-L305)

## WhChip Class Implementation

The `WhChip` class extends the abstract `TTChip` base class to provide Wormhole-specific implementations. Key characteristics:

| Method | Return Value | Description |
| --- | --- | --- |
| `min_fw_version()` | `0x2170000` | Minimum firmware version required for bundle support |
| `get_bundle_version()` | `FwVersion` | Delegates to `get_bundle_version_v1()` legacy method |
| `__repr__()` | `"Wormhole[{interface_id}]"` | String representation showing PCI interface ID |

The class uses firmware definitions loaded from `data/wormhole/fw_defines.yaml` to communicate with the chip's ARC processor.

**Sources**: [tt_flash/chip.py 297-305](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L297-L305)[tt_flash/chip.py 113-119](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L113-L119)

## Supported Board Variants

Wormhole platform supports six distinct board variants, differentiated by their UPI (Unique Part Identifier) values extracted from the 64-bit Board ID:

| Board Codename | UPI | Revision Check | Public Display Name | Reset Method |
| --- | --- | --- | --- | --- |
| `WH_UBB` | `0x35` | N/A | "Galaxy Wormhole" | IPMI |
| `E300_R2` | `0x1` | `0x2` | "e150" | Standard |
| `E300_R3` | `0x1` | `0x3` or `0x4` | "e150" | Standard |
| `E75` | `0x7` | N/A | "e75" | Standard |
| `E300_105` | `0x3` | N/A | "e150" | Standard |
| `E300_X2` | `0xA` | N/A | TBD | Standard |

The E300 series requires revision checking to distinguish between R2 and R3 variants. Board type resolution occurs in `get_board_type()` which maps the Board ID to an internal codename, later converted to user-friendly names via `change_to_public_name()`.

## Firmware Version Retrieval

Wormhole uses a legacy version retrieval method (`get_bundle_version_v1`) that queries the ARC processor with different `arg0` values to retrieve distinct version information:

### MSG_TYPE_FW_VERSION Query Protocol

| arg0 Value | Purpose | Return Format |
| --- | --- | --- |
| `0` | Base firmware version | Single 32-bit version number |
| `1` | Running bundle version | Packed: `(component << 24) | (major << 16) | (minor << 8) | patch` |
| `2` | SPI-stored bundle version | Packed: `(component << 24) | (major << 16) | (minor << 8) | patch` |

The function returns an `FwVersion` object with `allow_exception=True`, permitting graceful degradation for very old firmware versions that pre-date bundle support (<`0x2170000`). In such cases, the exception is captured but the flash process can continue with `--force` flag.

Special handling at [tt_flash/chip.py 64-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L64-L69) guards against firmware versions that incorrectly return the same value for running and SPI versions, using this mismatch detection to skip invalid SPI version queries.

## Reset Procedures

### Galaxy System Reset (WH_UBB)

Galaxy systems with WH_UBB boards require specialized IPMI-based reset procedures implemented in `glx_6u_trays_reset()`:

**Reset Parameters**:

*   `ubb_num`: UBB bitmap (default `0xF` = all 4 UBBs)
*   `dev_num`: Device bitmap (default `0xFF` = all 8 devices per UBB)
*   `op_mode`: `0x0` = assert/deassert with period, `0x1` = assert only, `0x2` = deassert only
*   `reset_time`: Duration in 10ms units (default `0xF` = 150ms)

**Post-Reset Validation**:

After reset, the system validates Ethernet connectivity by reading the debug buffer at offset `0x4*96` from address `0x12c0` for each of 16 Ethernet ports across 32 devices. A value of `10` (LINK_INACTIVE_FAIL_DUMMY_PACKET) indicates link failure. Full Galaxy resets (`ubb_num=0xF`) require complete link validation; partial resets skip this check as Ethernet training becomes desynchronized.

**Sources**: [tt_flash/flash.py 738-789](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L738-L789)[tt_flash/flash.py 700-736](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L700-L736)[tt_flash/flash.py 922-925](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L922-L925)

### Standard Wormhole Reset

Non-Galaxy Wormhole boards use the `WHChipReset.full_lds_reset()` method from `tt_tools_common`. The reset procedure includes:

**Adaptive M3 Boot Timeouts**:

The system dynamically adjusts post-reset wait times based on firmware version changes:

*   **Standard**: 20 seconds for same-major-version updates
*   **Extended**: 60 seconds when crossing major version boundaries (detected at [tt_flash/flash.py 894-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L894-L900))

**Reset Conditions**:

Standard WH boards are flagged for reset (`can_reset=True`) by default at [tt_flash/flash.py 484-485](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L484-L485) Reset is skipped if:

*   The `--no-reset` flag is present
*   Errors occurred during the flash stage (`rc != 0`)
*   Board is WH_UBB (handled by Galaxy reset path)

**Sources**: [tt_flash/flash.py 927-936](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L927-L936)[tt_flash/flash.py 890-900](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L890-L900)[tt_flash/flash.py 482-485](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L482-L485)

## ARC Message Types

Wormhole firmware communication uses the following ARC message types defined in `fw_defines.yaml`:

| Message Constant | Value | Purpose |
| --- | --- | --- |
| `MSG_TYPE_ARC_STATE0` | `0xa0` | Query ARC state (legacy) |
| `MSG_TYPE_ARC_STATE3` | `0xa3` | Query ARC state with extended info |
| `MSG_TYPE_FW_VERSION` | `0xb9` | Retrieve firmware version (with arg0 variants) |
| `MSG_TRIGGER_SPI_COPY_LtoR` | `0x50` | Trigger left-to-right SPI copy (unused on WH) |

**ARC State Pre-Check**:

Before version detection, the flash process sends `MSG_TYPE_ARC_STATE3` with a 0.1s timeout at [tt_flash/flash.py 215-221](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L215-L221) This pre-check is permitted to timeout without failure, serving as a communication warm-up.

**Version Query Implementation**:

The `MSG_TYPE_FW_VERSION` message uses a three-call pattern with different `arg0` values:

1.   Base version check (arg0=0)
2.   Running bundle version (arg0=1)
3.   SPI bundle version (arg0=2)

Each call returns a 32-bit packed integer decoded at [tt_flash/chip.py 57-62](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L57-L62) as `(component, major, minor, patch)` tuples.

## Minimum Firmware Requirements

Wormhole requires minimum firmware version `0x2170000` for bundle support, defined at [tt_flash/chip.py 298-299](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L298-L299) Firmware versions below this threshold:

*   Cannot provide bundle version information via `MSG_TYPE_FW_VERSION` with arg0 > 0
*   Return `FwVersion` with `exception` populated and `allow_exception=True`
*   Require `--force` flag to proceed with flash operation
*   Fall back to assuming update is needed (safe default for very old firmware)

The version check occurs at [tt_flash/chip.py 49](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L49-L49) and determines whether the legacy firmware supports the bundle query protocol introduced in later versions.
