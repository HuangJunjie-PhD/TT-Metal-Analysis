This section documents the suite of tools used to analyze the emulation surface of `tt-emule`, classify existing Tenstorrent kernels against that surface, and maintain the authoritative symbol index of the codebase. These tools bridge the gap between the "Natural Language Space" (architectural layers) and the "Code Entity Space" (specific headers and symbols).

## Overview of the Analysis Pipeline

The analysis infrastructure ensures that developers can determine whether a kernel is supported by `tt-emule` without needing to attempt a full build. It relies on a deterministic, pattern-matching approach rather than LLM-based heuristics.

## Kernel Classification (`classify_kernels.py`)

The `classify_kernels.py` script is a deterministic layer-1 classifier. It evaluates Tenstorrent kernels (typically `.cpp` files from `tt-metal` or `tt-mlir`) against the `tt-emule` surface to determine if they can be emulated.

### Classification Layers

The tool categorizes code into the following abstraction stack:

| Layer | Name | Description |
| --- | --- | --- |
| **Layer 1** | PoR API | High-level tile-op API (`add_tiles`, `cb_push_back`). |
| **Layer 1.5** | Config Capture | Transitional state capture (e.g., PACK sub-rect configs). |
| **Layer 2** | LLK Calls | Low-level kernel calls just below the PoR wrappers. |
| **Layer 3** | HW Model | Raw hardware instructions (`TTI_*`), SFPI vectors, or DST indices. |

### Verdicts and Buckets

Each kernel is assigned a verdict based on its deepest reach into the hardware stack:

1.   **`layer1`**: Fully modelable by `tt-emule` today.
2.   **`needs_stub`**: Uses a PoR-style `*_tile` function that exists in silicon but lacks a shim in `include/jit_hw/api/compute/`.
3.   **`ruled_out`**: Reaches Layer 2 or 3 in ways `tt-emule` cannot (or chooses not to) model.

**Rule-out Buckets (Ordered by difficulty):**

*   `sfpi_intrinsics` (Layer 3): Hand-written SFPU vector code.
*   `dst_register` (Layer 3): Raw `dst_reg[]` indexing.
*   `hw_instructions` (Layer 3): `TTI_*` macros or MOP runs.
*   `llk_headers` (Layer 2): Reaches a real LLK header tree (e.g., `/tt-llk/`).
*   `unshadowed_llk_call` (Layer 2): Calls an `llk_*` function that `tt-emule` does not yet shim.

## Surface Probing (`find_symbol.py`&`emule_surface.py`)

`emule_surface.py` acts as the shared denominator for what `jit_hw` provides. It is consumed by both the classifier and the developer-facing `find_symbol.py`.

### Symbol Detection Logic

The `Surface` class in `emule_surface.py` loads `structure.yaml` and categorizes symbols using regex patterns:

*   **Tile Symbols**: `^[a-z][a-z0-9_]*_tiles?$`[scripts/emule_surface.py 44](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/emule_surface.py#L44-L44)
*   **LLK Symbols**: `^_?llk_[a-z]\w*$`[scripts/emule_surface.py 45](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/emule_surface.py#L45-L45)

### `find_symbol.py` Probes

Developers use this tool to query the emulation surface:

*   `--supports <sym>`: Checks if a symbol is modelable (Layer 1) or requires a stub.
*   `--shadows <include>`: Checks if `tt-emule` provides a JIT shadow for a specific header (e.g., `compute_kernel_api/common.h`).
*   `--exit-status`: Returns non-zero if the symbol is not supported, allowing for CI/shell branching.

Sources: [scripts/emule_surface.py 48-129](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/emule_surface.py#L48-L129)[scripts/find_symbol.py 107-123](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/find_symbol.py#L107-L123)

## Structure Maintenance (`gen_structure.py`)

The `structure.yaml` file is the authoritative index of the `tt-emule` codebase. It is maintained by `gen_structure.py`, which is typically run as a pre-commit hook or CI gate.

### Implementation Details

*   **Symbol Extraction**: Uses a C++ declaration scanner (`extract_symbols`) to find namespaces, classes, functions, and macros [scripts/gen_structure.py 132-150](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/gen_structure.py#L132-L150)
*   **Comment Stripping**: Removes block and line comments before scanning to avoid false positives [scripts/gen_structure.py 77-81](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/gen_structure.py#L77-L81)
*   **Summary Preservation**: While symbols are auto-generated, the `summary` field for each file is editorial prose and must be manually maintained [scripts/gen_structure.py 19-22](https://github.com/tenstorrent/tt-emule/blob/a57cf072/scripts/gen_structure.py#L19-L22)

### CI/CD Integration

The tool supports a `--check` mode that performs a unified diff between the current `structure.yaml` and the actual state of the source tree. If they differ, the CI "Structure Index" gate fails, ensuring the documentation and classification logic never drift from the implementation.
