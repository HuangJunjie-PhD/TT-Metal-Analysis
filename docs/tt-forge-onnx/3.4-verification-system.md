This page describes the verification subsystem, which runs both the original framework model and the compiled Tenstorrent model on the same inputs and compares their outputs. This is distinct from intermediate graph verification during compilation (see page [3.1](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/3.1)); the focus here is on the post-compilation, end-to-end output comparison performed via the public `verify()` API.

For details on how model metadata is recorded during a test, see page [3.5](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/3.5)

* * *

## Overview

After a model is compiled with `forge.compile()`, the verification system confirms that the compiled model on Tenstorrent hardware produces numerically equivalent results to the original framework model running on CPU. Verification is the final step in the standard model test pattern.

The system is composed of:

| Component | Location | Role |
| --- | --- | --- |
| `verify()` | `forge/forge/verify/verify.py` | Public entry point for full verification |
| `_verify_backward()` | `forge/forge/verify/verify.py` | Backward pass gradient verification |
| `do_verify()` | `forge/forge/verify/verify.py` | Intermediate graph verification (used internally by the compiler pipeline) |
| `VerifyConfig` | `forge/forge/verify/config.py` | Configuration for all verification checks |
| `DeprecatedVerifyConfig` | `forge/forge/verify/config.py` | Legacy config used by `do_verify()` during intermediate compilation stages |
| `ValueChecker` subclasses | `forge/forge/verify/value_checkers.py` | Pluggable numeric comparison strategies |
| `compare_tensor_to_golden()` | `forge/forge/verify/compare.py` | Low-level tensor comparison |

## The `verify()` Function

`verify()`[forge/forge/verify/verify.py 420-560](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L420-L560) is the primary public API. It accepts the original framework model, the compiled model, and the input tensors. It runs both models and compares all outputs.

**Signature:**

**Returns:**`(framework_outputs, compiled_outputs)` — a tuple of tensor lists, or `(None, None)` if verification is disabled.

## VerifyConfig

`VerifyConfig`[forge/forge/verify/config.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/config.py) controls which checks are performed and how values are compared. All fields have sensible defaults so that `verify(inputs, fw_model, co_model)` works out of the box.

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `enabled` | `bool` | `True` | If `False`, skips all checks and returns outputs without asserting |
| `verify_size` | `bool` | `True` | Assert that output count from both models is equal |
| `verify_dtype` | `bool` | `True` | Assert dtype equivalence (accounts for Forge dtype narrowing) |
| `verify_shape` | `bool` | `True` | Assert tensor shapes match |
| `verify_values` | `bool` | `True` | Run numeric comparison via `value_checker` |
| `value_checker` | `ValueChecker` | `AutomaticValueChecker()` | Pluggable numeric comparison strategy |
| `verify_emitc_correctness` | `bool` | `False` | Also export model to a `.so` and run it via `testutils.test_so()` |
| `supported_tensor_types` | tuple | (torch.Tensor, tf.Tensor, ...) | Accepted input tensor types |
| `framework_model_types` | tuple | Module types | Accepted framework model types |
| `compiled_model_types` | tuple | `CompiledModel` | Accepted compiled model types |

The `ModelGroup.RED` flag in `verify()` automatically enables `verify_emitc_correctness`[forge/forge/verify/verify.py 447-449](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L447-L449)

### `DeprecatedVerifyConfig`

`DeprecatedVerifyConfig`[forge/forge/verify/config.py 92](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/config.py#L92-LNaN) is used internally by `do_verify()` during the compilation pipeline's intermediate graph verification stages. It has different fields oriented toward graph-level checking:

| Field | Description |
| --- | --- |
| `pcc` | Pearson correlation coefficient threshold |
| `relative_atol` | atol as fraction of max tensor value |
| `rtol`, `atol` | Per-data-format tolerances |
| `dump_tensors_path` | Optional path to dump tensors for inspection |

Sources: [forge/forge/verify/config.py](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/config.py)[forge/forge/verify/verify.py 447-449](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L447-L449)

## Value Checkers

The `value_checker` field of `VerifyConfig` accepts any subclass of `ValueChecker` from `forge/forge/verify/value_checkers.py`. The checker's `.check(golden, actual)` method is called for each pair of output tensors.

### `AutomaticValueChecker`

The default checker. Uses the Pearson Correlation Coefficient (PCC) to measure linear correlation between the golden and actual tensors. A `pcc` of `1.0` indicates exact correlation.

Example usage in tests:

Sources: [forge/forge/verify/verify.py 542-543](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L542-L543)[forge/test/models/jax/vision/resnet/test_resnet.py 46-51](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/models/jax/vision/resnet/test_resnet.py#L46-L51)

### `AllCloseValueChecker`

Uses `torch.allclose()` with configurable `rtol` and `atol`. Appropriate when the expected difference is bounded, not just correlated.

### `FullValueChecker`

Combines both PCC and `allclose`. Both conditions must pass. Use when both correlation and absolute closeness are required.

## Dtype Handling

`check_dtypes()`[forge/forge/verify/verify.py 274-301](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L274-L301) accounts for Forge's supported dtype set. Because Forge may internally narrow certain types (e.g. `torch.int64` → `torch.int32`, `torch.bool` → `torch.int32`), the check first round-trips the framework dtype through `pytorch_dtype_to_forge_dataformat()` → `forge_dataformat_to_pytorch_dtype()` before comparing to the compiled model output dtype. This avoids false mismatches due to expected narrowing.

Sources: [forge/forge/verify/verify.py 274-301](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L274-L301)[forge/forge/tensor.py 529-603](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tensor.py#L529-L603)

## Backward Pass Verification

When `with_backward=True` is passed to `verify()`, the private `_verify_backward()` function [forge/forge/verify/verify.py 304-418](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L304-L418) is called after forward verification. It:

1.   Runs `compiled_model.backward()` to get compiled gradient outputs.
2.   Runs `framework_output.backward(gradient=output_grad)` to get framework gradients.
3.   Compares gradient tensors by name (matching `grad_acc_*` and `output_grad_*` prefixes).
4.   Applies the same `verify_dtype`, `verify_shape`, and `verify_values` checks.

This is used when the model is compiled for training (`enable_training=True` in `CompilerConfig`).

Sources: [forge/forge/verify/verify.py 304-418](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L304-L418)

## Execution Stage Tracking

`verify()` updates the `ExecutionStage` enumeration (defined in `forge/forge/forge_property_utils.py`) at key points during its execution. This allows the property reporting system (see page [3.5](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/3.5)) to capture how far execution progressed.

**Diagram: `ExecutionStage` state transitions inside `verify()`**

These state transitions are recorded via `record_execution(ExecutionStage.X)` calls [forge/forge/verify/verify.py 480-557](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L480-L557) The current `ExecutionStage` is stored in `Tags.execution_stage` inside `ForgePropertyStore`, which is later read by the CI reporting layer.

Additionally, `record_consistency_limits(fw_out, co_out)`[forge/forge/verify/verify.py 517](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L517-L517) records the actual computed `pcc`, `atol`, and `rtol` values into `Tags.pcc`, `Tags.atol`, `Tags.rtol` for later analysis.

Sources: [forge/forge/verify/verify.py 480-557](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L480-L557)[forge/forge/forge_property_utils.py 279-348](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/forge_property_utils.py#L279-L348)

### Custom PCC Thresholds

Different model types use different PCC thresholds. A threshold of `0.99` is common for well-supported float32 models; `0.95` is used for models with bfloat16 or reduced-precision pipelines.

### Disabling Verification

Set `VerifyConfig(enabled=False)` to run both models and return outputs without any assertions. This is useful for benchmarking or when only runtime correctness (not numeric correctness) is being tested.

## `do_verify()` — Intermediate Graph Verification

`do_verify()`[forge/forge/verify/verify.py 120-253](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L120-L253) is distinct from `verify()`. It is called internally by the compilation pipeline (`forge/forge/compile.py`) to compare intermediate graph states at various `CompileDepth` stages against PyTorch golden outputs. It uses `DeprecatedVerifyConfig` and `compare_tensor_to_golden()` rather than the public `VerifyConfig`/`ValueChecker` interface. End users do not call this function directly.

Sources: [forge/forge/verify/verify.py 120-253](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/verify/verify.py#L120-L253)[forge/forge/compile.py 305-328](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/compile.py#L305-L328)
