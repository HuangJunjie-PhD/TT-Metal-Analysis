## Purpose and Scope

This document describes how the CI/CD pipeline executes tests for each hardware architecture (Wormhole B0, Blackhole, Quasar) with architecture-specific configurations, runners, and execution strategies. Each architecture has unique hardware characteristics, test requirements, and runner environments that require tailored execution workflows.

For information about the overall pipeline structure and job dependencies, see [GitHub Actions Pipeline Overview](https://deepwiki.com/tenstorrent/tt-llk/10.1-github-actions-pipeline-overview). For details on change detection that triggers these workflows, see [on-pr.yml Workflow and Change Detection](https://deepwiki.com/tenstorrent/tt-llk/10.2-on-pr.yml-workflow-and-change-detection). For the generic compilation and execution patterns, see [setup-and-test.yml: Compilation and Execution](https://deepwiki.com/tenstorrent/tt-llk/10.3-setup-and-test.yml:-compilation-and-execution-phases).

## Architecture-Specific Workflow Jobs

The CI pipeline defines separate workflow jobs for each hardware architecture, invoking either `setup-and-test.yml` or `build-quasar.yml` with architecture-specific parameters.

### Wormhole Test Execution

**Wormhole Test Execution Configuration**

The `setup-and-test-wormhole` job executes Wormhole-specific tests with the following configuration:

| Parameter | Value | Purpose |
| --- | --- | --- |
| `docker_image` | From `build-images` output | Container environment with SFPI toolchain |
| `runs_on` | `tt-ubuntu-2204-n150-stable` | Self-hosted runner with N150 Wormhole hardware |
| `test_splits` | `2` | Parallel test execution across 2 groups |
| `pytest_markers` | `"not perf and not nightly and not quasar"` | Exclude performance, nightly, and Quasar tests |
| `use_durations` | `true` | Enable duration-based test balancing |

The job conditionally executes when:

*   `needs.detect-changes.outputs.wormhole == 'true'` - Changes in `tt_llk_wormhole_b0/**`
*   `needs.detect-changes.outputs.tests == 'true'` - Changes in `tests/**` (excluding Quasar)
*   `needs.detect-changes.outputs.github == 'true'` - Changes in `.github/**`
*   Event is not a push to main (`github.event_name != 'push'`)

**Sources**: [.github/workflows/on-pr.yml 144-158](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L144-L158)

### Quasar Build Workflow

Quasar follows a different execution pattern: compilation-only without hardware execution. This is implemented through a separate workflow file.

**Quasar Compilation Configuration**

The `build-quasar` workflow invokes `build-quasar.yml` with Quasar-specific settings:

| Parameter | Value | Purpose |
| --- | --- | --- |
| `docker_image` | From `build-images` output | Container with SFPI toolchain |
| `runs_on` | `tt-ubuntu-2204-xlarge-stable` | Large CPU runner for parallel compilation |
| `test_splits` | `1` | Single compilation group |
| `pytest_markers` | `"quasar"` | Only tests marked with `quasar` |

**Compilation Execution Strategy**

Quasar tests use `pytest --compile-producer` mode without hardware execution:

`pytest -q --compile-producer -n 40 \      -m "quasar" \      --splits $TEST_SPLITS --group $matrix.test_group \      --override-ini="addopts=-v" --tb=short --timeout=60 \      --junitxml=pytest-report-quasar-compile.xml .`
Key aspects:

*   `-n 40`: 40 parallel pytest workers for compilation
*   `--compile-producer`: Compile-only mode, no execution
*   `CHIP_ARCH=quasar`: Environment variable set before compilation
*   `--timeout=60`: 60-second timeout per test
*   Conditional execution only when `needs.detect-changes.outputs.quasar == 'true'`

**Sources**: [.github/workflows/on-pr.yml 174-185](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L174-L185)[.github/workflows/build-quasar.yml 1-107](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/build-quasar.yml#L1-L107)

## Runner Configurations and Hardware Mapping

Each architecture maps to a specific self-hosted runner with corresponding hardware:

**Runner-to-Hardware Mapping Table**

| Runner Label | Hardware | Architecture | Execution Capability | Typical Use |
| --- | --- | --- | --- | --- |
| `tt-ubuntu-2204-n150-stable` | N150 | Wormhole B0 | Full execution on hardware | Wormhole tests with device interaction |
| `tt-ubuntu-2204-p150b-stable` | P150B | Blackhole | Full execution on hardware | Blackhole tests with device interaction |
| `tt-ubuntu-2204-xlarge-stable` | CPU only | Quasar | Compilation only | Quasar compilation without hardware |

The `CHIP_ARCH` environment variable determines which architecture-specific headers and libraries are used during compilation. This is set in the test environment setup or workflow steps.

## Pytest Marker Filtering

Each architecture-specific job uses pytest marker expressions to select appropriate tests:

**Standard Test Execution (Wormhole and Blackhole)**

```
pytest_markers: "not perf and not nightly and not quasar"
```

This marker expression:

*   Excludes performance tests (`not perf`) - handled separately in `perf-tests` job
*   Excludes nightly tests (`not nightly`) - for scheduled long-running tests
*   Excludes Quasar tests (`not quasar`) - different execution model

**Quasar Build Workflow**

```
pytest_markers: "quasar"
```

This marker expression:

*   Includes only tests marked with `@pytest.mark.quasar`
*   These tests are designed for compilation-only validation

**Performance Test Execution**

```
pytest_markers: "perf"
```

Performance tests use a separate matrix strategy executing on both architectures:

| Matrix Item | Runner | Test Splits |
| --- | --- | --- |
| Wormhole | `tt-ubuntu-2204-n150-stable` | 5 |
| Blackhole | `tt-ubuntu-2204-p150b-stable` | 5 |

Performance tests have:

*   `test_splits: 5` - More parallelization for longer-running tests
*   `timeout_minutes: 240` - Extended timeout (4 hours vs default)
*   `use_durations: true` - Duration-based balancing

**Sources**: [.github/workflows/on-pr.yml 156-205](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/on-pr.yml#L156-L205)

## Test Splitting and Parallelization Strategy

**Test Split Configuration by Architecture**

| Job | Architecture | Test Splits | Rationale |
| --- | --- | --- | --- |
| `setup-and-test-wormhole` | Wormhole B0 | 2 | Standard test suite, moderate parallelization |
| `setup-and-test-blackhole` | Blackhole | 2 | Standard test suite, moderate parallelization |
| `build-quasar` | Quasar | 1 | Compilation-only, no hardware bottleneck |
| `perf-tests` (Wormhole) | Wormhole B0 | 5 | Long-running tests, high parallelization |
| `perf-tests` (Blackhole) | Blackhole | 5 | Long-running tests, high parallelization |

**Quasar Parallel Compilation**

Quasar uses a different parallelization strategy:

*   Test splits: 1 (no test splitting)
*   Pytest workers: `-n 40` (40 parallel compilation workers)
*   Focus on CPU-bound compilation rather than hardware execution

The `generate-matrix` job creates the test group matrix:

`matrix=$(echo "[$(seq -s, 1 ${{ inputs.test_splits }})]")`
This generates `[1, 2, ..., N]` for parallel execution across multiple runners.

## Conditional Execution Based on Change Detection

Each architecture-specific job executes conditionally based on change detection output from the `detect-changes` job:

**Change Detection File Patterns**

The `files_yaml` configuration defines which file changes trigger each output:

`wormhole:  - 'tt_llk_wormhole_b0/**'blackhole:  - 'tt_llk_blackhole/**'quasar:  - 'tt_llk_quasar/**'  - 'tests/sources/quasar/**'  - 'tests/python_tests/quasar/**'tests:  - 'tests/**'  - '!tests/sources/quasar/**'  - '!tests/python_tests/quasar/**'github:  - '.github/**'`
**Conditional Execution Logic**

| Job | Condition |
| --- | --- |
| `setup-and-test-wormhole` | `(wormhole OR tests OR github) AND not push` |
| `setup-and-test-blackhole` | `(blackhole OR tests OR github) AND not push` |
| `build-quasar` | `quasar AND not push` |
| `perf-tests` | `performance AND not push` |

The `not push` condition (`github.event_name != 'push'`) ensures these jobs only run on pull requests and workflow dispatch events, not on direct pushes to main.

## Performance Test Execution Matrix

Performance tests execute on both Wormhole and Blackhole architectures using a matrix strategy:

**Performance Test Configuration Matrix**

`strategy:  fail-fast: false  matrix:    include:      - target: Wormhole        runs_on: tt-ubuntu-2204-n150-stable      - target: Blackhole        runs_on: tt-ubuntu-2204-p150b-stable`
**Performance-Specific Settings**

| Setting | Value | Purpose |
| --- | --- | --- |
| `test_splits` | `5` | Higher parallelization for long-running tests |
| `pytest_markers` | `"perf"` | Only performance-marked tests |
| `timeout_minutes` | `240` | 4-hour timeout vs standard timeout |
| `use_durations` | `true` | Balance tests by historical runtime |
| `fail-fast` | `false` | Continue both architectures if one fails |

The performance tests trigger only when:

*   `needs.detect-changes.outputs.performance == 'true'`
*   Changes match patterns: `**/*perf**` or `**/*profiler**`
*   Event is not a push to main

This allows performance validation across both hardware platforms while maintaining independent execution that doesn't block the other architecture if failures occur.
