Multi-host deployment enables distributed inference for Large Language Models (LLMs) across multiple Tenstorrent Galaxy systems. This architecture is required for models whose memory or compute requirements exceed a single Galaxy system (32 chips), such as DeepSeek-V3.

## Overview

The system uses a **Controller/Worker** model coordinated via MPI (Message Passing Interface) and SSH. A single orchestration script (`run.py`) manages the lifecycle of Docker containers across all physical hosts in the cluster.

### Supported Hardware Configurations

| Configuration | Hosts | Total Chips | Mesh Device | Use Case |
| --- | --- | --- | --- | --- |
| **Dual Galaxy** | 2 | 64 | `(8, 8)` | Models requiring 2 Galaxy systems |
| **Quad Galaxy** | 4 | 128 | `(8, 16)` | Large models (e.g., DeepSeek-V3) |

## Architecture and Execution Flow

The deployment architecture separates the API serving logic from the physical hardware workers.

### Container Roles

1.   **Controller (Rank-0 Host Only):** Runs the vLLM API server and the `mpirun` process. It acts as the orchestrator for the inference session.
2.   **Workers (All Hosts):** Run an `sshd` service on port 2200. These containers host the actual MPI processes that interact with the Tenstorrent hardware.

### Execution Flow

1.   **Orchestrator Setup:**`MultiHostOrchestrator` starts Worker containers on all remote and local hosts (sshd mode). `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L15-L16" min=15 max=16 file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`
2.   **Controller Launch:** The orchestrator starts the Controller container (vLLM server mode). `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L17-L17" min=17  file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`
3.   **MPI Initialization:** The Controller's vLLM engine calls `tt-run`, which spawns MPI processes on Workers via SSH. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L18-L18" min=18  file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`
4.   **Inference:** Distributed processes communicate via the network interface specified in `MPI_INTERFACE`.

## Implementation Details

### MultiHostOrchestrator

The `MultiHostOrchestrator` class (defined in `workflows/multihost_orchestrator.py`) is responsible for the container lifecycle.

*   **Worker Management:** Uses `ThreadPoolExecutor` to start/stop Worker containers in parallel across all hosts. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L32-L32" min=32  file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`
*   **Monitoring Loop:** Implements a health check loop that monitors the status of all containers. If any container fails, the orchestrator triggers a cleanup of the entire cluster.
*   **Cleanup:** An `atexit` handler and signal handlers (`SIGINT`, `SIGTERM`) ensure that remote containers are stopped and temporary configuration files are removed. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L114-L135" min=114 max=135 file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`

## Deployment Constraints

1.   **Rank-0 Requirement:** The `run.py` command **must** be executed on the host intended to be Rank-0 (the first host in `MULTIHOST_HOSTS`). This is necessary for the torch distributed TCP rendezvous. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/multihost_deployment.md?plain=1#L145-L145" min=145  file-path="docs/multihost_deployment.md">Hii</FileRef>`
2.   **User Permissions:** The shared storage directory must be world-writable or owned by `UID 1000` (container user) to allow the containerized application to write cache and config files. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/multihost_orchestrator.py#L106-L106" min=106  file-path="workflows/multihost_orchestrator.py">Hii</FileRef>`
3.   **Network:** All hosts must have the `MPI_INTERFACE` configured with high-speed connectivity (e.g., 100GbE) to support the low-latency communication required for distributed tensor parallelism. `<FileRef file-url="https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/multihost_deployment.md?plain=1#L78-L84" min=78 max=84 file-path="docs/multihost_deployment.md">Hii</FileRef>`
