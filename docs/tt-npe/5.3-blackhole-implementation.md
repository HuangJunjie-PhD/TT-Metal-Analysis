This page documents the `BlackholeDeviceModel` class, which provides architecture-specific parameters and behaviors for Tenstorrent's Blackhole chip family. This implementation includes support for both P100 and P150 variants and defines the NoC topology, bandwidth characteristics, and routing algorithms specific to the Blackhole architecture.

For the abstract device model interface, see [Device Model Interface](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface). For Wormhole B0 implementation details, see [Wormhole B0 Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.2-wormhole-b0-implementation). For multi-chip configurations, see [Multi-chip Device Models](https://deepwiki.com/tenstorrent/tt-npe/5.4-multi-chip-device-models).

## Overview

The `BlackholeDeviceModel` class inherits from `npeDeviceModel` and implements the device-specific behavior for Blackhole chips. Key distinguishing features include:

*   **Grid Dimensions**: 12 rows × 17 columns (compared to Wormhole's 12×10)
*   **AI Clock**: 1.35 GHz operating frequency
*   **Link Bandwidth**: 60.9 bytes/cycle (2× Wormhole's 30 bytes/cycle)
*   **Model Variants**: P100 (7 DRAM banks) and P150 (8 DRAM banks)
*   **Enhanced Ethernet**: 50 bytes/cycle single-direction bandwidth

## Grid Topology and Core Layout

The Blackhole device has a 12×17 grid with a distinct core layout:

| Core Type | Positions | Count |
| --- | --- | --- |
| DRAM | Columns 0, 9 (all rows) | 24 |
| ETH | Row 1, columns 1-7 and 10-16 | 14 |
| WORKER | Most interior positions | 143 |
| UNDEF | Various edge positions | 23 |

The layout provides:

*   **DRAM Channels**: 2 vertical strips at columns 0 and 9
*   **Ethernet Cores**: 14 cores at row 1 for multi-chip connectivity
*   **Worker Cores**: Dense compute grid in the interior
*   **Undefined Cores**: Gaps for alignment, primarily column 8 and row 0

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 480-600](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L480-L600)

## NoC Link and NIU Identification

The device model maintains bidirectional lookup tables for NoC links and Network Interface Units (NIUs):

The lookup tables are populated during initialization by iterating through all grid positions and link/NIU types:

*   **Link Types**: `NOC0_EAST`, `NOC0_SOUTH`, `NOC1_NORTH`, `NOC1_WEST`
*   **NIU Types**: `NOC0_SRC`, `NOC0_SINK`, `NOC1_SRC`, `NOC1_SINK`

Total links: 12 × 17 × 4 = 816 links

 Total NIUs: 12 × 17 × 4 = 816 NIUs

## Bandwidth Characteristics

### Transfer Bandwidth Table

The Blackhole transfer bandwidth table defines achieved bandwidth as a function of packet size:

| Packet Size (bytes) | Bandwidth (bytes/cycle) |
| --- | --- |
| 0 | 0.0 |
| 128 | 6.0 |
| 256 | 12.1 |
| 512 | 24.2 |
| 1024 | 48.0 |
| 2048 | 57.7 |
| 4096 | 58.7 |
| 8192 | 60.4 |
| 16384 | 60.9 |

This table shows that Blackhole achieves near-peak bandwidth (60.9 bytes/cycle) for packets larger than 8KB. The interpolation function `interpolateBW()` in `npeDeviceModelUtils.hpp` handles intermediate packet sizes.

### Injection and Absorption Rates

The device model defines per-core-type bandwidth limits:

The DRAM rates are divided by `ai_clk_ghz` (1.35) to convert from AI clock cycles to NoC cycles. The function `getSrcInjectionRateByCoreType()` and `getSinkAbsorptionRateByCoreType()` retrieve these values with fallback to WORKER rates if a core type is not found.

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 469-478](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L469-L478)[tt_npe/cpp/include/device_models/blackhole.hpp 292-319](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L292-L319)

### Aggregate Bandwidth Metrics

The model provides aggregate bandwidth calculations:

*   **Link Bandwidth**: 60.9 bytes/cycle per link
*   **Aggregate DRAM Bandwidth**: `NUM_BANKS × ((injection_rate + absorption_rate) / 2)`
    *   P100: 7 × 40.0 = 280 bytes/cycle
    *   P150: 8 × 40.0 = 320 bytes/cycle

*   **Ethernet Bandwidth**: 50 bytes/cycle per direction

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 246-253](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L246-L253)

## Routing Algorithm

The Blackhole routing algorithm implements dimension-ordered routing with wrap-around (toroidal topology):

### Unicast Routing

For NOC0 (X-Y routing):

1.   Move East (incrementing column with wraparound) until reaching destination column
2.   Move South (incrementing row with wraparound) until reaching destination row

For NOC1 (Y-X routing):

1.   Move North (decrementing row with wraparound) until reaching destination row
2.   Move West (decrementing column with wraparound) until reaching destination column

Each movement adds the corresponding link to the route vector.

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 321-358](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L321-L358)[tt_npe/cpp/include/device_models/blackhole.hpp 360-388](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L360-L388)

### Multicast Routing

For multicast destinations, the algorithm computes routes to multiple endpoints and collects unique links:

*   **NOC0**: Routes to each column in the multicast range at the end row
*   **NOC1**: Routes to each row in the multicast range at the end column

The `boost::unordered_flat_set` ensures only unique links are included in the final route.

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 364-387](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L364-L387)

## Latency Calculations

Blackhole provides hardcoded latency values tuned to the architecture:

### Read Latencies

| Configuration | Cycles |
| --- | --- |
| Same location (sx==dx && sy==dy) | 65 |
| Same row (sx==dx && sy!=dy) | 177 |
| Same column (sy==dy && sx!=dx) | 217 |
| Different row and column | 329 |

### Write Latencies

Write latencies are computed based on hop count:

```
WRITE_LATENCY = STARTUP_LATENCY + (hops × CYCLES_PER_HOP)
```

Where:

*   `STARTUP_LATENCY = 40` cycles
*   `CYCLES_PER_HOP = 11` cycles (approximately 11.5)

The hop count is determined by the dimension-ordered routing algorithm with toroidal wraparound.

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 421-440](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L421-L440)

## Congestion Modeling

The `modelCongestion()` function implements bandwidth derating based on resource contention:

The algorithm:

1.   **Accumulates demand** on all NoC resources (links and NIUs) from active transfers
2.   **Accounts for partial timesteps**: Transfers starting mid-timestep have proportionally reduced demand
3.   **Identifies bottlenecks**: For each transfer, finds the maximum-loaded resource on its path
4.   **Computes derate factors**: Bandwidth is reduced proportionally to oversubscription
5.   **Applies single-iteration update**: The algorithm runs once per timestep (no iterative refinement)

Special handling for multicast:

*   Demand accumulates at all sink NIUs
*   Only WORKER cores absorb multicast traffic
*   Bottleneck is determined by the slowest sink NIU

Sources: [tt_npe/cpp/include/device_models/blackhole.hpp 67-201](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp#L67-L201)

## Integration with Simulation Engine

The `computeCurrentTransferRate()` method integrates bandwidth calculation and congestion modeling:

The two-step process:

1.   **Initial Bandwidth**: `updateTransferBandwidth()` sets bandwidth based on packet size and injection rate
2.   **Congestion Derating**: `modelCongestion()` applies congestion-based reductions if enabled

The resulting `curr_bandwidth` value is used by the simulation engine to advance transfers in the current timestep.

## Model Variants: P100 vs P150

The `Model` enum distinguishes between two Blackhole SKUs:

| Variant | DRAM Banks | Aggregate DRAM BW (bytes/cycle) |
| --- | --- | --- |
| P100 | 7 | 280 |
| P150 | 8 | 320 |

The variant is selected at construction time and affects only the `NUM_BANKS` member, which propagates to aggregate DRAM bandwidth calculations via `getAggregateDRAMBandwidth()`.

## Key Differences from Wormhole B0

| Characteristic | Blackhole | Wormhole B0 |
| --- | --- | --- |
| **Grid Size** | 12 × 17 (204 cores) | 12 × 10 (120 cores) |
| **Link Bandwidth** | 60.9 bytes/cycle | 30 bytes/cycle |
| **AI Clock** | 1.35 GHz | N/A (implicit 1.0) |
| **Ethernet Bandwidth** | 50 bytes/cycle | 12.5 bytes/cycle |
| **DRAM Injection** | 40.0 bytes/cycle | 23.2 bytes/cycle |
| **DRAM Absorption** | 40.0 bytes/cycle | 24.0 bytes/cycle |
| **DRAM Banks** | 7 (P100) or 8 (P150) | 12 |
| **Read Latency (same loc)** | 65 cycles | 70 cycles |
| **Write CYCLES_PER_HOP** | 11 cycles | 10 cycles |
| **Ethernet Cores** | Row 1 (14 cores) | Rows 0,6 (16 cores) |

The Blackhole architecture provides approximately 2× the per-link bandwidth and 4× the ethernet bandwidth compared to Wormhole B0, while maintaining similar routing and congestion modeling algorithms.
