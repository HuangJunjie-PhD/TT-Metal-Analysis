# Chip Detection

Relevant source files
*   [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py)
*   [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py)

## Purpose and Scope

Chip detection is the process by which tt-flash discovers and enumerates Tenstorrent hardware connected to the host system via PCI. The detection layer bridges pyluwen's low-level PCI enumeration with tt-flash's chip abstraction classes, wrapping discovered devices in architecture-specific `WhChip` or `BhChip` objects.

This page documents the detection functions, progress reporting callbacks, and error handling mechanisms. For information about the chip abstraction classes themselves, see [TTChip Class Hierarchy](https://deepwiki.com/tenstorrent/tt-flash/6.1-ttchip-class-hierarchy). For information about how detection fits into the overall flashing workflow, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages).

**Sources:**[tt_flash/chip.py 1-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L1-L387)

## Detection Functions

tt-flash provides three primary functions for chip detection, each serving different use cases in the flashing and reset workflows.

### detect_local_chips()

`def detect_local_chips(ignore_ethernet: bool = False) -> list[Union[WhChip, BhChip]]`
This function discovers PCI-connected chips using a conservative approach that verifies communication with each device. It wraps pyluwen's `luwen_detect_chips_fallible()` function with continue-on-failure disabled, meaning detection stops immediately if any chip fails communication checks.

**Key characteristics:**

*   Forces `local_only=True` in pyluwen (only enumerates directly-connected PCI devices, no remote chips)
*   Uses `continue_on_failure=False` to ensure all chips are accessible
*   Raises an exception if any chip lacks communication: `"Do not have communication with {device}, you should reset or remove this device from your system before continuing"`
*   Provides a visual progress callback via `chip_detect_callback` when running in an interactive terminal
*   Returns architecture-wrapped chips (`WhChip` or `BhChip`) ready for flashing operations

**Sources:**[tt_flash/chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L308-L374)

### detect_chips()

`def detect_chips(local_only: bool = False) -> list[Union[WhChip, BhChip]]`
This function provides a simpler detection path without progress callbacks or communication verification. It wraps pyluwen's `luwen_detect_chips()` function and is used after reset operations to re-enumerate devices.

**Key characteristics:**

*   Optionally detects remote chips when `local_only=False` (e.g., chips connected via Ethernet on Galaxy systems)
*   No progress reporting or communication verification
*   Faster than `detect_local_chips()` but less safe for initial detection
*   Used in [tt_flash/flash.py 939](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L939-L939) after chip resets to quickly refresh the device list

**Sources:**[tt_flash/chip.py 377-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L377-L387)

### detect_chips_with_callback()

This is an external function from `tt_tools_common.utils_common.tools_utils` used specifically for Galaxy system reset workflows. It supports custom callback functions for progress reporting.

**Usage in tt-flash:**

*   [tt_flash/flash.py 777](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L777-L777) - Called after Galaxy reset with `local_only=True, ignore_ethernet=True` to avoid false hangs from reused Ethernet status codes

**Sources:**[tt_flash/flash.py 29](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L29-L29)[tt_flash/flash.py 777](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L777-L777)

## Progress Reporting with Callbacks

Both `detect_local_chips()` and the `TTChip.reinit()` method use callback functions to provide real-time progress updates during chip enumeration. The callback mechanism distinguishes between interactive (TTY) and non-interactive environments.

### Callback Implementation

The `chip_detect_callback` function tracks enumeration progress with three stateful variables:

`chip_count = 0      # Number of chips detected so farblock_count = 0     # Terminal lines used for displaylast_draw = 0       # Timestamp of last display update`
**Callback behavior:**

1.   **Chip counting:** Increments on `status.new_chip()`, decrements on `status.correct_down()`
2.   **Display throttling:** Updates at most every 100ms (0.1 seconds) to avoid terminal flicker
3.   **TTY mode:** Uses ANSI escape codes to create a live-updating display: 
    *   `\033<FileRef file-url="https://github.com/tenstorrent/tt-flash/blob/60c06032/{block_count}A` - Move cursor up\n - `\\033[J` - Clear from cursor to end of screen\n - `\\r` - Carriage return for line overwrite\n4. **Non-TTY mode#LNaN-LNaN" NaN file-path="{block_count}A`- Move cursor up\n   -`\033[J`- Clear from cursor to end of screen\n   -`\r` - Carriage return for line overwrite\n4. **Non-TTY mode">Hii [tt_flash/chip.py 320-348](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L320-L348)

### Status Information from PyLuwen

The callback receives a `status` object from pyluwen with methods:

*   `status.new_chip()` - Returns `True` when a new chip is detected
*   `status.correct_down()` - Returns `True` when chip count decreases (error correction)
*   `status.status_string()` - Returns multi-line string with detailed enumeration progress

This status information is only displayed in TTY mode; non-interactive runs print a final count: `Detected Chips: {chip_count}`[tt_flash/chip.py 372](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L372-L372)

**Sources:**[tt_flash/chip.py 130-158](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L130-L158)[tt_flash/chip.py 320-348](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L320-L348)

## Device Wrapping and Architecture Recognition

After pyluwen enumerates PCI devices, tt-flash wraps each device in an architecture-specific class based on device type identification.

### Device Type Detection

The wrapping process queries each device's architecture:

**Architecture-specific wrapping:**

| PyLuwen Method | Chip Type | tt-flash Wrapper | Defined In |
| --- | --- | --- | --- |
| `device.as_wh()` | Wormhole | `WhChip(device.as_wh())` | [tt_flash/chip.py 365](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L365-L365) |
| `device.as_bh()` | Blackhole | `BhChip(device.as_bh())` | [tt_flash/chip.py 367](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L367-L367) |

Each wrapped chip undergoes initialization in the constructor:

1.   **Store interface ID:**`self.interface_id = chip.pci_interface_id()`[tt_flash/chip.py 116](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L116-L116)

The firmware defines YAML file (`fw_defines.yaml`) is loaded from the appropriate data directory based on chip type (Wormhole vs Blackhole) [tt_flash/chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L109-L110)

## Error Handling and Communication Verification

### Communication Checks

The `detect_local_chips()` function performs strict communication verification using pyluwen's `have_comms()` method:

`if not device.have_comms():    raise Exception(        f"Do not have communication with {device}, you should reset or remove this device from your system before continuing."    )`
This check ensures that every detected chip is accessible via AXI/SPI before proceeding with flash operations. The error message explicitly guides users to reset or physically remove non-communicating devices.

**Sources:**[tt_flash/chip.py 357-360](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L357-L360)

### Force Upgrade

After communication verification, devices are "force upgraded" via `device.force_upgrade()`[tt_flash/chip.py 362](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L362-L362) This pyluwen operation ensures the device handle is upgraded to the most capable communication mode supported by both the hardware and the library.

**Sources:**[tt_flash/chip.py 362](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L362-L362)

### Unrecognized Device Handling

If a device cannot be identified as either Wormhole or Blackhole, both detection functions raise a `ValueError` with the message: `"Did not recognize board"`[tt_flash/chip.py 369](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L369-L369)[tt_flash/chip.py 385](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L385-L385)

This error typically indicates:

*   Unsupported hardware (e.g., legacy Grayskull chips removed in v3.4.7)
*   Corrupted device state requiring reset
*   Driver/library version mismatch

**Sources:**[tt_flash/chip.py 368-369](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L368-L369)[tt_flash/chip.py 384-385](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L384-L385)

## Integration with Flashing Workflow

Chip detection integrates into the flashing workflow at multiple stages:

### Initial Detection (DETECT Stage)

The main flashing routine calls detection to discover flashable devices:

`# From flash command handlerdevices = detect_local_chips()  # Safe, verified enumeration`
This occurs in the **DETECT** stage of the four-stage flash pipeline (see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages)).

**Sources:** Referenced in Diagram 3 (Firmware Flashing Workflow)

### Post-Reset Re-enumeration

After chip resets, the faster `detect_chips()` function re-enumerates devices without callbacks:

`if len(needs_reset_wh) > 0 or len(needs_reset_bh) > 0:    devices = detect_chips()`
This quick re-detection at [tt_flash/flash.py 939](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L939-L939) allows post-flash validation to proceed immediately after reset completion.

**Sources:**[tt_flash/flash.py 938-940](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L938-L940)

### Galaxy System Special Case

Galaxy systems use a different detection path after IPMI-based resets:

`chips = detect_chips_with_callback(local_only=True, ignore_ethernet=True)`
The `ignore_ethernet=True` parameter (mapped to `noc_safe`) prevents false hangs from Ethernet status codes that were reused to denote "connected" state, causing incorrect chip status interpretation during enumeration [tt_flash/flash.py 775-777](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L775-L777)

**Sources:**[tt_flash/flash.py 775-777](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L775-L777)

## Detection Function Comparison

| Feature | `detect_local_chips()` | `detect_chips()` | `detect_chips_with_callback()` |
| --- | --- | --- | --- |
| **Progress callback** | Yes (built-in) | No | Yes (custom) |
| **Communication verification** | Yes (strict) | No | Configurable |
| **Failure behavior** | Fail fast | Continue | Configurable |
| **Remote chip support** | No (always local) | Yes (optional) | Yes (optional) |
| **Primary use case** | Initial flash detection | Post-reset refresh | Galaxy reset workflows |
| **Performance** | Slower (full verification) | Faster | Depends on callback |
| **Source location** | [tt_flash/chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L308-L374) | [tt_flash/chip.py 377-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L377-L387) | `tt_tools_common` library |
