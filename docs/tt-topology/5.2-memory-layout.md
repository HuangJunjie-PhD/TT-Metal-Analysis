# Memory Layout

Relevant source files
*   [tt_topology/backend.py](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py)
*   [tt_topology/constants.py](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py)

This document describes the hardware memory layout and register definitions used by tt-topology for configuring Tenstorrent chips. It covers the organization of SPI flash memory regions, parameter tables, and the addressing scheme for dual-chip boards.

For information about how these memory regions are accessed programmatically, see [Hardware Interface](https://deepwiki.com/tenstorrent/tt-topology/4.2-hardware-interface). For details on specific board types and their memory characteristics, see [Supported Boards](https://deepwiki.com/tenstorrent/tt-topology/5.1-supported-boards).

## Memory Region Overview

The tt-topology system interacts with several distinct memory regions on Tenstorrent hardware, each serving specific purposes in topology configuration and status reporting.

### Memory Map

**Sources:**[tt_topology/constants.py 4-33](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py#L4-L33)

### Base Address Definitions

The memory layout is organized around four primary base addresses that define different functional regions:

| Region | Base Address | Purpose |
| --- | --- | --- |
| `ETH_L1_PARAM_BASE_ADDR` | `0x1000` | Board identification parameters |
| `ETH_TEST_RESULT_BASE_ADDR` | `0x1EC0` | Connection test results and neighbor info |
| `ETH_PARAM_BASE_ADDR` | `0x21100` | Configuration parameters |
| `ETH_FW_BASE_ADDR` | `0x23000` | Firmware region |

**Sources:**[tt_topology/constants.py 4-23](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py#L4-L23)

## Parameter Table Structure

The parameter table at `ETH_PARAM_BASE_ADDR` contains the primary configuration registers used by tt-topology. Each parameter occupies a 4-byte slot with specific offsets.

### Parameter Layout

**Sources:**[tt_topology/constants.py 8-13](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py#L8-L13)[tt_topology/backend.py 758-905](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L758-L905)

### Parameter Definitions

#### Core Configuration Parameters

*   **`ETH_PARAM_CHIP_COORD`** (`0x21100`): Stores X,Y coordinates as `bytearray([x, y, 0x0, 0x0])`
*   **`ETH_PARAM_PORT_DISABLE`** (`0x21108`): 16-bit mask disabling specific ethernet ports

#### Galaxy-Specific Parameters

*   **`ETH_PARAM_MOBO_ETH_EN`** (`0x211D0`): Set to `0xC3` to enable ethernet on Galaxy systems
*   **`ETH_PARAM_RACK_SHELF`** (`0x211D4`): Rack and shelf addressing for multi-Galaxy configurations

**Sources:**[tt_topology/constants.py 9-13](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py#L9-L13)[tt_topology/backend.py 991-1081](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L991-L1081)

## Dual Chip Support

N300 boards contain two Wormhole chips (left and right) that require separate parameter configurations. The memory layout accommodates this through address offsets.

### Left/Right Chip Addressing

**Sources:**[tt_topology/constants.py 14-16](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/constants.py#L14-L16)[tt_topology/backend.py 248-893](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L248-L893)

### Right Chip Configuration

The `ETH_PARAM_RIGHT_OFFSET` of `0x100` is added to base parameter addresses when configuring the right chip on N300 boards:

*   Right chip coordinate: `ETH_PARAM_CHIP_COORD + ETH_PARAM_RIGHT_OFFSET`
*   Right chip port disable: `ETH_PARAM_PORT_DISABLE + ETH_PARAM_RIGHT_OFFSET`
*   Right chip rack/shelf: `ETH_PARAM_RACK_SHELF + ETH_PARAM_RIGHT_OFFSET`

After configuration, the `MSG_TRIGGER_SPI_COPY_LtoR` ARC message synchronizes parameters between chips.

**Sources:**[tt_topology/backend.py 175-287](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L175-L287)

## Memory Access Patterns

The tt-topology system uses specific patterns for reading and writing to these memory regions based on the operation being performed.

### Configuration Workflow

**Sources:**[tt_topology/backend.py 142-905](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L142-L905)

### Connection Discovery

The system reads test result regions to build topology maps:

1.   **Board Identification**: Read `ETH_L1_PARAM_BOARD_TYPE` and `ETH_L1_PARAM_BOARD_ID`
2.   **Connection Scanning**: For each ethernet port (0-15), read remote chip information
3.   **Neighbor Mapping**: Build connection graph using `ETH_TEST_RESULT_REMOTE_TYPE/ID`

**Sources:**[tt_topology/backend.py 314-432](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/backend.py#L314-L432)
