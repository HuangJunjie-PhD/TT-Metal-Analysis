# Command Options

Relevant source files
*   [README.md](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/README.md?plain=1)
*   [tt_topology/tt_topology.py](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py)

This document provides comprehensive documentation of all command-line options available in the `tt-topology` CLI tool. These options control hardware topology configuration, output generation, and operational modes for Tenstorrent chip systems.

For information about the specific layout algorithms and coordinate generation, see [Topology Algorithms](https://deepwiki.com/tenstorrent/tt-topology/4.3-topology-algorithms). For details about octopus-specific configuration and Galaxy system support, see [Octopus Mode](https://deepwiki.com/tenstorrent/tt-topology/3.3-octopus-mode).

## Command Structure

The basic command structure follows the pattern:

`tt-topology [OPTIONS]`
All command-line options are processed through the `parse_args()` function and validated before execution begins.

## Core Topology Options

### Layout Selection (`-l, --layout`)

**Syntax:**`-l {linear,torus,mesh,isolated}` or `--layout {linear,torus,mesh,isolated}`

**Default:**`linear`

**Purpose:** Selects the chip coordinate layout algorithm

| Layout | Description | Use Case |
| --- | --- | --- |
| `linear` | Single-line connection of all chips | Sequential processing workloads |
| `torus` | Cyclic graph with wraparound connections | High-bandwidth circular topologies |
| `mesh` | Trivalent graph with maximum 3 connections per node | Complex multi-dimensional workloads |
| `isolated` | Default state with no inter-chip connections | Individual chip operation |

### Octopus Mode (`-o, --octopus`)

**Syntax:**`-o` or `--octopus`

**Purpose:** Enables Galaxy system configuration mode for TGG/TG setups

**Requirements:** Must be used with `--reset` option providing a valid JSON configuration

### Device Listing (`-ls, --list`)

**Syntax:**`-ls` or `--list`

**Purpose:** Lists all detected boards with their current coordinates and layout configuration

**Behavior:** Exits after displaying information without modifying any hardware

**Sources:**[tt_topology/tt_topology.py 40-85](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L40-L85)[tt_topology/tt_topology.py 472-475](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L472-L475)

## Output Control Options

### Log File Configuration (`--log`)

**Syntax:**`--log [filename]`

**Default:**`~/tt_topology_logs/<timestamp>_log.json`

**Purpose:** Specifies custom filename for the topology flash log containing SPI register states, connection maps, and coordinate assignments

### Plot Output (`-p, --plot_filename`)

**Syntax:**`-p [filename]` or `--plot_filename [filename]`

**Default:**`chip_layout.png`

**Purpose:** Generates a PNG visualization of the chip layout graph using matplotlib

### Test Log Filename (`-f, --filename`)

**Syntax:**`-f [filename]` or `--filename [filename]`

**Default:**`~/tt_smi/<timestamp>_snapshot.json`

**Purpose:** Changes the filename for test log output

**Sources:**[tt_topology/tt_topology.py 86-104](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L86-L104)

## Configuration Management Options

### Reset Configuration (`-r, --reset`)

**Syntax:**`-r [config.json ...]` or `--reset [config.json ...]`

**Purpose:** Provides reset JSON configuration for advanced hardware control

**Required for:** Octopus mode operations

**Format:** Must be a valid JSON file containing mobo reset configuration

### Generate Reset Template (`-g, --generate_reset_json`)

**Syntax:**`-g` or `--generate_reset_json`

**Purpose:** Creates a default reset JSON template that can be customized for use with `--reset`

**Output Location:**`~/.config/tenstorrent/reset_config.json`

**Workflow:** Generate → Edit → Use with `--reset`

**Sources:**[tt_topology/tt_topology.py 69-78](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L69-L78)[tt_topology/tt_topology.py 106-114](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/tt_topology/tt_topology.py#L106-L114)

## Common Option Combinations

### Standard Topology Configuration

`# Linear layout with custom plot outputtt-topology -l linear -p my_linear_layout.png # Mesh layout with custom log filett-topology -l mesh --log mesh_config.json`
### Galaxy System Setup

`# Generate configuration templatett-topology -g # Configure Galaxy systemtt-topology -o -r ~/.config/tenstorrent/reset_config.json`
### Diagnostic Operations

`# List current topologytt-topology -ls # Isolated mode (reset to defaults)tt-topology -l isolated`
## Validation and Error Handling

The system performs several validation steps:

1.   **Driver Validation:** Checks for tt-kmd driver availability via `get_driver_version()`
2.   **Hardware Detection:** Uses `detect_chips_with_callback()` to find supported devices
3.   **Board Compatibility:** Filters for supported board types (n300, n150, GALAXY)
4.   **Option Dependencies:** Validates that octopus mode includes required reset configuration

Invalid combinations or missing dependencies result in immediate exit with descriptive error messages.
