# Build Workflows

Relevant source files
*   [.github/workflows/bh-update-status-by-label.yml](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/bh-update-status-by-label.yml)

This document describes the GitHub Actions workflows that automate the building of tt-flash packages for multiple distribution channels. The build system supports both Debian package generation for Ubuntu variants and Python wheel creation for PyPI distribution. For information about the release orchestration and publishing process, see [Release Process](https://deepwiki.com/tenstorrent/tt-flash/9.1-release-process). For Debian-specific packaging details, see [Debian Packaging](https://deepwiki.com/tenstorrent/tt-flash/9.2-debian-packaging). For PyPI distribution specifics, see [PyPI Distribution](https://deepwiki.com/tenstorrent/tt-flash/9.3-pypi-distribution).

## Workflow Architecture

The build system uses a three-tier workflow architecture with `build-all.yml` serving as the orchestration layer that invokes parallel build workflows for Debian and PyPI packages.

### Build Workflow Orchestration

**Sources:**[.github/workflows/build-all.yml 1-53](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-all.yml#L1-L53)

The orchestration workflow accepts version component inputs (`MAJOR`, `MINOR`, `PATCH`, `NUMBER_OF_COMMITS_SINCE_TAG`) and a Git reference (`ref`), forwarding these to both build workflows. The `build_all_depends` job performs validation before triggering parallel builds with `needs` dependencies ensuring proper sequencing [.github/workflows/build-all.yml 25-52](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-all.yml#L25-L52)

| Workflow Parameter | Type | Required | Purpose |
| --- | --- | --- | --- |
| `ref` | string | No | Git reference to build from (default: `github.ref`) |
| `MAJOR` | string | No | Major version component |
| `MINOR` | string | No | Minor version component |
| `PATCH` | string | No | Patch version component |
| `NUMBER_OF_COMMITS_SINCE_TAG` | string | No | Commit count since last tag |

**Sources:**[.github/workflows/build-all.yml 6-22](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-all.yml#L6-L22)[.github/workflows/build-debian.yml 6-23](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L6-L23)

## Debian Package Build Workflow

The `build-debian.yml` workflow generates signed `.deb` packages for multiple Ubuntu versions using a matrix strategy.

### Build Matrix Configuration

**Sources:**[.github/workflows/build-debian.yml 26-35](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L26-L35)

The matrix strategy with `fail-fast: false` ensures all Ubuntu versions complete building even if one fails, maximizing artifact generation [.github/workflows/build-debian.yml 28-34](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L28-L34)

### Build Process Flow

**Sources:**[.github/workflows/build-debian.yml 36-178](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L36-L178)

### Dependency Installation

The workflow installs comprehensive Debian packaging dependencies before building:

```
build-essential      # GCC, make, and core build tools
debhelper           # Debian helper scripts
dh-python           # Python-specific Debian helpers
dh-sequence-python3 # Python3 packaging sequence
git-buildpackage    # gbp tools for Git-based packaging
gnupg               # GPG signing
libpython3-all-dev  # Python development headers
pybuild-plugin-pyproject  # PEP 517/518 build support
python3-all         # Python interpreters
python3-pip         # pip installer
python3-tomli       # TOML parsing for pyproject.toml
```

**Sources:**[.github/workflows/build-debian.yml 53-66](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L53-L66)

### GPG Signing Process

The workflow uses `crazy-max/ghaction-import-gpg@v6` to import the signing key from the `PKG_SIGNING_KEY_DEB` secret [.github/workflows/build-debian.yml 67-73](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L67-L73) The imported key's fingerprint is captured and passed to `gbp buildpackage` via the `DEBSIGN_KEYID` environment variable [.github/workflows/build-debian.yml 128-133](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L128-L133)

### Version Resolution

**Sources:**[.github/workflows/build-debian.yml 80-116](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L80-L116)

The workflow first attempts to use version components from workflow inputs [.github/workflows/build-debian.yml 84-107](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L84-L107) If inputs are not provided, it uses the `damienaicheh/extract-version-from-tag-action` to parse version information from Git tags [.github/workflows/build-debian.yml 80-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L80-L81) The resolved version is then passed to `gbp dch` to update the Debian changelog [.github/workflows/build-debian.yml 119-127](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L119-L127)

### Git Buildpackage Commands

The workflow executes two `git-buildpackage` commands in sequence:

#### Changelog Generation

**Options:**

*   `--debian-branch`: Specifies the branch to scan for changes
*   `-R`: Release version (marks as released, not UNRELEASED)
*   `-N`: Sets explicit version number
*   `--spawn-editor=never`: Automated mode without interactive editor

**Environment:**

*   `EMAIL: releases@tenstorrent.com`
*   `NAME: Tenstorrent Releases`

**Sources:**[.github/workflows/build-debian.yml 119-127](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L119-L127)

#### Package Building

**Options:**

*   `--git-ignore-new`: Proceeds even with uncommitted changes (necessary since `gbp dch` modifies `debian/changelog`)

**Environment:**

*   `DEBSIGN_KEYID`: GPG key fingerprint for package signing

**Sources:**[.github/workflows/build-debian.yml 128-133](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L128-L133)

### Artifact Naming and Upload

The workflow generates multiple artifacts with distribution-specific naming:

**Sources:**[.github/workflows/build-debian.yml 135-178](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L135-L178)

The distro identifier uses the matrix OS value directly [.github/workflows/build-debian.yml 143-149](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L143-L149) File renaming adds distro suffixes using `sed` transformations:

*   `.deb` files: `s/_all/_all-${DISTRO_IDENT}/`[.github/workflows/build-debian.yml 150-155](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L150-L155)
*   `.changes` files: `s/_amd64/_amd64-${DISTRO_IDENT}/`[.github/workflows/build-debian.yml 156-158](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L156-L158)

Four artifact uploads occur per matrix job:

| Artifact Name Pattern | Contents | Purpose |
| --- | --- | --- |
| `${DEB_FILENAME_DISTRO}` | Single `.deb` package | Installable package |
| `debian-changelog-${DISTRO_IDENT}` | `debian/changelog` file | Version history |
| `artifacts-${DISTRO_IDENT}.zip` | All files in `artifacts/` | Complete build output |
| `${DEB_FILENAME_CHANGES_DISTRO}` | `.changes` file | Package metadata and checksums |

**Sources:**[.github/workflows/build-debian.yml 159-178](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L159-L178)

## PyPI Wheel Build Workflow

The `build-pypi.yml` workflow generates Python wheel packages for distribution via TestPyPI and PyPI. While the complete workflow file is not shown in the provided sources, the orchestration in `build-all.yml` demonstrates its integration:

**Sources:**[.github/workflows/build-all.yml 46-52](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-all.yml#L46-L52)

## Reference Resolution

Both build workflows support flexible reference resolution to enable building from any Git reference:

**Sources:**[.github/workflows/build-debian.yml 42-51](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L42-L51)[.github/workflows/build-debian.yml 74-79](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L74-L79)

The reference resolution logic uses environment variable cascading [.github/workflows/build-debian.yml 42-51](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L42-L51):

1.   If `inputs.ref` is provided, use it
2.   Otherwise, fall back to `github.ref_name` from the workflow trigger context

The checkout action fetches full tag history (`fetch-tags: true`, `fetch-depth: 0`) to enable version extraction from Git tags [.github/workflows/build-debian.yml 74-79](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L74-L79)

## Workflow Triggers

All build workflows support dual trigger mechanisms:

**Sources:**[.github/workflows/build-debian.yml 3-23](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L3-L23)[.github/workflows/build-all.yml 3-22](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-all.yml#L3-L22)

The `workflow_dispatch` trigger enables manual execution from the GitHub Actions UI, while `workflow_call` allows the release workflow to invoke these builds programmatically with specific version parameters. This dual-trigger pattern provides both automation support and manual override capabilities for release engineering.
