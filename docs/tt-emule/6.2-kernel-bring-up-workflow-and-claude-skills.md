The kernel bring-up workflow in `tt-emule` is a structured process designed to bridge the gap between silicon hardware specifications and faithful software emulation. This process is accelerated by a suite of Claude "Skills" and "Agents" that automate hardware lookups, template generation, and verification.

## Bring-Up Workflow Overview

The bring-up of a new kernel or operation follows a rigorous loop: triaging missing symbols, looking up hardware specifications across architectures, implementing mocks at the appropriate API layer, and verifying against silicon baselines.

### The Bring-Up Loop

1.   **Triage**: Determine if the operation is missing or already covered in a shared shim using `scripts/find_symbol.py`[.claude/skills/compute-llk-bringup/SKILL.md 44-77](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/compute-llk-bringup/SKILL.md?plain=1#L44-L77)
2.   **Hardware Lookup**: Consult authoritative hardware specifications for Wormhole, Blackhole, and Quasar [.claude/skills/implement-mock/SKILL.md 22-30](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/implement-mock/SKILL.md?plain=1#L22-L30)
3.   **Strategy Selection**: Choose between Strategy A (Generic Stub), Strategy B (Per-op Patch), or Strategy C (Semantic Rewrite) [.claude/skills/implement-mock/SKILL.md 31-43](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/implement-mock/SKILL.md?plain=1#L31-L43)
4.   **Implementation**: Author the mock using standardized patterns, ensuring signature parity with silicon [.claude/skills/compute-llk-bringup/SKILL.md 116-144](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/compute-llk-bringup/SKILL.md?plain=1#L116-L144)
5.   **Verification**: Rebuild the JIT runner and execute regression suites to ensure PCC (Pearson Correlation Coefficient) matches silicon expectations [.claude/skills/verify-mock/SKILL.md 65-82](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/verify-mock/SKILL.md?plain=1#L65-L82)

## Claude Skills

Skills are specialized prompts and workflows that guide the Claude agent through complex tasks.

### `/implement-mock`&`/verify-mock`

The primary entry point for adding faithful mocks. It orchestrates the transition from hardware specification to verified implementation.

*   **Strategy A (Stub)**: Used for Layer-1 tile-ops (e.g., `noc_async_write`, `mul_tiles`). Stubs are placed in `include/jit_hw/`[.claude/skills/implement-mock/SKILL.md 44-62](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/implement-mock/SKILL.md?plain=1#L44-L62)
*   **Strategy B (Patch)**: Used for Layer-1.5/2 mixes where a consumer's header (e.g., `op.hpp`) is patched under `__EMULE_JIT_MODE` to route L1 addresses through `__emule_local_l1_to_ptr`[.claude/skills/implement-mock/SKILL.md 63-75](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/implement-mock/SKILL.md?plain=1#L63-L75)
*   **Strategy C (Rewrite)**: Used when the silicon implementation is too deeply bound to hardware-specific LLKs. The op is re-expressed at a higher level using `__emule_dst[]` math [.claude/skills/implement-mock/SKILL.md 76-84](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/implement-mock/SKILL.md?plain=1#L76-L84)

### `/compute-llk-bringup`

A specialization of Strategy A specifically for compute-kernel LLK shims located in `include/jit_hw/api/compute/`. It focuses on:

*   **Signature Locking**: Verifying parameters and types against `tt_metal/hw/inc/api/compute/`[.claude/skills/compute-llk-bringup/SKILL.md 98-99](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/compute-llk-bringup/SKILL.md?plain=1#L98-L99)
*   **Math Porting**: Translating polynomial coefficients and Cody-Waite reductions from `ckernel_sfpu_<name>.h` into C++ loops over `__emule_dst`[.claude/skills/compute-llk-bringup/SKILL.md 100-103](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/compute-llk-bringup/SKILL.md?plain=1#L100-L103)

### `/parallel-mock-implementation`

A batching primitive that utilizes the Claude Workflow tool to spawn sub-agents. Each sub-agent is responsible for authoring exactly one file in isolation, preventing race conditions on shared wiring files like `sfpu_split_includes.h`[.claude/skills/parallel-mock-implementation/SKILL.md 12-23](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/skills/parallel-mock-implementation/SKILL.md?plain=1#L12-L23)

## Uplift Skill

The `/uplift` skill is a specialized methodology for handling regressions caused by updating the `tt-metal` or `tt-umd` pins.

### Uplift Methodology

1.   **Triage**: Read build artifacts to identify if the failure is a device-open crash, JIT-compile error, or hang [.claude/CLAUDE.md 64](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/CLAUDE.md?plain=1#L64-L64)
2.   **Bisection**: Perform a two-stage build-and-run bisection to isolate whether the regression stems from `tt-metal` source changes or `tt-umd` driver updates [.claude/CLAUDE.md 91-97](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/CLAUDE.md?plain=1#L91-L97)
3.   **Fix Classification**: 
    *   **Shared-runtime behavioral regression**: Fixes in `sw_emule_chip.cpp` or `emulated_program_runner.cpp`.
    *   **jit_hw API-surface drift**: Updating stubs in `include/jit_hw/` to match new upstream signatures [.claude/CLAUDE.md 94-95](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/CLAUDE.md?plain=1#L94-L95)

4.   **Verification**: Use `--collect-only` to ensure membership in curated suites and verify with an "oracle" (the previous known-good pin) [.claude/CLAUDE.md 96-97](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.claude/CLAUDE.md?plain=1#L96-L97)
