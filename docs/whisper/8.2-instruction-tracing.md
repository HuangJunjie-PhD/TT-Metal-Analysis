Instruction tracing provides a detailed record of instruction execution, including register modifications, memory accesses, CSR updates, and optional page table walk information. This feature is essential for debugging, verification, and understanding program behavior at the architectural level.

For information about debug triggers and breakpoints, see [Debug Triggers and Breakpoints](https://deepwiki.com/tenstorrent/whisper/8.1-debug-triggers-and-breakpoints). For information about memory consistency verification, see [Memory Ordering Verification](https://deepwiki.com/tenstorrent/whisper/8.3-memory-ordering-verification).

## Overview

Whisper generates instruction traces that record the effects of each executed instruction. Traces can be output in two formats:

*   **Standard format**: Human-readable format with continuation lines for multiple changes
*   **CSV format**: Comma-separated format optimized for automated analysis and post-processing

Instruction tracing captures:

*   Program counter (virtual and physical addresses)
*   Instruction encoding and disassembly
*   Modified registers (integer, floating-point, vector, CSRs)
*   Memory accesses (loads and stores)
*   Privilege mode and virtual mode state
*   Optional page table walk information for address translation

## Trace Record Format

### Standard Format Structure

Each instruction trace consists of one or more lines, with the first line showing the primary change and subsequent lines (prefixed with `+`) showing additional changes:

```
#<tag> <hartid> <priv> <pc> <opcode> <resource> <addr> <value> <disassembly>
  +
#<tag> <hartid> <priv> <pc> <opcode> <resource> <addr> <value> <disassembly>
```

**Field Descriptions:**

| Field | Description | Example |
| --- | --- | --- |
| `tag` | Unique instruction sequence number | `#12345` |
| `hartid` | System hart index | `0` |
| `priv` | Privilege mode: M, HS, S, VS, VU, U, D (debug) | `M`, `VS` |
| `pc` | Program counter (hex) | `0x80000000` |
| `opcode` | Instruction encoding (hex) | `00000013` |
| `resource` | Type: `r` (int reg), `f` (fp reg), `v` (vec reg), `c` (CSR), `m` (memory) | `r` |
| `addr` | Register number or memory address | `0x05` |
| `value` | New value | `0x00001234` |
| `disassembly` | Instruction disassembly | `addi x5, x0, 0x1234` |

**Example Traces:**

```
# Integer register write
#100 0 M 0x80000000 00000013 r 05         00001234  addi x5, x0, 0x1234

# Load with memory address annotation
#101 0 M 0x80000004 00052503 r 0a         00000042  lw x10, 0(x10) [0x1000:0x1000]

# Store with memory write
#102 0 M 0x80000008 00a52023 m 00001000   00000042  sw x10, 0(x10)

# CSR write (multiple changes shown with +)
#103 0 M 0x8000000c 30102573 r 0a         00000001  csrrs x10, misa, x0
  +
#103 0 M 0x8000000c 30102573 c 0301       00000001  csrrs x10, misa, x0
```

Sources: [printTrace.cpp 95-151](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L95-L151)[printTrace.cpp 271-506](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L271-L506)

### CSV Format Structure

CSV traces provide a more structured format for automated analysis:

```
pc, inst, modified regs, source operands, memory, inst info, privilege, trap, disassembly, hartid
```

**CSV Field Descriptions:**

| Field | Description | Format |
| --- | --- | --- |
| `pc` | Virtual PC (`:physical PC` if different) | `0x80000000` or `0x80000000:0x40000000` |
| `inst` | Instruction encoding | `0x00000013` |
| `modified regs` | Changed registers (`;` separated) | `x5=0x1234;c0x300=0x1800` |
| `source operands` | Source registers/immediates | `x10;x11;i0x100` |
| `memory` | Memory addresses (`;` for multiple) | `0x1000=0x42` or `0x1000:0x2000` |
| `inst info` | Instruction category flags | `l`, `s`, `a`, `f`, `v`, `t`, `nt`, `j`, `c`, `r` |
| `privilege` | Privilege mode | `M`, `S`, `U`, `VS`, `VU` |
| `trap` | Exception/interrupt info | (see Trap Format below) |
| `disassembly` | Instruction disassembly | `addi x5, x0, 0x1234` |
| `hartid` | System hart index | `0` |

**Instruction Info Flags:**

*   `l` - Load instruction
*   `s` - Store instruction
*   `a` - Atomic instruction
*   `f` - Floating-point instruction
*   `v` - Vector instruction
*   `t` - Branch taken
*   `nt` - Branch not taken
*   `j` - Unconditional jump
*   `c` - Call instruction
*   `r` - Return instruction

## Trace Configuration

### Enabling Tracing

Tracing is configured through `Hart` class methods:

**Standard Trace Mode:**

`hart.enableTrace(true);  // Enable standard format tracing`
**CSV Trace Mode:**

`hart.enableCsvTrace(true);  // Enable CSV format tracing`
**Page Table Walk Logging:**

`hart.enablePageTableWalkTrace(true);  // Log address translation`
**Deferred Tracing:**

`hart.setInstructionLogStart(1000000);  // Start tracing after 1M instructions`

## Traced Components

### Integer Register Changes

The trace captures the last written integer register and its value. For instructions that modify two registers (e.g., `amocas.q` in RV32), both registers are recorded:

```
#100 0 M 0x80000000 12345678 r 0a         00000001  amocas.q x10, x12, (x14)
  +
#100 0 M 0x80000000 12345678 r 0b         00000002  amocas.q x10, x12, (x14)
```

Sources: [printTrace.cpp 378-398](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L378-L398)

### Floating-Point Register Changes

FP register traces include the bit pattern of the FP value. For RV32 with only F extension, only 32 bits are shown. For RV64 or when D extension is enabled, 64 bits are shown:

```
# 32-bit FP value
#200 0 M 0x80000100 00052507 f 0a         3f800000  flw f10, 0(x10)

# 64-bit FP value
#201 0 M 0x80000104 00053507 f 0a 3ff0000000000000  fld f10, 0(x10)
```

Sources: [printTrace.cpp 400-409](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L400-L409)[printTrace.cpp 153-209](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L153-L209)

### Vector Register Changes

Vector register traces show the entire register contents in hexadecimal, with bytes printed from most significant to least significant:

```
#300 0 M 0x80000200 02052507 v 0a 0102030405060708090a0b0c0d0e0f10...  vle32.v v10, (x10)
```

For vector register groups (when LMUL > 1), each register in the group is traced separately:

```
#301 0 M 0x80000204 22052507 v 0a 0102030405060708090a0b0c0d0e0f10...  vle32.v v10, (x10)
 +
#301 0 M 0x80000204 22052507 v 0b 1112131415161718191a1b1c1d1e1f20...  vle32.v v10, (x10)
```

Sources: [printTrace.cpp 411-427](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L411-L427)[printTrace.cpp 48-92](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L48-L92)

### CSR Changes

CSR modifications are traced with the CSR number and new value. Multiple CSRs may be modified by a single instruction (e.g., exceptions update `*EPC`, `*CAUSE`, `*TVAL`):

```
#400 0 M 0x80000300 30529073 c 0305       00000001  csrw mtvec, x5
  +
#400 0 M 0x80000300 30529073 c 0341       80000304  csrw mtvec, x5
  +
#400 0 M 0x80000300 30529073 c 0342       00000002  csrw mtvec, x5
```

CSRs are sorted by CSR number in the trace output for consistency.

Sources: [printTrace.cpp 450-478](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L450-L478)

### Memory Accesses

Memory traces show the address and data for loads and stores. Both virtual and physical addresses are shown when they differ:

**Load (no trace, address shown in brackets):**

```
#500 0 M 0x80000400 00052503 r 0a         00000042  lw x10, 0(x10) [0x1000:0x2000]
```

**Store (memory change traced):**

```
#501 0 M 0x80000404 00a52023 m 00001000   00000042  sw x10, 0(x10)
```

**Vector memory operations:**

```
#502 0 M 0x80000408 02850527 r 0a ... vle8.v v10, (x10) [0x1000:0x2000;0x1001:0x2001;...]
```

## Page Table Walk Logging

When page table walk tracing is enabled (`tracePtw_`), address translation information is appended to instruction traces:

### Page Table Walk Format

Page table walks are logged with the prefix `iptw` (instruction fetch) or `dptw` (data access):

```
#600 0 VS 0x80000500 00052503 r 0a         00000001  lw x10, 0(x10)
  +
satp mode: sv39
  +
vsatp mode: sv39
  +
hgatp mode: sv39
  +
dptw:
  +
  gva: 0x1000=0x200000a1, ma=read,write,exec,idempotent,amoother,amoswap,amological,rsrv,misalok,
  +
   gpa: 0x40001000=0x080000a1, ma=read,write,exec,idempotent,amoother,amoswap,amological,rsrv,misalok,
  +
    pa: 0x80002000=0x00000021, ma=read,write,exec,idempotent,amoother,amoswap,amological,rsrv,misalok,
```

**Walk Entry Format:**

Each walk entry shows:

*   **Address type**: `gva` (guest virtual), `gpa` (guest physical), or `pa` (physical)
*   **Address**: The address being translated
*   **PTE value**: Page table entry contents (for `pa` lines)
*   **Memory attributes** (`ma`): Physical memory attributes from PMA

Sources: [printTrace.cpp 212-266](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L212-L266)[printTrace.cpp 483-506](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L483-L506)

### Translation Modes

The translation mode for each level is logged:

| Mode | Description |
| --- | --- |
| `Bare` | No translation |
| `Sv32` | 32-bit virtual addressing |
| `Sv39` | 39-bit virtual addressing |
| `Sv48` | 48-bit virtual addressing |
| `Sv57` | 57-bit virtual addressing |

## Trace Output Control

### Buffering and Thread Safety

Whisper uses buffering to optimize trace output performance:

**Shared Buffer Mode (default):**

*   Single shared buffer for all harts
*   Mutex serialization prevents interleaved output
*   Used when `ownTrace_` is false

**Per-Hart Buffer Mode:**

*   Each hart has its own `PrintBuffer`
*   No serialization overhead
*   Used when `ownTrace_` is true
*   Must externally serialize final output

Sources: [printTrace.cpp 16-20](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L16-L20)[printTrace.cpp 318](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L318-L318)[printTrace.cpp 511-586](https://github.com/tenstorrent/whisper/blob/733fd921/printTrace.cpp#L511-L586)

### Trace Extension Hook

The tracer supports external extensions through a function pointer:

`extern void (*__tracerExtension)(void*);`
If set, this function is called for each instruction with a `TraceRecord<URV>` pointer, allowing external code to capture or process trace data before it's written to the output file.

### Privilege Mode Encoding

The privilege mode string indicates the execution context:

| String | Meaning |
| --- | --- |
| `M` | Machine mode |
| `HS` or `S` | Hypervisor/Supervisor mode (non-virtualized) |
| `U` | User mode (non-virtualized) |
| `VS` | Virtual Supervisor mode |
| `VU` | Virtual User mode |
| `D` | Debug mode |

The mode reflects the privilege at instruction fetch time, accounting for virtual mode (`virtMode_`) and hypervisor extension state.
