## Purpose and Scope

This page provides a comprehensive reference for all Matrix Unit (FPU) instructions available in the Tensix Coprocessor. The Matrix Unit performs low-precision (≤19-bit) arithmetic operations on matrices stored in [`SrcA`, `SrcB`](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/%60SrcA%60,%20%60SrcB%60) and [`Dst`](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/%60Dst%60) registers.

For information about the Matrix Unit architecture and pipeline, see [Matrix Unit (FPU) Architecture](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.4-matrix-unit-(fpu)-architecture). For Vector Unit instructions (higher precision, lower throughput), see [Vector Unit Core Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.3-vector-unit-core-instructions). For unpacker/packer instructions that move data to/from these registers, see [Unpacker Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.6-unpacker-instructions-(unpacr)) and [Packer Instructions](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.7-packer-instructions-(pacr)).

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 1-10](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L1-L10)

## Common Instruction Elements

### RWC Addressing

Most Matrix Unit instructions use RWCs (Row/Write Counters) to specify which rows to operate on. Each thread has its own set of RWCs:

`struct {  uint10_t Dst, Dst_Cr;  uint6_t SrcA, SrcA_Cr;  uint6_t SrcB, SrcB_Cr;  uint2_t FidelityPhase;  uint1_t ExtraAddrModBit;} RWCs[3];  // One per thread`
Instructions typically combine RWC base values with immediate offsets to compute final row addresses. The 2-bit `AddrMod` field in most instructions specifies how to increment RWCs after execution.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/RWCs.md 1-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/RWCs.md?plain=1#L1-L13)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 7-8](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L7-L8)

### Fidelity Phases

Floating-point operations may require multiple fidelity phases to achieve full precision. The `FidelityPhase` field in RWCs tracks which phase is active:

*   **1 phase:** Best performance, lower precision
*   **2 phases:** Medium performance/precision tradeoff
*   **3-4 phases:** Highest precision, lowest performance

Integer operations also use fidelity phases based on input magnitude ranges.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 33-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L33-L74)

## Compute Instructions

### Matrix Multiplication Instructions

| Instruction | Operation | Throughput | Latency | Key Characteristics |
| --- | --- | --- | --- | --- |
| `MVMUL` | Matrix multiply: `Dst += SrcA × SrcB` | 1 IPC | 5 cycles | Main workhorse, 4.096 TFLOP/s peak (1 phase, BroadcastSrcBRow=false) |
| `DOTPV` | Dot product with broadcast | 1 IPC | 5 cycles | 4.096 TFLOP/s peak, similar to MVMUL |
| `GAPOOL` | Accumulate with averaging | 1 IPC | 5 cycles | 2.048 TFLOP/s peak |

**Note:** Throughput is per fidelity phase. Multi-phase operations require multiple instructions, reducing effective IPC.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 22-33](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L22-L33)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 58-72](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L58-L72)

### Pooling Instructions

| Instruction | Operation | Throughput | Latency | Key Characteristics |
| --- | --- | --- | --- | --- |
| `GMPOOL` | Max reduction along columns | 1 IPC | 5 cycles | 0.256 TFLOP/s, does not use fidelity phases |

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 3-4](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L3-L4)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 67-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L67-L71)

### Element-wise Arithmetic Instructions

| Instruction | Operation | Throughput | Latency | AddDst Flag Effect |
| --- | --- | --- | --- | --- |
| `ELWMUL` | Element-wise multiply: `Dst += SrcA ⊙ SrcB` | 1 IPC | 5 cycles | 0.256 TFLOP/s (1 phase) |
| `ELWADD` | Element-wise add: `Dst = SrcA + SrcB` (or `Dst += SrcA + SrcB`) | 1 IPC | 5 cycles | 0.256 TFLOP/s if AddDst=true, else 0.128 |
| `ELWSUB` | Element-wise subtract: `Dst = SrcA - SrcB` (or `Dst += SrcA - SrcB`) | 1 IPC | 5 cycles | 0.256 TFLOP/s if AddDst=true, else 0.128 |

### Example: MOVB2A Instruction

The `MOVB2A` instruction demonstrates typical Matrix Unit instruction structure:

**Functional model excerpt:**

`// Wait for SrcB accesswhile (SrcB[MatrixUnit.SrcBBank].AllowedClient != MatrixUnit) {  wait;} // Calculate row addresses using RWCsSrcARow += RWCs[CurrentThread].SrcA;SrcBRow += RWCs[CurrentThread].SrcB; // Copy data with optional denormal flushingfor (unsigned Column = 0; Column < 16; ++Column) {  if (LaneConfig[Column / 2].BLOCK_DEST_MOV.Bit[Column & 1]) continue;  uint19_t SrcVal = SrcB[MatrixUnit.SrcBBank][SrcBRow][Column];  if (FlushDenormals && !(SrcVal & 0xff)) SrcVal = 0;  SrcA[MatrixUnit.SrcABank][SrcARow][Column] = SrcVal;} // Update RWCs using AddrModApplyAddrMod(AddrMod);`

## Performance Characteristics

### Peak Throughput (1 GHz, Wormhole B0)

The following table shows theoretical maximum performance for different instruction types and fidelity phase configurations:

| Instruction | 1 Phase | 2 Phases | 3 Phases | 4 Phases |
| --- | --- | --- | --- | --- |
| `MVMUL` (BroadcastSrcBRow=false) | 4.096 TFLOP/s | 2.048 TFLOP/s | 1.366 TFLOP/s | 1.024 TFLOP/s |
| `DOTPV` | 4.096 TFLOP/s | 2.048 TFLOP/s | 1.366 TFLOP/s | 1.024 TFLOP/s |
| `GAPOOL` | 2.048 TFLOP/s | 1.024 TFLOP/s | 0.683 TFLOP/s | 0.512 TFLOP/s |
| `MVMUL` (BroadcastSrcBRow=true) | 0.560 TFLOP/s | 0.280 TFLOP/s | 0.187 TFLOP/s | 0.140 TFLOP/s |
| `ELWMUL` | 0.256 TFLOP/s | 0.128 TFLOP/s | 0.085 TFLOP/s | 0.064 TFLOP/s |
| `GMPOOL` | 0.256 TFLOP/s | (same) | (same) | (same) |
| `ELWADD`/`ELWSUB` (AddDst=true) | 0.256 TFLOP/s | (same) | (same) | (same) |
| `ELWADD`/`ELWSUB` (AddDst=false) | 0.128 TFLOP/s | (same) | (same) | (same) |

**Notes:**

*   `GMPOOL`, `ELWADD`, and `ELWSUB` do not require multiple fidelity phases
*   Integer operations have the same TOP/s performance as floating-point TFLOP/s
*   For comparison, Vector Unit `SFPMAD` achieves 0.064 TFLOP/s at FP32 precision

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 56-74](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L56-L74)

### System-Level Performance

To calculate total ASIC performance, multiply the per-Matrix-Unit numbers by the tile count:

*   **Wormhole products:** 64, 72, or 80 Tensix tiles depending on SKU
*   **Example:** n150 board = 1 ASIC × 72 tiles = 294.912 TFLOP/s peak (MVMUL, 1 phase)
*   **Example:** n300 board = 2 ASICs × 64 tiles = 524.288 TFLOP/s peak (MVMUL, 1 phase)
*   **Example:** Galaxy server = 32 ASICs × 80 tiles = 10,485.76 TFLOP/s peak (MVMUL, 1 phase)

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 74-76](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L74-L76)

## Wait Gate and Instruction Scheduling

Matrix Unit instructions must respect register bank access protocols:

**Key points:**

*   Compute instructions (`MVMUL`, `ELWMUL`, etc.) automatically wait for `SrcB` bank access
*   Data movement to `SrcA` may require manual `STALLWAIT` instruction (condition code C10, block bit B6)
*   Some instructions have scheduling restrictions (e.g., `MOVB2A` limits next-cycle options)

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 22-28](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L22-L28)[WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 65-68](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L65-L68)

The following instructions exist for backward compatibility but perform minimal operations:

| Instruction | Actual Behavior | Status |
| --- | --- | --- |
| `MFCONV3S1`, `CONV3S1`, `CONV3S2` | Compute `Dst += 0` | NonContractualBehavior |

## Interaction with Other Execution Units

Matrix Unit instructions operate on registers that are also accessible by other execution units:

**Key interactions:**

*   **Unpackers:** Move data from L1 to `SrcA`, `SrcB`, or `Dst`
*   **Packers:** Move data from `Dst` to L1
*   **Vector Unit:** Operates on `Dst` via `LReg` intermediate storage
*   **Scalar Unit:** Can write individual elements to `SrcA`/`SrcB` via `STOREIND`

Proper coordination between units requires careful instruction scheduling to avoid data hazards.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 37-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L37-L46)

## Programming Considerations

### Instruction Throughput vs. Latency

While most Matrix Unit instructions have 1 IPC throughput, their 4-5 cycle latencies mean:

*   New instructions can be issued every cycle
*   Results are not available for several cycles
*   Software must schedule independent operations to hide latency
*   Pipeline stalls occur if dependent operations are scheduled too close together

### AddrMod Usage Strategy

Efficient kernels minimize standalone `SETRWC`/`INCRWC` instructions by:

1.   Pre-configuring `ThreadConfig.ADDR_MOD_*_SEC` arrays with common increment patterns
2.   Using the `AddrMod` field in compute instructions to auto-increment RWCs
3.   Exploiting `ExtraAddrModBit` to access 8 different increment patterns
4.   Structuring loops to align with available increment patterns

### Fidelity Phase Selection

Choose fidelity phase count based on precision requirements:

*   **1 phase:** Maximum performance for approximate results (common in neural network training)
*   **2 phases:** Common choice balancing performance and accuracy
*   **3-4 phases:** Full precision, use only when required
