# Dispatch System and Optimization

Relevant source files
*   [operations/dispatch_overhead.md](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1)

This document provides comprehensive guidance on optimizing the dispatch system in Tenstorrent hardware programming. It covers fast dispatch mechanisms, kernel initialization strategies, resource allocation optimization, and platform-specific considerations for maximizing performance while minimizing dispatch overhead.

For general TT-Metalium programming concepts, see [TT-Metalium Programming Model](https://deepwiki.com/tenstorrent/tt-tutorial/4.1-tt-metalium-programming-model). For broader performance optimization strategies, see [Performance and Optimization](https://deepwiki.com/tenstorrent/tt-tutorial/7-performance-and-optimization).

## Fast Dispatch Architecture

The fast dispatch system enables efficient kernel execution through a two-stage communication protocol between the dispatch unit and worker cores. Understanding this architecture is essential for optimizing kernel execution pipelines.

### Core Communication Protocol

**Launch Message Components:**

*   `k_id` (Kernel ID)
*   Number of circular buffers
*   Number of runtime arguments
*   Configuration metadata

**Go Message:**

*   Simple execution trigger
*   Sent only after previous kernel completion
*   Synchronous with kernel execution start

Sources: [operations/dispatch_overhead.md 13-36](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L13-L36)

### Fast Dispatch Timeline

The dispatch system operates on overlapping execution phases to maximize throughput:

**Timing Considerations:**

*   **Covered latency**: Worker initialization completes within handshake window (~700ns)
*   **Uncovered latency**: Initialization exceeds handshake time, causing execution delays
*   **Optimal case**: Launch message arrives before kernel N completion

Sources: [operations/dispatch_overhead.md 37-60](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L37-L60)

## Dispatch Performance Scenarios

### Best-Case Handshake

In optimal conditions, the dispatcher and worker cores maintain perfect synchronization with minimal latency.

### Dispatcher Behind Scenario

When multiple short-duration kernels execute rapidly, the dispatcher may lag behind worker completion.

**Impact**: Tight workloads with small kernels expose dispatcher lag, reducing overall throughput.

### Ring Buffer Overflow

When kernel N and N+1 cannot fit simultaneously in the worker's ring buffer:

**Symptoms**: Dispatch times in 7-15 microsecond range indicate ring buffer constraints.

Sources: [operations/dispatch_overhead.md 65-114](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L65-L114)

## Worker Core Initialization

### NCRISC Initialization (Wormhole)

On Wormhole platforms, `NCRISC` worker cores execute kernels from `IRAM`, requiring binary copy from L1 memory.

**Copy Performance**: ~10 bytes per cycle

| Binary Size | Estimated Cycles | Time |
| --- | --- | --- |
| ~2 KB (best) | ~200 | ~200 ns |
| ~4 KB (common) | ~400 | ~400 ns |
| ~16 KB (worst) | ~1600 | ~1600 ns |

**Optimization**: Place smaller binaries on `NCRISC` to minimize IRAM copy time.

Sources: [operations/dispatch_overhead.md 158-176](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L158-L176)

### BRISC vs NCRISC Role Assignment

Strategic core assignment optimizes initialization timing in reader/writer pipelines:

**Strategy**:

*   `NCRISC`: Reader (smaller, faster copy)
*   `BRISC`: Writer (larger, no IRAM copy)

**When to reverse roles**:

*   Dispatcher behind schedule
*   Ring buffer cannot hold N and N+1 simultaneously
*   Launch message delays prevent early copy

Sources: [operations/dispatch_overhead.md 177-217](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L177-L217)

### Circular Buffer Configuration

CB initialization adds measurable latency during worker setup:

| Platform | Time per CB | Worst Case (32 CBs) |
| --- | --- | --- |
| Wormhole | ~25 cycles | ~800 ns |
| Blackhole | ~20 cycles | ~640 ns |

**Optimization tips**:

*   Avoid unused CB requests
*   Pack CB indices efficiently
*   Minimize CB configuration overhead

Sources: [operations/dispatch_overhead.md 218-240](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L218-L240)

## Kernel Groups and Dispatch Scaling

### Kernel Group Definition

A kernel group consists of cores receiving identical kernel binaries. Variations in compile-time arguments create separate groups.

**Dispatch implications**:

*   Dispatcher loops through each kernel group
*   More groups = higher NoC bandwidth usage
*   Binary sizes up to 8KB per group

Sources: [operations/dispatch_overhead.md 241-262](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L241-L262)

### Physical Layout Constraints

Core arrangement affects multicast efficiency and kernel group count:

**Platform differences**:

*   **Wormhole**: Must split non-rectangular shapes into multiple rectangular groups
*   **Blackhole**: Supports L-shaped multicast, reducing group count

Sources: [operations/dispatch_overhead.md 263-284](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L263-L284)

## NoC Performance and Runtime Arguments

### NoC Flit Structure

Network-on-Chip communication uses Flow Control Units (flits) for data transmission:

| Platform | Flit Size | Bandwidth (1 GHz) |
| --- | --- | --- |
| Wormhole | 32 bytes | 32 GB/s |
| Blackhole | 64 bytes | 64 GB/s |

Sources: [operations/dispatch_overhead.md 285-303](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L285-L303)

### Runtime Arguments Performance

RTA dispatch cost exhibits characteristic patterns based on argument count:

**Key metrics**:

*   Each RTA: 4 bytes
*   First RTA adds full 32B flit
*   Additional RTAs free until flit boundary
*   Cost example: 256 RTAs × 4B × 3 kernels × 64 cores = 196 KB dispatch data

**Optimization strategies**:

*   **Multicast common RTAs**: Reduces bandwidth, increases dispatch cycles
*   **Unique RTAs**: Fewer dispatch cycles, higher bandwidth usage
*   Choose based on workload characteristics

Sources: [operations/dispatch_overhead.md 305-327](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L305-L327)

### RTAs vs Kernel Groups Trade-off

**Guidelines**:

*   Use RTAs for small kernel modifications with good code reuse
*   Avoid RTAs for large "god kernels" with minimal sharing
*   Consider bandwidth vs execution time trade-offs

Sources: [operations/dispatch_overhead.md 328-348](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L328-L348)

## Platform-Specific Considerations

### T3K Ethernet Dispatch Limitations

T3K platforms use Ethernet cores for dispatch, with significant performance constraints compared to Tensix dispatch:

**Performance impact**:

*   Ethernet dispatch will never match Tensix performance
*   Higher latency and reduced throughput
*   Design dispatch strategies to mitigate bottlenecks

Sources: [operations/dispatch_overhead.md 391-416](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L391-L416)

## Best Practices and Optimization Guidelines

### Global Initialization Optimization

Static initialization outperforms runtime assignment for global variables:

**Benefits**:

*   Densely packed during load time
*   Eliminates instruction overhead
*   Reduces binary size

Sources: [operations/dispatch_overhead.md 349-390](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L349-L390)

### Resource Contention Management

Fast dispatch creates resource contention between dispatch and execution:

**Mitigation strategies**:

*   Monitor dispatch costs in memory-heavy workloads
*   Balance dispatch timing with execution requirements
*   Consider workload characteristics when optimizing

Sources: [operations/dispatch_overhead.md 115-128](https://github.com/tenstorrent/tt-tutorial/blob/389c542c/operations/dispatch_overhead.md?plain=1#L115-L128)

The dispatch system optimization requires careful balancing of multiple factors including timing, resource usage, and platform constraints. Effective optimization involves profiling workload-specific behavior and applying targeted improvements based on bottleneck analysis.
