The D2M (Direct-to-Metal) regression framework provides automated validation of the `tt-mlir` compiler toolchain targeting `tt-emule`. It ensures that the MLIR-generated kernels, which are compiled "Direct-to-Metal" (bypassing high-level `ttnn` ops where possible), execute correctly on the emulated hardware model. This integration is critical for verifying the compiler's code generation, NOC addressing, and SFPU instruction mapping.

## Overview and Dependency Pinning

The D2M regression relies on a three-way dependency pin to ensure stability across three repositories. This prevents "cascading breakages" where an upstream change in `tt-metal` or `tt-mlir` breaks the emulator's CI.

1.   **tt-emule**: The root of the PR/testing.
2.   **tt-mlir-pin.txt**: Contains a specific SHA of `tenstorrent/tt-mlir`[tt-mlir-pin.txt 1-10](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tt-mlir-pin.txt#L1-L10)
3.   **Derived tt-metal SHA**: The CI dynamically extracts the `TT_METAL_VERSION` from `tt-mlir/third_party/CMakeLists.txt` at the pinned MLIR SHA [.github/workflows/pr-d2m-regression.yml 59-67](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L59-L67)

This ensures that `tt-emule` is always tested against a version of `tt-mlir` and `tt-metal` that are known to be compatible with each other.

## The Regression Runner (`run_d2m_regression.sh`)

The `run_d2m_regression.sh` script is the primary entry point for executing the D2M suite. It manages environment setup, parallel execution, and hang detection.

### Key Implementation Details

*   **Parallelism and Isolation**: Tests are run in parallel using `pytest` with the `--forked` flag to ensure process isolation [run_d2m_regression.sh 156-157](https://github.com/tenstorrent/tt-emule/blob/a57cf072/run_d2m_regression.sh#L156-L157)
*   **JIT Cache Isolation**: To prevent race conditions or stale kernel artifacts during parallel compilation, the script clears the JIT cache (`/tmp/tt_emule_jit_*`) and source directories [run_d2m_regression.sh 100-102](https://github.com/tenstorrent/tt-emule/blob/a57cf072/run_d2m_regression.sh#L100-L102)
*   **Two-Tier Timeouts**: 
    *   **Pytest Timeout**: A 300s (default) per-test-case limit enforced via `pytest-timeout`[run_d2m_regression.sh 27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/run_d2m_regression.sh#L27-L27)
    *   **File Timeout**: A 3600s (default) bash-level ceiling using the `timeout` command to catch hard hangs in native code that Python threads cannot interrupt [run_d2m_regression.sh 28-137](https://github.com/tenstorrent/tt-emule/blob/a57cf072/run_d2m_regression.sh#L28-L137)

*   **Architecture Selection**: Controlled via `TT_EMULE_ARCH`, which selects the appropriate cluster descriptor (e.g., `wormhole_N150.yaml` or `blackhole_P100.yaml`) [run_d2m_regression.sh 17-22](https://github.com/tenstorrent/tt-emule/blob/a57cf072/run_d2m_regression.sh#L17-L22)

Sources: `run_d2m_regression.sh:17-30`(), `run_d2m_regression.sh:100-112`(), `run_d2m_regression.sh:137-157`()

## Result Classification and Allowlists

Because the D2M suite contains thousands of tests, `tt-emule` uses an allowlist-based system to track known issues without blocking CI.

### Classification Logic (`classify-d2m-results.py`)

The classifier parses JUnit XML files generated by the runner and categorizes every test into one of four buckets [.github/scripts/classify-d2m-results.py 148-170](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/scripts/classify-d2m-results.py#L148-L170):

1.   **Passed**: Test passed and was either not in the allowlist or was marked as flaky.
2.   **New Failures**: Test failed but is not in the allowlist. **(Fails CI)**.
3.   **Newly-Passing**: Test passed but is still in the allowlist. This detects when a fix has improved the emulator or compiler. **(Fails CI)**.
4.   **Stale Entries**: An entry in the allowlist matched no tests in the current run (usually due to a test being renamed or removed). **(Fails CI)**.

### Allowlist Format

The allowlists (`.github/known-failures-d2m-wormhole.txt` and `blackhole.txt`) support `fnmatch` wildcards and a special `flaky:` prefix.

*   **`flaky:`**: Suppresses the "Newly-Passing" error. This is used for tests that oscillate between PASS and FAIL due to non-deterministic behavior or environment issues [.github/known-failures-d2m-blackhole.txt 6-9](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/known-failures-d2m-blackhole.txt#L6-L9)

## CI Workflows

The D2M regression is exercised in two primary GitHub Action workflows:

1.   **PR D2M Regression**: Runs on every pull request. It builds the pinned `tt-mlir` against the PR's `tt-emule` and fails if the classification script detects any regressions or stale allowlist entries [.github/workflows/pr-d2m-regression.yml 1-6](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/pr-d2m-regression.yml#L1-L6)
2.   **Nightly D2M Upstream**: Runs daily against the `main` tip of `tt-mlir`. This acts as an early warning system for upstream changes that might require an allowlist update or an emulator fix before the next pin bump [.github/workflows/nightly-d2m-upstream.yml 1-6](https://github.com/tenstorrent/tt-emule/blob/a57cf072/.github/workflows/nightly-d2m-upstream.yml#L1-L6)
