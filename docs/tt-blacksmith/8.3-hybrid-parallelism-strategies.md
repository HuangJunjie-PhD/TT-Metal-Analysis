## Purpose and Scope

This page documents how to combine data parallelism and tensor parallelism in TT-Blacksmith for large-scale model training on multi-chip hardware. Hybrid parallelism enables training models that require both model sharding (to fit the model in memory) and data parallelism (to maximize throughput).

For information about data parallelism alone, see [Data Parallelism](https://deepwiki.com/tenstorrent/tt-blacksmith/8.1-data-parallelism). For information about tensor parallelism alone, see [Tensor Parallelism](https://deepwiki.com/tenstorrent/tt-blacksmith/8.2-tensor-parallelism). For the underlying hardware abstraction layer, see [Device Management and Hardware Abstraction](https://deepwiki.com/tenstorrent/tt-blacksmith/5.2-device-management-and-hardware-abstraction).

## Overview

Hybrid parallelism combines two orthogonal parallelization strategies across a 2D device mesh:

*   **Data Parallelism**: Replicates the model across devices and shards the training data
*   **Tensor Parallelism**: Shards the model weights across devices and replicates the data

This is implemented through a 2D mesh with shape `[D, M]` where:

*   `D` = number of data parallel replicas
*   `M` = number of model parallel partitions
*   Total devices = `D * M`

The `DeviceManager` class determines which parallelization strategies are active based on mesh configuration and applies appropriate sharding operations.

**Sources:**[blacksmith/tools/device_manager.py 19-137](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L19-L137)[docs/src/experiments.md 1-58](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L1-L58)

## Mesh Configuration

### Mesh Shape and Axis Names

Hybrid parallelism is configured through two related parameters in `TrainingConfig`:

`mesh_shape: [8, 4]           # 8 devices in first dimension, 4 in secondmesh_axis_names: ["data", "model"]  # First dimension is data parallel, second is model parallel`
The `DeviceManager` creates a 2D mesh from these parameters:

**Sources:**[blacksmith/tools/device_manager.py 51-69](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L51-L69)[blacksmith/experiments/torch/llama/configs.py 76-83](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/configs.py#L76-L83)

### Axis Name Ordering

The `mesh_axis_names` can be specified in either order: `["data", "model"]` or `["model", "data"]`. The position in the list corresponds to the position in `mesh_shape`:

| mesh_shape | mesh_axis_names | Data Parallel Devices | Model Parallel Devices |
| --- | --- | --- | --- |
| [8, 4] | ["data", "model"] | 8 | 4 |
| [8, 4] | ["model", "data"] | 4 | 8 |
| [2, 4] | ["data", "model"] | 2 | 4 |

The `DeviceManager` determines which strategy is active by checking axis names, not positions:

`def is_data_parallel(self) -> bool:    return self.mesh is not None and "data" in self.mesh.axis_names and self.mesh.shape()["data"] > 1 def is_tensor_parallel(self) -> bool:    return self.mesh is not None and "model" in self.mesh.axis_names and self.mesh.shape()["model"] > 1`
**Sources:**[blacksmith/tools/device_manager.py 71-77](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L71-L77)[blacksmith/experiments/torch/llama/xla/lora/README.md 20-34](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L20-L34)

## Data Dimension Sharding

When `is_data_parallel()` returns `True`, the `DeviceManager.prepare_batch()` method shards input batches along the data dimension:

Each device receives `B / num_data_parallel_devices` samples. The model processes its local batch, computes gradients, and the gradients are all-reduced across the data parallel dimension during `optimizer_step()`.

**Sources:**[blacksmith/tools/device_manager.py 115-125](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L115-L125)[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 173-176](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L173-L176)

## Model Dimension Sharding

When `is_tensor_parallel()` returns `True`, the `DeviceManager.shard_model()` method applies model sharding patterns to the model weights:

Sharding patterns use regex to match module names and apply partition specifications:

`model_sharding_patterns:  - ['\.self_attn\.q_proj\.base_layer$',  ["model", null]]  # Column-parallel  - ['\.self_attn\.o_proj$',              [null, "model"]]  # Row-parallel  - ['\.mlp\.gate_proj$',                 ["model", null]]  # Column-parallel  - ['\.mlp\.down_proj$',                 [null, "model"]]  # Row-parallel`
*   `["model", null]`: Shard along dimension 0 (column-parallel)
*   `[null, "model"]`: Shard along dimension 1 (row-parallel)

**Sources:**[blacksmith/tools/device_manager.py 82-110](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L82-L110)[blacksmith/experiments/torch/llama/xla/lora/README.md 27-34](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/lora/README.md?plain=1#L27-L34)

## Communication Patterns

### All-Reduce in Data Parallel Dimension

During `optimizer_step()`, gradients are synchronized across the data parallel dimension:

The `xm.optimizer_step(optimizer, barrier=True)` call ensures gradients are properly synchronized:

**Sources:**[blacksmith/tools/device_manager.py 127-136](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L127-L136)

### All-Reduce in Model Parallel Dimension

Tensor parallel layers use implicit all-reduce operations during forward and backward passes:

*   **Column-parallel** layers (e.g., `q_proj`, `gate_proj`): No communication in forward pass, all-reduce in backward pass
*   **Row-parallel** layers (e.g., `o_proj`, `down_proj`): All-reduce in forward pass, no communication in backward pass

This follows the column-row-column pattern described in [Tensor Parallelism](https://deepwiki.com/tenstorrent/tt-blacksmith/8.2-tensor-parallelism).

**Sources:**[blacksmith/tools/device_manager.py 89-110](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/device_manager.py#L89-L110)

## Performance Considerations

### When to Use Hybrid Parallelism

| Scenario | Recommended Strategy | Reason |
| --- | --- | --- |
| Model fits on single chip | None | No parallelism needed |
| Model fits but want throughput | Data Parallel only | Maximize data throughput |
| Model doesn't fit on single chip | Tensor Parallel only | Required for memory |
| Large model + need throughput | Hybrid DP+TP | Combine memory and throughput benefits |
| Very large models (>7B) | Tensor Parallel required | Must shard to fit, add DP for throughput |

### Mesh Shape Selection

For a given number of devices `N`, consider:

*   **Square meshes** (e.g., [4, 4]): Balanced communication in both dimensions
*   **More data parallel** (e.g., [8, 2]): Better throughput if model fits with minimal sharding
*   **More tensor parallel** (e.g., [2, 8]): Required when model is too large

Example for 32-device Galaxy:

*   [8, 4]: Good balance for models like Llama 3.2 1B
*   [4, 8]: Better for larger models that need more sharding
*   [32, 1]: Maximum throughput (data parallel only) if model fits

## Environment Variables for Multi-chip

The `DeviceManager` sets required environment variables for multi-chip execution when a mesh is configured:

`def _setup_tt_environment(self):    os.environ["PJRT_DEVICE"] = "TT"    os.environ["XLA_STABLEHLO_COMPILE"] = "1"        # Additional setup for multichip    if hasattr(self.config, "mesh_shape") and self.config.mesh_shape is not None:        os.environ["XLA_ALWAYS_ALLREDUCE"] = "1"        os.environ["CONVERT_SHLO_TO_SHARDY"] = "1"        os.environ["DISABLE_NUMERIC_CC_TOKEN"] = "1"        xr.use_spmd()`
These enable SPMD (Single Program Multiple Data) compilation and proper collective communication.
