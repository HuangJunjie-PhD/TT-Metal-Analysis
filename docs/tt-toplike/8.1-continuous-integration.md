# Continuous Integration

Relevant source files
*   [.gitattributes](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.gitattributes)
*   [.github/workflows/ci.yml](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml)

The `tt-toplike` project utilizes GitHub Actions to maintain code quality, ensure build reproducibility, and enforce style consistency across its various components (TUI, GUI, and core library). The Continuous Integration (CI) pipeline is designed to be "contributor-friendly" by focusing enforcement on changed files while maintaining a strict zero-warning policy for new code.

## CI Workflow Overview

The CI process is triggered on every `push` to the `main` branch and for all `pull_request` events targeting `main`[.github/workflows/ci.yml 3-7](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L3-L7) The workflow runs on the `ubuntu-24.04` environment to ensure compatibility with modern toolchains and system dependencies [.github/workflows/ci.yml 12](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L12-L12)

### Workflow Stages

The CI pipeline follows a sequential execution of quality gates:

1.   **Environment Setup**: Initializes the Rust stable toolchain with `rustfmt` and `clippy` components [.github/workflows/ci.yml 19-22](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L19-L22)
2.   **Differential Formatting Check**: A specialized script that checks `rustfmt` compliance only for files modified in the current PR [.github/workflows/ci.yml 24-52](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L24-L52)
3.   **Static Analysis**: Runs `clippy` on the core library and TUI binary with `-D warnings` to deny all lints [.github/workflows/ci.yml 54-55](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L54-L55)
4.   **Unit Testing**: Executes library-level tests using the `--locked` flag to ensure `Cargo.lock` consistency [.github/workflows/ci.yml 57-58](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L57-L58)
5.   **Binary Compilation**: Performs parallel-ready builds of both the TUI and GUI binaries to verify that feature-flagged code paths compile without warnings [.github/workflows/ci.yml 60-69](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L60-L69)

### Differential Formatting Logic

To avoid blocking contributors with "formatting debt" (pre-existing style issues in files they did not touch), the CI implements a differential check. It calculates the merge base between the PR head and `origin/main` to identify specific `.rs` files changed in the changeset, excluding the `vendor/` directory [.github/workflows/ci.yml 29-31](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L29-L31)

## Build Targets and Feature Flags

The CI pipeline explicitly tests the primary entry points of the project using specific Cargo feature flags. This ensures that the decoupling between the telemetry backends and the frontend layers remains intact.

| Job Step | Command | Features | Goal |
| --- | --- | --- | --- |
| **Clippy** | `cargo clippy` | `tui` | Lint core logic and TUI implementation [.github/workflows/ci.yml 55](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L55-L55) |
| **Tests** | `cargo test` | `tui` | Verify library logic and mock backends [.github/workflows/ci.yml 58](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L58-L58) |
| **TUI Build** | `cargo build` | `tui` | Verify `tt-toplike-tui` binary compilation [.github/workflows/ci.yml 61](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L61-L61) |
| **GUI Build** | `cargo build` | `gui` | Verify `tt-toplike-gui` (iced) binary compilation [.github/workflows/ci.yml 66](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L66-L66) |

### Strict Warning Enforcement

For all build and clippy steps, the CI sets `RUSTFLAGS: -D warnings`[.github/workflows/ci.yml 63-68](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L63-L68) This transforms all compiler warnings (unused variables, dead code, etc.) into hard errors, maintaining a high standard for the codebase health.

## Build Reproducibility

The use of the `--locked` flag in `cargo clippy`, `cargo test`, and `cargo build` commands ensures that the CI environment uses the exact dependency versions specified in the `Cargo.lock` file [.github/workflows/ci.yml 55-66](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L55-L66) This prevents "it works on my machine" issues caused by upstream dependency updates. Additionally, the `.gitattributes` file marks the `vendor/` directory as `linguist-generated=true` to ensure it does not clutter diffs or analysis, while the CI formatting script explicitly ignores it to avoid checking third-party code [.gitattributes 1](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.gitattributes#L1-L1)[.github/workflows/ci.yml 31](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/.github/workflows/ci.yml#L31-L31)
