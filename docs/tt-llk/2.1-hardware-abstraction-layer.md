## Purpose and Scope

The Hardware Abstraction Layer (HAL) provides the foundational C++ API for controlling Tensix cores in the tt-llk library. This layer is defined primarily in [tt_llk_wormhole_b0/common/inc/ckernel.h](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h) and [tt_llk_blackhole/common/inc/ckernel.h](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h) and exposes primitives for:

*   **Memory access** with correct ordering semantics (blocking loads/stores)
*   **Configuration state management** for double-buffered ping-pong contexts
*   **Register access** for hardware configuration and status
*   **Hardware initialization** (zeroing accumulators, source registers)
*   **Address space mapping** across L1, L0, and configuration regions

For higher-level synchronization mechanisms (semaphores, mutexes), see [Synchronization Primitives](https://deepwiki.com/tenstorrent/tt-llk/2.2-synchronization-primitives). For specific hardware unit configuration, see the Unpacker ([#4](https://deepwiki.com/tenstorrent/tt-llk/4-unpacker-system)), Math ([#5](https://deepwiki.com/tenstorrent/tt-llk/5-math-unit-operations)), and Packer ([#6](https://deepwiki.com/tenstorrent/tt-llk/6-packer-system)) system documentation.

## Memory Access Primitives

The HAL provides memory ordering primitives that ensure correct transaction completion on RISC-V cores without hardware cache coherency.

### Blocking Load and Store Operations

The `load_blocking<T>()` and `store_blocking<T>()` templates enforce memory ordering through inline assembly sequences:

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 92-122](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L92-L122)[tt_llk_blackhole/common/inc/ckernel.h 101-168](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L101-L168)

**Key Implementation Details:**

| Function | Purpose | Wormhole B0 Sequence | Blackhole Sequence |
| --- | --- | --- | --- |
| `load_blocking<T>()` | Read with ordering | `lw; and %raw,%raw,%raw` | `lw; and x0,x0,%raw` |
| `store_blocking<T>()` | Write with ordering | `sw; lw; andi %raw,0` | `sw; lw; and x0,x0,%raw` |

The key difference is that Blackhole has a data cache (L0) that requires explicit invalidation semantics. The store flushes the cache line, and the subsequent load ensures the store completed to L1 memory.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 86-162](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L86-L162)[tt_llk_blackhole/common/inc/ckernel.h 95-168](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L95-L168)

### Synchronization Barriers

These functions provide synchronization points by writing to special PC buffer locations that force pipeline drains.

## Configuration State Management

The HAL implements a **ping-pong configuration context system** that allows double-buffering of hardware configuration registers. This enables preparation of the next operation's configuration while the current operation executes.

### Configuration State Architecture

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 72-334](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L72-L334)[tt_llk_blackhole/common/inc/ckernel.h 81-339](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L81-L339)

### Core State Management Functions

| Function | Purpose | Implementation |
| --- | --- | --- |
| `cfg_addr(cfg_addr32)` | Compute address for current state | `(cfg_state_id == 0) ? cfg_addr32 : (CFG_STATE_SIZE*4) + cfg_addr32` |
| `cfg_write(addr32, data)` | Write to config register | Uses `cfg_addr()` to select state |
| `cfg_read(addr32)` | Read from config register | Uses `cfg_addr()` to select state |
| `get_cfg_pointer()` | Get base pointer to current state | Returns `TENSIX_CFG_BASE` or `TENSIX_CFG_BASE + CFG_STATE_SIZE*16` |
| `flip_cfg_state_id()` | Toggle to alternate state | `cfg_state_id = 1 - cfg_state_id; TT_SETC16(...)` |
| `reset_cfg_state_id()` | Reset to state 0 | `cfg_state_id = 0` |

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 283-334](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L283-L334)[tt_llk_blackhole/common/inc/ckernel.h 288-339](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L288-L339)

## Architecture-Specific Extensions

### Blackhole-Specific Features

Blackhole introduces several advanced features not present in Wormhole B0:

#### Data Cache Management

Blackhole has an L0 data cache that requires explicit invalidation. The implementation uses a FENCE instruction to flush the cache.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 809-816](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L809-L816)

#### TRISC Synchronization Tracking

Blackhole provides hardware tracking of dependencies between TRISC memory accesses and Tensix instructions:

This eliminates many manual synchronization calls (`tensix_sync()`, `sync_regfile_write()`) by having hardware automatically stall when necessary.

**Sources:**[tt_llk_blackhole/common/inc/ckernel.h 553-581](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L553-L581)

#### Instruction Gathering Control

Controls whether the instruction fetch unit performs instruction gathering (coalescing multiple small instructions into wider fetch units). This is used when programming replay buffers.

## Summary

The Hardware Abstraction Layer in `ckernel.h` provides the foundational API for all LLK operations:

*   **Memory access primitives** ensure correct ordering on RISC-V cores without hardware coherency
*   **Configuration state management** implements double-buffered contexts for overlapping computation and configuration
*   **Destination register management** enables ping-pong buffering of results
*   **Register access functions** provide state-aware configuration and status reads
*   **Initialization functions** prepare hardware for operation
*   **Architecture-specific extensions** in Blackhole add cache management, dependency tracking, and advanced features

This layer is consumed by higher-level APIs for unpacking ([#4](https://deepwiki.com/tenstorrent/tt-llk/4-unpacker-system)), math ([#5](https://deepwiki.com/tenstorrent/tt-llk/5-math-unit-operations)), and packing ([#6](https://deepwiki.com/tenstorrent/tt-llk/6-packer-system)), which build operation-specific abstractions on top of these primitives.
