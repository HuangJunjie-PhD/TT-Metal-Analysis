# HugePages Configuration

Relevant source files
*   [README.md](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/README.md?plain=1)
*   [install.sh](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh)

## Purpose and Scope

This document describes the HugePages configuration process in the tt-installer system. HugePages are a critical component for optimizing memory access to Tenstorrent hardware accelerators. This page covers how HugePages are configured during installation, why they're necessary, and how to verify or troubleshoot the configuration.

For information about related components of the Tenstorrent software stack, see [Kernel-Mode Driver (KMD)](https://deepwiki.com/tenstorrent/tt-installer/5.1-kernel-mode-driver-(kmd)) and [tt-metalium Container](https://deepwiki.com/tenstorrent/tt-installer/5.5-tt-metalium-container).

## What are HugePages and Why They Matter

HugePages are a memory management feature that allows the operating system to establish memory pages larger than the default 4KB size. The tt-installer specifically configures 1GB HugePages for Tenstorrent hardware.

Sources: [install.sh 576-603](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L576-L603)[README.md 28](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/README.md?plain=1#L28-L28)

Benefits of HugePages for Tenstorrent hardware:

1.   **Reduced TLB Misses**: The Translation Lookaside Buffer (TLB) can cache more memory with larger page sizes
2.   **Lower Memory Management Overhead**: Fewer pages to track means less kernel overhead
3.   **Faster Device Memory Access**: Direct hardware access to large contiguous memory regions
4.   **Required for tt-metalium**: The Metalium container requires HugePages to be configured

## HugePages Configuration Process

The tt-installer sets up HugePages through the tenstorrent-tools package, which provides the necessary systemd services to configure HugePages on boot.

Sources: [install.sh 576-603](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L576-L603)

### Implementation Details

During installation, the script performs the following actions:

1.   Downloads the tenstorrent-tools package specific to your distribution:

    *   DEB package (Ubuntu/Debian): `tenstorrent-tools_{version}-1_all.deb`
    *   RPM package (Fedora/RHEL/CentOS): `tenstorrent-tools-{version}-1.noarch.rpm`

2.   Installs the package using the appropriate package manager (`dpkg` for Debian-based systems, `dnf` for RPM-based systems)

3.   Enables two systemd services:

    *   `tenstorrent-hugepages.service`: Configures the kernel to allocate HugePages
    *   `dev-hugepages\x2d1G.mount`: Mounts the HugePages filesystem at `/dev/hugepages-1G`

### Integration with tt-metalium

The tt-metalium container relies on HugePages for optimal performance. When running the container, the HugePages filesystem is mounted inside the container:

Sources: [install.sh 326-327](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L326-L327)

The Podman run command in the tt-metalium script mounts the HugePages directory with: `--volume=/dev/hugepages-1G:/dev/hugepages-1G`

## Customization and Control

### Skipping HugePages Configuration

You can skip the HugePages configuration during installation by setting the `TT_SKIP_INSTALL_HUGEPAGES` environment variable to `0`:

`TT_SKIP_INSTALL_HUGEPAGES=0 ./install.sh`
This is useful in scenarios where:

*   You've already configured HugePages manually
*   You're running in a container where HugePages should be configured on the host
*   You're installing in a testing environment where HugePages aren't needed

Sources: [install.sh 58](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L58-L58)[install.sh 576-603](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L576-L603)

### Automatic Configuration in Container Mode

When the installer is run in container mode (`TT_MODE_CONTAINER=0`), it automatically skips the HugePages configuration since HugePages must be set up on the host system, not inside a container.

Sources: [install.sh 98-100](https://github.com/tenstorrent/tt-installer/blob/19fdd6a2/install.sh#L98-L100)

## Verifying HugePages Configuration

After installation and system reboot, you can verify that HugePages are correctly configured using the following methods:

1.   **Check the mount point exists**:

`ls -la /dev/hugepages-1G`
2.   **Verify the systemd services are running**:

`systemctl status tenstorrent-hugepages.servicesystemctl status 'dev-hugepages\x2d1G.mount'`
3.   **Check kernel HugePages allocation**:

`cat /proc/meminfo | grep Huge`

## Troubleshooting

### Common Issues

1.   **Insufficient Memory**: If your system doesn't have enough free memory, HugePages allocation may fail.

2.   **Missing Mount Point**: If `/dev/hugepages-1G` doesn't exist after reboot, the mount service may have failed.

3.   **Container Access Issues**: If tt-metalium cannot access HugePages, check that the mount is properly shared with the container.

### Resolution Steps

1.   **Ensure sufficient free memory** before HugePages allocation.

2.   **Check systemd services**:

`journalctl -u tenstorrent-hugepages.servicejournalctl -u 'dev-hugepages\x2d1G.mount'`
3.   **Manually restart services**:

`sudo systemctl restart tenstorrent-hugepages.servicesudo systemctl restart 'dev-hugepages\x2d1G.mount'`
4.   **Reboot the system** to apply HugePages configuration changes.

## Technical Implementation Details

The HugePages configuration is implemented through systemd services that:

1.   Modify kernel parameters to allocate HugePages at boot time
2.   Mount the HugePages filesystem at `/dev/hugepages-1G`
3.   Set appropriate permissions for the mounted filesystem

This persistent configuration ensures that HugePages are available across system reboots.
