## Purpose and Scope

This page describes the TTNN layout and memory optimization system, which determines optimal tensor layouts and memory configurations for operations executing on Tenstorrent hardware. The system analyzes operation constraints, selects appropriate tensor layouts (row-major vs tile), buffer types (DRAM vs L1), memory layouts (interleaved vs sharded), and inserts memory reconfiguration operations as needed.

The core of this system is the `TTNNOptimizer` pass and its associated analysis policies, which leverage the `OpModel` and a greedy beam-search strategy to navigate the trade-offs between performance and memory capacity.

## Layout System Overview

### Layout Representation

The TTNN dialect represents tensor memory configuration through the `TTNNLayoutAttr` attribute, which encapsulates four key properties:

| Property | Description | Values |
| --- | --- | --- |
| **Layout** | Data layout in memory | `Layout::RowMajor`, `Layout::Tile` |
| **BufferType** | Memory space allocation | `BufferType::SystemMemory`, `BufferType::DRAM`, `BufferType::L1` |
| **TensorMemoryLayout** | Distribution across cores | `TensorMemoryLayout::Interleaved`, `HeightSharded`, `WidthSharded`, `BlockSharded` |
| **DataType** | Element data type | `DataType::Float32`, `BFloat16`, `BFP_BFloat8`, etc. |

Sources: [lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 114-121](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L114-L121)[include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h 47-53](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h#L47-L53)[lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp 69-81](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp#L69-L81)

### Layout Types

*   **Row-Major Layout**: Sequential memory layout where elements are stored row-by-row. Often required for host-device transfers or specific ops like pooling.
*   **Tile Layout**: Data organized in 32x32 tiles optimized for hardware compute kernels. Most compute-intensive operations (matmul, convolution) require tile layout for performance.

Sources: [lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 155-158](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L155-L158)[include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h 51-52](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h#L51-L52)

## Optimizer Pass System

The optimization pipeline is a sequence of transformations designed to find a valid and high-performance execution plan.

### TTNNOptimizer Pass

The `TTNNOptimizer` is the primary pass for making layout and sharding decisions. It uses an analysis-driven approach to assign `TTNNLayoutAttr` to tensors.

| Option | Type | Description |
| --- | --- | --- |
| `memoryLayoutAnalysisEnabled` | bool | Enables the search for optimal sharding configurations. |
| `memoryLayoutAnalysisPolicy` | PolicyType | Selects the algorithm (e.g., `DFSharding`). |
| `memReconfigEnabled` | bool | Allows the optimizer to insert `ttnn.to_layout` for resharding. |
| `maxLegalLayouts` | int64_t | Caps the search space for configurations (default 64). |

Sources: [include/ttmlir/Dialect/TTNN/Transforms/Optimizer.h 23-38](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Transforms/Optimizer.h#L23-L38)[include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h 25-31](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Utils/PassOverrides.h#L25-L31)

### Greedy Memory Layout Propagation

The `MemoryLayoutPropagation` analysis implements a beam-search based optimization strategy. It explores multiple candidate configurations (`BeamCandidate`) for each operation to find a path through the graph that minimizes DRAM access and NOC traffic.

*   **BeamCandidate**: Stores a configuration hint, input layouts, and a score. It also tracks `producerCandidateIndices` to reconstruct the optimal path [lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 168-179](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L168-L179)
*   **Scoring**: Candidates are scored based on factors like L1 residency, sharding, DRAM bytes transferred, and core utilization [lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 171-173](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L171-L173)
*   **Rule Books**: Per-operation logic is encapsulated in `OpRuleBook` subclasses (e.g., `MatmulRuleBook`, `Conv2dRuleBook`), which define input filters and output hints [lib/Dialect/TTNN/Analysis/OpRules/OpRuleBook.cpp 65-126](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/OpRules/OpRuleBook.cpp#L65-L126)

Sources: [lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 35-52](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L35-L52)[lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp 160-185](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/MemoryLayoutPropagation.cpp#L160-L185)[lib/Dialect/TTNN/Analysis/OpRules/OpRuleBook.cpp 65-126](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/OpRules/OpRuleBook.cpp#L65-L126)

### Data-Flow Sharding (DFSharding)

The `DFShardingPolicy` processes operations in a schedulable order and builds "L1 Chains"—sequences of operations that keep data in L1 memory to avoid DRAM spills. It uses a `ShardSolver` to reconcile constraints across these chains.

Sources: [lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp 40-49](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp#L40-L49)[lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp 141-160](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/DFShardingPolicy.cpp#L141-L160)

### L1 Chain Configuration

An `L1ChainConfig` tracks a group of operations intended to execute entirely within L1. It manages the state of the chain via `L1ChainState` (`InBuild`, `Built`, `Resolved`, `Completed`, `Failed`) and handles "spill" locations if the chain output must go to DRAM.

Sources: [include/ttmlir/Dialect/TTNN/Analysis/L1ChainConfig.h 44-50](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Analysis/L1ChainConfig.h#L44-L50)[include/ttmlir/Dialect/TTNN/Analysis/L1ChainConfig.h 120-133](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/include/ttmlir/Dialect/TTNN/Analysis/L1ChainConfig.h#L120-L133)

### L1 Spill Management

To handle memory pressure, the `L1SpillManagement` system includes a `SumL1MemoryTracker` that simulates the L1 memory address space. ## Layout Analysis and Selection

### ShardSolver and Bitset Propagation

The `ShardSolver` reconciles adjacent operation constraints using bitset-based constraint propagation. Each bit in the bitset represents a "legal config" (an `OpConfig`) for an operation.

*   **PathSet**: Represents valid pairs of (producer_config, consumer_config) across an edge.
*   **Constraint Propagation**: If a producer config has no valid consumer path, its bit is cleared. This propagates through the graph until a fixed point is reached or the set is empty [lib/Dialect/TTNN/Analysis/ShardSolver.cpp 106-120](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/lib/Dialect/TTNN/Analysis/ShardSolver.cpp#L106-L120)
