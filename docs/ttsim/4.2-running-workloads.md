# Running Workloads

Relevant source files
*   [README.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1)

## Purpose and Scope

This page explains how to execute TT-Metalium workloads on the ttsim simulator, including command-line usage patterns, supported workload types, and expected runtime behavior. This assumes you have already completed the configuration steps described in [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration). For information about simulation accuracy and numerical results, see [Numerical Accuracy](https://deepwiki.com/tenstorrent/ttsim/4.3-numerical-accuracy). For troubleshooting error messages, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference).

**Sources:**[README.md 1-121](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L1-L121)

## Prerequisites

Before running workloads, verify the following configuration is in place:

| Requirement | Description | Reference |
| --- | --- | --- |
| `TT_METAL_HOME` | Points to your TT-Metalium installation directory | [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| `TT_METAL_SIMULATOR` | Points to the simulator shared library (e.g., `~/sim/libttsim_wh.so`) | [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| `soc_descriptor.yaml` | SOC descriptor file co-located with the simulator library | [Configuration](https://deepwiki.com/tenstorrent/ttsim/4.1-configuration) |
| TT-Metalium built | Workload binaries must be built in your `$TT_METAL_HOME` | [Getting Started](https://deepwiki.com/tenstorrent/ttsim/2-getting-started) |

**Sources:**[README.md 23-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L23-L61)

## Basic Execution Pattern

### Command Structure

All ttsim workloads follow this execution pattern:

`TT_METAL_SLOW_DISPATCH_MODE=1 <workload_binary> [workload_arguments]`
The `TT_METAL_SLOW_DISPATCH_MODE=1` environment variable is **mandatory** for all simulator runs, as fast dispatch mode is not currently supported.

**Sources:**[README.md 51-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L51-L61)[README.md 64](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L64-L64)

## Supported Workload Types

ttsim supports multiple categories of TT-Metalium workloads:

### Metal Programming Examples

Metal programming examples demonstrate fundamental programming patterns using the Metal API. These are located in `$TT_METAL_HOME/build/programming_examples/`.

**Example: Add Two Integers in RISC-V**

`# Wormholeexport TT_METAL_SIMULATOR=~/sim/libttsim_wh.socd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 ./build/programming_examples/metal_example_add_2_integers_in_riscv``# Blackholeexport TT_METAL_SIMULATOR=~/sim/libttsim_bh.socd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 ./build/programming_examples/metal_example_add_2_integers_in_riscv`
**Sources:**[README.md 46-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L46-L61)

### ttnn Examples and Tests

The ttnn (Tenstorrent Neural Network) API provides higher-level neural network operations. Many ttnn examples and tests can run on ttsim in slow dispatch mode.

`cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 pytest tests/ttnn/<test_path>`
**Sources:**[README.md 18-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L18-L19)

### LLK Tests

Low-Level Kernel (LLK) tests validate individual compute kernel operations. Numerous LLK tests are supported by ttsim.

`cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 ./build/test/tt_metal/<llk_test_binary>`
**Sources:**[README.md 18-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L18-L19)

## Expected Behavior

### Normal Execution Output

Successful workload execution typically produces:

*   Progress messages from TT-Metalium
*   Workload-specific output (test results, computation results, etc.)
*   Clean exit with return code 0

Example successful output:

```
Device 0 opened
Running add_2_integers example...
Result: 42
Device 0 closed
```

### Performance Characteristics

ttsim is slower than running on silicon but maintains sufficient performance for productive development:

| Characteristic | Behavior |
| --- | --- |
| Speed | Slower than silicon, but "fast enough that you can run interesting workloads with good productivity" |
| Determinism | Results are deterministic for properly synchronized code |
| Scalability | Performance scales with workload complexity |

**Sources:**[README.md 3-6](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L3-L6)

### Dispatch Mode Limitation

Fast dispatch mode is not currently supported. Attempting to run without `TT_METAL_SLOW_DISPATCH_MODE=1` will result in an error:

`# This will fail:./build/programming_examples/metal_example_add_2_integers_in_riscv # This will succeed:TT_METAL_SLOW_DISPATCH_MODE=1 ./build/programming_examples/metal_example_add_2_integers_in_riscv`
**Sources:**[README.md 64](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L64-L64)

## Architecture-Specific Considerations

### Wormhole vs Blackhole

Different simulator binaries support different chip architectures:

**Architecture-Specific Configuration Diagram**

| Architecture | Simulator Binary | SOC Descriptor | Maturity |
| --- | --- | --- | --- |
| Wormhole | `libttsim_wh.so` | `wormhole_b0_80_arch.yaml` | More mature |
| Blackhole | `libttsim_bh.so` | `blackhole_140_arch.yaml` | Less mature |

The same workload binary can often run on either architecture by switching the `TT_METAL_SIMULATOR` environment variable, assuming the workload is compatible with the target architecture.

**Sources:**[README.md 18-19](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L18-L19)[README.md 36-37](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L36-L37)[README.md 46-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L46-L61)

## Common Execution Patterns

### Running Single Test

`cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 ./build/test/tt_metal/test_specific_binary`
### Running Python-Based Tests

`cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 pytest tests/ttnn/unit_tests/test_specific.py`
### Running Test Suites

`cd $TT_METAL_HOMETT_METAL_SLOW_DISPATCH_MODE=1 pytest tests/ttnn/unit_tests/ -v`
### Switching Architectures

`# Run on Wormholeexport TT_METAL_SIMULATOR=~/sim/libttsim_wh.soTT_METAL_SLOW_DISPATCH_MODE=1 <workload> # Run on Blackholeexport TT_METAL_SIMULATOR=~/sim/libttsim_bh.soTT_METAL_SLOW_DISPATCH_MODE=1 <workload>`
**Sources:**[README.md 46-61](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L46-L61)

## Error Handling During Execution

ttsim may report various error categories during workload execution. Common categories include:

| Error Category | Description | Action |
| --- | --- | --- |
| `UndefinedBehavior` | ISA specification violation | Fix workload code |
| `UnimplementedFunctionality` | Feature not yet supported | Check feature status or file issue |
| `ConfigurationError` | Environment/setup problem | Review configuration |
| `AssertionFailure` | Internal simulator bug | Report issue |

For detailed error information and remediation steps, see [Error Reference](https://deepwiki.com/tenstorrent/ttsim/4.4-error-reference).

**Sources:**[README.md 63-76](https://github.com/tenstorrent/ttsim/blob/67b1733e/README.md?plain=1#L63-L76)
