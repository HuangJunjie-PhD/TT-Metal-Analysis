## Purpose and Scope

The Failure Classification System provides automated categorization of test failures across the tt-forge-onnx compiler pipeline. It analyzes exception traces to determine which component (Metal, TTNN, MLIR, Forge, TVM) caused the failure and classifies the failure type (data mismatch, unsupported format, verification failure, etc.). This enables efficient debugging, regression tracking, and expected failure management (xfail) in the test infrastructure.

For information about the broader testing infrastructure, see [Testing Infrastructure](https://deepwiki.com/tenstorrent/tt-forge-onnx/6-testing-infrastructure). For operator testing patterns, see [Operator Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.2-operator-testing). For model testing workflows, see [Model Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.3-model-testing).

## System Architecture

The Failure Classification System consists of three layers: exception capture, pattern matching, and classification. Exceptions flow through pattern matchers that check component signatures and error messages, ultimately producing a categorized `FailingReason` that can be used for test management.

**Sources:**[forge/test/operators/utils/failing_reasons.py 1-320](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L1-L320)

## Core Data Structures

### ExceptionData

The `ExceptionData` dataclass captures structured information about an exception for pattern matching:

`@dataclassclass ExceptionData:    class_name: str      # e.g., "RuntimeError", "ValueError"    message: str         # Exception message text    error_log: str       # Full traceback with file paths and line numbers`
The `FailingReasonsFinder.build_ex_data()` method constructs `ExceptionData` from Python exceptions, normalizing the class name (removing `builtins.` prefix) and stripping ANSI color codes from tracebacks.

**Sources:**[forge/test/operators/utils/failing_reasons.py 18-24](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L18-L24)[forge/test/operators/utils/failing_reasons.py 268-288](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L268-L288)

* * *

### ExceptionCheck Pattern Matcher

The `ExceptionCheck` class defines a multi-criteria pattern matcher for exceptions:

| Field | Type | Purpose |
| --- | --- | --- |
| `class_name` | `Optional[str]` | Matches exact exception class name (e.g., "RuntimeError") |
| `component` | `Optional[ComponentChecker]` | Matches component via binary/file path signatures |
| `message` | `List[MessageCheckerType]` | List of message pattern matchers (all must match) |
| `error_log` | `List[MessageCheckerType]` | List of traceback pattern matchers (all must match) |

The `check()` method implements AND logic across all criteria: all specified matchers must pass for the check to succeed.

**Sources:**[forge/test/operators/utils/failing_reasons.py 29-53](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L29-L53)

* * *

### FailingReason Container

The `FailingReason` dataclass groups related `ExceptionCheck` patterns under a descriptive label:

`@dataclassclass FailingReason:    description: str                    # Human-readable failure description    checks: List[ExceptionCheck]        # Multiple patterns (OR logic)`
Multiple checks implement OR logic: any check matching causes the `FailingReason` to match. The `component_checker` property extracts the first non-NONE component from its checks, enabling component-based failure attribution.

**Sources:**[forge/test/operators/utils/failing_reasons.py 56-94](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L56-L94)

## Message Pattern Matching System

The `MessageChecker` class provides static factory methods for creating message matching functions:

### Common Pattern Examples

| Method | Usage Example | Purpose |
| --- | --- | --- |
| `M.contains("lib/libtt_metal.so")` | Detect Metal library in stack | Component identification |
| `M.starts_with("Data mismatch")` | Match error message prefix | Failure type detection |
| `M.regex(r"Tensor .* - data type mismatch")` | Flexible pattern matching | Complex error formats |
| `M.last_line(M.contains("forge/verify/"))` | Check final stack frame | Pinpoint error location |
| `M.neg(M.contains("lib/libtt_metal.so"))` | Exclude Metal | Negative filtering |
| `M.any(checker1, checker2)` | Multiple alternatives | OR logic |

The `M` alias (`M = MessageChecker`) provides concise syntax in failing reason definitions.

**Sources:**[forge/test/operators/utils/failing_reasons.py 97-129](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L97-L129)

## Component Classification System

The `ComponentChecker` enum identifies the compiler stack layer responsible for failures by analyzing binary/library names and file paths in exception tracebacks:

### Component Detection Logic

Each `ComponentChecker` value is a `FailingReason` with `ExceptionCheck` patterns that examine the `last_line()` of tracebacks:

*   **METAL**: Deepest in stack, includes `libtt_metal.so`, represents hardware/device layer
*   **TTNN**: Excludes Metal, includes `_ttnncpp.so`, represents TTNN operations layer
*   **MLIR**: Excludes Metal/TTNN, includes `libTTMLIRRuntime.so`, represents MLIR compilation
*   **FORGE**: Python/C++ code, matches `forge/verify/`, `forge/compile.py`, excludes lower layers
*   **TVM**: TVM frontend code, matches `tvm/relay/`, `forge/tvm_to_python.py`
*   **NONE**: Helper component for obsolete checks

The cascading negative checks (`M.neg(M.contains("lib/libtt_metal.so"))`) ensure single-component attribution by excluding lower layers.

**Sources:**[forge/test/operators/utils/failing_reasons.py 132-264](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/utils/failing_reasons.py#L132-L264)

## Failing Reasons Taxonomy

The `FailingReasons` enum defines 60+ failure categories organized by failure type. Each enum value is a `FailingReason` instance with descriptive text and pattern matching rules.

### Major Failure Categories

### Example Pattern Definitions

**DATA_MISMATCH_ALL_CLOSE** - Numerical accuracy failure:

`ExceptionCheck(    class_name="ValueError",    component=ComponentChecker.FORGE.value,    message=[        M.starts_with("Data mismatch -> AllCloseValueChecker (all_close)"),    ],    error_log=[        M.last_line(M.contains("forge/verify/value_checkers.py:")),    ],)`

## Integration with Test Framework

The Failure Classification System integrates with the test infrastructure through `TestPlan` and `TestCollection` objects:

### Failing Rules in Test Plans

Test plans define `failing_rules` as lists of `TestCollection` objects with `failing_reason` and optional `skip_reason` attributes:

`TestParamsData.test_plan = TestPlan(    verify=lambda test_device, test_vector: TestVerification.verify(...),    collections=[        TestCollection(            operators=["add", "mul"],            input_sources=TestCollectionCommon.all.input_sources,            input_shapes=TestCollectionCommon.all.input_shapes,        ),    ],    failing_rules=[        TestCollection(            operators=["add"],            dev_data_formats=[torch.float16],            failing_reason=FailingReasons.UNSUPPORTED_DATA_FORMAT,            skip_reason=FailingReasons.UNSUPPORTED_DATA_FORMAT,        ),        TestCollection(            operators=["mul"],            input_shapes=[(10, 10, 10000, 1)],            failing_reason=FailingReasons.OUT_OF_MEMORY,            skip_reason=FailingReasons.OUT_OF_MEMORY,        ),    ],)`
The test framework matches test vectors against failing rules to determine if failures are expected (`failing_reason`) or should be skipped (`skip_reason`).

**Sources:**[forge/test/operators/pytorch/eltwise_binary/test_binary.py 440-537](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_binary/test_binary.py#L440-L537)[forge/test/operators/pytorch/eltwise_unary/test_unary.py 284-343](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/eltwise_unary/test_unary.py#L284-L343)

## Expected Failure Management

### xfail vs skip Distinction

| Attribute | Behavior | Use Case |
| --- | --- | --- |
| `failing_reason` | Test runs, failure expected, marked xfail if matches | Known issues, tracked regressions |
| `skip_reason` | Test skipped entirely, doesn't execute | Crashes, hangs, fatal errors |

### Operator-Specific Failure Registers

The `FailingReasonsRegister` class maintains centralized lists of known failures per operator:

`class FailingReasonsRegister:    xfail = [        ("add", FailingReasons.DATA_MISMATCH_ALL_CLOSE),        ("add", FailingReasons.DATA_MISMATCH_PCC_LOW_RANGE),        ("conv2d", FailingReasons.CIRCULAR_BUFFER_EXCEEDS_L1),        ("matmul", FailingReasons.UNSUPPORTED_DATA_FORMAT),        # ... 100+ entries    ]        skip = [        ("matmul", FailingReasons.HIGH_MEMORY, None),        ("conv2d", FailingReasons.FATAL_ERROR, None),        ("pow", FailingReasons.FATAL_ERROR, None),        # ... entries with fatal issues    ]`
The `TestIdsDataLoader.build_failing_rules()` method converts these lists into `TestCollection` objects for inclusion in test plans:

`failing_rules = [    *TestIdsDataLoader.build_failing_rules(operators=["add", "mul", "div"]),    # Additional manual rules...]`
**Sources:**[forge/test/operators/pytorch/ids/failing_reasons_register.py 1-134](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/operators/pytorch/ids/failing_reasons_register.py#L1-L134)

## Usage Patterns in Tests

### Pattern 1: Operator Test with Failing Rules

Unary operator tests demonstrate comprehensive failure classification:

`TestParamsData.test_plan = TestPlan(    verify=lambda test_device, test_vector: TestVerification.verify(...),    collections=[        TestCollection(            operators=["relu", "sqrt", "exp", ...],            input_sources=TestCollectionCommon.all.input_sources,            input_shapes=TestCollectionCommon.all.input_shapes,        ),    ],    failing_rules=[        # Loaded from centralized register        *TestIdsDataLoader.build_failing_rules(            operators=["relu", "sqrt", "exp", ...]        ),        # Manual overrides        TestCollection(            dev_data_formats=[torch.float16],            failing_reason=FailingReasons.UNSUPPORTED_DATA_FORMAT,            skip_reason=FailingReasons.UNSUPPORTED_DATA_FORMAT,        ),    ],)`

## Component Attribution Statistics

The classification system enables per-component failure analysis through the `component_checker` property of `FailingReason` objects:

| Component | Typical Failure Types | Example File Paths |
| --- | --- | --- |
| **METAL** | Hardware errors, device crashes, TT_FATAL assertions | `lib/libtt_metal.so`, `lib/_ttnncpp.so` |
| **TTNN** | Operator validation, unsupported data formats, memory issues | `ttnn/cpp/ttnn/operations/` |
| **MLIR** | Lowering failures, IR verification, type mismatches | `lib/libTTMLIRRuntime.so` |
| **FORGE** | Data mismatches, PCC failures, verification errors | `forge/verify/`, `forge/compile.py` |
| **TVM** | Relay conversion, operator mapping, shape inference | `tvm/relay/`, `forge/tvm_to_python.py` |

This attribution appears in test reports and dashboards for prioritizing fixes by team/component.

## Validation and Maintenance

### Finding Unclassified Failures

The `UNCLASSIFIED` failing reason catches exceptions that don't match any pattern:

`UNCLASSIFIED = FailingReason(    description="Unclassified error",)  # No checks - catches all remaining failures`
Monitoring unclassified failures identifies gaps in the taxonomy requiring new pattern definitions.

### Obsoleting Old Patterns

The `ComponentChecker.NONE` value marks patterns that are no longer used:

`NONE = FailingReason(    description="None",    checks=[        ExceptionCheck(            error_log=[                M.contains("A non existing line in the error log"),            ],        ),    ],)`
Patterns with `component=ComponentChecker.NONE.value` are filtered out during `FailingReason` initialization.
