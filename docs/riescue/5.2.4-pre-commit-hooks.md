This page describes the pre-commit hook system used in RiESCUE to automate code quality checks before commits are finalized. Pre-commit hooks ensure that code formatting and linting standards are enforced locally, reducing CI/CD failures and maintaining code consistency.

For information about the individual linting tools (black, flake8, pyright), see [Black Code Formatter](https://deepwiki.com/tenstorrent/riescue/5.2.1-black-code-formatter), [Flake8 Linter](https://deepwiki.com/tenstorrent/riescue/5.2.2-flake8-linter), and [Pyright Type Checker](https://deepwiki.com/tenstorrent/riescue/5.2.3-pyright-type-checker). For information about the CI/CD pipeline that runs these checks on GitHub, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/riescue/5.3-cicd-pipeline).

## Overview

Pre-commit hooks are scripts that automatically run before a git commit is finalized. RiESCUE uses the `pre-commit` framework to manage these hooks, primarily to run `black` code formatting automatically on Python files before they are committed.

The pre-commit system provides:

*   **Automatic code formatting** before commits
*   **Early detection** of style issues
*   **Reduced CI failures** by catching problems locally
*   **Consistent code style** across contributors

## Configuration

### Pre-commit Configuration File

The repository includes a `.pre-commit-config.yaml` file that defines which hooks run and how they execute. This file specifies:

*   Hook repositories and versions
*   Which files to process
*   Command-line arguments for each tool
*   Execution order

Sources: [docs/lint.md 74-75](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L74-L75)[docs/lint.md 85](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L85-L85)

### Black Integration

By default, the pre-commit framework pulls black from its GitHub repository to ensure a reproducible environment. This means:

*   The black version is pinned in `.pre-commit-config.yaml`
*   Each developer gets the same black version regardless of their local installation
*   The environment is isolated from system-installed packages

However, if you prefer to use your locally installed black (via pip), you can configure a local hook. Note that this approach relies on your local environment and won't offer the same isolation or reproducibility as the default configuration.

Sources: [docs/lint.md 72-85](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L72-L85)

## Relationship to CI/CD

Pre-commit hooks provide a **local** enforcement layer that complements the **CI/CD** enforcement layer:

| Aspect | Pre-commit Hooks | CI/CD Pipeline |
| --- | --- | --- |
| **When** | Before local commit | After push to GitHub |
| **Scope** | Only changed files | All files in repository |
| **Tools** | black (formatting only) | black, flake8, pyright |
| **Enforcement** | Can be bypassed with `--no-verify` | Cannot be bypassed |
| **Purpose** | Developer convenience | Mandatory quality gate |
| **Speed** | Fast (local execution) | Slower (remote execution) |

The CI/CD pipeline runs the same checks in a containerized environment to ensure that all code meets quality standards, even if a developer bypasses local hooks.

Sources: [docs/lint.md 7-10](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L7-L10)[.github/CONTRIBUTING.md 36-63](https://github.com/tenstorrent/riescue/blob/07cefde3/.github/CONTRIBUTING.md?plain=1#L36-L63)

## Bypassing Hooks (Not Recommended)

In rare cases, you can bypass pre-commit hooks:

**Warning**: Bypassing hooks is **not recommended** because:

*   Your code will likely fail CI/CD checks
*   Other contributors will see inconsistent formatting
*   You'll need to fix issues later anyway

Only bypass hooks if you're making a temporary WIP commit that you plan to amend before pushing.

Sources: [docs/lint.md 66-68](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L66-L68)

## Troubleshooting

### Hook Installation Issues

If hooks aren't running after `pre-commit install`, verify:

### Black Version Conflicts

If you see errors about black versions:

1.   The pre-commit framework uses its own black installation by default
2.   Your local black version (if installed) doesn't affect pre-commit
3.   Check `.pre-commit-config.yaml` for the pinned black version

### Performance Issues

If pre-commit hooks are slow:

*   They only run on staged files (not all files)
*   Use `pre-commit run --files <path>` to test specific files
*   Consider the trade-off between local convenience and CI/CD speed

Sources: [docs/lint.md 72-85](https://github.com/tenstorrent/riescue/blob/07cefde3/docs/lint.md?plain=1#L72-L85)

## Contributing Standards

Contributors should:

1.   Install pre-commit hooks before starting development
2.   Let hooks run automatically on each commit
3.   Review and stage any auto-formatted changes
4.   Not bypass hooks except for temporary WIP commits
5.   Ensure code passes all checks before creating pull requests

Pull requests must pass all CI/CD checks, including linting, before they can be merged. Pre-commit hooks help you catch issues early.

## Related Tools

The pre-commit system currently focuses on `black` formatting, but the framework supports additional hooks:

*   **black**: Python code formatter (currently enabled)
*   **flake8**: Style guide enforcement (CI/CD only, not in pre-commit)
*   **pyright**: Static type checker (CI/CD only, not in pre-commit)
*   **Custom hooks**: Can be added to `.pre-commit-config.yaml`

For detailed information about running these tools manually, see:

*   [Black Code Formatter](https://deepwiki.com/tenstorrent/riescue/5.2.1-black-code-formatter)
