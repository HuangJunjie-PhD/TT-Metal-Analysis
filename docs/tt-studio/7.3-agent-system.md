The TT-Studio Agent System is a specialized service designed to provide advanced LLM-driven capabilities, including multi-model support, autonomous search integration, and robust health monitoring for inference containers. It acts as an orchestration layer between the user interface and the various LLM backends deployed on Tenstorrent hardware.

## Overview and Purpose

The agent service, located in `app/agent/`, is responsible for:

*   **Auto-discovery**: Identifying and connecting to LLM containers within the `tt_studio_network`.
*   **Resiliency**: Implementing circuit breakers and polling mechanisms to handle slow-starting or failing inference models.
*   **Tool Integration**: Providing the LLM with external tools, such as the Tavily-powered search agent.
*   **Multi-LLM Support**: Facilitating switching between local Tenstorrent-hosted models and cloud-based LLM fallbacks.

## Implementation Details

### LLM Polling and Circuit Breaker

The agent system includes a polling mechanism to ensure the target LLM container is ready to receive requests. This is critical because Tenstorrent models may take significant time to load weights into SRAM/DRAM.

*   **Polling**: Controlled by `AGENT_LLM_POLLING_TIMEOUT` and `AGENT_LLM_POLLING_MAX_ATTEMPTS`[app/docker-compose.yml 108-109](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L108-L109)
*   **Circuit Breaker**: If a model fails to respond within the `AGENT_LLM_POLLING_CIRCUIT_BREAKER_THRESHOLD`, the agent stops attempting to reach the container to prevent resource exhaustion and provides a failure state to the backend [app/docker-compose.yml 110](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L110-L110)

### Thinking Blocks and Reasoning

For models that support reasoning (like DeepSeek or R1 variants), the system processes `<think>` tags. The frontend component `StreamingMessage.tsx` implements a parser to extract these blocks for specialized UI rendering.

*   **Regex Parsing**: The function `processContent` uses `/ <think>(.*?)<\/think> /gis` to extract completed reasoning chains [app/frontend/src/components/chatui/StreamingMessage.tsx 26-30](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/chatui/StreamingMessage.tsx#L26-L30)
*   **Live Streaming**: The system detects incomplete thinking blocks using `/<think>(?!.*<\/think>)/is` to show a "Thinking..." state in the UI while the model is still generating its internal monologue [app/frontend/src/components/chatui/StreamingMessage.tsx 100-102](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/chatui/StreamingMessage.tsx#L100-L102)
*   **Content Cleaning**: The agent and frontend work together to strip metadata tokens like `<|eot_id|>` or `assistant` tags before displaying the final response to the user [app/frontend/src/components/chatui/StreamingMessage.tsx 33-46](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/frontend/src/components/chatui/StreamingMessage.tsx#L33-L46)

## Configuration

The agent's behavior is heavily influenced by environment variables defined in the `.env` file and passed via `docker-compose.yml`.

| Variable | Purpose |
| --- | --- |
| `USE_CLOUD_LLM` | Boolean flag to toggle between local TT hardware and cloud fallback [app/docker-compose.yml 105](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L105-L105) |
| `TAVILY_API_KEY` | Key for the search agent to perform real-time web lookups [app/docker-compose.yml 102](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L102-L102) |
| `LLM_CONTAINER_NAME` | The target container name the agent should communicate with [app/docker-compose.yml 106](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L106-L106) |
| `JWT_SECRET` | Used for authenticated communication between the agent and backend [app/docker-compose.yml 101](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L101-L101) |

### Search Agent Integration

When `TAVILY_API_KEY` is provided [app/.env.default 30](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/.env.default#L30-L30) the agent service can augment LLM prompts with real-time data. This integration allows the agent to:

1.   Intercept queries requiring external knowledge.
2.   Execute searches via the Tavily API.
3.   Inject search results into the LLM context window.

## Health Monitoring

The agent service includes its own health check to ensure it is ready to handle requests from the Django backend. It listens on port `8080` and provides a root endpoint for Docker health checks [app/docker-compose.yml 99-115](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L99-L115)

**Agent Health Check Configuration:**

*   **Test**: `curl -f http://localhost:8080/`[app/docker-compose.yml 115](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L115-L115)
*   **Interval**: 10 seconds [app/docker-compose.yml 117](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L117-L117)
*   **Start Period**: 30 seconds (to allow for service initialization) [app/docker-compose.yml 119](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/docker-compose.yml#L119-L119)
