## Purpose and Scope

This document provides a comprehensive overview of the GitHub Actions-based CI/CD pipeline for tt-tools-common. It covers all automated workflows including test execution, release orchestration, multi-target package building, and distribution publishing. For detailed information about the build system configuration and version management, see [Build and Distribution](https://deepwiki.com/tenstorrent/tt-tools-common/5-build-and-distribution). For testing methodology and pytest configuration, see [Testing Guide](https://deepwiki.com/tenstorrent/tt-tools-common/6.1-testing-guide).

## Test Automation Workflow

The `test.yml` workflow implements continuous integration testing with hardware and software test separation using pytest markers.

### Test Execution Flow

**Sources:**[.github/workflows/test.yml 13-67](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L13-L67)

### Test Job Configuration

The `test` job runs in parallel across three Python versions with `fail-fast: false` to ensure all versions are tested even if one fails:

`strategy:  fail-fast: false  matrix:    python-version: ["3.10", "3.11", "3.12"]`
**Sources:**[.github/workflows/test.yml 17-20](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L17-L20)

The `test-hardware` job runs on self-hosted runners with actual Tenstorrent hardware:

| Runner | Hardware Configuration |
| --- | --- |
| `tt-ubuntu-2204-n150-stable` | N150 card configuration |
| `tt-ubuntu-2204-p150b-stable` | P150B card configuration |

**Sources:**[.github/workflows/test.yml 40-48](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L40-L48)

### Test Marker Strategy

Tests are separated by the `requires_hardware` marker:

*   Software tests: `pytest -m "not requires_hardware"` - Run on standard GitHub runners
*   Hardware tests: `pytest -m requires_hardware` - Run on self-hosted runners with Tenstorrent chips

**Sources:**[.github/workflows/test.yml 36-38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L36-L38)[.github/workflows/test.yml 64-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L64-L66)

## Release Workflow

The `release.yml` workflow orchestrates the complete release process through a series of dependent jobs that modify version files, generate changelogs, build artifacts, and publish to multiple targets.

### Release Job Dependency Chain

**Sources:**[.github/workflows/release.yml 21-453](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L21-L453)

### Temporary Branch Isolation Pattern

The release workflow uses a temporary branch strategy to isolate version modifications before merging back to main. This prevents merge conflicts and ensures atomic version updates:

1.   **Branch Creation**: Job `create-temp-branch` generates a unique branch name: `rc-temp-{SHORT_SHA}-{TIMESTAMP}`
2.   **Version Modifications**: Jobs `versionchange` and `changelogs` commit to this temporary branch
3.   **Build Isolation**: All build workflows use this branch via `inputs.ref`
4.   **Merge Back**: Job `mergeback` rebases changes to main and deletes the temporary branch

**Sources:**[.github/workflows/release.yml 41-62](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L41-L62)[.github/workflows/release.yml 362-386](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L362-L386)

### Version Management Jobs

#### versionchange Job

The `versionchange` job modifies `pyproject.toml` using `bump2version` and extracts version components:

| Tool | Purpose | Configuration |
| --- | --- | --- |
| `toml-cli` | Extract current version from `pyproject.toml` | `toml get project.version` |
| `bump2version` | Increment version based on input | `--current-version` + `patch/minor/major` argument |
| `release-kit/semver@v2` | Parse version into MAJOR.MINOR.PATCH components | Used for downstream jobs |

**Sources:**[.github/workflows/release.yml 64-160](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L64-L160)

The job outputs 8 key values used by downstream jobs:

*   `git_hash`: Commit SHA after version update
*   `package_name`: Extracted from `project.name` in pyproject.toml
*   `package_version_new`: New version after bumping
*   `version_major`, `version_minor`, `version_patch`: Semantic version components
*   `number_of_commits_since_tag`: Commit count since last tag

**Sources:**[.github/workflows/release.yml 74-84](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L74-L84)

#### changelogs Job

Generates Debian changelog using `git-buildpackage` (gbp):

`gbp dch \  --debian-branch ${TEMP_BRANCH_REF} \  -R \  -N ${MAJOR}.${MINOR}.${PATCH} \  --spawn-editor=never`
**Sources:**[.github/workflows/release.yml 203-213](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L203-L213)

### Release Publication Jobs

#### generate-release Job

Creates GitHub Release with automated changelog generation:

1.   **Download Artifacts**: Retrieves all `.deb` and `.whl` files from build jobs
2.   **Generate Changelog**: Uses `mikepenz/release-changelog-builder-action@v5` in HYBRID mode to include both PRs and direct commits
3.   **Rename Artifacts**: Deduplicates `.deb` files by embedding distro name in filename
4.   **Create Release**: Uses `softprops/action-gh-release@v1` to attach all artifacts

**Sources:**[.github/workflows/release.yml 288-357](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L288-L357)

Changelog template configuration:

`{  "template": "#{{CHANGELOG}}\n\n## Contributors\n#{{CONTRIBUTORS}}",  "categories": [    {      "title": "## 🔄 Changes",      "labels": []    }  ],  "pr_template": "- #{{TITLE}} (#{{NUMBER}}) by @#{{AUTHOR}}",  "commit_template": "- #{{TITLE}} (#{{MERGE_SHA}}) by @#{{AUTHOR}}"}`
**Sources:**[.github/workflows/release.yml 305-321](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L305-L321)

## Build Workflows

### build-all.yml Orchestrator

The `build-all.yml` workflow serves as a coordination point that triggers both Debian and Python builds in parallel using `workflow_call`:

**Sources:**[.github/workflows/build-all.yml 24-53](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-all.yml#L24-L53)

*   `NUMBER_OF_COMMITS_SINCE_TAG`: For development builds

The Debian build workflow generates packages for three Ubuntu distributions in parallel.

#### Build Matrix Configuration

`strategy:  fail-fast: false  matrix:    os:      - ubuntu-22.04      - ubuntu-24.04      - ubuntu-latest`
**Sources:**[.github/workflows/build-debian.yml 26-34](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L26-L34)

#### Debian Build Pipeline

**Sources:**[.github/workflows/build-debian.yml 36-178](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L36-L178)

#### GPG Signing

Packages are signed using a GPG key stored in GitHub Secrets:

`- name: Import GPG key  id: gpg_key_import  uses: crazy-max/ghaction-import-gpg@v6  with:    gpg_private_key: ${{ secrets.PKG_SIGNING_KEY_DEB }}`
The fingerprint is passed to `gbp buildpackage` via `DEBSIGN_KEYID` environment variable.

**Sources:**[.github/workflows/build-debian.yml 67-133](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L67-L133)

#### Artifact Naming Strategy

Artifacts are renamed to embed distro identification to prevent filename collisions:

| Original Filename | Renamed Filename |
| --- | --- |
| `tt-tools-common_1.0.0_all.deb` | `tt-tools-common_1.0.0_all-ubuntu-22.04.deb` |
| `tt-tools-common_1.0.0_amd64.changes` | `tt-tools-common_1.0.0_amd64-ubuntu-22.04.changes` |

**Sources:**[.github/workflows/build-debian.yml 143-158](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/build-debian.yml#L143-L158)

## Publishing Workflows

### TestPyPI Staging

The `publish-to-testpypi` job validates packages on the TestPyPI staging server before production publication. This job uses trusted publishing via OIDC tokens, eliminating the need for API keys.

#### TestPyPI Configuration

`environment:  name: testpypi  url: https://test.pypi.org/p/tt-tools-common permissions:  id-token: write  # IMPORTANT: mandatory for trusted publishing`
**Sources:**[.github/workflows/release.yml 391-405](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L391-L405)

The job downloads wheel artifacts and publishes using `pypa/gh-action-pypi-publish@release/v1`:

`- name: Download Python distributions  uses: actions/download-artifact@v4  with:    pattern: tt-tools-common*.whl    name: release-dists    path: dist/    - name: Publish distribution 📦 to TestPyPI  uses: pypa/gh-action-pypi-publish@release/v1  with:    repository-url: https://test.pypi.org/legacy/    verbose: true`
**Sources:**[.github/workflows/release.yml 408-419](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L408-L419)

### Production PyPI Publication

The `publish-to-pypi` job runs only after TestPyPI validation succeeds. It uses identical configuration except for the target URL:

`needs:  - build_all_depends  - generate-release  - publish-to-testpypi  # Ensures TestPyPI succeeded first`
**Sources:**[.github/workflows/release.yml 424-432](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L424-L432)

### Trusted Publishing with OIDC

Both PyPI publishing jobs use GitHub's OIDC provider for authentication instead of long-lived API tokens. This requires:

1.   **GitHub Environment Configuration**: Environments named `testpypi` and `pypi` must be configured in repository settings
2.   **PyPI Trusted Publisher Setup**: PyPI projects must register the GitHub repository as a trusted publisher
3.   **Token Permissions**: Job must have `id-token: write` permission

**Sources:**[.github/workflows/release.yml 404-405](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L404-L405)[.github/workflows/release.yml 438-439](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L438-L439)

## Version Extraction for Packaging

The `setup.py` file provides backward compatibility for Ubuntu 22.04 packaging tools that don't fully support PEP 621 `pyproject.toml`. It dynamically extracts the version from `pyproject.toml` using `tomli`:

`#!/usr/bin/env python3"""Compatibility setup.py for Ubuntu 22.04 packaging tools.Uses setuptools_scm for dynamic versioning from git tags."""from setuptools import setup, find_packages import tomli if __name__ == "__main__":    with open("pyproject.toml", "rb") as f:        toml_data = tomli.load(f)     setup(        name="tt-tools-common",        version=toml_data['project']['version'],        packages=find_packages(),        python_requires=">=3.10",    )`
**Sources:**[setup.py 1-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/setup.py#L1-L21)

This approach ensures:

*   Single source of truth: Version only exists in `pyproject.toml`
*   Debian compatibility: `pybuild` can parse version information
*   Build reproducibility: Same version used for both `.deb` and `.whl` artifacts

## Workflow Execution Summary

### Complete Release Pipeline Flow

The following table summarizes the complete execution order during a release:

| Step | Job Name | Workflow | Key Actions | Artifacts |
| --- | --- | --- | --- | --- |
| 1 | `create-temp-branch` | release.yml | Creates `rc-temp-*` branch | Branch reference |
| 2 | `versionchange` | release.yml | Runs `bump2version`, extracts version components | Version outputs |
| 3 | `changelogs` | release.yml | Generates `debian/changelog` with `gbp dch` | Changelog commit |
| 4 | `build_all_depends` | build-all.yml | Triggers parallel builds | None (orchestrator) |
| 5a | `build-deb` | build-debian.yml | Matrix build for 3 Ubuntu versions | `.deb`, `.changes`, `changelog` |
| 5b | `buildpypi` | build-pypi.yml | Generates Python wheel | `.whl` |
| 6 | `tagrelease` | release.yml | Creates and pushes `v*.*.*` git tag | Git tag |
| 7 | `generate-release` | release.yml | Creates GitHub Release, attaches artifacts | GitHub Release |
| 8 | `publish-to-testpypi` | release.yml | Uploads to test.pypi.org | PyPI package |
| 9 | `publish-to-pypi` | release.yml | Uploads to pypi.org | PyPI package |
| 10 | `mergeback` | release.yml | Rebases to main, deletes temp branch | Merged commits |

**Sources:**[.github/workflows/release.yml 21-453](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/release.yml#L21-L453)
