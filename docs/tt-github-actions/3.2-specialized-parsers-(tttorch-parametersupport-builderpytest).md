This page provides a technical deep dive into the specialized parsers used within the `collect_data` action. While standard JUnit XML parsers handle basic test results, these specialized parsers are designed to extract rich metadata—such as tensor shapes, hardware card types, and compilation stages—from complex artifact formats like TAR archives and custom JSON reports.

## TTTorchModelTestsParser

The `TTTorchModelTestsParser` is specifically designed to parse operation-level compilation and execution data from the `tt-torch` project. Unlike other parsers that read a single file, this parser processes `.tar` archives containing multiple JSON files, each representing a model's operation test results `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L30-L31" min=30 max=31 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`.

### Implementation Details

*   **Extraction**: It extracts the TAR archive to `/tmp` and iterates through all contained `.json` files `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L46-L59" min=46 max=59 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`.
*   **Status Mapping**: It uses the `OpCompilationStatus``IntEnum` to determine test success. A test is only considered successful if it reaches the `EXECUTED` stage (value 7) `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L19-L75" min=19 max=75 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`.
*   **Tensor Descriptors**: It maps input and output tensor metadata (shape, data type, layout, etc.) into `TensorDesc` Pydantic models `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L123-L133" min=123 max=133 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`.

## ParameterSupportTestParser

The `ParameterSupportTestParser` processes JSON reports generated by the `tt-shield` project. These reports validate whether specific model parameters (e.g., `max_tokens`, `top_p`) are supported by the inference server `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parameter_support_test_parser.py#L37-L38" min=37 max=38 file-path=".github/actions/collect_data/src/parsers/parameter_support_test_parser.py">Hii</FileRef>`.

### Key Features

*   **Metadata Precedence**: The parser extracts global metadata (like `model_name` and `device`) and merges it with specific test results. Metadata values take precedence over values defined within the `parameter_support_tests` block, except for the `results` key itself `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parameter_support_test_parser.py#L70-L81" min=70 max=81 file-path=".github/actions/collect_data/src/parsers/parameter_support_test_parser.py">Hii</FileRef>`.
*   **Grouping**: Tests are grouped by the parameter being tested (e.g., `test_n`, `test_max_tokens`), which is mapped to the `group` field in the `Test` model `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parameter_support_test_parser.py#L83-L134" min=83 max=134 file-path=".github/actions/collect_data/src/parsers/parameter_support_test_parser.py">Hii</FileRef>`.
*   **Configuration**: Specific environment details like `endpoint_url` are stored in the `config` field via the `ParameterSupportTestConfig` dataclass `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parameter_support_test_parser.py#L20-L140" min=20 max=140 file-path=".github/actions/collect_data/src/parsers/parameter_support_test_parser.py">Hii</FileRef>`.

## BuilderPytestParser

The `BuilderPytestParser` is a specialized JUnit XML parser used for low-level operation tests (Builder tests). It identifies compatible files by looking for a specific `<property name="card" ...>` tag within the XML's `<properties>` section `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L25-L45" min=25 max=45 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.

### Advanced Metadata Extraction

*   **Hardware Context**: It extracts the `card` (e.g., n300, n150) and `git_sha` from the testsuite properties to provide hardware-specific context for failures `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L85-L106" min=85 max=106 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.
*   **Failure Stages**: It maps a custom `failure_stage` property to the `TestStatus` enum (e.g., `compile_failed`, `golden_failed`, `run_failed`) `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L17-L182" min=17 max=182 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.
*   **Dynamic Config**: Any XML property prefixed with `param_` is automatically stripped of the prefix and added to the test's `config` dictionary `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L163-L168" min=163 max=168 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`.

## Validation Patterns

All specialized parsers utilize Pydantic models to ensure data integrity before the data is uploaded to SFTP.

| Parser | Model Used | Validation Strategy |
| --- | --- | --- |
| **TTTorch** | `OpTest` | Validates `compilation_status` against `OpCompilationStatus` enum `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L70-L75" min=70 max=75 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`. |
| **ParameterSupport** | `Test` | Ensures `model_name` and `device` are present in metadata `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/parameter_support_test_parser.py#L20-L30" min=20 max=30 file-path=".github/actions/collect_data/src/parsers/parameter_support_test_parser.py">Hii</FileRef>`. |
| **BuilderPytest** | `OpTest` | Validates `backend` and `failure_stage` existence in XML properties `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L171-L182" min=171 max=182 file-path=".github/actions/collect_data/src/parsers/builder_pytest_parser.py">Hii</FileRef>`. |

If validation fails, the parsers call `failure_happened()` and log the error, but they often continue processing other tests to maximize data collection `<FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/python_unittest_parser.py#L102-L104" min=102 max=104 file-path=".github/actions/collect_data/src/parsers/python_unittest_parser.py">Hii</FileRef> <FileRef file-url="https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py#L117-L120" min=117 max=120 file-path=".github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py">Hii</FileRef>`.
