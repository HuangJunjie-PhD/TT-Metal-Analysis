## Build Toolchain: git-buildpackage

The packaging system uses **git-buildpackage** (gbp), a suite of tools for maintaining Debian packages in Git repositories. The two primary commands are:

| Command | Purpose | Usage in Workflow |
| --- | --- | --- |
| `gbp dch` | Generate/update `debian/changelog` from Git commit history | Automated version bumping before builds |
| `gbp buildpackage` | Build Debian package from Git repository | Creates `.deb`, `.changes`, and signature files |

The workflow uses `gbp dch` with the following flags:

*   `--debian-branch ${ref_name}`: Specifies the Git branch to work from
*   `-R` / `--release`: Mark the changelog entry as released
*   `-N ${MAJOR}.${MINOR}.${PATCH}`: Set the new version number
*   `--spawn-editor=never`: Non-interactive mode for CI environments

**Sources**: [.github/workflows/build-debian.yml 120-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L120-L124)

## GitHub Actions Workflow

The `build-debian.yml` workflow orchestrates Debian package creation for multiple Ubuntu versions. It can be triggered manually (`workflow_dispatch`) or called from other workflows (`workflow_call`).

**Input Parameters**:

*   `ref`: Git reference to build from (default: `${{ github.ref }}`)
*   `MAJOR`, `MINOR`, `PATCH`: Version components
*   `NUMBER_OF_COMMITS_SINCE_TAG`: Used for development versions

**Sources**: [.github/workflows/build-debian.yml 1-179](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L1-L179)

## Version Management and Changelog

The `debian/changelog` file follows Debian's strict format requirements and serves as both a version database and release history. Each entry records:

```
package-name (version) distribution; urgency=level

  [ Author Name ]
  * Change description
  * Another change

 -- Maintainer Name <email>  Date in RFC 2822 format
```

| `dh-python` | Debhelper add-on for Python packages |

| `gnupg` | GPG signing infrastructure |
| `pybuild-plugin-pyproject` | PEP 517/518 build backend support |
| `python3-tomli` | TOML parser for `pyproject.toml` |

### Build Rules

The `debian/rules` file defines the build recipe using `debhelper` automation:

[debian/rules 1-11](https://github.com/tenstorrent/tt-flash/blob/60c06032/debian/rules#L1-L11)

**Key Directives**:

*   `--buildsystem=pybuild`: Use `pybuild` for Python package building
*   `--with=python3`: Enable Python 3 integration
*   `override_dh_auto_test`: Disable test suite during packaging (tests run in separate CI jobs)

## Integration with Release Process

The Debian build workflow is called by the main release workflow ([Release Process](https://deepwiki.com/tenstorrent/tt-flash/9.1-release-process)). When invoked:

1.   The release workflow passes version components (`MAJOR`, `MINOR`, `PATCH`) as inputs
2.   The Debian workflow builds packages for all Ubuntu versions in parallel
3.   Artifacts are uploaded to GitHub and can be attached to release notes
4.   The signed `.changes` file enables package repository uploads

This modular design allows Debian packages to be built independently or as part of the full release pipeline.

**Sources**: [.github/workflows/build-debian.yml 3-23](https://github.com/tenstorrent/tt-flash/blob/60c06032/.github/workflows/build-debian.yml#L3-L23)
