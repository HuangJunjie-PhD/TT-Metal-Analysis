## Purpose and Scope

TIR Operators are the fundamental building blocks for constructing TensorIR (TIR) programs, which represent low-level tensor computations. These operators provide both standard mathematical operations (arithmetic, logical, bitwise) and hardware-specific intrinsics for accelerators (CUDA Tensor Cores, PTX instructions, barrier synchronization).

This page documents the complete set of operators available at the TIR level, their semantics, and how they are used in kernel construction. For information about TIR program structure and concepts, see [TIR Concepts and Structure](https://deepwiki.com/tenstorrent/tt-tvm/5.1-tir-concepts-and-structure). For information about scheduling primitives that transform TIR programs, see [Schedule API](https://deepwiki.com/tenstorrent/tt-tvm/5.3-schedule-api).

## Arithmetic and Mathematical Operators

### Basic Arithmetic

TIR provides standard arithmetic operations with automatic type matching and constant folding for index types (`int32`, `int64`).

| Operator | Python Function | C++ Function | Description |
| --- | --- | --- | --- |
| `+` | `add(a, b)` | `add(a, b)` | Addition with type matching |
| `-` (unary) | `neg(a)` | `neg(a)` | Negation |
| `-` (binary) | `sub(a, b)` | `sub(a, b)` | Subtraction |
| `*` | `mul(a, b)` | `mul(a, b)` | Multiplication |
| `/` | `div(a, b)` | `div(a, b)` | Division (C semantics) |
| `//` | `floordiv(a, b)` | `floordiv(a, b)` | Floor division |
| `%` | `truncmod(a, b)` | `truncmod(a, b)` | Truncated modulo |

**Type Matching Behavior**: When operands have different types, TIR applies the following rules ([src/tir/op/op.cc 103-180](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L103-L180)):

*   If both are floats: cast to higher precision (e.g., `float16 + float32` → `float32`)
*   If one is float: cast int to float
*   If both are integers: cast to higher bit width
*   If mixing signed/unsigned: cast to wider type

**Sources**: [src/tir/op/op.cc 103-180](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L103-L180)[src/tir/op/op.cc 343-456](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L343-L456)[include/tvm/tir/op.h 141-442](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/include/tvm/tir/op.h#L141-L442)

### Division Variants

TIR provides multiple division semantics for different use cases:

| Function | Semantics | Use Case |
| --- | --- | --- |
| `div(a, b)` | C-style division (truncation) | General purpose |
| `truncdiv(a, b)` | Explicit truncation toward zero | C semantics |
| `floordiv(a, b)` | Floor division | Indexing, iteration |
| `ceildiv(a, b)` | Ceiling division | Shape calculation |
| `indexdiv(a, b)` | Optimized for non-negative (=floordiv) | Index splitting |
| `shapediv(a, b)` | Optimized for non-negative (=ceildiv) | Shape splitting |

### Logical Operators

Logical operators require boolean operands and perform eager constant folding:

| Operator | Function | Truth Table |
| --- | --- | --- |
| `&&` | `logical_and(a, b)` | `false && x = false`, `true && x = x` |
| `||` | `logical_or(a, b)` | `true || x = true`, `false || x = x` |
| `!` | `logical_not(a)` | `!true = false`, `!false = true` |

**Type Safety**: Logical operators validate that inputs are boolean ([src/tir/op/op.cc 524-571](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L524-L571)):

## Type Operations

### Cast

`cast(dtype, value, span=Span())` converts a value to a different data type. For scalar types, it performs constant folding when possible ([src/tir/op/op.cc 292-332](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L292-L332)):

**Broadcasting**: When casting to a vector type from a scalar, TIR automatically broadcasts ([src/tir/op/op.cc 304-332](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L304-L332)):

### Reinterpret

`reinterpret(dtype, value, span=Span())` reinterprets the bit pattern of a value as a different type without conversion. Requires matching bit width ([src/tir/op/op.cc 335-340](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L335-L340)):

## Control Flow Operators

### if_then_else

`if_then_else(cond, true_value, false_value, span=Span())` is a ternary conditional expression that evaluates to one of two values based on a boolean condition ([src/tir/op/op.cc 459-473](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L459-L473)):

**Eager Evaluation**: If the condition is a constant, the operator immediately returns the appropriate branch.

### likely

`likely(cond, span=Span())` provides a hint to the compiler that a condition is likely to be true, enabling better branch prediction ([src/tir/op/op.cc 476-479](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L476-L479)):

### PTX Intrinsics

Low-level PTX instructions for advanced GPU programming:

**Matrix Operations**:

*   `ptx_mma(dtype, shape, A_layout, B_layout, ...)`: PTX MMA instruction
*   `ptx_mma_sp(dtype, shape, ...)`: Sparse MMA with metadata
*   `ptx_ldmatrix(dtype, trans, num, type, ...)`: Load matrix from shared memory
*   `mma_store(dtype, m, n, ...)`: Store MMA result
*   `mma_fill(dtype, local_size, ...)`: Fill MMA accumulator

**Async Copy**:

*   `ptx_cp_async(dtype, shared_ptr, ..., bytes)`: Asynchronous copy (cp.async)
*   `ptx_cp_async_bulk(dtype, shared_ptr, ..., barrier_id)`: Bulk async copy
*   `ptx_commit_group()`: Commit async copy group
*   `ptx_wait_group(num)`: Wait for async copy completion

**Barrier Synchronization**:

*   `ptx_init_barrier_thread_count(barrier_id, thread_count)`: Initialize barrier
*   `ptx_arrive_barrier(barrier_id)`: Arrive at barrier
*   `ptx_arrive_barrier_expect_tx(barrier_id, byte_count)`: Arrive with expected transaction
*   `ptx_wait_barrier(barrier_id)`: Wait for barrier completion
*   `ptx_cp_async_barrier(barrier_id)`: Arrive at barrier after async copy
*   `create_barriers(count)`: Create barrier shared memory

## Operator Implementation Details

### Constant Folding

TIR operators perform eager constant folding for index types during construction ([src/tir/op/op.cc 346-348](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L346-L348)):

This optimization simplifies expressions like `(3 + 4)` to `7` at IR construction time, reducing IR size and improving subsequent optimization passes.

### Operator Registration

TIR operators are registered through the operator registry system ([src/tir/op/op.cc 45-52](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/tir/op/op.cc#L45-L52)):

Operators are marked as `kPure` (no side effects), `kReadState`, `kUpdateState`, or `kOpaque` based on their behavior.

## Summary

TIR operators form the complete vocabulary for expressing tensor computations at the kernel level. They provide:

1.   **Standard Operations**: Full suite of arithmetic, logical, bitwise, and mathematical operations with automatic type matching
2.   **Hardware Intrinsics**: Direct access to accelerator features (Tensor Cores, PTX, barriers)
3.   **Memory Management**: Stack allocation, buffer access, workspace management
4.   **Call Interface**: Flexible mechanism for external functions and cross-module calls
5.   **Type Safety**: Validation and type checking at operator construction
6.   **Optimization**: Eager constant folding and simplification

The operators are used primarily through TVMScript for human-readable kernel authoring, and are transformed by the Schedule API during optimization. The final operators are lowered to target-specific code by the code generation backends.
