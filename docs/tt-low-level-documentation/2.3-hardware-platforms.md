## Purpose and Scope

This page describes the hardware platforms supported by TT-Metal, focusing on the physical characteristics, core configurations, and interface specifications that impact data movement performance. The two platforms documented are **Wormhole B0** and **Blackhole P100**, representing successive generations of Tenstorrent architecture.

For NOC architecture details that are common across both platforms, see [NOC Architecture](https://deepwiki.com/tenstorrent/tt-low-level-documentation/2.2-noc-architecture). For platform-specific performance metrics, see [Baseline Performance Metrics](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.1-baseline-performance-metrics). For detailed register-level differences, see [Register State Preservation](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.1-register-state-preservation) and [Platform Comparison](https://deepwiki.com/tenstorrent/tt-low-level-documentation/5.2-platform-comparison).

## Platform Overview

TT-Metal supports two hardware platforms with different scale and capabilities:

| Platform | Generation | Target Use Case | Architectural Similarity |
| --- | --- | --- | --- |
| **Wormhole B0** | Earlier generation | Production workloads | Baseline NOC architecture |
| **Blackhole P100** | Next generation | Higher performance workloads | NOC architecture nearly identical to WH |

Both platforms share the same fundamental NOC design: dual NOC instances (NOC0 and NOC1), 2D torus topology, packet-switched network with virtual channels, and RISC-V processor control (BRISC for NOC0, NCRISC for NOC1). The differences lie primarily in scale (core count) and interface width.

## Wormhole B0 Platform

### Core Configuration

### Technical Specifications

| Parameter | Value | Description |
| --- | --- | --- |
| **Tensix Cores** | 64 | Total number of compute cores |
| **Interface Width** | 256 bits/cycle | NOC data path width |
| **Interface Width (Bytes)** | 32 bytes/cycle | Equivalent byte throughput |
| **Maximum Packet Size** | 8 KB | Largest packet that can be transmitted |
| **NOC Instances** | 2 (NOC0, NOC1) | Independent network instances |
| **Virtual Channels** | 16 per NOC | For traffic management |
| **Addressing** | 48-bit | Address space size |
| **Topology** | 2D Torus | Network topology |

### Register Set

Wormhole B0 uses a specific set of NOC registers defined in the command buffers. Key registers include:

*   `NOC_TARG_ADDR_LO`, `NOC_TARG_ADDR_MID` (48-bit addressing, no `_HI`)
*   `NOC_RET_ADDR_LO`, `NOC_RET_ADDR_MID`
*   `NOC_PACKET_TAG`
*   `NOC_CTRL`
*   `NOC_AT_LEN_BE`
*   `NOC_CMD_CTRL`
*   `RISC_IF_STATUS_OFFSET` (present only on Wormhole)

## Blackhole P100 Platform

### Core Configuration

### Technical Specifications

| Parameter | Value | Description |
| --- | --- | --- |
| **Tensix Cores** | 110 | Total number of compute cores (+72% vs WH) |
| **Interface Width** | 512 bits/cycle | NOC data path width (2x WH) |
| **Interface Width (Bytes)** | 64 bytes/cycle | Equivalent byte throughput (2x WH) |
| **Maximum Packet Size** | 16 KB | Largest packet that can be transmitted (2x WH) |
| **NOC Instances** | 2 (NOC0, NOC1) | Independent network instances |
| **Virtual Channels** | 16 per NOC | For traffic management |
| **Addressing** | 64-bit | Address space size (extended from WH) |
| **Topology** | 2D Torus | Network topology |

### Enhanced Register Set

Blackhole introduces additional NOC registers for enhanced functionality:

**New Registers (not present in Wormhole):**

*   `NOC_TARG_ADDR_HI`, `NOC_RET_ADDR_HI` (for 64-bit addressing)
*   `NOC_AT_LEN_BE_1` (additional atomic length register)
*   `NOC_BRCST_EXCLUDE` (broadcast exclusion control)
*   `NOC_L1_ACC_AT_INSTRN` (L1 access atomic instruction)
*   `NOC_SEC_CTRL` (security control)
*   `CMD_BUF_AVAIL`, `CMD_BUF_OVFL` (command buffer status)

**Removed Registers:**

*   `RISC_IF_STATUS_OFFSET` (Wormhole only)

## Architectural Evolution

### Platform Progression

### Design Philosophy

The evolution from Wormhole B0 to Blackhole P100 follows a **scale-and-enhance** approach:

1.   **Preserve Core Architecture**: The fundamental NOC design remains unchanged (dual NOC instances, 2D torus, virtual channels, packet-switched network)
2.   **Scale Resources**: Increase core count from 64 to 110 (+72%)
3.   **Widen Interfaces**: Double the interface width from 256 to 512 bits/cycle
4.   **Enhance Capabilities**: Add registers for new features (broadcast exclusion, security control, atomic operations)
5.   **Maintain Compatibility**: Keep API and programming model consistent

This approach allows code written for Wormhole to run on Blackhole with minimal changes, while enabling Blackhole to achieve higher performance through increased parallelism and bandwidth.

### Performance Comparison

Performance measured in **Bytes/cycle** for common data movement primitives:

| Primitive | Wormhole B0 | Blackhole P100 | Improvement |
| --- | --- | --- | --- |
| DRAM Read | 22 B/cycle | 33 B/cycle | +50% |
| DRAM Write | 21 B/cycle | 34 B/cycle | +62% |
| One To One | 29 B/cycle | 60 B/cycle | +107% |
| One From One | 28 B/cycle | 60 B/cycle | +114% |
| One To All (Unicast) | 31 B/cycle | 62 B/cycle | +100% |
| One To All (Multicast) | 15 B/cycle | 24 B/cycle | +60% |
| One To All (Multicast + Linked) | 22 B/cycle | 41 B/cycle | +86% |
| One From All | 30 B/cycle | 60 B/cycle | +100% |

**Key Observations:**

*   Core-to-core operations (One To One, One From One, One From All) achieve near-2x speedup, matching the 2x interface width increase
*   DRAM operations show ~50-60% improvement, indicating DRAM bandwidth as a limiting factor
*   Multicast operations show variable improvements (60-86%) due to network congestion effects

## Register Availability Matrix

### Platform-Specific Register Differences

The following table shows register availability across platforms, highlighting which registers are platform-specific:

| Register | Wormhole B0 | Blackhole P100 | Notes |
| --- | --- | --- | --- |
| `NOC_TARG_ADDR_LO` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_TARG_ADDR_MID` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_TARG_ADDR_HI` | ✗ Unused | ✓ Stateful | **Blackhole only** (64-bit addressing) |
| `NOC_RET_ADDR_LO` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_RET_ADDR_MID` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_RET_ADDR_HI` | ✗ Unused | ✓ Stateful | **Blackhole only** (64-bit addressing) |
| `NOC_PACKET_TAG` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_CTRL` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_AT_LEN_BE` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_AT_LEN_BE_1` | ✗ Not present | ✓ Stateful | **Blackhole only** |
| `NOC_AT_DATA` | ✓ Stateful | ✓ Stateful | Both platforms |
| `NOC_BRCST_EXCLUDE` | ✗ Not present | ✓ Stateful | **Blackhole only** |
| `NOC_L1_ACC_AT_INSTRN` | ✗ Not present | ✓ Stateful | **Blackhole only** |
| `NOC_SEC_CTRL` | ✗ Not present | ✓ Stateful | **Blackhole only** |
| `NOC_CMD_CTRL` | ✓ Non-stateful | ✓ Non-stateful | Both platforms (control flag) |
| `NOC_NODE_ID` | ✓ Non-stateful | ✓ Non-stateful | Both platforms (control flag) |
| `NOC_ENDPOINT_ID` | ✓ Non-stateful | ✓ Non-stateful | Both platforms (control flag) |
| `RISC_IF_STATUS_OFFSET` | ✓ Non-stateful | ✗ Not present | **Wormhole only** |
| `CMD_BUF_AVAIL` | ✗ Not present | ✓ Non-stateful | **Blackhole only** |
| `CMD_BUF_OVFL` | ✗ Not present | ✓ Non-stateful | **Blackhole only** |

**Legend:**

*   ✓ = Register exists on platform
*   ✗ = Register does not exist on platform
*   **Stateful** = Register preserves value across API calls
*   **Non-stateful** = Register is modified or used as control flag in each API call

## Platform Selection Considerations

When choosing between Wormhole B0 and Blackhole P100 for development and deployment:

### Choose Wormhole B0 When:

*   Working with existing production code that targets Wormhole
*   64 cores are sufficient for the workload
*   48-bit addressing is adequate
*   Register compatibility with older codebases is required (`RISC_IF_STATUS_OFFSET`)

### Choose Blackhole P100 When:

*   Maximum performance is needed (2x interface width)
*   Workload benefits from 110 cores (72% more parallelism)
*   64-bit addressing is required for large memory spaces
*   Advanced features are needed: 
    *   Broadcast exclusion control (`NOC_BRCST_EXCLUDE`)
    *   Enhanced atomic operations (`NOC_AT_LEN_BE_1`, `NOC_L1_ACC_AT_INSTRN`)
    *   Security controls (`NOC_SEC_CTRL`)
    *   Command buffer overflow detection (`CMD_BUF_AVAIL`, `CMD_BUF_OVFL`)

### Code Portability:

Due to the preserved NOC architecture, most data movement code is portable between platforms. The primary considerations for porting are:

1.   Core count differences (adjust grid sizes from 7×7 to 9×9 max)
2.   Register availability (conditional compilation for platform-specific registers)
3.   Performance expectations (Blackhole achieves near-2x speedup for many operations)
