This page provides a comprehensive catalog of all data types supported by the Tenstorrent architecture, including their bit representations, storage locations, and characteristics. Each data type is described with its precision, range, and where it can reside within the memory hierarchy (L1 RAM, register files).

For information about conversions between data types, see [Format Conversion Pipeline](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.3-format-conversion-pipeline). For details about floating-point computation models and rounding behavior, see [Floating-Point Models and FMA](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.1-floating-point-models-and-fma). For information about compressed Block Floating-Point formats, see [Compression and BFP Formats](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.4-compression-and-bfp-formats).

## Overview

The Tenstorrent architecture supports three broad categories of data types:

1.   **Standard Floating-Point Types**: FP32, TF32, BF16, FP16, FP8
2.   **Block Floating-Point Types**: BFP8, BFP4, BFP2, BFP8a, BFP4a, BFP2a (compressed formats)
3.   **Integer Types**: INT32, INT16, INT8, UINT8

Data types have different representations depending on where they are stored:

*   **L1 Memory**: Standard IEEE754-like layouts (or compressed BFP formats)
*   **SrcA/SrcB Register Files**: Non-standard layouts optimized for Matrix Unit (FPU)
*   **Dst Register File**: Non-standard layouts with rearranged fields
*   **LReg Register File**: Standard IEEE754 layouts for Vector Unit (SFPU)

## Standard Floating-Point Types

### FP32 (32-bit IEEE754 Single Precision)

**Storage Locations**: L1, Dst, LReg

**Precision**: 1 sign bit, 8-bit exponent, 23-bit mantissa

**In L1 Memory (standard IEEE754 layout)**:

```
Bits: S EEEEEEEE MMMMMMMMMMMMMMMMMMMMMMM
      31 30    23 22                   0
```

**In Dst Register (non-standard layout)**: The coprocessor stores FP32 in Dst with rearranged bit fields. The high 16 bits store sign, mantissa high bits, and exponent in a BF16-like arrangement, while the low 16 bits store the remaining mantissa bits.

```
Bits: S MMMMMMM EEEEEEEE MMMMMMMMMMMMMMMM
      31 30   24 23    16 15             0
```

**RISC-V Access**: When RISC-V cores access Dst with `fmt=0`, automatic conversion between IEEE754 layout and Dst layout occurs unless `no_swizzle=true`.

**Notes**:

*   LReg always stores FP32 in standard IEEE754 layout
*   Unpacker can convert FP32â†’TF32, FP32â†’BF16, FP32â†’FP16 during data movement
*   Does not fully conform to IEEE754; see [Floating-Point Models and FMA](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.1-floating-point-models-and-fma) for divergences

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Dst.md 36-38](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L36-L38)[WormholeB0/TensixTile/TensixCoprocessor/Dst.md 79-81](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L79-L81)[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 49-51](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L49-L51)

### TF32 (TensorFloat-32)

**Storage Locations**: L1, SrcA/SrcB

**Precision**: 1 sign bit, 8-bit exponent, 10-bit mantissa (19 bits total)

**In L1 Memory**: TF32 is stored identically to FP32 (32 bits), with the low 13 mantissa bits typically zeroed by software.

**In SrcA/SrcB Register Files (non-standard layout)**:

```
Bits: S MMMMMMMMMM EEEEEEEE
      18 17       8 7      0
```

The fields are rearranged from sign-exponent-mantissa to sign-mantissa-exponent.

**Usage Notes**:

*   When unpacking TF32 from L1 to Dst, it becomes full FP32 precision
*   Matrix Unit (FPU) operates on TF32 with 10-bit mantissa precision
*   Conversion: FP32â†’TF32 truncates low 13 mantissa bits

### BF16 (BFloat16)

**Storage Locations**: L1, SrcA/SrcB, Dst

**Precision**: 1 sign bit, 8-bit exponent, 7-bit mantissa

**In L1 Memory (standard layout)**:

```
Bits: S EEEEEEEE MMMMMMM
      15 14     7 6     0
```

**In SrcA/SrcB Register Files (non-standard layout, 19 bits)**:

```
Bits: S MMMMMMM 00 EEEEEEEE
      18 17   11 10 9      0
```

Fields rearranged to sign-mantissa-exponent, zero-extended to 19 bits.

**In Dst Register (non-standard layout, 16 bits)**:

```
Bits: S MMMMMMM EEEEEEEE
      15 14    8 7      0
```

Fields rearranged to sign-mantissa-exponent.

**Notes**:

*   Unpackers can convert BFP8/BFP4/BFP2 losslessly to BF16
*   Direct truncation of FP32 (keep high 16 bits) approximates FP32â†’BF16 conversion
*   When accessing Dst via RISC-V with `fmt=3`, automatic bit rearrangement occurs

### FP16 (IEEE754 Half Precision)

**Storage Locations**: L1, SrcA/SrcB, Dst

**Precision**: 1 sign bit, 5-bit exponent, 10-bit mantissa

**In L1 Memory (standard IEEE754 layout)**:

```
Bits: S EEEEE MMMMMMMMMM
      15 14  10 9        0
```

**In SrcA/SrcB Register Files (non-standard layout, 19 bits)**:

```
Bits: S 000 MMMMMMMMMM EEEEE
      18 17 15 14      5 4   0
```

Sign bit moved to bit 18, mantissa in middle, exponent in low bits, zero-extended.

**In Dst Register (non-standard layout, 16 bits)**:

```
Bits: S MMMMMMMMMM EEEEE
      15 14       5 4   0
```

Fields rearranged to sign-mantissa-exponent.

**Infinity Representation**: Differs from IEEE754. The architecture uses exponent `0b11111` with non-zero mantissa for infinity.

**NaN**: Not supported by the architecture.

**Notes**:

*   Unpackers can convert FP8 (e5m2) losslessly to FP16
*   Unpackers can convert BFP8a/BFP4a/BFP2a losslessly to FP16
*   When accessing Dst via RISC-V with `fmt=2`, automatic bit rearrangement occurs

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Dst.md 43-53](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L43-L53)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 522-529](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L522-L529)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 563-565](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L563-L565)

### FP8 (8-bit Floating-Point, e5m2 format)

**Storage Locations**: L1

**Precision**: 1 sign bit, 5-bit exponent, 2-bit mantissa (implied leading 1)

**In L1 Memory**:

```
Bits: S EEEEE MM
      7 6   2 1 0
```

**Usage**:

*   Left-shifted by 8 bits during unpacking to become FP16
*   No direct computation support; must be converted to FP16 first
*   Used for ultra-low precision neural network weights/activations

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 452](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L452-L452)

## Block Floating-Point (BFP) Types

Block Floating-Point formats store multiple values sharing a common exponent, enabling compression. Each format has two variants: BFP (converts to BF16) and BFPa (converts to FP16).

### BFP8 and BFP8a

**Storage**: L1 only (compressed format)

**Size**: 1 byte per datum, plus 1 shared exponent per 16 datums

**Precision**: 8-bit sign-magnitude representation

**Conversion**:

*   BFP8 â†’ BF16: Normalize mantissa and apply shared exponent
*   BFP8a â†’ FP16: Similar normalization but FP16 exponent range

**Decompression Algorithm** (BFP8 â†’ BF16):

```
sign = datum[7]
mag  = datum[6:0] << 1
if (mag == 0):
    return sign ? 0xff80 : 0
leading_zeros = count_leading_zeros(mag)
mag <<= leading_zeros
exponent = shared_exp - leading_zeros
return (sign << 15) | (exponent << 7) | (mag & 0x7e)
```

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 495-520](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L495-L520)

### BFP4 and BFP4a

**Storage**: L1 only (compressed format)

**Size**: 0.5 bytes (4 bits) per datum, plus 1 shared exponent per 16 datums

**Precision**: 4-bit sign-magnitude representation

**Conversion**: Similar to BFP8, but datum is left-shifted by 4 bits before processing

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 454-455](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L454-L455)

### BFP2 and BFP2a

**Storage**: L1 only (compressed format)

**Size**: 0.25 bytes (2 bits) per datum, plus 1 shared exponent per 16 datums

**Precision**: 2-bit sign-magnitude representation

**Conversion**: Similar to BFP8, but datum is left-shifted by 6 bits before processing

**Performance Impact**: Always unpacks at x1 speed (16 bytes/cycle maximum)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 455-456](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L455-L456)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 593](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L593-L593)

**Storage Locations**: L1, Dst

**Format**: 1 sign bit, 31-bit magnitude (sign-magnitude representation)

**In L1 and Dst**:

```

      31 30                            0
```

### Integer "16" (16-bit Opaque Data)

**Storage Locations**: L1, SrcA/SrcB, Dst

**Format**: 1 sign bit, 15-bit magnitude

**In L1 Memory**:

```
Bits: S MMMMMMMMMMMMMMM
      15 14            0
```

**In SrcA/SrcB Register Files** (19 bits):

```
Bits: 000 MMMMMMMM 000 MMMMMMM
      18 16 15    8  7 5 6    0
```

Split into high and low bytes with zero-padding.

**In Dst Register**:

```
Bits: S MMMMMMMMMMMMMMM
      15 14            0
```

**Purpose**: Intended for opaque 16-bit data transfer, not computation. Can hold any 16-bit pattern including unsigned values.

**Notes**:

*   No computation instructions operate on this type
*   Useful for transferring data that will be reinterpreted as another type
*   When RISC-V accesses Dst with `fmt=4` and `unsigned_int=false`, conversion to two's complement occurs

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Dst.md 30-34](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L30-L34)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 487](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L487-L487)

### Integer "8" (10-bit Magnitude in 16-bit Container)

**Storage Locations**: L1, SrcA/SrcB, Dst

**Format**: 1 sign bit, 10-bit magnitude, overlaid onto FP16 representation

**In Dst Register** (16 bits):

```
Bits: S MMMMMMMMMM EEEEE
      15 14       5 4   0
```

Low 5 bits contain fixed exponent value (16 for non-zero, 0 for zero).

**L1 Storage**:

*   INT8 (sign-magnitude): 1 sign bit + 7-bit magnitude in 8 bits
*   UINT8 (unsigned): 8-bit magnitude

**Range**:

*   Full range in register: -1023 to +1023
*   Typical range after INT8/UINT8 conversion: -127 to +127 or 0 to +255

**Conversion from L1**:

*   INT8 or UINT8 formats specify interpretation
*   Configuration field `ALU_FORMAT_SPEC_REG0_SrcAUnsigned` or `SrcBUnsigned` controls signedness
*   Magnitude placed in bits [14:5], dummy exponent (16) in bits [4:0]

**RISC-V Access**:

*   With `fmt=5`, loads extract 7 or 8 bits of magnitude
*   Stores with `fmt=5` and `unsigned_int=false` have incorrect sign-magnitude conversion behavior

## Configuration Fields for Data Types

The following configuration fields control data type interpretation:

**In `THCON_SEC[0]` and `THCON_SEC[1]` (Unpacker Configuration):**

*   `REG0_TileDescriptor.InDataFormat` (4 bits): Source format in L1
*   `REG2_Out_data_format` (4 bits): Destination format in SrcA/SrcB/Dst
*   `REG7_Unpack_data_format_cntx[i]` (4 bits): Per-context input format override
*   `REG7_Unpack_out_data_format_cntx[i]` (4 bits): Per-context output format override

**Data Format Encoding (4 bits):**

## Architectural Notes

### Non-Standard Bit Layouts

The Tensix Coprocessor uses non-standard bit layouts for internal register files (SrcA, SrcB, Dst) that differ from both IEEE754 standards and L1 memory layouts. These rearrangements optimize hardware implementation:

*   **SrcA/SrcB**: Store sign-mantissa-exponent order (19 bits)
*   **Dst**: Store sign-mantissa-exponent order (16 or 32 bits)
*   **LReg**: Store standard IEEE754 layout (32 bits)

Automatic conversion happens during:

*   `UNPACR`: L1 â†’ SrcA/SrcB/Dst
*   `PACR`: Dst â†’ L1
*   RISC-V load/store: CPU registers â†” Dst (unless `no_swizzle=true`)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Dst.md 42-81](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L42-L81)[WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md 420-565](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/UNPACR_Regular.md?plain=1#L420-L565)

### Valid Bit Tracking

The Dst register file maintains valid bits for each row (`DstRowValid[1024]`). Rows are marked invalid by `ZEROACC` instruction and become valid when written. Reading invalid rows produces:

*   Packers: Read as zero
*   Matrix Unit: Read as identity element for operation (0 or -âˆž)
*   Vector Unit: Undefined behavior
*   Unpackers: Undefined behavior for partial writes

Sources: [WormholeB0/TensixTile/TensixCoprocessor/ZEROACC.md 1-86](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/ZEROACC.md?plain=1#L1-L86)[BlackholeA0/TensixTile/TensixCoprocessor/Dst.md 9-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/Dst.md?plain=1#L9-L13)

### IEEE754 Non-Conformance

The architecture deviates from IEEE754 in several ways:

*   FP16 infinity uses different bit pattern
*   FP16 NaN is not supported
*   Rounding modes differ from standard
*   Denormal handling may differ

See [Floating-Point Models and FMA](https://deepwiki.com/tenstorrent/tt-isa-documentation/6.1-floating-point-models-and-fma) for complete details on computational behavior.
