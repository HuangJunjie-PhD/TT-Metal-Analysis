# Constants and Configuration System

Relevant source files
*   [tt_smi/constants.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py)

## Purpose and Scope

This document describes the `constants.py` module, which serves as the centralized configuration system for TT-SMI. This module defines all static configuration data including telemetry field names, table headers, platform-specific mappings, hardware limits, and GUI configuration. The constants system separates backend data collection specifications from frontend display formatting.

For information about how these constants are used during data collection, see [Data Collection Pipeline](https://deepwiki.com/tenstorrent/tt-smi/6.3-data-collection-pipeline). For backend implementation details, see [Backend System (TTSMIBackend)](https://deepwiki.com/tenstorrent/tt-smi/6.2-backend-system-(ttsmibackend)). For hardware platform differences, see [Hardware Platform Support](https://deepwiki.com/tenstorrent/tt-smi/4-hardware-platform-support).

**Sources:**[tt_smi/constants.py 1-227](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L1-L227)

## Constants Module Structure

The `constants.py` file is organized into two major sections separated by comment blocks: Backend Constants (lines 4-161) and GUI Constants (lines 163-226). This separation ensures a clean boundary between data collection specifications and presentation layer configuration.

**Sources:**[tt_smi/constants.py 4-6](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L4-L6)[tt_smi/constants.py 162-164](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L162-L164)

## Backend Constants

### Telemetry Field Lists

The module defines two primary telemetry lists corresponding to different hardware platforms and their telemetry collection mechanisms.

#### SMBUS_TELEMETRY_LIST

`SMBUS_TELEMETRY_LIST` contains 54 telemetry fields for Wormhole platforms that use SMBus-based telemetry collection. This list defines the complete set of metrics available from the hardware monitoring system.

| Category | Field Examples | Purpose |
| --- | --- | --- |
| Board Identification | `BOARD_ID`, `BOARD_ID_HIGH`, `BOARD_ID_LOW`, `DEVICE_ID` | Unique board identification |
| Firmware Versions | `ARC0_FW_VERSION` through `ARC3_FW_VERSION`, `ETH_FW_VERSION`, `M3_BL_FW_VERSION`, `M3_APP_FW_VERSION` | Firmware version tracking |
| Memory Status | `DDR_SPEED`, `DDR_STATUS`, `GDDR_TRAIN_TEMP0`, `GDDR_TRAIN_TEMP1` | DRAM training and status |
| Clock Frequencies | `AICLK`, `AXICLK`, `ARCCLK` | Operating frequencies |
| Power Metrics | `VCORE`, `TDP`, `TDC`, `MVDDQ_POWER`, `INPUT_POWER` | Power consumption and limits |
| Thermal Metrics | `ASIC_TEMPERATURE`, `VREG_TEMPERATURE`, `BOARD_TEMPERATURE`, `ASIC_TMON0`, `ASIC_TMON1` | Temperature monitoring |
| Cooling | `FAN_SPEED`, `FAN_RPM` | Fan control and monitoring |
| Health Status | `ARC0_HEALTH` through `ARC3_HEALTH`, `FAULTS`, `THROTTLER` | System health indicators |
| Limits | `VDD_LIMITS`, `THM_LIMITS`, `BOARD_POWER_LIMIT` | Hardware safety limits |

**Sources:**[tt_smi/constants.py 8-64](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L64)

#### BH_TELEMETRY_LIST

`BH_TELEMETRY_LIST` contains 33 telemetry fields for Blackhole platforms using TAG-based telemetry. All fields are prefixed with `TAG_` to distinguish them from SMBus telemetry fields.

Notable differences from Wormhole telemetry:

*   Includes `L2CPUCLK0` through `L2CPUCLK3` for L2 CPU clock monitoring
*   Uses `TAG_DDR_FW_VERSION` instead of separate training temperature fields
*   Includes `TAG_CM_FW_VERSION` and `TAG_L2CPU_FW_VERSION` specific to Blackhole architecture
*   Contains `TAG_TIMER_HEARTBEAT` and `TAG_TELEM_ENUM_COUNT` for telemetry system monitoring

**Sources:**[tt_smi/constants.py 66-100](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L66-L100)

**Sources:**[tt_smi/constants.py 8-100](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L8-L100)

### Derived Field Lists

Four derived lists extract specific subsets of data for different purposes. These lists define which fields the backend should expose through specific interfaces.

#### TELEM_LIST

`TELEM_LIST` (lines 102-110) defines the 7 primary telemetry metrics displayed in the real-time telemetry tab:

*   `voltage` - Core voltage (mapped from `VCORE`)
*   `current` - Core current (calculated from power/voltage)
*   `aiclk` - AI clock frequency
*   `power` - Core power consumption
*   `asic_temperature` - ASIC die temperature
*   `fan_speed` - Cooling fan speed percentage
*   `heartbeat` - Firmware heartbeat counter

**Sources:**[tt_smi/constants.py 102-110](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L102-L110)

#### LIMITS

`LIMITS` (lines 112-121) defines the 8 hardware limit fields:

*   `vdd_min`, `vdd_max` - Voltage limits
*   `tdp_limit` - Thermal Design Power limit
*   `tdc_limit` - Thermal Design Current limit
*   `asic_fmax` - Maximum ASIC frequency
*   `therm_trip_l1_limit` - Level 1 thermal trip threshold
*   `thm_limit` - Thermal limit
*   `bus_peak_limit` - Bus peak power limit

**Sources:**[tt_smi/constants.py 112-121](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L112-L121)

#### FW_LIST

`FW_LIST` (lines 123-131) defines the 7 firmware version fields displayed in the Firmware Version tab:

*   `fw_bundle_version` - Complete firmware bundle version
*   `tt_flash_version` - TT Flash firmware version
*   `cm_fw` - Cluster Manager firmware version
*   `cm_fw_date` - Cluster Manager firmware build date
*   `eth_fw` - Ethernet firmware version
*   `bm_bl_fw` - Board Manager bootloader version
*   `bm_app_fw` - Board Manager application version

**Sources:**[tt_smi/constants.py 123-131](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L123-L131)

#### DEV_INFO_LIST

`DEV_INFO_LIST` (lines 133-142) defines the 8 device information fields displayed in the Device Information tab:

*   `bus_id` - PCI bus identifier
*   `board_type` - Board type string (e.g., "n150", "p150")
*   `board_id` - Unique board identifier
*   `coords` - Chip coordinates in multi-chip systems
*   `dram_status` - DRAM training status (boolean)
*   `dram_speed` - DRAM operating speed
*   `pcie_speed` - PCIe link speed generation
*   `pcie_width` - PCIe link width (lanes)

**Sources:**[tt_smi/constants.py 133-142](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L133-L142)

### PCI Properties Configuration

`PCI_PROPERTIES` (lines 144-149 and duplicated at lines 200-205) defines the four PCIe link properties collected from sysfs:

*   `current_link_speed` - Current negotiated link speed (GT/s)
*   `max_link_speed` - Maximum supported link speed (GT/s)
*   `current_link_width` - Current negotiated link width (lanes)
*   `max_link_width` - Maximum supported link width (lanes)

Two hardware constants define PCIe limits:

*   `MAX_PCIE_WIDTH = 16` - Maximum supported PCIe lane width
*   `MAX_PCIE_SPEED = 4` - Maximum PCIe generation (Gen 4)

## Platform-Specific Configuration

### Galaxy Board Types

`GLX_BOARD_TYPES` (line 151) defines the two supported Galaxy system board type strings:

*   `"tt-galaxy-wh"` - Galaxy Wormhole systems
*   `"tt-galaxy-bh"` - Galaxy Blackhole systems

These strings are used throughout the codebase to identify Galaxy systems and trigger Galaxy-specific reset and configuration logic.

**Sources:**[tt_smi/constants.py 151](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L151-L151)

### Galaxy UBB Bus ID Mappings

Galaxy 6U systems use unique bus ID mappings for each tray. Two dictionaries map tray numbers (1-4) to UBB (Universal Base Board) bus IDs, with different mappings for Wormhole and Blackhole platforms.

#### WH_UBB_BUS_IDS

`WH_UBB_BUS_IDS = {1: 0xC0, 2: 0x80, 3: 0x00, 4: 0x40}`
Maps Wormhole Galaxy tray numbers to bus IDs:

*   Tray 1 → Bus 0xC0
*   Tray 2 → Bus 0x80
*   Tray 3 → Bus 0x00
*   Tray 4 → Bus 0x40

**Sources:**[tt_smi/constants.py 154](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L154-L154)

#### BH_UBB_BUS_IDS

`BH_UBB_BUS_IDS = {1: 0x00, 2: 0x40, 3: 0xC0, 4: 0x80}`
Maps Blackhole Galaxy tray numbers to bus IDs:

*   Tray 1 → Bus 0x00
*   Tray 2 → Bus 0x40
*   Tray 3 → Bus 0xC0
*   Tray 4 → Bus 0x80

**Sources:**[tt_smi/constants.py 155](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L155-L155)

## Hardware Limits and Magic Values

Three constants define hardware-specific magic values and limits used in firmware communication and validation.

### Magic Firmware Values

*   `MAGIC_FW_VERSION = 0x01030000` (line 160) - Magic value used in firmware version validation
*   `MSG_TYPE_FW_VERSION = 0xB9` (line 161) - Message type identifier for firmware version requests

These values are used in low-level firmware communication protocols to identify specific message types and validate firmware data structures.

**Sources:**[tt_smi/constants.py 160-161](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L160-L161)

### GUI Update Interval

*   `GUI_INTERVAL_TIME = 0.1` (line 159) - Telemetry update interval in seconds (100ms)

This constant controls the refresh rate of the telemetry display in the TUI. The telemetry thread uses this value to determine how frequently to poll hardware sensors and update the display.

**Sources:**[tt_smi/constants.py 159](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L159-L159)

`HELP_MENU_MARKDOWN` (lines 209-226) contains a multi-line string with Markdown-formatted help documentation displayed when the user presses 'h' in the TUI. The content includes:

## Configuration System Design Patterns

The constants system follows several design patterns that promote maintainability and flexibility:

### Separation of Concerns

Backend constants (telemetry lists, platform mappings) are completely separated from GUI constants (table headers, help text). This allows backend changes without affecting the presentation layer.

### Platform Abstraction

Platform-specific configurations (SMBus vs TAG telemetry, Wormhole vs Blackhole UBB mappings) are isolated in dedicated constants, enabling easy addition of new platforms.

### Derived Lists Pattern

General telemetry lists (54 SMBus fields, 33 TAG fields) are reduced to smaller derived lists (7 TELEM_LIST fields, 8 LIMITS fields) for specific use cases, avoiding repeated field selection logic throughout the codebase.

### Single Source of Truth

All magic values, limits, and configuration strings are defined once in constants.py and imported throughout the application, preventing inconsistencies and simplifying updates.
