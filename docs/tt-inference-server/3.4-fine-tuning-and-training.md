The fine-tuning and training subsystem provides a robust infrastructure for executing Parameter-Efficient Fine-Tuning (PEFT) jobs, specifically LoRA, on Tenstorrent hardware. It utilizes a job-based architecture to manage long-running training tasks asynchronously, providing real-time metrics, logging, and checkpoint management through OpenAI-compatible endpoints.

## System Architecture

The training workflow is orchestrated by the `TrainingService`, which interacts with the `JobManager` to handle lifecycle states and persistence. Hardware execution is delegated to specialized `Forge` training runners that leverage `torch_xla` and `tt-forge-fe` to compile and run PyTorch models on Tenstorrent devices.

## Core Components

### TrainingService

The `TrainingService` (extending `BaseJobService`) acts as the high-level orchestrator for training tasks. It generates output paths for model adapters, initializes shared memory structures (using `multiprocessing.Manager`) for cross-process communication of logs and metrics, and maps the generic `TrainingRequest` to specific model runners.

*   **Job Creation:** It generates a unique task ID and creates a directory in `TRAINING_STORE_ADAPTERS_DIR` for checkpoints [tt-media-server/model_services/training_service.py 32-34](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L32-L34)
*   **Shared State:** It initializes `Manager.list()` for metrics, logs, and checkpoints to allow the training process (often a separate worker) to push updates back to the API layer [tt-media-server/model_services/training_service.py 39-41](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L39-L41)
*   **Metric Retrieval:** Implements `get_job_metrics` with an `after` parameter to allow the frontend to poll only new data points [tt-media-server/model_services/training_service.py 58-64](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L58-L64)

### JobManager and JobDatabase

The `JobManager` handles the lifecycle of all asynchronous tasks. It supports optional persistence via `JobDatabase`, a SQLite implementation.

*   **Job States:** Jobs transition through `QUEUED`, `IN_PROGRESS`, `COMPLETED`, `FAILED`, `CANCELLING`, and `CANCELLED`[tt-media-server/utils/job_manager.py 23-29](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/utils/job_manager.py#L23-L29)
*   **Persistence:** If `enable_job_persistence` is true, the `JobDatabase` stores job metadata, metrics, and logs across server restarts [tt-media-server/utils/job_database.py 44-98](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/utils/job_database.py#L44-L98)
*   **SQLite Schema:** The database includes tables for `jobs`, `metrics`, `checkpoints`, and `logs`, with foreign key constraints linked to the job ID [tt-media-server/utils/job_database.py 45-97](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/utils/job_database.py#L45-L97)

## Training Runners

Training runners implement the actual training loop. They are responsible for model sharding (for multi-chip), dataset loading via `get_dataset_loader`, and the optimization loop.

### Gemma LoRA Runner (`TrainingGemmaLoraRunner`)

Designed for single-chip execution of Gemma 1.1 2B.

*   **Setup:** Uses `AutoModelForCausalLM` and `get_peft_model` to wrap the base model with LoRA layers [tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py 38-118](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py#L38-L118)
*   **Compilation:** Employs `torch.compile` with the `tt` backend and legacy compile options [tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py 124-131](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py#L124-L131)
*   **Data Handling:** Uses `collate_fn_for_causal_lm` to prepare batches with pre-shifted labels for the cross-entropy loss [tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py 86-100](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_gemma_lora_runner.py#L86-L100)

### Llama LoRA Runner (`TrainingLoraRunner`)

Supports multi-chip execution (e.g., N300, T3K) using `torch_xla` SPMD for tensor parallelism.

*   **Sharding:** Defines `MODEL_SHARDING_PATTERNS` using regex to distribute weights across a device mesh (e.g., sharding `q_proj` on the "model" axis) [tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py 131-142](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py#L131-L142)
*   **Custom Loss:** Implements `_cross_entropy_loss` using manual `log_softmax` and one-hot encoding to work around XLA limitations and avoid expensive CCLs on the full vocabulary logits [tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py 41-60](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py#L41-L60)
*   **Scoped Steps:** The `_training_step_inner` function isolates large tensors (like logits) to prevent them from propagating through the computation graph [tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py 63-77](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/tt_model_runners/forge_training_runners/training_lora_runner.py#L63-L77)

## Implementation Details

### Multi-process Logging

To capture logs from the training runner (which may run in a spawned process) and serve them via the API, the system uses a custom `ListHandler`.

1.   The `TrainingService` creates a shared `multiprocessing.Manager().list()`[tt-media-server/model_services/training_service.py 40](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L40-L40)
2.   The `TTLogger` adds a `ListHandler` to this list [tt-media-server/utils/logger.py 143-147](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/utils/logger.py#L143-L147)
3.   The `ListHandler.emit()` method converts standard Python `LogRecord` objects into structured dictionaries: `{"id", "timestamp", "type", "step", "message"}`[tt-media-server/utils/logger.py 71-85](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/utils/logger.py#L71-L85)

### Checkpoint Management

Checkpoints are saved as model adapters (LoRA weights).

*   **Storage:** Saved in subdirectories named by `checkpoint_id` within the job's result path (e.g., `model_store/{job_id}/{checkpoint_id}`) [tt-media-server/model_services/training_service.py 87](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L87-L87)
*   **Verification:**`get_checkpoint_download_path` ensures the checkpoint ID exists in the job's checkpoint list before returning the path [tt-media-server/model_services/training_service.py 78-90](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/model_services/training_service.py#L78-L90)

### Training Request Parameters

The `TrainingRequest` object defines the hyperparameters and hardware configuration for the job:

*   **PEFT:**`lora_r`, `lora_alpha`, and `lora_target_modules` (defaulting to `["q_proj", "v_proj"]`) [tt-media-server/domain/training_request.py 25-27](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/domain/training_request.py#L25-L27)
*   **Optimizer:** Defaults to `ADAMW`[tt-media-server/domain/training_request.py 33](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/domain/training_request.py#L33-L33)
*   **Data:**`dataset_loader` (e.g., `SST2`) and `dataset_max_sequence_length`[tt-media-server/domain/training_request.py 14-15](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/tt-media-server/domain/training_request.py#L14-L15)
