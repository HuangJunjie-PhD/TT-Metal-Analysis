This document describes the V1 Scheduler implementation and the complete lifecycle of requests from arrival to completion. The Scheduler is responsible for orchestrating request execution by managing request queues, allocating KV cache resources, and applying scheduling policies (FCFS or priority-based). For information about the KV cache memory management details, see [Memory Management](https://deepwiki.com/tenstorrent/vllm/2.4-memory-management). For the overall engine architecture context, see [Engine Components](https://deepwiki.com/tenstorrent/vllm/2.2-engine-components).

* * *

## Request Lifecycle States

Requests in vLLM V1 transition through several states as they move from submission to completion. The `RequestStatus` enum defines these states.

### State Definitions

| State | Value | Description |
| --- | --- | --- |
| `WAITING` | 0 | Request is in the waiting queue, not yet scheduled |
| `RUNNING` | 1 | Request is actively being processed, has allocated KV cache blocks |
| `PREEMPTED` | 2 | Request was running but evicted due to resource constraints |
| `FINISHED_ABORTED` | 3 | Request terminated by client abort |
| `FINISHED_STOPPED` | 4 | Request completed normally (stop string/token or max_tokens reached) |
| `FINISHED_LENGTH_CAPPED` | 5 | Request finished due to length constraints |
| `FINISHED_IGNORED` | 6 | Request output ignored due to stop strings |
| `WAITING_FOR_REMOTE_KVS` | 7 | Request waiting for remote KV transfer (P/D disaggregation) |
| `WAITING_FOR_FSM` | 8 | Request waiting for structured output FSM compilation |

## Scheduling Algorithm

The `Scheduler.schedule()` method is invoked once per iteration to determine which requests to execute. The algorithm prioritizes running requests (continuation) before waiting requests (new work).

### High-Level Scheduling Flow

**Sources:**[vllm/v1/core/sched/scheduler.py 180-631](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L180-L631)

### Scheduling Constraints

The scheduler enforces several constraints during scheduling:

| Constraint | Configuration | Purpose |
| --- | --- | --- |
| Maximum running requests | `max_num_seqs` | Limits concurrent requests to prevent OOM |
| Maximum tokens per batch | `max_num_batched_tokens` | Controls batch size for efficient GPU utilization |
| Maximum encoder tokens | `max_num_encoder_input_tokens` | Limits encoder compute for multimodal models |
| Maximum LoRA adapters | `max_loras` | Constrains active LoRA models |
| KV cache blocks | `num_gpu_blocks` | Physical memory limit for KV cache |

## Request Queues and Scheduling Policies

The Scheduler supports two scheduling policies that determine the order in which waiting requests are processed.

### Policy Types

**Sources:**[vllm/v1/core/sched/scheduler.py 116-123](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L116-L123)[vllm/v1/core/sched/request_queue.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/request_queue.py)

### Request Queue Interface

The `waiting` queue is an abstract interface that supports different orderings:

**FCFS (First-Come First-Served):**

*   Requests dequeued in arrival order
*   Simple fairness guarantee
*   Default policy

**Priority-based:**

*   Requests dequeued by `(priority, arrival_time)` tuple
*   Lower priority value = higher priority
*   Ties broken by arrival time
*   Supports urgent requests and batch processing

**Common Operations:**

**Sources:**[vllm/v1/core/sched/request_queue.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/request_queue.py)[vllm/v1/core/sched/scheduler.py 116-127](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L116-L127)

## KV Cache Integration

The Scheduler works closely with the `KVCacheManager` to allocate and free memory blocks for request execution. This integration is critical for both performance (prefix caching) and correctness (preventing OOM).

### Allocation Flow

**Sources:**[vllm/v1/core/sched/scheduler.py 382-541](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L382-L541)[vllm/v1/core/kv_cache_manager.py 154-312](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_manager.py#L154-L312)

### Block Allocation Details

When allocating KV cache slots for a request, several factors are considered:

1.   **Prefix Cache Hits:** Reuse cached blocks from matching prefixes
2.   **External Cache Hits:** For P/D disaggregation, check remote cache
3.   **New Allocation:** Allocate blocks for uncached tokens
4.   **Sliding Window:** Free blocks outside attention window
5.   **Lookahead Tokens:** Pre-allocate for speculative decoding

**Block Hash Calculation:**

Blocks are hashed for prefix caching using a chain of hashes:

*   First block: `hash(NONE_HASH + tokens + cache_salt + mm_hashes)`
*   Subsequent blocks: `hash(parent_block_hash + tokens)`

This enables efficient cache lookup and sharing across requests with common prefixes.

**Sources:**[vllm/v1/core/kv_cache_manager.py 201-312](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_manager.py#L201-L312)[vllm/v1/core/kv_cache_utils.py 528-623](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L528-L623)

## Preemption

When the scheduler cannot allocate sufficient KV cache blocks for a running request, it must preempt (evict) other requests to free memory. Preemption is a critical mechanism for handling resource contention.

### Preemption Algorithm

**Sources:**[vllm/v1/core/sched/scheduler.py 256-293](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L256-L293)

### Preemption Behavior

**Priority Policy:**

*   Preempts request with **highest**`(priority, arrival_time)` value
*   Allows high-priority requests to evict low-priority ones
*   Within same priority, preempts oldest request

**FCFS Policy:**

*   Preempts the **last** request in the running queue (most recently added)
*   Effectively implements a stack discipline for resource contention
*   The current request may preempt itself if no other victim exists

**State After Preemption:**

*   Request status set to `PREEMPTED`
*   `num_computed_tokens` reset to 0 (must recompute from scratch)
*   `num_preemptions` counter incremented
*   Request moved to front of waiting queue (prepended)
*   All allocated KV cache blocks freed and returned to pool

**Sources:**[vllm/v1/core/sched/scheduler.py 268-290](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L268-L290)[vllm/v1/request.py 58-89](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/request.py#L58-L89)

## Advanced Scheduling Features

### Chunked Prefill

Chunked prefill allows breaking long prompt processing into multiple iterations to prevent single-request dominance of batch compute.

**Configuration:**

*   Enabled via `chunked_prefill_enabled` in `SchedulerConfig`
*   Chunk size controlled by `long_prefill_token_threshold`

**Behavior:**

*   When a request has more than `long_prefill_token_threshold` tokens to process, only that many are scheduled in one iteration
*   Remaining tokens scheduled in subsequent iterations
*   Allows interleaving decode of other requests with long prefill

**Implementation:**[vllm/v1/core/sched/scheduler.py 217-430](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L217-L430)

### Encoder Input Scheduling

For multimodal and encoder-decoder models, encoder inputs are scheduled separately with their own budget.

**Budget Tracking:**

*   `max_num_encoder_input_tokens` defines the encoder compute budget per iteration
*   Encoder cache manager tracks which encoder inputs have been processed
*   Encoder inputs freed when no longer needed (past the output attention window)

**Scheduling Logic:**[vllm/v1/core/sched/scheduler.py 230-452](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L230-L452)

### Speculative Decoding Integration

The scheduler supports speculative decoding where a draft model proposes tokens that are verified by the target model.

**Draft Token Handling:**

*   Draft tokens stored in `Request.spec_token_ids`
*   Scheduler allocates KV cache for both draft and target tokens
*   After verification, rejected draft tokens removed and state adjusted

**Update Logic:**[vllm/v1/core/sched/scheduler.py 306-314](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L306-L314)[vllm/v1/engine/core.py 294-299](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py#L294-L299)

### KV Transfer (Disaggregated Serving)

For prefill/decode disaggregation, requests may need to load KV cache from remote workers.

**States:**

*   `WAITING_FOR_REMOTE_KVS`: Request allocated memory but waiting for async KV receive
*   Connector tracks transfer progress and completion
*   Once complete, request transitions to `WAITING` and can be scheduled normally

**Flow:**[vllm/v1/core/sched/scheduler.py 346-421](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/sched/scheduler.py#L346-L421)
