This document describes the physical hardware test infrastructure used to validate tt-system-firmware on real Blackhole and Wormhole boards. This infrastructure executes smoke tests, stress tests, and TT-Metal integration tests on self-hosted GitHub Actions runners with JTAG access to hardware.

For information about unit testing with Zephyr Twister, see [Unit Testing with Twister](https://deepwiki.com/tenstorrent/tt-system-firmware/7.1-unit-testing-with-twister). For details on the test content and validation logic, see [Hardware Smoke Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.2-hardware-smoke-tests), [End-to-End Testing with pyluwen](https://deepwiki.com/tenstorrent/tt-system-firmware/7.3-end-to-end-testing-with-pyluwen), and [Stress and Long-Duration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.4-stress-and-long-duration-tests).

* * *

## Overview

The hardware test infrastructure consists of:

*   **Self-hosted GitHub Actions runners** attached to physical boards via JTAG
*   **Centralized board configuration** in `ci_boards.json` that defines test matrices
*   **Docker-based test containers** with Zephyr SDK, tt-kmd drivers, and Python test tools
*   **Multiple test workflows** for smoke tests, stress tests, and Metal integration

All hardware tests run on actual Tenstorrent boards, not simulators, providing validation of the complete firmware stack including SMC, DMC, and chip management firmware.

## Physical Test Runners and Board Types

### Runner Labels and Board Mapping

The infrastructure uses GitHub Actions self-hosted runners with specific labels that identify the board type and capabilities:

| Board Type | Runner Label | Product | Key Features |
| --- | --- | --- | --- |
| P100A | `p100a-jtag` | Blackhole | Single-chip, JTAG access |
| P150A | `p150a-jtag` | Blackhole | Single-chip, JTAG access |
| P300A | `p300a-jtag` | Blackhole | Dual-chip, JTAG access |
| Galaxy | `tt-galaxy-bh` | Blackhole | 32-chip system, JTAG access |
| N150 | `n150-jtag` | Wormhole | Single-chip, JTAG access |
| N300 | `n300-jtag` | Wormhole | Dual-chip, JTAG access |

**Sources:**[.github/ci_boards.json 1-46](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/ci_boards.json#L1-L46)

### Hardware Requirements

Each runner must provide:

*   **JTAG interface** via `/dev/bus/usb` for firmware flashing and debugging
*   **Tenstorrent device** accessible via `/dev/tenstorrent` and tt-kmd driver
*   **IPMI interface** via `/dev/ipmi0` for power management (Blackhole boards)
*   **Hugepages** mounted at `/dev/hugepages-1G` for PCIe DMA operations
*   **tt-kmd kernel module** builds at `/opt/tenstorrent/kmd-builds/`

**Sources:**[.github/workflows/hardware-smoke.yml 63](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml#L63-L63)[.github/workflows/hardware-long.yml 57](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-long.yml#L57-L57)

**Key Actions:**

1.   **Git Configuration** - Workaround for container UID mismatch
2.   **West Manifest Cleanup** - Remove stale repository state

5.   **Repository Update** - `west update --path-cache /tt-zephyr`
6.   **Python Package Installation** - Install pyluwen, tt-flash, tt-tools-common
7.   **Binary Blob Verification** - `west blobs fetch tt-system-firmware`

## Python Test Dependencies

The hardware tests rely on several Python packages installed in the container:

| Package | Version | Purpose |
| --- | --- | --- |
| `pyluwen` | >=0.8.1 | Hardware abstraction, chip detection, telemetry |
| `tt-flash` | ==3.6.2 | Firmware flashing and bundle management |
| `tt-smi` | >=3.0.23, <4.0.0 | System management interface |
| `tt-tools-common` | >=1.4.31 | Common utilities |
| `tt-burnin` | >=0.3.0 | Burn-in test utilities |
| `pytest-repeat` | - | Test repetition for stress tests |

**Sources:**[scripts/requirements.txt 1-21](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/requirements.txt#L1-L21)

### Version Constraints

The `tt-smi` package is restricted to versions below 4.0.0 to avoid issues with long reset times due to Ethernet training timeouts.

**Sources:**[scripts/requirements.txt 18-20](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/scripts/requirements.txt#L18-L20)

## Summary

The hardware test infrastructure provides comprehensive validation of tt-system-firmware on real hardware through:

*   **Self-hosted runners** with direct JTAG access to 6 different board configurations
*   **Centralized board configuration** enabling consistent test matrix generation
*   **Docker containers** providing reproducible build and test environments
*   **Multiple test workflows** covering smoke tests, stress tests, and ML integration
*   **Extensive artifact collection** for debugging and analysis

This infrastructure ensures that firmware changes are validated on actual hardware before deployment, catching issues that cannot be detected through unit tests or simulation alone.
