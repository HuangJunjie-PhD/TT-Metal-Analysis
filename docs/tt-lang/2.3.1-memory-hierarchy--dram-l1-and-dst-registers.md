## Purpose and Scope

This page describes the three-tier memory hierarchy used by Tenstorrent hardware and how `tt-lang` programs interact with each level: **DRAM** (off-chip persistent storage), **L1** (on-chip cache), and **DST registers** (compute destination registers). It covers memory configurations, data movement patterns between levels, and the register lifecycle for compute operations.

For information about Circular Buffers (which reside in L1), see [2.2.1](https://github.com/tenstorrent/tt-lang/blob/d76e6233/2.2.1) For data movement operations and copy patterns, see [2.3.2](https://github.com/tenstorrent/tt-lang/blob/d76e6233/2.3.2)

* * *

## Memory Hierarchy Overview

Tenstorrent accelerators use a three-level memory hierarchy where data flows through each level during kernel execution. `tt-lang` operates on chunks of tensors called **blocks**, and the compiler optimizes performance by partitioning computations into subblocks that fit within the various memory tiers.

## DRAM: Off-Chip Persistent Storage

DRAM is the primary storage for large tensor data. Tensors can be placed in DRAM and accessed by data movement threads (BRISC/NCRISC).

### Memory Configuration

Tensors are configured for DRAM placement using layout attributes. The `TTLLayoutTensor` type in the MLIR dialect represents these tensors, which include buffer type (DRAM/L1) and memory layout (Interleaved/Sharded) [include/ttlang/Dialect/TTL/IR/TTLOps.td 141-143](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L141-L143)

### DRAM Access in Kernels

Data movement kernels interact with DRAM via `ttl.copy`. This operation initiates an asynchronous transfer between a `tensor_slice` (representing a view into DRAM) and a `!ttl.cb` (representing a block in L1) [include/ttlang/Dialect/TTL/IR/TTLOps.td 122-136](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L122-L136) The transfer returns a `!ttl.transfer_handle` which must be synchronized using `ttl.wait`[include/ttlang/Dialect/TTL/IR/TTLOps.td 164-168](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L164-L168)

## L1: On-Chip Cache

L1 is on-chip memory local to each Tensix core. It serves as the storage for **Dataflow Buffers** (DFBs), also known as Circular Buffers (CBs).

### Dataflow Buffers (DFB)

DFBs are managed using the `ttl.bind_cb` operation, which binds a hardware circular buffer slot (0-31) to a logical handle [include/ttlang/Dialect/TTL/IR/TTLOps.td 26-35](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L26-L35)

*   **Capacity:** The hardware supports up to 32 circular buffers [include/ttlang/Dialect/TTL/IR/TTL.h 25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L25-L25)
*   **Double Buffering:** The `block_count` attribute (default 2) specifies the number of blocks to allocate, enabling double-buffering patterns where one block is computed while another is being transferred [include/ttlang/Dialect/TTL/IR/TTLOps.td 46-47](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L46-L47)
*   **Association:** The `ttl.attach_cb` operation associates a tensor SSA value with a CB handle, allowing the compiler to trace data provenance during lowering [include/ttlang/Dialect/TTL/IR/TTLOps.td 53-62](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L53-L62)

### Type Conversion

During the compilation pipeline, `!ttl.cb` types are lowered to `ttkernel::CBType`, which flattens the multidimensional tile shape into a total element count for the hardware API [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 70-73](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L70-L73)

## DST Registers: Compute Destination

DST (Destination) registers are specialized hardware registers where the MATH thread performs tile computations.

### DST Capacity and Subblocking

The hardware DST capacity is a critical constraint. To maximize performance, `tt-lang` partitions large compute operations into subblocks that fit within the available registers [docs/development/DST_Utilization.md 5-11](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Utilization.md?plain=1#L5-L11)

**Capacity Calculation:** The `unroll_factor` (subblock size) is calculated by comparing the total DST capacity $C$ against the register pressure $D$ of the inner compute loop [docs/development/DST_Utilization.md 100-108](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Utilization.md?plain=1#L100-L108)

*   **Blackhole/Wormhole:** Capacity is typically 16 or 32 tiles depending on the target architecture and data format [include/ttlang/Dialect/TTL/IR/TTL.h 47-49](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTL.h#L47-L49)
*   **Matmul Budget:** For matrix multiplication, only the parallel dimensions ($M \times N$) occupy DST slots; the reduction dimension ($K$) accumulates in-place [lib/Dialect/TTL/Transforms/TTLSubblockComputeForDST.cpp 145-154](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLSubblockComputeForDST.cpp#L145-L154)

## Register Allocation and Accumulation

The `TTLAssignDST` pass performs interval-based linear scan allocation to map logical tile results to physical DST indices [lib/Dialect/TTL/Transforms/TTLAssignDST.cpp 9-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLAssignDST.cpp#L9-L12)

### In-Place Merging

For unary operations, the allocator merges the input and output intervals into a single equivalence class. This ensures the operation is performed in-place on the same DST register, saving capacity [lib/Dialect/TTL/Transforms/TTLAssignDST.cpp 19-23](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLAssignDST.cpp#L19-L23)

### Accumulation Patterns

*   **Matmul Accumulation:** Multiple `matmul_tiles` calls to the same DST index accumulate results across the $K$ dimension. The DST footprint remains live for the entire $K$ loop, and `tile_regs_acquire` implicitly zeros the registers before accumulation starts [docs/development/DST_Allocation.md 89-101](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/development/DST_Allocation.md?plain=1#L89-L101)
*   **Copy Insertion:** If a value is needed by multiple consumers and one of them is an in-place operation, the pass inserts `ttl.copy_dst` to preserve the original value before it is clobbered [lib/Dialect/TTL/Transforms/TTLAssignDST.cpp 14-17](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLAssignDST.cpp#L14-L17)
