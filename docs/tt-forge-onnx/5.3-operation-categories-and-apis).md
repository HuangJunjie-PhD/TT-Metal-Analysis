This document catalogs the operation categories available in the Forge compiler and their Python APIs. Forge provides 80+ operations organized into six main categories: eltwise unary, eltwise binary, eltwise n-ary, tensor manipulation, neural network layers, and reductions. Each operation has a Python API function that constructs computation graph nodes through the `ForgeOp` wrapper system.

For information about how operations are implemented in C++ and dispatched, see [Operation Implementation Guide](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.2-operation-implementation-guide). For the overall operation system architecture, see [Operation Architecture](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.1-operation-architecture).

| `Squeeze` | `forge.op.Squeeze` | Remove dimension of size 1 |
| `Unsqueeze` | `forge.op.Unsqueeze` | Add dimension of size 1 |

## Operation API Patterns

All Forge operations follow consistent patterns in their Python API design.

**API Construction Flow**

**Common API Pattern**

Every operation follows this structure:

`def OperationName(name: str, operand: Tensor, **kwargs) -> Tensor:    """    Docstring with operation description.        Parameters    ----------    name : str        Op name, unique to the module, or leave blank to autoset    operand : Tensor        Input tensor    **kwargs : additional operation-specific parameters        Returns    -------    Tensor        Forge tensor (computation graph node)    """    return op(OpType.OperationName, name, operand, **kwargs).get_tensor()`
**Parameter Conventions**

1.   **name (str)**: Operation identifier in the graph. Empty string auto-generates name.
2.   **operand(s) (Tensor)**: Input tensors, positional arguments before attributes.
3.   **Attributes (**kwargs)**: Operation-specific configuration passed as keyword arguments.
4.   **Return (Tensor)**: Always returns a Tensor representing the output node.

**Attribute Handling**

Attributes are passed through the `ForgeOp` constructor and stored in the C++ `Op::attrs_` map:

`# Example: Conv2d with multiple attributesop(OpType.Conv2d, name, act, weights,    stride=[1, 1],    padding=[0, 0, 0, 0],   dilation=1,   groups=1,   channel_last=False).get_tensor()`
The C++ backend retrieves attributes using type-safe accessors:

`// From forge/csrc/ops/op.hpp:166-175template <typename T>const T& attr_as(std::string const& name) const {    return std::get<T>(attr(name));}`
**Dimension Indexing Conventions**

All operations support both positive and negative dimension indexing:

*   **Positive**: `dim=0` is first dimension (batch typically)
*   **Negative**: `dim=-1` is last dimension, `dim=-2` is second-to-last

Most APIs normalize to negative indexing internally:

`# From forge/forge/op/tm.py:32-37dims = len(operandA.shape.dims)if dim0 >= 0:    dim0 -= dimsif dim1 >= 0:    dim1 -= dims`

`// From forge/csrc/ops/op.cpp:37-131class OpTypeToString {    mapping_[OpType::Add] = "add";    mapping_[OpType::Matmul] = "matmul";    // ... 80+ mappings}; class StringToOpType {    mapping_["add"] = OpType::Add;    mapping_["matmul"] = OpType::Matmul;    // ... 80+ mappings};`
This enables:

*   Python string names → C++ enum dispatch
*   C++ enum → Python string for debugging/serialization

## Summary

Forge provides 80+ operations across six main categories, each with consistent Python APIs that construct computation graph nodes. The operation system uses:

*   **Python API Layer**: Category-specific modules in `forge/forge/op/`
*   **OpType Enum**: C++ enumeration in `forge/csrc/ops/op.hpp`
*   **C++ Implementation**: Individual `op_*.cpp` files dispatched by OpType
*   **TVM Integration**: Mapping tables for Relay IR translation

All operations follow the pattern:

`result = forge.op.OperationName(name, *operands, **attributes)`
This returns a `Tensor` object representing a node in the computation graph, which is later lowered through the MLIR backend (see [MLIR Backend and Code Generation](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.3-mlir-backend-and-code-generation)).
