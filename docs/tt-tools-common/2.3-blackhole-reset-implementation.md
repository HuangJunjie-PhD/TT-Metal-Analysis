## Purpose and Scope

This document details the Blackhole-specific chip reset implementation provided by the `BHChipReset` class. It covers the legacy IOCTL-based reset sequence, the M3 reset option, and the delegation pattern that routes to the modern unified reset system when available.

For the modern unified reset implementation used by both Blackhole and Wormhole when driver version >= 2.4.1, see [ChipReset: Modern Unified Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.2-chipreset:-modern-unified-reset). For reset architecture patterns and driver version requirements, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns). For reset configuration and logging, see [Reset Configuration and Logging](https://deepwiki.com/tenstorrent/tt-tools-common/2.6-reset-configuration-and-logging).

## Architecture Overview

The Blackhole reset system implements a **delegation pattern** that conditionally routes to different implementations based on driver version and virtualization environment. This enables gradual migration from the legacy IOCTL-based reset to the modern unified reset system while maintaining backward compatibility.

### Reset Implementation Selection Flow

**Sources:**[tt_tools_common/reset_common/bh_reset.py 68-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L68-L99)[tt_tools_common/reset_common/bh_reset.py 109-118](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L109-L118)

### Version Requirements

| Reset Implementation | Minimum Driver Version | Platform Restrictions |
| --- | --- | --- |
| Modern Unified (`ChipReset`) | 2.4.1 | No ARM support |
| Legacy IOCTL | 1.26.0 | No ARM support |
| Xen HVM Mode | 2.4.1 (uses `ChipReset`) | x86_64 only |

## Legacy IOCTL Reset Implementation

The legacy reset implementation directly communicates with the kernel driver via IOCTL commands. This path is used when driver version < 2.4.1 and not running under Xen HVM.

### IOCTL Interface

The `reset_device_ioctl()` method constructs a binary buffer matching the kernel driver's expected structure:

**Sources:**[tt_tools_common/reset_common/bh_reset.py 42-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L42-L66)

### Config Space Reset Sequence

The legacy reset monitors the PCIe config space to verify reset completion:

**Sources:**[tt_tools_common/reset_common/bh_reset.py 127-223](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L127-L223)

### Config Space Monitoring Details

The reset completion is detected by monitoring the **command.memory** register in PCIe config space:

| File Path | Offset | Bit | Meaning |
| --- | --- | --- | --- |
| `/sys/bus/pci/devices/{bdf}/config` | 4 bytes | bit 1 | 0 = reset complete, 1 = reset in progress |

The implementation uses `os.pread()` to read the byte at offset 4 without seeking, enabling efficient polling:

**Sources:**[tt_tools_common/reset_common/bh_reset.py 152-192](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L152-L192)

## M3 Reset Option

The M3 reset option was added in version 1.4.7 to enable resetting the M3 processor and DMC (DDR Memory Controller) along with the ASIC. This is useful for firmware upgrade scenarios or recovering from M3-related issues.

### M3 Reset Flow

**Sources:**[tt_tools_common/reset_common/bh_reset.py 137-144](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L137-L144)[CHANGELOG.md 55-60](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L55-L60)

### M3 Reset Timing Considerations

When `reset_m3=True`, the implementation adjusts timing parameters:

| Parameter | Standard Reset | M3 Reset | Reason |
| --- | --- | --- | --- |
| `post_reset_wait` | 2 seconds | `max(60, m3_delay)` | BMFW may self-upgrade during M3 reset |
| Early exit | Enabled after bit transition | Requires bit transition to 1 first | M3 reset has delayed effect |

The `can_early_exit` flag prevents premature completion detection when the ASIC takes time to enter reset state during M3 firmware operations.

**Sources:**[tt_tools_common/reset_common/bh_reset.py 129-192](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L129-L192)[CHANGELOG.md 48-51](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L48-L51)

## Migration to Modern Reset

Starting with driver version 2.4.1, the `BHChipReset` class delegates to the modern `ChipReset` implementation. This provides several advantages:

### Delegation Logic

**Sources:**[tt_tools_common/reset_common/bh_reset.py 90-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L90-L99)

### Modern Reset Advantages

When using `ChipReset`, Blackhole devices benefit from:

1.   **Secondary Bus Reset (SBR)**: Optional PCIe link-level reset before ASIC reset
2.   **Xen HVM Integration**: Xenstore coordination for host-guest PCIe hot-plug
3.   **Unified IOCTL Interface**: New `IoctlResetFlags` enum with more reset types
4.   **Improved Device Detection**: Waits for device to reappear on PCIe bus after hot-plug
5.   **Post-Reset Validation**: `POST_RESET` IOCTL flag for final validation

**Sources:**[tt_tools_common/reset_common/chip_reset.py 114-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L114-L288)

### Parameter Forwarding

All `full_lds_reset()` parameters are forwarded to `ChipReset`:

| Parameter | Type | Description | Default |
| --- | --- | --- | --- |
| `pci_interfaces` | `List[int]` | PCI interface IDs to reset | Required |
| `reset_m3` | `bool` | Enable M3/DMC reset | `False` |
| `silent` | `bool` | Suppress console output | `False` |
| `m3_delay` | `int` | Seconds to wait after M3 reset | `20` |
| `secondary_bus_reset` | `bool` | Enable SBR before ASIC reset | `True` |

When `reset_m3=True`, the modern implementation uses `IoctlResetFlags.ASIC_DMC_RESET` instead of `IoctlResetFlags.ASIC_RESET`.

**Sources:**[tt_tools_common/reset_common/bh_reset.py 68-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L68-L99)[tt_tools_common/reset_common/chip_reset.py 206-212](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L206-L212)

## Platform Restrictions

Both legacy and modern implementations enforce platform restrictions:

### ARM Platform Detection

ARM systems do not support PCIe device rescans after reset, making device recovery impossible. The implementation checks this early and exits with a clear error message directing users to reboot the system instead.

**Sources:**[tt_tools_common/reset_common/bh_reset.py 109-118](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L109-L118)[tt_tools_common/reset_common/chip_reset.py 139-148](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L139-L148)

## Error Handling

The implementation handles several error conditions:

| Error Condition | Detection | Handling |
| --- | --- | --- |
| ARM platform | `platform.startswith()` check | Exit with error message |
| Driver too old (< 1.26) | Implicit in ChipReset delegation | Falls back to legacy if < 2.4.1 |
| Device not found | `PciChip()` constructor failure | Exception propagated to caller |
| IOCTL failure | `fcntl.ioctl()` exception | OSError propagated to caller |
| Reset timeout | Config space polling loop | Marks device as failed, exits with count |
| Config space read error | `os.pread()` failure | Exception propagated to caller |

When reset completion fails, the implementation counts failures and exits with the failure count as the exit code:

**Sources:**[tt_tools_common/reset_common/bh_reset.py 150-213](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L150-L213)
