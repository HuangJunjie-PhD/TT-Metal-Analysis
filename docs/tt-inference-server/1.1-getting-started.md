This guide provides a step-by-step walkthrough for setting up the Tenstorrent (TT) Inference Server, installing prerequisites, and running your first model. The system is designed to orchestrate complex AI workloads on Tenstorrent hardware using either a host-side CLI (`run.py`) or direct Docker container execution.

## 1. Overview and Prerequisites

The `tt-inference-server` supports two primary execution paths:

1.   **Host-side Orchestration (`run.py`)**: Automates environment validation, model weight management, and Docker lifecycle [docs/workflows_user_guide.md 3-5](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L3-L5)
2.   **Direct Container Interface**: High-performance execution using pre-built images with self-resolving model specifications [docs/workflows_user_guide.md 5-6](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L5-L6)

### Hardware and Software Requirements

*   **Hardware**: Supports N150, N300, T3K, Galaxy, P-series, and BH LoudBox systems [README.md 27-41](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/README.md?plain=1#L27-L41)
*   **OS**: Linux (Ubuntu 20.04+ recommended).
*   **Python**: Version 3.8 or higher is required for the orchestration scripts [docs/workflows_user_guide.md 80](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L80-L80)
*   **Docker**: Required for containerized execution [docs/workflows_user_guide.md 113](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L113-L113)
*   **Drivers**: Latest Tenstorrent kernel module (KMD) and firmware must be installed on the host.

### Hugging Face Requirements

To access gated models (like Llama 3), you must provide an `HF_TOKEN` in your environment or `.env` file and accept terms on the Hugging Face model repository page [docs/workflows_user_guide.md 87-91](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L87-L91)

## 3. Running Your First Model with run.py

The `run.py` script is the primary entry point. It resolves the `ModelSpec` for a given model and hardware target, prepares weights, and launches the server or workflow.

### Basic Execution Command

To run a benchmark on a Llama-3.2-1B model using an N150 device:

`python3 run.py --model Llama-3.2-1B-Instruct --tt-device n150 --workflow benchmarks --docker-server`
[docs/workflows_user_guide.md 35-37](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L35-L37)

### Execution Flow

1.   **Bootstrap**: The script uses `bootstrap_uv` to ensure the `uv` package manager is available for creating isolated virtual environments for benchmarks or evals [run.py 16](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L16-L16)
2.   **Resolution**: `get_runtime_model_spec` matches CLI arguments against the `MODEL_SPECS` registry to find the correct `ModelSpec`[run.py 23-24](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L23-L24)
3.   **Validation**: `validate_setup` checks hardware availability (via PCI indices) and system software compatibility [run.py 48](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L48-L48)
4.   **Weight Management**: `setup_host` and the `HostSetupManager` download weights from Hugging Face or verify local paths [workflows/setup_host.py 39-67](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L39-L67)
5.   **Deployment**: `generate_docker_run_command` constructs the `docker run` call with device mappings (`/dev/tenstorrent`) and volume mounts [workflows/run_docker_server.py 124-176](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L124-L176)

## 4. Docker and Container Interface

For production or CI environments, you can interact directly with the Docker container. The system uses specific images for different engines (e.g., `vllm-tt-metal` for LLMs).

### Direct Docker Run

The container entrypoint resolves model configuration if `--model` and `--tt-device` are provided [docs/workflows_user_guide.md 15-16](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/docs/workflows_user_guide.md?plain=1#L15-L16)

`docker run --rm \  --device /dev/tenstorrent/0:/dev/tenstorrent/0 \  --mount type=bind,src=/dev/hugepages-1G,dst=/dev/hugepages-1G \  -v tt_volume:/home/container_app_user/cache_root \  ghcr.io/tenstorrent/tt-inference-server/vllm-tt-metal:latest \  --model Llama-3.2-1B-Instruct --tt-device n150`
[workflows/run_docker_server.py 153-176](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L153-L176)

### Volume Persistence Strategies

The `SetupConfig` class manages how model weights and caches are persisted [workflows/setup_host.py 39-67](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L39-L67):

*   **Docker Volume**: (Default) Consistent naming via `generate_docker_volume_name` based on implementation ID and model name [workflows/run_docker_server.py 91-96](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_docker_server.py#L91-L96)
*   **Host Volume**: Binds a specific host directory to the container's `cache_root` via `--host-volume`[workflows/setup_host.py 42](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L42-L42)
*   **Read-only Weights**: Mounts existing Hugging Face cache or local weight directories as read-only mounts [workflows/setup_host.py 43-46](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/setup_host.py#L43-L46)

## 5. Local Server Mode (Non-Docker)

For debugging or environments where Docker is unavailable, the server can run directly on the host using `--local-server`[run.py 127-128](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L127-L128)

### Prerequisites for Local Mode

1.   **Pre-built venv**: You must provide a path to a virtual environment that has `tt-metal` and the inference engine (e.g., vLLM) installed via `--tt-metal-python-venv-dir`[run.py 203-206](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L203-L206)
2.   **Environment Variables**: The orchestration logic configures the environment for the subprocess, including `TT_METAL_HOME` and `MODEL`[run.py 36](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L36-L36)[workflows/run_local_server.py 1-50](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/run_local_server.py#L1-L50)

### Command

`python3 run.py --model Llama-3.2-1B-Instruct --workflow benchmarks --local-server --tt-metal-python-venv-dir /path/to/your/venv`
### Key Components in Local Setup

*   `RuntimeConfig`: Holds the resolved paths and configuration for the local execution [run.py 38](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L38-L38)
*   `run_local_server()`: Orchestrates the host-side process launch using the specified venv [run.py 36](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/run.py#L36-L36)
