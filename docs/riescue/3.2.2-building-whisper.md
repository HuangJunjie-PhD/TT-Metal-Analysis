# Building Whisper

Relevant source files
*   [infra/build_spike.sh](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_spike.sh)
*   [infra/build_whisper.sh](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh)

## Purpose and Scope

This document describes how the Whisper Instruction Set Simulator (ISS) is built as part of the RiESCUE development environment. The build process is automated by the `build_whisper.sh` script, which supports both development and public/CI environments. For information about building Spike simulators, see [Building Spike](https://deepwiki.com/tenstorrent/riescue/3.2.3-building-spike). For system-level dependencies required before building Whisper, see [Base Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2.1-base-dependencies).

**Sources:**[infra/build_whisper.sh 1-73](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L1-L73)

## Build Modes

The `build_whisper.sh` script operates in two distinct modes depending on the presence of the Whisper submodule:

### Development Mode

When a `whisper` directory exists at the repository root (as a git submodule), the script builds in **development mode**. This mode is used by internal developers who have access to the Whisper submodule.

**Characteristics:**

*   Uses the local `whisper` submodule directory
*   Cleans previous build artifacts before building
*   Can build directly on the host or inside a Singularity container
*   Binary remains in the `whisper` directory after build

**Sources:**[infra/build_whisper.sh 18-35](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L18-L35)

### Public/CI Mode

When no `whisper` directory is found, the script operates in **public/CI mode**. This mode clones Whisper from the public GitHub repository and builds from a pinned commit.

**Characteristics:**

*   Clones from `https://github.com/tenstorrent/whisper.git`
*   Checks out commit SHA: `b24e30f238d462d3930744cb084d74129d0873a8`
*   Target directory depends on `NO_SINGULARITY` environment variable: 
    *   With `NO_SINGULARITY`: builds in `$HOME/whisper`
    *   Without `NO_SINGULARITY`: builds in `/usr/local/whisper` (container environment)

*   Installs binary to `/usr/bin/whisper`
*   Cleans up the cloned repository after installation

**Sources:**[infra/build_whisper.sh 36-71](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L36-L71)

## Build Configuration

### Compiler and Linker Flags

The build uses a standardized `make` command with specific configuration flags defined at [infra/build_whisper.sh 10](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L10-L10):

`make -j 20 BOOST_LIB_DIR=/usr/lib64/ BOOST_INC=/usr/include/boost/ STATIC_LINK=0 SOFT_FLOAT=1`
| Flag | Value | Purpose |
| --- | --- | --- |
| `-j 20` | 20 parallel jobs | Speeds up compilation using multiple CPU cores |
| `BOOST_LIB_DIR` | `/usr/lib64/` | Location of Boost libraries on Rocky Linux |
| `BOOST_INC` | `/usr/include/boost/` | Location of Boost headers |
| `STATIC_LINK` | `0` | Enables dynamic linking (reduces binary size) |
| `SOFT_FLOAT` | `1` | Uses software floating-point implementation |

**Sources:**[infra/build_whisper.sh 10](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L10-L10)

### Version Pinning

The public build mode pins Whisper to a specific commit to ensure reproducibility:

| Variable | Value |
| --- | --- |
| `whisper_remote` | `https://github.com/tenstorrent/whisper.git` |
| `whisper_sha` | `b24e30f238d462d3930744cb084d74129d0873a8` |

This pinning ensures that all developers and CI environments build the exact same version of Whisper, preventing version-related inconsistencies.

**Sources:**[infra/build_whisper.sh 12-13](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L12-L13)

## Build Process Details

### Development Mode Build Process

The development mode process ([infra/build_whisper.sh 19-35](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L19-L35)):

1.   **Navigate** to the `whisper` submodule directory
2.   **Clean** previous build artifacts: 
    *   Remove `build-Linux` directory
    *   Remove existing `whisper` binary
    *   Run `make clean`

3.   **Build** using either: 
    *   Direct execution on host (if Singularity unavailable or `NO_SINGULARITY` set)
    *   Containerized build via `container-run` script

4.   **Move** the compiled binary from `build-Linux/whisper` to `whisper`

**Sources:**[infra/build_whisper.sh 19-35](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L19-L35)

### Public/CI Mode Build Process

The public/CI mode process ([infra/build_whisper.sh 37-70](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L37-L70)):

1.   **Determine** target directory based on `NO_SINGULARITY` environment variable
2.   **Create** the target directory
3.   **Navigate** to the target directory
4.   **Clean** any existing state (leftover files from previous builds)
5.   **Initialize** a new git repository
6.   **Add** the Whisper remote repository
7.   **Fetch** only the specific commit (shallow clone with depth 1)
8.   **Checkout** the fetched commit
9.   **Build** using the standard build command
10.   **Move** the compiled binary to the root of the build directory
11.   **Install** the binary to `/usr/bin/whisper` for system-wide access
12.   **Clean up** by removing the entire build directory
13.   **Return** to the original directory

**Sources:**[infra/build_whisper.sh 37-70](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L37-L70)

## Container Integration

The build script integrates with the RiESCUE container system to ensure consistent build environments. The integration is controlled by two mechanisms:

### Container Detection

The script checks for Singularity availability using:

`if ! [ -x "$(command -v singularity)" ] || [ "$NO_SINGULARITY" ]; then`
This condition returns true (skips container) if:

*   The `singularity` command is not executable, OR
*   The `NO_SINGULARITY` environment variable is set (used in CI environments)

**Sources:**[infra/build_whisper.sh 26](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L26-L26)

### Container Build Execution

When building in a container, the script uses the `container-run` helper script located at [infra/build_whisper.sh 9](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L9-L9):

`container_run=$script_dir/container-run`
This script executes the build command inside the Singularity container, ensuring that:

*   The correct Boost library paths are available
*   All build dependencies are present
*   The build environment is identical across different host systems

For details on the container system, see [Container System](https://deepwiki.com/tenstorrent/riescue/3.1-container-system).

**Sources:**[infra/build_whisper.sh 9](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L9-L9)[infra/build_whisper.sh 26-32](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L26-L32)

## Installation Locations

The final location of the Whisper binary depends on the build mode:

| Build Mode | Binary Location | Access Scope |
| --- | --- | --- |
| **Development Mode** | `whisper/whisper` (relative to repo root) | Local to repository; requires manual addition to PATH |
| **Public/CI Mode** | `/usr/bin/whisper` | System-wide; automatically in PATH |

### Development Mode Usage

In development mode, the binary remains in the submodule directory. Users typically run it via:

*   Direct path: `./whisper/whisper <args>`
*   Adding to PATH: `export PATH=$PWD/whisper:$PATH`

### Public/CI Mode Usage

In public/CI mode, the binary is installed system-wide and can be invoked directly:

*   Command: `whisper <args>`

The RiESCUE toolchain abstraction layer locates the Whisper binary automatically. For details on how the framework invokes Whisper, see [Whisper Simulator](https://deepwiki.com/tenstorrent/riescue/4.2.1-whisper-simulator).

## Error Handling and Build Safety

The script employs several safety mechanisms to ensure reliable builds:

### Strict Error Handling

`set -eset -o pipefail`
These bash options ([infra/build_whisper.sh 15-16](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L15-L16)):

*   `set -e`: Exit immediately if any command fails
*   `set -o pipefail`: Fail if any command in a pipeline fails (not just the last one)

### Conditional Cleanup

The script uses conditional removal with fallback messages:

`rm -rf build-Linux || echo "build-Linux not found"rm -f whisper || echo "whisper not found"`
This approach ([infra/build_whisper.sh 22-23](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L22-L23)):

*   Prevents the script from failing if files don't exist
*   Provides informative output about what was found
*   Ensures clean state before building

### Clean Repository State

In public/CI mode, the script performs a thorough cleanup:

`rm -rf .git build-Linux whisper || true`
The `|| true` suffix ([infra/build_whisper.sh 54](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L54-L54)) prevents the script from failing if these files don't exist, ensuring the build can proceed even on first run.

**Sources:**[infra/build_whisper.sh 15-16](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L15-L16)[infra/build_whisper.sh 22-23](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L22-L23)[infra/build_whisper.sh 54](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L54-L54)

## Build Dependencies

Whisper requires several system-level dependencies to compile successfully. These are installed by the [Base Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2.1-base-dependencies) build script and include:

| Dependency | Purpose |

*   Libraries: `/usr/lib64/`

These paths are standard on Rocky Linux 9.1, the base OS used in the RiESCUE container. For details on how these dependencies are installed, see [Base Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2.1-base-dependencies).

## Integration with Container System

The Whisper build script is typically invoked during container construction. The container definition file calls this script as part of the `%post` section to ensure Whisper is available in the containerized development environment.

When the container is built:

1.   Base dependencies are installed first ([Base Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2.1-base-dependencies))
2.   `build_whisper.sh` is executed (this script)
3.   Whisper binary is installed to `/usr/bin/whisper`
4.   The binary is available for all future container runs

For details on container construction and the order of build operations, see [Container Definition and Build](https://deepwiki.com/tenstorrent/riescue/3.1.1-container-definition-and-build).

**Sources:**[infra/build_whisper.sh 1-73](https://github.com/tenstorrent/riescue/blob/07cefde3/infra/build_whisper.sh#L1-L73)

## Troubleshooting Common Build Issues

### Issue: Boost Libraries Not Found

**Symptoms:**

```
error: boost/... No such file or directory
```

**Solution:** Ensure base dependencies are installed. Run [Base Dependencies](https://deepwiki.com/tenstorrent/riescue/3.2.1-base-dependencies) build script before building Whisper.

### Issue: Make Job Fails with Memory Error

**Symptoms:**

```
c++: fatal error: Killed signal terminated program
```

**Solution:** The `-j 20` flag may exceed available memory. Manually run with fewer jobs:

`make -j 4 BOOST_LIB_DIR=/usr/lib64/ BOOST_INC=/usr/include/boost/ STATIC_LINK=0 SOFT_FLOAT=1`
### Issue: Permission Denied Installing to /usr/bin

**Symptoms:**

```
cp: cannot create regular file '/usr/bin/whisper': Permission denied
```

**Solution:** This should only occur in public/CI mode. Ensure the script is run with sufficient privileges (typically inside a container being built with root permissions).
