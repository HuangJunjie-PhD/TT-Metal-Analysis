## Purpose and Scope

This document describes the `ChipReset` class, the modern unified reset implementation for Blackhole and Wormhole chips introduced in driver version 2.4.1. `ChipReset` provides a standardized interface for performing chip-level resets using improved IOCTL commands and supports advanced features including secondary bus reset (SBR), M3/DMC reset, and Xen HVM virtualization coordination.

For chip-specific reset logic that delegates to `ChipReset`, see [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation) and [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation). For the overall reset architecture and migration strategy from legacy implementations, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns). For Grayskull reset procedures, which use a different approach, see [Grayskull Tensix Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.5-grayskull-tensix-reset).

**Sources:**[tt_tools_common/reset_common/chip_reset.py 1-289](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L1-L289)

## Class Overview

The `ChipReset` class is located in [tt_tools_common/reset_common/chip_reset.py 53-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L53-L288) and serves as the modern unified implementation for chip-level resets on both Blackhole and Wormhole PCIe boards. It is invoked by chip-specific reset classes when driver version >= 2.4.1 or when running in Xen HVM mode.

**Delegation Pattern**: Chip-specific classes (`BHChipReset`, `WHChipReset`) conditionally delegate to `ChipReset` based on driver version and platform capabilities. This enables gradual migration from legacy implementations while maintaining backward compatibility.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 53-54](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L53-L54)[tt_tools_common/reset_common/bh_reset.py 90-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L90-L99)

## IOCTL Interface

The `ChipReset` class communicates with the kernel driver via IOCTL commands to the `/dev/tenstorrent/{interface_id}` device file. The IOCTL interface is defined by magic numbers and structured flags.

### IOCTL Constants

| Constant | Value | Description |
| --- | --- | --- |
| `TENSTORRENT_IOCTL_MAGIC` | `0xFA` | Magic number identifying Tenstorrent IOCTL operations |
| `TENSTORRENT_IOCTL_RESET_DEVICE` | `0xFA06` | IOCTL command for device reset operations |

**Sources:**[tt_tools_common/reset_common/chip_reset.py 57-58](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L57-L58)

### IoctlResetFlags Enumeration

The `IoctlResetFlags` enum [tt_tools_common/reset_common/chip_reset.py 25-32](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L25-L32) defines the operation flags passed to the reset IOCTL:

| Flag | Value | Purpose |
| --- | --- | --- |
| `RESTORE_STATE` | 0 | Restore PCIe configuration state after reset |
| `RESET_PCIE_LINK` | 1 | Perform secondary bus reset (SBR) |
| `CONFIG_WRITE` | 2 | Legacy config space write reset |
| `USER_RESET` | 3 | User-initiated reset |
| `ASIC_RESET` | 4 | Reset ASIC only |
| `ASIC_DMC_RESET` | 5 | Reset ASIC including M3/DMC |
| `POST_RESET` | 6 | Post-reset cleanup and validation |

**Sources:**[tt_tools_common/reset_common/chip_reset.py 25-32](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L25-L32)[tt_tools_common/reset_common/chip_reset.py 60-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L85)

### reset_device_ioctl Method

The `reset_device_ioctl` method [tt_tools_common/reset_common/chip_reset.py 60-85](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L85) is the low-level interface for sending IOCTL commands:

**Method Signature:**

`def reset_device_ioctl(self, interface_id: int, flags: int) -> bool`
**Parameters:**

*   `interface_id`: PCI interface number (e.g., 0, 1, 2)
*   `flags`: `IoctlResetFlags` value specifying the operation

**Returns:**`True` if successful (result == 0), `False` otherwise

**Structure Format:** The method uses a structured binary format for IOCTL communication:

*   Input: `II` (2 unsigned integers: output_size_bytes, flags)
*   Output: `II` (2 unsigned integers: unused, result)

**Error Handling:** The method propagates exceptions from file operations (`FileNotFoundError`) and IOCTL calls (`OSError`). The file descriptor is guaranteed to close via the `finally` block.

## Secondary Bus Reset

Secondary Bus Reset (SBR) is a PCIe-level reset mechanism that resets all devices on a bus segment. `ChipReset` performs SBR as the first step in the reset sequence when `secondary_bus_reset=True`.

**Implementation:**[tt_tools_common/reset_common/chip_reset.py 195-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L195-L203)

The SBR step iterates through all PCI interfaces and calls `reset_device_ioctl` with `IoctlResetFlags.RESET_PCIE_LINK`. If SBR fails for a device, a warning is printed but the reset sequence continues. This non-fatal behavior ensures that ASIC reset can proceed even if SBR is not supported or fails on the platform.

**Why SBR First:** Performing SBR before ASIC reset helps clean up any PCIe link state issues and provides a more thorough reset of the PCIe hierarchy.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 195-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L195-L203)

## ASIC Reset Sequence

After optional secondary bus reset, `ChipReset` performs the ASIC-level reset using either `ASIC_RESET` or `ASIC_DMC_RESET` flags.

### Reset Flag Selection

The reset flag is selected based on the `reset_m3` parameter [tt_tools_common/reset_common/chip_reset.py 206-212](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L206-L212):

| Condition | Flag | Behavior |
| --- | --- | --- |
| `reset_m3=True` | `ASIC_DMC_RESET` | Resets ASIC including M3 management processor and DMC |
| `reset_m3=False` | `ASIC_RESET` | Resets ASIC only, preserving M3 state |

**M3 Reset Timing:** When `reset_m3=True`, the post-reset wait time is set to the `m3_delay` parameter (default 20 seconds) to allow time for M3 firmware to reinitialize. During board management firmware (BMFW) upgrades, the M3 reset can take 60 seconds or more [tt_tools_common/reset_common/bh_reset.py 138-140](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L138-L140)

**Sources:**[tt_tools_common/reset_common/chip_reset.py 206-212](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L206-L212)[tt_tools_common/reset_common/chip_reset.py 259-265](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L259-L265)

## Xen HVM Support

`ChipReset` includes special handling for Xen Hardware Virtual Machine (HVM) environments to coordinate PCIe device removal and reappearance with the hypervisor host.

### Xen HVM Detection

The `check_xen_hvm` function [tt_tools_common/reset_common/chip_reset.py 34-51](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L34-L51) determines if the system is running as a Xen HVM guest:

`def check_xen_hvm() -> bool:    """Check if the system is a Xen HVM guest"""`
**Detection Logic:**

1.   Read `/sys/hypervisor/type` → must be `"xen"`
2.   Read `/sys/hypervisor/guest_type` → must be `"HVM"`
3.   Return `True` if both conditions met, `False` otherwise

**Error Handling:**`FileNotFoundError` and `OSError` are caught and treated as "not Xen HVM". Other exceptions propagate.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 34-51](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L34-L51)

### Xenstore Coordination Protocol

When running in Xen HVM mode, `ChipReset` uses xenstore to coordinate with the host during PCIe hotplug operations:

**Xenstore Filename Format:**[tt_tools_common/reset_common/chip_reset.py 178-180](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L178-L180)

```
pci_hard_reset/{bus}_{device}-{function}
```

Example: `pci_hard_reset/01_00-0` for BDF `0000:01:00.0`

**Parallel Thread Waiting:**[tt_tools_common/reset_common/chip_reset.py 228-256](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L228-L256)

For multiple devices, `ChipReset` spawns parallel threads to wait for each xenstore file removal simultaneously. This prevents sequential timeout delays when resetting multiple devices.

**Timeout:** 5 minutes (300 seconds) per device. If the xenstore file is not removed within this time, the reset exits with an error [tt_tools_common/reset_common/chip_reset.py 216](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L216-L216)

**Sudo Requirement:** Xenstore operations require sudo privileges. The user is prompted once at the start [tt_tools_common/reset_common/chip_reset.py 170-174](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L170-L174) and credentials are revalidated before waiting [tt_tools_common/reset_common/chip_reset.py 223-227](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L223-L227)

**Sources:**[tt_tools_common/reset_common/chip_reset.py 167-256](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L167-L256)

## Device Reappearance

After ASIC reset, devices may be temporarily removed from the PCIe bus (hotplug removal) and then reappear with potentially new interface IDs. The `wait_for_device_to_reappear` method handles this scenario.

### wait_for_device_to_reappear Method

**Method Signature:**[tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111)

`def wait_for_device_to_reappear(self, bdf: str, timeout: int = 10) -> int`
**Parameters:**

*   `bdf`: PCIe Bus:Device.Function identifier (e.g., `"0000:01:00.0"`)
*   `timeout`: Maximum seconds to wait (default: 10)

**Returns:** New PCI interface ID for the reappeared device

**Algorithm:**

1.   Poll `/sys/bus/pci/devices/{bdf}/tenstorrent/tenstorrent!*` for device node
2.   Extract interface ID from filename (e.g., `tenstorrent!0` → `0`)
3.   Verify `/dev/tenstorrent/{interface_id}` exists
4.   Return interface ID when device is accessible
5.   Exit with error if timeout expires

**Why Interface IDs Change:** During hotplug removal and reappearance, the kernel driver may assign a different interface ID to the device. The original interface ID cannot be used after the device reappears.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 87-111](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L87-L111)

## Full LDS Reset Method

The `full_lds_reset` method [tt_tools_common/reset_common/chip_reset.py 114-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L114-L288) is the main entry point for performing a complete chip reset sequence.

### Method Signature

`def full_lds_reset(    self,    pci_interfaces: List[int],    reset_m3: bool = False,    silent: bool = False,    m3_delay: int = 20,    secondary_bus_reset: bool = True,) -> List[PciChip]:`
### Parameters

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `pci_interfaces` | `List[int]` | Required | List of PCI interface numbers to reset |
| `reset_m3` | `bool` | `False` | Whether to reset M3/DMC (management processor) |
| `silent` | `bool` | `False` | Suppress informational print statements |
| `m3_delay` | `int` | `20` | Seconds to wait after M3 reset |
| `secondary_bus_reset` | `bool` | `True` | Perform secondary bus reset before ASIC reset |

### Return Value

Returns a `List[PciChip]` containing pyluwen `PciChip` objects for all successfully reset devices.

### Complete Reset Flow

### Pre-Flight Validation

Before executing the reset, the method performs three critical checks [tt_tools_common/reset_common/chip_reset.py 136-148](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L136-L148):

1.   **Driver Version:** Calls `check_driver_version(operation="reset", minimum_required_version_str="2.4.1")` to ensure driver >= 2.4.1
2.   **Platform Check:** Rejects ARM platforms (`arm` or `aarch` prefix) as they don't support PCIe device rescans
3.   **Deduplication:** Removes duplicate PCI interfaces from the input list using `list(set(pci_interfaces))`

**Sources:**[tt_tools_common/reset_common/chip_reset.py 114-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L114-L288)

### BDF Collection and Cleanup

The method collects PCIe Bus:Device.Function (BDF) strings for all devices and explicitly deletes chip objects to close file descriptors [tt_tools_common/reset_common/chip_reset.py 157-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L157-L165):

`for pci_interface in pci_interfaces:    chip = PciChip(pci_interface=pci_interface)    bdf = chip.get_pci_bdf()    bdf_list.append(bdf)    # Force garbage collection of the chip object to close any open file descriptors    # This is necessary for dockerized containers    del chip`
This cleanup is critical in containerized environments where file descriptor limits may be restrictive.

### Post-Reset Wait Calculation

The sleep duration after ASIC reset depends on the reset type [tt_tools_common/reset_common/chip_reset.py 259](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L259-L259):

*   **M3 Reset:**`post_reset_wait = m3_delay` (default 20 seconds)
*   **Standard Reset:**`post_reset_wait = max(2, 0.4 * len(pci_interfaces))` (scales with device count)

This scaled timing accounts for staggered hotplug removal when multiple devices are reset.

## Platform and Driver Requirements

### Minimum Driver Version

`ChipReset` requires tt-kmd driver version **2.4.1 or higher**. This requirement is enforced at the start of `full_lds_reset` via [tt_tools_common/reset_common/chip_reset.py 137](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L137-L137):

`check_driver_version(operation="reset", minimum_required_version_str="2.4.1")`
If the driver version is insufficient, the method raises an exception and exits. For details on driver version checking, see [Driver Version Management](https://deepwiki.com/tenstorrent/tt-tools-common/3.1-driver-version-management).

### Platform Restrictions

**ARM Platform Limitation:**[tt_tools_common/reset_common/chip_reset.py 139-148](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L139-L148)

`ChipReset` does not support ARM or aarch64 platforms due to PCIe device rescan limitations on those architectures. If the platform is detected as ARM-based, the method prints an error and exits:

```
Cannot perform board reset on Arm systems, please reboot the system to reset the boards. Exiting...
```

**Platform Detection:** Uses `get_host_info()["Platform"]` to retrieve platform string and checks for `"arm"` or `"aarch"` prefix.

**Workaround:** On ARM systems, users must perform a full system reboot to reset Tenstorrent chips.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 137-148](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L137-L148)

## Error Handling and Exit Codes

### IOCTL Failures

The `reset_device_ioctl` method can raise exceptions:

*   **`FileNotFoundError`:** Device file `/dev/tenstorrent/{interface_id}` does not exist
*   **`OSError`:** IOCTL operation failed in kernel driver

These exceptions propagate to the caller and typically result in program termination.

### Secondary Bus Reset Warnings

SBR failures are non-fatal [tt_tools_common/reset_common/chip_reset.py 199-203](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L199-L203):

`if not self.reset_device_ioctl(pci_interface, IoctlResetFlags.RESET_PCIE_LINK):    print(CMD_LINE_COLOR.YELLOW,          f"Warning: Secondary bus reset not completed for device at PCI index {pci_interface}. Continuing with reset.",          CMD_LINE_COLOR.ENDC)`
The reset sequence continues even if SBR fails, as ASIC reset may still succeed.

### Device Reappearance Timeout

If a device fails to reappear within 10 seconds, the method exits [tt_tools_common/reset_common/chip_reset.py 103-109](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L103-L109):

`if not device_reappeared:    print(CMD_LINE_COLOR.RED,          f"Timeout waiting for device at PCI index {interface_id} to reappear.",          CMD_LINE_COLOR.ENDC)    sys.exit(1)`
### Xen Xenstore Timeout

In Xen HVM mode, if a xenstore file is not removed by the host within 5 minutes, the method exits [tt_tools_common/reset_common/chip_reset.py 245-248](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L245-L248):

`print(CMD_LINE_COLOR.RED,      f"Timeout waiting for xenstore file {xenstore_filename} to be removed. Exiting...",      CMD_LINE_COLOR.ENDC)sys.exit(1)`
### Post-Reset IOCTL Failure

If the `POST_RESET` IOCTL fails, the method prints an error and exits [tt_tools_common/reset_common/chip_reset.py 275-278](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L275-L278):

`else:    print(CMD_LINE_COLOR.RED,          f"Post-reset actions did not complete successfully for device at PCI index {pci_interface}.",          CMD_LINE_COLOR.ENDC)    sys.exit(1)`
**Exit Code Strategy:** The method uses `sys.exit(1)` for fatal errors, ensuring that calling scripts can detect failures via non-zero exit codes.

**Sources:**[tt_tools_common/reset_common/chip_reset.py 60-288](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/chip_reset.py#L60-L288)

## Integration with Chip-Specific Reset Classes

The `ChipReset` class is not typically invoked directly by end users. Instead, chip-specific reset classes delegate to it when appropriate conditions are met.

### BHChipReset Delegation

From [tt_tools_common/reset_common/bh_reset.py 92-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L92-L99):

`# Use new reset for driver version >= 2.4.1# or perform reset using ChipReset class which handles Xen HVM caseif is_driver_version_at_least(get_driver_version(), "2.4.1") or check_xen_hvm():    return ChipReset().full_lds_reset(        pci_interfaces,        reset_m3=reset_m3,        silent=silent,        m3_delay=m3_delay,        secondary_bus_reset=secondary_bus_reset    )`
**Delegation Conditions:**

1.   Driver version >= 2.4.1, OR
2.   Running in Xen HVM mode (regardless of driver version)

If neither condition is met, `BHChipReset` uses its legacy implementation with config space polling.

### WHChipReset Delegation

Wormhole reset follows the same delegation pattern, checking driver version and Xen HVM status before deciding between `ChipReset` and legacy ARC message-based reset. For details, see [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation).

**Sources:**[tt_tools_common/reset_common/bh_reset.py 92-99](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/bh_reset.py#L92-L99)

## Summary

The `ChipReset` class provides a modern, unified reset implementation with the following key features:

| Feature | Description |
| --- | --- |
| **IOCTL Interface** | Structured binary protocol with `IoctlResetFlags` for various reset operations |
| **Secondary Bus Reset** | Optional PCIe-level reset before ASIC reset |
| **M3/DMC Reset** | Support for management processor reset with configurable delays |
| **Xen HVM Support** | Xenstore coordination with parallel thread waiting for multi-device scenarios |
| **Device Hotplug** | Automatic detection of device reappearance with new interface IDs |
| **Platform Validation** | Driver version and architecture checks prevent unsupported operations |
| **Delegation Pattern** | Chip-specific classes transparently upgrade to `ChipReset` when available |

**Migration Path:** The architecture allows gradual adoption as users upgrade to driver version 2.4.1+, with legacy implementations maintained for backward compatibility.
