## Purpose and Scope

This document covers test scenarios that verify cache management operations in multi-processor (multi-hart) configurations within the RISC-V Cache Management Extensions (ZICBOM/ZICBOZ/ZICBOP). These tests validate cache coherence, synchronization between harts, and the interaction between cache operations and atomic memory operations (LR/SC).

For single-processor cache operation tests including CSR control and privilege mode testing, see [Cache Operation Scenarios](https://deepwiki.com/tenstorrent/riscv-coretp/4.1.1-cache-operation-scenarios). For the overall cache management extension test coverage, see [Cache Management Extensions](https://deepwiki.com/tenstorrent/riscv-coretp/4.1-cache-management-extensions-(zicbomzicbozzicbop)).

## Multi-Hart Coordination Primitives

Multi-processor cache tests use specialized TestStep primitives to coordinate execution across multiple hardware threads (harts). The framework provides explicit hart assignment and synchronization mechanisms.

### Hart Control Steps

| Step Type | Purpose | Key Parameters | Usage |
| --- | --- | --- | --- |
| `Hart` | Assign following steps to specific hart | `hart_index` | Switches execution context to specified hart |
| `HartExit` | Synchronization barrier or hart termination | `sync` | When `sync=True`, creates barrier point |
| `MemAccess` | Memory operations including atomics | `op` | Supports `lr.w`, `sc.w`, and CMO operations |

The `Hart` step switches execution context to a specific hardware thread. All subsequent steps execute on that hart until another `Hart` step changes the context. The `HartExit` step with `sync=True` creates a synchronization barrier where all harts must reach before any can proceed.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 764-796](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L764-L796)

## Test Scenario: Multi-Hart Cache Block Zeroing

`SID_ZICBO_021` validates that `cbo.zero` correctly zeroes an entire cache block and that the effect is visible across all harts in the system.

### Test Sequence

**SID_ZICBO_021 Test Flow**

### Key Validation Points

1.   **Cross-Hart Visibility**: Data written by Hart 0 must be visible to Hart 1 before the `cbo.zero` operation
2.   **Cache Block Alignment**: The `cbo.zero` operation zeros the entire cache block containing the target address, regardless of alignment
3.   **Memory Ordering**: The zeroed data must be visible to Hart 0 after the second synchronization barrier
4.   **CSR Configuration**: The test requires `menvcfg.CBZE`, `senvcfg.CBZE`, and `henvcfg.CBZE` to be set to allow execution

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 748-797](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L748-L797)

## LR/SC and Cache Operation Interactions

The RISC-V specification defines complex interactions between Load-Reserved/Store-Conditional (LR/SC) atomic operations and Cache Management Operations (CMO). Multi-processor tests validate that CMOs properly break or preserve LR/SC reservations.

### LR/SC Reservation Semantics with CMO

### Test Scenario: CMO Between LR and SC

`SID_ZICBO_049` tests the behavior when a cache operation executes between a Load-Reserved and Store-Conditional to the same address.

**CMO Between LR/SC Test Flow**

The test validates that:

1.   **CMO Execution**: The cache operation completes without raising exceptions
2.   **SC Behavior**: The Store-Conditional either succeeds or fails based on whether the reservation was broken
3.   **Memory Consistency**: Final memory state is consistent regardless of SC success/failure

## Test Scenario: Multi-Hart LR/SC with CMO

`SID_ZICBO_039` validates LR/SC behavior when another hart performs a cache operation to the same address between the LR and SC.

### Test Structure

**SID_ZICBO_039 Multi-Hart Atomic Test**

### Expected Behavior

The RISC-V architecture permits implementations to:

*   **Break Reservation**: Hart 1's `cbo.zero` may cause Hart 0's Store-Conditional to fail
*   **Preserve Reservation**: Some implementations may allow the SC to succeed if specific ordering guarantees exist

The test validates that regardless of the implementation choice:

1.   No exceptions occur during CMO execution
2.   The SC returns a valid result (0 or 1)
3.   Memory state remains consistent with the SC result

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 931-976](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L931-L976)

## Test Configuration Requirements

Multi-processor cache tests require specific environment configuration through `TestEnvCfg`:

`TestEnvCfg(    min_num_harts=2,  # Require at least 2 harts    priv_modes=[PrivilegeMode.M, PrivilegeMode.S, PrivilegeMode.U],    paging_modes=[PagingMode.SV39]  # Optional: for virtual memory tests)`
### CSR Configuration Pattern

All multi-processor cache tests follow this CSR initialization pattern:

`menvcfg_write = CsrWrite(csr_name="menvcfg", set_mask=0xF0)  # Enable all CMOsenvcfg_write = CsrWrite(csr_name="senvcfg", set_mask=0xF0)henvcfg_write = CsrWrite(csr_name="henvcfg", set_mask=0xF0)`
The bitmask `0xF0` sets:

*   Bit 4-5: `CBIE` (Cache Block Invalidate Enable)
*   Bit 6: `CBCFE` (Cache Block Clean and Flush Enable)
*   Bit 7: `CBZE` (Cache Block Zero Enable)

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 753-755](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L753-L755)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 781](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L781-L781)

## Summary of Multi-Processor Test Scenarios

| Test ID | Description | Harts | Key Operations | Validation |
| --- | --- | --- | --- | --- |
| SID_ZICBO_021 | Cache block zeroing visible across harts | 2 | Store → cbo.zero → Load | Data zeroed and visible |
| SID_ZICBO_039 | CMO does not affect LR/SC on different hart | 2 | lr.w/sc.w on Hart 0, cbo.zero on Hart 1 | Atomic operation completes |
| SID_ZICBO_049 | CMO between LR and SC on same hart | 1 | lr.w → cbo.flush → sc.w | SC result is valid |

### Common Test Patterns

1.   **Producer-Consumer**: One hart modifies memory, another performs CMO, third verifies effect
2.   **Interference Testing**: Execute LR/SC on one hart while another hart performs CMO to same address
3.   **Barrier Synchronization**: Use `HartExit(sync=True)` to enforce execution ordering
