## Purpose and Scope

This document describes the coherent DMA buffer allocation mechanism provided by the Tenstorrent kernel driver. DMA buffers are kernel-allocated, physically contiguous memory regions that can be shared between the host CPU and Tenstorrent devices for efficient data transfer.

This page covers:

*   Coherent DMA buffer allocation via `dma_alloc_coherent`.
*   The `TENSTORRENT_IOCTL_ALLOCATE_DMA_BUF` interface. [memory.c 456-457](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L456-L457)
*   Optional NOC DMA mapping using Outbound iATU regions. [memory.c 159-160](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L159-L160)
*   User-space memory mapping of DMA buffers via `mmap`. [memory.c 1236-1237](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1236-L1237)
*   Per-file descriptor buffer tracking and hash table management. [chardev_private.h 42-48](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L42-L48)

For user-provided page pinning, see [6.2 Page Pinning](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/6.2%20Page%20Pinning) For TLB window management, see [6.3 TLB Management](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/6.3%20TLB%20Management) For detailed iATU architecture, see [6.4 IATU and Address Translation](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/6.4%20IATU%20and%20Address%20Translation)

## Overview

DMA buffers in this driver are allocated from kernel memory using the DMA coherent API. They provide:

*   **Physical contiguity**: Guaranteed contiguous physical addresses (or IOVAs) for hardware DMA. [memory.c 497-499](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L497-L499)
*   **Cache coherence**: CPU and device see consistent data without explicit cache management.
*   **Zero-copy access**: User space can `mmap` the buffer for direct access. [memory.c 1255-1256](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1255-L1256)
*   **Optional NOC mapping**: Can be exposed to device NOC fabric via iATU translation. [memory.c 182-183](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L182-L183)

Each buffer is tracked by a `dmabuf` structure and owned by a specific file descriptor (`chardev_private`). Buffers persist until the file descriptor is closed, at which point they are automatically freed. [memory.c 1289-1319](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1289-L1319)

## Data Structures

### dmabuf Structure

The `dmabuf` structure represents a single allocated DMA buffer. It is defined internally in `memory.c`. [memory.c 30-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L30-L38)

**Field Descriptions:**

| Field | Type | Description |
| --- | --- | --- |
| `hash_chain` | `hlist_node` | Links buffer into `chardev_private.dmabufs` hash table. [memory.c 31](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L31-L31) |
| `ptr` | `void*` | Kernel virtual address for CPU access. [memory.c 32](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L32-L32) |
| `phys` | `dma_addr_t` | Physical address or IOVA (if IOMMU active). [memory.c 33](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L33-L33) |
| `size` | `u64` | Buffer size in bytes (always page-aligned). [memory.c 34](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L34-L34) |
| `index` | `u8` | User-assigned buffer index (0-254). [memory.c 35](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L35-L35) |
| `outbound_iatu_region` | `int` | iATU region number if NOC DMA enabled, or -1. [memory.c 36](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L36-L36) |

### Per-FD Tracking

Each open file descriptor maintains its own hash table of DMA buffers to ensure isolation and automatic cleanup. [chardev_private.h 42-48](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L42-L48)

The hash table uses the `index` field as the key, supporting up to 255 buffers per file descriptor with efficient lookup via `lookup_dmabuf_by_index`. [memory.c 442-450](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L442-L450)

Sources: [chardev_private.h 42-48](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev_private.h#L42-L48)[memory.c 442-450](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L442-L450)

## Allocation Flow

### IOCTL Interface

User space allocates DMA buffers via `TENSTORRENT_IOCTL_ALLOCATE_DMA_BUF`. The implementation is found in `ioctl_allocate_dma_buf`. [memory.c 456-543](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L456-L543)

Sources: [memory.c 456-543](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L456-L543)[memory.c 159-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L159-L187)

### Allocation Parameters and Constraints

**Defined Limits:**

*   `MAX_DMA_BUF_SIZE_LOG2 = 28` (256MB maximum). [memory.h 10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L10-L10)
*   `MAX_DMA_BUF_SIZE = 1 << 28`. [memory.c 240](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L240-L240)
*   All sizes must be page-aligned (`size % PAGE_SIZE == 0`). [memory.c 479-480](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L479-L480)
*   `TENSTORRENT_MAX_DMA_BUFS = 255`. [memory.c 476](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L476-L476)

**Input Validation:** The driver checks if the device is DMA-capable before proceeding. [memory.c 473-474](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L473-L474) It also verifies that the requested `buf_index` is not already present in the hash table. [memory.c 484-487](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L484-L487)

Sources: [memory.h 10](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L10-L10)[memory.c 240](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L240-L240)[memory.c 473-487](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L473-L487)

## NOC DMA Mapping

### Purpose

When the `TENSTORRENT_ALLOCATE_DMA_BUF_NOC_DMA` flag is set, the driver configures an Outbound iATU region to expose the DMA buffer to the device's Network-on-Chip (NOC) fabric. [memory.c 507-517](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L507-L517)

Sources: [memory.c 507-517](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L507-L517)[memory.c 182-183](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L182-L183)

### iATU Region Allocation and Gap Finding

The `setup_noc_dma` function allocates NOC address space using a top-down or bottom-up allocator based on the caller's requirement. For DMA buffers, it typically uses top-down. [memory.c 159-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L159-L187)

1.   **`find_iatu_region_top_down`**: Searches for a gap in the NOC address space starting from `noc_dma_limit` down to 0. [memory.c 54-86](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L54-L86)
2.   **`find_iatu_region_bottom_up`**: Searches for a gap starting from 0 up to `noc_dma_limit`. [memory.c 88-120](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L88-L120)
3.   **`configure_outbound_iatu`**: Programs the hardware registers via the device-specific `dev_class->configure_outbound_atu` function and records the mapping in `tenstorrent_device.outbound_iatus`. [memory.c 123-156](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L123-L156)

The hardware supports `TENSTORRENT_MAX_OUTBOUND_IATU_REGIONS` (16) concurrent regions. [memory.h 60](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L60-L60)

Sources: [memory.c 54-187](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L54-L187)[memory.h 60-66](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.h#L60-L66)

## Memory Mapping (mmap)

### mmap Offset Scheme

DMA buffers use a specific region of the `mmap` offset space to allow user-space to identify which buffer is being mapped. [memory.c 288-307](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L288-L307)

**Offset Encoding:**

*   `MMAP_OFFSET_DMA_BUF` is defined as `(PAGE_SIZE - 255 - 1) << 32`. [memory.c 288-290](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L288-L290)
*   Each buffer index occupies a 4GB window: `buf_index << 32`. [memory.c 301-303](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L301-L303)
*   The final offset provided to user-space is `MMAP_OFFSET_DMA_BUF | ((u64)buf->index << 32)`. [memory.c 526](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L526-L526)

Sources: [memory.c 288-307](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L288-L307)[memory.c 526](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L526-L526)

### Implementation

The `tenstorrent_mmap` function delegates to `vma_dmabuf_target` to identify if the offset corresponds to a DMA buffer. [memory.c 1253-1256](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1253-L1256) If a match is found in the per-FD hash table, it uses `dma_mmap_coherent` to map the memory. [memory.c 1255](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1255-L1255)

Sources: [memory.c 1236-1287](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1236-L1287)[memory.c 1016-1040](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1016-L1040)

## Resource Management and Cleanup

### Lifetime and Ownership

DMA buffers are strictly owned by the `chardev_private` instance (the file descriptor). They cannot be explicitly freed via IOCTL currently; `ioctl_free_dma_buf` returns `-EINVAL`. [memory.c 545-553](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L545-L553) This is a design choice to simplify mapping lifetime management, as user-space mappings refcount the `struct file`.

### Cleanup Process

When the file descriptor is closed, `tenstorrent_memory_cleanup` iterates through the hash table and releases all resources. [memory.c 1289-1319](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L1289-L1319)

**`teardown_outbound_iatu`**: This function clears the entry in the global `tt_dev->outbound_iatus` array and resets the hardware registers if a region was allocated. [memory.c 309-330](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/memory.c#L309-L330)
