## Purpose and Scope

This document introduces the **Data Movement** system in TT-Metal, which encompasses the mechanisms for transferring data between Tensix cores, DRAM, and other subsystems on Tenstorrent hardware. This page provides an overview of:

*   What data movement is and why it matters for operation writers
*   The role of the Network-on-Chip (NOC) as the primary data transport mechanism
*   The comprehensive test suite that validates and measures data movement performance
*   The testing framework used to profile and analyze data movement patterns

## What is Data Movement?

**Data movement** refers to all operations that transfer data between different locations in the TT-Metal system. These operations are fundamental to ML workloads, as computation on Tensix cores requires:

1.   **Loading data** from DRAM or other cores into local L1 memory
2.   **Distributing intermediate results** between cores for multi-core operations
3.   **Writing results** back to DRAM or forwarding to subsequent pipeline stages

Unlike general-purpose CPUs where data movement is implicit and managed by the memory hierarchy, TT-Metal requires operation writers to explicitly orchestrate data movement using NOC APIs. This explicit control enables:

*   **Overlapping computation with communication** through asynchronous operations
*   **Optimizing bandwidth** by selecting appropriate NOC instances and routing paths
*   **Implementing custom communication patterns** such as multicast, gather, and scatter

The efficiency of data movement directly impacts overall application performance, as bandwidth-limited operations can become bottlenecked by suboptimal data transfer patterns.

## The Network-on-Chip (NOC)

The **Network-on-Chip (NOC)** is the hardware infrastructure that implements all data movement in TT-Metal. The NOC is a high-bandwidth, packet-switched network that connects:

*   Tensix cores (containing compute engines and L1 memory)
*   DRAM controllers
*   PCIe interfaces
*   Ethernet endpoints (for multi-chip configurations)

### Key NOC Characteristics

**Diagram: NOC System Overview**

| Feature | Description |
| --- | --- |
| **Dual NOC Instances** | Two independent NOC instances (NOC0 and NOC1) controlled by BRISC and NCRISC processors respectively, providing redundancy and increased aggregate bandwidth |
| **2D Torus Topology** | Routers arranged in a 2D torus with wrap-around connections, ensuring uniform latency between nodes |
| **Packet-Switched** | Data transmitted in packets with maximum size of 8KB (Wormhole) or 16KB (Blackhole) |
| **Interface Width** | 256 bits/cycle (32 bytes/cycle) on Wormhole, 512 bits/cycle (64 bytes/cycle) on Blackhole |
| **Virtual Channels** | 16 virtual channels per NOC instance for traffic isolation and deadlock avoidance |
| **Operations** | Read, write (unicast/multicast), and atomic operations |

The NOC is **transparent to software** in the sense that operation writers use high-level APIs ([noc_async_read_barrier](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/noc_async_read_barrier)[noc_async_write](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/noc_async_write)) without needing to understand routing algorithms or flow control. However, understanding NOC characteristics is crucial for achieving optimal performance.

For detailed architecture including routing algorithms, virtual channel usage, and router design, see [NOC Architecture](https://deepwiki.com/tenstorrent/tt-low-level-documentation/2.2-noc-architecture).

## Data Movement Test Framework

The TT-Metal repository includes a comprehensive test suite located in [tests/tt_metal/tt_metal/data_movement/](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/tests/tt_metal/tt_metal/data_movement/) that validates NOC functionality and measures performance across diverse data movement patterns. This framework serves four primary purposes:

1.   **Functional Validation** - Verify correctness of data transfers across all supported patterns
2.   **Performance Measurement** - Quantify bandwidth and latency for each data movement primitive
3.   **Regression Detection** - Catch performance degradations or correctness issues in CI
4.   **Documentation** - Provide reference implementations and expected performance baselines

### Test Organization

**Diagram: Test Suite Organization and Data Flow**

Each test category is organized in its own subdirectory with:

*   Test implementation files (`test_*.cpp`)
*   Data movement kernels (`*_kernel.cpp`)
*   README documentation
*   Unique test IDs for regression tracking

**Sources:**[data_movement_doc/general/intro_to_dm.md 40-77](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/intro_to_dm.md?plain=1#L40-L77)

## Test Coverage by Pattern

The test suite covers all fundamental data movement patterns needed for ML operations:

| Pattern | Description | Test Directory | Use Cases |
| --- | --- | --- | --- |
| **DRAM Read** | Core reads from DRAM banks | `dram_read/` | Loading weights, activations |
| **DRAM Write** | Core writes to DRAM banks | `dram_write/` | Storing results, checkpointing |
| **One-to-One** | Single core writes to another core | `one_to_one/` | Pipeline transfers, core-to-core forwarding |
| **One-from-One** | Single core reads from another core | `one_from_one/` | Gather operations, data aggregation |
| **One-to-All (Unicast)** | Core sends to all other cores via individual writes | `one_to_all/` | Broadcasting data, weight distribution |
| **One-to-All (Multicast)** | Core sends to multiple cores with single operation | `one_to_all/` | Efficient broadcasting |
| **One-from-All** | Core reads from all other cores | `one_from_all/` | Reduction operations, gradient gathering |
| **Loopback** | Core writes to itself via NOC | `loopback/` | Testing NOC path, buffer management |
| **Multiple Cores** | Concurrent data movement from multiple sources | `multiple_cores/` | Real-world multi-core scenarios |

Each pattern includes multiple test variants:

*   Different transfer sizes (small, medium, large)
*   Different NOC instances (NOC0 vs NOC1)
*   Different virtual channels
*   Posted vs non-posted writes
*   Loopback enabled/disabled

## Performance Measurement and Analysis

### Test Execution Flow

**Diagram: Test Execution and Profiling Flow**

### Metrics Collection

The test framework uses cycle counters and NOC transaction counters to measure:

*   **Latency**: Time from operation issue to completion (cycles)
*   **Bandwidth**: Data transferred per cycle (bytes/cycle)
*   **Throughput**: Aggregate bandwidth across multiple cores

The [python/test_data_movement.py](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/python/test_data_movement.py) script orchestrates test execution, collects metrics, and generates visualizations. Results are organized by platform:

*   `wormhole_b0/` - Performance plots for Wormhole B0
*   `blackhole/` - Performance plots for Blackhole P100

### Ideal Performance Baselines

The following table shows **ideal performance** (maximum achievable bandwidth) for each data movement primitive, measured in bytes per cycle:

| Primitive | Wormhole B0 | Blackhole P100 |
| --- | --- | --- |
| DRAM Read | 22 | 33 |
| DRAM Write | 21 | 34 |
| One To One | 29 | 60 |
| One From One | 28 | 60 |
| One To All (Unicast) | 31 | 62 |
| One To All (Multicast) | 15 | 24 |
| One To All (Multicast + Linked) | 22 | 41 |
| One From All | 30 | 60 |

These baselines represent performance under optimal conditions and serve as targets for operation optimization. For detailed analysis of what factors limit performance below these ideals, see [Performance Analysis](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4-performance-analysis).

## Adding New Tests

The test framework is designed for extensibility. To add a new data movement test:

1.   **Create test directory** under [tests/tt_metal/tt_metal/data_movement/](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/tests/tt_metal/tt_metal/data_movement/)
2.   **Assign unique test ID** following the existing numbering scheme
3.   **Implement test harness** (`test_*.cpp`) using TT-Metal device APIs
4.   **Write dataflow kernels** (`*_kernel.cpp`) using NOC APIs
5.   **Add README** documenting test purpose, parameters, and expected behavior
6.   **Integrate with profiling** by adding test to [python/test_data_movement.py](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/python/test_data_movement.py)

For detailed guidelines, refer to [tests/tt_metal/tt_metal/data_movement/README.md](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/tests/tt_metal/tt_metal/data_movement/README.md?plain=1)

## Key Takeaways

1.   **Data movement is explicit** in TT-Metal - operation writers must orchestrate transfers using NOC APIs
2.   **The NOC is the primary transport** - all inter-core and DRAM communication goes through the NOC
3.   **Dual NOC instances** provide redundancy and increased bandwidth
4.   **Comprehensive test coverage** validates all data movement patterns used in ML workloads
5.   **Performance baselines** exist for each primitive, serving as optimization targets
6.   **Platform differences** between Wormhole and Blackhole affect bandwidth and capabilities

Understanding these fundamentals is essential for writing efficient TT-Metal operations that achieve high utilization of the NOC infrastructure.
