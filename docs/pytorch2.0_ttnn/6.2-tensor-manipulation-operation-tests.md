## Purpose and Scope

This page documents the test suite for tensor manipulation operations, which verify the correct lowering of PyTorch shape-transformation operations to their TTNN equivalents. Tensor manipulation operations include `view`, `reshape`, `slice`, and `squeeze` operations that change tensor shapes without modifying the underlying data values.

For tests of element-wise and pointwise operations, see [Element-wise and Pointwise Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.1-element-wise-and-pointwise-operation-tests). For tests of tensor creation operations like `full`, `ones`, and `arange`, see [Creation and Specialized Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.3-creation-and-specialized-operation-tests).

Sources: [tests/lowering/tensor_manipulation/test_view.py 1-105](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/tensor_manipulation/test_view.py#L1-L105)[tests/lowering/tensor_manipulation/test_squeeze.py 1-83](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/tensor_manipulation/test_squeeze.py#L1-L83)[tests/lowering/tensor_manipulation/test_slice.py 1-172](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/tensor_manipulation/test_slice.py#L1-L172)

## Overview of Tensor Manipulation Operations

Tensor manipulation operations in PyTorch are lowered to corresponding TTNN operations through the compilation pipeline. The primary operations tested in this category are:

| PyTorch Operation | TTNN Target | Primary Test File |
| --- | --- | --- |
| `torch.reshape` | `ttnn.reshape` | `test_reshape.py`, `test_view.py` |
| `torch.ops.aten.view.default` | `ttnn.experimental.view` | `test_view.py` |
| `torch.ops.aten.slice.Tensor` | `ttnn.slice` | `test_slice.py` |
| `torch.squeeze` | `ttnn.squeeze` | `test_squeeze.py` |

These operations are critical for neural network execution as they enable shape transformations between layers without data copying when possible.

## View and Reshape Operations

### Operation Semantics

The `view` and `reshape` operations both perform shape transformations but with different semantics:

*   **`torch.reshape`**: Flexible shape transformation that may copy data if memory layout requires it
*   **`torch.ops.aten.view.default`**: Strict view operation that requires contiguous memory and shares underlying storage

Both operations are lowered to TTNN operations, with `reshape` typically mapping to `ttnn.reshape` and view operations mapping to `ttnn.experimental.view`.

## Squeeze Operations

### Operation Semantics

The `squeeze` operation removes dimensions of size 1 from a tensor. PyTorch provides two variants:

*   **`torch.squeeze(input, dim)`** - Remove dimension `dim` if it has size 1
*   **`torch.squeeze(input)`** - Remove all dimensions with size 1

Both variants are lowered to `ttnn.squeeze`.

## Integration with Type Conversion

### `_to_copy` Operation

The `_to_copy` operation is used for dtype conversions and interacts closely with tensor manipulation operations. While primarily a type conversion operation, its behavior is tested in the context of shape operations.

**Diagram: _to_copy Integration Patterns**

The test cases `ToCopyViewModule` and `ToCopyExpand` verify that when `_to_copy` follows a view or expand operation and is subsequently used by TTNN operations, the entire sequence is properly lowered.

Sources: [tests/lowering/creation/test_to_copy.py 80-106](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/creation/test_to_copy.py#L80-L106)

### Conversion Logic

The lowering strategy for `_to_copy` is conditional:

1.   **Standalone `_to_copy`**: If there is no `ttnn.from_torch` following the operation, it remains as `aten._to_copy`[tests/lowering/creation/test_to_copy.py 29-51](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/creation/test_to_copy.py#L29-L51)
2.   **`_to_copy` with TTNN operations**: If followed by TTNN operations, the `_to_copy` is converted away and the dtype is handled by `ttnn.from_torch`[tests/lowering/creation/test_to_copy.py 54-77](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/creation/test_to_copy.py#L54-L77)

This ensures that dtype conversions are handled efficiently when the tensor is moved to the TTNN device.

## Data Movement and Layout Considerations

### Tile vs. Row-Major Layout

Tensor manipulation operations must handle layout transitions between different TTNN operation requirements. Most TTNN computational operations (like `abs`, `add`) require **tile layout**, while operations like `reshape` may use **row-major layout**.

The `ViewShareOpModule` specifically tests this interaction:

This pattern ensures the `NodeInputAligner` (see [Data Movement and Alignment](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.3-data-movement-and-alignment-(adddatamovepass))) correctly inserts layout conversion operations when necessary.

Sources: [tests/lowering/tensor_manipulation/test_view.py 48-56](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/tensor_manipulation/test_view.py#L48-L56)[tests/lowering/tensor_manipulation/test_view.py 46-47](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/tensor_manipulation/test_view.py#L46-L47)

### Integration with AddDataMovePass

The `AddDataMovePass` and `NodeInputAligner` work together to insert necessary data movement operations around tensor manipulation operations. This ensures:

1.   **Input alignment**: Tensors are in the correct layout/device before manipulation
2.   **Output alignment**: Results are in the expected layout for downstream operations
3.   **Device placement**: Operations execute on the appropriate device (host vs. TTNN device)

For more details on the data movement system, see [Data Movement and Alignment](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.3-data-movement-and-alignment-(adddatamovepass)).

**Comprehensive Tensor Manipulation Operation Mapping**
