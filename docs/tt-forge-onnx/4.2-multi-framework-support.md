## Purpose and Scope

This document explains how tt-forge-onnx provides unified compilation support for models from multiple deep learning frameworks: PyTorch, ONNX, TensorFlow, JAX, and PaddlePaddle. It covers the abstraction layers, module wrappers, tensor conversion utilities, and TVM frontend integration that enable framework-agnostic model compilation.

For details on the TVM integration and Relay IR processing, see [TVM Integration and Relay IR](https://deepwiki.com/tenstorrent/tt-forge-onnx/4.1-tvm-integration-and-relay-ir). For information on how TVM graphs are translated to Forge operations, see [Operation Translation (TVM to Python)](https://deepwiki.com/tenstorrent/tt-forge-onnx/4.3-operation-translation-(tvm-to-python)).

### PyTorchModule

The `PyTorchModule` wrapper encapsulates `torch.nn.Module` instances and provides forward redirection to handle out-of-order inputs:

**Key Methods:**

*   `__init__(name, module, redirect_forward)` - Creates wrapper with optional forward redirection
*   `forward(*args, **kwargs)` - Executes forward pass with input name mapping
*   `get_parameters()` - Extracts PyTorch parameters as Forge `Parameter` objects

**Sources:**

*   [forge/forge/module.py 79-185](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/module.py#L79-L185)

### OnnxModule

The `OnnxModule` wrapper handles ONNX models by extracting initializers as parameters:

**Key Methods:**

*   `__init__(name, onnx_model)` - Loads ONNX model and extracts initializers
*   `forward(*inputs)` - Runs ONNX inference session
*   `get_parameters()` - Converts ONNX initializers to Forge parameters

**Sources:**

*   [forge/forge/module.py 301-404](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/module.py#L301-L404)

### TFModule

The `TFModule` wrapper handles TensorFlow/Keras models:

**Key Methods:**

*   `__init__(name, module, inputs)` - Creates wrapper and traces concrete function
*   `forward(*args, **kwargs)` - Executes TF model with training=False
*   `get_parameters()` - Extracts trainable and non-trainable variables

**Sources:**

*   [forge/forge/module.py 188-298](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/module.py#L188-L298)

### JAXModule

The `JAXModule` wrapper supports Flax models via JAX-to-TensorFlow conversion:

**Key Methods:**

*   `__init__(name, module, params)` - Stores Flax model and parameters
*   `forward(*args)` - Executes Flax model forward pass
*   `get_parameters()` - Flattens nested JAX parameter dictionaries

**Sources:**

*   [forge/forge/module.py 407-505](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/module.py#L407-L505)

### PaddleModule

The `PaddleModule` wrapper handles PaddlePaddle models:

**Key Methods:**

*   `__init__(name, module)` - Wraps paddle.nn.Layer instance
*   `forward(*args)` - Executes Paddle forward pass
*   `get_parameters()` - Extracts Paddle parameters

**Sources:**

*   [forge/forge/module.py 508-583](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/module.py#L508-L583)

## TVM Frontend Integration

Each framework connects to TVM through dedicated frontend converters that produce Relay IR.

### Compilation Entry Points

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 176-249](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L176-L249)

### PyTorch Compilation Path

PyTorch models are traced using `torch.jit.trace` before conversion to Relay IR:

**Process:**

1.   Wrap model in evaluation mode
2.   Trace with sample inputs using `torch.jit.trace`
3.   Convert traced ScriptModule to Relay IR via `from_pytorch`
4.   Apply Relay passes and graph partitioning

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 560-670](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L560-L670)

### ONNX Compilation Path

ONNX models are loaded directly and converted to Relay IR:

**Process:**

1.   Load ONNX model proto
2.   Extract initializers as parameters
3.   Convert to Relay IR via `from_onnx`
4.   Verify against ONNX runtime if enabled
5.   Apply Relay passes and partitioning

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 708-860](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L708-L860)

### TensorFlow Compilation Path

TensorFlow models require concrete function generation:

**Process:**

1.   Get concrete function from Keras model
2.   Convert variables to constants if needed
3.   Extract GraphDef
4.   Convert to Relay IR via `from_tensorflow`
5.   Apply Relay passes and partitioning

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 862-1030](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L862-L1030)

### JAX Compilation Path

JAX models are converted through TensorFlow as an intermediate step:

**Process:**

1.   Convert Flax model to TensorFlow using `jax2tf`
2.   Wrap with input names
3.   Get concrete function
4.   Follow TensorFlow compilation path to Relay IR

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 1032-1140](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L1032-L1140)

### PaddlePaddle Compilation Path

PaddlePaddle models use static graph tracing:

**Process:**

1.   Convert to static graph using `jit.to_static`
2.   Extract parameter names
3.   Convert to Relay IR via `from_paddle`
4.   Apply Relay passes and partitioning

**Sources:**

*   [forge/forge/tvm_calls/forge_compile.py 672-706](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/tvm_calls/forge_compile.py#L672-L706)

## Parameter and Weight Management

### Parameter Extraction by Framework

| Framework | Source | Extraction Method |
| --- | --- | --- |

### Known Limitations

1.   **JAX Support**: Limited to Flax models via `jax2tf` conversion
2.   **TensorFlow**: Requires concrete function tracing, variable handling
3.   **Data Types**: Some frameworks require dtype mapping (e.g., tf.bfloat16)
4.   **Dynamic Shapes**: Limited support, requires tracing with fixed shapes
