This page explains the coordinate systems used on Tenstorrent chips, the `OnChipCoordinate` class that unifies them, how the `Device` class performs conversions, and how coordinates are consumed by memory access operations. For API-level documentation on reading and writing memory, see [Memory Access Operations](https://deepwiki.com/tenstorrent/tt-exalens/3.3-memory-access-operations). For device-level architecture and block types, see [Device Architecture](https://deepwiki.com/tenstorrent/tt-exalens/5-device-architecture).

## Coordinate Systems

Tenstorrent chips expose multiple coordinate systems that serve different purposes. All coordinate systems describe the same physical grid of on-chip blocks, but from different perspectives.

| Name | Notation | Depends on Harvesting | Description |
| --- | --- | --- | --- |
| `noc0` | `X-Y` | No | NOC 0 routing coordinates. A difference of 1 = 1 routing hop. Wraps around at grid edges. Primary internal representation. |
| `noc1` | `X-Y` | No | NOC 1 routing coordinates. Same structure as noc0 but the second NOC, which routes in the opposite direction. |
| `die` | `X-Y` | No | Physical die coordinates. Matches layout shown in architecture documentation and datasheets. Not typically used in software. |
| `logical` | `X,Y` | Yes | Harvesting-aware coordinate. Starts at `0,0` and counts only non-harvested cores. Prefixed by core type: `t0,0` (tensix), `e0,0` (eth), `d0,0` (dram). Unprefixed defaults to tensix. |
| `translated` | `X-Y` | Yes | NOC coordinate adjusted for harvesting by compressing harvested rows to the end. Used by the NOC hardware. On Wormhole, starts at offset 16. |

**Sources:**[ttexalens/coordinate.py 1-42](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/coordinate.py#L1-L42)

The canonical internal coordinate used throughout the codebase is **noc0**. All conversions pass through noc0, and all hardware operations ultimately receive noc0 `(x, y)` pairs.

## The `OnChipCoordinate` Class

`OnChipCoordinate` ([ttexalens/coordinate.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/coordinate.py)) is the single object that represents a location on chip. It carries:

*   An internal `_noc0_coord: tuple[int, int]` — the canonical noc0 form.
*   A reference to the `_device: Device` — required for coordinate conversion lookups.
*   An optional `_core_type: str` — the type of block at this location (e.g., `tensix`, `eth`, `dram`).

**Creating a coordinate:**

`OnChipCoordinate.create(coord_str, device, coord_type=None)` parses a string into an `OnChipCoordinate`. The separator determines the coordinate system:

| Input format | Example | Assumed system |
| --- | --- | --- |
| `X-Y` | `"1-2"` | `noc0` or `translated` |
| `X,Y` | `"0,0"` | `logical` (tensix) |
| `eX,Y` | `"e0,0"` | `logical` (eth) |
| `dX,Y` | `"d0,0"` | `logical` (dram) |
| `chN` | `"ch3"` | DRAM channel number |

**Converting a coordinate:**

`OnChipCoordinate.to(output_type)` returns a tuple `(x, y)` in the requested coordinate system.

`OnChipCoordinate.to_str(output_type)` returns a formatted string.

**Changing device:**

`OnChipCoordinate.change_device(device)` re-interprets the coordinate for a different `Device` instance.

**Sources:**[ttexalens/coordinate.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/coordinate.py)[docs/ttexalens-lib-docs.md 599-660](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/docs/ttexalens-lib-docs.md?plain=1#L599-L660)

## Coordinate Conversion Architecture

The `Device` class owns the coordinate conversion tables. During construction, `Device._init_coordinate_systems()` builds two lookup dictionaries:

*   `_from_noc0: dict[(noc0_tuple, coord_system), (converted_tuple, core_type)]` — convert FROM noc0 to any other system.
*   `_to_noc0: dict[(converted_tuple, coord_system, core_type), noc0_tuple]` — convert TO noc0 from any system.

**Diagram: Coordinate conversion data flow**

**Sources:**[ttexalens/device.py 400-458](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L400-L458)

The UMD layer (`UmdDevice.convert_from_noc0`) handles `noc1`, `logical`, and `translated` conversions. Die coordinates are computed locally using the `NOC_0_X_TO_DIE_X` and `NOC_0_Y_TO_DIE_Y` class-level lists defined on each architecture-specific `Device` subclass.

## Coordinate System Population

`Device._init_coordinate_systems()` is called in `Device.__init__`. It iterates over every known block location (indexed by block type), and for each noc0 location it:

1.   Calls `UmdDevice.convert_from_noc0()` for each of `noc1`, `logical`, and `translated`.
2.   Computes the `die` coordinate using `NOC_0_X_TO_DIE_X` / `NOC_0_Y_TO_DIE_Y`.
3.   Stores both forward and reverse mappings in `_from_noc0` and `_to_noc0`.

For `noc1` and `translated`, a wildcard `core_type="any"` reverse entry is also added, since those coordinate systems don't share values between different core types.

[ttexalens/device.py 405-437](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L405-L437)

## Coordinates in Memory Access Operations

All low-level memory reads and writes ultimately operate in noc0 space. The `OnChipCoordinate` provides the `_noc0_coord` tuple directly, which is passed to UMD:

```
noc_read(location, address, size_bytes, noc_id, ...)
  → noc_x, noc_y = location._noc0_coord
  → _umd_device.noc_read(noc_id, noc_x, noc_y, address, size_bytes, ...)
```

The `noc_id` parameter (0 or 1) selects which NOC to use for routing — this is separate from the coordinate system, which is always noc0 regardless of which NOC carries the transaction.

## Safe Mode and Memory Map Validation

When `safe_mode=True` (the default), every `noc_read` and `noc_write` call passes through `Device._validate_noc_access_is_safe()` before the operation executes.

This validation:

1.   Retrieves the `NocBlock` at the target location via `location.noc_block`.
2.   Looks up the block's `noc_memory_map` (a `MemoryMap` instance).
3.   For each byte range in the request, calls `MemoryMap.find_by_noc_address()` to find the containing `MemoryMapBlockInfo`.
4.   Checks `is_accessible`, `is_safe_to_read` / `is_safe_to_write` predicates on the block info.
5.   Raises `UnsafeAccessException` if any check fails.

**Sources:**[ttexalens/device.py 248-300](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L248-L300)

### MemoryMap and MemoryMapBlockInfo

Each `NocBlock` subclass builds its `noc_memory_map` by calling `noc_memory_map.add_blocks()` with a list of `MemoryMapBlockInfo` entries.

| Class | Key fields |
| --- | --- |
| `MemoryMap` | `_noc_addresses`, `_private_addresses`, `_bar0_addresses` (interval trees); `find_by_noc_address()`, `find_by_private_address()`, `find_by_name()` |
| `MemoryMapBlockInfo` | `name`, `memory_block` (a `MemoryBlock`), `safe_to_read`, `safe_to_write`, `access_check` |
| `MemoryBlock` | `size`, `address` (a `DeviceAddress` with `noc_address`, `private_address`, `bar0_address`) |

The `safe_to_read` and `safe_to_write` fields on `MemoryMapBlockInfo` accept either a `bool` (static) or a `Callable[[int, int], bool]` (address- and size-dependent).

**Sources:**[ttexalens/memory_map.py](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/memory_map.py)

## Block Type and Coordinate Association

The `Device._noc0_to_block_type` dictionary maps each known noc0 coordinate to a block type string (e.g., `"functional_workers"`, `"eth"`, `"dram"`, `"arc"`). This is built during `_init_coordinate_systems()` by iterating over `_block_locations`.

Block locations are populated from the UMD SoC descriptor via `get_cores()` and `get_harvested_cores()`, which return coordinates in `NOC0` space. These are wrapped in `OnChipCoordinate` objects immediately.

[ttexalens/device.py 532-548](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L532-L548)

## Rendering and Display

`Device.render(axis_coordinate, cell_renderer, legend)` renders a text grid of the chip, using any coordinate system as the axis labels. Supported values: `"noc0"`, `"noc1"`, `"translated"`, `"die"`, `"logical"` (and core-type-qualified variants like `"logical-tensix"`, `"logical-eth"`, `"logical-dram"`).

Internally it calls `loc.to(axis_coordinate)` for each block location and arranges cells on a 2D grid. The CLI `device` command exposes this directly.

**Sources:**[ttexalens/device.py 596-670](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L596-L670)

The `convert_coordinate()` library function and `OnChipCoordinate.create()` both accept the following string formats:

| Format | Example | Interpreted as |
| --- | --- | --- |
| `X-Y` | `"1-2"` | noc0 or translated (detected by range) |
| `X,Y` | `"0,0"` | logical tensix |
| `tX,Y` | `"t0,0"` | logical tensix (explicit) |
| `eX,Y` | `"e0,0"` | logical eth |
| `dX,Y` | `"d0,0"` | logical dram |
| `chN` | `"ch3"` | DRAM channel index |
