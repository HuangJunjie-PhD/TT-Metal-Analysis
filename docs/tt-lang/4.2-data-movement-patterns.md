This page describes common patterns for moving data in `tt-lang` kernels, focusing on reader and writer thread implementations, streaming techniques, tensor slicing, and pipelining. These patterns are the building blocks for implementing efficient data movement operations in `@ttl.datamovement()` decorated functions.

The programming model of `tt-lang` is centered around explicit specification of data movement and compute threads and explicit synchronization between them. This allows fine-grained control of the execution schedule and its performance implications.

## Core Data Movement Abstractions

Data movement in `tt-lang` relies on several key abstractions that bridge the gap between high-level Python and hardware-specific operations:

| Entity | Description | Code Reference |
| --- | --- | --- |
| **Dataflow Buffer (DFB)** | Also known as Circular Buffer (CB). A managed memory region in L1 for inter-thread communication. | [python/sim/dfb.py 24](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L24-L24) |
| **Block** | A rectangular grid of tiles within a DFB. It is the unit of reservation, waiting, and copying. | [python/sim/dfb.py 334](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L334-L334) |
| **Tensor Slice** | A view of a subset of a tensor, used as a source or destination for copy operations. | [test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir 51](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir#L51-L51) |
| **Transfer Handle** | An object (`CopyTransaction`) returned by `ttl.copy` used to synchronize asynchronous data transfers via `wait()`. | [python/sim/copy.py 21-32](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L21-L32) |

### Data Movement Thread Roles

A standard `tt-lang` kernel typically defines three threads:

1.   **DM0 (Reader)**: Usually maps to the **BRISC** processor. Responsible for moving data from DRAM/L1 tensors into DFBs. [python/sim/blockstate.py 25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L25-L25)
2.   **Compute**: Maps to the **TRISC** processors. Performs math operations on data in DFBs. [python/sim/decorators.py 81-90](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/decorators.py#L81-L90)
3.   **DM1 (Writer)**: Usually maps to the **NCRISC** processor. Responsible for moving results from DFBs back to tensors. [python/sim/blockstate.py 25](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L25-L25)

### Implementation Mechanics

When `ttl.copy` is called from a tensor to a `Block`, the system uses a registered handler to perform the transfer [python/sim/copy.py 54-55](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L54-L55)

*   **Validation**: The `TensorToBlockHandler` ensures that the source tensor layout matches the destination block layout [python/sim/copyhandlers.py 212-219](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L212-L219)
*   **Indexing**: For multi-tile transfers, the handler iterates through the requested tile range in the source tensor and populates the destination block [python/sim/copyhandlers.py 209-211](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L209-L211)
*   **State Machine**: Calling `copy` marks the block as a destination, preventing other threads from accessing it until `wait()` is called [python/sim/copy.py 64-68](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L64-L68)
*   **Hardware Mapping**: At the MLIR level, `ttl.copy` is lowered to `noc_async_read_tile` calls in the generated C++ code [test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir 41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir#L41-L41)

### Implementation Mechanics

*   **Handler**: Uses `BlockToTensorHandler` to move data from L1 blocks back to tensors [python/sim/copyhandlers.py 31](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L31-L31)
*   **Synchronization**: The `wait()` call on the `CopyTransaction` ensures the transfer is complete before the block is popped [python/sim/copy.py 119-121](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copy.py#L119-L121)
*   **Access Control**: The source block is locked against writes while the copy is in flight [python/sim/blockstate.py 118-124](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L118-L124)

## Efficient Data Movement Techniques

### Multi-Tile Transfers and Loop Generation

When copying multi-tile tensors (e.g., 2x2 tiles), the compiler automatically generates nested loops in the backend C++ to handle individual tile transfers [test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir 9-10](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir#L9-L10)

*   **Loop Bounds**: The generated tile loops (e.g., `for tile_y in 0..2`) are determined by the tensor layout, not the CB shape [test/ttlang/Translate/TTLToCpp/dma_multi_tile_same_layout_different_cb.mlir 13-14](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_same_layout_different_cb.mlir#L13-L14)
*   **Tile Indexing**: Tile offsets are calculated using row-major ordering: `tile_y * tiles_x + tile_x`[test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir 10](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir#L10-L10)
*   **User Loops vs. Tile Loops**: User-defined loops (e.g., `scf.for`) wrap these generated tile loops. Multiple multi-tile copies within a user loop will generate sequential tile loop nests unless fused [test/ttlang/Translate/TTLToCpp/dma_loop_multi_tile_nontrivial_cb.mlir 13-27](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_loop_multi_tile_nontrivial_cb.mlir#L13-L27)

### Tensor Slicing

`tt-lang` supports slicing syntax to define sub-regions for `ttl.copy`.

*   **Single Tile**: `tensor[0, 0]` selects one 32x32 tile [test/python/test_tensor_slice.py 189-195](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_tensor_slice.py#L189-L195)
*   **Tile Block**: `tensor[r1:r2, c1:c2]` selects a rectangular block of tiles [test/python/test_large_dram_streaming.py 67-70](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_large_dram_streaming.py#L67-L70)
*   **MLIR Lowering**: Slices are represented as `ttl.tensor_slice` operations which provide the base offsets for the `TensorAccessor`[test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir 51-52](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_read.mlir#L51-L52)

### Double-Buffering and Pipelining

To overlap data movement with computation, `block_count=2` is typically used when creating a DFB [test/python/test_elementwise_shapes.py 158](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L158-L158)

*   **Pipelining Pattern**: A common pattern is to issue a copy for the _next_ iteration before waiting for the _current_ iteration to finish [test/ttlang/Translate/TTLToCpp/dma_loop_single_tile.mlir 43-46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_loop_single_tile.mlir#L43-L46)
*   **Transfer Handles**: `ttl.copy` returns a handle. Synchronization via `ttl.wait` on these handles is lowered to `noc_async_read_barrier` or `noc_async_write_barrier` in C++ [test/ttlang/Translate/TTLToCpp/dma_single_tile_read.mlir 20-21](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_single_tile_read.mlir#L20-L21)

### Batching and Barrier Deduplication

When multiple `ttl.copy` operations are issued sequentially without intermediate `wait` calls, the compiler can batch them [test/ttlang/Translate/TTLToCpp/dma_multi_tile_batched_in_user_loop.mlir 115-120](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_batched_in_user_loop.mlir#L115-L120)

*   **Deduplication**: Consecutive barriers in the IR are deduplicated to a single `noc.async_read_barrier<Noc::BarrierMode::FULL>()` in the final code [test/ttlang/Translate/TTLToCpp/dma_multi_tile_batched_in_user_loop.mlir 96-97](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_multi_tile_batched_in_user_loop.mlir#L96-L97)
*   **Shared Barriers**: A single barrier can wait for all previously issued asynchronous NOC operations [test/ttlang/Translate/TTLToCpp/dma_batched_single_tile.mlir 29-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang/Translate/TTLToCpp/dma_batched_single_tile.mlir#L29-L30)
