## Purpose and Scope

This page covers NOC (Network-on-Chip) traffic analysis and DMA performance profiling for `tt-lang` kernels. It explains how to measure memory bandwidth, identify transfer bottlenecks, and analyze data movement patterns between DRAM and L1 memory across cores using the `perf_summary.py` tool.

For line-by-line cycle profiling, see [7.1 Auto-Profiling System](https://github.com/tenstorrent/tt-lang/blob/d76e6233/7.1%20Auto-Profiling%20System) For user-defined signpost zones, see [7.3 Signpost Profiling](https://github.com/tenstorrent/tt-lang/blob/d76e6233/7.3%20Signpost%20Profiling) For systematic optimization workflows using these tools, see [7.5 Optimization Workflow](https://github.com/tenstorrent/tt-lang/blob/d76e6233/7.5%20Optimization%20Workflow)

## Overview

NOC and DMA performance analysis focuses on understanding data movement costs. The Network-on-Chip handles all data transfers between DRAM, L1 memory, and compute cores. Analyzing NOC traffic reveals:

*   **DRAM bandwidth utilization**: How efficiently kernels use available memory bandwidth.
*   **Transfer patterns**: Size and frequency of DMA operations.
*   **L1 communication**: Inter-core data movement via pipes or shared memory.
*   **Locality Split**: The ratio of local vs. remote L1 accesses (crucial for sharded tensors).
*   **Bottlenecks**: Whether kernels are memory-bound or compute-bound.

Unlike cycle-level profiling which focuses on individual operations, NOC analysis provides a system-level view of data movement across the entire grid.

## Performance Dump Infrastructure

### Environment Variables

NOC profiling requires setting environment variables before kernel execution to enable hardware event capture and the tt-lang analysis script:

| Variable | Description |
| --- | --- |
| `TT_METAL_HOME` | Path to tt-metal (required) |
| `TT_METAL_DEVICE_PROFILER=1` | Enable device profiler |
| `TT_METAL_DEVICE_PROFILER_NOC_EVENTS=1` | Enable NOC event tracing (required for perf summary) |
| `TT_METAL_PROFILER_MID_RUN_DUMP=1` | Enable mid-run profiler dumps |
| `TTLANG_PERF_DUMP=1` | Print NOC traffic and per-kernel wall time summary |
| `TTLANG_PROFILE_CSV` | Optional custom path for profile results |

### Metrics Tracked

The analysis infrastructure tracks several critical metrics to determine kernel efficiency:

*   **Traffic Metrics**: DRAM bytes read/written, L1 bytes read/written.
*   **Effective Bandwidth**: Total payload divided by duration.
*   **Kernel Time**: Cycle counts for each RISC processor (BRISC, NCRISC, TRISC).
*   **Gather/Multicast Cost**: In `ksplit` matmuls, the `reduce_net` traffic overhead is measured.
*   **L1 Footprint**: Total SRAM usage per core vs. the budget (approx 1.44 MiB on Wormhole).

## Bandwidth Analysis

### Effective Bandwidth Calculation

Effective bandwidth measures payload throughput. It is calculated as: `effective BW = total payload / duration`

Efficiency is heavily influenced by transfer size; larger uniform sizes generally yield higher efficiency. In matmul kernels, block sizes (`bm`, `bn`, `bk`) determine the payload per DMA.

**Calibration findings:**

*   **Block Overhead**: Represented as `BLOCK_OVERHEAD_ALPHA = 64` tile-matmul units. Smaller blocks (low `bv`) are penalized by this constant overhead.
*   **Gather Penalty**: Represented as `KP_PENALTY_BETA = 0.035` per gather event. This penalty scales with the total number of gather events per core (`(Kp-1) * iter_per_core`).

## Common Performance Bottlenecks

Analysis of `ksplit` and `SUMMA` matmuls revealed several recurring bottlenecks:

| Symptom | Diagnosis | Fix |
| --- | --- | --- |
| **Regressed Wall Time (>8 dims)** | Sub-block decomposition producing lopsided blocks (e.g., 10 -> 8+2) | Keep `bm`, `bn`, `bk`<= 8 or use exact power-of-two multiples. |
| **High Gather Overhead** | Too many `Kp` parts relative to compute iterations | Reduce `Kp` or increase `iter_per_core` to amortize gather cost. |
| **PCC Failures (NaN)** | Accumulation timing or synchronization bugs in long-running kernels | Investigate `L1_ACC` loop annotations and barrier placement. |
| **NOC Contention** | Horizontal A-mcast and reduce-net traffic contending on NOC0 | Balance traffic by moving reduce-net or B-mcast to NOC1 (dm_write). |
| **L1 Budget Exhaustion** | Circular Buffer footprint exceeds ~1.44 MiB | Reduce `block_count` or shrink tile-block dimensions. |

## Multi-Core Performance Considerations

When analyzing performance on a grid, the summary aggregates data across all active nodes.

1.   **Grid Efficiency**: Throughput is roughly proportional to `cores / pad`. Padding waste occurs when tensor dimensions are not multiples of the block/grid size. `MAX_PAD` is typically capped at 1.25.
2.   **K-Split Scaling**: For large K dimensions (e.g., 2k x 8k x 2k), splitting the reduction across multiple cores (`Kp > 1`) can improve performance by shrinking each core's K-slab, provided the tall K amortizes the gather overhead.
3.   **Mcast Geometry**: The `ksplit_kernel` uses row-multicast for activations (A) and column-multicast for weights (B). Each A block is read once per row and each B block once per column.
