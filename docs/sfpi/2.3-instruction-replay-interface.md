# Instruction Replay Interface

Relevant source files
*   [include/lltt.h](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h)
*   [include/sfpi_fp16.h](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h)

## Purpose and Scope

The Instruction Replay Interface provides low-level access to the SFPU instruction recording and replay mechanism. This interface allows developers to record sequences of SFPU instructions into a hardware buffer and replay them efficiently, enabling optimized loop execution and repeated operation patterns. The interface is available exclusively on Wormhole and Blackhole architectures.

This page documents the `lltt` namespace and its functions for instruction replay management. For higher-level SFPI programming constructs, see [Core SFPI Programming Interface](https://deepwiki.com/tenstorrent/sfpi/2-core-sfpi-programming-interface). For platform-specific hardware intrinsics, see [Hardware Abstraction Layers](https://deepwiki.com/tenstorrent/sfpi/3-hardware-abstraction-layers).

**Sources:**[include/lltt.h 1-44](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L1-L44)

## Platform Availability

The instruction replay interface is conditionally compiled based on target architecture. It is only available on platforms with hardware replay support.

| Architecture | Availability | Preprocessor Macro |
| --- | --- | --- |
| Wormhole | ✓ Available | `__riscv_xtttensixwh` |
| Blackhole | ✓ Available | `__riscv_xtttensixbh` |
| Grayskull | ✗ Not Available | `__riscv_xtttensixgs` |

The entire `lltt.h` header is guarded by the condition `#if __riscv_xtttensixwh || __riscv_xtttensixbh`, ensuring it only compiles for supported architectures.

**Sources:**[include/lltt.h 11-12](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L11-L12)

## The Instruction Buffer

### Buffer Declaration

The instruction replay system uses a special hardware-backed buffer declared at file scope:

`extern volatile uint32_t __instrn_buffer[];`
This buffer is marked `volatile` to prevent compiler optimizations that could interfere with hardware operations. The buffer is externally linked and must be provided by the runtime environment.

**Sources:**[include/lltt.h 18](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L18-L18)

### Buffer Pointer with Hardware Annotation

Within the `lltt` namespace, a constexpr pointer to the instruction buffer is declared with a special GCC attribute:

`constexpr inline volatile uint32_t *[[gnu::rvtt_reg_ptr]] __instrn_buffer = ::__instrn_buffer;`
The `[[gnu::rvtt_reg_ptr]]` attribute indicates to the compiler that this pointer refers to a hardware register space or special memory-mapped region, enabling proper code generation for instruction replay operations.

**Sources:**[include/lltt.h 21](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L21-L21)

## API Functions

### Execution Mode Enumeration

The `ExecBool` enum provides type-safe execution mode selection:

`enum ExecBool : bool {NoExec, Exec};`
| Value | Meaning |
| --- | --- |
| `NoExec` | Record instructions without executing them |
| `Exec` | Execute instructions while recording them |

**Sources:**[include/lltt.h 23](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L23-L23)

### record() Function

The `record()` function template captures SFPU instructions into the hardware buffer for later replay.

**Function Behavior:**

*   Records SFPU instructions starting at offset `start` in the instruction buffer
*   Captures `length` instructions
*   Template parameter `E` controls whether instructions execute during recording 
    *   `lltt::NoExec` (default): Record only, do not execute
    *   `lltt::Exec`: Execute while recording

*   Marked `[[gnu::always_inline]]` to ensure zero overhead

**Sources:**[include/lltt.h 25-29](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L25-L29)

### replay() Function

The `replay()` function executes previously recorded instruction sequences from the hardware buffer.

`[[gnu::always_inline]] inline void replay(unsigned start, unsigned length)`
**Function Behavior:**

*   Replays `length` instructions starting at offset `start` in the buffer
*   Does not record new instructions (recording disabled)
*   Does not re-execute instructions that were already executed during recording
*   Always inlined for optimal performance

**Parameters:**

*   `start`: Buffer offset where the instruction sequence begins
*   `length`: Number of instructions to replay

**Sources:**[include/lltt.h 31-33](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L31-L33)

### replay_insn() Function

The `replay_insn()` function generates the encoded instruction value for a replay operation without executing it.

`[[gnu::always_inline]] constexpr std::uint32_treplay_insn(unsigned start, unsigned length)`
**Encoding Format:**

| Bits | Field | Value |
| --- | --- | --- |
| 31-24 | Opcode | `0x04` |
| 23-14 | Start Offset | `start << 14` |
| 13-4 | Length | `length << 4` |
| 3-0 | Reserved | 0 |

The encoding formula is:

```
(0x04 << 24) | (start << 14) | (length << 4)
```

**Use Cases:**

*   Manual instruction generation for advanced control flow
*   Integration with custom instruction scheduling
*   Direct hardware instruction buffer manipulation

**Sources:**[include/lltt.h 35-39](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L35-L39)

## Builtin Interface

### Macro Wrapper

The header defines a macro wrapper around the compiler builtin to simplify parameter passing:

`#define __builtin_rvtt_ttreplay(START, LENGTH, EXEC, RECORD) \  __builtin_rvtt_ttreplay(lltt::__instrn_buffer, LENGTH, 0, 0, START, EXEC, RECORD)`
This macro automatically injects the instruction buffer pointer and placeholder parameters, reducing the parameter count at call sites.

**Sources:**[include/lltt.h 15-16](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L15-L16)

### Builtin Function Parameters

The underlying `__builtin_rvtt_ttreplay()` intrinsic accepts the following parameters:

| Position | Parameter | Type | Purpose |
| --- | --- | --- | --- |
| 1 | buffer | `volatile uint32_t*` | Instruction buffer pointer |
| 2 | length | `unsigned` | Number of instructions |
| 3-4 | (unused) | - | Reserved parameters |
| 5 | start | `unsigned` | Buffer offset |
| 6 | exec | `bool` | Execute flag |
| 7 | record | `bool` | Record flag |

**Sources:**[include/lltt.h 15-16](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L15-L16)[include/lltt.h 28](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L28-L28)[include/lltt.h 32](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L32-L32)

## Usage Patterns

### Recording and Replay Flow

**Sources:**[include/lltt.h 25-33](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L25-L33)

### Typical Usage Scenarios

#### Scenario 1: Loop Optimization

Record the body of a loop once, then replay it multiple times without instruction decode overhead.

#### Scenario 2: Repeated Pattern Application

Capture a common operation pattern (e.g., activation function) and apply it to multiple data blocks.

#### Scenario 3: Execute-While-Record

Use `lltt::record<lltt::Exec>()` to execute instructions normally while simultaneously capturing them for later replay.

## Integration with SFPI

The instruction replay interface operates at a lower level than the main SFPI API documented in `sfpi.h`. While `sfpi.h` provides high-level vector operations and type wrappers, `lltt.h` gives direct control over instruction recording and playback.

### Relationship to Other SFPI Components

The replay interface can be used to optimize any SFPI code, regardless of whether it uses high-level types or low-level intrinsics.

**Sources:**[include/lltt.h 7-8](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L7-L8)

## Implementation Details

### Always Inline Attribute

All API functions are marked with `[[gnu::always_inline]]` to ensure they are inlined at all optimization levels. This is critical for performance, as the function call overhead would negate the benefits of instruction replay.

**Sources:**[include/lltt.h 26](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L26-L26)[include/lltt.h 31](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L31-L31)[include/lltt.h 35](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L35-L35)

### Constexpr Encoding

The `replay_insn()` function is marked `constexpr`, allowing the compiler to compute instruction encodings at compile time when given constant arguments. This enables zero-overhead instruction generation in performance-critical paths.

**Sources:**[include/lltt.h 35-39](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L35-L39)

### Namespace Organization

The `lltt` namespace name suggests "Low Level TensTorrent", indicating its position as a foundational interface layer beneath the higher-level SFPI abstractions.

**Sources:**[include/lltt.h 20](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L20-L20)[include/lltt.h 41](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/lltt.h#L41-L41)
