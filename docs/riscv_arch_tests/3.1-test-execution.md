# Test Execution

Relevant source files
*   [infra/quals.py](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/quals.py)
*   [infra/smoke.py](https://github.com/tenstorrent/riscv_arch_tests/blob/e54223b0/infra/smoke.py)

This document describes the test execution infrastructure that runs RISC-V architectural tests against instruction set simulators (ISS). The system provides both individual test execution and comprehensive smoke testing capabilities through two main Python scripts: `quals.py` for targeted test execution and `smoke.py` for automated comprehensive testing across all test suites.

For information about test list generation, see [Test List Generation](https://deepwiki.com/tenstorrent/riscv_arch_tests/3.2-test-list-generation). For details about simulator configurations, see [Whisper Configuration](https://deepwiki.com/tenstorrent/riscv_arch_tests/4.1-whisper-configuration) and [Spike Integration](https://deepwiki.com/tenstorrent/riscv_arch_tests/4.2-spike-integration).

## Core Components

The test execution system is built around several key classes that handle different aspects of test management and execution:

### Test Management Classes

The `QualTest` class represents an individual test case, containing a test name and associated runner. Each test tracks its execution result using the `PassFailEnum` enumeration that defines `PASS` and `FAILED` states.

The central `Runner` class orchestrates test execution by parsing test list files, managing ISS runners, and coordinating test compilation when needed. It supports configurable sampling rates, logging directories, and can work with both pre-compiled binaries and source assembly files.

### ISS Runner Classes

The system includes specialized runner classes for different instruction set simulators:

*   `WhisperRunner` - Executes tests using the Whisper ISS with configurable JSON configuration files
*   `SpikeRunner` - Executes tests using the Spike ISS with dynamically generated command-line arguments
*   `ISSRunner` - Base class providing common ISS execution functionality

### Test Compilation

The `TestCompiler` class handles on-demand compilation of assembly test files using `riscv64-unknown-elf-gcc` with appropriate RISC-V extensions and linking scripts.

## Individual Test Execution

Individual test execution is handled by the `quals.py` script, which can run a single test list file against a specified ISS. The script supports several key features:

### Command Line Interface

The script accepts the following primary arguments:

*   `--quals_file` - Path to the test list file (required)
*   `--iss` - Override ISS selection (whisper/spike)
*   `--whisper_path` / `--spike_path` - Custom ISS binary paths
*   `--pre-compiled` - Skip compilation and use existing binaries
*   `--track_test_num` - Display test progress during execution

### Test List Parsing

Test list files use a shell-like syntax where each line specifies test parameters:

```
name=test_name tool=whisper test=path/to/test.S opts=default
name=another_test tool=spike test=path/to/test.S opts="custom options"
```

The parser extracts test name, tool selection, test file path, execution options, and compilation flags from each line using `shlex.split()` for proper shell-like parsing.

### Result Reporting

Test results are presented in formatted tables using the `PrettyTable` library. Successful runs show only test names and PASS status, while failed runs include log file paths for debugging.

## Smoke Testing

The smoke testing functionality in `smoke.py` provides comprehensive automated testing across all test suites in the repository:

### Automated Test Discovery

The smoke tester uses a recursive generator function `suffix_file_generator()` to discover all `.list` files throughout the repository structure. This ensures comprehensive coverage without manual test list maintenance.

### Sampling Strategy

To balance coverage with execution time, smoke testing runs only 5% of tests from each test list (configurable via the `sample` parameter). This provides broad coverage across all test categories while maintaining reasonable execution times.

### Multi-ISS Testing

Each discovered test list is executed against both Spike and Whisper ISS, providing cross-validation of test results and ensuring compatibility across different simulation platforms.

## ISS Configuration and Execution

### Whisper Configuration

Whisper execution uses JSON configuration files to define ISS behavior. The system automatically selects appropriate configuration files based on test characteristics:

*   Default configuration: `infra/whisper_config.json`
*   Vector length 128: `infra/whisper_config_vlen_128.json`
*   Vector length 256: Uses default configuration

Whisper tests execute with standard options including maximum instruction limits, CSV output formatting, and large memory configurations.

### Spike Configuration

Spike execution uses dynamically generated command-line arguments based on test file paths and characteristics. The system automatically determines:

*   ISA string based on file path (e.g., `RV64IMAFDC` with extensions)
*   Privilege mode configuration (typically `msu` for maximum compatibility)
*   Vector architecture parameters when vector tests are detected
*   Memory alignment requirements based on test location

## Test Compilation Process

When tests are not pre-compiled, the system automatically compiles assembly source files using the RISC-V GCC toolchain:

### Compilation Configuration

The `TestCompiler` class uses `riscv64-unknown-elf-gcc` with default flags optimized for RISC-V architectural testing:

*   `-march=rv64gcv_zba_zbb_zbc_zbs` - Full ISA with common extensions
*   `-mabi=lp64` - 64-bit ABI
*   `-Wall -Werror` - Strict warning and error handling
*   `-O2` - Optimization level 2
*   `-nostartfiles` - Bare metal execution environment

### Build Process

Compilation requires matching `.S` assembly files and `.ld` linker script files. The compiler creates output binaries in a dedicated build directory, skipping compilation if binaries already exist to improve performance on repeated runs.

### Error Handling

Compilation failures are captured and reported, allowing the test execution to continue with remaining tests while providing detailed error information for debugging.
