## Purpose and Scope

This document describes the automated dependency management system for TTNN (Tenstorrent Neural Network) libraries and wheels. The system automatically fetches the latest TTNN pre-releases, updates dependency specifications across multiple configuration files, synchronizes git submodules, builds C++ extensions, and creates auto-merging pull requests. This automation ensures that the pytorch2.0_ttnn repository stays synchronized with the latest TTNN releases without manual intervention.

For information about the general CI/CD workflow orchestration, see [9.1](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.1-github-actions-workflows-overview). For details on Docker container infrastructure used during builds, see [9.4](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.4-docker-container-infrastructure). For development environment setup that consumes these dependencies, see [9.5](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.5-development-environment-setup).

## Overview of TTNN Dependency Lifecycle

The TTNN library serves as the hardware abstraction layer for Tenstorrent devices. The pytorch2.0_ttnn project depends on specific TTNN versions, which are distributed as Python wheels through an internal PyPI repository and GitHub releases. The dependency management system maintains synchronization between these TTNN versions and the compilation backend.

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 1-121](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L1-L121)

## TTNN Dependency Sources

### Internal PyPI Repository

The primary source for TTNN wheels is an internal PyPI server at `https://pypi.eng.aws.tenstorrent.com/ttnn/`. This repository contains pre-release versions built from the tt-metal repository's development branches.

The workflow queries this repository to identify the latest version:

`# Query internal PyPI and extract latest cp310 wheellatest_pre_release=$(curl -s $repo_url | sed -n 's/.*href="\([^"]*\).*/\1/p' | grep cp310 | sort -V | tail -n 1)`
**Sources:**[.github/workflows/update-ttnn-wheel.yaml 22-28](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L22-L28)

### GitHub Releases

For public distribution and setup.py/pyproject.toml specifications, the workflow references TTNN wheels hosted on GitHub releases in the `tenstorrent/tt-metal` repository. The URL format is:

```
https://github.com/tenstorrent/tt-metal/releases/download/v{version}/ttnn-{version}-cp310-cp310-linux_x86_64.whl
```

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 40-49](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L40-L49)

### Version Format Transformation

TTNN versions follow the format `X.Y.Z.devN+gHASH-rcM`. The workflow performs transformations between different version representations:

| Source Format | Transformation | Target Format | Usage |
| --- | --- | --- | --- |
| `0.0.1.dev123+gabcdef-rc1` | Remove `.dev`, change `+g` to `-g` | `v0.0.1-123-gabcdef-rc1` | Git submodule branch |
| `0.0.1.dev123+gabcdef-rc1` | Change `-rc` to `rc` | `0.0.1.dev123+gabcdefrc1` | GitHub release URL |
| `0.0.1.dev123+gabcdef-rc1` | No transformation | `0.0.1.dev123+gabcdef-rc1` | Internal PyPI URL |

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 27-114](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L27-L114)

## Dependency File Updates

### requirements.txt Update Strategy

The workflow removes any existing TTNN entries and appends the new version with Python version constraints:

`# Remove existing ttnn linessed -i '/^ttnn @ /d' requirements.txt # Append new version for Python 3.10echo "ttnn @ $repo_url$release ; python_version==\"3.10\"" >> requirements.txt`
The resulting entry format is:

```
ttnn @ https://pypi.eng.aws.tenstorrent.com/ttnn/{wheel_file} ; python_version=="3.10"
```

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 30-36](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L30-L36)

### setup.py and pyproject.toml Updates

Both files use GitHub release URLs with platform-specific constraints:

`ttnn_url="ttnn @ https://github.com/tenstorrent/tt-metal/releases/download/v${latest_version}/ttnn-$latest_version_short-cp310-cp310-linux_x86_64.whl"platform_constraint=" ; python_version=='3.10' and platform_system=='Linux' and platform_machine=='x86_64'" # Update setup.pysed -i "s|ttnn @ https://github.com/tenstorrent/tt-metal/releases/download/v[0-9.]*-rc[0-9]*/ttnn-[0-9.]*rc[0-9]*-cp310-cp310-linux_x86_64.whl.*|$ttnn_url$platform_constraint|g" setup.py # Update pyproject.tomlsed -i "s|ttnn @ https://github.com/tenstorrent/tt-metal/releases/download/v[0-9.]*-rc[0-9]*/ttnn-[0-9.]*rc[0-9]*-cp310-cp310-linux_x86_64.whl.*|$ttnn_url$platform_constraint|g" pyproject.toml`
The constraints ensure the TTNN wheel is only installed on compatible platforms (Linux x86_64 with Python 3.10).

## Pull Request Automation

### PR Creation with peter-evans/create-pull-request

The workflow uses the `peter-evans/create-pull-request@v7` action to atomically create PRs with the updated dependency files:

`- name: Create Pull Request  uses: peter-evans/create-pull-request@v7  id: create-pr  with:    branch: update-dependency    branch-suffix: timestamp    committer: github-actions[bot] <41898282+github-actions[bot]@users.noreply.github.com>    author: ${{ github.actor }} <${{ github.actor_id }}+${{ github.actor }}@users.noreply.github.com>    base: main    commit-message: "Update dependencies to ${{ steps.fetch_release.outputs.release }}"    title: "Update TT-NN to ${{ steps.fetch_release.outputs.release_short }}"    body: "This PR updates TT-NN wheel to the latest pre-release version."    labels: ttnn-wheel-update    delete-branch: true    token: ${{ secrets.GH_TOKEN }}`
**Branch naming:** The branch is named `update-dependency-{timestamp}` to avoid conflicts with concurrent updates.

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 51-65](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L51-L65)

### Auto-Approval and Auto-Merge

The workflow implements a fully automated approval and merge process:

**Approval step:**

`gh pr review ${{ steps.create-pr.outputs.pull-request-number }} --approve`
**Auto-merge step:**

`gh pr merge --merge --auto "${{ steps.create-pr.outputs.pull-request-number }}"`
The `--auto` flag enables GitHub's auto-merge feature, which waits for:

*   Required status checks to pass (from `before_merge.yaml` workflow)
*   Branch protection rules to be satisfied
*   No merge conflicts

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 67-80](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L67-L80)

## Integration with Testing Pipeline

### before_merge.yaml Workflow Trigger

When the auto-merge PR is added to the merge queue, the `before_merge.yaml` workflow is triggered:

`name: "Before Merge" on:  merge_group: jobs:  run-tests:    uses: ./.github/workflows/run-tests.yaml    secrets: inherit    with:      docker_tag: 'ghcr.io/tenstorrent/pytorch2.0_ttnn/ubuntu-22.04-amd64:latest'      commit_report: 'None'`
This workflow invokes the comprehensive test suite to validate the new TTNN version.

## Dependency Update Flow Summary

The complete dependency update lifecycle:

**Critical path timing:**

1.   **Version Detection:** 5-10 seconds (curl + sed parsing)
2.   **File Updates:** 1-2 seconds (sed operations)
3.   **PR Creation:** 5-10 seconds (GitHub API)
4.   **C++ Build:** 10-20 minutes (parallel, non-blocking)
5.   **Test Validation:** 60-120 minutes (full test suite)
6.   **Auto-merge:** Immediate after tests pass

**Sources:**[.github/workflows/update-ttnn-wheel.yaml 1-121](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/update-ttnn-wheel.yaml#L1-L121)[.github/workflows/before_merge.yaml 1-27](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/before_merge.yaml#L1-L27)[.github/workflows/run-tests.yaml 1-469](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L1-L469)

The workflow requires elevated permissions:

## Error Handling and Recovery

### Workflow Failure Scenarios

| Failure Point | Behavior | Recovery |
| --- | --- | --- |
| PyPI unavailable | Workflow fails, no PR created | Manual retry when service restored |
| Version parsing error | Workflow fails at sed step | Fix parsing logic, rerun |
| PR creation failure | No PR, no merge attempt | Check GH_TOKEN permissions |
| C++ build failure | PR still created, build job fails | Non-blocking, can merge if tests pass |
| Test failures | Auto-merge blocked | PR remains open for investigation |

The C++ extension build job is independent of the PR creation path, so build failures do not prevent dependency updates if the PR can be tested and merged successfully.
