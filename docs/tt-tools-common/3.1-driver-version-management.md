# Driver Version Management

## Purpose and Scope

This page documents the driver version management system in tt-tools-common, which provides detection, parsing, and validation of the Tenstorrent kernel driver (tt-kmd) version. The system ensures that operations requiring specific driver capabilities only execute when the installed driver meets minimum version requirements. This prevents runtime failures and provides clear error messages when version requirements are not met.

For information about the overall reset system that depends on driver version validation, see [Hardware Reset System](https://deepwiki.com/tenstorrent/tt-tools-common/2-hardware-reset-system). For details on host information gathering and compatibility checking, see [Host Information and Compatibility](https://deepwiki.com/tenstorrent/tt-tools-common/3.2-host-information-and-compatibility).

* * *

## System Overview

The driver version management system consists of four primary components:

1.   **Version Detection**: Reading the current driver version from the kernel module
2.   **Version Parsing**: Converting version strings into comparable numeric tuples
3.   **Version Comparison**: Determining if current version meets minimum requirements
4.   **Validation & Enforcement**: Checking requirements and failing gracefully with user guidance

The system supports semantic versioning (SemVer) with extensions for pre-release identifiers and build metadata commonly used in driver versioning schemes.

## Semantic Version Parsing

### Version String Format Support

The `_parse_version_string()` function handles multiple version string formats according to semantic versioning conventions:

**Semantic Version Parsing Pipeline**

### Parsing Rules

| Rule | Description | Example Input | Extracted Core | Output Tuple |
| --- | --- | --- | --- | --- |
| **Standard** | Three numeric parts | `"1.34.0"` | `"1.34.0"` | `(1, 34, 0)` |
| **Short** | Two or one parts | `"1.34"` | `"1.34"` | `(1, 34, 0)` |
| **Pre-release** | Version with `-identifier` | `"1.34.1-alpha"` | `"1.34.1"` | `(1, 34, 1)` |
| **Build metadata** | Version with `+metadata` | `"1.2.3+build456"` | `"1.2.3"` | `(1, 2, 3)` |
| **Combined** | Both pre-release and build | `"1.4.0-rc1+build42"` | `"1.4.0"` | `(1, 4, 0)` |
| **Extraversion** | Hyphenated extension | `"1.28-bh"` | `"1.28"` | `(1, 28, 0)` |

### Error Handling

The parser raises `ValueError` in the following cases:

*   Empty version string
*   Empty string after splitting
*   Non-integer version components

## Version Comparison Logic

### Comparison Function

The `is_driver_version_at_least()` function performs version comparison using Python's lexicographic tuple comparison:

**Version Comparison Decision Flow**

### Comparison Examples

| Current Version | Minimum Required | Current Tuple | Minimum Tuple | Result | Explanation |
| --- | --- | --- | --- | --- | --- |
| `"1.34.0"` | `"1.26.0"` | `(1, 34, 0)` | `(1, 26, 0)` | `True` | Current is newer |
| `"1.26.0"` | `"1.26.0"` | `(1, 26, 0)` | `(1, 26, 0)` | `True` | Versions match exactly |
| `"1.25.9"` | `"1.26.0"` | `(1, 25, 9)` | `(1, 26, 0)` | `False` | Current is older (minor) |
| `"2.0.0"` | `"1.35.0"` | `(2, 0, 0)` | `(1, 35, 0)` | `True` | Major version bump |
| `"1.26.0-alpha"` | `"1.26.0"` | `(1, 26, 0)` | `(1, 26, 0)` | `True` | Pre-release stripped |

Python's tuple comparison works lexicographically (left-to-right), which correctly implements semantic version comparison when tuples are ordered as `(major, minor, patch)`.

## Driver Version Validation

### Validation and Enforcement

The `check_driver_version()` function combines detection, parsing, and comparison into a single validation operation with automatic error handling:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `operation` | `str` | Required | Name of operation requiring driver version check |
| `minimum_required_version_str` | `str` | `"1.26.0"` | Minimum acceptable driver version |

**Behavior:**

1.   Calls `get_driver_version()` to detect current version
2.   Calls `is_driver_version_at_least()` to compare versions
3.   Returns silently if version is sufficient
4.   Prints red error messages and calls `sys.exit(1)` if insufficient or parsing fails

### Error Messages

The function provides three categories of error messages:

**1. Parsing Error:**

```
Error parsing driver version. {exception message}
```

**2. Insufficient Version:**

```
Current driver version: {current_version}
Operation '{operation}' requires Tenstorrent driver version {minimum_version} or greater. Your version is too old.
Please update your driver using tt-kmd: https://github.com/tenstorrent/tt-kmd
```

**3. No Driver Detected:**

```
No Tenstorrent driver detected! Please install the driver using tt-kmd: https://github.com/tenstorrent/tt-kmd
```

All error messages use `CMD_LINE_COLOR.RED` for terminal output visibility.

## Integration with Reset System

### Minimum Version Requirements

The reset system defines minimum driver version requirements as module-level constants:

| Constant | Value | Purpose |
| --- | --- | --- |
| `MINIMUM_DRIVER_VERSION_LDS_RESET` | `"1.26.0"` | Required for performing LDS (Link/Device/System) reset operations |

This constant is used as the default parameter value for `check_driver_version()`.

### Reset Integration Points

**Driver Version Check Integration in Reset System**

The reset system performs version checks at multiple levels:

1.   **Pre-flight validation**: Before executing any reset operation
2.   **Conditional routing**: To determine if modern `ChipReset` implementation can be used (requires driver >= 2.4.1)
3.   **Operation gating**: To ensure minimum version 1.26.0 for basic reset functionality

Sources: [tt_tools_common/utils_common/system_utils.py 17](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/system_utils.py#L17-L17)[CHANGELOG.md 103](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L103-L103)

## Usage Patterns

### Pattern 1: Pre-flight Validation

Used at the start of operations that require specific driver versions:

`# In reset operationfrom tt_tools_common.utils_common.system_utils import check_driver_version def perform_reset():    # Validate driver version, exit if insufficient    check_driver_version("reset", "1.26.0")        # Continue with reset operations...`
### Pattern 2: Conditional Feature Enablement

Used to determine which code path to execute based on driver capabilities:

`from tt_tools_common.utils_common.system_utils import (    get_driver_version,     is_driver_version_at_least) driver_version = get_driver_version()if is_driver_version_at_least(driver_version, "2.4.1"):    # Use modern ChipReset implementation    use_modern_reset_path()else:    # Fall back to legacy implementation    use_legacy_reset_path()`
### Pattern 3: Information Display

Used in diagnostic tools and compatibility menus:

`from tt_tools_common.utils_common.system_utils import get_driver_version def show_system_info():    driver_version = get_driver_version()    if driver_version:        print(f"TT-KMD {driver_version}")    else:        print("Driver not installed")`

## Version History and Evolution

### Critical Version Milestones

| Driver Version | Significance |
| --- | --- |
| `1.26.0` | Minimum version for LDS reset operations |
| `2.4.1` | Enables modern `ChipReset` unified implementation for Blackhole and Wormhole |

### Bug Fixes and Improvements

| Release | Issue | Fix |
| --- | --- | --- |
| **1.4.7** | Crash when parsing driver versions with extraversion field (e.g., `"1.28-bh"`) | Enhanced `_parse_version_string()` to strip pre-release identifiers before parsing |
| **1.4.0** | Driver version requirement not enforced for reset operations | Added `check_driver_version()` call with minimum version 1.26.0 |

The extraversion field fix in release 1.4.7 was particularly important as it enabled the system to handle Blackhole-specific driver builds that append chip identifiers to version strings.
