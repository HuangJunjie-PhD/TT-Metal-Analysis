The `tt-github-actions` repository serves as a centralized library of shared GitHub Actions and workflows for the Tenstorrent organization. It provides standardized tools for CI/CD telemetry collection, runner optimization, and community management.

## Repository Layout

The repository is structured to separate reusable action logic from the workflows that test or showcase them.

| Directory | Description |
| --- | --- |
| `.github/actions/` | Contains composite and Docker-based actions. Each subdirectory (e.g., `collect_data/`, `job_id/`) is a standalone action with its own `action.yml`. |
| `.github/workflows/` | Contains reusable workflows and CI pipelines that test the actions within this repository. |
| `.github/actions/collect_data/src/` | Python source code for the telemetry aggregation engine, including parsers and data models. |
| `.github/actions/collect_data/test/` | Pytest suite and golden-file fixtures for validating data transformation logic. |

**Sources:**[Readme.md 1-7](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/Readme.md?plain=1#L1-L7)[.github/actions/collect_data/Readme.md 1-8](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/Readme.md?plain=1#L1-L8)

## Using Actions in Other Workflows

To use an action from this repository in an external Tenstorrent project, reference the action using the `uses` keyword pointing to the specific path and branch/tag.

### Example: Using `collect_data`

The `collect_data` action is typically triggered by a `workflow_run` event to aggregate results after a primary CI pipeline finishes.

`jobs:  telemetry:    runs-on: ubuntu-latest    steps:      - name: Collect CI/CD data        uses: tenstorrent/tt-github-actions/.github/actions/collect_data@main        with:          repository: ${{ github.repository }}          run_id: ${{ github.event.workflow_run.id }}          run_attempt: ${{ github.event.workflow_run.run_attempt }}          sftp_host: ${{ secrets.SFTP_HOST }}          sftp_user: ${{ secrets.SFTP_USER }}          ssh-private-key: ${{ secrets.SFTP_SSH_KEY }}`
**Sources:**[.github/actions/collect_data/Readme.md 9-37](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/Readme.md?plain=1#L9-L37)

## Local Development & Testing

### Running Tests Locally

The primary logic for data collection is written in Python and can be tested locally using `pytest`. This is critical for validating changes to parsers or Pydantic models without pushing to GitHub.

1.   Navigate to the action directory: `cd .github/actions/collect_data`
2.   Install dependencies: `pip install -r requirements.txt`
3.   Execute tests: `pytest --junitxml=pytest.xml --cov-report=term-missing --cov=src test`

**Sources:**[.github/actions/collect_data/Readme.md 60-66](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/Readme.md?plain=1#L60-L66)[.github/actions/collect_data/requirements.txt 1-9](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/requirements.txt#L1-L9)[.github/actions/collect_data/pytest.ini 1-3](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/pytest.ini#L1-L3)

### Manual Data Workflow Trigger

You can manually trigger the data collection logic for a specific `run_id` to debug how it processes a particular CI run.

**Using the GitHub CLI:**

`gh workflow run "[internal] Produce analytic data" \  --ref main \  -f test_workflow_run_id=11253719387`
**Running Scripts Locally:** To simulate the action's execution on your machine, you must first download the raw GitHub API data and then run the generator:

1.   `download_workflow_data.sh <repo_name> <run_id> <attempt>`
2.   `GITHUB_EVENT_NAME=test python3 src/generate_data.py --run_id <run_id>`

**Sources:**[.github/actions/collect_data/Readme.md 68-82](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/Readme.md?plain=1#L68-L82)

## CI/CD for this Repository

This repository uses its own tools to ensure code quality:

*   **Pre-commit:** Runs `black` for formatting, `clang-format` for C/C++ files, and `check-copyright` to ensure SPDX headers are present.
*   **Test Collect Data:** A workflow that automatically runs the `pytest` suite whenever changes are made to the `collect_data` directory.
