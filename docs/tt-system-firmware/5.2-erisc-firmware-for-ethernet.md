## Purpose and Scope

This page documents the ERISC (Ethernet RISC) firmware components that control the Ethernet subsystem in Blackhole ASICs. ERISC firmware runs on dedicated RISC processors within each Ethernet core, managing Ethernet protocol operations, SerDes (Serializer/Deserializer) configuration, and link-layer communication.

For information about MRISC firmware that manages GDDR memory controllers, see [MRISC Firmware for GDDR](https://deepwiki.com/tenstorrent/tt-system-firmware/5.1-mrisc-firmware-for-gddr). For PCIe SerDes library documentation, see [PCIe SerDes Library](https://deepwiki.com/tenstorrent/tt-system-firmware/5.3-pcie-serdes-library).

* * *

## ERISC Architecture Overview

Blackhole ASICs contain 14 Ethernet instances, each with:

*   A dedicated ERISC processor running Ethernet firmware
*   512KB of L1 memory for firmware and data buffers
*   SerDes lanes for physical layer communication
*   NOC (Network-on-Chip) connectivity for data transfers

**Sources:**[lib/tenstorrent/bh_arc/eth.c 1-473](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L1-L473)[zephyr/module.yml 36-81](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/zephyr/module.yml#L36-L81)

## Ethernet Instance Selection and SerDes Mux

Not all 14 Ethernet instances are enabled on every board. The actual enabled instances depend on PCIe configuration and SerDes mux settings, as certain SerDes lanes can be allocated to either PCIe or Ethernet.

### SerDes Mux Configuration

The `SetupEthSerdesMux()` function configures `RESET_UNIT_PCIE_MISC_CNTL3` registers to select which ETH instances connect to which SerDes lanes. The `GetEthSel()` function determines the final bitmask of enabled ETH instances based on:

1.   PCIe configuration (disabled/1-lane/2-lane)
2.   SerDes mux settings
3.   Harvesting information
4.   Optional `eth_disable_mask` override

**Sources:**[lib/tenstorrent/bh_arc/eth.c 87-184](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L87-L184)

## MAC Address Generation

Each Ethernet instance receives a unique MAC address derived from the ASIC ID:

The organization ID (OUI) `0x208C47` is fixed, while the device-specific portion is derived from the ASIC ID multiplied by 12 (allowing up to 12 MAC addresses per ASIC for multi-port configurations).

## Reset Management

ERISC processors are held in reset during firmware loading and released after configuration is complete.

### Reset Control Registers

| Register | Address | Purpose |
| --- | --- | --- |
| `ETH_RESET_PC_0` | 0xFFB14000 | RISC0 reset program counter |
| `ETH_END_PC_0` | 0xFFB14004 | RISC0 end program counter |
| `ETH_RESET_PC_1` | 0xFFB14008 | RISC1 reset program counter |
| `ETH_END_PC_1` | 0xFFB1400C | RISC1 end program counter |
| `ETH_RISC_DEBUG_SOFT_RESET_0` | 0xFFB121B0 | Soft reset control |

The `ReleaseEthReset()` function releases RISC0 from reset:

```
soft_reset_0 &= ~(1 << 11);  // Clear bit 11 for RISC0 reset
```

RISC1 remains in reset (bit 11 cleared, other reset bits remain set).

**Sources:**[lib/tenstorrent/bh_arc/eth.c 38-42](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L38-L42)[lib/tenstorrent/bh_arc/eth.c 219-226](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L219-L226)

## Integration with Boot Sequence

ERISC firmware loading is integrated into the system initialization at priority level 106 (`eth_init_PRIO`), which occurs after:

*   **Priority 103**: PCIe initialization (`pcie_init_PRIO`)
*   **Priority 104**: Tensix initialization and L1 wipe (`tensix_init_PRIO`)
*   **Priority 105**: MRISC firmware loading (`InitMrisc_PRIO`)

And before:

*   **Priority 107**: SMBus target initialization (`InitSmbusTarget_PRIO`)
*   **Priority 114**: GDDR training (`gddr_training_PRIO`)

The initialization is performed by the `eth_init()` function, which is registered using the `SYS_INIT_APP()` macro at the end of [lib/tenstorrent/bh_arc/eth.c 473](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/eth.c#L473-L473)

## Summary

ERISC firmware provides Ethernet protocol management for Blackhole's 14 Ethernet instances. Key characteristics:

*   **Binary Components**: Four blob files (ERISC firmware, parameters, SerDes firmware, SerDes registers)
*   **Memory Layout**: Firmware at 0x70000, parameters at 0x7C000 in 512KB L1
*   **Instance Selection**: Dynamic based on PCIe configuration and SerDes mux settings
*   **Configuration**: Board-specific parameters including unique MAC addresses derived from ASIC ID
*   **Initialization**: Coordinated loading of SerDes and ERISC firmware via NOC DMA
*   **Version Management**: Firmware version retrievable at runtime from L1 memory

The firmware is loaded and managed by ARC chip management firmware during the boot sequence, with strict ordering relative to other subsystem initialization.
