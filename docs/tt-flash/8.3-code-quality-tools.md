# Code Quality Tools

Relevant source files
*   [pyproject.toml](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml)

This page documents the code quality tools used in tt-flash development: the `black` code formatter and the `basedpyright` static type checker. These tools enforce consistent code style and catch type-related errors during development.

For information about the overall development setup and build process, see [Building from Source](https://deepwiki.com/tenstorrent/tt-flash/8.1-building-from-source). For the project structure that these tools operate on, see [Project Structure](https://deepwiki.com/tenstorrent/tt-flash/8.2-project-structure).

## Overview

tt-flash uses two primary code quality tools configured in [pyproject.toml 37-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L37-L69):

*   **black**: An opinionated Python code formatter that ensures consistent style across the codebase
*   **basedpyright**: A static type checker (fork of Pyright) that validates type annotations and catches type errors

Both tools are configured as optional development dependencies and are not required for end-user installations.

## Black Code Formatter

### Version-Specific Configuration

Black is configured with Python version-specific constraints to handle compatibility across different Python runtimes. The configuration uses conditional version pinning in [pyproject.toml 38-41](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L38-L41):

`[project.optional-dependencies]dev = [  "black == 24.10.0; python_version >= '3.9'",  "black == 24.8.0; python_version == '3.8'",  "black == 23.3.0; python_version == '3.7'"]`
| Python Version | Black Version | Rationale |
| --- | --- | --- |
| 3.9+ | 24.10.0 | Latest black with full feature support |
| 3.8 | 24.8.0 | Last version with Python 3.8 support |
| 3.7 | 23.3.0 | Legacy support for older environments |

This tiered approach ensures that developers on different Python versions can use black while maintaining consistent formatting output across the project.

### Title: Black Configuration and Dependencies

**Sources**: [pyproject.toml 37-42](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L37-L42)

### Usage

To install black in your development environment:

`# Install with development dependenciespip install -e .[dev] # Or install black separatelypip install black==24.10.0  # for Python 3.9+`
To format the codebase:

`# Format all Python filesblack . # Check formatting without making changesblack --check . # Format specific files or directoriesblack tt_flash/ setup.py`
Black operates with default settings (no custom configuration required), using:

*   88 character line length (black's default)
*   Double quotes for strings
*   Trailing commas in multi-line structures
*   PEP 8 compliant formatting

**Sources**: [pyproject.toml 38-41](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L38-L41)

## Basedpyright Type Checker

### Configuration

Basedpyright is configured in [pyproject.toml 67-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L67-L69) with minimal settings to support broad Python version compatibility:

`[tool.basedpyright]pythonVersion = "3.7"typeCheckingMode = "standard"`
| Setting | Value | Purpose |
| --- | --- | --- |
| `pythonVersion` | `"3.7"` | Ensures type checking is compatible with the oldest supported Python version |
| `typeCheckingMode` | `"standard"` | Balanced type checking (not "basic" or "strict") |

The `pythonVersion = "3.7"` setting is critical because it ensures that type annotations and language features used in tt-flash are compatible with Python 3.7, even though the project's runtime requirement is `>=3.10`[pyproject.toml 6](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L6-L6) This provides forward compatibility for potential future support of older Python versions.

### Title: Basedpyright Configuration Architecture

**Sources**: [pyproject.toml 67-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L67-L69)

### Type Checking Modes

Basedpyright supports three type checking modes:

| Mode | Behavior | Use Case |
| --- | --- | --- |
| `basic` | Minimal type checking | Legacy codebases with few annotations |
| `standard` | Moderate strictness (tt-flash uses this) | Balance between safety and flexibility |
| `strict` | Maximum strictness | Fully type-annotated codebases |

The `standard` mode provides:

*   Basic type inference for unannotated functions
*   Type checking for annotated parameters and return types
*   Detection of common type errors (None vs value types)
*   Limited strictness on missing type annotations

**Sources**: [pyproject.toml 67-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L67-L69)

### Usage

Basedpyright is typically run manually during development or in CI/CD pipelines:

`# Install basedpyright (not in pyproject.toml dependencies)pip install basedpyright # Type check the entire codebasebasedpyright # Type check specific filesbasedpyright tt_flash/flash.py tt_flash/chip.py # Generate type checking reportbasedpyright --outputjson > type_report.json`
Basedpyright reads its configuration automatically from `[tool.basedpyright]` in [pyproject.toml 67-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L67-L69)

**Sources**: [pyproject.toml 67-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L67-L69)

## Notes on Tool Selection

### Black vs Other Formatters

tt-flash uses **black** rather than alternatives like `autopep8` or `yapf` because:

*   Opinionated formatting reduces debates about style
*   Zero configuration required
*   Widely adopted in the Python ecosystem
*   Fast execution on large codebases

### Basedpyright vs Pyright vs mypy

tt-flash uses **basedpyright** (a Pyright fork) rather than:

*   **Pyright**: Basedpyright offers community-driven enhancements
*   **mypy**: Pyright/basedpyright typically provides faster type checking and better integration with modern Python features

The choice of `typeCheckingMode = "standard"` rather than `"strict"` allows gradual adoption of type annotations without blocking development on existing code.
