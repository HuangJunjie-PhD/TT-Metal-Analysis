## Purpose and Scope

The `TTDevice` interface provides architecture-specific hardware operations for Tenstorrent devices. It serves as the abstraction layer between high-level chip management and low-level hardware communication. This layer encapsulates device-specific operations such as memory I/O, DMA transfers, ARC processor communication, and firmware interactions.

The interface utilizes a **DeviceProtocol** pattern to support multiple communication backends (PCIe, JTAG, and Remote Ethernet) while providing a unified API for architecture-specific implementations like `WormholeTTDevice` and `BlackholeTTDevice`.

**Sources:**[device/api/umd/device/tt_device/tt_device.hpp 68-170](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/tt_device.hpp#L68-L170)[device/tt_device/tt_device.cpp 57-98](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L57-L98)

## TTDevice Class Hierarchy

The `TTDevice` base class defines the common interface, while concrete subclasses implement architecture-specific logic. Communication logic is further decoupled into `DeviceProtocol` implementations.

**Sources:**[device/api/umd/device/tt_device/tt_device.hpp 68-343](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/tt_device.hpp#L68-L343)[device/api/umd/device/tt_device/wormhole_tt_device.hpp 24-69](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/wormhole_tt_device.hpp#L24-L69)[device/api/umd/device/tt_device/blackhole_tt_device.hpp 25-76](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/blackhole_tt_device.hpp#L25-L76)

## DeviceProtocol and Interface Pattern

`TTDevice` delegates low-level I/O to a `DeviceProtocol` object. This allows the same `WormholeTTDevice` logic to run over PCIe, JTAG, or a Remote connection by swapping the underlying protocol implementation.

### Protocol Implementations

| Protocol Class | Interface Supported | Backend |
| --- | --- | --- |
| `PcieProtocol` | `PcieInterface` | `PCIDevice` (Kernel Driver) |
| `JtagProtocol` | `JtagInterface` | `JtagDevice` (J-Link/Axi) |
| `RemoteProtocol` | `RemoteInterface` | `RemoteCommunication` (Ethernet Fabric) |

**Sources:**[device/api/umd/device/tt_device/tt_device.hpp 85-88](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/tt_device.hpp#L85-L88)[device/tt_device/tt_device.cpp 65-97](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L65-L97)[device/api/umd/device/tt_device/protocol/pcie_protocol.hpp 36-38](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/protocol/pcie_protocol.hpp#L36-L38)[device/api/umd/device/tt_device/protocol/jtag_protocol.hpp 23-25](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/protocol/jtag_protocol.hpp#L23-L25)

## Device Creation and Factory Pattern

`TTDevice` uses a static factory method to instantiate the appropriate architecture subclass based on the detected hardware.

1.   **Detection**: For PCIe, it probes the `PCIDevice` to get the architecture. For JTAG, it queries the `JtagDevice`.
2.   **Instantiation**: Returns a `unique_ptr` to either `WormholeTTDevice` or `BlackholeTTDevice`.
3.   **Protocol Setup**: During construction, the specific `DeviceProtocol` (PCIe/JTAG/Remote) is initialized and assigned.

**Sources:**[device/tt_device/tt_device.cpp 123-159](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L123-L159)[device/tt_device/tt_device.cpp 57-86](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L57-L86)

## Memory I/O Operations

### Unicast Read/Write

`TTDevice::read_from_device` and `TTDevice::write_to_device` provide basic memory access. In `PcieProtocol`, these use a cached TLB window to map the target core and address into the process's virtual memory space.

### NOC Multicast Write

Multicast writes allow broadcasting data to a range of cores.

*   **PCIe**: Uses `SiliconTlbWindow::noc_multicast_write_reconfigure` to set the `mcast` bit in the TLB entry.
*   **JTAG/Remote**: These protocols typically fallback to unicast loops if hardware multicast is not directly supported by the interface.

**Sources:**[device/tt_device/protocol/pcie_protocol.cpp 66-102](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/protocol/pcie_protocol.cpp#L66-L102)[device/tt_device/protocol/pcie_protocol.cpp 107-117](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/protocol/pcie_protocol.cpp#L107-L117)[device/tt_device/protocol/jtag_protocol.cpp 20-28](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/protocol/jtag_protocol.cpp#L20-L28)

## DMA Operations

DMA provides high-bandwidth transfers between host and device memory.

*   **Wormhole**: Supports DMA via the PCIe BAR2 DMA engine. `WormholeDmaTransfer` manages the hardware registers and completion polling.
*   **Blackhole**: Uses `BlackholeDmaTransfer` strategy.
*   **JTAG/Remote**: DMA is generally unsupported and will throw an exception.

**Sources:**[device/tt_device/protocol/pcie_protocol.cpp 29-38](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/protocol/pcie_protocol.cpp#L29-L38)[device/tt_device/protocol/pcie_protocol.cpp 143-158](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/protocol/pcie_protocol.cpp#L143-L158)[device/api/umd/device/tt_device/protocol/pcie_interface.hpp 31-43](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/protocol/pcie_interface.hpp#L31-L43)

## Firmware Interaction (ARC)

`TTDevice` manages communication with the ARC (Argonaut RISC Core) processor for system management tasks.

### ARC APB and CSM Access

ARC memory is divided into APB (Advanced Peripheral Bus) for registers and CSM (Configuration Space Memory).

*   **Wormhole**: Access is usually through BAR0 offsets.
*   **Blackhole**: Access may be via BAR0 or via NOC-to-AXI translation depending on whether the ARC core is available over the AXI bridge (determined by PCIe tile location).

### ARC Messenger and Telemetry

During `init_tt_device`, the driver creates:

*   `ArcMessenger`: For sending command messages to ARC firmware.
*   `ArcTelemetryReader`: For reading hardware telemetry (clocks, temperatures, harvesting masks).
*   `FirmwareInfoProvider`: For querying firmware versions and capabilities.

**Sources:**[device/tt_device/tt_device.cpp 100-121](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L100-L121)[device/tt_device/wormhole_tt_device.cpp 342-412](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L342-L412)[device/tt_device/blackhole_tt_device.cpp 304-311](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L304-L311)

## Architecture-Specific Implementations

### WormholeTTDevice

*   **iATU**: Configured via ARC messages to set up peer-to-peer regions.
*   **Clocks**: AICLK is retrieved via `ARC_MSG_GET_AICLK`.
*   **Harvesting**: Tensix harvesting masks are retrieved from ARC and shuffled into logical format.

**Sources:**[device/tt_device/wormhole_tt_device.cpp 100-111](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L100-L111)[device/tt_device/wormhole_tt_device.cpp 115-147](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L115-L147)

### BlackholeTTDevice

*   **iATU**: Directly configured via BAR2 registers (`ATU_OFFSET_IN_BH_BAR2`).
*   **Telemetry**: Extensive use of `ArcTelemetryReader` for harvesting masks (Tensix, DRAM, ETH, PCIe).
*   **NOC Translation**: Probes `NIU_CFG_NOC0_BAR_PCIE_ADDR` to check if hardware coordinate translation is enabled.

**Sources:**[device/tt_device/blackhole_tt_device.cpp 74-125](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L74-L125)[device/tt_device/blackhole_tt_device.cpp 127-138](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L127-L138)[device/tt_device/blackhole_tt_device.cpp 140-167](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/blackhole_tt_device.cpp#L140-L167)

## Hang Detection

`TTDevice` includes a `HangDetector` mechanism to identify if the PCIe link or NOC fabric has stopped responding.

*   `is_pcie_hung()`: Reads a known register and checks for the "all-ones" (`0xFFFFFFFF`) signature indicating a link failure.
*   `is_noc_hung()`: Attempts a NOC read and checks if it times out or returns a hang signature.
