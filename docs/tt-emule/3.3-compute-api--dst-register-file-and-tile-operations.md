The Compute API in `tt-emule` provides a C++ emulation of the Tenstorrent Tensix/Neo compute engine. It mirrors the Low-Level Kernel (LLK) and `ckernel` APIs used on hardware, allowing JIT-compiled kernels to perform tile-based math, data movement, and register management.

## DST Register File

In `tt-emule`, the hardware DST register file is emulated as a `thread_local` float array. This array serves as the primary accumulator and scratchpad for all compute operations.

### Capacity and Lifecycle

The physical DST on Wormhole and Blackhole architectures is 64 KB. `tt-emule` provides a kernel-visible ceiling that matches the "SyncFull" capacity of the hardware:

*   **Bfloat16 / INT32**: 16 tiles [include/jit_hw/api/compute/common.h 144](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L144-L144)
*   **FP32**: 8 tiles [include/jit_hw/api/compute/common.h 145](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L145-L145)

The register file follows a strict **Acquire -> Compute -> Commit -> Wait -> Release** lifecycle to match hardware synchronization semantics:

| Function | Emulation Behavior |
| --- | --- |
| `tile_regs_acquire()` | Zeroes out the target DST slots and marks them as "fresh" for operations like `reduce_tile`. [include/jit_hw/api/compute/common.h 182-192](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L182-L192) |
| `tile_regs_commit()` | No-op in emulation; hardware uses this to signal the PACK thread. [include/jit_hw/api/compute/common.h 194](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L194-L194) |
| `tile_regs_wait()` | No-op in emulation; hardware uses this to block UNPACK/MATH. [include/jit_hw/api/compute/common.h 195](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L195-L195) |
| `tile_regs_release()` | Marks DST slots as released. [include/jit_hw/api/compute/common.h 197-201](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L197-L201) |

### Data Representation

The underlying storage is `static thread_local float __emule_dst[16][1024]`[include/jit_hw/api/compute/common.h 148](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L148-L148)

*   **Bfloat16**: Stored as `float32` for high-precision intermediate math.
*   **INT32/UINT32**: Bit-patterns are cast and stored in the `float` array using `std::memcpy` to preserve exact bit-fidelity for integer operations [include/jit_hw/llk_math_eltwise_unary_datacopy.h 31-46](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/llk_math_eltwise_unary_datacopy.h#L31-L46)

## Macro Collapse & Thread Model

On real hardware, compute kernels are split across three RISC-V processors (TRISC0: Unpack, TRISC1: Math, TRISC2: Pack). In `tt-emule`, these are collapsed into a single execution thread. The `PACK(x)`, `MATH(x)`, and `UNPACK(x)` macros are defined to execute their contents sequentially in the same thread [include/jit_hw/api/compute/common.h 46-48](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L46-L48)

## Tile Operations API

### Element-wise Operations

Binary operations (`add_tiles`, `sub_tiles`, `mul_tiles`) perform scalar math across all 1024 elements of the specified DST tiles.

**Data Flow: Binary Operations**

*   **Accumulation**: These operations honor the `__emule_dest_accum_en` flag. If enabled, the result is added to the existing value in the destination tile [include/jit_hw/api/compute/common.h 218-245](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L218-L245)
*   **Sources**: [include/jit_hw/api/compute/common.h 218-245](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L218-L245)[include/jit_hw/api/compute/eltwise_binary.h 23-32](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/eltwise_binary.h#L23-L32)

### Reduction Operations

`reduce_tile` performs SUM or MAX reductions along `REDUCE_ROW`, `REDUCE_COL`, or `REDUCE_SCALAR` dimensions.

*   **Scaler Support**: It reads a scaler value from a separate CB tile (usually containing 1.0 or 1/N) to multiply against the reduced result [include/jit_hw/api/compute/reduce.h 73-78](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/reduce.h#L73-L78)
*   **Freshness Tracking**: To match hardware behavior where DST is undefined after acquisition, `tt-emule` tracks if a DST slot is "fresh." The first reduction into a fresh slot overwrites the zeroed DST; subsequent calls accumulate [include/jit_hw/api/compute/reduce.h 93-117](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/reduce.h#L93-L117)
*   **Sources**: [include/jit_hw/api/compute/reduce.h 63-165](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/reduce.h#L63-L165)

### Matrix Multiplication

`matmul_tiles` implements the FMA (Fused Multiply-Add) logic for 32x32 tiles.

*   **Transpose Support**: It respects the `__llk_matmul_transpose` flag (set during `mm_init`), which transposes the In1 (SrcB) tile before the dot product [include/jit_hw/internal/llk_state.h 49](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/internal/llk_state.h#L49-L49)
*   **Sources**: [include/jit_hw/api/compute/matmul.h](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/matmul.h)[include/jit_hw/internal/llk_state.h 44-49](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/internal/llk_state.h#L44-L49)

## Layout Conversion: nfaces vs. Row-Major

Tenstorrent hardware stores tiles in an `nfaces` (face-row-major) layout in L1 to optimize for the compute engine's 16x16 face structure. `tt-emule` handles the conversion between this hardware layout and the natural row-major layout used in DST.

### Layout Pipeline

1.   **Unpack**: `copy_tile` or `__emule_unpack_cb_tile_to` reads from L1, converts `nfaces` ->`row-major`, and stores in DST [include/jit_hw/api/compute/common.h 410-440](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L410-L440)
2.   **Compute**: Operations (e.g., `add_tiles`) operate on the `row-major` DST array.
3.   **Pack**: `pack_tile` reads from DST, converts `row-major` ->`nfaces`, and writes back to L1 [include/jit_hw/api/compute/common.h 476-508](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/common.h#L476-L508)

## Specialized Compute Kernels

`tt-emule` provides Layer-1 "shadows" for complex hardware-accelerated operations:

### TopK XL

The `topk_xl` API emulates the bitonic sort and merge hardware.

*   **Fused Encoding**: It decodes/encodes fused FP32 words containing a `bf16` value and a `u16` index [include/jit_hw/api/compute/topk_xl.h 48-71](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/topk_xl.h#L48-L71)
*   **Local Sort**: Uses `std::sort` over the flat DST sequence to emulate the hardware's top-K ranking [include/jit_hw/api/compute/topk_xl.h 173-188](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/topk_xl.h#L173-L188)
*   **Sources**: [include/jit_hw/api/compute/topk_xl.h 105-190](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/topk_xl.h#L105-L190)

### Mixture of Experts (MoE) Gates

Shadows for `deepseek_moe_gate`, `glm_moe_gate`, and `kimi_moe_gate` provide bit-accurate emulation of expert selection logic.

*   **Layout Dance**: These kernels often involve complex reshapes and transposes between the "Golden" view (e.g., 8x32) and the "Device" view (16x16 tiles) [include/jit_hw/api/compute/glm_moe_gate.h 51-64](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/glm_moe_gate.h#L51-L64)
*   **Normalization**: Includes fused sigmoid and scaling logic as defined by the specific model architecture [include/jit_hw/api/compute/kimi_moe_gate.h 105-113](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/kimi_moe_gate.h#L105-L113)
*   **Sources**: [include/jit_hw/api/compute/glm_moe_gate.h 39-124](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/glm_moe_gate.h#L39-L124)[include/jit_hw/api/compute/kimi_moe_gate.h 47-136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/kimi_moe_gate.h#L47-L136)

## Tilize and Untilize

`tt-emule` supports both hardware-accelerated and software-emulated tilization.

*   **Tilize Block**: Converts a horizontal strip of row-major data in an input CB into multiple `nfaces` tiles in an output CB [include/jit_hw/api/compute/tilize.h 66-145](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/tilize.h#L66-L145)
*   **Untilize Block**: Performs the inverse, scattering DST tiles back into a row-major linear buffer [include/jit_hw/api/compute/untilize.h 48-56](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/untilize.h#L48-L56)
*   **BFP8/BFP4 Support**: The tilizer includes specific paths for encoding shared-exponent block float formats during the conversion [include/jit_hw/api/compute/tilize.h 90-115](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/tilize.h#L90-L115)
*   **Sources**: [include/jit_hw/api/compute/tilize.h 26-155](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/tilize.h#L26-L155)[include/jit_hw/api/compute/untilize.h 24-60](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/compute/untilize.h#L24-L60)
