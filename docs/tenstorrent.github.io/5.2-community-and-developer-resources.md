This section provides a guide to the Tenstorrent ecosystem, detailing the various channels and platforms available for developer support, community engagement, and hardware access. It covers technical support pathways, cloud-based hardware evaluation, and legal terms for pre-installed model weights on Tenstorrent systems.

## Developer Ecosystem Overview

The Tenstorrent developer ecosystem is designed to support users across different stages of development, from initial hardware evaluation to low-level kernel optimization. The primary entry point for software documentation is the `core/index.rst` file, which serves as the portal for software guides, hardware specifications, and support resources [core/index.rst 1-50](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/index.rst#L1-L50)

### Community Channels

*   **Discord Community**: The primary hub for real-time interaction with Tenstorrent engineers and other developers. Users can ask questions, share projects, and receive updates on the software stack [core/os-model-terms.md 13](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/os-model-terms.md?plain=1#L13-L13)
*   **Support Request Portal**: For enterprise and technical support, users are directed to the formal support channels defined in the repository [core/index.rst 42](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/index.rst#L42-L42)
*   **Tenstorrent YouTube Channel**: Provides video tutorials, hardware deep-dives, and demonstrations of the software stack in action.
*   **TT-Awesome**: A curated catalog of community-contributed models, tools, and projects optimized for Tenstorrent hardware.

## Hardware Access & Cloud Resources

For developers who do not yet possess physical hardware, Tenstorrent provides cloud-based access and specialized hubs for information.

### Console Cloud Access

Tenstorrent Console allows developers to rent Tensix-based hardware (such as Wormhole n150 or n300 cards) in a cloud environment. This environment typically comes pre-configured with the necessary driver stack (TT-KMD) and development tools (TT-SMI).

### Developer Hub

The Developer Hub serves as a centralized repository for technical briefs, whitepapers, and getting-started guides. It links directly to the documentation projects for TT-Metalium and TT-NN [core/index.rst 31-33](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/index.rst#L31-L33)

## Pre-installed Models & OS Terms

Systems such as the **TT-QuietBox 2 (Blackhole)** may arrive with pre-installed open-weight AI models to facilitate immediate benchmarking and demonstration [core/os-model-terms.md 3-4](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/os-model-terms.md?plain=1#L3-L4)

### Supported Pre-installed Models

The following models are commonly provided for demonstration purposes:

| Model Category | Model Name | Source |
| --- | --- | --- |
| LLM | Qwen3-32B | HuggingFace |
| Speech-to-Text | Whisper-large-v3 | OpenAI |
| Text-to-Speech | SpeechT5-TTS | Microsoft |
| Text-to-Video | Wan2.2-T2V-A14B | Wan-AI |

### Usage Terms and Liability

Usage of these pre-installed models is subject to specific terms:

1.   **Demonstration Only**: Models are provided to showcase Tenstorrent hardware capabilities, not as a software product [core/os-model-terms.md 7](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/os-model-terms.md?plain=1#L7-L7)
2.   **No Warranty**: Models are provided "as-is" [core/os-model-terms.md 8](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/os-model-terms.md?plain=1#L8-L8)
3.   **Compliance**: Users must adhere to the original model licenses (e.g., Apache 2.0, MIT, or custom open-weight licenses) [core/os-model-terms.md 10](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/os-model-terms.md?plain=1#L10-L10)

## Documentation Feedback System

The documentation repository includes an integrated feedback mechanism to allow developers to report inaccuracies or suggest improvements directly from the browser.

### Implementation Logic

The feedback system is implemented via a template `feedback_widget.html` and integrated into the global Sphinx layout [shared/_templates/layout.html 39-41](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/shared/_templates/layout.html#L39-L41)

**Feedback Data Flow** The widget captures user sentiment and optional qualitative feedback, dispatching it to an analytics backend.

"Documentation Feedback Data Flow"

**Key Entities:**

*   `tt-feedback`: The root container for the widget [shared/_templates/feedback_widget.html 1](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/shared/_templates/feedback_widget.html#L1-L1)
*   `down_reason`: A radio group capturing specific failure modes: `inaccurate`, `unclear`, `missing`, or `irrelevant`[shared/_templates/feedback_widget.html 21-24](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/shared/_templates/feedback_widget.html#L21-L24)
*   `hide_feedback`: A metadata flag that can be set in `.rst` or `.md` files to disable the widget on specific pages [core/index.rst 3-4](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/core/index.rst#L3-L4)[shared/_templates/feedback_widget.html 83](https://github.com/tenstorrent/tenstorrent.github.io/blob/eda54884/shared/_templates/feedback_widget.html#L83-L83)
