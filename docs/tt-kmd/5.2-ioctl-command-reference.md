# IOCTL Command Reference

Relevant source files
*   [chardev.c](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c)
*   [ioctl.h](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h)

This page provides a comprehensive reference for all IOCTL commands exposed by the Tenstorrent kernel driver through the character device interface. These commands enable user-space applications to query device information, manage memory resources, configure address translation, control device resets, and manage power states.

For information about opening and closing the character device and per-FD state management, see [5.1 Character Device Operations](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/5.1%20Character%20Device%20Operations) For details about memory mapping operations via `mmap()`, see [5.3 Memory Mapping Interface](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/5.3%20Memory%20Mapping%20Interface)

## IOCTL Interface Overview

The driver uses the Linux IOCTL interface with magic number `0xFA` (defined as `TENSTORRENT_IOCTL_MAGIC`[ioctl.h 12](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L12-L12)) and implements 16 command codes (0-15). All IOCTLs follow a common pattern where input and output parameters are passed through structures containing `in` and `out` substructures. The IOCTL dispatcher is implemented in `tt_cdev_ioctl()`[chardev.c 465-576](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L465-L576)

### Command Dispatch Mechanism

The IOCTL dispatcher [chardev.c 465-576](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L465-L576) implements a multi-layer validation and synchronization mechanism:

1.   **Synchronization Layer**[chardev.c 470-475](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L470-L475): 
    *   Reset IOCTLs acquire `reset_rwsem` for exclusive write access (blocks all other operations).
    *   Non-reset IOCTLs acquire `reset_rwsem` for shared read access (allows concurrent execution).

2.   **Validation Checks**[chardev.c 477-498](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L477-L498): 
    *   **Detached check**: Returns `-ENODEV` if device has been hot-unplugged.
    *   **Reset generation check**: Returns `-ENODEV` if FD was opened before the last reset.
    *   **Hardware init check**: During a reset window, only info queries and reset operations are allowed.

3.   **Handler Dispatch**[chardev.c 500-567](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L500-L567): Dispatches to handlers that use `copy_from_user()` and `copy_to_user()` for data transfer.

Sources: [chardev.c 465-576](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L465-L576)

## Device Information Commands

### GET_DEVICE_INFO

Retrieves PCI device identification and configuration information.

**Command Code:**`TENSTORRENT_IOCTL_GET_DEVICE_INFO` (0) [ioctl.h 14](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L14-L14)

**Handler:**`ioctl_get_device_info()`[chardev.c 127-159](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L127-L159)

**Structure:**`struct tenstorrent_get_device_info`[ioctl.h 61-64](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L61-L64)

| Input Field | Type | Description |
| --- | --- | --- |
| `output_size_bytes` | `__u32` | Size of output buffer provided by caller |

| Output Field | Type | Description |
| --- | --- | --- |
| `output_size_bytes` | `__u32` | Actual size of output structure |
| `vendor_id` | `__u16` | PCI vendor ID (from `pdev->vendor`) |
| `device_id` | `__u16` | PCI device ID (from `pdev->device`) |
| `subsystem_vendor_id` | `__u16` | PCI subsystem vendor ID |
| `subsystem_id` | `__u16` | PCI subsystem device ID |
| `bus_dev_fn` | `__u16` | Packed bus/device/function: bits [0:2] function, [3:7] device, [8:15] bus |
| `max_dma_buf_size_log2` | `__u16` | Log2 of maximum DMA buffer size (28 = 256 MB) |
| `pci_domain` | `__u16` | PCI domain number |

Sources: [chardev.c 127-159](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L127-L159)[ioctl.h 45-64](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L45-L64)

## Memory Management Commands

### QUERY_MAPPINGS

Returns the available memory mapping regions (BARs) that can be accessed via `mmap()`.

**Command Code:**`TENSTORRENT_IOCTL_QUERY_MAPPINGS` (2) [ioctl.h 16](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L16-L16)

**Structure:**`struct tenstorrent_query_mappings`[ioctl.h 82-85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L82-L85)

**Mapping IDs:**[ioctl.h 32-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L32-L38)

*   `TENSTORRENT_MAPPING_RESOURCE0_UC` (1): BAR0 uncacheable.
*   `TENSTORRENT_MAPPING_RESOURCE0_WC` (2): BAR0 write-combining.
*   `TENSTORRENT_MAPPING_RESOURCE1_UC` (3): BAR2 uncacheable.
*   `TENSTORRENT_MAPPING_RESOURCE1_WC` (4): BAR2 write-combining.
*   `TENSTORRENT_MAPPING_RESOURCE2_UC` (5): BAR4 uncacheable.
*   `TENSTORRENT_MAPPING_RESOURCE2_WC` (6): BAR4 write-combining.

Sources: [ioctl.h 16](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L16-L16)[ioctl.h 32-38](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L32-L38)[ioctl.h 82-85](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L82-L85)

### ALLOCATE_DMA_BUF

Allocates a coherent DMA buffer in kernel memory and optionally configures an outbound iATU region for NOC DMA access.

**Command Code:**`TENSTORRENT_IOCTL_ALLOCATE_DMA_BUF` (3) [ioctl.h 17](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L17-L17)

**Structure:**`struct tenstorrent_allocate_dma_buf`[ioctl.h 107-110](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L107-L110)

**Flags:**

*   `TENSTORRENT_ALLOCATE_DMA_BUF_NOC_DMA` (2): Enable NOC DMA [ioctl.h 88](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L88-L88)

Sources: [ioctl.h 87-110](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L87-L110)

### PIN_PAGES

Pins user-space pages into physical memory for zero-copy DMA operations.

**Command Code:**`TENSTORRENT_IOCTL_PIN_PAGES` (7) [ioctl.h 21](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L21-L21)

**Structure:**`struct tenstorrent_pin_pages`[ioctl.h 204-207](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L204-L207)

**Flags:**[ioctl.h 168-171](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L168-L171)

*   `TENSTORRENT_PIN_PAGES_CONTIGUOUS` (1): App attests that the pages are physically contiguous.
*   `TENSTORRENT_PIN_PAGES_NOC_DMA` (2): App wants to use the pages for NOC DMA.
*   `TENSTORRENT_PIN_PAGES_NOC_TOP_DOWN` (4): NOC DMA will be allocated top-down.
*   `TENSTORRENT_PIN_PAGES_READ_ONLY` (8): Device will only read.

Sources: [ioctl.h 167-207](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L167-L207)

## Resource Lock Commands

### LOCK_CTL

Provides user-space resource locking with 64 independent locks.

**Command Code:**`TENSTORRENT_IOCTL_LOCK_CTL` (8) [ioctl.h 22](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L22-L22)

**Handler:**`ioctl_lock_ctl()`[chardev.c 270-329](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L270-L329)

**Structure:**`struct tenstorrent_lock_ctl`[ioctl.h 244-247](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L244-L247)

**Operations:**[ioctl.h 210-213](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/ioctl.h#L210-L213)

*   `TENSTORRENT_LOCK_CTL_ACQUIRE` (0): Try to acquire lock.
*   `TENSTORRENT_LOCK_CTL_RELEASE` (1): Release lock.
*   `TENSTORRENT_LOCK_CTL_TEST` (2): Test lock state.
*   `TENSTORRENT_LOCK_CTL_ACQUIRE_BLOCKING` (3): Blocks until acquired.
