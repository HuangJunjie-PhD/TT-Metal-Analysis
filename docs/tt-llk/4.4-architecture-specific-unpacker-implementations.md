## Purpose and Scope

This document compares the architecture-specific unpacker implementations between Wormhole B0 and Blackhole architectures. While both architectures share a common API contract defined in the LLK (Low-Level Kernel) layer, they differ significantly in their hardware capabilities, configuration structures, and optimization strategies.

**Key architectural differences:**

*   **Configuration size**: Wormhole B0 uses 2 DWORD (64-bit) structures, Blackhole uses 4 DWORD (128-bit)
*   **Processing model**: Wormhole B0 processes tiles face-by-face (16x16), Blackhole can process entire tiles (64-row mode)
*   **Synchronization**: Wormhole B0 uses basic semaphores, Blackhole adds TT Sync Tracking with CSR status queries
*   **Memory access**: Wormhole B0 requires manual address management, Blackhole provides hardware-assisted strided mode

For fundamental unpacker operations shared across architectures, see page 4.1. For tilization operations, see page 4.2. For broadcast patterns, see page 4.3.

### Narrow Tile Support

Wormhole B0 implements narrow tile optimization throughout its unpacker operations. Narrow tiles (16x32 or tiles with `num_faces < 4`) require different loop configurations:

| Configuration | Standard Tile (32x32) | Narrow Tile (16x32 or fewer faces) |
| --- | --- | --- |
| `outerloop` in MOP | 2 (for 4 faces) | 1 (for 2 faces) or variable |
| `innerloop` in MOP | Varies by operation | Often 1 |
| Address increments | Full tile stride | Half tile stride |
| Example usage | `num_faces == 4` | `num_faces == 1` or `num_faces == 2` |

Narrow tile handling in Wormhole B0:

*   Tilize operations: `const std::uint32_t outerloop = narrow_tile ? 1 : 2`[tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h 30](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_tilize.h#L30-L30)
*   Unpack AB: `narrow_tile` parameter affects MOP configuration [tt_llk_wormhole_b0/llk_lib/llk_unpack_AB.h 23](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_AB.h#L23-L23)
*   Reduce operations: Special handling for narrow tiles [tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h 32](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_math_reduce.h#L32-L32)

Blackhole narrow tile support is more limited:

*   Tilize: `LLK_ASSERT(!narrow_tile, "narrow_tile: this parameter is unused")`[tt_llk_blackhole/llk_lib/llk_unpack_tilize.h 23](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_unpack_tilize.h#L23-L23)
*   Pack operations accept `narrow_tile` parameter but include similar assertions [tt_llk_blackhole/llk_lib/llk_pack.h 84](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/llk_lib/llk_pack.h#L84-L84)
*   The 64-row unpacker architecture makes narrow tile optimization less relevant

### MOP Configuration Templates

Wormhole B0 uses template-based MOP (Micro-Operation Program) configuration with extensive compile-time specialization:

`template <    BroadcastType BType = BroadcastType::NONE,    bool acc_to_dest = false,    EltwiseBinaryReuseDestType binary_reuse_dest = EltwiseBinaryReuseDestType::NONE,    bool unpack_to_dest = false>inline void _llk_unpack_A_mop_config_(    const bool transpose_of_faces,     const std::uint32_t num_faces,     const std::uint32_t unpack_src_format,     const std::uint32_t unpack_dst_format = 0)`
#### MOP Programming Patterns

Key MOP programming patterns:

1.   **Replay Buffer Recording** - Instructions recorded once and replayed:

    *   `lltt::record(0, 4)` records 4 instructions [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 57](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L57-L57)
    *   `lltt::replay_insn(0, 2)` replays first 2 instructions [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 62](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L62-L62)
    *   Used for context switching between `unp_cfg_context` 0 and 1

2.   **Broadcast-Specific MOPs** - Different instruction sequences per broadcast type:

    *   **Column**: `srcb_set_z_2 = TT_OP_SETADCZW(p_setadc::UNP_B, 0, 0, 0, 2, 0b0001)`[tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 55](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L55-L55)
    *   **Row**: `unpack_srcb_unpack_srcb = lltt::replay_insn(2, 2)` with `srcb_clear_z` end op [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 111-117](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L111-L117)
    *   **Scalar**: `unpack_srcb_inc_z_0` with no Z increment [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 124](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L124-L124)

3.   **Face Transpose Handling** - Reorders face unpacking for transposed tiles:

    *   Records separate replay buffer at offset 4 [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 134](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L134-L134)
    *   Uses `TTI_UNPACR(SrcA, 0b10, ...)` or `0b01` for face increment [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 139-143](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L139-L143)
    *   `srca_set_z_1 = TT_OP_SETADCZW(p_setadc::UNP_A, 0, 0, 0, 1, 0b0001)` restores Z counter [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 54](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L54-L54)

4.   **Unpack-to-Dest Path** - Direct unpacking to destination register for 32-bit formats:

    *   `unpack_srca_to_dest = TT_OP_UNPACR(SrcA, 0b00010001, ...)` with CH0/CH1 Z increment [tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 40-41](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L40-L41)
    *   Special handling for broadcast types with column variant `0b00100010`[tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h 42-43](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/llk_lib/llk_unpack_A.h#L42-L43)

### Fast Tilize Operation (Wormhole B0 Only)

Wormhole B0 provides a specialized fast tilize path that uses both unpackers and the packer simultaneously to achieve higher throughput. This feature is **not available on Blackhole**.

#### Fast Tilize Architecture

#### Unit Dimension Parameter

Fast tilize uses a `unit_dim` parameter (1, 2, or 3) that determines how many tiles are processed per iteration:

| unit_dim | Tiles Per Unit | Top Faces (SrcA) | Bottom Faces (SrcB) | Packer Pattern |
| --- | --- | --- | --- | --- |
| 1 | 1 tile | 2 faces | 2 faces | ADDR_MOD_0 stride |
| 2 | 2 tiles | 4 faces | 4 faces | ADDR_MOD_0 + ADDR_MOD_1 |
| 3 | 3 tiles | 6 faces | 6 faces | ADDR_MOD_2 stride |

## Key Implementation Differences Summary

| Aspect | Wormhole B0 | Blackhole |
| --- | --- | --- |
| **Descriptor Complexity** | 2 DWORD (64-bit) | 4 DWORD (128-bit) with blob addressing |
| **Tile X Dimension** | 16 (face width) for face-by-face | 1024 (64-row mode) for single-pass |
| **Tile Z Dimension** | 4 (face iteration in tilize) | 1 (no iteration in 64-row mode) |
| **Synchronization Model** | `semaphore_post()`, `t6_semaphore_get()` | TT Sync + CSR + `store_then_load()` |
| **Context Count** | 2 contexts (double-buffering) | Up to 8 contexts (multi-buffering) |
| **Memory Ordering** | Software-managed stores | Hardware-assisted `store_then_load()` |
| **Replay Buffer Strategy** | `lltt::record()` in 16 slots | `lltt::record()` + `load_replay_buf()` |
| **Tilize Implementation** | `_llk_unpack_tilize_()` multi-pass | `_llk_unpack_tilize_()` single-pass |
| **Tilize X-End Config** | `face_r_dim * FACE_C_DIM - 1` | Fixed at `1023` (64 rows) |
| **Tilize Num Loops** | `num_faces / 2` or `2` (narrow) | Always `1` |
| **Fast Tilize** | Yes: `_llk_pack_fast_tilize_init_()` | No (not implemented) |
| **Untilize Pack** | Manual `TTI_ADDRCRXY()` per row | `DST_ACCESS_STRIDED_MODE` flag |
| **Untilize Addr Mods** | `ADDR_MOD_0` manual increment | Hardware-computed strides |
| **Partial Face Support** | Full with face-by-face iteration | Simplified with 64-row processing |
| **Broadcast Optimization** | Template-specialized MOPs | Template-specialized MOPs |
| **Narrow Tile Handling** | Extensive branches with `narrow_tile` | `LLK_ASSERT(!narrow_tile, ...)` |
| **Packer Config Struct** | 4 words in `pack_config_t` | 3 words in `pack_config_t` |
| **Packer Interface Control** | Fixed interface selection | `src_if_sel`, `pack_start_intf_pos` |

### Performance Implications

**Wormhole B0 Characteristics:**

*   Optimized for straightforward tile operations with minimal configuration overhead
*   Extensive narrow tile support (`narrow_tile` parameter) provides flexibility for sub-tile operations
*   Manual address management in untilize gives precise control but requires more instructions per row
*   Fast tilize feature achieves higher throughput using dual unpacker + 4 packer interfaces
*   Face-by-face processing requires 2-4 iterations per tile depending on `num_faces`

**Blackhole Improvements:**

*   64-row unpacker in `_llk_unpack_tilize_()` reduces tilize to single pass (4x fewer context switches)
*   Hardware strided mode in `_llk_pack_untilize_mop_config_()` reduces instruction count by ~50%
*   `store_then_load()` pattern ensures memory ordering without explicit synchronization barriers
*   Enhanced 128-bit configuration structures enable more complex data layouts with minimal overhead
*   Better pipeline utilization through hardware-assisted memory ordering
*   Multi-context support (8 contexts vs. 2) improves parallelism in complex workloads
