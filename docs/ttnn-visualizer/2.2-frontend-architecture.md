## Core Architecture

The TT-NN Visualizer frontend is built as a single-page application (SPA) using React. It communicates with the Flask backend through a REST API to retrieve and visualize profiler data, performance metrics, and network operation information.

Sources: [src/hooks/useAPI.tsx 1-914](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/hooks/useAPI.tsx#L1-L914)[src/components/Layout.tsx 1-95](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/components/Layout.tsx#L1-L95)[src/store/app.ts 1-48](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/store/app.ts#L1-L48)

## Component Hierarchy

The frontend is organized around a main Layout component that provides the structure for the application. The content is rendered through a router outlet system that allows for modal overlays.

Sources: [src/components/Layout.tsx 29-94](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/components/Layout.tsx#L29-L94)[src/components/FooterInfobar.tsx 25-133](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/components/FooterInfobar.tsx#L25-L133)

## State Management with Jotai

The application uses Jotai for state management, which provides a simpler alternative to Redux with atomic state. Key atoms are defined in `app.ts` and used throughout the application to maintain consistent state.

Sources: [src/store/app.ts 5-47](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/store/app.ts#L5-L47)[src/components/Layout.tsx 31-44](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/components/Layout.tsx#L31-L44)

## Data Fetching and API Integration

The application uses a combination of React Query and custom hooks defined in `useAPI.tsx` to fetch data from the backend API. These hooks provide a consistent interface for components to request and manage data.

Sources: [src/hooks/useAPI.tsx 53-892](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/hooks/useAPI.tsx#L53-L892)

## Data Models

The application uses TypeScript interfaces to define the shape of data throughout the application, ensuring type safety and consistent data structures. The key data models are defined in `APIData.ts`.

Key data models include:

| Data Model | Description | Key Properties |
| --- | --- | --- |
| `Operation` | Base operation information | id, name, inputs, outputs, stack_trace |
| `OperationDescription` | Extended operation details | duration, arguments, device_operations |
| `OperationDetailsData` | Detailed operation information | buffers, buffersSummary, l1_sizes |
| `Tensor` | Tensor data structure | address, shape, dtype, layout, producers, consumers |
| `Buffer` | Memory buffer information | address, buffer_type, device_id, size |
| `BufferData` | Extended buffer details | operation_id, max_size_per_bank, next_usage |
| `Node` | Device operation node | node_type, params, connections |
| `TabSession` | Session state | active_report, remote_connection |

Sources: [src/model/APIData.ts 9-278](https://github.com/tenstorrent/ttnn-visualizer/blob/b191aae4/src/model/APIData.ts#L9-L278)

## Conclusion

The TT-NN Visualizer frontend architecture is designed to provide a responsive and interactive interface for analyzing neural network operations. It leverages React for UI components, Jotai for state management, React Query for data fetching, and Blueprint JS for UI controls. The component-based structure allows for modular development and maintenance, while the custom hooks provide a clean abstraction over the backend API.

Key architectural strengths include:

*   Separation of concerns between UI, state management, and data fetching
*   Type-safe data models using TypeScript interfaces
*   Efficient state management with Jotai's atomic approach
*   Responsive UI with Blueprint JS components
*   Modular component structure for maintainability
