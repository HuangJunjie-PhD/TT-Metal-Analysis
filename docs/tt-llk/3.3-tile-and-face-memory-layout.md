## Purpose and Scope

This document explains the memory layout conventions for tiles and faces in tt-llk. A tile is the fundamental unit of data processing, organized as four 16×16 faces that form a 32×32 matrix. This page covers:

* Tile and face dimension constants and their usage in code
* Face ordering and memory organization in L1
* Stride calculations for different data formats
* Address counter mechanics for navigating tile memory
* Special cases: partial faces, narrow tiles, and BFP format layouts

For data format specifics, see [DataFormat Enumeration](/tenstorrent/tt-llk/3.1-dataformat-enumeration-and-support). For format conversion between stages, see [Format Conversion Pipeline](/tenstorrent/tt-llk/3.4-format-conversion-and-inference-pipeline).

## Tile and Face Dimensions

### Core Dimension Constants

The hardware processes data in **tiles** (32×32 matrices) subdivided into **faces** (16×16 sub-matrices). These constants are defined in `ckernel_defs.h`:

| Constant | Value | Description | Usage |
| --- | --- | --- | --- |
| `FACE_R_DIM` | 16 | Face row dimension | Unpacker/packer stride calculations |
| `FACE_C_DIM` | 16 | Face column dimension | X-counter end value configuration |
| `TILE_R_DIM` | 32 | Tile row dimension | Full tile processing loops |
| `TILE_C_DIM` | 32 | Tile column dimension | Full tile processing loops |
| `TILE_NUM_FACES` | 4 | Faces per standard tile | Z-counter range |

## Partial Faces and Narrow Tiles

### The num\_faces Parameter

Most LLK operations accept a `num_faces` parameter that specifies how many 16×16 faces are active in the tile:

| `num_faces` | Tile Shape | Active Faces | Typical Use Case |
| --- | --- | --- | --- |
| 1 | 16×16 | f0 only | Small matrix operations |
| 2 | 32×16 or 16×32 | f0, f1 or f0, f2 | Rectangular tiles |
| 4 | 32×32 | f0, f1, f2, f3 | Standard full tiles |

## Packer Memory Layout and Strides

### Packer Stride Calculation

The packer uses four-dimensional addressing (X, Y, Z, W) to write tiles to L1. Strides are calculated based on data format and face dimensions.

**Code Entity: `set_packer_strides(pack_src_format)`**

From `cpack_common.h`, this function calculates strides for destination register traversal:

**Stride Hierarchy:**

* **X stride**: Bytes per datum (1, 2, or 4)
* **Y stride**: Bytes per row = `FACE_R_DIM × x_stride`
* **Z stride**: Bytes per face = `FACE_C_DIM × y_stride`
* **W stride**: Bytes per tile = `TILE_NUM_FACES × z_stride`

**Diagram: Packer Stride Calculation for Float16**

### Packer L1 Address Offsets

Each of the four packers writes to a different face. The `set_packer_l1_offset()` function configures offset addresses for packers 1, 2, and 3 relative to packer 0.

**Code Entity: `set_packer_l1_offset(pack_dst_format, face_r_dim)`**

**L1 Offset Calculation Examples:**

| Format | face\_dim | Calculation | l1\_offset\_1 (16B units) |
| --- | --- | --- | --- |
| Float16 | 256 (16×16) | (256/16) × 2 - 1 | 31 |
| Float32 | 256 | (256/16) × 4 - 1 | 63 |
| Bfp8\_b | 256 | 1 - 1 | 0 (header only) |

**Diagram: Packer L1 Address Layout**

## BFP Format Memory Layout

### BFP Tile Structure

Block Floating-Point (BFP) formats store a shared exponent for each 16-element block, significantly reducing memory footprint.

**Diagram: BFP8 Tile Memory Layout**

### Exponential Section Size

The `exp_section_size` field in `pack_config_t` specifies the size of the exponent block. For BFP formats, this depends on `num_faces`:

**Code Entity: `cache_exponential_section_sizes_in_gprs()`**

From `cpack_common.h`, this function pre-computes exponent section sizes for different BFP formats:

**BFP Format Exponent Section Sizes (in 16-byte units):**

| Format | num\_faces | Exp Section Size | Calculation |
| --- | --- | --- | --- |
| Bfp8\_b | 2 | 17 (272B) | 1 header + 0 offset + 16 |
| Bfp8\_b | 4 | 19 (304B) | 1 header + 2 offset + 16 |
| Bfp4\_b | 2 | 9 (144B) | 1 header + 0 offset + 8 |

| Bfp2\_b | 2 | 5 (80B) | 1 header + 0 offset + 4 |

## Special Case: Tilize and Untilize

### Untilize Face Traversal

Untilize operations convert tiled data (faces f0-f3) to linear row-major layout. The unpacker processes faces by iterating through rows:

**Code Entity: `_llk_unpack_untilize_pass_()`**

This function from `llk_unpack_untilize.h` processes 16 rows per tile, working through faces sequentially:

The untilize operation requires two passes for a full 32×32 tile:

* **Pass 1**: Processes faces 0 and 1 (rows 0-15)
* **Pass 2**: Processes faces 2 and 3 (rows 16-31)
