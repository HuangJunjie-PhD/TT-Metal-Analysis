Release automation in `tt-inference-server` ensures that model implementations, hardware configurations, and performance/accuracy targets are synchronized with validated Docker artifacts. The workflow transitions code from development through pre-release to production using automated CI parsing, model specification updates, and image promotion.

## Release Workflow and Branching Strategy

The repository follows a structured branching strategy to maintain stability while integrating high-frequency updates from upstream dependencies like `tt-metal` and `vllm-tt-metal`.

*   **`dev`**: The active development branch where feature PRs are merged.
*   **`pre-release-vx.x.x`**: A stable branch cut from a passing Models CI commit on `dev`[scripts/release/README.md 42-46](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L42-L46)
*   **`rc-vx.x.x`**: Release Candidate branch created after merging `pre-release` into `dev`, used for final validation before `main`[scripts/release/README.md 148-149](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L148-L149)
*   **`main`**: The production-ready branch containing tagged releases.

## Key Automation Components

### 1. Models CI Reader (`models_ci_reader.py`)

This script serves as the bridge between the private `tt-shield` CI and the public `tt-inference-server`. It iterates through GitHub Action runs to identify the most recent successful executions for every model/device combination [scripts/release/models_ci_reader.py 52-54](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/models_ci_reader.py#L52-L54)

*   **Functions**: 
    *   `list_workflow_runs()`: Fetches history of the `on-nightly.yml` workflow using the GitHub API [scripts/release/models_ci_reader.py 136-141](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/models_ci_reader.py#L136-L141)
    *   `download_job_logs()`: Retrieves raw execution logs to verify model status [scripts/release/models_ci_reader.py 187-192](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/models_ci_reader.py#L187-L192)
    *   `parse_workflow_logs_dir()`: Utilizes `workflow_logs_parser.py` to extract `tt_metal_commit` and `vllm_commit` from the CI artifacts [scripts/release/workflow_logs_parser.py 22-24](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/workflow_logs_parser.py#L22-L24)

### 2. Model Spec Updater (`update_model_spec.py`)

Updates the central `workflows/model_spec.py` file with validated commit hashes. It ensures that `ModelSpecTemplate` objects reflect the exact versions of `tt-metal` and `vllm` that passed CI [scripts/release/update_model_spec.py 9-11](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/update_model_spec.py#L9-L11)

*   **Implementation**: 
    *   `map_perf_status_to_model_status()`: Maps CI `perf_status` (target, complete, functional) to `ModelStatusTypes` enum values [scripts/release/update_model_spec.py 47-55](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/update_model_spec.py#L47-L55)
    *   `find_template_in_content()`: Performs regex-based in-place replacement of commit hashes within the Python source file to avoid losing manual comments or structure [scripts/release/update_model_spec.py 63-75](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/update_model_spec.py#L63-L75)

*   **Documentation Generation**: After updating specs, the script can re-generate the Model Support `README.md` and `model_spec.json` using `generate_model_support_docs.py` or the `--output-only` flag in the updater [scripts/release/README.md 71-75](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L71-L75)[scripts/release/generate_model_support_docs.py 9-12](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/generate_model_support_docs.py#L9-L12)

### 3. Release Image Artifacts (`make_release_image_artifacts.py`)

This script manages the promotion of Docker images. It uses the `crane` CLI to perform server-side copies of images from the internal `tt-shield` registry to the public `ghcr.io/tenstorrent/tt-inference-server` repository [scripts/release/README.md 87-92](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L87-L92)

*   **Logic**: 
    *   Increments the `VERSION` file based on the `--increment` argument (`major`, `minor`, or `patch`) [scripts/release/README.md 99](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L99-L99)
    *   Constructs new tags following the pattern: `{VERSION}-{tt_metal_short_sha}-{vllm_short_sha}`[scripts/release/README.md 91-92](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L91-L92)
    *   Handles the `--dev` vs `--release` flag to target different image namespaces (e.g., `vllm-tt-metal-src-dev-...` vs `vllm-tt-metal-src-release-...`) [scripts/release/README.md 100-101](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/release/README.md?plain=1#L100-L101)

## Docker Image Management

While most images are promoted via `crane`, new or manual models may require local builds using `scripts/build_docker_images.py`.

### Resource-Aware Parallel Builds

The `build_docker_images.py` script includes a resource management system to prevent host exhaustion during parallel builds.

| Feature | Code Entity | Description |
| --- | --- | --- |
| **Memory Guard** | `get_available_memory_gb()` | Checks `/proc/meminfo` for `MemAvailable` to account for reclaimable buffers [scripts/build_docker_images.py 43-54](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/build_docker_images.py#L43-L54) |
| **Disk Guard** | `get_available_disk_gb()` | Queries `docker info` for the storage root directory to ensure space for layers [scripts/build_docker_images.py 85-98](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/build_docker_images.py#L85-L98) |
| **Concurrency** | `get_max_concurrent_builds()` | Calculates the optimal worker count based on physical CPU cores (default 4 per build) and available RAM [scripts/build_docker_images.py 100-155](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/build_docker_images.py#L100-L155) |

## Compliance and Quality Gates

### License Enforcement

The release process enforces license headers using `scripts/add_spdx_header.py`. This script scans `.py`, `.sh`, and `Dockerfile` files across major directories (e.g., `tt-media-server`, `workflows`, `benchmarking`) and prepends the Apache-2.0 SPDX header if missing [scripts/add_spdx_header.py 20-49](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/scripts/add_spdx_header.py#L20-L49) This is automated via the `SPDX Header Checker` GitHub Action.

### Community Interaction

The `.github/workflows/on-community-issue.yml` workflow automates the labeling of issues and PRs. It detects non-organization members by checking usernames (filtering for "TT" or "tt" strings) and applies a `community` label while notifying internal Slack channels via `slack-github-action`[on-community-issue.yml 37-69](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/on-community-issue.yml#L37-L69)
