## Purpose and Scope

This guide explains how to add support for a new hardware backend to tt-tvm's LLVM-based code generation infrastructure. It covers the architecture of the LLVM backend system, the class hierarchy, key methods to implement, and practical examples from existing backends.

For information about adding Relay operators or frontend converters, see [Adding Relay Operators](https://deepwiki.com/tenstorrent/tt-tvm/11.3-adding-relay-operators) and [Adding a Frontend Converter](https://deepwiki.com/tenstorrent/tt-tvm/11.2-adding-a-frontend-converter). For general LLVM backend configuration and optimization, see [LLVM Backend Architecture](https://deepwiki.com/tenstorrent/tt-tvm/6.1-llvm-backend-architecture) and [LLVM Optimization Pipeline](https://deepwiki.com/tenstorrent/tt-tvm/6.5-llvm-optimization-pipeline).

## Backend Architecture Overview

TVM's LLVM backend uses a hierarchical class structure where platform-specific code generators inherit from base classes and override methods to customize code generation behavior.

### Factory Registration Pattern

New backends are registered through TVM's packed function registry. The factory pattern is implemented in `CodeGenLLVM::Create()`:

The factory function searches for registered code generators by target name:

1.   First attempts: `"tvm.codegen.llvm.target_" + target_name`
2.   Falls back to: `"tvm.codegen.llvm.target_cpu"` for unknown CPU targets
3.   Returns nullptr if no factory is found

Sources: [src/target/llvm/codegen_llvm.cc 126-142](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L126-L142)[src/target/llvm/codegen_cpu.cc 691-706](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L691-L706)[src/target/llvm/codegen_nvptx.cc 318-330](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L318-L330)

## Step-by-Step Implementation Guide

### Step 1: Create Code Generator Class

Create a new file `src/target/llvm/codegen_<target>.cc` and define your code generator class:

Example skeleton:

*   Inherit from `CodeGenLLVM` for GPU/accelerator targets
*   Inherit from `CodeGenCPU` for CPU-like targets with TVM runtime
*   Declare constructor and destructor (can be defaulted)
*   Override virtual methods as needed

Sources: [src/target/llvm/codegen_nvptx.cc 70-86](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L70-L86)[src/target/llvm/codegen_amdgpu.cc 88-101](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L88-L101)[src/target/llvm/codegen_hexagon.cc 73-119](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_hexagon.cc#L73-L119)

### Step 2: Implement InitTarget()

Set platform-specific configuration in `InitTarget()`:

| Configuration | Purpose | Example |
| --- | --- | --- |
| `native_vector_bits_` | Native vector register width | 512 for AVX-512, 128 for NEON |
| Target triple | LLVM target specification | Set by `LLVMTarget` |
| Data layout | Memory layout configuration | Set by `TargetMachine` |

Example from Hexagon backend:

*   Sets `native_vector_bits_` based on HVX configuration (64 or 128 bytes)
*   Parses target features to detect HVX length
*   Falls back to 64-bit scalar vectors if HVX is not available

Sources: [src/target/llvm/codegen_hexagon.cc 127-146](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_hexagon.cc#L127-L146)[src/target/llvm/codegen_llvm.cc 171-230](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L171-L230)[src/target/llvm/codegen_arm.cc 48-52](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_arm.cc#L48-L52)

### Step 3: Implement Thread Index and Synchronization

For parallel execution targets, implement thread identification and synchronization:

Thread index implementation pattern for GPU backends:

| Thread Scope | Rank | NVPTX Intrinsic | AMDGPU Intrinsic |
| --- | --- | --- | --- |
| `threadIdx.x` | 1, dim=0 | `nvvm_read_ptx_sreg_tid_x` | `amdgcn_workitem_id_x` |
| `threadIdx.y` | 1, dim=1 | `nvvm_read_ptx_sreg_tid_y` | `amdgcn_workitem_id_y` |
| `threadIdx.z` | 1, dim=2 | `nvvm_read_ptx_sreg_tid_z` | `amdgcn_workitem_id_z` |
| `blockIdx.x` | 0, dim=0 | `nvvm_read_ptx_sreg_ctaid_x` | `amdgcn_workgroup_id_x` |
| `blockIdx.y` | 0, dim=1 | `nvvm_read_ptx_sreg_ctaid_y` | `amdgcn_workgroup_id_y` |
| `blockIdx.z` | 0, dim=2 | `nvvm_read_ptx_sreg_ctaid_z` | `amdgcn_workgroup_id_z` |

Sources: [src/target/llvm/codegen_nvptx.cc 143-178](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L143-L178)[src/target/llvm/codegen_amdgpu.cc 159-195](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L159-L195)[src/target/llvm/codegen_nvptx.cc 180-192](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L180-L192)[src/target/llvm/codegen_amdgpu.cc 197-208](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L197-L208)

### Step 4: Implement Memory Allocation

Override `VisitStmt_(const AllocateNode* op)` to handle target-specific memory allocation:

Key considerations:

*   **Address spaces**: GPU targets use different address spaces (e.g., 3 for shared memory in NVPTX/AMDGPU, 5 for local)
*   **Alignment**: GPU shared memory typically requires 16-byte alignment maximum
*   **Dynamic allocation**: Handle `.dyn` storage scope for dynamically-sized shared memory
*   **Constant size checking**: GPU targets require constant allocation sizes

Sources: [src/target/llvm/codegen_nvptx.cc 87-140](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L87-L140)[src/target/llvm/codegen_amdgpu.cc 102-156](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L102-L156)[src/target/llvm/codegen_llvm.cc 720-734](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L720-L734)

### Step 5: Implement Target-Specific Intrinsics

Override `CreateIntrinsic()` to handle platform-specific operations:

Common intrinsic categories to handle:

| Category | Operations | NVPTX Example | AMDGPU Example |
| --- | --- | --- | --- |
| Atomic | `atomic_add`, `atomic_cas` | `llvm::Intrinsic::nvvm_atomic_*` | `AtomicRMW::FAdd` |
| Warp/Wave | Shuffle, ballot, active mask | `nvvm_shfl_*`, inline asm | `amdgcn_ds_swizzle` |
| Math | Fast math, special functions | `nvvm_sqrt_*` | Standard LLVM |
| Memory | Prefetch, cache control | `nvvm_prefetch_*` | `amdgcn_s_waitcnt` |

Sources: [src/target/llvm/codegen_nvptx.cc 256-310](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L256-L310)[src/target/llvm/codegen_amdgpu.cc 219-277](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L219-L277)[src/target/llvm/codegen_arm.cc 59-68](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_arm.cc#L59-L68)

### Step 6: Add Function Annotations

Override `AddFunction()` to add target-specific function attributes:

Examples of function annotations:

*   **NVPTX**: Add `nvvm.annotations` metadata marking kernel functions
*   **AMDGPU**: Set `AMDGPU_KERNEL` calling convention and `amdgpu-flat-work-group-size` attribute
*   **CPU**: Add system library registration for exports

Sources: [src/target/llvm/codegen_nvptx.cc 76-85](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L76-L85)[src/target/llvm/codegen_amdgpu.cc 93-100](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L93-L100)[src/target/llvm/codegen_cpu.cc 226-239](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_cpu.cc#L226-L239)

### Step 7: Register the Backend

Register your code generator with TVM's packed function system:

Registration examples from existing backends:

**NVPTX registration:**

*   Registered as `"tvm.codegen.llvm.target_nvptx"`
*   Returns `new CodeGenNVPTX()` from factory lambda
*   Located at end of implementation file

**AMDGPU registration:**

*   Registered as `"tvm.codegen.llvm.target_amdgpu"`
*   Returns `new CodeGenAMDGPU()` from factory lambda

**CPU architecture variants:**

*   X86-64: `"tvm.codegen.llvm.target_x86-64"`
*   ARM: `"tvm.codegen.llvm.target_arm"`
*   Hexagon: `"tvm.codegen.llvm.target_hexagon"`

## Advanced Topics

### Custom Buffer Access Patterns

Some architectures require special handling for buffer access. Override `CreateBufferPtr()` for custom pointer arithmetic:

Hexagon example handles 2-D buffer indexing:

*   First index selects a buffer chunk pointer
*   Second index indexes within the chunk
*   Enables efficient handling of 2-D physical buffers

Sources: [src/target/llvm/codegen_hexagon.cc 255-278](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_hexagon.cc#L255-L278)[src/target/llvm/codegen_llvm.cc 961-994](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L961-L994)

### Optimization Pass Customization

For LLVM versions before 16.0, override `InitPassManagerBuilder()` to customize optimization passes:

Common customizations:

*   Loop vectorization settings
*   SLP vectorization
*   Inlining thresholds
*   Target-specific passes

Note: LLVM 16.0+ uses the new pass manager and requires different customization approaches.

Sources: [src/target/llvm/codegen_llvm.cc 502](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L502-L502)[src/target/llvm/codegen_llvm.cc 413-557](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L413-L557)[src/target/llvm/codegen_nvptx.cc 196-198](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L196-L198)

### External Function Calls

Override `CreateCallExtern()` to handle external library calls:

Hexagon example shows specialized handling:

*   Detects QHL (Qualcomm Hexagon Library) functions
*   Automatically splits oversized vectors
*   Calls function on each slice
*   Concatenates results back together

## Testing Your Backend

### Unit Test Structure

Create tests in `tests/python/unittest/test_target_codegen_<target>.py`:

Test pattern example:

1.   Use `te.placeholder()` and `te.compute()` to define computation
2.   Create schedule with `te.create_schedule()`
3.   Build with `tvm.build(schedule, [...], target)`
4.   Execute on device and verify results

## Common Patterns from Existing Backends

### Pattern 1: GPU Memory Hierarchy (NVPTX/AMDGPU)

Both GPU backends follow similar patterns:

**Shared memory allocation:**

*   Use address space 3
*   Set maximum 16-byte alignment
*   Handle dynamic shared memory with external linkage
*   Static shared memory with private linkage

**Local memory allocation:**

*   Use `CreateAlloca()` for stack allocation
*   Calculate alignment based on data size
*   Verify constant size at compile time

Sources: [src/target/llvm/codegen_nvptx.cc 87-140](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L87-L140)[src/target/llvm/codegen_amdgpu.cc 102-156](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L102-L156)

### Pattern 2: CPU Host Integration (x86/ARM)

CPU backends inherit from `CodeGenCPU` to get:

**TVM runtime integration:**

*   Packed function call infrastructure
*   Parallel launch support
*   Dynamic function lookup
*   System library registration

**Architecture-specific optimizations:**

*   Override `CreateIntrinsic()` for specialized intrinsics
*   Override `VisitExpr_()` for type conversions
*   Call vector intrinsics for SIMD operations

Sources: [src/target/llvm/codegen_x86_64.cc 47-122](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_x86_64.cc#L47-L122)[src/target/llvm/codegen_arm.cc 43-107](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_arm.cc#L43-L107)

### Pattern 3: DSP/Accelerator (Hexagon)

Hexagon backend shows patterns for specialized accelerators:

**Vector processing:**

*   Custom vector length detection from target features
*   Specialized vector lookup intrinsics
*   Multi-stage vector operations (shuffle, lookup, combine)

**Memory model:**

*   2-D buffer support for structured memory
*   Custom `CreateBufferPtr()` for complex indexing
*   Structured memory for better locality

**External libraries:**

*   Integration with vendor libraries (QHL)
*   Automatic vector splitting for library functions
*   Result concatenation after processing

Sources: [src/target/llvm/codegen_hexagon.cc 73-486](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_hexagon.cc#L73-L486)

## Checklist for New Backend

Before submitting a new backend, ensure:

*   - [x]  Class inherits from appropriate base (`CodeGenLLVM` or `CodeGenCPU`)
*   - [x] `InitTarget()` sets `native_vector_bits_` correctly
*   - [x] `GetThreadIndex()` returns correct thread identifiers
*   - [x] `CreateStorageSync()` implements required barriers
*   - [x] `VisitStmt_(AllocateNode*)` handles memory allocation
*   - [x] `AddFunction()` adds required function attributes
*   - [x]  Backend registered with `TVM_REGISTER_GLOBAL`
*   - [x]  Unit tests added for basic functionality
*   - [x]  Integration test verifies end-to-end compilation
*   - [x]  Documentation added to this wiki
