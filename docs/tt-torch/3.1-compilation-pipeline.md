This page describes the multi-stage MLIR-based compilation pipeline that transforms PyTorch models into executable flatbuffer binaries for Tenstorrent hardware. The pipeline converts PyTorch FX graphs through multiple intermediate representations (torch-mlir → StableHLO → TTIR → TTNN) before generating optimized binaries.

**Related pages:**

*   For detailed transformation stages and pass pipelines, see [PyTorch to TTNN Flow](https://deepwiki.com/tenstorrent/tt-torch/3.1.1-pytorch-to-ttnn-flow)
*   For op-by-op compilation and debugging modes, see [Op-by-Op Compilation](https://deepwiki.com/tenstorrent/tt-torch/3.1.2-op-by-op-compilation)
*   For configuration options and compiler controls, see [Configuration and Control](https://deepwiki.com/tenstorrent/tt-torch/3.1.3-configuration-and-control)

*   For execution of compiled binaries, see [Executor Framework](https://deepwiki.com/tenstorrent/tt-torch/3.3-executor-framework)

### Compilation Flow Overview

The compilation pipeline transforms models through multiple stages, with each stage producing progressively lower-level representations:

| Stage | Input | Output | Key Functions | File Location |
| --- | --- | --- | --- | --- |
| **FX Graph Processing** | PyTorch FX Graph | Processed FX Graph | `pass_pipeline()`, `split_onto_devices()` | `tt_torch/dynamo/passes.py` |
| **Torch MLIR** | FX Graph | Torch MLIR IR | `import_program()`, `verify_ir()` | `tt_torch/dynamo/torch_backend.py` |
| **Torch Backend IR** | Torch MLIR | Torch Backend MLIR | `run_pipeline_with_repro_report()` `torchdynamo-export-to-torch-backend-pipeline` | `torch_mlir.compiler_utils` |
| **StableHLO** | Torch Backend IR | StableHLO IR | `lower_mlir_module(OutputType.STABLEHLO)` | `torch_mlir.compiler_utils` |
| **TTIR** | StableHLO | TTIR Dialect | `tt_mlir.compile_stable_hlo_to_ttir()` `compileStableHLOToTTIR()` | `bindings.cpp`, `tt-mlir-interface.cpp` |
| **TTNN** | TTIR | TTNN Dialect | `tt_mlir.compile_ttir_to_bytestream()` `compileTTIRToTTNN()` | `bindings.cpp`, `tt-mlir-interface.cpp` |
| **Flatbuffer** | TTNN | Binary Executable | `ttnnToFlatbuffer()` (in tt-mlir) `create_binary_from_bytestream()` | `bindings.cpp` |

### FX Graph to StableHLO

The `torch_to_shlo()` function in `backend.py` handles the initial stages of compilation from FX graph to StableHLO IR:

**torch_to_shlo() Execution Steps**

| Step | Function Call | Purpose | Key Operations |
| --- | --- | --- | --- |
| 1. Graph passes | `pass_pipeline(gm, example_inputs, compiler_config)` | Apply decompositions, constant folding, device splitting | Returns `MultiChipGraph` with per-device programs |
| 2. Import to Torch MLIR | `import_program(program)` | Convert `ExportedProgram` to Torch MLIR `ModuleOp` | Imports FX graph as Torch dialect ops |
| 3. Verify IR | `verify_ir(module)` | Validate Torch MLIR IR structure | Ensures IR is well-formed |
| 4. Lower to backend | `run_pipeline_with_repro_report(module,` `"torchdynamo-export-to-torch-backend-pipeline")` | Convert Torch dialect to Torch Backend dialect | Applies torch-mlir passes |
| 5. Convert to StableHLO | `lower_mlir_module(False, OutputType.STABLEHLO, module)` | Lower Torch Backend to StableHLO | Final frontend lowering step |

The function iterates over each device in the `MultiChipGraph` and stores the resulting StableHLO module in `mcg.shlo_modules[device_idx]`.

**Sources:**[tt_torch/dynamo/backend.py 197-226](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L197-L226)[tt_torch/dynamo/passes.py 625-690](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/passes.py#L625-L690)

### Compilation Parameters

The TTIR to TTNN compilation (`compile_ttir_to_bytestream`) accepts several configuration parameters:

| Parameter | Type | Default | Purpose | Set By |
| --- | --- | --- | --- | --- |
| `system_desc_path` | `std::string_view` | Required | Path to hardware system description flatbuffer | Executor's `system_desc_paths[device_idx]` |
| `len_activations` | `size_t` | 0 | Number of activation tensors (user inputs + buffers) | `len(mcg.example_inputs[i]) + len(mcg.buffers[i])` |
| `len_graph_constants` | `size_t` | 0 | Number of constant/parameter tensors | `len(mcg.constant_inputs[i])` |
| `enable_consteval` | `bool` | `true` | Enable constant evaluation during compilation | `compiler_config.enable_consteval` |
| `enable_optimizer` | `bool` | `false` | Enable OpModel-based optimization passes (requires `TTMLIR_ENABLE_OPMODEL=ON`) | `compiler_config.enable_optimizer` |

The function returns a tuple of `(py::bytes binary, std::string ttnn_ir)` where the binary is a flatbuffer-encoded executable and the string is the TTNN dialect IR.

### Backend Selection Logic

The `backend()` function routes compilation based on `compile_depth` and `op_by_op_backend`:

**Backend Routing Decision Tree**

| `compile_depth` | `op_by_op_backend` | Handler Function | Executor Type | Behavior |
| --- | --- | --- | --- | --- |
| `COMPILE_OP_BY_OP` | `OpByOpBackend.TORCH` | `_torch_backend()` | `TorchExecutor` | Compile each FX node individually to binary |
| `COMPILE_OP_BY_OP` | `OpByOpBackend.STABLEHLO` | `torch_to_shlo()` + `_shlo_backend()` | `StablehloExecutor` | Convert to StableHLO, compile each op individually |
| `EXECUTE_OP_BY_OP` | `OpByOpBackend.TORCH` | `_torch_backend()` | `TorchExecutor` | Compile and execute each FX node |
| `EXECUTE_OP_BY_OP` | `OpByOpBackend.STABLEHLO` | `torch_to_shlo()` + `_shlo_backend()` | `StablehloExecutor` | Compile and execute each StableHLO op |
| `TORCH_FX` | N/A | `_base_backend()` | `Executor` | Return after FX graph processing, no MLIR compilation |
| `STABLEHLO` | N/A | `_base_backend()` | `Executor` | Compile to StableHLO only, populate `mcg.shlo_modules` |
| `TTNN_IR` | N/A | `_base_backend()` | `Executor` | Compile through TTNN, generate binaries in `mcg.binaries` |
| `EXECUTE` | N/A | `_base_backend()` | `Executor` | Full compilation and prepare for execution |

The `_base_backend()` function checks the achieved compile depth and records it via `compiler_config.record_property("achieved_compile_depth", depth)`.

**Sources:**[tt_torch/dynamo/backend.py 338-369](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L338-L369)[tt_torch/dynamo/backend.py 273-310](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L273-L310)[tt_torch/dynamo/backend.py 292-309](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L292-L309)

**Core Configuration Fields**

| --- | --- | --- | --- |

| `enable_consteval` | `bool` | `False` | Enables constant evaluation optimization pass |

| `automatic_parallelization` | `bool` | `False` | Enables automatic data parallelization via Shardy |
| `mesh_shape` | `List[int]` | `[1, 1]` | Mesh shape for automatic parallelization |

### IR Dumping and Debugging

The `dump_module()` and `dump_binary()` helper functions in `backend.py` save intermediate representations to disk based on configuration.

**IR Dumping Configuration**

| Environment Variable | Valid Values | Behavior | Output Location |
| --- | --- | --- | --- |
| `TT_TORCH_SAVE_MLIR` | Comma-separated: `"STABLEHLO"`, `"TTIR"`, `"TTNN"` | Saves specified IR dialects to files | `{output_mlir_dir}/{model_name}_{dialect}.mlir` |
| `TT_TORCH_SAVE_BINARY` | `1` or `0` | Saves compiled flatbuffer binary | `{output_binary_dir}/{model_name}.ttnn` |
| `TT_TORCH_IR_LOG_LEVEL` | `"DEBUG"` or `"INFO"` | Enables pass manager IR printing to stderr | Console output |

**dump_module() Function Behavior**

The `dump_module(module, name, compiler_config)` function:

1.   If `compiler_config.dump_info` is enabled, prints the module to stderr
2.   If `name.lower()` matches any entry in `compiler_config.save_mlir_override`, appends the module's MLIR to the corresponding file
3.   Uses `sanitize_filename()` to ensure safe filenames
4.   Creates output directories if they don't exist

**dump_binary() Function Behavior**

The `dump_binary(binary, compiler_config)` function:

1.   Only executes if `compiler_config.dump_binary` is `True`
2.   Appends the binary bytestream to `{output_binary_dir}/{model_name}.ttnn`
3.   Multiple graph modules compiled in sequence will append to the same file

**C++ PassManager IR Printing**

When `TT_TORCH_IR_LOG_LEVEL=DEBUG` is set:

*   C++ compilation functions check `std::getenv("TT_TORCH_IR_LOG_LEVEL")`
*   If set to `"DEBUG"`, they call `pm.enableIRPrinting()` on the MLIR `PassManager`
*   This prints IR before and after each pass to stderr
