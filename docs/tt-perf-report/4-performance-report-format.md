The Performance Report Format describes the structure, content, and interpretation of the output generated by the tt-perf-report tool. This document explains how performance data is organized, what metrics are presented, and how to understand the performance insights provided. For information on how the tool classifies operations, see [Performance Classification System](https://deepwiki.com/tenstorrent/tt-perf-report/5-performance-classification-system), and for details on implementing suggested optimizations, see [Optimization Advice System](https://deepwiki.com/tenstorrent/tt-perf-report/6-optimization-advice-system).

## 1. Report Structure Overview

The performance report consists of two main sections:

1.   **Performance Table**: A tabular representation of operations with metrics and classifications
2.   **Advice Section**: Targeted optimization recommendations based on the performance analysis

## 2. Performance Table Format

The performance table displays operations and their metrics in a columnar format. Each row represents an operation, and the columns provide different performance characteristics.

Here's a breakdown of the columns included in the performance table:

| Column | Description | Unit | Calculation |
| --- | --- | --- | --- |
| ID | Original row number from CSV trace | Integer | CSV row number + 2 (accounting for 1-based indexing and header) |
| Total % | Percentage of total execution time | % | `(device_time + op_to_op_gap) / total_duration * 100` |
| Bound | Performance classification | Text | Based on DRAM % and FLOPs % (DRAM, FLOP, BOTH, SLOW, or HOST) |
| OP Code | Operation type and parameters | Text | Enhanced with size information for specific ops |
| Device Time | Time spent executing on device | μs | Raw value from trace file / 1000 (ns to μs) |
| Op-to-Op Gap | Time between operations | μs | Raw value from trace file / 1000 (ns to μs) |
| Cores | Number of cores used | Integer | Raw value from trace file (adjusted for DRAM-sharded ops) |
| DRAM | Memory bandwidth achieved | GB/s | `total_data_size_bytes / duration_s / 1e9` |
| DRAM % | Percentage of peak DRAM bandwidth | % | `(dram_speed_gb_s / 288) * 100` (288 GB/s is Wormhole peak) |
| FLOPs | Compute throughput achieved | TFLOPs | Calculated based on operation dimensions and duration |
| FLOPs % | Percentage of peak compute | % | `(flops / peak_flops_value) * 100` |
| Math Fidelity | Precision configuration | Text | Format: `HiFi4/HiFi2/LoFi input_dtype x weight_dtype => output_dtype` |

The table includes a summary row at the bottom showing totals and aggregated statistics.

## 3. Bound Classifications

Operations are classified into different "bound" categories based on their resource utilization. This classification helps identify the nature of performance bottlenecks.

| Classification | Meaning | Criteria | Visual Indicator |
| --- | --- | --- | --- |
| DRAM | Memory bandwidth bound | DRAM% ≥ 65%, FLOPs% < 65% | Green DRAM metrics |
| FLOP | Compute bound | FLOPs% ≥ 65%, DRAM% < 65% | Green FLOPs metrics |
| BOTH | Both memory and compute bound | DRAM% ≥ 65% and FLOPs% ≥ 65% | Green DRAM and FLOPs metrics |
| SLOW | Neither memory nor compute bound | DRAM% < 65% and FLOPs% < 65% | Yellow metrics for whichever is higher |
| HOST | Running on host CPU | Operation contains "(torch)" | No specific color |

## 4. Visual Formatting and Color Coding

The performance report uses formatting and color coding to highlight important information and performance characteristics. This section explains the visual cues used in the report output.

### 4.1 Operation Type Coloring

Different operation types are assigned distinct colors to make them easy to identify:

| Operation Type | Color in Terminal |
| --- | --- |
| Matmul | Magenta |
| OptimizedConvNew | Orange |
| LayerNorm, AllGather, AllReduce | Cyan |
| ScaledDotProductAttention, NlpCreateHeads, etc. | Blue |
| Host operations (torch) | Red |
| Other operations | White |

### 4.2 Performance Indicator Coloring

Metrics are colored based on their performance characteristics:

| Metric | Coloring Rule |
| --- | --- |
| Cores | Red if < 10, Green if 64, Grey if N/A |
| DRAM & DRAM % | Green if DRAM-bound, Yellow if SLOW and DRAM% > FLOPs% |
| FLOPs & FLOPs % | Green if FLOP-bound, Yellow if SLOW and FLOPs% > DRAM% |
| Op-to-Op Gap | Red if > 6.5μs |
| Math Fidelity | Green if sufficient, Red if too high, Cyan if too low |

### 4.3 Low-Impact Operation Dimming

Operations that contribute less than a specified percentage (default: 0.5%) to the total execution time are dimmed (grey) to focus attention on significant operations.

## 5. Advice Section Format

The Advice section of the performance report provides optimization recommendations based on the analysis. It is divided into three subsections:

### 5.1 Fallback Advice

This section identifies operations running on the host CPU (marked with "(torch)") that should be moved to the device for better performance.

Example format:

```
Fallback
--------
[operation rows...]

These ops should be moved to run on device.
```

### 5.2 High Op-to-Op Gap Advice

This section identifies operations with high latency between operations (>6.5μs), which often indicates host overhead or inefficient kernel dispatching.

Example format:

```
High Op-to-Op Gap
----------------
[operation rows...]

These ops have a >6us gap since the previous operation. Running with tracing could save X us (Y% of overall time)
Alternatively ensure device is not waiting for the host and use device.enable_async(True). Experts can try moving runtime args in the kernels to compile-time args.
```

### 5.3 Matmul Optimization Advice

This section provides specific recommendations for matrix multiplication operations based on their performance characteristics.

Example format:

```
Matmul Optimization
-------------------
[operation row]
- [specific advice item 1]
- [specific advice item 2]
...or...
✅ Optimized

[next operation row]
...
```

## 6. CSV Output Format

When using the `--csv` option, the performance report can be exported to a CSV file. The CSV output includes all visible columns from the performance table, plus additional columns not shown in the standard console output:

| Additional Column | Description |
| --- | --- |
| Output Datatype | Datatype of the operation output |
| Input 0 Datatype | Datatype of the first input |
| Input 1 Datatype | Datatype of the second input |
| DRAM Sharded | Whether the operation uses DRAM sharding |
| Input 0 Memory | Memory type for first input |
| Inner Dim Block Size | in0_block_w from program config |
| Output Subblock H | out_subblock_h from program config |
| Output Subblock W | out_subblock_w from program config |
| Advice | Concatenated advice for the operation (if --no-advice is not used) |
| Raw OP Code | Unprocessed operation code (if --raw-op-codes is used) |

## 7. Interpretation Guide

This section provides guidance on how to interpret the performance report for different types of operations.

### 7.1 Understanding Matrix Multiplication Performance

Matrix multiplications (Matmul operations) are the most thoroughly analyzed operations in the report. Key aspects to look for:

*   **DRAM-bound operations**: Good utilization of memory bandwidth (green DRAM metrics)

    *   If not using DRAM sharding, consider enabling it
    *   Look for advice about math fidelity adjustments

*   **FLOP-bound operations**: Good utilization of compute resources (green FLOPs metrics)

    *   If using fewer than 64 cores, consider increasing the grid size
    *   Check if math fidelity can be reduced for higher throughput

*   **SLOW operations**: Neither memory nor compute bound (yellow metrics)

    *   Check memory layout (L1 vs DRAM)
    *   Examine block sizes (in0_block_w, out_subblock_h/w)
    *   Consider adjusting math fidelity

### 7.2 Understanding Convolution Performance

For OptimizedConvNew operations, the report includes additional configuration details in brackets:

*   **ABH**: Per-core output matrix height in tiles
*   **ADB**: ActivationDoubleBuffer enabled
*   **SR**: SplitReader enabled

### 7.3 Understanding Collective Operations

Collective operations like AllGather, AllReduce, and ReduceScatter show the average duration across all devices, helping identify communication bottlenecks in multi-device setups.

### 7.4 High Op-to-Op Gaps

High op-to-op gaps (>6.5μs, shown in red) indicate potential host overhead or kernel launch inefficiencies. The advice section calculates the potential time savings from reducing these gaps.

## Summary

The tt-perf-report tool provides a comprehensive performance analysis through its formatted output. The performance table clearly identifies operations, their resource utilization, and bottlenecks, while the advice section offers specific recommendations for optimization. Understanding this report format is essential for effectively optimizing machine learning workloads on Tenstorrent hardware.

For detailed information on how operations are classified, see the [Performance Classification System](https://deepwiki.com/tenstorrent/tt-perf-report/5-performance-classification-system) page, and for implementation details of recommended optimizations, refer to the [Optimization Advice System](https://deepwiki.com/tenstorrent/tt-perf-report/6-optimization-advice-system) page.
