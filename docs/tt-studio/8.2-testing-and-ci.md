This page documents the testing and continuous integration (CI) workflows implemented in TT-Studio. The CI system focuses on license compliance verification, code quality checks (linting), and providing automated feedback on Pull Requests.

## CI Workflow Architecture

TT-Studio employs GitHub Actions for continuous integration. The system ensures that all contributions adhere to the project's licensing standards and frontend coding conventions.

Sources: [.github/workflows/frontend-lint-license-checker.yml 1-15](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L1-L15)[.github/workflows/backend-license-checker.yml 1-16](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/backend-license-checker.yml#L1-L16)

## Frontend CI Workflow

The frontend CI workflow performs targeted linting and SPDX license header verification. To optimize runtime, it primarily analyzes files changed within the specific PR or push event.

### Implementation Details

1.   **Git History Fetching**: The workflow fetches full history to perform a reliable `git diff` between the PR branch and the base branch (e.g., `main`) [.github/workflows/frontend-lint-license-checker.yml 24-49](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L24-L49)
2.   **Targeted Linting**: It identifies changed files using `git diff --name-only --diff-filter=AM` and filters for frontend extensions (`.ts`, `.tsx`, `.js`, `.jsx`) [.github/workflows/frontend-lint-license-checker.yml 77-94](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L77-L94)
3.   **Header Verification**: Runs `npm run header:check:changed` which enforces the current year in the SPDX copyright notice [.github/workflows/frontend-lint-license-checker.yml 52-59](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L52-L59)
4.   **ESLint Execution**: Executes `npx eslint` on the filtered list of changed files [.github/workflows/frontend-lint-license-checker.yml 128-130](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L128-L130)

### Automated Feedback

The workflow parses ESLint output using `sed` to remove ANSI escape codes and `grep` to identify specific error lines [.github/workflows/frontend-lint-license-checker.yml 141-165](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L141-L165) If errors are found, it uses `actions/github-script@v7` to post a detailed comment on the Pull Request [.github/workflows/frontend-lint-license-checker.yml 180-210](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L180-L210)

Sources: [.github/workflows/frontend-lint-license-checker.yml 62-176](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L62-L176)

## Quality Assurance Procedures

### Backend Test Patterns

While the CI primarily checks licenses, the codebase includes patterns for backend testing, specifically for model inference servers. The `dummy_echo_model` serves as a reference for backend test implementation using `unittest.mock` and `queue.Queue` to simulate the inference pipeline.

*   **Mocking**: Use of `unittest.mock.Mock` and `patch` to isolate backend logic [.models/dummy_echo_model/src/test_dummy_backend.py 9](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.models/dummy_echo_model/src/test_dummy_backend.py#L9-L9)
*   **Pipeline Testing**: The `test_dummy_backend()` function demonstrates initializing `prompt_q`, `output_q`, and `status_q` to verify the `run_backend` loop [.models/dummy_echo_model/src/test_dummy_backend.py 16-24](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.models/dummy_echo_model/src/test_dummy_backend.py#L16-L24)

### Logging and Monitoring

Tests and production services use a standardized logging utility defined in `inference_logger.py`, which configures both `FileHandler` and `StreamHandler` for capturing execution details in `python_logs/`[.models/dummy_echo_model/src/inference_logger.py 14-35](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.models/dummy_echo_model/src/inference_logger.py#L14-L35)

## Local Verification

Developers should run checks locally before pushing to minimize CI cycles:

1.   **Frontend Linting**: Navigate to `app/frontend` and run `npm run lint`[.github/workflows/frontend-lint-license-checker.yml 133](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L133-L133)
2.   **License Headers**: 
    *   Frontend: `npm run header:check:changed`[.github/workflows/frontend-lint-license-checker.yml 59](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/frontend-lint-license-checker.yml#L59-L59)
    *   Backend: Install `check-copyright` and run `python -m check_copyright --config ./check_copyright_config.yaml .`[.github/workflows/backend-license-checker.yml 79](https://github.com/tenstorrent/tt-studio/blob/a6d347af/.github/workflows/backend-license-checker.yml#L79-L79)
