## Purpose and Scope

The Topology Discovery system is responsible for enumerating all Tenstorrent chips connected to the system and constructing a complete topology map. It detects both locally-attached chips (via PCIe or JTAG) and remotely-connected chips (via Ethernet fabric), validates firmware versions, and populates a `ClusterDescriptor` with the discovered topology information.

This page covers the multi-stage discovery pipeline, architecture-specific implementations for Wormhole and Blackhole, firmware verification, and Ethernet retraining logic.

## Discovery Pipeline Overview

The discovery process is initiated via the static `TopologyDiscovery::discover` method, which follows a four-stage pipeline.

**Topology Discovery Flow**

1.   **Hardware Enumeration**: Identifies all chips with direct host access (PCIe/JTAG).
2.   **Ethernet Retraining**: (Optional) Attempts to recover failed Ethernet links.
3.   **Remote Discovery**: Traverses the Ethernet fabric to find chips not directly connected to the host.
4.   **Descriptor Population**: Aggregates hardware data into a `ClusterDescriptor` for use by the UMD.

**Sources:**[device/topology/topology_discovery.cpp 89-115](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L89-L115)[device/api/umd/device/topology/topology_discovery.hpp 46-60](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery.hpp#L46-L60)

## Configuration and Options

The `TopologyDiscoveryOptions` struct controls the strictness and behavior of the discovery process.

| Option | Type | Default | Description |
| --- | --- | --- | --- |
| `cmfw_mismatch_action` | `Action` | `THROW` | Action if Chip Management FW version mismatch is found. |
| `eth_fw_mismatch_action` | `Action` | `THROW` | Action if Ethernet FW version mismatch is found. |
| `eth_fw_heartbeat_failure` | `Action` | `THROW` | Action if an Ethernet core heartbeat check fails. |
| `device_init_failure_action` | `Action` | `THROW` | If `IGNORE`, marks failed devices as "unhealthy" instead of halting. |
| `discover_remote_devices` | `bool` | `true` | Enables/disables Ethernet fabric traversal. |
| `perform_6u_eth_retrain` | `bool` | `false` | Enables retraining of failed links on 6U (Galaxy) systems. |

## Stage 1: Hardware Enumeration

The system first detects local devices using the specified `IODeviceType`.

**Hardware Enumeration Mapping**

*   **PCIe**: Calls `PCIDevice::enumerate_devices()` to find all Tenstorrent cards on the bus. [device/topology/topology_discovery.cpp 121-123](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L121-L123)
*   **JTAG**: Uses `JtagDevice` to count devices in the JTAG chain. [device/topology/topology_discovery.cpp 125-129](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L125-L129)
*   **Architecture Detection**: The `TopologyDiscovery::create_topology_discovery` factory determines whether to instantiate `TopologyDiscoveryWormhole` or `TopologyDiscoveryBlackhole` based on the architecture of the first discovered device. [device/topology/topology_discovery.cpp 45-83](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L45-L83)

## Stage 2: Ethernet Retraining

If `perform_6u_eth_retrain` is enabled, the driver attempts to fix links that failed training during boot.

*   **Wormhole**: Implementation checks `LINK_TRAIN_SUCCESS` status and can attempt up to 3 retries (`ETH_RETRAIN_ATTEMPT_COUNT`). It can also retrain `UNCONNECTED` ports if `RETRAIN_UNCONNECTED` is set. [device/api/umd/device/topology/topology_discovery_wormhole.hpp 75-86](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery_wormhole.hpp#L75-L86)
*   **Blackhole**: Currently implements a placeholder for retraining. [device/topology/topology_discovery_blackhole.cpp 63](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_blackhole.cpp#L63-L63)

## Stage 3: Remote Discovery

Remote discovery uses a breadth-first search (BFS) to traverse the Ethernet fabric.

1.   **Gateway Identification**: Local devices act as gateways.
2.   **Link Traversal**: For every active Ethernet port on a known chip, the driver reads the remote chip's `asic_id` and `board_id`.
3.   **Remote Device Creation**: If a new `asic_id` is found, a `TTDevice` with `RemoteCommunication` is created.
4.   **Firmware Verification**: 
    *   **Bundle Version**: All chips in the cluster must have matching firmware bundle versions. [device/api/umd/device/topology/topology_discovery.hpp 154-164](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery.hpp#L154-L164)
    *   **Heartbeat**: The driver verifies the `erisc` firmware heartbeat to ensure the Ethernet core is responsive. [device/api/umd/device/topology/topology_discovery.hpp 84-86](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery.hpp#L84-L86)

## Architecture-Specific Discovery

### Wormhole Implementation

Wormhole discovery relies on the `erisc` firmware's `RESULTS_BUF` and `NODE_INFO` structures.

*   **ASIC ID**: Fetched from `ERISC_LOCAL_BOARD_ID_LO_OFFSET`. [device/topology/topology_discovery_wormhole.cpp 67-81](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_wormhole.cpp#L67-L81)
*   **Ethernet Coordinates**: Uses `EthCoord` (rack, shelf, x, y) to identify position in Galaxy systems. [device/topology/topology_discovery_wormhole.cpp 99-112](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_wormhole.cpp#L99-L112)
*   **Remote Board ID**: Mangled ARC board ID is used to uniquely identify boards. [device/topology/topology_discovery_wormhole.cpp 37-50](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_wormhole.cpp#L37-L50)

**Sources:**[device/topology/topology_discovery_wormhole.cpp 33-162](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_wormhole.cpp#L33-L162)[device/api/umd/device/topology/topology_discovery_wormhole.hpp 19-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery_wormhole.hpp#L19-L87)

### Blackhole Implementation

Blackhole uses a different memory map and does not utilize `EthCoord`.

*   **Board ID**: Read from fixed offsets `0x7CFC8` (lo) and `0x7CFC4` (hi). [device/topology/topology_discovery_blackhole.cpp 67-84](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_blackhole.cpp#L67-L84)
*   **Mangled ASIC ID**: On non-6U systems, the ASIC ID is generated by mangling the board ID with the ASIC location: `(board_id << 5) | (asic_location & 0x1F)`. [device/topology/topology_discovery_blackhole.cpp 150-152](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_blackhole.cpp#L150-L152)
*   **Patching Connections**: `patch_eth_connections` is used to map physical Ethernet channels to logical channels, accounting for harvesting. [device/topology/topology_discovery_blackhole.cpp 154-173](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_blackhole.cpp#L154-L173)

**Sources:**[device/topology/topology_discovery_blackhole.cpp 35-152](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery_blackhole.cpp#L35-L152)[device/api/umd/device/topology/topology_discovery_blackhole.hpp 18-64](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/topology/topology_discovery_blackhole.hpp#L18-L64)

## Stage 4: ClusterDescriptor Population

The final stage converts the internal maps of `TTDevice` objects and Ethernet connections into a `ClusterDescriptor`.

*   **Chip ID Assignment**: Chips are assigned integer `ChipId` values. Local (MMIO) chips are typically assigned first. [device/topology/topology_discovery.cpp 230-245](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L230-L245)
*   **Harvesting Information**: Harvesting masks (Tensix, DRAM, ETH) are read from the devices and stored in the descriptor. [device/topology/topology_discovery.cpp 265-285](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L265-L285)
*   **Unhealthy Devices**: If `device_init_failure_action` is `IGNORE`, devices that failed to initialize are added to the unhealthy devices list. [device/topology/topology_discovery.cpp 173-181](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/topology/topology_discovery.cpp#L173-L181)
