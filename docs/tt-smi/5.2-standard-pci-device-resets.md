## Overview

This page documents the standard PCI device reset functionality in TT-SMI, specifically the `pci_board_reset` function and its device-specific reset handlers. Standard PCI resets are used for Wormhole and Blackhole devices connected via PCIe, and provide board-level or ASIC-level reset capabilities. For Galaxy 6U system resets (IPMI-based), see page 5.3.

## Supported Device Types

Standard PCI resets support the following Tenstorrent devices:

| Device Type | Reset Handler | Reset Level | Method |
| --- | --- | --- | --- |
| Wormhole (WH) | `WHChipReset` | Board-level | `full_lds_reset()` |
| Blackhole (BH) | `BHChipReset` | ASIC-level | `full_lds_reset()` |

**Note**: Grayskull devices are deprecated as of v3.0.35 and are no longer supported.

## pci_board_reset Function Architecture

The `pci_board_reset` function is the primary entry point for standard PCI device resets. It coordinates device-specific reset handlers based on detected device types:

**pci_board_reset Function Flow**:

1.   **Device Enumeration**: Iterate through provided PCI indices
2.   **Type Detection**: Initialize `PciChip` and check device type via `chip.as_wh()` or `chip.as_bh()`
3.   **List Segregation**: Separate Wormhole and Blackhole device indices
4.   **Galaxy Detection**: Check board types to determine if secondary bus reset should be used
5.   **Reset Execution**: Call appropriate reset handler with collected indices
6.   **Reinitialization**: Optionally re-detect chips after reset

Sources: [tt_smi/tt_smi_backend.py 733-807](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L733-L807)

**WHChipReset.full_lds_reset() Sequence**:

**Key Characteristics**:

*   **Ethernet Training**: Automatically retrains Ethernet connections post-reset

## BHChipReset Handler (Blackhole)

The `BHChipReset` class from `tt_tools_common.reset_common.bh_reset` provides ASIC-level reset functionality for Blackhole devices:

**BHChipReset.full_lds_reset() Sequence**:

**Key Characteristics**:

*   **Reset Level**: ASIC-level reset (more targeted than board-level)
*   **No Ethernet Training**: Blackhole uses different network topology
*   **PCIe Re-enumeration**: Waits for device to reappear on PCIe bus
*   **Parameter**: `secondary_bus_reset` flag controls whether secondary bus reset is performed

## Galaxy System Handling

For Galaxy 6U systems (board types `0x35` for Wormhole, `0x47` for Blackhole), the `pci_board_reset` function automatically adjusts reset behavior:

**Galaxy-Specific Behavior**:

*   **Secondary Bus Reset Disabled**: Galaxy systems require `secondary_bus_reset=False` due to hardware configuration
*   **CPLD Requirement**: Galaxy systems need CPLD FW v1.16+ for `tt-smi -r` to work
*   **Fallback Option**: If `tt-smi -r` fails, users should use `tt-smi -glx_reset` instead (see page 5.3)

Sources: [tt_smi/tt_smi_backend.py 768-780](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L768-L780)

## Post-Reset Reinitialization

After executing device resets, the `pci_board_reset` function can optionally reinitialize devices to verify successful reset:

**Reinitialization Flow**:

**Reinitialization Process**:

1.   **Enable Debugging**: Sets `RUST_BACKTRACE=full` environment variable
2.   **Chip Detection**: Calls `detect_chips_with_callback()` to re-detect devices
3.   **Validation**: Compares number of detected chips to pre-reset count
4.   **Error Handling**: Exits with error code 1 if reinitialization fails

## Reset Operation Constraints and Considerations

When performing reset operations, consider the following:

1.   **Grayskull Tensix Reset**: Maintains PCIe connection but resets computation cores
2.   **Wormhole/Blackhole Board Reset**: Performs a complete board-level reset
3.   **UBB Reset**: Affects all boards connected to the UBB
4.   **Galaxy/Motherboard Reset**: Resets entire motherboards containing multiple chips

Reset operations may affect running workloads and require reinitialization of the software stack after completion.
