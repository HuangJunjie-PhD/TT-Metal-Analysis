# Cache Operation Scenarios

## Purpose and Scope

This page documents test scenarios for RISC-V cache management operations defined by the ZICBOM (Cache Block Management), ZICBOZ (Cache Block Zero), and ZICBOP (Cache Block Prefetch) extensions. These scenarios validate CSR-based access control, cache operation semantics, privilege mode interactions, address alignment behavior, and exception conditions.

For multi-processor cache coherence and synchronization tests, see [Multi-Processor Cache Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.1.2-multi-processor-cache-tests). For the overall test plan structure, see [Cache Management Extensions](https://deepwiki.com/tenstorrent/riscv-coretp/4.1-cache-management-extensions-(zicbomzicbozzicbop)).

## Cache Management Overview

The RISC-V cache management extensions provide three categories of operations:

| Extension | Operations | Purpose |
| --- | --- | --- |
| **ZICBOM** | `cbo.clean`, `cbo.flush`, `cbo.inval` | Cache block management for coherence |
| **ZICBOZ** | `cbo.zero` | Efficient cache block zeroing |
| **ZICBOP** | `prefetch.i`, `prefetch.r`, `prefetch.w` | Cache prefetch hints |

These operations are controlled by CSR fields in `menvcfg`, `senvcfg`, and `henvcfg` registers, enabling hierarchical access control across privilege modes.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 1-100](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L1-L100)

## CSR Access Control Architecture

The CSR control hierarchy determines whether cache operations execute or trap with an illegal instruction exception. When a control bit is cleared (set to `00` for CBIE fields), the corresponding operation raises an exception in lower privilege modes.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 12-430](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L12-L430)

## Test Scenario Categories

### CSR Control Validation

These scenarios verify that CSR control bits correctly enable or disable cache operations across privilege modes:

Each scenario follows this pattern:

1.   Allocate a memory region with `Memory` step
2.   Configure CSR bits using `CsrWrite` to disable operations (`clear_mask`)
3.   Execute the cache operation with `MemAccess`
4.   Verify `ILLEGAL_INSTRUCTION` exception with `AssertException`

**Example:**`SID_ZICBO_001` validates that `cbo.inval` traps when `menvcfg.CBIE` is cleared:

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 12-43](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L12-L43)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 46-77](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L46-L77)

### Prefetch Operation Validation

Prefetch instructions are hint operations that must not cause exceptions:

| Scenario | Instruction | Test Focus |
| --- | --- | --- |
| `SID_ZICBO_013` | `prefetch.i`, `prefetch.r`, `prefetch.w` | No traps occur |
| `SID_ZICBO_015` | `prefetch.i` | Non-zero immediate `imm[11:5]` |
| `SID_ZICBO_016` | `prefetch.r`, `prefetch.w` | Various immediate values |

**Note:** Commented scenarios (SID_ZICBO_014) indicate compiler limitations prevent testing certain immediate encodings.

## Cache Block Zero (cbo.zero) Operations

### Basic Zeroing Behavior

The `cbo.zero` instruction zeroes an entire cache block efficiently:

**Implementation Details:**

*   `SID_ZICBO_017`: Configures all environment CSRs (`menvcfg`, `senvcfg`, `henvcfg`) with `set_mask=0x80` to enable `cbo.zero`
*   `SID_ZICBO_018`: Writes test data, executes `cbo.zero`, then verifies entire block is zeroed using `Load` and `AssertEqual`

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 559-634](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L559-L634)

### Virtual Address Aliasing

Cache operations must operate on the correct physical address when multiple virtual addresses map to the same physical location:

**Key Test Pattern:**

1.   Create two `Memory` regions with `base_pa=0x10000` (same physical address)
2.   Store data through first virtual address
3.   Execute `cbo.zero` on first virtual address
4.   Verify both virtual addresses read zero

This validates that the cache operation targets the physical address, not just the virtual address used.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 637-690](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L637-L690)

### Address Alignment Behavior

Cache operations automatically align addresses to cache block boundaries:

The implementation stores data at `offset=0x1` but the cache operation automatically aligns to the cache block start, zeroing the entire block including offset 0.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 693-745](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L693-L745)

## Exception Handling

### Store-like Exception Behavior

`cbo.zero` is treated as a store operation for exception purposes:

**Test Configuration:**

*   Memory allocated with `flags=PageFlags.READ` (no write permission)
*   Environment configured with `paging_modes=[PagingMode.SV39]`
*   Exception cause is `STORE_AMO_PAGE_FAULT`, not a load fault

**Note:** The commented code indicates future expansion for `STORE_AMO_ACCESS_FAULT` testing via PMP configuration.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 800-841](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L800-L841)

## Cache Management Operation Interactions

### CMO and Standard Memory Access

Cache operations interact correctly with normal memory accesses:

This scenario validates that:

*   Cache operations complete correctly
*   Subsequent memory accesses work properly
*   No ordering violations occur

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 1014-1048](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L1014-L1048)

### CMO and Atomic Operations

Cache operations must interact correctly with Load-Reserved/Store-Conditional (LR/SC) atomics:

**Test Purpose:** Validates the interaction between cache management operations and atomic memory operations. The RISC-V specification allows implementations to break LR/SC reservations on intervening cache operations, and this test verifies correct behavior.

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 1057-1100](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L1057-L1100)

## Instruction Definitions

The cache operation instructions are defined in the instruction catalog:

| Instruction | Extension | Operands | Category |
| --- | --- | --- | --- |
| `cbo.clean` | ZICBOM | `rs1` (address) | `CACHE_OPERATION` |
| `cbo.flush` | ZICBOM | `rs1` (address) | `CACHE_OPERATION` |
| `cbo.inval` | ZICBOM | `rs1` (address) | `CACHE_OPERATION` |
| `cbo.zero` | ZICBOZ | `rs1` (address) | `CACHE_OPERATION` |
| `prefetch.i` | ZICBOP | `rs1`, `imm` (offset) | `CACHE_OPERATION` |
| `prefetch.r` | ZICBOP | `rs1`, `imm` (offset) | `CACHE_OPERATION` |
| `prefetch.w` | ZICBOP | `rs1`, `imm` (offset) | `CACHE_OPERATION` |

All instructions use a single source register `rs1` containing the target address (with optional immediate offset for prefetch instructions).

**Sources:**[coretp/isa/instructions/misc.py 1285-1371](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instructions/misc.py#L1285-L1371)

## Test Step Construction Patterns

### Common Test Pattern

Most scenarios follow this construction pattern:

### CSR Configuration Methods

Two approaches for CSR manipulation:

| Method | Usage | Example |
| --- | --- | --- |
| `clear_mask` | Disable operations | `CsrWrite(csr_name="menvcfg", clear_mask=0x30)` clears CBIE |
| `set_mask` | Enable operations | `CsrWrite(csr_name="menvcfg", set_mask=0xF0)` sets all control bits |

**Bit Positions:**

*   CBIE[5:4]: `0x30` mask
*   CBCFE[6]: `0x40` mask
*   CBZE[7]: `0x80` mask
*   All CMO bits: `0xF0` mask

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 24-26](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L24-L26)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 572-574](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L572-L574)

## Environment Configuration

Test scenarios specify execution environments through `TestEnvCfg`:

**Examples:**

*   `TestEnvCfg(priv_modes=[PrivilegeMode.U])`: U-mode only (SID_ZICBO_002)
*   `TestEnvCfg(priv_modes=[PrivilegeMode.S, PrivilegeMode.U], virtualized=[True])`: VS/VU modes (SID_ZICBO_007)
*   `TestEnvCfg(paging_modes=[PagingMode.SV39])`: SV39 paging for exception tests (SID_ZICBO_022)
*   `TestEnvCfg()`: Default environment, all modes (SID_ZICBO_013)

**Sources:**[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 37](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L37-L37)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 248](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L248-L248)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 832](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L832-L832)

## Untestable Scenarios

Multiple scenarios are commented out with "Unable to be tested" annotations:

*   `SID_ZICBO_014`: Prefetch with `imm[4:0] != 0` (compiler rejects encoding)
*   `SID_ZICBO_023` through `SID_ZICBO_038`: Various hardware-specific behaviors
*   `SID_ZICBO_040` through `SID_ZICBO_045`: Implementation-dependent semantics
*   `SID_ZICBO_048`: Additional CMO interaction scenario

These scenarios represent test cases that:

1.   Require hardware-specific behavior not observable in software
2.   Depend on implementation details (cache size, timing)
3.   Use instruction encodings rejected by standard toolchains
