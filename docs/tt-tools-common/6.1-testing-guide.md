## Purpose and Scope

This document explains the test infrastructure for tt-tools-common, including the pytest-based test suite, hardware vs. software test separation, test fixtures, local test execution, and continuous integration testing. It provides guidance for both running existing tests and writing new tests.

For information about setting up a development environment before running tests, see [Development Environment Setup](https://deepwiki.com/tenstorrent/tt-tools-common/6.4-development-environment-setup). For details on the CI/CD pipeline that executes these tests, see [CI/CD Pipeline Overview](https://deepwiki.com/tenstorrent/tt-tools-common/6.3-cicd-pipeline-overview).

## Test Suite Architecture

The test suite is organized into a single directory with modular test files, a shared fixture configuration, and pytest markers to separate hardware-dependent tests from software-only tests.

**Sources:**[tt_tools_common/tests/conftest.py 1-12](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/conftest.py#L1-L12)[tt_tools_common/tests/test_reset.py 1-15](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L1-L15)[tt_tools_common/tests/test_detect_fallible.py 1-40](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L1-L40)[tt_tools_common/tests/test_reset_config.py 1-50](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L1-L50)

### Test File Organization

| Test File | Purpose | Hardware Required |
| --- | --- | --- |
| `test_reset.py` | Tests `ChipReset.full_lds_reset()` functionality | Yes |
| `test_detect_fallible.py` | Tests `detect_chips_with_callback()` before and after reset | Yes |
| `test_reset_config.py` | Tests reset configuration parsing and log generation | Mixed (some tests) |

All test files are located in `tt_tools_common/tests/`. The `conftest.py` file provides shared fixtures available to all tests.

**Sources:**[tt_tools_common/tests/test_reset.py 1-15](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L1-L15)[tt_tools_common/tests/test_detect_fallible.py 1-40](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L1-L40)[tt_tools_common/tests/test_reset_config.py 1-50](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L1-L50)

## Test Markers

The test suite uses pytest markers to categorize tests based on hardware requirements. This enables selective test execution in environments with or without Tenstorrent hardware.

### `requires_hardware` Marker

Tests that interact with physical Tenstorrent chips must be marked with `@pytest.mark.requires_hardware`. This marker is used to:

1.   **Skip tests in CI environments without hardware** - The standard test job runs with `-m "not requires_hardware"`
2.   **Select hardware tests for execution on dedicated hardware runners** - Hardware test jobs run with `-m requires_hardware`

**Sources:**[.github/workflows/test.yml 36-38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L36-L38)[.github/workflows/test.yml 64-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L64-L66)

### Example Usage

Hardware-dependent test example from [tt_tools_common/tests/test_reset.py 10-14](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L10-L14):

`@pytest.mark.requires_hardwaredef test_full_lds_reset(devices):    """Test full LDS reset on all detected chips."""    pci_interfaces = list(range(len(devices)))    ChipReset().full_lds_reset(pci_interfaces=pci_interfaces)`
Software-only test example from [tt_tools_common/tests/test_reset_config.py 16-20](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L16-L20):

`def test_parse_reset_input_pci_indices():    """Test parsing comma-separated PCI indices."""    result = parse_reset_input(["0,1,2,3"])    assert result.type == ResetType.ID_LIST    assert result.value == [0, 1, 2, 3]`
**Sources:**[tt_tools_common/tests/test_reset.py 10-14](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L10-L14)[tt_tools_common/tests/test_reset_config.py 16-20](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L16-L20)

## Test Fixtures

Fixtures provide reusable test setup and data. The test suite uses pytest fixtures defined in `conftest.py` and individual test files.

**Sources:**[tt_tools_common/tests/conftest.py 1-12](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/conftest.py#L1-L12)[tt_tools_common/tests/test_detect_fallible.py 13-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L13-L21)

### `devices` Fixture

Defined in [tt_tools_common/tests/conftest.py 8-11](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/conftest.py#L8-L11) this fixture provides a list of detected Tenstorrent chips:

`@pytest.fixture()def devices():    """Return devices detected on the system."""    return detect_chips()`
**Usage:** Tests that need to enumerate or interact with all detected chips should accept the `devices` parameter. Example from [tt_tools_common/tests/test_reset.py 11-14](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L11-L14)

**Sources:**[tt_tools_common/tests/conftest.py 8-11](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/conftest.py#L8-L11)

### `pci_indices` Fixture

Defined in [tt_tools_common/tests/test_detect_fallible.py 13-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L13-L21) this fixture provides a list of local (non-remote) chip PCI indices:

`@pytest.fixturedef pci_indices():    """Fixture that returns list of chip PCI indices"""    pci_idx = []    devices = detect_chips()    for i, dev in enumerate(devices):        if not dev.is_remote():            pci_idx.append(i)    return pci_idx`
**Usage:** Tests that perform operations requiring local PCI access should use this fixture. Example from [tt_tools_common/tests/test_detect_fallible.py 32-39](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L32-L39)

**Sources:**[tt_tools_common/tests/test_detect_fallible.py 13-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L13-L21)

## Running Tests Locally

### Installing Development Dependencies

Before running tests, install tt-tools-common in editable mode with development dependencies:

`pip install -e ".[dev]"`
This installs pytest and other development tools specified in `pyproject.toml`.

**Sources:**[.github/workflows/test.yml 32-34](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L32-L34)

### Running All Tests

Execute the entire test suite:

`pytest tt_tools_common/tests/ -v`
The `-v` flag enables verbose output showing individual test results.

**Sources:**[.github/workflows/test.yml 36-38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L36-L38)

### Running Software Tests Only

On systems without Tenstorrent hardware, run only software tests:

`pytest tt_tools_common/tests/ -v -m "not requires_hardware"`
This skips all tests marked with `@pytest.mark.requires_hardware`.

**Sources:**[.github/workflows/test.yml 36-38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L36-L38)

### Running Hardware Tests Only

On systems with Tenstorrent hardware and appropriate drivers installed:

`pytest tt_tools_common/tests/ -v -m requires_hardware`
This runs only tests that require physical hardware.

**Sources:**[.github/workflows/test.yml 64-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L64-L66)

### Running Specific Test Files

Execute tests from a single file:

`pytest tt_tools_common/tests/test_reset_config.py -v`
### Running Individual Tests

Run a specific test function:

`pytest tt_tools_common/tests/test_reset_config.py::test_parse_reset_input_pci_indices -v`
**Sources:**[tt_tools_common/tests/test_reset_config.py 16-20](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L16-L20)

## Continuous Integration Testing

The test suite runs automatically in GitHub Actions on every push to `main` and on all pull requests. The CI pipeline uses a matrix strategy to test across multiple Python versions and hardware configurations.

**Sources:**[.github/workflows/test.yml 1-67](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L1-L67)

### Test Workflow Configuration

The test workflow is defined in [.github/workflows/test.yml 1-67](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L1-L67) It consists of two jobs:

| Job | Python Versions | Runners | Test Selection |
| --- | --- | --- | --- |
| `test` | 3.10, 3.11, 3.12 | `ubuntu-latest` | `-m "not requires_hardware"` |
| `test-hardware` | 3.10 | `tt-ubuntu-2204-n150-stable` `tt-ubuntu-2204-p150b-stable` | `-m requires_hardware` |

Both jobs use `fail-fast: false` to ensure all matrix combinations run even if one fails.

**Sources:**[.github/workflows/test.yml 14-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L14-L21)[.github/workflows/test.yml 40-49](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L40-L49)

### Python Version Matrix

Software tests run on three Python versions to ensure compatibility:

*   Python 3.10
*   Python 3.11
*   Python 3.12

This is configured in [.github/workflows/test.yml 18-20](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L18-L20)

**Sources:**[.github/workflows/test.yml 14-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L14-L21)

### Hardware Test Infrastructure

Hardware tests execute on dedicated self-hosted runners with Tenstorrent hardware installed:

*   `tt-ubuntu-2204-n150-stable` - N150 hardware configuration
*   `tt-ubuntu-2204-p150b-stable` - P150B hardware configuration

These runners are configured in [.github/workflows/test.yml 45-48](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L45-L48) Hardware tests use Python 3.10 only, as shown in [.github/workflows/test.yml 54-57](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L54-L57)

**Sources:**[.github/workflows/test.yml 40-67](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L40-L67)

### Workflow Execution Steps

Each test job follows this sequence:

1.   **Checkout repository** - `actions/checkout@v4`
2.   **Set up Python** - `actions/setup-python@v5` with matrix version
3.   **Install dependencies** - `pip install -e ".[dev]"`
4.   **Run tests** - `pytest tt_tools_common/tests/ -v` with appropriate marker filter

**Sources:**[.github/workflows/test.yml 22-38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L22-L38)[.github/workflows/test.yml 50-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.github/workflows/test.yml#L50-L66)

## Writing New Tests

### Test File and Function Naming

*   Test files must begin with `test_` (e.g., `test_reset.py`)
*   Test functions must begin with `test_` (e.g., `test_full_lds_reset()`)
*   Use descriptive names that explain what is being tested

**Sources:**[tt_tools_common/tests/test_reset.py 11](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L11-L11)[tt_tools_common/tests/test_detect_fallible.py 25](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L25-L25)[tt_tools_common/tests/test_reset_config.py 16](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L16-L16)

### Using Test Markers

Mark hardware-dependent tests with the `requires_hardware` decorator:

`@pytest.mark.requires_hardwaredef test_hardware_operation(devices):    # Test implementation    pass`
Software-only tests require no marker:

`def test_configuration_parsing():    # Test implementation    pass`
**Sources:**[tt_tools_common/tests/test_reset.py 10](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset.py#L10-L10)[tt_tools_common/tests/test_reset_config.py 16](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L16-L16)

### Using Fixtures

Request fixtures as function parameters:

`@pytest.mark.requires_hardwaredef test_with_devices(devices):    """Test using the devices fixture."""    assert len(devices) > 0`
Create custom fixtures in your test file or in `conftest.py`:

`@pytest.fixturedef custom_fixture():    """Custom fixture for test setup."""    return setup_value`
**Sources:**[tt_tools_common/tests/conftest.py 8-11](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/conftest.py#L8-L11)[tt_tools_common/tests/test_detect_fallible.py 13-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L13-L21)

### Test Structure Pattern

Follow this pattern for hardware tests that may not have available hardware:

`@pytest.mark.requires_hardwaredef test_hardware_feature(pci_indices):    """Test description."""    if not pci_indices:        pytest.skip("No chips detected")        # Test implementation    operation(pci_interfaces=pci_indices)    assert expected_result`
This pattern is demonstrated in [tt_tools_common/tests/test_detect_fallible.py 32-39](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L32-L39)

**Sources:**[tt_tools_common/tests/test_detect_fallible.py 32-39](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_detect_fallible.py#L32-L39)

### Assertion Best Practices

1.   **Use clear assertion messages**: Include context when assertions might fail
2.   **Test multiple aspects**: Verify both return values and side effects
3.   **Check preconditions**: Validate fixtures provide expected data before testing

Example from [tt_tools_common/tests/test_reset_config.py 39-49](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L39-L49):

`@pytest.mark.requires_hardwaredef test_generate_reset_logs(devices):    """Test generating default reset logs."""    file_path = generate_reset_logs(devices)    assert file_path is not None    assert os.path.exists(file_path)            # Verify file can be parsed    parsed = parse_reset_input([file_path])    assert parsed is not None    assert parsed.type == ResetType.CONFIG_JSON`
**Sources:**[tt_tools_common/tests/test_reset_config.py 39-49](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/tests/test_reset_config.py#L39-L49)
