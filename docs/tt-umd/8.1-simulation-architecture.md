## Purpose and Scope

This document describes the simulation architecture in `tt-umd`, which enables device operations without physical hardware. The simulation system provides multiple distinct implementations: **TTSimTTDevice** for dynamic library-based simulation (TTSim), **RtlSimulationTTDevice** for external RTL simulator processes (Verilator/VCS), and **MockChip** / **SWEmuleChip** for testing and software-only emulation.

The architecture focuses on maintaining interface parity with physical hardware while providing specialized managers for simulation-specific memory and TLB allocation.

## Core Simulation Components

### TTSimTTDevice (Dynamic Library Simulation)

`TTSimTTDevice` loads a simulator as a shared library (`.so`) and communicates through the `TTSimCommunicator`.

*   **TTSimCommunicator**: Handles `dlopen`/`dlsym` for simulator functions. It supports a multichip ABI, allowing distinct `ChipId` values to be passed to the simulator to isolate I/O.
*   **DRAM Teleportation**: An optimization controlled by the `TT_SIMULATOR_DRAM_TELEPORT` environment variable. When enabled, it allows the driver to bypass the simulated NoC for DRAM accesses to improve performance.
*   **PCI Configuration**: Simulates PCI config space reads to determine Vendor ID (expected `0x1E52`) and Device ID (e.g., Wormhole or Blackhole).

**Sources:**[device/tt_device/tt_sim_tt_device.cpp 38-107](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_sim_tt_device.cpp#L38-L107)[device/api/umd/device/tt_device/tt_sim_tt_device.hpp 31-118](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/tt_sim_tt_device.hpp#L31-L118)

### RtlSimulationTTDevice (RTL Process Simulation)

`RtlSimulationTTDevice` communicates with an external RTL simulator process using the `RtlSimCommunicator`.

*   **RtlSimCommunicator**: Uses a FlatBuffer-based protocol to send commands (READ, WRITE, RESET) to a `SimulationHost`.
*   **Sysmem Callbacks**: Registers callbacks so the simulator can perform DMA-like operations into the host's `SimulationSysmemManager`.

**Sources:**[device/tt_device/rtl_simulation_tt_device.cpp 65-103](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L65-L103)[device/api/umd/device/tt_device/rtl_simulation_tt_device.hpp 30-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/rtl_simulation_tt_device.hpp#L30-L87)

## Simulation Memory Management

Simulation backends use specialized managers instead of the standard Silicon managers to handle the absence of physical BARs and hugepages.

### SimulationSysmemManager

Manages host memory channels for simulation. It provides a simulated PCIe base address and handles `read_from_sysmem` / `write_to_sysmem` operations by indexing into allocated memory buffers.

### SimulationTlbAllocator

Since simulation does not have physical PCIe TLB hardware, the `SimulationTlbAllocator` manages virtual TLB indices.

*   For **Wormhole/Blackhole**, it mimics the hardware TLB region sizes (16MB and 2MB respectively).
*   For **Quasar (Grendel)**, it uses a dummy 4GB window as the communicator handles I/O directly without real TLBs.

**Sources:**[device/tt_device/tt_sim_tt_device.cpp 124-156](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_sim_tt_device.cpp#L124-L156)[device/tt_device/rtl_simulation_tt_device.cpp 107-132](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/rtl_simulation_tt_device.cpp#L107-L132)[device/api/umd/device/chip_helpers/simulation_tlb_allocator.hpp 15-40](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/chip_helpers/simulation_tlb_allocator.hpp#L15-L40)

## Multi-Chip Simulation Support

The architecture supports multi-chip simulation through the `create_for_chip` factory.

1.   **Unique Chip IDs**: Each `TTSimTTDevice` is initialized with a specific `ChipId`.
2.   **Shared-dlopen Mode**: The `TTSimCommunicator` detects if the simulator library supports multi-chip ABI. If so, it allows multiple device instances to coexist, ensuring that I/O on `Chip 0` does not interfere with `Chip 1`.

**Sources:**[device/tt_device/tt_sim_tt_device.cpp 58-84](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_sim_tt_device.cpp#L58-L84)[device/api/umd/device/tt_device/tt_sim_tt_device.hpp 49-51](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/tt_device/tt_sim_tt_device.hpp#L49-L51)
