This page documents the emulation of **Circular Buffers (CBs)** and **Dataflow Buffers (DFBs)**. These APIs provide the primary synchronization and data movement primitives for JIT-compiled kernels in `tt-emule`. While Wormhole and Blackhole architectures rely on the Single-Producer Single-Consumer (SPSC) CB model, Quasar introduces the Multi-Producer Multi-Consumer (MPMC) DFB model using hardware tile counters.

## Circular Buffer (CB) API

In `tt-emule`, a CB is emulated as a ring buffer within a core's L1 memory. Synchronization is handled via a host-side `CBSyncState` structure that manages occupancy and thread signaling [docs/cb-emulation.md 7-10](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/cb-emulation.md?plain=1#L7-L10)

### CBSyncState and Synchronization

The `CBSyncState` struct represents the shared state of a CB that is visible across different RISC processors (Reader, Writer, and Compute) [include/tt_emule/cb_sync_state.hpp 8-15](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/cb_sync_state.hpp#L8-L15)

| `page_size` | `uint32_t` | Size of a single tile/page in bytes [include/tt_emule/cb_sync_state.hpp 30](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/cb_sync_state.hpp#L30-L30) |
| `num_pages` | `uint32_t` | Total capacity of the CB in tiles [include/tt_emule/cb_sync_state.hpp 31](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/tt_emule/cb_sync_state.hpp#L31-L31) |

### Per-RISC Pointer Model (LocalCBInterface)

To faithfully model silicon behavior and avoid race conditions (e.g., issue #139), `tt-emule` maintains per-thread (per-RISC) read and write pointers [include/jit_hw/internal/emule_cb_ptr.h 10-14](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/internal/emule_cb_ptr.h#L10-L14) These live in a `thread_local` array of `LocalCBInterface` structures [include/jit_hw/internal/emule_cb_ptr.h 81](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/internal/emule_cb_ptr.h#L81-L81)

*   **Fast Path:**`cb_reserve_back` and `cb_wait_front` include a lock-free fast path that checks the atomic `occupied` counter before acquiring the mutex [include/jit_hw/api/cb_api.h 107-111](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L107-L111)
*   **Pointer Encoding:** Following silicon convention, internal pointers are often stored in 16-byte units and shifted (`<< 4`) to produce host addresses [include/jit_hw/internal/emule_cb_ptr.h 50-53](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/internal/emule_cb_ptr.h#L50-L53)

### Core CB Functions

The following functions are exposed to JIT kernels via `jit_hw/api/cb_api.h`:

*   `cb_reserve_back(cb_id, n)`: Blocks until at least `n` slots are free in the CB [include/jit_hw/api/cb_api.h 98](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L98-L98)
*   `cb_push_back(cb_id, n)`: Advances the local write pointer and increments the shared `occupied` semaphore [include/jit_hw/api/cb_api.h 128-132](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L128-L132)
*   `cb_wait_front(cb_id, n)`: Blocks until at least `n` tiles are available for reading [include/jit_hw/api/cb_api.h 157](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L157-L157)
*   `cb_pop_front(cb_id, n)`: Advances the local read pointer and decrements the shared `occupied` semaphore [include/jit_hw/api/cb_api.h 183-187](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L183-L187)

## Dataflow Buffer (DFB) API (Quasar)

On the Quasar architecture, DFBs replace CBs. DFBs utilize `TileCounter` objects to track data flow across 8 Data Movement (DM) processors and 4 Compute engines [docs/QUASAR_EMULATION.md 20-27](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L20-L27)

### TileCounter and DFBSyncState

A `TileCounter` manages the `posted` (pushed) and `acked` (popped) counts for a buffer [include/jit_hw/api/dfb_api.h 151-164](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L151-L164)

*   **MPMC Support:** Unlike CBs, multiple producers can push to the same DFB, and multiple consumers can pop from it.
*   **Broadcast vs. Round-Robin:** DFBs support broadcasting data to multiple tile counters or distributing it via round-robin (RR) [include/jit_hw/api/dfb_api.h 42-55](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L42-L55)

### DFB Operations

*   `dfb_reserve_back(dfb_id, n)`: Waits for space in the target Tile Counter(s) [include/jit_hw/api/dfb_api.h 36](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L36-L36)
*   `dfb_push_back(dfb_id, n)`: Increments the `posted` count. This also triggers the **CB↔DFB Bridge**[include/jit_hw/api/dfb_api.h 73-84](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L73-L84)
*   `dfb_wait_front(dfb_id, n)`: Waits until `occupancy` (posted - acked) is at least `n`[include/jit_hw/api/dfb_api.h 105](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L105-L105)
*   `dfb_pop_front(dfb_id, n)`: Increments the `acked` count and handles pointer wrapping [include/jit_hw/api/dfb_api.h 122-136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L122-L136)

Sources: [include/jit_hw/api/dfb_api.h 36-150](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L36-L150)[docs/QUASAR_EMULATION.md 20-30](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/QUASAR_EMULATION.md?plain=1#L20-L30)

## CB↔DFB Bridge

For Quasar kernels that mix CB-style compute access with DFB-style data movement, `tt-emule` implements a bridge to keep both views coherent [docs/cb-emulation.md 127-134](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/cb-emulation.md?plain=1#L127-L134)

When a DFB operation occurs:

1.   `dfb_push_back` calls `cb_sync_push` on the corresponding CB ID [include/jit_hw/api/dfb_api.h 81-84](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L81-L84)
2.   `dfb_pop_front` calls `cb_sync_pop` on the corresponding CB ID [include/jit_hw/api/dfb_api.h 131-134](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L131-L134)
3.   This allows a Compute engine using `cb_wait_front` to synchronize with a DM engine using `dfb_push_back`.

Sources: [include/jit_hw/api/dfb_api.h 81-84](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L81-L84)[include/jit_hw/api/dfb_api.h 131-134](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L131-L134)[docs/cb-emulation.md 127-134](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/cb-emulation.md?plain=1#L127-L134)

## Hang Detection

Both CB and DFB APIs implement hang detection. If a wait operation exceeds a timeout (default 120s), the emulator prints diagnostic information including the core's logical/physical coordinates and the state of the buffer, then aborts [include/jit_hw/api/cb_api.h 113-122](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L113-L122)

*   **Environment Variables:**
    *   `TT_EMULE_CB_TIMEOUT`: Adjusts the timeout for CB operations [include/jit_hw/api/cb_api.h 90-96](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L90-L96)
    *   `TT_EMULE_DFB_TIMEOUT`: Adjusts the timeout for DFB operations [include/jit_hw/api/dfb_api.h 20-26](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L20-L26)
