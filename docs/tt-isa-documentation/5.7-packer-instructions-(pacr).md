This page provides an instruction set reference for packer-related instructions in the Tensix coprocessor. For detailed information about the packer system architecture, format conversion pipeline, compression schemes, and addressing mechanisms, see [Packer System (PACR)](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.6-packer-system-(pacr)).

The packer instructions are responsible for reading data from the `Dst` register file (or from L1 in the case of packer 0) and writing it to L1, optionally performing format conversion and compression during the transfer.

## Overview

There are two primary packer-related instructions:

1.   **`PACR`** - The main packing instruction that transfers data from `Dst` (or L1) to L1 with optional format conversion and compression
2.   **`PACR_SETREG`** - A configuration instruction that writes to specific packer configuration registers

Four independent packers (packers 0-3) can execute in parallel, with each packer capable of processing one instruction per cycle. The `PACR` instruction uses a 4-bit packer mask to specify which packers should execute.

## PACR Instruction

**Summary:** Transfer data from `Dst` register file (or from L1 for packer 0) to L1, with optional format conversion and compression.

**Backend execution unit:** Packers 0-3 (independent parallel execution)

### Instruction Fields

The `PACR` instruction contains the following fields:

| Field | Bits | Description |
| --- | --- | --- |
| `PackerMask` | 4 bits | Bit mask specifying which packers (0-3) should execute this instruction. Each bit enables one packer. |
| `AddrMod` | 2 bits | Selects one of four `ADDR_MOD_PACK_SEC` configurations for ADC updates. Indexes into `ThreadConfig[CurrentThread].ADDR_MOD_PACK_SEC[AddrMod]`. |
| `OvrdThreadId` | 1 bit | When set, uses `PackerIConfig.Addr_cnt_context` to select ADC instead of `CurrentThread`. |
| `Last` | 1 bit | Indicates last tile in a sequence; flushes output buffers, sets `NeedsNewAddress` flags, and triggers metadata accumulation. |
| `Flush` | 1 bit | Flushes output buffers and sets `NeedsNewAddress` flags. Sets `InputNumDatums = 0` to skip input reading. |
| `ZeroWrite` | 1 bit | Sets `InputSource = INPUT_SOURCE_DEV_NULL` to write zeros without reading input data. |

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 7-60](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L7-L60)[WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 72-78](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L72-L78)

### ADC Updates

After computing addresses, the ADCs are updated according to the `AddrMod` configuration selected by the 2-bit `AddrMod` field. Each thread has four pre-configured address modification configurations stored in `ThreadConfig[CurrentThread].ADDR_MOD_PACK_SEC[0-3]`.

The ADC system has two independent channels per thread:

*   **Channel[0]**: Used for input address generation and datum counting (modified by input address generator)
*   **Channel[1]**: Used for output address generation (modified by output address generator)

Each channel maintains separate X, Y, Z, W counters plus Y_Cr and Z_Cr carry registers. Note that Channel[1].X is used in the input datum count computation `ADCs[WhichADC].Packers.Channel[1].X - ADC.X + 1`, as documented in [WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 101](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L101-L101)

### Metadata Accumulation

Each packer maintains per-thread tile size accumulators and all-zero flags. When a `PACR` instruction completes with the `Last` bit set, metadata is updated:

**Metadata Fields:**

Each packer tracks the following metadata:

*   **`LastTileSize`**: Number of 16-byte chunks written by the most recent completed tile from any thread
*   **`AccTileSize[0-2]`**: Cumulative tile size per thread (indexed by thread 0-2), readable/resettable via TDMA-RISC
*   **`LastThread`**: Which thread (0-2) produced the most recent tile, used to determine which `AccTileSize` to update
*   **`AllZeroFlags`**: 32-bit flag word where each bit indicates if the corresponding output row was all zeros
*   **`MetadataFIFO`**: Per-packer FIFO storing `{TileSize, AllZeroFlags}` pairs for software polling

The metadata can be accessed via three mechanisms:

1.   **TDMA-RISC registers** at memory-mapped addresses starting at `RISCV_TDMA_REG_PACKED_SIZE` (0xFFB1_0000)
2.   **`SETDMAREG` instruction** with various `InputSource` field values (0-9) selecting different metadata views
3.   **Metadata FIFO reads** via TDMA-RISC at `RISCV_TDMA_REG_FIFO_PACKED_TILE_*` addresses (0xFFB1_1000-0xFFB1_1010)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 5-79](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L5-L79)[WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md 24-111](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md?plain=1#L24-L111)

## Performance Characteristics

### Throughput

Each packer can theoretically achieve the following throughput to L1:

*   **Normal write**: 1×128-bit write per cycle = 128 bits/cycle
*   **Atomic accumulate**: 1×128-bit accumulate (4 lanes of FP32/INT32 or 8 lanes of FP16/BF16) every 5 cycles = 25.6 bits/cycle
*   **Non-atomic accumulate**: 1×128-bit accumulate every 2 cycles = 64 bits/cycle (but requires software to ensure no overlapping addresses)

With four packers operating in parallel: 4×128 = 512 bits/cycle peak aggregate bandwidth to L1.

The accumulate operations are detailed in [WormholeB0/TensixTile/L1.md 13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L13-L13) which notes they operate on 4×FP32 or 4×INT32 (sign-magnitude with saturation) or 8×FP16 or 8×BF16.

### L1 Access Contention

L1 memory is organized as 16 banks accessible through 16 access ports. Packers share these ports with other clients, creating potential contention:

| Packer | Write Port | Shared With |
| --- | --- | --- |
| Packer 0 write | Ports #2, #5 | Unpacker 0 reads, Unpacker 1 reads, Packer 0 reads (L1-to-L1 mode), Packer 2 writes |
| Packer 0 read | Ports #2-#5 | (L1-to-L1 mode only) |
| Packer 1 write | Port #5 | Unpacker 1 reads, Packer 0 writes |
| Packer 2 write | Port #2 or #5 | Unpacker 0 reads, Packer 0 writes/reads |
| Packer 3 write | Port #6 | (dedicated) |

The exact port assignments are shown in the L1 port allocation diagram in [WormholeB0/TensixTile/L1.md 19](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L19-L19) Note that packers can write to L1 at ports #2, #5, or #6, as detailed in [WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)

**Contention Scenarios:**

1.   **L1-to-L1 packing**: When packer 0 operates in L1-to-L1 mode (`Source_interface_selection == 1`), it reads and writes via shared ports #2-#5, creating potential contention with its own reads and writes, and with unpacker operations
2.   **Unpacker-packer overlap**: When unpackers and packers operate simultaneously, they compete for shared ports #2 and #5, with unpackers achieving 4 or 5×128-bit reads per cycle when both unpackers are active, as noted in [WormholeB0/TensixTile/L1.md 36-38](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L36-L38)
3.   **Bank conflicts**: Multiple clients accessing the same bank in the same cycle force serialization, with only one client able to proceed while others wait, as described in [WormholeB0/TensixTile/L1.md 5-6](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L5-L6)

**Measured Performance:**

*   Theoretical: 128 bits/cycle per packer (4 packers × 128 = 512 bits/cycle total)
*   Practical (with contention): Varies based on access patterns, bank distribution, and concurrent client activity

The L1 memory system is organized as 16 banks of 91.5 KiB each, with any port able to access any bank, but bank conflicts causing serialization, as described in [WormholeB0/TensixTile/L1.md 3-6](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L3-L6)

Sources: [WormholeB0/TensixTile/L1.md 15-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L15-L46)

### Latency and Buffering

Each packer maintains three independent 16-byte output buffers, one for each output stream: RSI metadata stream, exponent stream, and data stream. Data accumulates in these buffers and is written to L1 only when:

1.   **Buffer fills to 16 bytes**: Triggers automatic write to L1, incrementing the corresponding `ByteAddress` by 16
2.   **`Last` flag set**: Forces flush of all partial buffers (zero-padded to 16 bytes), and sets all `NeedsNewAddress` flags to true for the next instruction
3.   **`Flush` flag set**: Forces flush of all partial buffers (zero-padded to 16 bytes), and sets all `NeedsNewAddress` flags to true for the next instruction

Note that while `Last` and `Flush` have the same effect on the output address generator (flushing buffers and setting `NeedsNewAddress`), they differ in the input address generator: `Flush` sets `InputNumDatums = 0`, skipping the input read phase.

**Pipeline Stages:**

1.   Read input data from Dst/L1 (unless `Flush` or `ZeroWrite` flag is set)
2.   Format conversion (detailed in the Packer System documentation)
3.   Compression (if enabled via configuration)
4.   Buffer accumulation (16-byte buffers per stream)
5.   L1 write (when buffer full or flushed)
6.   Metadata update (if `Last` flag set)

The buffering means output addresses advance in 16-byte increments, even if individual datums are smaller. This simplifies L1 access patterns (all writes are aligned 128-bit/16-byte writes) at the cost of potential padding overhead for very small tiles, as documented in [WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 106-111](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L106-L111)

Sources: [WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 106-115](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L106-L115)

## Instruction Scheduling Considerations

### Synchronization with Other Units

`PACR` instructions reading from `Dst` must wait until the `Dst` data is valid from preceding Matrix Unit or Vector Unit operations. Use `STALLWAIT` with appropriate block bits and condition codes to enforce dependencies:

*   **Block bits B3-B6**: Correspond to packers 0-3 respectively
*   **Condition codes C3-C6**: Packer busy status per packer (0-3 respectively)

The `STALLWAIT` instruction blocks specified backend units until a specified condition is satisfied. For packers, condition codes C3-C6 indicate whether the corresponding packer is busy.

## Metadata Access

### Via SETDMAREG Instruction

The `SETDMAREG` instruction (executed by the Scalar Unit / ThCon) reads packer state into Tensix GPRs. This is a specialized scalar instruction with the following encoding fields:

*   `ResultSize` (2 bits): Output size selector (0=16-bit, 1=32-bit, 2=128-bit, 3=128-bit tile header)
*   `WhichPackers` (4 bits): Packer selector mask (used by InputSource 8) or index (used by InputSource 6-7)
*   `InputSource` (4 bits): Selects which packer metadata to read (0-9, different metadata per value)
*   `InputHalfReg` (3 bits): Selects 16-bit or 32-bit sub-field when reading partial data
*   `ResultHalfReg` (7 bits): Target GPR half-register location

The instruction syntax is documented in [WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md 9-16](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md?plain=1#L9-L16) and the encoding diagram is referenced in [WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md 20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md?plain=1#L20-L20)

**Tile Header Structure:**

The exponent histogram (InputSource 6-7) is a 32-byte array of saturating counters tracking observed exponent values during BFP encoding. Each byte is a saturating counter serving as a histogram bin. ### Via TDMA-RISC Registers

RISC-V cores can read packer metadata via memory-mapped TDMA-RISC registers. See the TDMA-RISC Configuration section above for detailed examples.

Key addresses:

*   **`RISCV_TDMA_REG_PACKED_SIZE`** (0xFFB1_0000 + offsets): Last tile size per packer/thread
*   **`RISCV_TDMA_REG_ACC_PACKED_SIZE`** (0xFFB1_0400 + offsets): Accumulated tile size per packer/thread
*   **`RISCV_TDMA_REG_FIFO_PACKED_TILE_STATUS`** (0xFFB1_1038): FIFO status (empty/full flags)
*   **`RISCV_TDMA_REG_FIFO_PACKED_TILE_SIZE`** (0xFFB1_1000 + offsets): Read tile size from FIFO
*   **`RISCV_TDMA_REG_FIFO_PACKED_TILE_ZEROMASK`** (0xFFB1_1010 + offsets): Read zero flags from FIFO
