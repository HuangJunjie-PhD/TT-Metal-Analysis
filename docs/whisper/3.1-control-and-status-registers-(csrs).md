# Control and Status Registers (CSRs)

Relevant source files
*   [CsRegs.cpp](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp)
*   [CsRegs.hpp](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp)

This document describes the CSR (Control and Status Register) management system in Whisper, implemented by the `CsRegs` class and related components. CSRs are special registers defined by the RISC-V specification that control processor behavior, manage interrupts, configure memory protection, and provide status information.

**Scope**: This page covers CSR architecture, privilege-based access control, read/write masks, delegation mechanisms, interrupt handling (including AIA), hypervisor virtualization, and performance counter integration. For trigger-based debug breakpoints, see [Debug Triggers and Breakpoints](https://deepwiki.com/tenstorrent/whisper/8.1-debug-triggers-and-breakpoints). For performance counter event configuration, see [Performance Counters](https://deepwiki.com/tenstorrent/whisper/3.3-performance-counters). For vector register CSRs (`VSTART`, `VTYPE`, `VL`), see [Vector Register File](https://deepwiki.com/tenstorrent/whisper/3.2-vector-register-file).

## Core Data Structures

### CsrNumber Enumeration

The `CsrNumber` enum defines all CSR numbers according to the RISC-V privileged specification. CSRs are organized by privilege level, with the privilege mode encoded in bits [9:8] of the CSR address.

| Privilege Mode | Bits [9:8] | Example CSRs |
| --- | --- | --- |
| User/VU | `00` | `CYCLE (0xC00)`, `TIME (0xC01)`, `FFLAGS (0x001)` |
| Supervisor/VS | `01` | `SSTATUS (0x100)`, `SIE (0x104)`, `SATP (0x180)` |
| Hypervisor | `10` | `HSTATUS (0x600)`, `HGATP (0x680)` |
| Machine | `11` | `MSTATUS (0x300)`, `MIE (0x304)`, `MTVEC (0x305)` |

Read-only CSRs have bits [11:10] set to `11`.

### Mask Types

CSRs have three types of masks that control bit-level access:

1.   **Write Mask (`writeMask_`)**: Bits that CSR instructions (e.g., `CSRRW`) can modify. Read-only bits have corresponding mask bits set to 0.

2.   **Poke Mask (`pokeMask_`)**: Bits that can be modified by hardware or non-CSR instructions. Typically a superset of the write mask. Allows modification of bits that are read-only to CSR instructions but can be updated by hardware (e.g., `MEIP` in `MIP`).

3.   **Read Mask (`readMask_`)**: Bits that are readable. Used to enforce read-only-zero behavior for bits that may become readable/writable later based on configuration or extension enable/disable.

### Peek and Poke Operations

For internal or privileged access without CSR instruction semantics:

*   **`peek(CsrNumber, value)`**: Direct read without privilege checking. Returns the CSR value with `readMask_` applied. Used internally by the simulator.

*   **`poke(CsrNumber, value)`**: Direct write honoring only the poke mask. Used to modify bits that are hardware-modifiable but not writable via CSR instructions. Examples include setting interrupt pending bits from external sources.

## Privilege and Virtual Mode

### Privilege Mode Checking

CSR accessibility is determined by comparing the current privilege mode against the CSR's required privilege mode (encoded in bits [9:8] of the CSR number):

| Current Mode | Can Access |
| --- | --- |
| Machine | All CSRs |
| Supervisor | Supervisor and User CSRs (if enabled) |
| User | User CSRs (if counters enabled via MCOUNTEREN) |

Debug mode CSRs (`DCSR`, `DPC`, `DSCRATCH0/1`) are only accessible when the hart is in debug mode.

## Interrupt CSRs and Delegation

Interrupt management is one of the most complex aspects of CSR handling due to delegation, aliasing, and virtual mode interactions.

### MIP/MIE Structure

The `MIP` (Machine Interrupt Pending) and `MIE` (Machine Interrupt Enable) CSRs track pending and enabled interrupts:

| Bit | Interrupt Type | Description |
| --- | --- | --- |
| 0 | Reserved | - |
| 1 | `SSIP` | Supervisor software interrupt pending |
| 3 | `MSIP` | Machine software interrupt pending |
| 5 | `STIP` | Supervisor timer interrupt pending |
| 7 | `MTIP` | Machine timer interrupt pending |
| 9 | `SEIP` | Supervisor external interrupt pending |
| 11 | `MEIP` | Machine external interrupt pending |
| 13+ | Custom/AIA | Local interrupts, guest interrupts |

Sources: [CsRegs.hpp 37-66](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L37-L66)

### Interrupt Delegation with MIDELEG

`MIDELEG` (Machine Interrupt Delegation) delegates specific interrupts to supervisor mode. When bit _i_ of `MIDELEG` is set:

*   Interrupt _i_ is handled by supervisor mode (if enabled)
*   Bit _i_ of `MIP`/`MIE` becomes readable/writable from supervisor mode via `SIP`/`SIE`

**Diagram: Interrupt Delegation Hierarchy**

Sources: [CsRegs.cpp 175-223](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L175-L223)[CsRegs.cpp 1507-1562](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1507-L1562)

### Advanced Interrupt Architecture (AIA)

The AIA extension adds additional CSRs for fine-grained interrupt control:

**MVIEN/MVIP**: Machine virtual interrupt enable/pending. When `MVIEN[i]` is set and `MIDELEG[i]` is clear, `SIP[i]` becomes an alias to `MVIP[i]` instead of `MIP[i]`. This allows injecting supervisor-level interrupts without modifying `MIP`.

**HIDELEG/HVIEN/HVIP**: Hypervisor interrupt delegation and injection. Similar to MVIEN/MVIP but for VS-mode interrupts.

**MIREG/SIREG/VSIREG**: Indirect access registers for IMSIC (Incoming Message-Signaled Interrupt Controller). Reads/writes are redirected to IMSIC internal registers based on `MISELECT`/`SISELECT`/`VSISELECT`.

**MTOPEI/STOPEI/VSTOPEI**: Top External Interrupt registers. Reading returns the highest-priority pending and enabled external interrupt. Writing clears the pending bit of that interrupt.

Sources: [CsRegs.cpp 316-342](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L316-L342)[CsRegs.cpp 528-583](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L528-L583)[CsRegs.cpp 1711-1786](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1711-L1786)[CsRegs.hpp 389-437](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L389-L437)

### SIP/SIE Special Read/Write Logic

Reading `SIP` or `SIE` involves complex aliasing logic:

1.   **Base value**: Read from internal `MIP`/`MIE` and mask with `MIDELEG`
2.   **MVIP aliasing**: Where `MIDELEG[i]` is 0 and `MVIEN[i]` is 1, use `MVIP[i]` instead of `MIP[i]`
3.   **Shadow SIE**: When writing `SIE` where `MVIEN[i]` is 1 and `MIDELEG[i]` is 0, write to `shadowSie_` instead of `MIE`
4.   **Mask out virtual interrupts**: Bits 2, 6, 10 (`VSSIP`, `VSTIP`, `VSEIP`) and bit 12 (`SGEIP`) are always read-only zero in `SIP`/`SIE`

Sources: [CsRegs.cpp 175-223](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L175-L223)[CsRegs.cpp 1507-1623](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1507-L1623)

## Hypervisor CSRs

### HIP/HVIP Computed Values

`HIP` (Hypervisor Interrupt Pending) has several bits computed on-the-fly rather than stored:

**Bit 12 (SGEIP)**: Set if any bit in both `HGEIP` (guest external interrupt pending) and `HGEIE` (guest external interrupt enable) is set.

**Bit 10 (VSEIP)**: OR of `HVIP[10]` and the bit of `HGEIP` selected by `HSTATUS.VGEIN`.

**Bit 6 (VSTIP)**: OR of `HVIP[6]` and the virtual timer comparison (`time + htimedelta >= vstimecmp`).

Sources: [CsRegs.cpp 268-311](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L268-L311)

### VSIP Aliasing

`VSIP` (Virtual Supervisor Interrupt Pending) is entirely aliased and has no storage of its own:

*   Bits 1, 5, 9 (VSSIP, VSTIP, VSEIP): Alias to `HIP` bits 2, 6, 10 if `HIDELEG` is set; otherwise read as zero.
*   Bits 13-63: Alias to `SIP` if `HIDELEG` is set, or to `HVIP` if `HVIEN` is set and `HIDELEG` is clear.

Sources: [CsRegs.cpp 226-265](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L226-L265)

### HSTATUS Fields

Key fields in `HSTATUS`:

*   **VGEIN** (bits 17-12): Virtual guest external interrupt number. Selects which guest interrupt file is active.
*   **SPV** (bit 7): Supervisor previous virtualization mode. Saved/restored on trap entry/exit.
*   **SPVP** (bit 8): Supervisor previous virtual privilege.
*   **GVA** (bit 6): Guest virtual address. Set when a trap is caused by a guest virtual address.

Sources: [CsrFields.hpp 199-265](https://github.com/tenstorrent/whisper/blob/733fd921/CsrFields.hpp#L199-L265)

## Performance Counter CSRs

### Machine Performance Counters

Performance counters consist of:

*   **MCYCLE/MCYCLEH**: Cycle count (CSR 0xB00/0xB80)
*   **MINSTRET/MINSTRETH**: Retired instruction count (CSR 0xB02/0xB82)
*   **MHPMCOUNTER3-31/H**: Hardware performance monitor counters (CSR 0xB03-0xB1F/0xB83-0xB9F)

Each counter has a corresponding event selector:

*   **MHPMEVENT3-31**: Event selector for performance counters (CSR 0x323-0x33F)
*   **MHPMEVENTH3-31**: High 32 bits in RV32 (CSR 0x723-0x73F)

Sources: [CsRegs.hpp 104-228](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L104-L228)

### Counter Enable Control

Counter access from lower privilege modes is controlled by:

*   **MCOUNTEREN**: Machine counter enable for supervisor mode
*   **SCOUNTEREN**: Supervisor counter enable for user mode
*   **HCOUNTEREN**: Hypervisor counter enable for VS/VU mode

Each register has one bit per counter. Setting bit _i_ allows the next lower privilege level to access counter _i_.

Sources: [CsRegs.cpp 757-777](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L757-L777)[CsRegs.cpp 3181-3206](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L3181-L3206)

### Overflow Extension (Sscofpmf)

The Supervisor-level Counter Overflow and Privilege Mode Filtering extension adds:

*   **OF bit** in `MHPMEVENT`: Overflow indicator (bit 63)
*   **SCOUNTOVF**: Supervisor counter overflow status (CSR 0xDA0). Bits 3-31 mirror the OF bits of MHPMEVENT3-31, filtered by counter enable.
*   **LCOF interrupt**: Local counter overflow interrupt in `MIP`/`SIP` (bit 13)

When a counter overflows, the corresponding OF bit is set and the LCOF interrupt is raised.

Sources: [CsRegs.cpp 1026-1129](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1026-L1129)[CsrFields.hpp 695-713](https://github.com/tenstorrent/whisper/blob/733fd921/CsrFields.hpp#L695-L713)

## Special CSR Handling

### MSTATUS Legalization

Writing `MSTATUS` involves legalization to ensure valid state:

1.   **MPP field**: Must be a valid privilege mode (User=0, Supervisor=1, Machine=3). Reserved value (2) is converted to User.
2.   **SPP field**: Must be 0 or 1. Cannot be Supervisor if supervisor mode is disabled.
3.   **FS/VS fields**: If any is Dirty (3), the SD bit (bit 63 in RV64, bit 31 in RV32) is set to 1; otherwise, SD is 0.
4.   **Endianness**: UBE, SBE, MBE bits control byte order for each privilege level.

Sources: [CsRegs.cpp 1443-1473](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1443-L1473)

### MISA Legalization

Writing `MISA` enforces extension dependencies:

*   If F extension is disabled, D extension must also be disabled.
*   If F or D extension is disabled, V extension must be disabled.
*   If U extension is disabled, S extension must be disabled.
*   I and E extensions are mutually exclusive.

Sources: [CsRegs.cpp 1477-1502](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1477-L1502)

### Time CSRs with HTIMEDELTA

In virtual mode, reading `TIME` or `TIMEH` adds `HTIMEDELTA`/`HTIMEDELTAH` to the value:

```
TIME (VS mode) = TIME (actual) + HTIMEDELTA
```

This allows the hypervisor to virtualize time for guest operating systems.

Sources: [CsRegs.cpp 423-448](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L423-L448)[CsRegs.hpp 271-272](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L271-L272)

### PMPADDR NAPOT Encoding

When a PMP entry uses NAPOT (Naturally Aligned Power-Of-Two) addressing mode, the `PMPADDR` register encodes both the base address and size. Reading `PMPADDR` adjusts the value to reflect the actual address range.

Sources: [CsRegs.cpp 3297-3330](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L3297-L3330)

## Trigger CSRs

Trigger CSRs provide hardware debug breakpoint capabilities. They are managed separately by the `Triggers` subsystem but accessed through standard CSR operations.

### Trigger-Related CSRs

*   **TSELECT**: Selects which trigger (0 to N-1) is accessed by TDATA1/2/3
*   **TDATA1**: Primary trigger data (type, action, matching mode)
*   **TDATA2**: Secondary trigger data (match value or address)
*   **TDATA3**: Tertiary trigger data (additional matching criteria)
*   **TINFO**: Trigger information (read-only, describes trigger capabilities)
*   **TCONTROL**: Trigger control (machine mode trigger enable)
*   **MCONTEXT/SCONTEXT/HCONTEXT**: Context matching for triggers

Reading or writing these CSRs is redirected to the `Triggers` subsystem via `readTrigger()` and `writeTrigger()`.

## State Enable (Smstateen) Extension

The Smstateen extension provides fine-grained control over which features are available at different privilege levels.

### STATEEN CSR Hierarchy

```
MSTATEEN0-3  (Machine)
    ↓ (bits control accessibility)
HSTATEEN0-3  (Hypervisor)
    ↓ (bits control accessibility)
SSTATEEN0-3  (Supervisor)
```

Each bit in a STATEEN register controls access to a specific feature or CSR. For example:

*   Bit 0 (C): Controls access to context CSRs and Zcmt extension
*   Bit 1 (FCSR): Controls access to floating-point CSRs
*   Bit 55 (SRMCFG): Controls access to QoS ID configuration
*   Bit 63 (SEO): Controls supervisor execution-only pages

When a bit is 0 in a higher-privilege STATEEN, it becomes read-only zero in lower-privilege STATEEN registers.

Sources: [CsRegs.cpp 1209-1238](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L1209-L1238)[CsrFields.hpp 879-913](https://github.com/tenstorrent/whisper/blob/733fd921/CsrFields.hpp#L879-L913)

## Implementation Notes

### RV32 vs RV64

Many 64-bit CSRs are split into low and high halves in RV32:

*   `MSTATUS` (64-bit) → `MSTATUS` (bits 31:0) + `MSTATUSH` (bits 63:32)
*   `MCYCLE` (64-bit) → `MCYCLE` (bits 31:0) + `MCYCLEH` (bits 63:32)
*   `MHPMEVENT3` (64-bit) → `MHPMEVENT3` (bits 31:0) + `MHPMEVENTH3` (bits 63:32)

The `Csr` class tracks high/low relationships with `isHigh_`, `isLow_`, and `peer_` members.

Sources: [CsRegs.hpp 554-590](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L554-L590)[CsRegs.cpp 3756-3816](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.cpp#L3756-L3816)

### CSR Callbacks

CSRs can have pre- and post-callbacks for read, write, and poke operations:

*   **preWrite/prePoke**: Called before modifying the CSR value. Can modify the value to be written.
*   **postWrite/postPoke**: Called after modifying the CSR value. Used for side effects.
*   **postReset**: Called after CSR reset to initial value.

These callbacks enable complex CSR behaviors without cluttering the main read/write logic.

Sources: [CsRegs.hpp 650-673](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L650-L673)[CsRegs.hpp 804-812](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L804-L812)

### Fast Peek Methods

For frequently accessed CSRs, specialized `peek*()` methods bypass general CSR lookup:

*   `peekMip()`, `peekMie()`, `peekMvip()`, `peekMvien()`
*   `peekMstatus()`, `peekHstatus()`, `peekSstatus()`
*   `peekMideleg()`, `peekHideleg()`
*   `peekVstart()`, `peekMnstatus()`

These methods directly index into the `regs_` vector for performance.

Sources: [CsRegs.hpp 1527-1694](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L1527-L1694)

### Record Write Tracking

The `recordWrite()` method tracks which CSRs were modified by the last instruction. This is used for:

*   Instruction tracing
*   Detecting CSR changes for verification
*   Updating trigger hit bits

## CSR Integration with Hart Execution

The `Hart` class owns a `CsRegs` instance and uses it throughout instruction execution:

1.   **Privilege checking**: Before CSR instructions execute
2.   **Interrupt handling**: Checking MIE/MIP to determine if interrupts are pending
3.   **Exception handling**: Writing to MEPC/MCAUSE/MTVAL on traps
4.   **Address translation**: Reading SATP/HGATP for page table base
5.   **Performance counting**: Incrementing MCYCLE, MINSTRET, MHPMCOUNTER
6.   **Debug mode**: Entering debug mode based on DCSR configuration

The close integration ensures that CSR state accurately reflects the hart's execution state.

Sources: [CsRegs.hpp 915-917](https://github.com/tenstorrent/whisper/blob/733fd921/CsRegs.hpp#L915-L917)

**Key Takeaways**:

*   CSRs are managed by the `CsRegs<URV>` template class, with each CSR represented by a `Csr<URV>` object
*   Three types of masks (write, poke, read) control bit-level access permissions
*   Privilege checking and virtual mode remapping enable secure multi-level execution
*   Interrupt CSRs (MIP/MIE/SIP/SIE/HIP/VSIP) have complex aliasing and delegation logic
*   Special handling is required for MSTATUS, MISA, time CSRs, and interrupt-related CSRs
*   Extension enable/disable methods activate or deactivate groups of related CSRs
*   Bitfield structures in `CsrFields.hpp` provide convenient access to complex CSR fields
