This page documents the GitHub Actions workflows that automate building, testing, and releasing tt-exalens. It covers Docker image management, the build workflow chain, hardware test execution, and the PyPI release pipeline. For information about the build system itself (CMake, Makefile targets, SFPI), see [Build System Architecture](https://deepwiki.com/tenstorrent/tt-exalens/8.1-build-system-architecture). For information about test structure and categories, see [Test Categories and Coverage](https://deepwiki.com/tenstorrent/tt-exalens/8.3-test-categories-and-coverage).

* * *

## Overview

The pipeline is composed of several interrelated GitHub Actions workflows located in `.github/workflows/`. There are two entry-point workflows (`on-pr.yml`, `on-push.yml`) that delegate to a shared build-and-test workflow, which in turn calls a reusable build workflow. A separate release pipeline publishes Python wheels to GitHub Releases and PyPI.

**Workflow Summary**

| File | Role | Trigger |
| --- | --- | --- |
| `on-pr.yml` | Entry point for pull requests | PR opened against `main` |
| `on-push.yml` | Entry point for pushes to `main` | Push to `main` |
| `build-and-test.yml` | Docker build, compile, hardware tests | `workflow_call`, `workflow_dispatch` |
| `build-ttexalens.yml` | Reusable compile + wheel build | `workflow_call` |
| `release-latest.yml` | Triggers "latest" release build | Push to `main`, manual |
| `release-version.yml` | Triggers versioned release + PyPI publish | `VERSION` file changed on `main` |
| `release.yml` | Reusable wheel build + GitHub Release creation | `workflow_call` |
| `community-issue-tagging.yml` | Labels community issues/PRs | Issue/PR opened |
| `produce_data.yml` | Collects CI/CD metrics | Completion of "On PR", "On push", "Run all tests" |

## Trigger Workflows

Both entry-point workflows follow the same pattern: a matrix over Ubuntu versions (`22.04`, `24.04`) delegating to `build-and-test.yml`.

**`on-pr.yml`** fires on any pull request targeting `main`[.github/workflows/on-pr.yml 3-7](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/on-pr.yml#L3-L7) and requires `packages: write` permission for pushing Docker images [.github/workflows/on-pr.yml 8-9](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/on-pr.yml#L8-L9)

**`on-push.yml`** fires on any push to `main`[.github/workflows/on-push.yml 3-7](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/on-push.yml#L3-L7) using the same matrix and permissions.

Both pass `secrets: inherit` to the called workflow so downstream jobs can access `GITHUB_TOKEN` and other repository secrets.

Sources: [.github/workflows/on-pr.yml 1-19](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/on-pr.yml#L1-L19)[.github/workflows/on-push.yml 1-19](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/on-push.yml#L1-L19)

## Build and Test Workflow (`build-and-test.yml`)

This workflow coordinates three sequential stages: Docker image construction, library compilation, and hardware-backed test execution.

**Stages Diagram**

Sources: [.github/workflows/build-and-test.yml 17-168](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L17-L168)

### Job: `build-image`

Runs on `ubuntu-latest`. Checks out the repo with submodules, logs in to `ghcr.io`, then calls `.github/build-docker-images.sh` with the Ubuntu version [.github/workflows/build-and-test.yml 37-53](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L37-L53)

The script [.github/build-docker-images.sh 1-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/build-docker-images.sh#L1-L54) generates a content-based Docker tag by calling `.github/get-docker-tag.sh`, then conditionally builds and pushes the CI image only if the tag does not already exist in the registry.

The Docker tag is a `sha256` hash of the concatenation of:

*   `.github/Dockerfile.ci`
*   `ttexalens/requirements.txt`
*   `ttexalens/dev-requirements.txt`
*   `test/test_requirements.txt`

[.github/get-docker-tag.sh 7](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/get-docker-tag.sh#L7-L7) — the tag is prefixed with `dt-`.

The resulting image is named:

```
ghcr.io/tenstorrent/tt-exalens/tt-exalens-ci-ubuntu-<VERSION>:<DOCKER_TAG>
```

Two outputs are produced: `docker-image` (using the `harbor.ci.tenstorrent.net` mirror for shared runners) and `public-docker-image` (the `ghcr.io` address) [.github/workflows/build-and-test.yml 41-53](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L41-L53)

Sources: [.github/workflows/build-and-test.yml 24-53](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L24-L53)[.github/build-docker-images.sh 1-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/build-docker-images.sh#L1-L54)[.github/get-docker-tag.sh 1-8](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/get-docker-tag.sh#L1-L8)

### Job: `set-latest`

Executes only on push to `main`[.github/workflows/build-and-test.yml 67](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L67-L67) and only after `build-image` succeeds. Uses `skopeo copy` to tag the newly built image as `:latest` on `ghcr.io`[.github/workflows/build-and-test.yml 75-81](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L75-L81)

Sources: [.github/workflows/build-and-test.yml 63-81](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L63-L81)

### Job: `build`

Calls the reusable `build-ttexalens.yml` workflow, passing `runs-on: ubuntu-latest`, the `public-docker-image` output from `build-image`, and the `os` string [.github/workflows/build-and-test.yml 55-62](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L55-L62)

### Job: `run-tests`

Runs after both `build` and `build-image`. Uses a strategy matrix over three hardware boards:

| `arch` | `board` | Runner label |
| --- | --- | --- |
| `blackhole` | `p150b` | `tt-ubuntu-2204-p150b-stable` |
| `wormhole_b0` | `n150` | `tt-ubuntu-2204-n150-stable` |
| `wormhole_b0` | `n300` | `tt-ubuntu-2204-n300-stable` |

Each job runs inside the Docker container with `--device /dev/tenstorrent/0` and mounts hugepage and kernel module paths [.github/workflows/build-and-test.yml 100-108](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L100-L108)

Steps performed per board:

1.   Check out repo with submodules.
2.   Download build artifacts from the `build` job (artifact name: `ttexalens-build-ubuntu-<VERSION>`).
3.   Run library tests via `python3 -m xmlrunner discover` against `test/ttexalens/` and upload the XML report.
4.   Run app tests via `python3 -m unittest discover` against `test/app/`.
5.   Repeat steps 3–4 with `TTEXALENS_TESTS_USE_NOC1=1` — **skipped on `n300`** due to a known host reboot issue [.github/workflows/build-and-test.yml 147-158](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L147-L158)
6.   Install the built wheel from `build/ttexalens_wheel/*.whl`, then run `test/wheel/run-wheel.sh`.

Sources: [.github/workflows/build-and-test.yml 83-168](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L83-L168)

## Reusable Build Workflow (`build-ttexalens.yml`)

This workflow is the canonical way to compile tt-exalens inside CI. It is called by `build-and-test.yml` and by the release workflows.

Key inputs accepted by this workflow:

| Input | Default | Description |
| --- | --- | --- |
| `runs-on` | `tt-ubuntu-2204-large-stable` | Runner label |
| `os` | `ubuntu-22.04` | OS string used in artifact name |
| `config` | `Release` | Build configuration |
| `docker-image` | `ghcr.io/.../tt-exalens-ubuntu-22-04:latest` | Container image |

The SFPI toolchain is downloaded during `make` but removed from the uploaded artifact to save space — only the `riscv-tt-elf-gdb` binary is retained [.github/workflows/build-ttexalens.yml 52-59](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-ttexalens.yml#L52-L59)

The artifact `ttexalens-build-<os>` contains the entire `build/` directory (minus the large SFPI compiler binaries).

Sources: [.github/workflows/build-ttexalens.yml 1-65](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-ttexalens.yml#L1-L65)

## Release Pipeline

Three workflows together form the release pipeline:

**Release Workflow Diagram**

Sources: [.github/workflows/release-latest.yml 1-21](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-latest.yml#L1-L21)[.github/workflows/release-version.yml 1-59](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-version.yml#L1-L59)[.github/workflows/release.yml 1-96](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release.yml#L1-L96)

### `release-latest.yml`

Triggered on push to `main` or `workflow_dispatch`. Calls `release.yml` with `build-version: "latest"`[.github/workflows/release-latest.yml 15-21](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-latest.yml#L15-L21) This produces a GitHub Release tagged `latest` with prerelease status and no auto-generated release notes [.github/workflows/release.yml 76-85](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release.yml#L76-L85)

### `release-version.yml`

Triggered when the `VERSION` file is modified on `main`[.github/workflows/release-version.yml 7-9](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-version.yml#L7-L9) After calling `release.yml` (which reads the version from the `VERSION` file), it runs two additional publish jobs:

*   **`test-pypi-publish`**: uses the `testpypi-release` GitHub environment and `pypa/gh-action-pypi-publish` with Trusted Publishing (OIDC) to publish to TestPyPI [.github/workflows/release-version.yml 20-38](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-version.yml#L20-L38)
*   **`pypi-publish`**: uses the `Production PyPi release` GitHub environment to publish to `pypi.org/p/tt-exalens`[.github/workflows/release-version.yml 39-58](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-version.yml#L39-L58)

Both publish jobs use `id-token: write` permission for OIDC Trusted Publishing — no stored API tokens are required.

### `release.yml` (Reusable)

Accepts an optional `build-version` input. The `prepare-release` job:

1.   Runs inside `ghcr.io/tenstorrent/tt-exalens/tt-exalens-ci-ubuntu-22.04:latest`.
2.   Determines the version: uses the `build-version` input if set, otherwise reads from the `VERSION` file [.github/workflows/release.yml 43-49](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release.yml#L43-L49)
3.   Runs `make wheel` to produce `build/ttexalens_wheel/*.whl`.
4.   Uploads the wheel as the `ttexalens_wheel` artifact.

The `release` job then creates a GitHub Release using `softprops/action-gh-release@v2`[.github/workflows/release.yml 87-95](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release.yml#L87-L95) When `build-version` is `"latest"`, the release is tagged `latest` with `prerelease: true` and no generated notes; otherwise the tag is `v<VERSION>` with auto-generated release notes.

Sources: [.github/workflows/release.yml 1-96](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release.yml#L1-L96)[.github/workflows/release-version.yml 1-59](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-version.yml#L1-L59)[.github/workflows/release-latest.yml 1-21](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/release-latest.yml#L1-L21)

## Docker Image Management

The CI container image is content-addressed: its tag is derived from a SHA-256 hash of the files that determine its content.

**Tag Computation**

[.github/get-docker-tag.sh 7](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/get-docker-tag.sh#L7-L7) — The tag is generated by concatenating those four files and piping to `sha256sum`.

**Build and Push Logic** (`.github/build-docker-images.sh`)

The `build_and_push` function checks whether `<image>:<tag>` already exists via `docker manifest inspect`. If it does, the build is skipped entirely; otherwise the image is built and pushed to `ghcr.io`[.github/build-docker-images.sh 26-46](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/build-docker-images.sh#L26-L46) This means rebuilds only happen when one of the four source files changes.

The CI image is named:

```
ghcr.io/tenstorrent/tt-exalens/tt-exalens-ci-ubuntu-<VERSION>:<DOCKER_TAG>
```

On `main` branch, the `set-latest` job additionally tags this image as `:latest` using `skopeo copy`.

Sources: [.github/build-docker-images.sh 1-54](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/build-docker-images.sh#L1-L54)[.github/get-docker-tag.sh 1-8](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/get-docker-tag.sh#L1-L8)[.github/workflows/build-and-test.yml 63-81](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/build-and-test.yml#L63-L81)

## Supporting Workflows

### Community Issue Tagging (`community-issue-tagging.yml`)

Fires when an issue or PR is opened. Delegates to the central `tenstorrent/tt-github-actions` reusable workflow `on-community-issue.yml` to apply labels [.github/workflows/community-issue-tagging.yml 1-16](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/community-issue-tagging.yml#L1-L16) Requires `issues: write` and `pull-requests: write` permissions.

### CI/CD Data Collection (`produce_data.yml`)

Triggered on completion of "On PR", "On push", or "Run all tests" workflow runs. Uses the `tenstorrent/tt-github-actions``collect_data` action to ship run data to an SFTP host [.github/workflows/produce_data.yml 1-28](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/.github/workflows/produce_data.yml#L1-L28) Credentials are sourced from repository secrets (`SFTP_CICD_WRITER_HOSTNAME`, `SFTP_CICD_WRITER_USERNAME`, `SFTP_CICD_WRITER_KEY`).
