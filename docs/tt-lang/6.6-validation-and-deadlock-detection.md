**Purpose**: This page describes the validation mechanisms and deadlock detection capabilities of the tt-lang functional simulator. It covers how the simulator enforces correct `DataflowBuffer` (DFB) lifecycle usage, validates copy operations, and detects deadlock situations during multi-threaded kernel execution.

For information about the simulator's execution model, see [6.2 Program Execution Model](https://github.com/tenstorrent/tt-lang/blob/d76e6233/6.2%20Program%20Execution%20Model) For copy operation mechanics, see [6.5 Copy Handlers and Data Transfer](https://github.com/tenstorrent/tt-lang/blob/d76e6233/6.5%20Copy%20Handlers%20and%20Data%20Transfer)

* * *

## Overview

The simulator implements comprehensive validation to catch common programming errors early and provide actionable diagnostics. Validation occurs at multiple points in the simulation lifecycle:

1.   **DataflowBuffer state machine enforcement**: Ensures `reserve`, `wait`, `push`, and `pop` are called in the correct order for the specific thread type (Compute vs. DM) [python/sim/dfb.py 74-106](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L74-L106)
2.   **Copy operation validation**: Verifies shape compatibility between tensors, blocks, and pipes via specialized handlers in the `HANDLER_REGISTRY`[python/sim/copyhandlers.py 29-34](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L29-L34)
3.   **Resource limit validation**: Enforces hardware-specific constraints like L1 memory capacity and maximum DFB counts per node [python/sim/program.py 40-94](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L40-L94)
4.   **Deadlock detection**: Identifies when all greenlets are blocked waiting on resources (e.g., a full DFB or an empty Pipe) and none can make progress [python/sim/greenlet_scheduler.py 148-185](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L148-L185)

These validation mechanisms transform runtime hangs or silent data corruption that would occur on hardware into clear Python exceptions with source locations [python/sim/diagnostics.py 201-226](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/diagnostics.py#L201-L226)

## DataflowBuffer Lifecycle Validation

### State Machine Enforcement

Each `Block` returned by `DataflowBuffer.reserve()` or `.wait()` follows a strict state machine managed by `AccessState`. This enforces the correct sequence of operations based on the `KernelType` (DM or COMPUTE) and `BlockAcquisition` method [python/sim/blockstate.py 16-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L16-L48)

The `Block` class delegates all access-state logic to this state machine to verify that operations like `store()` or `copy()` are valid for the current state [test/sim/test_blockstate.py 51-84](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_blockstate.py#L51-L84)

#### Block State Transitions

The following diagram maps the logical DFB lifecycle to the internal state transitions defined in `AccessState`[python/sim/blockstate.py 16-33](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L16-L33)

**Key Transitions**:

*   **Must Write (MW)**: A newly reserved block that must be written to via `store` or `copy` before it can be pushed [python/sim/blockstate.py 19-21](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L19-L21)
*   **Must Read (MR)**: A block that has data but hasn't been consumed yet. It must be read or used as a source before `pop()` is allowed [python/sim/blockstate.py 22-24](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L22-L24)
*   **No Access While Writing (NAW)**: A block currently being used as a destination for an asynchronous copy; it is locked until `tx.wait()`[python/sim/blockstate.py 31](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L31-L31)

**Sources**: [python/sim/blockstate.py 16-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L16-L48)[test/sim/test_blockstate.py 51-84](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_blockstate.py#L51-L84)[python/sim/dfb.py 74-106](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L74-L106)

### Automatic Cleanup via Context Managers

The `Block` class implements `__enter__` and `__exit__` to support the `with` statement pattern [python/sim/dfb.py 138-163](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L138-L163)

If an exception occurs within the `with` block, the simulator skips the automatic `push()`/`pop()` to preserve the original error and avoid secondary state machine violations [python/sim/dfb.py 156-163](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L156-L163)

**Sources**: [python/sim/dfb.py 138-163](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/dfb.py#L138-L163)

## Copy Operation Validation

### Shape and Tile Count Validation

Copy operations are validated by specialized handlers in `python/sim/copyhandlers.py`. The simulator ensures that the source data fits exactly into the destination.

| Transfer | Validation Rule | File Reference |
| --- | --- | --- |
| **Tensor to Block** | `tensor.shape` must match `block.shape` (tile count) | [test/sim/test_copyhandlers.py 74-90](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_copyhandlers.py#L74-L90) |
| **Pipe to Block** | Pipe data shape must match destination Block shape | [test/sim/test_copyhandlers.py 149-184](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_copyhandlers.py#L149-L184) |
| **Block to Pipe** | Source block must contain valid data for pipe | [python/sim/copyhandlers.py 29-34](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/copyhandlers.py#L29-L34) |

If a mismatch is detected, a `ValueError` is raised detailing the expected vs. actual shapes [test/sim/test_copyhandlers.py 176-184](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_copyhandlers.py#L176-L184)

### Access Locking (NAW/ROR)

During an asynchronous copy, the involved blocks are locked to prevent race conditions. This is enforced by the `BlockStateMachine`:

*   **NAW (No Access while Writing)**: When a block is a `copy` destination, any `store()` or indexing attempt raises a `RuntimeError` until the transaction is waited [test/sim/test_blockstate.py 86-116](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_blockstate.py#L86-L116)
*   **ROR (Read-Only while Reading)**: When a block is a `copy` source, it can support multiple concurrent outgoing copies but cannot be overwritten [python/sim/blockstate.py 28-30](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/blockstate.py#L28-L30)

## Deadlock Detection

### Detection Logic in GreenletScheduler

The `GreenletScheduler` manages all simulated threads (COMPUTE, DM) as greenlets. A deadlock is defined as a state where no greenlets are "ready" to run, but some are still alive and blocked [python/sim/greenlet_scheduler.py 148-185](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L148-L185)

**Sources**: [python/sim/greenlet_scheduler.py 148-185](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L148-L185)[python/sim/diagnostics.py 61-90](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/diagnostics.py#L61-L90)[test/sim/test_greenlet_scheduler.py 177-215](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_greenlet_scheduler.py#L177-L215)

### Common Deadlock Scenarios

The simulator identifies specific blocking conditions to help debug deadlocks:

*   **Pipe Starvation**: A kernel calling `copy(pipe, block).wait()` blocks indefinitely because no sender has pushed data to that pipe address [test/sim/test_copyhandlers.py 114-141](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_copyhandlers.py#L114-L141)
*   **Buffer Exhaustion**: A producer calling `reserve()` on a `DataflowBuffer` where all blocks are full and the consumer is also blocked or terminated [test/sim/test_greenlet_scheduler.py 195-215](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/sim/test_greenlet_scheduler.py#L195-L215)

## Hardware Constraint Validation

The `Program` implementation validates that the kernel configuration respects physical hardware limits.

### L1 Memory and DFB Limits

| Constraint | Default Limit | logic |
| --- | --- | --- |
| **Max DFBs** | 32 | Checked via `get_max_dfbs()` during DFB creation [python/sim/program.py 40-63](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L40-L63) |
| **Max L1 Bytes** | 1,432 KiB | Sum of `capacity_bytes` across all DFBs on a node [python/sim/program.py 66-94](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L66-L94) |

The `SimulatorContext` tracks `kernel_l1_bytes` and `kernel_dfb_count` per kernel body [python/sim/context_types.py 82-101](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/context_types.py#L82-L101) If these are exceeded, the simulator issues diagnostic warnings using `print_diagnostic_warning`[python/sim/diagnostics.py 165-199](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/diagnostics.py#L165-L199)
