## Step-by-Step Guide: Creating a New Test

### Step 1: Create Test File Structure

Create a new file in `tests/python_tests/` named `test_<operation>.py`:

Sources: [tests/python_tests/test_reduce.py 1-30](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/test_reduce.py#L1-L30)

### Step 2: Define Test Parameters with @parametrize

The `@parametrize` decorator defines all test variants. It automatically resolves parameter dependencies:

The `parametrize` decorator supports:

*   **Simple lists**: `[value1, value2, value3]`
*   **Dependency functions**: `lambda param1: compute_valid_values(param1)`
*   **Format combinations**: `input_output_formats([DataFormat.Float16, ...])`

Sources: [tests/python_tests/helpers/param_config.py 292-319](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L292-L319)[tests/python_tests/test_reduce.py 40-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/test_reduce.py#L40-L68)

Use `generate_stimuli()` to create test input data:

The `generate_stimuli()` function returns:

*   `src_A`, `src_B`: PyTorch tensors with random data in the specified format
*   `tile_cnt_A`, `tile_cnt_B`: Number of 32×32 tiles in each tensor

## Template Parameters vs Runtime Parameters

Understanding the distinction between template and runtime parameters is crucial:

### Template Parameters (Compile-Time)

Template parameters are embedded into the compiled C++ code via the `build.h` header file. They define operation types, dimensions, and configuration that cannot change at runtime.

## Fused Operation Testing (L1-to-L1 Iterations)

Fused operations perform multiple pipeline passes, where the output of one iteration becomes the input of the next. Enable this with the `L1_to_L1_iterations` parameter:

The `L1_to_L1_iterations` parameter:

*   Generates multiple `FormatConfig` entries for format inference
*   Configures the C++ kernel to iterate multiple times
*   Adjusts validation tolerances for accumulated precision loss

For more details on fused operation testing, see [Fused Operation Testing](https://deepwiki.com/tenstorrent/tt-llk/8.9-fused-operation-testing).

## Validation and Tolerance Settings

The `passed_test()` function compares hardware output against golden references with format-appropriate tolerances:

Default tolerances vary by format:

*   **Float32**: Tighter tolerances (`rtol=1e-5`)
*   **Float16/Float16_b**: Moderate tolerances (`rtol=1e-3`)
*   **Bfp8_b**: Relaxed tolerances due to reduced precision

## Best Practices

### Parameter Organization

1.   **Order parameters by dependency**: Independent parameters first, dependent parameters later
2.   **Use meaningful constraint functions**: Make parameter relationships explicit
3.   **Document format limitations**: Add comments explaining format restrictions

### Test Isolation

1.   **Each test should be independent**: Tests should not depend on execution order
2.   **Clean L1 memory between tests**: The framework handles this automatically
3.   **Use fresh stimuli**: Generate new random data for each test variant

### Golden Model Accuracy

1.   **Match hardware precision**: Use PyTorch operations that mirror hardware behavior
2.   **Handle edge cases**: NaN, infinity, denormals should be handled consistently
3.   **Validate intermediate results**: For fused tests, validate each stage if possible

### Performance Considerations

1.   **Limit format sweeps for expensive tests**: Use focused format combinations
2.   **Skip large dimensions in coverage mode**: Large tests are slow with instrumentation
3.   **Use appropriate test splits in CI**: Balance compilation time vs execution time
