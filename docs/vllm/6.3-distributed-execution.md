This page documents vLLM's distributed execution system for running large language models across multiple GPUs, nodes, and heterogeneous hardware. It covers tensor parallelism (TP), pipeline parallelism (PP), data parallelism (DP), distributed KV cache transfer, and the multi-process worker architecture.

For information about the basic worker and executor architecture, see [Worker Architecture](/tenstorrent/vllm/2.5-worker-architecture). For KV cache management internals, see [Memory Management](/tenstorrent/vllm/2.4-memory-management). For platform-specific details, see [Platform Abstraction Layer](/tenstorrent/vllm/5.1-platform-abstraction-layer).

## Parallelism Strategies

vLLM v1 supports three primary parallelism strategies that can be combined to distribute model execution:

| Strategy | Purpose | Configuration | Use Case |
| --- | --- | --- | --- |
| **Tensor Parallelism (TP)** | Split model weights across GPUs within a node | `tensor_parallel_size` | Models too large for single GPU memory |
| **Pipeline Parallelism (PP)** | Split model layers across GPUs/nodes | `pipeline_parallel_size` | Very deep models, cross-node inference |
| **Data Parallelism (DP)** | Replicate model across independent instances | `data_parallel_size` | High throughput, load balancing |

The total world size is computed as: `world_size = TP × PP × DP`

**Sources:** [vllm/v1/executor/multiproc\_executor.py63-68](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/executor/multiproc_executor.py#L63-L68) [vllm/v1/worker/gpu\_worker.py78-83](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L78-L83) [vllm/config.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/config.py)

## Multi-Process Architecture

### Process Hierarchy

**Diagram: Multi-Process Communication Architecture**

The system uses three process types when `VLLM_ENABLE_V1_MULTIPROCESSING=1`:

1. **Client Process**: Contains user-facing API (`AsyncLLM` or `LLMEngine`)
2. **Engine Process**: Runs scheduler and KV cache management in busy loop (`EngineCoreProc`)
3. **Worker Processes**: Execute model forward passes (one per GPU for TP)

**Sources:** [vllm/v1/executor/multiproc\_executor.py49-134](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/executor/multiproc_executor.py#L49-L134) [vllm/v1/engine/core.py454-538](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py#L454-L538) [vllm/v1/engine/core\_client.py49-232](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core_client.py#L49-L232)

## Data Parallelism (DP)

Data parallelism replicates the entire model across multiple independent instances (engines), each serving different requests. DP enables horizontal scaling and load balancing.

### DP Architecture

**Diagram: Data Parallel Engine Topology**

### DP Client Types

vLLM provides different client implementations based on the DP configuration:

| Client Class | DP Size | Load Balancing | Use Case |
| --- | --- | --- | --- |
| `AsyncMPClient` | 1 | N/A | Single engine instance |
| `DPLBAsyncMPClient` | >1 | Internal (coordinator) | Shared frontend, round-robin |
| `DPAsyncMPClient` | >1 | External | Multiple API servers, client-side routing |

**Internal Load Balancing:** A `DPCoordinator` process runs alongside rank 0 engine, collecting queue statistics from all engines and routing requests via round-robin or least-loaded strategies.

**External Load Balancing:** Each engine has a colocated API server. External load balancer (e.g., nginx) routes requests to API servers.

**Sources:** [vllm/v1/engine/core\_client.py84-102](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core_client.py#L84-L102) [vllm/v1/engine/coordinator.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/coordinator.py) [vllm/v1/engine/core.py476-495](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/engine/core.py#L476-L495)

## Communication Primitives

### Message Queue Architecture

vLLM uses custom message queues built on shared memory for high-throughput, low-latency IPC:

**Diagram: Shared Memory Message Queue Topology**

### MessageQueue Implementation

The `MessageQueue` class provides:

* **Shared memory buffers**: Allocated via `/dev/shm` or POSIX shm
* **Chunking**: Large messages split into chunks (controlled by `VLLM_MQ_MAX_CHUNK_BYTES_MB`)
* **Non-blocking reads**: Workers poll without blocking
* **Handle export/import**: Processes share queue handles via pickled metadata

**Key Methods:**

* `enqueue(obj)`: Serialize and write object [vllm/distributed/device\_communicators/shm\_broadcast.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/distributed/device_communicators/shm_broadcast.py)

## Worker Coordination

### Execute Model Flow

**Diagram: Execute Model Sequence Across PP Stages**

### Synchronization Points

1. **Broadcast Barrier**: All workers must dequeue `SchedulerOutput` before execution begins
2. **PP Send/Recv**: Sequential stages synchronize via `send_tensor_dict()`/`recv_tensor_dict()`
3. **TP All-Reduce**: Within each stage, TP group synchronizes via NCCL collectives
4. **Response Collection**: Executor waits for response from output rank (last PP stage)

**Sources:** [vllm/v1/executor/multiproc\_executor.py173-200](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/executor/multiproc_executor.py#L173-L200) [vllm/v1/worker/gpu\_worker.py426-472](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/worker/gpu_worker.py#L426-L472)

## Configuration Examples

### Tensor Parallel Only (TP=4)

Splits model across 4 GPUs on single node. Each GPU has 1/4 of weights, synchronized via NCCL all-reduce.

### Pipeline Parallel Only (PP=2)

Splits layers across 2 GPUs/nodes. First GPU handles layers 0-N, second handles N+1-End. Requires sequential execution.

### TP + PP (TP=2, PP=2, Total=4 GPUs)

Creates 2 pipeline stages, each with 2-way tensor parallelism. Total world size = 4.

### Data Parallel (DP=2, TP=2, Total=4 GPUs)

Creates 2 independent replicas, each with 2-way tensor parallelism. Enables load balancing across replicas.
