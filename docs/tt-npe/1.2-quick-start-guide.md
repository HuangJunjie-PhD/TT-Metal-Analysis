This guide provides a minimal working example to get you running your first NoC performance simulation in under 5 minutes. It demonstrates the most common workflow: simulating a predefined workload and analyzing the results.

For complete installation instructions, see [Installation and Setup](https://deepwiki.com/tenstorrent/tt-npe/1.1-installation-and-setup). For detailed CLI options, see [Command Line Interface](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For programmatic usage, see [Python API](https://deepwiki.com/tenstorrent/tt-npe/3.2-creating-workloads-from-json).

## Prerequisites

*   Git and standard build tools (CMake, C++ compiler)
*   Python 3.8 or later
*   Basic command line familiarity

## Your First Simulation

### Step 1: Clone and Build

`git clone git@github.com:tenstorrent/tt-npe.gitcd tt-npe/./build-npe.sh`
The build process compiles the C++ core library ([tt_npe/cpp/src/](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/)) and Python bindings ([tt_npe/cpp/pybind/bindings.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp)). Build outputs are installed to `install/bin/` and `install/lib/`.

### Step 2: Environment Setup

`source ENV_SETUP`
> **Note**: If using with tt-metal, source `ENV_SETUP`**after** sourcing the tt-metal Python virtualenv setup.

This script ([ENV_SETUP](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/ENV_SETUP)) adds `install/bin/` to your `PATH`, making `tt_npe.py` and `npe_analyze_noc_trace_dir.py` available from anywhere.

### Step 3: Run Example Workload

`tt_npe.py -w tt_npe/workload/example_wl.json`
**Expected Output**:

## Understanding the Results

The `npe.Stats` object ([tt_npe/cpp/include/npeStats.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeStats.hpp)) returned by simulation contains:

| Field | Description | Typical Range |
| --- | --- | --- |
| `estimated_cycles` | Total simulated cycles | Workload dependent |
| `num_timesteps` | Simulation timesteps executed | `estimated_cycles / cycles_per_timestep` |
| `overall_avg_link_util` | Average NoC link utilization | 0-100% |
| `overall_max_link_util` | Peak NoC link utilization | 0-100% |
| `dram_bw_util` | DRAM bandwidth utilization | 0-100% |
| `getCongestionImpact()` | Congestion overhead percentage | 0-50% typical |

**Interpreting Results**:

*   **High link utilization (>70%)**: NoC may be bottleneck
*   **High DRAM utilization (>80%)**: Memory bandwidth limited
*   **High congestion impact (>20%)**: Significant contention between transfers

**Sources**: [tt_npe/cpp/pybind/bindings.cpp 39-66](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/bindings.cpp#L39-L66)[tt_npe/doc/tt_npe_pybind.html 288-365](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/doc/tt_npe_pybind.html#L288-L365)

## Common Usage Patterns

### Pattern 1: Profiler Integration (Recommended for Real Workloads)

For analyzing actual tt-metal programs:

`# Step 1: Capture NoC traces with tt-metal profilertt_metal/tools/profiler/profile_this.py --collect-noc-traces \  -c 'pytest tests/ttnn/integration_tests/resnet/test_resnet50.py' \  -o results # Step 2: Analyze all traces and generate timelinesnpe_analyze_noc_trace_dir.py results/.logs/ -e`
**Outputs**:

*   `results/reports/<timestamp>/ops_perf_results_<device>.csv` - Performance report with NPE columns (`DRAM BW UTIL`, `NOC UTIL`)
*   `results/npe_viz/*.npeviz` - Timeline files for ttnn-visualizer
*   `results/npe_viz/manifest.json` - Timeline file index

The profiler integration workflow is handled by [tt_npe/py/util/npe_analyze_noc_trace_dir.py 316-394](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L316-L394) which:

1.   Groups traces by operation ID ([npe_analyze_noc_trace_dir.py 286-314](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npe_analyze_noc_trace_dir.py#L286-L314))
2.   Merges multi-device traces using `fabric_post_process.py` ([npe_analyze_noc_trace_dir.py 354-366](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npe_analyze_noc_trace_dir.py#L354-L366))
3.   Runs NPE simulation in parallel ([npe_analyze_noc_trace_dir.py 372-382](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npe_analyze_noc_trace_dir.py#L372-L382))
4.   Generates summary statistics table ([npe_analyze_noc_trace_dir.py 198-226](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npe_analyze_noc_trace_dir.py#L198-L226))

### Pattern 3: Programmatic Workload Construction

For algorithmic workload generation, use the Python API:

`import tt_npe_pybind as npe # Create configurationcfg = npe.Config()cfg.device_name = "wormhole_b0"cfg.cycles_per_timestep = 32 # Build workloadwl = npe.Workload()phase = npe.Phase() # Add transfer: 8KB packet from (0,1,1) to (0,5,5)transfer = npe.Transfer(    packet_size=8192,      # bytes    num_packets=1,    src=npe.Coord(0, 1, 1),  # device_id, row, col    dst=npe.Coord(0, 5, 5),    injection_rate=0,      # auto-infer    phase_cycle_offset=0,    noc_type=npe.NocType.NOC_0)phase.addTransfer(transfer)wl.addPhase(phase) # Run simulationapi = npe.InitAPI(cfg)result = api.runNPE(wl) if isinstance(result, npe.Stats):    print(f"NoC Utilization: {result.overall_avg_link_util:.1f}%")`
See [tt_npe/py/examples/programmatic_workload_generation.py 21-75](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/examples/programmatic_workload_generation.py#L21-L75) for a complete example.

## Timeline Visualization

To visualize NoC activity over time, generate timeline files with the `-e` flag:

`tt_npe.py -w workload.json -e --timeline-json-filepath my_timeline.npeviz`
**Timeline File Contents**:

*   Transfer start/end times per timestep
*   Link and NIU demand/utilization grids
*   Congestion state evolution
*   Transfer group dependencies

**Viewing Timelines**:

1.   Visit [https://ttnn-visualizer.tenstorrent.com/](https://ttnn-visualizer.tenstorrent.com/)
2.   Load the `.npeviz` file (or `.npeviz.zst` if compressed)
3.   Navigate through timesteps to observe NoC behavior

The timeline format is defined by [tt_npe/cpp/src/npeStats.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp) and serialized by `v1TimelineSerialization()` ([npeStats.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/npeStats.cpp)).

## Essential CLI Options

| CLI Option | Config Field | Default | Purpose |
| --- | --- | --- | --- |
| `-w, --workload` | `workload_json_filepath` | **(required)** | Path to workload JSON file |
| `-d, --device` | `device_name` | `wormhole_b0` | Device architecture (`wormhole_b0`, `blackhole`, `N150`, `N300`, `T3K`, `P100`, `P150`) |
| `-c, --cycles-per-timestep` | `cycles_per_timestep` | `32` | Cycles per simulation timestep (lower = more accurate, slower) |
| `--cong-model` | `congestion_model_name` | `fast` | Congestion model (`fast` or `none`) |
| `-e, --emit-timeline-file` | `emit_timeline_file` | `false` | Generate `.npeviz` timeline file |
| `--timeline-json-filepath` | `timeline_filepath` | `npe_timeline.json` | Output path for timeline |
| `-t, --workload-is-noc-trace` | `workload_is_noc_trace` | `false` | Parse input as tt-metal NoC trace format |
| `-v, --verbose` | `verbosity` | `0` | Enable verbose logging (0-3) |

See [tt_npe/py/pycli/tt_npe.py 13-109](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/pycli/tt_npe.py#L13-L109) for the complete argument parser definition and [tt_npe/cpp/include/npeConfig.hpp 16-67](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeConfig.hpp#L16-L67) for configuration struct details.

## Troubleshooting Quick Reference

| Error | Cause | Solution |
| --- | --- | --- |
| `Must provide a tt-npe workload JSON file` | Missing `-w` argument | Add `-w path/to/workload.json` |
| `Could not create tt-npe workload from file` | Invalid JSON or format | Check JSON syntax; verify `is_noc_trace_format` matches file type |
| `WORKLOAD_VALIDATION_FAILED` | Invalid coordinates or parameters | Check coordinates are within device grid; verify packet sizes > 0 |
| Timeline file not generated | Missing `-e` flag | Add `-e` or `--emit-timeline-file` |
| Multi-device traces failing | Traces not merged | Run `fabric_post_process.py` on trace directory first |

For detailed error messages, run with `-v` or `--verbose` for increased logging verbosity.
