## Purpose and Scope

This page documents the Llama model testing infrastructure in tt-forge-onnx, covering both inference and training workflows. The system supports multiple Llama variants (OpenLLama 3B, Llama-3.2-1B) with different execution strategies for optimizing performance on Tenstorrent hardware.

**Scope:**

*   Inference execution modes: no cache, cache on host, cache on device
*   Prefill and decode phase separation
*   KV (key-value) cache management strategies
*   LoRA (Low-Rank Adaptation) training
*   Model wrappers for different execution patterns

For general LLM model support, see [7.2 Text and LLM Models](https://deepwiki.com/tenstorrent/tt-forge-onnx/7.2-text-and-llm-models). For testing infrastructure details, see [6.3 Model Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.3-model-testing).

## Inference Architecture Overview

Llama inference in tt-forge-onnx follows a two-phase execution model: **prefill** and **decode**. The prefill phase processes the initial prompt to generate the first token, while the decode phase iteratively generates subsequent tokens using cached key-values from previous computations.

### Prefill vs Decode Phases

**Sources:**[forge/test/mlir/llama/test_llama_inference.py 49-102](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/test_llama_inference.py#L49-L102)[forge/test/mlir/llama/tests/test_llama_prefil.py 26-52](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_prefil.py#L26-L52)

### KV Cache Strategies

The system supports three KV cache strategies with different performance characteristics:

| Strategy | Cache Location | Memory Usage | Performance | Compile Overhead |
| --- | --- | --- | --- | --- |
| **No Cache** | N/A | Low | Slowest (recomputes all KV pairs) | Low |
| **Cache on Host** | CPU Memory | Medium | Moderate (transfers KV to/from device) | Medium |
| **Cache on Device** | Device Memory | High | Fastest (KV stays on device) | High |

**Sources:**[forge/test/mlir/llama/tests/test_llama_decode.py 240-326](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L240-L326)[forge/test/mlir/llama/tests/test_llama_decode.py 363-491](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L363-L491)[forge/test/mlir/llama/tests/test_llama_decode.py 501-666](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L501-L666)

## Model Wrappers

The testing infrastructure provides three specialized model wrappers for different execution patterns.

### Model Wrapper Architecture

**Sources:**[forge/test/mlir/llama/tests/test_llama_prefil.py 15-23](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_prefil.py#L15-L23)[forge/test/mlir/llama/tests/test_llama_decode.py 70-133](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L70-L133)[forge/test/mlir/llama/tests/test_llama_decode.py 135-196](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L135-L196)

### LlamaPrefillModel

This wrapper isolates the decoder component (without `lm_head`) for testing prefill operations independently. The `lm_head` layer can then run on CPU to generate tokens from hidden states.

**Key characteristics:**

*   Returns `last_hidden_state` instead of logits
*   Used in split prefill/decode testing pattern
*   Enables on-device prefill with CPU-based decode

**Implementation:**[forge/test/mlir/llama/tests/test_llama_prefil.py 15-23](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_prefil.py#L15-L23)

### LlamaModelWrapper

General-purpose wrapper supporting both cached and non-cached inference. Handles attention mask preparation and KV cache format conversion between flat lists and nested tuples.

**Key features:**

*   Embeds tokens before model forward pass
*   Prepares 4D causal attention masks (fixes torch.jit.trace issues)
*   Converts between legacy cache format and `DynamicCache`
*   Flattens/unflattens KV pairs for TVM compatibility

**Implementation:**[forge/test/mlir/llama/tests/test_llama_decode.py 70-133](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L70-L133)

**Input/Output signatures:**

```
# Without cache:
forward(input_ids, attention_mask) -> [logits]

# With cache:
forward(input_ids, attention_mask, position_ids, *past_key_values) 
    -> [logits, key1, value1, ..., keyN, valueN]
```

### LlamaModelStaticCacheWrapper

Optimized wrapper for decode phase using `StaticCache` for on-device KV cache management. Pre-allocates cache tensors to maximum sequence length, enabling efficient device-resident caching.

**Key features:**

*   Uses `StaticCache` with pre-allocated tensors
*   Accepts `cache_position` to avoid `get_seq_length()` calls
*   Cache tensors passed as model inputs (TVM compatibility)
*   Returns only logits (cache managed externally)

**Implementation:**[forge/test/mlir/llama/tests/test_llama_decode.py 135-196](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L135-L196)

**Input signature:**

```
forward(input_ids, attention_mask, position_ids, cache_position, 
        k_cache0, v_cache0, k_cache1, v_cache1, ...)
```

**Sources:**[forge/test/mlir/llama/tests/test_llama_decode.py 135-196](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L135-L196)[forge/test/mlir/llama/tests/test_llama_decode.py 162-195](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L162-L195)

## Inference Without KV Cache

This execution mode recomputes all key-value pairs for every token generation, making it the slowest but simplest approach.

### Execution Flow

### Implementation Pattern

**Test function:**[forge/test/mlir/llama/tests/test_llama_decode.py 240-326](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L240-L326)

**Key aspects:**

1.   Load model with `use_cache=False`
2.   Pad input to `max_position_embeddings`
3.   Run prefill to get initial token
4.   Iteratively append tokens and run full forward pass
5.   Extract token from appropriate position in logits

**Padding strategy:** The input is padded to `MAX_POSITION_EMBEDDINGS` (2048) to maintain consistent tensor shapes across iterations. The attention mask tracks which positions contain actual content vs padding.

**Token extraction:**[forge/test/mlir/llama/tests/test_llama_decode.py 274-276](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/test/mlir/llama/tests/test_llama_decode.py#L274-L276)

## Summary

The Llama testing infrastructure demonstrates three key execution patterns for large language model inference:

1.   **No Cache**: Simplest implementation, recomputes all KV pairs per token (CPU testing)
2.   **Cache on Host**: Balanced approach with KV cache in CPU memory, transferred per decode step
3.   **Cache on Device**: Most performant, uses `StaticCache` with device-resident KV tensors

The system also supports LoRA training with forward and backward pass verification, enabling efficient fine-tuning on Tenstorrent hardware. All patterns utilize specialized model wrappers (`LlamaPrefillModel`, `LlamaModelWrapper`, `LlamaModelStaticCacheWrapper`) to handle different execution requirements and cache management strategies.

For related model support, see [7.2 Text and LLM Models](https://deepwiki.com/tenstorrent/tt-forge-onnx/7.2-text-and-llm-models). For general testing patterns, see [6.3 Model Testing](https://deepwiki.com/tenstorrent/tt-forge-onnx/6.3-model-testing).
