The Parser Test Suite ensures the reliability and accuracy of the data extraction logic used to transform raw test artifacts (JUnit XML, TAR archives, and JSON) into structured telemetry. It validates the parser selection logic, specialized data mapping for ML kernels (OpTests), and the handling of edge cases in various test report formats.

## Parser Selection & Orchestration

The `optests.py` module contains the logic for selecting the appropriate parser based on the environment and file metadata. This is critical for distinguishing between standard CI tests and specialized "Builder" jobs that require detailed hardware and tensor metadata.

### Selection Logic

*   **BuilderPytestParser**: Selected if the job name contains "builder" or the filename contains `_builder`, and the git branch is `main`[[.github/actions/collect_data/src/optests.py:14-39]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **TTTorchModelTestsParser**: Selected if the report file extension is `.tar`[[.github/actions/collect_data/src/optests.py:42-48]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **Dispatch**: The `create_optest_reports` function iterates through discovered reports and assigns them to the correct parser instance [[.github/actions/collect_data/src/optests.py:51-91]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).

## Specialized Parsers

### TTTorch Model Tests Parser

The `TTTorchModelTestsParser` handles `.tar` archives containing multiple JSON files, each representing a model's operation compilation status.

*   **Implementation**: It untars the archive to `/tmp/`, iterates through all JSON files, and maps the `compilation_status` integer to an `OpCompilationStatus` enum [[.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py:30-51]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **Data Mapping**: It extracts `input_tensors` and `output_tensors` into `TensorDesc` Pydantic models, capturing shape, data type, and layout [[.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py:123-133]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **Status Logic**: A test is considered "failed" if the compilation status is less than `OpCompilationStatus.EXECUTED` (value 7) [[.github/actions/collect_data/src/parsers/tt_torch_model_tests_parser.py:73-76]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).

### Parameter Support Parser

The `ParameterSupportTestParser` processes JSON reports from `tt-shield` that document which model parameters (like `max_tokens` or `n`) are supported by specific endpoints.

*   **Metadata Precedence**: Values found in the top-level `metadata` key of the JSON override values in the `parameter_support_tests` block, except for the `results` key itself [[.github/actions/collect_data/src/parsers/parameter_support_test_parser.py:70-80]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **Configuration**: It populates a `ParameterSupportTestConfig` containing `model_name`, `model_impl`, `device`, and `endpoint_url`[[.github/actions/collect_data/src/parsers/parameter_support_test_parser.py:20-30]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).

### Python Unittest Parser

This parser handles standard Python `unittest` XML outputs. A key feature is its "Timestamp Reconstruction" logic: if individual test cases lack timestamps, it uses the testsuite timestamp and adds the duration of preceding tests to ensure unique `test_start_ts` values, which is a requirement for the downstream data pipeline [[.github/actions/collect_data/src/parsers/python_unittest_parser.py:40-48]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)), [[.github/actions/collect_data/src/parsers/python_unittest_parser.py:80-83]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).

## Test Suite Implementation

The test suite uses `pytest` with fixtures and golden data files located in `test/data/`.

### Test Data Fixtures

*   **XML Fixtures**: `test/data/17415392798/artifacts/.../report_49443670279.xml` provides real-world examples of `BuilderPytestParser` outputs, including properties like `card`, `backend`, and `failure_stage`[[test/data/17415392798/artifacts/test-reports-builder-n150-tracy-silicon-49443670279/report_49443670279.xml:1-15]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
*   **TAR Fixtures**: `test/data/tt_torch_models/run2.tar` is used to verify the `TTTorchModelTestsParser`'s ability to extract and flatten nested JSON results [[.github/actions/collect_data/test/test_tt_torch_model_tests_parser.py:9-21]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).

### Validation Patterns

The tests verify several critical behaviors:

1.   **Parser Selection**: `test_optests.py` uses `@pytest.mark.parametrize` to check every permutation of branch name and job name against the `should_use_builder_pytest_parser` function [[.github/actions/collect_data/test/test_optests.py:31-48]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
2.   **Error Message Precedence**: `test_tt_torch_full_model_tests_parser.py` ensures that `pytest.skip` messages take precedence over custom error messages when both are present in the XML [[.github/actions/collect_data/test/test_tt_torch_full_model_tests_parser.py:23-37]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
3.   **Metadata Integrity**: `test_parameter_support_parser.py` validates that the parser correctly merges configuration while protecting the actual test results from being overwritten by metadata blocks [[.github/actions/collect_data/test/test_parameter_support_parser.py:114-191]](https://deepwiki.com/tenstorrent/tt-github-actions/7.2-parser-test-suite)).
