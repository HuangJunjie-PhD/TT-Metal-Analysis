## Purpose and Scope

This page documents the architecture of TT-XLA's GitHub Actions CI/CD workflows, including workflow triggers, job orchestration, and optimization strategies. It covers how workflows are structured, how they coordinate with each other, and how they optimize execution based on code changes.

For details on build jobs and artifact management, see [Build and Artifact Management](https://deepwiki.com/tenstorrent/tt-xla/7.2-build-and-artifact-management). For test matrix generation and execution, see [Test Matrix Generation and Execution](https://deepwiki.com/tenstorrent/tt-xla/7.3-test-matrix-generation-and-execution). For duration tracking and reporting, see [Duration Tracking and Reporting](https://deepwiki.com/tenstorrent/tt-xla/7.4-performance-benchmarking).

## Workflow Trigger Types

TT-XLA uses four types of workflow triggers, each designed for different CI/CD scenarios:

| Trigger Type | Workflows | Purpose | Frequency |
| --- | --- | --- | --- |
| **Pull Request** | `pr-main.yml` | Validate changes before merge | On PR to main |
| **Push** | `push-main.yml` | Validate merged code, deploy docs | On push to main |
| **Schedule** | `schedule-nightly.yml`, `schedule-nightly-experimental.yml`, `schedule-weekly.yml`, `schedule-weekly-training.yml` | Comprehensive testing, regression detection | Daily/Weekly cron |
| **Manual** | `manual-test.yml`, `manual-test-single.yml` | Developer testing, debugging | On-demand |

**Sources:**[.github/workflows/pr-main.yml 1-5](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L1-L5)[.github/workflows/push-main.yml 1-9](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/push-main.yml#L1-L9)[.github/workflows/schedule-nightly.yml 1-6](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/schedule-nightly.yml#L1-L6)[.github/workflows/manual-test.yml 1-11](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/manual-test.yml#L1-L11)

### Pull Request Workflow

The `pr-main.yml` workflow is the primary gatekeeper for code quality. It implements several optimization strategies:

**Key Features:**

*   **Concurrency Control**: Cancels in-progress runs when new commits are pushed ([.github/workflows/pr-main.yml 15-17](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L15-L17))
*   **Change Inspection**: Uses `inspect-changes` action to skip unnecessary jobs ([.github/workflows/pr-main.yml 24-35](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L24-L35))
*   **Conditional Execution**: Runs different test suites based on change type and PR labels ([.github/workflows/pr-main.yml 84-112](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L84-L112))

**Sources:**[.github/workflows/pr-main.yml 1-176](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L1-L176)

### Push Workflow

The `push-main.yml` workflow runs on successful merges to the main branch. It differs from PR workflow by:

*   Running unconditionally (no `inspect-changes` optimization)
*   Deploying documentation via `call-build-and-deploy-docs.yml`
*   Using full test suites without change-based filtering

**Sources:**[.github/workflows/push-main.yml 1-78](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/push-main.yml#L1-L78)

## Job Dependency Graph

### PR Workflow Job DAG

The `pr-main.yml` workflow orchestrates multiple parallel and sequential jobs:

**Dependency Patterns:**

1.   **Sequential**: `inspect-changes` → `build-image` → `build-ttxla-*` → `test-*`
2.   **Parallel Builds**: `build-ttxla-release`, `build-ttxla-explorer`, `build-ttxla-debug` run concurrently after `build-image`
3.   **Parallel Tests**: All test jobs run concurrently after builds complete
4.   **Final Gate**: `check-all-green` waits for all jobs using `if: always()` and `needs: [...]` ([.github/workflows/pr-main.yml 154-175](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L154-L175))

## The inspect-changes Optimization

The `inspect-changes` action analyzes PR changes to determine which jobs need to run, significantly reducing CI time for documentation-only or minor changes.

### Outputs Produced

The action produces five boolean flags:

| Output | Purpose | Example Usage |
| --- | --- | --- |
| `skip-build-and-test` | Skip all build/test jobs | Documentation-only changes |
| `run-pr-tests` | Run basic PR test suite | Any code change |
| `run-perf-tests` | Run performance benchmarks | Performance-sensitive changes |
| `only-run-forge-models` | Only test model changes | Changes to test configs |
| `run-uplift-model-tests` | Run extended model tests | Changes labeled `uplift-mlir` |

**Sources:**[.github/workflows/pr-main.yml 24-35](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L24-L35)

### Usage Pattern

This pattern enables:

*   **Early termination**: Entire build/test pipeline skips if only docs changed
*   **Selective execution**: Run only relevant test suites based on change type
*   **Label-based triggers**: Special handling for labeled PRs (e.g., `uplift-mlir`)

**Sources:**[.github/workflows/pr-main.yml 24-49](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L24-L49)

## Concurrency Control

The PR workflow uses GitHub's concurrency control to prevent resource waste:

**Behavior:**

*   **Group Key**: `On PR-refs/pull/123/merge` - unique per PR
*   **Cancellation**: New commit cancels previous run for same PR
*   **Scope**: Only affects PR workflow, not push or schedule workflows

This ensures:

*   **Resource Efficiency**: Don't test stale code
*   **Fast Feedback**: Latest changes tested first
*   **Clean State**: Each run starts fresh

**Not used in:**

*   `push-main.yml`: Every commit to main should be tested
*   `schedule-*.yml`: Scheduled runs don't conflict
*   `manual-*.yml`: User controls execution

**Sources:**[.github/workflows/pr-main.yml 15-17](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/pr-main.yml#L15-L17)

**Principle**: Request only necessary permissions per workflow

## Notification System

Scheduled workflows implement Slack notifications for failure/success:

**Implementation:**

1.   `fail-notify` job uses `needs: [all-test-jobs]` and `if: always()` to run regardless of test outcomes
2.   `alls-green` action checks if any jobs failed
3.   Branch check ensures notifications only for main branch
4.   Separate webhooks for different notification types (nightly, weekly, experimental)

**Channels:**

*   `SLACK_NIGHTLY_FAIL` / `SLACK_NIGHTLY_SUCCESS`: Nightly test results
*   `SLACK_WEEKLY_FAIL` / `SLACK_WEEKLY_SUCCESS`: Weekly test results
*   `SLACK_EXPERIMENTAL_NIGHTLY`: Experimental test results

**Sources:**[.github/workflows/schedule-nightly.yml 75-128](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/schedule-nightly.yml#L75-L128)[.github/workflows/schedule-weekly.yml 48-99](https://github.com/tenstorrent/tt-xla/blob/c77995f6/.github/workflows/schedule-weekly.yml#L48-L99)

## Test Suite Selection Strategy

Different workflows run different test suite presets optimized for their execution context:

| Workflow | Test Suites | Rationale |
| --- | --- | --- |
| **pr-main.yml** | `basic-test.json` `model-test-push.json` `model-test-extended.json` | Fast feedback on basic tests Comprehensive model validation for uplift PRs |
| **push-main.yml** | `basic-test.json` `model-test-push.json` | Same as PR, validates merged code |
| **schedule-nightly.yml** | `basic-test-nightly.json` `model-test-full.json` `model-test-passing.json` `model-test-xfail.json` | Full coverage including known failures Separate passing/xfail for clarity |
| **schedule-nightly-experimental.yml** | `model-test-experimental.json` | Unknown status models, expected to fail |
| **schedule-weekly.yml** | `model-test-passing-weekly.json` `model-test-xfail-weekly.json` | Comprehensive weekly validation |
| **schedule-weekly-training.yml** | `model-test-training-xfail-weekly.json` | Training-specific tests |
| **manual-test.yml** | User-selected preset or custom JSON | Flexible developer testing |
