tt-emule provides a high-fidelity software emulation of the Tenstorrent Quasar hardware architecture. This implementation models the **Tensix Neo** core, which introduces a significant departure from the Wormhole/Blackhole architectures by replacing Circular Buffers (CBs) with Dataflow Buffers (DFBs) and expanding the threading model to support 12 concurrent processors per core.

## Tensix Neo Architecture Overview

A single Quasar "Neo" core is emulated as a collection of 12 host threads sharing a unified 4 MB L1 SRAM region.

| Component | Hardware Specification | Emulation Implementation |
| --- | --- | --- |
| **Data Movement (DM)** | 8 Processors (DM0-DM7) | 8 Host Threads (via `get_dm_processors()`) |
| **Compute Engines** | 4 Engines (E0-E3) | 4 Host Threads (via `get_compute_processors()`) |
| **Shared L1 SRAM** | 4 MB Unified | 4 MB Host Heap (via `Core::l1_alloc()`) |
| **Synchronization** | Dataflow Buffers (DFBs) | `TileCounterArray` + `EmuleDFBInterface` |
| **Hart IDs** | 0-7 (DM), 8-11 (Compute) | Regex-patched `mhartid` to `__processor_id` TLS |

## Dataflow Buffers (DFBs)

The DFB is the primary synchronization mechanism in Quasar, designed to support Multi-Producer Multi-Consumer (MPMC) access patterns. Unlike the SPSC (Single-Producer Single-Consumer) model of Wormhole CBs, DFBs allow multiple DM and Compute threads to interact with the same L1 buffer using independent **Tile Counters (TC)**.

### Tile Counter Primitives

The core synchronization primitive is the `TileCounter`, which tracks the delta between `posted` (produced) and `acked` (consumed) tiles using atomics and condition variables.

Sources: [docs/DFB_EMULATION.md 19-42](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L19-L42)[include/jit_hw/api/dfb_api.h 105-120](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L105-L120)

### Access Patterns: STRIDED vs. BLOCKED

The emulator implements the two primary hardware access patterns for DFBs:

1.   **STRIDED**: Used when multiple producers/consumers interleave their access. The `stride_size` determines the pointer jump after each operation. Each thread round-robins through a subset of `tc_slots`.
2.   **BLOCKED**: Each producer/consumer is assigned a contiguous block of the buffer. In this mode, producers typically `broadcast_tc` to all consumer tile counters simultaneously.

Sources: [docs/DFB_EMULATION.md 102-109](https://github.com/tenstorrent/tt-emule/blob/a57cf072/docs/DFB_EMULATION.md?plain=1#L102-L109)[include/jit_hw/api/dfb_api.h 85-103](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L85-L103)

## JIT Implementation & CSR Patching

Because Quasar kernels rely on hardware Control and Status Registers (CSRs) for identity, the JIT compilation pipeline performs regex-based source patching to redirect hardware reads to Thread-Local Storage (TLS) variables.

### CSR Mapping Table

| Hardware CSR | Patched Emulation Variable | Purpose |
| --- | --- | --- |
| `mhartid` | `__processor_id` | Unique ID (0-11) for the thread within the Neo core. |
| `NEO_ID` | `__emule_neo_id` | Identifies which of the 4 compute engines is running. |
| `TRISC_ID` | `__emule_trisc_id` | Identifies the RISC slot (Unpack/Math/Pack). |

### The DFB-CB Bridge

To maintain compatibility with legacy kernels that may still use CB-style APIs on Quasar, `cb_api.h` and `dfb_api.h` include bridge logic. If a CB operation is performed on an ID that is configured as a DFB, the emulator automatically updates the corresponding Tile Counters.

*   **`cb_push_back`**: If the DFB interface is active for that ID, it calls `inc_posted` on the `TileCounterArray`. [include/jit_hw/api/cb_api.h 135-154](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/cb_api.h#L135-L154)
*   **`dfb_push_back`**: If a legacy CB structure exists for that ID, it advances the CB write pointer and signals the CB semaphore. [include/jit_hw/api/dfb_api.h 81-84](https://github.com/tenstorrent/tt-emule/blob/a57cf072/include/jit_hw/api/dfb_api.h#L81-L84)
