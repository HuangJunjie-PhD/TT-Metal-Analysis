## Purpose and Scope

This page documents the **instruction set architecture (ISA) representation and decoding system** in Whisper. It covers how RISC-V instructions are encoded, identified, decoded, and disassembled within the simulator. The focus is on the instruction table (`InstTable`), instruction metadata (`InstEntry`), instruction identification (`InstId`), and the decoding process.

For information about instruction **execution** within the Hart execution pipeline, see [Hart Execution Unit](https://deepwiki.com/tenstorrent/whisper/2.2-hart-execution-unit). For details on specific ISA extensions (Vector, Hypervisor, Bit Manipulation, Crypto), see [RISC-V Extensions](https://deepwiki.com/tenstorrent/whisper/5-risc-v-extensions).

## Instruction Identification System

### InstId Enumeration

Every RISC-V instruction supported by Whisper is assigned a unique identifier through the `InstId` enumeration. This enum provides a type-safe way to reference instructions throughout the codebase and serves as the primary key for instruction lookup.

`enum class InstId : uint32_t{  illegal,      // Invalid/illegal instruction    // Base instructions (RV32I/RV64I)  lui, auipc, jal, jalr,  beq, bne, blt, bge, bltu, bgeu,  lb, lh, lw, lbu, lhu,  sb, sh, sw,  addi, slti, sltiu, xori, ori, andi,  slli, srli, srai,  add, sub, sll, slt, sltu, xor_, srl, sra, or_, and_,  fence, pause, fence_tso, fence_i,  ecall, ebreak,    // CSR instructions  csrrw, csrrs, csrrc, csrrwi, csrrsi, csrrci,    // RV64I extensions  lwu, ld, sd, addiw, slliw, srliw, sraiw,  addw, subw, sllw, srlw, sraw,    // ... (additional extensions)    maxId = c_mop  // Last instruction ID};`
The `InstId` enum spans from `illegal` (0) to `maxId`, currently encompassing over 1000 distinct instruction identifiers covering:

*   Base ISA (I extension)
*   Multiply/Divide (M extension)
*   Atomic operations (A extension)
*   Floating-point (F, D, Zfh extensions)
*   Compressed instructions (C extension)
*   Vector operations (V extension)
*   Bit manipulation (Zb* extensions)
*   Cryptography (Zk* extensions)
*   Hypervisor (H extension)

## Instruction Decoding Process

### Hart Decoder Integration

The `Hart` class maintains a `Decoder` instance and a decode cache for efficient instruction decoding:

#### Decode Cache Mechanism

The Hart maintains a direct-mapped decode cache to avoid redundant decoding of frequently executed instructions:

*   **Cache size:** 128K entries (configurable via `decodeCacheSize_`)
*   **Cache indexing:** Direct-mapped using PC bits: `index = (pc >> 1) & decodeCacheMask_`
*   **Cache invalidation:** When memory is modified via `pokeMemory()`, overlapping cache entries are invalidated

`// From Hart.hpp lines 95-98uint64_t decodeCacheSize_ = 128*1024;uint64_t decodeCacheMask_ = decodeCacheSize_ - 1;std::vector<DecodedInst> decodeCache_;`

## Instruction Decoding Example

### Decoding the `add x3, x4, x5` Instruction

**Bit layout for `add x3, x4, x5` (encoding: `0x005202b3`):**

```
31    25 24   20 19   15 14 12 11    7 6      0
0000000  00101  00100  000  00011  0110011
funct7   rs2    rs1    f3   rd     opcode
  0       x5     x4     0    x3     0x33 (OP)
```

The decoder:

1.   Matches opcode `0x33` and funct3=0, funct7=0 to identify `add`
2.   Extracts rd=3, rs1=4, rs2=5 using the operand masks
3.   Returns a `DecodedInst` with these operands

**Sources:**[Hart.cpp 1334-1422](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L1334-L1422)[InstEntry.cpp 571-575](https://github.com/tenstorrent/whisper/blob/733fd921/InstEntry.cpp#L571-L575)

## Instruction Disassembly

### Disassembler Class

The `Disassembler` class converts `DecodedInst` objects into human-readable assembly text:

`// Usage in Hartvoid Hart<URV>::disassembleInst(const DecodedInst& di, std::string& str){  disas_.disassembleInst(di, str);}`
#### Disassembly Features

*   **Register naming:** Supports both numeric (x0-x31) and ABI names (zero, ra, sp, gp, tp, etc.)
*   **CSR names:** Translates CSR numbers to symbolic names (e.g., `mstatus`, `stvec`)
*   **Immediate formatting:** Displays immediates in decimal or hexadecimal based on context

### Extension Enabling

Extensions are enabled through the MISA CSR and individual extension enable flags:

`// From Hart.cpp lines 469-647void Hart<URV>::processExtensions(bool verbose){  URV value = 0;  if (not peekCsr(CsrNumber::MISA, value))    std::cerr << "Error: CSR MISA is not defined\n";   // Check MISA bits and enable corresponding extensions  bool flag = value & (URV(1) << ('s' - 'a'));  enableSupervisorMode(flag);    // ... enable other extensions based on MISA and configuration}`
Extension enablement affects:

1.   **Instruction availability:** Instructions from disabled extensions decode as illegal
2.   **CSR accessibility:** Extension-specific CSRs become available
3.   **Execution behavior:** Execution paths specific to extensions are activated

Compressed (16-bit) instructions are expanded to 32-bit equivalents during decoding:

*   `isCompressed()`: Returns true for 16-bit instructions
*   `isCompressedRv32()`: True for RV32-only compressed variants (c.flw, c.jal)

## Performance Considerations

### Decode Cache Performance

The decode cache significantly improves simulation performance by eliminating redundant decoding:

*   **Cache hit rate:** Typically >99% for loop-heavy code
*   **Cache line invalidation:** Only occurs when code memory is modified (rare in typical execution)
*   **Direct-mapped design:** Provides O(1) lookup with minimal overhead

`// From Hart.cpp lines 1095-1124void Hart<URV>::invalidateDecodeCache(uint64_t addr, unsigned size){  // Invalidate cache entries for modified memory region  for (unsigned i = 0; i < size; ++i)    {      uint64_t a = addr + i;      unsigned index = (a >> 1) & decodeCacheMask_;      decodeCache_[index].reset();    }}`
### Instruction Property Caching

Frequently-checked instruction properties (branch type, load/store size, etc.) are pre-computed during `InstTable` initialization to avoid runtime overhead.

**Sources:**[Hart.cpp 1095-1124](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L1095-L1124)[Hart.cpp 88-149](https://github.com/tenstorrent/whisper/blob/733fd921/Hart.cpp#L88-L149)

## Extension-Specific Instruction Details

### Vector Instructions

Vector instructions have additional properties tracked:

*   **Element width (SEW):** 8, 16, 32, 64 bits
*   **Group multiplier (LMUL):** 1/8, 1/4, 1/2, 1, 2, 4, 8
*   **Masking:** Masked vs. unmasked operations
*   **Segment count:** For segmented loads/stores

Vector extension instructions are identified during `InstTable` initialization:

`// From InstEntry.cpp lines 49-76for (unsigned i = 0; InstId(i) <= InstId::maxId; ++i){  auto& entry = instVec_.at(i);  auto ext = entry.extension();  switch(ext)  {    case RvExtension::V:    case RvExtension::Zvfh:    // ... other vector extensions      entry.setVector(true);      break;  }}`
### Floating-Point Instructions

FP instructions track:

*   **Rounding mode:** Static (encoded) or dynamic (from FCSR)
*   **FFLAGS modification:** Which FP exceptions may be raised
*   **NaN boxing:** Required for single-precision on RV64

## Summary

The Instruction Set Architecture system in Whisper provides:

1.   **Comprehensive instruction coverage:** 1100+ instructions across 40+ RISC-V extensions
2.   **Efficient decoding:** Direct-mapped decode cache with >99% hit rate
3.   **Flexible lookup:** By InstId, instruction name, or opcode pattern matching
4.   **Rich metadata:** Operand types, formats, extension association, and behavioral properties
5.   **Extensibility:** Clean separation between instruction definition and execution logic

The instruction table serves as the **central registry** for all ISA information, enabling the Hart to decode, execute, and trace instructions uniformly regardless of their extension or encoding format.
