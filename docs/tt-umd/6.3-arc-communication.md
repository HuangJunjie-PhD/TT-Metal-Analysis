## Overview

The ARC (Adaptive Runtime Controller) is an embedded RISC-V processor on Tenstorrent chips that runs firmware and provides control and monitoring services. The ARC communication system consists of several specialized components:

*   **ArcMessenger**: Implements a message-based command/response protocol for control operations (e.g., iATU configuration, RISC reset, power state changes).
*   **ArcTelemetryReader**: Provides runtime monitoring data (e.g., clock speeds, temperature, training status, harvesting masks) by reading a structured telemetry buffer.
*   **FirmwareInfoProvider**: A data-driven abstraction layer that queries firmware versions and board identification while maintaining backward compatibility across different firmware releases.
*   **SPITTDevice**: Provides access to the SPI device connected to the ARC, allowing for flash memory operations and board information retrieval.

This page covers ARC initialization, message protocols, telemetry mapping, and architecture-specific implementations for Wormhole and Blackhole.

## ArcMessenger: Control Protocol

`ArcMessenger` handles synchronous command execution. The implementation varies significantly between architectures.

### Wormhole Implementation

Wormhole uses a legacy mailbox-style communication where the host writes a message ID and arguments to scratch registers and polls for a response.

### Blackhole Implementation: Message Queues

Blackhole utilizes a more robust queue-based system managed by `BlackholeArcMessageQueue`. This implementation uses a circular buffer in ARC memory with read and write pointers (`rptr`/`wptr`).

*   **Queue Structure**: Each queue has a control block containing the base address and number of entries.
*   **Entry Size**: Fixed at 8 words (1 header/status word + 7 argument/return words).
*   **Trigger**: Firmware is notified of new requests via a firmware interrupt (`trigger_fw_int()`).

**Sources:**[device/arc/blackhole_arc_message_queue.cpp 53-75](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/blackhole_arc_message_queue.cpp#L53-L75)[device/arc/blackhole_arc_message_queue.cpp 103-142](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/blackhole_arc_message_queue.cpp#L103-L142)[device/arc/blackhole_arc_message_queue.cpp 144-167](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/blackhole_arc_message_queue.cpp#L144-L167)

## ArcTelemetryReader: Monitoring

`ArcTelemetryReader` provides a tag-based interface to device status. The mapping of tags to physical memory offsets is architecture and firmware-dependent.

### Telemetry Tags

Tags are defined in the `TelemetryTag` enum and include identifiers for clocks, temperatures, and hardware status.

| Tag | Enum Value | Description |
| --- | --- | --- |
| `AICLK` | 14 | AI Clock Frequency |
| `ASIC_TEMPERATURE` | 11 | Chip temperature |
| `BOARD_ID_HIGH` | 1 | High 32 bits of Board ID |
| `GDDR_STATUS` | 22 | DRAM training status |
| `ETH_FW_VERSION` | 24 | Ethernet firmware version |

**Sources:**[device/api/umd/device/types/telemetry.hpp 11-78](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/types/telemetry.hpp#L11-L78)

### SMBus Telemetry

For certain configurations, telemetry can be accessed via SMBus using `SmbusArcTelemetryReader`. This is primarily used for out-of-band monitoring.

**Sources:**[device/arc/smbus_arc_telemetry_reader.cpp 15-20](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/smbus_arc_telemetry_reader.cpp#L15-L20)

## FirmwareInfoProvider

`FirmwareInfoProvider` acts as a data-driven translation layer. It uses a `FirmwareFeatures` map to associate high-level features (like `AICLK`) with specific `TelemetryTag`s or `FixedValue`s based on the detected firmware version.

### Version Compatibility

The provider supports multiple firmware "bases":

*   **Legacy (18.3)**: Uses `WormholeTag` for Wormhole.
*   **Transitional (18.4 - 18.7)**: Introduces the `StandardTag` base.
*   **Modern (18.8+)**: Current feature set for Wormhole and Blackhole.

**Sources:**[device/firmware/firmware_info_provider.cpp 53-71](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/firmware/firmware_info_provider.cpp#L53-L71)[device/firmware/firmware_info_provider.cpp 76-117](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/firmware/firmware_info_provider.cpp#L76-L117)[device/api/umd/device/firmware/firmware_info_provider.hpp 201-213](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/firmware/firmware_info_provider.hpp#L201-L213)

## SPI Device Communication

The `SPITTDevice` hierarchy provides low-level access to the SPI flash controller connected to the ARC.

### Wormhole SPI (`WormholeSPITTDevice`)

Wormhole SPI access involves:

1.   **Pin Multiplexing**: Configuring GPIO2 pads for SPI functionality (SCK, CS, MOSI, MISO).
2.   **Clock Configuration**: Setting the baud rate based on the current `ARCCLK`.
3.   **Controller Setup**: Configuring the `SPI_CTRLR0` and `SPI_SSIENR` registers.

**Key Registers:**

*   `SPI_CNTL`: Master SPI enable/disable.
*   `SPI_DR`: Data register for TX/RX.
*   `SPI_SR`: Status register (Busy, TX Empty, RX Not Empty).

**Sources:**[device/arc/wormhole_spi_tt_device.cpp 29-42](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/wormhole_spi_tt_device.cpp#L29-L42)[device/arc/wormhole_spi_tt_device.cpp 129-164](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/wormhole_spi_tt_device.cpp#L129-L164)[device/arc/wormhole_spi_tt_device.cpp 171-200](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/arc/wormhole_spi_tt_device.cpp#L171-L200)

## Firmware Interaction Logic

### AI Clock (AICLK) Management

The driver can request the ARC to transition between power states using specific messages.

*   **Wormhole**: Uses `ARC_MSG_ARC_GO_BUSY` and `ARC_MSG_ARC_GO_LONG_IDLE`.
*   **Blackhole**: Uses `blackhole::ArcMessageType::AICLK_GO_BUSY` and `blackhole::ArcMessageType::AICLK_GO_LONG_IDLE`.

The driver typically waits for the clock to settle by polling the `AICLK` telemetry tag.

**Sources:**[tests/blackhole/test_arc_messages_bh.cpp 85-119](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/blackhole/test_arc_messages_bh.cpp#L85-L119)[tests/wormhole/test_arc_messages_wh.cpp 51-94](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/tests/wormhole/test_arc_messages_wh.cpp#L51-L94)

### DRAM Training Status

The `GDDR_STATUS` (or `DDR_STATUS`) tag contains a bitmask where each DRAM channel is represented by 4 bits (on Wormhole). The `FirmwareInfoProvider` parses these bits into `DramTrainingStatus` enums (`SUCCESS`, `FAIL`, `IN_PROGRESS`).

**Sources:**[device/firmware/wormhole_18_3_firmware_info_provider.cpp 60-87](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/firmware/wormhole_18_3_firmware_info_provider.cpp#L60-L87)
