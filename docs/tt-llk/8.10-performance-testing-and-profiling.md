Performance testing and profiling in the tt-llk repository provides infrastructure for benchmarking kernel execution and isolating performance characteristics of individual hardware pipeline stages (unpack, math, pack). This system enables precise cycle-count measurement and bottleneck identification for LLK operations.

For information about functional testing and validation, see page [8.2](https://deepwiki.com/tenstorrent/tt-llk/8.2-testconfig-and-test-orchestration) (TestConfig and Test Orchestration). For coverage collection, see page [8.11](https://deepwiki.com/tenstorrent/tt-llk/8.11-coverage-collection-and-analysis).

## Performance Testing Architecture

The performance testing system is built on top of the standard testing infrastructure but adds specialized profiling capabilities and multiple run types to isolate different aspects of kernel performance.

**Performance Test vs Functional Test**

Performance tests use `PerfConfig` instead of `TestConfig` and measure hardware cycle counts using on-chip profiler buffers. The key differences are:

| Aspect | Functional Test | Performance Test |
| --- | --- | --- |
| Configuration Class | `TestConfig` | `PerfConfig` |
| Primary Goal | Correctness validation | Cycle count measurement |
| Profiler Build | Optional | Always enabled (`ProfilerBuild.Yes`) |
| Run Types | Single execution | Multiple isolation modes |
| Output | Pass/fail + data validation | CSV reports with cycle counts |
| Test Marker | `@pytest.mark.test` | `@pytest.mark.perf` |

Sources: [tests/python_tests/helpers/perf.py 268-302](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/perf.py#L268-L302)

### PerfConfig Class

`PerfConfig` extends `TestConfig` with performance-specific functionality:

The constructor automatically:

## PerfRunType Enumeration

The `PerfRunType` enum defines different execution modes that isolate specific pipeline stages or measure end-to-end performance:

**PerfRunType Definition and Usage**

| Run Type | Stage Active | Purpose | Mock Operations |
| --- | --- | --- | --- |
| `L1_TO_L1` | All | End-to-end performance | None - full execution |
| `UNPACK_ISOLATE` | Unpack only | Measure unpacker bandwidth | Math clears valid bits, pack does nothing |
| `MATH_ISOLATE` | Math only | Measure compute performance | Unpack sets valid bits, pack does nothing |
| `PACK_ISOLATE` | Pack only | Measure packer bandwidth | Unpack/math do nothing |
| `L1_CONGESTION` | Math + Pack | Measure without unpack contention | Unpack does nothing |

The `perf.h` header provides mock functions that simulate data flow without actual computation:

**Unpack Mock Functions:**

## Profiler Infrastructure

The profiler system uses hardware performance counters in L1 memory to track cycle counts for different execution phases. Tests define profiling zones using macros that instrument the code.

### Profiler Macros

**ZONE_SCOPED(marker_name)**

Marks a code region for profiling. The profiler records cycle counts between the start and end of each zone:

**PROFILER_SYNC()**

Inserts a synchronization point ensuring all preceding hardware operations complete before the profiler records the timestamp. This prevents instruction pipelining from causing measurement inaccuracies.

## PerfReport System

`PerfReport` is a lazy-evaluation container for collecting, filtering, and exporting performance benchmark data across multiple test variants.

### PerfReport Class Structure

**Key Methods:**

*   `append(frame: pd.DataFrame)` - Adds performance data from a test run
*   `filter(column: str, value: Any)` - Creates filtered view without modifying original data

*   `post_process()` - Applies transformations like per-tile normalization

## Performance Data Output and Analysis

### CSV Report Structure

Performance tests generate CSV files in the `perf_data/` directory with the following structure:

**Column Categories:**

1.   **Parameter Columns** - Test configuration

    *   `formats.input_A`, `formats.input_B`, `formats.output`
    *   `unpack_to_dest`, `dest_acc`
    *   Custom parameters from `templates` and `runtimes` (e.g., `tile_count`, `math_fidelity`)

2.   **Marker Column** - Profiling zone name

    *   `INIT`
    *   `TILE_LOOP`
    *   Custom markers defined in kernel code

3.   **Statistic Columns** - Per run type

    *   `mean(L1_TO_L1)` - Average cycles for full pipeline
    *   `std(L1_TO_L1)` - Standard deviation
    *   `mean(UNPACK_ISOLATE)`, `std(UNPACK_ISOLATE)`
    *   `mean(MATH_ISOLATE)`, `std(MATH_ISOLATE)`
    *   `mean(PACK_ISOLATE)`, `std(PACK_ISOLATE)`
    *   `mean(L1_CONGESTION)`, `std(L1_CONGESTION)`

**Example CSV Row (conceptual):**

| formats.input_A | tile_count | marker | mean(L1_TO_L1) | mean(UNPACK_ISOLATE) | mean(MATH_ISOLATE) | mean(PACK_ISOLATE) |

## Profiler Buffer Memory Layout

The hardware profiler stores cycle count data in dedicated L1 memory regions:

| Memory Region | Address Range | Purpose |
| --- | --- | --- |
| Profiler Buffer 0 | 0x16B000 - 0x16C000 | TRISC0 (Unpack) profiling data |
| Profiler Buffer 1 | 0x16C000 - 0x16D000 | TRISC1 (Math) profiling data |
| Profiler Buffer 2 | 0x16D000 - 0x16E000 | TRISC2 (Pack) profiling data |

Each buffer stores timestamped entries for `ZONE_SCOPED` regions. The Python infrastructure reads these buffers via `device.read_from_device()` after kernel execution completes.
