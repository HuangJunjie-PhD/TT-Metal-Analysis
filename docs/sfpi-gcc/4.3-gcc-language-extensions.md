## Purpose and Scope

This page documents GCC's extensions to the C and C++ languages that are not part of the ISO standard. These extensions include additional numeric types, aggregate type extensions, and target-specific built-in functions. For C++ template-specific features, see [C++ Template System](https://deepwiki.com/tenstorrent/sfpi-gcc/4.1-c++-template-system). For information about the pass management system that processes these extensions, see [Pass Management and Optimization Pipeline](https://deepwiki.com/tenstorrent/sfpi-gcc/3.3-pass-management-and-optimization-pipeline).

GCC provides these extensions across multiple language frontends and target architectures. This page focuses on general language extensions and PowerPC-specific extensions, which serve as the primary example of target-specific built-in systems in this codebase.

## Additional Numeric Types

GCC extends the standard C type system with several additional numeric types to support larger integers, extended floating-point precision, and specialized arithmetic operations.

### Integer Extensions

| Type | Description | Availability |
| --- | --- | --- |
| `__int128` | 128-bit signed integer | Targets with 128-bit integer mode support |
| `unsigned __int128` | 128-bit unsigned integer | Same as __int128 |
| `long long int` | At least 64-bit signed integer | ISO C99, GCC extension in C90 mode |
| `unsigned long long int` | At least 64-bit unsigned integer | ISO C99, GCC extension in C90 mode |

The `__int128` type is supported on targets with an integer mode wide enough to hold 128 bits. Integer constants of these types use suffixes `LL`/`ULL` for long long and require special handling for `__int128` on platforms where `long long` is less than 128 bits.

### Complex Number Support

GCC supports complex integer and floating-point types using the `_Complex` keyword (or `__complex__` as a GNU extension). Complex types consist of real and imaginary parts of the same underlying type.

Complex constants use suffixes `i`, `I`, `j`, or `J` for the imaginary component:

**Complex Number Operations**

| Operation | Syntax | Description |
| --- | --- | --- |
| Extract real part | `__real__ exp` | Returns real component as lvalue |
| Extract imaginary part | `__imag__ exp` | Returns imaginary component as lvalue |
| Complex conjugation | `~exp` | Negates imaginary part |
| Build complex | `__builtin_complex(real, imag)` | Constructs complex from parts |

The `__real__` and `__imag__` operators work on lvalues, allowing both reading and writing of complex components. GCC may allocate complex variables non-contiguously with real and imaginary parts in different locations.

Sources: [gcc/doc/extend.texi 102-211](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/doc/extend.texi#L102-L211)

## Aggregate Type Extensions

GCC extends C's aggregate types (arrays, structures, unions) with several features that provide flexibility for system programming and dynamic memory management.

### Variable-Length Arrays

Variable-length arrays (VLAs) are arrays declared with a non-constant size expression, allocated on the stack:

VLAs are part of ISO C99 but accepted by GCC in C90 mode and C++ as an extension. Storage is allocated at declaration and deallocated when leaving scope.

**VLA Scope and Lifetime**

GCC extends VLAs to support:

*   VLAs as structure members
*   VLAs as function parameters
*   Parameter forward declarations for VLA dimensions

Sources: [gcc/doc/extend.texi 591-681](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/doc/extend.texi#L591-L681)

### Zero-Length and Flexible Array Members

GCC supports zero-length arrays as the last member of a structure, providing a pattern for variable-sized objects:

The ISO C99 standard introduced **flexible array members** with slightly different semantics:

**Differences Between Zero-Length Arrays and Flexible Array Members**

| Feature | Zero-Length Array | Flexible Array Member |
| --- | --- | --- |
| Syntax | `contents[0]` | `contents[]` |
| sizeof | Evaluates to 0 (quirk) | Undefined (incomplete type) |
| Position | Any position (discouraged) | Must be last member |
| In unions | Allowed (GCC extension) | Not allowed (GCC allows as extension) |
| Static initialization | Supported by GCC | Supported by GCC |

GCC allows flexible array members in positions not permitted by ISO C, with specific rules about their placement and initialization.

Sources: [gcc/doc/extend.texi 683-840](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/doc/extend.texi#L683-L840)

### Overloaded Built-in Functions

PowerPC built-ins support function overloading, where a single function name resolves to different implementations based on argument types. These are defined in `rs6000-overload.def`:

**Overload Resolution System**

Example overload definition:

```
[VEC_ABS, vec_abs, __builtin_vec_abs]
  vsc __builtin_vec_abs (vsc);
    ABS_V16QI
  vss __builtin_vec_abs (vss);
    ABS_V8HI
  vsi __builtin_vec_abs (vsi);
    ABS_V4SI
  vf __builtin_vec_abs (vf);
    ABS_V4SF
```

The overload system performs type-based resolution at compile time, selecting the appropriate low-level built-in based on argument types. If no exact match is found, implicit conversions may be applied to vector types.

Sources: [gcc/config/rs6000/rs6000-overload.def 1-80](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/rs6000/rs6000-overload.def#L1-L80)[gcc/config/rs6000/rs6000-overload.def 208-296](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/rs6000/rs6000-overload.def#L208-L296)

### AltiVec Vector Extensions

AltiVec provides SIMD vector types and operations for PowerPC. The extension is exposed through the `altivec.h` header and controlled by the `-maltivec` compiler flag.

**AltiVec Type System**

The AltiVec extension uses conditional macros that expand based on context. The `rs6000_macro_to_expand()` function in `rs6000-c.cc` peeks ahead at tokens to determine whether keywords like `vector`, `bool`, and `pixel` should be expanded to their `__`-prefixed forms.

**Vector Keyword Context Resolution**

1.   Checks if AltiVec is enabled (`TARGET_ALTIVEC`)
2.   Peeks at the next token(s)
3.   Determines if the next token is `bool`, `pixel`, or a valid type
4.   Expands `vector` to `__vector` if appropriate
5.   May also expand subsequent `bool` or `pixel` keywords

### Built-in Initialization and Registration

The PowerPC backend initializes built-ins through a multi-stage process:

**Built-in Initialization Flow**

The initialization process:

1.   **Type Initialization**: Creates fundamental types (vector types, opaque types, special types)
2.   **Built-in Registration**: Walks through generated tables, registering each built-in with appropriate type signatures
3.   **Overload Setup**: Initializes overload resolution tables for multi-type functions
4.   **Target-Specific Setup**: Configures MMA, HTM, and other specialized features

Each built-in is registered with a unique code in the `RS6000_BUILTIN_*` enumeration, a name, a type signature, and an ISA feature mask.

Sources: [gcc/config/rs6000/rs6000-builtin.cc 1-100](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/rs6000/rs6000-builtin.cc#L1-L100)

*   **No TOC**: `TARGET_HAS_TOC` is 0
*   **Different ABI**: Uses `ABI_DARWIN` instead of `ABI_AIX`
*   **128-bit long double**: Default to 128-bit even on older systems
*   **Stack alignment**: 128-bit boundary for both 32 and 64-bit
*   **Different register usage**: R2 and R13 available (not fixed)

## Summary

GCC's language extensions provide:

1.   **Additional Numeric Types**: Extended integer (`__int128`), complex, floating-point (`__float128`, `__ibm128`, decimal), and fixed-point types
2.   **Aggregate Extensions**: Variable-length arrays, zero-length arrays, flexible array members, empty structures, and unnamed fields
3.   **PowerPC Extensions**: Comprehensive vector types and 700+ built-in functions with overload resolution

These extensions are processed through target-specific hooks, type initialization systems, and specialized compiler passes. The PowerPC backend serves as a comprehensive example with its AltiVec/VSX vector types, MMA matrix operations, and hardware transactional memory support.

The extension system integrates with GCC's compilation pipeline through:

*   Frontend hooks for macro expansion and keyword handling
*   Type system initialization for vector and extended types
*   Built-in function registration and overload resolution
*   Target-specific machine descriptions for instruction patterns
*   Attribute processing for specialized built-in behavior
