## Purpose and Scope

This document describes the quantization strategies implemented in vLLM for reducing model memory footprint and improving inference throughput. Quantization techniques compress model weights and/or activations from their original precision (typically FP16/BF16) to lower-precision formats (FP8, INT8, INT4, etc.).

This page focuses on the quantization methods available, their configurations, and kernel implementations. For information about loading quantized models, see [Model Registry and Loading](https://deepwiki.com/tenstorrent/vllm/3.1-model-registry-and-loading). For MoE-specific implementation details, see [Mixture of Experts](https://deepwiki.com/tenstorrent/vllm/3.5-mixture-of-experts). For attention-specific quantization (KV cache), see [Attention Mechanisms](https://deepwiki.com/tenstorrent/vllm/2.6-attention-mechanisms).

## Quantization Method Overview

vLLM supports multiple quantization strategies that can be categorized by their approach to weight and activation quantization.

### Supported Quantization Methods

| Method | Weight Precision | Activation Precision | Group Size | Platforms |
| --- | --- | --- | --- | --- |
| FP8 | 8-bit FP8 | 8-bit FP8 or dynamic | per-tensor/per-block | CUDA (SM 8.9+), ROCm |
| INT8 | 8-bit INT8 | 16-bit or 8-bit | per-channel/per-tensor | CUDA, ROCm, CPU, XPU |
| INT4 (GPTQ) | 4-bit INT4 | 16-bit | per-group (32-128) | CUDA, ROCm |
| INT4 (AWQ) | 4-bit INT4 | 16-bit | per-group (64-128) | CUDA, ROCm |
| NVFP4 | 4-bit FP4 | varies | per-block | CUDA (SM 10.0+) |
| GGUF | 2-8 bit | 16-bit | varies | CUDA, ROCm, CPU |
| Compressed Tensors | varies | varies | varies | Multi-platform |

## Weight Quantization Methods

### FP8 Quantization (W8A8)

FP8 quantization uses 8-bit floating-point representation for both weights and activations. vLLM implements both per-tensor and block-wise FP8 quantization.

**Key Classes:**

*   `Fp8Config`: Configuration class that specifies FP8 parameters [vllm/model_executor/layers/quantization/fp8.py 71-141](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L71-L141)
*   `Fp8LinearMethod`: Implements FP8 linear layers [vllm/model_executor/layers/quantization/fp8.py 209-267](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L209-L267)
*   `Fp8MoEMethod`: FP8 quantization for MoE layers [vllm/model_executor/layers/quantization/fp8.py 392-579](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L392-L579)

**Activation Schemes:**

*   **Static**: Activation scales are pre-computed and stored in the checkpoint [vllm/model_executor/layers/quantization/fp8.py 336-342](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L336-L342)
*   **Dynamic**: Activation scales computed at runtime using per-tensor or per-token quantization [vllm/model_executor/layers/quantization/fp8.py 249-266](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L249-L266)

**Block-wise Quantization:** Block-wise quantization divides weight matrices into blocks and applies separate scaling factors to each block, improving accuracy at the cost of slightly higher memory usage [vllm/model_executor/layers/quantization/fp8.py 256-262](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L256-L262)

### INT4 Quantization (GPTQ/AWQ)

INT4 quantization compresses weights to 4 bits while keeping activations at 16 bits. Two primary methods are supported:

#### GPTQ (Group-wise Quantization)

GPTQ uses group-wise quantization with optional activation ordering. The implementation supports the Marlin kernel for efficient INT4 inference.

**Key Parameters:**

*   `group_size`: Number of weights sharing a scale (typically 32, 64, or 128) [vllm/model_executor/layers/quantization/gptq_marlin.py 88-91](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/gptq_marlin.py#L88-L91)
*   `desc_act`: Whether activation ordering is used [vllm/model_executor/layers/quantization/gptq_marlin.py 92](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/gptq_marlin.py#L92-L92)
*   `bits`: Quantization bits (4 or 8) [vllm/model_executor/layers/quantization/gptq_marlin.py 86](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/gptq_marlin.py#L86-L86)

**Sources:**[vllm/model_executor/layers/quantization/gptq_marlin.py 1-485](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/gptq_marlin.py#L1-L485)[csrc/quantization/gptq_marlin/](https://github.com/tenstorrent/vllm/blob/cd9e5b83/csrc/quantization/gptq_marlin/)

#### AWQ (Activation-aware Weight Quantization)

AWQ protects salient weight channels based on activation statistics, achieving better accuracy than naive quantization.

**Key Differences from GPTQ:**

*   Uses activation-aware channel importance
*   Typically uses larger group sizes (64-128)
*   Different packing format requiring AWQ-specific repacking

**Sources:**[vllm/model_executor/layers/quantization/awq_marlin.py 1-350](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/awq_marlin.py#L1-L350)[vllm/_custom_ops.py 342-360](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/_custom_ops.py#L342-L360)

### GGUF Quantization

GGUF supports multiple quantization formats from 2 to 8 bits, primarily for CPU inference but also with CUDA acceleration.

**Supported Formats:**

*   Q2_K, Q3_K, Q4_K, Q5_K, Q6_K: K-quantization variants
*   Q4_0, Q4_1: 4-bit with different zero-point strategies
*   Q8_0: 8-bit quantization

## Activation Quantization

Activation quantization strategies determine how intermediate activations are quantized during inference.

### Per-Tensor vs Per-Token

**Per-Tensor Quantization:**

*   Single scale factor for entire activation tensor
*   Lower memory overhead
*   Used by default for FP8 on older GPUs [vllm/model_executor/layers/quantization/fp8.py 252](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L252-L252)

**Per-Token Quantization:**

*   Separate scale factor for each token
*   Better accuracy for outlier tokens
*   Used on GPUs with CUTLASS support [vllm/model_executor/layers/quantization/fp8.py 250](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L250-L250)

**Per-Block Quantization:**

*   Scale factors for blocks of activations
*   Balance between accuracy and memory
*   Supported in block-wise FP8 [vllm/model_executor/layers/quantization/fp8.py 256-262](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L256-L262)

### Marlin Kernels

Marlin kernels provide efficient inference for INT4 and FP8 quantization on GPUs without native support.

**Supported Configurations:**

*   INT4 weights with INT16 activations (W4A16)
*   FP8 weights on GPUs without FP8 hardware support
*   Group sizes: 32, 64, 128, or channel-wise

**Weight Preprocessing:** Marlin requires weight repacking to optimize memory access patterns:

1.   Generate permutation matrix based on group size [csrc/quantization/gptq_marlin/gptq_marlin_repack.cu](https://github.com/tenstorrent/vllm/blob/cd9e5b83/csrc/quantization/gptq_marlin/gptq_marlin_repack.cu)
2.   Repack weights in Marlin-specific format
3.   Permute scales and zero-points if applicable

**MoE Support:** Marlin provides specialized kernels for MoE layers with expert-wise weight layout [CMakeLists.txt 915-935](https://github.com/tenstorrent/vllm/blob/cd9e5b83/CMakeLists.txt#L915-L935)

## Configuration and Selection

### Automatic Quantization Method Selection

vLLM automatically selects the appropriate quantization kernel based on the model configuration, hardware capabilities, and environment variables.

**Selection Criteria:**

1.   **Device Capability**: Check SM version for CUTLASS/Marlin support [vllm/platforms/cuda.py 89-96](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cuda.py#L89-L96)
2.   **Quantization Format**: FP8, INT4, INT8, etc. from config
3.   **Group Size**: For INT4, determines Marlin/Machete compatibility
4.   **Platform**: CUDA, ROCm, CPU, XPU have different kernel availability
5.   **Environment Variables**: `VLLM_USE_MARLIN`, `VLLM_ATTENTION_BACKEND`, etc.

**Sources:**[vllm/model_executor/layers/quantization/base_config.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/base_config.py)[vllm/model_executor/layers/quantization/fp8.py 166-186](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L166-L186)

## Platform-Specific Considerations

### CUDA Platform

**Supported Quantization:**

*   FP8 (SM 8.9+): Native support via CUTLASS
*   FP8 (SM < 8.9): Marlin kernels
*   INT4: Marlin (all), Machete (SM 9.0+)
*   NVFP4: Blackwell (SM 10.0+) only

**Capability Checks:** Platform capability checks determine available quantization methods [vllm/platforms/cuda.py 398-400](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/platforms/cuda.py#L398-L400):

### ROCm Platform

**Supported Quantization:**

*   FP8 via AIter (gfx9 architectures)
*   INT4 via Marlin or Triton
*   INT8 via Triton

*   Paged attention with FP8 KV cache

**Weight Shuffling:** ROCm requires weight shuffling for optimal AIter performance [vllm/model_executor/layers/quantization/fp8.py 418-426](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/layers/quantization/fp8.py#L418-L426)
