## Purpose and Scope

This document describes the test scenarios for the SSTC (Supervisor-mode Supervisor Timer Compare) extension, which provides supervisor-mode timer functionality through the `stimecmp` and `vstimecmp` CSRs. The SSTC test plan validates access control hierarchies across privilege modes, environment configuration bits, and interrupt pending register behavior.

For general test step primitives used in these scenarios, see [Test Steps Reference](https://deepwiki.com/tenstorrent/riscv-coretp/5-test-steps-reference). For information on counter enable tests without timer comparison, see [Counter Enable Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.2-counter-enable-tests). For PMU-related interrupt testing, see [PMU Overflow Tests](https://deepwiki.com/tenstorrent/riscv-coretp/4.3.3-pmu-overflow-tests-(sscofpmf)).

**Sources**: [coretp/plans/sstc/__init__.py 1-14](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/__init__.py#L1-L14)[coretp/plans/sstc/sstc_scenarios.py 1-11](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L1-L11)

## SSTC Extension Overview

The SSTC extension introduces supervisor-mode timer interrupts controlled by compare registers rather than software-managed timer values. Key components include:

| Component | Description | Access Control |
| --- | --- | --- |
| `stimecmp` | Supervisor timer compare register | Controlled by `mcounteren.TM`, `menvcfg.STCE` |
| `vstimecmp` | Virtual supervisor timer compare register | Controlled by `hcounteren.TM`, `henvcfg.STCE` |
| `time` | Read-only time counter CSR | Controlled by `mcounteren.TM`, `hcounteren.TM`, `scounteren.TM` |
| `menvcfg.STCE` | Machine environment config bit enabling SSTC | Bit 63, controls M-mode behavior |
| `henvcfg.STCE` | Hypervisor environment config bit | Bit 63, read-only zero when `menvcfg.STCE=0` |
| `mip.STIP` | Supervisor timer interrupt pending | Read-only when `menvcfg.STCE=1` |

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 12-101](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L12-L101)

## CSR Access Control Hierarchy

**Diagram: SSTC CSR Access Control Dependencies**

This diagram shows the hierarchical relationships between control CSRs and their effects on timer-related CSRs. The `mcounteren.TM` bit is the top-level gate, while `menvcfg.STCE` specifically enables SSTC functionality. The `henvcfg.STCE` bit is hardwired to zero when `menvcfg.STCE=0`, demonstrating WARL (Write Any value, Read Legal value) behavior.

## Test Scenarios

### SID_SSTC_01: Access Blocked When mcounteren.TM=0

**Test ID**: `1`

**Function**: `SID_SSTC_01()`

Tests that all timer-related CSRs are blocked when `mcounteren.TM=0`, regardless of `hcounteren.TM` and `scounteren.TM` settings.

| Configuration | Expected Behavior |
| --- | --- |
| `mcounteren.TM=0` | All accesses to `time`, `stimecmp`, `vstimecmp` raise `ILLEGAL_INSTRUCTION` |
| `mcounteren.TM=0`, `scounteren.TM=1` | Still raises `ILLEGAL_INSTRUCTION` |
| `mcounteren.TM=0`, `hcounteren.TM=1` | Still raises `ILLEGAL_INSTRUCTION` |
| `mcounteren.TM=0`, both set | Still raises `ILLEGAL_INSTRUCTION` |

**Test Environment**: `priv_modes=[S, U]`, `hypervisor=[True, False]`, `virtualized=[True, False]`

**Key Steps**:

1.   `CsrWrite(csr_name="mcounteren", clear_mask=0x2)` - Clear TM bit
2.   `CsrRead(csr_name="time", direct_read=True)` wrapped in `AssertException`
3.   Repeat for `stimecmp` and `vstimecmp`
4.   Set `scounteren.TM=1` and `hcounteren.TM=1` in various combinations
5.   Verify exceptions still occur

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 11-101](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L11-L101)

### SID_SSTC_02: Access Control with mcounteren.TM=1, hcounteren.TM=0

This scenario is split into three variants based on privilege mode:

#### SID_SSTC_02_M_HS (Test ID: 2)

**Function**: `SID_SSTC_02_M_HS()`

**Environment**: `priv_modes=[M, S]`, `hypervisor=[True]`

Tests that M-mode and HS-mode can access `stimecmp` and `time` even when `hcounteren.TM=0`.

**Key Steps**:

1.   `CsrWrite(csr_name="mcounteren", set_mask=0x2)` - Enable TM
2.   `CsrWrite(csr_name="hcounteren", clear_mask=0x2)` - Clear hypervisor TM
3.   `CsrRead(csr_name="stimecmp")` and `CsrRead(csr_name="time")` - Should succeed
4.   Set `scounteren.TM=1`
5.   Repeat reads - Should still succeed

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 104-144](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L104-L144)

#### SID_SSTC_02_HU (Test ID: 3)

**Function**: `SID_SSTC_02_HU()`

**Environment**: `priv_modes=[U]`, `hypervisor=[True]`

Tests that HU-mode (hypervisor user mode) accesses are blocked when `hcounteren.TM=0`.

**Key Assertion**: All CSR reads wrapped in `AssertException(cause=ExceptionCause.ILLEGAL_INSTRUCTION)`.

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 147-193](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L147-L193)

**Function**: `SID_SSTC_02_V()`

**Environment**: `priv_modes=[S, U]`, `virtualized=[True]`

Tests that VS-mode and VU-mode accesses are blocked when `hcounteren.TM=0`.

**Key Assertion**: All CSR reads wrapped in `AssertException(cause=ExceptionCause.ILLEGAL_INSTRUCTION)` for virtualized modes.

### SID_SSTC_04: menvcfg.STCE Control

#### SID_SSTC_04_NON_M (Test ID: 7)

**Function**: `SID_SSTC_04_NON_M()`

### SID_SSTC_06: henvcfg.STCE WARL Behavior

**Test ID**: `11`

**Function**: `SID_SSTC_06()`

**Environment**: `TestEnvCfg()` (default)

Tests that `henvcfg.STCE` is read-only zero when `menvcfg.STCE=0`, demonstrating WARL (Write Any, Read Legal) behavior.

**Test Steps**:

1.   `CsrWrite(csr_name="menvcfg", clear_mask=(1 << 63))` - Ensure STCE=0
2.   `CsrWrite(csr_name="henvcfg", set_mask=(1 << 63))` - Attempt to set henvcfg.STCE
3.   `CsrRead(csr_name="henvcfg")` - Read back value
4.   `Arithmetic(op="and", src1=henvcfg_read, src2=(1<<63))` - Extract STCE bit
5.   `AssertEqual(src1=henvcfg_masked, src2=0)` - Verify it's zero
6.   Attempt to clear (should remain zero)
7.   Repeat assertion

This demonstrates that writes to `henvcfg.STCE` are ignored when `menvcfg.STCE=0`.

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 444-488](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L444-L488)

### SID_SSTC_08: mip.STIP Read-Only Behavior

**Test ID**: `12`

**Function**: `SID_SSTC_08()`

**Environment**: `priv_modes=[M]`

Tests that `mip.STIP` (bit 5) becomes read-only when `menvcfg.STCE=1`, as timer interrupts are now controlled by the `stimecmp` comparison hardware.

**Test Steps**:

1.   `CsrWrite(csr_name="menvcfg", set_mask=(1 << 63))` - Enable STCE
2.   `CsrRead(csr_name="mip")` and `Arithmetic(op="andi", src1=mip, src2=(1<<5))` - Read initial STIP
3.   `CsrWrite(csr_name="mip", set_mask=(1 << 5))` - Attempt to set STIP
4.   Read mip again and verify STIP unchanged
5.   `CsrWrite(csr_name="mip", clear_mask=(1 << 5))` - Attempt to clear STIP
6.   Read mip again and verify STIP unchanged
7.   `AssertEqual` verifies all three reads return the same value

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 497-551](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L497-L551)

## Test Step Usage Patterns

### CSR Write Operations

The SSTC scenarios heavily use `CsrWrite` with mask operations:

**Diagram: CsrWrite Usage Patterns in SSTC Tests**

| Parameter | Purpose | Example from SSTC |
| --- | --- | --- |
| `set_mask` | Set specific bits to 1 | `CsrWrite(csr_name="mcounteren", set_mask=0x2)` |
| `clear_mask` | Clear specific bits to 0 | `CsrWrite(csr_name="hcounteren", clear_mask=0x2)` |
| `value` | Write complete value | Not used in SSTC scenarios |
| `direct_write` | Skip privilege transition | Defaults to `False` in SSTC tests |

**Sources**: [coretp/step/csr.py 12-42](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/csr.py#L12-L42)[coretp/plans/sstc/sstc_scenarios.py 19-514](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L19-L514)

### CSR Read and Assertion Patterns

SSTC tests use two main assertion patterns:

#### Exception Assertion Pattern

Used to verify access violations:

**Example**:

`time_read = CsrRead(csr_name="time", direct_read=True)assert_exception = AssertException(    cause=ExceptionCause.ILLEGAL_INSTRUCTION,     code=[time_read])`
**Sources**: [coretp/plans/sstc/sstc_scenarios.py 21-23](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L21-L23)

#### Value Equality Pattern

Used to verify CSR values (e.g., WARL behavior):

**Example from SID_SSTC_06**:

`henvcfg_read = CsrRead(csr_name="henvcfg")top = LoadImmediateStep(imm=(1 << 63))henvcfg_masked = Arithmetic(op="and", src1=henvcfg_read, src2=top)zero = LoadImmediateStep(imm=0)assert_equal = AssertEqual(src1=henvcfg_masked, src2=zero)`
**Sources**: [coretp/plans/sstc/sstc_scenarios.py 457-461](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L457-L461)

## Unimplemented Scenarios

Several scenario IDs (SID_SSTC_07, SID_SSTC_09 through SID_SSTC_15) are commented out in the source with the note "Unable to be implemented" and "Requires interrupt handler to be injected". These scenarios would test actual timer interrupt generation and handling, which requires:

1.   Setting up interrupt handlers in M/S mode
2.   Configuring `medeleg`/`mideleg` for interrupt delegation
3.   Waiting for timer interrupts to fire when `time >= stimecmp`
4.   Verifying interrupt delivery to the correct privilege level

These scenarios are beyond the current scope of the test step framework.

**Sources**: [coretp/plans/sstc/sstc_scenarios.py 491-600](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/sstc/sstc_scenarios.py#L491-L600)

## Summary

The SSTC test plan provides 12 implemented scenarios (test IDs 1-12, with 7 skipped) that comprehensively verify:

*   **Access Control**: CSR access permissions across privilege modes based on counter enable bits
*   **Environment Configuration**: SSTC enablement through `menvcfg.STCE` and `henvcfg.STCE`
*   **WARL Behavior**: Read-only behavior of `henvcfg.STCE` when `menvcfg.STCE=0`
*   **Interrupt Pending**: Read-only behavior of `mip.STIP` when SSTC is enabled
*   **Multi-mode Coverage**: Testing across M, S, U, HS, VS, VU modes with and without H extension

The scenarios use composition of `CsrWrite`, `CsrRead`, `AssertException`, `AssertEqual`, `Arithmetic`, and `LoadImmediateStep` test steps to verify specification compliance.
