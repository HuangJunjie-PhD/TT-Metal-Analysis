This page documents the model architectures supported by vLLM, the model registry system that manages them, and how models are detected, loaded, and validated. For information about installing vLLM and using models, see [Installation Guide](https://deepwiki.com/tenstorrent/vllm/1.1-installation-guide) and [Quick Start Guide](https://deepwiki.com/tenstorrent/vllm/1.3-quick-start-guide). For details on loading models from different sources, see the documentation in [docs/models/supported_models.md](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1)

## Model Categories

vLLM supports several categories of models, each serving different inference tasks. The model registry organizes these into distinct dictionaries that map HuggingFace architecture names to vLLM implementations.

**Text Generation Models** are decoder-only language models (e.g., LLaMA, Qwen, Mistral) that generate text autoregressively. The dictionary `_TEXT_GENERATION_MODELS` in [vllm/model_executor/models/registry.py 45-156](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L45-L156) contains over 100 architecture mappings like `"LlamaForCausalLM": ("llama", "LlamaForCausalLM")`.

**Embedding Models** produce vector representations for pooling tasks. The `_EMBEDDING_MODELS` dictionary [vllm/model_executor/models/registry.py 158-198](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L158-L198) includes text encoders (e.g., BERT, RoBERTa) and multimodal embedding models (e.g., CLIP-based).

**Multimodal Models** process vision, audio, or video inputs alongside text. The `_MULTIMODAL_MODELS` dictionary [vllm/model_executor/models/registry.py 215-280](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L215-L280) includes vision-language models like LLaVA, Qwen2-VL, and audio models like Ultravox.

**Speculative Decoding Models** are specialized architectures for draft model generation in speculative decoding. These are defined in `_SPECULATIVE_DECODING_MODELS`[vllm/model_executor/models/registry.py 282-300](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L282-L300) and include EAGLE, Medusa, and MTP variants.

**Transformers Backend Models** use HuggingFace Transformers implementations directly rather than vLLM-native code. This enables support for models not yet implemented in vLLM. See [docs/models/supported_models.md 16-156](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1#L16-L156) for details.

Sources: [vllm/model_executor/models/registry.py 45-330](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L45-L330)[docs/models/supported_models.md 1-50](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1#L1-L50)

## Model Registry Architecture

The model registry is implemented through the `_ModelRegistry` dataclass and related classes that handle model discovery, validation, and loading.

The `_ModelRegistry` class [vllm/model_executor/models/registry.py 560-756](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L560-L756) maintains a mapping of architecture names to model implementations. Each entry can be either:

1.   **_LazyRegisteredModel**[vllm/model_executor/models/registry.py 432-530](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L432-L530): Stores only the module path and class name. The actual model class is loaded on-demand to avoid initializing CUDA during import.

2.   **_RegisteredModel**[vllm/model_executor/models/registry.py 409-430](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L409-L430): Contains the already-loaded model class and its interface information.

Both types implement the `_BaseRegisteredModel` abstract class [vllm/model_executor/models/registry.py 398-407](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L398-L407) which requires `inspect_model_cls()` and `load_model_cls()` methods.

The `_ModelInfo` dataclass [vllm/model_executor/models/registry.py 354-396](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L354-L396) caches model capabilities:

*   `is_text_generation_model`: Whether it's a generative model
*   `supports_multimodal`: Whether it accepts multimodal inputs
*   `supports_pp`: Whether it supports pipeline parallelism
*   `is_attention_free`: Whether it uses attention-free architectures (e.g., Mamba)
*   `supports_v0_only`: Whether it only works with V0 engine

For lazy models, the inspection is performed in a subprocess [vllm/model_executor/models/registry.py 515-517](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L515-L517) to avoid CUDA initialization. The results are cached to disk [vllm/model_executor/models/registry.py 441-524](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L441-L524) using MD5 hashing of the model file to avoid redundant inspection.

Sources: [vllm/model_executor/models/registry.py 354-756](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L354-L756)[vllm/model_executor/models/registry.py 432-530](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L432-L530)

## Model Loading Flow

The process of loading a model from a HuggingFace repository or local path involves configuration parsing, architecture detection, and model initialization.

**Configuration Parsing** starts with detecting the config format. The `get_config_parser()` function [vllm/transformers_utils/config.py 211-216](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L211-L216) returns either `HFConfigParser` or `MistralConfigParser`:

*   **HFConfigParser**[vllm/transformers_utils/config.py 103-161](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L103-L161) loads standard HuggingFace `config.json` files. It checks if the model type is in `_CONFIG_REGISTRY`[vllm/transformers_utils/config.py 66-87](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L66-L87) which maps custom model types (e.g., `"chatglm"`, `"deepseek_v3"`) to vLLM-specific config classes in [vllm/transformers_utils/configs/](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/configs/)

*   **MistralConfigParser**[vllm/transformers_utils/config.py 163-197](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L163-L197) handles Mistral's proprietary format which uses `params.json` instead of `config.json`. It downloads the config via `_download_mistral_config_file()` and converts it using `adapt_config_dict()`.

**Architecture Detection** extracts the model architecture from the config's `"architectures"` field. For example, a Llama model config contains `"architectures": ["LlamaForCausalLM"]`.

**Registry Lookup** searches `_VLLM_MODELS`[vllm/model_executor/models/registry.py 321-329](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L321-L329) which combines all category dictionaries. The architecture name maps to a tuple `(module_name, class_name)`, e.g., `"LlamaForCausalLM": ("llama", "LlamaForCausalLM")`.

**Model Class Loading** happens via `_try_load_model_cls()`[vllm/model_executor/models/registry.py 532-544](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L532-L544) which:

1.   Verifies the architecture with `current_platform.verify_model_arch()`
2.   Calls `load_model_cls()` on the registered model
3.   For lazy models, this triggers `importlib.import_module()`[vllm/model_executor/models/registry.py 527-529](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L527-L529)

**Model Inspection** via `_try_inspect_model_cls()`[vllm/model_executor/models/registry.py 547-557](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L547-L557) returns `_ModelInfo` containing model capabilities. For lazy models, this may use cached results or run inspection in a subprocess to avoid CUDA initialization.

Sources: [vllm/transformers_utils/config.py 103-261](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L103-L261)[vllm/model_executor/models/registry.py 532-557](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L532-L557)[vllm/transformers_utils/config.py 211-216](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L211-L216)

## Model Architecture Support

### Text Generation Models

Text generation models are decoder-only architectures that generate text autoregressively. The `_TEXT_GENERATION_MODELS` dictionary [vllm/model_executor/models/registry.py 45-156](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L45-L156) contains approximately 100+ supported architectures.

| Architecture Family | Example Models | vLLM Module | Key Features |
| --- | --- | --- | --- |
| Llama Family | LlamaForCausalLM, MistralForCausalLM | `llama` | Base for many models, RoPE, GQA |
| Qwen Family | Qwen2ForCausalLM, Qwen3ForCausalLM | `qwen2`, `qwen3` | Multilingua, long context |
| DeepSeek | DeepseekV2ForCausalLM, DeepseekV3ForCausalLM | `deepseek_v2` | MLA, MoE variants |
| Mixtral | MixtralForCausalLM | `mixtral` | Mixture of Experts |
| Gemma | GemmaForCausalLM, Gemma2ForCausalLM | `gemma`, `gemma2` | Small efficient models |
| GPT-family | GPT2LMHeadModel, GPTNeoXForCausalLM | `gpt2`, `gpt_neox` | Classic architectures |
| Mamba | MambaForCausalLM, Mamba2ForCausalLM | `mamba`, `mamba2` | Attention-free SSMs |

Common pattern in the registry: `"ArchitectureName": ("module_name", "ClassName")`

For example:

`"LlamaForCausalLM": ("llama", "LlamaForCausalLM"),"MixtralForCausalLM": ("mixtral", "MixtralForCausalLM"),"DeepseekV3ForCausalLM": ("deepseek_v2", "DeepseekV3ForCausalLM"),`
Many architectures map to the same implementation. For instance, both `MistralForCausalLM` and `LlamaForCausalLM` use the `llama` module because Mistral is architecturally similar to Llama [vllm/model_executor/models/registry.py 121](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L121-L121)

**Attention-Free Architectures**: Models like Mamba [vllm/model_executor/models/registry.py 115-118](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L115-L118) use state-space models instead of attention. These are detected via the `is_attention_free()` interface method [vllm/model_executor/models/interfaces.py 104](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces.py#L104-L104)

**Hybrid Architectures**: Some models like Jamba combine attention and SSM layers [vllm/model_executor/models/registry.py 108](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L108-L108) These are marked with the `is_hybrid()` interface.

Sources: [vllm/model_executor/models/registry.py 45-156](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L45-L156)[tests/models/registry.py 165-359](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/registry.py#L165-L359)

### Embedding and Pooling Models

Embedding models produce fixed-size vector representations from variable-length input. The `_EMBEDDING_MODELS` dictionary [vllm/model_executor/models/registry.py 158-198](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L158-L198) includes both text-only and multimodal embedding architectures.

| Architecture | Task | Module | Default Pooling |
| --- | --- | --- | --- |
| BertModel | Text embedding | `bert` | CLS token |
| RobertaModel | Text embedding | `roberta` | CLS token |
| GteNewModel | Text embedding | `bert_with_rope` | MEAN pooling |
| LlamaModel | Embedding via base LLM | `llama` | LAST token |
| Phi3VForCausalLM | Multimodal embedding | `phi3v` | LAST token |
| Qwen2VLForConditionalGeneration | Multimodal embedding | `qwen2_vl` | LAST token |

**Text Embedding Models** use encoder-only or decoder-only architectures. BERT-style models [vllm/model_executor/models/registry.py 160](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L160-L160) typically use CLS token pooling, while LLaMA-based models [vllm/model_executor/models/registry.py 171-176](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L171-L176) use last token pooling.

**Multimodal Embedding Models**[vllm/model_executor/models/registry.py 189-197](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L189-L197) can produce embeddings from images and text. For example, `Phi3VForCausalLM` can serve as both a generative VLM and an embedding model.

**Pooling Type Detection**: The `get_default_pooling_type()` function [vllm/model_executor/models/interfaces_base.py 42-73](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces_base.py#L42-L73) determines the default pooling strategy from the model class. This is cached in `_ModelInfo.default_pooling_type`[vllm/model_executor/models/registry.py 359](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L359-L359)

**Geospatial Models**: Specialized models like `PrithviGeoSpatialMAE` and `Terratorch`[vllm/model_executor/models/registry.py 196-197](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L196-L197) process satellite imagery and produce embeddings. These require `skip_tokenizer_init=True`[tests/models/registry.py 399](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/registry.py#L399-L399) since they don't use text tokens.

Sources: [vllm/model_executor/models/registry.py 158-198](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L158-L198)[vllm/model_executor/models/interfaces_base.py 42-73](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces_base.py#L42-L73)

### Cross-Encoder and Classification Models

Cross-encoder models score pairs of inputs for tasks like reranking. These are defined in `_CROSS_ENCODER_MODELS`[vllm/model_executor/models/registry.py 200-213](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L200-L213)

**Native Cross-Encoders** like `BertForSequenceClassification`[vllm/model_executor/models/registry.py 201](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L201-L201) are specifically designed for classification. They typically output a single score or class probabilities.

**Automatic Conversion**: Models in `_AUTOMATIC_CONVERTED_MODELS`[tests/models/registry.py 429-438](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/registry.py#L429-L438) are base generative models converted to sequence classification via the `as_seq_cls_model` mechanism. This requires special `hf_overrides`[tests/models/registry.py 432-434](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/registry.py#L432-L434):

`"hf_overrides": {    "architectures": ["GemmaForSequenceClassification"],    "classifier_from_token": ["Yes"],    "method": "no_post_processing"}`
**Multimodal Cross-Encoders** like `JinaVLForRanking`[vllm/model_executor/models/registry.py 212](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L212-L212) can rank image-text pairs, enabling vision-language reranking tasks.

Sources: [vllm/model_executor/models/registry.py 200-213](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L200-L213)[tests/models/registry.py 413-438](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/registry.py#L413-L438)

### Multimodal Models

Multimodal models process vision, audio, or video inputs. The `_MULTIMODAL_MODELS` dictionary [vllm/model_executor/models/registry.py 215-280](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L215-L280) contains over 60 architectures spanning multiple modalities.

**Vision-Language Models** are the largest category. Key implementations:

*   **LLaVA family**[vllm/model_executor/models/registry.py 245-248](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L245-L248): Includes base LLaVA, LLaVA-NeXT, and LLaVA-OneVision. These use CLIP-style vision encoders with LLaMA-based language models.

*   **Qwen VL family**[vllm/model_executor/models/registry.py 264-271](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L264-L271): Qwen2-VL and Qwen2.5-VL support both images and videos. Qwen3-VL adds MoE variants.

*   **InternVL**[vllm/model_executor/models/registry.py 233-236](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L233-L236): Uses InternViT vision encoder with various language backbones.

*   **Llama4**[vllm/model_executor/models/registry.py 244](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L244-L244): Meta's multimodal Llama supporting images with cross-attention.

**Audio-Language Models** process spoken audio:

*   **Qwen2-Audio**[vllm/model_executor/models/registry.py 267](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L267-L267): Supports both speech and audio understanding.
*   **Ultravox**[vllm/model_executor/models/registry.py 276](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L276-L276): Speech-to-text model for voice interactions.
*   **Voxtral**[vllm/model_executor/models/registry.py 277](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L277-L277): Mistral's audio model (currently disabled in HF format).

**Video-Language Models** handle temporal sequences:

*   **LLaVA-NeXT-Video**[vllm/model_executor/models/registry.py 247](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L247-L247): Extends LLaVA for video understanding.
*   **GLM-4.1V and GLM-4.5V**[vllm/model_executor/models/registry.py 229-230](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L229-L230): Support both image and video inputs with metadata [tests/models/multimodal/processing/test_common.py 30-46](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/models/multimodal/processing/test_common.py#L30-L46)

**Encoder-Decoder Models**: Only `WhisperForConditionalGeneration`[vllm/model_executor/models/registry.py 279](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L279-L279) is currently supported as an encoder-decoder for transcription tasks. It's marked with `supports_transcription=True`[vllm/model_executor/models/interfaces.py 131](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces.py#L131-L131)

### Model Architecture Detection

Architecture detection determines which vLLM implementation to use for a given model.

**Architecture Resolution**[vllm/config.py 390-422](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/config.py#L390-L422) (referenced via `try_match_architecture_defaults`):

1.   Load config via `get_config()` which calls parser's `parse()` method
2.   Extract `architectures` list from config
3.   Try first architecture against `_VLLM_MODELS`
4.   If not found and `model_impl="auto"`, try Transformers backend
5.   If still not found, raise error with supported architectures

**Aliasing Support**[vllm/model_executor/models/registry.py 658-699](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L658-L699):

Some architectures have aliases:

`# "AquilaModel" and "AquilaForCausalLM" both map to llama"AquilaModel": ("llama", "LlamaForCausalLM"),"AquilaForCausalLM": ("llama", "LlamaForCausalLM"),`
**Platform Verification**[vllm/model_executor/models/registry.py 537-538](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L537-L538):

Before loading, `current_platform.verify_model_arch()` checks if the model is compatible with the current platform (CUDA, ROCm, CPU, TPU, etc.). This can raise errors for platform-specific models.

**Default Architecture Matching**[vllm/config.py 299-338](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/config.py#L299-L338) (via `iter_architecture_defaults`):

vLLM maintains default configurations for each architecture. When loading a model, it tries to match the architecture name against defaults and applies them:

`for name, default_config in iter_architecture_defaults():    if name == model_arch:        # Apply defaults        break`

### Model Interface Detection

Model capabilities are detected via interface protocols defined in [vllm/model_executor/models/interfaces.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces.py) and [vllm/model_executor/models/interfaces_base.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/interfaces_base.py)

**Core Interfaces**:

| Interface | Detection Method | Purpose |
| --- | --- | --- |
| `VllmModelForTextGeneration` | `is_text_generation_model()` | Generative models |
| `VllmModelForPooling` | `is_pooling_model()` | Embedding/classification |
| `SupportsMultiModal` | `supports_multimodal()` | Multimodal input support |
| `SupportsPP` | `supports_pp()` | Pipeline parallelism |
| `SupportsLoRA` | `supports_lora()` | LoRA adapter support |
