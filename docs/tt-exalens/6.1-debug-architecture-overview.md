This page describes the RISC-V core debug infrastructure in TTExaLens: the abstract interface, the concrete hardware implementation, how cores are identified, and how platform-specific behavior is layered on top.

For execution control operations (halt, step, continue, reset), see [Execution Control](https://deepwiki.com/tenstorrent/tt-exalens/6.2-execution-control). For memory and GPR access APIs, see [Memory and Register Access](https://deepwiki.com/tenstorrent/tt-exalens/6.3-memory-and-register-access). For watchpoint configuration, see [Watchpoints and Breakpoints](https://deepwiki.com/tenstorrent/tt-exalens/6.4-watchpoints-and-breakpoints). For platform-specific quirks and workarounds, see [Platform-Specific Debug Implementations](https://deepwiki.com/tenstorrent/tt-exalens/6.5-platform-specific-debug-implementations).

## Core Abstraction: `RiscDebug`

The entry point for all RISC-V core debugging is the abstract class `RiscDebug`, defined in [ttexalens/hardware/risc_debug.py 93-535](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L93-L535) Every debuggable RISC-V core—regardless of architecture—implements this interface.

The interface covers six functional areas:

| Area | Key Methods |
| --- | --- |
| Reset control | `is_in_reset`, `set_reset_signal` |
| Execution control | `halt`, `step`, `cont`, `is_halted`, `is_ebreak_hit`, `ensure_halted` |
| Register access | `read_gpr`, `write_gpr`, `get_pc` |
| Memory access | `read_memory`, `write_memory`, `read_memory_bytes`, `write_memory_bytes` |
| Watchpoints | `set_watchpoint_on_pc_address`, `set_watchpoint_on_memory_read/write/access`, `disable_watchpoint`, `read_watchpoints_state`, `read_watchpoint_address` |
| Callstack | `get_callstack` (implemented in base class using DWARF CFI) |

There is also a static factory method `RiscDebug.get_instance(risc_location)`[ttexalens/hardware/risc_debug.py 103-106](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L103-L106) that resolves the correct concrete instance by looking up the `NocBlock` for the given location and calling its `get_risc_debug` method.

Sources: [ttexalens/hardware/risc_debug.py 93-310](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L93-L310)

## Core Identity: `RiscLocation`

`RiscLocation`[ttexalens/hardware/risc_debug.py 25-35](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L25-L35) is a lightweight dataclass that uniquely identifies a single RISC-V core on a device:

| Field | Type | Description |
| --- | --- | --- |
| `location` | `OnChipCoordinate` | On-chip tile coordinate |
| `neo_id` | `int | None` | NEO sub-block index (Quasar only; `None` for WH/BH) |
| `risc_name` | `str` | Core name, e.g., `"brisc"`, `"trisc0"`, `"erisc"` |

`RiscLocation` is hashable and is used as a key when indexing debug instances. `RiscDebug.get_instance` takes a `RiscLocation` and returns the corresponding `RiscDebug`.

Sources: [ttexalens/hardware/risc_debug.py 25-35](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L25-L35)

## Core Configuration: `BabyRiscInfo`

`BabyRiscInfo`[ttexalens/hardware/baby_risc_info.py 12-98](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_info.py#L12-L98) is a dataclass that holds all static configuration for a single BabyRISC core. It is constructed once per core (typically inside a `NocBlock`) and passed to `BabyRiscDebug`.

| Field | Type | Purpose |
| --- | --- | --- |
| `risc_name` | `str` | Canonical lowercase name (`"brisc"`, `"trisc0"`, etc.) |
| `risc_id` | `int` | Numeric index used in hardware register encoding |
| `noc_block` | `NocBlock` | Owning block, provides `location` and `register_store` |
| `neo_id` | `int | None` | NEO sub-index (Quasar) |
| `l1` | `MemoryBlock` | L1 shared memory region |
| `max_watchpoints` | `int` | Number of hardware watchpoints |
| `reset_flag_shift` | `int` | Bit position of this core's reset bit in `RISCV_DEBUG_REG_SOFT_RESET_0` |
| `branch_prediction_register` | `str | None` | Register name to disable branch prediction (WH workaround) |
| `branch_prediction_mask` | `int | None` | Bitmask for the branch prediction field |
| `default_code_start_address` | `int | None` | Hard-coded reset vector (if not register-controlled) |
| `code_start_address_register` | `str | None` | Register holding the programmable reset PC |
| `code_start_address_enable_register` | `str | None` | Register that enables the override |
| `code_start_address_enable_bit` | `int | None` | Enable bit mask |
| `data_private_memory` | `MemoryBlock | None` | Core-private data SRAM (if present) |
| `code_private_memory` | `MemoryBlock | None` | Core-private instruction SRAM (older arch) |
| `debug_hardware_present` | `bool` | Whether hardware debug registers are wired up |

Key methods on `BabyRiscInfo`:

*   `get_code_start_address(register_store)` — reads the active reset PC (from register or default).
*   `set_code_start_address(register_store, address)` — programs the reset PC register, or writes a `JAL` stub if the core lacks a configurable register.
*   `translate_to_noc_address(addr)` — maps a private or L1 address to its NOC address.

Sources: [ttexalens/hardware/baby_risc_info.py 12-98](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_info.py#L12-L98)

## Hardware Protocol: `BabyRiscDebugHardware`

`BabyRiscDebugHardware`[ttexalens/hardware/baby_risc_debug.py 218-465](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L218-L465) implements the raw hardware handshake protocol for talking to the debug hardware registers on a BabyRISC core. It is not a `RiscDebug` subclass; it is a helper layer owned by `BabyRiscDebug`.

### Register Layout

Four NOC-accessible registers drive the debug interface:

| Constant | Index | Role |
| --- | --- | --- |
| `RISC_DBG_CNTL0` | — | Command trigger; selects read/write and target register address |
| `RISC_DBG_CNTL1` | — | Write data register |
| `RISC_DBG_STATUS0` | — | Contains the read-valid flag |
| `RISC_DBG_STATUS1` | — | Read data register |

The control value in `CNTL0` encodes direction and target:

*   Write: `0x80010000 + (risc_id << 17) + reg_addr`
*   Read: `0x80000000 + (risc_id << 17) + reg_addr`

### Internal Register Addresses

These are not NOC addresses — they are internal debug register indices passed through the CNTL0 protocol:

| Constant | Value | Purpose |
| --- | --- | --- |
| `REG_STATUS` | 0 | Read halt/watchpoint status |
| `REG_COMMAND` | 1 | Issue halt/step/continue/read/write commands |
| `REG_COMMAND_ARG_0` | 2 | First argument (address or register index) |
| `REG_COMMAND_ARG_1` | 3 | Second argument (write value) |
| `REG_COMMAND_RETURN_VALUE` | 4 | Result of read operations |
| `REG_HW_WATCHPOINT_SETTINGS` | 5 | Packed watchpoint enable/mode bits |
| `REG_HW_WATCHPOINT_0`–`_7` | 10–17 | Watchpoint address registers |

### Command Bits

Commands written to `REG_COMMAND` always include `COMMAND_DEBUG_MODE` (bit 31) combined with an operation flag:

| Constant | Bit | Meaning |
| --- | --- | --- |
| `COMMAND_DEBUG_MODE` | `0x80000000` | Must be set for all debug operations |
| `COMMAND_HALT` | `0x00000001` | Halt the core |
| `COMMAND_STEP` | `0x00000002` | Execute one instruction |
| `COMMAND_CONTINUE` | `0x00000004` | Resume execution |
| `COMMAND_READ_REGISTER` | `0x00000008` | Read a GPR |
| `COMMAND_WRITE_REGISTER` | `0x00000010` | Write a GPR |
| `COMMAND_READ_MEMORY` | `0x00000020` | Read a 32-bit word from memory |
| `COMMAND_WRITE_MEMORY` | `0x00000040` | Write a 32-bit word to memory |
| `COMMAND_FLUSH` | `0x00000100` | Flush pipeline and reset PC (Quasar) |

Sources: [ttexalens/hardware/baby_risc_debug.py 26-67](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L26-L67)[ttexalens/hardware/baby_risc_debug.py 218-465](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L218-L465)

## Obtaining a `RiscDebug` Instance

**Figure: Instance Resolution Path**

The two concrete paths are:

1.   **Wormhole / Blackhole**: `NocBlock.get_risc_debug` directly instantiates `WormholeBabyRiscDebug` or `BlackholeBabyRiscDebug`.
2.   **Quasar**: `NocBlock.get_risc_debug` delegates to a `QuasarFunctionalNeoBlock` (indexed by `neo_id`), which instantiates `QuasarBabyRiscDebug`.

Sources: [ttexalens/hardware/risc_debug.py 103-106](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/risc_debug.py#L103-L106)[ttexalens/hardware/quasar/functional_neo_block.py 188-199](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/functional_neo_block.py#L188-L199)

## Quasar NEO Structure

On Quasar, each functional worker block contains multiple NEO sub-blocks. Each NEO is represented by a `QuasarFunctionalNeoBlock`[ttexalens/hardware/quasar/functional_neo_block.py 58-199](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/functional_neo_block.py#L58-L199) which bundles:

*   A `RegisterStore` initialized from `register_map` with a NEO-relative base address.
*   A `DebugBusSignalStore` initialized from `debug_bus_signal_map`.
*   Four `BabyRiscInfo` instances: `trisc0`, `trisc1`, `trisc2`, `trisc3`.

Each TRISC in a NEO has separate `reset_flag_shift` values (11, 12, 13, 14 respectively), separate data-private memory regions, and separate reset PC override registers. The `neo_id` is used to distinguish between NEOs on the same tile.

Sources: [ttexalens/hardware/quasar/functional_neo_block.py 58-199](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/functional_neo_block.py#L58-L199)

## Memory Access via Debug Hardware

`BabyRiscDebug` provides unaligned memory access by building on the word-aligned hardware protocol. `read_memory_bytes` and `write_memory_bytes`[ttexalens/hardware/baby_risc_debug.py 703-760](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L703-L760) work as follows:

1.   Compute the word-aligned start and end addresses covering `<FileRef file-url="https://github.com/tenstorrent/tt-exalens/blob/046c35eb/address, address + size_bytes)`.\n2. Read all overlapping 4-byte words via `BabyRiscDebugHardware.read_memory`.\n3. For writes, reconstruct modified words by merging new bytes with existing words at the boundaries.\n4. Write all modified words back via `BabyRiscDebugHardware.write_memory`.\n\nAll memory operations through the debug hardware require the core to be halted. The `ensure_halted` context manager [ttexalens/hardware/baby_risc_debug.py#L323-L335" min=323 max=335 file-path="address, address + size_bytes)`.\n2. Read all overlapping 4-byte words via`BabyRiscDebugHardware.read_memory`.\n3. For writes, reconstruct modified words by merging new bytes with existing words at the boundaries.\n4. Write all modified words back via`BabyRiscDebugHardware.write_memory`.\n\nAll memory operations through the debug hardware require the core to be halted. The`ensure_halted`context manager [ttexalens/hardware/baby_risc_debug.py">Hii</FileRef> on`BabyRiscDebugHardware` halts the core if needed, performs the operation, and resumes if it halted it.

Access to private (core-local SRAM) memory when the core is in reset requires a special path via `ensure_private_memory_access`[ttexalens/hardware/baby_risc_debug.py 573-600](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L573-L600): a minimal `JAL x0, 0` loop is written to the reset vector, the core is taken out of reset, halted, and then the desired access is performed before resetting again.
