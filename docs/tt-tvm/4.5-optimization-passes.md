## Purpose and Scope

This page documents the major optimization passes in Relay IR that transform and optimize the computation graph. These passes operate at the graph level on Relay IRModule to improve performance, enable hardware-specific optimizations, and prepare code for lowering to TensorIR.

For information about the Relay operator library itself, see [Neural Network Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.1-neural-network-operators) and [Tensor Transformation Operators](https://deepwiki.com/tenstorrent/tt-tvm/4.2-tensor-transformation-operators). For layout transformation infrastructure, see [Layout Transformation System](https://deepwiki.com/tenstorrent/tt-tvm/4.4-layout-transformation-system). For operator implementation strategies, see [Strategy System](https://deepwiki.com/tenstorrent/tt-tvm/4.6-strategy-system).

## Legalize Pass

The Legalize pass converts operators into canonical forms or target-specific implementations. It queries the `FTVMLegalize` attribute registered on each operator to transform high-level operators into lower-level equivalents that are easier to schedule and optimize.

### Registration Pattern

Legalize functions are registered using the `@reg.register_legalize` decorator:

**Legalize Registration Examples**

### Common Legalize Implementations

| Operator | Legalize Function | TOPI Backend | Purpose |
| --- | --- | --- | --- |
| `nn.matmul` | `legalize_matmul` | `topi.nn.matmul_legalize` | Convert to batch_matmul or dense |
| `nn.dense` | `legalize_dense` | `topi.nn.dense_legalize` | Add layout transforms, pack weights |
| `nn.conv2d` | `legalize_conv2d` | `topi.nn.conv2d_legalize` | Layout conversion, weight packing |
| `nn.batch_matmul` | `legalize_batch_matmul` | `topi.nn.batch_matmul_legalize` | Optimize batch operations |
| `nn.conv2d_transpose` | `legalize_conv2d_transpose` | `topi.nn.conv2d_transpose_legalize` | Rewrite as upsampling + conv |
| `nn.conv3d_transpose` | `legalize_conv3d_transpose` | `topi.nn.conv3d_transpose_legalize` | 3D transpose convolution |
| `nn.deformable_conv2d` | `legalize_deformable_conv2d` | Returns None | No legalization needed |

## ConvertLayout Pass

The ConvertLayout pass forcibly converts operators to specified layouts by inserting layout transformation operators. This pass is more aggressive than AlterOpLayout and guarantees that the desired layout is achieved.

### ConvertLayout Mechanics

**ConvertLayout Insertion Logic**

### Registration and Implementation

ConvertLayout functions handle layout checking and transformation insertion:

`@reg.register_convert_op_layout("nn.conv2d")def convert_conv2d(attrs, inputs, tinfos, desired_layouts):    """Convert Layout pass registration for conv2d op.        Parameters    ----------    attrs : tvm.ir.Attrs        Attributes of current convolution    inputs : list of tvm.relay.Expr        The args of the Relay expr to be legalized    tinfos : list of types        List of input and output types    desired_layouts : list of layout strings        List of layouts defining our desired layout         for the data and kernel inputs respectively.            Returns    -------    result : tvm.relay.Expr        The transformed expr    """    data, weight = inputs    new_attrs = dict(attrs)        # Parse desired layouts    desired_data_layout, desired_kernel_layout = map(str, desired_layouts)    new_attrs["data_layout"] = desired_data_layout        # Handle default kernel layouts    if desired_data_layout == "NCHW":        new_attrs["kernel_layout"] = "OIHW"    elif desired_data_layout == "NHWC":        new_attrs["kernel_layout"] = "HWIO"            return relay.nn.conv2d(data, weight, **new_attrs)`

## FuseOps Pass

The FuseOps pass combines multiple operators into single fused kernels to reduce memory bandwidth and kernel launch overhead. This pass uses operator pattern tags (`TOpPattern`) to determine fusibility.

### Operator Pattern Categories

| Pattern | Value | Description | Examples |
| --- | --- | --- | --- |
| `kElemWise` | 0 | Element-wise operations | add, multiply, relu |
| `kBroadcast` | 1 | Broadcasting operations | broadcast_to, expand_dims |
| `kInjective` | 2 | One-to-one mapping | transpose, reshape |
| `kCommReduce` | 3 | Commutative reduction | sum, max, mean |
| `kOutEWiseFusable` | 4 | Output element-wise fusable | conv2d, dense |
| `kTuple` | 7 | Tuple operations | tuple, tuple_get_item |
| `kOpaque` | 8 | Opaque (not fusable) | sort, scan |

## Strategy System Integration

Optimization passes work closely with the strategy system to select hardware-specific implementations.

**Strategy System and Pass Interaction**

### Strategy Registration

Strategies are registered per-target and provide multiple implementation options:

`@conv2d_strategy.register(["cuda", "gpu"])def conv2d_strategy_cuda(attrs, inputs, out_type, target):    """conv2d cuda strategy"""    strategy = _op.OpStrategy()    data, kernel = inputs    layout = attrs.data_layout        if layout == "NCHW":        strategy.add_implementation(            wrap_compute_conv2d(topi.cuda.conv2d_nchw),            wrap_topi_schedule(topi.cuda.schedule_conv2d_nchw),            name="conv2d_nchw.cuda",        )        # Add winograd implementation        strategy.add_implementation(            wrap_compute_conv2d(topi.cuda.conv2d_nchw_winograd),            wrap_topi_schedule(topi.cuda.schedule_conv2d_nchw_winograd),            name="conv2d_nchw_winograd.cuda",            plevel=5,        )    return strategy`
Sources: [python/tvm/relay/op/strategy/cuda.py 138-207](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/python/tvm/relay/op/strategy/cuda.py#L138-L207)

### Pass Execution Order

Optimization passes typically execute in this order:

1.   **Legalize** - Convert to canonical forms
2.   **AlterOpLayout** - Suggest better layouts
3.   **ConvertLayout** - Force desired layouts
4.   **FuseOps** - Combine operators
5.   **FoldConstant** - Evaluate constants

The strategy system is queried during legalization and layout optimization to select target-specific implementations.
