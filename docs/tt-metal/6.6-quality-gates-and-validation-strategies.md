This document describes the multi-tier quality gate system that ensures code quality and correctness before changes are merged and released in the `tt-metal` repository. The system employs different validation strategies at different stages of the development lifecycle: Pull Request (PR) gate for fast feedback, Merge Gate for comprehensive validation, and post-commit/nightly pipelines for extended testing.

## Quality Gate Tiers

The repository implements a three-tier quality gate system with progressively increasing coverage and runtime requirements.

Title: Quality Gate Progression

**Sources:**[.github/workflows/pr-gate.yaml 1-5](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L1-L5)[.github/workflows/merge-gate.yaml 1-5](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/merge-gate.yaml#L1-L5)

### Design Constraints by Gate

| Gate | Max Duration | Individual Test Timeout | Build Type | Hardware Coverage |
| --- | --- | --- | --- | --- |
| **PR Gate** | 5 minutes | 1 second | ASan | N150 (minimal) |
| **Merge Gate** | 15 minutes | 5 seconds | TSan | N300, T3000, P150 (representative) |
| **Post-Commit** | Variable | Test-specific | Release, Debug, ASan, TSan, Profiler | All SKUs |

**Sources:**[.github/workflows/pr-gate.yaml 9-11](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L9-L11)[.github/workflows/merge-gate.yaml 7-9](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/merge-gate.yaml#L7-L9)

## PR Gate: Fast Validation

The PR gate provides rapid feedback on every commit to a pull request, emphasizing speed and early error detection through sanitizers and change detection.

### PR Gate Pipeline Architecture

Title: PR Gate Logic and Code Entities

**Key Characteristics:**

*   **AddressSanitizer (ASan)**: The default build type for the PR gate is `ASan`[[.github/workflows/pr-gate.yaml:132]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies), which detects memory errors like use-after-free and buffer overflows. It uses the Clang 20 toolchain [[.github/workflows/pr-gate.yaml:131]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Change Detection**: The action `find-changed-files` identifies which subsystems (Metalium, TTNN, LLK, etc.) have changed to optimize test selection [[.github/workflows/pr-gate.yaml:146-155]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Smoke Tests**: Validates core functionality on a single-chip platform to minimize hardware queue times. These are triggered based on file change detection outputs like `any-code-changed`[[.github/workflows/pr-gate.yaml:166]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

**Sources:**[.github/workflows/pr-gate.yaml 121-145](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L121-L145)[.github/workflows/pr-gate.yaml 146-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L146-L155)[.github/workflows/pr-gate.yaml 160-180](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L160-L180)

## Merge Gate: Comprehensive Validation

The merge gate is the final barrier before code lands on `main`. It runs comprehensive tests across multiple hardware platforms, often using ThreadSanitizer (TSan) to catch concurrency bugs.

### Merge Gate Pipeline

Title: Merge Gate System Components

**Sources:**[.github/workflows/merge-gate.yaml 56-62](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/merge-gate.yaml#L56-L62)[.github/workflows/merge-gate.yaml 137-153](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/merge-gate.yaml#L137-L153)

### ThreadSanitizer (TSan) Build

The merge gate utilizes a TSan build on Ubuntu 24.04 [[.github/workflows/merge-gate.yaml:151-152]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies). TSan is critical for identifying data races in the fast dispatch and multi-device command queue systems.

### Fail-Fast Strategy

To conserve CI resources, a `fail-fast` strategy is applied across workflows. For instance, in the PR Gate, the `concurrency` group is configured with `cancel-in-progress: true`[[.github/workflows/pr-gate.yaml:59-72]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies) to abort obsolete runs when new commits are pushed. Similarly, the Merge Gate uses a concurrency group to cancel runs when a PR leaves the queue [[.github/workflows/merge-gate.yaml:35-47]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

## Test Categories and Implementation

### Smoke Tests

Smoke tests are designed for high-speed validation of core components.

*   **Executables**: The validation suite produces tests for both `tt-metalium` and `tt-nn`[[.github/workflows/smoke.yaml:92]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Implementation**: These tests often validate basic hardware interactions and core dispatch logic [[.github/workflows/smoke.yaml:108]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Timeouts**: Individual smoke tests are strictly timed to ensure the gate remains fast [[.github/workflows/smoke.yaml:15-18]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

### Simulator Integration (TTSim)

For validation without physical hardware, the pipeline includes `ttsim.yaml`.

*   **Functionality**: It downloads the `libttsim.so` binary [[.github/workflows/ttsim.yaml:103]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies) and runs Metal examples like `matmul_multi_core` in a simulated environment [[.github/workflows/ttsim.yaml:122-145]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Implementation**: It sets `TT_METAL_SIMULATOR` and `TT_METAL_SLOW_DISPATCH_MODE=1` to run kernels through the simulator backend [[.github/workflows/ttsim.yaml:69-70]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Data Flow**: TTSim-compatible tests are dynamically extracted from `tests/pipeline_reorg/ttnn-tests.yaml` using a Python script that filters out incompatible configurations [[.github/workflows/ttsim.yaml:171-187]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

### LLK and Nightly Gates

Low-Level Kernel (LLK) and long-running tests are handled in extended gates:

*   **LLK Validation**: Changes to LLK components (Wormhole, Blackhole, or SFPI) trigger specific build and test logic [[.github/workflows/merge-gate.yaml:92-97]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Nightly L2 Tests**: The `tt-metal-l2-nightly.yaml` workflow runs exhaustive test suites, including DIDT tests, profiler regressions, and full model unit tests [[.github/workflows/tt-metal-l2-nightly.yaml:13-72]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

## Static Analysis and Sanitizer Options

### Clang Tidy Integration

Static analysis is performed via `code-analysis.yaml`.

*   **Incremental vs Full**: The system determines whether to run a full scan (on `main` or config changes) or an incremental scan (based on changed files) [[.github/workflows/code-analysis.yaml:105-121]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
*   **Parallelization**: Incremental scans are parallelized across targets like `tt_metal` and `ttnn`[[.github/workflows/code-analysis.yaml:172-199]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).

### Build-Time Trace

The build system supports `-ftime-trace` for Clang to analyze compilation bottlenecks [[CMakeLists.txt:151-158]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies). This is used to optimize the "Fast Dispatch" build times required for the 5-minute PR gate.

### Sanitizer Configuration in CMake

Sanitizers are integrated directly into the CMake build system [[CMakeLists.txt:67-69]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies). The `ASan` and `TSan` build types are first-class citizens in the project configuration, ensuring that quality gates use the same logic as developer local builds.

### Validation Strategies in Post-Commit

Post-commit tests on specific architectures (e.g., Blackhole) include health checks to ensure the hardware is in a valid state before execution [[dockerfile/upstream_test_images/run_upstream_tests_vanilla.sh:92-104]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies). This includes running `tt-smi -r` and `test_system_health`[[dockerfile/upstream_test_images/run_upstream_tests_vanilla.sh:94]](https://deepwiki.com/tenstorrent/tt-metal/6.6-quality-gates-and-validation-strategies).
