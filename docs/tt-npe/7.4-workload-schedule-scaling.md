## Purpose and Scope

This page documents the experimental workload schedule scaling feature in tt-npe. This feature allows users to linearly scale the temporal spacing of all transfers in a workload by multiplying their phase cycle offsets by a constant factor. This enables analysis of workload behavior under different timing assumptions, such as simulating slower or faster device operation.

For information about constructing workloads, see [Workload Data Model](https://deepwiki.com/tenstorrent/tt-npe/3.1-workload-data-model). For details on phase cycle offsets and transfer timing, see [Transfer State Lifecycle](https://deepwiki.com/tenstorrent/tt-npe/4.2-transfer-state-lifecycle).

## Overview

Workload schedule scaling modifies the timing of all transfers within a workload by applying a linear scaling factor to their `phase_cycle_offset` values. The `phase_cycle_offset` field determines when each transfer can begin execution relative to the start of its containing phase.

**Key Characteristics:**

*   **Linear scaling**: All phase cycle offsets are multiplied by the same factor
*   **Relative timing preserved**: The ratio between any two transfer start times remains constant
*   **No structural changes**: Transfer dependencies, routing, and all other workload properties remain unchanged
*   **Experimental status**: This feature is primarily intended for research and analysis purposes

Sources: [tt_npe/cpp/include/npeWorkload.hpp 108-109](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L108-L109)

## Scaling Factor Behavior

The scale factor determines how the workload schedule is transformed:

| Scale Factor | Effect | Use Case |
| --- | --- | --- |
| `< 1.0` | **Compression** - Transfers occur closer together in time | Simulate faster device operation or tighter scheduling |
| `= 1.0` | **No change** - Original timing preserved | Baseline simulation |
| `> 1.0` | **Expansion** - Transfers occur further apart in time | Simulate slower device operation or looser scheduling |

## Implementation Details

### What Gets Scaled

The scaling operation affects the `phase_cycle_offset` field of every `npeWorkloadTransfer` in the workload:

```
new_phase_cycle_offset = original_phase_cycle_offset × scale_factor
```

**Data Structure Context:**

Sources: [tt_npe/cpp/include/npeWorkload.hpp 25-95](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L25-L95)

### What Does NOT Get Scaled

The following workload properties remain unchanged:

*   **Transfer size**: `packet_size` and `num_packets` are not modified
*   **Source and destination coordinates**: Routing remains identical
*   **NoC type**: NOC_0 vs NOC_1 selection is preserved
*   **Injection rates**: Source injection rates are not scaled
*   **Transfer dependencies**: Parent-child relationships in transfer groups are preserved
*   **Golden cycles**: Reference cycle counts for validation are not modified

Sources: [tt_npe/cpp/include/npeWorkload.hpp 25-83](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L25-L83)

## Interaction with Trace Ingestion

When workloads are created from tt-metal NoC traces, the `phase_cycle_offset` values are derived from actual hardware timestamps. Understanding this origin is important for interpreting scaled results.

### Trace Timestamp Processing

During trace ingestion, timestamps undergo several transformations:

1.   **T0 Normalization**: All timestamps are made relative to the earliest event timestamp (`t0_timestamp`)

    *   See [tt_npe/cpp/src/npeWorkloadIngest.cpp 221-269](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L221-L269)

2.   **Latency Addition**: Device-specific read/write latencies are added to account for NoC protocol overhead

    *   Read latency: [tt_npe/cpp/src/npeWorkloadIngest.cpp 465-473](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L465-L473)
    *   Write latency: [tt_npe/cpp/src/npeWorkloadIngest.cpp 474-485](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L474-L485)

3.   **Fabric Path Latency**: For multi-chip transfers, additional latencies are added for the first hop

    *   See [tt_npe/cpp/src/npeWorkloadIngest.cpp 557-570](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L557-L570)

4.   **Scaling Application**: If `scaleWorkloadSchedule()` is called, the final offsets are multiplied by the scale factor

## Use Cases

### 1. Sensitivity Analysis

Test how workload performance scales with tighter or looser scheduling constraints:

`# Test multiple scale factorsscale_factors = [0.5, 0.75, 1.0, 1.25, 1.5, 2.0]results = [] for factor in scale_factors:    workload = npe.Workload.from_json("trace.json", "wormhole_b0", is_trace=True)    workload.scale_workload_schedule(factor)        config = npe.Config(device_name="wormhole_b0")    stats = npe.simulate(config, workload)        results.append({        'scale_factor': factor,        'total_cycles': stats.get_total_cycles(),        'noc_utilization': stats.get_average_noc_utilization()    })`
### 2. Ideal vs. Realistic Timing

Compare workload performance under ideal (compressed) and realistic timing:

`# Simulate with ideal timing (everything happens as fast as possible)workload_ideal = npe.Workload.from_json("trace.json", "wormhole_b0", is_trace=True)workload_ideal.scale_workload_schedule(0.1)  # 10x compressionstats_ideal = npe.simulate(config, workload_ideal) # Simulate with realistic timingworkload_realistic = npe.Workload.from_json("trace.json", "wormhole_b0", is_trace=True)# No scaling - use original timingstats_realistic = npe.simulate(config, workload_realistic) # Compute performance gapspeedup = stats_realistic.get_total_cycles() / stats_ideal.get_total_cycles()print(f"Realistic timing is {speedup:.2f}x slower than ideal")`
### 3. Clock Frequency Scaling Simulation

Model the effect of running the device at different clock frequencies:

`# Simulate at 2x clock frequency (everything happens 2x faster in cycles)# This is equivalent to scaling schedule by 0.5workload_fast = npe.Workload.from_json("trace.json", "wormhole_b0", is_trace=True)workload_fast.scale_workload_schedule(0.5) # Simulate at 0.5x clock frequency (everything happens 2x slower in cycles)workload_slow = npe.Workload.from_json("trace.json", "wormhole_b0", is_trace=True)workload_slow.scale_workload_schedule(2.0)`

## Limitations and Considerations

### Golden Cycle Accuracy

**Important**: The `golden_cycles` reference timing stored in the workload is **not scaled** when `scaleWorkloadSchedule()` is called. This means:

*   Cycle prediction error metrics may become invalid or misleading
*   Comparisons against golden results should be interpreted carefully
*   The golden cycles represent the original unscaled execution time

If you need accurate prediction error analysis with scaled workloads, you must manually scale the golden cycles as well.

Sources: [tt_npe/cpp/include/npeWorkload.hpp 111-125](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L111-L125)

### Transfer Group Dependencies

Transfer groups (used for multi-chip fabric transfers) maintain their parent-child relationships, but the temporal spacing between dependent transfers is scaled. This means:

*   Dependency ordering is preserved
*   Relative timing between parent and child transfers changes
*   This may affect congestion patterns in multi-hop routing scenarios

Sources: [tt_npe/cpp/include/npeWorkload.hpp 70-72](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L70-L72)[tt_npe/cpp/src/npeWorkloadIngest.cpp 520-638](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L520-L638)

### Injection Rate Interaction

The `injection_rate` field (which specifies how fast a source can inject data) is **not scaled**. This can lead to unrealistic scenarios:

```
If schedule is compressed (factor < 1.0) but injection_rate stays constant:
→ Transfers may become bandwidth-limited rather than schedule-limited

If schedule is expanded (factor > 1.0) but injection_rate stays constant:
→ Source bandwidth may become artificially limiting
```

For most accurate results when scaling, consider also adjusting injection rates or ensuring they are sufficiently high to not become the bottleneck.

Sources: [tt_npe/cpp/include/npeWorkload.hpp 62](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeWorkload.hpp#L62-L62)

### Experimental Status

This feature is marked as experimental, which means:

*   **Limited testing**: May not be validated across all workload types
*   **API stability**: Method signature or behavior may change in future versions
*   **Documentation gaps**: Some edge cases may not be fully documented
*   **Best effort support**: Issues may have lower priority than core features

**Recommendation**: Use this feature primarily for research and sensitivity analysis. For production performance estimates, rely on unscaled simulations with realistic timing.

## Scaling Algorithm Pseudocode

The conceptual implementation of `scaleWorkloadSchedule()`:

```
function scaleWorkloadSchedule(workload, scale_factor):
    for each phase in workload.phases:
        for each transfer in phase.transfers:
            transfer.phase_cycle_offset *= scale_factor
```

This simple linear transformation ensures:

*   All transfers are scaled uniformly
*   Relative ordering is preserved
*   Temporal density increases/decreases consistently

## Related Configuration Options

While `scaleWorkloadSchedule()` modifies the workload itself, several `npeConfig` options affect simulation timing and may interact with scaling:

| Config Option | Effect on Timing | Interaction with Scaling |
| --- | --- | --- |
| Device model selection | Changes link bandwidths and latencies | Independent - scaling affects schedule, device model affects execution |
| Congestion modeling | Affects actual transfer completion times | Independent - scaling changes when transfers start, not how fast they execute |
| Timestep size | Granularity of simulation | Independent - smaller timesteps give finer resolution of scaled schedule |

For more details on configuration options, see [Configuration Options](https://deepwiki.com/tenstorrent/tt-npe/6.3-configuration-options).
