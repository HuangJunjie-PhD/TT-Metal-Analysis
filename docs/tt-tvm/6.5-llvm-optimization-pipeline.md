## Purpose and Scope

The LLVM Optimization Pipeline in TVM applies a series of transformation passes to LLVM IR after code generation but before final machine code emission. This pipeline optimizes the generated IR for performance, applying both general-purpose and target-specific transformations. The optimization system manages two distinct pass manager implementations: the legacy pass manager (for LLVM < 16.0) and the new pass manager (for LLVM >= 16.0).

For information about the overall LLVM backend architecture, see [6.1](https://deepwiki.com/tenstorrent/tt-tvm/6.1-llvm-backend-architecture). For code generation that produces the IR being optimized, see [6.2](https://deepwiki.com/tenstorrent/tt-tvm/6.2-cpu-code-generation) for CPU and [6.3](https://deepwiki.com/tenstorrent/tt-tvm/6.3-gpu-code-generation) for GPU backends.

## Overview

The optimization pipeline executes during the `Finish()` method of code generation, after all TIR functions have been translated to LLVM IR but before the module is ready for execution or further processing. The pipeline respects target-specific optimization requirements and configuration.

## New Pass Manager (LLVM >= 16.0)

Starting with LLVM 16.0, TVM uses the new pass manager infrastructure which provides a more efficient and flexible pass execution model. The new pass manager uses a pipeline string to specify optimization passes.

### New Pass Manager Components

Sources: [src/target/llvm/codegen_llvm.cc 413-485](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L413-L485)

### Implementation Details

The new pass manager implementation in `CodeGenLLVM::Optimize()` follows this structure:

| Component | Purpose | Code Location |
| --- | --- | --- |
| `llvm::PassBuilder` | Constructs and configures the pass pipeline | [src/target/llvm/codegen_llvm.cc 421](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L421-L421) |
| Analysis Managers | Manage analysis results for different IR levels (loop, function, CGSCC, module) | [src/target/llvm/codegen_llvm.cc 423-430](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L423-L430) |
| `StandardInstrumentations` | Provides optional debugging and verification hooks | [src/target/llvm/codegen_llvm.cc 469-474](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L469-L474) |
| Pipeline String | Specifies which passes to run (e.g., `"default<O2>"`) | [src/target/llvm/codegen_llvm.cc 434-467](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L434-L467) |
| `ModulePassManager` | Executes the pass pipeline on the module | [src/target/llvm/codegen_llvm.cc 475-484](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L475-L484) |

The optimization level is determined from the target machine configuration and mapped to a pipeline string:

```
CodeGenOptLevel::None       → "default<O0>"
CodeGenOptLevel::Less       → "default<O1>"
CodeGenOptLevel::Default    → "default<O2>"
CodeGenOptLevel::Aggressive → "default<O3>"
```

## Legacy Pass Manager (LLVM < 16.0)

For LLVM versions prior to 16.0, TVM uses the legacy pass manager system which separates function-level and module-level passes into distinct managers.

### Legacy Pass Manager Architecture

Sources: [src/target/llvm/codegen_llvm.cc 504-557](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L504-L557)

### Pass Manager Configuration

The legacy pass manager is configured through the `PassManagerBuilder` class:

| Configuration | Value | Purpose |
| --- | --- | --- |
| `OptLevel` | 0-3 | Controls aggressiveness of optimization |
| `Inliner` | Function inlining pass | Configured based on optimization level |
| `LoopVectorize` | true | Enables loop vectorization |
| `SLPVectorize` | true | Enables straight-line code vectorization |
| Target Transform Info | From TargetMachine | Provides target-specific cost models |

Sources: [src/target/llvm/codegen_llvm.cc 513-549](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L513-L549)

### Execution Flow

The legacy pass manager executes in two phases:

1.   **Function Pass Phase**: `FunctionPassManager` runs optimization passes on each function individually

    *   Initialization: `fpass.doInitialization()`
    *   Per-function execution: `fpass.run(*it)` for each function
    *   Finalization: `fpass.doFinalization()`

2.   **Module Pass Phase**: `ModulePassManager` runs whole-module optimization passes

    *   Single execution: `mpass.run(*module_)`

Sources: [src/target/llvm/codegen_llvm.cc 551-556](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L551-L556)

Different target backends can customize the optimization pipeline to leverage target-specific optimizations or work around target-specific issues.

### Customization Mechanisms

Sources: [src/target/llvm/codegen_nvptx.cc 201-213](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L201-L213)[src/target/llvm/codegen_amdgpu.cc 210-214](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_amdgpu.cc#L210-L214)

### NVPTX-Specific Optimizations

The NVIDIA PTX backend includes custom pre-optimization to handle device library functions:

**Purpose**: Strip unused CUDA device library functions (prefixed with `__nv_`) to reduce code size and prevent linking issues.

**Implementation**: Before running the standard optimization pipeline, `CodeGenNVPTX::Optimize()` iterates through all functions in the module:

*   Functions with names starting with `__nv_` that are not declarations and not marked `NoInline` are set to `AvailableExternallyLinkage`
*   This allows the optimizer to inline actually-used device functions at call sites
*   Unused device functions are then removed during optimization

Sources: [src/target/llvm/codegen_nvptx.cc 201-213](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_nvptx.cc#L201-L213)

### Legacy Pass Manager Hooks

For the legacy pass manager, targets can override `InitPassManagerBuilder()` to customize pass configuration:

| Method | Default Behavior | Purpose |
| --- | --- | --- |
| `CodeGenLLVM::InitPassManagerBuilder()` | Empty (no customization) | Hook for subclasses to add target-specific passes |
| `CodeGenNVPTX::InitPassManagerBuilder()` | Empty | Placeholder for future NVPTX customizations |
| `CodeGenAMDGPU::InitPassManagerBuilder()` | Empty | Placeholder for future AMDGPU customizations |

## Integration with Compilation Flow

The optimization pipeline is tightly integrated into the overall compilation process managed by `CodeGenLLVM::Finish()`.

### Compilation Flow with Optimization

Sources: [src/target/llvm/codegen_llvm.cc 362-373](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L362-L373)[src/target/llvm/codegen_llvm.cc 413-558](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L413-L558)

### Module Verification

TVM performs verification before and after optimization to catch any IR malformations:

The `Verify()` method uses `llvm::verifyModule()` to check the IR for structural correctness and reports any errors with detailed diagnostic information.

Sources: [src/target/llvm/codegen_llvm.cc 354-360](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L354-L360)[src/target/llvm/codegen_llvm.cc 369-371](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L369-L371)

### Link Module Integration

Before optimization, `Finish()` links any additional LLVM modules that were added via `AddLinkModule()`. This typically includes:

*   Hand-written LLVM IR for intrinsics or special functions
*   Device library functions (e.g., math library functions)
*   External dependencies imported via `HandleImport()`

Linking occurs before optimization so that:

1.   Functions marked `AlwaysInline` from linked modules can be inlined during optimization
2.   Inter-procedural optimizations can work across module boundaries
3.   Unused functions from linked modules can be eliminated

Sources: [src/target/llvm/codegen_llvm.cc 364-368](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L364-L368)[src/target/llvm/codegen_llvm.cc 375-399](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/target/llvm/codegen_llvm.cc#L375-L399)

## Performance Considerations

The optimization pipeline significantly impacts compilation time and runtime performance:

| Factor | Impact | Configuration |
| --- | --- | --- |
| Optimization Level | Higher levels increase compile time but improve runtime performance | Target configuration |
| Vectorization | Enables SIMD instructions for better performance | Enabled by default in both pass managers |
| Inlining | Reduces call overhead at cost of code size | Controlled by optimization level |
| Target-Specific Passes | Can exploit architecture-specific features | Configured via `adjustPassManager()` |

### Debug vs Release Builds

For debugging, lower optimization levels preserve better correspondence between source and generated code:

*   `O0` (None): Minimal optimization, best for debugging
*   `O2` (Default): Recommended for production
*   `O3` (Aggressive): Maximum performance, may increase code size significantly
