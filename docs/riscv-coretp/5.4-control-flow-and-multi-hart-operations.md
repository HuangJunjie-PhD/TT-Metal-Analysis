## Purpose and Scope

This page documents TestStep types used for controlling execution flow and coordinating operations across multiple harts (hardware threads) in multi-processor test scenarios. These steps enable test scenarios to:

*   Switch execution context between different harts in multi-processor configurations
*   Synchronize operations across multiple harts
*   Implement control flow patterns for function calls
*   Coordinate complex interactions in cache coherence and atomic operation testing

For memory access operations, see [Memory Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.1-memory-operations). For arithmetic and computation steps, see [Assertions and Verification](https://deepwiki.com/tenstorrent/riscv-coretp/5.3-assertions-and-verification).

## Core Control Flow Steps

The coretp framework provides four primary TestStep types for control flow and multi-hart coordination:

| Step Type | Purpose | Key Parameters | Synchronization |
| --- | --- | --- | --- |
| `Hart` | Switch execution context to specified hart | `hart_index` | No |
| `HartExit` | Terminate hart execution | `sync` (optional) | Optional barrier |
| `Call` | Control flow for function calls | Function target | No |
| `Arithmetic` | Compute operations with control flow implications | Operation, operands | No |

## Hart Step - Execution Context Switching

### Overview

The `Hart` step designates that subsequent test steps should execute on a specific hardware thread (hart). This is fundamental for multi-processor test scenarios where operations must be distributed across multiple harts.

### Step Structure

**Parameters:**

*   `hart_index` (int): Zero-based index identifying the target hart

### Execution Semantics

When a test scenario encounters a `Hart` step:

1.   The test framework switches the execution context to the specified hart
2.   All subsequent steps execute on that hart until another `Hart` step is encountered
3.   Multiple `Hart` steps can reference the same hart index to return to that hart's execution context

## HartExit Step - Thread Termination and Synchronization

### Overview

The `HartExit` step terminates execution on the current hart and optionally provides a synchronization barrier across multiple harts. This is critical for ensuring memory visibility and ordering in multi-processor tests.

### Step Structure

**Parameters:**

*   `sync` (bool, optional): When `True`, creates a barrier where all harts wait until all have reached their `HartExit` points

### Synchronization Semantics

### Multi-Hart Synchronization Pattern

When `sync=True`, `HartExit` implements a barrier synchronization:

1.   Each hart executes until reaching its `HartExit(sync=True)` step
2.   Harts that arrive early wait at the barrier
3.   Once all harts reach their respective `HartExit` points, all proceed simultaneously
4.   This ensures memory operations from all harts are globally visible

**Sources:**[coretp/step/__init__.py 12](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/step/__init__.py#L12-L12)[coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py 766-791](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/plans/zicbom_zicboz_zicbop/zicbom_zicboz_zicbop_scenarios.py#L766-L791)

## Call Step - Function Control Flow

### Overview

The `Call` step provides control flow for invoking functions or code regions within test scenarios. This enables modular test construction and code reuse.

### Usage Context

While `Call` is imported and available in the step ecosystem, its primary use cases include:

*   Invoking helper functions defined in `CodePage` regions
*   Implementing subroutine-style test patterns
*   Creating reusable test fragments

## Complete Multi-Hart Test Example

### SID_ZICBO_021: Multi-Processor Cache Zero Test

This scenario demonstrates comprehensive multi-hart coordination for testing cache block zeroing operations across multiple processors.

**Test Execution Flow:**

1.   **Configuration Phase** (Lines 753-761):

    *   Set `min_num_harts=2` in environment configuration
    *   Configure CSRs to enable cache block operations
    *   Allocate shared memory region

2.   **Hart 0 Production Phase** (Lines 764-766):

    *   Switch to hart 0 context: `Hart(hart_index=0)`
    *   Write test data: `Store(memory=mem, value=0xDEADBEEF, offset=0)`
    *   Synchronize: `HartExit(sync=True)` - wait for all harts to reach barrier

3.   **Hart 1 Operation Phase** (Lines 768-770):

    *   Switch to hart 1 context: `Hart(hart_index=1)`
    *   Execute cache operation: `MemAccess(op="cbo.zero", memory=mem)`
    *   Synchronize: `HartExit(sync=True)` - ensure operation is globally visible

4.   **Hart 0 Verification Phase** (Lines 773-775):

    *   Return to hart 0: `Hart(hart_index=0)`
    *   Read memory: `Load(op="lw", memory=mem, offset=0)`
    *   Verify result: `AssertEqual(src1=load_verify, src2=0)`

## Common Multi-Hart Patterns Summary

### Pattern Categories

| Pattern | Use Case | Synchronization | Example Scenario |
| --- | --- | --- | --- |
| Producer-Consumer | One hart writes, another reads | Required | SID_ZICBO_021 |
| Concurrent Operations | Multiple harts access same resource | Optional | Cache coherence tests |
| Atomic Interference | Test atomic op interaction with other ops | Partial | SID_ZICBO_039 |
| Sequential Handoff | Harts execute in strict sequence | Required | Page table walk tests |

### Synchronization Guidelines

**Use `HartExit(sync=True)` when:**

*   Memory operations must be globally visible before proceeding
*   Testing cache coherence semantics
*   Ensuring ordering across multiple harts
*   Verifying results written by other harts

**Avoid synchronization when:**

*   Testing race conditions intentionally
*   Measuring concurrent performance
*   Simulating real-world unsynchronized scenarios
