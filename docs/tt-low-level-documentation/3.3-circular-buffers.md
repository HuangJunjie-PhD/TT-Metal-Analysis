# Circular Buffers

Relevant source files
*   [data_movement_doc/general/dm_best_practices.md](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1)

## Purpose and Scope

This document details circular buffer (CB) usage patterns for coordinating data movement between data movement kernels, compute kernels, and external memory. Circular buffers provide a producer-consumer synchronization mechanism that enables overlapped execution and efficient memory usage in TT-Metal kernels.

This page focuses specifically on the CB API functions and their correct usage patterns. For broader data movement concepts and asynchronous operations, see [Asynchronous Operations and Batching](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.2-asynchronous-operations-and-batching). For multi-core synchronization using CBs, refer to the examples in [Core API Concepts](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.1-core-api-concepts).

## Circular Buffer Architecture

Circular buffers are fixed-size memory regions in L1 that act as FIFO queues between kernel threads. Each circular buffer is divided into equal-sized pages (tiles), and operations always work in units of pages.

**Circular Buffer Memory Layout and Operations**

Sources: [data_movement_doc/general/dm_best_practices.md 63-88](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L63-L88)

**Key Architectural Constraints:**

| Constraint | Requirement | Reason |
| --- | --- | --- |
| Total Size | Must be multiple of page size | CB memory is divided evenly into pages |
| Page Count | `total_size / page_size` must be integer | No partial pages allowed |
| Operation Granularity | All operations work in page units | Hardware manages page boundaries |
| Reserve/Push Match | Reserved pages must equal pushed pages | State consistency requirement |
| Wait/Pop Match | Waited pages must equal popped pages | State consistency requirement |

## Producer-Consumer Synchronization Model

Circular buffers implement a blocking producer-consumer pattern where operations automatically synchronize based on buffer fullness.

**Producer-Consumer State Machine**

**Blocking Behavior:**

*   `cb_reserve_back(n)`: Blocks if fewer than `n` pages are free
*   `cb_wait_front(n)`: Blocks if fewer than `n` pages are filled
*   `cb_push_back(n)`: Non-blocking, updates fill count
*   `cb_pop_front(n)`: Non-blocking, updates free count

Sources: [data_movement_doc/general/dm_best_practices.md 63-88](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L63-L88)

## Producer Operations

The producer side writes data into the circular buffer using a three-step pattern: reserve, write, push.

**Producer Operation Flow**

### cb_reserve_back(cb_id, n)

**Purpose:** Reserve `n` pages at the back of the circular buffer for writing. This operation blocks until sufficient free space is available.

**Usage:**

`// Reserve 4 pages for writingcb_reserve_back(cb_id, 4);`
**Blocking Conditions:**

*   Current free pages <`n`: Blocks until consumer frees enough pages via `cb_pop_front()`
*   Buffer initially empty: Returns immediately

Sources: [data_movement_doc/general/dm_best_practices.md 69-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L69-L79)

### get_write_ptr(cb_id)

**Purpose:** Get the L1 memory address where the next write should occur. This address points to the first reserved page.

**Usage:**

`uint32_t write_addr = get_write_ptr(cb_id);// Use write_addr as destination for NOC operationsnoc_async_read(src_addr, write_addr, size);`
**Important:** The write pointer advances automatically with each `cb_push_back()`. If you reserve multiple pages, you must calculate offsets manually:

`cb_reserve_back(cb_id, 4);uint32_t base_addr = get_write_ptr(cb_id);constexpr uint32_t page_size = 2048;  // Example for (uint32_t i = 0; i < 4; i++) {    uint32_t page_addr = base_addr + i * page_size;    noc_async_read(src_addr + i * page_size, page_addr, page_size);}noc_async_read_barrier();cb_push_back(cb_id, 4);`
Sources: [data_movement_doc/general/dm_best_practices.md 74-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L74-L79)

### cb_push_back(cb_id, n)

**Purpose:** Mark `n` pages as filled and available for the consumer. This operation does not block.

**Critical Rule:** The sum of all `cb_push_back()` calls must exactly equal the number of pages reserved by the most recent `cb_reserve_back()`.

**Usage:**

`// Single push matching reservecb_reserve_back(cb_id, 2);// ... write data ...cb_push_back(cb_id, 2);  // Must match // Multiple pushes totaling reserved amountcb_reserve_back(cb_id, 4);// ... write first 2 pages ...cb_push_back(cb_id, 2);// ... write next 2 pages ...cb_push_back(cb_id, 2);  // Total: 2 + 2 = 4 (matches reserve)`
Sources: [data_movement_doc/general/dm_best_practices.md 67-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L67-L79)

## Consumer Operations

The consumer side reads data from the circular buffer using a three-step pattern: wait, read, pop.

**Consumer Operation Flow**

### cb_wait_front(cb_id, n)

**Purpose:** Wait until `n` pages are available at the front of the circular buffer for reading. This operation blocks until sufficient filled pages exist.

**Usage:**

`// Wait for 4 pages to be readycb_wait_front(cb_id, 4);`
**Blocking Conditions:**

*   Current filled pages <`n`: Blocks until producer fills enough pages via `cb_push_back()`
*   Buffer initially has data: Returns immediately if `n` pages already filled

Sources: [data_movement_doc/general/dm_best_practices.md 83-87](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L83-L87)

### get_read_ptr(cb_id)

**Purpose:** Get the L1 memory address where the next read should occur. This address points to the first filled page.

**Usage:**

`uint32_t read_addr = get_read_ptr(cb_id);// Use read_addr as source for processing or NOC operationsprocess_tile(read_addr);`
**Multi-page reads:**

`cb_wait_front(cb_id, 4);uint32_t base_addr = get_read_ptr(cb_id);constexpr uint32_t page_size = 2048; for (uint32_t i = 0; i < 4; i++) {    uint32_t page_addr = base_addr + i * page_size;    noc_async_write(page_addr, dst_addr + i * page_size, page_size);}cb_pop_front(cb_id, 4);`
Sources: [data_movement_doc/general/dm_best_practices.md 250-252](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L250-L252)

### cb_pop_front(cb_id, n)

**Purpose:** Mark `n` pages as consumed and free for reuse by the producer. This operation does not block.

**Critical Rule:** The sum of all `cb_pop_front()` calls must exactly equal the number of pages waited by the most recent `cb_wait_front()`.

**Usage:**

`// Single pop matching waitcb_wait_front(cb_id, 2);// ... read data ...cb_pop_front(cb_id, 2);  // Must match // Multiple pops totaling waited amountcb_wait_front(cb_id, 4);// ... read first 2 pages ...cb_pop_front(cb_id, 2);// ... read next 2 pages ...cb_pop_front(cb_id, 2);  // Total: 2 + 2 = 4 (matches wait)`

## Synchronization with NOC Operations

Circular buffer operations must be synchronized with asynchronous NOC operations to ensure data consistency.

**Synchronization Points in Producer-Consumer Pattern**

**Barrier Placement Rules:**

| Operation Type | Barrier Required | Location |
| --- | --- | --- |
| Producer reads into CB | `noc_async_read_barrier()` | After all reads, before `cb_push_back()` |
| Consumer writes from CB | `noc_async_write_barrier()` (optional) | After all writes, before `cb_pop_front()` or kernel end |
| Producer writes into CB | Manual write (no barrier) | Direct L1 write, no barrier needed |
| Consumer reads from CB | Manual read (no barrier) | Direct L1 read, no barrier needed |

**Critical Timing:**

`// WRONG - cb_push_back before data arrivescb_reserve_back(cb_id, 1);uint32_t addr = get_write_ptr(cb_id);noc_async_read(src_addr, addr, size);cb_push_back(cb_id, 1);  // BUG: Data might not be there yet! // CORRECT - Barrier ensures data is presentcb_reserve_back(cb_id, 1);uint32_t addr = get_write_ptr(cb_id);noc_async_read(src_addr, addr, size);noc_async_read_barrier();  // Wait for datacb_push_back(cb_id, 1);     // Now safe`

## Common Pitfalls and Anti-Patterns

### Pitfall 1: Mismatched Reserve/Push Counts

**Problem:** Reserving `n` pages but pushing a different number causes buffer state corruption.

`// WRONG - Reserved 2, pushed 1cb_reserve_back(cb_id, 2);uint32_t addr = get_write_ptr(cb_id);// ... write data to 2 pages ...cb_push_back(cb_id, 1);  // BUG: Only pushed 1 of 2 reserved pages // CORRECT - Match counts exactlycb_reserve_back(cb_id, 2);uint32_t addr = get_write_ptr(cb_id);// ... write data to 2 pages ...cb_push_back(cb_id, 2);  // Correct: matches reserve`
**Consequence:** Buffer accounting becomes incorrect, leading to deadlocks or data corruption. The producer may think pages are free when they're still reserved, or the consumer may think pages are filled when they're not.

Sources: [data_movement_doc/general/dm_best_practices.md 67-72](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L67-L72)

### Pitfall 2: Mismatched Wait/Pop Counts

**Problem:** Waiting for `n` pages but popping a different number.

`// WRONG - Waited 3, popped 2cb_wait_front(cb_id, 3);uint32_t addr = get_read_ptr(cb_id);// ... process 3 pages ...cb_pop_front(cb_id, 2);  // BUG: Only popped 2 of 3 waited pages // CORRECT - Match counts exactlycb_wait_front(cb_id, 3);uint32_t addr = get_read_ptr(cb_id);// ... process 3 pages ...cb_pop_front(cb_id, 3);  // Correct: matches wait`
Sources: [data_movement_doc/general/dm_best_practices.md 83-87](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L83-L87)

### Pitfall 3: Missing Barrier Before Push

**Problem:** Pushing pages to the circular buffer before NOC data arrives.

`// WRONG - Race conditioncb_reserve_back(cb_id, 1);uint32_t addr = get_write_ptr(cb_id);noc_async_read(src_addr, addr, size);cb_push_back(cb_id, 1);  // BUG: Data might not be there yet!// Consumer may read garbage data // CORRECT - Barrier ensures completioncb_reserve_back(cb_id, 1);uint32_t addr = get_write_ptr(cb_id);noc_async_read(src_addr, addr, size);noc_async_read_barrier();  // Wait for completioncb_push_back(cb_id, 1);     // Safe: data is present`
**Consequence:** Consumer reads incomplete or garbage data, causing incorrect computation results.

Sources: [data_movement_doc/general/dm_best_practices.md 12-22](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L12-L22)[data_movement_doc/general/dm_best_practices.md 76-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L76-L79)

### Pitfall 4: Incorrect Page Size Calculations

**Problem:** Buffer size not evenly divisible by page size, or incorrect offset calculations for multi-page operations.

`// WRONG - Size mismatchconstexpr uint32_t cb_size = 10000;      // Total sizeconstexpr uint32_t page_size = 3000;     // Page size// 10000 / 3000 = 3.33... pages - NOT ALLOWED! // CORRECT - Even divisionconstexpr uint32_t cb_size = 12000;      // Total sizeconstexpr uint32_t page_size = 3000;     // Page size// 12000 / 3000 = 4 pages exactly // WRONG - Manual offset calculationcb_reserve_back(cb_id, 4);uint32_t base = get_write_ptr(cb_id);for (uint32_t i = 0; i < 4; i++) {    // BUG: Wrong multiplier or page_size value    noc_async_read(src + i * wrong_size, base + i * wrong_size, page_size);} // CORRECT - Use configured page sizecb_reserve_back(cb_id, 4);uint32_t base = get_write_ptr(cb_id);constexpr uint32_t page_size = 2048;  // Must match CB configfor (uint32_t i = 0; i < 4; i++) {    noc_async_read(src + i * page_size, base + i * page_size, page_size);}`

## Complete Usage Patterns

### Pattern 1: Single-Buffered Read-Process-Write

Producer reads data into CB, consumer processes and writes out. Single page at a time.

`// Producer kernel (data movement)constexpr uint32_t cb_id_in = 0;constexpr uint32_t num_tiles = 32; for (uint32_t tile = 0; tile < num_tiles; tile++) {    cb_reserve_back(cb_id_in, 1);    uint32_t write_addr = get_write_ptr(cb_id_in);        noc_async_read(src_noc_addr + tile * tile_size, write_addr, tile_size);    noc_async_read_barrier();        cb_push_back(cb_id_in, 1);} // Consumer kernel (compute)constexpr uint32_t cb_id_in = 0;constexpr uint32_t cb_id_out = 1; for (uint32_t tile = 0; tile < num_tiles; tile++) {    // Read from input CB    cb_wait_front(cb_id_in, 1);    uint32_t read_addr = get_read_ptr(cb_id_in);        // Process into output CB    cb_reserve_back(cb_id_out, 1);    uint32_t write_addr = get_write_ptr(cb_id_out);        process_tile(read_addr, write_addr);        cb_pop_front(cb_id_in, 1);    cb_push_back(cb_id_out, 1);}`
Sources: [data_movement_doc/general/dm_best_practices.md 74-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L74-L79)

### Pattern 2: Batched Multi-Page Operations

Producer reads multiple tiles per iteration for better NOC utilization.

`// Producer kernel with batchingconstexpr uint32_t cb_id_in = 0;constexpr uint32_t batch_size = 4;constexpr uint32_t num_batches = 8;constexpr uint32_t tile_size = 2048; for (uint32_t batch = 0; batch < num_batches; batch++) {    cb_reserve_back(cb_id_in, batch_size);    uint32_t base_addr = get_write_ptr(cb_id_in);        // Issue batch of reads    for (uint32_t i = 0; i < batch_size; i++) {        uint32_t tile_addr = base_addr + i * tile_size;        uint32_t src_addr = src_noc_addr + (batch * batch_size + i) * tile_size;        noc_async_read(src_addr, tile_addr, tile_size);    }        // Single barrier for entire batch    noc_async_read_barrier();    cb_push_back(cb_id_in, batch_size);} // Consumer kernelfor (uint32_t batch = 0; batch < num_batches; batch++) {    cb_wait_front(cb_id_in, batch_size);    uint32_t base_addr = get_read_ptr(cb_id_in);        for (uint32_t i = 0; i < batch_size; i++) {        uint32_t tile_addr = base_addr + i * tile_size;        process_tile(tile_addr);    }        cb_pop_front(cb_id_in, batch_size);}`
Sources: [data_movement_doc/general/dm_best_practices.md 31-49](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L31-L49)[data_movement_doc/general/dm_best_practices.md 74-79](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L74-L79)

### Pattern 3: Double-Buffered Pipeline

Two circular buffers allow overlapped read and compute.

`// Producer kernel - fills both CBs alternatelyconstexpr uint32_t cb_id_0 = 0;constexpr uint32_t cb_id_1 = 1;constexpr uint32_t tiles_per_buffer = 8;constexpr uint32_t num_iterations = 16; for (uint32_t iter = 0; iter < num_iterations; iter++) {    uint32_t cb_id = (iter % 2 == 0) ? cb_id_0 : cb_id_1;        cb_reserve_back(cb_id, tiles_per_buffer);    uint32_t write_addr = get_write_ptr(cb_id);        for (uint32_t i = 0; i < tiles_per_buffer; i++) {        noc_async_read(src_addr + (iter * tiles_per_buffer + i) * tile_size,                      write_addr + i * tile_size, tile_size);    }        noc_async_read_barrier();    cb_push_back(cb_id, tiles_per_buffer);} // Consumer kernel - processes from alternating CBsfor (uint32_t iter = 0; iter < num_iterations; iter++) {    uint32_t cb_id = (iter % 2 == 0) ? cb_id_0 : cb_id_1;        cb_wait_front(cb_id, tiles_per_buffer);    uint32_t read_addr = get_read_ptr(cb_id);        // Process all tiles in this buffer    for (uint32_t i = 0; i < tiles_per_buffer; i++) {        process_tile(read_addr + i * tile_size);    }        cb_pop_front(cb_id, tiles_per_buffer);}`
**Benefit:** While consumer processes CB 0, producer can be filling CB 1, improving pipeline utilization.

Sources: [data_movement_doc/general/dm_best_practices.md 31-49](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/general/dm_best_practices.md?plain=1#L31-L49)

## Summary of CB API Functions

| Function | Purpose | Blocks? | Must Match |
| --- | --- | --- | --- |
| `cb_reserve_back(cb_id, n)` | Reserve n pages for writing | Yes (if not enough free) | `cb_push_back()` total |
| `get_write_ptr(cb_id)` | Get L1 address for writing | No | N/A |
| `cb_push_back(cb_id, n)` | Mark n pages as filled | No | `cb_reserve_back()` count |
| `cb_wait_front(cb_id, n)` | Wait for n pages to read | Yes (if not enough filled) | `cb_pop_front()` total |
| `get_read_ptr(cb_id)` | Get L1 address for reading | No | N/A |
| `cb_pop_front(cb_id, n)` | Mark n pages as freed | No | `cb_wait_front()` count |

**Key Synchronization Rule:** Always use `noc_async_read_barrier()` after issuing NOC reads and before calling `cb_push_back()` to ensure data has arrived in L1 memory.
