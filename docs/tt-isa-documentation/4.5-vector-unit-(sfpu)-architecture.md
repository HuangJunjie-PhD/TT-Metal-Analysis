The **Vector Unit (SFPU)** is one of three compute units within the Tensix Coprocessor, providing 32-wide SIMD FP32 operations. The Vector Unit consists of **five independent sub-units** that can execute in parallel: Load, Simple, MAD, Round, and Store. Each sub-unit handles a specific category of operations, and the `SFPLOADMACRO` instruction enables scheduling up to five instructions simultaneously (one per sub-unit) to achieve maximum throughput.

The Vector Unit operates on a dedicated register file called **LReg** (32 registers × 32 lanes) and can perform fused multiply-add (FMA), transcendental functions, conditional operations, and data movement between the Dst register file and LReg.

For information about other compute units, see Matrix Unit (FPU) Architecture (page 4.4) and Scalar Unit (ThCon) and Utilities (page 4.8). For instruction reference details, see Vector Unit Core Instructions (page 5.3), Vector Unit Data Movement Instructions (page 5.4), and Vector Unit Configuration and Control Instructions (page 5.5).

## Five Sub-Unit Architecture

The Vector Unit implements a **5-way parallel execution model** where instructions are dispatched to five independent sub-units. This architecture enables instruction-level parallelism: one instruction from each sub-unit can execute simultaneously on the same cycle.

**Diagram: Five Sub-Unit Parallel Execution**

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 1-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L1-L156)[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 1-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L1-L156) Diagram 3 from high-level context

### Sub-Unit Instruction Assignment

| Sub-Unit | Primary Instructions | Parallelism Constraint |
| --- | --- | --- |
| **Load** | `SFPLOAD`, `SFPLOADI`, `SFPLOADMACRO` | At most one per cycle |
| **Simple** | `SFPABS`, `SFPAND`, `SFPCAST`, `SFPCOMPC`, `SFPCONFIG`, `SFPDIVP2`, `SFPENCC`, `SFPEXEXP`, `SFPEXMAN`, `SFPIADD`, `SFPLZ`, `SFPMOV`, `SFPNOP`, `SFPNOT`, `SFPOR`, `SFPPOPC`, `SFPPUSHC`, `SFPSETCC`, `SFPSETEXP`, `SFPSETMAN`, `SFPSETSGN`, `SFPSHFT`, `SFPSWAP`, `SFPTRANSP`, `SFPXOR` | At most one per cycle; if both Simple and Round execute, one must have `VD == 16` |
| **MAD** | `SFPADD`, `SFPADDI`, `SFPLUT`, `SFPLUTFP32`, `SFPMAD`, `SFPMUL`, `SFPMULI`, `SFPMUL24` (Blackhole) | At most one per cycle |
| **Round** | `SFPSHFT2`, `SFPSTOCHRND` | At most one per cycle; if both Simple and Round execute, one must have `VD == 16` |
| **Store** | `SFPSTORE` | At most one per cycle |

**Special Case:** If `SFPSWAP` is scheduled to the Simple sub-unit, `SFPNOP` must be scheduled to the MAD sub-unit for the same cycle, and both Simple and Round must be idle or have `SFPNOP` on the next cycle.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 3-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L3-L13)[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 3-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L3-L13)

### Achieving 5-Way Parallelism with SFPLOADMACRO

Standard instruction push from RISC-V T1 can only execute one Vector Unit instruction per cycle. To achieve multi-instruction parallelism, use **`SFPLOADMACRO`**, which:

1.   Executes an `SFPLOAD` operation (moving data from Dst to LReg)
2.   Schedules up to **four additional instructions** for execution in future cycles (one each to Simple, MAD, Round, and Store sub-units)

**Configuration Requirements:**

*   Pre-configure scheduling state via `SFPCONFIG` writing to `LoadMacroConfig[32]`
*   `LoadMacroConfig` contains instruction templates, sequence timing, and delay configuration

**Example Parallelism Pattern:**

```
Cycle 0: SFPLOADMACRO executes on Load sub-unit, schedules 4 future instructions
Cycle 1: Simple + MAD + Round + Store all execute scheduled instructions
Cycle 2: SFPLOADMACRO executes again, schedules next batch
...
```

This enables sustained throughput of **5 instructions per cycle** when properly pipelined.

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 1-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L1-L156)[BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md 1-156](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPLOADMACRO.md?plain=1#L1-L156)

## Architecture Overview

The Vector Unit operates as a **32-lane SIMD processor**, with each lane processing one FP32 value independently. RISC-V T1 (the Math thread) controls the Vector Unit by pushing instructions to its instruction FIFO. Each instruction is dispatched to one of the five sub-units based on its opcode.

**Diagram: Vector Unit Data Flow**

**Lane-Parallel Execution Model:**

*   Each of 32 lanes has its own copy of all 17 LReg registers (LReg[0] through LReg[16])
*   Instructions execute **lanewise**: each lane processes its data independently
*   Lane enable masks control which lanes execute each instruction
*   `LaneConfig[Lane]` provides per-lane configuration

## LReg Register File

The Vector Unit has a dedicated register file called **LReg**, providing working storage for vector operations. LReg is replicated per lane, meaning each of the 32 lanes has its own independent copy of all registers.

**Register Organization:**

*   **Total registers:** 17 (indexed 0-16)
*   **Width:** 32 bits per register (typically FP32, but can hold any 32-bit value)
*   **Total capacity:** 17 registers × 32 lanes × 32 bits = 17,408 bits
*   **Per-lane storage:** Each lane maintains its own copy of all 17 LReg registers
*   **Register indices:** LReg[0] through LReg[16]

**Diagram: LReg Organization**

**Special Register Usage:**

**Access Characteristics:**

```
Read Ports:  Up to 4 (e.g., SFPMAD reads VA, VB, VC and writes VD)
Write Ports: 1 primary destination (VD)
Latency:     1 cycle (2 cycles for SFPMAD result)
Addressing:  Direct (register index in instruction) or indirect (via LReg[7])
```

## MAD Sub-Unit: SFPMAD Instruction

The **MAD sub-unit** executes fused multiply-add operations via the `SFPMAD` instruction, which performs: `VD = ±(VA * VB) ± VC`. This is the primary computational workhorse of the Vector Unit, achieving 0.0864 TFLOP/s at 1.35 GHz (Blackhole).

### Instruction Format

**Parameters:**

*   **VA:** Source register A index (4 bits, 0-15) - can be indirect
*   **VB:** Source register B index (4 bits, 0-15)
*   **VC:** Source register C index (4 bits, 0-15)
*   **VD:** Destination register index (4 bits, 0-15) - can be indirect
*   **Mod1:** Modifier bits controlling operation behavior

### Modifier Bits (Mod1)

| Bit | Constant | Description |
| --- | --- | --- |
| 0 | `SFPMAD_MOD1_NEGATE_VA` | Negate operand A (Blackhole only) |
| 1 | `SFPMAD_MOD1_NEGATE_VC` | Negate operand C (Blackhole only) |
| 2 | `SFPMAD_MOD1_INDIRECT_VA` | Use LReg[7][3:0] for VA index |
| 3 | `SFPMAD_MOD1_INDIRECT_VD` | Use LReg[7][3:0] for VD index |

## IEEE754 Conformance and Divergence

The Vector Unit's FMA implementation diverges from IEEE754 in several important ways. Understanding these differences is critical for numerical correctness.

### Common Divergences (Both Wormhole and Blackhole)

**Denormal Handling:**

*   **Input denormals:** Treated as signed zero (flushed on input)
*   **Output denormals:** Flushed to signed zero after rounding (Blackhole) or positive zero (Wormhole)
*   **Rationale:** Avoids complex normalization logic and variable shifts

**Partial Fusion:** The multiply and add are _partially_ fused but not _completely_ fused:

*   The product retains higher precision than FP32 (4 extra bits)
*   Not infinite precision as required by IEEE754
*   23 bits of precision are removed from the product rather than added to the addend

**Overflow Behavior:**

*   If `x * y` alone would overflow to infinity, the result is infinity even if the fully fused operation would be finite
*   If `x * y` alone would underflow to zero, the result is as if `x * y` was exactly zero

**NaN Handling:**

*   All NaN outputs are canonical (`0x7fc00000`) in Blackhole
*   Wormhole NaN outputs have at least bits `0x7f800001` set, but computation continues and can gain additional bits

**Rounding:**

*   Always round to nearest, ties to even (RNE)
*   Other rounding modes are not supported

| **Zero outputs** | Always positive zero | Can be negative zero |
| **NaN bit pattern** | `≥0x7f800001`, non-canonical | `0x7fc00000`, canonical |
| **Negative zero** | Flushed before rounding | Preserved through rounding |
| **Denormal rounding** | Has edge-case bugs | More correct |
| **Instruction modifiers** | No negation modifiers | `NEGATE_VA`, `NEGATE_VC` |

## Instruction Scheduling and Hazards

### Latency Characteristics

**SFPMAD Instruction:**

*   **Execution latency:** 2 cycles
*   **Issue rate:** 1 instruction per cycle (when no hazards)
*   **Throughput:** 32 FP32 operations per cycle (at steady state)

### Automatic Stalling (Blackhole Only)

Blackhole introduced automatic hazard detection for SFPMAD. Hardware ensures that instructions following SFPMAD do not read from locations written by SFPMAD in the previous cycle.

**Stalling Logic:**

*   Detects read-after-write (RAW) hazards automatically
*   Stalls consuming instruction for 1 cycle if hazard detected
*   Conservative assumptions when indirect addressing is used: 
    *   `SFPMAD_MOD1_INDIRECT_VA`: Assumes SFPMAD reads from every LReg
    *   `SFPMAD_MOD1_INDIRECT_VD`: Assumes SFPMAD writes to every LReg

**Automatic Stalling Limitations:**

The automatic stalling logic has known bugs and does not detect hazards for:

| Instruction | Issue | Workaround |
| --- | --- | --- |
| `SFPAND` with `SFPAND_MOD1_USE_VB` | Logic ignores modifier, thinks it reads VD not VB | Manual NOP insertion |
| `SFPOR` with `SFPOR_MOD1_USE_VB` | Logic ignores modifier, thinks it reads VD not VB | Manual NOP insertion |
| `SFPIADD` | Logic doesn't realize it reads VD | Manual NOP insertion |
| `SFPSHFT` | Logic doesn't realize it reads VD | Manual NOP insertion |
| `SFPCONFIG` | Logic doesn't realize it can read LReg[0] | Manual NOP insertion |
| `SFPSWAP` (non-swap modes) | Logic misses 1st cycle reads | Manual NOP insertion |
| `SFPSHFT2` (certain modes) | Logic doesn't detect reads or reads wrong register | Manual NOP insertion |

**SFPLOADMACRO Exception:**

*   Automatic stalling does NOT apply within `SFPLOADMACRO` sequences
*   Software must manually ensure 1-cycle gap between SFPMAD and consumers

**Sources:**[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-87](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L87)

## Performance Characteristics

### Throughput Analysis

**Per Instruction:**

*   32 FP32 multiplications
*   32 FP32 additions
*   **Total:** 64 FP32 operations per instruction

**Peak Throughput:**

| Architecture | Clock Rate | TFLOP/s per Vector Unit |
| --- | --- | --- |
| **Wormhole B0** | 1.0 GHz | 0.064 |
| **Blackhole A0** | 1.35 GHz | 0.0864 |

**Calculation:**

```
Wormhole:   64 ops/instruction × 1 instruction/cycle × 1.0 GHz = 64 GFLOP/s = 0.064 TFLOP/s
Blackhole:  64 ops/instruction × 1 instruction/cycle × 1.35 GHz = 86.4 GFLOP/s = 0.0864 TFLOP/s
```

### Achieving Peak Performance

To achieve peak throughput while simultaneously moving data in/out of the Vector Unit, use `SFPLOADMACRO`:

*   Combines load/compute/store operations
*   Hides data movement latency
*   Enables sustained throughput

**Performance Comparison with Matrix Unit:**

| Unit | Precision | Peak Throughput (Blackhole) | Use Case |
| --- | --- | --- | --- |
| Matrix Unit (FPU) | TF32/BF16/FP16 | 4.096 TFLOP/s | Matrix operations, high throughput |
| Vector Unit (SFPU) | FP32 | 0.0864 TFLOP/s | Element-wise ops, high precision |
| Ratio | - | ~47x faster (Matrix) | - |

**When to Use Vector Unit:**

*   Operations requiring FP32 precision
*   Element-wise operations (not matrix multiplication)
*   Transcendental functions (exp, log, sqrt, etc.)
*   Conditional/predicated operations
*   Operations after Matrix Unit produces FP32 results in Dst

## Wormhole vs Blackhole Enhancements

Blackhole introduced several significant improvements to the Vector Unit over Wormhole.

### Major Enhancements Summary

**Diagram: Blackhole Vector Unit Improvements**

### Detailed Improvements

**1. Negation Modifiers**

*   **Wormhole:** Must manually XOR sign bit before/after SFPMAD
*   **Blackhole:** Hardware negation via `SFPMAD_MOD1_NEGATE_VA` and `SFPMAD_MOD1_NEGATE_VC`
*   **Benefit:** Reduces instruction count, eliminates extra LReg usage

**2. Automatic Instruction Scheduling**

*   **Wormhole:** All hazards must be manually avoided with NOPs
*   **Blackhole:** Hardware detects most RAW hazards and auto-stalls
*   **Limitation:** Known bugs require manual NOPs for specific cases

**3. Improved IEEE754 Conformance**

*   **Wormhole Issues:**
    *   All zero outputs forced to positive zero
    *   Non-canonical NaN bit patterns
    *   Denormal rounding edge cases incorrect
    *   Sign bit leakage in NaN results

*   **Blackhole Improvements:**
    *   Negative zero properly preserved
    *   Canonical NaN outputs (`0x7fc00000`)
    *   Better denormal handling
    *   More predictable edge cases

**4. Higher Clock Frequency**

*   **Wormhole:** 1.0 GHz → 0.064 TFLOP/s
*   **Blackhole:** 1.35 GHz → 0.0864 TFLOP/s (35% improvement)

### Recommendation

For new code:

*   **Target Blackhole** when possible for better numerical behavior
*   **Test on both** if portability is required
*   **Use bit-perfect models** to validate numerical correctness
*   **Document assumptions** about zero/NaN handling

## Programming Considerations

### Best Practices

1.   **Understand FMA limitations:** Use bit-perfect models [Miscellaneous/FMA/fma.c](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/Miscellaneous/FMA/fma.c) to validate numerical behavior
2.   **Insert NOPs when needed:** On Blackhole, manually insert NOPs for instructions not covered by auto-stalling
3.   **Avoid indirect addressing in hot loops:** Conservative stalling assumptions hurt performance
4.   **Use SFPLOADMACRO for throughput:** Essential for hiding data movement latency
5.   **Leverage negation modifiers (Blackhole):** Reduces instruction count vs manual XOR
6.   **Test edge cases:** Zero, infinity, and NaN handling differs from IEEE754

### Performance Optimization

1.   **Minimize LReg pressure:** 32 registers shared across complex computations
2.   **Schedule independent operations:** Fill SFPMAD hazard bubbles with other work
3.   **Batch vector operations:** Amortize control overhead across 32 lanes
4.   **Use lane masking:** Disable unused lanes rather than branching
