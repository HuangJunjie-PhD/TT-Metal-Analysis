This page documents the various communication interfaces provided by the Luwen codebase for interacting with Tenstorrent AI accelerator chips. These interfaces form the foundation for sending and receiving data to and from the chips, enabling control, monitoring, and data transfer operations.

For information about specific chip implementations, see [Chip Support](https://deepwiki.com/tenstorrent/luwen/3-chip-support) and its child pages. For details about the core abstractions, see [Core Components](https://deepwiki.com/tenstorrent/luwen/2.2-core-components).

## Overview

The Luwen codebase provides several communication mechanisms to interact with Tenstorrent AI accelerator chips:

1.   **AXI (Advanced eXtensible Interface)** - For direct memory-mapped register access
2.   **NOC (Network on Chip)** - For accessing components within the chip via coordinates
3.   **Ethernet** - For inter-chip communication across the system
4.   **ARC Messaging** - For sending commands to the ARC processors on the chips

These interfaces are abstracted through the `HlComms` trait and implemented by various chip-specific implementations.

Sources: [crates/luwen-if/src/interface.rs 8-51](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/interface.rs#L8-L51)[crates/luwen-if/src/arc_msg.rs 5-9](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/arc_msg.rs#L5-L9)

## Interface Abstraction

Communication with Tenstorrent chips is abstracted through interfaces defined in the `luwen-if` crate and implemented in the `luwen-ref` crate. The primary interface trait is `HlComms` (High-Level Communications), which provides methods for different types of chip communication.

Sources: [crates/luwen-if/src/interface.rs 142-326](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/interface.rs#L142-L326)

## AXI Communication

The AXI (Advanced eXtensible Interface) bus provides direct memory-mapped access to chip registers. It's used for accessing control and status registers as well as other memory-mapped resources on the chip.

### AXI Operations

AXI supports both read and write operations of various sizes:

Sources: [crates/luwen-ref/src/lib.rs 286-335](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/lib.rs#L286-L335)[crates/luwen-if/src/interface.rs 40-51](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/interface.rs#L40-L51)

### Implementation Details

The AXI interface is implemented through the PCI device abstraction, which provides direct memory-mapped access to the chip's registers through the PCIe BAR (Base Address Register).

For small transfers (≤ 4 bytes), single register operations are used. For larger transfers, block transfer functions are employed, which may use DMA for efficiency when the size exceeds configured thresholds.

## NOC Communication

The NOC (Network on Chip) interface allows communication with components inside the chip via a coordinate-based addressing scheme. Each component has an (x,y) coordinate and a NOC ID.

### NOC Addressing

Sources: [crates/luwen-ref/src/lib.rs 337-426](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/lib.rs#L337-L426)[crates/luwen-if/src/interface.rs 8-31](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/interface.rs#L8-L31)

### NOC Operations

NOC supports read, write, and broadcast operations:

1.   **NOC Read** - Reads data from a specific (x,y) coordinate and NOC ID
2.   **NOC Write** - Writes data to a specific (x,y) coordinate and NOC ID
3.   **NOC Broadcast** - Broadcasts data to multiple coordinates within a specified range

Each NOC operation uses a TLB (Translation Lookaside Buffer) entry to translate the logical address to a physical address and set up the route for the communication.

Sources: [crates/luwen-ref/src/lib.rs 86-107](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/lib.rs#L86-L107)[crates/luwen-ref/src/lib.rs 227-251](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/lib.rs#L227-L251)[crates/ttkmd-if/src/tlb/mod.rs 88-105](https://github.com/tenstorrent/luwen/blob/55e35616/crates/ttkmd-if/src/tlb/mod.rs#L88-L105)

## Ethernet Communication

The Ethernet interface enables communication between different chips in a multi-chip system. It extends the NOC interface to work across chip boundaries.

### Ethernet Communication Flow

Sources: [crates/luwen-ref/src/lib.rs 427-605](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-ref/src/lib.rs#L427-L605)[crates/luwen-if/src/interface.rs 32-37](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/interface.rs#L32-L37)

### Implementation Details

Ethernet communication is implemented using command queues and DMA buffers for efficient data transfer:

1.   Command queues are set up for sending requests to remote chips
2.   DMA buffers are allocated for storing data during transfers
3.   Requests are packaged with coordinate information for the remote chip
4.   Timeouts are implemented to handle communication failures

For small transfers (≤ 4 bytes), single register operations are used. For larger transfers, block transfer functions employ DMA buffers for efficiency.

## ARC Messaging System

The ARC Messaging system provides a protocol for sending commands to the ARC processors on the chips. It uses a message-based approach with standardized message types and a well-defined protocol.

### ARC Message Structure

Sources: [crates/luwen-if/src/arc_msg.rs 11-143](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/arc_msg.rs#L11-L143)[crates/luwen-if/src/arc_msg.rs 248-252](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/arc_msg.rs#L248-L252)

### ARC Message Protocol

The ARC messaging protocol involves:

1.   Writing message code and arguments to scratch registers
2.   Triggering an interrupt to notify the ARC processor
3.   Optionally waiting for completion
4.   Reading response data from scratch registers

## PCIe DMA Transfers

For efficient data transfer between the host and the chip, Luwen uses PCIe DMA (Direct Memory Access) transfers. These are particularly useful for large data movements.

### DMA Transfer Flow

Sources: [crates/ttkmd-if/src/pci.rs 191-274](https://github.com/tenstorrent/luwen/blob/55e35616/crates/ttkmd-if/src/pci.rs#L191-L274)[crates/ttkmd-if/src/pci.rs 276-365](https://github.com/tenstorrent/luwen/blob/55e35616/crates/ttkmd-if/src/pci.rs#L276-L365)

### DMA Buffer Management

Luwen manages DMA buffers to efficiently transfer data between the host and the chip:

1.   Buffers are allocated using the kernel driver
2.   Physical addresses are tracked for DMA operations
3.   Different buffer sizes can be allocated based on transfer needs
4.   A threshold system determines when to use DMA vs. direct memory access

## Error Handling

Luwen provides a comprehensive error handling system for communication failures.

### Error Types Hierarchy

Sources: [crates/luwen-if/src/error.rs 10-103](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/error.rs#L10-L103)[crates/luwen-if/src/arc_msg.rs 187-223](https://github.com/tenstorrent/luwen/blob/55e35616/crates/luwen-if/src/arc_msg.rs#L187-L223)

Each error type includes detailed information about the failure, including:

1.   Error context (chip ID, address, etc.)
2.   Error type and specific error code
3.   Backtrace for debugging (when available)
4.   Human-readable error messages

This comprehensive error system allows developers to diagnose and resolve communication issues efficiently.

## Communication Interface Summary

The following table summarizes the main communication interfaces in Luwen:

| Interface | Purpose | Key Functions | Data Sizes | Addressing |
| --- | --- | --- | --- | --- |
| AXI | Register access | `axi_read32/write32` `axi_read/write` | 1-4 bytes (single) Multiple bytes (block) | 32-bit addresses |
| NOC | Intra-chip communication | `noc_read/write` `noc_broadcast` | Any size (uses TLB) | (x,y) coordinates + NOC ID |
| Ethernet | Inter-chip communication | `eth_noc_read/write` `eth_noc_broadcast` | Any size (uses DMA) | Ethernet address + (x,y) + NOC ID |
| ARC Messaging | Chip control | `arc_msg` | Fixed message format | Message codes + args |

These interfaces provide a comprehensive set of tools for interacting with Tenstorrent AI accelerator chips at various levels of abstraction.
