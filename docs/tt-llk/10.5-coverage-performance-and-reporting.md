This document describes the infrastructure for collecting code coverage, profiling performance, and reporting test results in the tt-llk CI/CD pipeline. It covers the mechanisms for instrumenting builds with coverage and profiler hooks, collecting data from hardware during test execution, merging results across parallel test splits, and uploading artifacts for analysis.

For information about test execution and orchestration, see [CI/CD Pipeline](/tenstorrent/tt-llk/10-cicd-pipeline). For details on the test framework itself, see [Testing Framework](/tenstorrent/tt-llk/8-testing-framework).

## Coverage Collection System

The tt-llk repository supports comprehensive code coverage collection through GCOV instrumentation, allowing developers to track which LLK operations are exercised by the test suite. Coverage collection is enabled via the `--coverage` flag and integrates with the compilation, execution, and reporting pipeline.

### Coverage Build Configuration

Coverage builds are controlled by the `CoverageBuild` enum and `WITH_COVERAGE` class variable in `TestConfig`:

**Coverage Build Control**

Sources: [tests/python\_tests/helpers/test\_config.py53-56](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L53-L56) [tests/python\_tests/helpers/test\_config.py118](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L118-L118) [tests/python\_tests/helpers/test\_config.py139-140](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L139-L140)

When coverage is enabled, the compilation system adds GCOV instrumentation flags and uses a debug memory layout that allocates additional space for coverage data structures. The `resolve_compile_options` method conditionally adds these flags:

[tests/python\_tests/helpers/test\_config.py546-553](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L546-L553)

The coverage instrumentation produces `.gcda` (coverage data) and `.gcno` (coverage notes) files during execution, which are later processed by `lcov` to generate human-readable coverage reports.

### Runtime Address Allocation

Coverage builds use a different L1 memory address for runtime parameters to avoid conflicts with coverage data structures:

| Build Type | Runtime Address | Symbol |
| --- | --- | --- |
| Normal | `0x20000` | `RUNTIME_ADDRESS_NON_COVERAGE` |
| Coverage | `0x64000` | `RUNTIME_ADDRESS_COVERAGE` |

[tests/python\_tests/helpers/test\_config.py139-140](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L139-L140)

The `write_runtimes_to_L1` method selects the appropriate address based on the `WITH_COVERAGE` flag:

[tests/python\_tests/helpers/test\_config.py479-486](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L479-L486)

This separation ensures that GCOV's internal data structures, which occupy lower memory regions, do not collide with test runtime parameters.

### GCOV Integration and Data Extraction

Coverage data is extracted from the device after test execution using the `pull_coverage_stream_from_tensix` function in `device.py`. The `process_coverage_run_artefacts` function in `TestConfig` orchestrates the complete coverage processing workflow:

**Coverage Data Extraction and Processing Flow**

Sources: [tests/python\_tests/helpers/device.py285-308](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L285-L308) [tests/python\_tests/helpers/test\_config.py1420-1580](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1420-L1580)

The extraction process reads the coverage data stream from L1 memory starting at the `__coverage_start` symbol, which is emitted by GCOV instrumentation. The coverage data is then processed into `.gcda` files and merged using `lcov`:

[tests/python\_tests/helpers/test\_config.py1420-1580](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L1420-L1580)

The `process_coverage_run_artefacts` function:

1. Locates all compiled ELF files in the artifact directory
2. Extracts coverage streams for each TRISC (unpack, math, pack)
3. Writes `.gcda` files alongside `.gcno` files from compilation
4. Invokes `lcov` to generate `.info` files
5. Merges all coverage data into `merged_coverage.info`

### Coverage Merging Workflow

The CI/CD pipeline executes tests in parallel splits, each producing independent coverage files. These must be merged to generate a complete coverage report:

**Coverage Merge Pipeline**

Sources: [.github/workflows/setup-and-test.yml217-275](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L217-L275)

The merge process is implemented in the `merge-coverage` job of the `setup-and-test.yml` workflow:

1. **Download artifacts**: Collects all `code-coverage-{arch}-*` artifacts from parallel runners
2. **Iterative merging**: Uses `lcov -a` to merge coverage data files sequentially
3. **Filtering**: Removes test infrastructure files from coverage (`lcov --remove '*tests/*'`)
4. **Upload**: Sends the merged `.info` file to Codecov for visualization

[.github/workflows/setup-and-test.yml244-265](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L244-L265)

The `process_coverage_run_artefacts` function in `TestConfig` orchestrates local coverage processing after test execution, generating the initial `.info` files for each test split.

## Performance Testing Infrastructure

Performance testing in tt-llk involves collecting hardware performance counters and profiling information during test execution. This system enables developers to analyze throughput, identify bottlenecks, and validate performance optimizations.

### Performance Counter Architecture

Performance counters are memory-mapped data structures in L1 memory that track hardware events during kernel execution. The `TestConfig` class defines a shared buffer layout for all three TRISC threads:

**Performance Counter Memory Layout (TestConfig Constants)**

Sources: [tests/python\_tests/helpers/test\_config.py175-204](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L175-L204)

The performance counter layout uses a single shared buffer accessed by all threads, plus dedicated per-thread profiler buffers:

**Shared Counter Buffer** (all threads use same addresses):

| Constant | Address | Size | Purpose |
| --- | --- | --- | --- |
| `PERF_COUNTERS_BASE_ADDR` | `0x16A000` | - | Base address |
| `PERF_COUNTERS_CONFIG_ADDR` | `0x16A000` | 344 bytes (86 words) | Configuration words |
| `PERF_COUNTERS_DATA_ADDR` | `0x16A158` | 688 bytes (172 words) | Counter data |
| `PERF_COUNTERS_SYNC_CTRL_ADDR` | `0x16A408` | 4 bytes | Sync control word |
| `PERF_COUNTERS_SIZE` | - | 1036 bytes | Total reservation |

[tests/python\_tests/helpers/test\_config.py185-204](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L185-L204)

**Per-Thread Profiler Buffers**:

| Thread | Constant | Address | Size |
| --- | --- | --- | --- |
| Unpack | `THREAD_PERFORMANCE_DATA_BUFFER[0]` | `0x16B000` | `0x400` (1024 bytes) |
| Math | `THREAD_PERFORMANCE_DATA_BUFFER[1]` | `0x16C000` | `0x400` (1024 bytes) |
| Pack | `THREAD_PERFORMANCE_DATA_BUFFER[2]` | `0x16D000` | `0x400` (1024 bytes) |
| Barrier | `TRISC_PROFILER_BARRIER_ADDRESS` | `0x16AFF4` | 4 bytes |

[tests/python\_tests/helpers/test\_config.py173-180](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L173-L180)

The shared buffer design allows all TRISCs to write performance data to the same memory region, while per-thread profiler buffers store thread-specific profiling information.

### Profiler Build Configuration

Profiler builds are separate from normal builds and require distinct shared artifacts because `trisc.cpp` is compiled with the `-DLLK_PROFILER` flag:

**Profiler Build Artifacts**

Sources: [tests/python\_tests/helpers/test\_config.py560-657](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L560-L657)

The `build_shared_artefacts` method maintains separate artifact directories and locks for profiler builds:

[tests/python\_tests/helpers/test\_config.py560-577](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L560-L577)

This separation ensures that profiler instrumentation does not affect normal test execution and prevents artifact caching conflicts.

### Performance Data Collection

Performance tests are marked with the `perf` pytest marker and use the `PerfConfig` class (subclass of `TestConfig`) with profiler builds enabled. The test infrastructure collects performance data through a multi-stage pipeline:

**Performance Test Execution Flow**

Sources: [tests/python\_tests/helpers/perf.py268-424](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/perf.py#L268-L424) [tests/python\_tests/conftest.py284-314](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L284-L314)

**PerfConfig Class Structure**

The `PerfConfig` class extends `TestConfig` with performance-specific functionality:

[tests/python\_tests/helpers/perf.py268-318](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/perf.py#L268-L318)

Key differences from `TestConfig`:

* Creates `run_configs` list with multiple run type variants

**PerfReport Lifecycle**

The module-scoped `perf_report` fixture manages performance data collection:

[tests/python\_tests/conftest.py284-314](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L284-L314)

1. **Data Collection**: `PerfConfig.run()` appends data to `PerfReport`
2. **Raw Dump**: Saves raw CSV with all collected metrics
3. **Post-Processing**: `post_process()` applies `_postprocess_tile_loop()` to normalize tile-loop metrics
4. **Post Dump**: Saves post-processed CSV with normalized values

**Session-Level Merging**

The `combine_perf_reports()` function merges performance data from all workers:

[tests/python\_tests/conftest.py316-328](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L316-L328) [tests/python\_tests/helpers/perf.py197-266](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/perf.py#L197-L266)

Merging process:

1. Finds all CSV files matching `*.gw*.csv` or `*.master*.csv` patterns
2. Extracts unique base names (e.g., `perf_unpack_untilize` from `perf_unpack_untilize.gw0.csv`)
3. Combines regular and post-processed files separately
4. Sorts by all columns for consistent ordering
5. Outputs to `perf_data/{base_name}/` directory

**Workflow Configuration**

Performance test workflow characteristics:

* Uses 5 test splits (higher parallelism for compute-intensive tests)
* Extends timeout to 240 minutes
* Runs on both Wormhole (`tt-ubuntu-2204-n150-stable`) and Blackhole (`tt-ubuntu-2204-p150b-stable`)
* Only triggers when `performance` files change (detected by `detect-changes` job)

## Test Result Validation

Test results are validated using statistical correlation metrics and tolerance-based comparison to account for numerical precision limitations in hardware implementations.

### PCC Calculation

The primary validation metric is Pearson Correlation Coefficient (PCC), implemented in the `calculate_pcc` function:

**PCC Calculation Flow**

Sources: [tests/python\_tests/helpers/utils.py85-148](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L85-L148)

The PCC calculation handles edge cases robustly:

[tests/python\_tests/helpers/utils.py104-147](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L104-L147)

Key features:

* **NaN handling**: Returns 1.0 if both tensors are all NaN, 0.0 if only one is
* **Zero detection**: Returns 0.0 if zero patterns don't match
* **Invalid masking**: Masks out NaN/Inf values before correlation
* **bfloat16 conversion**: Converts to float32 for accurate correlation calculation
* **Masked constant handling**: Returns 1.0 if correlation produces a masked constant

### Tolerance-Based Comparison

The `passed_test` function combines tolerance-based comparison with PCC validation:

**Test Validation Pipeline**

Sources: [tests/python\_tests/helpers/utils.py150-284](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L150-L284)

The validation logic:

[tests/python\_tests/helpers/utils.py158-192](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L158-L192)

**Format-Specific Tolerances:**

| DataFormat | Absolute Tolerance | Relative Tolerance | Notes |
| --- | --- | --- | --- |
| Float16/Float16\_b/Float32 | 0.05 | 0.05 | Standard floating point |
| Int8/Int32/UInt8/UInt16/UInt32 | 0 | 0 | Exact match required |
| Bfp8\_b | 0.1 | 0.2 | Higher tolerance for block FP |
| MxFp8R/MxFp8P | 0.2 | 0.3 | Highest tolerance for MX formats |

**PCC Requirements:**

* Normal tests: PCC > 0.99
* Fused tests (multiple L1-to-L1 iterations): PCC > 0.99^L1\_to\_L1\_iterations
  + Precision loss accumulates across iterations
  + Bfp8\_b format particularly sensitive to precision loss

The function also provides detailed error reporting with color-coded tile visualization when validation fails, showing exactly which elements diverge from golden values.

### Artifact Management

The CI/CD pipeline manages multiple artifact types across parallel test splits, with conditional upload logic to optimize storage:

**Artifact Types and Upload Strategy**

| Artifact Type | Naming Pattern | Upload Condition | Step Name | Retention |
| --- | --- | --- | --- | --- |
| JUnit Reports | `junit-report-{arch}-{group}` | `always()` | Upload JUnit report | Per workflow run |
| Performance JUnit | `perf-junit-report-{arch}-{group}` | `always() && contains(pytest_markers, 'perf')` | Upload JUnit report | Per workflow run |
| Coverage Data (per-split) | `code-coverage-{arch}-{group}` | `inputs.coverage` | Upload code coverage for this runner | Until merge job |
| Merged Coverage | `code_coverage_for_{arch}` | After merge job | Upload merged coverage .info file | Per workflow run |
| Performance Data | `perf-data-{arch}-{group}` | `always() && contains(pytest_markers, 'perf') && hashFiles('perf_data/**') != ''` | Upload performance data | Per workflow run |
| Test Logs | `test-logs-{arch}-{group}` | `failure()` | Upload test logs on failure | Per workflow run |

### Coverage Upload to Codecov

The merged coverage data is uploaded to Codecov for visualization, trend analysis, and PR annotations:

**Coverage Upload Pipeline**

Sources: [.github/workflows/setup-and-test.yml217-275](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L217-L275)

**Merge Job Configuration**

[.github/workflows/setup-and-test.yml217-236](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L217-L236)

The job runs conditionally:

* `needs: setup-and-test` - Waits for all test splits to complete
* `if: inputs.coverage == true` - Only executes when coverage is enabled
* `runs-on: ubuntu-latest` - Uses lightweight runner (no hardware needed)
* `container: ${{ inputs.docker_image }}` - Ensures consistent tooling

**LCOV Merging Logic**

[.github/workflows/setup-and-test.yml244-260](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L244-L260)

The merge process:

1. Finds all downloaded `.info` files recursively
2. Copies first file as base: `cp "$first" merged.info`
3. Iteratively merges remaining files: `lcov -q -a merged.info -a "$f" -o merged.info`
4. Filters test infrastructure: `lcov -q --remove merged.info '*tests/*'`
5. Outputs final file: `code_coverage_for_{arch}.info`

**Codecov Upload**

[.github/workflows/setup-and-test.yml267-275](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L267-L275)

Codecov integration:

* **Authentication**: Requires `CODECOV_TOKEN` secret from repository settings
* **File specification**: `files: ./code_coverage_for_{arch}.info`
* **Report naming**: `name: coverage-report-{arch}` (separates Wormhole/Blackhole)
* **Verbose logging**: `verbose: true` for debugging upload issues
* **Architecture tracking**: Separate reports enable architecture-specific coverage analysis

The upload provides:

* PR comments with coverage changes
* GitHub check annotations for uncovered lines
* Historical trend tracking
* Coverage badges for README
