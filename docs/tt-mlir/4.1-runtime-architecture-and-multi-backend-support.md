## Purpose and Scope

This document describes the runtime architecture of `tt-mlir`, focusing on its multi-backend dispatch system and runtime mode selection. The runtime provides a unified API that supports multiple device backends (TTNN and TTMetal) and host execution modes (Local and Distributed), allowing the same compiled programs to execute across different hardware configurations and runtime environments.

The runtime acts as the execution engine for Flatbuffer binaries generated by the `ttmlir-translate` tool. It manages device lifecycles, tensor memory, and program orchestration.

* * *

## Runtime Selection Architecture

The `tt-mlir` runtime supports multiple execution modes through a two-dimensional selection mechanism: **device runtime** (which hardware abstraction layer to use) and **host runtime** (how to execute across one or more host machines).

### Runtime Enumeration Types

The system defines specific enumerations to identify the active execution environment.

Title: Runtime Selection Enumerations

**Runtime Enumeration Types**

| Runtime Type | Values | Purpose |
| --- | --- | --- |
| `DeviceRuntime` | `TTNN`, `TTMetal`, `Disabled` | Selects which device backend to use for computation. |
| `HostRuntime` | `Local`, `Distributed` | Selects single-process or multi-process execution mode. |

## Multi-Backend Dispatch System

The runtime uses a macro-based dispatch system to route API calls to the appropriate backend implementation based on the current runtime selection stored in `RuntimeContext`.

### Dispatch Mechanism

The `DISPATCH_TO_CURRENT_RUNTIME` macro is the core of the routing logic. It performs a nested switch on the host and device runtimes.

Title: API Dispatch Logic

## Backend Implementations

### TTNN Backend

The TTNN backend leverages the `ttnn` library for high-level neural network execution. It manages tensor life-cycles, layout conversions, and program orchestration.

**Key Functions:**

*   `createOwnedTTNNTensor`: Creates tensors on the host, handling data type casting for unsupported types [runtime/lib/ttnn/runtime.cpp 43-97](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L43-L97)
*   `toHostSingleTensor`: Transfers device tensors to host, handling optional untilization and blocking/non-blocking behavior [runtime/lib/ttnn/runtime.cpp 99-170](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/runtime.cpp#L99-L170)
*   `ProgramExecutor`: Orchestrates the execution of operations defined in a TTNN Flatbuffer [runtime/lib/ttnn/program_executor.cpp 131-162](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L131-L162)

## Distributed Runtime Architecture

The distributed runtime enables multi-process execution across one or more host machines. It uses a Controller/Worker architecture to orchestrate execution.

### Controller and Worker Logic

*   **Controller:** Responsible for dispatching commands to workers. It manages the lifecycle of a distributed job.
*   **Worker:** Executes commands received from the controller. The command executor on the worker side interprets Flatbuffer commands and executes the corresponding runtime operations.

### Host Runtime Modes

The system supports different distributed modes:

*   `LocalSubprocess`: Single host, multiple processes (useful for debugging) [runtime/include/tt/runtime/types.h 48-51](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L48-L51)
*   `MultiProcess`: Multiple hosts/processes, typically managed via MPI [runtime/include/tt/runtime/types.h 53-56](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L53-L56)

## Type System and Runtime Checking

The runtime employs a "Runtime Checked" object system to ensure that objects (Tensors, Devices, Events) created by one backend are not accidentally used by another without explicit conversion.

Title: Type Hierarchy and Safety

**Type Safety Implementation:**

*   `RuntimeCheckedObjectImpl` stores an `associatedRuntime` field [runtime/include/tt/runtime/types.h 103](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L103-L103)
*   The `as<T>(DeviceRuntime)` method asserts that the requested runtime matches the object's associated runtime before casting the underlying `void*` handle [runtime/include/tt/runtime/types.h 121-125](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L121-L125)

## Binary Format and Compatibility

Binaries are serialized using Flatbuffers. The runtime provides utilities to inspect these binaries and determine compatibility.

*   **Binary Identity:** Each loaded binary is assigned a unique `binaryId` via an atomic counter [runtime/lib/binary.cpp 30-36](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L30-L36)[runtime/lib/binary.cpp 56-59](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L56-L59)
*   **Compatibility Check:**`setCompatibleDeviceRuntime` inspects the binary's Flatbuffer identifier to automatically set the `DeviceRuntime` to `TTNN` or `TTMetal`[runtime/lib/runtime.cpp 184-201](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/runtime.cpp#L184-L201)
*   **Schema Validation:** The runtime checks versioning and schema hashes to ensure the binary matches the runtime expectations [runtime/lib/binary.cpp 133-146](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L133-L146)
