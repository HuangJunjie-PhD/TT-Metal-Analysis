## @ttl.operation / @ttl.kernel

The `@ttl.operation` decorator transforms a Python function into a tt-lang kernel. It defines the computational grid and serves as the entry point for the `TTLGenericCompiler`[python/ttl/_src/ttl_ast.py 128-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L128-L130)

### Signature

### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `grid` | `Tuple[int, int]` | `str` | Defines the space of nodes (Tensix Cores). Must be a 2D tuple `(cols, rows)` or `"auto"`. Only 2D grids are supported; 3D grids or non-2D shapes cause a `ValueError` during MLIR type building [python/ttl/_src/ttl_ast.py 75-76](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L75-L76) |

### Execution Context Functions

Inside a kernel (and its nested threads), the following context functions are available via `ttl`:

*   **`ttl.grid_size(dims: int)`**: Returns the current grid dimensions as a tuple or integer depending on `dims`[docs/sphinx/specs/TTLangSpecification.md 107-114](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L107-L114)
*   **`ttl.node(dims: int)`**: Returns zero-based coordinates of the current node (core) [examples/elementwise-tutorial/step_3_multinode.py 93-96](https://github.com/tenstorrent/tt-lang/blob/d76e6233/examples/elementwise-tutorial/step_3_multinode.py#L93-L96)

### Implementation Details

The decorator triggers the registration of thread functions into a `_thread_registry` via `_register_thread`[python/ttl/ttl_api.py 102-104](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L102-L104) During compilation, the `TTLGenericCompiler` visits the AST to generate MLIR `ttl` dialect operations [python/ttl/_src/ttl_ast.py 128-145](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L128-L145) It handles `RankedTensorType` generation with `TTLLayoutAttr` encoding, ensuring tensors have at least 2 dimensions [python/ttl/_src/ttl_ast.py 69-83](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L69-L83)

## @ttl.compute

Defines the compute thread that executes on the TRISC processor. It is responsible for math operations on tiles within the Tensix core.

### Signature

### Key Operations and Lowering

*   **Math Functions**: Supports elementwise operations (e.g., `ttl.math.abs`, `ttl.math.neg`). These are lowered from TTL to `ttl.compute` blocks during the `ConvertTTLToCompute` pass [docs/sphinx/specs/TTLangSpecification.md 46](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L46-L46)[lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 29](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L29-L29)
*   **Accumulation**: The `TTLAnnotateL1AccLoops` pass detects user-written loops containing `ttl.store` ops with the `accumulate` attribute and marks them for hardware L1 accumulation [include/ttlang/Dialect/TTL/Passes.td 174-180](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L174-L180)
*   **DST Management**: The `TTLAssignDST` pass performs register allocation [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L36)
*   **Subblocking**: `TTLSubblockComputeForDST` optimizes compute for DST register capacity by partitioning work into subblocks [include/ttlang/Dialect/TTL/Passes.td 223-230](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L223-L230)

## @ttl.datamovement

Defines threads for moving data between DRAM/L1 and Circular Buffers (Dataflow Buffers). These execute on the BRISC (writer) and NCRISC (reader) processors.

### Signature

### Key Operations

*   **`ttl.copy(src, dst)`**: Initiates data transfer between tensors and blocks. The `TTLInsertCopyWait` pass ensures every copy has a corresponding wait if not explicitly provided [include/ttlang/Dialect/TTL/Passes.td 108-115](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L108-L115)
*   **Synchronization**: The `TTLInsertCBSync` pass automatically inserts missing `cb_push`/`cb_pop` for unmatched `cb_reserve`/`cb_wait`[include/ttlang/Dialect/TTL/Passes.td 6-23](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L6-L23)
*   **Coalescing**: `TTLCoalesceDFBAcquires` collapses consecutive same-DFB acquires into multi-tile operations for efficiency [include/ttlang/Dialect/TTL/Passes.td 28-48](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L28-L48)
*   **Intermediate Buffers**: `TTLInsertIntermediateDFBs` materializes fused expression chains to L1 via compiler-allocated DFBs at fusion split points [lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp 9-13](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/TTLInsertIntermediateDFBs.cpp#L9-L13)

### Dataflow Buffer Synchronization

The interaction between `datamovement` and `compute` is governed by the state of the `CircularBuffer` (DataflowBuffer).

| Operation | Thread Type | Effect |
| --- | --- | --- |
| `reserve()` | Producer | Acquires empty slots in CB [include/ttlang/Dialect/TTL/Passes.td 10-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L10-L12) |
| `push()` | Producer | Signals data is ready for consumer [include/ttlang/Dialect/TTL/Passes.td 10-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L10-L12) |
| `wait()` | Consumer | Blocks until data is available in CB [include/ttlang/Dialect/TTL/Passes.td 10-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L10-L12) |
| `pop()` | Consumer | Signals data has been consumed, freeing space [include/ttlang/Dialect/TTL/Passes.td 10-12](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/Passes.td#L10-L12) |

## Auto-Profiling and Context

When `TTLANG_AUTO_PROFILE=1` is set, the decorators and compiler automatically instrument the code for performance analysis [python/ttl/ttl_api.py 53-61](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L53-L61)

*   **`_run_profiling_pipeline`**: Called after kernel execution to read device profiler data via `ttnn.ReadDeviceProfiler` and display a report mapping cycles to Python source lines [python/ttl/ttl_api.py 166-206](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L166-L206)
*   **Line Mapping**: The system uses `get_line_mapper()` to correlate hardware cycles back to the original Python source file and line number [python/ttl/ttl_api.py 228-229](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L228-L229)
*   **Cache Management**: The system uses `_make_cache_key` to avoid redundant compilations, incorporating tensor shapes, dtypes, mesh configuration, and `CompilerOptions`[python/ttl/ttl_api.py 133-158](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L133-L158)
