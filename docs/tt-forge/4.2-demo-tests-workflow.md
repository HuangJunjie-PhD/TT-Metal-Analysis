## Purpose and Scope

The `demo-tests.yml` workflow is a comprehensive validation pipeline designed to ensure the functional correctness of real-world model demonstrations across the TT-Forge ecosystem. Unlike basic smoke tests, these tests execute complete end-to-end scripts—including model loading, compilation, and inference—on Tenstorrent hardware (N150/P150). It serves as a bridge between core compiler validation and user-facing application reliability.

## Workflow Architecture and Data Flow

The workflow utilizes a matrix-driven approach, where a central JSON configuration defines the test suite, and a filtering script selects specific tests based on project focus or hardware availability.

## Matrix Configuration and Selection

The workflow dynamically generates its execution strategy by parsing `models-matrix.json`. This file categorizes tests by project (e.g., `tt-forge-onnx`, `tt-xla`, `tt-torch`) and specifies their execution requirements.

### Code Entity Association: Matrix Structure

### Matrix Selection Logic

The `set-matrix` job invokes the filter script with the following inputs:

1.   **Project Filter:** Limits execution to a specific frontend (e.g., `tt-forge-onnx`).
2.   **Test Filter:** A string match to run a subset of tests.
3.   **Runner Flag:**`--sh-runner` is used to format outputs for GitHub Actions.

## Error Handling and Reporting

The workflow implements a multi-stage reporting system to handle failures and collect telemetry.

### Failure Notification Pipeline

1.   **fail-notify Job:** Evaluates the outcome of all matrix instances. It checks if the branch is `main` and if the run is a draft [.github/workflows/demo-tests.yml 133-172](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/demo-tests.yml#L133-L172)
2.   **fail-send-msg Job:** If a failure occurs on the `main` branch (and it is not a draft), a notification is sent to a dedicated Slack channel (`C088QN7E0R3`) [.github/workflows/demo-tests.yml 173-190](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/demo-tests.yml#L173-L190)
3.   **produce-data Job:** Triggers the `produce_data.yml` workflow to collect CI/CD metrics and performance data for long-term tracking [.github/workflows/demo-tests.yml 192-213](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/%20.github/workflows/demo-tests.yml#L192-L213)

### Data Collection Integration

The workflow calls `produce_data.yml` using the `trigger-workflow` action. This process archives logs and results to an internal SFTP server for analysis.

| Variable | Source | Purpose |
| --- | --- | --- |
| `parent_run_id` | Input | Links child demo tests to a parent release or CI run. |
| `version_tag` | Docker Image | Extracts the version from the image tag for reporting. |
| `failed` | `alls-green` | Aggregated status of the matrix. |

## Comparison with Basic Tests

While `basic-tests.yml` focuses on core frontend sanity checks (e.g., `basic_tests/tt-xla/demo_test.py`), `demo-tests.yml` validates the actual user experience by running the scripts found in the `demos/` directory.

| Feature | Basic Tests (`basic-tests.yml`) | Demo Tests (`demo-tests.yml`) |
| --- | --- | --- |
| **Scope** | Internal smoke tests | End-to-end model scripts |
| **Config** | Hardcoded logic in YAML | Matrix-based (`models-matrix.json`) |
| **Project Support** | `tt-xla`, `tt-forge` | `tt-forge-onnx`, `tt-xla`, `tt-torch` |
| **Complexity** | Low (single file) | High (multi-file, `requirements.txt`) |
