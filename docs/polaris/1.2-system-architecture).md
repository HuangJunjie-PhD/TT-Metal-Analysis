This document provides a comprehensive overview of the Polaris system architecture, describing the core components, their interactions, and how they work together to provide AI workload performance simulation capabilities. The architecture is designed around a modular framework that separates concerns between workload definition, hardware modeling, simulation execution, and performance analysis.

For information about specific workload models and examples, see [Workload Models and Examples](https://deepwiki.com/tenstorrent/polaris/4-workload-models-and-examples). For details about performance analysis tools and benchmarking, see [Performance Analysis and Benchmarking](https://deepwiki.com/tenstorrent/polaris/3-performance-analysis-and-benchmarking).

The Polaris system follows a layered architecture with clear separation between the simulation orchestration layer, the core simulation framework, and the configuration management system.

## Main Orchestrator Component

The `polaris.py` module serves as the central orchestrator that coordinates all simulation activities. It provides a command-line interface and manages the complete simulation workflow from configuration parsing to results generation.

### Orchestrator Architecture

**Sources:**[polaris.py 259-306](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polaris.py#L259-L306)[polaris.py 322-361](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polaris.py#L322-L361)[polaris.py 405-556](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polaris.py#L405-L556)[polaris.py 559-611](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polaris.py#L559-L611)

### Key Orchestrator Classes

| Class | Purpose | Key Methods |
| --- | --- | --- |
| `RangeArgument` | Manages parameter sweeps for frequency and batch size | `getvals()`, `check()` |
| `ReducedStats` | Aggregates performance statistics across operations | `summarize()` |

The configuration system provides a flexible, YAML-based approach to defining workloads, architectures, and their mappings. It uses Pydantic models for validation and type safety.

## Architecture Scalability and Extensibility

The system is designed with extensibility in mind, allowing for easy addition of new operations, hardware models, and workload types.

### Extension Points

| Extension Point | Interface | Implementation Examples |
| --- | --- | --- |
| New Operations | `SimOp` base class | `ConvOp`, `MatMulOp`, `LayerNormalizationOp` |
| New Hardware Models | `IPBlocksModel` configuration | Compute pipes, memory blocks |
| New Workload Types | `WorkloadCfgBlk` base class | `WorkloadTTSIM`, `WorkloadONNX` |
| New Output Formats | `OutputFormat` enum | YAML, JSON, Pickle, CSV |

**Sources:**[ttsim/ops/op.py 205-261](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/ops/op.py#L205-L261)[ttsim/config/simconfig.py 488-506](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/ttsim/config/simconfig.py#L488-L506)[polaris.py 34-48](https://github.com/tenstorrent/polaris/blob/ac6ed4a3/polaris.py#L34-L48)

The architecture provides a robust foundation for AI workload performance simulation with clear separation of concerns, comprehensive configuration management, and flexible extension capabilities.
