## Purpose and Scope

This page documents the multi-chip device model implementation in tt-npe, specifically the `WormholeMultichipDeviceModel` class that enables simulation of NoC traffic across multiple interconnected devices. Multi-chip models extend single-chip device models to support fabric-scale simulations with proper device ID tracking, cross-device routing, and congestion modeling.

For information about the device model interface and single-chip implementations, see [Device Model Interface](https://deepwiki.com/tenstorrent/tt-npe/5.1-device-model-interface), [Wormhole B0 Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.2-wormhole-b0-implementation), and [Blackhole Implementation](https://deepwiki.com/tenstorrent/tt-npe/5.3-blackhole-implementation). For details on topology configuration files used with multi-chip systems, see [Multi-chip Simulation Details](https://deepwiki.com/tenstorrent/tt-npe/7.2-multi-chip-simulation-details).

## Architecture Overview

The multi-chip device model uses a composition pattern, aggregating a single-chip model and extending its behavior to support multiple devices. This design reuses existing single-chip logic while adding multi-device coordination.

**Sources:**[tt_npe/cpp/include/device_models/wormhole_multichip.hpp 17-27](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_multichip.hpp#L17-L27)[tt_npe/cpp/include/device_models/wormhole_b0.hpp 16-26](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L16-L26)[tt_npe/cpp/include/npeDeviceModelIface.hpp 23-70](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeDeviceModelIface.hpp#L23-L70)

## Composition Pattern and Delegation

`WormholeMultichipDeviceModel` aggregates a `WormholeB0DeviceModel` instance and delegates most device-specific queries to it. This enables code reuse while avoiding the complexity of inheritance hierarchies.

| Method | Delegation Strategy |
| --- | --- |
| `getRows()` / `getCols()` | Direct delegation to `_wormhole_b0_model` |
| `getCoreType(coord)` | Direct delegation (device_id in coord is ignored) |
| `getSrcInjectionRate(coord)` | Direct delegation |
| `getSinkAbsorptionRate(coord)` | Direct delegation |
| `getLinkBandwidth(link_id)` | Direct delegation |
| `route(...)` | Delegates to `_wormhole_b0_model.route()`, then translates device IDs |
| `getLinkTypes()` / `getNIUTypes()` | Direct delegation |
| `getTransferBandwidthTable()` | Accessed via `_wormhole_b0_model` |

The key insight is that a multi-chip system consists of multiple identical single-chip grids, each with the same topology and characteristics, differentiated only by their `device_id`.

**Sources:**[tt_npe/cpp/include/device_models/wormhole_multichip.hpp 248-320](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_multichip.hpp#L248-L320)[tt_npe/cpp/include/device_models/wormhole_b0.hpp 18-25](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L18-L25)

## Cross-Device Routing

Multi-chip routing is accomplished through a two-step process:

1.   **Route on device 0**: Delegate to `_wormhole_b0_model.route()` to compute the path as if on device 0
2.   **Translate device IDs**: Use `changeRouteDeviceID()` to substitute the actual device ID into every link

The implementation enforces that routing is only performed within a single device (source and destination must have the same `device_id`). Cross-device routing is handled by the fabric routing layer during workload ingestion (see [Trace Preprocessing and Fabric Routing](https://deepwiki.com/tenstorrent/tt-npe/2.2-trace-preprocessing-and-fabric-routing)).

## Link and NIU Identifier Expansion

Single-chip models create link and NIU identifiers for one device grid. Multi-chip models expand these lookups to cover all devices.

### Link Lookup Population

For an 8-chip system with 12x10 grids and 4 link types per location:

*   Single-chip: 12 × 10 × 4 = 480 link IDs
*   Multi-chip: 8 × 12 × 10 × 4 = 3,840 link IDs

The ordering ensures link IDs are contiguous within each device, facilitating device-specific statistics collection.

### NIU Lookup Population

NIU lookups follow an identical pattern with NIU types (NOC0_SRC, NOC0_SINK, NOC1_SRC, NOC1_SINK):

*   Single-chip: 12 × 10 × 4 = 480 NIU IDs
*   Multi-chip: 8 × 12 × 10 × 4 = 3,840 NIU IDs

**Sources:**[tt_npe/cpp/include/device_models/wormhole_multichip.hpp 29-55](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_multichip.hpp#L29-L55)[tt_npe/cpp/include/device_models/wormhole_b0.hpp 27-49](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L27-L49)

## Congestion Modeling Across Devices

Multi-chip congestion modeling follows the same algorithm as single-chip but operates on expanded link and NIU grids. The model accounts for transfers on all devices simultaneously.

Key differences from single-chip congestion modeling:

1.   **Device-aware NIU lookup**: `getNIUID(device_id, row, col, type)` instead of `getNIUID(row, col, type)`
2.   **Expanded grid sizes**: Demand grids are sized for all devices (3,840 entries vs 480)
3.   **Cross-device isolation**: Transfers on different devices do not interfere (they have distinct link IDs and NIU IDs)

The algorithm uses first-order congestion modeling (single iteration) where bandwidth is derated by the most congested resource along each transfer's path.

**Sources:**[tt_npe/cpp/include/device_models/wormhole_multichip.hpp 90-216](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_multichip.hpp#L90-L216)[tt_npe/cpp/include/device_models/wormhole_b0.hpp 55-189](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp#L55-L189)

## Device State Initialization

Device state objects track runtime congestion state. Multi-chip models initialize device state with expanded dimensions:

The `npeDeviceState` contains:

*   `link_demand_grid`: Track demand on each link
*   `niu_demand_grid`: Track demand at each NIU
*   `multicast_write_link_demand_grid`: Track multicast-specific demand

These grids are reset at the start of each timestep's congestion modeling.

## Topology Configuration

Multi-chip systems use `topology.json` files to define inter-device connectivity. These files are processed during trace preprocessing by `fabric_post_process.py` to elaborate cross-device transfer paths.

### Topology File Structure

### Example: 4-Device Ring Topology

From [tt_npe/workload/T3K-fabric-traces/ring-4dev/topology.json](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/workload/T3K-fabric-traces/ring-4dev/topology.json):

### Topology Components

| Component | Purpose |
| --- | --- |
| `forwarding_channel_pairs` | Defines which ethernet channels on each device are paired for packet forwarding |
| `inter_device_channel_pairs` | Maps direct physical connections between devices |
| `eth_chan_to_coord` | Maps ethernet channel numbers to NoC coordinates on the device grid |

Topology files enable fabric routing algorithms to determine the multi-hop path for transfers between devices. For example, a transfer from device 0 to device 2 in a 4-device ring must be routed through device 3 or device 1, depending on the routing plane.

## Usage and Integration

The multi-chip device model is instantiated by the simulation engine based on workload requirements:

Key integration points:

1.   **Workload ingestion**: `convertNocTracesToNpeWorkload` creates transfer objects with device IDs populated from trace data
2.   **Device model selection**: Engine detects multi-device workloads and instantiates `WormholeMultichipDeviceModel`
3.   **Routing**: Each transfer is routed on its specific device
4.   **Statistics**: Per-device statistics can be collected by filtering based on device ID

The multi-chip model enables accurate simulation of large-scale fabric workloads while maintaining the simplicity and correctness of single-chip routing algorithms.
