This page describes the C++ backend implementation of individual operations in the Forge operation system. It covers the `Op` class interface, the per-op function signatures declared via `DECLARE_OP_INTERFACE`, the `OpType`-based dispatch mechanism in `op.cpp`, and the decomposition epoch model. For the Python-side wrapper functions and the `ForgeOp`/`OpType` dispatch pattern used to construct ops from Python, see [Operation Architecture](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.1-operation-architecture). For specific operation categories (tensor manipulation, convolution, element-wise, etc.), see pages [5.3](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.3-operation-categories-and-apis)–[5.5](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.2-operation-implementation-guide)#5.5).

## The `Op` Class

The central type is the `Op` class, defined in [forge/csrc/ops/op.hpp 152-247](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.hpp#L152-L247) It holds two data members:

*   `type_` — an `OpType` enum value identifying which operation this is
*   `attrs_` — an `Attrs` map (`std::map<std::string, Attr>`) holding named parameters for the operation

The `Attr` variant type accepts: `std::string`, `bool`, `int`, `float`, `std::vector<int>`, and two tuple-vector variants for broadcast specification. This covers all parameterizations needed by the full operation set.

**Attribute access helpers on `Op`:**

| Method | Description |
| --- | --- |
| `has_attr(name)` | Returns true if the attribute key exists |
| `attr_as<T>(name)` | Returns the attribute value, cast to type `T` |
| `set_attr(name, value)` | Sets or overwrites an attribute |
| `get_attr(name)` | Returns the `Attr` variant by key |
| `remove_attr(name)` | Removes an attribute, returns true if it existed |

Sources: [forge/csrc/ops/op.hpp 133-203](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.hpp#L133-L203)

## Required Interface Methods

Every operation must implement four functions. The `Op` class exposes these as public methods; they delegate to the per-op namespace implementations via a `switch` on `type_`.

**Diagram: Op Interface Methods**

Sources: [forge/csrc/ops/op.hpp 204-242](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.hpp#L204-L242)

### `eval`

```
at::Tensor eval(const std::vector<at::Tensor> &tensors) const
```

Computes the forward pass output as a PyTorch `at::Tensor`. The input `tensors` vector contains one tensor per operand. All operations use PyTorch's C++ ATen library for the actual computation. This is used during graph evaluation on the host (e.g., for reference outputs in verification), not during hardware execution.

### `shape`

```
std::tuple<graphlib::Shape, std::vector<graphlib::DimBroadcastTrampoline>>
    shape(const std::vector<std::vector<std::uint32_t>> &inputs) const
```

Infers the output shape from input shapes, without executing any computation. The second return value lists any broadcasting that occurred per dimension, encoded as `(operand_index, dim, broadcast_factor)` tuples. This is called during graph construction to propagate shape information through the graph.

### `backward`

```
tt::graphlib::NodeContext backward(
    tt::autograd::autograd_context &context,
    int operand,
    const std::vector<tt::graphlib::NodeContext> &inputs,
    const tt::graphlib::NodeContext &output,
    const tt::graphlib::NodeContext &gradient) const
```

Computes the gradient with respect to operand `operand`. The `autograd_context` provides access to graph-building utilities. Returns a `NodeContext` representing the computed gradient node in the autograd graph. Operations that are non-differentiable (e.g., comparison ops) typically assert or return a zero gradient in their implementation.

### Decomposition Methods

Decomposition expands a single composite op into a subgraph of primitive ops during specific compilation phases. There are three epochs:

| Method | `DecomposeEpoch` | When it runs |
| --- | --- | --- |
| `decompose_initial` | `Initial` | Before any optimization passes |
| `decompose_post_optimize` | `PostOptimize` | After graph optimization |
| `decompose_post_autograd` | `PostAutograd` | After autograd graph construction |

The template method `decompose<epoch>()` dispatches to the appropriate epoch method [forge/csrc/ops/op.cpp 527-540](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L527-L540) Most ops return early in all three epochs; only composite ops like `AvgPool1d`, `Broadcast`, `LogSoftmax`, `ReduceAvg`, `Resize2d`, and others provide non-trivial decompositions.

Sources: [forge/csrc/ops/op.cpp 526-633](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L526-L633)[forge/csrc/ops/op.hpp 230-241](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.hpp#L230-L241)

## The `DECLARE_OP_INTERFACE` Macro

Each operation lives in its own C++ namespace (e.g., `tt::ops::abs`, `tt::ops::conv_2d`) and its own `.cpp` file. The `DECLARE_OP_INTERFACE` macro in [forge/csrc/ops/op_interface.hpp 39-65](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op_interface.hpp#L39-L65) declares the six required functions for a given namespace:

```
DECLARE_OP_INTERFACE(abs)
```

expands to declarations for `abs::eval`, `abs::shape`, `abs::backward`, `abs::decompose_initial`, `abs::decompose_post_optimize`, `abs::decompose_post_autograd`, and `abs::initial_flops_estimate` — all accepting `const Op &op` as their first parameter.

This pattern means:

*   Every op implementation file is self-contained.
*   Adding a new op requires providing all six functions in a new namespace.
*   The `Op` class dispatch switch in `op.cpp` is the only place that connects an `OpType` enum value to its implementation namespace.

Sources: [forge/csrc/ops/op_interface.hpp 39-156](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op_interface.hpp#L39-L156)

## The Dispatch Mechanism

`Op::eval()`, `Op::shape()`, `Op::backward()`, and the three `decompose_*` methods all use the same pattern: a `switch (type_)` over the `OpType` enum that delegates to the appropriate per-op namespace function [forge/csrc/ops/op.cpp 243-332](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L243-L332)

The same structure is repeated independently for `shape()`, `backward()`, `decompose_initial()`, `decompose_post_optimize()`, and `decompose_post_autograd()`.

## Op Name ↔ OpType Mapping

Two singleton helper classes handle bidirectional string↔enum conversion:

*   `OpTypeToString` — maps `OpType` → `std::string` (e.g., `OpType::Conv2d` → `"conv2d"`)
*   `StringToOpType` — maps `std::string` → `OpType` (e.g., `"conv2d"` → `OpType::Conv2d`)

These are constructed once as static instances `type_to_string` and `string_to_type` at file scope [forge/csrc/ops/op.cpp 232-233](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L232-L233) The string names match the names used in the Python-side `ForgeOp` wrappers and in the `lowering_handler_map` in the MLIR lowering stage.

The `Op` class also has a string constructor: `Op(const std::string &op_name, Attrs attrs)` — this uses `string_to_type` to look up the `OpType` from a name string [forge/csrc/ops/op.cpp 235](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L235-L235)

## Op Classification Helpers

The `Op` class provides four boolean classification methods used by the graph optimization passes:

| Method | Returns true when |
| --- | --- |
| `is_tm()` | Op is a tensor-manipulation op (Transpose, Reshape, etc.) |
| `is_eltwise()` | Op is any element-wise op |
| `is_eltwise_unary()` | Op is a unary element-wise op |
| `is_eltwise_binary()` | Op is a binary element-wise op |
| `is_eltwise_nary()` | Op is an n-ary element-wise op |

These are defined in [forge/csrc/ops/op.hpp 221-226](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.hpp#L221-L226) and implemented in `op_common.cpp`. They are used by graph optimization passes (see [Graph Optimization Passes](https://deepwiki.com/tenstorrent/tt-forge-onnx/3.2-tvm-frontend-and-relay-ir)) to determine whether commutation rules apply.

## pybind11 Exposure

The `Op` class and `OpType` enum are exposed to Python via pybind11 in [forge/csrc/ops/python_bindings.cpp 19-125](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/python_bindings.cpp#L19-L125) The exposed interface is:

*   `Op.__init__(op_name: str, attrs: dict)` — constructs by name string
*   `Op.__init__(type: OpType, attrs: dict)` — constructs by enum value
*   `Op.eval(tensors)` — forward evaluation
*   `Op.shape(inputs)` — shape inference
*   `Op.type()` — returns the `OpType`
*   `Op.attrs()` — returns the attributes dict
*   `Op.__getattr__(name)` — attribute lookup by name
*   `Op.__setattr__(name, value)` — attribute assignment by name

The `OpType` enum values are exported with `.export_values()`, making them available as `forge._C.ops.OpType.Abs`, etc. Python wrapper functions in `forge/forge/op/` import `OpType` from `forge._C.ops` and pass it directly to `ForgeOp`[forge/forge/op/eltwise_unary.py 8](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/op/eltwise_unary.py#L8-L8)

Sources: [forge/csrc/ops/python_bindings.cpp 1-127](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/python_bindings.cpp#L1-L127)[forge/forge/op/eltwise_unary.py 1-47](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/op/eltwise_unary.py#L1-L47)
