# Contribution Process

Relevant source files
*   [CONTRIBUTING.md](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1)

This page explains the unique contribution model used by the ttsim project. Unlike typical open-source projects, ttsim does not accept pull requests. Instead, contributions flow through GitHub issues that inform internal development. This page covers the rationale behind this model, the relationship between public and internal repositories, and how community feedback is incorporated into releases.

For specific guidance on filing bug reports and feature requests, see [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues). For security vulnerabilities, see [Security Vulnerabilities](https://deepwiki.com/tenstorrent/ttsim/5.3-security-vulnerabilities).

**Purpose and Scope**: This document explains the architectural reasons for the issues-only contribution model, describes the code flow between internal and public repositories, and sets expectations for contributors about how their feedback is processed.

* * *

## Issues-Only Contribution Model

The ttsim project uses an **issues-only contribution model**. Community members can contribute through GitHub issues, but pull requests are not accepted.

| Contribution Type | Accepted? | Channel |
| --- | --- | --- |
| Bug reports | ✓ Yes | GitHub Issues |
| Feature requests | ✓ Yes | GitHub Issues |
| Questions | ✓ Yes | GitHub Issues |
| General feedback | ✓ Yes | GitHub Issues |
| Pull requests | ✗ No | Not accepted |
| Code patches | ✗ No | Submit as issue discussion instead |

This policy is documented in [CONTRIBUTING.md 31-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L31-L34) and applies both to the current binary-only releases and to future source code releases.

**Sources**: [CONTRIBUTING.md 1-41](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L1-L41)

## Repository Architecture

### Public and Internal Repository Relationship

The public ttsim repository at `github.com/tenstorrent/ttsim` is a **filtered export** from an internal Tenstorrent repository. This architectural decision has fundamental implications for the contribution model.

**Diagram: Repository Architecture and Code Flow**

The key architectural properties are:

1.   **Source of Truth**: The internal repository is the authoritative source for all code
2.   **One-Way Flow**: Code flows from internal → public only; there is no reverse integration path
3.   **Filtered Export**: The public view is derived from the internal repository through a filtering process that may exclude proprietary components, internal tooling, or unreleased features
4.   **Binary Delivery**: Current releases provide compiled binaries (`libttsim_wh.so`, `libttsim_bh.so`)
5.   **Future Source Delivery**: Planned source code releases will also be filtered exports, maintaining the one-way flow

**Sources**: [CONTRIBUTING.md 5](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L5-L5)[CONTRIBUTING.md 31-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L31-L34)

## Why Pull Requests Are Not Accepted

The technical reason pull requests cannot be accepted is architectural, not policy-based:

**Diagram: Contribution Model Comparison**

Since the public repository is a **derived artifact** rather than the source of truth, there is no integration path for external pull requests. Even if a PR were submitted to the public repository:

1.   The changes would not exist in the internal repository (source of truth)
2.   The next filtered export would overwrite any changes made to the public repository
3.   Manual porting of changes from public → internal would create maintenance overhead and potential conflicts

As stated in [CONTRIBUTING.md 32](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L32-L32) this policy remains in place even after source code is released because "the public source will be a filtered version of our internal repository that doesn't support the standard PR workflow."

**Sources**: [CONTRIBUTING.md 31-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L31-L34)

## How Community Feedback Is Handled

While direct code contributions are not possible, community feedback through issues is actively encouraged and incorporated into the development process.

**Diagram: Community Feedback Processing Workflow**

### Feedback Processing Steps

1.   **Submission**: Contributors file issues through GitHub (see [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues))
2.   **Deduplication**: Contributors should search existing issues per [CONTRIBUTING.md 28-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L28-L29)
3.   **Triage**: Internal team reviews issues to assess validity, priority, and scope
4.   **Implementation**: Accepted suggestions are implemented in the internal repository
5.   **Testing**: Changes are validated against the complete internal test suite
6.   **Release**: Changes are included in the next filtered export and binary release
7.   **Feedback Loop**: Issues are updated or closed when changes are released

**Required Information**: When filing issues, contributors should include per [CONTRIBUTING.md 22-26](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L22-L26):

*   Clear description of the problem or request
*   Steps to reproduce (for bugs)
*   Expected vs actual behavior
*   System information and `ttsim` version

**Sources**: [CONTRIBUTING.md 13-29](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L13-L29)

## Contributor Expectations

### What Contributors Can Do

| Action | Method | Response |
| --- | --- | --- |
| Report bugs | File GitHub issue | Triaged by internal team |
| Request features | File GitHub issue | Evaluated for roadmap inclusion |
| Ask questions | File GitHub issue | Answered by team/community |
| Provide feedback | File GitHub issue | Considered in planning |
| Search existing issues | GitHub search | Before filing new issues |
| Participate in discussions | Comment on issues | Community and team engagement |

### What Contributors Should Not Expect

| Action | Status | Reason |
| --- | --- | --- |
| Submit pull requests | ✗ Not accepted | Filtered export architecture |
| Directly modify source | ✗ Not possible | Internal repository is source of truth |
| Immediate implementation | ✗ Not guaranteed | Depends on priority and resources |
| External commit access | ✗ Not available | Single internal development team |

**Sources**: [CONTRIBUTING.md 13-34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L13-L34)

## Future Source Code Release

The ttsim project plans to release source code in the future under the Apache License 2.0, as stated in [CONTRIBUTING.md 5](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L5-L5) However, the contribution model will remain unchanged.

### Post-Source-Release Model

**Diagram: Contribution Model After Source Release**

### Benefits of Source Release

Even without accepting pull requests, source code release provides significant benefits:

| Benefit | Description |
| --- | --- |
| Transparency | Full visibility into simulator implementation |
| Local builds | Ability to compile custom versions for debugging |
| Deep debugging | Step through simulator code when investigating issues |
| Educational | Learn implementation techniques and architecture |
| Better bug reports | More detailed issue reports with specific code references |
| Code-based discussions | Issues can reference specific files, functions, and line numbers |

As noted in [CONTRIBUTING.md 34](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L34-L34) code-related suggestions can be discussed through issues, where contributors can reference specific implementation details once source code is available.

## License Agreement

By contributing to ttsim through issues or other channels, contributors agree that their contributions will be licensed under the Apache License 2.0, as stated in [CONTRIBUTING.md 39-40](https://github.com/tenstorrent/ttsim/blob/67b1733e/CONTRIBUTING.md?plain=1#L39-L40)

This means:

*   Feedback, suggestions, and ideas submitted through issues are available for incorporation into ttsim
*   Contributors do not retain proprietary rights over suggestions
*   The Apache License 2.0 governs usage of resulting code

For complete licensing details, see [Licensing](https://deepwiki.com/tenstorrent/ttsim/6.2-licensing).

## Summary

The ttsim contribution model is architecturally determined by the repository structure:

| Aspect | Detail |
| --- | --- |
| **Model Type** | Issues-only (no pull requests) |
| **Public Repository** | Filtered export from internal repository |
| **Code Flow** | Internal → Public only (one-way) |
| **Community Input** | GitHub issues (bugs, features, questions, feedback) |
| **Processing** | Internal team triages, implements, and releases |
| **Future Plans** | Source code release, but same contribution model |
| **License** | Apache License 2.0 |

This model ensures:

*   Clean separation between internal development and public distribution
*   Consistent code quality through internal validation
*   Clear communication channel through GitHub issues
*   Ability to filter proprietary or unreleased components

For practical guidance on filing effective issues, proceed to [Reporting Issues](https://deepwiki.com/tenstorrent/ttsim/5.2-reporting-issues).
