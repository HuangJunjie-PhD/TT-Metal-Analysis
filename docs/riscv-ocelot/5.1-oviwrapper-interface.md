The OviWrapper interface provides the critical integration point between the scalar BOOM core and the vector processing unit (VPU) in the RISC-V Ocelot processor. It functions as a translation layer that manages instruction flow, data movement, and synchronization between the scalar and vector domains. This document details the architecture, interfaces, and operational flow of the OviWrapper component.

For information about the internal architecture of the Vector Processing Unit itself, see [Vector Processing Unit](https://deepwiki.com/tenstorrent/riscv-ocelot/5.2-vector-processing-unit).

## Architecture Overview

The OviWrapper sits between the scalar core's execution units and the vector hardware implementation (`tt_vpu_ovi`). It manages the flow of vector instructions from the core, coordinates memory operations with the Load/Store Unit (LSU), and handles the completion of vector operations back to the core.

## Vector Processing Unit Interface

The OviWrapper interfaces with the `tt_vpu_ovi` blackbox module through a defined set of signals:

| Signal Group | Direction | Description |
| --- | --- | --- |
| Issue Interface | Input | Sends instructions and operands to VPU |
| Completion Interface | Output | Receives completed operation results |
| Memory Sync Interface | Input/Output | Coordinates memory operations |
| Load/Store Interface | Input/Output | Manages vector memory data transfer |
| Vector Register Updates | Output | Debug signals for vector register writes |

### Detailed Signal Listing:

```
- issue_valid, issue_inst, issue_sb_id, issue_scalar_opnd: Issue new instructions to VPU
- issue_vcsr, issue_vcsr_lmulb2: Vector configuration information
- completed_valid, completed_sb_id, completed_dest_reg: Result data from VPU
- completed_fflags: Floating-point flags from operations
- memop_sync_start/end: Synchronization for memory operations
- store_valid/data, load_valid/data: Vector memory data transfer
- mask_idx signals: Vector mask handling for predicated operations
```

## Operational Flow

### Vector Instruction Processing

When a vector instruction is issued from the core to the OviWrapper:

1.   The instruction enters through the `io.req` interface
2.   It's assigned a scoreboard ID (`next_sb_id`) and stored in the scoreboard
3.   The instruction is issued to the VPU via `issue_valid` and related signals
4.   When the VPU completes the operation, it asserts `completed_valid` with the matching `completed_sb_id`
5.   The OviWrapper prepares the response with result data and sends it back through `io.resp`

Sources: [src/main/scala/exu/ovi.scala 47-66](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/ovi.scala#L47-L66)[src/main/scala/exu/ovi.scala 478-507](https://github.com/tenstorrent/riscv-ocelot/blob/920fd904/src/main/scala/exu/ovi.scala#L478-L507)

### Vector Load/Store Operations

Vector memory operations require coordination with the LSU:

1.   Vector load/store instructions are queued in the `vLSIQueue`
2.   The OviWrapper manages address generation, data buffering, and sequencing
3.   Memory requests are sent through the `VGenIO` interface to the LSU
4.   For loads, returned data is processed and sent to the VPU
5.   For stores, data from the VPU is buffered and sent to memory
6.   Memory synchronization signals coordinate the process between components

## Integration with BOOM Core

The OviWrapper is integrated into the BOOM core through several components:

1.   It is instantiated as part of a vector execution unit when `usingVector` is enabled
2.   The execution unit framework connects it to the processor pipeline
3.   Its `vGenIO` interface is connected to the LSU to handle memory operations
4.   Vector CSR settings from the core are passed to configure vector operations

## Implementation Considerations

Several important aspects of the OviWrapper implementation are worth noting:

### Scoreboarding and Completion Tracking

The OviWrapper maintains a scoreboard of 32 entries to track in-flight vector operations. This allows the VPU to process multiple operations concurrently and complete them out-of-order while maintaining proper dependencies.

### Memory Synchronization

Memory operations use explicit synchronization signals between the OviWrapper and VPU:

*   `memop_sync_start`: Signals the start of a memory operation sequence
*   `memop_sync_end`: Signals the completion of a memory operation sequence

### Fake Load Entries

For masked-off elements in vector loads, the OviWrapper can generate "fake" load entries that provide predetermined data without actually performing memory accesses. This optimization improves performance for predicated vector operations.

### Vector Configuration

The OviWrapper receives vector configuration information from the CSR file, including:

*   `vconfig`: Vector length and other configuration parameters
*   `vxrm`: Vector rounding mode settings
*   `fcsr_rm`: Floating-point rounding mode

## Conclusion

The OviWrapper serves as the critical interface between the scalar BOOM core and the vector processing unit. It handles instruction flow, memory operations, and completion tracking, enabling efficient vector computation while maintaining integration with the out-of-order core architecture. The modular design with specialized components for address generation, data buffering, and memory synchronization allows for efficient handling of complex vector operations.
