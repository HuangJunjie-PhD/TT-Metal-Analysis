## Purpose and Scope

This document describes GCC's middle-end transformation passes, which convert and optimize the intermediate representation of compiled code. These passes operate on GIMPLE (tree-level) and RTL (register transfer language) representations, performing both intraprocedural and interprocedural optimizations. The pass infrastructure manages the sequencing and execution of hundreds of individual optimization passes.

For information about specific vectorization optimizations, see [Vectorization System](https://deepwiki.com/tenstorrent/sfpi-gcc/5.1-vectorization-system). For target-specific backend transformations, see [Target Backends](https://deepwiki.com/tenstorrent/sfpi-gcc/6-target-backends).

## Pass Infrastructure Overview

GCC's pass management system is defined by several core components:

*   **Pass definitions**: [gcc/passes.def 1-519](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L1-L519) defines the complete sequencing of all compiler passes
*   **Pass interface**: [gcc/tree-pass.h 1-786](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L1-L786) defines the base `opt_pass` class and pass types
*   **Pass execution**: The pass manager instantiates and executes passes according to the defined sequence

The pass infrastructure defines four fundamental pass types, all inheriting from `opt_pass`:

| Pass Type | Base Class | Purpose | Context |
| --- | --- | --- | --- |

| RTL pass | `rtl_opt_pass` | RTL-level optimizations | Per-function RTL |
| IPA pass | `ipa_opt_pass_d` | Whole-program analysis | Interprocedural |
| Simple IPA | `simple_ipa_opt_pass` | Lightweight IPA | Early analysis |

## GIMPLE Lowering Passes

The lowering passes convert high-level language constructs into GIMPLE form suitable for optimization. These passes always run first on a function.

### Lowering Pass Sequence

| Pass Name | Purpose | Implementation |
| --- | --- | --- |
| `pass_warn_unused_result` | Warn about unused function results | Early diagnostics |
| `pass_diagnose_omp_blocks` | Diagnose OpenMP blocks | OpenMP validation |
| `pass_lower_omp` | Lower OpenMP constructs | OpenMP expansion |
| `pass_lower_cf` | Lower control flow | Control flow lowering |
| `pass_lower_eh` | Lower exception handling | EH expansion |
| `pass_build_cfg` | Build control flow graph | CFG construction |
| `pass_expand_omp` | Expand OpenMP constructs | OpenMP parallelization |
| `pass_build_cgraph_edges` | Build call graph edges | IPA preparation |

## Interprocedural Analysis (IPA) Passes

IPA passes perform whole-program analysis and optimization across function boundaries. They are divided into three stages: early (small), main (regular), and late.

### Early IPA Passes (all_small_ipa_passes)

These lightweight passes prepare the program for main IPA analysis:

| Pass Name | Purpose | Phase |
| --- | --- | --- |
| `pass_ipa_free_lang_data` | Free language-specific data | Cleanup |
| `pass_ipa_function_and_variable_visibility` | Determine visibility | Analysis |
| `pass_build_ssa_passes` | Build SSA form | SSA construction |
| `pass_local_optimization_passes` | Early local opts | Optimization |

**Sources:**[gcc/passes.def 49-78](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/passes.def#L49-L78)

### Main IPA Passes (all_regular_ipa_passes)

The main IPA phase performs sophisticated interprocedural analysis and optimization:

| Pass Name | Purpose | Key Analysis |
| --- | --- | --- |
| `pass_ipa_whole_program_visibility` | Whole-program visibility | Symbol visibility |
| `pass_ipa_profile` | Profile-guided optimization | Edge profiling |
| `pass_ipa_icf` | Identical code folding | Function merging |
| `pass_ipa_devirt` | Devirtualization | Virtual call optimization |
| `pass_ipa_cp` | Constant propagation | Interprocedural constants |
| `pass_ipa_sra` | Scalar replacement | Argument optimization |
| `pass_ipa_fn_summary` | Function summaries | Cost analysis |
| `pass_ipa_inline` | Inlining | Call graph transformation |
| `pass_ipa_locality_cloning` | Locality-based cloning | Code layout optimization |

## Pass Properties and TODO Flags

Passes communicate requirements and changes through properties and TODO flags defined in [gcc/tree-pass.h 100-300](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L100-L300)

### Pass Properties

Properties indicate what the pass requires or provides:

| Property | Meaning | Value |
| --- | --- | --- |
| `PROP_gimple_any` | Any GIMPLE form accepted | 1 << 0 |
| `PROP_gimple_lcf` | Lowered control flow | 1 << 1 |
| `PROP_gimple_leh` | Lowered exception handling | 1 << 2 |
| `PROP_cfg` | Control flow graph built | 1 << 3 |
| `PROP_ssa` | SSA form | 1 << 5 |
| `PROP_loops` | Loop structures | 1 << 8 |
| `PROP_rtl` | RTL representation | 1 << 9 |

### TODO Flags

TODO flags indicate actions needed after pass execution:

Common TODO flags include:

*   `TODO_update_ssa`: Update SSA form after modifications
*   `TODO_cleanup_cfg`: Clean up control flow graph
*   `TODO_verify_ssa`: Verify SSA form correctness
*   `TODO_verify_flow`: Verify control flow graph
*   `TODO_verify_stmts`: Verify GIMPLE statement validity
*   `TODO_remove_unused_locals`: Remove unused local variables
*   `TODO_rebuild_alias`: Rebuild alias information

**Sources:**[gcc/tree-pass.h 100-300](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/tree-pass.h#L100-L300)

### Pass Manager Execution

### Pass Gate and Execute

Each pass implements two key methods:

1.   **Gate function**: Determines if the pass should run

    *   Checks optimization flags
    *   Verifies required properties
    *   Returns boolean

2.   **Execute function**: Performs the transformation

    *   Operates on current function or whole program
    *   Returns TODO flags
    *   May modify IR extensively

## Configuration and Control

Pass behavior is controlled through various mechanisms:

### Command-Line Options

Options from [gcc/common.opt 1-4000](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L1-L4000) control pass execution:

*   `-O0`, `-O1`, `-O2`, `-O3`, `-Os`, `-Ofast`: Optimization levels
*   `-fno-<pass-name>`: Disable specific pass
*   `-fdump-tree-<pass>`: Dump pass output
*   `-ftime-report`: Report pass timing
*   `-fipa-reorder-for-locality`: Enable IPA locality cloning

### Tuning Parameters

Parameters from [gcc/params.opt 1-1500](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/params.opt#L1-L1500) fine-tune pass behavior:

*   `--param max-inline-insns-single=N`: Control inlining
*   `--param lto-partition-locality-frequency-cutoff=N`: Locality cloning frequency threshold
*   `--param lto-partition-locality-size-cutoff=N`: Locality cloning size threshold
*   `--param max-locality-partition=N`: Maximum locality partition size
*   `--param max-combine-search-insns=N`: Limit combine search depth

### Pass-Specific Flags

Many passes have dedicated flags in [gcc/common.opt 1-4000](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L1-L4000):

*   `-ftree-vectorize`: Enable vectorization
*   `-ftree-loop-distribute-patterns`: Loop distribution
*   `-fipa-cp`: Interprocedural constant propagation
*   `-fdevirtualize`: Enable devirtualization
*   `-finline-functions`: Function inlining

**Sources:**[gcc/common.opt 1-4000](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/common.opt#L1-L4000)[gcc/params.opt 1-1500](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/params.opt#L1-L1500)
