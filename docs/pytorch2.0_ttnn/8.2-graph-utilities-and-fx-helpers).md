## Purpose and Scope

This page documents the graph manipulation utilities and FX helper functions provided by `torch_ttnn/utils.py`. These utilities enable efficient traversal, inspection, and modification of PyTorch FX graphs throughout the compilation pipeline. They provide abstractions for extracting metadata (shapes, dtypes), navigating graph structure, and registering custom TTNN objects within FX graphs.

For the broader compilation pipeline architecture, see [Core Compilation System](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2-core-compilation-system). For code export functionality that uses these utilities, see [Code Export and Standalone Script Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/8.1-code-export-and-standalone-script-generation).

## Custom Object Registration for TTNN Constants

FX graphs serialize all constants as part of their Python code generation. TTNN operations require special objects (device handles, layout configurations, memory configs) that cannot be directly serialized. The custom object registration system solves this by creating wrapper objects with unique string representations.

### TTNN Constant Wrapper Classes

[torch_ttnn/utils.py 136-175](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/utils.py#L136-L175)

**Diagram: TTNN Constant Wrapper Architecture**

Each wrapper class has a `__repr__` method that returns a unique identifier string. During code export, these strings are replaced with actual TTNN constants:

| Wrapper Class | `__repr__` Return | Replacement Target |
| --- | --- | --- |
| `TtnnDevice` | `"ttnn_Specified_Device"` | `"device"` |
| `TtnnRowMajorLayout` | `"ttnn_ROW_MAJOR_LAYOUT"` | `"ttnn.ROW_MAJOR_LAYOUT"` |
| `TtnnTileLayout` | `"ttnn_TILE_LAYOUT"` | `"ttnn.TILE_LAYOUT"` |
| `TtnnInt32` | `"ttnn_int32"` | `"ttnn.int32"` |
| `TtnnBfloat16` | `"ttnn_bfloat16"` | `"ttnn.bfloat16"` |
| `TtnnDramMemoryConfig` | `"ttnn_DRAM_MEMORY_CONFIG"` | `"ttnn.DRAM_MEMORY_CONFIG"` |
| `TtnnL1MemoryConfig` | `"ttnn_L1_MEMORY_CONFIG"` | `"ttnn.L1_MEMORY_CONFIG"` |

### Custom Object Registration API

The module provides two functions for registering custom objects in FX graphs:

#### get_add_custom_object_in_graph

[torch_ttnn/utils.py 182-211](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/utils.py#L182-L211)

**Diagram: Custom Object Registration Flow**

This function:

1.   Creates a `WrapperObj` class whose `__repr__` returns the provided key
2.   Registers the key as a custom builtin in `torch.fx.graph`
3.   Stores both the wrapper and original object in the registry
4.   Returns the wrapper object for use in FX graph nodes

#### get_emplace_custom_object_in_graph

[torch_ttnn/utils.py 213-263](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/utils.py#L213-L263)

This higher-level function builds on `get_add_custom_object_in_graph` by:

1.   Creating an instance of `object_type` with the provided `*args` and `**kwargs`
2.   Generating a unique key from the class name and sanitized parameters
3.   Registering the instance as a custom object

## Summary

The `torch_ttnn/utils.py` module provides essential infrastructure for working with PyTorch FX graphs in the compilation pipeline:

| Category | Functions | Purpose |
| --- | --- | --- |
| **Graph Manipulation** | `GraphCleanup` | Validate and clean graphs after transformations |
| **Node Inspection** | `get_opname`, `get_node_name`, `is_operation`, `users_have_getitem` | Extract node properties and navigate relationships |
| **Metadata Extraction** | `get_shape`, `get_dtype`, `get_arg`, `get_meta_val_attr` | Access tensor shapes, types, and attributes |
| **Graph Traversal** | `get_input_nodes`, `get_output_nodes` | Access graph boundaries |
| **TTNN Integration** | Custom object registration, constant wrappers | Enable TTNN-specific objects in FX graphs |
| **Validation** | `HasValidPageSize` | Ensure shapes meet hardware constraints |

These utilities abstract away FX graph complexity and provide a consistent API used by all compilation passes, transformation logic, and code export functionality.
