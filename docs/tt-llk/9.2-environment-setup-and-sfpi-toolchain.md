## Purpose and Scope

This document describes the environment setup infrastructure for tt-llk development and testing. The setup system handles:

1.   **SFPI Toolchain Management:** RISC-V GNU toolchain (`riscv-tt-elf-g++`) for compiling Tensix processor kernels
2.   **Python Virtual Environment:** Isolated Python environment with `uv` package installer for fast dependency resolution
3.   **Hardware Asset Downloads:** Architecture-specific headers and SFPU function files via `download_headers()` and `download_sfpu_files()`
4.   **System Dependencies:** APT package installation for required build tools
5.   **Pre-commit Hook Installation:** Code quality enforcement via `setup_precommit()`

The system provides two entry points:

*   **`setup_external_testing_env.sh`**: Local development setup with system dependency installation, virtual environment creation, and full initialization
*   **`setup_testing_env.sh`**: Core setup for SFPI toolchain and hardware assets (used by external script or CI environments)

The setup scripts support `--reuse`, `--clean`, and `--help` flags for flexible workflow management.

For detailed hardware header management, see page 9.3. For Docker-based CI environments, see page 9.4.

**Sources:**[tests/setup_testing_env.sh 1-252](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L1-L252)

## SFPI Toolchain Management

### SFPI Package Overview

The SFPI (Special Function Processor Instruction) toolchain is a RISC-V GNU cross-compiler targeting Tenstorrent's Tensix processor cores. The compiler binary is `riscv-tt-elf-g++` and is installed to `tests/sfpi/bin/`.

### Version File Structure

The `sfpi-version` file defines the toolchain version and SHA256 integrity hashes for all supported platforms:

**Supported Platform Matrix:**

| Host Architecture | Distribution | Package Formats | Example Hash Variable |
| --- | --- | --- | --- |
| `x86_64` | `debian` | `deb`, `txz` | `sfpi_x86_64_debian_deb_hash` `sfpi_x86_64_debian_txz_hash` |
| `x86_64` | `fedora` | `rpm`, `txz` | `sfpi_x86_64_fedora_rpm_hash` `sfpi_x86_64_fedora_txz_hash` |
| `aarch64` | `debian` | `deb`, `txz` | `sfpi_aarch64_debian_deb_hash` `sfpi_aarch64_debian_txz_hash` |

The setup script automatically selects the appropriate package based on detected host platform:

*   Architecture detection: `sfpi_arch=$(uname -m)`[tests/sfpi-info.sh 104](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-info.sh#L104-L104)
*   Distribution detection: Read `/etc/os-release` → set `sfpi_dist` from `ID` variable [tests/sfpi-info.sh 87-89](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-info.sh#L87-L89)
*   Distribution normalization: Check `ID_LIKE` for `debian` or `fedora` patterns [tests/sfpi-info.sh 91-97](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-info.sh#L91-L97)
*   Package format mapping: `sfpi_dist` → `sfpi_pkg` (`debian`→`deb`, `fedora`→`rpm`) [tests/sfpi-info.sh 99-102](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-info.sh#L99-L102)

**Sources:**[tests/sfpi-version 1-13](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-version#L1-L13)[tests/sfpi-info.sh 84-105](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/sfpi-info.sh#L84-L105)

## SFPI Installation Process

### Package Download and Verification

The installation sequence in `setup_testing_env.sh` follows a defensive verification pattern:

**Critical Verification Step:**

The hash verification uses the `sfpi_hashtype` variable (currently `sha256`) to ensure downloaded packages have not been corrupted or tampered with [tests/setup_testing_env.sh 227-232](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L227-L232)

**Installation Paths:**

| Component | Installation Path | Purpose |
| --- | --- | --- |
| Compiler Binary | `tests/sfpi/bin/riscv-tt-elf-g++` | RISC-V cross-compiler for Tensix cores |
| Linker | `tests/sfpi/bin/riscv-tt-elf-ld` | ELF linker for RISC-V binaries |
| Assembler | `tests/sfpi/bin/riscv-tt-elf-as` | RISC-V assembler |
| Headers | `tests/sfpi/lib/gcc/riscv-tt-elf/*/include/` | Standard library headers |
| Libraries | `tests/sfpi/lib/gcc/riscv-tt-elf/*/` | Runtime libraries (libc, libgcc) |
| Version File | `tests/sfpi/sfpi.version` | Installed version tracking stamp |

The extraction command removes any existing installation before unpacking: `rm -rf "${SCRIPT_DIR}/sfpi"`[tests/setup_testing_env.sh 236](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L236-L236)

## Stamp File Mechanism

Both header and SFPU downloads use stamp files to avoid redundant operations:

| Download Type | Stamp File Location | Verification |
| --- | --- | --- |
| Headers | `hw_specific/{arch}/inc/.headers_downloaded` | Checks file existence for each header |
| SFPU Files | `hw_specific/{arch}/metal_sfpu/.sfpu_downloaded` | Presence only (no content verification) |

The header download includes additional verification beyond stamp file presence:

This two-level check guards against partial downloads where the stamp file was created but some headers failed to download [tests/setup_testing_env.sh 53-66](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L53-L66)

**Header Download Source:**

Headers are downloaded from the `tt-metal` repository via `wget` with retry logic:

For Wormhole specifically, there's a fallback URL mechanism that tries `wormhole_b0_defines/` subdirectory if the primary URL fails [tests/setup_testing_env.sh 74-92](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/setup_testing_env.sh#L74-L92)
