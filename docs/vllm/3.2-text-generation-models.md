This page documents the text generation (decoder-only) models supported by vLLM, including the model registry system, architecture support, configuration options, and usage patterns for generating text from prompts.

**Scope**: This page covers models that perform autoregressive text generation (causal language modeling). For embedding/pooling models, see [Pooling Models](https://deepwiki.com/tenstorrent/vllm/3.3-multimodal-models). For multimodal models that accept images/audio/video, see [Multimodal Models](https://deepwiki.com/tenstorrent/vllm/3.3-multimodal-models). For the general model registry and loading mechanisms, see [Model Registry and Loading](https://deepwiki.com/tenstorrent/vllm/3.1-model-registry-and-loading).

## Supported Text Generation Architectures

vLLM natively supports over 100 text generation model architectures. These are decoder-only transformer models that generate text autoregressively given a prompt.

### Architecture Registry

Text generation models are registered in the `_TEXT_GENERATION_MODELS` dictionary, which maps HuggingFace architecture names to vLLM implementation details.

**Sources**: [vllm/model_executor/models/registry.py 45-156](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L45-L156)

### Major Model Families

The following table shows major model families and their architecture mappings:

| Model Family | HF Architecture | vLLM Module | Example Models |
| --- | --- | --- | --- |
| **Llama/Mistral** | `LlamaForCausalLM`, `MistralForCausalLM` | `llama` | Llama-3.2, Mistral-7B, InternLM |
| **GPT** | `GPT2LMHeadModel`, `GPTJForCausalLM`, `GPTNeoXForCausalLM` | `gpt2`, `gpt_j`, `gpt_neox` | GPT-2, GPT-J-6B, Pythia |
| **Qwen** | `Qwen2ForCausalLM`, `Qwen3ForCausalLM` | `qwen2`, `qwen3` | Qwen2.5, Qwen3 |
| **Gemma** | `GemmaForCausalLM`, `Gemma2ForCausalLM`, `Gemma3ForCausalLM` | `gemma`, `gemma2`, `gemma3` | Gemma-2B, Gemma-2-9B |
| **Mixture-of-Experts** | `MixtralForCausalLM`, `Qwen2MoeForCausalLM`, `DeepseekV2ForCausalLM` | `mixtral`, `qwen2_moe`, `deepseek_v2` | Mixtral-8x7B, DeepSeek-V2 |
| **Phi** | `PhiForCausalLM`, `Phi3ForCausalLM` | `phi`, `phi3` | Phi-2, Phi-3-mini |

### `LLM.generate()` Method

The `generate()` method performs synchronous text generation:

**Method Signature**:

**Input Formats**:

*   Plain strings: `"Hello, my name is"`
*   List of strings: `["Prompt 1", "Prompt 2"]`
*   `TextPrompt` objects with prompt and multi-modal data
*   Token IDs via `TokensPrompt`

**Output Structure**: Each `RequestOutput` contains:

*   `prompt`: Original prompt string
*   `outputs`: List of `CompletionOutput` objects (length = `n` from `SamplingParams`) 
    *   `text`: Generated text
    *   `token_ids`: Generated token IDs
    *   `cumulative_logprob`: Log probability
    *   `logprobs`: Per-token log probabilities (if requested)
    *   `finish_reason`: Why generation stopped ("stop", "length", etc.)

**Sources**: [vllm/entrypoints/llm.py 402-491](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/entrypoints/llm.py#L402-L491)[vllm/outputs.py 23-61](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/outputs.py#L23-L61)

## Sampling and Generation Control

### `SamplingParams` Configuration

`SamplingParams` controls the text generation behavior:

**Key Parameters**:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `n` | `int` | `1` | Number of completions to generate |
| `temperature` | `float` | `1.0` | Sampling temperature (0 = greedy) |
| `top_p` | `float` | `1.0` | Nucleus sampling threshold |
| `top_k` | `int` | `0` | Top-k sampling (0 = disabled) |
| `max_tokens` | `Optional[int]` | `None` | Maximum tokens to generate |
| `stop` | `Optional[Union[str, List[str]]]` | `None` | Stop strings |
| `stop_token_ids` | `Optional[List[int]]` | `None` | Stop token IDs |
| `repetition_penalty` | `float` | `1.0` | Penalize repeated tokens |
| `logprobs` | `Optional[int]` | `None` | Number of log probs to return |
| `structured_outputs` | `Optional[StructuredOutputsParams]` | `None` | Constrained decoding |

**Sources**: [vllm/sampling_params.py 85-272](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/sampling_params.py#L85-L272)

## Model-Specific Features

### Quantization Support

Text generation models support various quantization methods to reduce memory usage and improve throughput:

| Quantization Method | Supported Precisions | Implementation |
| --- | --- | --- |
| **FP8** | E4M3, E5M2 | CUTLASS, FlashInfer |
| **INT8** | W8A8 | CUTLASS |
| **INT4** | W4A16 | AWQ, GPTQ, Marlin |
| **AWQ** | 4-bit | Custom CUDA kernels |
| **GPTQ** | 2/3/4/8-bit | Custom CUDA kernels |
| **NVFP4** | 4-bit | NV-specific kernels (Blackwell) |

## Transformers Backend Fallback

For models not natively implemented in vLLM, the Transformers backend provides compatibility:

**Enabling Transformers Backend**:

**Compatibility Requirements**:

1.   Model must implement `_supports_attention_backend = True`
2.   Attention must use `ALL_ATTENTION_FUNCTIONS[_attn_implementation]`
3.   `kwargs` passed through from base model to attention layer

**Sources**: [docs/models/supported_models.md 16-145](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1#L16-L145)[vllm/model_executor/models/transformers.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/transformers.py)
