## Purpose and Scope

This page covers the fundamental concepts and terminology used throughout the Tracy profiling system. Understanding these concepts is essential for both using Tracy to profile applications and contributing to the Tracy codebase.

For information about the overall system architecture, see [Architecture Overview](https://deepwiki.com/tenstorrent/tracy/1.1-architecture-overview). For practical integration steps, see [Getting Started](https://deepwiki.com/tenstorrent/tracy/2-getting-started).

## Profiling Modes and Data Collection

### Real-time vs Sampling Profiling

Tracy operates as a **hybrid profiler** that combines two complementary approaches:

*   **Frame Profiler**: Manual instrumentation using zones to capture precise timing of code sections
*   **Sampling Profiler**: Periodic capture of call stacks for statistical analysis

The system achieves **nanosecond resolution** through hardware timer access via `rdtscp` instruction on x86/x64 and timer registers on ARM platforms.

**Sources**: [manual/tracy.tex 138-231](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L138-L231)[manual/techdoc.tex 159-179](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L159-L179)

### Client-Server Model

The **client** is the instrumented application that collects profiling data, while the **server** is the Tracy profiler GUI that processes and visualizes the data. This terminology reflects the fact that the server connects to the client to begin profiling.

**Sources**: [manual/tracy.tex 234-271](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L234-L271)[manual/techdoc.tex 360-377](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L360-L377)

### Profiling Operation Modes

| Mode | Description | Memory Usage | Reconnection |
| --- | --- | --- | --- |
| **Continuous** | Profiling starts immediately | High (accumulates until connection) | Single connection only |
| **On-demand** (`TRACY_ON_DEMAND`) | Profiling only when server connected | Low | Multiple connections possible |

**Sources**: [manual/tracy.tex 522-536](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L522-L536)

## Data Representation Concepts

### Zones

**Zones** are the fundamental unit of manual instrumentation, representing timed code sections:

Zone instrumentation is handled by the `ScopedZone` class which emits `QueueZoneBegin` in its constructor and `QueueZoneEnd` in its destructor.

**Sources**: [manual/techdoc.tex 466-472](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L466-L472)

### Frames

**Frames** represent work units, typically one iteration of a main loop. Frame profiling enables analysis of frame-to-frame performance variations:

*   `FrameMark` - Marks the end of a frame
*   `FrameMarkNamed(name)` - Creates named frame sets for different loops
*   Frame data enables timeline navigation and performance analysis

**Sources**: [manual/tracy.tex 223-225](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L223-L225)

### Source Locations

Source locations are represented by static `SourceLocationData` structures containing function name, file path, line number, and optional display name and color. Events store pointers to these structures rather than duplicating the data.

**Sources**: [manual/techdoc.tex 452-465](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L452-L465)

## File Formats and Protocols

### Tracy File Format

Tracy files (`.tracy`) use a compressed binary format with version headers:

The file format supports two compression schemes identified by magic headers:

*   `Lz4Header`: `{'t', 'l', 'Z', 4}`
*   `ZstdHeader`: `{'t', 'Z', 's', 't'}`

**Sources**: [server/TracyFileHeader.hpp 11-17](https://github.com/tenstorrent/tracy/blob/68c5cbd6/server/TracyFileHeader.hpp#L11-L17)

### Network Protocol

The client-server communication uses TCP port 8086 with LZ4 compression:

1.   **Handshake**: Exchange of `HandshakeShibboleth` and `ProtocolVersion`
2.   **Welcome Message**: Client sends timer calibration and system info
3.   **Event Stream**: Compressed queue items and auxiliary data
4.   **Query Protocol**: Server requests additional data (source locations, call stacks)

**Sources**: [manual/techdoc.tex 360-419](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L360-L419)

## Key Data Structures

### Queue Items

All profiling events are represented as 32-byte `QueueItem` structures:

Event types are grouped for efficient dispatch:

*   Events requiring auxiliary data transfer
*   String/data transfer events
*   Standard events

**Sources**: [manual/techdoc.tex 337-349](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L337-L349)

### Thread and Memory Management

Tracy uses separate memory management to avoid interfering with the profiled application:

*   **rpmalloc**: Custom allocator accessed via `tracy_malloc()`/`tracy_free()`
*   **Thread Identifiers**: 64-bit values sent with event batches
*   **Thread Local Storage**: Per-thread profiler state and queue access

**Sources**: [manual/techdoc.tex 154-157](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L154-L157)[manual/techdoc.tex 207-217](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L207-L217)

## Timing and Measurement

### Hardware Timer Access

Tracy achieves nanosecond resolution through direct hardware access:

| Platform | Timer Source | Resolution |
| --- | --- | --- |
| x86/x64 | `rdtscp` instruction | ~1-5 nanoseconds |
| ARM/ARM64 | `CNTVCT_EL0` register | ~40 nanoseconds |
| iOS | `mach_absolute_time()` | Hardware-dependent |
| Fallback | `std::chrono::high_resolution_clock` | ~300ns-1μs |

Timer readings are calibrated by comparing hardware timer values with system timer values to calculate a correction multiplier.

**Sources**: [manual/tracy.tex 159-180](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L159-L180)[manual/techdoc.tex 159-173](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/techdoc.tex#L159-L173)

### Performance Impact

Tracy's instrumentation overhead is approximately **2.25 nanoseconds per zone**, measured across millions of zone captures. This minimal impact is achieved through:

*   Lock-free queues (`moodycamel::ConcurrentQueue`)
*   Efficient memory pre-allocation
*   Minimal data copying
*   Hardware timer usage

**Sources**: [manual/tracy.tex 289-306](https://github.com/tenstorrent/tracy/blob/68c5cbd6/manual/tracy.tex#L289-L306)
