This document describes the first phase of the tt-lang compilation pipeline: converting Python kernel definitions to initial TTL MLIR. This phase is implemented by the `TTLGenericCompiler` class, which parses the Python Abstract Syntax Tree (AST) and emits TTL dialect operations representing the high-level structure of compute and data movement kernels. For subsequent transformation passes, see [3.3 TTL Dialect Transformations](https://github.com/tenstorrent/tt-lang/blob/d76e6233/3.3%20TTL%20Dialect%20Transformations) For the complete pipeline overview, see [3.1 Pipeline Overview](https://github.com/tenstorrent/tt-lang/blob/d76e6233/3.1%20Pipeline%20Overview)

## Purpose and Scope

The Python-to-TTL phase translates decorated Python functions (`@ttl.compute()` and `@ttl.datamovement()`) into MLIR `func.func` operations containing TTL dialect operations. This initial MLIR representation preserves the structure of the Python code while making it amenable to optimization passes. The compiler handles:

*   Function entry point generation with tensor accessor arguments [python/ttl/_src/ttl_ast.py 439-519](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L439-L519)
*   Circular buffer binding and lifecycle operations [python/ttl/_src/ttl_ast.py 425-437](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L425-L437)
*   Data movement operations (`ttl.copy`, tensor slicing) [python/ttl/operators.py 182-206](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/operators.py#L182-L206)
*   Tile math operations and operator overloading [python/ttl/operators.py 73-111](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/operators.py#L73-L111)
*   Control flow (loops, conditionals) [python/pykernel/_src/kernel_ast.py 121-204](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/pykernel/_src/kernel_ast.py#L121-L204)
*   Source location tracking for error reporting [python/ttl/_src/ttl_ast.py 33-39](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L33-L39)
*   Optional auto-profiling instrumentation [python/ttl/_src/ttl_ast.py 197-244](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L197-L244)

## Compilation Flow Overview

Title: Python to Initial MLIR Compilation Flow

The compilation is triggered when a decorated thread function is called. The decorator wraps the function and, when invoked, extracts the source code via `inspect.getsource`, cleans it using `_cleanup_source_code`, parses it to an AST, and passes it to `TTLGenericCompiler.visit()` which emits MLIR operations [python/ttl/ttl_api.py 780-853](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L780-L853)

### Initialization and Context

The compiler is initialized with runtime arguments and compilation context:

The `CompilerContext` (immutable dataclass) stores grid dimensions, memory space configuration, and layout information passed from the kernel decorator [python/ttl/_src/ttl_ast.py 119-127](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L119-L127):

| Field | Type | Purpose |
| --- | --- | --- |
| `grid` | `List[int]` | Grid dimensions `[cols, rows]` for multi-core execution |
| `memory_space` | `str` | `"L1"` or `"DRAM"` - determines addressing mode |
| `tiled` | `bool` | Whether tensors use tile layout (always `True` for TTL) |

Sources: [python/ttl/_src/ttl_ast.py 119-145](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L119-L145)

### Capture Mechanism

Thread functions capture variables from their enclosing scope. The `_collect_captures()` helper extracts closure variables and converts them to MLIR-compatible types [python/ttl/ttl_api.py 723-754](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L723-L754) Special handling is provided for `PipeNet` objects to preserve user-facing variable names for diagnostics [python/ttl/_src/ttl_ast.py 177-194](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L177-L194)

| Captured Type | Conversion | Usage |
| --- | --- | --- |
| `int` | Kept as Python int, emitted as `arith.constant` | Loop bounds, block counts |
| `CircularBuffer` | Kept as object, emitted via `_emit_cb_from_capture()` | CB binding |
| `PipeNet` | Tracked by ID for naming | Verifier diagnostics |
| TTNN tensor | Kept as object, becomes function argument | Tensor accessors |

Sources: [python/ttl/ttl_api.py 723-754](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L723-L754)[python/ttl/_src/ttl_ast.py 177-194](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L177-L194)

## TTL Operation Emission

### Circular Buffer Lifecycle Operations

The `with`-statement provides Python context manager syntax for CB acquire/release patterns. The `visit_With()` method emits corresponding TTL operations [python/ttl/_src/ttl_ast.py 578-698](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L578-L698):

| User Operation | TTL Operations Emitted | Description |
| --- | --- | --- |
| `dfb.wait()` | `ttl.cb_wait` → `ttl.attach_cb` | Wait for producer, attach CB to tensor |
| `dfb.reserve()` | `ttl.cb_reserve` → `ttl.attach_cb` | Reserve space, attach CB to tensor |
| Exit `with` (wait) | `ttl.cb_pop` | Release consumer block |
| Exit `with` (reserve) | `ttl.cb_push` | Commit producer block |
