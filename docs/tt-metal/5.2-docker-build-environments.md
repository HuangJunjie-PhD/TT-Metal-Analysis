## Docker Image Hierarchy

The `tt-metal` repository maintains several Docker images with distinct purposes, organized in a hierarchical dependency structure. The primary build logic is contained in `dockerfile/Dockerfile`. The build process is optimized using Docker Buildx Bake to manage complex multi-stage dependencies [dockerfile/Dockerfile 1-6](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L1-L6)

**Docker Stage and Image Relationship**

**Image Purpose Matrix**

| Image Name | Primary Use | Key Features | Typical Users |
| --- | --- | --- | --- |
| `base` | Runtime foundation | SFPI compiler, `uv`, Python venv, OpenMPI | All derived images |
| `ci-build` | Compilation | `ccache`, `mold` linker, `clang-20`, `yq` | CI build jobs |
| `ci-test` | Test execution | PyTorch CPU, `pytest`, `requirements-dev.txt` | CI test jobs |
| `dev` | Interactive development | `gdb`, debuggers, editors (vim/emacs) | Developers |
| `basic-dev` | Minimal Build | Lightweight alternative to `ci-build` | Local fast builds |

Sources: [dockerfile/Dockerfile 48-274](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L48-L274)[dockerfile/Dockerfile.basic-dev 1-10](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile.basic-dev#L1-L10)

## Image Tagging and Versioning Strategy

Docker images use content-addressable tagging based on SHA1 hashes of their dependencies. This ensures reproducible builds and efficient caching within the CI pipeline.

**Code Entity Mapping: Docker Hash Inputs**

**Tag Computation Logic** The `build-docker-artifact.yaml` workflow manages the creation of these tags. It outputs specific tags like `ci-build-tag`, `ci-test-tag`, and `dev-tag` which are then consumed by downstream workflows such as `pr-gate.yaml` and `merge-gate.yaml`[.github/workflows/pr-gate.yaml 113-116](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/pr-gate.yaml#L113-L116)[.github/workflows/merge-gate.yaml 129-134](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/merge-gate.yaml#L129-L134)

## Multi-Stage Build Breakdown

### Stage 1: `base` - Runtime Foundation

**Purpose**: Minimal runtime dependencies and environment configuration.

**Key Components**:

1.   **OpenMPI**: The `base` image includes OpenMPI (defaulting to `v5.0.7-ulfm`) copied from the `openmpi-layer`[dockerfile/Dockerfile 90-101](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L90-L101)
2.   **Python Management**: `uv` is used to install Python. Versions are pinned based on the Ubuntu distribution: Python 3.10 for Ubuntu 22.04 and Python 3.12 for Ubuntu 24.04 [dockerfile/Dockerfile 109-118](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L109-L118)
3.   **SFPI Toolchain**: The RISC-V SFPI toolchain is copied from the `sfpi-layer` into `/opt/tenstorrent/sfpi`[dockerfile/Dockerfile 127-130](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L127-L130)
4.   **System Dependencies**: Core libraries are installed via `install_dependencies.sh --docker`[dockerfile/Dockerfile 124](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L124-L124)

Sources: [dockerfile/Dockerfile 73-131](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L73-L131)[install_dependencies.sh 1-20](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/install_dependencies.sh#L1-L20)

### Stage 2: `ci-build` - Build Tools

**Purpose**: Optimized toolchain for high-performance C++ compilation.

**Notable Tools**:

*   **Linker**: Includes `mold` for faster linking [dockerfile/Dockerfile 154](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L154-L154)
*   **Caching**: Includes `ccache` to speed up repetitive builds in CI [dockerfile/Dockerfile 150](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L150-L150)
*   **Analysis**: Includes `doxygen`, `clangbuildanalyzer`, and `yq`[dockerfile/Dockerfile 151-153](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L151-L153)

Sources: [dockerfile/Dockerfile 136-173](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L136-L173)

### Stage 3: `ci-test` - Test Execution

**Purpose**: Environment for running the full test suite, including Python and C++ tests.

**Notable Features**:

*   **Python Dependencies**: Installs requirements from `requirements-dev.txt`[dockerfile/Dockerfile 192](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L192-L192)
*   **Torch Configuration**: Specifically targets CPU-only `torch` for x86_64 to avoid unnecessary GPU driver dependencies in the build environment [dockerfile/Dockerfile 203-206](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L203-L206)
*   **Environment**: Sets `PYTHONPATH` and virtual environment paths [dockerfile/Dockerfile 181-184](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L181-L184)

Sources: [dockerfile/Dockerfile 176-213](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L176-L213)

## Container Workflows and Testing

### CI/CD Workflow Integration

Docker images are typically resolved once per pipeline run to ensure consistency across build and test jobs.

**Workflow Logic (Build and Test Integration)**

**Key Behaviors**:

1.   **TTSim Integration**: The `ttsim.yaml` workflow uses the `basic-docker-image` to run integration tests for Wormhole and Blackhole architectures using the TTSim simulator [.github/workflows/ttsim.yaml 48-53](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/ttsim.yaml#L48-L53)
