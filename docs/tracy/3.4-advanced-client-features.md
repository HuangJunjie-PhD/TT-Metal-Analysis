## Callstack Capture

Tracy provides comprehensive callstack capture capabilities across multiple platforms and backends. The system automatically selects the most appropriate method based on the target platform and available libraries.

### Callstack Backend Architecture

The callstack system supports multiple collection methods:

*   **Windows (TRACY_HAS_CALLSTACK=1)**: Uses `RtlWalkFrameChain` for fast collection and DbgHelp for symbol resolution
*   **libunwind (TRACY_HAS_CALLSTACK=2,5)**: Uses `_Unwind_Backtrace` for portable unwinding
*   **execinfo (TRACY_HAS_CALLSTACK=3,4,6)**: Uses `backtrace()` with optional libunwind or libbacktrace backends

### Symbol Resolution Pipeline

Tracy implements a sophisticated symbol resolution system with multiple fallback mechanisms:

**Windows Symbol Resolution**:

*   Module cache with automatic loading via `EnumProcessModules`
*   Kernel driver symbol cache via `EnumDeviceDrivers`
*   DbgHelp integration with `SymFromAddr` and `SymGetLineFromAddr64`
*   Support for inline function resolution with `SymFromInlineContext`

**Unix Symbol Resolution**:

*   libbacktrace integration for DWARF processing
*   Dynamic library enumeration via `dl_iterate_phdr`
*   Kernel symbol loading from `/proc/kallsyms`
*   debuginfod support for remote debug symbol retrieval

## Lock Profiling

Tracy provides wrapper classes that automatically instrument lock operations while maintaining the original lock interface.

### Lock Instrumentation Architecture

### Lock Wrapper Implementation

The lock profiling system uses RAII wrappers that intercept lock operations:

| Lock Type | Wrapper Class | Queue Events | Use Case |
| --- | --- | --- | --- |
| Exclusive | `Lockable<T>` | Wait, Obtain, Release | std::mutex, std::recursive_mutex |
| Shared | `SharedLockable<T>` | SharedWait, SharedObtain, SharedRelease | std::shared_mutex |
| Custom | `LockableCtx` | Custom lock implementations | Manual instrumentation |

The system automatically generates unique lock IDs and tracks timing for:

*   **Wait time**: Duration spent waiting to acquire the lock
*   **Hold time**: Duration the lock was held
*   **Contention**: When multiple threads compete for the same lock

## Crash Handling

Tracy includes comprehensive crash handling to capture profiling data even when applications terminate unexpectedly.

### Crash Handler Architecture

### Cross-Platform Crash Handling

The crash handling system provides detailed crash information across platforms:

**Windows Crash Handling**:

*   Vectored exception handling with `SetUnhandledExceptionFilter`
*   Detailed exception analysis including memory access violations
*   Thread enumeration and suspension via `CreateToolhelp32Snapshot`
*   Integration with Windows Error Reporting

**Linux Crash Handling**:

*   Signal-based crash detection (`SIGSEGV`, `SIGILL`, `SIGFPE`, etc.)
*   Thread enumeration via `/proc/self/task`
*   Signal delivery to all threads for coordinated shutdown
*   Protection against recursive crashes

_Sources: [public/client/TracyProfiler.cpp 777-883](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/client/TracyProfiler.cpp#L777-L883)[public/client/TracyProfiler.cpp 898-1129](https://github.com/tenstorrent/tracy/blob/68c5cbd6/public/client/TracyProfiler.cpp#L898-L1129)[meson.build 96-98](https://github.com/tenstorrent/tracy/blob/68c5cbd6/meson.build#L96-L98)_

### Crash Data Preservation

The crash handler ensures maximum data preservation by:

*   Capturing a deep callstack (60 frames) at the crash point
*   Suspending all other threads to prevent data corruption
*   Flushing any remaining profiling data to the network
*   Providing a reasonable timeout to balance data preservation with responsiveness
