This page documents the backend services architecture of TT-Studio, covering the core service components, their responsibilities, and inter-service communication patterns. The backend consists of multiple containerized services that work together to provide model management, inference execution, and data persistence capabilities.

## Overview

TT-Studio's backend follows a microservices architecture with four primary service components, each running in separate Docker containers and communicating via HTTP APIs and shared volumes. The architecture supports dynamic model deployment, concurrent inference execution, and RAG-enhanced chat functionality.

## Core Backend Services

### Django API Server (tt_studio_backend)

The Django API server acts as the primary backend orchestrator, running on port 8000. It provides the main REST API endpoints consumed by the React frontend and coordinates interactions between all other backend services.

**Key Responsibilities:**

*   **Model Lifecycle Management**: Handles deployment, stopping, and starting of models using `run_container` and `stop_container`[app/backend/docker_control/docker_utils.py 157-300](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L157-L300)
*   **Docker Orchestration**: Interfaces with the Docker Control Service via `get_docker_client` to manage containers and networks [app/backend/docker_control/docker_control_client.py 18-35](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L18-L35)
*   **Hardware Abstraction**: Detects board types (e.g., N150, T3K, P150) and maps them to device names compatible with the inference server [app/backend/docker_control/docker_utils.py 122-155](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L122-L155)
*   **RAG Management**: Coordinates with ChromaDB for document ingestion and retrieval.

**Key Classes/Functions:**

*   `ContainersView`: Lists available models and current board status [app/backend/docker_control/views.py 151-220](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/views.py#L151-L220)
*   `InferenceView`: Proxies inference requests to the appropriate model container [app/backend/model_control/views.py 94-138](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/views.py#L94-L138)
*   `SystemResourceService`: Manages hardware resource reporting and `tt-smi` cache [app/backend/board_control/services.py 18](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/board_control/services.py#L18-L18)

### FastAPI Inference Server (tt_inference_server)

The FastAPI inference server on port 8001 provides a unified inference API that routes requests to dynamically deployed model containers. It handles the low-level logic of interacting with `tt-inference-server` workflows.

**Key Responsibilities:**

*   **Job Management**: Tracks deployment progress via `/run/progress/{job_id}`[inference-api/api.py 488-510](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L488-L510)
*   **Model Deployment**: Executes `run_main` from the `tt-inference-server` artifact to spin up model containers [inference-api/api.py 106](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L106-L106)
*   **Workflow Execution**: Supports various workflows like `server`, `demo`, and `vllm`[inference-api/api.py 106](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L106-L106)

### TT Agent Service (tt_studio_agent)

The TT Agent service on port 8080 provides AI agent capabilities including multi-LLM support and search integration.

**Key Responsibilities:**

*   **Agent Reasoning**: Processes requests via `AgentView`[app/backend/model_control/views.py 141-175](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/views.py#L141-L175)
*   **Auto-Discovery**: Can automatically find and use available LLM model containers for agent tasks [app/backend/model_control/views.py 165-167](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/views.py#L165-L167)

## Inference Request Flow

Inference requests are proxied through the Django backend to provide a consistent interface and handle RAG or Agent logic.

### Standard Inference vs. Agent Inference

1.   **Standard Inference**: `InferenceView` retrieves the internal URL of the model container from the `deploy_cache` and streams the response directly from the model's FastAPI/Flask endpoint [app/backend/model_control/views.py 94-138](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/views.py#L94-L138)
2.   **Agent Inference**: `AgentView` routes the request to `tt_studio_agent:8080`. The agent may then call the model container or external search APIs before returning a streamed response [app/backend/model_control/views.py 141-175](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/views.py#L141-L175)

## Security and Authentication

### Inter-Service Communication

*   **JWT**: The backend uses JWT tokens to authenticate requests to model containers. The `encoded_jwt` function generates tokens using the `JWT_SECRET` environment variable [app/backend/model_control/model_utils.py 47](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/model_control/model_utils.py#L47-L47)
*   **Docker Control**: Requests to the Docker Control Service are authenticated via a dedicated client that handles token headers [app/backend/docker_control/docker_control_client.py 18-35](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_control_client.py#L18-L35)

### Resource Protection

*   **Reset Logic**: When a model is stopped, the backend performs a hardware reset (`perform_reset`) to ensure the Tenstorrent chip is in a clean state for the next deployment [app/backend/docker_control/views.py 107-119](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/views.py#L107-L119)
*   **Timeout Management**: Deployments have a 5-hour timeout to accommodate large model downloads and compilation times [app/backend/docker_control/docker_utils.py 28](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L28-L28)
