## Purpose and Scope

This document presents an empirical evaluation of whether using multiple NoC virtual channels (VCs) improves throughput for unicast writes and read requests on Wormhole B0 and Blackhole platforms. The study tests one-to-one, one-to-all, and all-to-all traffic patterns across varying numbers of VCs to determine optimal usage patterns.

For general API usage guidance on virtual channels, see [Virtual Channels Usage](https://deepwiki.com/tenstorrent/tt-low-level-documentation/3.4-virtual-channels-usage). For baseline performance metrics of data movement primitives, see [Baseline Performance Metrics](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.1-baseline-performance-metrics).

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 1-7](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L1-L7)

## Key Finding

Alternating across multiple VCs within the same traffic class provides **no performance benefit** and often **degrades performance**. This counter-intuitive result stems from router hardware implementation details that make multiple VCs ineffective for parallelizing identical traffic streams.

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 4-6](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L4-L6)

## Virtual Channel Architecture

### Hardware Structure

Each physical NoC channel provides 16 virtual channels organized as:

*   **8 Non-Dateline VCs**: Primary routing paths
*   **8 Dateline VCs**: Paired with non-dateline VCs for deadlock prevention
*   **4 Unicast VCs**: Available within non-dateline group for software control

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 10-12](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L10-L12)

## Performance Results

### Wormhole B0 Results

| Test Type | Observation |
| --- | --- |
| **One-to-One (Write)** | No material difference across VC counts; no congestion present, so VCs provide no benefit |
| **One-to-All (Write)** | Similar to one-to-one; kernel serializes destinations, negating VC benefits |
| **All-to-All (Write)** | Irregular bandwidth variation with size and VC count; performance can degrade |
| **One-from-One (Read)** | No meaningful throughput gain from additional VCs |
| **One-from-All (Read)** | No meaningful throughput gain from additional VCs |
| **All-from-All (Read)** | Erratic bandwidth due to response ordering and hardware VC selection |

## Root Cause Analysis

### Problem 1: Router VC Allocation Ordering

**Issue:** The router arbitrates among VCs _before_ arbitrating among input/output ports. If a blocked VC is selected during VC arbitration, the router incurs idle cycles even if another VC could route successfully.

This design choice (made to meet timing constraints) means that:

*   Multiple VCs competing for the same path increase the chance of selecting a blocked VC
*   Idle cycles negate any intended throughput benefit
*   Single-VC operation avoids this overhead

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 79-83](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L79-L83)

### Problem 2: Shared Input Buffer Constraints

**Issue:** Input buffers are shared among VCs with capacity for only 3 active VCs at full bandwidth. When traffic uses multiple VCs:

*   **1 VC:** Effectively uses 2 VCs (non-dateline + paired dateline) → Full bandwidth
*   **2 VCs:** Effectively uses 4 VCs → Buffer pressure emerges, performance begins to degrade
*   **3-4 VCs:** Buffer saturation → All VCs drop to 1/16 bandwidth due to shared-buffer limits

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 84-86](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L84-L86)

## Traffic Pattern Analysis

### One-to-One Pattern

With no contention present, VC selection has no impact. All traffic flows through a single, uncongested path regardless of VC count.

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 88-89](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L88-L89)

### One-to-All Pattern

The evaluated kernel sends to one destination at a time (serialized iteration across destinations). Since destinations are processed sequentially, VC alternation provides no concurrency benefit.

**Potential Improvement:** Randomizing destination order could reduce unintended serialization effects.

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 89-99](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L89-L99)

### All-from-All Read Pattern

Erratic bandwidth stems from:

1.   **Response Ordering Perturbations:** Responses arrive in variable order depending on network state
2.   **Hardware VC Selection:** Read responses use hardware-selected VCs, creating unpredictable congestion patterns
3.   **Transient Congestion:** Variable response timing creates temporary hotspots

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 90-91](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L90-L91)

## Correct Usage Patterns

### Principle: Separate Traffic Classes, Not Same-Class Parallelization

**Correct:** Use multiple VCs to separate traffic with **different bandwidth characteristics**

*   DRAM vs. Tensix L1: Different bandwidth classes → Use different VCs
*   Tensix L1 vs. Ethernet: Same bandwidth class → Share same VC
*   Data vs. Flow Control: Different bandwidth classes → Use different VCs

**Incorrect:** Use multiple VCs to parallelize **identical traffic streams**

*   Debug vs. Production data to same destination: Same traffic class → Use same VC
*   Multiple data streams to same destination: Same traffic class → Use same VC

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 94-98](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L94-L98)

## Recommendations

| Recommendation | Rationale |
| --- | --- |
| **Use 1 VC for same-class traffic** | Avoids arbitration overhead and buffer contention |
| **Separate traffic by bandwidth class** | DRAM, L1, Ethernet, credits should use distinct VCs based on bandwidth characteristics |
| **Dedicate VC for flow control** | Credit/acknowledgement messages should use separate VC from data (e.g., VC 2 in Buda) |
| **Avoid splitting identical traffic** | Debug and production data targeting same destination should share VC |
| **Consider destination randomization** | For one-to-all patterns, randomize destination order to reduce serialization |

**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 92-99](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L92-L99)

## Open Issues and Future Work

### Small Transaction Count Bandwidth Drop

**Observation:** With only 4 transactions, bandwidth drops notably beyond:

*   4096 B/transaction on Wormhole (NoC 0)
*   2048 B/transaction on Blackhole (NoC 0)

**Likely Cause:** Unintended serialization at small transaction counts

**Proposed Solution:** Randomize destination and VC selection per iteration to break serialization patterns:

`// Example randomization approachuint32_t current_virtual_channel = (i + 9 * my_y + my_x) % num_virtual_channels;uint32_t sub_idx = (j + 9 * my_y + my_x) % num_subordinates;subordinate_x_coord = get_arg_val<uint32_t>(sub_idx * 2);subordinate_y_coord = get_arg_val<uint32_t>(sub_idx * 2 + 1);`
**Sources:**[data_movement_doc/virtual_channels/Virtual Channels.md 102-111](https://github.com/tenstorrent/tt-low-level-documentation/blob/ae7661c8/data_movement_doc/virtual_channels/Virtual%20Channels.md?plain=1#L102-L111)

### Planned Experiments

1.   **Re-run with randomization:** Execute one-to-all and all-to-all tests (4 transactions) with randomized destination/VC selection to quantify serialization impact

2.   **Flow-control VC validation:** Test configuration where flow-control messages use dedicated VC separate from data to validate recommendation

3.   **Buffer occupancy study:** Throttle injection rates to probe shared-buffer limits and determine saturation onset

4.   **Loopback VC sweep:** Add VC sweep for loopback traffic (low priority; implicitly covered by unicast tests)

## Relationship to Multicast Performance

While this study focused on unicast and read traffic, the findings complement the critical insights from the [Multicast Schemes Performance Study](https://deepwiki.com/tenstorrent/tt-low-level-documentation/4.2-multicast-schemes-performance-study):

*   Both studies reveal architectural constraints that limit expected performance gains
*   VC separation by traffic class aligns with avoiding self-interference in multicast schemes
*   Buffer sharing issues identified here also impact multicast acknowledge traffic
