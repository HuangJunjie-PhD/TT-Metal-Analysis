## Purpose and Scope

The PCIe SerDes Library (`tt_blackhole_libpciesd.a`) is a pre-compiled static library that provides low-level initialization and configuration routines for the PCIe SerDes (Serializer/Deserializer) hardware on Blackhole ASICs. This library handles physical layer configuration, link training, and SerDes-specific operations required to establish PCIe connectivity.

This document covers the library's interface, integration into the firmware build system, and usage patterns. For information about the overall PCIe subsystem initialization sequence, see [PCIe Subsystem](https://deepwiki.com/tenstorrent/tt-system-firmware/3.2-pcie-subsystem). For details about other binary firmware components, see [MRISC Firmware for GDDR](https://deepwiki.com/tenstorrent/tt-system-firmware/5.1-mrisc-firmware-for-gddr) and [ERISC Firmware for Ethernet](https://deepwiki.com/tenstorrent/tt-system-firmware/5.2-erisc-firmware-for-ethernet).

## Library Overview

The PCIe SerDes library is distributed as a pre-compiled static archive containing low-level SerDes configuration code. Unlike other binary firmware components that execute on dedicated RISC processors (MRISC, ERISC), this library is linked directly into the SMC/CMFW application and executes on the ARC processor during system initialization.

## Library Interface

The PCIe SerDes library exposes five primary functions through the `lib/tenstorrent/bh_arc/pciesd.h` header file.

### Functions

#### SerdesInit

`PCIeInitStatus SerdesInit(uint8_t pcie_inst, PCIeDeviceType device_type,                          uint8_t num_serdes_instance);`
Initializes the PCIe SerDes hardware for a given PCIe instance. This function configures the physical layer parameters, loads SerDes firmware, and prepares the hardware for link training.

**Parameters:**

*   `pcie_inst`: PCIe instance number (0 or 1 for Blackhole)
*   `device_type`: `EndPoint` (0) or `RootComplex` (1)
*   `num_serdes_instance`: Number of SerDes lanes to initialize (1 or 2)

**Returns:**`PCIeInitStatus` enumeration value indicating success or failure reason

#### CntlInitV2

`PCIeInitStatus CntlInitV2(const struct CntlInitV2Param *param);`
Second-generation controller initialization routine that configures PCIe controller registers, BARs, and link parameters based on board-specific configuration.

**Parameters:**

*   `param`: Pointer to `CntlInitV2Param` structure containing configuration

**Returns:**`PCIeInitStatus` enumeration value

#### CntlInit

`PCIeInitStatus CntlInit(uint64_t board_id, uint32_t vendor_id, uint8_t pcie_inst,                        uint8_t serdes_inst, uint8_t max_pcie_speed, uint8_t device_type,                        uint64_t region0_mask, uint64_t region2_mask, uint64_t region4_mask);`
Legacy controller initialization routine. The newer `CntlInitV2` function is preferred.

#### EnterLoopback / ExitLoopback

`void EnterLoopback(uint8_t pcie_inst);void ExitLoopback(uint8_t pcie_inst);`
Utility functions for enabling/disabling SerDes loopback mode, primarily used for testing and diagnostics.

**Sources:**[lib/tenstorrent/bh_arc/pciesd.h 38-48](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L38-L48)

### Data Structures

#### struct CntlInitV2Param

`struct CntlInitV2Param {    uint64_t board_id;          // Unique board identifier    uint32_t vendor_id;         // PCI vendor ID    uint8_t pcie_inst;          // PCIe instance (0 or 1)    uint8_t serdes_inst;        // SerDes instance    uint8_t max_pcie_speed;     // Maximum PCIe generation (1-4)    uint8_t device_type;        // EndPoint or RootComplex    uint64_t region0_mask;      // BAR0 size mask    uint64_t region2_mask;      // BAR2 size mask    uint64_t region4_mask;      // BAR4 size mask};`
This structure encapsulates all configuration parameters needed for PCIe controller initialization, including board identification, device role, and memory region sizing.

**Sources:**[lib/tenstorrent/bh_arc/pciesd.h 22-32](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L22-L32)

#### enum PCIeDeviceType

`typedef enum {    EndPoint = 0,    RootComplex = 1,} PCIeDeviceType;`
Defines the PCIe device's role in the topology. Blackhole boards typically operate in `EndPoint` mode when connected to a host system.

**Sources:**[lib/tenstorrent/bh_arc/pcie.h 13-16](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L13-L16)

#### enum PCIeInitStatus

`typedef enum {    PCIeInitOk = 0,    PCIeSerdesFWLoadTimeout = 1,    PCIeLinkTrainTimeout = 2,} PCIeInitStatus;`
Return codes from SerDes initialization functions indicating success or specific failure modes.

**Sources:**[lib/tenstorrent/bh_arc/pcie.h 18-22](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L18-L22)

## Usage in PCIe Initialization

The PCIe SerDes library is invoked during the system initialization sequence at priority level 103 (`pcie_init_PRIO`). The initialization flow is implemented in `lib/tenstorrent/bh_arc/pcie.c`.

### Configuration Parameters

The library receives board-specific configuration from the firmware table, including:

| Parameter | Source | Description |
| --- | --- | --- |
| `pcie_mode` | FW Table | Determines if PCIe is enabled or disabled for each instance |
| `num_serdes` | FW Table | Number of SerDes lanes (1 or 2) based on board layout |
| `max_pcie_speed` | FW Table | Maximum PCIe generation (Gen1-Gen4) |
| `board_id` | FW Table | Unique 64-bit board identifier |
| `vendor_id` | FW Table | PCI vendor ID (default: 0x1E52 for Tenstorrent) |
| `bar_sizes` | Constants | Default BAR sizes: BAR0=512MB, BAR2=1MB, BAR4=32GB |

### Multi-Instance Support

Blackhole ASICs feature two PCIe instances (PCIe0 and PCIe1), each potentially using multiple SerDes lanes. The library is called separately for each active PCIe instance based on the board's configuration.

## DMA Dependency

The PCIe SerDes library has a runtime dependency on the ARC DMA subsystem for transferring SerDes firmware and configuration data. The library expects the `ArcDmaTransfer()` function to be available in the linking environment.

### DMA Function Interface

`bool ArcDmaTransfer(const void *src, void *dst, uint32_t size);`
The `pciesd.h` header includes a compile-time verification that this function signature matches the library's expectations:

`__unused static bool (*verify_ArcDmaTransfer)(const void *, void *, uint32_t) = ArcDmaTransfer;`
This ensures that any signature mismatch between the library and the firmware's DMA implementation is caught at compile time.

**Sources:**[lib/tenstorrent/bh_arc/pciesd.h 35](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L35-L35)[lib/tenstorrent/bh_arc/pciesd.h 9](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pciesd.h#L9-L9)

## Platform-Specific Behavior

The PCIe SerDes library is only linked and used when building for ARC-based platforms. The `CMakeLists.txt` includes a conditional to prevent linking on non-ARC builds:

`if(CONFIG_ARC)  zephyr_library_link_libraries(${ZEPHYR_CURRENT_MODULE_DIR}/zephyr/blobs/tt_blackhole_libpciesd.a)endif()`
This ensures that recovery firmware builds (which use `CONFIG_TT_SMC_RECOVERY`) do not attempt to use PCIe functionality.

**Sources:**[lib/tenstorrent/bh_arc/CMakeLists.txt 68-70](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/CMakeLists.txt#L68-L70)

## Error Handling

The library functions return `PCIeInitStatus` values to indicate initialization outcomes:

| Status Code | Value | Meaning |
| --- | --- | --- |
| `PCIeInitOk` | 0 | Initialization completed successfully |
| `PCIeSerdesFWLoadTimeout` | 1 | SerDes firmware failed to load within timeout |
| `PCIeLinkTrainTimeout` | 2 | PCIe link training did not complete within timeout |

The calling code in `pcie.c` logs errors when non-zero status codes are returned and may adjust system behavior accordingly (e.g., disabling Ethernet on certain SerDes lanes if PCIe initialization fails).

**Sources:**[lib/tenstorrent/bh_arc/pcie.h 18-22](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.h#L18-L22)[lib/tenstorrent/bh_arc/pcie.c 250-400](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L250-L400)
