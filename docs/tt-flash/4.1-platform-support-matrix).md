This document provides a comprehensive reference of all hardware platforms supported by tt-flash, including chip architectures, board variants, and their respective identification codes. This page focuses on the enumeration and categorization of supported hardware.

For details on how board identification works internally (UPI extraction, revision parsing), see [Board Identification](https://deepwiki.com/tenstorrent/tt-flash/4.2-board-identification). For architecture-specific implementation details, see [Wormhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.3-wormhole-platform) and [Blackhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.4-blackhole-platform).

## Current Platform Support

tt-flash currently supports two chip architectures: **Wormhole** and **Blackhole**. Support for the legacy Grayskull architecture was removed in version 3.4.7 [README.md 152](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1#L152-L152)

## Legacy Platform Support

Grayskull chip support was officially removed in version 3.4.7 [debian/changelog 63](https://github.com/tenstorrent/tt-flash/blob/60c06032/debian/changelog#L63-L63) The last version of tt-flash that supports Grayskull devices is [v3.4.7](https://github.com/tenstorrent/tt-flash/blob/60c06032/v3.4.7)[README.md 152](https://github.com/tenstorrent/tt-flash/blob/60c06032/README.md?plain=1#L152-L152)

The removal included:

*   Deletion of Grayskull-specific chip implementation classes
*   Removal of Grayskull-specific firmware handling code
*   Removal of all Grayskull references from documentation

## Chip Architecture Implementation

Each supported chip architecture has a dedicated implementation class that inherits from the `TTChip` abstract base class:

*   Reset procedures
*   SPI flash operations

## Board Identification Workflow

When tt-flash detects a device, it follows this workflow to identify the board type:

1.   **Read Board ID**: Retrieve the 64-bit board ID from `chip.board_type`
2.   **Extract UPI**: Extract bits 36-55 using `(board_id >> 36) & 0xFFFFF`
3.   **Extract Revision**: Extract bits 32-35 using `(board_id >> 32) & 0xF`
4.   **Lookup Codename**: Use `get_board_type()` to map UPI (and revision if needed) to codename
5.   **Map Public Name**: Use `change_to_public_name()` to convert codename to display name
6.   **Select Implementation**: Route to `WhChip` or `BhChip` based on codename

For detailed information on this process, see [Board Identification](https://deepwiki.com/tenstorrent/tt-flash/4.2-board-identification).

Sources: [tt_flash/utility.py 39-131](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L39-L131)

## Special Cases

### Revision-Dependent Boards

The **E300 family** requires revision checking to distinguish between variants:

*   UPI `0x1` with revision `0x2` → `E300_R2`
*   UPI `0x1` with revision `0x3` or `0x4` → `E300_R3`

This logic is implemented in [tt_flash/utility.py 60-69](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/utility.py#L60-L69)

### P-Series Boards

The P-series boards (P100, P150, P300) use a range of UPI codes from `0x36` to `0x47`. Some variants (P300 series) require additional logic to identify left/right chip positioning on dual-chip boards [debian/changelog 147](https://github.com/tenstorrent/tt-flash/blob/60c06032/debian/changelog#L147-L147)

### Galaxy Systems

Both Wormhole (`WH_UBB`, UPI `0x35`) and Blackhole (`GALAXY-1`, UPI `0x47`) have Galaxy system variants that require special reset procedures involving IPMI commands rather than standard chip resets [debian/changelog 101-103](https://github.com/tenstorrent/tt-flash/blob/60c06032/debian/changelog#L101-L103)
