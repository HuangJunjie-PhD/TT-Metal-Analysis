## Overview

This page documents the Python library API functions for direct RISC-V core control, including memory access, reset management, and execution control. These functions provide low-level programmatic access to individual RISC-V cores on Tenstorrent devices.

**Scope:** This page covers the direct control API for reading/writing memory, managing reset signals, and controlling execution state. For loading complete ELF files and running firmware, see [ELF Management](https://deepwiki.com/tenstorrent/tt-exalens/3.5-elf-management). For comprehensive debugging infrastructure including callstacks and watchpoints, see [RISC-V Debugging System](https://deepwiki.com/tenstorrent/tt-exalens/6-risc-v-debugging-system).

**Target Audience:** Users who need to directly interact with RISC-V cores for testing, debugging, or low-level control scenarios.

## Core Control Architecture

The RISC-V core control API consists of high-level functions in `tt_exalens_lib.py` that delegate to the `RiscDebug` interface, which is implemented by `BabyRiscDebug` and platform-specific subclasses.

## Memory Access Functions

The primary API for reading and writing RISC-V memory provides access to both L1 and private memory regions.

### read_riscv_memory()

`def read_riscv_memory(    location: str | OnChipCoordinate,    address: int,    risc_name: str = "brisc",    context: Context | None = None,    device_id: int = 0,) -> int`
Reads a 32-bit word from RISC-V memory at the specified address. The function automatically handles:

*   Address validation
*   Private memory access requirements (halting/reset workarounds)
*   Platform-specific hardware differences

**Example Usage:**

`# Read from private memory addressvalue = read_riscv_memory("0,0", 0xFFB00000, "brisc") # Read from L1 memoryvalue = read_riscv_memory("1,0", 0x1000, "trisc0")`
**Sources:**[ttexalens/tt_exalens_lib.py 702-722](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L702-L722)[test/ttexalens/unit_tests/test_lib.py 574-603](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_lib.py#L574-L603)

### write_riscv_memory()

`def write_riscv_memory(    location: str | OnChipCoordinate,    address: int,    data: int,    risc_name: str = "brisc",    context: Context | None = None,    device_id: int = 0,) -> None`
Writes a 32-bit word to RISC-V memory at the specified address. Supports the same automatic handling as `read_riscv_memory()`.

**Example Usage:**

`# Write to private memorywrite_riscv_memory("0,0", 0xFFB00000, 0x12345678, "brisc")`
**Sources:**[ttexalens/tt_exalens_lib.py 725-748](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L725-L748)[test/ttexalens/unit_tests/test_lib.py 574-603](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_lib.py#L574-L603)

### Memory Address Spaces

RISC-V cores have access to multiple memory regions:

| Memory Region | Address Range | Access Method | Notes |
| --- | --- | --- | --- |
| **L1 Memory** | 0x0 - varies | NOC direct access | Shared across cores |
| **Data Private** | 0xFFB00000 - 0xFFB02000 | Debug interface | 8KB per core, requires halt/reset |
| **Code Private** | 0xFFC00000 - varies | Debug interface or NOC | Platform-dependent |

**Sources:**[ttexalens/hardware/baby_risc_debug.py 693-831](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L693-L831)[ttexalens/hardware/memory_block.py 8-43](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/memory_block.py#L8-L43)

### Byte-Level Memory Access

The underlying implementation provides byte-level access with automatic alignment handling:

The implementation automatically:

*   Aligns read/write operations to 4-byte boundaries
*   Performs read-modify-write for partial word writes
*   Handles cross-boundary access spanning multiple words

## Reset Control

RISC-V cores can be placed in and taken out of reset programmatically through the `RiscDebug` interface.

### set_reset_signal()

The `set_reset_signal(value: bool)` method controls the reset state of a RISC-V core:

`# Put core in resetrisc_debug.set_reset_signal(True) # Take core out of resetrisc_debug.set_reset_signal(False)`
**Implementation Details:**

The reset signal is controlled via the `SOFT_RESET_0` register, with each core having a specific bit position:

## Private Memory Access

Private memory regions (0xFFB00000 - data, 0xFFC00000 - code) require special handling because they are only accessible when the core is halted or in specific states.

### ensure_private_memory_access Context Manager

The `ensure_private_memory_access()` context manager provides safe access to private memory by automatically managing core state:

`with risc_debug.ensure_private_memory_access():    # Access private memory safely    value = risc_debug.read_memory(0xFFB00000)    risc_debug.write_memory(0xFFB00100, 0x42)`
### Access Patterns

**Key Implementation Details:**

1.   **Core Running:** If the core is not in reset, simply halt it, perform operations, and resume.

2.   **Core in Reset:** The implementation:

    *   Saves 4 bytes from the start address
    *   Writes an endless loop instruction (`JAL x0, 0` = 0x6F) to start address
    *   Takes core out of reset (it enters the endless loop)
    *   Halts the core
    *   Performs memory operations
    *   Returns core to reset
    *   Restores original bytes at start address

**Sources:**[ttexalens/hardware/baby_risc_debug.py 573-601](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L573-L601)[test/ttexalens/unit_tests/test_lib.py 574-603](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/test/ttexalens/unit_tests/test_lib.py#L574-L603)

### Automatic Private Memory Detection

The API functions automatically detect private memory addresses and apply the correct access pattern:

`# This automatically uses ensure_private_memory_access internallyvalue = read_riscv_memory("0,0", 0xFFB00100, "brisc")`

## Register Access

Direct access to RISC-V general-purpose registers (GPRs) and the program counter is provided through the `RiscDebug` interface.

### General Purpose Registers

`# Read GPR (x0-x31)value = risc_debug.read_gpr(10)  # Read register a0 (x10) # Write GPRrisc_debug.write_gpr(10, 0x1234)  # Write to register a0`
**Register Mapping:**

| Index | Name | ABI Name | Purpose |
| --- | --- | --- | --- |
| 0 | x0 | zero | Constant zero |
| 1 | x1 | ra | Return address |
| 2 | x2 | sp | Stack pointer |
| 8 | x8 | s0/fp | Frame pointer |
| 10-17 | x10-x17 | a0-a7 | Function arguments |
| 32 | - | pc | Program counter |

**Sources:**[ttexalens/hardware/baby_risc_debug.py 659-671](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L659-L671)[ttexalens/hardware/baby_risc_debug.py 69-173](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L69-L173)

### Program Counter Access

`# Get program counterpc = risc_debug.get_pc()`
The implementation uses two methods based on availability:

1.   **Debug Bus (Preferred):** If debug bus signals are available, reads PC directly without halting
2.   **GPR Read (Fallback):** Halts core temporarily and reads PC from register 32

**Note:** NCRISC and ERISC cores lose the topmost PC bit on the debug bus, requiring a correction by setting bit 31.

**Sources:**[ttexalens/hardware/baby_risc_debug.py 673-691](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L673-L691)[ttexalens/hardware/blackhole/baby_risc_debug.py 20-25](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L20-L25)

## Execution Control

Control of RISC-V core execution state is provided through halt, step, and continue operations.

### Core States and Transitions

### halt()

Halts the RISC-V core, allowing debug operations:

`risc_debug.halt() # Now safe to perform debug operationspc = risc_debug.read_gpr(32)`
**Implementation:** Writes `COMMAND_DEBUG_MODE + COMMAND_HALT` to the debug command register and verifies the halt succeeded.

**Sources:**[ttexalens/hardware/baby_risc_debug.py 629-634](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L629-L634)[ttexalens/hardware/baby_risc_debug.py 296-306](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L296-L306)

### step()

Executes a single instruction and re-halts:

`risc_debug.step()  # Execute one instruction`
**Platform Differences:**

| Platform | Behavior |
| --- | --- |
| Wormhole | Requires branch prediction disabled |
| Blackhole | Executes step twice due to hardware bug |
| Quasar | Standard behavior |

**Sources:**[ttexalens/hardware/baby_risc_debug.py 636-641](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L636-L641)[ttexalens/hardware/blackhole/baby_risc_debug.py 15-18](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L15-L18)[ttexalens/hardware/wormhole/baby_risc_debug.py 26-30](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L26-L30)

### cont()

Resumes core execution:

`risc_debug.cont()  # Resume execution`
**Platform Differences:**

| Platform | Implementation |
| --- | --- |
| Wormhole | Disables branch prediction, then continues with debug mode |
| Wormhole (ERISC) | Uses `continue_without_debug()` (no branch prediction control) |
| Blackhole | Standard continue with debug mode |
| Quasar | Standard continue with debug mode |

**Sources:**[ttexalens/hardware/baby_risc_debug.py 643-648](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L643-L648)[ttexalens/hardware/wormhole/baby_risc_debug.py 13-24](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L13-L24)

### ensure_halted Context Manager

Provides automatic halt/resume around operations:

`with risc_debug.ensure_halted():    # Core is guaranteed to be halted    value = risc_debug.read_gpr(10)    # Core automatically resumes after block (if it wasn't halted before)`
**Sources:**[ttexalens/hardware/baby_risc_debug.py 650-657](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L650-L657)[ttexalens/hardware/baby_risc_debug.py 323-335](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L323-L335)

## Debug Hardware Interface

The low-level debug hardware interface is implemented in `BabyRiscDebugHardware`, which communicates with RISC cores through dedicated debug registers.

### Debug Register Protocol

### Register Access Pattern

All debug operations follow a similar pattern:

1.   **Write Operation:**

```
Write data to CNTL1
Write command + address to CNTL0 (triggers operation)
Clear CNTL0 to acknowledge
```
2.   **Read Operation:**

```
Write address to CNTL0 (triggers read)
Clear CNTL0 to acknowledge
Read result from STATUS1
```

**Sources:**[ttexalens/hardware/baby_risc_debug.py 218-466](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L218-L466)

### Command Codes

| Command | Value | Purpose |
| --- | --- | --- |
| `COMMAND_HALT` | 0x00000001 | Halt core execution |
| `COMMAND_STEP` | 0x00000002 | Execute single instruction |
| `COMMAND_CONTINUE` | 0x00000004 | Resume execution |
| `COMMAND_READ_REGISTER` | 0x00000008 | Read GPR |
| `COMMAND_WRITE_REGISTER` | 0x00000010 | Write GPR |
| `COMMAND_READ_MEMORY` | 0x00000020 | Read memory word |
| `COMMAND_WRITE_MEMORY` | 0x00000040 | Write memory word |
| `COMMAND_FLUSH` | 0x00000100 | Flush pipeline |
| `COMMAND_DEBUG_MODE` | 0x80000000 | Enable debug mode |

**Sources:**[ttexalens/hardware/baby_risc_debug.py 48-58](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L48-L58)

## Platform-Specific Implementations

Each platform has specific quirks that require custom handling.

### Platform Implementation Classes

### Wormhole Specific

**Branch Prediction Workaround:** Must disable branch prediction before `step()` and `cont()` operations on functional worker cores. ERISC cores use `continue_without_debug()` instead.

**PC Read:** Uses debug bus signal for PC instead of GPR read.

**Sources:**[ttexalens/hardware/wormhole/baby_risc_debug.py 9-38](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L9-L38)

### Blackhole Specific

**Step Bug:** Hardware bug requires calling `step()` twice to execute a single instruction.

**TRISC2 Bug:** Accessing private memory addresses where `address % 16 > 4` fails due to hardware bug.

**PC Read:** Uses debug bus signal for PC.

**Sources:**[ttexalens/hardware/blackhole/baby_risc_debug.py 11-40](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L11-L40)

### Quasar Specific

**Instruction Cache:** Requires explicit flush after cache invalidation using the `flush()` command.

**PC Read:** Uses debug bus from NEO block.

**Sources:**[ttexalens/hardware/quasar/baby_risc_debug.py 9-35](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L9-L35)

## Usage Examples

### Example 1: Read and Modify Memory

`import ttexalens as lib # Initialize contextcontext = lib.init_ttexalens() # Read from private memoryvalue = lib.read_riscv_memory("0,0", 0xFFB00100, "brisc")print(f"Value at 0xFFB00100: 0x{value:08x}") # Modify the valuelib.write_riscv_memory("0,0", 0xFFB00100, value + 1, "brisc")`
### Example 2: Inspect Registers

`# Get coordinate and risc_debug objectlocation = lib.convert_coordinate("1,0")risc_debug = location.noc_block.get_risc_debug("trisc0") # Read all GPRswith risc_debug.ensure_halted():    for i in range(32):        reg_value = risc_debug.read_gpr(i)        print(f"x{i}: 0x{reg_value:08x}")        pc = risc_debug.read_gpr(32)    print(f"PC: 0x{pc:08x}")`
### Example 3: Single-Step Execution

`location = lib.convert_coordinate("0,0")risc_debug = location.noc_block.get_risc_debug("brisc") # Halt, step, and inspectrisc_debug.halt()pc_before = risc_debug.get_pc() risc_debug.step()pc_after = risc_debug.get_pc() print(f"PC before: 0x{pc_before:08x}")print(f"PC after:  0x{pc_after:08x}") risc_debug.cont()  # Resume execution`
### Example 4: Safe Private Memory Access

`location = lib.convert_coordinate("2,2")risc_debug = location.noc_block.get_risc_debug("ncrisc") # Works regardless of core state (running or in reset)with risc_debug.ensure_private_memory_access():    # Read private data memory    data = risc_debug.read_memory_bytes(0xFFB00000, 256)        # Modify and write back    modified = bytes([b ^ 0xFF for b in data])    risc_debug.write_memory_bytes(0xFFB00000, modified)`
