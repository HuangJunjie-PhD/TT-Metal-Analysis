## Purpose and Scope

This document describes the firmware signing infrastructure and MCUBoot integration in tt-system-firmware. It covers how firmware images are cryptographically signed during the build process, how MCUBoot validates signatures at boot time, and the recovery mechanisms that enable fallback when signature verification fails.

For information about the overall build system architecture, see [Build System](https://deepwiki.com/tenstorrent/tt-system-firmware/6-build-system). For details on firmware packaging and distribution, see [Combined Firmware Bundles](https://deepwiki.com/tenstorrent/tt-system-firmware/6.4-combined-firmware-bundles).

## MCUBoot Bootloader Overview

MCUBoot is an open-source secure bootloader used by tt-system-firmware to provide cryptographic verification of firmware images before execution. The system uses MCUBoot to ensure that only authenticated firmware can run on SMC and DMC controllers.

### MCUBoot Components

| Component | Description | Build Artifact |
| --- | --- | --- |
| **MCUBoot Bootloader** | First-stage bootloader that validates and loads application firmware | `mcuboot/zephyr/zephyr.elf` |
| **Main Application** | Primary firmware image with MCUBoot header | `smc/zephyr/zephyr.signed.bin` |
| **Recovery Firmware** | Minimal fallback firmware for recovery operations | `recovery/zephyr/zephyr.signed.bin` |

The sysbuild configuration orchestrates building all three components together, ensuring compatible signing keys and image formats.

## Build-Time Signing Process

Firmware signing occurs during the build process through integration with Zephyr's sysbuild system and the `imgtool` utility from MCUBoot.

### Signature Key Management

The signing workflow:

1.   **Key Provisioning**: The RSA-2048 private key is stored as a GitHub secret and decoded at build time
2.   **Sysbuild Configuration**: The key file path is passed via `-DSB_CONFIG_BOOT_SIGNATURE_KEY_FILE`
3.   **Image Signing**: `imgtool` generates RSA-2048 signatures and embeds them in TLV areas
4.   **Verification**: CI pipeline verifies all signed images before artifact upload

**Sources:**[.github/workflows/build-fw.yml 78-84](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L78-L84)[.github/workflows/build-fw.yml 94-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L94-L106)

### Build Command Integration

The build process passes the signature key through CMake configuration:

```
west build --sysbuild -b tt_blackhole@p100a/tt_blackhole/smc \
  -- -DSB_CONFIG_BOOT_SIGNATURE_KEY_FILE="$SIGNATURE_KEY_FILE"
```

For recovery firmware, additional configuration disables unsigned image generation:

```
west build -b tt_blackhole@p100a/tt_blackhole/smc \
  -- -DCONFIG_MCUBOOT_GENERATE_UNSIGNED_IMAGE=n \
     -DCONFIG_MCUBOOT_SIGNATURE_KEY_FILE="$SIGNATURE_KEY_FILE"
```

**Sources:**[.github/workflows/build-fw.yml 94-106](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L94-L106)[.github/workflows/build-fw.yml 159-176](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L159-L176)

### Post-Build Verification

After building, the CI pipeline verifies signatures using `imgtool verify`:

The `imgtool verify` command returns a non-zero exit code if:

*   The MCUBoot header is malformed
*   The signature is invalid
*   The key does not match the signature

## Boot-Time Verification Flow

MCUBoot performs cryptographic verification during the boot sequence before transferring control to the application firmware.

### MCUBoot Boot Sequence

### Signature Verification Algorithm

MCUBoot uses the following verification process:

1.   **Image Hash Calculation**: Compute SHA-256 hash of the application image
2.   **Signature Extraction**: Read RSA-2048 signature from TLV area
3.   **Public Key Verification**: Verify signature using embedded public key
4.   **Hash Comparison**: Compare computed hash with signed hash

The public key used for verification is embedded in the MCUBoot bootloader binary at build time, ensuring it matches the private key used to sign the application.

**Sources:**[app/smc/pytest/e2e_smoke.py 878-948](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L878-L948)

## Recovery and Failover Mechanism

The system implements a multi-level fallback strategy to maintain operability even when the main firmware image is corrupted or has invalid signatures.

### Recovery Boot Paths

### Recovery Mode Characteristics

When booting from recovery firmware, the system operates with reduced functionality:

| Feature | Main Firmware | Recovery Firmware |
| --- | --- | --- |
| **Telemetry** | ✓ Available via `get_telemetry()` | ✗ Not available |
| **Firmware Updates** | ✓ Supported via `tt-flash` | ✓ Supported via `tt-flash` |
| **Full SMC/DMC Communication** | ✓ All messages supported | ⚠ Limited message support |
| **Hardware Initialization** | ✓ Complete initialization | ⚠ Minimal initialization |

The recovery firmware is intentionally minimal to maximize reliability. Its primary purpose is to enable firmware updates that restore the main application.

**Sources:**[app/smc/pytest/e2e_smoke.py 913-915](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L913-L915)[app/smc/pytest/e2e_smoke.py 927-932](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L927-L932)

### Recovery Firmware Flash Process

When the main image is corrupted, the recovery firmware can still accept new firmware via `tt-flash`:

The test validation of this process:

1.   **Erase Main Header**: Intentionally corrupt the main image header at `0x29E000`
2.   **Trigger Reset**: Use `tt-smi -r` to reset the system
3.   **Verify Recovery Boot**: Confirm telemetry is unavailable (expected in recovery mode)
4.   **Flash Valid Image**: Use `tt-flash` to write a properly signed firmware bundle
5.   **Verify Main Boot**: Confirm telemetry is available after reset

## Testing Signature Verification

The firmware signing infrastructure includes comprehensive end-to-end tests to validate the entire signature verification and recovery flow.

### Test Coverage

### Test Implementation Details

The `test_mcuboot` function validates the complete recovery mechanism:

**Memory Addresses Used:**

*   `MCUBOOT_HEADER_ADDR = 0x29E000` - Main image header location
*   `MCUBOOT_MAGIC = 0x96F3B83D` - Expected magic number

**SPI Operations:**

*   `arc_chip.as_bh().spi_read(addr, buffer)` - Read from SPI flash
*   `arc_chip.as_bh().spi_write(addr, data)` - Write to SPI flash

**Validation Checks:**

*   Telemetry availability indicates boot mode (main vs. recovery)
*   MCUBoot magic presence indicates valid image header
*   Recovery mode successfully accepts firmware updates via `tt-flash`

**Sources:**[app/smc/pytest/e2e_smoke.py 878-948](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L878-L948)

### Bootloader Update Testing

For firmware upgrades that require bootloader replacement, the test workflow includes:

1.   **Flash Base Firmware**: Install older firmware version with different bootloader
2.   **Replace Bootloader**: Flash new MCUBoot bootloader using `west flash` for `mcuboot-bl2`
3.   **Verify DMC/SMC Boot**: Confirm both DMC and SMC boot with new bootloader
4.   **Flash Updated Firmware**: Install latest firmware signed with new key
5.   **Validate Telemetry**: Confirm firmware versions match expected values

This ensures that bootloader updates preserve the chain of trust and maintain compatibility with newly signed firmware images.

## Signature Verification in CI/CD

The build pipeline includes multiple verification checkpoints to ensure all released firmware is properly signed.

### CI Verification Workflow

### Assembly Test Firmware Signing

Assembly test firmware for manufacturing also uses MCUBoot signing:

**Build Configuration:**

```
west build --sysbuild -b tt_blackhole@p150a/tt_blackhole/dmc \
  -- -DCONFIG_TT_ASSEMBLY_TEST=y \
     -DCONFIG_MCUBOOT_GENERATE_UNSIGNED_IMAGE=n \
     -DSB_CONFIG_BOOT_SIGNATURE_KEY_FILE="$SIGNATURE_KEY_FILE"
```

**Verification:**

```
imgtool verify --key "$SIGNATURE_KEY_FILE" \
  build-assembly-test-p150/dmc/zephyr/zephyr.signed.bin
```

This ensures that even manufacturing test firmware maintains the security posture of production firmware.

**Sources:**[.github/workflows/build-fw.yml 242-274](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/build-fw.yml#L242-L274)

### Conditional Signing

The build system gracefully handles missing signature keys for development builds:

`SIGNATURE_KEY_FILE="$(mktemp)"echo "${{ secrets.SIGNATURE_KEY }}" | base64 -d > "$SIGNATURE_KEY_FILE"SZ=$(wc -c "$SIGNATURE_KEY_FILE" | awk '{print $1}')if [ $SZ -gt 42 ]; then  echo "SIGNATURE_KEY_FILE=$SIGNATURE_KEY_FILE" >> "$GITHUB_ENV"fi`
If the decoded key file is smaller than 42 bytes (indicating no valid key), the build proceeds without signing. This allows local development while ensuring all CI/CD builds are properly signed.

## Key Rotation and Security

### Public Key Embedding

The MCUBoot bootloader embeds the public key at build time. This means:

1.   **Key Immutability**: Once a bootloader is flashed, it can only verify images signed with the corresponding private key
2.   **Bootloader Updates Required**: Changing signing keys requires reflashing the MCUBoot bootloader
3.   **Chain of Trust**: The bootloader's public key establishes the root of trust for all subsequent firmware

### Development vs. Production Keys

| Environment | Key Type | Storage Location | Usage |
| --- | --- | --- | --- |
| **Development** | Optional | Not stored | Local testing without signing |
| **CI/CD** | Test Key | GitHub Secret `SIGNATURE_KEY` | Automated builds and testing |
| **Production** | Production Key | Secure storage (not in repo) | Release firmware signing |

The test key used in CI/CD is sufficient for validation but should not be used for production deployments. Production systems require a separate, securely managed signing key.
