## Purpose and Scope

The pytest test suite provides **parametric, hardware-executable tests** for the `tt-lang` Python DSL. These tests validate runtime correctness of tensor operations, multi-core execution, shape handling, and TTNN integration by executing kernels on actual hardware or in simulation. Unlike MLIR lit tests which validate compilation stages (see [8.4](https://github.com/tenstorrent/tt-lang/blob/d76e6233/8.4)), pytest tests focus on **end-to-end execution correctness** and numerical accuracy.

## Test Organization

### Test File Naming Convention

Pytest tests are identified by the `test_*.py` naming pattern under `test/python/` or `test/me2e/`. Files without this prefix (e.g., `simple_add.py`) are typically collected by `llvm-lit` instead for FileCheck-based IR validation of the compilation pipeline.

| Pattern | Runner | Purpose |
| --- | --- | --- |
| `test_*.py` | pytest | Parametric runtime tests with device execution |
| `*.py` (no `test_` prefix) | llvm-lit | Compilation/MLIR tests with `# RUN:` directives |

Sources: [test/python/test_elementwise_ops.py 12-14](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_ops.py#L12-L14)[test/python/test_elementwise_shapes.py 18-19](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L18-L19)[test/python/conftest.py 53-59](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/conftest.py#L53-L59)

The pytest suite covers several functional areas:

| Test Category | Example File | Focus Area |

### conftest.py and Fixtures

The `test/python/conftest.py` file provides shared infrastructure, notably the `device` fixture and temporary file management.

#### Temporary File Cleanup

Kernels generated via factories are written to temporary `.py` files. These are tracked in `temp_kernel_files` and cleaned up at exit using `atexit.register` to prevent disk clutter, unless `TTLANG_KEEP_GENERATED_KERNELS` is set.

Sources: [test/python/conftest.py 21-38](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/conftest.py#L21-L38)[test/python/test_elementwise_ops.py 24](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_ops.py#L24-L24)

#### Device Management

Tests typically require a `device` fixture which handles `ttnn.open_device()` and `ttnn.close_device()`. If the environment variable `TTLANG_COMPILE_ONLY` is set to `1` (or if no hardware is detected by `ttlang_test_utils`), tests requiring a device are skipped.

Sources: [test/python/conftest.py 84-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/conftest.py#L84-L102)[test/ttlang_test_utils.py 41-43](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang_test_utils.py#L41-L43)

#### Feature Detection

The `ttlang_test_utils.py` module handles lightweight feature detection for `ttnn` availability and hardware presence. It checks environment variables like `TT_METAL_SIMULATOR` and `TTLANG_HAS_DEVICE`, as well as device files in `/dev/tenstorrent*`.

## Kernel Factory Pattern

To avoid maintaining thousands of static test files for every operation and shape combination, the suite uses **Kernel Factories**. These functions take parameters (like operation type or tile shape), format a string template, write it to a temporary file, and dynamically import it.

### Architecture Mapping

**Natural Language to Code Entity Mapping**

Sources: [test/python/test_elementwise_shapes.py 157-190](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L157-L190)[test/python/test_elementwise_ops.py 133-151](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_ops.py#L133-L151)

### Implementation Example

The factory uses `importlib` to turn a generated string into a callable Python function:

`def make_binary_kernel(name: str, op: str):    """Generate a binary kernel by writing to temp file and importing."""    code = BINARY_KERNEL_TEMPLATE.format(name=name, op=op)     # Write to temp file (delete=False so we can import it)    with tempfile.NamedTemporaryFile(        mode="w", suffix=".py", delete=False, prefix=f"kernel_{name}_"    ) as f:        f.write(code)        temp_path = f.name     # Import the module    spec = importlib.util.spec_from_file_location(f"{name}_kernel_module", temp_path)    module = importlib.util.module_from_spec(spec)    spec.loader.exec_module(module)    temp_kernel_files.append(temp_path)     return getattr(module, f"{name}_kernel")`
Sources: [test/python/test_elementwise_ops.py 133-151](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_ops.py#L133-L151)[test/python/test_elementwise_shapes.py 157-190](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_elementwise_shapes.py#L157-L190)

### Template Types

| Template Name | File | Purpose |
| --- | --- | --- |
| `BINARY_KERNEL_TEMPLATE` | `test_elementwise_ops.py` | Standard infix ops (`l + r`) |
| `BINARY_FN_KERNEL_TEMPLATE` | `test_elementwise_ops.py` | Math function calls (`ttl.math.max(l, r)`) |
| `UNARY_KERNEL_TEMPLATE` | `test_elementwise_ops.py` | Unary ops (`ttl.math.exp(x)`) |
| `ADD_KERNEL_TEMPLATE` | `test_tensor_slice.py` | Loop-based slicing over multi-tile tensors |

## ME2E Test Infrastructure

The Model-End-to-End (ME2E) infrastructure provides a specialized runner for executing kernels compiled through the `tt-lang` pipeline. It is largely driven by `TTLElementwiseOps.def` to auto-generate test classes.

### Op Registry and Arity

The ME2E system parses `TTLElementwiseOps.def` to determine operation arity and name mappings. It maintains a mapping to PyTorch reference functions in `OP_TORCH_MAP`.

## Validation and Accuracy

### Precision Thresholds

The test suite uses specialized utilities to account for hardware SFPU/FPU approximations. For elementwise ops, thresholds are often overridden per-operation to match `tt-metal` quality gates.

| Metric | Function | Purpose |
| --- | --- | --- |
| **PCC** | `assert_pcc()` | Pearson Correlation Coefficient; primary quality gate (default 0.9999). |
| **Allclose** | `assert_allclose()` | Element-wise closeness within `rtol` and `atol`. |
| **ULP** | `assert_with_ulp()` | Unit in the Last Place validation for high-precision ops. |

Sources: [test/ttlang_test_utils.py 203-234](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang_test_utils.py#L203-L234)[test/ttlang_test_utils.py 237-242](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/ttlang_test_utils.py#L237-L242)[test/me2e/ops/__init__.py 87-140](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/me2e/ops/__init__.py#L87-L140)

### Tensor Slicing and Loops

The `test_tensor_slice.py` suite sweeps tensor shapes from 1x1 to 15x15 tiles. It validates that the `tensor[r, c]` syntax correctly calculates offsets for specific tiles within a larger memory allocation using nested loops in both compute and data movement threads.

Sources: [test/python/test_tensor_slice.py 28-29](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_tensor_slice.py#L28-L29)[test/python/test_tensor_slice.py 47-53](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_tensor_slice.py#L47-L53)[test/python/test_tensor_slice.py 57-63](https://github.com/tenstorrent/tt-lang/blob/d76e6233/test/python/test_tensor_slice.py#L57-L63)

### Specialized Correctness Tests

Tests like `test_axby.py` validate complex register allocation scenarios. For example, `axby_fused_kernel` ensures that multiple binary operations (`a*x` and `b*y`) followed by an addition correctly track intermediate results in the DST register file without operand corruption.
