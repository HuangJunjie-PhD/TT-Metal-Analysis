## Purpose and Scope

This page documents the `tt_mlir` Python module, which provides the low-level bindings between Python and the C++ MLIR compilation and runtime layers. These bindings expose MLIR compilation functions, device management operations, and runtime execution primitives that enable PyTorch tensors to be executed on Tenstorrent hardware.

The `tt_mlir` module is built as a pybind11 extension ([tt_torch/csrc/bindings.cpp](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp)) that wraps the tt-mlir runtime and compiler libraries. It serves as the foundation for the Executor framework and provides direct access to device operations, tensor conversions, and program execution.

For higher-level execution abstractions that build on these bindings, see [Executor Framework](https://deepwiki.com/tenstorrent/tt-torch/3.3-executor-framework). For multi-device strategies and graph splitting, see [Device Management and Parallelism](https://deepwiki.com/tenstorrent/tt-torch/3.5-device-management-and-parallelism). For the overall compilation pipeline, see [Compilation Pipeline](https://deepwiki.com/tenstorrent/tt-torch/3.1-compilation-pipeline).

## Runtime Operations

The module provides functions for tensor preprocessing, program execution, and output retrieval. The execution flow differs based on whether synchronous (`run`) or asynchronous (`run_async`) execution is used.

### Synchronous vs Asynchronous Execution

**Key Differences:**

*   **Synchronous (`run`)**: Blocks until execution completes and automatically transfers outputs to host as PyTorch tensors
*   **Asynchronous (`run_async`)**: Returns immediately with device-resident runtime tensors, allowing pipelining of operations

### Execution Flow Sequence

**Sources:**[tt_torch/csrc/bindings.cpp 226-282](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L226-L282)[tt_torch/csrc/bindings.cpp 368-390](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L368-L390)

### Function: `preprocess_inputs`

Converts PyTorch tensors to runtime tensors with proper layout, tiling, and padding for hardware execution.

**Signature:**

`def preprocess_inputs(    device: tt_mlir.Device,    inputs: List[torch.Tensor],    binary: tt_mlir.Binary,    program_idx: int,    tensor_start_idx: int) -> List[tt_mlir.Tensor]:`
**Implementation:**

*   Ensures all inputs are contiguous (converts in-place if needed)
*   Creates runtime tensors from PyTorch tensors via `create_tensor`
*   Queries expected layout from binary for each input
*   Applies layout transformations (tiling, padding) via `toLayout`
*   Marks tensors as retained to prevent premature deallocation

**Sources:**[tt_torch/csrc/bindings.cpp 226-258](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L226-L258)

### Function: `run`

Executes a compiled binary synchronously and returns PyTorch tensors on the host.

**Signature:**

`def run(    device: tt_mlir.Device,    binary: tt_mlir.Binary,    program_idx: int,    rt_inputs: List[tt_mlir.Tensor]) -> List[torch.Tensor]:`
**Implementation:**

*   Submits program to device via `tt::runtime::submit`
*   Creates PyTorch tensors from all output runtime tensors
*   Deallocates all runtime tensors after conversion
*   Handles cases where output tensors are reused from inputs

**Sources:**[tt_torch/csrc/bindings.cpp 368-390](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L368-L390)

### Function: `run_async`

Executes a compiled binary asynchronously and returns runtime tensors still on the device.

**Signature:**

`def run_async(    device: tt_mlir.Device,    binary: tt_mlir.Binary,    program_idx: int,    rt_inputs: List[tt_mlir.Tensor]) -> List[tt_mlir.Tensor]:`
**Implementation Details:**

*   Submits program to device via `tt::runtime::submit`
*   Returns runtime tensors immediately without untilization
*   Handles output tensor aliasing: if an output shares a handle with an input, creates a host copy to avoid data corruption
*   Does not deallocate tensors automatically
*   Enables device-to-device communication for multi-chip execution

**Key Differences from `run`:**

*   Returns `List[tt_mlir.Tensor]` instead of `List[torch.Tensor]`
*   Tensors remain on device (not untilized or copied to host)
*   User must manually call `to_host` to retrieve outputs when needed
*   Enables pipelining multiple operations on device
*   Used by Executor when `async_mode=True` or for intermediate outputs in multi-device graphs

**Sources:**[tt_torch/csrc/bindings.cpp 260-282](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L260-L282)[tt_torch/dynamo/executor.py 533-539](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L533-L539)

### Function: `to_host`

Converts runtime tensors to PyTorch tensors on the host, with optional deallocation.

**Signature:**

`def to_host(*args, deallocate_tensor: bool = True) -> Union[List[torch.Tensor], Dict]:`
**Behavior:**

*   Handles variable arguments (runtime tensors, tuples, or dict-like objects)
*   Untilizes each runtime tensor via `toHost(tensor, untilize=true)`
*   Creates PyTorch tensors with proper shape, stride, and data type
*   Optionally deallocates runtime tensors after conversion

### Conversion: PyTorch → Runtime Tensor

**Function:**`create_tensor` converts a PyTorch tensor to a runtime tensor:

*   Requires tensor to be contiguous
*   Computes accurate stride via `calculateStride` (handles edge cases)
*   Uses borrowed data pointer for efficiency (no copy)
*   Handles empty tensors by creating owned tensors instead

**Function:**`create_torch_tensor` converts a runtime tensor to PyTorch:

*   Untilizes tensor and brings to host memory
*   Creates empty PyTorch tensor with matching shape/stride/dtype
*   Copies data via `tt::runtime::memcpy` using temporary runtime tensor
*   Returns owned PyTorch tensor

## Tensor Lifecycle and Memory Management

Understanding tensor lifecycle is critical for efficient execution and avoiding memory leaks.

### Tensor States and Transitions

### Memory Ownership Models

The bindings support two ownership models for host tensors:

| Model | Created By | Behavior | Use Case |
| --- | --- | --- | --- |
| **Borrowed** | `createBorrowedHostTensor` | Points to existing PyTorch memory | Normal tensors with data |
| **Owned** | `createOwnedHostTensor` | Allocates new memory | Empty tensors or copies |

**Key Implementation Detail:** In [tt_torch/csrc/bindings.cpp 95-129](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L95-L129) the `create_tensor` function checks if the input tensor is empty (any dimension == 0). Empty tensors use owned memory because the borrowed pointer would be invalid. Non-empty tensors use borrowed memory to avoid unnecessary copies.

### Tensor Retain Flag

Runtime tensors have a retain flag that controls deallocation:

`# In preprocessing (bindings.cpp:253)tt::runtime::setTensorRetain(tensor, retain=True)`
This prevents premature deallocation when tensors are used as both inputs and outputs. The `run` function explicitly deallocates tensors after converting to PyTorch, while `run_async` leaves tensors on device for later retrieval.

### Executor Caching Strategy

The `Executor` class implements a 3-level caching strategy for runtime tensors to avoid repeated preprocessing:

**Cache Lifecycle:**

1.   **Initialization**: Executor creates cache entries for known buffers and constants
2.   **First Use**: `get_inputs` populates cache with runtime tensors
3.   **Subsequent Calls**: Runtime tensors reused without preprocessing
4.   **Invalidation**: Cache invalidated if constant set changes between graph breaks

**Sources:**[tt_torch/dynamo/executor.py 176-203](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L176-L203)[tt_torch/dynamo/executor.py 284-323](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L284-L323)[tt_torch/dynamo/executor.py 325-456](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L325-L456)

## System Description Management

The module provides functions to create and manage hardware system descriptors that describe the available Tenstorrent devices.

### Function: `create_system_desc`

Generates a system descriptor file for a given device, which is used by the TTIR-to-TTNN compiler to optimize for hardware characteristics.

**Signature:**

`def create_system_desc(device: tt_mlir.Device, descriptor_path: str):    """    device: Open device to describe    descriptor_path: Output path for descriptor file    """`
**Usage in Compilation:** The system descriptor is passed to `compile_ttir_to_bytestream` to inform compilation decisions about:

*   Number of available cores
*   Memory hierarchy (L1, DRAM)
*   Interconnect topology

## Binary Operations

The module provides utilities for working with compiled flatbuffer binaries that represent executable programs.

### Binary Class API

The `Binary` class wraps a flatbuffer-encoded program and provides introspection capabilities:

`import tt_mlir # Method 1: Create from bytestream (returned by compile_ttir_to_bytestream)binary = tt_mlir.create_binary_from_bytestream(bytestream) # Method 2: Direct compilation returns bytestreamttir = "..."  # TTIR IR stringsystem_desc = "/path/to/device.ttsys"bytestream, ttnn_str = tt_mlir.compile_ttir_to_bytestream(    ttir, system_desc,     len_activations=1,     len_graph_constants=2)binary = tt_mlir.create_binary_from_bytestream(bytestream) # Query program metadataprogram_idx = 0num_inputs = binary.getProgramInputs(program_idx)num_outputs = binary.getProgramOutputs(program_idx)print(f"Program {program_idx}: {num_inputs} inputs, {num_outputs} outputs") # Convert to JSON for debuggingjson_str = binary.asJson() # Or convert bytestream directly to JSON without Binary objectjson_str = tt_mlir.bytestream_to_json(bytestream)`
### Binary Lifecycle in Executor

The `Binary` object is used in multiple stages:

1.   **Preprocessing**: `getLayout` queries expected tensor layouts for each input
2.   **Execution**: Passed to `tt::runtime::submit` to execute the program
3.   **Introspection**: `getProgramInputs/Outputs` used for validation and debugging

## Environment and Configuration

### Build-Time Compilation Flags

The module behavior is controlled by preprocessor definitions set during build:

| Flag | Effect |
| --- | --- |

| `TT_RUNTIME_DEBUG=1` | Enables debug hooks and op introspection |
| `TTMLIR_ENABLE_OPMODEL=1` | Enables optimization pass in compiler |

## Module Installation

The `tt_mlir` module is installed via the wheel build process:

1.   **CMake Build**: Compiles `tt_mlir.so` from C++ sources
2.   **Library Linking**: Links against TTMLIRCompiler, TTMLIRRuntime, torch_python
3.   **RPATH Configuration**: Sets runtime library path to find dependencies
4.   **Wheel Packaging**: Includes module in tt_torch package

**Installation Locations:**

*   **Source build**: `install/lib/tt_mlir.so`
*   **Wheel install**: `site-packages/tt_mlir.so` (top-level, not in tt_torch package)
