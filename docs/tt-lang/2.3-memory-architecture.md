## Purpose and Scope

This page explains the memory hierarchy and data flow within the `tt-lang` ecosystem. It describes the multi-tier storage system—comprising DRAM, L1, and DST registers—and how the DSL abstracts these hardware components into programmable entities like Tensors, Dataflow Buffers (DFBs), and Tensor Slices.

For deep technical details on individual tiers and register capacity, see [Memory Hierarchy: DRAM, L1, and DST Registers](https://deepwiki.com/tenstorrent/tt-lang/2.3.1-memory-hierarchy:-dram-l1-and-dst-registers). For common optimization strategies like double-buffering and streaming, see [Data Movement Patterns](https://deepwiki.com/tenstorrent/tt-lang/2.3.2-data-movement-patterns).

## Memory Hierarchy Overview

Tenstorrent hardware utilizes a three-tier memory hierarchy designed for high-throughput tile-based computation. Data typically resides in large off-chip DRAM, is moved into on-chip L1 memory via Dataflow Buffers (historically called Circular Buffers), and is finally loaded into the compute engine's DST register file for mathematical operations [include/ttlang/Dialect/TTL/IR/TTLOps.td 28-33](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L28-L33)

## Memory Tiers

### DRAM (Off-Chip)

DRAM acts as the primary backing store for large tensors. In `tt-lang`, tensors in DRAM are represented with memory configurations specifying `DRAM` as the buffer type [python/ttl/ttl_api.py 123-127](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/ttl_api.py#L123-L127) Accessing DRAM requires asynchronous DMA transfers initiated by Data Movement threads [include/ttlang/Dialect/TTL/IR/TTLOps.td 125-129](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L125-L129)

*   **DSL Entity:**`ttnn.Tensor` with `ttnn.DRAM_MEMORY_CONFIG`.
*   **Access Pattern:** Accessed via `ttl.tensor_slice` and moved using `ttl.copy`[include/ttlang/Dialect/TTL/IR/TTLOps.td 80-85](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L80-L85)

### L1 Memory (On-Chip)

L1 is high-speed SRAM local to each Tensix core. It stores both persistent tensors (sharded or interleaved across cores) and transient Dataflow Buffers.

*   **DSL Entity:**`ttnn.Tensor` with `ttnn.L1_MEMORY_CONFIG`.
*   **Capacity:** Limited; the compiler validates that CB allocations do not exceed the core's L1 budget [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 57](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L57-L57)

### Dataflow Buffers (L1 Staging)

Dataflow Buffers (DFBs) are the primary mechanism for moving data between memory and compute. They are fixed-size L1 allocations that synchronize the hardware threads (NCRISC, BRISC, and MATH) [include/ttlang/Dialect/TTL/IR/TTLOps.td 28-33](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L28-L33) The `block_count` parameter determines how many blocks are allocated within the buffer for pipelining [include/ttlang/Dialect/TTL/IR/TTLOps.td 36-37](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L36-L37)

| `ttl.cb_reserve` | Producer thread acquires space in the buffer [include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h 112-115](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOpsUtils.h#L112-L115) |

### DST Registers (Compute Working Set)

The DST register file is the top of the hierarchy. All `ttl.math` operations execute on tiles currently residing in DST.

*   **Management:** Handled automatically by the compiler's `TTLAssignDST` pass, which performs linear scan allocation to maximize register reuse [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 36](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L36-L36)
*   **Optimization:** The `TTLSubblockComputeForDST` pass partitions computations into subblocks that fit within the physical DST capacity [lib/Dialect/TTL/Pipelines/TTLPipelines.cpp 38-42](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Pipelines/TTLPipelines.cpp#L38-L42)

## Data Movement and Slicing

Data flow is managed by creating "views" into larger tensors and copying them into DFBs.

### Tensor Slicing

The `ttl.tensor_slice` creates a view into a tensor at a specific tile position. This does not move data; it creates an SSA value used by `ttl.copy` to define the source or destination of a transfer [include/ttlang/Dialect/TTL/IR/TTLOps.td 80-85](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L80-L85)

### Asynchronous Data Flow

The memory architecture supports overlapping computation and data movement. While the MATH thread operates on a block in the DST registers, the Data Movement threads (BRISC/NCRISC) can fetch the next block from DRAM into a separate L1 slot within the same CB (controlled by `block_count=2` for double-buffering) [include/ttlang/Dialect/TTL/IR/TTLOps.td 36-37](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L36-L37)

## Summary of Memory Layouts

Tensors in the memory hierarchy are organized by layouts which define how tiles are distributed across memory banks and cores.

*   **Tile Layout:** Standard Tenstorrent 32x32 tile organization, required for compute operations [include/ttlang/Dialect/TTL/IR/TTLOps.td 87-91](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L87-L91)
*   **Interleaved/Sharded:** Tensors can be interleaved across DRAM banks or sharded across the L1 memory of multiple cores in a grid [lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp 159-161](https://github.com/tenstorrent/tt-lang/blob/d76e6233/lib/Dialect/TTL/Transforms/ConvertTTLToTTKernel.cpp#L159-L161)
*   **Circular Buffer Type:** Defines the shape, element type, and block count for L1 allocations [include/ttlang/Dialect/TTL/IR/TTLOps.td 40-41](https://github.com/tenstorrent/tt-lang/blob/d76e6233/include/ttlang/Dialect/TTL/IR/TTLOps.td#L40-L41)
