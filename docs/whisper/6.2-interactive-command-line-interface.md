## Purpose and Scope

The Interactive Command-Line Interface provides a human-friendly REPL (read-eval-print loop) for manually debugging RISC-V programs running in Whisper. Users can single-step through instructions, inspect and modify processor state, load programs, set breakpoints via triggers, and examine memory. This interface is distinct from the [Server API](https://deepwiki.com/tenstorrent/whisper/6.1-server-api-and-message-protocol) which is designed for programmatic control and lock-step verification, and from [GDB Remote Debugging](https://deepwiki.com/tenstorrent/whisper/6.4-gdb-remote-debugging) which provides source-level debugging through GDB.

## Execution Control Implementation

### Step Command

The `stepCommand()` method implements single-stepping:

Key behaviors:

*   If hart is in debug mode without a debug park loop, it exits debug mode before stepping and re-enters afterward
*   In MCM mode, each instruction is assigned a tag and retired explicitly using `system_.mcmRetire()`
*   The method supports stepping multiple instructions in a single command

**Sources:**[Interactive.cpp 236-287](https://github.com/tenstorrent/whisper/blob/733fd921/Interactive.cpp#L236-L287)

### Run Command

The `runCommand()` method executes until termination:

*   Exits debug mode if present (unless `dcsr.step` is set)
*   If `dcsr.step` is set, executes a single instruction then re-enters debug mode
*   Otherwise calls `hart.run(traceFile)` which executes until: 
    *   Program writes to `tohost` location
    *   `endpc` address is reached
    *   Maximum instruction count reached
    *   Breakpoint/trigger hit

**Sources:**[Interactive.cpp 217-232](https://github.com/tenstorrent/whisper/blob/733fd921/Interactive.cpp#L217-L232)

### Until Command

The `untilCommand()` method runs until a specified address:

*   Validates the target address is within memory range
*   Exits debug mode if present
*   Calls `hart.untilAddress(addr, traceFile)`
*   Handles `dcsr.step` by single-stepping if set

**Sources:**[Interactive.cpp 179-212](https://github.com/tenstorrent/whisper/blob/733fd921/Interactive.cpp#L179-L212)

## State Modification Implementation

### Poke Command Architecture

The `pokeCommand()` method modifies processor state:

### Vector Register Poke Format

Vector register values are specified as hexadecimal strings:

*   Must start with `0x` prefix
*   Must have even number of hex digits (each byte is 2 digits)
*   Bytes are parsed and reversed before writing to register
*   Example: `poke v v0 0x0123456789abcdef`

### Memory Poke Options

The memory poke command supports additional parameters:

```
poke m <address> <value> [size] [cache] [skipMem]
```

*   `size`: 1, 2, 4, or 8 bytes (default 4)
*   `cache`: 0 or 1 - whether to poke cache directly (for MCM mode)
*   `skipMem`: 0 or 1 - whether to skip actual memory write (for testing)

## Disassembly Commands

The `disassCommand()` method provides three disassembly modes:

### Mode 1: Disassemble Opcodes

```
disass opcode 0x13 0x8082
```

Disassembles individual instruction opcodes directly.

### Mode 2: Disassemble Function

```
disass function main
```

Looks up the function symbol and disassembles all instructions within its address range.

### Mode 3: Disassemble Address Range

```
disass 0x80000000 0x80000100
```

Disassembles all instructions between two addresses.

For each instruction, the output includes:

*   Address (program counter)
*   Instruction opcode (hex)
*   Disassembled text
*   For branches: `(T)` for taken, `(NT)` for not taken
*   For loads/stores: memory address in brackets

## Command Parsing and Keywords

Commands can include persistent keywords that affect subsequent commands:

```
hart=<id> time=<time> <command> [args...]
```

*   `hart=<id>`: Select which hart to operate on (in multi-hart systems)
*   `time=<time>`: Set timestamp for MCM operations

The `processKeywords()` method parses these from the command line and updates `hartId_` and `time_` member variables. These values persist across commands until explicitly changed.

Example:

```
hart=0 time=100 step 1 42
hart=1 peek r x1
```

## Integration with System Components

### Hart Selection

The interactive session maintains a `hartId_` member variable (default 0) that selects which hart to operate on in multi-hart systems. Most commands first call `system_.ithHart(hartId_)` to retrieve the target hart.

### Command Logging

When a `commandLog` file is provided, all executed commands are logged in a format that can be replayed:

```
hart=0 step #42 # ts=100
hart=0 peek r x1 # ts=100 tag=abc
hart=0 poke c mstatus 0x1800 # ts=101 tag=def
```

This enables:

*   Debugging session recording
*   Test case generation
*   Comparison between runs

## Error Handling

Commands return boolean success/failure status:

*   `true`: Command executed successfully
*   `false`: Command failed (invalid arguments, out-of-bounds access, etc.)

Error messages are printed to `std::cerr` with descriptive context:

```
Error: Invalid peek command: peek x
Error: Expecting: peek <item> <addr> or peek pc or peek all
```

The `interact()` loop continues after errors, allowing users to correct mistakes without restarting the session.
