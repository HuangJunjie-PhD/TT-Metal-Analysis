This guide walks developers through setting up a professional development environment for the `tt-metal` repository, building the stack from source, and verifying the installation with initial tests. This page focuses on the technical implementation of the build, test environment, and the CI/CD infrastructure that supports development.

## Overview

The `tt-metal` repository is a two-tier software stack designed for Tenstorrent hardware. It consists of **TT-Metalium** (the low-level runtime and kernel programming framework) and **TT-NN** (the high-level neural network operation library). Developers contributing to this repository work across C++ and Python, requiring a synchronized environment of drivers, firmware, toolchains, and runtime libraries that span from host drivers to device kernels.

The repository supports development for multiple Tenstorrent hardware platforms, enabling high-performance AI model execution using both low-level kernel and high-level neural network abstractions.

You need either physical Tenstorrent hardware or the `ttsim` simulator for development and testing. Supported platforms include:

## Repository Structure

The repository separates host runtime code, device kernel code, and neural network operations clearly:

*   **tt_metal/** holds the low-level runtime, HAL, kernel APIs, and hardware drivers.
*   **ttnn/** provides the TT-NN Python and C++ neural network operation libraries.
*   **tt-train/** contains the C++ training framework interfacing with TT-NN.
*   **tt_metal/third_party/umd/** includes the Userspace Metal Driver abstraction.
*   **tt_metal/hw/** and **tt_metal/tt-llk/** host hardware kernel source and low-level kernel APIs, respectively.

Sources: [INSTALLING.md 122-126](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/INSTALLING.md?plain=1#L122-L126)[CMakeLists.txt 28-36](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/CMakeLists.txt#L28-L36)[CMakeLists.txt 177-178](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/CMakeLists.txt#L177-L178)[tt-train/CMakeLists.txt 1-10](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt-train/CMakeLists.txt#L1-L10)

### 3. Build System Implementation Details

The CMake build targets C++20 and is optimized for developer iteration and static analysis:

*   Compiler debugging options: `-g3`, `-ggnu-pubnames` for rich macro info.
*   Frame pointers kept enabled in non-release builds for profiling (`-fno-omit-frame-pointer`).
*   The `mold` linker is integrated into Docker images for rapid linking (~seconds).
*   The build manages RISC-V SFPI toolchain for device kernel compilation.
*   Supports Unity builds for faster iteration and compiles test targets for Metalium and TTNN.
*   Clang-Tidy integration is enabled, with workflows for full and incremental scans of code parts.

## Build Options and Configuration

Several CMake options control features of the build system and runtime:

| Option | Description | Default |
| --- | --- | --- |
| `ENABLE_CCACHE` | Use ccache for incremental build acceleration | ON (top-level) |
| `TT_METAL_BUILD_TESTS` | Build Metalium Google Test suites | ON |
| `TT_LTO_ENABLED` | Enable Link-Time Optimization | OFF |
| `WITH_PYTHON_BINDINGS` | Build Python bindings | ON |
| `BUILD_SHARED_LIBS` | Build shared instead of static libraries | OFF |

These flags are settable via command line or environment during CMake configuration, allowing developer-tuned builds (debug, release, sanitizers).

Sources: [CMakeLists.txt 135-148](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/CMakeLists.txt#L135-L148)
