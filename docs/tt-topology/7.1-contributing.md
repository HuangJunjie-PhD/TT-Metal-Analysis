## Development Environment Setup

The tt-topology project requires a specific development environment to ensure consistency across contributors and maintain code quality standards.

### Prerequisites

Contributors must have the following dependencies available in their development environment:

| Component | Purpose | Notes |
| --- | --- | --- |
| Python 3.8+ | Runtime environment | Required for `black` and other tools |
| tt-tools-common | Tenstorrent hardware utilities | Core dependency |
| pyluwen v0.6.4 | Hardware interface library | Specific version required |
| networkx 3.1+ | Graph algorithms | For topology calculations |
| pydantic >=1.2 | Data validation | For structured logging |

### Code Quality Tools

The project uses automated code quality enforcement through pre-commit hooks defined in [.pre-commit-config.yaml 1-15](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/.pre-commit-config.yaml#L1-L15)

**Pre-commit Hook Configuration**

**Setting Up Pre-commit Hooks**

To install the pre-commit hooks in your local repository:

`pip install pre-commitpre-commit install`
The hooks will automatically run on each commit and enforce:

*   YAML syntax validation via `check-yaml`
*   File ending consistency via `end-of-file-fixer`
*   Whitespace cleanup via `trailing-whitespace`
*   TOML syntax validation via `check-toml`
*   Python code formatting via `black` (version 23.11.0)

Sources: [.pre-commit-config.yaml 1-15](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/.pre-commit-config.yaml#L1-L15)

Sources: [.github/workflows/community-issue-tagging.yml 1-13](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/.github/workflows/community-issue-tagging.yml#L1-L13)

### Contribution Guidelines

**Code Organization for Contributors**

When contributing to tt-topology, focus on these key areas:

| Component | File Path | Contribution Focus |
| --- | --- | --- |
| CLI Interface | `tt_topology/tt_topology.py` | Command parsing, user experience |
| Topology Logic | `tt_topology/backend.py` | Algorithm implementations, hardware coordination |
| Hardware Constants | `tt_topology/constants.py` | Memory mappings, register definitions |
| Logging Infrastructure | `tt_topology/log.py` | Structured logging, data models |

### Code Standards

All contributions must adhere to the following standards:

**Python Code Style**

*   Code must pass `black` formatting (version 23.11.0)
*   Python 3.8+ compatibility required
*   All imports must be organized and unused imports removed

**Configuration File Standards**

*   YAML files must pass syntax validation
*   TOML files must pass syntax validation
*   No trailing whitespace permitted
*   Consistent file endings required

**Documentation Requirements**

*   Functions that interface with hardware must include docstrings
*   Complex algorithms should include inline comments
*   Configuration changes require documentation updates

Sources: [.pre-commit-config.yaml 10-14](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/.pre-commit-config.yaml#L10-L14)

## Development Best Practices

### Hardware Interface Development

When working on hardware-related code in `tt_topology/backend.py` or `tt_topology/constants.py`, follow these practices:

*   Test changes on actual hardware when possible
*   Validate memory address calculations against hardware documentation
*   Ensure backward compatibility with existing board types (N150, N300, Galaxy)
*   Document any new hardware constants with their purpose and source

### Integration Testing

Before submitting contributions:

1.   Verify all pre-commit hooks pass locally
2.   Test CLI commands with various parameter combinations
3.   Validate log output format compatibility
4.   Ensure no regressions in supported hardware configurations

For detailed testing procedures, see [Testing Framework](https://deepwiki.com/tenstorrent/tt-topology/7.2-testing-framework).

### Submitting Changes

1.   Fork the repository
2.   Create a feature branch from main
3.   Make changes following code standards
4.   Ensure all pre-commit hooks pass
5.   Submit pull request with clear description
6.   Respond to automated labeling and review feedback

The automated workflow in [.github/workflows/community-issue-tagging.yml 10-11](https://github.com/tenstorrent/tt-topology/blob/f86e46c6/.github/workflows/community-issue-tagging.yml#L10-L11) will handle initial triage and labeling of your contribution.
