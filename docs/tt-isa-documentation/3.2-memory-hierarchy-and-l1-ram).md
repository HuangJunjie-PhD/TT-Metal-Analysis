## Purpose and Scope

This document describes the memory hierarchy within a Tensix tile, focusing on the organization, access patterns, and performance characteristics of L1 scratchpad RAM and core-local RAMs. For information about how RISC-V cores access memory through their Load/Store Units, see [Baby RISC-V Cores](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores). For details on how the Tensix Coprocessor's internal register files (SrcA, SrcB, Dst, LReg) interact with L1, see [Register Files: SrcA, SrcB, Dst](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.7-register-files:-srca-srcb-dst).

## Memory Hierarchy Overview

Each Tensix tile contains a multi-level memory hierarchy designed to support high-bandwidth parallel access from multiple clients:

**Sources:**[WormholeB0/TensixTile/L1.md 1-70](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L1-L70)[WormholeB0/TensixTile/BabyRISCV/README.md 76-106](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L76-L106)

The memory hierarchy consists of:

*   **L1 Scratchpad RAM**: 1464 KiB of high-bandwidth shared memory, organized as 16 banks with 16 access ports
*   **Local RAMs**: Small (2-4 KiB) low-latency memories private to each RISC-V core
*   **Coprocessor Register Files**: Internal staging buffers within the Tensix Coprocessor
*   **Off-Tile Memory**: GDDR6 DRAM and remote tile L1, accessible via NoC

## L1 Scratchpad RAM Organization

### Banking and Capacity

L1 is organized as **16 banks** of **91.5 KiB** each, totaling 1464 KiB. Each bank is an independent physical memory capable of one 128-bit (16-byte) read or one 128-bit write per cycle. The banking structure enables high aggregate bandwidth when accesses are distributed across multiple banks.

**Key Characteristics:**

*   **Total capacity**: 1464 KiB
*   **Bank size**: 91.5 KiB (93,696 bytes)
*   **Bank width**: 128 bits (16 bytes)
*   **Address range**: `MEM_L1_BASE` (0x0000_0000) to 0x0016_FFFF
*   **Bank throughput**: 1 read OR 1 write per cycle per bank

**Sources:**[WormholeB0/TensixTile/L1.md 1-14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L1-L14)

### Access Port Architecture

L1 is accessed through **16 access ports**, numbered 0-15. Any port can access any bank, providing full connectivity. However, **bank conflicts** occur when multiple ports attempt to access the same bank simultaneously—all but one port will stall.

Access ports provide functionality beyond basic reads and writes:

| Operation Type | Cycles | Description |
| --- | --- | --- |
| 128-bit read | 1 | Full bandwidth read |
| 128-bit write | 1 | Full bandwidth write |
| Narrow read (<128 bits) | 1 | Reads 128 bits, discards unwanted bits |
| Narrow write (<128 bits) | 5 | Atomic read-modify-write, blocks port and bank |
| Unconditional atomic | 5 | Atomic increment, add, etc. |
| Conditional atomic | 5 | Compare-and-increment, compare-and-set |
| Near-memory accumulate | 5 (atomic) 2 (non-atomic) | 4×FP32, 4×INT32, 8×FP16, 8×BF16 accumulation |

**Sources:**[WormholeB0/TensixTile/L1.md 5-14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L5-L14)

### Port Assignment to Clients

The 16 access ports are statically assigned to different clients within the Tensix tile. The diagram below shows the complete port allocation:

**Detailed Port Assignments:**

| Port | Primary Client(s) | Mux Depth | Notes |
| --- | --- | --- | --- |
| 0 | RISCV B load | 1 | Dedicated |
| 1 | RISCV B store, RISCV B ifetch | 2 | Round-robin mux |
| 2 | Unpacker 0, Packer 0 read, Packer 0 write | 3 | Shared between unpacking and packing |
| 3 | Unpacker 0, Unpacker 1 | 2 | Shared between both unpackers |
| 4 | Unpacker 1 | 1 | Dedicated to unpacker 1 |
| 5 | Packer 0 write, Packer 2 write | 2 | Shared between packer outputs |
| 6 | Packer 3 write | 1 | Dedicated |
| 7 | Mover write | 1 | Dedicated |
| 8 | Mover read | 1 | Dedicated |
| 9 | RISCV T0 load | 1 | Dedicated |
| 10 | RISCV T0 store, RISCV T0 ifetch, Scalar Unit | 3 | Heavily multiplexed |
| 11 | RISCV T1 load | 1 | Dedicated |
| 12 | RISCV T1 store, RISCV T1 ifetch | 2 | Round-robin mux |
| 13 | RISCV T2 load | 1 | Dedicated |
| 14 | RISCV T2 store, RISCV T2 ifetch, RISCV NC | 4 | Heavily multiplexed |
| 15 | NIU #0, NIU #1, Debug clients | 3+ | NoC access, rarely contended |

**Sources:**[WormholeB0/TensixTile/L1.md 15-20](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L15-L20)

Where a port has multiple clients, a round-robin arbiter resolves conflicts. For multi-level muxing (e.g., Port 10 with RISCV T0 store/ifetch/ThCon), independent arbitration occurs at each mux level.

## Bandwidth Characteristics by Client

### Baby RISC-V Cores

Baby RISC-V cores have limited L1 bandwidth due to their use of narrow operations:

| Operation | Bandwidth | Notes |
| --- | --- | --- |
| Store | 1×32-bit / 5 cycles | **6.4 bits/cycle** peak (narrow write overhead) |
| Load (sustained) | 4×32-bit / 7 cycles | **~18.3 bits/cycle** (limited by LSU queue depth) |
| Load (dependent) | 1×32-bit / 8 cycles | **4 bits/cycle** (latency not hidden) |
| Instruction fetch | 1×128-bit / 4+ cycles | Cached, infrequent if high hit rate |

**Recommendation:** RISC-V cores should delegate L1 access to other clients (Mover, Unpackers/Packers, NoC) whenever possible. Store call stacks in local RAM, not L1.

**Sources:**[WormholeB0/TensixTile/L1.md 21-30](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L21-L30)[WormholeB0/TensixTile/BabyRISCV/README.md 54-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L54-L71)

### Mover

The Mover has dedicated read and write ports:

| Operation | Bandwidth | Notes |
| --- | --- | --- |
| Read | 8×128-bit / 11 cycles | **~93.1 bits/cycle** measured |
| Write | 8×128-bit / 11 cycles | **~93.1 bits/cycle** measured |
| Combined | Read + Write simultaneously | Independent ports eliminate read/write conflicts |

**Sources:**[WormholeB0/TensixTile/L1.md 31-34](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L31-L34)

### Unpackers

Unpackers achieve high bandwidth for feeding the Tensix Coprocessor:

| Configuration | Bandwidth | Notes |
| --- | --- | --- |
| Single unpacker | 4×128-bit / cycle | Plus 1×128-bit for BFP exponents |
| Both unpackers | 5×128-bit / cycle | Ports 2/3/4 shared, plus exponent stream |

The theoretical peak may not be achievable if downstream bottlenecks (format conversion, decompression) throttle the unpacker.

**Sources:**[WormholeB0/TensixTile/L1.md 35-40](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L35-L40)

### Packers

Four packers write results back to L1:

| Operation | Bandwidth per Packer | Notes |
| --- | --- | --- |
| Write | 1×128-bit / cycle | Peak throughput |
| Atomic accumulate | 1×128-bit / 5 cycles | 4×FP32 or 8×FP16/BF16 |
| Non-atomic accumulate | 1×128-bit / 2 cycles | Software ensures no overlap |
| Packer 0 L1-to-L1 | 1×128-bit R + 1×128-bit W / cycle | Unique to Packer 0 |

**Sources:**[WormholeB0/TensixTile/L1.md 41-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L41-L46)

### Scalar Unit (ThCon)

The Tensix Coprocessor's Scalar Unit accesses L1 via Port 10:

| Operation | Bandwidth | Notes |
| --- | --- | --- |
| 128-bit write | 1×128-bit / 3 cycles | Preferred for throughput |
| 32-bit write | 1×32-bit / 5 cycles | Narrow write penalty |
| Read (32 or 128-bit) | Limited by 3-cycle issue rate | Shares Port 10 with T0 store/ifetch |

**Sources:**[WormholeB0/TensixTile/L1.md 47-50](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L47-L50)

### NoC

Each NoC has 4×128-bit connections to L1 (2 read, 2 write), sharing Port 15:

| Per NoC | Bandwidth | Notes |
| --- | --- | --- |
| Read | 256 bits/cycle | Theoretical peak |
| Write | 256 bits/cycle | Theoretical peak |
| Combined | 512 bits/cycle total | Port 15 rarely contended |

NoC connections are 256 bits wide between routers, matching the L1 interface bandwidth. Latency is at least 9 cycles per hop.

**Sources:**[WormholeB0/TensixTile/L1.md 51-54](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L51-L54)

## Atomic Operations

L1 supports several atomic operations for synchronization and coordination:

| Atomic Operation | Description | Available Via |
| --- | --- | --- |
| Atomic increment | Increment aligned 32-bit integer, return old value | `ATINCGET`, NoC |
| FIFO atomic | Wait for non-empty/non-full, optionally increment counter | `ATINCGETPTR` |
| Atomic compare-and-swap | Wait for 4-bit value match, then change | `ATCAS`, NoC |
| Atomic swap | Swap aligned 32-bit value | NoC only |

Alternative synchronization mechanisms exist outside L1:

*   **Mailboxes** ([BabyRISCV/Mailboxes.md](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BabyRISCV/Mailboxes.md?plain=1)): FIFO between RISC-V cores
*   **PCBufs** ([BabyRISCV/PCBufs.md](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BabyRISCV/PCBufs.md?plain=1)): Instruction buffers between cores
*   **Tensix Semaphores**: Atomic increment/decrement for Tensix threads
*   **Tensix Mutexes**: Atomic acquire/release for Tensix threads
*   **Backend Config RMWCIB**: Atomic bit manipulation via `RMWCIB` instruction
*   **NoC Overlay Registers**: Can be repurposed for atomic operations

**Sources:**[WormholeB0/TensixTile/L1.md 55-70](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L55-L70)

## Core-Local RAMs

Each of the five Baby RISC-V cores has a small, private, low-latency local RAM:

| RISC-V Core | Local RAM Size | Base Address | Latency |
| --- | --- | --- | --- |
| RISCV B | 4 KiB | `MEM_LOCAL_BASE` (0xFFB0_0000) | 2 cycles |
| RISCV T0 | 2 KiB | `MEM_LOCAL_BASE` (0xFFB0_0000) | 2 cycles |
| RISCV T1 | 2 KiB | `MEM_LOCAL_BASE` (0xFFB0_0000) | 2 cycles |
| RISCV T2 | 2 KiB | `MEM_LOCAL_BASE` (0xFFB0_0000) | 2 cycles |
| RISCV NC | 4 KiB | `MEM_LOCAL_BASE` (0xFFB0_0000) | 2 cycles |

**Key Characteristics:**

*   **Private Access**: Each RAM is accessible only via load/store from its associated core
*   **No Contention**: Never suffers bank conflicts or port contention
*   **Low Latency**: 2-cycle load latency fully hidden by 1 independent instruction
*   **Not NoC-Visible**: Local RAMs are not mapped into NoC address space

**Usage Recommendations:**

*   Place **call stacks** in local RAM (strongly recommended)
*   Store **thread-local variables** in local RAM
*   Cache **frequently-accessed read-only globals** in local RAM
*   Contrast with L1 load latency: ≥8 cycles, requiring 7 independent instructions to hide

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 117-120](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L117-L120)[WormholeB0/EthernetTile/BabyRISCV/README.md 88-91](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/EthernetTile/BabyRISCV/README.md?plain=1#L88-L91)

Additionally, RISCV NC has 16 KiB of **local instruction RAM (IRAM)** at `MEM_NCRISC_IRAM_BASE` (0xFFC0_0000), which is not accessible via load/store instructions but can be populated and executed from directly.

**Sources:**[WormholeB0/TensixTile/BabyRISCV/README.md 91-92](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L91-L92)

## Performance Considerations

### Bank Conflicts

Bank conflicts occur when multiple ports access the same L1 bank simultaneously. The bank address is determined by bits of the byte address:

```
Bank_ID = (ByteAddress >> 4) & 0xF  // Bits [7:4] determine bank
```

Each bank is 91.5 KiB, so addresses are distributed as:

*   Bank 0: 0x000000 - 0x016EFF, 0x017000 - 0x02DEFF, ...
*   Bank 1: 0x000010 - 0x016F0F, 0x017010 - 0x02DF0F, ...
*   ...

**Conflict Avoidance Strategies:**

1.   **Stride by 256 bytes**: Ensures consecutive accesses hit different banks
2.   **Align data structures**: Start buffers on bank boundaries
3.   **Interleave tensors**: Distribute tensor dimensions across banks
4.   **Separate read/write regions**: Use different address ranges for concurrent readers/writers

### Port Contention

Port contention occurs when multiple clients share the same access port. Particularly contentious ports:

| Port | Clients | Contention Risk |
| --- | --- | --- |
| 10 | RISCV T0 store, RISCV T0 ifetch, Scalar Unit | High (3-way mux) |
| 14 | RISCV T2 store, RISCV T2 ifetch, RISCV NC | High (4-way mux) |
| 2 | Unpacker 0, Packer 0 read, Packer 0 write | Medium (3-way mux) |
| 3 | Unpacker 0, Unpacker 1 | Medium (2-way mux) |
| 5 | Packer 0 write, Packer 2 write | Medium (2-way mux) |

**Mitigation Strategies:**

1.   **Schedule NoC transfers**: Move data when RISC-V cores are not heavily using L1
2.   **Use Mover for copies**: Dedicated ports (7, 8) avoid RISC-V contention
3.   **Minimize RISC-V L1 access**: Use local RAMs for stack and thread-local data
4.   **Coordinate Unpacker/Packer timing**: Avoid simultaneous operation on shared ports

### Narrow Operation Overhead

Narrow writes (< 128 bits) incur a **5× bandwidth penalty** due to atomic read-modify-write:

| Operation | Effective Cycles | Bandwidth Multiplier |
| --- | --- | --- |
| 128-bit write | 1 cycle | 1.0× (baseline) |
| 64-bit write | 5 cycles | 0.2× (5× slower) |
| 32-bit write | 5 cycles | 0.2× (5× slower) |
| 16-bit write | 5 cycles | 0.2× (5× slower) |
| 8-bit write | 5 cycles | 0.2× (5× slower) |

**Optimization:** Buffer data and write in 128-bit chunks whenever possible. This is especially critical for RISC-V store instructions and Scalar Unit operations.

**Sources:**[WormholeB0/TensixTile/L1.md 7-14](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L7-L14)[WormholeB0/TensixTile/BabyRISCV/README.md 68-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L68-L71)

### Recommended Access Patterns

**For RISC-V Cores:**

*   Use local RAMs for stack, thread-local storage, and hot variables
*   Delegate bulk transfers to Mover, NoC, or Unpackers/Packers
*   Minimize store operations; prefer 128-bit stores when necessary
*   Schedule loads to hide 8+ cycle latency with independent instructions

**For Tensix Coprocessor:**

*   Configure Unpackers to utilize all available ports (2-5)
*   Use Packer L1-to-L1 mode (Packer 0) for in-place transformations
*   Enable non-atomic accumulation when address ranges don't overlap
*   Align tensor buffers to 256-byte boundaries to avoid bank conflicts

**For NoC Transfers:**

*   Leverage full 256-bit read + 256-bit write bandwidth per NoC
*   Use both NoCs concurrently for maximum off-tile bandwidth
*   NoC-to-NoC transfers (within same tile) avoid network entirely
