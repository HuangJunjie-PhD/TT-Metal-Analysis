The Inference API in TT-Studio is a high-performance gateway built with **FastAPI** that bridges user requests to the **tt-inference-server** (Tenstorrent's optimized model execution engine). It handles model lifecycle execution, parameter validation, and real-time streaming of inference results via Server-Sent Events (SSE).

For details on how requests flow from the frontend through RAG context and agent logic, see [Request Processing](https://deepwiki.com/tenstorrent/tt-studio/7.2-request-processing). For information on the underlying execution engine, refer to the [tt-inference-server documentation](https://github.com/tenstorrent/tt-studio/blob/a6d347af/tt-inference-server%20documentation)

## API Architecture and Integration

The Inference API acts as a wrapper around the `tt-inference-server` workflows. It dynamically imports core logic from the inference server artifact and patches logging and path resolution to operate within the TT-Studio container environment.

## Core Components

### FastAPI Application

The main application entry point is defined in `inference-api/main.py`, which exports the `app` instance from `api.py`. The application includes complex patching logic to ensure that `tt-inference-server` modules correctly identify the artifact directory as the repository root [inference-api/api.py 56-63](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L56-L63)

### TT Inference Client

The Django backend communicates with this API using the `TTInferenceClient`. This client simplifies the process of starting deployments and polling for progress.

| Function | Purpose |
| --- | --- |
| `start_chat_deployment` | Triggers a `/run` request to start a model server workflow. Returns a `job_id`. |
| `TTInferenceRunResult` | A dataclass capturing the status, `job_id`, and API response from the server. |

## API Endpoints

The API exposes endpoints to manage the lifecycle of inference runs and retrieve real-time status.

### Endpoint Reference

| Method | Endpoint | Description |
| --- | --- | --- |
| `POST` | `/run` | Initiates a model workflow (e.g., `server`, `demo`). Supports background execution. |
| `GET` | `/run/progress/{job_id}` | Returns the current status and progress percentage of a deployment job. |
| `GET` | `/run/logs/{job_id}` | Streams real-time stdout/stderr logs for a specific job. |
| `POST` | `/stop/{job_id}` | Terminates a running job and cleans up associated resources. |

**Sources:**[inference-api/api.py 568-570](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L568-L570)[inference-api/api.py 654-656](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L654-L656)[inference-api/api.py 698-700](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L698-L700)[inference-api/api.py 730-732](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L730-L732)

## Deployment Workflow

When a model is deployed, the backend maps board types to specific Tenstorrent device names before calling the Inference API.

### Board Mapping Logic

The system uses `map_board_type_to_device_name` to ensure the `tt-inference-server` receives the correct hardware identifier.

| Board Type | Inference Server Device Name |
| --- | --- |
| `T3000` / `T3K` | `t3k` |
| `N300` | `n300` |
| `P300Cx2` | `p300x2` |
| `GALAXY` | `galaxy` |

**Sources:**[app/backend/docker_control/docker_utils.py 122-155](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L122-L155)

### Job Polling

Because model deployment (especially weight downloading) can take significant time, the API returns a `job_id` immediately. The backend uses `_poll_deployment_to_completion` to monitor the state until it reaches `completed` or `failed`.

**Sources:**[app/backend/docker_control/docker_utils.py 33-69](https://github.com/tenstorrent/tt-studio/blob/a6d347af/app/backend/docker_control/docker_utils.py#L33-L69)

## Streaming and Logging

The Inference API leverages FastAPI's `StreamingResponse` to provide low-latency feedback during long-running operations.

### Log Streaming

The `/run/logs/{job_id}` endpoint reads from a `deque` of log lines captured from the `tt-inference-server` execution thread. It uses a generator to yield lines as they are produced, allowing the frontend to display a live console of the deployment process.

`@app.get("/run/logs/{job_id}")async def run_logs(job_id: str):    # ... logic to find job ...    return StreamingResponse(        log_generator(job_record),        media_type="text/plain"    )`
**Sources:**[inference-api/api.py 698-728](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L698-L728)

### Implementation Details

*   **Thread Safety**: Deployment jobs are run in separate threads using `threading.Thread` to avoid blocking the FastAPI event loop [inference-api/api.py 596-610](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L596-L610)
*   **Log Management**: ANSI escape sequences are stripped from logs using `ANSI_ESCAPE_RE` to ensure clean display in the web UI [inference-api/api.py 119-120](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L119-L120)
*   **Artifact Handling**: The API checks multiple candidate paths (defined by `TT_INFERENCE_ARTIFACT_PATH`) to locate the required `tt-inference-server` workflows [inference-api/api.py 27-41](https://github.com/tenstorrent/tt-studio/blob/a6d347af/inference-api/api.py#L27-L41)
