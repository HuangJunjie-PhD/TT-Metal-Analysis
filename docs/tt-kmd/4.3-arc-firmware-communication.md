## Purpose and Scope

This document describes the protocols and mechanisms used by the kernel driver to communicate with the ARC processor firmware running on Tenstorrent devices. The ARC processor is an on-chip microcontroller responsible for device initialization, power management, reset handling, and telemetry data collection. Different hardware generations use different communication protocols—Wormhole uses a scratch register-based messaging protocol, while Blackhole implements a more sophisticated message queue system (QCB).

For hardware-specific implementation details beyond firmware communication, see [Wormhole Device Driver](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/Wormhole%20Device%20Driver) and [Blackhole Device Driver](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/Blackhole%20Device%20Driver)

* * *

## Overview

The ARC processor runs firmware that manages critical device functions. The kernel driver must communicate with this firmware to:

*   Configure device power states (ASIC states A0/A3)
*   Configure watchdog timeouts for automatic reset
*   Trigger device resets
*   Query telemetry data offsets
*   Send current date/time information
*   Configure PCIe settings

Both hardware generations expose the ARC through the NOC (Network-on-Chip) address space, but use fundamentally different message protocols.

## Wormhole Scratch Register Protocol

### Protocol Mechanism

Wormhole uses a synchronous scratch register-based protocol. The firmware monitors scratch register 5 for commands and responds by writing status back to the same register. Arguments are passed via scratch register 3, and return values are read from scratch register 3.

**Key Registers:**

*   `SCRATCH_REG(5)` at offset 0x74 - Message register (command + status) [wormhole.c 108-112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L108-L112)
*   `SCRATCH_REG(3)` at offset 0x6C - Argument/return value register [wormhole.c 208](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L208-L208)
*   `ARC_MISC_CNTL_REG` at offset 0x100 - Interrupt control [wormhole.c 105-106](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L105-L106)

**Message Format:**

| Bits 15-8 | Bits 7-0 |
| --- | --- |
| `0xAA` (magic) | Message ID |

Sources: [wormhole.c 90-112](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L90-L112)

### Implementation Functions

**Core Messaging Function:**

[wormhole.c 205-238](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L205-L238) - `wormhole_send_arc_fw_message_with_args()`

```
Parameters:
  - reset_unit_regs: Base address of reset unit (BAR4 mapped)
  - message_id: 8-bit command identifier
  - arg0, arg1: 16-bit arguments
  - timeout_us: Timeout in microseconds (0 = fire-and-forget)
  - exit_code: Output parameter for firmware return code

Returns:
  - true if message completed successfully
  - false if firmware not running or timeout occurred
```

**Pre-flight Checks:**

[wormhole.c 199-203](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L199-L203) - `arc_l2_is_running()`

*   Checks if ARC L2 firmware is operational.
*   Reads `POST_CODE_REG` from scratch register 0.
*   Verifies `(post_code & POST_CODE_ARC_L2_MASK) == POST_CODE_ARC_L2`.

**Polling Mechanism:**

[wormhole.c 163-197](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L163-L197) - `arc_msg_poll_completion()`

*   Adaptive polling: ~100 polls over timeout period, minimum 10μs intervals.
*   Detects hardware hang: calls `is_hardware_hung()` which reads `SCRATCH_REG(6)`[wormhole.c 129-138](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L129-L138)
*   Detects unrecognized messages: reads `0xFFFFFFFF` from message register only.

Sources: [wormhole.c 129-238](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L129-L238)

## Blackhole Message Queue Protocol

### Protocol Mechanism

Blackhole uses an asynchronous message queue system with separate request and response queues. Each message is 32 bytes (1 header word + 7 payload words). The firmware processes messages from the request queue and writes responses to the response queue.

**ARC Message Structure:**

[msgqueue.h 11-14](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.h#L11-L14)

`struct arc_msg {    u32 header;      // Message type and flags    u32 payload[7];  // Message-specific data};`
**Queue Control Block (QCB):**

The QCB pointer is stored in scratch register 11 (`ARC_MSG_QCB_PTR`). The driver uses `arc_msg_push` and `arc_msg_pop` to interact with this structure.

| Offset | Field Macro | Description |
| --- | --- | --- |
| +0x00 | `ARC_MSG_QUEUE_REQ_WPTR` | Driver increments after writing request |
| +0x04 | `ARC_MSG_QUEUE_RES_RPTR` | Driver increments after reading response |
| +0x10 | `ARC_MSG_QUEUE_REQ_RPTR` | Firmware increments after reading request |
| +0x14 | `ARC_MSG_QUEUE_RES_WPTR` | Firmware increments after writing response |

Sources: [blackhole.c 63-64](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L63-L64)[msgqueue.h 11-22](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.h#L11-L22)

### Message Flow

Sources: [msgqueue.c 12-132](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.c#L12-L132)

### Implementation Functions

**Request Queue Push:**

[msgqueue.c 12-70](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.c#L12-L70) - `arc_msg_push()`

*   Checks for space in the queue using `(wptr - rptr) % (2 * num_entries)`.
*   Writes the `arc_msg` header and payload to the calculated slot in CSM.
*   Updates the `REQ_WPTR` in the QCB.

**Response Queue Pop:**

[msgqueue.c 72-132](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.c#L72-L132) - `arc_msg_pop()`

*   Polls `RES_WPTR` until it differs from `RES_RPTR`.
*   Reads the 8-word response from the response queue base.
*   Updates the `RES_RPTR` in the QCB.

Sources: [msgqueue.c 12-132](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/msgqueue.c#L12-L132)

## CSM Access Interface

The ARC firmware resides in Common Shared Memory (CSM). The driver provides generation-specific wrappers to read and write this memory.

### Wormhole CSM Access

[wormhole.c 140-161](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L140-L161)

*   `wh_arc_addr_to_sysreg()`: Maps ARC internal addresses to BAR4 system register offsets.
*   `csm_read32()` / `csm_write32()`: Performs `ioread32`/`iowrite32` via `bar4_mapping`.

### Blackhole CSM Access

[blackhole.c 275-291](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L275-L291)

*   `csm_read32()` / `csm_write32()`: Performs NOC-based reads/writes to the ARC location at (8, 0).
*   Validates that the address is within the CSM range (0x0 to 0x1000000).

Sources: [wormhole.c 140-161](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L140-L161)[blackhole.c 56-58](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L56-L58)[blackhole.c 275-291](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L275-L291)

## Summary of Communication Parameters

| Feature | Wormhole | Blackhole |
| --- | --- | --- |
| **Primary Mechanism** | Scratch Registers | Message Queue (QCB) |
| **Command Register** | Scratch 5 | QCB + MSI FIFO |
| **Argument Register** | Scratch 3 | Queue Payload |
| **Queue Header Size** | N/A | 32 bytes |
| **Message Timeout** | Variable (adaptive) | 1000 ms (fixed) |
| **Max Payload** | 2x 16-bit args | 7x 32-bit words |
| **NOC Location** | Indirect (Reset Unit) | Direct (8, 0) |
