### CI Image (Dockerfile.ci)

The CI image is optimized for automated testing with minimal size and fast build times. It builds from the `tt-llk-base-ubuntu-22-04` image and focuses exclusively on testing dependencies.

| Characteristic | Value |
| --- | --- |
| Base Image | `ghcr.io/tenstorrent/tt-llk/tt-llk-base-ubuntu-22-04:${FROM_TAG}` |
| Primary Purpose | Automated CI/CD testing |
| Environment Variables | `GITHUB_ACTIONS="true"`, `PYTEST_RUN_PATH="tests/python_tests"` |
| Installed Tools | pytest ecosystem, coverage tools, basic Python packages |
| Optimization Strategy | Single-layer dependency installation with aggressive cleanup |

The Dockerfile follows a multi-stage approach with cleanup operations to minimize layer size [.github/Dockerfile.ci 13-25](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/Dockerfile.ci#L13-L25):

```
RUN set -e \
    && chmod +x /tmp/scripts/*.sh \
    && /tmp/scripts/install-tests-dependencies.sh \
    && rm -rf /tmp/* \
    && rm -rf /root/.cache/pip \
    && find /usr/local -name "*.pyc" -delete \
    && find /usr/local -name "__pycache__" -type d -exec rm -rf {} +
```

## Content-Addressed Tagging System

The repository implements a content-addressed tagging scheme that ensures Docker images are rebuilt only when relevant dependencies change. This provides efficient caching while guaranteeing reproducibility.

**Diagram: Docker Tag Generation and Cache Invalidation**

Sources: [.github/scripts/get-docker-tag.sh 1-17](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/scripts/get-docker-tag.sh#L1-L17)

The tag generation script computes a SHA-256 hash from seven critical files [.github/scripts/get-docker-tag.sh 8-16](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/scripts/get-docker-tag.sh#L8-L16):

This scheme ensures that:

1.   **Dockerfile changes** trigger rebuilds immediately
2.   **Dependency updates** in `requirements.txt` invalidate the cache
3.   **Build script modifications** force image regeneration
4.   **Unchanged configurations** reuse existing cached images

The `dt-` prefix distinguishes these as "Docker tag" identifiers in the registry.

Sources: [.github/scripts/get-docker-tag.sh 6-16](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/scripts/get-docker-tag.sh#L6-L16)

### UV Package Installer

The installation script uses `uv` (a fast Rust-based Python package installer) instead of vanilla `pip` for performance optimization [.github/scripts/install-tests-dependencies.sh 9](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/scripts/install-tests-dependencies.sh#L9-L9):

The `uv pip install` command accepts standard pip flags with additional performance characteristics:

*   `--index-strategy unsafe-best-match`: Aggressive dependency resolution
*   `--no-cache-dir`: Prevents cache storage in ephemeral containers
*   `-q`: Quiet output to reduce log verbosity

### Virtual Environment Detection

The script adapts to both containerized and local environments [.github/scripts/install-tests-dependencies.sh 11-17](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/scripts/install-tests-dependencies.sh#L11-L17):

*   **With `VIRTUAL_ENV`**: Installs to the activated virtual environment (standard in tt-metalium base image)
*   **Without `VIRTUAL_ENV`**: Uses `--system` flag to install to system Python

## Build Context Optimization

The `.dockerignore` file significantly reduces Docker build context size by excluding unnecessary files, improving build performance and reducing layer sizes.

**Diagram: Build Context Filtering**

Sources: [.github/.dockerignore 1-22](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/.dockerignore#L1-L22)

### Exclusion Rationale

The `.dockerignore` file implements strategic exclusions [.github/.dockerignore 1-22](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/.dockerignore#L1-L22):

| Excluded Path | Reason | Impact |
| --- | --- | --- |
| `.git/` | Git history is large and not needed in container | Major size reduction |
| `tt_llk_*/` | Source code mounted as volume at runtime | Avoids stale code in image |
| `tests/` | Test code mounted as volume at runtime | Enables live test development |
| `build/` | Artifacts generated during container execution | Prevents conflicts |
| `__pycache__/`, `.pytest_cache/` | Python cache regenerated in container | Prevents stale cache issues |
| `.github/scripts/` (default) | Most CI scripts not needed in container | Reduces unnecessary files |

### Selective Inclusion

The file uses negation patterns to include specific required files [.github/.dockerignore 20-21](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/.dockerignore#L20-L21):

```
# All CI scripts excluded by default
.github/scripts/

# Only keep what's needed for building the image
!.github/scripts/install-*.sh
!tests/requirements.txt
```

This ensures that:

*   `install-tests-dependencies.sh` is available for dependency installation
*   `requirements.txt` is accessible during image build
*   All other scripts remain excluded from the build context

## Docker Image Usage Patterns

### CI Pipeline Integration

The CI image is referenced in GitHub Actions workflows through the content-addressed tag system. The typical pattern in `.github/workflows/on-pr.yml` involves:

1.   **Tag Computation**: The `get-docker-tag.sh` script generates the current tag
2.   **Image Selection**: Jobs use the computed tag to pull the correct image
3.   **Cache Optimization**: If the tag exists in the registry, no rebuild occurs
4.   **Automatic Rebuild**: Tag mismatches trigger the `build-images` job

### Interactive Development Workflow

The IRD image supports interactive development sessions:

1.   **Container Launch**: `docker run -it --rm -v $(pwd):/workspace ghcr.io/tenstorrent/tt-llk/tt-llk-ird-ubuntu-22-04:TAG`
2.   **Source Mounting**: Local source code mounted for live editing
3.   **Device Access**: Hardware devices exposed through volume mounts
4.   **Persistent Storage**: User workspace preserved outside container

### Volume Mount Strategy

Both images expect runtime volume mounts for:

| Mount Point | Purpose | Excluded from Image |
| --- | --- | --- |
| `/workspace/tt_llk_*/` | Source code directories | Yes (see `.dockerignore`) |
| `/workspace/tests/` | Test suite | Yes (see `.dockerignore`) |
| `/workspace/build/` | Build artifacts | Yes (ephemeral) |
| `/dev/tenstorrent/*` | Hardware devices | N/A (host devices) |

This separation between image contents and runtime mounts ensures:

*   **Fresh source code**: Developers always work with current code
*   **Smaller images**: No stale or redundant code in layers
*   **Faster iteration**: Code changes don't require image rebuilds
