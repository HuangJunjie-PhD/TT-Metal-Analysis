## Purpose and Scope

This document describes the standardized training loop patterns used across all experiments in TT-Blacksmith. These patterns provide a consistent structure for implementing training scripts that work seamlessly with the manager classes and hardware abstraction layer. The patterns cover initialization, training epochs, validation, checkpointing, and error handling.

For information about the manager classes that the training loops interact with, see [Manager Classes](https://deepwiki.com/tenstorrent/tt-blacksmith/5.1-manager-classes). For details on device-specific operations, see [Device Management and Hardware Abstraction](https://deepwiki.com/tenstorrent/tt-blacksmith/5.2-device-management-and-hardware-abstraction).

## Training Loop Architecture

All training scripts in TT-Blacksmith follow a standardized three-tier architecture: entry point initialization, main training loop, and validation loop. This architecture ensures consistent behavior across different models and frameworks while allowing the device manager to handle hardware-specific operations transparently.

### Training Loop Implementation Pattern

The following code structure appears consistently across all training scripts:

| Component | Standard Implementation |
| --- | --- |
| **Error Handling** | `try-except-finally` block wrapping entire training |
| **Initial Validation** | `model.eval()` → `validate()` → `log_metrics()` → `model.train()` |
| **Epoch Loop** | `for epoch in range(config.num_epochs):` |
| **Batch Loop** | `for batch in tqdm(train_dataloader):` |
| **Gradient Zeroing** | `optimizer.zero_grad()` at start of each step |
| **Batch Preparation** | `batch = device_manager.prepare_batch(batch)` |
| **Model Sharding** | `device_manager.shard_model(model)` |
| **Loss Computation** | Framework-specific but always includes `.item()` for CPU conversion |
| **Backward Pass** | `loss.backward()` followed by `torch_xla.sync(wait=True)` if `use_tt` |
| **Optimizer Step** | `device_manager.optimizer_step(optimizer)` |
| **Metric Logging** | `logger.log_metrics()` with `commit=False` for accumulation, `commit=True` for sync |
| **Checkpointing** | `if checkpoint_manager.should_save_checkpoint(global_step):` |
| **Finalization** | `logger.finish()` in `finally` block |

## Validation Loop Pattern

Validation loops follow a simplified pattern without gradient computation or optimization. They accumulate metrics across the validation dataset and return averaged results.

### Validation Function Signature

### Validation Sequence

### Key Validation Patterns

| Pattern | Implementation | Purpose |
| --- | --- | --- |
| **Gradient Disabling** | `with torch.no_grad():` wrapper | Prevent gradient computation and save memory |
| **Evaluation Mode** | Called before validation: `model.eval()` | Disable dropout, batch norm updates |
| **Training Mode Restore** | Called after validation: `model.train()` | Re-enable training behaviors |
| **Batch Sharding** | `device_manager.prepare_batch(batch)` | Apply same data parallelism as training |
| **Model Sharding** | `device_manager.shard_model(model)` | Apply same tensor parallelism as training |
| **Loss Accumulation** | `total_loss += loss.item()` | Convert to Python scalar for CPU accumulation |
| **XLA Synchronization** | `torch_xla.sync(wait=True)` after loss | Ensure computation completes before .item() |
| **Metric Averaging** | `avg_loss = total_loss / num_batches` | Compute mean over dataset |

## Gradient Accumulation Pattern

For larger effective batch sizes without increasing memory consumption, gradient accumulation is used. This pattern accumulates gradients over multiple micro-batches before updating parameters.

### Gradient Accumulation Implementation

The Llama script demonstrates the canonical gradient accumulation pattern:

**Key Points:**

*   Loss is scaled by `1 / gradient_accumulation_steps` to maintain correct gradient magnitudes
*   `optimizer.zero_grad()` is called only at the start of accumulation cycles
*   `global_step` increments only after full accumulation completes
*   Validation and checkpointing occur based on `global_step`, not micro-batch count

## Memory Management Patterns

### Training Step Scoping

The Llama script extracts the training step into a separate function to ensure large tensors (particularly logits with vocabulary-sized dimensions) are scoped locally and released before gradient synchronization:

This pattern prevents expensive collective communication operations (CCLs) in multi-chip setups by ensuring large intermediate tensors don't propagate through the computation graph.

## Manager Interaction Pattern

The training loop coordinates with four manager classes. Understanding their interaction points is critical for implementing training scripts correctly.

### Manager Call Patterns

| Manager | Method | Call Frequency | Purpose |
| --- | --- | --- | --- |
| **ReproducibilityManager** | `setup()` | Once at script start | Set seeds, configure determinism |
| **DeviceManager** | `prepare_batch()` | Every training/validation step | Shard data for data parallelism |
| **DeviceManager** | `shard_model()` | Every training/validation step | Apply tensor parallelism patterns |
| **DeviceManager** | `optimizer_step()` | Every training step | Handle XLA synchronization |
| **TrainingLogger** | `log_metrics()` | Every `steps_freq` steps | Log training/validation metrics |
| **TrainingLogger** | `info()/error()` | As needed | Log status messages |
| **TrainingLogger** | `finish()` | Once at script end (in finally) | Finalize logging session |
| **CheckpointManager** | `should_save_checkpoint()` | Every step | Check if checkpoint needed |
| **CheckpointManager** | `save_checkpoint()` | When condition met | Save model state |
| **CheckpointManager** | `load_checkpoint()` | Once if resuming | Restore model state |

## Framework-Specific Considerations

### PyTorch XLA Patterns

When `config.use_tt=True`, PyTorch scripts execute on Tenstorrent hardware via torch_xla. This requires specific synchronization points:

| Operation | Synchronization Required | Reason |
| --- | --- | --- |
| After `loss.backward()` | `torch_xla.sync(wait=True)` | Ensure gradients computed before optimizer step |
| After `optimizer.step()` | Handled by `device_manager.optimizer_step()` | Synchronize parameter updates across devices |
| Before `loss.item()` | `torch_xla.sync(wait=True)` | Ensure computation completes before CPU transfer |
| At validation end | `torch_xla.sync(wait=True)` | Synchronize all device computations |

### Loss Function Patterns

Different models use different loss computation patterns based on their architecture:

**Causal Language Models (Llama, Qwen, Gemma, Phi):**

*   Shift logits by one position: `shift_logits = logits[:, :-1, :].contiguous()`
*   Compute loss on shifted predictions vs labels
*   Use `ignore_index=-100` to ignore padding tokens
*   Some use custom one-hot encoding for numerical stability

**Classification Models (ALBERT):**

*   Direct logits-to-labels comparison
*   Use standard `CrossEntropyLoss`
*   Track accuracy metrics in addition to loss

## Template for New Training Scripts

The repository provides a template script that implements all standard patterns:

[blacksmith/tools/templates/test_model_template.py 1-171](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/tools/templates/test_model_template.py#L1-L171)

This template should be used as the starting point for new experiments. It includes:

*   Complete entry point initialization
*   Standard training and validation loops
*   Proper manager interactions
*   Error handling and cleanup
*   Configuration-driven behavior

To create a new training script:

1.   Copy `test_model_template.py` to your experiment directory
2.   Customize model loading in the initialization phase
3.   Adjust loss computation for your task (classification vs generation)
4.   Modify dataset loading for your specific data format
5.   Test with single-chip configuration first, then add distributed training
