## Purpose and Scope

The Golden Model System provides reference implementations that compute mathematically correct results to validate hardware kernel execution. This system simulates the exact transformations that occur in hardware, including format conversions, precision loss, and operation-specific behaviors.

This page covers golden generator implementations, format conversion models, and validation mechanisms. For test execution and device interface, see [Device Interface and Hardware Execution](https://deepwiki.com/tenstorrent/tt-llk/8.5-device-interface-and-hardware-execution). For test orchestration and parameterization, see [TestConfig and Test Orchestration](https://deepwiki.com/tenstorrent/tt-llk/8.2-testconfig-and-test-orchestration) and [Test Parameterization and Format Sweeps](https://deepwiki.com/tenstorrent/tt-llk/8.3-test-parameterization-and-format-sweeps).

## System Architecture

The Golden Model System consists of four primary components: format conversion, fidelity simulation, operation-specific generators, and a registry system.

**Golden Model System Architecture**: The system transforms test inputs through format conversion and precision simulation to produce bit-exact reference results that match hardware behavior. Golden generators are registered via decorators and retrieved through the registry. Format conversion uses `SrcFormatModel` to simulate TF32 transformations, while `FidelityMasking` applies precision reduction to match hardware math modes.

**Sources**: [tests/python_tests/helpers/golden_generators.py 37-119](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L37-L119)[tests/python_tests/test_eltwise_unary_sfpu.py 232-241](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/test_eltwise_unary_sfpu.py#L232-L241)

## Format Conversion: SrcFormatModel

The `SrcFormatModel` class simulates hardware format conversion by decomposing floating-point values into sign, exponent, and mantissa components. This enables bit-exact matching with hardware unpacker behavior.

### TF32 Conversion Pipeline

**TF32 Format Conversion Pipeline**: Input formats are decomposed into sign/exponent/mantissa tuples, exponent biases are adjusted, and mantissa bits are padded or truncated to match the 10-bit TF32 mantissa width. The implicit leading 1 is explicitly stored in the mantissa field.

**Sources**: [tests/python_tests/helpers/golden_generators.py 232-413](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L232-L413)

### Format-Specific Conversion Methods

| Method | Format | Mantissa Width | Exponent Bias | Transformation |
| --- | --- | --- | --- | --- |
| `_fp32_to_tf32()` | Float32 | 23 → 10 bits | 127 | Truncate 13 LSBs |
| `_fp16b_to_tf32()` | Float16_b, Bfp8_b | 7 → 10 bits | 127 | Pad 3 LSBs with 0 |
| `_fp16_to_tf32()` | Float16 | 10 → 10 bits | 15 → 127 | Direct copy |
| `_mxfp8r_to_tf32()` | MxFp8R | Delegate to _fp16b_to_tf32() | 127 | Via bfloat16 |
| `_mxfp8p_to_tf32()` | MxFp8P | Delegate to _fp16b_to_tf32() | 127 | Via bfloat16 |

## Fidelity Masking

The `FidelityMasking` class applies mantissa bit masks to simulate hardware precision modes (LoFi, HiFi2, HiFi3, HiFi4). This is critical for matmul operations where hardware performs multiple fidelity iterations.

### Fidelity Iteration Masks

**Fidelity Iteration Masks**: Each fidelity mode performs multiple iterations with different mantissa masks applied to operands A and B. LoFi uses 5+7 bits, while HiFi4 accumulates all 10 bits across four iterations. Quasar uses different masks than Wormhole/Blackhole.

## Golden Generator Registry

The registry system uses decorators to register golden generator classes and provides lookup via `get_golden_generator()`.

### Registry Implementation

**Golden Generator Registry**: Classes decorated with `@register_golden` are instantiated and stored in `golden_registry`. Tests retrieve instances via `get_golden_generator(class)` and call them as functions via `__call__()`.

Computes matrix multiplication with fidelity masking applied across multiple iterations.

**Key Features**:

*   Supports multi-tile matmul with dimension validation
*   Iterates based on `MathFidelity` enum (LoFi=1, HiFi2=2, HiFi3=3, HiFi4=4 iterations)
*   Optional tilization of output

## MX Format Quantization

Microscaling (MX) formats use shared exponents across 32-element blocks, requiring special quantization handling.

### Quantization Process

**MX Format Quantization Pipeline**: Input tensors undergo pack→unpack roundtrip to simulate quantization effects. Large tensors are processed in chunks (4-face, 2-face, or 1-face) with appropriate padding. The quantized tensor matches what hardware unpacker produces from L1 memory.

**Sources**: [tests/python_tests/helpers/golden_generators.py 121-229](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L121-L229)

### Chunk Configuration

| Chunk Size | Min Elements | Num Faces | Use Case |
| --- | --- | --- | --- |
| 1024 | 1024 | 4 | Full tiles |
| 512 | 512 | 2 | Half tiles |
| 256 | 256 | 1 | Single face |
| 256 (padded) | < 256 | 1 | Partial face (with zero padding) |

**Implementation Note**: The chunked approach handles variable tensor sizes by selecting the largest valid chunk configuration for remaining elements. Padding is applied only when necessary to reach minimum chunk size.

**Sources**: [tests/python_tests/helpers/golden_generators.py 175-229](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L175-L229)

## Key Design Principles

1.   **Bit-Exact Matching**: Golden models replicate hardware transformations precisely, including format conversion artifacts and precision loss.

2.   **Format-Agnostic Operations**: Core operations use PyTorch, but all inputs/outputs pass through `SrcFormatModel` to match hardware behavior.

3.   **Composable Generators**: Golden generators can be chained (e.g., matmul → tilize → SFPU) to validate multi-stage pipelines.

4.   **Architecture-Specific Behavior**: Fidelity masks and operations adjust based on `ChipArchitecture` (Wormhole/Blackhole/Quasar).

5.   **Registry-Based Lookup**: Decorator pattern enables test-time generator selection without hardcoded imports.
