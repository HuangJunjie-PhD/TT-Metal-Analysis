This document details the comprehensive testing framework for individual operators in the PyBuda system. It covers how operators are verified across different configurations, input shapes, data formats, and mathematical fidelity levels. For information about testing complete models, see [Model Testing](https://deepwiki.com/tenstorrent/tt-buda/3.3-model-testing), and for automated graph generation testing, see [Random Graph Generator](https://deepwiki.com/tenstorrent/tt-buda/3.1-random-graph-generator).

## 1. Testing Framework Overview

The operator testing framework systematically validates the correctness and functionality of PyBuda operators through various test configurations. It follows a comprehensive test plan that ensures operators behave correctly across different:

*   Operand types and sources
*   Tensor shapes and dimensions
*   Data formats and math precision levels
*   Special attributes specific to each operator

**Test Plan Implementation**: The tests systematically verify different aspects of operators:

1.   **Operator implementation correctness** - Compares results against PyTorch's reference implementation
2.   **Hardware/backend support** - Ensures operators work properly on various backends
3.   **Shape handling** - Tests different tensor dimensions and sizes
4.   **Data flow validation** - Verifies correct behavior regardless of input source
5.   **Numerical accuracy** - Tests different data formats and math fidelity settings

## 2. Test Components and Architecture

The testing system consists of multiple components that work together to create comprehensive test coverage:

### 2.1 Test Models

Each test model is a PyBudaModule that implements a specific test scenario. The primary test models include:

*   **ModelFromAnotherOp**: Tests operators fed by outputs from other operators
*   **ModelFromHost**: Tests operators receiving inputs directly from the host
*   **ModelFromDramQueue**: Tests operators with inputs from DRAM queue
*   **ModelFromDramQueuePrologued**: Tests operators with prologued inputs
*   **ModelConstEvalPass**: Tests operators with constant inputs that can be evaluated at compile time
*   **ModelOpSrcFromTmEdge1/2**: Tests operators fed by tensor manipulation operations

These models enable testing operators with different input sources to ensure correct behavior in all scenarios.

Sources:

*   [pybuda/test/operators/eltwise_binary/test_eltwise_binary.py 84-235](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/test/operators/eltwise_binary/test_eltwise_binary.py#L84-L235)

### 2.2 Verification Utilities

The `VerifyUtils` class provides the core verification functionality:

`def verify(    model: PyBudaModule,    test_device: TestDevice,    input_shapes: List[TensorShape],    input_params: List[Dict] = [],    pcc: Optional[float] = None,    input_source_flag: InputSourceFlags = None,    dev_data_format: pybuda.DataFormat = None,    math_fidelity: pybuda.MathFidelity = None,)`
This function:

1.   Configures the compiler based on input parameters
2.   Sets up the verification configuration
3.   Runs the `verify_module` function to execute the PyBuda and reference implementations
4.   Compares the results for correctness

The `NetlistValidation` class enables checking the compiled netlist to verify certain properties:

`def get_value(self, key_path: str)`
This method retrieves values from the netlist to validate aspects like queue locations, op presence, and prologuing.

## 4. Supported Operator Types

The testing framework validates four main categories of operators:

### 4.1 Elementwise Binary Operators

These operators perform element-by-element operations on two input tensors. Examples include:

*   Arithmetic operators: Add, Subtract, Multiply, Divide, Power
*   Comparison operators: Greater, GreaterEqual, Less, LessEqual, Equal, NotEqual
*   Selection operators: Max, Min, Heaviside
*   Other: BinaryStack

## 5. Testing Process

The operator testing process follows these steps:

### 5.1 Test Case Setup

Each test case configures:

1.   The operator to test
2.   The test model (input source scenario)
3.   Input shapes
4.   Additional parameters like data format and math fidelity

For example, testing an Add operator with inputs from another operation:

`verify(    test_device=test_device,    model_type=ModelFromAnotherOp,    input_operator="Add",    input_shape=(1, 32, 64),    number_of_operands=2,    dev_data_format=pybuda.DataFormat.Float16_b,    math_fidelity=pybuda.MathFidelity.HiFi4,)`
Sources:

*   [pybuda/test/operators/eltwise_binary/test_eltwise_binary.py 409-463](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/test/operators/eltwise_binary/test_eltwise_binary.py#L409-L463)

### 5.2 Verification Process

The `verify_module` function:

1.   Creates input tensors with the specified shapes
2.   Executes the PyTorch reference implementation
3.   Compiles and runs the PyBuda implementation
4.   Compares the results for correctness

Success criteria include:

*   Results match within the specified precision (PCC threshold)
*   No compilation errors
*   No runtime errors
*   Netlist validation passes

Sources:

*   [pybuda/test/operators/utils/utils.py 71-123](https://github.com/tenstorrent/tt-buda/blob/74755d1f/pybuda/test/operators/utils/utils.py#L71-L123)

### 5.3 Netlist Validation

After compilation, tests validate the generated netlist to ensure properties like:

*   Correct input queue location (host vs. DRAM)
*   Proper prologuing for constants
*   Constant evaluation where appropriate

For example, checking that inputs are correctly placed in DRAM:

`netlist = NetlistValidation()assert netlist.get_value("/queues/x/loc") == 'dram'assert netlist.get_value("/queues/y/loc") == 'dram'`

## 7. Integration with PyBuda Framework

The operator testing system integrates with the broader PyBuda framework:

The operator testing system provides essential validation of individual components that are then used in higher-level testing:

1.   **Operator Repository Validation** - Ensures each operator functions correctly in isolation
2.   **Compilation Pipeline Validation** - Tests the compiler's handling of different operator configurations
3.   **Runtime System Validation** - Verifies correct execution of operators on the hardware

This layered testing approach ensures that issues are caught at the appropriate level, making debugging more straightforward and efficient.
