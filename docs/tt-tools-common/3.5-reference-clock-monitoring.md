# Reference Clock Monitoring

Relevant source files
*   [tt_tools_common/utils_common/tools_utils.py](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py)

## Purpose and Scope

The reference clock monitoring system provides functionality to read and validate the hardware reference clock counter on Tenstorrent chips. This system verifies that the `REFCLK_COUNTER` register pair is incrementing at the expected frequency, independent of the ARC clock (`ARCLK`). The monitoring system is primarily used during chip initialization and reset validation to ensure proper hardware operation.

For information about chip reset procedures that may use these validation functions, see [Hardware Reset System](https://deepwiki.com/tenstorrent/tt-tools-common/2-hardware-reset-system). For general chip detection and communication verification, see [Chip Detection and Communication](https://deepwiki.com/tenstorrent/tt-tools-common/3.3-chip-detection-and-communication).

## Reference Clock Counter Registers

The reference clock counter is maintained in hardware as a 64-bit value split across two 32-bit registers:

| Register | Address | Bits | Description |
| --- | --- | --- | --- |
| `ARC_RESET.REFCLK_COUNTER_HIGH` | AXI-translated | [63:32] | Upper 32 bits of counter |
| `ARC_RESET.REFCLK_COUNTER_LOW` | AXI-translated | [31:0] | Lower 32 bits of counter |

The counter increments continuously at the reference clock frequency, providing a stable timing reference independent of the ARC processor clock. This separation allows validation of clock behavior without dependency on ARC firmware operation.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 205-206](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L205-L206)

## Counter Reading Mechanism

### `read_refclk_counter(chip)` Function

The counter read operation must handle potential rollover of the lower 32-bit register during the read sequence. The implementation uses a double-read strategy:

The algorithm prevents race conditions when the low register rolls over to zero during the read sequence:

1.   Read high register (first sample)
2.   Read low register
3.   Read high register again (second sample)
4.   If high values differ, a rollover occurred during the read; re-read the low register
5.   Combine `high2` and `low` into a 64-bit value

**Chip Compatibility:** This function returns `None` for Grayskull chips, as they do not implement the `REFCLK_COUNTER` registers.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 202-214](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L202-L214)

## Rate Calculation

### `refclk_counter_rate(chip, delay_interval)` Function

The reference clock frequency is measured by sampling the counter twice with a known time delay:

| Parameter | Type | Default | Description |
| --- | --- | --- | --- |
| `chip` | `PciChip` | - | Target chip instance |
| `delay_interval` | `float` | `0.1` | Delay between samples in seconds |
| **Returns** | `float` | - | Measured frequency in MHz |

**Algorithm:**

**Formula:**

```
Rate (MHz) = (ΔCounter × 1000) / ΔTime(ns)
```

Where:

*   `ΔCounter` = `after_refclk - before_refclk` (counter ticks)
*   `ΔTime(ns)` = `after_ns - before_ns` (nanoseconds elapsed)
*   Factor of 1000 converts from GHz (counter ticks per ns) to MHz

**Sources:**[tt_tools_common/utils_common/tools_utils.py 218-227](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L218-L227)

## Validation Operations

The monitoring system provides two validation functions to detect hardware issues.

### Read Speed Validation

#### `check_refclk_counter_read_speed(chip)` Function

This function verifies that counter read operations complete within expected time bounds. Slow reads may indicate PCIe bus issues, driver problems, or hardware faults.

**Operation:**

1.   Performs 100 consecutive counter reads
2.   Measures total elapsed time
3.   Calculates average time per read
4.   Prints warning if reads exceed 100 microseconds each

**Expected Performance:** Each read should complete in under 100 microseconds. Exceeding this threshold suggests potential issues with:

*   PCIe communication latency
*   AXI bus contention
*   Driver overhead
*   Hardware access bottlenecks

**Sources:**[tt_tools_common/utils_common/tools_utils.py 230-240](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L230-L240)

### Rate Accuracy Validation

#### `check_refclk_counter_rate(chip, expected_refclk, accuracy)` Function

This function validates that the measured clock rate matches the expected hardware specification within a tolerance threshold.

| Parameter | Type | Description |
| --- | --- | --- |
| `chip` | `PciChip` | Target chip instance |
| `expected_refclk` | `float` | Expected reference clock frequency in MHz |
| `accuracy` | `float` | Acceptable deviation percentage (e.g., `5.0` for ±5%) |
| **Returns** | `str` or `None` | Error message if out of range, `None` if valid |

**Validation Logic:**

**Deviation Formula:**

```
deviation = |expected - observed| / min(expected, observed)
```

The minimum value is used in the denominator to ensure symmetric tolerance bounds regardless of whether the measured value is higher or lower than expected.

**Example:**

*   Expected: 100 MHz
*   Accuracy: 5.0 (±5%)
*   Measured: 94 MHz
*   Deviation: |100 - 94| / 94 = 6.38%
*   Result: "REFCLK_COUNTER outside of allowed range: 94"

**Sources:**[tt_tools_common/utils_common/tools_utils.py 243-252](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L243-L252)

## Chip Compatibility and Limitations

### Supported Architectures

| Chip Type | REFCLK_COUNTER Support | Notes |
| --- | --- | --- |
| Blackhole | ✓ Yes | Full support via `ARC_RESET` registers |
| Wormhole | ✓ Yes | Full support via `ARC_RESET` registers |
| Grayskull | ✗ No | `read_refclk_counter()` returns `None` |

The `read_refclk_counter()` function explicitly checks for Grayskull architecture and returns `None` if detected:

`if chip.as_gs():    return None`
This check prevents attempting to access non-existent registers on Grayskull hardware.

### AXI Translation Requirement

All register accesses use the `chip.axi_translate()` method to convert symbolic register names to physical addresses. This abstraction layer handles chip-specific address mapping and ensures portability across different hardware revisions.

**Sources:**[tt_tools_common/utils_common/tools_utils.py 203-206](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/utils_common/tools_utils.py#L203-L206)

## Integration with Reset and Validation Workflows

The reference clock monitoring functions are typically invoked during chip initialization and reset validation sequences. While these functions are defined in `tools_utils.py`, they are used by higher-level reset implementations to verify hardware state after reset operations.

**Common usage patterns:**

1.   **Post-reset validation:** After executing a chip reset, call `check_refclk_counter_rate()` to verify the reference clock resumed operation at the expected frequency.

2.   **Hardware diagnosis:** Use `check_refclk_counter_read_speed()` to detect PCIe communication issues before attempting complex operations.

3.   **Clock stability monitoring:** Periodically sample `refclk_counter_rate()` during long-running operations to detect clock drift or instability.
