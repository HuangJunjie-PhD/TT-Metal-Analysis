## Purpose and Scope

This page documents the TTNN C++ operation registration system, the template dispatch mechanism, and the device operation interface used for backend execution on Tenstorrent hardware. It explains how operations are implemented as C++ structs, how they are registered and exposed to Python via an automated system, and how device-level programs and kernels are created and dispatched through the `ttnn::device_operation` framework.

This is a LEAF page with no child pages. For a higher-level overview of the TTNN Python API, see section 4.1. For in-depth low-level runtime and kernel execution details, refer to section 2.4 Program and Kernel System.

* * *

## Operation Registration Pattern

### Overview

TTNN operations are implemented in C++ as specialized structs called _Execute Structs_ that encapsulate operation logic as static `invoke` methods. They are registered via a compile-time templated registration function, producing an operation object annotated with metadata for use in the Python binding layer.

At runtime, a Python auto-registration system scans the C++ submodules for such operation objects, extracts their metadata, and registers them into the Python `ttnn` module. This allows seamless dispatch from Python to the optimized C++ backend.

### Registration Workflow

Title: TTNN Operation Registration Flow

*   **Execute Struct**: Implements operation logic in C++.
*   **Registration Statement**: Using `ttnn::register_operation` template binds a name to the execute struct.
*   **Operation Object**: Annotated with `__ttnn_operation__`; visible to Python.
*   **Python Auto-Registration**: Scans C++ modules and registers operations into Python namespace.

**Sources:**[ttnn/api/ttnn/operation.hpp 1-10](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/operation.hpp#L1-L10)[ttnn/api/ttnn/device_operation.hpp 27](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L27-L27)

## Execute Struct Pattern

### Structure and Implementation

Operations in TTNN are implemented as C++ structs with one or more static `invoke` methods. These methods define the operation logic. The typical pattern supports multiple overloads depending on input parameters such as presence or absence of bias, activations, or memory configurations.

Title: Execute Struct Architecture

The return types can be:

*   `Tensor` for single output [ttnn/api/ttnn/device_operation.hpp 12](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L12-L12)
*   `std::vector<Tensor>` if multiple outputs are produced.
*   `std::optional<Tensor>` for optional outputs.

### Generic Operation Pattern

Inside the Execute struct, execution often involves the `generic_op` interface for arbitrary program execution, or a specialized `device_operation` dispatch.

The `ttnn::generic_op` interface allows for executing custom TT-Metal programs by taking a vector of `ttnn::Tensor` objects and a `tt::tt_metal::ProgramDescriptor`[tt_metal/api/tt-metalium/program_descriptors.hpp 28-33](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L28-L33) to define the program's kernels, circular buffers, and semaphores.

## Device Operation Interface and Program Factories

The core execution interface for device operations in TTNN uses a templated pattern in the `ttnn::device_operation` namespace. Key abstractions include:

*   **CachedProgram**: Wraps a TT-Metal `Program` with shared kernel handles and variables [ttnn/api/ttnn/device_operation.hpp 34-35](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L34-L35)
*   **CachedMeshWorkload**: Extends the program cache to multi-device mesh execution [ttnn/api/ttnn/device_operation.hpp 43-44](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L43-L44)
*   **Program Hash**: Computed using `compute_program_hash` to determine if a program can be reused from the cache [ttnn/api/ttnn/device_operation.hpp 57-69](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L57-L69)
*   **WorkloadFactory**: Provides `create_at()` or `create_mesh_workload()` to instantiate programs [ttnn/api/ttnn/device_operation.hpp 73-115](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L73-L115)

### Device Operation Dispatch Flow

The `ttnn::device_operation` namespace provides the infrastructure to map operation attributes and tensor arguments to a hardware program.

## Tensor and Memory Management in C++ Backend

The C++ backend manages tensors through the `ttnn::Tensor` class [ttnn/api/ttnn/device_operation.hpp 12](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/device_operation.hpp#L12-L12) Operations interact with tensor storage which can be `HostStorage` or `DeviceStorage`.

### Key Data Movement and Allocation

The system handles allocation and movement across host and device boundaries, specifically for mesh configurations.

### Core Selection for Sharded Tensors

The backend provides `get_optimal_worker_cores_for_sharded_tensor` to identify which cores in a grid actually contain data shards, preventing kernels from being launched on idle cores [ttnn/api/ttnn/tensor/tensor_utils.hpp 35-36](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor_utils.hpp#L35-L36)

## Build System and Program Optimization

TTNN operations are organized into modular C++ libraries via CMake [ttnn/CMakeLists.txt 92-93](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/CMakeLists.txt#L92-L93) The backend supports unity builds for faster compilation and uses precompiled headers (PCH) to reduce build times [ttnn/CMakeLists.txt 17-46](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/CMakeLists.txt#L17-L46)

### PCH Strategy

TTNN uses a tiered PCH strategy to balance build speed and developer iteration:

*   **PCHLean**: Contains only STL, `fmt`, and `json` headers. Immune to TT-Metalium header churn [ttnn/CMakeLists.txt 21-36](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/CMakeLists.txt#L21-L36)
*   **PCHFull**: Includes volatile headers like `device_operation.hpp`. Used only for targets implementing device operations [ttnn/CMakeLists.txt 39-73](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/CMakeLists.txt#L39-L73)

### Program Descriptor Patching

A critical optimization for cache hits is **program descriptor patching**. The system allows users to define `BufferBinding`[tt_metal/api/tt-metalium/program_descriptors.hpp 113-117](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L113-L117) and `CommonBufferBinding`[tt_metal/api/tt-metalium/program_descriptors.hpp 121-124](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L121-L124) within a `KernelDescriptor`.

These bindings allow the framework to resolve buffer base addresses that change on every dispatch. On a cache hit, the framework patches these addresses directly into the cached program, bypassing the need to recreate the full program descriptor [tt_metal/api/tt-metalium/program_descriptors.hpp 109-112](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L109-L112)

### Circular Buffer Configuration

The `CBDescriptor`[tt_metal/api/tt-metalium/program_descriptors.hpp 69-82](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L69-L82) is used to manage L1 memory allocations for data movement between kernels. The system uses `FormatDescriptors` to define data formats and page sizes for each circular buffer [tt_metal/api/tt-metalium/program_descriptors.hpp 70-75](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/program_descriptors.hpp#L70-L75)
