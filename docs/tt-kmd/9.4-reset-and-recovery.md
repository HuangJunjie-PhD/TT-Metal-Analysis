## Purpose and Scope

This page documents the driver's reset and recovery mechanisms, including reset types, generation tracking, synchronization primitives, and device-specific implementations. Reset operations allow recovery from hardware faults, firmware hangs, and error conditions while maintaining system stability.

For power management operations, see [Power Management](https://deepwiki.com/tenstorrent/tt-kmd/9.3-power-management). For general synchronization primitives, see [Synchronization Primitives](https://deepwiki.com/tenstorrent/tt-kmd/9.1-synchronization-primitives).

## Reset Types and Operations

The driver supports multiple reset types, each with different scopes and side effects. All reset operations are initiated via the `TENSTORRENT_IOCTL_RESET_DEVICE` ioctl.

### Reset Type Matrix

| Reset Type | Reset Gen | Hardware Action | FD Invalidation | Requires POST_RESET |
| --- | --- | --- | --- | --- |
| `RESTORE_STATE` | No | Restore saved PCI state | No | No |
| `RESET_PCIE_LINK` | No | PCIe hot reset | No | No |
| `CONFIG_WRITE` | Yes | Config write timer interrupt | Yes | No |
| `USER_RESET` | Yes | Set reset marker only | Yes | Yes |
| `ASIC_RESET` | Yes | Device-specific ASIC reset | Yes | Yes |
| `ASIC_DMC_RESET` | Yes | ASIC + memory controller reset | Yes | Yes |
| `POST_RESET` | No | Complete recovery sequence | No | N/A |

## Reset Generation Tracking

The `reset_gen` atomic counter implements generation-based file descriptor invalidation. When a reset occurs that requires FD invalidation, `reset_gen` is incremented, permanently invalidating all file descriptors opened before the reset.

**Reset Generation Architecture**

**Generation Checking in IOCTLs**

Every ioctl (except reset operations) validates the file descriptor generation before proceeding:

`// From chardev.c:483-487if (atomic_long_read(&priv->device->reset_gen) != priv->open_reset_gen) {	ret = -ENODEV;	goto out;}`
This ensures that file descriptors opened before a reset cannot perform operations after the reset, even if the device has been reinitialized.

## VMA Zapping

Virtual Memory Area (VMA) zapping invalidates all memory mappings created before a reset, preventing stale access to potentially remapped hardware resources.

**VMA Zapping Mechanism**

VMA zapping occurs for all reset types except `RESTORE_STATE`, ensuring that hardware remapping after reset doesn't expose stale mappings and user-space receives SIGBUS on access to pre-reset mappings.

Sources: [chardev.c 215-234](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L215-L234)

## Reset Synchronization

The `reset_rwsem` read-write semaphore coordinates reset operations with concurrent ioctl calls and mmap operations.

**Reset Synchronization Model**

**Key Synchronization Points:**

1.   **Normal IOCTLs** acquire read lock (shared), allowing concurrent operations.
2.   **Reset IOCTLs** acquire write lock (exclusive), blocking all other operations.
3.   **mmap** uses trylock to avoid ABBA deadlock with `mmap_lock`.
4.   **Device removal** acquires write lock to ensure safe cleanup.

Sources: [chardev.c 465-576](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L465-L576)[chardev.c 578-606](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L578-L606)[enumerate.c 353-358](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/enumerate.c#L353-L358)

The driver supports automatic device recovery via firmware watchdog mechanisms, controlled by the `auto_reset_timeout` module parameter.

**Watchdog Configuration**

| --- | --- | --- |
| Wormhole | ARC firmware watchdog | `WH_FW_MSG_UPDATE_M3_AUTO_RESET_TIMEOUT` |
| Blackhole | ARC message queue | `ARC_MSG_TYPE_SET_WDT_TIMEOUT` |
