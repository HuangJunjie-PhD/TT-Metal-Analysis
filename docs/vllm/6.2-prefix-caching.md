## Purpose and Scope

Prefix caching (also known as Automatic Prefix Caching or APC) is a performance optimization that reuses previously computed KV cache blocks across requests. When multiple requests share common prompt prefixes, or when a request is resumed after preemption, vLLM can avoid redundant computation by retrieving cached KV blocks instead of recomputing them.

This document covers:

*   The block hashing mechanism that enables cache lookups
*   Cache hit detection and block allocation flow during scheduling
*   Memory management and eviction policies
*   Support for multi-modal inputs, LoRA adapters, and specialized attention patterns

For information about the general KV cache architecture, see [Memory Management](https://deepwiki.com/tenstorrent/vllm/2.4-memory-management). For scheduler integration, see [Scheduler and Request Lifecycle](https://deepwiki.com/tenstorrent/vllm/2.3-scheduler-and-request-lifecycle).

## Core Concepts

### Block-Level Caching

vLLM organizes KV cache into fixed-size blocks (typically 16 or 32 tokens). Prefix caching operates at the block granularity:

*   **Full blocks** (completely filled with tokens) can be cached and reused
*   **Partial blocks** (not yet full) cannot be cached until completed
*   Each full block receives a cryptographic hash computed from its token IDs and parent block hash
*   Cache lookups use these hashes to identify matching blocks across requests

### Cache Hit and Miss

When scheduling a new request or resuming a preempted one:

1.   **Cache Hit**: The scheduler finds that one or more prefix blocks already exist in memory

    *   Blocks are reused immediately without recomputation
    *   `num_cached_tokens` tracks the number of tokens that hit the cache

2.   **Cache Miss**: No matching blocks exist

    *   New blocks are allocated and the tokens must be computed
    *   Once full, these blocks become eligible for caching

### Hash Chain

Block hashes form a chain where each block's hash depends on:

*   The parent block's hash (or `NONE_HASH` for the first block)
*   The current block's token IDs
*   Extra keys (multi-modal hashes, LoRA IDs, cache salt)

This creates a Merkle tree-like structure ensuring that identical prefixes produce identical hash chains.

## Block Hashing Mechanism

### Hash Computation Flow

**Sources:**[vllm/v1/core/kv_cache_utils.py 576-622](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L576-L622)[vllm/v1/core/kv_cache_utils.py 547-574](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L547-L574)[vllm/v1/core/kv_cache_utils.py 517-545](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L517-L545)

### Block Hash Structure

The hash for block `i` is computed as:

```
hash_i = hash_fn((parent_hash, token_ids, extra_keys))
```

Where:

*   `parent_hash`: `hash_{i-1}` or `NONE_HASH` if `i == 0`
*   `token_ids`: Tuple of token IDs in the block (length = `block_size`)
*   `extra_keys`: Optional tuple containing: 
    *   **Multi-modal hashes**: Identifiers for images/audio/video in the block range
    *   **LoRA ID**: `lora_request.lora_int_id` if using LoRA adapters
    *   **Cache salt**: User-provided salt for request-specific caching

The default hash function is `sha256` (from `vllm.utils`), but `sha256_cbor` is also supported for reproducibility.

**Sources:**[vllm/v1/core/kv_cache_utils.py 547-574](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L547-L574)[vllm/v1/core/kv_cache_utils.py 517-545](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L517-L545)[vllm/v1/core/kv_cache_utils.py 438-545](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L438-L545)

### Extra Keys for Multi-Modal and LoRA

Extra keys ensure that blocks with different contexts are not incorrectly shared:

*   **Multi-modal**: `_gen_mm_extra_hash_keys()` includes the `mm_feature.identifier` if the block overlaps with a multi-modal input position
*   **LoRA**: `_gen_lora_extra_hash_keys()` includes the LoRA integer ID if the request uses a LoRA adapter
*   **Cache salt**: Included only for the first block (token index 0) if `request.cache_salt` is set

**Sources:**[vllm/v1/core/kv_cache_utils.py 438-500](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L438-L500)[vllm/v1/core/kv_cache_utils.py 502-515](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L502-L515)[vllm/v1/core/kv_cache_utils.py 517-545](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L517-L545)

The `KVCacheManager.get_computed_blocks()` method implements cache lookup:

## Eviction and Memory Management

### LRU Eviction Policy

The `FreeKVCacheBlockQueue` maintains a doubly-linked list of free blocks ordered by eviction priority:

**Eviction order:**

1.   **Least recently used**: Blocks at the front of the queue (closest to `fake_head`) are evicted first
2.   **Longer sequences first**: When a request is freed, blocks are returned in reverse order, so tail blocks (with more cached tokens) are evicted before head blocks
3.   **Reference counting**: Only blocks with `ref_cnt == 0` can be evicted

**Sources:**[vllm/v1/core/kv_cache_utils.py 216-418](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/kv_cache_utils.py#L216-L418)[vllm/v1/core/block_pool.py 257-284](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/block_pool.py#L257-L284)

### Block Lifecycle

**Block states:**

*   **Free** (`ref_cnt == 0`, not in cache): In `FreeKVCacheBlockQueue`, available for allocation
*   **InUse** (`ref_cnt > 0`, not in cache): Allocated to one or more requests, but not yet full/cached
*   **Cached** (`ref_cnt >= 0`, in `BlockHashToBlockMap`): Full block with hash, may be in use or in free queue
*   **Evicted**: Cached block removed from `BlockHashToBlockMap` when allocated for new use

**Sources:**[vllm/v1/core/block_pool.py 286-315](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/block_pool.py#L286-L315)[vllm/v1/core/block_pool.py 188-256](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/block_pool.py#L188-L256)

### Touch Operation for Cache Hits

When a cache hit occurs, `BlockPool.touch()` moves cached blocks to the end of the free queue:

`# From vllm/v1/core/block_pool.py:332-365def touch(self, blocks: Iterable[list[KVCacheBlock]]) -> None:    """Touch the blocks to make them the last to be evicted."""    for blocks_per_group in blocks:        for block in blocks_per_group:            if block.ref_cnt == 0:                # Block is in free queue, move to end                self.free_block_queue.remove(block)                self.free_block_queue.append(block)            block.ref_cnt += 1`
This ensures that recently used blocks are less likely to be evicted.

## Advanced Features

### Sliding Window Attention Support

Models with sliding window attention (e.g., Mistral-7B) only need KV cache for the most recent `sliding_window` tokens. Prefix caching adapts:

**Key adaptations:**

*   `SlidingWindowManager.find_longest_cache_hit()` searches backward from the last block, returning only blocks within the sliding window
*   `SlidingWindowManager.remove_skipped_blocks()` replaces blocks outside the window with `null_block`
*   Prefix cache can still accelerate by caching the tail blocks that overlap across requests

**Sources:**[vllm/v1/core/single_type_kv_cache_manager.py 332-410](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/single_type_kv_cache_manager.py#L332-L410)[tests/v1/core/test_single_type_kv_cache_manager.py 97-160](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/v1/core/test_single_type_kv_cache_manager.py#L97-L160)

### Chunked Local Attention Support

Chunked local attention (e.g., Gemma-2) only attends to tokens within the current chunk plus the last `attention_chunk_size` tokens:

*   Similar to sliding window, but operates on chunk boundaries
*   `ChunkedLocalAttentionManager.find_longest_cache_hit()` returns blocks within the attention scope
*   Null blocks replace out-of-scope blocks

**Sources:**[vllm/v1/core/single_type_kv_cache_manager.py 413-518](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/v1/core/single_type_kv_cache_manager.py#L413-L518)[tests/v1/core/test_single_type_kv_cache_manager.py 33-95](https://github.com/tenstorrent/vllm/blob/cd9e5b83/tests/v1/core/test_single_type_kv_cache_manager.py#L33-L95)

### Hybrid KV Cache Coordinator

Models mixing multiple attention types (e.g., full attention + sliding window) use `HybridKVCacheCoordinator`:

**Cache hit logic:**

*   Must find cache hits in **all** KV cache groups
*   Returns the **minimum** hit length across groups (conservative estimate)
*   Ensures blocks remain synchronized across groups

## Configuration and Monitoring

### Enabling Prefix Caching

Prefix caching is controlled by the `enable_prefix_caching` parameter in `CacheConfig`:

`# Example initializationfrom vllm import LLM llm = LLM(    model="meta-llama/Llama-2-7b-hf",    enable_prefix_caching=True,  # Enable prefix caching    block_size=16,               # Tokens per block (affects granularity)    gpu_memory_utilization=0.9,  # Fraction of GPU memory for KV cache)`
**Configuration parameters:**

*   `enable_prefix_caching` (default: `False`): Enable/disable prefix caching globally
*   `block_size` (default: `16`): Number of tokens per block; smaller = finer granularity but more overhead
*   `gpu_memory_utilization`: Higher values allow more blocks to be cached

## Performance Considerations

### Block Size Trade-offs

| Block Size | Advantages | Disadvantages |
| --- | --- | --- |
| Smaller (8-16) | Finer cache granularity; more cache hits | More block overhead; more hash computations |
| Larger (32-64) | Less overhead; faster hashing | Coarser granularity; fewer cache hits |

**Recommendation**: Default `block_size=16` balances granularity and overhead for most workloads.

### Cache Hit Scenarios

**High cache hit rate:**

*   Multi-turn conversations with shared context
*   Batch processing with common prompt prefixes
*   Resumed requests after preemption

**Low cache hit rate:**

*   Unique prompts with no shared prefixes
*   Requests with prompt logprobs (explicitly disabled)
*   Rapidly changing prompts (e.g., streaming user input)

### Memory Overhead

Prefix caching adds minimal memory overhead:

*   `BlockHashToBlockMap`: ~40 bytes per cached block (hash + pointer)
*   `block_hash` field in `KVCacheBlock`: 32-36 bytes per block
*   Free queue pointers: 16 bytes per block (2 pointers)

Total: ~90 bytes per block, negligible compared to KV cache data (typically 8-64 KB per block).
