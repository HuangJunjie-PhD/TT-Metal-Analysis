# DUT Monitoring Interface

Relevant source files
*   [README.md](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/README.md?plain=1)
*   [mon/mon_instr/mon_instr_dpi.cc](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc)

The DUT Monitoring Interface provides the SystemVerilog Direct Programming Interface (DPI) layer that extracts architectural state from hardware testbenches and forwards it to the cosim-arch-checker framework. This interface serves as the primary data ingestion point for DUT execution monitoring in the lockstep verification system.

For information about the Bridge component that processes this monitored data, see [Bridge System](https://deepwiki.com/tenstorrent/cosim-arch-checker/3.1-bridge-system). For details about SystemVerilog DPI implementation specifics, see [SystemVerilog DPI](https://deepwiki.com/tenstorrent/cosim-arch-checker/3.4-systemverilog-dpi).

## Interface Registry System

The DUT monitoring interface uses a registry pattern to manage multiple named monitoring interfaces. This allows different testbench components or multiple DUT instances to register their own monitoring interfaces.

### Registry Architecture

**Sources:**[mon/mon_instr/mon_instr_dpi.cc 10-16](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L10-L16)

The registry system maintains a mapping from interface names to `cMonInstr` implementation objects. This enables multiple monitoring contexts to coexist and allows for runtime registration/deregistration of monitoring interfaces.

## DPI Function Interface

The monitoring interface provides five main DPI functions that capture different aspects of processor architectural state during execution.

### Function Signatures and Parameters

| Function | Purpose | Key Parameters |
| --- | --- | --- |
| `monitor_gpr` | General purpose register updates | `hart`, `addr`, `data` |
| `monitor_fpr` | Floating-point register updates | `hart`, `addr`, `data` |
| `monitor_vr` | Vector register updates | `hart`, `addr`, `*data` |
| `monitor_csr` | Control/status register updates | `hart`, `addr`, `data` |
| `monitor_retire` | Instruction retirement | `hart`, `tag`, `pc`, `opcode`, `trap` |

**Sources:**[README.md 70-76](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/README.md?plain=1#L70-L76)[mon/mon_instr/mon_instr_dpi.cc 18-51](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L18-L51)

### Function Dispatch Mechanism

**Sources:**[mon/mon_instr/mon_instr_dpi.cc 25-29](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L25-L29)[mon/mon_instr/mon_instr_dpi.cc 18-22](https://github.com/tenstorrent/cosim-arch-checker/blob/157c6637/mon/mon_instr/mon_instr_dpi.cc#L18-L22)

Each DPI function follows the same pattern: lookup the named interface in the registry, and if found, delegate to the appropriate method on the `cMonInstr` implementation object.

### Vector Register Handling

Vector registers require special handling due to their variable width. The `monitor_vr` function takes a pointer to `uint64_t` data, allowing for transfer of multi-word vector register contents based on the configured `k_VLen` parameter.

## Usage Pattern

The typical usage pattern involves the SystemVerilog testbench calling the monitoring functions whenever architectural state changes occur in the DUT.

### Testbench Integration Example

From the testbench perspective, the monitoring calls are made at specific points during instruction execution:

1.   **Register Updates**: Called when any architectural register is written
2.   **Instruction Retirement**: Called when an instruction completes execution

The interface name parameter allows multiple DUT instances or components to use the same monitoring infrastructure while maintaining separate state tracking.
