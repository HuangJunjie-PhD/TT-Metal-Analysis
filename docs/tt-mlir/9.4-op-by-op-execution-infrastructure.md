The Op-by-Op execution infrastructure provides a framework for isolating, executing, and validating individual operations within an MLIR module. This infrastructure is essential for debugging compilation regressions, verifying hardware kernels against golden references, and performing fine-grained performance analysis. It consists of two main components:

*   **Op-by-Op Infra**: a general-purpose module splitter and executor that manages operational decompositions and execution.
*   **Chisel**: a specialized differential debugging tool designed for golden comparisons and detailed runtime validation.

* * *

## Overview of Op-by-Op Infra

The Op-by-Op Infra enables a complex MLIR module (e.g., a complete neural network) to be broken down into its individual operations. Each operation is encapsulated into its own standalone MLIR module, which can be separately compiled and executed on the Tenstorrent device. The accompanying builders like `TTIRBuilder` and `StableHLOBuilder` support the management of "golden" tensors — reference outputs against which runtime results are compared.

### System Architecture and Data Flow

This infrastructure follows a pipeline that includes splitting the module by individual operations, compiling each operation module to hardware binaries, executing those binaries, and validating the results via golden references.

### Key Components

*   **`MLIRModuleSplitter`**:

 Parses the original MLIR module and extracts individual operations. Special handling is provided for control-flow ops like `func.call`, which may be inlined or replaced by callee processing, and for `stablehlo.composite` ops that require preserving decomposition functions. It excludes certain global or constant ops such as `ttnn.get_device`, `ttnn.deallocate`, and `stablehlo.constant` from isolated execution [tools/op-by-op-infra/op_by_op_infra/mlir_module_splitter.py 15-131](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/op-by-op-infra/op_by_op_infra/mlir_module_splitter.py#L15-L131)

*   **`MLIRModuleExecutor`**:

 Manages the execution lifecycle of each isolated op module, progressing it through phases such as generating TTIR, TTNN, flatbuffer binary, and running the flatbuffer. The execution results are wrapped in an `ExecutionResult` data structure [tools/op-by-op-infra/op_by_op_infra/mlir_module_executor.py 27-76](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/op-by-op-infra/op_by_op_infra/mlir_module_executor.py#L27-L76)[tools/op-by-op-infra/op_by_op_infra/execution_result.py 18-25](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/op-by-op-infra/op_by_op_infra/execution_result.py#L18-L25)

*   **Golden Tensor Management (`GoldenMapTensor`)**:

 This class manages reference (golden) tensors, supporting multi-device sharding. It implements the `__torch_function__` protocol to ensure that tensor operations get applied correctly and independently to shards, enabling precise and isolated golden reference computation [tools/golden/__init__.py 8-17](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/golden/__init__.py#L8-L17)

### Operation Filtering

Certain operations are not executed individually because they handle global hardware state or serve as constant definitions for other ops:

| Operation Name | Reason |
| --- | --- |
| `ttnn.get_device` | Manages global device state |
| `ttnn.deallocate` | Manages memory lifetime globally |
| `stablehlo.constant` | Bundled with user operations |

These exclusions are handled in the splitter logic to avoid invalid standalone execution.

## Chisel: Differential Debugging Tool

Chisel provides a powerful runtime framework for cross-validating the MLIR IR module against the flatbuffer binary produced by the backends. It enables detailed golden comparison of operation results during runtime, supporting debugging and regression isolation.

### Correlation Between IR and Binary Execution

Chisel creates a strong correlation between the IR module and the runtime binary execution using debug location information and ordered traversal of operations. It relies on the `IRModule` wrapper to load and traverse the MLIR source program, matching operations to binary execution steps.

*   The **`IRModule`** class loads MLIR source and provides efficient lookups on the IR function and operation contents.
*   **Operation utilities** help extract tensor inputs, outputs, and in-place mutation operands.
*   **`ChiselContext`** tracks the currently loaded binary commands, manages recorded golden comparison results, and orchestrates callbacks.
*   **Runtime hooks** provided by the `tt_runtime` Python API enable synchronized traversal and interception of binary operations during execution.

### Implementation Details

#### `IRModule` and Operation Utilities

*   `IRModule` caches functions and their operations by walking the MLIR in `PRE_ORDER`. It exposes methods for retrieving function operations, inputs, and outputs [tools/chisel/chisel/ops.py 133-156](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/ops.py#L133-L156)
*   `get_op_inputs` filters operands with tensor-like types (those having shape and element type) [tools/chisel/chisel/ops.py 67-69](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/ops.py#L67-L69)
*   `get_op_outputs` filters result values similarly [tools/chisel/chisel/ops.py 62-64](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/ops.py#L62-L64)
*   `get_flat_inplace_vals` identifies operands marked as mutated in-place, e.g., cache operands in certain TTNN ops [tools/chisel/chisel/ops.py 36-86](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/ops.py#L36-L86)

#### Runtime Execution and Callbacks

Chisel uses `tt_runtime.runtime.walk_program` to iterate over executable operations in a flatbuffer binary. During execution, callbacks intercept operation invocation before and after execution:

*   `_default_pre_op`: captures host copies of input tensors, ensuring that golden tensors can be seeded accurately from runtime values [tools/chisel/chisel/callbacks.py 91-127](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/callbacks.py#L91-L127)

*   `_default_post_op`: executes golden versions of the operation using the captured inputs, in two modes:

    *   **Isolated**: uses captured host inputs directly for reference execution.
    *   **Accumulated**: successively uses prior golden results as inputs for later ops.

After the golden execution, numerical correctness is verified via Pearson Correlation Coefficient (PCC) comparison with device results [tools/chisel/chisel/callbacks.py 156-207](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/callbacks.py#L156-L207)[tools/chisel/chisel/executor.py 15-30](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/chisel/chisel/executor.py#L15-L30)

#### Verification and Reporting

*   Chisel collects results in a `ChiselReport` as enumerated `ChiselRecord` entries.
*   PCC is used to measure numerical correlation between golden references and device outputs. If below threshold (typically 0.99), a `NUMERICS_FAIL` status is raised.
*   On failures, Chisel optionally dumps detailed debug artifacts — the MLIR source and offending flatbuffer binary — to a debug directory for offline investigation [test/python/chisel/test_builder_chisel_integration.py 40-174](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/python/chisel/test_builder_chisel_integration.py#L40-L174)

## Summary

The op-by-op execution infrastructure in the tt-mlir compiler stack allows precise isolation and validation of individual operations within MLIR-based AI workloads targeting Tenstorrent hardware. By splitting a complex MLIR graph into single-op modules, compiling and executing each separately, and verifying outputs against golden references, it provides a powerful framework for debugging, testing, and performance measurement.

The integral Chisel tool performs fine-grained runtime validation with detailed differential debugging capabilities, facilitating rapid detection and explanation of numerical mismatches or regressions.
