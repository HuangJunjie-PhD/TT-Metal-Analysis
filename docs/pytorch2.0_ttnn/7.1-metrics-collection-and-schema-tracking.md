## Purpose and Scope

This document explains the core data structures and collection mechanisms in the `torch_ttnn/metrics.py` module that capture operation signatures and schemas during model compilation. These structures form the foundation of the metrics system by recording what operations are encountered and how they are transformed through the compilation pipeline.

For information about how these metrics are aggregated and analyzed across test runs, see [Input Variation Analysis and Conversion Status](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.2-input-variation-analysis-and-conversion-status). For documentation generation from collected metrics, see [Documentation Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.3-documentation-generation).

**Sources:**[torch_ttnn/metrics.py 1-206](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L1-L206)

## Overview of the Metrics Schema System

The metrics collection system tracks operation signatures at two key points in the compilation pipeline: (1) when operations first enter the lowering passes in their original ATen form, and (2) after they are converted to TTNN operations. This dual tracking enables the system to determine conversion coverage and identify which input variations for each operation have been successfully lowered.

**Figure 1: Metrics Collection Flow Through Compilation Pipeline**

The system uses three primary data classes to represent operation schemas: `Inputs` for individual parameters, `InputVariation` for complete operation signatures before conversion, and `ConvertedInput` for signatures after conversion with linkage to the original.

**Sources:**[torch_ttnn/metrics.py 44-206](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L44-L206)[Diagram 5 from high-level architecture](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/Diagram%205%20from%20high-level%20architecture)

## Core Data Structures

### The Inputs Class

The `Inputs` class represents a single parameter to an operation, capturing its type, name, shape, and value. This class serves as the building block for constructing complete operation signatures.

**Figure 2: Core Data Classes for Schema Tracking**

The `Inputs` class performs shape extraction from `FakeTensor` metadata and value sanitization for serialization. Key attributes:

| Attribute | Raw Form | Processed Form | Purpose |
| --- | --- | --- | --- |
| `dtype` | Schema type | String | Parameter type from operation schema |
| `name` | Schema name | String | Parameter name from operation schema |
| `shape_` | FakeTensor or value | Original object | Raw shape metadata |
| `shape` | - | `"<[dim1, dim2, ...]>"` | Human-readable shape string |
| `value_` | Actual value | Original object | Raw parameter value |
| `value` | - | String representation | Serializable value string |

### The InputVariation Class

The `InputVariation` class represents a complete operation signature before any lowering transformations. It combines an operation name with a list of `Inputs` objects that describe each parameter.

#### Structure and Methods

The class provides three dictionary serialization formats:

| Method | Output Format | Use Case |
| --- | --- | --- |
| `dict()` | `{"opname": str, "inputs": List[str]}` | Human-readable reports |
| `dict_for_pickle()` | Includes `tensor_infos` with detailed metadata | Persistent storage, test generation |
| `get_input_str_list()` | `List[str]` of parameter descriptions | Intermediate processing |

### Node-Based Collection

The `collect_input_variation_from_node()` function handles collection from `torch.fx.Node` objects in the compiled graph. It checks if the node already has an `original_input_variations` metadata entry, indicating it has been transformed:

| Condition | Result Type | Meaning |
| --- | --- | --- |
| No `original_input_variations` metadata | `InputVariation` | First encounter of operation |
| Metadata exists, opname changed | `ConvertedInput` | Operation was converted |
| Metadata exists, opname unchanged | `InputVariation` | Operation passed through unchanged |

The function handles TTNN operations with and without the `ttnn.` prefix by checking for `aten.` prefix or using `target.__name__`:

## Serialization and Storage

### The save_pickle() Function

The `save_pickle()` function provides a standardized interface for persisting metrics to disk. It creates a nested directory structure under `metrics/` organized by model name:

```
metrics/
├── model_name/
│   ├── original-schema_list.pickle       # InputVariation objects from original graph
│   ├── compiled-schema_list.pickle       # ConvertedInput objects from compiled graph
│   ├── original-run_time_metrics.pickle  # Runtime metrics (not in this module)
│   └── compiled-run_time_metrics.pickle  # Runtime metrics (not in this module)
```

**Sources:**[torch_ttnn/metrics.py 12-18](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L12-L18)

## Integration with Compilation Pipeline

### Collection Points in ToTtPass

The `ToTtPass` (documented in [Operation Lowering](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass))) integrates metrics collection at the operation level. When the pass encounters ATen operations, it calls `collect_input_variation()` to capture the original signature and stores it in `node.meta["original_input_variations"]`:

**Figure 5: Metrics Collection During Operation Lowering**

After transformation, the replacement TTNN operation node inherits the metadata, allowing later passes to detect conversions by comparing the current operation name with the original.

**Sources:**[torch_ttnn/passes/lowering/to_tt_pass.py (referenced)](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_pass.py%20(referenced))[torch_ttnn/metrics.py 170-191](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L170-L191)

### Collection Points in Test Infrastructure

The test infrastructure (documented in [pytest Infrastructure and Fixtures](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.1-pytest-infrastructure-and-fixtures)) uses these collection functions in the `compile_and_run` fixture to capture schema lists at two points:

1.   **After AOT Autograd decomposition**: Captures the original ATen graph using `collect_input_variations_from_list_nodes()` on the decomposed graph nodes
2.   **After compilation**: Captures the compiled graph using the same function, where `ConvertedInput` objects are created for transformed operations

The fixture then serializes both lists using `save_pickle()` to `original-schema_list.pickle` and `compiled-schema_list.pickle` respectively.

**Sources:**[tests/conftest.py (referenced)](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/conftest.py%20(referenced))[torch_ttnn/metrics.py 193-206](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/metrics.py#L193-L206)

## Summary of Key Functions

| Function | Purpose | Returns | Integration Point |
| --- | --- | --- | --- |
| `collect_input_variation()` | Capture operation signature from schema | `InputVariation` or `None` | Called during ToTtPass for ATen ops |
| `collect_input_variation_from_node()` | Capture from FX node with conversion detection | `InputVariation`, `ConvertedInput`, or `None` | Called in test fixtures for both graphs |
| `collect_input_variations_from_list_nodes()` | Process entire FX graph | `List[InputVariation/ConvertedInput]` | Called in compile_and_run fixture |

## Relationship to Downstream Processing

The pickle files generated by this module serve as the input to the `tools/collect_metrics.py` aggregation script (documented in [Input Variation Analysis and Conversion Status](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.2-input-variation-analysis-and-conversion-status)). That script loads these files and constructs `InputVarPerOp` structures to:

1.   Match converted operations back to originals using the nested structure in `ConvertedInput.dict()`
2.   Determine conversion status (DONE, FALLBACK, NONE, REMOVED) for each input variation
3.   Generate documentation showing operation coverage across all tested models

The `tensor_infos` field in pickle files is used by the automated test generation system (documented in [Automated Test Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.4-automated-test-generation)) to create isolated operation tests with concrete input shapes extracted from real model executions.
