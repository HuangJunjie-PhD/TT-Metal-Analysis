# System Management Processor

Relevant source files
*   [README.md](https://github.com/tenstorrent/tt-firmware/blob/c9a325bb/README.md?plain=1)

## Purpose and Scope

This document describes the System Management Processor (SMP) in the Tenstorrent firmware architecture. The SMP is a critical component responsible for fundamental operations of Tenstorrent devices, including hardware initialization, I/O connectivity management, and power/thermal regulation. For detailed information about specific I/O connectivity features, see [I/O Connectivity](https://deepwiki.com/tenstorrent/tt-firmware/2.2-io-connectivity). For power and thermal management details, see [Power and Thermal Management](https://deepwiki.com/tenstorrent/tt-firmware/2.3-power-and-thermal-management).

Sources: [README.md 7-13](https://github.com/tenstorrent/tt-firmware/blob/c9a325bb/README.md?plain=1#L7-L13)

## Overview

The System Management Processor serves as the central control unit for Tenstorrent hardware, handling essential functions that enable the device to operate correctly and safely. It acts as an intermediary between the host system and the hardware, ensuring proper initialization, maintaining I/O connectivity, and enforcing power and thermal constraints.

## Key Responsibilities

### Hardware Initialization

The SMP is responsible for the startup sequence of Tenstorrent devices. This includes:

*   Power-on self-tests
*   Hardware component initialization
*   Subsystem configuration and setup
*   Establishing internal communication pathways

The initialization sequence is critical for ensuring all hardware components are properly configured before the device becomes available to the host system.

Sources: [README.md 10](https://github.com/tenstorrent/tt-firmware/blob/c9a325bb/README.md?plain=1#L10-L10)

### I/O Connectivity Management

The SMP manages all external I/O interfaces, ensuring reliable connectivity during device operation:

| Interface Type | Responsibility |
| --- | --- |
| PCIe | Host communication, enumeration, error handling |
| Ethernet | Network connectivity, link status monitoring |
| DRAM | Memory subsystem management, error correction |

The SMP monitors connection status and handles link maintenance to prevent disruptions during operation.

Diagram: I/O Connectivity Management Architecture

Sources: [README.md 11](https://github.com/tenstorrent/tt-firmware/blob/c9a325bb/README.md?plain=1#L11-L11)

### Power and Thermal Management

One of the SMP's critical functions is to ensure the hardware operates within its designed power and thermal envelope:

*   Active monitoring of power consumption
*   Temperature sensing and regulation
*   Implementing throttling mechanisms when limits are approached
*   Emergency shutdown procedures for thermal events

This functionality prevents damage to the hardware and ensures reliable operation across varying workloads.

## Board-Specific Implementations

The System Management Processor functionality is implemented across all supported Tenstorrent boards, with board-specific adaptations to accommodate hardware differences.

| Board Type | Notable SMP Features |
| --- | --- |
| Grayskull | Base SMP functionality |
| Wormhole | Base SMP functionality |
| Blackhole | Enhanced ERISC firmware (v1.4.0) with ETH mailbox support and Virtual UART |

For Blackhole boards specifically, the SMP includes an enhanced ERISC firmware with features like ETH mailbox messaging for link status checking and core control. The Virtual UART feature enables firmware debugging and observability from the host system.

For more details on Blackhole-specific features, see [Blackhole Specific Features](https://deepwiki.com/tenstorrent/tt-firmware/4-blackhole-specific-features).

Sources: [README.md 35-41](https://github.com/tenstorrent/tt-firmware/blob/c9a325bb/README.md?plain=1#L35-L41)

## Debugging and Observability

The SMP includes observability features that enable developers to monitor and debug the firmware behavior:

*   For Blackhole boards, the Virtual UART provides access to firmware debug messages
*   Debug messages can be viewed using the `tt-console` utility
*   Message types include `printk()` and `LOG_*()` outputs

These capabilities are particularly useful for troubleshooting issues related to hardware initialization, I/O connectivity, or thermal management.
