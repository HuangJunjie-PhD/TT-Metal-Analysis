# FP16 Conversions and Type System

## Purpose and Scope

This document describes the FP16 (16-bit floating-point) conversion system within SFPI, which provides dual-format support for converting between 32-bit floats and two distinct 16-bit floating-point representations: FP16A and FP16B. The system is implemented through a class hierarchy that enables type-safe conversions and seamless integration with the core SFPI vector types.

For information about the core vector types (`vFloat`, `vInt`, `vUInt`) and their operations, see [Vector Types and Operations](https://deepwiki.com/tenstorrent/sfpi/2.1-vector-types-and-operations). For details on how these conversions are used in hardware intrinsics, see [Hardware Intrinsics Overview](https://deepwiki.com/tenstorrent/sfpi/3.1-hardware-intrinsics-overview).

* * *

## FP16 Format Overview

The SFPI system supports two distinct 16-bit floating-point formats, each with different precision and range characteristics:

### Format Comparison

| Format | Encoding Method | Precision | Range | Use Case |
| --- | --- | --- | --- | --- |
| **FP16A** | IEEE-like with denormal handling | High | Standard | General-purpose FP16 with proper rounding |
| **FP16B** | Truncation (upper 16 bits of FP32) | Lower | Extended | Fast conversion, maintains FP32 exponent range |

**FP16A Format**: Uses a standard IEEE 754-like conversion algorithm that properly handles denormal numbers, rounding, and overflow cases. The conversion maintains precision through careful bit manipulation and conditional logic.

**FP16B Format**: Extracts the upper 16 bits of the FP32 representation, effectively truncating the lower mantissa bits. This provides a simpler, faster conversion at the cost of precision, but preserves the full exponent range of FP32.

## Class Hierarchy and Type System

**Diagram: FP16 Type Class Hierarchy**

### Base Class: `s2vFloat16`

The `s2vFloat16` class serves as the abstract base for all FP16 conversions, storing both the converted 16-bit value and the format identifier.

**Key Members**:

*   `value` (uint32_t): The 16-bit converted value stored in the lower bits
*   `format` (Format): Enum indicating whether this is FP16A or FP16B
*   `get()`: Returns the converted 16-bit value
*   `get_format()`: Returns the format enum as uint32_t (maps to `SFPLOADI_MOD0_*`)

**Format Enum Values**:

*   `fp16a = SFPLOADI_MOD0_FLOATA`: IEEE-like format
*   `fp16b = SFPLOADI_MOD0_FLOATB`: Truncated format

Sources: [include/sfpi_fp16.h 24-47](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L24-L47)

### Derived Classes: `s2vFloat16a` and `s2vFloat16b`

These derived classes provide explicit format selection through their type, eliminating the need for the format parameter in constructors.

**`s2vFloat16a`**: Always converts to FP16A format. Provides constructors for `float`, `double`, `int`, `int32_t`, `unsigned int`, and `uint32_t`.

**`s2vFloat16b`**: Always converts to FP16B format. Provides identical constructor overloads as `s2vFloat16a`.

Sources: [include/sfpi_fp16.h 50-69](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L50-L69)

## Conversion Algorithms

### FP32 to FP16A Conversion

The FP16A conversion implements a complete IEEE-like algorithm with proper handling of denormal numbers, rounding, and overflow cases.

**Diagram: FP16A Conversion Algorithm Flow**

**Algorithm Details**[include/sfpi_fp16.h 90-118](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L90-L118):

1.   **Add rounding bias**: `b = vui + 0x00001000` to handle rounding
2.   **Extract components**: 
    *   Exponent: `e = (b & 0x7F800000) >> 23`
    *   Mantissa: `m = b & 0x007FFFFF`

3.   **Conditional conversion**: 
    *   **Normal numbers** (`e > 112`): `((e - 112) << 10) & 0x7C00) | m >> 13`
    *   **Denormal numbers** (`101 < e < 113`): `((0x007FF000 + m) >> (125 - e) + 1) >> 1`
    *   **Overflow** (`e > 143`): `0x7FFF`

4.   **Combine with sign**: `(b & 0x80000000) >> 16 | <mantissa_result>`

This algorithm ensures accurate conversion with proper denormal handling, similar to IEEE 754 half-precision.

Sources: [include/sfpi_fp16.h 90-118](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L90-L118)

### FP32 to FP16B Conversion

The FP16B conversion is significantly simpler, performing a direct truncation of the FP32 representation.

**Diagram: FP16B Conversion Algorithm Flow**

**Algorithm Details**[include/sfpi_fp16.h 120-130](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L120-L130):

Simply extracts the upper 16 bits of the FP32 representation: `return tmp.vui >> 16`

This preserves:

*   The sign bit (bit 31 → bit 15)
*   The upper 8 exponent bits (bits 30-23 → bits 14-7)
*   The upper 7 mantissa bits (bits 22-16 → bits 6-0)

The lower 16 mantissa bits are discarded, resulting in reduced precision but maintaining the full FP32 exponent range.

Sources: [include/sfpi_fp16.h 120-130](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L120-L130)

## Integration with SFPI Vector Types

The FP16 conversion system integrates seamlessly with the core SFPI `vFloat` type, enabling FP16 values to be used anywhere FP32 values are expected.

**Diagram: FP16 Integration with SFPI Type System**

### `vFloat` Constructor

The `vFloat` class provides a direct constructor accepting `s2vFloat16`[include/sfpi.h 354](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L354-L354):

`vFloat(const s2vFloat16 f) { loadf16(f); }`
This constructor internally calls `loadf16()`[include/sfpi.h 840-843](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L840-L843):

`void vFloat::loadf16(const s2vFloat16 val) {    assign(__builtin_rvtt_sfpxloadi(val.get_format(), val.get()));}`
The format is passed to the hardware intrinsic, which uses it to select the appropriate decoding logic on the SFPU hardware.

Sources: [include/sfpi.h 354](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L354-L354)[include/sfpi.h 840-843](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L840-L843)

### `__vDReg` Assignment Operator

Destination registers can be directly assigned FP16 values [include/sfpi.h 205](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L205-L205):

`vFloat operator=(const s2vFloat16 f) const;`
This enables usage patterns like:

`dst_reg[0] = s2vFloat16a(3.14f);`
Sources: [include/sfpi.h 205](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L205-L205)

### Comparison Operations

All comparison operators work with FP16 values, supporting mixed-precision comparisons [include/sfpi.h 225-383](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L225-L383):

`// In __vDReg__vCond operator==(const s2vFloat16 x) const;__vCond operator<(const s2vFloat16 x) const;// ... and other comparison operators // In vFloat__vCond operator==(const s2vFloat16 x) const;__vCond operator<(const s2vFloat16 x) const;// ... and other comparison operators`
**Implementation Detail**[include/sfpi.h 809-814](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L809-L814): The comparison creates a `__vCond` object that encodes the format information:

`__vCond vFloat::operator==(const s2vFloat16 x) const {    return __vCond(__vCond::__vCondEQ, *this, x);}`
The `__vCond` constructor [include/sfpi.h 672-673](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L672-L673) then uses the format to determine the comparison mode:

`__vCond(const __vCondOpType t, const vFloat a, const s2vFloat16 b) {    result = __builtin_rvtt_sfpxfcmps(a.get(), b.get(),         t | ((b.get_format() == SFPLOADI_MOD0_FLOATA) ?              SFPXSCMP_MOD1_FMT_A : SFPXSCMP_MOD1_FMT_B));}`

### Arithmetic Operations

FP16 values participate in arithmetic operations through implicit conversion to `vFloat`[include/sfpi.h 368-807](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L368-L807):

`vFloat operator-(const s2vFloat16 b) const;`
The subtraction operator [include/sfpi.h 807](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L807-L807) handles FP16 negation:

`vFloat vFloat::operator-(const s2vFloat16 b) const {    return sfpi_int::fp_add(*this, b.negate());}`
The `negate()` method [include/sfpi_fp16.h 43](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L43-L43) flips the sign bit:

`s2vFloat16 negate() const {     return s2vFloat16(value ^ 0x8000, format); }`
Sources: [include/sfpi.h 368](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L368-L368)[include/sfpi.h 807](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L807-L807)[include/sfpi_fp16.h 43](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L43-L43)

### Integer to FP16 Conversion

Both classes support integer inputs [include/sfpi_fp16.h 54-68](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L54-L68):

`// Integer values are stored as-is (no conversion)s2vFloat16a int_val(42);        // 42 as int32_ts2vFloat16b uint_val(255u);     // 255 as uint32_t // The format still matters for hardware decodingvFloat v1 = int_val;  // SFPU decodes as FP16AvFloat v2 = uint_val; // SFPU decodes as FP16B`
Note: When constructing from integers [include/sfpi_fp16.h 78-82](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L78-L82) the value is stored directly without conversion. The format parameter still affects how the SFPU hardware interprets these bits when loaded.

## Format Selection Guidelines

| Scenario | Recommended Format | Rationale |
| --- | --- | --- |
| Standard ML inference | FP16A | Better precision with proper denormal handling |
| Wide dynamic range needs | FP16B | Maintains FP32 exponent range |
| Performance-critical paths | FP16B | Faster conversion (simple bit shift) |
| Constants and literals | FP16A | More predictable rounding behavior |
| Mixed-precision computation | FP16A | Better compatibility with IEEE standards |

**Hardware Constant Encoding**: The format enum values directly map to hardware encoding modes:

*   `SFPLOADI_MOD0_FLOATA` = 0
*   `SFPLOADI_MOD0_FLOATB` = 1

These values are passed to the `sfpxloadi` hardware intrinsic as part of the `mod0` field, instructing the SFPU on how to decode the 16-bit value when expanding back to 32-bit internal representation.

Sources: [include/sfpi_fp16.h 27-28](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi_fp16.h#L27-L28)[include/sfpi.h 840-843](https://github.com/tenstorrent/sfpi/blob/ed989b9d/include/sfpi.h#L840-L843)

## Type Safety and Compile-Time Guarantees

The class hierarchy provides compile-time type safety:

1.   **Explicit Format Selection**: `s2vFloat16a` and `s2vFloat16b` make format choice explicit in code
2.   **No Implicit Conversions**: Cannot accidentally mix formats without explicit construction
3.   **Format Tracking**: The `format` member ensures the hardware receives correct decoding instructions
4.   **Base Class Abstraction**: Code can accept `s2vFloat16` references to support both formats generically

Example demonstrating type safety:

`void process_fp16a_only(s2vFloat16a val) {    // Function only accepts FP16A format    vFloat v = val;    // ...} // s2vFloat16b cannot be passed without explicit conversion// process_fp16a_only(s2vFloat16b(1.0f));  // Compile error`
