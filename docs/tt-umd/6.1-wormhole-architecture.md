## Overview

This page documents the Wormhole B0 architecture implementation in `tt-umd`. Wormhole (`ARCH::WORMHOLE_B0`) is characterized by:

*   **Hardware DMA Engine**: BAR2-based PCIe DMA controller for bulk data transfers [device/tt_device/wormhole_tt_device.cpp 166-368](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L166-L368)
*   **Ethernet Training**: Two-phase link training with firmware version-dependent address maps [device/tt_device/wormhole_tt_device.cpp 406-506](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L406-L506)
*   **NOC Translation**: Optional coordinate remapping via NIU configuration [device/tt_device/wormhole_tt_device.cpp 73-79](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L73-L79)
*   **ARC Communication**: Dual access paths (PCIe BAR0 and JTAG) to ARC processor [device/tt_device/wormhole_tt_device.cpp 342-412](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L342-L412)
*   **iATU Configuration**: ARC message-based peer-to-peer address translation [device/tt_device/wormhole_tt_device.cpp 115-147](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L115-L147)

The implementation centers around `WormholeTTDevice`, which extends the base `TTDevice` interface to provide architecture-specific logic for memory access, clock management, and device initialization.

## DMA Engine

Wormhole features a dedicated PCIe DMA controller accessible via BAR2 registers. This allows high-throughput transfers between host hugepages and device memory (L1/DRAM).

### DMA Register Configuration

The driver interacts with specific register offsets defined in the hardware specification to trigger DMA transfers:

| Register Purpose | Offset (Write Engine) | Offset (Read Engine) |
| --- | --- | --- |
| Engine Enable | 0x00c | 0x02c |
| Completion Addr Low | 0x060 | 0x080 |
| Completion Addr High | 0x064 | 0x084 |
| Completion Value | 0x070 | 0x090 |
| Transfer Size | 0x208 | 0x308 |
| Source Addr Low | 0x20c | 0x30c |
| Destination Addr Low | 0x214 | 0x314 |
| Doorbell (Trigger) | 0x010 | 0x030 |

**Sources:**[device/tt_device/wormhole_tt_device.cpp 173-214](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L173-L214)[device/tt_device/wormhole_tt_device.cpp 252-293](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L252-L293)

### DMA Implementation Details

*   **Thread Safety**: All DMA operations are protected by `LockManager` using `MutexType::PCIE_DMA` to prevent concurrent access to the DMA hardware registers [device/tt_device/tt_device.cpp 68-69](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/tt_device.cpp#L68-L69)
*   **Polling and Timeouts**: The driver polls a completion flag in host memory. If the transfer does not complete within `DMA_TIMEOUT_MS` (10 seconds), a `DmaTimeoutError` is thrown [device/tt_device/wormhole_tt_device.cpp 223-239](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L223-L239)
*   **Zero-Copy**: Supports `dma_h2d_zero_copy` and `dma_d2h_zero_copy` which bypass intermediate driver buffers if the application provides pre-allocated physical memory addresses [device/tt_device/wormhole_tt_device.cpp 356-368](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L356-L368)

## Ethernet Training

Wormhole Ethernet cores require a training sequence managed by the `erisc` firmware. The `WormholeTTDevice` provides methods to monitor this status.

### Training Status Monitoring

The training process involves two phases:

1.   **Heartbeat Check**: Ensuring the `erisc` firmware is alive by monitoring a changing heartbeat value at address `0x1C`[device/tt_device/wormhole_tt_device.cpp 413-435](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L413-L435)
2.   **Status Polling**: Polling the training status register (typically at `0x1104`) until it reaches `SUCCESS` or `FAIL`[device/tt_device/wormhole_tt_device.cpp 443-467](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L443-L467)

**Sources:**[device/tt_device/wormhole_tt_device.cpp 406-506](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L406-L506)

## NOC Translation

Wormhole supports optional NOC (Network-on-Chip) coordinate translation. When enabled, the hardware automatically remaps logical coordinates to physical coordinates to handle harvested cores.

*   **Detection**: The driver queries bit 14 of the NIU configuration register (`NIU_CFG_0_OFFSET`) via ARC APB to determine if translation is active [device/tt_device/wormhole_tt_device.cpp 73-79](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L73-L79)
*   **Coordinate Mapping**: Lookup tables `NOC0_X_TO_NOC1_X` and `NOC0_Y_TO_NOC1_Y` are used to translate coordinates when `NOC1` is selected [device/api/umd/device/arch/wormhole_implementation.hpp 122-123](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/api/umd/device/arch/wormhole_implementation.hpp#L122-L123)

## iATU Configuration

The internal Address Translation Unit (iATU) is used for peer-to-peer (P2P) PCIe transfers. In Wormhole, this is configured by sending messages to the ARC processor.

**Sources:**[device/tt_device/wormhole_tt_device.cpp 115-147](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L115-L147)

## ARC Communication and Telemetry

Communication with the ARC processor is vital for power management and system health.

*   **ARC Messenger**: Used to send commands like `GET_AICLK` (to get the AI clock frequency) or `ARC_GET_HARVESTING` (to retrieve core harvesting masks) [device/tt_device/wormhole_tt_device.cpp 85-91](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L85-L91)[device/tt_device/wormhole_tt_device.cpp 103-111](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L103-L111)
*   **ARC APB/CSM Access**: Provides direct memory-mapped access to ARC's Advanced Peripheral Bus (APB) and Core Scratch Memory (CSM) via BAR0 or NOC [device/tt_device/wormhole_tt_device.cpp 149-164](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L149-L164)
*   **Harvesting**: Harvesting masks are retrieved from ARC and then shuffled via `CoordinateManager::shuffle_tensix_harvesting_mask` to align with the logical SoC layout [device/tt_device/wormhole_tt_device.cpp 94-95](https://github.com/tenstorrent/tt-umd/blob/9bd70c7a/device/tt_device/wormhole_tt_device.cpp#L94-L95)
