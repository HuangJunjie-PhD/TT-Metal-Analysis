## Purpose and Scope

This document describes the Docker container image building process within the TT-Forge release system. Docker images are created as part of the release workflow to provide pre-configured environments with all necessary dependencies and wheel packages installed. The build system creates two types of images: **base Python images** and **slim images** that package specific repository wheels.

For information about Python wheel building, see [Python Wheel Building](https://deepwiki.com/tenstorrent/tt-forge/5.4.1-python-wheel-building). For details on the overall release workflow that orchestrates Docker builds, see [Release Workflows](https://deepwiki.com/tenstorrent/tt-forge/5.3-release-workflows).

## Base Image Building

The base Python image is built using the `forge-ubuntu-22-04-py3-11` Dockerfile and provides a standardized Python 3.11 environment.

### Base Image Build Step

The base image build occurs early in the release workflow:

The build is executed in [.github/workflows/release.yml 152-170](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L152-L170):

*   **Condition**: Only builds if `python_version == '3.11'` (line 153)
*   **Action**: `docker-build-push` with specific parameters
*   **Caching**: Enabled via `cache_docker_tag: true` (line 158)
*   **Latest tag**: Always set to `true` for base images (line 159)

### Base Image Contents

The base image Dockerfile [.github/Dockerfile.ubuntu-22-04-py3-11](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/Dockerfile.ubuntu-22-04-py3-11) installs:

**System Dependencies** (lines 18-48):

*   Build tools: `make`, `build-essential`, `pkg-config`
*   Development libraries: `libssl-dev`, `zlib1g-dev`, `libreadline-dev`, `libffi-dev`
*   MPI library: OpenMPI-ULFM 5.0.7 (custom build for fault-tolerant MPI)
*   Utilities: `git`, `vim`, `wget`, `curl`, `jq`

**Python Environment** (lines 52-58):

*   Installs pyenv to `/root/.pyenv`
*   Installs Python 3.11 via pyenv
*   Sets Python 3.11 as global version
*   Upgrades pip to latest version

Sources: [.github/Dockerfile.ubuntu-22-04-py3-11 1-60](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/Dockerfile.ubuntu-22-04-py3-11#L1-L60)[.github/workflows/release.yml 152-170](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L152-L170)

## Slim Image Building

Slim images extend the base image by installing repository-specific wheel files. These images are used for testing and deployment.

### Slim Image Build Process

The slim image build occurs at [.github/workflows/release.yml 172-183](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L172-L183):

**Input Parameters**:

*   `image_name`: Constructed as `ghcr.io/tenstorrent/${repo_short}-slim`
*   `docker_tag`: The version tag from `gh_new_version_tag`
*   `make_latest`: Set to `false` during initial build (line 179)
*   `force_rebuild`: Equals `overwrite_releases` input (line 180)
*   `draft`: Draft mode flag (line 181)
*   `dockerfile`: `.github/Dockerfile.single-wheel-slim` (line 182)
*   `build_args`: Three build arguments passed to Dockerfile (line 183)

**Build Arguments** (line 183):

```
--build-arg REPO_SHORT=${{ repo_short }}
--build-arg WHEEL_FILES_PATH=${{ wheel_files_path }}
--build-arg DOCKER_BASE_IMAGE=${{ DOCKER_BASE_IMAGE }}
```

Sources: [.github/workflows/release.yml 172-183](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L172-L183)

## Image Tagging Strategy

The release workflow applies a two-phase tagging strategy to manage stable releases:

### Phase 1: Initial Build Tag

During the `build-release` job [.github/workflows/release.yml 172-183](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L172-L183):

*   Image is tagged with the version-specific tag (`gh_new_version_tag`)
*   `make_latest` is set to `false`
*   Image is built and pushed to registries

### Phase 2: Latest Tag Application

After successful publication [.github/workflows/release.yml 318-327](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L318-L327):

*   The `docker-build-push` action is called again
*   Uses the **same**`docker_tag` (no rebuild occurs)
*   `make_latest` is set based on `release_type`: 
    *   `stable` releases: `make_latest = true` → image tagged as `latest`
    *   Other release types: `make_latest = false` → no `latest` tag

The `make_latest` flag is determined in [.github/actions/set-release-facts/action.yaml 202-204](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L202-L204):

```
if [[ "${{ inputs.release_type }}" == "stable" ]]; then
  make_latest="true"
fi
```

Sources: [.github/workflows/release.yml 172-183](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L172-L183)[.github/workflows/release.yml 318-327](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L318-L327)[.github/actions/set-release-facts/action.yaml 202-204](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L202-L204)

## Build Conditions and Skip Logic

Not all repositories build Docker images. The build system uses the `skip_docker_build` flag to control this behavior.

### Skip Logic Configuration

The `skip_docker_build` flag is set per repository in [.github/actions/set-release-facts/action.yaml 217-248](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L217-L248):

| Repository | skip_docker_build | Reason |
| --- | --- | --- |
| `tt-forge-fe` | `false` | Frontend library needs Docker images for testing |
| `tt-mlir` | `true` | Compiler library, no Docker images needed |
| `tt-xla` | `false` | XLA frontend needs Docker images for testing |
| `tt-forge` | N/A | Meta-package, uses artifact uplift only |

### Conditional Build Execution

The conditional logic appears at [.github/workflows/release.yml 173](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L173-L173):

`if: ${{ (steps.set-release-facts.outputs.skip_docker_build == 'false' &&          steps.check_release.outputs.release_exists == 'false') ||         (steps.set-release-facts.outputs.skip_docker_build == 'false' &&          inputs.overwrite_releases) }}`
Sources: [.github/actions/set-release-facts/action.yaml 174-175](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L174-L175)[.github/actions/set-release-facts/action.yaml 217-248](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L217-L248)[.github/workflows/release.yml 173](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L173-L173)

## Integration with Testing Infrastructure

Built Docker images are the primary execution environment for CI/CD testing.

### Container Execution Configuration

Workflows such as `basic-tests.yml` and `demo-tests.yml` utilize the built images with specific hardware access configurations:

*   **Hardware Access**: Containers are launched with `--device /dev/tenstorrent`[.github/workflows/demo-tests.yml 97](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L97-L97)
*   **Volume Mounts**: Crucial system paths are mounted to enable hardware communication: 
    *   `/dev/hugepages` and `/dev/hugepages-1G` for memory management [.github/workflows/demo-tests.yml 99-100](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/demo-tests.yml#L99-L100)

## Draft Mode and Testing

Draft releases use modified Docker image configurations for testing purposes:

### Draft Mode Modifications

When `draft: true` is set, the following changes occur in [.github/actions/set-release-facts/action.yaml 256-295](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L256-L295):

1.   **Tag Prefix**: Images are tagged with `draft.${repo_short}.${version}` (line 278, 294)
2.   **Test Filtering**: Only specific tests run (e.g., `bge_m3`, `resnet`) (lines 259-260)
3.   **Basic Tests**: Limited to `n150` runners only (line 263)
4.   **Commit Nulling**: `build_release_latest_branch_commit` is cleared (line 269) to pick up artifacts from the branch rather than a specific commit

### Draft Image Publishing

Draft images are:

*   Built during `build-release` job
*   Pushed to registries with draft tag prefix
*   **Not** pushed to tt-pypi (see [.github/workflows/release.yml 305](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L305-L305))
*   Used for CI/CD workflow testing without affecting production releases

Sources: [.github/actions/set-release-facts/action.yaml 256-295](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/actions/set-release-facts/action.yaml#L256-L295)[.github/workflows/release.yml 305-306](https://github.com/tenstorrent/tt-forge/blob/6f2d9645/.github/workflows/release.yml#L305-L306)
