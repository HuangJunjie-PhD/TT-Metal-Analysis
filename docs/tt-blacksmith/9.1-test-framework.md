## Purpose and Scope

This document describes the test framework used to validate training experiments in TT-Blacksmith. The framework provides declarative test case definitions, subprocess-based execution, golden file validation, and marker-based test filtering for multi-hardware configurations.

For information about CI/CD pipelines that execute these tests, see [CI/CD Pipelines](https://deepwiki.com/tenstorrent/tt-blacksmith/9.2-cicd-pipelines). For information about the underlying training infrastructure being tested, see [Training Framework](https://deepwiki.com/tenstorrent/tt-blacksmith/5-training-framework).

## Test Case Definition System

Tests are defined declaratively in the `TRAINING_TEST_CASES` list using `pytest.param` objects. Each test case specifies the training script to execute, configuration files, timeout, and validation settings.

### Test Case Structure

**Sources:**[tests/training_test_cases.py 1-280](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/training_test_cases.py#L1-L280)

### Example Test Case Definition

A typical test case defines all necessary parameters for execution:

| Field | Type | Purpose |
| --- | --- | --- |
| `test_script` | `str` | Path to training script that will be executed as subprocess |
| `experiment_config` | `str` | Path to experiment YAML configuration |
| `test_config` | `str` (optional) | Path to test override YAML (defaults to `tests/configs/test_training_fast.yaml`) |
| `timeout` | `int` | Maximum execution time in seconds |
| `skip_loss_checks` | `bool` | If `True`, skip golden file validation |

## Pytest Markers System

The framework uses pytest markers to categorize tests along multiple dimensions, enabling fine-grained test filtering based on hardware availability, framework, parallelization strategy, and execution frequency.

### Marker Categories

The following markers are defined in `pytest.ini`:

| Category | Markers | Purpose |
| --- | --- | --- |
| **Frequency** | `push`, `uplift` | When tests run (every push vs. nightly) |
| **Hardware** | `n150`, `n300`, `n300_llmbox`, `galaxy` | Target hardware configuration |
| **Framework** | `torch`, `jax` | Which ML framework is used |
| **Parallelism** | `single_chip`, `data_parallel`, `tensor_parallel` | Distributed training strategy |
| **Split** | `split_0`, `split_1` | Test group for parallel execution |

**Sources:**[tests/pytest.ini 1-16](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/pytest.ini#L1-L16)

### Marker Application

Multiple markers are applied to each test case to describe its characteristics:

**Sources:**[tests/training_test_cases.py 14-19](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/training_test_cases.py#L14-L19)[tests/pytest.ini 2-15](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/pytest.ini#L2-L15)

### Marker-Based Filtering Examples

Tests can be filtered using pytest's `-m` option with boolean expressions:

`# Run only N150 single-chip testspytest -m "n150 and single_chip" # Run all PyTorch tests on N300pytest -m "n300 and torch" # Run push tests excluding Galaxypytest -m "push and not galaxy" # Run specific parallelism strategypytest -m "data_parallel or tensor_parallel"`

## Golden File Validation

Golden files store reference loss values from validated training runs. The framework compares new test runs against these references using pandas DataFrames with configurable tolerance.

### Validation Mechanism

**Sources:**[tests/test_all.py 29-32](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L29-L32)[tests/test_all.py 83-93](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L83-L93)[tests/test_all.py 127-134](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L127-L134)

### CSV File Format

Golden files contain step numbers and corresponding loss values:

| Column | Type | Description |
| --- | --- | --- |
| `_step` | `int` | Training step number |
| `train/loss` or `val/loss` | `float` | Loss value at that step |

Example golden file structure:

`_step,train/loss5,5.39192724227905310,4.49270844459533715,2.990625095367432`
**Sources:**[tests/golden_files/tt-llama_3_2_1b-sst2-n300-galaxy_train.csv 1-5](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/golden_files/tt-llama_3_2_1b-sst2-n300-galaxy_train.csv#L1-L5)

### Tolerance Configuration

Loss comparison uses relative tolerance via pandas' `rtol` parameter:

*   **Default tolerance**: `0.3` (30% relative difference allowed)
*   **Per-test override**: Specified in `setup_dict["tolerance"]`

`pd.testing.assert_frame_equal(golden_df, log_df, rtol=tolerance)`
**Sources:**[tests/test_all.py 29-32](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L29-L32)[tests/test_all.py 18](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L18-L18)

### Generating Golden Files

To create or update golden files:

`pytest tests/test_all.py --generate-golden-files -m "test_filter"`
This mode:

1.   Runs the training script
2.   Captures output logs
3.   Moves logs from `TEST_LOGS_DIR` to `GOLDEN_LOGS_DIR`
4.   Skips validation (all tests pass)

**Sources:**[tests/test_all.py 127-131](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L127-L131)

## Test Configuration Overrides

Test runs use a dedicated test configuration YAML that overrides experiment settings to create fast validation runs. This enables comprehensive testing without full training duration.

### Configuration Merge Strategy

**Sources:**[tests/configs/test_training_fast.yaml 1-9](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/configs/test_training_fast.yaml#L1-L9)[tests/test_all.py 17](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L17-L17)

### Default Test Configuration

The default test configuration (`tests/configs/test_training_fast.yaml`) provides:

| Field | Value | Purpose |
| --- | --- | --- |
| `test_config.max_steps_per_epoch` | `15` | Limit training to 15 steps per epoch |
| `steps_freq` | `5` | Log training metrics every 5 steps |
| `val_steps_freq` | `5` | Log validation metrics every 5 steps |
| `use_tt` | `true` | Ensure TT hardware is used |
| `log_on_wandb` | `false` | Disable W&B logging for tests |

**Sources:**[tests/configs/test_training_fast.yaml 1-9](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/configs/test_training_fast.yaml#L1-L9)

### Per-Test Configuration Override

Individual tests can specify custom test configurations:

`pytest.param(    {        "test_script": "...",        "experiment_config": "...",        "test_config": "tests/configs/custom_test.yaml",  # Custom override        "timeout": 3600,    },    ...)`
**Sources:**[tests/training_test_cases.py 203-207](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/training_test_cases.py#L203-L207)

## Running Tests

Tests can be executed using pytest with various filtering and configuration options.

### Basic Execution

`# Run all testspytest tests/test_all.py # Run specific test by IDpytest tests/test_all.py::test_training_script[tt-mlp-mnist-n150] # Run with verbose outputpytest tests/test_all.py -v # Run with captured output shownpytest tests/test_all.py -s`
### Filtering by Markers

`# Run only push testspytest tests/test_all.py -m "push" # Run N150 single-chip testspytest tests/test_all.py -m "n150 and single_chip" # Run all PyTorch testspytest tests/test_all.py -m "torch" # Run data parallel or tensor parallel testspytest tests/test_all.py -m "data_parallel or tensor_parallel" # Exclude Galaxy testspytest tests/test_all.py -m "not galaxy"`
**Sources:**[tests/pytest.ini 1-16](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/pytest.ini#L1-L16)

### Test Execution with Golden File Generation

`# Generate golden files for all testspytest tests/test_all.py --generate-golden-files # Generate golden files for specific markerspytest tests/test_all.py --generate-golden-files -m "n150 and push"`
**Sources:**[tests/test_all.py 127-131](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L127-L131)

### Timeout Handling

Each test has a configured timeout. If the training script exceeds this limit, the test fails with a timeout error:

`subprocess.TimeoutExpired → pytest.fail(f"Training script timed out after {timeout} seconds")`
**Sources:**[tests/test_all.py 79-80](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/tests/test_all.py#L79-L80)

## Test Case Examples

### Example 1: MNIST Tensor Parallel (N300)

`pytest.param(    {        "test_script": "blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py",        "experiment_config": "blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training_tp.yaml",        "timeout": 300,    },    marks=[        pytest.mark.push,        pytest.mark.n300,        pytest.mark.torch,        pytest.mark.tensor_parallel,    ],    id="tt-mlp-mnist-n300-tp",)`
This test:

*   Runs on every push
*   Requires N300 hardware (2 chips)
*   Uses PyTorch framework
*   Tests tensor parallel strategy
*   Has 5-minute timeout
*   Validates loss against golden files

## Integration with CI/CD

The test framework integrates with GitHub Actions CI/CD pipelines through marker-based filtering. CI workflows can execute different test suites based on trigger events:

*   **Push to main**: `pytest -m "push"`
*   **Nightly uplift**: `pytest -m "uplift"`
*   **Manual trigger**: Custom marker combinations

For detailed CI/CD pipeline documentation, see [CI/CD Pipelines](https://deepwiki.com/tenstorrent/tt-blacksmith/9.2-cicd-pipelines).
