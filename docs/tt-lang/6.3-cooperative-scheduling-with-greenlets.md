## Purpose and Scope

Cooperative scheduling is the execution model used in the `tt-lang` simulation framework to enable concurrent execution of compute and data movement functions across multiple cores. Unlike the compiler which targets hardware threads (MATH, BRISC, NCRISC), the simulator uses **Greenlets**—lightweight, cooperative coroutines—to interleave execution.

This page describes the `GreenletScheduler`, the mechanism for yielding on blocking operations (`wait`, `reserve`), and the fairness policies that govern thread execution.

For information about the overall simulation framework and `Program` execution model, see [6.2](https://github.com/tenstorrent/tt-lang/blob/d76e6233/6.2) For details on circular buffer simulation, see [6.4](https://github.com/tenstorrent/tt-lang/blob/d76e6233/6.4)

## Overview

In `tt-lang` simulation, each core executes up to three threads: one compute thread and up to two data movement threads (dm0 and dm1). These threads coordinate via `DataflowBuffer` (DFB) objects. To prevent simulation deadlocks when a thread waits for data not yet produced, the simulator employs a cooperative scheduler.

**Key characteristics:**

*   **Greenlet-based**: Each kernel (compute/DM) runs in its own `greenlet`[python/sim/greenlet_scheduler.py 7-10](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L7-L10)
*   **Stable Identity**: Kernels are identified by a `KernelId` consisting of `(linear_node, kind, func_name)`[python/sim/greenlet_scheduler.py 29-38](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L29-L38)
*   **Explicit Yielding**: Blocking operations like `wait()` or `reserve()` switch control back to the scheduler [python/sim/greenlet_scheduler.py 148-157](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L148-L157)
*   **Scheduling Policies**: Supports both `greedy` and `fair` (round-robin) algorithms [python/sim/greenlet_scheduler.py 58-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L58-L66)
*   **Deadlock Detection**: If all active kernels are blocked and no progress is made, the scheduler raises a diagnostic error [python/sim/greenlet_scheduler.py 430-454](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L430-L454)

## The GreenletScheduler

The `GreenletScheduler` manages a collection of active greenlets and executes them according to the configured algorithm.

### Kernel Registration

Kernels are added to the scheduler via `add_kernel(kernel_id, func)`[python/sim/greenlet_scheduler.py 104-108](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L104-L108) The scheduler enforces uniqueness; two kernels with the same `KernelId` (same node, same type, and same function name) on the same node will trigger a `RuntimeError`[python/sim/greenlet_scheduler.py 122-130](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L122-L130)

### Scheduling Algorithms

The simulator supports two policies [python/sim/greenlet_scheduler.py 58-66](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L58-L66):

1.   **Fair (Default)**: A round-robin approach. It selects the kernel that hasn't run for the longest time using a global timestamp to ensure all nodes and threads make progress [python/sim/greenlet_scheduler.py 321-344](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L321-L344)
2.   **Greedy**: Runs the first available unblocked kernel it finds in the active set [python/sim/greenlet_scheduler.py 311-319](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L311-L319)

### Execution Loop

The `run()` method is the heart of the scheduler. It continues as long as there are active greenlets in `self._active`[python/sim/greenlet_scheduler.py 280-285](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L280-L285) In each iteration:

*   It identifies "runnable" kernels (those not blocked, or whose blocking condition is now met via `can_wait`/`can_reserve`) [python/sim/greenlet_scheduler.py 293-305](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L293-L305)
*   It switches to the selected greenlet using `g.switch()`[python/sim/greenlet_scheduler.py 355-360](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L355-L360)
*   It monitors for progress via `self._has_made_progress` to detect deadlocks [python/sim/greenlet_scheduler.py 431-440](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L431-L440)

## Yielding on Blocking Operations

Blocking occurs when a kernel requires a resource (data or space) that is currently unavailable. Instead of a busy-wait, the kernel yields control back to the scheduler.

### block_if_needed

The function `block_if_needed(obj, op)` is the standard entry point for synchronization [python/sim/greenlet_scheduler.py 491-494](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L491-L494) It checks the object's `can_wait()` or `can_reserve()` status. If the check fails, it calls `block_current_kernel`[python/sim/greenlet_scheduler.py 513-521](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L513-L521)

### Implementation in Core Entities

| Entity | Operation | Blocking Condition |
| --- | --- | --- |
| `DataflowBuffer` | `wait()` | `can_wait()` returns `False` (insufficient ready blocks) [python/sim/greenlet_scheduler.py 513-514](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L513-L514) |
| `DataflowBuffer` | `reserve()` | `can_reserve()` returns `False` (insufficient free blocks) [python/sim/greenlet_scheduler.py 515-516](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L515-L516) |
| `CopyTransaction` | `wait()` | `handler.can_wait()` returns `False` (transfer in progress) [python/sim/greenlet_scheduler.py 513-514](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L513-L514) |

### Data Flow of a Blocked Kernel

When `block_current_kernel` is called, the scheduler captures the current source location via `find_user_code_location()` for diagnostics and switches back to the `_main_greenlet`[python/sim/greenlet_scheduler.py 164-185](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L164-L185)

## Deadlock Detection

A deadlock is defined as a state where:

1.   There are still active kernels in `self._active`[python/sim/greenlet_scheduler.py 431](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L431-L431)
2.   No kernel is able to make progress (all `can_wait`/`can_reserve` checks return `False` and no kernel has set `_has_made_progress`) [python/sim/greenlet_scheduler.py 432-435](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L432-L435)

When this occurs, the scheduler raises a `RuntimeError` with a detailed diagnostic report [python/sim/greenlet_scheduler.py 441-454](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L441-L454) This report includes:

*   The display name of each blocked kernel (e.g., `node0-mm_compute`) [python/sim/greenlet_scheduler.py 53-55](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L53-L55)
*   The operation it is blocked on (e.g., `wait` or `reserve`) [python/sim/greenlet_scheduler.py 174](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L174-L174)
*   The specific source code location (file and line number) where the block occurred [python/sim/greenlet_scheduler.py 165-167](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/greenlet_scheduler.py#L165-L167)

## Program Integration

The `Program` class (implemented via `ProgramImpl`) initializes the scheduler and populates it with kernels for every node in the specified grid [python/sim/program.py 107-151](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L107-L151)

### Per-Node Context

Before a kernel is added, the simulator builds a `node_context` via `_build_node_context(node)`[python/sim/program.py 153](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L153-L153) This involves:

*   **Deep-copying** state to ensure isolation between nodes using `copy.deepcopy`[python/sim/program.py 189](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L189-L189)
*   **Fresh DFBs**: Creating new `DataflowBuffer` instances for each node so that production/consumption is tracked per-node [python/sim/program.py 179-187](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L179-L187)
*   **Shared Tensors**: Keeping `ttnnsim.Tensor` references shared so that global data is accessible [python/sim/program.py 175-178](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L175-L178)

### Execution Lifecycle

The `_run_cooperative` method [python/sim/program.py 199](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L199-L199):

1.   Instantiates the `GreenletScheduler`[python/sim/program.py 212](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L212-L212)
2.   Iterates through the nodes, creating compute and DM kernels for each node [python/sim/program.py 214-222](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L214-L222)
3.   Sets the `_main_greenlet` and starts the scheduler via `scheduler.run()`[python/sim/program.py 224-225](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/sim/program.py#L224-L225)
