Relevant source files

This page provides immediate getting-started examples for the most common TT-SMI operations. It assumes TT-SMI is already installed on your system. For installation instructions, see [Installation and Setup](https://deepwiki.com/tenstorrent/tt-smi/1.1-installation-and-setup). For comprehensive documentation of all CLI options, see [Command-Line Interface](https://deepwiki.com/tenstorrent/tt-smi/2-command-line-interface). For detailed reset operation documentation, see [Reset Operations](https://deepwiki.com/tenstorrent/tt-smi/5-reset-operations).

## TT-SMI Operation Modes

TT-SMI supports three primary operation modes, each serving a distinct purpose:

| Mode | Command | Purpose | Output |
| --- | --- | --- | --- |
| **Interactive GUI** | `tt-smi` | Real-time monitoring with tabbed interface | Textual TUI application |
| **Reset** | `tt-smi -r [indices]` | Reset Tenstorrent devices | Status messages to terminal |
| **Snapshot** | `tt-smi -s` or `tt-smi -f` | Capture system state | JSON to stdout or file |

**Diagram: TT-SMI Execution Flow and Mode Selection**

The `main()` function [tt_smi.py 768-897](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L768-L897) orchestrates mode selection by checking command-line arguments in priority order. Reset operations bypass device detection and backend initialization for faster execution.

Sources: [tt_smi.py 768-897](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L768-L897)[tt_smi/tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py)

## Launching the Interactive GUI

The default mode displays a real-time monitoring interface with device information, telemetry, and firmware versions.

**Command:**

`tt-smi`
**Expected Behavior:**

*   Detects all Tenstorrent devices via `detect_chips_with_callback()`[tt_smi.py 874-883](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L874-L883)
*   Initializes `TTSMIBackend` with detected devices [tt_smi.py 891](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L891-L891)
*   Launches Textual TUI application via `TTSMI.run()`[tt_smi.py 725-731](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L725-L731)
*   Displays three tabs: Information (1), Telemetry (2), FW Version (3)
*   Telemetry updates automatically every 100ms when viewing tab 2

**GUI Navigation:**

*   Press `1`, `2`, or `3` to switch between tabs
*   Press `h` to display help menu
*   Press `c` to toggle sidebar (compact mode)
*   Press `q` to quit

**Application Structure:**

**Diagram: GUI Component Hierarchy**

The `TTSMI` class [tt_smi.py 67-575](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L67-L575) extends Textual's `App` and composes widgets defined in `compose()`[tt_smi.py 114-146](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L114-L146) When tab 2 is activated, `dispatch_telem_thread()`[tt_smi.py 550-567](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L550-L567) spawns a background thread that calls `update_telem_table()` every `GUI_INTERVAL_TIME` (0.1 seconds) [constants.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/constants.py#LNaN-LNaN)

Sources: [tt_smi.py 67-575](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L67-L575)[tt_smi.py 114-146](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L114-L146)[tt_smi.py 550-567](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L550-L567)

## Resetting Devices

TT-SMI supports resetting individual devices, all devices, or Galaxy 6U tray systems. Reset behavior varies by hardware platform.

### Standard PCI Device Reset

**Reset all devices:**

`tt-smi -r`
**Reset specific devices by PCI index:**

`tt-smi -r 0,1`
**Expected Output (Wormhole example):**

```
Starting reset on devices at PCI indices: 0
Waiting for 2 seconds for potential hotplug removal.
Waiting for devices to reappear on pci bus...
Reset successfully completed for device at PCI index 0.
Finishing reset on devices at PCI indices: 0
Re-initializing boards after reset....
Detected Chips: 2
```

**Reset Operation Flow:**

**Diagram: Standard Reset Operation Code Path**

The reset operation [tt_smi.py 790-803](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L790-L803) parses indices using `parse_reset_input()` from `tt_tools_common` and dispatches to `pci_board_reset()`[tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi_backend.py) The `--no_reinit` flag [tt_smi.py 676-680](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L676-L680) skips device re-detection for faster operation.

**Platform-Specific Reset Behavior:**

| Platform | Reset Type | Details |
| --- | --- | --- |
| **Wormhole** | Board-level | Power cycle + Ethernet retraining via `WHChipReset.full_lds_reset` |
| **Blackhole** | ASIC-level | ASIC reset via `BHChipReset.full_lds_reset` |
| **Grayskull** | ~~Tensix-level~~ | Deprecated as of v3.0.35 |

Sources: [tt_smi.py 790-803](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L790-L803)[README.md 145-178](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/README.md?plain=1#L145-L178)

### Galaxy 6U System Reset

Galaxy systems require specialized IPMI-based reset commands.

**Reset all Galaxy trays:**

`tt-smi -glx_reset`
**Reset with auto-retry (up to 3 attempts):**

`tt-smi -glx_reset_auto`
**Reset specific tray (1-4):**

`tt-smi -glx_reset_tray 3`
**Skip device re-initialization:**

`tt-smi -glx_reset --no_reinit`
**Expected Output:**

```
Resetting WH Galaxy trays with reset command...
Executing command: sudo ipmitool raw 0x30 0x8B 0xF 0xFF 0x0 0xF
Waiting for 30 seconds: 30
Driver loaded
Re-initializing boards after reset....
Detected Chips: 32
Re-initialized 32 boards after reset. Exiting...
```

**Galaxy Tray Mapping:**

To identify which devices belong to which tray:

`tt-smi -glx_list_tray_to_device`
**Output:**

```
Gathering Information ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 100% 0:00:00
```

Snapshots capture complete system state as JSON for logging, debugging, or monitoring integration.

**Default filename (timestamp-based):**

`tt-smi -f`
Saves to: `~/tt_smi/<timestamp>_snapshot.json`

**Custom filename:**

`tt-smi -f /path/to/custom_snapshot.json`
**Output:**

**Snapshot Generation Code Path:**

**Diagram: Snapshot Generation Data Flow**

Snapshots use the `TTSMILog` Pydantic model [log.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/log.py) for validation and serialization. The `create_tt_smi_log()` method aggregates data from `device_infos`, `device_telemetrys`, and `firmware_infos` dictionaries maintained by `TTSMIBackend`.

**Force Non-TTY Output:**

`tt-smi -s --snapshot_no_tty`

## Prerequisites Check

Before running TT-SMI operations, verify:

1.   **Driver installed:** TT-SMI requires `tt-kmd` driver ≥ 2.0.0 [tt_smi.py 777-784](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L777-L784)
2.   **Rust compiler:** Required for building `pyluwen` dependency (see [Installation and Setup](https://deepwiki.com/tenstorrent/tt-smi/1.1-installation-and-setup))
3.   **Hardware detected:** Run `tt-smi -ls` to confirm devices are visible
4.   **Permissions:** Galaxy reset operations require `sudo` for IPMI access

If no driver is detected, `main()` exits with error code 1 and displays installation instructions [tt_smi.py 777-784](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#L777-L784)
