TTNN-JIT is a specialized compilation infrastructure that allows Tenstorrent model developers to JIT-compile select portions of Python TTNN models using the Direct-To-Metal (D2M) pipeline [docs/src/ttnn-jit.md 1-4](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L1-L4) It bridges the gap between eager Python execution and optimized hardware-specific execution by tracing Python subgraphs, lowering them to MLIR (TTIR), and applying hardware-aware optimizations like operation fusion and grid selection [docs/src/ttnn-jit.md 136-145](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L136-L145)

### System Architecture and Data Flow

The JIT system follows a three-step process: Python tracing, D2M compilation, and runtime execution.

#### 1. Python Tracing and IR Generation

When a function is decorated with `@ttnn_jit.jit()`, it is wrapped in a `JitFunction` object [tools/ttnn-jit/api.py 37-46](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/api.py#L37-L46) Upon invocation, the system uses a tracing mechanism to intercept `ttnn.*` calls and translate them into TTIR operations [tools/ttnn-jit/_src/jit.py 145-151](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L145-L151) This process involves source code cleanup and redirection of `ttnn` calls through JIT-aware handlers [tools/ttnn-jit/_src/jit.py 44-46](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L44-L46)[docs/src/ttnn-jit.md 141-146](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L141-L146)

#### 2. D2M Compilation Pipeline

The generated TTIR module is passed to the `ttnn_to_ttmetal_pipeline`[tools/ttnn-jit/_src/jit.py 165](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L165-L165) This pipeline performs several critical tasks:

*   **Grid Selection:** Infers optimal core grids and sharding strategies for intermediate tensors [docs/src/ttnn-jit.md 165-170](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L165-L170)
*   **Op Fusion:** Fuses multiple elementwise operations into a single kernel to minimize data movement [docs/src/ttnn-jit.md 214-220](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L214-L220)
*   **Memory Analysis:** Analyzes available L1/DRAM ranges and output tensor requirements via the `MemoryAnalyzer` to guide bufferization [tools/ttnn-jit/_src/jit.py 154-159](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L154-L159)

#### 3. Runtime Execution

The final artifact is a Flatbuffer binary (or capsule) containing the compiled program [tools/ttnn-jit/_src/jit.py 187](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L187-L187) This binary is executed on the device using the `ttmlir` runtime dispatch system, specifically `run_binary` or `run_binary_from_capsule`[tools/ttnn-jit/_src/jit.py 180-188](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L180-L188)

### Key Classes and Implementation

#### JitFunction

The primary entry point for JIT execution. It manages the function's source code, output directories for generated artifacts, and the JIT cache [tools/ttnn-jit/_src/jit.py 30-68](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L30-L68)

*   `__call__`: Validates arguments using `inspect.signature`, queries the system descriptor via `_query_and_save_system_desc`, and triggers the `_compile_and_run` logic [tools/ttnn-jit/_src/jit.py 112-136](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L112-L136)
*   `_compile_and_run`: Orchestrates the transition from Python `ttnn` calls to MLIR IR via `generate_ir`, applies the backend pipeline, and dispatches the binary [tools/ttnn-jit/_src/jit.py 138-184](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L138-L184)

#### BaseOpHandler

A hierarchy of handlers in `jit_functions.py` used during tracing to resolve Python arguments (tensors or scalars) into MLIR values [tools/ttnn-jit/_src/jit_functions.py 59-109](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit_functions.py#L59-L109)

*   `_resolve_operand`: Handles the resolution order: Scalar constants ->`ResultWrapper` (intermediate results) ->`value_map` -> function block arguments [tools/ttnn-jit/_src/jit_functions.py 66-109](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit_functions.py#L66-L109)
*   `_create_scalar_tensor_constant`: Converts Python scalars into `ttir.full` operations using `FloatAttr` or `IntegerAttr` to enable broadcasting with other tensor operands [tools/ttnn-jit/_src/jit_functions.py 147-190](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit_functions.py#L147-L190)

#### JitCache

A caching mechanism that stores compiled Flatbuffer binaries indexed by the hash of the input tensor metadata (shapes, dtypes, and layouts) [tools/ttnn-jit/_src/jit.py 68-70](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L68-L70)

*   `contains`: Checks if a compiled version of the graph exists for the current input tensor metadata [tools/ttnn-jit/_src/jit.py 143](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L143-L143)
*   `compile_and_insert`: Compiles the IR string into a binary and stores it in the cache [tools/ttnn-jit/_src/jit.py 178-180](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L178-L180)

#### Layout and Memory Configuration

*   **Tensor Layouts:** Supports L1 Block/Height/Width sharded and DRAM interleaved [docs/src/ttnn-jit.md 105-117](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L105-L117)
*   **Automatic Grid Selection:** If `memory_config` is not provided to the decorator, the compiler defaults to a maximally L1 block-sharded layout [docs/src/ttnn-jit.md 86-87](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/docs/src/ttnn-jit.md?plain=1#L86-L87)[tools/ttnn-jit/api.py 28-29](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/api.py#L28-L29)

#### MLIR Conversion Patterns

Backend conversion from `ttnn` to `ttir` dialect is implemented using `OpConversionPattern`. ### Error Handling and Fallback

TTNN-JIT includes a robust fallback mechanism to ensure model execution continuity even when JIT compilation is not possible for specific subgraphs.

*   **Fallback Trigger:** Enabled via `fallback=True` in the `@ttnn_jit.jit` decorator [tools/ttnn-jit/api.py 17](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/api.py#L17-L17)
*   **Behavior:** If an exception occurs during `_compile_and_run` (e.g., unsupported sharding, memory allocation failure), the system catches it, sets `self.fallback_used = True`, and executes the original Python function eagerly through the standard `ttnn` path [tools/ttnn-jit/_src/jit.py 127-136](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L127-L136)
*   **Validation:** Fallback is strictly incompatible with `compile_only=True` because `compile_only` expects to return IR/flatbuffer artifacts rather than execution results [tools/ttnn-jit/_src/jit.py 52-57](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/tools/ttnn-jit/_src/jit.py#L52-L57)
