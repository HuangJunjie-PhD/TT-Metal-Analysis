This page documents how compiled flatbuffer binaries are loaded into the `tt-mlir` runtime and executed on Tenstorrent hardware. It covers the `Binary` loading API, the runtime dispatch mechanism, the TTNN-path `ProgramExecutor`, the TTMetal-path execution flow, result gathering, and trace management.

For background on how flatbuffer binaries are produced by the compiler, see **3.7 Flatbuffer Serialization**. For device and mesh lifecycle management, see **4.2 Device and Mesh Management**. For tensor creation and transfer, see **4.3 Tensor Management and Memory**.

* * *

## Binary Loading

### The `Binary` and `Flatbuffer` Types

The runtime represents a compiled artifact as a `Binary`, defined in [runtime/include/tt/runtime/types.h 30-36](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L30-L36)

`Binary` inherits from `Flatbuffer`, which holds the raw buffer as a `std::shared_ptr<void>`. Each `Binary` receives a monotonically-increasing `binaryId` (via `nextBinaryId()`) and a fresh `ProgramDescCache` on construction [runtime/lib/binary.cpp 30-36](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L30-L36) The cache is used by the TTNN runtime to avoid re-parsing program descriptors and is shared across instances [runtime/lib/binary.cpp 41-43](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L41-L43)

| Method | Description |
| --- | --- |
| `Flatbuffer::loadFromPath(path)` | Memory-maps or reads a `.ttnn`/`.ttm` file from disk [runtime/include/tt/runtime/types.h 273-357](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/types.h#L273-L357) |
| `Binary::nextBinaryId()` | Generates a unique atomic ID for the binary instance [runtime/lib/binary.cpp 56-59](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L56-L59) |
| `ttnn::getBinary(fb)` | Returns the root `TTNNBinary` table from a flatbuffer [runtime/lib/binary.cpp 126-131](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L126-L131) |
| `ttmetal::getBinary(fb)` | Returns the root `TTMetalBinary` table from a flatbuffer [runtime/lib/ttmetal/runtime.cpp 29-36](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L29-L36) |

### Format Detection and Runtime Selection

A flatbuffer binary carries a 4-byte file identifier. The runtime uses this to automatically configure the correct backend via `setCompatibleDeviceRuntime(binary)` in [runtime/lib/runtime.cpp 186-203](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/runtime.cpp#L186-L203)

*   **TTNN**: Checks for `SizePrefixedTTNNBinaryBufferHasIdentifier`[runtime/lib/binary.cpp 127-128](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/binary.cpp#L127-L128)
*   **TTMetal**: Checks for `SizePrefixedTTMetalBinaryBufferHasIdentifier`[runtime/lib/ttmetal/runtime.cpp 30-31](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L30-L31)

## TTNN Execution Path: `ProgramExecutor`

### Class Overview

`ProgramExecutor` is defined in [runtime/lib/ttnn/program_executor.cpp 131-161](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L131-L161) It encapsulates the state needed to run a specific program within a TTNN binary.

Its constructor:

1.   Obtains the `::tt::target::ttnn::Program*` from the binary using `utils::getProgram`[runtime/lib/ttnn/program_executor.cpp 136](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L136-L136)
2.   Maps each input's `global_id` to the runtime `Tensor*` in a `TensorPtrMap liveTensors`[runtime/lib/ttnn/program_executor.cpp 142-150](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L142-L150)
3.   Initializes a `ProgramContext` which manages the `DylibManager` for dynamic library symbols and the `ProgramTensorPool`[runtime/lib/ttnn/program_executor.cpp 158-161](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L158-L161)

## TTMetal Execution Path

For binaries targeting the TTMetal backend, the execution is handled by `ttmetal::submit` which uses an internal executor to process the binary commands [runtime/lib/ttmetal/runtime.cpp 152-155](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L152-L155)

### Command Processing

The TTMetal runtime processes commands defined in the `TTMetalBinary` flatbuffer via the `MCQExecutor` class (Metal Command Queue Executor). This includes:

*   **Program Launch**: Enqueues kernels to the device command queues.
*   **Memory Management**: Handles allocations and DMA transfers between host and device.
*   **Synchronization**: Handles host-side synchronization for events and buffer completions [runtime/lib/ttmetal/runtime.cpp 132-138](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttmetal/runtime.cpp#L132-L138)

## Trace Capture and Execution

TTNN supports capturing a sequence of operations into a hardware trace to minimize host overhead during repetitive execution. This is managed via specialized trace operations in the `ProgramExecutor`:

1.   **Begin Trace Capture**: Marks the start of a command sequence to be recorded on the device [runtime/lib/ttnn/program_executor.cpp 102](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L102-L102)
2.   **End Trace Capture**: Finalizes the recording and caches the trace [runtime/lib/ttnn/program_executor.cpp 104](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L104-L104)
3.   **Execute Trace**: Replays a previously captured command sequence [runtime/lib/ttnn/program_executor.cpp 105](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L105-L105)
4.   **Capture Or Execute Trace**: A hybrid operation that checks if a trace exists in the `TraceCache`; if not, it performs capture, otherwise it executes it [runtime/lib/ttnn/program_executor.cpp 103](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L103-L103)

Sources: [runtime/lib/ttnn/program_executor.cpp 102-105](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L102-L105)[runtime/include/tt/runtime/detail/ttnn/types/trace_cache.h](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/include/tt/runtime/detail/ttnn/types/trace_cache.h)

## Dynamic Library (dylib) Execution

The `DylibManager` handles the loading of external shared libraries specified in the flatbuffer. This is used for execution paths where parts of the program are compiled to native C++ code (e.g., via the EmitC backend).

*   **Symbol Loading**: Symbols are resolved using the `DylibManager` which manages the lifecycle of loaded libraries defined in the `dylibs` field of the flatbuffer `Program`[runtime/lib/ttnn/program_executor.cpp 158-161](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L158-L161)
*   **Native Call**: `FuncCallOp` retrieves the function pointer from the `DylibManager` and executes it, passing the relevant `liveTensors` from the `ProgramContext` as arguments [runtime/lib/ttnn/program_executor.cpp 80](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/runtime/lib/ttnn/program_executor.cpp#L80-L80)
