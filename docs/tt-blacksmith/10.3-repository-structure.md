This document describes the physical organization of the TT-Blacksmith repository, including the directory layout, Python package structure via `setup.py`, and how code is organized by framework. For information about the experiment execution flow and configuration system, see [Configuration System](https://deepwiki.com/tenstorrent/tt-blacksmith/10.3-repository-structure#3.3). ## Repository Layout

The TT-Blacksmith repository follows a three-tier organizational structure that separates datasets, models, and experiments. This design enables code reuse across different framework implementations and hardware targets.

**Sources:**[docs/src/experiments.md 43-56](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L43-L56)[.gitignore 1-20](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L1-L20)[setup.py 1-11](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/setup.py#L1-L11)

### Top-Level Directories

| Directory | Purpose | Key Contents |
| --- | --- | --- |
| `blacksmith/` | Main Python package containing all source code | `datasets/`, `models/`, `experiments/` subdirectories |
| `env/` | Environment management scripts and dependency specifications | `activate` script, `*_requirements.txt` files |
| `docs/` | Documentation source files | `src/` (markdown), `shared/` (images) |
| `.github/` | CI/CD configuration | `workflows/`, `test-matrix/` |
| `data/` | Runtime data directory (gitignored) | Downloaded datasets, cached files |
| `checkpoints/` | Model checkpoint storage (gitignored) | Saved model states during training |
| `wandb/` | Weights & Biases logging artifacts (gitignored) | Experiment tracking data |

**Sources:**[.gitignore 1-20](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L1-L20)[docs/src/experiments.md 43-56](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L43-L56)

## Python Package Structure

The repository is installed as a Python package named `blacksmith` via `setup.py`. This enables imports like `from blacksmith.datasets.torch.mnist import MNISTDataset` throughout the codebase.

**Sources:**[setup.py 1-11](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/setup.py#L1-L11)

### setup.py Configuration

The [setup.py 6-11](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/setup.py#L6-L11) file defines the package structure:

`setup(    name="blacksmith",    version="0.1",    description="Tenstorrent Python Blacksmith",    packages=find_packages(include=["blacksmith*"]),)`
The `find_packages(include=["blacksmith*"])` call automatically discovers all Python packages within the `blacksmith/` directory, including all subdirectories with `__init__.py` files. This means any new experiment, model, or dataset added under `blacksmith/` is automatically included in the package.

**Sources:**[setup.py 1-11](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/setup.py#L1-L11)

## Code Organization by Framework

The three core directories (`datasets/`, `models/`, `experiments/`) are organized first by AI framework (PyTorch, JAX, Flax), then by specific model or task. This structure allows framework-specific implementations while maintaining parallel organization.

**Sources:**[docs/src/experiments.md 43-56](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L43-L56)

### Framework Organization Patterns

The framework-first organization follows these patterns:

| Component | PyTorch Path | JAX Path | Purpose |
| --- | --- | --- | --- |
| Dataset loaders | `blacksmith/datasets/torch/{task}/` | `blacksmith/datasets/jax/{task}/` | Load and preprocess data for specific framework |
| Model definitions | `blacksmith/models/torch/{model}/` | `blacksmith/models/jax/{model}/` | Define model architectures or load pretrained models |
| Experiments | `blacksmith/experiments/torch/{model}/` | `blacksmith/experiments/jax/{model}/` | Training scripts, configs, and test files |

**Sources:**[docs/src/experiments.md 46-48](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L46-L48)

## Experiment Directory Structure

Within each experiment directory (e.g., `blacksmith/experiments/torch/mnist/`), code is organized by hardware configuration and parallelism strategy. This structure enables running the same model on different hardware targets with minimal code duplication.

**Sources:**[docs/src/experiments.md 48-56](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L48-L56)

### Experiment File Patterns

Each experiment directory contains:

1.   **configs.py**: Defines Pydantic configuration classes (e.g., `TrainingConfig`, `NetConfig`) that specify all experiment parameters
2.   **YAML files**: Provide concrete configuration values for specific experiment runs (e.g., `test_mnist_linear.yaml`)
3.   **Test scripts**: Python files (e.g., `test_mnist_training.py`) that execute the training loop using the defined configurations
4.   **Hardware subdirectories**: Organize experiments by target hardware (`single_chip/`, `galaxy/`, `quietbox/`, `t3k/`)
5.   **Parallelism subdirectories**: Within hardware directories, experiments may be organized by parallelism strategy (`data_parallel/`, `tensor_parallel/`, `hybrid/`)

**Sources:**[docs/src/experiments.md 48-56](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L48-L56)

### Hardware Configuration Mapping

The subdirectory structure maps to specific Tenstorrent hardware configurations:

| Directory | Hardware Target | Typical Usage |
| --- | --- | --- |
| `single_chip/` | N150 (single chip) | Small models, development, testing |
| `n300/` | N300 (2 chips) | Data parallel, small tensor parallel |
| `quietbox/` | QuietBox (4-8 chips) | Medium models with tensor parallelism |
| `t3k/` | T3K (8 chips) | Large models with hybrid parallelism |
| `galaxy/` | Galaxy (32 chips) | Very large models, maximum scale |

**Sources:**[docs/src/experiments.md 9-40](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/docs/src/experiments.md?plain=1#L9-L40)

## Excluded Directories and Artifacts

The [.gitignore 1-20](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L1-L20) file excludes runtime artifacts and generated files from version control. Understanding these exclusions is important for developers running experiments.

**Sources:**[.gitignore 1-20](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L1-L20)

### Runtime Artifacts

These directories are created during experiment execution and contain data specific to individual runs:

*   **data/**: Downloaded datasets (MNIST, SST-2, etc.) cached locally to avoid repeated downloads
*   **checkpoints/**: Model checkpoint files saved during training for resumption and evaluation
*   **wandb/**: Weights & Biases logging artifacts, including run metadata and uploaded artifacts
*   **results/**: Experiment output files, metrics, and generated artifacts
*   **logs/**: Training logs and debug output files
*   **model/**: Downloaded pretrained model files (e.g., from HuggingFace Hub)

**Sources:**[.gitignore 1-9](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L1-L9)

### Python and Build Artifacts

*   ****/__pycache__/**: Python bytecode files generated at runtime
*   ***.egg-info/**: Package metadata created during installation
*   **generated_modules/**: Dynamically generated Python modules
*   **generated/**: Other generated files from build processes

**Sources:**[.gitignore 2-10](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L2-L10)

### Environment Management

The `env/` directory follows a special pattern:

*   **env/***: All contents are excluded by default (including `xla_env/`, `ffe_env/`, `gpu_env/` virtual environments)
*   **Exceptions**: The `env/activate` script and all `*_requirements.txt` files are explicitly included in version control

This pattern ensures that virtual environments (which can be large and platform-specific) are not committed, while the scripts and dependency specifications needed to recreate them are preserved.

**Sources:**[.gitignore 14-19](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/.gitignore#L14-L19)

## Package Import Paths

After installation via `pip install -e .` (which runs `setup.py`), the following import patterns are available throughout the codebase:

| Import Pattern | Example | Purpose |
| --- | --- | --- |
| Dataset classes | `from blacksmith.datasets.torch.mnist import MNISTDataset` | Access dataset loaders |
| Model functions | `from blacksmith.models.torch.llama import get_model` | Load model architectures |
| Configuration classes | `from blacksmith.experiments.torch.mnist.configs import TrainingConfig` | Define experiment parameters |
| Experiment utilities | `from blacksmith.experiments.torch.mnist.single_chip import test_mnist_training` | Run experiments |

The `blacksmith` package prefix is consistent across all imports, making it clear which code belongs to the TT-Blacksmith project versus external dependencies.
