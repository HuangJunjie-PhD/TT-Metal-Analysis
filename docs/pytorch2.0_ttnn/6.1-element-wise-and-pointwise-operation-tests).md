## Purpose and Scope

This document describes the test infrastructure for validating element-wise and pointwise operation lowering from PyTorch ATen operations to TTNN operations. Element-wise operations apply a function independently to each element of one or more tensors (e.g., `relu`, `add`, `mul`), while pointwise operations are a subset that operate on corresponding elements at matching tensor positions.

These tests focus on individual operation transformations rather than complete model execution. For full model testing, see [Computer Vision Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.1-computer-vision-model-tests) and [Natural Language Processing Model Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5.2-natural-language-processing-model-tests). For tensor manipulation operations like `reshape`, `permute`, and `concat`, see [Tensor Manipulation Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.2-tensor-manipulation-operation-tests). For tensor creation operations like `arange`, `zeros`, and `ones`, see [Creation and Specialized Operation Tests](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.3-creation-and-specialized-operation-tests).

## Test Pattern and Framework

All element-wise and pointwise operation tests follow a consistent pattern that validates both graph transformation and numerical accuracy:

### Common Test Structure

Each test module implements the following structure:

1.   **Module Definition** - A `torch.nn.Module` subclass containing the operation under test
2.   **Input Generation** - Parametrized test inputs with various shapes and data types
3.   **Pre-compilation Execution** - Run the operation in PyTorch to get reference results
4.   **Compilation** - Apply `torch.compile` with `torch_ttnn.backend`
5.   **Post-compilation Execution** - Run the compiled module
6.   **Graph Validation** - Verify the operation was lowered to TTNN
7.   **Accuracy Validation** - Compare results using appropriate tolerances

**Sources:**[tests/lowering/eltwise/unary/test_hardsigmoid.py 12-64](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/unary/test_hardsigmoid.py#L12-L64)[tests/lowering/eltwise/binary/test_pow_scalar.py 12-45](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py#L12-L45)

## Unary Operation Tests

Unary operations transform a single input tensor element-wise. The test suite validates operations in the `tests/lowering/eltwise/unary/` directory.

### Activation Functions

### Test Example: Hardsigmoid

The `test_hardsigmoid.py` test demonstrates the standard pattern:

`class HardSigmoidModule(torch.nn.Module):    def forward(self, x):        return torch.nn.functional.hardsigmoid(x)`
The test validates multiple input shapes and verifies:

*   `torch.ops.aten.hardsigmoid.default` is removed from the graph
*   `ttnn.hardsigmoid` appears in the compiled graph
*   Results match within PCC (Pearson Correlation Coefficient) tolerance

**Parametrized Input Shapes:**

*   Various 4D shapes: `(1, 72, 1, 1)`, `(1, 120, 1, 1)`, `(1, 480, 1, 1)`, etc.
*   3D shapes: `(7, 7, 7)`
*   2D shapes: `(420, 69)`
*   1D shapes: `(1337,)`

## Binary Operation Tests

Binary operations combine two tensors element-wise. These tests are located in `tests/lowering/eltwise/binary/`.

### Test Example: Power Operation with Scalar

The `test_pow_scalar.py` demonstrates binary operation testing:

`class PowTensorScalarModule(torch.nn.Module):    def forward(self, input, exponent):        square = torch.ops.aten.pow.Scalar(input, exponent)        return square`
**Test Parameters:**

*   Scalar inputs: `3`, `0`, `2.2`
*   Exponent tensor shapes: `[2, 2]`, `[1, 1]`, `(2, 2)`

**Validation:**

*   Verifies `ttnn.pow` appears in compiled graph
*   Uses `assert_with_ulp` for most cases (ULP=5)
*   Uses `assert_allclose` with relaxed tolerances for edge cases (input=0)

**Sources:**[tests/lowering/eltwise/binary/test_pow_scalar.py 12-45](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py#L12-L45)

### Broadcasting and Shape Compatibility

Binary operation tests validate various broadcasting scenarios:

1.   **Same Shape** - Both tensors have identical shapes
2.   **Scalar Broadcasting** - One operand is a scalar, broadcast to tensor shape
3.   **Dimension Broadcasting** - Tensors with compatible shapes (e.g., `[3, 1, 5]` + `[3, 4, 1]`)
4.   **Rank Broadcasting** - Tensors with different ranks (e.g., `[2, 3, 4]` + `[4]`)

**Sources:**[tests/lowering/eltwise/binary/test_pow_scalar.py 21-45](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py#L21-L45)

## Guard System Integration

Element-wise and pointwise operations interact with the guard system to prevent lowering of incompatible input variations. The guard system has two components:

### Autogenerated Blocklists

The `to_tt_guard_autogen.py` file contains operation-specific blocklists derived from test failures:

**Example: `_log_softmax` Blocklist**

`aten__log_softmax_default_blocklist = [    ["Tensor<[19, 256008]> self = ?", "int dim = 1", "bool half_to_float = False"]]`
This blocks the specific input variation that causes issues during lowering.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard_autogen.py 12](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard_autogen.py#L12-L12)

### Manual Blocklists

The `to_tt_guard.py` file supplements autogenerated blocklists with manually curated entries, often with detailed explanations:

**Example: `gt.Scalar` Blocklist**

`# ttnn.from_torch(torch.tensor(False)) => ttnn.gt# ttnn.from_torch not support scalar of bool dtypeaten_gt_Scalar_blocklist = [["Tensor<[]> self = ?", "number other = 0"]]`
This prevents lowering operations that would require unsupported scalar conversions.

**Sources:**[torch_ttnn/passes/lowering/to_tt_guard.py 36](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/lowering/to_tt_guard.py#L36-L36)

## Constant Folding for Element-wise Operations

The `ConstantFoldingPass` evaluates element-wise operations with constant inputs at compile-time, integrating with the guard system to ensure TTNN compatibility:

The pass includes element-wise operations in its `foldable_ops` set:

`self.foldable_ops = {    torch.ops.aten.pow.Tensor_Tensor,    torch.ops.aten.add.Tensor,    torch.ops.aten.mul.Tensor,    torch.ops.aten.sub.Tensor,    # ... more operations}`
Before folding, the pass checks `can_lowering_to_ttnn(node)` to ensure the operation would be TTNN-compatible, preventing folding of operations that would fail during lowering.

**Sources:**[torch_ttnn/passes/constant_folding_pass.py 13-37](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L13-L37)[torch_ttnn/passes/constant_folding_pass.py 51-53](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/torch_ttnn/passes/constant_folding_pass.py#L51-L53)

## Accuracy Validation Methods

Element-wise and pointwise operation tests use multiple validation methods depending on the operation characteristics:

| Method | Usage | Tolerance | File |
| --- | --- | --- | --- |
| `assert_with_pcc` | Most operations | PCC ≥ threshold | `tests/utils.py` |
| `assert_with_ulp` | Operations with precision requirements | ULP ≤ threshold | `tests/utils.py` |
| `assert_allclose` | Operations with known numeric drift | rtol, atol parameters | PyTorch built-in |
| `torch.allclose` | Simple equality checks | Default tolerances | PyTorch built-in |

### PCC (Pearson Correlation Coefficient) Validation

Used for operations where relative relationships matter more than absolute values:

`assert_with_pcc(result_before, result_after)`
**Sources:**[tests/lowering/eltwise/unary/test_hardsigmoid.py 63](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/unary/test_hardsigmoid.py#L63-L63)

### ULP (Units in Last Place) Validation

Used for operations requiring specific floating-point precision:

`assert_with_ulp(result_before, result_after, 5)`
The ULP threshold (e.g., 5) specifies the maximum acceptable difference in the least significant bits.

**Sources:**[tests/lowering/eltwise/binary/test_pow_scalar.py 44](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py#L44-L44)

### Conditional Accuracy Validation

Tests may use different validation methods based on input characteristics:

`if input == 0:    assert_allclose(result_before, result_after, rtol=0.02, atol=1e-5)else:    assert_with_ulp(result_before, result_after, 5)`
This accounts for operations that behave differently at boundary conditions (e.g., `0^x`).

**Sources:**[tests/lowering/eltwise/binary/test_pow_scalar.py 41-44](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_pow_scalar.py#L41-L44)

## Test Coverage and Autogenerated Tests

The system automatically generates additional operation tests from metrics captured during model execution. The `generate_input_variation_test_from_models.py` tool creates tests for specific input variations that failed during model compilation:

1.   **Metrics Collection** - Model tests capture `InputVariation` data for each operation
2.   **Test Generation** - Tool generates parametrized tests for problematic variations
3.   **Validation** - Generated tests run in `tests/autogen-op/` directory
4.   **Guard Updates** - Failed variations are added to guard blocklists

This creates a feedback loop where model-level failures automatically generate focused operation-level tests.
