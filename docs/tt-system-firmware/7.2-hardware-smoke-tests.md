## Purpose and Scope

Hardware smoke tests validate basic firmware functionality on physical Blackhole and Wormhole boards. These tests run automatically on every pull request and push to verify that firmware changes do not break fundamental operations like chip enumeration, ARC message handling, DMC communication, and hardware initialization. The tests execute on real hardware using JTAG access and typically complete within minutes.

For comprehensive stress testing with 1000+ iterations, see [Stress and Long-Duration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.4-stress-and-long-duration-tests). For TT-Metal workload validation, see [TT-Metal Integration Tests](https://deepwiki.com/tenstorrent/tt-system-firmware/7.5-tt-metal-integration-tests). For unit tests running on simulated platforms, see [Unit Testing with Twister](https://deepwiki.com/tenstorrent/tt-system-firmware/7.1-unit-testing-with-twister).

## Test Categories

Hardware smoke tests are organized into three primary categories based on the firmware components and platforms they target:

| Test Category | Target Platform | Components Tested | Workflow Job | Typical Duration |
| --- | --- | --- | --- | --- |
| **DMC Smoke Tests** | Blackhole boards (p100a, p150a, p300a) | DMC firmware, JTAG communication, SMBus target | `hardware-smoke-test` → `run-dmc-tests` | ~2-3 minutes |
| **SMC Smoke Tests** | Blackhole boards (p100a, p150a, p300a) | SMC firmware, ARC messages, telemetry | `hardware-smoke-test` → `run-smc-tests` | ~2-3 minutes |
| **E2E Smoke Tests** | Blackhole and Wormhole boards | Full SMC→DMC→CMFW chain, firmware updates, hardware validation | `smoke-e2e-test`, `smoke-e2e-wh-test` | ~10-15 minutes |

Galaxy boards are excluded from smoke tests because they lack a recovery solution (see [.github/workflows/hardware-smoke.yml 40-41](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/hardware-smoke.yml#L40-L41)).

## Test Infrastructure Components

### Pytest Fixtures

The pytest framework uses a hierarchical fixture system where `unlaunched_dut` represents the unflashed firmware bundle, `launched_arc_dut` flashes it once per session, and `arc_chip_dut` ensures the chip is ready before each test function. The `wait_arc_boot` helper ([app/smc/pytest/e2e_smoke.py 146-182](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L146-L182)) validates boot status by checking that `ARC_STATUS` register contains the magic value `0xC0DE0000` plus a postcode ≥ `0x1D`.

**Sources:**[app/smc/pytest/e2e_smoke.py 128-198](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L128-L198)[app/smc/pytest/e2e_smoke.py 146-182](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L146-L182)[app/smc/pytest/e2e_smoke.py 200-208](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L200-L208)

### Reset and Recovery Tests

| Test Function | Reset Type | Recovery Mechanism | Success Criteria |
| --- | --- | --- | --- |
| `test_arc_watchdog` | ARC watchdog timeout | DMC monitors `ARC_HANG_PC_REG_ADDR` | Chip resets within 1100ms, hang PC ≠ 0 |
| `test_smi_reset` | SMI-initiated reset | `tt-smi -r` command | 10 successful resets, response code 0 |
| `test_dirty_reset` | DMC reset via JTAG | `dmc_reset.reset_dmc()` | SMC boots within 60s after unexpected reset |

The `arc_watchdog_test` helper ([app/smc/pytest/e2e_smoke.py 551-636](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L551-L636)) sets a 1000ms watchdog timeout, halts the ARC cores by writing `0xF0` to `ARC_MISC_CTRL`, then waits for the DMC to detect the hang and reset the chip. The test verifies timing: the reset should occur after 500ms but before 1100ms.

**Sources:**[app/smc/pytest/e2e_smoke.py 551-644](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L551-L644)[app/smc/pytest/e2e_smoke.py 707-738](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L707-L738)[app/smc/pytest/e2e_smoke.py 740-787](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L740-L787)

## Hardware-Specific Tests

### Tensix Reset Sequence Test

The `tensix_reset_sequence` ([app/smc/pytest/e2e_smoke.py 790-818](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L790-L818)) performs a careful sequence to reset all Tensix cores:

1.   Force `AICLK` to 250 MHz using `TT_SMC_MSG_FORCE_AICLK` (0x33)
2.   Clear 8 RISC reset registers at `0x80030040 + i*4`
3.   Send `TT_SMC_MSG_TOGGLE_TENSIX_RESET` (0xAF) and `TT_SMC_MSG_REINIT_TENSIX` (0x20)
4.   Write `0x47800` to `SOFT_RESET_ADDR` (0xFFB121B0) via NOC broadcast
5.   Set all RISC reset registers to `0xFFFFFFFF`
6.   Unforce AICLK (send `arg0=0`)

**Sources:**[app/smc/pytest/e2e_smoke.py 790-859](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L790-L859)[app/smc/pytest/e2e_smoke.py 821-859](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L821-L859)

* * *

### Flash Read/Write Test

The `test_flash_write` ([app/smc/pytest/e2e_smoke.py 478-512](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L478-L512)) validates SPI flash operations by:

1.   Reading the SPI RX training region at `0x13FFC` and verifying it contains `0xA5A55A5A`
2.   Writing 5 iterations of random data to the scratch region (`0x2800000` - `0x2810000`)
3.   Each iteration writes `0x8000` bytes at a time to simulate `tt-flash` behavior
4.   Reading back and comparing data to ensure write integrity

This test uses the `pyluwen` library's `chip.as_bh().spi_read()` and `chip.as_bh().spi_write()` methods, which are the same interfaces used by the `tt-flash` tool for firmware updates.

**Sources:**[app/smc/pytest/e2e_smoke.py 478-512](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/smc/pytest/e2e_smoke.py#L478-L512)

## Memory Addresses and Constants

The tests rely on well-known memory addresses and message codes:

### Register Addresses

```
ARC_STATUS                  = 0x80030060  # ARC firmware status and postcode
ARC_MISC_CTRL              = 0x80030100  # ARC core control (halt/resume)
BOOT_STATUS                = 0x80030408  # Hardware boot status
TELEMETRY_DATA_REG_ADDR    = 0x80030430  # Pointer to telemetry table
PCIE_INIT_CPL_TIME_REG     = 0x80030438  # PCIe init completion timestamp
CMFW_START_TIME_REG        = 0x8003043C  # CMFW start timestamp
ARC_START_TIME_REG         = 0x80030440  # Bootrom start timestamp
ARC_HANG_PC_REG            = 0x80030454  # Program counter at hang time
```

### ARC Message Codes

```
TT_SMC_MSG_TEST              = 0x90  # Echo test (response = arg + 1)
TT_SMC_MSG_READ_TS           = 0x1B  # Read temperature sensor
TT_SMC_MSG_READ_PD           = 0x1C  # Read process detector
TT_SMC_MSG_READ_VM           = 0x1D  # Read voltage monitor
TT_SMC_MSG_REINIT_TENSIX     = 0x20  # Reinitialize Tensix cores
TT_SMC_MSG_FORCE_AICLK       = 0x33  # Force AICLK frequency
```
