This document provides an overview of vLLM's support for multimodal models - AI models capable of processing and generating content based on multiple types of data (text, images, audio, video). We'll explore the architecture, interfaces, and implementation details of multimodal support in vLLM, as well as the specific multimodal models currently supported.

For information about specific model implementations like LLaMA or GPT, see [Text Generation Models](https://deepwiki.com/tenstorrent/vllm/3.2-text-generation-models). For details about the model registry and loading mechanism, see [Model Registry](https://deepwiki.com/tenstorrent/vllm/3.1-model-registry-and-loading).

## Architecture Overview

vLLM's multimodal support follows a modular architecture that extends the core language model inference capabilities with multimodal processing. The system is designed to be extensible, allowing new types of multimodal models to be integrated seamlessly.

## Core Interfaces and Components

### SupportsMultiModal Interface

Multimodal models in vLLM must implement the `SupportsMultiModal` interface, which defines the contract between the model and vLLM's multimodal processing system. This interface requires implementation of methods for handling multimodal inputs and integrating them with text inputs.

The key methods include:

*   `get_multimodal_embeddings`: Extracts embeddings from non-text inputs (images, audio, etc.)
*   `get_input_embeddings`: Generates input embeddings from text tokens
*   `get_language_model`: Returns the underlying language model

### MultiModal Processing Architecture

The multimodal processing pipeline is responsible for handling inputs like images, audio, and video, and preparing them for model consumption. It follows these key steps:

1.   **Input Parsing**: Converting raw inputs (e.g., image files) into a standardized format
2.   **Feature Extraction**: Extracting features using specialized encoders (e.g., vision encoder for images)
3.   **Embedding Generation**: Generating embeddings that can be consumed by the language model
4.   **Embedding Integration**: Integrating these embeddings with text embeddings

The main classes involved in this process are:

*   `BaseProcessingInfo`: Provides information about model configuration and capabilities
*   `BaseMultiModalProcessor`: Handles input processing and embedding generation
*   `MultiModalDataParser`: Parses multimodal inputs into a standardized format

### Multimodal Registry

vLLM uses a registry pattern to maintain a catalog of supported multimodal models and their associated processors. This registry, `MULTIMODAL_REGISTRY`, allows vLLM to dynamically discover and instantiate the appropriate processor for a given model.

Models are registered using the `@MULTIMODAL_REGISTRY.register_processor` decorator, which associates:

*   A model class (e.g., `LlavaForConditionalGeneration`)
*   A processor class (e.g., `LlavaMultiModalProcessor`)
*   An info class (e.g., `LlavaProcessingInfo`)
*   A dummy inputs builder (e.g., `LlavaDummyInputsBuilder`)

## Image Processing Pipeline

The image processing pipeline for vision-language models follows these general steps:

1.   **Image Loading and Preprocessing**: Loading images and performing initial preprocessing
2.   **Feature Extraction**: Using a vision encoder (like CLIP, SigLIP, etc.) to extract features
3.   **Projection**: Projecting vision features to align with language model dimensions
4.   **Integration**: Integrating vision embeddings with text embeddings

## Prompt Handling and Placeholder Tokens

A crucial aspect of multimodal processing is managing the prompt by inserting placeholder tokens that will be replaced with image/audio/video embeddings. vLLM provides two main strategies:

1.   **PromptReplacement**: Replaces portions of the prompt with placeholder tokens
2.   **PromptInsertion**: Inserts placeholder tokens at specified positions

These strategies are implemented through the `PromptUpdate` family of classes:

When processing a prompt, the multimodal processor:

1.   Identifies locations for multimodal content
2.   Inserts appropriate placeholder tokens
3.   Tracks the mapping between placeholder positions and multimodal inputs
4.   Passes this mapping to the model to enable embedding replacement

## Model-Specific Implementations

Let's look more closely at a few example model implementations:

### LLaVA (Large Language and Vision Assistant)

LLaVA combines a vision encoder (typically CLIP or SigLIP) with a language model (typically LLaMA). It processes images and generates text about them.

Key components:

*   `vision_tower`: The vision encoder (CLIP/SigLIP)
*   `multi_modal_projector`: Projects vision features to language model dimensions
*   `language_model`: The underlying LLM (LLaMA, etc.)

The LLaVA implementation provides mechanisms for:

*   Processing image pixels into features
*   Selecting relevant features based on strategy
*   Projecting these features to language model dimensions
*   Integrating vision embeddings with text embeddings

Sources:

*   [vllm/model_executor/models/llava.py 500-777](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/llava.py#L500-L777)

### Qwen2-VL (Vision-Language)

Qwen2-VL extends the Qwen2 language model with vision capabilities. It includes:

*   `vision_model`: Custom vision transformer for image processing
*   `vision_projector`: Projects vision features to language model dimensions
*   `language_model`: The Qwen2 language model

Unlike LLaVA, Qwen2-VL implements its own vision encoder and positional embedding strategy.

Sources:

*   [vllm/model_executor/models/qwen2_vl.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/qwen2_vl.py)

### Phi3-V (Microsoft's Vision-Language Model)

Phi3-V builds on Microsoft's Phi-3 family with vision capabilities:

*   `img_processor`: CLIP-based vision encoder
*   `vision_embed_tokens`: Custom embedding generator for images
*   `language_model`: The Phi-3 language model

A unique feature of Phi3-V is its HD transform for high-definition image processing.

## Extending vLLM with New Multimodal Models

To add support for a new multimodal model in vLLM, you need to:

1.   **Implement the Model Class**: Create a class implementing the `SupportsMultiModal` interface
2.   **Create a Processing Info Class**: Define a subclass of `BaseProcessingInfo` that provides model-specific information
3.   **Implement a Processor Class**: Create a subclass of `BaseMultiModalProcessor` for input handling
4.   **Implement a Dummy Inputs Builder**: Create a subclass of `BaseDummyInputsBuilder` for profiling
5.   **Register the Model**: Use the `@MULTIMODAL_REGISTRY.register_processor` decorator

## Conclusion

vLLM provides a robust and extensible framework for supporting multimodal models. The modular architecture, centered around the `SupportsMultiModal` interface and the `MultiModalProcessor` system, allows vLLM to handle a wide range of multimodal models while keeping the core components clean and maintainable.

The registration mechanism through `MULTIMODAL_REGISTRY` makes it straightforward to add support for new models, and the prompt handling system provides flexible ways to integrate multimodal content with text.

As multimodal AI continues to evolve, vLLM's architecture is positioned to incorporate new modalities and model architectures with minimal changes to the core system.
