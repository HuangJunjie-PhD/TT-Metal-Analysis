This page documents the complete boot and initialization sequence for Tenstorrent Blackhole systems, from power-on through hardware initialization to operational firmware. The sequence involves multiple firmware components: DMC (Device Management Controller) firmware running on RISC-V, CMFW (Chip Management Firmware) running on the ARC processor within the ASIC, and specialized firmware for MRISC, ERISC, and SerDes controllers.

For information about runtime management after initialization is complete, see DVFS and Throttling ([2.1.3](https://deepwiki.com/tenstorrent/tt-system-firmware/2.1.3-dvfs-and-throttling)). For details on the communication protocols established during initialization, see CM2DM Communication Protocol ([2.2.2](https://deepwiki.com/tenstorrent/tt-system-firmware/2.2.2-cm2dm-communication-protocol)).

## Overview of Boot Architecture

The boot process follows a hierarchical three-stage architecture:

**Stage 1: DMC Firmware Boot** - The DMC firmware boots on the board's RISC-V controller and performs board-level initialization including BIST, power detection, and GPIO setup.

**Stage 2: ASIC Reset and ARC Boot** - DMC loads bootrom code to the ASIC via JTAG and releases the ARC processor from reset, allowing CMFW to begin execution.

**Stage 3: Hardware Subsystem Initialization** - CMFW executes an ordered initialization sequence using Zephyr's `SYS_INIT` mechanism to bring up PCIe, Tensix, GDDR, and Ethernet subsystems.

## DMC Firmware Initialization

The DMC firmware runs on a RISC-V processor on the board and is responsible for managing chip-level operations including ASIC reset, bootrom loading, and communication with CMFW.

### DMC Main Function Entry

The DMC `main()` function performs the following initialization steps:

| Step | Function | Purpose |
| --- | --- | --- |
| 1 | `tt_bist()` | Built-in self-test (optional) |
| 2 | Fan speed initialization | Set initial fan speed to 35% |
| 3 | `boot_write_img_confirmed()` | Confirm firmware update via MCUBoot |
| 4 | SPI mux configuration | Force all SPI muxes to ARC control |
| 5 | GPIO setup | Configure board fault LED, therm trip, pgood interrupts |
| 6 | Bootrom loading | `jtag_bootrom_init()` and reset sequence |
| 7 | SMBus configuration | Configure SMBus in controller mode with PEC |
| 8 | Power detection | `detect_max_power()` reads PSU sense pins |
| 9 | Timer initialization | Start periodic timers for telemetry and events |
| 10 | Main event loop | `tt_event_wait()` processes events forever |

### DMC Event Loop

After initialization completes, the DMC enters an event-driven main loop that processes the following events:

*   **TT_EVENT_THERM_TRIP**: Thermal trip interrupt from ASIC (forces reset and 100% fan)
*   **TT_EVENT_WATCHDOG_EXPIRED**: ARC watchdog timeout (reads PC and resets)
*   **TT_EVENT_PERST**: PCIe reset signal (triggers ASIC reset)
*   **TT_EVENT_PGOOD**: Power good signal change (monitors power supply)
*   **TT_EVENT_BOARD_POWER_TO_SMC**: Update board power telemetry (1ms period)
*   **TT_EVENT_FAN_RPM_TO_SMC**: Update fan RPM telemetry (20ms period)
*   **TT_EVENT_CM2DM_POLL**: Poll for CM2DM messages from CMFW (20ms period)
*   **TT_EVENT_LOGS_TO_SMC**: Forward DMC logs to SMC (20ms period)

## Hardware Subsystem Initialization

### PCIe Subsystem Initialization

PCIe initialization configures SerDes, TLBs (Translation Lookaside Buffers), and performs link training. The `pcie_init()` function runs at priority 103.

**Key steps:**

1.   **Configure PCIe TLBs** - Setup TLB windows for SerDes registers, SII registers, and DBI access via `ConfigurePCIeTlbs()`
2.   **Setup DBI Access** - Configure NOC TLB for Direct Bus Interface access via `SetupDbiAccess()`
3.   **SerDes Initialization** - Call `SerdesInit()` from `libpciesd.a` to configure PHY
4.   **Controller Configuration** - Call `CntlInitV2()` to configure BARs, device type, speed
5.   **Setup Outbound TLBs** - Configure 8 outbound TLB windows via `SetupOutboundTlbs()`
6.   **Setup Inbound TLBs** - Configure inbound 4G TLB window via `SetupInboundTlbs()`
7.   **Link Training** - Wait up to 2000ms for link training to complete
8.   **Enable MSI** - Configure Message Signaled Interrupts
9.   **Register Reset Interrupt** - Connect PCIe error interrupt to `ChipResetRequest()`

Sources: [lib/tenstorrent/bh_arc/pcie.c 377-470](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L377-L470)[lib/tenstorrent/bh_arc/pcie.c 319-339](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L319-L339)[lib/tenstorrent/bh_arc/pcie.c 183-193](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L183-L193)[lib/tenstorrent/bh_arc/pcie.c 277-315](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/pcie.c#L277-L315)

### Tensix Compute Core Initialization

Tensix initialization enables clock gating, clears L1 memory, and wipes DEST registers. The `tensix_init()` function runs at priority 104.

**Initialization sequence:**

1.   **Enable Clock Gating** - Configure `CG_CTRL_EN` register to enable clock gating for all functional blocks (FPU, packers, unpackers, TRISCs, L1) with hysteresis of 2 cycles via `EnableTensixCG()`

2.   **Wipe L1 Memory** - Zero all L1 memory in enabled Tensix cores:

    *   Select arbitrary non-harvested Tensix core
    *   Zero first `SCRATCHPAD_SIZE` (128KB) using DMA from SRAM buffer
    *   Copy zeros to entire 1.5MB L1 using NOC DMA
    *   Broadcast cleared L1 to all other Tensix cores using multicast TLB

3.   **Wipe DEST Registers** - Clear DEST register in each Tensix core by running firmware on TRISC 0:

    *   Load `destwipe` firmware from SPI flash to all Tensix L1 using multicast
    *   Write counter coordinates to L1 mailbox
    *   Set TRISC 0 reset PC to firmware load address
    *   Release TRISC 0 from soft reset
    *   Wait for all cores to signal completion via atomic counter (10ms timeout)
    *   Re-assert TRISC 0 soft reset

Sources: [lib/tenstorrent/bh_arc/tensix_init.c 326-346](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L326-L346)[lib/tenstorrent/bh_arc/tensix_init.c 84-108](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L84-L108)[lib/tenstorrent/bh_arc/tensix_init.c 117-178](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L117-L178)[lib/tenstorrent/bh_arc/tensix_init.c 241-315](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/tensix_init.c#L241-L315)

MRISC firmware is loaded to GDDR controllers to manage memory training. The `InitMrisc()` function runs at priority 105, followed by `gddr_training()` at priority 111.

**MRISC Loading Sequence:**

**GDDR Training Sequence:**

1.   **Poll MRISC_INIT_STATUS** - Wait for each GDDR instance to signal `MRISC_INIT_FINISHED` (0xdeadbeef) or `MRISC_INIT_FAILED` (0xfa11) with 1000ms timeout

2.   **Run Hardware BIST** - Execute built-in self-test via MRISC message protocol:

    *   Check MRISC FW version (must be >= 2.7)
    *   Send `MRISC_MSG_TYPE_RUN_MEMTEST` message with test parameters

### Ethernet Subsystem Initialization

Ethernet initialization loads SerDes firmware and ERISC firmware for each enabled Ethernet instance. The `eth_init()` function runs at priority 106.

**Ethernet Initialization Steps:**

1.   **Setup SerDes Mux** - Configure `RESET_UNIT_PCIE_MISC_CNTL3` registers to route SerDes lanes to Ethernet instances based on PCIe configuration via `SetupEthSerdesMux()`

2.   **Initialize SerDes** - For each SerDes instance (0-5):

    *   Load SerDes register configuration from `ethsdreg` tag
    *   Load SerDes firmware from `ethsdfw` tag
    *   Configure SerDes for Ethernet mode

3.   **Wipe Ethernet L1** - Zero 512KB L1 memory in each enabled Ethernet instance using NOC DMA

4.   **Load ERISC Firmware** - For each enabled Ethernet instance:

    *   Load firmware from `ethfw` tag (starting at 0x70000)
    *   Set RISC0 reset PC to firmware load address
    *   Set RISC0 end PC to 0x7bffc

5.   **Load ERISC Configuration** - For each enabled Ethernet instance:

    *   Read configuration from `ethfwcfg` tag
    *   Insert dynamic values: `eth_sel`, speed override, board ID, MAC address base, ASIC ID
    *   Write configuration to L1 at 0x7c000
    *   Release RISC0 from reset

## Reset and Recovery Mechanisms

The system implements multiple reset mechanisms for fault recovery:

### ASIC Reset

Triggered by thermal trip, watchdog timeout, or host request. DMC performs:

1.   `bh_chip_cancel_bus_transfer_clear()` - Clear bus cancel flag
2.   `bharc_disable_i2cbus()` - Disable I2C bus to ARC
3.   `jtag_bootrom_reset_sequence()` - Full JTAG reset and bootrom reload
4.   `bharc_enable_i2cbus()` - Re-enable I2C bus

Sources: [lib/tenstorrent/bh_chip/bh_chip.c 172-190](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_chip/bh_chip.c#L172-L190)[app/dmc/src/main.c 452-482](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L452-L482)

### DMC Reset

Triggered by host request or severe fault. Causes DMC firmware to reboot via `sys_reboot(SYS_REBOOT_COLD)`, which also resets the ASIC.

Sources: [lib/tenstorrent/bh_arc/cm2dm_msg.c 110-116](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/lib/tenstorrent/bh_arc/cm2dm_msg.c#L110-L116)

### Watchdog Reset

If telemetry heartbeat stops for `auto_reset_timeout` milliseconds, DMC triggers watchdog reset:

1.   Read ARC program counter via JTAG (`RESET_UNIT_ARC_PC_CORE_0`)
2.   Store PC in `arc_hang_pc` for debugging
3.   Set fan to 100%
4.   Perform ASIC reset

Sources: [app/dmc/src/main.c 419-449](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/app/dmc/src/main.c#L419-L449)

## Initialization Completion

Initialization is considered complete when:

1.   All `SYS_INIT` functions have executed (priority 90-113)
2.   GDDR training has completed successfully
3.   SMBus target interface is operational
4.   CMFW sends `kCm2DmMsgIdReady` message to DMC
5.   DMC sends `dmStaticInfo` with version information to CMFW
6.   Telemetry system begins reporting valid data

At this point, the system transitions to operational state where DVFS, throttling, and runtime management take over.
