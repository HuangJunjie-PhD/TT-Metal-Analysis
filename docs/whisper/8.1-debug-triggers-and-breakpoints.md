## Purpose and Scope

This document describes the debug trigger and breakpoint subsystem in Whisper. Debug triggers provide hardware-assisted breakpoints and watchpoints for debugging RISC-V programs. They can monitor instruction execution, memory accesses (loads/stores), instruction counts, exceptions, and interrupts. When a trigger condition is met, the system can raise a breakpoint exception or enter debug mode.

For information about debug mode execution itself, see the debugging sections of the Hart documentation. For interactive debugging commands, see [Interactive Command-Line Interface](https://deepwiki.com/tenstorrent/whisper/6.2-interactive-command-line-interface). For GDB remote debugging, see [GDB Remote Debugging](https://deepwiki.com/tenstorrent/whisper/6.4-gdb-remote-debugging).

## Trigger Types

Whisper supports multiple trigger types defined by the RISC-V Debug Specification. Each trigger type serves a specific debugging purpose.

### Supported Trigger Types

| Type | Enum Value | Name | Purpose |
| --- | --- | --- | --- |
| 0 | `TriggerType::None` | None | No trigger configured |
| 2 | `TriggerType::Mcontrol` | Address/Data Match (legacy) | Match instruction addresses, load/store addresses, or data values |
| 3 | `TriggerType::Icount` | Instruction Count | Trigger after N instructions execute |
| 4 | `TriggerType::Itrigger` | Interrupt Trigger | Trigger on specific interrupt causes |
| 5 | `TriggerType::Etrigger` | Exception Trigger | Trigger on specific exception causes |
| 6 | `TriggerType::Mcontrol6` | Address/Data Match | Enhanced version of Mcontrol with additional features |
| 15 | `TriggerType::Disabled` | Disabled | Trigger is explicitly disabled |

**Sources:**[Triggers.hpp 31-35](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L31-L35)[Triggers.cpp 34-43](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L34-L43)

### Mcontrol and Mcontrol6 (Address/Data Triggers)

Address/data triggers match instruction addresses, load/store addresses, or load/store data values. They support various match types and can trigger before or after the triggering instruction.

#### Mcontrol/Mcontrol6 Configuration Fields

**Sources:**[Triggers.hpp 62-108](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L62-L108)[Triggers.hpp 112-163](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L112-L163)

**Key differences between Mcontrol and Mcontrol6:**

*   Mcontrol6 has separate `size` field (3 bits) vs. Mcontrol's `sizelo`/`sizehi`
*   Mcontrol6 supports virtual supervisor mode (`vs`, `vu` bits)
*   Mcontrol6 has `hit0`/`hit1` instead of single `hit` bit
*   Mcontrol6 timing is always "before" (no timing field)

### Icount (Instruction Count Trigger)

Icount triggers fire after a specified number of instructions have been executed. The count decrements on each instruction retirement.

**Sources:**[Triggers.hpp 166-182](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L166-L182)[Triggers.cpp 400-461](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L400-L461)

### Etrigger (Exception Trigger)

Exception triggers fire when an exception with a matching cause occurs. The TDATA2 register contains a bitmask where bit N corresponds to exception cause N.

**Sources:**[Triggers.hpp 185-202](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L185-L202)[Triggers.cpp 464-520](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L464-L520)

### Itrigger (Interrupt Trigger)

Interrupt triggers fire when an interrupt with a matching cause is taken. Like Etrigger, TDATA2 contains a bitmask of interrupt causes. The `nmi` bit determines if the trigger matches NMI or regular interrupts.

**Sources:**[Triggers.hpp 204-221](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L204-L221)[Triggers.cpp 523-584](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L523-L584)

## Trigger Components and CSRs

Triggers are accessed through Control and Status Registers (CSRs). Each trigger has multiple components that define its behavior.

### Trigger CSR Layout

| CSR | Name | Purpose | Offset |
| --- | --- | --- | --- |
| TSELECT | Trigger Select | Selects which trigger to access | N/A |
| TDATA1 | Trigger Data 1 | Control register (type, action, match, enables) | 0 |
| TDATA2 | Trigger Data 2 | Match value (address, data, or cause mask) | 1 |
| TDATA3 | Trigger Data 3 | Additional context (implementation-specific) | 2 |
| TINFO | Trigger Info | Reports supported trigger types | 3 |
| TCONTROL | Trigger Control | Controls trigger behavior in machine mode | N/A |

## Trigger Match Types

Address/data triggers (Mcontrol/Mcontrol6) support multiple match types to compare addresses or data values against TDATA2.

### Match Type Enumeration

| Value | Match Type | Comparison | Negated Version |
| --- | --- | --- | --- |
| 0 | `Match::Equal` | `item == tdata2` | 8: NotEqual |
| 1 | `Match::Masked` | `(item & mask) == (tdata2 & mask)` | 9: NotMasked |
| 2 | `Match::GE` | `item >= tdata2` | N/A |
| 3 | `Match::LT` | `item < tdata2` | N/A |
| 4 | `Match::MaskHighEqualLow` | Mask low half, compare | 12: NotMaskHighEqualLow |
| 5 | `Match::MaskLowEqualHigh` | Mask high half, compare | 13: NotMaskLowEqualHigh |

### NAPOT (Naturally Aligned Power-of-Two) Matching

For NAPOT matching, TDATA2 encodes both the base address and size. The least significant 1-bit in TDATA2 determines the size:

*   If TDATA2 = `0x1000`, size = 4KB (bit 12 is LSB 1)
*   If TDATA2 = `0x3000`, size = 8KB (bit 13 is LSB 1)

## Trigger Actions

When a trigger condition is met, one of several actions can be taken. The action is specified in the `action` field of TDATA1.

### Action Types

| Value | Action | Description |
| --- | --- | --- |
| 0 | `RaiseBreak` | Raise a breakpoint exception |
| 1 | `EnterDebug` | Enter debug mode (requires `dmode=1`) |
| 2 | `StartTrace` | Start trace collection |
| 3 | `StopTrace` | Stop trace collection |
| 4 | `EmitTrace` | Emit trace packet |
| 6 | `External0` | Implementation-defined external action 0 |
| 7 | `External1` | Implementation-defined external action 1 |

**Sources:**[Triggers.hpp 38-39](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L38-L39)[Triggers.cpp 46-54](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L46-L54)

### Action Selection Constraints

*   `EnterDebug` action requires `dmode=1` (debug mode only)
*   Action must be in the supported actions list (configurable)
*   If an unsupported action is written, the old action is preserved

**Sources:**[Triggers.cpp 200-205](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L200-L205)[Triggers.cpp 712-731](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L712-L731)

## Trigger Chaining

Multiple triggers can be chained together using the `chain` bit in TDATA1. In a chain, all triggers must match for the action to fire. This enables complex trigger conditions.

### Chain Behavior

**Sources:**[Triggers.cpp 253-278](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L253-L278)[Triggers.cpp 906-925](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L906-L925)

### Chain Boundaries

Chain boundaries are determined by the `chain` bit:

*   A chain starts after a trigger with `chain=0`
*   A chain continues while triggers have `chain=1`
*   A chain ends at a trigger with `chain=0`

The `defineChainBounds()` method computes these boundaries and stores them in each trigger.

**Sources:**[Triggers.cpp 906-925](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L906-L925)[Triggers.hpp 771-781](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L771-L781)

## Trigger Timing

Triggers can fire at two different timing points: **before** or **after** the triggering instruction executes.

### Timing Semantics

| Timing | When Checked | Effects |
| --- | --- | --- |
| `TriggerTiming::Before` | Before instruction executes | Instruction does not execute; PC not updated |
| `TriggerTiming::After` | After instruction completes | Instruction fully executes; side effects visible |

## Trigger Configuration

Triggers are configured through their CSR components. Configuration includes reset values, write masks, and poke masks.

### Configuration Methods

The `Triggers` class provides methods to configure individual triggers:

| Method | Purpose |
| --- | --- |
| `config(unsigned trigger, resets, masks, pokeMasks)` | Configure trigger with reset/mask values |
| `reset()` | Reset all triggers to configured reset values |
| `poke(trigger, v1, v2, v3)` | Direct write (bypasses masks) |
| `writeData1/2/3(trigger, debugMode, value)` | CSR write (respects masks) |

**Sources:**[Triggers.hpp 1010-1043](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L1010-L1043)[Triggers.cpp 587-630](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L587-L630)

### Configuration Constraints

When writing TDATA1:

*   Type changes to unsupported types are rejected
*   Action changes to unsupported actions are rejected
*   Writing type=0 (None) is converted to type=15 (Disabled)
*   `dmode=1` requires debug mode for writes
*   Chain bit changes trigger chain boundary recomputation

**Sources:**[Triggers.cpp 147-215](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L147-L215)

### Write and Poke Masks

Each TDATA register has two masks:

*   **Write Mask**: Controls which bits can be modified by CSR instructions
*   **Poke Mask**: Controls which bits can be modified by test bench pokes (typically wider than write mask)

**Sources:**[Triggers.hpp 474-484](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L474-L484)

The `Hart` class integrates trigger checks throughout the instruction execution pipeline.

### Cached Trigger State

For performance, the Hart maintains cached state about whether triggers are active:

| Cached Flag | Purpose |
| --- | --- |
| `hasActiveTrigger()` | Any trigger enabled |
| `hasActiveInstTrigger()` | Instruction address trigger enabled |
| `hasEnterDebugModeTripped()` | EnterDebug action triggered |
| `hasBreakpTripped()` | RaiseBreak action triggered |

These flags avoid checking all triggers on every instruction when no triggers are enabled.

## Privilege Mode Filtering

Triggers can be enabled or disabled for specific privilege modes using the mode bits in TDATA1:

| Bit | Privilege Mode |
| --- | --- |
| `m` | Machine mode |
| `s` | Supervisor mode |
| `u` | User mode |
| `vs` | Virtual supervisor mode (Mcontrol6, Icount, Etrigger, Itrigger) |
| `vu` | Virtual user mode (Mcontrol6, Icount, Etrigger, Itrigger) |

A trigger only matches if the corresponding mode bit is set for the current privilege mode.

**Sources:**[Triggers.cpp 979-1007](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L979-L1007)[Triggers.cpp 1095-1117](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L1095-L1117)

## Machine Mode Trigger Control (TCONTROL)

The TCONTROL CSR controls whether triggers fire in machine mode:

*   When `tcontrol.mte=0`: Triggers cannot fire in machine mode
*   When `tcontrol.mte=1`: Triggers with action `EnterDebug` can fire in machine mode

This prevents triggers with `RaiseBreak` action from firing in machine mode when interrupts are disabled, avoiding trigger reentrancy issues.

**Sources:**[Triggers.hpp 1148-1184](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.hpp#L1148-L1184)[Triggers.cpp 282-308](https://github.com/tenstorrent/whisper/blob/733fd921/Triggers.cpp#L282-L308)

## Example Trigger Configurations

### Instruction Address Breakpoint

To break on instruction at address 0x80000000:

```
TDATA1:
  type = 6 (Mcontrol6)
  action = 0 (RaiseBreak) or 1 (EnterDebug)
  execute = 1
  match = 0 (Equal)
  select = 0 (MatchAddress)
  m = 1, s = 1, u = 1  (all modes)
  
TDATA2:
  0x80000000
```

### Load/Store Watchpoint

To watch writes to address 0x10008:

```
TDATA1:
  type = 6 (Mcontrol6)
  action = 1 (EnterDebug)
  store = 1
  match = 0 (Equal)
  select = 0 (MatchAddress)
  size = 0 (any size)
  
TDATA2:
  0x10008
```

### Instruction Count Breakpoint

To break after 1000 instructions:

```
TDATA1:
  type = 3 (Icount)
  action = 1 (EnterDebug)
  count = 1000
  m = 1, s = 1, u = 1
```

### Chained Condition

To break when PC=0x1000 AND loading from 0x2000:

```
Trigger 0:
  TDATA1: type=6, execute=1, match=0, chain=1
  TDATA2: 0x1000
  
Trigger 1:
  TDATA1: type=6, load=1, match=0, chain=0
  TDATA2: 0x2000
```

## Summary

The debug trigger subsystem provides hardware-assisted debugging capabilities:

*   **Multiple trigger types**: Mcontrol/Mcontrol6 for address/data, Icount for instruction counts, Etrigger/Itrigger for exceptions/interrupts
*   **Flexible matching**: Equal, masked, range, NAPOT addressing
*   **Multiple actions**: Breakpoint exception, debug mode entry, trace control
*   **Chaining**: Complex conditions via trigger chains
*   **Privilege filtering**: Per-mode enable bits
*   **Integration**: Checks throughout Hart pipeline (fetch, execute, retire)

The trigger system balances functionality with performance through cached state and efficient matching logic.
