The OpTest data processing pipeline is responsible for orchestrating the collection and normalization of ML kernel operation test results. It specifically targets tests that validate low-level operations (ops) across different hardware backends and frontends, extracting rich metadata such as tensor descriptions and failure stages.

## Overview and Orchestration

The orchestration logic resides in `optests.py`, which identifies relevant test reports within a workflow's output directory and selects the appropriate parser based on file extensions, job naming conventions, and branch gating rules [.github/actions/collect_data/src/optests.py 1-13](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L1-L13)

### OpTest Processing Flow

The primary entry point is `create_optest_reports`, which performs the following steps:

1.   **Job Mapping**: Builds a lookup table from `github_job_id` to `name` using the provided `Pipeline` model [.github/actions/collect_data/src/optests.py 54-55](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L54-L55)
2.   **Report Discovery**: Searches for artifacts with `.tar` or `.xml` extensions using `get_github_job_id_to_test_reports`[.github/actions/collect_data/src/optests.py 61-63](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L61-L63)
3.   **Parser Selection**: Iterates through discovered reports and applies selection logic to instantiate either `BuilderPytestParser` or `TTTorchModelTestsParser`[.github/actions/collect_data/src/optests.py 75-84](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L75-L84)
4.   **Parsing**: Executes the `parser.parse()` method and aggregates the resulting list of `OpTest` Pydantic models [.github/actions/collect_data/src/optests.py 86-90](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L86-L90)

### Parser Selection Logic

The pipeline distinguishes between general model tests and specialized "builder" op tests.

*   **BuilderPytestParser**: Selected if the report is an `.xml` file AND the job name contains "builder" (or the filename contains "_builder") AND the current branch is `main`[.github/actions/collect_data/src/optests.py 14-39](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L14-L39) This branch gating ensures that high-fidelity op telemetry is only collected from the primary integration branch.
*   **TTTorchModelTestsParser**: Selected if the report has a `.tar` extension [.github/actions/collect_data/src/optests.py 42-49](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/optests.py#L42-L49)

## BuilderPytestParser Implementation

The `BuilderPytestParser` is designed to ingest JUnit XML reports generated by pytest that include extended hardware and operation metadata in `<properties>` tags [.github/actions/collect_data/src/parsers/builder_pytest_parser.py 25-27](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L25-L27)

### Hardware and Global Metadata

The parser extracts global run information from the `<testsuite>` properties:

*   **Card Type**: The `card` property (e.g., "n150", "n300") [.github/actions/collect_data/src/parsers/builder_pytest_parser.py 85-94](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L85-L94)
*   **Git SHA**: The `git_sha` property representing the specific commit tested [.github/actions/collect_data/src/parsers/builder_pytest_parser.py 97-106](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L97-L106)

### Operation Metadata Extraction

For every `<testcase>`, the parser extracts specific operation attributes from nested `<property>` tags:

*   **Backend/Frontend**: Identifies the software stack used (e.g., `ttnn`, `ttmetal` for backend; `ttir`, `shlo` for frontend) [.github/actions/collect_data/src/parsers/builder_pytest_parser.py 170-174](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L170-L174)
*   **Failure Stage**: Maps the `failure_stage` property to a `TestStatus` enum (e.g., `compile`, `golden`, `runtime`, `success`) [.github/actions/collect_data/src/parsers/builder_pytest_parser.py 17-22](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L17-L22)[.github/actions/collect_data/src/parsers/builder_pytest_parser.py 176-182](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/src/parsers/builder_pytest_parser.py#L176-L182)
*   **Tensor Descriptions**: Extracts `input_shapes` and `input_dtypes` to populate `TensorDesc` models [.github/actions/collect_data/test/data/builder_reports/binoptests.xml 9-12](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/test/data/builder_reports/binoptests.xml#L9-L12)

## Data Models and Types

The OpTest pipeline utilizes specialized fields within the `OpTest` Pydantic model (which inherits from `Test`).

| Field | Description | Source |
| --- | --- | --- |
| `card_type` | Specific hardware target (e.g., n150, n300). | `_get_card_type` |
| `backend` | The TT software backend (ttnn, ttmetal). | `properties.get("backend")` |
| `frontend` | The source frontend (ttir, shlo, tttorch). | `properties.get("frontend")` |
| `failure_stage` | The specific pipeline phase where a failure occurred. | `FAILURE_STAGE_TO_STATUS_ENUM` |
| `input_tensors` | List of `TensorDesc` (shape and dtype). | `properties.get("input_shapes/dtypes")` |

## Testing and Validation

The OpTest processing logic is validated via `test_optests.py`, which uses `MagicMock` to simulate pipeline objects and verifies the report discovery and parser selection logic [.github/actions/collect_data/test/test_optests.py 12-29](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/test/test_optests.py#L12-L29)

Specific parser accuracy is verified in `test_builder_pytest_parser.py`, which runs the `BuilderPytestParser` against static XML fixtures (e.g., `binoptests.xml`) and asserts the correct count and content of parsed tests [.github/actions/collect_data/test/test_builder_pytest_parser.py 14-20](https://github.com/tenstorrent/tt-github-actions/blob/3ecf8e49/.github/actions/collect_data/test/test_builder_pytest_parser.py#L14-L20)
