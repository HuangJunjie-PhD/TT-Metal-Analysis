## Data Flow Architecture

The following diagram shows how data flows from raw inputs through the simulation engine to final outputs:

**Figure 2: Data Flow Pipeline**

### Pipeline Stages

1.   **Preprocessing** ([tt_npe/py/util/npe_analyze_noc_trace_dir.py 333-414](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L333-L414)): Raw trace files are discovered by `group_traces_ttnn()` or `group_traces_metal()`, grouped by operation UID, and for multi-chip workloads, `fabric_post_process.py` function `process_traces()` elaborates high-level fabric send operations into hop-by-hop routing paths.

2.   **Ingestion** ([tt_npe/cpp/src/npeWorkloadIngest.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp)): JSON traces are parsed by function `convertNocTracesToNpeWorkload()`, which extracts zones using `ZoneIterator` for profiling context and constructs `npeWorkload` objects with transfer definitions and dependencies.

3.   **Simulation** ([tt_npe/cpp/src/npeEngine.cpp 202-361](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L202-L361)): Function `npeEngine::runSinglePerfSim()` calls `initTransferState()` to prepare transfers, then iterates through timesteps, calling `computeCurrentTransferRate()` for bandwidth modeling and tracking dependency satisfaction via `genDependencies()`.

4.   **Statistics Collection** ([tt_npe/cpp/src/npeStats.cpp 42-85](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L42-L85)): Metrics are accumulated by `insertTimestep()` at per-timestep granularity for temporal analysis, then aggregated per-device for multi-chip systems via `computeSummaryStats()`.

5.   **Output Generation** ([tt_npe/cpp/src/npeStats.cpp 203-331](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L203-L331)[tt_npe/cpp/src/npeStats.cpp 359-741](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L359-L741)): Results are formatted by `to_string()` for console summaries, `emitSimTimelineToFile()` for timeline JSON files (v1 format), CSV reports via Python orchestration, and `manifest.json` for batch runs.

**Sources:**[tt_npe/py/util/npe_analyze_noc_trace_dir.py 364-448](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L364-L448)[tt_npe/cpp/src/npeWorkloadIngest.cpp 303-600](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp#L303-L600)[tt_npe/cpp/src/npeEngine.cpp 202-360](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L202-L360)[tt_npe/cpp/src/npeStats.cpp 42-85](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeStats.cpp#L42-L85)

## Language Boundary and Integration

tt-npe's core simulation engine is implemented in C++ for performance, while orchestration, file I/O, and batch processing are handled in Python for flexibility. The integration is achieved through `pybind11`:

**Figure 3: Python-C++ Integration Architecture**

### Key Integration Points

*   **Module Import**: Python code imports the compiled `tt_npe_pybind.so` module via `import tt_npe_pybind as npe`, which exposes C++ classes using `PYBIND11_MODULE` macro
*   **Type Conversion**: pybind11 `py::class_<>` declarations automatically convert between Python and C++ types (e.g., `list` ↔ `std::vector`, `dict` ↔ `boost::unordered_flat_map`)
*   **Object Lifecycle**: C++ objects (e.g., `npeConfig`, `npeWorkload`, `npeStats`) are managed by Python's reference counting; pybind11 handles memory correctly across the boundary
*   **API Entry Point**: Function `npe.InitAPI(config)` returns an `npeAPI` handle that orchestrates all subsequent operations via method `runNPE(workload)`

**Sources:**[tt_npe/cpp/pybind/](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/)[tt_npe/py/util/npe_analyze_noc_trace_dir.py 205-230](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/npe_analyze_noc_trace_dir.py#L205-L230)

## Device Architecture Abstraction

The `npeDeviceModel` interface provides a clean abstraction layer that enables architecture-agnostic simulation:

**Figure 4: Device Model Hierarchy**

### Architecture-Specific Behavior

Each device model implementation encapsulates:

*   **Grid Dimensions**: Returned by `getRows()` and `getCols()` - e.g., 12×10 for Wormhole B0, 12×17 for Blackhole
*   **Routing Logic**: Implemented in `route(nocType, src, dst)` - returns `nocRoute` vector of `nocLinkID` representing path
*   **Latency Tables**: Static methods like `get_read_latency()` and `get_write_latency()` return cycle counts based on coordinates
*   **Injection Rates**: Returned by `getSrcInjectionRate(coord)` - varies by core type (DRAM, WORKER, ETH)
*   **Link Topology**: Managed via `getLinkAttributes(linkID)` returning `nocLinkAttr` structs with coordinates and types

The `npeEngine` queries these interface methods during simulation via polymorphism, allowing the same engine code to model different architectures.

**Sources:**[tt_npe/cpp/include/npeDeviceModelIface.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/npeDeviceModelIface.hpp)[tt_npe/cpp/include/device_models/wormhole_b0.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_b0.hpp)[tt_npe/cpp/include/device_models/blackhole.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/blackhole.hpp)[tt_npe/cpp/include/device_models/wormhole_multichip.hpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/include/device_models/wormhole_multichip.hpp)

### Input Formats

| Format | Description | Example Fields | Usage |
| --- | --- | --- | --- |
| **NoC Trace JSON** | Produced by tt-metal profiler; captures all noc_async_* API calls | `type`, `sx`, `sy`, `dx`, `dy`, `num_bytes`, `noc`, `timestamp`, `proc`, `zone` | Profiling real workloads |

## Build System and Dependencies

The build system uses CMake with CPM (CMake Package Manager) to fetch external dependencies:

**Figure 5: Build System Architecture**

### Build Process

1.   **Invocation**: User runs `./build-npe.sh` which invokes CMake with flags like `-DCMAKE_BUILD_TYPE=Release`
2.   **Dependency Resolution**: File `cmake/dependencies.cmake` uses CPM.cmake (`CPMAddPackage()`) to download and build external libraries
3.   **Compilation**: Core C++ library `libtt_npe.so` is compiled from sources in [tt_npe/cpp/src/](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/)
4.   **Python Bindings**: Module `tt_npe_pybind.so` is compiled from [tt_npe/cpp/pybind/](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/pybind/) and linked against `libtt_npe.so`
5.   **Installation**: CMake `install()` commands copy artifacts to `install/lib/`, `install/include/`, and `install/bin/`

## Multi-Chip Support

Multi-chip workloads require additional preprocessing to elaborate high-level fabric transfers into device-specific hops:

**Figure 6: Multi-Chip Processing Pipeline**

### Key Mechanisms

1.   **Topology Parsing** ([tt_npe/py/util/fabric_post_process.py](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py)): Class `TopologyGraph.__init__()` parses `topology.json` to build internal data structures: `device_id_to_fabric_node_id`, `routing_planes` dict, and `eth_chan_to_coord` mappings.

2.   **Path Elaboration** ([tt_npe/py/util/fabric_post_process.py](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py)): For each `fabric_send` operation in the trace, function `process_traces()` calls methods like `find_path_and_destination_unidirectional()` to compute the full path through intermediate devices, then `add_hops()` generates hop-by-hop transfer entries.

3.   **Transfer Groups** ([tt_npe/cpp/src/npeWorkloadIngest.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp)): Multi-hop fabric transfers are parsed from the `fabric_send.path` field and assigned a `transfer_group_id` with sequential `transfer_group_index` values, creating a linked sequence of transfers.

4.   **Dependency Chaining** ([tt_npe/cpp/src/npeEngine.cpp 125-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L125-L153)): Function `npeEngine::genDependencies()` creates `npeCheckpointID` objects between consecutive hops in a transfer group (identified by `transfer_group_parent` field), adding ethernet hop delays via constants `ETH_HOP_CYCLE_DELAY_BASE` and `ETH_HOP_CYCLE_DELAY_PER_BYTE`.

**Sources:**[tt_npe/py/util/fabric_post_process.py](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/py/util/fabric_post_process.py)[tt_npe/cpp/src/npeWorkloadIngest.cpp](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeWorkloadIngest.cpp)[tt_npe/cpp/src/npeEngine.cpp 125-153](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/tt_npe/cpp/src/npeEngine.cpp#L125-L153)
