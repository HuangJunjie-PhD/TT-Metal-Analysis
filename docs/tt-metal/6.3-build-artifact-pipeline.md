## Purpose and Scope

This page documents the build artifact pipeline responsible for producing Docker images, Debian packages, Python wheels, and tarballs for the `tt-metal` repository. The pipeline is primarily defined in `.github/workflows/build-artifact.yaml` and is a critical component of the CI/CD infrastructure, ensuring that consistent, containerized environments and compiled binaries are distributed to hardware test runners and release channels.

## Pipeline Architecture

The build artifact pipeline is an orchestrated set of GitHub Actions workflows that manage the lifecycle from source code to deployable artifacts. It handles platform-specific configurations, multi-stage Docker builds using `docker buildx bake`, and optimized C++ compilation via CMake.

## Platform Parsing and Configuration

The pipeline supports Ubuntu 22.04 and 24.04. The configuration is determined by the `platform` input, which maps to specific toolchains and Python versions.

### Platform Configuration Resolution

Python versions are auto-detected from the Ubuntu version: 3.10 for Ubuntu 22.04 and 3.12 for Ubuntu 24.04 [dockerfile/Dockerfile 108-119](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L108-L119) The default toolchain uses Clang 20 with `libstdc++`[.github/workflows/build-artifact.yaml 36-40](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L36-L40)

## Docker Image Pipeline

The repository uses a multi-stage `Dockerfile` optimized with `docker buildx bake` for tool and Python dependency caching. Images are pushed to GitHub Container Registry (GHCR) or a Harbor mirror.

### Docker Image Stages and Contents

| Stage | Purpose | Key Tools & Dependencies |
| --- | --- | --- |
| `base` | Runtime foundation | `uv`, Python, `cmake`, `zstd`, OpenMPI `v5.0.7`, SFPI [dockerfile/Dockerfile 73-131](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L73-L131) |
| `ci-build` | Compilation env | `ccache`, `doxygen`, `yq`, `mold` linker [dockerfile/Dockerfile 133-162](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L133-L162) |
| `ci-test` | Execution env | `ci-test-venv-layer` (Pytest, NumPy, Torch) [dockerfile/Dockerfile 164-183](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L164-L183) |
| `dev` | Dev sandbox | Venv activation, `git` configuration [dockerfile/Dockerfile 185-194](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L185-L194) |

The `ci-build` stage uses the `mold` linker to significantly reduce link times for the large `tt-metal` and `ttnn` C++ binaries [dockerfile/Dockerfile 154-162](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/dockerfile/Dockerfile#L154-L162)

## Build Artifact Job Execution

The `build-artifact` job performs the compilation of the `Metalium` project [CMakeLists.txt 30-38](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/CMakeLists.txt#L30-L38) It executes within the `ci-build` Docker container to ensure toolchain consistency.

### Key Build Options

The pipeline exposes several CMake variables through workflow inputs:

*   **Tracy**: Enabled via `tracy: true` to include profiling instrumentation [.github/workflows/build-artifact.yaml 17-21](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L17-L21)
*   **Distributed**: Enabled via `distributed: true` to include OpenMPI support for multi-host compute [.github/workflows/build-artifact.yaml 22-26](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L22-L26)
*   **LTO**: Link Time Optimization can be toggled via `enable-lto`[.github/workflows/build-artifact.yaml 67-71](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L67-L71)
*   **Build Type**: Supports `Release`, `Debug`, `ASan`, and `TSan`[.github/workflows/build-artifact.yaml 8-11](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L8-L11)

### Artifact Generation and Naming

The job produces Debian packages using CPack and Python wheels using `pip wheel`. Artifact names are constructed dynamically to include platform and configuration details:

*   `packages-artifact-name`: Includes distro, version, and build type [.github/workflows/build-artifact.yaml 158-164](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L158-L164)
*   `wheel-artifact-name`: Contains the `.whl` file [.github/workflows/build-artifact.yaml 172-178](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/build-artifact.yaml#L172-L178)

## TTSim Integration Tests

The pipeline also supports simulator-based validation using `ttsim.yaml`. This workflow downloads the generated Debian packages and installs them into a clean environment to run examples and unit tests on the TTSim simulator.

### TTSim Validation Flow

1.   **Download Artifacts**: Retrieves `.deb` packages produced by `build-artifact.yaml`[.github/workflows/ttsim.yaml 79-84](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/ttsim.yaml#L79-L84)
2.   **Install**: Installs `tt-metalium`, `tt-metalium-jit`, and `tt-metalium-dev`[.github/workflows/ttsim.yaml 86-91](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/ttsim.yaml#L86-L91)
3.   **Execute**: Runs a suite of examples (e.g., `matmul_multi_core`, `eltwise_sfpu`) against the `libttsim.so` backend [.github/workflows/ttsim.yaml 112-145](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/workflows/ttsim.yaml#L112-L145)
