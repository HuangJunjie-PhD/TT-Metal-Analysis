This page covers the mechanisms for executing programs across multiple Tenstorrent devices organized into a logical mesh. It specifically addresses how `MeshDevice` serves as a multi-device container, how `MeshWorkload` enables heterogeneous multi-program execution, and how the mesh coordinate system allows precise mapping of programs to device subsets.

*   For the `Device` class and single-device APIs, see [Device Abstraction Layer (2.3)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Device%20Abstraction%20Layer%20(2.3))
*   For program compilation and kernel dispatch on a single device, see [Program and Kernel System (2.4)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Program%20and%20Kernel%20System%20(2.4)) and [Fast Dispatch and Command Queue System (2.5)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Fast%20Dispatch%20and%20Command%20Queue%20System%20(2.5))

## Overview

`MeshDevice` is a collection of physical devices arranged in a logical N-dimensional grid (typically 2D). It implements the `IDevice` interface, presenting a unified surface for operations that can target individual devices, ranges of devices, or the entire mesh. [[tt_metal/api/tt-metalium/mesh_device.hpp 75-76](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/api/tt-metalium/mesh_device.hpp#L75-L76)]

`MeshWorkload` extends the single-device `Program` concept to heterogeneous multi-device execution. A single `MeshWorkload` can contain different programs mapped to different regions of the mesh, allowing for flexible workload distribution patterns such as pipeline parallelism or tensor parallelism. [[tt_metal/distributed/mesh_workload.cpp 88-97](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/distributed/mesh_workload.cpp#L88-L97)]

## Command Queues: FD vs SD

The repository supports two execution modes for MeshDevices: **Fast Dispatch (FD)** and **Slow Dispatch (SD)**.

### FDMeshCommandQueue (Fast Dispatch)

This is the high-performance path. It uses a `completion_queue_reader_thread_` to handle readback asynchronously. [[tt_metal/distributed/fd_mesh_command_queue.cpp 167](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/distributed/fd_mesh_command_queue.cpp#L167-L167)] It manages a `RingbufferCacheManager` for efficient command prefetching. [[tt_metal/distributed/fd_mesh_command_queue.cpp 154-155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/[tt_metal/distributed/fd_mesh_command_queue.cpp#L154-L155)]
