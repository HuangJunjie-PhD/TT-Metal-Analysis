## Device Detection and Initialization

### CUDA Platform Detection

CUDA platform detection uses NVML (NVIDIA Management Library) for stateless device queries that don't initialize CUDA contexts.

The `with_nvml_context` decorator ensures NVML is properly initialized and cleaned up for each operation:

`def with_nvml_context(fn: Callable[_P, _R]) -> Callable[_P, _R]:    @wraps(fn)    def wrapper(*args: _P.args, **kwargs: _P.kwargs) -> _R:        pynvml.nvmlInit()        try:            return fn(*args, **kwargs)        finally:            pynvml.nvmlShutdown()    return wrapper`

## Device Capability Management

CUDA devices are identified by their compute capability (SM version), which determines supported features and optimal kernel selection.

| Compute Capability | Architecture | Support Status |
| --- | --- | --- |
| 7.0, 7.2 | Volta | Supported (older CUDA) |
| 7.5 | Turing | Fully Supported |
| 8.0, 8.6, 8.7 | Ampere | Fully Supported |
| 8.9 | Ada Lovelace | Fully Supported |
| 9.0 | Hopper | Fully Supported |
| 10.0, 10.1, 11.0 | Blackwell | Fully Supported (CUDA 12.8+) |

### ROCm GCN Architecture

ROCm devices use GCN architecture names (e.g., gfx908, gfx942) instead of numeric compute capabilities.

Supported ROCm architectures:

*   **MI100/MI200 series**: gfx906, gfx908, gfx90a
*   **MI300 series**: gfx942 (MI300A/X), gfx950 (MI325X)
*   **RDNA2/3**: gfx1030, gfx1100, gfx1101
*   **RDNA4**: gfx1200, gfx1201, gfx1150, gfx1151

## Attention Backend Selection

### CUDA Attention Backends

CUDA platforms support multiple attention backends with automatic selection based on hardware capabilities:

**Backend Support Matrix:**

| Backend | Compute Capability | KV Cache Layout | Notes |
| --- | --- | --- | --- |
| FlashInfer | SM 10.0+ | HND | Default for Blackwell, optimal for SM 10.0 |
| Flash Attention | SM 8.0+ | Standard | Default for Ampere/Hopper |
| Triton Attention | SM 7.5+ | Standard | Fallback, supports FP8 KV on SM 8.9+ |
| FlexAttention | All | Standard | PyTorch native, fallback for older GPUs |
| CUTLASS MLA | SM 10.0 | Special | MLA only, block_size=128 |
| FlashMLA | Custom | Special | MLA only, block_size=64 |

## Summary

vLLM's CUDA and ROCm support provides:

1.   **Unified Platform Abstraction**: Common interface with platform-specific implementations for device detection, capability queries, and hardware management

2.   **Hardware Detection**:

    *   CUDA: NVML-based stateless queries for compute capability and device properties
    *   ROCm: AMDSMI for GCN architecture detection and device management

3.   **Custom Kernel Ecosystem**: Platform-specific extensions compiled conditionally based on hardware capability, including attention, quantization, and activation kernels

4.   **Flexible Attention**: Multiple backend options with automatic selection based on hardware capabilities, compute capability, and model requirements

5.   **FP8 Divergence**: Different FP8 formats (OCP vs FNUZ) with automatic detection and appropriate type selection

6.   **Advanced Quantization**: Extensive quantization support with platform-specific optimizations (Marlin/Machete for CUDA, Triton AWQ for ROCm)

7.   **Distributed Communication**: NCCL/RCCL integration with custom all-reduce support on capable hardware

8.   **Build Flexibility**: CMake-based conditional compilation supporting multiple CUDA versions (7.0 through 12.0) and ROCm architectures (gfx906 through gfx1201)
