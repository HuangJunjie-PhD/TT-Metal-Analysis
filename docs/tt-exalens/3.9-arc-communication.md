## Purpose and Scope

This page documents the Python library API for communicating with the **ARC (Auxiliary RISC Core)** on Tenstorrent devices. The ARC is a special-purpose firmware core that provides device-level services including telemetry, clock management, and system control.

This page covers the high-level communication functions `arc_msg()` and `read_arc_telemetry_entry()`. For information about the ARC hardware block itself and its low-level implementation, see [Hardware Blocks and Components](https://deepwiki.com/tenstorrent/tt-exalens/5.3-memory-maps-and-block-layout). For general device architecture, see [Device Architecture](https://deepwiki.com/tenstorrent/tt-exalens/5-device-architecture).

* * *

## ARC Overview

The ARC (Auxiliary RISC Core) is a dedicated firmware processor present on each Tenstorrent device. It runs firmware that manages device-level operations and exposes interfaces for:

*   **Message-based communication**: Sending command messages to the ARC with arguments and receiving responses
*   **Telemetry**: Reading system metrics like clock frequencies, temperatures, voltages, and other device state

### ARC in Device Hierarchy

**Sources:**[ttexalens/device.py 226-232](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L226-L232)

Each device has exactly one ARC block, accessible via the `arc_block` cached property on the `Device` class. This block provides the interface for all ARC communication operations.

## ARC Messaging API

### Function: `arc_msg()`

The `arc_msg()` function sends a message to the ARC firmware and optionally waits for a response. This is the primary mechanism for commanding the ARC to perform operations.

**Sources:**[ttexalens/tt_exalens_lib.py 417-449](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L417-L449)

#### Function Signature

`def arc_msg(    device_id: int,    msg_code: int,    wait_for_done: bool,    args: list[int],    timeout: datetime.timedelta,    context: Context | None = None,    noc_id: int | None = None,) -> list[int]`
#### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `device_id` | `int` | ID number of the device to send message to |
| `msg_code` | `int` | Message code identifying the operation to perform |
| `wait_for_done` | `bool` | If `True`, blocks until the ARC processes the message |
| `args` | `list[int]` | Arguments for the message (message-specific) |
| `timeout` | `datetime.timedelta` | Maximum time to wait for completion |
| `context` | `Context | None` | TTExaLens context (uses global context if `None`) |
| `noc_id` | `int | None` | NOC ID for communication (uses context default if `None`) |

#### Return Value

Returns `list[int]` containing three elements: `[return_code, reply0, reply1]`

*   `return_code`: Status code from the ARC
*   `reply0`, `reply1`: Message-specific response values

#### Implementation Flow

**Sources:**[ttexalens/tt_exalens_lib.py 417-449](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L417-L449)

#### Validation

The function performs several validation checks:

*   Context initialization via `check_context()`
*   Device ID validation via `validate_device_id()`
*   NOC ID normalization via `check_noc_id()`
*   Timeout must be non-negative

**Sources:**[ttexalens/tt_exalens_lib.py 442-447](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L442-L447)

## ARC Telemetry API

### Function: `read_arc_telemetry_entry()`

The `read_arc_telemetry_entry()` function reads telemetry data from the ARC. Telemetry provides real-time system metrics without requiring message round-trips.

**Sources:**[ttexalens/tt_exalens_lib.py 452-488](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L452-L488)

#### Function Signature

`def read_arc_telemetry_entry(    device_id: int,    telemetry_tag: int | str,    context: Context | None = None) -> int`
#### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `device_id` | `int` | ID number of device to read telemetry from |
| `telemetry_tag` | `int | str` | Telemetry tag name (e.g., `"ARC_MSG_GET_AICLK"`) or numeric ID |
| `context` | `Context | None` | TTExaLens context (uses global context if `None`) |

#### Return Value

Returns `int`: The current value of the requested telemetry entry.

#### Telemetry Tag Resolution

The function accepts telemetry tags as either strings or integers:

**String Tag Resolution:**

`if isinstance(telemetry_tag, str):    telemetry_tag_id = arc.get_telemetry_tag_id(telemetry_tag)    if telemetry_tag_id is None:        raise TTException(f"Telemetry tag {telemetry_tag} does not exist.")`
**Integer Tag Validation:**

`elif isinstance(telemetry_tag, int):    if not arc.has_telemetry_tag_id(telemetry_tag):        raise TTException(f"Telemetry tag ID {telemetry_tag} does not exist.")    telemetry_tag_id = telemetry_tag`
**Sources:**[ttexalens/tt_exalens_lib.py 477-486](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L477-L486)

#### Firmware Version Requirements

Telemetry reading requires firmware version 18.4 or higher:

**Sources:**[ttexalens/tt_exalens_lib.py 472-475](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L472-L475)

The function checks the device firmware version via `device._firmware_version` and raises a `TTException` if the version is below `CUTOFF_FIRMWARE_VERSION` (18.4).

## Device Integration

### Accessing the ARC Block

Each `Device` instance provides access to its ARC block through the `arc_block` cached property:

**Sources:**[ttexalens/device.py 225-232](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L225-L232)

#### Implementation

The `arc_block` property:

1.   Retrieves all blocks of type `"arc"` from the device
2.   Asserts exactly one ARC block exists
3.   Validates the block is an `ArcBlock` instance
4.   Returns the cached `ArcBlock` instance

`@cached_propertydef arc_block(self) -> ArcBlock:    arc_blocks = self.get_blocks(block_type="arc")    assert len(arc_blocks) == 1, "Expected a single ARC block"    assert isinstance(arc_blocks[0], ArcBlock), "Expected a single ARC block"    return arc_blocks[0]`
**Sources:**[ttexalens/device.py 225-232](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L225-L232)

### Architecture-Specific ARC Blocks

Different device architectures provide specialized `ArcBlock` implementations:

| Architecture | ARC Block Class |
| --- | --- |
| Wormhole | `WormholeArcBlock` |
| Blackhole | `BlackholeArcBlock` |
| Quasar | Not yet implemented |

**Sources:**[ttexalens/hw/tensix/wormhole/wormhole.py 58](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hw/tensix/wormhole/wormhole.py#L58-L58)[ttexalens/hw/tensix/blackhole/blackhole.py 59](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hw/tensix/blackhole/blackhole.py#L59-L59)

## Error Handling

### Common Exceptions

| Exception | Condition |
| --- | --- |
| `TTException` | Invalid device ID, negative timeout, unsupported firmware version |
| `ValueError` | Invalid telemetry tag type |
| `AssertionError` | No ARC block found on device (should never occur in valid configurations) |

### Firmware Version Check

**Sources:**[ttexalens/tt_exalens_lib.py 472-475](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L472-L475)

When reading telemetry, the function enforces a minimum firmware version of 18.4 (stored as `CUTOFF_FIRMWARE_VERSION` in `ttexalens.hardware.arc_block`). If the device firmware is older, a descriptive exception is raised with the actual firmware version.

## Communication Architecture

### Full Communication Stack

**Sources:**[ttexalens/tt_exalens_lib.py 417-488](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/tt_exalens_lib.py#L417-L488)[ttexalens/device.py 225-232](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/device.py#L225-L232)

Both `arc_msg()` and `read_arc_telemetry_entry()` ultimately communicate with the ARC hardware through the `server_ifc` (an instance of `TTExaLensCommunicator`), which interfaces with the Tenstorrent User Mode Driver (UMD) to access the hardware.
