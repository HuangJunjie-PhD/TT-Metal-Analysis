This document provides a high-level architectural overview of the Tracy profiling system, covering the client-server model, core data structures, and primary data flow paths. Tracy is a real-time C++ profiler consisting of a client library for instrumenting applications and a server application for collecting, processing, and visualizing profiling data.

For specific instrumentation APIs and integration details, see [Integration Guide](https://deepwiki.com/tenstorrent/tracy/2.2-integration-guide). For profiler GUI functionality and visualization components, see [Profiler Application](https://deepwiki.com/tenstorrent/tracy/4-profiler-application).

## System Architecture

Tracy implements a client-server architecture where instrumented applications act as clients, streaming profiling data to a standalone profiler server application. The system supports both live profiling sessions and offline analysis of saved trace files.

## Core Data Structures

### Event System

Tracy uses a strongly-typed event queue system to capture different types of profiling data:

| Event Type | Purpose | Key Fields |
| --- | --- | --- |
| `ZoneEvent` | Function/scope timing | start time, end time, source location |
| `GpuEvent` | GPU profiling | CPU start/end, GPU start/end, context |
| `LockEvent` | Lock contention tracking | time, thread, lock type |
| `MemEvent` | Memory allocation tracking | pointer, size, allocation/free time |
| `MessageData` | User messages/logs | time, text, thread, callstack |

## Threading Model

### Client Threading

The Tracy client library uses minimal threading to avoid impacting application performance:

*   **Main Application Threads**: Execute instrumented code and enqueue events
*   **Tracy Worker Thread**: Single background thread for network I/O and data compression
*   **Lock-Free Queues**: Enable thread-safe event submission without blocking application threads

### Server Threading

The profiler server employs multiple threads for responsive data processing:

*   **Network Thread**: Handles incoming connections and data reception
*   **Worker Thread**: Processes received events and builds data structures
*   **UI Thread**: Renders the ImGui-based profiler interface
*   **Background Tasks**: Optional threading for statistics computation and file I/O

## Network Protocol

Tracy implements a custom binary protocol over TCP for efficient data transmission:

### Protocol Components

| Component | Purpose | Implementation |
| --- | --- | --- |
| **Handshake** | Version negotiation and capability discovery | `HandshakeStatus` enum |
| **Frame Format** | LZ4-compressed event batches | `TargetFrameSize` = 256KB |
| **Discovery** | UDP broadcast for server discovery | `BroadcastMessage` |
| **Queries** | Server-initiated data requests | `ServerQuery` enum |
