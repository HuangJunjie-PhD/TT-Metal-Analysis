## Purpose and Scope

This document describes the `run-tests.yaml` GitHub Actions workflow, which implements the comprehensive testing pipeline for the `pytorch2.0_ttnn` project. It covers the parallel execution strategy that distributes tests across multiple runners, the artifact collection mechanism that aggregates results, and the coordination between different test categories. For information about specific test implementations and fixtures, see [Testing Infrastructure (conftest.py)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.1-pytest-infrastructure-and-fixtures). For details about the overall CI/CD workflow orchestration, see [GitHub Actions Workflows Overview](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/9.1-github-actions-workflows-overview).

## Workflow Structure and Inputs

The `run-tests.yaml` workflow is a reusable workflow that can be triggered by `workflow_call` or `workflow_dispatch`. It accepts three key inputs:

| Input | Type | Default | Purpose |
| --- | --- | --- | --- |
| `commit_report` | string | `'None'` | Controls which generated files to commit: `'None'`, `'Docs'`, `'Tests'`, or `'All'` |
| `docker_tag` | string | `'ghcr.io/tenstorrent/pytorch2.0_ttnn/ubuntu-22.04-amd64:latest'` | Specifies the Docker container image for test execution |
| `mock_run` | string | `""` | Allows running a subset of test groups for testing purposes |

The workflow outputs two values: `tests_passed` (0 for success, 1 for failure) and optionally `pull_request_number` if documentation updates are committed.

## Test Categories and Execution

### Tools Tests

The `tools-tests` job runs sequentially without parallelization, executing tests in the `tests/tools/` directory. It uses the standard container configuration with access to Tenstorrent hardware devices.

Configuration details:

*   **Container Image:** Uses the `docker_tag` input parameter
*   **Hardware Access:** Mounts `/dev/tenstorrent` device and huge pages
*   **Pytest Options:**`--github-report` flag for CI integration
*   **Report Title:**`"⭐️ Tools Tests"`

**Sources:**[.github/workflows/run-tests.yaml 48-69](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L48-L69)

### Lowering Tests (2 Parallel Groups)

The `lowering-tests` job executes operation lowering tests in 2 parallel groups using a static matrix strategy. Each group runs a subset of the tests in `tests/lowering/`.

The matrix is statically defined as `[1, 2]` in the workflow configuration.

**Sources:**[.github/workflows/run-tests.yaml 70-92](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L70-L92)

### Model Tests (Dynamic Parallel Groups)

The `model-tests` job implements dynamic parallelization based on the number of test files discovered in `tests/models/`. This is the most complex parallel execution in the pipeline.

#### Dynamic Test File Counting

The `count-test-files` job scans the test directory and generates a dynamic matrix:

The action generates a JSON array `[1, 2, 3, ..., N]` where N is the number of test files found.

**Sources:**[.github/workflows/run-tests.yaml 93-109](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L93-L109)

#### Model Test Execution Matrix

The `model-tests` job uses the dynamically generated matrix to parallelize execution:

| Configuration | Value |
| --- | --- |
| **Runners** | `["in-service", "nfs"]` |
| **Container** | Docker with Tenstorrent device access |
| **Volume Mounts** | `/mnt/tt-metal-pytorch-cache/.cache:/root/.cache` for model caching |
| **Matrix Groups** | `fromJson(needs.count-test-files.outputs.matrix)` |
| **Pytest Splits** | `--splits ${{ needs.count-test-files.outputs.num_files }} --group ${{ matrix.group }}` |

Each parallel job:

1.   Checks out the repository with LFS support
2.   Runs `docker system prune -a -f` to free space
3.   Executes `common_model_tests` action with its assigned group
4.   Uploads metrics artifacts as `model-tests-metrics-group-${{ matrix.group }}`

**Sources:**[.github/workflows/run-tests.yaml 110-153](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L110-L153)

### Multi-Device Tests

The `multi-device-tests` job runs on specialized N300 hardware with a mesh device configuration (1x2 grid). It executes tests that require multiple Tenstorrent devices.

Key differences from standard model tests:

*   **Hardware Requirements:** Runs on `tt-ubuntu-2204-n300-stable` or `["n300", "in-service", "nfs"]` runners
*   **Device Configuration:** Access to multi-device setup via `/dev/tenstorrent`
*   **Working Directory:** Uses `/work` as PYTHONPATH and working directory
*   **Report Title:**`"⭐️ Multi-Device Tests"`

**Sources:**[.github/workflows/run-tests.yaml 165-205](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L165-L205)

### Autogen Operation Tests

After model tests complete and generate metrics, the pipeline can optionally generate and run input variation tests for individual operations:

The autogen tests run with the same parallelization strategy as model tests, using pytest splits.

**Sources:**[.github/workflows/run-tests.yaml 206-307](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L206-L307)

## Parallelization Mechanisms

### Matrix Strategy Configuration

The workflow uses GitHub Actions' matrix strategy to parallelize test execution. The key pattern is:

`strategy:  matrix:    group: ${{ fromJson(needs.count-test-files.outputs.matrix) }}`
This converts the JSON array `[1, 2, 3, ..., N]` into N parallel jobs, each with a unique `matrix.group` value.

### Pytest Splits Integration

Each parallel job uses pytest-split to execute only its assigned portion of tests:

`pytest --splits N --group M`
Where:

*   **N** = Total number of parallel groups (from `num_files` output)
*   **M** = Current job's group number (from `matrix.group`)

This ensures each test file is executed exactly once across all parallel jobs.

**Sources:**[.github/workflows/run-tests.yaml 129-145](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L129-L145)

### Hardware Resource Configuration

All test jobs that require Tenstorrent hardware use consistent container options:

`options: >-  --rm   -v /dev/hugepages-1G:/dev/hugepages-1G   --device /dev/tenstorrent  -v ${{ github.workspace }}:${{ github.workspace }}   -w ${{ github.workspace }}`
For model tests with caching requirements:

`-v /mnt/tt-metal-pytorch-cache/.cache:/root/.cache`
This mounts a shared NFS cache for Hugging Face models, PyTorch models, and other downloaded artifacts, preventing redundant downloads across parallel jobs.

**Sources:**[.github/workflows/run-tests.yaml 118-121](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L118-L121)

## Job Dependencies and Execution Flow

The workflow defines a complex dependency graph to ensure proper ordering:

The critical path is:

1.   `tools-tests` → `lowering-tests`
2.   `lowering-tests` → `model-tests` (after `count-test-files`)
3.   `model-tests` → `push-autogen-op-tests` (if enabled)
4.   `push-autogen-op-tests` → `model-autogen-op-tests`
5.   `model-autogen-op-tests` → `collect-metrics`
6.   All test jobs → `tests-passed`

**Sources:**[.github/workflows/run-tests.yaml 47-469](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L47-L469)

## Final Validation and Exit Status

The `tests-passed` job runs conditionally with `if: ${{ always() }}` to ensure it executes even if previous jobs fail. It checks the results of all critical test jobs:

`model_result="${{ needs.model-tests.result}}"multi_device_result="${{ needs.multi-device-tests.result}}"tools_result="${{ needs.tools-tests.result}}"lowering_result="${{ needs.lowering-tests.result}}" if [[ ($tools_result == "success" || $tools_result == "skipped") &&       ($multi_device_result == "success" || $multi_device_result == "skipped") &&       ($lowering_result == "success" || $lowering_result == "skipped") &&       ($model_result == "success" || $model_result == "skipped") ]] ; then  echo "didpass=0" >> $GITHUB_OUTPUT  exit 0else  echo "didpass=1" >> $GITHUB_OUTPUT  exit 1fi`
The job allows both `"success"` and `"skipped"` as passing states, which accommodates conditional job execution. The output `didpass` is set to:

*   **0:** All tests passed
*   **1:** At least one test failed

This output is consumed by calling workflows like `before_merge.yaml` to determine whether to allow the merge.

**Sources:**[.github/workflows/run-tests.yaml 446-469](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L446-L469)

## Environment Configuration and Caching

### Environment Variables

All test jobs share common environment variables:

| Variable | Value | Purpose |
| --- | --- | --- |
| `pytest_verbosity` | `0` (models) or `2` (tools/lowering) | Controls pytest output detail |
| `pytest_report_title` | Job-specific title | Sets GitHub report heading |
| `PIP_CACHE_DIR` | `/root/pip_cache/.pip_cache` | Caches pip packages |
| `TORCH_HOME` | `/root/.cache/torch` | Caches PyTorch models |
| `HF_HOME` | `/root/.cache/huggingface` | Caches Hugging Face models |
| `HF_TOKEN` | `${{ secrets.HUGGINGFACE_TOKEN }}` | Authenticates for gated models |

**Sources:**[.github/workflows/run-tests.yaml 122-128](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L122-L128)

### Shared Cache Volume

Model tests and multi-device tests mount a shared NFS volume for caching:

`-v /mnt/tt-metal-pytorch-cache/.cache:/root/.cache`
This eliminates redundant model downloads across parallel jobs, significantly reducing:

*   Network bandwidth consumption
*   Test execution time
*   Storage requirements per runner

The cache persists across workflow runs, benefiting subsequent executions.

**Sources:**[.github/workflows/run-tests.yaml 121](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L121-L121)[.github/workflows/run-tests.yaml 183](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L183-L183)

## Container Image and Repository Setup

### Docker Container Configuration

All test execution happens inside a Docker container specified by the `docker_tag` input:

`container:  image: ${{ inputs.docker_tag }}  credentials:    username: ${{ github.actor }}    password: ${{ secrets.GH_TOKEN }}`
The default image is built from the `Dockerfile` which includes:

*   Ubuntu 22.04 base from `tt-metalium`
*   Python 3.10
*   System dependencies: `libgl1-mesa-glx`, `git-lfs`, `libsndfile1`, `docker.io`
*   Git LFS initialization

**Sources:**[.github/workflows/run-tests.yaml 55-62](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/workflows/run-tests.yaml#L55-L62)[dockerfile/Dockerfile 1-15](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/dockerfile/Dockerfile#L1-L15)

### Common Repository Setup Action

Each job uses the `common_repo_setup` action which:

1.   Sets up workflow telemetry tracking
2.   Creates pip cache directory if specified
3.   Installs Python 3.10
4.   Upgrades pip and configures PyTorch CPU index
5.   Installs dependencies from `requirements-dev.txt`
6.   Installs `pytest-github-report` for CI integration
7.   Runs cleanup actions

The action ensures consistent environment setup across all parallel jobs.

**Sources:**[.github/actions/common_repo_setup/action.yaml 1-34](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/.github/actions/common_repo_setup/action.yaml#L1-L34)

### Python Requirements

The `requirements.txt` pins specific versions:

*   **PyTorch:**`2.7.1+cpu` (CPU-only to reduce container size)
*   **TorchVision:**`0.22.1+cpu`
*   **ttnn:** Specific wheel from Tenstorrent's PyPI server

The CPU-only PyTorch installation keeps the container size manageable while the actual Tenstorrent hardware acceleration is provided by the ttnn library.
