## Purpose and Scope

This page documents the TTNN tensor abstraction: how tensors are described, created, stored, and managed across host and device memory. It covers the `Tensor` class, `TensorSpec`, `TensorLayout`, `PageConfig`, `MemoryConfig`, and the `Storage` variant types that sit on top of the TT-Metalium `Buffer` layer.

For details on the underlying allocator (`L1BankingAllocator`, DRAM/L1 bank management), see [Memory Management and Allocators (2.7)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Memory%20Management%20and%20Allocators%20(2.7)) For how TTNN Python operations are registered and dispatch to C++ device backends, see [Operation Registration and C++ Backend (4.2)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Operation%20Registration%20and%20C++%20Backend%20(4.2)) For collective tensor operations across devices, see [Collective Communication Operations (4.9)](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/Collective%20Communication%20Operations%20(4.9))

## Tensor Identity: What a Tensor Is

A `Tensor` in TTNN is the combination of:

1.   A **`TensorSpec`** — metadata describing what the tensor contains (shape, data type, memory placement policy) without owning any data [ttnn/api/ttnn/tensor/tensor.hpp 74-80](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L74-L80)
2.   A **`Storage`** variant — the actual location of the tensor's bytes, which may be host memory (`HostStorage`) or device memory (`DeviceStorage`) [ttnn/api/ttnn/tensor/tensor.hpp 54-57](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L54-L57)
3.   A **`TensorTopology`** — describes how the tensor is distributed across a mesh of devices [ttnn/api/ttnn/tensor/tensor.hpp 18-20](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L18-L20)

This separation allows `TensorSpec` to be used independently for pre-allocating tensors, validating shapes, and computing sizes before any data exists. Every `Tensor` is assigned a unique `tensor_id` via a global atomic counter [ttnn/core/tensor/tensor.cpp 46-47](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/tensor.cpp#L46-L47)[ttnn/core/tensor/tensor.cpp 84](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/tensor.cpp#L84-L84)[ttnn/core/tensor/tensor.cpp 160-164](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/tensor.cpp#L160-L164)

## TensorSpec

`TensorSpec` bundles a `Shape` and a `TensorLayout`. It is a lightweight value type used to describe tensor identity without owning data.

**Primary uses:**

*   Pre-allocating a tensor on device via `create_device_tensor(TensorSpec, MeshDevice*)`[ttnn/core/tensor/tensor_ops.cpp 46-85](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/tensor_ops.cpp#L46-L85)

## Storage Types

The `Storage` type is a `std::variant` that can hold `HostStorage` or `DeviceStorage`[ttnn/api/ttnn/tensor/storage.hpp 104-105](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/storage.hpp#L104-L105)

### HostStorage

Manages a `HostTensor`[ttnn/api/ttnn/tensor/tensor.hpp 56](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L56-L56) Data can be provided via `from_span`, `from_vector`, or "borrowed" via `from_borrowed_data`[ttnn/api/ttnn/tensor/tensor.hpp 82-142](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L82-L142) Borrowing uses `MemoryPin` to retain the lifetime of external buffers (e.g., Python numpy arrays or `mmap` regions) [ttnn/api/ttnn/tensor/tensor.hpp 91-110](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L91-L110)

### DeviceStorage

Manages a `MeshTensor` which wraps a `MeshBuffer`[ttnn/api/ttnn/tensor/tensor.hpp 57](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/api/ttnn/tensor/tensor.hpp#L57-L57) It tracks allocation status across a mesh of devices using a `MeshTensorHolder`[ttnn/core/tensor/storage.cpp 62-105](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/storage.cpp#L62-L105) Copying a `Tensor` performs a shallow copy of the `TensorAttributes` (a `shared_ptr`), sharing the underlying storage [ttnn/core/tensor/tensor.cpp 99-109](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/tensor.cpp#L99-L109)

A `DeviceStorage` instance can be the sole owner of device memory if its `mesh_tensor_holder_` use count is 1 and it doesn't have a separate `root_mesh_tensor_holder_`[ttnn/core/tensor/storage.cpp 166-171](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/tensor/storage.cpp#L166-L171)

## Layout Management and Sharding

### Sharding Strategies

TTNN supports advanced sharding where tensors are partitioned across a grid of cores.

*   **`ShardSpec`**: Defines the `CoreRangeSet` (grid), the `shape` of the shard, and the `orientation` (Row/Col Major) [ttnn/cpp/ttnn/operations/data_movement/sharded/interleaved_to_sharded/device/interleaved_to_sharded_program_factory.cpp 81-84](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/data_movement/sharded/interleaved_to_sharded/device/interleaved_to_sharded_program_factory.cpp#L81-L84)
*   **`InterleavedToSharded`**: Operation factory that creates programs to distribute interleaved data into shards. It calculates `num_units_per_shard` based on whether the layout is `TILE` or `ROW_MAJOR`[ttnn/cpp/ttnn/operations/data_movement/sharded/interleaved_to_sharded/device/interleaved_to_sharded_program_factory.cpp 97-140](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/data_movement/sharded/interleaved_to_sharded/device/interleaved_to_sharded_program_factory.cpp#L97-L140)
*   **`ShardedToInterleaved`**: Operation factory that collects sharded data back into an interleaved buffer [ttnn/cpp/ttnn/operations/data_movement/sharded/sharded_to_interleaved/device/sharded_to_interleaved_program_factory.cpp 47-50](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/data_movement/sharded/sharded_to_interleaved/device/sharded_to_interleaved_program_factory.cpp#L47-L50)

### Layout Conversion Logic

The `to_layout` operation manages transitions between `ROW_MAJOR` and `TILE` layouts.

*   **`untilize`**: Converts `TILE` layout to `ROW_MAJOR`. It handles `BFLOAT8_B` to `BFLOAT16` conversion natively [ttnn/cpp/ttnn/operations/core/to_layout/to_layout_op.cpp 42-43](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/core/to_layout/to_layout_op.cpp#L42-L43)
*   **`tilize`**: Converts `ROW_MAJOR` layout to `TILE`. It requires shard dimensions to be divisible by tile size [ttnn/cpp/ttnn/operations/core/to_layout/to_layout_op.cpp 102-105](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/core/to_layout/to_layout_op.cpp#L102-L105)
