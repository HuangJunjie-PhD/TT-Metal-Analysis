## Purpose and Scope

This document details the package configuration defined in `pyproject.toml`, which serves as the single source of truth for TT-SMI's metadata, dependencies, build requirements, and distribution settings. The configuration follows PEP 621 standards and defines how the `tt-smi` package is built, distributed, and installed across different platforms.

For information about the automated build and release workflows that consume this configuration, see [GitHub Release Workflow](https://deepwiki.com/tenstorrent/tt-smi/7.2-github-release-workflow) and [Debian Package Building](https://deepwiki.com/tenstorrent/tt-smi/7.3-debian-package-building). For details on the actual distribution channels, see [Distribution Channels](https://deepwiki.com/tenstorrent/tt-smi/7.4-distribution-channels).

## Dependency Specification

### Runtime Dependencies

The core runtime dependencies are declared at [pyproject.toml 26-36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L36) These are automatically installed when users run `pip install tt-smi`:

**Dependency Versioning Strategy**

The version specifications use two distinct strategies:

| Version Spec | Purpose | Libraries |
| --- | --- | --- |
| `~=X.Y.Z` | Compatible release (allows patch updates only) | `tt_tools_common`, `pyluwen` |
| `>=X.Y.Z` | Minimum version (allows any newer version) | `textual`, `rich`, `pydantic`, `distro`, `elasticsearch`, `pre-commit` |
| No version | Any version accepted | `tomli` |

The strict version pinning (`~=`) for Tenstorrent-specific libraries ensures hardware compatibility and synchronized releases across the Tenstorrent toolchain. For example, `pyluwen~=0.8.0` will accept `0.8.1` or `0.8.2` but reject `0.9.0`.

**Sources:**[pyproject.toml 26-36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L36)

### Critical Dependency Roles

**tt_tools_common (~=1.6.0)**

Provides shared utilities across Tenstorrent tools:

*   `detect_chips_with_callback()` - Device detection and enumeration
*   `WHChipReset.full_lds_reset()` - Wormhole board-level reset
*   `BHChipReset.full_lds_reset()` - Blackhole ASIC-level reset
*   Galaxy reset operations with IPMI integration

Referenced extensively in [tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi_backend.py) for hardware abstraction.

**pyluwen (~=0.8.0)**

Low-level hardware interface library:

*   `pci_scan()` - PCI device enumeration
*   `get_telemetry()` - Real-time telemetry collection
*   Chip detection and identification
*   Firmware version queries

Primary integration point at [tt_smi_backend.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi_backend.py) for all hardware communication.

**textual (>=0.59.0)**

Terminal User Interface framework:

*   `App` - Base application class extended by `TTSMI`
*   `DataTable` - Widget for device information and telemetry display
*   `TabbedContent` - Tab container for three-tab interface
*   Action system for keyboard shortcuts

Core UI implementation in [tt_smi.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py)

**Sources:**[pyproject.toml 26-36](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L26-L36) Diagrams 2 and 6 from context

## Entry Points and Executables

### Console Script Definition

The `tt-smi` executable is registered as a console script at [pyproject.toml 53-54](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L53-L54):

This configuration instructs setuptools to create a `tt-smi` executable that calls the `main()` function in the `tt_smi` module. When users run `tt-smi` from the command line, Python invokes [tt_smi.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi.py#LNaN-LNaN)

**Entry Point Resolution**

When pip installs the package:

1.   Reads `[project.scripts]` section
2.   Creates a wrapper script at `/usr/bin/tt-smi` (or virtual environment equivalent)
3.   The wrapper imports `tt_smi` and calls `main()`
4.   Handles Python interpreter resolution and environment setup

**Sources:**[pyproject.toml 53-54](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L53-L54)[setup.py 20-24](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/setup.py#L20-L24)

## Package Data Management

### Included Package Data

The `[tool.setuptools.package-data]` section at [pyproject.toml 59-65](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L59-L65) specifies non-Python files to include in the distribution:

| Pattern | Purpose | Example Files |
| --- | --- | --- |
| `data/version.txt` | Version information file | Build timestamp, Git hash |
| `*.css` | Textual TUI stylesheets | UI theming and layout |
| `data/*/*.yaml` | Hardware configuration files | Board definitions, limits |
| `data/*/*/*.yaml` | Nested configuration | Platform-specific settings |

The wildcard pattern `"*" = [...]` applies these rules to all packages found in the source tree.

**Sources:**[pyproject.toml 59-65](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L59-L65)

### Excluded Package Data

The `[tool.setuptools.exclude-package-data]` section at [pyproject.toml 67-72](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L67-L72) prevents certain directories from being packaged:

*   `build/` - Build artifacts directory
*   `debian/` - Debian packaging metadata (used by build workflow, not distributed)
*   `images/` - Documentation images (not needed at runtime)

These exclusions reduce package size and prevent distribution of development artifacts.

**Sources:**[pyproject.toml 67-72](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L67-L72)

### Package Discovery

The package discovery configuration at [pyproject.toml 74-75](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L74-L75):

This instructs setuptools to search the current directory (`.`) for Python packages. By default, setuptools finds all packages containing an `__init__.py` file.

**Sources:**[pyproject.toml 74-75](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L74-L75)

## Build System Configuration

### Build Backend Specification

The `[build-system]` section at [pyproject.toml 77-84](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L77-L84) declares PEP 517/518 build requirements:

**Build Backend: setuptools.build_meta**

The `setuptools.build_meta` backend implements PEP 517 standards, providing:

*   Isolated build environments
*   Consistent build interfaces
*   Support for source and wheel distributions

**Minimum Setuptools Version**

The requirement `setuptools>=43.0.0` ensures:

*   PEP 517 build backend support
*   `pyproject.toml` metadata parsing
*   Modern package discovery mechanisms

Version 43.0.0 was released in January 2020 and introduced critical PEP 517 fixes.

**Sources:**[pyproject.toml 77-84](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/pyproject.toml#L77-L84)

### Compatibility with setup.py

For systems with older packaging tools (notably Ubuntu 22.04's `python3-stdeb` and `gbp`), a compatibility `setup.py` exists at [setup.py 1-26](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/setup.py#L1-L26):

This script:

1.   Loads `pyproject.toml` using `tomli` parser
2.   Extracts version information
3.   Provides fallback `setup()` call for legacy tools
4.   Maintains consistency with `pyproject.toml` definitions

The Debian build workflow at [.github/workflows/build-debian.yml](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/.github/workflows/build-debian.yml) relies on this compatibility layer.

**Sources:**[setup.py 1-26](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/setup.py#L1-L26)

## Testing Configuration

### Test Dependencies and Fixtures

The test suite at [tests/test_reset.py](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tests/test_reset.py) uses dependencies defined in `optional-dependencies.test`:

Tests verify reset operations by:

1.   Detecting initial device count with `pci_scan()`
2.   Invoking reset functions from `tt_smi_backend`
3.   Re-scanning devices post-reset
4.   Asserting device count consistency

The stress tests perform `NUM_RESETS_STRESS_TEST = 10` consecutive resets to validate reliability.
