This page describes the Lit (LLVM Integrated Tester) testing framework and FileCheck-based verification used in `tt-mlir` to validate MLIR transformations, dialect conversions, and compiler passes.

## Purpose and Scope

Lit tests provide compilation-level verification of MLIR transformations by checking that intermediate representation (IR) produces expected textual output after passes. Unlike golden tests that validate numerical correctness against reference implementations, Lit tests verify:

*   **Dialect Conversion Correctness**: Operations are correctly lowered between dialects (StableHLO → TTIR → TTNN/TTMetal).
*   **Pass Transformations**: Optimization, canonicalization, and lowering passes produce expected IR patterns.
*   **Type System Validation**: Type inference, encoding attributes, and layout specifications are correct.
*   **Operation Folding**: Constant propagation and algebraic simplifications apply correctly.
*   **Verification Logic**: Operation verifiers correctly accept valid IR and reject invalid constructs.
*   **Hardware Workarounds**: Verification that specific hardware limitations are addressed via IR rewrites.

## Lit Configuration and Environment

The Lit testing environment is configured via `test/lit.cfg.py` and `test/lit.site.cfg.py.in`. These files define how tests are discovered, what tools are available, and how the system environment is mapped into the test runner.

### Feature Discovery and Targets

The configuration dynamically enables features based on the build settings and the hardware environment:

*   **Optional Dialects**: Features like `stablehlo`, `pykernel`, and `d2m-jit` are added if enabled during the build [test/lit.cfg.py 29-40](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L29-L40)
*   **Hardware Architecture**: If a `SYSTEM_DESC_PATH` is provided, `lit` uses `ttrt` to query the hardware architecture (e.g., `n150`, `n300`, `tg`, `llmbox`) and adds it to `available_features`[test/lit.cfg.py 68-86](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L68-L86)[test/lit.cfg.py 92-109](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L92-L109)
*   **Tool Substitutions**: Common MLIR and Tenstorrent tools like `mlir-opt`, `ttmlir-opt`, and `ttmlir-translate` are registered as substitutions for use in `RUN` lines [test/lit.cfg.py 165-169](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L165-L169)
*   **SFPI Support**: Lit checks for the existence of the `riscv32-tt-elf-g++` compiler to enable `sfpi` features [test/lit.cfg.py 65-66](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L65-L66)

### Parallelism Groups

To handle hardware resource constraints, `lit` defines parallelism groups:

*   **OpModel**: When `enable_opmodel` is set, tests that open physical devices are limited to a single thread to avoid resource contention [test/lit.cfg.py 56-59](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L56-L59)
*   **JIT Components**: Both `ttnn-jit` and `d2m-jit` tests are constrained to a single thread to prevent concurrent access issues [test/lit.cfg.py 141-143](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L141-L143)[test/lit.cfg.py 153-154](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L153-L154)

## MLIR FileCheck Testing Framework

### Test Structure and FileCheck Directives

Lit tests use FileCheck directives embedded in MLIR comments to specify expected output patterns. The test infrastructure invokes `ttmlir-opt` with specific passes and verifies output matches `CHECK` patterns.

**Example: Basic Opt Test**[test/ttmlir/ttmlir-opt.mlir 1-5](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/ttmlir/ttmlir-opt.mlir#L1-L5)

`// RUN: ttmlir-opt --show-dialects | FileCheck %s// CHECK: Available Dialects:// CHECK: ttcore// CHECK: ttir`
**FileCheck Directive Types**:

| Directive | Purpose |
| --- | --- |
| `CHECK` | Match pattern in order. |
| `CHECK-LABEL` | Match and reset scan position (used for functions/modules). |
| `CHECK-NOT` | Verify pattern does **not** appear between matches. |
| `CHECK-SAME` | Match on the same line as the previous match. |

**Sources:**[test/ttmlir/ttmlir-opt.mlir 1-5](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/ttmlir/ttmlir-opt.mlir#L1-L5)[test/lit.cfg.py 165-169](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L165-L169)

## Best Practices for MLIR Verification

1.   **Isolate Passes**: When testing a specific transformation, use `ttmlir-opt` with only that pass enabled to ensure the test is focused.
2.   **Target Awareness**: Use `REQUIRES` tags to ensure tests only run when specific environments (like `sfpi` or specific chips) are available [test/lit.cfg.py 65-66](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L65-L66)[test/lit.cfg.py 109](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L109-L109)
3.   **Environment Variables**: Ensure `TT_MLIR_HOME`, `TT_METAL_HOME`, and `TT_METAL_RUNTIME_ROOT` are correctly set, as the Lit configuration validates these during initialization [test/lit.cfg.py 178-196](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L178-L196)
4.   **Parallelism Control**: Use `parallelism_groups` in `lit.local.cfg` files for directories containing tests that access hardware [test/lit.cfg.py 53-59](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/lit.cfg.py#L53-L59)
5.   **Python Dependencies**: Ensure `requirements.txt` (including `lit`, `torch`, and `einops`) are installed for tests involving Python bindings or JIT components [test/python/requirements.txt 1-11](https://github.com/tenstorrent/tt-mlir/blob/c7d92e92/test/python/requirements.txt#L1-L11)
