## Purpose and Scope

This document describes the documentation generation system for tt-system-firmware. The system combines **Sphinx** for user-facing documentation with **Doxygen** for API reference documentation, automatically building and deploying to GitHub Pages. This page covers the documentation toolchain, configuration files, build processes, and CI/CD deployment workflows.

For information about contributing code or writing commit messages, see [Contributing Guidelines](https://deepwiki.com/tenstorrent/tt-system-firmware/9.3-coding-guidelines-and-standards). For instructions on building firmware locally, see [Getting Started](https://deepwiki.com/tenstorrent/tt-system-firmware/9.1-getting-started).

## Doxygen Configuration

### Configuration Overview

Doxygen generates API reference documentation from C/C++ source code:

**Sources:**[doc/Doxyfile 1-318](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/doc/Doxyfile#L1-L318)

### Key Doxygen Settings

| Setting | Value | Purpose |
| --- | --- | --- |
| `PROJECT_NAME` | `"TT-System-Firmware APIs"` | API reference title |
| `PROJECT_LOGO` | `images/tt_logo_white.png` | Logo for API docs |

| `INPUT` | `_doxygen/main.md`, `../include`, `../lib` | Source paths to document |
| `FILE_PATTERNS` | `*.c`, `*.h`, `*.md` | File types to process |
| `RECURSIVE` | `YES` | Recursively scan directories |
| `EXTRACT_ALL` | `YES` | Extract documentation from all entities |
| `HIDE_UNDOC_MEMBERS` | `YES` | Hide members without documentation |
| `GENERATE_XML` | `YES` | Generate XML for further processing |
| `OPTIMIZE_OUTPUT_FOR_C` | `YES` | Optimize for C code |

Doxygen integrates with upstream Zephyr documentation via tag files:

```

```

The build workflow downloads `zephyr.tag` from the upstream Zephyr documentation site, enabling cross-references between tt-system-firmware APIs and Zephyr APIs.

## CI/CD Pipeline

### Documentation Build Workflow

The documentation is automatically built and deployed via GitHub Actions:

**Sources:**[.github/workflows/docs.yml 1-271](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/docs.yml#L1-L271)

### CI Job Breakdown

#### 1. gen-config Job

Extracts board list from `ci_boards.json`:

**Sources:**[.github/workflows/docs.yml 22-31](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/docs.yml#L22-L31)

#### 2. ci-stats Jobs

Runs in parallel for each board (p100a, p150a, p300a):

*   Downloads past CI artifacts named `'Stress test results (${{ matrix.board }})'`
*   Parses `recording.csv` files from artifacts
*   Aggregates failure statistics by test name and workflow run
*   Generates HTML report using `render_ci_table.py`
*   Uploads CSV summaries and HTML report as artifacts

**Sources:**[.github/workflows/docs.yml 33-167](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/docs.yml#L33-L167)

#### 3. build Job

Main documentation build job:

**Step 1: Checkout and Dependencies**

**Step 2: Clone Zephyr**

**Step 3: Build Documentation**

**Step 4: Copy CI Stats**

**Sources:**[.github/workflows/docs.yml 168-250](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/docs.yml#L168-L250)

#### 4. deploy Job

Deploys to GitHub Pages (only on push to main):

*   Requires `pages: write` and `id-token: write` permissions
*   Uses `github-pages` environment
*   Uploads artifact via `actions/deploy-pages@v4`

**Sources:**[.github/workflows/docs.yml 257-271](https://github.com/tenstorrent/tt-system-firmware/blob/ab59b09c/.github/workflows/docs.yml#L257-L271)

## Summary

The tt-system-firmware documentation system is a production-grade dual-documentation setup that:

1.   **Combines Sphinx and Doxygen** to provide user guides and API reference in a unified deployment
2.   **Automates builds via GitHub Actions** with dependency caching and multi-job parallelization
3.   **Integrates CI statistics** into the deployed documentation for visibility
4.   **Supports cross-references** to upstream Zephyr documentation
5.   **Applies custom branding** via CSS and JavaScript for consistent Tenstorrent visual identity
6.   **Deploys to GitHub Pages** with environment protection and versioned artifacts
