## Purpose and Scope

This document explains the three-tier architecture that powers Forge's operation system: the **Python API layer** that provides user-facing functions, the **ForgeOp wrapper layer** that constructs operation instances, and the **C++ backend layer** that implements execution logic. This architecture enables efficient operation dispatch while maintaining a clean Python interface.

For details on implementing specific operations, see [Operation Implementation Guide](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.2-operation-implementation-guide). For a catalog of available operations organized by category, see [Operation Categories and APIs](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.3-operation-categories-and-apis).

### Attribute Handling

Operation attributes are passed as keyword arguments and stored in an `Attrs` map. The C++ `Op` class uses a variant type to support multiple attribute types:

| Attribute Type | C++ Type | Example Usage |
| --- | --- | --- |
| Integer | `int` | `dim=0`, `stride=1` |
| Float | `float` | `alpha=0.01`, `epsilon=1e-5` |
| Boolean | `bool` | `keepdim=True`, `channel_last=False` |
| String | `std::string` | `mode="constant"`, `approximate="tanh"` |
| Integer Vector | `std::vector<int>` | `shape=[1, 32, 224, 224]` |
| Tuple Vectors | `std::vector<std::tuple<...>>` | Complex padding specifications |

## Operation Lifecycle and Data Flow

The following diagram illustrates the complete data flow from a Python API call through operation construction, graph integration, and eventual C++ execution.

**Sources:**[forge/forge/op/eltwise_unary.py 46-48](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/forge/op/eltwise_unary.py#L46-L48)[forge/csrc/ops/op.cpp 243-332](https://github.com/tenstorrent/tt-forge-onnx/blob/e519819c/forge/csrc/ops/op.cpp#L243-L332)

### Key Data Flow Points

1.   **API Call**: User invokes Python function with operation name and operands
2.   **ForgeOp Construction**: Constructor creates C++ `Op` instance via pybind11
3.   **Tensor Creation**: `get_tensor()` returns a Forge `Tensor` object that references the operation
4.   **Graph Integration**: Tensor is added to the computation graph as an `OpNode`
5.   **Execution Dispatch**: During execution, `Op::eval()` dispatches to operation-specific implementation
6.   **Result Computation**: Implementation computes result using PyTorch (at::Tensor)

## Summary

The operation architecture implements a clean three-tier design:

1.   **Python API Layer** (`forge.op.*`): User-facing functions organized by category
2.   **ForgeOp Construction** (`op()` constructor): Bridges Python and C++, creates `Op` instances
3.   **C++ Backend** (`tt::ops::Op`): Implements dispatch and execution logic

This architecture provides:

*   **Clean separation**: Each tier has a distinct responsibility
*   **Type safety**: C++ OpType enum prevents invalid operations
*   **Efficient dispatch**: Switch-based dispatch in C++ for performance
*   **Extensibility**: Adding new operations requires changes at all three tiers
*   **Framework integration**: TVM mappings enable multi-framework support

For implementation details of specific operations, see [Operation Implementation Guide](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.2-operation-implementation-guide). For a complete catalog of available operations, see [Operation Categories and APIs](https://deepwiki.com/tenstorrent/tt-forge-onnx/5.3-operation-categories-and-apis).
