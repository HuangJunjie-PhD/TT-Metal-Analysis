## GitHub Actions Workflows

The tt-npe repository uses GitHub Actions for automated workflow management. Currently, the primary workflow handles community engagement and issue labeling.

### Community Issue and PR Labeling

The repository implements an automated labeling workflow that runs when issues or pull requests are opened by community members.

**Workflow Trigger Configuration:**

*   Triggers on issue creation: `on.issues.types: [opened]`
*   Triggers on PR creation: `on.pull_request.types: [opened]`
*   Delegates to centralized workflow: `tenstorrent/tt-github-actions/.github/workflows/on-community-issue.yml@main`

This workflow automatically tags contributions from external contributors to help maintainers identify and prioritize community engagement.

## Issue Templates

The repository provides structured issue templates to help contributors provide necessary information when reporting bugs, requesting features, or asking for help. All templates are located in `.github/workflows/ISSUE_TEMPLATE/` and use GitHub's YAML-based form syntax.

### Available Issue Types

| Template | Label | Purpose |
| --- | --- | --- |
| `bug_report.yml` | `bug` | Report defects or unexpected behavior |
| `feature_request.yml` | `feature` | Propose new functionality or enhancements |
| `documentation_request.yml` | `documentation` | Request missing or improved documentation |
| `how_to.yml` | `how to` | Ask for help with usage or implementation |

### Bug Report Template

The bug report template collects structured information about defects.

**Required Fields:**

*   **Bug Description** (`describe-bug`): Clear description of the issue and reproduction steps
*   **Reproduction Steps** (`reproduce-bug`): Step-by-step commands or actions that trigger the bug
*   **Expected Behavior** (`behavior`): What should happen instead
*   **Environment Information** (`environment-info`): OS version, Python version, framework version, software version

**Workflow: Bug Report Submission Flow**

Sources: [.github/workflows/ISSUE_TEMPLATE/bug_report.yml 1-44](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/.github/workflows/ISSUE_TEMPLATE/bug_report.yml#L1-L44)

### Feature Request Template

The feature request template structures proposals for new functionality.

**Form Structure:**

*   **Problem Description** (`feature-description`): Context for why the feature is needed
*   **Proposed Solution** (`feature-solution`): Description of desired functionality
*   **Alternatives** (`alternative-solutions`): Other approaches considered
*   **Additional Context** (`additional-context`): Supporting information or screenshots

The template encourages contributors to think through the problem space, proposed solution, and alternatives before submission.

Sources: [.github/workflows/ISSUE_TEMPLATE/feature_request.yml 1-36](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/.github/workflows/ISSUE_TEMPLATE/feature_request.yml#L1-L36)

### Documentation Request Template

For documentation gaps or improvements:

**Key Fields:**

*   **Documentation Issue** (`doc-issue`): What documentation is missing or unclear
*   **Desired Solution** (`describe-doc`): Specific content or presentation suggestions
*   **Visual Aids** (`describe-diagrams`): Requested diagrams or screenshots

Sources: [.github/workflows/ISSUE_TEMPLATE/documentation_request.yml 1-29](https://github.com/tenstorrent/tt-npe/blob/d1c8b91a/.github/workflows/ISSUE_TEMPLATE/documentation_request.yml#L1-L29)

### How-To/Help Template

For usage questions:

**Fields:**

*   **Help Topic** (`help-topic`): Specific question or problem
*   **Attempted Solutions** (`already`): What has been tried already
*   **Goal** (`goal`): Desired outcome

### Pull Request Process

When submitting pull requests:

1.   **Fork and Branch**: Create a feature branch from `main`
2.   **Local Development**: Build and test changes locally (see [Build System](https://deepwiki.com/tenstorrent/tt-npe/8.1-build-system) and [Testing Framework](https://deepwiki.com/tenstorrent/tt-npe/8.2-testing-framework))
3.   **Commit Messages**: Use clear, descriptive commit messages
4.   **Testing**: Ensure all tests pass before submission
5.   **Documentation**: Update relevant documentation for API changes
6.   **Submit PR**: Open pull request with clear description of changes
7.   **Community Label**: External contributors will be automatically labeled for maintainer review

## Best Practices for Contributors

### Code Quality Standards

When contributing code changes:

**Build Verification:**

*   Run `./build-npe.sh` to verify compilation
*   Ensure no compiler warnings are introduced
*   Test both Debug and Release builds if modifying performance-critical code

**Testing Requirements:**

*   Add unit tests for new C++ functionality in `tt_npe/cpp/test/`
*   Add Python tests for new API features in `tt_npe/py/test/`
*   Verify existing tests still pass: `pytest tt_npe/py/test/` and `./build/tt_npe_ut`
*   See [Testing Framework](https://deepwiki.com/tenstorrent/tt-npe/8.2-testing-framework) for detailed testing guidelines

**Documentation Updates:**

*   Update docstrings for modified Python APIs
*   Add examples for new features in relevant wiki pages
*   Update `README.md` if adding new user-facing functionality

### Issue Reporting Best Practices

**For Bug Reports:**

*   Include minimal reproducible example
*   Provide complete environment information (OS, Python version, build configuration)
*   Attach relevant log output or error messages
*   Specify expected vs. actual behavior

**For Feature Requests:**

*   Explain the use case and problem being solved
*   Consider performance implications for the simulation engine
*   Suggest API design if proposing new interfaces

**For Documentation Requests:**

*   Link to specific sections that need improvement
*   Provide examples of what information is missing or unclear

## CI/CD Infrastructure Summary

The current CI/CD infrastructure focuses on community engagement automation. The repository uses:

**Automation Components:**

*   `community-issue-tagging.yml`: Automatic labeling of community contributions
*   Centralized workflow delegation to `tenstorrent/tt-github-actions`

**Issue Management:**

*   Four structured issue templates for different contribution types
*   Automatic label assignment based on template selection
*   Required fields enforce complete information gathering

**Future CI/CD Enhancements:**

While not currently implemented, typical enhancements for a project of this type would include:

*   Automated build verification on pull requests
*   Test suite execution in CI pipeline
*   Static analysis and linting checks
*   Performance regression testing for simulation engine
*   Documentation build verification

## Repository Structure for Contributors

Understanding the repository layout helps contributors know where to make changes:

| Directory | Purpose | Relevant For |
| --- | --- | --- |
| `tt_npe/cpp/src/` | C++ implementation | Core simulation engine changes |
| `tt_npe/cpp/include/` | Public C++ headers | API modifications |
| `tt_npe/cpp/pybind/` | Python bindings | Python API changes |
| `tt_npe/py/` | Python tools and scripts | CLI and analysis tools |
| `tt_npe/cpp/test/` | C++ unit tests | Testing new C++ features |
| `tt_npe/py/test/` | Python tests | Testing Python API |
| `.github/workflows/` | CI/CD workflows | Automation changes |

See [System Architecture](https://deepwiki.com/tenstorrent/tt-npe/1.3-system-architecture) for detailed component relationships.

2.   **Search Existing Issues**: Look for similar problems or questions
3.   **File an Issue**: Use the appropriate issue template (`how_to.yml` for usage questions)
4.   **Provide Context**: Include relevant environment details and what you've tried
