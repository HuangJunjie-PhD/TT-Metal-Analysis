## Purpose and Scope

This document covers the reset configuration system and logging infrastructure in tt-tools-common. It explains how reset operations are specified (via command-line arguments, device IDs, or JSON configuration files), how reset configurations are parsed and validated, and how reset logs are generated in an Elasticsearch-compatible format for centralized monitoring.

For the actual reset execution logic for specific chip architectures, see [Blackhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.3-blackhole-reset-implementation), [Wormhole Reset Implementation](https://deepwiki.com/tenstorrent/tt-tools-common/2.4-wormhole-reset-implementation), and [Grayskull Tensix Reset](https://deepwiki.com/tenstorrent/tt-tools-common/2.5-grayskull-tensix-reset). For the overall reset architecture and design patterns, see [Reset Architecture and Design Patterns](https://deepwiki.com/tenstorrent/tt-tools-common/2.1-reset-architecture-and-design-patterns).

## Reset Input Types and Parsing

The reset system accepts three distinct input types for specifying which devices to reset, represented by the `ResetType` enumeration:

| Reset Type | Description | Example Input |
| --- | --- | --- |
| `ALL` | Reset all detected devices | `"all"` or no arguments |
| `ID_LIST` | Reset specific devices by PCI index | `"0,1,2"` or `0 1 2` |
| `CONFIG_JSON` | Reset according to JSON configuration file | `"path/to/reset_config.json"` |

The `ResetInput` dataclass encapsulates the parsed input:

`@dataclassclass ResetInput:    type: ResetType    value: Union[Dict, int, None]`
**Sources:**[tt_tools_common/reset_common/reset_utils.py 23-31](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L23-L31)

### Input Parsing Flow

**Sources:**[tt_tools_common/reset_common/reset_utils.py 33-87](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L33-L87)

### Parsing Logic Details

The `parse_reset_input` function implements a cascading parsing strategy:

1.   **Empty/All Check**: If input is empty or `["all"]`, return `ResetType.ALL` with no value
2.   **Single Argument Path**: 
    *   First, attempt to open as a file and parse JSON
    *   If file not found, try parsing as comma-separated integers (e.g., `"0,1,2"`)
    *   Return error if both fail

3.   **Multiple Arguments Path**: Parse as space-separated integers (e.g., `["0", "1", "2"]`)

The function deduplicates and sorts integer lists using `list(sorted(set(list_input)))` at [tt_tools_common/reset_common/reset_utils.py 66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L66-L66) and [tt_tools_common/reset_common/reset_utils.py 79](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L79-L79)

**Sources:**[tt_tools_common/reset_common/reset_utils.py 33-87](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L33-L87)

## Reset Configuration File Format

Reset configuration files use JSON format with a structured schema defined by the `HostResetLog` Pydantic model. These files specify which devices to reset, the reset methods to use, and logging preferences.

### Configuration File Structure

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 149-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L149-L185)

### Configuration Fields

| Field | Type | Purpose | Optional |
| --- | --- | --- | --- |
| `time` | `datetime` | Timestamp of configuration generation | Yes |
| `host_name` | `str` | Hostname where reset is executed | Yes |
| `gs_tensix_reset` | `PciResetDeviceInfo` | Grayskull devices to reset | Yes |
| `wh_link_reset` | `PciResetDeviceInfo` | Wormhole devices to reset via link reset | Yes |
| `re_init_devices` | `bool` | Whether to reinitialize devices after reset | Yes |
| `wh_mobo_reset` | `List[MoboReset]` | Galaxy motherboard reset configurations | Yes |
| `disable_serial_report` | `bool` | Flag to disable serial number reporting | Yes |
| `disable_sw_version_report` | `bool` | Flag to disable software version reporting | Yes |

All fields are marked optional via the `@optional` decorator at [tt_tools_common/reset_common/host_reset_log.py 167](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L167-L167) allowing partial configurations.

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 167-179](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L167-L179)[CHANGELOG.md 18-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L18-L21)

A generated configuration file contains:

`{    "time": "2024-03-15T10:30:00",    "host_name": "tt-host-01",    "gs_tensix_reset": {        "pci_index": []    },    "wh_link_reset": {        "pci_index": [0, 1]    },    "re_init_devices": true,    "disable_serial_report": false,    "disable_sw_version_report": false,    "wh_mobo_reset": [        {            "nb_host_pci_idx": [0, 1],            "mobo": "<MOBO NAME>",            "credo": ["<group id>:<credo id>", "<group id>:<credo id>"],            "disabled_ports": ["<group id>:<credo id>", "<group id>:<credo id>"]        }    ]}`

## Configuration File Generation

The `generate_reset_logs` function creates a reset configuration file based on detected devices.

### Generation Process

**Sources:**[tt_tools_common/reset_common/reset_utils.py 89-134](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L89-L134)

### Default Configuration Path

The default configuration directory is `~/.config/tenstorrent/` defined by the `LOG_FOLDER` constant at [tt_tools_common/reset_common/reset_utils.py 21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L21-L21) If this directory doesn't exist, it is created via `init_logging(LOG_FOLDER)` at [tt_tools_common/reset_common/reset_utils.py 132](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L132-L132)

### Sample Motherboard Reset Configuration

The generated configuration includes a sample `wh_mobo_reset` entry with placeholder values:

*   `nb_host_pci_idx`: List of Wormhole PCI indices that are nearest-neighbor hosts
*   `mobo`: Motherboard model name (placeholder: `"<MOBO NAME>"`)
*   `credo`: List of Credo retimer identifiers in format `"<group id>:<credo id>"`
*   `disabled_ports`: List of disabled port identifiers in same format

This sample configuration serves as a template for users to fill in actual Galaxy reset topology information.

## Elasticsearch-Compatible Logging

The logging infrastructure uses Pydantic models that generate Elasticsearch-compatible type mappings, enabling structured log ingestion into Elasticsearch clusters for centralized monitoring and analysis.

### ElasticModel Base Class

All logging models inherit from `ElasticModel`, which extends Pydantic's `BaseModel` with Elasticsearch mapping generation capabilities.

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 68-116](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L68-L116)[tt_tools_common/reset_common/host_reset_log.py 127-136](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L127-L136)

### Mapping Generation Process

The `ElasticModel.get_mapping()` class method generates Elasticsearch index mappings:

1.   Iterate over all Pydantic `__fields__` in the model
2.   For each field, call `field_to_mapping(info)` to determine ES type
3.   `field_to_mapping` delegates to `type_to_mapping(type)` for basic types
4.   Nested `ElasticModel` instances recursively generate object properties
5.   Return complete mapping dictionary

This enables automatic index creation with proper field types for efficient querying and aggregation.

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 127-136](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L127-L136)

### Special Elasticsearch Types

The module defines custom type aliases for precise Elasticsearch mapping control:

*   `Long(int)`: Maps to `long` type for 64-bit integers
*   `Keyword(str)`: Maps to `keyword` type for exact matching (not analyzed)
*   `Text(str)`: Maps to `text` type for full-text search
*   `Date(datetime.datetime)`: Maps to `date` type with configurable format
*   `Nested[T]`: Maps to `nested` type for array of objects

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 28-46](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L28-L46)[tt_tools_common/reset_common/host_reset_log.py 146](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L146-L146)

### Pydantic Version Compatibility

The module maintains compatibility with both Pydantic v1 and v2:

`try:    # Try the newer v2 pydantic and use that first    from pydantic.v1 import BaseModel    from pydantic.v1.fields import Fieldexcept:    # Assume we are on v1 and give that a go    from pydantic import BaseModel    from pydantic.fields import Field`
This fallback import strategy allows tt-tools-common to work in environments with either Pydantic major version, addressing compatibility issues noted in [CHANGELOG.md 78](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L78-L78)

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 15-22](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L15-L22)[CHANGELOG.md 74-78](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/CHANGELOG.md?plain=1#L74-L78)

## Pydantic Data Models

### PciResetDeviceInfo Model

Represents devices to be reset, identified by their PCI interface indices.

This model is used for both `gs_tensix_reset` and `wh_link_reset` fields in `HostResetLog`, containing the list of PCI indices for devices of that type.

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 149-154](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L149-L154)

### MoboReset Model

Represents a Galaxy motherboard reset configuration with topology information.

**Field Descriptions:**

*   `nb_host_pci_idx`: PCI indices of nearest-neighbor host devices (nullable)
*   `mobo`: Motherboard model identifier string
*   `credo`: Credo retimer identifiers in format `"<group id>:<credo id>"`
*   `disabled_ports`: Port identifiers to exclude from reset operations

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 156-165](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L156-L165)

### HostResetLog Model

The top-level model encapsulating complete reset configuration and metadata.

The `save_as_json` method serializes the model with proper formatting:

1.   Call Pydantic's `json()` method with `exclude_none=False` to include all fields
2.   Reload the JSON string to a Python dict for proper formatting
3.   Write to file with 4-space indentation for readability

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 180-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L180-L185)

### Optional Fields Decorator

All data models use the `@optional` decorator to make fields optional:

`@optionalclass PciResetDeviceInfo(ElasticModel):    pci_index: List[int]`
This decorator modifies the Pydantic model class to set `required = False` and `default = None` for specified fields, enabling partial configuration files where users only specify relevant fields.

**Sources:**[tt_tools_common/reset_common/host_reset_log.py 47-66](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L47-L66)

## Integration with Reset System

The configuration and logging system integrates with the broader reset infrastructure:

**Sources:**[tt_tools_common/reset_common/reset_utils.py 33-134](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/reset_utils.py#L33-L134)[tt_tools_common/reset_common/host_reset_log.py 180-185](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/tt_tools_common/reset_common/host_reset_log.py#L180-L185)

The configuration system serves dual purposes:

1.   **Input specification**: Define which devices to reset and how
2.   **Output logging**: Record reset operations for monitoring and debugging

This design enables both interactive reset operations (via command-line arguments) and automated reset workflows (via configuration files), while maintaining structured logging for operational visibility.
