## Purpose and Scope

This document describes the architecture of the comprehensive Python-based testing framework used to validate Low-Level Kernel (LLK) operations. The framework orchestrates the complete test lifecycle: from test definition and parameterization, through C++ compilation and hardware execution, to result validation using golden models.

The testing framework enables:

*   **Parameterized test generation** across thousands of format, dimension, and configuration combinations
*   **Two-phase compilation** (producer/consumer) for efficient parallel test execution in CI
*   **Hardware and simulator execution** with timeout handling and assertion detection
*   **Dual golden validation** using both hardware-aware and pure mathematical reference models

For specific details on golden model generation, see [Golden Model System](https://deepwiki.com/tenstorrent/tt-llk/8.4-golden-model-system). For device communication primitives, see [Device Interface and Hardware Execution](https://deepwiki.com/tenstorrent/tt-llk/8.5-device-interface-and-hardware-execution). For CI/CD pipeline orchestration, see [CI/CD Pipeline](https://deepwiki.com/tenstorrent/tt-llk/10-cicd-pipeline).

## Test Execution Modes

The testing framework operates in three distinct modes, controlled by the `TestMode` enumeration defined in [tests/python_tests/helpers/test_config.py 78-82](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L78-L82):

| Mode | Description | Use Case | CLI Flag |
| --- | --- | --- | --- |
| `TestMode.DEFAULT` | Sequential compile and execute | Local development, debugging individual tests | (none) |
| `TestMode.PRODUCE` | Compile test artifacts only, skip golden generation | CI phase 1: parallel compilation across workers | `--compile-producer` |
| `TestMode.CONSUME` | Execute pre-compiled artifacts only | CI phase 2: parallel execution across workers | `--compile-consumer` |

Mode selection logic in `TestConfig.setup_mode()`[tests/python_tests/helpers/test_config.py 343-359](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L343-L359):

*   Sets `golden_generators_module.get_golden_generator = dummy_golden_generator` in PRODUCE mode to skip validation
*   Clears artifact directory with `shutil.rmtree()` when compiling to ensure clean builds
*   Validates that both flags are not set simultaneously

The `PRODUCE`/`CONSUME` split enables efficient CI execution: tests are compiled in parallel by multiple workers, then consumed by separate execution workers. This separation avoids redundant compilation and enables test splitting based on execution time rather than source file organization.

## Test Lifecycle Overview

**Diagram: Complete test lifecycle showing key functions from test definition through validation**

The test lifecycle proceeds through five distinct phases:

1.   **Test Definition**: The `@parametrize` decorator ([tests/python_tests/helpers/param_config.py 54-83](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L54-L83)) generates test variants using `_params_solve_dependencies()` for constraint solving. Test functions call `generate_stimuli()` ([tests/python_tests/helpers/stimuli_generator.py 15-78](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/stimuli_generator.py#L15-L78)) to create input tensors and `get_golden_generator()` ([tests/python_tests/helpers/golden_generators.py 108-112](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L108-L112)) to retrieve the appropriate golden model class.

2.   **Configuration**: `TestConfig.__init__()` ([tests/python_tests/helpers/test_config.py 337-387](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L337-L387)) creates a test configuration, generates a SHA256 variant hash via `generate_variant_hash()` ([tests/python_tests/helpers/test_config.py 501-517](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L501-L517)), and calls `infer_data_formats()` ([tests/python_tests/helpers/test_config.py 659-790](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L659-L790)) to determine intermediate data formats through the `data_formats()` inference model.

3.   **Compilation**: `build_shared_artefacts()` ([tests/python_tests/helpers/test_config.py 560-657](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L560-L657)) compiles shared objects using `ThreadPoolExecutor`, then `build_elfs()` ([tests/python_tests/helpers/test_config.py 868-951](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L868-L951)) compiles per-variant kernels with `FileLock` coordination to prevent race conditions.

4.   **Execution**: The test resets mailboxes ([tests/python_tests/helpers/device.py 276-282](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L276-L282)), loads ELFs via `load_elf()`, serializes runtime parameters with `struct.pack()` ([tests/python_tests/helpers/test_config.py 476-486](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L476-L486)), releases cores from reset, and polls mailboxes with `wait_for_tensix_operations_finished()` ([tests/python_tests/helpers/device.py 229-273](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L229-L273)) using exponential backoff.

5.   **Validation**: Results are read via `read_from_device()` and unpacked with `unpack_res_tiles()` ([tests/python_tests/helpers/unpack.py 14-91](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/unpack.py#L14-L91)). The golden generator's `__call__()` method computes expected results. `calculate_pcc()` ([tests/python_tests/helpers/utils.py 85-147](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L85-L147)) computes Pearson Correlation Coefficient, and `passed_test()` ([tests/python_tests/helpers/utils.py 150-284](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/utils.py#L150-L284)) validates using `torch.isclose()` with format-specific tolerances and requires PCC > 0.99.

4.   Raises `CircularDependencyError` if cycles detected [tests/python_tests/helpers/param_config.py 39-48](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L39-L48)
5.   Raises `UnknownDependenciesError` if dependency references unknown parameter [tests/python_tests/helpers/param_config.py 32-36](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/param_config.py#L32-L36)

**Variant Hash Generation:**

The `generate_variant_hash()` method [tests/python_tests/helpers/test_config.py 576-593](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L576-L593) computes a SHA256 hash from compilation-affecting parameters:

The hash includes:

*   Format configuration (`formats`) - only if `compile_time_formats=True`
*   Template parameters (`templates`)
*   Build flags (`profiler_build`, `coverage_build`, `boot_mode`)
*   Destination accumulation mode (`dest_acc`)
*   L1 accumulation mode (`l1_acc`)
*   Unpack-to-dest flag (`unpack_to_dest`)
*   L1-to-L1 iteration count (`L1_to_L1_iterations`)

The hash enables:

*   **Deterministic artifact paths**: `/tmp/tt-llk-build/{test_name}/{variant_id}/`
*   **Build caching**: Skip compilation if `.build_complete` marker exists [tests/python_tests/helpers/test_config.py 877-887](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L877-L887)
*   **Parallel safety**: `FileLock` prevents concurrent builds of same variant [tests/python_tests/helpers/test_config.py 890-896](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L890-L896)

## Build System Integration

The compilation process splits into shared artifacts (built once per architecture) and per-variant artifacts (built for each test configuration):

**Diagram: Build system showing shared and per-variant artifact generation with lock coordination**

**Build Process:**

1.   **Shared Artifacts Build**[tests/python_tests/helpers/test_config.py 636-737](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L636-L737):

    *   Check for existing `.shared_complete` marker (fast path) at line 657
    *   Acquire `FileLock` on `/tmp/tt-llk-build-shared.lock` or `-shared-profiler.lock`
    *   Compile `tmu-crt0.o` from assembly [tests/helpers/tmu-crt0.S](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/tmu-crt0.S)
    *   Compile `brisc.o` from [tests/helpers/src/brisc.cpp](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/src/brisc.cpp) (if not Quasar)
    *   Compile `main_*.o` for each TRISC in parallel using `ThreadPoolExecutor` (max_workers=3) at line 716
    *   Link `brisc.elf` using linker scripts from [tests/helpers/ld/](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/)
    *   Write `.shared_complete` marker at line 732

2.   **Per-Variant Build**[tests/python_tests/helpers/test_config.py 872-1079](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L872-L1079):

    *   Generate `build.h` via `generate_build_header()`[tests/python_tests/helpers/test_config.py 824-871](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L824-L871)
    *   Check for existing `.build_complete` marker (fast path) at line 877
    *   Acquire `FileLock` on `/tmp/tt-llk-build/sync_primitives/{variant_id}.lock`
    *   Compile `kernel_*.o` for each TRISC with flags like `-DCOMPILE_FOR_TRISC=0/1/2`
    *   Link three ELFs: `unpack.elf`, `math.elf`, `pack.elf`
    *   Extract profiler metadata if enabled from `.profiler_meta` section via `objcopy`
    *   Write `.build_complete` marker

**Profiler Build Separation:**

Profiler builds require separate shared artifacts because `trisc.cpp` compiles differently with `-DLLK_PROFILER`. The framework maintains two sets of shared directories:

*   `TestConfig.SHARED_DIR` = `/tmp/tt-llk-build/shared/` for normal builds [line 263](https://github.com/tenstorrent/tt-llk/blob/366f58e2/line%20263)
*   `TestConfig.PROFILER_SHARED_DIR` = `/tmp/tt-llk-build/shared-profiler/` for profiler builds [line 267](https://github.com/tenstorrent/tt-llk/blob/366f58e2/line%20267)

Directory selection logic at [tests/python_tests/helpers/test_config.py 641-652](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L641-L652) uses `ProfilerBuild` enum.

**Memory Layout:**

Different linker scripts from [tests/helpers/ld/](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/) are used based on build configuration:

*   Normal build: `memory.{arch}.ld`
*   Coverage build: `memory.{arch}.debug.ld` (larger sections for instrumentation at lines 12-51)

Example coverage layout for Wormhole [tests/helpers/ld/memory.wormhole.debug.ld](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld):

*   `TRISC0_CODE: 64K`, `TRISC0_GCOV_MEM: 32K` (vs 8K in normal build)

**Mailbox Protocol:**

Each TRISC kernel writes `KERNEL_COMPLETE = 0xFF`[tests/python_tests/helpers/device.py 57](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L57-L57) to its designated mailbox. The test framework uses `Mailbox` enum values (either `MailboxesPerf` or `MailboxesDebug` from [tests/python_tests/helpers/llk_params.py](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/llk_params.py)) and polls all three:

*   Initial value: `0xA3` (written by `reset_mailboxes()`)
*   Completion value: `0xFF = KERNEL_COMPLETE`
*   Polling logic at [tests/python_tests/helpers/device.py 256-259](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L256-L259): checks if `read_word_from_device() == KERNEL_COMPLETE`

Only when all three mailboxes in the set `{Mailbox.Unpacker, Mailbox.Math, Mailbox.Packer}` contain `0xFF` does the test proceed to validation phase.

**Boot Mode Variants:**

The `BootMode` enum [tests/python_tests/helpers/device.py 60-65](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/device.py#L60-L65) and `CHIP_DEFAULT_BOOT_MODES` mapping [lines 67-71](https://github.com/tenstorrent/tt-llk/blob/366f58e2/lines%2067-71):

*   **BRISC**: BRISC loads and boots TRISC cores (Wormhole, Blackhole default)
*   **TRISC**: TRISC cores boot directly without BRISC (Quasar)
*   **EXALENS**: ttexalens directly injects instructions (debugging mode)
*   **DEFAULT**: Architecture-specific default mode

**Test Splitting Strategy:**

Tests are split across N parallel workers using duration-based balancing from `pytest-split` plugin [tests/requirements.txt 39](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/requirements.txt#L39-L39):

1.   **Duration Collection**: Nightly workflow collects test durations and uploads `.test_durations` artifact
2.   **Duration Download**: `dawidd6/action-download-artifact@v12`[.github/workflows/setup-and-test.yml 92-103](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L92-L103) downloads from main branch
3.   **Balanced Splitting**: `pytest-split` uses `--durations-path=.test_durations --splits N --group i` to distribute tests
4.   **Fallback**: If duration file unavailable, warning logged and lexicographic split used [lines 129-134](https://github.com/tenstorrent/tt-llk/blob/366f58e2/lines%20129-134)

This approach ensures similar wall-clock time per worker even when individual tests have vastly different execution times (e.g., matmul tests vs. SFPU tests).

**Compilation Producer Phase:**

Command from [.github/workflows/setup-and-test.yml 137-142](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L137-L142):

This phase:

*   Runs with 10 parallel processes (`-n 10` via `pytest-xdist`) for faster compilation
*   Generates all ELF artifacts for assigned test subset at `/tmp/tt-llk-build/`
*   Creates `.build_complete` markers via `build_elfs()`
*   Uses `dummy_golden_generator`[tests/python_tests/helpers/test_config.py 84-96](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L84-L96) to skip validation
*   Exits with skip status marked by `SKIP_JUST_FOR_COMPILE_MARKER`[line 163](https://github.com/tenstorrent/tt-llk/blob/366f58e2/line%20163)

**Execution Consumer Phase:**

Command from [.github/workflows/setup-and-test.yml 145-150](https://github.com/tenstorrent/tt-llk/blob/366f58e2/.github/workflows/setup-and-test.yml#L145-L150):

This phase:

*   Runs serially on hardware with fail-fast (`-x`)
*   Loads pre-compiled ELFs from `/tmp/tt-llk-build/{test_name}/{variant_id}/elf/`
*   Executes on device via `TestConfig.run()` and validates results
*   Generates JUnit XML reports via `--junitxml`
*   Merges compile and run reports using `junitparser`[line 152](https://github.com/tenstorrent/tt-llk/blob/366f58e2/line%20152)

**Conditional Execution:**

Jobs only run if relevant files changed:

*   Wormhole tests: `wormhole_any_changed` or `tests_any_changed`
*   Blackhole tests: `blackhole_any_changed` or `tests_any_changed`
*   Quasar tests: `quasar_any_changed` (compile-only, no execution)

## Artifact Management

The framework generates and manages several categories of artifacts during test execution:

| Artifact Type | Location | Purpose | Lifecycle |
| --- | --- | --- | --- |
| **Shared Objects** | `/tmp/tt-llk-build/shared/obj/` | Reusable compilation units | Built once per architecture per session |
| **Shared ELFs** | `/tmp/tt-llk-build/shared/elf/` | BRISC ELF (if applicable) | Built once per architecture per session |
| **Variant Objects** | `/tmp/tt-llk-build/{test_name}/{variant_id}/obj/` | Test-specific kernels | Built once per variant, cached across tests |
| **Variant ELFs** | `/tmp/tt-llk-build/{test_name}/{variant_id}/elf/` | Executable TRISC kernels | Built once per variant, cached across tests |
| **Build Headers** | `/tmp/tt-llk-build/{test_name}/{variant_id}/build.h` | Generated configuration | Generated for each variant |
| **Coverage Streams** | `/tmp/tt-llk-build/{test_name}/{variant_id}/*.stream` | gcov coverage data | Collected after each test run (coverage mode) |
| **Coverage Info** | `/tmp/tt-llk-build/merged_coverage.info` | Merged lcov format | Generated at session end (coverage mode) |
| **Performance Data** | `/tmp/tt-llk-build/temp_perf_data/*.csv` | Timing measurements | Collected per module (profiler mode) |
| **Profiler Metadata** | `/tmp/tt-llk-build/profiler_meta/{test_name}/{variant_id}/` | Profiler marker data | Extracted from ELF `.profiler_meta` section |

**Build Header Generation:**

The `generate_build_header()` method creates a C++ header with compile-time configuration:

This approach enables:

*   **Compile-time optimization**: Formats and buffer addresses as `constexpr`
*   **Type safety**: Enum types prevent invalid values
*   **Zero runtime overhead**: All configuration resolved at compile time

**File Lock Coordination:**

To prevent race conditions in parallel test execution:

*   **Shared artifact lock**: `/tmp/tt-llk-build-shared.lock` (or `-shared-profiler.lock`)
*   **Per-variant lock**: `/tmp/tt-llk-build/sync_primitives/{variant_id}.lock`
*   **Completion markers**: `.shared_complete`, `.build_complete` enable fast-path checks

The lock pattern:

1.   Check for completion marker (fast path, no lock needed)
2.   Acquire lock
3.   Check again (another process may have completed while waiting)
4.   Perform build
5.   Write completion marker
6.   Release lock
