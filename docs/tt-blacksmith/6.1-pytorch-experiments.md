This page covers the shared structure, conventions, and runtime patterns that apply to all experiments under `blacksmith/experiments/torch/`. It explains how the `torch_xla` backend is used to target Tenstorrent hardware via PJRT, and documents the patterns every PyTorch training script follows.

For details on individual model experiments, see the sub-pages: MNIST ([4.1.1](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.1.1)), Llama ([4.1.2](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.1.2)), Qwen ([4.1.3](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.1.3)), Gemma ([4.1.4](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.1.4)), and other models ([4.1.5](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.1.5)). For JAX experiments, see [4.2](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#4.2). For the configuration system shared across all experiments, see [3.3](https://deepwiki.com/tenstorrent/tt-blacksmith/6.1-pytorch-experiments#3.3). ### The `train()` Function

The `train()` function signature is consistent across all scripts:

`def train(    config: TrainingConfig,    device_manager: DeviceManager,    logger: TrainingLogger,    checkpoint_manager: CheckpointManager,)`
Inside `train()`, the sequence is always:

1.   Load model via `get_model(config, device_manager.device)` (LLM scripts) or direct model construction (MNIST/ViT scripts).
2.   Construct `optimizer` (AdamW for LLMs, SGD for MNIST).
3.   Optionally load a checkpoint via `checkpoint_manager.load_checkpoint(model, optimizer)` if `config.resume_from_checkpoint` is set.
4.   Load train and eval data loaders using `get_dataset(config, split=...)`.
5.   Run initial validation before the first training step.
6.   Iterate over epochs and batches, calling `device_manager.prepare_batch()`, `device_manager.shard_model()`, the forward/backward pass, and `device_manager.optimizer_step(optimizer)`.
7.   Log metrics with `logger.log_metrics()` at `steps_freq` intervals.
8.   Run validation at `val_steps_freq` intervals.
9.   Save checkpoints via `checkpoint_manager.save_checkpoint()`.
10.   Wrap the entire body in `try/except/finally`, calling `logger.finish()` in the `finally` block.

### `training_step_inner()` for LLM Scripts

LLM experiments (e.g. Llama) extract the forward and backward pass into a separate `training_step_inner()` function. This is intentional: large vocab-sized tensors such as `logits` are kept scoped locally so they do not persist in the computation graph across steps, which would cause unnecessary and expensive collective communication in multi-chip setups.

`def training_step_inner(batch, model, loss_fn, gradient_accumulation_steps):    output = model(input_ids=batch["input_ids"], attention_mask=batch["attention_mask"])    logits = output.logits    shift_logits = logits[:, :-1, :].contiguous()    loss = loss_fn(shift_logits, batch["expected_output"], batch["labels_mask"])    scaled_loss = loss / gradient_accumulation_steps    scaled_loss.backward()    return loss.detach()`
Simpler models (MNIST, Qwen, ViT) inline the forward/backward directly in the step loop.

Sources: [blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 90-103](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L90-L103)

## The `torch_xla` / PJRT Integration

All PyTorch experiments import `torch_xla` and use it to target Tenstorrent hardware through the PJRT plugin. The `config.use_tt` boolean controls whether TT-specific code paths are activated.

### Device Selection

`DeviceManager` resolves the device from `TrainingConfig`. When `use_tt` is `True`, the device is a `torch_xla` device backed by the PJRT plugin; otherwise it falls back to CPU or GPU. Experiments move models and tensors to `device_manager.device`.

### `torch_xla.sync()`

After the backward pass (and in validation loops), scripts call `torch_xla.sync(wait=True)` when `config.use_tt` is set. This flushes pending XLA operations and synchronizes execution before reading `.item()` values from tensors.

`if config.use_tt:    torch_xla.sync(wait=True)`
Sources: [blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 181-183](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L181-L183)[blacksmith/experiments/torch/qwen/test_qwen_finetuning.py 154-155](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/qwen/test_qwen_finetuning.py#L154-L155)[blacksmith/experiments/torch/vit/test_vit_finetuning.py 45-46](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/vit/test_vit_finetuning.py#L45-L46)

In the Llama XLA experiment, the computation graph cache is explicitly cleared after each optimizer step to prevent memory growth across steps:

`import torch_xla.runtime as xrif config.use_tt:    xr.clear_computation_cache()`

### Compile Options

Scripts that run on TT hardware may set precision and compilation options via `torch_xla.set_custom_compile_options()`. Two patterns are used:

| Pattern | Options | Used by |
| --- | --- | --- |
| High-precision LLM | `{"fp32_dest_acc_en": True, "math_fidelity": "hifi4"}` | Llama XLA |
| Export/debug | `{"export_path": "model", "export_tensors": True, "enable_const_eval": False}` | MNIST parallelism scripts |

*   `fp32_dest_acc_en`: Accumulates partial matrix-multiply results in FP32 to reduce precision loss.
*   `math_fidelity "hifi4"`: Uses all four mantissa phases for full-precision multiplications.

Sources: [blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 263-267](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L263-L267)[blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py 186-192](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/data_parallel/test_mnist_training.py#L186-L192)[blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py 196-202](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/mnist/tensor_parallel/test_mnist_training.py#L196-L202)

## Multi-chip Patterns

Scripts that support multi-chip (data parallel or tensor parallel) invoke two additional calls on the `DeviceManager` inside every step:

| Call | Purpose |
| --- | --- |
| `device_manager.prepare_batch(batch)` | Shards input tensors across devices along the data axis (data parallelism). |
| `device_manager.shard_model(model)` | Applies SPMD sharding annotations to model parameters using `model_sharding_patterns` (tensor parallelism). |
| `device_manager.optimizer_step(optimizer)` | Steps the optimizer; on XLA, wraps the step with `torch_xla.sync` to apply all-reduce gradients. |

Single-chip scripts still call `device_manager.optimizer_step(optimizer)` but skip `prepare_batch` sharding and `shard_model`.

## Gradient Accumulation

The Llama XLA script supports gradient accumulation controlled by `config.gradient_accumulation_steps`. The accumulation cycle resets `optimizer.zero_grad()` at the start of each cycle and only calls `device_manager.optimizer_step()` once `accumulation_step == gradient_accumulation_steps`. Loss is scaled by `1 / gradient_accumulation_steps` inside `training_step_inner`.

Other experiments (Qwen, MNIST, ViT) call `optimizer.zero_grad()` every step, so `gradient_accumulation_steps` is effectively 1.

Sources: [blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 155-196](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L155-L196)[blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py 95-103](https://github.com/tenstorrent/tt-blacksmith/blob/fe25eba0/blacksmith/experiments/torch/llama/xla/test_llama_fine_tuning_pure_torch.py#L95-L103)

## Metrics and Checkpointing

All scripts use the same `TrainingLogger` and `CheckpointManager` API:

| Action | Code | Timing |
| --- | --- | --- |
| Log training loss | `logger.log_metrics({"train/loss": avg_loss}, commit=False, step=global_step)` | every `config.steps_freq` steps |
| Log validation loss | `logger.log_metrics({"val/loss": val_loss}, commit=False, step=global_step)` | every `config.val_steps_freq` steps |
| Commit W&B row | `logger.log_metrics({}, commit=True, step=global_step)` | every optimizer step |
| Save checkpoint | `checkpoint_manager.save_checkpoint(model, global_step, epoch, optimizer)` | based on `should_save_checkpoint()` |
| Log artifact | `logger.log_artifact(final_model_path, artifact_type="model", name=...)` | end of training |
