## Purpose and Scope

This document describes the test suite that validates individual PyTorch operation lowering to TTNN equivalents. These tests focus on isolated operation behavior, ensuring that each `aten` operation correctly transforms to its corresponding `ttnn` operation with accurate numerical results. This is distinct from full model testing covered in [Model Test Coverage](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/5-model-test-coverage), which validates end-to-end model execution.

The operation lowering tests are organized in `tests/lowering/` and complement the automated test generation described in [Automated Test Generation](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.4-automated-test-generation). While autogenerated tests cover operation×input_variation combinations discovered during model execution, the tests documented here provide hand-crafted validation for critical operation patterns, edge cases, and known failure modes.

For information about the actual operation lowering implementation, see [Operation Lowering (ToTtPass)](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.2-operation-lowering-(tottpass)). For the guard system that prevents problematic lowerings, see [Guard System and Operation Blocklists](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/2.4-guard-system-and-operation-blocklists).

## Test Infrastructure and Structure

Operation lowering tests follow a standardized pattern that validates both graph transformation correctness and numerical accuracy. Each test creates a minimal `torch.nn.Module` that exercises a specific operation, compiles it through the `torch_ttnn` backend, and verifies the output.

### Test Module Pattern

Tests use simple module classes that wrap the operation under test:

```
class <Operation>Module(torch.nn.Module):
    def forward(self, *args, **kwargs):
        return <torch_operation>(*args, **kwargs)
```

This pattern isolates the operation from any surrounding computation, making it easier to verify that the specific operation is correctly lowered.

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 12-17](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L12-L17)[tests/lowering/eltwise/binary/test_div.py 11-16](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_div.py#L11-L16)[tests/lowering/eltwise/ternary/test_where.py 10-15](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/ternary/test_where.py#L10-L15)

### Compilation and Validation Flow

**Test Execution Steps:**

1.   **Eager Execution**: Run operation with PyTorch's native implementation to obtain reference output
2.   **Compilation**: Compile module through `torch_ttnn.backend` with device configuration
3.   **Compiled Execution**: Run operation through TTNN backend to obtain device output
4.   **Graph Validation**: Inspect transformed FX graph to verify operation lowering occurred
5.   **Accuracy Validation**: Compare outputs using PCC (Pearson Correlation Coefficient) or `allclose`

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 52-67](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L52-L67)[tests/lowering/eltwise/binary/test_div.py 52-69](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_div.py#L52-L69)

### Graph Validation Techniques

Tests verify that operations are correctly lowered by inspecting the compiled FX graph:

`# Extract operation targets from graph nodesnodes = list(option._out_fx_graphs[0].nodes)targets = [node.target for node in nodes] # Verify expected TTNN operation appearsassert targets.count(ttnn.<operation>) == 1 # Verify aten operation was removedassert torch.ops.aten.<operation> not in targets`
For complex operations that decompose into multiple TTNN operations (e.g., `baddbmm` → `matmul` + `add`), tests verify both the presence of expected operations and their ordering in the graph.

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 63-65](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L63-L65)[tests/lowering/matmul/test_baddbmm.py 44-56](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py#L44-L56)

### Accuracy Assertion Methods

Tests use validation utilities from `tests/utils.py`:

| Method | Purpose | Typical Threshold |
| --- | --- | --- |
| `torch.allclose()` | Absolute/relative tolerance check | Default: rtol=1e-5, atol=1e-8 |
| `assert_with_pcc()` | Pearson Correlation Coefficient | 0.99 - 0.999 |
| `assert_with_ulp()` | Units in Last Place (precision) | Not commonly used in lowering tests |

The PCC threshold varies by operation complexity: simple element-wise operations typically use 0.99, while matrix operations use 0.999.

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 90](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L90-L90)[tests/lowering/matmul/test_baddbmm.py 56](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py#L56-L56)

## Test Organization Hierarchy

**Directory Structure:**

*   **`eltwise/`**: Element-wise operations (broadcasted element-by-element computation) 
    *   `binary/`: Two-operand operations (add, sub, mul, div, pow)
    *   `unary/`: Single-operand operations (activations, transcendentals)
    *   `ternary/`: Three-operand operations (where, addcmul, addcdiv)

*   **`matmul/`**: Matrix multiplication and related operations (matmul, addmm, baddbmm, bmm)
*   **`pooling/`**: Pooling operations (max_pool, avg_pool, adaptive_avg_pool)
*   **`conv/`**: Convolution operations
*   **`attention/`**: Scaled dot-product attention
*   **`creation/`**: Tensor creation operations (arange, masked_fill)
*   **`embedding/`**: Embedding lookup operations
*   **`specialized/`**: Complex patterns and specialized operations

## Common Test Patterns

### Shape Parameterization

Tests use `@pytest.mark.parametrize` to validate operations across diverse input shape combinations:

**Common Shape Patterns:**

1.   **Matched Shapes**: Both operands have identical dimensions - simplest case, no broadcasting
2.   **Unidirectional Broadcasting**: One operand has trailing dimensions of size 1 or is lower rank
3.   **Scalar Broadcasting**: One operand is a scalar (shape `()`)
4.   **Problematic Bidirectional Broadcasting**: Both operands have non-matching dimensions requiring expansion in different positions - often marked with `xfail`

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 36-51](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L36-L51)[tests/lowering/eltwise/binary/test_div.py 20-51](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_div.py#L20-L51)[tests/lowering/eltwise/binary/test_mul.py 19-67](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_mul.py#L19-L67)

### Broadcasting Edge Cases and Known Issues

Many tests include `pytest.mark.xfail` annotations for broadcasting scenarios that are not yet supported:

`pytest.param(    ((64, 1), (1, 64)),    marks=pytest.mark.xfail(reason="broadcasting issues (#64)"))`
**Common Broadcasting Failures:**

| Pattern | Issue | Ticket Reference |
| --- | --- | --- |
| `((64, 1), (1, 64))` | Bidirectional broadcasting | `#64` |
| `((16, 1), (1, 1, 32))` | Rank mismatch with bidirectional | `tt-metal#12852, tt-metal#15808` |
| `((64, 3, 64, 64), (3, 1, 1))` | High-rank with trailing broadcast | `tt-metal#15808` |

These annotations document known limitations and prevent CI failures while fixes are in progress.

**Sources:**[tests/lowering/eltwise/binary/test_sub.py 46-49](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L46-L49)[tests/lowering/eltwise/binary/test_mul.py 26-65](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_mul.py#L26-L65)[tests/lowering/eltwise/ternary/test_where.py 25-28](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/ternary/test_where.py#L25-L28)

### Scalar Operand Handling

Many operations support mixed tensor-scalar operands, requiring separate test coverage:

Tests verify that:

*   Scalar operands are correctly handled without materializing intermediate tensors
*   Graph transformations differ based on operand positions (e.g., scalar numerator in division requires reciprocal+multiply decomposition)

**Example - Division with Scalar Numerator:**

When dividing a scalar by a tensor (`5.0 / tensor`), the operation decomposes into `reciprocal(tensor) * 5.0`:

`targets = [node.target for node in option._out_fx_graphs[0].nodes]assert targets.count(ttnn.reciprocal) == 1assert targets.count(ttnn.mul) == 1`
**Sources:**[tests/lowering/eltwise/binary/test_div.py 93-131](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_div.py#L93-L131)[tests/lowering/eltwise/binary/test_sub.py 126-162](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L126-L162)[tests/lowering/eltwise/binary/test_sub.py 70-91](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py#L70-L91)

### Multi-Argument Operations

Operations like `baddbmm` and `addmm` accept multiple tensor arguments plus scalar coefficients (`alpha`, `beta`):

```
torch.baddbmm(input, batch1, batch2, beta=β, alpha=α)
= β * input + α * (batch1 @ batch2)
```

Tests validate:

*   Correct decomposition into primitive operations (`matmul`, `multiply`, `add`)
*   Operation ordering in the graph
*   Special case handling (e.g., `beta=0` eliminates the addition)
*   Preservation of intermediate tensor metadata

**Validation Example:**

`target = [node.target for node in nodes]# Verify matmul occurs before multiplyassert target.index(ttnn.matmul) < target.index(ttnn.multiply)# Verify multiply result feeds into addassert nodes[target.index(ttnn.add)].args[1].target == ttnn.multiply# Verify intermediate shapes are preservedfor node in nodes:    if node.target == ttnn.matmul:        assert node.meta["val"].size() == expected_shape`
**Sources:**[tests/lowering/matmul/test_baddbmm.py 12-25](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py#L12-L25)[tests/lowering/matmul/test_baddbmm.py 44-56](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py#L44-L56)[tests/lowering/matmul/test_baddbmm.py 145-165](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py#L145-L165)

## Operation Categories Overview

The operation lowering test suite is divided into three main categories, each covered in detail in subsequent sections:

### Element-wise and Pointwise Operations ([Section 6.1](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.1-element-wise-and-pointwise-operation-tests))

Tests for operations that apply element-by-element computation with broadcasting support:

*   **Binary Operations**: `add`, `sub`, `mul`, `div`, `pow`
*   **Unary Operations**: `hardsigmoid`, `log_softmax`, `sigmoid`, `relu`
*   **Ternary Operations**: `where`, `addcmul`, `addcdiv`

These tests focus heavily on broadcasting semantics and scalar handling.

**Sources:**[tests/lowering/eltwise/binary/test_add.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_add.py)[tests/lowering/eltwise/binary/test_sub.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_sub.py)[tests/lowering/eltwise/binary/test_mul.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_mul.py)[tests/lowering/eltwise/binary/test_div.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/binary/test_div.py)[tests/lowering/eltwise/ternary/test_where.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/eltwise/ternary/test_where.py)

### Tensor Manipulation Operations ([Section 6.2](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.2-tensor-manipulation-operation-tests))

Tests for operations that perform complex tensor transformations:

*   **Matrix Operations**: `matmul`, `baddbmm`, `addmm`, `bmm`
*   **Pooling**: `max_pool2d`, `avg_pool2d`, `adaptive_avg_pool2d`
*   **Convolution**: `conv2d` with various configurations
*   **Attention**: `scaled_dot_product_attention`

These tests validate correct decomposition into TTNN primitives and preservation of tensor metadata through multi-operation transformations.

**Sources:**[tests/lowering/matmul/test_baddbmm.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_baddbmm.py)[tests/lowering/matmul/test_addmm.py](https://github.com/tenstorrent/pytorch2.0_ttnn/blob/7ba2867e/tests/lowering/matmul/test_addmm.py)

### Creation and Specialized Operations ([Section 6.3](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/6.3-creation-and-specialized-operation-tests))

Tests for tensor creation and specialized operations:

*   **Tensor Creation**: `arange`, `zeros`, `ones`, `full`
*   **Masking**: `masked_fill`, `masked_scatter`
*   **Embedding**: `embedding` lookup operations
*   **Specialized Patterns**: Complex operation combinations

These tests often validate handling of constant tensors and integration with the guard system.

## Test Execution and CI Integration

Operation lowering tests run as part of the `run-tests.yaml` workflow's lowering test matrix. The `compile_and_run` fixture (described in [pytest Infrastructure and Fixtures](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/4.1-pytest-infrastructure-and-fixtures)) automatically:

1.   Intercepts test execution
2.   Captures operation schemas and input variations
3.   Validates accuracy
4.   Collects metrics for documentation generation

Results feed into the metrics collection system ([Metrics Collection and Schema Tracking](https://deepwiki.com/tenstorrent/pytorch2.0_ttnn/7.1-metrics-collection-and-schema-tracking)), which generates comprehensive operation coverage reports.
