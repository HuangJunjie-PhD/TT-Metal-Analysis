## Purpose and Scope

The RISC-V SFPU Core provides the foundational infrastructure for Special Function Processing Unit (SFPU) support within the RISC-V backend of sfpi-gcc. This system implements the core built-in function framework, type definitions, and target availability mechanisms that enable SFPU instruction generation and optimization.

This page covers the base RISC-V SFPU implementation including built-in function definitions, function type systems, and integration with the GCC compilation pipeline. For information about Tenstorrent-specific SFPU extensions that build upon this foundation, see [Tenstorrent SFPU Extensions](https://deepwiki.com/tenstorrent/sfpi-gcc/2.2-tenstorrent-sfpu-extensions).

## Built-in Function Infrastructure

The RISC-V SFPU Core centers around a comprehensive built-in function system that exposes SFPU operations to C/C++ programs through intrinsic functions. The infrastructure is implemented primarily in the builtin system with support for both direct instruction mapping and more complex expansion patterns.

### Core Built-in Architecture

The built-in system uses a table-driven approach where each SFPU operation is described by a `riscv_builtin_description` structure containing the instruction code, function name, expansion type, prototype, and availability predicate.

Sources: [gcc/config/riscv/riscv-builtins.cc 79-95](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L79-L95)

### Built-in Registration and Initialization

The system registers built-in functions during compiler initialization through `riscv_init_builtins()`, which processes the builtin description table and creates appropriate function declarations:

The initialization process creates the specialized `V64SFmode` vector type for 64-element single-precision floating-point vectors, which serves as the primary data type for SFPU operations.

## Function Type System

The RISC-V SFPU Core implements an extensive function type system that supports the diverse signature requirements of SFPU operations. The type system is defined through a set of function prototype macros and type definitions.

### Core Type Definitions

| Type Symbol | Underlying Type | Purpose |
| --- | --- | --- |
| `RISCV_ATYPE_V64SF` | `v64SF_type_node` | 64-element vector of single-precision floats |
| `RISCV_ATYPE_USI` | `unsigned_intSI_type_node` | 32-bit unsigned integer |
| `RISCV_ATYPE_UHI` | `unsigned_intHI_type_node` | 16-bit unsigned integer |
| `RISCV_ATYPE_UQI` | `unsigned_intQI_type_node` | 8-bit unsigned integer |
| `RISCV_ATYPE_POINTER` | `ptr_type_node` | Generic pointer type |
| `RISCV_ATYPE_VOID` | `void_type_node` | Void return type |

Sources: [gcc/config/riscv/riscv-builtins.cc 143-151](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L143-L151)

### Function Prototype System

The function type system uses a macro-based approach to generate function prototypes with varying numbers of arguments. The system supports functions with 0-8 parameters through `RISCV_FTYPE_ATYPES` macros:

The type system defines over 50 different function signatures to accommodate the variety of SFPU operations, including simple vector operations, memory operations with multiple addressing modes, and complex multi-operand instructions.

Sources: [gcc/config/riscv/riscv-ftypes.def 32-125](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-ftypes.def#L32-L125)[gcc/config/riscv/riscv-builtins.cc 204-225](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L204-L225)

## Target Availability and Predicates

The RISC-V SFPU Core implements a target availability system that controls which built-in functions are exposed based on the target architecture configuration. This system ensures that SFPU instructions are only available when the appropriate hardware support is enabled.

### Availability Predicate System

The availability system uses function pointers in the builtin descriptions to check target capability at initialization time. This approach allows the same codebase to support multiple target configurations while only exposing appropriate functionality.

Sources: [gcc/config/riscv/riscv-builtins.cc 97-101](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L97-L101)[gcc/config/riscv/riscv-builtins.cc 234-248](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L234-L248)

## Integration Architecture

The RISC-V SFPU Core integrates with multiple layers of the GCC compilation system, from high-level built-in function calls down to RTL instruction generation.

### Compilation Pipeline Integration

The expansion system handles two primary builtin types: `RISCV_BUILTIN_DIRECT` for functions that return values, and `RISCV_BUILTIN_DIRECT_NO_TARGET` for void functions. Both types map directly to machine description patterns through instruction codes.

Sources: [gcc/config/riscv/riscv-builtins.cc 293-339](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/riscv-builtins.cc#L293-L339)

## Code Generation and Expansion

The RISC-V SFPU Core implements a sophisticated code generation system that converts built-in function calls into appropriate RTL instructions through pattern matching and operand preparation.

### Operand Processing System

The expansion system processes function arguments through a standardized operand preparation pipeline:

The operand processing system ensures that function arguments are properly converted to RTL operands with correct modes and constraints before attempting instruction expansion.

### Machine Description Integration

The core system integrates with GCC's machine description framework through instruction codes that map built-in functions to specific patterns. The system supports both target-producing and void instructions through the builtin type system.

The constraint system in [gcc/config/riscv/constraints.md 84-86](https://github.com/tenstorrent/sfpi-gcc/blob/763d6834/gcc/config/riscv/constraints.md?plain=1#L84-L86) provides an extension point for Tenstorrent-specific constraints, allowing the core RISC-V constraints to be augmented with additional SFPU-specific register and operand constraints.
