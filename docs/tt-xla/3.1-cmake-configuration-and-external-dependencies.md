# CMake Configuration and External Dependencies

Relevant source files
*   [CMakeLists.txt](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt)
*   [third_party/CMakeLists.txt](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt)

## Purpose and Scope

This document explains the CMake-based build system configuration for the tt-xla repository, focusing on the root build configuration, external dependency management, and build options. It covers how the project integrates tt-mlir via CMake's `ExternalProject_Add`, manages third-party dependencies like loguru, and provides various build configuration options for performance tracing, code coverage, and Explorer mode.

For information about the Python packaging layer that wraps this CMake build, see [Python Package Build and Distribution](https://deepwiki.com/tenstorrent/tt-xla/3.2-python-package-build-and-distribution). For Docker containerization of the build environment, see [Docker Build Infrastructure](https://deepwiki.com/tenstorrent/tt-xla/3.3-docker-build-infrastructure).

## CMake Project Structure

The tt-xla build system is organized as a hierarchical CMake project with the root [CMakeLists.txt 27-36](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L27-L36) defining the top-level project and including subdirectories:

| Subdirectory | Purpose |
| --- | --- |
| `pjrt_implementation/src` | Core PJRT plugin implementation |
| `third_party` | External dependencies (tt-mlir, loguru, pjrt_c_api) |
| `docs` | Documentation generation |
| `python_package` | Python wrapper installation |
| `tests/pjrt` | PJRT-level tests (enabled in Debug builds) |

## Environment Prerequisites and Compiler Configuration

The build system enforces several prerequisites at configuration time:

### Environment Activation Check

[CMakeLists.txt 29-31](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L29-L31) requires the tt-xla environment to be activated before running CMake:

`if (NOT DEFINED ENV{TTXLA_ENV_ACTIVATED})  message(FATAL_ERROR "tt-xla environment not activated. Please run 'source venv/activate'.")endif()`
This ensures that `TTMLIR_TOOLCHAIN_DIR` and other environment variables are properly set by the `venv/activate` script.

### Compiler and Language Standards

[CMakeLists.txt 39-44](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L39-L44) configures the project to use LLVM toolchain components:

| Setting | Value | Purpose |
| --- | --- | --- |
| `CMAKE_C_COMPILER` | `clang-17` | C compiler |
| `CMAKE_CXX_COMPILER` | `clang++-17` | C++ compiler |
| `CMAKE_CXX_STANDARD` | `17` | C++17 language features |
| `CMAKE_POSITION_INDEPENDENT_CODE` | `ON` | Enable PIC for shared libraries |
| `CMAKE_CXX_FLAGS` | `-fno-rtti` | Disable RTTI for compatibility with MLIR |
| `CMAKE_EXPORT_COMPILE_COMMANDS` | `ON` | Generate compile_commands.json |

**Sources:**[CMakeLists.txt 27-44](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L27-L44)

## Build Options and Configuration Variables

The build system provides several CMake options to control features, performance tracing, and debugging:

**Diagram: Build Configuration Options and Dependencies**

### Performance and Debugging Options

| Option | Default | Description | Effect |
| --- | --- | --- | --- |
| `TTMLIR_ENABLE_PERF_TRACE` | `ON` | Enable performance tracing in tt-mlir | Adds tracing instrumentation |
| `TTXLA_TRACY_ZONES` | `OFF` | Enable Tracy profiler zones in PJRT layer | Enables Tracy integration |
| `CODE_COVERAGE` | `OFF` | Enable code coverage reporting | Adds `--coverage` flags |
| `TTXLA_ENABLE_EXPLORER` | `OFF` | Build tt-mlir Explorer tool | Enables Python bindings and debug runtime |

[CMakeLists.txt 46-59](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L46-L59) defines these options, with Explorer mode automatically enabling Python bindings:

`option(TTXLA_ENABLE_EXPLORER "Enable Explorer feature in tt-mlir" OFF)if(TTXLA_ENABLE_EXPLORER)    set(TTMLIR_ENABLE_BINDINGS_PYTHON ON)endif()`
[third_party/CMakeLists.txt 41-44](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L41-L44) shows how Explorer mode also enables runtime debugging:

`set(TT_RUNTIME_DEBUG ${TT_RUNTIME_DEBUG} CACHE BOOL "Enable ttmlir runtime debugging")if(TTXLA_ENABLE_EXPLORER)    set(TT_RUNTIME_DEBUG ON)endif()`
### Build Type Configuration

[CMakeLists.txt 54-55](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L54-L55) defines `TTMLIR_BUILD_TYPE` as a separate cache variable from the main project's `CMAKE_BUILD_TYPE`:

`set(TTMLIR_BUILD_TYPE "Release" CACHE STRING "Build type for tt-mlir")set_property(CACHE TTMLIR_BUILD_TYPE PROPERTY STRINGS "Debug" "Release" "RelWithDebInfo" "MinSizeRel")`
This allows building tt-mlir in Release mode while the main project is in Debug mode, which can significantly improve compilation performance during development.

### Code Coverage Configuration

[CMakeLists.txt 65-79](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L65-L79) defines a `coverage_config` interface library that adds coverage flags when `CODE_COVERAGE=ON`:

`add_library(coverage_config INTERFACE)option(CODE_COVERAGE "Enable coverage reporting" OFF)if(CODE_COVERAGE AND CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")  target_compile_options(coverage_config INTERFACE    -O0        # no optimization    -g         # generate debug info    --coverage # sets all required flags  )`
Targets can link against `coverage_config` to automatically receive coverage instrumentation.

**Sources:**[CMakeLists.txt 46-79](https://github.com/tenstorrent/tt-xla/blob/c77995f6/CMakeLists.txt#L46-L79)

## External Dependencies Management

The [third_party/CMakeLists.txt 1-130](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L1-L130) file manages all external dependencies, with tt-mlir being the most complex integration.

### Version Pinning Strategy

[third_party/CMakeLists.txt 5-11](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L5-L11) pins dependency versions to specific Git commit hashes:

`option(USE_CUSTOM_TT_MLIR_VERSION "Flag to use TT_MLIR_VERSION set by the user" OFF) if (NOT DEFINED TT_MLIR_VERSION OR NOT USE_CUSTOM_TT_MLIR_VERSION)    set(TT_MLIR_VERSION "c7b8bd7784307cce46577bca1c2d33fdab31fccc")endif() set(LOGURU_VERSION "4adaa185883e3c04da25913579c451d3c32cfac1")`
Users can override `TT_MLIR_VERSION` by setting `USE_CUSTOM_TT_MLIR_VERSION=ON` and providing their own commit hash, enabling testing against development versions of tt-mlir.

**Sources:**[third_party/CMakeLists.txt 5-11](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L5-L11)

## tt-mlir Integration via ExternalProject

The tt-mlir compiler is integrated as a CMake `ExternalProject`, meaning it's built as part of the tt-xla build but in a separate CMake configuration scope.

### Build Directory Structure

[third_party/CMakeLists.txt 34-36](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L34-L36) establishes the directory layout:

`set(TTMLIR_SOURCE_DIR "${CMAKE_CURRENT_SOURCE_DIR}/tt-mlir")set(TTMLIR_BUILD_DIR "${TTMLIR_SOURCE_DIR}/src/tt-mlir/build")set(TTMLIR_LIB_DIR "${TTMLIR_SOURCE_DIR}/install")`
| Directory | Purpose |
| --- | --- |
| `third_party/tt-mlir/` | ExternalProject prefix |
| `third_party/tt-mlir/src/tt-mlir/` | Git checkout location |
| `third_party/tt-mlir/src/tt-mlir/build/` | CMake build directory |
| `third_party/tt-mlir/install/` | Install prefix for libraries |

### ExternalProject Configuration

[third_party/CMakeLists.txt 48-81](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L48-L81) defines the `ExternalProject_Add` invocation:

### Key CMake Arguments Passed to tt-mlir

[third_party/CMakeLists.txt 64-77](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L64-L77) passes configuration to the tt-mlir build:

| Argument | Value | Purpose |
| --- | --- | --- |
| `CMAKE_BUILD_TYPE` | `${TTMLIR_BUILD_TYPE}` | Independent build type control |
| `CMAKE_C_COMPILER` | `clang-17` | Match parent project compiler |
| `CMAKE_CXX_COMPILER` | `clang++-17` | Match parent project compiler |
| `CMAKE_CXX_COMPILER_LAUNCHER` | `ccache` | Enable compiler caching |
| `TT_RUNTIME_ENABLE_TTNN` | `ON` | Enable TTNN dialect and runtime |
| `TTMLIR_ENABLE_STABLEHLO` | `ON` | Enable StableHLO frontend |
| `TTMLIR_ENABLE_RUNTIME` | `ON` | Build runtime execution engine |
| `TT_RUNTIME_DEBUG` | Dynamic | Debug runtime (Explorer or manual) |

### tt-metal Runtime Root

[third_party/CMakeLists.txt 37-40](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L37-L40) handles `TT_METAL_RUNTIME_ROOT` environment variable:

`set(WITH_METAL_RUNTIME_ROOT_SET "")if(NOT DEFINED ENV{TT_METAL_RUNTIME_ROOT})    set(WITH_METAL_RUNTIME_ROOT_SET "TT_METAL_RUNTIME_ROOT=${TTPJRT_SOURCE_DIR}/third_party/tt-mlir/src/tt-mlir/third_party/tt-metal/src/tt-metal")endif()`
If not set in the environment, this points tt-mlir to use its own embedded tt-metal submodule. The variable is used in both the build command [third_party/CMakeLists.txt 59](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L59-L59) and optional Explorer build step [third_party/CMakeLists.txt 85](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L85-L85)

### Python Dependency Installation

[third_party/CMakeLists.txt 53-54](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L53-L54) installs tt-mlir's Python requirements before building:

`PATCH_COMMAND mkdir -p ${TTMLIR_BUILD_DIR}COMMAND TTPJRT_SOURCE_DIR=${TTPJRT_SOURCE_DIR} bash ${TTPJRT_SOURCE_DIR}/venv/install_ttmlir_requirements.sh`
This runs the `venv/install_ttmlir_requirements.sh` script in the `PATCH_COMMAND` phase, which occurs after Git checkout but before CMake configuration.

### Install Components

[third_party/CMakeLists.txt 61-62](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L61-L62) installs only the necessary components from tt-mlir:

`INSTALL_COMMAND ${CMAKE_COMMAND} --install <BINARY_DIR> --component SharedLibCOMMAND ${CMAKE_COMMAND} --install <BINARY_DIR> --component DistributedRuntime`
*   `SharedLib`: Compiler and runtime shared libraries (`.so` files)
*   `DistributedRuntime`: Distributed execution support

This selective installation keeps the tt-xla package size minimal by excluding headers, static libraries, and tools.

### Exposing tt-mlir Libraries

[third_party/CMakeLists.txt 95-110](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L95-L110) makes tt-mlir's shared libraries available to the tt-xla build:

`file(GLOB TTMLIR_LIBRARIES "${TTMLIR_LIB_DIR}/*.so")foreach(TTMLIR_LIBRARY ${TTMLIR_LIBRARIES})    get_filename_component(lib_name ${TTMLIR_LIBRARY} NAME_WE)    string(REPLACE "lib" "" lib_name ${lib_name})        add_library(${lib_name} SHARED IMPORTED GLOBAL)    set_target_properties(${lib_name} PROPERTIES        EXCLUDE_FROM_ALL TRUE        IMPORTED_LOCATION ${TTMLIR_LIBRARY}    )    add_dependencies(${lib_name} tt-mlir)endforeach()`
This creates CMake imported library targets (e.g., `TTMLIRCompiler`, `TTMLIRRuntime`) that other parts of the build can link against. The `EXCLUDE_FROM_ALL TRUE` property prevents these from being built unless actually needed as dependencies.

### Explorer Build Step

[third_party/CMakeLists.txt 83-89](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L83-L89) adds an optional build step for Explorer:

`if(TTXLA_ENABLE_EXPLORER)    ExternalProject_Add_Step(tt-mlir build_explorer        COMMAND env ${WITH_METAL_RUNTIME_ROOT_SET} ${CMAKE_COMMAND} --build <BINARY_DIR> -- explorer        DEPENDEES build        WORKING_DIRECTORY ${TTMLIR_BUILD_DIR}    )endif()`
The `explorer` target in tt-mlir builds the Explorer visualization and profiling tool.

**Sources:**[third_party/CMakeLists.txt 29-96](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L29-L96)

## loguru Logging Library Integration

The loguru library provides structured logging for the PJRT plugin implementation.

[third_party/CMakeLists.txt 114-127](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L114-L127) integrates loguru via `ExternalProject_Add`:

`ExternalProject_Add(    loguru    PREFIX ${TTPJRT_SOURCE_DIR}/third_party/loguru    CMAKE_GENERATOR Ninja    CMAKE_ARGS      -DCMAKE_BUILD_TYPE=Release      -DCMAKE_C_COMPILER=clang-17      -DCMAKE_CXX_COMPILER=clang++-17      -DCMAKE_CXX_FLAGS=${FLAGS}      -DCMAKE_INSTALL_PREFIX=${TTPJRT_SOURCE_DIR}/third_party/loguru/src/loguru-install    GIT_REPOSITORY https://github.com/emilk/loguru.git    GIT_TAG ${LOGURU_VERSION}    GIT_PROGRESS ON)`
Key differences from tt-mlir integration:

*   Always builds in Release mode for performance
*   Simpler configuration (no runtime or dialect options)
*   Pinned to commit `4adaa185883e3c04da25913579c451d3c32cfac1`

[third_party/CMakeLists.txt 112](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L112-L112) sets compiler flags for loguru:

`set(FLAGS "${CMAKE_CXX_FLAGS} -fPIC -Wno-undef -Wno-unsafe-buffer-usage -Wno-disabled-macro-expansion")`
These flags suppress common warnings in the loguru codebase that would otherwise fail the build.

**Sources:**[third_party/CMakeLists.txt 112-127](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L112-L127)

## TOOLCHAIN Build Mode

The [third_party/CMakeLists.txt 13-27](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L13-L27) section implements a special `TOOLCHAIN` mode for building only tt-mlir:

`if (TOOLCHAIN STREQUAL "ON")    cmake_minimum_required(VERSION 3.20)    project(ttmlir-toolchain LANGUAGES CXX C)    execute_process(        COMMAND git clone --recursive https://github.com/tenstorrent/tt-mlir.git ${PROJECT_SOURCE_DIR}/tt-mlir/src/tt-mlir        RESULT_VARIABLE result1        OUTPUT_VARIABLE output1        ERROR_VARIABLE error_output1    )    if (NOT ${result1} EQUAL 0)        message(FATAL_ERROR "Failed to clone tt-mlir: ${error_output1}")    endif()    execute_process(        COMMAND /bin/bash -c "cd ${PROJECT_SOURCE_DIR}/tt-mlir/src/tt-mlir && git checkout ${TT_MLIR_VERSION}"    )else()    # Normal build (lines 29-130)endif()`
### Purpose of TOOLCHAIN Mode

When `TOOLCHAIN=ON` is passed to CMake:

1.   Creates a minimal project that only clones and checks out tt-mlir
2.   Uses `git clone --recursive` to get all submodules
3.   Checks out the specific `TT_MLIR_VERSION` commit
4.   Skips all other tt-xla components

This mode is used by the Python package build system to pre-build tt-mlir as a separate step before building the main PJRT plugin. See [Python Package Build and Distribution](https://deepwiki.com/tenstorrent/tt-xla/3.2-python-package-build-and-distribution) for how this integrates with `setup.py`.

**Sources:**[third_party/CMakeLists.txt 13-27](https://github.com/tenstorrent/tt-xla/blob/c77995f6/third_party/CMakeLists.txt#L13-L27)
