## Purpose and Scope

This page documents the format conversion and inference systems that enable seamless data flow between Python test infrastructure and Tenstorrent hardware. The system consists of two primary subsystems:

1.   **Automatic Format Inference** - The `data_format_inference.py` module automatically determines intermediate data formats across the unpack→math→pack pipeline stages based on input/output format requirements and hardware constraints

2.   **Python ↔ Hardware Conversion** - The `pack.py` and `unpack.py` modules serialize PyTorch tensors into hardware-compatible byte arrays and deserialize hardware output back to PyTorch tensors

3.   **Golden Model Format Conversion** - The `SrcFormatModel` class provides precise bit-level format transformations for golden reference generation

This infrastructure enables:

*   Tests to specify only input/output formats while automatically handling intermediate conversions
*   Transparent serialization/deserialization between Python and hardware representations
*   Architecture-specific format inference (Wormhole vs Blackhole vs Quasar)
*   Accurate golden model generation matching hardware precision and rounding behavior

## Automatic Format Inference System

The format inference system (`data_format_inference.py`) automatically determines all intermediate data formats in the compute pipeline based on input/output format specifications and hardware constraints.

### Format Inference Entry Point

The `infer_data_formats()` function is the main entry point [tests/python_tests/helpers/data_format_inference.py 286-359](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/data_format_inference.py#L286-L359):

`def infer_data_formats(    input_format: DataFormat,    output_format: DataFormat,    is_fp32_dest_acc_en: DestAccumulation,    unpacking_to_dest: bool = False,    chip_arch: Optional[ChipArchitecture] = None,    input_format_B: DataFormat = None,) -> FormatConfig`
This function infers seven format configurations:

*   `unpack_A_src` - Source A format in L1 memory
*   `unpack_A_dst` - Source A format in registers (SrcA or Dest)
*   `unpack_B_src` - Source B format in L1 memory
*   `unpack_B_dst` - Source B format in registers (SrcB)
*   `math` - Computation format in destination registers
*   `pack_src` - Packer input format (from Dest registers)
*   `pack_dst` - Packer output format (to L1 memory)

### Math Format Inference

The `infer_math_format()` function determines the computation format [tests/python_tests/helpers/data_format_inference.py 264-283](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/data_format_inference.py#L264-L283):

**Rules:**

*   Both operands must belong to same exponent family (8-bit vs 5-bit)
*   Math format is set to first operand's format
*   Mixing exponent families (e.g., Float16_b + Float16) raises `ValueError`

`def infer_math_format(a: DataFormat, b: DataFormat = None) -> DataFormat:    if b is None:        return a        # Check exponent family compatibility    if (a.is_exponent_B() and not b.is_exponent_B()) or \       (not a.is_exponent_B() and b.is_exponent_B()):        raise ValueError(            f"Incompatible formats {a.name} and {b.name} "            f"with different exponent bit widths"        )        return a`

### Format Outlier Detection

The `is_format_combination_outlier()` function detects unsupported format combinations [tests/python_tests/helpers/data_format_inference.py 82-111](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/data_format_inference.py#L82-L111):

`def is_format_combination_outlier(    input_format: DataFormat,    output_format: DataFormat,    is_fp32_dest_acc_en: DestAccumulation,) -> bool:    """Check if combination requires workaround.        Outlier case: 8-bit exponent → Float16 without FP32 dest    Requires intermediate Float32 representation.    """    return (        input_format.is_exponent_B()        and not input_format.is_float32()        and output_format == DataFormat.Float16        and is_fp32_dest_acc_en == DestAccumulation.No    )`
This pattern occurs when converting Float16_b or Bfp8_b to Float16 without 32-bit destination accumulation. Hardware cannot perform this conversion directly, requiring `pack_src=Float32` as intermediate format.

### Multi-Stage Format Inference

The `data_formats()` function handles L1-to-L1 iteration pipelines [tests/python_tests/helpers/data_format_inference.py 390-482](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/data_format_inference.py#L390-L482):

`def data_formats(    input_format: DataFormat,    output_format: DataFormat,    is_fp32_dest_acc_en: DestAccumulation,    num_iterations: int,    unpacking_to_dest: bool = False,    chip_arch: Optional[ChipArchitecture] = None,    disable_format_inference: bool = False,    input_format_B: DataFormat = None,) -> List[FormatConfig]`
**Behavior:**

*   Single iteration: `[final_config]`
*   Multiple iterations: `[intermediate_config] * (n-1) + [final_config]`
*   Intermediate configs: Input and output formats both set to `input_format`
*   Final config: Output format set to `output_format`

This enables tests with multiple compute passes where intermediate results are stored back to L1 in the original input format, with final result in the desired output format.

### Block Floating-Point Packing (BFP8_b)

The `pack_bfp8_b()` function packs data into BFP8_b format with shared exponents [tests/python_tests/helpers/pack.py 116-159](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/pack.py#L116-L159):

**BFP8_b Structure:**

*   16-element blocks, each with 1 shared 8-bit exponent
*   16 × 8-bit mantissas (sign + 7-bit value)
*   Minimum 16 exponents total (padded if needed)

**Memory layout:**`[exponents...] + [mantissas...]`

### MX Format Packing (MxFp8R/P)

MX formats use 32-element blocks with shared E8M0 scales [tests/python_tests/helpers/pack.py 167-293](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/pack.py#L167-L293):

**MX Format Structure:**

*   32-element blocks per OCP MX specification
*   1 × 8-bit E8M0 scale per block
*   32 × 8-bit FP8 elements (E5M2 or E4M3)

**Memory layout:**`[all_scales] + [all_elements]` (fully separated)

**Diagram: MX Format Packing Pipeline**

Implementation [tests/python_tests/helpers/pack.py 167-240](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/pack.py#L167-L240):

`def _pack_mxfp8(tensor, fp8_dtype, element_max_normal, num_faces=4, face_r_dim=16):    # Reshape into 32-element blocks    num_blocks = len(fp32_array) // MXFP8_BLOCK_SIZE    blocks = fp32_array.reshape(num_blocks, MXFP8_BLOCK_SIZE)        # Calculate E8M0 scales (vectorized)    max_abs_values = np.max(np.abs(blocks), axis=1)    scale_ratio = max_abs_values / element_max_normal    exponents = np.ceil(np.log2(scale_ratio, where=(scale_ratio > 0)))    scales_e8m0 = np.clip(exponents, -127, 127) + 127        # Apply scales and convert to FP8    scale_factors = np.exp2(scales_e8m0 - 127.0)    scaled_blocks = blocks / scale_factors[:, np.newaxis]    fp8_blocks = scaled_blocks.astype(fp8_dtype)        # FULLY SEPARATED layout    return scales_e8m0.tolist() + fp8_blocks.tobytes()`

## Format-Specific Conversion Details

### Float16_b (bfloat16) to TF32

Float16_b format uses 1 sign bit, 8 exponent bits, and 7 mantissa bits. The conversion [tests/python_tests/helpers/golden_generators.py 268-301](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L268-L301):

`def _fp16b_to_tf32(tensor: torch.Tensor) -> tuple:    tensor_raw = tensor.to(torch.bfloat16).view(torch.uint16).to(torch.int64)        # Bit field widths    BFP16_MANT_WIDTH = 7    BFP16_EXP_WIDTH = 8        # Extract components using masks and shifts    sign = (tensor_raw & BFP16_SIGN_MASK) >> BFP16_SIGN_SHAMT    exp = (tensor_raw & BFP16_EXP_MASK) >> BFP16_EXP_SHAMT    mant = (tensor_raw & BFP16_MANT_MASK) >> BFP16_MANT_SHAMT        # Apply exponent bias (127 for 8-bit exponent)    exp = exp - 127        # Pad mantissa to 10 bits (7 → 10 requires 3 zero bits on right)    mant = mant << 3        # Include implied leading 1    mant = mant | (1 << 10)        return (sign, exp, mant)`
The key transformations are:

*   Exponent bias: Subtract 127 (standard for 8-bit exponents)
*   Mantissa padding: Left-shift by 3 to convert 7-bit mantissa to 10-bit
*   Implied 1: Set bit 10 to represent the implicit leading 1

### Float16 to TF32

Float16 format uses 1 sign bit, 5 exponent bits, and 10 mantissa bits. The conversion [tests/python_tests/helpers/golden_generators.py 303-333](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L303-L333):

`def _fp16_to_tf32(tensor: torch.Tensor) -> tuple:    tensor_raw = tensor.to(torch.float16).view(torch.uint16).to(torch.int64)        # Bit field widths    FP16_MANT_WIDTH = 10  # Already matches TF32 width    FP16_EXP_WIDTH = 5        # Extract components    sign = (tensor_raw & FP16_SIGN_MASK) >> FP16_SIGN_SHAMT    exp = (tensor_raw & FP16_EXP_MASK) >> FP16_EXP_SHAMT    mant = (tensor_raw & FP16_MANT_MASK) >> FP16_MANT_SHAMT        # Apply exponent bias (15 for 5-bit exponent)    exp = exp - 15        # No padding needed - mantissa is already 10 bits        # Include implied leading 1    mant = mant | (1 << FP16_MANT_WIDTH)        return (sign, exp, mant)`

Float32 format uses 1 sign bit, 8 exponent bits, and 23 mantissa bits. The conversion [tests/python_tests/helpers/golden_generators.py 335-370](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L335-L370):

## from_src_format: TF32 → DataFormat

The `from_src_format()` method converts TF32 components back to a PyTorch tensor in the target `DataFormat`. This is a universal converter that works for any format [tests/python_tests/helpers/golden_generators.py 398-413](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L398-L413):

2.   **Reconstruct value**: Apply the standard floating-point formula: `(-1)^sign × 2^exp × mantissa`
3.   **Cast to target format**: Use PyTorch's type conversion to the appropriate dtype

This universal converter works because:

*   The exponent is already unbiased and can be used directly with the power-of-two operation

## Exponent Bias Calculation

The `_exponent_bias()` helper function computes the standard IEEE 754 bias for a given exponent width [tests/python_tests/helpers/golden_generators.py 255-257](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/golden_generators.py#L255-L257):
