The `tt-metal` repository provides a comprehensive suite of triage and post-mortem analysis tools designed to diagnose hardware hangs, kernel panics, and NoC (Network-on-Chip) connectivity issues. The primary interface for these operations is `tt-triage`, a modular Python diagnostic framework that integrates with the `TTExaLens` debugger to inspect device state without requiring a fully functional runtime environment.

* * *

## Overview of `tt-triage`

The `tt-triage` tool (`tools/triage/triage.py`) runs a series of health checks on the Tenstorrent AI hardware, including ARC processor heartbeat checks, NoC link status, L1 memory/mailbox inspection, and detailed RISC-V core debugging via callstack analysis. It identifies currently running kernels, their states, and provides detailed callstack information to aid in troubleshooting failed or hung operations. `tt-triage` supports both local device connections and remote debugging via the TTExaLens server interface [tools/triage/triage.py 6-42](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/triage.py#L6-L42)

### Modular Script Framework and Data Flow

The triage system is architected as a collection of modular diagnostic scripts, orchestrated by the main `triage.py` runner. Each script performs one aspect of hardware inspection, for example, dumping callstacks, checking NoC status, or analyzing dispatcher state. The infrastructure uses singleton caching decorators (`@triage_singleton`) to guarantee efficient reuse of hardware reads and consistent state sampling across scripts [tools/triage/triage.py 136-160](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/triage.py#L136-L160)

## Automated Triage and Hang Detection

In CI and production environments, triage can be automatically triggered on device hangs detected by timeout conditions during fast dispatch operations. This avoids manual intervention and enables rapid root cause diagnostics.

The automatic triage configuration is controlled via the following environment variables set in GitHub Actions or runtime environments:

| Environment Variable | Description |
| --- | --- |
| `TT_METAL_DISPATCH_TIMEOUT_COMMAND_TO_EXECUTE` | Command executed automatically on timeout, typically runs `hang_report.py` and `tt-triage.py` scripts [.github/actions/setup-job/action.yml 157](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/actions/setup-job/action.yml#L157-L157) |
| `TT_METAL_OPERATION_TIMEOUT_SECONDS` | Timeout threshold in seconds before an operation is considered hung [.github/actions/setup-job/action.yml 158](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/actions/setup-job/action.yml#L158-L158) |
| `TT_TRIAGE_DUMP_RISC_DEBUG_SIGNALS_PATH` | File path where debug bus signal groups are dumped upon hang detection [.github/actions/setup-job/action.yml 149](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/actions/setup-job/action.yml#L149-L149) |
| `TT_TRIAGE_OUTPUT_PATH` | File path to store triage textual output [.github/actions/setup-job/action.yml 154](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/actions/setup-job/action.yml#L154-L154) |
| `TT_TRIAGE_LLM_OUTPUT_PATH` | File path to store machine-readable triage output for LLM or CI use [.github/actions/setup-job/action.yml 155](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/.github/actions/setup-job/action.yml#L155-L155) |

These variables enable automatic hardware state acquisition and triage summary generation when dispatch progress stalls beyond configured limits.

## Post-Mortem Analysis Components

### 1. Callstack Dumping

The `dump_callstacks.py` script extracts and displays callstacks from all RISC-V cores (BRISC, NCRISC, TRISCs, ERISC) across inspected devices. It reads the program counter (PC) from cores and decodes it against ELF debug symbols cached within `elfs_cache.py`. This mapping enables identifying the precise source code locations and functions executing at hang time [tools/triage/dump_callstacks.py 14-80](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump_callstacks.py#L14-L80)

*   By default, cores with `Go Message` set to `"DONE"` are filtered out as idle, controllable with `--all-cores`.
*   Verbosity flags increase the detail of displayed fields (top frame only, full callstack, GDB-based callstack).
*   Handles exceptional conditions such as PC out of range or timeouts gracefully, providing error messages.

### 2. Fast Dispatch State Inspection

Fast dispatch is an advanced command queue dispatch mechanism optimized for latency. `dump_fast_dispatch.py` reads critical state variables from the prefetch and dispatch cores to determine where command queue processing might be stalled [tools/triage/dump_fast_dispatch.py 13-14](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump_fast_dispatch.py#L13-L14)

The tool correlates kernel names running on cores with respective core information obtained from `dispatcher_data.py`. It fetches global variables indicating fence pointers, wait counts, and callback fences to detect pipeline stalls.

**Key Data Structures:**

This inspection aids in identifying the stage where the dispatch workflow is blocked, e.g., waiting on worker cores or internal data buffers.

_Sources: [tools/triage/dump\_fast\_dispatch.py 44-64](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_fast\_dispatch.py#L44-L64)[tools/triage/dispatcher\_data.py 47-88](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dispatcher\_data.py#L47-L88)[tools/triage/dump\_fast\_dispatch.py 88-124](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_fast\_dispatch.py#L88-L124)_

### 3. Running Operation Summaries

`dump_running_operations.py` aggregates and summarizes currently running operations across all inspected cores [tools/triage/dump_running_operations.py 14-22](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump_running_operations.py#L14-L22)

*   Maps host-assigned operation IDs to their corresponding operation names and parameters via the `operation_provider`.
*   Reports device and core counts executing each unique operation.
*   Truncates core listings by default for readability, expandable with `--full-cores`.

This overview helps identify if a particular operation is implicated in a cluster-wide hang condition.

_Sources: [tools/triage/dump\_running\_operations.py 14-32](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_running\_operations.py#L14-L32)[tools/triage/dump\_running\_operations.py 79-115](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_running\_operations.py#L79-L115)_

### 4. NoC Status and Transaction Checks

The `check_noc_status.py` script verifies the internal router synchronization by comparing firmware global variables that track NoC transactions against actual NoC hardware status registers [tools/triage/check_noc_status.py 1-13](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/check_noc_status.py#L1-L13)

*   Validates that the number of NoC requests and responses match in idle states.
*   Highlights issues such as queue mismatches, indicating possible stuck or corrupted NoC traffic.

These checks are crucial since NoC issues often manifest as system-wide hangs or deadlocks.

_Sources: [tools/triage/check\_noc\_status.py 1-13](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/check\_noc\_status.py#L1-L13)_

### 5. Debug Bus Signal Collection

For in-depth debugging, `dump_risc_debug_signals.py` collects hardware debug bus signal groups from cores identified as broken during triage runs [tools/triage/dump_risc_debug_signals.py 10-12](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump_risc_debug_signals.py#L10-L12)

*   Temporarily overwrites the first 16 bytes of the firmware text section in L1 memory to enable reading debug signals.
*   Restores original firmware text afterward to maintain hardware state integrity.
*   Outputs a JSON file that includes 128-bit signal values per group and the current git commit hash for traceability.

This hardware signal snapshot supports low-level diagnostics, often required by firmware engineers.

_Sources: [tools/triage/dump\_risc\_debug\_signals.py 68-108](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_risc\_debug\_signals.py#L68-L108)[tools/triage/dump\_risc\_debug\_signals.py 141-147](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/dump\_risc\_debug\_signals.py#L141-L147)_

## ARC Processor Health Monitoring

The ARC processor, responsible for power and system management on Tenstorrent devices, is monitored via the `check_arc.py` script [tools/triage/check_arc.py 10-12](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/check_arc.py#L10-L12)

*   Takes two heartbeat samples at a 2-second interval by reading the `TIMER_HEARTBEAT` telemetry register.
*   Verifies the heartbeat rate lies within expected bounds (9 to 11 beats per second).
*   Reads and validates the ARC `postcode` register to confirm firmware health (expected magic value `0xC0DE____` on Wormhole devices).
*   Provides uptime estimation and clock frequency information.

This ensures system management cores remain alive and responsive during hardware operation.

_Sources: [tools/triage/check\_arc.py 53-130](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tools/triage/check\_arc.py#L53-L130)_

# Summary

The `tt-metal` triage tools provide robust, extensible, and automated post-mortem hardware diagnostics for Tenstorrent AI accelerators. Core facilities include:

*   Deep inspection of all RISC cores with mapped callstack information.
*   Fast dispatch kernel internal state dumping to diagnose command queue issues.
*   Broad operation summaries aggregated cluster-wide.
*   NoC status integrity verification.
*   Debug bus signal group collection for broken cores.
*   Health checks on ARC system management processors.

Integration with the TTExaLens debugger package allows remote kernel and device state access without runtime dependencies. Automatic hang detection and triage invocation enable rapid CI/debugging feedback and improve hardware failure turnaround.
