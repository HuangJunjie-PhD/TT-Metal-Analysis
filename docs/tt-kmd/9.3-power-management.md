# Power Management

Relevant source files
*   [blackhole.c](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c)
*   [chardev.c](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c)

This document describes the power management subsystem in the Tenstorrent kernel driver. It covers how user-space applications request device power states, how multiple clients' power requirements are aggregated (MAX of settings, OR of flags), the idle power-down grace period mechanism, and the communication protocols with device firmware.

## Power State Model

The driver implements a **per-file-descriptor power state model**. Each open file descriptor specifies its power requirements independently via `struct tenstorrent_power_state`. The driver aggregates these requirements across all open file descriptors and communicates a unified power state to the device firmware.

**Sources:**[chardev.c 377-437](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L377-L437)[chardev.c 439-463](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L439-L463)

## Power-Aware vs Legacy Clients

The driver distinguishes between legacy and power-aware clients based on the `O_APPEND` flag during `open()`.

| Client Type | Detection (`open`) | Default State | Behavior |
| --- | --- | --- | --- |
| **Legacy** | `O_APPEND` not set | All flags ON except `MAX_AI_CLK` | Automatically triggers aggregation on open. |
| **Power-Aware** | `O_APPEND` set | All flags OFF | Starts at minimum power; must call IOCTL to request power. |

Legacy clients are initialized with `TT_POWER_FLAG_ALL & ~TT_POWER_FLAG_MAX_AI_CLK`[chardev.c 656-659](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L656-L659) Power-aware clients (using `O_APPEND`) start with zero flags [chardev.c 663-664](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L663-L664)

**Sources:**[chardev.c 629-673](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L629-L673)[chardev.c 656-659](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L656-L659)

## Power State Aggregation

When any client updates its power state or a file descriptor is closed, `tenstorrent_set_aggregated_power_state()` recomputes the device-wide state.

### Aggregation Rules

1.   **Flags (OR):** The aggregated `power_flags` is the bitwise OR of all clients' flags. If any client needs a feature, it stays on.
2.   **Settings (MAX):** For each entry in the `power_settings` array, the driver takes the `max()` value requested across all clients.
3.   **Unspecified Flags:** If a client provides fewer than 15 flags (based on `validity`), the unspecified bits are treated as **1** (ON) for backward compatibility [chardev.c 402-404](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L402-L404)

### Reset Generation Filtering

Clients opened before a device reset (where `priv->open_reset_gen != atomic_long_read(&tt_dev->reset_gen)`) are excluded from aggregation to ensure stale processes do not keep the hardware in a high-power state [chardev.c 391-392](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L391-L392)

**Sources:**[chardev.c 377-437](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L377-L437)[chardev.c 386-426](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L386-L426)

## Deferred Idle Power-Down

To avoid frequent power state oscillations, the driver supports a deferred power-down mechanism.

1.   **Grace Period:** Controlled by the `idle_power_down_grace_ms` module parameter.
2.   **Trigger:** When the last file descriptor is closed, if `defer_idle_powerdown` is enabled in the `tenstorrent_device_class`, the driver schedules `power_down_work`[chardev.c 704-712](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L704-L712)
3.   **Cancellation:** Any new `open()` or a device `reset` will call `cancel_delayed_work_sync(&tt_dev->power_down_work)` to prevent the device from powering down while active [chardev.c 210](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L210-L210)[chardev.c 637](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L637-L637)

**Sources:**[device.h 57-61](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/device.h#L57-L61)[chardev.c 704-712](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L704-L712)[chardev.c 210](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L210-L210)

## Device Implementation Details

### Wormhole (Legacy)

Wormhole devices do not support granular power state settings via this interface. `wormhole_set_power_state` is a no-op that always returns 0 [wormhole.c 1041-1044](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L1041-L1044) However, Wormhole uses scratch registers to signal basic states like `WH_FW_MSG_ASTATE0` (Active) and `WH_FW_MSG_ASTATE3` (Suspend) [wormhole.c 114-115](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/wormhole.c#L114-L115)

### Blackhole (Message Queue)

Blackhole uses the ARC message queue protocol. The `blackhole_set_power_state` function packs the aggregated `validity`, `flags`, and `settings` into an `arc_msg` with type `ARC_MSG_TYPE_POWER_SETTING` (0x21) and sends it to the ARC firmware [blackhole.c 1082-1097](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1082-L1097)

**Sources:**[blackhole.c 1082-1097](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L1082-L1097)[blackhole.c 71](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/blackhole.c#L71-L71)

## TT_POWER_FLAG Definitions

The following flags are used in `power_flags` to control specific hardware subsystems:

| Flag Constant | Bit | Description |
| --- | --- | --- |
| `TT_POWER_FLAG_MAX_AI_CLK` | 14 | Requests maximum AI clock frequency. |
| `TT_POWER_FLAG_ALL` | 0x7FFF | Helper mask representing all 15 supported flags [chardev.c 29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L29-L29) |

**Sources:**[chardev.c 29](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L29-L29)[chardev.c 402-404](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L402-L404)

## Synchronization

*   **`chardev_mutex`**: Protects the `open_fds_list` and ensures that only one power state aggregation occurs at a time [chardev.c 384](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L384)
*   **`priv->mutex`**: Protects the `power_state` within a specific `chardev_private` instance during IOCTL updates [chardev.c 458](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L458-L458)
*   **`reset_rwsem`**: Taken in read mode during power updates to ensure a device reset is not in progress [chardev.c 385](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L385-L385)

**Sources:**[chardev.c 384-385](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L384-L385)[chardev.c 458](https://github.com/tenstorrent/tt-kmd/blob/0ab170d7/chardev.c#L458-L458)
