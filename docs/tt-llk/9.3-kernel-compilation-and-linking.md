## Purpose and Scope

This document details the kernel compilation and linking process in the tt-llk repository. It covers how RISC-V source files are compiled into object files and linked into ELF executables for BRISC and TRISC cores. The page explains the role of `trisc.cpp` with its `LLKT_TRISC_UNPACK/MATH/PACK` flags, the assembly startup code in `tmu-crt0.S`, the BRISC kernel in `brisc.cpp`, the linker script configuration, and build flags that control features like coverage and profiling. For information about the build system architecture and artifact management, see [Compilation Architecture and Artifacts](https://deepwiki.com/tenstorrent/tt-llk/9.1-compilation-architecture-and-artifacts). For environment setup and toolchain installation, see [Environment Setup and SFPI Toolchain](https://deepwiki.com/tenstorrent/tt-llk/9.2-environment-setup-and-sfpi-toolchain).

## Compilation Process Overview

The kernel compilation process in tt-llk separates into two phases: **shared artifact compilation** and **variant-specific compilation**. This separation enables efficient parallel test execution by compiling common code once and reusing it across test variants.

### Compilation Phases

**Diagram: Kernel Compilation and Artifact Dependencies**

### Shared vs Variant Artifacts

| Artifact Type | Compiled Once Per | Reused Across | Build Directory |
| --- | --- | --- | --- |
| **Shared** | Architecture + profiler mode | All test variants | `SHARED_OBJ_DIR` or `PROFILER_SHARED_OBJ_DIR` |
| **Variant** | Test variant hash (formats, templates, runtimes) | Single test configuration | `ARTEFACTS_DIR/{test_name}/{variant_id}` |

The variant hash is computed via SHA256 over test configuration parameters, excluding runtime data and stimuli:

`temp_str = [str(value) for field_name, value in self.__dict__.items()            if field_name not in NON_COMPILATION_ARGUMENTS]self.variant_id = sha256(str(" | ".join(temp_str)).encode()).hexdigest()`

## Assembly Startup Code: tmu-crt0.S

The `tmu-crt0.S` file provides the low-level assembly startup code that initializes the RISC-V processor state before jumping to C++ code. It is compiled once as a shared artifact and linked into all ELF executables.

### Compilation Command

`${GXX} ${ARCH_NON_COMPUTE} ${OPTIONS_ALL} ${OPTIONS_COMPILE} \    -c -o ${SHARED_OBJ_DIR}/tmu-crt0.o \    ${HELPERS}/tmu-crt0.S`
| Flag | Value | Purpose |
| --- | --- | --- |
| `ARCH_NON_COMPUTE` | `-mcpu=tt-wh` (Wormhole) `-mcpu=tt-bh` (Blackhole) | Non-Tensix RISC-V target |
| `OPTIONS_ALL` | `-g -O3 -std=c++17 -ffast-math` | Debug symbols, optimization, C++17, fast math |
| `OPTIONS_COMPILE` | Architecture-specific includes and defines | Include paths and compilation flags |
| `-c` | Compile only (no linking) | Produces `.o` object file |

The `ARCH_NON_COMPUTE` flag is used instead of `ARCH_COMPUTE` because the startup code runs before the Tensix-specific features are initialized.

**Sources:**[tests/python_tests/helpers/test_config.py 605-608](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L605-L608)[tests/python_tests/helpers/test_config.py 186-208](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L186-L208)

### Assembly File Location

The startup assembly is located at `tests/helpers/tmu-crt0.S` and provides:

*   Stack pointer initialization
*   Zero-initialization of BSS section
*   Jump to `main()` function entry point
*   Exception handler setup

**Sources:**[tests/python_tests/helpers/test_config.py 103-104](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L103-L104)[tests/python_tests/helpers/test_config.py 605-608](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L605-L608)

## BRISC Kernel Compilation: brisc.cpp

The BRISC (Behavioral RISC) kernel manages kernel initialization and synchronization on non-Quasar architectures. On Quasar, TRISC cores boot directly and BRISC is not used.

### Compilation Command

`${GXX} ${ARCH_NON_COMPUTE} ${BRISC_DEFINE_COVERAGE} ${OPTIONS_ALL} \    ${NON_COVERAGE_OPTIONS_COMPILE} \    -c -o ${SHARED_OBJ_DIR}/brisc.o \    ${RISCV_SOURCES}/brisc.cpp`
| Flag | Value | Purpose |
| --- | --- | --- |
| `ARCH_NON_COMPUTE` | `-mcpu=tt-wh` or `-mcpu=tt-bh` | Non-Tensix RISC-V target |
| `BRISC_DEFINE_COVERAGE` | `-DCOVERAGE` (if coverage enabled) | Enable coverage instrumentation |
| `NON_COVERAGE_OPTIONS_COMPILE` | Base compile options without `-fprofile-arcs` | Avoid double instrumentation |

The BRISC kernel uses `NON_COVERAGE_OPTIONS_COMPILE` instead of `OPTIONS_COMPILE` because coverage instrumentation is handled separately via `coverage.o` linkage, not through compiler flags on `brisc.cpp` itself.

**Sources:**[tests/python_tests/helpers/test_config.py 610-616](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L610-L616)[tests/python_tests/helpers/test_config.py 103-104](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L103-L104)

### Architecture-Specific Behavior

**Diagram: Architecture-Specific BRISC Compilation**

On Quasar, the boot mode is set to `LLKT_BOOT_MODE_TRISC` and BRISC compilation is skipped:

`if TestConfig.CHIP_ARCH == ChipArchitecture.QUASAR:    OPTIONS_COMPILE += "-DLLK_BOOT_MODE_TRISC "else:    OPTIONS_COMPILE += "-DLLK_BOOT_MODE_BRISC "`

### BRISC Source Location

The BRISC kernel implementation is at `tests/helpers/src/brisc.cpp` and contains:

*   Kernel boot sequence
*   TRISC core initialization
*   Mailbox management
*   Coverage data collection (if enabled)

## TRISC Kernel Compilation: trisc.cpp

The TRISC kernels implement the three-stage pipeline (Unpack, Math, Pack). The `trisc.cpp` source is compiled three times with different preprocessor flags to generate specialized kernels for each stage.

### Parallel Compilation Process

**Diagram: Parallel TRISC Kernel Compilation**

The build system uses `ThreadPoolExecutor(max_workers=3)` to compile all three TRISC stages in parallel:

`with ThreadPoolExecutor(max_workers=3) as executor:    futures = [        executor.submit(build_kernel_part_main, name)        for name in TestConfig.KERNEL_COMPONENTS  # ["unpack", "math", "pack"]    ]    for fut in futures:        fut.result()`
**Sources:**[tests/python_tests/helpers/test_config.py 637-643](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L637-L643)[tests/python_tests/helpers/test_config.py 126](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L126-L126)

### TRISC Compilation Flags

| Kernel Stage | Preprocessor Flags | Purpose |
| --- | --- | --- |
| **Unpack** | `-DLLK_TRISC_UNPACK -DCOMPILE_FOR_TRISC=0` | Compiles unpacker LLK operations, disables math/pack code |
| **Math** | `-DLLK_TRISC_MATH -DCOMPILE_FOR_TRISC=1` | Compiles math LLK operations, disables unpack/pack code |
| **Pack** | `-DLLK_TRISC_PACK -DCOMPILE_FOR_TRISC=2` | Compiles packer LLK operations, disables unpack/math code |

The `COMPILE_FOR_TRISC` index maps to RISC core assignments:

*   `0` → TRISC0 (Unpacker)
*   `1` → TRISC1 (Math)
*   `2` → TRISC2 (Packer)

**Sources:**[tests/python_tests/helpers/test_config.py 627-635](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L627-L635)

### Compilation Command Template

`${GXX} ${ARCH_COMPUTE} ${OPTIONS_ALL} ${OPTIONS_COMPILE} \    -DCOMPILE_FOR_TRISC=${index} -DLLK_TRISC_${STAGE_UPPER} \    -c -o ${SHARED_OBJ_DIR}/main_${stage}.o \    ${RISCV_SOURCES}/trisc.cpp`
| Flag | Purpose |
| --- | --- |
| `ARCH_COMPUTE` | `-mcpu=tt-wh-tensix` or `-mcpu=tt-bh-tensix` - enables Tensix instructions |
| `OPTIONS_ALL` | `-g -O3 -std=c++17 -ffast-math` |
| `OPTIONS_COMPILE` | Architecture defines, include paths, coverage/profiler flags |

For Quasar, the `COMPILE_FOR_TRISC` flag is omitted as it uses a different kernel boot model.

**Sources:**[tests/python_tests/helpers/test_config.py 627-635](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L627-L635)[tests/python_tests/helpers/test_config.py 188-204](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L188-L204)

### Profiler Build Variation

When `profiler_build == ProfilerBuild.Yes`, the compilation adds `-DLLK_PROFILER` flag:

`if self.profiler_build == ProfilerBuild.Yes:    OPTIONS_COMPILE += "-DLLK_PROFILER "`
This enables performance counter instrumentation in the TRISC kernels. Profiler builds require separate shared artifacts stored in `PROFILER_SHARED_OBJ_DIR` because the code generation differs.

## Coverage Instrumentation Compilation: coverage.cpp

For coverage builds (`--coverage` flag), an additional `coverage.o` object is compiled and linked to collect code coverage data.

### Compilation Command

`${GXX} ${ARCH_NON_COMPUTE} ${OPTIONS_ALL} ${NON_COVERAGE_OPTIONS_COMPILE} \    -fno-strict-aliasing \    -c -o ${SHARED_OBJ_DIR}/coverage.o \    ${RISCV_SOURCES}/coverage.cpp`
The coverage object uses `NON_COVERAGE_OPTIONS_COMPILE` (without `-fprofile-arcs -ftest-coverage`) to avoid recursive instrumentation. The `-fno-strict-aliasing` flag is required for safe type punning in the coverage library.

**Sources:**[tests/python_tests/helpers/test_config.py 618-625](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L618-L625)

### Coverage Build Configuration

Coverage builds use different compilation options and memory layouts:

| Configuration | Coverage Build | Non-Coverage Build |
| --- | --- | --- |
| **Compile flags** | `-fprofile-arcs -ftest-coverage -fprofile-info-section -DCOVERAGE` | None |
| **Linker dependencies** | `coverage.o -lgcov` | None |
| **Memory layout** | `memory.{arch}.debug.ld` | `memory.{arch}.ld` |
| **Runtime address** | `0x64000` | `0x20000` |

The debug memory layout provides additional space for coverage data structures. The runtime address differs to avoid conflicts with coverage metadata.

**Sources:**[tests/python_tests/helpers/test_config.py 546-553](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L546-L553)[tests/python_tests/helpers/test_config.py 139-140](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L139-L140)

## Variant-Specific Kernel Compilation

After shared artifacts are compiled, each test variant compiles its own kernel objects that contain the test-specific LLK operation calls.

### Build Header Generation

Each variant generates a `build.h` header file containing:

*   Data format configurations (`UNPACK_A_IN`, `PACK_OUT`, etc.)
*   Template parameters (e.g., `LOOP_FACTOR`, `BLOCK_CT_DIM`)
*   Runtime parameter struct definition
*   Feature flags (`is_fp32_dest_acc_en`, `l1_acc_en`, `unpack_to_dest`)

`// Example generated build.h#pragma once#include "operand.h"#include "llk_defs.h" constexpr bool is_fp32_dest_acc_en = true;constexpr auto UNPACK_A_IN = ckernel::to_underlying(DataFormat::Float16_b);constexpr auto PACK_OUT = ckernel::to_underlying(DataFormat::Bfp8_b); struct RuntimeParams {    std::uint32_t TILE_SIZE_PACK;    std::uint32_t LOOP_FACTOR;    // ...};`
**Sources:**[tests/python_tests/helpers/test_config.py 792-866](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L792-L866)

### Kernel Object Compilation

**Diagram: Variant-Specific Kernel Compilation**

The `build_kernel_part()` function compiles each kernel stage:

`${GXX} ${ARCH_COMPUTE} ${OPTIONS_ALL} ${OPTIONS_COMPILE} \    -DCOMPILE_FOR_TRISC=${index} -DLLK_TRISC_${STAGE_UPPER} \    -I${VARIANT_DIR} \    -c -o ${VARIANT_OBJ_DIR}/kernel_${stage}.o \    ${TESTS_WORKING_DIR}/${test_name}.cpp`
The `-I${VARIANT_DIR}` flag ensures `build.h` is found during compilation.

**Sources:**[tests/python_tests/helpers/test_config.py 920-941](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L920-L941)

### Parallel Variant Compilation

Like shared artifacts, variant kernels are compiled in parallel:

`with ThreadPoolExecutor(max_workers=3) as executor:    futures = [        executor.submit(build_kernel_part, name)        for name in TestConfig.KERNEL_COMPONENTS    ]    for fut in futures:        fut.result()`
This maximizes CPU utilization during the compilation phase of test execution.

**Sources:**[tests/python_tests/helpers/test_config.py 943-949](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L943-L949)

## Linking Process and Linker Scripts

After compilation, object files are linked into executable ELF files using architecture-specific linker scripts that define memory regions and section placement.

### Linker Script Organization

**Diagram: Linker Script Hierarchy and Usage**

Linker scripts are located in `tests/helpers/ld/`:

*   `memory.{arch}.ld` - Normal memory layout
*   `memory.{arch}.debug.ld` - Extended memory for coverage/debug
*   `brisc.ld` - BRISC-specific entry and sections
*   `trisc.ld` - TRISC-specific entry and sections
*   `sections.ld` - Common section definitions (.text, .data, .bss, etc.)

**Sources:**[tests/python_tests/helpers/test_config.py 104](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L104-L104)[tests/python_tests/helpers/test_config.py 532-553](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L532-L553)[tests/helpers/ld/memory.blackhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L1-L68)[tests/helpers/ld/memory.wormhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L1-L68)

### Memory Layout Configuration

The linker scripts define precise memory regions for L0 (local memory) and L1 (shared memory) across architectures. Memory layouts differ between normal and coverage builds to accommodate instrumentation data.

#### L0 Memory Regions (Local Core Memory)

L0 memory is private per-core and primarily used for stack space:

| Architecture | L0 Base Address | BRISC Stack | TRISC Stack | NCRISC Stack |
| --- | --- | --- | --- | --- |
| **Wormhole B0** | `0xFFB00000` | 4K | 2K | 4K |
| **Blackhole** | `0xFFB00000` | 8K | 4K | 8K |

**Sources:**[tests/helpers/ld/memory.blackhole.debug.ld 7-11](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L7-L11)[tests/helpers/ld/memory.wormhole.debug.ld 7-11](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L7-L11)

Coverage builds use extended L1 regions to store gcov instrumentation data. The layout is sequential from L1 base address `0x0`:

| `TRISC0_GCOV_MEM` | `0x2A400` | 8K | TRISC0 coverage data |
| `TRISC1_CODE` | `0x2C400` | 64K | TRISC1 math kernel |
| `TRISC1_LOCAL_DATA_MEM` | `0x3C400` | 32K | TRISC1 data/BSS |
| `TRISC1_RUNTIME_ARGS` | `0x44400` | 1K | TRISC1 runtime params |

#### Wormhole vs Blackhole Differences

Wormhole uses smaller code regions and larger coverage regions:

| Region Type | Wormhole | Blackhole |
| --- | --- | --- |
| **BRISC L0 Stack** | 4K | 8K |
| **TRISC L0 Stack** | 2K | 4K |
| **TRISC GCOV_MEM** | 32K | 8K |
| **TRISC CODE** | 64K | 64K |

The larger GCOV regions on Wormhole accommodate additional coverage metadata for the architecture.

**Sources:**[tests/helpers/ld/memory.wormhole.debug.ld 34-51](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L34-L51)[tests/helpers/ld/memory.blackhole.debug.ld 34-52](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L34-L52)

#### Instruction Buffer Address

Both architectures define an instruction buffer symbol used for MOP (micro-operation) programming:

`PROVIDE(__instrn_buffer = 0xFFE40000);`
This address is used by the ckernel template system for recording and replaying instruction sequences.

### BRISC ELF Linking

`${GXX} ${ARCH_NON_COMPUTE} ${OPTIONS_ALL} ${OPTIONS_LINK} \    ${SHARED_OBJ_DIR}/tmu-crt0.o \    ${SHARED_OBJ_DIR}/brisc.o \    ${COVERAGE_DEPS} \    -T${MEMORY_LAYOUT_LD_SCRIPT} \    -T${LINKER_SCRIPTS}/brisc.ld \    -T${LINKER_SCRIPTS}/sections.ld \    -o ${SHARED_ELF_DIR}/brisc.elf`
| Component | Purpose |
| --- | --- |
| `tmu-crt0.o` | Startup assembly code |
| `brisc.o` | BRISC kernel implementation |
| `COVERAGE_DEPS` | `coverage.o -lgcov` if coverage enabled |
| `memory.*.ld` | Memory region definitions (`BRISC_CODE`, `BRISC_STACK_MEM`) |
| `brisc.ld` | BRISC entry point and section mappings |
| `sections.ld` | Common section definitions (.text, .data, .bss, .rodata) |

The `MEMORY_LAYOUT_LD_SCRIPT` variable resolves to:

*   `memory.{arch}.ld` for normal builds
*   `memory.{arch}.debug.ld` for coverage builds (via `-DCOVERAGE` flag)

This selection happens in `resolve_compile_options()`:

`if self.coverage_build == CoverageBuild.Yes:    OPTIONS_COMPILE += "-fprofile-arcs -ftest-coverage -fprofile-info-section -DCOVERAGE "    MEMORY_LAYOUT_LD_SCRIPT = f"{TestConfig.LINKER_SCRIPTS}/memory.{TestConfig.ARCH.value}.debug.ld"`

### TRISC ELF Linking

`${GXX} ${ARCH_COMPUTE} ${OPTIONS_ALL} ${OPTIONS_LINK} \    ${SHARED_OBJ_DIR}/tmu-crt0.o \    ${SHARED_OBJ_DIR}/main_${stage}.o \    ${VARIANT_OBJ_DIR}/kernel_${stage}.o \    -T${MEMORY_LAYOUT_LD_SCRIPT} \    -T${LINKER_SCRIPTS}/trisc.ld \    -T${LINKER_SCRIPTS}/sections.ld \    -o ${VARIANT_ELF_DIR}/${stage}.elf`
Each TRISC stage (unpack, math, pack) links:

1.   **Startup code**: `tmu-crt0.o` (shared)
2.   **Stage main**: `main_{stage}.o` (shared, stage-specific)
3.   **Test kernel**: `kernel_{stage}.o` (variant-specific)

The linker places sections into memory regions defined by the linker script:

*   `.text` → `TRISC{0,1,2}_CODE` (64K on both architectures)
*   `.data`, `.bss` → `TRISC{0,1,2}_LOCAL_DATA_MEM` (32K)
*   `.stack` → `TRISC{0,1,2}_STACK_MEM` (L0 memory, 2K-4K)
*   Runtime parameters → `TRISC{0,1,2}_RUNTIME_ARGS` (1K) in coverage builds

The `trisc.ld` script defines the entry point and section-to-region mappings specific to TRISC cores.

### Linker Options

The `OPTIONS_LINK` variable defines linker-specific flags applied to all ELF generation:

| Flag | Purpose |
| --- | --- |
| `-fexceptions` | Enable C++ exception handling |
| `-Wl,-z,max-page-size=16` | Set page alignment to 16 bytes (match hardware constraints) |
| `-Wl,-z,common-page-size=16` | Set common page size to 16 bytes |
| `-nostartfiles` | Don't use standard system startup files |
| `-Wl,--trace` | Print linker trace information for debugging |

The `-nostartfiles` flag is critical as the RISC-V target uses custom startup code (`tmu-crt0.S`) instead of standard C runtime initialization. This custom startup code:

*   Initializes the stack pointer from linker-defined symbols
*   Clears the BSS section
*   Calls constructors for global objects
*   Jumps to `main()` function

The 16-byte page alignment matches the hardware's memory access granularity and ensures efficient DMA transfers.

### ELF Output Organization

The build system organizes ELF files into two directory structures:

```
/tmp/tt-llk-build/
├── shared/
│   └── elf/
│       └── brisc.elf                    # Shared BRISC executable
├── shared-profiler/                      # Separate for profiler builds
│   └── elf/
│       └── brisc.elf
└── {test_name}/
    └── {variant_hash}/
        └── elf/
            ├── unpack.elf               # TRISC0 executable
            ├── math.elf                 # TRISC1 executable
            └── pack.elf                 # TRISC2 executable
```

The `variant_hash` is computed from test configuration parameters and ensures unique builds for different format/template/runtime combinations.

## Troubleshooting and Error Handling

### Common Issues and Solutions

| Issue | Symptom | Solution |
| --- | --- | --- |
| **Missing tt-smi** | `ERROR: tt-smi command failed or not found` | Install tt-smi or set `CHIP_ARCH` environment variable |
| **Header download failure** | `ERROR: Failed to download {header}` | Check internet connectivity, verify tt-metal repository accessibility |
| **SFPI hash mismatch** | `[ERROR] SFPI {filename} sha256 mismatch` | Re-download SFPI, check sfpi-version file integrity |
| **Wrong Python version** | `Error: Only Python 3.8 or 3.10 are supported` | Use Python 3.8 or 3.10, create new virtual environment |
| **Architecture detection failure** | `ERROR: Unable to determine chip architecture` | Set `CHIP_ARCH=wormhole` or `CHIP_ARCH=blackhole` manually |

### Manual Override Patterns

**Override chip architecture**:

`export CHIP_ARCH=wormhole./setup_testing_env.sh`
**Skip header download** (for Quasar or offline scenarios):

`CHIP_ARCH=quasar ./setup_testing_env.sh`
**Clean and reinstall**:

`rm -rf tests/sfpi tests/hw_specific/*/inc/.headers_downloaded./setup_testing_env.sh`
