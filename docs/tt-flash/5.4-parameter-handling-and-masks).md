# Parameter Handling and Masks

Relevant source files
*   [tt_flash/blackhole.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py)
*   [tt_flash/boot_fs.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py)
*   [tt_flash/flash.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py)

This document describes the parameter handling system in tt-flash, which allows firmware bundles to define dynamic values that are computed and written to specific SPI flash locations during the flashing process. This enables features like write-back of board configuration data, version stamping, and counter incrementing without requiring custom firmware images for each scenario.

For information about the overall flashing stages and workflow, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages). For details on firmware bundle structure, see [Firmware Bundle Format](https://deepwiki.com/tenstorrent/tt-flash/5.1-firmware-bundle-format).

## Overview

The parameter handling system operates on **mask files** (typically `mask.json`) that are bundled with firmware images. These masks define regions in the SPI flash where computed or dynamic values should be written, along with tags that specify which handler function should compute the value. The system supports different mask formats and handlers for Wormhole and Blackhole architectures.

**Sources:**[tt_flash/flash.py 33-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L148)[tt_flash/blackhole.py 49-82](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L49-L82)

## Mask File Format

### Wormhole Mask Format

For Wormhole chips, the mask file is a JSON array of parameter descriptor objects. Each object defines a contiguous region in the firmware image that requires special handling.

**Structure:**

| Field | Type | Description |
| --- | --- | --- |
| `start` | integer | Starting byte offset in the firmware image |
| `end` | integer | Ending byte offset (exclusive) in the firmware image |
| `tag` | string | Identifier for the handler function to use |

**Example:**

`[  {    "start": 4096,    "end": 4100,    "tag": "bundle_version"  },  {    "start": 8192,    "end": 8196,    "tag": "incr"  }]`
The mask is loaded and validated in [tt_flash/flash.py 341-366](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L341-L366) where the system extracts it from the firmware bundle tarfile and parses it as JSON.

**Sources:**[tt_flash/flash.py 368-424](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L368-L424)

### Blackhole Mask Format

Blackhole chips use a simplified mask format that only specifies handler tags without explicit byte ranges. This is because Blackhole uses a boot filesystem structure where handlers operate on entire filesystem objects rather than byte ranges.

**Structure:**

| Field | Type | Description |
| --- | --- | --- |
| `tag` | string | Identifier for the handler function to use |

**Example:**

`[  {    "tag": "write-boardcfg"  }]`
The Blackhole mask processing is handled by `boot_fs_write()` in [tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)

**Sources:**[tt_flash/blackhole.py 52-82](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L82)

## TAG_HANDLERS System

The parameter handling system uses a dispatch table called `TAG_HANDLERS` that maps tag strings to handler functions.

The Wormhole `TAG_HANDLERS` dictionary is defined at [tt_flash/flash.py 142-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L142-L148):

`TAG_HANDLERS: dict[str, Callable[[TTChip, bytearray, int, int, int], bytearray]] = {    "rmw": rmw_param,    "incr": incr_param,    "date": date_param,    "flash_version": flash_version,    "bundle_version": bundle_version,}`
The Blackhole `TAG_HANDLERS` dictionary is defined at [tt_flash/blackhole.py 49](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L49-L49):

`TAG_HANDLERS = {"write-boardcfg": writeback_boardcfg}`
**Sources:**[tt_flash/flash.py 142-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L142-L148)[tt_flash/blackhole.py 49](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L49-L49)

## Handler Function Signature

### Wormhole Handler Signature

Wormhole parameter handlers follow a standard signature that allows them to read from SPI flash and modify the firmware data buffer:

`def handler(    chip: TTChip,    data: bytearray,    spi_addr: int,    data_addr: int,    len: int) -> bytearray`
**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `chip` | `TTChip` | Chip object for reading SPI flash |
| `data` | `bytearray` | Firmware data buffer to modify |
| `spi_addr` | `int` | Absolute SPI flash address to read from |
| `data_addr` | `int` | Relative offset in data buffer to write to |
| `len` | `int` | Number of bytes to process |

**Returns:** Modified `bytearray` with the computed parameter written

**Sources:**[tt_flash/flash.py 33-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L124)

### Blackhole Handler Signature

Blackhole handlers operate on lists of `FlashWrite` objects:

`def handler(chip: BhChip, writes: list[FlashWrite]) -> list[FlashWrite]`
**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `chip` | `BhChip` | Blackhole chip object |
| `writes` | `list[FlashWrite]` | List of pending write operations |

**Returns:** Modified list of `FlashWrite` objects

**Sources:**[tt_flash/blackhole.py 18-82](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L18-L82)

## Wormhole Parameter Handlers

### rmw (Read-Modify-Write)

The `rmw_param` handler reads the existing value from SPI flash and preserves it in the new firmware image. This is used for preserving board-specific configuration data across firmware updates.

**Implementation:**[tt_flash/flash.py 33-42](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L42)

**Behavior:**

1.   Reads `len` bytes from SPI flash at `spi_addr`
2.   Writes the read bytes into `data` at offset `data_addr`
3.   Returns modified `data`

**Use Case:** Preserve board-specific calibration data or configuration that should not be overwritten by firmware updates.

**Sources:**[tt_flash/flash.py 33-42](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L33-L42)

### incr (Increment Counter)

The `incr_param` handler reads an integer counter from SPI flash, increments it, and writes the new value. This is useful for tracking the number of times firmware has been flashed.

**Implementation:**[tt_flash/flash.py 45-62](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L45-L62)

**Behavior:**

1.   Reads `len` bytes from SPI flash at `spi_addr`
2.   Interprets as little-endian integer
3.   Increments by 1 (or resets to 1 on overflow)
4.   Converts back to little-endian bytes
5.   Writes to `data` at offset `data_addr`

**Use Case:** Flash operation counter, boot counter, or version tracking.

**Sources:**[tt_flash/flash.py 45-62](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L45-L62)

### date (Date Stamp)

The `date_param` handler writes the current date in `0xYYYYMMDD` hexadecimal format. This provides a timestamp of when the firmware was flashed.

**Implementation:**[tt_flash/flash.py 65-74](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L65-L74)

**Behavior:**

1.   Gets current date using `date.today()`
2.   Formats as `YYYYMMDD` string
3.   Converts to integer with `0x` prefix (e.g., `0x20240315` for March 15, 2024)
4.   Converts to little-endian bytes
5.   Writes to `data` at offset `data_addr`

**Use Case:** Recording when firmware was last flashed for troubleshooting and tracking purposes.

**Sources:**[tt_flash/flash.py 65-74](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L65-L74)

### flash_version (tt-flash Version)

The `flash_version` handler embeds the version of tt-flash that performed the flashing operation into the firmware image.

**Implementation:**[tt_flash/flash.py 77-97](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L77-L97)

**Behavior:**

1.   Reads `tt_flash.__version__` string
2.   Splits version string by `.` delimiter
3.   Pads to 4 components with leading zeros if needed
4.   Truncates to first 4 components
5.   Reverses order: `[patch, minor, major, extra]`
6.   Writes as bytes to `data` at offset `data_addr`

**Format:** 4 bytes in reverse order: `[patch, minor, major, extra]`

**Use Case:** Identifying which version of tt-flash was used to flash the firmware, useful for debugging and support.

**Sources:**[tt_flash/flash.py 77-97](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L77-L97)

### bundle_version (Bundle Version)

The `bundle_version` handler embeds the firmware bundle version into the firmware image itself. This uses a global variable to store the semantic version extracted from the manifest.

**Implementation:**[tt_flash/flash.py 105-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L105-L124)

**Global State:**[tt_flash/flash.py 102](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L102-L102)

`__SEMANTIC_BUNDLE_VERSION = [0xFF, 0xFF, 0xFF, 0xFF]`
This global is set during manifest verification in [tt_flash/flash.py 694-695](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L694-L695)

**Behavior:**

1.   Reads from global `__SEMANTIC_BUNDLE_VERSION`
2.   Pads to 4 components if needed
3.   Reverses order: `[patch, minor, major, fwId]`
4.   Writes as bytes to `data` at offset `data_addr`

**Format:** 4 bytes in reverse order: `[patch, releaseId, fwId, debug]`

**Use Case:** Embedding the bundle version directly in firmware for verification and debugging.

**Sources:**[tt_flash/flash.py 100-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L100-L124)[tt_flash/flash.py 694-695](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L694-L695)

## Blackhole Parameter Handlers

### write-boardcfg (Board Configuration Write-back)

The `writeback_boardcfg` handler preserves board-specific configuration from the currently flashed firmware when writing new firmware. This is critical for Blackhole chips which use a boot filesystem structure.

**Implementation:**[tt_flash/blackhole.py 18-46](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L18-L46)

**Behavior:**

1.   Uses `boot_fs.read_tag()` to find `"boardcfg"` tag in current SPI flash
2.   Searches through pending `FlashWrite` objects to find the write containing `"boardcfg"`
3.   Extracts the board configuration data from SPI
4.   Overwrites the corresponding region in the new firmware image
5.   Verifies the write by re-reading and asserting equality

**Dependencies:** Uses the `boot_fs` module [tt_flash/boot_fs.py 135-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py#L135-L148) for reading boot filesystem structures.

**Use Case:** Preserving board-specific calibration, MAC addresses, or other configuration data that must persist across firmware updates on Blackhole chips.

**Sources:**[tt_flash/blackhole.py 18-46](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L18-L46)[tt_flash/boot_fs.py 135-148](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/boot_fs.py#L135-L148)

## Mask Processing Pipeline

The mask processing pipeline differs significantly between Wormhole and Blackhole architectures.

**Sources:**[tt_flash/flash.py 335-456](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L335-L456)[tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)

### Wormhole Processing Pipeline

The Wormhole pipeline is implemented in [tt_flash/flash.py 395-456](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L395-L456)

**Steps:**

1.   **Mask Validation**[tt_flash/flash.py 398-424](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L398-L424)

    *   Verifies each mask entry has `start`, `end`, and `tag` fields
    *   Ensures they are of correct types (int, int, str)
    *   Looks up tag in `TAG_HANDLERS` dictionary
    *   Builds list of `((start, end), handler)` tuples

2.   **Image Processing**[tt_flash/flash.py 428-454](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L428-L454)

    *   Iterates through lines in `image.bin` file
    *   Lines starting with `@` set the current address
    *   Other lines contain hex-encoded data (base16)
    *   For each data line: 
        *   Decodes hex to bytes
        *   Checks which parameter regions overlap
        *   Invokes handlers for overlapping regions
        *   Creates `FlashWrite(offset, data)` object

3.   **Split Region Error Check**[tt_flash/flash.py 445-448](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L445-L448)

    *   Raises error if a parameter region spans multiple write regions
    *   This ensures atomic parameter writes

**Sources:**[tt_flash/flash.py 395-456](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L395-L456)

### Blackhole Processing Pipeline

The Blackhole pipeline is implemented in [tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)

**Steps:**

1.   **Mask Validation**[tt_flash/blackhole.py 56-76](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L56-L76)

    *   Verifies each mask entry has `tag` field of type string
    *   Looks up tag in Blackhole-specific `TAG_HANDLERS`
    *   Builds list of handler functions

2.   **Image Parsing**[tt_flash/flash.py 377-389](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L377-L389)

    *   Same hex parsing as Wormhole
    *   Creates initial list of `FlashWrite` objects

3.   **Handler Application**[tt_flash/blackhole.py 78-80](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L78-L80)

    *   Calls `boot_fs_write()` which sequentially invokes handlers
    *   Each handler receives and returns the entire `writes` list
    *   Handlers can read, modify, or reorganize write operations

**Sources:**[tt_flash/blackhole.py 52-81](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/blackhole.py#L52-L81)[tt_flash/flash.py 374-393](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L374-L393)

## Parameter Write Sequence

After mask processing completes, the parameter-modified data is written to SPI flash as part of the standard write/verify cycle.

The parameter values computed by handlers are embedded in the `FlashWrite` objects returned from `flash_chip_stage1()`[tt_flash/flash.py 487-492](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L487-L492) These writes are then performed in `flash_chip_stage2()`[tt_flash/flash.py 495-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L495-L650) using the standard write/verify cycle documented in [Write and Verify Cycle](https://deepwiki.com/tenstorrent/tt-flash/5.5-write-and-verify-cycle).

**Key Points:**

*   Parameters are computed during Stage 1 (preparation)
*   The computed values are part of the data written in Stage 2
*   No special handling is needed during write/verify - parameters are just bytes in the data stream
*   SIGINT protection applies to parameter writes just like all other data

**Sources:**[tt_flash/flash.py 194-493](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L194-L493)[tt_flash/flash.py 495-650](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L495-L650)

## Error Handling

### Invalid Mask Format

If the mask format is invalid, errors are raised during validation:

*   **Missing or wrong-type fields:**[tt_flash/flash.py 403-410](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L403-L410)

    *   Raises `TTError` with descriptive message about expected format

*   **Unknown tag:**[tt_flash/flash.py 412-424](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L412-L424)

    *   Lists available tags in error message
    *   Prevents flashing with unrecognized handlers

### Split Region Error

For Wormhole chips, if a parameter region spans multiple writeable regions in the image file, this is detected and rejected at [tt_flash/flash.py 445-448](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L445-L448):

`raise TTError(    f"A parameter write ({start}:{end}) splits a writeable region "    f"({curr_addr}:{curr_stop}) in {boardname_to_display}! This is not supported.")`
This ensures that parameter writes are atomic and don't create inconsistent states.

### Handler Exceptions

If a handler function raises an exception during execution, it propagates up and causes the flash operation to fail. This is intentional - corrupt or missing parameter data should prevent flashing rather than silently produce invalid firmware.

**Sources:**[tt_flash/flash.py 403-448](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/flash.py#L403-L448)
