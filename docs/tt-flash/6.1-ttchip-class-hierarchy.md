# TTChip Class Hierarchy

Relevant source files
*   [tt_flash/chip.py](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py)

## Purpose and Scope

This document describes the class hierarchy used to abstract Tenstorrent chip hardware in tt-flash. The `TTChip` abstract base class and its concrete implementations (`WhChip`, `BhChip`) provide a unified interface for hardware operations while encapsulating chip-specific behaviors.

For information about chip detection and enumeration, see [Chip Detection](https://deepwiki.com/tenstorrent/tt-flash/6.2-chip-detection). For documentation of the underlying hardware communication methods (AXI, SPI, ARC messaging), see [Hardware Communication Interfaces](https://deepwiki.com/tenstorrent/tt-flash/6.3-hardware-communication-interfaces).

**Sources:**[tt_flash/chip.py 1-388](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L1-L388)

* * *

## Class Hierarchy Overview

The class hierarchy uses composition to wrap pyluwen's `PciChip` class, providing a higher-level abstraction with chip-specific implementations. `TTChip` serves as an abstract base class defining common operations, while `WhChip` and `BhChip` provide concrete implementations tailored to their respective hardware platforms.

**Sources:**[tt_flash/chip.py 113-306](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L113-L306)

* * *

## TTChip Abstract Base Class

### Initialization and Configuration

`TTChip` is initialized with a `PciChip` instance from pyluwen and stores the PCI interface ID for re-initialization [tt_flash/chip.py 114-118](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L114-L118) The constructor also loads chip-specific firmware definitions from `data/{prefix}/fw_defines.yaml` using `init_fw_defines()`[tt_flash/chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L109-L110)

| Attribute | Type | Purpose |
| --- | --- | --- |
| `luwen_chip` | `PciChip` | Wrapped pyluwen chip instance |
| `interface_id` | `int` | PCI interface identifier |
| `fw_defines` | `dict` | Firmware message type definitions from YAML |
| `telmetry_cache` | `Optional[Telemetry]` | Cached telemetry data |

**Sources:**[tt_flash/chip.py 113-124](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L113-L124)

### Re-initialization Support

The `reinit()` method recreates the `PciChip` connection and clears the telemetry cache [tt_flash/chip.py 122-161](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L122-L161) This is critical after chip resets during flashing operations. The method accepts an optional callback parameter for progress reporting during chip initialization:

The internal `chip_detect_callback()` provides terminal-based progress updates, displaying chip counts and status strings with ANSI escape sequences for dynamic updating [tt_flash/chip.py 130-161](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L130-L161)

**Sources:**[tt_flash/chip.py 122-161](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L122-L161)

### Common Hardware Interfaces

`TTChip` provides pass-through methods to the underlying `PciChip`, standardizing the interface across chip types:

| Method | Purpose | Signature |
| --- | --- | --- |
| `axi_read32()` | Read 32-bit value from AXI address | `(addr: int) -> int` |
| `axi_write32()` | Write 32-bit value to AXI address | `(addr: int, value: int)` |
| `axi_read()` | Read byte array from AXI address | `(addr: int, size: int) -> bytes` |
| `axi_write()` | Write byte array to AXI address | `(addr: int, data: bytes)` |
| `spi_read()` | Read from SPI flash | `(addr: int, size: int) -> bytes` |
| `spi_write()` | Write to SPI flash | `(addr: int, data: bytes)` |
| `arc_msg()` | Send ARC processor message | `(*args, **kwargs)` |
| `board_type()` | Get board type identifier | `() -> int` |

All AXI and SPI methods delegate directly to the wrapped `luwen_chip` instance [tt_flash/chip.py 205-230](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L205-L230)

**Sources:**[tt_flash/chip.py 202-230](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L202-L230)

### Telemetry Access

Two methods provide access to chip telemetry data:

*   **`get_telemetry()`**: Fetches fresh telemetry from the chip and updates the cache [tt_flash/chip.py 163-165](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L163-L165)
*   **`get_telemetry_unchanged()`**: Returns cached telemetry if available, otherwise fetches fresh data [tt_flash/chip.py 167-171](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L167-L171)

The cached approach improves performance when telemetry is accessed multiple times without intervening chip state changes.

**Sources:**[tt_flash/chip.py 163-171](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L163-L171)

### Firmware Version Extraction

The base class provides version extraction methods that read from telemetry and decode 32-bit version numbers into 4-tuple format `(component, major, minor, patch)`:

```
Version Encoding (32 bits):
┌─────────┬─────────┬─────────┬─────────┐
│ [31-24] │ [23-16] │ [15-8]  │ [7-0]   │
│ Comp    │ Major   │ Minor   │ Patch   │
└─────────┴─────────┴─────────┴─────────┘
```

Available version accessors [tt_flash/chip.py 181-191](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L181-L191):

*   `m3_fw_app_version()` - M3 application firmware version
*   `smbus_fw_version()` - SMBus firmware version (ARC1)
*   `arc_l2_fw_version()` - ARC L2 firmware version (ARC0)

**Sources:**[tt_flash/chip.py 173-191](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L173-L191)

### ASIC Location (P300 Support)

The `get_asic_location()` method returns the physical location of the ASIC on multi-chip boards like P300, where `0` indicates left (L) and `1` indicates right (R) [tt_flash/chip.py 193-200](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L193-L200) This is retrieved from telemetry.

**Sources:**[tt_flash/chip.py 193-200](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L193-L200)

### Abstract Methods

Subclasses must implement two abstract methods:

| Method | Purpose | Return Type |
| --- | --- | --- |
| `min_fw_version()` | Minimum firmware version required | `int` |
| `get_bundle_version()` | Retrieve running and SPI bundle versions | `FwVersion` |

These methods are declared abstract using the `@abstractmethod` decorator [tt_flash/chip.py 232-238](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L232-L238)

**Sources:**[tt_flash/chip.py 232-238](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L232-L238)

## WhChip: Wormhole Implementation

### Overview

`WhChip` implements `TTChip` for Wormhole architecture chips [tt_flash/chip.py 297-306](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L297-L306) The string representation identifies chips as `"Wormhole[{interface_id}]"`[tt_flash/chip.py 301-302](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L301-L302)

**Sources:**[tt_flash/chip.py 297-306](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L297-L306)

### Minimum Firmware Version

Wormhole requires firmware version `0x2170000` (version 33.23.0.0) or higher [tt_flash/chip.py 298-299](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L298-L299) This represents the minimum version that supports firmware bundles.

**Sources:**[tt_flash/chip.py 298-299](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L298-L299)

### Bundle Version Retrieval

`WhChip.get_bundle_version()` delegates to the legacy `get_bundle_version_v1()` function [tt_flash/chip.py 304-305](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L304-L305) This function uses ARC messages with the `MSG_TYPE_FW_VERSION` message type to query both running and SPI firmware versions.

#### Version Query Sequence

The function queries three different argument configurations:

1.   **arg0=0**: Base firmware version check [tt_flash/chip.py 42-44](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L42-L44)
2.   **arg0=1**: Running bundle version from RAM [tt_flash/chip.py 50-62](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L50-L62)
3.   **arg0=2**: SPI bundle version from flash [tt_flash/chip.py 70-82](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L70-L82)

Versions `0xFFFFFFFF` and `0xDEAD` are treated as error/invalid responses [tt_flash/chip.py 57-77](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L57-L77)

**Sources:**[tt_flash/chip.py 29-305](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L29-L305)

## BhChip: Blackhole Implementation

### Overview

`BhChip` implements `TTChip` for Blackhole architecture chips [tt_flash/chip.py 241-295](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L241-L295) The string representation identifies chips as `"Blackhole[{interface_id}]"`[tt_flash/chip.py 245-246](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L245-L246)

**Sources:**[tt_flash/chip.py 241-295](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L241-L295)

### Minimum Firmware Version

Blackhole chips have a minimum firmware version of `0x0`[tt_flash/chip.py 242-243](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L242-L243) indicating broader firmware compatibility compared to Wormhole.

**Sources:**[tt_flash/chip.py 242-243](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L242-L243)

### Bundle Version Retrieval

`BhChip` uses a more modern approach leveraging telemetry and boot filesystem structures [tt_flash/chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L248-L275):

#### Running Version (Telemetry)

The running firmware bundle version is read directly from telemetry's `fw_bundle_version` field [tt_flash/chip.py 254-260](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L254-L260):

#### SPI Version (Boot Filesystem)

The SPI flash version is retrieved by decoding the boot filesystem table entry `"cmfwcfg"`[tt_flash/chip.py 263-269](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L263-L269):

This method is more efficient than Wormhole's ARC message approach, utilizing the structured boot filesystem.

**Sources:**[tt_flash/chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L248-L275)

### Enhanced ASIC Location Support

`BhChip` overrides `get_asic_location()` with fallback logic for P300 boards [tt_flash/chip.py 277-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L277-L294) If telemetry retrieval fails, it reads the GPIO strap register directly:

| Register | Address | Purpose |
| --- | --- | --- |
| `GPIO_STRAP_REG_L` | `0x80030D20` | Records GPIO input states [0:31] at boot |

The ASIC location is determined from bit 6: if GPIO6 is high, the chip is on the left ASIC [tt_flash/chip.py 291-292](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L291-L292)

**Sources:**[tt_flash/chip.py 277-294](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L277-L294)

## FwVersion Dataclass

### Structure

`FwVersion` is a dataclass that encapsulates firmware version query results [tt_flash/chip.py 21-26](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L21-L26):

| Field | Type | Purpose |
| --- | --- | --- |
| `allow_exception` | `bool` | Whether exceptions during retrieval are acceptable |
| `exception` | `Optional[Exception]` | Captured exception if version query failed |
| `running` | `Optional[tuple[int, int, int, int]]` | Running firmware version (component, major, minor, patch) |
| `spi` | `Optional[tuple[int, int, int, int]]` | SPI flash firmware version (component, major, minor, patch) |

### Usage Pattern

Both chip implementations return `FwVersion` instances with `allow_exception=True`, allowing the flashing engine to decide how to handle version retrieval failures [tt_flash/chip.py 86-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L86-L275):

This design pattern enables graceful degradation when firmware is in an intermediate or corrupted state.

**Sources:**[tt_flash/chip.py 21-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L21-L275)

## Supporting Functions

### get_chip_data()

Loads chip-specific data files from the package data directories [tt_flash/chip.py 94-106](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L94-L106):

```
data/
├── wormhole/
│   └── fw_defines.yaml
└── blackhole/
    └── fw_defines.yaml
```

The function determines the chip type via `isinstance()` checks and constructs the appropriate path prefix (`"data/wormhole"` or `"data/blackhole"`). An `internal` parameter allows loading from `.ignored/` directories for development [tt_flash/chip.py 102-105](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L102-L105)

**Sources:**[tt_flash/chip.py 94-107](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L94-L107)

### init_fw_defines()

Loads and parses the chip-specific `fw_defines.yaml` file using `yaml.safe_load()`[tt_flash/chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L109-L110) This YAML file contains firmware message type definitions used in ARC communication (see [Firmware Message Types](https://deepwiki.com/tenstorrent/tt-flash/6.4-firmware-message-types)).

**Sources:**[tt_flash/chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L109-L110)

## Chip Factory Functions

### detect_local_chips()

The primary detection function for flashing operations [tt_flash/chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L308-L374) Key parameters:

*   `ignore_ethernet`: If `True`, enables `noc_safe` mode to avoid Ethernet-connected remote chips [tt_flash/chip.py 355](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L355-L355)

The function uses `luwen_detect_chips_fallible()` with a progress callback [tt_flash/chip.py 351-356](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L351-L356) Communication verification ensures that only accessible chips are returned [tt_flash/chip.py 357-360](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L357-L360) The `force_upgrade()` call promotes pyluwen's generic chip objects to specific Wormhole/Blackhole types [tt_flash/chip.py 362](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L362-L362)

**Sources:**[tt_flash/chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L308-L374)

### detect_chips()

A simplified detection function without progress callbacks or fallibility handling [tt_flash/chip.py 377-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L377-L387):

Used for verification operations where detailed progress reporting is unnecessary.

**Sources:**[tt_flash/chip.py 377-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/tt_flash/chip.py#L377-L387)

## Platform-Specific Behavior Summary

| Aspect | WhChip (Wormhole) | BhChip (Blackhole) |
| --- | --- | --- |
| **Min FW Version** | `0x2170000` (33.23.0.0) | `0x0` (any version) |
| **Version Retrieval** | ARC messages with 3 queries | Telemetry + boot filesystem |
| **SPI Version Source** | `MSG_TYPE_FW_VERSION` arg0=2 | `cmfwcfg` boot table entry |
| **ASIC Location** | Via telemetry only | Telemetry with GPIO fallback |
| **String Repr** | `"Wormhole[{id}]"` | `"Blackhole[{id}]"` |
| **Data Directory** | `data/wormhole/` | `data/blackhole/` |
