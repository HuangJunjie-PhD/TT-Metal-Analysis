## Purpose and Scope

The Tensix Coprocessor Frontend is the instruction ingestion and expansion system that sits between the Baby RISC-V cores and the Tensix backend execution units. This document describes the three-thread pipeline architecture, the macro-operation (MOP) expansion mechanism, the Replay buffering system, the Wait Gate for instruction scheduling, and the Auto TTSync feature introduced in Blackhole. The frontend's primary role is to allow RISC-V cores to push instructions at rates less than one per cycle while ensuring the backend receives one instruction per cycle per thread.

For information about the backend execution units that consume these instructions, see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system), [Matrix Unit](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.3-unpacker-system-(unpacr)), [Vector Unit](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.4-matrix-unit-(fpu)-architecture), [Unpacker System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.5-vector-unit-(sfpu)-architecture), [Packer System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.6-packer-system-(pacr)), and [Scalar Unit, Mover, and TDMA-RISC](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.8-soft-reset-and-system-control). For details about the RISC-V cores that produce instructions, see [Baby RISC-V Cores](https://deepwiki.com/tenstorrent/tt-isa-documentation/3.1-baby-risc-v-cores).

## Frontend Architecture Overview

The Tensix Coprocessor Frontend consists of three independent instruction pipelines, one for each Tensix thread (T0, T1, T2). Each pipeline includes instruction expansion hardware (MOP Expander and Replay Expander), buffering (29-32× 32-bit FIFO), conditional gating (Wait Gate), and in Blackhole, automatic synchronization (Auto TTSync).

## Instruction Sources and Routing

Instructions enter the Tensix Frontend from two types of sources:

### Direct Push from RISC-V T0/T1/T2

Each RISC-V T_i_ core can push instructions directly into the corresponding Tensix thread T_i_'s pipeline. RISC-V cores use custom `.ttinsn` pseudo-instructions to push 32-bit instruction words into the frontend. This is the primary mechanism for feeding instructions during normal operation.

### Broadcast Push from RISC-V B via PCBufs

RISC-V B (the master controller) can push instructions that are broadcast to all three thread pipelines simultaneously via the PCBuf mechanism. PCBufs are instruction FIFOs that sit before the MOP Expander stage. When RISC-V B pushes an instruction, it appears at the input to all three MOP Expanders.

**Key behavioral note:** When a RISC-V core enters soft reset, any instructions in its PCBuf or MOP Expander are discarded. Upon leaving soft reset, PCBufs are cleared. This is documented in [WormholeB0/TensixTile/SoftReset.md 37-41](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L37-L41)

## Pipeline Stages

Each thread pipeline processes instructions through the following stages:

### Stage 1: MOP Expansion

The **MOP Expander** is the first stage of each pipeline. It handles `MOP` and `MOP_CFG` instructions, expanding a single `MOP` instruction into up to 32,639 micro-instructions using two templates:

*   **Template 0:** Mask-based expansion with conditional execution
*   **Template 1:** Nested loop expansion with start/end/loop operations

All other instruction opcodes pass through unchanged. The MOP Expander configuration is mapped into each RISC-V T_i_'s address space at `TENSIX_MOP_CFG_BASE` as a write-only `uint32_t[9]` array. See the [MOP Expander section](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/MOP%20Expander%20section) below for detailed behavior.

### Stage 2: Replay Expansion

The **Replay Expander** sits after the MOP Expander and handles `REPLAY` instructions. It can buffer and replay up to 256 instructions from a 32× 32-bit buffer. Unlike the MOP Expander which is typically used for repetitive loops, the Replay Expander is designed for linear non-looping sequences. Instructions expanded by the MOP Expander can include `REPLAY` instructions, which will then be further expanded by the Replay Expander.

### Stage 3: Instruction FIFO Buffering

After expansion, instructions are buffered in a **29-32× 32-bit FIFO** (depth varies by architecture). This FIFO decouples the expansion rate from the dispatch rate and provides backpressure to upstream stages when the backend cannot accept instructions quickly enough.

### Stage 4: Wait Gate and Conditional Dispatch

The **Wait Gate** is the final stage before dispatch to backend execution units. Each instruction can optionally specify a `STALLWAIT` condition that must be satisfied before the instruction is dispatched. This allows fine-grained control over instruction scheduling based on backend resource availability, synchronization primitives, or other system state.

In **Blackhole**, the Wait Gate also includes **Auto TTSync** functionality, which automatically inserts synchronization between threads to prevent data hazards.

## MOP Expander (Macro-Operation Expansion)

The MOP Expander allows software to compress repetitive instruction sequences, reducing instruction fetch overhead and freeing RISC-V cores for other tasks. Each thread has an independent MOP Expander.

### Configuration Mapping

Configuration for each thread's MOP Expander is accessed by the corresponding RISC-V T_i_ core:

| RISC-V Core | Address | Maps to |
| --- | --- | --- |
| RISC-V T0 | `TENSIX_MOP_CFG_BASE` | Thread 0 MOP Expander |
| RISC-V T1 | `TENSIX_MOP_CFG_BASE` | Thread 1 MOP Expander |
| RISC-V T2 | `TENSIX_MOP_CFG_BASE` | Thread 2 MOP Expander |

The configuration space is **write-only**. Reading triggers undefined behavior.

### Template 0: Mask-Based Expansion

Template 0 uses a 32-bit mask (from `MOP_CFG` high 16 bits + `MOP` low 16 bits) to conditionally execute instructions. The functional model from [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 32-56](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L32-L56):

**Configuration fields for Template 0:**

| `MopCfg` Index | Usage |
| --- | --- |
| `MopCfg[0]` | Unused |
| `MopCfg[1]` | `Flags`: bit 0 = `HasB`, bit 1 = `HasA123` |
| `MopCfg[2]` | `InsnB` (used if `HasB`) |
| `MopCfg[3]` | `InsnA0` (always used) |
| `MopCfg[4]` | `InsnA1` (used if `HasA123`) |
| `MopCfg[5]` | `InsnA2` (used if `HasA123`) |
| `MopCfg[6]` | `InsnA3` (used if `HasA123`) |
| `MopCfg[7]` | `SkipA0` (used when mask bit is 1) |
| `MopCfg[8]` | `SkipB` (used if `HasB` and mask bit is 1) |

### Template 1: Nested Loop Expansion

Template 1 generates nested loops with start/end operations and alternating loop bodies. The functional model from [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 58-94](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L58-L94):

**Configuration fields for Template 1:**

| `MopCfg` Index | Usage |
| --- | --- |
| `MopCfg[0]` | `OuterCount` (low 7 bits) |
| `MopCfg[1]` | `InnerCount` (low 7 bits) |
| `MopCfg[2]` | `StartOp` (skipped if NOP) |
| `MopCfg[3]` | `EndOp0` (skipped if NOP) |
| `MopCfg[4]` | `EndOp1` (skipped if NOP or if `EndOp0` is NOP) |
| `MopCfg[5]` | `LoopOp` (main loop body instruction) |
| `MopCfg[6]` | `LoopOp1` (alternate loop body; if not NOP, doubles inner count) |
| `MopCfg[7]` | `Loop0Last` (override last inner iteration when last outer iteration) |
| `MopCfg[8]` | `Loop1Last` (override last inner iteration when not last outer iteration) |

### MOP Expander Performance and Synchronization

**Throughput:** The MOP Expander can emit one instruction per cycle during expansion. Control flow resolution (loops, conditionals) is effectively free. After expansion completes, if the next instruction is not a `MOP`, there is a **one-cycle transition penalty** during which no instructions are ingested or emitted.

**Synchronization:** Software must not modify `MopCfg` while an expansion is in progress. RISCV TTSync should be used to wait for expansions to finish before writing new configuration. The functional model shows `MopCfg` being sampled at the start of expansion, but actual hardware samples values during the expansion process, so mid-expansion modification causes undefined behavior.

**Configuration Access:** As noted in [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 98-112](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L98-L112) each RISC-V T_i_ core accesses its thread's `MopCfg` at `TENSIX_MOP_CFG_BASE`, which is mapped as `uint32_t[9]` but is **write-only**.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 112-121](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L112-L121)

## Replay Expander

The Replay Expander provides buffering and replay of linear instruction sequences. It sits after the MOP Expander in each thread pipeline and can store up to 256 instructions in a 32× 32-bit buffer. The Replay Expander is configured and controlled in-band using the `REPLAY` instruction.

**Key differences from MOP Expander:**

*   MOP Expander: Used for repetitive looping sequences
*   Replay Expander: Used for linear non-looping sequences that need to be executed multiple times

**Nesting:** MOP expansions can include `REPLAY` instructions, which will then be expanded by the Replay Expander. However, `REPLAY` expansions cannot contain `MOP` instructions (the MOP Expander sits before the Replay Expander in the pipeline).

**Configuration:** The Replay Expander is configured in-band using modes of the `REPLAY` instruction. See the Instruction Set Reference for `REPLAY` for detailed configuration options.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md 4-5](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOPExpander.md?plain=1#L4-L5)[WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md 78](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/BackendConfiguration.md?plain=1#L78-L78)

## Wait Gate and Instruction Scheduling

The Wait Gate is the final stage before instructions are dispatched to backend execution units. It implements conditional execution gating based on `STALLWAIT` conditions specified in each instruction.

### STALLWAIT Conditions

Instructions can specify conditions that must be satisfied before dispatch. Common conditions include:

*   Backend resource availability (e.g., C12 for mover idle, C13 for configuration synchronization)
*   Synchronization primitives (semaphores, mutexes)
*   Cross-thread coordination

### Block Masks

Each instruction can specify a block mask indicating which execution units it will use. The Wait Gate uses this information along with condition checks to determine when an instruction can safely execute.

### Dispatch Rate

When conditions are satisfied, the Wait Gate dispatches one instruction per cycle per thread to the appropriate backend execution units. The three threads operate independently, so the aggregate dispatch rate is up to three instructions per cycle across all threads.

## Auto TTSync (Blackhole)

**Blackhole introduces Auto TTSync**, an automatic synchronization mechanism in the Wait Gate that prevents data hazards between threads. When enabled, the hardware automatically inserts synchronization between threads based on dependencies inferred from instruction types and resource usage.

This feature reduces the burden on software to explicitly manage cross-thread synchronization using TTSync primitives, though explicit synchronization is still available and recommended for complex cases.

**Auto TTSync behavior:**

*   Automatically detects potential data hazards (e.g., T0 writing to resources that T1 will read)
*   Inserts implicit synchronization waits at the Wait Gate before dispatching dependent instructions
*   Operates transparently to software unless explicitly disabled

**Availability:** Auto TTSync is Blackhole-specific and not present in Wormhole B0.

## Thread-Specific Pipeline Configuration

Each of the three thread pipelines operates independently with thread-specific configuration:

### ThreadConfig and StateID

Backend configuration includes thread-specific state accessed via `ThreadConfig[CurrentThread]`, where `CurrentThread` is implicitly T0 for thread 0, T1 for thread 1, and T2 for thread 2. The `CFG_STATE_ID_StateID` field in `ThreadConfig` selects between two configuration banks (0 or 1) for certain thread-agnostic configuration.

### MOP Expander Configuration

As detailed above, each thread's MOP Expander has independent configuration accessible at `TENSIX_MOP_CFG_BASE` from the corresponding RISC-V T_i_ core.

### Replay Expander Configuration

Each thread has an independent Replay Expander configured via in-band `REPLAY` instructions pushed to that thread.

### Independent Operation

The three threads can execute completely different instruction sequences at different rates. One thread can be expanding a `MOP` while another is replaying a `REPLAY` sequence and the third is directly executing instructions without expansion.

## FIFO Depth and Buffering

Each thread pipeline includes a **29-32× 32-bit instruction FIFO** between the Replay Expander and the Wait Gate. The exact depth varies:

*   **Wormhole B0:** 29× 32-bit entries
*   **Blackhole A0:** 32× 32-bit entries (increased depth)

This buffering provides:

*   **Decoupling:** Expansion rate can differ from dispatch rate
*   **Backpressure:** When the backend is stalled or the Wait Gate is blocking, instructions accumulate in the FIFO
*   **Smoothing:** Transient rate mismatches (e.g., MOP transition penalty) don't stall the backend if the FIFO has space

**Overflow behavior:** If software pushes instructions faster than the backend can consume them, the FIFO will fill. Once full, the Replay Expander will stall, which will stall the MOP Expander, which will eventually backpressure the RISC-V core attempting to push instructions.

## Performance Characteristics

### Throughput

**Per-thread dispatch:** 1 instruction per cycle (when not stalled) **Aggregate dispatch:** Up to 3 instructions per cycle (one per thread)

**MOP Expansion:** 1 instruction emitted per cycle during expansion, with 1-cycle transition penalty after expansion completes

**Replay Expansion:** 1 instruction emitted per cycle during replay

### Latency

**Minimum ingestion-to-dispatch latency:** Several cycles through the pipeline stages (MOP Expander → Replay Expander → FIFO → Wait Gate)

**Wait Gate stall latency:** Variable, depends on when `STALLWAIT` conditions are satisfied

**PCBuf latency:** Instructions from RISC-V B via PCBufs add latency compared to direct push from RISC-V T_i_

### Optimization Strategies

1.   **Use MOP/REPLAY for repetitive sequences:** Reduces instruction push overhead, frees RISC-V cores
2.   **Ensure MOP expansions include at least one REPLAY:** Hides the 1-cycle transition penalty
3.   **Pre-configure MOP before use:** Use TTSync to ensure configuration is stable before pushing `MOP` instruction
4.   **Monitor FIFO depth:** Avoid overflow by balancing push rate with backend consumption rate
5.   **Use block masks and STALLWAIT wisely:** Minimize unnecessary stalls by accurately specifying resource dependencies

## Soft Reset Behavior

When RISC-V T_i_ enters soft reset:

*   Any instructions in the T_i_ MOP Expander or PCBuf are **discarded**
*   The instruction FIFO is likely **cleared** (implementation-dependent)
*   Upon leaving soft reset, PCBufs from RISC-V B to RISC-V T_i_ are **cleared**

This behavior is critical for reset sequences and debug scenarios. Software must re-initialize MOP Expander configuration and re-push instructions after taking a RISC-V T_i_ core out of reset.

See [WormholeB0/TensixTile/SoftReset.md 37-41](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L37-L41) for detailed soft reset behavior of RISC-V cores and their interaction with the frontend.

## Summary

The Tensix Coprocessor Frontend provides a sophisticated instruction pipeline architecture with three independent thread pipelines, each capable of macro-operation expansion, replay buffering, and conditional gating. Key design points:

*   **Three-thread parallelism:** Independent pipelines allow T0/T1/T2 to execute different instruction sequences concurrently
*   **Instruction expansion:** MOP and Replay expanders amortize instruction fetch overhead, achieving 1 instruction/cycle dispatch with <1 instruction/cycle push
*   **Flexible routing:** Instructions can come directly from RISC-V T_i_ or broadcast from RISC-V B via PCBufs
*   **Conditional execution:** Wait Gate implements fine-grained control over instruction dispatch based on backend state
*   **Blackhole enhancement:** Auto TTSync automatically prevents data hazards between threads

The frontend's design enables efficient orchestration of the Tensix backend execution units while minimizing instruction fetch bandwidth requirements on the Baby RISC-V cores.
