## Purpose and Scope

The Instruction Catalog System provides a comprehensive database of RISC-V instructions with filtering, extension management, and instruction instantiation capabilities. It manages the complete lifecycle from frozen instruction templates (`InstructionDef`) to mutable runtime instances (`Instruction`), supporting ISA-aware filtering based on extensions, operand types, and instruction categories.

For information about using instructions in test scenarios, see [Test Step System](https://deepwiki.com/tenstorrent/riscv-coretp/3.3-test-step-system). For details on creating executable code sequences, see [Memory Operations](https://deepwiki.com/tenstorrent/riscv-coretp/5.1-memory-operations).

## Instruction Definition Layer

### InstructionDef - Frozen Templates

`InstructionDef` is a frozen dataclass that serves as an immutable template for RISC-V instructions. Each definition specifies the instruction's name, required extensions, register width compatibility, functional category, and operand structure.

| Field | Type | Purpose |
| --- | --- | --- |
| `name` | `str` | Instruction mnemonic (e.g., "add", "sha256sig0") |
| `extension` | `Extension` | Required RISC-V extension(s) as flag enum |
| `xlen` | `Xlen` | Register width compatibility (XLEN32, XLEN64, or both) |
| `category` | `Category` | Functional classification (ARITHMETIC, LOGIC, etc.) |
| `destination` | `Optional[OperandSlot]` | Destination register slot specification |
| `source` | `list[OperandSlot]` | Source operand slot specifications |
| `formatter` | `str` | Assembly format string with `{name}` placeholders |

**Sources:**[coretp/isa/instruction.py 148-162](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instruction.py#L148-L162)

## Extension Hierarchy and Implication

The catalog automatically resolves extension dependencies through the `IMPLIED_EXTENSIONS` dictionary. When an ISA includes a composite extension like `G` or `ZKN`, all required sub-extensions are automatically added.

### Extension Implication Flow

## InstructionCatalog - ISA-Aware Filtering

The `InstructionCatalog` class provides the primary interface for querying instructions compatible with a specific ISA configuration.

### Initialization and Filtering Process

**Sources:**[coretp/isa/catalog.py 42-60](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L42-L60)

The constructor performs three key steps:

1.   **ISA Parsing**: Converts string ISA specifications to `RvArch` objects
2.   **Extension Expansion**: Resolves all implied extensions using `expand_implied_extensions()`
3.   **Initial Filtering**: Removes instructions incompatible with the ISA's xlen or missing required extensions

The compatibility check [coretp/isa/catalog.py 50-57](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L50-L57) ensures every extension bit required by an instruction is present in the ISA:

### Advanced Filtering API

The `filter()` method [coretp/isa/catalog.py 68-116](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L68-L116) supports fine-grained instruction queries:

| Parameter | Type | Description |
| --- | --- | --- |
| `extension` | `Optional[Extension]` | Match any instruction belonging to extension(s) |
| `category` | `Optional[Category]` | Match functional category (ARITHMETIC, LOGIC, etc.) |
| `has_destination` | `Optional[bool]` | Filter by presence of destination register |
| `source_reg_count` | `Optional[int]` | Minimum number of source register operands |
| `has_immediate` | `Optional[bool]` | Filter instructions with/without immediates |
| `destination_type` | `Optional[OperandType]` | Filter by destination type (GPR, FPR, VEC) |
| `source_type` | `Optional[OperandType]` | Filter by source operand type |
| `exclude_extensions` | `Optional[Extension]` | Exclude specific extensions |
| `xlen` | `Optional[Xlen]` | Filter by register width |

Each parameter generates a predicate function that is applied to the instruction list. The method returns a list of built `Instruction` instances, not `InstructionDef` templates.

**Sources:**[coretp/isa/catalog.py 68-116](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/catalog.py#L68-L116)

## Runtime Instruction Layer

### Instruction - Mutable Instances

Unlike frozen `InstructionDef` templates, `Instruction` objects are mutable and represent concrete instruction instances with assigned operand values.

**Key Differences from InstructionDef:**

| Aspect | InstructionDef | Instruction |
| --- | --- | --- |
| Mutability | Frozen dataclass | Mutable dataclass |
| Operands | `OperandSlot` (type only) | `Operand` (type + value) |
| Purpose | Template/definition | Runtime instance |
| Storage | `ALL_INSTRS` collection | Test scenario steps |
| ID | None | `instruction_id` field |

### Assembly Formatting

The `format()` method [coretp/isa/instruction.py 27-46](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instruction.py#L27-L46) generates RISC-V assembly strings using the formatter template:

*   Collects all operands (source + destination)
*   Formats integer offsets as hex (positive) or decimal (negative)
*   Substitutes operand values into formatter string using Python's `str.format()`

Example:

```
InstructionDef(formatter="andn {rd}, {rs1}, {rs2}")
→ Instruction with rd=x5, rs1=x10, rs2=x15
→ format() returns "andn x5, x10, x15"
```

## Label - Special Instruction Type

`Label` extends `Instruction` to represent assembly labels. It uses the `SYMBOL` operand type for the destination and formats as `{name}:`.

**Special Properties:**

*   `instruction_pointer`: Boolean flag indicating if label points to specific instruction
*   `label_name()`: Returns the symbol string from destination operand

**Sources:**[coretp/isa/instruction.py 116-146](https://github.com/tenstorrent/riscv-coretp/blob/e4fb9eea/coretp/isa/instruction.py#L116-L146)

## Summary

The Instruction Catalog System provides:

1.   **Immutable Templates**: `InstructionDef` frozen dataclasses for all RISC-V instructions
2.   **Extension Management**: Automatic resolution of implied extensions (G→I+M+A+F+D)
3.   **ISA-Aware Filtering**: `InstructionCatalog` removes incompatible instructions based on xlen and extensions
4.   **Flexible Queries**: Multi-parameter filtering by extension, category, operand types
5.   **Runtime Instances**: `Instruction` mutable objects with operand values for test execution
6.   **Assembly Generation**: Format strings produce RISC-V assembly from operand values

The system bridges the gap between abstract instruction specifications and concrete test scenarios, enabling ISA-compliant test generation while maintaining extensibility for new RISC-V extensions.
