## Purpose and Scope

This page documents the multi-stage compilation pipeline that transforms a PyTorch model into executable TTNN operations on Tenstorrent hardware. It covers the complete transformation sequence: **FX Graph → Torch MLIR → StableHLO → TTIR → TTNN → Flatbuffer**.

For information about how to configure this pipeline's behavior, see [Configuration and Control](https://deepwiki.com/tenstorrent/tt-torch/3.1.3-configuration-and-control). For details on operation-by-operation compilation as an alternative to full compilation, see [Op-by-Op Compilation](https://deepwiki.com/tenstorrent/tt-torch/3.1.2-op-by-op-compilation). For backend integration and pass execution, see [Backend Integration](https://deepwiki.com/tenstorrent/tt-torch/3.2-backend-integration).

## Stage 1: PyTorch to FX Graph

### Entry Point

When a model is compiled with `torch.compile(model, backend="tt-legacy")`, PyTorch's Dynamo captures the model execution as a graph representation.

Sources: [tt_torch/dynamo/backend.py 312-369](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L312-L369)

### Processing

The captured FX graph undergoes several optimization passes before MLIR conversion:

| Pass | Function | Purpose |
| --- | --- | --- |
| Constant Folding | `constant_fold()` | Evaluates constant expressions at compile time |
| Shape Propagation | `run_shape_prop()` | Propagates tensor shapes through graph |
| Redundant Cast Bypass | `bypass_redundant_cast()` | Removes unnecessary dtype conversions |
| Dtype Promotion Bypass | `bypass_dtype_promotion()` | Prevents unwanted float32 promotions |
| Dead Code Elimination | `reduce_graph()` | Removes unused nodes |

Sources: [tt_torch/dynamo/passes.py 24-135](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/passes.py#L24-L135)[tt_torch/dynamo/backend.py 186-226](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L186-L226)

**Configuration:** The `CompilerConfig.consteval_parameters` flag (set via `TT_TORCH_CONSTEVAL_PARAMETERS`) enables constant evaluation to include model parameters, not just embedded constants.

Sources: [tt_torch/tools/utils.py 411-417](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L411-L417)[tt_torch/tools/utils.py 438-440](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L438-L440)

## Stage 3: Torch MLIR to Torch Backend MLIR

### Pipeline Execution

The Torch dialect IR is lowered to Torch Backend dialect using a predefined MLIR pass pipeline.

Sources: [tt_torch/dynamo/backend.py 210-218](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L210-L218)

**Pipeline Name:**`builtin.module(torchdynamo-export-to-torch-backend-pipeline)`

This pipeline is provided by torch-mlir and performs decompositions and canonicalizations specific to the Torch Backend dialect. The `run_pipeline_with_repro_report()` utility generates crash reproducers if the pipeline fails, aiding debugging.

Sources: [tt_torch/dynamo/backend.py 210-218](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L210-L218)

## Stage 4: Torch Backend MLIR to StableHLO

### StableHLO Lowering

The conversion to StableHLO uses torch-mlir's `lower_mlir_module()` function with `OutputType.STABLEHLO`.

Sources: [tt_torch/dynamo/backend.py 220-224](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L220-L224)

### Optional Automatic Parallelization

When `compiler_config.automatic_parallelization` is enabled, the StableHLO IR undergoes automatic data parallelization using Shardy.

**Key Parameters:**

*   `mesh_shape`: Device grid configuration from `compiler_config.mesh_shape`
*   `len_activations`: Number of runtime inputs
*   `len_graph_constants`: Number of constant parameters

Sources: [tt_torch/dynamo/backend.py 241-249](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L241-L249)[tt_torch/csrc/tt-mlir-interface.cpp 73-132](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/tt-mlir-interface.cpp#L73-L132)[tt_torch/csrc/bindings.cpp 168-174](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L168-L174)

## Stage 5: StableHLO to TTIR

### C++ Compilation Interface

The StableHLO to TTIR transformation occurs in C++ through the tt-mlir compiler infrastructure.

Sources: [tt_torch/dynamo/backend.py 251-252](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L251-L252)[tt_torch/csrc/bindings.cpp 176-179](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L176-L179)[tt_torch/csrc/tt-mlir-interface.cpp 134-184](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/tt-mlir-interface.cpp#L134-L184)

### Pipeline Options

The `StableHLOToTTIRPipelineOptions` structure controls the conversion:

*   `arithDialectConversionsEnabled = true`: Enables arithmetic dialect conversions
*   `legalizeCompositeToCallEnabled = true`: Converts composite ops to function calls

Sources: [tt_torch/csrc/tt-mlir-interface.cpp 169-172](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/tt-mlir-interface.cpp#L169-L172)

### MLIR Context Setup

The compilation requires registering multiple dialects:

| Dialect | Purpose |
| --- | --- |
| `arith::ArithDialect` | Arithmetic operations |
| `func::FuncDialect` | Function definitions |
| `ml_program::MLProgramDialect` | ML program constructs |
| `tt::registerAllDialects()` | Tenstorrent-specific dialects (TTIR, TTNN, etc.) |
| `stablehlo::registerAllDialects()` | StableHLO operations |

Sources: [tt_torch/csrc/tt-mlir-interface.cpp 136-148](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/tt-mlir-interface.cpp#L136-L148)

## Stage 6: TTIR to TTNN and Flatbuffer

### Unified Compilation

The final stages (TTIR → TTNN → Flatbuffer) occur in a single C++ function call.

Sources: [tt_torch/dynamo/backend.py 259-267](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L259-L267)[tt_torch/csrc/bindings.cpp 181-202](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L181-L202)[tt_torch/csrc/tt-mlir-interface.cpp 194-291](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/tt-mlir-interface.cpp#L194-L291)

### Pipeline Configuration

The `TTIRToTTNNBackendPipelineOptions` controls backend-specific optimizations:

| Option | Default | Environment Variable | Purpose |
| --- | --- | --- | --- |
| `enableFusing` | `true` | - | Enables operation fusion |
| `enableConstEval` | from config | `TT_TORCH_CONSTEVAL` | Constant expression evaluation |
| `optimizerPassEnabled` | from config | `TT_TORCH_OPTIMIZER` | Enables optimizer (requires OPMODEL) |
| `argumentTypeMap` | computed | - | Marks inputs as Constant vs Input |
| `systemDescPath` | from runtime | - | Hardware system description |

## Stage 7: Flatbuffer to Execution

### Binary Creation and Device Setup

The compiled flatbuffer becomes an executable binary through the tt-mlir runtime.

Sources: [tt_torch/csrc/bindings.cpp 216-224](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L216-L224)[tt_torch/csrc/bindings.cpp 226-258](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L226-L258)[tt_torch/csrc/bindings.cpp 368-390](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L368-L390)[tt_torch/dynamo/executor.py](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py)

### Input Preprocessing

Before execution, inputs undergo layout transformations:

1.   **Contiguity Check:** Ensures tensors are contiguous in memory
2.   **Tilization:** Converts from row-major to hardware-friendly tiled layout
3.   **Device Transfer:** Copies data to device memory

Sources: [tt_torch/csrc/bindings.cpp 226-258](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/csrc/bindings.cpp#L226-L258)

## Module Dumping and Debugging

### Save MLIR Override

The `save_mlir_override` configuration enables saving intermediate MLIR representations to disk.

**Environment Variable:**`TT_TORCH_SAVE_MLIR=STABLEHLO,TTIR,TTNN`

**Output Location:**`{output_mlir_dir}/{sanitized_model_name}_{dialect}.mlir`

**Requirement:** Must build with `-DTT_RUNTIME_DEBUG=ON`
