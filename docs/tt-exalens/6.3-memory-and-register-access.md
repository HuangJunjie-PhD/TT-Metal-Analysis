This page covers the RISC-V core debug memory and register access operations provided by the `RiscDebug` interface and its `BabyRiscDebug` implementation. It describes how general-purpose registers (GPRs) and memory are read and written through the debug hardware, including alignment handling, private memory access, and safe mode checks.

For higher-level library wrappers (`read_riscv_memory`, `write_riscv_memory`) that call into these primitives, see [3.6](https://deepwiki.com/tenstorrent/tt-exalens/3.6-risc-v-core-control). For execution control (halt, step, continue) that must precede most of these operations, see [6.2](https://deepwiki.com/tenstorrent/tt-exalens/6.2-execution-control). For watchpoints and breakpoints, see [6.4](https://deepwiki.com/tenstorrent/tt-exalens/6.4-watchpoints-and-breakpoints). For platform-specific class inheritance, see [6.5](https://deepwiki.com/tenstorrent/tt-exalens/6.5-platform-specific-debug-implementations).

## General-Purpose Register (GPR) Access

### Register Index Convention

RISC-V has 32 integer registers (x0–x31) plus the program counter (PC). Within TTExaLens, registers are referenced by integer index 0–32, where index 32 represents the PC.

| Index | ABI Name | Notes |
| --- | --- | --- |
| 0 | `zero` | Always reads 0, writes are ignored |
| 1 | `ra` | Return address |
| 2 | `sp` | Stack pointer |
| 8 | `s0` / `fp` | Frame pointer (also called `s0`) |
| 10–17 | `a0`–`a7` | Function arguments / return values |
| 32 | `pc` | Program counter (special case) |

The full mapping is defined in `RISCV_REGISTER_NAMES_BY_INDEX` and `RISCV_REGISTER_INDEX_BY_NAME` in [ttexalens/hardware/baby_risc_debug.py 69-173](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L69-L173)

The helper functions `get_register_index(reg_index_or_name)` and `get_register_name(reg_index_or_name)` convert between names and indices [ttexalens/hardware/baby_risc_debug.py 176-188](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L176-L188)

## Program Counter (PC) Access

`get_pc()` in `BabyRiscDebug` uses the debug bus if a `pc` signal is available for the core, falling back to halting the core and reading GPR index 32.

Sources: [ttexalens/hardware/baby_risc_debug.py 673-691](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L673-L691)

### Platform Overrides for `read_gpr(32)`

All three architecture-specific subclasses override `read_gpr` to return the PC from the debug bus instead of the debug register protocol. This is because the debug bus gives the PC without requiring the core to be halted.

| Platform | Class | Override behavior |
| --- | --- | --- |
| Wormhole | `WormholeBabyRiscDebug` | For index 32: reads `risc_name + "_pc"` from `noc_block.debug_bus` |
| Blackhole | `BlackholeBabyRiscDebug` | For index 32: reads `risc_name + "_pc"` from `noc_block.debug_bus` |
| Quasar | `QuasarBabyRiscDebug` | For index 32: reads `risc_name + "_pc"` from `neo_block.debug_bus` |

Sources: [ttexalens/hardware/wormhole/baby_risc_debug.py 32-37](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/wormhole/baby_risc_debug.py#L32-L37)[ttexalens/hardware/blackhole/baby_risc_debug.py 20-25](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L20-L25)[ttexalens/hardware/quasar/baby_risc_debug.py 29-34](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/quasar/baby_risc_debug.py#L29-L34)

## Memory Access

### Word-Level Operations

`read_memory(address, safe_mode)` and `write_memory(address, data, safe_mode)` are 32-bit word operations. They are implemented as thin wrappers around the byte-level methods:

*   `read_memory` returns `int.from_bytes(read_memory_bytes(address, 4), 'little')`
*   `write_memory` calls `write_memory_bytes(address, data.to_bytes(4, 'little'))`

Sources: [ttexalens/hardware/baby_risc_debug.py 697-701](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L697-L701)

### Byte-Level Operations

`read_memory_bytes(address, size_bytes, safe_mode)` and `write_memory_bytes(address, data, safe_mode)` handle arbitrary byte counts and byte offsets. The underlying hardware only supports 4-byte aligned 32-bit word accesses, so the implementation manages alignment internally.

**Alignment handling diagram:**

Sources: [ttexalens/hardware/baby_risc_debug.py 703-770](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L703-L770)

### Hardware Protocol

`BabyRiscDebugHardware.read_memory(addr)` and `write_memory(addr, value)` implement the actual debug register protocol. Both **require the core to be halted** (enforced by `assert_halted()` when `enable_asserts=True`).

| Operation | Registers Written | Command Issued | Result Read |
| --- | --- | --- | --- |
| `read_memory(addr)` | `REG_COMMAND_ARG_0` = addr | `DEBUG_MODE | READ_MEMORY` | `REG_COMMAND_RETURN_VALUE` |
| `write_memory(addr, value)` | `REG_COMMAND_ARG_1` = value, `REG_COMMAND_ARG_0` = addr | `DEBUG_MODE | WRITE_MEMORY` | — |

Sources: [ttexalens/hardware/baby_risc_debug.py 411-427](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L411-L427)

## Private Memory Access

Some RISC cores have private memory (data or code) that is only accessible through the debug interface while the core is executing (not in reset). `ensure_private_memory_access()` is a context manager that ensures this precondition.

When the core is in reset, the context manager:

1.   Saves the word at the code start address.
2.   Writes a `JAL x0, 0` infinite loop instruction (`0x6F`) to the code start address.
3.   Takes the core out of reset so it begins executing the infinite loop.
4.   Halts the core through the debug hardware.
5.   Yields control to allow private memory access.
6.   Puts the core back into reset.
7.   Restores the original bytes at the code start address.

Sources: [ttexalens/hardware/baby_risc_debug.py 573-600](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L573-L600)

## Safe Mode

All memory access methods accept an optional `safe_mode: bool | None` parameter. When `None`, the value is inherited from `device._context.safe_mode`.

When `safe_mode=True`, `_validate_safe_access(address, size_bytes)` is called before any access. The current implementation is a placeholder (see [ttexalens/hardware/baby_risc_debug.py 693-695](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L693-L695)), but the hook is in place for future address range validation.

| Method | safe_mode check point |
| --- | --- |
| `read_memory_bytes` | Before alignment computation |
| `write_memory_bytes` | Before alignment computation |
| `read_memory` | Delegated via `read_memory_bytes` |
| `write_memory` | Delegated via `write_memory_bytes` |

Sources: [ttexalens/hardware/baby_risc_debug.py 703-735](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_debug.py#L703-L735)

## Platform-Specific Restrictions

### Blackhole: TRISC2 Private Memory Restriction

`BlackholeBabyRiscDebug` adds `assert_trisc2_address(address)` which raises `TTException` if the core is `trisc2` and `address % 16 > 4`. This is a hardware bug workaround documented in issue #528.

This check is applied in both `read_memory_bytes` and `write_memory_bytes` overrides.

Sources: [ttexalens/hardware/blackhole/baby_risc_debug.py 27-39](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/blackhole/baby_risc_debug.py#L27-L39)

## Memory Regions

`BabyRiscInfo` holds references to the memory blocks accessible to each core. The `MemoryBlock` class provides address translation between private (core-local) addresses and NOC addresses.

`BabyRiscInfo.translate_to_noc_address(addr)` attempts translation through `data_private_memory` first, then `l1`. This is used to verify or find the NOC equivalent of a private core address [ttexalens/hardware/baby_risc_info.py 90-98](https://github.com/tenstorrent/tt-exalens/blob/046c35eb/ttexalens/hardware/baby_risc_info.py#L90-L98)
