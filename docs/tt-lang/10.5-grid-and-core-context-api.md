## Purpose and Scope

This page documents the execution context API in `tt-lang`, which provides runtime information about the kernel's execution environment. These context-aware functions allow kernels to query their position in the grid, access grid dimensions, and facilitate work distribution. In the Tenstorrent architecture, a "node" corresponds to a single Tensix Core, which is the minimal unit capable of executing a `tt-lang` program [docs/sphinx/specs/TTLangSpecification.md 99-101](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L99-L101)

The context API is essential for Single-Program-Multiple-Data (SPMD) programming, where the same kernel code executes on multiple cores but operates on different data partitions based on its unique coordinates [docs/sphinx/specs/TTLangSpecification.md 101-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L101-L102)

## Overview of Context-Aware Functions

The execution context API provides functions that return information about the current execution environment. These functions are evaluated at runtime on each core and provide context-specific values.

| Function | Return Type | Description |
| --- | --- | --- |
| `ttl.node(dims=1)` | `int` or `tuple` | Current node coordinates. Returns a flattened index for `dims=1` and a `tuple` otherwise. [docs/sphinx/specs/TTLangSpecification.md 121-125](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L121-L125) |
| `ttl.grid_size(dims=1)` | `int` or `tuple` | Grid dimensions. Returns total core count for `dims=1` and a `tuple` otherwise. [docs/sphinx/specs/TTLangSpecification.md 106-113](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L106-L113) |

In the Python DSL implementation, these are handled by the `TTLGenericCompiler` which translates these calls into MLIR operations during the AST visiting process [python/ttl/_src/ttl_ast.py 166-175](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L166-L175)

## Context Function Details

### `ttl.node(dims)`

Returns the coordinates of the current node (Tensix core). Node coordinates are zero-based and contiguous, corresponding to a logical indexing scheme.

**Dimensionality Rules**:

*   If requested `dims` are smaller than grid dimensions, the highest rank dimension is flattened [docs/sphinx/specs/TTLangSpecification.md 121](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L121-L121)
*   If requested `dims` are greater than grid dimensions, highest rank dimensions are padded with zero [docs/sphinx/specs/TTLangSpecification.md 122](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L122-L122)

## Coordinate System and Indexing Conventions

### Grid vs Tensor Coordinate Systems

**Critical Convention**: Grid coordinates typically follow `(x, y)` notation (Column, Row), whereas tensor indexing follows `[Row, Column]` (or `[N, H, W, C]`) notation. This requires a coordinate swap when mapping grid positions to tensor tiles.

**Natural Language to Code Entity Space Mapping**

Title: Grid to Tensor Mapping Convention

**Mapping Rule**:

`# node_col = x, node_row = ynode_col, node_row = ttl.node(dims=2)  # To access the tile at this core's logical position in a 2D tensor:# Note: Tensor is [Row, Col]tile_slice = input_tensor[node_row, node_col]`

## Compilation and AST Handling

During the compilation of a `@ttl.operation` (the decorator for the top-level kernel entry point [docs/sphinx/specs/TTLangSpecification.md 45](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L45-L45)), the `TTLGenericCompiler` handles context functions.

### Context Resolution during AST Lowering

Title: Context Resolution during AST Lowering

**Compiler Behavior**: The `TTLGenericCompiler` maintains a `CompilerContext` which stores the `grid` dimensions [python/ttl/_src/ttl_ast.py 119-126](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L119-L126) This context is initialized during compiler instantiation [python/ttl/_src/ttl_ast.py 141-145](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L141-L145) When generating MLIR, the compiler uses these dimensions to build `RankedTensorType` with appropriate `TTLLayoutAttr` encoding [python/ttl/_src/ttl_ast.py 69-116](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L69-L116)

## SPMD and Multi-core Execution

In Single-Program-Multiple-Data (SPMD) mode, the grid remains two-dimensional while the operation is submitted to multiple chips [docs/sphinx/specs/TTLangSpecification.md 101-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L101-L102)

### Work Distribution Example

In a multi-node setup, kernels calculate their specific workload using `ttl.node()` and `ttl.grid_size()`:

`grid_cols, grid_rows = ttl.grid_size(dims=2)node_col, node_row = ttl.node(dims=2) # Calculate global offsetsrow = node_row * rows_per_node + local_rowcol = node_col * cols_per_node + local_col`
**Key Constants**:

*   `DEFAULT_TILE_SIZE`: 32x32 tiles are the standard unit of computation [python/ttl/_src/ttl_ast.py 15](https://github.com/tenstorrent/tt-lang/blob/d76e6233/python/ttl/_src/ttl_ast.py#L15-L15)
*   `SPMD Mode`: Simplifies multi-chip reasoning by keeping grid 2D while distributing data [docs/sphinx/specs/TTLangSpecification.md 101-102](https://github.com/tenstorrent/tt-lang/blob/d76e6233/docs/sphinx/specs/TTLangSpecification.md?plain=1#L101-L102)
