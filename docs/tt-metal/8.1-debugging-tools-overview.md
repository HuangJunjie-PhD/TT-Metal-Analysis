## Debugging Subsystem Architecture

The debugging stack in the TT-Metal framework is designed as a layered system involving coordination between host-side components and device-side instrumentation:

*   The **host side** includes background thread servers that continuously poll and analyze kernel execution states, error signals, and diagnostic buffers from devices.
*   The **device side** instrumentation writes runtime debugging data into designated L1 memory regions (such as mailboxes and ring buffers).

Control and lifecycle management of these debugging services are done through the singleton `MetalContext`, integrating with runtime configuration via `llrt::RunTimeOptions`.

### Architecture Diagram: Debugging Subsystem Components and Data Flow

**Notes:**

*   `WatcherServer` continuously polls device mailboxes, looking for error conditions and kernel status.
*   `WatcherDeviceReader` handles device-specific data reads for mailboxes and ring buffer analysis.
*   `DPrintServer` processes kernel-side debug prints (via DPRINT mechanism) for printf-style diagnostics.

## Watcher Debug System

Watcher is the core runtime debugging subsystem actively monitoring device kernel execution for error signals, corrupt states, and system hangs.

### Architecture and Operation

*   The `WatcherServer::Impl` starts a background thread (`poll_watcher_data()`) that polls all devices' mailboxes through `WatcherDeviceReader` instances, collecting debug events and errors [`tt_metal/impl/debug/watcher_server.cpp:112-143`].
*   Watcher readers decode kernel runtime assertions, hardware faults, and sanity check NoC traffic using `virtual_noc_coordinate()` and `get_noc_target_str()` helpers for address interpretation [`tt_metal/impl/debug/watcher_device_reader.cpp:131-160`].
*   Assert handling uses a detailed enum of `dev_msgs::debug_assert_type_t` to decipher cause and line number [`tt_metal/impl/debug/debug_helpers.hpp:110-150`].
*   Watcher handles multi-processor architectures (Blackhole, Wormhole, Quasar) with HAL-provided processor naming and coordinate translation [`tt_metal/impl/debug/watcher_device_reader.cpp:64-106`].
*   Stack overflow detection for RISC devices is integrated by tracking stack pointers at runtime with firmware hooking [`tt_metal/hw/firmware/src/tt-2xx/dm.cc:12-13`].

### Key Classes & Methods

| Class / Method | Description |
| --- | --- |
| `WatcherServer::Impl` | Host-side class that manages Watcher lifecycle and device polling. |
| `WatcherDeviceReader` | Reads device mailboxes and sanitizes NOC usage and asserts. |
| `poll_watcher_data()` | Thread function performing periodic device mailbox polling. |
| `virtual_noc_coordinate()` | Converts NoC coordinates to virtual core coordinates. |
| `get_noc_target_str()` | Gets human-readable target name for a NoC address. |

## DPRINT (Kernel Debugging via Print Buffers)

DPRINT enables in-kernel `printf` style debugging by writing messages into dedicated device print buffers in L1 memory, which the host-side `DPrintServer` polls and decodes.

### Features and Implementation

*   `DPrintServer::Impl` manages print buffer polling for all devices and cores [`tt_metal/impl/debug/dprint_server.cpp:159-190`].
*   Buffers are initialized with a magic value (`DEBUG_PRINT_SERVER_STARTING_MAGIC`) written to signal readiness via `WriteInitMagic()` [`tt_metal/impl/debug/dprint_server.cpp:122-150`].
*   Filtering by processor types is supported using runtime options to enable selective core printing [`tt_metal/impl/debug/dprint_server.cpp:106-110`].
*   Processor names are resolved dynamically via `GetRiscName()` based on device topology and HAL mappings [`tt_metal/impl/debug/dprint_server.cpp:93-104`].
*   Device print buffer addresses are retrieved through HAL calls, ensuring correct addressing across architectures [`tt_metal/impl/debug/debug_helpers.hpp:84-87`].

### Key Classes & Functions

| Class / Function | Role |
| --- | --- |
| `DPrintServer::Impl` | Hosts the print server thread and buffer management. |
| `WriteInitMagic()` | Writes synchronization magic values to device buffers. |
| `RiscEnabled()` | Runtime check for enabled RISCs for DPRINT. |
| `GetRiscName()` | Provides symbolic RISCV core names for messages. |

## Triage and Post-Mortem Analysis

The Triage system is a Python-based analysis toolset for diagnosing hardware hangs and runtime errors after the fact using collected dump files and live RPC queries.

### System Architecture

*   The main triage tool `tools/triage/triage.py` offers command-line automation and scripting to execute comprehensive health checks on devices, cores, and NoC traffic [`tools/triage/triage.py:6-42`].
*   The triage pipeline incorporates scripts for: 
    *   Inspecting dispatcher data and kernel states (`dispatcher_data.py`)
    *   Dumping key kernel variables from fast dispatch (`dump_fast_dispatch.py`)
    *   Running constitution checks across devices and cores (`run_checks.py`)

*   CI systems can automatically invoke triage on timeouts via configured GitHub Actions workflows [`.github/actions/setup-job/action.yml:140-161`].

| `run_checks` | Runs checks for device health, block and CPU core statuses. |
| CI Integration | Auto-triage triggered on hardware hang timeouts for automatic triage output. |
