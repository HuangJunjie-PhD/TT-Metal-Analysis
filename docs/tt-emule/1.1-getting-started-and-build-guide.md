This page provides a comprehensive guide to setting up the `tt-emule` development environment, building the emulator integrated with `tt-metal`, and verifying the installation using the tiered regression suite.

## Overview

`tt-emule` is designed to be built as a component of the `tt-metal` ecosystem. It replaces the hardware-dependent User-Mode Driver (UMD) and firmware layers with a C++ software model. The build process leverages `CMake` and `Ninja` to produce a version of `libtt_metal.so` that routes device commands to the emulator instead of silicon.

## Prerequisites

The following tools and libraries are required for a successful build. These versions match the `tt-mlir-ci-ubuntu-24-04` container used in CI.

| Tool | Minimum Version | Purpose |
| --- | --- | --- |
| **clang-20** | 20.x | Primary compiler for host and JIT kernels. |
| **libstdc++** | 13.x (gcc-13+) | Required for C++20 `std::ranges` support in `tt-metal`. |
| **CMake** | 3.24 | Build configuration. |
| **Ninja** | 1.10+ | Fast build execution. |
| **Python** | 3.10+ | Required for `ttnn` pytests and regression scripts. |

## Repository Setup & Pinning Strategy

`tt-emule` relies on a three-way dependency pinning strategy to ensure stability across `tt-metal` and `tt-mlir`.

1.   **Sibling Layout**: All repositories must be cloned as siblings in a common root directory.
2.   **Pinning Files**: `tt-emule` contains two authoritative pin files: 
    *   `tt-metal-pin.txt`: The specific SHA of `tt-metal` that this version of `tt-emule` is compatible with. [tt-metal-pin.txt 1-11](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tt-metal-pin.txt#L1-L11)
    *   `tt-mlir-pin.txt`: The SHA of `tt-mlir` used for Direct-to-Metal (D2M) regression. [tt-mlir-pin.txt 1-5](https://github.com/tenstorrent/tt-emule/blob/a57cf072/tt-mlir-pin.txt#L1-L5)

### Setup Commands

`export ROOT=$HOME/workmkdir -p "$ROOT" && cd "$ROOT" # Clone repositoriesgit clone git@github.com:tenstorrent/tt-emule.gitgit clone git@github.com:tenstorrent/tt-metal.git # Checkout the pinned tt-metal versioncd "$ROOT/tt-metal"git checkout $(grep -vE '^\s*(#|$)' "$ROOT/tt-emule/tt-metal-pin.txt" | head -1) # Initialize only required submodules (UMD and Tracy)git submodule update --init tt_metal/third_party/umdgit submodule update --init tt_metal/third_party/tracy`

## Building with Emulation Support

The emulator is activated via the `-DTT_METAL_USE_EMULE=ON` CMake flag. This flag triggers several internal changes:

*   Compiles `emulated_program_runner.cpp` within `tt-metal`. [BUILD_GUIDE.md 136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/BUILD_GUIDE.md?plain=1#L136-L136)
*   Defines `TT_METAL_USE_EMULE=1` for the `tt_metal`, `impl`, and `llrt` libraries. [BUILD_GUIDE.md 136](https://github.com/tenstorrent/tt-emule/blob/a57cf072/BUILD_GUIDE.md?plain=1#L136-L136)
*   Enables the emulation-specific dispatch branch in `tt_metal.cpp`. [IMPLEMENTATION_REPORT.md 112-117](https://github.com/tenstorrent/tt-emule/blob/a57cf072/IMPLEMENTATION_REPORT.md?plain=1#L112-L117)

### CMake Build Command

Run this command from the `tt-metal` source root:

`cmake -S . -B build_emule -G Ninja \    -DCMAKE_TOOLCHAIN_FILE=$ROOT/tt-metal/cmake/x86_64-linux-clang-20-libstdcpp-toolchain.cmake \    -DCMAKE_BUILD_TYPE=Release \    -DTT_METAL_USE_EMULE=ON \    -DTT_EMULE_PATH=$ROOT/tt-emule \    -DWITH_PYTHON_BINDINGS=ON \    -DTT_METAL_BUILD_TESTS=ON \    -DTTNN_BUILD_TESTS=ON \    -DENABLE_TRACY=OFF \    -DENABLE_DISTRIBUTED=ON \    -DCMAKE_INSTALL_PREFIX=$ROOT/tt-metal/build_emule cmake --build build_emule -j$(nproc)`
Sources: [BUILD_GUIDE.md 108-127](https://github.com/tenstorrent/tt-emule/blob/a57cf072/BUILD_GUIDE.md?plain=1#L108-L127)[GETTING_STARTED.md 33-51](https://github.com/tenstorrent/tt-emule/blob/a57cf072/GETTING_STARTED.md?plain=1#L33-L51)

### Post-Build Symlinks

To enable `import ttnn` and ensure `tt-mlir` can find the emulation libraries, four mandatory symlinks must be created:

`# Link the python binding for pytestln -sfn $ROOT/tt-metal/build_emule/ttnn/_ttnn.so \       $ROOT/tt-metal/ttnn/ttnn/_ttnn.so # Link libraries for RPATH resolutioncd $ROOT/tt-metal/build_emule/libln -sfn ../tt_metal/libtt_metal.so libtt_metal.soln -sfn ../tt_stl/libtt_stl.so libtt_stl.soln -sfn ../ttnn/_ttnncpp.so _ttnncpp.soln -sfn ../ttnn/_ttnn.so _ttnn.so`

## Verification

Verification is performed using per-architecture regression scripts located in the `tt-emule/scripts/` directory. These scripts should be run sequentially as they share a JIT cache.

### Regression Tiers

| Script | Arch | Tiers | Focus |
| --- | --- | --- | --- |
| `run_regression_wormhole.sh` | Wormhole | 2, 3, 3k, 3l, 5, 6 | Buffer ops, L1 transfers, JIT kernels, TTNN matmul. |
| `run_regression_blackhole.sh` | Blackhole | 4 | INT32 operations. |
| `run_regression_quasar.sh` | Quasar | 3b-3j | DFB synchronization, multi-core matmul PCC. |

Sources: [GETTING_STARTED.md 65-80](https://github.com/tenstorrent/tt-emule/blob/a57cf072/GETTING_STARTED.md?plain=1#L65-L80)[README.md 134-152](https://github.com/tenstorrent/tt-emule/blob/a57cf072/README.md?plain=1#L134-L152)

### Quick Verification Commands

`cd "$ROOT/tt-emule"export TT_METAL_DIR="$ROOT/tt-metal" # Run Wormhole regression (Zero-tolerance)bash scripts/run_regression_wormhole.sh # Run Quasar regression (Check against allowlist)bash scripts/run_regression_quasar.sh`
