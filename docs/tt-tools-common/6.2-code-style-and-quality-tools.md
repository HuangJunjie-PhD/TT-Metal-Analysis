## Code Formatting with Black

Black is the opinionated code formatter used for all Python source files in tt-tools-common. It enforces consistent style across the codebase, eliminating debates about formatting preferences.

### Black Configuration

The project uses Black version 23.11.0 or higher with default settings. Black's configuration is intentionally minimal to leverage its opinionated defaults:

| Configuration Aspect | Value | Rationale |
| --- | --- | --- |
| **Line Length** | 88 characters (default) | Black's optimized default |
| **String Quotes** | Double quotes (default) | Consistent string formatting |
| **Target Versions** | Python 3.10+ | Matches project requirement |
| **Skip String Normalization** | No | Enforces quote consistency |
| **Magic Trailing Comma** | Yes | Preserves intentional formatting |

**Sources:**[pyproject.toml 38](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L38-L38)

### Running Black

`# Format all Python files in the projectblack . # Check formatting without making changesblack --check . # Format specific files or directoriesblack tt_tools_common/ tests/ # Show diff without applying changesblack --diff .`
Black automatically discovers all `.py` files and applies consistent formatting. The `--check` flag is used in CI to verify formatting without modifying files.

## Pre-commit Hook Framework

The pre-commit framework automates code quality checks before commits reach the repository. It ensures that code meets style and quality standards before entering version control.

### Pre-commit Configuration

While the specific `.pre-commit-config.yaml` file is not provided in the source files, the project includes pre-commit as a development dependency, indicating its use for automated quality checks.

**Sources:**[pyproject.toml 39](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L39-L39)

### Installing Pre-commit Hooks

`# Install pre-commit in the repositorypre-commit install # Install hooks for all supported git hookspre-commit install --install-hooks # Uninstall hooks (if needed)pre-commit uninstall`
After installation, pre-commit runs automatically on `git commit`. The hooks execute only on staged files, making the checks fast and relevant to the current change.

### Running Pre-commit Manually

`# Run all hooks on staged filespre-commit run # Run all hooks on all files in the repositorypre-commit run --all-files # Run a specific hookpre-commit run black # Update hook repositories to latest versionspre-commit autoupdate`
Manual execution is useful for:

*   Formatting code before staging changes
*   Checking all files after updating hook configurations
*   Testing hook behavior without committing

## Development Dependency Details

### Black (Code Formatter)

*   **Version:**`>=23.11.0`
*   **Purpose:** Enforces consistent Python code formatting
*   **Usage:** Automatic via pre-commit; manual via `black .`
*   **Configuration:** Uses Black defaults (88 char line length)

### Pre-commit (Hook Framework)

*   **Version:**`>=3.5.0`
*   **Purpose:** Manages git hooks for quality checks
*   **Usage:** Install once with `pre-commit install`, runs on every commit
*   **Hooks:** Typically includes Black, trailing whitespace removal, YAML/JSON validation

### PyLuwen (Hardware Abstraction)

*   **Version:**`~=0.8.0`
*   **Purpose:** Required for hardware tests marked with `@pytest.mark.requires_hardware`
*   **Usage:** Import in tests that interact with Tenstorrent chips
*   **Note:** Not required for software-only tests

### Pytest (Test Framework)

*   **Version:**`>=7.0.0`
*   **Purpose:** Runs unit and integration tests
*   **Configuration:** Test markers defined in `[tool.pytest.ini_options]`
*   **Usage:** See [Testing Guide](https://deepwiki.com/tenstorrent/tt-tools-common/6.1-testing-guide) for detailed information

### Pytest-Asyncio (Async Test Support)

*   **Version:**`>=0.21.0`
*   **Purpose:** Enables testing of async functions in Textual widgets
*   **Usage:** Applied automatically to async test functions
*   **Integration:** Works with Textual's App.run_test() for TUI testing

## Python Version Compatibility

The project supports Python 3.10 and higher, with CI testing across multiple versions.

### Version Requirements

**Sources:**[pyproject.toml 3](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L3-L3)[pyproject.toml 19-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L19-L21)

### Version-Specific Considerations

| Python Version | Status | Notes |
| --- | --- | --- |
| **3.10** | Minimum required | All features supported; requires `tomli` for TOML parsing |
| **3.11** | Recommended | Better performance; `tomllib` built-in |
| **3.12** | Supported | Latest features; fully tested in CI |
| **3.9 or lower** | Not supported | Missing type hint features (PEP 604) |

The `requires-python = ">=3.10"` constraint in pyproject.toml prevents installation on incompatible Python versions. This ensures that modern syntax features can be used throughout the codebase without compatibility checks.

**Sources:**[pyproject.toml 3](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L3-L3)[pyproject.toml 18-21](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L18-L21)

## Code Quality in CI/CD

Code quality checks are integrated into the GitHub Actions CI pipeline to ensure all commits meet project standards.

### CI Quality Check Workflow

### Local vs CI Validation

| Check | Local (Pre-commit) | CI (GitHub Actions) |
| --- | --- | --- |
| **Black Formatting** | Automatic on commit | `black --check .` |
| **File Cleanup** | Trailing whitespace, EOF | Implicit via Black |
| **Test Execution** | Optional (developer choice) | Required for all PRs |
| **Hardware Tests** | Skipped (unless hardware available) | Skipped (`@pytest.mark.requires_hardware`) |

The local pre-commit hooks provide immediate feedback, while CI serves as the authoritative quality gate. Developers should run `pre-commit run --all-files` before pushing to catch issues early.

**Sources:**[pyproject.toml 38-43](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L38-L43)

## Package Data and Exclusions

The project includes specific package data while excluding build artifacts from distribution.

### Included Package Data

The `include-package-data = true` setting combined with specific patterns ensures that CSS themes and YAML configuration files are included in the wheel distribution, while build artifacts and Debian packaging files are excluded.

**Sources:**[pyproject.toml 55-69](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/pyproject.toml#L55-L69)

### Gitignore Patterns

The `.gitignore` file excludes common Python development artifacts:

| Pattern | Purpose |
| --- | --- |
| `__pycache__/` | Bytecode cache directories |
| `*.egg-info` | Package metadata |
| `.eggs/` | Setuptools egg cache |
| `build/` | Build output directory |
| `.venv/`, `venv/`, `.env/` | Virtual environment directories |
| `.pybuild/` | Python build directory |
| `debian/.debhelper`, `debian/files` | Debian build artifacts |
| `.vscode/` | VS Code configuration |

These patterns prevent committing transient build artifacts and developer-specific configurations to version control.

**Sources:**[.gitignore 1-17](https://github.com/tenstorrent/tt-tools-common/blob/892885f4/.gitignore#L1-L17)

## Best Practices for Contributors

### Development Workflow

1.   **Initial Setup**

`# Clone repositorygit clone https://github.com/tenstorrent/tt-tools-common.gitcd tt-tools-common # Create virtual environmentpython3.10 -m venv .venvsource .venv/bin/activate # Install with dev dependenciespip install -e ".[dev]" # Install pre-commit hookspre-commit install`
2.   **Making Changes**

`# Create feature branchgit checkout -b feature/my-improvement # Make code changesvim tt_tools_common/reset_common/chip_reset.py # Format code (optional, pre-commit will do this)black . # Run tests locallypytest -v # Commit (pre-commit runs automatically)git add .git commit -m "feat: improve reset logging"`
3.   **Before Pushing**

`# Run all pre-commit hooks on all filespre-commit run --all-files # Run full test suitepytest -v # Push to remotegit push origin feature/my-improvement`

### Code Quality Checklist

*   - [x]  All code is Black-formatted
*   - [x]  No trailing whitespace or incorrect line endings
*   - [x]  Type hints are used for function signatures
*   - [x]  Docstrings follow project conventions
*   - [x]  Tests pass locally (software tests minimum)
*   - [x]  No new imports without updating dependencies
*   - [x]  Changes are documented if API-facing
*   - [x]  Pre-commit hooks pass on all files

## Troubleshooting

### Black Formatting Conflicts

**Issue:** Pre-commit hook fails with Black formatting errors.

**Solution:**

`# Let Black fix the formattingblack . # Stage the formatted filesgit add . # Commit againgit commit`
### Pre-commit Hook Not Running

**Issue:** Commits succeed without running hooks.

**Solution:**

`# Reinstall hookspre-commit uninstallpre-commit install # Verify installationpre-commit run --all-files`
### Development Dependencies Not Found

**Issue:**`ModuleNotFoundError` for `black`, `pytest`, etc.

**Solution:**

`# Ensure dev dependencies are installedpip install -e ".[dev]" # Verify installationpip list | grep -E "black|pre-commit|pytest"`
### Python Version Mismatch

**Issue:** Package installation fails with Python version error.

**Solution:**

`# Check Python versionpython --version # Must be 3.10 or higher# Use pyenv or system package manager to install correct version`
