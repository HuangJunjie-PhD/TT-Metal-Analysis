## Purpose and Scope

This document describes the ping-pong configuration state management system in the LLK ckernel layer. This system enables double-buffering of hardware configuration registers and destination register space, allowing software to prepare the next operation's configuration while hardware executes the current operation. This mechanism is critical for achieving efficient pipeline operation in the unpack-math-pack stages.

For information about synchronization primitives that coordinate access to these states, see [Synchronization Primitives](/tenstorrent/tt-llk/2.2-synchronization-primitives). For details on register access patterns and TTI instructions, see [Register Access and TTI Instructions](/tenstorrent/tt-llk/2.4-register-access-and-tti-instructions).

## Overview

The configuration state management system maintains two independent ping-pong state variables that support double-buffered operation:

1. **Configuration State ID (`cfg_state_id`)** - Alternates between two hardware configuration register banks (0 and 1), enabling software to prepare the next kernel's configuration while the current kernel executes.
2. **Destination Offset ID (`dest_offset_id`)** - Alternates between two halves of the destination register file, supporting double-buffering of output data.

Both variables toggle between values 0 and 1, implementing a simple ping-pong buffering scheme that maximizes hardware utilization and minimizes pipeline stalls.

## Destination Register Offset Management (dest\_offset\_id)

The destination offset management system provides ping-pong buffering for the destination register file, allowing output data to be written to one half while the other half is being packed to L1 memory.

### Destination Buffer Addressing

**Destination Offset Calculation**

**Function: `get_dest_buffer_base()`**

## Usage Patterns and Best Practices

### Typical Kernel Lifecycle

### Key Usage Guidelines

1. **Always initialize states** - Call `reset_cfg_state_id()` and `reset_dest_offset_id()` at kernel entry to ensure consistent starting state.
2. **Flip states together** - When implementing double-buffering, flip both `cfg_state_id` and `dest_offset_id` at the same point in the pipeline to maintain synchronization.
3. **Use access functions** - Never directly access hardware configuration registers; always use `cfg_read()`, `cfg_write()`, or `get_cfg_pointer()` to ensure correct bank selection.
4. **Account for pipeline delay** - The two NOPs after `flip_cfg_state_id()` ensure the hardware state change propagates. Do not remove these unless you fully understand the pipeline implications.
5. **Coordinate with synchronization** - State flips should be coordinated with semaphore operations to ensure hardware has finished using the current state before switching. See [Synchronization Primitives](/tenstorrent/tt-llk/2.2-synchronization-primitives).

## Architecture-Specific Considerations

### Wormhole B0 vs Blackhole Differences

The core configuration state management system is architecturally consistent between Wormhole B0 and Blackhole, with identical APIs and state management logic. The primary differences lie in:

| Aspect | Wormhole B0 | Blackhole |
| --- | --- | --- |
| Memory Ordering | No FENCE instruction (NOP) | FENCE instruction available |
| Local Memory | 2-4K per TRISC | 3-4K per TRISC |
| Dest Register Size | Architecture-specific constant | Architecture-specific constant |
| Additional Features | - | TRISC synchronization tracking (see [Blackhole-Specific Extensions](/tenstorrent/tt-llk/2.5-blackhole-specific-extensions)) |

The `load_blocking()` and `store_blocking()` implementations differ slightly between architectures to account for memory ordering guarantees, but the configuration state management logic remains identical.

### Memory Layout Differences

The linker scripts define different memory layouts for local data storage:

**Wormhole B0 Layout:**

* TRISC Local Data: 1792 bytes each
* TRISC Stack: 256 bytes each
* Base Address: `LOCAL_MEM_BASE = 0xFFB00000`

**Blackhole Layout:**

* TRISC Local Data: 3584 bytes each (2x Wormhole)
* TRISC Stack: 512 bytes each (2x Wormhole)
* Base Address: `LOCAL_MEM_BASE = 0xFFB00000` (same)

These differences affect how much state can be maintained locally but do not impact the configuration state management APIs.
