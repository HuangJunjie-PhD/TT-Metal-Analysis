## Purpose and Scope

The Virtual Machine (VM) Backend is a code generation backend that compiles Relay IR modules into bytecode for execution on the Relay Virtual Machine. Unlike other backends that generate target-specific code (LLVM, CUDA, etc.), the VM backend produces platform-independent bytecode that is interpreted at runtime, enabling dynamic shape support, control flow, and late binding of operations.

This document covers the **compilation process** from Relay IR to VM bytecode. For information about the bytecode instruction set, see [VM Instruction Set](https://deepwiki.com/tenstorrent/tt-tvm/6.8-vm-instruction-set). For the executable format and serialization, see [VM Executable Format](https://deepwiki.com/tenstorrent/tt-tvm/6.9-vm-executable-format). For runtime execution details, see [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor).

## Optimization Passes

The VM backend applies a sequence of Relay optimization passes to prepare the IR for bytecode generation. These passes handle function inlining, closure conversion, memory planning, and allocation manifestation.

### VM-Specific Pass Pipeline

| Pass | Purpose |
| --- | --- |
| `LambdaLift` | Converts nested functions and closures into top-level globals |
| `LabelOps` | Tags operations with metadata for VM compilation |
| `MemoryPlan` | Plans memory allocation and reuse for tensors |
| `LiftConstants` | Moves large constants to a global constant pool |
| `ManifestAlloc` | Inserts explicit `memory.alloc_tensor` and `memory.alloc_storage` ops |

### Register Allocation

The VM uses a register-based execution model. Each function has a fixed-size register file, and the compiler assigns registers to intermediate values during compilation.

**Example:** When compiling `let x = a + b`, the compiler:

1.   Visits `a`, stores result in register R0
2.   Visits `b`, stores result in register R1
3.   Emits `InvokePacked(add_op, [R0, R1], R2)` where R2 = NewRegister()
4.   Binds variable `x` to register R2 in `var_register_map_`

**Sources:**[src/relay/backend/vm/compiler.cc 301-334](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L301-L334)

## Relay Expression to VM Instruction Mapping

Different Relay expression types are compiled to specific VM instruction sequences.

### Core Expression Mappings

| Relay Expression | VM Instructions | Notes |
| --- | --- | --- |
| `Constant` | `LoadConst(index, device, dst)` | Loads from constant pool |
| `Var` | (no instruction) | Reads from register in `var_register_map_` |
| `Tuple(fields)` | `AllocADT(tag, size, fields, dst)` | Allocates tuple with tag 0 |
| `TupleGetItem(tuple, i)` | `GetField(tuple_reg, i, dst)` | Extracts field i |
| `Let(var, value, body)` | Compile value, bind to register, compile body | Stateful binding |
| `If(cond, true, false)` | `If(test, target, true_off, false_off)` + branches | Control flow with offsets |
| `Call(op, args)` | Depends on op type (see below) | Multiple instruction types |
| `Function` | (closure) | `AllocClosure(func_idx, arity, free_vars, dst)` |

**Sources:**[src/relay/backend/vm/compiler.cc 368-708](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L368-L708)

### Pattern Matching Compilation

Pattern matching expressions (`Match`) are compiled into a decision tree with conditional branches.

**Example:** A pattern match on a list constructor:

```
match list {
  Cons(hd, tl) => hd
  Nil() => 0
}
```

Compiles to:

1.   `GetTag(list, R0)` - Extract constructor tag
2.   `LoadConsti(cons_tag, R1)` - Load expected tag
3.   `If(R0, R1, then_offset, else_offset)` - Branch on match
4.   Then branch: `GetField(list, 0, R2)` - Extract `hd`
5.   Else branch: Load constant 0

**Sources:**[src/relay/backend/vm/compiler.cc 167-212](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L167-L212)[src/relay/backend/vm/compiler.cc 757-807](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L757-L807)

## Compilation Context and State

The `VMCompilerContext` structure maintains shared state across the compilation process.

**Key Responsibilities:**

1.   **Function Indexing**: `global_map` assigns a unique integer index to each global function
2.   **Constant Management**: `constants` stores the constant pool, `const_device_indexes` tracks placement
3.   **Primitive Registration**: `primitive_map` maps primitive operation names to indices for runtime lookup
4.   **Device Management**: `virtual_devices_` tracks virtual device configurations for device-aware compilation

**Sources:**[src/relay/backend/vm/compiler.h 69-88](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.h#L69-L88)

## Closure and Higher-Order Function Handling

The VM backend supports first-class functions and closures through the `LambdaLift` pass and closure allocation instructions.

**Compilation Details:**

1.   **Lambda Lifting** ([src/relay/backend/vm/compiler.cc 68](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L68-L68)): Converts nested/anonymous functions to top-level globals with explicit free variable parameters
2.   **Closure Detection** ([src/relay/backend/vm/compiler.cc 252-292](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L252-L292)): Checks if a function has non-zero `Closure` attribute
3.   **Parameter Flattening**: Merges closure parameters with function parameters into a single parameter list
4.   **Closure Allocation**: Emits `AllocClosure` instruction with captured free variables
5.   **Closure Invocation**: Uses `InvokeClosure` instruction instead of `Invoke`

**Sources:**[src/relay/backend/vm/compiler.cc 252-292](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L252-L292)[src/relay/backend/vm/compiler.cc 682-698](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L682-L698)

## Device-Aware Compilation

The VM backend supports heterogeneous execution across multiple device types through device annotations and the `DeviceAwareExprFunctor`.

**Device Index Assignment:**

1.   Host device is always at index 0
2.   Additional devices are added to `virtual_devices_` as encountered during compilation
3.   `GetDeviceIndex()` returns the index for a given `VirtualDevice`
4.   VM instructions reference devices by index

**Sources:**[src/relay/backend/vm/compiler.cc 336-364](https://github.com/tenstorrent/tt-tvm/blob/02ad9c28/src/relay/backend/vm/compiler.cc#L336-L364)

## Summary

The VM Backend compiles Relay IR to platform-independent bytecode through the following process:

1.   **Optimization**: Apply VM-specific passes (LambdaLift, MemoryPlan, ManifestAlloc)
2.   **Bytecode Emission**: Traverse Relay IR with `VMFunctionCompiler`, emitting instructions
3.   **Register Allocation**: Assign virtual registers to intermediate values
4.   **Primitive Compilation**: Lower primitive operations to target code via TECompiler
5.   **Linking**: Assemble bytecode, constants, and compiled library into `Executable`

The result is a `vm::Executable` containing:

*   Bytecode for each function
*   Constant pool (NDArray values)
*   Compiled primitive library
*   Device and function metadata

This executable can be serialized, transferred over RPC, and executed by the `VirtualMachine` runtime (see [Virtual Machine Executor](https://deepwiki.com/tenstorrent/tt-tvm/7.2-virtual-machine-executor)).
