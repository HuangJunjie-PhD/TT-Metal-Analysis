This page documents the low-level register access mechanisms and Tensix Instruction Interface (TTI) used in the LLK. These facilities provide direct hardware control through memory-mapped configuration registers and specialized hardware intrinsics. The register access layer abstracts configuration state management and provides efficient read-modify-write operations for hardware control.

For information about higher-level configuration structures that use these primitives, see [Configuration State Management](https://deepwiki.com/tenstorrent/tt-llk/2.3-configuration-state-management). For synchronization primitives that use TTI instructions, see [Synchronization Primitives](https://deepwiki.com/tenstorrent/tt-llk/2.2-synchronization-primitives).

## Configuration Register Architecture

The Tensix hardware provides a memory-mapped configuration register space accessible from TRISC cores. This space is organized into two separate configuration states to enable ping-pong buffering, allowing configuration updates while hardware operates on the current state.

### Register Space Organization

**Configuration State Dual-Banking**

The hardware maintains two complete copies of configuration registers. The `cfg_state_id` variable tracks which state is active (0 or 1), and all register access functions automatically translate addresses to the appropriate state bank.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 69-70](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L69-L70)[tt_llk_wormhole_b0/common/inc/ckernel.h 189-217](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L189-L217)

### Address Translation Mechanism

The `cfg_addr()` function translates logical configuration addresses to physical addresses based on the current state:

| State ID | Address Calculation | Offset |
| --- | --- | --- |
| 0 | `cfg_addr32` | 0x0 |
| 1 | `cfg_addr32 + (CFG_STATE_SIZE * 4)` | CFG_STATE_SIZE * 16 bytes |

This translation ensures that configuration updates target the inactive state while hardware operates on the active state.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 189-192](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L189-L192)

## Register Access Functions

### Basic Configuration Register Access

The core register access functions provide read and write operations to configuration registers with automatic state translation:

**cfg_write() - Write to Configuration Register**

Writes a 32-bit value to a configuration register at the specified logical address. The address is automatically translated based on `cfg_state_id`.

**cfg_read() - Read from Configuration Register**

Reads a 32-bit value from a configuration register. Like writes, addresses are automatically translated.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 194-206](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L194-L206)[tt_llk_blackhole/common/inc/ckernel.h 202-214](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L202-L214)

### Configuration Register Pointers

**get_cfg_pointer() - Obtain Base Pointer**

Returns a pointer to the base of the current configuration state. This enables efficient sequential register access without repeated address translation.

**get_cfg16_pointer() - 16-bit Access**

Returns a 16-bit pointer for operations requiring 16-bit access granularity.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 209-227](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L209-L227)[tt_llk_blackhole/common/inc/ckernel.h 217-235](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L217-L235)

### General Register Operations

**reg_read() and reg_write() - Arbitrary Address Access**

These functions provide direct access to any memory-mapped register by absolute address, bypassing configuration state translation:

These are primarily used for accessing debug registers, wall clock counters, and other non-configuration hardware registers.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 268-278](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L268-L278)[tt_llk_blackhole/common/inc/ckernel.h 276-286](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L276-L286)

### MMIO Register Access

**mmio_register_write() - Register Space Write**

Writes to memory-mapped IO registers using register space addressing:

The function combines the register space identifier with a 6-bit address offset to form the final register address.

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 115-119](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L115-L119)[tt_llk_blackhole/common/inc/ckernel.h 123-127](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L123-L127)

## TTI Instruction Interface

TTI (Tensix Instruction Interface) instructions are hardware intrinsics that compile directly to Tensix machine instructions. These provide the lowest-level hardware control and are typically defined as compiler built-ins or inline assembly macros.

### TTI Instruction Categories

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 138-186](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L138-L186)[tt_llk_blackhole/common/inc/ckernel.h 146-194](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_blackhole/common/inc/ckernel.h#L146-L194)

### Common TTI Instructions

| Instruction | Purpose | Usage Context |
| --- | --- | --- |
| `TTI_SEMPOST(sem)` | Increment semaphore | Signal completion of work unit |
| `TTI_SEMGET(sem)` | Decrement semaphore | Acquire resource |
| `TTI_SEMWAIT(wait_res, sem, mode)` | Wait on semaphore condition | Block until zero or max |
| `TTI_SEMINIT(max, min, sem)` | Initialize semaphore | Set initial state |
| `TTI_STALLWAIT(stall_res, wait_res)` | Stall on resource | Wait for hardware unit |
| `TT_SETC16(addr, val)` | Write 16-bit config | Update hardware state |
| `TT_ZEROACC(mode, ...)` | Clear dest registers | Initialize accumulator |
| `TTI_ZEROSRC(...)` | Clear source registers | Reset input buffers |
| `TTI_MOP(type, count, zmask)` | Execute MOP | Run instruction sequence |
| `TTI_ATGETM(idx)` | Acquire mutex | Enter critical section |
| `TTI_ATRELM(idx)` | Release mutex | Exit critical section |
| `TTI_NOP` | No operation | Pipeline delay |

**Sources:**[tt_llk_wormhole_b0/common/inc/ckernel.h 138-309](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tt_llk_wormhole_b0/common/inc/ckernel.h#L138-L309)

### Hardware-Accelerated RMW - cfg_reg_rmw_tensix()

Uses TTI RMW instructions for byte-wise register modification:

The TTI instructions `TT_RMWCIB0` through `TT_RMWCIB3` each modify one byte of the 32-bit register. This approach is more efficient than software RMW because the hardware atomically performs the read-modify-write without CPU involvement.

**Template Parameters:**

*   `CfgAddr32`: Configuration register address (compile-time constant)
*   `Shamt`: Shift amount (compile-time constant)
*   `Mask`: Bit mask (compile-time constant)

The `load_replay_buf()` function on Blackhole enables recording instruction sequences:

Blackhole provides CSR access for monitoring queue and busy status:

When enabled, the hardware automatically stalls TRISC memory accesses and Tensix instructions to guarantee correct ordering, reducing the need for explicit synchronization calls.

**Synchronization Guidelines:**

## Summary

This page documented the register access and TTI instruction facilities that form the foundation of hardware control in the LLK:

*   **Configuration register architecture** with dual-state ping-pong buffering
*   **Register access functions** including cfg_write(), cfg_read(), and pointer-based access
*   **TTI instruction interface** providing hardware intrinsics for synchronization, configuration, and control
*   **Read-modify-write operations** with both software and hardware-accelerated implementations
*   **State management** for configuration and destination register contexts
*   **Architecture-specific extensions** in Blackhole including instruction gathering, replay buffers, and TTSYNC

These primitives are used throughout the higher-level LLK APIs to efficiently configure and control the Tensix hardware pipeline.
