This document provides a detailed technical overview of the test generation internals within the RiESCUE framework. It covers the interaction between the `Parser`, `Pool`, `Generator`, and `FeatMgr` components, explaining how assembly test files are processed, configured, and transformed into executable tests.

**Scope**: This page focuses on the internal architecture and data flow during test generation. For user-facing documentation on writing tests and using directives, see [Test Directives and Syntax](https://deepwiki.com/tenstorrent/riescue/2.1.2-test-directives-and-syntax). For information on how `FeatMgr` consolidates configuration, see [Feature Manager (FeatMgr)](https://deepwiki.com/tenstorrent/riescue/7.1.2-feature-manager-(featmgr)). For memory management specifics, see [Memory Management](https://deepwiki.com/tenstorrent/riescue/7.1.3-memory-management).

## Overview

Test generation in RiESCUE follows a multi-stage pipeline that transforms annotated assembly files (`.rasm` or `.s`) into fully executable ELF binaries. The process involves four primary components working together:

*   **`Parser`**: Scans the test file for RiESCUE directives and populates data structures
*   **`Pool`**: Central repository storing all parsed data (addresses, page mappings, test metadata)
*   **`FeatMgr`**: Configuration manager consolidating test requirements with system capabilities
*   **`Generator`**: Produces final assembly code, linker scripts, and page tables from parsed data

### Directive Parsing Examples

Each directive follows a specific pattern. Here are the key parsing patterns:

**Random Address Directive** (`;#random_addr(...)`):

*   Parsed by `parse_random_addr()` at [riescue/dtest_framework/parser.py 132-166](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L132-L166)
*   Creates `ParsedRandomAddress` dataclass at [riescue/dtest_framework/parser.py 719-734](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L719-L734)
*   Extracts: `name`, `type`, `size`, `align`, `and_mask`, `or_mask`, PMA attributes
*   Stored in `Pool.parsed_addrs` dictionary

**Page Mapping Directive** (`;#page_mapping(...)`):

*   Parsed by `parse_page_mappings()` at [riescue/dtest_framework/parser.py 168-572](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L168-L572)
*   Creates `ParsedPageMapping` dataclass at [riescue/dtest_framework/parser.py 736-845](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L736-L845)
*   Extracts: `lin_name`, `phys_name`, permission bits (r/w/x/u/a/d), `pagesizes`, PTE attributes
*   Supports complex nested attributes for guest-stage paging
*   Stored in `Pool.raw_page_mappings` list

**Test Header Directive** (`;#test.*`):

*   Parsed by `parse_test_header()` at [riescue/dtest_framework/parser.py 590-598](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L590-L598)
*   Creates `ParsedTestHeader` dataclass at [riescue/dtest_framework/parser.py 853-867](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L853-L867)
*   Extracts: `priv`, `env`, `paging`, `arch`, `features`, etc.
*   Single instance stored in `Pool.test_header`

## Generator Architecture

The `Generator` component is responsible for transforming parsed data and configuration into actual assembly code. While the `Generator` class itself is not shown in the provided files, we can infer its responsibilities from how it's invoked:

**Generator Initialization** (from [riescue/riescued.py 265](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/riescued.py#L265-L265)):

**Generator Responsibilities**:

| Component | Purpose | Input | Output |
| --- | --- | --- | --- |
| Address Resolution | Assigns physical addresses to random_addr directives | `Pool.parsed_addrs`, `FeatMgr.memory_map` | Resolved addresses in `Pool` |
| Page Table Generation | Creates multi-level page tables | `Pool.raw_page_mappings`, `FeatMgr.paging_mode` | Page table data structures |
| OS Code Generation | Generates trap handlers, interrupt vectors | `FeatMgr.priv_mode`, `FeatMgr.env` | Assembly code for handlers |
| Linker Script Generation | Creates memory layout | `FeatMgr.memory_map`, resolved addresses | `.ld` linker script |
| Assembly Generation | Produces final assembly | All above components | `.S` assembly file |

## Configuration Precedence

`FeatMgr` consolidates configuration from multiple sources with a specific precedence order:

The `FeatMgrBuilder` applies configuration in this order:

1.   `with_cpu_json()` - Loads base configuration from CPU config file
2.   `with_test_header()` - Overlays test-specific requirements
3.   `with_args()` - Applies CLI overrides (highest priority)

## Parsed Data Structures

The `Parser` creates strongly-typed dataclass instances for each directive type. These dataclasses are defined in [riescue/dtest_framework/parser.py 701-910](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/dtest_framework/parser.py#L701-L910):

**Key Dataclasses**:

| Dataclass | Directive | Key Fields | Purpose |
| --- | --- | --- | --- |
| `ParsedTestHeader` | `;#test.*` | `priv`, `env`, `paging`, `arch` | Test metadata and requirements |
| `ParsedRandomAddress` | `;#random_addr()` | `name`, `type`, `size`, `align`, `and_mask` | Address generation parameters |
| `ParsedRandomData` | `;#random_data()` | `name`, `type`, `and_mask`, `or_mask` | Random data generation |
| `ParsedPageMapping` | `;#page_mapping()` | `lin_name`, `phys_name`, `v`, `r`, `w`, `x` | Virtual-to-physical mappings |
| `ParsedPageMap` | `;#page_map()` | `name`, `mode` | Paging mode configuration |
| `ParsedReserveMemory` | `;#reserve_memory()` | `addr`, `size`, `addr_type` | Memory reservation |
| `ParsedVectoredInterrupt` | `;#vectored_interrupt()` | `index`, `name`, `label` | Interrupt vector setup |
| `ParsedCsrAccess` | `;#csr_rw()` | `csr_name`, `priv_mode`, `label` | CSR access code generation |
| `ParsedLeafPte` | `;#read_leaf_pte()` | `lin_name`, `paging_mode`, `label` | PTE reading code generation |

*   `add_test_header(test_header: ParsedTestHeader)`

*   `add_parsed_res_mem(res_mem: ParsedReserveMemory)`
*   `add_parsed_vectored_interrupt(interrupt: ParsedVectoredInterrupt)`

## RiescueC Integration

The test plan mode in RiescueC (`TpMode`) demonstrates how higher-level test generators use the same underlying infrastructure:

In test plan mode ([riescue/compliance/tp.py 72-113](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L72-L113)):

1.   Test plan is loaded from `coretp` library
2.   `TestPlanGenerator.build()` creates test scenarios
3.   `TestPlanGenerator.solve()` finds compatible environment using predicates from `FeatMgr`
4.   `TestPlanGenerator.generate()` produces assembly file
5.   The assembly file is passed to `RiescueD` which runs the standard pipeline

This demonstrates the composability of the test generation system - higher-level generators produce assembly with directives, which are then processed by the same Parser → Pool → Generator pipeline.

Sources: [riescue/compliance/tp.py 72-113](https://github.com/tenstorrent/riescue/blob/07cefde3/riescue/compliance/tp.py#L72-L113)

## Summary

The test generation internals follow a clean separation of concerns:

1.   **`Parser`** extracts directives and creates typed data structures
2.   **`Pool`** stores all parsed data in a central repository
3.   **`FeatMgr`** consolidates configuration from multiple sources with clear precedence
4.   **`Generator`** transforms parsed data and configuration into executable tests

This architecture enables:

*   **Extensibility**: New directive types can be added by implementing new parse methods and dataclasses
*   **Composability**: Higher-level generators (like `TpMode`) can produce assembly that feeds into the same pipeline
*   **Testability**: Each component has clear inputs/outputs and can be tested independently
*   **Flexibility**: Configuration precedence allows defaults to be overridden at multiple levels
