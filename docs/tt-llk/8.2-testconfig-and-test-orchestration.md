**Purpose**: This document describes the `TestConfig` class and the test orchestration infrastructure that manages the end-to-end lifecycle of LLK tests, from compilation to execution to validation. This includes variant hash generation, artifact caching, the two-phase PRODUCE/CONSUME split, and integration with pytest.

For information about specific test types and examples, see [Test Suite Examples and Coverage](https://deepwiki.com/tenstorrent/tt-llk/8.7-test-suite-examples-and-coverage). For details on the performance testing infrastructure, see [Performance Testing and Profiling](https://deepwiki.com/tenstorrent/tt-llk/8.10-performance-testing-and-profiling). For coverage collection mechanics, see [Coverage Collection and Analysis](https://deepwiki.com/tenstorrent/tt-llk/8.11-coverage-collection-and-analysis).

## System Overview

The test orchestration system is centered around the `TestConfig` class, which acts as a state machine and configuration manager for compiling and executing LLK kernel tests. It manages:

*   **Architecture-specific configuration** (Wormhole, Blackhole, Quasar)
*   **Compilation orchestration** with shared and variant-specific artifacts
*   **Variant identification** via SHA-256 hashing
*   **Two-phase execution** (PRODUCE/CONSUME modes) for parallelized CI/CD
*   **Runtime parameter marshalling** to L1 memory
*   **Coverage and profiling builds** with separate artifact paths

**Sources**: [tests/python_tests/helpers/test_config.py 105-442](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L105-L442)[tests/python_tests/conftest.py 98-153](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/conftest.py#L98-L153)

## Variant Identification and Hashing

Each test variant is uniquely identified by a SHA-256 hash of its compilation-relevant parameters. This enables artifact caching and reuse across test runs.

### Hash Generation Algorithm

The hash excludes runtime-only parameters like `variant_stimuli`, `runtimes`, and (optionally) `formats_config`, ensuring that only compilation-affecting parameters influence the variant ID:

**Sources**: [tests/python_tests/helpers/test_config.py 576-593](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L576-L593)

### Artifact Directory Structure

The variant hash determines the artifact directory layout:

```
/tmp/tt-llk-build/
├── shared/                          # Shared artifacts (all variants)
│   ├── obj/
│   │   ├── tmu-crt0.o
│   │   ├── brisc.o
│   │   ├── main_unpack.o
│   │   ├── main_math.o
│   │   └── main_pack.o
│   └── elf/
│       └── brisc.elf
├── shared-profiler/                 # Profiler-specific shared artifacts
│   ├── obj/
│   └── elf/
├── {test_name}/
│   └── {variant_hash}/              # Variant-specific artifacts
│       ├── obj/
│       │   ├── test_unpack.o
│       │   ├── test_math.o
│       │   └── test_pack.o
│       ├── elf/
│       │   ├── unpack.elf
│       │   ├── math.elf
│       │   └── pack.elf
│       └── .build_complete          # Marker file
└── sync_primitives/                 # File locks
    └── {variant_hash}.lock
```

## Compilation Orchestration

The compilation process is divided into **shared artifacts** (compiled once per architecture/mode) and **variant-specific artifacts** (compiled per test variant).

### Shared Artifact Build

Shared artifacts are built once and reused across all test variants. File locking ensures thread-safe compilation when using `pytest-xdist`:

The shared artifacts include:

*   `tmu-crt0.o`: Startup code for RISC-V processors
*   `brisc.o`: BRISC control processor code
*   `main_{unpack,math,pack}.o`: TRISC kernel wrapper code compiled with `-DCOMPILE_FOR_TRISC={0,1,2}`
*   `coverage.o`: Coverage instrumentation support (if enabled)
*   `brisc.elf`: Linked BRISC executable

**Sources**: [tests/python_tests/helpers/test_config.py 636-736](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L636-L736)

### Variant-Specific Build

For each test variant, the system compiles test-specific C++ files and links them with shared artifacts:

**Sources**: [tests/python_tests/helpers/test_config.py 872-976](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L872-L976)

### Build Header Generation

The `generate_build_header()` method creates a C++ header file (`build.h`) containing all compile-time configuration:

| Header Section | Content | Example |
| --- | --- | --- |
| **Format Config Struct** | `struct FormatConfig` definition | `FORMATS_CONFIG_STRUCT_COMPILETIME` or `FORMATS_CONFIG_STRUCT_RUNTIME` |
| **Template Parameters** | Compile-time constants | `constexpr bool APPROX_MODE = true;` |
| **Data Formats** | Format enumerations | `constexpr auto UNPACK_A_IN = DataFormat::Float16_b;` |
| **Runtime Args Struct** | `struct RuntimeParams` layout | Fields for tile sizes, formats, custom params |

## Profiler and Coverage Builds

The `TestConfig` class handles special build configurations for profiling and coverage:

### Profiler Build Configuration

Profiler builds require separate shared artifacts because `trisc.cpp` is compiled with `-DLLK_PROFILER`, which changes the code generation:

**Sources**: [tests/python_tests/helpers/test_config.py 638-675](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L638-L675)

### Coverage Build Configuration

Coverage builds modify:

*   **Compilation flags**: Add `-fprofile-arcs -ftest-coverage -fprofile-info-section -DCOVERAGE`
*   **Linker scripts**: Use `memory.{arch}.debug.ld` with expanded memory regions
*   **Runtime address**: Use `0x6E000` instead of `0x20000` to avoid data collisions
*   **Additional object**: Link with `coverage.o` and `-lgcov`

**Memory Layout Comparison**:

| Build Type | Runtime Args | Code Size | Data Size | GCOV Memory |
| --- | --- | --- | --- | --- |
| **Regular** | 0x20000 | 16K-32K | 8K-16K | N/A |
| **Coverage** | 0x6E000 | 32K-64K | 32K | 8K-32K per TRISC |

**Sources**: [tests/python_tests/helpers/test_config.py 622-629](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L622-L629)[tests/helpers/ld/memory.wormhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.wormhole.debug.ld#L1-L68)[tests/helpers/ld/memory.blackhole.debug.ld 1-68](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/helpers/ld/memory.blackhole.debug.ld#L1-L68)

## Artifact Caching and Lock Mechanism

The system uses file locks and marker files to enable safe parallel compilation with `pytest-xdist`:

### Lock Strategy

The two-level locking strategy ensures:

1.   Only one process builds shared artifacts at a time
2.   Multiple processes can build different variants simultaneously
3.   Marker files enable fast-path checks before acquiring locks

**Sources**: [tests/python_tests/helpers/test_config.py 665-675](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L665-L675)[tests/python_tests/helpers/test_config.py 889-896](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L889-L896)

### Cache Validation

Both shared and variant builds use marker files (`.shared_complete`, `.build_complete`) to indicate successful completion. The check-acquire-double-check pattern prevents race conditions:

**Sources**: [tests/python_tests/helpers/test_config.py 656-674](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L656-L674)[tests/python_tests/helpers/test_config.py 884-896](https://github.com/tenstorrent/tt-llk/blob/366f58e2/tests/python_tests/helpers/test_config.py#L884-L896)
