This document describes vLLM's model registry system and the mechanisms for loading and initializing models. The registry maintains a catalog of supported model architectures, handles configuration parsing from various formats, and manages the dynamic loading of model implementations.

For information about specific model implementations and their capabilities, see [Supported Models](https://deepwiki.com/tenstorrent/vllm/1.2-supported-models). For details about how models are executed after loading, see [Worker Architecture](https://deepwiki.com/tenstorrent/vllm/2.5-worker-architecture).

* * *

## Overview

vLLM's model registry serves as the central directory for all supported model architectures. It maps Hugging Face (HF) architecture names to vLLM model implementations and provides facilities for:

*   Registering native vLLM models and Transformers-backed models
*   Lazy loading of model classes to avoid CUDA initialization in the main process
*   Inspecting model capabilities without full instantiation
*   Parsing configuration files from multiple formats (HF, Mistral)
*   Managing custom configuration classes for special architectures

The registry is implemented in [vllm/model_executor/models/registry.py](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py) and exposed through the `ModelRegistry` singleton.

**Sources**: [vllm/model_executor/models/registry.py 560-696](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L560-L696)[vllm/model_executor/models/__init__.py 11-33](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/__init__.py#L11-L33)

## Registry Architecture

**Key Components**:

*   **`_ModelRegistry`**: The core registry class that maintains a `models` dictionary mapping architecture names to `_BaseRegisteredModel` instances
*   **`_RegisteredModel`**: Represents a model already imported in the main process, contains both the model class and pre-computed metadata
*   **`_LazyRegisteredModel`**: Represents a model not yet imported, stores only the module and class name for deferred loading
*   **`_ModelInfo`**: Dataclass containing model capability flags extracted via introspection

**Sources**: [vllm/model_executor/models/registry.py 354-396](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L354-L396)[vllm/model_executor/models/registry.py 409-530](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L409-L530)[vllm/model_executor/models/registry.py 560-696](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L560-L696)

## Model Registration

### Static Model Dictionaries

Models are initially registered through static dictionaries that map HF architecture names to module paths:

| Dictionary | Purpose | Example Entry |
| --- | --- | --- |
| `_TEXT_GENERATION_MODELS` | Decoder-only text generation models | `"LlamaForCausalLM": ("llama", "LlamaForCausalLM")` |
| `_EMBEDDING_MODELS` | Embedding and pooling models | `"BertModel": ("bert", "BertEmbeddingModel")` |
| `_CROSS_ENCODER_MODELS` | Cross-encoder/reranker models | `"BertForSequenceClassification": (...)` |
| `_MULTIMODAL_MODELS` | Vision, audio, video models | `"LlavaForConditionalGeneration": (...)` |
| `_SPECULATIVE_DECODING_MODELS` | Models for speculative decoding | `"MedusaModel": ("medusa", "Medusa")` |
| `_TRANSFORMERS_BACKEND_MODELS` | Generic Transformers backend wrappers | `"TransformersForCausalLM": (...)` |

All dictionaries are merged into `_VLLM_MODELS`[vllm/model_executor/models/registry.py 321-329](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L321-L329)

## Configuration Parsing and Loading

vLLM supports multiple configuration formats through a parser abstraction:

### Config Format Detection

Config format is determined via `get_config`[vllm/transformers_utils/config.py 469-528](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L469-L528):

1.   **Auto-detection**: Check for existence of `params.json` (Mistral) vs `config.json` (HF)
2.   **Explicit format**: User can specify `config_format="hf"` or `config_format="mistral"`

### Custom Config Registry

Special architectures require custom config classes registered in `_CONFIG_REGISTRY`[vllm/transformers_utils/config.py 66-87](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L66-L87):

The `LazyConfigDict` defers imports until access via `__getitem__`[vllm/transformers_utils/config.py 59-63](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L59-L63)

**Sources**: [vllm/transformers_utils/config.py 59-101](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L59-L101)[vllm/transformers_utils/config.py 103-202](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/config.py#L103-L202)[vllm/transformers_utils/configs/__init__.py 1-64](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/transformers_utils/configs/__init__.py#L1-L64)

## Model Class Loading

**Key Functions**:

*   **`_try_load_model_cls()`**[vllm/model_executor/models/registry.py 532-544](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L532-L544): Cached loader that verifies platform compatibility before loading
*   **`load_model_cls()`**[vllm/model_executor/models/registry.py 527-529](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L527-L529): Performs the actual import using `importlib.import_module()`
*   **Platform verification**: `current_platform.verify_model_arch()` checks if the model is supported on the current hardware

**LRU Caching**: Both `_try_load_model_cls()` and `_try_inspect_model_cls()` are decorated with `@lru_cache(maxsize=128)` to avoid redundant work.

**Sources**: [vllm/model_executor/models/registry.py 527-557](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L527-L557)

## Transformers Backend Integration

vLLM provides a "Transformers backend" that wraps arbitrary Transformers models without requiring native vLLM implementations:

**Backend Models** are registered in `_TRANSFORMERS_BACKEND_MODELS`[vllm/model_executor/models/registry.py 309-318](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L309-L318) and include:

*   `TransformersForCausalLM`: Generic causal LM wrapper
*   `TransformersForMultimodalLM`: Generic multimodal wrapper
*   `TransformersMoEForCausalLM`: MoE model wrapper
*   `TransformersEmbeddingModel`: Embedding model wrapper

Models can be explicitly routed to the Transformers backend using `model_impl="transformers"`[docs/models/supported_models.md 48](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1#L48-L48)

**Sources**: [vllm/model_executor/models/registry.py 302-318](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L302-L318)[docs/models/supported_models.md 16-121](https://github.com/tenstorrent/vllm/blob/cd9e5b83/docs/models/supported_models.md?plain=1#L16-L121)

## Registry Public API

The `ModelRegistry` singleton provides the following public methods:

| Method | Purpose |
| --- | --- |
| `register_model(arch, model_cls)` | Register a new model architecture |
| `get_supported_archs()` | Get set of all registered architectures |
| `is_text_generation_model(arch)` | Check if arch is text generation model |
| `is_pooling_model(arch)` | Check if arch is pooling/embedding model |
| `is_pp_supported_model(arch)` | Check if pipeline parallelism is supported |
| `is_multimodal_model(arch)` | Check if multimodal inputs are supported |
| `inspect_model_cls(arch)` | Get full `_ModelInfo` for architecture |
| `resolve_model_cls(model_config)` | Load and return the model class |

**Sources**: [vllm/model_executor/models/registry.py 697-837](https://github.com/tenstorrent/vllm/blob/cd9e5b83/vllm/model_executor/models/registry.py#L697-L837)
