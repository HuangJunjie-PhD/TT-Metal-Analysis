This document covers tt-torch's device management infrastructure and parallelism strategies for executing models across multiple Tenstorrent hardware devices. For information about the compilation pipeline that produces executables for these devices, see [Compilation Pipeline](https://deepwiki.com/tenstorrent/tt-torch/3.1-compilation-pipeline). For details on how executors use devices during model execution, see [Executor Framework](https://deepwiki.com/tenstorrent/tt-torch/3.3-executor-framework).

## Overview

tt-torch provides a hierarchical device management system through the `DeviceManager` class and supports multiple parallelism strategies including data parallelism, batch parallelism, and pipeline parallelism. The system is built on tt-mlir's mesh device abstraction, which allows flexible device topology configuration and efficient multi-device execution.

**Key Components:**

*   `DeviceManager`: Static class for acquiring and managing parent and sub-mesh devices
*   `Executor`: Base class that integrates device management into the execution flow
*   `MultiChipGraph`: Data structure for splitting computation across multiple devices
*   `BackendOptions`: Configuration object for passing device settings to compilation backend

## Device Hierarchy

tt-torch uses a two-level device hierarchy:

### Parent Mesh Device

A parent mesh device represents a logical grouping of physical devices arranged in a 2D mesh topology. The mesh shape is specified as `[rows, columns]` where the total number of devices equals `rows * columns`.

**Diagram: Parent Mesh Device with 2x2 Topology**

Sources: [tt_torch/tools/device_manager.py 49-79](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/device_manager.py#L49-L79)

### Sub-Mesh Devices

Sub-mesh devices are logical partitions of a parent mesh, allowing different parts of a model to execute on different device subsets. Each sub-mesh is defined by an offset and shape within the parent mesh.

**Diagram: Sub-Mesh Devices within Parent Mesh**

Sources: [tt_torch/tools/device_manager.py 111-140](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/device_manager.py#L111-L140)

## DeviceManager API

The `DeviceManager` class provides a centralized interface for device lifecycle management. All methods are static, maintaining a global registry of acquired devices.

### Core Operations

| Method | Purpose | Returns |
| --- | --- | --- |
| `create_parent_mesh_device()` | Acquire a new parent mesh with specified topology | `tt_mlir.Device` |
| `create_sub_mesh_device()` | Create a sub-mesh within a parent mesh | `tt_mlir.Device` |
| `release_parent_device()` | Close a parent mesh and optionally its sub-meshes | None |
| `release_sub_mesh_device()` | Close a specific sub-mesh device | None |
| `acquire_available_devices()` | Create parent mesh and 1x1 sub-meshes for all devices | `(parent, [devices])` |
| `get_num_available_devices()` | Query total available physical devices | `int` |

### Creating Parent Mesh Devices

**Diagram: Parent Mesh Device Creation Sequence**

The parent mesh is created with configurable options:

*   `mesh_shape`: 2D topology as `[rows, cols]`
*   `device_ids`: Specific physical device IDs to use
*   `num_hw_cqs`: Number of hardware command queues
*   `enable_program_cache`: Whether to cache compiled programs
*   `l1_small_size`: L1 cache size configuration
*   `dispatch_core_type`: Core type for dispatch operations

## Device Configuration in Executors

### BackendOptions Class

The `BackendOptions` class bridges user configuration to the compilation backend:

**Diagram: BackendOptions to Executor Relationship**

The `devices` field can be:

*   `None`: Executor creates and owns default devices
*   `[None]`: Single device with default mesh shape

**Diagram: Layer Assignment via Device Map**

Sources: [tt_torch/tools/utils.py 332-333](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/utils.py#L332-L333)[tt_torch/dynamo/passes.py 136-169](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/passes.py#L136-L169)

### Split Pipeline Process

The `split_onto_devices()` function implements the graph splitting logic:

**Diagram: Graph Splitting Process**

Key steps:

1.   **Validate and sort device_map**: Ensure inputs are placed before their consumers in the topological order
2.   **Create per-device graphs**: Empty subgraphs for each device
3.   **Partition nodes**: Assign each node to its target device based on `device_map`
4.   **Handle cross-device edges**: Insert `MultiChipOutput` and `MultiChipInput` for inter-device transfers
5.   **Link intermediate tensors**: Connect outputs to corresponding inputs

Sources: [tt_torch/dynamo/passes.py 343-532](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/passes.py#L343-L532)

### Node-to-Device Resolution

The `node_to_device()` function extracts device assignment from a node's metadata:

**Diagram: Resolving Node to Device Assignment**

The function progressively builds layer names (e.g., `model`, `model.layers`, `model.layers.15`) until finding a match in `device_map`.

## Parallelism Strategies

tt-torch supports multiple parallelism strategies depending on model architecture and hardware configuration.

### Data Parallelism

Data parallelism replicates the model across devices, with each device processing a different batch of data:

**Diagram: Data Parallel Execution**

Implementation in `ModelTester`:

**Diagram: Data Parallel Test Execution Flow**

The `data_parallel_mode` flag enables this behavior:

*   Compilation: One model compiled per device with `async_mode=True`
*   Execution: Each model runs asynchronously on its assigned device
*   Verification: Each device's output is verified independently against golden output

Sources: [tests/utils.py 186-200](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/utils.py#L186-L200)[tests/utils.py 394-401](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/utils.py#L394-L401)[tests/utils.py 672-708](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tests/utils.py#L672-L708)

### Pipeline Parallelism

Pipeline parallelism splits the model sequentially across devices, with each device processing a different stage:

**Diagram: Pipeline Parallel Execution**

Implemented via `device_map` splitting. Intermediate tensors flow through `MultiChipOutput` and `MultiChipInput` pairs.

Sources: [tt_torch/dynamo/passes.py 343-532](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/passes.py#L343-L532)

### Batch Parallelism (N300 Specific)

Batch parallelism is a specialized mode for N300 hardware where a batch is split across a 2D mesh:

**Diagram: Batch Parallel on N300 Hardware**

This is a hardware-specific optimization where the batch dimension is automatically partitioned by tt-mlir based on the mesh shape.

Sources: High-level diagrams mention this as "Batch Parallel N300 Specific"

### Automatic Parallelization

StableHLO IR can be automatically parallelized for mesh execution:

**Diagram: Automatic Parallelization Pipeline**

When `compiler_config.automatic_parallelization=True` and `mesh_shape` is configured, tt-mlir analyzes the StableHLO graph and inserts collective operations for data distribution and gathering across the mesh.

Sources: [tt_torch/dynamo/backend.py 241-249](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L241-L249)

The `create_device_mesh()` helper creates an XLA mesh:

**Diagram: Creating XLA Device Mesh**

The mesh is stored in `CompilerConfig.xla_mesh` and used by the XLA backend for distributed execution.

### XLA vs Legacy Backend Device Handling

| Aspect | tt-legacy Backend | tt-experimental Backend |
| --- | --- | --- |
| Device Management | Direct tt-mlir Device objects | XLA Mesh abstraction |
| Parallelism | Manual via device_map splitting | SPMD with mesh annotations |
| Multi-chip | MultiChipGraph splitting | XLA automatic sharding |

## Async Execution Mode

The `async_mode` flag enables device tensors to remain on-device between operations:

**Diagram: Async Mode Execution Flow**

Async mode is primarily used in data parallel execution where each device operates independently and the final `to_host()` transfer happens after all devices complete.

Sources: [tt_torch/dynamo/executor.py 458-568](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L458-L568)[tt_torch/tools/verify.py 365-419](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/tools/verify.py#L365-L419)

## Runtime Tensor Caching

Executors maintain caches for runtime tensors to avoid redundant preprocessing:

**Diagram: Shared Tensor Cache Architecture**

Cache behavior:

*   **Buffer cache**: For model buffers (e.g., static KV caches in transformers)
*   **Constant cache**: For model weights and constants
*   **Sharing**: Multiple executors can share the same cache by passing the same `BackendOptions` instance
*   **Invalidation**: Caches are validated against the current graph's constants and invalidated if mismatched

Sources: [tt_torch/dynamo/backend.py 21-54](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/backend.py#L21-L54)[tt_torch/dynamo/executor.py 176-203](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L176-L203)[tt_torch/dynamo/executor.py 284-324](https://github.com/tenstorrent/tt-torch/blob/f5e3955f/tt_torch/dynamo/executor.py#L284-L324)

## System Descriptor Creation

Each device requires a system descriptor file containing hardware-specific configuration:

**Diagram: System Descriptor Lifecycle**

System descriptors are created during executor initialization and cleaned up in the destructor. They contain device-specific information needed by the TTIR→TTNN compilation stage.
