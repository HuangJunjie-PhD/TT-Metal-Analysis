## Purpose and Scope

This page provides an in-depth technical overview of the multi-device and distributed execution capabilities of the `tt-metal` codebase. The primary goal is to explain the mechanisms supporting tensor parallelism (TP), data parallelism (DP), hybrid TP+DP configurations, and submesh strategies to scale large neural network models across multiple Tenstorrent hardware devices.

The infrastructure is centered around the `MeshDevice` abstraction, enabling seamless execution across many physical devices, from small 2-chip setups (N300/P300) to large 32-chip Galaxy clusters, and extends to multi-host clusters with mesh fabrics.

Detailed explanation of distributed tensor management, collective communication, sub-device partitioning, and multi-host scaling is provided, referencing key code entities and data flow.

* * *

## Multi-Device Abstraction: MeshDevice

The `MeshDevice` class is the cornerstone of the multi-device execution framework. It aggregates multiple physical devices into a single logical device interface compatible with the rest of the TTNN library.

*   **Inheritance and Interface:**`MeshDevice` implements the `IDevice` interface, allowing it to plug into existing code written for single devices without modification [tt_metal/api/tt-metalium/mesh_device.hpp 75-96](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/mesh_device.hpp#L75-L96)

*   **Key Components:**

    *   **MeshDeviceImpl:** The hidden implementation managing the lifecycle of physical devices, coordinate translation, and communication setup [tt_metal/distributed/mesh_device.cpp 98-114](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_device.cpp#L98-L114)
    *   **MeshDeviceView:** Logical sub-regions of the mesh exposing subsets of devices, enabling sub-mesh strategies that partition the device grid for different model shards or parallelism types [tt_metal/distributed/mesh_device.cpp 149-158](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_device.cpp#L149-L158)

*   **Command Queue:** A distributed `MeshCommandQueue` broadcasts or multicasts commands across devices or subsets of devices efficiently, abstracting the command dispatch layer from users [tt_metal/api/tt-metalium/mesh_device.hpp 63-67](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/mesh_device.hpp#L63-L67)

### Initialization Flow

`MeshDevice` creation goes through these steps:

1.   The user calls `MeshDevice::create()` with a configuration describing the shape and number of devices.
2.   Internally, `MeshDeviceImpl` acquires handles to physical devices discovered through runtime managers like `MetalContext` and `Cluster`[tt_metal/api/tt-metalium/mesh_device.hpp 163-170](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/api/tt-metalium/mesh_device.hpp#L163-L170)
3.   These device handles are wrapped in scoped pointers and logically arranged in the device mesh shape.
4.   Communication fabric and routing tables are configured for multi-device collective operations.

## Distributed Tensors and Parallelism

### Tensor Distribution Concepts

The `ttnn::distributed` namespace provides functionality to shard or replicate tensors across devices in the mesh to enable parallelism for large models.

*   **Distribution Modes:**

    *   `ROW_MAJOR`: Shards are mapped across devices using row-major coordinate ordering.
    *   `SUBMESH`: Tensors are mapped to a sub-region (submesh) of the full device mesh, allowing logical subdivisions of hardware for different parallelism tasks [ttnn/core/distributed/distributed_tensor.cpp 42-46](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/distributed/distributed_tensor.cpp#L42-L46)

*   **TensorToMesh:** This functor transforms host tensors into distributed tensors by applying the mesh mapping specified in `MeshMapperConfig`. It manages offsets, shapes, and memory pinning to deliver consistent shards onto devices [ttnn/core/distributed/distributed_tensor.cpp 110-121](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/core/distributed/distributed_tensor.cpp#L110-L121)

*   **MeshMapperConfig:** Defines dimension placements as `PlacementShard` or `PlacementReplicate`, specifying how particular tensor axes distribute across mesh dimensions [ttnn/ttnn/distributed/distributed.py 47-53](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/ttnn/distributed/distributed.py#L47-L53)

*   **TensorShardingInfo:** Provides helper computations and visualization for how tensors are partitioned and offsets for shards [ttnn/ttnn/distributed/distributed.py 22-42](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/ttnn/distributed/distributed.py#L22-L42)

### Parallelism Types Supported

*   **Tensor Parallelism (TP):** Splitting large tensors (e.g., embeddings, attention heads) across the mesh width or height, distributing the model's internal units.
*   **Data Parallelism (DP):** Replicating the full model copy across multiple devices, splitting batches across the mesh.
*   **Hybrid TP+DP:** Combinations where tensor dimensional axes and data batches are split across different dimensions of the mesh or via submeshes.

## Collective Communication Operations (CCL)

CCL operations provide the essential collective communication primitives required for synchronizing or exchanging data among multiple devices in TP and DP.

*   **Supported Topologies:** The system detects and supports various topologies defined in the fabric and mesh, including `Linear`, `Ring`, `Mesh`, and `Torus`[ttnn/cpp/ttnn/operations/ccl/ccl_common.cpp 27-44](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/ttnn/cpp/ttnn/operations/ccl/ccl_common.cpp#L27-L44)

*   **Boundary Modes:** The `get_boundary_mode` method infers whether communication wraps around (Torus) or not (Linear/Mesh), affecting algorithmic behavior of collectives.

*   **Implementation Components:**

    *   `CclOpTensorConfig` defines collective operation layouts and parameters.
    *   `LineTopology` and `RingTopology` encapsulate communication logic for those shapes.
    *   `CclCommandStreamBuilder` constructs command sequences for execution.
    *   `EriscDatamoverBuilder` and device kernels implement the low-level data movement.

## Sub-Device Management and Isolation

For large hardware devices, finer resource partitioning enables multi-tenancy or isolated sub-workloads within the same physical device.

*   **SubDeviceManager:** Manages partitioning of cores and memory banks into independent `SubDevices`, each with dedicated resources for isolation [tt_metal/impl/sub_device/sub_device_manager.cpp 49-91](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/impl/sub_device/sub_device_manager.cpp#L49-L91)

*   **Core Ranges and Resource Allocation:** SubDevices are assigned specific core ranges and L1 memory banks, enabling non-overlapping execution contexts.

*   **Stall Groups:** These are named groups of sub-devices that can be stalled simultaneously for synchronization or debugging.

*   **Independent Memory Allocators:** Each `SubDevice` uses its own allocator like `L1BankingAllocator` to avoid fragmentation and interference.

## Multi-Host and Scale-Out

`tt-metal` supports scaling beyond single hosts using multi-host fabric communication under the `tt-fabric` system.

*   **SystemMesh:** Provides a global device topology spanning all hosts, mapping fabric node IDs for inter-host routing [tt_metal/distributed/mesh_device.cpp 149-154](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_device.cpp#L149-L154)

*   **FabricNodeId:** Each device port on a host is assigned a fabric node ID, critical for multi-host collective routing.

*   **Coordinate Translation:** The function `compute_system_mesh_offset` maps local device coordinates in a host to global coordinates across hosts [tt_metal/distributed/mesh_device.cpp 149-158](https://github.com/tenstorrent/tt-metal/blob/f30f8df0/tt_metal/distributed/mesh_device.cpp#L149-L158)

## Hybrid Tensor and Data Parallelism Support

The system supports hybrid parallelism schemes combining tensor and data parallelism to optimize scaling for complex, large models:

*   Tensor axes (such as heads or embedding dimensions) are sharded across mesh axes (tensor parallelism).
*   The batch dimension is split for data parallelism, replicating weights but splitting input data.
*   The mesh can be partitioned into submeshes representing data parallel groups and tensor parallel groups.

## Summary

The `tt-metal` multi-device and distributed execution system is architected to support efficient large model scaling on Tenstorrent hardware. Through abstractions like `MeshDevice` and `MeshDeviceView` along with distributed tensor management via `TensorToMesh`, the system provides a flexible foundation for tensor parallelism, data parallelism, and hybrid approaches. Integration with collective communication operations (`TT_CCL`) and sub-device partitioning empowers isolation and performance optimization crucial for large cluster deployments. Rigorous testing fixtures capture multi-device scenarios affirming robustness.
