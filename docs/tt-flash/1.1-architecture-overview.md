## Purpose and Scope

This document describes the layered architecture of tt-flash, a firmware flashing utility for Tenstorrent AI accelerator chips. The architecture follows a four-layer design pattern that separates user interface concerns from hardware communication details through well-defined abstraction boundaries.

This page covers the structural organization of the codebase and the responsibilities of each architectural layer. For detailed information about specific aspects:

*   For supported hardware platforms and board identification, see [Platform Support Matrix](https://deepwiki.com/tenstorrent/tt-flash/4.1-platform-support-matrix) and [Board Identification](https://deepwiki.com/tenstorrent/tt-flash/4.2-board-identification)
*   For the flashing workflow and stages, see [Flash Stages](https://deepwiki.com/tenstorrent/tt-flash/5.2-flash-stages)
*   For chip-specific implementations, see [Wormhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.3-wormhole-platform) and [Blackhole Platform](https://deepwiki.com/tenstorrent/tt-flash/4.4-blackhole-platform)
*   For hardware communication protocols, see [Hardware Communication Interfaces](https://deepwiki.com/tenstorrent/tt-flash/6.3-hardware-communication-interfaces)

## Layer 3: Hardware Abstraction Layer

The Hardware Abstraction Layer provides a uniform interface for interacting with different Tenstorrent chip architectures through an inheritance hierarchy centered on the `TTChip` abstract base class.

### Class Hierarchy

### Chip Detection Functions

The abstraction layer provides two detection functions with different initialization guarantees:

| Function | Location | Initialization | Use Case |
| --- | --- | --- | --- |
| `detect_local_chips()` | [chip.py 308-374](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L308-L374) | Full chip initialization with callback | Primary detection for flash operations |
| `detect_chips()` | [chip.py 377-387](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L377-L387) | Basic communication only | Post-reset re-detection |

### Firmware Version Retrieval

Version retrieval differs between architectures:

**Wormhole:** Uses legacy ARC message protocol via `get_bundle_version_v1()`[chip.py 29-91](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L29-L91)

*   Queries `MSG_TYPE_FW_VERSION` with different `arg0` values to retrieve running and SPI versions
*   Minimum firmware version check: `0x2170000`[chip.py 298-299](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L298-L299)

**Blackhole:** Uses telemetry and boot filesystem [chip.py 248-275](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L248-L275)

*   Running version from `Telemetry.fw_bundle_version`[chip.py 254-260](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L254-L260)
*   SPI version from `decode_boot_fs_table("cmfwcfg")`[chip.py 263-269](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L263-L269)

## Layer 4: Hardware Communication Layer

The Hardware Communication Layer is provided entirely by the `pyluwen` library (version `~0.8.0` as specified in [pyproject.toml 26](https://github.com/tenstorrent/tt-flash/blob/60c06032/pyproject.toml#L26-L26)), which handles all low-level hardware interactions.

### pyluwen Integration Points

### Wrapped pyluwen Operations

The `TTChip` class wraps `PciChip` operations [chip.py 205-230](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L205-L230) with minimal transformation:

| TTChip Method | pyluwen Method | Data Flow |
| --- | --- | --- |
| `axi_write32()` | `PciChip.axi_write32()` | Direct passthrough |
| `axi_read32()` | `PciChip.axi_read32()` | Direct passthrough |
| `spi_write()` | `PciChip.spi_write()` | Direct passthrough |
| `spi_read()` | `PciChip.spi_read()` | Bytearray allocation wrapper |
| `arc_msg()` | `PciChip.arc_msg()` | Direct passthrough |
| `get_telemetry()` | `PciChip.get_telemetry()` | Cached wrapper [chip.py 163-165](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L163-L165) |

### Firmware Definitions

Hardware message types are loaded from `fw_defines.yaml`[chip.py 109-110](https://github.com/tenstorrent/tt-flash/blob/60c06032/chip.py#L109-L110) and stored in the `fw_defines` dictionary. These definitions map to ARC processor message types:

*   `MSG_TYPE_ARC_STATE3` - State query message
*   `MSG_TYPE_FW_VERSION` - Firmware version query
*   `MSG_TRIGGER_SPI_COPY_LtoR` - Trigger NEBULA_X2 local-to-remote SPI copy [flash.py 642](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L642-L642)
*   `MSG_CONFIRM_FLASHED_SPI` - Post-flash validation challenge [flash.py 946](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L946-L946)
*   `MSG_UPDATE_M3_AUTO_RESET_TIMEOUT` - M3 reset timeout adjustment [flash.py 630](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L630-L630)

## Cross-Layer Data Flow Patterns

### Firmware Bundle Loading and Extraction

The manifest loading sequence [main.py 179-201](https://github.com/tenstorrent/tt-flash/blob/60c06032/main.py#L179-L201) extracts version metadata, which is then validated by `verify_package()`[flash.py 663-697](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L663-L697) using version-specific validation functions defined in `BUNDLE_VALIDATION_FUNCS`[flash.py 658-661](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L658-L661)

### Board Identification and Name Resolution

Board identification flows through multiple transformations:

Implementation:

*   `get_board_type()`[utility.py 42-93](https://github.com/tenstorrent/tt-flash/blob/60c06032/utility.py#L42-L93) - Extracts UPI and revision, performs lookup
*   `change_to_public_name()`[utility.py 96-136](https://github.com/tenstorrent/tt-flash/blob/60c06032/utility.py#L96-L136) - Maps codenames to user-facing strings

### Version Comparison Logic

Version normalization and comparison spans multiple layers:

1.   **Retrieval** (Layer 3): Architecture-specific version retrieval
2.   **Normalization** (Layer 2): `normalize_fw_version()`[flash.py 127-139](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L127-L139) converts legacy `(80, major, minor, patch)` format to `(major, minor, patch, 0)`
3.   **Comparison** (Layer 2): `flash_chip_stage1()` performs multi-stage version comparison [flash.py 263-334](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L263-L334)
    *   Running vs manifest version
    *   SPI vs manifest version
    *   Major version boundary detection [flash.py 274-289](https://github.com/tenstorrent/tt-flash/blob/60c06032/flash.py#L274-L289)
