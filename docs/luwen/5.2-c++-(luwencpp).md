## Purpose and Scope

The luwencpp package provides C++ bindings for the Luwen framework, enabling C++ applications to interface with Tenstorrent AI accelerator chips. These bindings expose core Luwen functionality through a C-compatible API that can be consumed from C++ codebases. The luwencpp library compiles to a shared object (.so) file and provides a corresponding header file for inclusion in C++ projects.

For Python bindings to the same functionality, see [Python (pyluwen)](https://deepwiki.com/tenstorrent/luwen/5.1-python-(pyluwen)).

## Architecture Overview

The luwencpp bindings serve as a bridge between C++ applications and the Luwen framework's Rust implementation. They provide a thin wrapper around Luwen's core functionality, allowing C++ code to interact with Tenstorrent chips.

### Communication Interface (LuwenGlue)

The `LuwenGlue` structure forms the bridge between the C++ application and the Luwen framework. It contains function pointers for various operations that the C++ application must implement to enable communication with the chip.

The `LuwenGlue` structure provides the following key interfaces:

1.   **Device Information**: Function to retrieve basic information about the chip
2.   **AXI Operations**: Low-level register access (read/write)
3.   **Network-on-Chip (NOC) Operations**: Memory operations across the chip (read/write/broadcast)
4.   **Ethernet Operations**: Communication with remote chips (read/write/broadcast)

### Chip Lifecycle Management

The C++ bindings provide functions to create, manage, and release chip instances:

| Function | Description |
| --- | --- |
| `luwen_open` | Opens a connection to a local PCI chip |
| `luwen_open_remote` | Opens a connection to a remote chip via Ethernet |
| `luwen_close` | Closes a chip connection and releases resources |
| `chip_arc_msg` | Sends a message to the ARC processor on the chip |
| `chip_telemetry` | Retrieves telemetry data from the chip |

## Chip Initialization Process

When opening a chip using the C++ bindings, the Luwen framework performs a comprehensive initialization process to ensure the chip is ready for operation. This process involves checking the status of various chip components and waiting for them to be ready.

The initialization process checks the status of each component and creates appropriate status objects to track their state:

| Component | Status Information |
| --- | --- |
| Communication | Can communicate with the chip |
| ARC | ARC processor initialization status |
| DRAM | DRAM memory initialization and training status |
| Ethernet | Ethernet interface initialization status |

## Error Handling

The C++ bindings provide a structured approach to error handling through the `CResult` type, which encapsulates success values and error messages:

When functions like `chip_arc_msg` return a `CResult`, C++ applications should:

1.   Check the `tag` field to determine if the operation succeeded or failed
2.   If successful, use the `ok` value
3.   If failed, handle the error message in `err`

## Building and Integration

The luwencpp crate is compiled into a shared library (`libluwencpp.so`) and includes a C header file (`luwen.h`) that can be included in C++ projects. These files are generated during the build process:

1.   The Rust code is compiled into a dynamic library
2.   cbindgen generates the C header file from the Rust code

To use luwencpp in a C++ project:

1.   Include the generated header file: `#include "luwen.h"`
2.   Link against the shared library: `-lluwencpp`
3.   Implement the required callback functions
4.   Use the provided API to interact with Tenstorrent chips
