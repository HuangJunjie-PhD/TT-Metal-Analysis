The Packer System is the data output stage of the Tensix Coprocessor pipeline, responsible for reading computed results from internal buffers (`Dst` register file or L1 memory), performing format conversion, optional compression, optional accumulation, and writing the final output to L1 memory. The system consists of four independent packers (Packer 0-3), each executing `PACR` instructions to orchestrate multi-stage data output operations.

For detailed information about specific packer instructions, see [Packer Instructions (PACR)](https://deepwiki.com/tenstorrent/tt-isa-documentation/5.7-packer-instructions-(pacr)). For configuration mechanisms, see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system). For details about ADCs used in address generation, see [Backend Configuration System](https://deepwiki.com/tenstorrent/tt-isa-documentation/4.2-backend-configuration-system).

## System Architecture

The Tensix tile contains four independent packer units, numbered 0 through 3. Each packer operates as a pipeline with distinct stages for input address generation, data transformation, compression (optional), format conversion, accumulation (optional), and output address generation.

**Sources:**[WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)[WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md 1-102](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/InputAddressGenerator.md?plain=1#L1-L102)[WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md 1-121](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/Packers/OutputAddressGenerator.md?plain=1#L1-L121)

While all four packers share the same pipeline architecture, they differ in capabilities:

| **Packer 0** | `Dst` or L1 | Ports 2, 3, 4, 5 (read/write) | L1-to-L1 pack operations |
| **Packer 1** | `Dst` only | Ports 2, 3, 4, 5 (write only) | Standard packer |
| **Packer 2** | `Dst` only | Ports 2, 5 (write only) | Standard packer |
| **Packer 3** | `Dst` only | Port 6 (write only) | Standard packer |

## PACR Instruction Overview

The `PACR` instruction controls all four packers simultaneously, with per-packer enable bits in the `PackerMask` field. A single `PACR` instruction can operate zero, one, or multiple packers in parallel.

## Input Address Generation

Each packer's input address generation uses ADCs (Auto-incrementing Addressing Counters) to compute the starting address and determine how many datums to process.

### Address Calculation

The input address is computed using a 4D addressing scheme (X, Y, Z, W components):

```
InputAddr = Base + (ADC.X × Xstride) + (ADC.Y × Ystride) + (ADC.Z × Zstride) + (ADC.W × Wstride)
```

The specific ADC used is determined by:

*   `CurrentThread` (T0, T1, or T2) by default
*   `PackerIConfig.Addr_cnt_context` if `CurrentInstruction.OvrdThreadId` is set

## Pipeline Stages

### Stage 1: Input Address Generation

Described in the section above. Determines where to read input data from and how many datums to process.

### Stage 2: Transformation

Applies optional transformations to input data:

*   **Edge masking**: Zeroes out specific datums at tile boundaries
*   **ReLU activation**: `output = max(0, input)` for activation functions
*   **Downsampling**: Selects subset of input datums based on stride configuration

### Stage 3: Compression

For Block Floating-Point (BFP) output formats, generates:

*   **Row Start Indices (RSI)**: Metadata indicating where each row begins
*   **RLE (Run-Length Encoding)**: Counts of consecutive zero datums
*   **Shared exponents**: Common exponent for groups of mantissas

Compression can be disabled via `PackerIConfig.Disable_zero_compress` or globally via `Config[StateID].THCON_SEC0_REG1_All_pack_disable_zero_compress`.

### Stage 4: Format Conversion

Converts internal computation format to output format:

*   FP32 → BF16, FP16, FP8
*   TF32 → BF16, FP16
*   INT32 → INT16, INT8

Supports optional **stochastic rounding** using internal PRNGs for format conversions with precision loss.

### Stage 5: Accumulation

Optionally accumulates output with existing L1 data:

| Mode | Latency | Description |
| --- | --- | --- |
| **Atomic** | 5 cycles | Read-modify-write with automatic mutual exclusion |
| **Non-atomic** | 2 cycles | Faster but requires software synchronization |

Supports:

*   4×FP32 accumulation (128-bit)
*   4×INT32 accumulation with sign-magnitude representation and saturation
*   8×FP16 accumulation (128-bit)
*   8×BF16 accumulation (128-bit)

**Sources:**[WormholeB0/TensixTile/L1.md 10-13](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L10-L13)[WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)

### Stage 6: Output Address Generation

Generates separate addresses for up to three output streams. See next section for details.

### Output Alignment and Buffering

All packer output is 16-byte aligned:

```
ByteAddress = (Addr & 0x1ffff) << 4  // Always multiple of 16 bytes
```

Data is buffered in 16-byte chunks before being written to L1. If a buffer is flushed before containing 16 bytes (via `Last` or `Flush` flags), it is zero-padded to 16 bytes.

When a 16-byte buffer writes to L1, `ByteAddress` increments by 16.

## Configuration

Packer behavior is controlled by configuration stored in the `Config` and `ThreadConfig` arrays, accessible to both RISC-V cores and Tensix instructions.

### Per-Packer Configuration Structure

Each packer's configuration resides in `Config[StateID].Packers[i]` where `StateID` is determined by `ThreadConfig[CurrentThread].CFG_STATE_ID_StateID`:

**Key configuration fields:**

| Field | Location | Purpose |
| --- | --- | --- |
| `In_data_format` | `Config[StateID].Packers[i]` | Input data format (FP32, BF16, etc.) |
| `Out_data_format` | `Config[StateID].Packers[i]` | Output data format |
| `Source_interface_selection` | `Config[StateID].Packers[i]` | 0=Dst, 1=L1 (Packer 0 only) |
| `L1_source_addr` | `Config[StateID].Packers[i]` | High bits of L1 source address |
| `L1_Dest_addr` | `Config[StateID].Packers[i]` | Base output address in L1 |
| `Addr_cnt_context` | `Config[StateID].Packers[i]` | ADC selection override (0-2) |
| `Disable_zero_compress` | `Config[StateID].Packers[i]` | Disable compression for this packer |
| `Row_start_section_size` | `Config[StateID].Packers[i]` | Size of RSI metadata section |
| `Exp_section_size` | `Config[StateID].Packers[i]` | Size of exponent section |
| `Pack_fifo_size` | `Config[StateID].Packers[i]` | Circular buffer size (in 2-byte units) |
| `Pack_limit_address` | `Config[StateID].Packers[i]` | Circular buffer wrap threshold |
| `Add_l1_dest_addr_offset` | `Config[StateID].Packers[i]` | Enable TDMA-RISC offset addition |

### Address Modifier Configuration

The `AddrMod` field in `PACR` instructions selects one of multiple pre-configured address modification patterns:

Each `AddrMod` entry contains:

*   `YsrcIncr`, `YsrcClear`, `YsrcCR`: Input Y-coordinate control
*   `ZsrcIncr`, `ZsrcClear`: Input Z-coordinate control
*   `YdstIncr`, `YdstClear`, `YdstCR`: Output Y-coordinate control
*   `ZdstIncr`, `ZdstClear`: Output Z-coordinate control

## State Tracking and Introspection

Each packer maintains internal state that can be read using `SETDMAREG` instructions:

**Readable via `SETDMAREG`:**

*   `InputSource=0`: All four packers' `AccTileSize` and `LastTileSize`
*   `InputSource=1`: All four packers' `AllZeroFlags`
*   `InputSource=2-5`: Individual packer's tile header (size, format, flags)
*   `InputSource=6-7`: Individual packer's `ExponentHistogram` (32 bytes)
*   `InputSource=8`: All packers' first bit of `AllZeroFlags`, with optional reset
*   `InputSource=9`: Packer 0's maximum observed exponent

**Sources:**[WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md 1-116](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SETDMAREG_Special.md?plain=1#L1-L116)[WormholeB0/TensixTile/SoftReset.md 147-155](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L147-L155)

## Performance Characteristics

### Bandwidth Per Packer

| Operation | Bandwidth | Cycles per 128-bit |
| --- | --- | --- |
| **Standard write** | 128 bits/cycle | 1 cycle |
| **Atomic accumulate** | 25.6 bits/cycle | 5 cycles |
| **Non-atomic accumulate** | 64 bits/cycle | 2 cycles |

**Aggregate bandwidth (all 4 packers):**

*   Peak write: 512 bits/cycle = 64 bytes/cycle
*   Peak atomic accumulate: 102.4 bits/cycle
*   Peak non-atomic accumulate: 256 bits/cycle

**Sources:**[WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)

### L1 Port Contention

Packers compete for L1 access ports:

| Packer | Write Ports | Potential Conflicts |
| --- | --- | --- |
| **Packer 0** | 2, 5 | Unpackers 0/1 (port 2, 5), Packer 2 (port 2, 5) |
| **Packer 1** | 2, 5 | Unpackers 0/1 (port 2, 5), Packer 0 (port 2, 5), Packer 2 (port 2, 5) |
| **Packer 2** | 2, 5 | Unpackers 0/1 (port 2, 5), Packer 0 (port 2, 5), Packer 1 (port 2, 5) |
| **Packer 3** | 6 | None (dedicated port) |

**Packer 0 read operations** (L1-to-L1 mode) additionally use ports 2-5 for reading, creating potential conflicts with unpacker operations. Software should ensure unpackers are idle when performing L1-to-L1 packs.

**Sources:**[WormholeB0/TensixTile/L1.md 42-46](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L42-L46)

### Pipeline Latency

The complete packer pipeline (input → transformation → compression → format conversion → accumulation → output) is multi-cycle. Exact latency varies based on:

*   Data format conversions required
*   Compression enabled/disabled
*   Accumulation mode (none/atomic/non-atomic)
*   L1 bank conflicts

Software can use `STALLWAIT` conditions to synchronize with packer completion if needed.

**Sources:**[WormholeB0/TensixTile/L1.md 1-70](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/L1.md?plain=1#L1-L70)

## Soft Reset Behavior

Each packer has an independent soft reset bit in `RISCV_DEBUG_REG_SOFT_RESET_0`:

| Reset Bit | Effect |
| --- | --- |
| **Bit 2** | Packer 0 |
| **Bit 3** | Packer 1 |
| **Bit 4** | Packer 2 |
| **Bit 5** | Packer 3 |
| **Bit 17** | All packers' `Dst` connection and exponent histograms |

**Upon entering soft reset:**

*   In-flight `PACR` and `PACR_SETREG` instructions are aborted
*   Output buffers (before L1) are emptied
*   `AccTileSize` and `LastTileSize` set to zero
*   Metadata output FIFO emptied
*   (Bit 17 only) Exponent histograms reset, stochastic rounding PRNGs reset

**Upon leaving soft reset:**

*   `l1_dest_addr_offset` set to zero

**Sources:**[WormholeB0/TensixTile/SoftReset.md 146-173](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/SoftReset.md?plain=1#L146-L173)
