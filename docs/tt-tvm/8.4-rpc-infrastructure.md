## Purpose and Scope

The RPC (Remote Procedure Call) Infrastructure provides a distributed measurement and execution framework for TVM. It enables cross-compilation workflows where model compilation occurs on a host machine (e.g., x86 server) while performance measurements and execution happen on remote target devices (e.g., ARM boards, Android phones, embedded systems). This infrastructure is essential for auto-tuning on devices that differ from the compilation host.

For information about the auto-tuning systems that use RPC, see [AutoTVM](https://deepwiki.com/tenstorrent/tt-tvm/8.1-autotvm), [AutoScheduler](https://deepwiki.com/tenstorrent/tt-tvm/8.2-autoscheduler), and [Meta Schedule](https://deepwiki.com/tenstorrent/tt-tvm/8.3-meta-schedule). For deployment-focused runtime execution without tuning, see [Runtime and Execution](https://deepwiki.com/tenstorrent/tt-tvm/7-runtime-and-execution).

## RPC Tracker

The **RPC Tracker** is a centralized controller that manages device registration and session allocation. It maintains a registry of available devices and allocates them to clients on demand.

### Starting the Tracker

The tracker runs as a standalone process on the host machine:

`python -m tvm.exec.rpc_tracker --host=0.0.0.0 --port=9190`
Expected output:

```
INFO:RPCTracker:bind to 0.0.0.0:9190
```

### Querying Device Availability

Query the tracker to see registered devices:

`python -m tvm.exec.query_rpc_tracker --host=0.0.0.0 --port=9190`
Example output showing multiple device types:

```
Queue Status
----------------------------------
key          total  free  pending
----------------------------------
mate10pro    2      2     0
rk3399       2      2     0
rasp4b-64    11     11    0
----------------------------------
```

### Tracker Responsibilities

**Tracker Operation**: The tracker maintains device availability, queues client requests, and monitors session health through heartbeats.

## RPC Server

The **RPC Server** runs on each remote device and provides runtime execution capabilities. It registers with the tracker and handles incoming execution requests.

### Server Registration Modes

#### Direct Registration Mode

For Linux-based devices, the server connects directly to the tracker:

`python -m tvm.exec.rpc_server --tracker=[HOST_IP]:9190 --key=rasp4b-64`
Parameters:

*   `--tracker`: Tracker IP address and port
*   `--key`: Device identifier for grouping similar devices
*   `--port`: (Optional) Local port for incoming connections

#### Android RPC App

For Android devices, a dedicated RPC application is used. The app provides a GUI for configuration and automatically registers with the tracker when connected.

Configuration requirements:

*   Developer option: "Keep screen awake during charging" (for stable measurements)
*   Network connectivity to tracker
*   TVM runtime compiled with Android NDK

### Server Capabilities

**Server Capabilities**: The RPC server provides comprehensive device access including module loading, memory management, and execution on available compute devices.

## RPC Client and Session

The **RPC Client** API enables host programs to request devices from the tracker and execute operations remotely through **RPC Sessions**.

### Connecting to Tracker

`# Python APIfrom tvm import rpc # Connect to trackertracker = rpc.connect_tracker(tracker_host, tracker_port) # Request a device by keyremote = tracker.request(    key="rasp4b-64",      # Device key    priority=0,            # Request priority (lower = higher priority)    session_timeout=60     # Session timeout in seconds)`

For testing without remote devices:

`# Create a local session (simulates remote but executes locally)remote = rpc.LocalSession()`

## Integration with Auto-Tuning

The RPC infrastructure is tightly integrated with TVM's auto-tuning systems, enabling distributed measurement across heterogeneous devices.

### RPCRunner Configuration

The `RPCRunner` class provides the bridge between auto-tuning and RPC:

`from tvm import auto_scheduler # Configure RPCRunner for AutoSchedulerrunner = auto_scheduler.RPCRunner(    key="rasp4b-64",                  # Device key from tracker    host="127.0.0.1",                 # Tracker host    port=9190,                        # Tracker port    timeout=30,                       # Measurement timeout (seconds)    repeat=1,                         # Number of repeats per measurement    min_repeat_ms=200,                # Minimum time per repeat    enable_cpu_cache_flush=True       # Flush CPU cache for accurate timing) # Use in tuning optionstune_option = auto_scheduler.TuningOptions(    num_measure_trials=200,    builder=auto_scheduler.LocalBuilder(build_func="ndk"),  # or "default"    runner=runner,    measure_callbacks=[auto_scheduler.RecordToFile(log_file)])`
### Builder and Runner Separation

**Builder-Runner Architecture**: The builder compiles on the host (potentially with cross-compilation), while the runner uploads and measures on the actual target device via RPC.

### Parallel Measurements

The tracker enables parallel measurements across multiple devices:

`# Register multiple devices with the same key# Device 1: python -m tvm.exec.rpc_server --tracker=HOST:9190 --key=rasp4b-64# Device 2: python -m tvm.exec.rpc_server --tracker=HOST:9190 --key=rasp4b-64# ...# Device N: python -m tvm.exec.rpc_server --tracker=HOST:9190 --key=rasp4b-64 # The tuner automatically distributes trials across all available devicestuner = auto_scheduler.TaskScheduler(tasks, task_weights)tuner.tune(tune_option)  # Uses all registered devices in parallel`

## Advanced Features

### Proxy Mode for NAT Traversal

When devices are behind NAT or firewalls, proxy mode allows devices to connect out to the tracker instead of accepting incoming connections:

`# On host: Start tracker with proxy supportpython -m tvm.exec.rpc_tracker --host=0.0.0.0 --port=9190 # On device: Connect to tracker in proxy modepython -m tvm.exec.rpc_server \    --tracker=[PUBLIC_IP]:9190 \    --key=device_key \    --port=0  # Ephemeral port`
The device initiates the connection to the tracker and maintains it, allowing the tracker to forward client requests through the existing connection.

### Java RPC API

TVM provides a Java RPC API for integration with JVM-based systems:

`import org.apache.tvm.rpc.Server;import org.apache.tvm.rpc.Client;import org.apache.tvm.rpc.RPCSession; // Start standalone RPC serverServer server = new Server(port);server.start(); // Or connect to proxyServer server = new Server(proxyHost, proxyPort, "device_key");server.start(); // Client connectionRPCSession client = Client.connect("127.0.0.1", port);Function func = client.getFunction("test.rpc.strcat");String result = func.call("abc", 11L).asString();`
### Device-Specific Optimizations

Different device types require specific configurations:

| Device Type | Target String | Builder | Notes |
| --- | --- | --- | --- |
| ARM CPU | `llvm -mtriple=aarch64-linux-gnu -mattr=+neon` | `default` or `ndk` | Enable NEON for better performance |
| Android | `llvm -mtriple=aarch64-linux-android` | `ndk` | Requires NDK cross-compiler |
| Mali GPU | `opencl -device=mali` | `ndk` | Use OpenCL backend |
| CUDA GPU | `cuda` | `default` | Direct NVPTX compilation |
| Hexagon DSP | `hexagon` | Custom | Requires Hexagon SDK |

### Measurement Reliability

For accurate performance measurements on ARM devices:

`runner = auto_scheduler.RPCRunner(    key="device_key",    host="127.0.0.1",    port=9190,    repeat=3,                        # Multiple measurements    min_repeat_ms=500,               # Run at least 500ms    enable_cpu_cache_flush=True,     # Flush cache between runs    cooldown_interval=0.1            # Wait between measurements)`
For Android devices:

*   Enable "Keep screen awake during charging" in developer options
*   Ensure stable power supply (plugged in)
*   Disable CPU frequency scaling if possible
*   Close background applications

## Performance Considerations

### Tracker Scalability

The tracker is designed to handle large device pools:

*   **Concurrent Sessions**: Supports hundreds of concurrent RPC sessions
*   **Device Pooling**: Groups devices by key for efficient allocation
*   **Priority Queuing**: Higher-priority requests (lower priority value) are served first
*   **Session Timeouts**: Automatically reclaims hung sessions

### Network Overhead

RPC introduces network latency. Optimize by:

1.   **Batch Operations**: Upload all required files in one session
2.   **Binary Size**: Minimize library size (use AOT compilation when possible)
3.   **Local Tracker**: Run tracker on same network as devices
4.   **Measurement Duration**: Use `min_repeat_ms` ≥ 200ms to amortize overhead

### Multi-Device Tuning

When using multiple devices for auto-tuning:

`# Query available devices# python -m tvm.exec.query_rpc_tracker --host=0.0.0.0 --port=9190# Shows: rasp4b-64: total=11, free=11, pending=0 # Tuner automatically parallelizes across all 11 devicestuner = auto_scheduler.TaskScheduler(tasks, task_weights)tuner.tune(tune_option)  # Distributes 200 trials across 11 devices # Expected speedup: ~11x faster than single device# (with some overhead for coordination)`

## Summary

The RPC Infrastructure provides a robust, scalable framework for distributed compilation and measurement in TVM:

| Component | Purpose | Key Classes/Modules |
| --- | --- | --- |
| **RPC Tracker** | Device registry and session management | `tvm.exec.rpc_tracker` |
| **RPC Server** | Remote execution runtime | `tvm.exec.rpc_server` |
| **RPC Client/Session** | Host-side API for remote access | `tvm.rpc.connect_tracker()`, `tvm.rpc.RPCSession` |
| **RPCRunner** | Auto-tuning integration | `auto_scheduler.RPCRunner`, `autotvm.RPCRunner` |
| **Utilities** | Helper functions | `auto_scheduler.utils.request_remote()` |

This architecture enables efficient cross-compilation workflows, parallel auto-tuning, and deployment testing across heterogeneous hardware targets, making it essential for optimizing models for ARM devices, mobile phones, embedded systems, and accelerators that differ from the compilation host.
