This page describes instruction scheduling considerations and hazard management for both the Baby RISC-V cores and the Tensix Coprocessor. It covers instruction latencies, data hazards (RAW, WAW, WAR), and strategies for hiding latencies through careful instruction scheduling. For information about memory access patterns and bandwidth optimization, see [Memory Access Patterns and Optimization](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.1-memory-access-patterns-and-optimization). For information about structuring compute kernels and using the MOP/Replay Expanders, see [Kernel Structure and MOP Usage](https://deepwiki.com/tenstorrent/tt-isa-documentation/7.3-kernel-structure-and-mop-usage).

## Hazard Types

Data hazards occur when instructions have dependencies that prevent them from executing in the ideal pipelined fashion. Three types of data hazards exist:

**Read-After-Write (RAW)**: Also called a "true dependency". Instruction B reads a location that instruction A writes to. Instruction B must wait for instruction A to complete its write.

**Write-After-Write (WAW)**: Also called an "output dependency". Both instruction A and instruction B write to the same location. The writes must occur in program order to produce correct results.

**Write-After-Read (WAR)**: Also called an "anti-dependency". Instruction B writes to a location that instruction A reads from. Instruction B must wait for instruction A to complete its read before overwriting the data.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 28-73](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L28-L73)

## Baby RISC-V Pipeline and Instruction Latencies

### Pipeline Structure

Each Baby RISC-V core uses a 4-stage in-order pipeline with significant reordering capability in the Load/Store Unit. Instructions flow through: Frontend → Integer Unit → Load/Store Unit → Retire Unit.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 28-76](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L28-L76)

### Integer Unit Latencies

| Instruction Type | Cycles in Integer Unit | Notes |
| --- | --- | --- |
| Most instructions | 1 | Forwarding path allows latency 1 throughput 1 |
| `mul`, `mulh`, `mulhsu`, `mulhu` | 2 | Unit blocked during execution |
| `div`, `divu`, `rem`, `remu` | 2-33 | Depends on dividend/divisor values |
| Branch (mispredicted) | 3 (effective) | Includes 2-cycle bubble |

The forwarding path allows the result of an integer operation to be consumed by the very next instruction, appearing as latency 1 and throughput 1 from an external perspective.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 40-53](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L40-L53)

### Load/Store Unit Latencies

The Load/Store Unit can queue up to 8 instructions at any time: the oldest non-retired load, plus up to 7 subsequent instructions in program order. Load latency depends on the memory region accessed:

| Memory Region | Load Latency (cycles) | Max Loads In Flight |
| --- | --- | --- |
| Core-local data RAM | 2 | 8 |
| Mailboxes, PCBufs, TTSync, Tensix semaphores | ≥ 3 | 4 (aggregate) |
| Tensix GPRs, Tensix backend configuration | ≥ 4 | 4 (aggregate) |
| TDMA-RISC, Tile control, PIC, NoC config, NoC overlay | ≥ 7 | 4 (aggregate) |
| L1 scratchpad RAM (1464 KiB) | ≥ 8 | 4 (aggregate) |

**Latency Hiding**: A latency of `N` cycles means that `N - 1` independent instructions following the load will completely hide the latency.

**Throughput**:

*   Load latency < 5 cycles: 1 load per cycle sustained throughput
*   Load latency = `N` cycles (N ≥ 5): 4 loads every `N - 1` cycles

**Store Throughput**:

*   Stores to L1: At most 1 store every 5 cycles
*   Stores to other regions: 1 store per cycle (subject to conflicts)

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 54-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L54-L71)

## Tensix Coprocessor Instruction Latencies

### Matrix Unit (FPU) Instructions

The Matrix Unit executes instructions with varying latencies and throughput:

| Instruction(s) | Throughput (IPC) | Latency (cycles) | Notes |
| --- | --- | --- | --- |
| `MVMUL`, `DOTPV`, `GAPOOL`, `ELWMUL` | 1 (†) | 5 | † Reduced when multiple fidelity phases used |
| `GMPOOL`, `ELWADD`, `ELWSUB` | 1 | 5 |  |
| `SETRWC`, `INCRWC`, `CLEARDVALID`, `CLREXPHIST`, `GATESRCRST` | 1 | 1 |  |
| `SHIFTXA`, `ZEROACC`, `ZEROSRC`, `TRNSPSRCB` | 1 | 1 |  |
| `SHIFTXB` | 0.5 | 2 |  |
| `MOVD2A` | 1 | 2 (‡) | ‡ Only certain instructions can hide this |
| `MOVA2D`, `MOVDBGA2D`, `MOVB2D`, `MOVB2A` | 1 | 4 (‡) | ‡ Only certain instructions can hide this |

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 21-36](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L21-L36)

### Vector Unit (SFPU) Instructions

**SFPMAD Scheduling**: `SFPMAD` (Vector Unit multiply-add) requires 2 cycles to compute its result. The handling of this latency differs between architectures:

*   **Blackhole**: Hardware automatically stalls if the next instruction reads from a location written by `SFPMAD` (with some exceptions noted below)
*   **Wormhole**: Software must manually ensure a 1-cycle gap between `SFPMAD` and any instruction reading its output

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-87](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L87)[WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 61-64](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L61-L64)

### Unpacker and Packer Latencies

Unpacker and Packer operations are multi-cycle and bandwidth-limited:

*   **Unpackers**: Multi-cycle, dependent on L1 bandwidth and format conversion complexity
*   **Packers**: Multi-cycle with complex pipeline including format conversion, compression, and accumulation 
    *   Non-atomic accumulation: 2 cycles
    *   Atomic accumulation: 5 cycles

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 1-76](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L1-L76)

## Automatic vs Manual Stalling

### Blackhole: Automatic Instruction Scheduling

Blackhole introduces hardware-based automatic stalling for `SFPMAD` instructions. When an `SFPMAD` writes to an `LReg` and the next instruction attempts to read from that `LReg`, hardware automatically inserts a 1-cycle stall.

**Limitations of Automatic Stalling**:

Hardware does not detect all hazard cases. Software must manually insert `SFPNOP` for gaps when:

*   Using `SFPMAD_MOD1_INDIRECT_VA` or `SFPMAD_MOD1_INDIRECT_VD` (hardware conservatively assumes all `LReg` accessed)
*   Following `SFPMAD` with `SFPAND` (when `SFPAND_MOD1_USE_VB` is used)
*   Following `SFPMAD` with `SFPOR` (when `SFPOR_MOD1_USE_VB` is used)
*   Following `SFPMAD` with `SFPIADD` (hardware doesn't detect VD read)
*   Following `SFPMAD` with `SFPSHFT` (hardware doesn't detect VD read)
*   Following `SFPMAD` with `SFPCONFIG` (hardware doesn't detect potential `LReg[0]` read)
*   Following `SFPMAD` with `SFPSWAP` in certain modes (hardware doesn't detect 1st cycle reads)
*   Following `SFPMAD` with `SFPSHFT2` in certain modes (hardware doesn't detect reads)

**Exception for SFPLOADMACRO**: Automatic stalling does NOT apply to `SFPMAD` instructions executed as part of an `SFPLOADMACRO` sequence. Software must ensure manual gaps.

Sources: [BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 68-87](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L68-L87)

### Wormhole: Manual Instruction Scheduling

Wormhole requires software to manually manage all instruction scheduling for the Vector Unit. If `SFPMAD` is used, software must ensure that the next cycle does not execute a Vector Unit instruction that reads from any location written by `SFPMAD`. An `SFPNOP` instruction can be inserted to create this gap.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 61-64](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L61-L64)

## Specific Scheduling Requirements

### Matrix Unit Data Movement Instructions

Certain Matrix Unit instructions impose restrictions on what can follow them:

**MOVB2A Restrictions**:

*   If `MOVB2A` is used, on the next cycle the Matrix Unit can only accept `MOVD2A` or `MOVB2A`
*   Hardware automatically stalls the thread for 1 cycle if any other Matrix Unit instruction is presented
*   Software may wish to use `STALLWAIT` (block bit B6, condition code C10) prior to `MOVB2A` to ensure `SrcA[MatrixUnit.SrcABank].AllowedClient == SrcClient::MatrixUnit`

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 64-69](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L64-L69)

### Wait Gate Synchronization

The Tensix Coprocessor frontend includes a Wait Gate that enforces synchronization constraints by checking semaphores. Instructions may spend time waiting at the Wait Gate before being dispatched to backend execution units. This waiting is not visible as instruction latency but affects throughput.

Sources: [WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md 24-29](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MOVB2A.md?plain=1#L24-L29)

## Strategies for Hiding Latencies

### Independent Instruction Scheduling

The primary strategy for achieving high performance is to schedule independent instructions between a producer and consumer to hide latency:

**For Baby RISC-V**:

*   **2-cycle latency** (local RAM loads): 1 independent instruction needed
*   **8-cycle latency** (L1 loads): 7 independent instructions needed
*   Use local RAM for frequently accessed data (stack, thread-local variables) to minimize latency

**For Matrix Unit**:

*   **5-cycle latency** (most compute ops): Schedule 4-5 independent instructions or use multi-buffering
*   **2-cycle latency** (`MOVD2A`): Schedule 1-2 independent instructions
*   **4-cycle latency** (other MOV ops): Schedule 3-4 independent instructions

**For Vector Unit**:

*   **2-cycle latency** (`SFPMAD`): Schedule 1 independent instruction or use `SFPNOP` (Wormhole only; Blackhole auto-stalls)

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 117-120](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L117-L120)[WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md 21-36](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/MatrixUnit.md?plain=1#L21-L36)

### Multi-Buffering and Double-Buffering

When independent instructions are scarce, use multiple buffers to overlap computation with data movement:

**Dst Register Multi-Buffering**:

*   Use different rows of `Dst` for different operations
*   While computation operates on `Dst[0-15]`, packers write out `Dst[16-31]`
*   RWCs (Read-Write Counters) facilitate automatic cycling through buffer regions

**SrcA/SrcB Multi-Buffering**:

*   While Matrix Unit computes using `SrcA[0-15]` and `SrcB[0-15]`, unpackers fill `SrcA[16-31]` and `SrcB[32-47]`
*   Alternate between buffer regions using RWC manipulation

Sources: [WormholeB0/TensixTile/TensixCoprocessor/RWCs.md 1-78](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/RWCs.md?plain=1#L1-L78)

### Loop Unrolling

Unroll loops to create more opportunities for instruction reordering:

*   Increases the pool of independent instructions available for scheduling
*   Allows better overlap between different loop iterations
*   Use MOP Expander and Replay Expander to reduce RISC-V instruction fetch overhead

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 28-76](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L28-L76)

### Load/Store Unit Reordering

The Baby RISC-V Load/Store Unit can queue up to 8 instructions and reorder them internally:

*   Issue multiple loads early to allow them to execute in parallel
*   Place independent stores between loads
*   Rely on the unit's internal reordering to optimize execution

The Load/Store Unit maintains in-order entry and exit, but significant reordering occurs internally, which can help hide load latencies.

Sources: [WormholeB0/TensixTile/BabyRISCV/README.md 54-71](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/BabyRISCV/README.md?plain=1#L54-L71)

### SFPLOADMACRO for Vector Unit

Use `SFPLOADMACRO` to achieve sustained throughput on the Vector Unit by overlapping data movement with computation. This instruction sequence allows one `SFPMAD` per cycle while simultaneously loading/storing data, achieving the theoretical peak of 0.064 TFLOP/s (Wormhole) or 0.0864 TFLOP/s (Blackhole).

Sources: [WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md 65-68](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/WormholeB0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L65-L68)[BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md 88-91](https://github.com/tenstorrent/tt-isa-documentation/blob/9600a914/BlackholeA0/TensixTile/TensixCoprocessor/SFPMAD.md?plain=1#L88-L91)
