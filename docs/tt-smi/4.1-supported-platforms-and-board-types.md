This page documents the hardware platforms and board types supported by TT-SMI, including board identification mechanisms, platform-specific features, and Galaxy system architecture. For information about telemetry collection methods specific to each platform, see [Telemetry Collection and Monitoring](https://deepwiki.com/tenstorrent/tt-smi/4.2-telemetry-collection-and-monitoring). For reset operation differences across platforms, see [Reset Architecture and Types](https://deepwiki.com/tenstorrent/tt-smi/5.1-reset-architecture-and-types).

## Overview

TT-SMI supports three hardware generations from Tenstorrent: **Wormhole**, **Blackhole**, and **Galaxy systems** (both Wormhole and Blackhole variants). As of version 3.0.35, **Grayskull** support has been deprecated and removed from the codebase. Platform detection occurs through the `PciChip` interface from pyluwen, with board-specific identification derived from board ID values.

**Sources**: [README.md 15](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/README.md?plain=1#L15-L15)[CHANGELOG.md 8-15](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/CHANGELOG.md?plain=1#L8-L15)[tt_smi/tt_smi_backend.py 98-105](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L98-L105)

## Supported Hardware Generations

TT-SMI currently supports the following hardware platforms:

| Platform | Status | Device Detection | Primary Use Case | Reset Type |
| --- | --- | --- | --- | --- |
| **Wormhole** | Active | `device.as_wh()` | Standard PCIe boards (n-series, nb-series) | Board-level |
| **Blackhole** | Active | `device.as_bh()` | Standard PCIe boards (p-series) | ASIC-level |
| **Galaxy WH** | Active | `device.as_wh()` + board type `0x35` | Large-scale systems (32 devices, 4 trays) | IPMI-based |
| **Galaxy BH** | Active | `device.as_bh()` + board type `0x47` | Large-scale systems (32 devices, 4 trays) | IPMI-based |
| **Grayskull** | Deprecated (v3.0.35) | N/A | Legacy e-series boards | Tensix-level |

## Wormhole Board Types

Wormhole devices are identified via `device.as_wh()` and support the following board configurations:

| Board Type | UPI Value | Description | Device Count | Remote Device Support |
| --- | --- | --- | --- | --- |
| `nb_cb` | `0x8` | Nebula carrier board | 1 | No |
| `wh_4u` | `0xB` | 4U rack-mount configuration | Variable | No |
| `n300` | `0x14` | Dual-chip configuration | 2 | Yes (L/R suffix) |
| `n150` | `0x18` | Single-chip configuration | 1 | No |
| `tt-galaxy-wh` | `0x35` | Galaxy 6U system | 32 (4 trays × 8) | No |

### Wormhole-Specific Features

*   **Local/Remote Designation**: For `n300` boards, TT-SMI appends " L" (local) or " R" (remote) suffixes to distinguish the two ASICs on the same board.
*   **Remote Device Detection**: The `device.is_remote()` check identifies remote ASICs on multi-chip boards.
*   **Coordinate Reporting**: Wormhole devices report shelf coordinates via `device.as_wh().get_local_coord()`.

**Sources**: [tt_smi/tt_smi_backend.py 143-403](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L143-L403)[tt_smi/tt_smi_backend.py 678-689](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L678-L689)

## Blackhole Board Types

Blackhole devices are identified via `device.as_bh()` and include the following boards:

| Board Type | UPI Value | Description | Target Market |
| --- | --- | --- | --- |
| `bh-scrappy` | `0x36` | Early engineering board | Development |
| `p100a` | `0x43` | P100 variant A | Production |
| `p150a` | `0x40` | P150 variant A | Production |
| `p150b` | `0x41` | P150 variant B | Production |
| `p150c` | `0x42` | P150 variant C | Production |
| `p300a` | `0x45` | P300 variant A | Production |
| `p300b` | `0x44` | P300 variant B | Production |
| `p300c` | `0x46` | P300 variant C | Production |
| `tt-galaxy-bh` | `0x47` | Galaxy 6U system | Large-scale deployment |

### Blackhole-Specific Features

*   **TAG-based Telemetry**: Blackhole uses a different telemetry enumeration (`BH_TELEMETRY_LIST`) compared to Wormhole's SMBus-based system.
*   **ASIC-level Resets**: Blackhole boards support ASIC-level resets via `BHChipReset().full_lds_reset()`.
*   **No Remote Devices**: Unlike Wormhole n300, Blackhole boards do not have local/remote ASIC pairs.

**Sources**: [tt_smi/tt_smi_backend.py 691-710](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L691-L710)[tt_smi/constants.py 66-100](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/constants.py#L66-L100)

### Board Type Display in GUI

When displaying board information, TT-SMI applies special formatting:

1.   **For Wormhole n300 boards**: Appends " L" (local) or " R" (remote) suffix to distinguish the two ASICs
2.   **For standard boards**: Uses the board type string directly from `get_board_type()`
3.   **For invalid/unreadable boards**: Displays "N/A"

## Remote Device Handling (Wormhole Only)

Wormhole n300 boards contain two ASICs that communicate over an internal fabric. TT-SMI handles these as follows:

| Characteristic | Local Device | Remote Device |
| --- | --- | --- |
| PCI Interface ID | Available via `get_pci_interface_id()` | Returns "N/A" |
| Board Type Suffix | Appends " L" | Appends " R" |
| Reset Capability | Yes (via PCI index) | No (reset via local device) |
| Detection Method | `device.is_remote() == False` | `device.is_remote() == True` |
| PCI Properties | Read from sysfs | All "N/A" |

**Sources**: [tt_smi/tt_smi_backend.py 165-200](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L165-L200)[tt_smi/tt_smi_backend.py 312-313](https://github.com/tenstorrent/tt-smi/blob/a2791bfe/tt_smi/tt_smi_backend.py#L312-L313)
