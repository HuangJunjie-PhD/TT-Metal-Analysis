The **Model Specification Registry** is the central source of truth for all models supported by the `tt-inference-server`. It defines the relationship between model weights, inference engines, hardware configurations, and performance targets. This registry enables the automated orchestration of Docker containers, benchmarking suites, and evaluation pipelines by providing a structured schema for model metadata.

## Core Data Structures

The registry is built on a hierarchy of dataclasses that separate implementation details from hardware-specific constraints.

### ModelSpec and ModelSpecTemplate

The `ModelSpec` represents a fully resolved configuration for a specific model running on a specific device. To avoid redundancy, the registry primarily uses `ModelSpecTemplate` to define a model family once and then expand it across multiple hardware targets.

*   **ModelSpecTemplate**: Defines shared attributes like `tt_metal_commit`, `vllm_commit`, and the base `ImplSpec`. It contains a list of `DeviceModelSpec` objects [workflows/model_spec.py 277-313](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L277-L313)
*   **ModelSpec**: The expanded object containing the unique `model_id` (e.g., `id_vllm_llama-3.1-8b-instruct_n150`). It is generated by the `expand_to_specs()` method [workflows/model_spec.py 315-385](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L315-L385)

### DeviceModelSpec

This class defines how a model behaves on specific Tenstorrent hardware. It includes:

*   **Hardware Constraints**: `max_concurrency` and `max_context`[workflows/model_spec.py 195-196](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L195-L196)
*   **vLLM Arguments**: `vllm_args` (e.g., `num_scheduler_steps`, `block_size`) which are passed directly to the inference engine [workflows/model_spec.py 199](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L199-L199)
*   **TT Config**: `override_tt_config` for low-level metal parameters like `data_parallel` or `tensor_parallel`[workflows/model_spec.py 200](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L200-L200)
*   **System Requirements**: Hardware-specific firmware or KMD version requirements [workflows/model_spec.py 201](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L201-L201)

### ImplSpec

Defines the software implementation details, including the repository URL and the specific code path within the container used to execute the model [workflows/model_spec.py 143-152](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L143-L152)

## The MODEL_SPECS Registry

All models are registered in the `MODEL_SPECS` dictionary, which is populated by calling `get_model_spec_map()` on the `spec_templates` list [workflows/model_spec.py 1367-1375](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L1367-L1375)

### Hardware Inference and vllm_args Merging

When a user executes a model, the system resolves the `RuntimeConfig`. If specific `vllm_args` are provided via CLI, they are merged with the defaults defined in the `DeviceModelSpec`.

1.   **Selection**: The `MODEL_SPECS` dictionary is keyed by `model_id`[workflows/model_spec.py 433](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L433-L433)
2.   **Merging**: Runtime overrides from `RuntimeConfig` take precedence over `DeviceModelSpec` defaults during container command generation.
3.   **Hardware Inference**: The `DeviceTypes` enum [workflows/workflow_types.py 60-79](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/workflow_types.py#L60-L79) is used to validate that the requested model supports the target hardware, such as checking `is_wormhole()` or `is_blackhole()`[workflows/workflow_types.py 151-172](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/workflow_types.py#L151-L172)
4.   **Data Parallel Scaling**: The `get_perf_reference` function detects `data_parallel` settings in `override_tt_config` and adjusts the target device and performance metrics accordingly [workflows/model_spec.py 163-177](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L163-L177)

## Runtime Overrides and Versioning

The registry supports dynamic overrides at runtime through the `RuntimeConfig`. This allows developers to test different hardware configurations or commits without modifying the static registry.

*   **Dynamic JSON Loading**: The system can load a `ModelSpec` from a temporary JSON file via `ModelSpec.from_json()`[workflows/model_spec.py 440-453](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L440-L453)
*   **Docker Tagging**: Docker images are dynamically resolved using `generate_default_docker_link`, which combines the version, `tt_metal_commit`, and `vllm_commit` from the spec to ensure environment consistency [workflows/model_spec.py 45-62](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/workflows/model_spec.py#L45-L62)
*   **CI Configuration**: The `models-ci-config.json` file further maps these models to specific CI gates (nightly, weekly, release) and their required test devices [.github/workflows/models-ci-config.json 1-300](https://github.com/tenstorrent/tt-inference-server/blob/8ab207f9/.github/workflows/models-ci-config.json#L1-L300)
