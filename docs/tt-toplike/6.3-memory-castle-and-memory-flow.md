# Memory Castle and Memory Flow

Memory Castle and Memory Flow are full-screen ASCII visualizations that translate hardware telemetry into dynamic particle systems. While Memory Castle focuses on the vertical movement of data through the memory hierarchy (DDR → L2 → L1 → Tensix), Memory Flow focuses on the spatial distribution and NoC (Network on Chip) traffic between the DDR perimeter and the Tensix grid.

## Memory Castle (Memory Dungeon)

Memory Castle is a roguelike-inspired visualization where memory operations are represented as particles flowing through "rooms" or "vaults" corresponding to physical hardware layers [src/animation/memory_castle.rs 5-17](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_castle.rs#L5-L17)

### Particle System and Types

The system supports up to 600 `MemoryParticle` instances by default [src/config.rs 164-166](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/config.rs#L164-L166) Each particle has a `ParticleType` that determines its visual representation, velocity, and time-to-live (TTL).

| Type | Character Set | Velocity | TTL | Hardware Meaning |
| --- | --- | --- | --- | --- |
| `Read` | `◌`, `○`, `◉` | Fast (0.6) | 50 | Data moving from DDR to Tensix |
| `Write` | `□`, `▣`, `■` | Medium (0.4) | 60 | Data moving from Tensix to DDR |
| `CacheHit` | `◇`, `◈`, `◆` | Very Fast (0.8) | 40 | Rapid access from L1/L2 |
| `CacheMiss` | `∘`, `●`, `⬤` | Slow (0.3) | 70 | High latency access |

### Implementation Details

Particles are managed in a `Vec<MemoryParticle>` and updated every frame. They transition through four layers based on their Y-coordinate:

1.   **Layer 0:** DDR (Bottom)
2.   **Layer 1:** L2 Cache
3.   **Layer 2:** L1 SRAM
4.   **Layer 3:** Tensix Cores (Top)

The system implements a Z-order stack for rendering and a trail system that stores the last 8 positions of each particle to create a "fade-out" ribbon effect [src/animation/memory_castle.rs 62-63](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_castle.rs#L62-L63)[src/animation/memory_castle.rs 180-186](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_castle.rs#L180-L186)

## Memory Flow (MemoryFlowVis)

`MemoryFlowVis` provides a spatial representation of the chip, mapping DDR channels to the perimeter and the Tensix grid to the center. It visualizes memory access patterns and NoC traffic [src/animation/memory_flow.rs 5-15](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L5-L15)

### Flow Dynamics

Particles in this system, defined by `MemoryFlowParticle`, move between coordinates normalized from 0.0 to 1.0 [src/animation/memory_flow.rs 50-71](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L50-L71)

*   **Read Flow:** Particles spawn at the top edge (DDR) and move toward the center (Tensix) [src/animation/memory_flow.rs 74-96](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L74-L96)
*   **Write Flow:** Particles spawn in the center and move toward the bottom edge [src/animation/memory_flow.rs 98-118](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L98-L118)
*   **Intensity:** Driven by bandwidth utilization; higher activity results in more frequent spawns and brighter particles [src/animation/memory_flow.rs 18-22](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L18-L22)

### Heat Map Integration

The `MemoryFlowVis` struct maintains a `core_heat` vector (a grid of 0.0-1.0 values) [src/animation/memory_flow.rs 178-180](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L178-L180) This heat map is updated based on per-core power consumption, where the color of the "cores" in the visualization transitions from cool to hot based on `telemetry.asic_temp_c`[src/animation/memory_flow.rs 21-22](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L21-L22)

## Animation Sensitivity and Profiles

Both systems are governed by the `AnimationProfile` system, which adjusts the density and responsiveness of the particles [docs/superpowers/plans/2026-05-22-chip-portrait-particles.md 7-9](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/docs/superpowers/plans/2026-05-22-chip-portrait-particles.md?plain=1#L7-L9)

### Configuration Parameters

The `AnimConfig` struct translates profiles into multipliers:

*   **Sensitivity:** Multiplied into `AdaptiveBaseline::power_change()`. Range: 0.2 (AnomaliesOnly) to 5.0 (Paranoid) [src/config.rs 146-147](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/config.rs#L146-L147)
*   **Max Particles Scale:** Scales the particle caps. Range: 0.5 to 3.0 [src/config.rs 148-151](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/config.rs#L148-L151)

### Implementation in Viz Classes

Both `MemoryCastle` and `MemoryFlowVis` implement a `set_sensitivity(sensitivity: f32)` method that updates their internal `AdaptiveBaseline`[src/animation/memory_castle.rs 231-233](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_castle.rs#L231-L233)[src/animation/memory_flow.rs 214-216](https://github.com/tenstorrent/tt-toplike/blob/487bfe5a/src/animation/memory_flow.rs#L214-L216) This allows the UI to dynamically change animation vigor without re-instantiating the visualization objects.
